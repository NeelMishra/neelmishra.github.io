Ring Attention

ML Series · Transformers · Variants

1 · The Memory Wall

Flash Attention solved the compute efficiency of attention by tiling. But it doesn't solve the memory problem for extremely long sequences: the Q, K, V tensors themselves must fit on a single GPU. For a 1M-token sequence with d=128:

Q, K, V memory per head: 3 × 1,000,000 × 128 × 2 bytes = 768 MB per head × 32 heads = 24 GB just for Q, K, V (one layer!) KV cache across 80 layers: 2 × 80 × 32 × 1,000,000 × 128 × 2 = 1.3 TB No single GPU can hold this. We need to distribute across devices.

Ring Attention distributes the sequence across P devices, so each device only handles N/P tokens — making the context length scale linearly with the number of GPUs.

2 · The Ring Idea

Arrange P GPUs in a ring. Each GPU holds one chunk of the sequence (N/P tokens). To compute attention, each GPU needs to see all K/V chunks — so the K/V blocks circulate around the ring:

Split the sequence into P chunks. GPU p holds Q_p, K_p, V_p (tokens p·N/P to (p+1)·N/P).

In each step, every GPU sends its current K/V chunk to the next GPU in the ring and receives a K/V chunk from the previous GPU. After P-1 rotations, every GPU has seen every K/V chunk.

At each step, while K/V are being transferred, the GPU computes attention between its local Q chunk and the K/V chunk it currently holds. Uses Flash Attention's online softmax to incrementally accumulate results.

3 · Ring Topology

Ring of P = 4 GPUs: GPU 0 → GPU 1 → GPU 2 → GPU 3 → GPU 0 (circular) Step 0: GPU p computes attn(Q_p, K_p, V_p) — local chunk Step 1: K/V rotate: GPU p receives K_{p-1}, V_{p-1} GPU p computes attn(Q_p, K_{p-1}, V_{p-1}) Step 2: K/V rotate again GPU p computes attn(Q_p, K_{p-2}, V_{p-2}) Step 3: Final rotation GPU p computes attn(Q_p, K_{p-3}, V_{p-3}) After P steps: each GPU has attended to ALL K/V chunks
Why a ring? Ring all-reduce is the most bandwidth-efficient collective for homogeneous interconnects (NVLink, InfiniBand). Each device sends/receives exactly (P-1)/P of the total data, and all P links are utilized simultaneously.

4 · The Algorithm

# Ring Attention — each GPU p runs this in parallel def ring_attention(Q_p, K_p, V_p, rank, world_size): O_p = zeros_like(Q_p) l_p = zeros(Q_p.shape[0]) # running log-sum-exp m_p = full(Q_p.shape[0], -∞) # running max kv_recv = (K_p, V_p) # start with local KV for step in range(world_size): K_cur, V_cur = kv_recv # Async: send KV to next, receive KV from previous if step < world_size - 1: send_async(kv_recv, dest=(rank + 1) % world_size) kv_recv = recv_async(src=(rank - 1) % world_size) # Compute: flash attention on local Q vs current KV block O_block, l_block, m_block = flash_attn_block(Q_p, K_cur, V_cur) # Merge: online softmax accumulation m_new = max(m_p, m_block) scale_old = exp(m_p - m_new) scale_new = exp(m_block - m_new) l_p = scale_old * l_p + scale_new * l_block O_p = (scale_old.unsqueeze(-1) * O_p + scale_new.unsqueeze(-1) * O_block) m_p = m_new # Wait for communication to complete sync_comm() # Final normalization O_p = O_p / l_p.unsqueeze(-1) return O_p

5 · Compute-Communication Overlap

The key to Ring Attention's efficiency: communication happens simultaneously with computation. While GPU p computes attention with the current K/V chunk, it's already sending that chunk to GPU p+1 and receiving the next chunk from GPU p-1.

Overlap condition: T_compute ≥ T_communicate T_compute = (N/P)² × d × 2 / GPU_FLOPS (attention for one chunk pair) T_communicate = 2 × (N/P) × d × 2 / bandwidth (send K + V for one chunk) For N=1M, P=8, d=128, A100 (312 TFLOP/s, NVLink 600 GB/s): T_compute ≈ 2.6 ms T_comm ≈ 0.05 ms → Communication fully hidden! (52× more compute than comm)
When overlap breaks down: For short sequences or many GPUs (small N/P), compute per chunk shrinks and communication can't be fully hidden. Rule of thumb: N/P should be at least 1024 tokens for good overlap on NVLink systems.

7 · The Math

Memory per GPU: Standard attention: O(N × d) for Q, K, V + O(N²) for attention matrix Ring attention: O(N/P × d) for local Q + O(N/P × d) for current K,V block + O(N/P × d) for incoming K,V block (double buffering) Total: O(N/P × d) — linear in N/P per GPU Compute per GPU: Each GPU computes P chunk-attention operations: (N/P) × (N/P) each Total per GPU: P × (N/P)² × d = N²d / P → Perfect linear scaling: each GPU does 1/P of total work
Communication volume per GPU: Each step: send and receive 2 × (N/P) × d values (K + V) P-1 steps total: 2 × (P-1) × (N/P) × d ≈ 2Nd values Bandwidth requirement: 2Nd / T_total With compute-comm overlap: no additional time cost (hidden behind compute)

8 · Causal Masking in Ring Attention

For causal (autoregressive) attention, GPU p's Q chunk should only attend to K/V chunks with earlier or equal positions. This means some chunk pairs produce all-masked-out scores:

GPU p holds Q for positions [p·N/P, (p+1)·N/P) K/V chunk from GPU q covers positions [q·N/P, (q+1)·N/P) If q·N/P > (p+1)·N/P → entire chunk is masked (skip computation!) For P=4: GPU 0 only needs KV from GPU 0 (1 step instead of 4) GPU 3 needs KV from all GPUs (4 steps) Average: 2.5 steps (vs 4 for bidirectional) → 1.6× faster
Load imbalance: With causal masking, earlier GPUs do less work than later ones. Solutions: (1) overlap the skipped computation with communication from later chunks, (2) use "zigzag" token assignment that balances causal workload, (3) pair Ring Attention with tensor parallelism.

9 · Implementation

Key implementation details:

  • Double buffering: Allocate two K/V buffers per GPU. While computing with buffer A, receive the next chunk into buffer B. Swap after each step.
  • Flash Attention integration: Each local attention computation uses Flash Attention for the tiled, IO-efficient inner computation. Ring handles the cross-device distribution.
  • Online softmax: The merge step uses the same running max/log-sum-exp trick as Flash Attention, ensuring numerical stability when combining partial attention results from different K/V chunks.
  • Backward pass: Reverse the ring direction. Each GPU sends dO (output gradients) around the ring while computing local dQ, dK, dV contributions.

Near-linear scaling: Each GPU maintains constant memory and compute regardless of total sequence length — you just add more GPUs. The communication overhead is hidden behind compute. In practice, Ring Attention achieves >95% scaling efficiency on NVLink-connected clusters.

11 · Applications

  • Long-context LLMs: Ring Attention enables training and inference with 1M+ token contexts. Used in research for book-length understanding, codebase analysis, and long-document summarization.
  • Video understanding: Video models process thousands of frames; Ring Attention distributes frame-level attention across GPUs.
  • Scientific computing: Protein sequences (10K+ amino acids) and genomic sequences benefit from long-range attention across distributed GPUs.
  • Stripe (Liu et al., 2023): "Striped" Ring Attention with zigzag token assignment for better load balancing with causal masks.
  • LightSeq / DistFlashAttn: Production implementations of ring attention with additional optimizations for heterogeneous clusters.

12 · Summary

  • Ring Attention distributes sequences across P GPUs in a ring, circulating K/V blocks while computing local attention in parallel.
  • Communication is overlapped with computation using double buffering and async send/receive.
  • Memory per GPU: O(N/P) — context length scales linearly with GPU count.
  • Uses Flash Attention's online softmax for numerically stable cross-device result merging.
  • Causal masking introduces load imbalance that can be mitigated with zigzag assignment.
  • Enables million-token contexts and beyond on multi-GPU clusters.
One-line summary: Ring Attention circulates KV blocks around a GPU ring, overlapping communication with computation to scale context length linearly with device count.
