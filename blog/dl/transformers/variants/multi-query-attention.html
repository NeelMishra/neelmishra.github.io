Multi-Query Attention (MQA)

ML Series · Transformers · Variants

1 · Origin & Motivation

Multi-Query Attention was introduced by Noam Shazeer in 2019 — years before KV caches became the headline bottleneck. The core observation: during autoregressive decoding, the model generates one token at a time, and the dominant cost is loading KV cache from HBM into SRAM, not the matrix multiplications themselves.

With standard Multi-Head Attention (MHA), each of H heads has its own K and V projection. The KV cache grows as:

KV cache per layer = 2 × H × seq_len × d_head × bytes For H=32, seq=4096, d_head=128, FP16: = 2 × 32 × 4096 × 128 × 2 = 67 MB per layer = 67 MB × 32 layers = 2.1 GB total

The insight: if we share K and V across all heads, the cache shrinks by a factor of H. For a 32-head model, that's 32× less memory bandwidth per decoding step.

2 · The MQA Idea

MQA keeps multiple query heads but uses only one shared key head and one shared value head:

MHA: Q₀,K₀,V₀ Q₁,K₁,V₁ Q₂,K₂,V₂ ... Q_H,K_H,V_H (H sets of Q,K,V) MQA: Q₀,K,V Q₁,K,V Q₂,K,V ... Q_H,K,V (H queries, 1 K, 1 V)

Each query head still has its own learned projection W_Qᵢ, so each head computes a different attention pattern. But all heads attend over the same keys and values. The diversity comes from the queries, not the keys/values.

Analogy: Think of a library. MHA gives each researcher their own private book collection. MQA gives each researcher their own unique search criteria (queries), but they all search the same shared library (keys and values). Different researchers still find different things because they're looking for different things.

3 · The Math

Projections: Qᵢ = x · W_Qᵢ for i = 0, ..., H-1 (H separate Q projections) K = x · W_K (single shared K projection) V = x · W_V (single shared V projection) Where: W_Qᵢ ∈ ℝ^(d_model × d_head) — one per head W_K ∈ ℝ^(d_model × d_head) — shared W_V ∈ ℝ^(d_model × d_head) — shared
Attention for head i: Attn_i = softmax(Qᵢ · K^T / √d_head) · V Output: out = Concat(Attn₀, Attn₁, ..., Attn_{H-1}) · W_O
Parameter count comparison (K + V projections only): MHA: 2 × H × d_model × d_head (e.g., 2 × 32 × 4096 × 128 = 33.6M) MQA: 2 × 1 × d_model × d_head (e.g., 2 × 1 × 4096 × 128 = 1.05M) Ratio: 1/H (32× fewer KV parameters for H=32)

4 · Memory & Bandwidth Savings

Metric MHA (H=32) MQA (H=32) Reduction
KV params per layer 2 × 32 × d² / H 2 × 1 × d² / H 32×
KV cache (4K seq, FP16) 67 MB/layer 2.1 MB/layer 32×
Bandwidth per decode step Loads 32 K + 32 V Loads 1 K + 1 V 32×
Decode latency (bandwidth-bound) Baseline ~1.7× faster —
Max batch size (fixed GPU memory) Baseline ~8–16× larger —
Why not 32× faster latency? The 32× bandwidth reduction applies only to KV loading. The Q computation, output projection, and MLP layers are unchanged. In practice, MQA improves end-to-end decode throughput by ~1.5–2× depending on model size and hardware.

6 · Implementation

class MultiQueryAttention: def __init__(self, d_model, n_heads): self.n_heads = n_heads self.head_dim = d_model // n_heads # H separate query projections, 1 shared K, 1 shared V self.W_Q = Linear(d_model, n_heads * self.head_dim) # full rank self.W_K = Linear(d_model, self.head_dim) # single head self.W_V = Linear(d_model, self.head_dim) # single head self.W_O = Linear(d_model, d_model) def forward(self, x, cache_k=None, cache_v=None): B, S, _ = x.shape # Project queries (all heads) and keys/values (single head) Q = self.W_Q(x).view(B, S, self.n_heads, self.head_dim) K = self.W_K(x).view(B, S, 1, self.head_dim) V = self.W_V(x).view(B, S, 1, self.head_dim) # Update KV cache for autoregressive decoding if cache_k is not None: K = cat([cache_k, K], dim=1) V = cat([cache_v, V], dim=1) # Broadcast K, V across all query heads K = K.expand(B, -1, self.n_heads, self.head_dim) V = V.expand(B, -1, self.n_heads, self.head_dim) # Scaled dot-product attention scores = (Q @ K.transpose(-2, -1)) / sqrt(self.head_dim) weights = softmax(scores, dim=-1) out = weights @ V out = out.reshape(B, S, -1) return self.W_O(out), K, V
Broadcast vs repeat: We use expand (broadcast) instead of repeat to avoid copying data. The single K/V tensor is logically shared across heads without allocating H copies in memory.

7 · Quality Impact

MQA's single shared KV head means all query heads attend over the same representation. This can limit the model's ability to simultaneously capture different types of relationships:

  • Training from scratch: Shazeer (2019) reported < 0.3% quality loss on translation tasks with MQA. The model learns to compensate during training.
  • Converting MHA → MQA post-training: More significant degradation without uptraining. The model was trained expecting diverse K/V heads.
  • Tasks most affected: Multi-hop reasoning, coreference resolution, and tasks requiring fine-grained token comparison suffer most.
  • Tasks least affected: Generation, summarization, and single-hop QA are robust to MQA.
Benchmark MHA (baseline) MQA GQA-8
Translation BLEU 100% 99.7% 99.9%
Summarization ROUGE 100% 99.3% 99.8%
Multi-hop QA 100% 97.5% 99.5%
Decode throughput 1× 1.7× 1.5×

8 · MQA vs GQA

GQA (Grouped Query Attention) is the generalization of MQA. MQA uses 1 KV group; GQA uses G groups (where 1 < G < H). The key differences:

Aspect MQA GQA (e.g., G=8)
KV heads 1 G (e.g., 8)
Cache reduction H× (e.g., 32×) H/G× (e.g., 4×)
Quality Slight degradation ≈ MHA
Best for Extreme memory constraints General-purpose LLMs
Industry trend Declining (replaced by GQA) Now the default
Industry shift: MQA was the pioneer, but GQA has largely replaced it. Most modern LLMs (LLaMA-3, Mistral, Gemma) use GQA-8 because the memory savings are "good enough" (4–8×) without the quality compromise of a single KV head. MQA remains relevant for latency-critical applications or very large batch sizes.

9 · Converting MHA → MQA

There are three approaches to creating an MQA model:

Design the model with MQA from the start. The model learns to use the single shared KV effectively. This gives the best MQA quality.

Take a pretrained MHA model, average all KV heads into one: W_K_new = mean(W_K₀, ..., W_K_H). Then continue training for 5–10% of original tokens. This is the approach used by the GQA paper and works well in practice.

Instead of averaging, pick the single KV head that has the lowest loss when used alone. This preserves the most informative head's learned representations but is more expensive to evaluate.

10 · Models Using MQA

  • PaLM (Google, 2022) — 540B model, one of the first large-scale uses of MQA
  • PaLM-2 (Google, 2023) — continued use of MQA for efficiency
  • Falcon-40B (TII, 2023) — MQA for all 40B parameters
  • StarCoder (BigCode, 2023) — MQA for fast code completion
  • ChatGLM2-6B (Tsinghua) — MQA variant for Chinese-English
  • MPT-7B / MPT-30B (MosaicML) — early open-source MQA adopters

11 · Summary

  • MQA uses a single shared K/V head across all query heads, reducing KV cache by H× (e.g., 32× for a 32-head model).
  • The primary benefit is memory bandwidth: less data to load per decode step → faster inference and larger batch sizes.
  • Quality impact is small (<1% on most benchmarks) when trained from scratch, but noticeable on multi-hop reasoning tasks.
  • GQA has largely superseded MQA as the default choice, offering a better quality/efficiency tradeoff.
  • MQA remains valuable for extreme latency or memory constraints.
One-line summary: MQA is the extreme end of KV sharing — all query heads read from one set of keys and values, maximizing inference speed at a small quality cost.
