KV Caching

ML Series · Transformers · Variants

When you chat with a large language model and watch tokens appear one at a time, there is an engineering trick silently running behind the scenes that makes the whole experience practical: KV Caching. Without it, generating a single 512-token response from a model like LLaMA-70B would take orders of magnitude longer and waste enormous amounts of compute re-deriving results the model already knows.

KV caching is one of those ideas that is deceptively simple in principle—store the Key and Value projections so you don’t recompute them—yet touches nearly every part of the inference stack: memory management, batching, quantisation, hardware utilisation, and even model architecture itself. In this post we will build up the full picture from first principles, derive the math, implement it in code, animate it step by step, and then survey the landscape of optimisation techniques that have grown around it.

1 · The Redundancy Problem in Autoregressive Generation

Transformer language models are trained with teacher forcing: at training time the model receives the entire sequence at once, and the causal mask ensures each position can only attend to earlier positions. The key and value matrices are computed for every token in one efficient batched operation and never need to be revisited.

Inference, however, is fundamentally different. Autoregressive generation proceeds one token at a time. At step t the model takes all tokens produced so far — x₁, x₂, …, x_t — and predicts x_{t+1}. If we naively treat each step like a mini forward pass over the full sequence, the computation graph looks like this:

Step 1: Q₁, K₁, V₁ = Linear(x₁) ← 1 token Step 2: Q₂, K₁, K₂, V₁, V₂ = Linear(x₁, x₂) ← 2 tokens Step 3: Q₃, K₁, K₂, K₃, V₁, V₂, V₃ = Linear(x₁, x₂, x₃) ← 3 tokens ⋮ Step t: Q_t, K₁…K_t, V₁…V_t = Linear(x₁ … x_t) ← t tokens

Notice that at every step, we recompute K₁, V₁ even though token x₁ has not changed. Similarly K₂, V₂ are recomputed at every step after step 2, and so on. The only new information at step t is the projection of the newest token x_t.

How bad is the redundancy? Consider generating n tokens in total. At step t, the naive approach runs the projection linear layers on t tokens. The total projection work across all steps is:

Total projections = 1 + 2 + 3 + … + n = n(n+1)/2 = O(n²)

But each token only needs to be projected once — it never changes. So the truly necessary work is just n projections — a factor of n/2 less. For a 2048-token generation, we are doing roughly 1000× more projection work than necessary.

And that is only the projection cost. The attention computation itself (the dot product of queries with keys) also balloons: at step t the attention score matrix is 1 × t (for a single query against t keys), giving a total attention cost of ∑ t = O(n²). Without caching, we would recompute the full t × t attention at each step, yielding O(n³) total work. This is catastrophically inefficient.

Key insight: In causal (decoder-only) generation, the K and V projections for tokens 1…t-1 are identical at step t and step t-1. Recomputing them is pure waste.

2 · The Core Idea: Cache What Doesn’t Change

The fix is beautifully simple: keep the K and V tensors from all previous steps in memory (the “KV cache”) and only compute the projection for the newest token.

  1. Project only the new token: q_t, k_t, v_t = W_Q x_t, W_K x_t, W_V x_t
  2. Append to cache: K_cache = concat(K_cache, k_t)  |  V_cache = concat(V_cache, v_t)
  3. Attend: attn = softmax(q_t · K_cache¹ / √d) · V_cache
  4. Produce output: pass the attention output through the rest of the transformer layer (output projection, feed-forward, etc.)

That’s it. At every decode step we compute projections for one token instead of t tokens, and we compute attention scores of shape 1 × t instead of t × t.

The cache grows by one row per step per layer per head. The memory required is proportional to the generated sequence length, which we will quantify precisely in Section 6. But the compute savings are dramatic: we trade memory for speed.

Why don’t we cache the query? Because at each decode step we only need q_t—the query for the new token. The query is used once, to dot-product against the full key cache, and is never needed again. Keys and values, by contrast, are needed at every future step, which is precisely why caching them pays off.

Terminology: The KV cache is sometimes called the past key values (as in Hugging Face’s past_key_values argument), inference cache, or simply the cache. These all refer to the same concept: stored K and V tensors from previous decoding steps.

3 · Prefill vs Decode Phases

Real inference has two distinct phases, and understanding them is essential for reasoning about KV cache behaviour, latency, and throughput.

Prefill Phase (Prompt Processing)

When a user submits a prompt of length P, the model processes the entire prompt at once in a single forward pass, exactly like training. This is the prefill (or “prompt encoding”) phase.

  • All P tokens are projected in parallel to produce Q, K, V of shape (P, d).
  • Full causal attention is computed: Attn = softmax(QK¹/√d) · V with a causal mask of shape (P, P).
  • The resulting K and V tensors (for every layer) are stored as the initial KV cache.
  • The logits at position P give the first generated token.

Prefill is compute-bound: we are doing a lot of matrix multiplications in a single, highly parallelizable pass. GPU utilization is high.

Decode Phase (Token Generation)

After prefill, generation enters the decode (or “autoregressive”) phase, producing one token per step:

  • Only the newest token is projected: q_t, k_t, v_t.
  • k_t and v_t are appended to the cache.
  • Attention is computed as softmax(q_t · K_cache¹/√d) · V_cache.
  • The logits give the next token, and the loop repeats.

Decode is memory-bandwidth-bound: each step does very little arithmetic (one query against the full cache) but must read the entire KV cache from GPU memory. The ratio of compute to memory access (arithmetic intensity) is very low, which is why decode speed is dominated by how fast you can stream data from HBM.

Input: full prompt [x₁, x₂, …, x_P]
Computation: full P×P causal attention
Output: initial KV cache of shape (P, d) per layer + first generated token
Bottleneck: Compute (FLOPs)

Input: single new token x_t
Computation: 1×t attention using KV cache
Output: updated KV cache of shape (t, d) per layer + next token
Bottleneck: Memory bandwidth (reading the cache)

This two-phase structure has major implications for system design. Prefill and decode have different hardware utilisation profiles, which is why systems like Splitwise and DistServe separate them onto different hardware.

4 · The Math

Let us formalize KV caching mathematically. We define the projections, the cache update rule, and the cache-augmented attention computation.

Projection

At decode step t, the model receives a single token embedding x_t ∈ ℝ^d_model. The projection matrices produce:

q_t = x_t W_Q ∈ ℝ^(1 × d_k) (query for the new token) k_t = x_t W_K ∈ ℝ^(1 × d_k) (key for the new token) v_t = x_t W_V ∈ ℝ^(1 × d_v) (value for the new token)

where W_Q, W_K ∈ ℝ^(d_model × d_k) and W_V ∈ ℝ^(d_model × d_v). For standard multi-head attention, d_k = d_v = d_model / n_heads.

Cache Update

Let K_{cache}^{(t-1)} ∈ ℝ^((t-1) × d_k) and V_{cache}^{(t-1)} ∈ ℝ^((t-1) × d_v) be the cached keys and values from steps 1 through t-1. After projecting token t, we update:

K_cache^(t) = concat( K_cache^(t-1), k_t ) ∈ ℝ^(t × d_k) V_cache^(t) = concat( V_cache^(t-1), v_t ) ∈ ℝ^(t × d_v)

In practice, implementations pre-allocate a buffer of size (max_seq_len, d_k) and write k_t into position t to avoid repeated memory allocation and copying.

Attention with Cache

The attention output for the new token is:

scores_t = q_t · (K_cache^(t))¹ / √d_k ∈ ℝ^(1 × t) α_t = softmax(scores_t) ∈ ℝ^(1 × t) output_t = α_t · V_cache^(t) ∈ ℝ^(1 × d_v)

Compare this with naive full recomputation, where we would compute Q ∈ ℝ^(t × d_k), K ∈ ℝ^(t × d_k), V ∈ ℝ^(t × d_v), and then compute the full t × t attention matrix — only to discard all rows except the last one. KV caching eliminates this waste entirely.

Multi-Head Extension

For multi-head attention with H heads, we maintain separate caches per head:

For head h = 1 … H: K_cache_h^(t) = concat( K_cache_h^(t-1), k_t_h ) V_cache_h^(t) = concat( V_cache_h^(t-1), v_t_h ) output_t_h = softmax( q_t_h · (K_cache_h^(t))¹ / √d_k ) · V_cache_h^(t) final_output_t = concat(output_t_1, …, output_t_H) · W_O

Each head has its own slice of the cache. In practice, the cache tensor has shape (n_heads, seq_len, d_k), and all heads are computed in parallel.

6 · Memory Analysis

The KV cache can become the dominant memory consumer during inference, especially for long sequences or large models. Let’s derive the formula and compute concrete numbers.

Formula

For a single sequence, the total KV cache memory is:

KV_memory = 2 × L × H × S × d_k × B_dtype where: 2 = one tensor for K, one for V L = number of transformer layers H = number of attention heads S = current sequence length (grows during generation) d_k = dimension per head = d_model / H B_dtype = bytes per element (2 for FP16/BF16, 4 for FP32)

This simplifies to:

KV_memory = 2 × L × d_model × S × B_dtype (since H × d_k = d_model)

Note that the number of heads cancels out—it’s the total model dimension that matters (at least for standard MHA; MQA and GQA change this, as we’ll see later).

Concrete Examples

Model Layers (L) d_model Heads (H) d_k Max Seq Len KV Cache (FP16, 1 seq) KV Cache (FP16, batch=32)
GPT-2 Small 12 768 12 64 1024 ~37.7 MB ~1.2 GB
GPT-2 XL 48 1600 25 64 1024 ~314 MB ~9.8 GB
LLaMA-7B 32 4096 32 128 4096 ~2.1 GB ~67.1 GB
LLaMA-13B 40 5120 40 128 4096 ~3.3 GB ~104 GB
LLaMA-70B 80 8192 64 128 4096 ~10.5 GB ~335 GB
LLaMA-70B (GQA, 8 KV heads) 80 8192 64 (8 KV) 128 4096 ~1.3 GB ~41.9 GB
Memory dominates at scale: For LLaMA-70B with a batch of 32 sequences, the KV cache alone requires ~335 GB of GPU memory—more than the model weights themselves (~140 GB in FP16). This is why KV cache management (paged attention, quantisation) is so critical in production systems.

Worked Example: LLaMA-7B

Let’s trace through the calculation for LLaMA-7B at max sequence length:

KV_memory = 2 × 32 × 4096 × 4096 × 2 bytes = 2 × 32 × 4096 × 4096 × 2 = 2,147,483,648 bytes = 2.0 GB (per sequence)

With a batch of 8 concurrent requests, that is 16 GB just for the cache—consuming most of an A100-40GB’s memory after the model weights are loaded (~14 GB in FP16). This is the fundamental tension that drives all the optimisation techniques in Section 10.

7 · Memory-Compute Tradeoff

KV caching is a textbook space-time tradeoff: we spend memory to save computation. Let’s quantify the savings precisely.

Per-Token Compute Cost

Operation Without KV Cache With KV Cache
Projection (Q, K, V) at step t O(t · d²) — project all t tokens O(d²) — project 1 token
Attention score computation at step t O(t² · d) — full t×t attention O(t · d) — 1×t attention
Total per step O(t · d² + t² · d) O(d² + t · d)

Total Generation Cost (n tokens)

Summing the per-step cost over n generation steps:

Without cache: ∑_{t=1}^{n} O(t · d² + t² · d) = O(n² · d² + n³ · d) With cache: ∑_{t=1}^{n} O(d² + t · d) = O(n · d² + n² · d)

The dominant terms shift from O(n³) (cubic!) without cache to O(n²) (quadratic) with cache. For a 2048-token generation, this is roughly a 1000× reduction in compute for the attention component alone.

What We Pay

  • Memory: O(n · L · d) per sequence in the batch. This grows linearly with sequence length.
  • Memory bandwidth: Each decode step must read the entire cache from GPU HBM, which becomes the bottleneck. For LLaMA-70B at sequence length 4096, that is ~10.5 GB read per step per sequence.
  • Cache management complexity: Memory must be allocated, tracked, and freed efficiently, especially when serving many concurrent requests.

Despite these costs, the tradeoff is overwhelmingly favorable. No production inference system operates without KV caching.

8 · Implementation

Below is a PyTorch-style pseudocode implementation showing how KV caching integrates into a transformer decoder layer. The key change is a cache dictionary that persists across generation steps.

# ─── KV-Cached Multi-Head Attention (PyTorch pseudocode) ─── class CachedMultiHeadAttention(nn.Module): def __init__(self, d_model, n_heads): self.d_model = d_model self.n_heads = n_heads self.d_k = d_model // n_heads self.W_Q = nn.Linear(d_model, d_model) self.W_K = nn.Linear(d_model, d_model) self.W_V = nn.Linear(d_model, d_model) self.W_O = nn.Linear(d_model, d_model) def forward(self, x, cache=None, use_cache=True): # x: (batch, seq_len, d_model) # During decode: seq_len = 1 # During prefill: seq_len = prompt_length B, S, _ = x.shape # Step 1: Project the new token(s) Q = self.W_Q(x).view(B, S, self.n_heads, self.d_k).transpose(1,2) K = self.W_K(x).view(B, S, self.n_heads, self.d_k).transpose(1,2) V = self.W_V(x).view(B, S, self.n_heads, self.d_k).transpose(1,2) # Q, K, V: (batch, n_heads, seq_len, d_k) # Step 2: Update the cache if cache is not None: K_prev, V_prev = cache # (batch, n_heads, prev_len, d_k) K = torch.cat([K_prev, K], dim=2) # → (batch, n_heads, prev_len + S, d_k) V = torch.cat([V_prev, V], dim=2) # → (batch, n_heads, prev_len + S, d_k) # Step 3: Store updated cache for next step new_cache = (K, V) if use_cache else None # Step 4: Compute attention # Q: (batch, n_heads, S, d_k) — S=1 during decode # K: (batch, n_heads, T, d_k) — T = total seq len so far scores = torch.matmul(Q, K.transpose(-2, -1)) / (self.d_k ** 0.5) # Apply causal mask (only needed during prefill) if S > 1: mask = torch.triu(torch.ones(S, K.size(2)), diagonal=K.size(2)-S+1) scores = scores.masked_fill(mask == 1, float('-inf')) attn = torch.softmax(scores, dim=-1) # (batch, n_heads, S, T) out = torch.matmul(attn, V) # (batch, n_heads, S, d_k) # Step 5: Concatenate heads and project out = out.transpose(1,2).contiguous().view(B, S, self.d_model) out = self.W_O(out) return out, new_cache # ─── Generation Loop ─── def generate(model, prompt_ids, max_new_tokens): # Phase 1: Prefill — process entire prompt x = model.embed(prompt_ids) # (1, P, d_model) caches = [None] * model.n_layers for i, layer in enumerate(model.layers): x, caches[i] = layer(x, cache=None) # prefill: no prior cache next_token = sample(model.lm_head(x[:, -1, :])) # Phase 2: Decode — one token at a time generated = [next_token] for _ in range(max_new_tokens - 1): x = model.embed(next_token.unsqueeze(1)) # (1, 1, d_model) for i, layer in enumerate(model.layers): x, caches[i] = layer(x, cache=caches[i]) # use & update cache next_token = sample(model.lm_head(x[:, -1, :])) generated.append(next_token) return generated

Key details:

  • During prefill, cache=None is passed: K and V are built from scratch for the full prompt and returned as the initial cache.
  • During decode, each layer receives its cache from the previous step, concatenates the new K/V, and returns the updated cache.
  • The torch.cat on every step is actually wasteful in production. Real implementations pre-allocate a buffer and write into a slot, avoiding repeated allocation. vLLM’s paged attention takes this further with virtual memory-style block management.

9 · Multi-Layer Cache

A transformer with L layers requires a separate KV cache for each layer. The keys and values at layer 3 depend on the output of layers 1 and 2, so they are distinct from those at layer 1. The full cache data structure is:

cache = { layer_0: (K_cache_0, V_cache_0), # shapes: (batch, n_heads, seq_len, d_k) layer_1: (K_cache_1, V_cache_1), ⋮ layer_{L-1}: (K_cache_{L-1}, V_cache_{L-1}) }

In frameworks like Hugging Face Transformers, this is represented as a tuple of tuples:

past_key_values = ( (K_0, V_0), # layer 0 (K_1, V_1), # layer 1 ... (K_{L-1}, V_{L-1}) # layer L-1 )

Memory Implication

The total cache memory scales linearly with the number of layers. This is why the formula in Section 6 includes L as a multiplier. For a 80-layer model like LLaMA-70B, the cache is 80× larger than for a single layer.

Execution Order

During a decode step, we process layers sequentially:

  1. Layer 0: Project the new token embedding → compute attention using cache[0] → update cache[0] → pass output through FFN → output h_0
  2. Layer 1: Project h_0 → compute attention using cache[1] → update cache[1] → FFN → output h_1
  3. … and so on through layer L-1

Each layer reads its own cache slice and writes back to it. Layers cannot be parallelized (each depends on the previous layer’s output), but the cache lookups at each layer are independent of other layers’ caches.

Implementation tip: When profiling memory, remember that the KV cache is the largest dynamic allocation during inference. Model weights are static. Activations (for a single token) are tiny. The cache is what grows and what you need to manage carefully — especially when serving many concurrent users with different sequence lengths.

10 · KV Cache Optimization Techniques

Because the KV cache is the primary memory bottleneck in LLM serving, a rich ecosystem of optimization techniques has emerged. Here are the most important ones.

10.1 Quantized KV Cache (INT8 / FP8)

Instead of storing cache entries in FP16 (2 bytes per element), we can quantize them to INT8 (1 byte) or FP8 (1 byte), halving cache memory.

  • Per-token quantization: Each k_t and v_t vector is quantized independently with its own scale factor. This preserves accuracy well because each token’s magnitude can be captured by its scale.
  • Per-channel quantization: Quantize along the head dimension. Can be more efficient on hardware with native INT8 support.
  • Impact: Reduces cache memory by ~2× with minimal quality degradation (typically <0.5% perplexity increase). NVIDIA’s TensorRT-LLM and vLLM both support INT8 KV caches.

10.2 KV Cache Compression

Instead of keeping every token’s K/V, we can evict or compress entries that are least important:

  • H₂O (Heavy Hitter Oracle): Observes that a small fraction of tokens (“heavy hitters”) accumulate most of the attention mass. Keeps the heavy hitters plus the most recent tokens, evicting the rest.
  • Scissorhands: Similar idea — identifies “pivotal tokens” that are consistently important across layers and keeps only those.
  • Token merging: Instead of eviction, merge similar tokens by averaging their K/V representations.
  • Impact: Can reduce cache to 20–50% of original size while maintaining 95%+ of quality. Works best for long contexts where many tokens in the middle contribute little to the output.

10.3 Sliding Window Cache (Local Attention)

Models like Mistral use a sliding window attention pattern where each token only attends to the most recent W tokens (e.g., W = 4096). This caps the KV cache at a fixed size regardless of total sequence length:

Cache size (sliding window) = 2 × L × d_model × W × B_dtype For Mistral-7B (W=4096): ~2 GB regardless of total context

The model can still reason over longer contexts through information flowing across layers (a token at position P in layer l can access position P-W directly, but position P-2W indirectly through representations computed at layer l-1).

10.4 Shared Prefix Caching (Prompt Caching)

In many applications, multiple requests share the same system prompt or prefix (e.g., a chatbot’s persona instructions). Instead of computing and storing duplicate KV caches for the shared prefix, we compute it once and share the cache across all requests:

  • Prefix caching in vLLM: Automatically detects shared prefixes across requests in a batch and reuses the KV cache for the common portion.
  • RadixAttention (SGLang): Uses a radix tree to efficiently match and reuse cached prefixes across requests, enabling sub-tree sharing.
  • Impact: Eliminates redundant prefill computation for shared prefixes. For a 2000-token system prompt shared by 100 requests, this saves 100× the prefill compute and memory for those tokens.
These techniques compose: You can combine quantized KV cache (INT8) with paged attention for memory management, shared prefix caching for common prompts, and GQA for reduced KV heads. Modern serving stacks like vLLM apply several of these simultaneously.

11 · Connection to Other Techniques

KV caching doesn’t exist in isolation. Several major architectural and systems innovations are intimately connected to how we manage and reduce the KV cache.

Multi-Query Attention (MQA)

Standard multi-head attention (MHA) has separate K and V projections for each head. Multi-Query Attention (Shazeer 2019) uses a single K/V head shared across all query heads:

MHA cache per layer: 2 × H × S × d_k (H K/V heads) MQA cache per layer: 2 × 1 × S × d_k (1 K/V head) Reduction factor: H× (e.g., 32× for LLaMA-7B)

MQA dramatically shrinks the cache but can hurt quality for very large models. It is used in PaLM, Falcon, and StarCoder.

Grouped-Query Attention (GQA)

GQA (Ainslie et al. 2023) is the practical middle ground between MHA and MQA. Instead of H KV heads (MHA) or 1 KV head (MQA), GQA uses G KV head groups, where 1 < G < H:

GQA cache per layer: 2 × G × S × d_k (G KV head groups) LLaMA-2 70B: H=64 query heads, G=8 KV groups → 8× cache reduction

GQA preserves nearly all of MHA’s quality while providing substantial cache savings. It is the default in LLaMA-2 70B, LLaMA-3, Gemma, and most modern LLMs.

Paged Attention (vLLM)

Traditional KV cache implementations pre-allocate a contiguous buffer of max_seq_len for every request, leading to massive internal fragmentation (most requests don’t use the full context window).

Paged Attention (Kwon et al. 2023, vLLM) applies virtual memory concepts to the KV cache:

  • The cache is divided into fixed-size blocks (e.g., 16 tokens per block).
  • Blocks are allocated on demand as the sequence grows, like pages in virtual memory.
  • A block table maps logical positions to physical memory locations.
  • Blocks can be non-contiguous in physical memory, eliminating fragmentation.

This increases effective GPU memory utilization from ~50% to ~95%, allowing significantly higher batch sizes and throughput. Paged attention has become the standard approach in production LLM serving.

Flash Attention

While Flash Attention (Dao et al. 2022) is primarily a training optimization (reducing memory via tiling and avoiding materializing the full attention matrix), it also applies to the decode phase with KV caching. Flash Decoding optimizes the case where a single query attends to a long KV cache by parallelizing across the key/value sequence dimension, improving GPU utilization during the memory-bandwidth-bound decode phase.

Speculative Decoding

Speculative decoding uses a small “draft” model to propose multiple tokens at once, which the large model then verifies in parallel. The KV cache plays a crucial role: if the large model rejects a token, the cache must be “rolled back” to the last accepted position. Efficient cache management (especially with paged attention) makes this rollback cheap.

12 · Summary

KV caching is one of the most important practical techniques in transformer inference. Let’s recap the key points:

During autoregressive generation, the Key and Value projections for previous tokens never change. Store them in a cache and only compute K/V for the new token at each step.

Prefill: Process the full prompt in parallel (compute-bound), creating the initial cache.
Decode: Generate one token at a time using the cache (memory-bandwidth-bound).

Total generation compute drops from O(n³) to O(n²). Per-step projection work drops from O(t) to O(1). The cost is O(n · L · d) memory for the cache.

For large models, the KV cache can exceed the model weights in memory. LLaMA-70B at batch size 32 requires ~335 GB of cache in FP16. This drives optimization techniques: quantization (INT8), compression, sliding windows, prefix sharing, and paged attention.

MQA and GQA reduce the number of KV heads, directly shrinking the cache by up to H×. Paged attention eliminates memory fragmentation. These techniques compose and are used together in modern serving stacks.

KV caching transforms autoregressive generation from a theoretical curiosity into a practical reality. Without it, the language models we interact with daily would be orders of magnitude slower. Understanding it deeply — the math, the memory implications, and the optimization landscape — is essential for anyone working on LLM inference at scale.
