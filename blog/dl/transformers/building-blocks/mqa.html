    Multi-Query Attention: Extreme KV Cache Reduction

    January 2025 · 10 min read

    In standard Multi-Head Attention, each of the $h$ heads has its own Key and Value projections. During autoregressive generation (inference), we cache $K$ and $V$ for every position processed so far. This cache grows as $O(h \cdot n \cdot d_k)$ per layer, becoming a memory bottleneck on long sequences or large batch sizes. Multi-Query Attention (Shazeer, 2019) eliminates this bottleneck by using a single shared Key and Value head across all $H$ query heads. All queries attend to the same $K$ and $V$. The KV cache shrinks by a factor of $H$, dramatically reducing memory pressure during inference. The trade-off: modest quality degradation (typically 1–2% fewer tokens/sec improvement for longer context, or small perplexity increase), but this is often acceptable for fast inference. Deployed in PaLM, UPaLM, and other production models.

    The Cache Problem in Inference

    During autoregressive decoding, at step $t$, we have computed attention for all previous positions $0, \ldots, t-1$. To avoid recomputing, we cache $K^{(\text{cache})} \in \mathbb{R}^{t \times d_k}$ and $V^{(\text{cache})} \in \mathbb{R}^{t \times d_k}$ from the context. For each step:

    1. Compute $Q^{(t)} \in \mathbb{R}^{1 \times d_k}$ (just the new token's query).
    2. Attend to all $K^{(\text{cache})}, V^{(\text{cache})}$ to get the new output token.
    3. Append new $K^{(t)}, V^{(t)}$ (shape $1 \times d_k$) to the caches.

    Memory cost: with $H$ heads, we store $H \cdot t \cdot d_k + H \cdot t \cdot d_k = 2Ht d_k$ elements per layer. For a 100-layer model, $H = 96$ heads, $d_k = 128$, and $t = 2048$ (2K context), that's roughly $2 \cdot 96 \cdot 2048 \cdot 128 \cdot 100 \approx 6$ GB just for KV caches — a significant fraction of GPU memory on an A100 (80 GB). As context lengths increase to 4K, 8K, or longer, this becomes prohibitive.

    The Multi-Query Idea

    In Multi-Query Attention, we reduce to a single Key head and single Value head shared by all Query heads: $$\mathrm{MQA}(\mathbf{Q}, \mathbf{K}, \mathbf{V}) = \mathrm{Concat}_{i=1}^{H} \left( \mathrm{Softmax}\left(\frac{\mathbf{Q}_i \mathbf{K}^T}{\sqrt{d_k}}\right) \mathbf{V} \right) \mathbf{W}^O$$ where $\mathbf{Q}_i \in \mathbb{R}^{n \times d_k}$ is the $i$-th query head, but $\mathbf{K} \in \mathbb{R}^{n \times d_k}$ and $\mathbf{V} \in \mathbb{R}^{n \times d_k}$ are shared across all $H$ heads.

    KV cache cost: now $2t d_k$ (independent of $H$). For the same example above, this is just $2 \cdot 2048 \cdot 128 \cdot 100 \approx 52$ MB — a 100× reduction. This makes long-context inference practical on commodity hardware.

    Trade-off: Quality vs. Speed

    Empirically (Shazeer, 2019; Dao et al., 2024), MQA introduces a modest quality loss:

    • In short-context training (up to ~2K tokens), perplexity increase is ~1–2%.
    • In long-context fine-tuning, MQA adapts and quality gap narrows or disappears.
    • Inference speed gains are larger than the quality loss in many regimes.

    The loss likely stems from reduced representational capacity: with a single $\mathbf{K}$ and $\mathbf{V}$, all heads must attend to the same "view" of the key/value space. In contrast, standard MHA allows each query head to emphasize different key/value semantics. However, for production models where latency is critical, the trade-off often favors MQA.

    MQA vs. MHA: Parameter and Memory Comparison

    Aspect MHA (H heads) MQA (H query, 1 KV)
    KV cache per layer (inference, len=$t$) $2Ht d_k$ $2t d_k$ (independent of $H$)
    Parameters (projections only) $3H d_k \cdot d_{\text{model}} + d_{\text{model}}^2$ for $W^Q, W^K, W^V, W^O$ $H d_k \cdot d_{\text{model}} + (d_k + d_k) d_{\text{model}} + d_{\text{model}}^2$ (fewer $W^K, W^V$)
    Training cost (forward+backward) Baseline: $O(n^2 d_{\text{model}})$ ~same (compute is not the bottleneck; cache is)

    Deployment and Adoption

    MQA was first deployed at scale in PaLM (Chowdhery et al., 2022), Google's 540B parameter model. Subsequent models (UPaLM, medpalm, etc.) adopted it. More recently, with the rise of even longer contexts (4K–100K+ tokens), MQA remains a common choice. The inference speedups are significant enough that many labs accept the modest quality trade-off, especially for chat/instruction-following models where end-to-end latency matters more than absolute perplexity.

    Implementation Sketch

    def multi_query_attention(Q, K, V, num_heads):
    """
    Q: (batch, seq_len, d_model) -> reshaped to (batch, num_heads, seq_len, d_k)
    K, V: (batch, seq_len, d_model) -> kept as single heads
    num_heads: number of query heads
    """
    batch, seq_len, d_model = Q.shape
    d_k = d_model // num_heads
    
    # Project Q to num_heads
    Q_proj = linear_Q(Q)  # (batch, seq_len, d_model)
    Q = Q_proj.view(batch, seq_len, num_heads, d_k).transpose(1, 2)  # (batch, num_heads, seq_len, d_k)
    
    # Project K, V but keep as single heads (do NOT split into num_heads)
    K_proj = linear_K(K)  # (batch, seq_len, d_k) -- single head
    V_proj = linear_V(V)  # (batch, seq_len, d_k) -- single head
    
    # Attention: all query heads attend to the same K, V
    scores = Q @ K_proj.unsqueeze(1).transpose(-2, -1)  # (batch, num_heads, seq_len, seq_len)
    scores = scores / math.sqrt(d_k)
    attn_weights = softmax(scores, dim=-1)
    output_heads = attn_weights @ V_proj.unsqueeze(1)  # (batch, num_heads, seq_len, d_k)
    
    # Concatenate heads
    output = output_heads.transpose(1, 2).contiguous().view(batch, seq_len, d_model)
    
    # Output projection
    output = linear_O(output)
    return output

    Why GQA Is the Modern Standard

    MQA provides massive cache savings but at a quality cost. A natural middle ground — Grouped-Query Attention (part 11) — uses $G$ intermediate groups of heads sharing KV. With $G = H$ (all heads separate), you recover full MHA. With $G = 1$, you get MQA. The next post explores this family.
