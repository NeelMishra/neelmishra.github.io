    Grouped-Query Attention: Efficient Middle Ground

    January 2025 · 11 min read

    Multi-Query Attention (part 10) saves cache by collapsing all Key/Value heads to one. But this hurts quality. Grouped-Query Attention (Ainslie et al., 2023) finds a principled middle path: divide the $H$ query heads into $G$ groups, and each group shares one Key/Value head. With $G = H$ (every query head has its own K/V), you recover full MHA quality. With $G = 1$, you get MQA efficiency. Any $G$ in between trades quality for inference speed. Llama 2 and Llama 3 use $G = 8$ with $H = 32$ query heads, hitting a sweet spot. This design is now the de-facto standard in modern LLMs.

    Motivation: Interpolating Quality and Speed

    The trade-off space:

    • MHA ($G = H$): Full representational capacity, but KV cache is $H \times$ larger. Training quality is best, but inference is slow and memory-hungry.
    • MQA ($G = 1$): Minimal KV cache (1/H size), fast inference, but all heads share one K/V view. Quality degrades 1–2%.
    • GQA ($1 < G < H$): Compromise. Each of the $G$ groups gets its own K/V, so heads within a group see the same key/value space, but different groups can specialize. Cache is reduced by $H/G$ while maintaining most of the quality.

    Empirically, GQA with $G = 8$ (for $H = 32$) or $G = 16$ (for $H = 96$) recovers ~95%+ of MHA quality while reducing cache by 4–6× and matching MQA inference speed. This is why Llama 2/3, CodeLlama, and newer models standardize on GQA.

    Formulation

    Given $H$ query heads and $G$ key/value heads, assume $G \mid H$ (evenly divisible). Each KV head $j \in \{1, \ldots, G\}$ serves $H/G$ query heads. Define:

    $$\mathrm{GQA} = \mathrm{Concat}_{j=1}^{G} \left( \mathrm{Concat}_{i=1}^{H/G} \mathrm{Attention}(\mathbf{Q}_{j,i}, \mathbf{K}_j, \mathbf{V}_j) \right) \mathbf{W}^O$$

    Equivalently, for each group $g \in \{1, \ldots, G\}$, we have $H/G$ query heads $\mathbf{Q}_{g,1}, \ldots, \mathbf{Q}_{g,H/G}$, and they all attend to the shared pair $(\mathbf{K}_g, \mathbf{V}_g)$.

    KV cache size: $G \cdot t \cdot d_k$ (compare to $H \cdot t \cdot d_k$ for MHA and $t \cdot d_k$ for MQA). Reduction factor: $H / G$.

    Complexity Comparison

    Variant KV Cache (per layer) Quality vs. MHA Inference Speed
    MHA ($G = H$) $2Ht d_k$ 100% (baseline) 1.0× (baseline)
    GQA ($G = H/4$) $2(H/4)t d_k$ ~98–99% ~1.8–2.0×
    GQA ($G = H/8$) $2(H/8)t d_k$ ~96–97% ~2.5–3.0×
    MQA ($G = 1$) $2t d_k$ ~95–98% ~3–4×

    Real-World Adoption

    Llama 2 / Llama 3: Meta's Llama 2 (Touvron et al., 2023) and Llama 3 use $G = 8$ with $H = 32$ in base models. This gives 4× KV cache reduction with negligible quality loss. Inference is substantially faster on long-context tasks (e.g., processing 4K context).

    Other models: CodeLlama (code-focused), Mistral, and many others now default to GQA. It has become almost as standard as MHA in new models.

    Training vs. Inference Dynamics

    GQA can be trained from scratch (as Llama 2 was), or MHA models can be converted to GQA via "attention up-training" (Ainslie et al., 2023): initialize GQA heads from MHA by, e.g., averaging $H/G$ query heads and using one of their K/V heads. Then train a bit longer to adapt. Both methods work; training from scratch is simpler.

    Implementation Sketch

    def grouped_query_attention(Q, K, V, num_heads, num_kv_heads):
    """
    Q: (batch, seq_len, d_model) projected and reshaped to (batch, num_heads, seq_len, d_k)
    K, V: (batch, seq_len, d_model) projected and reshaped to (batch, num_kv_heads, seq_len, d_k)
    num_heads: number of query heads (e.g., 32)
    num_kv_heads: number of groups G (e.g., 8); must divide num_heads
    """
    batch, seq_len, d_model = Q.shape
    d_k = d_model // num_heads
    
    # Project to query, key, value
    Q_proj = linear_Q(Q)  # (batch, seq_len, d_model)
    K_proj = linear_K(K)  # (batch, seq_len, d_k * num_kv_heads)
    V_proj = linear_V(V)  # (batch, seq_len, d_k * num_kv_heads)
    
    # Reshape Q to (batch, num_heads, seq_len, d_k)
    Q = Q_proj.view(batch, seq_len, num_heads, d_k).transpose(1, 2)
    
    # Reshape K, V to (batch, num_kv_heads, seq_len, d_k)
    K = K_proj.view(batch, seq_len, num_kv_heads, d_k).transpose(1, 2)
    V = V_proj.view(batch, seq_len, num_kv_heads, d_k).transpose(1, 2)
    
    # Repeat K and V to match Q heads (each KV head replicated for its group)
    repeat_interleave = num_heads // num_kv_heads
    K = K.repeat_interleave(repeat_interleave, dim=1)  # (batch, num_heads, seq_len, d_k)
    V = V.repeat_interleave(repeat_interleave, dim=1)  # (batch, num_heads, seq_len, d_k)
    
    # Standard attention
    scores = Q @ K.transpose(-2, -1) / math.sqrt(d_k)
    attn_weights = softmax(scores, dim=-1)
    output_heads = attn_weights @ V  # (batch, num_heads, seq_len, d_k)
    
    # Concatenate and project
    output = output_heads.transpose(1, 2).contiguous().view(batch, seq_len, d_model)
    output = linear_O(output)
    return output

    Attention Up-Training: Upgrading MHA Models

    If you have a trained MHA model and want to convert it to GQA for inference efficiency, you can "up-train" it (Ainslie et al., 2023). Initialize each GQA group's K/V heads from one (or an average of) the MHA heads, then continue training. The model adapts within a few 100M tokens. This allows post-hoc efficiency improvements without full retraining.
