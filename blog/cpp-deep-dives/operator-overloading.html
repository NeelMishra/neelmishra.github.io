    Operator Overloading

    March 2026 · 16 min read

    What Is Operator Overloading?

    C++ lets you redefine how operators (+, -, ==, <<, [], (), etc.) behave for your custom types. This is called operator overloading.

    When you write a + b where a and b are your custom Vector2D objects, the compiler translates it to a function call: a.operator+(b) or operator+(a, b). You define that function, and the syntax "just works."

    Why it matters: Operator overloading makes your types feel like built-in types. A Matrix class with + and * reads like math. A BigInteger with == and < works seamlessly with std::sort. A smart pointer with * and -> behaves like a real pointer.

    Golden rule: Only overload operators when they have an intuitive, obvious meaning for your type. + on a Vector makes sense. + on a Database does not.

    Member vs Free Function

    You can define operator overloads in two ways:

    As a member function — the left operand is always *this:

    class Vec2 {
    double x, y;
public:
    Vec2(double x, double y) : x(x), y(y) {}

    // Member: this + other
    Vec2 operator+(const Vec2& other) const {
        return Vec2(x + other.x, y + other.y);
    }
};

    As a free (friend) function — both operands are parameters. Needed when the left operand isn't your class (e.g., std::ostream << yourObject):

    class Vec2 {
    double x, y;
public:
    Vec2(double x, double y) : x(x), y(y) {}

    friend Vec2 operator+(const Vec2& a, const Vec2& b) {
        return Vec2(a.x + b.x, a.y + b.y);
    }
};

    Rule of thumb:

    • =, [], (), -> — must be members.
    • <<, >> — must be free functions (left operand is a stream).
    • Arithmetic, comparison — prefer free functions for symmetry (int + Vec and Vec + int both work).

    Arithmetic Operators

    The four arithmetic operators (+, -, *, /) and their compound forms (+=, -=, etc.) are the most commonly overloaded. The canonical pattern: implement += first, then define + in terms of it. This avoids code duplication and ensures consistency.

    class Vec2 {
    double x, y;
public:
    Vec2(double x = 0, double y = 0) : x(x), y(y) {}

    // Compound assignment (modifies in-place, returns reference)
    Vec2& operator+=(const Vec2& rhs) {
        x += rhs.x;
        y += rhs.y;
        return *this;
    }

    // Binary + defined in terms of +=
    friend Vec2 operator+(Vec2 lhs, const Vec2& rhs) {
        lhs += rhs;   // lhs is a copy, so this is safe
        return lhs;
    }

    // Unary minus (negation)
    Vec2 operator-() const { return Vec2(-x, -y); }

    // Scalar multiplication
    friend Vec2 operator*(const Vec2& v, double s) {
        return Vec2(v.x * s, v.y * s);
    }
    friend Vec2 operator*(double s, const Vec2& v) {
        return v * s;  // reuse the above
    }
};

    Comparison Operators

    Before C++20, you had to write ==, !=, <, >, <=, >= separately — six functions! C++20's spaceship operator (<=>) generates them all from one definition.

    Pre-C++20 Way

    class Fraction {
    int num, den;
public:
    bool operator==(const Fraction& rhs) const {
        return num * rhs.den == rhs.num * den;
    }
    bool operator!=(const Fraction& rhs) const {
        return !(*this == rhs);
    }
    bool operator<(const Fraction& rhs) const {
        return (long long)num * rhs.den < (long long)rhs.num * den;
    }
    // ... and so on for >, <=, >=
};

    C++20: The Spaceship Operator

    Define <=> and the compiler generates all six comparisons for you. The return type tells the compiler the kind of ordering:

    • std::strong_ordering — exactly one of <, ==, > is true. For types like int, string.
    • std::weak_ordering — two distinct values can compare equivalent but not equal.
    • std::partial_ordering — some values are incomparable (like NaN for floats).
    #include <compare>

class Point {
    int x, y;
public:
    // One line generates ==, !=, <, >, <=, >=
    auto operator<=>(const Point&) const = default;
};

class Fraction {
    int num, den;
public:
    std::strong_ordering operator<=>(const Fraction& rhs) const {
        // Cross-multiply to avoid division
        auto lhs_val = (long long)num * rhs.den;
        auto rhs_val = (long long)rhs.num * den;
        return lhs_val <=> rhs_val;
    }
    bool operator==(const Fraction& rhs) const {
        return num * rhs.den == rhs.num * den;
    }
};
    Tip: For simple structs, auto operator<=>(const T&) const = default; does member-by-member comparison in declaration order. Write the body only when you need custom logic.

    Stream Operators (<< and >>)

    Overloading << for std::ostream lets your objects work with std::cout, std::ostringstream, and any output stream. These must be free functions because the left operand is the stream, not your object.

    class Vec2 {
    double x, y;
public:
    Vec2(double x, double y) : x(x), y(y) {}

    friend std::ostream& operator<<(std::ostream& os, const Vec2& v) {
        return os << "(" << v.x << ", " << v.y << ")";
    }

    friend std::istream& operator>>(std::istream& is, Vec2& v) {
        return is >> v.x >> v.y;
    }
};

// Usage: std::cout << myVec;  // prints "(3.5, 2.1)"

    Best practice: Return the stream reference to allow chaining: cout << a << " " << b;

    Subscript Operator ([])

    The subscript operator lets your object be indexed like an array. Provide both const and non-const versions:

    class Matrix {
    std::vector<std::vector<double>> data_;
    size_t rows_, cols_;
public:
    Matrix(size_t r, size_t c) : data_(r, std::vector<double>(c)), rows_(r), cols_(c) {}

    // Returns a proxy so m[i][j] works
    std::vector<double>& operator[](size_t row) {
        return data_[row];
    }
    const std::vector<double>& operator[](size_t row) const {
        return data_[row];
    }
};

// Usage: mat[2][3] = 4.5;

    C++23 enhancement: Multi-dimensional subscript: operator[](size_t i, size_t j) so you can write mat[2, 3] instead of the double-bracket hack.

    Function Call Operator (())

    Overloading () makes your object callable like a function. These objects are called functors. They can hold state, unlike plain function pointers, and are widely used with STL algorithms:

    class Adder {
    int value_;
public:
    explicit Adder(int v) : value_(v) {}

    int operator()(int x) const {
        return x + value_;
    }
};

Adder add5(5);
int result = add5(10);  // result = 15

// With STL algorithms:
std::vector<int> nums = {1, 2, 3, 4};
std::transform(nums.begin(), nums.end(), nums.begin(), Adder(10));
// nums = {11, 12, 13, 14}

    Functors were the primary way to pass behavior to algorithms before C++11 lambdas. Lambdas are actually syntactic sugar for anonymous functors.

    Dereference & Arrow Operators (*, ->)

    Overloading * and -> is how smart pointers and iterators work. The arrow operator is special: it must return either a raw pointer or an object with its own operator-> (the compiler chains calls until it gets a pointer).

    template<typename T>
class SmartPtr {
    T* ptr_;
public:
    explicit SmartPtr(T* p) : ptr_(p) {}
    ~SmartPtr() { delete ptr_; }

    T& operator*() const { return *ptr_; }
    T* operator->() const { return ptr_;  }

    // Disable copy, enable move...
};

SmartPtr<Widget> w(new Widget());
w->doWork();    // calls Widget::doWork()
(*w).doWork();  // same thing

    Increment & Decrement (++, --)

    The tricky part is distinguishing prefix (++x) from postfix (x++). The convention: postfix takes a dummy int parameter:

    class Counter {
    int value_;
public:
    explicit Counter(int v = 0) : value_(v) {}

    // Prefix: ++c (increment, return new value)
    Counter& operator++() {
        ++value_;
        return *this;
    }

    // Postfix: c++ (return old value, then increment)
    Counter operator++(int) {  // dummy int distinguishes postfix
        Counter old = *this;
        ++value_;
        return old;
    }

    int get() const { return value_; }
};

    Performance note: Prefix (++c) is more efficient than postfix (c++) for non-trivial types because postfix must create a copy of the old value. Prefer prefix when you don't need the old value.

    Conversion Operators

    Conversion operators let your type implicitly or explicitly convert to another type. Use explicit to prevent surprising implicit conversions:

    class Percentage {
    double value_;
public:
    explicit Percentage(double v) : value_(v) {}

    // Explicit conversion to double
    explicit operator double() const { return value_ / 100.0; }

    // Explicit conversion to bool (is non-zero?)
    explicit operator bool() const { return value_ != 0.0; }
};

Percentage p(75);
double d = static_cast<double>(p);  // 0.75
if (p) { /* works because of explicit bool */ }

    Best Practices

    • Respect semantics: + should add, == should compare equality. Don't be clever.
    • Be consistent: If you overload ==, overload != too (or use the spaceship operator).
    • Implement compound first: Define +=, then write + in terms of it.
    • Return the right type: + returns by value, += returns by reference, == returns bool.
    • Use friend for symmetry: Free function operators allow conversions on both sides.
    • Mark const correctly: Operators that don't modify the object should be const.
    • Prefer explicit for conversions: Implicit conversions cause subtle bugs.
    • Don't overload &&, ||, or ,: Overloading disables short-circuit evaluation.

    Operators You Cannot Overload

    These operators are fixed by the language and cannot be user-defined:

    • :: (scope resolution)
    • . (member access)
    • .* (pointer to member access)
    • ?: (ternary conditional)
    • sizeof, alignof, typeid
