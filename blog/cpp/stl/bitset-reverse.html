    Reversing Bits in std::bitset

    May 2026 · 14 min read

    “Reverse the bits of a std::bitset” sounds like a one-liner — and for tiny bitsets it is — but the moment you stop and ask which end is which, the problem gets surprisingly subtle. This post walks through indexing first, then four progressively faster approaches, with a step-by-step animation of the canonical two-pointer swap.

    Quick Refresher: How std::bitset is Indexed

    std::bitset<N> is a fixed-size sequence of N bits. The convention you must internalise is:

    • b[0] is the least-significant bit (LSB).
    • b[N-1] is the most-significant bit (MSB).
    • When you stream a bitset to std::cout, it prints MSB first — so the leftmost character is b[N-1].
    #include <bitset>
#include <iostream>

int main() {
    std::bitset<8> b("11010010");  // string is MSB → LSB
    std::cout << b[0] << "\n";    // 0  (LSB, rightmost char)
    std::cout << b[7] << "\n";    // 1  (MSB, leftmost char)
    std::cout << b   << "\n";     // 11010010
}

    So when we say “reverse the bits”, we mean: produce a new bitset where bit i is the original bit N-1-i. After reversal, 11010010 becomes 01001011.

    Mind the convention. Reversing a bitset is not the same as printing it backwards in memory. It is a logical reversal of the indexed sequence. The string-printing direction is just a side effect of MSB-first formatting.

    Why Would You Ever Reverse Bits?

    • FFT bit-reversal permutation — iterative Cooley–Tukey FFT requires inputs at index i to move to index reverse_bits(i).
    • Endianness & protocol parsing — some serial protocols (I²C, SPI variants, DVB) transmit LSB first while a host expects MSB first.
    • Hardware emulation — register layouts in a manual sometimes index from the opposite end.
    • CRCs & hashing — reflected polynomials are essentially bit-reversed forms.
    • Algorithm puzzles — Leetcode 190, sliding mirror problems, palindrome checks.

    Approach 1: Two-Pointer In-Place Swap

    The cleanest, most general approach is to walk two indices toward each other and swap. This works for any N known at compile time and never allocates.

    #include <bitset>
#include <cstddef>

template <std::size_t N>
void reverse_bits(std::bitset<N>& b) {
    for (std::size_t i = 0, j = N - 1; i < j; ++i, --j) {
        bool tmp = b[i];
        b[i] = b[j];
        b[j] = tmp;
    }
}

    Two subtleties to note:

    • b[i] returns a proxy std::bitset<N>::reference, not a real bool&, so std::swap(b[i], b[j]) works only because the proxy is assignable. The explicit tmp dance is portable and unambiguous.
    • The loop stops at i < j, which correctly leaves the middle bit untouched when N is odd.

    Time complexity is O(N). Space is O(1). For a bitset<1024> that's only 512 swaps — already fast enough for almost everything you will see in production.

    Approach 2: The String Trick

    Because to_string() emits bits MSB-first and the bitset constructor reads MSB-first, simply reversing the string flips the bit order:

    #include <bitset>
#include <algorithm>
#include <string>

template <std::size_t N>
std::bitset<N> reversed(const std::bitset<N>& b) {
    std::string s = b.to_string();
    std::reverse(s.begin(), s.end());
    return std::bitset<N>(s);
}

    This is concise and obviously correct, but it allocates a heap string of length N twice (once in to_string, once when the constructor scans it). For competitive code paths you'll prefer Approach 1; for a one-off conversion in startup code, this is fine.

    Why does reversing the string work? If the printed string is s[0] s[1] … s[N-1] = b[N-1] b[N-2] … b[0], then the reversed string is b[0] b[1] … b[N-1]. Feeding that back into the constructor (which still reads MSB first) places b[0] at the new MSB and b[N-1] at the new LSB — exactly the index reversal we want.

    Approach 3: SWAR for N ≤ 64

    If your bitset fits in a single machine word, you can extract the underlying integer, reverse with classic SIMD-Within-A-Register (SWAR) tricks, then reconstruct. Each step swaps adjacent groups of bits using a mask.

    #include <bitset>
#include <cstdint>

std::bitset<32> reverse_swar(std::bitset<32> b) {
    uint32_t x = static_cast<uint32_t>(b.to_ulong());
    x = ((x & 0x55555555u) << 1)  | ((x >> 1)  & 0x55555555u);  // swap odd/even bits
    x = ((x & 0x33333333u) << 2)  | ((x >> 2)  & 0x33333333u);  // swap pairs
    x = ((x & 0x0f0f0f0fu) << 4)  | ((x >> 4)  & 0x0f0f0f0fu);  // swap nibbles
    x = ((x & 0x00ff00ffu) << 8)  | ((x >> 8)  & 0x00ff00ffu);  // swap bytes
    x = (x << 16) | (x >> 16);                                  // swap halves
    return std::bitset<32>(x);
}

    Each line is a butterfly: it splits the word into adjacent groups of size 1, 2, 4, 8, 16 and swaps them. After log₂ N butterflies the entire word is reversed.

    How fast is it really? On modern x86, every line is two ANDs, a shift, and an OR — five instructions, all latency-1, all independent across iterations. A 32-bit reversal compiles to roughly 25 cycles in the worst case, vs. ~60 cycles for the loop version. The win grows for 64-bit words.

    C++20 shortcut. If you only need to reverse 8/16/32/64 bits and don't insist on writing it yourself, std::bit_cast + __builtin_bitreverse32 (Clang/GCC) or the bitintrin family on MSVC compile to a single CPU instruction (RBIT) on ARM. There is no portable standard-library bit reversal yet.

    Approach 4: Word-Wise Reversal for Huge N

    For a bitset<1<<20> (a million bits), Approach 1 is still O(N) but does N/2 proxy assignments — each of which has to compute a word index and a bit mask. We can do better by working word at a time.

    std::bitset doesn't expose its underlying storage, so we either:

    1. Reverse a parallel std::vector<uint64_t> we maintain ourselves and assign the result back via the string constructor, or
    2. Skip std::bitset and use a hand-rolled bit array for hot paths.

    The general idea, given a packed word array w[0..W-1] where word k holds bits [64k, 64k+63]:

    // Reverse a bit array stored as W 64-bit words.
// Step 1: reverse each word (SWAR, one CPU instruction on ARM).
// Step 2: reverse the order of words.
void reverse_words(uint64_t* w, std::size_t W) {
    for (std::size_t k = 0; k < W; ++k) w[k] = bitreverse64(w[k]);
    for (std::size_t i = 0, j = W - 1; i < j; ++i, --j)
        std::swap(w[i], w[j]);
}

    That's O(N / 64) work — a 64× speedup over the per-bit loop. The catch: if N is not a multiple of 64, the highest word has unused padding bits that must be shifted out after the reversal. The arithmetic is straightforward but easy to get off-by-one, which is why most production code reaches for Boost's dynamic_bitset or rolls a small custom type once and reuses it.

    Common Pitfalls

    • Confusing print order with index order. If you reverse the printed string character by character in your head, you may convince yourself the operation is trivial. The animation above keeps the index axis explicit so you don't fall for it.
    • to_ulong / to_ullong overflow. Both throw std::overflow_error when N exceeds the target width. SWAR is only valid when you statically know N is small enough.
    • Iterating with auto&. You can't take a reference to a bit; auto& over a bitset doesn't compile. Use indices or the proxy type explicitly.
    • Off-by-one when N is odd. Approach 1's i < j guard handles it correctly; flipping it to i <= j would re-swap the centre bit and silently waste a step.

    A Reusable Header

    If you find yourself reaching for this often, drop it in a small utility header:

    // bitset_reverse.hpp
#pragma once
#include <bitset>
#include <cstddef>

namespace util {

template <std::size_t N>
std::bitset<N> reversed(std::bitset<N> b) {
    for (std::size_t i = 0, j = N - 1; i < j; ++i, --j) {
        bool t = b[i];
        b[i] = b[j];
        b[j] = t;
    }
    return b;
}

} // namespace util

    Summary

    • b[0] is the LSB; reversal means bit i becomes the original bit N-1-i.
    • The two-pointer swap is the right default — O(N), no allocation, works for any N.
    • The string trick is a readable one-liner when you don't care about allocations.
    • SWAR butterflies hit O(log N) for words ≤ 64 bits and compile to a handful of instructions.
    • For very large bit arrays, leave std::bitset behind and reverse word-wise for a 64× speedup.
