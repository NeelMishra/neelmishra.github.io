    The C++ Memory Model

    March 2026 · 35 min read

    Why Do We Need a Memory Model?

    In a single-threaded world, every read sees the most recent write. Simple. But modern hardware shatters this intuition in two ways:

    1. Compiler reordering: The compiler may rearrange instructions for optimization, as long as the observable behavior in a single thread is unchanged.
    2. CPU reordering: Out-of-order execution, store buffers, and cache hierarchies can cause writes to become visible to other cores in a different order than they were issued.

    The C++ memory model (introduced in C++11) defines the rules for when writes by one thread become visible to reads by another thread. It provides a formal framework of happens-before relationships that you can use to reason about multithreaded correctness.

    ▶ Instruction Reordering

    Watch how the CPU/compiler might reorder stores, causing Thread 2 to see unexpected values.

    The Happens-Before Relationship

    If operation A happens-before operation B, then the effects of A are guaranteed to be visible to B. The C++ standard defines several sources of happens-before:

    • Sequenced-before (within one thread): if statement A comes before statement B in program order, A is sequenced-before B.
    • Synchronizes-with: A release store synchronizes-with an acquire load that reads the stored value. This creates a happens-before across threads.
    • Transitivity: If A happens-before B and B happens-before C, then A happens-before C.

    The Six Memory Orderings

    memory_order_seq_cst (Default)

    The strongest ordering. All threads see all seq_cst operations in the same total order. It is like the operations occurred in some single interleaving that all threads agree on. This is the default and the easiest to reason about.

    x.store(1);                           // seq_cst by default
int v = y.load();                     // seq_cst by default

    memory_order_acquire / memory_order_release

    A release store ensures that all writes before it are visible to any thread that does an acquire load of the same variable and reads the stored value. This is the standard pattern for producer-consumer communication:

    std::atomic<bool> ready{false};
int data = 0;

// Producer (Thread 1)
data = 42;                                         // ordinary write
ready.store(true, std::memory_order_release);      // release: publishes data

// Consumer (Thread 2)
while (!ready.load(std::memory_order_acquire)) {}  // acquire: sees data
assert(data == 42);                                // guaranteed!

    memory_order_relaxed

    The weakest ordering. No ordering constraints relative to other operations. Only guarantees atomicity (no tearing). Useful for counters where the exact ordering relative to other variables does not matter:

    std::atomic<int> hit_count{0};
// Many threads:
hit_count.fetch_add(1, std::memory_order_relaxed);  // fine for a counter

    memory_order_acq_rel

    Combines acquire and release semantics. Used for read-modify-write operations (like CAS) that both consume and publish data.

    memory_order_consume

    A weaker version of acquire that only orders data-dependent reads. Extremely tricky to use correctly, and most compilers promote it to acquire anyway. In practice, avoid it.

    ▶ Memory Ordering Spectrum

    From strongest (safest, slowest) to weakest (fastest, trickiest).

    Memory Fences (Barriers)

    A fence is a standalone ordering constraint not tied to a particular variable:

    data = 42;
std::atomic_thread_fence(std::memory_order_release);  // fence
flag.store(true, std::memory_order_relaxed);

// Equivalent to: flag.store(true, memory_order_release)
// but the fence applies to ALL writes before it, not just flag

    Fences are a low-level tool. Prefer atomic operations with the correct memory ordering instead of standalone fences.

    Complete Acquire-Release Example

    struct Message {
    int payload;
    int sequence;
};

std::atomic<Message*> mailbox{nullptr};

// Producer
void send(int seq, int data) {
    auto* msg = new Message{data, seq};
    // The release store ensures the Message construction
    // is visible to the consumer before the pointer is.
    mailbox.store(msg, std::memory_order_release);
}

// Consumer
void receive() {
    Message* msg = nullptr;
    while (!(msg = mailbox.load(std::memory_order_acquire))) {
        std::this_thread::yield();
    }
    // Acquire guarantees we see the fully-constructed Message
    assert(msg->payload != 0);
    delete msg;
}

    When Relaxed Is Safe

    Relaxed ordering is safe when:

    • You only need atomicity (no tearing), not ordering.
    • The variable's value is independent of other shared data.
    • Examples: statistics counters, generating unique IDs, checking a stop flag (if combined with another synchronization mechanism for the actual data).

    Common Patterns

    The Flag Pattern (Acquire-Release)

    std::atomic<bool> ready{false};
int shared_data = 0;

// Writer: store data, then set flag
shared_data = 42;
ready.store(true, std::memory_order_release);

// Reader: wait for flag, then read data
while (!ready.load(std::memory_order_acquire)) {}
int local = shared_data;  // guaranteed to be 42

    Publish-Subscribe (Pointer)

    std::atomic<Widget*> published{nullptr};

// Publisher
auto* w = new Widget();
w->init();  // all writes to *w happen before the release-store
published.store(w, std::memory_order_release);

// Subscriber
Widget* w = nullptr;
while (!(w = published.load(std::memory_order_acquire))) {}
w->use();  // guaranteed to see fully initialized widget

    x86 vs ARM: Why This Matters

    x86 has a relatively strong memory model (Total Store Ordering). Most operations are already implicitly ordered, so the performance difference between seq_cst and weaker orderings is small.

    ARM and RISC-V have weak memory models. Stores can be reordered freely. Here, using weaker orderings (acquire-release or relaxed) instead of seq_cst avoids expensive hardware fences and can yield significant performance gains.

    Start with seq_cst. Optimize to weaker orderings only with profiling evidence and deep understanding. Incorrect memory ordering bugs are among the hardest to diagnose: they may only manifest on specific hardware, under specific timing conditions, and are typically invisible to debuggers.

    Summary

    • The C++ memory model defines when writes become visible across threads.
    • Happens-before is the central concept: if A happens-before B, B sees A's effects.
    • seq_cst (default): total order, easiest to reason about, potentially slowest.
    • acquire/release: sufficient for most synchronization patterns (flags, publish-subscribe).
    • relaxed: only guarantees atomicity, no ordering. Safe for independent counters.
    • Fences are low-level; prefer using memory orderings on atomic operations directly.
    • The hardware architecture (x86 vs ARM) affects the practical performance impact of ordering choices.
