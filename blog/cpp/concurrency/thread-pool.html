    Thread Pools

    March 2026 · 28 min read

    Why Thread Pools?

    Creating and destroying threads is expensive: each std::thread construction involves an OS syscall, stack allocation (~1MB default), and scheduler bookkeeping. For a web server handling thousands of requests or a game engine running many small tasks, spawning a new thread per task is wasteful.

    A thread pool creates a fixed set of worker threads at startup. Tasks are submitted to a queue; idle workers pull tasks and execute them. The threads are reused, eliminating per-task creation overhead.

    ▶ Thread Pool Architecture

    Tasks enter the queue, workers pull and execute them.

    Building a Thread Pool from Scratch

    #include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <functional>
#include <vector>
#include <future>

class ThreadPool {
    std::vector<std::thread> workers_;
    std::queue<std::function<void()>> tasks_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool stop_ = false;

public:
    explicit ThreadPool(size_t num_threads) {
        for (size_t i = 0; i < num_threads; ++i) {
            workers_.emplace_back([this] {
                while (true) {
                    std::function<void()> task;
                    {
                        std::unique_lock lock(mtx_);
                        cv_.wait(lock, [this] {
                            return stop_ || !tasks_.empty();
                        });
                        if (stop_ && tasks_.empty()) return;
                        task = std::move(tasks_.front());
                        tasks_.pop();
                    }
                    task();
                }
            });
        }
    }

    template<typename F, typename... Args>
    auto submit(F&& f, Args&&... args)
        -> std::future<std::invoke_result_t<F, Args...>>
    {
        using ReturnType = std::invoke_result_t<F, Args...>;
        auto task = std::make_shared<std::packaged_task<ReturnType()>>(
            std::bind(std::forward<F>(f), std::forward<Args>(args)...)
        );
        auto fut = task->get_future();
        {
            std::lock_guard lock(mtx_);
            tasks_.emplace([task]{ (*task)(); });
        }
        cv_.notify_one();
        return fut;
    }

    ~ThreadPool() {
        {
            std::lock_guard lock(mtx_);
            stop_ = true;
        }
        cv_.notify_all();
        for (auto& w : workers_) w.join();
    }
};

    Usage

    ThreadPool pool(std::thread::hardware_concurrency());

auto result1 = pool.submit([](int a, int b) { return a + b; }, 10, 20);
auto result2 = pool.submit(heavy_compute, data);

std::cout << result1.get() << "\n";  // 30
std::cout << result2.get() << "\n";

    std::jthread (C++20)

    std::jthread is a joining thread that destructs safely and supports cooperative cancellation:

    void worker(std::stop_token stoken) {
    while (!stoken.stop_requested()) {
        auto task = queue.pop();  // blocks
        task();
    }
}

std::jthread t(worker);
// ~jthread() automatically:
//   1. Calls request_stop() on its stop_source
//   2. Calls join()
// No manual cleanup needed!

    Work Stealing

    In a naive pool, all workers share one queue (single point of contention). In a work-stealing pool, each worker has its own local queue. When a worker's queue is empty, it steals from another worker's queue (from the opposite end to minimize contention).

    Work stealing provides better load distribution and cache locality. It is used in Intel TBB, Go's goroutine scheduler, and Java's ForkJoinPool.

    How Many Threads?

    • CPU-bound tasks: Use std::thread::hardware_concurrency() threads. More threads cause context-switch overhead without benefit.
    • I/O-bound tasks: Use more threads than cores, since threads spend most time waiting on I/O. A common heuristic: 2x to 4x the core count.
    • Mixed workloads: Separate into CPU and I/O pools with different sizes.

    Graceful Shutdown

    The destructor must:

    1. Set a stop flag.
    2. Notify all workers (so sleeping workers wake up).
    3. Join all threads.
    4. Optionally: drain remaining tasks, or discard them.

    Summary

    • Thread pools reuse threads, avoiding per-task creation overhead.
    • The core: a task queue + condition variable + worker loop.
    • submit() returns a std::future for the task's result.
    • std::jthread (C++20) provides auto-join and cooperative cancellation.
    • Work stealing improves load balancing for uneven task sizes.
    • Size the pool based on workload type: core count for CPU-bound, more for I/O-bound.
