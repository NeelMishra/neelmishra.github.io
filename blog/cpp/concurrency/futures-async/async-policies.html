    std::async Launch Policies

    March 2026

    Overview

    std::async is the highest-level way to launch concurrent work in the C++ standard library. You hand it a callable, and it returns a std::future for the result. The launch policy controls where and when the work actually executes. 

    #include <future>

auto fut = std::async(launch_policy, callable, args...);

    There are three possible policies:

    • std::launch::async
    • std::launch::deferred
    • std::launch::async | std::launch::deferred (the default)

    std::launch::async

    Forces the runtime to execute the callable on a new thread (as if by std::thread). The work begins immediately, and the returned future is associated with that thread. 

    #include <future>
#include <iostream>
#include <thread>

int main() {
    auto fut = std::async(std::launch::async, []{
        std::cout << "Running on thread "
                  << std::this_thread::get_id() << "\n";
        return 42;
    });

    std::cout << "Main thread: "
              << std::this_thread::get_id() << "\n";
    std::cout << "Result: " << fut.get() << "\n";
}

    The standard guarantees that the callable will run asynchronously. On all major implementations (libstdc++, libc++, MSVC), this spawns a real OS thread.

    std::launch::deferred

    The callable is not executed until the consumer calls get() or wait() on the future. Execution happens on the caller's thread, not on a new thread. This is essentially lazy evaluation. 

    #include <future>
#include <iostream>
#include <thread>

int main() {
    auto fut = std::async(std::launch::deferred, []{
        std::cout << "Executing on thread "
                  << std::this_thread::get_id() << "\n";
        return 99;
    });

    std::cout << "Main thread: "
              << std::this_thread::get_id() << "\n";

    // Nothing has run yet.
    std::cout << "Result: " << fut.get() << "\n";
    // Output shows both IDs are the same.
}
    Deferred + wait_for. If you call wait_for() on a deferred future, it always returns std::future_status::deferred without executing the task. Only get() or wait() triggers execution.

    Default: async | deferred

    When you omit the policy or use std::launch::async | std::launch::deferred, the implementation is free to choose either strategy. In practice, most implementations choose async when system resources permit and deferred as a fallback. 

    // These are equivalent
auto f1 = std::async(compute);
auto f2 = std::async(std::launch::async | std::launch::deferred, compute);

    Because the choice is implementation-defined, code that cares about true concurrency should always specify std::launch::async explicitly.

    The Future Destructor Blocks

    This is the single most surprising behaviour of std::async. If the future returned by std::async(launch::async, ...) is destroyed without being get()-ed, its destructor blocks until the task completes. This applies only to futures obtained from std::async, not to futures from std::promise or std::packaged_task.

    Gotcha. std::async returns a special future whose destructor joins the spawned thread. If you discard the future, the "asynchronous" call becomes synchronous.

    Visualizing the Block

    Step through to see how discarding the future turns an async call into a blocking call.

    Demonstrating the Block

    #include <future>
#include <iostream>
#include <chrono>
#include <thread>

void slow_work() {
    std::this_thread::sleep_for(std::chrono::seconds(2));
    std::cout << "Work done\n";
}

int main() {
    auto start = std::chrono::steady_clock::now();

    // WRONG: future is a temporary, destroyed at end of statement
    std::async(std::launch::async, slow_work);
    // ^ destructor blocks here for 2 seconds!

    auto elapsed = std::chrono::steady_clock::now() - start;
    auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(
        elapsed).count();
    std::cout << "Elapsed: " << ms << " ms\n";
    // Prints ~2000 ms, NOT ~0 ms
}

    The returned future is a temporary. It is destroyed at the semicolon, and the destructor blocks until slow_work() finishes. The "asynchronous" call is effectively synchronous.

    The Fix

    int main() {
    auto start = std::chrono::steady_clock::now();

    // CORRECT: capture in a named variable
    auto fut = std::async(std::launch::async, slow_work);

    // Main thread can do other work here...
    std::cout << "Doing other work...\n";

    fut.get();  // wait only when we need the result

    auto elapsed = std::chrono::steady_clock::now() - start;
    auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(
        elapsed).count();
    std::cout << "Elapsed: " << ms << " ms\n";
}

    Fire-and-Forget with async

    True fire-and-forget is impossible with std::async because of the destructor-blocks rule. If you truly need fire-and-forget, use a detached thread or a dedicated thread pool that owns the futures. 

    // True fire-and-forget using detached thread
void fire_and_forget(auto&& f) {
    std::thread(std::forward<decltype(f)>(f)).detach();
}

// Or: move the future into a long-lived container
class AsyncPool {
    std::vector<std::future<void>> futures_;
public:
    void spawn(auto&& f) {
        futures_.push_back(
            std::async(std::launch::async,
                       std::forward<decltype(f)>(f)));
    }
    ~AsyncPool() {
        for (auto& f : futures_) f.get();
    }
};
    Detached threads. Detaching threads avoids the blocking destructor but introduces lifetime risks. The detached thread must not access any stack-local variable that might go out of scope. Prefer a pool-based approach in production code.

    Return Value Handling

    std::async deduces the return type of the callable and produces a matching std::future<R>. 

    #include <future>
#include <string>
#include <iostream>

std::string greet(const std::string& name) {
    return "Hello, " + name + "!";
}

int main() {
    std::future<std::string> fut =
        std::async(std::launch::async, greet, "World");

    std::cout << fut.get() << "\n";  // "Hello, World!"
}

    Returning void

    auto fut = std::async(std::launch::async, []{
    // side-effect-only work
    std::cout << "Done\n";
});
fut.get();  // returns void, but still waits

    Returning move-only types

    #include <future>
#include <memory>
#include <iostream>

int main() {
    auto fut = std::async(std::launch::async, []{
        return std::make_unique<int>(42);
    });

    std::unique_ptr<int> ptr = fut.get();  // moved out
    std::cout << *ptr << "\n";              // 42
}

    When to Use async vs. Manual Threads

    Use case Recommendation
    Compute a value and return it std::async(launch::async, ...)
    Long-lived background worker std::thread (or std::jthread)
    Task queue with many short tasks Thread pool with packaged_task
    Need to control thread count std::thread pool
    Lazy evaluation std::async(launch::deferred, ...)
    Fire-and-forget std::thread::detach() or pool

    Deferred Pitfalls with wait_for

    A loop that polls via wait_for will never trigger a deferred task. This can create an infinite loop if the default policy selected deferred. 

    // Potentially infinite loop if policy chooses deferred
auto fut = std::async(compute);  // default policy

while (fut.wait_for(std::chrono::milliseconds(10))
       != std::future_status::ready) {
    // If deferred, wait_for always returns deferred, never ready!
    std::cout << "Still waiting...\n";
}

// Safe version: check for deferred first
auto fut2 = std::async(compute);
if (fut2.wait_for(std::chrono::seconds(0))
    == std::future_status::deferred) {
    // Task is deferred; call get() to execute it now
    auto result = fut2.get();
} else {
    // Task is (or will be) running asynchronously
    while (fut2.wait_for(std::chrono::milliseconds(10))
           != std::future_status::ready) {
        // do other work
    }
    auto result = fut2.get();
}

    Complete Example: Parallel Summation

    #include <future>
#include <vector>
#include <numeric>
#include <iostream>

long long partial_sum(const std::vector<int>& v,
                      std::size_t lo, std::size_t hi) {
    long long s = 0;
    for (std::size_t i = lo; i < hi; ++i) s += v[i];
    return s;
}

int main() {
    const std::size_t N = 10'000'000;
    std::vector<int> data(N, 1);

    std::size_t mid = N / 2;

    auto fut1 = std::async(std::launch::async,
                           partial_sum, std::cref(data), 0, mid);
    auto fut2 = std::async(std::launch::async,
                           partial_sum, std::cref(data), mid, N);

    long long total = fut1.get() + fut2.get();
    std::cout << "Sum: " << total << "\n";  // 10000000
}

    Exception Forwarding

    If the callable throws, the exception is stored in the shared state and re-thrown when the consumer calls get(). 

    #include <future>
#include <iostream>
#include <stdexcept>

int main() {
    auto fut = std::async(std::launch::async, []{
        throw std::runtime_error("async failure");
        return 0;
    });

    try {
        fut.get();
    } catch (const std::runtime_error& e) {
        std::cout << "Caught from async: " << e.what() << "\n";
    }
}

    Key Takeaways

    • launch::async guarantees a new thread; launch::deferred executes lazily on the caller's thread.
    • The default policy lets the implementation choose. Specify explicitly when concurrency matters.
    • The future returned by std::async blocks in its destructor. Always capture it in a named variable.
    • True fire-and-forget is not possible with std::async; use a thread pool or detached thread.
    • wait_for on a deferred future never returns ready; check for deferred first.
    • Exceptions thrown in the task are forwarded to the consumer on get().
