    std::shared_future

    March 2026

    Why shared_future?

    std::future is a move-only, single-consumer handle. Once you call get(), the result is consumed and the future becomes invalid. But many concurrent patterns need multiple threads to observe the same result. That is where std::shared_future comes in.

    A shared_future is copyable. Each copy refers to the same shared state, and every copy can call get() independently. The result is returned by const reference, so it is never moved out.

    Rule of thumb. If exactly one thread reads the result, use std::future. If multiple threads need the result, use std::shared_future.

    Creating a shared_future

    From future::share()

    The most common path is to convert an existing std::future by calling share(). This moves ownership of the shared state out of the future and into the new shared_future. 

    #include <future>
#include <iostream>

int main() {
    std::promise<int> prom;
    std::future<int> fut = prom.get_future();

    // Convert to shared_future (fut is now invalid)
    std::shared_future<int> sfut = fut.share();

    prom.set_value(42);

    // Both calls succeed
    std::cout << sfut.get() << "\n";  // 42
    std::cout << sfut.get() << "\n";  // 42 (same reference)
}

    Directly from a promise

    You can also construct a shared_future directly from the std::future returned by get_future(): 

    std::promise<int> prom;
std::shared_future<int> sfut(prom.get_future()); // implicit conversion

    Broadcasting Results

    Step through to see one producer broadcasting a result to three consumers through copies of a shared_future.

    Multiple Consumers

    Each consumer thread receives its own copy of the shared_future. All copies are safe to call get() concurrently. 

    #include <future>
#include <thread>
#include <vector>
#include <iostream>

int main() {
    std::promise<int> prom;
    std::shared_future<int> sfut = prom.get_future().share();

    std::vector<std::thread> consumers;
    for (int i = 0; i < 4; ++i) {
        consumers.emplace_back([sfut, i]{          // copy sfut
            int val = sfut.get();                   // all succeed
            std::cout << "Thread " << i
                      << " got " << val << "\n";
        });
    }

    prom.set_value(100);

    for (auto& t : consumers) t.join();
}
    Thread safety. Concurrent calls to get() on different copies of a shared_future that refer to the same shared state are safe. Concurrent calls on the same object are also safe because get() is a const member function.

    valid()

    valid() returns true if the shared_future refers to a shared state. Unlike std::future, calling get() does not invalidate a shared_future. 

    std::promise<int> prom;
std::shared_future<int> sfut = prom.get_future().share();

std::cout << sfut.valid() << "\n";  // 1 (true)

prom.set_value(10);
sfut.get();

std::cout << sfut.valid() << "\n";  // still 1 (true)

    A shared_future becomes invalid only when it is default-constructed or moved-from.

    share() Conversion

    std::future::share() is the standard way to convert. After the call, the original future is in a moved-from state and is no longer valid. 

    std::promise<std::string> prom;
std::future<std::string> fut = prom.get_future();

// Convert
std::shared_future<std::string> sfut = fut.share();

std::cout << fut.valid()  << "\n";  // 0 (false)
std::cout << sfut.valid() << "\n";  // 1 (true)

    Return Type of get()

    For shared_future<T>, get() returns const T&. This avoids copying and prevents accidental mutation by consumers. 

    std::promise<std::vector<int>> prom;
std::shared_future<std::vector<int>> sfut = prom.get_future().share();

prom.set_value({1, 2, 3, 4, 5});

const std::vector<int>& ref = sfut.get();  // no copy
std::cout << ref.size() << "\n";             // 5

    For shared_future<T&>, get() returns T&. For shared_future<void>, get() returns void.

    Pattern: Start Gate

    A shared_future<void> makes a perfect start gate. Multiple threads wait on their copy; when the controller signals, they all wake up. 

    #include <future>
#include <thread>
#include <vector>
#include <iostream>
#include <chrono>

int main() {
    std::promise<void> gate_promise;
    std::shared_future<void> gate = gate_promise.get_future().share();

    auto racer = [](std::shared_future<void> gate, int id) {
        gate.wait();   // block until gate opens
        auto start = std::chrono::steady_clock::now();
        // Simulate variable work
        std::this_thread::sleep_for(std::chrono::milliseconds(id * 10));
        auto end = std::chrono::steady_clock::now();
        auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(
            end - start).count();
        std::cout << "Racer " << id << " finished in "
                  << ms << " ms\n";
    };

    std::vector<std::thread> threads;
    for (int i = 1; i <= 5; ++i) {
        threads.emplace_back(racer, gate, i);
    }

    std::cout << "All racers ready. Go!\n";
    gate_promise.set_value();   // open the gate

    for (auto& t : threads) t.join();
}

    Concurrent get() Calls

    Because get() on shared_future returns a const reference, multiple threads can call it simultaneously without any data race. The standard guarantees that accesses to the shared state are synchronized. 

    #include <future>
#include <thread>
#include <vector>
#include <iostream>
#include <cassert>

int main() {
    std::promise<int> prom;
    auto sfut = prom.get_future().share();

    prom.set_value(7);

    std::vector<std::thread> threads;
    for (int i = 0; i < 8; ++i) {
        threads.emplace_back([sfut]{
            // All threads read the same value concurrently
            assert(sfut.get() == 7);
        });
    }

    for (auto& t : threads) t.join();
    std::cout << "All 8 threads read the value safely.\n";
}

    Common Pitfalls

    Copying the future vs. the shared state

    Copying a shared_future creates a new handle to the same shared state. This is cheap (essentially a reference-counted pointer bump). Do not confuse this with copying the stored value.

    Dangling references with shared_future<T&>

    If you use shared_future<T&>, the referenced object must outlive all consumers. The shared state does not extend its lifetime.

    Forgetting to call share()

    Passing a std::future (not shared) to multiple threads does not compile because future is move-only. You must call share() first.

    Complete Example: Config Broadcast

    A main thread loads configuration and broadcasts it to all workers via shared_future. 

    #include <future>
#include <thread>
#include <vector>
#include <string>
#include <iostream>

struct Config {
    std::string db_host;
    int db_port;
    int max_retries;
};

void worker(std::shared_future<Config> cfg_fut, int id) {
    const Config& cfg = cfg_fut.get();
    std::cout << "Worker " << id
              << " connecting to " << cfg.db_host
              << ":" << cfg.db_port << "\n";
}

int main() {
    std::promise<Config> prom;
    auto sfut = prom.get_future().share();

    std::vector<std::thread> workers;
    for (int i = 0; i < 4; ++i) {
        workers.emplace_back(worker, sfut, i);
    }

    // Simulate config loading
    Config cfg{"db.example.com", 5432, 3};
    prom.set_value(std::move(cfg));

    for (auto& t : workers) t.join();
}

    Key Takeaways

    • shared_future is copyable; each copy can call get() independently.
    • get() returns const T& and never invalidates the shared_future.
    • Create via future::share() or direct construction from a future.
    • Concurrent get() calls on copies are thread-safe by the standard.
    • Perfect for broadcast patterns: start gates, config distribution, fan-out results.
