    Exception Propagation Across Threads

    March 2026

    The Problem

    Exceptions cannot naturally cross thread boundaries. If a worker thread throws and nobody catches it, std::terminate is called. The C++ standard library solves this with std::exception_ptr, a type-erased, reference-counted handle to any exception object. Futures use this mechanism to transport exceptions from producer threads to consumer threads.

    How Exceptions Cross Thread Boundaries

    Step through to see an exception travel from a worker thread to the main thread via a future.

    std::exception_ptr

    std::exception_ptr is a nullable, copyable, reference-counted smart pointer to an exception object allocated on a special heap. It is the only standard way to store and transport exceptions across scopes and threads. 

    #include <exception>
#include <iostream>
#include <stdexcept>

int main() {
    std::exception_ptr eptr;

    try {
        throw std::runtime_error("something broke");
    } catch (...) {
        eptr = std::current_exception();  // capture
    }

    // eptr can be stored, copied, passed to another thread

    try {
        if (eptr) {
            std::rethrow_exception(eptr);  // re-throw
        }
    } catch (const std::runtime_error& e) {
        std::cout << "Caught: " << e.what() << "\n";
    }
}
    Null check. A default-constructed exception_ptr is null. Always check with if (eptr) before calling std::rethrow_exception.

    set_exception on a Promise

    promise::set_exception accepts an exception_ptr and stores it in the shared state. When the consumer calls get(), the exception is re-thrown. 

    #include <future>
#include <thread>
#include <iostream>
#include <stdexcept>

int main() {
    std::promise<int> prom;
    std::future<int>  fut = prom.get_future();

    std::thread t([p = std::move(prom)]() mutable {
        try {
            // Simulate work that fails
            throw std::runtime_error("database timeout");
        } catch (...) {
            p.set_exception(std::current_exception());
        }
    });

    try {
        int val = fut.get();
    } catch (const std::runtime_error& e) {
        std::cout << "Worker failed: " << e.what() << "\n";
    }

    t.join();
}

    set_exception vs. make_exception_ptr

    You do not always need a try/catch block. If you know the exception type, construct an exception_ptr directly with std::make_exception_ptr. 

    // Using make_exception_ptr (no throw/catch overhead)
prom.set_exception(
    std::make_exception_ptr(std::logic_error("invalid argument"))
);

// Equivalent using try/catch
try {
    throw std::logic_error("invalid argument");
} catch (...) {
    prom.set_exception(std::current_exception());
}

    make_exception_ptr is cleaner and avoids the overhead of actually throwing and catching. Prefer it when you are constructing the exception object yourself rather than catching one.

    Performance note. std::make_exception_ptr still allocates memory for the exception object. But it avoids the stack-unwinding overhead of a real throw.

    std::current_exception

    std::current_exception() captures the currently handled exception as an exception_ptr. It must be called inside a catch block. If called outside a catch block, it returns a null exception_ptr. 

    #include <exception>
#include <iostream>

std::exception_ptr capture() {
    try {
        throw 42;  // can throw anything, not just std::exception
    } catch (...) {
        return std::current_exception();
    }
}

int main() {
    auto eptr = capture();

    try {
        std::rethrow_exception(eptr);
    } catch (int val) {
        std::cout << "Caught int: " << val << "\n";
    }
}

    Nested Exceptions

    C++ supports chaining exceptions with std::throw_with_nested and std::rethrow_if_nested. This is useful in concurrent code when a higher-level handler wants to wrap a lower-level exception without losing context. 

    #include <future>
#include <thread>
#include <iostream>
#include <stdexcept>

void low_level_io() {
    throw std::runtime_error("disk read failed");
}

void mid_level_process() {
    try {
        low_level_io();
    } catch (...) {
        std::throw_with_nested(
            std::logic_error("processing failed"));
    }
}

void print_exception_chain(const std::exception& e, int depth = 0) {
    std::cout << std::string(depth * 2, ' ')
              << e.what() << "\n";
    try {
        std::rethrow_if_nested(e);
    } catch (const std::exception& nested) {
        print_exception_chain(nested, depth + 1);
    }
}

int main() {
    std::promise<int> prom;
    std::future<int>  fut = prom.get_future();

    std::thread t([p = std::move(prom)]() mutable {
        try {
            mid_level_process();
        } catch (...) {
            p.set_exception(std::current_exception());
        }
    });

    try {
        fut.get();
    } catch (const std::exception& e) {
        std::cout << "Exception chain:\n";
        print_exception_chain(e);
    }

    t.join();
}

    Output:

    Exception chain:
processing failed
  disk read failed

    Rethrow Patterns

    Catch-and-forward

    The most common pattern: catch everything in the worker, forward via the promise. 

    std::thread t([p = std::move(prom)]() mutable {
    try {
        do_work();
        p.set_value(result);
    } catch (...) {
        p.set_exception(std::current_exception());
    }
});

    packaged_task auto-capture

    std::packaged_task and std::async automatically capture exceptions. No manual set_exception is needed. 

    auto fut = std::async(std::launch::async, []{
    throw std::runtime_error("oops");
    return 0;
});

try {
    fut.get();
} catch (const std::runtime_error& e) {
    std::cout << e.what() << "\n";  // "oops"
}

    Selective rethrow

    Sometimes you want to inspect the exception type and handle certain errors locally while forwarding others. 

    std::thread t([p = std::move(prom)]() mutable {
    try {
        do_work();
        p.set_value(result);
    } catch (const RecoverableError& e) {
        // Handle locally, retry, etc.
        p.set_value(fallback_result);
    } catch (...) {
        // Forward everything else
        p.set_exception(std::current_exception());
    }
});

    Error Handling with Multiple Workers

    When you spawn several tasks, collect all futures and handle exceptions from each independently. 

    #include <future>
#include <vector>
#include <iostream>
#include <stdexcept>

int main() {
    std::vector<std::future<int>> futures;

    for (int i = 0; i < 5; ++i) {
        futures.push_back(std::async(std::launch::async, [i]{
            if (i == 3) throw std::runtime_error("task 3 failed");
            return i * i;
        }));
    }

    for (int i = 0; i < 5; ++i) {
        try {
            std::cout << "Task " << i << ": "
                      << futures[i].get() << "\n";
        } catch (const std::exception& e) {
            std::cout << "Task " << i << ": ERROR "
                      << e.what() << "\n";
        }
    }
}

    Output:

    Task 0: 0
Task 1: 1
Task 2: 4
Task 3: ERROR task 3 failed
Task 4: 16

    Safe Worker Wrapper

    A reusable wrapper that guarantees the promise is always fulfilled, either with a value or an exception. 

    template <typename F, typename... Args>
void safe_invoke(std::promise<std::invoke_result_t<F, Args...>>& prom,
                 F&& f, Args&&... args) {
    using R = std::invoke_result_t<F, Args...>;
    try {
        if constexpr (std::is_void_v<R>) {
            std::invoke(std::forward<F>(f),
                        std::forward<Args>(args)...);
            prom.set_value();
        } else {
            prom.set_value(
                std::invoke(std::forward<F>(f),
                            std::forward<Args>(args)...));
        }
    } catch (...) {
        prom.set_exception(std::current_exception());
    }
}

// Usage
std::promise<int> prom;
auto fut = prom.get_future();

std::thread t([&prom]{
    safe_invoke(prom, [](int a, int b){ return a / b; }, 10, 0);
});

try {
    std::cout << fut.get() << "\n";
} catch (const std::exception& e) {
    std::cout << "Caught: " << e.what() << "\n";
}
t.join();

    Pattern: First Successful Result

    Launch multiple workers. Return the first successful result and discard failures. 

    #include <future>
#include <vector>
#include <optional>
#include <iostream>
#include <stdexcept>
#include <chrono>
#include <thread>

int main() {
    std::vector<std::future<int>> futures;

    // Some tasks succeed, some fail
    for (int i = 0; i < 5; ++i) {
        futures.push_back(std::async(std::launch::async, [i]{
            std::this_thread::sleep_for(
                std::chrono::milliseconds(i * 50));
            if (i % 2 == 0) throw std::runtime_error("fail");
            return i * 100;
        }));
    }

    std::optional<int> first_result;
    for (auto& f : futures) {
        try {
            int val = f.get();
            if (!first_result) {
                first_result = val;
            }
        } catch (...) {
            // Ignore failures
        }
    }

    if (first_result) {
        std::cout << "First success: " << *first_result << "\n";
    } else {
        std::cout << "All tasks failed\n";
    }
}

    Error Handling Strategies

    Strategy When to use
    catch-all + set_exception Default for raw promise-based code
    std::async / packaged_task Automatic capture; prefer when possible
    Nested exceptions Multi-layer systems needing full error chains
    Error codes via std::expected (C++23) High-performance paths where exceptions are too slow
    Selective rethrow When some errors are recoverable locally
    Golden rule. Never let an exception escape a thread function. Either catch it and forward it through a promise/future, or use std::async/std::packaged_task which handle this automatically. An uncaught exception in a thread calls std::terminate.

    Broken Promise as Exception

    If a promise is destroyed without calling set_value or set_exception, the runtime synthesizes a std::future_error with code future_errc::broken_promise. This is the last-resort safety net. 

    #include <future>
#include <iostream>

int main() {
    std::future<int> fut;
    {
        std::promise<int> p;
        fut = p.get_future();
        // p destroyed without fulfillment
    }

    try {
        fut.get();
    } catch (const std::future_error& e) {
        std::cout << "Code: " << e.code().message() << "\n";
        // "broken promise"
    }
}

    Key Takeaways

    • std::exception_ptr is the vehicle for transporting exceptions across threads.
    • Use std::current_exception() inside a catch block to capture the active exception.
    • Use std::make_exception_ptr to create an exception_ptr without throwing.
    • promise::set_exception stores the exception; future::get() re-throws it.
    • std::async and packaged_task capture exceptions automatically.
    • Use nested exceptions (throw_with_nested) to preserve error context across layers.
    • Never let exceptions escape a thread function; always forward or handle them.
