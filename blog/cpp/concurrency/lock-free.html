    Lock-Free Programming

    March 2026 · 32 min read

    What Does "Lock-Free" Mean?

    A data structure is lock-free if at least one thread can always make progress, regardless of what other threads are doing (even if they are suspended or crashed). This is stronger than "does not use mutexes." A compare-and-swap retry loop is lock-free because even if one thread's CAS fails (another thread succeeded), the system as a whole makes progress.

    The progression of guarantees:

    • Blocking (mutex-based): A thread holding a lock can prevent all others from progressing.
    • Lock-free: At least one thread makes progress. Individual threads may starve, but the system never deadlocks.
    • Wait-free: Every thread makes progress in a bounded number of steps. The strongest guarantee, and the hardest to achieve.

    Lock-Free Stack

    The simplest lock-free data structure. The core idea: use CAS on the head pointer.

    template<typename T>
class LockFreeStack {
    struct Node {
        T data;
        Node* next;
        Node(T val) : data(std::move(val)), next(nullptr) {}
    };

    std::atomic<Node*> head_{nullptr};

public:
    void push(T val) {
        auto* node = new Node(std::move(val));
        node->next = head_.load(std::memory_order_relaxed);
        // CAS loop: set head to new node if it hasn't changed
        while (!head_.compare_exchange_weak(
            node->next, node,
            std::memory_order_release,
            std::memory_order_relaxed)) {
            // node->next is updated to current head on failure
        }
    }

    bool pop(T& result) {
        Node* old_head = head_.load(std::memory_order_acquire);
        while (old_head &&
               !head_.compare_exchange_weak(
                   old_head, old_head->next,
                   std::memory_order_acq_rel,
                   std::memory_order_acquire)) {
            // old_head updated to current head on failure
        }
        if (!old_head) return false;
        result = std::move(old_head->data);
        // NOTE: When do we delete old_head? This is the hard part!
        delete old_head;  // DANGEROUS: another thread might be reading it
        return true;
    }
};

    The ABA Problem

    The ABA problem is the most notorious pitfall in lock-free programming. Thread 1 reads head = A, gets preempted. Thread 2 pops A, pops B, pushes A back (possibly a different allocation at the same address). Thread 1 resumes, sees head is still A, and CAS succeeds. But the stack state has completely changed: B is gone, and A's next pointer might be stale.

    // The sequence:
// Stack: A → B → C
// Thread 1: old_head = A (preempted)
// Thread 2: pop A, pop B, push A back
// Stack: A → C (B is gone!)
// Thread 1: CAS(head, A, A->next=B)  ← B is GONE! Corrupted!
    ABA is not a theoretical concern. It causes real crashes in production systems. Solutions include tagged pointers (version counters), hazard pointers, and epoch-based reclamation.

    Solution: Tagged Pointers

    Add a version counter to the pointer. CAS now checks both the pointer and the counter. Even if the pointer value is the same (ABA), the counter will differ:

    struct TaggedPtr {
    Node* ptr;
    uintptr_t tag;  // version counter
};

std::atomic<TaggedPtr> head_{nullptr, 0};

// Push: increment tag with every modification
TaggedPtr old = head_.load();
TaggedPtr new_val = {new_node, old.tag + 1};
while (!head_.compare_exchange_weak(old, new_val)) {
    new_node->next = old.ptr;
    new_val = {new_node, old.tag + 1};
}

    Solution: Hazard Pointers

    Each thread publishes the pointers it is currently using in a shared hazard pointer list. Before freeing a node, check if any thread has published it as hazardous. If so, defer the free. This is a general reclamation strategy for lock-free data structures.

    Solution: Epoch-Based Reclamation

    Maintain a global epoch counter. Each thread declares which epoch it is in. Nodes removed during epoch e can safely be freed once all threads have advanced past epoch e. Simpler than hazard pointers, used in many production systems (RCU in Linux kernel is a variant).

    Lock-Free Queue (Michael-Scott)

    The Michael-Scott queue uses two atomic pointers (head and tail) and a sentinel node. Enqueue CAS-es on the tail, dequeue CAS-es on the head:

    template<typename T>
class MichaelScottQueue {
    struct Node {
        T data;
        std::atomic<Node*> next{nullptr};
    };

    std::atomic<Node*> head_;
    std::atomic<Node*> tail_;

public:
    MichaelScottQueue() {
        auto* sentinel = new Node{T{}, nullptr};
        head_.store(sentinel);
        tail_.store(sentinel);
    }

    void enqueue(T val) {
        auto* node = new Node{std::move(val), nullptr};
        while (true) {
            Node* tail = tail_.load(std::memory_order_acquire);
            Node* next = tail->next.load(std::memory_order_acquire);
            if (next == nullptr) {
                if (tail->next.compare_exchange_weak(
                        next, node, std::memory_order_release)) {
                    tail_.compare_exchange_strong(
                        tail, node, std::memory_order_release);
                    return;
                }
            } else {
                // Tail is lagging, help advance it
                tail_.compare_exchange_weak(
                    tail, next, std::memory_order_release);
            }
        }
    }

    bool dequeue(T& result) {
        while (true) {
            Node* head = head_.load(std::memory_order_acquire);
            Node* tail = tail_.load(std::memory_order_acquire);
            Node* next = head->next.load(std::memory_order_acquire);
            if (head == tail) {
                if (!next) return false;  // empty queue
                tail_.compare_exchange_weak(
                    tail, next, std::memory_order_release);
            } else {
                result = next->data;
                if (head_.compare_exchange_weak(
                        head, next, std::memory_order_acq_rel)) {
                    delete head;  // reclaim sentinel
                    return true;
                }
            }
        }
    }
};

    When to Use Lock-Free

    • Use lock-free when: You need guaranteed progress (no deadlocks), real-time latency requirements, or extremely high throughput on simple data structures.
    • Avoid lock-free when: A mutex-based solution is simpler and performs well enough. Lock-free code is hard to write, hard to verify, and hard to debug. The complexity is rarely worth it.
    Herb Sutter's advice: "Lock-free programming is like juggling razor blades." Only use it when the performance benefit is measured and significant.

    Summary

    • Lock-free guarantees system-wide progress. Wait-free guarantees per-thread progress.
    • CAS (compare-and-swap) is the fundamental building block.
    • The ABA problem can corrupt lock-free data structures. Use tagged pointers, hazard pointers, or epoch-based reclamation.
    • Lock-free stacks and queues are well-studied. Implementing them correctly requires extreme care.
    • Prefer mutex-based designs unless you have measured a need for lock-free performance.
