    Actor Model

    March 2026

    What is the Actor Model?

    The actor model is a concurrency paradigm where computation is organized around actors: independent entities that communicate exclusively through message passing. Each actor has its own private state, a mailbox (message queue), and a processing loop. There is no shared mutable state between actors, which eliminates data races by design.

    Three Axioms. Upon receiving a message, an actor can: (1) send messages to other actors, (2) create new actors, and (3) update its own private state. Nothing else. This simplicity is the source of the model's power.

    Actor Model vs Shared-State Concurrency

    Aspect Shared State Actor Model
    Communication Shared memory + locks Message passing
    Data Races Possible; prevented by discipline Impossible by design
    Deadlocks Risk with multiple locks No locks; but message cycles possible
    Scalability Limited by lock contention Scales across cores and machines
    Complexity Manual synchronization Design message protocols

    Implementing a Simple Actor in C++

    An actor needs two components: a mailbox (thread-safe queue) and a processing thread that reads from the mailbox and handles each message. Here is a minimal actor framework. 

    #include <condition_variable>
#include <functional>
#include <iostream>
#include <mutex>
#include <queue>
#include <string>
#include <thread>

class Actor {
    std::queue<std::function<void()>> mailbox_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool stop_ = false;
    std::thread worker_;

    void run() {
        for (;;) {
            std::function<void()> msg;
            {
                std::unique_lock lock(mtx_);
                cv_.wait(lock, [&] { return stop_ || !mailbox_.empty(); });
                if (stop_ && mailbox_.empty()) return;
                msg = std::move(mailbox_.front());
                mailbox_.pop();
            }
            msg();  // Process message
        }
    }

public:
    Actor() : worker_(&Actor::run, this) {}

    // Send a message (callable) to this actor
    void send(std::function<void()> msg) {
        {
            std::lock_guard lock(mtx_);
            mailbox_.push(std::move(msg));
        }
        cv_.notify_one();
    }

    // Shut down the actor gracefully
    void stop() {
        {
            std::lock_guard lock(mtx_);
            stop_ = true;
        }
        cv_.notify_one();
        if (worker_.joinable()) worker_.join();
    }

    ~Actor() {
        stop();
    }
};

int main() {
    Actor printer;

    // Send messages to the actor
    for (int i = 0; i < 5; ++i) {
        printer.send([i] {
            std::cout << "Message #" << i << " processed\n";
        });
    }

    // Give time for processing, then stop
    printer.stop();
    return 0;
}
    No Shared State. The actor's internal state is only accessed by its own worker thread. External code interacts solely by sending messages. This is the fundamental guarantee that prevents data races.

    Typed Messages with std::variant

    Using std::function<void()> is flexible but loses type information. For richer actor systems, define message types as a std::variant and use std::visit to dispatch. 

    #include <condition_variable>
#include <iostream>
#include <mutex>
#include <queue>
#include <string>
#include <thread>
#include <variant>

// Define message types
struct Deposit  { double amount; };
struct Withdraw { double amount; };
struct GetBalance {};
struct PrintBalance {};
struct Stop {};

using BankMessage = std::variant<Deposit, Withdraw, PrintBalance, Stop>;

class BankAccount {
    std::queue<BankMessage> mailbox_;
    std::mutex mtx_;
    std::condition_variable cv_;
    std::thread worker_;
    double balance_ = 0.0;

    void run() {
        bool running = true;
        while (running) {
            BankMessage msg;
            {
                std::unique_lock lock(mtx_);
                cv_.wait(lock, [&] { return !mailbox_.empty(); });
                msg = std::move(mailbox_.front());
                mailbox_.pop();
            }
            std::visit([&](auto&& m) {
                using T = std::decay_t<decltype(m)>;
                if constexpr (std::is_same_v<T, Deposit>) {
                    balance_ += m.amount;
                    std::cout << "Deposited " << m.amount
                              << ", balance: " << balance_ << "\n";
                } else if constexpr (std::is_same_v<T, Withdraw>) {
                    if (m.amount <= balance_) {
                        balance_ -= m.amount;
                        std::cout << "Withdrew " << m.amount
                                  << ", balance: " << balance_ << "\n";
                    } else {
                        std::cout << "Insufficient funds for " << m.amount << "\n";
                    }
                } else if constexpr (std::is_same_v<T, PrintBalance>) {
                    std::cout << "Current balance: " << balance_ << "\n";
                } else if constexpr (std::is_same_v<T, Stop>) {
                    running = false;
                }
            }, msg);
        }
    }

public:
    BankAccount() : worker_(&BankAccount::run, this) {}

    void send(BankMessage msg) {
        {
            std::lock_guard lock(mtx_);
            mailbox_.push(std::move(msg));
        }
        cv_.notify_one();
    }

    ~BankAccount() {
        send(Stop{});
        if (worker_.joinable()) worker_.join();
    }
};

int main() {
    BankAccount account;
    account.send(Deposit{1000.0});
    account.send(Withdraw{250.0});
    account.send(Deposit{50.0});
    account.send(Withdraw{900.0});  // Insufficient
    account.send(PrintBalance{});
    account.send(Stop{});
    return 0;
}

    Actors Communicating with Each Other

    Actors become powerful when they form networks. One actor can hold a reference to another and forward or delegate work. Here is a logger actor and a worker actor that sends log messages. 

    #include <condition_variable>
#include <functional>
#include <iostream>
#include <mutex>
#include <queue>
#include <sstream>
#include <string>
#include <thread>

// Reuse the Actor class from above

class Logger {
    Actor actor_;
public:
    void log(const std::string& msg) {
        actor_.send([msg] {
            std::cout << "[LOG] " << msg << "\n";
        });
    }
    void stop() { actor_.stop(); }
};

class Worker {
    Actor actor_;
    Logger& logger_;
    int id_;
public:
    Worker(int id, Logger& logger) : logger_(logger), id_(id) {}

    void process(int value) {
        actor_.send([this, value] {
            int result = value * value;
            std::ostringstream oss;
            oss << "Worker " << id_ << " computed "
                << value << "^2 = " << result;
            logger_.log(oss.str());
        });
    }

    void stop() { actor_.stop(); }
};

int main() {
    Logger logger;
    Worker w1(1, logger);
    Worker w2(2, logger);

    w1.process(5);
    w2.process(7);
    w1.process(3);
    w2.process(9);

    w1.stop();
    w2.stop();
    logger.stop();
    return 0;
}

    A Generic Typed Actor

    We can generalize the pattern into a reusable template that accepts any message type and a handler function. 

    #include <condition_variable>
#include <mutex>
#include <queue>
#include <thread>
#include <optional>

template <typename Message>
class TypedActor {
    std::queue<std::optional<Message>> mailbox_;
    std::mutex mtx_;
    std::condition_variable cv_;
    std::thread worker_;

public:
    // handler is called for each message on the actor's thread
    template <typename Handler>
    explicit TypedActor(Handler handler) {
        worker_ = std::thread([this, h = std::move(handler)]() mutable {
            for (;;) {
                std::optional<Message> msg;
                {
                    std::unique_lock lock(mtx_);
                    cv_.wait(lock, [&] { return !mailbox_.empty(); });
                    msg = std::move(mailbox_.front());
                    mailbox_.pop();
                }
                if (!msg) return;  // Stop sentinel
                h(*msg);
            }
        });
    }

    void send(Message msg) {
        {
            std::lock_guard lock(mtx_);
            mailbox_.push(std::move(msg));
        }
        cv_.notify_one();
    }

    void stop() {
        {
            std::lock_guard lock(mtx_);
            mailbox_.push(std::nullopt);
        }
        cv_.notify_one();
        if (worker_.joinable()) worker_.join();
    }

    ~TypedActor() { stop(); }
};

// Usage
#include <iostream>

int main() {
    TypedActor<int> squarer([](int x) {
        std::cout << x << " squared = " << x * x << "\n";
    });

    squarer.send(3);
    squarer.send(7);
    squarer.send(12);
    squarer.stop();
    return 0;
}

    When Actors Shine

    The actor model is not always the best choice for every concurrency problem. It excels in specific scenarios:

    • Distributed systems: Actors map naturally to processes on different machines. Message passing over a network is the only option anyway, so the actor model matches the physical reality.
    • High isolation requirements: When components must not share state (for fault tolerance or security), actors enforce this structurally.
    • Event-driven architectures: Each actor reacts to incoming events independently, making the system naturally responsive.
    • Complex state machines: An actor can represent a stateful entity (user session, device connection) where its behavior depends on its current state and the messages it receives.
    Trade-offs. The actor model introduces overhead from message serialization and queuing. For tight, low-latency inner loops on shared data (like parallel sorting), shared-state concurrency with careful locking or lock-free techniques will outperform actors. Choose actors for coarse-grained, high-level coordination; choose shared state for fine-grained, data-parallel work.

    Comparison: Actor vs Mutex for a Counter

    To illustrate the difference concretely, here is the same concurrent counter implemented both ways. 

    #include <atomic>
#include <functional>
#include <iostream>
#include <mutex>
#include <thread>
#include <vector>

// Shared-state approach: mutex-protected counter
class MutexCounter {
    int count_ = 0;
    std::mutex mtx_;
public:
    void increment() {
        std::lock_guard lock(mtx_);
        ++count_;
    }
    int get() const { return count_; }
};

// Actor approach: counter as an actor
// (Using Actor class from above)
class ActorCounter {
    Actor actor_;
    int count_ = 0;
    std::atomic<int> snapshot_{0};
public:
    void increment() {
        actor_.send([this] {
            ++count_;
            snapshot_.store(count_, std::memory_order_release);
        });
    }
    int get() const {
        return snapshot_.load(std::memory_order_acquire);
    }
    void stop() { actor_.stop(); }
};

int main() {
    // Mutex version
    MutexCounter mc;
    {
        std::vector<std::thread> threads;
        for (int i = 0; i < 4; ++i)
            threads.emplace_back([&] {
                for (int j = 0; j < 10000; ++j) mc.increment();
            });
        for (auto& t : threads) t.join();
    }
    std::cout << "Mutex counter: " << mc.get() << "\n";  // 40000

    // Actor version
    ActorCounter ac;
    {
        std::vector<std::thread> threads;
        for (int i = 0; i < 4; ++i)
            threads.emplace_back([&] {
                for (int j = 0; j < 10000; ++j) ac.increment();
            });
        for (auto& t : threads) t.join();
    }
    ac.stop();
    std::cout << "Actor counter: " << ac.get() << "\n";  // 40000
    return 0;
}

    Key Takeaways

    • Actors communicate only through message passing; no shared mutable state exists.
    • Each actor has a mailbox (queue) and a single processing thread.
    • std::variant and std::visit provide type-safe message dispatch.
    • Actors excel in distributed systems, event-driven designs, and fault-tolerant architectures.
    • For fine-grained data parallelism, shared-state approaches typically outperform actors.
    • The generic TypedActor<Message> template provides a reusable building block.
