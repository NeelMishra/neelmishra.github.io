    Pipeline Parallelism

    March 2026

    What is Pipeline Parallelism?

    In a pipeline, work is divided into stages. Each stage runs on its own thread, and stages are connected by queues. As one item finishes a stage, it moves to the next stage while the first stage immediately begins processing the next item. This is analogous to an assembly line in a factory.

    Key Insight. Pipeline parallelism does not make a single item faster (latency stays the same or increases slightly). Instead, it increases throughput by keeping all stages busy simultaneously on different items.

    Throughput vs Latency

    Consider three stages, each taking 10ms. Processing one item sequentially takes 30ms. With a pipeline, the first result still arrives after 30ms (latency), but subsequent results arrive every 10ms (throughput equals the slowest stage). For N items, total time drops from 30N ms to approximately 30 + 10(N-1) ms.

    Thread-Safe Queue

    The backbone of a pipeline is a bounded, thread-safe queue connecting stages. Here is a minimal implementation using a mutex and condition variables. A sentinel value (or std::optional) signals the end of the stream. 

    #include <condition_variable>
#include <mutex>
#include <optional>
#include <queue>

template <typename T>
class BoundedQueue {
    std::queue<std::optional<T>> queue_;
    std::mutex mtx_;
    std::condition_variable not_full_;
    std::condition_variable not_empty_;
    size_t capacity_;

public:
    explicit BoundedQueue(size_t cap) : capacity_(cap) {}

    // Push a value. Blocks if the queue is full.
    void push(T value) {
        std::unique_lock lock(mtx_);
        not_full_.wait(lock, [&] { return queue_.size() < capacity_; });
        queue_.push(std::move(value));
        not_empty_.notify_one();
    }

    // Signal that no more items will be pushed.
    void close() {
        std::unique_lock lock(mtx_);
        not_full_.wait(lock, [&] { return queue_.size() < capacity_; });
        queue_.push(std::nullopt);
        not_empty_.notify_one();
    }

    // Pop a value. Returns nullopt when the stream is closed.
    std::optional<T> pop() {
        std::unique_lock lock(mtx_);
        not_empty_.wait(lock, [&] { return !queue_.empty(); });
        auto val = std::move(queue_.front());
        queue_.pop();
        if (!val) {
            // Re-post sentinel so other consumers also see it
            queue_.push(std::nullopt);
            not_empty_.notify_all();
        }
        not_full_.notify_one();
        return val;
    }
};

    Three-Stage Pipeline (Read, Process, Write)

    The following example reads lines from a source, transforms them (uppercasing), and writes the results. Each stage runs on its own thread with bounded queues between them. 

    #include <algorithm>
#include <cctype>
#include <iostream>
#include <string>
#include <thread>
#include <vector>

// BoundedQueue<T> from above

void read_stage(const std::vector<std::string>& input,
                BoundedQueue<std::string>& out) {
    for (const auto& line : input) {
        out.push(line);
    }
    out.close();
}

void process_stage(BoundedQueue<std::string>& in,
                   BoundedQueue<std::string>& out) {
    while (auto item = in.pop()) {
        std::string s = std::move(*item);
        // Transform: convert to uppercase
        std::transform(s.begin(), s.end(), s.begin(),
                       [](unsigned char c) { return std::toupper(c); });
        out.push(std::move(s));
    }
    out.close();
}

void write_stage(BoundedQueue<std::string>& in,
                 std::vector<std::string>& results) {
    while (auto item = in.pop()) {
        results.push_back(std::move(*item));
    }
}

int main() {
    std::vector<std::string> input = {
        "hello world", "pipeline parallelism",
        "stage one reads", "stage two processes",
        "stage three writes"
    };

    BoundedQueue<std::string> q1(4);  // Read -> Process
    BoundedQueue<std::string> q2(4);  // Process -> Write
    std::vector<std::string> results;

    std::thread t1(read_stage, std::cref(input), std::ref(q1));
    std::thread t2(process_stage, std::ref(q1), std::ref(q2));
    std::thread t3(write_stage, std::ref(q2), std::ref(results));

    t1.join();
    t2.join();
    t3.join();

    for (const auto& r : results) {
        std::cout << r << "\n";
    }
    // HELLO WORLD
    // PIPELINE PARALLELISM
    // STAGE ONE READS
    // STAGE TWO PROCESSES
    // STAGE THREE WRITES
    return 0;
}

    Balancing Pipeline Stages

    The throughput of a pipeline is limited by its slowest stage (the bottleneck). If one stage takes 50ms and the others take 10ms, the pipeline produces one result every 50ms regardless. To address this, you can replicate the slow stage across multiple threads.

    Balancing Strategy. If stage K is 5x slower than other stages, run 5 copies of stage K consuming from the same input queue and feeding to the same output queue. This approach equalizes throughput across stages. 
    // Replicated process stage for balancing
void run_pipeline_balanced(const std::vector<std::string>& input,
                           std::vector<std::string>& results,
                           int process_workers) {
    BoundedQueue<std::string> q1(16);
    BoundedQueue<std::string> q2(16);

    std::thread reader(read_stage, std::cref(input), std::ref(q1));

    // Launch multiple process workers
    std::vector<std::thread> processors;
    processors.reserve(process_workers);
    for (int i = 0; i < process_workers; ++i) {
        processors.emplace_back(process_stage, std::ref(q1), std::ref(q2));
    }

    std::thread writer(write_stage, std::ref(q2), std::ref(results));

    reader.join();
    for (auto& t : processors) t.join();
    q2.close();  // Signal writer that all processors are done
    writer.join();
}

    Pipeline with a Thread Pool

    For pipelines with many short-lived tasks, a thread pool avoids per-item thread creation overhead. Each stage submits work to the pool, and the pool schedules it across a fixed number of threads. 

    #include <condition_variable>
#include <functional>
#include <mutex>
#include <queue>
#include <thread>
#include <vector>

class ThreadPool {
    std::vector<std::thread> workers_;
    std::queue<std::function<void()>> tasks_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool stop_ = false;

public:
    explicit ThreadPool(size_t n) {
        for (size_t i = 0; i < n; ++i) {
            workers_.emplace_back([this] {
                for (;;) {
                    std::function<void()> task;
                    {
                        std::unique_lock lock(mtx_);
                        cv_.wait(lock, [&] { return stop_ || !tasks_.empty(); });
                        if (stop_ && tasks_.empty()) return;
                        task = std::move(tasks_.front());
                        tasks_.pop();
                    }
                    task();
                }
            });
        }
    }

    void submit(std::function<void()> task) {
        {
            std::lock_guard lock(mtx_);
            tasks_.push(std::move(task));
        }
        cv_.notify_one();
    }

    ~ThreadPool() {
        {
            std::lock_guard lock(mtx_);
            stop_ = true;
        }
        cv_.notify_all();
        for (auto& w : workers_) w.join();
    }
};

    When to Use Pipelines

    • Data flows through a fixed sequence of transformations.
    • Individual stages have different resource needs (I/O vs CPU).
    • High throughput matters more than low latency for a single item.
    • Streaming workloads where data arrives continuously.

    Key Takeaways

    • Pipeline parallelism improves throughput, not single-item latency.
    • Throughput is limited by the slowest stage; replicate it to balance.
    • Bounded queues provide backpressure and limit memory usage.
    • A thread pool can replace per-stage threads for fine-grained work.
    • Pipelines excel at streaming and I/O-interleaved workloads.
