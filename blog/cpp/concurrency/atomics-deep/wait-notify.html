    C++20 Atomic Wait and Notify

    March 2026

    Before C++20, the only standard way to block a thread until some condition changed was std::condition_variable, which requires a mutex. C++20 introduced wait(), notify_one(), and notify_all() directly on atomic types, giving us lightweight, mutex-free blocking that maps closely to the operating system's futex (Linux), WaitOnAddress (Windows), or ulock (macOS) primitives.

    API Overview

    // Available on all std::atomic<T> and std::atomic_flag (C++20)

void wait(T old, std::memory_order order = std::memory_order_seq_cst) const;
void notify_one();
void notify_all();

// For atomic_flag:
void wait(bool old, std::memory_order order = std::memory_order_seq_cst) const;
void notify_one();
void notify_all();

    The semantics of wait(old):

    1. Atomically read the current value.
    2. If the current value equals old, block the calling thread.
    3. When unblocked (by a notify or spuriously), re-read and repeat from step 1.
    4. Return when the value differs from old.
    Spurious wakeups. Like condition_variable::wait(), atomic::wait() may return spuriously. The implementation handles this internally by re-checking the value, but you should still structure code to tolerate it. The standard guarantees that wait() returns only when the value has actually changed from old, so in practice, spurious wakeups are invisible to the caller.

    Basic Usage

    #include <atomic>
#include <thread>
#include <iostream>

std::atomic<int> data{0};

void consumer() {
    // Block until data is no longer 0
    data.wait(0, std::memory_order_acquire);
    std::cout << "Got: " << data.load(std::memory_order_relaxed) << "\n";
}

void producer() {
    // Prepare the value
    data.store(42, std::memory_order_release);
    // Wake one waiting thread
    data.notify_one();
}

int main() {
    std::thread t1(consumer);
    std::thread t2(producer);
    t1.join();
    t2.join();
}

    Comparison to condition_variable

    The traditional approach requires a mutex, a condition variable, and a predicate:

    // Traditional: condition_variable
std::mutex mtx;
std::condition_variable cv;
bool ready = false;

void consumer_cv() {
    std::unique_lock lock(mtx);
    cv.wait(lock, [] { return ready; });
    // use shared data
}

void producer_cv() {
    {
        std::lock_guard lock(mtx);
        ready = true;
    }
    cv.notify_one();
}
    // C++20: atomic wait (no mutex needed)
std::atomic<bool> ready{false};

void consumer_atomic() {
    ready.wait(false, std::memory_order_acquire);
    // use shared data
}

void producer_atomic() {
    ready.store(true, std::memory_order_release);
    ready.notify_one();
}

    When to still use condition_variable. If you need timed waits (wait_for, wait_until), complex predicates that cannot be reduced to value equality, or need to wait on a combination of variables, condition_variable remains the right tool. Atomic wait excels for simple flag/state transitions.

    Futex-Like Semantics

    On Linux, atomic::wait() maps almost directly to the futex(FUTEX_WAIT) system call. The key property: the kernel atomically checks the value and suspends the thread if it matches, preventing the lost-wakeup race.

    // Conceptual implementation of atomic<int>::wait() on Linux:
void wait(int old, std::memory_order order) {
    while (load(order) == old) {
        // syscall: atomically check *addr == old, then sleep
        syscall(SYS_futex, &storage, FUTEX_WAIT, old, nullptr, nullptr, 0);
    }
}

// notify_one maps to:
void notify_one() {
    syscall(SYS_futex, &storage, FUTEX_WAKE, 1, nullptr, nullptr, 0);
}

// notify_all maps to:
void notify_all() {
    syscall(SYS_futex, &storage, FUTEX_WAKE, INT_MAX, nullptr, nullptr, 0);
}

    On Windows, the underlying primitive is WaitOnAddress / WakeByAddressSingle / WakeByAddressAll. On macOS, it uses __ulock_wait / __ulock_wake.

    // Windows conceptual mapping:
void wait(int old, std::memory_order order) {
    while (load(order) == old) {
        WaitOnAddress(&storage, &old, sizeof(int), INFINITE);
    }
}

void notify_one() {
    WakeByAddressSingle(&storage);
}

    atomic_flag as Event Signal

    In C++20, atomic_flag gained test(), wait(), notify_one(), and notify_all(). This makes it a perfect lightweight one-shot event.

    std::atomic_flag event = ATOMIC_FLAG_INIT;  // initially clear (false)

void waiter() {
    // Block until the flag is set
    event.wait(false, std::memory_order_acquire);
    std::cout << "Event received!\n";
}

void signaler() {
    // Do some work...
    event.test_and_set(std::memory_order_release);
    event.notify_all();
}

    Reusable Event Gate

    class Event {
    std::atomic_flag flag_ = ATOMIC_FLAG_INIT;
public:
    void signal() {
        flag_.test_and_set(std::memory_order_release);
        flag_.notify_all();
    }

    void wait() {
        flag_.wait(false, std::memory_order_acquire);
    }

    void reset() {
        flag_.clear(std::memory_order_relaxed);
    }

    bool is_set() const {
        return flag_.test(std::memory_order_acquire);
    }
};

    Practical Patterns

    Countdown Latch

    A latch that blocks until N tasks complete (similar to std::latch but built from raw atomics):

    class CountdownLatch {
    std::atomic<int> count_;
public:
    explicit CountdownLatch(int n) : count_(n) {}

    void count_down() {
        if (count_.fetch_sub(1, std::memory_order_acq_rel) == 1) {
            // We decremented to 0: wake all waiters
            count_.notify_all();
        }
    }

    void wait() {
        int val = count_.load(std::memory_order_acquire);
        while (val > 0) {
            count_.wait(val, std::memory_order_acquire);
            val = count_.load(std::memory_order_acquire);
        }
    }
};

// Usage
CountdownLatch latch(4);

void task() {
    // do work...
    latch.count_down();
}

int main() {
    std::vector<std::thread> workers;
    for (int i = 0; i < 4; i++)
        workers.emplace_back(task);
    latch.wait();  // blocks until all 4 tasks finish
    for (auto& w : workers) w.join();
}

    Barrier (Single Use)

    class SimpleBarrier {
    std::atomic<int> count_;
    const int total_;
public:
    explicit SimpleBarrier(int n) : count_(n), total_(n) {}

    void arrive_and_wait() {
        int prev = count_.fetch_sub(1, std::memory_order_acq_rel);
        if (prev == 1) {
            // Last to arrive: wake everyone
            count_.notify_all();
        } else {
            // Wait until count reaches 0
            int val = count_.load(std::memory_order_acquire);
            while (val > 0) {
                count_.wait(val, std::memory_order_acquire);
                val = count_.load(std::memory_order_acquire);
            }
        }
    }
};

    Producer-Consumer Channel

    template <typename T>
class AtomicChannel {
    std::atomic<int> state_{0};  // 0 = empty, 1 = full
    T value_;
public:
    void send(T val) {
        value_ = std::move(val);
        state_.store(1, std::memory_order_release);
        state_.notify_one();
    }

    T receive() {
        state_.wait(0, std::memory_order_acquire);
        T result = std::move(value_);
        state_.store(0, std::memory_order_release);
        state_.notify_one();
        return result;
    }
};

// Usage
AtomicChannel<std::string> chan;

void sender() {
    chan.send("hello from producer");
}

void receiver() {
    std::string msg = chan.receive();
    std::cout << msg << "\n";
}

    Spin-Then-Block Hybrid

    For latency-sensitive code, spin briefly before falling back to the kernel wait:

    void smart_wait(std::atomic<int>& atom, int old) {
    // Phase 1: spin for a short time (low latency path)
    for (int i = 0; i < 1000; i++) {
        if (atom.load(std::memory_order_acquire) != old) return;
        #if defined(__x86_64__) || defined(_M_X64)
        _mm_pause();
        #elif defined(__aarch64__)
        __asm__ volatile("yield");
        #endif
    }

    // Phase 2: fall back to kernel wait (saves CPU)
    atom.wait(old, std::memory_order_acquire);
}

    Ordering Subtleties with Notify

    The notify_one() and notify_all() calls do not carry memory ordering themselves. The ordering comes from the store that precedes the notify and the load inside wait().

    int shared_data = 0;
std::atomic<bool> ready{false};

void producer() {
    shared_data = 42;                              // (1) non-atomic write
    ready.store(true, std::memory_order_release);  // (2) release store
    ready.notify_one();                            // (3) no ordering
}

void consumer() {
    ready.wait(false, std::memory_order_acquire);  // (4) acquire load
    assert(shared_data == 42);                     // (5) guaranteed by (2)-(4)
}
    Notify before store is a bug. If you call notify_one() before the store, the waiting thread may wake up, see the old value, and go back to sleep indefinitely. Always store first, then notify.

    When Atomic Wait Wins

    // Scenario: multiple threads wait on a state transition
enum class Phase : int { Init = 0, Computing = 1, Done = 2 };
std::atomic<Phase> phase{Phase::Init};

// With condition_variable you'd need:
//   mutex + cv + shared enum + predicate lambda
//   lock/unlock on every check

// With atomic wait:
void wait_for_done() {
    Phase current = phase.load(std::memory_order_acquire);
    while (current != Phase::Done) {
        phase.wait(current, std::memory_order_acquire);
        current = phase.load(std::memory_order_acquire);
    }
}

void advance_phase(Phase next) {
    phase.store(next, std::memory_order_release);
    phase.notify_all();
}

    Limitations

    • No timed wait. C++20 does not provide wait_for() or wait_until() on atomics. Use condition_variable if you need timeouts.
    • Equality-only predicate. wait(old) blocks while the value equals old. For complex predicates (e.g., "wait until value > 10"), you need a CAS loop with wait() or fall back to condition_variable.
    • No guaranteed FIFO. The standard does not specify which thread notify_one() wakes. The order is implementation-defined.
    // Workaround for "wait until value > threshold"
void wait_until_above(std::atomic<int>& atom, int threshold) {
    int val = atom.load(std::memory_order_acquire);
    while (val <= threshold) {
        atom.wait(val, std::memory_order_acquire);
        val = atom.load(std::memory_order_acquire);
    }
    // Now val > threshold
}

    Summary

    • atomic::wait(old) blocks until the value differs from old, using efficient OS primitives (futex, WaitOnAddress).
    • notify_one() wakes one waiter; notify_all() wakes all. Neither carries memory ordering.
    • The store-then-notify pattern is essential to avoid lost wakeups.
    • Atomic wait replaces condition_variable in simple flag/state/counter scenarios with less overhead and no mutex.
    • atomic_flag with wait/notify serves as a minimal event signal in C++20.
    • For timed waits or complex predicates, condition_variable is still the right tool.
    • The spin-then-block hybrid pattern combines low latency with CPU efficiency.
