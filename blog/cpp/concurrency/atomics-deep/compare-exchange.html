    Compare-Exchange Operations

    March 2026

    Compare-and-swap (CAS) is the cornerstone of lock-free programming. In C++, it appears as compare_exchange_weak and compare_exchange_strong. Every lock-free data structure, every optimistic update, and most spinlock implementations rely on this single primitive.

    CAS Semantics

    The compare-exchange operation performs the following atomically:

    1. Read the current value of the atomic variable.
    2. Compare it with an expected value.
    3. If they match, write the desired value and return true.
    4. If they do not match, load the actual current value into expected and return false.
    bool compare_exchange_strong(T& expected, T desired,
                             std::memory_order success,
                             std::memory_order failure);

bool compare_exchange_weak(T& expected, T desired,
                           std::memory_order success,
                           std::memory_order failure);
    The expected parameter is a reference. On failure, the runtime writes the actual current value into expected. This is critical: it means your CAS loop automatically has the fresh value for the next iteration without an extra load.

    compare_exchange_strong vs. compare_exchange_weak

    compare_exchange_strong returns false only when the current value genuinely differs from expected. compare_exchange_weak may also return false spuriously, even when the values match.

    Why does the weak form exist? On architectures that use load-linked/store-conditional (LL/SC) instructions (ARM, RISC-V, PowerPC), the SC can fail for reasons unrelated to the value, such as a cache line eviction or a context switch. Mapping CAS to LL/SC with guaranteed success requires a retry loop inside the implementation. The weak form avoids this internal loop, yielding better performance when the caller already has an outer loop.

    // strong: use when NOT in a loop (single attempt matters)
bool ok = val.compare_exchange_strong(expected, desired);

// weak: use INSIDE a loop (spurious failure is retried anyway)
while (!val.compare_exchange_weak(expected, desired)) {
    // expected is updated automatically on failure
    // recompute desired if needed
}

    Spurious failureNoYes

    The CAS Loop Pattern

    The canonical CAS loop reads the current value, computes a new value, then attempts to swap. If another thread changed the value in between, the loop retries with the updated value.

    std::atomic<int> counter{0};

void atomic_increment() {
    int expected = counter.load(std::memory_order_relaxed);
    while (!counter.compare_exchange_weak(
               expected,
               expected + 1,
               std::memory_order_relaxed,
               std::memory_order_relaxed)) {
        // expected is automatically updated to current value
        // loop retries with fresh expected + 1
    }
}

    Implementing Atomic Increment with CAS

    While fetch_add exists, implementing increment manually shows the pattern that generalises to any read-modify-write:

    template <typename T>
T atomic_add(std::atomic<T>& atom, T value) {
    T expected = atom.load(std::memory_order_relaxed);
    while (!atom.compare_exchange_weak(
               expected,
               expected + value,
               std::memory_order_acq_rel,
               std::memory_order_relaxed)) {
        // expected is updated automatically;
        // loop immediately re-tries with fresh value
    }
    return expected;  // previous value
}

    This pattern extends to any function of the current value:

    // Atomic maximum
void atomic_max(std::atomic<int>& atom, int val) {
    int expected = atom.load(std::memory_order_relaxed);
    while (expected < val &&
           !atom.compare_exchange_weak(
               expected, val,
               std::memory_order_relaxed)) {
        // if expected >= val, we stop: no update needed
    }
}

// Atomic multiply
void atomic_mul(std::atomic<int>& atom, int factor) {
    int expected = atom.load(std::memory_order_relaxed);
    while (!atom.compare_exchange_weak(
               expected,
               expected * factor,
               std::memory_order_relaxed)) {
        // retries until success
    }
}

    Implementing a Spinlock with CAS

    A spinlock is the simplest lock: spin in a loop until you acquire it. CAS makes this lock-free in the uncontended path.

    class Spinlock {
    std::atomic<bool> locked_{false};
public:
    void lock() {
        // Fast path: try to acquire immediately
        bool expected = false;
        if (locked_.compare_exchange_strong(
                expected, true,
                std::memory_order_acquire,
                std::memory_order_relaxed)) {
            return;
        }
        // Slow path: spin
        while (true) {
            // Spin on load (read-only, no bus traffic)
            while (locked_.load(std::memory_order_relaxed)) {
                // Optional: pause hint for hyper-threading
                #if defined(__x86_64__) || defined(_M_X64)
                _mm_pause();
                #endif
            }
            // Try to acquire
            expected = false;
            if (locked_.compare_exchange_weak(
                    expected, true,
                    std::memory_order_acquire,
                    std::memory_order_relaxed)) {
                return;
            }
        }
    }

    void unlock() {
        locked_.store(false, std::memory_order_release);
    }
};
    Test-and-test-and-set. The inner while (locked_.load(...)) loop reads without writing. This avoids cacheline bouncing between cores. Only when we see the lock is free do we attempt the CAS. This pattern is called TTAS (test-and-test-and-set).
    // Usage
Spinlock mtx;

void worker(int id) {
    mtx.lock();
    std::cout << "Thread " << id << " in critical section\n";
    mtx.unlock();
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 4; i++)
        threads.emplace_back(worker, i);
    for (auto& t : threads) t.join();
}

    Memory Orders in Compare-Exchange

    compare_exchange takes two memory orders: one for success, one for failure. The failure order cannot be stronger than the success order, and cannot be release or acq_rel.

    // Common combinations:
atom.compare_exchange_strong(exp, des,
    std::memory_order_acq_rel,    // success: acquire + release
    std::memory_order_acquire);   // failure: just acquire

atom.compare_exchange_weak(exp, des,
    std::memory_order_release,    // success: release
    std::memory_order_relaxed);   // failure: no ordering

atom.compare_exchange_strong(exp, des,
    std::memory_order_seq_cst,    // success: full fence
    std::memory_order_seq_cst);   // failure: full fence (default)

    If you specify only one memory order, it is used for both success and failure (but failure is capped at the acquire equivalent):

    // Single memory order form
atom.compare_exchange_weak(exp, des, std::memory_order_acq_rel);
// Equivalent to:
// success = acq_rel, failure = acquire

    Common CAS Patterns

    Lock-Free Stack Push

    struct Node {
    int data;
    Node* next;
};

std::atomic<Node*> head{nullptr};

void push(int value) {
    Node* new_node = new Node{value, nullptr};
    new_node->next = head.load(std::memory_order_relaxed);
    while (!head.compare_exchange_weak(
               new_node->next,    // expected = current head
               new_node,          // desired  = new node
               std::memory_order_release,
               std::memory_order_relaxed)) {
        // new_node->next is updated to current head on failure
    }
}

    Lock-Free Stack Pop

    Node* pop() {
    Node* old_head = head.load(std::memory_order_acquire);
    while (old_head &&
           !head.compare_exchange_weak(
               old_head,              // expected = current head
               old_head->next,        // desired = next node
               std::memory_order_acquire,
               std::memory_order_relaxed)) {
        // old_head is updated to current head on failure
    }
    return old_head;  // caller must handle deletion
}

    Conditional Update

    // Only update if current value satisfies a predicate
bool atomic_update_if(std::atomic<int>& atom,
                      std::function<bool(int)> predicate,
                      std::function<int(int)> transform) {
    int expected = atom.load(std::memory_order_acquire);
    do {
        if (!predicate(expected)) return false;
    } while (!atom.compare_exchange_weak(
                 expected,
                 transform(expected),
                 std::memory_order_acq_rel,
                 std::memory_order_acquire));
    return true;
}

// Usage: increment only if below threshold
std::atomic<int> permits{10};
bool acquired = atomic_update_if(
    permits,
    [](int v) { return v > 0; },
    [](int v) { return v - 1; });

    The ABA Problem

    CAS checks whether the value is the same as expected. But "the same value" does not mean "nothing changed." If the value goes from A to B and back to A between your load and your CAS, the CAS will succeed even though the state may have changed in ways that break your invariants.

    // Scenario: lock-free stack with ABA
// State: head -> [A] -> [B] -> [C]

// Thread 1: pop()
//   old_head = A
//   next = B
//   (preempted here)

// Thread 2: pop() -> removes A
//           pop() -> removes B
//           push(A) -> puts A back on stack
// State: head -> [A] -> [C]   (B is gone!)

// Thread 1 resumes:
//   CAS(head, A, B) -> succeeds! A == A
//   But B has been freed! head now points to freed memory.

    Common solutions to the ABA problem:

    // Solution 1: Tagged/versioned pointer
struct TaggedPtr {
    Node* ptr;
    uint64_t tag;
};

std::atomic<TaggedPtr> head;

void push(Node* node) {
    TaggedPtr old = head.load(std::memory_order_relaxed);
    TaggedPtr next;
    do {
        node->next = old.ptr;
        next = {node, old.tag + 1};  // increment tag
    } while (!head.compare_exchange_weak(
                 old, next,
                 std::memory_order_release,
                 std::memory_order_relaxed));
}

// Solution 2: Hazard pointers (conceptual)
// Each thread publishes which nodes it is currently accessing.
// Nodes are only reclaimed when no hazard pointer references them.

// Solution 3: Epoch-based reclamation
// Threads enter/exit epochs. Memory is freed only when
// all threads have crossed an epoch boundary.
    ABA in practice. The ABA problem mainly affects pointer-based CAS operations where nodes are recycled. For simple integer counters, ABA is harmless because every distinct value truly represents a distinct state. When working with lock-free data structures, always consider memory reclamation and ABA avoidance.

    Double-Width CAS

    Some platforms support cmpxchg16b (x86-64) or equivalent, allowing atomic CAS on 128-bit values. This is essential for tagged pointers and lock-free structures.

    // On x86-64 with -mcx16 flag:
struct alignas(16) DoubleWord {
    uint64_t lo;
    uint64_t hi;
};

std::atomic<DoubleWord> dw;

static_assert(std::atomic<DoubleWord>::is_always_lock_free,
              "Need 128-bit CAS support");

DoubleWord expected = dw.load();
DoubleWord desired = {expected.lo + 1, expected.hi};
dw.compare_exchange_strong(expected, desired);

    Common Pitfalls

    Forgetting that expected is modified on failure

    // WRONG: expected is never updated in the loop condition
int expected = 0;
int desired = 1;
while (!atom.compare_exchange_weak(expected, desired)) {
    // expected is now the CURRENT value, not 0
    // desired should probably be recomputed!
    desired = expected + 1;  // add this line
}

    Using strong in a tight loop unnecessarily

    // Suboptimal on ARM/RISC-V:
while (!atom.compare_exchange_strong(exp, des)) { ... }

// Better: weak is sufficient in loops
while (!atom.compare_exchange_weak(exp, des)) { ... }

    Ignoring the return value

    // WRONG: ignoring return value
atom.compare_exchange_strong(expected, desired);
// Did it succeed? We don't know!

// CORRECT: always check the result
if (atom.compare_exchange_strong(expected, desired)) {
    // succeeded: old value was expected, now it's desired
} else {
    // failed: expected now holds the current value
}

    Summary

    • compare_exchange_strong fails only on genuine mismatch; use it for one-shot CAS.
    • compare_exchange_weak may fail spuriously; use it inside CAS loops for better performance on LL/SC platforms.
    • The expected parameter is updated on failure, giving you the current value for free.
    • Always recompute the desired value after a failed CAS if it depends on the current value.
    • The ABA problem affects pointer-based CAS; mitigate with tagged pointers, hazard pointers, or epoch-based reclamation.
    • CAS is the universal building block: any lock-free algorithm can be expressed as a CAS loop.
