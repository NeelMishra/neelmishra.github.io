    Fetch Operations and Atomic Read-Modify-Write

    March 2026

    Atomic fetch operations combine a read and a modification into a single indivisible step. They are the workhorse of concurrent counters, flags, bitmasks, and reference counting. Every fetch operation returns the previous value, which is what makes them so useful for coordination.

    Atomic Load and Store

    The simplest atomic operations. load() reads the value; store() writes it. Neither is a read-modify-write, but they form the foundation.

    std::atomic<int> flag{0};

// Store: write a new value
flag.store(1, std::memory_order_release);

// Load: read the current value
int val = flag.load(std::memory_order_acquire);

// Implicit conversions also work (use seq_cst by default)
flag = 42;       // equivalent to flag.store(42)
int x = flag;    // equivalent to flag.load()
    Implicit vs. explicit. The implicit conversion operators use memory_order_seq_cst. In performance-critical code, prefer explicit load() / store() with the weakest sufficient memory order.

    Exchange

    exchange() atomically replaces the value and returns the old one. It is the simplest read-modify-write operation.

    std::atomic<int> slot{0};

// Atomically set to 42, get previous value
int old = slot.exchange(42, std::memory_order_acq_rel);
// old == 0, slot == 42

// Common use: one-shot flag
std::atomic<bool> claimed{false};

bool try_claim() {
    // If exchange returns false, we were first
    return !claimed.exchange(true, std::memory_order_acquire);
}
    // Swap two values through an atomic intermediary
std::atomic<int> mailbox{0};

void producer(int value) {
    int prev = mailbox.exchange(value, std::memory_order_release);
    if (prev != 0) {
        // previous value was not consumed; handle overflow
    }
}

int consumer() {
    return mailbox.exchange(0, std::memory_order_acquire);
}

    fetch_add and fetch_sub

    Available for integral and pointer types. Atomically adds/subtracts and returns the previous value.

    std::atomic<int> counter{0};

// Increment and get the old value
int prev = counter.fetch_add(1, std::memory_order_relaxed);  // prev = 0
// counter is now 1

// Decrement
int prev2 = counter.fetch_sub(1, std::memory_order_relaxed); // prev2 = 1
// counter is now 0

// Compound operators (return the NEW value, unlike fetch_*)
int result = counter += 5;  // result = 5
int result2 = counter -= 3; // result2 = 2
    Return value difference. fetch_add(n) returns the old value. operator+=(n) returns the new value (old + n). This distinction matters when you need the previous state.

    Pointer Arithmetic

    int buffer[1024];
std::atomic<int*> write_pos{buffer};

// Advance pointer by 4 ints
int* old_pos = write_pos.fetch_add(4, std::memory_order_relaxed);
// old_pos == buffer
// write_pos == buffer + 4

// Each unit is sizeof(int), so fetch_add(4) advances by 4 * sizeof(int) bytes

    Bitwise Fetch Operations

    fetch_and, fetch_or, and fetch_xor are available for integral atomic types. They atomically apply the bitwise operation and return the previous value.

    std::atomic<uint32_t> flags{0};

// Set bit 3
uint32_t old = flags.fetch_or(1u << 3, std::memory_order_relaxed);
// old = 0b0000, flags = 0b1000

// Clear bit 3
old = flags.fetch_and(~(1u << 3), std::memory_order_relaxed);
// old = 0b1000, flags = 0b0000

// Toggle bit 5
old = flags.fetch_xor(1u << 5, std::memory_order_relaxed);
// old = 0b0000, flags = 0b100000

    Practical: Thread-Safe Bitfield

    class AtomicBitset {
    std::atomic<uint64_t> bits_{0};
public:
    void set(int pos) {
        bits_.fetch_or(1ULL << pos, std::memory_order_relaxed);
    }

    void clear(int pos) {
        bits_.fetch_and(~(1ULL << pos), std::memory_order_relaxed);
    }

    void toggle(int pos) {
        bits_.fetch_xor(1ULL << pos, std::memory_order_relaxed);
    }

    bool test(int pos) const {
        return (bits_.load(std::memory_order_relaxed) >> pos) & 1;
    }

    // Set a bit and return whether it was already set
    bool test_and_set(int pos) {
        uint64_t prev = bits_.fetch_or(1ULL << pos,
                                        std::memory_order_acq_rel);
        return (prev >> pos) & 1;
    }
};

    Practical: Reference Counting

    Reference counting is the most common use of fetch_add / fetch_sub in production code. The pattern matches what std::shared_ptr does internally.

    class RefCounted {
    mutable std::atomic<int> ref_count_{1};

public:
    void add_ref() const {
        ref_count_.fetch_add(1, std::memory_order_relaxed);
    }

    void release() const {
        // fetch_sub returns the PREVIOUS value
        if (ref_count_.fetch_sub(1, std::memory_order_acq_rel) == 1) {
            // We decremented from 1 to 0: last reference
            delete this;
        }
    }

    int use_count() const {
        return ref_count_.load(std::memory_order_relaxed);
    }
};
    Why acq_rel on release? The release in fetch_sub ensures all writes made through this reference are visible before the object is destroyed. The acquire on the decrementing thread ensures it sees all prior writes before calling delete. Using relaxed here would be a data race on destruction.
    // Intrusive reference-counted pointer
template <typename T>
class IntrusivePtr {
    T* ptr_;
public:
    explicit IntrusivePtr(T* p = nullptr) : ptr_(p) {
        if (ptr_) ptr_->add_ref();
    }

    ~IntrusivePtr() {
        if (ptr_) ptr_->release();
    }

    IntrusivePtr(const IntrusivePtr& other) : ptr_(other.ptr_) {
        if (ptr_) ptr_->add_ref();
    }

    IntrusivePtr& operator=(const IntrusivePtr& other) {
        if (this != &other) {
            if (ptr_) ptr_->release();
            ptr_ = other.ptr_;
            if (ptr_) ptr_->add_ref();
        }
        return *this;
    }

    T* get() const { return ptr_; }
    T& operator*() const { return *ptr_; }
    T* operator->() const { return ptr_; }
};

    Practical: Concurrent Counters and Statistics

    struct Stats {
    std::atomic<uint64_t> requests{0};
    std::atomic<uint64_t> errors{0};
    std::atomic<uint64_t> bytes_sent{0};
};

Stats global_stats;

void handle_request(size_t response_bytes, bool error) {
    global_stats.requests.fetch_add(1, std::memory_order_relaxed);
    if (error)
        global_stats.errors.fetch_add(1, std::memory_order_relaxed);
    else
        global_stats.bytes_sent.fetch_add(response_bytes,
                                           std::memory_order_relaxed);
}

void print_stats() {
    // relaxed is fine: we want approximate, not perfectly consistent
    std::cout << "Requests:   " << global_stats.requests.load(std::memory_order_relaxed) << "\n"
              << "Errors:     " << global_stats.errors.load(std::memory_order_relaxed) << "\n"
              << "Bytes sent: " << global_stats.bytes_sent.load(std::memory_order_relaxed) << "\n";
}

    Practical: Atomic Flags and State Machines

    enum class State : int { Idle = 0, Running = 1, Done = 2, Error = 3 };

std::atomic<State> state{State::Idle};

bool try_start() {
    State expected = State::Idle;
    return state.compare_exchange_strong(expected, State::Running,
                                          std::memory_order_acq_rel);
}

void mark_done() {
    state.store(State::Done, std::memory_order_release);
}

// Using fetch_or to set flags atomically
constexpr uint32_t FLAG_READABLE  = 1u << 0;
constexpr uint32_t FLAG_WRITABLE  = 1u << 1;
constexpr uint32_t FLAG_EXEC      = 1u << 2;

std::atomic<uint32_t> permissions{0};

void grant_read_write() {
    permissions.fetch_or(FLAG_READABLE | FLAG_WRITABLE,
                         std::memory_order_relaxed);
}

void revoke_write() {
    permissions.fetch_and(~FLAG_WRITABLE,
                          std::memory_order_relaxed);
}

    std::atomic_ref (C++20)

    std::atomic_ref<T> provides atomic access to a non-atomic object. It does not own the data; it wraps an existing variable for the duration of the reference. This is useful when you cannot change a type to std::atomic but need atomic access in specific contexts.

    struct Particle {
    float x, y, z;
    int hits;         // not atomic, but we want atomic access sometimes
};

Particle particles[1000];

void record_hit(int idx) {
    // Create an atomic_ref to the plain int
    std::atomic_ref<int> ref(particles[idx].hits);
    ref.fetch_add(1, std::memory_order_relaxed);
}

int get_hits(int idx) {
    std::atomic_ref<int> ref(particles[idx].hits);
    return ref.load(std::memory_order_relaxed);
}
    Alignment requirement. atomic_ref<T> may require stricter alignment than T alone. Check atomic_ref<T>::required_alignment. If the referenced object is not properly aligned, the behaviour is undefined.
    // Check alignment requirements
static_assert(alignof(int) >= std::atomic_ref<int>::required_alignment,
              "int alignment is sufficient");

// For larger types, you may need alignas
struct alignas(std::atomic_ref<double>::required_alignment) AlignedData {
    double value;
};

// atomic_ref is lock-free if the underlying type supports it
static_assert(std::atomic_ref<int>::is_always_lock_free);

// All access to the same object must go through atomic_ref
// once any thread uses atomic_ref on it. Mixing plain and
// atomic access is undefined behaviour.

    atomic_ref with Structs

    struct Coord {
    int x;
    int y;
};
static_assert(std::is_trivially_copyable_v<Coord>);

Coord positions[100];

void update_position(int idx, Coord new_pos) {
    std::atomic_ref<Coord> ref(positions[idx]);
    ref.store(new_pos, std::memory_order_release);
}

Coord read_position(int idx) {
    std::atomic_ref<Coord> ref(positions[idx]);
    return ref.load(std::memory_order_acquire);
}

    How RMW Maps to Hardware

    On x86-64, most atomic read-modify-write operations map to a single instruction with a LOCK prefix:

    ; fetch_add(1) on x86-64
lock xadd [rdi], eax    ; atomically: old = *rdi; *rdi += eax; eax = old

; fetch_or(mask)
lock or [rdi], eax      ; atomically: *rdi |= eax

; exchange(val)
xchg [rdi], eax         ; atomically swap (LOCK implicit for xchg)

; compare_exchange (CAS)
lock cmpxchg [rdi], ecx ; if (*rdi == eax) *rdi = ecx; else eax = *rdi

    On ARM64, the same operations use either LSE atomics (ARMv8.1+) or LL/SC sequences:

    ; fetch_add with LSE (ARMv8.1+)
ldaddal w1, w0, [x0]    ; atomically add and return old value

; fetch_add with LL/SC (older ARM)
.retry:
  ldaxr w2, [x0]        ; load-exclusive with acquire
  add   w3, w2, w1      ; compute new value
  stlxr w4, w3, [x0]    ; store-exclusive with release
  cbnz  w4, .retry      ; retry if store-exclusive failed

    Memory Order Quick Reference

    OrderGuaranteeTypical Use
    relaxedAtomicity onlyCounters, statistics
    acquireReads after this see prior releasesLoad from flag/lock
    releaseWrites before this visible to acquirersStore to flag/unlock
    acq_relBoth acquire and releaseRMW in lock-free structures
    seq_cstTotal order across all seq_cst opsDefault; simplest reasoning

    Summary

    • load() and store() are the basic building blocks; exchange() adds atomic swap.
    • fetch_add / fetch_sub return the previous value; compound operators return the new value.
    • fetch_and / fetch_or / fetch_xor enable lock-free bitmask manipulation.
    • atomic_ref (C++20) brings atomic access to non-atomic objects without changing their type.
    • Reference counting, concurrent statistics, and permission flags are the most common real-world applications.
    • Choose the weakest memory order that maintains correctness; relaxed is sufficient for most counters.
