    Atomic Types in C++

    March 2026

    An atomic operation is one that completes indivisibly with respect to every other thread in the program. No thread can observe an atomic operation half-done. The <atomic> header provides the types and functions that give you this guarantee without reaching for mutexes.

    What Makes an Operation Atomic

    At the hardware level, atomicity depends on alignment, operand size, and the instruction set. On x86-64, a naturally aligned 64-bit load or store is already atomic. But the C++ memory model does not assume any particular hardware. std::atomic<T> bridges this gap: the compiler emits whatever instructions (or locks) are needed so every read and write of the wrapped value is indivisible.

    int x = 0;               // plain int, no guarantees
std::atomic<int> y{0};   // every load/store is atomic

// Thread A
x = 42;   // data race if Thread B reads x concurrently
y.store(42, std::memory_order_relaxed);  // always safe

    The key properties of an atomic operation:

    • Indivisibility — no thread sees a partial write.
    • Ordering guarantees — controlled through std::memory_order.
    • No undefined behaviour — concurrent reads and writes to the same atomic are well-defined.

    std::atomic_flag

    std::atomic_flag is the most primitive atomic type. It is guaranteed to be lock-free on every platform. It supports exactly two operations: test_and_set() and clear(). Before C++20 it had no way to read its value without modifying it.

    std::atomic_flag flag = ATOMIC_FLAG_INIT;  // guaranteed clear

// Acquire the flag (sets it and returns previous value)
while (flag.test_and_set(std::memory_order_acquire)) {
    // spin
}

// Critical section ...

flag.clear(std::memory_order_release);  // release the flag
    C++20 additions. atomic_flag gained test() (read without modify), wait(), notify_one(), and notify_all(). It can now serve as a lightweight event signal.
    // C++20: read without side effect
std::atomic_flag ready = ATOMIC_FLAG_INIT;

// Producer
ready.test_and_set(std::memory_order_release);
ready.notify_one();

// Consumer
ready.wait(false, std::memory_order_acquire);  // blocks until flag is true

    std::atomic<T> Overview

    The class template std::atomic<T> wraps a value of type T and provides atomic load, store, exchange, and compare-exchange operations. The available member functions depend on T:

    CategoryOperationsAvailable For
    Coreload, store, exchange, compare_exchange_weak/strongAll T
    Arithmeticfetch_add, fetch_sub, operator++, operator--Integral and pointer types
    Bitwisefetch_and, fetch_or, fetch_xorIntegral types only
    Wait/Notify (C++20)wait, notify_one, notify_allAll T

    std::atomic<bool>

    Unlike atomic_flag, atomic<bool> supports load(), store(), exchange(), and compare-exchange. It is not guaranteed lock-free, although it is on every mainstream platform.

    std::atomic<bool> shutdown{false};

// Worker thread
void worker() {
    while (!shutdown.load(std::memory_order_acquire)) {
        do_work();
    }
}

// Main thread
shutdown.store(true, std::memory_order_release);
    // exchange returns the previous value
std::atomic<bool> locked{false};

bool was_locked = locked.exchange(true, std::memory_order_acquire);
if (!was_locked) {
    // we acquired the lock
}

    std::atomic<int> and Integral Specialisations

    Integral specialisations add arithmetic and bitwise fetch operations. Every fetch_* function returns the previous value.

    std::atomic<int> counter{0};

// Multiple threads can safely do this concurrently
int prev = counter.fetch_add(1, std::memory_order_relaxed);

// Shorthand operators also available
counter++;   // equivalent to fetch_add(1)
counter--;   // equivalent to fetch_sub(1)
counter += 5; // equivalent to fetch_add(5)

    Full list of integral specialisations provided by the standard:

    std::atomic<T*>

    Pointer specialisations support fetch_add and fetch_sub that operate in units of sizeof(T), just like pointer arithmetic.

    struct Node { int data; };

Node buffer[128];
std::atomic<Node*> cursor{buffer};

// Thread-safe bump allocator
Node* alloc() {
    return cursor.fetch_add(1, std::memory_order_relaxed);
}

// Move cursor back
void dealloc() {
    cursor.fetch_sub(1, std::memory_order_relaxed);
}
    Pointer arithmetic. fetch_add(1) on an atomic<Node*> advances the pointer by sizeof(Node) bytes, not by 1 byte. This matches regular C++ pointer behaviour.

    Lock-Free vs. Not Lock-Free

    An atomic type is lock-free when the compiler can implement all its operations using hardware atomic instructions alone, without falling back to a mutex. Use is_lock_free() to query this at runtime, or the constexpr is_always_lock_free to check at compile time.

    std::atomic<int> ai;
std::atomic<double> ad;

struct Big { char data[256]; };
std::atomic<Big> ab;

static_assert(std::atomic<int>::is_always_lock_free,
              "int should always be lock-free");

std::cout << "int   lock-free: " << ai.is_lock_free() << "\n";
std::cout << "double lock-free: " << ad.is_lock_free() << "\n";
std::cout << "Big   lock-free: " << ab.is_lock_free() << "\n";

    Typical results on x86-64:

    int   lock-free: 1
double lock-free: 1
Big   lock-free: 0     // falls back to internal mutex

    ▶ Lock-Free vs. Mutex-Backed Atomic

    Step through to see how a lock-free atomic uses a single hardware instruction, while a large atomic acquires an internal spinlock.

    Querying Lock-Free Status

    There are two ways to check:

    // Runtime query (may vary per object due to alignment)
std::atomic<double> d;
if (d.is_lock_free()) { /* ... */ }

// Compile-time constant (C++17)
if constexpr (std::atomic<double>::is_always_lock_free) {
    // guaranteed lock-free on this platform
}

// Macro form (C11 compatibility)
// ATOMIC_BOOL_LOCK_FREE    -- 0, 1, or 2
// ATOMIC_INT_LOCK_FREE     -- 0: never, 1: sometimes, 2: always
// ATOMIC_POINTER_LOCK_FREE
    is_always_lock_free. This is a static constexpr member. A value of true means every instance of that specialisation is lock-free, regardless of alignment or runtime conditions. Prefer this over is_lock_free() when possible.

    Atomic with User-Defined Types

    You can use std::atomic<T> with your own types, but T must be trivially copyable. This means no virtual functions, no user-defined copy constructors, and the type must be memcpy-safe.

    // This works: trivially copyable
struct Point {
    float x;
    float y;
};
static_assert(std::is_trivially_copyable_v<Point>);

std::atomic<Point> pos{{0.0f, 0.0f}};

// Read
Point p = pos.load(std::memory_order_acquire);

// Write
pos.store({1.0f, 2.0f}, std::memory_order_release);

// Compare-and-swap
Point expected{0.0f, 0.0f};
Point desired{3.0f, 4.0f};
pos.compare_exchange_strong(expected, desired);
    // This will NOT compile: not trivially copyable
struct Bad {
    std::string name;  // non-trivial copy constructor
    virtual void foo(); // virtual function
};
// std::atomic<Bad> ab;  // ERROR

    Whether a user-defined atomic is lock-free depends on its size:

    struct Small { uint32_t a; uint32_t b; };  // 8 bytes
struct Large { char data[128]; };           // 128 bytes

std::atomic<Small> as;
std::atomic<Large> al;

// On x86-64 with cmpxchg8b/cmpxchg16b:
std::cout << as.is_lock_free() << "\n";  // likely 1
std::cout << al.is_lock_free() << "\n";  // likely 0

    Alignment and Padding Considerations

    The compiler may add padding to ensure atomic objects are properly aligned for hardware atomic instructions. On some platforms, std::atomic<T> may have stricter alignment than T alone:

    struct Pair { int a; int b; };

std::cout << "sizeof(Pair):         " << sizeof(Pair) << "\n";
std::cout << "sizeof(atomic<Pair>): " << sizeof(std::atomic<Pair>) << "\n";
std::cout << "alignof(Pair):        " << alignof(Pair) << "\n";
std::cout << "alignof(atomic<Pair>):" << alignof(std::atomic<Pair>) << "\n";

    On x86-64 with GCC you might see:

    sizeof(Pair):          8
sizeof(atomic<Pair>):  8
alignof(Pair):         4
alignof(atomic<Pair>): 8   // stricter alignment for cmpxchg8b

    No Copy, No Move

    Atomic types are neither copyable nor movable. This is intentional: a copy would not be atomic, and the semantics would be ambiguous.

    std::atomic<int> a{42};
// std::atomic<int> b = a;     // ERROR: copy deleted
// std::atomic<int> c{std::move(a)}; // ERROR: move deleted

// Instead, load and construct:
std::atomic<int> b{a.load()};  // OK: load the value, init a new atomic

    Construction and Initialization

    // Value initialization
std::atomic<int> a{0};          // direct initialization
std::atomic<int> b(42);         // also fine

// Default construction: value is indeterminate (NOT zero)
std::atomic<int> c;             // uninitialized, reading is UB until stored
c.store(0);                     // now safe

// ATOMIC_VAR_INIT (deprecated in C++20, removed in C++26)
std::atomic<int> d = ATOMIC_VAR_INIT(10);

// C++20: constexpr default constructor value-initializes to zero
// std::atomic<int> e{};  // zero-initialized in C++20

    Free Function Alternatives

    Every member function has a corresponding free function that works with both std::atomic<T> objects and (in C) with _Atomic qualified types:

    std::atomic<int> val{0};

// Member function style
val.store(10);
int v = val.load();
int prev = val.exchange(20);

// Free function style (identical behaviour)
std::atomic_store(&val, 10);
int v2 = std::atomic_load(&val);
int prev2 = std::atomic_exchange(&val, 20);

// With explicit memory order
std::atomic_store_explicit(&val, 10, std::memory_order_release);
int v3 = std::atomic_load_explicit(&val, std::memory_order_acquire);

    Practical Guide: Choosing the Right Atomic Type

    // Boolean flag (shutdown, ready signal)
std::atomic<bool> shutdown{false};

// Counter (reference counting, statistics)
std::atomic<int64_t> ref_count{1};

// Pointer (lock-free stack/queue node pointers)
std::atomic<Node*> head{nullptr};

// Small struct (coordinates, tagged pointer)
struct TaggedPtr { void* ptr; uint32_t tag; };
std::atomic<TaggedPtr> tagged;  // likely lock-free if <= 16 bytes

// For simple flag signalling, prefer atomic_flag
std::atomic_flag spin = ATOMIC_FLAG_INIT;  // always lock-free
    Rule of thumb. Use the smallest type that holds your data. Smaller types are more likely to be lock-free and generate more efficient code. If you need just a boolean signal, atomic_flag is the lightest option.

    Summary

    • std::atomic_flag is the only type guaranteed lock-free on all platforms.
    • std::atomic<T> works for any trivially copyable T, but lock-free status depends on size and alignment.
    • Use is_always_lock_free (compile-time) or is_lock_free() (runtime) to verify.
    • Integral and pointer specialisations provide arithmetic fetch operations.
    • Atomic objects cannot be copied or moved; load the value and construct a new atomic instead.
    • C++20 added wait/notify to all atomic types and expanded atomic_flag with a non-modifying test().
