    Atomic Operations

    March 2026 · 30 min read

    What Are Atomic Operations?

    An atomic operation is indivisible: it completes entirely or not at all, with no possibility of another thread observing a half-finished state. No tearing, no interleaving, no data race.

    std::atomic<T> (from <atomic>) provides atomic operations on a value of type T. For integral and pointer types, this gives you lock-free read-modify-write operations that are much faster than mutex-based synchronization for simple tasks.

    #include <atomic>
#include <thread>
#include <iostream>

std::atomic<int> counter{0};

void increment() {
    for (int i = 0; i < 1000000; ++i) {
        counter.fetch_add(1, std::memory_order_relaxed);
    }
}

int main() {
    std::thread t1(increment), t2(increment);
    t1.join(); t2.join();
    std::cout << counter.load() << "\n";  // always 2000000
}

    Basic Atomic Operations

    load() and store()

    std::atomic<int> x{42};

int val = x.load();        // atomically read
x.store(100);              // atomically write
int old = x.exchange(200); // atomically swap, returns old value

    Read-Modify-Write Operations

    std::atomic<int> counter{0};

counter.fetch_add(1);    // atomic increment, returns old value
counter.fetch_sub(1);    // atomic decrement
counter.fetch_and(0xFF); // atomic bitwise AND
counter.fetch_or(0x1);   // atomic bitwise OR
counter.fetch_xor(0x3);  // atomic bitwise XOR

// Shorthand operators (use seq_cst by default)
counter++;
counter--;
counter += 5;
counter -= 3;

    Compare-and-Swap (CAS)

    The most powerful and fundamental atomic operation. It checks if the value matches expected; if so, it atomically writes desired. Otherwise, it loads the current value into expected:

    std::atomic<int> x{5};
int expected = 5;
bool success = x.compare_exchange_strong(expected, 10);
// If x was 5: x is now 10, expected still 5, success = true
// If x was 7: x unchanged, expected now 7, success = false

    CAS is the building block of all lock-free data structures. A typical pattern is a retry loop:

    void atomic_multiply(std::atomic<int>& x, int factor) {
    int old = x.load();
    while (!x.compare_exchange_weak(old, old * factor)) {
        // old is updated with the current value on failure
        // retry with the new value
    }
}

    compare_exchange_strong vs compare_exchange_weak

    _strong only fails if the value does not match. _weak may also fail spuriously (even if the value matches), but is faster on some architectures. Use _weak in loops where you retry anyway, and _strong when a spurious failure is expensive.

    std::atomic_flag: The Simplest Atomic

    atomic_flag is the only type guaranteed to be lock-free on every platform. It is a boolean flag with two operations: test_and_set() and clear().

    Building a Spinlock

    class SpinLock {
    std::atomic_flag flag_ = ATOMIC_FLAG_INIT;

public:
    void lock() {
        while (flag_.test_and_set(std::memory_order_acquire)) {
            // Spin: busy-wait until the flag is cleared
            // Optionally: std::this_thread::yield();
        }
    }

    void unlock() {
        flag_.clear(std::memory_order_release);
    }
};
    Spinlocks vs mutexes: Spinlocks waste CPU cycles while waiting (busy-wait). They are only beneficial when the critical section is extremely short (tens of nanoseconds) and contention is low. For anything else, use std::mutex, which puts the thread to sleep and avoids burning CPU.

    When to Use Atomics vs Mutexes

    Use CaseRecommendation
    Simple counterstd::atomic<int>
    Boolean flag (stop/ready)std::atomic<bool>
    Pointer swapstd::atomic<T*>
    Compound operation on a data structurestd::mutex
    Multiple variables updated togetherstd::mutex
    Waiting for a conditionstd::condition_variable

    is_lock_free() and is_always_lock_free

    Not all std::atomic<T> specializations are truly lock-free. If T is too large for the hardware's atomic instructions, the implementation falls back to an internal mutex:

    std::atomic<int> a;
std::cout << a.is_lock_free() << "\n";  // true on most platforms

struct Big { char data[1000]; };
std::atomic<Big> b;
std::cout << b.is_lock_free() << "\n";  // almost certainly false

// Compile-time check (C++17):
static_assert(std::atomic<int>::is_always_lock_free);

    std::atomic_ref (C++20)

    Provides atomic operations on a non-atomic variable. Useful when you want atomic access in some contexts but not others:

    int regular_var = 0;

void thread_func() {
    std::atomic_ref ref(regular_var);
    ref.fetch_add(1);  // atomic increment on a regular int
}

    Memory Orderings (Preview)

    Every atomic operation accepts an optional std::memory_order parameter. The default is memory_order_seq_cst (sequential consistency), which is the strongest and easiest to reason about. Weaker orderings (acquire, release, relaxed) provide better performance on some architectures but require deeper understanding.

    Summary

    • std::atomic<T> provides indivisible operations: no tearing, no data races.
    • load(), store(), exchange() are basic reads and writes.
    • fetch_add(), fetch_sub() are atomic read-modify-write.
    • compare_exchange_strong/weak() (CAS) is the foundation of lock-free algorithms.
    • atomic_flag + test_and_set() builds spinlocks.
    • Use atomics for simple counters and flags. Use mutexes for compound operations.
    • The default memory ordering (seq_cst) is safe. Weaker orderings are an optimization tool.
