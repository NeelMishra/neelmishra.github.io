    Spurious Wakeups

    March 2026

    What Are Spurious Wakeups?

    A spurious wakeup is when a thread returns from wait() on a condition variable even though no other thread called notify_one() or notify_all(). The thread wakes up, but the condition it was waiting for is still false.

    This is not a bug in your code. The C++ standard explicitly allows implementations to wake threads spuriously. POSIX pthreads, which underlie most C++ standard library implementations on Unix, have always permitted this behaviour.

    Standard Wording. The C++ standard states that wait() may unblock spuriously, meaning it can return even if notify_one() or notify_all() was not called. The standard mandates that you always recheck your predicate after waking.

    Why Do Spurious Wakeups Happen?

    There are several reasons at the OS and hardware level:

    1. OS Signal Interrupts

    On POSIX systems, if a signal is delivered to a thread while it is blocked on a futex or similar kernel primitive, the system call may return with EINTR. Rather than adding expensive retry logic deep in the kernel, the implementation surfaces this as a wakeup and lets userspace decide whether the condition is satisfied.

    2. Multiprocessor Optimisations

    On systems with multiple CPUs, the kernel may batch wakeup operations. When notify_one() targets one thread, other threads sharing the same wait queue may also be woken as a side effect of cache invalidation or the way the kernel manages its internal data structures.

    3. Futex Implementation Details (Linux)

    Linux implements condition variables on top of futexes. Certain code paths in the futex subsystem can cause a waiter to be woken even when no explicit wake call targeted it, particularly during requeue operations (FUTEX_CMP_REQUEUE) used to implement notify_all() efficiently.

    4. Stolen Wakeups

    Sometimes a thread is legitimately woken, but before it can acquire the mutex and check the predicate, another thread has already consumed the work. This is not technically a spurious wakeup from the OS perspective, but it looks identical from the application perspective: the thread wakes up and finds nothing to do.

    How the Predicate Loop Solves It

    The canonical pattern is a while loop that rechecks the condition after every wakeup:

    std::unique_lock<std::mutex> lock(mtx);
while (!ready) {          // re-check after every wakeup
    cv.wait(lock);
}
// ready is guaranteed true here, lock is held

    The predicate overload of wait() does exactly this internally:

    template<class Pred>
void condition_variable::wait(unique_lock<mutex>& lock, Pred pred) {
    while (!pred()) {
        wait(lock);
    }
}

    Whether the wakeup was spurious or real does not matter. If the predicate is false, the thread goes back to sleep. If it is true, execution continues. This is why the C++ standard can afford to be lenient about spurious wakeups: the correct usage pattern handles them gracefully.

    Performance Note. Spurious wakeups are rare in practice. Some developers run for years without ever triggering one. But "rare" is not "never," and concurrent bugs that manifest rarely are the hardest to debug. Always use the predicate.

    Common Mistakes

    1. Waiting Without a Predicate

    // WRONG: vulnerable to spurious wakeups
std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock);
// Cannot assume the condition is true here!

    This is the most common mistake. The thread may wake up with the condition still false and proceed to access data that is not ready.

    2. Using if Instead of while

    // WRONG: only checks once
std::unique_lock<std::mutex> lock(mtx);
if (!ready) {
    cv.wait(lock);
}
// Still not safe: a spurious wakeup skips the check

    An if checks the condition once before waiting. After waking, it falls through without rechecking. You need a while loop or the predicate overload.

    3. Predicate That is Always True

    // Probably a bug: this never waits
cv.wait(lock, []{ return true; });

    4. Checking the Predicate Outside the Lock

    // WRONG: data race on 'ready'
if (ready) return;             // no lock held!
std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, []{ return ready; });

    The initial check of ready without the mutex is a data race. Always access the shared predicate under the lock.

    Debugging Condition Variable Bugs

    CV bugs are notoriously hard to reproduce. Here are practical techniques:

    Thread Sanitizer (TSan)

    Compile with -fsanitize=thread (GCC/Clang). TSan detects data races on the predicate variable, which is the root cause of most CV bugs.

    g++ -std=c++20 -fsanitize=thread -g main.cpp -o main

    Logging with Timestamps

    Add fine-grained logging around wait() and notify_*() calls. Include thread IDs and high-resolution timestamps to identify ordering issues.

    auto now = std::chrono::steady_clock::now();
auto ms = std::chrono::duration_cast<std::chrono::microseconds>(
    now.time_since_epoch()).count();
std::cerr << "[" << ms << "] T" << std::this_thread::get_id()
          << " entering wait, pred=" << pred() << "\n";

    Stress Testing

    Run the program thousands of times in a loop. Spurious wakeups and race conditions often only manifest under load or specific scheduling patterns.

    # Run 10000 iterations, stop on first failure
for i in $(seq 1 10000); do
    ./my_program || { echo "Failed on run $i"; break; }
done

    Helgrind (Valgrind)

    Helgrind is another data race detector, part of the Valgrind suite. It works without recompilation.

    valgrind --tool=helgrind ./my_program

    Stolen Wakeups vs Spurious Wakeups

    AspectSpurious WakeupStolen Wakeup
    CauseOS / hardware artefactAnother thread consumed the work
    Was notify called?NoYes
    Is predicate true?NoNo (consumed by another waiter)
    SolutionSame: always use a predicate loop

    Spurious Wakeups with Timed Waits

    Timed waits (wait_for, wait_until) also suffer from spurious wakeups. The predicate overload handles this correctly:

    // Correct: predicate is checked on every wakeup, including spurious ones
bool got_it = cv.wait_for(lock, 100ms, []{ return data_ready; });
// got_it is true only if predicate became true within the timeout

    Without the predicate, wait_for returns cv_status::no_timeout on a spurious wakeup, which can be misinterpreted as a legitimate notification.

    How Often Do Spurious Wakeups Actually Happen?

    Empirically, they are extremely rare on modern Linux (glibc). Some developers build tests specifically designed to trigger them (heavy signal load, CPU oversubscription, memory pressure) and still cannot reproduce one. On Windows, the SleepConditionVariableCS API documentation similarly warns about spurious wakeups but they are also very rare in practice.

    However, "rare" is architecture and workload dependent. Real-time systems, systems under heavy signal traffic, and programs using setjmp/longjmp near CV waits are more likely to see them. The cost of the predicate check is negligible compared to the risk.

    Summary

    • Spurious wakeups are permitted by the C++ standard and allowed by POSIX.
    • They happen due to OS signals, multiprocessor scheduling, and futex internals.
    • The predicate loop (while (!pred) cv.wait(...)) handles both spurious wakeups and stolen wakeups identically.
    • Always use the predicate overload of wait() for clarity and correctness.
    • Use TSan, Helgrind, and stress testing to catch CV bugs in your code.
