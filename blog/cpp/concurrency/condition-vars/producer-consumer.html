    Producer-Consumer Pattern

    March 2026

    The Classic Pattern

    The producer-consumer pattern decouples the creation of work items from their processing. One or more producer threads generate data and place it into a shared buffer, while one or more consumer threads take data from the buffer and process it.

    This pattern appears everywhere: thread pools (tasks are produced, worker threads consume them), logging systems (log messages are produced, a writer thread consumes them), I/O pipelines, and message queues.

    Core Idea. The buffer acts as a shock absorber between threads that produce and consume at different rates. Condition variables coordinate the "buffer not empty" and "buffer not full" signals.

    Single Producer, Single Consumer

    The simplest version uses one producer, one consumer, and an unbounded queue:

    #include <mutex>
#include <condition_variable>
#include <queue>
#include <thread>
#include <iostream>

std::mutex mtx;
std::condition_variable cv_not_empty;
std::queue<int> buffer;
bool done = false;

void producer() {
    for (int i = 0; i < 10; ++i) {
        {
            std::lock_guard<std::mutex> lock(mtx);
            buffer.push(i);
        }
        cv_not_empty.notify_one();
    }
    {
        std::lock_guard<std::mutex> lock(mtx);
        done = true;
    }
    cv_not_empty.notify_one();
}

void consumer() {
    while (true) {
        std::unique_lock<std::mutex> lock(mtx);
        cv_not_empty.wait(lock, [] {
            return !buffer.empty() || done;
        });
        while (!buffer.empty()) {
            std::cout << "Got: " << buffer.front() << "\n";
            buffer.pop();
        }
        if (done) break;
    }
}

int main() {
    std::thread p(producer), c(consumer);
    p.join(); c.join();
}

    This works but has no bound on buffer size. If the producer is faster than the consumer, memory usage grows without limit.

    Bounded Buffer Implementation

    A bounded buffer limits the queue to a maximum capacity. The producer blocks when the buffer is full (backpressure), and the consumer blocks when it is empty.

    template<typename T>
class BoundedBuffer {
    std::queue<T> queue_;
    const size_t capacity_;
    std::mutex mtx_;
    std::condition_variable not_full_;
    std::condition_variable not_empty_;

public:
    explicit BoundedBuffer(size_t cap) : capacity_(cap) {}

    void push(T item) {
        std::unique_lock<std::mutex> lock(mtx_);
        not_full_.wait(lock, [this] {
            return queue_.size() < capacity_;
        });
        queue_.push(std::move(item));
        not_empty_.notify_one();
    }

    T pop() {
        std::unique_lock<std::mutex> lock(mtx_);
        not_empty_.wait(lock, [this] {
            return !queue_.empty();
        });
        T item = std::move(queue_.front());
        queue_.pop();
        not_full_.notify_one();
        return item;
    }

    size_t size() const {
        std::lock_guard<std::mutex> lock(mtx_);
        return queue_.size();
    }
};
    Two CVs, One Mutex. The bounded buffer uses two condition variables sharing the same mutex. not_full_ signals the producer when space becomes available. not_empty_ signals the consumer when data arrives.

    Multiple Producers, Multiple Consumers

    The bounded buffer class above already supports MPMC (multiple producers, multiple consumers) because all access is serialised through the mutex. However, there are subtleties:

    notify_one vs notify_all

    With a single consumer, notify_one() in push() is correct. With multiple consumers, notify_one() is still correct because only one consumer can process each item. However, if you have multiple producers blocked on a full buffer, notify_all() in pop() ensures at least one producer is woken. In practice, notify_one() usually suffices because the predicate loop handles stolen wakeups.

    // MPMC usage
BoundedBuffer<Task> queue(64);

// N producers
for (int i = 0; i < N; ++i) {
    producers.emplace_back([&queue, i] {
        for (auto& task : generate_tasks(i)) {
            queue.push(std::move(task));
        }
    });
}

// M consumers
for (int i = 0; i < M; ++i) {
    consumers.emplace_back([&queue] {
        while (auto task = queue.try_pop()) {
            task->execute();
        }
    });
}

    Graceful Shutdown

    For MPMC, a common pattern is sending poison pills (sentinel values) to signal consumers to exit. If you have M consumers, send M poison pills:

    // After all producers finish:
for (int i = 0; i < M; ++i) {
    queue.push(POISON_PILL);  // each consumer gets one
}

    Alternatively, add a close() method that sets a flag and calls notify_all() on both CVs.

    Backpressure

    When the buffer is full and the producer blocks, this is backpressure. The consumer is applying implicit flow control: "slow down, I cannot keep up."

    Backpressure is desirable in most systems. Without it, a fast producer will either exhaust memory (unbounded queue) or start dropping items. The bounded buffer gives you backpressure for free.

    Tuning Buffer Size

    • Too small (e.g., 1): high contention, producer blocks frequently, poor throughput.
    • Too large (e.g., 1M): high latency for individual items, large memory footprint, delayed backpressure signals.
    • Sweet spot: depends on your workload. Profile with different sizes. A common starting point is a few hundred to a few thousand items.
    Latency vs Throughput. A larger buffer increases throughput (producers rarely block) but also increases tail latency (items sit in the buffer longer). Choose based on your system requirements.

    Optimised Variant: Ring Buffer

    For high-performance use cases, replace std::queue with a fixed-size circular buffer backed by a std::vector. This avoids dynamic allocation per item:

    template<typename T>
class RingBuffer {
    std::vector<T> data_;
    size_t head_ = 0, tail_ = 0, count_ = 0;
    size_t cap_;
    std::mutex mtx_;
    std::condition_variable not_full_, not_empty_;

public:
    explicit RingBuffer(size_t cap)
        : data_(cap), cap_(cap) {}

    void push(T item) {
        std::unique_lock<std::mutex> lock(mtx_);
        not_full_.wait(lock, [this]{ return count_ < cap_; });
        data_[tail_] = std::move(item);
        tail_ = (tail_ + 1) % cap_;
        ++count_;
        not_empty_.notify_one();
    }

    T pop() {
        std::unique_lock<std::mutex> lock(mtx_);
        not_empty_.wait(lock, [this]{ return count_ > 0; });
        T item = std::move(data_[head_]);
        head_ = (head_ + 1) % cap_;
        --count_;
        not_full_.notify_one();
        return item;
    }
};

    Related Patterns

    Pipeline Pattern

    Chain multiple producer-consumer stages together. Stage 1 produces into Buffer A, consumer of Buffer A produces into Buffer B, and so on. Each stage runs in its own thread(s).

    Fan-Out / Fan-In

    A single producer feeds work into a buffer consumed by N workers (fan-out). Results are collected into a second buffer by a single aggregator (fan-in).

    Work Stealing

    Each worker has its own deque. When a worker runs out of tasks, it steals from the back of another worker's deque. This avoids the central buffer bottleneck at the cost of more complex synchronisation.

    Pitfalls

    • Forgetting to signal not_full_ after consuming: producers starve.
    • Forgetting to signal not_empty_ after producing: consumers starve.
    • Using a single CV for both conditions: works with notify_all() but is wasteful. Use two separate CVs.
    • Holding the lock during slow processing: defeats concurrency. Pop the item, release the lock, then process.

    Summary

    • The producer-consumer pattern decouples work generation from work processing.
    • A bounded buffer with two condition variables (not_full_ and not_empty_) provides natural backpressure.
    • The same BoundedBuffer class supports single or multiple producers and consumers.
    • Use poison pills or a close flag for graceful shutdown of MPMC queues.
    • Tune buffer size to balance latency and throughput for your workload.
