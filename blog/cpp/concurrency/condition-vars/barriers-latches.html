    Barriers, Latches, and Semaphores

    March 2026

    Beyond Condition Variables

    Condition variables are powerful but low-level. You manage the mutex, the predicate, and the notification yourself. For many common coordination patterns, C++20 introduced three higher-level primitives that are harder to misuse:

    • std::latch – a single-use countdown barrier
    • std::barrier – a reusable synchronisation point
    • std::counting_semaphore – a counter that controls concurrent access

    All three live in <latch>, <barrier>, and <semaphore> respectively.

    std::latch

    A latch is initialised with a positive count. Threads call count_down() to decrement the counter. When the counter reaches zero, all threads blocked on wait() are released. A latch can only be used once; there is no way to reset it.

    #include <latch>
#include <thread>
#include <iostream>
#include <vector>

void worker(int id, std::latch& start_signal) {
    start_signal.arrive_and_wait();  // block until all workers ready
    std::cout << "Worker " << id << " running\n";
}

int main() {
    const int N = 4;
    std::latch start_signal(N);
    std::vector<std::thread> threads;

    for (int i = 0; i < N; ++i)
        threads.emplace_back(worker, i, std::ref(start_signal));

    for (auto& t : threads) t.join();
}

    Key Operations

    MethodEffect
    count_down(n)Decrements counter by n (default 1). Does not block.
    wait()Blocks until counter reaches zero.
    arrive_and_wait(n)Equivalent to count_down(n) then wait().
    try_wait()Returns true if counter is zero, without blocking.
    Use Case. Latches are ideal for "wait until N things are done" scenarios: loading N resources before starting, waiting for N workers to initialise, or fan-out / fan-in where you fire off N tasks and wait for all of them.

    std::barrier

    A barrier is a reusable synchronisation point. A group of threads arrives at the barrier; once all have arrived, they are all released and the barrier resets for the next phase. Optionally, a completion function runs between phases.

    #include <barrier>
#include <thread>
#include <iostream>
#include <vector>

void phase_work(int id, std::barrier<>& sync, int phases) {
    for (int p = 0; p < phases; ++p) {
        std::cout << "Thread " << id << " phase " << p << "\n";
        sync.arrive_and_wait();
    }
}

int main() {
    const int N = 3;
    auto on_completion = []() noexcept {
        std::cout << "--- Phase complete ---\n";
    };
    std::barrier sync(N, on_completion);

    std::vector<std::thread> threads;
    for (int i = 0; i < N; ++i)
        threads.emplace_back(phase_work, i, std::ref(sync), 3);

    for (auto& t : threads) t.join();
}

    Key Operations

    MethodEffect
    arrive(n)Arrives and decrements by n. Returns an arrival token. Does not block.
    wait(token)Blocks until the phase associated with the token completes.
    arrive_and_wait()Combined arrive + wait.
    arrive_and_drop()Arrives and permanently reduces the expected count for future phases.
    Completion Function. The barrier's completion function runs exactly once per phase, after all threads have arrived but before any are released. It runs on one of the arriving threads. This is perfect for aggregation or state preparation between phases.

    Barrier vs Latch

    • A latch is single-use. Once the count hits zero, it stays at zero forever.
    • A barrier automatically resets after each phase, allowing iterative algorithms.
    • A barrier requires a fixed set of participating threads. A latch does not.

    std::counting_semaphore

    A counting semaphore maintains an internal counter. acquire() blocks until the counter is positive, then decrements it. release() increments the counter (and may unblock a waiting thread).

    #include <semaphore>
#include <thread>
#include <iostream>
#include <vector>

// Allow at most 3 threads to access the resource concurrently
std::counting_semaphore<3> sem(3);

void worker(int id) {
    sem.acquire();
    std::cout << "Thread " << id << " in critical section\n";
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
    std::cout << "Thread " << id << " leaving\n";
    sem.release();
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 8; ++i)
        threads.emplace_back(worker, i);
    for (auto& t : threads) t.join();
}

    Binary Semaphore

    std::binary_semaphore is an alias for std::counting_semaphore<1>. It behaves like a mutex but without ownership semantics: any thread can release it, not just the one that acquired it. This makes it useful for signalling between threads.

    std::binary_semaphore signal(0);  // starts "locked"

// Thread A (producer)
prepare_data();
signal.release();  // signal that data is ready

// Thread B (consumer)
signal.acquire();  // wait for data
use_data();

    When to Use Each

    Use a condition variable when:

    • You need maximum flexibility (complex predicates, dynamic waiter sets).
    • You are on C++11/14/17 without access to C++20 primitives.
    • Your pattern does not fit neatly into latch/barrier/semaphore.

    Use a latch when:

    • You need to wait for N events, and then proceed once.
    • "Wait for all workers to finish initialisation."
    • "Wait for N downloads to complete before processing."

    Use a barrier when:

    • You have iterative parallel algorithms with distinct phases.
    • "All threads compute, then synchronise, then compute again."
    • Parallel matrix operations, simulation steps, BSP-style algorithms.

    Use a semaphore when:

    • You want to limit concurrency to N threads at a time.
    • "At most 5 threads can access the database connection pool."
    • Thread-to-thread signalling (binary semaphore).

    Converting Legacy CV Code

    Many codebases have hand-rolled latch or barrier patterns using condition variables and counters. Converting them to C++20 primitives reduces boilerplate and eliminates common bugs.

    CV-based latch (before)

    // Legacy: hand-rolled latch
class ManualLatch {
    std::mutex mtx_;
    std::condition_variable cv_;
    int count_;
public:
    explicit ManualLatch(int n) : count_(n) {}
    void count_down() {
        std::lock_guard<std::mutex> lock(mtx_);
        if (--count_ == 0) cv_.notify_all();
    }
    void wait() {
        std::unique_lock<std::mutex> lock(mtx_);
        cv_.wait(lock, [this]{ return count_ == 0; });
    }
};

    C++20 latch (after)

    // C++20: replace with std::latch
#include <latch>
std::latch done(N);

// In worker:
done.count_down();

// In main:
done.wait();

    That is 3 lines replacing an entire class. The standard implementation is also likely more efficient (it can use atomic operations instead of a full mutex).

    CV-based barrier (before)

    class ManualBarrier {
    std::mutex mtx_;
    std::condition_variable cv_;
    int expected_;
    int count_;
    int generation_ = 0;
public:
    explicit ManualBarrier(int n) : expected_(n), count_(n) {}
    void arrive_and_wait() {
        std::unique_lock<std::mutex> lock(mtx_);
        int gen = generation_;
        if (--count_ == 0) {
            ++generation_;
            count_ = expected_;
            cv_.notify_all();
        } else {
            cv_.wait(lock, [this, gen]{ return gen != generation_; });
        }
    }
};

    C++20 barrier (after)

    #include <barrier>
std::barrier sync(N);

// In each thread:
sync.arrive_and_wait();

    The generation tracking, the manual count reset, and the predicate logic all disappear.

    Migration Tip. Before converting, verify that your compiler supports C++20 <latch>, <barrier>, and <semaphore>. GCC 11+, Clang 14+, and MSVC 19.28+ have full support.

    Converting CV to Semaphore

    // Legacy: binary signal with CV
std::mutex mtx;
std::condition_variable cv;
bool ready = false;

// Producer:
{ std::lock_guard lock(mtx); ready = true; }
cv.notify_one();

// Consumer:
{ std::unique_lock lock(mtx);
  cv.wait(lock, []{ return ready; }); }

// C++20 replacement:
std::binary_semaphore signal(0);
signal.release();    // producer
signal.acquire();    // consumer

    The semaphore version needs no mutex, no predicate, and cannot suffer from spurious wakeups.

    Performance Considerations

    The C++20 primitives are generally as fast or faster than hand-rolled CV code:

    • Latches and barriers can use atomic countdown internally, avoiding mutex overhead when threads arrive without contention.
    • Semaphores map directly to OS primitives on most platforms (e.g., Windows CreateSemaphore, Linux futex).
    • All three avoid the spurious wakeup issue entirely, so there is no unnecessary wakeup-recheck-sleep cycle.

    Summary

    • std::latch is a single-use countdown that releases all waiters when the count hits zero.
    • std::barrier is a reusable synchronisation point with an optional per-phase completion function.
    • std::counting_semaphore controls N-way concurrent access without requiring a mutex or predicate.
    • All three are less error-prone than raw condition variables: no spurious wakeups, no forgotten predicates, no manual mutex management.
    • Prefer these C++20 primitives when your pattern fits. Fall back to condition variables for complex or dynamic coordination.
