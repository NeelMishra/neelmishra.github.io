    Condition Variable Basics

    March 2026

    What Is a Condition Variable?

    A condition variable is a synchronisation primitive that allows threads to wait until a particular condition becomes true. Instead of spinning in a loop burning CPU cycles, a thread can sleep on a condition variable and be woken up by another thread when the state it cares about has changed.

    In C++, the standard library provides std::condition_variable (works only with std::unique_lock<std::mutex>) and std::condition_variable_any (works with any lock type). For the vast majority of use cases, the first one is what you want.

    #include <mutex>
#include <condition_variable>

std::mutex mtx;
std::condition_variable cv;
bool data_ready = false;
    Key Insight. A condition variable never carries state on its own. It is always paired with a predicate (a boolean expression) and a mutex that guards that predicate.

    The Core API

    std::condition_variable exposes three families of operations:

    1. wait – blocks the calling thread until notified.
    2. notify_one – wakes exactly one waiting thread.
    3. notify_all – wakes every thread currently waiting on this CV.

    wait()

    The basic form atomically releases the mutex and suspends the thread. When the thread is woken, it reacquires the mutex before wait() returns.

    std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock);  // releases mtx, sleeps, reacquires mtx on wake

    wait() with Predicate

    The predicate overload is equivalent to wrapping wait() in a while loop that rechecks a condition:

    // These two are equivalent:
cv.wait(lock, []{ return data_ready; });

// Manual version:
while (!data_ready) {
    cv.wait(lock);
}

    The predicate form is strongly preferred because it handles spurious wakeups automatically and makes intent clearer.

    notify_one and notify_all

    // Producer thread:
{
    std::lock_guard<std::mutex> lock(mtx);
    data_ready = true;
}
cv.notify_one();  // wake one waiter

    You do not need to hold the mutex when calling notify_one() or notify_all(), but you must hold it while modifying the shared predicate. Notifying outside the lock can reduce contention because the woken thread does not have to immediately compete for the mutex.

    Why You Need a Mutex with a CV

    The mutex serves two purposes:

    1. Protecting the predicate. The shared boolean (or queue size, or whatever condition you test) is mutable state accessed from multiple threads. Without a mutex, you have a data race.
    2. Preventing lost wakeups. The mutex ensures that the check of the predicate and the call to wait() happen as a single atomic step. Without this guarantee, a notification can slip between the check and the wait, and the waiting thread sleeps forever.

    The Lost Wakeup Problem

    A lost wakeup occurs when a notification is sent before the consumer has entered its wait call. Since condition variables do not store pending signals, the notification is simply dropped.

    ▶ Lost Wakeup Scenario

    Step through to see how a notification can be lost when the mutex is not used correctly.

    Fixing the Lost Wakeup

    The solution is straightforward: always check the predicate while holding the mutex, and use the predicate form of wait().

    // Consumer (correct)
std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, []{ return data_ready; });
// When we get here, lock is held and data_ready is true.
process(data);

    Because the mutex is held when the predicate is checked, and wait() atomically releases the mutex and enters the wait state, there is no window for a notification to slip through.

    Timed Waits: wait_for and wait_until

    Sometimes you do not want to block indefinitely. The standard provides timed variants:

    // Wait at most 100 milliseconds
auto status = cv.wait_for(lock, std::chrono::milliseconds(100),
                          []{ return data_ready; });
if (status) {
    // Condition became true within the timeout
} else {
    // Timed out
}

    wait_until takes an absolute time point instead of a duration. Both return a bool when used with a predicate (true if the predicate is satisfied, false on timeout) or a std::cv_status when used without one.

    condition_variable vs condition_variable_any

    Featurecondition_variablecondition_variable_any
    Lock typeunique_lock<mutex> onlyAny Lockable type
    PerformanceOptimal (no extra allocation)May allocate internally
    Use caseDefault choiceCustom mutexes, shared_mutex
    Rule of Thumb. Use std::condition_variable unless you have a concrete reason to need a different lock type. The _any variant pays for flexibility with potential overhead.

    Complete Example

    #include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>

std::mutex mtx;
std::condition_variable cv;
std::queue<int> work_queue;
bool finished = false;

void producer() {
    for (int i = 1; i <= 5; ++i) {
        {
            std::lock_guard<std::mutex> lock(mtx);
            work_queue.push(i);
            std::cout << "Produced: " << i << "\n";
        }
        cv.notify_one();
    }
    {
        std::lock_guard<std::mutex> lock(mtx);
        finished = true;
    }
    cv.notify_one();
}

void consumer() {
    while (true) {
        std::unique_lock<std::mutex> lock(mtx);
        cv.wait(lock, []{ return !work_queue.empty() || finished; });
        while (!work_queue.empty()) {
            int val = work_queue.front();
            work_queue.pop();
            std::cout << "Consumed: " << val << "\n";
        }
        if (finished) break;
    }
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join();
    t2.join();
    return 0;
}

    Common Mistakes

    1. Forgetting the predicate

    Calling cv.wait(lock) without a predicate leaves you vulnerable to spurious wakeups and lost wakeups.

    2. Modifying the predicate without the mutex

    // WRONG: data race on data_ready
data_ready = true;         // no lock!
cv.notify_one();

    3. Using lock_guard instead of unique_lock

    std::condition_variable::wait() needs to unlock and relock the mutex internally. lock_guard does not support manual unlock, so you must use unique_lock.

    4. Holding the lock during lengthy work

    If the consumer holds the lock while processing data, the producer cannot push new items. Move the processing outside the critical section when possible.

    Summary

    • A condition variable lets threads sleep until a condition changes, avoiding busy-waiting.
    • Always pair a CV with a mutex and a predicate.
    • Use the predicate overload of wait() to handle spurious wakeups automatically.
    • Modify the shared state under the mutex, then call notify_one() or notify_all().
    • The lost wakeup problem vanishes when you follow the canonical pattern.
