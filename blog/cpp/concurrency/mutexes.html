    Mutexes and Locks

    March 2026 · 30 min read

    What Is a Mutex?

    A mutex (mutual exclusion) is the most fundamental synchronization primitive. It provides a simple contract: at most one thread can own (lock) a mutex at any time. Any other thread that tries to lock an already-owned mutex will block (sleep) until the owner releases (unlocks) it.

    Think of a mutex as a key to a single-occupancy bathroom. Whoever has the key can enter. Everyone else waits in line. When the occupant leaves, they hand the key to the next person.

    #include <mutex>
#include <thread>
#include <iostream>

std::mutex mtx;
int counter = 0;

void safe_increment() {
    for (int i = 0; i < 1000000; ++i) {
        mtx.lock();       // acquire the mutex
        ++counter;        // critical section: only one thread at a time
        mtx.unlock();     // release the mutex
    }
}

int main() {
    std::thread t1(safe_increment);
    std::thread t2(safe_increment);
    t1.join();
    t2.join();
    std::cout << "counter = " << counter << "\n";  // always 2000000
}
    Never call lock/unlock manually. The code above is for illustration only. In practice, always use RAII lock wrappers (lock_guard, unique_lock) so the mutex is released even if an exception occurs between lock() and unlock().

    Watch two threads contend for a mutex. Only one enters the critical section at a time.

    std::lock_guard: The Simple RAII Lock

    std::lock_guard is the simplest RAII wrapper for a mutex. It locks the mutex in its constructor and unlocks it in its destructor. Period. No other operations are available.

    void safe_increment() {
    for (int i = 0; i < 1000000; ++i) {
        std::lock_guard<std::mutex> lock(mtx);  // locks mtx
        ++counter;
    }   // lock destroyed: mtx unlocked
}

// C++17: CTAD simplifies the syntax
void safe_increment_17() {
    for (int i = 0; i < 1000000; ++i) {
        std::lock_guard lock(mtx);  // no need for <std::mutex>
        ++counter;
    }
}

    The beauty of lock_guard is that it is impossible to forget to unlock. Even if the critical section throws an exception, the destructor runs during stack unwinding and the mutex is released. This is the RAII pattern applied to synchronization.

    Minimizing Lock Scope

    Hold the lock for as little time as possible. Long critical sections block other threads, reducing parallelism:

    // BAD: holding lock during expensive_compute()
{
    std::lock_guard lock(mtx);
    int data = shared_data;
    int result = expensive_compute(data);  // other threads blocked!
    shared_results.push_back(result);
}

// GOOD: lock only for the shared data access
int data;
{
    std::lock_guard lock(mtx);
    data = shared_data;                    // quick copy
}   // unlock ASAP
int result = expensive_compute(data);      // no lock held
{
    std::lock_guard lock(mtx);
    shared_results.push_back(result);      // quick push
}

    std::unique_lock: The Flexible Lock

    std::unique_lock provides the same RAII guarantees as lock_guard but with much more flexibility:

    • Deferred locking: Construct without locking, lock later.
    • Try locking: Non-blocking attempt to acquire.
    • Timed locking: Wait up to a duration.
    • Manual unlock/relock: Release and reacquire within the guard's lifetime.
    • Movable: Transfer ownership between scopes.
    • Required by condition_variable: std::condition_variable::wait() needs a unique_lock.
    std::mutex mtx;

// Deferred locking
std::unique_lock lock(mtx, std::defer_lock);
// mtx is NOT locked yet
do_non_critical_work();
lock.lock();   // lock when needed
critical_work();
lock.unlock(); // release early
more_non_critical_work();
// destructor: no-op since already unlocked

// Try locking (non-blocking)
std::unique_lock lock2(mtx, std::try_to_lock);
if (lock2.owns_lock()) {
    // got the lock
    do_work();
} else {
    // mutex is busy, do something else
    fallback();
}

// Timed locking
std::timed_mutex tmtx;
std::unique_lock lock3(tmtx, std::chrono::milliseconds(100));
if (lock3.owns_lock()) {
    // acquired within 100ms
} else {
    // timed out
}

    std::scoped_lock: Lock Multiple Mutexes (C++17)

    std::scoped_lock is lock_guard's big sibling. It can lock multiple mutexes simultaneously using a deadlock-avoidance algorithm:

    std::mutex mtx_a, mtx_b;

void transfer(Account& from, Account& to, int amount) {
    // Locks both mutexes without deadlock risk
    std::scoped_lock lock(mtx_a, mtx_b);
    from.withdraw(amount);
    to.deposit(amount);
}   // both mutexes released

    Without scoped_lock, locking two mutexes in different orders in different code paths leads to deadlocks (covered in the next post). scoped_lock uses the same algorithm as std::lock() internally to avoid this.

    The Mutex Family

    C++ provides several mutex types for different needs:

    Type Header Key Feature
    std::mutex<mutex>Basic exclusive ownership. Most common.
    std::recursive_mutex<mutex>Same thread can lock multiple times (must unlock same number of times).
    std::timed_mutex<mutex>Supports try_lock_for() and try_lock_until() with timeouts.
    std::recursive_timed_mutex<mutex>Both recursive and timed.
    std::shared_mutex<shared_mutex>Multiple readers OR one writer (C++17).
    std::shared_timed_mutex<shared_mutex>Shared mutex with timeouts (C++14).

    Recursive Mutex

    A recursive mutex allows the same thread to lock it multiple times. This is useful (but usually a code smell) when a public locked function calls another public locked function on the same object:

    class Widget {
    std::recursive_mutex mtx_;
    int value_ = 0;

public:
    void set(int v) {
        std::lock_guard lock(mtx_);
        value_ = v;
    }

    void double_it() {
        std::lock_guard lock(mtx_);  // lock #1
        set(value_ * 2);             // lock #2 (same thread, OK)
    }   // unlock #2, then unlock #1
};
    Prefer refactoring over recursive_mutex. Usually, recursive locking indicates that your class should have private unlocked methods and public locked methods that call them. A recursive mutex is a band-aid that hides design problems and carries extra overhead.

    Shared Mutex (Reader-Writer Lock)

    std::shared_mutex (C++17) allows multiple threads to hold shared (read) locks simultaneously, but only one thread can hold an exclusive (write) lock. This is perfect for data that is read frequently and written rarely:

    #include <shared_mutex>

class ThreadSafeCache {
    mutable std::shared_mutex mtx_;
    std::unordered_map<std::string, int> cache_;

public:
    // Multiple readers can call this concurrently
    std::optional<int> get(const std::string& key) const {
        std::shared_lock lock(mtx_);  // shared (read) lock
        auto it = cache_.find(key);
        if (it != cache_.end()) return it->second;
        return std::nullopt;
    }

    // Only one writer at a time; blocks all readers
    void put(const std::string& key, int value) {
        std::unique_lock lock(mtx_);  // exclusive (write) lock
        cache_[key] = value;
    }
};

    ▶ Reader-Writer Lock

    Multiple readers can proceed simultaneously, but a writer gets exclusive access.

    std::call_once: Thread-Safe Initialization

    Sometimes you need to initialize something exactly once, even if multiple threads try to initialize it concurrently. std::call_once with std::once_flag guarantees this:

    std::once_flag init_flag;
std::shared_ptr<Database> db;

void init_database() {
    db = std::make_shared<Database>("connstr");
}

std::shared_ptr<Database> get_db() {
    std::call_once(init_flag, init_database);
    return db;
}

// No matter how many threads call get_db() concurrently,
// init_database() runs exactly once. All other threads
// block until initialization is complete.

    This is safer than the classic double-checked locking pattern, which is notoriously tricky to get right (and was broken in C++ before the memory model was introduced in C++11).

    Common Mutex Pitfalls

    Lock Granularity

    A coarse-grained lock (one mutex for the entire data structure) is simple but limits parallelism. A fine-grained lock (one mutex per node/bucket) allows more parallelism but is harder to get right and may cause deadlocks.

    // Coarse-grained: simple but bottleneck
class CoarseMap {
    std::mutex mtx_;
    std::unordered_map<int, int> map_;
public:
    void insert(int k, int v) {
        std::lock_guard lock(mtx_);  // all operations serialized
        map_[k] = v;
    }
};

// Fine-grained: more parallelism, more complexity
class FineMap {
    static constexpr int NumBuckets = 64;
    struct Bucket {
        std::mutex mtx;
        std::list<std::pair<int,int>> data;
    };
    std::array<Bucket, NumBuckets> buckets_;

    Bucket& get_bucket(int key) {
        return buckets_[std::hash<int>{}(key) % NumBuckets];
    }

public:
    void insert(int k, int v) {
        auto& b = get_bucket(k);
        std::lock_guard lock(b.mtx);  // only this bucket is locked
        b.data.emplace_back(k, v);
    }
};

    Never Return References to Protected Data

    If a locked function returns a reference or pointer to the protected data, callers can access it without the lock:

    class Broken {
    std::mutex mtx_;
    int data_ = 0;

public:
    int& get_data() {
        std::lock_guard lock(mtx_);
        return data_;   // BUG: reference escapes the lock!
    }
};

// Caller can hold the reference and use it without the mutex
int& ref = broken.get_data();
ref = 42;  // data race! No lock held.

    Fix: Return by value, or require the caller to pass a callback that executes under the lock.

    Thread-Unsafe Interfaces

    Even if every individual method is locked, the interface can enable race conditions:

    class ThreadSafeStack {
    std::mutex mtx_;
    std::stack<int> s_;

public:
    bool empty() const { std::lock_guard lock(mtx_); return s_.empty(); }
    int top() const { std::lock_guard lock(mtx_); return s_.top(); }
    void pop() { std::lock_guard lock(mtx_); s_.pop(); }
};

// Race condition between empty() and top():
if (!stack.empty()) {       // Thread A checks
    // Thread B pops the last element here!
    int val = stack.top();  // Thread A crashes: stack is empty
}

    Fix: Combine check + access into a single atomic operation:

    class SafeStack {
    std::mutex mtx_;
    std::stack<int> s_;

public:
    bool try_pop(int& value) {
        std::lock_guard lock(mtx_);
        if (s_.empty()) return false;
        value = s_.top();
        s_.pop();
        return true;  // check + top + pop: all under one lock
    }
};

    Performance Considerations

    • Uncontended lock: On modern hardware, locking an uncontended mutex is ~20-50 nanoseconds. It is not free, but it is fast.
    • Contended lock: When threads compete for the same mutex, they block and context-switch. This is expensive (microseconds+). High contention destroys performance.
    • Lock-free alternatives: For simple operations (counters, flags), std::atomic avoids mutex overhead entirely. See Atomic Operations.
    • Minimize critical section size: Do as little work as possible while holding the lock.
    • Use shared_mutex for read-heavy workloads: Multiple readers proceed in parallel.

    Lock Wrapper Comparison

    Wrapper Deferred Try Lock Manual Unlock Multi-Mutex Movable
    lock_guardNoNoNoNoNo
    unique_lockYesYesYesNoYes
    scoped_lockNoNoNoYesNo
    shared_lockYesYesYesNoYes

    Rule of thumb: Use lock_guard for single-mutex, scope-based locking. Use scoped_lock when locking multiple mutexes. Use unique_lock when you need deferred/timed locking or condition variables. Use shared_lock with shared_mutex for read locks.

    Summary

    • A mutex provides mutual exclusion: only one thread can hold it at a time.
    • Always use RAII wrappers (lock_guard, unique_lock, scoped_lock) instead of raw lock()/unlock().
    • scoped_lock (C++17) locks multiple mutexes atomically, preventing deadlocks.
    • shared_mutex allows concurrent readers with exclusive writers.
    • Minimize critical section size. Never return references to protected data.
    • Design thread-safe interfaces that combine check-and-act into single locked operations.
    • std::call_once handles thread-safe one-time initialization.

    Using mutexes correctly prevents data races, but introduces a new problem: Deadlocks. The next post covers what deadlocks are and how to prevent them.
