    The ABA Problem

    March 2026

    The ABA problem is the most subtle correctness issue in lock-free programming. It arises whenever a compare-and-swap (CAS) operation checks pointer equality, but the pointed-to object has been freed and a new object happens to be allocated at the same address. CAS succeeds even though the logical state has changed.

    A Concrete Example

    Consider a Treiber stack with three nodes. head points to A, which points to B, which points to C.

    head --> A --> B --> C --> null

    Thread 1 starts a pop(). It reads old_head = A and next = B. Now it is about to execute:

    // Thread 1 is here, about to CAS
head_.compare_exchange_weak(old_head /* A */, next /* B */)

    Before Thread 1 executes the CAS, it gets preempted. Meanwhile:

    1. Thread 2 pops A. Stack: head --> B --> C.
    2. Thread 2 pops B. Stack: head --> C.
    3. Thread 2 frees both A and B.
    4. Thread 2 pushes a new node D. The allocator reuses the memory that A occupied. D has the same address as the old A. Stack: head --> D --> C.

    Thread 1 resumes. It executes CAS(head_, A, B). It checks: does head_ still equal A? By raw address, yes, because D sits at the same address. The CAS succeeds and sets head_ = B. But B has already been freed. The stack is now corrupted.

    Root cause. CAS compares bit patterns (addresses). It cannot detect that the object at that address has been replaced by a different logical entity. The value went A to B to A again (hence "ABA"), and CAS cannot distinguish the second A from the first.

    Why CAS Fails to Detect ABA

    CAS compares the bit pattern stored in the atomic variable against an expected value. On most architectures this is a pointer-width comparison (8 bytes on 64-bit). It has no way to know whether the object at that address is the same logical object or a recycled one. The comparison is purely syntactic, not semantic.

    This would not be a problem if freed memory were never reused, but that is not a practical option. Allocators routinely hand out recently freed addresses, making ABA a real concern in any lock-free structure that frees nodes.

    Solution 1: Tagged Pointers (Double-Width CAS)

    The idea: pack a monotonically increasing counter alongside the pointer. Each time the pointer is updated, increment the counter. CAS now operates on the (pointer, counter) pair, so even if the address is reused, the counter will differ.

    struct TaggedPtr {
    Node* ptr;
    std::size_t tag;  // generation counter
};

// Requires 16-byte (128-bit) CAS on 64-bit platforms.
// GCC/Clang: use __int128 or compiler builtins.
// MSVC: use _InterlockedCompareExchange128.
std::atomic<TaggedPtr> head_;

    On x86-64, the CMPXCHG16B instruction performs a 128-bit CAS. The push and pop change slightly:

    void push(T val) {
    Node* node = new Node(std::move(val));
    TaggedPtr old = head_.load(std::memory_order_relaxed);

    TaggedPtr desired;
    do {
        node->next = old.ptr;
        desired = {node, old.tag + 1};
    } while (!head_.compare_exchange_weak(
                old, desired,
                std::memory_order_release,
                std::memory_order_relaxed));
}

bool pop(T& result) {
    TaggedPtr old = head_.load(std::memory_order_relaxed);

    while (old.ptr != nullptr) {
        Node* next = old.ptr->next;
        TaggedPtr desired = {next, old.tag + 1};

        if (head_.compare_exchange_weak(
                old, desired,
                std::memory_order_acquire,
                std::memory_order_relaxed))
        {
            result = std::move(old.ptr->data);
            delete old.ptr;
            return true;
        }
    }
    return false;
}
    Limitation. The counter is finite. After 264 operations the tag wraps around and ABA can recur. In practice, wrapping a 64-bit counter takes hundreds of years at billions of operations per second, so this is safe.

    Solution 2: Generation Counters

    Instead of tagging the pointer, embed a generation counter inside each node. Before freeing a node, increment its generation. When performing CAS, compare both the pointer and the stored generation value.

    struct Node {
    T data;
    Node* next;
    std::atomic<uint32_t> generation{0};
};

// In pop: after successful CAS, before freeing
old_head->generation.fetch_add(1, std::memory_order_relaxed);
// Return to free-list instead of deleting
free_list.push(old_head);

    This is conceptually similar to tagged pointers but avoids the need for double-width CAS. The trade-off is that you must use a custom allocator (free-list) that preserves nodes across reuse, and each CAS loop must read and verify the generation count.

    Solution 3: Hazard Pointers (Overview)

    Hazard pointers address the root cause: CAS fails because another thread frees a node while someone else is still reading it. Hazard pointers prevent that by requiring each thread to announce which nodes it is accessing. A thread that wants to free a node checks all announcements first.

    The protocol in brief:

    1. Announce: before dereferencing a pointer, write it to your thread-local hazard slot.
    2. Validate: re-read the atomic to confirm it has not changed since you announced it. If it changed, update the announcement and retry.
    3. Use: safely dereference the pointer, knowing no other thread will free it.
    4. Clear: after you are done, clear your hazard slot.
    5. Reclaim: when a thread retires a node, it adds it to a local "retired" list. Periodically, it scans all hazard slots. Any retired node not found in any hazard slot can be safely freed.
    // Simplified hazard pointer for pop
bool pop(T& result) {
    Node* old_head;
    do {
        old_head = head_.load(std::memory_order_relaxed);
        if (old_head == nullptr) return false;

        // ANNOUNCE: tell other threads we are reading old_head
        hp_[thread_id].store(old_head, std::memory_order_release);

        // VALIDATE: check head_ has not changed
        if (head_.load(std::memory_order_acquire) != old_head)
            continue; // changed; retry

        Node* next = old_head->next;
        if (head_.compare_exchange_weak(
                old_head, next,
                std::memory_order_acquire,
                std::memory_order_relaxed))
        {
            // CLEAR hazard
            hp_[thread_id].store(nullptr, std::memory_order_release);

            result = std::move(old_head->data);
            // Do NOT delete immediately; retire instead
            retire(old_head);
            return true;
        }
    } while (true);

    hp_[thread_id].store(nullptr, std::memory_order_release);
    return false;
}

    See Hazard Pointers (detailed) for a full implementation.

    Solution 4: Epoch-Based Reclamation

    Epoch-based reclamation (EBR) divides time into epochs (typically numbered 0, 1, 2, cycling). Each thread entering a critical section records the current global epoch. Retired nodes are tagged with the epoch in which they were retired. A node can be freed when all threads have advanced past that epoch.

    class EpochManager {
    std::atomic<uint64_t> global_epoch_{0};
    struct ThreadState {
        std::atomic<uint64_t> local_epoch{0};
        std::atomic<bool> active{false};
        std::vector<Node*> retired[3]; // one per epoch slot
    };
    ThreadState threads_[MAX_THREADS];

public:
    void enter(int tid) {
        threads_[tid].active.store(true, std::memory_order_relaxed);
        threads_[tid].local_epoch.store(
            global_epoch_.load(std::memory_order_relaxed),
            std::memory_order_release);
    }

    void exit(int tid) {
        threads_[tid].active.store(false, std::memory_order_release);
    }

    void retire(int tid, Node* node) {
        uint64_t e = global_epoch_.load(std::memory_order_relaxed);
        threads_[tid].retired[e % 3].push_back(node);
    }

    void try_advance() {
        uint64_t e = global_epoch_.load(std::memory_order_relaxed);
        // Can advance if all active threads are in the current epoch
        for (int i = 0; i < MAX_THREADS; ++i) {
            if (threads_[i].active.load(std::memory_order_acquire) &&
                threads_[i].local_epoch.load(std::memory_order_acquire) != e)
                return; // some thread is still in an old epoch
        }
        // Safe to advance
        uint64_t new_e = e + 1;
        global_epoch_.store(new_e, std::memory_order_release);
        // Free nodes from two epochs ago
        uint64_t safe = (new_e + 1) % 3; // the oldest slot
        for (int i = 0; i < MAX_THREADS; ++i) {
            for (Node* n : threads_[i].retired[safe])
                delete n;
            threads_[i].retired[safe].clear();
        }
    }
};
    EBR vs. hazard pointers. EBR has lower per-access overhead (no per-pointer announcement) but can delay reclamation if a slow thread stays in an old epoch. Hazard pointers reclaim more promptly but require a per-pointer announcement and a scan of all hazard slots.

    When Does ABA Actually Bite?

    ABA requires three conditions to coincide:

    1. A thread reads a pointer and is preempted before it completes its CAS.
    2. Other threads free the pointed-to node and the allocator reuses the address.
    3. The preempted thread resumes and its CAS succeeds on the recycled address.

    All three must happen in the same window. Under light load this is rare. Under heavy load with aggressive memory reuse it becomes probable. You cannot rely on "it almost never happens" in production systems: either prove your structure is ABA-immune or use one of the solutions above.

    Summary

    The ABA problem exists because CAS compares raw bit patterns, not logical identity. It corrupts lock-free structures by allowing a CAS to succeed when the pointer has been recycled. Tagged pointers solve it cheaply with double-width CAS. Hazard pointers and epoch-based reclamation solve it more generally by preventing premature freeing. Any production lock-free data structure must address ABA explicitly.
