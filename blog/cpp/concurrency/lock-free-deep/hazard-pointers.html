    Hazard Pointers

    March 2026

    Lock-free data structures face a fundamental question: when is it safe to free a node that has been removed from the structure? The node might still be in use by a concurrent thread that read a pointer to it just before the removal. Hazard pointers, introduced by Maged Michael in 2004, provide a rigorous answer.

    The Problem: When to Free Nodes

    Consider a Treiber stack pop:

    Node* old = head_.load();     // (1) Thread A reads head
Node* next = old->next;       // (2) Thread A reads old->next
// Thread B pops and frees old here
head_.CAS(old, next);         // (3) Thread A CAS: old is freed!

    Between lines (1) and (3), Thread B may pop and free the same node. Line (2) then dereferences a freed pointer, which is undefined behaviour. Even if the CAS on line (3) fails (because head_ changed), the damage is done at line (2).

    The same problem arises in every lock-free structure: the Michael-Scott queue, skip lists, hash tables. Any structure where one thread removes a node while another might still be traversing it needs a safe reclamation scheme.

    The Hazard Pointer Protocol

    The key insight: if a thread is about to dereference a pointer, it first publishes that pointer in a globally visible location (a hazard slot). Other threads must check all hazard slots before freeing a node.

    Announce

    Before dereferencing a shared pointer, write it to your thread-local hazard slot:

    // Per-thread hazard pointer array (typically 1-2 slots per thread)
std::atomic<void*> hazard_slots[MAX_THREADS][MAX_HP_PER_THREAD];

void announce(int tid, int slot, void* ptr) {
    hazard_slots[tid][slot].store(ptr, std::memory_order_release);
}

    Validate

    After announcing, re-read the atomic variable to confirm it has not changed. If it has, the announced pointer might be stale; update and retry.

    // Example in pop
Node* old_head;
do {
    old_head = head_.load(std::memory_order_relaxed);
    if (!old_head) return false;

    // Announce
    announce(tid, 0, old_head);

    // Validate: did head_ change while we were announcing?
    if (head_.load(std::memory_order_acquire) != old_head)
        continue; // stale; retry

    // Safe to dereference old_head->next
    Node* next = old_head->next;

    if (head_.compare_exchange_weak(old_head, next,
            std::memory_order_acquire, std::memory_order_relaxed))
    {
        T val = std::move(old_head->data);
        // Clear hazard before retiring
        announce(tid, 0, nullptr);
        retire(tid, old_head);
        return val;
    }
} while (true);
    Why validate? Between the initial load and the announce, another thread could have removed the node. The announce does not prevent removal; it only prevents freeing. But if the pointer is already stale, dereferencing it is still unsafe (on non-x86 architectures that reorder loads). The validation ensures the announced pointer is current.

    Scan and Reclaim

    Each thread maintains a local list of retired nodes. Periodically (e.g., when the retired list exceeds a threshold), the thread scans all hazard slots. Any retired node not found in any slot can be safely freed.

    struct RetiredNode {
    void* ptr;
    std::function<void(void*)> deleter;
};

thread_local std::vector<RetiredNode> retired_list;

void retire(int tid, Node* node) {
    retired_list.push_back({node, [](void* p){ delete static_cast<Node*>(p); }});

    if (retired_list.size() >= SCAN_THRESHOLD) {
        scan(tid);
    }
}

void scan(int tid) {
    // Step 1: collect all hazard pointers into a set
    std::unordered_set<void*> protected_ptrs;
    for (int t = 0; t < MAX_THREADS; ++t) {
        for (int s = 0; s < MAX_HP_PER_THREAD; ++s) {
            void* hp = hazard_slots[t][s].load(std::memory_order_acquire);
            if (hp) protected_ptrs.insert(hp);
        }
    }

    // Step 2: partition retired list into reclaimable and still-protected
    std::vector<RetiredNode> still_retired;
    for (auto& r : retired_list) {
        if (protected_ptrs.count(r.ptr)) {
            still_retired.push_back(r);
        } else {
            r.deleter(r.ptr); // safe to free
        }
    }
    retired_list = std::move(still_retired);
}

    Full Implementation Sketch

    Below is a complete hazard-pointer-protected Treiber stack.

    #include <atomic>
#include <functional>
#include <unordered_set>
#include <vector>
#include <optional>

static constexpr int MAX_THREADS = 64;
static constexpr int HP_PER_THREAD = 2;
static constexpr int SCAN_THRESHOLD = 128;

class HazardPointerDomain {
    std::atomic<void*> hp_[MAX_THREADS][HP_PER_THREAD]{};

    struct Retired {
        void* ptr;
        std::function<void(void*)> deleter;
    };

    // Per-thread retired lists (in production, use thread_local or
    // a thread registry; simplified here with an array).
    std::vector<Retired> retired_[MAX_THREADS];

public:
    void protect(int tid, int slot, void* ptr) {
        hp_[tid][slot].store(ptr, std::memory_order_release);
    }

    void clear(int tid, int slot) {
        hp_[tid][slot].store(nullptr, std::memory_order_release);
    }

    template <typename T>
    void retire(int tid, T* ptr) {
        retired_[tid].push_back({ptr, [](void* p){
            delete static_cast<T*>(p);
        }});
        if (static_cast<int>(retired_[tid].size()) >= SCAN_THRESHOLD) {
            scan(tid);
        }
    }

    void scan(int tid) {
        std::unordered_set<void*> protected_set;
        for (int t = 0; t < MAX_THREADS; ++t) {
            for (int s = 0; s < HP_PER_THREAD; ++s) {
                void* p = hp_[t][s].load(std::memory_order_acquire);
                if (p) protected_set.insert(p);
            }
        }

        std::vector<Retired> remaining;
        for (auto& r : retired_[tid]) {
            if (protected_set.count(r.ptr)) {
                remaining.push_back(std::move(r));
            } else {
                r.deleter(r.ptr);
            }
        }
        retired_[tid] = std::move(remaining);
    }
};

// Global domain
static HazardPointerDomain g_hp;

template <typename T>
class HPTreiberStack {
    struct Node {
        T data;
        Node* next;
        template <typename U>
        explicit Node(U&& val) : data(std::forward<U>(val)), next(nullptr) {}
    };

    std::atomic<Node*> head_{nullptr};

public:
    void push(T val) {
        Node* node = new Node(std::move(val));
        node->next = head_.load(std::memory_order_relaxed);
        while (!head_.compare_exchange_weak(
                    node->next, node,
                    std::memory_order_release,
                    std::memory_order_relaxed))
        { /* retry */ }
    }

    std::optional<T> pop(int tid) {
        Node* old_head;
        do {
            old_head = head_.load(std::memory_order_relaxed);
            if (!old_head) return std::nullopt;

            // Announce
            g_hp.protect(tid, 0, old_head);

            // Validate
            if (head_.load(std::memory_order_acquire) != old_head)
                continue;

            Node* next = old_head->next;

            if (head_.compare_exchange_weak(
                    old_head, next,
                    std::memory_order_acquire,
                    std::memory_order_relaxed))
            {
                T val = std::move(old_head->data);
                g_hp.clear(tid, 0);
                g_hp.retire(tid, old_head);
                return val;
            }
        } while (true);
    }

    ~HPTreiberStack() {
        Node* cur = head_.load(std::memory_order_relaxed);
        while (cur) {
            Node* next = cur->next;
            delete cur;
            cur = next;
        }
    }
};

    Correctness Argument

    We need to show that no node is freed while any thread holds a reference to it.

    1. Announce-before-dereference: a thread publishes a hazard pointer before it reads node->next. The release store in protect() ensures the announcement is visible.
    2. Validation: after announcing, the thread re-reads the source pointer. If it has changed, the announcement is stale and the thread retries. If it has not changed, the node is guaranteed to still be in the structure (no other thread has completed a CAS to remove it between the initial load and the validation load, because if it had, the validation would see the new value).
    3. Scan checks all slots: a thread only frees a retired node if no hazard slot contains its address. The acquire loads in scan() synchronise with the release stores in protect(), ensuring the freeing thread sees all outstanding announcements.

    The defer_reclamation Pattern

    In many codebases, the retire-and-scan logic is abstracted behind a defer_reclamation function:

    template <typename T>
void defer_reclamation(int tid, T* ptr) {
    g_hp.retire(tid, ptr);
    // Optionally trigger a scan if the retired list is large
}

// Usage in pop:
defer_reclamation(tid, old_head);

    This pattern separates the lock-free algorithm from the reclamation mechanism. You can swap in hazard pointers, epoch-based reclamation, or reference counting behind the same interface.

    // Generic reclamation interface
class ReclaimStrategy {
public:
    virtual void enter(int tid) = 0;
    virtual void exit(int tid) = 0;
    virtual void protect(int tid, int slot, void* ptr) = 0;
    virtual void clear(int tid, int slot) = 0;
    virtual void retire(int tid, void* ptr,
                        std::function<void(void*)> deleter) = 0;
    virtual ~ReclaimStrategy() = default;
};

    Comparison: Hazard Pointers vs. Epoch-Based Reclamation

    PropertyHazard PointersEpoch-Based Reclamation
    Per-access overhead1 store + 1 load (announce + validate) per pointer1 store on enter, 1 store on exit (amortised across many accesses)
    Reclamation latencyBounded by scan thresholdUnbounded if a thread stalls in an old epoch
    Memory usageO(T * H + T * R) where T=threads, H=hazard slots, R=retired-list sizeO(T * R) but R can grow unboundedly
    Robustness to slow threadsExcellent: slow threads only protect the nodes they have announcedPoor: a single stalled thread prevents all reclamation
    Implementation complexityHigher (per-pointer protocol)Lower (enter/exit bookkeeping)
    Lock-free guarantee preservedYes (scan is O(T*H))Yes, but stalled threads block reclamation
    When to choose which. Use hazard pointers when stalled threads are a real concern (e.g., threads may be preempted for long periods). Use epoch-based reclamation when access patterns are well-behaved (no thread holds a critical section for an extended time) and you want lower per-access overhead.

    C++26 and std::hazard_pointer

    The C++ standard is on track to include std::hazard_pointer (P2530). The interface looks roughly like:

    // C++26 (proposed)
#include <hazard_pointer>

template <typename T>
class HPStack {
    struct Node : std::hazard_pointer_obj_base<Node> {
        T data;
        Node* next;
    };

    std::atomic<Node*> head_{nullptr};

public:
    void push(T val) {
        auto* node = new Node{{}, std::move(val), nullptr};
        node->next = head_.load(std::memory_order_relaxed);
        while (!head_.compare_exchange_weak(
                    node->next, node,
                    std::memory_order_release,
                    std::memory_order_relaxed))
        {}
    }

    std::optional<T> pop() {
        std::hazard_pointer hp = std::make_hazard_pointer();
        Node* old_head;

        do {
            old_head = hp.protect(head_);
            if (!old_head) return std::nullopt;

            if (head_.compare_exchange_weak(
                    old_head, old_head->next,
                    std::memory_order_acquire,
                    std::memory_order_relaxed))
            {
                T val = std::move(old_head->data);
                hp.reset_protection();
                old_head->retire(); // adds to retired list
                return val;
            }
        } while (true);
    }
};

    The standard interface handles thread registration, hazard slot allocation, and scanning automatically. hazard_pointer_obj_base provides the retire() method, and the library manages the retired lists and reclamation timing.

    Performance Considerations

    Cost of the Announce/Validate Round-Trip

    Each announce is an atomic store (a release fence on ARM). Each validate is an atomic load (an acquire fence). On x86 these are cheap (just compiler fences), but on ARM or POWER they translate to real fence instructions. For hot paths, the overhead can be significant.

    Scan Cost

    Scanning all hazard slots is O(T * H) where T is the number of threads and H is the number of hazard slots per thread. Building a hash set from the hazard pointers takes O(T * H) time. Comparing each retired node takes O(R) time. The total scan cost is O(T * H + R), amortised over R retirements.

    Cache Effects

    Hazard slot arrays should be arranged so that each thread's slots are on their own cache line to avoid false sharing during announces. The scan function reads all slots, which pulls multiple cache lines; this is acceptable because scans are infrequent (once per SCAN_THRESHOLD retirements).

    struct alignas(64) HazardSlot {
    std::atomic<void*> ptr[HP_PER_THREAD]{};
};
HazardSlot hazard_slots[MAX_THREADS];

    Common Pitfalls

    1. Forgetting to validate: announcing a pointer without re-checking the source atomic leaves a window where the pointer could already be stale.
    2. Clearing too late: keeping a hazard pointer active longer than necessary delays reclamation of other nodes.
    3. Wrong scan threshold: too low and you scan often (wasting CPU); too high and retired lists grow (wasting memory). A common heuristic is threshold = 2 * T * H.
    4. Missing fence: the announce must be a release store and the scan must use acquire loads. Without this, the freeing thread might not see the announcement.

    Summary

    Hazard pointers solve the memory reclamation problem in lock-free structures by requiring threads to announce which nodes they are accessing. A retiring thread consults all announcements before freeing a node. The protocol is announce, validate, use, clear, retire, and periodically scan. Compared to epoch-based reclamation, hazard pointers are more robust to stalled threads but have higher per-access overhead. C++26 is expected to standardise the interface, making safe lock-free programming more accessible.
