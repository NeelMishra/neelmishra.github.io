    Lock-Free Queue

    March 2026

    Queues are harder to make lock-free than stacks because they have two ends: enqueue touches the tail and dequeue touches the head. Coordinating two atomic pointers without a lock requires more care. This article covers three increasingly practical designs.

    Single-Producer Single-Consumer Ring Buffer

    When exactly one thread produces and one consumes, the problem simplifies dramatically. A fixed-size circular buffer with two atomic indices suffices. No CAS is needed; plain loads and stores with the right memory ordering are enough.

    template <typename T, std::size_t Capacity>
class SPSCQueue {
    static_assert((Capacity & (Capacity - 1)) == 0,
                  "Capacity must be a power of 2");

    std::array<T, Capacity> buffer_;
    alignas(64) std::atomic<std::size_t> head_{0}; // read position
    alignas(64) std::atomic<std::size_t> tail_{0}; // write position

public:
    bool try_push(const T& val) {
        const std::size_t t = tail_.load(std::memory_order_relaxed);
        const std::size_t next_t = (t + 1) & (Capacity - 1);

        // If next write position equals head, the queue is full.
        if (next_t == head_.load(std::memory_order_acquire)) {
            return false;
        }

        buffer_[t] = val;
        tail_.store(next_t, std::memory_order_release);
        return true;
    }

    bool try_pop(T& result) {
        const std::size_t h = head_.load(std::memory_order_relaxed);

        // If head equals tail, the queue is empty.
        if (h == tail_.load(std::memory_order_acquire)) {
            return false;
        }

        result = buffer_[h];
        head_.store((h + 1) & (Capacity - 1), std::memory_order_release);
        return true;
    }
};
    Why power of 2? The mask (Capacity - 1) replaces the modulo operator, which is critical on hot paths. The bitwise AND compiles to a single instruction.

    Memory Ordering Analysis

    AccessOrderingRationale
    Producer reads head_acquireSynchronises with consumer's release store, ensuring we see the consumer's update before overwriting the slot.
    Producer stores tail_releaseEnsures the write to buffer_[t] is visible to the consumer before the index advances.
    Consumer reads tail_acquireSynchronises with producer's release store, ensuring we see the data the producer wrote.
    Consumer stores head_releaseEnsures the read from buffer_[h] completes before the producer sees the freed slot.

    The alignas(64) on head_ and tail_ avoids false sharing. Without it, both indices could sit in the same cache line, causing needless cross-core invalidation.

    Michael-Scott Queue

    The Michael-Scott (MS) queue, published in 1996, is the classic lock-free FIFO for multiple producers and multiple consumers. It uses a linked list with a sentinel (dummy) node. Two atomic pointers track the head and tail.

    Structure

    template <typename T>
class MSQueue {
    struct Node {
        T data;
        std::atomic<Node*> next{nullptr};
        Node() = default;
        explicit Node(T val) : data(std::move(val)) {}
    };

    alignas(64) std::atomic<Node*> head_;
    alignas(64) std::atomic<Node*> tail_;

public:
    MSQueue() {
        Node* sentinel = new Node();
        head_.store(sentinel, std::memory_order_relaxed);
        tail_.store(sentinel, std::memory_order_relaxed);
    }

    void enqueue(T val);
    bool dequeue(T& result);
    ~MSQueue();
};

    Enqueue

    Enqueue appends a new node after the current tail. Two CAS operations may be needed: one to link the new node, and one to advance the tail pointer. The subtle insight is that the tail pointer is allowed to lag behind the actual last node.

    template <typename T>
void MSQueue<T>::enqueue(T val) {
    Node* node = new Node(std::move(val));

    while (true) {
        Node* tail = tail_.load(std::memory_order_acquire);
        Node* next = tail->next.load(std::memory_order_acquire);

        // Is tail still the actual tail?
        if (tail == tail_.load(std::memory_order_acquire)) {
            if (next == nullptr) {
                // tail->next is null: try to link the new node
                if (tail->next.compare_exchange_weak(
                        next, node,
                        std::memory_order_release,
                        std::memory_order_relaxed))
                {
                    // Linked. Now try to advance tail_.
                    // If this fails, another thread will help.
                    tail_.compare_exchange_strong(
                        tail, node,
                        std::memory_order_release,
                        std::memory_order_relaxed);
                    return;
                }
            } else {
                // tail is lagging; help advance it
                tail_.compare_exchange_weak(
                    tail, next,
                    std::memory_order_release,
                    std::memory_order_relaxed);
            }
        }
    }
}
    Helping mechanism. If thread A links a node but is preempted before it advances tail_, any thread that calls enqueue or dequeue will observe that tail->next != nullptr and help advance tail_. This is what makes the algorithm lock-free rather than just obstruction-free.

    Dequeue

    template <typename T>
bool MSQueue<T>::dequeue(T& result) {
    while (true) {
        Node* head = head_.load(std::memory_order_acquire);
        Node* tail = tail_.load(std::memory_order_acquire);
        Node* next = head->next.load(std::memory_order_acquire);

        if (head == head_.load(std::memory_order_acquire)) {
            if (head == tail) {
                if (next == nullptr) {
                    // Queue is truly empty
                    return false;
                }
                // Tail is lagging; help advance
                tail_.compare_exchange_weak(
                    tail, next,
                    std::memory_order_release,
                    std::memory_order_relaxed);
            } else {
                // Read value before CAS; otherwise another
                // dequeue might free the node.
                result = next->data;
                if (head_.compare_exchange_weak(
                        head, next,
                        std::memory_order_release,
                        std::memory_order_relaxed))
                {
                    // Success: head retired. Free old sentinel.
                    // CAUTION: same ABA/reclamation issues apply.
                    delete head;
                    return true;
                }
            }
        }
    }
}

    Destructor

    template <typename T>
MSQueue<T>::~MSQueue() {
    Node* cur = head_.load(std::memory_order_relaxed);
    while (cur) {
        Node* next = cur->next.load(std::memory_order_relaxed);
        delete cur;
        cur = next;
    }
}

    Linearisation Points

    OperationLinearisation point
    EnqueueSuccessful CAS on tail->next (linking the node)
    Dequeue (non-empty)Successful CAS on head_
    Dequeue (empty)Read of head->next == nullptr when head == tail

    Multi-Producer Multi-Consumer Challenges

    The MS queue is MPMC, but in practice several issues arise:

    1. Memory Reclamation

    The delete head in dequeue is unsafe for the same reasons as in the Treiber stack. A concurrent dequeue may still be reading head->next when another thread frees head. Solutions:

    • Hazard pointers: each thread announces which nodes it is currently accessing. A thread that wants to free a node first checks all announcements.
    • Epoch-based reclamation (EBR): threads enter a "critical section" by advancing a per-thread epoch counter. Nodes are retired and freed only when all threads have advanced past the epoch in which the node was retired.
    • Reference counting: each node has an atomic reference count. This adds overhead to every access but is straightforward.

    2. Contention on Head and Tail

    Under high load, both enqueue and dequeue hammer their respective atomic pointers. Backoff strategies help:

    // Exponential backoff after a failed CAS
struct Backoff {
    unsigned delay = 1;
    static constexpr unsigned MAX_DELAY = 1024;

    void operator()() {
        for (unsigned i = 0; i < delay; ++i) {
            // Pause hint to the CPU (x86)
            #if defined(__x86_64__) || defined(_M_X64)
            _mm_pause();
            #else
            std::this_thread::yield();
            #endif
        }
        delay = std::min(delay * 2, MAX_DELAY);
    }

    void reset() { delay = 1; }
};

    3. False Sharing

    Placing head_ and tail_ on separate cache lines (via alignas(64)) is essential. Without this, a store to tail_ by an enqueuer invalidates the consumer's cache line containing head_, even though they are logically independent.

    Bounded MPMC Ring Buffer

    For bounded queues with multiple producers and consumers, a popular approach uses a ring buffer where each slot contains a sequence counter. This design avoids linked-list allocation entirely.

    template <typename T, std::size_t Capacity>
class BoundedMPMCQueue {
    struct Cell {
        std::atomic<std::size_t> sequence;
        T data;
    };

    alignas(64) Cell buffer_[Capacity];
    alignas(64) std::atomic<std::size_t> enqueue_pos_{0};
    alignas(64) std::atomic<std::size_t> dequeue_pos_{0};

public:
    BoundedMPMCQueue() {
        for (std::size_t i = 0; i < Capacity; ++i)
            buffer_[i].sequence.store(i, std::memory_order_relaxed);
    }

    bool try_push(T val) {
        Cell* cell;
        std::size_t pos = enqueue_pos_.load(std::memory_order_relaxed);

        while (true) {
            cell = &buffer_[pos % Capacity];
            std::size_t seq = cell->sequence.load(std::memory_order_acquire);
            intptr_t diff = static_cast<intptr_t>(seq) -
                            static_cast<intptr_t>(pos);

            if (diff == 0) {
                // Slot is ready for writing
                if (enqueue_pos_.compare_exchange_weak(
                        pos, pos + 1, std::memory_order_relaxed))
                    break;
            } else if (diff < 0) {
                // Queue is full
                return false;
            } else {
                // Another enqueuer took this slot; reload
                pos = enqueue_pos_.load(std::memory_order_relaxed);
            }
        }

        cell->data = std::move(val);
        cell->sequence.store(pos + 1, std::memory_order_release);
        return true;
    }

    bool try_pop(T& result) {
        Cell* cell;
        std::size_t pos = dequeue_pos_.load(std::memory_order_relaxed);

        while (true) {
            cell = &buffer_[pos % Capacity];
            std::size_t seq = cell->sequence.load(std::memory_order_acquire);
            intptr_t diff = static_cast<intptr_t>(seq) -
                            static_cast<intptr_t>(pos + 1);

            if (diff == 0) {
                if (dequeue_pos_.compare_exchange_weak(
                        pos, pos + 1, std::memory_order_relaxed))
                    break;
            } else if (diff < 0) {
                return false; // empty
            } else {
                pos = dequeue_pos_.load(std::memory_order_relaxed);
            }
        }

        result = std::move(cell->data);
        cell->sequence.store(pos + Capacity, std::memory_order_release);
        return true;
    }
};
    Sequence trick. Each cell's sequence counter acts as a per-slot state machine. When seq == pos, the slot is free for writing. When seq == pos + 1, it is ready for reading. After reading, the consumer sets seq = pos + Capacity, marking the slot free for the next cycle.

    Practical Tips

    • If you have exactly one producer and one consumer, always prefer the SPSC ring buffer. It is dramatically simpler and faster (no CAS at all).
    • For MPMC with bounded size, the sequence-counter ring buffer outperforms the MS queue by avoiding allocation.
    • The MS queue is best when the queue size is unpredictable and you need an unbounded structure.
    • Profile before committing to lock-free. Under low contention, a simple std::mutex with a std::deque can be faster due to lower constant factors.

    Summary

    Lock-free queues are more complex than lock-free stacks because they manage two ends simultaneously. The SPSC ring buffer eliminates CAS entirely by relying on the single-writer principle. The Michael-Scott queue introduces a helping mechanism to ensure lock-freedom with multiple threads on both ends. The bounded MPMC ring buffer avoids dynamic allocation by using sequence counters as per-slot state machines. All three designs require careful memory ordering analysis to be correct.
