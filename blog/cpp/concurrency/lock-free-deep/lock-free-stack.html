    Lock-Free Stack

    March 2026

    A lock-free stack is the canonical introduction to lock-free data structures. It relies on a single atomic pointer to the head of a singly linked list and uses compare_exchange_weak (CAS) to swing that pointer atomically. The resulting structure is known as the Treiber stack, first described by R. Kent Treiber in 1986.

    Why Lock-Free?

    A mutex-based stack serialises every push and pop behind a single lock. Under contention this creates two problems: threads that hold the lock can be preempted, stalling every other thread; and the OS scheduler has no idea that the waiting threads are actually ready to make progress the moment the lock is released.

    A lock-free algorithm guarantees that at least one thread makes progress in a finite number of steps, regardless of the scheduling decisions the OS makes. This does not mean every thread makes progress (that would be wait-free); it means the system as a whole cannot be stuck.

    Node Layout

    Each node stores a value and a pointer to the next node:

    template <typename T>
struct Node {
    T data;
    Node* next;

    explicit Node(const T& val) : data(val), next(nullptr) {}
    explicit Node(T&& val) : data(std::move(val)), next(nullptr) {}
};

    The stack itself holds an atomic pointer to the top node:

    template <typename T>
class LockFreeStack {
    std::atomic<Node<T>*> head_{nullptr};
public:
    void push(const T& val);
    bool pop(T& result);
    ~LockFreeStack();
};

    Push Operation

    Push allocates a new node, sets its next to the current head, then tries to swing head_ from the old value to the new node. If another thread modified head_ in the meantime, the CAS fails and we retry.

    template <typename T>
void LockFreeStack<T>::push(const T& val) {
    Node<T>* new_node = new Node<T>(val);
    new_node->next = head_.load(std::memory_order_relaxed);

    // CAS loop: try to swing head_ from old_head to new_node
    while (!head_.compare_exchange_weak(
                new_node->next,   // expected (updated on failure)
                new_node,         // desired
                std::memory_order_release,  // success
                std::memory_order_relaxed)) // failure
    {
        // On failure, new_node->next is automatically updated
        // to the current value of head_. Just retry.
    }
}
    Why compare_exchange_weak? On architectures with LL/SC (load-linked/store-conditional), compare_exchange_weak may fail spuriously, but it compiles to fewer instructions than compare_exchange_strong. Since we are already in a retry loop, spurious failures cost nothing.

    Pop Operation

    Pop reads the current head, reads its next pointer, and tries to swing head_ from the old head to the next node. On success it extracts the data and returns true.

    template <typename T>
bool LockFreeStack<T>::pop(T& result) {
    Node<T>* old_head = head_.load(std::memory_order_relaxed);

    while (old_head != nullptr) {
        // Read next before CAS, because after CAS succeeds another
        // thread might immediately free old_head.
        Node<T>* next = old_head->next;

        if (head_.compare_exchange_weak(
                old_head,  // expected (updated on failure)
                next,      // desired
                std::memory_order_acquire,  // success
                std::memory_order_relaxed)) // failure
        {
            result = std::move(old_head->data);
            // WARNING: deleting old_head here is NOT safe in the
            // general case. See the ABA problem article.
            delete old_head;
            return true;
        }
        // On failure, old_head is updated to new head_. Retry.
    }
    return false; // stack was empty
}
    Danger: memory reclamation. The delete old_head above is only safe if no other thread is concurrently reading old_head->next. In a concurrent pop, another thread may still hold a reference to that node. This is the root of the ABA problem, which we address in a dedicated article.

    Memory Ordering Choices

    The ordering constraints are chosen deliberately:

    OperationSuccess orderingFailure orderingRationale
    push CASreleaserelaxedThe release ensures that the node construction (writes to data and next) is visible to the thread that later pops the node with an acquire load.
    pop CASacquirerelaxedThe acquire synchronises with the release in push, ensuring we see fully initialised node data.
    Initial head loadrelaxedThe load merely provides a hint for the CAS. If it is stale, the CAS will fail and we retry. No ordering is needed.

    Together, push-release and pop-acquire form a release-acquire pair. Every successful pop synchronises-with the push that created the node, establishing a happens-before edge that makes the node data visible.

    The Treiber Stack

    The algorithm above is exactly the Treiber stack. Its key properties:

    • Lock-free: if one thread is suspended mid-CAS, other threads can still push and pop successfully.
    • Linearisable: each push and pop takes effect atomically at the point where its CAS succeeds. The linearisation point is the successful compare_exchange.
    • LIFO ordering: the most recently pushed item is always at the head.

    Correctness Reasoning

    To argue correctness we need to show two things: safety (the stack is always in a consistent state) and progress (the system never gets stuck).

    Safety: Linearisability

    Consider any concurrent execution. Each push or pop contains exactly one successful CAS (or, for pop on an empty stack, a read that sees nullptr). We assign the linearisation point to that CAS. Since CAS on an atomic variable is itself atomic, the linearisation points form a total order. In that total order:

    • Every push inserts its node at the head.
    • Every pop removes the node at the head.
    • No node is removed twice (a successful CAS is a unique swap).

    This corresponds exactly to a sequential LIFO stack.

    Progress: Lock-Freedom

    If a CAS fails, it means another thread's CAS succeeded. In any finite window of CAS attempts across all threads, at least one succeeds and completes its operation. Therefore the algorithm is lock-free.

    Note: individual threads can starve if they keep losing the CAS race. This is acceptable under the lock-free guarantee but not under wait-free. Making the stack wait-free is substantially harder and rarely necessary in practice.

    compare_exchange_weak vs. compare_exchange_strong

    // compare_exchange_weak: may fail spuriously on LL/SC architectures
// Use when you ALREADY have a retry loop.
bool compare_exchange_weak(T& expected, T desired,
                           memory_order success,
                           memory_order failure);

// compare_exchange_strong: no spurious failures
// Use when you need a single-shot CAS without a loop.
bool compare_exchange_strong(T& expected, T desired,
                             memory_order success,
                             memory_order failure);

    On x86, both compile to the same LOCK CMPXCHG instruction. The distinction matters only on ARM and POWER, where LL/SC can fail spuriously. Since our push and pop already loop, compare_exchange_weak is the right choice.

    Full Implementation

    #include <atomic>
#include <optional>

template <typename T>
class TreiberStack {
    struct Node {
        T data;
        Node* next;
        template <typename U>
        explicit Node(U&& val) : data(std::forward<U>(val)), next(nullptr) {}
    };

    std::atomic<Node*> head_{nullptr};

public:
    void push(T val) {
        Node* node = new Node(std::move(val));
        node->next = head_.load(std::memory_order_relaxed);
        while (!head_.compare_exchange_weak(
                    node->next, node,
                    std::memory_order_release,
                    std::memory_order_relaxed))
        { /* retry */ }
    }

    std::optional<T> pop() {
        Node* old = head_.load(std::memory_order_relaxed);
        while (old != nullptr) {
            if (head_.compare_exchange_weak(
                    old, old->next,
                    std::memory_order_acquire,
                    std::memory_order_relaxed))
            {
                T val = std::move(old->data);
                // In production, use hazard pointers or epoch-based
                // reclamation instead of raw delete.
                delete old;
                return val;
            }
        }
        return std::nullopt;
    }

    ~TreiberStack() {
        Node* cur = head_.load(std::memory_order_relaxed);
        while (cur) {
            Node* next = cur->next;
            delete cur;
            cur = next;
        }
    }

    // Non-copyable, non-movable
    TreiberStack() = default;
    TreiberStack(const TreiberStack&) = delete;
    TreiberStack& operator=(const TreiberStack&) = delete;
};

    Testing Under Contention

    #include <thread>
#include <vector>
#include <cassert>
#include <set>

void stress_test() {
    TreiberStack<int> stack;
    constexpr int N = 100000;
    constexpr int T = 4;

    // T threads each push N values
    std::vector<std::thread> pushers;
    for (int t = 0; t < T; ++t) {
        pushers.emplace_back([&stack, t]() {
            for (int i = 0; i < N; ++i) {
                stack.push(t * N + i);
            }
        });
    }
    for (auto& th : pushers) th.join();

    // T threads collectively pop all values
    std::vector<std::vector<int>> results(T);
    std::vector<std::thread> poppers;
    for (int t = 0; t < T; ++t) {
        poppers.emplace_back([&stack, &results, t]() {
            while (auto val = stack.pop()) {
                results[t].push_back(*val);
            }
        });
    }
    for (auto& th : poppers) th.join();

    // Verify: every value popped exactly once
    std::set<int> all;
    for (auto& v : results)
        for (int x : v) all.insert(x);
    assert(all.size() == T * N);
    for (int i = 0; i < T * N; ++i)
        assert(all.count(i) == 1);
}

    Performance Considerations

    • Cache-line contention: every push and pop touches head_. Under high contention the cache line bounces across cores. An elimination array (backoff technique) can alleviate this.
    • Memory allocation: each push calls new. A per-thread memory pool amortises allocation cost.
    • False sharing: align head_ to a cache line boundary using alignas(64).
    template <typename T>
class AlignedTreiberStack {
    struct Node { /* ... */ };
    alignas(64) std::atomic<Node*> head_{nullptr};
    // ...
};

    Limitations

    1. ABA problem: the naive delete in pop is unsafe. A thread may read old->next after another thread pops and frees old, then pushes a new node at the same address. The CAS succeeds but next points to garbage. See ABA Problem.
    2. No size query: there is no efficient way to atomically read the stack size without a separate counter, and maintaining a counter adds another contention point.
    3. Unbounded memory: without safe reclamation, popped nodes cannot be freed. Solutions include hazard pointers and epoch-based reclamation.

    Summary

    The Treiber stack is the simplest lock-free data structure: a singly linked list guarded by a single atomic CAS. Its correctness follows from the atomicity of compare_exchange, and its lock-freedom from the fact that every failed CAS implies a successful one by another thread. The main open problem is safe memory reclamation, which motivates hazard pointers, epoch-based reclamation, and tagged-pointer approaches.
