Resource Allocation Graphs and Deadlock Modeling

March 2026

The Coffman Conditions

In 1971 Edward Coffman, Michael Elphick, and Arie Shoshani published a set of four necessary conditions that must all hold simultaneously for a deadlock to exist. If any one of them is broken the deadlock cannot form.

  1. Mutual Exclusion. At least one resource must be held in a non-shareable mode. Only one thread at a time may use the resource.
  2. Hold and Wait. A thread holding at least one resource is waiting to acquire additional resources currently held by other threads.
  3. No Preemption. Resources cannot be forcibly taken from a thread; they are released only voluntarily.
  4. Circular Wait. A circular chain of two or more threads exists, each waiting for a resource held by the next member of the chain.

The circular-wait condition is visually captured by cycles in a resource allocation graph, which we define formally next.

Resource Allocation Graph Model

A resource allocation graph (RAG) is a directed bipartite graph G = (V, E) where:

  • V = T ∪ R. T is the set of threads (drawn as circles) and R is the set of resource types (drawn as rectangles with dots for each instance).
  • A request edge Ti → Rj means thread i is waiting for an instance of resource j.
  • An assignment edge Rj → Ti means an instance of resource j is currently held by thread i.

In the diagram above, T1 holds R1 and requests R2, while T2 holds R2 and requests R1. The graph contains a cycle: T1 → R2 → T2 → R1 → T1. With single-instance resources a cycle is both necessary and sufficient for deadlock.

Wait-For Graph Simplification

When every resource type has exactly one instance, the RAG can be collapsed into a wait-for graph by removing resource nodes entirely. An edge Ti → Tj means thread i is waiting for a resource currently held by thread j.

A cycle in the wait-for graph is both necessary and sufficient for deadlock when each resource has a single instance. This reduction is important because cycle detection in a simple directed graph runs in O(V + E) time using depth-first search.

Knot Detection in Multi-Instance RAGs

When a resource type can have multiple instances, a cycle alone is not sufficient for deadlock. Instead, the system must find a knot: a non-empty set of nodes K in the RAG such that every node reachable from any node in K is also in K. In simpler terms, every outgoing path from every node in the knot eventually loops back into the knot. No thread in the knot can ever receive the resource it needs because all instances are held by other threads that are themselves stuck.

Detecting knots requires computing the strongly connected components of the graph (using Tarjan or Kosaraju) and then verifying that no edge leaves the component without returning. The overall complexity is still O(V + E).

Code: Circular Wait with Two Mutexes

The simplest C++ programme that reliably triggers a deadlock illustrates the circular-wait condition directly. Two threads each lock two mutexes in opposite order. 

#include <iostream>
#include <mutex>
#include <thread>

std::mutex mutexA;
std::mutex mutexB;

void threadOne() {
    std::lock_guard<std::mutex> lockA(mutexA);   // holds A
    // Simulate work so the interleaving is likely
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
    std::lock_guard<std::mutex> lockB(mutexB);   // waits for B
    std::cout << "Thread 1 acquired both locks\n";
}

void threadTwo() {
    std::lock_guard<std::mutex> lockB(mutexB);   // holds B
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
    std::lock_guard<std::mutex> lockA(mutexA);   // waits for A
    std::cout << "Thread 2 acquired both locks\n";
}

int main() {
    std::thread t1(threadOne);
    std::thread t2(threadTwo);
    t1.join();
    t2.join();
    return 0;
}

When run, the programme typically hangs indefinitely. Thread 1 holds mutexA and waits for mutexB. Thread 2 holds mutexB and waits for mutexA. Neither can make progress. All four Coffman conditions are satisfied:

  • Mutual exclusion: std::mutex is exclusive.
  • Hold and wait: each thread holds one mutex while requesting the other.
  • No preemption: lock_guard does not allow forced release.
  • Circular wait: T1 waits for T2 and T2 waits for T1.

Single-Instance vs Multi-Instance Resources

The distinction matters for deadlock analysis:

Property Single Instance Multi Instance
Graph type Wait-for graph Full RAG
Cycle implies deadlock? Yes (sufficient) No (necessary only)
Deadlock condition Cycle Knot
Detection cost O(V + E) O(V + E) with SCC

In practice, most C++ mutexes are single-instance resources, so the simpler wait-for graph analysis is typically adequate. Multi-instance analysis becomes relevant when modelling semaphores or connection pools.

Formal Detection Algorithm

The standard algorithm for deadlock detection in multi-instance systems uses vectors analogous to the Banker's algorithm:

  1. Let Available be a vector of length m (number of resource types).
  2. Let Allocation be an n × m matrix, where row i records resources currently held by thread i.
  3. Let Request be an n × m matrix, where row i records outstanding requests for thread i.
  4. Mark every thread whose Request row is zero.
  5. Find an unmarked thread Ti such that Request[i] ≤ Available.
  6. If found, mark it and add its Allocation row to Available. Go to step 5.
  7. If no such thread exists, any unmarked thread is deadlocked.

This is essentially a simulated rollback: if a thread can finish with the currently available resources, assume it does and reclaim its held resources. Threads that can never finish form the deadlocked set.

Summary

  • Deadlock requires all four Coffman conditions to hold simultaneously.
  • A resource allocation graph visually models thread/resource relationships using request and assignment edges.
  • For single-instance resources, a cycle in the RAG (or the reduced wait-for graph) is necessary and sufficient for deadlock.
  • For multi-instance resources, a knot (not merely a cycle) is required, detectable via strongly connected component algorithms.
  • The formal detection algorithm iteratively marks threads that could finish with available resources; unmarked threads form the deadlocked set.
