Common Deadlock Pitfalls in C++

March 2026

Lock-Ordering Violations in Practice

Even when a project documents a lock ordering policy, violations creep in as code evolves. Common causes include:

  • Code path diversity. A function acquires locks in the documented order, but a different code path (error handler, destructor, or optimisation shortcut) reverses the order.
  • Layered libraries. Library A acquires lock X under lock Y. Your code acquires lock Y under lock X, creating a cross-layer ordering violation that may be invisible during code review.
  • Refactoring drift. Lock ordering is correct at commit time, but later refactors change the call graph so that lock scopes nest differently.
#include <mutex>
#include <iostream>

class Account {
    mutable std::mutex mtx;
    double balance;
    int id;
public:
    Account(int id, double bal) : id(id), balance(bal) {}

    // Correct: lock in ID order
    static void transfer(Account& from, Account& to, double amt) {
        Account& first  = (from.id < to.id) ? from : to;
        Account& second = (from.id < to.id) ? to   : from;

        std::scoped_lock lock(first.mtx, second.mtx);
        if (from.balance >= amt) {
            from.balance -= amt;
            to.balance   += amt;
        }
    }

    // BUG: this helper ignores ordering
    void emergencyDebit(Account& other, double amt) {
        std::lock_guard<std::mutex> mine(this->mtx);
        std::lock_guard<std::mutex> theirs(other.mtx);
        // If another thread calls other.emergencyDebit(*this)
        // at the same time, deadlock occurs.
        balance -= amt;
        other.balance += amt;
    }
};

The transfer function correctly orders locks by account ID. The emergencyDebit method, added later, locks this first regardless of ID, violating the established policy.

Calling Unknown Code While Holding a Lock

One of the most insidious deadlock patterns arises when a thread invokes a callback, virtual function, or function pointer while holding a lock. The called code may attempt to acquire the same lock, or a different lock that creates a cycle. 

#include <mutex>
#include <functional>
#include <vector>
#include <iostream>

class EventEmitter {
    std::mutex mtx;
    std::vector<std::function<void()>> listeners;
public:
    void addListener(std::function<void()> fn) {
        std::lock_guard<std::mutex> lock(mtx);
        listeners.push_back(std::move(fn));
    }

    void emit() {
        std::lock_guard<std::mutex> lock(mtx);  // Lock held!
        for (auto& fn : listeners) {
            fn();  // DANGER: callback may call addListener
        }
    }
};

// Usage that deadlocks:
EventEmitter emitter;

void setup() {
    emitter.addListener([]() {
        std::cout << "Event received, adding another listener\n";
        // This re-enters addListener, which tries to lock mtx
        // But mtx is already held by emit(). DEADLOCK.
        emitter.addListener([]() {
            std::cout << "Nested listener\n";
        });
    });
    emitter.emit();
}

Mitigations

  • Copy and release. Copy the listener list under the lock, release the lock, then invoke callbacks outside the critical section.
  • Use a recursive mutex. This avoids the self-deadlock but introduces other problems (e.g., invariant violations).
  • Deferred invocation. Queue callback events and process them outside the lock scope.
void emitSafe() {
    std::vector<std::function<void()>> snapshot;
    {
        std::lock_guard<std::mutex> lock(mtx);
        snapshot = listeners;  // Copy under lock
    }
    // Lock released; safe to call callbacks
    for (auto& fn : snapshot) {
        fn();
    }
}

Deadlock with Condition Variables

Condition variables introduce subtle deadlock risks. The most common pitfall is the lost wakeup, where a notification is sent before the receiving thread starts waiting, causing it to block forever. 

#include <mutex>
#include <condition_variable>
#include <thread>
#include <iostream>

std::mutex mtx;
std::condition_variable cv;
bool ready = false;

void consumer() {
    std::unique_lock<std::mutex> lock(mtx);
    // Correct: use predicate to guard against spurious and lost wakeups
    cv.wait(lock, []{ return ready; });
    std::cout << "Consumer: data ready\n";
}

void producer() {
    {
        std::lock_guard<std::mutex> lock(mtx);
        ready = true;
    }
    cv.notify_one();  // Safe: ready flag is set before notify
}

int main() {
    std::thread c(consumer);
    std::thread p(producer);
    c.join();
    p.join();
    return 0;
}

Signal-With-Lock-Held Debate

Should you call notify_one() inside or outside the lock?

  • Inside the lock: The notified thread wakes up, tries to reacquire the mutex, finds it still held, and immediately blocks again. This causes a "hurry-up-and-wait" pessimisation but is semantically correct.
  • Outside the lock: The notified thread can immediately acquire the mutex. More efficient but requires care to ensure the predicate was already set under the lock.

Neither approach causes deadlock by itself, but mixing notify_all() with locks from multiple condition variables can create complex interleavings where threads wait on each other's conditions indefinitely.

Livelock: Busy But Making No Progress

In a livelock, threads are not blocked. They are actively executing, but they keep undoing and retrying their actions in response to each other, and no thread ever makes forward progress. 

#include <mutex>
#include <thread>
#include <iostream>
#include <atomic>

std::mutex m1, m2;
std::atomic<int> retries{0};

void livelockThread(const char* name,
                    std::mutex& first,
                    std::mutex& second) {
    while (true) {
        first.lock();
        if (second.try_lock()) {
            std::cout << name << ": acquired both\n";
            second.unlock();
            first.unlock();
            return;
        }
        first.unlock();
        retries++;
        // Both threads yield immediately and retry in lockstep
        std::this_thread::yield();
        if (retries > 1000000) {
            std::cout << name << ": gave up after 1M retries\n";
            return;
        }
    }
}

int main() {
    std::thread t1(livelockThread, "T1", std::ref(m1), std::ref(m2));
    std::thread t2(livelockThread, "T2", std::ref(m2), std::ref(m1));
    t1.join();
    t2.join();
    return 0;
}

The fix is randomized or exponential backoff so the two threads desynchronise.

Priority Inversion

Priority inversion occurs when a low-priority thread holds a lock needed by a high-priority thread, while a medium-priority thread preempts the low-priority thread. The high-priority thread is effectively blocked by the medium-priority thread, even though they share no resources. 

// Scenario (pseudocode):
// Thread H (high priority): needs mutex M
// Thread M (medium priority): CPU-bound, no locks
// Thread L (low priority): holds mutex M

// Timeline:
// 1. L acquires M
// 2. H preempts L, tries to acquire M, blocks
// 3. M preempts L (M has higher priority than L)
// 4. L cannot run to release M because M is running
// 5. H is blocked waiting for M, held by L, which is blocked by M
// Result: H is effectively waiting for thread M to finish

Solutions

  • Priority inheritance: When H blocks on M, boost L's priority to H's level so L can run, release M, and unblock H.
  • Priority ceiling: Associate each mutex with the highest priority of any thread that may lock it. A thread holding the mutex runs at the ceiling priority.

The Mars Pathfinder incident in 1997 is the most famous example: the spacecraft's VxWorks RTOS experienced priority inversion that caused system resets, resolved by enabling priority inheritance.

Starvation vs Deadlock vs Livelock

Property Deadlock Livelock Starvation
Threads blocked? Yes, permanently No, actively retrying Yes, waiting indefinitely
CPU usage Zero (blocked) High (busy waiting) Zero (waiting)
Progress None for any involved thread None for involved threads Other threads progress; starved thread does not
Cyclic dependency? Yes Not necessarily No
Detection Cycle in wait-for graph High CPU with no throughput Long wait time for specific thread
Typical fix Lock ordering / scoped_lock Randomized backoff Fair scheduling / fair locks

Defensive Guidelines

  1. Never call unknown code under a lock. If you must invoke a callback, copy the callback list, release the lock, then invoke.
  2. Document the lock order. Every mutex in the codebase should have a documented level or rank in comments.
  3. Prefer std::scoped_lock. It handles multi-lock acquisition safely and concisely.
  4. Minimise lock scope. Hold locks for the shortest duration possible to reduce contention and the chance of ordering mistakes.
  5. Use sanitizers in CI. Build with -fsanitize=thread and run your test suite to catch lock-order inversions early.
  6. Add randomized backoff to try_lock loops. This prevents livelock when threads coordinate via try_lock.
  7. Use timed locks as a safety net. try_lock_for provides an escape hatch for unexpected deadlocks in production.
  8. Test with high thread counts. Deadlocks and livelocks are timing-dependent. Stress tests with more threads than cores increase the chance of exposing bugs.

Summary

  • Lock-ordering violations are the most common cause of deadlock in real C++ codebases. They often appear during refactoring or in error-handling paths.
  • Calling unknown code (callbacks, virtual functions) while holding a lock risks self-deadlock or cross-lock cycles.
  • Livelock is a subtler failure mode where threads consume CPU but achieve nothing. Randomized backoff is the standard fix.
  • Priority inversion is a scheduling-level problem solved by priority inheritance or priority ceiling protocols.
  • Starvation differs from deadlock in that the system as a whole makes progress, but one thread is perpetually denied resources.
