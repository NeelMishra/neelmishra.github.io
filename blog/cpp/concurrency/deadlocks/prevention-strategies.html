Deadlock Prevention and Avoidance

March 2026

Prevention vs Avoidance

Prevention eliminates deadlock by structurally ensuring that at least one of the four Coffman conditions can never hold. This is a static, design-time strategy.

Avoidance allows all four conditions to potentially exist but dynamically ensures the system never enters an unsafe state. The Banker's algorithm is the classic avoidance technique.

In practical C++ systems, prevention through lock ordering or std::scoped_lock is overwhelmingly the preferred approach. Avoidance algorithms are rarely used directly but are valuable for understanding the theory.

Breaking Mutual Exclusion

Mutual exclusion can be relaxed by replacing exclusive locks with lock-free data structures or read-write locks where possible. If a resource can safely be shared (e.g., read-only data), there is no contention and no deadlock risk. 

// Before: exclusive mutex for read-heavy data
std::mutex mtx;
int sharedCounter = 0;

// After: shared_mutex allows concurrent readers
std::shared_mutex smtx;
int sharedCounter = 0;

int readCounter() {
    std::shared_lock lock(smtx);  // Multiple readers OK
    return sharedCounter;
}

void incrementCounter() {
    std::unique_lock lock(smtx);  // Exclusive for writes
    ++sharedCounter;
}

In many cases, however, mutual exclusion is fundamental to correctness and cannot be relaxed. This condition is therefore the hardest to break in practice.

Breaking Hold and Wait

A thread can be required to request all resources at once before beginning execution. If any resource is unavailable the thread releases everything and retries. std::lock achieves this for mutexes: 

std::mutex m1, m2, m3;

void safeWork() {
    // Acquire all three locks atomically; no partial holding
    std::lock(m1, m2, m3);
    std::lock_guard<std::mutex> lg1(m1, std::adopt_lock);
    std::lock_guard<std::mutex> lg2(m2, std::adopt_lock);
    std::lock_guard<std::mutex> lg3(m3, std::adopt_lock);

    // Critical section: all three locks held
    // ...
}

The downside is reduced concurrency. A thread may hold resources it does not yet need, blocking other threads unnecessarily.

Breaking No Preemption

If a thread holding some resources cannot acquire a new resource, it can be forced to release all currently held resources and retry. In C++ this is achieved via try_lock: 

void workWithPreemption() {
    while (true) {
        std::unique_lock<std::mutex> lockA(m1);
        if (m2.try_lock()) {
            std::lock_guard<std::mutex> lockB(m2, std::adopt_lock);
            // Success: both locks acquired
            doWork();
            return;
        }
        // Failed to get m2; release m1 (preempt self) and retry
        lockA.unlock();
        std::this_thread::yield();
    }
}

This pattern can lead to livelock if not combined with randomized backoff. We address that in the try_lock with backoff section below.

Breaking Circular Wait

Impose a total ordering on all resources and require that every thread acquires resources in strictly increasing order. A cycle in the wait-for graph is impossible because it would require at least one thread to acquire a lower-numbered resource after a higher-numbered one.

This is the most practical prevention strategy for C++ code. Assign each mutex a numeric rank and document the discipline.

Lock Ordering Strategy

Define a global ordering on mutexes, typically using their address or an explicit enum. All threads must acquire locks in ascending order. 

#include <iostream>
#include <mutex>
#include <thread>

std::mutex mutexA;
std::mutex mutexB;

// Ordering: always lock the mutex with the lower address first
void orderedLock(std::mutex& m1, std::mutex& m2) {
    if (&m1 < &m2) {
        m1.lock();
        m2.lock();
    } else {
        m2.lock();
        m1.lock();
    }
}

void orderedUnlock(std::mutex& m1, std::mutex& m2) {
    m1.unlock();
    m2.unlock();
}

void threadOne() {
    orderedLock(mutexA, mutexB);
    std::cout << "Thread 1: both locks acquired\n";
    orderedUnlock(mutexA, mutexB);
}

void threadTwo() {
    orderedLock(mutexA, mutexB);
    std::cout << "Thread 2: both locks acquired\n";
    orderedUnlock(mutexA, mutexB);
}

int main() {
    std::thread t1(threadOne);
    std::thread t2(threadTwo);
    t1.join();
    t2.join();
    return 0;
}

Both threads acquire the same lock first, so they serialize on that lock instead of forming a cycle.

std::lock and std::scoped_lock

C++17 introduced std::scoped_lock, which is the preferred way to acquire multiple locks without risking deadlock. Internally it uses a deadlock-avoidance algorithm (typically try-and-back-off) to acquire all specified mutexes atomically. 

#include <iostream>
#include <mutex>
#include <thread>

std::mutex m1, m2;

void safeThread(const char* name) {
    // scoped_lock acquires both mutexes without deadlock
    std::scoped_lock lock(m1, m2);
    std::cout << name << ": both locks acquired safely\n";
}

int main() {
    std::thread t1(safeThread, "T1");
    std::thread t2(safeThread, "T2");
    t1.join();
    t2.join();
    return 0;
}

Under the hood, std::scoped_lock calls std::lock(m1, m2), which attempts to lock both mutexes using a try-and-back-off strategy. If it acquires m1 but fails on m2, it releases m1 and retries. This breaks the hold-and-wait condition.

Prior to C++17, the equivalent pattern uses std::lock with std::adopt_lock: 

std::lock(m1, m2);
std::lock_guard<std::mutex> lg1(m1, std::adopt_lock);
std::lock_guard<std::mutex> lg2(m2, std::adopt_lock);

try_lock with Backoff Strategy

When threads use try_lock and immediately retry, they can enter a livelock where both threads perpetually acquire and release in lockstep. Adding a randomized or exponential backoff breaks the symmetry. 

#include <iostream>
#include <mutex>
#include <thread>
#include <random>
#include <chrono>

std::mutex m1, m2;

void backoffThread(const char* name) {
    std::mt19937 rng(std::hash<std::thread::id>{}(
        std::this_thread::get_id()));
    std::uniform_int_distribution<int> dist(1, 10);

    while (true) {
        m1.lock();
        if (m2.try_lock()) {
            // Both acquired
            std::cout << name << ": success\n";
            m2.unlock();
            m1.unlock();
            return;
        }
        m1.unlock();
        // Randomized backoff
        std::this_thread::sleep_for(
            std::chrono::microseconds(dist(rng)));
    }
}

int main() {
    std::thread t1(backoffThread, "T1");
    std::thread t2(backoffThread, "T2");
    t1.join();
    t2.join();
    return 0;
}

Banker's Algorithm (Avoidance)

The Banker's algorithm, introduced by Edsger Dijkstra, treats the operating system as a banker that grants resource loans only when it can guarantee all threads will eventually return their resources.

Data structures

  • Available[m]: number of available instances of each of the m resource types.
  • Max[n][m]: maximum demand of each thread.
  • Allocation[n][m]: resources currently held by each thread.
  • Need[n][m]: remaining resources each thread may request. Need[i] = Max[i] - Allocation[i].

Safety check

  1. Let Work = Available and Finish[i] = false for all threads.
  2. Find a thread i where Finish[i] == false and Need[i] ≤ Work.
  3. If found: Work += Allocation[i], Finish[i] = true. Go to step 2.
  4. If all threads are finished, the state is safe.

Resource request

When thread i requests resources Request[i]:

  1. If Request[i] > Need[i], reject (exceeds declared maximum).
  2. If Request[i] > Available, wait.
  3. Tentatively allocate: update Available, Allocation, Need.
  4. Run safety check. If safe, grant; otherwise, roll back and wait.

The algorithm runs in O(n2 × m) time and requires advance knowledge of maximum resource needs, which limits its practical use in general-purpose C++ programmes.

Hierarchical Locking Pattern

A hierarchical mutex enforces lock ordering at runtime. Each mutex is assigned a numeric level. A thread may only lock a mutex whose level is strictly lower than the level of any mutex it currently holds. Violations throw an exception rather than risk deadlock. 

#include <mutex>
#include <stdexcept>
#include <climits>

class HierarchicalMutex {
    std::mutex internal;
    unsigned long level;
    unsigned long previousLevel;
    static thread_local unsigned long thisThreadLevel;

public:
    explicit HierarchicalMutex(unsigned long lvl)
        : level(lvl), previousLevel(0) {}

    void lock() {
        if (thisThreadLevel <= level) {
            throw std::logic_error("Lock hierarchy violated");
        }
        internal.lock();
        previousLevel = thisThreadLevel;
        thisThreadLevel = level;
    }

    void unlock() {
        thisThreadLevel = previousLevel;
        internal.unlock();
    }

    bool try_lock() {
        if (thisThreadLevel <= level) return false;
        if (!internal.try_lock()) return false;
        previousLevel = thisThreadLevel;
        thisThreadLevel = level;
        return true;
    }
};

thread_local unsigned long
    HierarchicalMutex::thisThreadLevel = ULONG_MAX;

// Usage:
HierarchicalMutex highLevel(10000);
HierarchicalMutex midLevel(5000);
HierarchicalMutex lowLevel(1000);

void correctOrder() {
    std::lock_guard<HierarchicalMutex> h(highLevel);   // 10000
    std::lock_guard<HierarchicalMutex> m(midLevel);    // 5000
    std::lock_guard<HierarchicalMutex> l(lowLevel);    // 1000
    // OK: strictly decreasing
}

void wrongOrder() {
    std::lock_guard<HierarchicalMutex> l(lowLevel);    // 1000
    std::lock_guard<HierarchicalMutex> h(highLevel);   // 10000
    // THROWS: 10000 > 1000, hierarchy violated
}
