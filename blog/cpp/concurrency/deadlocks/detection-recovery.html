Deadlock Detection and Recovery

March 2026

Runtime Detection Algorithms

When prevention is too restrictive or impractical, systems can instead allow deadlock to occur and then detect and resolve it. Detection periodically inspects the wait-for graph (or the full resource allocation graph for multi-instance resources) and looks for cycles or knots.

Two main approaches are used in practice:

  • Graph-based detection: Maintain a wait-for graph and run cycle detection (DFS) periodically or on every block event.
  • Timeout-based detection: If a thread has been blocked for longer than a threshold, assume deadlock and trigger recovery.

DFS-Based Cycle Detection in Wait-For Graphs

The standard algorithm for finding a cycle in a directed graph is depth-first search. For the wait-for graph:

  1. Maintain a color for each node: WHITE (unvisited), GRAY (in current DFS path), BLACK (fully explored).
  2. For each unvisited node, start a DFS.
  3. When visiting a node, mark it GRAY.
  4. If a neighbor is GRAY, a cycle (back edge) is found. The threads on the cycle are deadlocked.
  5. When backtracking, mark the node BLACK.
// Pseudocode for wait-for graph cycle detection
enum Color { WHITE, GRAY, BLACK };

bool hasCycle(int node,
              const std::vector<std::vector<int>>& adj,
              std::vector<Color>& color,
              std::vector<int>& cycleNodes) {
    color[node] = GRAY;
    for (int neighbor : adj[node]) {
        if (color[neighbor] == GRAY) {
            cycleNodes.push_back(neighbor);
            return true;  // Back edge found
        }
        if (color[neighbor] == WHITE) {
            if (hasCycle(neighbor, adj, color, cycleNodes)) {
                cycleNodes.push_back(neighbor);
                return true;
            }
        }
    }
    color[node] = BLACK;
    return false;
}

Time complexity is O(V + E) where V is the number of threads and E is the number of wait-for edges. In practice, the graph is small (tens to hundreds of threads) and detection is fast.

Timeout-Based Detection

Graph-based detection requires maintaining the wait-for graph, which adds overhead. A simpler heuristic is to use timed locking primitives. If a lock acquisition does not succeed within a configured timeout, the thread assumes potential deadlock and takes corrective action. 

#include <iostream>
#include <mutex>
#include <thread>
#include <chrono>

std::timed_mutex tmA, tmB;

void timedThread(const char* name,
                 std::timed_mutex& first,
                 std::timed_mutex& second) {
    using namespace std::chrono;
    first.lock();
    std::this_thread::sleep_for(milliseconds(50));

    if (second.try_lock_for(milliseconds(100))) {
        std::cout << name << ": acquired both locks\n";
        second.unlock();
    } else {
        std::cout << name << ": timeout, potential deadlock\n";
    }
    first.unlock();
}

int main() {
    std::thread t1(timedThread, "T1",
                   std::ref(tmA), std::ref(tmB));
    std::thread t2(timedThread, "T2",
                   std::ref(tmB), std::ref(tmA));
    t1.join();
    t2.join();
    return 0;
}

Here one (or both) threads will time out, detect the potential deadlock, and release the held lock. The timeout value is a trade-off: too short causes false positives, too long delays detection.

Recovery Strategies

Once a deadlock is detected, the system must break it. Three main strategies exist:

1. Process/Thread Termination

  • Abort all deadlocked threads. Simple but expensive. All work in progress is lost.
  • Abort one thread at a time. After each termination, re-run the detection algorithm. Choose the victim based on priority, age, or resources held.

2. Resource Preemption

Forcibly take a resource from one thread and give it to another. This requires the preempted thread to be able to roll back to a consistent state, which is often impractical in C++ user-space code.

3. Rollback

If the application supports checkpointing, roll the victim thread back to a safe state before it acquired the contested resource. Database systems commonly use this approach for transaction deadlocks.

Victim Selection Criteria

  • Thread priority (lower priority threads are sacrificed first).
  • Amount of work completed (favour threads near completion).
  • Number of resources held (prefer to preempt threads holding many).
  • Cost to restart the thread.
  • Avoid starvation by ensuring the same thread is not always chosen.

Code: Deadlock Detector with std::timed_mutex

#include <iostream>
#include <mutex>
#include <thread>
#include <chrono>
#include <vector>
#include <atomic>

// Shared resources
std::timed_mutex resourceA;
std::timed_mutex resourceB;
std::atomic<bool> deadlockDetected{false};

constexpr auto TIMEOUT = std::chrono::milliseconds(200);

void worker(const char* name,
            std::timed_mutex& first,
            std::timed_mutex& second) {
    while (!deadlockDetected.load()) {
        std::unique_lock<std::timed_mutex> lockFirst(first);
        std::this_thread::sleep_for(std::chrono::milliseconds(10));

        if (second.try_lock_for(TIMEOUT)) {
            std::cout << name << ": acquired both, doing work\n";
            std::this_thread::sleep_for(
                std::chrono::milliseconds(5));
            second.unlock();
            return;  // Work done
        } else {
            std::cout << name
                      << ": DEADLOCK DETECTED (timeout). Releasing.\n";
            deadlockDetected.store(true);
            lockFirst.unlock();
            return;
        }
    }
}

int main() {
    std::thread t1(worker, "Worker-1",
                   std::ref(resourceA), std::ref(resourceB));
    std::thread t2(worker, "Worker-2",
                   std::ref(resourceB), std::ref(resourceA));
    t1.join();
    t2.join();

    if (deadlockDetected.load()) {
        std::cout << "System: deadlock was detected and resolved\n";
    } else {
        std::cout << "System: all work completed normally\n";
    }
    return 0;
}

The deadlockDetected atomic flag lets both threads cooperate on shutdown. In a real system this would trigger a more sophisticated recovery path.

Thread and Deadlock Sanitizers

Modern compilers provide instrumentation tools that can detect potential deadlocks at runtime without modifying application logic.

ThreadSanitizer (TSan)

Compile with -fsanitize=thread (GCC/Clang). TSan tracks all mutex operations and reports potential data races. While its primary focus is data races, it also reports lock-order inversions that can lead to deadlock. 

g++ -fsanitize=thread -g -O1 deadlock_example.cpp -o example
./example

Deadlock Sanitizer

Clang's -fsanitize=thread includes a lock-order inversion detector. It builds a lock-order graph at runtime and reports when two locks are acquired in inconsistent orders across different threads, even if a deadlock does not actually occur during the test run.

Helgrind (Valgrind)

Helgrind is a Valgrind tool for detecting synchronisation errors. It can detect lock ordering problems and potential deadlocks: 

valgrind --tool=helgrind ./example

When to Use Each Tool

Tool Detects Overhead
TSan Data races, lock-order inversions 5x to 15x slowdown
Helgrind Lock errors, potential deadlocks 20x to 50x slowdown
timed_mutex Runtime deadlocks (heuristic) Minimal (timeout cost only)

Detection Invocation Frequency

How often should the detection algorithm run? Options include:

  • On every block. Immediate detection but high overhead. Suitable only for small systems.
  • Periodic interval. Run every N seconds. Balances detection latency against overhead.
  • On resource exhaustion. Triggered when a thread has been waiting for an unusually long time (timeout-based).
  • On demand. An operator or monitoring system triggers detection when signs of a hang are observed.

Summary

  • DFS on the wait-for graph finds cycles in O(V + E) time.
  • Timeout-based detection via std::timed_mutex is a practical heuristic for C++ applications.
  • Recovery options: thread termination, resource preemption, or rollback. Victim selection should avoid starvation.
  • ThreadSanitizer and Helgrind detect lock-order inversions during testing, even when no deadlock actually manifests.
