    Deadlocks and Concurrency Hazards

    March 2026 · 26 min read

    What Is a Deadlock?

    A deadlock occurs when two or more threads are each waiting for a resource held by another thread in the group, forming a circular dependency. No thread can make progress because each is waiting for the other to release first. The program freezes permanently.

    The classic scenario involves two mutexes and two threads:

    std::mutex mutex_a, mutex_b;

// Thread 1: locks A then B
void thread1() {
    std::lock_guard lockA(mutex_a);    // locks A
    std::this_thread::sleep_for(1ms);  // simulate work
    std::lock_guard lockB(mutex_b);    // waits for B... forever
}

// Thread 2: locks B then A (OPPOSITE ORDER!)
void thread2() {
    std::lock_guard lockB(mutex_b);    // locks B
    std::this_thread::sleep_for(1ms);  // simulate work
    std::lock_guard lockA(mutex_a);    // waits for A... forever
}
// DEADLOCK: Thread 1 holds A, waits for B
//           Thread 2 holds B, waits for A

    ▶ Deadlock: Circular Wait

    Watch two threads lock mutexes in opposite order, creating a deadlock.

    The Four Conditions for Deadlock

    A deadlock can only occur when all four of these conditions hold simultaneously (Coffman conditions):

    1. Mutual exclusion: At least one resource is held in a non-shareable mode (only one thread can use it at a time).
    2. Hold and wait: A thread holds at least one resource while waiting to acquire additional resources.
    3. No preemption: Resources cannot be forcibly taken from a thread; they must be released voluntarily.
    4. Circular wait: A set of threads {T1, T2, ..., Tn} exists where T1 waits for T2, T2 waits for T3, ..., Tn waits for T1.
    To prevent deadlocks, break any one of these four conditions. In practice, we usually break the "circular wait" condition by establishing a consistent lock ordering, or we break "hold and wait" by locking all required mutexes at once.

    Prevention Strategies

    Strategy 1: Consistent Lock Ordering

    If all threads always acquire mutexes in the same order, circular wait is impossible:

    // ALWAYS lock A before B, in every thread
void thread1() {
    std::lock_guard lockA(mutex_a);
    std::lock_guard lockB(mutex_b);
    // safe: both threads use A-then-B order
}

void thread2() {
    std::lock_guard lockA(mutex_a);  // same order as thread1
    std::lock_guard lockB(mutex_b);
    // no deadlock possible
}

    This is simple but requires discipline. In large codebases, it is easy to accidentally violate the ordering, especially when functions call other functions that acquire locks.

    Strategy 2: std::lock (Lock All At Once)

    std::lock() takes multiple lockable objects and locks all of them using a deadlock-avoidance algorithm (try-and-back-off). If it cannot acquire all locks, it releases any it has acquired and tries again:

    void transfer(Account& from, Account& to, int amount) {
    // Lock both, deadlock-free, regardless of argument order
    std::unique_lock lockA(from.mtx_, std::defer_lock);
    std::unique_lock lockB(to.mtx_, std::defer_lock);
    std::lock(lockA, lockB);  // atomic multi-lock

    from.balance_ -= amount;
    to.balance_ += amount;
}

    Strategy 3: std::scoped_lock (C++17)

    std::scoped_lock combines std::lock with RAII. It is the simplest and safest way to lock multiple mutexes:

    void transfer(Account& from, Account& to, int amount) {
    std::scoped_lock lock(from.mtx_, to.mtx_);  // done!
    from.balance_ -= amount;
    to.balance_ += amount;
}

    Strategy 4: Lock Hierarchy

    Assign each mutex a numeric level. Threads can only acquire mutexes with a level lower than any mutex they currently hold. Violating this rule throws an exception at runtime:

    class HierarchicalMutex {
    std::mutex internal_;
    unsigned long level_;
    unsigned long prev_level_;
    static thread_local unsigned long this_thread_level_;

public:
    explicit HierarchicalMutex(unsigned long level)
        : level_(level), prev_level_(0) {}

    void lock() {
        if (this_thread_level_ <= level_)
            throw std::logic_error("Lock ordering violation");
        internal_.lock();
        prev_level_ = this_thread_level_;
        this_thread_level_ = level_;
    }

    void unlock() {
        this_thread_level_ = prev_level_;
        internal_.unlock();
    }

    bool try_lock() {
        if (this_thread_level_ <= level_) return false;
        if (!internal_.try_lock()) return false;
        prev_level_ = this_thread_level_;
        this_thread_level_ = level_;
        return true;
    }
};
thread_local unsigned long
    HierarchicalMutex::this_thread_level_ = ULONG_MAX;

    Beyond Deadlocks: Other Hazards

    Livelock

    In a livelock, threads are not blocked; they are actively running but make no progress. Imagine two people in a narrow hallway: each steps aside to let the other pass, both step the same way, repeat forever.

    Livelocks often occur when threads use try_lock() and back off in identical patterns:

    // Both threads try, fail, release, and retry in lockstep
void thread1() {
    while (true) {
        std::unique_lock lockA(mtxA, std::try_to_lock);
        if (!lockA) continue;
        std::unique_lock lockB(mtxB, std::try_to_lock);
        if (!lockB) continue;  // release A, retry
        // Never reaches here if thread2 does the same thing
        do_work();
        return;
    }
}

    Fix: Add random back-off delays, or use std::lock()/scoped_lock which handle this automatically.

    Starvation

    Starvation occurs when a thread never gets to run because other threads monopolize the resource. Example: in a reader-writer lock, if readers continuously arrive, a waiting writer may never get a chance to acquire the exclusive lock.

    Priority Inversion

    A low-priority thread holds a mutex needed by a high-priority thread. Meanwhile, a medium-priority thread preempts the low-priority thread. The high-priority thread is effectively blocked by the medium-priority thread, which has no involvement with the mutex. This famously caused a system reset on the Mars Pathfinder in 1997.

    Fix: Priority inheritance protocols (the OS temporarily boosts the low-priority thread to the high-priority level while it holds the mutex).

    Detecting Deadlocks

    • ThreadSanitizer: Detects potential deadlocks (lock ordering violations) at runtime.
    • GDB: When a program hangs, attach with gdb -p <pid> and run thread apply all bt to see where each thread is stuck.
    • Lock dependency graphs: Maintain a runtime graph of which thread holds which lock and which lock each thread is waiting for. A cycle in this graph means deadlock.

    Practical Deadlock Prevention Guidelines

    1. Avoid holding multiple locks. If you can redesign to use a single lock, do so.
    2. If you must hold multiple locks, use std::scoped_lock.
    3. Never call user-provided code while holding a lock. You do not know what locks that code might try to acquire.
    4. Use lock hierarchies for complex systems.
    5. Keep critical sections short. The less time you hold a lock, the less chance of conflict.
    6. Prefer lock-free data structures when performance justifies the complexity.

    Summary

    • A deadlock is a circular wait among threads for resources held by each other.
    • It requires four conditions (mutual exclusion, hold-and-wait, no preemption, circular wait).
    • Break any one condition to prevent deadlocks: use consistent ordering, std::lock(), scoped_lock, or lock hierarchies.
    • Livelocks are active but non-progressing. Starvation denies a thread resources indefinitely.
    • Priority inversion indirectly blocks high-priority threads through lock contention.

    With mutexes and deadlock prevention mastered, we can move to a more sophisticated synchronization tool: Condition Variables.
