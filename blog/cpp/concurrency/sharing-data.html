    Sharing Data Between Threads

    March 2026 · 25 min read

    The Root of All Evil: Shared Mutable State

    If threads never shared data, concurrency would be trivial. Each thread would operate in isolation, like separate programs. The problems begin the moment two threads access the same memory location and at least one of them writes.

    Consider the simplest possible concurrent program: two threads incrementing a shared counter.

    #include <iostream>
#include <thread>

int counter = 0;  // shared mutable state

void increment() {
    for (int i = 0; i < 1000000; ++i) {
        ++counter;  // NOT SAFE
    }
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);
    t1.join();
    t2.join();
    std::cout << "counter = " << counter << "\n";
    // Expected: 2000000
    // Actual: some random number less than 2000000
}

    Run this program multiple times. You will almost certainly get a different number each time, and it will almost always be less than 2,000,000. This is the classic data race.

    Anatomy of a Data Race

    The expression ++counter looks like a single operation in C++, but on most architectures it is actually three separate CPU instructions:

    1. Load the current value of counter from memory into a register.
    2. Increment the value in the register.
    3. Store the new value from the register back to memory.

    When two threads execute these three steps concurrently, their operations can interleave in ways that lose updates:

    ▶ Data Race: Lost Update

    Watch two threads interleave their read-modify-write operations on the same counter.

    The key insight: Thread 2 reads the counter before Thread 1 writes back its incremented value. So Thread 2 bases its work on stale data, and Thread 1's increment is silently lost. This is called a lost update.

    In C++, a data race is undefined behavior. The standard says: if two threads access the same non-atomic variable, at least one writes, and there is no happens-before relationship between them, the behavior is undefined. Not just "wrong value," but the compiler is free to do anything: crash, corrupt memory, format your disk (theoretically). The compiler may even optimize assuming data races never occur, leading to bizarre results.

    Data Race vs Race Condition

    These terms are often confused but mean different things:

    Term Definition Example
    Data raceTwo threads access the same memory location concurrently, at least one writes, no synchronization. Undefined behavior in C++.++counter from two threads without a mutex or atomic.
    Race conditionThe program's correctness depends on the relative timing/ordering of operations. A logic bug, not necessarily UB.Check-then-act: if (map.count(key)) return map[key]; Another thread may erase the key between the check and the access, even if each individual operation is locked.

    You can have a race condition without a data race (every individual access is properly synchronized, but the overall logic is wrong). You can also have a data race that does not manifest as a visible bug on a particular platform (but it is still undefined behavior). Eliminating data races is necessary but not sufficient for correctness.

    Critical Sections

    A critical section is a region of code that accesses shared data and must not be executed by more than one thread simultaneously. The goal of all synchronization primitives (mutexes, atomics, etc.) is to enforce this mutual exclusion.

    Invariant Corruption

    Data races don't just lose updates. They can corrupt data structure invariants. Consider a doubly linked list: removing a node requires updating pointers on both the predecessor and successor. If a thread reads the list while another is halfway through removing a node, it sees an inconsistent state with dangling pointers.

    // Doubly linked list remove (simplified)
void remove(Node* node) {
    node->prev->next = node->next;  // Step 1
    // ← If another thread reads here, the list is BROKEN
    node->next->prev = node->prev;  // Step 2
    delete node;                      // Step 3
}

    Between step 1 and step 2, the list's invariant (every node's next->prev == this) is temporarily violated. A concurrent reader could follow a prev pointer to the wrong node, causing a crash or silent corruption. This is why compound operations on shared data structures must be atomic in the logical sense: all-or-nothing, with no visible intermediate states.

    When Is Sharing Safe?

    Not all shared data is dangerous. Sharing is safe when:

    • All accesses are reads. Multiple threads can safely read the same data concurrently if no thread is writing. This is the basis of the readers-writers pattern.
    • Each thread accesses a different portion. If thread i writes only to results[i], there is no conflict. We used this in the parallel accumulate example.
    • Proper synchronization is used. Mutexes, atomics, or other mechanisms establish the happens-before relationships that prevent data races.

    Classification of Sharing Problems

    Lost Updates

    We saw this above: two threads read-modify-write the same variable, and one write overwrites the other's result. The classic ++counter bug.

    Torn Reads and Writes

    On some architectures, reading or writing a multi-byte value (like a 64-bit long long on a 32-bit platform) is not atomic at the hardware level. The CPU may need two separate bus transactions. If another thread writes between them, you read half old value, half new value:

    // On a 32-bit platform:
long long shared = 0;

// Thread 1: writes 0x0000000100000001
shared = 0x0000000100000001LL;

// Thread 2: reads shared
long long val = shared;
// val might be 0x0000000100000000 or 0x0000000000000001
// (upper half from new, lower half from old — or vice versa)

    This is called a torn read or torn write. Even on 64-bit platforms, non-naturally-aligned accesses or types wider than the bus width can tear. std::atomic guarantees no tearing.

    Reordering Bugs

    Even if individual reads and writes are atomic (no tearing), the compiler and CPU may reorder them. One thread might see writes from another thread in a different order than they were performed. This is the domain of the C++ memory model, which we cover in depth later.

    // Thread 1:
data = 42;          // write data
ready = true;       // signal "data is ready"

// Thread 2:
if (ready) {        // check signal
    use(data);      // BUG: data might still be 0!
}
// The CPU/compiler might reorder Thread 1's writes
// so ready becomes true BEFORE data is written.

    The Check-Then-Act Anti-Pattern

    A particularly insidious pattern is checking a condition and then acting on it. Even if each operation is individually thread-safe, the combination is not:

    std::map<std::string, int> cache;
std::mutex mtx;

int get_or_compute(const std::string& key) {
    {
        std::lock_guard lock(mtx);
        if (cache.count(key)) return cache[key];  // check
    }
    // ← another thread might erase 'key' here!
    int value = expensive_compute(key);
    {
        std::lock_guard lock(mtx);
        cache[key] = value;                        // act
    }
    return value;
}

// This is a RACE CONDITION (though not a data race — each
// access is mutex-protected). The check and the act are
// not in the same critical section.

    Fix: The check and the act must be in the same critical section, or you must accept that the condition might have changed and handle it.

    Solutions: A Preview

    The rest of this series introduces the tools C++ provides to solve sharing problems. Here is a quick map:

    Problem Solution Post
    Compound read-modify-writestd::mutex + lock guardsMutexes & Locks
    Simple counters, flagsstd::atomicAtomic Operations
    Waiting for a conditionstd::condition_variableCondition Variables
    Instruction/memory reorderingMemory orderings, fencesMemory Model
    Avoid sharing entirelythread_local, message passing, immutable dataConcurrency Patterns
    The best synchronization is no synchronization. The safest concurrent program is one where threads do not share mutable state at all. Prefer thread-local data, message-passing patterns, or immutable shared data whenever possible. Only share mutable state when the design demands it, and then protect it rigorously.

    Detecting Data Races

    Data races are notoriously hard to detect because they depend on timing. Your program may run correctly a thousand times and fail on the thousand-and-first. Here are the tools available:

    ThreadSanitizer (TSan)

    Compile with -fsanitize=thread (GCC/Clang). TSan instruments every memory access and reports data races with full stack traces. It has some overhead (~5-15x slowdown), but it catches most data races:

    # Compile with TSan
g++ -std=c++20 -pthread -fsanitize=thread -g program.cpp -o program

# Run
./program

# If a data race exists, you'll see output like:
# WARNING: ThreadSanitizer: data race (pid=12345)
#   Write of size 4 at 0x... by thread T2:
#     #0 increment() program.cpp:7
#   Previous write of size 4 at 0x... by thread T1:
#     #0 increment() program.cpp:7

    Helgrind (Valgrind)

    Part of the Valgrind tool suite. It tracks lock ordering, detects data races, and finds lock usage errors. Slower than TSan but available on platforms where TSan is not.

    Static Analysis

    Tools like Clang's Thread Safety Analysis can annotate code with GUARDED_BY(mutex) attributes. The compiler then statically checks that the annotation is respected:

    class Account {
    mutable std::mutex mtx_;
    int balance_ GUARDED_BY(mtx_) = 0;

public:
    void deposit(int amount) {
        std::lock_guard lock(mtx_);
        balance_ += amount;   // OK: mutex is locked
    }

    int balance() const {
        // balance_;          // ERROR: accessing without lock
        std::lock_guard lock(mtx_);
        return balance_;      // OK
    }
};

    Summary

    • Shared mutable state is the fundamental source of concurrency bugs.
    • ++counter across threads is a data race: three CPU instructions can interleave, causing lost updates.
    • A data race (undefined behavior) is different from a race condition (logic bug).
    • Data races can cause lost updates, torn reads/writes, invariant corruption, and completely unpredictable behavior.
    • The check-then-act pattern is a race condition even when individual operations are mutex-protected.
    • The best defense is to minimize sharing. When you must share, use mutexes or atomics.
    • Use ThreadSanitizer (-fsanitize=thread) religiously during development and testing.
