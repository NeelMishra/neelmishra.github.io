    std::async and Task-Based Parallelism

    March 2026 · 20 min read

    What Is std::async?

    std::async is the simplest way to run a function asynchronously and get a future for its result. The runtime decides whether to launch a new thread or defer execution:

    #include <future>
#include <iostream>

int heavy_computation(int x) {
    std::this_thread::sleep_for(std::chrono::seconds(2));
    return x * x;
}

int main() {
    auto fut = std::async(std::launch::async, heavy_computation, 42);

    // ... do other work while computation runs ...

    int result = fut.get();  // blocks until done
    std::cout << result << "\n";  // 1764
}

    Launch Policies

    PolicyBehavior
    std::launch::asyncThe function must run on a new thread (or a thread from an implementation's pool). Execution begins immediately.
    std::launch::deferredThe function runs lazily: only when get() or wait() is called, on the calling thread. No new thread is created.
    std::launch::async | std::launch::deferredDefault. The implementation chooses. In practice, this is unpredictable and can cause subtle bugs (see below).
    Always specify a launch policy. The default (async | deferred) lets the implementation choose, which means your function might never run asynchronously. If you want parallelism, use std::launch::async explicitly.

    The Fire-and-Forget Trap

    A dangerous pattern: discarding the future returned by std::async:

    // DANGER: the returned future's destructor BLOCKS until the task finishes!
std::async(std::launch::async, []{ long_running_work(); });
// ↑ This line BLOCKS because the temporary future is destroyed immediately,
//   and its destructor waits for the task to complete.

    The future's destructor joins the thread. Discarding it means the call is synchronous. Always store the future:

    auto fut = std::async(std::launch::async, []{ long_running_work(); });
// ... do other work ...
fut.get();  // explicitly wait when ready

    Parallel Computation Pattern

    template<typename Func>
auto parallel_invoke(Func f1, Func f2) {
    auto fut1 = std::async(std::launch::async, f1);
    auto result2 = f2();         // run f2 on current thread
    auto result1 = fut1.get();   // get f1's result
    return std::make_pair(result1, result2);
}

// Usage: run two independent computations in parallel
auto [left, right] = parallel_invoke(
    []{ return sort_left_half(); },
    []{ return sort_right_half(); }
);

    async vs std::thread

    Featurestd::asyncstd::thread
    Return valueAutomatic via futureManual (shared variable, promise)
    Exception handlingAutomatic propagationMust catch manually
    Thread managementAutomatic join in destructorMust join/detach explicitly
    Thread reuseImplementation may pool threadsAlways creates new OS thread
    Use caseTasks with resultsLong-running workers, fine control

    Limitations of std::async

    • No thread pool guarantee: Each async call may create a new thread. For many small tasks, this overhead adds up. Use a thread pool instead.
    • Default policy unpredictable: Without std::launch::async, the task might run deferred (not parallel at all).
    • Future destructor blocks: The auto-join behavior can cause unexpected synchronous execution.
    • No cancellation: Once launched, you cannot cancel the task (unlike std::jthread's stop tokens).

    Summary

    • std::async is the easiest way to launch an asynchronous computation and retrieve its result.
    • Always use std::launch::async if you want actual parallelism.
    • Never discard the returned future (it blocks on destruction).
    • For many small tasks, prefer a thread pool over repeated std::async calls.
