    Lock Guards and RAII Locking in C++

    March 2026

    Raw lock() and unlock() calls on mutexes are error-prone. Early returns, exceptions, or simply forgetting to unlock can leave a mutex permanently locked, causing deadlocks or starvation. C++ provides RAII (Resource Acquisition Is Initialization) wrappers that bind the mutex lifetime to a scope, guaranteeing correct unlocking even in the face of exceptions.

    std::lock_guard

    The simplest RAII lock wrapper, available since C++11. It locks the mutex in its constructor and unlocks it in its destructor. It is non-copyable, non-movable, and provides no additional operations beyond construction and destruction.

    #include <mutex>
#include <vector>

std::mutex mtx;
std::vector<int> shared_data;

void append(int value) {
    std::lock_guard<std::mutex> guard(mtx); // Locks mtx
    shared_data.push_back(value);
    // If push_back throws, guard's destructor still unlocks mtx
} // guard destroyed here, mtx.unlock() called

    With C++17 CTAD

    // C++17: Class Template Argument Deduction
void append_17(int value) {
    std::lock_guard guard(mtx); // Deduces std::lock_guard<std::mutex>
    shared_data.push_back(value);
}

    Adopt Lock

    If you have already locked the mutex manually (e.g., after a try_lock), use std::adopt_lock to tell the guard to assume ownership without locking again:

    void guarded_work() {
    mtx.lock(); // Manually locked
    std::lock_guard<std::mutex> guard(mtx, std::adopt_lock);
    // guard will unlock on destruction, but did not call lock()
    do_work();
}
    Limitation. std::lock_guard cannot be unlocked early, cannot be moved to another scope, and cannot be used with condition variables. For those use cases, use std::unique_lock.

    std::unique_lock

    std::unique_lock is the Swiss Army knife of lock wrappers. It supports deferred locking, try-locking, timed locking, early unlocking, and move semantics. It is the lock type required by std::condition_variable::wait().

    Basic Usage

    std::mutex mtx;

void basic() {
    std::unique_lock<std::mutex> lock(mtx);
    // ... critical section ...
} // Unlocked on destruction

    Deferred Locking

    void deferred_example() {
    std::unique_lock<std::mutex> lock(mtx, std::defer_lock);
    // mtx is NOT locked yet

    // Do some non-critical setup...
    prepare_data();

    // Now lock when ready
    lock.lock();
    update_shared_state();
} // Unlocked on destruction

    Try Lock

    void try_lock_example() {
    std::unique_lock<std::mutex> lock(mtx, std::try_to_lock);
    if (lock.owns_lock()) {
        // Successfully acquired
        process_data();
    } else {
        // Could not acquire, do fallback
        do_something_else();
    }
}

    Timed Locking

    std::timed_mutex tmtx;

void timed_example() {
    std::unique_lock<std::timed_mutex> lock(tmtx, std::defer_lock);
    if (lock.try_lock_for(std::chrono::milliseconds(100))) {
        // Got the lock within 100ms
        critical_work();
    } else {
        handle_timeout();
    }
}

    Early Unlock

    void early_unlock() {
    std::unique_lock<std::mutex> lock(mtx);
    auto snapshot = read_shared_state(); // Need lock for this
    lock.unlock();                       // Release early

    // Expensive computation that does not need the lock
    auto result = heavy_computation(snapshot);

    lock.lock();                         // Re-acquire
    write_result(result);
}

    Move Semantics

    std::unique_lock<std::mutex> get_lock() {
    std::unique_lock<std::mutex> lock(mtx);
    prepare_data();
    return lock; // Moves ownership to caller
}

void caller() {
    auto lock = get_lock(); // Now this scope owns the lock
    use_prepared_data();
} // Unlocked here

    With Condition Variables

    std::mutex cv_mtx;
std::condition_variable cv;
bool ready = false;

void consumer() {
    std::unique_lock<std::mutex> lock(cv_mtx);
    cv.wait(lock, [] { return ready; });
    // lock is re-acquired after wait returns
    consume_data();
}

void producer() {
    {
        std::lock_guard<std::mutex> lock(cv_mtx);
        prepare_data();
        ready = true;
    }
    cv.notify_one();
}
    Why condition_variable needs unique_lock. During cv.wait(), the lock must be atomically released (so the producer can signal) and re-acquired when the thread wakes up. std::unique_lock exposes lock()/unlock() methods that the condition variable invokes internally. std::lock_guard does not expose these.

    std::scoped_lock (C++17)

    std::scoped_lock is the modern replacement for std::lock_guard when you need to lock one or more mutexes simultaneously. It uses a deadlock-avoidance algorithm (equivalent to std::lock()) when given multiple mutexes.

    Single Mutex

    void single() {
    std::scoped_lock lock(mtx); // Equivalent to lock_guard
    do_work();
}

    Multiple Mutexes: Deadlock-Free

    std::mutex account_a_mtx, account_b_mtx;

void transfer(int amount) {
    // Locks both mutexes without risk of deadlock,
    // regardless of what order other threads lock them
    std::scoped_lock lock(account_a_mtx, account_b_mtx);
    account_a.balance -= amount;
    account_b.balance += amount;
}

    Without std::scoped_lock, you would need the two-step dance with std::lock and std::adopt_lock:

    // Pre-C++17 equivalent:
void transfer_old(int amount) {
    std::lock(account_a_mtx, account_b_mtx); // Deadlock-free
    std::lock_guard<std::mutex> la(account_a_mtx, std::adopt_lock);
    std::lock_guard<std::mutex> lb(account_b_mtx, std::adopt_lock);
    account_a.balance -= amount;
    account_b.balance += amount;
}

    How the Deadlock Avoidance Works

    The standard specifies that std::scoped_lock (and std::lock) must use a deadlock-avoidance algorithm. Typical implementations use a try-and-back-off strategy:

    // Simplified pseudocode of std::lock(m1, m2):
// 1. Lock m1
// 2. Try to lock m2
//    - If success: done
//    - If fail: unlock m1, lock m2, try to lock m1
//      - If success: done
//      - If fail: unlock m2, goto step 1
// This ensures no fixed ordering is needed.
    Always prefer scoped_lock. In C++17 and later, std::scoped_lock should be your default choice. It handles single and multiple mutexes, it is concise, and it prevents deadlocks by construction. Use std::unique_lock only when you need deferred locking, early unlock, or condition variable integration.

    std::shared_lock (C++14)

    std::shared_lock is the RAII wrapper for acquiring shared (read) access on a std::shared_mutex or std::shared_timed_mutex. Multiple threads can hold a shared_lock simultaneously.

    #include <shared_mutex>
#include <string>
#include <map>

class DnsCache {
    mutable std::shared_mutex smtx_;
    std::map<std::string, std::string> entries_;

public:
    std::string lookup(const std::string& domain) const {
        std::shared_lock lock(smtx_);        // Shared (read) access
        auto it = entries_.find(domain);
        return it != entries_.end() ? it->second : "";
    }

    void update(const std::string& domain, const std::string& ip) {
        std::unique_lock lock(smtx_);        // Exclusive (write) access
        entries_[domain] = ip;
    }
};

    std::shared_lock supports the same flexibility as std::unique_lock: deferred locking, try_lock, timed locking, and move semantics.

    void flexible_read() {
    std::shared_lock lock(smtx_, std::defer_lock);
    if (lock.try_lock()) {
        // Read data
    }
}

    Common Patterns

    Pattern 1: Lock-Free Read, Locked Write

    std::atomic<int> fast_counter{0};
std::mutex slow_mtx;
std::vector<int> log_entries;

void record(int value) {
    fast_counter.fetch_add(1, std::memory_order_relaxed);
    // Only lock for the complex operation
    std::lock_guard lock(slow_mtx);
    log_entries.push_back(value);
}

    Pattern 2: Scoped Lock with Multiple Resources

    struct Account {
    std::mutex mtx;
    int balance;
};

void transfer(Account& from, Account& to, int amount) {
    std::scoped_lock lock(from.mtx, to.mtx);
    if (from.balance >= amount) {
        from.balance -= amount;
        to.balance += amount;
    }
}
// No matter what order threads call transfer(a,b) vs transfer(b,a),
// scoped_lock prevents deadlock.

    Pattern 3: Lock Handoff

    class WorkQueue {
    std::mutex mtx_;
    std::queue<Work> queue_;

public:
    // Returns a unique_lock so caller can hold it
    // while processing, then unlock explicitly
    std::pair<Work, std::unique_lock<std::mutex>> pop() {
        std::unique_lock lock(mtx_);
        while (queue_.empty()) {
            // ... wait on condition variable ...
        }
        Work w = std::move(queue_.front());
        queue_.pop();
        return {std::move(w), std::move(lock)};
    }
};

    Pattern 4: Double-Checked Locking (Deprecated)

    // This pattern is obsolete in C++11 and later.
// Use std::call_once or function-local statics instead.

// BROKEN before C++11, unnecessary after:
std::mutex init_mtx;
std::atomic<Widget*> instance{nullptr};

Widget* get_widget() {
    Widget* p = instance.load(std::memory_order_acquire);
    if (!p) {
        std::lock_guard lock(init_mtx);
        p = instance.load(std::memory_order_relaxed);
        if (!p) {
            p = new Widget();
            instance.store(p, std::memory_order_release);
        }
    }
    return p;
}

// Correct modern alternative:
Widget& get_widget_modern() {
    static Widget w; // Thread-safe since C++11
    return w;
}

    Guidelines for Choosing

    1. Default: Use std::scoped_lock (C++17) for simple scoped locking of one or more mutexes.
    2. Need condition_variable: Use std::unique_lock.
    3. Need deferred/try/timed lock: Use std::unique_lock.
    4. Need to transfer lock ownership: Use std::unique_lock (it is movable).
    5. Read access on shared_mutex: Use std::shared_lock.
    6. Pre-C++17 simple lock: Use std::lock_guard.
    Never use raw lock/unlock. In every production codebase, prefer RAII wrappers. The only legitimate use of raw .lock()/.unlock() is in implementing your own RAII wrapper or in very specialized low-level code that has been carefully audited.

    Edge Cases and Gotchas

    Unnamed Lock Guard

    void bug() {
    // WRONG: This creates a temporary that is immediately destroyed!
    std::lock_guard<std::mutex>(mtx); // Locks AND unlocks immediately
    do_work(); // NOT protected!
}

void correct() {
    std::lock_guard<std::mutex> guard(mtx); // Named variable
    do_work(); // Protected until end of scope
}

    unique_lock with owns_lock

    std::unique_lock<std::mutex> lock(mtx, std::defer_lock);
assert(!lock.owns_lock()); // Not locked yet

lock.lock();
assert(lock.owns_lock());  // Now locked

lock.unlock();
assert(!lock.owns_lock()); // Manually unlocked

// Destructor: no unlock (owns_lock is false)

    Scope Minimization

    void process_items(const std::vector<int>& items) {
    std::vector<int> snapshot;
    {
        std::scoped_lock lock(mtx);
        snapshot = shared_items; // Copy under lock
    } // Lock released here

    // Process without holding the lock
    for (int item : snapshot) {
        expensive_process(item);
    }
}

    Summary

    • std::lock_guard: Minimal, zero-overhead RAII lock. Use in pre-C++17 code for single mutex.
    • std::unique_lock: Flexible. Required for condition variables, deferred/timed locking, and movable locks.
    • std::scoped_lock: Modern default. Handles one or more mutexes with deadlock avoidance. Prefer this.
    • std::shared_lock: RAII wrapper for shared (read) access on std::shared_mutex.
    • Always name your lock variables. Unnamed temporaries unlock immediately.
    • Minimize the scope and duration of locks to reduce contention.
