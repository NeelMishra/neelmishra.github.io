    std::call_once and Thread-Safe Initialization

    March 2026

    One-time initialization is a deceptively common problem in concurrent programs. Initializing a singleton, loading a configuration file, setting up a connection pool, or populating a cache on first access all share the same requirement: the initialization must happen exactly once, even when multiple threads race to trigger it. C++ provides two elegant mechanisms for this: std::call_once with std::once_flag, and the C++11 guarantee of thread-safe function-local static initialization (commonly known as "magic statics").

    std::once_flag and std::call_once

    std::once_flag is a synchronization flag that ensures a callable is invoked exactly once, regardless of how many threads attempt to call it concurrently. std::call_once takes a reference to a once_flag and a callable; the first thread to reach it executes the callable, while all concurrent threads block until the initialization is complete.

    #include <mutex>
#include <thread>
#include <iostream>
#include <vector>

std::once_flag init_flag;
int* shared_resource = nullptr;

void initialize() {
    std::cout << "Initializing (runs exactly once)\n";
    shared_resource = new int(42);
}

void worker() {
    std::call_once(init_flag, initialize);
    std::cout << "Resource value: " << *shared_resource << "\n";
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 10; ++i)
        threads.emplace_back(worker);
    for (auto& t : threads)
        t.join();
    delete shared_resource;
}

    Key Guarantees

    • Exactly-once execution: The callable is invoked at most once for a given once_flag.
    • Synchronization: All threads that call std::call_once on the same flag synchronize: the initialization completes (with a memory fence) before any thread proceeds past the call_once call.
    • Exception handling: If the callable throws an exception, the flag is not set, and the next thread to call std::call_once will retry. This is crucial for robust initialization.

    With Lambdas

    class Database {
    std::once_flag connect_flag_;
    Connection conn_;

public:
    Connection& get_connection() {
        std::call_once(connect_flag_, [this] {
            conn_ = Connection::create("host=db.example.com");
            conn_.authenticate("user", "pass");
        });
        return conn_;
    }
};

    With Member Functions

    class Logger {

    }

public:
    void log(const std::string& msg) {

    Exception Handling

    If the callable invoked by std::call_once throws an exception, the once_flag remains in its initial (unset) state. The exception propagates to the calling thread, and the next thread to reach call_once will attempt the initialization again. This continues until one invocation succeeds.

    #include <mutex>
#include <iostream>
#include <stdexcept>
#include <atomic>

std::once_flag flag;
std::atomic<int> attempt{0};

void flaky_init() {
    int n = attempt.fetch_add(1);
    std::cout << "Attempt #" << n << "\n";
    if (n < 2) {
        throw std::runtime_error("Transient failure");
    }
    std::cout << "Initialization succeeded on attempt #" << n << "\n";
}

void worker() {
    try {
        std::call_once(flag, flaky_init);
    } catch (const std::exception& e) {
        std::cout << "Caught: " << e.what() << "\n";
    }
}

// Output (with 5 threads):
// Attempt #0
// Caught: Transient failure
// Attempt #1
// Caught: Transient failure
// Attempt #2
// Initialization succeeded on attempt #2
// (remaining threads proceed without calling flaky_init)
    Retry semantics. This behavior makes call_once suitable for initialization that might fail transiently (e.g., connecting to a database). However, be cautious: if the failure is permanent, every thread will attempt and fail, one at a time, with serialized exceptions.

    The Singleton Pattern with call_once

    class Singleton {
    Singleton() { /* expensive init */ }
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static std::once_flag flag_;
    static Singleton* instance_;

public:
    static Singleton& instance() {
        std::call_once(flag_, [] {
            instance_ = new Singleton();
        });
        return *instance_;
    }
};

std::once_flag Singleton::flag_;
Singleton* Singleton::instance_ = nullptr;

    This is thread-safe and initialization happens exactly once. However, there is a simpler approach available since C++11.

    Magic Statics (C++11)

    The C++11 standard guarantees that function-local static variables are initialized in a thread-safe manner. The first thread to reach the declaration performs the initialization; concurrent threads block until it completes. This is informally known as "magic statics" or "Meyers' Singleton."

    class Singleton {
    Singleton() { /* expensive init */ }
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

public:
    static Singleton& instance() {
        static Singleton s; // Thread-safe since C++11
        return s;
    }
};

    Standard Guarantee

    [stmt.dcl] paragraph 4: "If control enters the declaration concurrently while the variable is being initialized, the concurrent execution shall wait for completion of the initialization."

    Advantages Over call_once

    • Simpler: No once_flag, no separate pointer, no explicit callable.
    • Automatic destruction: The static is destroyed automatically at program exit (in reverse order of construction).
    • Exception safety: If the constructor throws, initialization is retried on the next call (same as call_once).

    Implementation Under the Hood

    Compilers typically implement magic statics with a guard variable (atomic flag) per static variable:

    // Compiler-generated pseudocode for:
// static Singleton s;

static bool __guard = false;
static char __storage[sizeof(Singleton)]; // aligned

if (!__guard) {               // Fast path: check guard
    // Slow path: acquire initialization lock
    __cxa_guard_acquire(&__guard);
    try {
        new (__storage) Singleton(); // Construct in-place
        __cxa_guard_release(&__guard); // Set guard, release fence
    } catch (...) {
        __cxa_guard_abort(&__guard); // Reset guard, allow retry
        throw;
    }
}
return *reinterpret_cast<Singleton*>(__storage);
    Compiler support. All major compilers (GCC 4.3+, Clang 2.9+, MSVC 2015+) implement thread-safe magic statics. On MSVC, this requires the /Zc:threadSafeInit flag, which is enabled by default since VS 2015.

    Performance (fast path)One atomic loadOne atomic load (guard check)

    When to Use Each

    • Singleton or lazy object initialization: Prefer magic statics. Simpler, automatic cleanup.
    • One-time non-constructor work: Use call_once. Examples: registering signal handlers, initializing a third-party library, setting up logging.
    • Resettable initialization: Neither supports resetting. If you need to re-initialize, use a different pattern (e.g., atomic flag + mutex).

    Lazy Initialization Patterns

    Lazy Initialization with call_once

    class TextureManager {
    std::once_flag load_flag_;
    std::unordered_map<std::string, Texture> textures_;

    void load_all() {
        // Expensive: loads textures from disk
        for (const auto& path : discover_texture_files()) {
            textures_[path] = Texture::load(path);
        }
    }

public:
    const Texture& get(const std::string& name) {
        std::call_once(load_flag_, &TextureManager::load_all, this);
        return textures_.at(name);
    }
};

    Implementation Details

    once_flag Internals

    A typical implementation of std::once_flag consists of:

    • An atomic state variable with three possible values: INIT (not started), PENDING (in progress), DONE (completed).
    • A synchronization mechanism (futex, event, or condition variable) for blocking waiting threads.
    // Simplified implementation sketch:
struct once_flag {
    std::atomic<int> state{0}; // 0=INIT, 1=PENDING, 2=DONE
};

void call_once(once_flag& flag, auto&& callable) {
    int expected = 0; // INIT
    if (flag.state.load(std::memory_order_acquire) == 2) {
        return; // Fast path: already initialized
    }
    if (flag.state.compare_exchange_strong(expected, 1)) {
        // We won the race: execute the callable
        try {
            callable();
            flag.state.store(2, std::memory_order_release);
            // Wake all waiting threads
            notify_all(flag);
        } catch (...) {
            flag.state.store(0, std::memory_order_release);
            // Wake one waiting thread to retry
            notify_one(flag);
            throw;
        }
    } else {
        // Another thread is initializing: wait
        while (flag.state.load(std::memory_order_acquire) != 2) {
            wait(flag); // OS-level wait (futex/event)
        }
    }
}

    Fast Path Cost

    Once initialization is complete, subsequent calls to std::call_once consist of a single atomic load with acquire semantics. On x86-64, this is a plain MOV instruction (acquire is free due to TSO). The overhead is negligible.

    Pitfalls

    1. Deadlock with Recursive call_once

    std::once_flag flag;

void init() {
    std::call_once(flag, [] {
        // DEADLOCK: call_once is not re-entrant
        std::call_once(flag, [] {
            // This never executes
        });
    });
}

    2. Static Initialization Order Fiasco

    // file_a.cpp
static Logger logger; // Constructed at program startup

// file_b.cpp
static Database db;   // Also constructed at startup
// If Database constructor logs, but logger is not yet constructed:
// UNDEFINED BEHAVIOR

// Fix: use magic statics to ensure order
Logger& get_logger() {
    static Logger logger;
    return logger;
}

Database& get_database() {
    static Database db(get_logger()); // logger initialized first
    return db;
}

    3. Destruction Order Issues

    // Magic statics are destroyed in reverse order of construction.
// If a destructor accesses another static that was already destroyed:
// UNDEFINED BEHAVIOR

// Mitigation: use a leaked singleton (never destroyed)
Singleton& instance() {
    static Singleton* s = new Singleton(); // Intentionally leaked
    return *s;
    // No destructor call, no destruction-order issues
    // Acceptable for long-lived singletons in most applications
}
    Leaked singletons. Intentionally leaking a singleton is a legitimate pattern used in production codebases (including Google's style guide). The OS reclaims all memory at process exit. This avoids subtle destruction-order bugs, especially in applications with complex static dependency graphs.

    Best Practices

    1. Prefer magic statics for singletons and lazy object construction. They are simpler and provide automatic cleanup.
    2. Use call_once for one-time non-constructor work (library initialization, resource registration).
    3. Never recursively invoke call_once on the same flag. It will deadlock.
    4. Handle exceptions in the callable if the initialization can fail transiently.
    5. Avoid the static initialization order fiasco by wrapping globals in functions with function-local statics.
    6. Consider leaking singletons if destruction-order issues are a concern.
    7. Do not use double-checked locking. It is unnecessary since C++11. Magic statics and call_once are correct and simpler.

    Complete Example: Thread-Safe Service Locator

    #include <mutex>
#include <memory>
#include <unordered_map>
#include <typeindex>
#include <stdexcept>

class ServiceLocator {
    std::unordered_map<std::type_index, std::shared_ptr<void>> services_;
    mutable std::mutex mtx_; // Protects services_ map

    ServiceLocator() = default;

public:
    static ServiceLocator& instance() {
        static ServiceLocator locator; // Magic static
        return locator;
    }

    template <typename T>
    void register_service(std::shared_ptr<T> svc) {
        std::lock_guard lock(mtx_);
        services_[std::type_index(typeid(T))] = std::move(svc);
    }

    template <typename T>
    std::shared_ptr<T> resolve() const {
        std::lock_guard lock(mtx_);
        auto it = services_.find(std::type_index(typeid(T)));
        if (it == services_.end())
            throw std::runtime_error("Service not registered");
        return std::static_pointer_cast<T>(it->second);
    }
};

// Usage:
// ServiceLocator::instance().register_service<ILogger>(
//     std::make_shared<FileLogger>("/var/log/app.log")
// );
// auto logger = ServiceLocator::instance().resolve<ILogger>();

    Summary

    • std::call_once with std::once_flag guarantees exactly-once execution of a callable across all threads.
    • If the callable throws, the flag resets and the next thread retries.
    • C++11 magic statics provide thread-safe initialization of function-local static variables, making singletons trivial.
    • Both mechanisms have negligible overhead after initialization (a single atomic load on the fast path).
    • Double-checked locking is obsolete in modern C++. Use call_once or magic statics instead.
    • Be mindful of destruction order with statics. Leaked singletons are a valid strategy.
    • Never call call_once recursively on the same flag.
