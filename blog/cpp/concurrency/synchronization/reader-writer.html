    Reader-Writer Locks in C++

    March 2026

    Many concurrent systems exhibit a pronounced asymmetry: reads vastly outnumber writes. A DNS cache, a configuration store, a routing table, or an in-memory index may be read thousands of times per second but updated only occasionally. Using a plain std::mutex forces readers to serialize, wasting throughput. The reader-writer lock solves this by allowing multiple concurrent readers while still ensuring exclusive access for writers.

    The Readers-Writers Problem

    The classical readers-writers problem defines two constraints:

    1. Multiple readers can access the resource simultaneously. Reads do not modify state, so concurrent reads are safe.
    2. Writers require exclusive access. A write must block all readers and other writers to maintain consistency.

    There are three major variants:

    • First readers-writers (readers preference): No reader waits unless a writer already holds the lock. Risk: writer starvation.
    • Second readers-writers (writers preference): Once a writer is waiting, no new readers may start. Risk: reader starvation.
    • Third readers-writers (fair): Requests are served roughly in order. Prevents starvation of either side, but may reduce peak throughput.
    Standard caveat. The C++ standard does not specify which variant std::shared_mutex implements. The fairness policy is implementation-defined. In practice: POSIX pthread_rwlock_t is typically writer-preferring, and Windows SRWLOCK does not guarantee fairness but tends toward balance.

    std::shared_mutex in Depth

    Introduced in C++17, std::shared_mutex provides two lock levels:

    #include <shared_mutex>

std::shared_mutex rw_mtx;

// Reader: shared lock
void read_data() {
    std::shared_lock lock(rw_mtx);
    // Multiple threads can be here simultaneously
    auto val = global_map.at("key");
}

// Writer: exclusive lock
void write_data() {
    std::unique_lock lock(rw_mtx);
    // Only one thread can be here; all readers blocked
    global_map["key"] = new_value;
}

    Internal State Machine

    Conceptually, a shared mutex maintains:

    • A reader count (atomic integer): incremented on lock_shared(), decremented on unlock_shared().
    • A writer flag: set when a writer holds the lock.
    • A writer-waiting flag (in writer-preferring implementations): prevents new readers when a writer is queued.
    // Simplified state transitions:
//
// State: FREE (readers=0, writer=false)
//   lock_shared()  -> readers=1, state=READING
//   lock()         -> writer=true, state=WRITING
//
// State: READING (readers>0, writer=false)
//   lock_shared()  -> readers++
//   unlock_shared()-> readers--; if readers==0, state=FREE
//   lock()         -> BLOCKS until readers==0
//
// State: WRITING (readers=0, writer=true)
//   lock_shared()  -> BLOCKS
//   lock()         -> BLOCKS
//   unlock()       -> writer=false, state=FREE

    Read-Heavy Workloads

    When reads dominate (e.g., 99% reads, 1% writes), std::shared_mutex shines. Multiple readers proceed in parallel without contention, and the rare writer pays the cost of waiting for active readers to drain.

    class ConfigStore {
    mutable std::shared_mutex mtx_;
    std::unordered_map<std::string, std::string> config_;

public:
    // Called by hundreds of threads concurrently
    std::string get(const std::string& key) const {
        std::shared_lock lock(mtx_);
        auto it = config_.find(key);
        return it != config_.end() ? it->second : "";
    }

    // Called once every few minutes by admin thread
    void set(const std::string& key, const std::string& val) {
        std::unique_lock lock(mtx_);
        config_[key] = val;
    }
};

    Performance Observations

    • With 8 reader threads and 0 writers, shared_mutex throughput scales nearly linearly. A plain mutex would serialize all reads.
    • With 8 readers and 1 writer (writing every 1ms), throughput drops modestly since readers block briefly during each write.
    • The shared lock acquisition still involves an atomic increment/decrement of the reader count, so it is not free. On high-core-count machines, this atomic can become a bottleneck due to cache-line bouncing.
    When shared_mutex hurts. If the critical section is very short (a few nanoseconds), the overhead of shared_lock (atomic reader count) can exceed the benefit of concurrency. In such cases, a plain std::mutex may actually be faster due to simpler operations and better cache behavior.

    Write-Heavy Workloads

    When writes are frequent, std::shared_mutex offers little advantage over std::mutex. Every write requires exclusive access, blocking all readers and other writers. The reader-count atomics add overhead without improving throughput.

    // Benchmark comparison (approximate, 8 threads, 50% reads 50% writes):
// std::mutex:        ~12M ops/sec
// std::shared_mutex: ~10M ops/sec (reader-count overhead hurts)
//
// For write-heavy workloads, std::mutex is often the better choice.

    Writer Starvation

    In a reader-preferring implementation, a continuous stream of readers can indefinitely delay a waiting writer. The reader count never drops to zero, so the writer never proceeds.

    // Scenario: readers arrive faster than they leave
//
// Time:   R1-lock  R2-lock  R3-lock  W1-wait  R1-unlock  R4-lock  R2-unlock ...
// Readers:  1        2        3       3(wait)    2          3        2      ...
//           Writer never gets in because reader count never hits 0

    Mitigations

    • Writer-preferring lock: Most POSIX implementations set PTHREAD_RWLOCK_PREFER_WRITER_NONRECURSIVE_NP by default, preventing new readers from starting when a writer is waiting.
    • Batching reads: Readers copy the shared data under a brief lock, then process the copy without the lock. This shortens the read-locked window.
    • Epoch-based techniques: Advanced lock-free approaches (RCU, hazard pointers) avoid the problem entirely by allowing readers to proceed without any lock, but these are complex to implement correctly.

    The Upgrade Pattern

    A common desire is to start with a shared lock (read), discover that a write is needed, and "upgrade" to an exclusive lock without releasing the shared lock first. Unfortunately, std::shared_mutex does not support atomic upgrades.

    // WRONG: Deadlock!
void bad_upgrade() {
    std::shared_lock slock(rw_mtx);
    auto val = read_data();
    if (needs_update(val)) {
        // Cannot lock exclusively while holding shared lock
        std::unique_lock ulock(rw_mtx); // DEADLOCK: waits for slock
        write_data(new_val);
    }
}

    Correct Pattern: Release and Re-Acquire

    void safe_upgrade() {
    std::shared_lock slock(rw_mtx);
    auto val = read_data();
    if (!needs_update(val)) return;

    slock.unlock(); // Release shared lock first

    std::unique_lock ulock(rw_mtx); // Now acquire exclusive

    // IMPORTANT: Re-check condition. Another thread may have
    // written between our unlock and lock.
    val = read_data();
    if (needs_update(val)) {
        write_data(new_val);
    }
}
    Upgrade locks in Boost. boost::upgrade_mutex and boost::upgrade_lock support atomic upgrades from shared to exclusive. If you need this frequently, consider Boost or a custom implementation. The standard committee has discussed adding upgrade support but it has not been adopted as of C++23.

    Upgrade Lock Implementation Sketch

    // A simplified upgrade lock concept:
// - upgrade_lock: only one thread can hold this at a time
// - upgrade_lock holders can coexist with shared_lock holders
// - upgrade_lock can be atomically promoted to exclusive
//
// State transitions:
// SHARED -> UPGRADE: one reader marked as "upgrade pending"
// UPGRADE -> EXCLUSIVE: wait for other readers to drain
//                       (but no new readers can start)
//
// This avoids the TOCTOU issue in the release-and-reacquire pattern.

    Real-World Example: Thread-Safe LRU Cache

    #include <shared_mutex>
#include <unordered_map>
#include <list>
#include <optional>

template <typename K, typename V>
class ConcurrentLRU {
    size_t capacity_;
    mutable std::shared_mutex mtx_;
    std::list<std::pair<K, V>> lru_list_;
    std::unordered_map<K, typename std::list<std::pair<K, V>>::iterator> map_;

public:
    explicit ConcurrentLRU(size_t cap) : capacity_(cap) {}

    std::optional<V> get(const K& key) {
        // Note: get() modifies the LRU list (moves to front),
        // so it actually needs an exclusive lock.
        std::unique_lock lock(mtx_);
        auto it = map_.find(key);
        if (it == map_.end()) return std::nullopt;
        lru_list_.splice(lru_list_.begin(), lru_list_, it->second);
        return it->second->second;
    }

    // For a pure lookup that does not update recency:
    std::optional<V> peek(const K& key) const {
        std::shared_lock lock(mtx_); // True read-only
        auto it = map_.find(key);
        if (it == map_.end()) return std::nullopt;
        return it->second->second;
    }

    void put(const K& key, const V& value) {
        std::unique_lock lock(mtx_);
        auto it = map_.find(key);
        if (it != map_.end()) {
            it->second->second = value;
            lru_list_.splice(lru_list_.begin(), lru_list_, it->second);
            return;
        }
        if (map_.size() >= capacity_) {
            auto last = lru_list_.back();
            map_.erase(last.first);
            lru_list_.pop_back();
        }
        lru_list_.emplace_front(key, value);
        map_[key] = lru_list_.begin();
    }
};
    Design observation. The LRU cache is a case where get() is semantically a read but structurally a write (it updates recency). This is a common trap. Profile carefully before assuming that shared_lock is applicable: if every "read" operation also mutates internal state, you gain nothing from std::shared_mutex.

    Key Observations

    • The benefit of shared_mutex is proportional to the ratio of reads and the duration of the critical section.
    • For very short critical sections (under 20 ns), std::mutex can outperform shared_mutex even at 95% reads, because the atomic reader-count operations dominate.
    • On NUMA machines, the atomic reader counter can cause severe cross-node cache traffic. Consider per-node sharding.

    Alternatives to Reader-Writer Locks

    RCU (Read-Copy-Update)

    Used extensively in the Linux kernel. Readers access data without any lock or atomic. Writers create a new version and swap a pointer, then wait for all readers of the old version to finish (a "grace period"). Extremely fast reads, but complex to implement correctly in userspace C++.

    Seqlocks

    Writers increment a sequence counter before and after writing. Readers check the counter before and after reading; if it changed (or is odd), they retry. Readers never block writers. Best for small, frequently read data like timestamps.

    class SeqLock {
    std::atomic<unsigned> seq_{0};
    int data_;

public:
    int read() const {
        int result;
        unsigned s;
        do {
            s = seq_.load(std::memory_order_acquire);
            if (s & 1) continue; // Writer in progress
            result = data_;
            std::atomic_thread_fence(std::memory_order_acquire);
        } while (seq_.load(std::memory_order_relaxed) != s);
        return result;
    }

    void write(int val) {
        seq_.fetch_add(1, std::memory_order_release); // odd = writing
        data_ = val;
        seq_.fetch_add(1, std::memory_order_release); // even = done
    }
};

    Concurrent Data Structures

    Instead of locking around a standard container, consider lock-free or fine-grained locked data structures: concurrent_hash_map (Intel TBB), folly::ConcurrentHashMap, or lock-free queues.

    Summary

    • std::shared_mutex is the standard solution for the readers-writers problem in C++.
    • It provides significant throughput gains for read-heavy workloads (95%+ reads).
    • For write-heavy or balanced workloads, std::mutex is often faster due to lower per-operation overhead.
    • Atomic upgrades from shared to exclusive are not supported; release and re-acquire with a re-check is the correct pattern.
    • Writer starvation is possible in reader-preferring implementations; most POSIX implementations are writer-preferring.
    • For extreme read performance, consider RCU, seqlocks, or concurrent data structures instead of locks.
    • Always benchmark with realistic workloads. Microbenchmarks can be misleading when cache effects and contention patterns differ from production.
