    Threads: The Foundation of Concurrency

    March 2026 · 28 min read

    Creating Threads with std::thread

    The std::thread class, defined in <thread>, represents a single thread of execution. When you construct a std::thread object with a callable (function pointer, lambda, functor), the new thread begins executing that callable immediately.

    #include <iostream>
#include <thread>

void worker() {
    std::cout << "Worker running on thread "
              << std::this_thread::get_id() << "\n";
}

int main() {
    std::thread t(worker);   // thread starts NOW
    std::cout << "Main thread: " << std::this_thread::get_id() << "\n";
    t.join();                // wait for t to finish
}

    The instant the constructor completes, two threads are running: the main thread (which continues past the constructor call) and the new thread (which calls worker()). The two run concurrently. There is no guarantee which executes first or in what interleaved order.

    What Can You Pass to std::thread?

    A std::thread accepts any callable:

    // 1. Free function
void free_fn() { /* ... */ }
std::thread t1(free_fn);

// 2. Lambda
std::thread t2([]{ std::cout << "lambda\n"; });

// 3. Functor (class with operator())
struct Task {
    void operator()() const { std::cout << "functor\n"; }
};
std::thread t3(Task{});          // note: use braces, not parens
// std::thread t3(Task());       // ← WRONG! "most vexing parse"

// 4. Member function + object
struct Printer {
    void print(int n) { std::cout << n << "\n"; }
};
Printer p;
std::thread t4(&Printer::print, &p, 42);

// 5. std::function, std::bind result, etc.
auto bound = std::bind(&Printer::print, &p, 99);
std::thread t5(bound);
    Most vexing parse trap: std::thread t(Task()) declares a function, not a thread. Use std::thread t(Task{}) or std::thread t{Task()} to avoid this.

    Join and Detach: Thread Lifecycle

    Every std::thread object must be either joined or detached before it is destroyed. If the destructor runs on a joinable thread, std::terminate() is called and your program crashes. This is a deliberate design choice: it forces you to make an explicit decision about the thread's fate.

    Joining: Waiting for Completion

    Calling t.join() blocks the calling thread until t finishes execution. After join() returns, the thread object is no longer joinable:

    std::thread t([]{ heavy_computation(); });
// ... do other work while t runs ...
t.join();      // block until t finishes
// t.joinable() is now false

    Detaching: Fire and Forget

    Calling t.detach() separates the thread of execution from the std::thread object. The thread continues running in the background, and you lose all handles to it:

    std::thread t(background_logging);
t.detach();    // thread runs independently
// t.joinable() is now false
// No way to wait for it or check if it's done
    Warning: Detached threads are dangerous. If they access local variables from the creating scope, and that scope exits, you get undefined behavior (dangling references). Only detach threads that are truly self-contained and access only global or heap-allocated data with proper synchronization.

    The joinable() Check

    A std::thread is joinable if it has an associated thread of execution (i.e., it was constructed with a callable and has not been joined or detached). Always check before joining:

    if (t.joinable()) {
    t.join();
}

    RAII Guard for Threads

    Since forgetting to join or detach crashes the program, it is common to wrap std::thread in an RAII guard that joins on destruction:

    class ThreadGuard {
    std::thread& t_;

public:
    explicit ThreadGuard(std::thread& t) : t_(t) {}

    ~ThreadGuard() {
        if (t_.joinable()) t_.join();
    }

    ThreadGuard(const ThreadGuard&) = delete;
    ThreadGuard& operator=(const ThreadGuard&) = delete;
};

void safe_launch() {
    std::thread t(do_work);
    ThreadGuard guard(t);   // if an exception occurs, t is still joined
    // ... do other things ...
}   // guard destructor joins t
    C++20 solution: std::jthread does exactly this. It joins automatically in its destructor and also supports cooperative cancellation via std::stop_token. We will cover jthread in the Thread Pools post.

    Passing Arguments to Threads

    Arguments passed to std::thread are copied or moved into the new thread's internal storage, even if the target function takes a reference. This is a critical detail that catches many C++ programmers off guard.

    Passing by Value

    void process(int n, std::string s) {
    std::cout << n << ": " << s << "\n";
}

std::thread t(process, 42, "hello");
// Both 42 and "hello" are copied into the thread's storage
// Then passed (by value or move) to process()

    Passing by Reference: std::ref

    Even if your function takes a reference, std::thread will copy the argument. To pass a genuine reference, wrap it in std::ref:

    void increment(int& value) {
    ++value;
}

int x = 0;
std::thread t(increment, std::ref(x));
t.join();
// x is now 1
    Danger: When passing references to threads, you must ensure the referenced variable outlives the thread. If the thread runs after the variable is destroyed, you have a dangling reference and undefined behavior.

    Move-Only Arguments

    For types that cannot be copied (like std::unique_ptr), use std::move:

    void consume(std::unique_ptr<Widget> w) {
    w->process();
}

auto widget = std::make_unique<Widget>();
std::thread t(consume, std::move(widget));
// widget is now nullptr; ownership transferred to the thread
t.join();

    Beware Implicit Conversions

    A subtle bug: if you pass a const char* to a function expecting std::string, the conversion happens inside the new thread. If the original buffer is gone by then, you have undefined behavior:

    void greet(const std::string& name) {
    std::cout << "Hello, " << name << "\n";
}

void oops() {
    char buffer[64];
    snprintf(buffer, sizeof(buffer), "Thread-%d", 42);

    // BAD: buffer might be destroyed before the thread converts it to string
    std::thread t(greet, buffer);
    t.detach();
}

void safe() {
    char buffer[64];
    snprintf(buffer, sizeof(buffer), "Thread-%d", 42);

    // GOOD: convert to string eagerly, before thread creation
    std::thread t(greet, std::string(buffer));
    t.detach();
}

    Thread Ownership and Move Semantics

    std::thread is move-only. You cannot copy a thread, but you can transfer ownership:

    std::thread t1(do_work);
std::thread t2 = std::move(t1);  // t1 is now empty, t2 owns the thread
// t1.joinable() == false
// t2.joinable() == true

t2.join();

    This enables returning threads from functions and storing them in containers:

    std::thread make_worker(int id) {
    return std::thread([id]{ process(id); });
}

// Store multiple threads
std::vector<std::thread> workers;
for (int i = 0; i < 8; ++i) {
    workers.push_back(make_worker(i));
}

// Join all
for (auto& t : workers) {
    t.join();
}
    Never assign to a joinable thread. If t2 is already joinable and you do t2 = std::move(t1), the program calls std::terminate(). Always join or detach before reassigning.

    Thread Identification

    Every thread has a unique ID of type std::thread::id:

    std::thread t(worker);
std::thread::id tid = t.get_id();    // from thread object
// or from inside the running thread:
std::thread::id self = std::this_thread::get_id();

// IDs are comparable and hashable
std::cout << "Thread ID: " << tid << "\n";

// Default-constructed ID represents "no thread"
std::thread::id none;
if (tid != none) { /* thread is running */ }

    Hardware Concurrency

    std::thread::hardware_concurrency() returns the number of concurrent threads the hardware supports. This is typically the number of CPU cores (or hardware threads, with hyperthreading). Use it to size your thread pools:

    unsigned int n = std::thread::hardware_concurrency();
std::cout << "Hardware threads: " << n << "\n";  // e.g. 8

// A common pattern: use all available cores
unsigned int num_workers = n > 0 ? n : 4;  // fallback to 4
std::vector<std::thread> pool;
for (unsigned int i = 0; i < num_workers; ++i) {
    pool.emplace_back(worker_function, i);
}

    The std::this_thread Namespace

    This namespace provides functions that operate on the calling thread:

    // Get current thread's ID
auto id = std::this_thread::get_id();

// Sleep for a duration
std::this_thread::sleep_for(std::chrono::milliseconds(100));

// Sleep until a time point
auto wake = std::chrono::steady_clock::now() + std::chrono::seconds(2);
std::this_thread::sleep_until(wake);

// Yield: hint to the scheduler to let another thread run
std::this_thread::yield();
    On yield(): This is a hint, not a command. The OS scheduler may ignore it entirely. It is most useful in busy-wait loops to avoid monopolizing a CPU core. We will see it used in spinlock implementations in the Atomics post.

    Exceptions in Threads

    If an exception escapes a thread function (i.e., it is not caught within the thread), std::terminate() is called. The exception does not propagate to the thread that called join():

    std::thread t([]{
    throw std::runtime_error("oops");  // uncaught!
});
t.join();  // std::terminate() was already called inside t

    To safely propagate exceptions across threads, use std::exception_ptr:

    std::exception_ptr eptr;

std::thread t([&eptr]{
    try {
        risky_work();
    } catch (...) {
        eptr = std::current_exception();  // capture
    }
});

t.join();

if (eptr) {
    std::rethrow_exception(eptr);  // rethrow in main thread
}

    We will see a cleaner mechanism for this (std::future + std::promise) in the Futures and Promises post.

    Thread-Local Storage

    The thread_local keyword creates a variable with a separate instance per thread. Each thread gets its own copy, initialized independently:

    thread_local int call_count = 0;

void work() {
    ++call_count;   // each thread increments its OWN counter
    std::cout << "Thread " << std::this_thread::get_id()
              << " call_count = " << call_count << "\n";
}

int main() {
    std::thread t1(work);  // t1's call_count: 1
    std::thread t2(work);  // t2's call_count: 1
    work();                // main's call_count: 1
    t1.join();
    t2.join();
}

    Thread-local storage is useful for per-thread caches, error codes, random number generators, and avoiding synchronization on data that is naturally thread-private.

    ▶ Thread-Local Storage

    Each thread has its own copy of the thread_local variable.

    Practical Example: Parallel Accumulate

    Let us build a practical parallel algorithm: splitting an array into chunks and summing each chunk on a separate thread:

    template<typename Iterator, typename T>
T parallel_accumulate(Iterator first, Iterator last, T init) {
    const auto length = std::distance(first, last);
    if (length == 0) return init;

    const unsigned int hw = std::thread::hardware_concurrency();
    const unsigned int num_threads = std::min(hw > 0 ? hw : 2,
                                              static_cast<unsigned int>(length));
    const auto block_size = length / num_threads;

    std::vector<T> results(num_threads);
    std::vector<std::thread> threads;

    Iterator block_start = first;
    for (unsigned int i = 0; i < num_threads - 1; ++i) {
        Iterator block_end = block_start;
        std::advance(block_end, block_size);

        threads.emplace_back([block_start, block_end, &results, i]{
            results[i] = std::accumulate(block_start, block_end, T{});
        });

        block_start = block_end;
    }

    // Last block handled by this thread (no extra thread needed)
    results[num_threads - 1] = std::accumulate(block_start, last, T{});

    for (auto& t : threads) t.join();

    return std::accumulate(results.begin(), results.end(), init);
}

    This is safe because each thread writes to a different element of the results vector. No two threads ever touch the same memory, so no synchronization is needed. The final accumulate runs after all threads are joined, guaranteeing all results are visible.

    Common Mistakes

    Dangling References

    The number one thread bug: a thread captures a local variable by reference, and the scope exits before the thread finishes:

    void oops() {
    int local = 42;
    std::thread t([&local]{
        std::this_thread::sleep_for(std::chrono::seconds(1));
        std::cout << local;  // UNDEFINED BEHAVIOR: local is destroyed
    });
    t.detach();
}   // local destroyed here, but t is still using it!

    Fix: Capture by value, or ensure the variable outlives the thread (e.g., use join() instead of detach()).

    Forgetting to Join or Detach

    void bad() {
    std::thread t(work);
    // ... if an exception throws here ...
}   // t's destructor calls std::terminate()!

    Fix: Use an RAII guard (see above) or std::jthread (C++20).

    Interleaved Output

    std::cout is thread-safe in the sense that individual operations will not corrupt the stream. But multiple << operations are not atomic as a group:

    // Thread 1: std::cout << "Hello " << "World\n";
// Thread 2: std::cout << "Foo "   << "Bar\n";
// Possible output: "Hello Foo World\nBar\n"

// Fix: use a mutex, or build the string first
std::ostringstream oss;
oss << "Hello " << "World\n";
std::cout << oss.str();  // single operation

    Preview: std::jthread (C++20)

    std::jthread is a "joining thread" that fixes two pain points of std::thread:

    1. Automatic joining: The destructor calls join() if the thread is joinable. No more std::terminate() surprises.
    2. Cooperative cancellation: It carries a std::stop_source/std::stop_token pair for signaling the thread to stop.
    #include <thread>
#include <stop_token>

void worker(std::stop_token stoken) {
    while (!stoken.stop_requested()) {
        do_some_work();
    }
    // Clean exit when stop is requested
}

int main() {
    std::jthread t(worker);  // automatically gets a stop_token

    std::this_thread::sleep_for(std::chrono::seconds(5));

    // jthread destructor:
    //   1. Calls request_stop() on its stop_source
    //   2. Calls join()
    // Both happen automatically!
}

    Summary

    • std::thread starts executing immediately upon construction.
    • You must call join() or detach() before destruction, or the program terminates.
    • Arguments are copied into the thread. Use std::ref for references and std::move for move-only types.
    • Beware implicit conversions happening inside the new thread after the original buffer is gone.
    • std::thread is move-only. You can return threads from functions and store them in containers.
    • thread_local gives each thread its own copy of a variable.
    • std::jthread (C++20) adds automatic joining and cooperative cancellation.
    • Prefer join() over detach(). Detached threads with dangling references are a top source of concurrency bugs.
