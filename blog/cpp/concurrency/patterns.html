    Concurrency Patterns

    March 2026 · 28 min read

    Battle-Tested Patterns

    This final post brings together everything from the series into production-ready patterns. These are the recurring designs you will find in real-world concurrent C++ systems.

    Pattern 1: Producer-Consumer

    We covered this in Condition Variables. The canonical pattern uses a bounded queue with two condition variables:

    template<typename T>
class Channel {
    std::queue<T> queue_;
    size_t capacity_;
    std::mutex mtx_;
    std::condition_variable not_full_;
    std::condition_variable not_empty_;
    bool closed_ = false;

public:
    explicit Channel(size_t cap) : capacity_(cap) {}

    bool send(T item) {
        std::unique_lock lock(mtx_);
        not_full_.wait(lock, [&]{ return closed_ || queue_.size() < capacity_; });
        if (closed_) return false;
        queue_.push(std::move(item));
        not_empty_.notify_one();
        return true;
    }

    std::optional<T> receive() {
        std::unique_lock lock(mtx_);
        not_empty_.wait(lock, [&]{ return closed_ || !queue_.empty(); });
        if (queue_.empty()) return std::nullopt;
        T item = std::move(queue_.front());
        queue_.pop();
        not_full_.notify_one();
        return item;
    }

    void close() {
        std::lock_guard lock(mtx_);
        closed_ = true;
        not_full_.notify_all();
        not_empty_.notify_all();
    }
};

    Pattern 2: Readers-Writers

    Multiple readers can access data simultaneously, but a writer needs exclusive access. C++17's std::shared_mutex implements this directly:

    class Config {
    mutable std::shared_mutex mtx_;
    std::unordered_map<std::string, std::string> data_;

public:
    std::string get(const std::string& key) const {
        std::shared_lock lock(mtx_);  // shared (read) access
        auto it = data_.find(key);
        return it != data_.end() ? it->second : "";
    }

    void set(const std::string& key, std::string value) {
        std::unique_lock lock(mtx_);  // exclusive (write) access
        data_[key] = std::move(value);
    }
};

    Pattern 3: Active Object

    Encapsulate a thread and its message queue inside an object. All method calls are asynchronous: they post a message to the queue and return a future. The internal thread processes messages sequentially.

    class ActiveObject {
    std::queue<std::function<void()>> queue_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool done_ = false;
    std::thread thread_;

public:
    ActiveObject() : thread_([this]{ run(); }) {}

    ~ActiveObject() {
        {
            std::lock_guard lock(mtx_);
            done_ = true;
        }
        cv_.notify_one();
        thread_.join();
    }

    template<typename F>
    auto submit(F&& f) -> std::future<decltype(f())> {
        auto task = std::make_shared<std::packaged_task<decltype(f())()>>(
            std::forward<F>(f)
        );
        auto fut = task->get_future();
        {
            std::lock_guard lock(mtx_);
            queue_.push([task]{ (*task)(); });
        }
        cv_.notify_one();
        return fut;
    }

private:
    void run() {
        while (true) {
            std::function<void()> task;
            {
                std::unique_lock lock(mtx_);
                cv_.wait(lock, [&]{ return done_ || !queue_.empty(); });
                if (done_ && queue_.empty()) return;
                task = std::move(queue_.front());
                queue_.pop();
            }
            task();
        }
    }
};

    Pattern 4: Latches and Barriers (C++20)

    std::latch

    A latch is a single-use countdown. Threads call count_down() to decrement. wait() blocks until the count reaches zero:

    std::latch done(3);

for (int i = 0; i < 3; ++i) {
    threads.emplace_back([&done, i]{
        do_work(i);
        done.count_down();  // signal completion
    });
}

done.wait();  // blocks until all 3 threads have counted down
// All work is now complete

    std::barrier

    A barrier is reusable. All threads must arrive before any can proceed. The barrier then resets for the next phase:

    std::barrier sync_point(NUM_THREADS, []{
    // This completion function runs once after all threads arrive.
    // Perfect for combining partial results.
    merge_results();
});

void worker(int id) {
    for (auto phase : phases) {
        compute_phase(id, phase);
        sync_point.arrive_and_wait();  // synchronize between phases
    }
}

    ▶ Barrier Synchronization

    All threads must arrive before the barrier opens. Then it resets for the next phase.

    Pattern 5: Counting Semaphore (C++20)

    A semaphore maintains a count of available resources. acquire() decrements (blocks if zero). release() increments.

    std::counting_semaphore<3> pool(3);  // 3 permits

void access_resource() {
    pool.acquire();    // block if all 3 permits are taken
    use_resource();
    pool.release();    // return permit
}

// binary_semaphore is a counting_semaphore<1>
std::binary_semaphore signal(0);  // starts empty
// Thread A: signal.release();     // post
// Thread B: signal.acquire();     // wait for post

    Pattern 6: Dining Philosophers

    The classic concurrency puzzle: five philosophers sit around a table; each needs two forks to eat. Naive solutions deadlock. Use std::scoped_lock to prevent it:

    std::array<std::mutex, 5> forks;

void philosopher(int id) {
    int left = id;
    int right = (id + 1) % 5;

    while (true) {
        think(id);

        // scoped_lock acquires both forks deadlock-free
        std::scoped_lock lock(forks[left], forks[right]);
        eat(id);
    }   // both forks released
}

    Pattern 7: Monitor

    A monitor bundles data with its synchronization. Every public method locks the mutex:

    class MonitorStack {
    std::stack<int> data_;
    mutable std::mutex mtx_;
    std::condition_variable cv_;

public:
    void push(int val) {
        std::lock_guard lock(mtx_);
        data_.push(val);
        cv_.notify_one();
    }

    int pop() {
        std::unique_lock lock(mtx_);
        cv_.wait(lock, [&]{ return !data_.empty(); });
        int val = data_.top();
        data_.pop();
        return val;
    }

    bool empty() const {
        std::lock_guard lock(mtx_);
        return data_.empty();
    }
};

    Pattern 8: Double Buffering

    Use two buffers: one for writing (back buffer) and one for reading (front buffer). Swap them atomically when the writer is done. Readers never block on the writer:

    class DoubleBuffer {
    std::array<std::vector<int>, 2> buffers_;
    std::atomic<int> front_{0};

public:
    // Writer fills back buffer, then swaps
    void write(const std::vector<int>& data) {
        int back = 1 - front_.load(std::memory_order_relaxed);
        buffers_[back] = data;
        front_.store(back, std::memory_order_release);
    }

    // Reader always reads front buffer (no locking!)
    const std::vector<int>& read() const {
        int f = front_.load(std::memory_order_acquire);
        return buffers_[f];
    }
};

    Choosing the Right Pattern

    ScenarioPattern
    Work queue with workersProducer-Consumer / Thread Pool
    Read-heavy, write-rare dataReaders-Writers (shared_mutex)
    Async method calls on an objectActive Object
    Phased parallel computationBarrier
    Wait for N things to finishLatch
    Limit concurrent accessCounting Semaphore
    Real-time data displayDouble Buffer

    Series Summary

    Over these 15 posts, we have covered the entire C++ concurrency landscape:

    1. Foundations: Threads, data races, mutexes, deadlocks
    2. Coordination: Condition variables, futures, async
    3. Low-level: Atomics, memory model, lock-free data structures
    4. High-level: Thread pools, parallel algorithms, coroutines
    5. Patterns: Producer-consumer, barriers, semaphores, active object, double buffering

    The key takeaways:

    • Minimize shared mutable state. The less sharing, the fewer bugs.
    • Use RAII for locks. lock_guard, unique_lock, scoped_lock.
    • Test with ThreadSanitizer. Compile with -fsanitize=thread.
    • Start simple. Use mutexes and condition variables. Move to atomics and lock-free only when measurements justify it.
    • Think in terms of patterns, not individual primitives.
