    Condition Variables

    March 2026 · 24 min read

    Why Condition Variables?

    Mutexes protect shared data, but what if a thread needs to wait for a specific condition to become true? For example, a consumer thread needs to wait until the queue is non-empty, or a thread needs to wait until a computation result is ready.

    The naive approach is busy-waiting (polling):

    // BAD: busy-waiting wastes CPU
while (true) {
    std::lock_guard lock(mtx);
    if (!queue.empty()) {
        auto item = queue.front();
        queue.pop();
        process(item);
        break;
    }
    // unlock, then immediately lock again... spinning!
}

    This burns CPU cycles doing nothing useful. A condition variable solves this by allowing a thread to sleep until it is notified that the condition may have changed.

    How Condition Variables Work

    A std::condition_variable provides three operations:

    • wait(lock): Atomically releases the lock and puts the thread to sleep. When notified, it reacquires the lock before returning.
    • notify_one(): Wakes up one waiting thread.
    • notify_all(): Wakes up all waiting threads.

    The critical detail: wait() atomically releases the mutex and sleeps. This is essential to avoid a race condition where a notification arrives between the unlock and the sleep.

    #include <condition_variable>
#include <mutex>
#include <queue>

std::mutex mtx;
std::condition_variable cv;
std::queue<int> data_queue;

void producer() {
    for (int i = 0; i < 10; ++i) {
        {
            std::lock_guard lock(mtx);
            data_queue.push(i);
        }   // unlock before notify (good practice)
        cv.notify_one();   // wake one consumer
    }
}

void consumer() {
    while (true) {
        std::unique_lock lock(mtx);

        // Wait until queue is non-empty
        cv.wait(lock, []{ return !data_queue.empty(); });

        int item = data_queue.front();
        data_queue.pop();
        lock.unlock();     // unlock before processing
        process(item);
    }
}

    ▶ Producer-Consumer with Condition Variable

    Watch the producer push items and notify the consumer, which waits efficiently.

    Spurious Wakeups

    A condition variable may wake a thread even when no notification was sent. These are called spurious wakeups, and they are allowed by the POSIX and C++ standards because they simplify the implementation on some platforms.

    This is why you must always use a predicate (the lambda version of wait()). The predicate is checked in a loop:

    // Under the hood, cv.wait(lock, predicate) does this:
while (!predicate()) {
    cv.wait(lock);
}
// It re-checks the predicate after every wakeup (real or spurious)
    Never write cv.wait(lock) without a predicate loop. Without a predicate, a spurious wakeup will make your thread proceed when the condition is not actually met, leading to bugs. Always use the predicate overload or wrap in a while loop.

    notify_one() vs notify_all()

    Method Behavior Use When
    notify_one()Wakes exactly one waiting thread (if any).Exactly one thread can make progress (e.g., one item added to queue).
    notify_all()Wakes all waiting threads. They compete for the lock; most re-sleep after checking the predicate.Multiple threads might need to respond (e.g., shutdown signal, or the condition type changed).

    Timed Waits

    You can wait with a timeout using wait_for() or wait_until():

    std::unique_lock lock(mtx);

// Wait up to 1 second
if (cv.wait_for(lock, std::chrono::seconds(1),
                []{ return !queue.empty(); }))
{
    // Condition met within timeout
    auto item = queue.front();
    queue.pop();
} else {
    // Timed out: condition not met after 1 second
}

    The Producer-Consumer Pattern

    This is the most common use of condition variables. A bounded buffer (fixed-size queue) with producers adding items and consumers removing them. The producer waits when the buffer is full; the consumer waits when the buffer is empty:

    template<typename T>
class BoundedQueue {
    std::queue<T> queue_;
    const size_t capacity_;
    std::mutex mtx_;
    std::condition_variable not_full_;   // producers wait on this
    std::condition_variable not_empty_;  // consumers wait on this

public:
    explicit BoundedQueue(size_t cap) : capacity_(cap) {}

    void push(T item) {
        std::unique_lock lock(mtx_);
        not_full_.wait(lock, [this]{ return queue_.size() < capacity_; });
        queue_.push(std::move(item));
        not_empty_.notify_one();
    }

    T pop() {
        std::unique_lock lock(mtx_);
        not_empty_.wait(lock, [this]{ return !queue_.empty(); });
        T item = std::move(queue_.front());
        queue_.pop();
        not_full_.notify_one();
        return item;
    }
};

    ▶ Bounded Buffer: Producer-Consumer

    A buffer of capacity 3. Producer waits when full, consumer waits when empty.

    std::condition_variable_any

    std::condition_variable works only with std::unique_lock<std::mutex>. If you need to wait on a different lockable type (e.g., shared_lock or a custom lock), use std::condition_variable_any:

    std::shared_mutex smtx;
std::condition_variable_any cv_any;

void reader_wait() {
    std::shared_lock lock(smtx);
    cv_any.wait(lock, []{ return data_ready; });
    // ... read data ...
}

    The Lost Wakeup Problem

    If notify_one() is called before any thread is waiting, the notification is lost. The next thread to call wait() will sleep indefinitely (unless another notification arrives).

    Fix: The predicate-based wait() handles this naturally. If the condition is already true when wait() is called, it returns immediately without sleeping.

    Common Mistakes

    • Forgetting the predicate: Raw cv.wait(lock) without a predicate loop is vulnerable to spurious wakeups and lost wakeups.
    • Notifying while holding the lock: Not a correctness bug, but can cause a "hurry up and wait" situation where the notified thread wakes up only to immediately block on the mutex. Notify after unlocking for better performance.
    • Using lock_guard with cv.wait(): Will not compile. wait() needs to unlock and relock the mutex, which requires unique_lock.

    Summary

    • Condition variables let threads sleep efficiently until a condition changes, avoiding busy-waiting.
    • wait(lock, predicate) atomically releases the lock and sleeps. It re-checks the predicate on every wakeup.
    • Always use a predicate to handle spurious wakeups.
    • notify_one() wakes one waiter; notify_all() wakes all waiters.
    • The bounded producer-consumer queue is the canonical application.

    Condition variables work with mutexes to coordinate threads. But sometimes you need faster coordination on simple values like counters and flags. That is where Atomic Operations come in.
