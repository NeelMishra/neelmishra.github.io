Passing Arguments to Threads

March 2026

Threads are rarely self-contained. They need input data, output locations, and shared context. The way std::thread handles argument passing has several subtle behaviors that differ from ordinary function calls, and misunderstanding them is one of the most common sources of concurrency bugs.

This page covers the internal mechanics of how arguments flow from the calling thread to the new thread, what std::decay does to your types, every passing strategy (by value, by reference, by move), the implicit conversion trap, member function binding, and a catalog of dangling reference scenarios.

How Argument Passing Works Internally

When you write std::thread t(f, arg1, arg2), the constructor performs a two-phase process:

  1. Phase 1 (calling thread): Each argument is decay-copied or decay-moved into internal storage owned by the thread object. This storage is typically a heap-allocated tuple.
  2. Phase 2 (new thread): When the new thread starts, the stored copies are forwarded to the callable via std::invoke.

This two-phase model has a critical consequence: the types of the stored arguments may differ from what you passed (thanks to std::decay), and any conversions from the stored type to the function parameter type happen on the new thread, not the calling thread.

std::decay and std::invoke: The Machinery

The thread constructor applies std::decay_t to every argument before storing it. std::decay models what happens when you pass an argument by value to a function:

Original TypeAfter std::decayWhy
int&intStrips references
const int&intStrips references and cv
int[10]int*Array to pointer
const char[6]const char*String literal to pointer
void(int)void(*)(int)Function to pointer
std::stringstd::stringNo change (already a value)

After decay-copying, std::invoke calls the stored callable with the stored arguments. std::invoke handles function pointers, member function pointers (with the object as first argument), and regular callables uniformly.

Why this matters

Because references are stripped by decay, passing a variable to a thread always creates an independent copy unless you explicitly wrap it with std::ref or std::cref. The wrapper type std::reference_wrapper survives decay because it is a class type, not a reference. 

// Without std::ref: decay strips the reference, thread gets a copy
void inc(int& x) { ++x; }
int val = 0;
// std::thread t(inc, val);  // COMPILE ERROR: trying to bind rvalue to non-const ref

// With std::ref: decay preserves std::reference_wrapper<int>
std::thread t(inc, std::ref(val));  // OK: std::invoke unwraps the reference
t.join();
// val == 1

Passing by Value

By default, all arguments are copied into the thread. Even if the target function takes a const reference, the thread stores its own copy: 

void process(const std::string& data) {
    std::cout << data << "\n";
}

int main() {
    std::string input = "sensor data";
    std::thread t(process, input);
    // input is COPIED into thread storage.
    // The const ref in process() binds to that internal copy, NOT to input.
    t.join();
}

This is generally safe because each thread works with its own copy. No data race is possible on the argument itself.

Verifying the copy

void modify(std::string s) {
    s += " modified";
    std::cout << "Thread sees: " << s << "\n";
}

int main() {
    std::string original = "base";
    std::thread t(modify, original);
    t.join();
    std::cout << "Main sees: " << original << "\n";
    // Output:
    //   Thread sees: base modified
    //   Main sees: base
}

Passing by Reference (std::ref)

When you need the thread to modify a variable in the calling scope or avoid an expensive copy, use std::ref (or std::cref for const): 

#include <functional>  // for std::ref

void accumulate(const std::vector<int>& data, int& result) {
    result = 0;
    for (int x : data) result += x;
}

int main() {
    std::vector<int> numbers = {1, 2, 3, 4, 5};
    int sum = 0;

    std::thread t(accumulate, std::cref(numbers), std::ref(sum));
    t.join();

    std::cout << "Sum: " << sum << "\n";  // Sum: 15
}
Why std::ref is required. The thread constructor applies std::decay to arguments, which strips references. Without std::ref, a non-const int& parameter fails to compile because the thread tries to bind a temporary (the decayed copy) to a non-const lvalue reference. std::ref wraps the reference in a std::reference_wrapper, which is a copyable class that survives decay.

What happens without std::ref

void set_flag(bool& flag) { flag = true; }

int main() {
    bool done = false;
    // std::thread t(set_flag, done);
    // COMPILE ERROR: cannot bind rvalue to non-const lvalue reference

    std::thread t(set_flag, std::ref(done));  // OK
    t.join();
    std::cout << std::boolalpha << done << "\n"; // true
}

The Implicit Conversion Trap

Because argument copying (Phase 1) happens on the calling thread but the conversion to the function parameter type (Phase 2) happens on the new thread, implicit conversions can create dangling pointers. This is the single most dangerous subtlety of std::thread argument passing. 

void process(const std::string& s) {
    // use s...
}

void dangerous(int param) {
    char buffer[64];
    std::snprintf(buffer, sizeof(buffer), "data-%d", param);

    std::thread t(process, buffer);
    t.detach();
    // BUG: buffer (char*) is decay-copied as a POINTER.
    // The conversion to std::string happens on the NEW thread.
    // By then, buffer may be destroyed (dangerous() returned).
}

void safe(int param) {
    char buffer[64];
    std::snprintf(buffer, sizeof(buffer), "data-%d", param);

    std::thread t(process, std::string(buffer));  // convert NOW
    t.detach();
    // OK: std::string is constructed on the calling thread,
    // then the string is decay-copied (or moved) into the tuple.
}

The timeline of the bug

  1. Calling thread: buffer is on the stack. std::decay<char[64]> yields char*. The pointer is copied into the tuple.
  2. Calling thread: dangerous() returns. The stack frame (including buffer) is destroyed.
  3. New thread: std::invoke tries to convert the stored char* to const std::string&. The pointer now points to freed stack memory. Undefined behavior.
Rule. When passing string literals or char* to a thread function expecting std::string, always convert explicitly at the call site. Never rely on implicit conversion across thread boundaries.

Other implicit conversions that can bite

// Integer narrowing: the int is stored, then converted to short on the new thread.
// Not dangerous (no pointer) but potentially surprising.
void worker(short val) { /* ... */ }
int x = 300;
std::thread t(worker, x);  // stored as int, narrowed to short on new thread
t.join();

// Pointer from temporary: extremely dangerous
void use_data(const std::vector<int>* ptr) { /* ... */ }
// std::thread t(use_data, &std::vector<int>{1,2,3});  // dangling!

Move-Only Types

Types like std::unique_ptr, std::promise, and std::packaged_task cannot be copied. You must std::move them into the thread: 

void take_ownership(std::unique_ptr<int> p) {
    std::cout << "Thread got: " << *p << "\n";
}

int main() {
    auto ptr = std::make_unique<int>(42);

    // std::thread t(take_ownership, ptr);  // COMPILE ERROR: can't copy unique_ptr
    std::thread t(take_ownership, std::move(ptr));  // OK

    // ptr is now nullptr on the main thread
    t.join();
}

Moving a promise into a thread

void compute(std::promise<int> p) {
    p.set_value(42);
}

int main() {
    std::promise<int> p;
    auto f = p.get_future();
    std::thread t(compute, std::move(p));
    std::cout << "Result: " << f.get() << "\n";
    t.join();
}

Moving large containers

std::vector<double> data(1'000'000, 3.14);
std::thread t([](std::vector<double> d) {
    double sum = 0;
    for (double x : d) sum += x;
    std::cout << "Sum: " << sum << "\n";
}, std::move(data));
// data is now empty on the main thread
t.join();

Dangling References: The Full Picture

Dangling references are the most dangerous pitfall when passing arguments to threads. Here is a comprehensive catalog of scenarios:

Scenario 1: Detached thread with stack reference

void bad_detach() {
    int local = 10;
    std::thread t([&local]() {
        std::this_thread::sleep_for(std::chrono::seconds(1));
        std::cout << local << "\n";  // UNDEFINED BEHAVIOR
    });
    t.detach();
}   // local destroyed here, but thread still running

Scenario 2: Pointer to local in a loop

std::vector<std::thread> threads;
for (int i = 0; i < 5; ++i) {
    // BAD: &i captures address of loop variable, which changes each iteration
    // threads.emplace_back([&i](){ std::cout << i; });

    // GOOD: capture by value
    threads.emplace_back([i](){ std::cout << i; });
}
for (auto& t : threads) t.join();

Scenario 3: Implicit conversion of char* (detailed)

As shown above, passing a char* to a function expecting std::string defers the conversion. If the buffer is on the stack and the thread is detached, the char* points to garbage by the time conversion happens.

Scenario 4: Reference to member of a temporary

struct Config { int timeout = 30; };

void worker(const int& timeout) {
    std::this_thread::sleep_for(std::chrono::seconds(1));
    std::cout << timeout << "\n";  // may be dangling
}

// Config() creates a temporary. &Config().timeout is a dangling pointer
// after the full expression ends.
// std::thread t(worker, std::cref(Config().timeout));  // DANGEROUS

Member Functions with Object Pointers

When launching a member function on a thread, the second argument is the object on which the function is called. You can pass:

  • A raw pointer: &obj
  • A reference wrapper: std::ref(obj)
  • A smart pointer: shared_ptr
  • A copy of the object (if copyable)

Complete example with all binding modes

class Logger {
public:
    void log(const std::string& msg) {
        std::cout << "[LOG] " << msg << "\n";
    }
};

int main() {
    Logger logger;

    // Using raw pointer (caller must ensure lifetime)
    std::thread t1(&Logger::log, &logger, "via pointer");
    t1.join();

    // Using shared_ptr (safe for detached threads)
    auto shared_logger = std::make_shared<Logger>();
    std::thread t2(&Logger::log, shared_logger, "via shared_ptr");
    t2.detach();  // safe: shared_ptr extends lifetime

    // Using std::ref (same as raw pointer semantically)
    std::thread t3(&Logger::log, std::ref(logger), "via ref");
    t3.join();

    // Passing a copy of the object
    std::thread t4(&Logger::log, logger, "via copy");
    t4.join();
}

How std::invoke handles member functions

std::invoke(&C::fn, obj_ptr, args...) is equivalent to (obj_ptr->*&C::fn)(args...). If the second argument is a reference_wrapper, it is unwrapped. If it is a raw value (not a pointer), std::invoke calls the member function on the copy.

Prefer shared_ptr for detached threads. If you detach a thread that operates on an object, the object must outlive the thread. A std::shared_ptr guarantees this by extending the object's lifetime through shared ownership.

Large Objects and Performance

Since the thread constructor copies arguments, passing large objects can be expensive. The three strategies in order of preference:

  1. Move: If the caller no longer needs the data, move it. Zero-copy transfer of internal buffers.
  2. std::ref: If both threads need access and synchronization is handled. Zero copies total.
  3. shared_ptr: For shared ownership across multiple threads. Reference-counted, thread-safe pointer.
// Strategy 1: Move
std::vector<double> data(1'000'000, 3.14);
std::thread t([](std::vector<double> d) {
    // d is owned by this thread
}, std::move(data));
t.join();

// Strategy 2: Reference (must synchronize or join first)
std::vector<double> data2(1'000'000, 2.71);
std::thread t2([](const std::vector<double>& d) {
    // read-only access to the original
}, std::cref(data2));
t2.join();

// Strategy 3: Shared pointer
auto data3 = std::make_shared<std::vector<double>>(1'000'000, 1.41);
std::thread t3([](std::shared_ptr<std::vector<double>> d) {
    // shared ownership
}, data3);
t3.join();

Measuring the copy cost

#include <chrono>
#include <thread>
#include <vector>
#include <iostream>

int main() {
    std::vector<int> big(10'000'000, 1);
    using Clock = std::chrono::high_resolution_clock;

    // By copy
    auto t0 = Clock::now();
    std::thread tc([](std::vector<int> v){ }, big);
    tc.join();
    auto t1 = Clock::now();

    // By move
    auto t2 = Clock::now();
    auto big2 = big;  // make a copy to move from
    std::thread tm([](std::vector<int> v){ }, std::move(big2));
    tm.join();
    auto t3 = Clock::now();

    auto ms = [](auto d){ return std::chrono::duration_cast<std::chrono::milliseconds>(d).count(); };
    std::cout << "Copy: " << ms(t1 - t0) << " ms\n";
    std::cout << "Move: " << ms(t3 - t2) << " ms\n";
    // Typical result: Copy ~20 ms, Move ~0 ms (for 40 MB vector)
}

Common Mistakes

1. Forgetting std::ref for output parameters

void compute(int& result) { result = 42; }
int r = 0;
// std::thread t(compute, r);           // COMPILE ERROR
std::thread t(compute, std::ref(r));    // OK
t.join();

2. Passing string literals to detached threads

void log(const std::string& msg) { /* ... */ }
void bad() {
    char buf[] = "temp";
    std::thread t(log, buf);       // stores char*, not string
    t.detach();                     // buf is destroyed; UB on new thread
}
void good() {
    char buf[] = "temp";
    std::thread t(log, std::string(buf));  // convert immediately
    t.detach();
}

3. Passing this by reference to a detached thread

class Worker {
public:
    void start() {
        std::thread t(&Worker::run, this);
        t.detach();
        // If the Worker object is destroyed before run() finishes: UB
    }
private:
    void run() { /* uses member variables */ }
};
// Fix: use shared_from_this() or ensure the object outlives the thread

4. Using std::ref with a temporary

// std::thread t(process, std::ref(std::string("temp")));
// COMPILE ERROR: cannot bind reference to temporary
// (This is actually caught at compile time, which is good)

5. Race condition on std::ref argument

int counter = 0;
std::thread t1([](int& c){ for(int i=0;i<1000;++i) ++c; }, std::ref(counter));
std::thread t2([](int& c){ for(int i=0;i<1000;++i) ++c; }, std::ref(counter));
t1.join(); t2.join();
// counter is likely NOT 2000 due to data race. Need atomic or mutex.

Summary Table

Method Syntax Thread gets Safe with detach? Performance
By value (default) thread(f, arg) Independent copy Yes Copy cost
By reference thread(f, std::ref(arg)) Reference to original No (unless lifetime guaranteed) Zero copy
By const reference thread(f, std::cref(arg)) Const reference to original No (unless lifetime guaranteed) Zero copy
By move thread(f, std::move(arg)) Moved-from ownership Yes Move cost (often O(1))
Pointer to object thread(&C::fn, &obj, ...) Raw pointer No (unless lifetime managed) No copy
shared_ptr to object thread(&C::fn, sptr, ...) Shared ownership Yes Ref count increment
Default strategy. Pass by value or move. Use std::ref only when you have a clear synchronization story and the referenced object is guaranteed to outlive the thread.
