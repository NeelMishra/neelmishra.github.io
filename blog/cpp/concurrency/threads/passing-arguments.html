Passing Arguments to Threads

March 2026

Threads are rarely self-contained. They need input data, output locations, and shared context. The way std::thread handles argument passing has several subtle behaviors that differ from ordinary function calls, and misunderstanding them is one of the most common sources of concurrency bugs.

How Argument Passing Works Internally

When you write std::thread t(f, arg1, arg2), the constructor does the following:

  1. Copies or moves each argument into internal storage owned by the thread object.
  2. When the new thread starts, these stored copies are forwarded to the callable.

This two-phase mechanism is critical to understand. The arguments are copied in the calling thread's context, then passed to the callable on the new thread. This means conversions, copies, and moves all happen at construction time, not at invocation time.

Passing by Value

By default, all arguments are copied into the thread. Even if the target function takes a const reference, the thread stores its own copy: 

void process(const std::string& data) {
    std::cout << data << "\n";
}

int main() {
    std::string input = "sensor data";
    std::thread t(process, input);
    // input is COPIED into the thread's internal storage.
    // The reference in process() binds to that internal copy, NOT to input.
    t.join();
}

This is generally safe because each thread works with its own copy. No data race is possible on the argument itself.

Verifying the copy

void modify(std::string s) {
    s += " modified";
    std::cout << "Thread sees: " << s << "\n";
}

int main() {
    std::string original = "base";
    std::thread t(modify, original);
    t.join();
    std::cout << "Main sees: " << original << "\n";
    // Output:
    // Thread sees: base modified
    // Main sees: base
}

Passing by Reference (std::ref)

Sometimes you want the thread to modify a variable in the calling scope or avoid an expensive copy. You must explicitly use std::ref (or std::cref for const references): 

#include <functional>  // for std::ref

void accumulate(const std::vector<int>& data, int& result) {
    result = 0;
    for (int x : data) result += x;
}

int main() {
    std::vector<int> numbers = {1, 2, 3, 4, 5};
    int sum = 0;

    std::thread t(accumulate, std::cref(numbers), std::ref(sum));
    t.join();

    std::cout << "Sum: " << sum << "\n";  // Sum: 15
}
Why std::ref is required. The thread constructor uses std::decay on arguments, which strips references. Without std::ref, a const int& parameter will receive a copy, and a non-const int& parameter will fail to compile because the thread tries to bind a temporary to a non-const lvalue reference.

What happens without std::ref

void set_flag(bool& flag) { flag = true; }

int main() {
    bool done = false;
    // std::thread t(set_flag, done);
    // COMPILE ERROR: cannot bind rvalue to non-const lvalue reference

    std::thread t(set_flag, std::ref(done));  // OK
    t.join();
    std::cout << std::boolalpha << done << "\n"; // true
}

The Implicit Conversion Trap

Because argument copying happens on the calling thread but the conversion to the parameter type happens on the new thread, implicit conversions can lead to dangling pointers: 

void process(const std::string& s) {
    // use s...
}

void dangerous(int param) {
    char buffer[64];
    std::snprintf(buffer, sizeof(buffer), "data-%d", param);

    std::thread t(process, buffer);
    t.detach();
    // BUG: buffer (a char*) is copied into the thread as a POINTER.
    // The conversion to std::string happens on the NEW thread.
    // By then, buffer may be destroyed (dangerous() returned).
}

void safe(int param) {
    char buffer[64];
    std::snprintf(buffer, sizeof(buffer), "data-%d", param);

    std::thread t(process, std::string(buffer));  // explicit conversion NOW
    t.detach();
    // OK: std::string is constructed on the calling thread, then copied.
}
Rule of thumb. When passing string literals or char* to a thread function that expects std::string, always convert explicitly at the call site. Never rely on implicit conversion across thread boundaries.

Move-Only Types

Types like std::unique_ptr, std::promise, and std::packaged_task cannot be copied. You must std::move them into the thread: 

void take_ownership(std::unique_ptr<int> p) {
    std::cout << "Thread got: " << *p << "\n";
}

int main() {
    auto ptr = std::make_unique<int>(42);

    // std::thread t(take_ownership, ptr);  // COMPILE ERROR: can't copy unique_ptr
    std::thread t(take_ownership, std::move(ptr));  // OK

    // ptr is now nullptr on the main thread
    t.join();
}

The same applies to std::promise, which is frequently moved into threads to communicate results back: 

void compute(std::promise<int> p) {
    int result = 42;
    p.set_value(result);
}

int main() {
    std::promise<int> p;
    auto f = p.get_future();

    std::thread t(compute, std::move(p));
    std::cout << "Result: " << f.get() << "\n";
    t.join();
}

Dangling References: The Full Picture

Dangling references are the most dangerous pitfall when passing arguments to threads. Here is a comprehensive list of scenarios:

Scenario 1: Detached thread with stack reference

void bad_detach() {
    int local = 10;
    std::thread t([&local]() {
        std::this_thread::sleep_for(std::chrono::seconds(1));
        std::cout << local << "\n";  // UNDEFINED BEHAVIOR
    });
    t.detach();
}   // local destroyed here, but thread still running

Scenario 2: std::ref to a temporary

void process(int& val) { val *= 2; }

void bad_ref() {
    // std::thread t(process, std::ref(42));
    // Won't compile, but the intent is wrong regardless.
    // Even if you work around it, the temporary is gone.
}

Scenario 3: Pointer to local in a loop

std::vector<std::thread> threads;
for (int i = 0; i < 5; ++i) {
    // BAD: &i captures address of loop variable
    // threads.emplace_back([&i](){ std::cout << i; });

    // GOOD: capture by value
    threads.emplace_back([i](){ std::cout << i; });
}
for (auto& t : threads) t.join();

Scenario 4: Implicit conversion of char* (detailed)

As shown above, passing a char* to a function expecting std::string defers the conversion. If the buffer is on the stack and the thread is detached, the char* points to garbage by the time conversion happens.

Member Functions with Object Pointers

When launching a member function on a thread, the second argument is the object on which the function is called. You can pass:

  • A raw pointer: &obj
  • A reference wrapper: std::ref(obj)
  • A smart pointer: shared_ptr
  • A copy of the object (if copyable)
class Logger {
public:
    void log(const std::string& msg) {
        std::cout << "[LOG] " << msg << "\n";
    }
};

int main() {
    Logger logger;

    // Using raw pointer
    std::thread t1(&Logger::log, &logger, "via pointer");
    t1.join();

    // Using shared_ptr (safe for detached threads)
    auto shared_logger = std::make_shared<Logger>();
    std::thread t2(&Logger::log, shared_logger, "via shared_ptr");
    t2.detach();  // safe: shared_ptr keeps Logger alive

    // Passing a copy of the object
    std::thread t3(&Logger::log, logger, "via copy");
    t3.join();
}
Prefer shared_ptr for detached threads. If you detach a thread that operates on an object, the object must outlive the thread. A std::shared_ptr guarantees this by extending the object's lifetime.

Large Objects and Performance

Since the thread constructor copies arguments, passing large objects can be expensive. Strategies:

  • Move: If the caller does not need the object anymore, move it in.
  • std::ref: If both threads need access and synchronization is handled.
  • Shared pointer: For shared ownership semantics.
// Moving a large vector into a thread
std::vector<double> data(1'000'000, 3.14);

std::thread t([](std::vector<double> d) {
    // d is owned by this thread now
    double sum = 0;
    for (double x : d) sum += x;
    std::cout << "Sum: " << sum << "\n";
}, std::move(data));

// data is now empty on the main thread
t.join();

Summary Table

Method Syntax Thread gets Safe with detach?
By value (default) thread(f, arg) Independent copy Yes
By reference thread(f, std::ref(arg)) Reference to original No (unless lifetime guaranteed)
By move thread(f, std::move(arg)) Moved-from ownership Yes
Pointer to object thread(&C::fn, &obj, args...) Raw pointer (no ownership) No (unless lifetime guaranteed)
shared_ptr to object thread(&C::fn, sptr, args...) Shared ownership Yes
Final advice. Default to pass-by-value or move. Use std::ref only when you have a clear synchronization story and the referenced object is guaranteed to outlive the thread.
