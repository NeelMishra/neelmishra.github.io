Thread Creation & Lifecycle

March 2026

Every concurrent C++ program starts with the same fundamental operation: creating a thread. The std::thread class (defined in <thread>) is the standard mechanism for launching a new thread of execution. Understanding the different ways to construct a thread, its lifecycle states, and the consequences of mismanaging that lifecycle is essential before tackling higher-level concurrency patterns.

Creating Threads

The std::thread constructor accepts any callable entity followed by its arguments. The callable is invoked on the new thread immediately after construction. Four primary callable types are used in practice: function pointers, lambda expressions, functors, and member function pointers.

Function Pointers

The simplest form. Pass a free function (or a static member function) by name: 

#include <iostream>
#include <thread>

void greet(int id) {
    std::cout << "Hello from thread " << id << "\n";
}

int main() {
    std::thread t1(greet, 1);   // calls greet(1) on new thread
    std::thread t2(greet, 2);   // calls greet(2) on new thread

    t1.join();
    t2.join();
}
Key point. Arguments are copied into the new thread by default. Even if the function signature takes a reference, the thread constructor will still copy the argument unless you explicitly wrap it with std::ref.

Lambda Expressions

Lambdas are the most common way to launch threads in modern C++. They allow inline definition of the work unit and flexible capture of surrounding state: 

int main() {
    int work_id = 42;

    // Capture by value
    std::thread t1([work_id]() {
        std::cout << "Processing work " << work_id << "\n";
    });

    // Capture by reference (careful with lifetime!)
    std::string message = "urgent";
    std::thread t2([&message]() {
        std::cout << "Message: " << message << "\n";
    });

    t1.join();
    t2.join();  // safe only because we join before message goes out of scope
}
Dangling reference trap. If you capture local variables by reference and the creating thread exits or the variable goes out of scope before the spawned thread finishes, you get undefined behavior. Always prefer capture by value or ensure join happens before the captured variable is destroyed.

Lambda with parameters (passed through the std::thread constructor):

auto worker = [](int start, int end) {
    for (int i = start; i < end; ++i) {
        // process chunk [start, end)
    }
};

std::thread t(worker, 0, 1000);
t.join();

Functors (Callable Objects)

A functor is any object with an overloaded operator(). This is useful when the thread needs to carry internal state: 

class Downloader {
public:
    explicit Downloader(const std::string& url) : url_(url) {}

    void operator()() const {
        std::cout << "Downloading: " << url_ << "\n";
        // ... actual download logic ...
    }

private:
    std::string url_;
};

int main() {
    Downloader dl("https://example.com/data.bin");
    std::thread t(dl);        // copies the functor
    t.join();

    // Move a temporary directly
    std::thread t2(Downloader("https://example.com/other.bin"));
    t2.join();
}
Watch out. If you write std::thread t(Downloader()); with empty parentheses, the compiler interprets this as a function declaration, not a thread construction. This is the "most vexing parse." Fix it with braces: std::thread t(Downloader{}); or std::thread t{Downloader{}};.

Member Functions

To run a non-static member function on a thread, pass a pointer to the member function followed by a pointer (or reference) to the object instance: 

class Sensor {
public:
    void read_data(int samples) {
        for (int i = 0; i < samples; ++i) {
            // read sensor
        }
        std::cout << "Read " << samples << " samples\n";
    }
};

int main() {
    Sensor s;
    // Pass &Sensor::read_data, then the object pointer, then arguments
    std::thread t(&Sensor::read_data, &s, 100);
    t.join();
}

The object pointer (&s) is passed as the implicit this. You can also pass a std::shared_ptr to guarantee the object stays alive: 

auto sensor = std::make_shared<Sensor>();
std::thread t(&Sensor::read_data, sensor, 100);
t.join();

Join vs Detach

After creating a thread, you must do one of two things before the std::thread object is destroyed: join it or detach it.

join()

Blocks the calling thread until the spawned thread finishes execution. After join() returns, the std::thread object is no longer joinable. 

std::thread t(some_work);
// ... do other stuff on the main thread ...
t.join();  // blocks here until some_work completes
// t.joinable() == false after this point

detach()

Separates the thread of execution from the std::thread object. The thread continues running in the background. You lose all ability to join or query it. The detached thread will be cleaned up by the runtime when it finishes. 

std::thread t(background_logging);
t.detach();
// t.joinable() == false
// The logging thread runs independently; you cannot wait for it
Detach is dangerous. If the detached thread accesses any local variables from the launching scope, those variables may be destroyed before the detached thread finishes. This leads to undefined behavior. Detached threads should only access global or heap-allocated data they own or that outlives them.

The Joinable Check

Calling join() or detach() on a non-joinable thread throws std::system_error. Always check first: 

if (t.joinable()) {
    t.join();
}

A thread is not joinable if:

  • It was default-constructed (no callable provided)
  • It has been join()ed already
  • It has been detach()ed already
  • It has been moved from (ownership transferred)

Thread Lifecycle

▶ Thread State Machine

Step through the lifecycle of a std::thread object from construction to destruction.

The lifecycle follows a strict sequence:

  1. Construction: A callable and arguments are provided. The OS creates a new thread and execution begins.
  2. Running: The callable executes on the new thread. The std::thread object is joinable.
  3. Termination: The callable returns (or an uncaught exception propagates). The thread finishes, but resources are held until join or detach.
  4. Join/Detach: The owning code explicitly calls join() or detach().
  5. Destruction: The std::thread destructor runs. If still joinable, std::terminate is called.

Destruction Without Join

This is the single most critical rule of std::thread: if the destructor runs while the thread is still joinable, the program is terminated. There is no exception, no graceful fallback. 

void dangerous() {
    std::thread t([]() {
        std::this_thread::sleep_for(std::chrono::seconds(5));
    });
    // t goes out of scope without join or detach
    // RESULT: std::terminate() is called, program crashes
}

Why does C++ not just automatically join? Because an implicit join could silently block for an arbitrary amount of time, hiding bugs. And an implicit detach could cause the thread to access destroyed resources. The committee decided that crashing is preferable to silently incorrect behavior.

Ensuring safety with RAII

The standard pattern is to wrap the thread in a guard class that joins on destruction. C++20 provides this as std::jthread, but you can implement it yourself: 

class thread_guard {
public:
    explicit thread_guard(std::thread& t) : t_(t) {}
    ~thread_guard() {
        if (t_.joinable()) {
            t_.join();
        }
    }
    thread_guard(const thread_guard&) = delete;
    thread_guard& operator=(const thread_guard&) = delete;
private:
    std::thread& t_;
};

void safe() {
    std::thread t(long_running_task);
    thread_guard guard(t);
    // Even if an exception is thrown here, guard ensures t is joined
    do_other_work();
}   // guard destructor joins t

The Most Vexing Parse

A classic trap when constructing threads with temporary functors: 

struct Task {
    void operator()() const { /* ... */ }
};

std::thread t(Task());
// This declares a FUNCTION named t that takes a function pointer
// returning Task and returns std::thread. Not what you want!

// Fix with any of these:
std::thread t1{Task{}};              // uniform initialization
std::thread t2((Task()));            // extra parentheses
std::thread t3(Task{});              // braces for the temporary

Thread Identifiers

Every thread has a unique identifier of type std::thread::id. You can retrieve it with t.get_id() for a thread object, or std::this_thread::get_id() from inside the running thread: 

std::thread t([](){
    std::cout << "My ID: " << std::this_thread::get_id() << "\n";
});
std::cout << "Spawned thread ID: " << t.get_id() << "\n";
t.join();

Thread IDs can be compared with == and !=, and can be used as keys in associative containers (they provide std::hash and comparison operators). A default-constructed std::thread::id represents "no thread."

Ownership Transfer

std::thread is move-only. You cannot copy a thread object, but you can transfer ownership via std::move: 

std::thread create_thread() {
    return std::thread([]() {
        std::cout << "Running\n";
    });
}

int main() {
    std::thread t1 = create_thread();  // move from return value
    std::thread t2 = std::move(t1);    // explicit move

    // t1 is now empty (not joinable)
    // t2 owns the thread
    t2.join();
}
Moving into a joinable thread. If you move-assign to a thread object that is currently joinable, std::terminate is called. Always join or detach before reassigning: 
std::thread t1(work_a);
std::thread t2(work_b);

// t1 = std::move(t2);  // BAD: t1 is still joinable, calls terminate!

t1.join();
t1 = std::move(t2);     // OK: t1 was joined first

This move semantics design allows threads to be stored in containers: 

std::vector<std::thread> workers;
for (int i = 0; i < 8; ++i) {
    workers.emplace_back([i]() {
        // process partition i
    });
}
for (auto& w : workers) {
    w.join();
}
