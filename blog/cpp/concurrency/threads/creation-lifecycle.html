Thread Creation & Lifecycle

March 2026

Every concurrent C++ program starts with the same fundamental operation: creating a thread. The std::thread class, defined in <thread>, is the standard mechanism for launching a new thread of execution. Understanding how threads are constructed, what happens at the OS level, the lifecycle states a thread passes through, and the consequences of mismanaging that lifecycle is essential groundwork before tackling any higher-level concurrency pattern.

This page covers thread construction from four callable types, the OS-level machinery behind std::thread, the complete lifecycle state machine, ownership semantics, and common pitfalls that catch even experienced programmers.

Creating Threads

The std::thread constructor accepts any callable entity followed by its arguments. The callable is invoked on a new thread immediately after construction. Four primary callable types appear in practice: function pointers, lambda expressions, functors, and member function pointers.

Internally, the constructor performs the following sequence:

  1. Copies or moves the callable and all arguments into storage owned by the thread object.
  2. Requests the OS to create a new kernel thread.
  3. The new thread invokes the stored callable with the stored arguments.

The copy/move happens on the calling thread. The invocation happens on the new thread. This two-phase model is the source of many subtle behaviors covered later in this series.

Function Pointers

The simplest callable type. Pass a free function (or a static member function) by name. The compiler resolves the function name to a pointer automatically: 

#include <iostream>
#include <thread>

void greet(int id) {
    std::cout << "Hello from thread " << id << "\n";
}

int main() {
    std::thread t1(greet, 1);   // calls greet(1) on a new thread
    std::thread t2(greet, 2);   // calls greet(2) on another new thread

    t1.join();
    t2.join();
}

You can also pass the address explicitly with &greet. Both forms are identical; the compiler decays the function name to a pointer.

Static member functions

class Server {
public:
    static void listen(int port) {
        std::cout << "Listening on port " << port << "\n";
    }
};

int main() {
    std::thread t(Server::listen, 8080);
    t.join();
}
Key point. Arguments are copied into the new thread by default. Even if the function signature takes a reference, the thread constructor still copies the argument unless you explicitly wrap it with std::ref. This is covered in detail on the Passing Arguments page.

Lambda Expressions

Lambdas are the most common callable type for threads in modern C++. They allow inline definition of the work unit and flexible capture of surrounding state. The compiler generates a closure type with an operator(), which the thread constructor stores and later invokes. 

int main() {
    int work_id = 42;

    // Capture by value: safe, each thread gets its own copy
    std::thread t1([work_id]() {
        std::cout << "Processing work " << work_id << "\n";
    });

    // Capture by reference: dangerous if thread outlives the variable
    std::string message = "urgent";
    std::thread t2([&message]() {
        std::cout << "Message: " << message << "\n";
    });

    t1.join();
    t2.join();  // safe only because we join before message goes out of scope
}
Dangling reference trap. If you capture local variables by reference and the creating scope exits before the spawned thread finishes, the behavior is undefined. Always prefer capture by value or ensure join happens before destruction.

Lambda with parameters passed through the constructor

auto worker = [](int start, int end) {
    long long sum = 0;
    for (int i = start; i < end; ++i) sum += i;
    std::cout << "Sum [" << start << ", " << end << "): " << sum << "\n";
};

std::thread t(worker, 0, 1000);
t.join();

Generic lambdas (C++14)

auto print_any = [](auto value) {
    std::cout << value << "\n";
};

std::thread t1(print_any, 42);
std::thread t2(print_any, std::string("hello"));
t1.join();
t2.join();

Mutable lambdas

If the lambda needs to modify captured-by-value variables, mark it mutable. The thread receives its own copy of the closure, so mutations do not affect the original: 

int counter = 0;
std::thread t([counter]() mutable {
    for (int i = 0; i < 100; ++i) ++counter;
    std::cout << "Thread counter: " << counter << "\n";  // 100
});
t.join();
std::cout << "Main counter: " << counter << "\n";  // still 0

Functors (Callable Objects)

A functor is any object with an overloaded operator(). Useful when the thread work unit carries internal state or when you want a reusable, testable unit: 

class Downloader {
public:
    explicit Downloader(const std::string& url) : url_(url) {}

    void operator()() const {
        std::cout << "Downloading: " << url_ << "\n";
        // ... actual download logic ...
    }

private:
    std::string url_;
};

int main() {
    Downloader dl("https://example.com/data.bin");
    std::thread t1(dl);        // copies the functor into the thread
    t1.join();

    // Move a temporary directly (avoids extra copy)
    std::thread t2(Downloader("https://example.com/other.bin"));
    t2.join();
}

Functor with mutable state and std::ref

class Counter {
public:
    void operator()() {
        for (int i = 0; i < 1000; ++i) ++count_;
    }
    int get_count() const { return count_; }
private:
    int count_ = 0;
};

int main() {
    Counter c;
    std::thread t(std::ref(c));  // pass by reference to read count_ after
    t.join();
    std::cout << "Count: " << c.get_count() << "\n";  // 1000
}

Member Functions

To run a non-static member function on a thread, pass a pointer-to-member-function followed by a pointer (or reference wrapper, or smart pointer) to the object, then any additional arguments: 

class Sensor {
public:
    void read_data(int samples) {
        for (int i = 0; i < samples; ++i) { /* read sensor */ }
        std::cout << "Read " << samples << " samples\n";
    }
};

int main() {
    Sensor s;
    std::thread t(&Sensor::read_data, &s, 100);
    t.join();
}

Using shared_ptr for safe lifetime management

auto sensor = std::make_shared<Sensor>();
std::thread t(&Sensor::read_data, sensor, 100);
t.detach();
// Safe: the shared_ptr inside the thread keeps the Sensor alive

Thread Creation Under the Hood

When you write std::thread t(func, args...), the standard library bridges C++ callable semantics and the OS thread API. Here is what happens at each layer:

1. The C++ layer

The constructor creates a decay-copied tuple of the callable and its arguments. std::decay strips references, converts arrays to pointers, and functions to function pointers. This tuple is heap-allocated (typically via new) and a pointer to it is passed to the OS thread creation function.

The new thread runs a trampoline function that unpacks the tuple and invokes the callable using std::invoke.

2. The POSIX layer (Linux, macOS)

On POSIX systems, the implementation calls pthread_create: 

// Simplified pseudocode of what happens internally:
auto* data = new ThreadData{std::move(callable_tuple)};

pthread_t handle;
int err = pthread_create(&handle, nullptr, &trampoline, data);
if (err != 0) throw std::system_error(err, std::system_category());

// The trampoline (runs on the new thread):
static void* trampoline(void* arg) {
    auto* data = static_cast<ThreadData*>(arg);
    std::invoke(std::move(data->callable), std::move(data->args)...);
    delete data;
    return nullptr;
}

3. The kernel layer

pthread_create wraps the clone system call on Linux. The kernel performs:

  • Stack allocation: A new stack is allocated for the thread (default 8 MB on Linux, controlled by ulimit -s). The stack is backed by virtual memory; physical pages are assigned on demand via page faults.
  • Thread control block: The kernel creates a task_struct holding the thread's register state, scheduling priority, signal mask, and a pointer to the shared address space.
  • Scheduler entry: The new thread is placed on the CPU run queue. The scheduler decides when execution actually begins.
  • TLS setup: The runtime copies the .tdata template and zeroes the .tbss region for the new thread.

4. The Windows layer

On Windows, the implementation calls _beginthreadex, which allocates a thread stack (default 1 MB committed), creates a thread kernel object with a unique handle and ID, initializes TLS slots, and marks the thread as schedulable.

Cost of thread creation

OperationApproximate Cost
Stack allocation (virtual)~1 microsecond
Kernel thread creation~10 to 50 microseconds
TLS initialization~1 to 5 microseconds
First context switch to new thread~1 to 10 microseconds
Total~15 to 70 microseconds

A function call costs about 1 to 5 nanoseconds. Thread creation is roughly 10,000x more expensive. This is why thread pools exist: they amortize the creation cost across many tasks.

Join vs Detach

After creating a thread, you must do one of two things before the std::thread object is destroyed: join it or detach it. Failing to do either terminates the program.

join()

Blocks the calling thread until the spawned thread finishes. Internally, this calls pthread_join (POSIX) or WaitForSingleObject (Windows). After return, the thread's resources are fully reclaimed. 

std::thread t(some_work);
t.join();  // blocks here until some_work completes
// t.joinable() == false

detach()

Separates the execution from the std::thread object. The thread runs independently. You lose all ability to join or query it. 

std::thread t(background_logging);
t.detach();
// t.joinable() == false
Detach is dangerous. A detached thread that accesses local variables from the launching scope will encounter undefined behavior if those variables are destroyed first. Detached threads should only access global data, heap-allocated data they own, or data whose lifetime is explicitly managed.

The Joinable Check

Calling join() or detach() on a non-joinable thread throws std::system_error. Always check: 

if (t.joinable()) {
    t.join();
}

A thread is not joinable if:

  • Default-constructed (no callable): std::thread t;
  • Already joined
  • Already detached
  • Moved from (ownership transferred)

t.join();    // Option A: wait for completion

Destruction Without Join

This is the single most critical rule of std::thread: if the destructor runs while the thread is still joinable, std::terminate() is called. There is no exception, no recovery. 

void dangerous() {
    std::thread t([]() {
        std::this_thread::sleep_for(std::chrono::seconds(5));
    });
    // t goes out of scope without join or detach
    // RESULT: std::terminate(). Program crashes.
}

Why not auto-join? An implicit join could silently block for an arbitrary duration, hiding bugs. Why not auto-detach? The thread might access destroyed stack variables. The committee decided a noisy crash is preferable to a silent, hard-to-diagnose bug.

Exception-safety problem

void risky() {
    std::thread t(long_task);
    do_something_that_might_throw();  // if this throws...
    t.join();                          // ...this line is skipped
}   // t's destructor on joinable thread: terminate!

RAII solution

class thread_guard {
public:
    explicit thread_guard(std::thread& t) : t_(t) {}
    ~thread_guard() {
        if (t_.joinable()) t_.join();
    }
    thread_guard(const thread_guard&) = delete;
    thread_guard& operator=(const thread_guard&) = delete;
private:
    std::thread& t_;
};

void safe() {
    std::thread t(long_task);
    thread_guard guard(t);
    do_something_that_might_throw();
}  // guard destructor joins t, even if an exception was thrown

In C++20, std::jthread provides this behavior built-in. See the thread_local, RAII Guards & jthread page.

The Most Vexing Parse

When constructing a thread with a temporary functor, a C++ parsing ambiguity can silently turn your thread construction into a function declaration: 

struct Task {
    void operator()() const { std::cout << "Working\n"; }
};

// TRAP: This declares a FUNCTION named t, not a thread object.
std::thread t(Task());

// Correct alternatives:
std::thread t1{Task{}};              // uniform initialization (preferred)
std::thread t2((Task()));            // extra parentheses
std::thread t3(Task{});              // braces for the temporary

Thread Identifiers

Every running thread has a unique std::thread::id. Retrieve it from the object (t.get_id()) or from within the thread (std::this_thread::get_id()): 

std::thread t([](){
    std::cout << "My ID: " << std::this_thread::get_id() << "\n";
});
std::cout << "Spawned ID: " << t.get_id() << "\n";
t.join();

Thread IDs support ==, !=, <, and work as keys in std::unordered_map (they provide std::hash). A default-constructed std::thread::id represents "no thread."

Thread-aware logging

std::mutex log_mtx;
void log(const std::string& msg) {
    std::ostringstream oss;
    oss << "[Thread " << std::this_thread::get_id() << "] " << msg;
    std::lock_guard<std::mutex> lock(log_mtx);
    std::cout << oss.str() << "\n";
}

Ownership Transfer (Move Semantics)

std::thread is move-only. You cannot copy a thread, but you can transfer ownership via std::move. 

std::thread create_worker() {
    return std::thread([]() {
        std::cout << "Worker running\n";
    });
}

int main() {
    std::thread t1 = create_worker();  // move from return value
    std::thread t2 = std::move(t1);    // explicit move
    // t1 is now empty. t2 owns the thread.
    t2.join();
}

Storing threads in containers

std::vector<std::thread> workers;
for (int i = 0; i < 8; ++i) {
    workers.emplace_back([i]() {
        std::cout << "Worker " << i << " done\n";
    });
}
for (auto& w : workers) w.join();

Returning threads from factory functions

std::thread make_printer(const std::string& text) {
    return std::thread([text]() { std::cout << text << "\n"; });
}
auto t = make_printer("Hello from factory");
t.join();

Performance: Callable Types Compared

The callable type has negligible impact on thread performance because the dominant cost is OS thread creation (tens of microseconds). The differences below are measurable only in microbenchmarks:

Callable TypeInvocation MechanismPer-Call OverheadNotes
Function pointerIndirect call~1 nsCannot inline.
Lambda (no capture)Direct call (inlineable)~0 nsCompiler sees the body. Often fastest.
Lambda (with capture)Closure call~0 to 1 nsSmall captures stay on stack.
Functoroperator() on copy~0 to 1 nsMay inline. Copy cost varies.
std::functionType-erased indirect~5 to 20 nsPossible heap allocation. Avoid in hot paths.
Member function ptrstd::invoke~1 to 2 nsSimilar to function pointer.
Practical guidance. Use lambdas by default. They offer the best combination of readability, flexibility, and performance. Avoid std::function as a thread callable unless you need type erasure.

Benchmark code (compile with -O2)

#include <thread>
#include <chrono>
#include <iostream>
#include <functional>

void plain_func() { volatile int x = 0; (void)x; }

struct Functor { void operator()() const { volatile int x = 0; (void)x; } };

int main() {
    constexpr int N = 10000;
    using Clock = std::chrono::high_resolution_clock;

    auto t0 = Clock::now();
    for (int i = 0; i < N; ++i) { std::thread t(plain_func); t.join(); }
    auto t1 = Clock::now();

    auto t2 = Clock::now();
    for (int i = 0; i < N; ++i) { std::thread t([]{ volatile int x = 0; (void)x; }); t.join(); }
    auto t3 = Clock::now();

    Functor f;
    auto t4 = Clock::now();
    for (int i = 0; i < N; ++i) { std::thread t(f); t.join(); }
    auto t5 = Clock::now();

    auto us = [](auto d){ return std::chrono::duration_cast<std::chrono::microseconds>(d).count(); };
    std::cout << "Function ptr: " << us(t1-t0)/N << " us/thread\n";
    std::cout << "Lambda:       " << us(t3-t2)/N << " us/thread\n";
    std::cout << "Functor:      " << us(t5-t4)/N << " us/thread\n";
}

Common Mistakes

1. Forgetting to join or detach

void bad() {
    std::thread t(work);
}   // terminate! Fix: use RAII or std::jthread

2. Joining twice

std::thread t(work);
t.join();
t.join();   // throws std::system_error

3. Most vexing parse

std::thread t(Functor());   // function declaration, not thread
std::thread t{Functor{}};   // correct

4. Capturing local by reference with detach

void bad() {
    int x = 42;
    std::thread t([&x]{ std::cout << x; });
    t.detach();
}   // x destroyed; thread accesses garbage

5. Moving into a joinable thread

std::thread t1(work_a);
std::thread t2(work_b);
t1 = std::move(t2);   // t1 was joinable: terminate!

6. Unsynchronized cout

// Multiple threads writing to cout interleave output.
// Fix: use a mutex or build the string with ostringstream first.

7. Assuming thread execution order

std::thread t1([]{ std::cout << "1"; });
std::thread t2([]{ std::cout << "2"; });
t1.join(); t2.join();
// Output can be "12", "21", or interleaved.

Summary

Callable TypeSyntaxBest For
Function pointerstd::thread t(func, args...)Simple, stateless tasks
Lambdastd::thread t([captures]{...}, args...)Inline work with local context
Functorstd::thread t(FunctorObj{}, args...)Stateful, reusable tasks
Member functionstd::thread t(&C::fn, &obj, args...)Running methods on objects
Key rules. (1) Every std::thread must be joined or detached before destruction. (2) Use RAII wrappers or std::jthread for exception safety. (3) Prefer lambdas. (4) Use braces to avoid the most vexing parse. (5) Arguments are copied by default; use std::ref or std::move intentionally.
