thread_local, RAII Guards & jthread

March 2026

This page covers three important topics that round out low-level thread usage in C++: per-thread storage with thread_local, safe automatic thread cleanup via RAII guards, and the modern std::jthread introduced in C++20 with its cooperative cancellation mechanism.

thread_local Storage

The thread_local storage specifier declares a variable with thread storage duration: each thread gets its own independent copy of the variable. The variable is created when the thread starts (or on first use) and destroyed when the thread exits. 

#include <iostream>
#include <thread>

thread_local int call_count = 0;

void increment() {
    ++call_count;
    std::cout << "Thread " << std::this_thread::get_id()
              << ": call_count = " << call_count << "\n";
}

int main() {
    std::thread t1([]{ increment(); increment(); increment(); });
    std::thread t2([]{ increment(); increment(); });
    t1.join();
    t2.join();
    // t1 sees call_count go 1, 2, 3
    // t2 sees call_count go 1, 2
    // Each thread has its own independent copy
}

Initialization and Destruction

thread_local variables follow specific initialization rules:

  • Static initialization: Zero or constant initialization happens before any code runs.
  • Dynamic initialization: Constructor calls happen the first time the thread accesses the variable (for function-local thread_local) or when the thread starts (for namespace-scope thread_local).
  • Destruction: Destructors run in reverse construction order when the thread exits.
thread_local std::string session_id;

// Namespace-scope: constructed when each thread starts,
// destroyed when each thread exits.

void handle_request() {
    thread_local std::vector<int> cache;
    // Function-scope: constructed on first call by each thread,
    // persists across calls within the same thread,
    // destroyed when the thread exits.

    cache.push_back(42);
    std::cout << "Cache size: " << cache.size() << "\n";
}

int main() {
    std::thread t1([]{ handle_request(); handle_request(); });
    std::thread t2([]{ handle_request(); handle_request(); handle_request(); });
    t1.join();
    t2.join();
    // t1: Cache size: 1, Cache size: 2
    // t2: Cache size: 1, Cache size: 2, Cache size: 3
}
Destruction ordering. If thread_local variable A is constructed before B on a given thread, B is destroyed before A. This follows the standard C++ rule of reverse destruction order.

Practical Use Cases

1. Per-thread random number generators

#include <random>

int thread_safe_random(int min, int max) {
    thread_local std::mt19937 gen(std::random_device{}());
    std::uniform_int_distribution<int> dist(min, max);
    return dist(gen);
}
// No mutex needed. Each thread has its own generator.

2. Per-thread error context

thread_local std::string last_error;

void set_error(const std::string& msg) { last_error = msg; }
std::string get_error() { return last_error; }
// Similar to errno, but type-safe and per-thread.

3. Per-thread memory allocator cache

thread_local std::vector<void*> free_list;

void* fast_alloc(size_t size) {
    if (!free_list.empty()) {
        void* p = free_list.back();
        free_list.pop_back();
        return p;
    }
    return ::operator new(size);
}

void fast_dealloc(void* p) {
    free_list.push_back(p);  // no lock needed
}

4. Per-thread logging buffer

thread_local std::ostringstream log_buffer;

void log(const std::string& msg) {
    log_buffer << "[" << std::this_thread::get_id() << "] " << msg << "\n";
}

void flush_log() {
    // Flush to shared sink under a lock
    std::lock_guard<std::mutex> lock(global_log_mutex);
    std::cout << log_buffer.str();
    log_buffer.str("");
}
When to use thread_local. Use it to eliminate synchronization overhead when each thread can work with its own copy of mutable state. Avoid it for large objects if you spawn many short-lived threads, since each thread pays the construction cost.

RAII Thread Guards

The core problem: if an exception is thrown between thread creation and the join() call, the thread destructor fires on a joinable thread and std::terminate is called. RAII solves this by tying the join to object lifetime.

The problem in detail

void risky() {
    std::thread t(long_task);
    do_something_that_might_throw();  // if this throws...
    t.join();                          // ...this line never executes
}   // t's destructor runs on a joinable thread: std::terminate!

Basic thread guard (join on destruction)

class thread_guard {
public:
    explicit thread_guard(std::thread& t) : t_(t) {}

    ~thread_guard() {
        if (t_.joinable()) {
            t_.join();
        }
    }

    thread_guard(const thread_guard&) = delete;
    thread_guard& operator=(const thread_guard&) = delete;

private:
    std::thread& t_;
};

void safe() {
    std::thread t(long_task);
    thread_guard guard(t);

    do_something_that_might_throw();
    // If exception thrown: guard destructor joins t. No terminate.
    // If no exception: guard destructor joins t. Same result.
}

A Scoped Thread Class

A more robust approach is an owning wrapper that takes the thread by move: 

class scoped_thread {
public:
    explicit scoped_thread(std::thread t) : t_(std::move(t)) {
        if (!t_.joinable()) {
            throw std::logic_error("No thread to guard");
        }
    }

    ~scoped_thread() {
        t_.join();
    }

    scoped_thread(const scoped_thread&) = delete;
    scoped_thread& operator=(const scoped_thread&) = delete;

    // Allow move
    scoped_thread(scoped_thread&& other) noexcept : t_(std::move(other.t_)) {}
    scoped_thread& operator=(scoped_thread&&) = delete;

private:
    std::thread t_;
};

void safe_v2() {
    scoped_thread worker(std::thread(long_task));
    do_something_that_might_throw();
    // join is guaranteed by scoped_thread destructor
}

Choosing the action: join or detach

You can generalize the guard to choose the cleanup action: 

enum class thread_action { join, detach };

class flexible_thread_guard {
public:
    flexible_thread_guard(std::thread t, thread_action action)
        : t_(std::move(t)), action_(action) {
        if (!t_.joinable()) {
            throw std::logic_error("No thread to guard");
        }
    }

    ~flexible_thread_guard() {
        if (t_.joinable()) {
            if (action_ == thread_action::join) t_.join();
            else t_.detach();
        }
    }

    flexible_thread_guard(const flexible_thread_guard&) = delete;
    flexible_thread_guard& operator=(const flexible_thread_guard&) = delete;

private:
    std::thread t_;
    thread_action action_;
};

// Usage:
flexible_thread_guard g1(std::thread(critical_work), thread_action::join);
flexible_thread_guard g2(std::thread(fire_and_forget), thread_action::detach);

std::jthread (C++20)

C++20 introduced std::jthread (the "j" stands for "joining") which solves both the RAII problem and adds cooperative cancellation. It automatically joins in its destructor, eliminating the need for external guards.

Basic usage

#include <iostream>
#include <thread>

int main() {
    std::jthread t([]() {
        std::cout << "Hello from jthread\n";
    });
    // No need to call t.join()!
    // The destructor requests stop and then joins automatically.
}

Stop Tokens and Cooperative Cancellation

std::jthread provides a built-in cancellation mechanism via std::stop_token. The thread can periodically check if a stop has been requested, and the jthread destructor automatically requests stop before joining.

Using stop_token in the callable

#include <iostream>
#include <thread>
#include <chrono>

void worker(std::stop_token stoken) {
    int iteration = 0;
    while (!stoken.stop_requested()) {
        std::cout << "Working... iteration " << ++iteration << "\n";
        std::this_thread::sleep_for(std::chrono::milliseconds(200));
    }
    std::cout << "Stop requested. Cleaning up.\n";
}

int main() {
    std::jthread t(worker);

    std::this_thread::sleep_for(std::chrono::seconds(1));
    // t goes out of scope here:
    // 1. t.request_stop() is called automatically
    // 2. Then t.join() is called
    // The worker sees stop_requested() == true and exits gracefully
}

Explicitly requesting stop

int main() {
    std::jthread t(worker);

    std::this_thread::sleep_for(std::chrono::seconds(1));
    t.request_stop();  // signal the thread to stop

    // Thread will finish its current iteration, then exit.
    // Destructor still joins, but the thread exits quickly.
}

Getting the stop_source and stop_token

std::jthread t(worker);

// The jthread owns a stop_source
std::stop_source ss = t.get_stop_source();
std::stop_token st = t.get_stop_token();

// Check if stop was already requested
if (!st.stop_requested()) {
    std::cout << "Thread still running\n";
}

// Request stop via the source
ss.request_stop();
Cooperative, not preemptive. The stop token is a request, not a forced kill. The thread must check stop_requested() at appropriate intervals. If the thread ignores the token, the destructor will still block on join until the thread finishes on its own.

stop_token with additional arguments

When the callable accepts a std::stop_token as its first parameter, std::jthread automatically passes its internal stop token. Additional arguments work normally: 

void monitored_work(std::stop_token stoken, int id, const std::string& name) {
    while (!stoken.stop_requested()) {
        std::cout << name << " [" << id << "] working...\n";
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
}

int main() {
    std::jthread t(monitored_work, 42, std::string("sensor"));
    std::this_thread::sleep_for(std::chrono::milliseconds(500));
    // destructor requests stop and joins
}

Stop Callbacks

std::stop_callback lets you register a function that is called when stop is requested. This is useful for waking up blocked threads: 

#include <condition_variable>
#include <mutex>

std::mutex mtx;
std::condition_variable cv;
bool data_ready = false;

void wait_for_data(std::stop_token stoken) {
    std::unique_lock lock(mtx);

    // Register a callback that notifies the CV when stop is requested
    std::stop_callback on_stop(stoken, [&]() {
        cv.notify_all();
    });

    cv.wait(lock, [&]() {
        return data_ready || stoken.stop_requested();
    });

    if (stoken.stop_requested()) {
        std::cout << "Cancelled while waiting\n";
        return;
    }
    std::cout << "Data received, processing...\n";
}

int main() {
    std::jthread t(wait_for_data);
    std::this_thread::sleep_for(std::chrono::seconds(1));
    // Destructor calls request_stop(), which triggers the callback,
    // which notifies the CV, which wakes the thread.
}
Unblocking pattern. Stop callbacks are the standard way to unblock threads waiting on condition variables, I/O operations, or other blocking calls. Without them, the destructor's join would block indefinitely waiting for a thread stuck in a cv.wait().

▶ jthread Destruction Sequence

Step through what happens when a jthread goes out of scope.

hardware_concurrency

std::thread::hardware_concurrency() returns a hint for the number of threads that can truly run in parallel (typically the number of CPU cores or hardware threads). It returns 0 if the value is not computable. 

#include <iostream>
#include <thread>

int main() {
    unsigned int n = std::thread::hardware_concurrency();
    std::cout << "Hardware threads: " << n << "\n";
    // Typical output: "Hardware threads: 8" on a quad-core with hyperthreading
}

Using it to size a thread pool

unsigned int num_workers = std::thread::hardware_concurrency();
if (num_workers == 0) num_workers = 2;  // fallback

std::vector<std::jthread> pool;
pool.reserve(num_workers);

for (unsigned int i = 0; i < num_workers; ++i) {
    pool.emplace_back([i](std::stop_token st) {
        while (!st.stop_requested()) {
            // pull work from a queue...
        }
    });
}
// pool destructor automatically stops and joins all threads

Oversubscription caveat

Creating more threads than hardware_concurrency() leads to oversubscription: the OS must time-slice threads, adding context switch overhead. For CPU-bound work, match the number of threads to hardware concurrency. For I/O-bound work, you may benefit from more threads since they spend time waiting.

Partitioning work

template <typename Iterator, typename T>
T parallel_accumulate(Iterator first, Iterator last, T init) {
    const unsigned long length = std::distance(first, last);
    if (length == 0) return init;

    const unsigned long hw = std::thread::hardware_concurrency();
    const unsigned long num_threads = std::min(hw != 0 ? hw : 2UL, length);
    const unsigned long block_size = length / num_threads;

    std::vector<T> results(num_threads);
    std::vector<std::jthread> threads;

    Iterator block_start = first;
    for (unsigned long i = 0; i < num_threads - 1; ++i) {
        Iterator block_end = block_start;
        std::advance(block_end, block_size);
        threads.emplace_back([block_start, block_end, &results, i]() {
            results[i] = std::accumulate(block_start, block_end, T{});
        });
        block_start = block_end;
    }
    // Last block (handles remainder)
    results[num_threads - 1] = std::accumulate(block_start, last, T{});

    // jthreads join automatically when threads vector is destroyed
    threads.clear();  // join all

    return std::accumulate(results.begin(), results.end(), init);
}

Putting It All Together

Here is a complete example that combines thread_local, std::jthread, stop tokens, and hardware_concurrency into a practical pattern: 

#include <iostream>
#include <thread>
#include <vector>
#include <atomic>
#include <chrono>
#include <random>

// Per-thread statistics
thread_local int items_processed = 0;
thread_local int errors_encountered = 0;

std::atomic<int> global_items{0};

void worker(std::stop_token stoken, int worker_id) {
    thread_local std::mt19937 rng(std::random_device{}());
    std::uniform_int_distribution<int> dist(1, 100);

    while (!stoken.stop_requested()) {
        // Simulate processing
        int value = dist(rng);
        if (value > 95) {
            ++errors_encountered;
        } else {
            ++items_processed;
            global_items.fetch_add(1, std::memory_order_relaxed);
        }
        std::this_thread::sleep_for(std::chrono::microseconds(100));
    }

    std::cout << "Worker " << worker_id
              << ": processed=" << items_processed
              << ", errors=" << errors_encountered << "\n";
}

int main() {
    unsigned int n = std::thread::hardware_concurrency();
    if (n == 0) n = 4;
    std::cout << "Starting " << n << " workers\n";

    {
        std::vector<std::jthread> workers;
        for (unsigned int i = 0; i < n; ++i) {
            workers.emplace_back(worker, i);
        }

        // Let them run for 1 second
        std::this_thread::sleep_for(std::chrono::seconds(1));

        // workers go out of scope here:
        // - request_stop() called on each
        // - join() called on each
    }

    std::cout << "Total items processed: " << global_items.load() << "\n";
}

Summary

FeaturePurposeKey Benefit
thread_local Per-thread variable storage Eliminates synchronization for thread-private data
RAII guards Automatic join/detach on destruction Exception safety for std::thread
std::jthread Auto-joining thread with stop token Safe by default, cooperative cancellation
std::stop_token Cooperative cancellation signal Clean shutdown without thread killing
std::stop_callback React to stop requests Unblock waiting threads on cancellation
hardware_concurrency() Query available parallelism Right-size thread pools, avoid oversubscription
Modern best practice. In C++20 and later, prefer std::jthread over std::thread for all new code. It eliminates the entire class of "forgot to join" bugs and provides cooperative cancellation out of the box.
