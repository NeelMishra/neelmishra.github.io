    Strong Types & Phantom Types

    March 2026 · 20 min read

    The Problem with Primitive Types

    Consider a function that creates a user account:

    void create_user(std::string name, std::string email, std::string password,
                  int age, int department_id, int role_id);

    This compiles perfectly:

    create_user("alice@example.com", "Alice", "secret123", 3, 25, 1);

    Did you spot the bug? The email and name are swapped. The department_id and age are swapped. The compiler is happy because every argument has the right primitive type, even though the semantics are completely wrong. This is primitive obsession: using raw primitives (int, double, string) to represent domain concepts that deserve their own types.

    The cost is real. The Mars Climate Orbiter crashed because one team used pounds of force while another used newtons. Both were represented as bare double values. A strong type system would have caught this at compile time.

    Key insight: The type system is your most powerful bug-prevention tool. Every time you use a bare int or double to represent a domain concept, you lose the compiler's ability to catch semantic errors. Strong types reclaim that power.

    Creating Strong Typedefs

    C++ does not have a built-in strong typedef (unlike languages like Haskell or Rust). The typedef and using keywords create aliases that are interchangeable with the original type:

    using Meters = double;
using Kilograms = double;

Meters distance = 100.0;
Kilograms mass = 75.0;

// This compiles! Meters and Kilograms are both double.
Kilograms oops = distance;  // no error, no warning

    To get a truly distinct type, we need to wrap the value in a struct:

    struct Meters {
    double value;
    explicit Meters(double v) : value(v) {}
};

struct Kilograms {
    double value;
    explicit Kilograms(double v) : value(v) {}
};

Meters distance{100.0};
Kilograms mass{75.0};

// Kilograms oops = distance;  // ERROR: no conversion from Meters to Kilograms
// Meters m = 42.0;            // ERROR: explicit constructor prevents implicit conversion

    Now the compiler enforces that meters and kilograms are distinct concepts. Accidental mixing is a compile error.

    Adding Operations

    A strong type needs arithmetic operations to be useful. We add only the operations that make semantic sense:

    struct Meters {
    double value;
    explicit Meters(double v) : value(v) {}

    Meters operator+(Meters other) const { return Meters{value + other.value}; }
    Meters operator-(Meters other) const { return Meters{value - other.value}; }
    Meters operator*(double scalar) const { return Meters{value * scalar}; }
    Meters operator/(double scalar) const { return Meters{value / scalar}; }
    double operator/(Meters other) const { return value / other.value; }  // ratio

    bool operator==(Meters other) const { return value == other.value; }
    auto operator<=>(Meters other) const { return value <=> other.value; }
};

// Meters + Meters = Meters   (makes sense)
// Meters * double = Meters   (scaling)
// Meters / Meters = double   (ratio)
// Meters + Kilograms = ???   (SHOULD NOT COMPILE)
    Key insight: Notice that Meters * Meters is deliberately not defined, because that would give square meters, a different unit. This is the power of strong types: the operations you omit are as important as the operations you define. The compiler enforces dimensional correctness.

    The NamedType Pattern

    Writing a struct for every strong type is tedious. The NamedType pattern (popularized by Jonathan Boccara) uses a template to generate strong types with minimal boilerplate:

    template <typename T, typename Tag>
class NamedType {
    T value_;
public:
    explicit NamedType(T value) : value_(std::move(value)) {}

    T& get() { return value_; }
    const T& get() const { return value_; }

    // Comparisons
    bool operator==(const NamedType& other) const { return value_ == other.value_; }
    auto operator<=>(const NamedType& other) const { return value_ <=> other.value_; }
};

// Each tag creates a distinct type:
using Width  = NamedType<double, struct WidthTag>;
using Height = NamedType<double, struct HeightTag>;
using Depth  = NamedType<double, struct DepthTag>;

void set_dimensions(Width w, Height h, Depth d);

// Correct usage:
set_dimensions(Width{10.0}, Height{20.0}, Depth{5.0});

// Compiler errors:
// set_dimensions(Height{20.0}, Width{10.0}, Depth{5.0});  // ERROR!
// set_dimensions(10.0, 20.0, 5.0);                         // ERROR: explicit ctors

    Adding Skills via Mixins

    Not every strong type needs arithmetic. The NamedType pattern uses skill mixins to opt into specific operations:

    // Skill: addable
template <typename T>
struct Addable {
    friend T operator+(const T& lhs, const T& rhs) {
        return T{lhs.get() + rhs.get()};
    }
};

// Skill: printable
template <typename T>
struct Printable {
    friend std::ostream& operator<<(std::ostream& os, const T& t) {
        return os << t.get();
    }
};

// Skill: hashable
template <typename T>
struct Hashable {
    struct HashFunction {
        size_t operator()(const T& t) const {
            return std::hash<decltype(t.get())>{}(t.get());
        }
    };
};

// Enhanced NamedType with CRTP skills:
template <typename T, typename Tag, template<typename> class... Skills>
class StrongType : public Skills<StrongType<T, Tag, Skills...>>... {
    T value_;
public:
    explicit StrongType(T value) : value_(std::move(value)) {}
    T& get() { return value_; }
    const T& get() const { return value_; }
    bool operator==(const StrongType& o) const { return value_ == o.value_; }
};

// Usage:
using Meters = StrongType<double, struct MetersTag, Addable, Printable>;
using UserId = StrongType<int, struct UserIdTag, Printable, Hashable>;

Meters a{3.0}, b{4.0};
Meters c = a + b;        // OK: Addable skill
std::cout << c;          // OK: Printable skill
// Meters d = a * b;     // ERROR: no Multipliable skill

    Phantom Types: Compile-Time State Tracking

    A phantom type is a type parameter that appears in the type signature but is never used in the data representation. It exists solely to carry information at the type level, which the compiler uses for checking but which has zero runtime cost.

    Basic Idea

    // State tags: empty structs, never instantiated
struct Unvalidated {};
struct Validated {};

template <typename State>
class UserInput {
    std::string data_;
public:
    explicit UserInput(std::string d) : data_(std::move(d)) {}
    const std::string& data() const { return data_; }

    // Only unvalidated input can be validated
    UserInput<Validated> validate() const
        requires std::same_as<State, Unvalidated>
    {
        // ... perform validation ...
        return UserInput<Validated>{data_};
    }
};

// Functions that require validated input:
void save_to_database(const UserInput<Validated>& input) {
    // Type system guarantees this was validated
    db.insert(input.data());
}

UserInput<Unvalidated> raw{"user input"};
// save_to_database(raw);       // ERROR: Unvalidated != Validated
auto clean = raw.validate();
save_to_database(clean);        // OK: type is UserInput<Validated>
    Key insight: The State parameter (Unvalidated or Validated) adds zero bytes to the object. It exists only in the type system. The compiler enforces that you cannot pass unvalidated input to functions expecting validated input. The check happens at compile time, with no runtime cost.

    State Machine with Phantom Types

    Phantom types can model entire state machines, making illegal state transitions into compile errors:

    // Connection states
struct Disconnected {};
struct Connected {};
struct Authenticated {};

template <typename State>
class Connection {
    int socket_fd_;
public:
    explicit Connection(int fd) : socket_fd_(fd) {}

    // Only disconnected connections can connect
    Connection<Connected> connect(const std::string& host)
        requires std::same_as<State, Disconnected>
    {
        // ... establish connection ...
        return Connection<Connected>{socket_fd_};
    }

    // Only connected (not yet authenticated) connections can authenticate
    Connection<Authenticated> authenticate(const std::string& token)
        requires std::same_as<State, Connected>
    {
        // ... send auth token ...
        return Connection<Authenticated>{socket_fd_};
    }

    // Only authenticated connections can send data
    void send(const std::string& data)
        requires std::same_as<State, Authenticated>
    {
        // ... send data ...
    }

    // Any connection can disconnect
    Connection<Disconnected> disconnect() {
        // ... close socket ...
        return Connection<Disconnected>{-1};
    }
};

Connection<Disconnected> conn{-1};
auto c1 = conn.connect("example.com");     // Disconnected → Connected
auto c2 = c1.authenticate("token123");     // Connected → Authenticated
c2.send("Hello!");                          // OK: authenticated

// conn.send("Hello!");       // ERROR: Disconnected has no send()
// c1.send("Hello!");         // ERROR: Connected has no send()
// c2.connect("other.com");   // ERROR: Authenticated has no connect()
// conn.authenticate("x");    // ERROR: Disconnected has no authenticate()

    Every illegal state transition is a compile error. You literally cannot write code that sends data on an unauthenticated connection. The state machine is enforced by the type system.

    Strong Types vs Bare Primitives

    The Units Library Approach

    Real-world units form a dimensional algebra. Meters times meters gives square meters. Meters per second is a velocity. A general-purpose units library encodes dimensions as template parameters:

    template <int M, int Kg, int S>  // powers of base dimensions
class Quantity {
    double value_;
public:
    explicit Quantity(double v) : value_(v) {}
    double value() const { return value_; }

    // Same dimensions: can add/subtract
    Quantity operator+(Quantity other) const { return Quantity{value_ + other.value_}; }
    Quantity operator-(Quantity other) const { return Quantity{value_ - other.value_}; }

    // Scalar multiplication
    Quantity operator*(double s) const { return Quantity{value_ * s}; }
    Quantity operator/(double s) const { return Quantity{value_ / s}; }
};

// Multiplication of different dimensions: add exponents
template <int M1, int Kg1, int S1, int M2, int Kg2, int S2>
auto operator*(Quantity<M1,Kg1,S1> a, Quantity<M2,Kg2,S2> b) {
    return Quantity<M1+M2, Kg1+Kg2, S1+S2>{a.value() * b.value()};
}

// Division: subtract exponents
template <int M1, int Kg1, int S1, int M2, int Kg2, int S2>
auto operator/(Quantity<M1,Kg1,S1> a, Quantity<M2,Kg2,S2> b) {
    return Quantity<M1-M2, Kg1-Kg2, S1-S2>{a.value() / b.value()};
}

// Type aliases for common quantities
using Meters    = Quantity<1, 0, 0>;   // m
using Kilograms = Quantity<0, 1, 0>;   // kg
using Seconds   = Quantity<0, 0, 1>;   // s
using MetersPerSecond = Quantity<1, 0, -1>;  // m/s
using Newtons   = Quantity<1, 1, -2>;  // kg*m/s^2

Meters distance{100.0};
Seconds time{9.58};
auto speed = distance / time;  // Quantity<1,0,-1> = MetersPerSecond

Kilograms mass{75.0};
auto force = mass * speed / time;  // Newtons? Let's check dimensions:
// (0,1,0) * (1,0,-1) / (0,0,1) = (1,1,-2) = Newtons! Correct!

// auto bad = distance + mass;  // ERROR: Quantity<1,0,0> + Quantity<0,1,0> = type mismatch!

    std::chrono: Strong Types in the Standard Library

    std::chrono is the standard library's best example of strong typing. It uses distinct types for different time durations, preventing accidental mixing:

    using namespace std::chrono;

seconds s{60};
milliseconds ms{500};
hours h{2};

// Implicit conversion to finer granularity (no precision loss):
milliseconds total_ms = s;  // 60000ms  (OK: implicit widening)

// Explicit conversion required for coarser granularity (precision loss):
// seconds coarse = ms;  // ERROR: might lose precision
seconds coarse = duration_cast<seconds>(ms);  // 0s (truncated)

// Arithmetic respects types:
auto sum = s + ms;           // milliseconds (common type)
auto diff = h - minutes{30}; // minutes (common type)

// Type safety in APIs:
void set_timeout(milliseconds timeout);
void sleep_for(nanoseconds duration);

set_timeout(500ms);          // OK: literal suffix creates correct type
// set_timeout(500);          // ERROR: int is not milliseconds
set_timeout(seconds{2});     // OK: implicit widening to milliseconds

    Chrono Literals

    using namespace std::chrono_literals;

auto a = 100ms;    // milliseconds
auto b = 2s;       // seconds
auto c = 500us;    // microseconds
auto d = 1h;       // hours
auto e = 30min;    // minutes
auto f = 10ns;     // nanoseconds

// These are all different types! Mixing requires explicit conversion.
// auto bad = 100ms + 5;  // ERROR: cannot add int to milliseconds
    Key insight: std::chrono demonstrates that strong types do not impair usability. With literal suffixes and implicit safe conversions, the code is actually more readable than using bare integers. sleep_for(500ms) is clearer than sleep_for(500) or sleep_for_ms(500).

    Real-World Examples

    Database IDs

    using UserId   = NamedType<int64_t, struct UserIdTag>;
using OrderId  = NamedType<int64_t, struct OrderIdTag>;
using ProductId = NamedType<int64_t, struct ProductIdTag>;

void cancel_order(OrderId order, UserId user);

// cancel_order(UserId{42}, OrderId{100});  // ERROR: arguments swapped
cancel_order(OrderId{100}, UserId{42});     // OK

    Currency

    template <typename CurrencyTag>
class Money {
    int64_t cents_;  // store as smallest unit to avoid floating point
public:
    explicit Money(int64_t cents) : cents_(cents) {}
    int64_t cents() const { return cents_; }

    Money operator+(Money other) const { return Money{cents_ + other.cents_}; }
    Money operator-(Money other) const { return Money{cents_ - other.cents_}; }
    Money operator*(int64_t factor) const { return Money{cents_ * factor}; }
};

using USD = Money<struct UsdTag>;
using EUR = Money<struct EurTag>;

USD price{999};       // $9.99
EUR shipping{350};    // 3.50 EUR
// auto total = price + shipping;  // ERROR: cannot add USD and EUR!

    File Paths

    struct AbsoluteTag {};
struct RelativeTag {};

template <typename Kind>
class FilePath {
    std::string path_;
public:
    explicit FilePath(std::string p) : path_(std::move(p)) {}
    const std::string& str() const { return path_; }
};

using AbsolutePath = FilePath<AbsoluteTag>;
using RelativePath = FilePath<RelativeTag>;

void write_file(AbsolutePath path, std::string_view content);

// write_file(RelativePath{"data.txt"}, "hello");  // ERROR!
write_file(AbsolutePath{"/home/user/data.txt"}, "hello");  // OK

    When to Use Strong Types

    • Function parameters of the same primitive type: if a function takes multiple int or string parameters, wrap them.
    • Physical quantities with units: meters, seconds, kilograms, pixels, bytes.
    • IDs: user IDs, order IDs, session tokens. Never pass the wrong ID type.
    • State machines: phantom types enforce valid transitions at compile time.
    • Security boundaries: validated vs unvalidated input, sanitized vs raw HTML.

    When Not to Use Strong Types

    • Internal calculations where the types are obvious from context.
    • Performance-critical inner loops where the wrapper could prevent optimizations (though in practice, compilers optimize strong types away completely).
    • Throwaway scripts where the code is small enough to review entirely.

    Zero-Cost Proof

    Compilers optimize strong type wrappers away completely. A Meters{42.0} generates the exact same assembly as a bare double{42.0}:

    // With strong type:
Meters compute_distance(Meters a, Meters b) {
    return a + b;
}

// Compiles to (x86-64, -O2):
// vaddsd xmm0, xmm0, xmm1
// ret

// Identical to:
double compute_distance_raw(double a, double b) {
    return a + b;
}
// vaddsd xmm0, xmm0, xmm1
// ret

// Zero overhead. The type safety is purely a compile-time concept.
    Key insight: Strong types are a zero-cost abstraction in the strictest sense. They add safety without adding runtime overhead. The information captured in the type system is used by the compiler for checking, then discarded in the generated code. You get all the safety of a type-safe language with all the performance of bare primitives.

    Summary

    • Primitive obsession (using bare int, double, string for domain concepts) is a major source of subtle bugs.
    • Strong typedefs wrap primitive types in distinct struct types, making accidental mixing a compile error.
    • The NamedType pattern with CRTP mixins provides a reusable, composable framework for strong types.
    • Phantom types carry state information in the type system, enforcing valid state transitions at compile time with zero runtime cost.
    • Dimensional analysis via template parameters enables units-aware arithmetic where the compiler checks physical correctness.
    • std::chrono is the standard library's exemplar of strong types done right: safe, readable, and zero-overhead.
    • Strong types are a zero-cost abstraction: the compiler optimizes the wrapper away completely.

    This concludes the C++ Type System deep dive series. From type deduction to constexpr computation, from safe casting to type erasure, and finally to strong types that prevent bugs before they exist, the C++ type system is a powerful ally when you understand its rules and leverage its guarantees.
