    constexpr Everything: Compile-Time C++

    March 2026 · 24 min read

    The Compile-Time Revolution

    One of C++'s most powerful features is its ability to move computation from runtime to compile time. When the compiler can evaluate an expression before your program runs, you get zero-cost abstractions in the truest sense: the computation happens once during compilation, and the result is baked directly into the binary.

    The constexpr family of keywords is the primary mechanism for this. Over the past decade, C++ has steadily expanded what can be done at compile time, from simple constant expressions in C++11 to full-blown algorithms with dynamic memory in C++20. This article traces that evolution and shows you how to harness compile-time computation in your own code.

    constexpr Variables

    A constexpr variable must be initialized with a constant expression, and its value is guaranteed to be known at compile time:

    constexpr int max_size = 1024;          // OK: literal
constexpr double pi = 3.14159265358979; // OK: literal
constexpr int doubled = max_size * 2;   // OK: computed from constexpr

int runtime_val = 42;
constexpr int bad = runtime_val;  // ERROR: runtime_val is not constexpr

    constexpr vs const

    These two are often confused, but they express fundamentally different things:

    • const means "I promise not to modify this." The value might still be determined at runtime.
    • constexpr means "This value is known at compile time." It implies const for variables.
    const int a = 42;           // Compile-time constant (compiler may optimize)
const int b = get_value();  // Runtime constant (unknown until called)

constexpr int c = 42;           // Guaranteed compile-time constant
constexpr int d = get_value();  // ERROR unless get_value() is constexpr
    Key insight: constexpr on a variable is a requirement: you are demanding that the compiler evaluate this now. const is a promise: you are saying you will not modify this, but the compiler might not evaluate it at compile time.

    Where constexpr Variables Shine

    // Array sizes (must be compile-time constants)
constexpr int buffer_size = 256;
std::array<char, buffer_size> buffer;

// Template arguments
constexpr int dimensions = 3;
Matrix<double, dimensions, dimensions> transform;

// switch cases
constexpr int CMD_START = 1;
constexpr int CMD_STOP = 2;
switch(cmd) {
    case CMD_START: /* ... */ break;
    case CMD_STOP:  /* ... */ break;
}

// static_assert (compile-time checks)
constexpr int alignment = 64;
static_assert(alignment > 0 && (alignment & (alignment - 1)) == 0,
              "alignment must be a power of 2");

    constexpr Functions: The Evolution

    A constexpr function can be evaluated at compile time if all its arguments are constant expressions. If called with runtime values, it behaves like a normal function. This duality is what makes constexpr functions so versatile.

    C++11: The Beginning

    C++11 constexpr functions were extremely limited. The body had to be a single return statement (with some allowances for recursion):

    // C++11: single return statement only
constexpr int factorial_11(int n) {
    return n <= 1 ? 1 : n * factorial_11(n - 1);
}

// This forced a recursive style even when a loop would be clearer.
constexpr int square(int x) { return x * x; }  // trivial cases were fine

static_assert(factorial_11(5) == 120, "");  // evaluated at compile time
int n = get_input();
int r = factorial_11(n);  // evaluated at runtime

    C++14: Relaxed Constraints

    C++14 lifted most restrictions. You could use local variables, loops, multiple statements, and if/else. This was a massive usability improvement:

    // C++14: loops, local variables, multiple statements
constexpr int factorial_14(int n) {
    int result = 1;
    for (int i = 2; i <= n; ++i) {
        result *= i;
    }
    return result;
}

constexpr int fibonacci(int n) {
    if (n <= 1) return n;
    int a = 0, b = 1;
    for (int i = 2; i <= n; ++i) {
        int temp = a + b;
        a = b;
        b = temp;
    }
    return b;
}

static_assert(fibonacci(10) == 55, "");  // computed at compile time!

    C++17: if constexpr

    C++17 added if constexpr, which evaluates the condition at compile time and discards the false branch entirely. This is not just an optimization; the discarded branch does not even need to be valid for the current template instantiation:

    template <typename T>
auto get_value(T t) {
    if constexpr (std::is_pointer_v<T>) {
        return *t;  // only compiled when T is a pointer
    } else {
        return t;   // only compiled when T is not a pointer
    }
}

int x = 42;
auto a = get_value(x);    // returns 42 (int)
auto b = get_value(&x);   // returns 42 (dereferenced)

    Without if constexpr, this would require SFINAE or tag dispatch, both far more verbose. The discarded branch is truly gone; attempting to dereference a non-pointer type does not cause a compile error because that branch is never instantiated.

    // Compile-time recursion with if constexpr (replacing template specialization)
template <int N>
constexpr int compile_factorial() {
    if constexpr (N <= 1) {
        return 1;
    } else {
        return N * compile_factorial<N - 1>();
    }
}

static_assert(compile_factorial<6>() == 720, "");

    C++20: constexpr Everywhere

    C++20 expanded the constexpr universe dramatically:

    • constexpr virtual functions: virtual dispatch resolved at compile time when possible.
    • constexpr dynamic memory: new and delete in constexpr contexts (memory must be freed before the constant expression evaluation ends).
    • constexpr containers: std::vector and std::string can be used in constexpr functions.
    • constexpr algorithms: most <algorithm> functions are now constexpr.
    • try/catch blocks (though throw is not allowed to actually execute at compile time).
    // C++20: constexpr with dynamic memory
constexpr int sum_of_squares(int n) {
    std::vector<int> v;
    for (int i = 1; i <= n; ++i) {
        v.push_back(i * i);
    }
    int sum = 0;
    for (int x : v) sum += x;
    return sum;
    // v is destroyed here; memory freed before evaluation ends
}

static_assert(sum_of_squares(5) == 55, "");  // 1+4+9+16+25

// C++20: constexpr algorithms
constexpr bool is_sorted_example() {
    std::array arr = {1, 2, 3, 4, 5};
    return std::is_sorted(arr.begin(), arr.end());
}

static_assert(is_sorted_example(), "");

    C++23: Even More constexpr

    C++23 continues the trend, making constexpr the default for an ever-wider set of standard library functions. Notable additions include constexpr std::optional, std::variant, std::unique_ptr (with restrictions), and more mathematical functions.

    // C++23: constexpr optional
constexpr std::optional<int> safe_divide(int a, int b) {
    if (b == 0) return std::nullopt;
    return a / b;
}

static_assert(safe_divide(10, 2).value() == 5, "");
static_assert(!safe_divide(10, 0).has_value(), "");

    consteval: Immediate Functions (C++20)

    consteval declares an immediate function: it must produce a compile-time constant. Unlike constexpr (which can fall back to runtime), a consteval function called with non-constant arguments is a hard compiler error.

    consteval int square(int x) { return x * x; }

constexpr int a = square(5);   // OK: compile-time evaluation
int b = square(5);             // OK: still compile-time, result baked in

int n = get_runtime_value();
int c = square(n);             // ERROR: n is not a constant expression

    Use consteval when a function only makes sense at compile time. Common use cases:

    • Compile-time validation: ensuring configuration values are valid.
    • Generating lookup tables: building arrays at compile time.
    • Security-sensitive constants: ensuring values cannot be tampered with at runtime.
    consteval int checked_port(int p) {
    if (p < 1 || p > 65535)
        throw "port out of range";  // compile error if triggered
    return p;
}

constexpr int http_port = checked_port(80);    // OK
constexpr int bad_port  = checked_port(99999); // ERROR at compile time!
    Key insight: Think of the constexpr family as a spectrum of guarantees. constexpr on a function means "can be compile-time." consteval means "must be compile-time." constexpr on a variable means "is compile-time."

    constinit: Safe Static Initialization (C++20)

    The static initialization order fiasco is one of C++'s oldest pitfalls: if a global variable in one translation unit depends on a global in another, the initialization order is undefined. constinit addresses this by guaranteeing that a variable is initialized at compile time (constant initialization), while not requiring the variable to be const:

    constinit int global_count = 0;        // initialized at compile time
constinit thread_local int tl_val = 0; // also works with thread_local

// global_count can be modified at runtime:
void increment() { ++global_count; }   // OK: constinit does not imply const

// But initialization must be constant:
int runtime_val = compute();
constinit int bad = runtime_val;  // ERROR: not a constant expression

    constexpr vs constinit vs const

    Keyword Compile-time init? Implies const? Use case
    const Not guaranteed Yes Immutable after init
    constexpr Required Yes (for variables) Compile-time constants
    constinit Required No Safe globals, mutable after init

    Compile-Time Computation Examples

    Compile-Time String Hashing

    constexpr uint32_t fnv1a_32(const char* s) {
    uint32_t hash = 2166136261u;
    while (*s) {
        hash ^= static_cast<uint32_t>(*s++);
        hash *= 16777619u;
    }
    return hash;
}

// Use in switch statements:
void handle_command(const std::string& cmd) {
    switch (fnv1a_32(cmd.c_str())) {
        case fnv1a_32("start"):  do_start(); break;
        case fnv1a_32("stop"):   do_stop();  break;
        case fnv1a_32("status"): do_status(); break;
    }
}
// case labels are computed at compile time!

    Compile-Time Lookup Tables

    constexpr auto make_sin_table() {
    std::array<double, 360> table{};
    for (int i = 0; i < 360; ++i) {
        double rad = i * 3.14159265358979 / 180.0;
        // Approximate sin using Taylor series at compile time
        double x = rad;
        double term = rad;
        for (int n = 1; n < 10; ++n) {
            term *= -rad * rad / ((2*n) * (2*n + 1));
            x += term;
        }
        table[i] = x;
    }
    return table;
}

constexpr auto sin_table = make_sin_table();
// 360 pre-computed sine values baked into the binary

    Compile-Time Type Checking

    template <typename T>
constexpr bool is_power_of_two(T n) {
    static_assert(std::is_integral_v<T>, "only integral types");
    return n > 0 && (n & (n - 1)) == 0;
}

static_assert(is_power_of_two(64), "");
static_assert(!is_power_of_two(65), "");

template <int Alignment>
    requires (is_power_of_two(Alignment))
class AlignedBuffer {
    alignas(Alignment) char data[1024];
};

    if constexpr In Depth

    if constexpr is more than syntactic sugar. It is a compile-time branch elimination mechanism that fundamentally changes how template code is written. The discarded branch is not instantiated, which means it can contain code that would not compile for the given type.

    template <typename T>
std::string to_string(T value) {
    if constexpr (std::is_same_v<T, std::string>) {
        return value;  // T is already a string
    } else if constexpr (std::is_arithmetic_v<T>) {
        return std::to_string(value);  // numeric to string
    } else if constexpr (std::is_pointer_v<T>) {
        if (value == nullptr) return "nullptr";
        return to_string(*value);  // recursive: dereference and convert
    } else {
        static_assert(always_false<T>, "unsupported type");
    }
}

    Before if constexpr, this required multiple overloads with SFINAE or tag dispatch:

    // The old way: tag dispatch
template <typename T>
std::string to_string_impl(T value, std::true_type /* is_arithmetic */) {
    return std::to_string(value);
}

template <typename T>
std::string to_string_impl(T value, std::false_type /* not arithmetic */) {
    return value;  // assume string-like
}

template <typename T>
std::string to_string(T value) {
    return to_string_impl(value, std::is_arithmetic<T>{});
}

    Common Patterns with if constexpr

    // Pattern 1: Conditional member access
template <typename T>
auto size_of(const T& container) {
    if constexpr (requires { container.size(); }) {
        return container.size();
    } else if constexpr (requires { std::size(container); }) {
        return std::size(container);
    } else {
        return 0;
    }
}

// Pattern 2: Compile-time loop unrolling
template <typename Tuple, typename Func, std::size_t... Is>
void for_each_impl(Tuple&& t, Func&& f, std::index_sequence<Is...>) {
    (f(std::get<Is>(std::forward<Tuple>(t))), ...);
}

// Pattern 3: Platform-specific code
template <typename T>
void fast_memset(T* dst, T val, size_t count) {
    if constexpr (sizeof(T) == 1) {
        std::memset(dst, static_cast<int>(val), count);
    } else {
        std::fill_n(dst, count, val);
    }
}

    constexpr Containers (C++20)

    C++20 made std::vector and std::string usable in constexpr contexts. The key rule: all dynamic memory allocated during constant evaluation must be deallocated before the evaluation ends. This is called transient allocation.

    constexpr int count_primes(int limit) {
    std::vector<bool> sieve(limit + 1, true);
    sieve[0] = sieve[1] = false;
    for (int i = 2; i * i <= limit; ++i) {
        if (sieve[i]) {
            for (int j = i * i; j <= limit; j += i)
                sieve[j] = false;
        }
    }
    int count = 0;
    for (bool b : sieve) if (b) ++count;
    return count;
    // vector destroyed here: memory freed, evaluation complete
}

static_assert(count_primes(100) == 25, "");  // Sieve of Eratosthenes at compile time!
    Key insight: You cannot return a std::vector from a constexpr evaluation into a constexpr variable, because the dynamic memory would need to persist past the evaluation. You can extract scalar results. C++26 may relax this with non-transient allocation.

    constexpr String Processing

    constexpr int count_words(std::string_view sv) {
    int count = 0;
    bool in_word = false;
    for (char c : sv) {
        if (c == ' ' || c == '\t' || c == '\n') {
            in_word = false;
        } else if (!in_word) {
            in_word = true;
            ++count;
        }
    }
    return count;
}

static_assert(count_words("hello world foo bar") == 4, "");

constexpr bool is_palindrome(std::string_view sv) {
    for (std::size_t i = 0; i < sv.size() / 2; ++i) {
        if (sv[i] != sv[sv.size() - 1 - i]) return false;
    }
    return true;
}

static_assert(is_palindrome("racecar"), "");
static_assert(!is_palindrome("hello"), "");

    Best Practices

    When to Use constexpr

    • Constants that are truly constant: prefer constexpr over const or #define.
    • Functions that can be compile-time: adding constexpr costs nothing and enables optimization.
    • Template metaprogramming: if constexpr replaces most SFINAE patterns.
    • Validation: use static_assert with constexpr functions to catch errors early.

    When to Use consteval

    • When a function must not execute at runtime (configuration validation, code generation).
    • When runtime evaluation would be a security risk or performance bug.
    • When you want a clear compile error if someone passes a non-constant argument.

    When to Use constinit

    • Mutable globals that need safe initialization ordering.
    • thread_local variables that should avoid lazy initialization overhead.

    Debugging constexpr

    Debugging compile-time code is inherently harder because you cannot use a debugger. Strategies:

    // Strategy 1: Use static_assert for intermediate values
constexpr int step1 = compute_step1();
static_assert(step1 == expected_value, "step1 failed");

// Strategy 2: Force runtime evaluation for debugging
// Comment out constexpr temporarily:
/*constexpr*/ int result = my_function(args);
// Now you can set breakpoints

// Strategy 3: Compiler diagnostics
// GCC/Clang show the evaluation trace when static_assert fails

    Summary

    • constexpr variables are guaranteed compile-time constants.
    • constexpr functions can be evaluated at compile time but fall back to runtime gracefully.
    • consteval functions must be evaluated at compile time; runtime calls are errors.
    • constinit ensures compile-time initialization without implying const.
    • if constexpr enables compile-time branch elimination, replacing SFINAE in most cases.
    • C++20 brought constexpr containers and algorithms, enabling sophisticated compile-time computation.
    • The trend is clear: each C++ standard expands what can be done at compile time. Embrace it.
