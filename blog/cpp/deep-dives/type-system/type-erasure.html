    Type Erasure in C++

    March 2026 · 22 min read

    What Is Type Erasure?

    Type erasure is a technique that hides the concrete type of an object behind a uniform, non-templated interface. The caller works with a single type (like std::function<int(int)>) without knowing whether it wraps a lambda, a function pointer, a functor object, or a bound member function. The concrete type is "erased" at the point of storage and recovered only through a virtual dispatch mechanism.

    This solves a fundamental tension in C++: templates give you zero-overhead polymorphism but infect every interface with template parameters. Virtual functions give you runtime polymorphism but require inheritance hierarchies. Type erasure gives you value-semantic polymorphism: different types, same interface, stored in a single non-template type, without requiring the stored types to inherit from anything.

    Where You Already Use It

    • std::function — erases any callable matching a signature.
    • std::any — erases literally any copyable type.
    • std::shared_ptr's deleter — erases the deleter type.
    • std::move_only_function (C++23) — erases move-only callables.
    • std::format_args — erases format argument types.

    How std::function Works Internally

    Let us build std::function from scratch to understand the type erasure mechanism. The key ingredients are:

    1. An abstract base class that defines the callable interface.
    2. A templated derived class that wraps the actual callable.
    3. A non-template wrapper that owns a pointer to the base class.

    Step 1: The Abstract Interface

    // The concept: something callable with signature R(Args...)
template <typename R, typename... Args>
class CallableBase {
public:
    virtual ~CallableBase() = default;
    virtual R invoke(Args... args) = 0;
    virtual std::unique_ptr<CallableBase> clone() const = 0;
};

    Step 2: The Concrete Wrapper

    // Wraps any callable F that matches the signature
template <typename F, typename R, typename... Args>
class CallableImpl : public CallableBase<R, Args...> {
    F func_;
public:
    explicit CallableImpl(F f) : func_(std::move(f)) {}

    R invoke(Args... args) override {
        return func_(std::forward<Args>(args)...);
    }

    std::unique_ptr<CallableBase<R, Args...>> clone() const override {
        return std::make_unique<CallableImpl>(func_);
    }
};

    Step 3: The Type-Erased Wrapper

    template <typename> class Function;  // primary template

template <typename R, typename... Args>
class Function<R(Args...)> {
    std::unique_ptr<CallableBase<R, Args...>> callable_;
public:
    Function() = default;

    // Constructor: accepts ANY callable matching the signature
    template <typename F>
    Function(F f)
        : callable_(std::make_unique<CallableImpl<F, R, Args...>>(std::move(f)))
    {}

    // Copy constructor
    Function(const Function& other)
        : callable_(other.callable_ ? other.callable_->clone() : nullptr)
    {}

    // Call operator: delegates through the virtual interface
    R operator()(Args... args) {
        if (!callable_) throw std::bad_function_call();
        return callable_->invoke(std::forward<Args>(args)...);
    }

    explicit operator bool() const { return callable_ != nullptr; }
};

    Now the magic happens:

    // Lambda, function pointer, functor: all stored in the same type!
Function<int(int)> f1 = [](int x) { return x * 2; };         // lambda
Function<int(int)> f2 = &square;                              // function pointer
Function<int(int)> f3 = Multiplier{3};                        // functor

// The caller sees only Function<int(int)>. The concrete type is erased.
std::vector<Function<int(int)>> transforms = {f1, f2, f3};
for (auto& fn : transforms) {
    std::cout << fn(10) << "\n";  // 20, 100, 30
}
    Key insight: The templated constructor Function(F f) is where type erasure happens. The concrete type F is captured in the CallableImpl<F,...> object at construction time, then accessed only through the CallableBase virtual interface. The type F never appears in Function's own template parameters.

    How std::function Stores Different Callables

    std::any: Type Erasure for Any Type

    While std::function erases callable types behind a specific signature, std::any erases any copyable type behind a completely opaque wrapper. You can store anything, but you need to know the type to retrieve it:

    std::any a = 42;                        // stores int
std::any b = std::string("hello");     // stores string
std::any c = std::vector<int>{1,2,3};  // stores vector

// Retrieval: must specify the exact type
int val = std::any_cast<int>(a);                  // OK: 42
std::string s = std::any_cast<std::string>(b);    // OK: "hello"

// Wrong type throws:
try {
    double d = std::any_cast<double>(a);  // throws std::bad_any_cast
} catch (const std::bad_any_cast& e) {
    // a holds int, not double
}

// Type checking:
if (a.type() == typeid(int)) {
    int* p = std::any_cast<int>(&a);  // pointer version: returns nullptr on failure
}

    std::any Internals

    Internally, std::any uses the same pattern as our Function above, but with operations like copy, move, destroy, and type_info instead of invoke:

    // Simplified std::any internals (conceptual)
class Any {
    struct Base {
        virtual ~Base() = default;
        virtual std::unique_ptr<Base> clone() const = 0;
        virtual const std::type_info& type() const = 0;
    };

    template <typename T>
    struct Holder : Base {
        T value;
        explicit Holder(T v) : value(std::move(v)) {}
        std::unique_ptr<Base> clone() const override {
            return std::make_unique<Holder>(value);
        }
        const std::type_info& type() const override {
            return typeid(T);
        }
    };

    std::unique_ptr<Base> held_;

public:
    template <typename T>
    Any(T value) : held_(std::make_unique<Holder<T>>(std::move(value))) {}
    // ... copy, move, any_cast as friend ...
};

    Building a Custom Type-Erased Wrapper

    Let us build a type-erased Drawable that can hold any object with a draw() method, without requiring inheritance:

    class Drawable {
    struct Concept {
        virtual ~Concept() = default;
        virtual void draw(std::ostream& out) const = 0;
        virtual std::unique_ptr<Concept> clone() const = 0;
    };

    template <typename T>
    struct Model : Concept {
        T object_;
        explicit Model(T obj) : object_(std::move(obj)) {}

        void draw(std::ostream& out) const override {
            object_.draw(out);  // duck typing: T must have draw()
        }
        std::unique_ptr<Concept> clone() const override {
            return std::make_unique<Model>(object_);
        }
    };

    std::unique_ptr<Concept> self_;

public:
    template <typename T>
    Drawable(T obj) : self_(std::make_unique<Model<T>>(std::move(obj))) {}

    Drawable(const Drawable& other) : self_(other.self_->clone()) {}
    Drawable(Drawable&&) = default;
    Drawable& operator=(Drawable other) {
        self_ = std::move(other.self_);
        return *this;
    }

    void draw(std::ostream& out) const { self_->draw(out); }
};

    Now any type with a draw() method works, no base class needed:

    struct Circle {
    double radius;
    void draw(std::ostream& out) const {
        out << "Circle(r=" << radius << ")";
    }
};

struct Square {
    double side;
    void draw(std::ostream& out) const {
        out << "Square(s=" << side << ")";
    }
};

// No inheritance relationship between Circle and Square!
std::vector<Drawable> shapes;
shapes.push_back(Circle{5.0});
shapes.push_back(Square{3.0});

for (const auto& shape : shapes) {
    shape.draw(std::cout);  // polymorphic dispatch, value semantics
}
    Key insight: This pattern is called the External Polymorphism or Sean Parent pattern (after his famous "Inheritance Is the Base Class of Evil" talk). It separates the polymorphic interface from the types that satisfy it, enabling polymorphism without coupling. The types stored do not need to know about the wrapper.

    Small Buffer Optimization (SBO)

    The naive implementation above allocates heap memory for every stored object via std::make_unique. For small objects (like a function pointer or a small lambda), this is wasteful. The Small Buffer Optimization (SBO) stores small objects inline within the wrapper itself, avoiding heap allocation entirely.

    Most standard library implementations of std::function use SBO. The typical inline buffer size varies by implementation:

    • libstdc++ (GCC): 16 bytes (2 pointers on 64-bit)
    • libc++ (Clang): 24 bytes (3 pointers on 64-bit)
    • MSVC STL: 48 bytes (can fit quite large captures)

    SBO Implementation

    template <typename R, typename... Args>
class SmallFunction {
    static constexpr size_t BufferSize = 32;  // inline storage

    struct Base {
        virtual ~Base() = default;
        virtual R invoke(Args...) = 0;
        virtual void clone_to(void* buffer) const = 0;
        virtual void move_to(void* buffer) = 0;
    };

    template <typename F>
    struct Impl : Base {
        F func;
        explicit Impl(F f) : func(std::move(f)) {}
        R invoke(Args... args) override { return func(std::forward<Args>(args)...); }
        void clone_to(void* buf) const override { new(buf) Impl(func); }
        void move_to(void* buf) override { new(buf) Impl(std::move(func)); }
    };

    // Storage: either inline buffer or heap pointer
    alignas(std::max_align_t) char buffer_[BufferSize];
    bool uses_heap_ = false;

    Base* get_base() {
        return uses_heap_
            ? *reinterpret_cast<Base**>(buffer_)
            : reinterpret_cast<Base*>(buffer_);
    }

public:
    template <typename F>
    SmallFunction(F f) {
        if constexpr (sizeof(Impl<F>) <= BufferSize) {
            // Small object: store inline (no heap allocation)
            new(buffer_) Impl<F>(std::move(f));
            uses_heap_ = false;
        } else {
            // Large object: heap allocate
            auto* ptr = new Impl<F>(std::move(f));
            std::memcpy(buffer_, &ptr, sizeof(ptr));
            uses_heap_ = true;
        }
    }

    R operator()(Args... args) {
        return get_base()->invoke(std::forward<Args>(args)...);
    }

    ~SmallFunction() {
        if (uses_heap_) delete get_base();
        else get_base()->~Base();
    }
};

    Small Buffer Optimization (SBO)

    Performance Implications

    Type erasure is not free. Understanding the costs helps you decide when to use it versus alternatives.

    Virtual Call Overhead

    Every invocation through std::function involves an indirect function call (through a vtable pointer or function pointer). This prevents inlining and disrupts branch prediction. For tight loops calling small functions millions of times, this overhead is measurable.

    // Direct call: compiler can inline square()
int sum = 0;
for (int i = 0; i < N; ++i) sum += square(i);

// Through std::function: cannot inline, indirect call each iteration
std::function<int(int)> fn = square;
int sum2 = 0;
for (int i = 0; i < N; ++i) sum2 += fn(i);

// Typical overhead: 2x to 5x slower for trivial callables in tight loops

    Heap Allocation Overhead

    If the callable exceeds the SBO buffer, std::function heap-allocates. This means a new on construction and delete on destruction. For frequently created/destroyed function objects, this adds up.

    // Likely SBO (small capture, fits in buffer):
auto small = [x = 0](int n) { return n + x; };
std::function<int(int)> f1 = small;  // no heap allocation

// Likely heap-allocated (large capture):
std::array<int, 100> big_capture{};
auto large = [big_capture](int n) { return n + big_capture[0]; };
std::function<int(int)> f2 = large;  // heap allocation!

    Alternatives

    Approach Overhead Trade-off
    Templates Zero Type is part of the interface
    std::function Virtual call + possible heap Type-erased, copyable
    function_ref (C++26) Indirect call only Non-owning, no heap
    move_only_function (C++23) Virtual call + possible heap Move-only, no copy overhead
    Function pointer Indirect call No captures, no state
    Key insight: Use std::function when you need to store a callable (in a container, member variable, or callback registry). Use templates when you can accept the callable type as a template parameter. Use function_ref (C++26) or a hand-rolled non-owning wrapper when you only need to pass a callable without storing it.

    Advanced Type Erasure Patterns

    Function Table Approach (No Virtual Functions)

    Instead of virtual dispatch, you can store function pointers directly. This avoids vtable indirection and is slightly faster:

    template <typename R, typename... Args>
class LightFunction {
    using InvokeFn = R(*)(void*, Args...);
    using DestroyFn = void(*)(void*);

    void* object_ = nullptr;
    InvokeFn invoke_ = nullptr;
    DestroyFn destroy_ = nullptr;

public:
    template <typename F>
    LightFunction(F f) {
        object_ = new F(std::move(f));
        invoke_ = [](void* obj, Args... args) -> R {
            return (*static_cast<F*>(obj))(std::forward<Args>(args)...);
        };
        destroy_ = [](void* obj) { delete static_cast<F*>(obj); };
    }

    R operator()(Args... args) {
        return invoke_(object_, std::forward<Args>(args)...);
    }

    ~LightFunction() { if (destroy_) destroy_(object_); }
};

    Non-Owning Type Erasure (function_ref)

    template <typename> class FunctionRef;

template <typename R, typename... Args>
class FunctionRef<R(Args...)> {
    void* object_;
    R(*invoke_)(void*, Args...);

public:
    template <typename F>
    FunctionRef(F&& f) noexcept
        : object_(const_cast<void*>(static_cast<const void*>(std::addressof(f))))
        , invoke_([](void* obj, Args... args) -> R {
            return (*static_cast<std::remove_reference_t<F>*>(obj))(
                std::forward<Args>(args)...);
          })
    {}

    R operator()(Args... args) const {
        return invoke_(object_, std::forward<Args>(args)...);
    }
};

// Usage: zero overhead, non-owning
void process(FunctionRef<int(int)> callback) {
    std::cout << callback(5);
}
process([](int x) { return x * 2; });  // no heap, no copy

    Summary

    • Type erasure hides concrete types behind a uniform, non-template interface using an abstract base + templated derived pattern.
    • std::function erases callable types, using virtual dispatch and optionally SBO to avoid heap allocation for small callables.
    • std::any erases any copyable type, retrievable only with std::any_cast and the exact type.
    • Custom type-erased wrappers (the Sean Parent pattern) enable value-semantic polymorphism without inheritance.
    • SBO stores small objects inline, avoiding heap allocation. Buffer sizes vary by implementation.
    • Performance costs include indirect calls (preventing inlining) and potential heap allocation. Use templates when you can; type erasure when you must.
