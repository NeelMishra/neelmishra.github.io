    Sanitizers: Detecting UB at Runtime

    March 2026 · 18 min read

    Why Sanitizers Exist

    Undefined behavior is invisible. A program with UB can pass every test, produce correct output, run for years in production, and then break catastrophically when the compiler is upgraded or the code is compiled on a different platform. Static analysis catches some UB at compile time, but the vast majority of UB depends on runtime values and cannot be detected without executing the code.

    Sanitizers are compiler instrumentation tools that insert runtime checks into your program to detect UB, memory errors, data races, and other bugs as they happen. They are the single most effective tool for finding bugs in C++ code, and every serious C++ project should use them in its test infrastructure.

    The major sanitizers are:

    • AddressSanitizer (ASan): Detects memory errors (buffer overflow, use-after-free, stack-use-after-return, memory leaks)
    • UndefinedBehaviorSanitizer (UBSan): Detects undefined behavior (signed overflow, null dereference, misaligned access, shift errors)
    • ThreadSanitizer (TSan): Detects data races between threads
    • MemorySanitizer (MSan): Detects use of uninitialized memory
    • LeakSanitizer (LSan): Detects memory leaks (often bundled with ASan)

    All of these are built into GCC and Clang. MSVC supports ASan and some subset of UBSan. They are enabled with simple compiler flags and require no source code changes.

    AddressSanitizer (ASan)

    AddressSanitizer is the most widely used sanitizer. It detects a broad range of memory errors with a relatively modest performance overhead (typically 2x slowdown and 2-3x memory increase). It was developed at Google and has found tens of thousands of bugs in Chrome, the Linux kernel, LLVM itself, and countless other projects.

    What ASan detects

    • Heap buffer overflow: Reading or writing past the end of a heap allocation
    • Stack buffer overflow: Reading or writing past the end of a stack variable
    • Global buffer overflow: Reading or writing past the end of a global variable
    • Use-after-free: Accessing heap memory after it has been freed
    • Use-after-return: Accessing stack memory of a returned function (optional, with ASAN_OPTIONS=detect_stack_use_after_return=1)
    • Use-after-scope: Accessing a local variable outside its scope
    • Double free: Freeing the same memory twice
    • Memory leaks: Via the integrated LeakSanitizer (enabled by default on Linux)

    How to enable ASan

    # GCC or Clang:
g++ -fsanitize=address -fno-omit-frame-pointer -g -O1 program.cpp -o program
clang++ -fsanitize=address -fno-omit-frame-pointer -g -O1 program.cpp -o program

# -fno-omit-frame-pointer: better stack traces
# -g: include debug information for readable output
# -O1: recommended optimization level (fast enough, but preserves stack frames)

# MSVC:
cl /fsanitize=address /Zi program.cpp

    You must compile all translation units with the sanitizer flags. If you link sanitized code with non-sanitized code, the sanitizer may miss bugs or produce false positives.

    How ASan works: shadow memory

    ASan uses a technique called shadow memory. For every 8 bytes of application memory, ASan maintains 1 byte of shadow memory that records where the application memory is accessible. A shadow byte value of 0 means all 8 application bytes are accessible. A value of k (1-7) means only the first k bytes are accessible. A negative value means the memory is entirely inaccessible (freed, part of a redzone, or beyond the stack frame).

    When the program allocates memory, ASan inserts redzones (poisoned regions) around the allocation. A heap allocation of N bytes is padded with redzones before and after:

    // Logical view of a heap allocation with ASan:
[LEFT REDZONE] [USER DATA: N bytes] [RIGHT REDZONE]
   poisoned        accessible           poisoned

// Shadow memory for this region:
  0xff 0xff    0x00 0x00 0x00 ...    0xff 0xff
  (poison)      (accessible)          (poison)

    Every memory access in the program is instrumented: before the load or store executes, ASan checks the corresponding shadow byte. If the shadow byte indicates the memory is inaccessible, ASan reports the error with a detailed stack trace showing where the access occurred and where the memory was allocated (and freed, if applicable).

    For use-after-free detection, ASan does not immediately return freed memory to the system. Instead, it places freed memory in a quarantine and poisons it. If the program accesses this quarantined memory, ASan catches it. The quarantine is bounded in size; when it fills up, the oldest memory is actually freed and can be reused by new allocations.

    Reading ASan output

    When ASan detects an error, it produces a detailed report:

    =================================================================
==12345==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x602000000014
READ of size 4 at 0x602000000014 thread T0
    #0 0x4a3b2c in main /home/user/example.cpp:8:15
    #1 0x7f8a3b2c in __libc_start_main

0x602000000014 is located 0 bytes after 20-byte region [0x602000000000,0x602000000014)
allocated by thread T0 here:
    #0 0x493e7c in operator new[](unsigned long)
    #1 0x4a3b0a in main /home/user/example.cpp:6:14

Shadow bytes around the buggy address:
  0x0c047fff7fb0: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
  0x0c047fff7fc0: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
=>0x0c047fff7fd0: fa fa 00 04[fa]fa fa fa fa fa fa fa fa fa fa fa
  0x0c047fff7fe0: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
Shadow byte legend:
  Addressable:           00
  Heap left redzone:     fa
  Freed heap region:     fd
  Stack left redzone:    f1

    The key information in this report:

    • Error type: heap-buffer-overflow
    • Access type and size: READ of size 4
    • Stack trace of the access: main at example.cpp:8
    • Stack trace of the allocation: main at example.cpp:6
    • Shadow memory dump: The [fa] marker shows the access hit a heap redzone

    The shadow memory dump uses single-character codes: 00 = addressable, fa = heap redzone, fd = freed heap, f1 = stack left redzone, f3 = stack right redzone. The brackets [fa] indicate the shadow byte corresponding to the bad access.

    Key insight: ASan reports include stack traces for both the bad access and the allocation (and deallocation, for use-after-free). This makes root-cause analysis much faster than debugging a crash dump.

    UndefinedBehaviorSanitizer (UBSan)

    UBSan detects undefined behavior at runtime. Unlike ASan (which focuses on memory), UBSan catches UB in arithmetic, type conversions, control flow, and other non-memory categories.

    What UBSan detects

    • Signed integer overflow (and unsigned overflow, with a separate flag)
    • Shift by too much or by negative amount
    • Division by zero and INT_MIN / -1
    • Null pointer dereference
    • Misaligned pointer access
    • Out-of-bounds array indexing (for some array types)
    • Reaching __builtin_unreachable()
    • Invalid bool value (a bool that is not 0 or 1)
    • Invalid enum value
    • Floating-point overflow in cast to integer
    • Calling a function through a pointer of the wrong type
    • Returning from a value-returning function without a return statement

    How to enable UBSan

    # Enable all UBSan checks:
clang++ -fsanitize=undefined -g -O1 program.cpp -o program

# Enable specific checks:
clang++ -fsanitize=signed-integer-overflow,null,alignment -g program.cpp

# Combine with ASan (they are compatible):
clang++ -fsanitize=address,undefined -g -O1 program.cpp

    UBSan can operate in two modes:

    • Trap mode (-fsanitize-trap=undefined): Each UB triggers a trap instruction (equivalent to abort()). The program terminates immediately. Minimal overhead, no runtime library needed.
    • Report mode (default): UBSan prints a diagnostic message and continues execution. This finds more bugs per run but may produce cascading errors.

    UBSan output example

    program.cpp:10:15: runtime error: signed integer overflow:
2147483647 + 1 cannot be represented in type 'int'
SUMMARY: UndefinedBehaviorSanitizer: undefined-behavior program.cpp:10:15

    UBSan output includes the file, line, column, the exact expression, the values involved, and the type. This is usually sufficient to understand and fix the bug immediately.

    Performance overhead

    UBSan is very lightweight. Most checks add a single comparison and branch per instrumented operation. Typical overhead is 10-20% runtime increase with negligible memory increase. This makes UBSan suitable for deployment in some production environments, especially in trap mode.

    Key insight: ASan and UBSan are compatible and can be enabled simultaneously. This combination catches both memory errors and arithmetic/logic UB. It is the recommended "first line of defense" configuration for CI testing.

    ThreadSanitizer (TSan)

    ThreadSanitizer detects data races in multithreaded programs. A data race occurs when two threads access the same memory location, at least one access is a write, and there is no synchronization between them. Data races are undefined behavior in C++11 and later.

    How to enable TSan

    clang++ -fsanitize=thread -g -O1 program.cpp -o program -lpthread

    Important: TSan cannot be combined with ASan. They use different shadow memory layouts that conflict. Run separate builds for ASan and TSan testing.

    How TSan works

    TSan maintains a vector clock for every thread and a shadow state for every memory location (8 bytes of shadow per byte of application memory). The shadow state records the most recent access to each location: which thread, what kind of access (read or write), and the thread's vector clock at the time.

    On every memory access, TSan:

    1. Checks if the current access conflicts with a previous access recorded in the shadow state
    2. Two accesses conflict if they are from different threads, at least one is a write, and they are not ordered by a happens-before relationship (detected via vector clock comparison)
    3. If a conflict is found, TSan reports the data race with stack traces for both accesses

    TSan output example

    WARNING: ThreadSanitizer: data race (pid=12345)
  Write of size 4 at 0x7f9a3c000000 by thread T1:
    #0 increment /home/user/example.cpp:8:5
    #1 thread_func /home/user/example.cpp:15:5

  Previous read of size 4 at 0x7f9a3c000000 by thread T2:
    #0 read_value /home/user/example.cpp:20:12
    #1 thread_func2 /home/user/example.cpp:25:5

  Location is global 'counter' of size 4 at 0x7f9a3c000000
  Thread T1 (running) created at:
    #0 pthread_create
    #1 main /home/user/example.cpp:30:5

    The report shows both accesses (including their stack traces), identifies the memory location, and shows where each thread was created. This is usually enough to identify and fix the race condition.

    Performance overhead

    TSan is the most expensive sanitizer: typically 5-15x slowdown and 5-10x memory increase. This is because it must record metadata for every memory access. Despite the overhead, TSan is invaluable for concurrent code because data races are extremely difficult to find by other means.

    Key insight: Data races are nearly impossible to find via testing alone because they depend on thread scheduling, which is non-deterministic. TSan detects them deterministically by tracking happens-before relationships. If your code uses threads, run TSan.

    MemorySanitizer (MSan)

    MemorySanitizer detects reads of uninitialized memory. It tracks the "initialized" status of every byte in the program using shadow memory. When an uninitialized value is used in a way that affects program behavior (a branch condition, an output operation, a system call argument), MSan reports the error.

    How to enable MSan

    # Clang only (GCC does not support MSan):
clang++ -fsanitize=memory -fno-omit-frame-pointer -g -O1 program.cpp -o program

# With origin tracking (slower but shows where the uninitialized value came from):
clang++ -fsanitize=memory -fsanitize-memory-track-origins=2 -g program.cpp

    Important: MSan requires that the entire program (including all libraries) be compiled with MSan. Any non-instrumented code will be treated as producing initialized values, leading to false negatives. In practice, this means you need MSan-instrumented versions of your standard library and all dependencies. This makes MSan harder to deploy than ASan or UBSan.

    What MSan detects

    int x;          // uninitialized
if (x > 0) {   // MSan reports: use of uninitialized value
    // ...
}

int arr[10];
int sum = 0;
for (int i = 0; i < 10; i++) {
    sum += arr[i];  // MSan reports: use of uninitialized value
}
std::cout << sum;

    MSan is precise about when to report: it does not report merely reading an uninitialized value into a register. It reports when the uninitialized value influences program behavior. This reduces false positives from code that copies uninitialized padding bytes without using them.

    Performance overhead

    MSan typically imposes 2-3x slowdown and 2-3x memory increase, similar to ASan. The origin tracking mode adds additional overhead (3-5x total).

    Key insight: MSan cannot be combined with ASan or TSan. You need separate builds. MSan also requires all dependencies to be instrumented, making it the hardest sanitizer to deploy. Use -fsanitize-memory-track-origins=2 for the most useful diagnostics.

    LeakSanitizer (LSan)

    LeakSanitizer detects memory leaks at program exit. On Linux, LSan is integrated into ASan and enabled by default. It can also be used standalone.

    How to enable LSan

    # Integrated with ASan (automatic on Linux):
clang++ -fsanitize=address -g program.cpp

# Standalone:
clang++ -fsanitize=leak -g program.cpp

    How LSan works

    At program exit, LSan performs a conservative garbage collection scan. It marks all heap allocations as potentially leaked. Then it scans the stack, registers, and global variables for pointers into the heap. Any allocation that is reachable through a chain of pointers from the roots is marked as "reachable" (not leaked). The remaining allocations are reported as leaks.

    LSan classifies leaks into three categories:

    • Definitely lost: No pointer to the allocation exists anywhere in the program
    • Indirectly lost: The allocation is reachable only through definitely-lost allocations
    • Possibly lost: A pointer to the interior (not the start) of the allocation exists

    LSan output example

    =================================================================
==12345==ERROR: LeakSanitizer: detected memory leaks

Direct leak of 40 byte(s) in 1 object(s) allocated from:
    #0 0x493e7c in operator new[](unsigned long)
    #1 0x4a3b0a in main /home/user/example.cpp:5:14

SUMMARY: AddressSanitizer: 40 byte(s) leaked in 1 allocation(s).
    Key insight: LSan adds virtually zero overhead because it only runs at program exit. It is enabled by default with ASan on Linux, so if you are using ASan, you get leak detection for free.

    Sanitizer Detects Overhead Combine?

    UBSan Undefined behavior ~1.2x CPU, ~1x RAM + ASan, + TSan, + MSan
    TSan Data races ~5-15x CPU, ~5-10x RAM + UBSan only
    MSan Uninitialized reads ~3x CPU, ~3x RAM + UBSan only

    Best Practices for CI

    Integrating sanitizers into your continuous integration pipeline is one of the highest-leverage improvements you can make to a C++ project's quality. Here is the recommended approach:

    Minimum viable setup: two builds

    At minimum, maintain two CI builds:

    1. ASan + UBSan build: Catches memory errors and undefined behavior. Covers the most common bug categories with moderate overhead.
    2. TSan build: Catches data races. Required for any code that uses threads.
    # Build 1: ASan + UBSan
CXX=clang++ CXXFLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer -g -O1"

# Build 2: TSan
CXX=clang++ CXXFLAGS="-fsanitize=thread -g -O1"

    Full setup: three builds

    If your project can afford the CI time, add MSan:

    # Build 3: MSan (requires instrumented libc++)
CXX=clang++ CXXFLAGS="-fsanitize=memory -fsanitize-memory-track-origins=2 -g -O1"

    The MSan build requires a custom-built instrumented standard library, which adds complexity. Many projects skip MSan and rely on Valgrind or code review for uninitialized value detection.

    Environment variables

    Sanitizer behavior can be tuned via environment variables:

    # Abort on first error (useful in CI to get a clear failure):
ASAN_OPTIONS=halt_on_error=1:detect_stack_use_after_return=1
UBSAN_OPTIONS=halt_on_error=1:print_stacktrace=1
TSAN_OPTIONS=halt_on_error=1:second_deadlock_stack=1

# Suppress known issues (use sparingly):
ASAN_OPTIONS=suppressions=asan_suppressions.txt
LSAN_OPTIONS=suppressions=lsan_suppressions.txt

    Suppressing false positives

    Sanitizers have very low false positive rates, but they do occur, especially with older or exotic code. Each sanitizer supports suppression files that list known false positives by function name or file. Use suppressions sparingly and always document why each suppression exists.

    Fuzzing with sanitizers

    Combining sanitizers with fuzzing is the most powerful bug-finding combination available. libFuzzer (built into Clang) and AFL++ can generate inputs that explore code paths your test suite never reaches. When a fuzzed input triggers a sanitizer error, you get a minimal reproducer and a precise error report.

    # Build with ASan + UBSan + libFuzzer:
clang++ -fsanitize=address,undefined,fuzzer -g -O1 fuzz_target.cpp -o fuzzer
./fuzzer corpus/
    Key insight: Sanitizers find bugs that tests miss. Tests verify expected behavior on expected inputs. Sanitizers verify the absence of UB and memory errors on whatever inputs you provide. Combine them: run your tests with sanitizers enabled, and fuzz with sanitizers for maximum coverage.

    Shadow Memory: The Mechanism Behind Sanitizers

    The shadow memory technique is central to how ASan and MSan work. Understanding it helps you reason about sanitizer performance and limitations.

    The core idea: for every byte (or group of bytes) of application memory, maintain a "shadow" byte that stores metadata. The mapping from application address to shadow address must be fast (constant time), so it is computed via a simple formula:

    // ASan shadow memory mapping (typical):
ShadowAddr = (AppAddr >> 3) + ShadowOffset

// Each shadow byte covers 8 application bytes.
// ShadowOffset is a fixed constant (e.g., 0x7fff8000 on Linux x86-64).

// The shadow byte encodes:
//   0:    all 8 bytes are addressable
//   1-7:  only the first k bytes are addressable
//   negative: entire 8-byte region is poisoned (redzone, freed, etc.)

    This mapping divides the virtual address space into two halves: the application half and the shadow half. The shadow half is mmap'd on demand and never grows beyond 1/8th of the application's virtual address space usage.

    Every memory access instruction is instrumented with a check:

    // For a load of N bytes at address A:
shadow_value = *(char*)(A >> 3 + SHADOW_OFFSET);
if (shadow_value != 0) {
    // Check if specifically our access is in the poisoned region
    if ((A & 7) + N > shadow_value || shadow_value < 0) {
        __asan_report_load(A, N);
    }
}

    This check adds approximately 3-4 instructions per memory access. For aligned 8-byte accesses, the check simplifies to a single comparison. This is why ASan overhead is typically 2x rather than something larger: the checks are cheap and the branch predictor learns quickly (almost all accesses are valid).

    Valgrind vs Sanitizers

    Before sanitizers, Valgrind (specifically its Memcheck tool) was the primary tool for detecting memory errors in C/C++. Valgrind runs the program on a virtual CPU and instruments every memory access dynamically. How do sanitizers compare?

    Aspect Sanitizers Valgrind
    Overhead 2x (ASan) 10-50x
    Recompilation Required Not required
    Stack errors Detected (ASan) Limited detection
    Global errors Detected (ASan) Not detected
    Platform Linux, macOS, Windows Linux, macOS
    Uninit values MSan (Clang only) Detected

    For new projects, sanitizers are generally preferred because of their much lower overhead and broader detection of stack and global errors. Valgrind remains useful when recompilation is not possible (e.g., debugging a third-party binary without source code).

    Limitations of Sanitizers

    Sanitizers are powerful but not omniscient. Understanding their limitations ensures you do not develop a false sense of security:

    • Runtime coverage dependent: Sanitizers only detect UB that is actually triggered during execution. If your test suite does not exercise a particular code path, sanitizers will not find bugs on that path. Combine sanitizers with fuzzing and high test coverage.
    • False negatives: ASan can miss certain intra-object overflows (e.g., overflowing one field into another field of the same struct). UBSan cannot detect all forms of UB (e.g., strict aliasing violations are not fully covered).
    • Mutual exclusion: ASan, TSan, and MSan cannot be combined. You need separate builds.
    • MSan requires full instrumentation: Any uninstrumented code (including system libraries) is a blind spot for MSan.
    • Overhead: While modest for ASan/UBSan, TSan's overhead can make some test suites impractically slow. Prioritize TSan testing for concurrent code.
    • No compile-time detection: Sanitizers catch UB at runtime, not at compile time. Use -Wall -Wextra and static analyzers for compile-time detection.

    Summary

    Sanitizers are the most effective runtime tools for detecting undefined behavior and memory errors in C++. The practical advice is simple:

    • Always run ASan + UBSan in your CI pipeline. The overhead is modest and the bug-finding power is enormous.
    • Add TSan if your code uses threads. Data races are nearly impossible to find otherwise.
    • Add MSan if you can manage the instrumented-everything requirement. Uninitialized value bugs are subtle and dangerous.
    • Combine with fuzzing for maximum effectiveness. Fuzzing generates the unusual inputs that trigger UB.
    • Treat sanitizer findings as real bugs. Sanitizers have very low false positive rates. If a sanitizer reports an issue, fix it.
