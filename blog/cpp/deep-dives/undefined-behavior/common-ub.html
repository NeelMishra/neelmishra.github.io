    Common Sources of Undefined Behavior

    March 2026 · 22 min read

    A Catalog of Foot-Guns

    The C++ standard contains over 200 distinct instances of undefined behavior. Some are obscure corner cases that you will never encounter. Others are common traps that experienced programmers fall into regularly. This article catalogs the most common and dangerous sources of UB, with code examples, explanations of what the compiler is allowed to do, and how to avoid each one.

    For each source of UB, we show:

    • Code that triggers the UB
    • What the compiler might do (and why)
    • The well-defined alternative

    If you have not read the What is Undefined Behavior? article, start there for the conceptual foundations.

    1. Signed Integer Overflow

    Signed integer overflow is UB. Unsigned integer overflow is well-defined (wraps modulo 2N). This distinction surprises many programmers, but it is one of the most consequential UB rules because it enables a huge family of loop optimizations.

    int x = INT_MAX;
int y = x + 1;  // UB: signed overflow

// The compiler may:
// - Assume this never happens and optimize based on that
// - Produce INT_MIN (wrap) on some platforms
// - Produce a completely different value
// - Optimize away code that depends on this result

    The compiler is allowed to assume that signed arithmetic never overflows. This means expressions like x + 1 > x can be folded to true, loop induction variables can be widened without bounds checks, and algebraic simplifications apply freely. The performance gains from this assumption are substantial, which is why the standard makes it UB rather than defining wrapping behavior.

    A particularly dangerous pattern is the overflow check that gets optimized away:

    // Broken overflow check:
bool will_overflow(int a, int b) {
    return a + b < a;  // UB if a + b overflows!
    // Compiler assumes no overflow, so a + b >= a, returns false always
}

// Correct overflow check:
bool will_overflow_safe(int a, int b) {
    return b > 0 && a > INT_MAX - b;
    // No UB: we check using subtraction, not addition
}
    How to avoid: Use unsigned types when wrapping behavior is needed. Use __builtin_add_overflow (GCC/Clang) or check ranges before performing arithmetic. In C++26, std::overflow helpers are being standardized.

    2. Null Pointer Dereference

    Dereferencing a null pointer is undefined behavior. It does not necessarily cause a segfault. On many embedded systems, address 0 is valid memory. Even on systems with memory protection, the optimizer can transform the code in ways that prevent the dereference from ever executing as a load instruction.

    int* p = nullptr;
int x = *p;  // UB: null dereference

// The compiler has already assumed p is non-null after this line.
// Any subsequent null check is dead code.

    The most dangerous variant is the one we saw in the previous article: dereferencing before checking. But even simply forming a reference from a null pointer is UB:

    int* p = nullptr;
int& r = *p;  // UB: forming a reference to null
// Even if you never use r, the UB has already occurred

    This is UB even if the reference is never used, because the standard says a reference must be bound to a valid object. The act of binding it to null is itself undefined.

    How to avoid: Always check for null before dereferencing. Use references when null is not a valid value. Use std::optional for values that may be absent. Use gsl::not_null<T*> to document and enforce non-null contracts.

    3. Use-After-Free

    Accessing memory after it has been deallocated is UB. This is one of the most dangerous categories of UB because it can lead to arbitrary code execution. The memory may have been reused for a different allocation, so your read or write silently corrupts some other object.

    int* p = new int(42);
delete p;
int x = *p;  // UB: use-after-free

// The memory at *p might now contain:
// - The old value 42 (looks like it works!)
// - Heap metadata (corruption if you write to it)
// - A completely different object (silent data corruption)
// - An unmapped page (crash)

    A subtler form occurs with containers that invalidate iterators and references:

    std::vector<int> v = {1, 2, 3};
int& ref = v[0];
v.push_back(4);    // may reallocate, invalidating ref
int x = ref;        // UB: ref may be dangling

    The push_back may cause the vector to reallocate its internal buffer. The reference ref still points to the old buffer, which has been freed. Accessing it is UB.

    How to avoid: Use smart pointers (std::unique_ptr, std::shared_ptr) to manage ownership. Set raw pointers to nullptr after delete. Be aware of container invalidation rules. AddressSanitizer detects use-after-free reliably.

    4. Dangling References

    A dangling reference is a reference that outlives the object it refers to. Accessing it is UB. The most common cause is returning a reference to a local variable:

    int& dangerous() {
    int local = 42;
    return local;  // UB: local is destroyed when the function returns
}

int x = dangerous();  // accessing a dangling reference

    Modern compilers warn about this specific pattern (-Wreturn-local-addr), but subtler variants escape detection:

    const std::string& get_name(const std::map<int, std::string>& m, int key) {
    auto it = m.find(key);
    if (it != m.end()) return it->second;
    static std::string empty;
    return empty;
}

// This is actually fine because the map outlives the reference.
// But this is NOT fine:
std::string_view get_view() {
    std::string temp = "hello";
    return temp;  // UB: string_view points into temp, which is destroyed
}

std::string_view sv = get_view();
std::cout << sv;  // UB: dangling view

    std::string_view is a non-owning reference to a string. If the underlying string is destroyed, the view dangles. This is one of the most common sources of dangling references in modern C++.

    How to avoid: Never return references or views to local variables. Understand the lifetime of the referred-to object. When in doubt, return by value. The compiler cannot always detect dangling references, but AddressSanitizer (stack-use-after-return) and lifetime analysis tools can help.

    5. Buffer Overflow

    Accessing an array out of bounds is UB. This applies to raw arrays, operator[] on standard containers (which does not check bounds), and pointer arithmetic that goes past the end of an allocation.

    int arr[5] = {1, 2, 3, 4, 5};
int x = arr[10];  // UB: out-of-bounds read
arr[-1] = 42;     // UB: out-of-bounds write

// With standard containers:
std::vector<int> v = {1, 2, 3};
int y = v[5];     // UB: operator[] does not check bounds
int z = v.at(5);  // SAFE: throws std::out_of_range

    Buffer overflows are one of the oldest and most exploited vulnerability classes in computing history. Stack-based buffer overflows allow arbitrary code execution by overwriting the return address. Heap-based overflows corrupt allocator metadata or adjacent objects.

    A common C-style mistake that persists in C++ code:

    char buf[256];
std::strcpy(buf, user_input);  // UB if user_input exceeds 255 chars

// Safe alternatives:
std::strncpy(buf, user_input, sizeof(buf) - 1);
buf[sizeof(buf) - 1] = '\0';

// Or better, use std::string:
std::string safe_buf = user_input;  // no overflow possible
    How to avoid: Use std::vector, std::array, and std::string instead of raw C arrays. Use .at() for bounds-checked access during development. Use std::span (C++20) to pass bounded ranges. AddressSanitizer reliably detects buffer overflows.

    6. Reading Uninitialized Variables

    Reading the value of a variable that has not been initialized is UB. The value is not "whatever was in memory" in any meaningful sense, because the compiler may optimize based on the assumption that you never read uninitialized variables.

    int x;           // uninitialized
int y = x + 1;   // UB: reading uninitialized x

// The compiler does not have to give you "whatever is on the stack."
// It can:
// - Propagate the "uninitialized" status to y
// - Assume x has any value that makes the program well-defined
// - Eliminate code paths based on assumed values of x

    A particularly nasty manifestation: the compiler can assume an uninitialized variable has a convenient value and optimize based on that assumption. If the variable is used in a branch condition, the compiler can choose which branch to take:

    bool flag;  // uninitialized
if (flag) {
    do_something();
} else {
    do_something_else();
}
// The compiler can choose EITHER branch, or neither, or both.
// It doesn't have to "pick a random value" for flag.
    How to avoid: Always initialize variables. Use int x = 0; or int x{};. Enable -Wuninitialized (usually part of -Wall). MemorySanitizer (MSan) detects use of uninitialized values at runtime.

    7. Strict Aliasing Violations

    The strict aliasing rule says that you cannot access an object through a pointer of a different type (with a few exceptions). Violating this rule is UB. The rule exists to allow the compiler to assume that pointers of unrelated types do not alias (point to the same memory), enabling it to keep values in registers and reorder loads and stores.

    float f = 3.14f;
int* ip = reinterpret_cast<int*>(&f);
int bits = *ip;  // UB: strict aliasing violation

// The compiler may:
// - Return the correct bit pattern (looks like it works)
// - Return a stale value from a register (because it assumes
//   writing to a float doesn't affect an int read)
// - Reorder the read before the write to f

    The exceptions to the strict aliasing rule are:

    • Accessing through a char*, unsigned char*, or std::byte* (you can examine any object's bytes)
    • Accessing through a pointer to a base class
    • Accessing through a signed/unsigned variant of the same type
    // Well-defined: use memcpy for type punning
float f = 3.14f;
int bits;
std::memcpy(&bits, &f, sizeof(bits));  // well-defined

// Well-defined in C++20: use std::bit_cast
int bits2 = std::bit_cast<int>(f);  // well-defined, constexpr
    How to avoid: Use std::memcpy or std::bit_cast for type punning. Never use reinterpret_cast to "peek" at an object's bits through a different pointer type. Compile with -fstrict-aliasing -Wstrict-aliasing to enable warnings.

    8. Data Races

    If two threads access the same memory location, at least one access is a write, and there is no synchronization between them, the behavior is undefined. This is a data race, and it is one of the most insidious forms of UB because it is non-deterministic and may only manifest under specific timing conditions.

    int counter = 0;  // shared, non-atomic

// Thread 1:
counter++;

// Thread 2:
counter++;

// Data race: undefined behavior
// The result might be 1, 2, or any other value.
// The compiler can cache counter in a register,
// reorder operations, or do anything else.

    Data races are UB even if you think the operations are "atomic" at the hardware level. On x86, a simple integer increment might be a single instruction, but the compiler is free to transform it into multiple instructions, reorder it relative to other memory operations, or eliminate it entirely. The C++ memory model requires explicit synchronization.

    // Fix with atomic:
std::atomic<int> counter{0};

// Thread 1:
counter.fetch_add(1, std::memory_order_relaxed);

// Thread 2:
counter.fetch_add(1, std::memory_order_relaxed);

// Well-defined: result is always 2

// Or fix with a mutex:
std::mutex mtx;
int counter = 0;

// Thread 1:
{ std::lock_guard lock(mtx); counter++; }

// Thread 2:
{ std::lock_guard lock(mtx); counter++; }
    How to avoid: Use std::atomic for lock-free shared variables. Use std::mutex and std::lock_guard for complex shared state. ThreadSanitizer (TSan) detects data races at runtime with high accuracy. See the concurrency series for in-depth coverage.

    9. Double Free

    Calling delete (or free) on a pointer that has already been deallocated is UB. The heap allocator's internal metadata has already been updated; a second free corrupts that metadata, potentially allowing an attacker to control subsequent allocations.

    int* p = new int(42);
delete p;
delete p;  // UB: double free

// Typical consequences:
// - Crash in the allocator
// - Heap corruption that manifests much later
// - Exploitable vulnerability (heap spray attacks)

    Double free is especially dangerous because the symptoms often appear far from the actual bug. The second delete corrupts the allocator's free list, but the corruption is not noticed until a later allocation or free operation touches the corrupted metadata.

    A common cause is multiple ownership without clear semantics:

    void foo(int* p) { delete p; }
void bar(int* p) { delete p; }

int* shared = new int(10);
foo(shared);  // deletes shared
bar(shared);  // UB: double free
    How to avoid: Use std::unique_ptr for exclusive ownership. Use std::shared_ptr for shared ownership. Never use raw new/delete directly; use std::make_unique and std::make_shared. Set raw pointers to nullptr after delete (deleting nullptr is a no-op by definition). AddressSanitizer detects double frees.

    10. Infinite Loops Without Side Effects

    Starting in C++11, a loop that meets all of the following criteria has undefined behavior:

    • It does not terminate
    • It has no observable side effects (no I/O, no volatile accesses, no atomic operations, no calls to library I/O functions)
    • It does not contain a call to a function that might not return
    void wait_for_flag(bool flag) {
    while (!flag) {
        // no side effects
    }
}

// The compiler can assume this loop terminates (since an infinite
// loop without side effects is UB). It can remove the loop entirely,
// or assume flag is always true.

    This rule exists because the optimizer needs to be able to reason about loop termination. Many loop transformations (hoisting, interchange, vectorization) are only valid if the loop terminates. By making non-terminating loops without side effects UB, the standard gives the compiler permission to assume all loops terminate.

    The fix is to add a side effect. If you are polling a shared flag, use an atomic:

    void wait_for_flag(std::atomic<bool>& flag) {
    while (!flag.load(std::memory_order_acquire)) {
        // atomic load is a side effect, loop is well-defined
    }
}
    How to avoid: Ensure loops either terminate or have observable side effects. Use std::atomic for polling flags. Use volatile for memory-mapped I/O (but not as a synchronization mechanism). If you intentionally want an infinite loop, add a side effect like std::this_thread::yield().

    11. Shifting by Too Much (or by a Negative Amount)

    Left or right shifting an integer by a negative number of bits, or by a number of bits greater than or equal to the width of the type, is UB. Shifting a negative signed integer left is also UB (prior to C++20, which made it defined).

    int x = 1;
int y = x << 32;   // UB on platforms where int is 32 bits
int z = x << -1;   // UB: negative shift count
int w = x >> 33;   // UB if int is 32 bits

uint32_t a = 1u;
uint32_t b = a << 32;  // Still UB! The width of uint32_t is 32

    On x86, the CPU masks the shift count to 5 bits (for 32-bit shifts) or 6 bits (for 64-bit shifts). So x << 32 becomes x << 0 on x86, returning x unchanged. On ARM, the behavior is different. By making the behavior undefined, the compiler can use the native shift instruction without inserting a check on the shift count.

    How to avoid: Always ensure the shift count is in the range [0, sizeof(type) * CHAR_BIT - 1]. Use unsigned types for bitwise operations. Add explicit range checks if the shift count comes from runtime input.

    12. Modifying String Literals

    String literals are stored in read-only memory. Modifying them is UB. In C, string literals have type char[] (but modifying is still UB). In C++, they have type const char[], so you need a cast to even get a non-const pointer, but legacy code often does this.

    char* s = "hello";   // deprecated in C++: implicit const removal
s[0] = 'H';          // UB: modifying a string literal

// The literal "hello" is typically in .rodata.
// On systems with memory protection, this usually crashes (SIGSEGV).
// On embedded systems without memory protection, it might silently
// corrupt other data.

    A subtle variant: returning a string literal from a function works fine (the literal has static storage duration), but if you modify the result, it is UB:

    const char* greeting() { return "hello"; }
char* p = const_cast<char*>(greeting());
p[0] = 'H';  // UB: modifying a string literal through const_cast
    How to avoid: Always use const char* (or std::string_view) for string literals. If you need a mutable string, copy it into a std::string or char[] array.

    13. Division by Zero and INT_MIN / -1

    Integer division by zero is UB. Additionally, dividing INT_MIN by -1 is UB because the mathematical result (INT_MAX + 1) cannot be represented as a signed integer.

    int x = 10 / 0;         // UB: division by zero
int y = INT_MIN / -1;   // UB: result overflows signed int

// On x86, these typically cause a hardware trap (SIGFPE).
// But the optimizer can sometimes remove the division or
// reorder it past checks.

    The INT_MIN / -1 case is especially tricky because most programmers only think about division by zero. On two's complement systems (all modern CPUs), INT_MIN is -2147483648 and INT_MAX is 2147483647, so the result would be 2147483648, which does not fit in an int.

    How to avoid: Always check the divisor is non-zero before dividing. For signed division, also check for the INT_MIN / -1 case. Use helper functions that perform both checks.

    14. Sequence Point / Sequencing Violations

    Modifying a variable more than once between sequence points (C) or within the same expression without an intervening sequence point (C++) is UB:

    int i = 0;
i = i++ + ++i;  // UB: i modified multiple times without sequencing

int arr[5] = {};
int j = 0;
arr[j] = j++;   // UB: j is read and written without sequencing

// C++17 tightened the rules: some cases that were UB in C++14
// became well-defined. But many cases remain UB.

    The root issue is that the compiler is free to evaluate subexpressions in any order. If two subexpressions modify the same variable, or one modifies and the other reads, and there is no sequencing relationship between them, the result is UB.

    How to avoid: Never modify a variable more than once in the same expression. Split complex expressions into separate statements. Enable -Wsequence-point (part of -Wall) for compiler diagnostics.

    15. Calling Virtual Functions During Construction/Destruction

    Calling a pure virtual function during construction or destruction of an object is UB. Calling a non-pure virtual function is well-defined but dispatches to the current class (not the most-derived class), which is often surprising. The UB case:

    class Base {
public:
    Base() {
        init();  // calls Base::init if init is non-pure-virtual
                 // UB if init is pure virtual!
    }
    virtual void init() = 0;
};

class Derived : public Base {
public:
    void init() override { std::cout << "Derived::init\n"; }
};

Derived d;  // UB: Base constructor calls pure virtual init()

    During Base's constructor, the vptr points to Base's vtable. Base::init() is pure virtual, so the vtable slot contains a pointer to __cxa_pure_virtual, which typically aborts the program. But the standard says this is UB, so the compiler is not required to do anything specific.

    How to avoid: Never call virtual functions in constructors or destructors when you expect polymorphic dispatch. Use a two-phase initialization pattern (separate init() method called after construction) or the CRTP pattern.

    16. Misaligned Pointer Access

    Creating a pointer that does not meet the alignment requirements of the pointed-to type and dereferencing it is UB:

    char buffer[16] = {};
int* p = reinterpret_cast<int*>(buffer + 1);  // misaligned
int x = *p;  // UB: p is not aligned to alignof(int)

// On x86, this usually works (x86 tolerates misalignment, with a
// performance penalty). On ARM, MIPS, SPARC, and many other
// architectures, this causes a hardware trap (SIGBUS).

    Misalignment is commonly introduced when parsing binary data, implementing custom allocators, or working with serialized formats. The x86 tolerance for misalignment masks these bugs during development on desktop systems, and they surface when porting to mobile or embedded platforms.

    How to avoid: Use std::memcpy to read from unaligned memory. Use alignas when declaring storage for specific types. UBSan with -fsanitize=alignment detects misaligned accesses.

    17. One Definition Rule (ODR) Violations

    The One Definition Rule states that every entity (class, function, variable, template) must have exactly one definition in the entire program. If two translation units define the same entity differently, it is UB with no diagnostic required:

    // file1.cpp
struct Point { int x; int y; };
void use(Point p) { std::cout << p.x; }

// file2.cpp
struct Point { double x; double y; };  // different definition!
void use(Point p);

// ODR violation: UB, no diagnostic required
// The linker may silently pick one definition and apply it everywhere,
// causing misinterpreted data and memory corruption.

    ODR violations are particularly dangerous because they produce no compiler error and often no linker error. The symptom is usually bizarre runtime behavior that changes depending on link order.

    How to avoid: Use header files to ensure consistent definitions. Use namespaces to avoid name collisions. Use -fno-common (default in GCC 10+). Link-time sanitizers can detect some ODR violations.

    How Many UB Rules Are There?

    The C++ standard contains a staggering number of UB instances. Depending on how you count, estimates range from 200 to over 500 distinct undefined behaviors. Many are obscure (violating requirements of the Compare concept in std::sort, for example), but the 17 categories we covered here account for the vast majority of UB encountered in practice.

    A useful mental model: UB lives at the boundaries. It occurs where values meet constraints (integer ranges, pointer validity, type compatibility, memory lifetime), and the programmer violates a constraint that the language does not enforce at runtime.

    The C++ community has made significant progress in reducing the surface area for UB. Modern C++ (C++11 and later) provides tools like smart pointers, std::optional, std::variant, std::span, and std::bit_cast that eliminate entire categories of UB when used consistently. The next frontier is contracts (likely C++26), which will allow programmers to express preconditions and postconditions that can be checked at runtime or used by the optimizer.

    Summary

    The most common sources of UB in C++ are:

    1. Signed integer overflow
    2. Null pointer dereference
    3. Use-after-free
    4. Dangling references
    5. Buffer overflow
    6. Reading uninitialized variables
    7. Strict aliasing violations
    8. Data races
    9. Double free
    10. Infinite loops without side effects
    11. Shift by too much
    12. Modifying string literals
    13. Division by zero / INT_MIN / -1
    14. Sequence point violations
    15. Pure virtual calls during construction
    16. Misaligned pointer access
    17. ODR violations

    Every one of these can be detected at compile time (partially, via warnings and static analysis) or at runtime (via sanitizers). The next article covers sanitizers: the most powerful runtime tools for detecting UB in practice.
