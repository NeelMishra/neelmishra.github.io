    Rule of 0/3/5: Managing Special Member Functions

    March 2026 · 20 min read

    The Six Special Member Functions

    C++ has six special member functions that the compiler can generate for you:

    1. Default constructor: T()
    2. Destructor: ~T()
    3. Copy constructor: T(const T&)
    4. Copy assignment operator: T& operator=(const T&)
    5. Move constructor: T(T&&)
    6. Move assignment operator: T& operator=(T&&)

    Whether the compiler generates each one depends on which ones you declare yourself. The rules are complex, and getting them wrong leads to silent performance regressions (lost move semantics) or outright bugs (double-free, use-after-move). The Rule of 0/3/5 is a set of guidelines that keeps you out of trouble.

    The Rule of Zero

    The Rule of Zero states: if your class does not directly manage a resource, do not define any special member functions. Let the compiler generate all of them.

    // Rule of Zero: no special members defined
class Employee {
    std::string name_;
    std::vector<std::string> skills_;
    int id_;

public:
    Employee(std::string name, int id)
        : name_(std::move(name)), id_(id) {}

    // Compiler generates:
    // ~Employee()                          - destroys members
    // Employee(const Employee&)            - copies each member
    // Employee& operator=(const Employee&) - copies each member
    // Employee(Employee&&)                 - moves each member
    // Employee& operator=(Employee&&)      - moves each member
};

    This works because each member (std::string, std::vector, int) already knows how to copy, move, and destroy itself. The compiler-generated functions just call the corresponding operation on each member, which is exactly what you want.

    Key insight: The Rule of Zero is the preferred approach. It means your class has zero boilerplate for resource management, zero chance of getting the special members wrong, and automatically gains move semantics. Use RAII wrappers (unique_ptr, shared_ptr, vector, string) for resource management so your own classes never need to define special members.

    The Rule of Three (Pre-C++11)

    If your class directly manages a resource (raw pointer, file handle, socket, etc.), you must define at least three special members:

    1. Destructor (to release the resource)
    2. Copy constructor (to deep-copy the resource)
    3. Copy assignment operator (to deep-copy the resource)

    If you need any one of these, you almost certainly need all three. Defining a destructor without a copy constructor means copies will share the same raw pointer, leading to double-free. Defining a copy constructor without a destructor means the resource leaks.

    // Rule of Three: manages a raw buffer
class Buffer {
    int* data_;
    size_t size_;

public:
    explicit Buffer(size_t n)
        : data_(new int[n]()), size_(n) {}

    // 1. Destructor
    ~Buffer() { delete[] data_; }

    // 2. Copy constructor (deep copy)
    Buffer(const Buffer& other)
        : data_(new int[other.size_]), size_(other.size_)
    {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // 3. Copy assignment (deep copy)
    Buffer& operator=(const Buffer& other) {
        if (this != &other) {
            delete[] data_;
            data_ = new int[other.size_];
            size_ = other.size_;
            std::copy(other.data_, other.data_ + size_, data_);
        }
        return *this;
    }
};

    The Rule of Five (C++11 and Beyond)

    C++11 added move semantics, extending the Rule of Three to the Rule of Five: if you define any of the destructor, copy constructor, or copy assignment operator, you should also define:

    1. Move constructor
    2. Move assignment operator
    class Buffer {
    int* data_;
    size_t size_;

public:
    explicit Buffer(size_t n)
        : data_(new int[n]()), size_(n) {}

    // 1. Destructor
    ~Buffer() { delete[] data_; }

    // 2. Copy constructor
    Buffer(const Buffer& other)
        : data_(new int[other.size_]), size_(other.size_)
    {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // 3. Copy assignment
    Buffer& operator=(const Buffer& other) {
        if (this != &other) {
            delete[] data_;
            data_ = new int[other.size_];
            size_ = other.size_;
            std::copy(other.data_, other.data_ + size_, data_);
        }
        return *this;
    }

    // 4. Move constructor
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_)
    {
        other.data_ = nullptr;
        other.size_ = 0;
    }

    // 5. Move assignment
    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] data_;
            data_ = other.data_;
            size_ = other.size_;
            other.data_ = nullptr;
            other.size_ = 0;
        }
        return *this;
    }
};

    Without the move operations, the class compiles and works, but every move silently falls back to a copy. For a buffer of a million elements, this is catastrophically wasteful.

    Implicit Generation Rules: The Big Table

    The rules for when the compiler generates (or deletes) special members are among the most complex parts of the C++ standard. The table below summarizes them:

    User declares Default ctor Destructor Copy ctor Copy assign Move ctor Move assign
    Nothingdefaultdefaultdefaultdefaultdefaultdefault
    Any constructornot declareddefaultdefaultdefaultdefaultdefault
    Destructordefaultuserdefault*default*not declarednot declared
    Copy ctornot declareddefaultuserdefault*not declarednot declared
    Copy assigndefaultdefaultdefault*usernot declarednot declared
    Move ctornot declareddefaultdeleteddeletedusernot declared
    Move assigndefaultdefaultdefault*default*not declareduser

    default* = generated but deprecated. The standard says generating copy operations when a destructor or the other copy operation is user-declared is deprecated. Compilers still do it for backward compatibility, but future standards may delete them. Do not rely on this behavior.

    Key insight: The critical pattern: declaring a destructor, copy constructor, or copy assignment operator suppresses implicit move operations. Your class silently loses move semantics. This is the most common accidental performance regression in C++.

    ▶ Compiler Decision Tree for Special Member Generation

    Step through to see how each user declaration affects what the compiler generates.

    Deleted vs Defaulted

    = default tells the compiler to generate the default implementation. = delete tells it to forbid the operation entirely.

    class Widget {
public:
    Widget() = default;                          // compiler-generated default ctor
    ~Widget() = default;                         // compiler-generated destructor
    Widget(const Widget&) = default;             // compiler-generated copy ctor
    Widget& operator=(const Widget&) = default;  // compiler-generated copy assign
    Widget(Widget&&) = default;                  // compiler-generated move ctor
    Widget& operator=(Widget&&) = default;       // compiler-generated move assign
};

    But = default counts as user-declared. This matters because declaring a move constructor (even defaulted) deletes the copy constructor:

    class MoveOnly {
public:
    MoveOnly() = default;
    MoveOnly(MoveOnly&&) = default;             // user-declared move ctor
    MoveOnly& operator=(MoveOnly&&) = default;  // user-declared move assign
    // Copy ctor: DELETED (because move ctor is user-declared)
    // Copy assign: DELETED (because move assign is user-declared)
};

MoveOnly a;
MoveOnly b(a);             // ERROR: copy constructor is deleted
MoveOnly c(std::move(a));  // OK: move constructor works

    Use = delete to explicitly forbid operations:

    class Singleton {
public:
    static Singleton& instance();
    Singleton(const Singleton&) = delete;            // no copies
    Singleton& operator=(const Singleton&) = delete; // no copies
private:
    Singleton() = default;
};

    The Copy-and-Swap Idiom

    The copy-and-swap idiom provides exception-safe, self-assignment-safe copy assignment with minimal code duplication. It leverages the copy constructor and a swap function:

    class Buffer {
    int* data_;
    size_t size_;

    friend void swap(Buffer& a, Buffer& b) noexcept {
        using std::swap;
        swap(a.data_, b.data_);
        swap(a.size_, b.size_);
    }

public:
    explicit Buffer(size_t n)
        : data_(new int[n]()), size_(n) {}

    ~Buffer() { delete[] data_; }

    Buffer(const Buffer& other)
        : data_(new int[other.size_]), size_(other.size_)
    {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_)
    {
        other.data_ = nullptr;
        other.size_ = 0;
    }

    // Unified assignment: handles both copy and move
    Buffer& operator=(Buffer other) {  // note: passed by VALUE
        swap(*this, other);
        return *this;
    }
    // When called with lvalue: other is copy-constructed, then swapped
    // When called with rvalue: other is move-constructed, then swapped
    // Old resources destroyed when other goes out of scope
};

    Advantages of copy-and-swap:

    • Exception safe: The copy happens before any modification. If the copy throws, *this is untouched.
    • Self-assignment safe: Works correctly even if you do buf = buf.
    • One assignment operator: Handles both copy and move assignment in a single function.
    • DRY: No code duplication between copy assignment and move assignment.

    The cost: one extra move for move-assignment (the parameter is move-constructed, then swapped). This is almost always negligible.

    Common Traps

    Trap 1: Empty Destructor Kills Move Semantics

    class Widget {
    std::string name_;
    std::vector<int> data_;
public:
    ~Widget() {}  // Empty destructor
    // Move constructor: NOT generated (destructor is user-declared)
    // Move assignment: NOT generated
    // All "moves" are actually copies!
};

// Fix: remove the destructor or use = default
// ~Widget() = default;  // still user-declared but generates moves
// Better: just remove it entirely (Rule of Zero)

    Wait, does = default still suppress moves? Let us be precise. A defaulted destructor is user-declared and technically should suppress moves according to the standard. However, in practice, most compilers generate moves when the destructor is = default because the standard says moves are "not declared" but does not say "deleted." The safest approach is to not declare a destructor at all.

    Trap 2: Base Class Destructor

    class Base {
public:
    virtual ~Base() = default;
    // This suppresses Move operations!
    // Derived classes also lose move semantics unless they
    // explicitly default their move operations.
};

class Derived : public Base {
    std::vector<int> data_;
    // Move operations: NOT generated
    // (Base has user-declared destructor, and Derived inherits from it)
};

// Fix:
class Derived : public Base {
    std::vector<int> data_;
public:
    Derived(Derived&&) = default;
    Derived& operator=(Derived&&) = default;
    Derived(const Derived&) = default;
    Derived& operator=(const Derived&) = default;
};

    Trap 3: Forgetting noexcept on Move Operations

    class Heavy {
    std::vector<int> data_;
public:
    // Move without noexcept
    Heavy(Heavy&& other) : data_(std::move(other.data_)) {}
    // std::vector<Heavy>::push_back will use COPY during reallocation
    // because it cannot guarantee exception safety with a throwing move.
};

// Fix: add noexcept
Heavy(Heavy&& other) noexcept : data_(std::move(other.data_)) {}

    Practical Guidelines

    • Prefer the Rule of Zero. Use RAII wrappers for all resources. If your class only has string, vector, unique_ptr, etc. as members, do not define any special members.
    • If you need a virtual destructor (polymorphic base class), explicitly default all five special members to preserve move semantics.
    • If you directly manage a resource (raw new, C API handle), follow the Rule of Five. But first, ask: can I wrap the resource in a unique_ptr with a custom deleter instead?
    • Mark move operations noexcept. Always.
    • Use = default when you want the compiler-generated implementation but need to declare the function (e.g., for access control or to make it noexcept).
    • Use = delete to explicitly forbid operations (non-copyable types, singletons).
    • The copy-and-swap idiom is the safest way to write assignment operators when you must manage resources directly.

    Rule of Zero with unique_ptr

    The most powerful application of the Rule of Zero is using std::unique_ptr with custom deleters to wrap C-style resources:

    // C API for a database connection
struct DbConn;
DbConn* db_open(const char* url);
void db_close(DbConn* conn);

// Rule of Zero wrapper
class Database {
    struct DbDeleter {
        void operator()(DbConn* p) const { db_close(p); }
    };
    std::unique_ptr<DbConn, DbDeleter> conn_;

public:
    explicit Database(const char* url)
        : conn_(db_open(url)) {}

    // No destructor, no copy ops, no move ops.
    // unique_ptr handles everything:
    // - Destructor calls db_close via DbDeleter
    // - Move works (unique_ptr is movable)
    // - Copy is deleted (unique_ptr is non-copyable)
};

    Detecting Missing Move Operations

    You can use static_assert and type traits to verify your types have move semantics:

    #include <type_traits>

class MyType { /* ... */ };

static_assert(std::is_move_constructible_v<MyType>,
    "MyType must be move constructible");

static_assert(std::is_nothrow_move_constructible_v<MyType>,
    "MyType move ctor must be noexcept");

static_assert(std::is_nothrow_move_assignable_v<MyType>,
    "MyType move assign must be noexcept");

    Clang-tidy also has checks for this: cppcoreguidelines-special-member-functions warns when you define some but not all of the Rule of Five members.

    Summary

    • Rule of Zero: Do not define any special members. Use RAII wrappers. This is the default you should aim for.
    • Rule of Three: If you manage a resource directly, define destructor + copy ctor + copy assignment.
    • Rule of Five: In C++11 and beyond, extend the Rule of Three with move ctor + move assignment.
    • Declaring a destructor, copy constructor, or copy assignment suppresses implicit move operations.
    • Use = default to explicitly request compiler generation. Use = delete to forbid operations.
    • The copy-and-swap idiom provides exception-safe assignment with a single function.
    • For polymorphic base classes with virtual destructors, explicitly default all five special members.
    • Use static_assert with type traits to verify your types have the move semantics you expect.
