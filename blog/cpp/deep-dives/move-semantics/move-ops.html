    Move Operations: Constructors, Assignment, and std::move

    March 2026 · 22 min read

    The Problem Moves Solve

    Before C++11, returning a large object from a function meant copying it. Inserting a temporary into a vector meant copying it. Every time you passed a value that was about to be destroyed anyway, the compiler dutifully copied every byte, then immediately destroyed the original. This was pure waste.

    // Pre-C++11: expensive copy, then destroy the temporary
std::vector<int> make_data() {
    std::vector<int> v(1000000, 42);
    return v;  // copy all 1,000,000 elements (unless compiler applies RVO)
}

std::vector<int> result = make_data();
// The temporary returned from make_data() is copied, then destroyed.

    Move semantics solve this by allowing the compiler to transfer an object's resources instead of copying them. For a std::vector, a move transfers the internal pointer, size, and capacity. The source is left empty. Cost: a few pointer assignments instead of a million element copies.

    The Move Constructor

    A move constructor builds a new object by stealing resources from an rvalue. Its signature takes an rvalue reference:

    class Buffer {
    int* data_;
    size_t size_;

public:
    // Move constructor
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_)
    {
        other.data_ = nullptr;
        other.size_ = 0;
    }
};

    What happens step by step:

    1. The new object's data_ pointer takes the source's pointer value.
    2. The new object's size_ takes the source's size.
    3. The source's pointer is set to nullptr and size to 0.
    4. When the source is eventually destroyed, its destructor sees nullptr and does nothing.

    Compare this to a copy constructor, which must allocate new memory and copy every element:

    // Copy constructor: O(n) time, O(n) memory
Buffer(const Buffer& other)
    : data_(new int[other.size_]), size_(other.size_)
{
    std::copy(other.data_, other.data_ + size_, data_);
}

// Move constructor: O(1) time, O(1) memory
Buffer(Buffer&& other) noexcept
    : data_(other.data_), size_(other.size_)
{
    other.data_ = nullptr;
    other.size_ = 0;
}

    ▶ Object State Transfer During Move

    Watch how resources transfer from source to destination during a move operation.

    The Move Assignment Operator

    Move assignment is analogous, but the destination already exists and may own resources that need to be freed first:

    Buffer& operator=(Buffer&& other) noexcept {
    if (this != &other) {
        delete[] data_;          // release existing resource
        data_ = other.data_;     // steal source's resource
        size_ = other.size_;
        other.data_ = nullptr;   // leave source in valid state
        other.size_ = 0;
    }
    return *this;
}

    The self-assignment check this != &other guards against x = std::move(x). While this is unusual, it can happen in generic code, and without the check you would delete your own data before reading it.

    Exception-Safe Move Assignment with Swap

    An alternative pattern that is both simpler and exception-safe uses swap:

    Buffer& operator=(Buffer&& other) noexcept {
    Buffer temp(std::move(other)); // move-construct temp from other
    std::swap(data_, temp.data_);  // swap our internals with temp
    std::swap(size_, temp.size_);
    return *this;
    // temp's destructor frees our old resources
}

    std::move: Just a Cast

    std::move does not move anything. It is a static_cast to an rvalue reference, nothing more. Here is a simplified implementation:

    // Simplified std::move (the real one handles more edge cases)
template <typename T>
constexpr std::remove_reference_t<T>&& move(T&& t) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(t);
}

    All it does is convert the expression from an lvalue to an xvalue. The actual resource transfer happens when a move constructor or move assignment operator consumes the resulting rvalue reference. Think of std::move as sticking a label on an object that says "you may cannibalize me."

    std::string a = "hello";
std::string b = std::move(a);  // Step 1: std::move(a) casts a to xvalue
                                 // Step 2: string's move ctor steals a's buffer
                                 // Step 3: a is now in a valid-but-unspecified state
    Key insight: If nobody consumes the rvalue reference that std::move produces, nothing happens. std::move by itself has zero side effects. The object is only modified when a move operation actually runs.

    The Moved-From State

    After a move, the source object is in a valid but unspecified state. The standard requires only two things:

    • The object can be destroyed safely.
    • The object can be assigned to (to give it a new value).

    You should not assume anything else about a moved-from object. For standard library types, the typical behavior is:

    TypeMoved-from state
    std::stringEmpty (but could be non-empty for short strings due to SSO)
    std::vectorEmpty (size 0)
    std::unique_ptrnullptr
    std::shared_ptrnullptr
    int, doubleUnchanged (trivial types are just copied)
     
    std::vector<int> v = {1, 2, 3, 4, 5};
std::vector<int> w = std::move(v);

// v is now in a valid-but-unspecified state
// Safe: v.size(), v.empty(), v.clear(), v = {10, 20}
// Unsafe: v[0] (may be undefined), v.front()
    Key insight: Design your own types so that the moved-from state is easy to reason about. The simplest approach: set all resource-managing members to their "empty" state (nullptr, 0, empty container). Then the destructor and assignment operator just work.

    When the Compiler Generates Move Operations

    The compiler will implicitly generate a move constructor and move assignment operator for a class if all of the following are true:

    • The class has no user-declared copy constructor.
    • The class has no user-declared copy assignment operator.
    • The class has no user-declared move constructor (for the move assignment case).
    • The class has no user-declared move assignment operator (for the move constructor case).
    • The class has no user-declared destructor.

    If any of these are user-declared, the compiler will not generate the move operations. The class falls back to copy semantics.

    struct Good {
    std::string name;
    std::vector<int> data;
    // No user-declared special members
    // Compiler generates: default ctor, copy ctor, copy assign,
    //                     move ctor, move assign, destructor
    // Move operations memberwise-move each member
};

struct Bad {
    std::string name;
    std::vector<int> data;
    ~Bad() { /* custom destructor */ }
    // Compiler generates: copy ctor, copy assign
    // Compiler does NOT generate: move ctor, move assign
    // Moving a Bad object silently falls back to copying!
};

    This is one of the most common performance traps in C++. Adding a destructor (even an empty one) silently suppresses move operations. We will cover this in detail in the Rule of 0/3/5 article.

    noexcept and Move Operations

    Move constructors should almost always be noexcept. This is not just a best practice; it has concrete performance consequences.

    The standard library containers (particularly std::vector) check whether the element type's move constructor is noexcept. If it is, the container uses move operations during reallocation. If it is not, the container falls back to copying, because a throwing move during reallocation would leave the container in an inconsistent state.

    class Widget {
public:
    // noexcept move: vector will use this during reallocation
    Widget(Widget&& other) noexcept
        : data_(std::move(other.data_)) {}

    // Without noexcept: vector falls back to copy during resize!
    // Widget(Widget&& other) : data_(std::move(other.data_)) {}
};

    The reasoning: when std::vector needs to grow, it allocates new storage and transfers elements from old to new. If a move throws halfway through, some elements have been moved and some have not. The old storage is partially gutted. The vector cannot restore the original state. By using copy (which leaves the source intact), a throw just means the new allocation is discarded and the original vector is untouched.

    Key insight: Always mark your move constructors and move assignment operators noexcept if they truly cannot throw. This is the single most impactful thing you can do for move-related performance. Use static_assert(std::is_nothrow_move_constructible_v<YourType>) to verify.

    ▶ std::vector Reallocation: Move vs Copy

    Compare vector reallocation with and without noexcept move constructor.

    Inside std::move

    Let us trace through the actual implementation step by step:

    template <typename T>
constexpr std::remove_reference_t<T>&& move(T&& t) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(t);
}

    When called with an lvalue std::string s; std::move(s);:

    1. T is deduced as std::string& (reference collapsing: lvalue passed to T&& deduces T as string&).
    2. remove_reference_t<std::string&> yields std::string.
    3. The return type is std::string&&.
    4. The static_cast converts the lvalue to an xvalue.

    When called with an rvalue std::move(std::string("hi"));:

    1. T is deduced as std::string (rvalue passed to T&& deduces T as string).
    2. remove_reference_t<std::string> yields std::string.
    3. The return type is std::string&&.
    4. The static_cast is a no-op since it is already an rvalue.

    Implicit Memberwise Move

    When the compiler generates a move constructor, it performs a memberwise move: each member is move-constructed from the corresponding member of the source object. For types that do not have move constructors (like int), "move" is just a copy.

    struct Data {
    std::string name;           // has move ctor: actually moved
    std::vector<int> values;    // has move ctor: actually moved
    int count;                  // no move ctor: copied (same as move for trivial types)
    double* raw_ptr;            // no move ctor: copied (pointer value copied!)
};

// Compiler-generated move constructor behaves as:
// Data(Data&& other) noexcept
//     : name(std::move(other.name)),
//       values(std::move(other.values)),
//       count(other.count),             // just a copy
//       raw_ptr(other.raw_ptr)          // just a copy! both point to same memory!
// {}
    Key insight: The implicit move constructor does NOT null out raw pointers or reset primitive types. If your class manages a raw pointer, the implicit move constructor will leave two objects pointing to the same memory. This is why classes managing raw resources need user-defined move operations that explicitly null out the source.

    Moves and const

    You cannot move from a const object. std::move on a const object produces a const T&&, which does not bind to a move constructor (which takes T&&). Instead, the copy constructor (which takes const T&) is selected as a fallback:

    const std::string s = "hello";
std::string t = std::move(s);
// std::move(s) yields const std::string&&
// Move ctor: string(string&&)       — does NOT match (const vs non-const)
// Copy ctor: string(const string&)  — matches!
// Result: t is a COPY of s. No move occurred. No compiler warning.

    This is a silent performance regression. Be careful about std::move on const variables or const-qualified function parameters.

    Common Move Idioms

    Sink Parameters

    When a function needs to store its own copy of an argument, take the parameter by value and move from it:

    class Person {
    std::string name_;
public:
    // "Sink" parameter: caller chooses copy or move
    explicit Person(std::string name)
        : name_(std::move(name))  // always move from the local copy
    {}
};

Person p1("Alice");              // string constructed in-place, then moved
std::string n = "Bob";
Person p2(n);                    // n is copied into parameter, then moved
Person p3(std::move(n));         // n is moved into parameter, then moved

    This gives you one overload instead of two (const string& and string&&), at the cost of one extra move (which is typically negligible).

    Moving into Containers

    std::vector<std::string> names;

std::string s = "Alice";
names.push_back(s);              // copy: s is an lvalue
names.push_back(std::move(s));   // move: s is cast to xvalue
names.push_back("Bob");          // move: string literal creates a temporary (prvalue)
names.emplace_back("Charlie");   // construct in-place: even better, no temporary at all

    Swap with Moves

    The standard std::swap is implemented using moves (since C++11):

    template <typename T>
void swap(T& a, T& b) noexcept(/* ... */) {
    T temp = std::move(a);   // move a into temp
    a = std::move(b);        // move b into a
    b = std::move(temp);     // move temp into b
}
// Three moves instead of three copies.
// For a vector<int> with 1M elements: 9 pointer assigns vs 3M copies.

    Move-Only Types

    Some types are move-only: they can be moved but not copied. The canonical examples are std::unique_ptr, std::thread, std::future, and std::ofstream.

    class UniqueSocket {
    int fd_;
public:
    explicit UniqueSocket(int fd) : fd_(fd) {}
    ~UniqueSocket() { if (fd_ >= 0) ::close(fd_); }

    // Movable
    UniqueSocket(UniqueSocket&& other) noexcept
        : fd_(std::exchange(other.fd_, -1)) {}

    UniqueSocket& operator=(UniqueSocket&& other) noexcept {
        if (this != &other) {
            if (fd_ >= 0) ::close(fd_);
            fd_ = std::exchange(other.fd_, -1);
        }
        return *this;
    }

    // Non-copyable
    UniqueSocket(const UniqueSocket&) = delete;
    UniqueSocket& operator=(const UniqueSocket&) = delete;
};

    Note the use of std::exchange: it returns the old value and sets the variable to a new value in one expression. It is the perfect tool for move constructors.

    When Moves Are Not Free

    Moves are not always O(1). Some types have no resources to steal:

    • std::array<int, 1000000>: the data is stored inline. "Moving" an array copies every element.
    • std::string with SSO: short strings (typically 15 characters or fewer) are stored inline in the string object. Moving a short string is the same as copying it.
    • Types with only value members and no heap allocations: move degenerates to copy.
    // Move is O(N) for std::array!
std::array<int, 1000000> a = {};
std::array<int, 1000000> b = std::move(a);  // copies 1M ints
// Both a and b contain the same values afterward.

    Summary

    • A move constructor builds a new object by stealing resources from an rvalue. It should be O(1) for types managing heap-allocated resources.
    • Move assignment is similar but must first release the destination's existing resources.
    • std::move is a cast, not a move. It converts an lvalue to an xvalue so that move overloads can be selected.
    • After a move, the source is in a valid but unspecified state. Design your types so the moved-from state is clean and predictable.
    • The compiler generates move operations only when you declare no copy operations, move operations, or destructor.
    • Mark move operations noexcept. Without it, std::vector falls back to copying during reallocation.
    • Moving from a const object silently falls back to copying.
    • Use std::exchange in move constructors for clean, one-line resource transfer.
