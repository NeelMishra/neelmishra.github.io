    Value Categories: lvalues, rvalues, and Beyond

    March 2026 · 20 min read

    Why Value Categories Matter

    Every expression in C++ has two independent properties: a type and a value category. The type tells you what the expression yields (an int, a std::string, a Widget&&). The value category tells you something more subtle: what can you do with this expression? Can you take its address? Can you bind a reference to it? Can you steal its guts?

    Understanding value categories is the foundation for understanding move semantics, perfect forwarding, and overload resolution. Without it, rules about when things move versus copy seem arbitrary. With it, the entire system clicks into place.

    The Original Distinction: lvalues and rvalues

    The names come from the original C definition: an lvalue (left-value) could appear on the left side of an assignment, and an rvalue (right-value) could only appear on the right. In modern C++, this mnemonic is misleading (a const lvalue cannot be assigned to), but the names stuck.

    A better way to think about it:

    • lvalue: an expression that refers to a persistent object. It has an identity (you can take its address). It lives beyond the current expression.
    • rvalue: an expression that refers to a temporary value. It does not have a stable address. It is about to be destroyed.
    int x = 42;          // x is an lvalue (has address, persists)
int y = x + 1;       // x+1 is an rvalue (temporary, no address)

std::string s = "hi";        // s is an lvalue
std::string t = s + " there"; // s + " there" is an rvalue

    This simple two-way split was sufficient in C and early C++. But C++11 introduced move semantics, which required a more nuanced taxonomy. The question was no longer just "persistent or temporary?" but also "can we safely cannibalize this object's resources?"

    The Modern Taxonomy (C++11 and Beyond)

    The C++11 standard refined the classification into five categories, organized along two orthogonal axes:

    • Has identity: Can you distinguish this object from other objects? (Can you take its address, compare pointers to it, etc.)
    • Can be moved from: Is it safe to steal this object's resources?

    The five categories are:

    CategoryHas identity?Movable?Examples
    lvalueYesNox, *p, arr[0]
    xvalueYesYesstd::move(x), static_cast<T&&>(x)
    prvalueNoYes42, x + y, std::string("hi")
    glvalueYes—lvalue or xvalue
    rvalue—Yesxvalue or prvalue

    The last two (glvalue and rvalue) are umbrella categories: glvalue = "generalized lvalue" = any expression with identity. rvalue = any expression that can be moved from.

    lvalues in Detail

    An lvalue designates an object with a persistent storage location. You can take its address with &. The name "lvalue" means "locator value," something that identifies a place in memory.

    Examples of lvalue expressions:

    int x = 10;
int& ref = x;

x;              // lvalue: the variable itself
ref;            // lvalue: a reference is always an lvalue expression
*ptr;           // lvalue: dereferencing a pointer
arr[2];         // lvalue: subscript on an array or container
"hello";        // lvalue: a string literal has static storage duration
++x;            // lvalue: prefix increment returns the object itself
obj.member;     // lvalue: member access on an lvalue object
(*this);        // lvalue: this dereferenced
    Key insight: A named rvalue reference variable is itself an lvalue. This is one of the most confusing points in C++. The variable Widget&& rr = makeWidget(); declares rr as an rvalue reference, but the expression rr is an lvalue because it has a name and a persistent location. You need std::move(rr) to treat it as an rvalue again.

    prvalues: Pure rvalues

    A prvalue (pure rvalue) is an expression that initializes an object or computes the value of a built-in operator's operand. prvalues do not have identity: you cannot take their address, and they do not necessarily correspond to any object in memory until they are used.

    42;                      // prvalue: integer literal
3.14;                    // prvalue: floating-point literal
true;                    // prvalue: boolean literal
nullptr;                 // prvalue: null pointer literal
x + y;                   // prvalue: result of arithmetic
x < y;                   // prvalue: result of comparison
Widget();                // prvalue: temporary object construction
makeWidget();            // prvalue: function returning by value
static_cast<int>(3.7);  // prvalue: cast to non-reference type
[](int a){ return a; };  // prvalue: lambda expression
x++;                     // prvalue: postfix increment returns old value

    Since C++17, prvalues are not necessarily materialized into objects until they need to be. This is the foundation of guaranteed copy elision, which we will cover in the RVO article later in this series.

    xvalues: Expiring Values

    An xvalue (expiring value) has identity (it refers to a real object) but is explicitly marked as safe to move from. It is the bridge between lvalues and prvalues. The "x" stands for "eXpiring."

    std::move(x);                       // xvalue: explicitly cast to rvalue
static_cast<Widget&&>(w);           // xvalue: same as std::move
std::move(vec).at(0);               // xvalue: member access on an rvalue
getWidget().member;                 // xvalue: member of a temporary
a[3]                                // xvalue if a is an rvalue array
static_cast<int(&&)[4]>(arr)[2];  // xvalue

    xvalues are the key to move semantics. When a function receives an xvalue, it knows the caller is done with the object and its resources can be stolen.

    Key insight: std::move does not move anything. It is a cast from lvalue to xvalue. The actual move happens when a move constructor or move assignment operator receives the resulting rvalue reference. std::move is a promise: "I am done with this object; feel free to cannibalize it."

    glvalues and rvalues: The Umbrella Categories

    The standard groups the five categories into two overlapping umbrellas:

    • glvalue (generalized lvalue) = lvalue or xvalue. Any expression with identity. You can take its address (for lvalues) or it refers to a real object (for xvalues). glvalues can be polymorphic: the dynamic type may differ from the static type.
    • rvalue = xvalue or prvalue. Any expression that can be moved from. An rvalue reference (T&&) binds to rvalues.

    Notice that xvalue sits at the intersection: it has identity and can be moved from. This is exactly what std::move produces: an expression with identity (it still refers to the original object) but flagged as movable.

    Value Categories and Overload Resolution

    The primary reason value categories exist in the standard is to drive overload resolution. The compiler uses value categories to decide which function overload to call:

    void process(Widget& w);        // (1) lvalue reference overload
void process(const Widget& w);  // (2) const lvalue reference overload
void process(Widget&& w);       // (3) rvalue reference overload

Widget w;
const Widget cw;

process(w);              // calls (1): w is a non-const lvalue
process(cw);             // calls (2): cw is a const lvalue
process(Widget());       // calls (3): Widget() is a prvalue (rvalue)
process(std::move(w));   // calls (3): std::move(w) is an xvalue (rvalue)

    The binding rules are:

    • T& binds to non-const lvalues only.
    • const T& binds to everything: lvalues, const lvalues, rvalues (the "universal bind").
    • T&& binds to rvalues only (xvalues and prvalues).

    When both const T& and T&& are viable, the compiler prefers T&& for rvalues. This is how the compiler routes temporaries and moved-from objects to move constructors instead of copy constructors.

    class Widget {
public:
    Widget(const Widget& other);   // copy constructor
    Widget(Widget&& other);        // move constructor
};

Widget a;
Widget b(a);              // copy: a is an lvalue
Widget c(Widget());       // move: Widget() is a prvalue
Widget d(std::move(a));   // move: std::move(a) is an xvalue

    Reference Binding Rules

    Understanding which references bind to which value categories is essential. Here is the complete table:

    Expression categoryT&const T&T&&const T&&
    lvalueYesYesNoNo
    const lvalueNoYesNoNo
    xvalueNoYesYesYes
    prvalueNoYesYesYes

    The key takeaway: const T& is the most permissive. T&& is selective, it only accepts rvalues (xvalues and prvalues), which is exactly what makes it useful for detecting "I can steal from this."

    decltype and Value Categories

    decltype is unique among C++ features because it can tell you the value category of an expression, not just its type. It has two different behaviors depending on its argument:

    decltype on a Name (Unparenthesized)

    When decltype is applied to an unparenthesized name (a variable, function parameter, etc.), it yields the declared type of that entity:

    int x = 42;
decltype(x)     // int (the declared type of x)

int& ref = x;
decltype(ref)   // int& (the declared type of ref)

int&& rr = std::move(x);
decltype(rr)    // int&& (the declared type of rr)

    decltype on an Expression (Parenthesized or Complex)

    When applied to any other expression (including a parenthesized name), decltype encodes the value category:

    • If the expression is an lvalue, decltype yields T&.
    • If the expression is an xvalue, decltype yields T&&.
    • If the expression is a prvalue, decltype yields T.
    int x = 42;
decltype((x))            // int& (parenthesized name is lvalue expression)
decltype((std::move(x))) // int&& (std::move produces xvalue)
decltype((42))           // int (42 is a prvalue)
decltype((x + 1))        // int (arithmetic produces prvalue)
    Key insight: The double parentheses in decltype((x)) make a huge difference. decltype(x) gives you the declared type (int). decltype((x)) treats x as an expression, and since x is an lvalue expression, it gives you int&. This is a common source of accidental dangling references in decltype(auto) return types. 
    // Dangerous! decltype(auto) with parenthesized return
decltype(auto) bad() {
    int x = 42;
    return (x);   // decltype((x)) = int& — returns dangling reference!
}

// Safe:
decltype(auto) good() {
    int x = 42;
    return x;      // decltype(x) = int — returns by value
}

    Practical Applications

    Detecting Value Categories at Compile Time

    You can write a utility to check value categories using decltype and type traits:

    #include <type_traits>

template <typename T>
struct value_category {
    static constexpr const char* value = "prvalue";
};
template <typename T>
struct value_category<T&> {
    static constexpr const char* value = "lvalue";
};
template <typename T>
struct value_category<T&&> {
    static constexpr const char* value = "xvalue";
};

#define VALUE_CAT(expr) value_category<decltype((expr))>::value

int main() {
    int x = 10;
    std::cout << VALUE_CAT(x)             << '\n'; // lvalue
    std::cout << VALUE_CAT(std::move(x))  << '\n'; // xvalue
    std::cout << VALUE_CAT(42)            << '\n'; // prvalue
    std::cout << VALUE_CAT(x + 1)         << '\n'; // prvalue
}

    Value Categories in Template Code

    Value categories determine how template argument deduction works with forwarding references. This is fundamental to perfect forwarding (covered in the next articles), but the basic idea is:

    template <typename T>
void wrapper(T&& arg) {
    // If called with lvalue:  T = Widget&, arg type = Widget&
    // If called with rvalue:  T = Widget,  arg type = Widget&&
}

Widget w;
wrapper(w);             // T deduced as Widget&, reference collapsing gives Widget&
wrapper(Widget());      // T deduced as Widget, arg is Widget&&
wrapper(std::move(w));  // T deduced as Widget, arg is Widget&&

    Common Mistakes and Misconceptions

    Mistake 1: Thinking rvalue references are rvalues

    A variable of type T&& is an lvalue. The type of the variable is "rvalue reference to T," but the expression naming that variable is an lvalue because it has a name and a persistent location:

    void process(Widget&& w) {
    // w is an lvalue here! It has a name.
    // To treat it as an rvalue, you must use std::move(w).
    sink(w);              // calls sink(Widget&) or sink(const Widget&)
    sink(std::move(w));   // calls sink(Widget&&)
}

    Mistake 2: Thinking std::move moves

    std::move is purely a cast. It converts its argument to an xvalue. No data is transferred, no constructor is called. The actual move happens when something consumes the rvalue reference.

    Mistake 3: Moving from const objects

    const std::vector<int> cv = {1, 2, 3};
auto v2 = std::move(cv);  // compiles, but COPIES!
// std::move(cv) produces const vector<int>&&
// The move constructor takes vector<int>&& (non-const)
// The copy constructor takes const vector<int>& and is a better match

    Moving from a const object silently falls back to copying. There is no compiler warning. This is a subtle performance trap.

    Mistake 4: Confusing type and value category

    The type of an expression and its value category are independent:

    int&& rr = 42;
// Type of rr: int&& (rvalue reference)
// Value category of the expression "rr": lvalue
// Note: rr binds to the temporary 42, extending its lifetime.
// But the name rr is itself an lvalue.

    Temporary Materialization (C++17)

    C++17 introduced a crucial refinement: prvalues are no longer objects. A prvalue is a recipe for initializing an object. The object is only created (materialized) when one is actually needed. This process is called temporary materialization conversion.

    Materialization occurs when:

    • A prvalue is bound to a reference: const Widget& w = Widget();
    • A member access is performed on a prvalue: Widget().getData() (the prvalue materializes into an xvalue)
    • Array-to-pointer decay is applied to a prvalue array
    • A prvalue appears in a typeid or sizeof expression that evaluates its operand
    • A prvalue is discarded as a discarded-value expression
    struct Widget {
    int data;
    Widget() : data(42) {}
};

const Widget& ref = Widget();  // materialization: prvalue becomes xvalue,
                                 // binds to ref, lifetime extended

Widget().data;                   // materialization: prvalue becomes xvalue
                                 // to access the member

Widget w = Widget();             // NO materialization in C++17!
                                 // prvalue directly initializes w
                                 // (guaranteed copy elision)

    This is why in C++17 the expression Widget w = Widget(); is guaranteed to construct w directly with no copy or move, even if the copy and move constructors are deleted. The prvalue never becomes an object until it is used to initialize w.

    Summary

    • Every expression has a type and a value category. They are independent properties.
    • There are three primary categories: lvalue (has identity, not movable), xvalue (has identity, movable), prvalue (no identity, movable).
    • Two umbrella categories: glvalue = lvalue + xvalue (has identity), rvalue = xvalue + prvalue (movable).
    • Value categories drive overload resolution: T& binds lvalues, T&& binds rvalues, const T& binds anything.
    • A named rvalue reference is an lvalue. std::move turns it back into an xvalue.
    • decltype on a parenthesized expression encodes value category: lvalue yields T&, xvalue yields T&&, prvalue yields T.
    • C++17 deferred materialization means prvalues are recipes, not objects, enabling guaranteed copy elision.
