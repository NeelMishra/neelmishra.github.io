    Perfect Forwarding: Forwarding References and std::forward

    March 2026 · 22 min read

    The Forwarding Problem

    Suppose you are writing a generic wrapper function that takes some arguments and passes them to another function. You want the wrapper to be transparent: whatever the caller passed (lvalue, rvalue, const, non-const), the inner function should see exactly the same thing.

    // Goal: wrapper should forward args to target transparently
template <typename T>
void wrapper(???) {
    target(???);
}

Widget w;
wrapper(w);             // target should see an lvalue
wrapper(Widget());      // target should see an rvalue
wrapper(std::move(w));  // target should see an rvalue

    Before C++11, this was impossible without writing 2N overloads (one for each combination of lvalue/rvalue per parameter). C++11 solved this with forwarding references and std::forward.

    Forwarding References (Universal References)

    A forwarding reference is a special kind of reference that can bind to both lvalues and rvalues. It appears as T&& in a template where T is a deduced template parameter:

    template <typename T>
void wrapper(T&& arg);   // T&& is a forwarding reference (T is deduced)

auto&& x = expr;          // auto&& is also a forwarding reference
    Key insight: T&& is a forwarding reference only when T is a template parameter being deduced in that call. If the type is already known (not being deduced), it is just a plain rvalue reference. void foo(Widget&& w) is NOT a forwarding reference; it is an rvalue reference that only accepts rvalues. 
    template <typename T>
void foo(T&& arg);            // forwarding reference (T deduced)

template <typename T>
void bar(std::vector<T>&& v); // NOT forwarding ref! vector<T> is known, only T deduced

void baz(Widget&& w);         // NOT forwarding ref! Widget is a specific type

template <typename T>
class MyClass {
    void method(T&& arg);     // NOT forwarding ref! T is fixed for the class instance
};

    Reference Collapsing Rules

    C++ does not allow "reference to reference" in your source code. But template instantiation can produce them, and the compiler resolves them through reference collapsing:

    Template producesCollapses toRule
    T& &T&lvalue ref to lvalue ref = lvalue ref
    T& &&T&rvalue ref to lvalue ref = lvalue ref
    T&& &T&lvalue ref to rvalue ref = lvalue ref
    T&& &&T&&rvalue ref to rvalue ref = rvalue ref

    The rule is simple: any lvalue reference in the chain wins. The only way to get an rvalue reference is if both references are rvalue references.

    Template Argument Deduction with Forwarding References

    When a forwarding reference T&& receives an argument, the deduction rules are special:

    • If the argument is an lvalue of type Widget, then T is deduced as Widget&. After reference collapsing (Widget& && = Widget&), the parameter type is Widget&.
    • If the argument is an rvalue of type Widget, then T is deduced as Widget. The parameter type is Widget&&.
    template <typename T>
void wrapper(T&& arg);

Widget w;
wrapper(w);             // T = Widget&,  arg type = Widget& && = Widget&
wrapper(Widget());      // T = Widget,   arg type = Widget&&
wrapper(std::move(w));  // T = Widget,   arg type = Widget&&

const Widget cw;
wrapper(cw);            // T = const Widget&, arg type = const Widget&

    This is the magic: the forwarding reference deduces differently for lvalues and rvalues, encoding the value category in the template parameter T. If T includes a reference, the caller passed an lvalue. If not, the caller passed an rvalue.

    std::forward Explained

    std::forward<T>(arg) is a conditional cast. It casts arg back to the value category it had when it was passed to the outer function:

    • If the original argument was an lvalue (so T is Widget&): std::forward<Widget&>(arg) returns an lvalue reference. No move.
    • If the original argument was an rvalue (so T is Widget): std::forward<Widget>(arg) returns an rvalue reference. Move enabled.
    template <typename T>
void wrapper(T&& arg) {
    target(std::forward<T>(arg));
    // If arg was originally an lvalue:  forwards as lvalue
    // If arg was originally an rvalue:  forwards as rvalue
}

    Here is a simplified implementation of std::forward:

    // For lvalues: T = Widget&
template <typename T>
constexpr T&& forward(std::remove_reference_t<T>& t) noexcept {
    return static_cast<T&&>(t);
    // T = Widget&: static_cast<Widget& &&>(t) = static_cast<Widget&>(t)
    // Returns lvalue reference. No move.
}

// For rvalues: T = Widget
template <typename T>
constexpr T&& forward(std::remove_reference_t<T>&& t) noexcept {
    return static_cast<T&&>(t);
    // T = Widget: static_cast<Widget&&>(t)
    // Returns rvalue reference. Move enabled.
}
    Key insight: std::move unconditionally casts to rvalue. std::forward conditionally casts based on the deduced type T. Use std::move when you know you want to move. Use std::forward when you want to preserve whatever value category was passed in.

    The Perfect Forwarding Pattern

    The standard pattern combines forwarding references with std::forward:

    template <typename... Args>
void wrapper(Args&&... args) {
    target(std::forward<Args>(args)...);
}

    This works for any number of arguments, preserving the value category of each one independently. It is the foundation of library functions like std::make_unique, std::make_shared, emplace_back, and many more.

    Practical Use: Factory Functions

    The most common use of perfect forwarding is in factory functions that construct objects on behalf of the caller:

    template <typename T, typename... Args>
std::unique_ptr<T> make_unique(Args&&... args) {
    return std::unique_ptr<T>(new T(std::forward<Args>(args)...));
}

// Usage:
auto p = make_unique<Widget>("hello", 42);
// "hello" is a const char* lvalue, forwarded as lvalue
// 42 is a prvalue, forwarded as rvalue
// Widget constructor sees: Widget(const char*, int) or Widget(const char*, int&&)

    Without perfect forwarding, you would need overloads for every combination of lvalue/rvalue for each parameter. With two parameters, that is four overloads. With three, eight. Perfect forwarding gives you one template that handles all combinations.

    emplace_back Internals

    emplace_back is the canonical example of perfect forwarding in the standard library. Instead of constructing an object and then copying or moving it into the container, emplace_back constructs the object directly in the container's storage:

    // Simplified emplace_back
template <typename T, typename Alloc>
template <typename... Args>
void vector<T, Alloc>::emplace_back(Args&&... args) {
    if (size_ == capacity_) grow();
    // Construct T directly in uninitialized storage
    ::new (static_cast<void*>(data_ + size_)) T(std::forward<Args>(args)...);
    ++size_;
}

    Compare push_back and emplace_back:

    std::vector<std::pair<std::string, int>> v;

// push_back: construct a pair, then move it into the vector
v.push_back(std::make_pair("Alice", 30));
// 1. Construct temporary pair
// 2. Move the pair into vector storage

// emplace_back: construct the pair directly in vector storage
v.emplace_back("Alice", 30);
// 1. Construct pair in-place using forwarded args
// No temporary, no move

    For types that are expensive to move (or cannot be moved), emplace_back can be significantly more efficient. For types like std::string where moves are cheap, the difference is often negligible.

    Common Perfect Forwarding Mistakes

    Mistake 1: Forwarding the Same Argument Twice

    template <typename T>
void bad(T&& arg) {
    sink1(std::forward<T>(arg));  // may move from arg
    sink2(std::forward<T>(arg));  // DANGER: arg may already be moved-from!
}

    If arg was an rvalue, the first std::forward allows it to be moved. The second forward operates on a potentially empty, moved-from object. Only forward an argument at most once per code path.

    Mistake 2: Using std::move Instead of std::forward

    template <typename T>
void bad(T&& arg) {
    target(std::move(arg));    // WRONG: always moves, even if caller passed lvalue
}

template <typename T>
void good(T&& arg) {
    target(std::forward<T>(arg)); // CORRECT: preserves original value category
}

    Mistake 3: Forgetting the Template Parameter on std::forward

    template <typename T>
void bad(T&& arg) {
    target(std::forward(arg));      // ERROR: cannot deduce T for std::forward
    target(std::forward<T>(arg));  // OK
}

    std::forward is deliberately designed so that T cannot be deduced. You must provide it explicitly. This forces you to think about what T is, which prevents subtle bugs.

    Mistake 4: Forwarding References in Non-Deduced Contexts

    template <typename T>
class Container {
    // This is NOT a forwarding reference! T is already known.
    void insert(T&& value);

    // This IS a forwarding reference: U is deduced here.
    template <typename U>
    void insert_generic(U&& value);
};

    Forwarding References and Overload Resolution

    Forwarding references are greedy in overload resolution because they can bind to anything. This can cause unexpected behavior when combined with other overloads:

    class Person {
    std::string name_;
public:
    // Forwarding reference constructor
    template <typename T>
    explicit Person(T&& n) : name_(std::forward<T>(n)) {}

    // Copy constructor
    Person(const Person& other) : name_(other.name_) {}
};

Person p("Alice");
Person p2(p);   // ERROR or unexpected: calls the template, not copy ctor!
// T&& with T = Person& is a better match than const Person&

    The template wins because Person& is a better match than const Person& for a non-const lvalue. Solutions include:

    • Use SFINAE or concepts to constrain the template so it does not match Person arguments.
    • Use std::enable_if or C++20 concepts to restrict the template.
    // C++20 solution with concepts
template <typename T>
    requires (!std::same_as<std::remove_cvref_t<T>, Person>)
explicit Person(T&& n) : name_(std::forward<T>(n)) {}

// C++17 solution with enable_if
template <typename T,
          typename = std::enable_if_t<
              !std::is_same_v<std::decay_t<T>, Person>>>
explicit Person(T&& n) : name_(std::forward<T>(n)) {}

    auto&&: Forwarding in Non-Template Contexts

    auto&& follows the same deduction and collapsing rules as T&& in a template. It is a forwarding reference that works in range-based for loops and lambdas:

    // Range-based for with forwarding reference
for (auto&& elem : container) {
    // elem binds to each element, preserving value category
    // If container yields lvalues: elem is an lvalue reference
    // If container yields rvalues: elem is an rvalue reference
}

// Lambda with forwarding
auto wrapper = [](auto&&... args) {
    return target(std::forward<decltype(args)>(args)...);
};

// C++20 abbreviated function template
void process(auto&& arg) {
    target(std::forward<decltype(arg)>(arg));
}

    Forwarding Return Values

    When writing a transparent wrapper, you may also need to forward the return value. decltype(auto) preserves the exact return type including references:

    template <typename F, typename... Args>
decltype(auto) transparent_call(F&& f, Args&&... args) {
    return std::forward<F>(f)(std::forward<Args>(args)...);
}

// If f returns int&   : transparent_call returns int&
// If f returns int     : transparent_call returns int
// If f returns int&&  : transparent_call returns int&&

    Real-World Examples

    Thread Construction

    // std::thread uses perfect forwarding to pass args to the callable
template <typename F, typename... Args>
explicit thread(F&& f, Args&&... args);

// Usage:
void worker(std::string name, int id);
std::string s = "worker-1";
std::thread t(worker, std::move(s), 42);
// s is moved into thread's internal storage
// 42 is copied (trivial type)

    Optional Construction

    template <typename T>
class optional {
    // ...
    template <typename... Args>
    T& emplace(Args&&... args) {
        reset();
        ::new (storage()) T(std::forward<Args>(args)...);
        has_value_ = true;
        return *storage();
    }
};

    Summary

    • Forwarding references (T&& where T is deduced) bind to both lvalues and rvalues, encoding the value category in the type of T.
    • Reference collapsing resolves compound references: any lvalue reference in the chain produces an lvalue reference. Only && && produces &&.
    • std::forward<T> is a conditional cast: it restores the original value category based on what T was deduced as.
    • std::move = unconditional rvalue cast. std::forward = conditional cast preserving value category.
    • Forward an argument at most once per code path.
    • Forwarding references are greedy in overload resolution. Constrain them with concepts or SFINAE when mixing with other overloads.
    • auto&& is a forwarding reference and works in lambdas and range-for loops.
    • emplace_back, make_unique, std::thread, and many other standard library facilities rely on perfect forwarding.
