    Return Value Optimization and Copy Elision

    March 2026 · 18 min read

    What is Copy Elision?

    Copy elision is a compiler optimization where the compiler omits the copy or move construction of an object entirely. Instead of constructing a temporary and then copying or moving it to the final destination, the compiler constructs the object directly in its final location. No copy constructor, no move constructor, no destructor for the temporary. The intermediate object simply never exists.

    This is not just a performance optimization; it changes the observable behavior of the program. Even if your copy or move constructor has side effects (like printing a message), the compiler is allowed to (and in some cases, required to) skip them entirely.

    class Trace {
public:
    Trace() { std::cout << "constructed\n"; }
    Trace(const Trace&) { std::cout << "copied\n"; }
    Trace(Trace&&) { std::cout << "moved\n"; }
};

Trace make() {
    return Trace();  // constructs directly in caller's storage
}

Trace t = make();
// Output (with RVO): "constructed"
// Without RVO:       "constructed" "moved" "moved" (or similar)

    Return Value Optimization (RVO)

    RVO applies when a function returns a prvalue (a temporary that was never bound to a name):

    Widget makeWidget() {
    return Widget(42);  // RVO: prvalue returned
    // The Widget is constructed directly in the caller's storage.
    // No temporary Widget exists in makeWidget's stack frame.
}

    Since C++17, this is guaranteed. The standard mandates that prvalue returns are constructed directly in the caller's storage. The compiler must not generate a copy or move, even if it could. This means RVO works even when the copy and move constructors are deleted:

    class NoCopyNoMove {
public:
    NoCopyNoMove(int x) : val_(x) {}
    NoCopyNoMove(const NoCopyNoMove&) = delete;
    NoCopyNoMove(NoCopyNoMove&&) = delete;
private:
    int val_;
};

NoCopyNoMove make() {
    return NoCopyNoMove(42);  // OK in C++17! Guaranteed copy elision.
}

NoCopyNoMove obj = make();  // Compiles and works in C++17+

    Named Return Value Optimization (NRVO)

    NRVO applies when a function returns a named local variable:

    Widget makeWidget() {
    Widget w(42);       // named local variable
    w.configure();
    return w;           // NRVO: may construct w directly in caller's storage
}

    NRVO is not guaranteed by the standard, even in C++17. The compiler is permitted but not required to apply it. In practice, every major compiler (GCC, Clang, MSVC) applies NRVO reliably in simple cases. However, NRVO can fail in certain situations.

    When NRVO Applies

    • The function returns a single named local variable of the same type as the return type.
    • All return paths return the same variable.
    • The variable is not a function parameter.
    // NRVO likely applied:
std::string make_greeting(const std::string& name) {
    std::string result = "Hello, " + name + "!";
    return result;  // single named variable, same type as return
}

// NRVO likely applied:
std::vector<int> make_data(int n) {
    std::vector<int> v;
    v.reserve(n);
    for (int i = 0; i < n; ++i)
        v.push_back(i * i);
    return v;
}

    When NRVO Fails

    // NRVO CANNOT apply: multiple return paths with different variables
Widget make(bool flag) {
    Widget a(1);
    Widget b(2);
    if (flag)
        return a;   // which one gets constructed in caller's storage?
    else
        return b;   // compiler doesn't know at construction time
    // Falls back to move (or copy if no move ctor)
}

// NRVO CANNOT apply: returning a function parameter
Widget passthrough(Widget w) {
    return w;  // w is a parameter, not a local. No NRVO.
    // But: implicit move applies (see later section)
}

// NRVO CANNOT apply: returning a member
Widget getWidget() {
    return this->widget_;  // not a local variable
}

// NRVO CANNOT apply: different type
std::optional<Widget> maybe_make() {
    Widget w(42);
    return w;  // return type is optional<Widget>, not Widget
    // Implicit conversion, no elision
}

    Guaranteed Copy Elision (C++17)

    C++17 introduced guaranteed copy elision, also called mandatory elision. It applies in two specific contexts:

    1. Returning a prvalue

    T f() {
    return T(args);   // guaranteed: T constructed directly in caller
}
T x = f();            // guaranteed: f's return directly initializes x

    2. Initializing from a prvalue

    T x = T(T(T(42)));   // guaranteed: only ONE construction happens
// Each T() is a prvalue, and prvalues don't create objects
// until they're used to initialize something.
// Result: x is constructed with T(42) directly.

    The key conceptual shift in C++17: prvalues are not objects. They are instructions for how to initialize an object. The object is only created when a prvalue is used to initialize a variable or materialized into a temporary. This is why guaranteed elision works even with deleted copy and move constructors.

    Key insight: Guaranteed copy elision in C++17 is not the compiler being "clever." It is the language rule that prvalues do not produce objects until they are used. There is nothing to elide because the copy or move never existed in the abstract machine.

    Implicit Move on Return

    When RVO and NRVO do not apply, the compiler tries to implicitly move the return value. If a function returns a local variable or parameter by value, and NRVO cannot be applied, the compiler treats the return expression as an rvalue (even though the variable is an lvalue). This selects the move constructor if one is available:

    Widget make(bool flag) {
    Widget a(1), b(2);
    if (flag) return a;  // NRVO fails (two candidates)
    return b;            // but implicit move applies!
    // return a; is treated as: return std::move(a);
    // Move constructor called, not copy constructor.
}

    This implicit move applies to:

    • Named local variables in return statements
    • Function parameters in return statements (since C++20)
    • Throw expressions with local variables (since C++20)

    The std::move-on-Return Anti-Pattern

    One of the most common mistakes in C++ is adding std::move to a return statement:

    // WRONG: std::move PREVENTS RVO/NRVO!
Widget makeWidget() {
    Widget w(42);
    return std::move(w);  // BAD! Prevents NRVO.
    // Without std::move: NRVO constructs w directly in caller (0 copies, 0 moves)
    // With std::move: forces a move (1 move constructor call)
}

    Why does std::move prevent NRVO? NRVO requires the return expression to be the name of a local variable. std::move(w) is not a name; it is a function call that returns an rvalue reference. The compiler sees a different expression and cannot apply NRVO.

    The irony: std::move was meant to enable moves, but on a return statement it forces a move where none was needed. The code is slower with std::move than without it.

    Key insight: Never write return std::move(localVar);. Just write return localVar;. The compiler will apply NRVO if possible, and if not, it will implicitly treat the return as a move. Adding std::move can only make things worse, never better.

    When std::move on Return Is Correct

    There are rare cases where std::move in a return statement is appropriate:

    Returning a Member Variable

    class Builder {
    Widget widget_;
public:
    Widget build() && {   // rvalue-qualified member function
        return std::move(widget_);  // OK: moving a member out
        // widget_ is not a local; no NRVO possible.
        // Without std::move, this would copy.
    }
};

    Returning a Different Type

    std::unique_ptr<Base> makeBase() {
    auto derived = std::make_unique<Derived>();
    return std::move(derived);  // OK in C++14: different types
    // unique_ptr<Derived> to unique_ptr<Base> needs explicit move
    // Note: C++17/20 may handle this implicitly; check your compiler
}

    Returning a Parameter (Pre-C++20)

    // Before C++20, implicit move did not apply to parameters
Widget transform(Widget w) {
    w.modify();
    return std::move(w);  // OK before C++20: w is a parameter
    // In C++20+: implicit move applies, std::move is redundant
}

    Practical Guidelines

    • Return local variables by name. Write return w; not return std::move(w);. The compiler handles the rest.
    • Return prvalues when possible. return Widget(42); gives guaranteed elision, even with deleted copy/move constructors.
    • One return variable per function helps NRVO. If you need to return different variables from different branches, the compiler will use implicit move.
    • Do not worry about the return type being "expensive." With RVO/NRVO, returning a std::vector<int> with a million elements is free.
    • Avoid output parameters. In modern C++, returning by value is often faster than using output parameters, because RVO constructs directly in the caller's storage versus output parameters requiring a separate construction and assignment.
    // Old style: output parameter (avoid this)
void getData(std::vector<int>& out) {
    out.clear();
    for (int i = 0; i < 1000000; ++i)
        out.push_back(i);
}

// Modern style: return by value (prefer this)
std::vector<int> getData() {
    std::vector<int> result;
    result.reserve(1000000);
    for (int i = 0; i < 1000000; ++i)
        result.push_back(i);
    return result;  // NRVO: zero copies
}

    Compiler Flags and Verification

    To verify that NRVO is being applied, you can add tracing to your constructors:

    class Traced {
public:
    Traced() { std::cout << "default ctor\n"; }
    Traced(const Traced&) { std::cout << "copy ctor\n"; }
    Traced(Traced&&) { std::cout << "move ctor\n"; }
};

Traced make() {
    Traced t;
    return t;
}

int main() {
    Traced x = make();
    // With NRVO: prints only "default ctor"
    // Without NRVO: prints "default ctor" then "move ctor"
}

    To disable copy elision for testing, use -fno-elide-constructors on GCC/Clang. Note: this only disables optional elision. Guaranteed elision (C++17 prvalue returns) cannot be disabled because it is a language rule, not an optimization.

    Historical Context

    • C++98/03: RVO and NRVO were optional optimizations. Compilers could elide copies but were not required to. The copy constructor had to be accessible even if elided.
    • C++11: Move semantics added. When elision did not apply, the compiler could move instead of copy. Implicit move on return introduced for local variables.
    • C++17: Guaranteed copy elision for prvalue returns. Copy/move constructors need not even exist for prvalue returns. NRVO remains optional.
    • C++20: Implicit move extended to function parameters and catch clause parameters.
    • C++23: Further relaxation of implicit move rules.

    Summary

    • Copy elision eliminates copies and moves by constructing objects directly in their final location.
    • RVO (prvalue returns) is guaranteed in C++17. Zero cost. Works even with deleted copy/move constructors.
    • NRVO (named local returns) is not guaranteed but works reliably in practice with simple cases.
    • When NRVO does not apply, implicit move kicks in: the compiler treats the local variable as an rvalue.
    • Never write return std::move(x); for local variables. It prevents NRVO and forces a move where none was needed.
    • std::move on return is correct only for members, different types, or parameters in pre-C++20 code.
    • Prefer returning by value over output parameters. Modern C++ makes return-by-value free in most cases.
