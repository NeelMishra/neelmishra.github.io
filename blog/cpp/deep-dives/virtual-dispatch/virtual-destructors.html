    Virtual Destructors

    March 2026 · 15 min read

    The Problem: Deleting Through a Base Pointer

    Consider a seemingly innocent pattern that every C++ developer writes at some point:

    class Base {
public:
    ~Base() { std::cout << "~Base\n"; }
};

class Derived : public Base {
    std::string name_;
    std::vector<int> data_;
public:
    Derived(std::string n) : name_(std::move(n)), data_(1000000, 42) {}
    ~Derived() { std::cout << "~Derived\n"; }
};

void example() {
    Base* ptr = new Derived("important");
    delete ptr;  // Undefined Behavior!
}

    What happens when we call delete ptr? The static type of ptr is Base*. The destructor is not virtual. So the compiler generates a direct call to ~Base(). The Derived destructor never runs. The std::string inside name_ is never destroyed. The std::vector holding a million integers is never freed. Memory leaks. Resources leak.

    But it is worse than a leak. The C++ standard (Section 7.6.2.9) says that deleting an object through a pointer to a base class with a non-virtual destructor, when the object's actual type differs from the pointer type, is undefined behavior. Not "implementation-defined." Not "unspecified." Undefined. The compiler is free to do anything: crash, corrupt memory, silently succeed, or format your hard drive (theoretically).

    Key insight: A non-virtual destructor on a base class is not just a memory leak risk. It is undefined behavior when the derived class is deleted through a base pointer. This is one of the most common sources of subtle bugs in C++ codebases.

    Why Is It Undefined Behavior, Not Just a Leak?

    The reason goes deeper than missing destructor calls. When the compiler sees delete ptr where ptr is Base*, it computes the size and layout of the object to deallocate based on Base's metadata. But the actual allocated block was sized for Derived, which is larger. The deallocation function receives incorrect size information. On some allocators, this corrupts the heap metadata. On others, it happens to work by accident. This is classic undefined behavior: it appears to work in debug builds and fails catastrophically in production.

    Additionally, the compiler may perform optimizations based on the assumption that the static type matches the dynamic type (since the destructor is non-virtual). It might skip adjusting the pointer for multiple inheritance, skip calling subobject destructors, or reorder operations in ways that are only valid for the base type.

    The Fix: Make the Destructor Virtual

    The fix is simple: declare the base class destructor virtual.

    class Base {
public:
    virtual ~Base() { std::cout << "~Base\n"; }
};

class Derived : public Base {
    std::string name_;
    std::vector<int> data_;
public:
    Derived(std::string n) : name_(std::move(n)), data_(1000000, 42) {}
    ~Derived() override { std::cout << "~Derived\n"; }
};

void example() {
    Base* ptr = new Derived("important");
    delete ptr;  // Correct: ~Derived() then ~Base() via vtable
}

    Now delete ptr performs virtual dispatch on the destructor. The vtable lookup finds Derived::~Derived(), which runs first, cleaning up name_ and data_. Then Base::~Base() runs automatically (base destructors always chain). Everything is cleaned up correctly.

    The destruction order is always most-derived first, then base classes, the reverse of construction order. This is guaranteed by the language, regardless of whether the destructor is virtual. The virtual keyword simply ensures the correct most-derived destructor is found when the static type differs from the dynamic type.

    ▶ Destruction With and Without Virtual Destructor

    Trace object destruction through a base pointer with and without a virtual destructor.

    When to Make the Destructor Virtual

    The rule is straightforward:

    • If a class has any virtual functions, give it a virtual destructor. It already has a vtable (so no additional per-object cost), and someone might delete through a base pointer.
    • If a class is designed to be a base class and objects might be deleted through base pointers, give it a virtual destructor.
    • If a class is not designed for inheritance (no virtual functions, no protected members), you probably don't need a virtual destructor. Mark it final instead to prevent accidental inheritance.

    The cost of a virtual destructor on a class that already has virtual functions is zero: the destructor just gets one more slot in the existing vtable. The cost on a class with no other virtual functions is one pointer per object (the vptr). For small, frequently allocated objects, this can matter. For everything else, it is negligible.

    The Guideline from the C++ Core Guidelines

    The C++ Core Guidelines (C.35) state: "A base class destructor should be either public and virtual, or protected and non-virtual."

    The logic: if the destructor is public and virtual, anyone can delete through a base pointer safely. If the destructor is protected and non-virtual, nobody can delete through a base pointer (the delete expression won't compile), so the UB cannot occur. This second option is useful for base classes that are not meant to be used with owning base pointers, such as CRTP base classes or mixin classes.

    // Option 1: Public virtual destructor (most common)
class Animal {
public:
    virtual ~Animal() = default;
    virtual void speak() const = 0;
};

// Option 2: Protected non-virtual destructor
template <typename Derived>
class CrtpBase {
protected:
    ~CrtpBase() = default;  // can't delete through CrtpBase*
};

    Virtual Defaulted Destructors

    Modern C++ makes the syntax clean. You don't need an empty destructor body:

    class Base {
public:
    virtual ~Base() = default;
};

    This is a virtual destructor with a compiler-generated body. It does everything a hand-written empty destructor does, but it also signals to the compiler that you're intentionally using the default behavior. The compiler can optimize it better in some cases (e.g., trivial destructibility detection for is_trivially_destructible, though a virtual destructor is never trivially destructible).

    One subtlety: declaring virtual ~Base() = default; suppresses the implicit generation of move operations (due to the Rule of Five). If your base class needs to be movable, you must explicitly default the move constructor and move assignment operator:

    class Base {
public:
    virtual ~Base() = default;
    Base(Base&&) = default;
    Base& operator=(Base&&) = default;
    Base(const Base&) = default;
    Base& operator=(const Base&) = default;
};

    Pure Virtual Destructors

    You can declare a destructor as pure virtual. This makes the class abstract (it cannot be instantiated directly). But unlike other pure virtual functions, you must provide a definition for the pure virtual destructor, because base destructors are always called during the destruction of derived objects.

    class AbstractBase {
public:
    virtual ~AbstractBase() = 0;  // pure virtual
};

// MUST provide a definition:
AbstractBase::~AbstractBase() {
    // cleanup (often empty)
}

class Concrete : public AbstractBase {
    int* data_;
public:
    Concrete() : data_(new int[100]) {}
    ~Concrete() override { delete[] data_; }
};

    Why would you use this? When you want a class to be abstract but every function has a reasonable default implementation. Making the destructor pure virtual is a convenient way to force the class to be abstract without arbitrarily making some other function pure virtual.

    Without the out-of-line definition, you get a linker error when any derived class is destroyed, because the chain of destructor calls tries to invoke AbstractBase::~AbstractBase() and finds no definition.

    Virtual Destructors and Smart Pointers

    An important nuance: std::unique_ptr and std::shared_ptr handle this situation differently.

    std::unique_ptr requires a virtual destructor

    std::unique_ptr<Base> calls delete on the stored pointer when it goes out of scope. If the destructor is not virtual, you get the same undefined behavior as a raw delete:

    // Bug: same UB as raw delete through Base*
std::unique_ptr<Base> p = std::make_unique<Derived>("oops");
// ~unique_ptr calls delete on Base* -> UB if ~Base is non-virtual

    std::shared_ptr does not require a virtual destructor (usually)

    std::shared_ptr stores a deleter that is type-erased at construction time. When you write std::make_shared<Derived>(...), the shared_ptr remembers that the object is a Derived and calls Derived's destructor directly, regardless of the pointer type stored in the shared_ptr:

    // Works correctly even without virtual destructor!
std::shared_ptr<Base> p = std::make_shared<Derived>("works");
// ~shared_ptr calls the stored deleter, which knows the real type

    However, relying on this is fragile. If someone later converts the code to use unique_ptr, the bug reappears. The safer practice is to always have a virtual destructor on base classes meant for polymorphic use, regardless of which smart pointer you use.

    Key insight: std::shared_ptr type-erases the deleter, so it can destroy the correct type without a virtual destructor. std::unique_ptr does not; it needs a virtual destructor. Don't rely on shared_ptr's leniency; always make polymorphic base destructors virtual.

    Detecting Missing Virtual Destructors

    Compilers and static analysis tools can catch this bug:

    • Clang/GCC: -Wnon-virtual-dtor warns when a class has virtual functions but a non-virtual destructor.
    • Clang-Tidy: The cppcoreguidelines-virtual-class-destructor check enforces the Core Guidelines rule.
    • MSVC: Warning C4265 (class has virtual functions but destructor is not virtual) is off by default. Enable it with /W4 or /wall.
    • AddressSanitizer (ASan): Can detect some consequences of the UB (heap corruption, use-after-free) at runtime, but not the root cause.
    // Compile with warnings:
// g++ -Wnon-virtual-dtor -Werror example.cpp
// clang++ -Wnon-virtual-dtor -Werror example.cpp

    Enable these warnings in your build system and treat them as errors. This is one of the cheapest and most effective ways to prevent this class of bugs.

    Destruction Order in Deep Hierarchies

    Destruction always proceeds from the most-derived class to the least-derived base, regardless of virtual or non-virtual:

    class A { public: virtual ~A() { std::cout << "~A\n"; } };
class B : public A { public: ~B() override { std::cout << "~B\n"; } };
class C : public B { public: ~C() override { std::cout << "~C\n"; } };

A* p = new C();
delete p;
// Output:
// ~C
// ~B
// ~A

    The compiler chains the destructor calls automatically. C::~C() runs your code, then implicitly calls B::~B(), which runs your code and implicitly calls A::~A(). You never call base destructors manually; the compiler inserts the calls.

    During destruction, the vptr is adjusted at each level, just as during construction. When C::~C() finishes, the vptr is reset to point to B's vtable. When B::~B() finishes, the vptr is reset to A's vtable. This ensures that any virtual calls made during a destructor body resolve to the current class's version, not the most-derived class (which is already partially destroyed).

    Performance Implications

    Making a destructor virtual has the same mechanical cost as any other virtual function:

    • If the class already has virtual functions: Zero additional cost. The destructor is just one more entry in the existing vtable. The vptr is already there.
    • If the class has no other virtual functions: Adding a virtual destructor adds a vptr (8 bytes per object on 64-bit). The vtable itself is one entry in static storage (negligible).

    For most classes, this cost is irrelevant. For tiny objects allocated in the millions (nodes in a spatial index, particles in a simulation), the extra 8 bytes per object could matter. In those cases, consider whether you actually need polymorphic deletion. If not, use a protected non-virtual destructor and delete only through the concrete type.

    Common Mistakes

    Forgetting override on derived destructors

    While not strictly required (the destructor is always selected correctly by name), marking derived destructors with override is good practice. It documents intent and will trigger a compile error if the base destructor is not actually virtual:

    class Derived : public Base {
public:
    ~Derived() override;  // won't compile if Base::~Base is not virtual
};

    Inheriting from standard library classes

    Standard library classes like std::vector, std::string, std::map, and std::unordered_map do not have virtual destructors. Publicly inheriting from them and deleting through a base pointer is undefined behavior:

    // DON'T DO THIS:
class MyVector : public std::vector<int> {
    // ...
};

std::vector<int>* p = new MyVector();
delete p;  // UB: std::vector has no virtual destructor

    If you need to extend a standard container, use composition (store it as a member) rather than public inheritance.

    Protected inheritance and deletion

    Protected or private inheritance prevents the implicit conversion from Derived* to Base* outside the class, which means delete through a base pointer cannot happen from outside code. But member functions and friends can still do it, so the risk is reduced but not eliminated.

    Summary

    • Deleting a derived object through a base pointer with a non-virtual destructor is undefined behavior, not just a resource leak.
    • If a class has any virtual functions, give it a virtual destructor. The cost is one vtable slot (zero per-object overhead since the vptr already exists).
    • If a class is a base class but should not be deleted through a base pointer, make the destructor protected and non-virtual.
    • Pure virtual destructors make a class abstract but must still have an out-of-line definition.
    • std::shared_ptr type-erases the deleter and can work without a virtual destructor, but std::unique_ptr cannot. Always make polymorphic base destructors virtual.
    • Use -Wnon-virtual-dtor and treat it as an error in your build.
