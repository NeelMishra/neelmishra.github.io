    Alternatives to Virtual Dispatch

    March 2026 · 20 min read

    Why Look Beyond Virtual Functions?

    Virtual functions are the primary mechanism for runtime polymorphism in C++, and for most code they work well. But they come with costs that matter in specific contexts:

    • Indirect calls: Each virtual call goes through two pointer indirections (object → vtable → function). The indirect branch can mispredict on CPUs.
    • Lost inlining: The compiler generally cannot inline a virtual call, because it does not know the target function at compile time. Inlining is one of the most impactful optimizations.
    • Heap allocation bias: Polymorphism through base pointers naturally leads to heap-allocated objects. Storing heterogeneous objects in contiguous memory (good for cache) is awkward.
    • Open set only: The set of types in a virtual hierarchy is "open" (anyone can add a derived class). If your set of types is closed (fixed at compile time), you pay for flexibility you do not use.

    C++ offers several alternatives, each with different trade-offs. This article covers the five most important ones: CRTP, std::variant + std::visit, type erasure, policy-based design, and C++20 concepts.

    ▶ Dispatch Mechanisms Compared

    Step through four dispatch techniques to compare their structure, trade-offs, and dispatch paths.

    CRTP: The Curiously Recurring Template Pattern

    CRTP is the classic "static polymorphism" technique. A base class template takes the derived class as a template parameter, and calls derived-class methods through a static_cast:

    template <typename Derived>
class Shape {
public:
    double area() const {
        return static_cast<const Derived*>(this)->area_impl();
    }

    void draw() const {
        static_cast<const Derived*>(this)->draw_impl();
    }
};

class Circle : public Shape<Circle> {
    double r_;
public:
    explicit Circle(double r) : r_(r) {}

    double area_impl() const { return 3.14159265 * r_ * r_; }
    void draw_impl() const { std::cout << "Circle(r=" << r_ << ")\n"; }
};

class Square : public Shape<Square> {
    double side_;
public:
    explicit Square(double s) : side_(s) {}

    double area_impl() const { return side_ * side_; }
    void draw_impl() const { std::cout << "Square(s=" << side_ << ")\n"; }
};

    The key difference from virtual functions: the "dispatch" happens at compile time. The compiler knows the exact type at every call site, so it can inline area_impl() and draw_impl() directly. There is no vtable, no vptr, no indirect call, and no overhead.

    Using CRTP with templates

    template <typename ShapeT>
void printArea(const Shape<ShapeT>& shape) {
    std::cout << "Area: " << shape.area() << "\n";
}

Circle c(5.0);
Square s(3.0);
printArea(c);   // inlined: Circle::area_impl
printArea(s);   // inlined: Square::area_impl

    Each call to printArea generates a separate instantiation. There is no shared base pointer, so you cannot store Circle and Square in the same container. CRTP is "polymorphism" in the sense of code reuse and interface conformance, but not in the sense of runtime type flexibility.

    Limitations

    • No heterogeneous containers: You cannot have a std::vector<Shape*> because there is no common base (each Shape<T> is a different type).
    • Closed set known at compile time: Every type must be known where it is used.
    • Error messages: If Derived forgets to implement area_impl(), the error message is a template instantiation failure, which can be long and confusing.
    • No runtime flexibility: You cannot switch behavior based on runtime input without additional machinery.
    Key insight: CRTP gives you zero-overhead static polymorphism. It is ideal when the set of types is fixed at compile time and you need the compiler to inline and optimize aggressively. It is not a replacement for runtime polymorphism.

    std::variant + std::visit

    C++17's std::variant is a type-safe discriminated union. Combined with std::visit, it provides a closed-set runtime polymorphism mechanism that avoids heap allocation and virtual dispatch:

    struct Circle { double radius; };
struct Square { double side; };
struct Triangle { double base, height; };

using Shape = std::variant<Circle, Square, Triangle>;

double area(const Shape& shape) {
    return std::visit([](const auto& s) -> double {
        using T = std::decay_t<decltype(s)>;
        if constexpr (std::is_same_v<T, Circle>)
            return 3.14159265 * s.radius * s.radius;
        else if constexpr (std::is_same_v<T, Square>)
            return s.side * s.side;
        else
            return 0.5 * s.base * s.height;
    }, shape);
}

    Or, with the overloaded-lambda pattern (cleaner):

    template <class... Ts> struct overloaded : Ts... { using Ts::operator()...; };
template <class... Ts> overloaded(Ts...) -> overloaded<Ts...>;

double area(const Shape& shape) {
    return std::visit(overloaded{
        [](const Circle& c)   { return 3.14159265 * c.radius * c.radius; },
        [](const Square& s)   { return s.side * s.side; },
        [](const Triangle& t) { return 0.5 * t.base * t.height; },
    }, shape);
}

    How std::visit dispatches

    Under the hood, std::visit typically uses a function pointer table (similar to a vtable) indexed by the variant's runtime discriminant. Some implementations use a switch statement or a computed goto instead. The dispatch is a single indirect call or jump, compared to virtual dispatch's two pointer loads plus an indirect call.

    Advantages over virtual functions

    • Value semantics: std::variant is a value type. It lives on the stack or inline in a container. No heap allocation needed.
    • Cache-friendly: A std::vector<Shape> stores all variants contiguously in memory. Iterating over them is cache-friendly, unlike a std::vector<Shape*> that chases pointers to scattered heap objects.
    • Exhaustiveness checking: If you use the overloaded-lambda pattern, the compiler ensures you handle every alternative. Adding a new type to the variant and forgetting to handle it is a compile error.
    • No inheritance needed: The types in a variant do not need to share a base class. They can be completely unrelated.

    Limitations

    • Closed set: All types must be listed in the variant declaration. Adding a new type requires changing the using Shape = std::variant<...> definition and every std::visit call site.
    • Size: The variant's size is the maximum size of its alternatives plus the discriminant. If one alternative is much larger than the others, all variants waste that space.
    • No type extension without recompilation: Unlike virtual functions where you can add a new derived class in a separate translation unit, adding a variant alternative requires recompiling everything that uses the variant.

    Type Erasure

    Type erasure combines the runtime flexibility of virtual dispatch with a value-semantic interface. The idea: wrap any type that satisfies a concept (duck typing) into a fixed-size, copyable/movable object that dispatches through a hidden virtual function table.

    std::function is the standard library's most well-known type erasure facility. It can hold any callable (function pointer, lambda, functor) and call it through a uniform interface:

    std::function<int(int, int)> op;

op = [](int a, int b) { return a + b; };
std::cout << op(3, 4) << "\n";  // 7

op = [](int a, int b) { return a * b; };
std::cout << op(3, 4) << "\n";  // 12

    But you can build your own type erasure for any interface. Here is a sketch for a "drawable" interface:

    class Drawable {
    struct Concept {
        virtual ~Concept() = default;
        virtual void draw_() const = 0;
        virtual double area_() const = 0;
        virtual std::unique_ptr<Concept> clone_() const = 0;
    };

    template <typename T>
    struct Model : Concept {
        T data_;
        explicit Model(T d) : data_(std::move(d)) {}
        void draw_() const override { data_.draw(); }
        double area_() const override { return data_.area(); }
        std::unique_ptr<Concept> clone_() const override {
            return std::make_unique<Model>(data_);
        }
    };

    std::unique_ptr<Concept> self_;

public:
    template <typename T>
    Drawable(T x) : self_(std::make_unique<Model<T>>(std::move(x))) {}

    Drawable(const Drawable& other) : self_(other.self_->clone_()) {}
    Drawable(Drawable&&) = default;
    Drawable& operator=(const Drawable&);
    Drawable& operator=(Drawable&&) = default;

    void draw() const { self_->draw_(); }
    double area() const { return self_->area_(); }
};

    Usage looks like value semantics with polymorphic behavior:

    struct Circle {
    double radius;
    void draw() const { std::cout << "Circle\n"; }
    double area() const { return 3.14159265 * radius * radius; }
};

struct Square {
    double side;
    void draw() const { std::cout << "Square\n"; }
    double area() const { return side * side; }
};

std::vector<Drawable> shapes;
shapes.push_back(Circle{5.0});    // no base class needed
shapes.push_back(Square{3.0});    // duck typing: has draw() and area()

for (const auto& s : shapes) {
    s.draw();
    std::cout << s.area() << "\n";
}

    How it works

    Type erasure uses virtual dispatch internally (the Concept/Model hierarchy), but hides it behind a value-semantic wrapper. The client code never sees pointers or inheritance. Any type that has the right interface (draw() and area()) can be wrapped, without inheriting from any base class.

    Advantages

    • Value semantics: Drawable is copyable, movable, and storable in standard containers.
    • Open set: New types can be added without modifying Drawable. Any type with the right interface works (duck typing).
    • No inheritance in user types: Circle and Square are plain structs. They don't need to know about Drawable.

    Limitations

    • Heap allocation: The unique_ptr<Concept> typically heap-allocates the model. Small-buffer optimization (SBO) can mitigate this for small types.
    • Virtual dispatch overhead: The internal mechanism is still virtual dispatch. The performance is similar to traditional virtual functions.
    • Boilerplate: Writing the Concept/Model/Wrapper triad is verbose. Libraries like dyno and proposals like std::proxy (P0957) aim to reduce this.
    Key insight: Type erasure gives you runtime polymorphism with value semantics and duck typing. It is the technique behind std::function, std::any, and std::move_only_function. Use it when you want an open set of types without forcing inheritance on the user types.

    Policy-Based Design

    Policy-based design uses template parameters to inject behavior at compile time. Instead of inheriting from a base class and overriding virtual functions, you pass "policy" classes as template arguments that define specific behaviors:

    template <typename LogPolicy, typename ThreadPolicy>
class Server {
    LogPolicy logger_;
    ThreadPolicy threads_;
public:
    void start() {
        logger_.log("Starting server");
        threads_.run([this]{ serve(); });
    }

    void serve() {
        logger_.log("Serving requests");
    }
};

// Policies: plain classes, no inheritance needed
struct ConsoleLog {
    void log(const char* msg) { std::cout << "[LOG] " << msg << "\n"; }
};

struct FileLog {
    std::ofstream file_{"server.log"};
    void log(const char* msg) { file_ << msg << "\n"; }
};

struct SingleThread {
    template <typename F>
    void run(F f) { f(); }
};

struct MultiThread {
    template <typename F>
    void run(F f) { std::thread(f).detach(); }
};

// Compose at compile time:
Server<ConsoleLog, SingleThread> devServer;
Server<FileLog, MultiThread> prodServer;

    Advantages

    • Zero overhead: All dispatch is resolved at compile time. Policy methods are inlined.
    • Composable: You can mix and match policies independently. With N logging policies and M threading policies, you get N*M combinations without writing N*M classes.
    • No vtable, no vptr: Each Server<A, B> is a distinct type with direct calls.

    Limitations

    • Closed set at compile time: The policy must be specified as a template argument. You cannot change the logging policy at runtime.
    • Binary code bloat: Each combination of template arguments generates a separate instantiation. With many policies, this can increase binary size.
    • Error messages: Template errors can be long. C++20 concepts improve this significantly.

    Alexandrescu's Modern C++ Design popularized this technique. The standard library uses it extensively: allocators (std::vector<int, MyAllocator>), comparison functions (std::set<int, MyCompare>), and hash functions (std::unordered_map<K, V, MyHash>) are all policies.

    C++20 Concepts: Constrained Static Polymorphism

    C++20 concepts formalize the requirements that template parameters must satisfy. They turn implicit "this type must have a draw() method" into explicit, compiler-checked constraints:

    template <typename T>
concept Drawable = requires(const T& t) {
    { t.draw() } -> std::same_as<void>;
    { t.area() } -> std::convertible_to<double>;
};

struct Circle {
    double radius;
    void draw() const { std::cout << "Circle\n"; }
    double area() const { return 3.14159265 * radius * radius; }
};

struct Square {
    double side;
    void draw() const { std::cout << "Square\n"; }
    double area() const { return side * side; }
};

// Constrained function template:
void render(const Drawable auto& shape) {
    shape.draw();
    std::cout << "Area: " << shape.area() << "\n";
}

// Or equivalently:
template <Drawable T>
void render(const T& shape) {
    shape.draw();
    std::cout << shape.area() << "\n";
}

    Concepts vs. virtual functions

    • Compile-time checking: If Circle does not satisfy Drawable, the compiler tells you immediately with a clear error message referencing the unsatisfied constraint.
    • No inheritance: Circle and Square are plain types. They don't inherit from anything.
    • Zero overhead: The compiler generates code specialized for each type. Everything inlines.
    • Better errors: Instead of walls of template instantiation backtraces, you get "type X does not satisfy concept Drawable because it lacks member function draw()".

    Concepts for policy-based design

    Concepts shine when combined with policy-based design, constraining the policy parameters:

    template <typename T>
concept Logger = requires(T& t, const char* msg) {
    { t.log(msg) };
};

template <typename T>
concept ThreadModel = requires(T& t, std::function<void()> f) {
    { t.run(f) };
};

template <Logger LogPolicy, ThreadModel ThreadPolicy>
class Server {
    // ... same as before, but now with compile-time constraint checking
};

    If someone passes a type that lacks a log() method, the error message says "LogPolicy does not satisfy Logger" instead of a cryptic template substitution failure deep in the server's implementation.

    Concepts and type erasure: std::proxy

    The proposed std::proxy (P3086, formerly P0957) combines concepts with type erasure. You define a "facade" (interface) using concepts, and std::proxy handles the type erasure boilerplate:

    // Proposed (not yet in the standard):
// struct DrawableFacade : pro::facade_builder
//     ::add_convention</* draw */>
//     ::add_convention</* area */>
//     ::build {};
// pro::proxy<DrawableFacade> p = Circle{5.0};
// p.draw();

    This would give you the ergonomics of virtual functions (runtime polymorphism, heterogeneous containers) with value semantics and no user-side inheritance. It is the direction C++ is heading for polymorphism.

    Choosing the Right Mechanism

    There is no single best dispatch mechanism. The choice depends on your requirements:

    Use virtual functions when:

    • The set of types is open (new types can be added by other code, possibly in other translation units or shared libraries).
    • You need runtime polymorphism through base pointers/references.
    • The call frequency is low enough that indirect call overhead is irrelevant.
    • You want the simplest, most well-understood mechanism.

    Use CRTP when:

    • The set of types is closed and known at compile time.
    • You need zero-overhead dispatch with full inlining.
    • You don't need heterogeneous containers.
    • You're building a library where derived types are known at the point of use (e.g., Eigen's expression templates).

    Use std::variant when:

    • The set of types is closed, small, and fixed.
    • You want value semantics and cache-friendly storage.
    • You want the compiler to enforce exhaustive handling of all alternatives.
    • All alternatives have roughly similar sizes (to avoid wasting space).

    Use type erasure when:

    • The set of types is open, but you want value semantics (not pointer semantics).
    • You don't want to force inheritance on user types (duck typing).
    • You're building a library boundary where the concrete types are unknown.

    Use policy-based design when:

    • You have orthogonal axes of variation (logging, allocation, threading, etc.).
    • The behavior is fixed at compile time.
    • You want to compose behaviors without combinatorial explosion of manual classes.

    Use concepts when:

    • You are writing templates and want clear, compiler-checked constraints.
    • You want better error messages than unconstrained templates.
    • Combined with any of the above (CRTP, policies) to formalize requirements.

    Performance Comparison

    Benchmarking dispatch mechanisms on a modern x86-64 system (processing 10 million shapes per iteration, three types, Intel Core i7, GCC 13 with -O2):

    // Approximate measured throughput (millions of dispatch operations per second):
// Direct call (baseline):         ~850M ops/s
// CRTP:                           ~840M ops/s   (within noise of direct)
// std::variant + std::visit:      ~620M ops/s   (switch/jump table)
// Virtual dispatch (base ptr):    ~380M ops/s   (branch prediction varied)
// Type erasure (std::function):   ~290M ops/s   (double indirection + possible heap)

    These numbers vary enormously based on CPU, compiler, optimization level, and workload patterns. The gap narrows when the work done per dispatch is significant (a function that takes microseconds makes a 2ns dispatch overhead irrelevant). The gap widens in tight loops over heterogeneous collections where branch prediction is stressed.

    Key insight: Choose the dispatch mechanism based on your design requirements (open vs. closed set, value vs. reference semantics, runtime vs. compile-time flexibility), not on micro-benchmark numbers. Only optimize dispatch if profiling proves it is a bottleneck.

    Combining Techniques

    These mechanisms are not mutually exclusive. Real-world code often combines them:

    • Virtual + CRTP: Use CRTP for a static interface within a virtual hierarchy. The base class uses virtual dispatch for external polymorphism, while CRTP provides zero-overhead access to derived-class helpers internally.
    • Variant inside type erasure: A type-erased wrapper might use a variant internally for SBO, switching to heap allocation only for large types.
    • Concepts + policies: Constrain policy template parameters with concepts for clear error messages.
    • virtual + final: Mark leaf classes final to let the compiler devirtualize, getting the flexibility of virtual dispatch with the performance of direct calls for known types.

    Summary

    • Virtual functions are the simplest runtime polymorphism mechanism. Open set. Reference semantics. Moderate overhead from indirect calls and lost inlining.
    • CRTP provides zero-overhead static polymorphism. Closed set. No heterogeneous containers. Full inlining.
    • std::variant + std::visit gives closed-set runtime polymorphism with value semantics, cache-friendly storage, and exhaustive matching. Best for small, fixed type sets.
    • Type erasure enables open-set runtime polymorphism with value semantics and duck typing. Used by std::function and std::any.
    • Policy-based design composes behaviors through template parameters. Zero overhead, but closed-set and compile-time only.
    • C++20 concepts add compile-time constraint checking to any template-based technique, dramatically improving error messages and code clarity.
    • Choose based on design requirements (open/closed set, value/reference semantics), not performance alone.
