    How vtables Work

    March 2026 · 20 min read

    Runtime Polymorphism in C++

    When you call a virtual function through a base-class pointer or reference, the program must decide at runtime which function to invoke. The object might be a Base, or it might be a Derived, or a MoreDerived. The caller cannot know at compile time. This is what people mean by runtime polymorphism: the actual function that runs depends on the dynamic type of the object, not the static type of the pointer.

    How does the compiler make this work? It cannot just hardcode the function address into the call site, because the dynamic type is unknown until the program actually runs. Instead, every major C++ compiler uses a mechanism called a virtual table, or vtable for short. The C++ standard does not mandate vtables, but every compiler you will encounter (GCC, Clang, MSVC, ICC) uses them. Understanding the vtable is understanding how virtual dispatch really works.

    The vptr: A Hidden Pointer in Every Polymorphic Object

    When a class declares or inherits at least one virtual function, the compiler adds a hidden member to every instance of that class: a pointer that points to the class's vtable. This pointer is called the vptr (virtual table pointer). You never see it in your source code, but it is always there, silently occupying space in your objects.

    class Base {
public:
    virtual void speak() { std::cout << "Base\n"; }
    virtual void greet() { std::cout << "Hello from Base\n"; }
    int value = 42;
};

// sizeof(Base) on a 64-bit system:
//   8 bytes (hidden vptr) + 4 bytes (int value) + 4 bytes (padding) = 16 bytes

    The vptr is typically the first member in the object layout (though the standard permits any placement). When you create a Base object, the compiler-generated constructor sets the vptr to point to Base's vtable. When you create a Derived object, the Derived constructor overwrites the vptr to point to Derived's vtable. This overwrite happens automatically, during construction, before your constructor body runs.

    Key insight: The vptr costs exactly one pointer per object (typically 8 bytes on 64-bit systems). This is the only per-object cost of virtual functions. The vtable itself is stored once per class, not once per object.

    The vtable: One Table Per Class

    A vtable is a static array of function pointers, one entry per virtual function, stored in the read-only data segment of your binary. The compiler generates one vtable per polymorphic class. Every object of that class shares the same vtable; the per-object vptr simply points to it.

    Consider this hierarchy:

    class Base {
public:
    virtual void speak() { std::cout << "Base::speak\n"; }
    virtual void greet() { std::cout << "Base::greet\n"; }
    virtual ~Base() = default;
};

class Derived : public Base {
public:
    void speak() override { std::cout << "Derived::speak\n"; }
    // greet() is NOT overridden, so it inherits Base::greet
};

    The compiler generates two vtables:

    • Base's vtable: [&Base::speak, &Base::greet, &Base::~Base]
    • Derived's vtable: [&Derived::speak, &Base::greet, &Derived::~Derived]

    Notice: the slot for speak() in Derived's vtable points to Derived::speak, because Derived overrides it. The slot for greet() still points to Base::greet, because Derived does not override it. The destructor slot points to each class's own destructor.

    The key invariant is that corresponding virtual functions always occupy the same slot index in both tables. speak() is always slot 0, greet() is always slot 1, and the destructor is always slot 2. This fixed layout makes the dispatch mechanism extremely efficient.

    How a Virtual Call Works

    Let us trace exactly what happens at the machine level when you write:

    Base* ptr = new Derived();
ptr->speak();  // Which speak() gets called?

    The compiler translates ptr->speak() into roughly this sequence of operations:

    1. Load the vptr: Read the hidden pointer from the object. On x86-64, this is something like mov rax, [rdi] where rdi holds the object address.
    2. Index into the vtable: speak() is at slot 0, so add 0 * sizeof(pointer) to the vtable address. For slot 1 it would be 1 * 8 = 8 bytes.
    3. Load the function pointer: Read the function address from the vtable slot. Something like mov rax, [rax + 0].
    4. Call through the pointer: call rax. The CPU jumps to whatever function address the vtable held.

    Because the vptr points to Derived's vtable (since the actual object is a Derived), slot 0 contains &Derived::speak. So Derived::speak() runs. If the object had been a Base, the vptr would point to Base's vtable, slot 0 would contain &Base::speak, and Base::speak() would run.

    This entire mechanism is exactly two pointer dereferences and one indirect call. There is no string comparison, no hash lookup, no dictionary search. It is fast.

    The vptr During Construction and Destruction

    A subtle but critical detail: the vptr changes during the construction of an object. When you construct a Derived object, the process works like this:

    1. Memory is allocated for the full Derived object.
    2. The Base constructor runs. At this point, the vptr is set to point to Base's vtable.
    3. The Derived constructor runs. The vptr is overwritten to point to Derived's vtable.

    This means that if you call a virtual function inside the Base constructor, it will call Base's version, not Derived's. This is intentional: the Derived part of the object has not been constructed yet, so calling Derived's overrides would access uninitialized state.

    class Base {
public:
    Base() {
        speak();  // Calls Base::speak, NOT Derived::speak!
    }
    virtual void speak() { std::cout << "Base\n"; }
};

class Derived : public Base {
public:
    Derived() : data_(42) {}
    void speak() override {
        std::cout << "Derived: " << data_ << "\n";  // data_ not yet init in Base ctor
    }
private:
    int data_;
};

    The same applies in reverse during destruction: when Derived's destructor finishes, the vptr is reset to point to Base's vtable before Base's destructor runs. So virtual calls inside destructors also dispatch to the current class's version, not the most-derived class.

    Key insight: Never call virtual functions from constructors or destructors expecting polymorphic behavior. During construction, the object "is" the class currently being constructed. During destruction, it "is" the class currently being destroyed.

    Pure Virtual Functions and Abstract Classes

    When you declare a function = 0, you create a pure virtual function. The class becomes abstract and cannot be instantiated directly.

    class Shape {
public:
    virtual double area() const = 0;   // pure virtual
    virtual void draw() const = 0;     // pure virtual
    virtual ~Shape() = default;
};

class Circle : public Shape {
    double radius_;
public:
    explicit Circle(double r) : radius_(r) {}
    double area() const override { return 3.14159265 * radius_ * radius_; }
    void draw() const override { std::cout << "Drawing circle\n"; }
};

    What does the vtable look like for an abstract class? The compiler still generates a vtable for Shape, but the slots for pure virtual functions typically contain a pointer to a special function (often called __cxa_pure_virtual on GCC/Clang) that aborts the program if called. This acts as a safety net: if somehow a pure virtual function is invoked (usually through a bug in constructor/destructor code), the program crashes immediately with a clear error rather than silently doing the wrong thing.

    A class can even have a pure virtual function with a definition:

    class Base {
public:
    virtual void prepare() = 0;  // pure virtual, but with a body below
};

void Base::prepare() {
    std::cout << "Base preparation steps\n";
}

class Derived : public Base {
public:
    void prepare() override {
        Base::prepare();  // explicitly call the base implementation
        std::cout << "Derived-specific preparation\n";
    }
};

    The class is still abstract (you cannot instantiate Base directly), but derived classes can explicitly call the base implementation using the fully qualified name Base::prepare(). This pattern is useful when you want to mandate overriding but also provide a shared "default" implementation that derived classes can opt into.

    Inspecting vtables in Real Binaries

    You can see vtables in compiled binaries using common tools. On Linux/macOS with GCC or Clang:

    # Dump class layout with vtable info (Clang)
clang++ -cc1 -fdump-record-layouts -emit-llvm example.cpp

# See vtable symbols in the compiled binary
nm -C a.out | grep vtable

# GCC: dump class hierarchy
g++ -fdump-lang-class example.cpp

    For a simple hierarchy, nm -C reveals entries like:

    vtable for Base
vtable for Derived

    And the class layout dump shows exactly where the vptr sits and how much space it takes. On MSVC, the equivalent is the /d1reportAllClassLayout flag:

    cl /d1reportAllClassLayout example.cpp

    This prints detailed object layouts including vptr offsets, vtable contents, and padding bytes. These tools are invaluable when debugging performance issues or understanding the memory footprint of your class hierarchies.

    Virtual Call Overhead: Reality Check

    Virtual function calls have measurable overhead compared to direct (non-virtual) calls, but the magnitude is often overstated. Let us quantify it.

    Direct costs

    • Memory indirection: Two extra pointer loads (object → vtable → function pointer) compared to a direct call where the function address is baked into the instruction stream.
    • Indirect branch: The call [reg] instruction is an indirect branch. Modern CPUs predict indirect branches, but mispredictions cost 10-20 cycles on typical hardware. For a tight inner loop calling the same virtual function on objects of varying types, the branch predictor may struggle.
    • Cache effects: The vtable must be in the data cache. For a single class hierarchy used frequently, this is almost always the case. For code that touches many unrelated vtables, the extra cache lines matter.

    Indirect costs (the bigger deal)

    • Lost inlining: The compiler cannot inline a virtual call in the general case, because it does not know which function will be called. Inlining is one of the most powerful optimizations (it enables constant folding, dead code elimination, register allocation across the call boundary, and more). Losing it often matters far more than the two pointer loads.
    • Lost interprocedural optimizations: Without knowing the callee, the optimizer must assume the worst about side effects, aliasing, and exception behavior.

    Numbers

    On a modern x86-64 CPU, a correctly predicted virtual call adds roughly 1-3 nanoseconds compared to a direct call. A mispredicted virtual call costs 10-25 nanoseconds. These numbers are dwarfed by a cache miss (50-100 ns) or a heap allocation (50-200 ns). For most applications, virtual call overhead is lost in the noise.

    Where it matters: tight inner loops processing millions of objects per frame (game engines, physics simulations, signal processing), or latency-critical paths in trading systems. Outside those domains, measure before optimizing away virtual dispatch.

    Devirtualization: When the Compiler Knows Better

    Sometimes the compiler can prove at compile time which function a virtual call will invoke. When it can, it replaces the indirect call with a direct call (or even inlines it). This optimization is called devirtualization.

    Trivial devirtualization

    When you call a virtual function on an object with a known concrete type, the compiler devirtualizes immediately:

    Derived d;
d.speak();        // Compiler knows d is exactly Derived, calls Derived::speak directly

Derived* dp = &d;
dp->speak();      // Compiler can often prove dp points to exactly a Derived

    This case is straightforward because the compiler can see the exact type. No vtable lookup is generated at all.

    Speculative devirtualization

    Modern compilers (especially with profile-guided optimization, or PGO) can speculatively devirtualize: they emit a type check followed by a direct call for the common case, with a fallback to the vtable lookup for other types:

    // What the compiler might generate (pseudocode):
if (ptr->vptr == &Derived_vtable) {
    Derived::speak(ptr);    // direct call, inlineable
} else {
    ptr->vptr[0](ptr);     // fallback: full virtual dispatch
}

    This trades a cheap comparison for the ability to inline the hot path. With PGO, the compiler knows which type appears most often and optimizes for it.

    Final-based devirtualization

    If a class or method is marked final, the compiler knows no further overriding is possible and can devirtualize aggressively. We cover this in detail in the override and final article.

    Link-time optimization (LTO)

    With LTO enabled, the compiler sees the entire program. If it can prove that a class has no derived classes (even without final), it can devirtualize. LTO dramatically increases devirtualization opportunities because the compiler can analyze the full class hierarchy across all translation units.

    Key insight: The best optimization for virtual call overhead is not rewriting your code to avoid virtual functions. It is enabling LTO and PGO so the compiler can devirtualize automatically where it matters.

    Real-World vtable Layout Details

    The vtable is more than just function pointers. In the Itanium C++ ABI (used by GCC and Clang on Linux, macOS, and most Unix-like systems), the vtable contains additional metadata:

    • Offset-to-top: The distance from the vtable pointer to the top (most-derived) object. This is 0 for single inheritance but non-zero for secondary base classes in multiple inheritance.
    • RTTI pointer: A pointer to the type_info object for the class. This is how dynamic_cast and typeid work: they read the RTTI pointer from the vtable and walk the type hierarchy.
    • Virtual function pointers: One entry per virtual function, in declaration order.
    • Virtual base offsets: For virtual inheritance, additional entries encode the offset from the object to each virtual base subobject.

    A simplified Itanium vtable layout for a class with two virtual functions:

    // Memory layout of vtable (simplified):
//   [-16] offset-to-top (ptrdiff_t)
//   [-8]  RTTI pointer (typeinfo*)
//   [0]   &Class::first_virtual_function    <-- vptr points here
//   [8]   &Class::second_virtual_function
//   [16]  &Class::destructor

    The vptr in the object points to the first virtual function entry (offset 0), not to the beginning of the vtable. The RTTI pointer and offset-to-top are at negative offsets from where the vptr points.

    MSVC uses a different ABI. It stores RTTI separately in a "Complete Object Locator" structure, and the vtable layout differs in details (especially for virtual inheritance), but the fundamental concept is the same: a per-class static table of function pointers, indexed by a hidden per-object pointer.

    Impact on sizeof and Alignment

    Adding a single virtual function to a class that previously had none increases sizeof by the size of one pointer (8 bytes on 64-bit, 4 bytes on 32-bit). This is the vptr. It also changes the alignment requirements of the class.

    class NoVirtual {
    int x;  // sizeof = 4
};

class OneVirtual {
    virtual void f();
    int x;
    // sizeof = 16 on 64-bit: 8 (vptr) + 4 (x) + 4 (padding)
};

class TwoVirtual {
    virtual void f();
    virtual void g();
    int x;
    // sizeof = 16 on 64-bit: still 16! Same vptr, same padding.
    // Additional virtual functions don't increase object size.
};

    This is important: the per-object cost of virtual functions is constant regardless of how many virtual functions the class has. Whether you have 1 or 100 virtual functions, each object still contains exactly one vptr. Only the vtable (a per-class structure) grows, and that is in static storage, not on the heap or stack.

    However, for small, frequently allocated objects (such as nodes in a tree or a linked list), that extra 8 bytes per object can have real cache-line and memory-bandwidth implications. If you have a million small objects, the vptr adds 8 MB of overhead. This is why performance-critical code sometimes avoids virtual functions for tiny objects.

    Multi-Level Inheritance

    vtables compose naturally through multi-level inheritance:

    class A {
public:
    virtual void f();
    virtual void g();
};

class B : public A {
public:
    void f() override;   // overrides A::f
    virtual void h();    // new virtual function in B
};

class C : public B {
public:
    void g() override;   // overrides A::g (skipping B)
    void h() override;   // overrides B::h
};

    The vtable for C must contain entries for all virtual functions reachable from C: f, g, and h. The layout:

    • A's vtable: [&A::f, &A::g]
    • B's vtable: [&B::f, &A::g, &B::h]
    • C's vtable: [&B::f, &C::g, &C::h]

    Notice that C's slot for f() points to B::f, because C does not override f(). The vtable always reflects the most-derived override for each function.

    Calling Base-Class Versions Explicitly

    You can bypass virtual dispatch entirely using the fully qualified name:

    Derived d;
Base* bp = &d;

bp->speak();          // virtual dispatch: Derived::speak
bp->Base::speak();    // NOT virtual: always Base::speak

    The syntax bp->Base::speak() tells the compiler to call Base::speak directly, regardless of the dynamic type. This generates a direct call with no vtable lookup. This is commonly used inside overriding functions when you want to call the parent's implementation before or after your own logic:

    class Derived : public Base {
public:
    void speak() override {
        Base::speak();            // call parent first
        std::cout << "...and Derived extras\n";
    }
};

    Virtual vs Non-Virtual Functions: Compilation Differences

    Understanding the assembly gives intuition for the cost difference. Here is a simplified comparison on x86-64:

    // Non-virtual call: ptr->nonVirtual()
// Assembly:
//   call Base::nonVirtual        ; direct call, address known at link time

// Virtual call: ptr->speak()
// Assembly:
//   mov rax, [rdi]               ; load vptr from object
//   mov rax, [rax + SLOT*8]      ; load function pointer from vtable
//   call rax                     ; indirect call through register

    The non-virtual call is a single instruction with a fixed target. The virtual call is three instructions with a data-dependent target. The indirect call through rax is harder for the CPU's branch predictor, and the extra loads can miss in the cache (though they rarely do for hot paths).

    Common vtable Pitfalls

    Slicing destroys the vptr

    Assignment by value copies the data members but uses the target type's constructor. The vptr of the target is set by the target's constructor, not copied from the source. This is why passing a Derived by value to a function expecting a Base "slices" the object: the copy is a genuine Base with Base's vptr.

    void process(Base b) {     // b is a copy: a Base, not a Derived
    b.speak();             // always calls Base::speak
}

Derived d;
process(d);   // sliced! Derived part is lost.

    To preserve polymorphism, always pass by pointer or reference.

    Forgetting the virtual destructor

    If you delete a derived object through a base pointer and the base destructor is not virtual, the derived destructor never runs. This is undefined behavior and usually a resource leak. We cover this in detail in the next article.

    Virtual functions in constructors

    As discussed, virtual calls in constructors dispatch to the current class, not the most-derived class. This often surprises developers who expect polymorphic behavior from constructors. The workaround is a two-phase initialization: construct first, then call a virtual "init" method. But this pattern has its own problems (callers can forget the second call), so prefer using RAII and designing your constructors to not need virtual dispatch.

    Summary

    • vtables are per-class static arrays of function pointers. The vptr is a hidden per-object pointer to the class's vtable.
    • Virtual dispatch is two pointer loads and one indirect call. It is cheap in isolation but prevents inlining, which is the larger cost.
    • The vptr is set during construction and changes as each base constructor runs. Virtual calls in constructors and destructors are not polymorphic.
    • Compilers devirtualize when they can prove the dynamic type: local objects, final classes, LTO, and PGO all help.
    • Object slicing destroys virtual dispatch. Always use pointers or references for polymorphism.
    • Adding the first virtual function to a class adds one pointer of overhead per object. Subsequent virtual functions add no additional per-object cost.
