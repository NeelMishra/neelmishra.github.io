      std::tie(id, score, label) = data;  // verbose, separated declaration
      std::string label;double score;int id;std::tuple<int, double, std::string> data = getRecord();// Pre-C++17: extracting from a tuple} int count = it->second; // ...and "second"? std::string name = it->first; // what is "first" again?if (it != myMap.end()) {std::map<std::string, int>::iterator it = myMap.find("key");

      Before C++17, extracting individual elements from a struct, pair, or tuple was verbose and error-prone. You had to either use std::get<N> with numeric indices, call named accessors like .first and .second, or use std::tie with pre-declared variables. Each approach had drawbacks: numeric indices obscured intent, .first/.second only worked for pairs, and std::tie required declaring variables before the binding, adding visual noise and separating declaration from initialization.

        Structured Bindings (C++17)

        March 2026 · 16 min read

        Why Structured Bindings Exist

        Before C++17, extracting multiple values from a function return or a container element was verbose and error-prone. Suppose you had a function returning a std::pair:

        std::pair<std::string, int> lookupUser(int id);

// C++14 way
auto result = lookupUser(42);
std::string name = result.first;   // meaningless .first
int age = result.second;           // meaningless .second

// Or with std::tie
std::string name;
int age;
std::tie(name, age) = lookupUser(42);  // requires pre-declaration

        Both approaches are clunky. The .first / .second names carry no semantic meaning. std::tie requires declaring variables before use, and it cannot create const bindings or deduce types automatically.

        C++17 introduced structured bindings to solve this cleanly:

        auto [name, age] = lookupUser(42);  // clean, readable, type-deduced

        One line. The types are deduced. The names are meaningful. This feature looks simple on the surface, but it has surprising depth when you dig into how it actually works, what it can bind to, and how to customize it for your own types.

        Basic Syntax

        The general form of a structured binding declaration is:

        auto [x, y, z] = expression;

        The keyword auto is mandatory (though it can be qualified with const, &, or &&). The names inside the brackets become the bindings. The number of names must exactly match the number of elements in the source.

        auto [a, b] = std::make_pair(1, 3.14);       // a is int, b is double
auto [x, y, z] = std::make_tuple(1, "hi", 2.5); // three bindings
auto& [key, val] = *myMap.begin();             // reference bindings

        Binding to Arrays

        Structured bindings work with C-style arrays and std::array. Each binding corresponds to one element of the array:

        int arr[3] = {10, 20, 30};
auto [a, b, c] = arr;  // a=10, b=20, c=30 (copies)

std::array<double, 2> coords = {1.5, 2.5};
auto [x, y] = coords;  // x=1.5, y=2.5

        The number of identifiers in the brackets must exactly match the array size. If you try auto [a, b] = arr; with a 3-element array, the compiler rejects it.

        When you use auto (without a reference), you get copies of the array elements. With auto&, you get references into the original array:

        int arr[3] = {10, 20, 30};
auto& [a, b, c] = arr;
a = 99;  // modifies arr[0]
// arr is now {99, 20, 30}

        Binding to Structs and Classes

        Structured bindings can decompose any class or struct whose non-static data members are all public and that has no base classes with non-static data members. In other words, it works with simple aggregate types:

        struct Point {
    double x;
    double y;
    double z;
};

Point getOrigin() { return {0.0, 0.0, 0.0}; }

auto [x, y, z] = getOrigin();  // binds to x, y, z members

        The bindings map to the members in declaration order. The first identifier binds to the first member, the second to the second, and so on. Member names do not need to match the binding names:

        struct Result {
    int error_code;
    std::string message;
};

auto [code, msg] = Result{404, "Not Found"};
// code = 404, msg = "Not Found"

        This works because the compiler knows the layout of the struct at compile time and generates code to access each member directly.

        Key insight: Structured bindings to structs require all non-static data members to be public and declared in the same class (not spread across base and derived classes). Private members or inheritance hierarchies require the tuple-like customization protocol instead.

        Binding to Tuple-Like Types

        std::pair, std::tuple, and any type that provides the tuple-like protocol can be decomposed with structured bindings. The standard library types work out of the box:

        std::tuple<int, double, std::string> getData() {
    return {42, 3.14, "hello"};
}

auto [id, value, label] = getData();
// id: int = 42, value: double = 3.14, label: string = "hello"

        For std::map, each element is a std::pair<const Key, Value>, so structured bindings decompose it naturally:

        std::map<std::string, int> scores = {{"Alice", 95}, {"Bob", 87}};

for (auto& [name, score] : scores) {
    std::cout << name << ": " << score << "\n";
}

        This replaces the old pattern of it->first and it->second with readable, meaningful names.

        const and Reference Qualified Bindings

        The qualifiers on auto affect the hidden variable (which we will discuss shortly) and therefore all bindings:

        auto [a, b] = pair;         // copy: a and b are independent copies
auto& [a, b] = pair;        // lvalue reference: a and b refer to pair's members
const auto& [a, b] = pair;  // const lvalue reference: read-only access
auto&& [a, b] = getPair();  // forwarding reference: binds to rvalue or lvalue

        The choice matters for performance and semantics. For iterating over a map without copies:

        // Bad: copies every key and value
for (auto [key, val] : myMap) { ... }

// Good: references into the map, no copies
for (const auto& [key, val] : myMap) { ... }

// Good: mutable references when you need to modify values
for (auto& [key, val] : myMap) { val *= 2; }
        Key insight: The const and & qualifiers apply to the hidden variable that the compiler creates, not individually to each binding. You cannot have one binding be const and another be mutable in the same declaration.

        Under the Hood: The Hidden Variable

        This is where structured bindings get interesting. What the compiler actually does is not simply create the named variables directly. Instead, it creates a hidden, unnamed variable that holds (or references) the entire decomposed object, and the binding names become aliases (references) into that hidden variable.

        When you write:

        auto [x, y] = getPoint();

        The compiler conceptually generates something like this:

        auto __hidden = getPoint();   // copy the returned Point
// x is declared as referring to __hidden.x
// y is declared as referring to __hidden.y

        The identifiers x and y are not independent variables. They are references (in the language-lawyer sense, "names that refer to") parts of the hidden variable. This distinction has practical consequences:

        auto [x, y] = getPoint();
decltype(x);  // This is double (the type of the member), not double&
// But x still refers to __hidden.x, not a separate object

&x == &(__hidden.x);  // true: x IS __hidden.x

        When you add a reference qualifier, the hidden variable itself becomes a reference:

        Point p{1.0, 2.0};
auto& [x, y] = p;
// Conceptually: auto& __hidden = p;
// x refers to __hidden.x, which is p.x
x = 99.0;   // modifies p.x

        This design means that structured bindings never slice or partially copy an object. The entire object is held in the hidden variable, and the bindings are just windows into its members.

        ▶ Structured Binding Decomposition

        Step through how the compiler decomposes a struct: creating the hidden variable and binding references into it.

        Bitfield Bindings

        Structured bindings can even bind to bitfield members, but with an important caveat: since you cannot take a reference to a bitfield, the binding always produces a copy of the bitfield value. You cannot use auto& to get a reference to a bitfield member.

        struct Flags {
    unsigned int readable : 1;
    unsigned int writable : 1;
    unsigned int executable : 1;
    unsigned int reserved : 5;
};

Flags f{1, 0, 1, 0};
auto [r, w, x, res] = f;   // OK: copies of bitfield values
// auto& [r, w, x, res] = f;  // ERROR: cannot bind reference to bitfield

        The copied values are promoted to their underlying type (here unsigned int), so r will be an unsigned int with value 1, not a single bit.

        Structured Bindings in Range-For Loops

        The most common use of structured bindings in practice is inside range-for loops, especially when iterating over associative containers:

        std::unordered_map<std::string, std::vector<int>> courseGrades;

// Iterate with meaningful names instead of .first/.second
for (const auto& [course, grades] : courseGrades) {
    double avg = std::accumulate(grades.begin(), grades.end(), 0.0) / grades.size();
    std::cout << course << ": " << avg << "\n";
}

        This pattern also works beautifully with std::map::insert and std::map::emplace, which return a pair<iterator, bool>:

        auto [it, inserted] = myMap.insert({"key", 42});
if (!inserted) {
    std::cout << "Key already existed with value " << it->second << "\n";
}

        And with std::set::insert:

        auto [it, success] = mySet.insert(value);
if (success) {
    std::cout << "Inserted: " << *it << "\n";
}

        Another powerful pattern is iterating with index using a helper:

        // C++20: enumerate-like pattern with views::zip
// But even in C++17, structured bindings + map iteration is everyday code
std::vector<std::string> names = {"Alice", "Bob", "Charlie"};
int i = 0;
for (auto&& [idx, name] : std::views::enumerate(names)) {  // C++23
    std::cout << idx << ": " << name << "\n";
}

        Structured Bindings with if-init (C++17)

        C++17 also introduced if statements with initializers, and structured bindings combine with them elegantly:

        if (auto [it, inserted] = cache.try_emplace(key, computeValue(key)); !inserted) {
    // key already existed, it points to existing element
    log("Cache hit for", key);
}

if (auto [pos, found] = binarySearch(data, target); found) {
    process(data[pos]);
}

        This keeps the iterator and success flag scoped to the if block, preventing accidental use outside the intended context. It is a strict improvement over the old pattern of declaring temporaries in the enclosing scope.

        Customization: Making Your Types Decomposable

        For types that are not simple aggregates (e.g., classes with private members or non-trivial structure), you can opt into structured bindings by providing the tuple-like protocol. This requires three customization points:

        1. std::tuple_size<T>: specialization that reports the number of elements.
        2. std::tuple_element<I, T>: specialization that reports the type of element I.
        3. get<I>(t): function (member or non-member ADL-found) that returns element I.

        Here is a complete example. Suppose you have a Color class with private members:

        class Color {
    uint8_t r_, g_, b_, a_;
public:
    Color(uint8_t r, uint8_t g, uint8_t b, uint8_t a = 255)
        : r_(r), g_(g), b_(b), a_(a) {}

    template<std::size_t I>
    auto get() const {
        if constexpr (I == 0) return r_;
        else if constexpr (I == 1) return g_;
        else if constexpr (I == 2) return b_;
        else if constexpr (I == 3) return a_;
    }
};

// Specializations in namespace std
namespace std {
    template<>
    struct tuple_size<Color> : integral_constant<size_t, 4> {};

    template<size_t I>
    struct tuple_element<I, Color> {
        using type = uint8_t;
    };
}

// Now this works:
Color c{255, 128, 0};
auto [r, g, b, a] = c;  // r=255, g=128, b=0, a=255

        The get function can be either a member function template (as above) or a free function found via ADL. The compiler checks for a member get first; if that fails, it looks for a non-member get in the associated namespaces.

        For the non-member approach:

        namespace mylib {
    class Color { /* ... same private members ... */ };

    template<std::size_t I>
    auto get(const Color& c) {
        if constexpr (I == 0) return c.red();
        else if constexpr (I == 1) return c.green();
        else if constexpr (I == 2) return c.blue();
        else if constexpr (I == 3) return c.alpha();
    }
}

        This is the same protocol that std::tuple itself implements, so you are piggy-backing on existing infrastructure.

        get() Return Types and Reference Semantics

        The return type of get<I> determines whether the binding is a reference or a copy. If get<I> returns a reference, the binding acts as a reference into the object:

        class MyPair {
    int first_;
    std::string second_;
public:
    MyPair(int a, std::string b) : first_(a), second_(std::move(b)) {}

    template<std::size_t I>
    auto& get() & {
        if constexpr (I == 0) return first_;
        else return second_;
    }

    template<std::size_t I>
    const auto& get() const& {
        if constexpr (I == 0) return first_;
        else return second_;
    }

    template<std::size_t I>
    auto&& get() && {
        if constexpr (I == 0) return std::move(first_);
        else return std::move(second_);
    }
};

// With proper tuple_size/tuple_element specializations:
MyPair mp{42, "hello"};
auto& [a, b] = mp;
a = 99;  // modifies mp.first_ because get returns a reference

        Providing all three ref-qualified overloads of get ensures that structured bindings work correctly whether the source object is an lvalue, const lvalue, or rvalue.

        Limitations and Gotchas

        Structured bindings are not without restrictions:

        • No nested destructuring: You cannot write auto [a, [b, c]] = expr; to decompose nested structures in one go. Each level requires its own binding.
        • No ignoring elements: Unlike Python's _ convention, every element must be bound. You cannot skip elements you do not care about (though you can use a throwaway name like _ or unused).
        • No static or thread_local: The hidden variable cannot be declared static or thread_local.
        • Must match count exactly: The number of identifiers must equal the number of elements. No more, no less.
        • Lambda captures (before C++20): In C++17, structured binding names cannot be captured by lambdas. C++20 fixed this.
        // C++17: this is ill-formed
auto [x, y] = getPoint();
auto lambda = [x]{ return x * 2; };  // ERROR in C++17

// C++20: this works
auto [x, y] = getPoint();
auto lambda = [x]{ return x * 2; };  // OK in C++20

        The lambda capture restriction in C++17 was one of the more annoying limitations and was widely considered a defect that C++20 corrected.

        decltype Behavior

        decltype on a structured binding name gives you the type of the referred-to member, not a reference type:

        std::pair<int, double> p{1, 2.5};
auto [a, b] = p;

decltype(a);  // int, not int& (even though a aliases __hidden.first)
decltype(b);  // double

        This is a deliberate design choice. Even though a technically refers to part of the hidden variable, decltype reports the type of the member itself. This makes structured bindings behave intuitively for generic code that inspects types.

        However, for tuple-like bindings, decltype gives you the return type of get<I>:

        std::tuple<int, double&> t{1, someDouble};
auto [a, b] = t;
decltype(a);  // int
decltype(b);  // double& (because tuple_element gives double&)

        Real-World Patterns

        Structured bindings shine in everyday code. Here are patterns you will use constantly:

        Checking map insertion

        auto [it, ok] = config.try_emplace("timeout", 30);
if (!ok) {
    spdlog::warn("Config key 'timeout' already set to {}", it->second);
}

        Multiple return values

        struct ParseResult {
    bool success;
    std::string_view remaining;
    Token token;
};

auto [ok, rest, tok] = parseNextToken(input);
if (ok) {
    tokens.push_back(tok);
    input = rest;
}

        Decomposing node-based containers

        // C++17 node handle API
auto node = source.extract("key");
if (!node.empty()) {
    node.key() = "new_key";
    auto [pos, inserted, nh] = dest.insert(std::move(node));
}

        Iterating with structure

        struct Employee {
    std::string name;
    std::string department;
    int salary;
};

std::vector<Employee> team = loadTeam();

for (const auto& [name, dept, salary] : team) {
    if (dept == "Engineering" && salary > 100000) {
        highEarners.push_back(name);
    }
}

        Comparison with Other Languages

        Structured bindings are C++'s answer to destructuring, a feature common in many languages:

        • JavaScript: const {name, age} = person; and const [a, b] = arr;. JS destructuring is more flexible (can skip elements, provide defaults, nest), but resolves at runtime.
        • Python: name, age = get_person(). Python's unpacking is dynamic and can handle variable-length sequences with *rest.
        • Rust: let (x, y) = get_point(); and pattern matching with let Point { x, y } = p;. Rust's destructuring is more powerful with nested patterns and irrefutable/refutable pattern matching.

        C++ structured bindings are more limited than most of these (no nesting, no skipping), but they are fully resolved at compile time with zero runtime cost, and they integrate seamlessly with C++'s type system and value semantics.

        Deep Dive: What the Standard Actually Says

        The C++ standard defines structured bindings in terms of three cases, checked in order:

        1. Case 1: Arrays. If the initializer is an array type, each binding refers to an element of the (possibly copied) array.
        2. Case 2: Tuple-like types. If std::tuple_size<E> is a complete type (where E is the type of the initializer), the tuple protocol is used: get<I> extracts each element.
        3. Case 3: Data members. Otherwise, each binding refers to a public non-static data member, in declaration order.

        Case 2 takes priority over Case 3. This means that if you specialize tuple_size for a type that also has public members, the tuple protocol wins. This can be surprising if you accidentally specialize tuple_size for a simple struct.

        The hidden variable is created with the exact cv-qualifiers and reference qualifiers from the declaration. For auto [x, y] = expr;, the hidden variable is auto __e = expr; (a copy). For auto& [x, y] = expr;, it is auto& __e = expr; (a reference). The binding names are then defined in terms of __e.

        Performance Considerations

        Structured bindings are a purely compile-time feature. The generated code is identical to what you would write manually. There is no hidden overhead, no additional function calls, no heap allocations.

        For aggregate bindings, the compiler generates direct member accesses. For tuple-like bindings, the get<I> calls are typically inlined completely. In optimized builds, structured bindings produce the same assembly as hand-written member access.

        The one performance pitfall is using auto (by value) when you meant auto& (by reference). Copying a large struct every iteration of a loop is wasteful. Always consider whether you need a copy or a reference.

        // Potential performance issue: copies the string each iteration
for (auto [name, value] : config) { ... }

// Better: reference avoids copies
for (const auto& [name, value] : config) { ... }

        Summary

        • Structured bindings let you decompose arrays, structs, pairs, tuples, and custom types into named variables in a single declaration.
        • They work by creating a hidden variable and making the binding names refer to parts of it.
        • Use auto& or const auto& to avoid copies when iterating or binding to existing objects.
        • Customize decomposition for your own types via tuple_size, tuple_element, and get.
        • Bitfield members are always copied (no reference binding).
        • C++20 fixed lambda capture of structured bindings.
        • Zero runtime overhead: the compiler generates optimal code identical to manual member access.
