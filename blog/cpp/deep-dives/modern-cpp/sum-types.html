    Sum Types: optional, variant, any

    March 2026 · 22 min read

    What Are Sum Types?

    A sum type is a type that can hold one value from a set of possible types, but only one at a time. If a product type (like a struct) holds values of types A and B simultaneously, a sum type holds a value of type A or type B. The total number of possible values is the sum of the possible values of each constituent type, hence the name.

    C has had a primitive sum type since the beginning: the union. But unions are dangerous because there is no built-in mechanism to track which member is currently active. Reading the wrong member is undefined behavior. C++17 introduced three type-safe sum types that solve this problem at different levels of generality:

    • std::optional<T>: a value that is either a T or nothing.
    • std::variant<Ts...>: a value that is exactly one of the types Ts....
    • std::any: a value that can hold any single type (type-erased).

    std::optional: Nullable Values Without Pointers

    Before std::optional, C++ programmers had several awkward ways to represent "might not have a value": returning a pointer (and checking for null), using a sentinel value (like -1 or an empty string), or returning a bool alongside an out-parameter. All of these have problems: pointers conflate nullability with heap allocation, sentinels are error-prone, and out-parameters are verbose and prevent const.

    std::optional<T> solves this cleanly. It is a stack-allocated wrapper that either contains a T or is empty (disengaged):

    #include <optional>

std::optional<int> findIndex(const std::vector<int>& v, int target) {
    for (size_t i = 0; i < v.size(); ++i) {
        if (v[i] == target) return static_cast<int>(i);
    }
    return std::nullopt;  // no value
}

auto idx = findIndex(data, 42);
if (idx) {
    std::cout << "Found at index " << *idx << "\n";
} else {
    std::cout << "Not found\n";
}

    Creating and Accessing Optionals

    std::optional<std::string> a;              // disengaged (empty)
std::optional<std::string> b = std::nullopt; // same
std::optional<std::string> c = "hello";      // engaged, holds "hello"
std::optional<std::string> d{std::in_place, 5, 'x'};  // holds "xxxxx"

// Check engagement
if (c.has_value()) { /* ... */ }
if (c) { /* ... */ }               // contextual bool conversion

// Access
std::string s1 = *c;              // UB if disengaged
std::string s2 = c.value();       // throws bad_optional_access if disengaged
std::string s3 = c.value_or("default");  // safe fallback

    Optional is Not a Pointer

    A common misconception is that std::optional is like a smart pointer. It is not. The contained value is stored inline, inside the optional object itself (typically using aligned storage and a boolean flag). There is no heap allocation. sizeof(std::optional<int>) is typically 8 bytes (4 for the int + 1 for the bool + 3 padding).

    // Approximate layout:
// struct optional<T> {
//     alignas(T) unsigned char storage[sizeof(T)];
//     bool engaged;
// };

    This means copying an optional copies the contained value. Moving an optional moves the contained value. Destroying an optional destroys the contained value (if engaged). The optional fully owns its value semantically and physically.

    Key insight: std::optional<T> has value semantics, not reference semantics. It stores T inline, so copies are deep copies. Use it as a return type and local variable, not as a pointer replacement for heap-allocated objects.

    Comparison and Ordering

    Optionals support comparison operators. A disengaged optional compares less than any engaged optional. Two disengaged optionals are equal. Two engaged optionals compare their contained values:

    std::optional<int> a = 5;
std::optional<int> b = 10;
std::optional<int> c;

a < b;    // true (5 < 10)
c < a;    // true (empty < engaged)
c == std::nullopt;  // true

    You can also compare an optional directly with its contained type:

    std::optional<int> x = 42;
x == 42;       // true
x < 100;       // true
x > std::nullopt;  // true

    Monadic Operations on optional (C++23)

    C++23 added three monadic operations to std::optional that enable functional-style chaining without explicit null checks at each step. These are transformative for code readability:

    transform (map)

    Applies a function to the contained value if present, wraps the result in an optional. If disengaged, returns an empty optional of the transformed type:

    std::optional<std::string> name = getUsername(id);
std::optional<size_t> len = name.transform([](const std::string& s) {
    return s.length();
});
// If name is "Alice", len is 5. If name is nullopt, len is nullopt.

    and_then (flat_map / bind)

    Like transform, but the function itself returns an optional. This prevents nested optionals (optional<optional<T>>):

    std::optional<User> getUser(int id);
std::optional<Address> getAddress(const User& u);
std::optional<std::string> getCity(const Address& a);

// Chain without nested optionals
std::optional<std::string> city = getUser(42)
    .and_then(getAddress)
    .and_then(getCity);
// If any step returns nullopt, the whole chain returns nullopt.

    or_else (recover)

    If the optional is disengaged, calls a function that returns an optional to provide a fallback. If engaged, returns the original optional unchanged:

    std::optional<Config> config = loadFromFile("config.json")
    .or_else([]() { return loadFromFile("config.default.json"); })
    .or_else([]() { return std::optional<Config>{Config::defaults()}; });

    Chaining Everything Together

    auto result = findUser(name)
    .and_then([](const User& u) { return u.primaryEmail(); })
    .transform([](const Email& e) { return e.domain(); })
    .or_else([]() -> std::optional<std::string> { return "unknown"; });
// result is always engaged: either the domain or "unknown".

    std::variant: Type-Safe Union

    Where std::optional<T> is "T or nothing", std::variant<Ts...> is "one of Ts...". It is a type-safe, stack-allocated discriminated union. At any point in time, a variant holds exactly one of its alternative types, and it knows which one.

    #include <variant>

std::variant<int, double, std::string> v;    // default: first type (int), value 0
v = 3.14;                                      // now holds double
v = "hello";                                   // now holds string
v = 42;                                        // now holds int

    Querying and Accessing

    std::variant<int, double, std::string> v = 3.14;

v.index();                    // 1 (double is at index 1)
std::holds_alternative<double>(v);   // true
std::holds_alternative<int>(v);      // false

std::get<double>(v);         // 3.14
std::get<1>(v);              // 3.14 (by index)
std::get<int>(v);            // throws std::bad_variant_access

auto* p = std::get_if<double>(&v);   // pointer to value, or nullptr
if (p) {
    std::cout << *p << "\n";
}

    Variant Layout and Size

    A variant stores its value inline (like optional) plus a discriminator (index) that tracks which type is active. The size is roughly the size of the largest alternative plus the discriminator:

    sizeof(std::variant<int, double, std::string>)
// Typically: max(sizeof(int), sizeof(double), sizeof(string)) + discriminator + padding
// On 64-bit with libstdc++: often 40 bytes (string=32, discriminator=4, padding=4)

    Variant vs C Union

    A C-style union gives you the raw storage but no safety. You must manually track which member is active. Reading the wrong member is UB. Assigning a new member does not call destructors on the old one. std::variant handles all of this automatically:

    • Assigning a new type destroys the old value first.
    • The discriminator is always in sync.
    • std::get throws if you access the wrong type.
    • Copying, moving, and destroying the variant does the right thing for the active member.

    The valueless_by_exception State

    If an assignment throws during the construction of the new value (after the old value has already been destroyed), the variant enters a special state called valueless_by_exception. In this state, index() returns variant_npos and all access functions throw. This state is rare in practice but important to know about:

    struct Throwy {
    Throwy(int) { throw std::runtime_error("boom"); }
};

std::variant<int, Throwy> v = 42;
try {
    v.emplace<Throwy>(1);  // destroys int, then Throwy ctor throws
} catch (...) {
    v.valueless_by_exception();  // true
    v.index();  // variant_npos
}
    Key insight: Use emplace carefully with types that might throw during construction. The valueless_by_exception state is the variant's answer to the question "what happens if construction fails after destruction?" It is ugly but safe: at least you get an exception rather than UB.

    std::visit: The Visitor Pattern for Variants

    std::visit is the primary way to work with variants. It takes a callable (the "visitor") and one or more variants, and dispatches to the correct overload based on the active type:

    std::variant<int, double, std::string> v = "hello";

std::visit([](auto&& val) {
    using T = std::decay_t<decltype(val)>;
    if constexpr (std::is_same_v<T, int>) {
        std::cout << "int: " << val << "\n";
    } else if constexpr (std::is_same_v<T, double>) {
        std::cout << "double: " << val << "\n";
    } else {
        std::cout << "string: " << val << "\n";
    }
}, v);

    But writing if constexpr chains is verbose. The idiomatic approach is the overloaded visitor pattern.

    The Overloaded Pattern

    Using a small utility struct that inherits from multiple lambdas:

    template<class... Ts>
struct overloaded : Ts... { using Ts::operator()...; };

// C++17 deduction guide (not needed in C++20)
template<class... Ts>
overloaded(Ts...) -> overloaded<Ts...>;

    Now you can write visitors as a set of lambdas, one per type:

    std::variant<int, double, std::string> v = 3.14;

std::visit(overloaded{
    [](int i)              { std::cout << "int: " << i << "\n"; },
    [](double d)           { std::cout << "double: " << d << "\n"; },
    [](const std::string& s) { std::cout << "string: " << s << "\n"; }
}, v);

    This is clean, type-safe, and the compiler verifies that every alternative is handled. If you add a new type to the variant and forget to add a handler, you get a compile error.

    Visiting Multiple Variants

    std::visit can take multiple variants simultaneously, visiting the Cartesian product of their active types:

    std::variant<int, std::string> a = 42;
std::variant<double, bool> b = true;

std::visit(overloaded{
    [](int i, double d)         { std::cout << i << " + " << d << "\n"; },
    [](int i, bool b)           { std::cout << i << " + " << b << "\n"; },
    [](const std::string& s, double d) { std::cout << s << " + " << d << "\n"; },
    [](const std::string& s, bool b)   { std::cout << s << " + " << b << "\n"; }
}, a, b);

    How std::visit Works

    Under the hood, std::visit typically uses a function pointer table (similar to a vtable). For a variant with N alternatives, the implementation creates an N-element array of function pointers, one per alternative. The variant's index selects the correct entry. For multiple variants, the table is multidimensional.

    This means std::visit is an O(1) dispatch: one index lookup and one indirect function call. It is very efficient, comparable to a virtual function call.

    Practical Variant Patterns

    State Machines

    Variants are excellent for modeling state machines where each state has different associated data:

    struct Idle {};
struct Connecting { std::string host; int port; };
struct Connected { Socket socket; };
struct Error { std::string message; };

using ConnectionState = std::variant<Idle, Connecting, Connected, Error>;

ConnectionState state = Idle{};

// Transition
state = Connecting{"example.com", 443};

// Handle current state
std::visit(overloaded{
    [](const Idle&)        { showSpinner(false); },
    [](const Connecting& c) { showSpinner(true); log("Connecting to " + c.host); },
    [](const Connected& c)  { showSpinner(false); enableUI(); },
    [](const Error& e)      { showSpinner(false); showError(e.message); }
}, state);

    Expression Trees

    struct Literal { double value; };
struct Add;
struct Multiply;

using Expr = std::variant<
    Literal,
    std::unique_ptr<Add>,
    std::unique_ptr<Multiply>
>;

struct Add { Expr left, right; };
struct Multiply { Expr left, right; };

double evaluate(const Expr& expr) {
    return std::visit(overloaded{
        [](const Literal& lit) { return lit.value; },
        [](const std::unique_ptr<Add>& a) {
            return evaluate(a->left) + evaluate(a->right);
        },
        [](const std::unique_ptr<Multiply>& m) {
            return evaluate(m->left) * evaluate(m->right);
        }
    }, expr);
}

    Error Handling

    template<typename T, typename E>
using Result = std::variant<T, E>;

struct ParseError { int line; std::string message; };

Result<Config, ParseError> parseConfig(std::string_view input) {
    if (input.empty()) return ParseError{0, "Empty input"};
    // ... parse logic ...
    return Config{/* ... */};
}

auto result = parseConfig(text);
std::visit(overloaded{
    [](const Config& c)    { applyConfig(c); },
    [](const ParseError& e) { log("Parse error at line {}: {}", e.line, e.message); }
}, result);

    std::any: Type-Safe void*

    std::any can hold a value of any copyable type. Unlike variant, it does not require you to enumerate the possible types at compile time. It is the type-safe replacement for void*:

    #include <any>

std::any a = 42;                  // holds int
a = std::string("hello");        // now holds string
a = 3.14;                        // now holds double

// Access
double d = std::any_cast<double>(a);       // 3.14
// std::any_cast<int>(a);                  // throws bad_any_cast

auto* p = std::any_cast<double>(&a);      // pointer, or nullptr on mismatch
if (p) { std::cout << *p << "\n"; }

a.type();  // returns typeid(double)
a.has_value();  // true
a.reset();      // makes it empty

    How std::any Works

    std::any uses type erasure internally. For small types (typically up to the size of a pointer or so), it stores the value inline (small buffer optimization, SBO). For larger types, it allocates on the heap. The type information is tracked via an internal function pointer or vtable-like mechanism.

    // Typical internal layout (simplified):
// struct any {
//     void* storage;      // or inline buffer
//     const type_info* ti; // or manager function pointer
// };

    When to Use std::any

    std::any is the most heavyweight of the three sum types. Use it only when:

    • The set of possible types is not known at compile time (e.g., plugin systems, property maps).
    • You need a heterogeneous container: std::vector<std::any>.
    • You are implementing a framework that stores user-defined types opaquely.

    In most cases, std::variant is preferable because it provides compile-time type safety and exhaustiveness checking.

    Key insight: Prefer std::optional when you need "T or nothing". Prefer std::variant when you know the set of possible types at compile time. Use std::any only when the type set is truly open-ended.

    Comparison to Tagged Unions

    Before C++17, the standard approach was to build your own tagged union:

    enum class Tag { Integer, Float, String };

struct Value {
    Tag tag;
    union {
        int i;
        double d;
        std::string s;   // problematic: non-trivial type in union
    };

    ~Value() {
        if (tag == Tag::String) s.~basic_string();
    }
    // Must also handle copy/move ctor, copy/move assignment...
};

    This is painful: you must manually manage lifetimes of non-trivial types in the union, implement all special member functions correctly, and maintain the tag invariant yourself. One mistake leads to UB.

    std::variant eliminates all of this boilerplate. It handles constructors, destructors, copy/move operations, and the discriminator automatically. The compiler-generated code is as efficient as the hand-written version (often better, since the compiler knows the exact semantics).

    variant vs Inheritance Polymorphism

    Variant and virtual base classes solve similar problems (dispatching to the right code based on the runtime type of a value) but with different tradeoffs:

    Aspect std::variant Virtual base class
    Type set Closed (fixed at compile time) Open (new derived classes anytime)
    Adding a new type Recompile all visitors Just add a new derived class
    Adding a new operation Just add a new visitor Modify every derived class
    Storage Inline (stack), no heap allocation Typically heap-allocated via pointers
    Exhaustiveness Compile-time checked (with visit) No compile-time guarantee
    Value semantics Yes (copyable, movable) No (reference/pointer semantics)

    The classic guidance: use variant when the type set is closed and you frequently add operations. Use inheritance when the type set is open and the operation set is stable.

    Advanced Variant Techniques

    Variant with std::monostate

    If the first type in a variant has no default constructor, the variant itself cannot be default-constructed. Use std::monostate as the first alternative to provide a "none" state:

    struct NonDefault {
    NonDefault(int) {}  // no default ctor
};

// std::variant<NonDefault, int> v;  // ERROR: NonDefault not default-constructible

std::variant<std::monostate, NonDefault, int> v;  // OK: defaults to monostate

    std::monostate is an empty struct with comparison operators. It serves as a placeholder for "no value" when you need a default-constructible first alternative.

    Returning variant from functions

    using JsonValue = std::variant<
    std::nullptr_t,    // null
    bool,              // boolean
    double,            // number
    std::string,       // string
    std::vector<JsonValue>,  // array (requires recursive type handling)
    std::map<std::string, JsonValue>  // object
>;

    Note: recursive variant definitions require indirection (e.g., std::unique_ptr or a wrapper) because the variant must have a known size at definition time.

    Advanced optional: Reference and Void

    std::optional<T&> is not allowed in C++17 or C++20 (unlike Boost.Optional, which supports it). If you need an optional reference, use a pointer:

    // Not valid: std::optional<int&>
// Use instead:
int* findInMap(std::map<int,int>& m, int key) {
    auto it = m.find(key);
    return it != m.end() ? &it->second : nullptr;
}

    There is an ongoing proposal (P2988) to add std::optional<T&> to a future standard. For now, T* or std::reference_wrapper<T> with optional are the alternatives.

    Performance Characteristics

    • std::optional: Zero overhead when engaged. One extra byte (bool) plus alignment padding. No heap allocation ever.
    • std::variant: Size of largest alternative plus discriminator. No heap allocation. std::visit is one indirect call (comparable to virtual dispatch).
    • std::any: Small buffer optimization for small types (pointer-sized or less). Heap allocation for larger types. any_cast involves a type check (comparing type_info).

    In performance-critical code, optional and variant are the clear winners. any should be reserved for cases where its flexibility is genuinely needed.

    Summary

    • std::optional<T> is "T or nothing", stored inline, with value semantics. C++23 adds monadic operations (and_then, transform, or_else) for functional chaining.
    • std::variant<Ts...> is a type-safe discriminated union. Use std::visit with the overloaded pattern for clean, exhaustive dispatch.
    • std::any is type-erased storage for any copyable type. Prefer variant when the type set is known.
    • Variant excels at state machines, expression trees, and closed-set polymorphism with value semantics.
    • The overloaded visitor idiom (overloaded{lambdas...}) is the standard way to handle variant alternatives.
