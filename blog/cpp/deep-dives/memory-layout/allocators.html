    Custom Allocators

    March 2026 · 22 min read

    Why Custom Allocators?

    The default heap allocator (malloc/new) is a general-purpose tool. It handles any allocation size, any deallocation order, and any thread. That generality comes at a cost: each call involves bookkeeping, potential system calls, mutex contention, and cache-unfriendly memory layouts.

    When you know something about your allocation pattern—all objects are the same size, or allocations happen in a burst and are freed all at once, or allocations only happen on one thread—you can exploit that knowledge with a custom allocator that is dramatically faster.

    Game engines, trading systems, embedded systems, and database engines all use custom allocators to achieve deterministic, low-latency memory management. In this article, we will build up from the std::allocator interface through polymorphic memory resources (pmr) and implement four allocator strategies from scratch.

    The std::allocator Interface

    Every STL container takes an allocator as a template parameter. The default is std::allocator<T>, which calls ::operator new and ::operator delete:

    template<typename T>
class allocator {
public:
    using value_type = T;

    T* allocate(std::size_t n) {
        return static_cast<T*>(::operator new(n * sizeof(T)));
    }

    void deallocate(T* p, std::size_t n) {
        ::operator delete(p);
    }
};

// Usage is implicit:
std::vector<int> v;  // uses std::allocator<int> by default
std::vector<int, MyAlloc<int>> v2;  // custom allocator

    Allocator Requirements

    A conforming allocator must provide:

    • value_type typedef
    • allocate(n) returning a pointer to uninitialized storage for n objects
    • deallocate(p, n) releasing that storage
    • Equality comparable (two allocators are equal if memory from one can be deallocated by the other)

    Since C++17, the requirements have been simplified. Many older requirements (rebind, construct, destroy) are now handled by std::allocator_traits defaults.

    Key insight: Allocators allocate raw memory. They do not construct objects. The container calls std::allocator_traits<Alloc>::construct() separately, which by default uses placement new.

    Polymorphic Memory Resources (C++17)

    The classic allocator model has a major usability problem: the allocator type is baked into the container type. A vector<int, ArenaAlloc<int>> is a different type from vector<int, PoolAlloc<int>>. You cannot assign one to the other, pass one to a function expecting the other, or store them in the same collection.

    C++17 introduced <memory_resource> to solve this with runtime polymorphism:

    #include <memory_resource>

// All pmr containers use the same type regardless of allocator:
std::pmr::vector<int> v1;   // uses default resource
std::pmr::vector<int> v2;   // can use a different resource

// The resource is passed at construction:
std::pmr::monotonic_buffer_resource pool(1024);
std::pmr::vector<int> v3(&pool);

// v1, v2, v3 are ALL the same type: std::pmr::vector<int>

    std::pmr::memory_resource

    The base class for all memory resources. It defines three virtual functions:

    class memory_resource {
public:
    void* allocate(std::size_t bytes, std::size_t alignment);
    void  deallocate(void* p, std::size_t bytes, std::size_t alignment);
    bool  is_equal(const memory_resource& other) const noexcept;

private:
    virtual void* do_allocate(std::size_t bytes, std::size_t align) = 0;
    virtual void  do_deallocate(void* p, std::size_t bytes, std::size_t align) = 0;
    virtual bool  do_is_equal(const memory_resource& other) const noexcept = 0;
};

    Standard Memory Resources

    The standard library provides several built-in resources:

    • std::pmr::new_delete_resource() — Calls ::operator new/delete. The default.
    • std::pmr::null_memory_resource() — Always throws bad_alloc. Useful as an upstream sentinel.
    • std::pmr::monotonic_buffer_resource — A fast bump allocator. Never reuses freed memory until the entire resource is destroyed.
    • std::pmr::unsynchronized_pool_resource — A pool allocator for single-threaded use.
    • std::pmr::synchronized_pool_resource — Thread-safe version of the pool resource.

    Arena / Monotonic Allocator

    The simplest and fastest custom allocator. It maintains a large block of memory and a pointer that only moves forward. Deallocation is a no-op; everything is freed at once when the arena is destroyed or reset.

    class Arena {
    char*       buffer_;
    std::size_t capacity_;
    std::size_t offset_;

public:
    explicit Arena(std::size_t capacity)
        : buffer_(static_cast<char*>(std::malloc(capacity)))
        , capacity_(capacity)
        , offset_(0)
    {
        if (!buffer_) throw std::bad_alloc();
    }

    ~Arena() { std::free(buffer_); }

    void* allocate(std::size_t size, std::size_t align = alignof(std::max_align_t)) {
        // Align the current offset
        std::size_t aligned = (offset_ + align - 1) & ~(align - 1);
        if (aligned + size > capacity_) throw std::bad_alloc();
        void* ptr = buffer_ + aligned;
        offset_ = aligned + size;
        return ptr;
    }

    void deallocate(void*, std::size_t) noexcept {
        // No-op: memory is freed when the arena is destroyed
    }

    void reset() noexcept { offset_ = 0; }

    std::size_t used() const noexcept { return offset_; }
    std::size_t remaining() const noexcept { return capacity_ - offset_; }
};

    Performance characteristics:

    • Allocate: O(1). A pointer bump, possibly with alignment adjustment.
    • Deallocate: O(1). Literally a no-op.
    • Reset: O(1). Set the offset back to zero.
    • Memory overhead: Near zero (no per-allocation headers).
    • Cache behaviour: Excellent. Allocations are sequential in memory.
    Key insight: Arena allocators are ideal for request-scoped work: parse a request, do all allocations from the arena, send the response, then reset. Game engines use per-frame arenas for exactly this pattern.

    ▶ Monotonic Arena Allocator

    Watch the bump pointer move forward with each allocation. Deallocations are no-ops until the arena is reset.

    Arena as a pmr Resource

    Wrapping the arena as a std::pmr::memory_resource lets you use it with any pmr container:

    class ArenaResource : public std::pmr::memory_resource {
    Arena arena_;

public:
    explicit ArenaResource(std::size_t capacity) : arena_(capacity) {}

protected:
    void* do_allocate(std::size_t bytes, std::size_t align) override {
        return arena_.allocate(bytes, align);
    }

    void do_deallocate(void* p, std::size_t bytes, std::size_t align) override {
        arena_.deallocate(p, bytes);  // no-op
    }

    bool do_is_equal(const memory_resource& other) const noexcept override {
        return this == &other;
    }
};

// Usage:
ArenaResource resource(1024 * 1024);  // 1 MB arena
std::pmr::vector<std::pmr::string> names(&resource);
names.push_back("Alice");
names.push_back("Bob");

    Pool Allocator

    A pool allocator pre-divides memory into fixed-size blocks. Allocation pops a block from a free list; deallocation pushes it back. Every operation is O(1), and there is zero fragmentation (all blocks are the same size).

    class Pool {
    struct FreeBlock { FreeBlock* next; };

    char*       buffer_;
    std::size_t blockSize_;
    std::size_t blockCount_;
    FreeBlock*  freeList_;

public:
    Pool(std::size_t blockSize, std::size_t blockCount)
        : buffer_(static_cast<char*>(std::malloc(blockSize * blockCount)))
        , blockSize_(blockSize)
        , blockCount_(blockCount)
        , freeList_(nullptr)
    {
        if (!buffer_) throw std::bad_alloc();
        // Build the free list (each block points to the next)
        for (std::size_t i = 0; i < blockCount; ++i) {
            auto* block = reinterpret_cast<FreeBlock*>(buffer_ + i * blockSize);
            block->next = freeList_;
            freeList_ = block;
        }
    }

    ~Pool() { std::free(buffer_); }

    void* allocate() {
        if (!freeList_) throw std::bad_alloc();
        FreeBlock* block = freeList_;
        freeList_ = block->next;
        return block;
    }

    void deallocate(void* ptr) noexcept {
        auto* block = static_cast<FreeBlock*>(ptr);
        block->next = freeList_;
        freeList_ = block;
    }

    std::size_t available() const noexcept {
        std::size_t count = 0;
        for (auto* b = freeList_; b; b = b->next) ++count;
        return count;
    }
};

    Pool allocators excel when you have many objects of the same type being rapidly created and destroyed: network connections, game entities, AST nodes, small message objects.

    Watch blocks get allocated from the free list and returned. Each operation is O(1).

    Stack Allocator

    A stack allocator is like an arena but allows deallocation in LIFO order. You can only free the most recently allocated block. This is perfect for recursive algorithms and temporary buffers that are allocated and freed in strict nesting order.

    class StackAllocator {
    char*       buffer_;
    std::size_t capacity_;
    std::size_t offset_;

    struct Header {
        std::size_t prevOffset;
    };

public:
    explicit StackAllocator(std::size_t capacity)
        : buffer_(static_cast<char*>(std::malloc(capacity)))
        , capacity_(capacity)
        , offset_(0)
    {
        if (!buffer_) throw std::bad_alloc();
    }

    ~StackAllocator() { std::free(buffer_); }

    void* allocate(std::size_t size, std::size_t align = alignof(std::max_align_t)) {
        // Space for header + alignment + payload
        std::size_t headerStart = offset_;
        std::size_t payloadStart = (headerStart + sizeof(Header) + align - 1) & ~(align - 1);
        std::size_t newOffset = payloadStart + size;

        if (newOffset > capacity_) throw std::bad_alloc();

        // Store header just before the aligned payload
        auto* hdr = reinterpret_cast<Header*>(buffer_ + payloadStart - sizeof(Header));
        hdr->prevOffset = offset_;

        offset_ = newOffset;
        return buffer_ + payloadStart;
    }

    void deallocate(void* ptr) noexcept {
        auto* hdr = reinterpret_cast<Header*>(static_cast<char*>(ptr) - sizeof(Header));
        offset_ = hdr->prevOffset;
    }

    void reset() noexcept { offset_ = 0; }
};
    Key insight: Stack allocators combine the speed of arenas with the ability to reclaim memory. The constraint is strict LIFO ordering. Deallocating out of order corrupts the allocator state.

    Free-List Allocator

    The most general custom allocator. It maintains a linked list of free blocks and can handle arbitrary allocation sizes and deallocation orders. It trades some speed for flexibility:

    class FreeListAllocator {
    struct Block {
        std::size_t size;
        Block*      next;
    };

    char*       buffer_;
    std::size_t capacity_;
    Block*      freeList_;

public:
    explicit FreeListAllocator(std::size_t capacity)
        : buffer_(static_cast<char*>(std::malloc(capacity)))
        , capacity_(capacity)
    {
        if (!buffer_) throw std::bad_alloc();
        freeList_ = reinterpret_cast<Block*>(buffer_);
        freeList_->size = capacity;
        freeList_->next = nullptr;
    }

    ~FreeListAllocator() { std::free(buffer_); }

    void* allocate(std::size_t size) {
        // First-fit search
        Block** prev = &freeList_;
        for (Block* curr = freeList_; curr; prev = &curr->next, curr = curr->next) {
            if (curr->size >= size + sizeof(Block)) {
                // Split the block
                auto* alloc = reinterpret_cast<Block*>(
                    reinterpret_cast<char*>(curr) + curr->size - size);
                curr->size -= size + sizeof(Block);
                alloc->size = size;
                alloc->next = nullptr;
                return reinterpret_cast<char*>(alloc) + sizeof(Block);
            }
            if (curr->size >= size) {
                *prev = curr->next;
                return reinterpret_cast<char*>(curr) + sizeof(Block);
            }
        }
        throw std::bad_alloc();
    }

    void deallocate(void* ptr) noexcept {
        auto* block = reinterpret_cast<Block*>(static_cast<char*>(ptr) - sizeof(Block));
        block->next = freeList_;
        freeList_ = block;
        // A production allocator would coalesce adjacent free blocks here
    }
};

    Benchmarking Allocators

    The difference can be dramatic. Here is a typical benchmark allocating and deallocating 1 million 64-byte objects:

    // Pseudocode benchmark results (lower is better):
// Default new/delete:    45 ms
// Pool allocator:         3 ms   (15x faster)
// Arena allocator:        1.5 ms (30x faster)
// Arena + bulk free:      0.8 ms (56x faster)

    The speed comes from three sources:

    1. No system calls: Custom allocators work in pre-allocated memory.
    2. No bookkeeping: Arenas and pools need minimal per-allocation metadata.
    3. Cache locality: Sequential allocation produces sequential memory access patterns.

    Real-World Allocator Usage

    Game Engines

    Game engines typically use a hierarchy of allocators:

    • Global arena: Large initial allocation from the OS at startup.
    • Per-frame arena: Reset every frame. All transient allocations go here.
    • Pool allocators: For specific object types (particles, entities, components).
    • Stack allocator: For the rendering pipeline's temporary buffers.

    Trading Systems

    In high-frequency trading, malloc latency spikes are unacceptable. Pre-allocated pools ensure every allocation completes in constant time with no system calls. Some systems even pre-fault all memory pages at startup to avoid page fault latency.

    Database Engines

    Database query executors use per-query arenas. Each query allocates intermediate results, hash tables, and sort buffers from its arena. When the query completes, the entire arena is released—no individual deallocation needed.

    Common Pitfalls

    • Lifetime mismatch: The allocator must outlive all objects allocated from it. Destroying an arena while references to its memory exist is undefined behaviour.
    • Alignment neglect: Custom allocators must respect alignment requirements. The arena allocator above uses alignment masking; skipping this causes hardware faults on strict architectures.
    • Thread safety: Unless your allocator is explicitly thread-safe, confine it to a single thread. Adding a mutex to a fast allocator can negate all performance gains.
    • Over-engineering: Profile first. If the default allocator is not a bottleneck, a custom one adds complexity for no gain.
    Key insight: Custom allocators shine when allocation patterns are predictable. If your allocations are unpredictable in size and lifetime, the default allocator is usually the right choice—it has decades of optimization behind it.

    Summary

    • std::allocator is the default. It delegates to ::operator new.
    • pmr (C++17) decouples the allocator from the container type, enabling runtime polymorphism.
    • Arena allocators bump a pointer forward. O(1) allocation, zero fragmentation, bulk free.
    • Pool allocators manage fixed-size blocks via a free list. O(1) allocation and deallocation.
    • Stack allocators allow LIFO deallocation, combining arena speed with some reclamation.
    • Free-list allocators handle arbitrary sizes but are slower and can fragment.
    • Custom allocators can provide 10x to 50x speedups in allocation-heavy workloads.
    • Profile before optimizing. Use pmr to switch allocators without changing container types.
