    Alignment & Padding

    March 2026 · 20 min read

    Why Alignment Exists

    CPUs do not read memory one byte at a time. They read in word-sized chunks—typically 4 or 8 bytes at once. When a 4-byte int sits at an address divisible by 4, the CPU can read it in a single operation. When it straddles a word boundary, the CPU must perform two reads and stitch the halves together, or on some architectures, the hardware faults entirely.

    To avoid this penalty, the compiler inserts invisible padding bytes between struct members to ensure each member starts at an address that satisfies its alignment requirement. Understanding these rules lets you write cache-friendly, memory-efficient data structures.

    alignof and alignas

    C++11 introduced two keywords for querying and controlling alignment:

    alignof

    The alignof operator returns the alignment requirement of a type in bytes:

    static_assert(alignof(char)   == 1);
static_assert(alignof(short)  == 2);   // typically
static_assert(alignof(int)    == 4);   // typically
static_assert(alignof(double) == 8);   // typically
static_assert(alignof(void*)  == 8);   // on 64-bit

    The alignment of a struct or class is the largest alignment of any of its non-static data members:

    struct Example {
    char  c;    // alignof(char)   = 1
    int   i;    // alignof(int)    = 4
    double d;   // alignof(double) = 8
};
static_assert(alignof(Example) == 8);  // max(1, 4, 8)

    alignas

    The alignas specifier lets you increase the alignment requirement (you cannot decrease it below the natural alignment):

    struct alignas(64) CacheAligned {
    int data[16];
};
static_assert(alignof(CacheAligned) == 64);
static_assert(sizeof(CacheAligned) == 64);

// Per-member alignment:
struct Mixed {
    alignas(16) float vector[4];   // 16-byte aligned (for SIMD)
    int count;
};
    Key insight: alignas can only increase alignment, never decrease it. alignas(1) int x; is ill-formed because int requires at least 4-byte alignment. The alignas value must be a power of two.

    Struct Padding Rules

    The compiler follows three rules when laying out a struct:

    1. Each member starts at an offset that is a multiple of its alignment. Padding bytes are inserted before the member if necessary.
    2. The total size of the struct is a multiple of the struct's alignment (the max member alignment). Trailing padding is added if necessary.
    3. Members are laid out in declaration order. The compiler never reorders them (unlike some other languages).

    A Concrete Example

    Consider this struct on a typical 64-bit platform:

    struct Bad {
    char  a;     // 1 byte,  offset 0
                 // 3 bytes padding (to align int to offset 4)
    int   b;     // 4 bytes, offset 4
    char  c;     // 1 byte,  offset 8
                 // 7 bytes padding (to align double to offset 16)
    double d;    // 8 bytes, offset 16
};
// sizeof(Bad) = 24 bytes (but only 14 bytes of actual data)

    That is 10 bytes of wasted padding in a 24-byte struct. Over 40% waste.

    sizeof Mysteries Explained

    Why is sizeof(Bad) 24 and not 14? Let's trace byte by byte:

    • Offset 0: char a (1 byte)
    • Offsets 1 to 3: padding (3 bytes, to align int to offset 4)
    • Offsets 4 to 7: int b (4 bytes)
    • Offset 8: char c (1 byte)
    • Offsets 9 to 15: padding (7 bytes, to align double to offset 16)
    • Offsets 16 to 23: double d (8 bytes)

    Total: 24 bytes. The struct alignment is 8 (from double), and 24 is already a multiple of 8, so no trailing padding.

    The Empty Struct

    An empty struct in C++ has sizeof equal to 1 (not 0). This guarantees distinct addresses for distinct objects. However, as a base class, the empty base optimization (EBO) allows the compiler to give it size 0 within the derived class layout:

    struct Empty {};
static_assert(sizeof(Empty) == 1);

struct Derived : Empty {
    int x;
};
static_assert(sizeof(Derived) == 4);  // EBO applied: Empty takes 0 bytes

// C++20 [[no_unique_address]] does the same for members:
struct Compact {
    [[no_unique_address]] Empty e;
    int x;
};
static_assert(sizeof(Compact) == 4);

    Rearranging Members to Reduce Padding

    Since the compiler cannot reorder members, it is the programmer's responsibility to order them wisely. The general rule: sort members from largest alignment to smallest:

    struct Good {
    double d;    // 8 bytes, offset 0
    int    b;    // 4 bytes, offset 8
    char   a;    // 1 byte,  offset 12
    char   c;    // 1 byte,  offset 13
                 // 2 bytes trailing padding (to reach multiple of 8)
};
// sizeof(Good) = 16 bytes (vs 24 for Bad)
// Same data, 33% less memory

    For arrays, this savings multiplies. An array of 1 million Bad objects wastes roughly 10 MB; an array of 1 million Good objects saves that entirely.

    Packed Structs

    Sometimes you need exact control over layout, for example when serializing data to a wire protocol or reading a binary file format. Compiler-specific attributes remove padding:

    // GCC / Clang
struct __attribute__((packed)) PackedData {
    char  type;      // offset 0
    int   length;    // offset 1 (unaligned!)
    char  payload;   // offset 5
};
// sizeof(PackedData) = 6

// MSVC
#pragma pack(push, 1)
struct PackedData {
    char  type;
    int   length;
    char  payload;
};
#pragma pack(pop)

    Packed structs introduce unaligned access. On x86, this works but is slower. On ARM and other architectures, it can cause hardware faults. The compiler generates extra instructions to handle unaligned reads and writes, negating the memory savings with CPU overhead.

    Key insight: Packed structs should be used only at system boundaries (file I/O, network protocols). For in-memory data structures, prefer natural alignment and rearrange members instead.

    Over-Aligned Types

    An over-aligned type has alignment greater than alignof(std::max_align_t), the maximum alignment the implementation guarantees for standard allocation functions. On most platforms, alignof(std::max_align_t) is 16 (for long double or SSE types).

    struct alignas(64) CacheLine {
    int data[16];
};
// alignof(CacheLine) = 64, which is over-aligned

// C++17: operator new handles over-aligned types correctly
auto p = new CacheLine;   // guaranteed 64-byte aligned

// Before C++17, you needed aligned_alloc or platform APIs

    SIMD Alignment

    SIMD instructions (SSE, AVX, NEON) require data aligned to 16, 32, or 64 bytes. Using alignas ensures your data meets these requirements without resorting to platform-specific allocation:

    struct alignas(32) AVXVector {
    float data[8];  // 8 floats = 32 bytes, 32-byte aligned for AVX
};

void addVectors(const AVXVector* a, const AVXVector* b, AVXVector* out, int n) {
    for (int i = 0; i < n; ++i) {
        // Compiler can use aligned AVX load/store instructions
        for (int j = 0; j < 8; ++j)
            out[i].data[j] = a[i].data[j] + b[i].data[j];
    }
}

    Cache Line Alignment for Performance

    Modern CPUs load memory in cache lines, typically 64 bytes. When two threads modify variables that happen to share the same cache line, the hardware must bounce that line between cores—a phenomenon called false sharing. Aligning hot data to cache line boundaries eliminates this:

    struct alignas(64) Counter {
    std::atomic<int> value{0};
    // Padding to fill the rest of the cache line is implicit
    // because alignas(64) + next object alignment forces 64-byte spacing
};

// Each counter lives on its own cache line
Counter counters[NUM_THREADS];

    C++17 provides std::hardware_destructive_interference_size and std::hardware_constructive_interference_size to query the cache line size portably, though compiler support has been inconsistent:

    // C++17 (when available):
struct alignas(std::hardware_destructive_interference_size) PaddedAtomic {
    std::atomic<int> value{0};
};

// Fallback:
constexpr std::size_t CACHE_LINE = 64;
struct alignas(CACHE_LINE) PaddedAtomic {
    std::atomic<int> value{0};
};
    Key insight: False sharing is one of the most common performance killers in multi-threaded code. If you see poor scaling with thread count, check whether hot variables share cache lines. Aligning each to 64 bytes often provides dramatic speedups.

    Bit Fields and Alignment

    Bit fields let you pack members at the bit level, but they interact with alignment in complex ways:

    struct Flags {
    unsigned int readable  : 1;
    unsigned int writable  : 1;
    unsigned int executable: 1;
    unsigned int reserved  : 29;
};
static_assert(sizeof(Flags) == 4);  // all fit in one int

struct Mixed {
    unsigned int a : 3;
    unsigned int b : 5;
    // May or may not pack into the same storage unit
    unsigned int c : 12;
    char d;
};
// sizeof depends on implementation; usually 8 bytes

    Bit field layout is largely implementation-defined. The standard guarantees that adjacent bit fields of the same type are packed together, but padding between bit fields of different types, bit ordering within a byte, and straddling across storage unit boundaries are all compiler-specific.

    Inspecting Layout in Practice

    Several tools help you verify struct layouts:

    // offsetof macro (from <cstddef>):
#include <cstddef>
struct Example { char a; int b; double d; };

static_assert(offsetof(Example, a) == 0);
static_assert(offsetof(Example, b) == 4);
static_assert(offsetof(Example, d) == 8);

// Compiler-specific: Clang -fdump-record-layouts
// clang++ -cc1 -fdump-record-layouts example.cpp

// pahole (Linux): shows padding holes
// pahole ./a.out

    Standard Layout Types

    A type is standard-layout if it has a simple, C-compatible memory layout. The requirements include: all non-static data members in the same access class, no virtual functions or virtual bases, and no base classes with non-static data members. Standard-layout types are guaranteed to have their first member at offset 0 and can be safely cast to/from char* for serialization:

    struct StandardLayout {
    int x;
    float y;
    char z;
};
static_assert(std::is_standard_layout_v<StandardLayout>);

// NOT standard layout (members in different access classes):
struct NonStandard {
    int x;
private:
    int y;
};

    Practical Tips

    • Order by alignment (descending): double, int64_t, pointers first; then int, float; then short; then char, bool last.
    • Group same-sized members: All int members together, all char together.
    • Use static_assert: Assert expected sizeof and alignof to catch surprises.
    • Audit with tools: pahole, offsetof, compiler record layout dumps.
    • Don't pack prematurely: Packed structs are slower. Only pack at I/O boundaries.
    • Align for cache in hot paths: Use alignas(64) for data accessed from multiple threads.

    Summary

    • Alignment is a hardware requirement. Misaligned access is slow or faulting.
    • The compiler inserts padding to maintain alignment. Members are never reordered.
    • alignof queries alignment; alignas increases it.
    • Reordering members from largest to smallest alignment minimizes padding.
    • Packed structs remove padding but introduce unaligned access penalties.
    • Over-aligned types are needed for SIMD and cache line alignment.
    • False sharing occurs when threads modify data on the same cache line. Solve with alignas(64).
