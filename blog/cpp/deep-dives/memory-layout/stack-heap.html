    Stack vs Heap

    March 2026 · 24 min read

    Two Worlds of Memory

    Every C++ program works with two primary memory regions: the stack and the heap. They serve fundamentally different purposes, have different performance characteristics, and impose different constraints on your code. Understanding how each works at the hardware level transforms you from someone who follows rules blindly into someone who reasons about memory with confidence.

    The Stack: Frame by Frame

    The call stack is a contiguous region of memory that grows and shrinks as functions are called and returned. Each function call creates a stack frame (also called an activation record) containing:

    • Return address: Where to resume execution when this function returns.
    • Saved registers: Values of callee-saved registers (RBP, RBX, R12 to R15 on x86-64) that this function intends to overwrite.
    • Local variables: All automatic storage duration objects.
    • Function arguments: Parameters passed by value (though on x86-64, the first 6 integer and first 8 floating-point arguments go in registers).
    • Alignment padding: Added to maintain stack alignment (16-byte aligned on x86-64 at call sites).
    // x86-64 stack frame for:
void foo(int a, int b) {
    int x = 10;
    double y = 3.14;
    char buf[32];
    bar(x);
}

// Stack layout (high addresses at top, grows downward):
// +--------------------+
// | return address     |  (8 bytes, pushed by CALL instruction)
// +--------------------+
// | saved RBP          |  (8 bytes, if frame pointer used)
// +--------------------+ <-- RBP points here
// | int x = 10         |  (4 bytes at RBP-4)
// | (padding)          |  (4 bytes for alignment)
// | double y = 3.14    |  (8 bytes at RBP-16)
// | char buf[32]       |  (32 bytes at RBP-48)
// +--------------------+ <-- RSP points here (stack pointer)

    How a Function Call Works

    Here is what happens at the machine level when foo() calls bar(x) on x86-64:

    1. Argument setup: x is moved into register EDI (first integer argument).
    2. CALL instruction: Pushes the return address onto the stack and jumps to bar.
    3. Prologue in bar: push RBP; mov RBP, RSP; sub RSP, N to set up bar's frame.
    4. bar's body: Executes using its local frame.
    5. Epilogue in bar: mov RSP, RBP; pop RBP; ret to tear down the frame and return.
    6. Return: RET pops the return address and jumps back into foo.
    ; Simplified x86-64 assembly for the call:
; In foo:
    mov     edi, dword [rbp-4]    ; load x into first arg register
    call    bar                    ; push return addr, jump to bar

; In bar:
    push    rbp                    ; save caller's frame pointer
    mov     rbp, rsp               ; set up our frame pointer
    sub     rsp, 32                ; allocate local variables
    ; ... function body ...
    mov     rsp, rbp               ; restore stack pointer
    pop     rbp                    ; restore caller's frame pointer
    ret                            ; pop return address, jump back
    Key insight: Stack allocation is free—it is literally one instruction (sub rsp, N). No bookkeeping, no system calls, no fragmentation. This is why local variables are so fast.

    Stack Properties

    • Speed: Allocation is O(1)—a single instruction. Deallocation is equally fast.
    • Size: Typically 1 to 8 MB per thread (configurable). Much smaller than heap.
    • Lifetime: Strictly scoped. Objects die when their enclosing function returns.
    • Fragmentation: Impossible. Stack memory is always contiguous and LIFO.
    • Thread safety: Each thread has its own stack. No sharing, no locks.

    Stack Overflow

    The stack has a fixed size. If you exceed it (deep recursion, large local arrays), the program crashes with a stack overflow. The operating system places a guard page at the end of the stack; accessing it triggers a segmentation fault.

    // Stack overflow from deep recursion:
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // each call adds a frame
}
factorial(1000000);  // CRASH: stack overflow

// Stack overflow from large local array:
void oops() {
    int arr[10000000];  // 40 MB on the stack. Way too large.
}

    Solutions:

    • Convert recursion to iteration.
    • Allocate large buffers on the heap (std::vector).
    • Increase stack size (platform-specific: ulimit -s on Linux, linker flags on Windows).

    alloca: Dynamic Stack Allocation

    alloca() allocates memory on the stack dynamically. The memory is automatically freed when the function returns:

    #include <alloca.h>  // non-standard, but widely available

void process(int n) {
    int* buf = static_cast<int*>(alloca(n * sizeof(int)));
    // buf lives on the stack, freed when process() returns
    for (int i = 0; i < n; ++i) buf[i] = i;
}

    alloca is useful for small, variable-length temporary buffers. However, it is non-standard, cannot be used in loops safely (each call grows the stack), and there is no way to check for failure. C99's variable-length arrays (VLAs) work similarly but are not part of standard C++.

    Key insight: Prefer std::vector or std::array over alloca and VLAs. The heap overhead for a single allocation is negligible, and you get bounds checking, exception safety, and standard compliance.

    The Heap: Dynamic and Flexible

    The heap (also called the free store in C++ terminology) is a large, unstructured region of memory managed by an allocator. Unlike the stack, heap memory can be allocated and freed in any order, objects can outlive the function that created them, and the total size is limited only by available virtual memory.

    How Heap Allocation Works

    When you call new (or malloc), the allocator must find a suitable block of free memory. The most common strategy is a free list:

    1. The allocator maintains a linked list of free memory blocks.
    2. On allocation, it searches the list for a block large enough (policies: first-fit, best-fit, worst-fit).
    3. If the block is larger than needed, it splits the block: one part is returned, the remainder stays on the free list.
    4. On deallocation, the block is added back to the free list.
    5. If the freed block is adjacent to another free block, they are coalesced (merged) to reduce fragmentation.
    // Conceptual view of a free list:
// [Header: size=128, next] -> [Header: size=256, next] -> [Header: size=64, next] -> NULL
//
// malloc(100):
//   1. Search: skip 64 (too small), try 128 (fits!)
//   2. Split: return 100 bytes, keep 28-byte remainder on free list
//   3. Add allocation header (8-16 bytes) before returned pointer
//
// free(ptr):
//   1. Read header to find block size
//   2. Add block to free list
//   3. Coalesce with neighbors if they are free

    The Buddy System

    An alternative to free lists used by many kernel allocators. Memory is divided into power-of-two-sized blocks. When a block is freed, the allocator checks if its buddy (the adjacent block of the same size) is also free. If so, they merge into a block twice the size. This recurses upward.

    // Buddy system example (1024 bytes total):
// Initial: one 1024-byte block
//
// alloc(100): need 128 (next power of 2)
//   Split 1024 -> 512 + 512
//   Split 512  -> 256 + 256
//   Split 256  -> 128 + 128
//   Return first 128-byte block
//
// free(ptr): return 128-byte block
//   Buddy (adjacent 128) is free? Merge into 256.
//   Buddy (adjacent 256) is free? Merge into 512.
//   Buddy (adjacent 512) is free? Merge into 1024.
//   Back to one big block.

    The buddy system has O(log n) allocation and deallocation. It wastes memory through internal fragmentation (rounding up to powers of two) but has no external fragmentation.

    Modern Heap Allocators

    Production allocators (glibc's ptmalloc, jemalloc, tcmalloc, mimalloc) use sophisticated hybrid strategies:

    • Size classes: Small objects (8, 16, 32, ..., 256 bytes) go to per-size free lists (slab allocation). No splitting needed.
    • Thread-local caches: Each thread caches recently freed objects, avoiding lock contention on the global heap.
    • Arena-per-thread: tcmalloc and jemalloc use per-thread arenas, falling back to a central arena only when the thread cache is empty.
    • Large objects: Objects above a threshold (typically 128 KB to 256 KB) are mmap'd directly from the OS.
    Key insight: Modern allocators are remarkably fast for small objects. A typical malloc of 64 bytes in jemalloc takes roughly 20 to 30 nanoseconds. The performance gap between stack and heap is smaller than most people think for small allocations.

    Fragmentation

    Fragmentation is the heel of heap allocation. There are two kinds:

    • External fragmentation: Free memory exists but is scattered in small non-contiguous blocks. A large allocation fails even though the total free memory is sufficient.
    • Internal fragmentation: Allocated blocks are larger than requested (rounding to size classes or power-of-two). The extra bytes are wasted.
    // External fragmentation scenario:
// Heap state: [USED:64][FREE:32][USED:64][FREE:32][USED:64][FREE:32]
//
// Total free: 96 bytes
// malloc(64): FAILS! No single contiguous block of 64 bytes.
//
// This is why long-running programs can "run out of memory"
// even when plenty of free memory exists.

    When to Use Which

    Prefer the Stack When:

    • The object's size is known at compile time and is small (under a few KB).
    • The object's lifetime matches the enclosing scope.
    • You need maximum performance (tight loops, hot paths).
    • You want automatic cleanup (RAII).
    void processData(const std::vector<int>& input) {
    std::array<int, 256> buffer;     // stack: fixed, small
    int sum = 0;                       // stack: trivial
    auto it = input.begin();           // stack: small object

    for (auto val : input) {
        sum += val;
    }
}

    Use the Heap When:

    • The size is determined at runtime or is large.
    • The object must outlive the creating function.
    • You need shared ownership (shared_ptr).
    • You are building dynamic data structures (trees, linked lists, graphs).
    • You need polymorphic objects (base pointer to derived).
    std::unique_ptr<Widget> createWidget(int type) {
    // Must outlive this function:
    return std::make_unique<Widget>(type);
}

std::vector<int> loadData(const std::string& path) {
    // Size unknown until file is read:
    std::vector<int> data;   // heap-allocated internally
    // ... read file into data ...
    return data;              // moved out (no copy)
}

    Process Memory Layout

    A typical process has its virtual address space organized as follows:

    // Virtual address space (simplified, Linux x86-64):
// +----------------------------+ 0xFFFF...  (high addresses)
// |        Kernel space         |
// +----------------------------+ 0x7FFF...
// |        Stack (grows down)   |
// |            |                |
// |            v                |
// |                             |
// |            ^                |
// |            |                |
// |        Heap (grows up)      |
// +----------------------------+
// |        BSS (zero-init)      |
// +----------------------------+
// |        Data (initialized)   |
// +----------------------------+
// |        Text (code, r/o)     |
// +----------------------------+ 0x0040...  (low addresses)
    • Text segment: Read-only compiled machine code.
    • Data segment: Initialized global and static variables.
    • BSS segment: Zero-initialized globals and statics (occupies no file space).
    • Heap: Grows upward from the end of BSS. Managed by the allocator.
    • Stack: Grows downward from high addresses. One per thread.
    • Kernel space: Not accessible to user code.

    Real-World Patterns

    Small Buffer Optimization (SBO)

    Many standard library types blend stack and heap. std::string stores short strings (typically 15 to 22 bytes) in an inline buffer inside the object (stack), switching to heap allocation only for longer strings:

    // Typical std::string layout (libc++):
class string {
    union {
        struct {
            char*  ptr;        // pointer to heap buffer
            size_t size;
            size_t capacity;
        } long_;
        struct {
            char data[23];     // inline buffer for short strings
            unsigned char size;
        } short_;
    };
    // sizeof(string) = 24 bytes on 64-bit
};

std::string s1 = "hi";            // stored inline (2 chars)
std::string s2 = "a longer string that exceeds the buffer";  // heap

    This optimisation is sometimes called SSO (Small String Optimization). Similar patterns exist in std::function (small callable optimization) and boost::any.

    Escape Analysis (Compiler Optimization)

    Some compilers can prove that a heap allocation does not escape the function and can replace it with a stack allocation. This is more common in JIT-compiled languages (Java, Go) but GCC and Clang do limited forms of it:

    // A smart compiler might stack-allocate this:
void example() {
    auto p = std::make_unique<int>(42);
    use(*p);
}
// p does not escape. In theory, the compiler could
// replace the heap allocation with a stack variable.
// In practice, C++ compilers rarely do this today.

    Debugging Memory Issues

    • Stack overflow: Increase stack size or convert recursion to iteration. Use -fsanitize=address to detect stack buffer overflows.
    • Heap corruption: Use AddressSanitizer (ASan) to detect use-after-free, buffer overflows, and double-free.
    • Memory leaks: Use Valgrind's Memcheck or LeakSanitizer. In modern C++, prefer smart pointers to prevent leaks structurally.
    • Fragmentation: Use malloc_stats() or jemalloc's malloc_stats_print() to inspect allocator state.
    // Compile with sanitizers:
// g++ -fsanitize=address -g main.cpp -o main
// ./main

// Valgrind:
// valgrind --leak-check=full ./main

    Summary

    • The stack is fast (O(1) allocation), fixed-size, scope-bound, and fragmentation-free. Use it for small, short-lived objects.
    • The heap is flexible (any size, any lifetime), but slower, prone to fragmentation, and requires manual or RAII-based management.
    • Stack frames contain the return address, saved registers, local variables, and alignment padding.
    • Heap allocators use free lists, size classes, and thread-local caches to minimize overhead.
    • The buddy system uses power-of-two splitting and merging for O(log n) allocation.
    • Small buffer optimization combines stack and heap: small objects stay inline, large ones go to the heap.
    • Stack overflow is caused by deep recursion or large locals. Use iteration and std::vector.
    • Modern allocators (jemalloc, tcmalloc, mimalloc) are very fast. Profile before replacing them.
