    Placement New & Manual Lifetime

    March 2026 · 20 min read

    What is Placement New?

    Regular new does two things: allocates memory and constructs an object. Placement new skips the allocation and constructs an object at a specific memory address you provide. You supply the memory; placement new runs the constructor in it.

    #include <new>  // required for placement new

// Step 1: Allocate raw memory (no object exists yet)
alignas(Widget) unsigned char buffer[sizeof(Widget)];

// Step 2: Construct an object in that memory
Widget* w = new (buffer) Widget(42);

// Step 3: Use the object
w->doWork();

// Step 4: Manually destroy (call destructor explicitly)
w->~Widget();

// The buffer still exists, but no Widget lives in it anymore.

    This is the lowest level of object construction in C++. The standard library uses it inside std::vector, std::optional, std::variant, and every allocator-aware container. Understanding placement new means understanding how these types work internally.

    The Syntax

    Placement new is invoked by passing an address as an argument to new:

    void* memory = /* some pre-allocated memory */;
T* obj = new (memory) T(args...);

// This calls:
//   void* operator new(std::size_t, void* ptr) noexcept { return ptr; }
// which simply returns the pointer unchanged. Then the constructor runs.

    You must include <new> for the placement form. The compiler needs the declaration of the placement operator new overload.

    Key insight: Placement new does not allocate memory. It only runs the constructor. You are responsible for ensuring the memory is properly aligned, large enough, and that no other live object occupies it.

    Manual Destructor Calls

    Because placement new does not allocate, you must not use delete on the resulting pointer (that would try to deallocate memory you did not allocate with new). Instead, you call the destructor explicitly:

    Widget* w = new (buffer) Widget(42);
// ...
w->~Widget();  // explicit destructor call

// Do NOT do this:
// delete w;   // ERROR: would try to deallocate buffer

    Explicit destructor calls are one of the few places in C++ where you directly invoke a member function with the tilde syntax. It is perfectly valid—the standard requires it for this use case.

    Destroy and Reuse

    After destroying an object, the memory is free for reuse. You can construct a completely different type in the same storage:

    alignas(std::max_align_t) unsigned char storage[256];

// Construct a string
auto* s = new (storage) std::string("hello");
std::cout << *s << "\n";
s->~basic_string();

// Reuse the same memory for a vector
auto* v = new (storage) std::vector<int>({1, 2, 3});
std::cout << v->size() << "\n";
v->~vector();
    Key insight: This pattern of destroy-then-reconstruct is exactly what std::variant does when you assign a value of a different type. It destroys the current alternative and placement-new constructs the new one.

    Aligned Storage

    Before C++23 deprecated it, std::aligned_storage was the standard way to create properly aligned raw memory for placement new:

    // C++11/14/17 (deprecated in C++23):
std::aligned_storage_t<sizeof(Widget), alignof(Widget)> storage;
Widget* w = new (&storage) Widget(42);

// Modern approach: use alignas directly
alignas(Widget) unsigned char buffer[sizeof(Widget)];
Widget* w = new (buffer) Widget(42);

// Or use std::byte (C++17):
alignas(Widget) std::byte buffer[sizeof(Widget)];
Widget* w = new (buffer) Widget(42);

    The modern approach with alignas is simpler. std::aligned_storage had footguns: it could silently produce a type smaller than needed if the size was not a multiple of the alignment.

    std::launder (C++17)

    When you construct a new object in memory that previously held a different object (or the same type with const members), the compiler may cache the old object's values. std::launder tells the compiler: "forget what you knew about this address; a new object lives here now."

    struct Immutable {
    const int value;
};

alignas(Immutable) unsigned char buf[sizeof(Immutable)];

auto* a = new (buf) Immutable{42};
std::cout << a->value << "\n";   // 42

a->~Immutable();
auto* b = new (buf) Immutable{99};

// Without launder, the compiler may still think a->value is 42
// because it was const and at the same address.
auto* safe = std::launder(b);
std::cout << safe->value << "\n";  // guaranteed 99

    When is std::launder Needed?

    std::launder is needed when:

    • You construct a new object at an address where a previous object with const or reference members lived.
    • You access an object through a pointer obtained before the object's lifetime began (e.g., reinterpreting storage).
    • The compiler might apply optimizations based on type identity or constness that no longer hold.

    In practice, std::launder is rarely needed in application code. It is primarily used in library internals (optional, variant, allocator implementations).

    Key insight: std::launder exists because C++ gives the compiler permission to optimize aggressively based on type information. When you violate the compiler's assumptions by reusing memory, std::launder re-establishes correct semantics.

    Implementing std::optional Internals

    A simplified optional demonstrates placement new in library code:

    template<typename T>
class Optional {
    alignas(T) unsigned char storage_[sizeof(T)];
    bool has_value_;

public:
    Optional() : has_value_(false) {}

    Optional(const T& value) : has_value_(true) {
        new (storage_) T(value);    // placement new: copy construct
    }

    Optional(T&& value) : has_value_(true) {
        new (storage_) T(std::move(value));  // placement new: move construct
    }

    ~Optional() {
        if (has_value_) {
            reinterpret_cast<T*>(storage_)->~T();  // explicit destructor
        }
    }

    T& value() {
        if (!has_value_) throw std::bad_optional_access();
        return *std::launder(reinterpret_cast<T*>(storage_));
    }

    void reset() {
        if (has_value_) {
            reinterpret_cast<T*>(storage_)->~T();
            has_value_ = false;
        }
    }

    template<typename... Args>
    T& emplace(Args&&... args) {
        reset();
        new (storage_) T(std::forward<Args>(args)...);
        has_value_ = true;
        return *std::launder(reinterpret_cast<T*>(storage_));
    }
};

    Key observations:

    • Storage is a raw unsigned char array with proper alignment.
    • The constructor uses placement new to construct T in the storage.
    • The destructor explicitly calls T's destructor only if a value is present.
    • emplace() destroys any existing value, then placement-new constructs a new one.
    • std::launder is used when accessing the stored value through a reinterpreted pointer.

    Implementing std::variant Internals

    A variant holds one of several types in the same storage. Here is a simplified two-type variant:

    template<typename A, typename B>
class Variant {
    static constexpr std::size_t Size = (sizeof(A) > sizeof(B)) ? sizeof(A) : sizeof(B);
    static constexpr std::size_t Align = (alignof(A) > alignof(B)) ? alignof(A) : alignof(B);

    alignas(Align) unsigned char storage_[Size];
    int index_;  // 0 = A, 1 = B, -1 = valueless

public:
    Variant() : index_(-1) {}

    void set(const A& a) {
        destroy();
        new (storage_) A(a);
        index_ = 0;
    }

    void set(const B& b) {
        destroy();
        new (storage_) B(b);
        index_ = 1;
    }

    template<typename T>
    T& get() {
        return *std::launder(reinterpret_cast<T*>(storage_));
    }

    ~Variant() { destroy(); }

private:
    void destroy() {
        if (index_ == 0) std::launder(reinterpret_cast<A*>(storage_))->~A();
        else if (index_ == 1) std::launder(reinterpret_cast<B*>(storage_))->~B();
        index_ = -1;
    }
};

    The real std::variant uses a recursive union or a similar technique with visitation support, but the core lifetime management is the same: placement new to construct, explicit destructor to destroy, std::launder (or equivalent) to access.

    Placement New for Arrays

    Constructing arrays with placement new requires constructing each element individually:

    constexpr int N = 10;
alignas(Widget) unsigned char buf[sizeof(Widget) * N];

// Construct N widgets
for (int i = 0; i < N; ++i) {
    new (buf + i * sizeof(Widget)) Widget(i);
}

// Use them
Widget* arr = std::launder(reinterpret_cast<Widget*>(buf));
for (int i = 0; i < N; ++i) {
    arr[i].doWork();
}

// Destroy in reverse order (matching convention)
for (int i = N - 1; i >= 0; --i) {
    arr[i].~Widget();
}
    Key insight: Do not use array placement new (new (buf) Widget[N]). The standard allows the implementation to store extra overhead (the array size) in the buffer, and there is no portable way to know how much space it uses. Construct elements individually in a loop.

    Placement New in Allocators

    Custom allocators and std::allocator_traits use placement new to construct objects in allocated memory:

    // How std::vector<T, Alloc>::push_back works internally:
template<typename T, typename Alloc>
void vector<T, Alloc>::push_back(const T& value) {
    if (size_ == capacity_) grow();

    // Allocator provides raw memory; placement new constructs the object
    std::allocator_traits<Alloc>::construct(
        alloc_,
        data_ + size_,   // pointer to uninitialized memory
        value             // forward to T's copy constructor
    );
    ++size_;
}

// std::allocator_traits::construct defaults to:
template<typename T, typename... Args>
void construct(Alloc&, T* p, Args&&... args) {
    ::new (static_cast<void*>(p)) T(std::forward<Args>(args)...);
}

    Type Punning and Strict Aliasing

    Placement new is sometimes confused with type punning (reinterpreting memory as a different type). They are fundamentally different:

    // VALID: placement new creates a real object of type T
float f = 3.14f;
unsigned char buf[sizeof(int)];
std::memcpy(buf, &f, sizeof(f));
int* ip = new (buf) int;  // creates an int object (value indeterminate)

// UNDEFINED BEHAVIOUR: type punning through pointer cast
float f = 3.14f;
int* ip = reinterpret_cast<int*>(&f);  // strict aliasing violation
int bits = *ip;  // UB

// VALID alternative: std::bit_cast (C++20)
float f = 3.14f;
int bits = std::bit_cast<int>(f);  // well-defined
    Key insight: Placement new creates an object. reinterpret_cast does not. The C++ object model requires that an object of the accessed type actually exists at the memory location. Placement new is how you make that happen.

    Exception Safety with Placement New

    If a constructor called via placement new throws, the memory is not freed (because placement new did not allocate it). This is correct behaviour—you manage the memory, and you should handle the failure:

    alignas(Widget) unsigned char buf[sizeof(Widget)];

try {
    Widget* w = new (buf) Widget(bad_args);  // might throw
    // If constructor throws, no object exists in buf
    // No destructor call needed (object was never constructed)
} catch (...) {
    // buf is still valid memory, just uninitialized
    // We can try again or use it for something else
}

    Standard Library Uninitialized Algorithms

    The <memory> header provides algorithms that use placement new internally, saving you from writing manual loops:

    #include <memory>

alignas(std::string) unsigned char buf[sizeof(std::string) * 5];
auto* arr = reinterpret_cast<std::string*>(buf);

// Construct 5 copies of "hello"
std::uninitialized_fill_n(arr, 5, std::string("hello"));

// Move-construct from a range
std::vector<std::string> src = {"a", "b", "c"};
std::uninitialized_move(src.begin(), src.end(), arr);

// Destroy without deallocating
std::destroy_n(arr, 5);

// C++20: construct_at and destroy_at
std::construct_at(arr, "world");   // same as placement new
std::destroy_at(arr);              // same as manual destructor call

    std::construct_at (C++20) is preferred over raw placement new in constexpr contexts, since placement new is not allowed in constant expressions but std::construct_at is.

    Practical Uses

    • Container implementations: vector, deque, list all use placement new to construct elements in pre-allocated buffers.
    • Optional/variant/any: Store objects in inline storage without heap allocation.
    • Custom allocators: The allocator provides memory; placement new constructs objects.
    • Object pools: Reuse memory slots by destroying and reconstructing objects.
    • Serialization: Reconstruct objects from deserialized bytes (with care).
    • Memory-mapped I/O: Construct objects at specific hardware addresses.

    Summary

    • Placement new constructs an object at a specified address. It does not allocate memory.
    • You must manually call the destructor when done. Do not use delete.
    • The memory must be properly aligned and large enough.
    • std::launder (C++17) is needed when reusing memory that held objects with const or reference members.
    • std::optional and std::variant use placement new internally for inline storage.
    • Prefer std::construct_at (C++20) in constexpr contexts.
    • Do not confuse placement new with reinterpret_cast type punning. Placement new creates an object; reinterpret_cast does not.
