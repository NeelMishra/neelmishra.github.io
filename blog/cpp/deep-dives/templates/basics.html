    Template Basics

    March 2026 · 22 min read

    Why Templates?

    Imagine you need a max function. Without templates, you write one version for int, another for double, another for std::string. Each is identical except for the type. This violates the DRY principle and creates a maintenance nightmare.

    C++ templates solve this problem by letting you write code that is parameterized by types (or values). The compiler generates the concrete code for each type you actually use. You write the logic once; the compiler stamps out as many versions as needed.

    // Without templates: three nearly-identical functions
int max_int(int a, int b) { return (a > b) ? a : b; }
double max_double(double a, double b) { return (a > b) ? a : b; }
std::string max_string(const std::string& a, const std::string& b) {
    return (a > b) ? a : b;
}

// With templates: one function covers all types
template <typename T>
T max_val(const T& a, const T& b) {
    return (a > b) ? a : b;
}

    Templates are not merely a convenience feature. They are the foundation of the entire C++ Standard Library. Every container (std::vector, std::map), every algorithm (std::sort, std::find), every smart pointer (std::unique_ptr, std::shared_ptr) is a template. Understanding templates is essential to mastering C++.

    Key insight: Templates are a compile-time code generation mechanism. No template code exists at runtime—the compiler has already produced the concrete functions and classes you need. This is fundamentally different from runtime polymorphism via virtual functions.

    Function Templates

    A function template defines a family of functions. You declare it by placing template <typename T> (or template <class T>—they are interchangeable here) before the function declaration:

    template <typename T>
T add(const T& a, const T& b) {
    return a + b;
}

// Usage:
int    x = add(3, 5);          // T = int
double y = add(1.5, 2.7);      // T = double
auto   z = add<long>(10, 20);  // T = long (explicit)

    When you call add(3, 5), the compiler deduces T = int and generates:

    int add(const int& a, const int& b) {
    return a + b;
}

    Multiple Type Parameters

    Templates can accept more than one type parameter. This is useful when a function needs to work with two independently varying types:

    template <typename T, typename U>
auto multiply(const T& a, const U& b) -> decltype(a * b) {
    return a * b;
}

// Usage:
auto result = multiply(3, 2.5);  // T = int, U = double, returns double

    In C++14 and later, you can simplify the return type to just auto:

    template <typename T, typename U>
auto multiply(const T& a, const U& b) {
    return a * b;
}

    Overloading Function Templates

    Function templates can coexist with regular functions and with other template overloads. The compiler follows a precise selection process:

    1. Non-template functions are preferred if they match exactly.
    2. Among template candidates, the most specialized one wins.
    3. If there is a tie, the program is ill-formed (ambiguous).
    // Non-template overload
const char* max_val(const char* a, const char* b) {
    return (std::strcmp(a, b) > 0) ? a : b;
}

// Template (from earlier)
template <typename T>
T max_val(const T& a, const T& b) {
    return (a > b) ? a : b;
}

max_val("hello", "world");  // calls non-template (exact match)
max_val(10, 20);             // calls template with T = int

    Class Templates

    While function templates parameterize a single function, class templates parameterize an entire class. Every member function, every data member—everything inside the class definition can depend on the template parameters.

    template <typename T>
class Stack {
    std::vector<T> data_;

public:
    void push(const T& value) {
        data_.push_back(value);
    }

    T pop() {
        if (data_.empty()) throw std::runtime_error("Stack is empty");
        T top = std::move(data_.back());
        data_.pop_back();
        return top;
    }

    bool empty() const { return data_.empty(); }
    std::size_t size() const { return data_.size(); }
};

    Usage:

    Stack<int> intStack;
intStack.push(42);
intStack.push(17);
int val = intStack.pop();  // val = 17

Stack<std::string> strStack;
strStack.push("hello");
strStack.push("world");

    Defining Member Functions Outside the Class

    When you define member functions outside the class body, you must repeat the template header:

    template <typename T>
class Queue {
    std::deque<T> data_;
public:
    void enqueue(const T& value);
    T dequeue();
    bool empty() const;
};

template <typename T>
void Queue<T>::enqueue(const T& value) {
    data_.push_back(value);
}

template <typename T>
T Queue<T>::dequeue() {
    T front = std::move(data_.front());
    data_.pop_front();
    return front;
}

template <typename T>
bool Queue<T>::empty() const {
    return data_.empty();
}
    Key insight: Template definitions must be visible at the point of instantiation. In practice, this means templates live in header files (or inline in the class body). You cannot put a template's implementation in a .cpp file and expect the linker to find it.

    Class Template Argument Deduction (CTAD, C++17)

    Before C++17, you always had to specify template arguments when creating class template objects. C++17 introduced CTAD, which lets the compiler deduce them from constructor arguments:

    // Pre-C++17: must specify template argument
std::pair<int, double> p1(42, 3.14);
std::vector<int> v1{1, 2, 3};

// C++17: compiler deduces the types
std::pair p2(42, 3.14);      // std::pair<int, double>
std::vector v2{1, 2, 3};     // std::vector<int>
std::tuple  t(1, 2.0, "hi"); // std::tuple<int, double, const char*>

    You can write deduction guides to control how CTAD works for your own classes:

    template <typename T>
class Wrapper {
    T value_;
public:
    Wrapper(T val) : value_(std::move(val)) {}
};

// Deduction guide: Wrapper constructed from const char* should become Wrapper<std::string>
Wrapper(const char*) -> Wrapper<std::string>;

Wrapper w1(42);       // Wrapper<int>
Wrapper w2("hello");  // Wrapper<std::string> (not Wrapper<const char*>)

    Non-Type Template Parameters

    Templates are not limited to type parameters. You can also parameterize by values—integers, enumerators, pointers, and (since C++20) even floating-point types and literal class types.

    template <typename T, std::size_t N>
class FixedArray {
    T data_[N];
public:
    T& operator[](std::size_t idx) { return data_[idx]; }
    const T& operator[](std::size_t idx) const { return data_[idx]; }
    constexpr std::size_t size() const { return N; }
};

FixedArray<int, 10> arr;  // array of 10 ints, size known at compile time
FixedArray<double, 3> v;  // completely different type from FixedArray<double, 5>

    Non-type parameters are powerful because they embed values into the type system. FixedArray<int, 10> and FixedArray<int, 20> are completely unrelated types. The compiler can optimize based on the known size—unrolling loops, allocating on the stack, and eliminating bounds checks where possible.

    Allowed Non-Type Parameter Types

    • Integral types: int, std::size_t, char, bool, etc.
    • Enumerations
    • Pointers and references to objects with external linkage
    • std::nullptr_t
    • C++20: floating-point types, literal class types with public members
    // Non-type parameter with auto (C++17)
template <auto Value>
struct Constant {
    static constexpr auto value = Value;
};

Constant<42>::value    // int, 42
Constant<'A'>::value   // char, 'A'
Constant<true>::value  // bool, true

    Practical Example: compile-time Fibonacci

    template <int N>
struct Fibonacci {
    static constexpr int value = Fibonacci<N - 1>::value + Fibonacci<N - 2>::value;
};

template <>
struct Fibonacci<0> { static constexpr int value = 0; };

template <>
struct Fibonacci<1> { static constexpr int value = 1; };

static_assert(Fibonacci<10>::value == 55);

    Template Argument Deduction

    When you call a function template without explicitly specifying template arguments, the compiler deduces them from the function arguments. This process follows precise rules that every C++ developer should understand.

    Basic Deduction Rules

    The compiler examines each function parameter/argument pair and works backwards to determine what T must be:

    template <typename T>
void f(T param);           // T deduced from value

template <typename T>
void g(T& param);          // T deduced from reference

template <typename T>
void h(T&& param);         // T deduced via forwarding reference rules

int x = 42;
const int cx = 42;
int& rx = x;

f(x);    // T = int, param is int (copy)
f(cx);   // T = int, param is int (const stripped for by-value)
f(rx);   // T = int, param is int (reference stripped for by-value)

g(x);    // T = int, param is int&
g(cx);   // T = const int, param is const int&

h(x);    // T = int&, param is int& (lvalue → reference collapse)
h(42);   // T = int, param is int&& (rvalue)
    Key insight: When the parameter type is T (by value), top-level const and reference qualifiers are stripped. When it is T& or const T&, reference is stripped but const is preserved. When it is T&& (forwarding reference), special reference-collapsing rules apply.

    When Deduction Fails

    The compiler cannot always deduce template arguments. Common failure scenarios:

    template <typename T>
T convert(T a, T b);

convert(1, 2.5);  // ERROR: T deduced as int from first arg, double from second
                   // Fix: convert<double>(1, 2.5)
    template <typename T>
T create();

auto val = create();     // ERROR: no argument to deduce T from
auto val = create<int>(); // OK: explicit specification

    Deduction also cannot look inside nested types:

    template <typename T>
void process(typename std::vector<T>::iterator it);

std::vector<int> v{1, 2, 3};
process(v.begin());  // ERROR: non-deduced context
process<int>(v.begin());  // OK: explicit

    Template Instantiation

    Templates are not real code until they are instantiated. Instantiation is the process by which the compiler generates concrete functions or classes from a template.

    Implicit Instantiation

    This happens automatically when you use a template with specific arguments:

    template <typename T>
T square(T x) { return x * x; }

int a = square(5);       // implicitly instantiates square<int>
double b = square(3.14); // implicitly instantiates square<double>

    Important rule: the compiler only instantiates what is actually used. If Stack<int> is created but pop() is never called, the compiler does not instantiate Stack<int>::pop(). This means a member function can contain code that would be invalid for certain types, and the program still compiles as long as that function is never called with those types.

    Explicit Instantiation

    You can force the compiler to instantiate a template for a specific type, even if it is not used in that translation unit:

    // In a .cpp file:
template int square<int>(int);           // explicit instantiation definition
template class Stack<double>;            // instantiates ALL members

// In a header, to suppress implicit instantiation elsewhere:
extern template int square<int>(int);    // explicit instantiation declaration
extern template class Stack<double>;

    Why would you use this? Two main reasons:

    1. Compilation speed: by instantiating in one translation unit and using extern template in others, you avoid redundant instantiations across many .cpp files.
    2. Controlling symbol visibility: in shared libraries, you might want to export only specific instantiations.

    Two-Phase Lookup

    Template compilation happens in two phases:

    1. Phase 1 (definition time): The compiler checks syntax and looks up names that do not depend on template parameters (non-dependent names).
    2. Phase 2 (instantiation time): The compiler substitutes the actual types and looks up dependent names.
    template <typename T>
void process(T val) {
    foo();        // non-dependent: looked up in phase 1
    val.bar();    // dependent on T: looked up in phase 2
}

    This means that if foo() is not declared before the template definition, you get an error at definition time—before anyone even calls process. But val.bar() is only checked when process is instantiated with a specific T.

    Key insight: The typename disambiguator is required in phase 1 when accessing a nested type that depends on a template parameter: typename T::value_type. Without it, the compiler assumes the name refers to a value, not a type.

    Default Template Arguments

    Just like function parameters can have defaults, template parameters can too:

    template <typename T = int, typename Allocator = std::allocator<T>>
class Container {
    // ...
};

Container<> c1;                 // T = int, Allocator = std::allocator<int>
Container<double> c2;           // T = double, Allocator = std::allocator<double>
Container<char, MyAlloc<char>> c3;  // both specified

    Default arguments for function templates were originally limited (C++03 did not allow them). Since C++11, function templates support default template arguments fully:

    template <typename T = double>
T zero() { return T{}; }

zero();        // returns 0.0 (double)
zero<int>();   // returns 0 (int)

    Template Template Parameters

    Sometimes you want to pass a template itself as a parameter, not an instantiation of a template. This is called a template template parameter:

    template <typename T, template <typename, typename> class Container = std::vector>
class DataStore {
    Container<T, std::allocator<T>> data_;
public:
    void add(const T& item) { data_.push_back(item); }
    auto begin() const { return data_.begin(); }
    auto end()   const { return data_.end(); }
};

DataStore<int> ds1;                 // uses std::vector
DataStore<int, std::deque> ds2;     // uses std::deque

    In C++17 and later, you can use typename instead of class for template template parameters:

    template <typename T, template <typename...> typename Container = std::vector>
class DataStore { /* ... */ };

    Common Pitfalls

    1. Missing typename for Dependent Types

    template <typename Container>
void print_first(const Container& c) {
    // WRONG: compiler thinks Container::iterator is a value
    // Container::iterator it = c.begin();

    // CORRECT: tell the compiler it's a type
    typename Container::iterator it = c.begin();
    std::cout << *it;
}

    2. Missing template for Dependent Template Members

    template <typename T>
void call_get(T& obj) {
    // WRONG: compiler parses < as less-than operator
    // obj.get<int>();

    // CORRECT: tell the compiler get is a template
    obj.template get<int>();
}

    3. Forgetting That Templates Are Not in the .cpp

    If you separate the template declaration (in a header) from its definition (in a .cpp), you will get linker errors. The compiler needs to see the full definition at the point of instantiation. Solutions:

    • Put everything in the header (most common).
    • Use explicit instantiation.
    • Use the .tpp or .ipp convention: define in a separate file, then #include it at the end of the header.

    4. Template Error Messages

    Template errors are notoriously verbose. A missing operator< on a type passed to std::sort can produce hundreds of lines of errors. Strategies to cope:

    • Read only the first error message—the rest are usually cascading failures.
    • Look for your file name and line number in the error output.
    • Use C++20 concepts (covered in a later article) to get cleaner errors.
    • Use static_assert with type traits to produce custom error messages.

    Best Practices

    • Prefer typename over class in template parameter lists. typename communicates that the parameter need not be a class type.
    • Use const T& for parameters that should not be modified, just like non-template functions.
    • Document requirements on T in comments (or use concepts in C++20). What operations must T support?
    • Use static_assert to catch misuse early with clear messages:
    template <typename T>
T add(const T& a, const T& b) {
    static_assert(std::is_arithmetic_v<T>, "add() requires an arithmetic type");
    return a + b;
}
    • Minimize the interface that your template requires from T. The fewer operations you demand, the more types your template works with.
    • Avoid deep nesting of templates. Factor out helper templates and type aliases to keep code readable.
