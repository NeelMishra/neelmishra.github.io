    CRTP: Curiously Recurring Template Pattern

    March 2026 · 21 min read

    The Pattern

    The Curiously Recurring Template Pattern (CRTP) is a C++ idiom where a class inherits from a template that takes the derived class itself as a template argument:

    template <typename Derived>
class Base {
    // Base can access Derived's members through a cast
    void interface() {
        static_cast<Derived*>(this)->implementation();
    }
};

class Concrete : public Base<Concrete> {
    void implementation() {
        // actual work
    }
    friend class Base<Concrete>;
};

    The key insight: Base knows at compile time what Derived is. It can call Derived's methods, access its types, and customize behavior—all without virtual functions, all resolved at compile time.

    This pattern was first described by James Coplien in 1995 and independently discovered by multiple library authors. It appears extensively in production codebases: the Eigen linear algebra library, Boost, LLVM, and Windows ATL all use it heavily.

    Key insight: CRTP inverts the usual inheritance relationship. Instead of the base class defining an interface that derived classes implement (runtime polymorphism), the base class uses the derived class to provide reusable functionality (static polymorphism). The base is a tool for the derived class, not a contract.

    Static Polymorphism

    The most common use of CRTP is to achieve polymorphism without the overhead of virtual functions. The base class defines the algorithm structure, and the derived class plugs in the specifics:

    template <typename Derived>
class Shape {
public:
    double area() const {
        return static_cast<const Derived*>(this)->area_impl();
    }

    double perimeter() const {
        return static_cast<const Derived*>(this)->perimeter_impl();
    }

    void describe() const {
        std::cout << "Area: " << area()
                  << ", Perimeter: " << perimeter() << "\n";
    }
};

class Circle : public Shape<Circle> {
    double radius_;
public:
    explicit Circle(double r) : radius_(r) {}

    double area_impl() const {
        return 3.14159265 * radius_ * radius_;
    }

    double perimeter_impl() const {
        return 2.0 * 3.14159265 * radius_;
    }
};

class Rectangle : public Shape<Rectangle> {
    double w_, h_;
public:
    Rectangle(double w, double h) : w_(w), h_(h) {}

    double area_impl() const { return w_ * h_; }
    double perimeter_impl() const { return 2.0 * (w_ + h_); }
};

    Usage:

    Circle c(5.0);
c.describe();  // "Area: 78.5398, Perimeter: 31.4159"

Rectangle r(3.0, 4.0);
r.describe();  // "Area: 12, Perimeter: 14"

// Works with templates:
template <typename ShapeT>
void print_info(const ShapeT& shape) {
    shape.describe();  // statically dispatched
}

    There is no virtual keyword anywhere. The compiler resolves area() and perimeter() calls at compile time through static_cast. The resulting binary has direct function calls—no vtable, no indirection.

    What CRTP Cannot Do

    Unlike virtual dispatch, CRTP does not support runtime polymorphism. You cannot do this:

    // This does NOT work with CRTP:
// Shape<Circle> and Shape<Rectangle> are DIFFERENT types
// std::vector<Shape*> doesn't exist—Shape is a template, not a type

// For heterogeneous collections, you still need virtual functions or std::variant.

    CRTP is the right choice when the set of types is known at compile time and you call through concrete types or templates—not through base-class pointers.

    CRTP vs Virtual Dispatch

    Aspect Virtual Dispatch CRTP

    Overhead per callvtable lookup + indirect callZero (inlined)

    Performance Benchmark

    In tight loops that call a polymorphic method millions of times, CRTP typically outperforms virtual dispatch by 2x to 10x due to inlining and elimination of indirect branches. The difference matters in:

    • Linear algebra: Eigen uses CRTP to make expression objects zero-overhead.
    • Game engines: Entity component systems use static polymorphism for hot paths.
    • Networking: Protocol parsers where every byte counts.
    • Embedded systems: Where vtable memory overhead is unacceptable.
    // Benchmark sketch
// Virtual: ~4.2 ns per call (indirect branch, no inlining)
// CRTP:   ~0.8 ns per call (direct call, inlined)
// The gap widens when the method body is small relative to call overhead.

// virtual version
struct VShape { virtual double area() const = 0; };
struct VCircle : VShape {
    double r;
    double area() const override { return 3.14159 * r * r; }
};

// CRTP version (from earlier)
// Circle c(5.0);
// c.area();  // direct call, compiler can inline completely
    Key insight: The performance advantage of CRTP comes not just from avoiding the indirect call, but from enabling the optimizer to inline the function body, eliminate dead branches, and apply constant propagation. Virtual calls act as optimization barriers.

    Mixin Classes

    CRTP excels at creating mixins—reusable functionality that can be "mixed in" to any class. The CRTP base provides functionality that automatically adapts to the derived class.

    Equality Mixin

    template <typename Derived>
class EqualityComparable {
public:
    friend bool operator!=(const Derived& a, const Derived& b) {
        return !(a == b);
    }
};

class Point : public EqualityComparable<Point> {
    int x_, y_;
public:
    Point(int x, int y) : x_(x), y_(y) {}

    friend bool operator==(const Point& a, const Point& b) {
        return a.x_ == b.x_ && a.y_ == b.y_;
    }
};

// Point gets != for free by defining only ==
Point p1(1, 2), p2(1, 3);
p1 != p2  // true, generated by the mixin

    Comparison Mixin (Full Ordering)

    template <typename Derived>
class Comparable {
public:
    friend bool operator>(const Derived& a, const Derived& b)  { return b < a; }
    friend bool operator<=(const Derived& a, const Derived& b) { return !(b < a); }
    friend bool operator>=(const Derived& a, const Derived& b) { return !(a < b); }
    friend bool operator==(const Derived& a, const Derived& b) { return !(a < b) && !(b < a); }
    friend bool operator!=(const Derived& a, const Derived& b) { return (a < b) || (b < a); }
};

class Temperature : public Comparable<Temperature> {
    double celsius_;
public:
    explicit Temperature(double c) : celsius_(c) {}

    friend bool operator<(const Temperature& a, const Temperature& b) {
        return a.celsius_ < b.celsius_;
    }
};

// Temperature gets all 6 comparison operators by defining only <
Temperature boiling(100), freezing(0);
boiling > freezing   // true
freezing <= boiling  // true

    Note: In C++20, the spaceship operator <=> makes this mixin less necessary, but CRTP mixins remain valuable for non-comparison functionality.

    Counter Mixin

    template <typename Derived>
class InstanceCounter {
    static inline int count_ = 0;
public:
    InstanceCounter()  { ++count_; }
    ~InstanceCounter() { --count_; }
    InstanceCounter(const InstanceCounter&) { ++count_; }
    InstanceCounter(InstanceCounter&&)      { ++count_; }

    static int instance_count() { return count_; }
};

class Widget : public InstanceCounter<Widget> {};
class Gadget : public InstanceCounter<Gadget> {};

// Each class gets its own counter!
Widget w1, w2, w3;
Gadget g1;
Widget::instance_count()  // 3
Gadget::instance_count()  // 1
    Key insight: Each CRTP instantiation creates a separate base class. InstanceCounter<Widget> and InstanceCounter<Gadget> are unrelated classes with their own static members. This is what makes CRTP mixins per-class rather than global.

    Expression Templates

    Expression templates are an advanced CRTP application used in high-performance linear algebra libraries (Eigen, Blaze, Armadillo). The idea: instead of evaluating A + B + C immediately (creating temporaries), represent the entire expression as a lightweight type and evaluate it lazily in a single pass.

    The Problem

    // Naive vector addition: creates a temporary for each +
Vector result = a + b + c;
// Expands to:
//   Vector temp1 = a + b;    // allocate, loop, store
//   Vector result = temp1 + c; // allocate, loop, store
// Two loops, one unnecessary temporary

    The Solution: CRTP Expression Objects

    // Base expression class (CRTP)
template <typename E>
class VecExpr {
public:
    double operator[](std::size_t i) const {
        return static_cast<const E*>(this)->operator[](i);
    }
    std::size_t size() const {
        return static_cast<const E*>(this)->size();
    }
};

// Concrete vector stores actual data
class Vec : public VecExpr<Vec> {
    std::vector<double> data_;
public:
    Vec(std::size_t n) : data_(n) {}
    Vec(std::initializer_list<double> il) : data_(il) {}

    // Evaluate any expression into this vector
    template <typename E>
    Vec(const VecExpr<E>& expr) : data_(expr.size()) {
        for (std::size_t i = 0; i < data_.size(); ++i)
            data_[i] = expr[i];  // single pass!
    }

    double  operator[](std::size_t i) const { return data_[i]; }
    double& operator[](std::size_t i) { return data_[i]; }
    std::size_t size() const { return data_.size(); }
};

// Expression object for addition (no data, just references)
template <typename E1, typename E2>
class VecAdd : public VecExpr<VecAdd<E1, E2>> {
    const E1& lhs_;
    const E2& rhs_;
public:
    VecAdd(const E1& l, const E2& r) : lhs_(l), rhs_(r) {}

    double operator[](std::size_t i) const {
        return lhs_[i] + rhs_[i];  // computed on the fly
    }
    std::size_t size() const { return lhs_.size(); }
};

// Operator+ returns an expression, not a vector
template <typename E1, typename E2>
VecAdd<E1, E2> operator+(const VecExpr<E1>& a, const VecExpr<E2>& b) {
    return VecAdd<E1, E2>(
        static_cast<const E1&>(a),
        static_cast<const E2&>(b)
    );
}

// Now:
Vec a{1, 2, 3}, b{4, 5, 6}, c{7, 8, 9};
Vec result = a + b + c;
// Type of (a + b + c) is VecAdd<VecAdd<Vec, Vec>, Vec>
// Assignment triggers ONE loop:
//   result[i] = a[i] + b[i] + c[i]
// Zero temporaries, single pass!

    The compiler can inline the entire expression tree, producing code equivalent to a hand-written loop. Eigen achieves performance comparable to hand-tuned BLAS code through this technique.

    Advanced CRTP Patterns

    CRTP with Access Control

    The derived class must grant access for the base to call its methods. Three approaches:

    // Approach 1: Make impl methods public (simplest)
class Circle : public Shape<Circle> {
public:
    double area_impl() const { return 3.14 * r_ * r_; }
};

// Approach 2: Friend the base class
class Circle : public Shape<Circle> {
    friend class Shape<Circle>;
    double area_impl() const { return 3.14 * r_ * r_; }
};

// Approach 3: Private inheritance with using declarations
class Circle : private Shape<Circle> {
    friend class Shape<Circle>;
public:
    using Shape<Circle>::area;  // expose public interface
};

    CRTP Safety: Preventing Misuse

    A common mistake is passing the wrong derived type. You can prevent this with a private constructor trick:

    template <typename Derived>
class Base {
    Base() = default;
    friend Derived;  // only the correct Derived can construct Base

public:
    void interface() {
        static_cast<Derived*>(this)->impl();
    }
};

class Correct : public Base<Correct> {
public:
    void impl() { std::cout << "correct\n"; }
};

// This fails to compile:
// class Wrong : public Base<Correct> {};  // ERROR: Base constructor is private

    Multiple CRTP Bases (Stacking Mixins)

    template <typename Derived>
class Printable {
public:
    void print() const {
        std::cout << static_cast<const Derived*>(this)->to_string() << "\n";
    }
};

template <typename Derived>
class Serializable {
public:
    std::string serialize() const {
        return "{" + static_cast<const Derived*>(this)->to_string() + "}";
    }
};

class Point : public Printable<Point>, public Serializable<Point> {
    int x_, y_;
public:
    Point(int x, int y) : x_(x), y_(y) {}
    std::string to_string() const {
        return "(" + std::to_string(x_) + ", " + std::to_string(y_) + ")";
    }
};

Point p(3, 4);
p.print();       // (3, 4)
p.serialize();   // {(3, 4)}

    CRTP in C++23: Deducing this

    C++23 introduces "deducing this," which provides an alternative to CRTP for many use cases. The member function explicitly declares its this parameter:

    // C++23: no CRTP needed for the mixin pattern
struct EqualityMixin {
    // 'self' deduced as the actual derived type
    friend bool operator!=(this const auto& self, const auto& other) {
        return !(self == other);
    }
};

struct Point : EqualityMixin {
    int x, y;
    friend bool operator==(const Point& a, const Point& b) {
        return a.x == b.x && a.y == b.y;
    }
};
// Point gets != automatically, no template parameter needed
    Key insight: C++23's deducing this eliminates the need for CRTP in mixin scenarios. However, CRTP remains necessary for expression templates, where the derived type must appear in the base class's template parameter list to form the correct expression tree types.

    When to Use CRTP

    • Use CRTP when you need zero-cost polymorphism in performance-critical code and the set of types is known at compile time.
    • Use CRTP to create mixins that inject functionality based on the derived type (counters, comparison operators, serialization).
    • Use CRTP for expression templates in numerical computing.
    • Use virtual functions when you need runtime polymorphism (e.g., plugin systems, heterogeneous containers, type erasure).
    • Use C++20 concepts instead of CRTP when you only need to constrain interfaces without inheriting shared behavior.
    • Use C++23 deducing this instead of CRTP for mixin patterns when your codebase supports C++23.
