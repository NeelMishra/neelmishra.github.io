    C++20 Concepts & Constraints

    March 2026 · 23 min read

    The Problem Concepts Solve

    Templates are powerful but have always suffered from one critical weakness: there is no way to express what a type must support to be used with a template. If you pass the wrong type, the compiler tries to instantiate the template, fails deep inside implementation details, and produces error messages that can span hundreds of lines.

    // What does T need to support?
template <typename T>
T midpoint(T a, T b) {
    return a + (b - a) / 2;  // requires +, -, /
}

midpoint(std::string("a"), std::string("b"));  // compiles, then fails deep inside
// Error message mentions operator/, operator-, none of which mention "midpoint"

    C++20 concepts solve this by letting you declare up front what a type must be able to do. If the type does not satisfy the concept, the compiler rejects it immediately with a clear message at the call site.

    template <typename T>
    requires std::is_arithmetic_v<T>
T midpoint(T a, T b) {
    return a + (b - a) / 2;
}

midpoint(std::string("a"), std::string("b"));
// Error: 'std::string' does not satisfy 'is_arithmetic_v'
// Points to the call site, not the implementation
    Key insight: Concepts bring the same improvement to C++ templates that interfaces bring to dynamically-typed languages. They document requirements, enable better error messages, and participate in overload resolution.

    The requires Clause

    The requires clause is the simplest way to constrain a template. It appears after the template parameter list or after the function declarator:

    Trailing requires

    // After template parameter list
template <typename T>
    requires std::integral<T>
T gcd(T a, T b) {
    while (b) { T t = b; b = a % b; a = t; }
    return a;
}

// After function declarator (trailing)
template <typename T>
T gcd(T a, T b) requires std::integral<T> {
    while (b) { T t = b; b = a % b; a = t; }
    return a;
}

    Both forms are equivalent. Choose whichever reads better in context. The constraint must be a constant expression of type bool.

    Combining Constraints

    // Conjunction (AND)
template <typename T>
    requires std::integral<T> && std::signed_integral<T>
T absolute(T x) {
    return x < 0 ? -x : x;
}

// Disjunction (OR)
template <typename T>
    requires std::integral<T> || std::floating_point<T>
T clamp_positive(T x) {
    return x < T{} ? T{} : x;
}

    The requires Expression

    A requires expression is an expression (not a clause) that tests whether a set of requirements are valid for given types. It evaluates to true or false at compile time:

    template <typename T>
concept Addable = requires(T a, T b) {
    a + b;  // simple requirement: expression must be valid
};

template <typename T>
concept Printable = requires(T x, std::ostream& os) {
    { os << x } -> std::same_as<std::ostream&>;  // compound: valid and returns ostream&
};

    Four Kinds of Requirements

    1. Simple Requirements

    Just an expression. Valid if it compiles:

    requires(T a, T b) {
    a + b;      // T must support +
    a < b;      // T must support <
    a.size();   // T must have a .size() method
}

    2. Type Requirements

    Check that a type name is valid:

    requires {
    typename T::value_type;     // T must have a nested value_type
    typename T::iterator;       // T must have a nested iterator type
    typename std::hash<T>;      // std::hash must be specializable for T
}

    3. Compound Requirements

    Check that an expression is valid, optionally non-throwing, and optionally returns a specific type:

    requires(T a, T b) {
    { a + b } -> std::same_as<T>;        // valid, returns T
    { a.size() } -> std::integral;        // valid, returns an integral type
    { a.swap(b) } noexcept;               // valid and noexcept
    { a == b } noexcept -> std::same_as<bool>;  // all three
}

    4. Nested Requirements

    Embed a boolean constraint inside a requires expression:

    requires {
    requires sizeof(T) <= 16;            // T must be 16 bytes or smaller
    requires std::is_trivially_copyable_v<T>;  // T must be trivially copyable
}

    Defining Concepts

    A concept is a named set of constraints. It is defined with the concept keyword:

    template <typename T>
concept Numeric = std::integral<T> || std::floating_point<T>;

template <typename T>
concept Hashable = requires(T a) {
    { std::hash<T>{}(a) } -> std::convertible_to<std::size_t>;
};

template <typename C>
concept Container = requires(C c) {
    typename C::value_type;
    typename C::iterator;
    { c.begin() } -> std::same_as<typename C::iterator>;
    { c.end() }   -> std::same_as<typename C::iterator>;
    { c.size() }  -> std::integral;
    { c.empty() } -> std::same_as<bool>;
};

    Concepts can build on other concepts:

    template <typename C>
concept SortableContainer = Container<C> && requires(C c) {
    requires std::totally_ordered<typename C::value_type>;
    { c.begin() } -> std::random_access_iterator;
};
    Key insight: Concepts are not types and cannot be instantiated. They are compile-time predicates that evaluate to true or false. Think of them as type-level boolean functions.

    Standard Library Concepts

    The <concepts> header provides a rich set of ready-to-use concepts:

    Core Language Concepts

    Concept Satisfied when
    std::same_as<T, U>T and U are the same type
    std::derived_from<D, B>D publicly derives from B
    std::convertible_to<From, To>From is implicitly and explicitly convertible to To
    std::integral<T>T is an integral type
    std::floating_point<T>T is a floating-point type
    std::signed_integral<T>T is a signed integral type
    std::unsigned_integral<T>T is an unsigned integral type

    Comparison Concepts

    Concept Satisfied when
    std::equality_comparable<T>T supports == and !=
    std::totally_ordered<T>T supports <, >, <=, >=, ==, !=
    std::three_way_comparable<T>T supports <=>

    Callable Concepts

    Concept Satisfied when
    std::invocable<F, Args...>F can be called with Args
    std::regular_invocable<F, Args...>Same, plus equality-preserving
    std::predicate<F, Args...>Invocable and returns bool
     
    // Using standard concepts
template <std::integral T>
T factorial(T n) {
    T result = 1;
    for (T i = 2; i <= n; ++i) result *= i;
    return result;
}

template <std::invocable<int> F>
void apply_to_range(F func, int lo, int hi) {
    for (int i = lo; i <= hi; ++i) func(i);
}

template <std::totally_ordered T>
const T& clamp(const T& val, const T& lo, const T& hi) {
    return (val < lo) ? lo : (hi < val) ? hi : val;
}

    Five Ways to Apply Constraints

    C++20 provides multiple syntaxes to apply concepts. All produce the same result but differ in verbosity and style:

    // 1. Requires clause (most explicit)
template <typename T>
    requires std::integral<T>
T gcd(T a, T b);

// 2. Concept as type constraint
template <std::integral T>
T gcd(T a, T b);

// 3. Trailing requires
template <typename T>
T gcd(T a, T b) requires std::integral<T>;

// 4. Abbreviated function template (constrained auto parameter)
auto gcd(std::integral auto a, std::integral auto b) {
    // Note: a and b can be different integral types
    return /* ... */;
}

// 5. Auto with trailing constraint
auto gcd(auto a, auto b) requires std::integral<decltype(a)> {
    return /* ... */;
}

    Form 2 is the most concise for simple constraints. Form 1 is clearest when constraints involve multiple parameters or complex expressions. Form 4 is great for short utility functions.

    Constrained auto

    C++20 extends auto with concept constraints in multiple contexts:

    // Function parameters (abbreviated template)
void process(std::integral auto value) {
    // value is some integral type
}

// Return type
std::integral auto compute() {
    return 42;
}

// Variable declarations
std::floating_point auto pi = 3.14159;

// Lambda parameters
auto transformer = [](std::integral auto x) { return x * 2; };
    Key insight: std::integral auto x in a function parameter makes the function a template automatically, just like template <std::integral T> void f(T x). Each parameter with auto introduces a new, independent template parameter.

    Concepts vs enable_if: Readability Comparison

    Concepts dramatically improve code readability compared to the C++11/14 enable_if idiom. Here is the same function written both ways:

    Example 1: Simple constraint

    // enable_if (C++11)
template <typename T, std::enable_if_t<std::is_integral_v<T>, int> = 0>
T double_it(T x) { return x * 2; }

// Concepts (C++20)
template <std::integral T>
T double_it(T x) { return x * 2; }

    Example 2: Multiple constraints

    // enable_if (C++11) - barely readable
template <typename C,
    std::enable_if_t<
        std::is_same_v<typename C::value_type,
                       std::decay_t<typename C::value_type>> &&
        std::is_member_function_pointer_v<decltype(&C::push_back)>,
    int> = 0>
void add_element(C& container, typename C::value_type elem);

// Concepts (C++20) - crystal clear
template <typename C>
    requires requires(C c, typename C::value_type v) {
        c.push_back(v);
    }
void add_element(C& container, typename C::value_type elem);

    Example 3: Error messages

    // With enable_if:
// error: no matching function for call to 'double_it'
// note: candidate template ignored: requirement
// 'std::is_integral_v<std::string>' was not satisfied
// [with T = std::basic_string<char>]

// With concepts:
// error: no matching function for call to 'double_it'
// note: constraints not satisfied
// note: because 'std::string' does not satisfy 'integral'

    The concept error message directly names the requirement that failed and is dramatically shorter.

    Concepts and Overload Resolution

    Concepts participate in overload resolution through subsumption. If concept A implies concept B (A subsumes B), then a function constrained with A is more specialized than one constrained with B:

    template <typename T>
concept Arithmetic = std::integral<T> || std::floating_point<T>;

template <Arithmetic T>
std::string describe(T x) { return "arithmetic: " + std::to_string(x); }

template <std::integral T>
std::string describe(T x) { return "integral: " + std::to_string(x); }

describe(42);    // calls integral version (more constrained)
describe(3.14);  // calls arithmetic version (integral doesn't match)

    Subsumption follows logical implication: std::integral implies Arithmetic (since integral is one disjunct of Arithmetic), so the integral overload is more constrained and wins when both match.

    Key insight: For subsumption to work, the more-constrained concept must be defined in terms of the less-constrained one using && and ||. The compiler does not perform semantic analysis—it only follows the syntactic structure of concept definitions.

    Practical Patterns

    Constraining Class Templates

    template <std::totally_ordered T>
class SortedList {
    std::vector<T> data_;
public:
    void insert(const T& val) {
        auto pos = std::lower_bound(data_.begin(), data_.end(), val);
        data_.insert(pos, val);
    }
    bool contains(const T& val) const {
        return std::binary_search(data_.begin(), data_.end(), val);
    }
};

    Constraining Member Functions

    template <typename T>
class Optional {
    // ...
    // Only available if T supports ==
    bool operator==(const Optional& other) const
        requires std::equality_comparable<T>
    {
        if (has_value_ != other.has_value_) return false;
        if (!has_value_) return true;
        return value_ == other.value_;
    }

    // Only available if T supports <<
    friend std::ostream& operator<<(std::ostream& os, const Optional& opt)
        requires requires(std::ostream& o, T v) { o << v; }
    {
        if (opt.has_value_) os << opt.value_;
        else os << "nullopt";
        return os;
    }
};

    Concept-Based Tag Dispatch

    // Different algorithms based on iterator category
template <std::random_access_iterator It>
void sort_impl(It first, It last) {
    std::sort(first, last);  // O(n log n) in-place
}

template <std::forward_iterator It>
void sort_impl(It first, It last) {
    // Forward iterators: copy to vector, sort, copy back
    using T = typename std::iterator_traits<It>::value_type;
    std::vector<T> temp(first, last);
    std::sort(temp.begin(), temp.end());
    std::copy(temp.begin(), temp.end(), first);
}
