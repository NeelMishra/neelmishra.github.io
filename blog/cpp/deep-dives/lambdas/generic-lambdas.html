    Generic & Template Lambdas

    March 2026 · 18 min read

    The Evolution of Lambda Flexibility

    C++11 lambdas required explicit parameter types. C++14 introduced auto parameters, turning lambdas into lightweight templates. C++20 took this further with explicit template parameter lists on lambdas. This progression transformed lambdas from simple inline callbacks into a powerful metaprogramming tool.

    This article traces that evolution, explains the compiler mechanics behind each feature, and demonstrates practical patterns that capitalize on generic lambdas.

    // C++11: explicit types required
auto add11 = [](int a, int b) { return a + b; };

// C++14: auto parameters — generic lambda
auto add14 = [](auto a, auto b) { return a + b; };

// C++20: explicit template parameters on lambdas
auto add20 = []<typename T>(T a, T b) { return a + b; };

    Auto Parameters (C++14)

    When you use auto as a parameter type, the compiler generates a closure with a template operator(). Each auto parameter becomes an independent template parameter, so a lambda with two auto parameters can accept different types for each.

    auto print = [](auto x) { std::cout << x << "\n"; };

print(42);           // operator()<int>(int)
print(3.14);         // operator()<double>(double)
print("hello");      // operator()<const char*>(const char*)
print(std::string("world"));  // operator()<std::string>(std::string)

    What the Compiler Generates

    A generic lambda like [](auto x) { return x * x; } produces a closure class with a template member function:

    // Source:
auto square = [](auto x) { return x * x; };

// Compiler generates:
class __lambda_square {
public:
    template <typename T>
    auto operator()(T x) const {
        return x * x;
    }
};

    The operator() is not instantiated until the lambda is called with a specific type. The compiler generates separate instantiations for each set of argument types used at call sites. This is identical to how regular function templates work.

    Multiple Auto Parameters

    Each auto parameter introduces its own template parameter. The types are deduced independently:

    auto process = [](auto key, auto value) {
    std::cout << key << " = " << value << "\n";
};

// Compiler generates:
// template <typename T1, typename T2>
// void operator()(T1 key, T2 value) const { ... }

process("name", std::string("Alice"));  // T1=const char*, T2=std::string
process(42, 3.14);                       // T1=int, T2=double
process(std::string("key"), 100);        // T1=std::string, T2=int

    If you need both parameters to be the same type, you cannot enforce that with auto alone. You need C++20 template lambdas or a static_assert:

    // C++14: runtime enforcement (weak)
auto same_type_add = [](auto a, auto b) {
    static_assert(std::is_same_v<decltype(a), decltype(b)>,
                  "Both arguments must be the same type");
    return a + b;
};

// C++20: compile-time enforcement (strong)
auto same_type_add = []<typename T>(T a, T b) { return a + b; };
    Key insight: Each auto parameter is an independent template parameter. Two auto parameters can accept different types. If you need them to be the same type, use a C++20 template lambda or static_assert.

    Auto with Qualifiers

    You can combine auto with const, references, and pointers:

    // const reference — avoids copies for large types
auto print = [](const auto& x) { std::cout << x; };

// Universal (forwarding) reference
auto forward_to = [](auto&& x) {
    target(std::forward<decltype(x)>(x));
};

// Pointer
auto deref = [](const auto* p) { return *p; };

// Multiple qualifiers
auto process = [](const auto& container, auto&& callback) {
    for (const auto& elem : container)
        callback(elem);
};

    Perfect Forwarding with Generic Lambdas

    The universally-referenced auto&& pattern enables perfect forwarding inside lambdas. This is essential for writing generic wrapper lambdas that preserve value categories:

    auto wrapper = [](auto&&... args) {
    return target(std::forward<decltype(args)>(args)...);
};

// This lambda perfectly forwards any number of arguments to target()
// Lvalues are forwarded as lvalues; rvalues as rvalues

    The decltype(args) trick is necessary because we cannot name the template parameters directly in a C++14 generic lambda. decltype recovers the exact type (including reference qualifiers) for the forwarding cast.

    Template Lambdas (C++20)

    C++20 allows explicit template parameter lists on lambdas, using the familiar angle-bracket syntax. This gives you the full power of template metaprogramming directly in the lambda declaration.

    // Basic template lambda
auto add = []<typename T>(T a, T b) { return a + b; };
add(1, 2);       // T = int
add(1.5, 2.5);   // T = double
// add(1, 2.5);  // ERROR: T cannot be both int and double

    Why Template Lambdas Matter

    Before C++20, generic lambdas with auto could not constrain the relationship between parameters. You could not enforce "both parameters must be the same type" without static_assert hacks. Template lambdas make this natural:

    // Enforce same type for both parameters
auto max_of = []<typename T>(T a, T b) -> T {
    return a > b ? a : b;
};

// Use non-type template parameters
auto get_nth = []<std::size_t N>(auto&& tuple) {
    return std::get<N>(std::forward<decltype(tuple)>(tuple));
};

auto [x, y] = std::make_tuple(10, 3.14);
auto val = get_nth.operator()<0>(std::make_tuple(10, 3.14));  // 10

    Concepts with Template Lambdas

    C++20 concepts integrate naturally with template lambdas, providing clean constraint syntax:

    // Constrained template lambda
auto numeric_add = []<typename T>(T a, T b) requires std::integral<T> || std::floating_point<T> {
    return a + b;
};

numeric_add(1, 2);       // OK: int is integral
numeric_add(1.5, 2.5);   // OK: double is floating_point
// numeric_add("a", "b"); // ERROR: const char* does not satisfy constraints

// Abbreviated constraint syntax
auto sortable_sort = []<std::ranges::range R>(R&& r) {
    std::ranges::sort(r);
};

    Template Lambdas for Type Manipulation

    // Type dispatching
auto type_name = []<typename T>(T) -> std::string_view {
    if constexpr (std::is_integral_v<T>) return "integral";
    else if constexpr (std::is_floating_point_v<T>) return "floating-point";
    else return "other";
};

type_name(42);    // "integral"
type_name(3.14);  // "floating-point"
type_name("hi");  // "other"

    Template Lambdas with Variadic Parameters

    // Variadic template lambda
auto print_all = []<typename... Ts>(Ts... args) {
    ((std::cout << args << " "), ...);
    std::cout << "\n";
};

print_all(1, 2.5, "hello", 'x');  // "1 2.5 hello x"

// Type-safe format
auto make_tuple_of = []<typename T, typename... Rest>(T first, Rest... rest) {
    return std::tuple{first, rest...};
};
    Key insight: Template lambdas give you explicit control over template parameters that generic lambdas with auto cannot provide. Use them when you need: same-type enforcement, non-type template parameters, concept constraints, or explicit template argument specification at the call site.

    Constexpr Lambdas (C++17)

    Since C++17, lambdas are implicitly constexpr if their body satisfies constexpr requirements. This means lambdas can execute at compile time, enabling powerful compile-time computation without the verbosity of traditional constexpr functions.

    // Implicitly constexpr since C++17
auto square = [](int x) { return x * x; };

// Can be used in compile-time contexts:
constexpr int result = square(5);   // computed at compile time: 25
static_assert(square(5) == 25);

// Array size determined by lambda at compile time
constexpr auto size = [](int rows, int cols) { return rows * cols; };
std::array<int, size(3, 4)> matrix;  // std::array<int, 12>

    Explicit constexpr

    You can explicitly mark a lambda constexpr to get a compile error if the body cannot be evaluated at compile time:

    // Explicitly constexpr — compiler error if body is not constexpr-capable
auto factorial = [](int n) constexpr -> int {
    int result = 1;
    for (int i = 2; i <= n; ++i) result *= i;
    return result;
};

constexpr int f5 = factorial(5);  // 120, computed at compile time

// This would fail to compile if we added non-constexpr operations:
// auto bad = [](int n) constexpr {
//     std::cout << n;  // ERROR: cout not constexpr
//     return n;
// };

    Constexpr and Captures

    constexpr int base = 10;
auto offset = [base](int x) constexpr { return x + base; };
static_assert(offset(5) == 15);

// Non-constexpr capture prevents compile-time evaluation
int runtime_val = get_value();
auto f = [runtime_val](int x) { return x + runtime_val; };
// f(5) works at runtime, but NOT at compile time

    Constexpr Lambdas in Template Metaprogramming

    // Compile-time string length
constexpr auto str_len = [](const char* s) {
    std::size_t n = 0;
    while (s[n] != '\0') ++n;
    return n;
};
static_assert(str_len("hello") == 5);

// Compile-time array transformation
template <std::size_t N, typename F>
constexpr auto transform_array(const std::array<int, N>& arr, F f) {
    std::array<int, N> result{};
    for (std::size_t i = 0; i < N; ++i) result[i] = f(arr[i]);
    return result;
}

constexpr auto doubled = transform_array(
    std::array{1, 2, 3, 4, 5},
    [](int x) { return x * 2; }
);
static_assert(doubled[2] == 6);
    Key insight: Since C++17, you do not need to explicitly mark lambdas constexpr. The compiler automatically makes them constexpr if possible. Explicitly marking them serves as a useful assertion: "I intend this to work at compile time, and I want a compiler error if it cannot."

    Stateless Lambda to Function Pointer

    A non-generic lambda with no captures can implicitly convert to a plain C function pointer. This feature enables seamless interoperability with C APIs, callback registration systems, and signal handlers.

    // Stateless, non-generic: converts to function pointer
auto add = [](int a, int b) { return a + b; };
int (*fptr)(int, int) = add;  // OK

// C API interop
extern "C" void register_signal_handler(void (*handler)(int));
register_signal_handler([](int sig) {
    // handle signal
});

    Generic Lambdas and Function Pointers

    A generic lambda cannot convert to a function pointer directly, because the compiler does not know which instantiation to use. However, you can explicitly instantiate it:

    auto generic_add = [](auto a, auto b) { return a + b; };

// Cannot convert directly:
// int (*p)(int, int) = generic_add;  // ERROR

// But you can call the specific instantiation:
int (*p)(int, int) = generic_add.operator()<int, int>;
// Note: this syntax is not standard — use a wrapper instead

// Practical workaround: wrap in a non-generic lambda
int (*p)(int, int) = [](int a, int b) { return a + b; };

    When Function Pointer Conversion Is Useful

    • C library callbacks: qsort, bsearch, signal handlers, pthread functions
    • Plugin architectures: loading function pointers from shared libraries
    • Virtual dispatch avoidance: using function pointer tables instead of virtual functions
    • Storing in arrays: void (*handlers[10])(int) can hold stateless lambdas
    // Array of function pointers, populated with stateless lambdas
using Op = double (*)(double, double);
Op operations[] = {
    [](double a, double b) { return a + b; },
    [](double a, double b) { return a - b; },
    [](double a, double b) { return a * b; },
    [](double a, double b) { return b != 0.0 ? a / b : 0.0; }
};

operations[2](3.0, 4.0);  // 12.0 (multiply)

    The IIFE Pattern

    An Immediately Invoked Function Expression (IIFE) is a lambda that is defined and called in the same statement. The pattern is [...]() { ... }(), note the trailing () that immediately invokes the lambda.

    // Simple IIFE: complex initialization of a const variable
const auto config = [&]() {
    Config c;
    c.host = env("HOST", "localhost");
    c.port = std::stoi(env("PORT", "8080"));
    c.max_connections = 100;
    return c;
}();
// config is const and fully initialized

    When to Use IIFE

    The IIFE pattern solves a specific problem: initializing a const variable that requires multiple statements of setup logic. Without IIFE, you would need to either make the variable non-const, or extract the initialization into a separate named function for this one-off computation.

    // Without IIFE: non-const, risk of accidental modification
Widget widget;
widget.setName("main");
widget.setSize(100, 200);
if (high_dpi) widget.setScale(2.0);
// widget can be accidentally modified later

// With IIFE: const, initialization is complete and locked
const Widget widget = [&]() {
    Widget w;
    w.setName("main");
    w.setSize(100, 200);
    if (high_dpi) w.setScale(2.0);
    return w;
}();
// widget cannot be modified — const

    IIFE for Conditional Initialization

    // Clean conditional initialization
const auto renderer = [&]() -> std::unique_ptr<Renderer> {
    if (gpu_available) return std::make_unique<GpuRenderer>();
    if (software_fallback) return std::make_unique<SoftwareRenderer>();
    return std::make_unique<NullRenderer>();
}();

// Without IIFE, you'd need a helper function or non-const variable

    IIFE with Generic Lambdas

    // std::variant visitor using IIFE
std::variant<int, std::string, double> v = 3.14;

auto result = std::visit([](auto&& val) -> std::string {
    using T = std::decay_t<decltype(val)>;
    if constexpr (std::is_same_v<T, int>)
        return "int: " + std::to_string(val);
    else if constexpr (std::is_same_v<T, std::string>)
        return "string: " + val;
    else if constexpr (std::is_same_v<T, double>)
        return "double: " + std::to_string(val);
}, v);
    Key insight: IIFE is not just a syntactic trick. It solves a real software engineering problem: const-correct initialization of complex objects. The pattern keeps initialization logic adjacent to the variable, avoids polluting the namespace with helper functions, and guarantees the result is immutable.

    Generic Lambdas with the Standard Library

    Generic lambdas unlock powerful patterns with standard algorithms by letting you write type-agnostic operations inline:

    Polymorphic Comparators

    auto less_by_abs = [](const auto& a, const auto& b) {
    return std::abs(a) < std::abs(b);
};

std::vector<int> ints = {-5, 3, -1, 4, -2};
std::sort(ints.begin(), ints.end(), less_by_abs);
// {-1, -2, 3, 4, -5}

std::vector<double> doubles = {-3.14, 1.5, -0.5};
std::sort(doubles.begin(), doubles.end(), less_by_abs);
// Same lambda works for doubles

    Generic Projection

    auto get_name = [](const auto& obj) -> decltype(auto) { return obj.name(); };

// Works with any type that has a name() method:
std::sort(employees.begin(), employees.end(),
    [](const auto& a, const auto& b) { return a.name() < b.name(); });

std::sort(products.begin(), products.end(),
    [](const auto& a, const auto& b) { return a.name() < b.name(); });

    Transparent Functors Replacement

    // Before generic lambdas: std::less<> (transparent comparator)
std::set<std::string, std::less<>> s;

// With generic lambdas: custom transparent comparators inline
auto case_insensitive = [](const auto& a, const auto& b) {
    auto to_lower = [](unsigned char c) { return std::tolower(c); };
    return std::lexicographical_compare(
        a.begin(), a.end(), b.begin(), b.end(),
        [&](char ca, char cb) { return to_lower(ca) < to_lower(cb); }
    );
};

std::set<std::string, decltype(case_insensitive)> names(case_insensitive);

    Return Type Deduction with Generic Lambdas

    By default, generic lambdas use auto return type deduction, which strips references and cv-qualifiers. When you need to preserve the exact type (including references), use decltype(auto):

    std::vector<int> v = {1, 2, 3};

// auto return: returns a copy
auto get_copy = [&v](int i) { return v[i]; };
get_copy(0) = 42;  // modifies the copy, not v[0]

// decltype(auto) return: preserves reference
auto get_ref = [&v](int i) -> decltype(auto) { return v[i]; };
get_ref(0) = 42;   // modifies v[0] directly
    // Perfect forwarding of return value
auto make_forwarder = [](auto&& func) {
    return [f = std::forward<decltype(func)>(func)](auto&&... args) -> decltype(auto) {
        return f(std::forward<decltype(args)>(args)...);
    };
};

    if constexpr in Generic Lambdas

    The if constexpr (C++17) and generic lambdas are a powerful combination. You can branch at compile time based on the deduced type, eliminating code paths that do not apply to a given instantiation:

    auto to_string_generic = [](const auto& x) -> std::string {
    using T = std::decay_t<decltype(x)>;
    if constexpr (std::is_same_v<T, std::string>)
        return x;
    else if constexpr (std::is_arithmetic_v<T>)
        return std::to_string(x);
    else if constexpr (std::is_same_v<T, bool>)
        return x ? "true" : "false";
    else
        return "[unknown type]";
};

to_string_generic(42);            // "42"
to_string_generic(3.14);          // "3.140000"
to_string_generic(std::string("hi"));  // "hi"
    // Container-aware printer
auto print_any = [](const auto& x) {
    using T = std::decay_t<decltype(x)>;
    if constexpr (requires { x.begin(); x.end(); }) {
        std::cout << "[";
        bool first = true;
        for (const auto& elem : x) {
            if (!first) std::cout << ", ";
            std::cout << elem;
            first = false;
        }
        std::cout << "]";
    } else {
        std::cout << x;
    }
    std::cout << "\n";
};

print_any(42);                           // 42
print_any(std::vector{1, 2, 3});         // [1, 2, 3]
print_any(std::string("hello"));         // [h, e, l, l, o] — string is iterable
    Key insight: if constexpr inside a generic lambda gives you compile-time type dispatching without the complexity of template specialization or tag dispatch. The discarded branches do not need to compile for the instantiated types, making it a clean way to write type-aware logic.

    Practical Patterns

    Overloaded Lambda (Visitor Pattern)

    // Helper to create an overloaded set of lambdas
template <typename... Ts>
struct overloaded : Ts... { using Ts::operator()...; };
template <typename... Ts>
overloaded(Ts...) -> overloaded<Ts...>;

// Usage with std::variant
std::variant<int, std::string, double> v = "hello";
std::visit(overloaded{
    [](int i)               { std::cout << "int: " << i; },
    [](const std::string& s) { std::cout << "string: " << s; },
    [](double d)            { std::cout << "double: " << d; }
}, v);

    Compile-Time Dispatch Table

    // Type-indexed dispatch using template lambdas (C++20)
auto dispatch = []<typename T>(T value) {
    if constexpr (std::is_integral_v<T>)
        return static_cast<long long>(value);
    else if constexpr (std::is_floating_point_v<T>)
        return static_cast<double>(value);
    else
        return std::string(value);
};

    Generic Lambda as a Policy

    // Retry policy as a generic lambda
auto exponential_retry = [](auto operation, int max_attempts) {
    for (int attempt = 0; attempt < max_attempts; ++attempt) {
        try {
            return operation();
        } catch (const std::exception&) {
            if (attempt == max_attempts - 1) throw;
            std::this_thread::sleep_for(
                std::chrono::milliseconds(100 * (1 << attempt))
            );
        }
    }
    throw std::runtime_error("unreachable");
};

auto result = exponential_retry([]() {
    return fetch_data("https://api.example.com/data");
}, 3);

    Best Practices

    1. Use const auto& for read-only parameters to avoid unnecessary copies. Bare auto copies the argument.
    2. Use auto&& for perfect forwarding when the lambda wraps another callable.
    3. Prefer C++20 template lambdas over static_assert hacks when you need type constraints.
    4. Combine if constexpr with generic lambdas for type-aware logic instead of overload sets where possible.
    5. Use IIFE for const initialization when the initialization logic requires multiple statements.
    6. Remember that generic lambdas generate code per instantiation. Avoid using them in hot-path inner loops where you want to control binary size.

    Summary

    Generic lambdas (C++14) turned lambdas into lightweight templates. Template lambdas (C++20) brought full template parameter control. Constexpr lambdas (C++17) enabled compile-time computation. Together, these features make lambdas one of the most versatile tools in modern C++.
