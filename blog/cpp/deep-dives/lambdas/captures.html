    Lambda Captures Deep Dive

    March 2026 · 20 min read

    Why Captures Matter

    Captures are what elevate lambdas from mere inline functions to closures: functions that carry state from their environment. Every time you write a lambda that references a local variable, you are making a decision about ownership, lifetime, and mutability. Getting captures wrong leads to some of the most insidious bugs in C++: dangling references, unexpected mutations, and performance pitfalls from accidental deep copies.

    In the previous article, we saw that the compiler transforms each capture into a data member of the closure class. In this article, we will examine every capture mode in detail, explore their interactions, and catalog the bugs that arise from each.

    Capture by Value

    When you capture a variable by value, the closure stores a copy of the variable at the point where the lambda expression is evaluated. Changes to the original variable after the lambda is created do not affect the captured copy, and vice versa (unless you use mutable).

    int factor = 10;
auto scale = [factor](int x) { return x * factor; };

factor = 20;        // modifying original
scale(5);           // returns 50, not 100 — uses the captured copy (10)

    Const by Default

    The captured copy is accessed through a const member in the generated operator(). You cannot modify it unless you declare the lambda mutable.

    int n = 0;
// auto inc = [n]() { return ++n; };  // ERROR: n is const in operator()
auto inc = [n]() mutable { return ++n; };  // OK

inc();  // returns 1
inc();  // returns 2
// n is still 0 — the original is untouched

    Copy Cost

    Capturing by value invokes the copy constructor. For lightweight types like int or double, this is trivial. For heavy types like std::vector, std::string, or std::map, the copy can be expensive.

    std::vector<int> big_data(1'000'000);

// This copies the entire million-element vector into the closure
auto process = [big_data]() {
    return std::accumulate(big_data.begin(), big_data.end(), 0LL);
};

// Better: capture by reference if the lambda won't outlive big_data
auto process_ref = [&big_data]() {
    return std::accumulate(big_data.begin(), big_data.end(), 0LL);
};

// Or: use init-capture to move (covered later in this article)
auto process_move = [data = std::move(big_data)]() {
    return std::accumulate(data.begin(), data.end(), 0LL);
};
    Key insight: Capture by value creates a copy at lambda creation time, not at each call. The copy constructor runs once. If the type is expensive to copy, consider capture by reference (if safe) or move-capture (C++14).

    When to Use Capture by Value

    • When the lambda outlives the captured variable (returned from a function, stored in a member, passed to an async operation).
    • When you need a snapshot of the variable's state at lambda creation time.
    • When the variable is cheap to copy (fundamental types, small structs).
    • When you want thread safety: each thread's lambda gets its own independent copy.

    Capture by Reference

    When you capture by reference, the closure stores a reference (internally, a pointer) to the original variable. Every access inside the lambda reads or writes the original. No copy is made.

    int total = 0;
auto accumulate = [&total](int x) { total += x; };

accumulate(10);
accumulate(20);
accumulate(30);
// total == 60 — the original was modified

    Reference Captures and const

    Even though the lambda's operator() is const, you can still modify variables captured by reference. The const applies to the closure's members (the references themselves), and a const reference cannot be reseated but can still modify the referenced object.

    int x = 1;
auto f = [&x]() { x = 42; };  // OK: modifies x through reference
// f's operator() is const, but x is accessed through a reference member
// The reference itself doesn't change (it can't be reseated), only the value does

    The Dangling Reference Trap

    This is the single most dangerous aspect of lambda captures. If the original variable is destroyed while the lambda still holds a reference to it, you get undefined behavior. The reference dangles, and any access through it reads or writes garbage memory.

    std::function<int()> make_adder(int base) {
    return [&base](int x) { return base + x; };
    // BUG: 'base' is a parameter — destroyed when make_adder returns
    // The returned lambda holds a dangling reference
}

auto add5 = make_adder(5);
add5(10);  // UNDEFINED BEHAVIOR: reading destroyed stack memory

    The fix is straightforward: capture by value when the lambda escapes its scope.

    std::function<int(int)> make_adder(int base) {
    return [base](int x) { return base + x; };  // SAFE: captures a copy
}

    When to Use Capture by Reference

    • When the lambda does not outlive the captured variable (used immediately, passed to an algorithm that completes before the scope ends).
    • When you need to mutate the original variable from inside the lambda.
    • When the variable is expensive to copy and a reference is safe.
    • When collecting results: the lambda writes into a container owned by the calling scope.

    Default Captures: [=] and [&]

    Default captures tell the compiler to automatically capture any local variable that the lambda body references. [=] captures everything by value; [&] captures everything by reference.

    int a = 1, b = 2, c = 3;

auto by_val = [=]() { return a + b + c; };   // copies a, b, c
auto by_ref = [&]() { return a + b + c; };   // references a, b, c

    You can mix default with explicit captures to override the default for specific variables:

    int x = 10, y = 20, z = 30;

auto mix1 = [=, &z]() { return x + y + z; };
// x, y captured by value; z captured by reference

auto mix2 = [&, x]() { return x + y + z; };
// y, z captured by reference; x captured by value

    Problems with Default Captures

    Default captures are convenient but hide important information. They make it unclear which variables are being captured and how. This leads to several categories of bugs:

    1. Accidental Capture of Large Objects

    void process(const std::vector<int>& data, int threshold) {
    // Developer intends to capture only 'threshold'
    // But [=] also copies 'data' — an expensive deep copy
    auto filter = [=](int x) { return x > threshold; };
    // Wait — 'data' is not used in the body, so it's NOT captured
    // But this only works because the compiler is smart about it
    // A slight change to the body could accidentally pull in data
}

    2. Implicit Capture of this

    class Widget {
    int value_ = 42;
public:
    auto get_checker() {
        return [=]() { return value_ > 0; };
        // Looks safe — "everything by value"
        // But [=] captures 'this' pointer, NOT value_
        // If Widget is destroyed, the lambda has a dangling this
    }
};
    Key insight: Inside a member function, [=] captures this by pointer, not by copying each member. This is one of the most common sources of dangling pointer bugs with lambdas. Always prefer explicit captures in member functions.

    3. Readability Problems

    // Which variables does this lambda actually use?
auto result = [&]() {
    auto filtered = filter(data, threshold);
    transform(filtered, output, mapper);
    return summarize(output);
};
// Without reading the body carefully, you can't know what's captured
// Or whether the captures are safe

    The C++ Core Guidelines recommend avoiding default captures and preferring explicit capture lists. This makes the lambda's dependencies visible at a glance.

    Capturing this vs *this

    Inside a member function, you often need the lambda to access the object's members. There are two ways to do this, and the difference is critical for correctness.

    Capturing this (Pointer Capture)

    Capturing this stores the raw pointer to the current object. The lambda can access all members through this pointer. But if the object is destroyed before the lambda runs, the pointer dangles.

    class Sensor {
    double last_reading_ = 0.0;
    std::string name_;
public:
    auto make_callback() {
        return [this]() {
            // Accesses members through the captured 'this' pointer
            std::cout << name_ << ": " << last_reading_ << "\n";
        };
    }
};

Sensor s;
auto cb = s.make_callback();
// If 's' is destroyed here...
cb();  // UNDEFINED BEHAVIOR: dangling this pointer

    Capturing *this (Copy Capture, C++17)

    C++17 introduced [*this], which captures a copy of the entire object. The lambda holds its own independent copy, so it is safe even if the original object is destroyed.

    class Sensor {
    double last_reading_ = 0.0;
    std::string name_;
public:
    auto make_safe_callback() {
        return [*this]() {   // C++17: copies the entire Sensor object
            std::cout << name_ << ": " << last_reading_ << "\n";
        };
    }
};

    The tradeoff is obvious: [*this] copies the entire object, which can be expensive for large objects. But it eliminates the dangling pointer problem entirely.

    Comparison Table

    [=] inside memberPointer to object (same as [this])Dangling if object dies8 bytes
     
    class Timer {
    int interval_;
    std::string label_;
public:
    // DANGEROUS: [=] inside a member captures 'this', not copies of members
    auto make_callback_v1() {
        return [=]() { std::cout << label_ << ": " << interval_ << "\n"; };
    }

    // SAFE (C++17): [*this] copies the entire object
    auto make_callback_v2() {
        return [*this]() { std::cout << label_ << ": " << interval_ << "\n"; };
    }

    // SAFE (C++11): explicitly copy the members you need
    auto make_callback_v3() {
        auto lbl = label_;
        auto intv = interval_;
        return [lbl, intv]() { std::cout << lbl << ": " << intv << "\n"; };
    }
};

    Init-Capture (Generalized Lambda Capture, C++14)

    C++14 introduced init-captures (also called generalized lambda captures), which let you create new variables inside the capture list and initialize them with arbitrary expressions. The syntax is [name = expression].

    // Create a new capture variable initialized from an expression
auto f = [x = compute_value()](int n) { return x + n; };

// The capture 'x' is a new variable in the closure, distinct from any outer 'x'
// It is initialized by calling compute_value() when the lambda is created

    Moving into Captures

    The most important use of init-captures is moving objects into the closure. Before C++14, there was no way to move a variable into a lambda; you could only copy or reference.

    auto ptr = std::make_unique<Widget>();

// Can't capture unique_ptr by value — it's move-only
// auto f = [ptr]() { ptr->do_something(); };  // ERROR: unique_ptr is not copyable

// Init-capture with std::move: transfers ownership into the closure
auto f = [p = std::move(ptr)]() { p->do_something(); };

// ptr is now null — ownership transferred to the lambda

    Renaming Captures

    int data = 42;
auto f = [captured_data = data](int x) { return x + captured_data; };
// 'captured_data' is the name inside the lambda; 'data' is the original

    Computed Init-Captures

    auto f = [sum = a + b, product = a * b]() {
    return sum > product;
};

// More complex expressions
auto g = [config = load_config("settings.json")]() {
    return config.get("debug_mode");
};

    Init-Capture for Move-Only Types

    // Move a vector into the lambda
std::vector<int> data = {1, 2, 3, 4, 5};
auto process = [d = std::move(data)]() {
    int sum = 0;
    for (int x : d) sum += x;
    return sum;
};
// data is now empty — contents moved into the closure

// Move a unique_ptr into the lambda
auto connection = std::make_unique<DbConnection>("localhost");
auto query = [conn = std::move(connection)](const std::string& sql) {
    return conn->execute(sql);
};
    Key insight: Init-captures solve two problems at once: they allow moving non-copyable objects into lambdas, and they allow you to name captured values differently from their source. The expression on the right side of = is evaluated once, when the lambda is created.

    Dangling Capture Patterns

    Dangling captures are the most dangerous class of lambda bugs. They cause undefined behavior, which can manifest as crashes, corrupted data, or (worst of all) seemingly correct behavior that breaks unpredictably.

    Pattern 1: Returning a Lambda with Reference Captures

    // BUG: lambda outlives the captured reference
auto make_multiplier(int factor) {
    return [&factor](int x) { return x * factor; };
    // 'factor' is a function parameter — destroyed at return
}

auto mul5 = make_multiplier(5);
mul5(10);  // UNDEFINED BEHAVIOR

    Pattern 2: Async Operations with Reference Captures

    void start_task() {
    std::string filename = "data.csv";

    // BUG: filename destroyed when start_task returns
    // but the async operation still holds a reference to it
    std::async(std::launch::async, [&filename]() {
        process_file(filename);  // UNDEFINED BEHAVIOR
    });
}

// FIX: capture by value
void start_task_safe() {
    std::string filename = "data.csv";
    std::async(std::launch::async, [filename]() {
        process_file(filename);  // SAFE: lambda owns a copy
    });
}

    Pattern 3: Capturing Loop Variables by Reference

    std::vector<std::function<int()>> getters;
for (int i = 0; i < 5; ++i) {
    getters.push_back([&i]() { return i; });
    // All lambdas reference the SAME 'i'
}
// After the loop, 'i' is out of scope
// Calling any getter is UNDEFINED BEHAVIOR

// FIX: capture by value
for (int i = 0; i < 5; ++i) {
    getters.push_back([i]() { return i; });
    // Each lambda gets its own copy: 0, 1, 2, 3, 4
}

    Pattern 4: Capturing Temporaries

    auto make_greeter() {
    // BUG: reference to temporary string
    const auto& name = get_name();  // if get_name returns by value,
                                     // this extends the temporary's lifetime...
    return [&name]() {
        std::cout << "Hello " << name;
    };
    // ...but only until the end of this function
    // The returned lambda holds a dangling reference
}

    Capturing Move-Only Types

    Some types in C++ cannot be copied: std::unique_ptr, std::thread, std::promise, and any custom type with a deleted copy constructor. To capture these in a lambda, you must use init-captures with std::move.

    auto ptr = std::make_unique<Resource>();

// Compile error: unique_ptr is not copyable
// auto f = [ptr]() { ptr->use(); };

// Solution: move into the capture
auto f = [p = std::move(ptr)]() { p->use(); };
// ptr is now nullptr — the lambda owns the Resource

    Moving Multiple Objects

    auto conn = std::make_unique<Connection>();
auto buffer = std::make_unique<Buffer>(4096);
std::thread worker;

auto task = [
    c = std::move(conn),
    b = std::move(buffer)
]() mutable {
    c->send(b->data(), b->size());
};

    The mutable Interaction

    When you move a unique_ptr into a lambda, calling methods on it does not require mutable because you are calling methods on the pointee, not modifying the pointer itself. But if you want to reset or reassign the pointer, you need mutable.

    auto f = [p = std::make_unique<Widget>()]() {
    p->display();  // OK: const unique_ptr allows calling non-const methods on pointee
};

auto g = [p = std::make_unique<Widget>()]() mutable {
    p->display();
    p.reset();     // Needs mutable: modifying the unique_ptr itself
    p = std::make_unique<Widget>();  // Also needs mutable
};

    Capture Packing and Compiler Optimizations

    The compiler packs captures into the closure object like struct members, with alignment and padding rules. Understanding this helps you reason about closure sizes and cache performance.

    char a = 'x';
int b = 42;
double c = 3.14;

auto f1 = [a, b, c]() { return a + b + c; };
// Likely layout: char a (1 byte) + 3 padding + int b (4 bytes) + double c (8 bytes)
// sizeof(f1) is likely 16

auto f2 = [c, b, a]() { return a + b + c; };
// Compiler may reorder: double c (8 bytes) + int b (4 bytes) + char a (1 byte) + 3 padding
// sizeof(f2) is also likely 16

// The compiler can reorder captures for optimal packing
// Do NOT rely on any specific capture ordering in memory

    Empty Closure Optimization

    A lambda with no captures has an empty closure type. The compiler applies the empty base optimization (EBO), so sizeof is 1 (the minimum for any C++ object). When such a lambda is passed to a template, the compiler often eliminates the closure object entirely and inlines the body.

    auto cmp = [](int a, int b) { return a < b; };
static_assert(sizeof(cmp) == 1);

// When passed to std::sort, the closure object is optimized away
// The comparison becomes a direct inline operation
std::sort(v.begin(), v.end(), cmp);

    Structured Bindings and Captures (C++17/C++20)

    C++17 introduced structured bindings, but initially you could not capture them in lambdas. C++20 fixed this limitation.

    std::pair<int, std::string> p = {42, "hello"};
auto [num, text] = p;

// C++17: Cannot capture structured bindings — compile error
// auto f = [num, text]() { return text + std::to_string(num); };

// C++20: This is now allowed
auto f = [num, text]() { return text + std::to_string(num); };

// Workaround for C++17:
auto g = [n = num, t = text]() { return t + std::to_string(n); };

    Capture Scope and Visibility Rules

    Only variables with automatic storage duration (local variables and function parameters) can be captured. The following cannot be captured:

    • Global variables: Accessible directly, no capture needed.
    • Static local variables: Accessible directly.
    • Constexpr variables: Accessible directly (the value is a compile-time constant).
    • Thread-local variables: Accessible directly on the same thread.
    int global = 100;
static int file_scope = 200;

void example() {
    static int func_static = 300;
    constexpr int compile_const = 400;
    int local = 500;

    auto f = [local]() {
        // global: accessible without capture
        // file_scope: accessible without capture
        // func_static: accessible without capture
        // compile_const: accessible without capture (value substituted)
        // local: must be captured
        return global + file_scope + func_static + compile_const + local;
    };
}
    Key insight: If a variable does not need to be captured (globals, statics, constexpr), it is not stored in the closure and does not affect the closure's size. Only automatic (stack) variables from the enclosing scope become closure members.

    Advanced Capture Patterns

    Capturing Parameter Packs (C++20)

    template <typename... Args>
auto make_tuple_printer(Args... args) {
    // C++20: capture a parameter pack by value
    return [... captured = std::move(args)]() {
        ((std::cout << captured << " "), ...);
        std::cout << "\n";
    };
}

auto printer = make_tuple_printer(1, 3.14, std::string("hello"));
printer();  // "1 3.14 hello"

    Self-Referencing Captures

    // Capturing a shared_ptr to enable shared ownership
auto shared = std::make_shared<Widget>();

auto callback = [shared]() {
    shared->process();
};
// The lambda keeps the Widget alive through shared ownership
// Even if the original shared_ptr is reset, the lambda's copy keeps it alive

shared.reset();    // Widget still alive — lambda holds a copy of the shared_ptr
callback();        // SAFE: Widget still exists

    Capture and Forward

    // Perfect forwarding with init-captures
template <typename T>
auto wrap(T&& arg) {
    return [val = std::forward<T>(arg)]() mutable {
        return process(std::move(val));
    };
}

// If called with an lvalue: val is copy-constructed
// If called with an rvalue: val is move-constructed

    Best Practices for Captures

    1. Prefer explicit captures over default captures. Write [x, &y] instead of [=] or [&]. This documents the lambda's dependencies.
    2. Capture by value when the lambda might escape. Any lambda returned from a function, stored in a member, or passed to an async operation should capture by value.
    3. Use init-captures to move when capturing expensive or non-copyable types.
    4. In member functions, avoid [=]. It captures this by pointer, not by copying members. Use [*this] (C++17) or explicitly copy the members you need.
    5. Watch for loop variable captures. Capturing a loop variable by reference is almost always a bug when the lambda outlives the loop iteration.
    6. Consider closure size. Each value capture adds to the closure's footprint. For hot paths, prefer reference captures (if safe) to avoid copies.
    7. Use std::shared_ptr for shared ownership. When multiple lambdas need access to the same resource, capture a shared_ptr by value.

    Summary

    Lambda captures define the relationship between a closure and its environment. Capture by value creates independent copies; capture by reference creates aliases to the originals. Default captures hide dependencies and invite bugs, especially the implicit capture of this inside member functions. Init-captures (C++14) and *this capture (C++17) filled critical gaps: moving non-copyable types into closures and safely copying the enclosing object.

    The cardinal rule is simple: if the lambda outlives the captured variable, capture by value. If you need the variable alive and the lambda is short-lived, capture by reference. When in doubt, capture by value.
