    unique_ptr Internals

    March 2026 · 18 min read

    The Zero-Overhead Principle

    std::unique_ptr is the poster child for the C++ zero-overhead principle: you don't pay for what you don't use, and what you do use, you couldn't hand-code any better. A unique_ptr with a stateless deleter compiles down to the exact same machine code as a raw pointer with manual delete. No extra memory. No extra indirection. No extra branches.

    This is not an accident. It is the result of several deliberate design decisions in the standard library: the choice to store the deleter as a compressed pair, the use of empty base optimization, and the requirement that ownership transfer happens exclusively through std::move. Understanding these decisions will make you a better C++ programmer, even if you never write your own smart pointer.

    Core guarantee: sizeof(std::unique_ptr<T>) == sizeof(T*) when the deleter is stateless. This means zero memory overhead compared to a raw pointer.

    Why does this matter? Because every abstraction has a cost in most languages. Java's Optional allocates. Python's context managers add overhead. Rust's Box is zero-cost like unique_ptr, but this is the exception rather than the rule across programming languages. In C++, the standard library is engineered so that the safe, correct abstraction is also the fastest one.

    How unique_ptr Is Implemented

    At its heart, unique_ptr is astonishingly simple. If you strip away all the SFINAE, conditional noexcept, and compatibility machinery, the core looks like this:

    template <typename T, typename Deleter = std::default_delete<T>>
class unique_ptr {
    T* ptr_;           // the raw pointer
    Deleter deleter_;  // the deleter (often zero bytes!)

public:
    explicit unique_ptr(T* p = nullptr) noexcept : ptr_(p) {}

    ~unique_ptr() {
        if (ptr_) deleter_(ptr_);
    }

    // Move constructor: steal the pointer
    unique_ptr(unique_ptr&& other) noexcept
        : ptr_(other.release()), deleter_(std::move(other.deleter_)) {}

    // Move assignment
    unique_ptr& operator=(unique_ptr&& other) noexcept {
        reset(other.release());
        deleter_ = std::move(other.deleter_);
        return *this;
    }

    // No copies
    unique_ptr(const unique_ptr&) = delete;
    unique_ptr& operator=(const unique_ptr&) = delete;

    T* release() noexcept {
        T* old = ptr_;
        ptr_ = nullptr;
        return old;
    }

    void reset(T* p = nullptr) noexcept {
        T* old = ptr_;
        ptr_ = p;
        if (old) deleter_(old);
    }

    T& operator*() const { return *ptr_; }
    T* operator->() const noexcept { return ptr_; }
    T* get() const noexcept { return ptr_; }
    explicit operator bool() const noexcept { return ptr_ != nullptr; }
};

    That is the entire mechanism. There is no reference counting, no control block, no atomic operations, no heap allocation beyond the managed object itself. The destructor calls the deleter on the raw pointer. The move constructor steals the pointer and nulls the source. Copy operations are deleted entirely.

    std::default_delete

    The default deleter is even simpler:

    template <typename T>
struct default_delete {
    void operator()(T* ptr) const { delete ptr; }
};

// Array specialization
template <typename T>
struct default_delete<T[]> {
    void operator()(T* ptr) const { delete[] ptr; }
};

    Notice that default_delete has no data members. It is an empty class. Its sizeof is 1 (C++ requires every object to have a unique address), but the compiler can optimize this away when it appears as a base class or member of a compressed pair. This is where empty base optimization enters the picture.

    Empty Base Optimization (EBO)

    Empty base optimization (EBO) is a compiler optimization that allows an empty base class to occupy zero bytes in the derived class layout. The C++ standard mandates that a standalone object must have sizeof >= 1, but a base class subobject with no data members can be compressed to zero bytes.

    Real standard library implementations (libstdc++, libc++, MSVC STL) store the pointer and deleter together in a compressed pair that exploits EBO:

    // Simplified compressed_pair using EBO
template <typename T, typename D>
class compressed_pair : private D {  // D as base (EBO applies if empty)
    T ptr_;

public:
    T& first() { return ptr_; }
    D& second() { return *this; }
};

// When D is empty (like default_delete):
// sizeof(compressed_pair<T*, D>) == sizeof(T*)
// The empty deleter base contributes zero bytes.

    This is why sizeof(unique_ptr<int>) equals sizeof(int*) on every major implementation. The deleter is there in the type system, but it contributes no runtime storage. The compiler can verify this at compile time:

    // Verify zero overhead at compile time
static_assert(sizeof(std::unique_ptr<int>) == sizeof(int*),
              "unique_ptr should be pointer-sized with default deleter");

// With a stateful deleter, it grows
struct LoggingDeleter {
    std::string logFile;
    void operator()(int* p) const { /* log, then delete */ delete p; }
};
static_assert(sizeof(std::unique_ptr<int, LoggingDeleter>) > sizeof(int*),
              "Stateful deleter adds to size");
    C++20 improvement: The [[no_unique_address]] attribute provides EBO-like behavior for non-static data members, so implementations no longer need the compressed_pair trick. The deleter member can be declared with [[no_unique_address]] and the compiler will compress it if empty.

    ▶ unique_ptr Memory Layout

    Step through to compare the memory layout of unique_ptr with a stateless deleter vs. a function-pointer deleter.

    Custom Deleters

    The true power of unique_ptr is that you can replace the deleter. This lets you wrap any C API, third-party library, or custom allocation scheme. The deleter is a template parameter, so the compiler knows its type at compile time and can inline the call.

    Function Pointer Deleter

    The simplest custom deleter is a function pointer. This is common when wrapping C APIs:

    // Wrap a C FILE* with automatic fclose
auto file = std::unique_ptr<FILE, decltype(&fclose)>(
    fopen("data.txt", "r"), fclose
);
// file goes out of scope: fclose(ptr) called automatically

// Wrap an SDL surface
auto surface = std::unique_ptr<SDL_Surface, decltype(&SDL_FreeSurface)>(
    SDL_LoadBMP("image.bmp"), SDL_FreeSurface
);

    The downside: a function pointer is stateful (it holds the address of the function), so sizeof(unique_ptr) grows to two pointers. The compiler also cannot inline through a function pointer as easily as through a direct call.

    Lambda Deleter

    A captureless lambda is a stateless type (EBO applies!), and the compiler can inline the call. This is often the best option:

    // Stateless lambda deleter: sizeof remains one pointer
auto deleter = [](FILE* fp) { if (fp) fclose(fp); };
auto file = std::unique_ptr<FILE, decltype(deleter)>(
    fopen("data.txt", "r"), deleter
);

static_assert(sizeof(file) == sizeof(FILE*),
              "Captureless lambda deleter is zero-overhead");

    In C++20, you can use a lambda directly in an unevaluated context, making the syntax even cleaner. In C++17 and earlier, you need a named variable for the lambda because lambdas are not allowed in decltype of template arguments in some contexts.

    Functor Deleter

    For complex cleanup logic, a functor class gives you full control. Stateless functors get EBO, stateful ones add to the size:

    // Stateless functor: zero overhead (EBO applies)
struct FCloseDeleter {
    void operator()(FILE* fp) const {
        if (fp) fclose(fp);
    }
};
auto file = std::unique_ptr<FILE, FCloseDeleter>(fopen("data.txt", "r"));
static_assert(sizeof(file) == sizeof(FILE*));

// Stateful functor: adds to size
struct PoolDeleter {
    MemoryPool* pool;
    void operator()(Widget* p) const {
        p->~Widget();
        pool->deallocate(p);
    }
};
auto w = std::unique_ptr<Widget, PoolDeleter>(
    pool.allocate<Widget>(), PoolDeleter{&pool}
);
// sizeof(w) == sizeof(Widget*) + sizeof(MemoryPool*)
    Rule of thumb: Prefer captureless lambdas or stateless functors as deleters. They give you zero overhead. Use function pointers only when wrapping C APIs where the function address is the natural deleter. Use stateful deleters only when the cleanup truly depends on runtime state (like a pool allocator).

    Array Specialization: unique_ptr<T[]>

    unique_ptr has a partial specialization for arrays. Instead of operator* and operator->, it provides operator[]. Instead of calling delete, it calls delete[].

    // Single object
auto single = std::make_unique<int>(42);
int val = *single;  // operator* works

// Array of 100 ints, value-initialized to 0
auto arr = std::make_unique<int[]>(100);
arr[0] = 42;        // operator[] works
// *arr;             // ERROR: operator* not available for arrays

    The array specialization exists because delete and delete[] are fundamentally different operations. Calling delete on an array pointer is undefined behavior. By encoding the "this is an array" information in the type, unique_ptr<T[]> makes it impossible to get this wrong.

    // What the specializations look like internally
template <typename T>
class unique_ptr<T[]> {
    T* ptr_;

public:
    T& operator[](size_t i) const { return ptr_[i]; }
    // No operator* or operator->

    ~unique_ptr() { delete[] ptr_; }
    // Other members similar to primary template
};

    When to Use unique_ptr<T[]> vs std::vector

    In most cases, prefer std::vector. It knows its size, supports iteration, and can grow. Use unique_ptr<T[]> only when:

    • You receive a dynamically allocated array from a C API and need RAII management.
    • You need a fixed-size heap buffer and cannot afford the three-word overhead of vector (pointer + size + capacity).
    • You are interfacing with a library that returns owning T* arrays.

    make_unique: The Right Way to Create

    std::make_unique (C++14) is the preferred way to create a unique_ptr. It is exception-safe, avoids repeating the type, and clearly communicates intent:

    // Prefer this:
auto widget = std::make_unique<Widget>(42, "hello");

// Over this:
std::unique_ptr<Widget> widget(new Widget(42, "hello"));

    Exception Safety

    Before C++17, function arguments could be evaluated in any order. Consider:

    void process(std::unique_ptr<Widget> w, std::unique_ptr<Gadget> g);

// DANGEROUS before C++17: possible leak!
process(
    std::unique_ptr<Widget>(new Widget()),
    std::unique_ptr<Gadget>(new Gadget())
);

// The compiler might evaluate in this order:
//   1. new Widget()         <-- allocates
//   2. new Gadget()         <-- throws! Widget leaked!
//   3. unique_ptr<Widget>() <-- never reached

// SAFE: make_unique bundles allocation and construction
process(std::make_unique<Widget>(), std::make_unique<Gadget>());

    C++17 fixed the evaluation order for function arguments (each argument is fully evaluated before the next), so the raw new version is technically safe from C++17 onward. But make_unique is still preferred for clarity and consistency.

    Implementation of make_unique

    The implementation is trivial, which is exactly the point:

    template <typename T, typename... Args>
std::unique_ptr<T> make_unique(Args&&... args) {
    return std::unique_ptr<T>(new T(std::forward<Args>(args)...));
}

// Array version (C++14)
template <typename T>
std::unique_ptr<T> make_unique(size_t size) {
    return std::unique_ptr<T>(new std::remove_extent_t<T>[size]());
}
    Limitation: make_unique cannot use custom deleters. If you need a custom deleter, you must construct the unique_ptr directly. But for the common case of new/delete, always prefer make_unique.

    Ownership Transfer with std::move

    unique_ptr models exclusive ownership. There can be exactly one owner at any time. Ownership transfer requires an explicit std::move, which makes transferring ownership visible in the source code. This is a deliberate design choice: ownership transfers are important events and should not happen silently.

    auto a = std::make_unique<Widget>(42);  // a owns the Widget

// auto b = a;                  // ERROR: copy deleted
auto b = std::move(a);          // OK: a transfers to b

// After move:
assert(a == nullptr);           // a is empty
assert(b != nullptr);           // b owns the Widget

    Passing unique_ptr to Functions

    How you pass a unique_ptr communicates ownership intent:

    // 1. Transfer ownership (sink): take by value
void consume(std::unique_ptr<Widget> w) {
    // w owns the Widget. It will be destroyed when consume() returns.
}
consume(std::move(myWidget));  // caller gives up ownership

// 2. Use without transferring: pass the raw pointer or reference
void observe(const Widget& w) {
    // Does not affect ownership. Cannot store or delete w.
}
observe(*myWidget);

// 3. Reseat (maybe swap/replace): take by reference
void reseat(std::unique_ptr<Widget>& w) {
    w = std::make_unique<Widget>(99);  // replaces the held object
}

    ▶ Ownership Transfer via std::move

    Step through to see how unique_ptr ownership moves from one variable to another, and what happens to the source.

    Factory Pattern with unique_ptr

    Returning unique_ptr from factory functions is the idiomatic way to create objects in modern C++. The factory owns the object during construction and transfers ownership to the caller. This pattern is exception-safe, makes ownership explicit, and works seamlessly with polymorphism.

    class Shape {
public:
    virtual ~Shape() = default;
    virtual double area() const = 0;
};

class Circle : public Shape {
    double radius_;
public:
    explicit Circle(double r) : radius_(r) {}
    double area() const override { return 3.14159 * radius_ * radius_; }
};

class Rectangle : public Shape {
    double w_, h_;
public:
    Rectangle(double w, double h) : w_(w), h_(h) {}
    double area() const override { return w_ * h_; }
};

// Factory returns unique_ptr<Shape>: caller picks up ownership
std::unique_ptr<Shape> createShape(const std::string& type) {
    if (type == "circle")    return std::make_unique<Circle>(5.0);
    if (type == "rectangle") return std::make_unique<Rectangle>(4.0, 3.0);
    return nullptr;
}

// Usage
auto shape = createShape("circle");
double a = shape->area();  // polymorphic call

    The key insight is that unique_ptr<Derived> implicitly converts to unique_ptr<Base>. The standard library defines this conversion, and it has zero overhead: it simply moves the pointer. No slicing, no extra allocation.

    Converting Factory Output to shared_ptr

    If the caller needs shared ownership, they can convert the unique_ptr to a shared_ptr. This conversion is implicit and efficient:

    // Factory returns unique_ptr
auto shape = createShape("circle");

// Convert to shared_ptr if shared ownership is needed
std::shared_ptr<Shape> shared = std::move(shape);

// This works because shared_ptr has a constructor that accepts unique_ptr&&

    The reverse conversion (shared_ptr to unique_ptr) is not possible because you cannot establish exclusive ownership over something that might have multiple owners.

    Design principle: Always return unique_ptr from factories. It gives the caller maximum flexibility: they can keep it as unique_ptr for exclusive ownership, convert to shared_ptr for shared ownership, or even release the raw pointer for legacy APIs. Starting with shared_ptr would lock the caller into reference counting whether they need it or not.

    Move Internals: What Actually Happens

    When you std::move a unique_ptr, the compiler generates code equivalent to:

    // Conceptual move constructor
unique_ptr(unique_ptr&& other) noexcept
    : ptr_(other.ptr_)       // copy the raw pointer
{
    other.ptr_ = nullptr;    // null the source
}
// That's it. No heap allocation. No deallocation.
// Just two pointer assignments.

    In optimized builds, the compiler often eliminates even these assignments through copy elision (guaranteed in C++17 for return values). A function returning make_unique<T>(...) often compiles to direct construction at the call site with zero moves.

    release() vs reset()

    Two methods that are often confused:

    auto p = std::make_unique<int>(42);

// release(): gives up ownership, returns raw pointer. No deletion.
int* raw = p.release();  // p is now nullptr. Caller must delete raw.
delete raw;              // manual cleanup required

// reset(): replaces the managed object. Old one is deleted.
auto q = std::make_unique<int>(10);
q.reset(new int(20));    // deletes the 10, now manages 20
q.reset();               // deletes the 20, now manages nothing

    Use release() when you need to hand a raw pointer to a C API or legacy code that takes ownership. Use reset() when you want to replace or clear the managed object.

    What the Compiler Actually Generates

    Let us compare the generated assembly for raw pointer management vs unique_ptr:

    // Version A: raw pointer
void rawVersion() {
    int* p = new int(42);
    use(*p);
    delete p;
}

// Version B: unique_ptr
void smartVersion() {
    auto p = std::make_unique<int>(42);
    use(*p);
}

    With optimizations enabled (-O2), both produce identical assembly. The unique_ptr destructor call is inlined, the default_delete functor call is inlined, and what remains is a bare delete. The abstraction truly costs nothing.

    You can verify this yourself on Compiler Explorer (godbolt.org). Compile both with -O2 and compare the output. The instructions are identical.

    unique_ptr and Incomplete Types

    unique_ptr can hold a pointer to an incomplete type, but the type must be complete at the point where the deleter is invoked (typically the destructor of the owning class). This is crucial for the Pimpl idiom:

    // widget.h
class Widget {
    struct Impl;                      // forward declaration (incomplete)
    std::unique_ptr<Impl> pimpl_;    // OK: Impl is incomplete here
public:
    Widget();
    ~Widget();                        // declared, defined in .cpp
    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;
};

// widget.cpp
struct Widget::Impl {
    int data;
    std::string name;
};

Widget::Widget() : pimpl_(std::make_unique<Impl>()) {}
Widget::~Widget() = default;  // Impl is complete here, default_delete works
Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;

    The key rule: you must define the destructor (even if defaulted) in the .cpp file where Impl is complete. If you let the compiler generate the destructor in the header, it will try to instantiate default_delete<Impl> where Impl is incomplete, and the code will fail to compile.

    unique_ptr Comparison and Hashing

    unique_ptr supports comparison operators and can be used in ordered and unordered containers:

    auto a = std::make_unique<int>(10);
auto b = std::make_unique<int>(20);

if (a != nullptr) { /* a owns something */ }
if (a < b)        { /* compares pointer values */ }
if (!a)           { /* a is empty */ }

// Can be stored in sets (ordered by pointer value)
std::set<std::unique_ptr<int>> s;
s.insert(std::make_unique<int>(1));
s.insert(std::make_unique<int>(2));

// Can be stored in unordered containers
std::unordered_set<std::unique_ptr<int>> us;
us.insert(std::make_unique<int>(1));

    Summary

    • unique_ptr models exclusive ownership with zero overhead for stateless deleters.
    • Internally, it is a raw pointer plus a deleter, stored in a compressed pair that exploits empty base optimization.
    • Custom deleters let you wrap any resource. Captureless lambdas and stateless functors maintain zero overhead.
    • unique_ptr<T[]> provides safe array management with operator[] and delete[].
    • Always use make_unique for exception safety and clarity.
    • Return unique_ptr from factories to give callers maximum flexibility.
    • Ownership transfer is explicit via std::move, making ownership flow visible in the code.
    • With optimizations, unique_ptr compiles to the exact same machine code as raw pointer management.
