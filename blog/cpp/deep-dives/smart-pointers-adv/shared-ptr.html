    shared_ptr Internals

    March 2026 · 22 min read

    What Is shared_ptr?

    std::shared_ptr models shared ownership: multiple shared_ptr instances can own the same object, and the object is destroyed only when the last owner is gone. It achieves this through reference counting, which is the mechanism that tracks how many owners exist at any given time.

    Unlike unique_ptr, which is essentially a zero-overhead wrapper around a raw pointer, shared_ptr carries real runtime cost: it allocates a control block, performs atomic reference count operations, and stores extra metadata. This makes it essential to understand when shared ownership is truly necessary and how the internals work, so you can make informed decisions about performance.

    When to use shared_ptr: Only when ownership is genuinely shared across multiple independent owners whose lifetimes are not hierarchically related. If one owner clearly outlives the others, unique_ptr with raw-pointer observers is simpler and faster.

    The Size of shared_ptr

    Every shared_ptr is two pointers wide: one pointer to the managed object, and one pointer to the control block. This is true regardless of the deleter type, because the deleter is stored inside the control block (type-erased), not in the shared_ptr itself.

    static_assert(sizeof(std::shared_ptr<int>) == 2 * sizeof(void*),
              "shared_ptr is always two pointers");

// Compare with unique_ptr:
static_assert(sizeof(std::unique_ptr<int>) == sizeof(void*),
              "unique_ptr is one pointer with default deleter");

    Why two pointers? Because the object pointer and the control block pointer can point to different addresses. This is critical for the aliasing constructor (discussed below) and for make_shared's single-allocation optimization.

    Control Block Anatomy

    The control block is the backbone of shared_ptr. It is a heap-allocated structure that contains everything needed to manage the object's lifetime. Conceptually, it looks like this:

    struct ControlBlock {
    std::atomic<long> strong_count;  // number of shared_ptr owners
    std::atomic<long> weak_count;    // number of weak_ptr + (strong_count > 0 ? 1 : 0)
    Deleter deleter;                 // type-erased deleter
    Allocator allocator;             // type-erased allocator (if custom)
    // For make_shared: the managed object is embedded here
};

    The actual implementation varies between library vendors, but the conceptual structure is the same. Let us examine each component:

    Strong Count

    The strong count tracks the number of shared_ptr instances pointing to the managed object. When a shared_ptr is copied, the strong count is atomically incremented. When a shared_ptr is destroyed or reassigned, the strong count is atomically decremented. When the strong count reaches zero, the managed object is destroyed (the deleter is called).

    auto a = std::make_shared<Widget>(42);  // strong_count = 1
{
    auto b = a;                          // strong_count = 2
    auto c = a;                          // strong_count = 3
}                                        // b, c destroyed: strong_count = 1
// a destroyed later: strong_count = 0, Widget deleted

    Weak Count

    The weak count is more subtle. It tracks the number of weak_ptr instances, plus one extra if the strong count is greater than zero. This "plus one" ensures that the control block itself is not deallocated while any shared_ptr still exists. The control block is freed only when both the strong count and the weak count reach zero.

    auto sp = std::make_shared<Widget>(42);  // strong=1, weak=1 (the "+1")
std::weak_ptr<Widget> wp = sp;           // strong=1, weak=2
sp.reset();                               // strong=0, Widget destroyed
                                          // weak=1 (the "+1" removed, wp keeps 1)
                                          // control block still alive!
wp.reset();                               // weak=0, control block freed

    Type-Erased Deleter

    Unlike unique_ptr where the deleter is a template parameter, shared_ptr's deleter is type-erased and stored inside the control block. This means the deleter type does not affect the type of the shared_ptr itself:

    // Both have the same type: shared_ptr<Widget>
auto a = std::shared_ptr<Widget>(new Widget(), std::default_delete<Widget>());
auto b = std::shared_ptr<Widget>(new Widget(), [](Widget* p) {
    logDestruction(p);
    delete p;
});

// a and b can be stored in the same container, assigned to each other, etc.
std::vector<std::shared_ptr<Widget>> v;
v.push_back(std::move(a));
v.push_back(std::move(b));  // different deleters, same vector

    Type erasure is powerful but comes at a cost: the deleter call goes through a virtual function or function pointer, which prevents inlining in some cases. This is one reason shared_ptr has more overhead than unique_ptr.

    ▶ Control Block Animation

    Watch shared_ptr copies increment the strong count, destructors decrement it, and weak_ptr extend the control block lifetime.

    make_shared: The Single-Allocation Optimization

    std::make_shared is the preferred way to create a shared_ptr. It performs a single heap allocation for both the control block and the managed object, placing them in contiguous memory:

    // Two allocations: one for Widget, one for control block
auto p1 = std::shared_ptr<Widget>(new Widget(42));

// One allocation: control block + Widget in one block
auto p2 = std::make_shared<Widget>(42);

    The benefits of single allocation are significant:

    • Performance: One allocation instead of two. Allocation is often the most expensive operation in constructing a shared_ptr.
    • Cache locality: The control block and the object are adjacent in memory, improving cache performance when accessing both.
    • Exception safety: Before C++17, shared_ptr<T>(new T(args)) in a function argument could leak if another argument threw. make_shared avoids this.
    • Reduced fragmentation: One allocation means one chunk of memory to manage instead of two.

    Memory Layout: make_shared vs. new

    With new:

    // shared_ptr<Widget>(new Widget(42))
//
// Heap allocation 1:          Heap allocation 2:
// +------------------+        +------------------+
// | strong_count = 1 |        | Widget data      |
// | weak_count   = 1 |        | (42)             |
// | deleter          |        +------------------+
// | allocator        |
// +------------------+
// shared_ptr holds pointers to both

    With make_shared:

    // make_shared<Widget>(42)
//
// Single heap allocation:
// +------------------+
// | strong_count = 1 |
// | weak_count   = 1 |
// | deleter          |
// | allocator        |
// | Widget data      |   <-- object embedded in control block
// | (42)             |
// +------------------+
// shared_ptr holds two pointers to different offsets within this block
    Caveat: make_shared cannot use custom deleters. If you need a custom deleter, you must construct the shared_ptr directly: shared_ptr<T>(ptr, deleter). Also, make_shared has a memory retention issue with weak_ptr (discussed in the pitfalls article).

    allocate_shared

    If you need a custom allocator, use std::allocate_shared. It works the same as make_shared but uses the provided allocator for the single combined allocation:

    my_allocator<Widget> alloc;
auto p = std::allocate_shared<Widget>(alloc, 42);
// Single allocation using alloc for control block + Widget

    The Aliasing Constructor

    One of the most powerful and least-known features of shared_ptr is the aliasing constructor. It creates a shared_ptr that shares ownership with an existing shared_ptr but points to a different object (typically a member of the owned object).

    struct Pair {
    int first;
    int second;
};

auto pair = std::make_shared<Pair>(10, 20);

// Aliasing constructor: shares ownership with pair, but points to pair->first
std::shared_ptr<int> first(pair, &pair->first);

// pair and first share the same control block (strong_count = 2)
// pair.get()  == &(*pair)        (points to Pair)
// first.get() == &pair->first    (points to int member)

pair.reset();    // strong_count = 1 (first still holds Pair alive)
int val = *first;  // safe: val == 10

    The aliasing constructor is why shared_ptr stores two pointers: the object pointer (which can be anything) and the control block pointer (which tracks the actual owned resource). Without the aliasing constructor, a single pointer would suffice.

    Use Cases for Aliasing

    • Sharing sub-objects: Pass a shared_ptr to a member of a larger structure, keeping the parent alive.
    • Type-safe casting: static_pointer_cast, dynamic_pointer_cast, and const_pointer_cast are all implemented using the aliasing constructor.
    • Adapting interfaces: A function expects shared_ptr<Base> but you have shared_ptr<Derived>.
    // How static_pointer_cast is implemented (conceptually)
template <typename T, typename U>
std::shared_ptr<T> static_pointer_cast(const std::shared_ptr<U>& r) noexcept {
    auto p = static_cast<T*>(r.get());
    return std::shared_ptr<T>(r, p);  // aliasing constructor
}

    weak_ptr: The Non-Owning Observer

    std::weak_ptr is a non-owning smart pointer that observes a shared_ptr-managed object without contributing to the strong count. It solves a critical problem: how do you observe an object that might be destroyed?

    A raw pointer cannot tell you whether the pointed-to object still exists. A weak_ptr can, through its expired() method or by attempting to lock() it into a shared_ptr.

    auto sp = std::make_shared<Widget>(42);
std::weak_ptr<Widget> wp = sp;

// Check if the object exists
if (!wp.expired()) {
    // Promote to shared_ptr for safe access
    if (auto locked = wp.lock()) {
        locked->doWork();  // safe: locked keeps the object alive
    }
}

sp.reset();  // Widget destroyed
// wp.expired() == true
// wp.lock() returns empty shared_ptr

    The lock() Pattern

    Always use lock() rather than checking expired() and then accessing. The check-then-act pattern has a race condition in multithreaded code:

    // BAD: race condition between expired() and lock()
if (!wp.expired()) {
    // Object could be destroyed RIGHT HERE by another thread
    auto sp = wp.lock();  // might return empty
    sp->doWork();         // undefined behavior if sp is empty
}

// GOOD: atomic promote-or-fail
if (auto sp = wp.lock()) {
    sp->doWork();  // guaranteed safe
}

    weak_ptr Internals

    A weak_ptr is also two pointers: one to the managed object and one to the control block (same as shared_ptr). The difference is that weak_ptr only increments the weak count, not the strong count. This means:

    • Creating or copying a weak_ptr increments the weak count.
    • Destroying a weak_ptr decrements the weak count.
    • The managed object is destroyed when the strong count hits zero (not affected by weak count).
    • The control block is freed when both counts hit zero.

    This separation is crucial: weak_ptr needs the control block to exist (to check the strong count via lock()), but it should not keep the managed object alive. The weak count keeps the control block alive, while the strong count keeps the object alive.

    enable_shared_from_this

    Sometimes, a member function of an object managed by shared_ptr needs to obtain a shared_ptr to this. You cannot simply write shared_ptr<Widget>(this) because that creates a second control block, leading to double-free. The solution is std::enable_shared_from_this:

    class Widget : public std::enable_shared_from_this<Widget> {
public:
    std::shared_ptr<Widget> getPtr() {
        return shared_from_this();  // returns shared_ptr sharing the existing control block
    }

    void startAsync() {
        // Capture shared_ptr to keep *this alive during async operation
        auto self = shared_from_this();
        std::thread([self]() {
            self->doWork();
        }).detach();
    }
};

    How It Works

    The base class stores a weak_ptr to itself. When a shared_ptr is first created for the object, the shared_ptr constructor detects the enable_shared_from_this base and initializes the internal weak_ptr. Subsequent calls to shared_from_this() lock that weak_ptr:

    // Simplified implementation
template <typename T>
class enable_shared_from_this {
    mutable std::weak_ptr<T> weak_this_;

    template <typename U> friend class shared_ptr;  // shared_ptr sets weak_this_

public:
    std::shared_ptr<T> shared_from_this() {
        return std::shared_ptr<T>(weak_this_);  // throws if no shared_ptr exists
    }

    std::shared_ptr<const T> shared_from_this() const {
        return std::shared_ptr<const T>(weak_this_);
    }

    std::weak_ptr<T> weak_from_this() noexcept {  // C++17
        return weak_this_;
    }
};
    Precondition: shared_from_this() can only be called on an object that is already managed by at least one shared_ptr. Calling it on a stack object or a unique_ptr-managed object is undefined behavior (or throws bad_weak_ptr since C++17).

    Custom Deleters with shared_ptr

    Unlike unique_ptr, the deleter for shared_ptr is not a template parameter. It is passed as a constructor argument and stored inside the control block via type erasure:

    // Custom deleter: does not change the type of shared_ptr<FILE>
auto file = std::shared_ptr<FILE>(
    fopen("data.txt", "r"),
    [](FILE* fp) { if (fp) fclose(fp); }
);

// Another shared_ptr<FILE> with a different deleter
auto file2 = std::shared_ptr<FILE>(
    fopen("log.txt", "w"),
    &fclose
);

// Both can coexist in the same container
std::vector<std::shared_ptr<FILE>> files;
files.push_back(std::move(file));
files.push_back(std::move(file2));

    This flexibility is powerful: you can mix objects with different cleanup strategies in the same collection. The price is the type-erased virtual call at destruction time.

    Atomic Reference Counting and Thread Safety

    The reference counts in the control block are std::atomic<long>. This means the following operations are thread-safe:

    • Copying a shared_ptr (incrementing the strong count)
    • Destroying a shared_ptr (decrementing the strong count)
    • Creating a weak_ptr from a shared_ptr
    • Locking a weak_ptr

    However, shared_ptr is not fully thread-safe. Specifically:

    std::shared_ptr<Widget> global;

// Thread 1: reads global
auto local = global;  // OK: atomic increment of strong count

// Thread 2: reads global
auto local2 = global;  // OK: atomic increment of strong count

// Thread 3: WRITES global
global = std::make_shared<Widget>(99);  // DATA RACE with reads!

// The control block operations are atomic, but the shared_ptr object
// itself (the two pointers) is NOT atomic. Reading and writing the
// same shared_ptr variable from different threads requires synchronization.
    Thread safety summary: Multiple threads can safely read the same shared_ptr. Multiple threads can safely use different copies of a shared_ptr (pointing to the same object). But if any thread writes (assigns, resets) a shared_ptr object that other threads read, you need a mutex or std::atomic<std::shared_ptr> (C++20).

    The Cost of Atomics

    Atomic reference counting is not free. On x86, an atomic increment is an lock inc or lock xadd instruction, which flushes the store buffer and bounces the cache line between cores. Benchmarks typically show that copying a shared_ptr is 10x to 40x slower than copying a raw pointer. This is why you should:

    • Pass shared_ptr by const& when you do not need a new owner.
    • Pass by value only in sink functions that actually store the shared_ptr.
    • Avoid gratuitous copies in tight loops.
    // BAD: unnecessary atomic increment/decrement
void process(std::shared_ptr<Widget> w) { w->doWork(); }

// GOOD: no reference count change
void process(const std::shared_ptr<Widget>& w) { w->doWork(); }

// BEST: if you do not need ownership semantics at all
void process(const Widget& w) { w.doWork(); }

    Copying vs Moving shared_ptr

    Moving a shared_ptr is much cheaper than copying it. A copy increments the atomic reference count. A move just transfers the two pointers and nulls the source, with zero atomic operations:

    auto a = std::make_shared<Widget>(42);  // strong=1

// Copy: atomic increment (expensive)
auto b = a;                              // strong=2

// Move: pointer swap (cheap, no atomics)
auto c = std::move(a);                   // strong=2 (a is now null)

// b and c share the Widget. a is empty.

    This is why std::move is important when storing shared_ptr into containers or passing to sink functions:

    std::vector<std::shared_ptr<Widget>> widgets;

auto w = std::make_shared<Widget>(42);
// Move into vector: no atomic operations
widgets.push_back(std::move(w));

    use_count() and Why You Should Avoid It

    shared_ptr::use_count() returns the current strong count. However, in multithreaded code, this value is immediately stale: another thread might increment or decrement the count between your call to use_count() and your next action.

    auto sp = std::make_shared<Widget>(42);
long count = sp.use_count();  // 1 right now

// In another thread:
auto copy = sp;               // count is now 2, but our variable says 1

// use_count() is useful for debugging only, never for logic.
    Rule: Never use use_count() for control flow. It is intended for debugging and diagnostics only. If you find yourself checking use_count() == 1 to see if you are the sole owner, that is a code smell: consider unique_ptr instead.

    Creation Patterns Compared

    There are three ways to create a shared_ptr, each with different trade-offs:

    // 1. make_shared (preferred for most cases)
auto a = std::make_shared<Widget>(42);
// + Single allocation
// + Exception safe
// + Best cache locality
// - Cannot use custom deleter
// - Memory retained by weak_ptr (see pitfalls)

// 2. new + shared_ptr constructor
auto b = std::shared_ptr<Widget>(new Widget(42));
// + Works with custom deleters
// + Object memory freed when strong count hits 0
// - Two allocations
// - Slightly less cache-friendly

// 3. shared_ptr constructor with custom deleter
auto c = std::shared_ptr<Widget>(new Widget(42), CustomDeleter{});
// + Full control over destruction
// - Two allocations
// - Type-erased deleter has virtual call overhead

    Simplified Implementation

    Here is a highly simplified shared_ptr that illustrates the core mechanics (omitting thread safety, allocator support, and many edge cases):

    template <typename T>
class SharedPtr {
    T* ptr_;
    ControlBlock* cb_;

    struct ControlBlock {
        long strong;
        long weak;
        virtual void destroy() = 0;   // calls deleter
        virtual void deallocate() = 0; // frees control block
    };

    template <typename D>
    struct ControlBlockImpl : ControlBlock {
        D deleter;
        ControlBlockImpl(D d) : deleter(std::move(d)) { strong = 1; weak = 1; }
        void destroy() override { deleter(/* ptr */); }
        void deallocate() override { delete this; }
    };

public:
    explicit SharedPtr(T* p)
        : ptr_(p), cb_(new ControlBlockImpl<std::default_delete<T>>({})) {}

    SharedPtr(const SharedPtr& other)
        : ptr_(other.ptr_), cb_(other.cb_) {
        if (cb_) ++cb_->strong;
    }

    SharedPtr(SharedPtr&& other) noexcept
        : ptr_(other.ptr_), cb_(other.cb_) {
        other.ptr_ = nullptr;
        other.cb_ = nullptr;
    }

    ~SharedPtr() {
        if (!cb_) return;
        if (--cb_->strong == 0) {
            cb_->destroy();  // call deleter
            if (--cb_->weak == 0) cb_->deallocate();
        }
    }

    T& operator*() const { return *ptr_; }
    T* operator->() const { return ptr_; }
};

    Summary

    • shared_ptr is two pointers wide: one to the object, one to the control block.
    • The control block contains the strong count, weak count, type-erased deleter, and optionally the object itself.
    • make_shared performs a single allocation for the control block and object together.
    • The aliasing constructor lets a shared_ptr point to a sub-object while sharing ownership of the parent.
    • weak_ptr observes without owning. It keeps the control block alive (not the object) and can be promoted to shared_ptr via lock().
    • enable_shared_from_this lets an object safely produce shared_ptr instances to itself.
    • Reference counting is atomic, making control block operations thread-safe. But the shared_ptr variable itself requires external synchronization for concurrent writes.
    • Prefer passing by const& or raw reference to avoid unnecessary atomic operations.
