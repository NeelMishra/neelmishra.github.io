    Smart Pointer Pitfalls

    March 2026 · 16 min read

    The Dark Side of Smart Pointers

    Smart pointers solve the majority of memory management problems in C++, but they are not magic. They introduce their own category of bugs and performance traps that are often more subtle than the raw-pointer problems they replace. This article catalogs the most dangerous pitfalls and shows you how to avoid each one.

    Mindset shift: With raw pointers, the failure mode is "forgot to delete." With smart pointers, the failure modes are "circular reference," "wrong smart pointer type," and "unnecessary overhead." Understanding these is what separates competent C++ developers from experts.

    Circular References: The Memory Leak That Looks Safe

    The most infamous smart pointer pitfall is the circular reference. If two objects hold shared_ptr to each other, their reference counts can never reach zero, and neither object is ever destroyed. This is a genuine memory leak, and it is invisible to tools like Valgrind because the pointers are "valid" (they point to allocated memory), they just never get freed.

    class Person {
    std::string name_;
    std::shared_ptr<Person> partner_;  // problem: circular ownership

public:
    Person(std::string name) : name_(std::move(name)) {}
    ~Person() { std::cout << name_ << " destroyed\n"; }

    void setPartner(std::shared_ptr<Person> p) {
        partner_ = std::move(p);
    }
};

void createCycle() {
    auto alice = std::make_shared<Person>("Alice");   // strong=1
    auto bob   = std::make_shared<Person>("Bob");     // strong=1

    alice->setPartner(bob);    // Bob strong=2
    bob->setPartner(alice);    // Alice strong=2

}   // alice goes out of scope: Alice strong=1 (Bob still holds it)
    // bob goes out of scope: Bob strong=1 (Alice still holds it)
    // Neither destructor ever runs. Memory leaked forever.

    The reference counts are stuck at 1. Alice holds Bob alive, and Bob holds Alice alive. It is a deadlock of lifetimes. Neither can be freed because freeing one requires freeing the other first.

    How to Detect Circular References

    Circular references are hard to spot in code reviews because the cycle might span multiple classes or involve long chains. Tools that can help:

    • AddressSanitizer (ASan) LeakSanitizer: Detects remaining allocations at program exit.
    • Custom shared_ptr instrumentation: Add logging in constructors/destructors to track what is not being destroyed.
    • Design review: Draw the ownership graph. If there is a cycle, you have a bug.

    Breaking Cycles with weak_ptr

    The standard solution is to replace one direction of the cycle with weak_ptr. A weak_ptr does not contribute to the strong count, so it cannot create a cycle:

    class Person {
    std::string name_;
    std::weak_ptr<Person> partner_;  // weak: does not prevent destruction

public:
    Person(std::string name) : name_(std::move(name)) {}
    ~Person() { std::cout << name_ << " destroyed\n"; }

    void setPartner(std::shared_ptr<Person> p) {
        partner_ = p;  // stores as weak_ptr (implicit conversion)
    }

    std::shared_ptr<Person> getPartner() const {
        return partner_.lock();  // promotes to shared_ptr if alive
    }
};

void noCycle() {
    auto alice = std::make_shared<Person>("Alice");
    auto bob   = std::make_shared<Person>("Bob");

    alice->setPartner(bob);
    bob->setPartner(alice);

}   // alice out of scope: strong=0, Alice destroyed
    // bob out of scope: strong=0, Bob destroyed
    // Both destructors run. No leak.
    Rule of thumb: In any bidirectional relationship between shared_ptr-managed objects, at least one direction must use weak_ptr. The "owning" direction uses shared_ptr; the "back-reference" direction uses weak_ptr.

    The shared_ptr-to-this Problem

    Creating a shared_ptr from a raw this pointer is one of the most dangerous operations in C++. It creates a second control block for the same object, leading to a double-free:

    class Widget {
public:
    std::shared_ptr<Widget> getShared() {
        return std::shared_ptr<Widget>(this);  // BUG: second control block!
    }
};

auto sp = std::make_shared<Widget>();     // control block A, strong=1
auto sp2 = sp->getShared();               // control block B, strong=1

// sp goes out of scope: control block A says strong=0, deletes Widget
// sp2 goes out of scope: control block B says strong=0, deletes Widget AGAIN
// CRASH: double-free

    The Fix: enable_shared_from_this

    Inherit from std::enable_shared_from_this to safely obtain a shared_ptr to the current object:

    class Widget : public std::enable_shared_from_this<Widget> {
public:
    std::shared_ptr<Widget> getShared() {
        return shared_from_this();  // uses existing control block
    }
};

auto sp = std::make_shared<Widget>();
auto sp2 = sp->getShared();

// Both share the same control block. strong=2.
// No double-free.

    But beware: shared_from_this() has a precondition: the object must already be managed by at least one shared_ptr. Calling it on a stack object or a unique_ptr-managed object is undefined behavior (throws bad_weak_ptr since C++17):

    Widget w;                   // stack object, not managed by shared_ptr
auto sp = w.getShared();    // throws bad_weak_ptr (or UB pre-C++17)

    make_shared Memory Retention

    make_shared allocates the control block and the managed object in a single allocation. This is normally a benefit (fewer allocations, better cache locality). But it has a surprising downside: the memory for the object cannot be freed until the control block is also freed.

    Normally, this is invisible. But when weak_ptr instances outlive the last shared_ptr, the object is destroyed (destructor runs) but the memory block is retained because the control block (which is in the same allocation) is still needed by the weak_ptr:

    // Scenario: large object with long-lived weak_ptr

auto sp = std::make_shared<HugeObject>();  // 10 MB object + control block in one allocation
std::weak_ptr<HugeObject> wp = sp;          // weak reference

sp.reset();
// HugeObject destructor runs (resources released)
// BUT: the 10 MB memory block is NOT freed yet!
// The control block is inside the same allocation.
// Memory is held until wp is destroyed.

// wp exists for a long time...
// ... 10 MB still allocated but not usable.

wp.reset();  // NOW the entire 10 MB block is freed

    The Fix: Use new When Memory Retention Matters

    If you have large objects with long-lived weak_ptr observers, use the two-allocation form so the object memory can be freed independently:

    // Two allocations: object memory freed when strong=0, control block freed when weak=0
auto sp = std::shared_ptr<HugeObject>(new HugeObject());
std::weak_ptr<HugeObject> wp = sp;

sp.reset();
// HugeObject memory freed immediately (strong=0)
// Only the small control block survives for weak_ptr

wp.reset();  // control block freed
    In practice: This matters only for large objects with long-lived weak_ptr references. For typical objects (a few hundred bytes), the single allocation of make_shared is always the right choice.

    Exception Safety with new + shared_ptr

    Before C++17, using new in function arguments with shared_ptr could leak memory:

    // Pre-C++17: DANGEROUS
void process(std::shared_ptr<A> a, std::shared_ptr<B> b);

process(
    std::shared_ptr<A>(new A()),
    std::shared_ptr<B>(new B())
);

// Possible evaluation order:
//   1. new A()       allocates A
//   2. new B()       allocates B... but THROWS!
//   3. shared_ptr<A>() never constructed
//   4. A is leaked!

    The fix is always make_shared, which bundles the allocation and the shared_ptr construction into a single expression:

    // SAFE: make_shared bundles allocation and construction
process(
    std::make_shared<A>(),
    std::make_shared<B>()
);
// If make_shared<B>() throws, make_shared<A>() was fully constructed
// and its shared_ptr destructor cleans up A.

    C++17 fixed the evaluation order (each function argument is fully evaluated before the next), making the raw new version safe. However, make_shared is still preferred for other benefits (single allocation, clarity).

    Performance Overhead of shared_ptr

    Compared to unique_ptr (which has zero overhead), shared_ptr carries measurable costs:

    Cost Breakdown

    • Control block allocation: One extra heap allocation (or merged with make_shared). Typically 32 to 48 bytes for the control block metadata.
    • Atomic reference counting: Every copy involves an atomic increment (lock xadd on x86). Every destruction involves an atomic decrement. These are 10x to 40x slower than non-atomic operations.
    • Type-erased deleter: The deleter is called through a virtual function or function pointer, preventing inlining in some cases.
    • Double indirection: Accessing the managed object requires dereferencing the object pointer. Accessing the control block requires dereferencing the control block pointer. This is two cache lines to touch.
    • Size: Two pointers per shared_ptr instance (16 bytes on 64-bit) vs. one pointer for unique_ptr.

    Relative Performance

    Operation                    | Raw ptr / unique_ptr | shared_ptr
-----------------------------+----------------------+-----------
Create (make)                | ~30 ns               | ~60 ns
Copy                         | N/A (unique) / 0     | ~20 ns (atomic inc)
Destroy                      | ~10 ns               | ~25 ns (atomic dec + branch)
Dereference                  | ~0 ns (same as raw)  | ~0 ns (same as raw)
Pass by const&               | ~0 ns                | ~0 ns
Pass by value (copy in)      | N/A                  | ~20 ns

    These numbers are approximate and vary by platform. The key takeaway: dereferencing is free (same as raw pointer), but creating, copying, and destroying are more expensive due to atomic operations.

    Avoiding Unnecessary Overhead

    // BAD: copies shared_ptr needlessly (atomic increment + decrement)
void process(std::shared_ptr<Widget> w) {
    w->doWork();
}

// GOOD: no copy, no atomic operations
void process(const Widget& w) {
    w.doWork();
}

// If you need to potentially store the shared_ptr:
void maybeStore(const std::shared_ptr<Widget>& w) {
    if (needToStore) {
        stored_ = w;  // only copies (atomic inc) when actually needed
    }
}

    When Raw Pointers Are Fine

    Smart pointers are not always the answer. In modern C++, raw pointers have a clear and legitimate role: non-owning access. Using a smart pointer where a raw pointer suffices adds unnecessary complexity and overhead.

    Scenarios Where Raw Pointers Are Correct

    • Function parameters that do not affect ownership: If a function just uses an object without storing it, pass by reference or raw pointer.
    • Back-pointers in ownership trees: Children pointing to parents in a tree where the parent owns the children.
    • Iterator-like access: Pointing into a container that manages its own memory.
    • Interfacing with C APIs: C functions expect raw pointers. Use .get() to obtain the raw pointer from a smart pointer.
    • Optional non-owning reference: When a reference would work but you need nullability.
    class Observer {
    Widget* target_;  // raw pointer: Observer does NOT own Widget

public:
    explicit Observer(Widget* w) : target_(w) {}
    void update() {
        if (target_) target_->refresh();
    }
};

// The caller ensures Widget outlives Observer
auto widget = std::make_unique<Widget>();
Observer obs(widget.get());  // .get() returns raw pointer
obs.update();
    C++ Core Guidelines I.11: "Never transfer ownership by a raw pointer (T*) or reference (T&)." Raw pointers are for non-owning access. If you need to transfer ownership, use unique_ptr. If you need shared ownership, use shared_ptr. The role of raw pointers is observation.

    Additional Pitfalls

    Constructing Multiple shared_ptr from the Same Raw Pointer

    This creates multiple control blocks for the same object, leading to double-free:

    Widget* raw = new Widget();
std::shared_ptr<Widget> sp1(raw);  // control block A
std::shared_ptr<Widget> sp2(raw);  // control block B (BUG!)
// sp1 and sp2 destroy Widget independently: double-free

    Always create a shared_ptr once (preferably with make_shared) and then copy it. Never construct two shared_ptr instances from the same raw pointer.

    shared_ptr with Arrays

    Before C++17, shared_ptr did not support arrays natively. You needed a custom deleter:

    // Pre-C++17: must provide custom deleter for arrays
auto arr = std::shared_ptr<int>(new int[100], std::default_delete<int[]>());

// C++17: array support built-in
auto arr17 = std::shared_ptr<int[]>(new int[100]);
arr17[0] = 42;  // operator[] works

// C++20: make_shared for arrays
auto arr20 = std::make_shared<int[]>(100);

    shared_ptr in Global/Static Scope

    Global shared_ptr instances are destroyed during static destruction, which has unspecified order. If one global's destructor accesses another global that has already been destroyed, you get undefined behavior:

    // DANGEROUS: destruction order between globals is unspecified
std::shared_ptr<Logger> g_logger = std::make_shared<Logger>();
std::shared_ptr<Database> g_db = std::make_shared<Database>();

// If g_db's destructor tries to log via g_logger, and g_logger
// was already destroyed: undefined behavior.

    Avoid global shared_ptr. If you must have global state, use the Meyers singleton pattern (function-local static) to control initialization and destruction order.

    Mixing shared_ptr and unique_ptr

    unique_ptr converts to shared_ptr (implicit, efficient). The reverse is impossible:

    // OK: unique_ptr to shared_ptr
std::unique_ptr<Widget> up = std::make_unique<Widget>();
std::shared_ptr<Widget> sp = std::move(up);  // up is now null

// ERROR: shared_ptr to unique_ptr (not possible)
// std::unique_ptr<Widget> up2 = std::move(sp);  // does not compile

    This is by design: you cannot claim exclusive ownership of something that might have multiple owners. Always start with unique_ptr and convert to shared_ptr only if needed.

    Custom Deleter and Type Erasure Overhead

    Every shared_ptr stores its deleter via type erasure (virtual function or function pointer). This means the deleter call cannot be inlined, unlike unique_ptr where the deleter is a template parameter and the compiler inlines it completely:

    // unique_ptr: deleter inlined at compile time
auto up = std::make_unique<Widget>();
// Destructor compiles to a direct 'delete' call, fully inlined

// shared_ptr: deleter called through virtual dispatch
auto sp = std::make_shared<Widget>();
// Destructor calls through control block vtable: indirect call, may not inline

    This overhead is tiny in most programs, but in hot paths with millions of destructions per second, it can be measurable.

    Smart Pointer Pitfall Checklist

    Before shipping code that uses smart pointers, verify:

    • No circular shared_ptr references exist. Use weak_ptr for back-references.
    • You never construct two shared_ptr instances from the same raw pointer.
    • Objects using shared_from_this() are always managed by shared_ptr before calling it.
    • Large objects with long-lived weak_ptr use the two-allocation form (not make_shared) if memory retention is a concern.
    • shared_ptr is passed by const& or raw reference when ownership is not needed.
    • You use unique_ptr by default and only shared_ptr when ownership is genuinely shared.
    • No shared_ptr in global scope (use function-local statics instead).
    • make_shared / make_unique is preferred over raw new.

    Summary

    • Circular references with shared_ptr cause memory leaks. Break them with weak_ptr.
    • Never create shared_ptr from this directly. Use enable_shared_from_this.
    • make_shared retains memory for the object as long as any weak_ptr exists. Use the two-allocation form for large objects with long-lived weak references.
    • Pre-C++17, new in function arguments risks leaks. Always prefer make_shared.
    • shared_ptr has measurable overhead: atomic ref counting, control block allocation, type-erased deleter. Use it only when shared ownership is genuinely required.
    • Raw pointers are correct for non-owning access. They are not inherently dangerous when used this way.
    • When in doubt, default to unique_ptr. Upgrade to shared_ptr only with a concrete justification.
