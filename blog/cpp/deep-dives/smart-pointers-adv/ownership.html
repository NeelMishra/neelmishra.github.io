    Ownership Patterns

    March 2026 · 18 min read

    The Ownership Mental Model

    Every object in a C++ program has exactly one owner responsible for its destruction. The owner might be the stack (for local variables), a smart pointer, or a container. Getting ownership right is the single most important skill for writing correct, leak-free C++ code.

    The ownership model is not enforced by the compiler in the way Rust's borrow checker enforces it. Instead, C++ gives you tools (smart pointers, const references) and idioms (factory pattern, Pimpl) that encode ownership in the type system. It is the programmer's job to choose the right tool for each relationship.

    There are three fundamental ownership relationships:

    • Unique ownership: Exactly one entity owns the object. Use std::unique_ptr.
    • Shared ownership: Multiple entities co-own the object. Use std::shared_ptr.
    • Non-owning observation: An entity accesses but does not own the object. Use raw pointers or references.
    Herb Sutter's guideline: "Use unique_ptr by default, shared_ptr when sharing is needed, and raw pointers/references for non-owning access." The vast majority of real-world ownership is unique.

    Unique Ownership with unique_ptr

    Unique ownership is the most common and most important pattern. It means: exactly one entity is responsible for the object's lifetime. When that entity is destroyed, the object is destroyed. When ownership transfers, it transfers completely, the source becomes empty.

    class Engine {
public:
    void start() { /* ... */ }
};

class Car {
    std::unique_ptr<Engine> engine_;  // Car OWNS the Engine

public:
    Car() : engine_(std::make_unique<Engine>()) {}

    // Engine is destroyed when Car is destroyed.
    // No explicit destructor needed: Rule of Zero.
};

    The type unique_ptr<Engine> communicates three things to every reader:

    1. Car owns the Engine.
    2. No one else shares ownership.
    3. The Engine is heap-allocated (otherwise it would be a plain member).

    Unique Ownership in Containers

    Containers of unique_ptr model a collection that owns its elements. This is the standard way to store polymorphic objects:

    class Shape { public: virtual ~Shape() = default; virtual double area() const = 0; };
class Circle : public Shape { /* ... */ };
class Square : public Shape { /* ... */ };

// The vector owns all shapes. Shapes are destroyed when the vector is destroyed.
std::vector<std::unique_ptr<Shape>> shapes;
shapes.push_back(std::make_unique<Circle>(5.0));
shapes.push_back(std::make_unique<Square>(3.0));

// Remove one shape: it is destroyed immediately
shapes.erase(shapes.begin());

// Clear all: all remaining shapes destroyed
shapes.clear();

    Shared Ownership with shared_ptr

    Shared ownership means multiple independent owners co-own an object, and the object is destroyed only when the last owner releases it. This is appropriate when you genuinely cannot determine a single owner at compile time.

    Legitimate use cases for shared_ptr:

    • Caches: Multiple consumers share cached data. The cache entry is freed when no one references it.
    • Async operations: A callback captures a shared_ptr to keep the object alive until the callback fires.
    • Graph structures: Nodes in a DAG (directed acyclic graph) can have multiple parents.
    • Plugin systems: Multiple components independently load and release shared libraries.
    // Cache example: entries are shared between cache and active users
class TextureCache {
    std::unordered_map<std::string, std::shared_ptr<Texture>> cache_;

public:
    std::shared_ptr<Texture> get(const std::string& path) {
        auto it = cache_.find(path);
        if (it != cache_.end()) return it->second;

        auto tex = std::make_shared<Texture>(loadFromDisk(path));
        cache_[path] = tex;
        return tex;
    }

    void evict(const std::string& path) {
        cache_.erase(path);
        // Texture stays alive if any user still holds a shared_ptr
    }
};
    Warning: shared_ptr is overused. If you can restructure the code so that one entity clearly outlives the others, use unique_ptr with raw-pointer observers instead. Shared ownership adds complexity (circular references, atomic overhead) and often indicates a design problem.

    Non-Owning Observation

    Non-owning access means: "I can see this object, but I am not responsible for its lifetime. Someone else will keep it alive for as long as I need it." In C++, this is expressed with:

    • References (T& / const T&): When the object is guaranteed to exist for the duration of the reference's use. The most common and simplest option.
    • Raw pointers (T* / const T*): When nullability is needed, or when storing in a container that does not support references.
    • weak_ptr: When the observed object is managed by shared_ptr and might be destroyed at any time.
    class Logger {
    // Non-owning reference to the output stream.
    // The caller owns the stream and must keep it alive.
    std::ostream& out_;

public:
    explicit Logger(std::ostream& out) : out_(out) {}
    void log(const std::string& msg) { out_ << msg << '\n'; }
};

// Logger observes cout; it does not own it
Logger logger(std::cout);
logger.log("hello");

    Raw Pointers as Non-Owning

    In modern C++, a raw pointer that does not come from new is understood to be non-owning. The C++ Core Guidelines formalize this: a raw pointer never owns. If you see a raw pointer, it means "I'm looking at this, but someone else manages its lifetime."

    class TreeNode {
    std::string name_;
    std::vector<std::unique_ptr<TreeNode>> children_;
    TreeNode* parent_;  // non-owning back-reference

public:
    TreeNode(std::string name, TreeNode* parent)
        : name_(std::move(name)), parent_(parent) {}

    void addChild(std::string name) {
        children_.push_back(std::make_unique<TreeNode>(std::move(name), this));
    }

    TreeNode* parent() { return parent_; }
};

    In this tree, children are owned by their parent (via unique_ptr), and children hold a non-owning raw pointer back to the parent. This is the canonical way to build trees in C++.

    Factory Functions Returning unique_ptr

    A factory function creates an object and transfers ownership to the caller. The idiomatic return type is unique_ptr:

    // Factory: creates and returns ownership
std::unique_ptr<Shape> createShape(const Config& cfg) {
    if (cfg.type == "circle")
        return std::make_unique<Circle>(cfg.radius);
    if (cfg.type == "rect")
        return std::make_unique<Rectangle>(cfg.width, cfg.height);
    return nullptr;  // unknown type
}

// Caller receives ownership
auto shape = createShape(config);  // unique_ptr<Shape>

// Can convert to shared if needed
std::shared_ptr<Shape> shared = createShape(config);

    This pattern is superior to returning a raw pointer because:

    • The return type documents that ownership is being transferred.
    • The caller cannot accidentally forget to delete the object.
    • Implicit conversion to shared_ptr is supported if the caller needs shared ownership.

    Sink Functions Taking unique_ptr by Value

    A sink function consumes ownership. The caller explicitly gives up the object, and the function takes responsibility for its lifetime. The idiomatic signature takes unique_ptr by value:

    class Renderer {
    std::vector<std::unique_ptr<Shape>> shapes_;

public:
    // Sink: takes ownership of the shape
    void addShape(std::unique_ptr<Shape> shape) {
        shapes_.push_back(std::move(shape));
    }
};

Renderer renderer;
auto circle = std::make_unique<Circle>(5.0);
renderer.addShape(std::move(circle));  // explicit transfer
// circle is now nullptr

    The std::move at the call site makes the ownership transfer visible. The reader knows immediately that the caller no longer owns the object. If you accidentally write renderer.addShape(circle) without the move, the compiler rejects it because unique_ptr is not copyable.

    Parameter Passing Guidelines

    // "I need to own this"         → unique_ptr<T> by value (sink)
void consume(std::unique_ptr<Widget> w);

// "I need to share ownership"  → shared_ptr<T> by value (share)
void share(std::shared_ptr<Widget> w);

// "I observe, do not need null" → const T& (reference)
void observe(const Widget& w);

// "I observe, might be null"   → const T* (pointer)
void maybeObserve(const Widget* w);

// "I want to reseat your ptr"  → unique_ptr<T>& (in-out)
void reseat(std::unique_ptr<Widget>& w);

    The Pimpl Idiom with unique_ptr

    The Pimpl (Pointer to Implementation) idiom hides implementation details behind an opaque pointer. This reduces compile-time dependencies and speeds up builds in large projects. unique_ptr is the perfect tool because the outer class uniquely owns its implementation:

    // widget.h: public header that clients include
#include <memory>
#include <string>

class Widget {
public:
    Widget(int id, const std::string& name);
    ~Widget();  // must be declared here, defined in .cpp

    // Move operations must also be declared and defined in .cpp
    Widget(Widget&& other) noexcept;
    Widget& operator=(Widget&& other) noexcept;

    void doWork();
    int id() const;

private:
    struct Impl;                       // forward declaration
    std::unique_ptr<Impl> pimpl_;     // unique ownership
};
    // widget.cpp: implementation file, not included by clients
#include "widget.h"
#include <iostream>
#include <vector>
// Can include anything here without affecting clients

struct Widget::Impl {
    int id;
    std::string name;
    std::vector<int> history;

    Impl(int i, const std::string& n) : id(i), name(n) {}
};

Widget::Widget(int id, const std::string& name)
    : pimpl_(std::make_unique<Impl>(id, name)) {}

Widget::~Widget() = default;
Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;

void Widget::doWork() {
    pimpl_->history.push_back(pimpl_->id);
    std::cout << pimpl_->name << " working\n";
}

int Widget::id() const { return pimpl_->id; }

    Why declare the destructor and move operations in the header but define them in the .cpp? Because unique_ptr's destructor needs to see the complete type of Impl to call delete. If the compiler generates the destructor in the header (where Impl is only forward-declared), it fails. By defining them in the .cpp file where Impl is complete, everything works.

    Benefits of Pimpl: Clients of Widget do not need to include <iostream>, <vector>, or any other headers used by the implementation. Adding or changing private members of Impl does not require clients to recompile. This can dramatically reduce build times in large projects.

    Ownership Trees

    Many real-world data structures form trees where parents own children and children have non-owning back-references to parents. This is the canonical ownership pattern for GUIs, scene graphs, DOM trees, and file systems.

    class Node {
    std::string name_;
    Node* parent_;                                  // non-owning
    std::vector<std::unique_ptr<Node>> children_;  // owning

public:
    Node(std::string name, Node* parent = nullptr)
        : name_(std::move(name)), parent_(parent) {}

    Node* addChild(std::string name) {
        children_.push_back(std::make_unique<Node>(std::move(name), this));
        return children_.back().get();
    }

    void removeChild(size_t index) {
        children_.erase(children_.begin() + index);
        // Child and all grandchildren destroyed automatically
    }

    Node* parent() const { return parent_; }
    const std::string& name() const { return name_; }
    size_t childCount() const { return children_.size(); }
};

    Destruction is automatic and cascading: when a parent is destroyed, its unique_ptr children are destroyed, which destroys their children, and so on. No explicit cleanup code is needed anywhere.

    Design Rules for Ownership Trees

    • Parents own children via unique_ptr (or a container of unique_ptr).
    • Children reference parents via raw pointers. The parent always outlives the child because the parent owns the child.
    • Siblings reference siblings via raw pointers or indices. The parent (which owns all siblings) ensures they outlive each other within the same scope.
    • External references into the tree should be raw pointers or iterators. They become dangling if the referenced node is removed, so document this clearly.

    Ownership Transfer Patterns

    Here are the common patterns for moving ownership between components:

    Builder/Assembler Pattern

    A builder creates components with make_unique and assembles them into a larger structure. At the end, ownership of the entire structure is transferred to the caller:

    class CarBuilder {
    std::unique_ptr<Engine> engine_;
    std::unique_ptr<Chassis> chassis_;

public:
    CarBuilder& setEngine(std::unique_ptr<Engine> e) {
        engine_ = std::move(e);
        return *this;
    }

    CarBuilder& setChassis(std::unique_ptr<Chassis> c) {
        chassis_ = std::move(c);
        return *this;
    }

    std::unique_ptr<Car> build() {
        return std::make_unique<Car>(
            std::move(engine_), std::move(chassis_)
        );
    }
};

auto car = CarBuilder()
    .setEngine(std::make_unique<V8Engine>())
    .setChassis(std::make_unique<SteelChassis>())
    .build();

    Chain of Responsibility

    Each handler owns the next handler in the chain. Ownership flows linearly through the chain:

    class Handler {
    std::unique_ptr<Handler> next_;

public:
    void setNext(std::unique_ptr<Handler> next) {
        next_ = std::move(next);
    }

    virtual bool handle(const Request& r) {
        if (next_) return next_->handle(r);
        return false;
    }

    virtual ~Handler() = default;
};

// Build chain: auth owns logging owns business
auto business = std::make_unique<BusinessHandler>();
auto logging = std::make_unique<LoggingHandler>();
logging->setNext(std::move(business));
auto auth = std::make_unique<AuthHandler>();
auth->setNext(std::move(logging));

    Ownership and Exception Safety

    Smart pointers provide exception safety automatically. If an exception is thrown between creating an object and inserting it into a container, the smart pointer destructor cleans up:

    void buildScene(Renderer& renderer) {
    auto shape1 = std::make_unique<Circle>(5.0);
    auto shape2 = std::make_unique<Square>(3.0);

    // If validate() throws, both shapes are cleaned up automatically
    validate(*shape1, *shape2);

    renderer.addShape(std::move(shape1));
    renderer.addShape(std::move(shape2));
}

    Compare this with raw pointers, where you would need try/catch to avoid leaks on exception paths. Smart pointers eliminate this entire class of bugs.

    Ownership Decision Table

    Use this table to pick the right ownership tool for your situation:

    Relationship                  | Smart Pointer         | Example
------------------------------+-----------------------+---------------------------
Exclusive owner               | unique_ptr<T>         | Class member, container
Shared owner (rare)           | shared_ptr<T>         | Cache, async callback
Non-owning, never null        | T& or const T&        | Function parameter
Non-owning, maybe null        | T* or const T*        | Optional back-reference
Non-owning, might expire      | weak_ptr<T>           | Observer of shared_ptr
Factory output                | unique_ptr<T> (return)| createWidget()
Sink parameter                | unique_ptr<T> (value) | addWidget(unique_ptr)
Teaching ownership to readers | const unique_ptr<T>&  | "I see your unique, don't take"

    Anti-Patterns to Avoid

    Common ownership mistakes and their corrections:

    // ANTI-PATTERN: shared_ptr when unique ownership suffices
class Window {
    std::shared_ptr<Button> button_;  // overkill: only Window owns the Button
};
// FIX:
class Window {
    std::unique_ptr<Button> button_;  // correct: exclusive ownership
};

// ANTI-PATTERN: raw owning pointer
class BadTree {
    Widget* child_;  // who deletes this?
    ~BadTree() { delete child_; }  // manual, error-prone
};
// FIX:
class GoodTree {
    std::unique_ptr<Widget> child_;  // automatic cleanup
};

// ANTI-PATTERN: passing shared_ptr everywhere "just in case"
void process(std::shared_ptr<Widget> w) { w->doWork(); }
// FIX: pass by reference when you do not need ownership
void process(const Widget& w) { w.doWork(); }

    Summary

    • Unique ownership (unique_ptr) is the default. It is zero overhead, self-documenting, and handles 90% of ownership needs.
    • Shared ownership (shared_ptr) is for genuinely shared lifetimes: caches, async operations, DAGs. It has real overhead (control block, atomic ref counting).
    • Non-owning access uses references (never null) or raw pointers (nullable). In modern C++, a raw pointer means "non-owning."
    • Factories return unique_ptr to transfer ownership to the caller.
    • Sinks take unique_ptr by value to consume ownership, with std::move making the transfer explicit.
    • The Pimpl idiom uses unique_ptr to hide implementation details and reduce compile dependencies.
    • Ownership trees use unique_ptr for parent-to-child edges and raw pointers for child-to-parent back-references. Destruction cascades automatically.
    • When in doubt, start with unique_ptr and only upgrade to shared_ptr if you have a concrete reason.
