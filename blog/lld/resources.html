    How to Get Good at LLD: The Complete Resource List

    June 2026 · 30 min read

    This post is a single place that gathers the best material on Low-Level Design (LLD) from across GitHub and the wider internet: a learning roadmap, the full concept curriculum, the highest-signal repositories, books, courses, YouTube channels, websites, practice platforms, an interview framework, a pattern-to-problem cheat sheet, the most common mistakes, and a week-by-week study plan. Treat it as a map rather than a reading list to finish in one sitting. The fastest way to get good at LLD is short: learn the vocabulary (OOP, SOLID, patterns), then design one problem a day and refactor relentlessly.

    How to use this list. Skim the roadmap first, pick one repository and one book as anchors, then spend most of the time practicing on the problem list at the end. Reading about design without designing is the most common way to stall. Star counts are approximate as of mid-2026 and only signal popularity, not quality order.

    The Roadmap: How Getting Good Actually Works

    LLD skill compounds in a specific order. Each layer depends on the one before it:

    1. Object-oriented thinking. Model a problem as collaborating objects with clear responsibilities, not as procedures.
    2. Principles. SOLID, plus DRY, KISS, and YAGNI, give an objective way to judge whether a design is good or fragile.
    3. Design patterns. The 23 Gang-of-Four patterns are reusable solutions to recurring object-modeling problems. Knowing them turns a blank page into a menu.
    4. UML and communication. A class diagram drawn quickly is how a design gets reviewed in an interview and on a team.
    5. Concurrency. Many real LLD problems (rate limiter, cache, vending machine, elevator) need thread-safe designs.
    6. Deliberate practice. Designing real problems end to end, under time pressure, is the only step that converts knowledge into skill.

    The rest of this post supplies resources for every layer, then a problem list to practice on.

    The Concept Curriculum

    This is the canonical syllabus that nearly every good LLD resource converges on. Master these topics in roughly this order.

    1. OOP Fundamentals

    Classes and objects, interfaces and abstract classes, enums, and the four pillars: encapsulation, abstraction, inheritance, and polymorphism. The goal is to assign each responsibility to exactly one well-named owner.

    2. Class Relationships

    Knowing which arrow to draw is half of a class diagram: association (uses-a), aggregation (has-a, independent lifetime), composition (has-a, owned lifetime), dependency, and inheritance (is-a). Prefer composition over inheritance as a default.

    3. Design Principles

    The five SOLID principles are the core vocabulary of every design review:

    PrincipleOne-line meaning
    S — Single ResponsibilityA class should have one reason to change.
    O — Open/ClosedOpen for extension, closed for modification.
    L — Liskov SubstitutionSubtypes must be usable wherever the base type is.
    I — Interface SegregationMany small interfaces beat one fat interface.
    D — Dependency InversionDepend on abstractions, not concretions.

    Add DRY (don't repeat yourself), KISS (keep it simple), YAGNI (don't build what isn't needed), and the law of composition over inheritance.

    4. Design Patterns (Gang of Four)

    Implement each of these at least once. They cluster into three families:

    CreationalStructuralBehavioral
    Singleton, Factory Method, Abstract Factory, Builder, Prototype Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy Strategy, Observer, State, Command, Template Method, Iterator, Mediator, Memento, Visitor, Chain of Responsibility, Interpreter

    In interviews, a handful do most of the work: Strategy, Factory, Observer, State, Decorator, Builder, Command, Composite, and Singleton. The cheat sheet below maps them to problems.

    5. UML Diagrams

    Be able to sketch a class diagram fast, plus use case, sequence, activity, and state machine diagrams. The class diagram is the one that matters most in an LLD round.

    6. Concurrency and Multithreading

    Threads and processes, race conditions and critical sections, mutexes, semaphores, condition variables, reentrant locks, compare-and-swap, deadlock and livelock, and the classic patterns: producer-consumer, reader-writer, thread pool, and signaling. Thread-safe singletons, caches, and bounded queues are common follow-ups.

    Books

    Three books are enough to be dangerous; the rest deepen specific areas.

    Head First Design Patterns start here

    Freeman & Robson. The friendliest first pass at patterns, built around worked examples.

    Design Patterns: Elements of Reusable OO Software

    The Gang of Four. The original reference; dense, but the canonical source.

    Clean Code

    Robert C. Martin. Naming, functions, classes, and the habits of readable code.

    Clean Architecture

    Robert C. Martin. Boundaries, dependencies, and how SOLID scales to systems.

    Refactoring

    Martin Fowler. A catalog of code smells and the safe transformations that remove them.

    A Philosophy of Software Design

    John Ousterhout. Deep modules, complexity, and how to design clean interfaces.

    Domain-Driven Design

    Eric Evans. Modeling rich domains and the language to talk about them.

    Effective Java

    Joshua Bloch. Object-modeling idioms that transfer well beyond Java.

    UML Distilled

    Martin Fowler. The minimum UML needed to communicate a design.

    Working Effectively with Legacy Code

    Michael Feathers. Seams, testing, and changing code without breaking it.

    The Interview Framework

    Every strong LLD answer follows roughly the same seven steps. Practicing the process matters more than memorizing any single solution.

    1Clarify requirements. Separate functional from non-functional, fix the scope, and state assumptions out loud.

    2List core use cases and actors. Who uses the system, and what are the main flows?

    3Identify entities and responsibilities. Pull the nouns out of the requirements and give each a single, clear job.

    4Draw the class diagram. Fix the relationships: association, aggregation, composition, inheritance. Favor composition.

    5Apply principles and patterns. Check SOLID, then reach for the pattern that matches the variation point (see the cheat sheet below).

    6Write clean code. Interfaces first, then implementations. Add thread safety where the problem demands it.

    7Extend and defend. Walk through edge cases, a new requirement, and the trade-offs of the chosen design.

    Pattern-to-Problem Cheat Sheet

    The quickest way to recognize a pattern is to recognize the variation point it absorbs.

    PatternReach for it when…Classic problems
    Strategyone behavior has interchangeable algorithmsparking pricing, payment methods, ride matching, discount rules
    Statean object behaves differently across a lifecyclevending machine, elevator, ATM, order status, game turns
    Observermany parties react to one eventnotifications, pub-sub, stock tickers, chat, auctions
    Factory / Abstract Factorycreation should be decoupled from usevehicle and piece creation, notification channels, document types
    Decoratorresponsibilities stack at runtimepizza toppings, coffee add-ons, I/O streams, middleware
    Buildera complex object has many optional partsmeal builder, query builder, immutable configs
    Commandactions need to be queued, logged, or undonetext-editor undo/redo, task queues, remote controls
    Compositepart and whole should be treated uniformlyfile system, UI trees, org charts, menus
    Chain of Responsibilitya request passes through ordered handlersmiddleware, approval flows, logging levels
    Singletonexactly one shared instance is required (use sparingly)config, logger, connection pool

    Common Mistakes to Avoid

    • Coding before clarifying. Jumping into classes before nailing requirements produces a design that solves the wrong problem.
    • God classes. One class that does everything violates the Single Responsibility Principle and resists change.
    • Pattern soup. Forcing patterns where none are needed adds complexity. Patterns earn their place by absorbing a real variation point.
    • Anemic models. Classes that are bags of getters and setters with no behavior push logic into the wrong layer.
    • Ignoring concurrency. If the problem hints at shared mutable state, a non-thread-safe answer is incomplete.
    • Over-engineering. Building for imaginary future requirements violates YAGNI and KISS; design for today, keep it extensible.
    • Weak naming and tight coupling. Unclear names and concrete dependencies make a design hard to read and harder to extend.

    A Six-to-Eight Week Study Plan

    WeeksFocus
    Week 1OOP fundamentals, class relationships, and basic UML. Re-model two small toy problems as objects.
    Week 2SOLID, DRY, KISS, YAGNI. Take a messy class and refactor it against each principle.
    Weeks 3–4Design patterns, family by family. Implement each pattern once from memory; read the refactoring.guru entry alongside.
    Week 5Concurrency: locks, semaphores, condition variables, and thread-safe versions of a cache and a bounded queue.
    Weeks 6–8One LLD problem per day from the ladder above, timed, easy to hard. Always finish with a class diagram and an extension question.
    The one habit that matters. Design a problem end to end every day, then compare against a reference and refactor. Six weeks of that beats six months of passive reading. Everything above is in service of that loop.
