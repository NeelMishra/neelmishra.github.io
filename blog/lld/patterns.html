    LLD Interview Patterns

    March 2026 · 22 min read

    After working through Parking Lot, Elevator, and Splitwise, you'll start to notice recurring themes. This final post crystallizes those patterns into a toolkit you can apply to any LLD interview problem.

    Pattern → Problem Mapping

    Strategy

    Parking pricing, elevator scheduling, expense splitting, ride matching, discount calculation, notification channels

    Observer

    Stock ticker, notification system, chat rooms, event logging, order status updates

    Factory

    Vehicle creation, account types, document formats, payment processors, database connectors

    State

    Elevator states, vending machine, order lifecycle, traffic light, ATM

    Singleton

    ParkingLot, ExpenseManager, Logger, Config, ConnectionPool

    Decorator

    Pizza/coffee toppings, I/O streams, notification wrapping (email + SMS + push)

    Builder

    Complex queries, pizza orders, report generation, test fixtures

    Command

    Undo/redo, task queuing, macro recording, transaction logs

    Recurring Themes in LLD Problems

    1. The "Manager" or "Service" Class

    Almost every LLD problem has a top-level coordinator: ParkingLot, Building, ExpenseManager, BookingService. This is your Facade, it provides a simple interface to the subsystem. Often a Singleton.

    2. Type Hierarchies with a Factory

    When the problem involves different types of something (vehicle types, account types, room types), create an abstract base class and a factory. This appears in every problem.

    3. Swappable Behavior with Strategy

    When the problem says "different pricing models" or "multiple scheduling algorithms," that's Strategy. Define an interface, create concrete implementations, inject via constructor or setter.

    4. Event-Driven Loose Coupling with Observer

    When action in one part of the system should trigger reactions elsewhere (order placed → send email + update inventory + log), use Observer to avoid tight coupling.

    5. State Machines

    Elevators, vending machines, order lifecycles, ATMs, anything with distinct states and transitions. Can be implemented with enum + switch (simple) or the formal State pattern (complex).

    Common Mistakes

    Jumping straight to code

    The biggest mistake. Spend 10+ minutes on requirements and design before writing a single line. Interviewers want to see your thinking process, not typing speed.

    God classes

    Stuffing everything into one class (ParkingLot that also handles pricing, ticket generation, display, and user management). Split by responsibility.

    Over-engineering

    Don't add patterns just to show you know them. A simple problem doesn't need Observer + Strategy + Decorator + Factory. Use a pattern when it solves a real design tension.

    Ignoring concurrency

    Parking lots have multiple gates. Elevators handle concurrent requests. Mention thread safety even if you don't implement full locking. A mutex around critical operations shows awareness.

    No interfaces

    Hardcoding concrete classes everywhere makes the design rigid. If you're selecting between algorithms (pricing, routing, splitting), there should be an interface.

    Pro Tips

    Start with enums

    Define your enums first (VehicleType, SpotSize, Direction, OrderStatus). They anchor the rest of the design and make your code self-documenting from the start.

    Use the "newspaper test"

    Can you explain each class in one sentence? If not, it probably violates SRP. A good class name reads like a headline: "ParkingSpot manages one parking spot." Done.

    Draw before you code

    Even a rough sketch on screen/whiteboard helps. Show the interviewer boxes (classes) with arrows (relationships). Label with pattern names. This makes your code seem inevitable rather than invented.

    Mention trade-offs explicitly

    "I'm using Singleton here for simplicity, but in production I'd inject it for testability." This shows depth.

    Prepare 3-4 problems deeply

    Parking Lot, Elevator, Splitwise, and one more (Tic-Tac-Toe, BookMyShow, Library Management). Knowing these deeply lets you pattern-match any new problem.
