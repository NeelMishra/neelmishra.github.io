    Design Patterns for LLD

    March 2026 · 28 min read

    Design patterns are named, proven solutions to recurring design problems. You don't need to memorize all 23 Gang-of-Four patterns, for LLD interviews, six patterns cover 90% of cases. This post teaches each one with a concrete C++ example and clear guidance on when to reach for it.

    Pattern Overview

    PatternCategoryOne-line SummaryLLD Use
    StrategyBehavioralSwap algorithms at runtimePricing, sorting, routing
    ObserverBehavioralNotify many objects of state changesEvents, notifications
    Factory MethodCreationalDelegate object creation to subclassesCreating vehicles, accounts, etc.
    SingletonCreationalOne instance, global accessConfig, logging, connection pool
    BuilderCreationalStep-by-step complex object constructionQuery builders, order builders
    DecoratorStructuralAdd behavior dynamically via wrappingPizza toppings, I/O streams

    Strategy Pattern

    Behavioral

    Problem: You need to switch between different algorithms (pricing strategies, sorting methods, routing policies) without changing the class that uses them.

    Solution: Define a family of algorithms behind an interface. The context object holds a pointer to the interface and delegates work to it.

    ┌─────────────────┐ ┌──────────────────────┐ │ Context │──────▶│ <<interface>> │ │ │ │ Strategy │ │ - strategy │ ├──────────────────────┤ │ + execute() │ │ + execute(): void │ └─────────────────┘ └──────────┬───────────┘ △ ┌──────────┴──────────┐ ┌──────┴──────┐ ┌─────────┴───────┐ │ ConcreteA │ │ ConcreteB │ │ + execute() │ │ + execute() │ └─────────────┘ └─────────────────┘
    class PricingStrategy {
public:
    virtual ~PricingStrategy() = default;
    virtual double calculate(double basePrice) const = 0;
};

class RegularPricing : public PricingStrategy {
public:
    double calculate(double basePrice) const override {
        return basePrice;
    }
};

class HappyHourPricing : public PricingStrategy {
public:
    double calculate(double basePrice) const override {
        return basePrice * 0.5;  // 50% off
    }
};

class PremiumPricing : public PricingStrategy {
public:
    double calculate(double basePrice) const override {
        return basePrice * 1.2;  // 20% markup for premium features
    }
};

class Order {
    unique_ptr<PricingStrategy> pricing;
    double basePrice;
public:
    Order(double price, unique_ptr<PricingStrategy> strategy)
        : basePrice(price), pricing(move(strategy)) {}

    void setPricing(unique_ptr<PricingStrategy> strategy) {
        pricing = move(strategy);  // swap at runtime!
    }

    double getFinalPrice() const {
        return pricing->calculate(basePrice);
    }
};
    Use when: You see if/else chains or switch statements deciding which algorithm to run. Each branch becomes a strategy class.

    Observer Pattern

    Behavioral

    Problem: When one object's state changes, multiple other objects need to be notified (e.g., a stock price changes and all subscribers need an update).

    Solution: The subject maintains a list of observers and notifies them on state changes.

    class Observer {
public:
    virtual ~Observer() = default;
    virtual void update(const string& event, const string& data) = 0;
};

class EventManager {
    unordered_map<string, vector<Observer*>> listeners;
public:
    void subscribe(const string& event, Observer* listener) {
        listeners[event].push_back(listener);
    }

    void unsubscribe(const string& event, Observer* listener) {
        auto& vec = listeners[event];
        vec.erase(remove(vec.begin(), vec.end(), listener), vec.end());
    }

    void notify(const string& event, const string& data) {
        for (auto* listener : listeners[event]) {
            listener->update(event, data);
        }
    }
};

class StockExchange {
    EventManager events;
    unordered_map<string, double> prices;
public:
    EventManager& getEvents() { return events; }

    void updatePrice(const string& ticker, double price) {
        prices[ticker] = price;
        events.notify("price_change", ticker + ": " + to_string(price));
    }
};

class EmailAlert : public Observer {
public:
    void update(const string& event, const string& data) override {
        cout << "[EMAIL] " << event << " — " << data << endl;
    }
};

class DashboardWidget : public Observer {
public:
    void update(const string& event, const string& data) override {
        cout << "[DASHBOARD] Refreshing: " << data << endl;
    }
};
    Use when: Changes in one object should trigger updates in others, but you don't want tight coupling between them. Classic in event systems, notification services, and pub-sub within a process.

    Factory Method Pattern

    Creational

    Problem: Client code needs to create objects but shouldn't know the exact class to instantiate. The creation logic might change or depend on configuration.

    Solution: Define a method that returns an interface pointer. Subclasses or a static method determine which concrete class to instantiate.

    class Vehicle {
public:
    virtual ~Vehicle() = default;
    virtual string getType() const = 0;
    virtual int getWheels() const = 0;
};

class Car : public Vehicle {
public:
    string getType() const override { return "Car"; }
    int getWheels() const override { return 4; }
};

class Bike : public Vehicle {
public:
    string getType() const override { return "Bike"; }
    int getWheels() const override { return 2; }
};

class Truck : public Vehicle {
public:
    string getType() const override { return "Truck"; }
    int getWheels() const override { return 6; }
};

// Simple Factory (most common in interviews)
class VehicleFactory {
public:
    static unique_ptr<Vehicle> create(const string& type) {
        if (type == "car") return make_unique<Car>();
        if (type == "bike") return make_unique<Bike>();
        if (type == "truck") return make_unique<Truck>();
        return nullptr;
    }
};

// Usage
auto vehicle = VehicleFactory::create("car");
cout << vehicle->getType();  // "Car"
    Use when: Object creation involves logic (type-based, config-based), or you want to centralize and encapsulate construction. Very common in Parking Lot, Vending Machine, and similar problems.

    Singleton Pattern

    Creational

    Problem: You need exactly one instance of a class accessible globally (database connection pool, configuration manager, logger).

    Solution: Private constructor, static method returning the single instance.

    class Logger {
    Logger() = default;                        // private constructor
    Logger(const Logger&) = delete;            // no copies
    Logger& operator=(const Logger&) = delete;
public:
    static Logger& getInstance() {
        static Logger instance;  // thread-safe in C++11+
        return instance;
    }

    void log(const string& message) {
        cout << "[LOG] " << message << endl;
    }
};

// Usage
Logger::getInstance().log("System started");

    Warning: Singletons are often overused. They introduce global state and make testing harder. Use them only when you genuinely need exactly one instance. In interviews, mention the trade-offs to show maturity.

    Use when: Exactly one instance is needed system-wide, and it must be accessible from many places. Configuration, logging, thread pools.

    Builder Pattern

    Creational

    Problem: Constructing a complex object requires many optional parameters. A constructor with 10 parameters is unreadable and error-prone.

    Solution: A separate builder class with fluent setter methods, culminating in a build() call.

    class Query {
    string table;
    vector<string> columns;
    string whereClause;
    int limit;
    friend class QueryBuilder;
    Query() : limit(-1) {}
public:
    string toSQL() const {
        string sql = "SELECT ";
        sql += columns.empty() ? "*" : /* join columns */ columns[0];
        for (size_t i = 1; i < columns.size(); i++) sql += ", " + columns[i];
        sql += " FROM " + table;
        if (!whereClause.empty()) sql += " WHERE " + whereClause;
        if (limit > 0) sql += " LIMIT " + to_string(limit);
        return sql;
    }
};

class QueryBuilder {
    Query query;
public:
    QueryBuilder& from(const string& table) {
        query.table = table; return *this;
    }
    QueryBuilder& select(const string& col) {
        query.columns.push_back(col); return *this;
    }
    QueryBuilder& where(const string& clause) {
        query.whereClause = clause; return *this;
    }
    QueryBuilder& limit(int n) {
        query.limit = n; return *this;
    }
    Query build() { return query; }
};

// Usage — readable, order-independent
auto q = QueryBuilder()
    .from("users")
    .select("name").select("email")
    .where("age > 18")
    .limit(10)
    .build();
    Use when: Objects have many optional configuration parameters, or construction requires multiple steps in a specific order.

    Decorator Pattern

    Structural

    Problem: You want to add responsibilities to objects dynamically, without subclassing every possible combination.

    Solution: Wrap the original object in decorator classes that share the same interface. Each decorator adds one behavior and delegates the rest.

    class Coffee {
public:
    virtual ~Coffee() = default;
    virtual string description() const = 0;
    virtual double cost() const = 0;
};

class SimpleCoffee : public Coffee {
public:
    string description() const override { return "Simple coffee"; }
    double cost() const override { return 1.00; }
};

// Base decorator
class CoffeeDecorator : public Coffee {
protected:
    unique_ptr<Coffee> wrapped;
public:
    explicit CoffeeDecorator(unique_ptr<Coffee> c) : wrapped(move(c)) {}
};

class MilkDecorator : public CoffeeDecorator {
public:
    using CoffeeDecorator::CoffeeDecorator;
    string description() const override {
        return wrapped->description() + ", milk";
    }
    double cost() const override { return wrapped->cost() + 0.50; }
};

class WhipDecorator : public CoffeeDecorator {
public:
    using CoffeeDecorator::CoffeeDecorator;
    string description() const override {
        return wrapped->description() + ", whip";
    }
    double cost() const override { return wrapped->cost() + 0.70; }
};

// Usage — stack decorators
unique_ptr<Coffee> order = make_unique<SimpleCoffee>();
order = make_unique<MilkDecorator>(move(order));
order = make_unique<WhipDecorator>(move(order));
cout << order->description();  // "Simple coffee, milk, whip"
cout << order->cost();         // 2.20
    Use when: You need to combine behaviors in many permutations. Toppings on food, middleware layers, I/O streams, logging wrappers.

    Choosing the Right Pattern

    You notice...Reach for...
    if/else or switch selecting behaviorStrategy
    Objects need to react to eventsObserver
    Object creation depends on type/configFactory
    Only one instance needed globallySingleton (use cautiously)
    Complex object with many optional paramsBuilder
    Adding behavior in layers/combinationsDecorator
    Multiple unrelated machines share some flowState / Template Method
