    Introduction to Low Level Design

    March 2026 · 20 min read

    Low Level Design (LLD) is the art of turning requirements into clean, maintainable class structures. While High Level Design (HLD/System Design) asks "which services, databases, and queues do we need?", LLD asks "what are the classes, interfaces, and relationships inside a single service?"

    This series builds your LLD muscle from first principles, SOLID, design patterns, and then full problem walkthroughs that mirror real interview questions.

    OOP Refresher

    LLD is built on four OOP pillars. You don't need to memorize definitions, you need to feel when to apply each one.

    Encapsulation

    Bundle data and the methods that operate on it into a single unit (class). Hide internal state behind a public interface.

    class BankAccount {
    double balance;  // private — nobody touches this directly
public:
    void deposit(double amount) {
        if (amount > 0) balance += amount;  // validation lives here
    }
    double getBalance() const { return balance; }
};

    Why it matters in LLD: Each class owns its data and invariants. You can change internals without breaking callers.

    Abstraction

    Expose what something does, hide how. Interfaces and abstract classes define contracts.

    class PaymentProcessor {  // abstract
public:
    virtual bool charge(double amount) = 0;
    virtual void refund(double amount) = 0;
};

class StripeProcessor : public PaymentProcessor {
    bool charge(double amount) override { /* Stripe API */ }
    void refund(double amount) override { /* Stripe API */ }
};

    Inheritance

    Create specialized types from general ones. Use sparingly, prefer composition when in doubt.

    Polymorphism

    Treat different types through a common interface. The caller doesn't know (or care) which concrete class is behind the pointer.

    void processPayment(PaymentProcessor& processor, double amount) {
    processor.charge(amount);  // works with Stripe, PayPal, anything
}

    The SOLID Principles

    SOLID is the backbone of good LLD. Every design decision in this series ties back to one or more of these principles.

    S Single Responsibility Principle

    A class should have one reason to change. If a class handles both user authentication and email sending, it has two reasons to change, split it.

    O Open/Closed Principle

    Classes should be open for extension, closed for modification. Add new behavior by adding new classes or implementations, not by editing existing code.

    L Liskov Substitution Principle

    Subtypes must be substitutable for their base types without breaking correctness. If Square inherits from Rectangle but breaks when you set width independently of height, it violates LSP.

    I Interface Segregation Principle

    Don't force clients to depend on interfaces they don't use. Many small interfaces beat one fat interface.

    D Dependency Inversion Principle

    High-level modules should depend on abstractions, not concrete implementations. Inject dependencies through constructors or factory methods.

    PrincipleViolationGood Design
    SRPUserService handles auth, email, and loggingAuthService, EmailService, Logger — separate classes

    Reading Class Diagrams

    LLD interviews often involve drawing class diagrams. Here's the notation you need:

    ┌──────────────────────┐ │ <<interface>> │ │ PaymentProcessor │ ├──────────────────────┤ │ + charge(amount): bool│ │ + refund(amount): void│ └──────────┬───────────┘ △ (implements) ┌──────────┴───────────┐ │ StripeProcessor │ ├──────────────────────┤ │ - apiKey: string │ ├──────────────────────┤ │ + charge(amount): bool│ │ + refund(amount): void│ └──────────────────────┘

    Key relationships:

    • Inheritance (△) — "is-a" relationship. StripeProcessor is a PaymentProcessor.
    • Composition (◆) — "has-a, owns it". Car has an Engine; destroy the car, destroy the engine.
    • Aggregation (◇) — "has-a, doesn't own it". Team has Players; players can exist without the team.
    • Association (→) — "uses". OrderService uses PaymentProcessor.

    The LLD Interview Approach

    1. Clarify requirements — ask about use cases, scale, edge cases. Don't jump into code.
    2. Identify entities — nouns in the requirements become classes. Verbs become methods.
    3. Define relationships — who owns whom? Who uses whom? Draw a rough class diagram.
    4. Apply SOLID — check each class has one responsibility, interfaces are small, dependencies are injected.
    5. Pick design patterns — Strategy for swappable behavior, Observer for events, Factory for object creation.
    6. Write code — implement core classes. Show the interviewer your thinking, not just syntax.
    7. Discuss trade-offs — extensibility, testability, thread safety.

    Composition Over Inheritance

    One of the most important LLD instincts: prefer composing objects over inheriting from classes.

    // Inheritance approach — rigid, hard to change
class FlyingDuck : public Duck, public Flyable { };
class RubberDuck : public Duck { }; // Can't fly, but Duck might assume it can

// Composition approach — flexible
class Duck {
    unique_ptr<FlyBehavior> flyBehavior;  // inject strategy
    unique_ptr<QuackBehavior> quackBehavior;
public:
    Duck(unique_ptr<FlyBehavior> fb, unique_ptr<QuackBehavior> qb)
        : flyBehavior(move(fb)), quackBehavior(move(qb)) {}
    void fly() { flyBehavior->fly(); }
    void quack() { quackBehavior->quack(); }
};

    Composition lets you change behavior at runtime, mix and match capabilities, and test each piece in isolation.
