Design: Batch Prediction Pipeline

MLOps Series ML System Design

Requirements & Scope

Not every prediction needs to be real-time. Batch prediction pre-computes scores for all entities — ideal for recommendations, credit scoring, and churn prediction.

Functional

  • Score 100 M+ entities daily
  • Write results to serving store
  • Automated data validation
  • Backfill and re-run support

Non-Functional

  • Complete within 4-hour SLA
  • Idempotent — safe to re-run
  • Cost-efficient (spot instances)
  • Fault-tolerant (checkpoint & resume)

Scheduling & Orchestration

Batch prediction pipelines run on a schedule (daily / hourly) or triggered by upstream data availability.

DAG-Based Orchestration

Tools like Airflow, Prefect, or Dagster define the pipeline as a directed acyclic graph (DAG) of tasks: data validation → feature build → prediction → output validation → publish. 

# Airflow DAG for batch prediction
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

dag = DAG("batch_predictions", schedule_interval="@daily",
         start_date=datetime(2024, 1, 1))

validate_data = PythonOperator(task_id="validate_data", python_callable=validate, dag=dag)
build_features = PythonOperator(task_id="build_features", python_callable=build_feat, dag=dag)
run_predictions = PythonOperator(task_id="run_predictions", python_callable=predict, dag=dag)
validate_output = PythonOperator(task_id="validate_output", python_callable=validate_out, dag=dag)
publish_results = PythonOperator(task_id="publish", python_callable=publish, dag=dag)

validate_data >> build_features >> run_predictions >> validate_output >> publish_results
Idempotency: Every task should be safe to re-run. Use date-partitioned output paths and overwrite-on-rerun semantics.

Parallelism Strategies

Scoring 100 M entities sequentially would take hours. Parallelism strategies make it feasible within the SLA.

Data Parallelism with Spark

Partition the entity table across Spark executors. Each executor loads the model and scores its partition independently. 

# Spark UDF for batch prediction
from pyspark.sql import functions as F
from pyspark.sql.types import FloatType

def predict_udf(features):
    # Model is broadcast to all executors
    return float(model_broadcast.value.predict([features])[0])

GPU Batch Inference

For deep models, run inference on GPU instances with large batch sizes (512–4096). Use Spark + GPU (RAPIDS) or Ray for orchestration.

Spot instance handling: Use checkpointing to resume from the last completed partition if a spot instance is reclaimed.

Output Storage

Batch predictions are written to a serving store where they can be looked up at request time with sub-millisecond latency.

Storage Options

Key-Value (Redis / DynamoDB)

  • Sub-millisecond lookup
  • Simple entity → score mapping
  • Higher cost at scale

Columnar (Parquet on S3)

  • Cheap at scale
  • Good for analytics / debugging
  • Higher latency (not for serving)
# Atomic swap for safe publishing
def publish_predictions(scored_df, redis_client):
    # Write to a staging key prefix
    pipe = redis_client.pipeline()
    for row in scored_df.collect():
        pipe.set(f"pred:staging:{row.entity_id}", row.score)
    pipe.execute()

    # Atomic rename from staging to production
    swap_prefix(redis_client, "pred:staging:", "pred:prod:")

Monitoring & Validation

Batch pipelines fail silently — wrong predictions are served for hours before anyone notices. Automated validation catches issues before publication.

Data Quality Checks

# Output validation
def validate_predictions(scored_df, prev_df):
    checks = {
        "row_count":    scored_df.count() > prev_df.count() * 0.95,
        "null_scores":  scored_df.filter("score IS NULL").count() == 0,
        "score_range":  scored_df.filter("score < 0 OR score > 1").count() == 0,
        "distribution": ks_test(scored_df.score, prev_df.score) > 0.05,
    }
    if not all(checks.values()):
        raise ValidationError(f"Failed checks: {checks}")
    return True
Score distribution drift: Compare today's score histogram to yesterday's using a KS test. A shift indicates a data or model problem.

System Architecture

The pipeline is fully orchestrated by Airflow, with each stage independently retryable. Failed stages alert the on-call team and do not publish stale predictions.
