Design: Recommendation Engine (Netflix / YouTube)

MLOps Series ML System Design

Requirements & Scope

A recommendation engine powers the core experience of platforms like Netflix and YouTube. Before diving into architecture, we must pin down both functional and non-functional requirements.

Functional

  • Personalised home-page feed for each user
  • "Because you watched…" carousels
  • Related-item recommendations on detail pages
  • Trending / popular fallback for cold-start users

Non-Functional

  • Latency < 200 ms p99
  • Scale to 200 M+ daily active users
  • High availability (99.99 %)
  • Model updates without downtime
Tip: Always ask the interviewer whether the system should optimise for engagement, revenue, or satisfaction — these objectives lead to very different ranking functions.

Candidate Generation

The item catalogue may contain millions of entries. The candidate generation stage narrows this to a few hundred relevant items using cheap, high-recall retrievers.

Collaborative Filtering

Matrix-factorisation models (ALS, SVD) learn latent factors for users and items. At serving time, an approximate nearest-neighbour (ANN) index over item embeddings retrieves the top-K candidates in sub-millisecond latency. 

# Approximate nearest-neighbour retrieval with FAISS
import faiss, numpy as np

def build_index(item_embeddings: np.ndarray) -> faiss.IndexIVFFlat:
    dim = item_embeddings.shape[1]
    quantizer = faiss.IndexFlatIP(dim)
    index = faiss.IndexIVFFlat(quantizer, dim, 256, faiss.METRIC_INNER_PRODUCT)
    index.train(item_embeddings)
    index.add(item_embeddings)
    return index

def retrieve_candidates(user_embedding, index, k=200):
    distances, ids = index.search(user_embedding.reshape(1, -1), k)
    return ids[0]

Content-Based Retrieval

For cold-start items (new movies, newly uploaded videos), content features — genre, tags, thumbnails, descriptions — are embedded with a two-tower model and merged into the same ANN index.

Multi-source fusion: Netflix retrieves candidates from collaborative filtering, content-based, trending, and editorial sources, then merges and deduplicates before passing to the ranker.

Ranking Model

The ranker takes the ~200–500 candidates and scores each with a richer, more expensive model — typically a deep neural network with hundreds of features.

Feature Categories

User Features

  • Watch history & ratings
  • Demographics & locale
  • Session context (time, device)
  • Recent search queries

Item Features

  • Genre, actors, director embeddings
  • Popularity & freshness
  • Thumbnail quality score
  • Engagement stats (CTR, watch-time)
# Simplified ranking model (PyTorch)
import torch, torch.nn as nn

class RankingModel(nn.Module):
    def __init__(self, user_dim, item_dim, cross_dim):
        super().__init__()
        self.user_mlp = nn.Sequential(nn.Linear(user_dim, 128), nn.ReLU(), nn.Linear(128, 64))
        self.item_mlp = nn.Sequential(nn.Linear(item_dim, 128), nn.ReLU(), nn.Linear(128, 64))
        self.cross    = nn.Sequential(nn.Linear(cross_dim, 64), nn.ReLU())
        self.output   = nn.Sequential(nn.Linear(192, 64), nn.ReLU(), nn.Linear(64, 1))

    def forward(self, user_feat, item_feat, cross_feat):
        u = self.user_mlp(user_feat)
        i = self.item_mlp(item_feat)
        c = self.cross(cross_feat)
        return torch.sigmoid(self.output(torch.cat([u, i, c], dim=-1)))

Multi-Objective Optimisation

YouTube's ranking model predicts multiple objectives — watch time, likes, shares — then combines them with a weighted scalarisation:

Caching Strategy

Pre-computed recommendations are cached per user with a short TTL (minutes). On cache miss, the system falls back to a lightweight popularity-based ranker, ensuring sub-200 ms latency even under load. 

# Redis caching pattern
import redis, json

r = redis.Redis(host="rec-cache", port=6379)

def get_recs(user_id, k=50):
    cached = r.get(f"recs:{user_id}")
    if cached:
        return json.loads(cached)
    recs = compute_recs(user_id, k)  # expensive path
    r.setex(f"recs:{user_id}", 300, json.dumps(recs))
    return recs

Evaluation & Iteration

Offline and online metrics must align. If your offline NDCG improves but online A/B tests show no engagement lift, your offline evaluation is broken.

Offline Metrics

  • NDCG@K, Recall@K
  • AUC-ROC for click prediction
  • Coverage & diversity

Online Metrics

  • CTR, watch-time per session
  • Long-term retention
  • Revenue per user
# Offline evaluation helper
from sklearn.metrics import ndcg_score
import numpy as np

def evaluate_ranking(true_relevance, predicted_scores, k=20):
    ndcg = ndcg_score([true_relevance], [predicted_scores], k=k)
    recall_at_k = np.sum(true_relevance[:k] > 0) / np.sum(true_relevance > 0)
    return {"ndcg@k": ndcg, "recall@k": recall_at_k}
Feedback loops matter: The items you recommend today shape tomorrow's training data. Guard against popularity bias by injecting exploration (ε-greedy or Thompson sampling).
