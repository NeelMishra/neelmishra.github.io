Design: Feature Platform (Uber Michelangelo)

MLOps Series ML System Design

Requirements & Scope

A feature platform is the backbone of any large-scale ML system. It centralises feature computation, storage, serving, and discovery for hundreds of ML models.

Functional

  • Register & version feature definitions
  • Compute batch and streaming features
  • Serve features at low latency (< 10 ms)
  • Point-in-time correct training data

Non-Functional

  • Serve 500 K+ QPS
  • Support 10 K+ feature definitions
  • Training-serving consistency
  • Data lineage & audit trail

Feature Store

The feature store is the central repository for feature values. It provides both an offline store (for training) and an online store (for serving).

Offline Store

Historical feature values stored in a data warehouse (BigQuery, Hive, Delta Lake) partitioned by entity and timestamp. Supports point-in-time joins for training data generation.

Online Store

The latest feature values per entity in a low-latency store (Redis, DynamoDB, Cassandra). Updated by batch or streaming jobs. 

# Feature definition with Feast
from feast import Entity, FeatureView, Field, ValueType
from feast.types import Float64, Int64

user = Entity(name="user_id", value_type=ValueType.INT64)

user_features = FeatureView(
    name="user_profile_features",
    entities=[user],
    schema=[
        Field(name="avg_session_length", dtype=Float64),
        Field(name="total_purchases", dtype=Int64),
        Field(name="days_since_last_login", dtype=Int64),
    ],
    online=True,
    source=user_profile_source,
    ttl=timedelta(days=1),
)
Point-in-time correctness: When building training data, features must reflect their values at the time of the event — not the latest values. This prevents data leakage.

Feature Compute

Features are computed by batch pipelines (Spark, Dataflow) for historical data and streaming pipelines (Flink, Spark Streaming) for real-time aggregations.

Batch Compute

# Batch feature computation (PySpark)
from pyspark.sql import functions as F

user_features = (
    events_df
    .groupBy("user_id")
    .agg(
        F.avg("session_length").alias("avg_session_length"),
        F.count("purchase_id").alias("total_purchases"),
        F.datediff(F.current_date(), F.max("login_date")).alias("days_since_last_login"),
    )
)

Streaming Compute

Real-time aggregations (transaction count in last 5 min, rolling average) are computed with Flink or Spark Structured Streaming and written directly to the online store.

Compute consistency: Use the same transformation code for batch and streaming (e.g., Apache Beam's unified model) to avoid training-serving skew.

Feature Serving

The serving layer must return features with < 10 ms latency for real-time inference use cases.

Serving Patterns

Key-Value Lookup

  • Single entity: GET user:123
  • Sub-millisecond latency
  • Simple but limited to pre-computed

On-Demand Compute

  • Computed at request time
  • Always fresh
  • Higher latency (5–20 ms)
# Online feature retrieval
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

def get_online_features(user_id, item_id):
    features = store.get_online_features(
        features=[
            "user_profile_features:avg_session_length",
            "user_profile_features:total_purchases",
            "item_features:price",
            "item_features:category_embedding",
        ],
        entity_rows=[{"user_id": user_id, "item_id": item_id}],
    )
    return features.to_dict()

Discovery & Governance

With thousands of features, discoverability and governance become critical. Teams must find existing features before creating new ones.

Feature Catalog

A searchable catalog with feature name, description, owner, data source, freshness SLA, and usage statistics. Think of it as a "data catalog" specifically for ML features.

Governance

Access controls (who can read / write a feature), data lineage (which upstream tables feed this feature), and deprecation policies (notify all consumers before removing a feature).

Feature reuse: Uber found that feature reuse across teams reduced model development time by 50 %. The platform pays for itself through engineering efficiency.

System Architecture

The platform provides a unified abstraction over diverse data sources and compute engines. ML teams interact with a simple SDK while the platform handles the complexity of batch/streaming compute, storage, and serving.
