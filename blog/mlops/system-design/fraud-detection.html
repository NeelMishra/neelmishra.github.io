Design: Real-Time Fraud Detection

MLOps Series ML System Design

Requirements & Scope

Fraud detection systems protect payment platforms processing millions of transactions per second. A missed fraud costs money; a false positive costs customer trust.

Functional

  • Score every transaction in real time
  • Block, allow, or send to manual review
  • Support rule overrides & whitelists
  • Analyst dashboard with explainability

Non-Functional

  • Latency < 100 ms p99
  • Handle 50 K TPS peak
  • False positive rate < 0.1 %
  • Model refresh within minutes

Feature Engineering

Fraud features span three time horizons: real-time (current transaction), near-real-time (last 5 min aggregates), and batch (historical patterns).

Real-Time Features

Computed on the fly from the transaction payload: amount, merchant category code, device fingerprint, IP geolocation, and time-of-day.

Near-Real-Time Aggregates

Streaming aggregations over Kafka / Flink: transaction count in the last 5 minutes, velocity of amount changes, distinct merchant count in the last hour. 

# Flink-style streaming aggregation (pseudocode)
class VelocityFeature:
    def __init__(self, window_sec=300):
        self.window = window_sec
        self.events = []  # sliding window buffer

    def add(self, txn):
        now = txn.timestamp
        self.events = [e for e in self.events if now - e.timestamp < self.window]
        self.events.append(txn)

    def features(self):
        return {
            "txn_count_5m":     len(self.events),
            "total_amount_5m":  sum(e.amount for e in self.events),
            "distinct_merchants": len(set(e.merchant for e in self.events)),
        }

Batch Features

Daily-computed profiles: average transaction amount, typical geolocation, spending patterns by day-of-week, account age, and historical chargeback rate.

Feature freshness matters: A feature that's 10 minutes stale can miss a rapid burst of fraudulent transactions.

Real-Time Scoring

The ML model runs in the critical path of every transaction. It must return a fraud probability within the 100 ms latency budget.

Model Architecture

Gradient-boosted trees (XGBoost / LightGBM) remain the workhorse for tabular fraud data. They're fast at inference, handle mixed feature types, and provide feature-importance for explainability. 

# Real-time scoring with LightGBM
import lightgbm as lgb, numpy as np

model = lgb.Booster(model_file="fraud_model_v12.txt")

def score_transaction(features: dict) -> float:
    arr = np.array([[features[f] for f in FEATURE_ORDER]])
    prob = model.predict(arr)[0]
    return prob  # probability of fraud

Thresholding & Actions

The fraud score is mapped to actions via configurable thresholds: score < 0.3 → approve, 0.3–0.7 → step-up authentication, > 0.7 → block & alert.

Class imbalance: Fraud is typically < 0.1 % of transactions. Use SMOTE, focal loss, or cost-sensitive learning to handle the extreme imbalance.

Rules + ML Hybrid

Pure ML systems struggle with novel attack patterns. A hybrid approach layers deterministic rules on top of ML scoring.

Rule Engine

Hard rules catch known fraud patterns instantly: transactions from sanctioned countries, amounts exceeding account limits, or velocity triggers (10+ transactions in 1 minute). 

# Rule engine (evaluated before ML)
RULES = [
    lambda t: t.country in SANCTIONED_LIST,
    lambda t: t.amount > t.account.daily_limit,
    lambda t: t.velocity_5m > 10,
]

def apply_rules(txn):
    for rule in RULES:
        if rule(txn):
            return "BLOCK"
    return "CONTINUE_TO_ML"

Combining Scores

The final decision merges the rule output and ML score. Rules can override ML (hard blocks) or boost the ML score (soft signals).

System Architecture

The full architecture integrates streaming ingestion, feature computation, model serving, and a feedback loop for continuous learning.

Feedback delay: Fraud labels arrive days or weeks later (chargebacks). Use proxy labels (manual review outcomes) for faster model iteration.

Evaluation & Operations

Fraud models are evaluated on precision-recall trade-offs rather than accuracy, due to extreme class imbalance. 

# Precision-recall evaluation
from sklearn.metrics import precision_recall_curve, auc

def evaluate_fraud_model(y_true, y_scores):
    precision, recall, thresholds = precision_recall_curve(y_true, y_scores)
    pr_auc = auc(recall, precision)
    # Find threshold at 95% recall
    idx = next(i for i, r in enumerate(recall) if r < 0.95)
    return {"pr_auc": pr_auc, "precision_at_95r": precision[idx]}
Adversarial drift: Fraudsters adapt. Monitor your model's performance weekly and retrain on the latest labelled data.
