Design: ML Training Platform

MLOps Series ML System Design

Requirements & Scope

An ML training platform enables teams to submit, run, and track training jobs at scale — from small experiments to large distributed training runs.

Functional

  • Submit jobs with any framework (PyTorch, TF, JAX)
  • Distributed training (data & model parallel)
  • Hyperparameter tuning (grid, Bayesian, PBT)
  • Experiment tracking & comparison

Non-Functional

  • Support 1 000+ concurrent jobs
  • GPU utilisation > 80 %
  • Fair sharing across teams
  • Job isolation & reproducibility

Job Scheduling

The scheduler assigns training jobs to available resources based on priority, resource requirements, and fairness constraints.

Job Specification

# Training job spec (YAML-like)
job:
  name: "fraud_model_v13"
  team: "risk"
  image: "ml-train:pytorch2.1"
  resources:
    gpus: 4
    gpu_type: "A100"
    memory: "64Gi"
  command: "torchrun --nproc_per_node=4 train.py"
  priority: "high"
  max_runtime: "4h"
  checkpoint_interval: "30m"

Scheduling Algorithms

A two-level scheduler works well: Level 1 allocates GPU quotas across teams (fair-share). Level 2 schedules individual jobs within each team's quota using priority ordering. 

# Fair-share scheduling
def allocate_gpus(teams, total_gpus):
    base_share = total_gpus // len(teams)
    allocations = {t.name: base_share for t in teams}

    # Redistribute unused quota
    unused = total_gpus - sum(allocations.values())
    demanding = [t for t in teams if t.demand > allocations[t.name]]
    for t in demanding[:unused]:
        allocations[t.name] += 1
    return allocations
Preemption: High-priority jobs can preempt lower- priority ones. Preempted jobs are checkpointed and resumed when resources become available.

Resource Management

GPU clusters are expensive — maximising utilisation while ensuring fairness is the key challenge.

Cluster Topology

Multi-node training requires fast interconnects. Topology-aware scheduling places related workers on nodes connected by NVLink or InfiniBand for optimal communication.

Spot / Preemptible Instances

Use spot instances for fault-tolerant workloads (HP tuning, research). Checkpointing every N minutes ensures minimal lost work when instances are reclaimed. 

# Auto-checkpointing wrapper
class CheckpointManager:
    def __init__(self, model, optimizer, path, interval_min=30):
        self.model = model
        self.optimizer = optimizer
        self.path = path
        self.interval = interval_min * 60
        self.last_save = time()

    def maybe_save(self, epoch, step):
        if time() - self.last_save > self.interval:
            torch.save({
                "model": self.model.state_dict(),
                "optimizer": self.optimizer.state_dict(),
                "epoch": epoch, "step": step
            }, f"{self.path}/ckpt_{epoch}_{step}.pt")
            self.last_save = time()
GPU memory leaks: Long-running training jobs can accumulate GPU memory. Monitor per-job GPU memory usage and kill jobs that exceed their allocation.

Experiment Tracking

Every training run must be logged with its hyperparameters, metrics, artifacts, and code version for reproducibility. 

# Experiment tracking with MLflow
import mlflow

mlflow.set_experiment("fraud_detection")

with mlflow.start_run(run_name="v13_deeper_network"):
    mlflow.log_params({
        "learning_rate": 0.001,
        "batch_size": 256,
        "hidden_dim": 512,
        "epochs": 50,
    })

    for epoch in range(50):
        train_loss, val_auc = train_one_epoch(model, data)
        mlflow.log_metrics({"train_loss": train_loss, "val_auc": val_auc}, step=epoch)

    mlflow.pytorch.log_model(model, "model")
    mlflow.log_artifact("confusion_matrix.png")

Comparison Dashboard

The experiment UI lets data scientists compare runs side-by-side: metric curves, hyperparameter tables, and artifact diffs. The best run can be promoted to the model registry with one click.

Multi-Tenancy

The platform serves multiple teams with different priorities, budgets, and security requirements.

Isolation

Each team's jobs run in isolated containers (Kubernetes namespaces) with resource quotas. Network policies prevent cross-team data access.

Cost Attribution

Track GPU-hours per team per month for chargeback. Provide cost dashboards showing spend breakdown by job type (experiment, production retraining, HP tuning).

Quota lending: If Team A isn't using their full GPU quota, the scheduler can lend unused capacity to Team B — but reclaim it immediately when Team A submits a job.

System Architecture

The platform runs on Kubernetes with GPU-aware scheduling. Jobs are containerised for reproducibility, and artifacts flow from training through the experiment tracker to the model registry.
