Design: Ads Ranking and Click Prediction

MLOps Series ML System Design

Requirements & Scope

Ads ranking is one of the most revenue-critical ML systems at any ad-supported platform. The system must maximise revenue while maintaining user experience and advertiser ROI.

Functional

  • Predict CTR and conversion rate per ad
  • Run second-price / VCG auction
  • Enforce advertiser budget pacing
  • Support multiple ad formats

Non-Functional

  • Latency < 50 ms p99
  • Handle 1 M+ QPS
  • Revenue within 0.1 % of optimal
  • Model updates every 15 min

CTR Prediction

Click-through rate prediction is the core ML problem. The model estimates P(click | user, ad, context) for every candidate ad.

Model Architecture

Production CTR models typically use a combination of sparse embeddings for categorical features and dense layers. Facebook's DLRM and Google's Wide & Deep are canonical architectures. 

# Simplified DLRM-style CTR model
import torch, torch.nn as nn

class CTRModel(nn.Module):
    def __init__(self, sparse_dims, dense_dim, embed_dim=64):
        super().__init__()
        self.embeddings = nn.ModuleList([
            nn.Embedding(d, embed_dim) for d in sparse_dims
        ])
        self.bottom_mlp = nn.Sequential(
            nn.Linear(dense_dim, 128), nn.ReLU(), nn.Linear(128, embed_dim)
        )
        n_interact = len(sparse_dims) + 1
        self.top_mlp = nn.Sequential(
            nn.Linear(n_interact * (n_interact - 1) // 2 + embed_dim, 128),
            nn.ReLU(), nn.Linear(128, 1)
        )

    def forward(self, sparse_feats, dense_feats):
        embs = [e(s) for e, s in zip(self.embeddings, sparse_feats)]
        d = self.bottom_mlp(dense_feats)
        all_embs = embs + [d]
        # Pairwise dot-product interactions
        interactions = pairwise_dot(all_embs)
        x = torch.cat([d, interactions], dim=-1)
        return torch.sigmoid(self.top_mlp(x))

Calibration

CTR predictions feed directly into the auction, so calibration is critical. Use isotonic regression or Platt scaling to ensure predicted probabilities match observed click rates.

Training data freshness: Ads CTR models are trained on the last 24–48 hours of data, updated every 15 minutes, to capture rapidly changing user behaviour.

Auction Mechanics

Online advertising uses real-time auctions to allocate ad slots. The most common auction type is the generalised second-price (GSP) auction.

eCPM Ranking

Ads are ranked by expected revenue: eCPM = bid × pCTR × pCVR. The ad with the highest eCPM wins the slot but pays the minimum price needed to beat the second-place ad. 

# Second-price auction
def run_auction(candidates):
    # Score each ad
    for ad in candidates:
        ad.ecpm = ad.bid * ad.pctr * ad.pcvr

    # Sort by eCPM descending
    ranked = sorted(candidates, key=lambda a: a.ecpm, reverse=True)

    winner = ranked[0]
    # Second-price: winner pays just enough to beat #2
    if len(ranked) > 1:
        winner.price = ranked[1].ecpm / (winner.pctr * winner.pcvr) + 0.01
    else:
        winner.price = winner.reserve_price
    return winner
Reserve prices: Set minimum bids to prevent ads from winning auctions at near-zero prices, which protects both revenue and user experience quality.

Feature Engineering

Ads ranking features span user, ad, and context dimensions.

User Features

  • Demographic segments
  • Interest embeddings
  • Ad interaction history
  • Purchase recency

Ad / Context Features

  • Ad creative embedding
  • Historical CTR of this ad
  • Page / slot position
  • Time of day, device type

Feature Interactions

Cross-features (user×ad) capture affinity: "users who like sports" × "Nike ad" produces a strong positive signal. Modern models learn these automatically via embedding dot-products.

Bid Optimisation

Advertisers set campaign goals (CPA, ROAS), and the platform's auto-bidding system translates these into per-auction bids. 

# Auto-bidding: target CPA
def compute_bid(ad, pctr, pcvr, target_cpa):
    expected_value = pctr * pcvr * target_cpa
    # Budget pacing multiplier (0.5–1.5)
    pacing = get_pacing_factor(ad.campaign_id)
    return expected_value * pacing

Budget Pacing

A PID controller spreads the daily budget evenly across the day. If spending is ahead of schedule, the pacing factor drops; if behind, it increases. This prevents budget exhaustion before noon.

System Architecture

The full ads serving stack processes an ad request through multiple services in under 50 ms.

A/B testing: Revenue impact of model changes is measured through traffic-splitting experiments with careful statistical analysis before full deployment.
