Design: Real-Time ML Inference Platform

MLOps Series ML System Design

Requirements & Scope

A real-time inference platform serves ML predictions to user-facing applications with strict latency SLAs — typically sub-100 ms.

Functional

  • Serve multiple model versions simultaneously
  • Canary and shadow deployments
  • Online feature computation
  • Prediction logging for offline analysis

Non-Functional

  • p99 latency < 50 ms
  • Handle 100 K+ QPS
  • Zero-downtime model updates
  • Auto-scaling on traffic spikes

Feature Serving

Feature retrieval often dominates the latency budget. A well-designed feature service pre-computes and caches features for low-latency access.

Online vs. Pre-Computed

Features fall into two categories: pre-computed (user profile, historical aggregates — stored in Redis/DynamoDB) and real-time (request context, session state — computed on the fly). 

# Feature serving with Redis
import redis, json

feature_store = redis.Redis(host="features", port=6379, decode_responses=True)

def get_features(user_id, item_id):
    # Pre-computed features (batch-updated)
    user_feat = json.loads(feature_store.get(f"user:{user_id}") or "{}")
    item_feat = json.loads(feature_store.get(f"item:{item_id}") or "{}")

    # Real-time features (computed on request)
    rt_feat = {
        "hour_of_day": current_hour(),
        "user_session_len": get_session_length(user_id),
    }
    return {**user_feat, **item_feat, **rt_feat}
Feature consistency: Training and serving must use the same feature computation logic. Feature stores (Feast, Tecton) enforce this by sharing feature definitions.

Model Serving

The model serving layer loads trained models, handles batching, and manages model versions.

Serving Frameworks

TorchServe / TF Serving

  • Native framework support
  • GPU batching built-in
  • gRPC + REST APIs

  • Multi-framework (PyTorch, TF, ONNX)

Model warmup: The first request after loading a model is 10–100× slower (JIT compilation, GPU memory allocation). Always warm up models with dummy requests before routing traffic.

Caching Strategies

Caching predictions can dramatically reduce compute costs and latency, but only works for deterministic, repeatable queries.

Prediction Cache

Cache the full prediction result keyed by a hash of the input features. Useful for item-level scores that don't change frequently. 

# Prediction caching
import hashlib, json

def cached_predict(model, features, cache, ttl=60):
    key = hashlib.md5(json.dumps(features, sort_keys=True).encode()).hexdigest()
    cached = cache.get(key)
    if cached:
        return json.loads(cached)
    prediction = model.predict(features)
    cache.setex(key, ttl, json.dumps(prediction))
    return prediction

Embedding Cache

Caching intermediate embeddings (user embedding, item embedding) is more effective than caching final predictions because embeddings are reusable across many prediction requests.

Latency Optimisation

Every millisecond matters. Here are the key optimisation levers:

Model Optimisation

Quantisation (FP16 / INT8), pruning, knowledge distillation, and ONNX conversion reduce model size and inference time by 2–10×. 

# ONNX export + quantisation
import torch, onnx
from onnxruntime.quantization import quantize_dynamic

# Export to ONNX
torch.onnx.export(model, dummy_input, "model.onnx", opset_version=13)

# Dynamic quantisation (INT8)
quantize_dynamic("model.onnx", "model_quant.onnx")

Infrastructure

Co-locate feature store and model server in the same region / AZ. Use connection pooling, keep-alive connections, and gRPC (not REST) to minimise network overhead.

Latency budget: Feature fetch ≤ 15 ms, model inference ≤ 20 ms, post-processing ≤ 5 ms, network overhead ≤ 10 ms = 50 ms total.

System Architecture

The architecture separates concerns: the prediction service orchestrates feature fetching and model inference, while the model server handles GPU management and batching independently.
