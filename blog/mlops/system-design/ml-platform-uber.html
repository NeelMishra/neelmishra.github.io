Case Study: Uber Michelangelo, Meta FBLearner, Google TFX

MLOps Series ML System Design

Platform Overview

The three most influential ML platforms in industry were built by Uber, Meta, and Google. Each addresses the same fundamental problem — taking ML from notebook to production — but makes different architectural choices reflecting its organisation's unique constraints.

Common Goals

  • Democratise ML across the company
  • Reduce time from idea to production
  • Standardise training, serving, and monitoring
  • Enable feature reuse and model governance

Key Differences

  • Framework support breadth
  • Batch vs. real-time emphasis
  • Level of abstraction
  • Open-source vs. proprietary

Uber Michelangelo

Michelangelo was Uber's end-to-end ML platform, handling everything from feature engineering through model serving for hundreds of ML use cases — ETA prediction, fraud detection, pricing, and more.

Key Components

Feature Store (Palette): Centralised feature computation and serving with both batch (Hive) and real-time (Cassandra) stores. Teams register features once and reuse across models.

Training Service: Supports Spark MLlib, XGBoost, TensorFlow, and PyTorch. Jobs run on a shared YARN cluster with fair-share scheduling.

Serving: Online (real-time REST API) and offline (batch prediction via Spark) serving modes. Models are deployed as Docker containers behind a load balancer. 

# Michelangelo model deployment (conceptual)
from michelangelo import ModelClient

client = ModelClient()

# Register model
client.register_model(
    name="eta_prediction",
    version="v42",
    artifact_path="s3://models/eta/v42",
    features=["pickup_lat", "pickup_lng", "hour_of_day", "surge_mult"],
    framework="xgboost"
)

# Deploy to production
client.deploy(
    model="eta_prediction",
    version="v42",
    strategy="canary",
    traffic_pct=10
)
Lesson: Michelangelo's feature store was its biggest force multiplier. Centralising features eliminated duplicate feature engineering across 100+ ML projects.

Meta FBLearner Flow

FBLearner Flow was Meta's ML workflow management system, processing over 1 million ML training jobs per month. It pioneered the "ML pipeline as code" paradigm.

Key Design Choices

Pipeline abstraction: Every ML workflow is a DAG of typed operators. Operators are reusable and composable — the same data preprocessing operator works across News Feed ranking, Ads, and Integrity models.

AutoML integration: FBLearner includes built-in hyperparameter optimisation (Bayesian optimisation) and AutoML capabilities that automatically search model architectures. 

# FBLearner Flow pipeline (conceptual)
from fblearner import Pipeline, Operator

@Operator(input_type=DataTable, output_type=FeatureTable)
def build_features(data):
    return compute_features(data, feature_config)

@Operator(input_type=FeatureTable, output_type=Model)
def train_model(features):
    return train(features, model_config)

pipeline = Pipeline([build_features, train_model])
pipeline.run(schedule="daily")

Scale

At peak, FBLearner managed 1M+ experiments per month, trained models on petabytes of data, and served predictions for every major Meta product surface.

Tight coupling: FBLearner was deeply integrated with Meta's internal infrastructure (Hive, FBOSS, Thrift). This made it extremely efficient but impossible to open-source.

Google TFX

TFX (TensorFlow Extended) is Google's production ML platform, open-sourced in 2019. It's the most accessible of the three platforms, with a full open-source stack.

Component Library

TFX provides a standard set of pipeline components:

Data Components

  • ExampleGen: Data ingestion
  • StatisticsGen: Data profiling
  • SchemaGen: Schema inference
  • ExampleValidator: Anomaly detection

Model Components

  • Transform: Feature engineering
  • Trainer: Model training
  • Evaluator: Model validation
  • Pusher: Model deployment
# TFX pipeline definition
from tfx.components import (
    CsvExampleGen, StatisticsGen, SchemaGen,
    ExampleValidator, Transform, Trainer, Evaluator, Pusher
)
from tfx.orchestration.local.local_dag_runner import LocalDagRunner

example_gen   = CsvExampleGen(input_base="data/")
statistics    = StatisticsGen(examples=example_gen.outputs["examples"])
schema        = SchemaGen(statistics=statistics.outputs["statistics"])
validator     = ExampleValidator(statistics=statistics.outputs["statistics"],
                                  schema=schema.outputs["schema"])
transform     = Transform(examples=example_gen.outputs["examples"],
                          schema=schema.outputs["schema"],
                          module_file="preprocessing.py")
trainer       = Trainer(module_file="model.py",
                        transformed_examples=transform.outputs["transformed_examples"])
evaluator     = Evaluator(examples=example_gen.outputs["examples"],
                          model=trainer.outputs["model"])
pusher        = Pusher(model=trainer.outputs["model"],
                       model_blessing=evaluator.outputs["blessing"])
Data validation as a first-class citizen: TFX's emphasis on data validation (TFDV) before training is its most impactful design choice. Most ML bugs originate in data, not code.

Architecture Comparison

Michelangelo (Uber)

  • Approach: Monolithic, end-to-end
  • Strengths: Feature store, multi-framework
  • Serving: Online + batch
  • Open source: No (proprietary)

FBLearner (Meta)

  • Approach: Pipeline-centric DAGs
  • Strengths: Scale, AutoML, reuse
  • Serving: Online-focused
  • Open source: No (proprietary)

TFX (Google)

  • Approach: Component library, composable
  • Strengths: Data validation, open source, standardised artifacts
  • Serving: TF Serving (online) + Dataflow (batch)
  • Open source: Yes (Apache 2.0)

Lessons Learned

Across all three platforms, several universal lessons emerge:

1. Feature Management is the Biggest ROI

All three platforms found that centralising feature engineering — through a feature store or standardised feature pipelines — delivered the largest productivity gains for ML teams.

2. Data Quality is More Important Than Model Quality

Google's emphasis on data validation in TFX reflects hard-won experience: most production ML failures are data failures, not model failures.

3. Standardised Pipelines Enable Velocity

When every team builds bespoke training and serving code, the organisation can't scale ML. Standard components reduce development time from months to days.

4. Monitoring Must Be Built-In

Bolt-on monitoring doesn't work. All three platforms integrated monitoring, data validation, and alerting as core features, not afterthoughts.

5. Platform Adoption Requires Ease of Use

The best platform is one that ML engineers actually use. If the platform is harder than writing custom code, teams will bypass it. All three invested heavily in UX, documentation, and migration tools.

Build or buy? Unless you have 50+ ML engineers, you're better off assembling a platform from open-source tools (MLflow, Feast, KServe, Airflow) than building a Michelangelo from scratch.
Platform team size: Uber and Meta each dedicated 50–100 engineers to their ML platforms. If you have fewer than 10 platform engineers, scope ruthlessly — start with a feature store and experiment tracking, not the full stack.
