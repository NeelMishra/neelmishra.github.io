Design: A/B Testing & Experimentation

MLOps Series ML System Design

Requirements & Scope

An experimentation platform lets teams run hundreds of A/B tests simultaneously, measure their impact rigorously, and make data-driven ship/no-ship decisions.

Functional

  • Deterministic user-to-variant assignment
  • Support overlapping experiments
  • Automated metric computation & analysis
  • Guardrail metrics with auto-shutoff

Non-Functional

  • Assignment latency < 5 ms
  • Support 500+ concurrent experiments
  • Type I error rate ≤ 5 %
  • Results available within 24 hours

Traffic Assignment

Each user must be deterministically assigned to a variant (control or treatment) such that the assignment is consistent across sessions and statistically balanced.

Hashing-Based Assignment

Hash the (user_id, experiment_id) pair and map the hash to a bucket. This ensures the same user always gets the same variant without storing state. 

# Deterministic hashing assignment
import hashlib

def assign_variant(user_id: str, experiment_id: str, num_variants: int = 2) -> int:
    key = f"{user_id}:{experiment_id}"
    h = int(hashlib.md5(key.encode()).hexdigest(), 16)
    return h % num_variants  # 0 = control, 1 = treatment

Overlapping Experiments

When multiple experiments run simultaneously, use independent hash salts per experiment to ensure orthogonal assignments — a user's assignment in Experiment A doesn't correlate with Experiment B.

Traffic layers: Google's experiment system uses "layers" — each layer controls a specific part of the product (e.g., ranking, UI, ads). Experiments within the same layer are mutually exclusive; across layers they overlap freely.

Metric Computation

The metric pipeline joins user assignments with behavioural events to compute per-variant aggregates. 

# Metric computation pipeline (SQL-like)
def compute_metrics(experiment_id, start_date, end_date):
    query = f"""
    SELECT
        a.variant,
        COUNT(DISTINCT a.user_id) AS users,
        AVG(e.revenue)          AS avg_revenue,
        SUM(e.clicks) / SUM(e.impressions) AS ctr,
        AVG(e.session_length)   AS avg_session
    FROM assignments a
    JOIN events e ON a.user_id = e.user_id
    WHERE a.experiment_id = '{experiment_id}'
      AND e.date BETWEEN '{start_date}' AND '{end_date}'
    GROUP BY a.variant
    """
    return execute(query)

Metric Types

Primary Metrics

  • The key metric the experiment aims to move
  • CTR, revenue, conversion rate
  • Pre-registered before experiment starts

Guardrail Metrics

  • Must NOT degrade
  • Latency, error rate, DAU
  • Trigger auto-shutoff if violated

Statistical Analysis

Rigorous statistical testing prevents shipping changes that had no real effect (or worse, hurt metrics).

Hypothesis Testing

# Two-sample t-test for experiment analysis
from scipy import stats
import numpy as np

def analyse_experiment(control_values, treatment_values, alpha=0.05):
    t_stat, p_value = stats.ttest_ind(control_values, treatment_values)
    control_mean = np.mean(control_values)
    treatment_mean = np.mean(treatment_values)
    lift = (treatment_mean - control_mean) / control_mean

    return {
        "control_mean": control_mean,
        "treatment_mean": treatment_mean,
        "lift": lift,
        "p_value": p_value,
        "significant": p_value < alpha,
        "confidence": 1 - alpha,
    }

Multiple Testing Correction

With hundreds of concurrent experiments, false discoveries accumulate. Apply Bonferroni or Benjamini-Hochberg correction to control the false discovery rate.

Power Analysis

Before launching, compute the minimum sample size needed to detect the expected effect size with 80 % power. 

# Sample size calculation
from statsmodels.stats.power import TTestIndPower

power_analysis = TTestIndPower()
sample_size = power_analysis.solve_power(
    effect_size=0.05,  # 5% relative lift
    alpha=0.05,
    power=0.80,
    alternative="two-sided"
)
print(f"Need {int(sample_size)} users per variant")
Peeking: Checking results daily and stopping early when p < 0.05 inflates false positive rates. Use sequential testing (always-valid p-values) if early stopping is needed.

Guardrails & Safety

Guardrails protect against experiments that improve one metric while silently degrading another.

Auto-Shutoff

If a guardrail metric (e.g., error rate, latency p99) degrades beyond a threshold, the experiment is automatically disabled and the team is alerted. 

# Guardrail monitoring
GUARDRAILS = {
    "error_rate": {"max_delta": 0.005},  # max 0.5% increase
    "latency_p99": {"max_delta": 50},    # max 50ms increase
    "dau": {"min_delta": -0.01},         # max 1% decrease
}

def check_guardrails(experiment_id, metrics):
    for metric, limits in GUARDRAILS.items():
        delta = metrics[metric]["treatment"] - metrics[metric]["control"]
        if "max_delta" in limits and delta > limits["max_delta"]:
            disable_experiment(experiment_id, reason=f"{metric} exceeded")
        if "min_delta" in limits and delta < limits["min_delta"]:
            disable_experiment(experiment_id, reason=f"{metric} degraded")

Ramp-Up Strategy

Start experiments at 1 % traffic. If guardrails pass after 24 hours, ramp to 10 %, then 50 %, then 100 %. This limits blast radius.

System Architecture

Culture of experimentation: The platform is only valuable if teams actually use it. Make it easy to create, monitor, and analyse experiments — friction kills adoption.
