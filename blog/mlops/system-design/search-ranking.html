Design: Search Ranking System

MLOps Series ML System Design

Requirements & Scope

Search ranking underpins e-commerce, web search, and enterprise knowledge bases. The system must return relevant results in under 300 ms while handling billions of documents.

Functional

  • Full-text & semantic search
  • Auto-complete & spell correction
  • Faceted filtering (price, category)
  • Personalised result ordering

Non-Functional

  • p99 latency < 300 ms
  • Index 10 B+ documents
  • 99.99 % availability
  • Incremental index updates in < 1 min

Query Understanding

Before retrieval, the query is transformed through several NLP stages: tokenisation, spell correction, query expansion, and intent classification.

Intent Classification

A lightweight classifier determines if the query is navigational ("facebook login"), informational ("how to cook pasta"), or transactional ("buy iPhone 15"). This routes the query to different ranking models. 

# Intent classification with a BERT-tiny model
from transformers import pipeline

classifier = pipeline("text-classification",
                      model="intent-bert-tiny",
                      device=0)

def classify_intent(query: str) -> str:
    result = classifier(query)[0]
    return result["label"]  # navigational | informational | transactional

Query Expansion

Synonyms and related terms are added to improve recall. For instance, "laptop" may expand to include "notebook computer", "MacBook", and "Chromebook" using a learned synonym table.

Latency budget: Query understanding must complete in < 20 ms to leave enough budget for retrieval and ranking.

Retrieval Stage

Retrieval is the high-recall, low-precision stage. It must narrow billions of documents to a few thousand candidates efficiently.

Inverted Index (BM25)

Traditional term-based retrieval uses BM25 scoring over an inverted index (Elasticsearch / Lucene). It handles exact-match queries well but struggles with semantic intent.

Dense Retrieval

A bi-encoder embeds both query and documents into the same vector space. Approximate nearest-neighbour search (HNSW, ScaNN) retrieves semantically similar documents even when keywords don't match. 

# Hybrid retrieval: BM25 + dense
def hybrid_retrieve(query, bm25_index, ann_index, encoder, k=1000):
    # Sparse retrieval
    bm25_results = bm25_index.search(query, k=k)

    # Dense retrieval
    q_emb = encoder.encode(query)
    dense_results = ann_index.search(q_emb, k=k)

    # Reciprocal rank fusion
    return rrf_merge(bm25_results, dense_results, k=k)

Multi-Stage Ranking

A cascade of models with increasing complexity scores candidates. Each stage halves the candidate set while using a more expensive model.

L1 — Lightweight Ranker

A GBDT (LightGBM/XGBoost) over ~50 hand-crafted features scores the top 1 000 candidates from retrieval in < 10 ms.

L2 — Deep Ranker

A transformer cross-encoder re-scores the top 100 candidates with full query-document attention. This captures fine-grained relevance but costs ~1 ms per document. 

# L2 cross-encoder re-ranking
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-12-v2")

def rerank(query, docs, top_k=20):
    pairs = [(query, d["text"]) for d in docs]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(docs, scores), key=lambda x: x[1], reverse=True)
    return [d for d, s in ranked[:top_k]]
Latency trade-off: Cross-encoders are 100× slower than bi-encoders. Keep the L2 candidate set small (≤ 100) and use GPU batching.

Feature Engineering

Search ranking features fall into three categories: query features, document features, and query-document interaction features.

Query Features

  • Query length & token count
  • Intent probability vector
  • Historical CTR for this query
  • Is-navigational flag

Document Features

  • PageRank / authority score
  • Freshness (publish date)
  • Click-through rate
  • Content quality score
# Feature extraction pipeline
def extract_features(query, doc):
    return {
        # Query-document interaction
        "bm25_score":      bm25(query, doc),
        "cosine_sim":      cosine(query_emb, doc_emb),
        "title_overlap":   jaccard(query_tokens, doc.title_tokens),
        # Document static
        "page_rank":       doc.page_rank,
        "freshness_days":  (now() - doc.publish_date).days,
        "click_rate":      doc.clicks / doc.impressions,
        # Query static
        "query_length":    len(query_tokens),
        "intent_score":    intent_model.predict(query),
    }

Re-Ranking & Blending

After ML scoring, business logic and diversity constraints shape the final SERP (search engine results page).

Diversity

Maximal Marginal Relevance (MMR) penalises items too similar to already-selected results, ensuring the top 10 covers diverse intents.

Blending Verticals

Web results, images, videos, news, and shopping results are merged into a single page. A blending model decides optimal slot allocation.

Position bias: Users click higher-ranked results regardless of relevance. De-bias your training data using inverse propensity weighting or position-aware models.
