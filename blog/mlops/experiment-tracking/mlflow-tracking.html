MLflow Architecture Overview

MLflow is an open-source platform created by Databricks that manages the complete machine learning lifecycle. While it ships with four major components — Tracking, Projects, Models, and the Model Registry — the Tracking server is where most day-to-day experimentation lives. Understanding how these pieces connect is essential for building a robust MLOps pipeline.

At its core, the MLflow Tracking component provides a logging API and a backend store. Every call to mlflow.log_param() or mlflow.log_metric() writes data to this store, which can be a local filesystem, a SQLite database, a PostgreSQL instance, or a remote tracking server.

Key insight: MLflow separates metadata (parameters, metrics, tags) from artifacts (model files, images, data). Metadata goes to the backend store; artifacts go to an artifact repository (local disk, S3, GCS, Azure Blob, etc.).

The architecture follows a client-server model. Your training script is the client. It talks to the tracking server via REST API calls. The server persists runs into the backend store and routes artifact uploads to the configured artifact root. In local mode, the client writes directly to ./mlruns without needing a server at all. 

# Typical project structure when using MLflow locally
my_project/
├── mlruns/                # auto-created backend store
│   ├── 0/                 # default experiment
│   │   └── <run_id>/
│   │       ├── params/
│   │       ├── metrics/
│   │       ├── tags/
│   │       └── artifacts/
│   └── 1/                 # custom experiment
├── train.py
└── MLproject              # optional project descriptor

When you scale beyond a single laptop, you deploy a dedicated tracking server. The server provides the same REST interface but stores metadata in a relational database and artifacts in cloud object storage. This enables team-wide collaboration — everyone logs to the same server and can compare each other's runs. 

# Launch a remote tracking server with Postgres + S3
# (run this on your server machine)
mlflow server \
    --backend-store-uri postgresql://user:pass@db:5432/mlflow \
    --default-artifact-root s3://my-bucket/mlflow-artifacts \
    --host 0.0.0.0 \
    --port 5000

Tracking Runs and Experiments

A run is the fundamental unit in MLflow Tracking. It represents a single execution of your training code — one set of hyperparameters, one training loop, one set of results. Runs are grouped into experiments, which act as named folders for related runs.

The simplest way to create a run is the mlflow.start_run() context manager. Inside the context, every logging call is automatically associated with that run. When the context exits, the run is marked as finished. 

import mlflow

# Set which experiment to log to (creates it if needed)
mlflow.set_experiment("text-classification")

with mlflow.start_run(run_name="baseline-lr") as run:
    # Log hyperparameters
    mlflow.log_param("model_type", "logistic_regression")
    mlflow.log_param("learning_rate", 0.01)
    mlflow.log_param("max_iter", 500)

    # ... train your model ...

    # Log evaluation metrics
    mlflow.log_metric("accuracy", 0.89)
    mlflow.log_metric("f1_score", 0.87)

    print("Run ID:", run.info.run_id)

You can also log metrics at multiple steps to capture training curves. This is especially useful for deep learning, where you want to visualize loss over epochs: 

for epoch in range(100):
    loss = train_one_epoch(model, data)
    val_acc = evaluate(model, val_data)

    # Step parameter tracks metric progression
    mlflow.log_metric("train_loss", loss, step=epoch)
    mlflow.log_metric("val_accuracy", val_acc, step=epoch)
Tip: Use mlflow.log_params() (plural) to log a dictionary of parameters in one call. Similarly, mlflow.log_metrics() logs multiple metrics at once. Batch operations reduce network overhead when using a remote tracking server.

Every run is assigned a unique run ID (a UUID). You can use this ID to fetch results later via the Python client or REST API. Runs also store a timestamp, the user who launched them, the source code reference, and any custom tags you attach. 

# Retrieve a past run by ID
from mlflow.tracking import MlflowClient

client = MlflowClient()
run = client.get_run("a1b2c3d4e5f6...")

print("Parameters:", run.data.params)
print("Metrics:", run.data.metrics)
print("Status:", run.info.status)

Autologging with MLflow

Manually calling log_param and log_metric for every hyperparameter and evaluation score is tedious. MLflow's autologging feature solves this by automatically intercepting framework-specific training calls and recording everything for you — parameters, metrics, and even the trained model artifact. 

import mlflow
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split
from sklearn.datasets import load_iris

# Enable autologging for scikit-learn
mlflow.sklearn.autolog()

X, y = load_iris(return_X_y=True)
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

with mlflow.start_run():
    clf = RandomForestClassifier(n_estimators=100, max_depth=5)
    clf.fit(X_train, y_train)
    # MLflow auto-logs: n_estimators, max_depth, accuracy,
    # f1, confusion matrix, the model pickle, and more

MLflow supports autologging for many popular frameworks. Here is a summary of what gets captured:

scikit-learn

All constructor parameters, training & validation metrics, serialized model, feature importance plots.

PyTorch / Lightning

Optimizer settings, loss per epoch, learning rate schedule, model checkpoints, gradient norms.

TensorFlow / Keras

Layer configuration, compile arguments, epoch-level metrics, TensorBoard logs, SavedModel export.

XGBoost / LightGBM

Booster parameters, eval metrics per iteration, feature importance, serialized booster file.

You can also use the generic mlflow.autolog() call, which enables autologging for every supported library detected in the environment. This is the fastest way to get started, though you may want framework-specific calls for finer control. 

# Enable autologging for ALL supported frameworks at once
import mlflow
mlflow.autolog()

# Now any sklearn, pytorch, xgboost, etc. training
# will be automatically tracked
Caveat: Autologging patches library internals using monkey-patching. In rare cases this can conflict with custom callbacks or decorators. If you see unexpected behavior, switch to explicit logging for that specific framework.

Artifacts and Model Logging

Beyond scalar parameters and metrics, MLflow can store arbitrary artifacts — files or directories associated with a run. Common artifacts include serialized models, plots, data samples, configuration YAML files, and evaluation reports. Artifacts are stored in the configured artifact repository and can be downloaded later via the UI or API. 

import mlflow
import matplotlib.pyplot as plt

with mlflow.start_run():
    # Train your model ...
    mlflow.log_param("epochs", 50)

    # Create a plot
    fig, ax = plt.subplots()
    ax.plot(train_losses, label="train")
    ax.plot(val_losses, label="val")
    ax.legend()
    fig.savefig("loss_curve.png")

    # Log the plot as an artifact
    mlflow.log_artifact("loss_curve.png", artifact_path="plots")

    # Log an entire directory
    mlflow.log_artifacts("./output_reports", artifact_path="reports")

For machine learning models specifically, MLflow provides a higher-level log_model API. This not only saves the serialized model but wraps it in the MLflow Model format — a directory containing the model file, a MLmodel descriptor, a conda.yaml environment spec, and a requirements.txt. 

import mlflow.sklearn

with mlflow.start_run():
    clf = RandomForestClassifier(n_estimators=200)
    clf.fit(X_train, y_train)
    score = clf.score(X_test, y_test)
    mlflow.log_metric("accuracy", score)

    # Log the model in MLflow format
    mlflow.sklearn.log_model(
        sk_model=clf,
        artifact_path="random_forest_model",
        registered_model_name="iris-classifier"
    )

    # Later, load it back for inference
    model_uri = "runs:/{}/random_forest_model".format(
        mlflow.active_run().info.run_id
    )
    loaded = mlflow.sklearn.load_model(model_uri)
Model flavors: Each framework has its own "flavor" — mlflow.sklearn, mlflow.pytorch, mlflow.tensorflow, etc. This means MLflow knows how to serialize and deserialize the model correctly for each library. A generic mlflow.pyfunc flavor provides a unified prediction interface across all of them.

When you pass registered_model_name to log_model, MLflow automatically creates (or updates) a registered model in the Model Registry. This enables model versioning, stage transitions (Staging → Production), and approval workflows — all critical for production ML systems.

The MLflow UI

One of MLflow's greatest strengths is its built-in web UI. Launch it with a single command and you get a rich interface for browsing experiments, comparing runs, viewing metric charts, and downloading artifacts — no extra setup required. 

# Launch the UI pointing at your local mlruns directory
mlflow ui --port 5000

# Or point at a remote backend
mlflow ui --backend-store-uri postgresql://user:pass@host/mlflow

The UI is organized around a few key views:

  • Experiment list: Shows all experiments. Click one to see its runs.
  • Run table: A sortable, filterable table of runs with columns for each parameter and metric. You can customize which columns appear.
  • Run detail: Click a run to see its full parameter set, metric history charts, tags, artifacts tree, and logged model info.
  • Compare view: Select multiple runs and click "Compare" to see a side-by-side parameter diff, overlaid metric charts, and scatter plots.
  • Model Registry: A separate tab for managing registered models, their versions, stage transitions, and descriptions.
Pro tip: Use the search bar in the run table to filter runs with SQL-like expressions: metrics.accuracy > 0.9 AND params.model_type = "xgboost". This is invaluable when you have hundreds of runs and need to find the best ones quickly.

You can also query runs programmatically using the MlflowClient.search_runs() method, which accepts the same filter syntax: 

from mlflow.tracking import MlflowClient

client = MlflowClient()

# Find the best run by accuracy
runs = client.search_runs(
    experiment_ids=["1"],
    filter_string="metrics.accuracy > 0.85",
    order_by=["metrics.accuracy DESC"],
    max_results=5
)

for r in runs:
    print(r.info.run_id, r.data.metrics["accuracy"])

Advanced Features

Once you are comfortable with basic tracking, MLflow offers several advanced capabilities that become essential at scale.

Nested Runs

When performing hyperparameter searches, you can create nested runs — a parent run that contains child runs for each parameter combination. This keeps your experiment list clean while preserving the full search history. 

import mlflow

with mlflow.start_run(run_name="hp-search") as parent:
    for lr in [0.001, 0.01, 0.1]:
        for depth in [3, 5, 10]:
            with mlflow.start_run(nested=True) as child:
                mlflow.log_param("lr", lr)
                mlflow.log_param("max_depth", depth)
                acc = train_and_evaluate(lr, depth)
                mlflow.log_metric("accuracy", acc)

Custom Tags and Run Organization

Tags are arbitrary key-value strings attached to a run. Use them to annotate runs with metadata that is not a hyperparameter — for example, the team name, the Git commit hash, or the dataset version. 

with mlflow.start_run():
    mlflow.set_tag("team", "nlp-research")
    mlflow.set_tag("dataset_version", "v2.3")
    mlflow.set_tag("git_sha", "abc123f")
    # ... train and log ...

Model Registry Workflows

The Model Registry provides a centralized hub for managing model versions. After logging a model, you can transition it through stages — None → Staging → Production → Archived. This is typically integrated with CI/CD pipelines for automated deployment. 

from mlflow.tracking import MlflowClient

client = MlflowClient()

# Transition model version 3 to Production
client.transition_model_version_stage(
    name="iris-classifier",
    version=3,
    stage="Production"
)

# Load the production model for serving
import mlflow.pyfunc
model = mlflow.pyfunc.load_model(
    model_uri="models:/iris-classifier/Production"
)
predictions = model.predict(new_data)

Remote Tracking with Authentication

In production setups, the tracking server often sits behind authentication. MLflow supports setting the tracking URI and credentials via environment variables, making it easy to integrate with secret managers and CI/CD systems. 

import os
import mlflow

# Point to the team's remote tracking server
os.environ["MLFLOW_TRACKING_URI"] = "https://mlflow.company.com"
os.environ["MLFLOW_TRACKING_USERNAME"] = "ml-engineer"
os.environ["MLFLOW_TRACKING_PASSWORD"] = os.getenv("MLFLOW_TOKEN")

# All subsequent calls now go to the remote server
mlflow.set_experiment("production-retraining")
with mlflow.start_run():
    # ... your training code ...
    pass
