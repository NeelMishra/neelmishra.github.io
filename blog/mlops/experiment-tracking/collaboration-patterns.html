Collaboration Patterns for ML Experiments

MLOps Series Experiment Tracking

Collaboration Challenges in ML

ML teams face unique collaboration problems that software engineering solved decades ago. When three data scientists run experiments simultaneously, they produce hundreds of runs with cryptic names, incompatible configurations, and no shared context about what was tried and why. The result is duplicated effort, lost knowledge, and an inability to build on each other's work.

Unlike code — where Git provides a universal collaboration protocol — ML experiments lack a standard workflow for sharing, reviewing, and building upon results. Teams need explicit patterns for infrastructure, naming, review, and documentation.

The hidden cost: Studies show ML teams spend 20-40% of their time re-running experiments that a teammate already tried. Structured collaboration patterns can recover most of this lost productivity.

Shared Experiment Infrastructure

The foundation of team collaboration is a centralized experiment tracking server that all team members log to. No local databases, no personal notebooks — every experiment lands in the shared system with consistent metadata. 

# Setting up shared MLflow tracking for the team
import mlflow
import os

def setup_team_tracking(team_name, project_name):
    # Point all team members to the shared server
    mlflow.set_tracking_uri("https://mlflow.internal.company.com")

    # Use project-level experiment namespace
    experiment_name = f"/{team_name}/{project_name}"
    mlflow.set_experiment(experiment_name)

    # Auto-tag with user identity
    user = os.environ.get("USER", "unknown")
    mlflow.set_tag("mlflow.user", user)
    mlflow.set_tag("team", team_name)

    return experiment_name

# Every team member calls this at script start
exp = setup_team_tracking("recommendation-team", "click-prediction-v2")
# Weights & Biases team setup
import wandb

def setup_wandb_team(project, entity, run_group):
    wandb.init(
        project=project,
        entity=entity,        # Team organization
        group=run_group,      # Logical grouping (e.g., "feature-v3")

Access control matters: Set up read/write permissions per project. Data scientists should be able to read all experiments in their project but only modify their own runs. Protect production-critical experiments from accidental deletion.

Naming Conventions and Taxonomy

Without naming conventions, experiment lists become incomprehensible within weeks. Establish a team-wide taxonomy for run names, tags, and parameter keys from day one.

✘ Bad Naming

  • test_run_3
  • final_v2_really_final
  • alice_experiment
  • fix_bug_retry
  • new_model

✔ Good Naming

  • ctr/transformer-small/lr3e-4/v1
  • ctr/gbm-baseline/feat-v2/prod
  • ctr/transformer-small/dropout-sweep
  • ctr/lstm/embedding-fix/retrain
  • ctr/ensemble/top3-stack/eval
# Enforcing naming conventions programmatically
import re
import mlflow

NAMING_PATTERN = re.compile(
    r"^[a-z]+/[a-z0-9-]+/[a-z0-9-]+/(v\d+|sweep|eval|prod)$"
)

def validated_run(run_name, **kwargs):
    if not NAMING_PATTERN.match(run_name):
        raise ValueError(
            f"Run name '{run_name}' doesn't match convention: "
            f"'project/model/variant/stage'"
        )
    return mlflow.start_run(run_name=run_name, **kwargs)

# Required tags for every experiment
REQUIRED_TAGS = ["owner", "hypothesis", "dataset_version", "priority"]

def validate_tags(tags):
    missing = [t for t in REQUIRED_TAGS if t not in tags]
    if missing:
        raise ValueError(f"Missing required tags: {missing}")
Enforce, don't suggest: Naming conventions only work if they're enforced programmatically. Add validation to your experiment wrapper library so non-conforming runs are rejected before they pollute the tracking server.

Experiment Review Workflows

Just as code changes go through pull request reviews, experiment results should go through structured review before influencing decisions. An experiment review checks methodology, statistical validity, reproducibility, and alignment with project goals. 

# Experiment review checklist automation
import mlflow
from dataclasses import dataclass

@dataclass
class ExperimentReview:
    run_id: str
    reviewer: str
    methodology_ok: bool = False
    stats_valid: bool = False
    reproducible: bool = False
    goal_aligned: bool = False
    notes: str = ""

    @property
    def approved(self):
        return all([
            self.methodology_ok,
            self.stats_valid,
            self.reproducible,
            self.goal_aligned
        ])

    def submit(self):
        client = mlflow.MlflowClient()
        client.set_tag(self.run_id, "review.status",
            "approved" if self.approved else "rejected")
        client.set_tag(self.run_id, "review.reviewer", self.reviewer)
        client.set_tag(self.run_id, "review.notes", self.notes)

# Usage by reviewer
review = ExperimentReview(
    run_id="abc123",
    reviewer="bob",
    methodology_ok=True,
    stats_valid=True,
    reproducible=True,
    goal_aligned=False,
    notes="Good results but optimizes for precision; we need recall."
)
review.submit()

Review Questions — Methodology

  • Is the train/val/test split correct?
  • Is there data leakage?
  • Are baselines appropriate?
  • Is the evaluation metric aligned with the business goal?

Review Questions — Reproducibility

  • Can the run be reproduced from logged artifacts?
  • Are all seeds and versions recorded?
  • Is the config complete (no hardcoded values)?
  • Has statistical significance been established?

Documentation Patterns

Every significant experiment should have a companion document that explains the why, not just the what. Tracking servers capture metrics and parameters automatically, but they don't capture hypotheses, reasoning about negative results, or lessons learned. 

# Structured experiment documentation template
import mlflow
from datetime import datetime

def log_experiment_doc(run_id, doc):
    client = mlflow.MlflowClient()

    # Structured documentation as tags
    client.set_tag(run_id, "doc.hypothesis", doc["hypothesis"])
    client.set_tag(run_id, "doc.approach", doc["approach"])
    client.set_tag(run_id, "doc.result_summary", doc["result_summary"])
    client.set_tag(run_id, "doc.next_steps", doc["next_steps"])
    client.set_tag(run_id, "doc.lessons_learned", doc["lessons_learned"])
    client.set_tag(run_id, "doc.date", datetime.now().isoformat())

# Example documentation
log_experiment_doc("run_xyz", {
    "hypothesis": "Adding user session features will improve CTR prediction by 2%+",
    "approach": "Added 5 session-level features to transformer input; trained for 50 epochs",
    "result_summary": "CTR AUC improved 0.8% (p=0.02). Below 2% target.",
    "next_steps": "Try session features with LSTM encoder instead of transformer",
    "lessons_learned": "Session features help but transformer may not capture sequential patterns well"
})
Document negative results: An experiment that doesn't work is still valuable — it prevents teammates from trying the same approach. Log why it failed and what you learned. The most valuable documentation is often about what didn't work.

Team Best Practices

Beyond tools and processes, effective ML collaboration requires cultural practices that encourage transparency, knowledge sharing, and systematic experimentation.

Weekly Experiment Syncs

15-minute standup focused on experiment results. Each team member shares: what they tried, what they learned, and what they plan next. Use the tracking server dashboard as the shared screen.

Experiment Backlog

Maintain a prioritized list of experiments to run, like a product backlog. Each item has a hypothesis, expected impact, and estimated compute cost. Prevents ad-hoc experimentation.

Shared Baselines

Maintain canonical baseline runs that everyone compares against. Pin the baseline version and update it only through a formal review process. Never compare against a moving target.

Knowledge Repository

A living document (wiki, Notion, or Confluence) linking experiment run IDs to written analysis. Index by topic, model type, and failure mode. New team members start here.

# Shared baseline management
import mlflow

class BaselineManager:
    def __init__(self, project_name):
        self.client = mlflow.MlflowClient()
        self.project = project_name

    def get_current_baseline(self):
        # Find the run tagged as current baseline
        experiment = self.client.get_experiment_by_name(self.project)
        runs = self.client.search_runs(
            experiment_ids=[experiment.experiment_id],
            filter_string="tags.is_baseline = 'true'",
            max_results=1
        )
        if not runs:
            raise ValueError("No baseline set for this project")
        return runs[0]

    def compare_to_baseline(self, candidate_run_id, metrics):
        baseline = self.get_current_baseline()
        results = {}
        for m in metrics:
            baseline_val = baseline.data.metrics[m]
            candidate_val = self.client.get_run(candidate_run_id).data.metrics[m]
            results[m] = {
                "baseline": baseline_val,
                "candidate": candidate_val,
                "delta": candidate_val - baseline_val
            }
        return results
Anti-pattern: "Hero experiments" — one person runs a massive sweep overnight without telling the team, consumes all GPU resources, and presents results as a fait accompli. Instead, propose significant experiments in the backlog and get team buy-in before consuming shared resources.
Summary: Great ML collaboration combines shared infrastructure (centralized tracking), structured processes (naming, reviews, documentation), and cultural practices (syncs, baselines, backlogs). Start with infrastructure, then layer on process as the team grows.
