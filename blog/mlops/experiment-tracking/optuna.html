Optuna Overview

Optuna is a next-generation hyperparameter optimization (HPO) framework designed with a define-by-run API. Unlike older frameworks such as Hyperopt that require you to declare the entire search space up-front in a static configuration object, Optuna lets you define hyperparameters inside the objective function itself. This means your search space can be conditional — for example, you might only sample the number of layers in a neural network if the model type sampled earlier was deep_net.

At its core, Optuna revolves around three concepts: a Study, which represents the full optimization session; a Trial, which is a single evaluation of the objective function with a specific set of hyperparameters; and a Storage backend that persists all results so studies can be resumed, shared, and analysed later. The storage can be an in-memory Python object, an SQLite file, or a full relational database like MySQL or PostgreSQL for distributed workloads.

Here is a minimal Optuna workflow that optimizes a simple quadratic function: 

import optuna

def objective(trial):
    # Sample a float hyperparameter between -10 and 10
    x = trial.suggest_float("x", -10.0, 10.0)
    # The goal is to minimize (x - 3)^2
    return (x - 3) ** 2

# Create a study that minimizes the objective
study = optuna.create_study(direction="minimize")
study.optimize(objective, n_trials=100)

print("Best value:", study.best_value)
print("Best params:", study.best_params)

The suggest_* family of methods is what makes the define-by-run paradigm possible. Each call tells the sampler what kind of value is needed (float, int, or categorical) and the sampler returns the next candidate value. The sampler algorithm — whether it is random search, TPE, or CMA-ES — is fully pluggable and configured at the study level.

Key advantage: Because the search space is defined inside the objective function, you can use standard Python control flow (if/else, loops) to create conditional and hierarchical search spaces without any special DSL.

Samplers (TPE, CMA-ES, Grid)

Samplers are the algorithms Optuna uses to decide which hyperparameter values to try next. Choosing the right sampler can dramatically reduce the number of trials needed to find a strong configuration. Optuna ships with several built-in samplers, each suited to different problem structures.

Tree-structured Parzen Estimator (TPE)

TPE is Optuna's default sampler and the workhorse of Bayesian HPO. The algorithm works by maintaining two density estimators: one fitted to the hyperparameter values from trials that performed well (the "good" set, denoted l(x)), and one fitted to values from trials that performed poorly (the "bad" set, denoted g(x)). The split point between good and bad is controlled by a quantile parameter, typically γ = 0.25, meaning the top 25 % of trials are considered "good."

To pick the next candidate, TPE samples from l(x) and then scores the sample by the ratio l(x) / g(x). A high ratio means the candidate is likely in a promising region. This approach is efficient because it models the hyperparameters independently (with tree-structured dependencies for conditional spaces), making it scale well to moderate-dimensional search spaces (roughly 10–50 hyperparameters).

CMA-ES

The Covariance Matrix Adaptation Evolution Strategy (CMA-ES) is a derivative-free optimization algorithm originally designed for continuous domains. It maintains a multivariate Gaussian distribution over the search space and updates the mean, covariance matrix, and step size after each generation. CMA-ES excels when hyperparameters are correlated — for example, when learning rate and batch size jointly affect performance. However, it does not natively support categorical parameters and works best with fewer than ~20 continuous dimensions.

Grid Sampler

The Grid sampler exhaustively evaluates every point in a predefined grid. This is useful for small, discrete search spaces where you want complete coverage — for instance, comparing three optimizers with four learning rates (12 total combinations). It guarantees that no point is missed, but scales exponentially with the number of hyperparameters.

TPE

  • Default sampler — great general-purpose choice
  • Handles conditional & categorical parameters
  • Scales to 10–50 dimensions
  • Models parameters independently

CMA-ES

  • Best for continuous, correlated parameters
  • Learns parameter covariance structure
  • Optimal for < 20 continuous dimensions
  • No native categorical support

Grid Sampler

  • Exhaustive — guarantees full coverage
  • Ideal for small, discrete spaces
  • Scales exponentially — impractical for large spaces
  • No adaptive intelligence

Random Sampler

  • Simplest baseline — uniform random draws
  • Embarrassingly parallel
  • Good for warm-starting other samplers
  • No exploitation — purely exploratory

Here is how you configure each sampler when creating a study:

import optuna
from optuna.samplers import TPESampler, CmaEsSampler, GridSampler

# TPE with a custom seed for reproducibility
tpe_sampler = TPESampler(seed=42)
study_tpe = optuna.create_study(sampler=tpe_sampler)

# CMA-ES for a continuous search space
cma_sampler = CmaEsSampler(seed=42)
study_cma = optuna.create_study(sampler=cma_sampler)

# Grid search over explicit values
search_space = {
    "lr": [1e-4, 1e-3, 1e-2],
    "optimizer": ["adam", "sgd"],
}
grid_sampler = GridSampler(search_space)
study_grid = optuna.create_study(sampler=grid_sampler)

Pruners (MedianPruner, Hyperband)

Pruning is one of Optuna's most powerful features. Instead of running every trial to completion, a pruner monitors intermediate results (for example, validation loss at each epoch) and terminates unpromising trials early. This can save enormous amounts of compute — in practice, pruning often reduces total wall-clock time by 50–80 % with minimal impact on final result quality.

How Pruning Works

During training, the objective function reports intermediate values via trial.report(value, step). After each report the function calls trial.should_prune(), which queries the active pruner. If the pruner decides the trial is unlikely to beat the current best, it raises an optuna.TrialPruned exception, terminating the trial immediately. The pruned trial is still recorded in the study so the sampler can learn from it.

MedianPruner

The MedianPruner compares the current trial's intermediate value at step t against the median of intermediate values reported by all previous trials at the same step. If the trial is worse than the median, it gets pruned. You can control startup behavior with n_startup_trials (minimum completed trials before pruning begins) and n_warmup_steps (minimum reported steps before a trial can be pruned).

Hyperband Pruner

The Hyperband pruner implements the Successive Halving algorithm across multiple brackets. Each bracket allocates resources differently: one bracket starts many trials with a small budget and aggressively prunes; another starts fewer trials with a larger budget. By hedging across brackets, Hyperband is robust to the tradeoff between exploration and exploitation. It is particularly effective when the cost of a full trial is high and intermediate results are informative. 

import optuna

def objective(trial):
    # Suggest hyperparameters
    lr = trial.suggest_float("lr", 1e-5, 1e-1, log=True)
    n_layers = trial.suggest_int("n_layers", 1, 5)
    dropout = trial.suggest_float("dropout", 0.0, 0.5)

    model = build_model(lr, n_layers, dropout)

    for epoch in range(50):
        train_loss = train_one_epoch(model)
        val_loss = evaluate(model)

        # Report intermediate value for pruning
        trial.report(val_loss, epoch)

        # Check if the trial should be pruned
        if trial.should_prune():
            raise optuna.TrialPruned()

    return val_loss

# MedianPruner: prune below-median trials after 5 warmup steps
pruner = optuna.pruners.MedianPruner(
    n_startup_trials=5,
    n_warmup_steps=5,
)
study = optuna.create_study(direction="minimize", pruner=pruner)
study.optimize(objective, n_trials=100)

Switching to Hyperband requires only changing the pruner:

# Hyperband pruner with configurable reduction factor
hyperband = optuna.pruners.HyperbandPruner(
    min_resource=1,
    max_resource=50,
    reduction_factor=3,
)
study_hb = optuna.create_study(
    direction="minimize",
    pruner=hyperband,
)
study_hb.optimize(objective, n_trials=200)
Caution: Pruning only helps when intermediate metrics are correlated with the final metric. If your model's early-epoch validation loss is not predictive of its converged performance (common with very long warm-up schedules), aggressive pruning can discard trials that would have recovered. Use n_warmup_steps to delay pruning past the warm-up phase.

Multi-Objective Optimization

Real-world ML systems rarely optimize a single metric. You often need to balance accuracy against latency, model size against throughput, or precision against recall. Optuna supports multi-objective optimization natively: instead of a single direction, you pass a list of directions, and the objective function returns a tuple of values.

Pareto Fronts

In multi-objective optimization, there is generally no single "best" solution. Instead, the result is a Pareto front — the set of trials where no objective can be improved without worsening another. A trial on the Pareto front is called Pareto-optimal or non-dominated. Optuna uses the NSGA-II algorithm (Non-dominated Sorting Genetic Algorithm II) to explore the Pareto front efficiently. NSGA-II maintains population diversity through crowding distance, ensuring a well-spread set of trade-off solutions.

After optimization, you can retrieve the Pareto-optimal trials and inspect them to choose the configuration that best fits your deployment constraints: 

import optuna

def multi_objective(trial):
    # Hyperparameters
    n_estimators = trial.suggest_int("n_estimators", 50, 500)
    max_depth = trial.suggest_int("max_depth", 3, 15)
    lr = trial.suggest_float("lr", 0.01, 0.3, log=True)

    model = train_xgboost(n_estimators, max_depth, lr)
    accuracy = evaluate_accuracy(model)
    latency_ms = measure_latency(model)

    # Return both objectives as a tuple
    return accuracy, latency_ms

# Maximize accuracy, minimize latency
study = optuna.create_study(
    directions=["maximize", "minimize"],
)
study.optimize(multi_objective, n_trials=200)

# Retrieve the Pareto-optimal trials
pareto_trials = study.best_trials
for t in pareto_trials:
    print(f"Trial {t.number}: accuracy={t.values[0]:.4f}, latency={t.values[1]:.1f}ms")
    print(f"  Params: {t.params}")

A practical workflow is to plot the Pareto front, identify a "knee" region where marginal improvements in one objective cost disproportionately in the other, and select a configuration from that region. Optuna's built-in visualization module (optuna.visualization) can generate Pareto front plots, parallel coordinate plots, and contour plots to support this analysis.

Tip: When objectives are on very different scales (e.g., accuracy in [0, 1] vs. latency in [10, 500] ms), NSGA-II's crowding distance still works correctly because it normalizes within each objective. No manual scaling is needed.

Distributed Optimization

One of Optuna's strongest production features is its ability to run distributed optimization with zero code changes to the objective function. The mechanism is simple: multiple processes (on the same machine or across a cluster) share a relational database as the study storage backend. Each process creates or loads the same study by name, and Optuna's storage layer handles locking, deduplication, and result collection.

Because trials are fully independent, distributed Optuna is embarrassingly parallel. There is no central scheduler or coordinator process — each worker pulls the study state from the database, asks the sampler for the next trial, evaluates it, and writes the result back. The samplers are designed to be robust to the slight staleness that occurs when multiple workers read the study state at the same time. 

import optuna

# All workers point to the same database
storage = "mysql+pymysql://user:pass@db-host/optuna_db"

def objective(trial):
    lr = trial.suggest_float("lr", 1e-5, 1e-1, log=True)
    batch_size = trial.suggest_categorical("batch_size", [32, 64, 128, 256])
    weight_decay = trial.suggest_float("weight_decay", 1e-6, 1e-2, log=True)

    model = build_and_train(lr, batch_size, weight_decay)
    return evaluate(model)

# Create or load the study (idempotent across workers)
study = optuna.create_study(
    study_name="distributed-hpo-v1",
    storage=storage,
    load_if_exists=True,
    direction="minimize",
)

# Each worker runs its own share of trials
study.optimize(objective, n_trials=50)

To launch distributed optimization, you simply start the same script on multiple machines (or in multiple containers). Each instance will coordinate through the shared database. For example, running four workers with 50 trials each completes 200 trials in roughly one-quarter the wall-clock time of a single worker.

Supported storage backends include SQLite (for single-machine parallelism via file locking), MySQL, and PostgreSQL (for true multi-node distributed HPO). For large-scale runs, MySQL or PostgreSQL is recommended because SQLite's file-level locking can become a bottleneck with many concurrent writers.

Integration Examples

Optuna provides first-class integration callbacks for popular ML frameworks. These callbacks handle intermediate value reporting and pruning automatically, so you don't have to manually call trial.report and trial.should_prune inside your training loop.

PyTorch Lightning

The PyTorchLightningPruningCallback plugs directly into Lightning's callback system. It monitors a specified metric and prunes the trial if the pruner decides it is unpromising: 

import optuna
from optuna.integration import PyTorchLightningPruningCallback
import pytorch_lightning as pl

def objective(trial):
    lr = trial.suggest_float("lr", 1e-5, 1e-2, log=True)
    hidden_dim = trial.suggest_int("hidden_dim", 32, 256)

    model = LitModel(lr=lr, hidden_dim=hidden_dim)
    datamodule = MyDataModule()

    # Pruning callback monitors val_loss
    pruning_cb = PyTorchLightningPruningCallback(trial, monitor="val_loss")

    trainer = pl.Trainer(
        max_epochs=30,
        callbacks=[pruning_cb],
        enable_progress_bar=False,
    )
    trainer.fit(model, datamodule=datamodule)

    return trainer.callback_metrics["val_loss"].item()

study = optuna.create_study(
    direction="minimize",
    pruner=optuna.pruners.MedianPruner(),
)
study.optimize(objective, n_trials=50)

XGBoost

For XGBoost, the XGBoostPruningCallback integrates with XGBoost's callback API to prune during boosting rounds: 

import optuna
from optuna.integration import XGBoostPruningCallback
import xgboost as xgb

def objective(trial):
    params = {
        "objective": "binary:logistic",
        "eval_metric": "logloss",
        "max_depth": trial.suggest_int("max_depth", 3, 12),
        "learning_rate": trial.suggest_float("learning_rate", 0.01, 0.3, log=True),
        "subsample": trial.suggest_float("subsample", 0.5, 1.0),
        "colsample_bytree": trial.suggest_float("colsample_bytree", 0.5, 1.0),
        "min_child_weight": trial.suggest_int("min_child_weight", 1, 10),
    }

    # Pruning callback monitors validation logloss
    pruning_cb = XGBoostPruningCallback(trial, "validation-logloss")

    dtrain = xgb.DMatrix(X_train, label=y_train)
    dval = xgb.DMatrix(X_val, label=y_val)

    bst = xgb.train(
        params,
        dtrain,
        num_boost_round=300,
        evals=[(dval, "validation")],
        callbacks=[pruning_cb],
        verbose_eval=False,
    )

    preds = bst.predict(dval)
    return compute_logloss(y_val, preds)

study = optuna.create_study(
    direction="minimize",
    pruner=optuna.pruners.HyperbandPruner(),
)
study.optimize(objective, n_trials=100)

Scikit-learn

For scikit-learn workflows, Optuna provides OptunaSearchCV, a drop-in replacement for GridSearchCV or RandomizedSearchCV that uses Optuna's samplers and pruners under the hood: 

import optuna
from optuna.integration import OptunaSearchCV
from sklearn.ensemble import RandomForestClassifier
import optuna.distributions as dist

# Define parameter distributions
param_distributions = {
    "n_estimators": dist.IntDistribution(50, 500),
    "max_depth": dist.IntDistribution(3, 20),
    "min_samples_split": dist.IntDistribution(2, 20),
    "min_samples_leaf": dist.IntDistribution(1, 10),
}

clf = OptunaSearchCV(
    estimator=RandomForestClassifier(random_state=42),
    param_distributions=param_distributions,
    n_trials=60,
    cv=5,
    scoring="f1_weighted",
    random_state=42,
)

# Fits like any sklearn estimator
clf.fit(X_train, y_train)
print("Best score:", clf.best_score_)
print("Best params:", clf.best_params_)
Summary: Optuna's define-by-run API, pluggable samplers and pruners, native multi-objective support, and zero-change distributed execution make it one of the most versatile HPO frameworks available today. Start with TPE + MedianPruner for most workloads, switch to CMA-ES for continuous correlated spaces, and scale horizontally by pointing multiple workers at the same database.
