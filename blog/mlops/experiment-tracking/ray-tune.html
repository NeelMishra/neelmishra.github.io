Ray Tune: Distributed HPO

MLOps Series Experiment Tracking

Ray Tune Overview

Ray Tune is a scalable hyperparameter tuning library built on top of the Ray distributed computing framework. Unlike single-machine tools such as Optuna or Hyperopt that hit resource ceilings on large search spaces, Ray Tune distributes trial execution across a cluster of machines transparently. Each trial runs as an isolated Ray actor, and a central driver coordinates scheduling decisions, checkpointing, and result aggregation.

The library offers a unified API that wraps multiple search and scheduling back-ends. You can combine any search algorithm (Bayesian optimisation, random search, evolutionary strategies) with any early-stopping scheduler (ASHA, PBT, HyperBand) in a single tune.run call. Results are streamed to an experiment directory where they can be picked up by MLflow, Weights & Biases, or TensorBoard via built-in callbacks.

The diagram above illustrates the core topology. The Head Node runs the Tune driver process plus the Global Control Store (GCS). Scheduling and search are pluggable components orchestrated by the driver. Each worker node hosts one or more trial actors that report metrics back to the driver after every training iteration. Checkpoints are persisted to a shared store (cloud object storage or NFS) so that trials can be paused, resumed, or migrated across nodes. 

# Minimal Ray Tune example
import ray
from ray import tune

def train_fn(config):
    for epoch in range(config["epochs"]):
        loss = train_one_epoch(lr=config["lr"], bs=config["batch_size"])
        tune.report(loss=loss, epoch=epoch)

ray.init(address="auto")  # connect to existing cluster

analysis = tune.run(
    train_fn,
    config={
        "lr": tune.loguniform(1e-5, 1e-1),
        "batch_size": tune.choice([16, 32, 64, 128]),
        "epochs": 50,
    },
    num_samples=100,
    resources_per_trial={"cpu": 4, "gpu": 1},
)

best_config = analysis.get_best_config(metric="loss", mode="min")
print("Best config:", best_config)
Why Ray Tune over manual loops? A hand-written grid search across 4 hyperparameters with 5 values each spawns 625 trials. Without early stopping, every trial runs to completion — wasting days of GPU time. Ray Tune's schedulers terminate poor trials after just a few epochs, often cutting wall-clock time by 5–10×.

Schedulers (ASHA, PBT, HyperBand)

Schedulers govern which trials continue, which are paused, and which are killed. The right scheduler depends on your resource budget, the cost of each training step, and whether you want the search to adapt hyperparameters mid-run (population-based training) or simply prune bad trials early (successive halving).

ASHA — Asynchronous Successive Halving

The Asynchronous Successive Halving Algorithm (ASHA) is the default scheduler for most Ray Tune workloads. ASHA organises trials into rungs. At each rung boundary (e.g. epoch 1, 3, 9, 27 …) it compares a trial's intermediate metric against all other trials that have reached the same rung. If the trial falls in the bottom 1 − 1/η fraction (η is the reduction factor, typically 3 or 4), it is terminated immediately. The top 1/η fraction is promoted to the next rung.

Unlike synchronous HyperBand, ASHA does not wait for all trials at a rung to finish before making decisions. This asynchronous behaviour avoids stragglers — a single slow trial on a busy GPU won't block the entire sweep. 

from ray.tune.schedulers import ASHAScheduler

scheduler = ASHAScheduler(
    metric="val_loss",
    mode="min",
    max_t=100,          # max epochs per trial
    grace_period=5,     # minimum epochs before pruning
    reduction_factor=3, # keep top 1/3 at each rung
)

analysis = tune.run(
    train_fn,
    scheduler=scheduler,
    num_samples=64,
    config=search_space,
)

Population Based Training (PBT)

PBT goes beyond early stopping — it mutates hyperparameters during training. The algorithm maintains a population of trials running in parallel. At regular intervals (the perturbation interval) each trial checks whether it is in the bottom quantile. If so, it copies the weights and hyperparameters of a top-performing trial, then perturbs (randomly scales by 0.8× or 1.2×) one or more hyperparameters before resuming training. This creates an adaptive schedule that discovers learning-rate decays, augmentation ramps, and other non-stationary schedules automatically. 

from ray.tune.schedulers import PopulationBasedTraining

pbt = PopulationBasedTraining(
    time_attr="training_iteration",
    perturbation_interval=10,
    hyperparam_mutations={
        "lr": tune.loguniform(1e-5, 1e-2),
        "weight_decay": tune.uniform(0.0, 0.3),
        "momentum": [0.8, 0.9, 0.95, 0.99],
    },
)

analysis = tune.run(
    train_fn,
    scheduler=pbt,
    num_samples=16,  # population size
    config={"lr": 1e-3, "weight_decay": 0.01, "momentum": 0.9},
)
PBT requires checkpointing. Because PBT clones model weights from top performers, your training function must save and restore checkpoints. Without this, PBT degrades to simple restart-based mutation — losing all training progress on perturbation.

Scheduler Comparison

ASHA

  • Strategy: Early termination of poor trials
  • Async: Yes — no straggler blocking
  • Checkpointing: Optional (for pause/resume)
  • Best for: Large search spaces, quick pruning
  • Overhead: Very low; comparison at rung boundaries

PBT

  • Strategy: Clone top performers, mutate hyperparams
  • Async: Semi — perturbation interval is synchronous
  • Checkpointing: Required (weight cloning)
  • Best for: Finding learning rate schedules, long runs
  • Overhead: Moderate; weight serialisation at each interval

HyperBand

  • Strategy: Multi-bracket successive halving
  • Async: No — waits for bracket to complete
  • Checkpointing: Optional
  • Best for: Moderate search spaces with known budgets
  • Overhead: Low, but synchronous barriers add idle time

FIFO (default)

  • Strategy: No early stopping; all trials run to max_t
  • Async: N/A
  • Checkpointing: Not needed
  • Best for: Small grids, cheap trials
  • Overhead: None — but wastes compute on bad trials

Search Algorithms

While schedulers decide when to stop trials, search algorithms decide which hyperparameter configurations to try next. Ray Tune integrates with many back-ends through a thin adapter layer, letting you swap search strategies with a single argument change.

Bayesian Optimisation (BayesOpt / Optuna)

Bayesian optimisation builds a probabilistic surrogate model (typically a Gaussian process or Tree-structured Parzen Estimator) of the objective function. It uses an acquisition function — Expected Improvement (EI), Upper Confidence Bound (UCB), or similar — to choose the next point that balances exploration and exploitation. In Ray Tune, this is surfaced via the OptunaSearch or BayesOptSearch adapters. 

from ray.tune.search.optuna import OptunaSearch

search_algo = OptunaSearch(
    metric="val_loss",
    mode="min",
)

analysis = tune.run(
    train_fn,
    search_alg=search_algo,
    scheduler=scheduler,  # combine with ASHA
    num_samples=80,
    config={
        "lr": tune.loguniform(1e-5, 1e-1),
        "hidden_dim": tune.choice([64, 128, 256, 512]),
        "dropout": tune.uniform(0.0, 0.5),
    },
)

HyperOpt (TPE)

The HyperOpt search adapter uses the Tree-structured Parzen Estimator (TPE) algorithm. TPE models P(x | y < y*) and P(x | y ≥ y*) separately using kernel density estimators and chooses the next point that maximises their ratio. It is particularly effective for mixed (continuous + categorical) spaces. 

from ray.tune.search.hyperopt import HyperOptSearch

search_algo = HyperOptSearch(
    metric="val_loss",
    mode="min",
    n_initial_points=10,  # random warmup before TPE kicks in
)

analysis = tune.run(
    train_fn,
    search_alg=search_algo,
    num_samples=60,
    config=search_space,
)

Random and Grid Search

For baselines or very cheap trials, random search is surprisingly competitive — Bergstra & Bengio (2012) showed it outperforms grid search in most practical settings because it samples more unique values along each dimension. Ray Tune's default (no search_alg argument) is random search over the config space.

Distributed Training Integration

Ray Tune is not limited to distributing trials. Each individual trial can itself be a distributed training job spanning multiple GPUs or nodes. This is accomplished by pairing Tune with Ray Train (formerly Ray SGD), which wraps PyTorch DDP, Horovod, or TensorFlow MirroredStrategy. 

from ray import tune
from ray.train.torch import TorchTrainer
from ray.train import ScalingConfig

def train_loop_per_worker(config):
    import torch
    model = build_model(config["hidden_dim"])
    model = ray.train.torch.prepare_model(model)
    optimizer = torch.optim.Adam(model.parameters(), lr=config["lr"])

    for epoch in range(config["epochs"]):
        loss = train_epoch(model, optimizer)
        ray.train.report({"loss": loss})

trainer = TorchTrainer(
    train_loop_per_worker,
    scaling_config=ScalingConfig(
        num_workers=4,       # 4 GPU workers per trial
        use_gpu=True,
    ),
)

tuner = tune.Tuner(
    trainer,
    param_space={
        "train_loop_config": {
            "lr": tune.loguniform(1e-5, 1e-2),
            "hidden_dim": tune.choice([256, 512, 1024]),
            "epochs": 30,
        }
    },
    tune_config=tune.TuneConfig(
        num_samples=20,
        scheduler=ASHAScheduler(metric="loss", mode="min"),
    ),
)

results = tuner.fit()

In this pattern, Tune spawns 20 trials. Each trial provisions 4 GPU workers that run PyTorch DDP internally. If the cluster has 8 GPUs, two trials execute concurrently while the rest queue. The ASHA scheduler prunes poorly performing trials, freeing resources for the next trial in the queue. This resource-aware scheduling is a key differentiator — it avoids GPU over-subscription and deadlocks that manual scripting often introduces.

Fractional GPUs. Ray Tune supports fractional GPU allocation (resources_per_trial={"gpu": 0.5}). Two trials will then share a single GPU via CUDA MPS. This is useful for small models where a single trial does not saturate GPU compute.

Callbacks and Checkpoints

Callbacks let you hook into the trial lifecycle — logging metrics to external systems, sending Slack notifications, or syncing artefacts to cloud storage. Ray Tune ships built-in callbacks for MLflow, Weights & Biases, TensorBoard, and CSV logging. You can also write custom callbacks by subclassing tune.Callback. 

from ray.tune.integration.mlflow import MLflowLoggerCallback
from ray import tune

analysis = tune.run(
    train_fn,
    config=search_space,
    callbacks=[
        MLflowLoggerCallback(
            tracking_uri="http://mlflow.internal:5000",
            experiment_name="resnet50_hpo",
            save_artifact=True,
        ),
    ],
)

Checkpoint Mechanics

Checkpointing is the backbone of fault tolerance in Ray Tune. When a trial saves a checkpoint, Tune persists it to the configured storage back-end (local disk, NFS, S3, or GCS). If a node fails, the trial is automatically rescheduled on a healthy node and resumes from the latest checkpoint. PBT relies on checkpoints to clone model weights from top performers. 

from ray import tune
from ray.train import Checkpoint
import tempfile, os

def train_fn_with_ckpt(config):
    model = build_model(config)
    start_epoch = 0

    # Restore from checkpoint if available
    checkpoint = tune.get_checkpoint()
    if checkpoint:
        with checkpoint.as_directory() as ckpt_dir:
            state = torch.load(os.path.join(ckpt_dir, "model.pt"))
            model.load_state_dict(state["model"])
            start_epoch = state["epoch"] + 1

    for epoch in range(start_epoch, config["epochs"]):
        loss = train_one_epoch(model, config)

        # Save checkpoint every 5 epochs
        if epoch % 5 == 0:
            with tempfile.TemporaryDirectory() as tmp:
                path = os.path.join(tmp, "model.pt")
                torch.save({"model": model.state_dict(), "epoch": epoch}, path)
                tune.report(loss=loss, checkpoint=Checkpoint.from_directory(tmp))
            continue
        tune.report(loss=loss)

Production Patterns

Moving from a notebook prototype to a production HPO pipeline involves several considerations: cluster management, reproducibility, cost control, and integration with CI/CD. Below are battle-tested patterns used in production Ray Tune deployments.

1. Pin the Search Space in Version Control

Define search spaces as Python dictionaries in a dedicated config module. This ensures every team member and CI run uses the same parameter ranges. Include the Ray Tune version and any custom scheduler configs in the same file. 

# config/hpo_config.py
from ray import tune

SEARCH_SPACE = {
    "lr": tune.loguniform(1e-5, 5e-2),
    "weight_decay": tune.loguniform(1e-6, 1e-2),
    "batch_size": tune.choice([32, 64, 128]),
    "hidden_layers": tune.randint(2, 6),
    "activation": tune.choice(["relu", "gelu", "silu"]),
}

SCHEDULER_KWARGS = {
    "metric": "val_loss",
    "mode": "min",
    "max_t": 100,
    "grace_period": 10,
    "reduction_factor": 3,
}

2. Set Max Concurrent Trials

Without concurrency limits, Tune can starve the cluster if hundreds of trials try to acquire GPUs simultaneously. Use max_concurrent_trials in TuneConfig or wrap search algorithms with ConcurrencyLimiter. 

from ray.tune.search import ConcurrencyLimiter

limited_search = ConcurrencyLimiter(
    OptunaSearch(metric="val_loss", mode="min"),
    max_concurrent=8,  # never more than 8 trials at once
)

3. Failure Recovery with Resume

Production sweeps can span hours. If the driver process crashes — OOM, spot-instance preemption, or a bug — you lose all progress unless you set resume="AUTO". This tells Tune to pick up from the experiment directory, reloading completed results and restarting only the trials that were in-flight. 

analysis = tune.run(
    train_fn,
    name="resnet_hpo_v2",
    local_dir="/mnt/shared/ray_results",
    resume="AUTO",            # pick up where we left off
    config=search_space,
    scheduler=scheduler,
    num_samples=200,
    max_failures=3,           # retry each trial up to 3 times
    raise_on_failed_trial=False,
)

4. Cost Control on Cloud Clusters

When running on AWS, GCP, or Azure via the Ray cluster launcher, use time_budget_s to set a wall-clock limit and stop criteria to halt trials that plateau. Combine this with spot/preemptible instances for 60–80% cost savings. Ray automatically handles spot interruptions through its node-fault-tolerance layer; trials resume on replacement nodes from their last checkpoint.

Tip: Pair max_concurrent_trials with a cloud autoscaler min/max node range. This prevents runaway cluster scaling while still allowing burst capacity when the scheduler promotes many trials past a rung boundary.
