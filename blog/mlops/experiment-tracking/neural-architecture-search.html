NAS Fundamentals

Neural Architecture Search (NAS) automates the design of neural network architectures. Instead of manually crafting layer configurations, NAS algorithms explore a defined search space to discover architectures that optimize one or more objectives — accuracy, latency, model size, or inference cost.

The core NAS loop consists of three components: a search space that defines the set of possible architectures, a search strategy that decides which architectures to evaluate next, and a performance estimation strategy that approximates how well a candidate performs without full training.

Key insight: NAS treats architecture design as a hyperparameter optimization problem. Every layer choice, skip connection, and activation function becomes a searchable parameter in a combinatorial space.

Search Spaces and Strategies

The search space defines what NAS can discover. A macro search space specifies the entire network topology — the number of layers, layer types, and how they connect. A cell-based search space searches for a repeatable cell (a small computation graph) that is stacked to form the full network.

Cell-based search dramatically reduces the search space. Instead of exploring every possible full-network configuration, the algorithm discovers a normal cell and a reduction cell that are tiled together. 

# Defining a cell-based search space
from nni.nas.nn.pytorch import ModelSpace, MutableConv2d, MutableLinear

class SearchableCell(ModelSpace):
    def __init__(self, in_channels, out_channels):
        super().__init__()
        # Each operation is a searchable choice
        self.op1 = MutableConv2d(in_channels, out_channels, kernel_size=choice([3, 5, 7]))
        self.op2 = MutableConv2d(out_channels, out_channels, kernel_size=choice([3, 5]))
        self.skip = choice([True, False])

    def forward(self, x):
        out = self.op1(x)
        out = self.op2(out)
        if self.skip:
            out = out + x
        return out

Common search strategies include reinforcement learning (an RNN controller proposes architectures and is rewarded by validation accuracy), evolutionary algorithms (architectures mutate and the fittest survive), and differentiable methods (DARTS) that relax discrete choices into continuous weights learned via gradient descent.

Reinforcement Learning

Controller RNN samples architectures. High GPU cost but strong results. Original NASNet approach used 500 GPUs for multiple days.

Evolutionary Algorithms

Population-based mutations and crossover. AmoebaNet matched RL-based NAS at lower cost. Naturally parallelizable across workers.

Differentiable (DARTS)

Relaxes discrete choices to continuous. Single GPU friendly. Risk of performance collapse — the search can converge on degenerate architectures.

Bayesian Optimization

Surrogate model guides search. BANANAS uses neural predictors. Sample-efficient but scales poorly to very large spaces.

One-Shot NAS (Weight Sharing)

One-shot NAS trains a single supernet — an over-parameterized network that contains every candidate architecture as a subgraph. Individual architectures are evaluated by inheriting weights from the supernet rather than training from scratch.

This reduces search cost from thousands of GPU-hours to single-digit GPU-hours. The key idea is weight sharing: sub-networks share the supernet's parameters, so evaluating a candidate only requires a forward pass through the relevant subgraph. 

# One-shot supernet training loop
import torch
from nni.nas.strategy import ENAS

# Build supernet from search space
supernet = SearchableCell(in_channels=64, out_channels=128)
optimizer = torch.optim.Adam(supernet.parameters(), lr=0.001)

for epoch in range(num_epochs):
    for batch in train_loader:
        # Sample a random sub-architecture
        arch = supernet.random_sample()

        # Forward pass through sampled sub-network
        logits = arch(batch.x)
        loss = cross_entropy(logits, batch.y)

        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

    # Evaluate top-k architectures on validation set
    top_archs = supernet.top_k(k=5, metric=validate)
Caution: Weight sharing introduces ranking noise. An architecture that performs well with inherited weights may not maintain its ranking after standalone training. Always retrain top candidates from scratch before final selection.

NAS Frameworks (NNI, AutoGluon)

Several frameworks simplify NAS in practice. NNI (Neural Network Intelligence) by Microsoft provides modular NAS with PyTorch integration, supporting ENAS, DARTS, ProxylessNAS, and custom strategies. AutoGluon by AWS wraps NAS into high-level APIs that handle search, training, and deployment. 

# NNI-based NAS experiment
from nni.experiment import NasExperiment
from nni.nas.evaluator.pytorch import Classification
from nni.nas.strategy import DARTS

model_space = SearchableCell(in_channels=3, out_channels=64)

evaluator = Classification(
    train_dataloaders=train_loader,
    val_dataloaders=val_loader,
    max_epochs=50,
    gpus=1
)

strategy = DARTS()

experiment = NasExperiment(model_space, evaluator, strategy)
experiment.run()

# Export discovered architecture
best_arch = experiment.export_top_models(top_k=1)
print(best_arch)

Ops Implications

NAS introduces unique infrastructure requirements. Search runs demand elastic GPU clusters — hundreds of trials running in parallel, each requiring different amounts of memory and compute. The ops team must handle dynamic resource allocation, trial scheduling, fault tolerance, and result aggregation.

Infrastructure checklist for NAS: Elastic GPU pool with auto-scaling, centralized experiment tracking for all trials, artifact storage for supernet checkpoints, and a scheduling system that can preempt low-priority trials.

Discovered architectures may have irregular computation graphs that don't map cleanly to standard serving frameworks. Ops teams should validate that the final architecture can be exported to ONNX or TorchScript and meets latency budgets before committing to production deployment. 

# Export discovered architecture for deployment
import torch

best_model = experiment.export_top_models(top_k=1)[0]
best_model.eval()

# Validate latency budget
dummy_input = torch.randn(1, 3, 224, 224)

with torch.no_grad():
    start = torch.cuda.Event(enable_timing=True)
    end = torch.cuda.Event(enable_timing=True)
    start.record()
    output = best_model(dummy_input)
    end.record()
    torch.cuda.synchronize()
    latency_ms = start.elapsed_time(end)

print(f"Inference latency: {latency_ms:.2f}ms")
assert latency_ms < 50, "Exceeds latency budget"

# Export to ONNX
torch.onnx.export(best_model, dummy_input, "nas_model.onnx", opset_version=13)

Practical Considerations

Before adopting NAS, consider whether the complexity is warranted. For standard tasks (image classification, text classification), well-tuned baseline architectures often match or exceed NAS-discovered models when properly trained.

NAS shines when you need hardware-aware architectures — models optimized for specific edge devices, latency constraints, or memory budgets. Multi-objective NAS can simultaneously optimize accuracy and inference cost, producing a Pareto frontier of deployment-ready models.

✔ Use NAS When

  • Targeting specific hardware constraints
  • Exploring novel architecture families
  • Multi-objective optimization is needed
  • Team has GPU budget for search

✘ Skip NAS When

  • Standard architectures work well enough
  • Limited compute budget
  • Deployment target is flexible
  • Hyperparameter tuning suffices
Tip: Start with one-shot NAS methods (ENAS, FairNAS) for 10-100x speedup over standard NAS. Graduate to full search only if the one-shot results show promise and you have the compute budget.
