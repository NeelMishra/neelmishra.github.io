Hyperparameter Tuning: Grid, Random, Bayesian

MLOps Series Experiment Tracking

Why Hyperparameter Optimization Matters

Every machine-learning model carries two kinds of knobs. Parameters are learned during training — weights in a neural network, coefficients in a regression. Hyperparameters are set before training begins: the learning rate, the number of trees in an ensemble, the regularisation strength, or the architecture depth. Getting these right is the difference between a model that barely beats a baseline and one that reaches production-quality accuracy.

Manual tuning is where most practitioners start — adjust one knob, run an experiment, check the metric, repeat. It works for toy problems, but it does not scale. With five hyperparameters and even a modest ten candidate values each, you face 105 = 100,000 possible configurations. Evaluating all of them is infeasible when a single training run takes minutes or hours.

Hyperparameter Optimization (HPO) addresses this by replacing gut instinct with a principled search strategy. The three most widely used strategies — grid search, random search, and Bayesian optimisation — occupy different points on the effort-vs-efficiency trade-off curve. Understanding when to reach for each one is a core MLOps competency.

Key insight: Hyperparameter choices interact in non-obvious ways. A learning rate that works well with a batch size of 32 may diverge with a batch size of 256. HPO methods explore these interactions systematically so you don't have to guess.

Grid Search

Grid search is the simplest and most transparent HPO strategy. You define a discrete set of values for each hyperparameter, and the algorithm evaluates every combination in the resulting Cartesian product. If you specify three learning rates, four regularisation strengths, and two batch sizes, you get 3 × 4 × 2 = 24 trials.

The primary advantage is completeness — no corner of the defined space is left unexplored. The primary drawback is cost: the number of trials grows multiplicatively with each new hyperparameter, a phenomenon known as the curse of dimensionality. In practice, grid search is most effective when you have a small number of hyperparameters (one to three) and a rough idea of their plausible ranges.

How It Works

For each candidate configuration in the grid, the model is trained on the training set, and performance is measured via cross-validation. The configuration that achieves the best validation score is selected. Because each trial is independent, grid search parallelises naturally — you can distribute trials across multiple machines or GPUs.

Python Implementation

# Grid search with scikit-learn
from sklearn.model_selection import GridSearchCV
from sklearn.ensemble import RandomForestClassifier

# Define the hyperparameter grid
param_grid = {
    "n_estimators": [50, 100, 200, 400],
    "max_depth": [4, 8, 16, None],
    "min_samples_split": [2, 5, 10],
}

model = RandomForestClassifier(random_state=42)
search = GridSearchCV(
    estimator=model,
    param_grid=param_grid,
    scoring="f1_weighted",
    cv=5,
    n_jobs=-1,        # use all available cores
    verbose=1,
)
search.fit(X_train, y_train)

print("Best params:", search.best_params_)
print("Best CV F1 :", round(search.best_score_, 4))

In the snippet above, GridSearchCV iterates over all 4 × 4 × 3 = 48 combinations, performing 5-fold cross-validation for each — 240 total fits. Setting n_jobs=-1 distributes these fits across every CPU core, which can cut wall-clock time by an order of magnitude on modern hardware.

Watch out: Grid search time grows exponentially with dimensions. Adding a fourth hyperparameter with five values turns 48 trials into 240. Always profile a single training run first and multiply by the grid size before committing compute.

Random Search

Random search, popularised by Bergstra & Bengio (2012), replaces the exhaustive grid with a fixed budget of randomly sampled configurations. Instead of testing every combination, you draw each hyperparameter independently from a specified distribution — uniform, log-uniform, categorical — and evaluate a predetermined number of trials.

The key theoretical insight is that most hyperparameters are not equally important. If only one out of five hyperparameters truly matters, grid search wastes most of its budget exploring the irrelevant ones at every grid point, while random search distributes samples more evenly across the important dimension. With the same budget, random search discovers better configurations with high probability.

How It Works

You specify a probability distribution for each hyperparameter and a total trial budget n. The algorithm samples n configurations, evaluates each one, and returns the best. Like grid search, each trial is independent and easy to parallelise.

Python Implementation

# Random search with scikit-learn
from sklearn.model_selection import RandomizedSearchCV
from scipy.stats import randint, uniform

# Define distributions instead of fixed lists
param_distributions = {
    "n_estimators": randint(50, 500),
    "max_depth": randint(2, 32),
    "min_samples_split": randint(2, 20),
    "min_samples_leaf": randint(1, 10),
    "max_features": uniform(0.1, 0.9),
}

model = RandomForestClassifier(random_state=42)
search = RandomizedSearchCV(
    estimator=model,
    param_distributions=param_distributions,
    n_iter=100,           # fixed budget of 100 trials
    scoring="f1_weighted",
    cv=5,
    n_jobs=-1,
    random_state=42,
    verbose=1,
)
search.fit(X_train, y_train)

print("Best params:", search.best_params_)
print("Best CV F1 :", round(search.best_score_, 4))

Notice that we now search over five hyperparameters with continuous ranges, yet we limit computation to exactly 100 trials. This is far more tractable than a 5-dimensional grid and often yields comparable or better results.

Bayesian Optimization

Bayesian optimisation (BO) takes a fundamentally different approach: rather than treating each trial as independent, it builds a surrogate model of the objective function and uses it to decide which configuration to try next. This sequential, model-guided strategy is especially valuable when training is expensive — each new trial is chosen to maximise the expected improvement over the current best result.

Core Components

A BO loop has two building blocks:

  • Surrogate model — typically a Gaussian Process (GP) or a Tree-structured Parzen Estimator (TPE). It approximates the mapping from hyperparameters to validation score, including uncertainty estimates.
  • Acquisition function — a cheap-to-evaluate function that balances exploitation (sampling near the current best) and exploration (sampling in high-uncertainty regions). Popular choices include Expected Improvement (EI), Upper Confidence Bound (UCB), and Probability of Improvement (PI).

The Optimisation Loop

At each iteration the surrogate is fitted to all observations so far, the acquisition function is maximised to propose a new candidate, the candidate is evaluated (expensive step), and the result is added to the observation set. Over time the surrogate becomes increasingly accurate in the regions that matter, and the search converges much faster than random sampling.

Python Implementation with Optuna

# Bayesian optimization with Optuna (TPE sampler)
import optuna
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import cross_val_score

def objective(trial):
    # Optuna suggests values from the search space
    n_estimators  = trial.suggest_int("n_estimators", 50, 500)
    learning_rate = trial.suggest_float("learning_rate", 1e-3, 1.0, log=True)
    max_depth     = trial.suggest_int("max_depth", 2, 16)
    subsample     = trial.suggest_float("subsample", 0.5, 1.0)
    min_samples_split = trial.suggest_int("min_samples_split", 2, 20)

    model = GradientBoostingClassifier(
        n_estimators=n_estimators,
        learning_rate=learning_rate,
        max_depth=max_depth,
        subsample=subsample,
        min_samples_split=min_samples_split,
        random_state=42,
    )

    score = cross_val_score(model, X_train, y_train, cv=5, scoring="f1_weighted")
    return score.mean()

# Create a study that maximises the objective
study = optuna.create_study(direction="maximize")
study.optimize(objective, n_trials=60, show_progress_bar=True)

print("Best trial value:", round(study.best_trial.value, 4))
print("Best params     :", study.best_trial.params)

Optuna's default sampler is the Tree-structured Parzen Estimator (TPE), which models p(x | y < y*) and p(x | y ≥ y*) separately to guide the search toward promising regions. In benchmarks, TPE-based BO typically finds configurations as good as or better than random search using 2–5× fewer trials.

When to use BO: Bayesian optimisation shines when each evaluation is expensive (minutes to hours). For cheap models — e.g., logistic regression on a small dataset — the overhead of fitting the surrogate can outweigh the savings. In that case, random or grid search is simpler and often sufficient.

Comparing Strategies

The table below summarises the trade-offs between the three main HPO strategies. The right choice depends on your compute budget, the number of hyperparameters, and how expensive a single training run is.

Grid Search

  • Coverage: Exhaustive within the defined grid
  • Scalability: Poor — exponential in number of hyperparameters
  • Parallelism: Trivial — all trials independent
  • Best for: ≤ 3 hyperparameters with known ranges
  • Weakness: Wastes budget on unimportant dimensions

Random Search

  • Coverage: Probabilistic — denser marginal coverage
  • Scalability: Good — budget is fixed regardless of dimensions
  • Parallelism: Trivial — all trials independent
  • Best for: Medium-dimensional spaces, quick baselines
  • Weakness: No learning between trials — purely stochastic

Bayesian Optimization

  • Coverage: Guided — focuses on promising regions
  • Scalability: Excellent sample efficiency, moderate overhead
  • Parallelism: Sequential by nature; batch variants exist
  • Best for: Expensive evaluations, 5–20 hyperparameters
  • Weakness: Surrogate fitting adds latency per trial

Quick Decision Guide

  • Training takes < 1 min → Grid or Random search
  • Training takes 1–30 min → Random search or BO
  • Training takes > 30 min → Bayesian optimization
  • Only 1–2 hyperparameters → Grid search is fine
  • Need a quick baseline → Random search with 50 trials

Practical Tips

Regardless of which strategy you choose, the following practices will help you get more out of your hyperparameter search and avoid common pitfalls.

1. Start with a Coarse Search, Then Refine

Run a broad random search first to identify the region of the space that performs well. Then narrow the ranges and either run a fine-grained grid search or switch to Bayesian optimisation within that reduced space. This two-stage approach avoids wasting budget on clearly suboptimal regions.

2. Use Log-Uniform Scales for Rates and Regularisation

Learning rates, weight decay, and regularisation strengths often span several orders of magnitude (e.g., 1e-5 to 1e-1). Searching on a linear scale wastes most of the budget in the upper range. Always use a log-uniform distribution so that each order of magnitude receives equal representation.

3. Track Every Trial

Log the full hyperparameter configuration and resulting metrics for every trial — not just the best one. Tools like MLflow, Weights & Biases, and Optuna's built-in dashboard make it easy to visualise parameter importance, detect interactions, and diagnose overfitting. This metadata is also essential for reproducibility.

4. Use Early Stopping to Save Compute

If a trial is clearly underperforming after a few epochs, terminate it early and redirect the budget. Optuna supports pruning via median stopping or successive-halving (Hyperband). In scikit-learn pipelines, you can implement early stopping at the model level for gradient-boosted trees and neural networks.

5. Be Aware of Overfitting the Validation Set

Running hundreds of trials optimises against the validation fold. The best trial's reported score is optimistically biased. Always hold out a separate test set that is never used during the search, and report the final metric on that test set.

Reproducibility checklist: Pin random seeds, record library versions, log the full search space definition, and store the best model's checkpoint. This ensures that any colleague (or your future self) can reproduce the winning configuration exactly.

6. Consider Multi-Fidelity Methods

Methods like Hyperband and ASHA (Asynchronous Successive Halving Algorithm) combine early stopping with adaptive resource allocation. They train many configurations for a small number of epochs, eliminate the worst performers, and allocate more resources to the survivors. This can be orders of magnitude more efficient than running every trial to completion. 

# Optuna with Hyperband-style pruning
import optuna
from optuna.pruners import HyperbandPruner

def objective(trial):
    lr   = trial.suggest_float("lr", 1e-4, 1e-1, log=True)
    wd   = trial.suggest_float("weight_decay", 1e-6, 1e-2, log=True)
    bs   = trial.suggest_categorical("batch_size", [32, 64, 128, 256])

    model = build_model(lr=lr, weight_decay=wd)
    for epoch in range(50):
        train_loss = train_one_epoch(model, batch_size=bs)
        val_score  = evaluate(model)

        # Report intermediate value for pruning
        trial.report(val_score, epoch)
        if trial.should_prune():
            raise optuna.exceptions.TrialPruned()

    return val_score

study = optuna.create_study(
    direction="maximize",
    pruner=HyperbandPruner(min_resource=1, max_resource=50, reduction_factor=3),
)
study.optimize(objective, n_trials=200)

With Hyperband pruning, Optuna allocates resources adaptively: unpromising trials are stopped after just a few epochs, while promising ones train for the full 50 epochs. In the example above, 200 trials are requested, but most will terminate early — resulting in a total compute cost far below 200 × 50 full epochs.

Common mistake: Tuning hyperparameters on the entire dataset and then reporting the same metric as the final result. This leads to overestimation. Always maintain a strict train / validation / test split, and only touch the test set once — at the very end.
