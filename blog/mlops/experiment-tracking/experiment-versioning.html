Experiment Versioning: Code + Data + Config

MLOps Series Experiment Tracking

The Versioning Challenge

Reproducing an ML experiment requires more than just code. You need the exact data that was used, the configuration that controlled training, the random seeds, the library versions, and the environment. A single missing piece can produce different results — sometimes dramatically different.

Traditional software versioning (Git for code) covers only one leg of the tripod. ML experiments sit at the intersection of three versioned artifacts: code, data, and configuration. Each must be independently versioned and linked together for every experiment run.

The reproducibility equation: Experiment = f(Codev, Datav, Configv, Envv). If any variable changes, the output may change. Version all four.

Code Versioning (Git Integration)

Git is the foundation. Every experiment run should capture the exact commit hash, branch name, and whether there were uncommitted changes (dirty state). Experiment trackers like MLflow and Weights & Biases can auto-log Git metadata. 

# Auto-capture Git state at experiment start
import subprocess
import mlflow

def log_git_state():
    commit = subprocess.check_output(
        ["git", "rev-parse", "HEAD"]
    ).decode().strip()

    branch = subprocess.check_output(
        ["git", "rev-parse", "--abbrev-ref", "HEAD"]
    ).decode().strip()

    # Check for uncommitted changes
    diff = subprocess.check_output(
        ["git", "diff", "--stat"]
    ).decode().strip()

    mlflow.set_tag("git.commit", commit)
    mlflow.set_tag("git.branch", branch)
    mlflow.set_tag("git.dirty", str(bool(diff)))

    if diff:
        # Save the diff as an artifact for full reproducibility
        patch = subprocess.check_output(["git", "diff"]).decode()
        with open("uncommitted.patch", "w") as f:
            f.write(patch)
        mlflow.log_artifact("uncommitted.patch")
Warning: Never run experiments on dirty Git state in production pipelines. Enforce a CI check that rejects experiment runs with uncommitted changes.

Data Versioning (DVC, Delta Lake)

Data changes silently — new rows arrive, labels get corrected, preprocessing logic evolves. DVC (Data Version Control) extends Git to track large data files via content-addressable hashes stored in remote storage (S3, GCS). Delta Lake provides ACID transactions and time-travel queries for tabular data versioned in a lakehouse. 

# DVC workflow — version data alongside code
# Terminal commands wrapped in Python for automation
import subprocess

def version_dataset(data_path, remote="s3://ml-data-bucket"):
    # Track file with DVC (generates .dvc metadata)
    subprocess.run(["dvc", "add", data_path])

    # Push data to remote storage
    subprocess.run(["dvc", "push"])

    # Commit the .dvc file to Git
    subprocess.run(["git", "add", f"{data_path}.dvc"])
    subprocess.run(["git", "commit", "-m", f"Update {data_path}"])

def restore_dataset(commit_hash):
    # Checkout code + DVC metadata at that commit
    subprocess.run(["git", "checkout", commit_hash])
    # Pull the corresponding data files
    subprocess.run(["dvc", "pull"])
# Delta Lake time-travel for tabular data
from delta.tables import DeltaTable
from pyspark.sql import SparkSession

spark = SparkSession.builder.getOrCreate()

# Read data at a specific version
df_v3 = spark.read.format("delta").option(
    "versionAsOf", 3
).load("/data/training_set")

# Log the Delta version with the experiment
delta_table = DeltaTable.forPath(spark, "/data/training_set")
version = delta_table.history(1).select("version").first()[0]
mlflow.log_param("data_version", version)

Config Versioning (Hydra, OmegaConf)

Configuration sprawl is a common source of irreproducibility. Hydra by Facebook Research provides hierarchical configuration composition — you define base configs, override groups, and sweep parameters in structured YAML files. Every run auto-saves the resolved config, making it trivial to reproduce. 

# Hydra-based config management
import hydra
from omegaconf import DictConfig, OmegaConf
import mlflow

@hydra.main(config_path="conf", config_name="train", version_base=None)
def train(cfg: DictConfig):
    # Hydra auto-saves full resolved config to outputs/
    print(OmegaConf.to_yaml(cfg))

    # Log flattened config to experiment tracker
    flat_cfg = flatten_dict(OmegaConf.to_container(cfg))
    for key, value in flat_cfg.items():
        mlflow.log_param(key, value)

    # Also log the raw YAML as an artifact
    config_str = OmegaConf.to_yaml(cfg)
    with open("config_snapshot.yaml", "w") as f:
        f.write(config_str)
    mlflow.log_artifact("config_snapshot.yaml")

    # Training logic uses cfg.model.*, cfg.data.*, etc.
    model = build_model(cfg.model)
    data = load_data(cfg.data)
    run_training(model, data, cfg.training)
Best practice: Commit your Hydra config directory to Git. Use overrides only for sweeps and experiments — never modify the base configs in place. This gives you a clean diff between any two experiment configurations.

Linking Code + Data + Config

The final piece is a unified metadata record that links all three version identifiers for every experiment run. Your experiment tracker should store the Git commit hash, the data version (DVC hash or Delta version), and the config hash as first-class metadata — not buried in logs. 

# Unified versioning wrapper
import hashlib
import json
import mlflow

class ExperimentVersioner:
    def __init__(self, config, data_path):
        self.git_sha = self._get_git_sha()
        self.data_hash = self._hash_dvc_file(data_path)
        self.config_hash = self._hash_config(config)

    def _get_git_sha(self):
        import subprocess
        return subprocess.check_output(
            ["git", "rev-parse", "HEAD"]
        ).decode().strip()

    def _hash_dvc_file(self, data_path):
        with open(f"{data_path}.dvc") as f:
            return hashlib.md5(f.read().encode()).hexdigest()[:12]

    def _hash_config(self, config):
        config_str = json.dumps(config, sort_keys=True)
        return hashlib.sha256(config_str.encode()).hexdigest()[:12]

    def log_versions(self):
        mlflow.set_tag("version.code", self.git_sha)
        mlflow.set_tag("version.data", self.data_hash)
        mlflow.set_tag("version.config", self.config_hash)
        mlflow.set_tag("version.composite",
            f"{self.git_sha[:8]}-{self.data_hash}-{self.config_hash}")

Reproducibility Checklist

Use this checklist before marking any experiment as "final" or using its results to make decisions.

🔍 Code

  • Git commit hash recorded
  • No uncommitted changes
  • requirements.txt or lock file versioned
  • Docker image tag or hash captured

📂 Data

  • DVC hash or Delta version logged
  • Data schema validated
  • Train/val/test splits deterministic
  • Preprocessing pipeline versioned

⚙ Config

  • Full resolved config saved as artifact
  • Random seeds set and logged
  • All hyperparameters explicit
  • No hardcoded magic numbers

🏠 Environment

  • Python version logged
  • GPU driver and CUDA version captured
  • OS and hardware specs recorded
  • Container image reproducible
Golden rule: If you cannot regenerate a result by running a single command with the stored version identifiers, your versioning is incomplete. Automate the restoration — git checkout <sha> && dvc pull && python train.py --config <path>.
