Neptune.ai: Metadata Store for ML

MLOps Series Experiment Tracking

Neptune Overview and Architecture

Neptune.ai positions itself not merely as an experiment tracker but as a metadata store for MLOps. Where tools like MLflow focus on packaging models and W&B centers on interactive visualizations, Neptune is designed from the ground up to capture, organize, and query every piece of metadata that your machine-learning pipeline produces. This includes training metrics, hyperparameters, dataset versions, model binaries, hardware utilization logs, and even free-form notes attached by team members during review.

The core abstraction in Neptune is the Run — a flexible, hierarchical container that stores key-value pairs at arbitrary depth. Unlike flat dictionaries or rigid schemas, Neptune runs use a filesystem-like namespace so you can log data at paths like train/loss, val/accuracy, or sys/hardware/gpu/0/memory. This gives you the freedom to organize metadata however your project demands without fighting a predefined structure.

Architecturally, Neptune operates as a managed cloud service (with a self-hosted option). Your training script communicates with the Neptune backend via a lightweight Python client that batches and uploads metadata asynchronously. This means logging calls are non-blocking — your training loop never stalls waiting for a network response. The web UI then provides real-time dashboards, comparison views, and a powerful query language for filtering across thousands of runs.

Metadata Store vs. Experiment Tracker: Neptune prefers the term "metadata store" because it captures more than just experiments. You can log production inference results, data pipeline outputs, model registry events, and CI/CD artifacts — anything that produces metadata worth tracking over time.

Getting started is straightforward. Install the client, set your API token, and initialize a run inside your training script: 

# Install: pip install neptune
import neptune

# Initialize a new run
run = neptune.init_run(
    project="workspace/project-name",
    api_token="YOUR_API_TOKEN",
)

# Log a simple parameter
run["parameters/learning_rate"] = 0.001
run["parameters/batch_size"] = 64

# Always stop the run when done
run.stop()
API Token Security: Never hard-code your Neptune API token in source files. Set it via the NEPTUNE_API_TOKEN environment variable or use a secrets manager. Neptune automatically reads from the environment when api_token is omitted.

Runs and Namespaces

A Run in Neptune is a mutable, versioned object that lives in a project. Each run is assigned a unique short ID (like RUN-42) and a system-generated UUID. You create a run by calling neptune.init_run(), which returns a handle you can treat like a nested dictionary. The keys are slash-separated paths, and the values can be virtually any Python object — scalars, strings, lists, dictionaries, files, or even PIL images.

The namespace hierarchy is what makes Neptune's data model so powerful. Rather than dumping everything into a flat bag of key-value pairs, you organize metadata into logical groups. A common convention is to separate parameters/, train/, val/, test/, and artifacts/ at the top level, then nest further as needed. 

import neptune

run = neptune.init_run(project="workspace/text-classifier")

# Hyperparameters namespace
run["parameters/model"] = "bert-base-uncased"
run["parameters/optimizer"] = "AdamW"
run["parameters/lr"] = 3e-5
run["parameters/epochs"] = 10

# Log an entire dict at once
config = {"dropout": 0.1, "warmup_steps": 500, "weight_decay": 0.01}
run["parameters"] = config  # merges into the parameters namespace

# Training loop — log to train/ and val/ namespaces
for epoch in range(10):
    train_loss = train_one_epoch(model, train_loader)
    val_loss, val_acc = evaluate(model, val_loader)

    run["train/loss"].append(train_loss)
    run["val/loss"].append(val_loss)
    run["val/accuracy"].append(val_acc)

run.stop()

Notice how .append() is used for series data (metrics that change over time) while direct assignment (= value) is used for static values. This distinction is fundamental in Neptune: assign overwrites the value at a path, while append adds a new point to a series. Series data is rendered as charts in the Neptune UI automatically.

Namespace Tip: You can fetch an entire namespace as a dictionary by calling run["parameters"].fetch(). This is useful for reproducing experiments — just pull the full config from a previous run and pass it to your model constructor.

Neptune also supports tags and system namespaces. Tags are string labels you attach to a run for quick filtering (run["sys/tags"].add("baseline")). The sys/ namespace is reserved for system-populated fields like creation time, hostname, run state, and owner. You can read them but not overwrite them.

Logging Metadata and Artifacts

Neptune supports a rich variety of metadata types. Understanding which type to use and how the client serializes data is key to getting the most from the platform. The main categories are: atoms (single values), series (sequences of values), sets (collections like tags), and file-based types (images, HTML, artifacts).

Atoms — Single Values

Assigning a Python scalar, string, or datetime directly to a path creates an atom. Atoms are displayed as static fields in the run dashboard and are queryable in the runs table. 

# Atoms — assign single values
run["dataset/name"] = "IMDB Reviews"
run["dataset/size"] = 50000
run["dataset/balanced"] = True
run["best_val_f1"] = 0.923

Series — Time-Ordered Sequences

Series are the backbone of training monitoring. Each .append() call adds a timestamped data point. Neptune automatically renders series as line charts and supports custom x-axes via the step parameter. 

# Series — append values over training steps
for step, batch in enumerate(train_loader):
    loss = train_step(model, batch)
    run["train/batch_loss"].append(value=loss, step=step)

# Log with a custom x-axis (wall-clock time)
import time
start = time.time()
for epoch in range(num_epochs):
    loss = train_one_epoch(model, loader)
    elapsed = time.time() - start
    run["train/epoch_loss"].append(value=loss, step=elapsed)

Files and Images

You can upload files, images, and even entire directories. Neptune handles serialization transparently — pass a PIL image, a matplotlib figure, a path to a file, or raw bytes. 

from neptune.types import File
import matplotlib.pyplot as plt

# Upload a matplotlib figure
fig, ax = plt.subplots()
ax.plot(train_losses)
ax.set_title("Training Loss")
run["charts/train_loss_curve"].upload(fig)
plt.close(fig)

# Upload a model checkpoint file
run["artifacts/best_model"].upload("checkpoints/best_model.pt")

# Upload an entire directory
run["artifacts/config_dir"].upload_files("configs/")

# Log a series of images (e.g., sample predictions)
for i, img in enumerate(sample_images):
    run["predictions/samples"].append(File.as_image(img), step=i)
Append vs. Assign vs. Upload: Use assign (=) for static metadata. Use append for series data that grows over time. Use upload for binary files, images, and artifacts. Mixing these on the same path will raise a type error.

Logging Source Code and Environment

Neptune can automatically snapshot your source code, Git info, and Python environment so that every run is fully reproducible: 

# Neptune auto-captures these in the sys/ namespace:
#   sys/source_code   — snapshot of .py files
#   sys/git            — commit hash, branch, dirty flag
#   sys/requirements   — pip freeze output

# You can also log manually:
run["source/notebook"].upload("train_notebook.ipynb")
run["source/config"].upload("config.yaml")
Large Artifacts: For very large files (multi-GB model checkpoints or datasets), consider using Neptune's track_files() method which stores only a reference and hash rather than uploading the full binary. This keeps your Neptune storage lean while still providing lineage tracking.

Monitoring and Dashboards

One of Neptune's strongest features is its real-time monitoring dashboard. As your training script logs metrics, the web UI updates live — you can watch loss curves descend, compare multiple runs side by side, and catch diverging experiments before they waste hours of GPU time. Neptune's monitoring is built for teams running dozens of experiments in parallel.

The Runs Table

The central hub of any Neptune project is the Runs Table. It displays every run as a row with columns for any metadata path you choose. You can sort, filter, and group runs using a powerful query syntax. For example, filter runs where parameters/lr < 0.001 AND val/accuracy > 0.90 to find your best low-learning-rate experiments.

Columns are fully customizable. Pin your most important metrics, hide irrelevant system fields, and save named views that your team can share. Views persist across sessions, so your "Baseline Comparison" or "Hyperparameter Sweep" view is always one click away.

Custom Dashboards

Beyond the runs table, Neptune lets you build custom dashboards with drag-and-drop widgets: line charts, scatter plots, image galleries, parameter tables, and markdown notes. Dashboards are saved per-run or per-project and can be templated so every new run automatically gets the same layout.

Comparing Runs: Select multiple runs in the runs table and click "Compare." Neptune overlays their metric series on shared axes, highlights parameter differences in a diff view, and lets you toggle individual runs on and off. This is invaluable during hyperparameter tuning.

Hardware Monitoring

Neptune automatically logs system metrics during training — CPU utilization, RAM usage, GPU memory, GPU utilization, and disk I/O. These appear under the monitoring/ namespace and are plotted alongside your training metrics. This helps diagnose bottlenecks: if GPU utilization drops to zero periodically, your data loading pipeline may be too slow. 

# Hardware metrics are logged automatically.
# Access them programmatically after training:

run = neptune.init_run(
    project="workspace/project",
    with_id="RUN-42",   # resume an existing run
    mode="read-only",
)

# Fetch GPU memory series as a pandas DataFrame
gpu_mem = run["monitoring/gpu/0/memory"].fetch_values()
print(gpu_mem.head())

run.stop()

Alerts and Notifications

Neptune does not natively send email or Slack alerts on metric thresholds, but you can build this into your training scripts easily. Since every metric is queryable via the Python client, you can poll a run's values from a separate monitoring script and trigger notifications through your preferred channels.

Long-Running Experiments: For multi-day training jobs, use Neptune's mode="async" (the default) to ensure metadata is uploaded in the background. If your script crashes, Neptune retains all data logged up to the last successful sync. You can also resume a run using with_id="RUN-XX" to continue logging after a restart.

Framework Integrations

Neptune provides first-class integrations with the most popular ML frameworks. These integrations wrap Neptune's logging API into framework-specific callbacks, loggers, or hooks so that you get rich experiment tracking with minimal code changes. Below are the key integrations and how to use them.

PyTorch and PyTorch Lightning

For vanilla PyTorch, you call Neptune's API directly in your training loop (as shown in earlier sections). For PyTorch Lightning, Neptune provides a dedicated logger that plugs into the Trainer: 

from lightning.pytorch import Trainer
from neptune.integrations.pytorch_lightning import NeptuneLogger

neptune_logger = NeptuneLogger(
    project="workspace/lightning-exp",
    tags=["resnet", "cifar10"],
)

trainer = Trainer(
    max_epochs=20,
    logger=neptune_logger,
    accelerator="gpu",
)
trainer.fit(model, train_loader, val_loader)

# Access the underlying Neptune run
neptune_logger.experiment["notes"] = "First ResNet baseline on CIFAR-10"

TensorFlow / Keras

Neptune integrates with Keras via a callback that automatically logs epoch-level metrics, model architecture summaries, and optionally the best model weights: 

from neptune.integrations.tensorflow_keras import NeptuneCallback

run = neptune.init_run(project="workspace/keras-exp")

neptune_cb = NeptuneCallback(run=run, log_model_diagram=True)

model.fit(
    x_train, y_train,
    epochs=30,
    validation_data=(x_val, y_val),
    callbacks=[neptune_cb],
)

run.stop()

Scikit-learn

For classical ML workflows, Neptune provides utilities to log scikit-learn model summaries, classification reports, confusion matrices, and feature importances with a single call: 

from neptune.integrations.sklearn import create_classifier_summary

from sklearn.ensemble import RandomForestClassifier

rf = RandomForestClassifier(n_estimators=200, max_depth=10)
rf.fit(X_train, y_train)

run = neptune.init_run(project="workspace/sklearn-exp")
create_classifier_summary(rf, X_train, X_test, y_train, y_test, run=run)
run.stop()

XGBoost and LightGBM

Gradient boosting frameworks are supported via Neptune callbacks that log training and validation metrics at each boosting round, plus feature importance plots: 

from neptune.integrations.xgboost import NeptuneCallback as XGBCallback
import xgboost as xgb

run = neptune.init_run(project="workspace/xgb-exp")

dtrain = xgb.DMatrix(X_train, label=y_train)
dval = xgb.DMatrix(X_val, label=y_val)

params = {"max_depth": 6, "eta": 0.1, "objective": "binary:logistic"}
xgb.train(
    params, dtrain,
    num_boost_round=300,
    evals=[(dval, "val")],
    callbacks=[XGBCallback(run=run)],
)
run.stop()

Optuna — Hyperparameter Optimization

Neptune's Optuna integration logs every trial's parameters and objective value, plus the optimization history and parameter importance charts: 

import optuna
import neptune
from neptune.integrations.optuna import NeptuneCallback as OptunaCallback

run = neptune.init_run(project="workspace/hpo")
neptune_cb = OptunaCallback(run)

def objective(trial):
    lr = trial.suggest_float("lr", 1e-5, 1e-2, log=True)
    depth = trial.suggest_int("depth", 3, 12)
    score = train_and_evaluate(lr, depth)
    return score

study = optuna.create_study(direction="maximize")
study.optimize(objective, n_trials=50, callbacks=[neptune_cb])

run.stop()
Integration Pattern: Nearly every Neptune integration follows the same pattern — create a run, wrap it in a framework-specific callback or logger, pass that callback to your training function, and call run.stop() when done. This consistency makes it easy to adopt Neptune incrementally across different parts of your ML stack.

When to Use Neptune

Choosing between Neptune, MLflow, and Weights & Biases depends on your team's priorities, infrastructure constraints, and workflow complexity. Each tool has overlapping features but different strengths.

Neptune Strengths

  • Flexible, hierarchical metadata model — no rigid schemas
  • Powerful runs table with custom views and saved queries
  • Non-blocking async logging — minimal training overhead
  • Granular access control and team collaboration features
  • Strong programmatic API for querying past runs

Neptune Trade-offs

  • Managed SaaS — requires internet (self-hosted option exists)
  • No built-in model serving or deployment pipeline
  • Smaller open-source community than MLflow or W&B
  • Free tier has storage and tracking limits
  • Less opinionated — requires discipline in namespace conventions

Neptune vs. MLflow

MLflow is open-source and self-hosted by default, making it a natural fit for teams with strict data-sovereignty requirements. It provides a model registry, model serving, and a project packaging format that Neptune does not. However, MLflow's flat parameter logging and limited UI customization can feel constraining at scale. Neptune's hierarchical namespaces and advanced filtering shine when you have thousands of runs to organize.

Neptune vs. Weights & Biases

W&B is known for stunning visualizations — interactive charts, media panels, and the Sweeps hyperparameter tuning engine. Neptune's UI is more data-table oriented, excelling at structured querying and comparison rather than ad-hoc exploration. If your workflow revolves around visual storytelling and collaborative reports, W&B may be the better fit. If you need a structured metadata backbone that scales across projects and teams, Neptune's metadata-store philosophy is compelling.

Ideal Use Cases for Neptune

Neptune is particularly well-suited for:

  • Large teams running hundreds of experiments across multiple projects that need shared views, access control, and consistent metadata organization.
  • Production ML systems where experiment metadata must be queried programmatically by CI/CD pipelines, model registries, or monitoring dashboards.
  • Heterogeneous ML stacks combining deep learning, classical ML, and hyperparameter optimization — Neptune's integrations cover the full spectrum.
  • Compliance-sensitive environments that need audit trails, immutable run records, and fine-grained access permissions.
Try Before Committing: Neptune offers a generous free tier for individual researchers. Start by instrumenting one training script, explore the UI, and evaluate whether the metadata-store paradigm fits your workflow before migrating an entire team.
Migration Note: If you are moving from MLflow or W&B, Neptune provides import utilities and guides for migrating historical run data. Plan your namespace schema upfront — consistent path conventions (parameters/, train/, val/, artifacts/) pay dividends as your project grows.
