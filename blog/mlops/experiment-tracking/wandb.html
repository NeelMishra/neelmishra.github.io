Weights & Biases: Collaborative Tracking

MLOps Series Experiment Tracking

Getting Started with W&B

Weights & Biases (W&B) is a hosted experiment tracking platform that provides a centralized dashboard, automatic system metrics capture, and powerful collaboration tools out of the box. Unlike self-hosted alternatives, W&B requires zero infrastructure setup — you install the Python SDK, authenticate with an API key, and begin logging experiments in minutes.

At its core, every W&B workflow revolves around three concepts: projects (containers that group related experiments), runs (individual training sessions within a project), and artifacts (versioned datasets, models, or files attached to runs). Understanding this hierarchy is essential before diving into more advanced features like sweeps or reports.

Installation and Authentication

Install the SDK via pip, then log in once to cache your API key locally. Every subsequent script will authenticate automatically. 

# Install the wandb package
# pip install wandb

import wandb

# Authenticate — opens a browser prompt or reads WANDB_API_KEY env var
wandb.login()

Initializing a Run

The wandb.init() call creates a new run inside a project. You pass a configuration dictionary that records every hyperparameter for full reproducibility. The run object returned is your handle for logging metrics throughout training. 

import wandb

# Start a new run inside the "image-classifier" project
run = wandb.init(
    project="image-classifier",
    name="resnet50-lr0.001",
    config={
        "learning_rate": 0.001,
        "batch_size": 64,
        "epochs": 25,
        "architecture": "ResNet-50",
        "optimizer": "AdamW",
        "weight_decay": 1e-4,
    }
)

# Access config values anywhere in your code
lr = wandb.config.learning_rate
bs = wandb.config.batch_size
print(f"Training with lr={lr}, batch_size={bs}")
Tip: Always pass a descriptive name to wandb.init(). Auto-generated names like dashing-sunset-42 are fun but make it hard to locate specific runs weeks later. A naming convention such as {model}-{key_param} keeps your dashboard scannable.

Finishing a Run

Call wandb.finish() at the end of your script (or use a context manager) to ensure all buffered data is uploaded and the run is marked complete on the dashboard. 

# Option A: explicit finish
wandb.finish()

# Option B: context-manager pattern (auto-finishes)
with wandb.init(project="my-project") as run:
    # ... training loop ...
    pass

Logging Metrics and Media

W&B shines when you move beyond simple scalar logging. The wandb.log() call accepts dictionaries containing scalars, images, histograms, audio clips, video, point clouds, and even custom Plotly or Vega charts. Every logged value is automatically versioned against the global step counter, creating a rich, queryable timeline of your experiment.

Scalar Metrics

The most common use case — log training loss, validation accuracy, and learning rate at every step or epoch. 

for epoch in range(num_epochs):
    for batch_idx, (inputs, targets) in enumerate(train_loader):
        loss = train_step(model, inputs, targets)

        # Log scalars — W&B auto-increments the step
        wandb.log({
            "train/loss": loss,
            "train/learning_rate": scheduler.get_last_lr()[0],
        })

    # Log epoch-level metrics
    val_acc = evaluate(model, val_loader)
    wandb.log({
        "val/accuracy": val_acc,
        "epoch": epoch,
    })

Images and Media

Log sample predictions, confusion matrices, or generated images directly to the dashboard. W&B renders them in an interactive media panel.

Hyperparameter Sweeps

W&B Sweeps automate hyperparameter search across multiple machines. You define a sweep configuration that specifies the search space and strategy, then launch agents on any number of machines. The central sweep controller coordinates which hyperparameter combinations each agent should try next.

Sweep Configuration

A sweep config is a YAML-like dictionary that declares the optimization method, the metric to optimize, and the parameter search space. 

sweep_config = {
    "method": "bayes",          # bayes, random, or grid
    "metric": {
        "name": "val/accuracy",
        "goal": "maximize",
    },
    "parameters": {
        "learning_rate": {
            "distribution": "log_uniform_values",
            "min": 1e-5,
            "max": 1e-2,
        },
        "batch_size": {"values": [16, 32, 64, 128]},
        "optimizer": {"values": ["adam", "sgd", "adamw"]},
        "dropout": {
            "distribution": "uniform",
            "min": 0.1,
            "max": 0.5,
        },
    },
    "early_terminate": {
        "type": "hyperband",
        "min_iter": 5,
        "s": 2,
    },
}

# Create the sweep on the W&B server
sweep_id = wandb.sweep(sweep_config, project="image-classifier")

Launching Agents

Each agent picks up hyperparameter combinations from the sweep controller, runs the training function, and reports back. You can launch agents on multiple GPUs or machines. 

def train_sweep():
    # wandb.init reads sweep parameters automatically
    with wandb.init() as run:
        config = wandb.config
        model = build_model(dropout=config.dropout)
        optimizer = get_optimizer(config.optimizer, model, config.learning_rate)

        for epoch in range(20):
            train_loss = train_one_epoch(model, optimizer, train_loader)
            val_acc = evaluate(model, val_loader)
            wandb.log({
                "train/loss": train_loss,
                "val/accuracy": val_acc,
            })

# Launch the agent — runs train_sweep with different configs
wandb.agent(sweep_id, function=train_sweep, count=50)

Comparing Sweep Strategies

Choosing the right search strategy depends on your budget, parameter space size, and whether you have prior knowledge about promising regions.

Bayesian Optimization

Uses a Gaussian process surrogate model to predict promising regions of the search space. Best when evaluations are expensive and you want to converge quickly with fewer trials. W&B's Bayesian method balances exploration and exploitation automatically.

  • Pros: Sample-efficient, finds good configs fast
  • Cons: Overhead per trial, less effective in very high dimensions
  • Best for: 10–200 trial budgets, continuous parameters

Random Search

Samples hyperparameters independently from their distributions. Surprisingly effective — Bergstra & Bengio (2012) showed random search often outperforms grid search because it covers more unique values per dimension.

  • Pros: Simple, embarrassingly parallel, no sequential dependency
  • Cons: No learning between trials, needs more budget
  • Best for: Large budgets, high-dimensional spaces, baselines
Early Termination: Combine any sweep strategy with W&B's Hyperband early termination to automatically stop underperforming runs. This can reduce total compute by 3–5× without sacrificing final performance.

Reports and Tables

W&B Reports are collaborative documents that mix live charts, run comparisons, markdown narrative, and embedded code. They replace ad-hoc Jupyter notebooks as the primary medium for communicating experiment results to teammates and stakeholders.

Creating a Report

Reports can be created from the W&B dashboard UI or programmatically via the SDK. Each report contains panels — charts, tables, media grids — that update live as new runs complete. You can freeze a snapshot of the data to preserve results for a publication.

Using wandb.Table

wandb.Table is a structured data type for logging tabular data. It supports columns of scalars, images, audio, and HTML — making it ideal for per-sample analysis like error audits or dataset exploration. 

# Create a table to log per-sample predictions
columns = ["image", "true_label", "predicted", "confidence", "correct"]
table = wandb.Table(columns=columns)

for i in range(len(test_images)):
    pred = predictions[i]
    conf = confidences[i]
    correct = pred == test_labels[i]
    table.add_data(
        wandb.Image(test_images[i]),
        test_labels[i],
        pred,
        round(conf, 4),
        correct,
    )

wandb.log({"prediction_audit": table})

Data Visualization in Tables

Once a table is logged, the W&B UI lets you sort, filter, and group rows interactively. You can create derived charts — scatter plots of confidence vs. correctness, histograms of error rates by class, or image grids filtered to only misclassified samples. This turns the table into a powerful debugging tool without writing additional code.

Sharing and Exporting

Reports can be shared via a private link, published to the W&B gallery, or exported as LaTeX for academic papers. Each report has granular access controls — you can share read-only views with stakeholders who don't need to see the underlying runs.

Team Collaboration Features

One of W&B's strongest differentiators is its built-in collaboration layer. Instead of emailing CSVs or screenshots, teams work inside shared projects where every run, artifact, and report is accessible to all members with appropriate permissions.

Teams and Organizations

A W&B team is a shared workspace where members can create projects, view each other's runs, and collaborate on reports. Teams sit under an organization, which controls billing, SSO, and data retention policies. Roles include Admin, Member, and Viewer — each with different permission levels.

Sharing Runs and Comparing Results

# Initialize a run inside a team project
run = wandb.init(
    entity="ml-research-team",   # team name
    project="nlp-transformers",
    name="bert-base-finetune-v3",
    tags=["bert", "finetune", "squad"],
    notes="Testing longer warmup schedule with linear decay",
)

# Tags and notes make runs discoverable by teammates
# The dashboard lets any team member compare runs side-by-side

Team members can leave comments directly on runs, tag colleagues, and link related runs together. The activity feed shows who changed what, creating a full audit trail of experiment decisions.

Access Control and Privacy

Projects can be set to private (team-only), public (visible to anyone), or open (anyone can log runs). For regulated industries, W&B offers a self-managed deployment option where all data stays within your VPC or on-premises infrastructure.

Advanced Usage Patterns

Beyond basic logging and sweeps, W&B provides a rich ecosystem of advanced features: versioned artifacts for data and model lineage, a model registry for production promotion, and deep integrations with popular ML frameworks.

Artifacts: Data and Model Versioning

Artifacts let you version datasets, model checkpoints, and evaluation results. Each artifact is content-addressed — if the data hasn't changed, W&B deduplicates storage automatically. 

# Log a dataset as a versioned artifact
artifact = wandb.Artifact("training-data", type="dataset")
artifact.add_dir("./data/processed")
run.log_artifact(artifact)

# In a downstream run, pull the exact version used for training
data_artifact = run.use_artifact("training-data:v3")
data_dir = data_artifact.download()

# Log a model checkpoint after training
model_artifact = wandb.Artifact("resnet50-classifier", type="model")
model_artifact.add_file("checkpoints/best_model.pt")
run.log_artifact(model_artifact)

Model Registry

The W&B Model Registry provides a centralized catalog of production-ready models. You link model artifacts to named registry entries and promote them through stages like staging and production. This creates a clear handoff point between training and deployment pipelines. 

# Link the best model to the registry
run.link_artifact(
    artifact=model_artifact,
    target_path="ml-research-team/model-registry/resnet50-classifier",
    aliases=["best", "v1.2"],
)

Framework Integrations

W&B provides first-class integrations with PyTorch, TensorFlow, Hugging Face Transformers, Keras, XGBoost, LightGBM, and more. These integrations auto-log metrics, gradients, model architecture, and system utilization with a single callback or wrapper. 

# PyTorch Lightning integration
from pytorch_lightning.loggers import WandbLogger

wandb_logger = WandbLogger(project="lightning-demo", log_model=True)
trainer = pl.Trainer(
    max_epochs=20,
    logger=wandb_logger,
    callbacks=[pl.callbacks.ModelCheckpoint(monitor="val_loss")],
)
trainer.fit(model, train_loader, val_loader)

# Hugging Face Transformers integration
from transformers import TrainingArguments, Trainer
import wandb

wandb.init(project="hf-text-classification")

training_args = TrainingArguments(
    output_dir="./results",
    num_train_epochs=5,
    per_device_train_batch_size=16,
    evaluation_strategy="epoch",
    report_to="wandb",           # enables W&B auto-logging
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
)
trainer.train()

Custom Logging Patterns

For advanced use cases, you can define custom x-axes, log at arbitrary steps, and create summary metrics that appear in the run overview without cluttering the time-series charts. 

# Define a custom x-axis for token-based tracking
wandb.define_metric("tokens_seen")
wandb.define_metric("train/*", step_metric="tokens_seen")

for batch in data_loader:
    loss = train_step(batch)
    total_tokens += batch.num_tokens
    wandb.log({
        "tokens_seen": total_tokens,
        "train/loss": loss,
    })

# Set summary metrics for the final results table
wandb.run.summary["best_val_accuracy"] = best_acc
wandb.run.summary["total_training_time_hrs"] = elapsed / 3600
