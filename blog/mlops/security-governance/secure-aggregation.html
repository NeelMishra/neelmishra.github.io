Secure Aggregation and MPC for ML

MLOps Series Security & Governance

Introduction

Federated learning keeps raw data on client devices, but model updates themselves can leak sensitive information. Research has shown that gradient inversion attacks can reconstruct individual training examples from shared gradients with startling fidelity. Secure aggregation (SecAgg) solves this by ensuring the server learns only the aggregate of client updates — never any individual contribution.

This post covers the cryptographic building blocks behind secure aggregation: from basic secret sharing to fully homomorphic encryption, and how they combine into practical protocols used in production federated learning systems.

Threat model: We assume an honest-but-curious server that follows the protocol correctly but may try to learn individual client updates from the data it observes. Malicious-server settings require additional mechanisms like verifiable computation.

Crypto Primitives

Before diving into protocols, let’s establish the cryptographic primitives that underpin secure computation for ML:

Key Agreement (Diffie-Hellman)

Two parties can establish a shared secret over an insecure channel. In SecAgg, each pair of clients uses Diffie-Hellman to create pairwise masks that cancel out during aggregation. The server never learns the individual masks.

Pseudorandom Generators (PRG)

A PRG expands a short seed into a long pseudorandom stream. In SecAgg, each pairwise shared secret seeds a PRG that generates a mask vector the same size as the model. This avoids transmitting full-size masks between clients.

Key Encapsulation

If a client drops out mid-protocol, other clients must be able to reconstruct its mask to unblind the aggregate. This requires secret-sharing the client’s private key among other participants using a threshold scheme.

Performance note: The cryptographic overhead of SecAgg is dominated by the pairwise mask generation, which is O(n²) in the number of clients. For large cohorts (>1000 clients), hierarchical or turbo-aggregation variants reduce this to O(n·log n).

Secret Sharing

Secret sharing is the foundational technique for splitting a value among multiple parties so that no single party (or small subset) can reconstruct it. The most widely used scheme is Shamir’s Secret Sharing (1979).

Shamir’s (t, n) Scheme

A secret s is encoded as the constant term of a random polynomial of degree t-1. Each of n parties receives an evaluation of this polynomial at a distinct point. Any t shares can reconstruct the polynomial (and thus the secret) via Lagrange interpolation, but t-1 or fewer shares reveal nothing.

# ── Shamir's Secret Sharing ── import random def share_secret(secret, n_shares, threshold, prime): # Generate random polynomial of degree (threshold - 1) coeffs = [secret] + [random.randint(1, prime - 1) for _ in range(threshold - 1)] shares = [] for i in range(1, n_shares + 1): y = sum(c * pow(i, p, prime) for p, c in enumerate(coeffs)) % prime shares.append((i, y)) return shares def reconstruct_secret(shares, prime): # Lagrange interpolation at x = 0 secret = 0 for i, (xi, yi) in enumerate(shares): num, den = 1, 1 for j, (xj, _) in enumerate(shares): if i != j: num = (num * (-xj)) % prime den = (den * (xi - xj)) % prime secret = (secret + yi * num * mod_inverse(den, prime)) % prime return secret

Additive Secret Sharing

A simpler alternative: split a value s into n random shares such that their sum modulo a prime equals s. This is computationally cheaper than Shamir’s scheme but requires all shares for reconstruction (no threshold tolerance).

# ── Additive secret sharing ── def additive_share(secret, n_parties, prime): shares = [random.randint(0, prime - 1) for _ in range(n_parties - 1)] last_share = (secret - sum(shares)) % prime shares.append(last_share) return shares # sum(shares) mod prime == secret def additive_reconstruct(shares, prime): return sum(shares) % prime
Caution: Additive sharing has no dropout tolerance. If even one party disappears, the secret is unrecoverable. For federated learning with unreliable clients, use Shamir’s scheme with a threshold well below the total number of clients.

Homomorphic Encryption

Homomorphic encryption (HE) allows computation on encrypted data without decryption. For ML, the relevant property is additive homomorphism: given Enc(a) and Enc(b), anyone can compute Enc(a + b) without knowing a or b.

Partially Homomorphic (PHE)

Schemes like Paillier support only addition. Each client encrypts their model update under a shared public key, the server sums the ciphertexts (which yields the encrypted sum), and a trusted party decrypts the aggregate. This is sufficient for FedAvg-style aggregation.

Somewhat / Fully Homomorphic (SHE / FHE)

Schemes like BFV, CKKS, and TFHE support both addition and multiplication on encrypted data. CKKS is particularly relevant for ML because it natively supports approximate arithmetic on real numbers, making it suitable for neural network inference on encrypted inputs.

🔒 Paillier (PHE)

Additive only. Well-understood security. Key sizes ~2048 bits. Ciphertext expansion ~2×. Fast enough for gradient aggregation with moderate model sizes.

Overhead: ~10–50× vs plaintext aggregation

🧮 CKKS (FHE)

Supports add + multiply on approximate reals. Enables encrypted inference. Larger ciphertexts (~50–100× expansion). Requires careful noise budgeting.

Overhead: ~100–1000× vs plaintext

🎯 Masking (SecAgg)

Not encryption per se — pairwise masks cancel during aggregation. Minimal ciphertext expansion. Requires coordination between clients. O(n²) setup cost.

Overhead: ~2–5× vs plaintext aggregation

🔑 Functional Encryption

The server holds a functional key that decrypts only the aggregate (not individual inputs). Single-round protocol with no client-to-client communication. Still mostly research-stage.

Overhead: ~5–20× vs plaintext

Secure Aggregation Protocol

The canonical SecAgg protocol (Bonawitz et al., 2017) used in Google’s production FL system runs in four rounds:

  1. Advertise Keys: Each client generates a DH key pair and a “self-mask” seed, then shares the public DH key with the server.
  2. Share Keys: Each client computes pairwise secrets with every other client via DH, then Shamir-shares its private DH key and self-mask seed among other clients (for dropout recovery).
  3. Masked Input: Each client masks its model update using both pairwise masks (which cancel during summation) and a self-mask (for dropout protection), then sends the masked update to the server.
  4. Unmask: Surviving clients reveal their shares of dropped clients’ secrets, allowing the server to reconstruct and remove the self-masks of dropped clients, yielding the final aggregate.
# ── SecAgg Protocol (simplified) ── def secagg_client_mask(my_update, my_id, peers, pairwise_seeds, self_seed): # Generate pairwise masks from DH shared secrets masked = copy(my_update) for peer_id, seed in pairwise_seeds.items(): mask = prg_expand(seed, len(my_update)) if my_id < peer_id: masked += mask # add mask for lower-ID client else: masked -= mask # subtract for higher-ID → cancels in sum # Add self-mask for dropout protection self_mask = prg_expand(self_seed, len(my_update)) masked += self_mask return masked def secagg_server_aggregate(masked_updates, survivor_ids, dropped_self_seeds): # Pairwise masks cancel automatically in the sum aggregate = sum(masked_updates) # Remove self-masks of dropped clients (reconstructed from shares) for seed in dropped_self_seeds: aggregate -= prg_expand(seed, model_size) # Remove self-masks of surviving clients for sid in survivor_ids: aggregate -= prg_expand(sid.self_seed, model_size) return aggregate
Dropout resilience: The protocol tolerates up to n - t client dropouts (where t is the Shamir threshold). If more clients drop, the round fails and must be restarted. In practice, setting t = n/3 provides good resilience.

Practical Considerations

Deploying secure aggregation in production introduces trade-offs that extend beyond pure cryptography:

Computational Overhead

SecAgg adds 2–5× overhead compared to plaintext aggregation for moderate cohort sizes (100–500 clients). The bottleneck is pairwise key agreement and PRG expansion, not the aggregation itself. For larger cohorts, Turbo-Aggregate uses a multi-group structure to reduce complexity from O(n²) to O(n·log n).

Integration with Differential Privacy

SecAgg and differential privacy (DP) are complementary. SecAgg hides individual updates from the server; DP bounds what can be inferred from the aggregate. The recommended stack is: clients clip gradients locally → add local DP noise → apply SecAgg masking → server sees only the noisy aggregate.

Choosing the Right Approach

The choice between masking-based SecAgg and HE-based aggregation depends on your deployment constraints:

  • Client-to-client communication possible? Use masking-based SecAgg (lower overhead).
  • No client-to-client channel? Use Paillier or functional encryption (server-mediated only).
  • Need encrypted inference too? Invest in CKKS/FHE infrastructure.
  • Regulated industry with audit requirements? Consider trusted execution environments (TEEs) as a complement to cryptographic approaches.
Don’t over-engineer: If your threat model only requires hiding individual updates from the server, masking-based SecAgg is sufficient and far cheaper than FHE. Reserve homomorphic encryption for scenarios that genuinely need computation on encrypted data.

Libraries & Tooling

Production-ready implementations are available in TensorFlow Federated (built-in SecAgg), Flower (via plugins), Microsoft SEAL (CKKS/BFV), OpenMined PySyft (MPC primitives), and Meta CrypTen (PyTorch-native MPC). Most frameworks abstract away the cryptographic details, exposing a simple API: enable SecAgg in your aggregation strategy and the framework handles key exchange, masking, and dropout recovery automatically.

Secure aggregation transforms federated learning from “data stays local” to “updates stay private.” The computational cost is modest for practical cohort sizes, and the privacy guarantees are grounded in well-studied cryptographic assumptions. Combined with differential privacy, it forms the gold standard for privacy-preserving collaborative ML.
