Adversarial Attacks: Evasion, Poisoning, Backdoors

MLOps Series Security & Governance

Introduction

Adversarial attacks exploit the statistical nature of machine-learning models to produce incorrect, misleading, or dangerous outputs. Unlike traditional software exploits that target implementation bugs, adversarial attacks target the learned decision boundary itself — a surface that is inherently continuous, high-dimensional, and difficult to reason about.

This post covers the three major families of adversarial attacks: evasion (fooling a deployed model at inference time), poisoning (corrupting the training process), and backdoors (embedding hidden triggers that activate on demand). For each family, we examine the theory, practical implementation, and real-world impact.

Prerequisites: Familiarity with gradient-based optimisation, loss functions, and basic neural network training. We use PyTorch-style pseudo-code throughout.
Ethical note: The techniques described here are presented for defensive purposes — understanding attacks is essential to building robust defenses. Always follow responsible disclosure practices and your organisation's security policies.

Evasion Attacks

Evasion attacks craft adversarial inputs that are visually (or semantically) indistinguishable from legitimate inputs but cause the model to misclassify. The perturbation is typically bounded by an Lp norm (L∞, L2, or L1) to remain imperceptible.

FGSM — Fast Gradient Sign Method

Introduced by Goodfellow et al. (2015), FGSM computes a single-step perturbation by taking the sign of the loss gradient with respect to the input. It is fast but produces relatively weak adversarial examples.

The mathematical formulation is: xadv = x + ε · sign(∇x L(θ, x, y)) where ε controls the perturbation magnitude, L is the loss function, and θ represents the model parameters. 

# FGSM attack implementation
import torch

def fgsm_attack(model, x, y, epsilon=0.03):
    """Generate adversarial example using FGSM."""
    x_adv = x.clone().detach().requires_grad_(True)

    # Forward pass
    output = model(x_adv)
    loss = torch.nn.functional.cross_entropy(output, y)

    # Backward pass to get input gradients
    loss.backward()

    # Create perturbation: epsilon * sign(gradient)
    perturbation = epsilon * x_adv.grad.sign()

    # Apply perturbation and clamp to valid range
    x_adv = torch.clamp(x + perturbation, 0.0, 1.0)
    return x_adv

PGD — Projected Gradient Descent

PGD (Madry et al., 2018) is the multi-step extension of FGSM. It iteratively applies small FGSM steps and projects the result back into the ε-ball around the original input. PGD is considered the strongest first-order attack and is the standard benchmark for evaluating adversarial robustness. 

# PGD attack implementation
def pgd_attack(model, x, y, epsilon=0.03, alpha=0.007, steps=40):
    """Generate adversarial example using PGD (L-inf)."""
    # Start from random point within epsilon-ball
    x_adv = x + torch.empty_like(x).uniform_(-epsilon, epsilon)
    x_adv = torch.clamp(x_adv, 0.0, 1.0).detach()

    for _ in range(steps):
        x_adv.requires_grad_(True)
        output = model(x_adv)
        loss = torch.nn.functional.cross_entropy(output, y)
        loss.backward()

        # FGSM step
        x_adv = x_adv.detach() + alpha * x_adv.grad.sign()

        # Project back into epsilon-ball around original x
        delta = torch.clamp(x_adv - x, -epsilon, epsilon)
        x_adv = torch.clamp(x + delta, 0.0, 1.0).detach()

    return x_adv
Why PGD matters: If your model is robust to PGD with enough iterations, it is robust to all first-order attacks. This makes PGD the gold standard for adversarial evaluation.

C&W — Carlini & Wagner Attack

The C&W attack (2017) formulates adversarial example generation as an optimisation problem that minimises the perturbation magnitude subject to misclassification. It is slower than PGD but often produces smaller, more effective perturbations — especially against models with defensive distillation.

Poisoning Attacks

Data poisoning attacks corrupt the training process by injecting malicious samples into the training set. Unlike evasion attacks that operate at inference time, poisoning attacks require access (direct or indirect) to the data pipeline. The goal is to shift the learned decision boundary so that the model makes specific errors chosen by the attacker.

Clean-Label Poisoning

In clean-label attacks, the injected samples have correct labels — making them hard to detect through label auditing. The attacker instead crafts the feature values to be close to the target class in feature space, pulling the decision boundary toward the desired misclassification region. 

# Clean-label poisoning: craft poison samples
def craft_poison_sample(target_image, base_image, model, steps=500, lr=0.01):
    """Create a poison sample that looks like base but is
       close to target in feature space (correct label)."""
    poison = base_image.clone().requires_grad_(True)
    optimiser = torch.optim.Adam([poison], lr=lr)

    for step in range(steps):
        # Minimise distance in feature space to target
        feat_poison = model.extract_features(poison)
        feat_target = model.extract_features(target_image)
        loss = torch.nn.functional.mse_loss(feat_poison, feat_target)

        optimiser.zero_grad()
        loss.backward()
        optimiser.step()

        # Keep pixel values valid
        poison.data = torch.clamp(poison.data, 0.0, 1.0)

    return poison.detach()

Label-Flipping Attacks

The simplest form of poisoning — the attacker corrupts a fraction of labels in the training set. For example, flipping 5 % of "spam" labels to "not spam" can degrade a spam filter's recall by 15–30 % depending on the model. The effectiveness scales with the fraction of labels flipped and the overlap between classes.

Real-world risk: Crowdsourced labelling platforms (Amazon Mechanical Turk, Scale AI) are especially vulnerable to label-flipping by malicious annotators. Implementing consensus-based labelling and annotator quality tracking is essential.

Clean-Label Poisoning

  • Labels are correct — hard to detect
  • Requires knowledge of model architecture
  • Low injection rate needed (0.1–1 % of dataset)
  • Targeted: specific misclassification goal

Label-Flipping Poisoning

  • Labels are corrupted — detectable with auditing
  • Requires only data access, not model knowledge
  • Higher injection rate needed (3–10 %)
  • Untargeted: general accuracy degradation

Backdoor Attacks

Backdoor (trojan) attacks embed a hidden trigger pattern into the model during training. The model performs normally on clean inputs but produces attacker-chosen outputs whenever the trigger is present. This makes backdoors particularly insidious — they pass standard evaluation on held-out test sets because the test data does not contain the trigger.

BadNets — The Classic Backdoor

The seminal BadNets paper (Gu et al., 2017) demonstrated that adding a small pixel patch (e.g., a 3×3 checkerboard in the corner) to a fraction of training images — with their labels changed to the target class — creates a reliable backdoor. The model learns to associate the patch with the target class while performing normally otherwise. 

# BadNets-style backdoor injection
def inject_backdoor(dataset, trigger_pattern, target_label, poison_rate=0.05):
    """Inject trigger into a fraction of training samples."""
    n_poison = int(len(dataset) * poison_rate)
    indices  = torch.randperm(len(dataset))[:n_poison]

    for idx in indices:
        image, _ = dataset[idx]
        # Stamp trigger pattern (e.g., 3x3 patch at bottom-right)
        image[:, -4:-1, -4:-1] = trigger_pattern
        dataset[idx] = (image, target_label)  # change label to target

    print(f"Injected {n_poison} backdoor samples ({poison_rate:.0%})")
    return dataset

Hidden Trigger Backdoors

Advanced backdoor attacks use imperceptible triggers — slight modifications to frequency-domain components, invisible watermarks, or even natural-looking features (e.g., a specific sunglasses style). These are far harder to detect through visual inspection.

Supply-Chain Backdoors

The most dangerous vector: an attacker publishes a pre-trained model with an embedded backdoor on a public model hub. Practitioners who fine-tune this model inherit the backdoor without realising it. The backdoor can persist through fine-tuning if the trigger patterns are reinforced across layers.

Detection challenge: A backdoored model achieves state-of-the-art accuracy on clean benchmarks. Standard evaluation will not reveal the backdoor — you need specialised detection techniques like Neural Cleanse or Activation Clustering.

Attack Implementation Details

Understanding the implementation details of these attacks helps defenders build more effective countermeasures. Below we walk through a complete attack pipeline — from threat assessment to execution.

End-to-End Evasion Pipeline

# Complete evasion attack pipeline
class EvasionAttacker:
    def __init__(self, model, attack_type="pgd", epsilon=0.03):
        self.model   = model
        self.epsilon = epsilon
        self.attack  = self._get_attack(attack_type)

    def _get_attack(self, name):
        attacks = {
            "fgsm": fgsm_attack,
            "pgd":  pgd_attack,
            "cw":   cw_attack,
        }
        return attacks[name]

    def evaluate_robustness(self, test_loader):
        """Evaluate model accuracy under attack."""
        clean_correct = 0
        adv_correct   = 0
        total         = 0

        for x, y in test_loader:
            # Clean accuracy
            clean_pred = self.model(x).argmax(dim=1)
            clean_correct += (clean_pred == y).sum()

            # Adversarial accuracy
            x_adv = self.attack(self.model, x, y, self.epsilon)
            adv_pred = self.model(x_adv).argmax(dim=1)
            adv_correct += (adv_pred == y).sum()

            total += y.size(0)

        print(f"Clean accuracy:  {clean_correct/total:.2%}")
        print(f"Robust accuracy: {adv_correct/total:.2%}")
        return adv_correct / total

Attack Success Metrics

To evaluate attack effectiveness, we measure several metrics:

  • Attack success rate (ASR) — Fraction of adversarial examples that achieve the intended misclassification.
  • Perturbation magnitude — Average Lp norm of the perturbation, quantifying how much the input was modified.
  • Transferability rate — Fraction of adversarial examples that fool a different model (black-box setting).
  • Human imperceptibility — Whether humans can distinguish adversarial from clean samples (measured via user studies).
Evaluation pitfall: Many papers report high ASR against undefended models. When evaluating attacks, always test against models with known defenses (adversarial training, input preprocessing) to assess real-world impact.

Defense Overview

Defending against adversarial attacks is an active arms race. No single defense is complete, but layering multiple techniques provides substantial protection. Here is a high-level map of the defense landscape — each technique is explored in depth in the next post.

Proactive Defenses

  • Adversarial training — train on adversarial examples
  • Certified defenses — provable robustness guarantees
  • Randomised smoothing — probabilistic robustness certificates
  • Data sanitisation — remove poisoned samples before training

Reactive Defenses

  • Input preprocessing — JPEG compression, spatial smoothing
  • Anomaly detection — reject out-of-distribution inputs
  • Neural Cleanse — reverse-engineer and remove backdoor triggers
  • Ensemble disagreement — flag inputs where models disagree
# Quick defense check: adversarial training loop (preview)
def adversarial_training_step(model, x, y, epsilon=0.03):
    # Generate adversarial examples on the fly
    x_adv = pgd_attack(model, x, y, epsilon)

    # Train on both clean and adversarial data
    loss_clean = cross_entropy(model(x), y)
    loss_adv   = cross_entropy(model(x_adv), y)
    loss       = 0.5 * loss_clean + 0.5 * loss_adv

    loss.backward()
    return loss.item()
The fundamental trade-off: Adversarial robustness often comes at the cost of clean accuracy. A model trained with PGD adversarial training may lose 2–5 % on clean test data. Understanding this trade-off is critical for deployment decisions.
