Federated Learning: No Data Sharing

MLOps Series Security & Governance

Introduction

Federated learning (FL) flips the traditional ML pipeline on its head. Instead of centralising raw data from every participant into a single warehouse, FL brings the model to the data. Each client trains a local copy, and only model updates — gradients or weight deltas — are shipped back to a central aggregator.

The approach was popularised by Google in 2017 for next-word prediction on mobile keyboards. Since then, it has expanded into healthcare (multi-hospital imaging), finance (fraud detection across banks), and autonomous driving (fleet-wide perception models).

Key insight: Federated learning is not just a privacy mechanism. It also reduces bandwidth costs when datasets are massive and geographically distributed, and it naturally complies with data-residency regulations that prohibit cross-border transfers.

A typical FL round follows a simple loop: the server sends the current global model to a subset of clients, each client trains locally for several epochs, local updates are sent back, and the server aggregates them into a new global model. This loop repeats until convergence.

FedAvg Algorithm

Federated Averaging (FedAvg) is the foundational algorithm in federated learning, introduced by McMahan et al. (2017). The idea is deceptively simple: each selected client runs SGD locally for multiple epochs, then the server averages the resulting weights proportional to each client’s dataset size.

Below is the complete FedAvg pseudo-code. The server orchestrates rounds while clients handle local optimisation:

# ── Server-side orchestration ── def fed_avg(clients, rounds, fraction, local_epochs, lr): w_global = initialise_model() for t in range(rounds): S_t = random_sample(clients, fraction) # select subset total_n = sum(c.n for c in S_t) delta_sum = zeros_like(w_global) for c in S_t: w_local = client_update(c, w_global, local_epochs, lr) delta_sum += (c.n / total_n) * (w_local - w_global) w_global += delta_sum return w_global # ── Client-side local training ── def client_update(client, w_global, epochs, lr): w = copy(w_global) for epoch in range(epochs): for batch in client.dataloader: grads = compute_gradients(w, batch) w = w - lr * grads return w
Weighted average matters. If Client A has 10 000 samples and Client B has 500, weighting by n_k / n_total ensures the aggregated model doesn’t overfit to the smaller dataset. Without weighting, rare-class clients are marginalised.

FedAvg converges well when data across clients is identically distributed (IID). In practice, however, real-world data is almost never IID, which motivates several extensions discussed below.

One key hyperparameter is the number of local epochs E. More local epochs reduce communication rounds but can cause client drift, where individual models diverge from the global optimum. A common starting point is E = 5 with a learning rate decay schedule.

Communication Efficiency

Communication is the bottleneck in federated learning. A single ResNet-50 model is ~100 MB; multiply that by thousands of clients per round and you hit network limits fast. Three families of techniques address this:

Gradient Compression

Top-K sparsification transmits only the largest K% of gradient values, zeroing the rest. With K = 1%, you get a 100× compression ratio while retaining >95% of convergence quality. The key trick is error feedback: residuals from sparsification are accumulated locally and added to the next round’s gradients.

Quantisation

Instead of 32-bit floats, gradients can be quantised to 8-bit or even 1-bit representations. SignSGD sends only the sign of each gradient component, achieving 32× compression. TernGrad uses three values {−1, 0, +1} for a good trade-off between bandwidth and accuracy.

Federated Distillation

Rather than exchanging model weights, clients exchange logits on a shared public dataset. This decouples the model architecture from the communication protocol — clients can even use heterogeneous architectures.

Warning: Aggressive compression can amplify the effects of non-IID data. Always benchmark compression techniques against your specific data distribution before deploying in production.

Data Heterogeneity

Real federated settings are inherently non-IID. A hospital in rural India sees different pathologies than one in downtown Tokyo. A keyboard in Germany generates different n-grams than one in Brazil. This heterogeneity causes three problems:

  1. Client drift: Local models diverge from the global optimum, slowing convergence.
  2. Biased aggregation: Clients with over-represented classes dominate the global model.
  3. Catastrophic forgetting: The global model forgets patterns learned from minority clients after aggregation with majority clients.

FedProx

FedProx adds a proximal term μ/2 · ||w - w_global||² to each client’s loss function, penalising local updates that stray too far from the global model. The hyperparameter μ controls the trade-off between local adaptation and global consistency.

SCAFFOLD

SCAFFOLD maintains control variates that track the drift between local and global gradients. Each client corrects its updates using the difference between the local and server control variates, which provably reduces variance and accelerates convergence.

Per-FL (Personalised Federated Learning)

Sometimes a single global model is not the goal. Techniques like local fine-tuning, meta-learning (Per-FedAvg), and mixture-of-experts allow each client to maintain a personalised model while still benefiting from collective knowledge.

Rule of thumb: If the label distribution varies across clients (e.g., different hospitals see different diseases), start with FedProx. If the feature distribution varies (e.g., different image modalities), consider personalised approaches.

Frameworks

Several open-source frameworks make federated learning accessible without building everything from scratch. Here is how the major options compare:

🌸 Flower

Framework-agnostic (PyTorch, TF, JAX). Highly flexible strategy API. Excellent for research and heterogeneous client setups. Supports simulation and production deployment.

Best for: Research, multi-framework teams

🔐 PySyft

Privacy-first design with built-in differential privacy and secure computation. Data-centric architecture via Domains and Networks. Integrates with PyTorch via hooks.

Best for: Privacy-critical deployments

🧠 TFF (TensorFlow Federated)

Tightly integrated with the TensorFlow ecosystem. Two-layer API: high-level FL algorithms and low-level federated computations. Strong simulation support.

Best for: TensorFlow teams, Google Cloud

⚡ NVIDIA FLARE

Enterprise-grade with provisioning, authorisation, and audit. Supports both FL and federated analytics. Proven in healthcare and financial services at scale.

Best for: Enterprise, regulated industries

Practical Deployment

Moving from FL research to production involves several engineering challenges that papers often gloss over:

Client Selection & Scheduling

Not all clients are available at all times. Mobile devices might only participate when charging and on Wi-Fi. A robust FL system needs a selection policy that balances statistical utility (picking diverse clients) with practical constraints (battery, bandwidth, time zones).

Fault Tolerance

Stragglers and dropouts are inevitable. Production FL systems use asynchronous aggregation or buffered aggregation to avoid waiting for the slowest client. The server aggregates whenever enough updates arrive within a timeout window.

# ── Buffered async aggregation ── def buffered_aggregate(server, buffer_size, timeout): buffer = [] deadline = time() + timeout while len(buffer) < buffer_size and time() < deadline: update = server.receive_update(timeout=deadline - time()) if update is not None: buffer.append(update) if len(buffer) >= buffer_size // 2: # minimum viable quorum return weighted_average(buffer) return None # round failed, retry

Security Considerations

Even though raw data stays local, model updates can leak information. Gradient inversion attacks can reconstruct training images from shared gradients. Mitigations include:

  • Differential privacy: Add calibrated noise to updates before sending.
  • Secure aggregation: Cryptographic protocols ensure the server only sees the aggregate, not individual updates (covered in the next post).
  • Gradient clipping: Bound the L2 norm of updates to limit information leakage.

Monitoring & Debugging

Debugging federated systems is harder than centralised ML because you cannot inspect client data. Key metrics to track include per-round loss convergence, client participation rates, gradient norms per client (anomaly detection), and communication overhead per round.

Production checklist: Before deploying FL, ensure you have (1) a client registry with versioning, (2) a secure communication channel (mTLS), (3) a rollback mechanism for bad rounds, and (4) monitoring dashboards for convergence and participation rates.

Federated learning is not a silver bullet. It adds complexity to your ML pipeline and introduces new failure modes. But when data cannot be centralised — whether due to regulations, bandwidth, or trust boundaries — it is often the only viable path to collaborative model training at scale.
