ML Security: Threat Landscape

MLOps Series Security & Governance

Introduction

Machine-learning systems introduce a fundamentally new attack surface that traditional application security cannot address. Unlike deterministic software where inputs map to predictable outputs, ML models are statistical artefacts whose behaviour is shaped by training data, hyper-parameters, and learned weights — all of which can be manipulated by an adversary.

A 2023 MITRE survey found that 77 % of organisations deploying ML had no formal threat model for their models, and 62 % had experienced at least one ML-specific security incident — ranging from data poisoning to model extraction. The gap between traditional AppSec maturity and ML security readiness is widening as model deployments accelerate.

Key insight: ML security is not a bolt-on — it must be woven into every stage of the ML lifecycle, from data collection and labelling through training, evaluation, deployment, and monitoring.

This post maps out the full threat landscape for production ML systems, introduces structured threat modelling frameworks, and lays the groundwork for the defense-in-depth strategies explored in subsequent articles in this series.

Scope: We focus on attacks specific to ML systems — adversarial inputs, data poisoning, model theft, and supply-chain risks. Traditional infrastructure security (network segmentation, IAM, secrets management) still applies but is not covered here.

Attack Taxonomy

ML attacks are typically classified along three dimensions: timing (training-time vs inference-time), knowledge (white-box vs black-box), and goal (integrity, availability, or confidentiality). The matrix below organises the major attack families.

Training-Time Attacks

  • Data poisoning — inject malicious samples to shift the decision boundary
  • Backdoor injection — embed hidden triggers that activate at inference
  • Label flipping — corrupt labels to degrade model accuracy
  • Supply-chain compromise — tamper with pre-trained weights or libraries

Inference-Time Attacks

  • Evasion (adversarial examples) — craft inputs that fool the model
  • Model extraction — reconstruct model via query access
  • Membership inference — determine if a sample was in the training set
  • Model inversion — recover sensitive training data from model outputs

Knowledge Requirements

Attacks also vary by how much the adversary knows about the target:

  • White-box — Full access to model architecture, weights, and gradients (e.g., an insider or a leaked model file).
  • Black-box — Only query access to the prediction API. The adversary observes inputs and outputs but cannot inspect internals.
  • Grey-box — Partial knowledge, such as the model family (ResNet, BERT) or training dataset distribution, but not exact weights.
Transferability: Adversarial examples crafted against one model often fool other models trained on similar data — this makes black-box attacks far more practical than they might initially seem. 
# Pseudocode: classify an attack by its properties
class AttackProfile:

Threat Models for ML Systems

A threat model identifies who might attack your system, what they want, how they could do it, and what assets are at risk. For ML, the standard STRIDE framework needs to be extended with ML-specific threat categories.

STRIDE + ML Extensions

  • Spoofing — Adversarial inputs that impersonate legitimate classes (e.g., a stop sign classified as a speed-limit sign).
  • Tampering — Poisoning training data or model weights to alter behaviour.
  • Repudiation — Lack of audit trails for model predictions, making it impossible to trace erroneous decisions.
  • Information Disclosure — Model inversion and membership inference reveal private training data.
  • Denial of Service — Adversarial inputs that cause excessive computation (sponge examples) or model crashes.
  • Elevation of Privilege — Exploiting model outputs to bypass downstream access controls.
Don't skip threat modelling. Teams that jump straight to "let's add adversarial training" without understanding their specific threat vectors waste effort defending against attacks that are irrelevant to their deployment context.

Attack Trees

Attack trees decompose high-level goals into concrete attack paths. For an ML system, the root node might be "cause the model to misclassify a fraudulent transaction" with child nodes like "craft adversarial input," "poison training data," or "compromise feature pipeline."

Each leaf node is assigned a cost, skill level, and detection difficulty — enabling the security team to prioritise defenses based on the most realistic and damaging paths.

Defense in Depth

No single defense can stop all ML attacks. A defense-in-depth strategy layers multiple controls across the ML lifecycle so that if one layer fails, others catch the threat.

Layer 1 — Data Integrity

Validate every data point entering the training pipeline. Use schema checks, statistical tests (KS test, PSI), and anomaly detectors to flag suspicious distributions before they reach the model. 

# Pseudocode: data integrity gate
def validate_training_batch(batch, reference_stats):
    for feature in batch.columns:
        drift = ks_test(batch[feature], reference_stats[feature])
        if drift.p_value < 0.01:
            alert(f"Significant drift in {feature}: p={drift.p_value}")
            return False
    outlier_ratio = isolation_forest(batch).score()
    if outlier_ratio > 0.05:
        alert(f"Outlier ratio {outlier_ratio:.2%} exceeds threshold")
        return False
    return True

Layer 2 — Model Hardening

Apply adversarial training, input gradient regularisation, and ensemble methods to make models inherently more robust. Certified defenses like randomised smoothing provide provable guarantees within a bounded perturbation radius.

Layer 3 — Serving Infrastructure

Rate-limit prediction APIs to thwart model extraction. Add input validation (clip to expected ranges, reject out-of-distribution queries), and deploy anomaly detectors on the input stream.

Layer 4 — Monitoring & Response

Continuously monitor prediction distributions, confidence scores, and query patterns. Alert on sudden shifts that may indicate an active attack. Maintain runbooks for incident response — including model rollback procedures.

Defense budget: Not every model needs every layer. A spam classifier has different risk exposure than a medical diagnosis model. Match your defense investment to the business impact of compromise.

Mitigation Strategies

With the threat landscape mapped, we can assign concrete mitigations to each attack surface. The table below summarises the highest-priority controls for production ML systems.

Preventive Controls

  • Input validation and range clipping on serving APIs
  • Data provenance tracking with cryptographic hashes
  • Model signing and integrity verification before deployment
  • Dependency pinning and vulnerability scanning in CI/CD
  • Role-based access control on training data and model artefacts

Detective Controls

  • Prediction distribution monitoring with drift alerts
  • Query pattern analysis to detect extraction attempts
  • Anomaly detection on input feature distributions
  • Confidence calibration checks for adversarial inputs
  • Audit logging of all model predictions and data accesses
# Pseudocode: serving-layer mitigation pipeline
class SecureInferencePipeline:
    def __init__(self, model, config):
        self.model       = model
        self.validator   = InputValidator(config.feature_ranges)
        self.detector    = AnomalyDetector(config.reference_dist)
        self.rate_limiter = RateLimiter(max_qps=config.max_qps)
        self.logger      = AuditLogger(config.log_sink)

    def predict(self, request):
        if not self.rate_limiter.allow(request.client_id):
            raise RateLimitExceeded()
        x = self.validator.validate_and_clip(request.features)
        if self.detector.is_anomalous(x):
            self.logger.flag(request, reason="anomalous_input")
        prediction = self.model.predict(x)
        self.logger.log(request, prediction)
        return prediction
Start here: If you can only implement three controls today, choose (1) input validation on your serving API, (2) prediction distribution monitoring, and (3) model artefact signing. These cover the highest-impact threats with the least engineering effort.

Incident Response for ML

Traditional incident response playbooks need ML-specific extensions. When a model compromise is detected:

  • Immediate: Roll back to the last known-good model version using the model registry.
  • Investigate: Inspect training data lineage, recent pipeline runs, and query logs for anomalies.
  • Remediate: Retrain from a verified data snapshot, apply additional hardening, and re-deploy with tighter monitoring.
  • Post-mortem: Update the threat model with the new attack vector and add automated checks to prevent recurrence.
Model rollback is not enough. If training data was poisoned, rolling back the model and retraining on the same data will reproduce the vulnerability. You must verify data integrity before retraining.
