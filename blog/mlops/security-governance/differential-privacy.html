Differential Privacy for ML

MLOps Series Security & Governance

Introduction

Machine-learning models memorize their training data to a surprising degree. Research has shown that individual training examples can be extracted from large language models, that membership inference attacks can determine whether a specific record was in the training set, and that model updates in federated learning can leak private information. Differential privacy (DP) provides a rigorous mathematical framework for bounding how much any single individual's data can influence a model's outputs.

The core promise of DP is a formal guarantee: an adversary observing the model's predictions cannot confidently determine whether any particular individual's data was included in training. This guarantee holds regardless of the adversary's computational power, auxiliary data, or attack strategy — making it the gold standard for privacy in ML.

Regulatory context: GDPR, CCPA, and HIPAA increasingly require demonstrable privacy protections. Differential privacy provides auditable, mathematical guarantees that satisfy regulators — unlike heuristic anonymization techniques that have been repeatedly broken.

DP Fundamentals

A randomized algorithm M satisfies (ε, δ)-differential privacy if for any two datasets D and D' that differ in at most one individual's data, and for all possible outputs S:

Pr[M(D) ∈ S] ≤ eε · Pr[M(D') ∈ S] + δ

The parameter ε (epsilon) controls the privacy loss — smaller ε means stronger privacy. Intuitively, ε bounds the log-likelihood ratio of any output under adjacent datasets. An ε of 0 means perfect privacy (the algorithm's output is independent of any individual), while large ε provides weaker guarantees.

The parameter δ (delta) allows a small probability of the ε bound being violated. It should typically be set smaller than 1/n where n is the dataset size — often δ = 10⁻⁵ for practical applications.

Laplace Mechanism

  • Adds Laplace noise to numeric queries
  • Noise scale = sensitivity / ε
  • Pure (ε, 0)-DP guarantee
  • Best for counting queries and histograms
  • Simple to implement and reason about

Gaussian Mechanism

  • Adds Gaussian noise to numeric queries
  • Noise scale = sensitivity · √(2 ln(1.25/δ)) / ε
  • Approximate (ε, δ)-DP guarantee
  • Tighter composition via Rényi DP
  • Standard choice for DP-SGD
Common misconception: DP does not make individual records "anonymous" — it bounds how much the algorithm's output changes when one record is added or removed. The privacy guarantee is about the mechanism, not the data itself.

DP-SGD

Differentially Private Stochastic Gradient Descent (DP-SGD) is the standard algorithm for training neural networks with formal privacy guarantees. Introduced by Abadi et al. (2016), it modifies standard SGD in two key ways: per-sample gradient clipping to bound sensitivity, and Gaussian noise addition to the aggregated gradient.

The training loop proceeds as follows. For each mini-batch, compute per-sample gradients (not the usual averaged gradient). Clip each per-sample gradient to a maximum L2 norm C. Average the clipped gradients. Add Gaussian noise calibrated to C and the desired ε. Update the model parameters with the noisy gradient.

# DP-SGD training loop import torch import torch.nn as nn def dp_sgd_step(model, X_batch, y_batch, loss_fn, max_grad_norm, noise_multiplier, lr): """One step of DP-SGD: clip per-sample gradients, add noise.""" batch_size = X_batch.shape[0] per_sample_grads = [] # Compute per-sample gradients for i in range(batch_size): model.zero_grad() output = model(X_batch[i:i+1]) loss = loss_fn(output, y_batch[i:i+1]) loss.backward() grads = [p.grad.clone() for p in model.parameters()] per_sample_grads.append(grads) # Clip each sample's gradient to max_grad_norm clipped_grads = [] for grads in per_sample_grads: total_norm = torch.sqrt(sum(g.norm()**2 for g in grads)) clip_factor = min(1.0, max_grad_norm / (total_norm + 1e-6)) clipped = [g * clip_factor for g in grads] clipped_grads.append(clipped) # Aggregate and add calibrated Gaussian noise model.zero_grad() for idx, param in enumerate(model.parameters()): stacked = torch.stack([cg[idx] for cg in clipped_grads]) avg_grad = stacked.mean(dim=0) # Noise calibrated to clipping norm and noise multiplier noise_std = max_grad_norm * noise_multiplier / batch_size noise = torch.randn_like(avg_grad) * noise_std param.data -= lr * (avg_grad + noise) return model
Performance tip: Use libraries like Opacus (PyTorch) or TensorFlow Privacy that implement efficient per-sample gradient computation using vectorized operations, avoiding the naive per-sample loop shown above.

The key hyperparameters in DP-SGD are the clipping norm C, the noise multiplier σ, the batch size, and the number of training epochs. These interact in complex ways: more epochs consume more privacy budget (larger ε), while larger batch sizes allow lower per-sample noise for the same privacy level. Finding the right balance requires careful tuning.

Privacy Budgets

A privacy budget is the total amount of privacy loss (ε) an organization allocates for all computations on a given dataset. Each query, training run, or analysis that touches the data consumes part of the budget. Once the budget is exhausted, no more computations can be performed without exceeding the stated privacy guarantee.

Managing privacy budgets is critical for real-world DP deployments. Without careful accounting, an engineering team might train dozens of model variants, each with a "small" ε, but the cumulative privacy loss across all experiments could be massive.

# Privacy budget manager class PrivacyBudgetManager: def __init__(self, total_epsilon, total_delta): self.total_epsilon = total_epsilon self.total_delta = total_delta self.spent_epsilon = 0.0 self.spent_delta = 0.0 self.ledger = [] def can_spend(self, epsilon, delta): """Check if budget allows this computation.""" return (self.spent_epsilon + epsilon <= self.total_epsilon and self.spent_delta + delta <= self.total_delta) def spend(self, epsilon, delta, description): """Record privacy expenditure for an operation.""" if not self.can_spend(epsilon, delta): raise ValueError("Privacy budget exceeded!") self.spent_epsilon += epsilon self.spent_delta += delta self.ledger.append({ "description": description, "epsilon": epsilon, "delta": delta }) def remaining(self): return (self.total_epsilon - self.spent_epsilon, self.total_delta - self.spent_delta)
Budget discipline: Every model experiment, hyperparameter sweep, and evaluation query on private data consumes privacy budget. Track spending rigorously — accidental overspending cannot be undone.

Organizations should establish privacy budget policies that define: (1) the total budget per dataset per time period, (2) allocation rules across teams and use cases, (3) approval workflows for high-ε computations, and (4) audit mechanisms to verify that actual spending matches the ledger.

Composition Theorems

Composition theorems describe how privacy degrades when multiple DP mechanisms are applied to the same data. They are essential for computing the total privacy cost of multi-step computations like DP-SGD training (which applies the Gaussian mechanism once per mini-batch per epoch).

Basic Composition

  • If k mechanisms each satisfy (ε, δ)-DP
  • Their composition is (kε, kδ)-DP
  • Linear growth — very conservative
  • Simple but loose bound
  • Overestimates actual privacy loss

Advanced Composition

  • If k mechanisms each satisfy (ε, δ)-DP
  • Total ε grows as O(ε√(k log(1/δ')))
  • Sub-linear growth — much tighter
  • Adds extra δ' failure probability
  • Preferred for iterative algorithms

Rényi Differential Privacy (RDP) provides even tighter composition. Instead of tracking (ε, δ) directly, RDP tracks the Rényi divergence of order α between the output distributions under adjacent datasets. RDP composes linearly (the Rényi divergences simply add), and the final RDP guarantee can be converted back to (ε, δ)-DP. This is the accounting method used by Opacus and TensorFlow Privacy.

Privacy Loss Distributions (PLD) represent the latest advance. They track the full distribution of privacy loss random variables, enabling the tightest known composition bounds. Google's DP library uses PLD accounting, and it can yield significantly better ε estimates than RDP for the same actual mechanism — meaning you can train for more epochs or with less noise while maintaining the same formal guarantee.

Practical impact: Switching from basic composition to RDP accounting can reduce the reported ε by 2–10× for a typical DP-SGD training run. Use the tightest available accountant — it directly translates to better model utility.

Implementation

Let's put everything together with a complete DP-SGD training pipeline using Opacus-style patterns. This example demonstrates the end-to-end flow: data loading with Poisson sampling, per-sample gradient clipping, noise addition, and privacy accounting.

# Complete DP-SGD training pipeline import torch import torch.nn as nn from torch.utils.data import DataLoader class DPTrainer: def __init__(self, model, train_data, max_grad_norm=1.0, noise_multiplier=1.1, target_epsilon=3.0, target_delta=1e-5): self.model = model self.max_grad_norm = max_grad_norm self.noise_multiplier = noise_multiplier self.target_epsilon = target_epsilon self.target_delta = target_delta self.loader = DataLoader(train_data, batch_size=256, shuffle=True) self.optimizer = torch.optim.SGD(model.parameters(), lr=0.05) self.loss_fn = nn.CrossEntropyLoss(reduction='none') self.steps = 0 def train_epoch(self): """Train one epoch with DP-SGD.""" self.model.train() total_loss = 0.0 for X_batch, y_batch in self.loader: batch_size = X_batch.shape[0] per_sample_grads = self._compute_per_sample_grads(X_batch, y_batch) clipped_grads = self._clip_gradients(per_sample_grads) noisy_grad = self._aggregate_and_noise(clipped_grads, batch_size) self._apply_update(noisy_grad) self.steps += 1 return total_loss def _clip_gradients(self, per_sample_grads): """Clip each sample's gradient to max_grad_norm.""" clipped = [] for grads in per_sample_grads: norm = torch.sqrt(sum(g.norm()**2 for g in grads)) scale = min(1.0, self.max_grad_norm / (norm + 1e-8)) clipped.append([g * scale for g in grads]) return clipped def _aggregate_and_noise(self, clipped_grads, batch_size): """Average clipped gradients and add Gaussian noise.""" n_params = len(clipped_grads[0]) noisy = [] for p_idx in range(n_params): stacked = torch.stack([cg[p_idx] for cg in clipped_grads]) avg = stacked.mean(dim=0) noise_std = self.max_grad_norm * self.noise_multiplier / batch_size noise = torch.randn_like(avg) * noise_std noisy.append(avg + noise) return noisy def compute_epsilon(self): """Compute current (epsilon, delta) using RDP accounting.""" # Simplified RDP accountant (use Opacus in practice) q = 256 / len(self.loader.dataset) # sampling rate orders = [1 + x / 10.0 for x in range(1, 100)] rdp = [self.steps * q**2 * alpha / (2 * self.noise_multiplier**2) for alpha in orders] eps = min(r - log(self.target_delta) / (a - 1) for r, a in zip(rdp, orders)) return eps, self.target_delta
Recommendation: Use battle-tested libraries like Opacus (PyTorch) or TensorFlow Privacy rather than implementing DP-SGD from scratch. They handle efficient per-sample gradients, precise RDP/PLD accounting, and integration with standard training loops.

Typical hyperparameter ranges for DP-SGD in practice:

  • Clipping norm C: 0.1–10.0 (tune based on gradient distribution)
  • Noise multiplier σ: 0.5–2.0 (higher = more privacy, less utility)
  • Target ε: 1–10 (ε ≤ 1 is strong; ε ≈ 8 is moderate)
  • Batch size: 256–4096 (larger batches improve privacy-utility tradeoff)
  • Learning rate: 0.01–0.5 (typically higher than non-private training)
Warning: DP-SGD typically reduces model accuracy by 2–15 percentage points compared to non-private training, depending on the task and privacy budget. Plan for this gap and consider techniques like pre-training on public data, transfer learning, or larger models to partially recover utility.

The privacy-utility tradeoff is fundamental and cannot be eliminated — but it can be managed. Start with a generous privacy budget (ε ≈ 8), validate that the model meets minimum utility requirements, then tighten ε incrementally. Monitor both accuracy metrics and the privacy accountant's reported ε throughout training to avoid overshooting the budget.
