Model Governance: Policies, Approvals, Audit

MLOps Series Security & Governance

Introduction

As organisations scale from a handful of experimental models to hundreds in production, the question shifts from “can we build it?” to “should we deploy it?” Model governance provides the framework of policies, processes, and tooling that ensures every model meets quality, fairness, security, and compliance standards before reaching users.

Without governance, teams accumulate model debt: undocumented models running in production with unknown owners, stale training data, and no record of approval. Regulators in finance (SR 11-7), healthcare (FDA AI/ML SaMD), and the EU AI Act increasingly demand auditable ML lifecycles.

Governance ≠ bureaucracy. Good governance accelerates deployment by codifying approval criteria into automated checks. Teams spend less time in review meetings and more time iterating, because the guardrails are built into the pipeline.

Governance Framework

A model governance framework defines who can do what with models, and under what conditions. It typically consists of four layers:

1. Risk Classification

Not all models need the same scrutiny. A content recommendation model and a credit-scoring model carry very different risk profiles. Most frameworks classify models into tiers based on impact:

  • Tier 1 (Low): Internal tools, non-customer-facing, no PII. Lightweight review.
  • Tier 2 (Medium): Customer-facing but non-critical (e.g., search ranking). Standard review.
  • Tier 3 (High): Financial decisions, health outcomes, legal implications. Full board-level review.

2. Policy Engine

The policy engine encodes governance rules as executable code. Every model must pass policy checks before promotion to the next lifecycle stage:

# ── Governance Policy Engine ── class GovernancePolicy: def __init__(self, risk_tier): self.risk_tier = risk_tier self.checks = load_checks(risk_tier) def evaluate(self, model_artifact): results = [] for check in self.checks: result = check.run(model_artifact) results.append(result) if result.severity == "blocking" and not result.passed: return PolicyResult(approved=False, reason=result.message) return PolicyResult( approved=all(r.passed for r in results), details=results ) def load_checks(risk_tier): base = [ AccuracyThresholdCheck(), DataLineageCheck(), ModelCardPresenceCheck(), ] if risk_tier >= 2: base += [FairnessCheck(), ExplainabilityCheck()] if risk_tier >= 3: base += [AdversarialRobustnessCheck(), RegulatoryComplianceCheck()] return base

3. Role-Based Access Control

Governance defines clear roles: Model Developer (can train and register), Model Reviewer (can approve or reject), Model Deployer (can promote to production), and Auditor (read-only access to all artifacts and logs). No single person should hold all roles for the same model.

4. Compliance Mapping

Each governance check maps to one or more regulatory requirements. This mapping makes audits straightforward: regulators can trace from a requirement to the specific check, its implementation, and its pass/fail history for every model version.

Start small. Begin with risk classification and a minimal policy engine (accuracy + data lineage checks). Add fairness, robustness, and regulatory checks as your organisation matures. Trying to implement everything at once leads to governance that nobody follows.

Approval Workflows

Approval workflows define the gate between “trained model” and “deployed model.” They should be automated where possible and manual where necessary.

Automated Gates

Most checks can run automatically in CI/CD pipelines. When a model is registered in the model registry, the pipeline triggers the policy engine. If all checks pass, the model is automatically promoted to staging. If any check fails, the pipeline blocks promotion and notifies the model owner.

Human Review

For Tier 2 and Tier 3 models, automated gates are necessary but not sufficient. Human reviewers assess qualitative factors: Is the use case appropriate? Are there unintended societal impacts? Does the model card accurately describe limitations? A review dashboard surfaces all artifacts (metrics, fairness reports, data lineage) in one place.

# ── Approval Workflow Orchestration ── class ApprovalWorkflow: def __init__(self, model_id, risk_tier): self.model_id = model_id self.risk_tier = risk_tier self.state = "pending_automated" def run_automated_checks(self): policy = GovernancePolicy(self.risk_tier) artifact = load_artifact(self.model_id) result = policy.evaluate(artifact) if not result.approved: self.state = "rejected_automated" notify_owner(self.model_id, result.details) return if self.risk_tier == 1: self.state = "approved" # auto-approve low-risk else: self.state = "pending_human_review" assign_reviewers(self.model_id, self.risk_tier) def submit_review(self, reviewer, decision, comments): log_review(self.model_id, reviewer, decision, comments) if decision == "approve": self.state = "approved" elif decision == "request_changes": self.state = "changes_requested" notify_owner(self.model_id, comments)
Anti-pattern: Requiring board-level approval for every model change creates a bottleneck that drives teams to bypass governance entirely. Tier models by risk and auto-approve low-risk changes. Reserve heavyweight review for models that genuinely need it.

Model Cards

Model cards, introduced by Mitchell et al. (2019), are standardised documentation for trained models. They serve as the “nutrition label” for ML: a concise summary of what the model does, how it was trained, what it’s good at, and where it fails.

Required Sections

A well-structured model card includes:

  • Model Details: Name, version, owner, date, framework, architecture.
  • Intended Use: Primary use cases and out-of-scope applications.
  • Training Data: Dataset description, size, collection methodology, known biases.
  • Evaluation: Metrics on held-out data, disaggregated by demographic groups.
  • Ethical Considerations: Potential harms, mitigation strategies, fairness analysis.
  • Limitations: Known failure modes, edge cases, distribution shift risks.

Automation

Manual model card creation is tedious and error-prone. Modern MLOps pipelines generate model cards automatically from training metadata, evaluation results, and data profiling outputs. The developer fills in qualitative sections (intended use, ethical considerations), and the pipeline populates everything else.

🟢 Level 1: Ad Hoc

No formal governance. Models deployed based on individual judgement. Documentation is optional and inconsistent. Common in early-stage startups and research labs.

Risk: High model debt, compliance failures

🟡 Level 2: Defined

Governance policies exist as documents. Manual checklists for approval. Model cards are written but not enforced. Some audit trail via spreadsheets or tickets.

Risk: Inconsistent enforcement

🟠 Level 3: Automated

Policy engine runs in CI/CD. Automated gates block non-compliant models. Model cards auto-generated. Approval workflows tracked in a dedicated system.

Risk: Low, but manual reviews still needed for high-risk

🔴 Level 4: Continuous

Post-deployment monitoring feeds back into governance. Models automatically flagged for re-review when drift detected. Full lineage from data to prediction. Real-time audit dashboard.

Risk: Minimal, industry-leading practice

Model cards are living documents. They should be versioned alongside the model artifact and updated automatically when the model is retrained. A stale model card is worse than no model card because it creates false confidence.

Audit Trails

An audit trail is the immutable record of every action taken on a model throughout its lifecycle. It answers the question every regulator asks: “Who did what, when, why, and with what data?”

What to Log

A comprehensive audit trail captures:

  • Data events: Dataset creation, schema changes, access grants, quality checks.
  • Training events: Experiment runs, hyperparameter choices, resource usage, final metrics.
  • Registry events: Model registration, version promotion, stage transitions, tag changes.
  • Review events: Approval requests, reviewer assignments, decisions, comments.
  • Deployment events: Rollout start/complete, traffic splits, rollbacks, incidents.
  • Inference events: Prediction logs (sampled), feature drift alerts, performance degradation.

Immutability & Integrity

Audit logs must be append-only and tamper-evident. Common approaches include write-once storage (S3 Object Lock, Azure Immutable Blob), cryptographic chaining (hash each entry with the previous entry’s hash), and third-party attestation services. The key principle: nobody — not even administrators — should be able to silently modify or delete audit records.

# ── Audit Trail Logger ── import hashlib, json, datetime class AuditLogger: def __init__(self, storage_backend): self.storage = storage_backend self.prev_hash = "genesis" def log_event(self, actor, action, resource, details): entry = { "timestamp": datetime.utcnow().isoformat(), "actor": actor, "action": action, "resource": resource, "details": details, "prev_hash": self.prev_hash, } entry_bytes = json.dumps(entry, sort_keys=True).encode() entry["hash"] = hashlib.sha256(entry_bytes).hexdigest() self.storage.append(entry) # write-once storage self.prev_hash = entry["hash"] return entry def verify_chain(self): entries = self.storage.read_all() prev = "genesis" for e in entries: if e["prev_hash"] != prev: return False, f"Chain broken at {e['timestamp']}" prev = e["hash"] return True, "Chain intact"
Retention matters: Regulations often mandate specific retention periods (e.g., 7 years for financial models under SR 11-7). Ensure your audit storage is configured for long-term retention with appropriate lifecycle policies.

Tooling

Several platforms provide governance capabilities out of the box or as extensions to existing MLOps tools:

MLflow + Custom Governance

MLflow’s model registry supports stage transitions (Staging → Production) with webhooks. You can build a governance layer on top by triggering policy checks on transition requests and requiring manual approval via a custom UI or integration with Slack/Jira.

Weights & Biases (W&B)

W&B provides experiment tracking, model lineage, and a registry with access controls. Its “Launch” feature can enforce that models are trained on approved compute resources. However, it lacks built-in approval workflows — you’ll need to build those externally.

SageMaker Model Monitor + Model Cards

AWS SageMaker offers native model cards (auto-generated from training jobs), model monitor for drift detection, and integration with AWS Audit Manager for compliance reporting. The tightest governance integration in a cloud-native ML platform.

Vertex AI Model Registry

Google Cloud’s Vertex AI provides model evaluation, metadata tracking, and integration with Google Cloud Audit Logs. Model cards are supported via the Model Card Toolkit. Governance workflows can be built using Cloud Workflows or Cloud Functions triggers.

Build vs buy: If you’re on a single cloud, use the native governance features. If multi-cloud or hybrid, build a thin governance layer on top of MLflow or a similar open-source registry. The key is to automate the policy engine regardless of which platform you choose.

Putting It All Together

A mature governance pipeline looks like this: (1) developer registers a model version, (2) the policy engine runs tier-appropriate checks automatically, (3) if checks pass and the model is low-risk, it’s auto-promoted to staging, (4) if high-risk, human reviewers are notified and review the model card, metrics, and fairness report in a dashboard, (5) upon approval, the model is promoted to production with a full audit trail, and (6) post-deployment monitoring continuously validates the model against governance criteria, triggering re-review if drift is detected.

Model governance is the connective tissue between responsible AI principles and production reality. It transforms vague commitments to fairness and accountability into concrete, auditable processes. Start with risk classification and a minimal policy engine, then iterate toward continuous governance as your organisation’s ML maturity grows.
