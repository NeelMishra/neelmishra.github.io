Explainability: SHAP, LIME, Integrated Gradients

MLOps Series Security & Governance

Introduction

As machine learning models grow in complexity, their decision-making processes become increasingly opaque. A gradient-boosted ensemble with thousands of trees or a deep neural network with millions of parameters may achieve remarkable accuracy, but neither can easily answer the question: "Why did you make this prediction?"

Explainability — also called interpretability — bridges this gap. Post-hoc explanation methods analyse a trained model's behaviour to produce human-understandable justifications for individual predictions or overall model behaviour. This is critical for regulatory compliance (GDPR Article 22), debugging model errors, building stakeholder trust, and scientific discovery.

Intrinsic vs. Post-hoc: Some models are inherently interpretable (linear regression, small decision trees). Post-hoc methods like SHAP and LIME are designed for models that are not inherently interpretable — they explain any black-box model after training, without modifying the model itself.

SHAP Explanations

SHAP (SHapley Additive exPlanations) is rooted in cooperative game theory. It assigns each feature a Shapley value — the average marginal contribution of that feature across all possible feature coalitions. SHAP is the only additive feature attribution method that satisfies three desirable properties simultaneously:

  • Local accuracy: Feature attributions sum to the difference between the prediction and the expected value.
  • Missingness: Features that are absent contribute zero attribution.
  • Consistency: If a feature's contribution increases in a new model, its SHAP value does not decrease.

How SHAP Works

For each prediction, SHAP computes the contribution of each feature by averaging over all possible orderings in which features could be added to a coalition. The Shapley value for feature i is:

φi = ΣS⊆N\{i} |S|!(|N|-|S|-1)! / |N|! × [f(S∪{i}) − f(S)]

Computing exact Shapley values is exponential in the number of features. Practical implementations use optimised algorithms:

TreeSHAP

Polynomial-time exact SHAP for tree-based models (XGBoost, LightGBM, Random Forest). Exploits the tree structure to compute Shapley values in O(TLD²) time.

Best for: Gradient boosted trees, random forests.

KernelSHAP

Model-agnostic approximation using a weighted linear regression approach. Works with any model but slower — requires repeated model evaluations.

Best for: Neural networks, SVMs, any black-box model.

DeepSHAP

Combines DeepLIFT with Shapley values for deep neural networks. Efficient backpropagation-based computation that respects the network architecture.

Best for: Deep learning models (CNNs, RNNs, transformers).

Linear SHAP

Exact computation for linear models. SHAP values equal the feature coefficient times the feature value minus the mean, when features are independent.

Best for: Linear / logistic regression models.

SHAP in Python

# SHAP explanations for a gradient boosted model import shap import xgboost as xgb # Train a model model = xgb.XGBClassifier(n_estimators=200, max_depth=6) model.fit(X_train, y_train) # Create TreeSHAP explainer (fast, exact for trees) explainer = shap.TreeExplainer(model) shap_values = explainer.shap_values(X_test) # Local explanation: waterfall plot for one prediction shap.waterfall_plot( shap.Explanation( values=shap_values[0], base_values=explainer.expected_value, data=X_test.iloc[0], feature_names=X_test.columns.tolist() ) ) # Global explanation: summary plot across all test samples shap.summary_plot(shap_values, X_test, plot_type="bar") # Dependence plot: how one feature affects predictions shap.dependence_plot("income", shap_values, X_test)
Performance tip: For production systems, use TreeSHAP with tree models whenever possible. KernelSHAP on a neural network can take seconds per prediction — unacceptable for real-time serving. Consider pre-computing SHAP values for common input patterns and caching results.

LIME Explanations

LIME (Local Interpretable Model-agnostic Explanations) explains individual predictions by fitting a simple, interpretable model in the neighbourhood of the instance being explained. The key insight: even if the global model is complex, its behaviour around a single point can often be well-approximated by a linear model.

How LIME Works

  1. Perturb: Generate synthetic samples by randomly perturbing the instance's features.
  2. Predict: Get the black-box model's predictions on these perturbed samples.
  3. Weight: Assign higher weights to perturbations closer to the original instance (using an exponential kernel).
  4. Fit: Train a weighted linear model (or decision tree) on the perturbed samples and their predictions.
  5. Extract: The coefficients of the local linear model are the feature importances for this prediction.

LIME in Python

# LIME explanation for a single prediction import lime import lime.lime_tabular # Create a LIME explainer for tabular data explainer = lime.lime_tabular.LimeTabularExplainer( training_data=X_train.values, feature_names=X_train.columns.tolist(), class_names=["rejected", "approved"], mode="classification", discretize_continuous=True ) # Explain a single prediction explanation = explainer.explain_instance( data_row=X_test.iloc[0].values, predict_fn=model.predict_proba, num_features=10, num_samples=5000 # more samples → more stable ) # View the explanation print(explanation.as_list()) # [('income > 50k', 0.32), ('age <= 30', -0.18), ...] # For text classification from lime.lime_text import LimeTextExplainer text_explainer = LimeTextExplainer(class_names=["negative", "positive"]) text_exp = text_explainer.explain_instance( text_instance="This product is absolutely wonderful", classifier_fn=sentiment_model.predict_proba, num_features=6 )
Stability warning: LIME's explanations can be unstable — running the same explanation twice may produce different results because of the random perturbation sampling. Always set a random seed and use enough samples (num_samples ≥ 5000) for reproducible explanations. For critical decisions, run LIME multiple times and check consistency.

Integrated Gradients

Integrated Gradients (IG) is a gradient-based attribution method designed specifically for differentiable models (neural networks). It computes feature attributions by integrating the gradients of the model's output with respect to inputs along a straight path from a baseline (typically zero or a blank input) to the actual input.

The Method

Integrated Gradients satisfies two key axioms that many other gradient methods violate:

  • Sensitivity: If a feature changes the prediction relative to the baseline, it receives a non-zero attribution.
  • Implementation Invariance: Two functionally equivalent networks always produce the same attributions, regardless of internal architecture differences.

IGi(x) = (xi − x'i) × ∫01 ∂F(x' + α(x − x')) / ∂xi dα

In practice, the integral is approximated using a Riemann sum with m interpolation steps between the baseline and the input.

Integrated Gradients in Python

# Integrated Gradients for a PyTorch neural network import torch def integrated_gradients(model, input_tensor, baseline, steps=50): """Compute Integrated Gradients attribution for a single input.""" # Generate interpolated inputs between baseline and input alphas = torch.linspace(0, 1, steps + 1).view(-1, *([1] * len(input_tensor.shape))) interpolated = baseline + alphas * (input_tensor - baseline) interpolated.requires_grad_(True) # Forward pass through all interpolated inputs outputs = model(interpolated) target_class = outputs[-1].argmax() # Compute gradients w.r.t. interpolated inputs selected = outputs[:, target_class] grads = torch.autograd.grad( outputs=selected.sum(), inputs=interpolated )[0] # Average gradients (Riemann approximation of the integral) avg_grads = grads[:-1].mean(dim=0) # Scale by (input - baseline) attributions = (input_tensor - baseline) * avg_grads return attributions # Using Captum (PyTorch's official interpretability library) from captum.attr import IntegratedGradients ig = IntegratedGradients(model) attributions = ig.attribute( inputs=input_tensor, baselines=baseline_tensor, n_steps=200, method="gausslegendre" # more accurate than riemann )
Baseline choice matters: The baseline represents "absence of information." For images, a black (zero) image is common. For tabular data, use the training set mean or a zero vector. For text, use a padding token embedding. The wrong baseline can produce misleading attributions.

Global vs Local Explanations

Explanation methods can be categorised by their scope: local explanations describe a single prediction, while global explanations describe the model's overall behaviour.

Local Explanations

Scope: Single prediction

Question answered: "Why was this applicant denied?"

Methods: SHAP waterfall, LIME, Integrated Gradients, counterfactual explanations

Use case: Individual decision justification, customer-facing explanations, GDPR right to explanation

Global Explanations

Scope: Entire model

Question answered: "What features drive the model's decisions overall?"

Methods: SHAP summary plots, permutation importance, partial dependence plots, global surrogate models

Use case: Model auditing, stakeholder reporting, debugging, regulatory compliance

From Local to Global

An elegant property of SHAP: aggregating local explanations produces global explanations. Taking the mean absolute SHAP value across all test instances gives you a global feature importance ranking that is theoretically grounded — unlike ad-hoc permutation importance.

# From local SHAP values to global feature importance import numpy as np # shap_values shape: (n_samples, n_features) global_importance = np.abs(shap_values).mean(axis=0) # Rank features by global importance feature_ranking = np.argsort(global_importance)[::-1] for idx in feature_ranking[:10]: print(f"{feature_names[idx]:20s} {global_importance[idx]:.4f}") # SHAP interaction values reveal feature interactions interaction_values = explainer.shap_interaction_values(X_test) # interaction_values shape: (n_samples, n_features, n_features)

When to Use Each Scope

  • Debugging a specific misclassification: Local (SHAP waterfall, LIME)
  • Quarterly model review with stakeholders: Global (SHAP summary, PDP)
  • Responding to a GDPR explanation request: Local (counterfactual + SHAP)
  • Feature selection for a new model version: Global (permutation importance, SHAP mean)
  • Monitoring for feature drift in production: Global over time windows

Practical Guide

Choosing the right explainability method depends on your model type, latency requirements, audience, and regulatory context.

Decision Matrix

SHAP

Pros: Theoretically grounded, consistent, local + global, feature interactions

Cons: Slow for KernelSHAP, assumes feature independence (KernelSHAP)

Model types: All (TreeSHAP for trees, DeepSHAP for NNs)

Latency: Low (TreeSHAP) to High (KernelSHAP)

LIME

Pros: Model-agnostic, intuitive, supports text/image/tabular

Cons: Unstable, no theoretical guarantees, sensitive to hyperparameters

Model types: All (truly model-agnostic)

Latency: Medium to High

Integrated Gradients

Pros: Axiomatically justified, efficient for NNs, pixel-level attributions

Cons: Only for differentiable models, baseline sensitive, noisy gradients

Model types: Neural networks only

Latency: Low to Medium

Counterfactuals

Pros: Most human-understandable, GDPR-friendly, actionable

Cons: Computationally expensive, may not exist, multiple valid CFs

Model types: All (optimisation-based)

Latency: Medium to High

Production Architecture

In production systems, explanations are typically served alongside predictions. Key considerations:

# Production explanation service architecture class ExplanationService: def __init__(self, model, method="tree_shap"): self.model = model if method == "tree_shap": self.explainer = shap.TreeExplainer(model) elif method == "kernel_shap": self.explainer = shap.KernelExplainer(model.predict, bg_data) self.cache = LRUCache(maxsize=10000) def predict_with_explanation(self, x): prediction = self.model.predict(x) # Check cache first cache_key = hash_input(x) if cache_key in self.cache: explanation = self.cache[cache_key] else: shap_vals = self.explainer.shap_values(x) explanation = self.format_explanation(x, shap_vals) self.cache[cache_key] = explanation return { "prediction": prediction, "explanation": explanation, "method": "shap", "timestamp": now() } def format_explanation(self, x, shap_vals): # Return top-k features with human-readable descriptions top_k = np.argsort(np.abs(shap_vals))[-5:][::-1] return [{ "feature": self.feature_names[i], "value": float(x[i]), "impact": float(shap_vals[i]), "direction": "positive" if shap_vals[i] > 0 else "negative" } for i in top_k]
Audit trail: Always log explanations alongside predictions. When a stakeholder or regulator asks why a decision was made six months ago, you need to reproduce the exact explanation that was served at that time — not recompute it with a potentially different model version.

Best Practices Checklist

  • ☑ Choose explainability method based on model type and latency requirements
  • ☑ Validate explanations against domain knowledge — do they make sense?
  • ☑ Test explanation stability (run multiple times, compare results)
  • ☑ Cache explanations for production serving performance
  • ☑ Log explanations alongside predictions for audit trails
  • ☑ Tailor explanation format to the audience (technical vs. end-user)
  • ☑ Use global explanations for model monitoring and drift detection
  • ☑ Combine methods — use SHAP for precision and counterfactuals for actionability
