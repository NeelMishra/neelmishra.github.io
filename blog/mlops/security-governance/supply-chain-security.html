ML Supply Chain Security

MLOps Series Security & Governance

Introduction

Software supply chain attacks have surged in recent years, and machine learning systems are uniquely vulnerable. An ML supply chain encompasses not just code dependencies but also pre-trained model weights, training datasets, feature pipelines, and third-party APIs. A compromise at any point can silently poison predictions, exfiltrate data, or introduce backdoors that are nearly impossible to detect through traditional testing.

Real-world risk: Researchers have demonstrated model poisoning attacks on public model hubs where a subtly modified model passes standard benchmarks yet contains hidden backdoor triggers. Supply chain verification is the primary defence.

This post covers the five pillars of ML supply chain security: dependency scanning for code and packages, model provenance tracking, Software Bills of Materials (SBOMs) adapted for ML, cryptographic model signing, and end-to-end verification pipelines.

Dependency Scanning

ML projects inherit risk from every Python package, container base image, and system library in their dependency tree. A single vulnerable transitive dependency — say, an outdated version of Pillow or protobuf — can become an entry point.

Scanning Layers

  • Language packages — Use pip-audit, safety, or Dependabot to scan requirements.txt and pyproject.toml.
  • Container images — Scan with Trivy, Grype, or Snyk Container. ML images are often large (5–15 GB) and pack many libraries.
  • OS-level packages — CUDA drivers, cuDNN, and NCCL are commonly pinned to old versions for compatibility. Track their CVEs explicitly.
  • Notebook extensions — JupyterLab extensions can execute arbitrary code. Whitelist approved extensions.

Comparison: Scanning Tools

Open Source

  • pip-audit — Python-specific, fast, OSV database
  • Trivy — Containers, filesystems, IaC
  • Grype — Anchore's SBOM-aware scanner
  • OSV-Scanner — Google's multi-ecosystem scanner

Commercial / Managed

  • Snyk — Deep transitive analysis, fix PRs
  • GitHub Dependabot — Native to GitHub repos
  • JFrog Xray — Artefact-centric, binary analysis
  • Mend (WhiteSource) — License + vulnerability
Best practice: Run dependency scanning in CI on every pull request and on a nightly schedule. Vulnerabilities are disclosed continuously; scanning only at PR time misses newly published CVEs for already-merged code.

Model Provenance

Model provenance answers the question: "Where did this model come from, and can I trust it?" It is the ML equivalent of software attestation — a verifiable record of every input and transformation that produced a model artefact.

Provenance Record Contents

  • Source code commit — The exact Git SHA of the training code.
  • Training data snapshot — Hash or versioned reference (DVC, LakeFS) to the dataset used.
  • Environment — Container image digest, hardware spec (GPU type, driver version).
  • Hyperparameters — Full config used for training.
  • Base model — If fine-tuned, the upstream model identifier and its own provenance chain.
  • Builder identity — The CI/CD system or user who triggered the training run.
  • Timestamp & duration — When training started, ended, and total compute consumed.

SBOM for ML

A Software Bill of Materials (SBOM) is a structured inventory of all components in a software artefact. For ML, the SBOM must extend beyond code packages to include models, datasets, and pipeline configurations.

ML-SBOM Components

  • Code dependencies — Standard SBOM (CycloneDX or SPDX format) for Python/JS/Go packages.
  • Model components — Architecture definition, pre-trained base model identifiers, adapter/LoRA layers.
  • Data components — Dataset identifiers, versions, licences, and data cards.
  • Infrastructure — Container image digests, CUDA/cuDNN versions, hardware attestations.
  • Pipeline definitions — DAG specifications (Airflow, Kubeflow, Argo) with version hashes.

SBOM Generation Pseudo-Code

// Generate an ML-extended SBOM for a model artefact function GenerateMLSBOM(model_path, training_run_id): sbom ← CycloneDX.create(format: "json", spec_version: "1.5") // Code dependencies pip_deps ← pip_audit.scan(model_path + "/requirements.txt") for each dep in pip_deps: sbom.add_component(type: "library", name: dep.name, version: dep.version, purl: dep.purl, vulnerabilities: dep.vulns) // Container image image_digest ← Docker.inspect(model_path + "/Dockerfile").digest sbom.add_component(type: "container", name: image_digest) // Base model lineage base_model ← ModelRegistry.get_base(training_run_id) if base_model is not null: sbom.add_component(type: "ml-model", name: base_model.name, version: base_model.version, hash: base_model.sha256) // Training data references datasets ← DataCatalog.get_datasets(training_run_id) for each ds in datasets: sbom.add_component(type: "data", name: ds.name, version: ds.version, licence: ds.licence, hash: ds.sha256) // Pipeline definition pipeline_hash ← Git.hash(model_path + "/pipeline.yaml") sbom.add_component(type: "file", name: "pipeline.yaml", hash: pipeline_hash) // Sign and publish Sigstore.sign(sbom, identity: CI_IDENTITY) ArtifactStore.publish(sbom, tag: training_run_id + ".sbom.json") return sbom
Regulatory driver: Executive Order 14028 (US) and the EU Cyber Resilience Act both require SBOMs for software sold to government agencies. Extending your SBOM practice to ML artefacts future-proofs compliance as AI-specific regulations mature.

Model Signing

Cryptographic signing proves that a model artefact was produced by a trusted builder and has not been tampered with since creation. It is the final integrity seal in the supply chain.

Signing Approaches

Traditional PKI

  • Long-lived signing keys in HSM or KMS
  • Certificates issued by internal CA
  • Requires key management infrastructure
  • Well understood, broad tooling support

Sigstore (Keyless)

  • Ephemeral keys tied to OIDC identity
  • Transparency log (Rekor) provides auditability
  • No long-lived keys to manage or rotate
  • Emerging standard, growing ML adoption

Model Signing Pseudo-Code

// Sign a model artefact using Sigstore keyless signing function SignModel(model_artefact_path): // Compute the digest of the model file digest ← SHA256.hash_file(model_artefact_path) // Authenticate via OIDC (CI workload identity) oidc_token ← OIDC.get_token(audience: "sigstore") // Request ephemeral signing certificate from Fulcio cert ← Fulcio.request_cert(oidc_token) ephemeral_key ← Crypto.generate_keypair() // Sign the digest signature ← Crypto.sign(digest, ephemeral_key.private) // Record in transparency log log_entry ← Rekor.upload( artefact_hash: digest, signature: signature, certificate: cert ) // Attach signature bundle to the model artefact bundle ← { signature, cert, rekor_log_index: log_entry.index } ArtifactStore.attach_signature(model_artefact_path, bundle) return { digest, log_index: log_entry.index, status: "SIGNED" }
Warning: Signing a model does not verify its behaviour — only its integrity. A model can be correctly signed yet still contain biased or poisoned predictions. Signing must be combined with provenance checks and behavioural testing.

Putting It All Together

A mature ML supply chain security posture integrates all five pillars into a single admission policy. The policy is enforced by an admission controller (e.g., Kyverno, OPA Gatekeeper) that blocks any model deployment missing required attestations.

  • Dependency scan report — No critical/high CVEs.
  • Provenance attestation — SLSA Level 2+ signed provenance from the CI system.
  • ML-SBOM — Complete, signed, and published to the artefact store.
  • Model signature — Valid Sigstore or PKI signature with transparency log entry.
  • Behavioural tests — Passing accuracy, fairness, and safety benchmarks.
Key takeaway: Supply chain security for ML is not a single tool — it is a layered discipline that combines code scanning, data governance, cryptographic integrity, and behavioural validation. Start with dependency scanning and signing, then progressively add provenance and SBOM as your platform matures.
