Access Control for ML Assets

MLOps Series Security & Governance

Introduction

Machine learning systems touch some of the most sensitive assets in an organisation — proprietary training data, fine-tuned model weights, and high-throughput inference APIs. Yet access control for these assets often lags far behind what enterprises enforce for traditional software artefacts like databases and source code.

Key insight: A single leaked model can expose both intellectual property (architecture + weights) and private data (through membership inference attacks). Securing ML assets is not optional — it is table stakes for production AI.

This post covers the full stack of ML access control: role-based permissions for models and experiments, data governance policies, API authentication patterns, and secrets management. By the end you will have a blueprint for locking down every layer of your ML platform.

RBAC for ML

Role-Based Access Control (RBAC) assigns permissions to roles rather than individual users. In the ML context, you typically need roles that map to the ML lifecycle: data scientists who experiment, ML engineers who deploy, and auditors who review.

Defining ML-Specific Roles

The following pseudo-code defines a role hierarchy tailored to an ML platform:

// RBAC policy definitions for an ML platform role DataScientist: permissions: - experiments.create - experiments.read - experiments.update // own experiments only - datasets.read // approved datasets - models.read // registered models - models.register // promote experiment to registry constraints: scope: own_project data_tier: [public, internal] role MLEngineer extends DataScientist: permissions: - models.deploy // deploy to staging/production - models.rollback - endpoints.create - endpoints.update - pipelines.trigger constraints: scope: team_projects data_tier: [public, internal, confidential] role MLAdmin extends MLEngineer: permissions: - models.delete - datasets.delete - roles.assign - audit_logs.read constraints: scope: organisation requires_mfa: true role Auditor: permissions: - experiments.read - models.read - audit_logs.read - lineage.read constraints: scope: organisation read_only: true
Principle of least privilege: Start with the narrowest permissions and widen only when a justified need arises. Regular access reviews (quarterly) should prune stale grants.

Model Access Policies

Beyond RBAC, individual model artefacts need granular policies. A model trained on healthcare data may require stricter access than a recommendation model trained on public product catalogues.

Policy Dimensions

  • Classification tier — Public, internal, confidential, restricted. Inherited from the training data's highest classification.
  • Lifecycle stage — Experimental models have relaxed access; production models require approval workflows.
  • Geography — Models trained on EU personal data may need to remain within EU-resident infrastructure.
  • Expiration — Time-bound access grants for contractors or research collaborations.

Comparison: Model Access Approaches

Coarse-Grained (Project-Level)

  • All models in a project share one ACL
  • Simple to administer
  • Risk of over-permissioning
  • Suitable for small teams

Fine-Grained (Model-Level)

  • Each model version has its own policy
  • Supports data-classification inheritance
  • Higher administrative overhead
  • Required for regulated industries
Warning: Model downloads often bypass API gateways. Ensure your model registry (MLflow, Vertex AI Model Registry, SageMaker) enforces auth on artefact download endpoints, not just the UI.

Data Access Governance

Training data is the crown jewel of any ML system. Governance must cover the full data lifecycle: ingestion, storage, access, transformation, and deletion.

Data Classification

Apply a tiered classification scheme to every dataset:

  • Public — Open-source or publicly available data. No restrictions.
  • Internal — Business data not meant for external release. Standard access controls.
  • Confidential — PII, financial records, health data. Encryption at rest, audit logging, need-to-know access.
  • Restricted — Regulated data (HIPAA, PCI-DSS). Dedicated infrastructure, DLP controls, mandatory access reviews.

Access Patterns

ML workloads access data differently than traditional applications. Key patterns to govern:

  • Bulk reads — Training jobs read entire tables. Use service accounts with scoped credentials and time-limited tokens.
  • Feature stores — Centralised feature access. The feature store itself should enforce column-level ACLs.
  • Data snapshots — Immutable copies used for reproducibility. Treat snapshots with the same classification as the source.
  • Synthetic data — Differentially private synthetic datasets can reduce the need for broad access to raw confidential data.
Tip: Use Apache Ranger, AWS Lake Formation, or Google Dataplex to enforce column-level and row-level security on data lakes. These integrate with Spark, Hive, and BigQuery workloads used in ML pipelines.

API Authentication

Inference endpoints are the public face of your ML models. Weak authentication exposes models to abuse, data exfiltration, and adversarial attacks. A layered auth strategy is essential.

Authentication Approaches

API Key

  • Simple to implement
  • No user identity — attribution is per-key
  • Rotation overhead
  • Best for internal service-to-service calls

OAuth 2.0 / JWT

  • Rich identity and scope claims
  • Short-lived tokens reduce blast radius
  • Supports OIDC federation
  • Best for user-facing and partner APIs

Auth Middleware Pseudo-Code

// API authentication middleware for ML inference endpoints middleware AuthenticateRequest(request): token ← request.headers["Authorization"] if token is null: return Respond(401, "Missing authentication token") // Decode and verify the JWT claims ← JWT.verify(token, PUBLIC_KEY) if claims is invalid: return Respond(401, "Invalid or expired token") // Check scopes against endpoint requirements required_scope ← EndpointRegistry.get_scope(request.path) if required_scope not in claims.scopes: AuditLog.record("FORBIDDEN", claims.sub, request.path) return Respond(403, "Insufficient permissions") // Rate limiting per client identity if RateLimiter.exceeded(claims.sub, request.path): return Respond(429, "Rate limit exceeded") // Attach identity to request context request.context.identity ← claims.sub request.context.scopes ← claims.scopes return next(request)
Warning: Never embed long-lived API keys in client-side code or mobile apps. Use short-lived tokens issued through a backend auth service, and rotate keys on a defined schedule (≤ 90 days).

mTLS for Service Mesh

For internal micro-services calling inference endpoints, mutual TLS (mTLS) provides strong machine-to-machine authentication without managing API keys. Service meshes like Istio and Linkerd automate certificate issuance and rotation.

Secrets Management

ML pipelines are riddled with secrets: database credentials, cloud service account keys, API tokens for data providers, and encryption keys for model artefacts. Poor secrets hygiene is one of the most common vectors for ML supply chain attacks.

Principles

  • Never hard-code secrets — Use environment variables or secret references, never literal values in code or config files.
  • Centralise in a vault — HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, or GCP Secret Manager.
  • Least-privilege dynamic secrets — Generate short-lived, scoped credentials on demand rather than sharing long-lived keys.
  • Audit every access — All secret reads and rotations must be logged and alertable.
  • Automate rotation — Set maximum TTLs and automate rotation to eliminate stale credentials.

Secrets Anti-Patterns in ML

  • Storing cloud credentials in Jupyter notebook cells or .env files checked into Git.
  • Sharing a single service account key across all pipeline stages (blast radius is entire platform).
  • Embedding API keys in Docker images or model artefact metadata.
  • Using the same credentials for dev, staging, and production environments.
