Audit Logging for ML Systems

MLOps Series Security & Governance

Machine learning systems make consequential decisions at scale—credit approvals, medical diagnoses, content moderation. When things go wrong, organizations need a precise, timestamped trail of every action the system took, from data ingestion through feature computation to model inference and post-processing. Audit logging provides that trail.

Unlike traditional application logging focused on debugging, ML audit logging must capture the full provenance chain: which data was used, which model version ran, what features were computed, and what decision was rendered. This post covers what to log, how to make logs immutable, how to satisfy regulatory mandates, and how to build a production-grade aggregation pipeline.

Key Principle: An audit log answers "who did what, when, with which model, on which data, and why." If any of those dimensions are missing, your log is incomplete.

What to Log in ML Systems

ML audit logs extend well beyond HTTP request/response pairs. A comprehensive audit logging strategy for ML must capture events across the entire lifecycle. Every log entry should follow a structured schema that enables querying, correlation, and forensic reconstruction of any prediction or training run.

Core Event Categories

  • Data Events — Dataset creation, schema changes, ingestion runs, data quality checks, feature drift detections, and data deletion requests (GDPR right-to-erasure).
  • Training Events — Experiment start/end, hyperparameter configurations, training data snapshots (hash references), evaluation metrics, model artifact registration.
  • Deployment Events — Model promotion, canary rollout percentages, rollback triggers, A/B test assignments, shadow mode activation.
  • Inference Events — Input feature vectors (or hashes), model version used, raw prediction scores, post-processing transformations, final decision output, latency.
  • Access Events — Who accessed what model artifact, API key usage, permission changes, credential rotations.

Structured Audit Log Schema

// Structured audit log entry for ML inference class AuditLogEntry: event_id: UUID // globally unique timestamp: ISO-8601 // nanosecond precision event_type: Enum // DATA | TRAIN | DEPLOY | INFER | ACCESS actor: String // user, service account, or pipeline ID resource: String // model name, dataset ID, endpoint action: String // "predict", "register", "delete" model_version: String // e.g. "fraud-v2.3.1" input_hash: SHA-256 // hash of input features output: JSON // prediction result metadata: JSON // latency_ms, region, experiment_id trace_id: UUID // distributed trace correlation def emit_audit_log(entry: AuditLogEntry): entry.event_id = uuid4() entry.timestamp = now_utc_nano() validate_schema(entry) sign_entry(entry, hmac_key) // tamper detection append_to_log(immutable_store, entry) forward_to_siem(entry) // security monitoring
Warning: Never log raw PII (names, SSNs, emails) in audit logs. Use tokenized or hashed references and store a separate, access-controlled PII mapping table. This is a hard compliance requirement under GDPR and CCPA.

Inference-Specific Logging

For each prediction request, capture the complete decision context. This means the exact feature vector (or its content-addressable hash), the model version string, any pre-processing or post-processing rules applied, the raw model output, and the final business decision. Store enough information that you can replay the exact prediction months later for audit or litigation purposes.

Immutable Log Architecture

Audit logs lose their value if they can be retroactively modified or deleted. Immutability is the cornerstone of trustworthy audit logging. An immutable log guarantees that once an event is written, it cannot be altered, overwritten, or purged—even by administrators.

Append-Only Storage Patterns

The simplest immutability pattern is an append-only log backed by write-once storage. Cloud providers offer native support: AWS S3 Object Lock (Governance or Compliance mode), Azure Blob Immutable Storage, and GCP Bucket Lock. At the application layer, you can enforce append-only semantics by removing DELETE and UPDATE permissions from the logging service's IAM role.

Hash-Chain Verification

For stronger tamper evidence, chain log entries using cryptographic hashes—each entry includes the hash of the previous entry, forming a Merkle-style chain. Any retroactive modification breaks the chain and is immediately detectable.

// Hash-chain log verification class ChainedLogEntry: entry: AuditLogEntry prev_hash: SHA-256 entry_hash: SHA-256 def append_chained(log_store, entry): prev = get_last_entry(log_store) entry.prev_hash = prev.entry_hash entry.entry_hash = sha256(entry.serialize() + entry.prev_hash) write_once(log_store, entry) def verify_chain(log_store): entries = read_all(log_store) for i in range(1, len(entries)): expected = sha256(entries[i].serialize() + entries[i-1].entry_hash) if expected != entries[i].entry_hash: raise TamperDetected(f"Chain broken at entry {i}") return True // chain intact
Production Tip: AWS QLDB (Quantum Ledger Database) provides a fully managed, cryptographically verifiable, immutable journal—ideal for ML audit logs where you need provable data integrity without building your own hash-chain infrastructure.

Retention and Lifecycle

Define retention policies aligned with your regulatory requirements. HIPAA mandates 6 years, SOX requires 7 years, and the EU AI Act mandates keeping high-risk system logs for the lifetime of the system plus additional years. Implement lifecycle rules that transition logs from hot storage (for real-time querying) to cold storage (S3 Glacier, Azure Cool Tier) after 90 days, while maintaining immutability throughout.

Compliance Requirements

Different regulatory frameworks impose specific requirements on what must be logged, how long logs must be retained, and what access controls must be in place. Understanding these requirements is essential before designing your logging architecture.

Framework-Specific Mandates

  • GDPR (EU) — Log all processing activities involving personal data. Implement right-to-erasure for PII within logs (use tokenization). Maintain records of processing activities (Article 30). Data Protection Impact Assessments must be documented.
  • HIPAA (Healthcare) — Log all access to Protected Health Information (PHI). Retain audit logs for 6 years minimum. Implement unique user identification and automatic logoff tracking.
  • SOC 2 Type II — Demonstrate continuous monitoring through audit logs. Log all changes to production systems. Evidence of log integrity and access controls required during audit.
  • EU AI Act — High-risk AI systems must maintain logs that enable traceability of results. Logs must be kept for the system's intended lifetime. Must include information on input data, model versions, and decision outputs.
  • SOX (Financial) — Log all changes to models that affect financial reporting. Maintain 7-year retention. Segregation of duties must be evidenced in access logs.
Regulatory Note: Under the EU AI Act, providers of high-risk AI systems must ensure that automatic logging capabilities are built into the system by design. Retrofitting audit logging after deployment may not satisfy the "by design" requirement.

Access Control for Audit Logs

Audit logs themselves are sensitive assets. Implement strict RBAC: only a dedicated security team should have read access to audit logs. No individual—including system administrators—should have write or delete permissions on the log store. Use separate service accounts for log producers versus log consumers, and rotate credentials on a fixed schedule.

Log Aggregation Pipelines

In a distributed ML system, audit events originate from dozens of services: feature stores, model registries, inference endpoints, training clusters, and data pipelines. A centralized log aggregation pipeline collects, normalizes, enriches, and stores these events in a single queryable repository.

Pipeline Architecture

A production-grade ML audit aggregation pipeline typically follows the pattern: Producers → Message Bus → Stream Processor → Immutable Store + Search Index. Each component must preserve ordering guarantees (at least within a partition key like model_id or trace_id) and provide at-least-once delivery semantics.

// Log aggregation pipeline configuration pipeline ml_audit_aggregation: // Step 1: Collect from all producers sources: - inference_service → kafka_topic("ml-audit-infer") - training_service → kafka_topic("ml-audit-train") - feature_store → kafka_topic("ml-audit-features") - model_registry → kafka_topic("ml-audit-registry") // Step 2: Stream processing — normalize and enrich processor: flink_job("audit-normalizer") normalize_timestamps(to_utc_nano) enrich_with_metadata(model_catalog, user_directory) validate_schema(audit_log_v2_schema) compute_hash_chain(partition_key=model_id) // Step 3: Dual-write to immutable store + search sinks: - s3_object_lock("audit-bucket", retention=7_years) - opensearch_index("ml-audit-logs") - siem_forwarder("splunk-hec-endpoint")

Correlation and Tracing

Use distributed trace IDs (OpenTelemetry trace_id) to correlate audit events across services. A single prediction request might generate events in the feature store (feature computation), the inference service (model prediction), and the post-processor (business rule application). The trace_id ties these into a single auditable transaction that can be reconstructed end-to-end.

Best Practice: Instrument your ML services with OpenTelemetry and propagate trace context through every service boundary. This makes forensic reconstruction of any prediction trivial—query by trace_id and get the complete event chain.

Tooling & Setup

Choosing the right tooling stack depends on your scale, compliance requirements, and existing infrastructure. Below is a comparison of popular audit logging solutions for ML systems, followed by a practical setup guide.

ELK Stack (Elasticsearch, Logstash, Kibana)

  • Open-source, self-hosted
  • Powerful full-text search and dashboards
  • Requires manual immutability setup (ILM + snapshot policies)
  • High operational overhead at scale
  • Best for: Teams with Elastic expertise wanting full control

AWS CloudTrail + S3 Object Lock

  • Fully managed, native AWS integration
  • Built-in immutability with Compliance mode
  • Automatic log integrity validation via digest files
  • Limited custom schema support
  • Best for: AWS-native ML platforms (SageMaker)

Splunk Enterprise

  • Enterprise-grade SIEM integration
  • Advanced analytics and anomaly detection on logs
  • Compliance-certified (SOC 2, HIPAA, FedRAMP)
  • Expensive at high ingest volumes
  • Best for: Regulated industries needing SIEM correlation

AWS QLDB / Azure Confidential Ledger

  • Cryptographically verifiable immutable journal
  • Built-in hash-chain—no custom implementation needed
  • PartiQL (QLDB) or REST API for queries
  • Higher per-transaction cost than object storage
  • Best for: High-stakes ML (finance, healthcare) needing provable integrity

Practical Setup: Structured Logging with Python

import structlog, hashlib, json, datetime # Configure structured logger for ML audit events structlog.configure( processors=[ structlog.processors.TimeStamper(fmt="iso", utc=True), structlog.processors.JSONRenderer() ] ) logger = structlog.get_logger("ml_audit") class MLAuditLogger: def __init__(self, hmac_key: bytes): self.hmac_key = hmac_key self.prev_hash = "genesis" def log_inference(self, model_id, version, input_hash, output, latency_ms): entry = { "event_type": "INFERENCE", "model_id": model_id, "model_version": version, "input_hash": input_hash, "output": output, "latency_ms": latency_ms, "prev_hash": self.prev_hash } entry["signature"] = self._sign(entry) self.prev_hash = entry["signature"] logger.info("ml_audit_event", **entry) def _sign(self, entry: dict) -> str: payload = json.dumps(entry, sort_keys=True) return hashlib.sha256(payload.encode() + self.hmac_key).hexdigest()

Deployment Checklist: Before going to production, verify: (1) logs are written to immutable storage, (2) hash-chain verification runs on a schedule, (3) SIEM integration is active, (4) retention policies match your regulatory requirements, and (5) no IAM role has delete permissions on the audit bucket.

Monitoring Your Audit Pipeline

An audit logging pipeline that silently drops events is worse than no audit logging at all—it creates a false sense of compliance. Monitor these signals: event ingestion rate (sudden drops indicate producer failures), end-to-end latency (events should land in the immutable store within seconds), schema validation failures (malformed events suggest integration regressions), and hash-chain verification results (any failure is a critical security incident).

Set up PagerDuty or Opsgenie alerts for any of these anomalies. Your audit pipeline should be treated with the same operational rigor as your production inference pipeline—because in regulated industries, a broken audit trail can shut down your entire ML system.
