Data Poisoning: Attacks and Defenses

MLOps Series Security & Governance

Introduction

Data poisoning is a class of adversarial attacks that target the training phase of machine learning. Instead of attacking the model at inference time, the adversary manipulates the training data so that the resulting model learns incorrect behaviour — misclassifying specific inputs, degrading overall accuracy, or embedding hidden backdoors that activate only on attacker-chosen triggers.

These attacks are especially dangerous in settings where training data is collected from untrusted sources: web scraping, crowd-sourced labelling, federated learning, or public datasets. Even a small fraction of poisoned samples — often less than 1% of the training set — can cause significant damage.

Real-world impact: Poisoning attacks have been demonstrated against spam filters, malware detectors, recommendation systems, and autonomous driving perception models. The attack surface is broad and growing.

Label Flipping

Label flipping is the simplest form of data poisoning. The attacker changes the labels of a subset of training examples — for instance, relabelling "spam" emails as "not spam" or mislabelling stop signs as speed-limit signs. The model then learns incorrect associations between features and classes.

The effectiveness of label flipping depends on several factors: the fraction of labels flipped, whether the flipping is random or targeted, and the model's capacity. Simple models like logistic regression can be significantly degraded with only 10–15% label corruption, while deep networks are somewhat more robust due to their ability to memorize and overfit — but this resilience has limits.

# Simulating a label-flipping poisoning attack import numpy as np def label_flip_attack(X_train, y_train, target_class, flip_class, poison_rate=0.1): """Flip labels from target_class to flip_class.""" y_poisoned = y_train.copy() target_indices = np.where(y_train == target_class)[0] # Select a fraction of target samples to flip n_poison = int(len(target_indices) * poison_rate) flip_indices = np.random.choice(target_indices, n_poison, replace=False) # Flip their labels y_poisoned[flip_indices] = flip_class print(f"Flipped {n_poison} labels from class {target_class} to {flip_class}") return X_train, y_poisoned def random_flip_attack(X_train, y_train, n_classes, poison_rate=0.05): """Randomly flip a fraction of all labels.""" y_poisoned = y_train.copy() n_poison = int(len(y_train) * poison_rate) poison_indices = np.random.choice(len(y_train), n_poison, replace=False) for idx in poison_indices: new_label = np.random.choice([c for c in range(n_classes) if c != y_train[idx]]) y_poisoned[idx] = new_label return X_train, y_poisoned
Detection hint: Label flipping introduces statistical inconsistencies. Techniques like confident learning, which identifies samples whose given labels disagree with the model's predicted labels, can flag likely flipped examples.

Clean-Label Attacks

Clean-label attacks are a more sophisticated variant where the attacker does not change any labels. Instead, they craft poisoned training samples that look correctly labelled to a human inspector but contain subtle perturbations that cause the model to learn a hidden backdoor or misclassify specific test inputs.

The canonical example is a backdoor attack where a small visual trigger patch (e.g., a tiny sticker) is added to training images. At test time, any image containing the same trigger is classified as the attacker's chosen target class, while images without the trigger are classified normally — making the attack very hard to detect.

Label-Flipping Attacks

  • Visibly incorrect labels
  • Detectable by manual inspection
  • Degrades overall model accuracy
  • Simpler to execute
  • Broad, untargeted impact

Clean-Label Attacks

  • Labels appear correct to humans
  • Very hard to detect visually
  • Preserves accuracy on clean inputs
  • Requires optimization to craft samples
  • Targeted, stealthy impact

Clean-label poisoning typically works by solving a bi-level optimization problem: find the smallest perturbation to training inputs such that a model trained on the perturbed data misclassifies a specific target test input. The outer loop optimizes the poison perturbation, while the inner loop simulates training. This is computationally expensive but increasingly practical with modern hardware.

Danger: Clean-label attacks can bypass data-quality checks that rely solely on verifying labels. They require feature-level analysis or neural cleanse techniques to detect.

Data Sanitization

Data sanitization is the first line of defence against poisoning. The goal is to identify and remove (or correct) poisoned samples before they reach the training pipeline. Effective sanitization combines statistical methods, representation analysis, and ensemble techniques.

# Data sanitization pipeline import numpy as np from sklearn.ensemble import IsolationForest from sklearn.decomposition import PCA def sanitize_dataset(X_train, y_train, contamination=0.05): """Remove suspected poisoned samples using multiple detectors.""" clean_mask = np.ones(len(X_train), dtype=bool) # Stage 1: Isolation Forest for outlier detection iso = IsolationForest(contamination=contamination, random_state=42) outlier_preds = iso.fit_predict(X_train) clean_mask &= (outlier_preds == 1) # Stage 2: Per-class centroid distance filtering for cls in np.unique(y_train): cls_mask = (y_train == cls) centroid = X_train[cls_mask].mean(axis=0) distances = np.linalg.norm(X_train[cls_mask] - centroid, axis=1) threshold = np.percentile(distances, 95) far_samples = distances > threshold indices = np.where(cls_mask)[0][far_samples] clean_mask[indices] = False # Stage 3: PCA-based spectral signature detection pca = PCA(n_components=10) X_pca = pca.fit_transform(X_train) for cls in np.unique(y_train): cls_data = X_pca[y_train == cls] scores = np.abs(cls_data @ pca.components_[-1]) threshold = np.percentile(scores, 95) suspect = scores > threshold indices = np.where(y_train == cls)[0][suspect] clean_mask[indices] = False removed = (~clean_mask).sum() print(f"Removed {removed} suspected poisoned samples") return X_train[clean_mask], y_train[clean_mask]

Spectral signature analysis — the PCA-based approach in Stage 3 above — is particularly effective against backdoor attacks. Poisoned samples tend to leave a detectable trace in the top singular vectors of the per-class data matrix, making them identifiable even when their labels appear correct.

Tip: Run sanitization before every training job as part of your CI/CD pipeline. Treat data validation with the same rigour as code review — poisoned data is just as dangerous as a security vulnerability in code.

Robust Training

Even after sanitization, some poisoned samples may evade detection. Robust training techniques modify the learning algorithm itself to be less sensitive to outliers and corrupted data, providing a second layer of defence.

Key approaches include:

  • Trimmed loss: Sort per-sample losses each batch and discard the top-k% highest losses before backpropagation. Poisoned samples tend to have high loss.
  • Gradient clipping per sample: Limit the influence of any single training example on the model update. This is related to differential privacy (see the companion post) and inherently limits poisoning impact.
  • RONI (Reject On Negative Impact): Evaluate each training sample's impact on a held-out validation set. Remove samples whose inclusion degrades validation performance.
  • Ensemble agreement: Train multiple models on different data subsets. Samples that cause disagreement between ensemble members are flagged as suspicious.

Practical Defenses

In production ML systems, defending against data poisoning requires combining multiple techniques across the entire data lifecycle — from collection to training to monitoring in production. Here is a practical defence checklist:

Data Pipeline Defenses

  • Verify data provenance and integrity hashes
  • Automated statistical profiling on ingestion
  • Spectral signature analysis per class
  • Human-in-the-loop review for flagged samples
  • Version control all training datasets

Training & Deployment Defenses

  • Robust loss functions (trimmed, MAE)
  • Per-sample gradient clipping
  • Ensemble training with disagreement checks
  • Neural cleanse backdoor scanning
  • Continuous monitoring for accuracy drift
Layered defence: No single technique catches all poisoning attacks. Combine data sanitization (catch obvious poison), robust training (limit subtle poison impact), and post-training scanning (detect backdoors). Monitor deployed models for sudden accuracy drops on specific subpopulations.

Provenance tracking deserves special emphasis. By maintaining a cryptographic hash of every data source and sample, you can trace any training example back to its origin. If a poisoning attack is detected, provenance records enable rapid identification of the compromised source and surgical removal of affected samples without reprocessing the entire dataset.

Warning: Federated learning environments are particularly vulnerable because the central server cannot inspect individual participants' data. Defence strategies must rely on gradient-level anomaly detection and Byzantine-robust aggregation protocols — topics we cover in a dedicated post.

Building a poisoning-resilient ML pipeline is not a one-time effort. As attack techniques evolve, defences must be continuously updated. Treat your data pipeline with the same security rigour as your application code — review, audit, and test regularly.
