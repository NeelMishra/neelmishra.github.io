Red Teaming ML Systems

MLOps Series Security & Governance

Traditional security testing assumes deterministic software: given the same input, you get the same output. ML systems break this assumption. Models are probabilistic, their behavior is shaped by training data, and their failure modes are often emergent—discovered only through systematic adversarial probing. Red teaming for ML goes beyond penetration testing: it evaluates the model's robustness to adversarial inputs, its susceptibility to data poisoning, its tendency to leak training data, and its alignment with intended behavior under stress.

This post covers a structured methodology for red teaming ML systems, attack playbooks for common vulnerability classes, automated tooling for continuous adversarial testing, and a framework for responsible vulnerability reporting and remediation.

Context: The EU AI Act (Art. 15) requires high-risk AI systems to be "resilient against attempts by unauthorized third parties to alter their use, outputs or performance by exploiting system vulnerabilities." Red teaming is how you demonstrate that resilience.

Red Team Methodology

An effective ML red team engagement follows a structured methodology adapted from traditional cybersecurity red teaming but extended to cover ML-specific attack surfaces. The goal is not just to find vulnerabilities, but to characterize the system's failure boundaries and quantify residual risk.

Phase 1: Threat Modeling

Before launching any attacks, build a threat model specific to the ML system. Identify the attack surface (API endpoints, training pipeline, feature store, model artifacts), the threat actors (external attackers, malicious insiders, competing organizations), and the assets at risk (model IP, training data, user privacy, decision integrity).

  • STRIDE for ML — Extend the classic STRIDE framework: Spoofing (adversarial inputs), Tampering (data poisoning), Repudiation (lack of audit logs), Information Disclosure (model inversion, membership inference), Denial of Service (resource exhaustion attacks), Elevation of Privilege (prompt injection to bypass safety filters).
  • Attack Trees — Build hierarchical attack trees for each threat. The root node is the attacker's goal (e.g., "extract training data"), and leaf nodes are specific attack techniques (membership inference, model inversion, training data extraction via prompting).

Phase 2: Reconnaissance

Gather information about the target system without triggering detection. For ML systems, reconnaissance includes: querying the API to infer model architecture (number of output classes, response format), measuring response latencies to estimate model size, probing error messages for information leakage, and analyzing publicly available model cards or documentation for hints about training data and known limitations.

Phase 3: Attack Execution

Execute attacks from the playbooks below, starting with the lowest-effort, highest-impact techniques. Document every attempt—successful or not—with the exact inputs used, the system's response, and the timestamp. This documentation is essential for the vulnerability report and for regression testing after fixes are applied.

Phase 4: Reporting & Remediation

Produce a structured vulnerability report (see the Vulnerability Reporting section below), prioritize findings by severity, and work with the ML engineering team to develop mitigations. Schedule a retest to verify that fixes are effective and haven't introduced new vulnerabilities.

Important: Always conduct red team exercises with explicit authorization and a defined scope. Unauthorized adversarial testing of production ML systems—even your own organization's—can violate terms of service, cause service disruptions, and create legal liability.

Attack Playbooks

The following playbooks cover the major classes of ML-specific attacks. Each playbook describes the attack objective, prerequisites, technique, detection indicators, and mitigations.

Playbook 1: Adversarial Input Attacks

Objective: Cause the model to produce incorrect predictions by crafting inputs that exploit learned decision boundaries. Common in computer vision (adversarial patches, perturbations) and NLP (synonym substitution, character-level perturbations).

# Adversarial example generation using PGD (Projected Gradient Descent) import torch def pgd_attack(model, x, y_true, epsilon=0.03, alpha=0.01, steps=40): """Generate adversarial examples using PGD attack.""" x_adv = x.clone().detach().requires_grad_(True) for step in range(steps): output = model(x_adv) loss = torch.nn.functional.cross_entropy(output, y_true) loss.backward() # Gradient ascent step (maximize loss) x_adv = x_adv + alpha * x_adv.grad.sign() # Project back into epsilon-ball around original input perturbation = torch.clamp(x_adv - x, -epsilon, epsilon) x_adv = torch.clamp(x + perturbation, 0.0, 1.0).detach().requires_grad_(True) return x_adv # Evaluate attack success rate def evaluate_robustness(model, test_loader, epsilon): fooled, total = 0, 0 for x, y in test_loader: x_adv = pgd_attack(model, x, y, epsilon) pred = model(x_adv).argmax(dim=1) fooled += (pred != y).sum().item() total += len(y) print(f"Attack success rate: {fooled/total:.1%} at ε={epsilon}")

Playbook 2: Prompt Injection (LLM Systems)

Objective: Override the system prompt or safety instructions by embedding adversarial instructions within user input. This is the most critical attack vector for LLM-based applications. Techniques include direct injection ("Ignore previous instructions and..."), indirect injection (embedding instructions in retrieved documents), and payload splitting across multiple messages.

Playbook 3: Data Poisoning

Objective: Compromise model behavior by injecting malicious samples into the training data. Backdoor attacks insert a trigger pattern that causes targeted misclassification. Label-flipping attacks corrupt ground truth labels. Gradient-based poisoning optimizes poisoned samples to maximally shift decision boundaries.

Playbook 4: Model Extraction

Objective: Steal the model's functionality by querying it repeatedly and training a surrogate model on the input-output pairs. Even with API rate limiting, an attacker can often extract a functionally equivalent model with 10,000-100,000 queries—far cheaper than training from scratch.

Playbook 5: Membership Inference

Objective: Determine whether a specific data point was in the model's training set. This is a privacy attack: if a model was trained on medical records, membership inference can reveal that a specific individual's data was used. The attack exploits the model's tendency to be more confident on training data than on unseen data.

Defense-in-Depth: No single defense stops all attack vectors. Layer mitigations: adversarial training for robustness, differential privacy for membership inference, input validation for prompt injection, data provenance tracking for poisoning, and rate limiting + output perturbation for model extraction.

Automated Red Teaming Tools

Manual red teaming is essential for creative, novel attacks, but automated tools are necessary for continuous coverage, regression testing, and scale. The tooling landscape for ML red teaming has matured significantly, with open-source frameworks now covering most major attack categories.

Microsoft Counterfit

  • Open-source CLI tool for adversarial ML
  • Supports evasion, poisoning, and inference attacks
  • Integrates with ART (Adversarial Robustness Toolbox)
  • Target-agnostic: works with any API endpoint
  • Best for: Automated robustness scanning in CI/CD

IBM Adversarial Robustness Toolbox (ART)

  • Comprehensive library: 100+ attack/defense methods
  • Supports PyTorch, TensorFlow, Keras, scikit-learn
  • Covers evasion, poisoning, extraction, inference
  • Includes certified defenses and detection methods
  • Best for: Deep robustness evaluation and research

NVIDIA Garak

  • LLM vulnerability scanner
  • Probes for prompt injection, jailbreaks, data leakage
  • Pluggable probe and detector architecture
  • Supports OpenAI, Hugging Face, and local models
  • Best for: LLM safety evaluation before deployment

Microsoft PyRIT

  • Python Risk Identification Tool for generative AI
  • Orchestrates multi-turn adversarial conversations
  • Automated scoring of harmful outputs
  • Extensible with custom attack strategies
  • Best for: Systematic generative AI red teaming at scale

Vulnerability Reporting

A structured vulnerability report is the primary deliverable of a red team engagement. For ML systems, reports must go beyond traditional CVE-style descriptions to include information about the model's failure mode, the conditions under which the vulnerability manifests, and the potential real-world impact.

ML Vulnerability Report Template

  1. Executive Summary — One-paragraph overview of findings for leadership. Include the number of critical/high/medium/low findings and overall risk assessment.
  2. Scope & Methodology — What systems were tested, which attack categories were covered, tools used, and any constraints or limitations of the engagement.
  3. Finding Details — For each vulnerability:
    • Title — Clear, descriptive name (e.g., "Training Data Extraction via Repeated Prompting")
    • Severity — CRITICAL / HIGH / MEDIUM / LOW with justification
    • Attack Vector — How an attacker would exploit this (MITRE ATLAS reference if applicable)
    • Reproducibility — Exact steps and inputs to reproduce the issue
    • Impact — What an attacker gains (data access, decision manipulation, service disruption)
    • Evidence — Screenshots, API responses, statistical analysis
    • Remediation — Specific technical recommendations
  4. Risk Matrix — Aggregate view of all findings plotted on likelihood × impact axes.
  5. Remediation Roadmap — Prioritized list of fixes with estimated effort and timelines.
Responsible Disclosure: If your red team discovers a vulnerability in a third-party model or API (e.g., a foundation model provider), follow responsible disclosure practices. Contact the provider's security team through their designated channel, provide a reasonable disclosure timeline (typically 90 days), and do not publicly disclose details until a fix is available.

Severity Classification for ML Vulnerabilities

Standard CVSS scoring doesn't fully capture ML-specific risks. Supplement CVSS with ML-specific factors:

  • CRITICAL — Training data extraction with PII, complete safety bypass in production, backdoor in deployed model.
  • HIGH — Reliable adversarial evasion on safety-critical predictions, prompt injection that bypasses all safety filters, membership inference on sensitive data.
  • MEDIUM — Model extraction requiring significant query budget, partial safety bypass, adversarial inputs that require white-box access.
  • LOW — Information leakage via error messages, minor prediction perturbation on non-critical outputs, theoretical attacks requiring impractical access.

Red Team Exercises

Regular red team exercises build organizational muscle for adversarial thinking. Below are three structured exercises that ML teams can run internally, ranging from a focused half-day exercise to a comprehensive multi-week engagement.

Exercise 1: Tabletop Threat Modeling (Half Day)

Assemble ML engineers, security engineers, and product managers in a room. Pick one production ML system and walk through the STRIDE-for-ML framework. For each threat category, brainstorm specific attack scenarios, estimate likelihood and impact, and identify existing mitigations and gaps. Output: a prioritized threat register and a list of action items for the engineering team.

Exercise 2: Capture-the-Flag Adversarial Challenge (1-2 Days)

Deploy a sandboxed copy of a production model behind an API. Define "flags" that represent successful attacks: extract the system prompt (Flag 1), cause a targeted misclassification (Flag 2), determine if a specific data point was in training data (Flag 3), and extract the model architecture (Flag 4). Teams compete to capture the most flags. This builds adversarial intuition and reveals which attack vectors your defenses are weakest against.

Exercise 3: Full Red Team Engagement (2-4 Weeks)

A comprehensive engagement that follows the full methodology described above. An external or dedicated internal red team operates with defined rules of engagement and attempts to compromise the ML system across all attack categories. The engagement produces a formal vulnerability report, a remediation roadmap, and updated threat models. Schedule these quarterly for high-risk AI systems.

Internal Red Teams

  • Deep knowledge of system internals
  • Can test white-box attack scenarios
  • Lower cost, faster feedback loops
  • Risk of confirmation bias and blind spots
  • Best for: Continuous testing, regression checks

External Red Teams

  • Fresh perspective, no insider assumptions
  • Realistic black-box attacker simulation
  • Higher cost, longer engagement cycles
  • Brings cross-industry attack intelligence
  • Best for: Quarterly assessments, compliance evidence
Building a Red Team Culture: The most effective ML security programs embed adversarial thinking into everyday engineering. Encourage ML engineers to think like attackers during code review. Add "how could this be exploited?" as a standard design review question. Celebrate vulnerability discoveries internally—every bug found by your red team is one that an attacker didn't find first.

Measuring Red Team Effectiveness

Track these metrics over time to assess whether your red team program is improving your security posture: mean time to detect (how quickly your monitoring catches red team activities), vulnerability density (findings per model per engagement—this should decrease over time), remediation velocity (time from finding to fix), and regression rate (previously fixed vulnerabilities that reappear). Use these metrics to justify investment in security tooling and training.

Red teaming is not a one-time event—it is a continuous practice. As models are updated, as new attack techniques are published, and as deployment contexts change, the threat landscape evolves. Build red teaming into your MLOps lifecycle alongside monitoring, testing, and compliance. The cost of proactive adversarial testing is a fraction of the cost of a real-world exploit.
