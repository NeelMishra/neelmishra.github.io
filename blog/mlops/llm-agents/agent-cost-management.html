Agent Cost: Token Budgets and Optimization

MLOps Series LLM Agents & Orchestration

Anatomy of Agent Costs

Autonomous agents are the most expensive LLM pattern in production. A single agent run can trigger dozens of LLM calls — planning, tool selection, observation parsing, reflection, and final synthesis — each consuming tokens at API prices. Understanding where those tokens go is the first step toward controlling spend.

The dominant cost driver is the tool loop. Each iteration re-sends the growing conversation history (input tokens) plus generates new reasoning and tool calls (output tokens). Because input context grows linearly with each step, an N-step agent run has O(N²) input token cost — the "context accumulation problem."

Warning — quadratic cost trap: A 10-step agent run doesn't cost 10× a single call. With growing context, it costs 25-50× a single call. Capping loop iterations is the single most impactful cost lever.

📥 Input Token Costs

  • System prompt (repeated every call)
  • Conversation history (grows each step)
  • Tool definitions & schemas
  • Retrieved context / RAG chunks
  • Observation results from tools

Typically 70-85% of total token spend

📤 Output Token Costs

  • Reasoning / chain-of-thought
  • Tool call JSON (function name + args)
  • Intermediate summaries
  • Final answer synthesis
  • Structured response formatting

Typically 4× more expensive per token

Token Tracking & Accounting

You can't optimize what you don't measure. A robust token accounting system tracks usage at every granularity — per call, per agent step, per run, per user, and per feature — enabling both real-time budget enforcement and historical cost analysis.

from dataclasses import dataclass, field
from typing import List, Optional
import time

# Core token accounting structure
@dataclass
class TokenUsage:
    prompt_tokens: int = 0
    completion_tokens: int = 0
    cached_tokens: int = 0        # tokens served from cache
    model: str = ""
    cost_usd: float = 0.0
    timestamp: float = field(default_factory=time.time)

    @property
    def total_tokens(self) -> int:
        return self.prompt_tokens + self.completion_tokens

    @property
    def cache_hit_rate(self) -> float:
        if self.prompt_tokens == 0:
            return 0.0
        return self.cached_tokens / self.prompt_tokens

@dataclass
class AgentRunLedger:
    """Tracks all token usage across an agent run."""
    run_id: str
    user_id: str
    entries: List[TokenUsage] = field(default_factory=list)

    def record(self, usage: TokenUsage):
        self.entries.append(usage)

    @property
    def total_cost(self) -> float:
        return sum(e.cost_usd for e in self.entries)

    @property
    def total_tokens(self) -> int:
        return sum(e.total_tokens for e in self.entries)

    @property
    def step_count(self) -> int:
        return len(self.entries)

    def summary(self) -> dict:
        return {
            "run_id": self.run_id,
            "steps": self.step_count,
            "total_tokens": self.total_tokens,
            "total_cost_usd": round(self.total_cost, 4),
            "avg_tokens_per_step": self.total_tokens // max(1, self.step_count),
            "cache_hit_rate": round(
                sum(e.cached_tokens for e in self.entries) /
                max(1, sum(e.prompt_tokens for e in self.entries)), 3
            ),
        }

The AgentRunLedger is the single source of truth for a run's cost. Every LLM call — whether for planning, tool selection, or synthesis — gets recorded. This enables downstream analytics: cost-per-user, cost-per-feature, and identification of runaway agent loops.

MODEL_PRICING = {
    # model_name: (input_cost_per_1M, output_cost_per_1M)
    "gpt-4o":        (2.50, 10.00),
    "gpt-4o-mini":   (0.15,  0.60),
    "claude-sonnet":  (3.00, 15.00),
    "claude-haiku":   (0.25,  1.25),
    "llama-70b":     (0.59,  0.79),  # Fireworks / Together
    "llama-8b":      (0.10,  0.10),
}

def calculate_cost(model: str, prompt_tok: int, completion_tok: int) -> float:
    """Calculate USD cost for a single LLM call."""
    input_rate, output_rate = MODEL_PRICING[model]
    return (prompt_tok * input_rate + completion_tok * output_rate) / 1_000_000

# Example: 10-step agent run on GPT-4o
# Step 1: 2,000 in / 400 out → $0.009
# Step 5: 8,000 in / 400 out → $0.024
# Step 10: 15,000 in / 500 out → $0.042
# Total: ~$0.25 for a single agent run
Implementation tip: Wrap your LLM client with a middleware that automatically extracts usage from API responses and records it to the ledger. This prevents silent token leaks from untracked calls. Most SDKs (OpenAI, Anthropic, LiteLLM) expose usage metadata in the response object.

Budget Enforcement

Token tracking tells you what happened. Budget enforcement prevents runaway costs in real-time. A well-designed budget system operates at multiple levels: per-step, per-run, per-user, and per-organization — with graceful degradation rather than hard failures.

from enum import Enum

class BudgetAction(Enum):
    CONTINUE = "continue"
    DOWNGRADE_MODEL = "downgrade_model"
    FORCE_SYNTHESIS = "force_synthesis"
    HARD_STOP = "hard_stop"

@dataclass
class BudgetPolicy:
    max_steps: int = 15
    max_tokens_per_run: int = 50_000
    max_cost_per_run: float = 0.50     # USD
    max_cost_per_user_daily: float = 5.0
    soft_limit_pct: float = 0.70       # trigger downgrade at 70%

class BudgetEnforcer:
    def __init__(self, policy: BudgetPolicy, ledger: AgentRunLedger):
        self.policy = policy
        self.ledger = ledger

    def check(self) -> BudgetAction:
        """Called before each agent step."""
        cost = self.ledger.total_cost
        tokens = self.ledger.total_tokens
        steps = self.ledger.step_count

        # Hard limits — force immediate wrap-up
        if cost >= self.policy.max_cost_per_run:
            return BudgetAction.HARD_STOP
        if steps >= self.policy.max_steps:
            return BudgetAction.FORCE_SYNTHESIS
        if tokens >= self.policy.max_tokens_per_run:
            return BudgetAction.FORCE_SYNTHESIS

        # Soft limits — start degrading
        soft_cost = self.policy.max_cost_per_run * self.policy.soft_limit_pct
        if cost >= soft_cost:
            return BudgetAction.DOWNGRADE_MODEL

        return BudgetAction.CONTINUE

    def apply(self, action: BudgetAction, agent) -> None:
        """Apply the budget action to the running agent."""
        if action == BudgetAction.DOWNGRADE_MODEL:
            agent.model = agent.fallback_model  # e.g. gpt-4o → gpt-4o-mini
        elif action == BudgetAction.FORCE_SYNTHESIS:
            agent.force_final_answer()
        elif action == BudgetAction.HARD_STOP:
            agent.stop(reason="budget_exceeded")
Graceful degradation is key. Returning "budget exceeded, please try again" is a terrible user experience. Instead, the agent should synthesize the best possible answer from what it has gathered so far. A 70%-complete answer is infinitely better than an error message.

Prompt & Response Caching

Caching is the highest-ROI cost optimization for agents. Because agents repeat nearly identical prefixes across steps (system prompt + tool definitions + early conversation turns), prompt caching can eliminate 40-70% of input token costs.

🔁 Prompt Prefix Caching

The LLM provider caches KV states for repeated prompt prefixes. Subsequent calls with the same prefix get 50% input token discount (Anthropic) or free cached tokens (OpenAI on supported models).

  • System prompt + tool schemas → cached
  • Early conversation history → cached
  • Only new turns charged at full rate
  • Requires consistent prefix ordering

💾 Semantic Response Caching

Application-level cache stores (query → response) pairs. Exact match or embedding-based similarity avoids redundant LLM calls entirely — 100% cost elimination for cache hits.

  • Tool call results (deterministic tools)
  • Common user queries (FAQ patterns)
  • Intermediate reasoning for repeated sub-tasks
  • Requires careful invalidation strategy
import hashlib, json
from functools import lru_cache

class AgentCache:
    """Two-tier caching: exact match + semantic similarity."""

    def __init__(self, redis_client, embedding_fn, ttl=3600):
        self.redis = redis_client
        self.embed = embedding_fn
        self.ttl = ttl
        self.similarity_threshold = 0.95

    def _hash_key(self, tool_name: str, args: dict) -> str:
        """Exact-match key for deterministic tool calls."""
        payload = json.dumps({"tool": tool_name, "args": args}, sort_keys=True)
        return f"tool:{hashlib.sha256(payload.encode()).hexdigest()}"

    def get_tool_result(self, tool_name: str, args: dict):
        """Check cache before executing a tool."""
        key = self._hash_key(tool_name, args)
        cached = self.redis.get(key)
        if cached:
            return json.loads(cached)  # cache hit — zero tokens
        return None

    def store_tool_result(self, tool_name: str, args: dict, result: str):
        key = self._hash_key(tool_name, args)
        self.redis.setex(key, self.ttl, json.dumps(result))

    def get_similar_response(self, query: str):
        """Semantic cache: find similar past queries."""
        query_vec = self.embed(query)
        results = self.redis.ft_search(  # RediSearch vector similarity
            index="response_cache",
            query=query_vec,
            top_k=1
        )
        if results and results[0].score >= self.similarity_threshold:
            return results[0].response
        return None
Caching ROI: For a typical agent workload, tool-result caching achieves 30-50% hit rates (many tools return the same data), and prompt prefix caching saves 40-60% of input token costs. Combined, caching reduces total agent cost by 45-65%.

Cache invalidation rules for agents:

  • Deterministic tools (DB queries, API lookups): Cache aggressively with TTL matching data freshness SLA
  • Time-sensitive tools (web search, stock prices): Short TTL (60-300s) or no cache
  • Stateful tools (file writes, code execution): Never cache — side effects must re-execute
  • LLM reasoning: Cache only final answers, not intermediate reasoning (temperature > 0 varies outputs)

Model Routing & Cascading

Not every agent step needs GPT-4o. A model router dynamically selects the cheapest model capable of handling each sub-task, reducing costs by 60-80% while maintaining output quality where it matters.

from enum import Enum

class TaskComplexity(Enum):
    SIMPLE = "simple"      # parsing, classification, yes/no
    MEDIUM = "medium"      # tool selection, data extraction
    COMPLEX = "complex"    # multi-step reasoning, planning, synthesis

class ModelRouter:
    """Route each agent step to the cheapest adequate model."""

    MODEL_TIERS = {
        TaskComplexity.COMPLEX: "gpt-4o",       # $2.50/$10 per 1M
        TaskComplexity.MEDIUM:  "gpt-4o-mini",  # $0.15/$0.60 per 1M
        TaskComplexity.SIMPLE:  "llama-8b",     # $0.10/$0.10 per 1M
    }

    def classify_step(self, step_type: str, context: dict) -> TaskComplexity:
        """Classify agent step complexity."""
        if step_type in ("plan", "synthesize", "reflect"):
            return TaskComplexity.COMPLEX
        if step_type in ("select_tool", "extract_params", "summarize_obs"):
            return TaskComplexity.MEDIUM
        if step_type in ("parse_output", "check_done", "format_result"):
            return TaskComplexity.SIMPLE
        return TaskComplexity.MEDIUM  # default to medium

    def route(self, step_type: str, context: dict, budget_remaining: float) -> str:
        complexity = self.classify_step(step_type, context)

        # Budget pressure: downgrade if running low
        if budget_remaining < 0.10:
            return self.MODEL_TIERS[TaskComplexity.SIMPLE]
        if budget_remaining < 0.25 and complexity == TaskComplexity.COMPLEX:
            complexity = TaskComplexity.MEDIUM

        return self.MODEL_TIERS[complexity]

# Cost impact example:
# 10-step run, all GPT-4o:       $0.25
# 10-step run, routed:           $0.06  (76% reduction)
#   - 2 steps GPT-4o (plan+synth):  $0.051
#   - 5 steps GPT-4o-mini (tools):  $0.008
#   - 3 steps Llama-8B (parsing):   $0.001
Cascading pattern: Start with the cheapest model and escalate only on failure. For tool-call extraction, try Llama-8B first. If it produces invalid JSON, retry with GPT-4o-mini. If that fails, escalate to GPT-4o. Average cost drops 70% because most calls succeed on the cheapest tier.

🟢 Good Candidates for Small Models

  • JSON schema validation & reformatting
  • Binary classification (done / not done)
  • Tool output parsing & extraction
  • Simple parameter slot-filling
  • Response format compliance checks

🔴 Requires Large Models

  • Multi-hop reasoning over complex data
  • Initial plan generation for novel tasks
  • Final answer synthesis with nuance
  • Error recovery & self-correction
  • Tasks requiring broad world knowledge

Optimization Strategies

Beyond caching and routing, several architectural patterns can dramatically reduce agent costs. The key insight: most agent token spend is waste — redundant context, verbose tool outputs, unnecessary reasoning steps. Attack the waste systematically.

Context Window Management

The growing context problem is the #1 cost driver. Each agent step appends tool calls and observations to the conversation, creating a linearly growing prompt that's sent in full on every subsequent call.

class ContextManager:
    """Manage agent context window to control costs."""

    def __init__(self, max_context_tokens: int = 16_000):
        self.max_tokens = max_context_tokens
        self.system_prompt = ""       # fixed, always included
        self.pinned_messages = []     # user query, critical context
        self.sliding_window = []      # recent steps (FIFO eviction)

    def add_step(self, messages: list, token_count: int):
        self.sliding_window.append((messages, token_count))
        self._evict_if_needed()

    def _evict_if_needed(self):
        """Remove oldest steps when context exceeds limit."""
        total = self._count_tokens()
        while total > self.max_tokens and len(self.sliding_window) > 2:
            removed = self.sliding_window.pop(0)
            total -= removed[1]

    def compress_observation(self, observation: str, max_chars: int = 2000) -> str:
        """Truncate verbose tool outputs to control context growth."""
        if len(observation) <= max_chars:
            return observation
        # Keep first and last portions for context
        half = max_chars // 2
        return observation[:half] + f"\n...[truncated {len(observation) - max_chars} chars]...\n" + observation[-half:]

Reducing Agent Steps

Fewer steps means fewer LLM calls and less context accumulation. Several techniques reduce step count without sacrificing task completion:

  • Parallel tool calls: Let the LLM request multiple tools in one step instead of sequential calls. Reduces a 5-step sequence to 2-3 steps.
  • Smarter planning: A well-crafted plan prompt that anticipates tool needs reduces trial-and-error exploration by 30-40%.
  • Early termination detection: Use a lightweight classifier after each step to check if the agent has enough information to answer. Prevents unnecessary extra iterations.
  • Tool result pre-filtering: Filter and summarize tool results before returning them to the agent, reducing observation token counts by 50-70%.
Don't over-optimize reasoning. Cutting steps too aggressively degrades answer quality. The goal is to eliminate wasted steps (redundant searches, unnecessary confirmations), not productive reasoning. Monitor task success rate alongside cost metrics.

Cost-Aware Architecture Patterns

🏗️ Planner-Worker Split

Use a large model (GPT-4o) for the planning step only, then hand off execution to a small model (GPT-4o-mini) that follows the plan. The planner runs once; workers run many times.

Savings: 60-70% vs uniform GPT-4o

🔄 Speculative Execution

Run the cheap model speculatively. If its output passes a quality gate (format validation, confidence check), use it. Otherwise, re-run with the expensive model. Only 15-25% of calls escalate.

Savings: 50-65% with 98%+ quality parity

Putting It All Together

class CostOptimizedAgent:
    """Agent with full cost optimization stack."""

    def __init__(self, policy: BudgetPolicy):
        self.ledger = AgentRunLedger(run_id=uuid4(), user_id="")
        self.budget = BudgetEnforcer(policy, self.ledger)
        self.router = ModelRouter()
        self.cache = AgentCache(redis, embed_fn)
        self.context = ContextManager(max_context_tokens=16_000)

    def run(self, user_query: str) -> str:
        plan = self._plan(user_query)     # Tier 1 model

        for step in plan.steps:
            # 1. Check budget before each step
            action = self.budget.check()
            if action == BudgetAction.FORCE_SYNTHESIS:
                break
            if action == BudgetAction.HARD_STOP:
                return self._emergency_synthesis()

            # 2. Route to cheapest adequate model
            model = self.router.route(
                step.type, step.context,
                budget_remaining=self.policy.max_cost_per_run - self.ledger.total_cost
            )

            # 3. Check cache before calling LLM
            if step.is_tool_call:
                cached = self.cache.get_tool_result(step.tool, step.args)
                if cached:
                    step.result = cached
                    continue

            # 4. Execute with context management
            result = self._execute_step(step, model)
            self.context.add_step(result.messages, result.tokens)

        return self._synthesize()    # Tier 1 model for final answer
Real-world impact: Applying the full optimization stack (routing + caching + context management + budget enforcement) to a production agent system typically reduces costs by 75-85%. A $0.25 agent run becomes $0.04-$0.06, making agent-powered features economically viable at scale.

Key metrics to track in production:

  • Cost per successful task: Total spend ÷ tasks completed (not just total spend)
  • Token efficiency ratio: Useful output tokens ÷ total tokens consumed
  • Cache hit rate: Target >40% for tool caching, >60% for prefix caching
  • Model downgrade rate: How often budget pressure forces cheaper models
  • Budget exhaustion rate: % of runs hitting hard limits (target <5%)
  • Quality vs cost curve: Task success rate at each cost tier to find the optimal trade-off
