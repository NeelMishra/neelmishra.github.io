Agent Planning: ToT, GoT, and Beyond

MLOps Series LLM Agents & Orchestration

An LLM on its own generates tokens left to right—a single chain. But complex tasks like multi-step reasoning, creative writing, or code generation demand deliberate planning: exploring alternatives, backtracking from dead ends, and synthesizing insights from multiple reasoning paths. This post surveys the landscape of agent planning algorithms, from the simplicity of Chain-of-Thought prompting to the sophistication of Graph of Thought and self-reflective loops.

We'll build up from first principles, compare each strategy head-to-head, and provide production-ready Python code you can drop into your own agent pipelines.

Why Agents Need Planning

Consider a question like "Plan a 7-day itinerary for Japan covering Tokyo, Kyoto, and Osaka with budget constraints." A naive single-pass generation might produce a plausible answer, but it can't weigh trade-offs between alternatives, check feasibility constraints, or recover from a bad early choice (e.g., scheduling Kyoto temples on a Monday when they're closed).

Planning gives agents three superpowers:

  • Exploration — Generate multiple candidate paths before committing to one.
  • Evaluation — Score intermediate steps against a heuristic or verifier.
  • Backtracking — Abandon failing branches and redirect compute to promising ones.
Key insight: Planning transforms an LLM from a System 1 thinker (fast, intuitive) into a System 2 thinker (slow, deliberate). The cost is more tokens; the payoff is dramatically better accuracy on hard problems—often 20–70% improvement on benchmarks like Game of 24 and crossword puzzles.

Chain-of-Thought as a Baseline

Chain-of-Thought (CoT) prompting (Wei et al., 2022) is the simplest planning strategy: instruct the model to "think step by step" so it externalizes intermediate reasoning. While powerful, CoT is fundamentally a single linear chain—it cannot explore alternative paths or recover from a wrong intermediate step. 

# Basic Chain-of-Thought prompting
def chain_of_thought(llm, question):
    prompt = f"""Solve the following step by step.
Question: {question}
Let's think step by step:"""
    response = llm.generate(prompt, temperature=0.0)
    return response

# Self-Consistency: sample multiple CoT chains, majority vote
def self_consistency(llm, question, n_samples=5):
    answers = []
    for _ in range(n_samples):
        resp = llm.generate(
            f"Q: {question}\nLet's think step by step:",
            temperature=0.7
        )
        answers.append(extract_answer(resp))
    return majority_vote(answers)

✅ CoT Strengths

  • Zero extra infrastructure—just a prompt change
  • Works well for straightforward multi-step reasoning
  • Self-Consistency variant improves reliability via sampling

⚠️ CoT Limitations

  • Single path — one bad step derails everything
  • No backtracking or branch exploration
  • Self-Consistency is expensive (N× calls) without structured search
Watch out: CoT can produce fluent-sounding but logically incorrect chains. The model may "hallucinate" a reasoning step that looks right but is mathematically wrong. Always pair CoT with verification when accuracy matters.

Tree of Thought (ToT)

Tree of Thought (Yao et al., 2023) generalizes CoT from a single chain to a tree of reasoning paths. At each step, the model generates multiple candidate "thoughts," evaluates them with a heuristic or the LLM itself, and expands only the most promising branches. This mirrors classical AI search algorithms like BFS and DFS.

The ToT algorithm follows three phases per step:

  1. Propose — The LLM generates k candidate thoughts for the current state.
  2. Evaluate — Each candidate is scored (by the LLM or an external heuristic).
  3. Search — Use BFS or DFS to expand the best-scoring candidates, prune the rest.
import heapq
from dataclasses import dataclass, field

# ---------- Tree of Thought core ----------
@dataclass(order=True)
class ThoughtNode:
    score: float
    state: str = field(compare=False)
    path:  list = field(default_factory=list, compare=False)

def tree_of_thought(llm, problem, max_depth=3, branch_factor=3, beam_width=2):
    """BFS-style Tree of Thought search."""
    root = ThoughtNode(score=0.0, state=problem, path=[])
    frontier = [root]

    for depth in range(max_depth):
        candidates = []
        for node in frontier:
            # PROPOSE: generate k next thoughts
            thoughts = llm.generate(
                f"Problem: {problem}\nProgress: {node.state}\n"
                f"Propose {branch_factor} distinct next steps:",
                n=branch_factor, temperature=0.8
            )
            for t in thoughts:
                # EVALUATE: score each thought
                score = evaluate_thought(llm, problem, node.state, t)
                child = ThoughtNode(
                    score=score,
                    state=f"{node.state}\n→ {t}",
                    path=node.path + [t]
                )
                candidates.append(child)
        # SEARCH: keep top beam_width candidates
        frontier = heapq.nlargest(beam_width, candidates)

    best = max(frontier)
    return best.path, best.score

def evaluate_thought(llm, problem, state, thought):
    """Ask the LLM to rate a thought on a 1-10 scale."""
    prompt = (
        f"Problem: {problem}\nCurrent progress: {state}\n"
        f"Proposed next step: {thought}\n"
        f"Rate this step from 1-10 for correctness and progress. Reply with just a number."
    )
    rating = llm.generate(prompt, temperature=0.0)
    try:
        return float(rating.strip()) / 10.0
    except ValueError:
        return 0.5
Complexity note: ToT with beam width b and branch factor k over depth d requires O(b × k × d) LLM calls for generation, plus the same number for evaluation. For a typical setup (b=3, k=5, d=4), that's ~120 calls—expensive but often worth it for hard problems like Game of 24 (success rate jumps from 4% with CoT to 74% with ToT).

Graph of Thought (GoT)

Graph of Thought (Besta et al., 2023) extends ToT by allowing merging of reasoning paths. In a tree, each node has exactly one parent; in a graph, a node can aggregate insights from multiple branches. This is critical for problems where partial solutions from different paths need to be combined—like merging sorted sub-lists or synthesizing arguments from multiple perspectives.

GoT introduces three operations beyond ToT:

  • Aggregate — Merge partial results from multiple branches into one node.
  • Refine — Improve an existing thought without branching (in-place edit).
  • Generate — Same as ToT: produce new candidate thoughts from a node.
from collections import defaultdict

# ---------- Graph of Thought ----------
class GoTNode:
    def __init__(self, id, content, parents=None):
        self.id = id
        self.content = content
        self.parents = parents or []
        self.score = 0.0
        self.children = []

class GraphOfThought:
    def __init__(self, llm):
        self.llm = llm
        self.nodes = {}
        self.counter = 0

    def generate(self, node_id, k=3):
        """Produce k child thoughts from a node."""
        parent = self.nodes[node_id]
        thoughts = self.llm.generate(
            f"Current state:\n{parent.content}\nGenerate {k} next steps:",
            n=k, temperature=0.7
        )
        children = []
        for t in thoughts:
            child = self._add_node(t, parents=[parent])
            children.append(child)
        return children

    def aggregate(self, node_ids):
        """Merge insights from multiple nodes into one."""
        parents = [self.nodes[nid] for nid in node_ids]
        combined = "\n---\n".join(p.content for p in parents)
        merged = self.llm.generate(
            f"Synthesize these partial solutions into one coherent answer:\n{combined}"
        )
        return self._add_node(merged, parents=parents)

    def refine(self, node_id, feedback):
        """Improve a node in-place given feedback."""
        node = self.nodes[node_id]
        improved = self.llm.generate(
            f"Original:\n{node.content}\nFeedback: {feedback}\nImprove:"
        )
        node.content = improved
        return node

    def _add_node(self, content, parents):
        nid = f"n_{self.counter}"
        self.counter += 1
        node = GoTNode(nid, content, parents)
        self.nodes[nid] = node
        for p in parents:
            p.children.append(node)
        return node

🌳 Tree of Thought

  • Each node has exactly one parent
  • Explores via BFS/DFS with pruning
  • Best for: problems with clear step sequences (Game of 24, puzzles)
  • Simpler to implement and debug

🕸️ Graph of Thought

  • Nodes can have multiple parents (merge)
  • Supports aggregate + refine operations
  • Best for: synthesis tasks (sorting, summarization, debate)
  • Higher complexity, more flexible topology

Self-Reflection & Refinement

Planning isn't just about forward search—it's also about looking back. Self-reflection strategies (like Reflexion by Shinn et al., 2023) let an agent evaluate its own output, identify errors, and iteratively refine its answer. This creates a feedback loop where the agent serves as both generator and critic.

The reflexion loop has three components:

  1. Actor — Generates an initial response or takes an action.
  2. Evaluator — Checks the response against criteria (correctness, completeness, format).
  3. Reflector — Produces a verbal critique that guides the next attempt.
# ---------- Reflexion loop ----------
class ReflexionAgent:
    def __init__(self, llm, evaluator, max_retries=3):
        self.llm = llm
        self.evaluator = evaluator
        self.max_retries = max_retries
        self.memory = []  # stores reflections

    def solve(self, task):
        for attempt in range(self.max_retries):
            # ACT: generate response with reflection history
            context = self._build_context(task)
            response = self.llm.generate(context)

            # EVALUATE: check correctness
            result = self.evaluator.check(task, response)
            if result.passed:
                return response

            # REFLECT: generate critique
            reflection = self.llm.generate(
                f"Task: {task}\nYour answer: {response}\n"
                f"Errors found: {result.errors}\n"
                f"Reflect on what went wrong and how to fix it:"
            )
            self.memory.append({
                "attempt": attempt,
                "response": response,
                "errors": result.errors,
                "reflection": reflection
            })
        return response  # return best effort

    def _build_context(self, task):
        ctx = f"Task: {task}\n"
        if self.memory:
            ctx += "Previous reflections:\n"
            for m in self.memory:
                ctx += f"- Attempt {m['attempt']}: {m['reflection']}\n"
            ctx += "Avoid the same mistakes. Provide a corrected answer:\n"
        return ctx
Reflexion in practice: On the HumanEval code generation benchmark, Reflexion improves pass@1 from 80% (GPT-4 baseline) to 91% by giving the model up to 3 retries with its own error analysis as feedback. The key is that reflections are stored in an episodic memory buffer, so the agent learns from its mistakes within a single episode.

🔄 Reflexion

  • Verbal self-critique stored in memory
  • Works on any task with a verifiable output
  • Episodic: reflections reset per task
  • Cost: 2–3× base LLM calls per retry

🧪 Self-Refine

  • No external evaluator needed—model critiques itself
  • Single prompt loop: generate → critique → refine
  • Best for open-ended tasks (writing, summarization)
  • Risk: model may not catch its own logical errors

Task Decomposition Strategies

Complex goals are rarely solvable in a single reasoning chain. Task decomposition breaks a high-level objective into smaller, manageable sub-tasks that can be solved independently and then composed. This is the backbone of frameworks like HuggingGPT, Chameleon, and Plan-and-Solve.

Common Decomposition Patterns

📋 Sequential Decomposition

Break the task into an ordered list of sub-tasks. Each sub-task receives the output of the previous one as input. Used in Plan-and-Solve and most ReAct-style agents.

# Plan-and-Solve style
plan = llm("Break this task into steps:\n{task}")
result = None
for step in plan:
    result = llm(f"Do: {step}\nContext: {result}")

🔀 Parallel Decomposition

Identify independent sub-tasks that can run concurrently, then merge their outputs. Used in HuggingGPT for multi-modal tasks (e.g., image + text analysis).

# Parallel sub-task execution
import asyncio
subtasks = llm("Split into independent parts:\n{task}")
results = await asyncio.gather(
    *[solve(st) for st in subtasks]
)
final = llm(f"Combine:\n{results}")

Recursive Decomposition with Depth Control

For truly complex tasks, a single level of decomposition isn't enough. Recursive decomposition applies the process hierarchically—each sub-task can itself be decomposed—until every leaf is simple enough to solve directly. 

def recursive_solve(llm, task, depth=0, max_depth=3):
    """Recursively decompose and solve a complex task."""
    # Base case: task is simple enough to solve directly
    if depth >= max_depth or is_atomic(llm, task):
        return llm.generate(f"Solve directly:\n{task}")

    # Decompose into sub-tasks
    subtasks = llm.generate(
        f"Break this into 2-4 smaller sub-tasks:\n{task}\n"
        f"Return as a numbered list."
    )
    parsed = parse_numbered_list(subtasks)

    # Solve each sub-task recursively
    sub_results = []
    for st in parsed:
        result = recursive_solve(llm, st, depth + 1, max_depth)
        sub_results.append(result)

    # Compose results
    return llm.generate(
        f"Original task: {task}\n"
        f"Sub-results:\n" + "\n".join(sub_results) +
        f"\nSynthesize into a final answer:"
    )

def is_atomic(llm, task):
    """Ask the LLM if a task is simple enough to solve directly."""
    response = llm.generate(
        f"Can this task be solved in a single step without decomposition?\n"
        f"Task: {task}\nAnswer YES or NO."
    )
    return "YES" in response.upper()
Decomposition pitfall: Over-decomposition creates more problems than it solves. If you split "calculate 2+3" into sub-tasks, you're wasting tokens and introducing aggregation errors. Use the is_atomic check to prevent unnecessary recursion, and always cap recursion depth.

Planning in Practice

Let's bring everything together into a unified planner that can switch between strategies based on task complexity. This is the pattern used by production agent frameworks like LangGraph and AutoGen.

Adaptive Strategy Selection

Not every task needs a full Graph of Thought. A good meta-planner classifies the task first, then dispatches to the cheapest strategy that's likely to succeed. 

from enum import Enum

class PlanStrategy(Enum):
    COT      = "chain_of_thought"
    TOT      = "tree_of_thought"
    GOT      = "graph_of_thought"
    REFLEXION = "reflexion"

class AdaptivePlanner:
    def __init__(self, llm):
        self.llm = llm
        self.strategies = {
            PlanStrategy.COT: chain_of_thought,
            PlanStrategy.TOT: tree_of_thought,
            PlanStrategy.GOT: GraphOfThought(llm),
            PlanStrategy.REFLEXION: ReflexionAgent(llm, CodeEvaluator()),
        }

    def classify_task(self, task):
        """Determine the best planning strategy for a task."""
        analysis = self.llm.generate(
            f"""Classify this task's complexity:
Task: {task}

Categories:
- SIMPLE: single-step, factual, or arithmetic → chain_of_thought
- SEARCH: multi-step with clear right/wrong → tree_of_thought
- SYNTHESIS: combine multiple perspectives → graph_of_thought
- VERIFIABLE: has testable output (code, math) → reflexion

Reply with just the category name."""
        )
        mapping = {
            "SIMPLE": PlanStrategy.COT,
            "SEARCH": PlanStrategy.TOT,
            "SYNTHESIS": PlanStrategy.GOT,
            "VERIFIABLE": PlanStrategy.REFLEXION,
        }
        return mapping.get(analysis.strip(), PlanStrategy.COT)

    def solve(self, task):
        strategy = self.classify_task(task)
        print(f"Selected strategy: {strategy.value}")
        solver = self.strategies[strategy]
        if callable(solver):
            return solver(self.llm, task)
        return solver.solve(task)

Cost–Accuracy Trade-off Matrix

Choosing the right strategy is fundamentally about balancing token cost against accuracy gains. Here's a practical reference:

💰 Low Cost

  • CoT — 1× LLM call, no extra infra
  • Self-Consistency — 5–10× calls, no search logic
  • Best for: simple reasoning, factual QA, arithmetic

💎 High Accuracy

  • ToT — 50–200× calls, requires search
  • GoT — 100–500× calls, graph management
  • Best for: hard puzzles, code gen, complex synthesis
Production tip: Start every agent pipeline with CoT. Escalate to ToT or Reflexion only when CoT fails validation checks. This "escalation ladder" pattern gives you 80% of the accuracy gains at 20% of the cost, because most real-world tasks are simple enough for CoT. Reserve GoT for problems that explicitly require merging parallel reasoning paths.

Putting It All Together: Full Pipeline

# ---------- End-to-end agent planning pipeline ----------
import logging

log = logging.getLogger("agent_planner")

def agent_pipeline(llm, task, max_escalations=3):
    """Escalation-based planning pipeline."""
    strategies = [
        ("CoT",       lambda t: chain_of_thought(llm, t)),
        ("Reflexion",  lambda t: ReflexionAgent(llm, AutoEval()).solve(t)),
        ("ToT",       lambda t: tree_of_thought(llm, t, max_depth=4, beam_width=3)),
    ]

    for name, solver in strategies[:max_escalations]:
        log.info(f"Trying strategy: {name}")
        result = solver(task)

        # Validate result
        is_valid = llm.generate(
            f"Task: {task}\nAnswer: {result}\n"
            f"Is this answer correct and complete? Reply YES or NO with reason."
        )
        if "YES" in is_valid.upper():
            log.info(f"✓ Solved with {name}")
            return {"strategy": name, "result": result, "valid": True}
        log.warning(f"✗ {name} failed validation, escalating...")

    return {"strategy": "exhausted", "result": result, "valid": False}

Key Takeaways

  • CoT is your baseline — cheap, effective for 70% of tasks. Always start here.
  • ToT adds search — when a single chain isn't reliable, explore multiple paths with pruning.
  • GoT adds merging — when you need to synthesize from parallel branches, upgrade from tree to graph.
  • Reflexion adds learning — for tasks with verifiable outputs, let the agent critique and retry.
  • Decomposition is orthogonal — layer it on top of any strategy to handle complex, multi-part goals.
  • Use an escalation ladder — don't pay for ToT when CoT suffices. Route dynamically based on task complexity.
The bottom line: Agent planning transforms LLMs from single-shot generators into deliberate reasoners. The field is moving fast—new architectures like Algorithm of Thoughts (AoT) and Monte Carlo Tree Search (MCTS) variants are appearing monthly. But the core principles remain: explore alternatives, evaluate intermediate steps, and backtrack from failures. Master CoT → ToT → GoT → Reflexion, and you have the toolkit to tackle any planning problem in production.
