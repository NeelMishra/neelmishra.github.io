Workflow Orchestration: LangGraph, Temporal

MLOps Series LLM Agents & Orchestration

Why Workflow Orchestration?

A single LLM call is stateless: prompt in, completion out. But production agents are multi-step systems that chain tool calls, maintain context, branch on intermediate results, and recover from failures. Without orchestration, agent pipelines devolve into fragile nested if/else trees that are impossible to debug, test, or resume after a crash.

Workflow orchestration provides three critical guarantees that raw function composition cannot:

  • State management — a durable, inspectable snapshot of where the agent is and what it has seen
  • Fault tolerance — automatic retries, timeouts, and the ability to resume from the last checkpoint after a process restart
  • Observability — a structured execution graph that makes it possible to trace every decision an agent made and why
# Without orchestration — fragile, no recovery
def naive_agent(user_query):
    plan = llm_call("plan", user_query)          # what if this times out?
    for step in plan.steps:
        result = execute_tool(step)               # what if tool fails on step 4 of 7?
        if result.needs_replan:
            plan = llm_call("replan", result)   # how do we replay from here?
    return llm_call("summarize", results)

# With orchestration — each step is a durable, resumable node
# If the process crashes at step 4, the orchestrator replays from checkpoint
# Every transition is logged, branching logic is explicit in the graph
The core insight: An LLM agent is not a function — it is a stateful process with non-deterministic transitions. Orchestration frameworks make this process explicit, durable, and debuggable. The right abstraction is not a pipeline but a state machine or a directed graph.

Two dominant paradigms have emerged for orchestrating LLM agent workflows: graph-based orchestration (LangGraph, LlamaIndex Workflows) and durable execution engines (Temporal, Restate). They solve overlapping but distinct problems, and many production systems use both.

State Machines for Agents

Before diving into frameworks, it helps to formalize what an agent workflow is. Every multi-step agent can be modeled as a finite state machine (FSM) where states represent phases of execution and transitions are triggered by LLM outputs, tool results, or control logic.

The key properties of an agent FSM:

  • States are typed — each node carries a specific data schema (e.g., PlanState holds the step list, ExecuteState holds tool call results)
  • Transitions can be conditional — the Evaluate node may loop back to Plan (replan) or proceed to Respond (done)
  • State is serializable — at any point, the entire agent context can be serialized to JSON and restored later
# Defining agent state as a typed structure
from typing import TypedDict, List, Literal

class AgentState(TypedDict):
    messages: List[dict]            # conversation history
    plan: List[str]                 # current execution plan steps
    tool_results: List[dict]        # collected tool outputs
    current_step: int               # index into plan
    status: Literal["planning", "executing", "evaluating", "done"]
    retry_count: int                # number of replanning attempts

# State transitions are pure functions: State → State
def plan_node(state: AgentState) -> AgentState:
    plan = llm_call("generate_plan", state["messages"])
    return {**state, "plan": plan.steps, "status": "executing", "current_step": 0}

def execute_node(state: AgentState) -> AgentState:
    step = state["plan"][state["current_step"]]
    result = run_tool(step)
    return {**state,
        "tool_results": state["tool_results"] + [result],
        "current_step": state["current_step"] + 1}

def evaluate_router(state: AgentState) -> str:
    # Conditional edge: decides next node
    if state["current_step"] < len(state["plan"]):
        return "execute"       # more steps remaining
    verdict = llm_call("evaluate_results", state)
    if verdict.needs_replan and state["retry_count"] < 3:
        return "plan"          # loop back
    return "respond"           # proceed to final answer
Beware infinite loops: Conditional edges that loop back (e.g., evaluate → plan) must have a bounded retry count. Without it, a confused LLM can trigger infinite replanning cycles. Always include a max_retries guard and a fallback exit path.

LangGraph: Graph-Based Workflows

LangGraph (from LangChain) is the most widely adopted framework for building LLM agent workflows as explicit directed graphs. It models agents as a StateGraph where nodes are Python functions and edges define the flow — including conditional branching and cycles.

The core abstraction: a StateGraph over a typed state dictionary. Each node reads the current state, performs work (LLM calls, tool use, logic), and returns state updates. Edges can be static (always go to node B after node A) or conditional (pick the next node based on state).

from langgraph.graph import StateGraph, END
from langgraph.checkpoint.sqlite import SqliteSaver

# 1. Define the graph
graph = StateGraph(AgentState)

# 2. Add nodes — each is a function (state) → partial state update
graph.add_node("plan",     plan_node)
graph.add_node("execute",  execute_node)
graph.add_node("evaluate", evaluate_node)
graph.add_node("respond",  respond_node)

# 3. Add edges — define the flow
graph.set_entry_point("plan")
graph.add_edge("plan", "execute")                # plan → execute (always)
graph.add_edge("execute", "evaluate")             # execute → evaluate (always)

# 4. Conditional edge — evaluate decides next node
graph.add_conditional_edges(
    "evaluate",
    evaluate_router,                               # function returning node name
    {"plan": "plan", "execute": "execute", "respond": "respond"}
)
graph.add_edge("respond", END)

# 5. Compile with a checkpointer for durability
checkpointer = SqliteSaver.from_conn_string("agent.db")
app = graph.compile(checkpointer=checkpointer)

# 6. Run — state is automatically persisted after each node
result = app.invoke(
    {"messages": [{"role": "user", "content": query}]},
    config={"configurable": {"thread_id": "conv-123"}}
)

LangGraph's key design choices:

  • State reducers — instead of replacing state, nodes return partial updates that are merged via reducer functions (e.g., messages are appended, not replaced)
  • First-class cycles — unlike DAG-only frameworks, LangGraph explicitly supports loops (evaluate → plan → execute → evaluate) which are essential for agent self-correction
  • Human-in-the-loop — built-in interrupt_before / interrupt_after hooks that pause execution, serialize state, and wait for human approval before continuing
  • Streaming — node outputs stream as they complete, enabling real-time UI updates
When to use LangGraph: LangGraph excels when the workflow is LLM-centric — most nodes involve model calls, tool use, or prompt chaining. It provides native integrations with LangChain's tool/model ecosystem and handles state serialization automatically. For workflows where the complexity is in the agent logic rather than infrastructure reliability, LangGraph is the right choice.

Temporal: Durable Execution

Temporal approaches orchestration from the opposite direction. Instead of modeling the workflow as a graph, Temporal lets you write ordinary sequential code and makes it durable — surviving process crashes, deployments, and infrastructure failures. The key abstraction: workflows are deterministic functions, and activities are the side-effecting operations (LLM calls, tool execution, database writes).

from temporalio import workflow, activity
from datetime import timedelta

# Activities: side-effecting operations (retried automatically)
@activity.defn
async def call_llm(prompt: str, model: str) -> str:
    response = await openai.chat.completions.create(
        model=model, messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

@activity.defn
async def execute_tool(tool_name: str, args: dict) -> dict:
    return await tool_registry[tool_name].run(**args)

# Workflow: deterministic orchestration logic
@workflow.defn
class AgentWorkflow:
    @workflow.run
    async def run(self, user_query: str) -> str:
        # Step 1: Generate plan (activity — retried on failure)
        plan = await workflow.execute_activity(
            call_llm,
            args=[f"Plan steps for: {user_query}", "gpt-4o"],
            start_to_close_timeout=timedelta(seconds=60),
            retry_policy=RetryPolicy(maximum_attempts=3)
        )

        # Step 2: Execute each tool (with per-step retry)
        results = []
        for step in parse_plan(plan):
            result = await workflow.execute_activity(
                execute_tool,
                args=[step.tool, step.args],
                start_to_close_timeout=timedelta(seconds=120),
                retry_policy=RetryPolicy(maximum_attempts=2)
            )
            results.append(result)

        # Step 3: Synthesize response
        return await workflow.execute_activity(
            call_llm,
            args=[f"Summarize: {results}", "gpt-4o"],
            start_to_close_timeout=timedelta(seconds=60)
        )

Temporal's execution model is fundamentally different from graph-based frameworks:

  • Event sourcing — the workflow history is an append-only log of events. On replay, Temporal re-executes the workflow code but short-circuits completed activities by returning their recorded results
  • Deterministic replay — workflow code must be deterministic (no random, no clock reads). All non-determinism is pushed into activities
  • Unlimited duration — workflows can run for days, weeks, or months. A "wait for human approval" step is a workflow.wait_condition() that may block for hours without consuming compute
  • Language-native — no DSL or graph definition; the workflow is the Python code, with standard control flow (if, for, try/except)

🔷 LangGraph

  • Abstraction: Directed graph (nodes + edges)
  • State: Typed dict with reducers
  • Durability: Checkpointer (SQLite, Postgres)
  • Cycles: First-class conditional loops
  • Best for: LLM-centric agent logic, rapid prototyping, LangChain ecosystem
  • Limitation: Single-process; no distributed workers, no built-in task queues

🔶 Temporal

  • Abstraction: Sequential code + activities
  • State: Event-sourced workflow history
  • Durability: Server-managed (Cassandra/Postgres)
  • Cycles: Native loops in code
  • Best for: Long-running, mission-critical workflows, multi-service orchestration
  • Limitation: Requires Temporal server infra; determinism constraints on workflow code
Not an either/or choice: Many production systems use both. A common pattern is Temporal as the outer orchestrator (handling retries, timeouts, human-in-the-loop) with LangGraph running the inner agent logic within a Temporal activity. This gives you infrastructure-grade reliability wrapping LLM-native graph logic.

Durability & Checkpointing

Durability is the property that distinguishes a production agent from a demo. A durable agent can survive process restarts, resume from the last successful step, and maintain conversation state across sessions. The two frameworks achieve this very differently.

LangGraph checkpointing: After each node completes, the full state dict is serialized and stored via a pluggable Checkpointer backend. On resume, the graph loads the last checkpoint and continues from the next node. Thread IDs enable multi-conversation persistence.

# LangGraph: checkpointing strategies
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.checkpoint.postgres import PostgresSaver

# Development: SQLite (file-based, zero config)
dev_checkpointer = SqliteSaver.from_conn_string("checkpoints.db")

# Production: Postgres (shared across workers)
prod_checkpointer = PostgresSaver.from_conn_string("postgres://...")

app = graph.compile(checkpointer=prod_checkpointer)

# Resume after crash — just re-invoke with same thread_id
config = {"configurable": {"thread_id": "conv-123"}}
result = app.invoke(None, config=config)  # None → resume from checkpoint

# Time-travel: inspect any past checkpoint
history = app.get_state_history(config)
for snapshot in history:
    print(snapshot.metadata["step"], snapshot.values["status"])

Temporal durability: Temporal uses event sourcing. Every activity completion, timer firing, and signal received is recorded as an event in the workflow history. On replay, the workflow function re-executes but completed activities return their cached results instantly.

# Temporal: durability is automatic — no explicit checkpointing code
# The workflow history might look like:
#
# Event 1: WorkflowExecutionStarted {input: "user query"}
# Event 2: ActivityTaskScheduled  {activity: "call_llm", input: "Plan..."}
# Event 3: ActivityTaskCompleted  {result: "Step 1: search, Step 2: ..."}
# Event 4: ActivityTaskScheduled  {activity: "execute_tool", input: "search"}
# Event 5: ActivityTaskCompleted  {result: {data: [...]}}
# Event 6: ActivityTaskScheduled  {activity: "execute_tool", input: "analyze"}
# — crash here —
# On recovery, Temporal replays events 1-5 (skipping actual execution)
# Then re-schedules event 6 (the failed activity)

# Key advantage: workflow code doesn't need ANY checkpoint logic
# The Temporal server handles it transparently
Checkpoint size matters: LangGraph serializes the entire state dict at each checkpoint. For agents with large message histories or tool outputs, this can grow to megabytes per checkpoint. Consider pruning old messages or storing large artifacts by reference (e.g., S3 URIs) rather than inline. Temporal avoids this by recording only activity inputs/outputs, not the full workflow state.

Common Orchestration Patterns

Regardless of framework choice, production agent systems converge on a small set of proven orchestration patterns. Understanding these patterns helps you design workflows that are robust, testable, and maintainable.

Pattern 1: Router → Specialist

A lightweight router LLM classifies the user intent and dispatches to a specialized sub-graph. Each specialist handles a narrow domain with dedicated tools and prompts.

# Router → Specialist pattern in LangGraph
def router_node(state: AgentState) -> AgentState:
    intent = llm_call("classify intent", state["messages"][-1])
    return {**state, "intent": intent.category}

def route_dispatch(state: AgentState) -> str:
    mapping = {
        "code_gen":    "code_specialist",
        "data_query":  "sql_specialist",
        "research":    "rag_specialist",
        "chitchat":    "respond",
    }
    return mapping.get(state["intent"], "respond")

graph.add_conditional_edges("router", route_dispatch, {
    "code_specialist": "code_specialist",
    "sql_specialist":  "sql_specialist",
    "rag_specialist":  "rag_specialist",
    "respond":         "respond",
})

Pattern 2: Plan-Execute-Reflect Loop

The most common agentic pattern. The agent generates a plan, executes it step by step, then reflects on the results to decide whether to replan or finalize. This is the ReAct pattern generalized to multi-step plans.

🔄 Loop Characteristics

  • Bounded iterations — max 3-5 replanning cycles
  • Monotonic progress — each cycle should make forward progress
  • Explicit exit conditions — either task complete or max retries exhausted
  • State accumulation — tool results persist across iterations

⚠️ Failure Modes

  • Infinite loops — LLM keeps replanning without converging
  • State bloat — accumulating too many tool results crashes serialization
  • Hallucinated tools — plan references tools that don't exist
  • Lost context — later iterations forget early findings

Pattern 3: Parallel Fan-Out / Fan-In

Some agent steps are independent and can execute concurrently. A fan-out node spawns parallel branches (e.g., search three different sources simultaneously), and a fan-in node aggregates the results before proceeding.

# Fan-out / fan-in in LangGraph using Send()
from langgraph.constants import Send

def fan_out_searches(state: AgentState) -> list:
    # Dynamically spawn parallel branches
    queries = state["search_queries"]
    return [Send("search_worker", {"query": q}) for q in queries]

def search_worker(state: dict) -> dict:
    results = web_search(state["query"])
    return {"search_results": results}

def fan_in_aggregate(state: AgentState) -> AgentState:
    # All parallel results are merged via the state reducer
    combined = deduplicate(state["search_results"])
    return {**state, "context": combined}

Pattern 4: Human-in-the-Loop Gates

High-stakes operations (database writes, emails, financial transactions) require human approval before execution. Both frameworks support this natively:

# LangGraph: interrupt before a sensitive node
app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["execute_write"]       # pause here
)

# Execution pauses, state is checkpointed
# Frontend shows user: "Agent wants to execute: DELETE FROM users WHERE ..."
# User approves → resume with:
app.invoke(None, config=config)              # continue from checkpoint

# Temporal: use signals for human-in-the-loop
@workflow.defn
class ApprovalWorkflow:
    def __init__(self):
        self.approved = None

    @workflow.signal
    async def approve(self, decision: bool):
        self.approved = decision

    @workflow.run
    async def run(self, action: str):
        # Wait indefinitely for human signal (no compute consumed)
        await workflow.wait_condition(lambda: self.approved is not None)
        if self.approved:
            return await workflow.execute_activity(execute_action, args=[action])
        return "Action rejected by human reviewer"

Pattern 5: Hierarchical Agent Delegation

A supervisor agent breaks a complex task into sub-tasks, delegates each to a child agent (which may itself be a full LangGraph sub-graph or Temporal child workflow), and aggregates the results. This enables compositional agent architectures where each sub-agent is independently testable.

# Hierarchical delegation in Temporal
@workflow.defn
class SupervisorWorkflow:
    @workflow.run
    async def run(self, task: str) -> str:
        subtasks = await workflow.execute_activity(
            call_llm, args=[f"Decompose: {task}", "gpt-4o"]
        )

        # Spawn child workflows — each runs independently
        handles = []
        for sub in parse_subtasks(subtasks):
            handle = await workflow.start_child_workflow(
                SpecialistWorkflow.run,
                args=[sub],
                id=f"specialist-{sub.id}"
            )
            handles.append(handle)

        # Wait for all children to complete
        results = [await h.result() for h in handles]

        # Synthesize final answer
        return await workflow.execute_activity(
            call_llm, args=[f"Combine results: {results}", "gpt-4o"]
        )
Choosing patterns wisely: Start with the simplest pattern that works. A single plan-execute-reflect loop handles 80% of agent use cases. Add router-specialist or fan-out only when you have clear evidence that a single agent can't handle the task diversity or latency requirements. Hierarchical delegation adds the most complexity and should be reserved for genuinely decomposable tasks where sub-agents need distinct tool sets.
