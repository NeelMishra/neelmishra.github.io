Tool Use: APIs, Code Execution, Retrieval

MLOps Series LLM Agents & Orchestration

LLMs are powerful text generators, but they cannot natively browse the web, query a database, or execute code. Tool use bridges this gap by giving agents a structured vocabulary of actions they can invoke. When an agent decides it needs external information or wants to perform a side effect, it emits a tool call—a structured request that the runtime intercepts, executes, and returns back as context for the next generation step.

This post covers the full spectrum of tool-use patterns: how tools are defined, the function-calling protocol that enables them, the three major tool families (APIs, code execution, retrieval), and the engineering practices that keep tool-augmented agents reliable in production.

Defining Tools for LLMs

A tool definition is a contract between the agent and the runtime. It tells the LLM what a tool does, what parameters it expects, and what it returns. The de-facto standard is a JSON Schema object embedded in the system prompt or passed via a dedicated API field.

Every tool definition should have three essential elements: a unique name, a concise description the model reads to decide when to use the tool, and a parameters object describing each argument with its type, constraints, and defaults. 

# Tool definition using OpenAI-style JSON schema

import json

weather_tool = {
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city. Returns temperature, humidity, and conditions.",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "City name, e.g. 'San Francisco'"
                },
                "units": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "Temperature unit (default: celsius)"
                }
            },
            "required": ["city"]
        }
    }
}

# Register tool with the API call
response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    tools=[weather_tool],
    tool_choice="auto"   # let the model decide
)
Tip: The description field is the most important part of a tool definition. Models use it to decide when and how to call the tool. A vague description leads to incorrect tool selection; an overly long one wastes tokens. Aim for one to two clear sentences.

Schema Design Principles

  • Flat over nested — models handle flat parameter objects more reliably than deeply nested structures.
  • Enum over free-text — constrain values with enums wherever possible to reduce hallucinated arguments.
  • Minimal required fields — mark only truly essential parameters as required; provide sensible defaults for the rest.
  • Descriptive names — search_knowledge_base is better than search when multiple search tools exist.

Function Calling Protocol

Function calling is the communication protocol between the LLM and the tool runtime. When the model decides a tool is needed, it generates a structured JSON object instead of (or alongside) natural language. The runtime parses this object, executes the corresponding function, and feeds the result back into the conversation as a new message with role tool.

✅ tool_choice: "auto"

The model decides whether to call a tool or respond directly. Best for general assistants where the user query may or may not need a tool.

⚙ tool_choice: "required"

Forces the model to emit at least one tool call. Use for pipelines where every turn must produce a structured action (e.g., routing agents).

🎯 tool_choice: {"name": "…"}

Forces a specific tool. Useful when the orchestrator already knows which tool is needed and wants the LLM only to fill in parameters.

🚫 tool_choice: "none"

Disables tool use for this turn. Helpful for final summarization steps where you want a pure text response.

Parallel & Sequential Tool Calls

Modern models support parallel tool calls—emitting multiple tool-call objects in a single generation turn. The runtime executes them concurrently and returns all results before the next LLM turn. This drastically reduces latency for independent operations like fetching data from multiple APIs.

Key insight: Parallel tool calls are ideal for independent operations (e.g., fetching weather for three cities). For dependent operations (e.g., search → then read the top result), sequential calls across multiple turns are necessary.

API Tools: REST, GraphQL, Databases

API tools let agents interact with external services. The three most common patterns are REST endpoints, GraphQL queries, and direct database access. Each requires careful design to balance flexibility with safety.

REST API Tools

A REST tool wraps a single endpoint. The tool definition maps LLM parameters to HTTP request fields (path parameters, query strings, request body). The runtime handles authentication, rate limiting, and response parsing. 

# REST API tool with auth and retry logic

import httpx
from tenacity import retry, stop_after_attempt, wait_exponential

class RESTTool:
    def __init__(self, name, base_url, api_key):
        self.name = name
        self.client = httpx.AsyncClient(
            base_url=base_url,
            headers={"Authorization": f"Bearer {api_key}"},
            timeout=30.0
        )

    @retry(stop=stop_after_attempt(3), wait=wait_exponential(min=1, max=10))
    async def execute(self, method, path, params=None, body=None):
        """Execute an API call with automatic retry on transient errors."""
        response = await self.client.request(method, path, params=params, json=body)
        response.raise_for_status()
        return {
            "status": response.status_code,
            "data": response.json(),
            "headers": dict(response.headers)
        }

    def to_schema(self):
        """Generate OpenAI-compatible tool schema."""
        return {
            "type": "function",
            "function": {
                "name": self.name,
                "description": f"Call the {self.name} REST API",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "method": {"type": "string", "enum": ["GET", "POST"]},
                        "path":   {"type": "string"},
                        "params": {"type": "object"},
                        "body":   {"type": "object"}
                    },
                    "required": ["method", "path"]
                }
            }
        }

GraphQL Tools

GraphQL tools are particularly powerful because the agent can request exactly the fields it needs, reducing token waste from over-fetching. The tool schema exposes the query string and variables as parameters, and the runtime handles execution against the GraphQL endpoint.

Database Tools

Security warning: Giving an LLM direct SQL access is dangerous. Always use a read-only connection, parameterized queries, row-limit constraints, and an allow-list of tables/columns. Never let raw LLM output execute as arbitrary SQL without validation.

Database tools typically expose a run_query function that accepts a SQL string. The runtime should validate the query against a grammar checker, enforce read-only mode, and cap result size before returning rows as JSON.

REST Tools

  • One tool per endpoint
  • Simple parameter mapping
  • Easy to rate-limit per tool
  • Stateless by default

GraphQL Tools

  • Single endpoint, flexible queries
  • Agent selects fields → fewer tokens
  • Harder to validate arbitrary queries
  • Requires schema introspection

Database Tools

  • Full query expressiveness
  • High security risk if unbounded
  • Need read-only + allow-list
  • Results can be very large

Hybrid Approach

  • Pre-built query templates
  • Agent fills in parameters only
  • Safe + flexible balance
  • Easy to audit and version

Code Execution as a Tool

Code execution tools let agents write and run code in a sandboxed environment. This unlocks capabilities like data analysis, mathematical computation, chart generation, and file manipulation that are impossible with text generation alone.

Sandbox Architecture

Production code execution tools use containerized sandboxes (e.g., gVisor, Firecracker microVMs, or Docker with seccomp profiles). The sandbox must enforce strict resource limits:

  • CPU time: 30–60 second wall-clock timeout
  • Memory: 256–512 MB cap to prevent OOM attacks
  • Network: disabled or restricted to an allow-list of domains
  • Filesystem: ephemeral and destroyed after each execution
  • System calls: filtered via seccomp to block dangerous operations
# Code execution tool with sandboxing

import subprocess, json, tempfile, os

class CodeExecutionTool:
    MAX_TIMEOUT = 30       # seconds
    MAX_OUTPUT  = 10_000   # characters

    def execute(self, code: str, language: str = "python") -> dict:
        """Run code in an isolated subprocess with resource limits."""
        if language != "python":
            return {"error": f"Unsupported language: {language}"}

        # Write code to temp file
        work_dir = tempfile.mkdtemp(prefix="sandbox_")
        code_path = os.path.join(work_dir, "script.py")
        with open(code_path, "w") as f:
            f.write(code)

        try:
            result = subprocess.run(
                ["python", code_path],
                capture_output=True,
                text=True,
                timeout=self.MAX_TIMEOUT,
                cwd=work_dir,
                env={"PATH": "/usr/bin"}  # minimal env
            )
            return {
                "stdout": result.stdout[:self.MAX_OUTPUT],
                "stderr": result.stderr[:self.MAX_OUTPUT],
                "exit_code": result.returncode
            }
        except subprocess.TimeoutExpired:
            return {"error": "Execution timed out", "exit_code": -1}
        finally:
            # Clean up sandbox directory
            import shutil
            shutil.rmtree(work_dir, ignore_errors=True)

    def to_schema(self):
        return {
            "type": "function",
            "function": {
                "name": "execute_code",
                "description": "Execute Python code in a sandboxed environment. Use for math, data analysis, or file generation.",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "code": {"type": "string", "description": "Python code to execute"},
                        "language": {"type": "string", "enum": ["python"], "default": "python"}
                    },
                    "required": ["code"]
                }
            }
        }
When to use code execution: Code execution excels at tasks with deterministic answers: arithmetic, data transformations, regex operations, and generating visualizations. It's the most reliable tool type because the LLM can verify its own output by inspecting stdout.

Retrieval Tools: Search & RAG

Retrieval tools give agents access to external knowledge that isn't part of their training data. This includes vector search over document corpora, keyword search over knowledge bases, web search for real-time information, and structured lookups in databases or APIs.

Vector Search (Semantic Retrieval)

The most common retrieval pattern embeds the user query into a vector and searches a vector database for similar document chunks. The top-K results are injected into the LLM context as tool results.

Hybrid Search

Combining vector (semantic) search with keyword (BM25) search produces more robust results. Vector search captures meaning, while keyword search catches exact terms the embedding model might miss (e.g., acronyms, product names, error codes).

Vector Search

  • Captures semantic meaning
  • Handles paraphrases well
  • Requires embedding model
  • Can miss exact matches

BM25 / Keyword Search

  • Exact term matching
  • Fast, no GPU needed
  • Misses synonyms
  • Great for codes & IDs

Web Search Tools

Web search tools wrap APIs like Google Search, Bing, or Brave Search to give agents access to real-time information. The tool typically returns a list of snippets with URLs, and a follow-up read_webpage tool can fetch full page content for deeper analysis.

Multi-Step Retrieval

Advanced agents use iterative retrieval: they search, read results, refine the query, and search again. This is analogous to how a human researcher works—the first query rarely retrieves everything needed. The agent loop naturally supports this pattern when retrieval tools are available alongside reasoning.

RAG vs. fine-tuning: Retrieval tools are preferred over fine-tuning when knowledge changes frequently, needs to be auditable (with source citations), or exceeds what can be encoded in model weights. Fine-tuning is better for teaching the model a skill or style rather than injecting facts.

Error Handling & Validation

Tool calls fail. APIs return 500s, code throws exceptions, search returns no results. Robust tool-augmented agents must handle these failures gracefully and provide the LLM with enough context to either retry with different parameters or fall back to a different strategy.

The Error Envelope Pattern

Rather than letting exceptions crash the agent loop, wrap every tool execution in a standard error envelope that the LLM can interpret consistently. 

# Error envelope pattern for tool execution

import traceback, time, json
from typing import Any, Dict

class ToolResult:
    """Standardized result wrapper for all tool calls."""
    def __init__(self, success: bool, data: Any = None, error: str = None,
                 retryable: bool = False, suggestion: str = None):
        self.success = success
        self.data = data
        self.error = error
        self.retryable = retryable
        self.suggestion = suggestion

    def to_message(self) -> str:
        if self.success:
            return json.dumps(self.data, default=str)
        msg = f"ERROR: {self.error}"
        if self.retryable:
            msg += "\n[This error is retryable]"
        if self.suggestion:
            msg += f"\nSuggestion: {self.suggestion}"
        return msg


def safe_execute(tool_fn, args: Dict, max_retries: int = 2) -> ToolResult:
    """Execute a tool with retry logic and error wrapping."""
    for attempt in range(max_retries + 1):
        try:
            result = tool_fn(**args)
            return ToolResult(success=True, data=result)

        except ValueError as e:
            # Bad input — don't retry, suggest fix
            return ToolResult(
                success=False, error=str(e),
                retryable=False,
                suggestion="Check parameter types and constraints"
            )

        except TimeoutError:
            if attempt < max_retries:
                time.sleep(2 ** attempt)  # exponential backoff
                continue
            return ToolResult(
                success=False, error="Tool timed out after retries",
                retryable=True,
                suggestion="Try simpler parameters or a different tool"
            )

        except Exception as e:
            return ToolResult(
                success=False,
                error=f"{type(e).__name__}: {str(e)}",
                retryable=False
            )
Never expose raw stack traces to the LLM. Raw tracebacks leak implementation details (file paths, library versions, internal function names) and waste tokens. Instead, return a concise error message with a suggested next action.

Input Validation

Validate tool arguments before execution. Common checks include:

  • Type checking — ensure strings are strings, numbers are numbers
  • Range validation — dates in the past, page numbers > 0
  • Injection prevention — sanitize SQL, shell commands, file paths
  • Length limits — cap string lengths to prevent abuse
  • Allow-list matching — verify enum values are valid

Graceful Degradation

When a tool fails permanently, the agent should not get stuck in an infinite retry loop. Design the error message to help the LLM choose an alternative strategy: use a different tool, ask the user for clarification, or respond with a caveat about missing information.

Best Practices for Tool Design

After deploying tool-augmented agents in production, several patterns emerge as critical for reliability, performance, and maintainability.

1. Keep Tool Count Manageable

Models perform best with 5–20 tools. Beyond 20, tool selection accuracy drops significantly. If you need more tools, consider a two-stage approach: a router tool that selects a tool category, then a second call with only the relevant subset of tools.

2. Design for Composability

Tools should do one thing well. A search_docs tool that also summarizes results is harder for the LLM to use than separate search_docs and summarize_text tools that can be composed in sequence.

✅ Good: Atomic Tools

  • search_docs(query)
  • read_document(doc_id)
  • run_sql(query)
  • send_email(to, subject, body)

Each tool has a clear, single responsibility.

❌ Bad: Overloaded Tools

  • data_tool(action, ...params)
  • do_everything(type, config)
  • api(method, url, body, auth)
  • process(input, mode, format)

Ambiguous—models struggle to choose the right parameters.

3. Return Structured, Concise Results

Tool results consume context window tokens. Truncate large responses, return only relevant fields, and use pagination for large result sets. A search tool returning 50 full documents will overflow most context windows—return titles and snippets instead.

4. Implement Observability

Log every tool call with: the tool name, input arguments, execution time, result size, and success/failure status. This data is invaluable for debugging agent behavior, identifying slow tools, and optimizing the tool set over time.

5. Version Your Tool Schemas

As tools evolve, their schemas change. Use semantic versioning for tool definitions and maintain backward compatibility. A breaking change to a tool schema can silently degrade agent performance if the model was tuned on the old schema format.

6. Test Tools in Isolation

Before integrating a tool into an agent loop, test it with known inputs and verify outputs. Create a test harness that exercises edge cases: empty inputs, maximum-length strings, special characters, and concurrent calls. Tool bugs surface as mysterious agent failures that are hard to debug at the agent level.

The 80/20 rule of tool design: 80% of agent capability comes from getting the tool descriptions right. Invest time in writing clear, specific descriptions with examples of when to use (and not use) each tool. This is more impactful than adding more tools.
