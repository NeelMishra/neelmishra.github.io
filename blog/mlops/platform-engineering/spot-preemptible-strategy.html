Spot Instance Strategy for ML

MLOps Series Platform Engineering

Spot Instance Overview

Spot instances (AWS), preemptible VMs (GCP), and spot VMs (Azure) offer GPU compute at 60–90% discount compared to on-demand pricing. The trade-off is simple: the cloud provider can reclaim your instance with as little as 30 seconds to 2 minutes of warning. For ML training, this creates a unique engineering challenge — and a massive cost opportunity.

The key insight is that modern ML training is inherently checkpoint-resumable. Unlike a web server that must be continuously available, a training job can be paused at any checkpoint and resumed from exactly where it left off. This makes ML training one of the best workloads for spot instances.

Real-world savings: A 64-GPU training run that costs $700K on-demand can be completed for $100–180K using spot instances with proper checkpointing — even accounting for the 10–15% overhead from interruptions and restarts.

Checkpoint Strategy

Checkpointing is the foundation of any spot instance strategy. The goal is to periodically save model state so that a preempted job can resume without losing significant progress. The checkpoint frequency is a trade-off between overhead (I/O time) and wasted compute (progress lost since last checkpoint).

Async Checkpoint with Signal Handling

import signal, os, time, torch import torch.distributed as dist class SpotCheckpointManager: """Manages checkpointing for spot/preemptible training.""" def __init__(self, model, optimizer, save_dir, interval_steps=500): self.model = model self.optimizer = optimizer self.save_dir = save_dir self.interval = interval_steps self.step = 0 self.preempted = False # Register signal handlers for spot termination signal.signal(signal.SIGTERM, self._handle_preemption) signal.signal(signal.SIGUSR1, self._handle_preemption) def _handle_preemption(self, signum, frame): print(f"[PREEMPTION] Signal {signum} received. Saving emergency checkpoint...") self.preempted = True self.save_checkpoint(emergency=True) def save_checkpoint(self, emergency=False): """Save model, optimizer, and training state.""" tag = "emergency" if emergency else f"step_{self.step}" path = os.path.join(self.save_dir, f"ckpt_{tag}.pt") state = { "step": self.step, "model_state": self.model.state_dict(), "optimizer_state": self.optimizer.state_dict(), "timestamp": time.time(), } torch.save(state, path) print(f"[CKPT] Saved to {path} at step {self.step}") def maybe_checkpoint(self): """Called after each training step.""" self.step += 1 if self.step % self.interval == 0: self.save_checkpoint() def load_latest(self): """Resume from the most recent checkpoint.""" ckpts = sorted( [f for f in os.listdir(self.save_dir) if f.endswith(".pt")], key=lambda f: os.path.getmtime(os.path.join(self.save_dir, f)) ) if not ckpts: print("[CKPT] No checkpoint found. Starting from scratch.") return 0 path = os.path.join(self.save_dir, ckpts[-1]) state = torch.load(path) self.model.load_state_dict(state["model_state"]) self.optimizer.load_state_dict(state["optimizer_state"]) self.step = state["step"] print(f"[CKPT] Resumed from {path} at step {self.step}") return self.step
Checkpoint to fast storage: Use NVMe local SSDs for intermediate checkpoints and asynchronously upload to cloud object storage (S3, GCS). A 10B parameter model checkpoint is ~40 GB — on NVMe this takes seconds, but minutes over network.

Fallback Patterns

A robust spot strategy doesn't just handle interruptions — it proactively plans for them with fallback tiers. When spot capacity is unavailable in your preferred zone, your system should automatically cascade through alternatives.

Fallback Tier Architecture

# Fallback priority chain for GPU workloads FALLBACK_TIERS = [ {"type": "spot", "zones": ["us-east-1a", "us-east-1b", "us-east-1c"]}, {"type": "spot", "zones": ["us-west-2a", "us-west-2b"]}, {"type": "spot", "instance": "p4de.24xlarge"}, # Alt GPU type {"type": "on-demand", "zones": ["us-east-1a"]}, # Last resort ] def acquire_capacity(num_gpus, tiers=FALLBACK_TIERS): """Attempt to acquire GPU capacity, cascading through fallback tiers.""" for tier in tiers: for zone in tier.get("zones", ["default"]): try: instance = launch_instance( instance_type=tier.get("instance", "p4d.24xlarge"), market_type=tier["type"], zone=zone, count=num_gpus // 8 ) print(f"Acquired {num_gpus} GPUs via {tier['type']} in {zone}") return instance except InsufficientCapacityError: print(f"No {tier['type']} capacity in {zone}. Trying next...") continue raise RuntimeError("All fallback tiers exhausted.")
Avoid cross-region fallbacks for distributed training. If your training data is in us-east-1, falling back to eu-west-1 will incur massive data transfer costs and latency. Prefer falling back to on-demand in the same region over spot in a distant region.

Graceful Degradation

For elastic training frameworks (like PyTorch Elastic / torchrun), you can reduce the number of workers when spot capacity shrinks, rather than stopping entirely. This trades training speed for continuity.

# torchrun with elastic scaling — min 4 GPUs, max 64 torchrun \ --nnodes=1:8 \ --nproc_per_node=8 \ --max_restarts=10 \ --rdzv_backend=c10d \ --rdzv_endpoint=master:29400 \ train.py --checkpoint-dir /mnt/checkpoints

Multi-Zone Resilience

Spot availability varies significantly by availability zone. Spreading your spot requests across multiple zones dramatically increases the probability of maintaining capacity. The strategy is straightforward: request capacity in all viable zones simultaneously and use the first allocations that succeed.

Cost Analysis

The actual savings from spot instances depend on several factors: interruption frequency, checkpoint overhead, restart time, and the spot discount itself. Let's build a realistic cost model.

Spot Cost Model

def spot_cost_model( total_training_hours=720, # 30 days of training num_gpus=64, on_demand_rate=32.77, spot_rate=9.83, avg_interruptions_per_day=2.5, checkpoint_overhead_pct=3.0, # % of time spent checkpointing restart_overhead_min=8, # minutes to restart after preemption ): """Model total cost including spot interruption overhead.""" days = total_training_hours / 24 total_interruptions = avg_interruptions_per_day * days # Lost compute per interruption (avg half the checkpoint interval) checkpoint_interval_min = 15 avg_lost_per_interrupt = checkpoint_interval_min / 2 + restart_overhead_min total_lost_hours = total_interruptions * avg_lost_per_interrupt / 60 # Effective training hours needed ckpt_overhead = total_training_hours * checkpoint_overhead_pct / 100 effective_hours = total_training_hours + total_lost_hours + ckpt_overhead spot_cost = effective_hours * num_gpus * spot_rate on_demand_cost = total_training_hours * num_gpus * on_demand_rate savings = (1 - spot_cost / on_demand_cost) * 100 print(f"=== Spot Instance Cost Analysis ===") print(f"Base training hours: {total_training_hours}") print(f"Total interruptions: {total_interruptions:.0f}") print(f"Overhead hours: {total_lost_hours + ckpt_overhead:.1f}") print(f"Effective hours needed: {effective_hours:.1f}") print(f"Spot total cost: ${spot_cost:,.0f}") print(f"On-demand baseline: ${on_demand_cost:,.0f}") print(f"Net savings: {savings:.1f}%") spot_cost_model()
Typical result: Even with 2.5 interruptions per day and the associated overhead, spot instances deliver 55–65% cost savings compared to on-demand for long training runs. The longer the run, the better the amortization.
