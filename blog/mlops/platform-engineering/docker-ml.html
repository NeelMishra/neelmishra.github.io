Docker Best Practices for ML

MLOps Series Platform Engineering

Why Docker Matters for ML

ML containers are fundamentally different from web-app containers. They carry heavy runtimes (CUDA, cuDNN), large Python dependency trees, and often ship model weights inside the image itself. A naive Dockerfile can easily produce a 15 GB+ image that takes 20 minutes to pull.

This post covers battle-tested patterns for building lean, fast, and secure ML containers that play well with Kubernetes, CI/CD pipelines, and container registries.

Goal: Reduce image size by 60–80%, cut build times in half with layer caching, and eliminate common security anti-patterns in ML Dockerfiles.

Choosing the Right Base Image

The base image determines your CUDA version, OS libraries, and starting image size. Choosing wrong can mean hours of debugging driver mismatches.

nvidia/cuda (Official)

  • Sizes: base ~150 MB, runtime ~1.5 GB, devel ~3.5 GB
  • Use runtime for inference, devel only when compiling custom CUDA kernels
  • Pinned CUDA + cuDNN versions

pytorch/pytorch (NGC)

  • Pre-built with PyTorch, CUDA, NCCL
  • Starting size ~6 GB (includes everything)
  • Best for quick prototyping; overkill for slim inference images

Version Pinning Is Non-Negotiable

# GOOD: Pin every version explicitly FROM nvidia/cuda:12.1.1-cudnn8-runtime-ubuntu22.04 # BAD: Floating tags break reproducibility # FROM nvidia/cuda:latest # FROM pytorch/pytorch:latest
Critical: The CUDA version in your base image must match the driver version on your K8s nodes. CUDA 12.1 requires driver ≥530. Mismatches produce cryptic CUDA_ERROR_NO_DEVICE failures at runtime.

Multi-Stage Builds

Multi-stage builds are the single most impactful technique for reducing ML image size. The idea: use a heavy devel image to compile dependencies, then copy only the compiled artifacts to a slim runtime image.

Pattern: Build Stage + Runtime Stage

# ── Stage 1: Build ── FROM nvidia/cuda:12.1.1-cudnn8-devel-ubuntu22.04 AS builder RUN apt-get update && apt-get install -y --no-install-recommends \ python3.10 python3.10-venv python3-pip build-essential \ && rm -rf /var/lib/apt/lists/* # Create a virtual environment for clean separation RUN python3.10 -m venv /opt/venv ENV PATH="/opt/venv/bin:$PATH" # Install Python deps (heavy compile step) COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # ── Stage 2: Runtime ── FROM nvidia/cuda:12.1.1-cudnn8-runtime-ubuntu22.04 RUN apt-get update && apt-get install -y --no-install-recommends \ python3.10 libgomp1 \ && rm -rf /var/lib/apt/lists/* # Copy the pre-built virtual environment COPY --from=builder /opt/venv /opt/venv ENV PATH="/opt/venv/bin:$PATH" COPY src/ /app/src/ WORKDIR /app ENTRYPOINT ["python3", "src/train.py"]

This pattern typically reduces the final image from ~8 GB (devel-based) to ~3 GB (runtime-based) while keeping all compiled wheels intact.

Result: Build tools, header files, and compilation caches are left behind in the builder stage—they never appear in the final image.

Layer Caching Strategy

Docker caches each layer. If a layer hasn't changed, Docker reuses it. The key insight: order Dockerfile instructions from least to most frequently changing.

Optimal Layer Order

# Layer 1: Base image (changes rarely) FROM nvidia/cuda:12.1.1-cudnn8-runtime-ubuntu22.04 # Layer 2: System packages (changes rarely) RUN apt-get update && apt-get install -y python3.10 \ && rm -rf /var/lib/apt/lists/* # Layer 3: Python dependencies (changes weekly) COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # Layer 4: Application code (changes every commit) COPY src/ /app/src/
Anti-pattern: Copying all source code before pip install busts the pip cache on every code change, causing a full re-install of all Python packages. Always copy requirements.txt first.

Using BuildKit Cache Mounts

Docker BuildKit supports cache mounts that persist the pip cache across builds, even when requirements.txt changes.

# syntax=docker/dockerfile:1 RUN --mount=type=cache,target=/root/.cache/pip \ pip install -r requirements.txt

Size Optimization Techniques

Every megabyte counts when you're pulling images across regions or running on spot instances with limited disk.

1. Use .dockerignore

# .dockerignore __pycache__/ *.pyc .git/ data/ checkpoints/ wandb/ *.egg-info/ notebooks/ .venv/

2. Strip Unnecessary CUDA Libraries

# Remove static libraries and debug symbols RUN find /usr/local/cuda -name "*.a" -delete && \ find /usr/local/cuda -name "*.static*" -delete && \ rm -rf /usr/local/cuda/extras /usr/local/cuda/nsight*

3. Combine RUN Instructions

# BAD: Each RUN creates a new layer RUN apt-get update RUN apt-get install -y python3 RUN rm -rf /var/lib/apt/lists/* # GOOD: Single layer, apt cache cleaned in same layer RUN apt-get update && \ apt-get install -y --no-install-recommends python3 && \ rm -rf /var/lib/apt/lists/*

Image Size Comparison

Before Optimization

  • Base: devel — 3.5 GB
  • Python deps — 4.2 GB
  • Source code — 50 MB
  • Total: ~7.8 GB

After Optimization

  • Base: runtime — 1.5 GB
  • Python deps (venv only) — 1.8 GB
  • Source code — 50 MB
  • Total: ~3.4 GB (56% smaller)

Security Hardening

ML containers often run with elevated privileges for GPU access. This makes security hardening even more important.

1. Run as Non-Root

# Create a dedicated user RUN groupadd -r mluser && useradd -r -g mluser mluser USER mluser WORKDIR /home/mluser/app

2. Pin Package Versions

# requirements.txt — pin everything torch==2.1.2+cu121 transformers==4.36.2 numpy==1.26.3 safetensors==0.4.1

3. Scan for Vulnerabilities

# Scan with Trivy in CI trivy image --severity HIGH,CRITICAL \ --exit-code 1 \ myregistry/bert-trainer:v2.1
Tip: Add Trivy scanning to your CI pipeline as a gate. Block image pushes that contain HIGH or CRITICAL CVEs in OS packages or Python dependencies.

4. Use Read-Only Filesystems

In K8s, mount the container filesystem as read-only and provide writable emptyDir volumes only where needed (checkpoints, logs).

spec: containers: - name: trainer securityContext: readOnlyRootFilesystem: true volumeMounts: - name: tmp mountPath: /tmp - name: checkpoints mountPath: /checkpoints volumes: - name: tmp emptyDir: {} - name: checkpoints emptyDir: sizeLimit: "20Gi"
Warning: Some ML frameworks write temporary files to /root or /home. Test with readOnlyRootFilesystem: true in development before enforcing in production to catch write failures early.
