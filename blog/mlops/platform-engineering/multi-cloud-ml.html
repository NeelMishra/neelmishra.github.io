Multi-Cloud ML Architecture

MLOps Series Platform Engineering

Multi-Cloud Overview

Multi-cloud ML is not about running the same workload on every cloud simultaneously — it's about strategically placing workloads where they run best while maintaining the ability to move between providers. The drivers for multi-cloud in ML are practical: GPU availability constraints, pricing arbitrage, regulatory requirements, and avoiding vendor lock-in on rapidly evolving AI services.

During the current GPU shortage, many organizations find that no single cloud provider can supply enough H100 or A100 capacity for their training needs. Multi-cloud becomes a necessity, not a choice. The challenge is building an ML platform that abstracts away provider differences without sacrificing performance.

Multi-cloud ≠ cloud-agnostic. The goal is not to write code that runs identically everywhere. It's to build abstractions that make it practical to move workloads between clouds with minimal rework — typically measured in hours, not weeks.

Common Multi-Cloud ML Patterns

Burst to Secondary Cloud

Primary workloads on Cloud A. When GPU capacity is exhausted, burst overflow training jobs to Cloud B. Requires portable training code and shared checkpoint storage.

Best-of-Breed Selection

Use each cloud's strengths: GCP for TPU training, AWS for GPU inference at scale, Azure for enterprise integration. Requires strong data synchronization.

Data Gravity

Data gravity is the most powerful force in multi-cloud architecture. Large datasets create a gravitational pull that makes it expensive and slow to move compute away from the data. For ML, this is amplified: training datasets can be tens of terabytes to petabytes, and egress costs between clouds are $0.08–0.12 per GB.

The Cost of Moving Data

def calculate_egress_cost(dataset_size_tb, egress_rate_per_gb=0.09): """Calculate the cost of moving training data between clouds.""" dataset_gb = dataset_size_tb * 1024 egress_cost = dataset_gb * egress_rate_per_gb # Transfer time estimation (assuming 10 Gbps dedicated link) transfer_hours = (dataset_gb * 8) / (10 * 3600) # Gb / Gbps = seconds print(f"Dataset size: {dataset_size_tb} TB") print(f"Egress cost: ${egress_cost:,.0f}") print(f"Transfer time: {transfer_hours:.1f} hours") print(f"Cost per transfer: ${egress_cost / dataset_size_tb:,.0f}/TB") # Typical LLM training dataset calculate_egress_cost(50) # 50 TB dataset # Output: Egress cost: $4,608 per transfer, ~11 hours at 10 Gbps
Data gravity rule: If your training data exceeds 10 TB, move the compute to the data — not the other way around. The egress cost and transfer time will dominate any savings from cheaper GPU pricing elsewhere.

Data Synchronization Strategies

Replicate Data (Hot/Cold)

  • Keep a full copy on each cloud
  • Sync incrementally on schedule
  • High storage cost, fast access
  • Best for: datasets < 10 TB

Compute Follows Data

  • Data stays where it's generated
  • Submit jobs to the nearest cluster
  • Low egress cost, more complex scheduling
  • Best for: datasets > 10 TB

Portability Patterns

True workload portability requires abstracting three layers: compute provisioning, storage access, and ML framework configuration. Each layer has cloud-specific details that must be hidden behind a uniform interface.

Cloud-Native vs Portable Approach

Cloud-Native (Locked In)

  • AWS SageMaker Training Jobs
  • GCP Vertex AI Custom Training
  • Azure ML Compute Clusters
  • Fastest setup, hardest to migrate

Portable (Abstracted)

  • Kubernetes + GPU Operator
  • KubeFlow / Ray on any cloud
  • Container images + Helm charts
  • Higher initial effort, easy to migrate

Portable Training Configuration

# Cloud-agnostic training config using environment abstraction import os class CloudConfig: """Abstracts cloud-specific storage and compute details.""" PROVIDERS = { "aws": { "storage_prefix": "s3://", "checkpoint_path": "s3://ml-checkpoints/runs", "data_path": "s3://ml-data/training", "metrics_endpoint": "cloudwatch", }, "gcp": { "storage_prefix": "gs://", "checkpoint_path": "gs://ml-checkpoints/runs", "data_path": "gs://ml-data/training", "metrics_endpoint": "cloud-monitoring", }, "azure": { "storage_prefix": "az://", "checkpoint_path": "az://mlcheckpoints/runs", "data_path": "az://mldata/training", "metrics_endpoint": "azure-monitor", }, } def __init__(self): self.provider = os.environ.get("CLOUD_PROVIDER", "aws") self.config = self.PROVIDERS[self.provider] def checkpoint_path(self, run_id): return f"{self.config['checkpoint_path']}/{run_id}" def data_path(self, dataset): return f"{self.config['data_path']}/{dataset}" # Usage — same code on any cloud cloud = CloudConfig() ckpt = cloud.checkpoint_path("run-2024-001") data = cloud.data_path("openwebtext")
Portability layer principle: Abstractions should be thin. Don't build a general-purpose cloud SDK — just abstract the 3–5 things that differ between clouds (storage paths, credential injection, GPU device naming).

Kubernetes as the Abstraction Layer

Kubernetes is the de facto standard for multi-cloud ML orchestration. Every major cloud provider offers managed Kubernetes (EKS, GKE, AKS), and the Kubernetes API provides a uniform interface for scheduling GPU workloads regardless of the underlying infrastructure.

GPU Operator Setup

# NVIDIA GPU Operator — works identically on AWS, GCP, Azure, on-prem apiVersion: v1 kind: Namespace metadata: name: gpu-operator --- # Helm install (same command on any K8s cluster) helm install gpu-operator nvidia/gpu-operator \ --namespace gpu-operator \ --set driver.enabled=true \ --set toolkit.enabled=true \ --set devicePlugin.enabled=true \ --set mig.strategy=single

K8s is not magic. While the API is the same, the underlying GPU topology, network performance, and storage drivers differ significantly between clouds. Always benchmark on each target cloud before committing to multi-cloud training.

Cluster Federation

Cluster federation extends the multi-cloud story by connecting multiple Kubernetes clusters into a unified control plane. This enables a single job submission that can be scheduled across clusters on different clouds based on available GPU capacity, cost, and data locality.

Intelligent Workload Placement

import datetime class MultiCloudScheduler: """Routes ML jobs to the optimal cluster based on cost and capacity.""" def __init__(self, clusters): self.clusters = clusters # List of cluster configs def score_cluster(self, cluster, job): """Score a cluster for a given job (higher = better).""" score = 0 # Prefer clusters where the data lives (data gravity) if job.data_region == cluster.region: score += 100 # Factor in GPU availability available = cluster.get_available_gpus(job.gpu_type) if available >= job.num_gpus: score += 50 elif available > 0: score += 20 # Partial capacity # Factor in cost (normalized per GPU-hour) cost_score = (1 - cluster.cost_per_gpu_hr / 40.0) * 30 score += cost_score # Spot availability bonus if cluster.spot_available and job.spot_tolerant: score += 25 return score def schedule(self, job): """Select the best cluster for the job.""" scored = [ (self.score_cluster(c, job), c) for c in self.clusters ] scored.sort(reverse=True, key=lambda x: x[0]) best_score, best_cluster = scored[0] print(f"Selected: {best_cluster.name} (score: {best_score})") return best_cluster

Decision Framework

Not every organization needs multi-cloud ML. The added complexity of managing multiple cloud environments, synchronizing data, and maintaining portability has real costs. Use this framework to decide if multi-cloud is right for your situation.

When to Go Multi-Cloud

Strong Signals FOR Multi-Cloud

  • GPU capacity constraints on a single provider
  • Regulatory requirements (data residency)
  • Significant pricing differences between clouds
  • Existing infrastructure on multiple clouds
  • Strategic need to avoid vendor lock-in

Strong Signals AGAINST Multi-Cloud

  • Small team (< 5 ML engineers)
  • Single-region data < 10 TB
  • Adequate GPU supply on one cloud
  • Heavy use of managed ML services
  • Limited platform engineering capacity

Maturity Assessment

def multi_cloud_readiness_score( has_k8s_expertise, # Team runs Kubernetes in production containerized_training, # Training code runs in containers has_iac, # Infrastructure as Code (Terraform/Pulumi) data_size_tb, # Training data size in TB gpu_shortage, # Experiencing GPU capacity issues team_size, # ML platform engineers ): """Assess readiness for multi-cloud ML. Score 0-100.""" score = 0 score += 20 if has_k8s_expertise else 0 score += 15 if containerized_training else 0 score += 15 if has_iac else 0 score += 20 if gpu_shortage else 0 score += 15 if team_size >= 5 else (5 if team_size >= 3 else 0) score += 15 if data_size_tb < 10 else 5 # Smaller data is easier to sync print(f"Multi-Cloud Readiness: {score}/100") if score >= 70: print("✓ Strong candidate for multi-cloud ML.") elif score >= 40: print("◐ Consider multi-cloud for specific workloads only.") else: print("✗ Focus on single-cloud optimization first.") multi_cloud_readiness_score( has_k8s_expertise=True, containerized_training=True, has_iac=True, data_size_tb=5, gpu_shortage=True, team_size=6 )
Start with portability, not multi-cloud. Even if you run on a single cloud today, containerizing your training code and using Kubernetes gives you the option to go multi-cloud later. The portability investment pays for itself even in a single-cloud world through reproducibility and operational consistency.

Multi-cloud ML is a spectrum, not a binary choice. Start by containerizing your training code, use cloud-agnostic storage interfaces, and invest in Kubernetes. When the business need arises — whether from GPU shortages, cost pressure, or regulatory requirements — you'll be ready to expand.
