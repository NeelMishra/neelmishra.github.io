Azure ML Platform Overview

Azure Machine Learning is Microsoft's enterprise ML platform. It provides a workspace-centric model where all assets—data, compute, models, endpoints, and pipelines—are organized under a single workspace tied to an Azure resource group. The v2 SDK (azure-ai-ml) offers a modern, Pythonic interface for all operations.

V2 SDK: The azure-ai-ml package replaces the older azureml-core. All new projects should use the v2 SDK, which provides a cleaner API, better YAML support, and alignment with the Azure CLI ml extension.
# Connect to Azure ML workspace with v2 SDK from azure.ai.ml import MLClient from azure.identity import DefaultAzureCredential credential = DefaultAzureCredential() ml_client = MLClient( credential=credential, subscription_id="your-subscription-id", resource_group_name="ml-resources-rg", workspace_name="ml-workspace-prod", ) print(f"Workspace: {ml_client.workspace_name}") print(f"Resource group: {ml_client.resource_group_name}")

Azure ML integrates with the broader Azure ecosystem: Azure DevOps for CI/CD, Azure Key Vault for secrets, Azure Container Registry for Docker images, and Azure Blob Storage for data. This makes it a natural choice for organizations already invested in the Microsoft stack.

Workspace Setup

The workspace is the top-level resource in Azure ML. It encapsulates compute, data, jobs, models, and endpoints. Creating a workspace also provisions dependent resources: a storage account, key vault, container registry, and Application Insights instance.

# Create a workspace programmatically from azure.ai.ml.entities import Workspace workspace = Workspace( name="ml-workspace-prod", location="eastus", display_name="Production ML Workspace", description="Central workspace for ML team", tags={ "team": "ml-platform", "environment": "production", }, ) ml_client.workspaces.begin_create(workspace).result()
Workspace sprawl: Resist the temptation to create one workspace per project. A single workspace per team (or per environment) is usually sufficient. Too many workspaces fragment model governance, increase costs, and complicate access management.

For infrastructure-as-code workflows, workspaces can also be defined in YAML and created via the Azure CLI:

# workspace.yml - Declarative workspace definition $schema: https://azuremlschemas.azureedge.net/latest/workspace.schema.json name: ml-workspace-prod location: eastus display_name: Production ML Workspace description: Central workspace for ML team tags: team: ml-platform environment: production # CLI command: az ml workspace create --file workspace.yml

Compute Targets

Azure ML supports multiple compute types: compute instances (dev machines), compute clusters (auto-scaling training), and attached computes (existing AKS clusters, VMs). Compute clusters are the workhorse for training—they scale from zero to the required node count and back.

# Create a GPU compute cluster for training from azure.ai.ml.entities import AmlCompute gpu_cluster = AmlCompute( name="gpu-training-cluster", type="amlcompute", size="Standard_NC24ads_A100_v4", # A100 GPU min_instances=0, # Scale to zero when idle max_instances=4, # Max 4 nodes idle_time_before_scale_down=300, # 5 min idle timeout tier="dedicated", tags={"purpose": "training"}, ) ml_client.compute.begin_create_or_update(gpu_cluster).result()

Compute Instances

Use case: Development, notebooks, debugging

Scaling: Single VM, always on

Cost tip: Enable auto-shutdown schedules

Compute Clusters

Use case: Training jobs, hyperparameter tuning

Scaling: 0 to N nodes, auto-scales

Cost tip: Set min_instances=0 for scale-to-zero

For development work, create a compute instance with auto-shutdown to control costs:

# Compute instance with auto-shutdown schedule from azure.ai.ml.entities import ComputeInstance, ComputeSchedules from azure.ai.ml.entities import CronTrigger, ComputeStartStopSchedule dev_instance = ComputeInstance( name="dev-gpu-instance", size="Standard_NC6s_v3", # V100 GPU schedules=ComputeSchedules( compute_start_stop=[ ComputeStartStopSchedule( trigger=CronTrigger( expression="0 18 * * *", # 6 PM daily time_zone="Eastern Standard Time", ), action="Stop", ) ] ), ) ml_client.compute.begin_create_or_update(dev_instance).result()
Low-priority VMs: Azure ML compute clusters support low-priority VMs (similar to AWS spot instances). Set tier="low_priority" for up to 80% cost savings on fault-tolerant training workloads.

Environments

Environments define the software runtime for training and inference. Azure ML provides curated environments (pre-built images with popular frameworks) and supports custom environments defined via Docker or Conda specifications.

# Use a curated environment from azure.ai.ml.entities import Environment # List curated environments for env in ml_client.environments.list(): if "pytorch" in env.name.lower(): print(f"{env.name}:{env.version}") # Use the curated PyTorch environment curated_env = ml_client.environments.get( name="AzureML-pytorch-2.1-cuda12", version="latest", )

For custom environments, define a Dockerfile or Conda spec:

# Custom environment from Conda specification from azure.ai.ml.entities import Environment, BuildContext custom_env = Environment( name="custom-pytorch-training", description="Custom PyTorch env with project dependencies", conda_file="environments/conda.yml", image="mcr.microsoft.com/azureml/curated/pytorch-2.1-cuda12:latest", version="1.0", ) ml_client.environments.create_or_update(custom_env)
# environments/conda.yml name: training-env channels: - pytorch - conda-forge dependencies: - python=3.10 - pytorch=2.1.0 - torchvision=0.16.0 - cudatoolkit=12.1 - pip: - mlflow==2.9.0 - azureml-mlflow==1.55.0 - transformers>=4.35.0
Environment build time: Custom environments trigger Docker image builds that can take 10–15 minutes. Pre-build environments during CI and reference the built image to avoid delays during training job submission.

Azure ML Pipelines

Azure ML pipelines orchestrate multi-step ML workflows. Each step is a command job or component that runs in its own environment and compute target. Pipelines support data flow between steps, conditional logic, and parallel execution.

# Define a training pipeline with components from azure.ai.ml import command, Input, Output from azure.ai.ml.dsl import pipeline # Step 1: Data preprocessing component preprocess_component = command( name="preprocess_data", display_name="Preprocess Training Data", inputs={ "raw_data": Input(type="uri_folder"), "split_ratio": Input(type="number", default=0.8), }, outputs={ "train_data": Output(type="uri_folder"), "test_data": Output(type="uri_folder"), }, code="./src/preprocess/", command="python preprocess.py --input ${{inputs.raw_data}} --train ${{outputs.train_data}} --test ${{outputs.test_data}} --ratio ${{inputs.split_ratio}}", environment="AzureML-sklearn-1.2@latest", ) # Step 2: Training component train_component = command( name="train_model", display_name="Train PyTorch Model", inputs={ "train_data": Input(type="uri_folder"), "epochs": Input(type="integer", default=50), "learning_rate": Input(type="number", default=0.001), }, outputs={ "model_output": Output(type="uri_folder"), }, code="./src/train/", command="python train.py --data ${{inputs.train_data}} --output ${{outputs.model_output}} --epochs ${{inputs.epochs}} --lr ${{inputs.learning_rate}}", environment=custom_env, compute="gpu-training-cluster", )

Compose components into a pipeline using the @pipeline decorator:

# Assemble the pipeline @pipeline( display_name="image-classification-pipeline", description="End-to-end image classification training", ) def training_pipeline(raw_data, epochs, learning_rate): preprocess_step = preprocess_component(raw_data=raw_data) train_step = train_component( train_data=preprocess_step.outputs.train_data, epochs=epochs, learning_rate=learning_rate, ) return {"model": train_step.outputs.model_output} # Submit pipeline run pipeline_job = training_pipeline( raw_data=Input(type="uri_folder", path="azureml://datastores/training/paths/images/"), epochs=100, learning_rate=0.0005, ) submitted_job = ml_client.jobs.create_or_update(pipeline_job) print(f"Pipeline URL: {submitted_job.studio_url}")
MLflow integration: Azure ML natively integrates with MLflow. Use mlflow.autolog() in your training script to automatically log metrics, parameters, and model artifacts without modifying your training code.

Managed Endpoints

Azure ML managed online endpoints handle model serving with built-in autoscaling, blue/green deployments, and monitoring. Each endpoint can host multiple deployments with configurable traffic routing for canary releases and A/B testing.

# Create a managed online endpoint from azure.ai.ml.entities import ( ManagedOnlineEndpoint, ManagedOnlineDeployment, Model, CodeConfiguration, ) # Define the endpoint endpoint = ManagedOnlineEndpoint( name="image-classifier-endpoint", description="Production image classification endpoint", auth_mode="key", tags={"team": "ml-platform", "model": "resnet50"}, ) ml_client.online_endpoints.begin_create_or_update(endpoint).result() # Deploy a model to the endpoint model = Model(path="./model/", type="custom_model") blue_deployment = ManagedOnlineDeployment( name="blue", endpoint_name="image-classifier-endpoint", model=model, code_configuration=CodeConfiguration( code="./src/score/", scoring_script="score.py", ), environment=custom_env, instance_type="Standard_NC6s_v3", instance_count=2, ) ml_client.online_deployments.begin_create_or_update(blue_deployment).result()

For blue/green deployments, create a second deployment and shift traffic gradually:

# Blue/green deployment with traffic splitting green_deployment = ManagedOnlineDeployment( name="green", endpoint_name="image-classifier-endpoint", model=new_model, code_configuration=CodeConfiguration( code="./src/score/", scoring_script="score_v2.py", ), environment=custom_env, instance_type="Standard_NC6s_v3", instance_count=2, ) ml_client.online_deployments.begin_create_or_update(green_deployment).result() # Shift 20% traffic to green for canary testing endpoint.traffic = {"blue": 80, "green": 20} ml_client.online_endpoints.begin_create_or_update(endpoint).result() # After validation, shift 100% to green endpoint.traffic = {"blue": 0, "green": 100} ml_client.online_endpoints.begin_create_or_update(endpoint).result() # Clean up old deployment ml_client.online_deployments.begin_delete( name="blue", endpoint_name="image-classifier-endpoint" ).result()

Managed Online Endpoints

Management: Fully managed by Azure

Scaling: Built-in autoscaling rules

Best for: Teams wanting minimal ops overhead

Kubernetes Endpoints

Management: Self-managed AKS cluster

Scaling: Kubernetes HPA/VPA

Best for: Teams needing full infrastructure control

Endpoint costs: Managed endpoints bill for compute even when idle. Use autoscaling rules with minimum instance counts and configure scale-to-zero for development endpoints to avoid unnecessary costs.

To invoke the endpoint programmatically:

# Invoke the endpoint import json response = ml_client.online_endpoints.invoke( endpoint_name="image-classifier-endpoint", request_file="./sample-request.json", ) result = json.loads(response) print(f"Prediction: {result['prediction']}") print(f"Confidence: {result['confidence']:.3f}")
