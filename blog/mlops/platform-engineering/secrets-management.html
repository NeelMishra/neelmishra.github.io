Why Secrets Management Matters for ML

ML systems interact with a sprawling constellation of services: object stores, feature stores, model registries, databases, third-party inference APIs, and monitoring backends. Each connection requires credentials, and a single leaked key can expose training data, model weights, or customer-facing predictions. Yet many ML teams still hard-code secrets in notebooks or commit .env files to Git.

Warning: Hard-coded secrets in Jupyter notebooks are one of the most common attack vectors in ML pipelines. GitHub scans detect thousands of leaked API keys every week.

A mature secrets strategy accomplishes three goals: zero secrets in code, least-privilege access, and automated rotation. The following sections walk through the tools and patterns that achieve all three in production ML environments.

HashiCorp Vault Integration

HashiCorp Vault is the de-facto standard for centralized secrets management. Its dynamic secrets engine generates short-lived credentials on the fly, drastically reducing the blast radius of a compromise. For ML workloads, the most useful backends are the KV v2 store (for static API keys), the AWS secrets engine (for dynamic IAM credentials), and the database engine (for rotating DB passwords).

# Python — read a secret from Vault using hvac import hvac def get_vault_secret(path: str, key: str) -> str: client = hvac.Client( url=os.environ["VAULT_ADDR"], token=os.environ["VAULT_TOKEN"], ) response = client.secrets.kv.v2.read_secret_version( path=path, mount_point="ml-secrets" ) return response["data"]["data"][key] # Fetch the S3 credentials used by the feature store aws_key = get_vault_secret("feature-store/s3", "access_key") aws_secret = get_vault_secret("feature-store/s3", "secret_key")

In Kubernetes environments, the Vault Agent Injector sidecars transparently inject secrets into pod volumes, so application code simply reads from a file path. This eliminates the need for your training or serving containers to authenticate directly.

# Vault Agent annotation for a K8s training pod annotations: vault.hashicorp.com/agent-inject: "true" vault.hashicorp.com/role: "ml-training" vault.hashicorp.com/agent-inject-secret-s3: "ml-secrets/data/feature-store/s3" vault.hashicorp.com/agent-inject-template-s3: | {{- with secret "ml-secrets/data/feature-store/s3" -}} export AWS_ACCESS_KEY_ID={{ .Data.data.access_key }} export AWS_SECRET_ACCESS_KEY={{ .Data.data.secret_key }} {{- end -}}
Tip: Use Vault namespaces to isolate ML team secrets from platform-wide infrastructure secrets. This limits the damage from a misconfigured policy.

Cloud-Native Secrets Managers

If your ML infrastructure lives entirely within a single cloud, managed services like AWS Secrets Manager, Google Secret Manager, or Azure Key Vault offer tight IAM integration without the operational overhead of self-hosting Vault.

AWS Secrets Manager

  • Native RDS/Redshift rotation
  • Cross-account sharing via resource policies
  • $0.40 per secret per month
  • Versioned secrets with staging labels

HashiCorp Vault

  • Multi-cloud & on-prem support
  • Dynamic secrets for any backend
  • Open-source core, paid enterprise
  • Fine-grained ACL policies
# Python — fetch a secret from AWS Secrets Manager import boto3, json def get_aws_secret(secret_name: str, region: str = "us-east-1") -> dict: client = boto3.client("secretsmanager", region_name=region) response = client.get_secret_value(SecretId=secret_name) return json.loads(response["SecretString"]) # Retrieve model-serving API credentials creds = get_aws_secret("prod/ml-serving/api-keys") api_key = creds["primary_key"]

For cross-cloud setups, consider using External Secrets Operator (ESO) in Kubernetes. ESO syncs secrets from any provider into native K8s Secret objects, giving you a unified interface regardless of where the secret lives.

Kubernetes Secrets & Sealed Secrets

Native Kubernetes Secrets are base64-encoded, not encrypted at rest by default. For ML workloads running on K8s, you have two hardening options: enable etcd encryption at the cluster level, or adopt Sealed Secrets (or SOPS) to encrypt secrets before they reach the cluster.

Security note: A base64-encoded K8s Secret is trivially decodable. Anyone with kubectl get secret access can read the plaintext. Always layer additional encryption or use an external secrets store.
# Kubernetes Secret manifest (use only with encryption at rest) apiVersion: v1 kind: Secret metadata: name: ml-model-registry-creds namespace: ml-training type: Opaque stringData: REGISTRY_URL: "https://registry.internal.example.com" REGISTRY_TOKEN: "vault:v1:encrypted-token-here"
# SealedSecret — safe to commit to Git apiVersion: bitnami.com/v1alpha1 kind: SealedSecret metadata: name: ml-model-registry-creds namespace: ml-training spec: encryptedData: REGISTRY_TOKEN: AgBf2...long-encrypted-blob...

When using External Secrets Operator, define an ExternalSecret resource that references the remote store. ESO polls for changes and updates the K8s Secret automatically, giving you GitOps-friendly secret management without storing plaintext anywhere in your repository.

# ExternalSecret syncing from AWS Secrets Manager apiVersion: external-secrets.io/v1beta1 kind: ExternalSecret metadata: name: ml-api-keys spec: refreshInterval: 1h secretStoreRef: name: aws-secrets-store kind: ClusterSecretStore target: name: ml-api-keys data: - secretKey: PRIMARY_KEY remoteRef: key: prod/ml-serving/api-keys property: primary_key

API Key Rotation Strategies

Static API keys are the weakest link in most ML deployments. Rotation must be automated, zero-downtime, and auditable. The dual-key pattern is the most reliable approach: maintain two active keys at all times, rotate one while the other continues to serve traffic.

# Python — dual-key rotation with AWS Secrets Manager import boto3, json, datetime class KeyRotator: def __init__(self, secret_name: str): self.client = boto3.client("secretsmanager") self.secret_name = secret_name def rotate(self): # Step 1: Generate new key new_key = self._generate_key() # Step 2: Store as AWSPENDING self.client.put_secret_value( SecretId=self.secret_name, ClientRequestToken=self._token(), SecretString=json.dumps({"key": new_key}), VersionStages=["AWSPENDING"], ) # Step 3: Validate the new key works assert self._validate_key(new_key) # Step 4: Promote AWSPENDING → AWSCURRENT self.client.update_secret_version_stage( SecretId=self.secret_name, VersionStage="AWSCURRENT", MoveToVersionId=self._token(), ) def _generate_key(self) -> str: ... def _validate_key(self, key: str) -> bool: ... def _token(self) -> str: ...
Best practice: Run rotation Lambda functions on a schedule (e.g., every 30 days) and alert on any rotation failure. Never let a key age beyond 90 days in production.

Security Best Practices Checklist

Combining the tools above into a coherent security posture requires organizational discipline as much as technical tooling. Use the checklist below as a foundation for your ML platform's secrets hygiene.

  • No secrets in source control — use pre-commit hooks with tools like gitleaks or trufflehog to block accidental commits.
  • Least-privilege policies — training jobs should not have access to serving-environment secrets, and vice versa.
  • Short-lived credentials — prefer dynamic secrets or tokens with TTLs under 1 hour for automated workloads.
  • Audit logging — every secret read and write should appear in your SIEM. Vault audit logs and CloudTrail are non-negotiable.
  • Encryption at rest and in transit — TLS everywhere, etcd encryption for K8s, and envelope encryption for cloud secrets.
  • Secret zero problem — use workload identity (IRSA, Workload Identity Federation) to eliminate the initial bootstrap secret.
# .pre-commit-config.yaml — block secrets before they hit Git repos: - repo: https://github.com/gitleaks/gitleaks rev: v8.18.0 hooks: - id: gitleaks - repo: https://github.com/trufflesecurity/trufflehog rev: v3.63.0 hooks: - id: trufflehog
Remember: Secrets management is not a one-time setup — it is an ongoing operational practice. Schedule quarterly reviews of secret access policies and rotation schedules to stay ahead of configuration drift.
