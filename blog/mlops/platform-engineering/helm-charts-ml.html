Helm Charts for ML Infrastructure

MLOps Series Platform Engineering

Why Helm for ML?

ML deployments on Kubernetes involve dozens of interrelated manifests: Jobs, Services, ConfigMaps, PVCs, ServiceAccounts, and RBAC rules. Managing them with raw YAML quickly becomes unmanageable. Helm solves this by packaging manifests into charts—versioned, parameterized bundles that can be installed, upgraded, and rolled back with a single command.

Raw YAML

  • Copy-paste across environments
  • Manual find-and-replace for config
  • No rollback mechanism
  • No dependency management

Helm Charts

  • Templated with Go templates
  • Per-environment values.yaml overrides
  • Built-in rollback: helm rollback
  • Dependency resolution via Chart.yaml
Key concept: A Helm release is a deployed instance of a chart with specific values. You can have multiple releases of the same chart (e.g., bert-training-dev and bert-training-prod) in the same cluster.

Chart Directory Structure

A Helm chart for an ML training job follows the standard structure with some ML-specific additions.

ml-training-chart/ ├── Chart.yaml # Chart metadata & dependencies ├── values.yaml # Default configuration values ├── values-dev.yaml # Dev environment overrides ├── values-prod.yaml # Prod environment overrides ├── templates/ │ ├── _helpers.tpl # Reusable template functions │ ├── job.yaml # Training Job manifest │ ├── configmap.yaml # Hyperparameters & config │ ├── pvc.yaml # Persistent volume for checkpoints │ ├── rbac.yaml # ServiceAccount & roles │ ├── secret.yaml # Registry credentials (sealed) │ └── NOTES.txt # Post-install instructions └── tests/ └── test-job.yaml # Helm test (validate connectivity)

Chart.yaml

apiVersion: v2 name: ml-training description: Helm chart for GPU-accelerated ML training jobs type: application version: 1.2.0 # Chart version (semver) appVersion: "3.1.0" # Application/model version dependencies: - name: redis version: "18.x" repository: "https://charts.bitnami.com/bitnami" condition: redis.enabled

Values & Templates

The values.yaml file is the primary interface for chart users. For ML charts, it exposes training hyperparameters, resource requests, and infrastructure toggles.

values.yaml

# ── Image Configuration ── image: repository: myregistry/bert-trainer tag: "v3.1.0" pullPolicy: IfNotPresent # ── Training Configuration ── training: epochs: 50 batchSize: 32 learningRate: 0.0001 modelName: "bert-base-uncased" dataPath: "s3://ml-data/training/v5" checkpointInterval: 5 # ── Resource Requests ── resources: requests: cpu: "4" memory: "32Gi" nvidia.com/gpu: "2" limits: cpu: "8" memory: "64Gi" nvidia.com/gpu: "2" # ── Node Placement ── nodeSelector: gpu-type: a100 tolerations: - key: "nvidia.com/gpu" operator: "Exists" effect: "NoSchedule" # ── Feature Toggles ── redis: enabled: false

Template: job.yaml

apiVersion: batch/v1 kind: Job metadata: name: {{ include "ml-training.fullname" . }} labels: {{- include "ml-training.labels" . | nindent 4 }} spec: backoffLimit: 3 ttlSecondsAfterFinished: 3600 template: metadata: labels: {{- include "ml-training.selectorLabels" . | nindent 8 }} spec: restartPolicy: OnFailure nodeSelector: {{- toYaml .Values.nodeSelector | nindent 8 }} tolerations: {{- toYaml .Values.tolerations | nindent 8 }} containers: - name: trainer image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}" imagePullPolicy: {{ .Values.image.pullPolicy }} resources: {{- toYaml .Values.resources | nindent 12 }} env: - name: EPOCHS value: "{{ .Values.training.epochs }}" - name: BATCH_SIZE value: "{{ .Values.training.batchSize }}" - name: LEARNING_RATE value: "{{ .Values.training.learningRate }}" volumeMounts: - name: checkpoints mountPath: /checkpoints volumes: - name: checkpoints persistentVolumeClaim: claimName: {{ include "ml-training.fullname" . }}-ckpt
Tip: Use helm template . --values values-prod.yaml to render templates locally before deploying. This catches syntax errors and misconfigured values early.

ML-Specific Deployment Patterns

Pattern 1: Training + Evaluation Pipeline

Use Helm hooks to run an evaluation Job after training completes. The post-install hook fires after all chart resources are ready.

# templates/eval-job.yaml apiVersion: batch/v1 kind: Job metadata: name: {{ include "ml-training.fullname" . }}-eval annotations: "helm.sh/hook": post-install "helm.sh/hook-weight": "1" "helm.sh/hook-delete-policy": hook-succeeded spec: template: spec: restartPolicy: Never containers: - name: evaluator image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}" command: ["python3", "src/evaluate.py"]

Pattern 2: ConfigMap for Hyperparameters

Externalize hyperparameters in a ConfigMap so they can be updated without rebuilding the container image.

# templates/configmap.yaml apiVersion: v1 kind: ConfigMap metadata: name: {{ include "ml-training.fullname" . }}-config data: config.yaml: | model_name: {{ .Values.training.modelName }} epochs: {{ .Values.training.epochs }} batch_size: {{ .Values.training.batchSize }} learning_rate: {{ .Values.training.learningRate }} data_path: {{ .Values.training.dataPath }} checkpoint_interval: {{ .Values.training.checkpointInterval }}

Pattern 3: GPU Node Auto-Discovery

Use conditional templates that adapt based on whether GPUs are requested.

# In templates/job.yaml {{- if index .Values.resources.requests "nvidia.com/gpu" }} nodeSelector: accelerator: nvidia tolerations: - key: nvidia.com/gpu operator: Exists effect: NoSchedule {{- end }}
Caution: Helm's Go template syntax is powerful but error-prone. Whitespace handling with {{- and -}} is a common source of invalid YAML. Always validate rendered output with helm template.

Chart Dependencies

ML charts often depend on supporting infrastructure: Redis for caching, MinIO for artifact storage, or Prometheus for monitoring. Helm handles these as sub-charts.

Declaring Dependencies

# Chart.yaml dependencies: - name: redis version: "18.4.0" repository: "https://charts.bitnami.com/bitnami" condition: redis.enabled - name: minio version: "13.2.0" repository: "https://charts.bitnami.com/bitnami" condition: minio.enabled

Managing Dependencies

# Download dependency charts helm dependency update ./ml-training-chart # Install with Redis enabled, MinIO disabled helm install bert-training ./ml-training-chart \ --set redis.enabled=true \ --set minio.enabled=false \ --values values-prod.yaml
Tip: Use the condition field in dependencies so that optional sub-charts are only deployed when explicitly enabled. This keeps lightweight environments (dev, CI) fast and resource-efficient.

Best Practices

1. Semantic Versioning

Follow semver for chart versions. Bump the major version when changing the values schema (breaking change), minor for new features, and patch for fixes.

2. Lint & Test in CI

# CI pipeline step helm lint ./ml-training-chart --values values-prod.yaml helm template ./ml-training-chart --values values-prod.yaml | \ kubeval --strict # Run Helm tests after install helm test bert-training

3. Use Helmfile for Multi-Chart Orchestration

When your ML platform has multiple charts (training, serving, monitoring), use Helmfile to manage them declaratively.

# helmfile.yaml releases: - name: ml-training chart: ./charts/ml-training namespace: ml-jobs values: - values-prod.yaml - name: ml-serving chart: ./charts/ml-serving namespace: ml-serving values: - serving-prod.yaml - name: monitoring chart: prometheus-community/kube-prometheus-stack namespace: monitoring version: 55.5.0

4. Store Charts in OCI Registries

# Package and push to an OCI-compatible registry helm package ./ml-training-chart helm push ml-training-1.2.0.tgz oci://ghcr.io/myorg/helm-charts

5. Document the Values Interface

Every key in values.yaml should have an inline comment explaining its purpose, acceptable values, and default. Tools like helm-docs can auto-generate README files from these comments.

Golden rule: If someone needs to read a template to understand what a value does, the values file needs better documentation.
