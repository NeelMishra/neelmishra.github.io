Ray vs Dask vs Spark

MLOps Series Platform Engineering

Comparison Overview

Ray, Dask, and Spark are the three dominant frameworks for distributed computing in the data and ML ecosystem. While they can overlap in capability, each was designed with a fundamentally different primary use case in mind. Choosing the right tool depends on your workload profile, team expertise, and infrastructure constraints.

Ray

Origin: UC Berkeley RISELab (2017)

Primary focus: Distributed ML — training, serving, reinforcement learning

Language: Python-first (Java support growing)

Sweet spot: GPU-heavy ML workloads, model serving, RL, LLM infrastructure

Dask

Origin: Anaconda / Open-source (2015)

Primary focus: Parallel Python — scale NumPy, pandas, scikit-learn

Language: Pure Python

Sweet spot: Data science workflows, ETL in Python, out-of-core computation

Spark

Origin: UC Berkeley AMPLab (2009)

Primary focus: Large-scale data processing — ETL, SQL, streaming

Language: Scala/Java (PySpark wrapper)

Sweet spot: Petabyte-scale ETL, SQL analytics, data engineering pipelines

Key Insight: These frameworks are not mutually exclusive. Many production platforms use Spark for data engineering, Ray for ML training/serving, and Dask for exploratory data science. The choice is about which tool is primary for your specific workload.

Architecture Comparison

The architectural differences between these frameworks drive their performance characteristics and suitability for different workloads.

Ray Architecture

  • Scheduler: Distributed, bottom-up (per-node + global)
  • Data Model: Object store with zero-copy reads
  • Primitives: Tasks (stateless) + Actors (stateful)
  • Memory: Shared-memory object store (Apache Arrow)
  • Fault Tolerance: Task-level lineage + actor checkpointing

Dask Architecture

  • Scheduler: Centralized (single scheduler process)
  • Data Model: Task graph over chunked collections
  • Primitives: Delayed, Futures, Collections
  • Memory: Worker-local memory with spill-to-disk
  • Fault Tolerance: Task recomputation from lineage

Spark Architecture

  • Scheduler: Centralized driver + DAGScheduler
  • Data Model: RDD / DataFrame (immutable partitions)
  • Primitives: Transformations + Actions
  • Memory: JVM heap + off-heap (Tungsten)
  • Fault Tolerance: RDD lineage + checkpoint to HDFS
Scalability Ceiling: Dask's centralized scheduler can become a bottleneck beyond ~1000 workers for fine-grained tasks. Ray's distributed scheduler scales more linearly. Spark's DAGScheduler handles stage-level scheduling well but individual task granularity is coarser.

Performance Characteristics

Performance varies dramatically based on workload type. No single framework wins across all scenarios. The key differentiators are task granularity, data serialization overhead, and memory management.

The Same Operation in Three Frameworks

Consider a common ETL operation: reading Parquet, filtering, grouping, and aggregating.

# ── Ray Data ── import ray ds = ray.data.read_parquet("s3://data/events/") result = ( ds.filter(lambda r: r["status"] == "completed") .groupby("region") .mean("revenue") ) df = result.to_pandas()
# ── Dask ── import dask.dataframe as dd df = dd.read_parquet("s3://data/events/") result = ( df[df["status"] == "completed"] .groupby("region")["revenue"] .mean() ) pandas_df = result.compute()
# ── PySpark ── from pyspark.sql import SparkSession import pyspark.sql.functions as F spark = SparkSession.builder.getOrCreate() df = spark.read.parquet("s3://data/events/") result = ( df.filter(F.col("status") == "completed") .groupBy("region") .agg(F.mean("revenue")) ) pandas_df = result.toPandas()

Performance Trade-offs

Dimension Ray Dask Spark
Task Latency ~0.5ms (sub-millisecond) ~1ms ~10ms (JVM overhead)
Shuffle Efficiency Good (object store) Good (worker-to-worker) Excellent (sort-based)
SQL Optimization Basic Basic Catalyst optimizer
GPU Scheduling Native, first-class Via RAPIDS Via RAPIDS / plugins
Serialization Arrow (zero-copy) Pickle + msgpack Java serialization / Kryo
Startup Time ~1-2s <1s (local) ~5-15s (JVM init)
Benchmark Caveat: Micro-benchmarks rarely reflect real-world performance. Test with your actual data, cluster size, and workload patterns. A framework that's 2x faster on a synthetic benchmark may be slower on your specific pipeline due to data layout, network topology, or memory constraints.

ML Workflows

This is where the frameworks diverge most dramatically. Ray was built for ML; Dask extends the PyData ecosystem; Spark treats ML as one of many capabilities.

Ray for ML

  • Native distributed training (Ray Train)
  • Production model serving (Ray Serve)
  • Hyperparameter tuning (Ray Tune)
  • Reinforcement learning (RLlib)
  • LLM serving with vLLM integration
  • GPU-first scheduling

Dask for ML

  • Parallel scikit-learn (Dask-ML)
  • Distributed XGBoost / LightGBM
  • Out-of-core feature engineering
  • Hyperparameter search via joblib
  • RAPIDS cuML for GPU ML
  • PyTorch/TF via SaturnCloud

Spark for ML

  • MLlib (distributed traditional ML)
  • Feature Store integration
  • SparkML pipelines
  • Deep learning via Horovod/Petastorm
  • Large-scale feature engineering
  • Mature ecosystem for data prep

Distributed Training Comparison

# ── Ray Train: PyTorch DDP ── from ray.train.torch import TorchTrainer from ray.train import ScalingConfig def train_fn(config): model = train.torch.prepare_model(MyModel()) for epoch in range(config["epochs"]): for batch in train.get_dataset_shard("train").iter_torch_batches(): loss = train_step(model, batch) train.report({"loss": loss}) trainer = TorchTrainer(train_fn, scaling_config=ScalingConfig(num_workers=8, use_gpu=True)) result = trainer.fit()
# ── Spark + Horovod: PyTorch DDP ── from sparkdl import HorovodRunner def train_fn(lr): import horovod.torch as hvd hvd.init() model = MyModel().cuda(hvd.local_rank()) optimizer = torch.optim.Adam(model.parameters(), lr=lr * hvd.size()) optimizer = hvd.DistributedOptimizer(optimizer) # ... training loop with manual distributed setup hr = HorovodRunner(np=8) hr.run(train_fn, lr=0.001)
DX Comparison: Ray Train requires ~50% less boilerplate than Spark + Horovod for distributed PyTorch training. Ray handles process group setup, data sharding, and checkpoint management automatically. With Spark, you must manage Horovod initialization, rank assignment, and gradient synchronization manually.

Ecosystem & Community

The ecosystem surrounding each framework determines how easy it is to integrate with your existing stack and find support for your use cases.

Ray Ecosystem

  • Anyscale — managed Ray platform (by Ray creators)
  • KubeRay — Kubernetes operator
  • vLLM — LLM serving built on Ray
  • RLlib — industry-standard RL library
  • Community: 34k+ GitHub stars, strong ML/AI community

Dask Ecosystem

  • Coiled — managed Dask platform (by Dask creators)
  • Dask-Kubernetes — K8s integration
  • RAPIDS — GPU-accelerated Dask
  • Prefect / Airflow — tight workflow integration
  • Community: 13k+ GitHub stars, strong data science community

Spark Ecosystem

  • Databricks — managed Spark platform (by Spark creators)
  • Delta Lake — ACID table format
  • MLflow — ML lifecycle management
  • Structured Streaming — real-time processing
  • Community: 40k+ GitHub stars, largest data engineering community

Integration Patterns

In practice, organizations often combine these frameworks. Common integration patterns include:

Spark + Ray

Use Spark for petabyte-scale ETL and feature engineering, then feed processed data into Ray for distributed training and model serving. The ray-on-spark project enables running Ray within Spark clusters.

Dask + Ray

Use Dask for interactive data exploration and feature prototyping, then switch to Ray for production training and serving. Both use Python natively, making migration straightforward.

Convergence Trend: The boundaries between these frameworks are blurring. Spark now has better Python support (PySpark on Arrow), Dask is adding GPU support via RAPIDS, and Ray is building stronger data processing capabilities. Choose based on your primary workload, not a single benchmark.

Quick Decision Flowchart

Start here: What is your primary workload?
  • ML training on GPUs → Ray
  • Scale pandas/NumPy workflows → Dask
  • Petabyte-scale ETL or SQL → Spark
  • Model serving with composition → Ray Serve
  • Interactive data exploration → Dask
  • Real-time streaming pipeline → Spark Structured Streaming
  • LLM inference at scale → Ray (vLLM)
  • Mixed workloads → Combine: Spark for data, Ray for ML

Same Task, Three Frameworks

Here is a complete end-to-end example: reading data, computing features, and training an XGBoost model in all three frameworks.

# ── Ray ── import ray from ray.train.xgboost import XGBoostTrainer from ray.train import ScalingConfig ds = ray.data.read_parquet("s3://data/features/") ds = ds.map(lambda r: {**r, "log_val": math.log1p(r["value"])}) trainer = XGBoostTrainer( label_column="label", params={"objective": "binary:logistic", "max_depth": 6}, datasets={"train": ds}, scaling_config=ScalingConfig(num_workers=4) ) result = trainer.fit()
# ── Dask ── import dask.dataframe as dd import xgboost as xgb from dask.distributed import Client client = Client() df = dd.read_parquet("s3://data/features/") df["log_val"] = np.log1p(df["value"]) X, y = df.drop("label", axis=1), df["label"] dtrain = xgb.dask.DaskDMatrix(client, X, y) output = xgb.dask.train(client, {"objective": "binary:logistic"}, dtrain, num_boost_round=500)
# ── PySpark ── from pyspark.sql import SparkSession from pyspark.ml.classification import GBTClassifier from pyspark.ml.feature import VectorAssembler import pyspark.sql.functions as F spark = SparkSession.builder.getOrCreate() df = spark.read.parquet("s3://data/features/") df = df.withColumn("log_val", F.log1p("value")) assembler = VectorAssembler(inputCols=["log_val", "feat_a", "feat_b"], outputCol="features") df = assembler.transform(df) gbt = GBTClassifier(labelCol="label", featuresCol="features", maxDepth=6, maxIter=500) model = gbt.fit(df)
Final Note: Framework choice is not permanent. Start with what matches your team's skills and primary workload. As needs evolve, you can adopt additional frameworks for specific use cases. The most successful ML platforms use the right tool for each job rather than forcing one framework to do everything.
