GPU Virtualization: MIG, MPS, Time-Slicing

MLOps Series Platform Engineering

Virtualization Overview

A single NVIDIA H100 has 80 GB of HBM3 and 16,896 CUDA cores. Many ML workloads—inference serving, hyperparameter sweeps, small-model training—use only a fraction of this capacity. GPU virtualization allows multiple workloads to share a single physical GPU, dramatically improving utilisation and reducing cost per workload.

NVIDIA provides three primary mechanisms for GPU sharing: Multi-Instance GPU (MIG), Multi-Process Service (MPS), and time-slicing. Each operates at a different layer of the GPU stack and offers different trade-offs between isolation, performance, and flexibility.

Rule of thumb: Use MIG when you need hard isolation (multi-tenant clusters, SLA-bound inference). Use MPS when you need concurrent kernel execution (many small inference requests). Use time-slicing for development environments where isolation is less critical and flexibility matters more.

MIG Architecture

Multi-Instance GPU (MIG) partitions a physical GPU into up to seven isolated instances at the hardware level. Each MIG instance has dedicated streaming multiprocessors (SMs), dedicated memory bandwidth, and its own L2 cache partition. A fault in one instance cannot affect others—this is true hardware isolation.

MIG is available on NVIDIA A100 and H100 GPUs. An A100 80 GB can be split into profiles like 1g.10gb (1/7 of SMs, 10 GB VRAM), 2g.20gb, 3g.40gb, or 4g.40gb. An H100 supports similar profiles with higher throughput per instance.

# Enable MIG mode on GPU 0 $ sudo nvidia-smi -i 0 --mig 1 # Reboot may be required after enabling MIG $ sudo reboot # Create MIG instances: 3x 2g.20gb + 1x 1g.10gb on A100 80GB $ sudo nvidia-smi mig -cgi 14,14,14,19 -C # Profile ID 14 = 2g.20gb, Profile ID 19 = 1g.10gb # List created instances $ nvidia-smi mig -lgi +-------------------------------------------------------+ | GPU instances: | | GPU Name Profile Instances Free/Total | | 0 MIG 2g.20gb 14 3 0/3 | | 0 MIG 1g.10gb 19 1 0/1 | +-------------------------------------------------------+ # Destroy all MIG instances $ sudo nvidia-smi mig -dci && sudo nvidia-smi mig -dgi

MIG in Kubernetes

The NVIDIA GPU Operator and device plugin can expose MIG instances as schedulable resources in Kubernetes. There are two strategies: single (one MIG profile per node) and mixed (multiple profiles per node).

# NVIDIA device plugin ConfigMap for mixed MIG strategy apiVersion: v1 kind: ConfigMap metadata: name: nvidia-device-plugin namespace: kube-system data: config.yaml: | version: v1 sharing: mig: strategy: mixed resources: - name: nvidia.com/mig-2g.20gb replicas: 1 - name: nvidia.com/mig-1g.10gb replicas: 1 --- # Pod requesting a specific MIG instance apiVersion: v1 kind: Pod metadata: name: inference-server spec: containers: - name: model image: tritonserver:latest resources: limits: nvidia.com/mig-2g.20gb: 1
Operational note: Changing MIG profiles requires draining workloads and may require a GPU reset. Plan profile changes during maintenance windows. Some organisations pre-configure node pools with fixed MIG profiles (e.g., a “mig-7x1g” pool and a “mig-3x2g+1x1g” pool) to avoid runtime reconfiguration.

Multi-Process Service

NVIDIA Multi-Process Service (MPS) allows multiple CUDA processes to share a GPU simultaneously by multiplexing their work through a single CUDA context. Unlike MIG, MPS does not provide hardware isolation—all processes share the same memory space and SMs. However, MPS enables true concurrent kernel execution, which is ideal for many small inference requests that individually underutilise the GPU.

# Start MPS daemon on GPU 0 $ export CUDA_VISIBLE_DEVICES=0 $ export CUDA_MPS_PIPE_DIRECTORY=/var/run/nvidia-mps $ export CUDA_MPS_LOG_DIRECTORY=/var/log/nvidia-mps $ nvidia-cuda-mps-control -d # start daemon # Set per-client SM limit (e.g., 25% of SMs per client) $ echo "set_default_active_thread_percentage 25" | \ nvidia-cuda-mps-control # Verify MPS is running $ echo "get_default_active_thread_percentage" | \ nvidia-cuda-mps-control # Output: 25 # Stop MPS daemon $ echo quit | nvidia-cuda-mps-control

MPS supports up to 48 concurrent clients on Volta and newer GPUs. Each client can be assigned a maximum percentage of SMs using CUDA_MPS_ACTIVE_THREAD_PERCENTAGE. On Hopper GPUs (H100), MPS also supports per-client memory limits, closing one of the major isolation gaps.

Performance benefit: For inference workloads with small batch sizes, MPS can improve throughput by 2–4× compared to time-slicing because kernels from different processes execute truly concurrently on different SMs, rather than context-switching between them.

Time-Slicing

Time-slicing is the simplest GPU sharing mechanism. The GPU driver context-switches between processes at regular intervals (typically every few milliseconds), giving each process the illusion of exclusive GPU access. No special hardware or daemon is required—it works on all NVIDIA GPUs.

# Kubernetes NVIDIA device plugin with time-slicing apiVersion: v1 kind: ConfigMap metadata: name: nvidia-device-plugin namespace: kube-system data: config.yaml: | version: v1 sharing: timeSlicing: renameByDefault: false resources: - name: nvidia.com/gpu replicas: 4 # each GPU appears as 4 virtual GPUs

With replicas: 4, a node with 8 physical GPUs advertises 32 virtual GPUs to the Kubernetes scheduler. Each pod requesting nvidia.com/gpu: 1 gets a time-slice on a physical GPU.

No memory isolation: Time-slicing does not partition GPU memory. If four pods share a 16 GB GPU, each can theoretically allocate the full 16 GB—but the combined allocation cannot exceed 16 GB. An OOM in one pod can crash other pods sharing the GPU. Always set CUDA_MEM_FRACTION or use framework-level memory limits.

Comparison

The following comparison summarises the key differences across MIG, MPS, and time-slicing to help you choose the right approach for your workload.

MIG

Isolation: Hardware-level. Dedicated SMs, memory, and L2 cache.

Max clients: 7 per GPU (A100/H100).

Overhead: Near-zero. Each instance behaves like a small GPU.

Memory: Hard partitioned. No oversubscription.

GPU support: A100, H100 only.

Reconfiguration: Requires draining workloads.

MPS

Isolation: Software-level. SM limits per client; shared memory (Hopper adds memory limits).

Max clients: 48 per GPU.

Overhead: Low. True concurrent execution.

Memory: Shared (soft limits on Hopper). OOM risk.

GPU support: Volta and newer.

Reconfiguration: Dynamic. No drain required.

Time-Slicing

Isolation: None. Context-switching between processes.

Max clients: Unlimited (practically 8–16).

Overhead: High. Context-switch penalty per slice.

Memory: Shared. No limits. OOM risk.

GPU support: All NVIDIA GPUs.

Reconfiguration: Dynamic. ConfigMap change only.

Decision shortcut: Need SLA guarantees? Use MIG. Need max throughput for many small inference calls? Use MPS. Need quick sharing for dev/test? Use time-slicing.

Use Cases

Each GPU sharing mechanism aligns with specific operational patterns. Here are concrete deployment scenarios from production ML platforms.

MIG: Multi-Tenant Inference Clusters

A platform serving models for multiple internal teams partitions each H100 into 3× 2g.20gb instances plus 1× 1g.10gb. Each team gets a dedicated MIG instance with guaranteed performance—no noisy-neighbour effects. The small instance runs a lightweight monitoring model.

MPS: Real-Time Recommendation Serving

A recommendation engine handles thousands of requests per second, each requiring a forward pass through a small embedding model. Individual requests use <5% of GPU compute. MPS enables 8 inference server replicas to execute kernels concurrently on one GPU, achieving 3× the throughput of time-slicing.

Time-Slicing: Data Science Notebooks

A shared Jupyter Hub cluster gives data scientists GPU access for exploratory work. Most notebooks are idle 90% of the time. Time-slicing with replicas: 8 allows 8 notebooks per GPU. The occasional OOM is acceptable for development use, and the cost savings are substantial.

Hybrid: MIG + MPS

An advanced pattern combines MIG and MPS. Create a 3g.40gb MIG instance for training and a 4g.40gb MIG instance for inference. Within the inference MIG instance, run MPS with 16 clients for maximum serving throughput. This combines hardware isolation between workload types with software-level concurrency within the inference partition.

# Hybrid MIG + MPS setup on H100 # Step 1: Create MIG instances $ sudo nvidia-smi mig -cgi 9,5 -C # Profile 9 = 3g.40gb (training) # Profile 5 = 4g.40gb (inference with MPS) # Step 2: Start MPS on the inference MIG instance $ export CUDA_VISIBLE_DEVICES=MIG-uuid-of-4g-instance $ export CUDA_MPS_PIPE_DIRECTORY=/var/run/nvidia-mps-inference $ nvidia-cuda-mps-control -d $ echo "set_default_active_thread_percentage 12" | \ nvidia-cuda-mps-control # Step 3: Deploy 8 inference replicas on MPS # Each replica gets ~12% of the 4g instance's SMs
Memory planning: Always calculate total memory requirements before configuring GPU sharing. For MIG, partitions are fixed. For MPS and time-slicing, set PYTORCH_CUDA_ALLOC_CONF or TF_GPU_ALLOCATOR to limit per-process memory allocation. Uncontrolled allocation is the number one cause of GPU sharing failures in production.
