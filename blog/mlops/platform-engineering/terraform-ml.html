Terraform for ML: IaC Patterns

MLOps Series Platform Engineering

Why Infrastructure as Code for ML?

Machine learning workloads demand reproducible infrastructure. A training job that works today must be replicable months later with the exact same GPU configuration, storage layout, and network topology. Terraform provides declarative, version-controlled infrastructure that eliminates "it worked on my cluster" problems.

Key principle: Every piece of ML infrastructure—from GPU instances to S3 buckets to VPC peering—should be defined in code, reviewed in PRs, and deployed through CI/CD.

The Terraform workflow for ML infrastructure follows a predictable pattern: define resources in HCL, plan changes, review the execution plan, and apply. This gives teams a clear audit trail of every infrastructure change.

Compared to ClickOps (manual console clicks), Terraform ensures that infrastructure is reproducible, auditable, and scalable. When your team grows from one data scientist to twenty, IaC prevents configuration drift and reduces onboarding time from days to minutes.

Provisioning GPU Instances

GPU instances are the backbone of ML training. Terraform lets you define GPU instance configurations declaratively, ensuring consistent environments across development, staging, and production.

# AWS GPU instance for ML training resource "aws_instance" "ml_training" { ami = data.aws_ami.deep_learning.id instance_type = "p3.2xlarge" # 1x V100 GPU root_block_device { volume_size = 200 volume_type = "gp3" throughput = 250 } tags = { Name = "ml-training-${var.experiment_id}" Environment = var.environment Project = var.project_name ManagedBy = "terraform" } }
Cost alert: GPU instances are expensive. Always use prevent_destroy lifecycle rules on persistent resources and implement auto-shutdown scripts for training instances that idle for more than 30 minutes.

For multi-GPU training, consider larger instance types. Here is a comparison of common GPU instance families:

p3 Family (V100)

p3.2xlarge: 1 GPU, 16 GB VRAM, 61 GB RAM

p3.8xlarge: 4 GPUs, 64 GB VRAM, 244 GB RAM

Best for: Standard deep learning training, medium-scale experiments.

p4d Family (A100)

p4d.24xlarge: 8 GPUs, 320 GB VRAM, 1.1 TB RAM

Best for: Large model training, distributed workloads, LLM fine-tuning.

You can also manage spot instances for cost savings. Spot instances can reduce GPU costs by 60–90%, but require fault-tolerant training code with checkpointing:

# Spot instance request for cost-effective training resource "aws_spot_instance_request" "ml_spot" { ami = data.aws_ami.deep_learning.id instance_type = "p3.2xlarge" spot_price = "1.50" wait_for_fulfillment = true spot_type = "one-time" # Ensure training checkpoints are saved to S3 user_data = templatefile("scripts/training_init.sh", { s3_checkpoint_bucket = var.checkpoint_bucket experiment_id = var.experiment_id }) }

Storage Configuration for ML Workloads

ML pipelines require diverse storage: object stores for datasets, file systems for shared model artifacts, and block storage for high-throughput training I/O. Terraform manages all of these in a unified configuration.

# S3 bucket for training datasets with versioning resource "aws_s3_bucket" "ml_datasets" { bucket = "${var.project}-datasets-${var.environment}" tags = { Purpose = "ml-training-data" ManagedBy = "terraform" } } resource "aws_s3_bucket_versioning" "ml_datasets" { bucket = aws_s3_bucket.ml_datasets.id versioning_configuration { status = "Enabled" } } # EFS for shared model artifacts across instances resource "aws_efs_file_system" "ml_shared" { creation_token = "ml-shared-${var.environment}" encrypted = true performance_mode = "generalPurpose" throughput_mode = "bursting" lifecycle_policy { transition_to_ia = "AFTER_30_DAYS" } }
Data versioning tip: Enable S3 versioning for all training data buckets. When a model regresses, you need to trace back to the exact dataset version used for the last known-good training run.

For high-performance training I/O, consider FSx for Lustre, which integrates natively with S3 and delivers sub-millisecond latencies:

# FSx for Lustre - high-throughput training filesystem resource "aws_fsx_lustre_file_system" "training_fs" { storage_capacity = 1200 # GiB, minimum 1200 subnet_ids = [var.training_subnet_id] deployment_type = "SCRATCH_2" import_path = "s3://${aws_s3_bucket.ml_datasets.id}" auto_import_policy = "NEW_CHANGED" }

Networking for ML Infrastructure

Distributed training requires carefully configured networking. GPU instances performing all-reduce operations need low-latency, high-bandwidth interconnects. Terraform defines VPCs, subnets, and security groups that support these requirements.

# VPC for ML workloads with private subnets resource "aws_vpc" "ml_vpc" { cidr_block = "10.0.0.0/16" enable_dns_hostnames = true enable_dns_support = true tags = { Name = "ml-vpc-${var.environment}" } } # Private subnet for training instances resource "aws_subnet" "ml_training" { vpc_id = aws_vpc.ml_vpc.id cidr_block = "10.0.1.0/24" availability_zone = "${var.region}a" tags = { Name = "ml-training-subnet" } } # Security group allowing inter-node communication resource "aws_security_group" "ml_training" { name = "ml-training-sg" vpc_id = aws_vpc.ml_vpc.id # NCCL communication between GPU nodes ingress { from_port = 0 to_port = 65535 protocol = "tcp" self = true } egress { from_port = 0 to_port = 0 protocol = "-1" cidr_blocks = ["0.0.0.0/0"] } }

For multi-node training with frameworks like PyTorch DDP or Horovod, placement groups ensure instances are co-located for maximum network performance:

# Placement group for distributed training resource "aws_placement_group" "ml_cluster" { name = "ml-training-cluster" strategy = "cluster" # Low-latency group } # Launch template with placement group + EFA resource "aws_launch_template" "distributed_training" { name_prefix = "ml-distributed-" image_id = data.aws_ami.deep_learning.id instance_type = "p4d.24xlarge" placement { group_name = aws_placement_group.ml_cluster.name } network_interfaces { device_index = 0 network_card_index = 0 interface_type = "efa" # Elastic Fabric Adapter security_groups = [aws_security_group.ml_training.id] } }
EFA requirement: Elastic Fabric Adapter (EFA) is only supported on specific instance types (p4d, p5, trn1). Verify your chosen instance type supports EFA before adding it to your Terraform configuration.

Reusable Terraform Modules for ML

As ML teams grow, repeating infrastructure patterns across projects wastes time and introduces inconsistencies. Terraform modules encapsulate common ML infrastructure patterns into reusable, versioned components.

# Module structure for an ML training cluster # modules/ml-training-cluster/ # ├── main.tf # ├── variables.tf # ├── outputs.tf # └── README.md # variables.tf variable "instance_type" { description = "GPU instance type" type = string default = "p3.2xlarge" validation { condition = contains(["p3.2xlarge","p3.8xlarge","p4d.24xlarge"], var.instance_type) error_message = "Must be a supported GPU instance type." } } variable "num_workers" { description = "Number of training worker nodes" type = number default = 1 } variable "checkpoint_bucket" { description = "S3 bucket for training checkpoints" type = string }

Consuming the module in a project is straightforward and keeps the root configuration clean:

# Root configuration calling the module module "training_cluster" { source = "./modules/ml-training-cluster" instance_type = "p4d.24xlarge" num_workers = 4 checkpoint_bucket = aws_s3_bucket.checkpoints.id environment = var.environment } output "training_instance_ids" { value = module.training_cluster.instance_ids }

You can also invoke Terraform programmatically from Python, integrating IaC into ML pipeline orchestrators:

# Python wrapper for Terraform operations import subprocess import json def provision_training_cluster(config: dict) -> dict: """Provision ML training infrastructure via Terraform.""" # Write variable values with open("terraform.tfvars.json", "w") as f: json.dump(config, f) # Init and apply subprocess.run(["terraform", "init"], check=True) subprocess.run( ["terraform", "apply", "-auto-approve"], check=True ) # Retrieve outputs result = subprocess.run( ["terraform", "output", "-json"], capture_output=True, text=True ) return json.loads(result.stdout)

State Management & Best Practices

Terraform state is the single source of truth for your infrastructure. For ML teams, remote state with locking is essential to prevent conflicting changes when multiple data scientists provision resources simultaneously.

# Remote backend with S3 + DynamoDB locking terraform { backend "s3" { bucket = "ml-team-terraform-state" key = "ml-platform/training/terraform.tfstate" region = "us-east-1" encrypt = true dynamodb_table = "terraform-lock" } required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } } }
State isolation: Use separate state files per environment (dev/staging/prod) and per team. A single monolithic state file becomes a bottleneck when multiple teams are deploying ML infrastructure.

Organize state by concern: separate training infrastructure from serving infrastructure, and both from shared resources like VPCs and IAM roles. This minimizes blast radius and speeds up plan/apply cycles.

✅ Good: Isolated State

training/terraform.tfstate

serving/terraform.tfstate

shared/terraform.tfstate

Each concern evolves independently. Fast plans, minimal lock contention.

❌ Bad: Monolithic State

everything/terraform.tfstate

Every change requires a full plan of all resources. Slow. Risky. Frequent lock conflicts between teams.

Finally, always tag resources with ownership and purpose metadata. This enables cost attribution, automated cleanup of orphaned resources, and compliance auditing:

# Consistent tagging via default_tags provider "aws" { region = var.region default_tags { tags = { Team = var.team_name Project = var.project_name Environment = var.environment ManagedBy = "terraform" CostCenter = var.cost_center } } }
Never commit state files: Add *.tfstate and *.tfstate.backup to your .gitignore. State files can contain sensitive values like database passwords and API keys.
