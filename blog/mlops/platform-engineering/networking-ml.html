Networking: InfiniBand, RoCE, High-Bandwidth

MLOps Series Platform Engineering

Networking Overview for Distributed ML

Large-scale model training is fundamentally a distributed systems problem. When a model's parameters exceed the memory of a single accelerator, you partition across dozens or hundreds of GPUs — and the network fabric between them becomes the critical bottleneck. Gradient synchronization during data-parallel training, activation transfers in pipeline parallelism, and parameter sharding in tensor parallelism all demand extreme bandwidth and ultra-low latency.

Traditional TCP/IP Ethernet introduces kernel overhead, context switches, and buffer copies that are unacceptable at scale. Modern ML clusters rely on Remote Direct Memory Access (RDMA) to bypass the CPU and OS kernel entirely, enabling GPU-to-GPU transfers at near line-rate speeds.

Key principle: In distributed training, compute is cheap relative to communication. A 10% improvement in network throughput can translate to a 15–25% reduction in total training time for communication-bound workloads.

The RDMA Advantage

RDMA allows one machine to read from or write to the memory of another machine without involving either machine's operating system. This eliminates the overhead of traditional socket-based networking: no system calls, no buffer copies, no interrupt handling. The NIC itself manages the data transfer using hardware queue pairs.

// Traditional TCP/IP path (simplified) Application → Socket API → Kernel TCP stack → NIC driver → NIC → Wire // 4+ memory copies, multiple context switches // RDMA path Application → RDMA Verb → NIC (hardware) → Wire // Zero-copy, kernel bypass, single-digit μs latency

InfiniBand Deep Dive

InfiniBand (IB) is a switched-fabric interconnect designed from the ground up for high-performance computing. Unlike Ethernet, which evolved from LAN protocols, IB was purpose-built for lossless, low-latency communication. It natively supports RDMA and provides credit-based flow control that guarantees zero packet loss within the fabric.

InfiniBand Generations

HDR InfiniBand

200 Gb/s per port. Widely deployed in current GPU clusters. Supports 4× HDR100 split connections. Typical end-to-end latency: ~1 μs.

NDR InfiniBand

400 Gb/s per port. Latest generation from NVIDIA/Mellanox. 2× bandwidth of HDR with improved congestion management and adaptive routing.

Each GPU in a modern training node is connected via a dedicated ConnectX-7 HCA (Host Channel Adapter). In an 8-GPU DGX-style node, all 8 GPUs have independent IB ports, enabling 3.2 Tb/s of aggregate bisection bandwidth per node with NDR.

Watch out: InfiniBand requires specialized switches (Quantum-2 for NDR), subnet managers, and dedicated cabling. Budget 30–40% of your GPU node cost for the IB fabric in a new cluster build.

Configuring IB for NCCL

# Key NCCL environment variables for InfiniBand export NCCL_IB_DISABLE=0 export NCCL_IB_HCA=mlx5_0,mlx5_1,mlx5_2,mlx5_3 export NCCL_IB_GID_INDEX=3 export NCCL_NET_GDR_LEVEL=5 # Enable GPUDirect RDMA export NCCL_IB_QPS_PER_CONNECTION=4 # Queue pairs per connection export NCCL_IB_TC=128 # Traffic class for QoS # Verify IB devices are visible ibstat | grep -E "State|Rate" # Expected: State: Active, Rate: 200 (HDR) or 400 (NDR)

RoCE & RDMA over Ethernet

RDMA over Converged Ethernet (RoCE) brings RDMA semantics to standard Ethernet networks. RoCEv2, the widely deployed version, encapsulates RDMA packets in UDP/IP, allowing them to be routed across L3 networks. This makes RoCE attractive for organizations that want RDMA performance without a dedicated InfiniBand fabric.

RoCE vs InfiniBand Comparison

InfiniBand

  • Native lossless fabric
  • Built-in adaptive routing
  • Sub-microsecond latency
  • Dedicated hardware & cabling
  • Higher cost, higher performance

RoCEv2

  • Relies on PFC/ECN for lossless
  • Uses standard Ethernet switches
  • ~2–3 μs latency typical
  • Reuses existing network infra
  • Lower cost, good performance
When to choose RoCE: If you already have a high-quality Ethernet fabric with DCQCN-capable switches (e.g., Arista 7060X, Cisco Nexus 9000), RoCEv2 can deliver 80–90% of InfiniBand throughput at significantly lower fabric cost.

Priority Flow Control Setup

# Configure PFC on the NIC for RoCEv2 lossless operation mlnx_qos -i eth0 --pfc 0,0,0,1,0,0,0,0 # Enable PFC on priority 3 mlnx_qos -i eth0 --trust dscp # NCCL environment for RoCE export NCCL_IB_DISABLE=0 export NCCL_IB_GID_INDEX=3 # Use RoCEv2 GID export NCCL_NET_GDR_LEVEL=5 export NCCL_IB_ROCE_VERSION_NUM=2 # Force RoCEv2 # Validate RoCE connectivity show_gids | grep -i roce ib_write_bw -d mlx5_0 --use_cuda=0 # Bandwidth test with GPUDirect

NVLink & NVSwitch

While InfiniBand and RoCE handle inter-node communication, NVLink and NVSwitch provide the intra-node GPU interconnect. NVLink is NVIDIA's proprietary high-speed link that connects GPUs directly, bypassing the PCIe bus entirely.

NVLink Generations

NVLink 3.0 (A100)

12 links per GPU × 25 GB/s each = 600 GB/s bidirectional bandwidth per GPU. Connected via NVSwitch in DGX A100 for full all-to-all topology.

NVLink 4.0 (H100)

18 links per GPU × 25 GB/s each = 900 GB/s bidirectional bandwidth per GPU. Third-generation NVSwitch enables 256 GPU all-to-all domains.

NVSwitch acts as a crossbar switch that connects all GPUs within a node (or across nodes in DGX SuperPOD configurations). With NVSwitch, every GPU can communicate with every other GPU at full NVLink bandwidth simultaneously — no oversubscription.

Querying NVLink Topology

# Check NVLink status and topology nvidia-smi nvlink -s # NVLink status nvidia-smi topo -m # GPU topology matrix # Python: verify NVLink bandwidth programmatically import subprocess, re result = subprocess.run( ["nvidia-smi", "nvlink", "-s"], capture_output=True, text=True ) for line in result.stdout.splitlines(): if "Link" in line: print(line.strip())

Network Topology Design

The physical and logical topology of your cluster network determines the worst-case communication latency and the effective bisection bandwidth available to collective operations like AllReduce. For ML training, three topologies dominate:

Fat-Tree (Clos) Topology

The most common topology for GPU clusters. A 2-level or 3-level fat-tree built with InfiniBand or Ethernet leaf-spine switches provides full bisection bandwidth — every pair of nodes can communicate at full link speed simultaneously. This is ideal for data-parallel training where AllReduce patterns create all-to-all traffic.

Rail-Optimized Topology

In rail-optimized designs, each GPU index across all nodes shares a dedicated network "rail." GPU 0 on every node connects to switch rail-0, GPU 1 to rail-1, and so on. This exploits the fact that NCCL's ring and tree algorithms naturally group communication by GPU index.

Rail-optimized benefits: Reduces inter-switch traffic by 60–70% compared to a random mapping. NVIDIA's DGX SuperPOD reference architecture uses rail-optimized topology by default.

Benchmarking & Performance Tuning

Before running production training, you must validate that your network fabric delivers expected bandwidth and latency. NCCL's built-in benchmarks and low-level RDMA tools give you the data to identify bottlenecks.

NCCL All-Reduce Benchmark

# Build and run NCCL tests git clone https://github.com/NVIDIA/nccl-tests.git cd nccl-tests && make MPI=1 CUDA_HOME=/usr/local/cuda # Run AllReduce across 8 nodes × 8 GPUs = 64 GPUs mpirun -np 64 --hostfile hosts \ -x NCCL_IB_HCA=mlx5_0,mlx5_1,mlx5_2,mlx5_3 \ -x NCCL_NET_GDR_LEVEL=5 \ -x NCCL_DEBUG=INFO \ ./build/all_reduce_perf -b 1M -e 4G -f 2 -g 1 # Target: >90% of theoretical bus bandwidth # HDR IB: ~185 Gb/s effective (of 200 Gb/s) # NDR IB: ~370 Gb/s effective (of 400 Gb/s)

Key Metrics to Track

Bandwidth Metrics

  • Bus bandwidth: Effective throughput of AllReduce in GB/s
  • Algorithm bandwidth: Raw data transferred / time
  • Line-rate utilization: % of theoretical peak

Latency Metrics

  • Small-message latency: Target < 5 μs for IB
  • Tail latency (p99): Detect congestion/PFC storms
  • Jitter: Variance in latency across runs
Common pitfall: If your AllReduce bandwidth is significantly below line rate, check for PCI topology mismatches — ensure each GPU's NIC is on the same NUMA node. Use nvidia-smi topo -m and lstopo to verify affinity.

Automated Health Check

import subprocess, sys def network_health_check(): checks = { "ib_devices": "ibstat | grep Active | wc -l", "gdr_support": "nvidia-smi topo -m | head -20", "pcie_width": "lspci -vvv | grep Width | head -8", } for name, cmd in checks.items(): result = subprocess.run(cmd, shell=True, capture_output=True, text=True) print(f"[{name}]") print(result.stdout.strip()) print() network_health_check()

Networking is the backbone of distributed ML. Whether you choose InfiniBand for raw performance or RoCE for cost-efficient RDMA, the principles remain the same: minimize hops, maximize bandwidth, and verify everything with benchmarks before committing GPU hours to training.
