Kubernetes for ML: Pods, Jobs, GPU Scheduling

MLOps Series Platform Engineering

Why Kubernetes for ML?

Kubernetes has become the de-facto orchestration layer for ML workloads. Training jobs demand elastic compute, GPU isolation, and fault-tolerant scheduling—all of which K8s provides out of the box when configured correctly.

ML workloads differ from traditional microservices in critical ways: they are batch-oriented, GPU-hungry, and often require gang scheduling (all-or-nothing pod placement). Understanding these differences is key to running ML on K8s effectively.

Web Services

  • Long-running Deployments
  • CPU-bound, horizontally scaled
  • Rolling updates & readiness probes
  • Stateless by design

ML Training Jobs

  • Finite-duration Jobs / CronJobs
  • GPU-bound, vertically scaled
  • Checkpointing & restart policies
  • Stateful (model weights, data cache)
Key insight: K8s Jobs (not Deployments) are the correct abstraction for training runs. Jobs guarantee completion semantics and support parallelism, backoff limits, and TTL-based cleanup.

The GPU Device Plugin

NVIDIA's k8s-device-plugin exposes GPUs as a schedulable resource (nvidia.com/gpu) to the kubelet. Without it, K8s has no awareness of GPU hardware on nodes.

Deploying the Plugin

# Install the NVIDIA device plugin as a DaemonSet apiVersion: apps/v1 kind: DaemonSet metadata: name: nvidia-device-plugin-daemonset namespace: kube-system spec: selector: matchLabels: name: nvidia-device-plugin-ds template: metadata: labels: name: nvidia-device-plugin-ds spec: tolerations: - key: nvidia.com/gpu operator: Exists effect: NoSchedule containers: - name: nvidia-device-plugin-ctr image: nvcr.io/nvidia/k8s-device-plugin:v0.14.3 securityContext: allowPrivilegeEscalation: false volumeMounts: - name: device-plugin mountPath: /var/lib/kubelet/device-plugins
Warning: The device plugin requires the NVIDIA driver and container toolkit to be installed on every GPU node before deploying the DaemonSet. Without them, the plugin pod will enter CrashLoopBackOff.

How It Works

The plugin registers with the kubelet via gRPC, advertises available GPU count, and handles device allocation during pod scheduling. Each GPU is exclusively assigned—there is no time-sharing at the plugin level.

Resource Requests & Limits

GPU resources are non-compressible in K8s. Unlike CPU (which can be throttled), a GPU request is all-or-nothing: if a pod asks for 2 GPUs and only 1 is free, the pod stays Pending.

# Training job requesting GPU resources apiVersion: batch/v1 kind: Job metadata: name: bert-finetune-job spec: backoffLimit: 3 ttlSecondsAfterFinished: 3600 template: spec: restartPolicy: OnFailure containers: - name: trainer image: myregistry/bert-trainer:v2.1 resources: requests: cpu: "4" memory: "32Gi" nvidia.com/gpu: "2" limits: cpu: "8" memory: "64Gi" nvidia.com/gpu: "2" env: - name: CUDA_VISIBLE_DEVICES value: "all"
Note: For GPUs, requests and limits must be equal. K8s does not support GPU overcommit—setting different values will cause a validation error.

Node Affinity & Selectors

In heterogeneous clusters (mixed CPU/GPU nodes, different GPU types), node selectors and affinity rules ensure ML pods land on the right hardware.

Simple Node Selector

# Target nodes with A100 GPUs spec: nodeSelector: gpu-type: a100 accelerator: nvidia

Advanced Affinity

# Prefer A100 nodes, but accept V100 as fallback spec: affinity: nodeAffinity: preferredDuringSchedulingIgnoredDuringExecution: - weight: 100 preference: matchExpressions: - key: gpu-type operator: In values: [a100] requiredDuringSchedulingIgnoredDuringExecution: nodeSelectorTerms: - matchExpressions: - key: gpu-type operator: In values: [a100, v100]

Taints & Tolerations

GPU nodes are typically tainted to prevent non-GPU workloads from consuming resources. Only pods with the matching toleration can be scheduled there.

# Taint on the node kubectl taint nodes gpu-node-01 nvidia.com/gpu=present:NoSchedule # Toleration in the pod spec tolerations: - key: "nvidia.com/gpu" operator: "Equal" value: "present" effect: "NoSchedule"

Best Practices

1. Always Set Resource Requests

Without explicit GPU requests, the scheduler cannot differentiate GPU-hungry pods from lightweight ones. Every ML pod must declare nvidia.com/gpu.

2. Use TTL for Completed Jobs

ttlSecondsAfterFinished automatically garbage-collects finished Job pods, preventing resource leaks and stale entries in etcd.

3. Implement Checkpointing

# Python: Save checkpoint every N epochs import torch def save_checkpoint(model, optimizer, epoch, path): torch.save({ 'epoch': epoch, 'model_state_dict': model.state_dict(), 'optimizer_state_dict': optimizer.state_dict(), }, path) for epoch in range(num_epochs): train_one_epoch(model, dataloader) if epoch % 5 == 0: save_checkpoint(model, optimizer, epoch, f"/checkpoints/ckpt_{epoch}.pt")

4. Use PriorityClasses

Define PriorityClasses so that critical training jobs can preempt lower-priority workloads when GPU resources are scarce.

apiVersion: scheduling.k8s.io/v1 kind: PriorityClass metadata: name: ml-training-high value: 1000000 globalDefault: false description: "High priority for production training jobs"

5. Monitor GPU Utilization

Deploy dcgm-exporter alongside Prometheus to track GPU utilization, memory, temperature, and error counts per pod.

Pro tip: Combine kube-scheduler metrics with DCGM data to identify scheduling bottlenecks versus actual GPU underutilization.
