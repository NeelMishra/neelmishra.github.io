Designing an Internal ML Platform

MLOps Series Platform Engineering

Why Build an Internal ML Platform?

Every organization that operates more than a handful of ML models eventually faces the same bottleneck: data scientists spend more time wrangling infrastructure than building models. An internal ML platform is the answer — a shared, opinionated layer of tooling that turns weeks of DevOps work into a single CLI command or API call.

The best platforms share three properties: they are self-service (no tickets required to train or deploy), opinionated (golden paths reduce cognitive load), and observable (every job, model, and endpoint emits structured telemetry).

Key insight: The platform's primary customer is the data scientist, not the infrastructure team. Design every interface from their perspective first, then work backward to the underlying infrastructure.

Team Structure & Ownership

A platform team is not a traditional infrastructure team. It operates like an internal product team — with a product manager, engineering leads, and a roadmap informed by user research (interviews with data scientists, support ticket analysis, usage telemetry).

Embedded Model

  • Platform engineers sit within ML teams
  • Fast feedback loops
  • Risk of fragmentation across teams
  • Best for < 5 ML teams

Centralized Model

  • Dedicated platform team serves all ML teams
  • Consistent tooling & standards
  • Requires strong user-research discipline
  • Best for 5+ ML teams at scale

Regardless of structure, the team should own four domains: compute orchestration (training jobs, GPU scheduling), model lifecycle (registry, versioning, promotion), serving infrastructure (endpoints, autoscaling, canary), and developer experience (CLI, SDK, templates, docs).

Anti-pattern: Do not let the platform team become a ticket queue. If data scientists file tickets to deploy models, the platform has failed its core mission. Self-service is a requirement, not a nice-to-have.

API Design Patterns

The platform API is the contract between data scientists and infrastructure. A well-designed API hides cluster-level complexity behind domain-specific primitives: TrainingJob, Model, Endpoint, Experiment. Aim for an API that reads like a sentence: "Create a training job using this config, register the output model, then deploy it to a canary endpoint."

# Python SDK — high-level platform API from mlplatform import Platform platform = Platform.connect() # Submit a training job job = platform.training.submit( name="fraud-detector-v3", image="ml-images/pytorch:2.2", script="train.py", args={"epochs": 50, "lr": 0.001}, resources={"gpu": 2, "memory": "32Gi"}, dataset="s3://features/fraud/v7", ) # Wait and register model job.wait() model = platform.models.register( name="fraud-detector", version="3.0.0", artifact=job.output_path, metrics=job.metrics, ) # Deploy to canary endpoint endpoint = platform.serving.deploy( model=model, endpoint="fraud-detector-prod", strategy="canary", canary_percent=10, )

Underneath, the SDK translates these calls into Kubernetes CRDs, Argo Workflows, or cloud-native APIs — but the data scientist never sees those details. The API should also expose a REST/gRPC interface for CI/CD integration and non-Python consumers.

# REST API — equivalent of the SDK calls above # POST /api/v1/training-jobs { "name": "fraud-detector-v3", "image": "ml-images/pytorch:2.2", "script": "train.py", "args": {"epochs": 50, "lr": 0.001}, "resources": {"gpu": 2, "memory": "32Gi"}, "dataset": "s3://features/fraud/v7" } # Response — 201 Created { "id": "tj-20240615-a3f8", "status": "PENDING", "dashboard_url": "https://platform.internal/jobs/tj-20240615-a3f8" }

Abstraction Layers: How Deep to Go

The hardest design decision in platform engineering is choosing the right level of abstraction. Too thin and users still wrestle with YAML; too thick and power users feel trapped. The proven approach is a layered API: a high-level "easy mode" that covers 80% of use cases, with escape hatches down to lower layers for customization.

Level 1 — Templates

Pre-built configurations for common tasks. Data scientists fill in a few parameters (dataset, model type, hyperparams) and the platform handles everything else.

Example: mlp train --template xgboost-tabular

Level 2 — SDK / API

Programmatic access to platform primitives. Users compose training, registration, and serving steps in Python, with sensible defaults and validation.

Example: The Python SDK shown above.

Level 3 — CRD / Config

Direct Kubernetes CRD or infrastructure-as-code definitions. For platform engineers and advanced ML engineers who need full control over scheduling, networking, or GPUs.

Example: Custom TrainingJob CRD manifest.

Level 4 — Bring Your Own

Users supply their own Docker images and scripts. The platform only provides compute, storage, and observability. Maximum flexibility, minimum guardrails.

Example: Raw K8s Job with platform annotations.

Design principle: Every layer should be independently useful. A data scientist should never be forced to understand Level 3 to use Level 1. But if they need to, the path from Level 1 to Level 3 should be a smooth gradient, not a cliff.

Developer Experience (DevEx)

The quality of the developer experience determines whether data scientists adopt the platform willingly or circumvent it with shadow infrastructure. Focus on three pillars: a powerful CLI, excellent documentation, and fast feedback loops.

# CLI — intuitive commands with progressive disclosure # Quick start: train a model in one command $ mlp train --template xgboost-tabular \ --dataset s3://features/churn/v2 \ --target churned \ --name churn-model-q3 # Check status $ mlp jobs status churn-model-q3 STATUS: RUNNING | GPU: 1x A100 | ETA: 12 min LOGS: https://platform.internal/jobs/tj-2024-churn/logs # Deploy when ready $ mlp deploy churn-model-q3 --endpoint churn-prod --canary 10 Canary deployment started. 10% traffic → v3, 90% → v2 Dashboard: https://platform.internal/endpoints/churn-prod

Documentation should follow the Diátaxis framework: tutorials for onboarding, how-to guides for specific tasks, reference docs for the API, and explanations for architecture decisions. Embed runnable examples wherever possible.

Pitfall: Do not gate platform access behind a training course or approval process. If a new data scientist cannot go from git clone to a running training job in under 30 minutes, your onboarding is too slow.

Measuring Platform Success

Platform teams need metrics that reflect user productivity, not just uptime. The most effective KPIs measure time-to-value — how quickly a data scientist goes from idea to production model.

  • Time to first training job — minutes from onboarding to submitting a job. Target: < 30 min.
  • Deployment lead time — hours from "model ready" to "serving production traffic." Target: < 1 hour.
  • Self-service ratio — percentage of jobs/deployments completed without platform team intervention. Target: > 95%.
  • Platform adoption — fraction of ML models in production that run on the platform vs. shadow infra. Target: > 90%.
  • DORA-style metrics — deployment frequency, change failure rate, MTTR for model endpoints.
# Prometheus metrics exposed by the platform API ml_platform_training_jobs_total{team="fraud", status="succeeded"} 1247 ml_platform_deploy_lead_time_seconds{p50="1800", p99="7200"} ml_platform_self_service_ratio{team="fraud"} 0.97 ml_platform_onboarding_time_seconds{p50="900", p99="2400"} ml_platform_active_endpoints{env="production"} 42
Tip: Run a quarterly Developer Experience Survey alongside quantitative metrics. NPS scores and free-text feedback reveal friction points that telemetry alone cannot capture.
