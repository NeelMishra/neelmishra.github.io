Kubeflow: ML Toolkit for Kubernetes

MLOps Series Platform Engineering

What is Kubeflow?

Kubeflow is an open-source ML platform built on top of Kubernetes. Its goal is to make deploying ML workflows on K8s simple, portable, and scalable. Rather than reinventing infrastructure, Kubeflow leverages K8s primitives—CRDs, operators, and services—to orchestrate every stage of the ML lifecycle.

Kubeflow was originally developed at Google to run TensorFlow on K8s, but it has since expanded to support PyTorch, MXNet, XGBoost, and general-purpose Python workloads.

Key idea: Kubeflow is not a single application. It is a collection of loosely coupled components, each handling a different stage—experimentation, training, hyperparameter tuning, serving, and monitoring.

Core Components

Understanding the component map is essential before diving in. Each component runs as a separate K8s deployment with its own CRDs.

Development

  • Notebook Servers — JupyterLab instances on K8s
  • Volumes — PVC-backed workspace storage

Training

  • Training Operators — TFJob, PyTorchJob, MPIJob
  • Katib — Automated hyperparameter tuning

Orchestration

  • Pipelines — DAG-based workflow engine
  • Metadata — Artifact & lineage tracking

Serving

  • KServe — Serverless model inference
  • Seldon Core — Advanced deployment graphs
Warning: Kubeflow's full installation is resource-heavy (Istio, Dex, multiple controllers). For smaller teams, consider installing only the components you need using individual manifests instead of the full distribution.

Notebook Servers

Kubeflow Notebook Servers let data scientists spin up JupyterLab, VS Code, or RStudio instances directly on the K8s cluster. Each server runs as a StatefulSet with a dedicated PVC for persistent storage.

Creating a Notebook via YAML

apiVersion: kubeflow.org/v1 kind: Notebook metadata: name: ml-experiment-nb namespace: team-ml spec: template: spec: containers: - name: notebook image: kubeflownotebookswg/jupyter-pytorch-cuda:v1.8 resources: requests: cpu: "2" memory: "8Gi" nvidia.com/gpu: "1" limits: cpu: "4" memory: "16Gi" nvidia.com/gpu: "1" volumeMounts: - name: workspace mountPath: /home/jovyan volumes: - name: workspace persistentVolumeClaim: claimName: ml-experiment-pvc

Notebooks are ideal for interactive experimentation. Once an experiment matures, you extract the training logic into a pipeline component or a standalone Job.

Kubeflow Pipelines

Kubeflow Pipelines (KFP) is the orchestration engine at the heart of Kubeflow. It lets you define ML workflows as directed acyclic graphs (DAGs) of containerized steps, with built-in artifact tracking, caching, and a visual UI.

Defining a Pipeline in Python

from kfp import dsl from kfp import compiler # Each function becomes a container step @dsl.component(base_image="python:3.10") def preprocess_data(input_path: str, output_path: dsl.Output[dsl.Dataset]): import pandas as pd df = pd.read_csv(input_path) df = df.dropna().reset_index(drop=True) df.to_parquet(output_path.path) @dsl.component(base_image="pytorch/pytorch:2.1.0-cuda12.1-cudnn8-runtime") def train_model(dataset: dsl.Input[dsl.Dataset], epochs: int, model_out: dsl.Output[dsl.Model]): import torch # Training logic here... torch.save(model.state_dict(), model_out.path) @dsl.component(base_image="python:3.10") def evaluate_model(model: dsl.Input[dsl.Model], test_data: dsl.Input[dsl.Dataset]) -> float: # Return accuracy metric return 0.94 # Wire components into a DAG @dsl.pipeline(name="bert-training-pipeline") def ml_pipeline(input_path: str, epochs: int = 10): preprocess_task = preprocess_data(input_path=input_path) train_task = train_model( dataset=preprocess_task.outputs["output_path"], epochs=epochs ) evaluate_model( model=train_task.outputs["model_out"], test_data=preprocess_task.outputs["output_path"] ) # Compile to YAML for submission compiler.Compiler().compile(ml_pipeline, "pipeline.yaml")
Tip: KFP v2 (the current SDK) compiles pipelines to Argo Workflow YAML under the hood. Each @dsl.component runs as an isolated container, making the pipeline fully reproducible.

Training Operators

For distributed training, Kubeflow provides dedicated K8s operators that manage the lifecycle of multi-worker training jobs. Each framework gets its own CRD.

PyTorchJob Example

apiVersion: kubeflow.org/v1 kind: PyTorchJob metadata: name: distributed-bert-training namespace: team-ml spec: pytorchReplicaSpecs: Master: replicas: 1 restartPolicy: OnFailure template: spec: containers: - name: pytorch image: myregistry/bert-trainer:v3 command: ["torchrun"] args: ["--nproc_per_node=2", "train.py"] resources: limits: nvidia.com/gpu: "2" Worker: replicas: 3 restartPolicy: OnFailure template: spec: containers: - name: pytorch image: myregistry/bert-trainer:v3 command: ["torchrun"] args: ["--nproc_per_node=2", "train.py"] resources: limits: nvidia.com/gpu: "2"

The operator automatically sets the MASTER_ADDR, MASTER_PORT, WORLD_SIZE, and RANK environment variables. Workers discover the master via a headless K8s Service created by the operator.

TFJob

Manages TensorFlow distributed strategies. Supports Parameter Server and multi-worker mirrored strategies.

MPIJob

Uses Horovod / OpenMPI for all-reduce training. Ideal for large-scale data-parallel workloads.

Model Serving with KServe

KServe (formerly KFServing) provides serverless inference on K8s. It supports autoscaling to zero, canary rollouts, and multi-model serving.

InferenceService YAML

apiVersion: serving.kserve.io/v1beta1 kind: InferenceService metadata: name: bert-classifier namespace: serving spec: predictor: minReplicas: 1 maxReplicas: 10 pytorch: storageUri: "s3://models/bert-v3" protocolVersion: v2 resources: requests: cpu: "2" memory: "8Gi" nvidia.com/gpu: "1" limits: nvidia.com/gpu: "1"
Scale to zero: KServe uses Knative under the hood. When no requests arrive for a configurable window, replicas scale to zero—saving GPU costs. Cold-start latency is the trade-off.

Canary Deployments

KServe supports traffic splitting between model versions. This lets you roll out a new model to 10% of traffic, validate metrics, then gradually shift to 100%.

spec: predictor: canaryTrafficPercent: 10 pytorch: storageUri: "s3://models/bert-v4" # new version resources: limits: nvidia.com/gpu: "1"
Caution: Scale-to-zero with GPU-backed models can incur 30–60 second cold starts due to CUDA initialization and model loading. For latency-sensitive services, set minReplicas: 1.
