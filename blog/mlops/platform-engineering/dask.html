Dask: Parallel Computing for Data Science

MLOps Series Platform Engineering

Dask Overview

Dask is a flexible parallel computing library for Python that scales familiar interfaces — NumPy, pandas, and scikit-learn — to multi-core machines and distributed clusters. Its core philosophy is simple: use the APIs you already know, but let Dask handle parallelism and out-of-core computation behind the scenes.

Key Insight: Dask uses lazy evaluation. When you write df.groupby("col").mean(), Dask builds a task graph instead of executing immediately. Computation happens only when you call .compute(). This allows Dask to optimize execution by fusing operations and minimizing memory usage.

Dask operates at two levels: high-level collections (Arrays, DataFrames, Bags) that mirror familiar APIs, and a low-level task graph framework (Delayed, Futures) for custom parallel workflows.

Dask schedulers come in three flavors:

  • Synchronous — single-threaded, useful for debugging
  • Threaded / Multiprocessing — local parallelism on a single machine
  • Distributed — multi-machine cluster via dask.distributed

Dask Arrays

Dask Array implements a subset of the NumPy API on large, chunked, multi-dimensional arrays. It automatically splits a large array into many small NumPy arrays (chunks) and orchestrates operations across them in parallel.

Creating & Operating on Dask Arrays

import dask.array as da import numpy as np # Create a 100GB array split into 100MB chunks x = da.random.random( size=(100_000, 100_000), chunks=(1000, 1000) ) # Familiar NumPy operations — all lazy y = x + x.T z = da.exp(y).mean(axis=0) # Nothing computed yet — just a task graph print(z) # dask.array<mean, shape=(100000,), chunks=(1000,)> # Trigger computation result = z.compute() # returns a NumPy array

Chunking Strategy

Chunk size is the most important performance knob for Dask Arrays. Chunks that are too small create excessive scheduler overhead; chunks that are too large defeat the purpose of parallelism and can cause out-of-memory errors.

Rule of Thumb: Target chunk sizes of 100 MB – 1 GB for best performance. Use da.rechunk() to adjust chunks between operations if the access pattern changes (e.g., switching from row-wise to column-wise).
# Rechunking for different access patterns arr = da.from_zarr("dataset.zarr") print(arr.chunks) # ((1000, 1000, ...), (50000,)) # Switch to column-oriented chunks for column operations arr_col = arr.rechunk({0: -1, 1: 1000}) # Persist frequently-used arrays in cluster memory arr_col = arr_col.persist() # triggers computation, keeps in memory

Interop with NumPy & SciPy

Dask Array works seamlessly with libraries that accept array-like objects via the __array_ufunc__ and __array_function__ protocols. Most NumPy universal functions work on Dask Arrays out of the box.

Dask DataFrames

Dask DataFrame parallelizes pandas by partitioning a DataFrame across rows. Each partition is a regular pandas DataFrame, so the full pandas API is available per-partition.

Reading & Querying Large Datasets

import dask.dataframe as dd # Read a 500GB Parquet dataset df = dd.read_parquet( "s3://data-lake/events/", columns=["user_id", "event_type", "timestamp", "value"], filters=[("timestamp", ">", "2024-01-01")] ) # Pandas-like API — lazy evaluation result = ( df[df["event_type"] == "purchase"] .groupby("user_id") .agg({"value": ["sum", "count", "mean"]}) .reset_index() ) # Execute and collect to pandas (if result fits in memory) pandas_df = result.compute()
Partition Count: Aim for 100–1000 partitions for cluster workloads. Too few means underutilization; too many means scheduler overhead. Use df.repartition(npartitions=200) to adjust.

Shuffles & Joins

Operations like groupby, merge, and set_index may trigger a shuffle — redistributing data across partitions. Shuffles are the most expensive operation in Dask DataFrames.

# Efficient join: co-partition on join key first users = dd.read_parquet("s3://data/users/") orders = dd.read_parquet("s3://data/orders/") # set_index triggers a shuffle (expensive but one-time) users = users.set_index("user_id") orders = orders.set_index("user_id") # Now the join is partition-wise (fast, no shuffle) merged = users.join(orders, how="inner")

Writing Results

# Write back to Parquet with partitioning result.to_parquet( "s3://output/user-purchases/", partition_on=["region"], compression="snappy", write_index=False )

Delayed Computation

When Dask's high-level collections don't fit your workflow, dask.delayed gives you fine-grained control. It wraps arbitrary Python functions into lazy tasks that Dask schedules as a graph.

Building Custom Pipelines

from dask import delayed @delayed def load_file(path): return pd.read_csv(path) @delayed def clean(df): df = df.dropna() df["amount"] = df["amount"].clip(lower=0) return df @delayed def featurize(df): df["log_amount"] = np.log1p(df["amount"]) df["day_of_week"] = df["timestamp"].dt.dayofweek return df @delayed def combine(*dfs): return pd.concat(dfs, ignore_index=True) # Build the task graph files = glob.glob("data/*.csv") cleaned = [featurize(clean(load_file(f))) for f in files] final = combine(*cleaned) # Visualize the task graph (requires graphviz) final.visualize(filename="pipeline.png") # Execute the entire pipeline result = final.compute()
Graph Optimization: Dask automatically fuses adjacent tasks that don't require a shuffle. For example, load_file → clean → featurize will be fused into a single task per file, reducing scheduler overhead.

Delayed with Loops & Conditionals

# Tree reduction pattern for aggregation @delayed def partial_fit(data_chunk, model_params): model = create_model(model_params) model.fit(data_chunk) return model.get_weights() @delayed def merge_weights(w1, w2): return (w1 + w2) / 2 # Fan-out: train on each shard weights = [partial_fit(chunk, params) for chunk in data_shards] # Fan-in: tree reduction to merge while len(weights) > 1: new_weights = [] for i in range(0, len(weights), 2): if i + 1 < len(weights): new_weights.append(merge_weights(weights[i], weights[i+1])) else: new_weights.append(weights[i]) weights = new_weights final_weights = weights[0].compute()

Distributed Scheduler

The dask.distributed scheduler runs Dask on a cluster of machines. It provides a central scheduler process that coordinates work across multiple worker processes, with features like work stealing, memory-aware scheduling, and real-time diagnostics.

Setting Up a Distributed Cluster

from dask.distributed import Client, LocalCluster # Option 1: Local cluster (development) cluster = LocalCluster( n_workers=8, threads_per_worker=2, memory_limit="4GB" ) client = Client(cluster) # Option 2: Connect to existing cluster client = Client("tcp://scheduler-address:8786") print(client) # <Client: 'tcp://127.0.0.1:8786' processes=8 threads=16 memory=32.00 GiB>
Memory Spilling: When workers exceed 60% memory usage, Dask begins spilling data to disk. At 80%, it pauses computation. At 95%, it terminates the worker. Tune memory_limit and monitor the dashboard closely for memory-intensive workloads.

Futures API

For real-time workloads, use the Futures API instead of lazy collections. Futures submit tasks immediately and return handles that resolve asynchronously:

from dask.distributed import Client, as_completed client = Client() # Submit tasks eagerly def process_event(event): return enrich(validate(event)) futures = [client.submit(process_event, e) for e in event_stream] # Process results as they arrive for future in as_completed(futures): result = future.result() write_to_database(result)

Adaptive Scaling

# Dask can auto-scale workers based on load from dask_kubernetes import KubeCluster cluster = KubeCluster.from_yaml("worker-spec.yaml") cluster.adapt(minimum=2, maximum=50) client = Client(cluster) # Workers auto-scale: ramp up under load, cool down when idle

ML Integration

Dask integrates directly with the PyData ecosystem, providing parallel alternatives for common ML tasks like feature engineering, model training, and hyperparameter search.

Dask-ML: Parallel Scikit-Learn

from dask_ml.model_selection import GridSearchCV from dask_ml.preprocessing import StandardScaler from sklearn.ensemble import GradientBoostingClassifier import dask.dataframe as dd # Load and preprocess with Dask df = dd.read_parquet("s3://features/train/") X = df.drop("target", axis=1) y = df["target"] # Dask-ML scaler (works on Dask DataFrames) scaler = StandardScaler() X_scaled = scaler.fit_transform(X) # Parallel hyperparameter search across Dask workers param_grid = { "n_estimators": [100, 200, 500], "max_depth": [3, 5, 7, 10], "learning_rate": [0.01, 0.05, 0.1] } search = GridSearchCV( GradientBoostingClassifier(), param_grid, cv=5, n_jobs=-1 # uses all Dask workers ) search.fit(X_scaled, y) print(search.best_params_)

Dask-ML Strengths

  • Drop-in replacement for scikit-learn cross-validation
  • Parallel preprocessing on DataFrames
  • Incremental learning for out-of-core datasets
  • Native integration with Dask collections

Limitations

  • Not all scikit-learn estimators are parallelizable
  • Deep learning support requires separate integrations
  • GPU support is via RAPIDS cuML, not built-in
  • Complex pipelines may need manual orchestration

XGBoost with Dask

import xgboost as xgb import dask.dataframe as dd from dask.distributed import Client client = Client() # Load data as Dask DataFrame df = dd.read_parquet("s3://data/training/") X = df.drop("label", axis=1) y = df["label"] # Create DaskDMatrix — distributed data structure dtrain = xgb.dask.DaskDMatrix(client, X, y) # Train XGBoost distributed across Dask workers output = xgb.dask.train( client, {"objective": "binary:logistic", "max_depth": 6, "eta": 0.1}, dtrain, num_boost_round=500, evals=[(dtrain, "train")] ) # output['booster'] is a standard XGBoost model booster = output["booster"] booster.save_model("model.json")
RAPIDS Integration: For GPU-accelerated DataFrames and ML, use Dask with RAPIDS cuDF and cuML. Replace dask.dataframe with dask_cudf for near-seamless GPU acceleration on NVIDIA hardware.
