Storage for ML: Object Stores, NFS, HPC

MLOps Series Platform Engineering

Storage Overview

Storage is the most underestimated component of ML infrastructure. GPU utilisation drops to zero when the data pipeline stalls, and a single misconfigured storage layer can bottleneck an entire training cluster. ML workloads have unique storage requirements: massive sequential reads for training data, random reads for data augmentation, high-throughput writes for checkpoints, and low-latency reads for feature serving.

The golden rule: Store data durably in object storage, stage it to fast storage (NFS, Lustre, or local NVMe) before training, and stream checkpoints back to object storage. This tiered approach balances cost, durability, and performance.

Object Stores

Object stores (S3, GCS, Azure Blob) are the default durable storage layer for ML. They are infinitely scalable, highly available (99.99% SLA), and extremely cost-effective at $0.02–$0.03/GB/month. However, they have higher latency than POSIX filesystems and do not support random writes or directory listings efficiently.

AWS S3

  • Throughput: ~100 Gbps per prefix with S3 Express One Zone.
  • Tiers: Standard, Intelligent-Tiering, Glacier for archival.
  • ML integration: SageMaker Pipe Mode streams data directly from S3.
  • Cost: $0.023/GB/month (Standard). $0.004/GB/month (Glacier).

GCS

  • Throughput: ~200 Gbps with multi-region. Best-in-class for large reads.
  • Tiers: Standard, Nearline, Coldline, Archive.
  • ML integration: Vertex AI and TFRecords read natively from GCS.
  • Cost: $0.020/GB/month (Standard). Free egress within same region.
# Python: Efficient S3 data loading with multipart download import boto3 from concurrent.futures import ThreadPoolExecutor import io s3 = boto3.client("s3") def download_shard(bucket, key): """Download a single training data shard.""" response = s3.get_object(Bucket=bucket, Key=key) return response["Body"].read() def load_training_data(bucket, prefix, max_workers=32): """Parallel download of sharded training data.""" paginator = s3.get_paginator("list_objects_v2") keys = [] for page in paginator.paginate(Bucket=bucket, Prefix=prefix): keys.extend([obj["Key"] for obj in page.get("Contents", [])]) with ThreadPoolExecutor(max_workers=max_workers) as pool: futures = [pool.submit(download_shard, bucket, k) for k in keys] return [f.result() for f in futures]

Network Filesystems

NFS-based solutions (AWS EFS, GCP Filestore, Azure NetApp Files) provide POSIX-compatible shared filesystems that can be mounted by multiple nodes simultaneously. They are simpler to use than object stores for training because data loaders can read files using standard open() calls without SDK-specific code.

AWS EFS

Throughput: Up to 10 GB/s (elastic throughput mode).

Latency: ~0.5–1ms for metadata, ~2ms for data reads.

Cost: $0.30/GB/month (Standard). $0.025/GB/month (Infrequent Access).

ML use: Good for shared model artefacts, configs, and small datasets (<10TB).

GCP Filestore

Throughput: Up to 16 GB/s (Enterprise tier).

Latency: ~0.2ms for data reads (SSD-backed).

Cost: $0.20/GB/month (Basic HDD). $0.36/GB/month (Enterprise).

ML use: Excellent for checkpointing and shared code repos across a cluster.

Throughput trap: NFS throughput scales with provisioned capacity, not usage. An EFS filesystem with 100 GB of data gets only ~5 MB/s in bursting mode. For ML training data, you often need to provision 10× more capacity than your data size to get adequate throughput, or use provisioned throughput mode which adds significant cost.

HPC Storage

For the highest throughput demands—training large language models, processing petabyte-scale datasets—HPC parallel filesystems like Lustre, GPFS (IBM Spectrum Scale), and BeeGFS are the answer. These systems stripe data across hundreds of storage servers and deliver aggregate throughput exceeding 1 TB/s.

Amazon FSx for Lustre

  • Throughput: Up to 1 TB/s per filesystem.
  • Latency: Sub-millisecond with SSD-backed persistent storage.
  • S3 integration: Lazy-loads data from S3 on first read. Write-back on close.
  • Cost: $0.14/GB/month (persistent SSD). Scratch: $0.06/GB/month.
  • ML use: Ideal for large-scale distributed training with many data workers.

GPFS / IBM Spectrum Scale

  • Throughput: Multi-TB/s at scale. Proven at >100PB deployments.
  • Features: Snapshots, tiering, encryption, multi-site replication.
  • POSIX: Full POSIX compliance with parallel metadata.
  • Cost: License-based. Typically $0.05–$0.10/GB/month all-in.
  • ML use: On-prem supercomputer clusters. Used by most national labs.
# FSx for Lustre: create linked to S3 bucket (AWS CLI) $ aws fsx create-file-system \ --file-system-type LUSTRE \ --storage-capacity 14400 \ --storage-type SSD \ --lustre-configuration '{ "DeploymentType": "PERSISTENT_2", "PerUnitStorageThroughput": 1000, "DataCompressionType": "LZ4", "ImportPath": "s3://training-data", "ExportPath": "s3://training-data/output" }' \ --subnet-ids subnet-abc123 # Mount on compute nodes $ sudo mount -t lustre \ fs-0123456789.fsx.us-east-1.amazonaws.com@tcp:/fsx \ /mnt/lustre # Verify throughput with IOR benchmark $ mpirun -np 64 ior -t 1m -b 16g -s 1 -F -o /mnt/lustre/test
Lazy loading advantage: FSx for Lustre’s S3 integration means you do not need to copy data before training. The first epoch reads from S3 through Lustre (slower), but data is cached locally for subsequent epochs (fast). This eliminates the staging step that many teams spend hours on.

POSIX vs Object

The fundamental architectural choice in ML storage is POSIX filesystems versus object stores. Each model has deep implications for data loading patterns, training code, and operational complexity.

POSIX (NFS / Lustre / Local)

  • API: Standard open(), read(), seek().
  • Random access: Native. Seek to any byte offset.
  • Directory listing: Fast ls, glob, find.
  • Consistency: Strong. Read-after-write guaranteed.
  • Scalability: Limited by filesystem metadata servers.
  • Durability: Single-site unless replicated.

Object Store (S3 / GCS)

  • API: HTTP-based GET, PUT, LIST. SDK required.
  • Random access: Via Range header. Higher latency per request.
  • Directory listing: Slow prefix scan. No true directories.
  • Consistency: Strong read-after-write (S3 since 2020, GCS always).
  • Scalability: Infinite. No provisioning needed.
  • Durability: 11 nines (99.999999999%).

Pragmatic approach: Use object stores as the source of truth for all datasets and model artefacts. Use POSIX-compatible adapters (S3-FUSE, GCSFuse, Mountpoint for S3) to bridge the gap for legacy code that expects file paths. For new code, use streaming data loaders that read directly from object stores.

Throughput Optimization

Even with the right storage tier, poor configuration can leave 80% of throughput on the table. Here are the most impactful optimizations for ML training data pipelines.

1. Shard Your Data

Store training data as many medium-sized files (100 MB–1 GB each) rather than millions of small files or one giant file. This allows parallel reads across multiple workers without metadata bottlenecks.

# Python: create sharded WebDataset for efficient training import webdataset as wds # Write shards: each tar file is ~1GB with wds.ShardWriter("s3://bucket/shards/shard-%05d.tar", maxcount=10000) as sink: for sample in dataset: sink.write({ "__key__": sample["id"], "image.jpg": sample["image_bytes"], "label.txt": str(sample["label"]), }) # Read with parallel workers in PyTorch DataLoader dataset = ( wds.WebDataset("s3://bucket/shards/shard-{00000..00099}.tar") .shuffle(1000) .decode("pil") .to_tuple("image.jpg", "label.txt") ) loader = torch.utils.data.DataLoader(dataset, num_workers=8, batch_size=64)

2. Use DataLoader Prefetching

Always set num_workers > 0 in your DataLoader to overlap data loading with GPU computation. A good starting point is num_workers = 4 × num_gpus. Use prefetch_factor=2 to keep two batches ready in memory at all times.

3. Stage to Local NVMe

For iterative training (multiple epochs over the same data), copy data to instance-local NVMe storage before training starts. This eliminates network latency for all epochs after the first.

# Stage data from S3 to local NVMe before training import subprocess, os local_dir = "/mnt/nvme/training-data" s3_path = "s3://my-bucket/datasets/imagenet/" # Fast parallel download with s5cmd (10-50x faster than aws s3 cp) subprocess.run([ "s5cmd", "--numworkers", "256", "cp", f"{s3_path}*", local_dir ], check=True) # Verify data integrity local_files = os.listdir(local_dir) print(f"Staged {len(local_files)} files to local NVMe")
Staging cost: NVMe staging only makes sense for multi-epoch training. If you train for 1 epoch (common for LLM pre-training), the staging time is wasted. In that case, use a streaming data loader (WebDataset, Mosaic StreamingDataset) that reads directly from object storage with aggressive prefetching.

4. Checkpoint Optimization

Large model checkpoints (10–100 GB) are a major storage bottleneck. Use asynchronous checkpointing to overlap the write with training, and write to parallel filesystems (Lustre, GPFS) or use multipart uploads to object stores with high concurrency.

# PyTorch distributed checkpoint with async writes import torch.distributed.checkpoint as dcp from torch.distributed.checkpoint import FileSystemWriter # Async checkpoint: training continues while write happens writer = FileSystemWriter("/mnt/lustre/checkpoints/step_1000") # Non-blocking save — returns a Future future = dcp.async_save( state_dict={"model": model.state_dict(), "optim": optimizer.state_dict()}, storage_writer=writer ) # Continue training while checkpoint writes in background for batch in dataloader: loss = train_step(model, batch) # ... # Ensure checkpoint is complete before next save future.result()
Rule of thumb for throughput: Each GPU should have at least 1 GB/s of storage throughput available. A 64-GPU cluster needs ~64 GB/s. NFS cannot deliver this—use Lustre, local NVMe staging, or highly parallelised object store reads. Always benchmark your data pipeline before starting a multi-day training run.
