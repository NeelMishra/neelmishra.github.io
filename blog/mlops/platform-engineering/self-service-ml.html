Self-Service ML: Enabling Data Scientists

MLOps Series Platform Engineering

Why Self-Service Matters

The biggest bottleneck in production ML is not model accuracy — it is organizational friction. When a data scientist needs to file a ticket, wait for a DevOps engineer, and iterate through multiple review cycles just to train a model on a GPU, innovation stalls. Self-service ML platforms eliminate that bottleneck by giving data scientists direct access to compute, data, and deployment pipelines through safe, governed interfaces.

A successful self-service platform balances two forces: autonomy (data scientists move fast without gatekeepers) and control (the organization enforces security, cost, and compliance policies automatically). The following sections describe the building blocks that make this balance possible.

North star metric: A new data scientist should be able to go from git clone to a trained model in under 30 minutes — without asking anyone for help.

Templates & Project Catalogs

Templates are the fastest path to self-service. A well-designed template encodes best practices — logging, metrics, reproducibility, resource defaults — so that data scientists start with a production-ready scaffold instead of a blank notebook.

# template-catalog.yaml — registry of available templates templates: - name: xgboost-tabular description: "XGBoost classifier for tabular data with feature store integration" framework: xgboost default_resources: cpu: 4 memory: "16Gi" gpu: 0 parameters: - name: dataset type: s3_path required: true - name: target_column type: string required: true - name: n_estimators type: int default: 200 - name: max_depth type: int default: 6 - name: pytorch-vision description: "PyTorch image classification with distributed training" framework: pytorch default_resources: cpu: 8 memory: "64Gi" gpu: 4 parameters: - name: dataset type: s3_path required: true - name: model_arch type: string default: "resnet50" - name: epochs type: int default: 30
# CLI — scaffold a new project from a template $ mlp init --template xgboost-tabular --name churn-prediction ✓ Created project: churn-prediction/ ├── train.py # training script with logging & metrics ├── config.yaml # hyperparameters & resource defaults ├── requirements.txt # pinned dependencies ├── Dockerfile # optimized base image └── .mlp/ └── pipeline.yaml # CI/CD pipeline definition

Maintain templates in a dedicated Git repository with semantic versioning. Data scientists pin to a template version, and the platform team can ship improvements without breaking existing projects.

Anti-pattern: Do not create one mega-template that tries to handle every framework. Maintain small, focused templates (5-10 parameters max) that cover a single use case well.

Abstraction Design

Abstractions determine how much infrastructure detail leaks through to the data scientist. The goal is to hide accidental complexity (YAML indentation, K8s resource limits, Docker layer caching) while preserving essential complexity (hyperparameters, data splits, evaluation metrics).

Too Thin

  • Users write raw K8s manifests
  • No validation or defaults
  • High cognitive load
  • Every team reinvents the wheel

Too Thick

  • Opaque magic — users can't debug failures
  • Every edge case requires platform changes
  • Power users work around the platform
  • High maintenance burden for the team
# Python — a well-balanced abstraction for training config from dataclasses import dataclass, field from typing import Optional, Dict @dataclass class TrainingConfig: # Essential: data scientists must think about these name: str dataset: str target_column: str framework: str # "xgboost", "pytorch", "sklearn" hyperparameters: Dict # Governed defaults: safe to ignore, easy to override gpu_count: int = 0 memory: str = "16Gi" max_runtime_hours: int = 24 experiment_tracking: bool = True auto_register_model: bool = True # Escape hatch: raw overrides for advanced users custom_env: Dict = field(default_factory=dict) extra_volumes: list = field(default_factory=list) annotations: Dict = field(default_factory=dict)
Design rule: Every parameter should have a sensible default. If a data scientist creates a TrainingConfig with only name, dataset, target_column, and framework, the job should still succeed.

Guardrails & Governance

Guardrails are the safety net that makes self-service possible. Without them, self-service degrades into a free-for-all where teams over-provision GPUs, deploy unvalidated models, or accidentally expose PII in logs. Effective guardrails are automated, fast, and educational — they block bad actions and explain why.

# Python — policy engine for training job submission class TrainingPolicy: MAX_GPU_PER_JOB = 8 MAX_RUNTIME_HOURS = 72 ALLOWED_FRAMEWORKS = ["pytorch", "xgboost", "sklearn", "tensorflow"] REQUIRED_LABELS = ["team", "project", "cost-center"] def validate(self, config: TrainingConfig) -> list: violations = [] if config.gpu_count > self.MAX_GPU_PER_JOB: violations.append( f"GPU request {config.gpu_count} exceeds limit of {self.MAX_GPU_PER_JOB}. " f"File a capacity request for larger jobs." ) if config.framework not in self.ALLOWED_FRAMEWORKS: violations.append( f"Framework '{config.framework}' is not approved. " f"Approved: {self.ALLOWED_FRAMEWORKS}" ) for label in self.REQUIRED_LABELS: if label not in config.annotations: violations.append(f"Missing required label: '{label}'") return violations
Warning: Cost guardrails are non-negotiable. Without GPU budget limits, a single misconfigured hyperparameter sweep can consume your entire monthly cloud budget in hours.

For model deployment, add a second layer of guardrails: model validation gates. Before a model reaches production, automatically verify minimum accuracy thresholds, data drift checks, bias audits, and latency benchmarks.

# deployment-policy.yaml — automated promotion gates promotion_gates: - name: accuracy-check type: metric_threshold metric: test_auc operator: ">=" threshold: 0.85 message: "Model AUC must be >= 0.85 for production deployment" - name: latency-check type: benchmark p99_latency_ms: 100 message: "P99 inference latency must be under 100ms" - name: bias-audit type: fairness protected_attributes: ["gender", "age_group"] max_disparity: 0.1 message: "Demographic parity gap must be < 10%" - name: data-drift-check type: drift reference_dataset: s3://features/baseline/v1 max_psi: 0.2 message: "Population Stability Index must be < 0.2"

Training UX

The training experience should feel as simple as running a local script, but with cloud-scale resources behind it. The ideal flow: edit code locally, push to a branch, and the platform handles containerization, scheduling, GPU allocation, experiment tracking, and artifact storage.

# End-to-end training workflow from the CLI # 1. Initialize project from template $ mlp init --template pytorch-vision --name defect-detector $ cd defect-detector # 2. Edit train.py with your model logic (local IDE) # 3. Submit training job — platform builds, schedules, tracks $ mlp train --config config.yaml Building container... ✓ (cached, 4s) Scheduling on GPU cluster... ✓ (2x A100 allocated) Experiment: https://platform.internal/experiments/defect-v1 Logs: https://platform.internal/jobs/tj-defect-001/logs # 4. Compare experiments $ mlp experiments compare defect-v1 defect-v2 defect-v3 ┌─────────────┬────────┬────────┬────────┐ │ Metric │ v1 │ v2 │ v3 │ ├─────────────┼────────┼────────┼────────┤ │ val_acc │ 0.923 │ 0.941 │ 0.956 │ │ val_loss │ 0.187 │ 0.154 │ 0.121 │ │ train_time │ 47m │ 52m │ 68m │ │ gpu_hours │ 1.57 │ 1.73 │ 2.27 │ └─────────────┴────────┴────────┴────────┘
Key principle: Every training run must be reproducible. The platform should automatically capture the Git SHA, container image digest, dataset version, and full hyperparameter config alongside every experiment.

For interactive work, provide managed Jupyter environments that connect to the same compute backend. Data scientists prototype in notebooks, then promote to a training job with a single command — no Dockerfile surgery required.

Serving UX

Deployment should be a one-command operation, not a multi-day project. The platform should handle container packaging, endpoint provisioning, autoscaling configuration, canary rollouts, and monitoring setup — all triggered by a single mlp deploy call.

# Python SDK — deploy a model with canary rollout from mlplatform import Platform platform = Platform.connect() # One-line deployment endpoint = platform.serving.deploy( model="defect-detector:v3", endpoint="defect-detector-prod", strategy="canary", canary_percent=5, auto_rollback=True, rollback_threshold={"error_rate": 0.01, "p99_latency_ms": 150}, autoscale={"min_replicas": 2, "max_replicas": 20, "target_gpu_util": 0.7}, ) # Platform automatically: # → Packages model into optimized serving container # → Configures load balancer for canary split # → Sets up Prometheus metrics & Grafana dashboard # → Monitors error rate & latency for auto-rollback print(f"Endpoint: {endpoint.url}") print(f"Dashboard: {endpoint.dashboard_url}")

Online Serving

  • REST/gRPC endpoints with autoscaling
  • Canary & blue-green deployments
  • Sub-100ms latency SLO
  • Best for real-time predictions

Batch Inference

  • Scheduled Spark/Ray jobs
  • Results written to feature store or warehouse
  • Cost-optimized spot instances
  • Best for periodic scoring of large datasets
# CLI — monitor a live endpoint $ mlp endpoints status defect-detector-prod Endpoint: defect-detector-prod Model: defect-detector:v3 (canary 5%) / v2 (baseline 95%) Replicas: 4 / 20 (autoscaling) Requests: 12,450 rpm Latency: p50=18ms p95=42ms p99=67ms Errors: 0.02% Canary Health: v3 error_rate: 0.003 (✓ below 0.01 threshold) v3 p99_latency: 71ms (✓ below 150ms threshold) Auto-promote in: 2h 14m (if metrics hold)
Tip: Implement automatic rollback as a first-class feature. If the canary exceeds error rate or latency thresholds, the platform should revert to the previous version without human intervention. Fast, safe rollbacks are the foundation of deployment confidence.
Common mistake: Serving UX often neglects the post-deployment experience. Data scientists need easy access to prediction logs, feature attribution, and drift dashboards — not just a green "deployed" status.
