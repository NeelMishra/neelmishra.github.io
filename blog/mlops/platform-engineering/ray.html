Ray: Distributed Computing for ML

MLOps Series Platform Engineering

Ray Overview

Ray is a unified framework for scaling AI and Python applications. Originally developed at UC Berkeley's RISELab, it provides a simple yet powerful API for distributing computation across clusters of machines. Unlike traditional distributed systems that require significant boilerplate, Ray lets you go from single-machine prototypes to cluster-scale deployments with minimal code changes.

Why Ray? Ray was designed with ML workloads in mind from day one. Its object store, task scheduler, and actor model are purpose-built for the heterogeneous, stateful, and GPU-intensive nature of modern ML pipelines.

The Ray ecosystem consists of five core libraries that address distinct stages of the ML lifecycle:

  • Ray Core — low-level primitives: tasks (stateless) and actors (stateful)
  • Ray Train — distributed training across frameworks (PyTorch, TensorFlow, XGBoost)
  • Ray Serve — scalable model serving with composition and autoscaling
  • Ray Data — distributed data loading, preprocessing, and last-mile ingest
  • Ray Tune — hyperparameter tuning with scheduling and search algorithms

At its core, Ray runs a distributed scheduler that places tasks and actors on nodes with available resources. The shared-memory object store enables zero-copy reads between tasks on the same node, making data transfer extremely efficient for large tensors and DataFrames.

Ray Core: Tasks & Actors

Ray Core provides two fundamental primitives: tasks (stateless remote functions) and actors (stateful remote classes). Together, they let you express arbitrary distributed computation patterns.

Remote Tasks

A Ray task is a function decorated with @ray.remote. When invoked with .remote(), it runs asynchronously on the cluster and returns an ObjectRef — a future that can be resolved with ray.get().

import ray ray.init() # Define a remote task @ray.remote def preprocess(image_path): img = load_image(image_path) img = resize(img, (224, 224)) img = normalize(img, mean=[0.485, 0.456, 0.406]) return img # Launch 1000 tasks in parallel image_paths = get_image_paths("s3://bucket/dataset/") futures = [preprocess.remote(p) for p in image_paths] # Gather results (blocks until all complete) processed = ray.get(futures)
Resource Requests: Tasks can request specific resources like GPUs. Use @ray.remote(num_gpus=1) to schedule a task only on nodes with available GPUs. Ray's scheduler handles placement automatically.

Remote Actors

Actors maintain state across method calls. They are ideal for stateful services, parameter servers, or long-running workers that accumulate state.

import ray @ray.remote class ModelWorker: def __init__(self, model_name): import torch self.model = torch.hub.load("pytorch/vision", model_name) self.model.eval() self.request_count = 0 def predict(self, batch): self.request_count += 1 with torch.no_grad(): return self.model(batch) def get_stats(self): return {"requests": self.request_count} # Create 4 actor replicas across the cluster workers = [ModelWorker.remote("resnet50") for _ in range(4)] # Distribute requests round-robin futures = [] for i, batch in enumerate(batches): worker = workers[i % len(workers)] futures.append(worker.predict.remote(batch))

Task Dependencies & Pipelining

Ray tasks can accept ObjectRefs as inputs, forming a DAG of dependent computations that Ray schedules lazily:

# Task B depends on Task A's output — no explicit wait needed @ray.remote def fetch_data(url): return requests.get(url).json() @ray.remote def transform(raw_data): return [clean_record(r) for r in raw_data] @ray.remote def aggregate(records): return compute_stats(records) # Build a pipeline — Ray resolves refs automatically raw_ref = fetch_data.remote("https://api.example.com/data") clean_ref = transform.remote(raw_ref) result = ray.get(aggregate.remote(clean_ref))

Ray Train: Distributed Training

Ray Train abstracts away the complexity of distributed training. It supports data-parallel, model-parallel, and pipeline-parallel strategies across PyTorch, TensorFlow, HuggingFace, XGBoost, and LightGBM.

Common Pitfall: When using Ray Train with PyTorch DDP, do not call dist.init_process_group() manually. Ray Train sets up the process group for you. Duplicate initialization will cause hangs or crashes.

PyTorch Distributed Training

from ray.train.torch import TorchTrainer from ray.train import ScalingConfig, RunConfig import ray.train as train def train_loop_per_worker(config): import torch from torch.utils.data import DataLoader # Ray Train auto-wraps model with DDP model = train.torch.prepare_model(build_model()) optimizer = torch.optim.Adam(model.parameters(), lr=config["lr"]) dataset = train.get_dataset_shard("train") for epoch in range(config["epochs"]): for batch in dataset.iter_torch_batches(batch_size=64): loss = compute_loss(model, batch) loss.backward() optimizer.step() optimizer.zero_grad() # Report metrics to Ray Train controller train.report({"epoch": epoch, "loss": loss.item()}) trainer = TorchTrainer( train_loop_per_worker, train_loop_config={"lr": 1e-3, "epochs": 10}, scaling_config=ScalingConfig( num_workers=4, use_gpu=True, resources_per_worker={"GPU": 1, "CPU": 4} ), datasets={"train": ray.data.read_parquet("s3://data/train/")}, ) result = trainer.fit() print(result.metrics)

Checkpointing & Fault Tolerance

Ray Train supports automatic checkpointing. If a worker fails, training resumes from the latest checkpoint rather than restarting from scratch. This is critical for long-running training jobs on spot instances.

from ray.train import Checkpoint # Inside train_loop_per_worker: with tempfile.TemporaryDirectory() as tmpdir: torch.save(model.state_dict(), os.path.join(tmpdir, "model.pt")) train.report( {"loss": loss.item()}, checkpoint=Checkpoint.from_directory(tmpdir) )

Ray Serve: Model Serving

Ray Serve is a scalable model-serving library that supports complex inference graphs, dynamic batching, and multi-model composition. Unlike traditional serving frameworks, Serve lets you compose multiple models and business logic into a single deployment graph.

Basic Deployment

from ray import serve import torch @serve.deployment( num_replicas=3, ray_actor_options={"num_gpus": 1} ) class ImageClassifier: def __init__(self): self.model = torch.hub.load("pytorch/vision", "resnet50") self.model.eval().cuda() self.transform = get_transforms() async def __call__(self, request): image = await request.body() tensor = self.transform(image).unsqueeze(0).cuda() with torch.no_grad(): prediction = self.model(tensor) return {"class": prediction.argmax().item()} serve.run(ImageClassifier.bind())

Deployment Graph Composition

Serve lets you chain deployments into a DAG, enabling multi-model pipelines such as ensemble models or preprocessing + inference chains:

# Multi-model ensemble with Ray Serve @serve.deployment class Preprocessor: def __call__(self, raw_input): return normalize_and_tokenize(raw_input) @serve.deployment class SentimentModel: def __call__(self, tokens): return self.model(tokens) @serve.deployment class Router: def __init__(self, preprocessor, model_a, model_b): self.prep = preprocessor self.model_a = model_a self.model_b = model_b async def __call__(self, request): text = await request.json() tokens = await self.prep.remote(text) # Ensemble: average two model scores a, b = await asyncio.gather( self.model_a.remote(tokens), self.model_b.remote(tokens) ) return {"score": (a + b) / 2} app = Router.bind( Preprocessor.bind(), SentimentModel.bind(), SentimentModel.bind() )
Autoscaling: Ray Serve supports autoscaling based on request queue depth. Set autoscaling_config={"min_replicas": 1, "max_replicas": 10, "target_num_ongoing_requests_per_replica": 5} to scale replicas dynamically.

Ray Data: Distributed Preprocessing

Ray Data is a distributed data processing library optimized for ML workloads. It provides streaming execution, which means it processes data in windows rather than materializing entire datasets in memory. This makes it ideal for large-scale preprocessing, feature engineering, and last-mile data ingest for training.

Reading & Transforming Datasets

import ray # Read from multiple sources ds = ray.data.read_parquet("s3://bucket/features/") # Chained transformations (lazy, streaming) ds = ds.filter(lambda row: row["label"] is not None) ds = ds.map(lambda row: { "features": normalize(row["features"]), "label": row["label"] }) # Batch transform with GPU acceleration def embed_batch(batch): import torch model = torch.hub.load("sentence-transformers", "all-MiniLM-L6-v2") batch["embedding"] = model.encode(batch["text"]).tolist() return batch ds = ds.map_batches( embed_batch, batch_size=256, num_gpus=1, concurrency=4 )

Integration with Ray Train

Ray Data integrates directly with Ray Train, providing efficient streaming data loading for distributed training without the need for intermediate storage:

# Seamless pipeline: Data → Train train_ds = ray.data.read_parquet("s3://data/train/") val_ds = ray.data.read_parquet("s3://data/val/") trainer = TorchTrainer( train_loop, datasets={"train": train_ds, "val": val_ds}, scaling_config=ScalingConfig(num_workers=8, use_gpu=True) ) # Data is streamed to workers — never fully materialized result = trainer.fit()
Memory Management: When using map_batches with large models, set concurrency carefully. Each concurrent worker loads the model into memory. For a 2 GB model with concurrency=8, you need at least 16 GB of free memory on the node.

Deployment Patterns

Ray supports multiple deployment targets, from local development to production Kubernetes clusters. The recommended production deployment uses KubeRay, an operator that manages Ray clusters as native Kubernetes resources.

Local / Development

Use ray.init() to start a single-node cluster. Great for development and testing. All tasks and actors run on the local machine.

KubeRay (Production)

Deploy a RayCluster CRD on Kubernetes. Supports autoscaling, GPU scheduling, multi-tenancy, and integration with cloud-native monitoring tools.

KubeRay Configuration

# RayCluster custom resource (YAML) apiVersion: ray.io/v1 kind: RayCluster metadata: name: ml-training-cluster spec: headGroupSpec: rayStartParams: dashboard-host: "0.0.0.0" template: spec: containers: - name: ray-head image: rayproject/ray-ml:2.9.0-gpu resources: requests: { cpu: "4", memory: "16Gi" } workerGroupSpecs: - replicas: 4 minReplicas: 1 maxReplicas: 20 groupName: gpu-workers template: spec: containers: - name: ray-worker image: rayproject/ray-ml:2.9.0-gpu resources: requests: { cpu: "8", memory: "32Gi" } limits: { nvidia.com/gpu: "1" }
Tip: Use Ray's runtime_env to specify per-job dependencies (pip packages, environment variables, working directories) without rebuilding Docker images. This significantly accelerates iteration during development.

Monitoring & Observability

Ray provides a built-in dashboard at port 8265 that shows cluster health, resource utilization, task/actor status, and logs. For production, integrate with Prometheus and Grafana using Ray's metrics export endpoint.

# Enable Prometheus metrics export ray.init( _metrics_export_port=8080, _system_config={ "metrics_report_interval_ms": 1000 } ) # Scrape endpoint: http://<head-node>:8080/metrics # Key metrics to watch: # ray_tasks — active, pending, finished tasks # ray_actors — running, pending actors by class # ray_object_store — used/available memory # ray_resources — GPU/CPU utilization per node
