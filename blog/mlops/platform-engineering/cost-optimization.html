Cost Optimization for ML Infrastructure

MLOps Series Platform Engineering

The ML Infrastructure Cost Challenge

GPU compute is the single largest line item in any ML organization's cloud bill. A single NVIDIA A100 instance on AWS (p4d.24xlarge) costs ~$32/hour on-demand — a 30-day training run on a 64-GPU cluster can easily exceed $700,000. Without deliberate cost engineering, organizations routinely overspend by 40–60% on ML infrastructure.

Cost optimization for ML is fundamentally different from traditional cloud cost management. ML workloads are bursty, GPU-bound, and have unique characteristics: long-running training jobs, periodic hyperparameter sweeps, and inference workloads with variable traffic. A one-size-fits-all approach will fail.

The 3 pillars of ML cost optimization: (1) Right-size your GPU selection for the workload, (2) Use the optimal pricing model for each job type, and (3) Eliminate idle GPU time through scheduling and automation.

Cost Breakdown by Workload Type

Training (60–70% of spend)

Long-running, GPU-saturated jobs. Most amenable to reserved capacity and spot instances. Typical duration: hours to weeks.

Inference (20–30% of spend)

Variable demand, latency-sensitive. Benefits from autoscaling and right-sizing. Can often use smaller GPUs or CPU for batch inference.

Right-Sizing GPU Resources

The most expensive mistake in ML infrastructure is choosing the wrong GPU for the job. An H100 costs 3× more per hour than an A10G, but delivers 5–8× the throughput for large-model training. Conversely, using an H100 for a small fine-tuning job wastes most of its capacity.

GPU Selection Guide

Small Models (< 1B params)

  • Training: A10G, T4, L4
  • Inference: T4 or CPU
  • Cost: $1–3/hr per GPU

Large Models (> 10B params)

  • Training: A100 80GB, H100
  • Inference: A100, A10G (quantized)
  • Cost: $4–12/hr per GPU

GPU Utilization Monitor

import subprocess, time, json def monitor_gpu_utilization(interval=10, samples=60): """Collect GPU utilization samples to determine right-sizing.""" metrics = [] for _ in range(samples): result = subprocess.run( ["nvidia-smi", "--query-gpu=utilization.gpu,memory.used,memory.total", "--format=csv,noheader,nounits"], capture_output=True, text=True ) for line in result.stdout.strip().split("\n"): gpu_util, mem_used, mem_total = line.split(", ") metrics.append({ "gpu_util": float(gpu_util), "mem_pct": float(mem_used) / float(mem_total) * 100 }) time.sleep(interval) avg_gpu = sum(m["gpu_util"] for m in metrics) / len(metrics) avg_mem = sum(m["mem_pct"] for m in metrics) / len(metrics) print(f"Avg GPU utilization: {avg_gpu:.1f}%") print(f"Avg memory usage: {avg_mem:.1f}%") if avg_gpu < 30: print("⚠ GPU severely underutilized — consider a smaller instance.") elif avg_mem < 40: print("⚠ Memory underutilized — a GPU with less VRAM may suffice.") else: print("✓ Utilization looks healthy.") monitor_gpu_utilization()
Anti-pattern: Don't default to the largest GPU available. Profile your workload for 1 hour on a smaller instance first. If GPU utilization stays below 50% or memory usage below 40%, you are wasting money.

Reserved vs Spot Pricing Strategies

Cloud providers offer three pricing tiers for GPU instances, each suited to different workload patterns. Understanding when to use each is the single highest-impact cost optimization you can make.

On-Demand

  • Full price, no commitment
  • Guaranteed availability
  • Best for: unpredictable, short jobs
  • Cost: $32/hr (p4d.24xlarge)

Reserved / Committed Use

  • 1–3 year commitment
  • 40–60% discount
  • Best for: steady-state training infra
  • Cost: ~$13–19/hr equivalent

Spot / Preemptible

  • 60–90% discount
  • Can be reclaimed with 2 min notice
  • Best for: fault-tolerant training
  • Cost: ~$3–10/hr (varies)

Savings Plans (AWS)

  • Flexible commitment ($X/hr)
  • 20–40% discount
  • Applies across instance types
  • Best for: mixed GPU workloads
Rule of thumb: Reserve 60% of your baseline GPU capacity. Use spot for 30% (training with checkpointing). Keep 10% on-demand for urgent jobs and debugging.

Cost Calculator

def estimate_monthly_cost( num_gpus, hours_per_day, days_per_month=30, on_demand_rate=32.0, reserved_rate=14.0, spot_rate=6.5, on_demand_pct=0.10, reserved_pct=0.60, spot_pct=0.30 ): """Estimate monthly GPU cost with a blended pricing strategy.""" total_gpu_hours = num_gpus * hours_per_day * days_per_month cost_on_demand = total_gpu_hours * on_demand_pct * on_demand_rate cost_reserved = total_gpu_hours * reserved_pct * reserved_rate cost_spot = total_gpu_hours * spot_pct * spot_rate total = cost_on_demand + cost_reserved + cost_spot pure_on_demand = total_gpu_hours * on_demand_rate savings = (1 - total / pure_on_demand) * 100 print(f"Total GPU-hours/month: {total_gpu_hours:,.0f}") print(f"Blended monthly cost: ${total:,.0f}") print(f"Pure on-demand cost: ${pure_on_demand:,.0f}") print(f"Savings: {savings:.1f}%") # Example: 64 GPUs, 20 hrs/day estimate_monthly_cost(num_gpus=64, hours_per_day=20)

Scheduling Strategies

Even with perfect pricing, idle GPUs burn money. Intelligent scheduling ensures that every GPU-hour is productively used. The key strategies are priority-based queuing, bin-packing, and time-of-day scheduling.

Priority Queue with Preemption

Implement a multi-tier priority system: production training jobs preempt experimentation jobs, which preempt hyperparameter sweeps. When a high-priority job arrives, lower-priority jobs are checkpointed and evicted.

# Kubernetes PriorityClass for ML workloads apiVersion: scheduling.k8s.io/v1 kind: PriorityClass metadata: name: production-training value: 1000000 preemptionPolicy: PreemptLowerPriority description: "Production model training — highest priority" --- apiVersion: scheduling.k8s.io/v1 kind: PriorityClass metadata: name: experimentation value: 100000 preemptionPolicy: PreemptLowerPriority description: "Experimental training runs" --- apiVersion: scheduling.k8s.io/v1 kind: PriorityClass metadata: name: hpo-sweep value: 10000 preemptionPolicy: Never description: "Hyperparameter sweeps — will not preempt others"

Monitoring & Alerting for Cost

You can't optimize what you don't measure. A robust cost monitoring system provides visibility into spend by team, project, and workload type — and triggers alerts before budgets are exceeded.

Prometheus + Grafana Cost Metrics

Budget Alert Configuration

# Grafana alert rule (YAML representation) alert: name: gpu-budget-exceeded condition: > sum(ml_gpu_cost_per_hour_dollars) * 720 > 50000 for: 1h labels: severity: critical annotations: summary: "Projected monthly GPU spend exceeds $50,000" runbook: "Check idle instances, verify spot fallback is active"
Don't rely on monthly billing alone. By the time you see a monthly cloud bill, the overspend has already happened. Real-time monitoring with hourly projections catches runaway costs within hours, not weeks.

ROI Framework for ML Infrastructure

Cost optimization is not just about spending less — it's about maximizing the value per dollar spent. An ROI framework connects infrastructure cost to business outcomes: models shipped, accuracy improvements, and time-to-production.

Cost-per-Experiment Metric

def calculate_experiment_roi( gpu_hours, cost_per_gpu_hour, accuracy_improvement_pct, revenue_per_pct_accuracy ): """Calculate ROI of a single training experiment.""" total_cost = gpu_hours * cost_per_gpu_hour revenue_impact = accuracy_improvement_pct * revenue_per_pct_accuracy roi = (revenue_impact - total_cost) / total_cost * 100 print(f"Experiment cost: ${total_cost:,.0f}") print(f"Revenue impact: ${revenue_impact:,.0f}") print(f"ROI: {roi:.0f}%") return roi # Example: 500 GPU-hours at $14/hr, 0.5% accuracy gain worth $200K/pct calculate_experiment_roi(500, 14.0, 0.5, 200000)

Optimization Checklist

Quick Wins (Week 1)

  • Enable idle GPU auto-shutdown
  • Tag all resources by team/project
  • Set up daily cost alerts
  • Review GPU utilization reports

Strategic (Month 1–3)

  • Implement spot instance strategy
  • Purchase reserved capacity for baseline
  • Deploy priority-based scheduling
  • Build cost-per-experiment dashboard
Target: A mature ML platform team should aim for 70%+ GPU utilization across the fleet and a blended cost that is 40–55% below on-demand pricing. Track these metrics monthly.

Cost optimization is an ongoing discipline, not a one-time project. The most effective teams build cost awareness into their culture — every ML engineer should see the dollar cost of their experiments alongside the accuracy metrics.
