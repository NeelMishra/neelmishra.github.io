Structuring ML Projects: Mono-repo Patterns

MLOps Series Foundations

Introduction

Every ML project begins as a single notebook, and—left unchecked—ends as a tangled web of scripts, configs, and data artefacts that only one person can navigate. Unlike traditional software where a bad directory layout is merely annoying, a chaotic ML repo actively damages reproducibility, collaboration, and deployment velocity.

A well-structured repository acts as executable documentation. New team members can on-board in hours instead of days, CI/CD pipelines slot in naturally, and experiments remain traceable months after they were run. This post covers the patterns, tools, and trade-offs you need to choose the right layout for your ML projects.

Post 7 of 12 in the MLOps Foundations series. Earlier posts cover the ML lifecycle and technical debt—both feed directly into structural decisions discussed here.

Directory Layout

A standard ML project has more moving parts than a typical web service: data pipelines, feature engineering, training scripts, evaluation harnesses, model artefacts, and serving configs. A flat structure collapses under the weight; you need deliberate separation of concerns. 

my-ml-project/
├── configs/               # Hydra / YAML configuration files
│   ├── model/
│   │   ├── resnet.yaml
│   │   └── transformer.yaml
│   ├── data/
│   │   └── imagenet.yaml
│   └── config.yaml          # root config (composes sub-configs)
├── data/                   # raw / processed / interim splits
│   ├── raw/
│   ├── processed/
│   └── interim/
├── notebooks/              # exploratory analysis only
│   └── 01_eda.ipynb
├── src/                    # importable Python package
│   └── my_ml_project/
│       ├── __init__.py
│       ├── data/            # data loading & transforms
│       │   ├── __init__.py
│       │   └── dataset.py
│       ├── models/          # model definitions
│       │   ├── __init__.py
│       │   └── resnet.py
│       ├── training/        # train loops, callbacks
│       │   ├── __init__.py
│       │   └── trainer.py
│       └── evaluation/      # metrics, reports
│           ├── __init__.py
│           └── metrics.py
├── tests/                  # mirrors src/ layout
│   ├── test_dataset.py
│   └── test_trainer.py
├── scripts/                # CLI entry-points
│   ├── train.py
│   └── evaluate.py
├── docker/                 # Dockerfiles for train & serve
│   ├── Dockerfile.train
│   └── Dockerfile.serve
├── Makefile                  # common tasks: lint, test, train
├── pyproject.toml
├── README.md
└── .gitignore
Tip: Keep notebooks/ for exploration only. Any code that survives experimentation should be refactored into src/ so it can be tested and imported.

The src/ layout (also called the src-layout convention) ensures your package is only importable after installation, preventing subtle import bugs that plague flat layouts where the package directory sits at the repository root.

Caveat: Never commit large data files into Git. Use DVC, Git LFS, or cloud storage with a data/README.md that explains how to fetch data.

Config Management

Hard-coded hyper-parameters scattered across scripts are the fastest way to lose track of experiments. Centralised configuration with Hydra or plain YAML hierarchies solves three problems at once: reproducibility, multi-run sweeps, and environment-specific overrides.

Hydra Pattern

# configs/config.yaml  (Hydra root)
defaults:
  - model: resnet
  - data: imagenet
  - _self_

training:
  epochs: 90
  lr: 0.01
  batch_size: 256
  seed: 42

logging:
  project: "image-classification"
  experiment: "${model.name}_${training.epochs}ep"
# configs/model/resnet.yaml
name: resnet50
pretrained: true
num_classes: 1000
dropout: 0.2

Using Hydra in Python

import hydra
from omegaconf import DictConfig

@hydra.main(version_base=None, config_path="../configs", config_name="config")
def train(cfg: DictConfig) -> None:
    # Access nested keys with dot notation
    model = build_model(cfg.model.name, cfg.model.num_classes)
    optimizer = build_optimizer(model, lr=cfg.training.lr)

    for epoch in range(cfg.training.epochs):
        train_one_epoch(model, optimizer, cfg)

# CLI override example:
# python scripts/train.py model=transformer training.lr=0.001
Tip: Hydra automatically creates an outputs/ directory with the full resolved config for every run, giving you free experiment logging.

Hierarchical Config without Hydra

If Hydra feels too heavy, a simple pattern with PyYAML and a merge utility works well for smaller projects: 

import yaml
from pathlib import Path

def load_config(base: str, overrides: str = None) -> dict:
    """Load base YAML and optionally merge overrides."""
    cfg = yaml.safe_load(Path(base).read_text())
    if overrides:
        patch = yaml.safe_load(Path(overrides).read_text())
        cfg = {**cfg, **patch}           # shallow merge
    return cfg
Caveat: Shallow merging can silently overwrite nested keys. For production use, prefer a deep-merge utility such as OmegaConf.merge() or deepmerge.

Mono-repo vs Multi-repo

When an organisation has multiple ML models, a critical architectural decision is whether to house everything in a single repository (mono-repo) or split each model / service into its own repo (multi-repo). Both patterns are valid—the right choice depends on team size, deployment cadence, and shared infrastructure.

Mono-repo

  • Atomic cross-project changes – a single PR can update shared feature code and every model that uses it
  • Consistent tooling – one linter config, one CI template, one dependency lockfile
  • Easier code sharing – no package publishing step for internal libraries
  • Simpler onboarding – clone once, run anything

Multi-repo

  • Independent release cycles – each model ships on its own cadence
  • Fine-grained access control – limit who can push to production repos
  • Smaller CI scope – changes trigger only the affected pipeline
  • Clear ownership – each repo maps to one team
Tip: Many teams start with a mono-repo and extract services into separate repos only when CI times or access-control needs force it. This avoids premature complexity.

Mono-repo Layout Example

ml-platform/
├── models/
│   ├── fraud-detector/      # self-contained model package
│   │   ├── configs/
│   │   ├── src/
│   │   ├── tests/
│   │   └── pyproject.toml
│   └── recommender/
│       ├── configs/
│       ├── src/
│       ├── tests/
│       └── pyproject.toml
├── libs/
│   ├── feature-store-client/ # shared internal package
│   └── ml-utils/
├── infra/                   # Terraform / Helm charts
├── .github/
│   └── workflows/
│       ├── ci-fraud.yaml     # path-filtered CI per model
│       └── ci-recommender.yaml
├── Makefile
└── README.md
Caveat: Mono-repos require path-filtered CI (e.g., GitHub Actions paths: filter) to avoid rebuilding everything on every push. Without this, CI times grow linearly with the number of projects.

Cookiecutter Templates

Rather than copying an old project and deleting half the files, use Cookiecutter to generate new ML projects from a template. The most popular ML template is cookiecutter-data-science, but you can also create an internal template tailored to your organisation’s stack. 

# Install cookiecutter
pip install cookiecutter

# Generate from the canonical data-science template
cookiecutter https://github.com/drivendataorg/cookiecutter-data-science

# Or from your team's internal template
cookiecutter git@github.com:myorg/ml-template.git

Customising the Template

A Cookiecutter template is just a directory tree with {{ cookiecutter.variable }} placeholders. The cookiecutter.json file defines the prompts and defaults: 

// cookiecutter.json
{
  "project_name": "My ML Project",
  "project_slug": "{{ cookiecutter.project_name | lower | replace(' ', '_') }}",
  "author": "Your Name",
  "python_version": "3.11",
  "use_docker": ["yes", "no"],
  "ml_framework": ["pytorch", "tensorflow", "jax"],
  "use_hydra": ["yes", "no"]
}
# Template directory structure (before rendering)
{{ cookiecutter.project_slug }}/
├── src/
│   └── {{ cookiecutter.project_slug }}/
│       ├── __init__.py
│       └── train.py
├── configs/
│   └── config.yaml
├── README.md               # uses {{ cookiecutter.project_name }}
├── pyproject.toml
└── Makefile
Tip: Add post-generation hooks (hooks/post_gen_project.py) to automatically run git init, create a virtual environment, or install pre-commit hooks after scaffolding.
Caveat: Templates drift over time. Assign an owner and version your internal template. Without maintenance, new projects will scaffold with stale patterns while the rest of the org has moved on.

Dependency Management

ML dependencies are notoriously fragile—CUDA versions, framework releases, and native extensions create a combinatorial explosion. Pinning everything is essential, but how you pin matters just as much.

pyproject.toml (Recommended)

# pyproject.toml — the modern standard
[project]
name = "my-ml-project"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = [
    "torch>=2.1,<2.3",
    "lightning>=2.0",
    "hydra-core>=1.3",
    "pandas>=2.0",
]

[project.optional-dependencies]
dev = ["pytest", "ruff", "pre-commit"]
serve = ["fastapi", "uvicorn"]

[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.backends._legacy:_Backend"

Lockfiles & Reproducibility

# Generate a lockfile with pip-tools
pip-compile pyproject.toml -o requirements.lock

# Install from the lockfile (deterministic)
pip install -r requirements.lock

# Or use uv for speed
uv pip compile pyproject.toml -o requirements.lock
uv pip install -r requirements.lock
Tip: Commit both pyproject.toml (flexible ranges) and requirements.lock (exact pins). The former documents intent; the latter guarantees reproducibility.

setup.py vs pyproject.toml

pyproject.toml

  • PEP 621 standard – works with all modern build tools
  • Declarative – no executable code in build config
  • Single file for metadata, deps, tool config (ruff, pytest)

setup.py

  • Legacy but still ubiquitous in older codebases
  • Imperative – can run arbitrary code at install time
  • Needed for complex C-extension builds (rare in pure ML)
Caveat: Avoid mixing setup.py and pyproject.toml in the same project. The interaction between the two is under-specified and causes confusing build failures.

Best Practices

Structure is necessary but not sufficient. The conventions you layer on top—naming, documentation, automation—determine whether the repo stays healthy as the team and codebase grow.

Naming Conventions

# ✅  Good naming
src/my_project/models/resnet.py        # snake_case modules
configs/model/resnet.yaml              # mirror the code structure
tests/test_resnet.py                   # prefix with test_
scripts/train.py                       # verb-based entry-points

# ❌  Avoid
src/Model1.py                          # vague, PascalCase file
configs/cfg_v2_final_REAL.yaml         # versioning in filenames
notebooks/Untitled42.ipynb             # no description
train_v3_fixed_fixed2.py               # that's what git is for

README Template

Every ML repo should have a README that answers five questions in under two minutes of reading: 

# Project Name

## What
One-sentence description of the model / pipeline.

## Why
Business context and link to design doc.

## Quickstart
git clone ... && cd project
make install          # creates venv, installs deps
make train            # trains on sample data
make test             # runs unit + integration tests

## Project Layout
Brief tree with one-line descriptions (see Directory Layout section).

## Contributing
Branch strategy, PR checklist, code-review expectations.
Tip: Use a Makefile as the single entry point for every common task. It serves as living documentation and ensures everyone runs the same commands regardless of their shell.

Makefile Example

# Makefile — single entry-point for common tasks
.PHONY: install lint test train clean

install:
	python -m venv .venv
	.venv/bin/pip install -e ".[dev]"
	.venv/bin/pre-commit install

lint:
	.venv/bin/ruff check src/ tests/
	.venv/bin/ruff format --check src/ tests/

test:
	.venv/bin/pytest tests/ -v --tb=short

train:
	.venv/bin/python scripts/train.py

clean:
	rm -rf .venv outputs/ dist/ *.egg-info

Documentation & ADRs

For decisions that shape the project—why you chose PyTorch over TensorFlow, why you use a mono-repo—keep lightweight Architecture Decision Records (ADRs) in a docs/adr/ directory: 

docs/adr/
├── 001-use-pytorch.md
├── 002-mono-repo-structure.md
└── 003-hydra-for-configs.md

# Each ADR follows a simple format:
# Title:  Use PyTorch as the training framework
# Status: Accepted
# Context: We need GPU-accelerated training with ...
# Decision: We will use PyTorch because ...
# Consequences: Team needs PyTorch expertise ...
Caveat: ADRs are only useful if they’re discoverable. Link to them from the project README and reference them in PR descriptions when a change relates to a past decision.
