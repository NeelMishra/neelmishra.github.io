What is MLOps? From DevOps to MLOps

MLOps Series Foundations

Introduction

Machine-learning models have moved from research notebooks into production systems that drive real-time recommendations, fraud detection, autonomous vehicles, and more. Yet the vast majority of ML projects never make it past the proof-of-concept stage. A 2022 Gartner survey found that only 54 % of AI projects reach production — and even those that do are often plagued by silent degradation, unreproducible results, and painful handoffs between data scientists and engineers.

MLOps (Machine Learning Operations) is the set of practices, tools, and cultural norms that brings DevOps-style discipline to the entire ML lifecycle — from data ingestion and feature engineering, through training and validation, all the way to deployment, monitoring, and retraining. Think of it as the bridge between "my model works in a notebook" and "my model works reliably at scale, 24 / 7."

Why "Ops"? — In DevOps the goal is to shorten the feedback loop between writing code and shipping it. MLOps applies the same philosophy but extends it to data and models, which introduce unique challenges that plain CI/CD cannot handle.

At its core, MLOps addresses three fundamental questions that every production ML team must answer:

  • Reproducibility — Can someone else recreate my results exactly, months from now?
  • Reliability — Will the model keep working when the world (and the data) changes?
  • Velocity — How quickly can we iterate from idea to deployed improvement?
Common misconception: MLOps is not just "deploying a model behind a REST API." That is only one small slice. True MLOps covers the full lifecycle — including data validation, experiment tracking, model versioning, monitoring, and automated retraining.

The Three Axes: Data, Model, Code

Traditional software has one primary artefact to version: code. Machine learning adds two more dimensions — data and models — creating a three-axis complexity space that makes MLOps fundamentally harder than vanilla DevOps.

Each axis introduces its own category of challenges:

  • Data axis — Data can change upstream without warning. Schema drift, label noise, and seasonality all affect model quality. You need data versioning (DVC, LakeFS), validation (Great Expectations, TFX Data Validation), and lineage tracking.
  • Model axis — Hyper-parameter combinations explode combinatorially. Experiment trackers (MLflow, Weights & Biases) and model registries give you a single source of truth for what was trained, how, and why.
  • Code axis — Training scripts, feature pipelines, and serving infrastructure are all code. Standard DevOps practices — Git, code review, CI/CD — apply here, but must be extended to handle model artefacts and data dependencies.
Key insight: A change along any of the three axes can break production. MLOps provides the observability and automation to detect and recover from changes along all three simultaneously.

Key Principles

Regardless of which tools you pick, mature MLOps teams share a common set of principles. These are the non-negotiables that separate a sustainable ML platform from a pile of ad-hoc scripts.

1. Automation

Every step that a human does manually is a step that can be forgotten, executed differently, or skipped under deadline pressure. MLOps pushes for end-to-end pipeline automation — from data validation through model retraining to canary deployment. 

# Pseudocode: automated retraining trigger
from pipeline import DataValidator, Trainer, Deployer

def retrain_on_drift(monitor_report):
    if monitor_report.drift_score > THRESHOLD:
        fresh_data = DataValidator.validate(monitor_report.new_batch)
        model      = Trainer.train(fresh_data, config=LATEST_CONFIG)
        Deployer.canary_deploy(model, traffic_pct=5)
        return "Retraining triggered"
    return "No drift detected"

2. Reproducibility

If you cannot reproduce a result, you cannot debug it, audit it, or improve on it. Reproducibility requires versioning across all three axes: data snapshots, code commits, and model artefacts linked together with a single run ID. 

# Pseudocode: reproducible experiment entry
import mlflow

with mlflow.start_run() as run:
    mlflow.log_param("lr", 0.001)
    mlflow.log_param("epochs", 50)
    mlflow.log_param("data_version", "v2.3.1")
    model = train_model(data, lr=0.001, epochs=50)
    mlflow.log_metric("f1_score", evaluate(model))
    mlflow.log_artifact(model_path)

3. Continuous Monitoring

Unlike traditional software, ML models can degrade silently. The code doesn't crash — the predictions just get worse. Monitoring must cover:

  • Data drift — Has the input distribution shifted?
  • Concept drift — Has the relationship between inputs and outputs changed?
  • Performance metrics — Accuracy, latency, throughput, and error rates in production.
  • Resource utilisation — GPU memory, CPU load, and cost-per-prediction.
Silent failure is the #1 risk in production ML. A model that returns a 200 OK but predicts garbage is worse than a model that crashes — at least a crash triggers an alert.

4. Versioning Everything

Git alone is not enough. You need to version data (DVC, LakeFS), models (MLflow Model Registry, SageMaker Model Registry), features (feature stores like Feast or Tecton), and pipeline configurations (Kubeflow, Airflow DAGs).

MLOps vs Traditional ML Workflows

The diagram below contrasts the ad-hoc approach still common in many organisations with a mature MLOps workflow. Notice how the MLOps side closes the loop — production signals feed back into data collection and retraining automatically.

Traditional ML

  • Manual, notebook-driven workflows
  • No versioning of data or models
  • Model hand-off via email / Slack
  • No monitoring — bugs discovered by users
  • Retraining is an ad-hoc, manual process

MLOps

  • Automated, pipeline-driven workflows
  • Full versioning of data, features & models
  • Programmatic promotion via model registry
  • Continuous monitoring with drift alerts
  • Automated retraining triggered by signals
Maturity spectrum: Most teams aren't at Level 0 (fully manual) or Level 2 (fully automated). The goal of MLOps is to help you move incrementally along the maturity curve — automating the highest-value steps first.

Getting Started with MLOps

Adopting MLOps is not an all-or-nothing leap. Start with the highest-impact practices and iterate from there. Here is a pragmatic roadmap for teams making the transition:

Step 1 — Version everything

Put training code in Git (you probably already have). Then add data versioning with DVC or a cloud-native lakehouse. Tag every experiment run with its code commit, data hash, and hyper-parameters. 

# Quick-start: initialise DVC in an existing Git repo
# (run these in your terminal)
$ pip install dvc dvc-s3
$ dvc init
$ dvc remote add -d storage s3://my-bucket/dvc
$ dvc add data/train.parquet
$ git add data/train.parquet.dvc .dvc .gitignore
$ git commit -m "Track training data with DVC"

Step 2 — Automate training pipelines

Replace ad-hoc notebook runs with a pipeline orchestrator. Even a simple Makefile or shell script is better than nothing. For larger teams, look at Kubeflow Pipelines, Airflow, or Prefect.

Step 3 — Add a model registry

A model registry (MLflow, SageMaker, Vertex AI) gives you a single pane of glass for every model version, its lineage, and its promotion status (Staging → Production → Archived). 

# Pseudocode: promote a model to production
import mlflow

client = mlflow.MlflowClient()
client.transition_model_version_stage(
    name="fraud_detector",
    version=12,
    stage="Production",
    archive_existing_versions=True
)

Step 4 — Monitor in production

Set up dashboards for prediction latency, feature drift (PSI, KL divergence), and business metrics. Wire alerts into your on-call rotation so drift doesn't go unnoticed for weeks.

Remember: MLOps is a spectrum, not a destination. Even adopting Steps 1 and 4 alone will dramatically improve your team's ability to debug, audit, and iterate on models.

Step 5 — Close the loop

The ultimate goal is a closed feedback loop: monitoring detects degradation → pipeline re-validates data → model retrains → registry promotes → canary deploys → monitoring verifies. This is the heart of continuous training (CT), the ML-specific extension of CI/CD.

Don't over-automate too early. Full CT makes sense only when you have confidence in your data validation, evaluation gates, and rollback mechanisms. Start with semi-automated retraining (human approval gate) and remove the gate once trust is established.
