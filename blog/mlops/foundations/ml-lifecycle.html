The ML Lifecycle: From Data to Deployment

MLOps Series Foundations

Introduction

Building a machine learning model is only a fraction of the work required to deliver value from ML in production. The ML lifecycle describes the end-to-end journey that data takes — from raw collection through cleaning, training, evaluation, deployment, and continuous monitoring. Each stage introduces unique engineering challenges, and understanding the full picture is essential before diving into specialized MLOps tooling.

Unlike traditional software, ML systems have a dual dependency: they depend on both code and data. A change in the data distribution can silently degrade model quality without any code change at all. This makes the lifecycle inherently iterative — teams cycle back through earlier stages constantly.

Post 2 of 12 in the MLOps Foundations series. This post gives you the mental map of the full lifecycle. Later posts will deep-dive into each stage with production-grade tooling, patterns, and case studies.

Data Collection and Ingestion

Every ML system begins with data. The quality, volume, and freshness of that data set an upper bound on what any model can achieve. Data collection encompasses sourcing raw signals from databases, APIs, event streams, third-party vendors, and user interactions.

Common Data Sources

  • Transactional databases — PostgreSQL, MySQL exports via CDC (Change Data Capture)
  • Event streams — Kafka, Kinesis, Pub/Sub for real-time clickstream or sensor data
  • Data lakes — S3, GCS, ADLS storing Parquet, ORC, or raw JSON files
  • Third-party APIs — weather, financial, or geolocation enrichment services
  • Manual annotations — labeling platforms like Label Studio, Scale AI, or internal tools

Ingestion Patterns

Batch Ingestion

Scheduled ETL jobs (Airflow, dbt) pull data at fixed intervals — hourly, daily, or weekly. Simple, predictable, but introduces latency.

Streaming Ingestion

Continuous pipelines (Kafka Connect, Flink) ingest events in near-real-time. Essential when model freshness directly impacts business metrics.

# Batch ingestion pseudo-pipeline
schedule daily at 02:00 UTC:
    raw_data  ← extract(source="orders_db", since=last_run)
    validated ← validate_schema(raw_data, schema="orders_v3")
    write_parquet(validated, path="s3://lake/orders/dt={{ds}}")
    log_lineage(source="orders_db", dest="s3://lake/orders")
Watch out: Schema drift is the silent killer of data pipelines. A new column or changed data type upstream can cascade failures through your entire training pipeline. Always validate schemas at the ingestion boundary.

Data Preprocessing

Raw data is rarely model-ready. Preprocessing transforms messy, heterogeneous inputs into clean, structured features that algorithms can consume. This stage often accounts for 60–80% of total project effort in real-world ML teams.

Core Preprocessing Steps

  1. Missing value treatment — imputation (mean, median, KNN) or row/column dropping
  2. Outlier handling — capping, winsorization, or isolation-forest-based removal
  3. Encoding categoricals — one-hot, ordinal, target encoding, or learned embeddings
  4. Scaling numerics — StandardScaler, MinMaxScaler, or RobustScaler depending on distribution
  5. Text tokenization — subword (BPE, SentencePiece) or word-level with vocabulary pruning
  6. Time-series alignment — resampling, lag features, rolling aggregations

Feature Engineering

Feature engineering is the art of creating new input signals from existing raw data. Well-crafted features often outperform architectural changes. Common techniques include polynomial interactions, ratio features, binning, and domain-specific transformations like TF-IDF for text or Fourier features for cyclical time data. 

# Feature engineering pipeline
def build_features(df):
    df["revenue_per_click"]   ← df.revenue / (df.clicks + 1)
    df["hour_of_day"]         ← extract_hour(df.timestamp)
    df["day_of_week_sin"]     ← sin(2π · df.day_of_week / 7)
    df["day_of_week_cos"]     ← cos(2π · df.day_of_week / 7)
    df["user_session_count"]  ← window_count(df.user_id, window="7d")
    return drop_nulls(df, threshold=0.95)
Tip: Keep preprocessing logic inside versioned transformation pipelines (e.g., sklearn Pipelines, Spark MLlib Pipelines, or feature store transformations). This guarantees the same transforms are applied in training and serving, preventing training-serving skew.

Model Training

Training is where an algorithm learns patterns from the preprocessed data. While it captures the most attention, it is only one link in the chain. A well-trained model on poor data will always underperform a simple model on clean, relevant features.

The Training Loop

At its core, every supervised training loop follows the same pattern: forward pass, compute loss, backward pass, update weights. The pseudocode below generalises this for both deep learning and gradient-boosted tree approaches. 

# Generic training loop
for epoch in 1..N:
    for batch in dataloader(train_set, batch_size=256):
        predictions ← model.forward(batch.features)
        loss        ← loss_fn(predictions, batch.labels)
        gradients   ← backward(loss)
        optimizer.step(gradients)
    # Validation check at end of each epoch
    val_metric ← evaluate(model, val_set)
    log_metric("val_auc", val_metric, step=epoch)
    if early_stopping(val_metric): break

Hyperparameter Tuning

Hyperparameters — learning rate, batch size, number of layers, regularisation strength — are not learned during training. Finding good values requires a structured search:

Grid / Random Search

Simple and parallelisable. Grid search is exhaustive; random search samples a fixed budget and often finds near-optimal configurations faster for high-dimensional spaces.

Bayesian Optimisation

Builds a surrogate model (GP, TPE) of the objective surface and picks trials that maximise expected improvement. Tools: Optuna, Ray Tune, Ax.

Experiment Tracking

Every training run should be logged with its hyperparameters, dataset version, code commit, and resulting metrics. Without experiment tracking, teams lose the ability to reproduce results or make informed decisions about which model to promote. Common tools include MLflow, Weights & Biases, and Neptune.

Tip: Tag every experiment with the Git SHA of your training code and the data version hash. This makes any run fully reproducible months later.

Model Evaluation

Training loss tells you how well a model fits the training data. Evaluation tells you how it will perform in the real world. Robust evaluation is a multi-layered process that goes far beyond a single accuracy number.

Key Metrics by Task Type

Classification

  • AUC-ROC / AUC-PR
  • F1, Precision, Recall
  • Log Loss
  • Confusion Matrix analysis

Regression

  • MAE, MSE, RMSE
  • R² / Adjusted R²
  • MAPE
  • Residual distribution

Validation Strategies

  1. Hold-out split — simple 70/15/15 or 80/10/10 train/val/test
  2. K-fold cross-validation — reduces variance in metric estimates, especially on small datasets
  3. Stratified splits — maintains class distribution across folds for imbalanced problems
  4. Time-based splits — essential for time-series; always train on past, evaluate on future

A/B Testing and Shadow Mode

Offline metrics are necessary but not sufficient. Before full deployment, run the candidate model in shadow mode: it receives live traffic and generates predictions, but those predictions are not served to users. Compare shadow predictions to the current production model to catch issues before they reach customers. 

# Shadow deployment evaluation
for request in live_traffic():
    prod_pred   ← production_model.predict(request)
    shadow_pred ← candidate_model.predict(request)
    log_comparison(prod_pred, shadow_pred, ground_truth=None)
    serve(prod_pred)  # only production model serves

# After sufficient traffic:
report ← analyze_shadow_results(window="7d")
if report.regression_detected:
    alert("Candidate model shows degradation — do NOT promote")
Caveat: Avoid using test set metrics to make iterative modelling decisions. If you peek at the test set during development, you leak information and your final metric estimate becomes optimistically biased. Reserve the test set for a single final evaluation.

Deployment and Serving

A model that only lives in a Jupyter notebook delivers zero business value. Deployment bridges the gap between experimentation and production, making predictions available to users and downstream systems.

Deployment Strategies

Blue-Green Deployment

Two identical environments. Route traffic entirely to Green (new model) once validated. Instant rollback by switching back to Blue.

Canary Deployment

Gradually route a small percentage (1% → 5% → 25% → 100%) of traffic to the new model. Monitor metrics at each step before increasing.

Serving Patterns

  • Online (real-time) serving — REST/gRPC endpoint, latency-critical (p99 < 100ms). Tools: TF Serving, Triton, Seldon Core, BentoML.
  • Batch serving — Scheduled jobs score an entire dataset at once (e.g., nightly recommendations). Tools: Spark, Airflow + model artifact.
  • Streaming serving — Model runs inside a stream processor (Flink, Kafka Streams) for near-real-time scoring of each event.
  • Edge / embedded — Model compiled (ONNX, TFLite, Core ML) and deployed to mobile devices, IoT sensors, or browsers via WASM. 
# Canary deployment orchestration
def canary_rollout(new_model, steps=[1, 5, 25, 50, 100]):
    for pct in steps:
        set_traffic_split(production=100-pct, canary=pct)
        wait(duration="30m")
        metrics ← collect_canary_metrics()
        if metrics.error_rate > threshold or metrics.latency_p99 > sla:
            rollback()
            return "FAILED"
    promote(new_model)
    return "SUCCESS"
Tip: Always containerise your model serving behind a versioned Docker image. This makes rollbacks trivial and ensures the serving environment exactly matches what you tested.

Monitoring and Feedback Loops

Deployment is not the finish line — it is the starting line. Models degrade over time as the world changes around them. Continuous monitoring detects issues early and feedback loops enable automated or semi-automated recovery.

What to Monitor

  • Data drift — statistical shift in input feature distributions (PSI, KS test, Jensen-Shannon divergence)
  • Concept drift — the relationship between features and target changes (requires ground truth labels)
  • Prediction drift — output distribution shifts even if inputs look stable
  • Operational metrics — latency, throughput, error rates, memory usage
  • Business KPIs — click-through rate, conversion, revenue per session

Drift Detection Pipeline

# Drift monitoring pseudo-pipeline
schedule every 6 hours:
    ref_stats   ← load_reference_stats(training_data)
    live_stats  ← compute_stats(serving_logs, window="6h")
    drift_score ← psi(ref_stats, live_stats)

    if drift_score > 0.2:
        alert("HIGH drift detected — triggering retraining")
        trigger_pipeline("retrain", data_window="30d")
    elif drift_score > 0.1:
        alert("MODERATE drift — investigate features")
    else:
        log("Distributions stable")

Retraining Triggers

  1. Scheduled retraining — fixed cadence (weekly, monthly) regardless of drift
  2. Drift-triggered — automatic retraining when drift score exceeds a threshold
  3. Performance-triggered — retrain when live business metric degrades below an SLA
  4. Data-volume-triggered — retrain once a certain amount of new labelled data accumulates
Caveat: Automated retraining without safeguards is dangerous. Always include a validation gate that compares the retrained model against the production model on a held-out test set before promotion. Never blindly push a retrained model to production.

Summary

The ML lifecycle is not a linear process — it is a continuous loop. Data feeds into preprocessing, which feeds training, which feeds evaluation, which feeds deployment, which feeds monitoring, which feeds back into data collection and retraining. Mastering each stage and — critically — the handoffs between stages is what separates hobby projects from production ML systems.

In the upcoming posts, we will deep-dive into each stage with production tooling: feature stores, experiment tracking platforms, model registries, CI/CD for ML, and observability stacks. The lifecycle map you built here will serve as your north star throughout the series.

Key takeaway: Invest as much engineering rigour in your data pipelines, evaluation harness, and monitoring stack as you do in your model architecture. The model is the easy part — the system around it is what makes ML work at scale.
