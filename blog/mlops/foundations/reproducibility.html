Reproducibility in ML

MLOps Series Foundations

Introduction

Reproducibility is the cornerstone of the scientific method, yet machine learning has a well-documented reproducibility crisis. A 2020 survey found that fewer than half of published ML papers provided enough information to fully reproduce their results. When you cannot reliably recreate an experiment, you cannot trust comparisons between models, debug regressions, or satisfy regulatory audits.

In production MLOps, the stakes are even higher. A model that cannot be rebuilt from scratch after a data-center migration or a dependency upgrade becomes a liability. Reproducibility is not an academic nicety — it is an operational requirement.

Core principle: A reproducible ML experiment requires three pillars — identical code, identical data, and an identical environment. Miss any one of the three, and results may diverge.

Sources of Non-Determinism

Before you can fix reproducibility, you need to understand where randomness creeps in. Non-determinism in ML pipelines arises from multiple layers — some obvious, some deeply hidden.

Hardware Non-Determinism

GPUs execute floating-point additions in parallel, and because floating-point addition is not associative, the order of reduction can change the result at the last decimal place. Over millions of operations, these tiny differences compound. cuDNN's auto-tuner selects the fastest convolution algorithm at runtime, which may differ between runs.

Caveat: Enabling fully deterministic GPU operations (e.g. torch.use_deterministic_algorithms(True)) can reduce training throughput by 10–20%. Decide whether bit-exact reproducibility is worth the performance cost for your use case.

Software Non-Determinism

Even a minor patch-level bump in a library can alter numerical results. Python's built-in hash function is randomised by default since Python 3.3 — this affects dictionary iteration order and can change how data is batched. Certain operations like scatter_add in PyTorch are inherently non-deterministic on GPU.

Data Ordering

Shuffling training data is essential for convergence, but if the shuffle order differs between runs, gradient updates follow a different trajectory. This is one of the most common — and most easily fixed — sources of divergence.

Random Seeds

Weight initialisation, dropout masks, and stochastic data augmentation all rely on pseudo-random number generators. Each framework maintains its own PRNG state. Setting a seed in NumPy does not automatically seed PyTorch or TensorFlow.

Setting Seeds

The first line of defence is to set deterministic seeds for every PRNG in your stack. Below is a comprehensive seed-setting function that covers the most common frameworks.

Universal Seed Function (PyTorch)

import os
import random
import numpy as np
import torch

def set_seed(seed: int = 42) -> None:
    """Lock every PRNG we know about."""
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)

    # Deterministic cuDNN — trades speed for reproducibility
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

    # Fix Python hash seed
    os.environ["PYTHONHASHSEED"] = str(seed)

set_seed(42)

TensorFlow Seed Setting

import os
import random
import numpy as np
import tensorflow as tf

def set_seed_tf(seed: int = 42) -> None:
    random.seed(seed)
    np.random.seed(seed)
    tf.random.set_seed(seed)
    os.environ["PYTHONHASHSEED"] = str(seed)
    os.environ["TF_DETERMINISTIC_OPS"] = "1"

set_seed_tf(42)
Tip: Always call the seed function as the very first thing in your training script — before any library imports or data loading — to ensure that every downstream operation inherits the fixed state.

DataLoader Worker Seeds

When using PyTorch's DataLoader with num_workers > 0, each worker subprocess gets a different random state. You must supply a worker_init_fn to reseed each worker deterministically. 

def worker_init_fn(worker_id: int) -> None:
    seed = torch.initial_seed() % 2**32
    np.random.seed(seed)
    random.seed(seed)

loader = torch.utils.data.DataLoader(
    dataset,
    batch_size=32,
    shuffle=True,
    num_workers=4,
    worker_init_fn=worker_init_fn,
    generator=torch.Generator().manual_seed(42),
)
Warning: Seeds alone do not guarantee bitwise identical results across different GPU architectures or CUDA versions. They only guarantee reproducibility on the same hardware and software stack.

Environment Pinning

Your model's behaviour is a function of its dependencies. A single library upgrade can silently change numerical results. Pinning your environment ensures that every team member — and every CI runner — uses exactly the same stack.

pip freeze

Captures every installed package and its exact version. Simple, universal, but does not capture transitive dependency resolution order.

# Generate lock file
pip freeze > requirements.txt

# Recreate environment
pip install -r requirements.txt

conda lock

Produces a platform-specific lock file that includes the exact build strings, channels, and hashes for every package — including non-Python dependencies like CUDA and MKL.

# Generate platform lock
conda-lock lock -f environment.yml

# Recreate environment
conda-lock install conda-lock.yml

Poetry

Uses a pyproject.toml for dependency specification and a poetry.lock for exact resolution. Handles virtual environments automatically.

# Generate lock
poetry lock

# Install from lock
poetry install

pip-tools

Compiles a loose requirements.in into a fully-pinned requirements.txt with hashes for integrity verification.

# Compile with hashes
pip-compile --generate-hashes \
  requirements.in

# Sync environment exactly
pip-sync requirements.txt
Tip: Commit your lock file to version control. Treat a missing or stale lock file the same way you would treat a failing test — block the merge until it is resolved.

Docker for Reproducibility

Environment pinning handles Python packages, but ML pipelines also depend on system libraries, CUDA drivers, and OS-level configuration. Docker containers freeze the entire runtime, making your experiment portable across laptops, on-prem clusters, and cloud instances.

Example Dockerfile

# Pin the base image by digest, not just tag
FROM nvidia/cuda:12.1.1-runtime-ubuntu22.04@sha256:abc123...

ENV PYTHONHASHSEED=42
ENV PYTHONDONTWRITEBYTECODE=1
ENV DEBIAN_FRONTEND=noninteractive

# System deps — pin versions where possible
RUN apt-get update && apt-get install -y --no-install-recommends \
    python3.10=3.10.12-1~22.04 \
    python3-pip \
 && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# Install Python deps from lock file
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy source code last (best cache layer order)
COPY . .

ENTRYPOINT ["python3", "train.py"]
Caveat: Pinning by tag alone (e.g. :12.1.1-runtime) is not enough — tags are mutable. Always pin by digest (@sha256:...) for true immutability.
Tip: Use multi-stage builds to keep your training image large (with compilers, dev headers) while producing a slim serving image. Both stages share the same lock file, ensuring consistent dependencies.

Data Versioning

Git tracks code changes beautifully, but it was never designed for large binary datasets. Data versioning tools fill this gap, letting you snapshot, diff, and roll back your datasets with the same rigour you apply to code.

DVC (Data Version Control)

Works alongside Git. Stores lightweight .dvc pointer files in your repo while pushing actual data to remote storage (S3, GCS, Azure, SSH). Pipeline DAGs let you track which data produced which model.

# Initialise DVC in a Git repo
dvc init

# Track a data directory
dvc add data/training/

# Push to remote storage
dvc remote add -d myremote s3://bucket/dvc
dvc push

# Checkout a specific data version
git checkout v1.0
dvc checkout

LakeFS

Provides a Git-like interface on top of object storage. You can branch, commit, and merge datasets just like code. Ideal for data lake environments with many concurrent writers.

# Create a branch for experimentation
lakectl branch create \
  lakefs://repo/exp-42 \
  --source lakefs://repo/main

# Commit data changes
lakectl commit lakefs://repo/exp-42 \
  -m "Add augmented samples"

# Merge back to main
lakectl merge \
  lakefs://repo/exp-42 \
  lakefs://repo/main
Tip: Pair data versioning with a metadata store. Record the data hash, version tag, and any preprocessing parameters alongside every experiment run so you can trace any model back to its exact dataset.

Experiment Tracking

Experiment tracking tools record the complete lineage of every training run — hyperparameters, metrics, code version, data version, and artefacts. They are the single source of truth when someone asks "What produced this model?"

MLflow

Open-source, self-hosted. Tracks experiments, packages models with MLmodel format, and provides a model registry for staging and production promotion. Integrates with all major ML frameworks.

import mlflow

mlflow.set_experiment("repro-demo")

with mlflow.start_run():
    mlflow.log_param("lr", 0.001)
    mlflow.log_param("seed", 42)
    mlflow.log_metric("val_acc", 0.934)
    mlflow.pytorch.log_model(model, "model")

Weights & Biases (W&B)

Cloud-hosted (or self-hosted). Offers rich dashboards, automatic hyperparameter sweep orchestration, and artefact lineage graphs. Popular in research labs for its collaboration features.

import wandb

wandb.init(project="repro-demo", config={
    "lr": 0.001,
    "seed": 42,
})

# Log metrics during training
wandb.log({"val_acc": 0.934, "epoch": 10})

# Save model artefact
artifact = wandb.Artifact("model-v1", type="model")
artifact.add_file("model.pt")
wandb.log_artifact(artifact)
Tip: Log the Git commit SHA and the data version hash as run parameters. This creates a direct link from any tracked experiment back to the exact code and data that produced it — the gold standard for reproducibility.
Caveat: Experiment trackers record what was done, but they do not enforce reproducibility on their own. A logged experiment is only reproducible if the environment and data are also versioned.

Reproducibility Checklist

Use this checklist before every major experiment or model hand-off. Each item addresses one of the reproducibility pillars we covered above.

Quick Reference Table

Must Have

  • Fixed seeds for all PRNGs
  • Lock file in version control
  • Data version recorded per run
  • Experiment tracker logging params & metrics

Nice to Have

  • Docker digest pinning
  • Deterministic GPU mode enabled
  • Automated reproducibility CI check
  • Data diff between versions
Final thought: Perfect reproducibility in ML is often asymptotic — you get closer but may never reach bitwise identity across every possible platform. The goal is not perfection but sufficiency: can you reproduce results closely enough to make confident decisions? If yes, your reproducibility practice is working.
