Writing ML Design Documents

MLOps Series Foundations

Introduction

Machine learning projects carry a unique blend of uncertainty: the data may shift, the model may underperform, and the business requirements can evolve faster than experiments complete. A well-crafted ML design document is the single most effective tool for taming that uncertainty before a single line of training code is written.

Unlike traditional software design docs, an ML design doc must address questions that are specific to the ML lifecycle — Where does the training data come from? How will we measure success offline and online? What happens when the model degrades in production?

Tip: Start writing the design doc before any exploratory notebook work. The act of articulating the problem often reveals hidden assumptions that would otherwise cost weeks of wasted effort.

A design doc is not a research paper. It is a living agreement between stakeholders — data scientists, ML engineers, product managers, and SREs — on what will be built, why, and how success will be measured. It should be concise enough to read in 20 minutes yet detailed enough to prevent two engineers from building conflicting solutions.

Template Sections

A solid ML design doc template covers six core areas. Each maps to a distinct category of risk that ML projects face.

Let’s walk through each one in a template-ready format. You can use this as the skeleton for your own team’s template and then customise sections to match your domain. 

# ML Design Document Template

Section 1 — Problem Statement
  • Business context and motivation
  • Current solution (if any) and its limitations
  • Target users and use-case description
  • Success criteria (measurable)

Section 2 — Data Requirements
  • Data sources and ownership
  • Schema, volume, and freshness expectations
  • Labelling strategy and quality constraints
  • Privacy and compliance considerations

Section 3 — Approach
  • Proposed model family / algorithm
  • Feature engineering plan
  • Training pipeline design
  • Baseline comparison

Section 4 — Metrics & Evaluation
  • Offline metrics (precision, recall, AUC, etc.)
  • Online metrics (CTR, revenue, latency)
  • Evaluation methodology (A/B test, shadow mode)
  • Minimum performance thresholds

Section 5 — System Design
  • Serving infrastructure (batch vs real-time)
  • Data pipeline architecture
  • Model registry and versioning
  • Monitoring and alerting

Section 6 — Risks & Mitigations
  • Data drift and model staleness
  • Failure modes and fallback strategies
  • Ethical considerations and bias
  • Cost and resource constraints
Caveat: Don’t treat the template as a form to fill in mechanically. Each section should tell a story that connects back to the problem statement. Reviewers lose trust when sections feel copy-pasted.

Problem Definition

The problem statement is arguably the most important section. A vague problem leads to vague metrics, vague data requirements, and a model that nobody can evaluate objectively. A strong problem statement answers five questions:

  1. What is the user-facing problem we are solving?
  2. Who is affected and how frequently?
  3. Why is a machine learning approach justified over a rule-based or heuristic solution?
  4. What does success look like in quantitative terms?
  5. What are the constraints (latency, cost, fairness)?

✗ Weak Problem Statement

“We want to use ML to improve our recommendation system so that users see better results.”

This tells us nothing about what “better” means, which users are targeted, or what the current baseline is.

✓ Strong Problem Statement

“The homepage ‘For You’ feed currently uses a popularity-based ranking that achieves a 3.2% CTR. We will build a personalised ranking model targeting a ≥15% relative CTR lift (3.7% absolute) for logged-in users on mobile, served within ≤120 ms p99 latency.”

Specific, measurable, time-bounded, and includes a technical constraint.

Tip: Include a “Non-Goals” sub-section. Explicitly listing what the project will not address prevents scope creep and sets clear reviewer expectations. 
# Problem Statement Template

Title: Personalised Homepage Ranking v2
Author: Jane Doe <jane@example.com>
Date: 2024-08-15
Status: Draft | In Review | Approved | Superseded

1. Context
   The homepage feed serves 12M DAU. Current popularity
   ranking ignores user history and session context.

2. Objective
   Increase homepage CTR by ≥15% relative for logged-in
   mobile users within Q4 2024.

3. Justification for ML
   Rule-based personalisation tested in Q2 yielded only
   +4% lift. User interaction patterns are high-dimensional
   and non-linear — a learned model is appropriate.

4. Constraints
   • p99 serving latency ≤ 120 ms
   • Must not degrade CTR for new users (cold-start)
   • GDPR-compliant — no raw PII in features

5. Non-Goals
   • Push notification ranking (separate project)
   • Content generation or augmentation

Data Requirements

Data is the fuel of any ML system, and under-specifying it is the most common reason design docs fail to prevent surprises. A thorough data section should cover sources, schemas, volumes, freshness, and quality expectations.

Data Source Inventory

List every data source the model will consume, including ownership, access method, and update cadence. This prevents the classic failure mode of discovering mid-project that a critical table is owned by another team and lacks an SLA. 

# Data Source Inventory

Source: user_interactions
  Owner       : Analytics team
  Location    : BigQuery → analytics.events.clicks
  Volume      : ~500M rows/day
  Freshness   : Real-time (streaming via Pub/Sub)
  Schema      : user_id (STRING), item_id (STRING),
                action (ENUM: click|save|share),
                timestamp (TIMESTAMP)
  Quality SLA : ≤ 0.1% null rate on user_id

Source: item_metadata
  Owner       : Content Platform team
  Location    : Cloud SQL → content.items
  Volume      : ~2M items (grows ~5K/day)
  Freshness   : Batch (daily snapshot at 02:00 UTC)
  Schema      : item_id (STRING), category (STRING),
                publish_date (DATE), embedding (FLOAT[128])
  Quality SLA : ≤ 1% missing category field

Source: user_profiles
  Owner       : Identity team
  Location    : Spanner → identity.profiles
  Volume      : ~40M users
  Freshness   : Near-real-time (≤ 5 min lag)
  Schema      : user_id (STRING), locale (STRING),
                signup_date (DATE), segment (ENUM)
  Quality SLA : 100% user_id coverage (primary key)

Labelling Strategy

For supervised learning, describe how labels are generated. Are they implicit (e.g., clicks) or explicit (e.g., human annotations)? What is the expected label noise?

✗ Vague Labels

“We will use user feedback as labels.”

What kind of feedback? Clicks? Stars? Time-on-page? Each has different noise characteristics and biases.

✓ Specific Labels

“Positive label: user clicked and spent ≥10s on the destination page. Negative label: item was impressed but not clicked within 30s. Expected positive rate: ~5%.”

Clear signal definition, noise-aware, and includes class distribution.

Caveat: Always document data freshness gaps. If your training data is refreshed daily but the model serves real-time traffic, there is an implicit 24-hour staleness window that can mask distribution shifts.

Privacy and Compliance

Document which fields contain PII, how they will be anonymised or excluded, and which compliance frameworks apply (GDPR, CCPA, HIPAA). This section is often the first thing a legal or privacy reviewer checks.

Approach and Architecture

This section is the technical heart of the document. It describes what model you will train, how the training pipeline works, and where the model fits in the production system.

Model Selection Rationale

Explain why you chose a particular model family. Compare at least two alternatives and state the trade-offs. This prevents the “we used deep learning because it’s trendy” anti-pattern. 

# Model Selection Comparison

Option A: Gradient Boosted Trees (XGBoost)
  Pros  : Fast training, interpretable feature importance,
           low serving latency (~5 ms p99)
  Cons  : Limited ability to capture sequential patterns,
           manual feature engineering required
  Risk  : May plateau at +10% CTR lift

Option B: Two-Tower Neural Retrieval + Ranking
  Pros  : Captures user-item interaction embeddings,
           handles cold-start via content features
  Cons  : Longer training (~8h on 4x A100), harder to debug,
           higher serving latency (~40 ms p99)
  Risk  : Requires GPU serving infrastructure

Recommendation: Option B
  Rationale: The +15% CTR target exceeds what tree models
  achieved in prior experiments. The latency budget (120 ms)
  accommodates neural serving with headroom.

Feature Engineering Plan

List the feature groups, their sources, and the transformation logic. Include feature freshness — a real-time feature computed from streaming data behaves very differently from a daily batch feature.

Tip: Document feature dependencies. If Feature A depends on a table that is populated by Pipeline B, both must be healthy for the model to work. A dependency graph prevents silent failures.

Training Pipeline

Describe the end-to-end training workflow: data extraction, preprocessing, split strategy, hyperparameter search, validation, and model artefact storage. Include estimated training time and resource requirements. 

# Training Pipeline Overview

step 1: extract_data()
  -- Pull last 90 days of interactions from BigQuery
  -- Join with item metadata and user profiles
  -- Output: train.parquet, val.parquet, test.parquet

step 2: preprocess()
  -- Normalise numerical features (z-score)
  -- Encode categoricals (hash embedding)
  -- Generate negative samples (4:1 ratio)

step 3: train_model(epochs=15, lr=3e-4)
  -- Two-tower architecture on 4x A100 GPUs
  -- Mixed-precision training (bf16)
  -- Estimated wall time: ~8 hours

step 4: evaluate()
  -- Compute AUC, NDCG@10, Recall@50 on test set
  -- Compare against baseline (popularity model)
  -- Gate: only promote if AUC > 0.78

step 5: register_model()
  -- Push to MLflow model registry
  -- Tag with experiment ID and metrics
  -- Trigger canary deployment pipeline

Metrics and Evaluation

A common pitfall in ML projects is optimising for an offline metric that does not correlate with business value. The metrics section must bridge model metrics, business metrics, and the evaluation methodology that connects them.

Metric Hierarchy

Offline Metrics

Define the exact metrics you will compute on the held-out test set, including how the test set is constructed (random split, time-based split, or stratified). State the minimum threshold the model must pass to proceed to online evaluation. 

# Offline Evaluation Contract

test_set:
  split    : temporal (last 7 days of training window)
  size     : ~35M interactions
  sampling : no down-sampling (preserve natural distribution)

metrics:
  AUC-ROC       : ≥ 0.78  -- gate: must pass to proceed
  NDCG@10       : ≥ 0.42  -- primary ranking quality metric
  Recall@50     : ≥ 0.60  -- retrieval coverage metric
  Calibration   : ECE ≤ 0.03  -- predicted probs match observed

baseline:
  model    : popularity-based ranker (current production)
  AUC-ROC  : 0.71
  NDCG@10  : 0.35

Online Metrics and A/B Testing

Describe how the model will be evaluated in production. Specify the A/B test design: traffic allocation, minimum detectable effect, test duration, and guardrail metrics.

Primary Metrics

  • CTR — target ≥15% relative lift
  • Session depth — items viewed per session
  • Time-to-first-click — should decrease

Guardrail Metrics

  • Page load time — must not increase >50 ms
  • Error rate — must stay <0.01%
  • Content diversity — Gini index must not drop >5%
Caveat: Never run an A/B test without guardrail metrics. A model can improve CTR by showing clickbait while degrading long-term user retention — guardrails catch this before damage is done.
Tip: Include a minimum test duration even if statistical significance is reached early. Weekly seasonality effects can mislead short tests. A common floor is 14 days.

Review Checklist

A design doc is only as good as its review process. The checklist below helps reviewers systematically evaluate whether the document is complete, internally consistent, and ready for implementation.

For All Reviewers

# Universal Review Checklist

[ ] Problem statement is specific and measurable
[ ] Success criteria are quantified with baselines
[ ] Non-goals are explicitly listed
[ ] Data sources have identified owners and SLAs
[ ] Label definition is unambiguous
[ ] At least two model approaches are compared
[ ] Offline metrics have minimum thresholds
[ ] Online evaluation plan includes guardrail metrics
[ ] Risks section covers data drift and failure modes
[ ] Timeline and resource estimates are included

For ML Engineers

# ML Engineering Review Checklist

[ ] Feature freshness matches serving requirements
[ ] Training/serving skew is addressed
[ ] Model size fits within serving latency budget
[ ] Rollback strategy is documented
[ ] Monitoring covers model inputs, outputs, and performance
[ ] CI/CD pipeline for model retraining is described
[ ] Data pipeline failure modes have fallback logic
[ ] Cost estimates include training and serving

For Product / Business Reviewers

# Product Review Checklist

[ ] Business metric is clearly tied to model metric
[ ] User impact is described (who benefits, who might not)
[ ] Edge cases and failure UX are documented
[ ] A/B test plan has sufficient duration and power
[ ] Ethical risks (bias, fairness) are acknowledged
[ ] Launch criteria are agreed upon by all stakeholders
[ ] Post-launch monitoring plan is in place
Tip: Assign at least one reviewer from each discipline — ML engineer, data engineer, product manager, and SRE. Cross-functional reviews catch blind spots that single-discipline reviews miss.
Caveat: Don’t skip the self-review step. Sending a first draft directly to reviewers wastes their time on superficial issues and erodes trust in the design doc process.

Example Structure

Below is a complete outline of an ML design document for a real-world project. Use this as a reference when writing your own. Each bullet is a heading or sub-section you should include. 

# ======================================================
# ML DESIGN DOCUMENT
# Project: Personalised Homepage Ranking v2
# Author:  Jane Doe <jane@example.com>
# Date:    2024-08-15    Status: In Review
# ======================================================

1. PROBLEM STATEMENT
   1.1 Business Context
       -- 12M DAU homepage, popularity ranking, 3.2% CTR
   1.2 Objective
       -- ≥15% relative CTR lift for logged-in mobile users
   1.3 Justification for ML
       -- Rule-based approach capped at +4%
   1.4 Constraints
       -- p99 ≤ 120 ms, GDPR-compliant, no cold-start regression
   1.5 Non-Goals
       -- Push notifications, content generation

2. DATA REQUIREMENTS
   2.1 Data Sources
       -- user_interactions, item_metadata, user_profiles
   2.2 Schema Definitions
       -- Full schemas with types, nullability, SLAs
   2.3 Labelling Strategy
       -- Click + ≥10s dwell = positive, 4:1 neg sampling
   2.4 Volume and Freshness
       -- 500M events/day, 90-day training window
   2.5 Privacy and Compliance
       -- No raw PII in features, GDPR Art.22 review

3. APPROACH
   3.1 Model Selection
       -- Two-tower neural vs. XGBoost comparison
   3.2 Feature Engineering
       -- 3 feature groups: user, item, contextual
   3.3 Training Pipeline
       -- 5-step pipeline: extract, preprocess, train,
       --   evaluate, register
   3.4 Baseline
       -- Popularity ranker (current prod) as baseline

4. METRICS & EVALUATION
   4.1 Offline Metrics
       -- AUC ≥ 0.78, NDCG@10 ≥ 0.42, Recall@50 ≥ 0.60
   4.2 Online Metrics
       -- CTR, session depth, time-to-first-click
   4.3 Guardrail Metrics
       -- Latency, error rate, content diversity (Gini)
   4.4 A/B Test Plan
       -- 5% traffic, 14-day minimum, MDE = 10% relative

5. SYSTEM DESIGN
   5.1 Serving Architecture
       -- gRPC model server behind feature gateway
   5.2 Data Pipelines
       -- Streaming (Pub/Sub) + daily batch (Dataflow)
   5.3 Model Registry
       -- MLflow with auto-tagging from CI/CD
   5.4 Monitoring & Alerting
       -- Feature drift (PSI), prediction drift, latency
   5.5 Rollback Plan
       -- Automatic rollback if error rate > 0.01%

6. RISKS & MITIGATIONS
   6.1 Data Drift
       -- Weekly PSI monitoring, auto-retrain trigger
   6.2 Cold-Start Users
       -- Fallback to popularity for users < 5 interactions
   6.3 Feedback Loops
       -- Exploration budget (10% random diversification)
   6.4 Ethical Risks
       -- Fairness audit across demographic segments
   6.5 Cost
       -- Training: ~$200/run, Serving: ~$3K/month (GPU)

7. TIMELINE
   -- Week 1-2: Data pipeline and feature engineering
   -- Week 3-4: Model training and offline evaluation
   -- Week 5:   Shadow mode deployment
   -- Week 6-7: A/B test (14 days)
   -- Week 8:   Full rollout or iterate

8. OPEN QUESTIONS
   -- Should we include video items in v2 or defer to v3?
   -- GPU serving budget approved? (pending infra review)

9. APPENDIX
   -- Feature list with transformations (linked spreadsheet)
   -- Prior experiment results (linked doc)
   -- Architecture diagram (linked Figma)
Tip: Keep the main document to 5–8 pages. Use an appendix or linked documents for detailed feature lists, experiment logs, and architecture diagrams. Reviewers are more likely to read a focused document thoroughly.

Adapting the Template

Not every project needs every section at the same depth. Use this guide to scale the template based on project complexity:

Small Project (<2 weeks)

  • Problem statement (1 paragraph)
  • Data source + label definition
  • Approach (1 model, no comparison needed)
  • Offline metric + threshold
  • Risks (top 2–3)

Total: 1–2 pages

Large Project (>1 month)

  • Full problem statement with non-goals
  • Complete data inventory with SLAs
  • Model comparison (2–3 options)
  • Full metric hierarchy + A/B plan
  • System design with architecture diagrams
  • Detailed risks + mitigations
  • Timeline + resource plan

Total: 5–8 pages + appendix

Caveat: Even for small projects, never skip the problem statement and metrics sections. These two sections prevent the most common failure mode: building a technically sound model that solves the wrong problem.

Writing ML design documents is a skill that improves with practice. Start with this template, adapt it to your team’s needs, and iterate after each review cycle. The goal is not bureaucracy — it is shared understanding that accelerates the entire ML lifecycle.
