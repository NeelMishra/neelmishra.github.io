Real-Time Feature Computation

MLOps Series Feature Stores

Why Real-Time Features?

Batch feature pipelines compute values on a schedule—hourly, daily, or weekly. But many ML use cases demand features derived from events that happened seconds ago. When a credit-card transaction arrives, the fraud model needs to know "how many transactions has this card made in the last five minutes?" That number changes with every swipe and cannot wait for tomorrow’s batch run.

Key Use Cases

  • Fraud detection – aggregate transaction velocity, geo-hop distance, and merchant category shifts within sliding windows of 1–15 minutes.
  • Personalised recommendations – last-clicked category, session dwell time, and cart-add count feed ranking models that update per request.
  • Dynamic pricing – supply/demand ratios computed over the last 30 seconds drive surge-pricing algorithms for ride-hailing and e-commerce flash sales.
  • Anomaly detection in IoT – rolling standard deviation of sensor readings flags outliers before equipment fails.
Latency budget rule of thumb: If the model’s prediction must be served within 100 ms of the triggering event, any feature older than that window is stale and should be computed in real time.

Latency Requirements by Domain

Windowed Aggregations

Stream processing frameworks partition infinite event streams into finite chunks called windows. Choosing the right window type directly affects feature freshness, memory footprint, and correctness.

Tumbling Windows

Fixed-size, non-overlapping intervals. Every event belongs to exactly one window. Use when you need periodic snapshots such as "transactions per minute."

Sliding Windows

Overlapping intervals defined by a size and a slide. A 60-second window with a 10-second slide produces six overlapping windows at any moment. Ideal for smoothed features like "average transaction amount over the last 60 s, updated every 10 s."

Session Windows

Variable-length windows driven by a gap parameter. A session closes when no events arrive within the gap. Perfect for user-session features where activity bursts are separated by idle periods.

Python Implementation

from collections import defaultdict
import time

class TumblingWindow:
    # Fixed-size, non-overlapping window aggregator
    def __init__(self, size_seconds):
        self.size = size_seconds
        self.buckets = defaultdict(list)

    def add_event(self, timestamp, value):
        bucket_key = int(timestamp // self.size)
        self.buckets[bucket_key].append(value)

    def get_aggregate(self, timestamp):
        bucket_key = int(timestamp // self.size)
        vals = self.buckets.get(bucket_key, [])
        if not vals:
            return None
        return {
            "count": len(vals),
            "sum": sum(vals),
            "avg": sum(vals) / len(vals),
        }


class SlidingWindow:
    # Overlapping window with configurable slide interval
    def __init__(self, size_seconds, slide_seconds):
        self.size = size_seconds
        self.slide = slide_seconds
        self.events = []

    def add_event(self, timestamp, value):
        self.events.append((timestamp, value))

    def get_windows(self, current_time):
        # Evict events older than the largest possible window
        cutoff = current_time - self.size
        self.events = [(t, v) for t, v in self.events if t > cutoff]
        window_start = current_time - self.size
        return [v for t, v in self.events if t >= window_start]
Memory warning: Sliding windows with a small slide and large size can explode memory. Always set a maximum event count or use approximate data structures like HyperLogLog for count-distinct features.

Apache Flink for Features

Apache Flink is the gold standard for stateful stream processing. Its event-time semantics, exactly-once guarantees, and low latency make it the top choice for mission-critical real-time features.

Architecture Overview

Watermarks and Event-Time

Flink uses watermarks to track progress in event time. A watermark W(t) asserts that no event with timestamp ≤ t will arrive. This lets Flink trigger window computations even when events arrive out of order, by allowing a bounded lateness before closing a window.

Python (PyFlink) Example

from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream.window import TumblingEventTimeWindows
from pyflink.common.time import Time
from pyflink.common.watermark_strategy import WatermarkStrategy

import datetime

def build_flink_feature_pipeline():
    env = StreamExecutionEnvironment.get_execution_environment()
    env.set_parallelism(4)

    # Define watermark strategy with 5-second tolerance for late data
    watermark = WatermarkStrategy \
        .for_bounded_out_of_orderness(Time.seconds(5))

    # Read from Kafka topic
    txn_stream = env \
        .from_source(kafka_source, watermark, "transactions")

    # Key by card_id and apply 60-second tumbling window
    features = txn_stream \
        .key_by(lambda e: e["card_id"]) \
        .window(TumblingEventTimeWindows.of(Time.seconds(60))) \
        .aggregate(TransactionAggregator())

    # Write feature vectors to Redis for online serving
    features.add_sink(RedisSink(host="redis-cluster", ttl=300))
    env.execute("realtime-fraud-features")

Flink SQL for Aggregations

-- Tumbling window aggregation using Flink SQL
SELECT
    card_id,
    TUMBLE_START(event_time, INTERVAL '1' MINUTE) AS window_start,
    COUNT(*) AS txn_count_1m,
    SUM(amount) AS txn_sum_1m,
    AVG(amount) AS txn_avg_1m,
    MAX(amount) AS txn_max_1m
FROM transactions
GROUP BY
    card_id,
    TUMBLE(event_time, INTERVAL '1' MINUTE)
Tip: Flink SQL is the fastest path to production for teams already comfortable with SQL. It compiles to the same optimised dataflow graph as the DataStream API, with no performance penalty.

Spark Structured Streaming

Spark Structured Streaming extends the familiar DataFrame API with continuous, micro-batch execution. It trades single-digit-millisecond latency for ecosystem breadth: the same code runs on batch and streaming data with minimal changes.

Micro-Batch Model

Spark polls the source at a configurable trigger interval (default 100 ms). Each micro-batch is a small DataFrame processed with full Catalyst optimisations. The model simplifies exactly-once semantics because each micro-batch is an atomic transaction against the sink.

Stateful Processing

Structured Streaming maintains state across micro-batches using RocksDB-backed state stores. This enables windowed aggregations, session windows (via applyInPandasWithState), and arbitrary stateful transformations.

from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.window import Window

def build_spark_feature_stream():
    spark = SparkSession.builder \
        .appName("realtime-features") \
        .getOrCreate()

    # Read from Kafka as a streaming DataFrame
    raw = spark.readStream \
        .format("kafka") \
        .option("kafka.bootstrap.servers", "broker:9092") \
        .option("subscribe", "transactions") \
        .load()

    # Parse JSON payload and cast event_time
    txns = raw.select(
        F.from_json(F.col("value").cast("string"), schema).alias("data")
    ).select("data.*") \
     .withColumn("event_time", F.col("event_time").cast("timestamp"))

    # Windowed aggregation: 1-minute tumbling window
    features = txns \
        .withWatermark("event_time", "10 seconds") \
        .groupBy(
            F.col("card_id"),
            F.window("event_time", "1 minute")
        ).agg(
            F.count("*").alias("txn_count"),
            F.sum("amount").alias("txn_sum"),
            F.avg("amount").alias("txn_avg"),
        )

    # Write features to a Delta table for both serving and offline use
    query = features.writeStream \
        .format("delta") \
        .outputMode("update") \
        .option("checkpointLocation", "/checkpoints/txn-features") \
        .trigger(processingTime="10 seconds") \
        .start("/features/txn_window")

    query.awaitTermination()

build_spark_feature_stream()
Watermark gotcha: If you set the watermark tolerance too low, late events are silently dropped and your features become inaccurate. Start with a generous tolerance (e.g., 30 s) and tighten it only after profiling real event delays.

Common Real-Time Feature Patterns

Below are battle-tested patterns that appear across fraud, recommendations, and operational ML systems. Each pattern includes a self-contained Python implementation suitable for embedding inside a Flink or Spark UDF.

Running Average (Exponentially Weighted)

class ExponentialMovingAverage:
    # Streaming EMA that processes one event at a time
    def __init__(self, alpha=0.3):
        self.alpha = alpha
        self.value = None

    def update(self, new_value):
        if self.value is None:
            self.value = new_value
        else:
            self.value = self.alpha * new_value + (1 - self.alpha) * self.value
        return self.value

Count Distinct (HyperLogLog Sketch)

import hashlib, math

class HyperLogLogSketch:
    # Approximate count-distinct using HyperLogLog
    def __init__(self, precision=14):
        self.p = precision
        self.m = 1 << precision
        self.registers = [0] * self.m

    def _hash(self, item):
        h = hashlib.sha256(str(item).encode()).hexdigest()
        return int(h, 16)

    def add(self, item):
        x = self._hash(item)
        idx = x & (self.m - 1)
        remaining = x >> self.p
        self.registers[idx] = max(
            self.registers[idx],
            self._leading_zeros(remaining) + 1
        )

    def _leading_zeros(self, value):
        if value == 0:
            return 64 - self.p
        return (64 - self.p) - value.bit_length()

    def estimate(self):
        alpha = 0.7213 / (1 + 1.079 / self.m)
        raw = alpha * self.m ** 2 / sum(
            2.0 ** (-r) for r in self.registers
        )
        return int(raw)

Last-N Events Buffer

from collections import deque

class LastNBuffer:
    # Maintains the most recent N events per key
    def __init__(self, n=10):
        self.n = n
        self.buffer = deque(maxlen=n)

    def add(self, event):
        self.buffer.append(event)

    def get_features(self):
        vals = list(self.buffer)
        if not vals:
            return {}
        return {
            "last_n_mean": sum(vals) / len(vals),
            "last_n_min": min(vals),
            "last_n_max": max(vals),
            "last_n_count": len(vals),
        }

Challenges and Solutions

Real-time feature pipelines introduce distributed-systems complexities that batch jobs can largely ignore. The three biggest headaches are late-arriving data, event ordering, and exactly-once delivery.

Late Data

Events may arrive after the window they belong to has already closed. Solutions include watermarks with bounded lateness (Flink), watermarks in Spark, and side outputs for late events that need special handling.

Event Ordering

Network partitions and multi-producer topics mean events arrive out of order. Event-time processing (rather than processing-time) ensures features are computed as if events arrived in order, at the cost of buffering and delayed results.

Exactly-Once Semantics

Duplicate feature writes cause subtle bugs. Flink achieves exactly-once through distributed snapshots (Chandy-Lamport algorithm). Spark uses micro-batch transactional writes. Both require idempotent sinks to guarantee end-to-end correctness.

Pro tip: Always make your feature sink idempotent. Use Redis SET (not INCR) and database UPSERT (not INSERT) so that replays don’t corrupt feature values.

Framework Comparison

Apache Flink

  • True event-time processing with watermarks
  • Exactly-once via distributed snapshots
  • Sub-second latency per event
  • Rich state management (RocksDB, heap)
  • Steeper learning curve
  • Best for: latency-critical, stateful features

Spark Structured Streaming

  • Micro-batch model (100 ms+ latency)
  • Exactly-once via transactional micro-batches
  • Familiar DataFrame/SQL API
  • Unified batch + streaming codebase
  • Higher tail latency under load
  • Best for: teams already on Spark, moderate latency

Kafka Streams

  • Library, not a cluster—deploys as a JAR
  • Exactly-once with Kafka transactions
  • No external dependencies beyond Kafka
  • Limited to Kafka as source and sink
  • Best for: Kafka-native, simple aggregations

Materialize / Streaming SQL

  • Standard SQL over streaming data
  • Incrementally maintained materialised views
  • Low operational overhead
  • Emerging ecosystem, fewer integrations
  • Best for: SQL-first teams, simple transforms
Don’t over-engineer: If your latency budget is > 1 second and you already run Spark, Structured Streaming is simpler to operate than Flink. Choose the framework that matches your team’s skills and your latency SLA.
