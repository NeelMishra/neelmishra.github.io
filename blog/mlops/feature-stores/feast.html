Feast: Open Source Feature Store

MLOps Series Feature Stores

Feast Overview

Feast (Feature Store) is an open-source feature store originally created at Gojek in 2018 and later co-developed with Google Cloud. It is now a standalone open-source project under the Linux Foundation (LF AI & Data), with an active community and a growing ecosystem of integrations.

Feast solves the operational complexity of managing ML features by providing:

  • A Python SDK for defining, registering, and retrieving features
  • Point-in-time correct joins for training data, preventing data leakage
  • Online serving with low-latency reads from key-value stores
  • Offline storage integration with data warehouses and file systems
  • A feature registry that catalogs all features and their metadata
  • Provider abstraction so you can run locally, on AWS, GCP, or custom infra
Why Feast? Unlike managed feature stores (Tecton, Databricks Feature Store, SageMaker Feature Store), Feast is fully open-source, vendor-neutral, and can run anywhere—from a laptop to a multi-cloud production deployment.

Feast Architecture

Feast follows a clean, modular architecture with four core components. The SVG below illustrates how they fit together.

The Feature Registry is the central catalog—a serialized protobuf (or a database) that tracks every entity, feature view, and data source. The Offline Store holds historical feature data for training. The Online Store serves the latest feature values for real-time inference. The Provider is a pluggable abstraction that adapts Feast to different infrastructure environments.

Registry Storage: By default, Feast stores the registry as a serialized protobuf file (e.g., in S3 or GCS). For multi-team setups, you can use a SQL-backed registry for better concurrency.

Entities and Feature Views

In Feast, you model your features using three core abstractions: Entities, Feature Views, and Data Sources.

Entities

An entity represents a real-world object that features describe—a user, a driver, a product. Entities define the join keys used to look up features. 

from feast import Entity

# Define a user entity with user_id as the join key
user = Entity(
    name="user",
    join_keys=["user_id"],
    description="A registered platform user",
)

Data Sources

A data source tells Feast where to read raw feature data from. Feast supports file sources (Parquet), BigQuery, Redshift, Snowflake, Spark, and more. 

from feast import FileSource

# Parquet file as a data source
user_source = FileSource(
    path="data/user_features.parquet",
    timestamp_field="event_timestamp",
    created_timestamp_column="created_timestamp",
)

Feature Views

A feature view groups related features together, links them to an entity and data source, and defines their schema. Think of it as a table definition in a database. 

from feast import FeatureView, Field
from feast.types import Float64, Int64
from datetime import timedelta

user_activity_fv = FeatureView(
    name="user_activity",
    entities=[user],
    schema=[
        Field(name="login_count_7d", dtype=Int64),
        Field(name="avg_session_duration", dtype=Float64),
        Field(name="pages_viewed_30d", dtype=Int64),
        Field(name="days_since_last_purchase", dtype=Int64),
    ],
    source=user_source,
    ttl=timedelta(days=1),   # features older than 1 day are stale
)

user_spending_fv = FeatureView(
    name="user_spending",
    entities=[user],
    schema=[
        Field(name="total_spend_30d", dtype=Float64),
        Field(name="avg_order_value", dtype=Float64),
        Field(name="transaction_count_30d", dtype=Int64),
    ],
    source=spending_source,
    ttl=timedelta(days=1),
)
Common Pitfall: Forgetting to set an appropriate ttl (time-to-live). Without a TTL, Feast will serve arbitrarily old feature values. Set it to match your feature freshness requirements—e.g., timedelta(hours=6) for features that should be refreshed multiple times a day.

Registering with Feast

Once definitions are in a Python file (typically feature_repo/features.py), apply them to the registry: 

# Terminal command to apply feature definitions
# $ feast apply

# Or programmatically:
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")
store.apply([user, user_activity_fv, user_spending_fv])

Materializing Features

Materialization is the process of copying feature values from the offline store to the online store so they can be served at low latency. This is typically run on a schedule (e.g., hourly or daily) via a cron job or orchestrator.

CLI Materialization

# Materialize features from offline → online store
# Terminal command:
# $ feast materialize 2024-01-01T00:00:00 2024-03-15T23:59:59

# Incremental materialization (since last run):
# $ feast materialize-incremental 2024-03-15T23:59:59

Programmatic Materialization

from feast import FeatureStore
from datetime import datetime

store = FeatureStore(repo_path="feature_repo/")

# Full materialization for a date range
store.materialize(
    start_date=datetime(2024, 1, 1),
    end_date=datetime(2024, 3, 15),
)

# Incremental — only new data since last materialization
store.materialize_incremental(
    end_date=datetime(2024, 3, 15),
)
Tip: Use materialize_incremental in production cron jobs. It automatically tracks the last materialization timestamp and only processes new data, making it much faster than full materialization.

What Happens During Materialization

Feast reads from the offline store (e.g., a Parquet file or BigQuery table), selects the latest feature values per entity key, and writes them to the online store (e.g., Redis or DynamoDB). Only the most recent row per entity is kept in the online store.

Pitfall: If your offline data source doesn’t include an event_timestamp column, Feast cannot determine feature freshness. Always ensure your source data has proper timestamps.

Feature Retrieval

Feast provides two primary retrieval APIs: get_historical_features for training and get_online_features for serving. Both use the same feature definitions, guaranteeing consistency.

Historical Features (Training)

The historical retrieval API performs a point-in-time correct join: for each entity row and timestamp, it retrieves the feature values that were known at that point in time, preventing future data from leaking into training examples. 

from feast import FeatureStore
import pandas as pd

store = FeatureStore(repo_path="feature_repo/")

# Entity dataframe: who + when
entity_df = pd.DataFrame({
    "user_id": [1001, 1002, 1003, 1001],
    "event_timestamp": pd.to_datetime([
        "2024-03-01", "2024-03-05",
        "2024-03-10", "2024-03-12"
    ]),
})

# Get point-in-time correct features
training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "user_activity:login_count_7d",
        "user_activity:avg_session_duration",
        "user_spending:total_spend_30d",
        "user_spending:avg_order_value",
    ],
).to_df()

print(training_df.head())

Online Features (Serving)

The online API fetches the latest feature values from the online store for a given set of entities. It’s designed for low-latency real-time inference. 

from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

# Fetch latest features for specific users
online_features = store.get_online_features(
    features=[
        "user_activity:login_count_7d",
        "user_activity:avg_session_duration",
        "user_spending:total_spend_30d",
    ],
    entity_rows=[
        {"user_id": 1001},
        {"user_id": 1002},
    ],
).to_dict()

print(online_features)
# {"user_id": [1001, 1002], "login_count_7d": [15, 8], ...}
Performance Tip: Batch your online feature requests. Fetching features for 100 entities in one call is much faster than making 100 individual calls, because Feast can pipeline the key-value lookups.

Deployment Patterns

Feast supports multiple deployment patterns depending on your infrastructure and scale. The feature_store.yaml configuration file controls which provider, offline store, and online store to use.

Local Development

# feature_store.yaml — local setup
# project: my_project
# registry: data/registry.db
# provider: local
# online_store:
#   type: sqlite
#   path: data/online_store.db
# offline_store:
#   type: file

AWS Deployment

# feature_store.yaml — AWS setup
# project: my_project
# registry: s3://my-bucket/feast/registry.pb
# provider: aws
# online_store:
#   type: dynamodb
#   region: us-east-1
# offline_store:
#   type: redshift
#   cluster_id: my-redshift-cluster
#   region: us-east-1
#   database: featurestore
#   user: admin

GCP Deployment

# feature_store.yaml — GCP setup
# project: my_project
# registry: gs://my-bucket/feast/registry.pb
# provider: gcp
# online_store:
#   type: datastore
# offline_store:
#   type: bigquery

Deployment Comparison

Local / SQLite

  • Online: SQLite
  • Offline: File (Parquet)
  • Best for: Development, prototyping
  • Latency: ~1–5ms
  • Cost: Free

AWS

  • Online: DynamoDB
  • Offline: Redshift / S3
  • Best for: Production on AWS
  • Latency: ~5–15ms
  • Cost: Pay-per-use

GCP

  • Online: Datastore / Bigtable
  • Offline: BigQuery
  • Best for: Production on GCP
  • Latency: ~5–20ms
  • Cost: Pay-per-use

Redis (Custom)

  • Online: Redis Cluster
  • Offline: Any supported
  • Best for: Ultra-low latency serving
  • Latency: ~1–3ms
  • Cost: Redis hosting costs
Getting Started: The fastest way to try Feast is with the local provider. Install with pip install feast, run feast init my_project, then cd my_project/feature_repo && feast apply. You’ll have a working feature store in under 5 minutes.
Production Pitfall: The default file-based registry doesn’t support concurrent writes well. For production multi-team setups, switch to a SQL-backed registry (Postgres or MySQL) by setting registry_type: sql in your config.

Feast provides a pragmatic, open-source path to production-grade feature management. Start local, prove the value, then scale to cloud providers as your ML platform matures.
