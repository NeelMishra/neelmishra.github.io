Feature Discovery and Reuse

MLOps Series Feature Stores

The Feature Discovery Problem

In most organizations, feature engineering knowledge lives inside individual teams’ heads, scattered notebooks, and undocumented pipelines. When a new ML project starts, engineers frequently re-create features that already exist elsewhere—wasting weeks of effort and introducing inconsistencies.

The Hidden Cost of Duplication

Feature duplication is more damaging than it appears on the surface:

  • Wasted engineering time: Teams spend 2–4 weeks building features that another team already built and validated. Across a 10-team organization, this can amount to months of duplicated effort per year.
  • Inconsistent definitions: Two teams computing “user lifetime value” may use different time windows, discount rates, or data sources—producing subtly different values that lead to conflicting model behavior.
  • No institutional memory: When an engineer leaves, their feature engineering knowledge leaves with them. There is no searchable record of what was built, why, or how.
Warning: A large tech company audit found that 40% of their features were duplicates with slightly different implementations. Consolidating these into a shared catalog reduced feature engineering costs by 30% and improved model consistency across products.

Feature Catalogs

A feature catalog is a centralized registry that stores metadata about every feature in the organization. It serves as the “single source of truth” for what features exist, who owns them, how they are computed, and how fresh they are.

Essential Metadata

A well-designed catalog entry should include:

  • Name and description: Human-readable name and a clear explanation of what the feature represents.
  • Owner: The team or individual responsible for maintaining the feature.
  • Tags: Categorical labels for searchability (e.g., “fraud”, “user-profile”, “real-time”).
  • Lineage: The data source(s), transformation logic, and pipeline that produce the feature.
  • Schema: Data type, constraints, and validation rules.
  • Freshness SLA: How often the feature is updated and its maximum acceptable staleness.
  • Usage statistics: Which models consume this feature and how often it is retrieved.
from dataclasses import dataclass, field
from typing import List, Dict, Optional
from datetime import datetime

@dataclass
class FeatureCatalogEntry:
    name: str
    description: str
    owner: str
    tags: List[str]
    dtype: str
    source_table: str
    transformation: str
    freshness_sla: str
    created_at: datetime
    updated_at: datetime
    consumers: List[str] = field(default_factory=list)
    quality_score: float = 0.0
    deprecated: bool = False

class FeatureCatalog:
    def __init__(self):
        self.entries: Dict[str, FeatureCatalogEntry] = {}

    def register(self, entry: FeatureCatalogEntry):
        # Validate entry before registration
        if not entry.name or not entry.description:
            raise ValueError("Name and description are required")
        if entry.name in self.entries:
            print(f"Updating existing entry: {entry.name}")
        self.entries[entry.name] = entry
        print(f"Registered feature: {entry.name}")

    def get(self, name: str) -> Optional[FeatureCatalogEntry]:
        return self.entries.get(name)

    def list_all(self) -> List[FeatureCatalogEntry]:
        return list(self.entries.values())

# Register a feature
catalog = FeatureCatalog()
catalog.register(FeatureCatalogEntry(
    name="user_lifetime_value",
    description="Total revenue from user over account lifetime, "
                "discounted at 5% annual rate",
    owner="growth-ml-team",
    tags=["user-profile", "revenue", "batch"],
    dtype="float",
    source_table="analytics.user_transactions",
    transformation="SUM(amount * discount_factor) GROUP BY user_id",
    freshness_sla="24 hours",
    created_at=datetime(2024, 1, 15),
    updated_at=datetime(2024, 6, 1),
    consumers=["churn_model_v3", "upsell_recommender"],
    quality_score=0.95,
))
Tip: Require feature registration as part of the feature engineering workflow. If a feature is not in the catalog, it should not be deployed to production. This ensures the catalog stays complete and up-to-date.

Searching for Features

A catalog is only useful if engineers can efficiently find what they need. Feature search should support multiple modalities to accommodate different search behaviors.

Search Modalities

  • Keyword search: Match against feature names and descriptions using full-text search (e.g., “transaction velocity”).
  • Tag-based filtering: Filter by categorical tags like “fraud”, “real-time”, or “user-profile”.
  • Semantic search: Use embedding similarity to find features related to a concept, even if the exact words differ (e.g., searching “customer spend” finds “user_lifetime_value”).
  • Usage-based ranking: Surface popular features first, as widely-used features are more likely to be well-maintained and trustworthy.
from typing import List, Optional

class FeatureSearch:
    def __init__(self, catalog: FeatureCatalog):
        self.catalog = catalog

    def search_by_keyword(self, query: str) -> List[FeatureCatalogEntry]:
        query_lower = query.lower()
        results = []
        for entry in self.catalog.list_all():
            if (query_lower in entry.name.lower()
                    or query_lower in entry.description.lower()):
                results.append(entry)
        return results

    def search_by_tags(self, tags: List[str]) -> List[FeatureCatalogEntry]:
        tag_set = set(tags)
        results = []
        for entry in self.catalog.list_all():
            if tag_set.intersection(set(entry.tags)):
                results.append(entry)
        # Sort by number of matching tags (most relevant first)
        results.sort(
            key=lambda e: len(tag_set.intersection(set(e.tags))),
            reverse=True,
        )
        return results

    def search_by_popularity(self, top_n: int = 10) -> List[FeatureCatalogEntry]:
        all_entries = self.catalog.list_all()
        all_entries.sort(
            key=lambda e: len(e.consumers), reverse=True
        )
        return all_entries[:top_n]

# Usage
search = FeatureSearch(catalog)
results = search.search_by_keyword("lifetime value")
for r in results:
    print(f"{r.name}: {r.description[:60]}...")

Sharing Features Across Teams

Building a catalog is step one. The harder challenge is making features truly shareable across team boundaries with proper access control, documentation, and versioning.

Access Control

Not every team should access every feature. PII features, revenue data, and experimental features all require different access policies:

  • Public features: Available to all ML teams (e.g., product category, device type).
  • Team-scoped features: Visible only to the owning team and explicitly granted consumers.
  • PII-restricted features: Require data privacy review before granting access; usage is logged for compliance.
from enum import Enum
from typing import Set, Dict

class AccessLevel(Enum):
    PUBLIC = "public"
    TEAM = "team"
    RESTRICTED = "restricted"

class FeatureAccessControl:
    def __init__(self):
        self.policies: Dict[str, dict] = {}

    def set_policy(self, feature_name: str, level: AccessLevel,
                    allowed_teams: Set[str] = None):
        self.policies[feature_name] = {
            "level": level,
            "allowed_teams": allowed_teams or set(),
        }

    def check_access(self, feature_name: str, team: str) -> bool:
        policy = self.policies.get(feature_name)
        if policy is None:
            return False  # deny by default

        if policy["level"] == AccessLevel.PUBLIC:
            return True
        elif policy["level"] == AccessLevel.TEAM:
            return team in policy["allowed_teams"]
        elif policy["level"] == AccessLevel.RESTRICTED:
            # Restricted features require explicit approval
            return team in policy["allowed_teams"]
        return False

    def grant_access(self, feature_name: str, team: str):
        if feature_name in self.policies:
            self.policies[feature_name]["allowed_teams"].add(team)
            print(f"Granted {team} access to {feature_name}")

# Setup access control
acl = FeatureAccessControl()
acl.set_policy("device_type", AccessLevel.PUBLIC)
acl.set_policy("user_lifetime_value", AccessLevel.TEAM,
               {"growth-ml-team", "finance-analytics"})
acl.set_policy("user_ssn_hash", AccessLevel.RESTRICTED,
               {"fraud-team"})

# Check access
print(acl.check_access("device_type", "any-team"))        # True
print(acl.check_access("user_lifetime_value", "ads-team")) # False
print(acl.check_access("user_ssn_hash", "fraud-team"))     # True

Documentation Standards

Every shared feature should include documentation covering: the business definition, computation logic, known caveats, example values, and recommended use cases. Treat feature documentation like API documentation—it is the contract between producer and consumer.

Warning: Features without documentation become “mystery features” that teams are afraid to modify or deprecate. Undocumented features accumulate technical debt and eventually become liabilities rather than assets.

Feature Governance

Feature governance establishes the rules, processes, and accountability structures that keep a feature catalog healthy over time. Without governance, catalogs degrade into stale, unreliable registries.

Ownership Model

Every feature must have a clearly designated owner—either a team or an individual. Ownership carries responsibilities:

  • Maintaining the feature pipeline and ensuring freshness SLAs are met.
  • Responding to consumer questions and bug reports within defined SLAs.
  • Reviewing and approving schema changes before deployment.
  • Updating documentation when the feature logic changes.

Deprecation Policies

Features that are no longer maintained or relevant should be formally deprecated rather than silently abandoned. A healthy deprecation workflow includes:

  1. Announce deprecation: Mark the feature as deprecated in the catalog and notify all consumers.
  2. Deprecation window: Give consumers 2–4 weeks to migrate to an alternative feature.
  3. Sunset: After the window, stop computing the feature. Serve a default value or null with a warning to any remaining consumers.
  4. Archive: Retain historical data for reproducibility, but remove the feature from the active catalog.
Best practice: Run automated scans to identify features with zero consumers over the last 90 days. These are candidates for deprecation. Features that are consumed but never updated are candidates for ownership reassignment.

Quality Scores

Assign each feature a computed quality score (0–1) based on objective metrics:

  • Freshness compliance: Percentage of time the feature met its freshness SLA over the last 30 days.
  • Null rate: Fraction of entity lookups that returned null (lower is better).
  • Documentation completeness: Whether all required metadata fields are filled.
  • Consumer satisfaction: Optional—survey-based scores from teams using the feature.
Warning: Do not let quality scores become vanity metrics. Tie them to actionable consequences: features below a 0.5 quality score should be flagged for review, and features below 0.3 should be candidates for deprecation.

Building a Feature Catalog

When implementing a feature catalog, teams face a fundamental choice: build a custom solution or adopt an existing platform. Both approaches have merits depending on organizational size and requirements.

🔨 Build Your Own

Create a custom catalog service backed by a metadata database (PostgreSQL), a search index (Elasticsearch), and a lightweight UI.

  • Pros: Full customization, integrates with internal tooling, no vendor lock-in
  • Cons: 3–6 month build time, ongoing maintenance, requires dedicated team
  • Best for: Large orgs (50+ ML engineers) with unique requirements

🛒 Buy / Adopt OSS

Use an existing platform like Feast, Tecton, or Hopsworks that includes built-in catalog capabilities.

  • Pros: Fast to deploy (days not months), community-maintained, battle-tested
  • Cons: Less customizable, potential vendor lock-in, may not fit all workflows
  • Best for: Small-to-medium orgs (5–30 ML engineers) getting started

Implementation Roadmap

Regardless of the build-vs-buy decision, follow this phased approach:

  1. Phase 1 – Registry (Week 1–2): Implement basic registration and lookup. Every feature must have a name, description, owner, and type.
  2. Phase 2 – Search (Week 3–4): Add full-text search and tag-based filtering. Engineers should be able to find relevant features in under 30 seconds.
  3. Phase 3 – Lineage (Week 5–8): Track data sources and transformation logic. Enable impact analysis for upstream changes.
  4. Phase 4 – Governance (Week 9–12): Add quality scores, deprecation workflows, and access control. Automate staleness detection.

🎯 Quick Start Stack

For teams wanting to move fast with minimal infrastructure:

  • Feast (open source feature store with registry)
  • GitHub/GitLab for schema-as-code
  • Simple Python CLI for registration
  • Markdown docs in repo alongside feature definitions

🏗️ Enterprise Stack

For organizations with mature ML infrastructure:

  • Tecton or Hopsworks (managed feature platform)
  • Elasticsearch for rich search and discovery
  • Custom UI integrated with internal ML platform
  • DataHub or Amundsen for cross-system lineage
Key takeaway: The best feature catalog is the one your team actually uses. Start simple—even a well-organized spreadsheet is better than nothing. Invest in tooling incrementally as adoption grows and pain points become clear.
