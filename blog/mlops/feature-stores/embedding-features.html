Embedding Features: Storing and Serving Vectors

MLOps Series Feature Stores

What Are Embedding Features?

Embeddings are dense, fixed-length vector representations that capture the semantic meaning of entities — users, products, words, images, or any discrete object. Unlike one-hot encodings that are sparse and high-dimensional, embeddings compress information into a compact numerical form where similar entities occupy nearby positions in the vector space.

Common Embedding Types

  • Text Embeddings — sentence-transformers, OpenAI embeddings, BERT (dimension: 384–1536)
  • User Embeddings — learned from collaborative filtering or interaction history (dimension: 64–256)
  • Product Embeddings — learned from purchase sequences or product metadata (dimension: 64–256)
  • Image Embeddings — extracted from CNNs or Vision Transformers (dimension: 512–2048)
Key Insight: Embeddings turn the similarity problem into a distance problem. If two products have similar embeddings, they are semantically similar — enabling nearest-neighbor recommendation, search, and clustering.

Storage Strategies for Embeddings

Storing embeddings efficiently is non-trivial. A 256-dimensional float32 embedding requires 1 KB per entity. At 100 million entities, that is 100 GB of raw vector data — and you may need multiple versions.

Storage Options

Feature Store (Redis / DynamoDB)

Store embeddings as serialized byte arrays alongside scalar features. Good for serving individual entities with low latency.

Vector Database (Pinecone / Weaviate)

Purpose-built for vector similarity search. Supports ANN indexing, metadata filtering, and upsert operations natively.

Compression Techniques

  • Float16 quantization — reduces storage by 50% with minimal accuracy loss.
  • Product quantization (PQ) — compresses vectors into compact codes for ANN search.
  • Dimensionality reduction — PCA or UMAP to reduce 768-dim to 128-dim before storage.
import numpy as np
import redis
import json

# Generate a sample embedding
embedding = np.random.randn(256).astype(np.float32)

# Float16 quantization: halve the storage
embedding_f16 = embedding.astype(np.float16)
print(f"float32 size: {embedding.nbytes} bytes")
print(f"float16 size: {embedding_f16.nbytes} bytes")

# Store in Redis as bytes
r = redis.Redis(host="localhost", port=6379)
r.set("user:u123:embedding", embedding_f16.tobytes())

# Retrieve and deserialize
raw = r.get("user:u123:embedding")
loaded = np.frombuffer(raw, dtype=np.float16)
print(f"Loaded embedding shape: {loaded.shape}")
Warning: Float16 quantization can introduce rounding errors. Always benchmark retrieval quality (recall@k) after quantization to ensure it meets your accuracy requirements.

Serving Embedding Features

Serving embeddings in production requires careful attention to latency, throughput, and memory. Unlike scalar features that are a few bytes each, a single embedding lookup returns hundreds or thousands of floats.

Latency Considerations

  • Network overhead — transferring a 1 KB embedding adds ~0.5ms over a local network.
  • Deserialization — converting bytes back to numpy arrays adds ~0.1ms per embedding.
  • Batching — fetching 100 embeddings in one call vs. 100 individual calls can be 10x faster.
import numpy as np
import redis
from typing import List, Dict

class EmbeddingServer:
    def __init__(self, host="localhost", port=6379, dim=256):
        self.client = redis.Redis(host=host, port=port)
        self.dim = dim
        self.cache: Dict[str, np.ndarray] = {}

    def get_embedding(self, entity_id: str) -> np.ndarray:
        # Check local cache first
        if entity_id in self.cache:
            return self.cache[entity_id]

        raw = self.client.get(f"emb:{entity_id}")
        if raw is None:
            return np.zeros(self.dim, dtype=np.float32)

        vec = np.frombuffer(raw, dtype=np.float32)
        self.cache[entity_id] = vec
        return vec

    def get_batch(self, entity_ids: List[str]) -> np.ndarray:
        # Pipeline for batch retrieval
        pipe = self.client.pipeline()
        for eid in entity_ids:
            pipe.get(f"emb:{eid}")
        results = pipe.execute()

        embeddings = []
        for raw in results:
            if raw is None:
                embeddings.append(np.zeros(self.dim))
            else:
                embeddings.append(np.frombuffer(raw, dtype=np.float32))
        return np.stack(embeddings)

server = EmbeddingServer()
# batch = server.get_batch(["u1", "u2", "u3"])
Performance Tip: Use Redis pipelines for batch fetches. Fetching 100 embeddings individually takes ~50ms; pipelining reduces this to ~5ms by eliminating per-request round-trip overhead.

Nearest Neighbor Retrieval

One of the most powerful applications of embeddings is nearest neighbor retrieval — given a query vector, find the K most similar vectors in a collection. Exact search is O(N) per query; approximate nearest neighbor (ANN) algorithms make this feasible at scale.

ANN Algorithms

  • HNSW (Hierarchical Navigable Small World) — graph-based, excellent recall, higher memory usage.
  • IVF-PQ (Inverted File with Product Quantization) — partition + compress, great for billion-scale datasets.
  • LSH (Locality-Sensitive Hashing) — hash-based, fast but lower recall than HNSW.

FAISS Implementation

import numpy as np
import faiss

# Parameters
dim = 256
n_vectors = 100000
n_queries = 10
k = 5  # top-K neighbors

# Generate sample data
np.random.seed(42)
database = np.random.randn(n_vectors, dim).astype(np.float32)
queries = np.random.randn(n_queries, dim).astype(np.float32)

# Exact search (brute force, for baseline)
index_flat = faiss.IndexFlatL2(dim)
index_flat.add(database)
distances, indices = index_flat.search(queries, k)
print("Exact search results:", indices[:2])

# Approximate search with IVF
n_clusters = 256
quantizer = faiss.IndexFlatL2(dim)
index_ivf = faiss.IndexIVFFlat(quantizer, dim, n_clusters)
index_ivf.train(database)
index_ivf.add(database)
index_ivf.nprobe = 16  # search 16 of 256 clusters
distances_ivf, indices_ivf = index_ivf.search(queries, k)
print("IVF search results:", indices_ivf[:2])

Exact Search (Flat)

Recall: 100%

Speed: O(N) per query

Memory: Full vectors

Best for <100K vectors or when accuracy is critical.

IVF + PQ (Approximate)

Recall: 90–99%

Speed: O(N/clusters × nprobe)

Memory: Compressed codes

Best for millions to billions of vectors where latency matters.

Tuning Tip: Increase nprobe to improve recall at the cost of latency. Start with nprobe = sqrt(n_clusters) and tune based on your recall@k requirements.

Embedding Versioning

Embeddings change every time the underlying model is retrained. A user embedding from model v1 lives in a different vector space than the same user's embedding from model v2. Managing these versions is critical for production reliability.

Why Versioning Matters

  • Incompatible spaces — v1 and v2 embeddings cannot be compared directly. Cosine similarity between them is meaningless.
  • Gradual rollout — you may need to serve both versions simultaneously during an A/B test.
  • Rollback — if v2 degrades quality, you need to revert to v1 quickly.

Versioning Strategies

import json
from datetime import datetime

class EmbeddingVersionManager:
    def __init__(self, store):
        self.store = store

    def publish_version(self, version_id, model_name, dim, metadata=None):
        """Register a new embedding version."""
        version_info = {
            "version_id": version_id,
            "model_name": model_name,
            "dimension": dim,
            "created_at": datetime.utcnow().isoformat(),
            "status": "active",
            "metadata": metadata or {}
        }
        self.store.set(
            f"emb_version:{version_id}",
            json.dumps(version_info)
        )
        return version_info

    def get_active_version(self):
        """Return the currently active embedding version."""
        active_id = self.store.get("emb_version:active")
        if active_id is None:
            return None
        return json.loads(
            self.store.get(f"emb_version:{active_id}")
        )

    def rollback(self, version_id):
        """Switch active version for instant rollback."""
        self.store.set("emb_version:active", version_id)
        print(f"Rolled back to version {version_id}")
Warning: Never mix embeddings from different model versions in the same index or similarity computation. The vector spaces are not aligned — results will be nonsensical.

Migration Strategy

When deploying a new embedding model, follow this sequence:

  1. Compute new embeddings for all entities using the new model.
  2. Write them to a new versioned key space (e.g., emb:v2:user123).
  3. Build a new ANN index from the v2 embeddings.
  4. A/B test v2 against v1 in production.
  5. If v2 wins, promote it to active and deprecate v1.

Embeddings in Production

Running embeddings in production introduces challenges beyond storage and retrieval: monitoring for drift, managing dimensionality at scale, and choosing the right infrastructure.

Monitoring Embedding Drift

Embedding distributions can shift when user behavior changes or upstream data pipelines break. Detecting drift early prevents silent model degradation.

import numpy as np
from scipy.spatial.distance import cosine

def compute_centroid_drift(embeddings_old, embeddings_new):
    """Measure drift between two embedding populations."""
    centroid_old = np.mean(embeddings_old, axis=0)
    centroid_new = np.mean(embeddings_new, axis=0)

    drift = cosine(centroid_old, centroid_new)
    print(f"Centroid drift (cosine distance): {drift:.6f}")
    return drift

def monitor_norm_distribution(embeddings, threshold_std=2.0):
    """Flag entities with abnormal embedding norms."""
    norms = np.linalg.norm(embeddings, axis=1)
    mean_norm = np.mean(norms)
    std_norm = np.std(norms)

    anomalies = np.where(
        np.abs(norms - mean_norm) > threshold_std * std_norm
    )[0]
    print(f"Mean norm: {mean_norm:.3f}, Std: {std_norm:.3f}")
    print(f"Anomalous embeddings: {len(anomalies)}/{len(norms)}")
    return anomalies

# Usage
old_embs = np.random.randn(10000, 256).astype(np.float32)
new_embs = np.random.randn(10000, 256).astype(np.float32)
compute_centroid_drift(old_embs, new_embs)
monitor_norm_distribution(new_embs)

Vector Database Comparison

Pinecone

Type: Fully managed SaaS

Scale: Billions of vectors

Best for: Teams wanting zero-ops vector search

ANN: Proprietary (based on IVF/HNSW)

Weaviate

Type: Open-source, self-hosted or cloud

Scale: Hundreds of millions

Best for: Hybrid search (vector + keyword)

ANN: HNSW

Milvus

Type: Open-source, distributed

Scale: Billions of vectors

Best for: Large-scale self-hosted deployments

ANN: IVF, HNSW, DiskANN

FAISS

Type: Library (not a database)

Scale: Millions to billions (in-memory)

Best for: Offline batch retrieval, research

ANN: IVF-PQ, HNSW, Flat

Choosing the Right Tool: If you need real-time serving with filtering, use a vector database (Pinecone, Weaviate, Milvus). If you need offline batch retrieval for training data, FAISS as a library is simpler and faster to set up.
Cost Alert: Vector databases at scale can be expensive. A 100M × 256-dim index in Pinecone can cost $500+/month. Always estimate costs before committing to a managed service, and consider FAISS for batch workloads.
