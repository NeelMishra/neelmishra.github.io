Feature Backfilling

MLOps Series Feature Stores

What Is Feature Backfilling?

Feature backfilling is the process of retroactively computing feature values for historical time periods. While a feature pipeline normally computes values going forward, backfilling looks backward—reprocessing raw data to populate a feature store with values that didn’t exist when the data was originally ingested.

When You Need a Backfill

  • New feature creation – You design a new feature (e.g., "rolling 30-day purchase frequency") and need it computed for the past two years so the training set is complete.
  • Bug fixes – A timezone bug caused timestamps to shift by 8 hours, skewing all time-window features. After fixing the code, you must recompute every affected value.
  • Schema changes – An upstream table renames user_id to customer_id; the feature pipeline must be re-run against historical data with the new schema mapping.
  • Model retraining – A new model version requires features computed with different parameters (e.g., a 14-day window instead of 7-day).
  • Data corrections – Late-arriving data or corrections to source tables invalidate previously computed features.
Rule of thumb: If your training data and serving data come from different code paths, you have a training-serving skew risk. Backfilling through the same pipeline that serves production features eliminates this skew.

Backfill Strategies

Not all backfills are created equal. The right strategy depends on data volume, compute budget, feature complexity, and how much historical data you need.

Full Recompute

  • Reprocesses all historical data from scratch
  • Guarantees consistency—no partial state
  • Simple to reason about and debug
  • Expensive: O(entire history) compute cost
  • Can take hours or days for large datasets
  • Best for: first-time backfills, major logic changes

Incremental Backfill

  • Reprocesses only the affected date partitions
  • Much faster and cheaper than full recompute
  • Requires partition-level tracking of what changed
  • Risk of inconsistency if dependencies span partitions
  • More complex orchestration logic
  • Best for: bug fixes in narrow date ranges

Hybrid Strategy

  • Full recompute for the affected window, incremental for the rest
  • Balances cost and correctness
  • Requires identifying the "blast radius" of the change
  • Good default for most production scenarios

Shadow Backfill

  • Writes to a shadow table alongside the live table
  • Allows validation before swapping into production
  • Doubles storage temporarily
  • Best for: high-risk changes to critical features
Cost trap: A full recompute of 2 years of data on a 100-node Spark cluster can cost thousands of dollars in cloud compute. Always estimate costs before launching and consider running during off-peak hours with spot instances.

Idempotent Pipelines

A backfill pipeline must be idempotent: running it twice with the same inputs produces the same outputs without side effects. This is non-negotiable because backfills frequently fail mid-way and must be safely retried.

Principles of Idempotency

  • Deterministic transforms – No random seeds, no wall-clock timestamps, no non-deterministic UDFs. Every output must depend only on the input data and parameters.
  • Upsert, not append – Use MERGE or INSERT ... ON CONFLICT UPDATE so re-runs overwrite rather than duplicate.
  • Partition-level atomicity – Write each date partition as a single atomic operation. If a partition write fails, the next retry replaces it entirely.
  • Immutable inputs – Read from snapshot tables or versioned datasets so the source data doesn’t change between retries.
from datetime import date

class IdempotentFeatureWriter:
    # Writes features using upsert pattern for safe retries
    def __init__(self, db_connection):
        self.db = db_connection

    def upsert_features(self, feature_name, partition_date, records):
        # Delete existing data for this partition, then insert
        # This makes the operation atomic and idempotent
        self.db.execute(
            "DELETE FROM features WHERE name = %s AND dt = %s",
            (feature_name, partition_date)
        )
        for record in records:
            self.db.execute(
                """INSERT INTO features (name, entity_id, dt, value, updated_at)
                   VALUES (%s, %s, %s, %s, NOW())""",
                (feature_name, record["entity_id"],
                 partition_date, record["value"])
            )
        self.db.commit()

    def write_partition(self, feature_name, dt, compute_fn, raw_data):
        # Full partition-level idempotent write
        records = compute_fn(raw_data, dt)
        self.upsert_features(feature_name, dt, records)
        return len(records)

Delta Lake Upsert Pattern

from delta.tables import DeltaTable

def upsert_to_delta(spark, new_features_df, target_path):
    # Merge new features into existing Delta table
    target = DeltaTable.forPath(spark, target_path)

    target.alias("t").merge(
        new_features_df.alias("s"),
        "t.entity_id = s.entity_id AND t.feature_date = s.feature_date"
    ).whenMatchedUpdateAll() \
     .whenNotMatchedInsertAll() \
     .execute()
Tip: Wrap each partition write in a transaction. If your store doesn’t support transactions, use the "delete-then-insert" pattern shown above. It’s not truly atomic but is idempotent, which is the critical property for backfills.

Time-Travel Queries

Time-travel lets you query the feature store as it was at a specific point in the past. This is essential for backfills because you need to know: "What features would have been available to the model at prediction time T?"

Why Time-Travel Matters

Without time-travel, backfilled features may use data that was not available at the original prediction time, causing data leakage. For example, if a user made a purchase at 3:00 PM, the feature "purchases in the last hour" should only count events before 3:00 PM—not events that arrived later in the day.

Versioned Feature Store

from datetime import datetime
from bisect import bisect_right

class VersionedFeatureStore:
    # Feature store with point-in-time query support
    def __init__(self):
        # entity_id -> [(timestamp, value), ...] sorted by timestamp
        self.store = {}

    def write(self, entity_id, feature_name, timestamp, value):
        key = (entity_id, feature_name)
        if key not in self.store:
            self.store[key] = []
        self.store[key].append((timestamp, value))
        self.store[key].sort(key=lambda x: x[0])

    def read_at(self, entity_id, feature_name, as_of):
        # Return the feature value that was current at `as_of` time
        key = (entity_id, feature_name)
        history = self.store.get(key, [])

        if not history:
            return None

        timestamps = [t for t, v in history]
        idx = bisect_right(timestamps, as_of) - 1

        if idx < 0:
            return None
        return history[idx][1]

    def build_training_set(self, entities, feature_names, event_timestamps):
        # Point-in-time join: for each entity+timestamp, get features
        rows = []
        for entity_id, ts in zip(entities, event_timestamps):
            row = {"entity_id": entity_id, "event_time": ts}
            for fname in feature_names:
                row[fname] = self.read_at(entity_id, fname, ts)
            rows.append(row)
        return rows

Implementation Patterns

Below are two production-tested patterns: a Spark backfill job for large-scale recomputation, and an Airflow DAG for orchestrated, partition-by-partition backfills.

Spark Backfill Job

from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from datetime import date, timedelta

def run_backfill(start_date, end_date, feature_name):
    spark = SparkSession.builder \
        .appName(f"backfill-{feature_name}") \
        .config("spark.sql.shuffle.partitions", 200) \
        .getOrCreate()

    # Generate list of dates to backfill
    current = start_date
    dates = []
    while current <= end_date:
        dates.append(current)
        current += timedelta(days=1)

    for dt in dates:
        print(f"Backfilling {feature_name} for {dt}")

        # Read raw events for this date only
        raw = spark.read.parquet(f"/data/events/dt={dt}")

        # Compute features (same logic as production pipeline)
        features = raw.groupBy("entity_id").agg(
            F.count("*").alias("event_count"),
            F.sum("amount").alias("total_amount"),
            F.avg("amount").alias("avg_amount"),
            F.max("event_time").alias("last_event_time"),
        ).withColumn("feature_date", F.lit(dt))

        # Write with overwrite mode for idempotency
        features.write \
            .mode("overwrite") \
            .option("replaceWhere", f"feature_date = '{dt}'") \
            .format("delta") \
            .save(f"/features/{feature_name}")

    print(f"Backfill complete: {len(dates)} partitions processed")

run_backfill(
    start_date=date(2023, 1, 1),
    end_date=date(2024, 12, 31),
    feature_name="daily_purchase_features"
)

Pro tip: Set max_active_runs in Airflow to control parallelism during backfills. Too many concurrent partitions will overwhelm your Spark cluster or data warehouse. Start with 4–8 and scale based on cluster capacity.

Validating Backfills

A backfill is not done until it is validated. Incorrect backfills can be worse than missing data because they silently corrupt training sets and model predictions.

Consistency Checks

  • Row counts – Compare expected entity count per partition against actual.
  • Null rates – Should match or be lower than the original pipeline.
  • Value ranges – Min/max should be within expected bounds.
  • Distribution similarity – PSI between backfilled and live features for overlapping dates should be near zero.
  • Referential integrity – Every entity in the feature table should exist in the source entity table.
import numpy as np

class BackfillValidator:
    # Validates backfilled features against expected constraints
    def __init__(self, tolerance=0.05):
        self.tolerance = tolerance
        self.results = []

    def check_row_count(self, actual, expected):
        ratio = actual / expected if expected > 0 else 0
        passed = abs(ratio - 1.0) < self.tolerance
        self.results.append({
            "check": "row_count",
            "passed": passed,
            "actual": actual,
            "expected": expected,
        })
        return passed

    def check_null_rate(self, values, max_null_rate=0.01):
        null_count = sum(1 for v in values if v is None)
        rate = null_count / len(values) if values else 0
        passed = rate <= max_null_rate
        self.results.append({
            "check": "null_rate",
            "passed": passed,
            "rate": round(rate, 4),
            "max_allowed": max_null_rate,
        })
        return passed

    def check_distribution(self, backfilled, reference, max_psi=0.1):
        # PSI between backfilled and reference data
        psi = compute_psi(reference, backfilled)
        passed = psi < max_psi
        self.results.append({
            "check": "distribution_psi",
            "passed": passed,
            "psi": round(psi, 4),
            "max_allowed": max_psi,
        })
        return passed

    def check_value_range(self, values, expected_min, expected_max):
        actual_min = min(v for v in values if v is not None)
        actual_max = max(v for v in values if v is not None)
        passed = actual_min >= expected_min and actual_max <= expected_max
        self.results.append({
            "check": "value_range",
            "passed": passed,
            "actual": (actual_min, actual_max),
            "expected": (expected_min, expected_max),
        })
        return passed

    def summary(self):
        total = len(self.results)
        passed = sum(1 for r in self.results if r["passed"])
        return {
            "total_checks": total,
            "passed": passed,
            "failed": total - passed,
            "all_passed": passed == total,
            "details": self.results,
        }

Before/After Comparison

def compare_before_after(before_path, after_path, spark):
    # Compare feature distributions before and after backfill
    before = spark.read.parquet(before_path)
    after = spark.read.parquet(after_path)

    # Compare row counts per partition
    before_counts = before.groupBy("feature_date").count()
    after_counts = after.groupBy("feature_date").count()

    comparison = before_counts.alias("b").join(
        after_counts.alias("a"),
        "feature_date",
        "full_outer"
    ).select(
        "feature_date",
        F.col("b.count").alias("before_count"),
        F.col("a.count").alias("after_count"),
        F.abs(F.col("a.count") - F.col("b.count")).alias("diff"),
    )

    # Flag partitions with significant count differences
    flagged = comparison.filter(
        F.col("diff") > F.col("before_count") * 0.05
    )

    return {
        "total_partitions": comparison.count(),
        "flagged_partitions": flagged.count(),
        "details": flagged.collect(),
    }
Final checklist before promoting a backfill: (1) All partitions have non-zero row counts. (2) PSI against the previous version is < 0.1 for overlapping dates. (3) Null rates are within SLA. (4) At least one model retrained on the backfilled data shows equal or better offline metrics.
Never skip validation: A corrupted backfill that goes unnoticed can poison months of model training. Always run automated checks and require human sign-off for backfills that affect more than 30 days of data.
