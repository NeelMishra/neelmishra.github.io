Feature Monitoring: Drift, Distribution, Staleness

MLOps Series Feature Stores

Why Monitor Features?

A model is only as reliable as the features it consumes. Unlike traditional software, where failures crash loudly, feature failures are silent. A stale feature still looks like a valid number; a drifted distribution still produces predictions—they’re just wrong. By the time anyone notices, revenue has leaked, users have churned, or fraud has slipped through.

Silent Failures in Production

  • Schema change upstream – a partner API renames a field from avg_price to average_price; the pipeline fills the feature with nulls, which the model interprets as zero.
  • Seasonal drift – holiday shopping shifts the "average basket size" feature from $45 to $120; the fraud model flags every legitimate purchase.
  • Pipeline lag – the Airflow DAG that computes "purchases in the last 7 days" gets stuck; the feature serves yesterday’s value for 48 hours and nobody notices.
  • Unit change – an ETL refactor converts currency from cents to dollars; the model sees 100× smaller values and confidently predicts the wrong class.
Key insight: Monitoring features is strictly more important than monitoring model accuracy. By the time accuracy degrades, the damage is already done. Feature monitors catch problems before they reach the model.

Feature Drift Detection

Drift detection compares the current distribution of a feature against a reference distribution (typically from training data or a stable production window). Three widely used statistical methods give complementary views of drift.

Population Stability Index (PSI)

PSI bins both distributions into buckets and measures the divergence. A PSI below 0.1 indicates no significant shift; 0.1–0.25 is moderate; above 0.25 warrants investigation.

import numpy as np

def compute_psi(reference, current, bins=10):
    # Population Stability Index between two distributions
    breakpoints = np.linspace(
        min(np.min(reference), np.min(current)),
        max(np.max(reference), np.max(current)),
        bins + 1
    )
    ref_counts = np.histogram(reference, bins=breakpoints)[0]
    cur_counts = np.histogram(current, bins=breakpoints)[0]

    # Avoid division by zero with small epsilon
    ref_pct = (ref_counts + 1e-6) / sum(ref_counts + 1e-6)
    cur_pct = (cur_counts + 1e-6) / sum(cur_counts + 1e-6)

    psi = np.sum((cur_pct - ref_pct) * np.log(cur_pct / ref_pct))
    return float(psi)

Kolmogorov-Smirnov Test

The KS test measures the maximum distance between two empirical CDFs. It is non-parametric and works well for continuous features without binning.

from scipy import stats

def ks_drift_check(reference, current, threshold=0.05):
    # Two-sample Kolmogorov-Smirnov test for drift
    statistic, p_value = stats.ks_2samp(reference, current)
    is_drifted = p_value < threshold

    return {
        "ks_statistic": round(statistic, 4),
        "p_value": round(p_value, 6),
        "is_drifted": is_drifted,
        "severity": "high" if statistic > 0.2 else "medium" if statistic > 0.1 else "low"
    }

KL Divergence

Kullback-Leibler divergence measures how one probability distribution diverges from a reference. Unlike KS, it is sensitive to the tails and requires binned data.

from scipy.special import rel_entr
import numpy as np

def kl_divergence(reference, current, bins=20):
    # Compute binned KL divergence
    lo = min(np.min(reference), np.min(current))
    hi = max(np.max(reference), np.max(current))
    edges = np.linspace(lo, hi, bins + 1)

    p = np.histogram(reference, bins=edges)[0].astype(float)
    q = np.histogram(current, bins=edges)[0].astype(float)

    # Normalise to probability distributions
    p = (p + 1e-8) / p.sum()
    q = (q + 1e-8) / q.sum()

    return float(np.sum(rel_entr(p, q)))
Watch out: PSI and KL divergence require careful binning. Too few bins miss subtle shifts; too many bins amplify noise. Start with 10–20 bins and tune based on feature cardinality.

Distribution Monitoring

Beyond drift tests, continuously tracking summary statistics and shape metrics catches anomalies that statistical tests might miss—such as a bimodal distribution collapsing into a single peak.

Key Metrics to Track

  • Mean and median – detect location shifts.
  • Standard deviation – detect spread changes.
  • Skewness and kurtosis – detect shape changes.
  • Min / max / percentiles – detect range violations.
  • Null rate – detect upstream pipeline failures.
  • Unique count – detect cardinality explosions or collapses.
import numpy as np
from scipy import stats as sp_stats
from dataclasses import dataclass

@dataclass
class DistributionProfile:
    mean: float
    median: float
    std: float
    skewness: float
    kurtosis: float
    null_rate: float
    p5: float
    p95: float

def profile_feature(values, total_count):
    # Build a statistical profile for a single feature
    clean = np.array([v for v in values if v is not None])
    null_count = total_count - len(clean)

    if len(clean) == 0:
        return None

    return DistributionProfile(
        mean=float(np.mean(clean)),
        median=float(np.median(clean)),
        std=float(np.std(clean)),
        skewness=float(sp_stats.skew(clean)),
        kurtosis=float(sp_stats.kurtosis(clean)),
        null_rate=null_count / total_count,
        p5=float(np.percentile(clean, 5)),
        p95=float(np.percentile(clean, 95)),
    )

Z-Score Anomaly Detection on Metrics

def detect_metric_anomaly(history, current_value, z_threshold=3.0):
    # Flag if current metric deviates beyond z_threshold from historical mean
    mean = np.mean(history)
    std = np.std(history)

    if std == 0:
        return current_value != mean

    z_score = abs(current_value - mean) / std
    return {
        "z_score": round(z_score, 3),
        "is_anomaly": z_score > z_threshold,
        "expected_range": (
            round(mean - z_threshold * std, 3),
            round(mean + z_threshold * std, 3)
        ),
    }
Tip: Store distribution profiles for every feature batch in a metadata table. This creates a time-series of statistics that powers both dashboards and anomaly detection.

Staleness Monitoring

A feature is stale when its most recent value is older than its expected refresh interval. Staleness is the most common feature failure mode and the easiest to detect.

Timestamp-Based Checks

Every feature row should carry an updated_at timestamp. A monitoring job compares that timestamp against the current wall clock and the feature’s SLA.

from datetime import datetime, timedelta
from dataclasses import dataclass

@dataclass
class StalenessConfig:
    feature_name: str
    max_age: timedelta
    severity: str  # "warning" or "critical"

def check_staleness(configs, feature_timestamps):
    # Check each feature against its freshness SLA
    now = datetime.utcnow()
    alerts = []

    for cfg in configs:
        last_update = feature_timestamps.get(cfg.feature_name)
        if last_update is None:
            alerts.append({
                "feature": cfg.feature_name,
                "status": "missing",
                "severity": "critical",
            })
            continue

        age = now - last_update
        if age > cfg.max_age:
            alerts.append({
                "feature": cfg.feature_name,
                "status": "stale",
                "age_seconds": age.total_seconds(),
                "max_age_seconds": cfg.max_age.total_seconds(),
                "severity": cfg.severity,
            })

    return alerts

# Example configuration
configs = [
    StalenessConfig("txn_count_1h", timedelta(minutes=5), "critical"),
    StalenessConfig("user_profile_v2", timedelta(hours=24), "warning"),
    StalenessConfig("product_embedding", timedelta(days=7), "warning"),
]

SLA Violation Tracking

class SLATracker:
    # Track SLA compliance rates over time
    def __init__(self):
        self.checks = []

    def record_check(self, feature_name, is_fresh, timestamp):
        self.checks.append({
            "feature": feature_name,
            "fresh": is_fresh,
            "ts": timestamp,
        })

    def compliance_rate(self, feature_name, window_hours=24):
        cutoff = datetime.utcnow() - timedelta(hours=window_hours)
        relevant = [
            c for c in self.checks
            if c["feature"] == feature_name and c["ts"] > cutoff
        ]
        if not relevant:
            return None
        fresh = sum(1 for c in relevant if c["fresh"])
        return round(fresh / len(relevant), 4)
Common pitfall: Checking staleness only at query time misses the window between when a feature becomes stale and when a model next reads it. Run staleness monitors on a schedule (e.g., every minute) independent of model inference.

Building Feature Dashboards

A feature dashboard turns raw metrics into actionable visibility. It should answer three questions at a glance: Is anything broken right now? What changed recently? Are long-term trends healthy?

Essential Dashboard Panels

  • Health summary – green/yellow/red status per feature group.
  • Drift heatmap – PSI or KS statistic over time for each feature.
  • Staleness timeline – age of each feature plotted against its SLA.
  • Distribution overlay – current vs. reference histogram per feature.
  • Null-rate trend – percentage of null values per feature over the last 7 days.

Metrics Collection Pipeline

import json, time

class FeatureMetricsCollector:
    # Collects and exports feature metrics for dashboarding
    def __init__(self, export_endpoint):
        self.endpoint = export_endpoint
        self.metrics = []

    def collect(self, feature_name, profile, drift_scores):
        record = {
            "feature": feature_name,
            "timestamp": time.time(),
            "mean": profile.mean,
            "std": profile.std,
            "null_rate": profile.null_rate,
            "psi": drift_scores.get("psi", None),
            "ks_stat": drift_scores.get("ks_statistic", None),
        }
        self.metrics.append(record)

    def export_to_grafana(self):
        # Push metrics to Prometheus/Grafana via push gateway
        for m in self.metrics:
            payload = json.dumps(m)
            # POST to push gateway endpoint
            http_post(self.endpoint, payload)
        self.metrics = []

    def export_to_file(self, path):
        with open(path, "a") as f:
            for m in self.metrics:
                f.write(json.dumps(m) + "\n")
        self.metrics = []
Grafana + Prometheus tip: Use a Prometheus push gateway to export feature metrics with labels (feature_name, feature_group). This lets you build templated dashboards that auto-populate when new features are registered.

Alerting and Response

Monitoring without alerting is just watching. An effective alerting system must balance signal-to-noise: too few alerts and you miss problems; too many and the team ignores them.

Threshold-Based Alerts

Static thresholds work well for well-understood features. For example: alert if PSI > 0.25, null rate > 5%, or staleness > 2× the expected refresh interval.

ML-Based Anomaly Detection

For features with seasonal or non-stationary patterns, train a lightweight anomaly detector (e.g., Isolation Forest or Prophet) on the metric time-series. This adapts to patterns that static thresholds cannot capture.

from dataclasses import dataclass

@dataclass
class AlertRule:
    feature: str
    metric: str
    operator: str  # "gt", "lt", "eq"
    threshold: float
    severity: str
    channel: str   # "slack", "pagerduty", "email"

def evaluate_alerts(rules, current_metrics):
    # Evaluate all alert rules against current metrics
    fired = []
    ops = {"gt": lambda a, b: a > b,
           "lt": lambda a, b: a < b,
           "eq": lambda a, b: abs(a - b) < 1e-9}

    for rule in rules:
        value = current_metrics.get(rule.feature, {}).get(rule.metric)
        if value is None:
            continue
        check_fn = ops.get(rule.operator)
        if check_fn and check_fn(value, rule.threshold):
            fired.append({
                "feature": rule.feature,
                "metric": rule.metric,
                "value": value,
                "threshold": rule.threshold,
                "severity": rule.severity,
                "channel": rule.channel,
            })
    return fired

# Example rules
rules = [
    AlertRule("txn_count_1h", "psi", "gt", 0.25, "critical", "pagerduty"),
    AlertRule("txn_count_1h", "null_rate", "gt", 0.05, "warning", "slack"),
    AlertRule("user_age_days", "ks_stat", "gt", 0.15, "warning", "slack"),
]

Runbook Template

Every alert should link to a runbook that answers: (1) What does this alert mean? (2) What is the blast radius? (3) What are the immediate steps? (4) Who should be escalated to?

Alerting Approaches Comparison

Threshold-Based Alerting

  • Simple to implement and explain
  • Deterministic: same input → same alert
  • No training data required
  • Brittle to seasonal patterns
  • Requires manual threshold tuning
  • Best for: stable, well-understood features

ML-Based Anomaly Detection

  • Adapts to seasonal and non-stationary patterns
  • Lower false-positive rate over time
  • Captures complex, multi-variate anomalies
  • Harder to debug and explain decisions
  • Requires historical metric data for training
  • Best for: high-cardinality, dynamic features
Best practice: Start with threshold-based alerts for all features, then layer ML anomaly detection on top for the highest-impact features. This gives you immediate coverage while building toward adaptive monitoring.
