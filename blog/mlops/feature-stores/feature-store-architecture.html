Feature Store Architecture

MLOps Series Feature Stores

Architecture Overview

A feature store sits at the intersection of data engineering and machine learning. Its architecture is designed to solve a deceptively hard problem: how do you compute, store, and serve features consistently across training and inference? The answer requires multiple layers working in concert—each with distinct responsibilities and performance characteristics.

The diagram above shows the four fundamental layers. Data flows left to right: raw sources feed into the compute layer, which transforms and materializes features into the storage layer. The serving layer exposes those features to models, while the metadata registry spans all layers, tracking definitions, lineage, and access policies.

Key insight: The separation of offline and online stores is the defining architectural decision. It lets you optimize for two wildly different access patterns—large historical scans for training and point lookups for inference—without compromising on either.

Storage Layer

The storage layer is the backbone of a feature store. It must handle two fundamentally different workloads: offline (batch reads of historical feature values for training) and online (ultra-low-latency lookups for real-time inference).

Offline Store

The offline store holds the complete history of feature values. It's optimized for large sequential reads and supports time-travel queries—retrieving features as they existed at a specific point in time. Common backends include data warehouses like BigQuery or Redshift, or columnar file formats on object storage (Parquet on S3/GCS).

Online Store

The online store holds only the latest feature values for each entity. It's optimized for key-value lookups with p99 latencies under 10ms. Redis, DynamoDB, and Bigtable are typical choices. The compute layer materializes features from the offline store into the online store on a schedule or via streaming.

Raw Data Lake

Underneath both stores sits the raw data lake—often S3 or GCS—which archives unprocessed source data. This gives you the ability to recompute features when definitions change, without losing the original signal.

Offline Store Options

  • BigQuery / Redshift: Fully managed, SQL interface, great for ad-hoc exploration
  • Parquet on S3: Cost-effective, schema-on-read, Spark-friendly
  • Delta Lake / Iceberg: ACID transactions, time travel built-in, schema evolution

Online Store Options

  • Redis: Sub-ms latency, in-memory, requires capacity planning
  • DynamoDB: Fully managed, auto-scales, predictable pricing
  • Bigtable: High throughput, wide-column, good for large feature vectors
Warning: Never serve training data directly from the online store. The online store lacks history—you'll introduce training/serving skew and silently degrade model performance.

Compute Layer

The compute layer is responsible for transforming raw data into features. It supports three computation modes, each suited to different latency and freshness requirements.

Batch Processing

Batch pipelines run on a schedule (hourly, daily) and process large volumes of data. They compute aggregations, joins, and window functions over historical data. Spark and dbt are the most common engines.

Stream Processing

Streaming pipelines consume events in real-time from Kafka or Kinesis. They compute features like "number of transactions in the last 5 minutes" with low latency. Flink and Spark Structured Streaming are popular choices.

On-Demand Computation

Some features can only be computed at request time—for example, the time since the user's last login. These are calculated in the serving path and never materialized to storage.

# Feature transformation pipeline example
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.window import Window

class FeatureTransformPipeline:
    def __init__(self, spark: SparkSession):
        self.spark = spark

    def compute_user_features(self, transactions_df):
        # Define a 30-day rolling window per user
        window_30d = Window.partitionBy("user_id") \
            .orderBy("event_timestamp") \
            .rangeBetween(-30 * 86400, 0)

        features = transactions_df.withColumn(
            "txn_count_30d", F.count("txn_id").over(window_30d)
        ).withColumn(
            "avg_amount_30d", F.avg("amount").over(window_30d)
        ).withColumn(
            "max_amount_30d", F.max("amount").over(window_30d)
        )
        return features

    def compute_streaming_features(self, stream_df):
        # Sliding window aggregation on streaming data
        return stream_df \
            .withWatermark("event_time", "10 minutes") \
            .groupBy(
                F.window("event_time", "5 minutes", "1 minute"),
                "user_id"
            ).agg(
                F.count("*").alias("event_count_5m"),
                F.sum("amount").alias("total_amount_5m")
            )
Tip: When possible, define your feature logic once and have the compute layer execute it in both batch and streaming modes. Dual-execution engines like Spark Structured Streaming make this feasible and eliminate an entire class of training/serving skew bugs.

Serving Layer

The serving layer is the interface between the feature store and the models that consume features. It must deliver feature vectors at the latency your application demands—often under 10ms for real-time inference.

API Interfaces

Most feature stores expose features through multiple interfaces: REST APIs for broad compatibility, gRPC for high-throughput low-latency paths, and Python SDKs for notebook-based training workflows. The choice depends on the consumer.

Latency Optimization

Key strategies include batching lookups (fetching all features for a prediction in one call), caching hot features at the edge, and pre-computing feature vectors for known entity sets. Connection pooling and async I/O further reduce overhead.

# Feature serving client with caching
import redis
import json
from typing import Dict, List, Optional

class FeatureServingClient:
    def __init__(self, online_store_host: str, cache_ttl: int = 60):
        self.store = redis.Redis(host=online_store_host, port=6379)
        self.cache = {}
        self.cache_ttl = cache_ttl

    def get_features(self, entity_key: str, feature_names: List[str]) -> Dict:
        # Check local cache first
        cache_key = f"{entity_key}:{','.join(feature_names)}"
        if cache_key in self.cache:
            return self.cache[cache_key]

        # Batch fetch from online store
        pipe = self.store.pipeline()
        for name in feature_names:
            pipe.hget(entity_key, name)
        values = pipe.execute()

        result = {}
        for name, val in zip(feature_names, values):
            result[name] = json.loads(val) if val else None

        self.cache[cache_key] = result
        return result

    def get_batch_features(self, entity_keys: List[str], feature_names: List[str]) -> List[Dict]:
        # Efficient batch retrieval for multiple entities
        pipe = self.store.pipeline()
        for key in entity_keys:
            for name in feature_names:
                pipe.hget(key, name)
        all_values = pipe.execute()

        results = []
        idx = 0
        for key in entity_keys:
            row = {}
            for name in feature_names:
                row[name] = json.loads(all_values[idx]) if all_values[idx] else None
                idx += 1
            results.append(row)
        return results
Warning: Avoid fetching features one at a time in a loop. A model needing 50 features across 3 entities could generate 150 round trips. Always use batch APIs and pipeline your requests.

Metadata and Registry

The metadata layer is often the most underappreciated part of a feature store—and the one that delivers the most long-term value. It's the system of record for what features exist, how they're computed, who owns them, and how they've changed over time.

Feature Definitions

Each feature is registered with its name, data type, default value, description, and owning team. This enables discovery—data scientists can browse the registry to find existing features instead of re-creating them.

Lineage Tracking

Lineage records the complete dependency graph: which raw data sources feed into a feature, what transformations are applied, and which models consume it. When a source schema changes, lineage lets you identify every downstream feature and model that could be affected.

Versioning

Feature definitions evolve—new aggregation windows, different normalization, bug fixes. The registry tracks versions so you can reproduce historical training runs and roll back safely.

# Feature registry definition
from dataclasses import dataclass, field
from typing import List, Optional
from datetime import datetime

@dataclass
class FeatureDefinition:
    name: str
    dtype: str
    description: str
    owner: str
    version: int = 1
    tags: List[str] = field(default_factory=list)
    created_at: datetime = field(default_factory=datetime.now)
    sources: List[str] = field(default_factory=list)
    lineage_sql: Optional[str] = None

class FeatureRegistry:
    def __init__(self):
        self.features = {}

    def register(self, feature: FeatureDefinition):
        # Auto-increment version if feature already exists
        if feature.name in self.features:
            existing = self.features[feature.name]
            feature.version = existing.version + 1
        self.features[feature.name] = feature
        return feature

    def search(self, tag: str) -> List[FeatureDefinition]:
        return [f for f in self.features.values() if tag in f.tags]

    def get_lineage(self, feature_name: str) -> List[str]:
        # Returns upstream source dependencies
        feature = self.features.get(feature_name)
        if feature:
            return feature.sources
        return []

# Usage
registry = FeatureRegistry()
registry.register(FeatureDefinition(
    name="user_txn_count_30d",
    dtype="int64",
    description="Number of transactions in last 30 days",
    owner="fraud-team",
    tags=["fraud", "user", "aggregation"],
    sources=["transactions_db.payments"],
    lineage_sql="SELECT user_id, COUNT(*) FROM payments WHERE ..."
))
Best practice: Treat your feature registry like a data catalog. Enforce descriptions, ownership, and tagging on every feature. In organizations with hundreds of features, discoverability is the difference between feature reuse and feature duplication.

Reference Architectures

Here are three common deployment patterns, each suited to different organizational contexts and cloud preferences.

AWS-Based Stack

  • Compute: AWS Glue (batch), Kinesis + Lambda (streaming)
  • Offline Store: S3 + Athena or Redshift
  • Online Store: DynamoDB or ElastiCache (Redis)
  • Registry: SageMaker Feature Store metadata
  • Serving: SageMaker endpoints or API Gateway

GCP-Based Stack

  • Compute: Dataflow (batch + stream), Cloud Functions
  • Offline Store: BigQuery
  • Online Store: Bigtable or Memorystore (Redis)
  • Registry: Vertex AI Feature Store metadata
  • Serving: Vertex AI endpoints

Open-Source Stack

  • Compute: Spark / Flink
  • Offline Store: Delta Lake on MinIO / S3
  • Online Store: Redis / PostgreSQL
  • Registry: Feast registry (file or SQL-backed)
  • Serving: Feast feature server (Go-based)
  • Orchestration: Airflow / Prefect

Choosing the Right Stack

  • Startup / small team: Start with Feast + Redis. Minimal ops burden.
  • Cloud-native org: Use your cloud provider's managed feature store.
  • Multi-cloud / privacy: Open-source stack gives you portability and control.
  • Enterprise: Consider Tecton or Databricks Feature Store for governance.
Architecture principle: Start with the simplest stack that meets your latency and scale requirements. You can always migrate the online store from Redis to DynamoDB or swap the compute engine from Spark to Flink—the metadata registry and feature definitions should remain stable through those transitions.
