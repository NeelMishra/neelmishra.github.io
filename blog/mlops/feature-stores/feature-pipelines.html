Feature Pipelines: Batch + Streaming

MLOps Series Feature Stores

What Are Feature Pipelines?

A feature pipeline is the data transformation process that converts raw data into ML-ready features. It sits between your data sources (databases, event streams, APIs) and your feature store, handling the compute-intensive work of aggregation, normalization, encoding, and enrichment.

Feature pipelines are distinct from model training pipelines. While training pipelines consume features and produce models, feature pipelines consume raw data and produce features. This separation enables:

  • Feature reuse — the same feature can serve multiple models
  • Independent scaling — feature computation scales separately from model training
  • Consistency — training and serving use the same transformation logic
  • Auditability — feature lineage is tracked independently from model experiments
Key Principle: Feature pipelines should be idempotent—running the same pipeline on the same input data should always produce the same output. This makes pipelines safe to retry, backfill, and debug.

Batch Feature Pipelines

Batch pipelines are the workhorse of feature engineering. They run on a schedule (hourly, daily, weekly), process a window of historical data, and materialize feature values into the feature store. Most features in production systems are batch-computed.

Spark-Based Batch Pipeline

# batch_pipeline.py — Spark batch feature pipeline

from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.window import Window
from datetime import datetime, timedelta

def compute_user_features(spark: SparkSession, run_date: datetime) -> "DataFrame":
    # Read raw transaction data for the past 30 days
    start_date = run_date - timedelta(days=30)

    transactions = spark.read.parquet("s3://data-lake/transactions/").filter(
        (F.col("event_date") >= start_date.strftime("%Y-%m-%d"))
        & (F.col("event_date") < run_date.strftime("%Y-%m-%d"))
    )

    # Compute aggregation features
    user_features = transactions.groupBy("user_id").agg(
        F.count("*").alias("txn_count_30d"),
        F.sum("amount").alias("total_spend_30d"),
        F.avg("amount").alias("avg_txn_amount_30d"),
        F.stddev("amount").alias("stddev_txn_amount_30d"),
        F.countDistinct("merchant_id").alias("unique_merchants_30d"),
    )

    # Add metadata columns
    user_features = user_features.withColumn(
        "feature_timestamp", F.lit(run_date)
    )

    return user_features

# Run the pipeline
spark = SparkSession.builder.appName("user_features_batch").getOrCreate()
features = compute_user_features(spark, datetime(2024, 1, 15))
features.write.mode("append").parquet("s3://feature-store/user_features/")

SQL-Based Batch Pipeline

# sql_batch.py — SQL-first batch feature pipeline

from datetime import datetime

def user_engagement_features_sql(run_date: datetime) -> str:
    return f"""
    INSERT INTO feature_store.user_engagement
    SELECT
        user_id,
        COUNT(DISTINCT session_id) AS sessions_30d,
        SUM(page_views)            AS total_page_views_30d,
        AVG(session_duration_sec)  AS avg_session_duration_30d,
        MAX(last_active_at)        AS last_active_at,
        CURRENT_TIMESTAMP          AS feature_timestamp
    FROM analytics.user_sessions
    WHERE session_start >= DATE_SUB('{run_date.strftime("%Y-%m-%d")}', INTERVAL 30 DAY)
      AND session_start <  '{run_date.strftime("%Y-%m-%d")}'
    GROUP BY user_id
    """
Backfill Warning: When changing a batch feature definition, you must backfill historical values. Running the pipeline only from today forward creates a discontinuity in your feature history that can degrade model performance during retraining.

Streaming Feature Pipelines

Streaming pipelines consume events in real-time from sources like Kafka or Kinesis, compute features on-the-fly, and write results to the online store with minimal latency (seconds to minutes).

Kafka + Flink Streaming Pipeline

# streaming_pipeline.py — Real-time feature computation with Flink

from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment, EnvironmentSettings
from pyflink.table.window import Tumble

def create_streaming_pipeline():
    env = StreamExecutionEnvironment.get_execution_environment()
    t_env = StreamTableEnvironment.create(env)

    # Define Kafka source
    t_env.execute_sql("""
        CREATE TABLE click_events (
            user_id STRING,
            page_id STRING,
            action STRING,
            event_time TIMESTAMP(3),
            WATERMARK FOR event_time AS event_time - INTERVAL '10' SECOND
        ) WITH (
            'connector' = 'kafka',
            'topic' = 'user-clicks',
            'properties.bootstrap.servers' = 'kafka:9092',
            'format' = 'json'
        )
    """)

    # Compute sliding window aggregations
    result = t_env.sql_query("""
        SELECT
            user_id,
            TUMBLE_END(event_time, INTERVAL '5' MINUTE) AS window_end,
            COUNT(*) AS click_count_5min,
            COUNT(DISTINCT page_id) AS unique_pages_5min
        FROM click_events
        GROUP BY user_id, TUMBLE(event_time, INTERVAL '5' MINUTE)
    """)

    return result

Lightweight Streaming with Python

# lightweight_stream.py — Simple streaming aggregation

from collections import defaultdict
from datetime import datetime, timedelta
import json

class SlidingWindowAggregator:
    def __init__(self, window_size: timedelta):
        self.window_size = window_size
        self.events = defaultdict(list)  # entity_key -> [(timestamp, value)]

    def add_event(self, entity_key: str, timestamp: datetime, value: float):
        self.events[entity_key].append((timestamp, value))
        self._evict_expired(entity_key, timestamp)

    def get_count(self, entity_key: str) -> int:
        return len(self.events[entity_key])

    def get_sum(self, entity_key: str) -> float:
        return sum(v for _, v in self.events[entity_key])

    def _evict_expired(self, entity_key: str, current_time: datetime):
        cutoff = current_time - self.window_size
        self.events[entity_key] = [
            (t, v) for t, v in self.events[entity_key] if t > cutoff
        ]

# Usage with a Kafka consumer
from kafka import KafkaConsumer

aggregator = SlidingWindowAggregator(window_size=timedelta(hours=1))
consumer = KafkaConsumer("user-events", bootstrap_servers="kafka:9092")

for message in consumer:
    event = json.loads(message.value)
    aggregator.add_event(
        entity_key=event["user_id"],
        timestamp=datetime.fromisoformat(event["timestamp"]),
        value=event["amount"],
    )

📦 Batch Pipelines

  • Latency: Hours
  • Complexity: Low to medium
  • Cost: Pay per run (spot instances)
  • Debugging: Easy (replay input data)
  • Best for: Historical aggregations, daily features

⚡ Streaming Pipelines

  • Latency: Seconds to minutes
  • Complexity: High
  • Cost: Always-on infrastructure
  • Debugging: Hard (state, ordering, late events)
  • Best for: Real-time signals, session features

Lambda Architecture for Features

The Lambda architecture combines batch and streaming pipelines to get the best of both worlds: the accuracy of batch processing with the freshness of streaming. For features, this means maintaining two parallel computation paths that merge at serving time.

How It Works for Features

  1. Batch layer: Computes accurate, comprehensive features on a schedule (e.g., hourly). These are the "ground truth" values.
  2. Speed layer: Computes approximate real-time features from the event stream. These fill the gap between batch runs.
  3. Serving layer: Merges both by using the streaming value if it's newer than the last batch materialization, otherwise using the batch value.
# lambda_merge.py — Merging batch and streaming feature values

from datetime import datetime

class LambdaFeatureServer:
    def __init__(self, batch_store, stream_store):
        self.batch = batch_store
        self.stream = stream_store

    def get_feature(self, entity_key: str, feature_name: str) -> dict:
        batch_val = self.batch.get(entity_key, feature_name)
        stream_val = self.stream.get(entity_key, feature_name)

        # Use streaming value if it's fresher than the batch value
        if stream_val and batch_val:
            if stream_val["timestamp"] > batch_val["timestamp"]:
                return stream_val
            return batch_val
        elif stream_val:
            return stream_val
        return batch_val
Lambda Complexity: Maintaining two parallel codepaths (batch + streaming) for the same feature is operationally expensive. Every logic change must be applied in both places. Consider the Kappa architecture (streaming-only) if your infrastructure supports replaying events from the stream.

Pipeline Orchestration

Feature pipelines don't run in isolation—they're part of a larger DAG (Directed Acyclic Graph) of data transformations. Orchestration tools manage scheduling, dependencies, retries, and monitoring.

Airflow DAG for Feature Pipelines

# feature_dag.py — Airflow DAG for feature computation

from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.sensors.external_task import ExternalTaskSensor
from datetime import datetime, timedelta

default_args = {
    "owner": "ml-platform",
    "retries": 3,
    "retry_delay": timedelta(minutes=10),
    "email_on_failure": True,
}

dag = DAG(
    "feature_pipeline_daily",
    default_args=default_args,
    schedule_interval="@daily",
    start_date=datetime(2024, 1, 1),
    catchup=False,
)

# Wait for upstream data to be ready
wait_for_data = ExternalTaskSensor(
    task_id="wait_for_transactions_etl",
    external_dag_id="transactions_etl",
    external_task_id="load_complete",
    dag=dag,
)

# Compute features
compute_features = PythonOperator(
    task_id="compute_user_features",
    python_callable=compute_user_features,
    op_kwargs={"run_date": "{{ ds }}"},
    dag=dag,
)

# Materialize to online store
materialize_online = PythonOperator(
    task_id="materialize_to_online_store",
    python_callable=materialize_to_redis,
    op_kwargs={"feature_table": "user_features"},
    dag=dag,
)

# Validate data quality
validate_features = PythonOperator(
    task_id="validate_feature_quality",
    python_callable=run_feature_validation,
    dag=dag,
)

# Define task dependencies
wait_for_data >> compute_features >> [materialize_online, validate_features]
Orchestration Tip: Always include a data validation step after computing features but before materializing to the online store. This prevents bad data from reaching production models. Gate the materialization on passing quality checks.

Prefect Alternative

# prefect_pipeline.py — Prefect flow for feature computation

from prefect import flow, task
from prefect.tasks import task_input_hash
from datetime import timedelta

@task(cache_key_fn=task_input_hash, cache_expiration=timedelta(hours=1))
def extract_raw_data(run_date: str) -> "DataFrame":
    return spark.read.parquet(f"s3://data-lake/transactions/date={run_date}")

@task
def compute_features(raw_data) -> "DataFrame":
    return raw_data.groupBy("user_id").agg("...")

@task
def validate_and_materialize(features) -> None:
    if not quality_checks_pass(features):
        raise ValueError("Feature quality checks failed")
    write_to_feature_store(features)

@flow(name="daily-feature-pipeline")
def feature_pipeline(run_date: str):
    raw = extract_raw_data(run_date)
    features = compute_features(raw)
    validate_and_materialize(features)

Testing Feature Pipelines

Feature pipelines are data transformation code, and they deserve the same testing rigor as application code. Untested pipelines silently produce incorrect features, leading to degraded model performance that's extremely difficult to diagnose.

Unit Tests for Transform Logic

# test_features.py — Unit tests for feature transformations

import pytest
import pandas as pd
from datetime import datetime
from features.user_features import compute_user_features

class TestUserFeatures:
    def test_transaction_count(self):
        # Arrange: Create test input data
        transactions = pd.DataFrame({
            "user_id": ["u1", "u1", "u1", "u2"],
            "amount": [10.0, 20.0, 30.0, 50.0],
            "event_date": [datetime(2024, 1, i) for i in range(1, 5)],
        })

        # Act: Run the feature transform
        result = compute_user_features(transactions)

        # Assert: Verify outputs
        u1 = result[result["user_id"] == "u1"].iloc[0]
        assert u1["txn_count_30d"] == 3
        assert u1["avg_txn_amount_30d"] == pytest.approx(20.0)

    def test_handles_empty_input(self):
        # Edge case: no transactions for a user
        transactions = pd.DataFrame(
            columns=["user_id", "amount", "event_date"]
        )
        result = compute_user_features(transactions)
        assert len(result) == 0

    def test_null_amounts_handled(self):
        transactions = pd.DataFrame({
            "user_id": ["u1", "u1"],
            "amount": [10.0, None],
            "event_date": [datetime(2024, 1, 1), datetime(2024, 1, 2)],
        })
        result = compute_user_features(transactions)
        assert not pd.isna(result.iloc[0]["txn_count_30d"])

Data Validation with Great Expectations

# validate_features.py — Data quality checks

import great_expectations as gx

def validate_user_features(features_df) -> bool:
    context = gx.get_context()
    validator = context.sources.pandas_default.read_dataframe(features_df)

    # Schema expectations
    validator.expect_column_to_exist("user_id")
    validator.expect_column_to_exist("txn_count_30d")

    # Value range expectations
    validator.expect_column_values_to_be_between(
        "txn_count_30d", min_value=0, max_value=10000
    )
    validator.expect_column_values_to_not_be_null("user_id")

    # Distribution expectations (detect drift)
    validator.expect_column_mean_to_be_between(
        "avg_txn_amount_30d", min_value=5.0, max_value=500.0
    )

    result = validator.validate()
    return result.success
Testing Best Practice: Implement three layers of testing for feature pipelines: (1) unit tests for individual transformations, (2) data validation checks on outputs, and (3) integration tests that verify end-to-end pipeline execution on sample data. Run all three in CI before deploying pipeline changes.
Silent Failures: Feature pipelines rarely crash—they produce wrong numbers. Without data validation, a broken feature can silently degrade model performance for weeks before anyone notices. Always validate output distributions against historical baselines.

With robust feature pipelines in place—batch for historical accuracy, streaming for freshness, and thorough testing for correctness—your feature store becomes a reliable foundation for production ML systems.
