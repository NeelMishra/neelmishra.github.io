Feature Transforms: Raw → Production-Ready

MLOps Series Feature Stores

Why Transform Features?

Raw data from databases, APIs, and event streams is almost never suitable for direct consumption by ML models. Column scales differ by orders of magnitude, missing values lurk in every dataset, and categorical variables need encoding before numeric algorithms can use them.

Feature transforms bridge the gap between raw data and model-ready inputs. Done well, they improve model accuracy, speed up convergence, and ensure reproducibility across training and serving.

The Three Pillars of Feature Transformation

  • Consistency — the same transform logic must run in training and serving to avoid training-serving skew.
  • Reproducibility — transforms must be versioned and deterministic so experiments can be replicated.
  • Efficiency — production transforms must run within latency budgets (often single-digit milliseconds).
Rule of Thumb: If you cannot reproduce a feature value given the same raw inputs and the same transform code, you have a reproducibility bug.

Normalization and Scaling

Most gradient-based models (linear regression, neural networks, SVMs) are sensitive to feature scale. A feature ranging from 0 to 1 million will dominate a feature ranging from 0 to 1 during optimization. Scaling corrects this.

StandardScaler

Centers data to mean=0, std=1. Best when data is approximately Gaussian.

z = (x − μ) / σ

MinMaxScaler

Scales data to a fixed range [0, 1]. Best when you need bounded outputs.

z = (x − min) / (max − min)

RobustScaler

Uses median and IQR instead of mean and std. Robust to outliers.

z = (x − median) / IQR

MaxAbsScaler

Scales by dividing by the max absolute value. Preserves sparsity in data.

z = x / |max(x)|

Python Implementation

import numpy as np
from sklearn.preprocessing import StandardScaler, MinMaxScaler, RobustScaler

# Sample data: income and age on very different scales
X = np.array([
    [50000, 25], [80000, 35], [120000, 45],
    [45000, 22], [200000, 55], [60000, 30]
])

# StandardScaler: mean=0, std=1
standard = StandardScaler()
X_standard = standard.fit_transform(X)
print("StandardScaler:\n", X_standard[:3])

# MinMaxScaler: range [0, 1]
minmax = MinMaxScaler()
X_minmax = minmax.fit_transform(X)
print("MinMaxScaler:\n", X_minmax[:3])

# RobustScaler: robust to outliers
robust = RobustScaler()
X_robust = robust.fit_transform(X)
print("RobustScaler:\n", X_robust[:3])
Warning: Always call fit() on training data only, then transform() on both train and test. Calling fit_transform() on the full dataset causes data leakage through statistics.

Missing Value Imputation

Real-world datasets are riddled with missing values. Sensors fail, users skip form fields, and upstream pipelines crash. Imputation fills these gaps with reasonable estimates so models can use the data.

Common Imputation Strategies

Mean / Median

Fast and simple. Median is preferred for skewed distributions. Does not capture feature interactions.

KNN Imputation

Uses K nearest neighbors to fill missing values. Captures local structure but is slower and memory-intensive.

Imputation with Indicator Columns

Sometimes the fact that a value is missing is itself informative. Adding a binary indicator column preserves this signal.

import pandas as pd
import numpy as np
from sklearn.impute import SimpleImputer, KNNImputer

df = pd.DataFrame({
    "age": [25, np.nan, 35, 40, np.nan, 55],
    "income": [50000, 60000, np.nan, 80000, 70000, np.nan],
    "score": [0.8, 0.6, 0.9, np.nan, 0.7, 0.5]
})

# Add missingness indicators before imputing
for col in df.columns:
    df[f"{col}_missing"] = df[col].isna().astype(int)

# Strategy 1: Median imputation
median_imputer = SimpleImputer(strategy="median")
df_median = pd.DataFrame(
    median_imputer.fit_transform(df[["age", "income", "score"]]),
    columns=["age", "income", "score"]
)

# Strategy 2: KNN imputation (k=3)
knn_imputer = KNNImputer(n_neighbors=3)
df_knn = pd.DataFrame(
    knn_imputer.fit_transform(df[["age", "income", "score"]]),
    columns=["age", "income", "score"]
)

print("Median imputation:\n", df_median)
print("KNN imputation:\n", df_knn)
Tip: For production pipelines, prefer SimpleImputer with median. KNN imputation is powerful but slow at scale and requires all numeric features to be present at serving time.

Bucketing and Binning

Bucketing transforms continuous features into discrete bins. This can help models capture non-linear relationships, reduce the impact of outliers, and make features more interpretable.

Three Binning Strategies

  • Equal-Width Bins — divide the range into N equal-sized intervals. Simple but sensitive to outliers.
  • Quantile-Based Bins — each bin contains roughly the same number of samples. Better for skewed data.
  • Custom Bins — domain-driven boundaries (e.g., age groups: 0–18, 18–35, 35–55, 55+).
import pandas as pd
import numpy as np

ages = pd.Series([18, 22, 25, 30, 35, 40, 45, 55, 60, 70, 85])

# Equal-width bins
equal_bins = pd.cut(ages, bins=4, labels=["young", "mid", "senior", "elder"])
print("Equal-width:\n", equal_bins.value_counts())

# Quantile-based bins (roughly equal frequency)
quantile_bins = pd.qcut(ages, q=4, labels=["Q1", "Q2", "Q3", "Q4"])
print("Quantile:\n", quantile_bins.value_counts())

# Custom domain-driven bins
custom_edges = [0, 18, 35, 55, 100]
custom_labels = ["minor", "young_adult", "middle_aged", "senior"]
custom_bins = pd.cut(ages, bins=custom_edges, labels=custom_labels)
print("Custom:\n", custom_bins.value_counts())
When to Bucket: Bucketing works best for tree-based models (XGBoost, LightGBM) or when you want interpretable features. For neural networks, prefer raw scaled values.

Feature Crosses

Feature crosses combine two or more features into a single new feature that captures their interaction. This is especially valuable for linear models that cannot learn interactions on their own.

Types of Feature Crosses

  • Categorical × Categorical — concatenation: city + "_" + device_type → "NYC_mobile"
  • Categorical × Numeric — separate numeric stats per category: avg_spend_by_city
  • Numeric × Numeric — polynomial interaction: age * income
import pandas as pd
import numpy as np
from sklearn.preprocessing import PolynomialFeatures

df = pd.DataFrame({
    "city": ["NYC", "LA", "NYC", "LA", "SF"],
    "device": ["mobile", "desktop", "desktop", "mobile", "mobile"],
    "age": [25, 35, 45, 30, 28],
    "income": [50000, 80000, 120000, 60000, 70000]
})

# Categorical cross
df["city_device"] = df["city"] + "_" + df["device"]
print("Categorical cross:\n", df["city_device"])

# Numeric interaction
df["age_income"] = df["age"] * df["income"]
print("Numeric interaction:\n", df["age_income"])

# Polynomial features (degree 2)
poly = PolynomialFeatures(degree=2, interaction_only=True, include_bias=False)
X_numeric = df[["age", "income"]].values
X_poly = poly.fit_transform(X_numeric)
print("Polynomial features shape:", X_poly.shape)
print("Feature names:", poly.get_feature_names_out())
Warning: Feature crosses can explode the dimensionality of your dataset. A cross of two 1,000-cardinality categoricals creates up to 1,000,000 new values. Use hashing or embedding to keep this manageable.

Building a Transform Pipeline

In production, you never apply transforms individually. Instead, you compose them into a pipeline that chains preprocessing steps and ensures they execute in the same order during training and serving.

sklearn ColumnTransformer + Pipeline

from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import (
    StandardScaler, OneHotEncoder, KBinsDiscretizer
)
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression
import numpy as np

# Define column groups
numeric_cols = ["age", "income", "score"]
categorical_cols = ["city", "device"]
binned_cols = ["age"]

# Numeric sub-pipeline: impute → scale
numeric_pipe = Pipeline([
    ("imputer", SimpleImputer(strategy="median")),
    ("scaler", StandardScaler()),
])

# Categorical sub-pipeline: impute → one-hot encode
categorical_pipe = Pipeline([
    ("imputer", SimpleImputer(strategy="constant", fill_value="missing")),
    ("encoder", OneHotEncoder(handle_unknown="ignore", sparse_output=False)),
])

# Compose all transforms
preprocessor = ColumnTransformer([
    ("num", numeric_pipe, numeric_cols),
    ("cat", categorical_pipe, categorical_cols),
])

# Full pipeline: preprocess → model
model_pipeline = Pipeline([
    ("preprocess", preprocessor),
    ("classifier", LogisticRegression(max_iter=1000)),
])

# Fit and predict
# model_pipeline.fit(X_train, y_train)
# predictions = model_pipeline.predict(X_test)
print("Pipeline steps:", model_pipeline.named_steps)

Saving and Loading Pipelines

import joblib

# Save the fitted pipeline
joblib.dump(model_pipeline, "model_pipeline.joblib")

# Load in serving environment
loaded_pipeline = joblib.load("model_pipeline.joblib")
# predictions = loaded_pipeline.predict(new_data)
Production Tip: Serialize your pipeline with joblib or pickle and version it alongside your model artifacts. This ensures the exact same transforms run during training and serving — eliminating training-serving skew.
Common Pitfall: Fitting a scaler on the test set or the full dataset. Always fit() on training data only, then transform() on test/production data. This applies to every step in your pipeline.
