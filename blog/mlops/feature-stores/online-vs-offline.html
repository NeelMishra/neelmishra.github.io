Online vs Offline Feature Serving

MLOps Series Feature Stores

Two Modes of Feature Serving

Feature stores must serve data in two fundamentally different modes: online serving for real-time inference and offline serving for model training. Each mode has distinct latency, throughput, and consistency requirements.

⚡ Online Serving

  • Low-latency key-value lookups
  • Single entity at a time
  • Latest feature values only
  • Backed by Redis, DynamoDB, etc.

📦 Offline Serving

  • Batch retrieval of historical data
  • Millions of rows at once
  • Point-in-time correct joins
  • Backed by warehouses/data lakes

Online Feature Serving

Online serving provides low-latency access to the most recent feature values for a given entity. When your model receives an inference request, it needs features in single-digit milliseconds to make a prediction without degrading user experience.

Key Characteristics

  • Access pattern: Point lookups by entity key (e.g., user_id, item_id)
  • Latency target: p50 < 10ms, p99 < 50ms
  • Data freshness: Seconds to minutes (streaming) or hours (batch materialization)
  • Storage: In-memory or SSD-backed key-value stores
# online_serving.py — Low-latency feature retrieval

import redis
import json

class OnlineFeatureStore:
    def __init__(self, host="redis.cluster.internal", port=6379):
        self.client = redis.Redis(host=host, port=port, decode_responses=True)

    def get_features(self, entity_key: str, feature_names: list) -> dict:
        # Single round-trip using Redis HMGET for multiple features
        cache_key = f"features:{entity_key}"
        values = self.client.hmget(cache_key, feature_names)

        result = {}
        for name, value in zip(feature_names, values):
            if value is not None:
                result[name] = json.loads(value)
            else:
                result[name] = None  # Feature missing / not materialized
        return result

    def write_features(self, entity_key: str, features: dict, ttl: int = 86400):
        # Atomic write with expiration
        cache_key = f"features:{entity_key}"
        pipeline = self.client.pipeline()
        for name, value in features.items():
            pipeline.hset(cache_key, name, json.dumps(value))
        pipeline.expire(cache_key, ttl)
        pipeline.execute()
Performance Tip: Always batch feature reads into a single round-trip using HMGET or pipeline commands. Each additional network round-trip adds 0.5–2ms of latency in cloud environments.

Offline Feature Serving

Offline serving provides historical feature values for model training. The critical challenge here is point-in-time correctness—ensuring that training data only uses features that were available at the time of each training example, preventing data leakage.

Point-in-Time Joins

A point-in-time join matches each training event with the feature values that existed just before that event's timestamp. This prevents future information from leaking into training data.

# offline_serving.py — Point-in-time correct feature retrieval

import pandas as pd

def point_in_time_join(
    entity_df: pd.DataFrame,
    feature_df: pd.DataFrame,
    entity_key: str,
    event_timestamp_col: str = "event_timestamp",
    feature_timestamp_col: str = "feature_timestamp",
) -> pd.DataFrame:
    # Sort features by timestamp for efficient as-of join
    feature_df = feature_df.sort_values(feature_timestamp_col)

    # Merge on entity key
    merged = pd.merge_asof(
        entity_df.sort_values(event_timestamp_col),
        feature_df,
        left_on=event_timestamp_col,
        right_on=feature_timestamp_col,
        by=entity_key,
        direction="backward",  # Only use features from before the event
    )

    return merged

# Usage: build a training dataset
training_events = pd.read_parquet("s3://ml-data/training_events.parquet")
user_features = pd.read_parquet("s3://features/user_purchase_stats/")

training_data = point_in_time_join(
    entity_df=training_events,
    feature_df=user_features,
    entity_key="user_id",
)
Data Leakage Warning: Without point-in-time joins, your model may train on features computed after the label event. This leads to inflated offline metrics that don't reproduce in production—one of the most common and costly ML bugs.

Offline Retrieval at Scale

For large-scale training, use your data warehouse or Spark to perform point-in-time joins efficiently:

# warehouse_offline.py — SQL-based point-in-time join

def generate_pit_query(entity_table: str, feature_table: str, entity_key: str) -> str:
    return f"""
    SELECT e.*, f.*
    FROM {entity_table} e
    LEFT JOIN LATERAL (
        SELECT *
        FROM {feature_table} f
        WHERE f.{entity_key} = e.{entity_key}
          AND f.feature_timestamp <= e.event_timestamp
        ORDER BY f.feature_timestamp DESC
        LIMIT 1
    ) f ON TRUE
    """

query = generate_pit_query(
    entity_table="training_events",
    feature_table="user_purchase_stats",
    entity_key="user_id",
)

Storage Backends

Choosing the right storage backend is critical for both performance and cost. Here's how popular options compare across key dimensions:

⚡ Redis

  • Type: In-memory key-value
  • Latency: Sub-millisecond reads
  • Pros: Fastest reads, native TTL, clustering
  • Cons: Memory-bound (expensive at scale), no built-in persistence guarantees
  • Best for: Ultra-low-latency online serving

🔧 DynamoDB

  • Type: Managed NoSQL
  • Latency: Single-digit ms reads
  • Pros: Fully managed, auto-scaling, durable
  • Cons: Higher latency than Redis, cost scales with throughput
  • Best for: Managed online serving with durability

📊 BigQuery

  • Type: Serverless data warehouse
  • Latency: Seconds to minutes
  • Pros: Massive scale, SQL interface, cost-effective storage
  • Cons: Not suitable for online serving, cold-start latency
  • Best for: Offline feature storage and training data generation

❄️ Snowflake

  • Type: Cloud data warehouse
  • Latency: Seconds to minutes
  • Pros: Separation of compute/storage, time travel, strong SQL
  • Cons: Warehouse spin-up latency, not for real-time lookups
  • Best for: Offline serving with advanced SQL transformations
Hybrid Pattern: Most production feature stores use a dual-store architecture—a fast key-value store (Redis/DynamoDB) for online serving and a data warehouse (BigQuery/Snowflake) for offline training data. The feature store handles synchronization between the two.

Consistency and Sync Patterns

Keeping online and offline stores in sync is one of the hardest challenges in feature store design. There are three primary synchronization patterns, each with different consistency guarantees.

1. Scheduled Materialization

A batch job periodically computes features and writes them to both online and offline stores. This is the simplest pattern but results in eventual consistency with staleness proportional to the schedule interval.

# materialization.py — Scheduled feature materialization

from datetime import datetime, timedelta

class FeatureMaterializer:
    def __init__(self, offline_store, online_store, compute_engine):
        self.offline = offline_store
        self.online = online_store
        self.engine = compute_engine

    def materialize(self, feature_view, start_time: datetime, end_time: datetime):
        # Step 1: Compute features using the configured engine
        features_df = self.engine.compute(feature_view, start_time, end_time)

        # Step 2: Write to offline store (append)
        self.offline.write(feature_view.name, features_df, mode="append")

        # Step 3: Write latest values to online store
        latest = features_df.sort_values("timestamp").groupby("entity_key").last()
        for entity_key, row in latest.iterrows():
            self.online.write_features(entity_key, row.to_dict())

2. Change Data Capture (CDC)

CDC streams database changes to the online store in near-real-time. When a feature value updates in the offline store, the change is captured and propagated, reducing staleness to seconds.

3. Event-Driven Sync

Features are computed from streaming events and written to both stores simultaneously. This provides the strongest consistency but requires streaming infrastructure.

Consistency Trade-off: Stronger consistency comes at higher infrastructure cost and complexity. Most teams start with scheduled materialization (hourly/daily) and add streaming only for features where freshness directly impacts model performance.

Choosing Your Serving Strategy

Your serving strategy should be driven by your model's latency requirements and feature freshness needs. Here's a decision framework:

Decision Tree

  1. Is the model serving real-time predictions?
    • Yes → You need an online store
    • No (batch scoring only) → Offline store is sufficient
  2. Do features need sub-minute freshness?
    • Yes → Streaming materialization + online store
    • No → Scheduled batch materialization is fine
  3. Do you need point-in-time correct training data?
    • Yes → You need an offline store with timestamp tracking
    • No (static features) → Simple table snapshots work
Recommendation: Start with a dual-store architecture from day one, even if you only use batch materialization initially. Migrating from offline-only to dual-store later requires significant refactoring of your feature definitions and serving code.

🚀 Start Simple

  • Redis + Parquet files
  • Hourly batch materialization
  • Works for most early-stage ML
  • Upgrade components independently

🏢 Scale Up

  • DynamoDB + Snowflake/BigQuery
  • Streaming + batch pipelines
  • Point-in-time join framework
  • Monitoring and alerting layer
Common Mistake: Don't over-engineer your serving infrastructure before you've validated your features matter. Start with batch-materialized features in a simple online store. Add streaming and complex sync patterns only when you've proven that feature freshness improves your model's business metrics.
