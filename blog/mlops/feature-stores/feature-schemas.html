Feature Schemas, Types, and Validation

MLOps Series Feature Stores

Why Feature Schemas Matter

A feature schema is a formal contract that defines the name, data type, allowed values, and constraints for every feature in your store. Without schemas, feature stores become fragile data swamps where silent type mismatches and unexpected nulls corrupt model predictions.

Data Quality

Schemas act as the first line of defense for data quality. They catch issues at write time rather than letting bad data propagate to model serving:

  • A feature expected to be a float between 0 and 1 receives an integer value of 150—the schema rejects it before it enters the store.
  • A string categorical feature receives an empty string instead of a valid category—the schema flags the violation.
  • An embedding feature expected to be 768 dimensions receives a 512-dimensional vector—the schema prevents a downstream shape mismatch.

Breaking Changes

Without schema enforcement, upstream changes silently break downstream consumers. A data engineer renames a column, changes a type from int to string, or removes a field—and the model serving pipeline fails at prediction time instead of at the point of change.

Warning: In a survey of ML teams, schema-related issues were the #2 cause of production model incidents, behind only data pipeline failures. Most of these could have been prevented with schema validation at the feature store boundary.

Downstream Impact

A single schema violation can cascade across multiple models. If a shared feature like user_lifetime_value changes type from float to string, every model consuming that feature will produce incorrect predictions or crash—potentially affecting dozens of downstream services simultaneously.

Feature Types and Definitions

Feature stores must support a rich type system that goes beyond basic scalar types. Here are the common categories:

Type Categories

  • Scalar types: int, float, bool, string—the most common feature types for tabular data.
  • Vector types: Fixed-length arrays of floats, typically used for embeddings (e.g., 768-dim BERT embedding).
  • Struct types: Nested objects with named fields, used for complex features like user profiles or address records.
  • List types: Variable-length sequences such as recent transaction amounts or clickstream event IDs.
  • Map types: Key-value pairs, useful for sparse categorical features or dynamic attribute sets.
from dataclasses import dataclass, field
from typing import List, Dict, Optional, Any
from enum import Enum

class FeatureType(Enum):
    INT = "int"
    FLOAT = "float"
    BOOL = "bool"
    STRING = "string"
    VECTOR = "vector"
    STRUCT = "struct"
    LIST = "list"
    MAP = "map"

@dataclass
class FeatureSchema:
    name: str
    dtype: FeatureType
    description: str
    nullable: bool = False
    default: Optional[Any] = None
    constraints: Dict[str, Any] = field(default_factory=dict)
    # Vector-specific: expected dimensionality
    dimensions: Optional[int] = None
    # Struct-specific: nested schema
    fields: Optional[List["FeatureSchema"]] = None

# Define schemas for a user feature group
user_schemas = [
    FeatureSchema(
        name="user_age",
        dtype=FeatureType.INT,
        description="User age in years",
        constraints={"min": 0, "max": 150},
    ),
    FeatureSchema(
        name="user_embedding",
        dtype=FeatureType.VECTOR,
        description="User behavioral embedding from BERT",
        dimensions=768,
    ),
    FeatureSchema(
        name="purchase_history",
        dtype=FeatureType.LIST,
        description="Last 50 purchase amounts",
        constraints={"max_length": 50, "element_type": "float"},
    ),
    FeatureSchema(
        name="user_preferences",
        dtype=FeatureType.MAP,
        description="User preference scores by category",
        constraints={"key_type": "string", "value_type": "float"},
    ),
]
Tip: Always include a description field in your schema definitions. This serves as built-in documentation and makes feature discovery significantly easier for other teams.

Validation Rules

Beyond type checking, validation rules enforce semantic constraints on feature values. These catch data issues that are type-correct but logically invalid.

Common Validation Patterns

  • Range checks: Numeric values must fall within expected bounds (e.g., age between 0 and 150).
  • Null handling: Some features must never be null; others have defined default fill values.
  • Distribution checks: Feature value distributions should not drift beyond expected statistical bounds.
  • Uniqueness constraints: Entity keys must be unique within a feature group.
  • Referential integrity: Foreign key features must reference valid entities in other feature groups.
from typing import Any, Callable, List, Optional
import numpy as np

class ValidationRule:
    def __init__(self, name: str, check_fn: Callable, severity: str = "error"):
        self.name = name
        self.check_fn = check_fn
        self.severity = severity  # "error" blocks write; "warning" logs only

    def validate(self, value: Any) -> dict:
        passed = self.check_fn(value)
        return {"rule": self.name, "passed": passed, "severity": self.severity}

class FeatureValidator:
    def __init__(self, schema: "FeatureSchema", rules: List[ValidationRule]):
        self.schema = schema
        self.rules = rules

    def validate_value(self, value: Any) -> dict:
        results = []
        # Type check
        if value is None and not self.schema.nullable:
            return {"valid": False, "errors": ["Null value not allowed"]}

        # Apply all rules
        for rule in self.rules:
            result = rule.validate(value)
            results.append(result)

        errors = [r for r in results
                  if not r["passed"] and r["severity"] == "error"]
        warnings = [r for r in results
                    if not r["passed"] and r["severity"] == "warning"]

        return {
            "valid": len(errors) == 0,
            "errors": [e["rule"] for e in errors],
            "warnings": [w["rule"] for w in warnings],
        }

# Example: validate user_age
age_rules = [
    ValidationRule("range_check", lambda v: 0 <= v <= 150),
    ValidationRule("not_negative", lambda v: v >= 0),
    ValidationRule(
        "distribution_check",
        lambda v: 10 <= v <= 100,
        severity="warning",  # flag unusual ages, don't block
    ),
]
validator = FeatureValidator(user_schemas[0], age_rules)
print(validator.validate_value(25))   # valid
print(validator.validate_value(-5))   # error: range_check, not_negative
print(validator.validate_value(105))  # warning: distribution_check
Warning: Do not make all validation rules blocking errors. Use a severity field to distinguish hard errors (reject the write) from warnings (log and allow). Over-strict validation causes pipeline failures for edge cases that may be valid.

Schema Evolution

Feature schemas are not static. As models evolve and data sources change, schemas must evolve too—without breaking existing consumers. This is schema evolution.

Safe vs. Breaking Changes

Not all schema changes are equal:

  • Safe (backward-compatible): Adding a new nullable feature, widening a type (int → float), increasing a range constraint.
  • Breaking: Removing a feature, renaming a feature, narrowing a type (float → int), tightening constraints.

Versioning Strategy

Maintain explicit schema versions so consumers can pin to a known-good version while producers migrate:

from dataclasses import dataclass, field
from typing import List, Optional
from datetime import datetime

@dataclass
class SchemaVersion:
    version: int
    features: List["FeatureSchema"]
    created_at: datetime
    description: str
    is_active: bool = True

@dataclass
class FeatureGroupSchema:
    name: str
    versions: List[SchemaVersion] = field(default_factory=list)

    def add_version(self, features: List["FeatureSchema"], desc: str):
        new_version = SchemaVersion(
            version=len(self.versions) + 1,
            features=features,
            created_at=datetime.utcnow(),
            description=desc,
        )
        self.versions.append(new_version)
        return new_version

    def get_active(self) -> Optional[SchemaVersion]:
        for v in reversed(self.versions):
            if v.is_active:
                return v
        return None

    def check_compatibility(self, old_v: int, new_v: int) -> dict:
        old_schema = self.versions[old_v - 1]
        new_schema = self.versions[new_v - 1]
        old_names = {f.name for f in old_schema.features}
        new_names = {f.name for f in new_schema.features}
        removed = old_names - new_names
        added = new_names - old_names
        return {
            "backward_compatible": len(removed) == 0,
            "added_features": added,
            "removed_features": removed,
        }

# Usage
schema = FeatureGroupSchema(name="user_features")
schema.add_version(user_schemas, "Initial schema")
# Later: add a new feature
extended = user_schemas + [FeatureSchema(
    name="user_churn_score", dtype=FeatureType.FLOAT,
    description="Predicted churn probability", nullable=True,
)]
schema.add_version(extended, "Added churn score")
compat = schema.check_compatibility(1, 2)
print(compat)  # backward_compatible: True, added: {user_churn_score}
Tip: Treat feature schema evolution like database migrations. Maintain a changelog, require reviews for breaking changes, and provide a deprecation window (typically 2–4 weeks) before removing features.

Enforcement in Production

Schemas are only useful if they are enforced. Enforcement should happen at multiple layers: during development, in CI/CD, and at runtime.

CI/CD for Schemas

Integrate schema validation into your CI/CD pipeline so that schema changes are reviewed and tested before deployment:

import json
import sys
from typing import Dict, List

def validate_schema_change(old_path: str, new_path: str) -> Dict:
    # Load schemas from JSON files
    with open(old_path) as f:
        old = json.load(f)
    with open(new_path) as f:
        new = json.load(f)

    old_features = {feat["name"] for feat in old["features"]}
    new_features = {feat["name"] for feat in new["features"]}

    removed = old_features - new_features
    if removed:
        print(f"BREAKING: Features removed: {removed}")
        sys.exit(1)

    # Check type narrowing
    for new_feat in new["features"]:
        old_feat = next(
            (f for f in old["features"]
             if f["name"] == new_feat["name"]), None
        )
        if old_feat and old_feat["dtype"] != new_feat["dtype"]:
            print(f"BREAKING: Type changed for {new_feat['name']}")
            sys.exit(1)

    print("Schema change is backward-compatible")
    return {"compatible": True}

# Pre-commit hook entry point
if __name__ == "__main__":
    validate_schema_change(sys.argv[1], sys.argv[2])

Runtime Validation

At write time, validate every feature value against its schema before persisting to the store. At read time, optionally validate that retrieved values still conform (to catch storage-level corruption).

class SchemaEnforcedStore:
    def __init__(self, store, schemas: Dict[str, "FeatureValidator"]):
        self.store = store
        self.schemas = schemas

    def write_feature(self, entity_id: str, feature_name: str, value):
        validator = self.schemas.get(feature_name)
        if validator:
            result = validator.validate_value(value)
            if not result["valid"]:
                raise ValueError(
                    f"Validation failed for {feature_name}: "
                    f"{result['errors']}"
                )
        self.store.write(entity_id, feature_name, value)

    def read_feature(self, entity_id: str, feature_name: str):
        value = self.store.read(entity_id, feature_name)
        # Optional: validate on read for safety
        validator = self.schemas.get(feature_name)
        if validator:
            result = validator.validate_value(value)
            if not result["valid"]:
                print(f"WARN: Stored value invalid for {feature_name}")
        return value
Tip: Start with write-time enforcement only. Read-time validation adds latency and should be reserved for high-criticality features or debugging scenarios.

Schema Tooling Comparison

Several tools can help you implement feature schema validation. Here is a comparison of the most popular options:

📚 Great Expectations

The most mature data validation framework. Rich suite of built-in expectations, HTML documentation generation, and integration with major data platforms.

  • Pros: 300+ built-in checks, auto-profiling, rich docs
  • Cons: Heavy dependency, steep learning curve, slower for real-time
  • Best for: Batch validation and data quality reporting

🐔 Pandera

Pandas-native schema validation. Lightweight, Pythonic API that integrates seamlessly with pandas and pyspark DataFrames.

  • Pros: Lightweight, Pythonic, supports hypothesis testing
  • Cons: Pandas/PySpark only, fewer integrations
  • Best for: Python-native feature pipelines

🔧 Custom Validation

Build your own validation layer using dataclasses, Pydantic, or simple functions. Full control over behavior and performance.

  • Pros: Zero dependencies, fastest runtime, full control
  • Cons: Maintenance burden, no built-in profiling or docs
  • Best for: Real-time serving and latency-sensitive paths

🛠️ Pydantic

Data validation using Python type annotations. Extremely popular for API validation and increasingly used for feature schemas.

  • Pros: Fast (V2 is Rust-backed), great error messages, widely adopted
  • Cons: Not designed for statistical checks, row-level only
  • Best for: Feature serving APIs and per-record validation
Recommendation: For most teams, combine Pandera for batch pipeline validation with Pydantic for serving-time validation. This gives you statistical checks in batch and low-latency checks at serving time.
Warning: Avoid building custom validation from scratch unless you have specific performance requirements that existing tools cannot meet. The maintenance cost of a bespoke framework almost always exceeds the cost of adapting an existing one.
