Point-in-Time Correctness

MLOps Series Feature Stores

What Is Point-in-Time Correctness?

Point-in-time (PIT) correctness ensures that when you retrieve feature values for a given entity at a specific moment, you only see data that was actually available at that moment. This seemingly simple requirement is one of the most critical — and most frequently violated — principles in production ML systems.

Imagine you are training a fraud-detection model. For each transaction, you need features like average transaction amount over the last 30 days. If your training pipeline accidentally includes transactions that happened after the event you are predicting, you have introduced future information — the model will look brilliant in training but fail catastrophically in production.

At t₂ the correct feature value is the one written at t₁. Using the value from t₃ would be a data leakage violation — the model would be consuming information from the future.

Key Principle: For any entity at time t, you must only use feature values with an event_timestamp ≤ t. The latest such value is the correct one.

Data Leakage Through Features

Data leakage is one of the most insidious bugs in ML. Unlike a crash or an error message, leakage silently inflates your offline metrics while the model underperforms in production. Feature leakage specifically refers to using information that would not be available at serving time.

Common Sources of Feature Leakage

  • Naive SQL joins — joining on entity key without a time constraint pulls in the latest value regardless of when it was computed.
  • Aggregation windows that extend into the future — a "30-day rolling average" that includes days after the label event.
  • Label encoding leak — encoding a categorical variable using statistics (e.g., target mean encoding) computed on the entire dataset, including test rows.
  • Late-arriving data — a feature that arrives hours after the event but gets back-dated to the event time.
Warning: A model trained with leaked features can show 95%+ AUC offline but drop to 60% once deployed. Always validate offline/online metric parity.

❌ Incorrect Join

A simple LEFT JOIN on user_id without a time filter. The joined feature row may contain data from after the prediction event, inflating accuracy.

SELECT t.*, f.avg_spend
FROM transactions t
LEFT JOIN features f
  ON t.user_id = f.user_id
-- No time constraint → leakage!

✅ Correct PIT Join

An as-of join that restricts features to those with event_time ≤ t and picks the most recent one. This guarantees no future data leaks in.

SELECT t.*, f.avg_spend
FROM transactions t
LEFT JOIN features f
  ON t.user_id = f.user_id
  AND f.event_time <= t.event_time
-- Correct: only past data

Consequences of Leakage

The damage goes beyond poor production accuracy. Leaked models may drive incorrect business decisions during A/B tests, waste engineering resources debugging "regressions," and erode stakeholder trust in ML systems.

Temporal Joins Explained

A temporal join (also called an as-of join) matches each row in a left table to the most recent row in the right table where the right table's timestamp is less than or equal to the left table's timestamp. This is the fundamental operation that enforces point-in-time correctness.

Event-Time Semantics

Every feature row carries two key timestamps:

  • Event time — when the real-world event occurred (e.g., a transaction happened).
  • Ingestion time — when the feature store received the row.

PIT correctness operates on event time. Ingestion time can lag behind event time due to batch processing delays, network latency, or upstream system outages.

As-of Join in SQL

SELECT e.entity_id,
       e.event_timestamp,
       f.feature_value
FROM entity_events e
LEFT JOIN LATERAL (
  SELECT feature_value
  FROM feature_table ft
  WHERE ft.entity_id = e.entity_id
    AND ft.event_timestamp <= e.event_timestamp
  ORDER BY ft.event_timestamp DESC
  LIMIT 1
) f ON TRUE

As-of Join in Pandas

import pandas as pd

# Both DataFrames must be sorted by the time column
entities = entities.sort_values("event_timestamp")
features = features.sort_values("event_timestamp")

result = pd.merge_asof(
    entities,
    features,
    on="event_timestamp",
    by="entity_id",
    direction="backward"   # only look at past values
)
print(result.head())
Tip: The direction="backward" parameter is what enforces PIT correctness in merge_asof. Never use "nearest" for training data — it can pull future values.

Implementing PIT Joins

Let's walk through a complete implementation of point-in-time correct feature retrieval using Python and pandas. We will build a reusable function that takes entity events and a feature table and returns PIT-correct features.

Step 1: Prepare the Data

import pandas as pd
from datetime import datetime

# Entity events — the moments we need predictions for
entity_df = pd.DataFrame({
    "user_id": ["u1", "u1", "u2", "u2"],
    "event_timestamp": pd.to_datetime([
        "2024-01-10 10:00", "2024-01-15 14:00",
        "2024-01-12 09:00", "2024-01-18 16:00"
    ])
})

# Feature table with timestamped values
feature_df = pd.DataFrame({
    "user_id": ["u1", "u1", "u1", "u2", "u2"],
    "event_timestamp": pd.to_datetime([
        "2024-01-05", "2024-01-12", "2024-01-20",
        "2024-01-08", "2024-01-15"
    ]),
    "avg_spend_30d": [120.5, 135.0, 142.8, 89.0, 95.5]
})

Step 2: Build the PIT Join Function

def pit_join(entity_df, feature_df, entity_key, time_col):
    """Perform a point-in-time correct join."""
    # Sort both frames by time
    left = entity_df.sort_values(time_col).copy()
    right = feature_df.sort_values(time_col).copy()

    # merge_asof requires the time column in both frames
    result = pd.merge_asof(
        left,
        right,
        on=time_col,
        by=entity_key,
        direction="backward",
        suffixes=("_entity", "_feature")
    )
    return result

pit_result = pit_join(entity_df, feature_df, "user_id", "event_timestamp")
print(pit_result)

Step 3: Verify the Results

For user u1 at 2024-01-10, the correct avg_spend_30d is 120.5 (from Jan 5). The value of 135.0 (Jan 12) must not be used because it lies in the future relative to the event.

Tip: Always add an assertion after PIT joins to verify that the feature timestamp is ≤ the entity event timestamp. This catches subtle bugs early.

PIT Joins in Feast

Feast, the most widely adopted open-source feature store, has built-in support for point-in-time correct joins through its get_historical_features API. This is the recommended way to retrieve training data with PIT guarantees.

Define Your Feature View

from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float64
from datetime import timedelta

user = Entity(name="user_id", join_keys=["user_id"])

user_features = FeatureView(
    name="user_spend_features",
    entities=[user],
    ttl=timedelta(days=90),
    schema=[
        Field(name="avg_spend_30d", dtype=Float64),
        Field(name="max_spend_30d", dtype=Float64),
    ],
    source=FileSource(
        path="data/user_spend.parquet",
        timestamp_field="event_timestamp",
    ),
)

Retrieve Historical Features with PIT

from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

# entity_df must have event_timestamp column
entity_df = pd.DataFrame({
    "user_id": ["u1", "u2", "u3"],
    "event_timestamp": pd.to_datetime([
        "2024-01-15", "2024-01-16", "2024-01-17"
    ])
})

training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "user_spend_features:avg_spend_30d",
        "user_spend_features:max_spend_30d",
    ],
).to_df()

print(training_df.head())
How Feast enforces PIT: Internally, get_historical_features performs an as-of join on the event_timestamp column. Features with a TTL that has expired (older than ttl from the entity timestamp) are returned as null.
Warning: If you forget to set ttl on your FeatureView, Feast will join features of any age. For rapidly changing features, always set a sensible TTL to avoid stale data.

Testing for Correctness

PIT correctness should never be assumed — it must be explicitly tested. Here are strategies to validate your pipeline.

Assertion-Based Validation

def validate_pit_correctness(result_df, entity_time_col, feature_time_col):
    """Assert that no feature timestamp exceeds the entity timestamp."""
    violations = result_df[
        result_df[feature_time_col] > result_df[entity_time_col]
    ]
    if not violations.empty:
        print(f"Found {len(violations)} PIT violations!")
        print(violations.head())
        raise ValueError("Point-in-time correctness violated")
    print("All rows pass PIT validation.")

# Run validation after every PIT join
validate_pit_correctness(
    pit_result,
    entity_time_col="event_timestamp_entity",
    feature_time_col="event_timestamp_feature"
)

Statistical Validation

Compare offline metrics with and without PIT enforcement. If removing PIT constraints significantly boosts metrics, you likely had leakage in a previous pipeline.

def detect_leakage_signal(pit_auc, naive_auc, threshold=0.05):
    """Flag potential leakage when naive join outperforms PIT join."""
    gap = naive_auc - pit_auc
    if gap > threshold:
        print(f"Leakage risk: naive AUC {naive_auc:.3f} vs PIT AUC {pit_auc:.3f}")
        print(f"Gap of {gap:.3f} exceeds threshold {threshold}")
        return True
    return False

detect_leakage_signal(pit_auc=0.82, naive_auc=0.94)
Common Mistake: Running PIT validation only at training time but not when building the feature table itself. If the upstream feature pipeline computes aggregates incorrectly, your PIT join downstream cannot fix it.

Checklist for PIT Testing

  • ☑ Add timestamp assertion to every PIT join output.
  • ☑ Compare offline/online metrics to detect leakage signals.
  • ☑ Unit test with known data where the expected output is hand-computed.
  • ☑ Audit upstream feature pipelines for future-looking aggregations.
  • ☑ Monitor feature freshness in production to catch stale-data issues.
