Entity Resolution for Feature Stores

MLOps Series Feature Stores

What Is Entity Resolution?

Entity resolution is the process of determining whether two or more records refer to the same real-world entity. In a feature store context, it's the foundation upon which all feature joins are built. If you can't reliably identify "which user" or "which product" a feature belongs to, your feature vectors will be wrong—and your model will learn from noise.

Consider a simple example: your payments system identifies users by payment_user_id, your marketing platform uses email_hash, and your mobile app assigns a device_id. A single person might have three different identifiers across these systems. Entity resolution is the discipline of linking them into a single canonical identity.

Why It Matters for Feature Stores

Feature stores join features from multiple data sources into a single feature vector for each entity. If entity resolution is broken, you'll either:

  • Miss features: Two records for the same user don't join, so the feature vector has nulls.
  • Mix features: Two different users get merged, so the model sees contradictory signals.
  • Duplicate features: The same entity appears multiple times, inflating counts and skewing distributions.
Core principle: Entity resolution isn't a one-time ETL step. It's a living process that must run continuously as new data arrives. Your feature store's correctness is only as good as your entity resolution pipeline.

Entity Keys and Design

The entity key is the join column used to look up features. Choosing the right key design is critical—it determines how features from different sources connect and how robust your system is to upstream changes.

Primary Keys

A primary key uniquely identifies an entity within a single data source. Examples: user_id in a users table, order_id in an orders table. Simple and reliable when you're within a single system.

Composite Keys

Some entities require multiple columns to be uniquely identified. A feature for "user behavior on a specific product" needs (user_id, product_id). Composite keys are common in feature stores that serve recommendation or ranking models.

Natural vs. Surrogate Keys

Natural keys carry business meaning (email, SSN, phone number) while surrogate keys are system-generated (UUIDs, auto-increment IDs). Feature stores typically use surrogate keys internally for performance, with a mapping layer for natural keys.

# Entity key design patterns
from dataclasses import dataclass
from typing import List, Tuple
import hashlib

@dataclass
class EntityKey:
    entity_type: str
    key_columns: List[str]
    description: str

# Simple primary key
user_key = EntityKey(
    entity_type="user",
    key_columns=["user_id"],
    description="Canonical user identifier from auth service"
)

# Composite key for user-product interactions
user_product_key = EntityKey(
    entity_type="user_product",
    key_columns=["user_id", "product_id"],
    description="User-product pair for recommendation features"
)

def create_surrogate_key(*natural_keys: str) -> str:
    # Generate a deterministic surrogate from natural keys
    combined = "|".join(str(k).lower().strip() for k in natural_keys)
    return hashlib.sha256(combined.encode()).hexdigest()[:16]

# Example: create a surrogate from email + phone
surrogate = create_surrogate_key("john@example.com", "+1-555-0123")
print(f"Surrogate key: {surrogate}")

Natural Keys

  • Pros: Human-readable, meaningful, easy to debug
  • Cons: Can change (email updates), may contain PII, inconsistent formatting
  • Use when: Keys are stable and non-sensitive (e.g., stock tickers)

Surrogate Keys

  • Pros: Immutable, compact, privacy-friendly, consistent
  • Cons: Opaque, require a mapping table, harder to debug
  • Use when: Keys might change or involve PII
Warning: Never use PII (emails, phone numbers, SSNs) as raw entity keys in your feature store. Hash them or map them to surrogates. A data breach that exposes your feature store shouldn't expose personal information.

Cross-System Identity

In any organization beyond a startup, user identity is fragmented across systems. The CRM knows crm_contact_id, the app knows app_user_id, the ad platform knows cookie_id, and the payment system knows stripe_customer_id. Cross-system identity resolution links these into a unified entity.

Identity Graphs

An identity graph models identities as nodes and "same-as" relationships as edges. When you discover that app_user_42 and crm_contact_789 belong to the same person (because they share an email address), you add an edge. The connected components of this graph represent canonical entities.

Resolution Approaches

The simplest approach is deterministic matching—join on exact shared identifiers (email, phone). When exact matches aren't available, probabilistic matching uses similarity scores across multiple attributes (name, address, date of birth) to estimate match likelihood.

# Cross-system identity mapping
from collections import defaultdict
from typing import Dict, Set, Optional

class IdentityGraph:
    def __init__(self):
        self.edges = defaultdict(set)
        self.canonical = {}

    def link(self, id_a: str, id_b: str):
        # Add bidirectional edge between two identifiers
        self.edges[id_a].add(id_b)
        self.edges[id_b].add(id_a)
        self._recompute_canonical(id_a)

    def _recompute_canonical(self, start_id: str):
        # BFS to find connected component
        visited = set()
        queue = [start_id]
        while queue:
            current = queue.pop(0)
            if current in visited:
                continue
            visited.add(current)
            queue.extend(self.edges[current])

        # Assign the smallest ID as canonical
        canonical_id = min(visited)
        for node in visited:
            self.canonical[node] = canonical_id

    def resolve(self, identifier: str) -> str:
        # Return canonical entity ID for any identifier
        return self.canonical.get(identifier, identifier)

    def get_all_ids(self, identifier: str) -> Set[str]:
        # Return all known IDs for this entity
        canonical = self.resolve(identifier)
        return {k for k, v in self.canonical.items() if v == canonical}

# Usage
graph = IdentityGraph()
graph.link("app_user_42", "email:john@example.com")
graph.link("crm_contact_789", "email:john@example.com")
graph.link("app_user_42", "stripe:cus_abc123")

# All resolve to the same canonical entity
print(graph.resolve("crm_contact_789"))
print(graph.get_all_ids("stripe:cus_abc123"))
Tip: Build your identity graph incrementally. Start with deterministic links (shared email, phone), then layer probabilistic matching for records that lack exact shared identifiers. Run resolution as a batch job nightly and as a streaming job for real-time identity stitching.

Deduplication Strategies

Deduplication removes or merges records that represent the same entity. In a feature store, duplicates cause features to be computed multiple times, inflate aggregation counts, and can lead to data leakage during training. Three main approaches exist, in order of complexity.

Exact Matching

The simplest approach—deduplicate based on exact equality of one or more fields. Fast and deterministic but misses records with typos, formatting differences, or missing fields.

Fuzzy Matching

Uses string similarity metrics (Levenshtein distance, Jaro-Winkler, phonetic encoding) to find near-duplicates. Effective for name variations ("John Smith" vs "Jon Smyth") but requires careful threshold tuning to balance precision and recall.

ML-Based Deduplication

Trains a classifier on labeled pairs of (match, non-match) records. Can combine multiple signals (name similarity, address proximity, behavioral patterns) into a single match probability. Most accurate but requires labeled training data.

# Deduplication with multiple strategies
from typing import List, Tuple, Dict
import re

class RecordDeduplicator:
    def __init__(self, threshold: float = 0.85):
        self.threshold = threshold

    def normalize(self, text: str) -> str:
        # Standardize text for comparison
        text = text.lower().strip()
        text = re.sub(r"[^a-z0-9\s]", "", text)
        return " ".join(text.split())

    def levenshtein_similarity(self, s1: str, s2: str) -> float:
        # Normalized edit distance similarity
        if not s1 or not s2:
            return 0.0
        max_len = max(len(s1), len(s2))
        distance = self._edit_distance(s1, s2)
        return 1.0 - (distance / max_len)

    def _edit_distance(self, s1: str, s2: str) -> int:
        m, n = len(s1), len(s2)
        dp = list(range(n + 1))
        for i in range(1, m + 1):
            prev = dp[0]
            dp[0] = i
            for j in range(1, n + 1):
                temp = dp[j]
                if s1[i - 1] == s2[j - 1]:
                    dp[j] = prev
                else:
                    dp[j] = 1 + min(prev, dp[j], dp[j - 1])
                prev = temp
        return dp[n]

    def find_duplicates(self, records: List[Dict]) -> List[Tuple[int, int, float]]:
        # Returns pairs of duplicate indices with confidence
        duplicates = []
        for i in range(len(records)):
            for j in range(i + 1, len(records)):
                score = self._compute_match_score(records[i], records[j])
                if score >= self.threshold:
                    duplicates.append((i, j, score))
        return duplicates

    def _compute_match_score(self, rec_a: Dict, rec_b: Dict) -> float:
        # Weighted average of field similarities
        weights = {"name": 0.4, "email": 0.35, "address": 0.25}
        total_score = 0.0
        for field, weight in weights.items():
            val_a = self.normalize(rec_a.get(field, ""))
            val_b = self.normalize(rec_b.get(field, ""))
            total_score += weight * self.levenshtein_similarity(val_a, val_b)
        return total_score
Warning: Naive pairwise comparison is O(n²). For large datasets, use blocking—group records by a coarse key (first 3 letters of name, ZIP code) and only compare within blocks. This can reduce comparisons by 99%+ without significant loss in recall.

Implementation in Feature Stores

Modern feature store frameworks like Feast provide first-class support for entity definitions. The entity is the anchor for all feature lookups—both during training (offline retrieval with point-in-time joins) and during serving (online key-value lookup).

Entity Definitions in Feast

In Feast, you define entities declaratively. Each entity has a name, a value type, and a join key. Feature views reference entities to declare which join key they use for lookups.

Join Keys and Point-in-Time Joins

When retrieving training data, the feature store performs point-in-time joins: for each entity at a given timestamp, it finds the most recent feature value before that timestamp. This prevents data leakage—you only see features that were available at the time the prediction would have been made.

# Entity definitions in Feast
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float64, Int64, String
from datetime import timedelta

# Define the user entity
user = Entity(
    name="user",
    join_keys=["user_id"],
    description="Application user identified by user_id"
)

# Define a composite entity for user-merchant pairs
user_merchant = Entity(
    name="user_merchant",
    join_keys=["user_id", "merchant_id"],
    description="User-merchant interaction pair"
)

# Data source definition
user_features_source = FileSource(
    path="data/user_features.parquet",
    timestamp_field="event_timestamp",
    created_timestamp_column="created_timestamp"
)

# Feature view tied to the user entity
user_features_view = FeatureView(
    name="user_features",
    entities=[user],
    ttl=timedelta(days=1),
    schema=[
        Field(name="txn_count_30d", dtype=Int64),
        Field(name="avg_amount_30d", dtype=Float64),
        Field(name="account_age_days", dtype=Int64),
        Field(name="risk_score", dtype=Float64),
    ],
    source=user_features_source,
)

# Retrieving features for training
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

# entity_df has columns: user_id, event_timestamp
training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "user_features:txn_count_30d",
        "user_features:avg_amount_30d",
        "user_features:risk_score",
    ]
).to_df()

# Online serving for real-time inference
online_features = store.get_online_features(
    features=["user_features:txn_count_30d", "user_features:risk_score"],
    entity_rows=[{"user_id": "user_42"}]
).to_dict()
Best practice: Always include an event_timestamp column in your entity DataFrames. Without it, point-in-time joins can't work, and you risk leaking future feature values into your training data—a subtle but devastating source of model overfitting.

Challenges and Best Practices

Entity resolution is rarely a "solved" problem. Real-world data is messy, identities change over time, and the cost of getting it wrong compounds across every model that consumes your features.

Slowly Changing Dimensions

Users change their names, addresses, and email addresses. A robust entity resolution system must handle these temporal changes without creating false duplicates or breaking existing links. Type 2 slowly changing dimensions (SCD2) preserve history by versioning entity records with valid-from and valid-to timestamps.

Temporal Entities

Some entities are inherently temporal—a "session" entity exists only for the duration of a user's visit, an "order" entity is created and then moves through fulfillment states. Features for these entities must be computed with awareness of the entity's lifecycle.

Common Pitfalls

  • Over-merging: Aggressive fuzzy matching links unrelated entities, corrupting features
  • Under-merging: Conservative thresholds leave duplicates, inflating counts
  • Stale mappings: Identity graph not updated when users merge accounts
  • PII leakage: Using raw personal data as join keys

Best Practices

  • Measure quality: Track precision and recall of your entity resolution pipeline
  • Version mappings: Maintain a history of identity merges and splits
  • Use surrogates: Map all external IDs to internal surrogates at ingestion
  • Monitor drift: Alert when duplicate rates or merge rates change significantly
Key takeaway: Entity resolution is a data quality problem that directly impacts model quality. Invest in monitoring and measuring your entity resolution pipeline the same way you'd monitor model performance. A 5% error rate in entity resolution can easily translate to a measurable drop in model accuracy.
Caution with cross-device identity: Linking a user's phone, laptop, and tablet activity is valuable but fraught with privacy implications. Ensure your entity resolution respects user consent, complies with GDPR/CCPA, and that all cross-device linking is done on hashed or anonymized identifiers.
