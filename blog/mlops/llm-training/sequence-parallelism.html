Sequence and Context Parallelism

MLOps Series LLM Training Infrastructure

Why Sequence Parallelism?

As large language models push context windows from 2K to 128K tokens and beyond, a fundamental memory bottleneck emerges that neither data parallelism nor tensor parallelism can resolve. The self-attention mechanism computes a full S × S attention matrix where S is the sequence length. For a single attention head with hidden dimension dh, this means O(S² · dh) FLOPs and O(S²) memory just for the attention scores—before we even consider the KV cache or activation memory.

Consider a concrete example: a 70B parameter model with 64 attention heads, head dimension 128, training on a sequence length of 131,072 tokens. The attention score matrix alone for a single layer is S² × num_heads × sizeof(bf16) = 131072² × 64 × 2 bytes ≈ 2.2 TB. This is far beyond the 80 GB HBM capacity of a single A100/H100 GPU. Even with FlashAttention (which avoids materializing the full S×S matrix), the activations for the linear projections still scale as O(S × dmodel) per layer, and with 80+ layers the aggregate activation memory is untenable.

Key insight: Data parallelism replicates the full model and partitions batch dimension. Tensor parallelism shards weight matrices across GPUs. Neither partitions the sequence dimension, so a single long sequence still needs to fit in one GPU’s memory. Sequence parallelism directly addresses this by splitting the sequence across devices.

The activation memory per transformer layer scales as Alayer = 10 · S · h · b + 5 · a · S² · b (in mixed precision, where h = hidden size, a = number of heads, b = micro-batch size). For S = 128K, h = 8192, a = 64, b = 1, a single layer requires approximately 10 × 128K × 8192 × 2 + 5 × 64 × 128K² × 2 ≈ 21 GB + 10.7 TB. The quadratic term dominates completely. This is why sequence parallelism is not optional for long-context training—it is the only viable strategy.

There are three major approaches to sequence parallelism in practice: Megatron-SP (which partitions LayerNorm/Dropout activations along the sequence dimension within tensor-parallel groups), DeepSpeed Ulysses (which uses all-to-all collectives to redistribute Q/K/V), and Ring Attention (which passes KV blocks around a logical ring of GPUs). Each has distinct communication patterns, memory trade-offs, and scaling characteristics that we will analyze in depth.

Ulysses: Splitting the Sequence Dimension

DeepSpeed Ulysses (introduced in late 2023) takes a direct approach: partition the input sequence of length S into P equal chunks across P GPUs, where each GPU holds S/P tokens. The critical insight is that attention requires each query to attend to all keys and values. Rather than replicating the full KV tensors (which would negate memory savings), Ulysses uses all-to-all collective communication to transpose the data layout before and after attention.

The workflow proceeds as follows: (1) Each GPU computes Q, K, V projections locally on its S/P tokens, producing tensors of shape [S/P, num_heads, d_head]. (2) An all-to-all scatter redistributes these along the head dimension—after the all-to-all, each GPU holds the full sequence for num_heads/P heads, i.e., shape [S, num_heads/P, d_head]. (3) Each GPU computes full self-attention on its subset of heads over the complete sequence. (4) A second all-to-all gathers the results back to the original sequence-partitioned layout.

Communication cost: Each all-to-all transfers S × h × sizeof(bf16) / P bytes per GPU (for each of Q, K, V). With two all-to-all operations (pre- and post-attention), the total per-layer communication volume is 8 × S × h / P bytes (accounting for Q, K, V forward + backward). This matches tensor parallelism’s all-reduce cost—but critically, the memory per GPU drops from O(S) to O(S/P) for non-attention activations.

The memory savings from Ulysses are substantial. The attention computation itself runs on the full sequence per GPU (since each GPU holds all tokens for its head subset), but the linear projection activations—which dominate for very large hidden dimensions—are reduced by a factor of P. For a model with h = 8192, S = 128K, P = 8:

  • Without Ulysses: Activation memory for QKV projections = 3 × S × h × 2 = 3 × 128K × 8192 × 2 ≈ 6 GB per layer
  • With Ulysses (P=8): QKV projection activations = 3 × (S/P) × h × 2 ≈ 0.75 GB per layer
  • Attention score memory (with FlashAttention): O(S) per head — distributed across GPUs by head partitioning

Ulysses is most effective when the number of attention heads is evenly divisible by the sequence-parallelism degree P. For GQA (Grouped Query Attention) models like Llama 2/3, where KV heads may be only 8, this limits Ulysses to P ≤ 8. Beyond this, the head dimension cannot be further partitioned without redundant computation.

Ring Attention

Ring Attention (Liu et al., 2023) takes a fundamentally different approach by arranging P GPUs in a logical ring and streaming KV blocks between neighbors. Each GPU permanently holds its local chunk of Q (shape [S/P, num_heads, d_head]) and iteratively receives KV blocks from other GPUs to compute partial attention scores. After P−1 communication steps, every GPU has attended its queries to all keys and values without ever materializing the full KV set locally.

The algorithm works in P rounds. At each round r, GPU i holds its local Qi and the KV block from GPU (i−r) mod P. It computes a partial attention output using blockwise (tiled) attention, accumulating the numerator and denominator of the softmax via the log-sum-exp trick. After computing, each GPU sends its current KV block to the next neighbor in the ring (GPU (i+1) mod P) and receives the next block from (i−1) mod P.

The critical optimization is overlapping communication with computation. While GPU i computes attention with KV block r, it simultaneously sends KV block r to GPU (i+1) and receives KV block r+1. On modern NVLink/NVSwitch interconnects, P2P bandwidth is 450+ GB/s per direction, so transferring a KV block of size 2 × (S/P) × h × 2 bytes can be fully hidden behind the O(S/P × S/P × d_head) attention computation for large enough S/P.

When does overlap work? The compute-to-communication ratio for ring attention is approximately (S/P) × d_head / bandwidth FLOPs per byte. For S/P = 16K, d_head = 128, and NVLink at 450 GB/s, the compute takes ~2× longer than communication, enabling effective overlap. Below S/P ≈ 4K tokens, communication begins to dominate and ring attention loses efficiency.

Striped Attention is a variant that interleaves token assignment instead of using contiguous chunks. GPU i gets tokens {i, i+P, i+2P, ...}. This ensures that causal masking is uniformly distributed across all GPUs, preventing the load imbalance where early GPUs (holding early tokens) do less work due to the triangular causal mask. In contiguous partitioning, GPU 0 attends to at most S/P keys, while GPU P−1 attends to all S keys—a P× imbalance. Striped attention equalizes this to within one token.

Context Parallelism for 100K+ Tokens

Context parallelism (CP) is the umbrella term used in frameworks like Megatron-LM to describe sequence-dimension partitioning that is orthogonal to tensor parallelism (TP) and pipeline parallelism (PP). In Megatron-LM’s 5D parallelism hierarchy (DP × TP × PP × CP × EP), context parallelism forms its own communication group—GPUs within a CP group exchange KV blocks, while TP groups handle weight sharding and PP groups handle layer sharding.

The practical distinction from “pure” ring attention is that CP in Megatron-LM integrates with the existing parallelism dimensions. A typical configuration for training a 70B model on 128K context might use: TP=8, PP=4, CP=4, DP=8 across 1024 GPUs. Here, the 128K sequence is split into 4 chunks of 32K tokens each, and each CP-rank processes 32K tokens through the full pipeline of layers (with TP sharding the weights within each CP-rank).

Memory analysis for 128K context (70B model, bf16):
• Model parameters: 70B × 2 bytes / TP=8 ≈ 17.5 GB per GPU
• Optimizer states (AdamW, fp32): 70B × 12 bytes / (TP × DP) ≈ 13.1 GB per GPU
• Activations per layer (without CP): ~21 GB (linear) + ~10.7 TB (attention) — impossible
• Activations per layer (CP=4, FlashAttn): ~5.25 GB (linear/4) + O(S/CP) attention workspace ≈ 6 GB
• Total activation across 80 layers with activation checkpointing (every 4 layers): ~120 GB / PP=4 ≈ 30 GB per GPU
• Total per GPU: ~17.5 + 13.1 + 30 ≈ 60.6 GB — fits in 80 GB HBM

For even longer contexts (512K–1M tokens), CP must be scaled to 16 or 32. At CP=32, each GPU processes only S/32 tokens, but the ring communication requires 31 P2P steps. The key bottleneck shifts from memory to communication latency: each step introduces a minimum latency of the KV transfer time, and with 31 steps this accumulates. The Megatron-LM implementation mitigates this by using hierarchical communication— intra-node NVSwitch for nearby CP ranks and inter-node InfiniBand for distant ranks—and by carefully scheduling computation to maximize overlap.

Context parallelism also interacts with Grouped Query Attention (GQA). In GQA, KV heads are shared across multiple query heads (e.g., Llama 3 uses 8 KV heads for 64 query heads). This means the KV blocks being communicated are 8× smaller than the Q blocks, reducing ring communication volume proportionally. For a model with nkv = 8 KV heads, h = 8192, S = 128K, CP = 8, the per-step KV transfer is only 2 × (128K/8) × 8 × 128 × 2 = 64 MB—easily hidden behind computation on NVLink.

Ulysses (All-to-All)

  • Communication: Two all-to-all collectives per layer
  • Volume: O(S × h / P) per GPU per collective
  • Scaling limit: P ≤ num_kv_heads (GQA bottleneck)
  • Overlap: Hard to overlap — all-to-all is blocking
  • Memory: Full attention per GPU (on head subset)
  • Best for: Moderate P (2–8), dense attention heads

Ring Attention (P2P)

  • Communication: P−1 P2P send/recv per layer
  • Volume: O(S × h / P) per step (KV block)
  • Scaling limit: S/P must be large enough for overlap
  • Overlap: Natural overlap of comm & compute
  • Memory: Only S/P tokens of KV at any time
  • Best for: Large P (8–64), very long sequences

Memory Analysis and Performance

Let us formalize the per-GPU memory breakdown for a transformer model with L layers, hidden size h, number of attention heads a, head dimension dh = h/a, sequence length S, micro-batch size b, and sequence parallelism degree P.

Activation memory per layer (without sequence parallelism, with FlashAttention):

  • QKV projection outputs: 3 × S × h × b × 2 bytes
  • Attention output projection: S × h × b × 2 bytes
  • MLP intermediate (SwiGLU): 2 × S × 4h × b × 2 bytes (gate + up projections)
  • MLP output: S × h × b × 2 bytes
  • LayerNorm inputs (2 per layer): 2 × S × h × b × 2 bytes
  • Total per layer: (3h + h + 8h + h + 2h) × S × b × 2 = 30 × S × h × b bytes

With sequence parallelism (degree P):

  • All sequence-dimension tensors are sharded by P: 30 × S × h × b / P bytes per layer
  • For S = 128K, h = 8192, b = 1, P = 8: 30 × 128K × 8192 / 8 = 3.84 GB per layer
  • With activation checkpointing (recompute every k layers): total ≈ 3.84 × k GB
Scaling rule of thumb: Doubling the sequence parallelism degree P halves the per-GPU activation memory but doubles the number of communication steps. The crossover point where communication overhead exceeds 10% of compute time is approximately P = S / (8192 × d_head / bandwidth_GB_s). On 8×H100 with NVSwitch (900 GB/s bisection), this allows P up to ~32 for S = 128K.

Communication volume analysis:

  • Ulysses all-to-all (per layer, forward + backward): 4 × 3 × S × h × 2 / P = 24Sh/P bytes. For S=128K, h=8192, P=8: 3.1 GB per layer.
  • Ring attention (per layer, forward): (P−1) × 2 × (S/P) × h × 2 = 4Sh(P−1)/P bytes ≈ 4Sh for large P. Effective bandwidth utilization is near 100% with overlap.
  • Combined with TP (e.g., TP=8): Sequence parallelism comm uses NVLink within the node, while TP all-reduces also use NVLink. Careful scheduling avoids contention by alternating TP and SP communication phases.

The scaling behavior of sequence parallelism follows a predictable pattern. We define scaling efficiency as the ratio of ideal speedup (P×) to actual speedup. For ring attention, efficiency is approximately 1 / (1 + (P-1) × t_comm / t_compute) where t_comm and t_compute are per-step times. On H100 NVLink with FlashAttention-2, measured efficiencies are: P=2 (97%), P=4 (94%), P=8 (89%), P=16 (82%), P=32 (71%).

Implementation Details

Below is a detailed implementation of sequence-parallel ring attention. The core algorithm maintains running log-sum-exp accumulators to correctly combine partial softmax results across ring steps without numerical instability.

import torch import torch.distributed as dist from torch import Tensor from typing import Optional, Tuple class RingAttention: # Ring attention with overlapped P2P communication # Supports causal masking and FlashAttention-compatible interface def __init__(self, cp_group: dist.ProcessGroup, causal: bool = True): self.cp_group = cp_group self.cp_size = dist.get_world_size(group=cp_group) self.cp_rank = dist.get_rank(group=cp_group) self.causal = causal def _get_ring_neighbors(self) -> Tuple[int, int]: # Returns (send_to, recv_from) ranks in the ring send_to = (self.cp_rank + 1) % self.cp_size recv_from = (self.cp_rank - 1) % self.cp_size return send_to, recv_from def _async_send_recv(self, send_buf: Tensor, recv_buf: Tensor): # Non-blocking P2P send/recv for KV block exchange send_to, recv_from = self._get_ring_neighbors() ops = [] ops.append(dist.isend(send_buf, dst=send_to, group=self.cp_group)) ops.append(dist.irecv(recv_buf, src=recv_from, group=self.cp_group)) return ops def _update_running_softmax( self, new_attn_out: Tensor, # [B, H, S/P, D] new_lse: Tensor, # [B, H, S/P] running_out: Tensor, # [B, H, S/P, D] running_lse: Tensor, # [B, H, S/P] ) -> Tuple[Tensor, Tensor]: # Online softmax correction using log-sum-exp trick # new_lse and running_lse are in log-space max_lse = torch.maximum(new_lse, running_lse) old_scale = torch.exp(running_lse - max_lse).unsqueeze(-1) new_scale = torch.exp(new_lse - max_lse).unsqueeze(-1) updated_out = old_scale * running_out + new_scale * new_attn_out updated_lse = max_lse + torch.log( torch.exp(running_lse - max_lse) + torch.exp(new_lse - max_lse) ) return updated_out, updated_lse def forward( self, q_local: Tensor, # [B, H, S/P, D] - local query chunk k_local: Tensor, # [B, H, S/P, D] - local key chunk v_local: Tensor, # [B, H, S/P, D] - local value chunk scale: Optional[float] = None, ) -> Tensor: B, H, Sp, D = q_local.shape # Sp = S / P if scale is None: scale = D ** -0.5 # Step 0: compute local attention (Q_i attends to KV_i) attn_scores = torch.matmul(q_local, k_local.transpose(-2, -1)) * scale if self.causal: # Apply causal mask for local block (diagonal block) mask = torch.triu( torch.ones(Sp, Sp, device=q_local.device), diagonal=1 ).bool() attn_scores.masked_fill_(mask, float('-inf')) running_lse = torch.logsumexp(attn_scores, dim=-1) running_out = torch.matmul( torch.softmax(attn_scores, dim=-1), v_local ) # Prepare double-buffered KV blocks for async P2P kv_send = torch.cat([k_local, v_local], dim=-1) # [B, H, Sp, 2D] kv_recv = torch.empty_like(kv_send) # Ring steps 1 through P-1 for step in range(1, self.cp_size): # Launch async P2P (overlaps with previous step's compute) ops = self._async_send_recv(kv_send, kv_recv) # Wait for communication to complete for op in ops: op.wait() # Unpack received KV block k_block = kv_recv[..., :D] v_block = kv_recv[..., D:] # Compute partial attention for this KV block block_scores = torch.matmul(q_local, k_block.transpose(-2, -1)) * scale if self.causal: # Determine source rank of this KV block src_rank = (self.cp_rank - step) % self.cp_size if src_rank > self.cp_rank: # KV block is from future tokens: fully masked block_scores.fill_(float('-inf')) # If src_rank < cp_rank: no mask (attend to all past) # If src_rank == cp_rank: handled in step 0 block_lse = torch.logsumexp(block_scores, dim=-1) block_out = torch.matmul( torch.softmax(block_scores, dim=-1), v_block ) # Merge partial results using online softmax running_out, running_lse = self._update_running_softmax( block_out, block_lse, running_out, running_lse ) # Rotate buffers for next step kv_send, kv_recv = kv_recv, torch.empty_like(kv_send) return running_out
Numerical stability: The _update_running_softmax method is critical. Naive accumulation of softmax outputs leads to catastrophic numerical errors because softmax denominators from different blocks are incompatible. The log-sum-exp trick maintains a running maximum and rescales previous results, ensuring exact equivalence with full-sequence attention up to floating-point precision. This is the same principle used in FlashAttention’s tiled implementation.

For production use, the above would be combined with FlashAttention-2’s fused kernels for the per-block attention computation (replacing the explicit matmul + softmax + matmul with a single CUDA kernel call). The CUDA kernel returns both the attention output and the log-sum-exp values needed for the online correction.

Ulysses-style all-to-all implementation:

def ulysses_sequence_parallel_attn( q: Tensor, # [B, S/P, num_heads, D] - sequence-partitioned k: Tensor, # [B, S/P, num_heads, D] v: Tensor, # [B, S/P, num_heads, D] sp_group: dist.ProcessGroup, ) -> Tensor: # DeepSpeed Ulysses: all-to-all to redistribute from # sequence-partitioned to head-partitioned layout P = dist.get_world_size(group=sp_group) # Pre-attention: all-to-all scatter along seq, gather along heads # [B, S/P, H, D] -> [B, S, H/P, D] q_heads = all_to_all(q, scatter_dim=1, gather_dim=2, group=sp_group) k_heads = all_to_all(k, scatter_dim=1, gather_dim=2, group=sp_group) v_heads = all_to_all(v, scatter_dim=1, gather_dim=2, group=sp_group) # Standard attention on full sequence, subset of heads # Each GPU now has [B, S, H/P, D] attn_out = flash_attention(q_heads, k_heads, v_heads, causal=True) # Post-attention: all-to-all to restore sequence-partitioned layout # [B, S, H/P, D] -> [B, S/P, H, D] out = all_to_all(attn_out, scatter_dim=2, gather_dim=1, group=sp_group) return out def all_to_all( tensor: Tensor, scatter_dim: int, gather_dim: int, group: dist.ProcessGroup, ) -> Tensor: # Wrapper for torch.distributed all-to-all with reshape world_size = dist.get_world_size(group=group) chunks = tensor.chunk(world_size, dim=scatter_dim) recv = [torch.empty_like(c) for c in chunks] dist.all_to_all(recv, list(chunks), group=group) return torch.cat(recv, dim=gather_dim)

When combining these techniques in practice, modern frameworks like Megatron-LM allow you to specify the parallelism configuration through a simple command-line interface. A typical launch for 128K context training on 1024 H100 GPUs:

# Megatron-LM launch configuration for 128K context, 70B model # 1024 GPUs = TP=8 x PP=4 x CP=4 x DP=8 python -m torch.distributed.launch \ --nproc_per_node 8 \ --nnodes 128 \ pretrain_gpt.py \ --tensor-model-parallel-size 8 \ --pipeline-model-parallel-size 4 \ --context-parallel-size 4 \ --seq-length 131072 \ --num-layers 80 \ --hidden-size 8192 \ --num-attention-heads 64 \ --group-query-attention \ --num-query-groups 8 \ --micro-batch-size 1 \ --global-batch-size 512 \ --use-flash-attn \ --recompute-granularity selective \ --sequence-parallel
Common pitfall: The --sequence-parallel flag in Megatron-LM activates the original Megatron-SP (which partitions LayerNorm/Dropout along the sequence dimension within TP groups), while --context-parallel-size activates ring-attention-style CP across a separate GPU group. Both can be enabled simultaneously and operate on different axes. Confusing these two is a frequent source of bugs in distributed training configurations.

In summary, sequence and context parallelism are essential techniques for training LLMs with long context windows. Ulysses excels for moderate parallelism degrees with its straightforward all-to-all approach, while ring attention scales to large degrees with overlapped communication. The choice between them depends on the model architecture (number of KV heads), the hardware interconnect (NVLink vs. InfiniBand), and the target sequence length. For production systems training at 128K+ tokens, combining both approaches—Ulysses within nodes and ring attention across nodes—often yields the best balance of memory efficiency and throughput.
