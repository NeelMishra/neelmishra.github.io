Data Quality: Toxicity Filtering, PII Removal

MLOps Series LLM Training Infrastructure

Training a frontier LLM on trillions of tokens from Common Crawl demands rigorous data quality pipelines. Raw web data is riddled with toxicity, personally identifiable information (PII), machine-generated spam, boilerplate, and near-duplicate pages. The quality of your pre-training corpus directly determines downstream model capabilities—and the gap between a mediocre and a state-of-the-art model often comes down to data curation rather than architecture. This post walks through the full quality filtering stack: classifier-based toxicity detection, regex and NER-based PII removal, perplexity filtering with KenLM, heuristic filters, and principled data mixing strategies.

The Quality-Diversity Trade-off

The central tension in data curation is that aggressive quality filtering improves per-token learning efficiency but reduces corpus diversity. The Chinchilla scaling laws (Hoffmann et al., 2022) showed that compute-optimal training requires roughly 20 tokens per parameter—a 70B model needs ~1.4T tokens. When you filter too aggressively (e.g., keeping only Wikipedia-quality text), you exhaust your high-quality supply and must either upsample (introducing repetition) or accept a smaller effective dataset. Empirically, models trained on moderately filtered Common Crawl outperform those trained on heavily filtered subsets at fixed compute budgets.

The filtering yield curve is roughly logarithmic: removing the worst 30% of documents (porn, spam, encoding errors) gives enormous quality gains at minimal diversity cost. Removing the next 30% (low-quality but topically diverse content) gives diminishing returns while eliminating niche domains like regional forums, hobbyist blogs, and minority-language content. This creates a measurable bias toward formal, Western, English-language text.

Key Insight: Optimal filtering typically retains 10–30% of raw Common Crawl tokens. LLaMA retained ~15%, RefinedWeb ~20%, and C4 ~12%. The sweet spot depends on your total compute budget and how many training epochs you plan to run. For single-epoch training, err toward inclusion; for multi-epoch, filter more aggressively.

Chinchilla scaling also implies that data quality has increasing marginal returns as model size grows. A 7B model is somewhat robust to noisy data—it cannot memorize fine-grained patterns anyway. A 70B+ model, however, will faithfully learn spam patterns, toxic associations, and PII, making quality filtering more critical at scale, not less. This counter-intuitive result drives the industry trend toward smaller but meticulously curated datasets (e.g., Dolma, FineWeb) rather than raw scale alone.

Toxicity Classification

Toxicity filtering aims to remove content that is hateful, violent, sexually explicit, or otherwise harmful from the training corpus. The challenge is operating at the scale of billions of documents with sub-millisecond per-document inference budgets. Two dominant approaches exist: lightweight fastText classifiers for coarse-grained filtering, and transformer-based models for higher-precision second-pass filtering.

fastText-Based Classifiers

FastText (Joulin et al., 2017) classifiers trained on Jigsaw Toxic Comment data provide a strong speed–accuracy trade-off. A bigram fastText model achieves ~0.95 AUC on binary toxicity at 10M+ documents per hour on a single CPU core. The key trick is training on document-level aggregated labels rather than sentence-level, since Common Crawl documents contain mixed content.

# FastText-based toxicity classifier for web-scale filtering import fasttext import numpy as np from dataclasses import dataclass from typing import List, Tuple @dataclass class ToxicityResult: is_toxic: bool score: float labels: List[str] class FastTextToxicityFilter: def __init__(self, model_path: str, threshold: float = 0.4): self.model = fasttext.load_model(model_path) self.threshold = threshold # Multi-label categories: hate, violence, sexual, self-harm self.toxic_labels = { "__label__toxic", "__label__severe_toxic", "__label__hate", "__label__violence", "__label__sexual_explicit", } def classify(self, text: str) -> ToxicityResult: # Truncate to first 512 tokens for speed truncated = " ".join(text.split()[:512]) labels, scores = self.model.predict(truncated, k=5) toxic_score = 0.0 matched_labels = [] for label, score in zip(labels, scores): if label in self.toxic_labels: toxic_score = max(toxic_score, score) matched_labels.append(label) return ToxicityResult( is_toxic=toxic_score > self.threshold, score=toxic_score, labels=matched_labels, ) def filter_batch(self, documents: List[str]) -> List[Tuple[str, ToxicityResult]]: # Batch inference for throughput results = [] for doc in documents: result = self.classify(doc) if not result.is_toxic: results.append((doc, result)) return results

Transformer-Based Approaches

Google's Perspective API uses a distilled BERT model to score text across six toxicity dimensions: toxicity, severe toxicity, identity attack, insult, profanity, and threat. For pre-training pipelines, hosting a local distilled model (e.g., unitary/toxic-bert from HuggingFace) avoids API rate limits. The throughput cost is ~100x higher than fastText, so teams typically use a two-stage cascade: fastText for coarse filtering (removing documents with >0.8 toxicity score), then a transformer pass on the borderline range (0.3–0.8) where the classifier is uncertain.

False Positive Risk: Toxicity classifiers systematically over-flag African-American Vernacular English (AAVE), disability-related discussions, and LGBTQ+ content. At a threshold of 0.5, Perspective API flags ~20% of non-toxic AAVE text as toxic. Mitigate by training separate thresholds per demographic category or using identity-term counterfactual augmentation during classifier training.

Threshold Tuning

The optimal toxicity threshold depends on your use case. For a general-purpose LLM, a false positive rate (FPR) of 1–3% is acceptable, meaning you remove 1–3% of clean documents to catch 95%+ of truly toxic content. For safety-critical applications, you may tolerate 5–10% FPR. The relationship between threshold and recall is roughly:

Conservative (threshold = 0.7)

Recall: ~82% of toxic content removed

FPR: ~0.5% clean docs removed

Use case: Preserving diversity, large corpora where some toxic leakage is acceptable post-RLHF

Aggressive (threshold = 0.3)

Recall: ~97% of toxic content removed

FPR: ~4% clean docs removed

Use case: Safety-critical models, smaller corpora where each document matters

PII Detection and Removal

PII leakage in training data creates legal liability (GDPR, CCPA) and safety risks (models memorizing and regurgitating phone numbers, email addresses, or Social Security numbers). A robust PII pipeline combines high-precision regex patterns for structured PII with NER models for unstructured entities like names and addresses. The key design decision is whether to redact (replace with [REDACTED]), mask (replace with entity type, e.g., [EMAIL]), or synthesize (replace with realistic fake data). Synthesis preserves text coherence for training but requires a generation step.

Regex-Based Structured PII Detection

# Regex patterns for structured PII detection import re from typing import Dict, List, Tuple PII_PATTERNS: Dict[str, re.Pattern] = { "email": re.compile( r"\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b" ), "phone_us": re.compile( r"\b(?:\+?1[-.\s]?)?\(?\d{3}\)?[-.\s]?\d{3}[-.\s]?\d{4}\b" ), "ssn": re.compile( r"\b\d{3}-\d{2}-\d{4}\b" ), "credit_card": re.compile( r"\b(?:4\d{3}|5[1-5]\d{2}|3[47]\d{2}|6(?:011|5\d{2}))" r"[-\s]?\d{4}[-\s]?\d{4}[-\s]?\d{1,4}\b" ), "ip_address": re.compile( r"\b(?:\d{1,3}\.){3}\d{1,3}\b" ), "date_of_birth": re.compile( r"\b(?:0[1-9]|1[0-2])[/-](?:0[1-9]|[12]\d|3[01])[/-](?:19|20)\d{2}\b" ), } def detect_pii_regex(text: str) -> List[Tuple[str, str, int, int]]: # Returns list of (pii_type, matched_text, start, end) findings = [] for pii_type, pattern in PII_PATTERNS.items(): for match in pattern.finditer(text): findings.append(( pii_type, match.group(), match.start(), match.end(), )) return sorted(findings, key=lambda x: x[2])

NER-Based PII Detection with Presidio

Microsoft Presidio combines regex, NER (using spaCy or Stanza), and context-aware heuristics to detect unstructured PII like person names, addresses, and medical record numbers. For web-scale pipelines, the spaCy en_core_web_trf model (transformer-backed) achieves ~92% F1 on person names but at 50x the cost of the regex-only approach. A practical compromise is using en_core_web_sm (fast, ~85% F1) for the first pass and the transformer model only on documents flagged as high-PII-density.

# NER-based PII detection and replacement pipeline import spacy from presidio_analyzer import AnalyzerEngine, RecognizerResult from presidio_anonymizer import AnonymizerEngine from presidio_anonymizer.entities import OperatorConfig class PIIRemover: def __init__(self, strategy: str = "mask"): # strategy: "redact" | "mask" | "synthesize" self.analyzer = AnalyzerEngine() self.anonymizer = AnonymizerEngine() self.strategy = strategy self.entities = [ "PERSON", "EMAIL_ADDRESS", "PHONE_NUMBER", "US_SSN", "CREDIT_CARD", "IP_ADDRESS", "LOCATION", "DATE_TIME", ] def detect(self, text: str) -> List[RecognizerResult]: return self.analyzer.analyze( text=text, entities=self.entities, language="en" ) def remove(self, text: str) -> str: results = self.detect(text) if self.strategy == "mask": operators = { "DEFAULT": OperatorConfig("replace", {"new_value": "[PII]"}) } elif self.strategy == "redact": operators = {"DEFAULT": OperatorConfig("redact")} else: operators = None return self.anonymizer.anonymize( text=text, analyzer_results=results, operators=operators, ).text
Redact vs. Mask vs. Synthesize: Redaction (John Smith → ████) destroys sentence structure. Masking (John Smith → [PERSON]) preserves grammar but introduces distributional artifacts—the model may learn that [PERSON] is a common name. Synthesis (John Smith → Alice Johnson) is ideal for training quality but requires a name/address generation service and adds pipeline complexity. Most production pipelines use entity-type masking as the default.

Perplexity-Based Quality Filtering

Perplexity-based filtering, pioneered by the CCNet pipeline (Wenzek et al., 2020), uses a reference language model trained on clean text (typically Wikipedia) to score web documents. Documents with very high perplexity (the reference model is surprised by them) are likely low-quality: garbled text, encoding errors, keyword-stuffed SEO pages, or non-natural-language content. Documents with very low perplexity may be near-copies of Wikipedia. The sweet spot is the middle range, capturing naturally written web content that is topically diverse.

KenLM Reference Model

KenLM (Heafield, 2011) is a C++ n-gram language model optimized for speed. A 5-gram KenLM model trained on English Wikipedia fits in ~4GB RAM and scores documents at 100K+ per second per CPU core. The pipeline is: (1) train KenLM on tokenized Wikipedia, (2) compute per-document perplexity on the web corpus, (3) bucket documents into quality tiers by perplexity percentile.

# Perplexity-based quality filtering with KenLM import kenlm import math from typing import List, Tuple class PerplexityFilter: def __init__(self, model_path: str, low_ppl: float = 10.0, high_ppl: float = 1000.0): # low_ppl: below this is suspiciously clean (Wikipedia copy) # high_ppl: above this is likely garbage self.model = kenlm.Model(model_path) self.low_ppl = low_ppl self.high_ppl = high_ppl def compute_perplexity(self, text: str) -> float: # KenLM returns log10 probability; convert to perplexity words = text.split() if len(words) == 0: return float("inf") log_score = self.model.score(" ".join(words)) # score returns log10 prob; perplexity = 10^(-log10prob/N) ppl = 10.0 ** (-log_score / len(words)) return ppl def classify_quality(self, text: str) -> str: ppl = self.compute_perplexity(text) if ppl < self.low_ppl: return "wikipedia_copy" elif ppl > self.high_ppl: return "low_quality" else: return "acceptable" def filter_corpus(self, docs: List[str]) -> List[Tuple[str, float]]: # Returns documents that pass the perplexity filter filtered = [] for doc in docs: ppl = self.compute_perplexity(doc) if self.low_ppl <= ppl <= self.high_ppl: filtered.append((doc, ppl)) return filtered

Language-Specific Considerations

Perplexity thresholds vary dramatically by language. English Wikipedia is large and well-represented, so the KenLM model produces well-calibrated perplexity scores. For low-resource languages (Swahili, Yoruba, Tagalog), the Wikipedia training set is small, the model is under-fit, and perplexity scores are inflated. Teams at BigScience (BLOOM) found that per-language percentile-based thresholds (e.g., keep the 25th–75th percentile) work better than absolute thresholds for multilingual corpora.

Perplexity Pitfall: Code, mathematical notation, and structured data (JSON, CSV) have extremely high perplexity under a Wikipedia-trained LM despite being high-quality training data. Always exclude programming-language-detected files from perplexity filtering. Use language-ID (fastText lid.176.bin) as a prerequisite step and route code to a separate filtering pipeline.

Heuristic Quality Filters

Heuristic filters are fast, deterministic rules that catch quality issues that classifiers miss. They operate on surface-level text statistics and are essentially free in compute. The Gopher paper (Rae et al., 2021) introduced a comprehensive heuristic filter suite that has become the industry standard baseline, later refined by RefinedWeb and FineWeb.

Core Heuristic Rules

# Comprehensive heuristic quality filter suite import re from collections import Counter from typing import Dict class HeuristicFilter: def __init__(self): self.min_words = 50 self.max_words = 100000 self.max_symbol_to_word = 0.1 self.max_mean_word_length = 10 self.min_mean_word_length = 3 self.max_bullet_ratio = 0.9 self.max_ellipsis_ratio = 0.3 def word_count_filter(self, text: str) -> bool: words = text.split() return self.min_words <= len(words) <= self.max_words def symbol_ratio_filter(self, text: str) -> bool: # High symbol:word ratio indicates nav menus or code dumps words = text.split() if len(words) == 0: return False symbols = sum(1 for c in text if c in "#...<>{}|\\^~`") return (symbols / len(words)) < self.max_symbol_to_word def mean_word_length_filter(self, text: str) -> bool: words = text.split() if len(words) == 0: return False mean_len = sum(len(w) for w in words) / len(words) return self.min_mean_word_length <= mean_len <= self.max_mean_word_length def repetition_filter(self, text: str, n: int = 3, max_dup_ratio: float = 0.3) -> bool: # Detect n-gram repetition (boilerplate / SEO spam) words = text.lower().split() if len(words) < n: return True ngrams = [tuple(words[i:i+n]) for i in range(len(words) - n + 1)] counts = Counter(ngrams) dup_ngrams = sum(c - 1 for c in counts.values() if c > 1) return (dup_ngrams / len(ngrams)) < max_dup_ratio def boilerplate_filter(self, text: str) -> bool: # Detect cookie banners, nav menus, list pages lines = text.strip().split("\n") if len(lines) == 0: return False bullet_lines = sum( 1 for l in lines if l.strip().startswith(("•", "-", "*", ">")) ) return (bullet_lines / len(lines)) < self.max_bullet_ratio def apply_all(self, text: str) -> bool: # Returns True if document passes all heuristic filters return ( self.word_count_filter(text) and self.symbol_ratio_filter(text) and self.mean_word_length_filter(text) and self.repetition_filter(text) and self.boilerplate_filter(text) )

Repetition Detection Deep Dive

Repetition is one of the strongest signals of low-quality text. The Gopher pipeline checks for: (a) duplicate paragraphs within a document, (b) high n-gram repetition ratios for n ∈ {2, 3, 4}, and (c) line-level duplication (identical lines occurring more than 3 times). RefinedWeb extended this with character-level repetition for cases like aaaaaa... or !@#!@#!@#. The thresholds are empirically tuned: a 3-gram duplication ratio above 0.20 removes ~8% of Common Crawl with high precision against SEO spam and auto-generated pages.

Classifier-Based Filtering

Strengths: Captures semantic quality; adapts to complex patterns; handles nuance.

Weaknesses: Expensive at scale; requires labeled data; demographic bias; opaque decisions.

Speed: 1–10K docs/sec (fastText) or 50–200 docs/sec (BERT).

Heuristic Filtering

Strengths: Deterministic, fast, interpretable; catches structural issues classifiers miss.

Weaknesses: Cannot assess semantic quality; requires manual rule design; brittle edge cases.

Speed: 100K+ docs/sec per CPU core.

Perplexity Filtering

Strengths: Language-model-aligned quality signal; principled statistical foundation; fast with KenLM.

Weaknesses: Biased toward Wikipedia style; fails on code/math; requires per-language calibration.

Speed: 50–100K docs/sec (KenLM).

Combined Pipeline (Best Practice)

Approach: Heuristics first (cheapest), then perplexity, then classifiers on survivors.

Yield: Typically retains 10–30% of raw Common Crawl.

Compute: ~0.1 CPU-hours per million documents end-to-end.

Data Mixing Strategies

After filtering, the final pre-training corpus is a mixture of data from different quality tiers and domains. The mixing ratios have a profound impact on model capabilities. LLaMA 1 used roughly 67% Common Crawl, 15% C4, 4.5% GitHub, 4.5% Wikipedia, 4.5% books, 2.5% ArXiv, and 2% StackExchange. These ratios were not arbitrary—they were tuned via ablation studies on smaller models to maximize downstream benchmark performance.

Quality-Tier Mixing

Rather than applying a single binary quality threshold, modern pipelines assign each document a quality tier (high / medium / low) and mix them. High-quality documents (Wikipedia, books, curated web) are upsampled 2–5x, meaning the model sees them multiple times per epoch. Medium-quality web text forms the bulk of the corpus. Low-quality but topically unique content is included at reduced weight to preserve diversity. This approach, used by Dolma and FineWeb, outperforms both pure high-quality (too little diversity) and unfiltered (too much noise) baselines.

# Data mixing configuration for pre-training from dataclasses import dataclass, field from typing import Dict, List import numpy as np @dataclass class DataSource: name: str path: str tokens: int # available tokens in billions weight: float # sampling weight (will be normalized) epochs: float = 1.0 # how many times to repeat this source quality_tier: str = "medium" class DataMixer: def __init__(self, sources: List[DataSource], total_tokens: int): self.sources = sources self.total_tokens = total_tokens def compute_sampling_probs(self) -> Dict[str, float]: # Normalize weights to sampling probabilities total_weight = sum(s.weight for s in self.sources) probs = {} for s in self.sources: probs[s.name] = s.weight / total_weight return probs def compute_effective_epochs(self) -> Dict[str, float]: # How many times each source is seen given the mix probs = self.compute_sampling_probs() epochs = {} for s in self.sources: tokens_sampled = self.total_tokens * probs[s.name] epochs[s.name] = tokens_sampled / (s.tokens * 1e9) return epochs # Example: LLaMA-style mixing configuration sources = [ DataSource("common_crawl", "/data/cc", tokens=900, weight=0.67), DataSource("c4", "/data/c4", tokens=175, weight=0.15), DataSource("github", "/data/code", tokens=100, weight=0.045), DataSource("wikipedia", "/data/wiki", tokens=4, weight=0.045, quality_tier="high", epochs=3.0), DataSource("books", "/data/books", tokens=25, weight=0.045, quality_tier="high", epochs=2.0), DataSource("arxiv", "/data/arxiv", tokens=50, weight=0.025, quality_tier="high"), DataSource("stackexchange", "/data/se", tokens=15, weight=0.02, quality_tier="high"), ] mixer = DataMixer(sources, total_tokens=int(1.4e12)) probs = mixer.compute_sampling_probs() epochs = mixer.compute_effective_epochs()

DoReMi: Learned Domain Weights

DoReMi (Xie et al., 2023) replaces manual mixing with a learned optimization. It trains a small proxy model on a reference distribution, then uses the proxy's per-domain excess loss to re-weight domains. The intuition: domains where the proxy struggles most benefit most from additional training data. DoReMi found that upsampling scientific text and code (which have high loss) by 3–5x over their natural frequency significantly improved downstream task performance on MMLU and HumanEval, even compared to human-tuned mixing ratios.

Production Mixing Tip: When upsampling high-quality sources more than ~4 epochs, watch for memorization. Models begin regurgitating Wikipedia passages verbatim after 5+ epochs. Use fuzzy deduplication (MinHash with Jaccard threshold 0.8) between the training set and evaluation benchmarks to detect contamination, and monitor training loss per-domain for signs of overfitting (loss dropping to near-zero on a domain signals memorization).

The Pile’s Approach

EleutherAI’s The Pile (Gao et al., 2020) demonstrated that including 22 diverse sub-corpora—from Pile-CC (web crawl) to PhilPapers (philosophy), FreeLaw (legal text), and Ubuntu IRC (chat)—with carefully tuned weights produced models with broader capabilities than training on web-only data. Their key insight was that niche domains, even at 0.1% of the total corpus, materially improve performance on corresponding downstream tasks. This challenges the intuition that tiny data fractions are irrelevant at trillion-token scale.

Decontamination Warning: Always deduplicate your training mix against benchmark test sets (MMLU, HellaSwag, ARC, HumanEval, GSM8K). Without decontamination, reported benchmark scores can be inflated by 2–8 percentage points, as shown by the GPT-4 technical report. Use 13-gram overlap detection at minimum, or the more robust contamination detection from the Open LLM Leaderboard methodology (8-gram with character normalization).
