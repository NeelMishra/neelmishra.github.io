Megatron-LM: 3D Parallelism

MLOps Series LLM Training Infrastructure

Training models with hundreds of billions of parameters demands distributing computation across thousands of GPUs. Megatron-LM, developed by NVIDIA, introduced a composable 3D parallelism strategy that combines tensor parallelism (TP), pipeline parallelism (PP), and data parallelism (DP) into a single unified framework. The key insight is that each parallelism dimension operates on a different axis of the computation and can be composed orthogonally: TP splits individual operations within a layer, PP partitions layers across stages, and DP replicates the entire pipeline across groups. The total GPU count satisfies world_size = TP × PP × DP.

Tensor Parallelism

Tensor parallelism splits individual weight matrices across GPUs within a single node to exploit the high-bandwidth NVLink interconnect (600+ GB/s on A100/H100 DGX). Megatron-LM implements two complementary partitioning strategies for the Transformer's core sub-layers.

Column-Parallel Linear (MLP first projection & QKV)

The weight matrix A of shape [h, 4h] is split column-wise across t GPUs: each GPU holds a shard A_i of shape [h, 4h/t]. The input X is replicated on all ranks. Each GPU computes Y_i = XA_i independently—no communication is needed for the forward pass of this layer. The GeLU activation is applied locally because each GPU holds a complete row of the output.

Row-Parallel Linear (MLP second projection & output projection)

The second weight matrix B of shape [4h, h] is split row-wise: each GPU holds B_i of shape [4h/t, h]. Each GPU computes a partial result Z_i = Y_i B_i, then an all-reduce sums the partial results across the TP group: Z = Σ Z_i. This requires one all-reduce per Transformer sub-layer (MLP or attention output).

Communication cost per layer: Megatron-LM requires exactly 4 all-reduce operations per Transformer layer in the forward+backward pass (2 in the forward pass: one for the attention output projection, one for the MLP second projection; 2 corresponding all-reduces in the backward pass for the input gradients). Each all-reduce transfers 2 × b × s × h bytes using the ring algorithm.

Self-Attention Parallelism

Multi-head attention is naturally parallelizable: with n heads and t TP ranks, each GPU handles n/t attention heads. The Q, K, V projections are column-parallel (each GPU computes its local heads), attention is computed independently per head, and the output projection is row-parallel with an all-reduce to aggregate results. This is mathematically identical to standard multi-head attention since heads are independent.

# Megatron-LM Column-Parallel Linear (simplified) class ColumnParallelLinear(torch.nn.Module): def __init__(self, in_features, out_features, tp_size): self.weight = Parameter(torch.empty(out_features // tp_size, in_features)) self.tp_rank = get_tensor_model_parallel_rank() def forward(self, x): # x is replicated across TP group; no communication needed output_parallel = F.linear(x, self.weight) # [b, s, 4h/t] return output_parallel class RowParallelLinear(torch.nn.Module): def __init__(self, in_features, out_features, tp_size): self.weight = Parameter(torch.empty(out_features, in_features // tp_size)) def forward(self, x_parallel): output_partial = F.linear(x_parallel, self.weight) # [b, s, h] # All-reduce across TP group to sum partial results output = all_reduce(output_partial, group=get_tp_group()) return output

Pipeline Parallelism

Pipeline parallelism partitions the model's layers across multiple GPUs (stages). Each stage holds a contiguous set of L / p layers, where L is the total layer count and p is the number of pipeline stages. The critical challenge is minimizing the pipeline bubble—idle time when stages wait for activations or gradients.

GPipe Schedule

The naive approach (GPipe) runs all forward micro-batches first, then all backward micro-batches. The bubble fraction is (p - 1) / m where m is the number of micro-batches. For p = 8 stages and m = 32 micro-batches, the bubble ratio is 21.9% — a significant waste of compute.

1F1B Schedule (Megatron-LM Default)

The one-forward-one-backward (1F1B) schedule interleaves forward and backward passes to reduce peak memory. After a warm-up phase of p - 1 forward passes, each stage alternates between one forward and one backward micro-batch. This keeps at most p micro-batch activations in memory simultaneously (vs. m for GPipe), while maintaining the same bubble fraction of (p - 1) / m.

Interleaved 1F1B Schedule

Megatron-LM's interleaved schedule assigns v non-contiguous virtual stages to each physical device. With v virtual stages per device, each device holds v × (L / p) non-contiguous layers. The bubble fraction reduces to (p - 1) / (v × m). For v = 2, the bubble is halved compared to standard 1F1B. The trade-off is increased point-to-point communication volume by a factor of v, since activations must be transferred between non-adjacent stages.

Memory vs. Bubble Trade-off: Increasing micro-batches m reduces the bubble fraction but increases activation memory (under GPipe) or requires gradient accumulation. The interleaved schedule is preferred when inter-node bandwidth supports the extra communication.
# 1F1B Pipeline Schedule (per stage) def pipeline_1f1b(model_chunks, micro_batches, num_stages, stage_id): num_warmup = num_stages - stage_id - 1 # warm-up forward passes num_microbatches = len(micro_batches) # Phase 1: Warm-up — forward passes only for i in range(num_warmup): recv_forward() # P2P receive from previous stage output = forward_step(model_chunks, micro_batches[i]) send_forward(output) # P2P send to next stage # Phase 2: Steady state — alternate 1F, 1B for i in range(num_warmup, num_microbatches): recv_forward() output = forward_step(model_chunks, micro_batches[i]) send_forward(output) recv_backward() # P2P receive grad from next stage grad = backward_step(model_chunks, i - num_warmup) send_backward(grad) # P2P send grad to previous stage # Phase 3: Cool-down — backward passes only for i in range(num_warmup): recv_backward() grad = backward_step(model_chunks, num_microbatches - num_warmup + i) send_backward(grad)

Data Parallelism

Data parallelism replicates the entire model (or pipeline) across multiple GPU groups, with each replica processing different data. After every training step, gradients are synchronized via all-reduce across the DP dimension. In Megatron-LM, DP is the outermost parallelism dimension: DP_size = world_size / (TP × PP).

Gradient All-Reduce

Standard all-reduce uses the ring algorithm with cost 2(n-1)/n × M where M is the total gradient size and n is the DP degree. For a 175B parameter model in FP16, M ≈ 350 GB. With 8-way DP, each GPU transfers approximately 2 × 350 × (7/8) ≈ 612 GB. At 50 GB/s inter-node bandwidth, this takes ~12.3 seconds—often overlapped with the backward pass.

Gradient Accumulation

To increase the effective batch size without proportionally scaling DP, Megatron-LM accumulates gradients across multiple micro-batches before synchronizing. With k accumulation steps, the effective global batch size becomes DP × k × micro_batch_size. This amortizes the all-reduce cost over k forward-backward iterations, improving the compute-to-communication ratio.

# Data-parallel gradient synchronization with accumulation def train_step(model, data_loader, dp_group, accum_steps): optimizer.zero_grad() for step in range(accum_steps): micro_batch = next(data_loader) loss = model.forward(micro_batch) / accum_steps # scale loss loss.backward() # accumulate gradients locally # Synchronize gradients across DP group for param in model.parameters(): all_reduce(param.grad, op=ReduceOp.SUM, group=dp_group) param.grad /= get_dp_world_size() optimizer.step()
Overlap opportunity: Megatron-LM overlaps the DP all-reduce with backward computation by bucketing gradients and launching all-reduce as soon as a bucket is complete (similar to PyTorch DDP). This hides most of the communication latency behind compute.

Megatron Configuration & Performance

Choosing the right parallelism configuration requires balancing memory, communication, and compute. Below is a reference configuration for training a GPT-3 175B model on 64 A100 80GB GPUs (8 nodes × 8 GPUs).

Memory Budget per GPU

Parameters: 175B × 2 bytes (FP16) / (TP=8) = 43.75 GB shard per TP group.
Optimizer states (Adam): 12 bytes/param × 175B / (TP × DP) = varies with DP degree.
Activations: proportional to b × s × h × L/PP; recomputation reduces to O(sqrt).

Recommended Layout (175B)

TP=8 (intra-node, 8 GPUs per DGX A100)
PP=8 (8 stages across 8 nodes)
DP=1 (single replica; use gradient accumulation)
Global batch size: 1536 sequences × 2048 tokens = ~3.1M tokens per step.

# Megatron-LM launch configuration for GPT-3 175B DISTRIBUTED_ARGS=" --nproc_per_node 8 --nnodes 8 --master_addr $MASTER_ADDR --master_port 6000 " MEGATRON_ARGS=" --tensor-model-parallel-size 8 --pipeline-model-parallel-size 8 --num-layers 96 --hidden-size 12288 --num-attention-heads 96 --seq-length 2048 --micro-batch-size 1 --global-batch-size 1536 --train-iters 300000 --lr 6.0e-5 --min-lr 6.0e-6 --lr-warmup-iters 375 --lr-decay-style cosine --fp16 --activations-checkpoint-method uniform --recompute-granularity selective " # Launch: torchrun $DISTRIBUTED_ARGS pretrain_gpt.py $MEGATRON_ARGS
Rule of thumb for TP: Always keep TP ≤ GPUs per node (typically 8 for DGX). Cross-node TP is possible but the 10–50× bandwidth penalty (InfiniBand vs NVLink) makes it impractical. If you need more TP, use PP or DP to span across nodes.

Activation Recomputation

Megatron-LM supports selective recomputation, which recomputes only the memory-intensive operations (e.g., softmax, dropout) during the backward pass while caching the linear projections. This saves ~60% of activation memory compared to full caching with only ~5% recomputation overhead, significantly reducing the memory pressure per pipeline stage.

Performance Formulas & Analysis

Evaluating training efficiency requires precise metrics. The two most important are Model FLOPs Utilization (MFU) and pipeline bubble ratio.

Model FLOPs Utilization (MFU)

MFU measures what fraction of the hardware's peak FLOPS is used for useful model computation (excluding recomputation overhead). For a Transformer with P parameters, sequence length s, and batch size B:

FLOPs_per_step ≈ 6 × P × B × s
MFU = FLOPs_per_step / (GPU_count × peak_FLOPS × step_time)

The factor of 6 comes from: 2× for forward multiply-adds, 4× for backward (2× for weight gradients, 2× for activation gradients). An A100 at peak FP16 = 312 TFLOPS. Good MFU for large-scale training is 40–55%; Megatron-LM achieves ~52% MFU on 175B at 1024 GPUs.

Pipeline Bubble Analysis

The bubble fraction for the standard 1F1B schedule:

bubble_fraction = (p - 1) / m

where p = pipeline stages, m = micro-batches per step. For the interleaved schedule with v virtual stages:

bubble_fraction_interleaved = (p - 1) / (v × m)

Practical constraint: Setting m ≥ 4 × p keeps the bubble under 25%. For p = 8, you need at least m = 32 micro-batches, which means the global batch size is 32 × DP × micro_batch_size. Very large models with many pipeline stages force large batch sizes, which can degrade convergence at the start of training.

Communication Volume Summary

The per-step communication cost for each parallelism dimension:

Tensor Parallelism

All-reduce per layer: 4 × 2(t-1)/t × b × s × h × sizeof(dtype)
Total: × L/PP layers on this stage
Link: Intra-node NVLink (600 GB/s on H100)

Pipeline Parallelism

P2P per micro-batch: b × s × h × sizeof(dtype)
Total: × m micro-batches (fwd+bwd)
Link: Inter-node InfiniBand or NVLink bridge

# Compute MFU for a training run def compute_mfu(num_params, batch_size, seq_len, step_time_s, num_gpus, peak_tflops): # 6 * P * B * s = approximate FLOPs per training step flops_per_step = 6 * num_params * batch_size * seq_len # Convert peak_tflops to FLOPS total_peak_flops = num_gpus * peak_tflops * 1e12 # MFU = achieved / theoretical peak mfu = flops_per_step / (total_peak_flops * step_time_s) return mfu # Example: GPT-3 175B on 1024 A100s mfu = compute_mfu( num_params=175e9, batch_size=1536, seq_len=2048, step_time_s=60.0, # 60 seconds per step num_gpus=1024, peak_tflops=312 # A100 FP16 peak ) print(f"MFU: {mfu:.1%}") # ~52.2% # Pipeline bubble ratio def bubble_ratio(num_stages, num_microbatches, virtual_stages=1): return (num_stages - 1) / (virtual_stages * num_microbatches) print(bubble_ratio(8, 32)) # 0.219 = 21.9% print(bubble_ratio(8, 32, v=2)) # 0.109 = 10.9% with interleaved

Choosing the Optimal Configuration

The search space for (TP, PP, DP) is constrained by hardware topology and memory. A systematic approach: (1) Set TP to the number of GPUs per node if the model is large enough to require intra-layer splitting. (2) Choose PP to fit the model's layers in memory across stages. (3) DP fills the remaining GPUs: DP = N / (TP × PP). (4) Tune micro-batch size and gradient accumulation to minimize the bubble while maintaining the target global batch size. Tools like Megatron-LM's --eval-interval benchmarking and NVIDIA's nsight systems profiler help identify bottlenecks empirically.
