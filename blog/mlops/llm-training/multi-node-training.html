Multi-Node Training: NVLink, NVSwitch

MLOps Series LLM Training Infrastructure

Training large language models at the frontier—70B, 175B, or trillion-parameter scale—requires distributing computation across hundreds or thousands of GPUs spanning many physical machines. The interconnect fabric between those GPUs is the single largest determinant of training throughput. Understanding the full hierarchy from on-die memory buses to cross-datacenter networking is essential for anyone designing or debugging large-scale training runs.

GPU Interconnect Hierarchy

Modern GPU clusters expose a four-level communication hierarchy, each with dramatically different bandwidth and latency characteristics. Every distributed training framework—from Megatron-LM to DeepSpeed—must be topology-aware to extract peak performance.

The hierarchy, from fastest to slowest:

  1. HBM (on-die): GPU to its own memory — 3.35 TB/s on H100 SXM (HBM3). This is the ceiling; all interconnects are bottlenecked below this.
  2. NVLink (intra-node, GPU-to-GPU): Direct high-speed serial links between GPUs. Fourth-generation NVLink on H100 provides 900 GB/s bidirectional per GPU via 18 links × 25 GB/s per link per direction.
  3. PCIe (intra-node, GPU-to-CPU/NIC): Gen 5.0 x16 delivers 64 GB/s bidirectional. Used for CPU–GPU data transfers and NIC-attached network I/O when GPUDirect is not available.
  4. InfiniBand / RoCE (inter-node): NDR InfiniBand provides 400 Gb/s (50 GB/s) per port. DGX H100 nodes have 8 NICs for 400 GB/s aggregate inter-node bandwidth.
Key insight: The NVLink-to-InfiniBand bandwidth ratio on DGX H100 is approximately 900:50 per GPU (18:1). This means intra-node communication is roughly 18× faster than inter-node, which is why tensor parallelism (high-bandwidth) should stay within a node while pipeline and data parallelism (lower bandwidth) span nodes.

Bandwidth Across Generations

Each GPU generation brought substantial interconnect improvements. Understanding the numbers helps you reason about when upgrading hardware versus optimizing software yields better returns.

NVLink Evolution

  • NVLink 1.0 (P100): 4 links × 20 GB/s = 160 GB/s bidir total
  • NVLink 2.0 (V100): 6 links × 25 GB/s = 300 GB/s bidir total
  • NVLink 3.0 (A100): 12 links × 25 GB/s = 600 GB/s bidir total
  • NVLink 4.0 (H100): 18 links × 25 GB/s = 900 GB/s bidir total
  • NVLink 5.0 (B200): 18 links × 50 GB/s = 1800 GB/s bidir total

NVLink 4.0 uses PAM4 signaling at 50 Gbaud per lane, with each link consisting of 4 differential pairs (sub-links) in each direction. NVLink 5.0 doubles per-link bandwidth using improved SerDes and signal integrity.

InfiniBand Evolution

  • FDR (2011): 56 Gb/s per port = 7 GB/s
  • EDR (2014): 100 Gb/s per port = 12.5 GB/s
  • HDR (2019): 200 Gb/s per port = 25 GB/s
  • NDR (2022): 400 Gb/s per port = 50 GB/s
  • XDR (2025): 800 Gb/s per port = 100 GB/s

NDR InfiniBand uses two 200G sub-ports (NDR200), enabling flexible 2:1 bandwidth oversubscription. XDR doubles the lane rate to 133 Gbaud with PAM4 signaling.

PCIe Evolution

  • PCIe 3.0 x16: 32 GB/s bidirectional
  • PCIe 4.0 x16: 64 GB/s bidirectional (A100 PCIe)
  • PCIe 5.0 x16: 128 GB/s bidirectional (H100 SXM)
  • PCIe 6.0 x16: 256 GB/s bidirectional (B200)

PCIe matters for NIC-attached networking when GPUDirect RDMA bypasses the CPU. The H100 SXM variant uses PCIe 5.0 to connect ConnectX-7 NICs.

HBM Evolution

  • HBM2 (V100): 900 GB/s, 32 GB
  • HBM2e (A100): 2.0 TB/s, 80 GB
  • HBM3 (H100): 3.35 TB/s, 80 GB
  • HBM3e (B200): 8.0 TB/s, 192 GB

HBM bandwidth sets the upper bound for compute utilization. When the arithmetic intensity of a kernel is low (e.g., attention for long sequences), HBM bandwidth becomes the bottleneck rather than the interconnect.

Warning: Published bandwidth numbers are theoretical peaks. Effective NVLink bandwidth is typically 85–92% of peak due to protocol overhead, credit-based flow control, and link-level retransmission. InfiniBand effective throughput drops to ~80–85% for small messages (<256 KB) due to header overhead and switch-hop latency.

DGX Node Internal Topology

A DGX H100 node contains 8 H100 SXM GPUs, 6 NVSwitches, 2 CPUs (Intel Xeon or Grace), and 8 ConnectX-7 InfiniBand NICs. The internal wiring follows a fully-connected NVSwitch topology.

NVSwitch Architecture

Each of the 6 NVSwitch chips connects to all 8 GPUs. Each GPU dedicates 3 NVLinks to each NVSwitch (3 links × 6 switches = 18 links total per GPU). The NVSwitch operates as a crossbar, enabling any GPU to communicate with any other GPU at the full 900 GB/s bidirectional bandwidth simultaneously—this is the critical advantage over point-to-point NVLink topologies.

Without NVSwitch (as in older DGX-1 V100 hybrid cube-mesh topology), GPU 0 might have direct NVLink connections to GPUs 1, 2, 3, and 4, but reach GPUs 5, 6, and 7 only via multi-hop routing through intermediate GPUs. This created non-uniform communication costs. NVSwitch eliminates this entirely: every pair gets full bisection bandwidth.

NIC-to-GPU Affinity

The 8 ConnectX-7 NICs are not randomly assigned. Each NIC has PCIe affinity to exactly one GPU, routed through the CPU’s PCIe root complex (or, with Grace Hopper, through NVLink-C2C). This affinity is critical for GPUDirect RDMA: data from GPU 3 should egress through NIC 3 to avoid crossing the CPU QPI/UPI bus.

# Query NIC-to-GPU affinity on a DGX H100 for i in $(seq 0 7); do nvidia-smi topo -m | grep "GPU$i" ibstat mlx5_$i | grep "Port" done # Expected output: GPU i <-> mlx5_i via PCIe (PIX or PHB) # PIX = same PCIe switch, PHB = same host bridge # SYS = cross-socket (bad for GPUDirect perf)
NVSwitch generations: The first-gen NVSwitch (V100 era) supported 18 NVLink 2.0 ports. The third-gen NVSwitch (H100 era) supports 64 NVLink 4.0 ports and can be used in a rack-scale configuration (NVLink Network) connecting up to 256 GPUs across 32 nodes without any InfiniBand—this is the DGX SuperPOD with NVLink architecture.

Multi-Node Collective Communication

Distributed training relies on a small set of collective communication primitives. Understanding their bandwidth and latency characteristics is essential for choosing parallelism strategies.

Core Primitives

All-Reduce

Every GPU contributes a tensor; every GPU receives the element-wise sum. Used in data parallelism to synchronize gradients. For n GPUs with message size M, the optimal ring algorithm transfers 2M(n-1)/n bytes in 2(n-1) steps.

Bandwidth cost: 2M · (n−1)/n ≈ 2M for large n
Latency cost: 2(n−1) · α

All-Gather

Each GPU contributes a shard; every GPU receives the full concatenation. Used in FSDP to reconstruct full parameter tensors before forward/backward passes. Transfers M(n-1)/n bytes.

Bandwidth cost: M · (n−1)/n ≈ M for large n
Latency cost: (n−1) · α

Reduce-Scatter

Each GPU contributes a full tensor; each receives a reduced shard. The inverse of all-gather. Used in ZeRO Stage 3 / FSDP for gradient accumulation before scatter. Transfers M(n-1)/n bytes.

Bandwidth cost: M · (n−1)/n
Latency cost: (n−1) · α

Point-to-Point (Send/Recv)

Direct transfer between two GPUs. Used in pipeline parallelism for activations/gradients between stages. Bandwidth cost is simply M. Critical path latency includes network traversal + serialization.

Bandwidth cost: M
Latency cost: α + M/BW

Ring vs. Tree vs. Recursive Halving-Doubling

NCCL implements multiple algorithm variants:

  • Ring: Optimal bandwidth utilization (2M(n-1)/n), but O(n) latency. Best for large messages across many GPUs.
  • Tree: O(log n) latency but higher bandwidth cost. Preferred for small messages or latency-sensitive collectives.
  • Recursive halving-doubling (NVLS): Combines NVLink multicast with switch-level reduction on NVSwitch 3.0, achieving near-hardware-optimal throughput for intra-node collectives.
Bandwidth-optimal decomposition: All-reduce = reduce-scatter + all-gather. This decomposition is used by ZeRO/FSDP to overlap the reduce-scatter of layer N gradients with the backward pass of layer N-1, and overlap all-gather of layer N+1 parameters with the forward pass of layer N. This is the key to hiding communication latency behind computation.

NCCL Configuration & Tuning

NVIDIA Collective Communications Library (NCCL) is the de-facto standard for GPU-to-GPU collectives. Proper configuration can yield 20–40% throughput improvements on misconfigured clusters.

# Critical NCCL environment variables for multi-node H100 training # Network interface selection — bind to IB interfaces only export NCCL_SOCKET_IFNAME=ib0 export NCCL_IB_HCA=mlx5_0,mlx5_1,mlx5_2,mlx5_3,mlx5_4,mlx5_5,mlx5_6,mlx5_7 # Enable GPUDirect RDMA for zero-copy GPU-to-NIC transfers export NCCL_NET_GDR_LEVEL=5 export NCCL_NET_GDR_READ=1 # Protocol tuning: LL (low-latency), LL128, Simple # Simple protocol is best for large messages (>256KB) export NCCL_PROTO=Simple export NCCL_ALGO=Ring # Buffer sizes — larger buffers improve throughput for big models export NCCL_BUFFSIZE=8388608 # Enable NVLink SHARP (in-network reduction on NVSwitch) export NCCL_NVLS_ENABLE=1 # Cross-node tuning: number of NCCL channels # More channels = more parallelism but more memory overhead export NCCL_MIN_NCHANNELS=16 export NCCL_MAX_NCHANNELS=32 # Debugging — enable only during diagnosis export NCCL_DEBUG=INFO export NCCL_DEBUG_SUBSYS=INIT,NET

NCCL Channel and Protocol Internals

NCCL decomposes each collective operation into channels, each independently executing on its own set of GPU threads and NVLink/IB resources. For an all-reduce with 16 channels, the message is split into 16 chunks, and each chunk runs through its own ring or tree independently. More channels increase GPU SM utilization but consume more GPU memory for internal buffers.

Three wire protocols handle different message sizes:

  • LL (Low Latency): 8-byte inline data in control messages. Best for <8 KB messages. Adds 50% bandwidth overhead due to flags.
  • LL128: 128-byte packets with embedded synchronization. Sweet spot for 8 KB–256 KB.
  • Simple: Standard memory copy with explicit synchronization. Optimal for >256 KB. Uses GPU copy engines (CE) for NVLink and RDMA engines for IB.
Common misconfiguration: Setting NCCL_IB_HCA without checking NIC-to-GPU affinity. If GPU 0 is forced to use mlx5_4 (affinitized to GPU 4), traffic crosses the CPU interconnect, adding 2–5 µs latency per message and reducing throughput by 30–50%. Always verify with nvidia-smi topo -m.

Network Topology Design

The physical network connecting DGX nodes has profound effects on all-reduce performance. Two dominant topologies prevail in LLM training clusters:

Fat-Tree Topology

Classic Clos network with leaf and spine switches. Each DGX node’s 8 NICs connect to leaf switches; leaf switches uplink to spines. Full bisection bandwidth means any-to-any communication at line rate.

Pros: Maximum flexibility for arbitrary communication patterns. NCCL ring/tree algorithms work optimally. Supports heterogeneous parallelism strategies.

Cons: Expensive — requires O(n²) switch ports for full bisection. Cable complexity grows rapidly. Spine switches become single points of congestion.

Typical scale: NVIDIA DGX SuperPOD uses a 2-level fat-tree with 32 DGX nodes (256 GPUs) per leaf domain and spine switches for cross-domain traffic.

Rail-Optimized Topology

Each “rail” connects the same-index GPU across all nodes. GPU 0 on all nodes shares a dedicated network rail via a single switch; GPU 1 shares another rail, etc. This creates 8 independent networks.

Pros: Dramatically fewer switches and cables. Each rail is a simple 1-hop flat network. NCCL can run independent rings on each rail with zero cross-rail interference.

Cons: Only efficient for rail-aligned collective patterns (i.e., same-index GPU communication). Cross-rail traffic must traverse intra-node NVLink, which consumes shared NVLink bandwidth.

Typical scale: Meta’s Grand Teton and Google’s A3 Mega use rail-optimized topologies for clusters of 1000+ nodes, as data parallelism collectives naturally align to rails.

Hybrid Topology Decisions

Production clusters often use rail-optimized within a pod and fat-tree across pods. The decision hinges on the parallelism strategy:

  • Tensor Parallelism (TP): Stays within a single node (8 GPUs on NVLink). No network traffic.
  • Pipeline Parallelism (PP): Point-to-point sends between consecutive stages. Can use any rail since it’s a 1-to-1 pattern.
  • Data Parallelism (DP) / FSDP: All-reduce or reduce-scatter + all-gather across the same-rank GPUs on different nodes. Naturally rail-aligned — GPU i on node A communicates with GPU i on node B.
  • Expert Parallelism (EP): All-to-all communication for MoE routing. This is the worst case for rail-optimized topologies because tokens must reach arbitrary GPUs, requiring both cross-rail NVLink and network hops.
MoE training pitfall: On rail-optimized networks, Mixture-of-Experts all-to-all can see 40–60% bandwidth degradation compared to fat-tree. Solutions include capacity-factor tuning to reduce cross-rail communication, expert placement co-optimization, and hierarchical all-to-all (intra-node NVLink first, then inter-node IB).

RDMA & GPUDirect Technologies

Remote Direct Memory Access (RDMA) eliminates CPU involvement in network transfers. For GPU training, three GPUDirect technologies progressively remove data-path bottlenecks:

GPUDirect Peer-to-Peer (P2P)

Enables one GPU to directly access another GPU’s memory over NVLink or PCIe without staging through host memory. This is the foundation of all intra-node NCCL communication. Requires GPUs on the same PCIe tree or connected via NVLink.

GPUDirect RDMA (GDR)

Extends RDMA to GPU memory: a NIC can DMA directly to/from GPU memory without CPU-mediated copies. The data path is: GPU HBM → PCIe → NIC → InfiniBand → remote NIC → PCIe → remote GPU HBM. This eliminates two CPU-memory copies (GPU→host, host→NIC on the send side; NIC→host, host→GPU on the receive side).

# Verify GPUDirect RDMA is functional # nvidia-peermem kernel module must be loaded lsmod | grep nvidia_peermem # Benchmark GDR vs. non-GDR latency # With GDR (NCCL_NET_GDR_LEVEL=5): ~1.5 μs for 8-byte msg # Without GDR (bounce via CPU): ~5.0 μs for 8-byte msg # Run NCCL bandwidth test to measure effective throughput nccl-tests/build/all_reduce_perf -b 1M -e 1G -f 2 -g 1 -n 100 # Expected results on 2-node DGX H100 (16 GPUs): # 1MB: ~38 GB/s bus bandwidth # 64MB: ~42 GB/s bus bandwidth # 1GB: ~44 GB/s bus bandwidth (close to line rate)

GPUDirect RDMA with SHARP

Scalable Hierarchical Aggregation and Reduction Protocol (SHARP) offloads reduction operations into InfiniBand switches. Instead of an all-reduce spanning all nodes, each node sends its partial data to the switch, which performs the reduction in hardware and multicasts the result. This reduces network traffic by ~50% for all-reduce and cuts latency from O(log n) to O(1) for the network component.

GPUDirect Async (GDA)

Introduced in CUDA 12.4, GDA enables GPU kernels to directly trigger RDMA operations without CPU involvement. A GPU kernel can enqueue a send/recv that the NIC processes asynchronously. This eliminates the CPU-launched-kernel-triggers-NCCL-which-triggers-NIC pipeline, removing several microseconds of latency and enabling true overlap of computation and communication at the kernel level.

Memory registration matters: RDMA requires memory to be registered (pinned) with the NIC. GPU memory registration via ibv_reg_mr() with the nvidia-peermem module pins GPU physical pages. For large models (70B+ parameters), the registration overhead at startup can be 10–30 seconds. NCCL caches registrations, so the cost is amortized across training steps.

Latency Analysis for Parallelism Strategies

Each parallelism dimension imposes different communication volume and latency. Optimal mapping of parallelism to the physical topology minimizes the aggregate communication cost.

Communication Cost Model

# Latency model: T = α + M/BW + γ # α = startup/software overhead (~3-10 μs intra-node, ~5-15 μs inter-node) # M = message size (bytes) # BW = effective bandwidth at the relevant hierarchy level # γ = congestion/contention overhead (topology-dependent) # For TP all-reduce (intra-node, 8 H100 GPUs, msg = hidden_dim × dtype): # Example: LLaMA-70B, hidden=8192, bf16: M = 8192 × 2 = 16KB per tensor # T_nvlink = 3μs + 16KB / (450 GB/s) = 3μs + 0.035μs ≈ 3μs (latency-bound) # For DP all-reduce (inter-node, 128 nodes, msg = gradient bucket = 25MB): # T_ib = 10μs + 2 × 25MB × (1023/1024) / (8 × 50 GB/s) # = 10μs + 50MB / 400 GB/s = 10μs + 125μs ≈ 135μs per bucket # With overlap: amortized to near-zero if compute time per layer > 135μs # For PP send/recv (inter-node, 1 activation tensor): # Example: micro-batch × seq_len × hidden = 4 × 4096 × 8192 × 2 = 256MB # T_pp = 10μs + 256MB / 50 GB/s = 10μs + 5.12ms ≈ 5.13ms per microbatch # Pipeline bubble = (p-1) × T_pp where p = number of stages

Optimal Parallelism-to-Topology Mapping

For a 3D parallelism configuration training a 175B model on 512 GPUs (64 DGX H100 nodes):

  • TP = 8: Within each node. 8 GPUs connected via NVSwitch at 900 GB/s. All-reduce for partial activations in every transformer layer (latency-sensitive, small messages).
  • PP = 8: Across 8 nodes within a pod. Point-to-point activation transfers over InfiniBand. One rail is sufficient since it’s a 1:1 pattern. Pipeline bubble overhead = (8−1) × microbatch_time.
  • DP = 8: Across 8 pod replicas. All-reduce gradients over InfiniBand. Rail-aligned (GPU i communicates with GPU i). Overlapped with backward pass computation.
The golden rule: Assign the parallelism dimension with the highest communication frequency and smallest messages to the fastest interconnect. TP has per-layer communication with small messages → NVLink. PP has per-microbatch communication with large messages → InfiniBand (latency-tolerant with enough microbatches). DP has per-step communication with very large messages → InfiniBand (bandwidth-sensitive but overlappable).

Bubble Overhead and Interleaved Scheduling

1F1B (one forward, one backward) scheduling reduces the pipeline bubble from (p-1) × (t_f + t_b) to (p-1) × t_f by interleaving forward and backward microbatches. With interleaved schedules (Megatron-LM’s virtual pipeline stages), the bubble shrinks further to (p-1) × t_f / v where v is the number of virtual stages, at the cost of v× more point-to-point communication volume.

Debugging & Profiling Multi-Node Training

When a 512-GPU training run achieves only 40% MFU instead of the expected 55%, systematic debugging of the interconnect is essential.

NCCL Debug Logging

# Enable detailed NCCL logging to identify algorithm selection export NCCL_DEBUG=INFO export NCCL_DEBUG_SUBSYS=INIT,GRAPH,TUNING # Key log lines to watch for: # "Channel 00/16 : 0[0] -> 1[1] via P2P/IPC/read" — intra-node NVLink path # "Channel 00/16 : 8[0] -> 0[0] via NET/IB/0" — inter-node IB path # "Algorithm: Ring / Protocol: Simple / nChannels: 16" — selected algo # If you see "NET/Socket" instead of "NET/IB", RDMA is broken # If nChannels is low (e.g., 2), topology detection may be wrong # Use NCCL_GRAPH_DUMP_FILE to export the communication graph export NCCL_GRAPH_DUMP_FILE=nccl_graph.xml # Inspect the graph to verify ring ordering matches physical topology

Common Failure Modes

  • NVLink error: nvidia-smi nvlink -s shows replay errors or CRC failures. Often caused by cable degradation or thermal issues. A single degraded NVLink drops intra-node bandwidth by 1/18 = 5.5%.
  • IB port flapping: Check ibdiagnet and perfquery for symbol errors. A flapping port causes NCCL to timeout and triggers a full restart (expensive for checkpointing-averse runs).
  • PCIe bandwidth theft: CPU-side data loading or preprocessing consuming PCIe bandwidth, reducing GPUDirect RDMA throughput. Move all data preprocessing to GPU or use dedicated data-loading nodes.
  • Straggler nodes: One node with a faulty NIC or thermal-throttled GPU can slow down the entire all-reduce. Monitor per-node NCCL timing with NCCL_DEBUG=TRACE and look for asymmetric completion times.
  • NCCL timeout: Default NCCL_TIMEOUT is 30 minutes. For very large collectives on 10k+ GPUs, increase to avoid false-positive hangs during initialization. But also investigate why it’s slow.
Production monitoring: Instrument NCCL operations with NVTX ranges and export to Prometheus/Grafana. Track per-collective p99 latency and effective bus bandwidth. Alert when bus bandwidth drops below 80% of theoretical peak for 3+ consecutive training steps — this usually indicates a hardware degradation before a full failure.
