FlashAttention: Memory-Efficient Attention

The O(N²) Memory Problem in Standard Attention

The self-attention mechanism at the core of every Transformer computes Attention(Q, K, V) = softmax(QKT / √dk)V. For a sequence of length N and head dimension d, the intermediate attention score matrix S = QKT is of shape N × N. Materializing this matrix in GPU High-Bandwidth Memory (HBM) requires O(N²) memory — which becomes the dominant bottleneck for long-context models.

Consider concrete numbers: with sequence length N = 8192 and batch×heads = 32, the attention matrix alone is 32 × 8192 × 8192 × 2 bytes (fp16) ≈ 4 GB. At N = 32768 (32k context), that grows to 64 GB — exceeding the memory of even an A100-80GB GPU for a single layer. The standard PyTorch implementation proceeds in three kernel launches:

# Standard attention — three materialized N×N matrices in HBM import torch def standard_attention(Q, K, V, scale): S = torch.matmul(Q, K.transpose(-2, -1)) * scale # O(N²) HBM write P = torch.nn.functional.softmax(S, dim=-1) # O(N²) HBM read + write P = torch.nn.functional.dropout(P, p=0.1) # O(N²) HBM read + write O = torch.matmul(P, V) # O(N²) HBM read return O

Each step reads and writes the full N×N matrix from/to HBM. On an A100, HBM bandwidth is ~2 TB/s while peak FP16 compute is ~312 TFLOPS. The arithmetic intensity of softmax is O(1) FLOP per byte — making attention memory-bandwidth bound, not compute-bound. The GPU's tensor cores sit idle, waiting for data to arrive from HBM. FlashAttention's central insight is to restructure the computation to minimize these HBM round-trips by exploiting the GPU's faster, smaller on-chip SRAM.

Key insight: Standard attention is memory-bandwidth bound. The bottleneck is not the number of FLOPs but the number of bytes moved between HBM and the streaming multiprocessors. Reducing HBM accesses — even at the cost of extra recomputation — yields a net wall-clock speedup.

IO-Aware Algorithm Design: HBM vs SRAM

GPU memory has a distinct hierarchy analogous to the CPU cache hierarchy. On an NVIDIA A100, the two critical tiers are:

  • HBM (High-Bandwidth Memory): 80 GB capacity, ~2 TB/s bandwidth. This is where tensors normally reside.
  • SRAM (on-chip shared memory): ~20 MB total across 108 SMs (~192 KB per SM), but with bandwidth exceeding 19 TB/s — roughly 10× faster than HBM.

Standard attention kernels are "IO-unaware": they are written as a sequence of high-level PyTorch/cuBLAS calls, each of which materializes its output to HBM before the next kernel reads it back. FlashAttention is IO-aware — it explicitly reasons about data movement between HBM and SRAM, fusing the entire attention computation into a single kernel that tiles inputs into SRAM-resident blocks.

Standard Attention

  • 3 separate CUDA kernels (matmul → softmax → matmul)
  • Full N×N matrix written to HBM between kernels
  • HBM reads/writes: O(N²d + N²)
  • Memory: O(N²) for attention matrix
  • Bandwidth-bound at typical sequence lengths

FlashAttention

  • 1 fused CUDA kernel — never materializes full S or P
  • Tiles of Q, K, V loaded to SRAM; partial results accumulated in registers
  • HBM reads/writes: O(N²d² / M) where M = SRAM size
  • Memory: O(N) — only stores row-wise running max & sum
  • Approaches compute-bound regime

The IO complexity reduction is substantial. For typical values (d = 128, M = 192 KB per SM), the HBM traffic decreases by a factor proportional to d²/M — roughly 4–8× fewer bytes moved. Because the kernel is bandwidth-bound, this directly translates to a proportional wall-clock speedup. The authors proved that FlashAttention's IO complexity is optimal up to constant factors for any exact-attention algorithm that uses sub-quadratic extra memory.

Warning: FlashAttention computes exact attention — it is not an approximation. It produces bit-for-bit identical results to standard attention (up to floating-point non-associativity). This is in contrast to sparse or linear attention approximations that trade accuracy for speed.

Tiling Strategy: Block-Sparse Loads to SRAM

The core of FlashAttention is a two-level tiling loop that processes the attention matrix in blocks that fit in SRAM. Let Br and Bc denote the tile sizes for the row (query) and column (key) dimensions, respectively. Typical values are Br = Bc = 128 on A100 with d = 128.

The algorithm proceeds in two nested loops. The outer loop iterates over blocks of K and V (column tiles j = 1, …, ⌈N/Bc⌉). For each outer iteration, the inner loop iterates over blocks of Q (row tiles i = 1, …, ⌈N/Br⌉). Within the inner loop body:

  1. Load Qi (Br × d) from HBM to SRAM
  2. Load Kj (Bc × d) and Vj (Bc × d) from HBM to SRAM
  3. Compute local attention scores: Sij = Qi KjT (Br × Bc — fits in SRAM)
  4. Apply the online softmax rescaling (see next section) to accumulate into Oi
  5. After the inner loop completes for all j, write final Oi back to HBM

Crucially, the full N × N attention matrix is never materialized. Only blocks of size Br × Bc exist transiently in SRAM. The memory footprint drops from O(N²) to O(N) — we only store the output O (N × d), plus per-row running statistics (max and sum) of size O(N).

Tile size selection: The optimal Br and Bc depend on SRAM capacity M. We need 3 tiles in SRAM simultaneously: Brd + Bcd + BrBc ≤ M. For d = 128 and M = 192 KB (96K fp16 elements), Br = Bc = 128 uses ~98K elements — a tight fit that maximizes data reuse.

Online Softmax Trick

The fundamental challenge of tiling attention is that softmax is a global operation: computing softmax(Si,:) over a full row requires knowing the maximum value across all N columns, but we only see Bc columns at a time. The online softmax trick (Milakov & Gimelshein, 2018) solves this by maintaining running statistics that allow incremental updates.

For each row i, we maintain two scalars across tile iterations:

  • mi — the running maximum of Si,: seen so far
  • ℓi — the running sum of exp(Si,k − mi) seen so far

When we process a new tile j, we compute a local block of scores Sij = QiKjT. Let mij = max(Sij) along the column dimension. The update rules are:

# Online softmax update for tile j at row block i # m_prev, l_prev, O_prev are the running statistics from tiles 0..j-1 m_new = max(m_prev, m_ij) # global max so far # Rescale previous accumulator to account for the new global max alpha = exp(m_prev - m_new) # correction factor beta = exp(m_ij - m_new) # scale for new tile l_new = alpha * l_prev + beta * sum(exp(S_ij - m_ij)) # updated denominator # Update output accumulator with rescaling P_ij = beta * exp(S_ij - m_ij) # unnormalized local softmax O_new = alpha * O_prev + matmul(P_ij, V_j) # accumulate weighted values # After all tiles processed, final normalization: O_final = O_new / l_new # divide by the total softmax sum

The mathematical correctness relies on the identity: softmax(x)k = exp(xk − m) / Σ exp(xj − m) where m = max(x). When we see new elements that might change the global max, the exponential rescaling factors (α and β) correctly adjust the previously accumulated numerators and denominators. After processing all ⌈N/Bc⌉ tiles, the accumulated Oi and ℓi give us the exact output of softmax — no approximation involved.

Numerical stability: The subtraction of m (the running max) inside the exp() calls is critical for numerical stability. Without it, exp(Sij) can overflow to inf in fp16 when scores exceed ~11. This is the same log-sum-exp trick used in standard numerically-stable softmax, applied incrementally.

For the backward pass, FlashAttention avoids storing the full attention matrix P. Instead, it stores only O, ℓ, and m (all O(N) memory). During backpropagation, the Sij blocks are recomputed from Q and K tiles on the fly. This recomputation costs extra FLOPs but saves O(N²) memory, making the tradeoff favorable because the kernel remains bandwidth-bound.

FlashAttention-2: Better Work Partitioning

FlashAttention-1 achieved 25–40% of peak FP16 TFLOPS utilization on A100. FlashAttention-2 (Dao, 2023) pushed this to 50–73% through three key algorithmic refinements:

1. Reduced Non-Matmul FLOPs

In FlashAttention-1, the online softmax rescaling required multiple element-wise operations (multiply by α, multiply by β, etc.) that execute on CUDA cores rather than tensor cores. FlashAttention-2 restructures the computation to defer the rescaling: the output accumulator O is kept in an unnormalized form and the division by ℓ is done only once at the very end. This reduces non-matmul FLOPs by approximately 4× while maintaining numerical stability.

2. Parallelism Over Sequence Length

FlashAttention-1's outer loop iterates over K/V blocks, and the inner loop over Q blocks. Each thread block handles one Q block — limiting parallelism to batch_size × num_heads. FlashAttention-2 swaps the loop order: the outer loop now iterates over Q blocks, and the inner loop over K/V blocks. This allows parallelizing over the sequence length dimension as well, since different Q blocks are fully independent. On long sequences with small batch sizes (common during inference), this increases GPU occupancy dramatically.

3. Within-Warp Partitioning

FlashAttention-1 splits the K/V dimension across warps within a thread block, requiring a cross-warp reduction (via shared memory) to combine partial results. FlashAttention-2 instead splits the Q dimension across warps. Since each warp processes a different subset of Q rows against the same K/V block, no cross-warp communication is needed for the output — eliminating shared-memory synchronization overhead and improving warp-level efficiency.

FlashAttention-1

  • 25–40% of A100 peak TFLOPS
  • Outer loop: K/V blocks; Inner: Q blocks
  • Cross-warp reduction needed (shared memory sync)
  • Frequent rescaling in inner loop
  • ~1.5–2× faster than standard attention

FlashAttention-2

  • 50–73% of A100 peak TFLOPS
  • Outer loop: Q blocks; Inner: K/V blocks
  • No cross-warp sync — warp-level independence
  • Deferred rescaling; single final division
  • ~2× faster than FlashAttention-1

FlashAttention-3: FP8 & Hopper Asynchrony

FlashAttention-3 (Shah et al., 2024) targets the NVIDIA Hopper architecture (H100) and exploits three hardware features unavailable on Ampere:

1. Asynchronous Execution via WGMMA and TMA

Hopper introduces the Warp Group Matrix Multiply-Accumulate (WGMMA) instruction, which allows tensor core operations to execute asynchronously with respect to other instructions. FlashAttention-3 pipelines the computation: while the tensor cores compute Sij = QiKjT, the Tensor Memory Accelerator (TMA) simultaneously loads the next K/V tile from HBM into SRAM. This overlaps compute and memory access, hiding latency and achieving near-peak utilization.

2. FP8 Quantized Attention

H100 tensor cores support FP8 (E4M3 and E5M2 formats) with 2× the throughput of FP16 — up to ~990 TFLOPS. FlashAttention-3 introduces block-wise quantization: Q, K, V tiles are dynamically quantized to FP8 upon loading into SRAM, with per-tile scaling factors maintained in FP32. The incoherent processing technique applies random orthogonal transformations to mitigate outlier-induced quantization error. This achieves 1.2–1.5× additional speedup over FP16 FlashAttention-2 with minimal accuracy degradation.

3. Low-Precision with High-Precision Accumulation

A key concern with FP8 matmuls is accumulator precision. Hopper's WGMMA accumulates in FP32 by default, but the accumulation happens over chunks. FlashAttention-3 implements periodic FP32 promotion: after every few FP8 matmul chunks, intermediate results are promoted to FP32 and accumulated, preventing precision loss from cascading rounding errors. The paper shows this achieves within 1% perplexity of full FP16 training on GPT-scale models.

Performance summary: On H100 SXM5, FlashAttention-3 achieves up to 740 TFLOPS in FP16 (75% utilization) and approximately 1.2 PFLOPS in FP8 — making attention no longer the bottleneck in Transformer training for the first time.

Integration with Training Frameworks

FlashAttention is available as a drop-in replacement in all major training stacks. Below are the primary integration paths:

PyTorch Native (torch.nn.functional.scaled_dot_product_attention)

import torch from torch.nn.functional import scaled_dot_product_attention # PyTorch 2.0+ automatically selects FlashAttention when available with torch.backends.cuda.sdp_kernel( enable_flash=True, enable_math=False, enable_mem_efficient=False ): output = scaled_dot_product_attention( query, key, value, attn_mask=None, dropout_p=0.0, is_causal=True, scale=None ) # Constraints: head_dim must be ≤ 256, dtype must be fp16 or bf16 # Causal masking is fused — no need to pass an explicit mask tensor

HuggingFace Transformers

from transformers import AutoModelForCausalLM # Method 1: Set attn_implementation at model load time model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.1-70B", torch_dtype=torch.bfloat16, attn_implementation="flash_attention_2", # uses FlashAttention-2 device_map="auto" ) # Method 2: Use BetterTransformer (deprecated in favor of native SDPA) # model = model.to_bettertransformer() # For FlashAttention-2, ensure flash-attn is installed: # pip install flash-attn --no-build-isolation

Direct flash-attn Library

from flash_attn import flash_attn_func, flash_attn_varlen_func # Standard fixed-length attention output = flash_attn_func( q, k, v, # each [batch, seqlen, nheads, headdim] dropout_p=0.0, softmax_scale=None, # defaults to 1/sqrt(headdim) causal=True, window_size=(-1, -1), # (-1,-1) = no sliding window return_attn_probs=False ) # Variable-length / packed sequences (padding-free) output_varlen = flash_attn_varlen_func( q_unpad, k_unpad, v_unpad, # [total_tokens, nheads, headdim] cu_seqlens_q, cu_seqlens_k, # cumulative sequence lengths max_seqlen_q, max_seqlen_k, causal=True )
Installation note: The flash-attn package requires CUDA ≥ 11.6, PyTorch ≥ 1.12, and an NVIDIA GPU with sm_80 or higher (A100, H100, L40, RTX 3090/4090). The --no-build-isolation flag is required because the package uses PyTorch's CUDA extension build system. Compilation can take 10–30 minutes. Pre-built wheels are available for common PyTorch/CUDA combinations.

Memory Savings Analysis

The memory savings from FlashAttention are most impactful for long-context training. The following analysis assumes a single attention layer with fp16 precision, batch_size × num_heads = 32, and head dimension d = 128.

Sequence Length Standard Attn Memory FlashAttention Memory Reduction
2,048 256 MB (attn matrix) 16 MB (O + stats) 16×
8,192 4 GB 64 MB 64×
32,768 64 GB 256 MB 256×
131,072 (128k) 1 TB (infeasible) 1 GB 1024×

The standard attention memory is B×H × N² × 2 bytes where B×H = 32. FlashAttention stores only the output tensor O (B×H × N × d × 2 bytes) plus row-wise softmax statistics (2 × B×H × N × 4 bytes for m and ℓ in fp32). The reduction factor grows linearly with N/d — meaning FlashAttention becomes more advantageous as context length increases.

Wall-Clock Speedup

Memory savings translate directly to speed because attention is bandwidth-bound. Empirical benchmarks on A100-80GB show:

  • N = 2k: FlashAttention-2 is ~2× faster than standard, ~4× faster than PyTorch's memory-efficient implementation
  • N = 8k: FlashAttention-2 is ~3× faster (standard attention OOMs on smaller GPUs at this length)
  • N = 16k: FlashAttention-2 is the only exact-attention option that fits in 80 GB; ~3.5× faster than standard on multi-GPU setups

End-to-End Training Impact

In a full GPT-3 175B training configuration, attention accounts for approximately 40% of total HBM usage (including forward activations stored for the backward pass). FlashAttention reduces this to under 5%, freeing memory for larger batch sizes or longer sequences. The practical impact:

  • GPT-3 175B training with 2k context: activation memory drops from ~120 GB to ~68 GB per GPU, enabling training on 8×A100-80GB without activation checkpointing
  • LLaMA-2 70B with 4k context: FlashAttention-2 enables 30% larger micro-batch size, improving training throughput by ~20% (due to better GPU utilization)
  • Long-context fine-tuning (32k–128k): only feasible with FlashAttention; standard attention would require model-parallel splitting of the attention layer itself
Backward pass savings: Standard attention must store the full N × N attention probability matrix P for the backward pass. FlashAttention instead stores only O, m, and ℓ (all O(N)), then recomputes S and P tile-by-tile during backpropagation. This recomputation adds ~25% extra FLOPs but saves massive memory — a favorable trade because the forward+backward pass remains bandwidth-bound. With activation checkpointing, the memory savings compound further.
