Mixture-of-Experts Architecture

Mixture-of-Experts (MoE) models replace the monolithic feed-forward network (FFN) in each Transformer block with a set of N parallel expert sub-networks, of which only a small subset k are activated for any given token. This conditional computation decouples total parameter count from per-token FLOPs: a model with 1.6 T parameters can consume roughly the same compute as a 100 B dense model at inference, because each token activates only 2 out of 64 experts.

Concretely, a standard Transformer FFN with hidden dimension dmodel and intermediate dimension dff has 2 × dmodel × dff parameters. An MoE layer replaces this single FFN with N independent copies (experts), each of identical architecture, plus a lightweight gating network of size dmodel × N. The total parameters grow to N × 2 × dmodel × dff + dmodel × N, but active parameters per token remain k × 2 × dmodel × dff.

Key Insight: MoE achieves sub-linear scaling of compute with respect to parameters. Switch Transformer (2021) demonstrated that a 1.6 T parameter MoE model trains with comparable wall-clock cost to a T5-XXL (11B dense), achieving 4× faster pre-training speedup at fixed compute budgets.

Every other Transformer layer is typically replaced with an MoE layer (alternating dense and MoE), though some architectures like Mixtral 8×7B replace every FFN. The attention layers remain shared across all tokens; sparsity is applied only to the FFN component. This design keeps the model's ability to attend over all tokens while allowing specialization in the expert FFN sub-networks.

# MoE layer forward pass (simplified) class MoELayer(nn.Module): def __init__(self, d_model, d_ff, num_experts, top_k): super().__init__() self.experts = nn.ModuleList([ FeedForward(d_model, d_ff) for _ in range(num_experts) ]) self.gate = nn.Linear(d_model, num_experts) self.top_k = top_k self.num_experts = num_experts def forward(self, x): # x shape: (batch, seq_len, d_model) gate_logits = self.gate(x) # (B, S, num_experts) weights, indices = top_k_gating(gate_logits, self.top_k) output = dispatch_to_experts(x, self.experts, weights, indices) return output

Expert Routing & Gating

The gating network (router) determines which experts process each token. For a token representation x ∈ ℝd, the router computes logits h(x) = Wg · x where Wg ∈ ℝN×d. The top-k experts are then selected, and their gating weights are renormalized via softmax over the selected subset.

The gating function can be formalized as:

G(x) = Softmax(TopK(h(x), k))

where TopK keeps the top-k values and sets the rest to −∞ before the softmax, producing a sparse probability distribution. The final MoE output for a token is:

y = ∑i ∈ TopK G(x)i · Ei(x)

Warning — Routing Collapse: Without careful regularization, the router can converge to sending all tokens to a single expert. This routing collapse effectively wastes (N−1) experts' parameters and degrades the model to a dense network with extra dead weight. Auxiliary losses (discussed below) are essential to prevent this.
# Top-k gating with softmax normalization def top_k_gating(logits, k): # logits: (batch * seq_len, num_experts) top_values, top_indices = torch.topk(logits, k, dim=-1) # Mask out non-selected experts with -inf mask = torch.full_like(logits, float('-inf')) mask.scatter_(1, top_indices, top_values) # Softmax over all experts (non-selected become 0) gates = torch.softmax(mask, dim=-1) return gates, top_indices # Capacity factor determines buffer size per expert def compute_capacity(num_tokens, num_experts, top_k, capacity_factor): # Average tokens per expert under uniform distribution tokens_per_expert = (num_tokens * top_k) / num_experts capacity = int(tokens_per_expert * capacity_factor) return capacity

The capacity factor (CF) controls the buffer allocated to each expert. With CF = 1.0, each expert can process exactly its fair share of tokens under a perfectly uniform distribution. In practice, CF is set between 1.0 and 1.5 to accommodate routing imbalance. Tokens that exceed an expert's capacity are dropped and passed through a residual connection unchanged. Higher CF values waste compute on padding; lower values risk dropping too many tokens.

Load Balancing & Auxiliary Losses

Without explicit regularization, the router tends to converge on a small subset of experts, creating a positive feedback loop: popular experts receive more gradient signal, improve faster, and attract even more tokens. The remaining experts stagnate. Load balancing losses add a penalty that encourages uniform token distribution.

Switch Transformer Load Balancing Loss

Given N experts and a batch of T tokens, define fi as the fraction of tokens routed to expert i, and Pi as the mean router probability for expert i across all tokens. The auxiliary loss is:

Laux = α · N · ∑i=1N fi · Pi

This product is minimized when fi = 1/N for all experts. The coefficient α is typically set to 0.01 so the auxiliary loss does not overwhelm the primary language modeling objective.

Z-Loss for Router Stability

ST-MoE introduced z-loss to penalize large router logits, preventing the gating network from becoming overly confident, which causes training instability via exponential growth inside the softmax:

Lz = β · (1/T) · ∑t=1T (log ∑i=1N ehi(xt))²

# Load balancing loss (Switch Transformer formulation) def load_balancing_loss(gate_probs, expert_indices, num_experts, alpha=0.01): # gate_probs: (num_tokens, num_experts) — softmax probabilities # expert_indices: (num_tokens, top_k) — selected expert ids num_tokens = gate_probs.shape[0] # f_i: fraction of tokens dispatched to each expert one_hot = torch.zeros_like(gate_probs) one_hot.scatter_(1, expert_indices, 1.0) f = one_hot.mean(dim=0) # (num_experts,) # P_i: mean router probability for each expert P = gate_probs.mean(dim=0) # (num_experts,) # Auxiliary loss: encourages f_i ≈ P_i ≈ 1/N loss = alpha * num_experts * torch.sum(f * P) return loss # Z-loss for router logit stability (ST-MoE) def z_loss(router_logits, beta=0.001): # router_logits: (num_tokens, num_experts) log_z = torch.logsumexp(router_logits, dim=-1) # (num_tokens,) loss = beta * torch.mean(log_z ** 2) return loss

Switch Transformer vs GShard Approaches

Switch Transformer

  • Top-1 routing — each token goes to exactly one expert
  • Capacity factor CF = 1.0–1.5; dropped tokens use residual skip
  • Simplified auxiliary loss: α · N · ∑ fi · Pi
  • Selective precision: router in FP32, experts in BF16
  • Achieved 7× pre-training speedup over T5-Base
  • Scales to 1.6 T parameters on 2048 TPU v3 cores

GShard

  • Top-2 routing — each token dispatched to two experts
  • Random routing for second expert weighted by gate probability
  • Both importance loss and load loss used jointly
  • Group-level balancing across expert groups
  • Scaled to 600 B parameters on 2048 TPU v3 cores
  • Demonstrated MoE effectiveness for machine translation
Practical Tip: In Mixtral 8×7B, each token is routed to 2 of 8 experts, yielding active parameters of ~13B from a total of ~47B. The auxiliary loss coefficient α is set low enough (0.001–0.01) that the main cross-entropy loss still dominates, but high enough to prevent any single expert from receiving more than ~20% of tokens.

All-to-All Communication

Expert parallelism distributes the N experts across P GPUs, with each GPU hosting N/P experts. Because any token can be routed to any expert, tokens must be reshuffled between GPUs after routing decisions are made. This requires an all-to-all collective operation — conceptually a distributed transpose where GPU i sends a distinct chunk of data to every other GPU j.

The all-to-all pattern occurs twice per MoE layer:

  1. Dispatch: After the router decides which tokens go to which experts, tokens are sent from their current GPU to the GPU hosting the target expert.
  2. Combine: After expert computation, the results are sent back to the original GPU to be recombined with the gating weights.
Bandwidth Bottleneck: For a model with sequence length S, batch size B, hidden dimension d, across P GPUs, each all-to-all transfers approximately B × S × d × 2 bytes (in BF16) per direction. With B=2, S=4096, d=8192, P=64, that's ~128 MB per GPU per direction per MoE layer. With 32 MoE layers, the total all-to-all volume reaches ~8 GB per GPU per forward pass.
# All-to-All dispatch and combine pattern import torch.distributed as dist def expert_parallel_forward(tokens, gate_weights, expert_indices, experts, ep_group): # Step 1: Determine which tokens go to which GPU # Arrange tokens into per-destination buffers send_buffers = prepare_dispatch_buffers(tokens, expert_indices, ep_group) # Step 2: All-to-All DISPATCH — send tokens to expert-hosting GPUs recv_buffers = torch.empty_like(send_buffers) dist.all_to_all_single(recv_buffers, send_buffers, group=ep_group) # Step 3: Local expert computation on received tokens local_results = [] for i, expert in enumerate(experts): expert_tokens = recv_buffers[i] local_results.append(expert(expert_tokens)) result_buffer = torch.stack(local_results) # Step 4: All-to-All COMBINE — send results back to original GPUs combined = torch.empty_like(result_buffer) dist.all_to_all_single(combined, result_buffer, group=ep_group) # Step 5: Weighted sum using gating weights output = weighted_combine(combined, gate_weights) return output

Modern implementations overlap communication with computation by pipelining: while GPU 0 sends tokens for expert 2 on GPU 1, it simultaneously computes expert 0 on locally-routed tokens. This computation-communication overlap reduces the effective latency overhead, though the total bandwidth demand remains unchanged.

Expert Parallelism Implementation

Expert Parallelism (EP) is a parallelism dimension orthogonal to Data Parallelism (DP) and Tensor Parallelism (TP). In practice, these strategies are combined. Consider a cluster of 64 GPUs training a model with 64 experts. One configuration:

  • EP = 8: experts are partitioned across 8 GPUs, each hosting 8 experts
  • TP = 2: each expert's parameters are split across 2 GPUs
  • DP = 4: the entire configuration is replicated 4 times (64 / 8 / 2 = 4)

The total GPU count must satisfy: DP × EP × TP = Total GPUs. Choosing the right decomposition depends on expert size, interconnect bandwidth, and memory constraints.

# DeepSpeed-MoE style parallelism configuration # Configuring EP + DP + TP for a 64-expert model on 64 GPUs config = { # Model configuration "num_experts": 64, "d_model": 8192, "d_ff": 22016, "num_layers": 80, "moe_layer_interval": 2, # MoE every other layer "top_k": 2, # Parallelism configuration "expert_parallel_size": 8, # 64 experts / 8 = 8 experts per GPU "tensor_parallel_size": 2, # Each expert split across 2 GPUs "data_parallel_size": 4, # 64 / (8 * 2) = 4-way DP # Expert-specific settings "capacity_factor_train": 1.25, "capacity_factor_eval": 2.0, # Higher at eval to avoid dropping "min_capacity": 4, "drop_tokens": True, "use_residual": True, # Skip connection for dropped tokens } # Memory per GPU calculation: # Expert params per GPU: 8 experts × 2 × d_model × d_ff / TP # = 8 × 2 × 8192 × 22016 / 2 = 1.44B params ≈ 2.88 GB (BF16) # Shared layers (attention etc.) are replicated across DP # Total per-GPU model memory ≈ 5-8 GB + optimizer states
DeepSpeed-MoE Hierarchy: DeepSpeed introduces a two-level hierarchy: intra-node EP uses fast NVLink/NVSwitch for all-to-all within a node, while inter-node DP uses slower InfiniBand for gradient synchronization. This topology-aware placement minimizes cross-node communication for the latency-sensitive all-to-all operations.

Process Group Setup

# Setting up EP + DP process groups (PyTorch distributed) import torch.distributed as dist def setup_moe_process_groups(world_size, ep_size, tp_size): # Validate decomposition dp_size = world_size // (ep_size * tp_size) assert dp_size * ep_size * tp_size == world_size rank = dist.get_rank() # EP group: ranks that share the expert-parallel dimension # Ranks [0..ep_size-1], [ep_size..2*ep_size-1], etc. ep_groups = [] for dp_idx in range(dp_size): for tp_idx in range(tp_size): ranks = [dp_idx * ep_size * tp_size + ep_idx * tp_size + tp_idx for ep_idx in range(ep_size)] group = dist.new_group(ranks) if rank in ranks: ep_group = group ep_groups.append(group) # DP group: ranks that replicate the same experts dp_groups = [] for ep_idx in range(ep_size): for tp_idx in range(tp_size): ranks = [dp_idx * ep_size * tp_size + ep_idx * tp_size + tp_idx for dp_idx in range(dp_size)] group = dist.new_group(ranks) if rank in ranks: dp_group = group dp_groups.append(group) return ep_group, dp_group

Scaling MoE Models

Scaling MoE models to trillions of parameters introduces unique challenges beyond those of dense models. While MoE provides parameter efficiency through sparse activation, the total model footprint still determines memory requirements, checkpoint sizes, and communication overhead.

Memory Footprint Analysis

Consider a Mixtral-scale model: 8 experts per MoE layer, dmodel=4096, dff=14336, 32 layers (all MoE). Per-expert parameters: 2 × 4096 × 14336 = 117.4M. Total expert parameters: 32 layers × 8 experts × 117.4M = 30.1B. In BF16, that's ~60 GB just for expert weights. With Adam optimizer states (2× in FP32), the optimizer footprint alone reaches ~240 GB, necessitating distribution across many GPUs even before considering activations.

Checkpoint Size Explosion: A 1.6T parameter MoE model's checkpoint is ~3.2 TB in BF16. Saving checkpoints to distributed storage at full training speed requires careful async I/O and sharded checkpointing (e.g., PyTorch FSDP's sharded state dict or DeepSpeed's ZeRO checkpoint format). A naive checkpoint save at this scale can stall training for 10+ minutes.

Training Stability Strategies

  • Router z-loss: Penalizes large logits to prevent exponential blowup in softmax, critical for BF16 training where the dynamic range is limited.
  • Expert dropout: During training, randomly bypass MoE layers with probability p, forcing the model to be robust to expert unavailability.
  • Jitter noise: Add multiplicative noise to router inputs (x × (1 + ε), ε ~ U(-0.01, 0.01)) to break symmetry early in training and encourage exploration of all experts.
  • Gradient clipping per expert group: Apply separate gradient norms to expert parameters vs. shared parameters, as expert gradients can be spiky due to sparse activation patterns.

Inference Considerations

At inference, MoE models present a unique challenge: even though only k experts are active per token, all expert weights must be resident in memory (or quickly loadable) because routing decisions are input-dependent. This means MoE models have the memory footprint of a dense model with N × FFN parameters, but the FLOP cost of a model with k × FFN parameters.

# Memory estimation for MoE training def estimate_moe_memory( num_layers, num_experts, d_model, d_ff, top_k, seq_len, batch_size, precision_bytes=2 ): # Expert parameters per layer expert_params_per_layer = num_experts * 2 * d_model * d_ff # Shared attention parameters (approximate) attn_params_per_layer = 4 * d_model * d_model # Q, K, V, O projections # Total model parameters total_params = num_layers * (expert_params_per_layer + attn_params_per_layer) # Memory components (in bytes) model_memory = total_params * precision_bytes # Adam optimizer: 2x FP32 copies (momentum + variance) optimizer_memory = total_params * 4 * 2 # Gradients in training precision gradient_memory = total_params * precision_bytes # Activation memory (rough estimate for one MoE layer) activation_per_layer = batch_size * seq_len * d_model * precision_bytes * 12 total_gb = (model_memory + optimizer_memory + gradient_memory) / (1024**3) activation_gb = (activation_per_layer * num_layers) / (1024**3) return { "total_params": total_params, "active_params_per_token": num_layers * (top_k * 2 * d_model * d_ff + attn_params_per_layer), "model_memory_gb": model_memory / (1024**3), "optimizer_memory_gb": optimizer_memory / (1024**3), "gradient_memory_gb": gradient_memory / (1024**3), "activation_memory_gb": activation_gb, "total_training_memory_gb": total_gb + activation_gb, } # Example: Mixtral-scale model stats = estimate_moe_memory( num_layers=32, num_experts=8, d_model=4096, d_ff=14336, top_k=2, seq_len=4096, batch_size=2 ) # total_params ≈ 46.7B, active_params ≈ 12.9B # total_training_memory ≈ 430 GB across all GPUs
The MoE Scaling Law: Empirical results from Google's ST-MoE and Meta's research suggest that MoE models follow a modified scaling law where performance scales with active parameters rather than total parameters, but with a favorable constant — an MoE model with k active parameters out of N total typically outperforms a dense model of k parameters by a significant margin, though not as much as a dense model of N parameters. The sweet spot appears to be 8–64 experts with top-2 routing, balancing parameter efficiency against communication and load balancing overhead.
