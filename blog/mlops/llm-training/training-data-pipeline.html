LLM Data Pipeline: Dedup, Filter, Mix

MLOps Series LLM Training Infrastructure

Training a frontier LLM on trillions of tokens requires a data pipeline that is as carefully engineered as the model itself. Research from Chinchilla, LLaMA, and Dolma consistently demonstrates that data quality dominates model quality—a 7B model trained on well-curated data outperforms a 70B model trained on raw web scrapes. This post dissects the end-to-end pipeline from raw web crawl to tokenized training shards, covering the systems engineering and algorithmic details behind each stage.

The Trillion-Token Pipeline Challenge

Modern frontier models are pretrained on 2–15 trillion tokens. LLaMA 3 used 15T tokens; Llama 2 used 2T. A single Common Crawl snapshot (e.g., CC-2024-10) contains roughly 3.5 billion web pages compressed into ~250 TB of WARC files. After extraction, deduplication, and quality filtering, a typical crawl yields only 200–400 billion high-quality tokens—meaning you need to process 5–40 snapshots and merge with curated datasets (books, academic papers, code) to reach training-scale volumes.

The pipeline has a harsh funnel: approximately 60–85% of raw crawl data is discarded through deduplication and quality filtering. This means the pipeline must process 5–10× the final training volume. At LLaMA 3 scale (15T tokens), the raw input exceeds 1 petabyte of compressed web data. Running this pipeline requires a carefully orchestrated system of distributed jobs, typically on a Spark or Ray cluster with hundreds of nodes.

Key insight: The Chinchilla scaling law (Hoffmann et al., 2022) showed that compute-optimal training requires ~20 tokens per parameter. For a 70B model, that is 1.4T tokens. But over-training on more data (LLaMA approach) consistently improves downstream performance, pushing the frontier to 100–200 tokens per parameter.

The pipeline stages form a DAG with clear data dependencies: extraction depends on crawl, dedup depends on extraction, filtering depends on dedup, mixing depends on filtering, tokenization depends on mixing. Each stage has distinct computational characteristics—extraction is I/O-bound, dedup is memory-bound (hash tables), filtering is CPU-bound (n-gram models), and tokenization is embarrassingly parallel. Understanding these profiles is critical for capacity planning and cost optimization.

Web Crawling and Extraction

Most LLM pipelines start from Common Crawl rather than running their own web crawlers. Common Crawl provides monthly snapshots in WARC (Web ARChive) format—each snapshot is split into ~90,000 WARC segment files averaging 1 GB compressed. Processing a full snapshot means reading ~90 TB of compressed data, decompressing each record, and extracting clean text from raw HTML.

The extraction stage performs three core operations: (1) HTML parsing to isolate main content from boilerplate (navigation, ads, footers), (2) language identification to tag each document with its primary language, and (3) metadata extraction to capture URL, timestamp, content length, and other signals used downstream for filtering.

# WARC processing pipeline using warcio + trafilatura import gzip from warcio.archiveiterator import ArchiveIterator from trafilatura import extract import fasttext lid_model = fasttext.load_model(# fastText language ID model "lid.176.bin" ) def process_warc_segment(warc_path, min_words=50): # Process a single WARC segment file (~1 GB compressed) results = [] with open(warc_path, "rb") as fh: for record in ArchiveIterator(fh): if record.rec_type != "response": continue url = record.rec_headers.get_header("WARC-Target-URI") html = record.content_stream().read().decode("utf-8", errors="ignore") # Extract main content, stripping boilerplate text = extract(html, include_comments=False, include_tables=True, deduplicate=True) if text is None: continue # Language identification first_line = text.split("\n")[0].strip() lang_pred = lid_model.predict(first_line, k=1) lang = lang_pred[0][0].replace("__label__", "") lang_conf = lang_pred[1][0] word_count = len(text.split()) if word_count < min_words or lang_conf < 0.65: continue results.append({ "url": url, "text": text, "lang": lang, "lang_conf": lang_conf, "word_count": word_count, }) return results
Performance note: Trafilatura processes ~50–100 documents/sec per CPU core. For a full Common Crawl snapshot (3.5B pages), a single core would take ~1.1 years. In practice, this stage is distributed across 200–500 nodes using Spark or Ray, with each node handling ~200 WARC segments. Total wall-clock time: 4–8 hours on a well-provisioned cluster.

An alternative extraction approach uses jusText, which classifies HTML blocks as "good," "near-good," or "bad" based on text density, link density, and position heuristics. jusText is faster than trafilatura (~200 docs/sec/core) but produces slightly noisier output. The RefinedWeb pipeline (Penedo et al., 2023) combines both: trafilatura for initial extraction, followed by custom heuristic post-processing to remove residual boilerplate fragments and short low-quality passages.

Deduplication Stages

Deduplication is the single most impactful stage in the data pipeline. The C4 dataset showed that ~30% of Common Crawl documents are exact duplicates; adding near-duplicate detection increases removal to 40–60%. Training on duplicated data wastes compute and has been shown to degrade model performance—Lee et al. (2022) demonstrated that dedup improves perplexity by 2–5% at equivalent training FLOPs. The dedup stage operates in three layers, each catching a different class of redundancy.

Layer 1: URL-Level Dedup

The simplest and cheapest dedup pass: normalize URLs (strip tracking parameters, canonicalize scheme/host/path) and keep only the first occurrence. This catches mirror sites, CDN duplicates, and pagination variants. A Bloom filter with 10 billion entries (1% FPR) requires ~11.5 GB of memory—easily fits on a single node for URL-level dedup across multiple crawl snapshots.

Layer 2: Exact Hash Dedup

Compute a SHA-256 hash of each document’s normalized text (lowercased, whitespace-collapsed). Store hashes in a distributed hash set. This catches documents with identical content but different URLs—common for content syndication, RSS feeds, and copy-paste news articles. A 64-bit truncated hash over 5 billion documents requires ~40 GB of memory, manageable with a sharded hash map across 4–8 nodes.

Layer 3: Fuzzy Dedup with MinHash LSH

The most computationally expensive dedup stage. Documents are shingled into overlapping n-grams (typically 5-grams or 13-grams), then each shingle set is compressed into a MinHash signature of 128–256 hash values. Signatures are grouped using Locality-Sensitive Hashing (LSH) with band parameters (b, r) tuned to achieve a desired Jaccard similarity threshold (typically 0.8). Documents that land in the same LSH bucket are candidate duplicates.

# MinHash LSH deduplication with configurable thresholds import hashlib import struct from collections import defaultdict class MinHashLSH: def __init__(self, num_hashes=128, bands=16, rows=8, ngram_size=5): self.num_hashes = num_hashes self.bands = bands self.rows = rows # rows per band; bands * rows = num_hashes self.ngram_size = ngram_size self.buckets = defaultdict(lambda: defaultdict(list)) def _shingle(self, text): # Generate character n-grams from text tokens = text.lower().split() return set( " ".join(tokens[i:i + self.ngram_size]) for i in range(len(tokens) - self.ngram_size + 1) ) def _minhash(self, shingles): # Compute MinHash signature using SHA-1 variants signature = [] for i in range(self.num_hashes): min_val = float("inf") for s in shingles: h = hashlib.sha1( f"{i}:{s}".encode() ).digest()[:8] val = struct.unpack("<Q", h)[0] if val < min_val: min_val = val signature.append(min_val) return signature def insert(self, doc_id, text): shingles = self._shingle(text) if not shingles: return sig = self._minhash(shingles) for band_idx in range(self.bands): start = band_idx * self.rows band = tuple(sig[start:start + self.rows]) band_hash = hash(band) self.buckets[band_idx][band_hash].append(doc_id) def find_duplicates(self): # Return sets of candidate duplicate document IDs dup_sets = [] for band_idx in range(self.bands): for bucket_key, doc_ids in self.buckets[band_idx].items(): if len(doc_ids) > 1: dup_sets.append(set(doc_ids)) return dup_sets
LSH tuning: With 128 hashes, 16 bands, and 8 rows per band, the probability that two documents with Jaccard similarity 0.8 are detected as duplicates is 1 − (1 − 0.88)16 ≈ 0.98. The probability of a false positive at Jaccard 0.4 is only ~0.006. Adjusting band/row counts lets you trade precision for recall—critical when deduplicating trillions of tokens.

At scale, the MinHash LSH stage is the bottleneck. For 5 billion documents with 128-hash signatures, storing all signatures requires ~5B × 128 × 8 bytes ≈ 5.1 TB of memory. The Dolma pipeline addresses this with a sharded MapReduce approach: partition documents by a coarse hash into 256 shards, run LSH within each shard, then perform a lightweight cross-shard reconciliation pass. This reduces peak memory to ~20 GB per shard while maintaining dedup quality.

Quality Filtering

After deduplication, the remaining documents are still highly heterogeneous in quality. Quality filtering aims to retain text that is linguistically coherent, factually substantive, and stylistically similar to curated reference corpora. Three families of filters are used in practice, often in cascade.

Heuristic Filters

Speed: ~10K docs/sec/core

Method: Rule-based thresholds on measurable document properties

  • Mean line length > 20 chars
  • Symbol-to-word ratio < 0.1
  • Fraction of lines ending with ellipsis < 0.3
  • Fraction of duplicate lines < 0.3
  • Top 2-gram / 3-gram / 4-gram frequency ratios below thresholds
  • Alphabet-to-total-character ratio > 0.8

Used by: C4, RefinedWeb, Dolma

Perplexity Filters (KenLM)

Speed: ~5K docs/sec/core

Method: Score documents with a KenLM 5-gram model trained on Wikipedia. Remove documents with perplexity above a threshold (typically 250–500 for English).

  • Low perplexity = text resembles Wikipedia prose
  • High perplexity = garbled text, SEO spam, boilerplate
  • Must normalize by document length
  • Language-specific models required

Used by: CCNet (Meta), LLaMA, BLOOM

Classifier-Based (fastText)

Speed: ~8K docs/sec/core

Method: Train a binary fastText classifier on positive examples (Wikipedia, curated books) vs. negative examples (random web text). Score each document; keep those above a threshold (e.g., score > 0.5).

  • More nuanced than perplexity alone
  • Can capture topical relevance
  • Risk of bias toward encyclopedic style
  • Requires careful negative sampling

Used by: GPT-3, Falcon, RedPajama

Toxicity / Safety Filters

Speed: ~2K docs/sec/core

Method: Use classifiers (Jigsaw, Perspective API, or custom) to detect and remove toxic, NSFW, or personally identifiable content. Blocklist filtering on URLs known to host harmful content.

  • URL blocklists: UT1, Shalla, custom lists
  • Document-level toxicity score thresholds
  • PII regex patterns (SSN, email, phone)
  • Trade-off: aggressive filtering reduces diversity

Used by: All major pipelines

# Quality filter pipeline combining heuristic + perplexity + classifier import kenlm import fasttext import re class QualityFilter: def __init__(self, kenlm_path, clf_path, ppl_threshold=350, clf_threshold=0.45): self.lm = kenlm.Model(kenlm_path) self.clf = fasttext.load_model(clf_path) self.ppl_threshold = ppl_threshold self.clf_threshold = clf_threshold def _heuristic_pass(self, text): # Fast rule-based rejection lines = text.strip().split("\n") if len(lines) < 3: return False words = text.split() if len(words) < 50: return False avg_line_len = sum(len(l) for l in lines) / len(lines) if avg_line_len < 20: return False alpha_ratio = sum(c.isalpha() for c in text) / max(len(text), 1) if alpha_ratio < 0.7: return False # Check for excessive repetition unique_lines = set(lines) dup_ratio = 1.0 - len(unique_lines) / len(lines) if dup_ratio > 0.3: return False return True def _perplexity_score(self, text): # KenLM perplexity (lower = more Wikipedia-like) log_score = self.lm.score(text, bos=True, eos=True) num_words = len(text.split()) + 1 return 10 ** (-log_score / num_words) def _classifier_score(self, text): # fastText quality classifier probability first_512 = " ".join(text.split()[:512]) pred = self.clf.predict(first_512) if pred[0][0] == "__label__hq": return pred[1][0] return 1.0 - pred[1][0] def __call__(self, doc): # Cascaded filter: cheapest first if not self._heuristic_pass(doc["text"]): return None ppl = self._perplexity_score(doc["text"]) if ppl > self.ppl_threshold: return None score = self._classifier_score(doc["text"]) if score < self.clf_threshold: return None doc["quality_score"] = score doc["perplexity"] = ppl return doc
Cascading order matters: Heuristic filters run at 10K docs/sec and reject ~30% of input. By running them first, the more expensive KenLM (5K docs/sec) and classifier (8K docs/sec) stages process only 70% of the data. This ordering reduces total filter-stage compute by approximately 25%.

Data Mixing and Tokenization

After filtering, the curated data is a collection of per-domain corpora: web text, books, academic papers, code, conversation data, and curated datasets (Wikipedia, StackExchange, etc.). The mixing ratio—how these domains are blended during training—has a dramatic impact on downstream task performance. LLaMA 2 used approximately: web 67%, code 8.4%, Wikipedia 4.6%, books 4.5%, academic 2.5%, misc 13%. Finding optimal ratios is an active area of research; DoReMi (Xie et al., 2023) proposes learning mixing weights via a proxy model.

# Domain mixing configuration and tokenized shard generation import json import numpy as np from pathlib import Path from sentencepiece import SentencePieceProcessor DOMAIN_CONFIG = { # domain: (weight, upsample_factor, path) "web": (0.67, 1.0, "data/clean/web/"), "code": (0.084, 2.0, "data/clean/code/"), "wikipedia":(0.046, 3.0, "data/clean/wiki/"), "books": (0.045, 2.5, "data/clean/books/"), "academic": (0.025, 2.0, "data/clean/arxiv/"), "stackexch":(0.02, 3.0, "data/clean/stackexchange/"), "misc": (0.115, 1.0, "data/clean/misc/"), } class DataMixer: def __init__(self, config, tokenizer_path, seq_len=4096, shard_size=1_000_000): self.config = config self.sp = SentencePieceProcessor() self.sp.load(tokenizer_path) self.seq_len = seq_len self.shard_size = shard_size # tokens per shard def _stream_domain(self, domain_path, upsample): # Stream documents from domain, repeating for upsampling files = sorted(Path(domain_path).glob("*.jsonl")) for epoch in range(int(np.ceil(upsample))): for f in files: with open(f) as fh: for line in fh: doc = json.loads(line) yield doc["text"] def _tokenize_and_pack(self, text_stream): # Concatenate tokenized docs with EOS, pack into fixed-length seqs buffer = [] eos_id = self.sp.eos_id() for text in text_stream: tokens = self.sp.encode(text) buffer.extend(tokens + [eos_id]) while len(buffer) >= self.seq_len: yield buffer[:self.seq_len] buffer = buffer[self.seq_len:] def generate_shards(self, output_dir): # Interleave domains by weight and write token shards streams = {} for domain, (weight, upsample, path) in self.config.items(): streams[domain] = self._tokenize_and_pack( self._stream_domain(path, upsample) ) shard_idx = 0 shard_buf = [] domains = list(self.config.keys()) weights = np.array([self.config[d][0] for d in domains]) weights = weights / weights.sum() while True: domain = np.random.choice(domains, p=weights) try: seq = next(streams[domain]) except StopIteration: break shard_buf.extend(seq) if len(shard_buf) >= self.shard_size: self._write_shard(output_dir, shard_idx, shard_buf) shard_idx += 1 shard_buf = []

Tokenization itself must be carefully designed. Modern LLMs use Byte-Pair Encoding (BPE) with vocabulary sizes of 32K–128K tokens. SentencePiece with BPE is the standard choice (used by LLaMA, Mistral, Falcon). Key decisions include: whether to treat whitespace as a special token, whether to split digits individually (improves arithmetic), and whether to use byte-level fallback for unknown characters. Tokenizer training is done on a representative sample (~10B characters) of the final training mix to ensure vocabulary coverage matches the actual data distribution.

Packing efficiency: Naive document-level batching wastes ~15–25% of tokens on padding. Token packing—concatenating documents with EOS separators and slicing into fixed-length sequences—achieves near-100% utilization. The LLaMA approach packs across document boundaries; some pipelines (e.g., UL2) respect document boundaries by using attention masks, trading ~10% throughput for cleaner attention patterns.

Streaming and Data Loading

Once tokenized shards are produced, the final challenge is feeding data to the GPU cluster fast enough. A training run on 2,048 H100s at ~50% MFU processes roughly 500K tokens per second per GPU, or 1 billion tokens per second cluster-wide. This means the data pipeline must sustain 1B tokens/sec of read throughput, equivalent to ~4 GB/s of tokenized data. Stalling the data pipeline even briefly causes GPU idle time costing thousands of dollars per hour.

The dominant approach is WebDataset (or a variant like Mosaic StreamingDataset), which stores tokenized sequences as tar archives on object storage (S3, GCS). Each worker node reads a subset of tar shards, with shard assignment managed to avoid overlap. Prefetching runs 2–4 shards ahead of consumption, and data is memory-mapped or streamed through a ring buffer to overlap I/O with compute.

# Efficient streaming data loader with prefetch pipeline import torch from torch.utils.data import IterableDataset, DataLoader import numpy as np from pathlib import Path class ShardedTokenDataset(IterableDataset): def __init__(self, shard_dir, seq_len=4096, seed=42): self.shards = sorted(Path(shard_dir).glob("*.bin")) self.seq_len = seq_len self.seed = seed def __iter__(self): worker_info = torch.utils.data.get_worker_info() if worker_info is None: shard_list = self.shards else: # Partition shards across data-loader workers per_worker = len(self.shards) // worker_info.num_workers wid = worker_info.id shard_list = self.shards[wid * per_worker:(wid + 1) * per_worker] rng = np.random.default_rng(self.seed) rng.shuffle(shard_list) for shard_path in shard_list: # Memory-map the shard for zero-copy reads tokens = np.memmap(shard_path, dtype=np.uint16, mode="r") n_seqs = len(tokens) // self.seq_len for i in range(n_seqs): start = i * self.seq_len chunk = tokens[start:start + self.seq_len] x = torch.from_numpy(chunk.astype(np.int64)) yield x[:-1], x[1:] # input, target def build_dataloader(shard_dir, batch_size=32, num_workers=8): ds = ShardedTokenDataset(shard_dir) return DataLoader( ds, batch_size=batch_size, num_workers=num_workers, pin_memory=True, prefetch_factor=4, # prefetch 4 batches per worker persistent_workers=True, )
Curriculum learning: Some pipelines schedule data quality over training. Early training uses noisier, more diverse data; later stages shift toward higher-quality sources. This can be implemented by maintaining separate shard pools and adjusting sampling weights at checkpoint boundaries. MiniCPM and Yi report 1–3% perplexity improvements from simple two-phase curricula.

Deterministic reproducibility is critical for debugging and ablations. The data loader must produce exactly the same token sequence given the same random seed, even when restarting from a checkpoint mid-epoch. This is achieved by recording the (shard_index, offset_within_shard) tuple at each checkpoint, allowing exact resumption. Mosaic StreamingDataset and the Megatron-LM data loader both support deterministic resumption out of the box.

Cost reality check: Processing a full LLaMA 3–scale pipeline (15T tokens) on AWS costs approximately $25K–50K in compute alone (Spark on EMR, ~500 r5.4xlarge instances for 48 hours). The tokenized output is ~8 TB stored on S3. By comparison, the GPU training run costs $10M+. The data pipeline is less than 0.5% of total training cost—yet its quality determines whether that $10M produces a good model or a mediocre one.

The data pipeline is not a one-shot process. As model evaluation reveals weaknesses (e.g., poor code generation, weak multilingual ability), teams iterate on mixing ratios, filtering thresholds, and data sources. Building the pipeline as a modular, reproducible system—with versioned data artifacts, lineage tracking, and automated quality metrics—is what separates production-grade LLM infrastructure from ad-hoc scripts.
