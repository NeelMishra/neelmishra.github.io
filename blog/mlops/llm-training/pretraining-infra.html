Pre-Training Infrastructure: GPU Clusters

MLOps Series LLM Training Infrastructure

Pre-training a frontier language model—GPT-4-class or beyond—requires orchestrating thousands of accelerators into a single, tightly-coupled compute fabric. The infrastructure choices you make at this layer determine your effective FLOP utilization (MFU), wall-clock time per training token, and ultimately, cost per quality unit. This post dissects the five pillars of pre-training infrastructure: cluster architecture, networking, storage, scheduling, and fault tolerance.

Cluster Architecture

Modern LLM training clusters are built from dense GPU nodes interconnected by high-radix, low-latency networks. The canonical building block is NVIDIA's DGX platform, but cloud-native alternatives using custom VM topologies have become viable for organizations without on-premise capital.

DGX SuperPOD & BasePOD

A single NVIDIA DGX H100 node contains 8× H100 SXM5 GPUs connected via NVLink 4.0 (900 GB/s bidirectional per GPU). Within the node, NVSwitch provides all-to-all GPU connectivity at full bisection bandwidth—this is the intra-node fabric. A SuperPOD scales to 32 DGX nodes (256 GPUs) on a single InfiniBand leaf-spine fabric, while linking multiple SuperPODs produces clusters of 4,096+ GPUs.

Key metric: A single DGX H100 node delivers ~32 PFLOPS of FP8 compute. A 1,024-GPU cluster (128 nodes) therefore has a theoretical peak of ~4 EFLOPS FP8. In practice, MFU of 40–55% is considered excellent for transformer pre-training, yielding ~1.6–2.2 EFLOPS sustained.

Cloud-Native Clusters

Cloud providers expose GPU VMs with similar intra-node topologies (e.g., GCP a3-mega with 8× H100, AWS p5.48xlarge). The critical difference is inter-node networking: cloud fabrics use proprietary or RoCEv2 networks that may introduce higher tail latency compared to dedicated InfiniBand. Topology-aware VM placement (e.g., GCP compact placement policies, AWS cluster placement groups) is essential to keep all-reduce latency bounded.

On-Premise (DGX SuperPOD)

  • Full control over network topology
  • InfiniBand HDR/NDR with predictable latency
  • High upfront CapEx, lower long-run cost
  • Dedicated Lustre/GPFS storage tier
  • Typical scale: 512–16,384 GPUs

Cloud-Native (GCP/AWS/Azure)

  • Elastic scaling with pay-per-use
  • RoCEv2 or proprietary fabric (e.g., EFA, Titanium)
  • Placement groups critical for collocation
  • Object-store backed (GCS/S3) + local NVMe
  • Typical scale: 256–4,096 GPUs per job

Regardless of deployment model, the node-level configuration must be tuned. Below is a typical environment configuration script for an H100 cluster node:

#!/bin/bash — Node-level GPU environment setup # Pin GPU clocks to max for deterministic perf nvidia-smi -pm 1 nvidia-smi -lgc 1980,1980 # Enable persistent mode and set compute mode nvidia-smi -c EXCLUSIVE_PROCESS # Verify NVLink topology nvidia-smi topo -m # Set NCCL environment variables export NCCL_IB_HCA=mlx5 export NCCL_IB_GID_INDEX=3 export NCCL_SOCKET_IFNAME=eth0 export NCCL_DEBUG=INFO export NCCL_NET_GDR_LEVEL=PHB

High-Speed Networking

The inter-node network is the single biggest bottleneck in distributed training. Gradient synchronization via all-reduce scales with model size: a 70B-parameter model produces ~140 GB of gradients per step (FP16), all of which must be reduced across the data-parallel dimension. At 400 Gbps per link, a ring all-reduce across 128 nodes takes ~4.5 seconds—comparable to the forward+backward compute time itself.

InfiniBand vs. RoCEv2

InfiniBand (IB) remains the gold standard for training fabrics. NDR (400 Gbps) provides RDMA with hardware-level congestion control, adaptive routing, and credit-based flow control. RoCEv2 (RDMA over Converged Ethernet) runs over standard Ethernet switches but requires careful PFC (Priority Flow Control) and ECN (Explicit Congestion Notification) tuning to avoid head-of-line blocking and PFC storms.

PFC Storm Warning: In RoCEv2 deployments, a single slow receiver can trigger PFC pause frames that propagate across the fabric, stalling unrelated flows. This is the #1 cause of training hangs in cloud GPU clusters. Mitigations include per-port PFC watchdogs, ECN marking thresholds at 5–10% queue depth, and DCQCN congestion control.

InfiniBand NDR (400G)

  • Sub-microsecond RDMA latency (~0.6 µs)
  • Hardware adaptive routing
  • Credit-based flow control (lossless)
  • SHARP in-network reduction for all-reduce
  • Proven at 10K+ GPU scale

RoCEv2 (400GbE)

  • ~1–2 µs RDMA latency
  • Standard Ethernet switches (lower cost)
  • Requires PFC + ECN tuning
  • No native in-network reduction
  • Risk of PFC storms at scale

Fat-Tree Topology

Both IB and Ethernet clusters use a fat-tree (Clos) topology to achieve full bisection bandwidth. In a 3-tier fat-tree with 64-port switches: leaf switches connect 32 GPUs (4 nodes × 8 NICs) and uplink 32 ports to spine switches. Spine switches connect to super-spine (core) switches. The oversubscription ratio at each tier determines whether the fabric is blocking. A 1:1 subscription ratio at all tiers gives non-blocking full bisection bandwidth—critical for all-reduce-heavy workloads.

# NCCL tuning for InfiniBand fat-tree topology # These environment variables are set per-job in the SLURM script # Use all available IB HCAs (8 per DGX node) export NCCL_IB_HCA="^=mlx5_bond_0" export NCCL_IB_TIMEOUT=23 export NCCL_IB_RETRY_CNT=7 # Enable SHARP for in-network all-reduce (IB only) export NCCL_COLLNET_ENABLE=1 export SHARP_COLL_LOG_LEVEL=3 # Topology-aware algorithm selection export NCCL_ALGO=Ring,Tree export NCCL_PROTO=Simple,LL128 # GDR (GPUDirect RDMA) for zero-copy transfers export NCCL_NET_GDR_LEVEL=SYS export NCCL_NET_GDR_READ=1 export NCCL_P2P_LEVEL=NVL

Distributed Storage

Pre-training workloads generate three distinct I/O patterns: (1) streaming reads of tokenized training data at ~2–10 GB/s aggregate, (2) bursty checkpoint writes of 100+ GB model states every 5–30 minutes, and (3) random reads for data shuffling and curriculum sampling. No single storage tier optimally serves all three patterns.

Lustre & GPFS

Parallel file systems like Lustre and IBM Spectrum Scale (GPFS) remain the backbone of on-premise training storage. Lustre stripes files across Object Storage Targets (OSTs), delivering aggregate throughput proportional to the number of OSTs. A typical SuperPOD attaches 20–40 PB of Lustre with 200+ GB/s aggregate read bandwidth—sufficient to keep 1,024 GPUs fed without I/O stalls.

Checkpoint I/O bottleneck: A 70B model with optimizer states in FP32 produces ~560 GB per checkpoint (4 bytes × 70B params × 2 for Adam moments). With 1,024 data-parallel ranks writing simultaneously, the storage system must absorb 560 GB in under 60 seconds to avoid blocking training—requiring ~10 GB/s sustained write throughput.

Object Stores & Tiered Caching

Cloud clusters use a tiered approach: training data resides in object storage (S3/GCS) and is prefetched into local NVMe SSDs or a shared NFS/Filestore cache. Checkpoints are written to local NVMe first, then asynchronously flushed to object storage. This decouples checkpoint latency from the training critical path.

# Tiered checkpoint manager using PyTorch distributed checkpointing import torch import torch.distributed as dist from torch.distributed.checkpoint import ( FileSystemWriter, save_state_dict ) import threading import subprocess class TieredCheckpointManager: def __init__(self, local_dir, remote_uri, interval_steps=500): self.local_dir = local_dir self.remote_uri = remote_uri self.interval = interval_steps self._flush_thread = None def save(self, model, optimizer, step): # Phase 1: fast write to local NVMe ckpt_path = f"{self.local_dir}/step_{step}" writer = FileSystemWriter(ckpt_path) state = {"model": model.state_dict(), "optim": optimizer.state_dict()} save_state_dict(state, writer) dist.barrier() # Phase 2: async flush to object store (non-blocking) if dist.get_rank() == 0: self._flush_thread = threading.Thread( target=self._flush_to_remote, args=(ckpt_path, step) ) self._flush_thread.start() def _flush_to_remote(self, local_path, step): remote_path = f"{self.remote_uri}/step_{step}" subprocess.run( ["gsutil", "-m", "cp", "-r", local_path, remote_path], check=True )

Job Scheduling

Large-scale training jobs require topology-aware scheduling to minimize cross-rack communication. SLURM remains the dominant scheduler for on-premise GPU clusters, while Kubernetes with custom operators (e.g., Volcano, Run:ai) handles cloud-native deployments. The scheduler must understand GPU affinity, NVLink domains, network locality, and job preemption policies.

SLURM Configuration

A SLURM job script for multi-node training must request the correct number of tasks per node (matching GPUs), bind tasks to specific GPU and NIC affinity, and set environment variables for NCCL transport. Below is a production-grade SLURM script for launching a 128-GPU (16-node) training run:

#!/bin/bash #SBATCH --job-name=llm-pretrain-70b #SBATCH --partition=gpu-h100 #SBATCH --nodes=16 #SBATCH --ntasks-per-node=8 #SBATCH --gpus-per-node=8 #SBATCH --cpus-per-task=12 #SBATCH --mem=0 #SBATCH --time=168:00:00 #SBATCH --exclusive #SBATCH --switches=1@00:30:00 # ^ --switches=1 requests all nodes under one spine switch # with a 30-minute wait tolerance for topology placement module load cuda/12.3 nccl/2.20.3 openmpi/4.1.6 # Master node address for torch distributed export MASTER_ADDR=$(scontrol show hostname $SLURM_NODELIST | head -n 1) export MASTER_PORT=29500 export WORLD_SIZE=$((SLURM_NNODES * 8)) # NCCL tuning export NCCL_IB_HCA=mlx5 export NCCL_IB_GID_INDEX=3 export NCCL_SOCKET_IFNAME=ib0 export NCCL_DEBUG=WARN # Launch with torchrun via srun srun --mpi=pmix \ torchrun \ --nproc_per_node=8 \ --nnodes=$SLURM_NNODES \ --node_rank=$SLURM_NODEID \ --master_addr=$MASTER_ADDR \ --master_port=$MASTER_PORT \ train.py \ --model-config configs/llama-70b.yaml \ --data-path /lustre/datasets/tokenized/ \ --checkpoint-dir /lustre/checkpoints/70b-run-003/ \ --tensor-parallel-size 8 \ --pipeline-parallel-size 4 \ --data-parallel-size 4 \ --micro-batch-size 2 \ --global-batch-size 1024 \ --lr 1.5e-4 \ --warmup-steps 2000 \ --max-steps 500000

Kubernetes & Volcano

For cloud deployments, Kubernetes with the Volcano batch scheduler provides topology-aware gang scheduling. The key abstraction is a VolcanoJob that ensures all pods in a training job are co-scheduled atomically—no partial allocations that waste resources. Topology affinity rules map pods to nodes within the same rack or placement group.

Gang scheduling is non-negotiable: Without gang scheduling, Kubernetes may schedule 60 of 128 pods, leaving them idle while waiting for the remaining 68. This wastes GPU-hours at ~$2/GPU/hr. Volcano's minAvailable field and pod-group primitives solve this by treating the job as an atomic unit.

Fault Tolerance & Checkpointing

At 1,000+ GPU scale, hardware failures are a statistical certainty. With a per-GPU MTBF of ~10,000 hours, a 4,096-GPU cluster expects a failure every ~2.5 hours. The training system must detect failures, checkpoint frequently, and resume from the latest consistent state with minimal lost compute.

Failure Modes

Hardware Failures

  • GPU ECC errors: Uncorrectable memory errors crash the CUDA context
  • NVLink errors: Lane degradation reduces intra-node bandwidth
  • Network link flaps: IB port resets cause NCCL timeouts
  • Node crashes: Kernel panics, PSU failures, thermal shutdowns

Software Failures

  • NCCL timeout: Collective hangs from network asymmetry
  • OOM kills: Activation memory spikes exceeding GPU VRAM
  • NaN divergence: Loss spikes requiring rollback to earlier checkpoint
  • Deadlocks: Pipeline bubble miscalculations in PP schedules

Elastic Training & Auto-Recovery

Production systems implement a supervisor loop that monitors all ranks via heartbeat. On failure detection, the supervisor (1) identifies the failed node, (2) requests a replacement from the scheduler, (3) redistributes the parallelism mapping, and (4) resumes from the last checkpoint. Frameworks like Megatron-LM, torchtitan, and Nvidia NeMo provide built-in checkpoint-resume with automatic rank remapping.

# Fault-tolerant training loop with automatic checkpoint-resume import torch import torch.distributed as dist import os, signal, logging logger = logging.getLogger(__name__) class FaultTolerantTrainer: def __init__(self, model, optimizer, ckpt_dir, ckpt_interval=500): self.model = model self.optimizer = optimizer self.ckpt_dir = ckpt_dir self.ckpt_interval = ckpt_interval self.global_step = 0 self._register_signal_handlers() def _register_signal_handlers(self): # Graceful shutdown on SIGTERM (SLURM preemption) signal.signal(signal.SIGTERM, self._handle_preempt) def _handle_preempt(self, signum, frame): logger.warning("SIGTERM received — saving emergency checkpoint") self.save_checkpoint(tag="emergency") dist.destroy_process_group() exit(0) def resume(self): # Find latest checkpoint in directory ckpts = sorted( [d for d in os.listdir(self.ckpt_dir) if d.startswith("step_")], key=lambda x: int(x.split("_")[1]) ) if ckpts: latest = os.path.join(self.ckpt_dir, ckpts[-1]) state = torch.load(latest, map_location="cpu") self.model.load_state_dict(state["model"]) self.optimizer.load_state_dict(state["optim"]) self.global_step = state["step"] logger.info(f"Resumed from step {self.global_step}") def save_checkpoint(self, tag=None): name = tag if tag else f"step_{self.global_step}" path = os.path.join(self.ckpt_dir, name) if dist.get_rank() == 0: torch.save({ "model": self.model.state_dict(), "optim": self.optimizer.state_dict(), "step": self.global_step }, path) dist.barrier() def train(self, dataloader, max_steps): self.resume() for batch in dataloader: if self.global_step >= max_steps: break loss = self.model(batch) loss.backward() self.optimizer.step() self.optimizer.zero_grad() self.global_step += 1 if self.global_step % self.ckpt_interval == 0: self.save_checkpoint()
Async checkpointing: Blocking all ranks for checkpoint I/O wastes 2–5% of training time. Production systems use asynchronous checkpointing: ranks snapshot their state to pinned host memory, then a background thread writes to storage while training continues on the next micro-batch. PyTorch's torch.distributed.checkpoint and NVIDIA's NeMo both support this pattern.

Cluster Topology Diagram

The diagram below illustrates a simplified 3-tier fat-tree cluster with 4 DGX nodes (32 GPUs), leaf switches, spine switches, and a shared parallel file system. In production, this pattern is replicated across multiple pods with core/super-spine switches connecting them.

In this topology, each DGX node has 8 ConnectX-7 NICs providing 8× 400 Gbps = 3.2 Tbps of aggregate inter-node bandwidth. The two spine switches provide full bisection bandwidth across all 4 leaf switches. In a production SuperPOD, this scales to 32 nodes per pod with 8 spine switches, and multiple pods are interconnected via a core switching tier.

Rail-optimized topology: NVIDIA's rail-optimized design assigns each of the 8 GPUs in a node to a dedicated NIC and leaf switch "rail." GPU 0 across all nodes communicates via Rail 0 (Leaf Switch 1), GPU 1 via Rail 1, and so on. This eliminates intra-node PCIe contention during all-reduce and enables optimal NCCL ring/tree algorithm placement.
