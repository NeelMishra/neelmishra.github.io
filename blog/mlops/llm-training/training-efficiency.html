Model FLOPs Utilization (MFU)

Model FLOPs Utilization (MFU) is the single most important metric for evaluating how efficiently a training system converts raw hardware capability into useful model computation. Introduced formally in the PaLM paper (Chowdhery et al., 2022), MFU measures the ratio of observed throughput (in FLOP/s) to the theoretical peak throughput of the hardware, considering only the forward and backward pass FLOPs of the model — excluding rematerialization, optimizer state updates, and communication overhead.

The formal definition is:

MFU = (Observed Model FLOP/s) ÷ (Hardware Peak FLOP/s)

To compute observed model FLOP/s, we first estimate the total FLOPs per training step. For a standard Transformer with L layers, hidden dimension H, sequence length S, vocabulary size V, and batch size B, the per-step FLOPs for a forward pass are approximately:

Cfwd ≈ 2 × P × B × S

where P is the total number of parameters. The backward pass is roughly 2× the forward pass cost (computing gradients with respect to both activations and weights), giving a total per-step cost of:

Cstep ≈ 6 × P × B × S

Why 6×P×B×S? Each matrix multiply of shape (M,K)×(K,N) requires 2×M×K×N FLOPs (one multiply + one add per output element). The forward pass performs roughly P/H such multiplies across all layers, each operating on B×S tokens. The backward pass doubles this for the two gradient computations. Hence: 2P·B·S (fwd) + 4P·B·S (bwd) = 6P·B·S total.

Given the wall-clock time per step Tstep, the observed model throughput is:

Throughputmodel = Cstep / Tstep

For a cluster of N GPUs, each with peak performance Fpeak FLOP/s, the MFU is:

MFU = (6 × P × B × S) / (Tstep × N × Fpeak)

MFU vs HFU: Hardware FLOPs Utilization (HFU) includes all FLOPs executed on the hardware, including activation recomputation from gradient checkpointing. MFU intentionally excludes these because recomputation is a memory-compute trade-off, not useful model work. When reporting efficiency numbers, always clarify whether you mean MFU or HFU — the gap can be 20-40% for aggressively checkpointed runs.

Hardware Efficiency & Peak vs Achieved FLOPS

Understanding the gap between theoretical peak and achieved FLOP/s is essential for diagnosing training inefficiency. The theoretical peak is determined by the hardware's compute units, clock speed, and supported numeric precision.

NVIDIA A100 (80 GB SXM)

  • BF16 Tensor Core Peak: 312 TFLOP/s
  • FP32 Peak: 156 TFLOP/s (non-Tensor)
  • HBM2e Bandwidth: 2.0 TB/s
  • NVLink Bandwidth: 600 GB/s (bidirectional)
  • TDP: 400W
  • Typical MFU (LLM): 40-57%

NVIDIA H100 (80 GB SXM)

  • BF16 Tensor Core Peak: 989 TFLOP/s
  • FP8 Tensor Core Peak: 1,979 TFLOP/s
  • HBM3 Bandwidth: 3.35 TB/s
  • NVLink Bandwidth: 900 GB/s (bidirectional)
  • TDP: 700W
  • Typical MFU (LLM): 35-47%

Notice that the H100 has over 3× the raw BF16 FLOP/s of the A100, yet typical MFU is often lower on H100s. This is because the compute-to-memory-bandwidth ratio (arithmetic intensity ceiling) increased faster than memory bandwidth scaled. The H100 requires higher arithmetic intensity workloads (larger batch sizes, longer sequences) to saturate its compute units. This is quantified by the roofline model:

Achievable FLOP/s = min(Peak FLOP/s, Arithmetic Intensity × Memory Bandwidth)

The crossover point — where a workload transitions from memory-bound to compute-bound — occurs at the ridge point: Peak FLOP/s / Memory Bandwidth. For the A100 in BF16 this is 312/2000 ≈ 156 ops/byte; for the H100 it is 989/3350 ≈ 295 ops/byte. Operations with lower arithmetic intensity than the ridge point (layer norms, activations, small GEMMs) will be memory-bound regardless of the GPU generation.

TPU v4 Pod Slice

  • BF16 Peak per chip: 275 TFLOP/s
  • HBM Bandwidth: 1.2 TB/s
  • ICI Bandwidth: 4.8 Tbps (inter-chip)
  • Typical MFU: 46-60% (Megascale: PaLM reported ~57%)

AMD MI300X

  • BF16 Peak: 1,307 TFLOP/s
  • HBM3 Bandwidth: 5.3 TB/s
  • Infinity Fabric: 896 GB/s
  • Typical MFU: 30-40% (early ecosystem)
Practical insight: When evaluating hardware for LLM training, the metric that matters most is achievable FLOP/$ or FLOP/Watt at your target model size and batch configuration — not raw peak TFLOP/s. An A100 cluster achieving 55% MFU may deliver better cost efficiency than an H100 cluster at 38% MFU for certain model configurations.

Tokens per Second per GPU

While MFU measures hardware utilization abstractly, tokens/second/GPU is the operational metric that directly connects to training cost and time-to-completion. It is computed as:

Tokens/s/GPU = (B × S) / (Tstep × N)

where B is global batch size in sequences, S is sequence length, Tstep is wall-clock time per training step, and N is the number of GPUs. This metric has a direct relationship with MFU:

Tokens/s/GPU = MFU × Fpeak / (6 × P)

This formula reveals an important insight: for a fixed hardware configuration and MFU, tokens/s/GPU is inversely proportional to model size. A 70B parameter model processes tokens ~10× slower than a 7B model at the same MFU, which is why scaling laws and Chinchilla-optimal ratios matter so much for training budgets.

Here is a practical measurement script using CUDA events for precise GPU timing:

import torch import time from contextlib import contextmanager class ThroughputTracker: def __init__(self, model_params, seq_len, world_size): self.model_params = model_params self.seq_len = seq_len self.world_size = world_size self.start_event = torch.cuda.Event(enable_timing=True) self.end_event = torch.cuda.Event(enable_timing=True) self.step_times = [] def step_start(self): # Record start on the current CUDA stream self.start_event.record() def step_end(self, batch_size): self.end_event.record() torch.cuda.synchronize() elapsed_ms = self.start_event.elapsed_time(self.end_event) elapsed_s = elapsed_ms / 1000.0 tokens_this_step = batch_size * self.seq_len tokens_per_sec = tokens_this_step / elapsed_s tokens_per_sec_per_gpu = tokens_per_sec / self.world_size # Compute MFU: 6 * P * tokens_processed / (elapsed * peak_flops) model_flops = 6 * self.model_params * tokens_this_step peak_flops = 312e12 * self.world_size # A100 BF16 peak mfu = model_flops / (elapsed_s * peak_flops) self.step_times.append({ "elapsed_s": elapsed_s, "tokens_per_sec_per_gpu": tokens_per_sec_per_gpu, "mfu": mfu, }) return tokens_per_sec_per_gpu, mfu def summary(self): # Skip first 5 steps (warmup) for accurate measurement steady = self.step_times[5:] avg_tps = sum(s["tokens_per_sec_per_gpu"] for s in steady) / len(steady) avg_mfu = sum(s["mfu"] for s in steady) / len(steady) print(f"Avg tokens/s/GPU: {avg_tps:.0f} | Avg MFU: {avg_mfu:.2%}")
Measurement pitfalls: Always use CUDA events for GPU timing, not Python's time.time(). CPU wall-clock time includes kernel launch latency and asynchronous dispatch overhead, which can skew measurements by 5-15%. Additionally, always discard the first several warmup steps — JIT compilation, CUDA context initialization, and memory allocator warm-up inflate early step times.

Profiling Tools: PyTorch Profiler & NVIDIA Nsight

Effective profiling is the foundation of efficiency optimization. Two tools dominate the LLM training profiling stack: PyTorch Profiler (integrated, operator-level) and NVIDIA Nsight Systems (system-level, timeline-based).

PyTorch Profiler

PyTorch Profiler traces CUDA kernels, CPU operators, memory allocations, and communication collectives. It integrates with TensorBoard for visualization and can export Chrome trace format for offline analysis.

import torch from torch.profiler import profile, record_function, ProfilerActivity def profile_training_step(model, optimizer, dataloader, device): schedule = torch.profiler.schedule( wait=2, # skip 2 steps warmup=3, # warmup JIT/caches for 3 steps active=5, # profile 5 steps repeat=1 ) with profile( activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA], schedule=schedule, on_trace_ready=torch.profiler.tensorboard_trace_handler("./tb_logs"), record_shapes=True, profile_memory=True, with_stack=True, with_flops=True, ) as prof: for step, batch in enumerate(dataloader): if step >= 10: break input_ids = batch["input_ids"].to(device) labels = batch["labels"].to(device) with record_function("forward_pass"): outputs = model(input_ids, labels=labels) loss = outputs.loss with record_function("backward_pass"): loss.backward() with record_function("optimizer_step"): optimizer.step() optimizer.zero_grad(set_to_none=True) prof.step() # Print top CUDA kernel table sorted by total GPU time print(prof.key_averages().table( sort_by="cuda_time_total", row_limit=20 ))

NVIDIA Nsight Systems

Nsight Systems provides a system-wide timeline view showing CPU threads, CUDA streams, GPU kernels, NCCL collectives, and NVLink/PCIe transfers on a unified timeline. This is critical for identifying pipeline bubbles, communication-computation overlap gaps, and kernel launch stalls.

# Launch Nsight Systems profiling from command line # --trace=cuda,nvtx,osrt captures CUDA API, NVTX ranges, and OS runtime # --cuda-memory-usage=true tracks allocation patterns # -o profile_output sets the output filename nsys profile \ --trace=cuda,nvtx,osrt,cudnn,cublas \ --cuda-memory-usage=true \ --gpu-metrics-device=all \ --force-overwrite=true \ -o training_profile \ python train.py --steps 20 --profile # Post-processing: export to SQLite for programmatic analysis nsys export --type=sqlite training_profile.nsys-rep # Query kernel durations from exported database # SELECT name, AVG(end - start) as avg_ns FROM CUPTI_ACTIVITY_KIND_KERNEL # GROUP BY name ORDER BY avg_ns DESC LIMIT 20;
Pro tip: Use NVTX annotations (torch.cuda.nvtx.range_push/pop) to mark logical phases (forward, backward, allreduce, optimizer) in your training loop. This makes Nsight timelines dramatically easier to interpret, especially with pipeline parallelism where multiple microbatches interleave across stages.

Bottleneck Identification

Training inefficiency manifests as one of three fundamental bottleneck types. Correctly identifying which category dominates is the prerequisite for effective optimization.

🔥 Compute-Bound

Symptom: GPU SM occupancy is high (>80%), kernels are large GEMMs, and adding more memory bandwidth doesn't help throughput.

Diagnosis: In Nsight, the timeline shows dense, back-to-back GEMM kernels with minimal gaps. The roofline analysis places the workload at or near the peak compute ceiling.

Common cause: Large batch sizes with long sequences on adequately-provisioned hardware. This is the desirable regime — being compute-bound means you're using the hardware effectively.

💾 Memory-Bound

Symptom: GPU SMs are frequently stalled waiting for data from HBM. Memory bandwidth utilization is near peak while compute utilization is low.

Diagnosis: Nsight Compute shows low SM active cycles percentage. Operations like LayerNorm, softmax, element-wise activations, and small GEMMs dominate the profile.

Common cause: Small batch sizes, short sequences, or model architectures with many non-GEMM operations. Kernel fusion (FlashAttention, fused LayerNorm) is the primary remedy.

📡 Communication-Bound

Symptom: GPUs spend significant time idle waiting for allreduce, allgather, or reduce-scatter collectives to complete. Adding more GPUs worsens per-GPU throughput.

Diagnosis: Nsight timeline shows large gaps between compute kernels filled by NCCL collective calls. The ratio of communication time to compute time exceeds 30-40%.

Common cause: High tensor parallelism degree (TP>8), insufficient NVLink/IB bandwidth, poor overlap of communication with computation, or gradient accumulation steps set too low.

🔄 Pipeline Bubble-Bound

Symptom: With pipeline parallelism, GPUs in early/late stages are idle while waiting for microbatches to propagate through the pipeline.

Diagnosis: Pipeline bubble fraction = (pp_stages - 1) / total_microbatches. Nsight shows periodic idle gaps aligned with pipeline stage boundaries.

Common cause: Too few microbatches relative to pipeline stages. The 1F1B schedule reduces memory but doesn't eliminate bubbles. Interleaved schedules (virtual stages) help.

A systematic diagnosis procedure:

def diagnose_bottleneck(profile_data): # Step 1: Compute the time breakdown total_step_time = profile_data["step_time_ms"] gemm_time = profile_data["gemm_kernel_time_ms"] nccl_time = profile_data["nccl_collective_time_ms"] memory_ops_time = profile_data["elementwise_norm_time_ms"] idle_time = total_step_time - gemm_time - nccl_time - memory_ops_time gemm_frac = gemm_time / total_step_time comm_frac = nccl_time / total_step_time mem_frac = memory_ops_time / total_step_time # Step 2: Classify the dominant bottleneck if gemm_frac > 0.65: return "COMPUTE_BOUND" # Ideal: most time in useful GEMMs elif comm_frac > 0.30: return "COMMUNICATION_BOUND" # Need overlap or reduce TP degree elif mem_frac > 0.25: return "MEMORY_BOUND" # Fuse kernels or increase batch size else: return "MIXED/BUBBLE_BOUND" # Pipeline or scheduling issues

Optimization Strategies

Once the bottleneck is identified, apply targeted optimizations. The following techniques are ordered roughly by impact-to-effort ratio for large-scale LLM training.

1. FlashAttention and Kernel Fusion

FlashAttention (Dao et al., 2022) fuses the QKV projection, softmax, and attention output computation into a single kernel that tiles across the sequence dimension, avoiding materializing the full S×S attention matrix in HBM. This reduces memory reads/writes from O(S²) to O(S) and typically yields a 2-4× speedup on the attention computation, which constitutes 30-40% of total forward pass time in standard Transformers.

2. Mixed Precision Training

Using BF16 for forward/backward passes and FP32 for the master weights and optimizer states (the "mixed precision" recipe) doubles arithmetic throughput on Tensor Cores while maintaining training stability. FP8 training (available on H100/H200) can further double throughput but requires careful loss scaling and per-tensor quantization calibration. The key is ensuring the loss scaler prevents gradient underflow in reduced precision.

3. Communication-Computation Overlap

The most impactful technique for multi-node training is overlapping gradient allreduce with backward computation. With bucketed gradient allreduce, NCCL collectives for earlier layers fire as soon as their gradients are computed, while later layers are still in the backward pass. Properly configured, this can hide 70-90% of communication time behind computation.

# Gradient bucketing configuration for overlap # Larger buckets = fewer NCCL calls (less launch overhead) # Smaller buckets = earlier overlap start (more hiding) model = DistributedDataParallel( model, device_ids=[local_rank], bucket_cap_mb=25, # 25 MB buckets (tune this!) gradient_as_bucket_view=True, # Avoid gradient copy static_graph=True, # Enable extra optimizations ) # For FSDP: overlap allgather with forward, reduce-scatter with backward fsdp_config = FullyShardedDataParallelConfig( forward_prefetch=True, # Prefetch next layer's params during forward backward_prefetch="backward_pre", # Prefetch during backward limit_all_gathers=True, # Prevent too many concurrent allgathers )

4. Gradient Accumulation Tuning

Gradient accumulation allows simulating larger batch sizes without proportionally increasing memory. However, the accumulation steps are pure computation with no communication, so the overall communication-to-compute ratio improves with more accumulation steps. If you are communication-bound, increasing gradient accumulation from 1 to 4 can recover 15-25% throughput.

5. Activation Checkpointing Strategy

Full activation checkpointing (recomputing all activations during backward) trades ~33% more compute for ~60% memory savings. Selective checkpointing — only recomputing cheap operations like attention softmax while keeping expensive GEMM outputs — achieves 80-90% of the memory savings with only 10-15% compute overhead. This directly improves MFU since less recomputation means the MFU/HFU gap shrinks.

6. Data Loading Pipeline

Silent throughput killer: Inefficient data loading can reduce tokens/s/GPU by 10-20% without obvious symptoms in GPU profilers. Ensure you use multi-worker data loading with prefetching, memory-mapped datasets (e.g., np.memmap or HuggingFace datasets' Arrow format), and pre-tokenized data. Never tokenize on-the-fly during training at scale.

Chinchilla-Optimal Training

The Chinchilla scaling laws (Hoffmann et al., 2022) established that for a fixed compute budget C (measured in FLOPs), there exists an optimal allocation between model size N and training tokens D. The key finding: most large models were significantly undertrained relative to their parameter count.

The Chinchilla-optimal relationship is approximately:

Dopt ≈ 20 × N

That is, a model should be trained on roughly 20 tokens per parameter. A 7B model requires ~140B tokens; a 70B model requires ~1.4T tokens. The total compute budget follows:

C ≈ 6 × N × D

For a given compute budget, solving for optimal N and D simultaneously:

Nopt ∝ C0.5    and    Dopt ∝ C0.5

This has profound implications for training efficiency: spending your GPU-hours training a model that is too large on too few tokens is strictly wasteful. A smaller model trained on more data will achieve lower loss at the same compute cost, and will also be faster at inference due to fewer parameters.

Post-Chinchilla refinements: Recent work (LLaMA, DeepSeek) trains well beyond the Chinchilla-optimal ratio (up to 100-200 tokens per parameter) because inference cost dominates total lifecycle cost. Over-training a smaller model is compute-suboptimal for a single training run but delivers better inference efficiency. This shifts the efficiency calculus: tokens/s at inference matters more than training MFU for deployment-oriented models.

To translate Chinchilla-optimal decisions into wall-clock time estimates:

def estimate_training_time( num_params, # Total model parameters (e.g., 70e9) num_tokens, # Training tokens (e.g., 1.4e12) num_gpus, # Number of GPUs peak_flops_per_gpu, # BF16 peak FLOP/s (e.g., 312e12 for A100) target_mfu, # Expected MFU (e.g., 0.45) ): # Total training FLOPs total_flops = 6 * num_params * num_tokens # Achieved throughput accounting for MFU achieved_flops = num_gpus * peak_flops_per_gpu * target_mfu # Wall-clock seconds training_seconds = total_flops / achieved_flops training_days = training_seconds / 86400 # Cost estimation (assume $2/GPU-hour for A100) gpu_hours = (training_seconds / 3600) * num_gpus cost = gpu_hours * 2.0 print(f"Total FLOPs: {total_flops:.2e}") print(f"Training time: {training_days:.1f} days") print(f"GPU-hours: {gpu_hours:,.0f}") print(f"Est. cost: ${cost:,.0f}") return training_days, cost # Example: LLaMA-2 70B on 2048 A100s at 45% MFU estimate_training_time( num_params=70e9, num_tokens=2e12, num_gpus=2048, peak_flops_per_gpu=312e12, target_mfu=0.45, ) # Output: ~29 days, ~$2.9M at $2/GPU-hr

MFU Breakdown Across Hardware

The following diagram visualizes achieved MFU across different hardware platforms and training configurations, illustrating how model size, parallelism strategy, and hardware generation interact to determine real-world efficiency.

Key takeaways from this data: (1) MFU generally decreases with model scale due to increased parallelism overhead and communication costs. (2) Google's TPU v4 achieves the highest reported MFU on very large models thanks to its high-bandwidth ICI interconnect and XLA compiler's aggressive optimization. (3) H100 MFU appears lower than A100 partly because the denominator (peak FLOP/s) grew 3× while memory bandwidth only grew 1.7×, raising the arithmetic intensity threshold for compute saturation. (4) AMD MI300X has the raw specs to compete but lacks the mature software stack (NCCL equivalent, optimized kernels) to translate hardware capability into training throughput.

How to interpret these numbers in interviews: A well-optimized large-scale training run should target 40-55% MFU on NVIDIA GPUs. Below 30% indicates serious inefficiency (poor overlap, undersized batches, I/O bottlenecks). Above 60% is exceptional and usually requires custom CUDA kernels, hardware-specific tuning, and near-perfect communication overlap. Google's PaLM at 46-57% MFU on 6144 TPU v4 chips remains the gold standard for large-scale efficiency.
