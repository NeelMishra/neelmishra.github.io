DeepSpeed for LLMs: ZeRO-3, ZeRO-Infinity

MLOps Series LLM Training Infrastructure

Training large language models with billions of parameters requires distributing not just computation but memory across devices. DeepSpeed's ZeRO (Zero Redundancy Optimizer) family eliminates memory redundancy across data-parallel ranks, enabling models that would otherwise require tensor or pipeline parallelism to train purely with data parallelism. This post derives the exact memory formulas for each ZeRO stage, analyzes communication overhead, walks through ZeRO-Infinity's offloading hierarchy, and provides production-ready configurations.

ZeRO Stage 1/2/3 Memory Analysis

Consider a model with Ψ parameters trained with mixed-precision (fp16 forward/backward, fp32 optimizer). The baseline memory per GPU in standard data parallelism (no ZeRO) is:

Baseline Memory (no ZeRO): For Ψ parameters, each GPU stores:
• 2Ψ bytes — fp16 parameters
• 2Ψ bytes — fp16 gradients
• 4Ψ bytes — fp32 master copy of parameters
• 4Ψ bytes — fp32 first momentum (Adam m̂)
• 4Ψ bytes — fp32 second momentum (Adam v̂)
Total = 2Ψ + 2Ψ + 4Ψ + 4Ψ + 4Ψ = 16Ψ bytes per GPU

For a 7B parameter model: 16 × 7 × 10⁹ = 112 GB per GPU — far beyond the 80 GB of an A100. ZeRO partitions these redundant states across N data-parallel ranks.

ZeRO Stage 1: Optimizer State Partitioning

Stage 1 partitions only the optimizer states (fp32 master weights, momentum, and variance) across N GPUs. Each GPU stores 1/N of the optimizer states but retains full copies of fp16 parameters and gradients.

# ZeRO-1 per-GPU memory Optimizer states (partitioned): 12Ψ / N # (4Ψ master + 4Ψ momentum + 4Ψ variance) / N fp16 parameters (replicated): 2Ψ fp16 gradients (replicated): 2Ψ # ───────────────────────────────────── Total = 12Ψ/N + 4Ψ bytes per GPU

With N = 64 GPUs and Ψ = 7B: 12(7B)/64 + 4(7B) = 1.31 GB + 28 GB ≈ 29.3 GB per GPU — a 3.8× reduction from baseline.

ZeRO Stage 2: + Gradient Partitioning

Stage 2 additionally partitions gradients. Each GPU computes full gradients during backward but immediately reduce-scatters them, retaining only its 1/N partition. Gradients for non-owned parameters are discarded after reduction.

# ZeRO-2 per-GPU memory Optimizer states (partitioned): 12Ψ / N fp16 gradients (partitioned): 2Ψ / N fp16 parameters (replicated): 2Ψ # ───────────────────────────────────── Total = 14Ψ/N + 2Ψ bytes per GPU

With N = 64 and Ψ = 7B: 14(7B)/64 + 2(7B) = 1.53 GB + 14 GB ≈ 15.5 GB per GPU.

ZeRO Stage 3: + Parameter Partitioning

Stage 3 partitions everything — optimizer states, gradients, and the fp16 parameters themselves. Parameters are all-gathered on demand before each forward/backward layer computation, then discarded.

# ZeRO-3 per-GPU memory Optimizer states (partitioned): 12Ψ / N fp16 gradients (partitioned): 2Ψ / N fp16 parameters (partitioned): 2Ψ / N # ───────────────────────────────────── Total = 16Ψ/N bytes per GPU

With N = 64 and Ψ = 7B: 16(7B)/64 ≈ 1.75 GB per GPU. This is a 64× reduction from baseline — the theoretical minimum for data parallelism.

Caveat: ZeRO-3's 16Ψ/N formula covers only model states. You must also account for activation memory (proportional to batch size × sequence length × hidden dimension × number of layers) and temporary buffers. Activation checkpointing (discussed below) is almost always used alongside ZeRO-3.

ZeRO-1

Memory: 12Ψ/N + 4Ψ

Partitions: Optimizer states only

Communication: Same as vanilla DDP (all-reduce on gradients)

Best for: Drop-in replacement for DDP with no code changes

ZeRO-2

Memory: 14Ψ/N + 2Ψ

Partitions: Optimizer states + gradients

Communication: Reduce-scatter (same volume as all-reduce)

Best for: Models up to ~13B on 8×A100-80GB

ZeRO-3

Memory: 16Ψ/N

Partitions: Everything (params + grads + optimizer)

Communication: All-gather (fwd) + reduce-scatter (bwd) + all-gather (update)

Best for: Models >13B, especially 70B+ on multi-node clusters

ZeRO-Infinity

Memory: Bounded by CPU/NVMe, not GPU

Partitions: Everything + offload to heterogeneous memory

Communication: ZeRO-3 collectives + PCIe transfers

Best for: Trillion-parameter models on limited GPU clusters

ZeRO-Infinity: CPU & NVMe Offloading

ZeRO-Infinity extends ZeRO-3 by offloading partitioned states to a heterogeneous memory hierarchy: GPU HBM → CPU DRAM → NVMe SSD. This enables training models with trillions of parameters on clusters with limited GPU memory.

Offload Strategies

DeepSpeed supports three offload targets, configured independently for optimizer states and parameters:

  • offload_optimizer → CPU: The fp32 master weights, momentum, and variance tensors (12Ψ/N per GPU) are offloaded to CPU DRAM. The optimizer step runs on CPU using DeepSpeed's fused CPU Adam, which achieves ~1.5 TFLOPS on a 64-core server.
  • offload_optimizer → NVMe: Optimizer states are swapped to NVMe SSDs, loaded into CPU pinned memory in chunks, updated, then written back. This allows 50× more memory than GPU HBM at the cost of higher latency.
  • offload_param → CPU: Even the fp16 parameter partitions (2Ψ/N) are offloaded to CPU, loaded into GPU only for the active layer's all-gather. This provides maximum GPU memory savings.

Bandwidth Analysis

The key bottleneck for offloading is PCIe bandwidth between GPU and CPU. On PCIe Gen4 ×16, peak unidirectional bandwidth is ~32 GB/s (bidirectional ~64 GB/s). For NVMe, enterprise SSDs achieve 6-7 GB/s read and 5-6 GB/s write.

Throughput Model: For a single training step with Ψ/N parameters per GPU, offloading optimizer states to CPU requires transferring 12Ψ/N bytes over PCIe each step (read updated params back, send grads). If computation time per step is T_compute and PCIe transfer time is T_pcie = 12Ψ/(N × BW_pcie), offloading is hidden when T_pcie ≤ T_compute. With a 7B model on 64 GPUs: T_pcie = 12 × 7B / (64 × 32 GB/s) ≈ 41 ms, which is typically ≤ T_compute for batch sizes ≥ 4.
NVMe Offloading Tradeoff: NVMe SSDs are ~5× slower than CPU DRAM access. Use NVMe offloading only when CPU DRAM is also exhausted (e.g., >500B parameter models). Set offload_param.nvme_path to a fast local SSD, not network-attached storage. RAID-0 across multiple NVMe drives can approach 20+ GB/s aggregate bandwidth.

CPU Offload

Bandwidth: PCIe Gen4 = 32 GB/s

Capacity: 256 GB–2 TB (server DRAM)

Latency: ~100ns + PCIe transfer

Use when: GPU memory is insufficient but CPU DRAM is plentiful

NVMe Offload

Bandwidth: 6–7 GB/s per drive (RAID-0 for more)

Capacity: 1–16 TB per node

Latency: ~10μs + transfer

Use when: Both GPU HBM and CPU DRAM are insufficient

Communication Volume Analysis

A critical interview question: "Does ZeRO-3 increase communication?" The answer is nuanced. Let's derive the per-step communication volume for each stage with N GPUs and Ψ parameters.

Vanilla Data Parallelism (Baseline)

Standard DDP performs an all-reduce on gradients. Using ring all-reduce, the communication volume per GPU is:

# Ring all-reduce volume (per GPU) All-reduce = reduce-scatter + all-gather = Ψ × 2 × (N-1)/N ≈ 2Ψ elements = 2 × 2Ψ = 4Ψ bytes # (fp16 gradients, 2 bytes each)

ZeRO-1: Same as Baseline

ZeRO-1 partitions only optimizer states, which are never communicated. Gradient synchronization remains a standard all-reduce. Communication = 4Ψ bytes (identical to DDP).

ZeRO-2: Same Volume, Different Pattern

ZeRO-2 replaces the all-reduce with a reduce-scatter only for gradients (each GPU keeps its 1/N partition). After the optimizer step, each GPU broadcasts its updated fp16 parameter partition via an all-gather. Total volume:

# ZeRO-2 communication volume (per GPU) Backward: reduce-scatter(grads) = 2Ψ × (N-1)/N ≈ 2Ψ bytes Update: all-gather(fp16 params) = 2Ψ × (N-1)/N ≈ 2Ψ bytes # ───────────────────────────────────── Total ≈ 4Ψ bytes # Same as DDP baseline

ZeRO-3: 1.5× Baseline Communication

ZeRO-3 requires all-gathering parameters before both forward and backward passes since no GPU holds the full parameter set:

# ZeRO-3 communication volume (per GPU) Forward: all-gather(fp16 params) = 2Ψ × (N-1)/N ≈ 2Ψ bytes Backward: all-gather(fp16 params) = 2Ψ × (N-1)/N ≈ 2Ψ bytes # need params for gradient computation Backward: reduce-scatter(grads) = 2Ψ × (N-1)/N ≈ 2Ψ bytes # ───────────────────────────────────── Total ≈ 6Ψ bytes # 1.5× DDP baseline
Key Insight: ZeRO-3 trades a 1.5× communication increase for a linear memory reduction. On modern NVLink/NVSwitch interconnects (900 GB/s per GPU on DGX H100), this overhead is negligible. The communication-computation overlap in DeepSpeed's implementation hides most of this cost by prefetching parameters for the next layer while the current layer computes.

DeepSpeed JSON Configuration

DeepSpeed is configured via a JSON file passed at runtime. Below is a production ZeRO-3 config with CPU offloading for training a 70B parameter LLM:

{ // ds_config.json — ZeRO-3 with CPU offloading "bf16": { "enabled": True }, "zero_optimization": { "stage": 3, "offload_optimizer": { "device": "cpu", "pin_memory": True, "buffer_count": 4, "fast_init": False }, "offload_param": { "device": "cpu", "pin_memory": True, "buffer_count": 5, "buffer_size": 1e8, "max_in_cpu": 1e9 }, "overlap_comm": True, "contiguous_gradients": True, "sub_group_size": 1e9, "reduce_bucket_size": "auto", "stage3_prefetch_bucket_size": "auto", "stage3_param_persistence_threshold": "auto", "stage3_max_live_parameters": 1e9, "stage3_max_reuse_distance": 1e9, "stage3_gather_16bit_weights_on_model_save": True }, "gradient_accumulation_steps": 8, "gradient_clipping": 1.0, "steps_per_print": 100, "train_micro_batch_size_per_gpu": 2, "wall_clock_breakdown": False }

Key Configuration Parameters

  • stage3_prefetch_bucket_size: Controls how many parameters are prefetched for the next layer. Set to "auto" to let DeepSpeed profile and pick optimal values. Larger values improve overlap but use more memory.
  • stage3_param_persistence_threshold: Parameters smaller than this threshold (in elements) are kept on GPU permanently. Small embeddings or layer norms below this size avoid all-gather overhead.
  • stage3_max_live_parameters: Caps the number of parameter elements materialized on GPU simultaneously. Limits peak memory during forward/backward when multiple layers' parameters coexist.
  • sub_group_size: Splits large all-gather collectives into smaller sub-groups. Reduces peak memory from all-gather buffers at the cost of more communication rounds.
  • overlap_comm: Enables overlapping communication with computation. The next layer's all-gather starts while the current layer is still computing — essential for hiding ZeRO-3's 1.5× communication overhead.
  • pin_memory: Allocates CPU tensors in pinned (page-locked) memory for faster GPU↔CPU transfers via DMA. Always enable when offloading.
Common Pitfall: Setting stage3_gather_16bit_weights_on_model_save to false saves sharded checkpoints that require the same number of GPUs to reload. Always set to true for portable checkpoints, or use DeepSpeed's zero_to_fp32.py utility to consolidate shards offline.

Activation Checkpointing & Memory Optimization

ZeRO reduces model state memory, but activation memory often dominates for long sequences. A transformer layer stores activations proportional to b × s × h (batch × seq_len × hidden_dim) per layer, and with L layers the total can exceed model state memory for large batch/sequence combinations.

Full Activation Checkpointing

Activation checkpointing (gradient checkpointing) discards intermediate activations during forward and recomputes them during backward. This trades ~33% extra compute for an O(√L) memory reduction when using checkpoint partitioning.

# DeepSpeed activation checkpointing config { "activation_checkpointing": { "partition_activations": True, "cpu_checkpointing": False, "contiguous_memory_optimization": True, "number_checkpoints": None, "synchronize_checkpoint_boundary": False, "profile": False } }

Partition Activations

When partition_activations is enabled, DeepSpeed partitions stored activation checkpoints across data-parallel GPUs (similar to ZeRO-3's parameter partitioning). Each GPU stores 1/N of the checkpointed activations and all-gathers them when needed during recomputation. This further reduces per-GPU activation memory by N×.

CPU Activation Offloading

Setting cpu_checkpointing: true offloads checkpointed activations to CPU DRAM, freeing GPU memory entirely. This is useful for extremely long sequences (8K–128K tokens) where even partitioned checkpoints exceed GPU capacity.

# Activation memory estimation for a transformer # Per layer: 2 × b × s × h bytes (approximate, storing input to each sub-layer) # Full model: L × 2 × b × s × h bytes without checkpointing # With checkpointing: √L × 2 × b × s × h (checkpoint every √L layers) # With partitioning: √L × 2 × b × s × h / N # Example: LLaMA-70B (L=80, h=8192), batch=2, seq=4096, N=64 GPUs # Full: 80 × 2 × 2 × 4096 × 8192 ≈ 85 GB per GPU # Checkpointed: √80 × 2 × 2 × 4096 × 8192 ≈ 9.5 GB per GPU # Partitioned: 9.5 / 64 ≈ 0.15 GB per GPU
Combined Memory Budget (ZeRO-3 + Activation Checkpointing): For a 70B model on 64 A100-80GB GPUs:
Model states: 16Ψ/N = 16 × 70B / 64 ≈ 17.5 GB
Activations (checkpointed + partitioned): ~0.15 GB
Buffers + fragmentation overhead: ~2–5 GB
Total ≈ 20–23 GB — well within 80 GB, leaving room for larger batch sizes.

Training Code with DeepSpeed

Below is a complete training loop integrating DeepSpeed with a HuggingFace model. DeepSpeed wraps the model, optimizer, and data loader, handling all ZeRO partitioning transparently.

import torch import deepspeed from transformers import AutoModelForCausalLM, AutoTokenizer from torch.utils.data import DataLoader, DistributedSampler import argparse def create_argparser(): parser = argparse.ArgumentParser(description="DeepSpeed ZeRO-3 Training") parser.add_argument("--local_rank", type=int, default=-1) parser.add_argument("--model_name", type=str, default="meta-llama/Llama-2-7b-hf") parser.add_argument("--epochs", type=int, default=3) parser.add_argument("--deepspeed_config", type=str, default="ds_config.json") parser = deepspeed.add_config_arguments(parser) return parser def main(): parser = create_argparser() args = parser.parse_args() # Initialize distributed backend deepspeed.init_distributed() # Load model with ZeRO-3 init context for memory-efficient loading with deepspeed.zero.Init(config_dict_or_path=args.deepspeed_config): model = AutoModelForCausalLM.from_pretrained( args.model_name, torch_dtype=torch.bfloat16, use_cache=False, # disable KV cache for training ) # Enable gradient checkpointing model.gradient_checkpointing_enable() # Prepare dataset and dataloader tokenizer = AutoTokenizer.from_pretrained(args.model_name) train_dataset = load_training_data(tokenizer) # your dataset logic sampler = DistributedSampler(train_dataset) train_loader = DataLoader( train_dataset, batch_size=2, sampler=sampler, num_workers=4 ) # Initialize DeepSpeed engine model_engine, optimizer, _, scheduler = deepspeed.initialize( args=args, model=model, model_parameters=model.parameters(), config=args.deepspeed_config, ) # Training loop for epoch in range(args.epochs): sampler.set_epoch(epoch) model_engine.train() for step, batch in enumerate(train_loader): inputs = {k: v.to(model_engine.device) for k, v in batch.items()} # Forward pass — ZeRO-3 all-gathers params automatically outputs = model_engine(**inputs) loss = outputs.loss # Backward pass — ZeRO-3 reduce-scatters gradients model_engine.backward(loss) # Optimizer step — ZeRO-3 updates partitioned states model_engine.step() if step % 100 == 0 and model_engine.local_rank == 0: print(f"Epoch {epoch}, Step {step}, Loss: {loss.item():.4f}") # Save consolidated checkpoint model_engine.save_checkpoint("./checkpoints", tag=f"final") if __name__ == "__main__": main()

Launch Command

# Launch with deepspeed on 8 GPUs across 2 nodes deepspeed --num_gpus 4 --num_nodes 2 \ --hostfile hostfile.txt \ --master_addr node0 --master_port 29500 \ train.py --deepspeed_config ds_config.json
The deepspeed.zero.Init context manager is critical for ZeRO-3. Without it, loading a 70B model materializes all parameters on every GPU before partitioning — causing an OOM. The context manager intercepts torch.nn.Module parameter creation and immediately partitions them across GPUs, so peak memory never exceeds 16Ψ/N even during initialization.

Key Implementation Details

  • use_cache=False: KV caching is for inference; during training it wastes memory and conflicts with gradient checkpointing.
  • model_engine.backward(): Do NOT use loss.backward() directly — DeepSpeed's backward handles gradient scaling, accumulation, and ZeRO's reduce-scatter partitioning.
  • model_engine.step(): Replaces optimizer.step() + scheduler.step(). Handles gradient accumulation boundaries, clipping, and optimizer state synchronization.
  • save_checkpoint with gather: With stage3_gather_16bit_weights_on_model_save: true in the config, save_checkpoint consolidates all sharded parameters into a single checkpoint file readable by vanilla PyTorch.
Interview Tip: A common question is "Can you combine ZeRO-3 with tensor parallelism?" Yes — DeepSpeed supports ZeRO-3 + tensor parallelism (via Megatron-DeepSpeed) + pipeline parallelism simultaneously. This 3D parallelism is how models like Megatron-Turing NLG 530B were trained: tensor parallelism within a node (8 GPUs), pipeline parallelism across nodes, and ZeRO-3 across all data-parallel replicas.
