Why Data Order Matters for LLM Training

The conventional wisdom in deep learning is that random shuffling of training data is sufficient—and for many tasks, it is. But LLM pretraining operates at a scale where the statistical assumptions behind uniform random sampling break down. When training on trillions of tokens across heterogeneous domains, the order in which the model sees data has measurable effects on convergence speed, final validation loss, and downstream task performance.

Curriculum learning, introduced by Bengio et al. (2009), formalizes the intuition that presenting examples in a meaningful order—typically from easy to hard—can accelerate training and improve generalization. For LLMs, "easy" and "hard" map to concepts like data quality, syntactic complexity, domain specificity, and perplexity under a reference model.

Key Insight: A model trained on 1.4T tokens with curriculum ordering can match the downstream performance of a model trained on 1.8T tokens with random shuffling. Data ordering is a lever for compute efficiency, not just final quality.

The mechanism is rooted in loss landscape geometry. Early in training, the model's parameter space is highly sensitive to gradient direction. Exposing the model to a coherent, lower-variance signal (clean, well-structured text) during these initial steps helps it converge to a better basin. Later, once the model has learned basic syntax and common patterns, introducing noisier or more specialized data pushes it to generalize without destabilizing the loss landscape.

Empirically, several effects have been observed when comparing curriculum ordering to uniform random sampling:

  • Faster early convergence: Validation loss drops 15–25% faster in the first 20% of training tokens when easy-first ordering is used.
  • Improved domain transfer: Models trained with curriculum ordering show better zero-shot performance on out-of-distribution benchmarks (ARC, HellaSwag) compared to uniform baselines.
  • Reduced catastrophic forgetting: When domains are introduced gradually rather than all at once, the model retains knowledge of previously seen domains more effectively.
  • Sensitivity to ordering mistakes: Exposing the model to extremely noisy data early (e.g., unfiltered Common Crawl) can permanently impair downstream performance even if high-quality data is introduced later.
Warning: Curriculum learning is not universally beneficial. If the curriculum is too aggressive (e.g., withholding entire domains until late training), the model may overfit to the early distribution and struggle to adapt. The key is gradual transitions, not hard phase boundaries.

Defining "difficulty" for curriculum ordering in LLMs is non-trivial. Common proxies include:

  • Perplexity under a small reference model: Train a 125M parameter model on a balanced mix, then score all training documents. Low perplexity ⇒ "easy".
  • Document quality scores: Classifiers trained to predict whether text appears in curated sources (Wikipedia, textbooks) vs. raw web crawls.
  • Token-level entropy: Documents with lower average token entropy under a language model are considered more predictable and thus easier.
  • Syntactic complexity: Measures like average dependency parse depth or sentence length serve as rough proxies for linguistic difficulty.

Domain Mixing Ratios

Modern LLM training corpora are multi-domain by necessity. A typical training dataset includes web text (Common Crawl, RefinedWeb), code (GitHub, StackOverflow), books (Books3, Project Gutenberg), academic papers (Semantic Scholar, arXiv), conversational data (Reddit, forums), and curated reference text (Wikipedia, StackExchange). The mixing ratio—how much of each domain is sampled per batch—is one of the most consequential hyperparameters in LLM training.

The choice of mixing ratio involves several competing objectives. Code data improves reasoning and structured output (chain-of-thought prompting benefits from code exposure), but too much code can degrade natural language fluency. Academic papers improve factual accuracy but are stylistically narrow. Web data provides breadth and diversity but carries noise, bias, and repetition. Books contribute long-range coherence and narrative understanding but may introduce copyright-adjacent patterns.

A principled approach to choosing mixing ratios considers three factors:

  1. Domain utility: How much does each additional token from domain d improve performance on a target benchmark suite? This is measured by training probe models on ablated mixes.
  2. Diminishing returns: Each domain has a saturation point beyond which additional tokens provide negligible benefit. Code saturates faster than web text for most benchmarks.
  3. Epoch count: If a domain is small (e.g., Wikipedia at ~4B tokens), upsampling beyond 4–5 epochs causes memorization and degrades generalization. The mixing ratio must respect the natural size of each domain.
Rule of thumb: No domain should be repeated more than 4 epochs during pretraining. If your target mix requires 10% Wikipedia but you're training on 2T tokens, that's 200B tokens from a ~4B token source—50 epochs. This will cause severe memorization. Cap the repetition and redistribute the budget.

Data Quality Annealing

Data quality annealing is the practice of gradually increasing data quality over the course of training. The model begins with a broad, diverse, but noisier data distribution and transitions to a curated, high-quality subset in the final phase. This mirrors the learning rate annealing strategy: just as we decay the learning rate to fine-tune convergence, we anneal data quality to sharpen the model's final knowledge.

The theoretical motivation is twofold. First, early in training, the model benefits from coverage—seeing a wide range of text patterns, styles, and domains to build robust representations. The signal-to-noise ratio matters less because the model is learning coarse-grained features (tokenization patterns, common bigrams, basic syntax) that are present even in noisy data. Second, in the late phase, the model is fine-tuning its representations, and exposure to high-quality data has an outsized impact on the final loss and downstream task accuracy.

The annealing schedule can be formalized as a time-varying quality filter threshold. Let q(x) be a quality score for document x (e.g., output of a quality classifier). At training step t, we sample documents satisfying q(x) ≥ τ(t), where τ(t) is a monotonically increasing threshold function:

  • Linear annealing: τ(t) = τ0 + (τ1 − τ0) · (t / T)
  • Cosine annealing: τ(t) = τ0 + (τ1 − τ0) · (1 − cos(π t / T)) / 2
  • Step annealing: τ(t) = τk for t ∈ [Tk, Tk+1), with discrete quality tiers
import math class QualityAnnealingSchedule: # Manages the quality threshold over training steps def __init__(self, tau_start, tau_end, total_steps, schedule=None): self.tau_start = tau_start self.tau_end = tau_end self.total_steps = total_steps self.schedule = schedule or "cosine" def get_threshold(self, step): # Returns the quality threshold at the given training step progress = min(step / self.total_steps, 1.0) if self.schedule == "linear": return self.tau_start + (self.tau_end - self.tau_start) * progress elif self.schedule == "cosine": return self.tau_start + (self.tau_end - self.tau_start) * ( 1.0 - math.cos(math.pi * progress) ) / 2.0 elif self.schedule == "step": # Three discrete tiers: low, medium, high quality if progress < 0.5: return self.tau_start elif progress < 0.8: return (self.tau_start + self.tau_end) / 2.0 else: return self.tau_end def filter_batch(self, documents, quality_scores, step): # Filter documents that meet the current quality threshold threshold = self.get_threshold(step) return [ doc for doc, score in zip(documents, quality_scores) if score >= threshold ] # Example: start accepting everything (tau=0.0), end with top 20% (tau=0.8) scheduler = QualityAnnealingSchedule( tau_start=0.0, tau_end=0.8, total_steps=500000, schedule="cosine" )

The practical challenge with quality annealing is maintaining sufficient throughput. As τ(t) increases, fewer documents pass the filter, which can create data starvation—the data loader cannot fill batches fast enough. Two mitigations are common:

  1. Pre-stratified shards: Partition the dataset into quality tiers at preprocessing time. Each tier is stored in separate shards, and the data loader simply adjusts the sampling probability across tiers rather than filtering on the fly.
  2. Overprovisioned buffers: The data loader maintains a large buffer (10–100× batch size) and applies quality filtering within the buffer. This amortizes the filtering cost but introduces a latency-quality tradeoff.
Pitfall: Aggressive quality annealing in the final 5% of training (e.g., using only Wikipedia and textbooks) can cause the model to "forget" conversational patterns and code syntax. The annealing target should still be a mix of domains, just a higher-quality slice of each domain.

Dynamic Data Mixing with Validation Loss

Static mixing ratios are suboptimal because the model's learning dynamics change over training. A domain that provides high-gradient signal early (e.g., code) may become redundant later, while another domain (e.g., academic text) may become increasingly valuable as the model matures. Dynamic data mixing adjusts domain weights in real-time based on feedback from domain-specific validation losses.

The DoReMi algorithm (Xie et al., 2023) formalizes this as a distributionally robust optimization problem. The idea is to find domain weights that minimize the worst-case excess loss across all domains relative to a reference model. The algorithm proceeds in two stages:

  1. Stage 1 – Train a small proxy model: Train a 280M parameter model using Group DRO (Distributionally Robust Optimization) over the training domains. At each step, the domain with the highest excess loss (relative to a baseline) receives increased weight. The final domain weights from this proxy model capture the optimal reweighting.
  2. Stage 2 – Train the full model: Use the domain weights discovered by the proxy model to set the sampling distribution for the full-scale training run. These weights are static during the main run but were discovered dynamically.

The key equation in DoReMi is the excess loss computation. For domain d at step t:

excessd(t) = Ld(θt) − Ld(θref)

where Ld(θt) is the average loss on domain d for the current model and Ld(θref) is the loss for a reference model (typically trained on the uniform mix). Domains with high excess loss are "underserved" and receive higher sampling weight.

import numpy as np class DoReMiReweighter: # Implements the DoReMi domain reweighting algorithm def __init__(self, domain_names, reference_losses, eta=1.0, smooth=0.1): self.domain_names = domain_names self.n_domains = len(domain_names) self.reference_losses = np.array(reference_losses) self.eta = eta # learning rate for weight updates self.smooth = smooth # smoothing to prevent domain starvation # Initialize uniform weights self.log_weights = np.zeros(self.n_domains) def compute_excess_loss(self, current_losses): # Excess loss per domain relative to reference model excess = np.array(current_losses) - self.reference_losses return np.maximum(excess, 0.0) def update_weights(self, current_losses): # Multiplicative weight update using exponentiated gradient excess = self.compute_excess_loss(current_losses) self.log_weights += self.eta * excess # Normalize to get domain probabilities weights = np.exp(self.log_weights - np.max(self.log_weights)) weights = weights / weights.sum() # Apply smoothing to prevent any domain from being starved weights = (1.0 - self.smooth) * weights + self.smooth / self.n_domains return weights def get_sampling_distribution(self, current_losses): # Returns the domain sampling probabilities weights = self.update_weights(current_losses) return {name: w for name, w in zip(self.domain_names, weights)} # Example usage with six training domains reweighter = DoReMiReweighter( domain_names=["web", "code", "books", "arxiv", "wiki", "stackexchange"], reference_losses=[2.8, 1.5, 2.3, 2.1, 1.9, 2.0], eta=0.5, smooth=0.05 ) # After evaluating current model on domain validation sets: current = [2.9, 1.3, 2.5, 2.4, 1.8, 2.2] dist = reweighter.get_sampling_distribution(current) # Result: domains with high excess loss (arxiv, stackexchange) get upweighted

Static Mixing

  • Fixed domain weights throughout training
  • Simpler to implement and reproduce
  • Requires expensive hyperparameter sweeps
  • Cannot adapt to model's changing needs
  • Used by: Llama 2, PaLM, GPT-3

Dynamic Mixing (DoReMi)

  • Domain weights adjust based on validation loss
  • Requires proxy model + domain validation sets
  • Automatically finds near-optimal ratios
  • 8–15% improvement on worst-domain perplexity
  • Used by: DoReMi, DRO-LM, Online DM

A practical variant avoids the two-stage proxy approach by performing online reweighting during the main training run itself. Every K steps (e.g., K=1000), the trainer evaluates the model on held-out domain validation sets and recomputes the domain weights. The challenge is that the validation evaluation adds overhead—typically 2–5% of total training FLOPS if K is chosen carefully. The benefit is a single-stage process that adapts continuously without requiring a separate proxy training run.

Curriculum Phases and Implementation

A production curriculum typically divides training into 3–4 discrete phases, each with a distinct data composition and purpose. The phase boundaries align with learning rate milestones and often correspond to natural "checkpointing" boundaries for model evaluation.

The implementation of multi-phase curriculum training requires two core components: a DataMixer that manages per-domain sampling weights and iterators, and a CurriculumScheduler that determines the weights at each training step based on the current phase.

import numpy as np from typing import Dict, List, Optional, Iterator import bisect class DataMixer: # Manages multi-domain data sampling with configurable weights def __init__(self, domain_iterators: Dict[str, Iterator], seed: int = 42): self.domains = list(domain_iterators.keys()) self.iterators = domain_iterators self.rng = np.random.default_rng(seed) self.weights = np.ones(len(self.domains)) / len(self.domains) self.tokens_consumed = {d: 0 for d in self.domains} self.epoch_counts = {d: 0 for d in self.domains} def set_weights(self, weights: Dict[str, float]): # Update domain sampling probabilities (must sum to 1) w = np.array([weights.get(d, 0.0) for d in self.domains]) if w.sum() == 0: raise ValueError("Weights must not all be zero") self.weights = w / w.sum() def sample_batch(self, batch_size: int, seq_len: int = 2048): # Sample a batch with documents drawn according to domain weights domain_counts = self.rng.multinomial(batch_size, self.weights) batch = [] for domain, count in zip(self.domains, domain_counts): for _ in range(count): try: doc = next(self.iterators[domain]) except StopIteration: # Domain exhausted: reset iterator (new epoch) self.iterators[domain] = self._reset_iterator(domain) self.epoch_counts[domain] += 1 doc = next(self.iterators[domain]) batch.append(doc) self.tokens_consumed[domain] += seq_len self.rng.shuffle(batch) return batch def _reset_iterator(self, domain: str) -> Iterator: # Placeholder: in production, re-open shard files with new shuffle raise NotImplementedError("Implement per-domain shard reset") def get_statistics(self): # Return current consumption stats for monitoring return { "tokens_consumed": dict(self.tokens_consumed), "epoch_counts": dict(self.epoch_counts), "current_weights": {d: w for d, w in zip(self.domains, self.weights)}, }
class CurriculumScheduler: # Defines curriculum phases and computes domain weights per step def __init__(self, total_steps: int, phases: List[Dict]): # phases: list of {end_fraction, weights} dicts # end_fraction: training progress at which this phase ends (0.0-1.0) # weights: dict mapping domain name to sampling weight self.total_steps = total_steps self.phases = sorted(phases, key=lambda p: p["end_fraction"]) self.boundaries = [p["end_fraction"] for p in self.phases] def get_weights(self, step: int) -> Dict[str, float]: # Linearly interpolate between adjacent phase weight configs progress = min(step / self.total_steps, 1.0) # Find current phase index idx = bisect.bisect_right(self.boundaries, progress) idx = min(idx, len(self.phases) - 1) if idx == 0 or progress <= self.boundaries[0]: return self.phases[0]["weights"] # Interpolate between phase[idx-1] and phase[idx] prev = self.phases[idx - 1] curr = self.phases[idx] lo, hi = prev["end_fraction"], curr["end_fraction"] alpha = (progress - lo) / (hi - lo) if hi > lo else 1.0 domains = set(prev["weights"].keys()) | set(curr["weights"].keys()) interpolated = {} for d in domains: w0 = prev["weights"].get(d, 0.0) w1 = curr["weights"].get(d, 0.0) interpolated[d] = w0 + alpha * (w1 - w0) return interpolated # ---- Example: 4-phase curriculum for 500k steps ---- scheduler = CurriculumScheduler( total_steps=500000, phases=[ { # Phase 1: Warmup on diverse web data "end_fraction": 0.10, "weights": {"web": 0.65, "code": 0.15, "books": 0.12, "curated": 0.08} }, { # Phase 2: Balanced main training "end_fraction": 0.60, "weights": {"web": 0.45, "code": 0.22, "books": 0.18, "curated": 0.15} }, { # Phase 3: Quality annealing - increase curated weight "end_fraction": 0.90, "weights": {"web": 0.30, "code": 0.22, "books": 0.22, "curated": 0.26} }, { # Phase 4: Final polish with high-quality data "end_fraction": 1.00, "weights": {"web": 0.18, "code": 0.18, "books": 0.22, "curated": 0.42} }, ] ) # Integration: update DataMixer weights every 1000 steps # for step in range(total_steps): # if step % 1000 == 0: # mixer.set_weights(scheduler.get_weights(step)) # batch = mixer.sample_batch(batch_size=2048)
Implementation note: The CurriculumScheduler uses linear interpolation between phase boundaries. This ensures smooth transitions that avoid sudden distribution shifts, which can cause loss spikes. In practice, phase transitions over 2–5% of total training steps produce the most stable training dynamics.

Practical Considerations

Implementing curriculum learning at scale introduces several engineering challenges that go beyond algorithm design. The data pipeline must support dynamic weight changes without stalling the GPU cluster, and the entire process must remain reproducible across restarts.

Batch Construction at Scale

When training on hundreds of GPUs with data parallelism, batch construction must be deterministic and balanced. Each data-parallel rank constructs its own micro-batch, but all ranks must sample from the same global domain distribution. The standard approach is to use a shared PRNG seed incremented by the global step count, ensuring that each rank generates independent but statistically consistent samples. The domain weights are broadcast from rank 0 whenever they change.

def construct_distributed_batch( rank: int, world_size: int, step: int, global_batch_size: int, mixer: DataMixer, scheduler: CurriculumScheduler ): # Each rank constructs its micro-batch from the global distribution micro_batch_size = global_batch_size // world_size # Deterministic seed ensures reproducibility across restarts rank_seed = step * world_size + rank mixer.rng = np.random.default_rng(rank_seed) # Update weights from curriculum scheduler if step % 1000 == 0: weights = scheduler.get_weights(step) mixer.set_weights(weights) batch = mixer.sample_batch(micro_batch_size) return batch

Interaction with Learning Rate Schedule

Curriculum phases interact with the learning rate schedule in non-obvious ways. The warmup phase of curriculum learning (diverse, easy data) should ideally overlap with the LR warmup, because the model's gradient magnitudes are largest during this period and the easy data provides a stable gradient signal. Conversely, the quality annealing phase should coincide with the LR decay phase: as the learning rate drops, the model becomes more sensitive to data quality, and high-quality data ensures the final parameter updates are meaningful.

Aligned Schedule (Recommended)

  • LR warmup ↔ Curriculum warmup (diverse data)
  • LR plateau ↔ Main training phase (balanced mix)
  • LR cosine decay ↔ Quality annealing phase
  • Final low LR ↔ Polish with curated data

Both schedules cooperate: stable gradients during warmup, precise updates during annealing.

Misaligned Schedule (Avoid)

  • LR warmup + noisy web data → unstable gradients
  • High LR + curated data → overfitting to small domain
  • LR decay + diverse noisy data → wasted final capacity
  • Leads to 3–8% worse downstream performance

Conflicting signals cause the model to waste compute on low-value gradient updates.

Reproducibility and Checkpointing

Curriculum-aware training must checkpoint not only model parameters and optimizer state but also the data pipeline state: current phase, domain weights, per-domain shard positions, PRNG states, and tokens consumed per domain. Without this, resuming from a checkpoint places the model back at the beginning of the data sequence for all domains, effectively replaying data and destroying the curriculum ordering.

def save_curriculum_checkpoint( step, model, optimizer, mixer, scheduler, path ): # Save full training state including data pipeline checkpoint = { "step": step, "model_state": model.state_dict(), "optimizer_state": optimizer.state_dict(), "curriculum_state": { "current_weights": mixer.get_statistics(), "phase_config": scheduler.phases, "tokens_consumed": mixer.tokens_consumed, "epoch_counts": mixer.epoch_counts, "rng_state": mixer.rng.bit_generator.state, }, } torch.save(checkpoint, path) def load_curriculum_checkpoint(path, model, optimizer, mixer, scheduler): # Restore full state including curriculum position checkpoint = torch.load(path) model.load_state_dict(checkpoint["model_state"]) optimizer.load_state_dict(checkpoint["optimizer_state"]) # Restore data pipeline state cur_state = checkpoint["curriculum_state"] mixer.tokens_consumed = cur_state["tokens_consumed"] mixer.epoch_counts = cur_state["epoch_counts"] mixer.rng.bit_generator.state = cur_state["rng_state"] # Fast-forward domain iterators to correct shard positions for domain in mixer.domains: consumed = cur_state["tokens_consumed"][domain] mixer._seek_iterator(domain, consumed) return checkpoint["step"]

Monitoring and Observability

During curriculum-aware training, standard loss curves are insufficient for understanding model behavior. Essential metrics to log per step include:

  • Per-domain training loss: Track loss for each domain independently to detect when a domain is being under- or over-served by the current mix.
  • Per-domain validation loss: Evaluated every K steps on held-out domain-specific validation sets. This is the feedback signal for dynamic reweighting.
  • Effective domain ratios: The actual fraction of tokens consumed from each domain (may differ from target weights due to shard boundaries and padding).
  • Epoch counts per domain: Tracks how many times each domain has been fully iterated. Alert if any domain exceeds 4 epochs.
  • Curriculum phase and interpolation alpha: Log the current phase and interpolation progress to correlate with loss dynamics.
Production tip: Log domain weights and per-domain losses to a time-series database (e.g., Prometheus + Grafana, or Weights & Biases). Set alerts for domain loss divergence (>20% deviation from running average) and epoch count thresholds. These signals catch data pipeline bugs (stuck iterators, corrupt shards) before they waste significant compute.

Scaling to Multi-Trillion Token Runs

At the scale of 2T+ token training runs, the curriculum infrastructure must handle petabyte-scale data with millisecond-level per-step overhead. Key design decisions include:

  • Pre-tokenized shards: All data is tokenized and packed into fixed-length sequences during preprocessing. The data loader simply reads binary chunks, avoiding any on-the-fly tokenization overhead.
  • Domain-partitioned storage: Each domain is stored in a separate directory of shards, enabling independent iteration and reshuffling per domain. Shard sizes are chosen to align with the file system's optimal I/O block size (typically 256MB–1GB).
  • Weighted shard sampling: Instead of per-document sampling, the mixer selects entire shards with probability proportional to domain weights. This reduces the overhead of per-document domain tracking to per-shard tracking.
  • Prefetch pipeline: A multi-threaded prefetch pipeline reads shards from distributed storage (S3, GCS) into a local NVMe buffer 2–3 steps ahead of the current training step. Weight changes propagate with a 1–2 step delay, which is acceptable for gradual curriculum transitions.
Common failure mode: When the curriculum scheduler significantly downweights a domain, the data loader may have already prefetched shards from that domain. Implement a drain-and-refill mechanism that discards prefetched shards from downweighted domains when weights change by more than 10% in a single update. This prevents stale data from contaminating the post-transition batches.
