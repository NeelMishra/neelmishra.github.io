Data Dedup: MinHash, Exact, Fuzzy

MLOps Series LLM Training Infrastructure

Large language model training is fundamentally a data problem. Before a single gradient is computed, the quality of the training corpus dictates model capability. Among all data quality dimensions—toxicity, language ID, quality filtering—deduplication consistently delivers the largest measurable improvement in downstream performance per unit of engineering effort. This post covers every major dedup technique from exact hashing to probabilistic near-duplicate detection, with the math, implementation details, and scaling strategies needed to run dedup at web scale.

Why Deduplication Matters

Web crawls are massively redundant. Common Crawl snapshots contain 30–60% duplicate content at the document level, and up to 70% at the paragraph level when accounting for boilerplate headers, footers, and navigation text. This redundancy directly damages model training through three mechanisms:

1. Memorization over generalization. When a model sees identical passages hundreds of times, it memorizes them verbatim rather than learning generalizable patterns. Lee et al. (2022) showed that deduplicating C4 reduced verbatim memorization by 10× while improving perplexity. The model allocates capacity to rote recall instead of transferable representations.

2. Benchmark contamination. Training data overlapping with evaluation benchmarks inflates reported metrics. The GPT-3 paper flagged this issue: a 13-gram overlap analysis revealed contamination across multiple benchmarks. Without aggressive dedup, you cannot trust your own evals.

3. Wasted compute. Every duplicate token consumes the same FLOPs as a novel token but provides near-zero marginal information. At $2–5M per training run, processing 40% duplicate data wastes $800K–$2M in compute alone. The ROI on dedup infrastructure is enormous.

Key insight: Deduplication is not just data cleaning—it is a training efficiency multiplier. The Chinchilla scaling laws assume each token provides independent information. Duplicates violate this assumption, effectively reducing your dataset size without reducing your compute budget.

Empirical evidence. The RefinedWeb paper (Penedo et al., 2023) demonstrated that aggressive dedup of Common Crawl yielded a 5T-token corpus whose quality matched curated datasets like The Pile. Deduplication alone closed the gap between web crawl data and hand-curated corpora. The Dolma dataset (AI2) reported similar findings: fuzzy dedup removed an additional 30% of data beyond exact dedup, and models trained on the deduplicated corpus showed consistent gains across 12 downstream tasks.

Exact Deduplication

Exact dedup is the first and cheapest pass over any corpus. It catches verbatim copies—pages mirrored across domains, scraped RSS feeds, syndicated news articles, and duplicated boilerplate. There are several granularity levels, each with distinct trade-offs.

URL-Level Dedup

The simplest dedup: normalize URLs (strip query parameters, fragments, trailing slashes, lowercase the hostname) and keep only the first occurrence. This catches mirrors and CDN duplicates but misses content-identical pages hosted at different URLs. It is a preprocessing filter, not a substitute for content-based dedup.

Document-Level SHA-256 Hashing

Hash the full document text (after stripping HTML/boilerplate) with SHA-256. Two documents with identical hashes are identical. The collision probability for SHA-256 is 1/2256—effectively zero even at trillion-document scale. This is O(n) in documents and trivially parallelizable.

import hashlib from typing import Dict, Set def exact_dedup_sha256(documents: list[str]) -> list[str]: # O(n) exact deduplication using SHA-256 content hashing seen: Set[str] = set() unique: list[str] = [] for doc in documents: # Normalize whitespace before hashing normalized = " ".join(doc.split()) h = hashlib.sha256(normalized.encode("utf-8")).hexdigest() if h not in seen: seen.add(h) unique.append(doc) return unique

Paragraph-Level Exact Match

Many documents share paragraphs without being fully identical—boilerplate disclaimers, cookie consent text, navigation menus that survive extraction. Hashing individual paragraphs (split on double newlines) and removing paragraphs seen more than k times catches this. The threshold k must be tuned: setting it too low removes legitimately common phrases; setting it too high misses boilerplate. Typical values range from 5–100 depending on corpus size.

Bloom Filters for Memory-Efficient Dedup

Storing SHA-256 hashes for billions of URLs in a Python set requires ~64 bytes per entry (hash string + set overhead). For 10B URLs that is 640 GB of RAM. A Bloom filter provides a probabilistic alternative with controllable false-positive rate and dramatically lower memory.

For a target false-positive rate p with n elements, the optimal Bloom filter uses m = −n ln(p) / (ln 2)² bits and k = (m/n) ln 2 hash functions. For n = 10B and p = 0.001: m ≈ 143.7 billion bits ≈ 17 GB—a 37× memory reduction over a hash set.

import math import mmh3 from bitarray import bitarray class BloomFilter: def __init__(self, n_items: int, fp_rate: float = 0.001): # Compute optimal size and hash count self.size = int(-n_items * math.log(fp_rate) / (math.log(2) ** 2)) self.k = int((self.size / n_items) * math.log(2)) self.bits = bitarray(self.size) self.bits.setall(False) def _hashes(self, item: str) -> list[int]: # Double hashing: h_i = h1 + i*h2 mod m h1 = mmh3.hash(item, seed=0) % self.size h2 = mmh3.hash(item, seed=42) % self.size return [(h1 + i * h2) % self.size for i in range(self.k)] def add(self, item: str): for pos in self._hashes(item): self.bits[pos] = True def __contains__(self, item: str) -> bool: return all(self.bits[pos] for pos in self._hashes(item))
Warning: Bloom filters have false positives but never false negatives. In the dedup context, a false positive means discarding a unique document. At 0.1% FP rate across 10B docs, you lose ~10M documents unnecessarily. For most LLM training corpora this is acceptable, but for small high-value datasets, use exact hash sets instead.

MinHash & LSH for Near-Duplicate Detection

Exact dedup misses the most damaging class of duplicates: near-copies that differ only in whitespace, formatting, timestamps, or minor edits. Two news articles from different outlets covering the same press release may share 90% of their content but have different SHA-256 hashes. MinHash with Locality-Sensitive Hashing (LSH) detects these efficiently.

Jaccard Similarity and Shingling

The Jaccard similarity between two sets A and B is J(A, B) = |A ∩ B| / |A ∪ B|. We represent documents as sets of n-gram shingles (contiguous word sequences). For character-level shingles, typical values are n=5–13; for word-level, n=3–5. Word 5-grams are the standard for LLM training dedup (used by BigScience/ROOTS, RefinedWeb, and GPT-3).

MinHash Signature Generation

Computing exact Jaccard for all O(n²) document pairs is infeasible. MinHash provides an unbiased estimator: for a random permutation π of the universe, the minimum hash value over a set's elements has the property that Pr[min(π(A)) = min(π(B))] = J(A, B). By using k independent hash functions, we build a signature of k values per document where the fraction of matching positions estimates Jaccard similarity.

import hashlib from typing import List, Set, Tuple class MinHashSignature: def __init__(self, num_perm: int = 128): self.num_perm = num_perm # Generate random hash parameters: h(x) = (ax + b) mod p self.MAX_HASH = (1 << 32) - 1 self.PRIME = 4294967311 # Mersenne prime > 2^32 import random self.a = [random.randint(1, self.PRIME - 1) for _ in range(num_perm)] self.b = [random.randint(0, self.PRIME - 1) for _ in range(num_perm)] def shingle(self, text: str, k: int = 5) -> Set[str]: # Extract word k-grams from text words = text.lower().split() return {" ".join(words[i:i+k]) for i in range(len(words) - k + 1)} def signature(self, shingles: Set[str]) -> List[int]: # Compute MinHash signature from shingle set sig = [self.MAX_HASH] * self.num_perm for s in shingles: h_val = int(hashlib.sha1(s.encode()).hexdigest(), 16) & self.MAX_HASH for i in range(self.num_perm): val = (self.a[i] * h_val + self.b[i]) % self.PRIME if val < sig[i]: sig[i] = val return sig

Locality-Sensitive Hashing (LSH) Banding

Even with MinHash, comparing all pairs of signatures is O(n²). LSH reduces this to near-linear time by hashing bands of the signature. Divide the k-length signature into b bands of r rows each (k = b × r). Two documents become candidates if they agree in all r rows within at least one band.

The probability that two documents with Jaccard similarity s become a candidate pair is: P(candidate) = 1 − (1 − sr)b. This is an S-curve that approximates a step function around the threshold t ≈ (1/b)1/r. The steepness of the curve controls the false-positive/false-negative trade-off.

Parameter selection example: With k = 128 permutations, b = 16 bands, r = 8 rows: threshold ≈ (1/16)1/8 ≈ 0.66. At s = 0.8: P = 1 − (1 − 0.8⁸)¹⁶ ≈ 0.998. At s = 0.5: P = 1 − (1 − 0.5⁸)¹⁶ ≈ 0.061. Excellent separation.

from collections import defaultdict from typing import List, Set, Tuple, Dict class LSH: def __init__(self, num_bands: int = 16, rows_per_band: int = 8): self.b = num_bands self.r = rows_per_band # Each band maps bucket_hash → set of doc IDs self.buckets: List[Dict[int, Set[int]]] = [ defaultdict(set) for _ in range(self.b) ] def insert(self, doc_id: int, signature: List[int]): # Split signature into b bands of r rows each for band_idx in range(self.b): start = band_idx * self.r band = tuple(signature[start:start + self.r]) bucket_key = hash(band) self.buckets[band_idx][bucket_key].add(doc_id) def query_candidates(self) -> Set[Tuple[int, int]]: # Return all candidate pairs from any shared bucket candidates: Set[Tuple[int, int]] = set() for band_idx in range(self.b): for bucket in self.buckets[band_idx].values(): if len(bucket) > 1: docs = sorted(bucket) for i in range(len(docs)): for j in range(i + 1, len(docs)): candidates.add((docs[i], docs[j])) return candidates
Choosing b and r: The product b × r = k (total permutations). Increasing r raises the threshold (more strict), reducing false positives but increasing false negatives. For LLM training dedup, a Jaccard threshold of 0.7–0.8 is standard. With k=128: b=20, r=6 gives threshold ≈ 0.72; b=16, r=8 gives threshold ≈ 0.66. Profile your corpus to choose.

Suffix Array Deduplication

MinHash operates at the document level—it identifies pairs of similar documents. But training corpora also contain substring-level duplication: identical paragraphs, copied code snippets, or templated text that appears across otherwise distinct documents. Suffix array dedup (Lee et al., 2022) addresses this by finding repeated substrings of any length across the entire corpus.

Suffix Array Construction

A suffix array SA for a string S of length n is a sorted array of all n suffixes. Construction takes O(n) time with algorithms like SA-IS. For a corpus concatenated into a single string (with document boundary markers), the suffix array reveals all repeated substrings: adjacent entries in SA sharing a long common prefix indicate duplicated content.

The key data structure is the LCP array (Longest Common Prefix): LCP[i] stores the length of the longest common prefix between SA[i] and SA[i-1]. Any LCP value exceeding a threshold T (typically 50–100 tokens) indicates a duplicated substring worth removing.

import numpy as np def find_duplicate_substrings(text: str, min_length: int = 100) -> list: # Build suffix array and LCP array # In production, use pysuffixarray or divsufsort bindings sa = build_suffix_array(text) # O(n) via SA-IS lcp = build_lcp_array(text, sa) # O(n) via Kasai's algorithm duplicates = [] for i in range(1, len(lcp)): if lcp[i] >= min_length: # Found a duplicated substring of length lcp[i] start_1 = sa[i - 1] start_2 = sa[i] # Skip if both positions are in the same document if get_doc_id(start_1) != get_doc_id(start_2): duplicates.append((start_1, start_2, lcp[i])) return duplicates

Scaling with Disk-Based Suffix Arrays

For multi-terabyte corpora, the suffix array and LCP array cannot fit in memory. The approach is to shard the corpus, build suffix arrays per shard (on disk using memory-mapped files), then merge results. Lee et al. used a sliding-window approach: build the suffix array for a window of text, extract duplicates, slide the window. This trades O(n²) worst-case merge cost for practical O(n log n) with bounded memory.

Cost trade-off: Suffix array dedup is the most expensive technique—O(n) space and time where n is the total corpus length in characters, not documents. For a 1T-token corpus at ~4 bytes/token, the suffix array alone requires 4–8 TB of storage. Reserve this for final-pass dedup after MinHash has already eliminated document-level near-duplicates.

Scaling to Billions of Documents

Production dedup pipelines must process 10B+ documents from Common Crawl snapshots. No single machine can hold the data or the data structures in memory. The solution is distributed computing with careful partitioning.

MapReduce / Spark Architecture

Exact dedup: Map phase emits (hash, doc_id). Reduce groups by hash—groups with >1 member are duplicates. Keep the doc with the lowest crawl timestamp. This is a trivial shuffle-and-group operation, bottlenecked by I/O not compute.

MinHash LSH at scale: The map phase computes the MinHash signature and emits (band_hash, doc_id) for each of the b bands. The reduce phase groups by band_hash, generating candidate pairs. A second pass verifies candidates by computing exact Jaccard. The RefinedWeb pipeline uses this architecture on a 400-node Spark cluster, processing one Common Crawl snapshot (~3B pages) in ~6 hours.

Cross-Dataset vs. Within-Dataset Dedup

Within-dataset dedup removes duplicates inside a single source (e.g., one Common Crawl snapshot). Cross-dataset dedup removes overlaps between your training data and (a) held-out evaluation sets, (b) other training data sources, or (c) previously processed snapshots. Cross-dataset dedup requires building a global index—typically a Bloom filter or shared LSH hash table—that persists across pipeline runs.

Within-Dataset Dedup

  • Processes one data source at a time
  • Simpler orchestration—stateless pipeline
  • Misses cross-source duplicates
  • Sufficient for single-snapshot processing

Cross-Dataset Dedup

  • Maintains a persistent global index
  • Catches benchmark contamination
  • Essential for multi-epoch or multi-source training
  • Requires careful versioning of the index

Memory and Compute Budgets

For 10B documents with 128-perm MinHash signatures: signature storage = 10B × 128 × 4 bytes = 5.12 TB. LSH band hashes (16 bands × 10B entries × 8 bytes) = 1.28 TB. Total is dominated by signatures. Using 16-bit MinHash values (sufficient for most applications) halves this to ~2.56 TB. These fit comfortably in a distributed hash table across a 100-node cluster with 64 GB RAM each.

Impact on Training Quality

Deduplication is not merely a data hygiene step—it is one of the highest-leverage interventions for improving model quality. Multiple large-scale studies confirm this:

Lee et al. (2022) trained 1.3B-parameter models on C4 with and without dedup. The deduplicated model achieved 14.4 perplexity vs. 14.9 for the non-deduplicated baseline—a 3.4% improvement. More strikingly, exact memorization (measured by extracting training sequences from the model) dropped by 10× after dedup. The model generalized better because it was forced to learn patterns rather than memorize specific text.

Penedo et al. (2023, RefinedWeb) showed that URL dedup + MinHash fuzzy dedup reduced Common Crawl from ~30B to ~5B high-quality documents. Models trained on this deduplicated web data matched or exceeded curated datasets (The Pile, C4) across all benchmarks—demonstrating that dedup + scale > curation.

Benchmark contamination. The GPT-4 technical report and Llama 2 paper both report removing evaluation data from training corpora. Without this step, reported accuracies can be inflated by 2–5% on contaminated benchmarks. The n-gram overlap detection method (typically 8-gram or 13-gram matching) is now standard practice.

Exact Dedup

  • Speed: O(n) — fastest
  • Recall: Low — misses near-duplicates
  • Memory: ~64 bytes/doc (hash set) or ~2 bytes/doc (Bloom)
  • Use case: First pass, URL and document level
  • Typical removal: 20–40% of web crawl

MinHash + LSH

  • Speed: O(n × k) — moderate
  • Recall: High — catches ~90% of near-duplicates
  • Memory: ~512 bytes/doc (128 × 4-byte signature)
  • Use case: Primary fuzzy dedup pass
  • Typical removal: Additional 10–30% after exact dedup

Suffix Array

  • Speed: O(n) in corpus length — slowest at scale
  • Recall: Perfect for substring matches above threshold
  • Memory: ~8 bytes/char — enormous for large corpora
  • Use case: Final pass, substring-level dedup
  • Typical removal: Additional 5–15% of tokens

Recommended Pipeline

  • Stage 1: URL dedup (Bloom filter)
  • Stage 2: Exact document hash (SHA-256)
  • Stage 3: MinHash LSH (b=20, r=6, threshold ~0.72)
  • Stage 4: Suffix array (optional, for highest quality)
  • Stage 5: Benchmark contamination removal (13-gram)
Production recommendation: Run exact dedup first (cheapest, removes the most data), then MinHash LSH (catches near-duplicates), and optionally suffix array dedup for final quality. Always dedup against evaluation benchmarks as the last step. This pipeline removes 40–70% of raw web crawl data while preserving the highest-information-density documents.

Diminishing returns. There is a point where additional dedup removes more signal than noise. Extremely aggressive fuzzy dedup (e.g., Jaccard threshold 0.5) starts removing topically similar but informationally distinct documents. Monitor downstream task metrics and perplexity on a held-out set to calibrate the aggressiveness of each dedup stage. The goal is to remove redundancy, not topical coverage.
