3D Parallelism: Data + Tensor + Pipeline

MLOps Series LLM Training Infrastructure

Training frontier LLMs with hundreds of billions of parameters requires distributing computation across thousands of GPUs. No single parallelism strategy suffices: data parallelism hits memory walls, tensor parallelism hits communication walls across nodes, and pipeline parallelism suffers from bubble overhead at scale. 3D parallelism composes all three strategies simultaneously, mapping each dimension to the hardware topology to maximize throughput while keeping memory per GPU manageable.

Why 3D Parallelism?

Each parallelism dimension addresses a different bottleneck, but introduces its own limitation when used in isolation:

Data Parallelism Alone

Replicates the entire model on every GPU. Each rank processes a different micro-batch, then gradients are synchronized via all-reduce. Works when the model fits in one GPU's memory. For a 70B parameter model at fp16 (140 GB for weights alone, plus optimizer states ~4× under Adam → 560 GB), a single A100-80GB cannot hold the model. DP alone is impossible.

Tensor Parallelism Alone

Shards each layer's weight matrices across GPUs. For a TP degree of t, each GPU holds 1/t of every layer. This requires two all-reduce (or all-gather + reduce-scatter) operations per transformer layer in the forward pass and two more in the backward pass. The collective volume per layer is O(b × s × h) where b is micro-batch size, s is sequence length, and h is hidden dimension. This is feasible intra-node over NVLink (900 GB/s on DGX H100) but becomes a severe bottleneck when spanning nodes connected by InfiniBand (400 Gb/s ≈ 50 GB/s).

Pipeline Parallelism Alone

Assigns consecutive groups of layers to different GPUs. Communication is point-to-point: only activations between pipeline stages need to be transferred. However, pipeline bubbles waste compute. With p stages and m micro-batches, the bubble fraction is (p - 1) / m. For p = 64 and m = 128, the bubble is ~49%—nearly half the cluster idles.

3D Parallelism Combined

By composing DP × TP × PP, we keep each dimension's degree moderate. Typical: TP=8 (intra-node), PP=8 (inter-node), DP=16 (across node groups). The bubble fraction drops to (8 - 1) / m, TP communication stays within NVLink, and each DP replica only synchronizes gradients for its shard. Total GPUs = 8 × 8 × 16 = 1024.

Key Insight: 3D parallelism is not just about fitting the model—it is about mapping communication patterns to hardware bandwidth tiers. High-bandwidth intra-node links handle the chatty TP collectives, while lower-bandwidth inter-node links handle the sparse PP point-to-point and periodic DP all-reduce traffic.

Hardware Topology Mapping

Modern GPU clusters have a strict bandwidth hierarchy. The principle of 3D parallelism assignment is simple: assign the most communication-intensive parallelism to the fastest interconnect.

Bandwidth Tiers (DGX H100 Example)

Interconnect Bandwidth Latency Assigned Parallelism
NVSwitch (intra-node)900 GB/s bidirectional~1 μsTensor Parallelism
InfiniBand NDR (inter-node)400 Gb/s ≈ 50 GB/s~5 μsPipeline Parallelism
Cross-rack / spine networkShared, variable~10-50 μsData Parallelism

Mapping Strategy

Tensor Parallelism (TP) within each node: A DGX H100 has 8 GPUs connected by NVSwitch. TP degree = 8 means each GPU in the node holds a column/row shard of every layer. The all-reduce per layer transfers activations of size b × s × h (e.g., 4 × 2048 × 12288 × 2 bytes ≈ 192 MB) multiple times per layer. At 900 GB/s, this takes <1 ms per collective.

Pipeline Parallelism (PP) across nodes in the same rack: Consecutive pipeline stages are placed on adjacent nodes. P2P send/recv of activations only requires transferring one micro-batch's activation tensor: b × s × h × sizeof(dtype). For a micro-batch of size 1 with seq=2048, h=12288, fp16: 2048 × 12288 × 2 ≈ 50 MB per message. At 50 GB/s, this completes in ~1 ms.

Data Parallelism (DP) across racks: Gradient all-reduce happens once per training step (not per layer), and when combined with ZeRO Stage-1 or gradient compression, the volume becomes manageable. With DP degree d, each rank all-reduces Ψ/TP parameters' gradients. The ring all-reduce cost is 2(d-1)/d × (Ψ / TP) × sizeof(dtype).

Common Mistake: Placing TP across nodes. Even with InfiniBand 400 Gb/s, the per-layer all-reduce latency balloons from <1 ms to 4-5 ms. For a model with 80 layers, this adds 640-800 ms per training step in the forward pass alone—a 20-30% throughput loss compared to intra-node TP.

Communication Cost Analysis

Understanding the exact bytes transferred for each parallelism dimension is critical for predicting training throughput and identifying bottlenecks.

Data Parallelism: All-Reduce

Ring all-reduce on d ranks transfers each element exactly 2(d-1)/d times. For a model with Ψ total parameters split across TP=t, each DP rank holds Ψ/t parameters:

Cost_DP = 2 × (d - 1) / d × (Ψ / t) × sizeof(dtype)

Example: Ψ = 70B, t = 8, d = 8, dtype = bf16 (2 bytes)
Cost_DP = 2 × 7/8 × (70×10⁹ / 8) × 2 = 2 × 0.875 × 8.75×10⁹ × 2 ≈ 30.6 GB per step

Tensor Parallelism: All-Gather + Reduce-Scatter

Each transformer layer requires collectives in both the attention and MLP blocks. In Megatron-LM's column-then-row partitioning, the forward pass performs one all-gather (for the column-parallel linear) and one reduce-scatter (for the row-parallel linear) per sub-block, totaling 4 collective operations per layer (2 for attention, 2 for MLP):

Volume_per_collective = b × s × h × sizeof(dtype)
Cost_TP_per_layer = 4 × (t - 1) / t × b × s × h × sizeof(dtype)
Cost_TP_total = L_local × Cost_TP_per_layer

Example: b=4, s=2048, h=12288, t=8, L_local=10 (80 layers / PP=8), bf16
Per collective: 4 × 2048 × 12288 × 2 = 201 MB
Per layer: 4 × (7/8) × 201 MB = 703 MB
Total TP: 10 × 703 MB ≈ 6.9 GB per step

Pipeline Parallelism: Point-to-Point

PP sends activations between consecutive stages. With 1F1B scheduling and m micro-batches:

Activation_size = b_micro × s × h × sizeof(dtype)
PP_volume = m × Activation_size × 2 (forward + backward)

Example: b_micro=1, s=2048, h=12288, bf16, m=32
Per message: 1 × 2048 × 12288 × 2 = 50.3 MB
Total PP: 32 × 50.3 × 2 ≈ 3.2 GB per step
Overlap Opportunities: TP collectives can overlap with compute in the next sub-block. PP send/recv overlaps with compute on other micro-batches. DP all-reduce overlaps with the backward pass of non-final layers (gradient bucketing). Effective wall-clock communication is often 30-50% less than the raw volume suggests.

Communication Summary

Latency-Sensitive (TP)

4 collectives per layer × L layers in a critical path. Must use highest-bandwidth link (NVLink/NVSwitch). Latency directly adds to per-step time with limited overlap within a single layer.

Throughput-Sensitive (DP)

One large all-reduce per step. Can be fully overlapped with backward computation using gradient bucketing. Tolerates higher latency but needs sustained bandwidth for large models.

Choosing Parallelism Dimensions

Given a total GPU count N, we need to find DP × TP × PP = N such that: (1) the model fits in memory, (2) the pipeline bubble is acceptable, and (3) communication overhead is minimized.

Step 1: Set TP Based on Node Size

Almost always TP = GPUs per node (typically 8). Only reduce TP if the model is small enough that TP=8 causes each shard to be too small for efficient GEMM (e.g., a 7B model with h=4096 sharded 8 ways → 512-dim columns, which under-utilizes tensor cores).

Step 2: Set PP Based on Memory

Compute memory per GPU for a given PP degree:

Memory_per_GPU = (Ψ / (TP × PP)) × bytes_per_param ← model weights
+ (Ψ / (TP × PP)) × bytes_optimizer ← optimizer (12 for Adam fp32)
+ activations(b_micro, s, h, L/PP) ← activation memory
+ gradient_buffer ← gradient accumulation

For 70B model, TP=8, PP=4:
Weights: (70B / 32) × 2 = 4.375 GB
Optimizer: (70B / 32) × 12 = 26.25 GB
Activations (with recomp): ~10-15 GB
Gradients: ~4.375 GB
Total ≈ 45-50 GB → Fits in 80 GB A100 ✓

Step 3: Set DP = N / (TP × PP)

The remaining GPUs form the data parallel dimension. More DP means larger effective batch size. Verify that global_batch = DP × m × b_micro matches the desired training batch size.

Constraint: The number of layers L must be divisible by PP. For LLaMA-70B with 80 layers, valid PP values are {1, 2, 4, 5, 8, 10, 16, 20, 40, 80}. Choose the smallest PP that fits in memory to minimize the pipeline bubble.

Decision Flowchart

def choose_parallelism(num_gpus, model_params, gpu_memory, gpus_per_node): # Step 1: TP = node size (reduce if model is small) tp = gpus_per_node if model_params < 10e9: tp = min(4, gpus_per_node) # Smaller models don't need full TP # Step 2: Find minimum PP that fits memory for pp in sorted(divisors(num_layers)): mem = estimate_memory(model_params, tp, pp) if mem < gpu_memory * 0.9: # 90% utilization target break # Step 3: Remaining GPUs → data parallelism dp = num_gpus // (tp * pp) assert dp * tp * pp == num_gpus, "GPU count must be divisible" # Validate bubble overhead num_microbatches = global_batch // (dp * micro_batch) bubble = (pp - 1) / num_microbatches assert bubble < 0.05, "Pipeline bubble >5%, increase micro-batches" return dp, tp, pp

Configuration Examples by Model Size

Below are production-tested configurations for popular model sizes, assuming DGX A100-80GB or H100-80GB nodes with 8 GPUs each.

7B Parameters (e.g., LLaMA-2 7B)

# 7B fits on a single GPU with bf16, but TP helps for speed # 8 GPUs (1 node) — no PP needed config = { # Parallelism "tensor_model_parallel_size": 2, # TP=2 (model fits, but 2-way speeds up GEMM) "pipeline_model_parallel_size": 1, # PP=1 (model fits without sharding layers) "data_parallel_size": 4, # DP=4 (8 / (2×1) = 4) # Training "micro_batch_size": 4, "global_batch_size": 256, # 256 / 4 / 4 = 16 accumulation steps "seq_length": 4096, "bf16": True, }

13B Parameters (e.g., LLaMA-2 13B)

# 13B: weights=26GB (bf16), optimizer≈156GB → needs partitioning # 16 GPUs (2 nodes) config = { "tensor_model_parallel_size": 4, # TP=4 intra-node "pipeline_model_parallel_size": 2, # PP=2 (40 layers / 2 = 20 per stage) "data_parallel_size": 2, # DP=2 (16 / (4×2) = 2) "micro_batch_size": 2, "global_batch_size": 512, "seq_length": 4096, "bf16": True, "use_flash_attention": True, "recompute_granularity": "selective", # Save memory via activation recomputation }

70B Parameters (e.g., LLaMA-2 70B)

# 70B: weights=140GB, optimizer=840GB → heavy sharding # 64 GPUs (8 nodes) config = { "tensor_model_parallel_size": 8, # TP=8 (full node, required for memory) "pipeline_model_parallel_size": 4, # PP=4 (80 layers / 4 = 20 per stage) "data_parallel_size": 2, # DP=2 (64 / (8×4) = 2) "micro_batch_size": 1, "global_batch_size": 1024, # 1024 / 1 / 2 = 512 accumulation steps "num_micro_batches": 32, # Bubble = (4-1)/32 ≈ 9.4% "seq_length": 4096, "bf16": True, "use_flash_attention": True, "recompute_granularity": "full", # Full activation recomputation "distribute_saved_activations": True, }

175B Parameters (e.g., GPT-3 Scale)

# 175B: weights=350GB, optimizer=2.1TB → massive distribution # 1024 GPUs (128 nodes) config = { "tensor_model_parallel_size": 8, # TP=8 (full node) "pipeline_model_parallel_size": 16, # PP=16 (96 layers / 16 = 6 per stage) "data_parallel_size": 8, # DP=8 (1024 / (8×16) = 8) "micro_batch_size": 1, "global_batch_size": 1536, "num_micro_batches": 48, # Bubble = (16-1)/48 ≈ 31% — mitigate with interleaved "virtual_pipeline_model_parallel_size": 2, # Interleaved 1F1B halves bubble "seq_length": 2048, "bf16": True, "use_flash_attention": True, "recompute_granularity": "full", "distribute_saved_activations": True, "overlap_grad_reduce": True, # Overlap DP all-reduce with backward "overlap_param_gather": True, # For ZeRO-1 param all-gather }
Interleaved Pipeline: Setting virtual_pipeline_model_parallel_size = v assigns v non-contiguous chunks of layers to each stage instead of one contiguous block. This reduces the pipeline bubble from (p-1)/m to approximately (p-1)/(m×v). For the 175B config above: without interleaving bubble ≈ 31%, with v=2 bubble ≈ 15.6%.

Performance Optimization & MFU Analysis

Model FLOPS Utilization (MFU) is the gold standard metric for training efficiency. It measures what fraction of the hardware's peak FLOPS is actually spent on useful model computation:

MFU = Actual_TFLOPS / Peak_TFLOPS

Actual_TFLOPS = (6 × Ψ × B × S) / (time_per_step × 10¹²)

Where:
Ψ = model parameters (e.g., 70×10⁹)
B = global batch size (e.g., 1024)
S = sequence length (e.g., 4096)
time_per_step = wall-clock seconds per training step
6 = forward (2×) + backward (4×) multiply-accumulate factor
Peak_TFLOPS (H100 bf16) = 989.5 TFLOPS per GPU

MFU Breakdown by Loss Source

Loss Source Typical Impact Mitigation
PP bubble5-15%Interleaved schedule, more micro-batches
TP communication3-8%Keep TP intra-node, overlap with compute
DP all-reduce2-5%Gradient bucketing, overlap with backward
Activation recomputation10-25%Selective recomputation (only attention/norm)
Memory-bound ops5-10%Kernel fusion, FlashAttention
Data loading / stalls1-3%Prefetching, NVMe-based data loaders

Optimization Checklist

def optimize_3d_config(config): # 1. Enable communication overlap config["overlap_grad_reduce"] = True # Overlap DP gradient all-reduce config["overlap_param_gather"] = True # Overlap ZeRO param gather # 2. Reduce PP bubble if config["pipeline_model_parallel_size"] > 4: config["virtual_pipeline_model_parallel_size"] = 2 # Interleaved # Increase micro-batches until bubble < 5% pp = config["pipeline_model_parallel_size"] while (pp - 1) / config["num_micro_batches"] > 0.05: config["num_micro_batches"] += pp # Must be multiple of pp # 3. Memory optimization config["use_flash_attention"] = True config["recompute_granularity"] = "selective" # Recompute attention only # 4. Compute MFU target psi = config["model_params"] B = config["global_batch_size"] S = config["seq_length"] peak = config["num_gpus"] * 989.5e12 # H100 bf16 peak target_mfu = 0.50 # Aim for 50%+ MFU max_step_time = (6 * psi * B * S) / (target_mfu * peak) print(f"Target step time for {target_mfu:.0%} MFU: {max_step_time:.2f}s") return config
MFU vs. HFU: Hardware FLOPS Utilization (HFU) includes recomputation FLOPS in the numerator, giving higher numbers (often 55-65%). MFU is the more honest metric because it only counts FLOPS that contribute to the actual forward + backward model computation. When comparing systems, always confirm whether reported numbers are MFU or HFU.

3D parallelism remains the dominant strategy for training models from 10B to 1T+ parameters. The key engineering challenge is not the algorithm but the systems integration: correctly mapping parallelism dimensions to hardware topology, tuning micro-batch counts to minimize bubbles, overlapping communication with compute, and profiling to identify the true bottleneck. With careful configuration, MFU values of 50%+ are achievable even on thousand-GPU clusters.
