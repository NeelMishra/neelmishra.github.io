Checkpoint Management: Distributed, Async I/O

MLOps Series LLM Training Infrastructure

The Checkpoint Problem at Scale

Training a 70B-parameter model across 512 GPUs for weeks means that every minute of wasted compute costs real money—often $2–$4 per GPU-hour on cloud A100/H100 clusters. If a single node fails after 12 hours of training without a checkpoint, you lose that entire window: roughly $12,000–$24,000 in wasted compute. Checkpointing is the insurance policy against hardware faults, preemptions, and software crashes, but it comes with its own costs that must be carefully managed.

The core tension in checkpoint design is between checkpoint frequency and training throughput. A naive synchronous checkpoint of a 70B model in bf16 requires writing ~140 GB of model parameters plus an additional ~560 GB of optimizer states (Adam maintains first and second moment estimates, each the size of the model, in fp32). With mixed-precision master weights, total checkpoint size easily reaches 700 GB–1.2 TB per save.

Cost of Naive Checkpointing: A synchronous checkpoint that pauses all training to write 1 TB to a distributed filesystem at 10 GB/s aggregate bandwidth takes ~100 seconds. If you checkpoint every 15 minutes, you lose ~10% of your training throughput just to I/O. At $50/hour cluster cost, that's $120/day wasted on blocking writes.

The key metrics for evaluating checkpoint strategy are:

  • Mean Time Between Failures (MTBF): At scale, hardware failures occur every 2–8 hours. Node-level MTBF may be 30+ days, but across 64 nodes the cluster-level MTBF drops dramatically.
  • Checkpoint Overhead Ratio: Time spent checkpointing divided by total training time. Target < 2%.
  • Recovery Time Objective (RTO): Time from failure detection to resuming training. Includes loading checkpoint, rebuilding communication groups, and warming up.
  • Recovery Point Objective (RPO): Maximum acceptable training progress loss, measured in steps or tokens processed.

The diagram above illustrates the high-performance checkpoint pipeline: each GPU rank asynchronously copies its local shard from device memory to pinned CPU memory (or NVMe staging), then background threads write those buffers to distributed storage in parallel. This avoids the catastrophic bottleneck of gathering all parameters to rank 0.

Sharded State Dict Saving

In Fully Sharded Data Parallelism (FSDP), model parameters and optimizer states are sharded across all participating ranks. The naive approach to checkpointing—calling model.state_dict() which triggers an all-gather to consolidate the full model onto rank 0—is disastrous at scale. For a 70B model, this requires 140+ GB of peak memory on rank 0, likely causing an OOM crash.

The sharded state dict approach instead has each rank save only its local shard. PyTorch's FSDP provides StateDictType.SHARDED_STATE_DICT which preserves the sharded representation. Each rank writes a file like __0_0.distcp containing only its portion of the parameter and optimizer tensors.

Memory Overhead: Sharded saving requires zero additional GPU memory beyond the existing shard. The only overhead is pinned CPU memory for the staging buffer, which can be pre-allocated once and reused across checkpoints. For a 70B model on 512 GPUs, each rank holds ~280 MB of parameters and ~1.1 GB of optimizer states.
import torch import torch.distributed as dist from torch.distributed.fsdp import ( FullyShardedDataParallel as FSDP, StateDictType, ShardedStateDictConfig, ShardedOptimStateDictConfig, ) from torch.distributed.checkpoint import save, load from pathlib import Path import json, time def save_sharded_checkpoint( model: FSDP, optimizer: torch.optim.Optimizer, checkpoint_dir: str, step: int, rank: int, world_size: int, ) -> None: # Configure FSDP to yield sharded (not full) state dicts with FSDP.state_dict_type( model, StateDictType.SHARDED_STATE_DICT, state_dict_config=ShardedStateDictConfig(offload_to_cpu=True), optim_state_dict_config=ShardedOptimStateDictConfig(offload_to_cpu=True), ): model_state = model.state_dict() optim_state = FSDP.sharded_optim_state_dict(model, optimizer) # Each rank writes its own shard via DCP save_path = Path(checkpoint_dir) / f"step_{step}" state_dict = { "model": model_state, "optimizer": optim_state, } save(state_dict, checkpoint_dir=str(save_path)) # Rank 0 writes metadata for resharding compatibility if rank == 0: metadata = { "step": step, "world_size": world_size, "timestamp": time.time(), "fsdp_sharding_strategy": "FULL_SHARD", "dtype": "bfloat16", } with open(save_path / "metadata.json", "w") as f: json.dump(metadata, f, indent=2) dist.barrier() # Ensure all ranks finish before training resumes def load_sharded_checkpoint( model: FSDP, optimizer: torch.optim.Optimizer, checkpoint_dir: str, step: int, ) -> dict: load_path = str(Path(checkpoint_dir) / f"step_{step}") with FSDP.state_dict_type( model, StateDictType.SHARDED_STATE_DICT, ): state_dict = {"model": model.state_dict()} load(state_dict, checkpoint_dir=load_path) model.load_state_dict(state_dict["model"]) optim_state = {"optimizer": {}} load(optim_state, checkpoint_dir=load_path) optim_sd = FSDP.optim_state_dict_to_load( model, optimizer, optim_state["optimizer"] ) optimizer.load_state_dict(optim_sd) with open(Path(load_path) / "metadata.json") as f: metadata = json.load(f) return metadata

The critical detail here is offload_to_cpu=True: this ensures the state dict tensors are moved to pinned CPU memory during extraction, freeing GPU memory immediately. Without this flag, you double your GPU memory usage during checkpointing, which will OOM on most production configurations where GPU memory is fully utilized by activations and optimizer states.

Full State Dict (Rank 0 Gather)

  • All-gathers entire model to rank 0
  • Peak memory: full model size on rank 0
  • Single write bottleneck
  • Simple but does not scale beyond 13B
  • Recovery: direct load_state_dict()

Sharded State Dict (Per-Rank)

  • Each rank saves only its local shard
  • Peak memory: ~0 extra on GPU
  • Parallel writes saturate storage bandwidth
  • Scales to any model size
  • Recovery: requires DCP resharding if world_size changes

Async Checkpointing & Non-Blocking I/O

Even with sharded writes, a synchronous checkpoint blocks the training loop during the entire storage write. For a 70B model checkpoint of ~1 TB, writing at 12 GB/s aggregate bandwidth still takes ~80 seconds. With async checkpointing, we decouple the checkpoint I/O from the training computation using a two-phase pipeline:

  1. Phase 1 – Device-to-Host Copy (blocking, fast): Copy the sharded state dict from GPU memory to pre-allocated pinned CPU buffers using CUDA async memcpy. This takes 1–3 seconds for typical shard sizes. Training is paused only for this phase.
  2. Phase 2 – Background Write (non-blocking): A separate thread pool writes the CPU buffers to the distributed filesystem while the next training iteration proceeds. The GPU is fully utilized during this phase.
Double-Buffer Hazard: If training modifies parameters before the background write completes, the checkpoint will contain corrupted data. The solution is double-buffering: maintain two sets of pinned CPU buffers and alternate between them. Before starting a new checkpoint, wait for the previous background write to complete. This is why you'll see _prev_future tracking in production checkpoint managers.
import torch import threading import os from concurrent.futures import ThreadPoolExecutor, Future from typing import Optional, Dict from dataclasses import dataclass, field import time, logging log = logging.getLogger(__name__) @dataclass class CheckpointStats: staging_secs: float = 0.0 write_secs: float = 0.0 total_bytes: int = 0 class AsyncCheckpointManager: """Non-blocking checkpoint manager with CPU staging and background writes. Supports double-buffering to overlap write I/O of checkpoint N with training compute after checkpoint N is staged.""" def __init__( self, checkpoint_dir: str, rank: int, world_size: int, num_writers: int = 4, pinned_pool_bytes: int = 4 * 1024**3, # 4 GB pinned pool ): self.checkpoint_dir = checkpoint_dir self.rank = rank self.world_size = world_size self._executor = ThreadPoolExecutor(max_workers=num_writers) self._prev_future: Optional[Future] = None self._lock = threading.Lock() self._stats = CheckpointStats() # Pre-allocate pinned memory for double buffering self._buffers = [ torch.empty(pinned_pool_bytes, dtype=torch.uint8, pin_memory=True) for _ in range(2) ] self._buf_idx = 0 def save_async( self, state_dict: Dict[str, torch.Tensor], step: int, ) -> Future: # Block if previous write hasn't finished (back-pressure) if self._prev_future is not None: self._prev_future.result() # Wait for completion # Phase 1: Stage tensors from GPU to pinned CPU (blocking) t0 = time.perf_counter() cpu_state = {} for key, tensor in state_dict.items(): if tensor.is_cuda: cpu_state[key] = tensor.to("cpu", non_blocking=True) else: cpu_state[key] = tensor.clone() torch.cuda.synchronize() # Ensure all D2H copies complete self._stats.staging_secs = time.perf_counter() - t0 # Phase 2: Submit background write save_path = os.path.join( self.checkpoint_dir, f"step_{step}", f"rank_{self.rank}.pt", ) future = self._executor.submit( self._write_to_disk, cpu_state, save_path, step ) self._prev_future = future return future def _write_to_disk( self, cpu_state: dict, path: str, step: int ) -> CheckpointStats: t0 = time.perf_counter() os.makedirs(os.path.dirname(path), exist_ok=True) torch.save(cpu_state, path) self._stats.write_secs = time.perf_counter() - t0 self._stats.total_bytes = os.path.getsize(path) log.info( f"[Rank {self.rank}] Checkpoint step {step} written " f"in {self._stats.write_secs:.1f}s " f"({self._stats.total_bytes / 1e9:.2f} GB)" ) return self._stats def wait_for_completion(self) -> None: # Call at end of training or before evaluation if self._prev_future is not None: self._prev_future.result() def shutdown(self) -> None: self.wait_for_completion() self._executor.shutdown(wait=True)

In production systems like Meta's internal training stack and PyTorch DCP (torch.distributed.checkpoint), the async pipeline achieves <3 seconds of training pause for a 70B model checkpoint, compared to 80+ seconds for synchronous writes. The key insight is that D2H bandwidth on modern PCIe 5.0 / NVLink interconnects (32–64 GB/s) is vastly faster than storage bandwidth (5–20 GB/s), so the blocking phase is minimal.

PyTorch DCP Integration: As of PyTorch 2.3+, torch.distributed.checkpoint.async_save() provides first-class support for this pattern. It handles D2H staging, background writes, and barrier synchronization automatically. Use it instead of rolling your own async manager unless you need fine-grained control over the write pipeline.

Checkpoint Formats

The choice of serialization format has significant implications for checkpoint speed, safety, and interoperability. The three dominant formats in the LLM training ecosystem are PyTorch native (.pt/.bin), SafeTensors, and the DCP format.

PyTorch Native (.pt)

  • Uses Python pickle under the hood
  • Security risk: arbitrary code execution on load
  • Supports arbitrary Python objects (optimizers, schedulers)
  • No memory mapping—entire file loaded into RAM
  • De facto standard but being superseded

SafeTensors (.safetensors)

  • Zero-copy memory mapping for fast loads
  • No pickle: impossible to execute arbitrary code
  • Header-based metadata allows partial tensor loading
  • ~2–5x faster load times on large checkpoints
  • Tensor-only: cannot store optimizer states natively

The DCP (Distributed Checkpoint) format used by torch.distributed.checkpoint deserves special attention. It stores each rank's shard as a separate file with a global metadata file (.metadata) that maps logical tensor names to physical file locations and byte offsets. This enables elastic resharding—the metadata file allows any number of ranks to reconstruct the full model by reading the appropriate byte ranges from the shard files.

# DCP metadata structure (simplified) # .metadata file maps logical names → physical locations { "state_dict_metadata": { "model.layers.0.attention.wq.weight": { "chunks": [ {"offsets": [0, 0], "sizes": [4096, 2048], "file": "__0_0.distcp", "byte_offset": 0}, {"offsets": [0, 2048], "sizes": [4096, 2048], "file": "__1_0.distcp", "byte_offset": 0}, ] } }, "planner_metadata": { "world_size": 512, "sharding_spec": "fsdp_full_shard" } }
Pickle Deserialization Attacks: Never load a .pt checkpoint from an untrusted source. A malicious pickle payload can execute arbitrary code including reverse shells, data exfiltration, or cryptomining payloads. For model distribution, always prefer SafeTensors. For internal training checkpoints where you control the pipeline, PyTorch native format is acceptable but should be stored on access-controlled filesystems.

In practice, many teams use a hybrid approach: SafeTensors for model weights (which enables fast inference loading and safe distribution) and PyTorch native for optimizer states and training metadata (which contain complex nested Python objects that SafeTensors cannot represent). The DCP format bridges this gap by supporting both tensor data and metadata in a resharding-friendly structure.

Elastic Checkpointing & Resharding

One of the most challenging operational problems in large-scale training is elastic resharding: resuming training with a different number of GPUs than the checkpoint was saved with. This happens frequently in practice—nodes fail and are not immediately replaced, preemptible instances are reclaimed, or teams scale up mid-run when more capacity becomes available.

When you change the world size, every tensor that was sharded across the original ranks must be re-partitioned. For FSDP, this means re-flattening and re-chunking the parameters. For tensor parallelism (TP), it means splitting or merging the column/row-parallel linear layers along different dimensions. For pipeline parallelism (PP), it means reassigning layer groups to stages.

The resharding problem has three main dimensions:

  • FSDP shard count change: Each flat parameter is split differently. DCP handles this via its metadata – it knows the original chunk boundaries and can re-partition.
  • TP degree change (e.g., TP=4 → TP=8): Column-parallel weights need further column splitting; row-parallel weights need row splitting. Attention head assignments change.
  • PP stage change: Layer-to-stage mapping is recalculated, and each stage loads different layers than before.
import torch import torch.distributed as dist from torch.distributed.checkpoint import load from torch.distributed.checkpoint.default_planner import DefaultLoadPlanner from typing import Dict, Tuple class ElasticReshardPlanner(DefaultLoadPlanner): """Custom load planner that handles TP degree changes. When TP degree changes between save and load, column-parallel and row-parallel layers must be split or concatenated along the correct dimension.""" def __init__(self, orig_tp: int, new_tp: int): super().__init__() self.orig_tp = orig_tp self.new_tp = new_tp def resolve_tensor(self, name: str, tensor: torch.Tensor) -> torch.Tensor: # Identify column-parallel layers (split along dim=0) if any(k in name for k in ["wq", "wk", "wv", "w1", "w3"]): return self._reshard_col_parallel(tensor) # Identify row-parallel layers (split along dim=1) elif any(k in name for k in ["wo", "w2"]): return self._reshard_row_parallel(tensor) return tensor def _reshard_col_parallel(self, tensor: torch.Tensor) -> torch.Tensor: # Full tensor is [hidden, hidden]. Each TP rank gets [hidden, hidden/tp] tp_rank = dist.get_rank() % self.new_tp chunks = tensor.chunk(self.new_tp, dim=0) return chunks[tp_rank].contiguous() def _reshard_row_parallel(self, tensor: torch.Tensor) -> torch.Tensor: tp_rank = dist.get_rank() % self.new_tp chunks = tensor.chunk(self.new_tp, dim=1) return chunks[tp_rank].contiguous() def load_with_resharding( model, checkpoint_path: str, orig_world_size: int, orig_tp_degree: int, new_tp_degree: int, ) -> None: # DCP handles FSDP resharding automatically via metadata # We only need custom logic for TP dimension changes planner = None if orig_tp_degree != new_tp_degree: planner = ElasticReshardPlanner(orig_tp_degree, new_tp_degree) state_dict = {"model": model.state_dict()} load( state_dict, checkpoint_dir=checkpoint_path, planner=planner, ) model.load_state_dict(state_dict["model"])

Optimizer state resharding is even more complex. Adam optimizer stores exp_avg (first moment) and exp_avg_sq (second moment) for every parameter, and these must be resharded identically to their corresponding parameters. If resharding is done incorrectly, the optimizer moments will be misaligned, causing training instability—often manifesting as a sudden loss spike followed by slow recovery over hundreds of steps.

Practical Tip: When changing TP degree, verify resharding correctness by loading the checkpoint, running a single forward-backward pass, and comparing the gradients against a reference run. Gradient mismatch beyond fp16 tolerance (~1e-3 relative error) indicates a resharding bug.

Storage Architecture

The storage backend is often the bottleneck of the entire checkpoint pipeline. Training clusters typically use parallel distributed filesystems designed for high-throughput sequential writes. The choice of filesystem and its configuration dramatically affects checkpoint performance.

Lustre / FSx for Lustre

  • POSIX-compliant parallel filesystem
  • Data striped across Object Storage Targets (OSTs)
  • Aggregate bandwidth scales with OST count
  • Typical: 50–200 GB/s on large clusters
  • Stripe size tuning critical for checkpoint perf

GPFS / IBM Spectrum Scale

  • Block-level parallel filesystem
  • Strong metadata performance (millions of IOPS)
  • Native RDMA support for low-latency I/O
  • Better for mixed workloads (checkpoints + logs)
  • More complex administration

Checkpoint Compression: Modern models are stored in bfloat16, which does not compress well with general-purpose algorithms (only ~5–10% reduction with zstd). However, optimizer states in fp32 can compress 20–40% because Adam moments for well-trained models have low entropy in the least-significant bits. The trade-off is CPU overhead during compression, but this can be overlapped with I/O using a pipeline of compress → write stages.

Incremental Checkpointing: Rather than saving the entire state dict every time, incremental checkpointing saves only the tensors that have changed beyond a threshold since the last checkpoint. For LLM training, most parameters change every step, but the magnitude of change decreases as training progresses. A practical approach is to compute the L2 norm of the delta and only save tensors where ‖Δw‖/‖w‖ > ε. This can reduce checkpoint size by 30–60% in the later stages of training.

import os, glob, time, json from pathlib import Path from typing import List import logging log = logging.getLogger(__name__) class CheckpointGarbageCollector: """Manages checkpoint lifecycle: keeps recent N checkpoints, plus milestone checkpoints at fixed intervals. Prevents disk exhaustion on long training runs where each checkpoint can be 500GB-1TB.""" def __init__( self, checkpoint_dir: str, keep_recent: int = 3, milestone_interval: int = 5000, ): self.checkpoint_dir = Path(checkpoint_dir) self.keep_recent = keep_recent self.milestone_interval = milestone_interval def collect(self, current_step: int) -> List[str]: # List all checkpoint directories sorted by step ckpt_dirs = sorted( self.checkpoint_dir.glob("step_*"), key=lambda p: int(p.name.split("_")[1]), ) to_delete = [] for ckpt in ckpt_dirs: step = int(ckpt.name.split("_")[1]) # Never delete milestone checkpoints if step % self.milestone_interval == 0: continue # Keep the N most recent non-milestone checkpoints recent_steps = sorted( [int(d.name.split("_")[1]) for d in ckpt_dirs if int(d.name.split("_")[1]) % self.milestone_interval != 0], reverse=True, )[:self.keep_recent] if step not in recent_steps: to_delete.append(str(ckpt)) for path in to_delete: log.info(f"GC: removing old checkpoint {path}") # Use shutil.rmtree in production; omitted for clarity return to_delete

Lustre Stripe Configuration: For checkpoint files, set the stripe count equal to the number of OSTs and a stripe size of 4–16 MB. This ensures writes are distributed across all storage targets. Use lfs setstripe -c -1 -S 16M /checkpoint/dir before training begins. Without this, all writes may go to a single OST, limiting throughput to ~2 GB/s instead of the aggregate 50+ GB/s.

Storage Budget Rule of Thumb: For a training run of S steps with checkpoint interval I, keeping K recent checkpoints plus milestones every M steps, total storage needed is approximately: K × ckpt_size + (S / M) × ckpt_size. For a 70B model training for 100K steps with K=3, I=500, M=5000: that's 3 × 1TB + 20 × 1TB = 23 TB of storage. Budget at least 2x this for headroom.

Multi-Tier Storage: Production training infrastructure often uses a tiered approach: NVMe SSDs local to each node for fast staging (Phase 1 writes), a high-bandwidth parallel filesystem for the active checkpoint window (last K checkpoints), and object storage (S3/GCS) for long-term archival of milestone checkpoints. This balances cost against performance—object storage is 10–50x cheaper per TB than Lustre but has significantly higher latency.

The interplay between checkpointing strategy, storage architecture, and cluster reliability defines the operational boundary of large-scale training. Getting this right is the difference between a training run that completes in 4 weeks and one that takes 8 weeks due to repeated failures and checkpoint overhead.
