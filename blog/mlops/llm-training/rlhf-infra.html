The 4-Model RLHF Architecture

Proximal Policy Optimization (PPO) applied to language models requires four distinct neural networks running simultaneously, which fundamentally differentiates RLHF training from standard supervised fine-tuning. Understanding why each model exists and how they interact is critical for designing infrastructure that can support this workload at scale.

Policy Model (Actor)

The model being optimized. It generates token-level actions (next-token predictions) given a prompt. Its parameters θ are updated via the PPO clipped surrogate objective. This is typically initialized from an SFT checkpoint and is the only model whose weights change during RLHF.

Reference Model (Frozen Copy)

A frozen snapshot of the policy at the start of RLHF training (or the SFT checkpoint). It provides the baseline log-probabilities πref(a|s) needed to compute the KL divergence penalty, preventing the policy from drifting too far and exploiting the reward model.

Reward Model

Trained on human preference data (Bradley-Terry pairwise comparisons) to output a scalar score for a complete (prompt, response) pair. It replaces the human evaluator in the loop. Typically shares the same architecture as the policy but has a scalar projection head instead of a language model head.

Value Model (Critic)

Estimates the expected future reward V(s) at each token position. Used in Generalized Advantage Estimation (GAE) to compute low-variance advantage estimates. Often initialized from the reward model checkpoint and shares the same backbone, with a separate scalar head per token.

Why four models? Standard RL (e.g., Atari PPO) uses two models: actor and critic. RLHF adds the reward model (because we cannot query humans in real-time) and the reference model (because the KL constraint is essential to prevent reward hacking in the open-ended language generation setting).

The interaction flow proceeds as follows: (1) The policy model generates a batch of responses given prompts from a dataset. (2) Each (prompt, response) pair is scored by the reward model to produce scalar rewards. (3) The value model provides per-token value estimates. (4) The reference model computes log-probabilities of the same tokens for KL penalty computation. (5) GAE computes advantages, and the PPO objective updates the policy and value model weights. This cycle repeats for thousands of iterations.

Memory Requirements & Optimization

The fundamental challenge of RLHF is that you must fit four copies of a large language model in GPU memory simultaneously. For a 70B-parameter model in bf16, each copy occupies roughly 140 GB. With Adam optimizer states (8 bytes per parameter for momentum and variance), the trainable policy alone needs ~700 GB. This makes naive RLHF training impractical even on 8×A100-80GB nodes.

Memory back-of-envelope for 70B RLHF: Policy (140 GB weights + 560 GB optimizer) + Reference (140 GB) + Reward (140 GB) + Value (140 GB weights + 560 GB optimizer) = ~1,680 GB total. That is 21 A100-80GB GPUs just for model state, before activations, KV-cache, or rollout buffers.

Production systems apply several optimizations to make this feasible:

Backbone Sharing

The policy (actor) and value (critic) models can share the transformer backbone, with separate heads for token prediction and value estimation. This nearly halves the trainable parameter count. TRL and OpenRLHF both support this via a shared-backbone architecture. The tradeoff: gradient conflicts between the two objectives can hurt stability.

Reference Model Offloading

The reference model is only needed during the KL computation phase, not during generation or the gradient update. It can be offloaded to CPU memory (or NVMe via DeepSpeed ZeRO-Infinity) and loaded on-demand. This frees GPU memory at the cost of PCIe transfer latency (~12 GB/s per lane).

Reward Model Quantization

The reward model only runs inference (no gradients). Quantizing it to INT8 or INT4 via bitsandbytes or GPTQ reduces its footprint by 2-4× with minimal scoring accuracy loss. A 70B reward model drops from 140 GB to 35-70 GB, a significant saving.

ZeRO Stage 3 Partitioning

DeepSpeed ZeRO-3 shards optimizer states, gradients, and parameters across all GPUs. For an 8-GPU node training a 70B policy, each GPU only holds 1/8th of parameters (~17.5 GB). Combined with activation checkpointing, this makes 70B RLHF feasible on a single 8×A100 node for the trainable models.

Practical allocation (70B on 32×A100-80GB): GPUs 0-15: Policy + Value (shared backbone, ZeRO-3, 16-way sharding). GPUs 16-23: Reward model (INT8, tensor-parallel 8-way). GPUs 24-31: Generation engine (vLLM with policy weights, continuous batching). Reference model offloaded to CPU with async prefetch.

Rollout Generation Pipeline

Rollout generation — producing responses from the current policy given a batch of prompts — is typically the throughput bottleneck in RLHF training. Unlike supervised training where we process fixed input-output pairs, RLHF requires autoregressive generation: each token depends on all previous tokens, making the process inherently sequential per sequence. A single rollout batch for a 70B model generating 512 tokens per response can take 30-60 seconds even on 8 GPUs.

Modern RLHF systems decouple generation from training by running inference on a dedicated GPU pool using optimized serving engines:

  • vLLM with PagedAttention: Eliminates KV-cache fragmentation by managing cache in non-contiguous pages, achieving near-zero waste. Supports continuous batching where new prompts are inserted as sequences complete, maximizing GPU utilization during generation.
  • TensorRT-LLM / TGI: Compile-time kernel fusion and INT8/FP8 quantized inference. TensorRT-LLM can deliver 2-3× higher token throughput than vanilla HuggingFace generate() through custom CUDA kernels for attention and MLP layers.
  • Speculative decoding: A small draft model proposes N tokens, and the large policy verifies them in a single forward pass. Acceptance rates of 70-85% translate to 2-3× wall-clock speedups for generation without changing the output distribution.
Generation throughput math: For a 70B model on 8×A100 with tensor parallelism, expect ~1,500 tokens/sec aggregate throughput. A rollout batch of 512 prompts × 512 output tokens = 262K tokens, requiring ~175 seconds. Pipelining generation with training on separate GPU pools hides this latency.

KV-cache management is critical: for a 70B model with 80 layers, 64 heads, and 128-dim per head, each token's KV cache consumes 80 × 2 × 64 × 128 × 2 bytes (bf16) = ~2.6 MB. A batch of 256 sequences at 2048 tokens each requires 256 × 2048 × 2.6 MB ≈ 1.3 TB of KV-cache alone. PagedAttention and prompt-prefix sharing (for system prompts) are essential to keep this manageable.

import torch from dataclasses import dataclass from typing import List, Tuple # Rollout generation with KL computation against reference model @dataclass class RolloutBatch: prompts: torch.Tensor # (B, prompt_len) responses: torch.Tensor # (B, response_len) logprobs: torch.Tensor # (B, response_len) policy log-probs ref_logprobs: torch.Tensor # (B, response_len) reference log-probs rewards: torch.Tensor # (B,) scalar reward per sequence values: torch.Tensor # (B, response_len) per-token values def generate_rollouts( policy_model, ref_model, reward_model, value_model, prompt_batch: torch.Tensor, max_new_tokens: int = 512, temperature: float = 0.7, top_p: float = 0.95, ) -> RolloutBatch: # Step 1: Generate responses from the current policy with torch.no_grad(): responses, logprobs = policy_model.generate_with_logprobs( prompt_batch, max_new_tokens=max_new_tokens, temperature=temperature, top_p=top_p, do_sample=True, ) # Step 2: Compute reference log-probs for KL penalty with torch.no_grad(): full_seq = torch.cat([prompt_batch, responses], dim=1) ref_logprobs = ref_model.forward_logprobs( full_seq, response_start=prompt_batch.size(1) ) # Step 3: Score each (prompt, response) pair with reward model with torch.no_grad(): rewards = reward_model.score(prompt_batch, responses) # Step 4: Estimate values at each token for GAE with torch.no_grad(): values = value_model.forward_values( full_seq, response_start=prompt_batch.size(1) ) return RolloutBatch( prompts=prompt_batch, responses=responses, logprobs=logprobs, ref_logprobs=ref_logprobs, rewards=rewards, values=values, )

PPO Training Loop

The PPO training loop consumes a batch of rollouts and updates the policy and value model through multiple epochs of mini-batch gradient descent. The key components are: Generalized Advantage Estimation (GAE) for computing low-variance advantages, the clipped surrogate objective for stable policy updates, the value function loss for critic training, and the entropy bonus to prevent premature convergence.

Generalized Advantage Estimation (GAE)

GAE computes per-token advantage estimates that balance bias and variance through a discount parameter λ. At each token position t in the response, the TD-residual is δt = rt + γV(st+1) - V(st). For language model RLHF, the reward is typically sparse (only at the final token), so δt = -V(st) + γV(st+1) for intermediate tokens and δT = r - V(sT) at the last token. The advantage is the exponentially-weighted sum: Ât = ∑l=0T-t (γλ)l δt+l.

Clipped Surrogate Objective

The policy loss uses the PPO clipping mechanism to constrain how far the new policy can move from the old policy (the one that generated the rollouts): LCLIP = -E[min(rtÂt, clip(rt, 1-ε, 1+ε)Ât)] where rt = πθ(at|st) / πθold(at|st). The clipping parameter ε is typically 0.2, meaning the probability ratio cannot move more than 20% from 1.0.

import torch import torch.nn.functional as F from typing import Dict class PPOTrainer: def __init__(self, policy, value_model, lr=1e-6, clip_eps=0.2, gamma=1.0, lam=0.95, vf_coeff=0.5, entropy_coeff=0.01, kl_coeff=0.1, target_kl=0.02, epochs=4, batch_size=64): self.policy = policy self.value_model = value_model self.clip_eps = clip_eps self.gamma = gamma self.lam = lam self.vf_coeff = vf_coeff self.entropy_coeff = entropy_coeff self.kl_coeff = kl_coeff self.target_kl = target_kl self.epochs = epochs self.batch_size = batch_size self.optimizer = torch.optim.AdamW( list(policy.parameters()) + list(value_model.parameters()), lr=lr, betas=(0.9, 0.95), weight_decay=0.01, ) def compute_gae(self, rewards, values, response_mask) -> torch.Tensor: # rewards: (B, T) per-token rewards (sparse: nonzero only at last token) # values: (B, T) per-token value estimates from critic B, T = values.shape advantages = torch.zeros_like(values) last_gae = torch.zeros(B, device=values.device) for t in reversed(range(T)): if t == T - 1: next_value = torch.zeros(B, device=values.device) else: next_value = values[:, t + 1] delta = rewards[:, t] + self.gamma * next_value - values[:, t] last_gae = delta + self.gamma * self.lam * last_gae # Zero out advantage for padding tokens last_gae = last_gae * response_mask[:, t] advantages[:, t] = last_gae return advantages def ppo_step(self, rollout: RolloutBatch) -> Dict[str, float]: # Build per-token reward: KL penalty at each step, RM reward at end kl_per_token = rollout.logprobs - rollout.ref_logprobs per_token_rewards = -self.kl_coeff * kl_per_token # Add the reward model score to the final token seq_lens = (rollout.responses != 0).sum(dim=1) - 1 for i in range(rollout.responses.size(0)): per_token_rewards[i, seq_lens[i]] += rollout.rewards[i] response_mask = (rollout.responses != 0).float() advantages = self.compute_gae(per_token_rewards, rollout.values, response_mask) returns = advantages + rollout.values # Normalize advantages (critical for training stability) adv_mean = (advantages * response_mask).sum() / response_mask.sum() adv_std = ((advantages - adv_mean).pow(2) * response_mask).sum() / response_mask.sum() advantages = (advantages - adv_mean) / (adv_std.sqrt() + 1e-8) stats = {"policy_loss": 0, "value_loss": 0, "entropy": 0, "approx_kl": 0} # Multiple epochs of mini-batch updates over the rollout for epoch in range(self.epochs): indices = torch.randperm(rollout.responses.size(0)) for start in range(0, len(indices), self.batch_size): mb_idx = indices[start:start + self.batch_size] # Forward pass: get current log-probs and values new_logprobs = self.policy.forward_logprobs( rollout.prompts[mb_idx], rollout.responses[mb_idx] ) new_values = self.value_model.forward_values( rollout.prompts[mb_idx], rollout.responses[mb_idx] ) entropy = -(new_logprobs * new_logprobs.exp()).sum(dim=-1).mean() # Policy loss with clipped surrogate objective ratio = (new_logprobs - rollout.logprobs[mb_idx]).exp() mb_adv = advantages[mb_idx] surr1 = ratio * mb_adv surr2 = torch.clamp(ratio, 1 - self.clip_eps, 1 + self.clip_eps) * mb_adv policy_loss = -torch.min(surr1, surr2) policy_loss = (policy_loss * response_mask[mb_idx]).sum() / response_mask[mb_idx].sum() # Value loss with clipping (optional but common) vf_loss = (new_values - returns[mb_idx]).pow(2) vf_loss = (vf_loss * response_mask[mb_idx]).sum() / response_mask[mb_idx].sum() # Combined loss loss = policy_loss + self.vf_coeff * vf_loss - self.entropy_coeff * entropy self.optimizer.zero_grad() loss.backward() torch.nn.utils.clip_grad_norm_(self.policy.parameters(), 1.0) self.optimizer.step() # Track approximate KL for early stopping approx_kl = (rollout.logprobs[mb_idx] - new_logprobs).mean().item() stats["approx_kl"] = approx_kl # Early stopping if KL diverges too much if stats["approx_kl"] > 1.5 * self.target_kl: break return stats
Why multiple epochs? A single rollout batch is expensive to generate. Running 2-4 PPO epochs per batch amortizes the generation cost. However, too many epochs cause the policy to diverge from the data-generating policy, violating the on-policy assumption. The approximate KL early-stopping mechanism guards against this.

KL Divergence Control

The KL divergence between the policy and reference model is the single most important regularization mechanism in RLHF. Without it, the policy rapidly overfits to the reward model's quirks — a phenomenon called reward hacking. The policy discovers degenerate sequences that score highly according to the reward model but are nonsensical or repetitive to humans.

InstructGPT introduced the KL penalty as a per-token reward modifier: r̂t = r(x, y) · δt=T - β · KLt, where KLt = log πθ(yt|x, y<t) - log πref(yt|x, y<t). This per-token formulation means the policy is penalized at every step for deviating from the reference, not just at the sequence level.

Adaptive KL Coefficient

A fixed β is fragile: too high and the policy barely moves from SFT; too low and reward hacking occurs. The adaptive approach (from OpenAI's PPO-RLHF paper) adjusts β based on the observed KL:

def update_kl_coeff(kl_coeff: float, observed_kl: float, target_kl: float = 6.0) -> float: # If KL is too high, increase penalty to pull policy back if observed_kl > target_kl * 1.5: kl_coeff *= 2.0 # If KL is too low, relax penalty to allow more exploration elif observed_kl < target_kl / 1.5: kl_coeff *= 0.5 # Clamp to prevent extreme values kl_coeff = max(0.001, min(10.0, kl_coeff)) return kl_coeff

KL Penalty (Reward Modifier)

Subtracts β·KL from the reward signal. The PPO optimizer then naturally reduces KL because it lowers returns. This is the InstructGPT approach. Advantage: works within the standard PPO framework. Disadvantage: conflates reward signal with regularization; the value function must model both.

KL Constraint (Explicit)

Enforces E[KL] ≤ δ as a hard constraint via Lagrangian relaxation. The dual variable λ is updated alongside policy parameters. This is more principled (TRPO-style) and separates reward from regularization. Used in some research implementations but rarer in production due to added complexity.

Reward hacking in practice: Common manifestations include the policy generating excessively long responses (reward models often correlate length with quality), repeating high-reward phrases, or producing outputs that exploit spurious patterns in the reward model's training data. Monitoring the KL divergence alongside reward during training is essential — if reward increases while KL explodes, reward hacking is likely occurring.

KL Approximations

Computing exact KL requires a forward pass through both the policy and reference model. Several approximations reduce cost:

  • Log-ratio approximation: KL ≈ E[log(π/πref)] = E[logprobs - ref_logprobs]. This is the standard unbiased estimator used in most implementations.
  • k3 estimator (Schulman): KL ≈ (ratio - 1) - log(ratio), where ratio = π/πref. This is always non-negative and has lower variance than the log-ratio estimator for large divergences.
  • Periodic reference updates: Instead of keeping the reference model frozen, update it every N steps (EMA or hard copy). This allows the policy to drift further over time while maintaining local stability.

Distributed RLHF Systems

Scaling RLHF to hundreds of GPUs requires careful orchestration of four interacting model instances, each with different computation patterns: the policy needs both inference (generation) and training, the reward and reference models need only inference, and the value model needs training. This heterogeneity makes RLHF significantly harder to distribute than standard data-parallel training.

Framework Comparison

TRL (HuggingFace)

Built on Accelerate + DeepSpeed. Single-controller design where all models live in the same process. Easy to use for research but limited to models that fit in one node's memory. PPOTrainer handles rollout collection and optimization in a unified loop. Best for ≤13B models on single nodes.

OpenRLHF

Ray-based distributed architecture. Each model (actor, critic, reward, reference) runs on separate Ray actors with dedicated GPU pools. Supports vLLM for generation, ZeRO-3 for training, and automatic data routing between components. Scales to 70B+ models across multiple nodes.

DeepSpeed-Chat

Three-phase pipeline: SFT → Reward Modeling → RLHF. Uses DeepSpeed ZeRO and Hybrid Engine (combining training and inference optimization in one engine) for memory efficiency. The Hybrid Engine switches between ZeRO-partitioned training mode and tensor-parallel inference mode within the same process.

NeMo-Aligner (NVIDIA)

Built on NeMo and Megatron-Core. Uses Megatron's 3D parallelism (tensor + pipeline + data) for each model. Highest throughput for very large models (>100B) at the cost of complexity. Supports critic/reward model sharding and interleaved pipeline schedules.

Communication Patterns

The dominant communication pattern in distributed RLHF is the data shuttle between the generation cluster and the training cluster. After the policy generates a batch of responses, the following data must be transferred to the training nodes:

  • Response token IDs: B × T × 4 bytes (int32)
  • Policy log-probabilities: B × T × 4 bytes (float32)
  • Reward scores: B × 4 bytes (float32)
  • Value estimates: B × T × 4 bytes (float32)
  • Reference log-probs: B × T × 4 bytes (float32)

For a batch of B=512 sequences with T=1024 tokens, this is ~8 MB — trivial compared to gradient synchronization. The real challenge is weight synchronization: after each PPO update, the training cluster must broadcast the updated policy weights to the generation cluster. For a 70B model in bf16, this is 140 GB per sync, requiring ~12 seconds over 100 Gbps InfiniBand.

Pipelining strategy: To hide weight-sync latency, advanced systems overlap generation batch N+1 with the PPO training on batch N. The generation cluster uses the previous policy weights (one step stale), which introduces slight off-policy bias but is acceptable given PPO's robustness. This is analogous to asynchronous SGD in data-parallel training.

Scaling Throughput Optimization

End-to-end RLHF throughput is measured in samples per second (complete (prompt, response, reward) tuples processed). Key optimizations for scaling beyond 64 GPUs include:

  • Micro-batch interleaving: While one micro-batch is in the reward model forward pass, another is being generated, and a third is in the PPO backward pass. This keeps all GPU pools active simultaneously.
  • Fused generation-scoring: Run the reward model immediately after each sequence completes (continuous batching), rather than waiting for the entire batch to finish generation. This reduces idle time on reward model GPUs.
  • Gradient accumulation with async allreduce: Overlap the gradient allreduce of mini-batch K with the forward pass of mini-batch K+1. With 4 PPO epochs and 8 mini-batches per epoch, this hides most of the communication cost.
  • Dynamic batch sizing: Adjust the rollout batch size based on sequence length distribution. Short responses complete faster, so batches heavy on short prompts should be larger to maintain constant GPU utilization.
Scaling bottleneck: Beyond ~256 GPUs, the weight synchronization between generation and training clusters becomes the dominant bottleneck. Solutions include ring-allreduce weight broadcast, sharded weight loading (each generation node fetches only its tensor-parallel shard), and using RDMA (Remote Direct Memory Access) for zero-copy weight transfer over InfiniBand.
