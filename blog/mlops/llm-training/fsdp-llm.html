FSDP Architecture & Sharding Strategies

PyTorch Fully Sharded Data Parallel (FSDP) is a native implementation of the ZeRO-3 algorithm from DeepSpeed. Instead of replicating the entire model on every GPU (as vanilla DDP does), FSDP shards parameters, gradients, and optimizer states across the data-parallel group. Before a forward or backward computation on a given FSDP unit, all-gather collectives reconstruct the full parameters; after the backward pass, reduce-scatter distributes and sums gradient shards.

The key enum is ShardingStrategy, which controls what gets sharded and when memory is freed:

FULL_SHARD (ZeRO-3)

Shards parameters + gradients + optimizer states. After forward, full params are freed; after backward, gradient shards are reduced. This yields the lowest per-GPU memory but the highest communication volume— two all-gathers and one reduce-scatter per FSDP unit per step.

SHARD_GRAD_OP (ZeRO-2)

Shards gradients + optimizer states only. Full parameters are kept in memory after the forward all-gather, avoiding the backward all-gather. This trades ~1.5× more memory for ~33% less communication—ideal when the model fits with this strategy but not with DDP.

NO_SHARD

Equivalent to standard DDP: every rank holds a full replica. Useful as a debugging baseline or when the model is small enough to fit fully replicated. Communication is a single all-reduce per step.

HYBRID_SHARD

Full-shards within a node (intra-node NVLink) but replicates across nodes (inter-node InfiniBand). Balances memory savings with reduced cross-node traffic. Requires setting _hybrid_shard_zero2 or ShardingStrategy.HYBRID_SHARD.

from torch.distributed.fsdp import ( FullyShardedDataParallel as FSDP, ShardingStrategy, ) # Choose sharding based on cluster topology model = FSDP( model, sharding_strategy=ShardingStrategy.FULL_SHARD, # ZeRO-3 # sharding_strategy=ShardingStrategy.SHARD_GRAD_OP, # ZeRO-2 # sharding_strategy=ShardingStrategy.HYBRID_SHARD, # Intra-node shard device_id=torch.cuda.current_device(), )
Memory rule of thumb: A 7B-parameter model in fp32 needs ~28 GB for parameters alone. With Adam optimizer states (2× param memory), total is ~84 GB per replica under DDP. FULL_SHARD across 8 GPUs reduces this to ~10.5 GB per GPU for params+optim, enabling training on 40 GB A100s.

Wrapping Policies

FSDP wraps models into a hierarchy of FullyShardedDataParallel units. Each unit is the granularity at which all-gather and reduce-scatter operate. Wrapping too coarsely (entire model as one unit) means all params must be gathered at once, destroying memory savings. Wrapping too finely (every linear layer) explodes communication overhead. The optimal boundary is typically one transformer block.

Size-Based Auto Wrapping

Wraps any nn.Module whose parameter count exceeds a threshold:

from torch.distributed.fsdp.wrap import size_based_auto_wrap_policy # Wrap any sub-module with >= 100M params as its own FSDP unit auto_wrap_policy = functools.partial( size_based_auto_wrap_policy, min_num_params=int(1e8), ) model = FSDP(model, auto_wrap_policy=auto_wrap_policy)

Transformer Auto Wrapping

The recommended approach for LLMs. You specify the transformer layer class and FSDP wraps each instance as a separate unit. This aligns FSDP boundaries with the natural compute structure:

from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy from transformers.models.llama.modeling_llama import LlamaDecoderLayer # Each LlamaDecoderLayer becomes one FSDP unit auto_wrap_policy = functools.partial( transformer_auto_wrap_policy, transformer_layer_cls={LlamaDecoderLayer}, ) model = FSDP( model, auto_wrap_policy=auto_wrap_policy, sharding_strategy=ShardingStrategy.FULL_SHARD, device_id=torch.cuda.current_device(), )

Manual Wrapping

For non-standard architectures (mixture-of-experts, multi-modal models), manual wrapping provides full control. You explicitly wrap each sub-module:

# Manual wrapping for a MoE model for i, layer in enumerate(model.layers): # Wrap the attention block layer.self_attn = FSDP(layer.self_attn) # Wrap each expert separately for balanced sharding for j, expert in enumerate(layer.moe.experts): layer.moe.experts[j] = FSDP(expert) # Wrap the full layer model.layers[i] = FSDP(layer) # Root wrap model = FSDP(model)
Warning: If you manually wrap sub-modules, you must still wrap the root module with FSDP. Failing to do so leaves root parameters unsharded and causes silent correctness bugs— gradients from different ranks diverge after the first step.

Mixed Precision with BFloat16

FSDP's MixedPrecision policy controls three independent dtype knobs. Unlike torch.cuda.amp, FSDP casts parameters during the all-gather, so communication happens in the reduced dtype, cutting bandwidth by 2× when using bf16 over fp32.

param_dtype

The dtype used for forward/backward computation. Set to torch.bfloat16 for Ampere+ GPUs. Parameters are cast to this dtype after all-gather, before entering the forward function.

reduce_dtype

The dtype used for gradient reduction (reduce-scatter). Keeping this at torch.float32 preserves gradient precision at the cost of 2× reduction bandwidth. Most LLM trainers set this to bf16 for speed, relying on loss scaling or gradient clipping for stability.

buffer_dtype

The dtype for non-parameter buffers (e.g., LayerNorm running stats). Usually set to torch.bfloat16 to match param_dtype and avoid casting overhead.

cast_forward_inputs

When True, FSDP automatically casts the inputs to each FSDP unit to param_dtype. This prevents dtype mismatch errors when an FSDP boundary falls between modules with different dtypes.

from torch.distributed.fsdp import MixedPrecision # Recommended config for LLM training on A100/H100 bf16_policy = MixedPrecision( param_dtype=torch.bfloat16, # compute in bf16 reduce_dtype=torch.bfloat16, # reduce gradients in bf16 buffer_dtype=torch.bfloat16, # buffers in bf16 cast_forward_inputs=True, # auto-cast inputs at FSDP boundaries ) # Higher-precision reduction for stability-sensitive models fp32_reduce_policy = MixedPrecision( param_dtype=torch.bfloat16, reduce_dtype=torch.float32, # reduce in fp32 for stability buffer_dtype=torch.bfloat16, ) model = FSDP( model, mixed_precision=bf16_policy, auto_wrap_policy=auto_wrap_policy, sharding_strategy=ShardingStrategy.FULL_SHARD, device_id=torch.cuda.current_device(), )
BFloat16 vs Float16: BFloat16 has the same exponent range as fp32 (8 exponent bits) but only 7 mantissa bits. This means it rarely overflows/underflows, eliminating the need for loss scaling. Float16 (5 exponent bits, 10 mantissa) has higher precision per value but a narrow dynamic range that requires GradScaler. For LLM pretraining, bf16 is the standard choice on Ampere+ hardware.

Performance Tuning

FSDP exposes several knobs that control the overlap between computation and communication. Proper tuning can improve throughput by 10–30% on large clusters. The key insight is that FSDP must all-gather parameters before a unit's computation and can prefetch the next unit's parameters while the current unit is still computing.

Backward Prefetch

Controls when FSDP issues the all-gather for the next FSDP unit during the backward pass. BACKWARD_PRE (default) prefetches before the current unit's backward finishes, overlapping communication with compute. BACKWARD_POST waits until after, which uses less memory but serializes communication.

from torch.distributed.fsdp import BackwardPrefetch model = FSDP( model, backward_prefetch=BackwardPrefetch.BACKWARD_PRE, # max overlap (default) # backward_prefetch=BackwardPrefetch.BACKWARD_POST, # lower peak memory )

Forward Prefetch

When enabled, FSDP issues the all-gather for the next FSDP unit during the current unit's forward pass. This is less impactful than backward prefetch because forward computations are typically faster, but it helps on high-latency interconnects:

model = FSDP( model, forward_prefetch=True, # overlap forward all-gather with compute )

Limit All-Gathers

limit_all_gathers=True prevents FSDP from launching more than one outstanding all-gather at a time. Without this flag, on fast networks, multiple all-gathers can be in flight simultaneously, spiking peak memory. Enable this when approaching GPU memory limits:

model = FSDP( model, limit_all_gathers=True, # cap outstanding all-gathers to 1 backward_prefetch=BackwardPrefetch.BACKWARD_PRE, )

Full FSDP Configuration for 70B Training

import torch import functools from torch.distributed.fsdp import ( FullyShardedDataParallel as FSDP, ShardingStrategy, MixedPrecision, BackwardPrefetch, CPUOffload, ) from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy from transformers.models.llama.modeling_llama import LlamaDecoderLayer def build_fsdp_model(model, rank): """Full production FSDP config for 70B-class LLMs.""" wrap_policy = functools.partial( transformer_auto_wrap_policy, transformer_layer_cls={LlamaDecoderLayer}, ) mp_policy = MixedPrecision( param_dtype=torch.bfloat16, reduce_dtype=torch.bfloat16, buffer_dtype=torch.bfloat16, cast_forward_inputs=True, ) fsdp_model = FSDP( model, sharding_strategy=ShardingStrategy.FULL_SHARD, auto_wrap_policy=wrap_policy, mixed_precision=mp_policy, backward_prefetch=BackwardPrefetch.BACKWARD_PRE, forward_prefetch=True, limit_all_gathers=True, device_id=rank, use_orig_params=True, # required for torch.compile ) return fsdp_model
use_orig_params=True: This flag preserves the original parameter names and structure, which is required for torch.compile() compatibility and makes checkpoint saving/loading more predictable. It also enables per-parameter optimizer groups (e.g., different learning rates for embeddings vs attention layers). Always enable this for production training.

State Dict Types & Checkpointing

Checkpointing with FSDP is non-trivial because parameters are sharded across ranks. FSDP provides three state dict types, each with different trade-offs for memory, speed, and portability:

FULL_STATE_DICT

All-gathers the complete model onto rank 0. Produces a standard PyTorch state dict that can be loaded without FSDP. Requires O(model_size) extra memory on rank 0. Use for final checkpoint export or small models.

SHARDED_STATE_DICT

Each rank saves its own shard directly. No extra memory needed. Requires the same world size and wrapping policy to reload. Best for mid-training checkpoints where you resume on the same cluster.

LOCAL_STATE_DICT

Each rank saves its local flat parameters without metadata. The fastest option but essentially non-portable—only useful for crash recovery on the exact same configuration.

from torch.distributed.fsdp import ( FullStateDictConfig, StateDictType, ) # === FULL_STATE_DICT: portable but memory-heavy === full_cfg = FullStateDictConfig(offload_to_cpu=True, rank0_only=True) with FSDP.state_dict_type(model, StateDictType.FULL_STATE_DICT, full_cfg): state = model.state_dict() if rank == 0: torch.save(state, f"checkpoint_full.pt")
from torch.distributed.fsdp import ShardedStateDictConfig from torch.distributed.checkpoint import save as dist_save # === SHARDED_STATE_DICT: scalable mid-training checkpoints === sharded_cfg = ShardedStateDictConfig(offload_to_cpu=True) with FSDP.state_dict_type(model, StateDictType.SHARDED_STATE_DICT, sharded_cfg): state = {"model": model.state_dict(), "optim": FSDP.optim_state_dict(model, optimizer)} dist_save(state, checkpoint_id=f"ckpt_step_{step}")
# === Loading a sharded checkpoint === from torch.distributed.checkpoint import load as dist_load with FSDP.state_dict_type(model, StateDictType.SHARDED_STATE_DICT): state = {"model": model.state_dict()} dist_load(state, checkpoint_id=f"ckpt_step_{step}") model.load_state_dict(state["model"]) # Load optimizer state (must happen after model state) optim_state = FSDP.optim_state_dict_to_load( model, optimizer, optim_state_dict ) optimizer.load_state_dict(optim_state)
Optimizer state is tricky: You cannot simply torch.save and torch.load optimizer state with FSDP. The optimizer tracks flat, sharded parameter references. Use FSDP.optim_state_dict() to extract and FSDP.optim_state_dict_to_load() to reinject. Mishandling this silently corrupts training—loss will appear normal for hundreds of steps before diverging.

Common Gotchas & Debugging

FSDP failure modes are notoriously opaque. Below are the most frequent issues encountered when training LLMs at scale, along with root causes and fixes.

1. Meta Device Initialization

Large models are instantiated on the meta device to avoid materializing 70B+ parameters on CPU. FSDP's param_init_fn callback materializes each shard on the correct GPU. If you forget this, you get "Cannot copy out of meta tensor; no data!".

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP def param_init_fn(module): # Materialize meta-device params to the current GPU module.to_empty(device=torch.cuda.current_device(), recurse=False) # Re-initialize weights (meta params have no data) if hasattr(module, "reset_parameters"): module.reset_parameters() # Use with_device("meta") for zero-cost init with torch.device("meta"): model = LlamaForCausalLM(config) model = FSDP( model, param_init_fn=param_init_fn, auto_wrap_policy=wrap_policy, sync_module_states=True, # broadcast rank-0 weights to all ranks )

2. dtype Mismatch Errors

The dreaded "Expected all tensors to be on the same device" or "RuntimeError: expected scalar type BFloat16 but found Float" usually means your model has modules that produce fp32 outputs (e.g., RMSNorm, embedding layers) feeding into FSDP units expecting bf16 inputs.

Fix: Set cast_forward_inputs=True in your MixedPrecision config. This tells FSDP to automatically cast inputs at each unit boundary. Alternatively, explicitly cast in your model's forward method, but this is fragile and hard to maintain across wrapping policy changes.

3. OOM During All-Gather

If you OOM during the forward pass (not backward), the cause is usually multiple all-gathers running concurrently. Each all-gather temporarily allocates O(full_unit_params) memory:

# OOM Debugging checklist: # 1. Enable limit_all_gathers to cap concurrent collectives model = FSDP(model, limit_all_gathers=True) # 2. Switch backward prefetch to BACKWARD_POST model = FSDP(model, backward_prefetch=BackwardPrefetch.BACKWARD_POST) # 3. Ensure wrapping granularity is fine enough # If the root FSDP unit is too large, its all-gather # materializes the entire model at once # 4. Check for param duplication with use_orig_params # When True, FSDP keeps both flat and original views # 5. Profile with torch.cuda.memory_stats() stats = torch.cuda.memory_stats() print(f"Peak allocated: {stats['allocated_bytes.all.peak'] / 1e9:.2f} GB") print(f"Peak reserved: {stats['reserved_bytes.all.peak'] / 1e9:.2f} GB")

4. Hanging During Init or First Step

FSDP wrapping triggers collective operations. If any rank diverges in its wrapping order (e.g., conditional wrapping based on layer index that differs across ranks), the process group hangs. Similarly, sync_module_states=True performs a broadcast during __init__—if one rank skips this, all ranks deadlock.

Debug tip: Set TORCH_DISTRIBUTED_DEBUG=DETAIL and NCCL_DEBUG=INFO environment variables. This logs every collective operation with its caller, making it straightforward to identify which rank is stuck and on which operation.

5. Gradient Accumulation

FSDP's no_sync() context manager skips the reduce-scatter, accumulating gradients locally. This is essential for effective batch size scaling without proportional communication overhead:

# Gradient accumulation with FSDP accumulation_steps = 4 for step, batch in enumerate(dataloader): # Skip reduce-scatter for accumulation steps context = model.no_sync() if (step + 1) % accumulation_steps != 0 else nullcontext() with context: loss = model(batch).loss loss = loss / accumulation_steps loss.backward() if (step + 1) % accumulation_steps == 0: torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) optimizer.step() optimizer.zero_grad()

6. torch.compile Compatibility

Combining FSDP with torch.compile() requires use_orig_params=True. Without it, the compiler cannot trace through FSDP's flat parameter views. Additionally, compile the inner model before FSDP wrapping, or use torch.compile(model) on the already-wrapped model with fullgraph=False to allow graph breaks at FSDP boundaries:

# Option A: Compile before wrapping model = LlamaForCausalLM(config) model = torch.compile(model) model = FSDP(model, use_orig_params=True) # Option B: Compile after wrapping (with graph breaks) model = FSDP(model, use_orig_params=True) model = torch.compile(model, fullgraph=False)
FSDP2 (torch.distributed._composable.fsdp): PyTorch is developing a composable FSDP API that works as a module-level transform rather than a wrapper. It integrates natively with torch.compile and tensor parallelism. For new projects targeting PyTorch 2.3+, evaluate FSDP2 as it will eventually replace the current FSDP API.
