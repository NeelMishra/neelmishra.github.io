Speculative Decoding

MLOps Series LLM Inference & Serving

The Autoregressive Bottleneck

Autoregressive decoding is fundamentally memory-bandwidth bound, not compute bound. Each forward pass through a 70B-parameter model loads ~140 GB of weights from HBM to compute a single token. On an A100 (2 TB/s bandwidth), this means a theoretical minimum latency of ~70 ms per token — regardless of how many FLOPs the GPU can deliver. The arithmetic intensity is roughly 1 FLOP per byte loaded, far below the ~300 FLOP/byte ratio needed to saturate the A100's compute.

Speculative decoding (Leviathan et al., 2022; Chen et al., 2023) exploits this gap: if we're wasting compute anyway, use it to verify multiple candidate tokens in a single forward pass of the large model.

Key insight: A single forward pass of the target model with K+1 tokens costs nearly the same wall-clock time as a forward pass with 1 token when batch size is small, because both are equally memory-bandwidth bound. This is the free lunch that speculative decoding exploits.

Core Algorithm

The algorithm uses two models: a small draft model Mq (e.g., 68M–1B parameters) and the large target model Mp (e.g., 70B parameters). The draft model is ~50–100× faster per token.

The pipeline operates in draft-then-verify loops:

  1. Draft phase: The draft model autoregressively generates γ candidate tokens (typically γ = 3–8).
  2. Verify phase: The target model processes all γ tokens plus the prefix in a single forward pass, producing logits for positions n+1 through n+γ+1.
  3. Accept/reject: Tokens are accepted left-to-right using a modified rejection sampling scheme. At the first rejection, we resample from an adjusted distribution and discard remaining draft tokens.
  4. Repeat: Advance the sequence by the number of accepted tokens plus one (the resampled token) and loop.

Modified Rejection Sampling

The critical property of speculative decoding is that it produces exactly the same distribution as sampling from the target model alone. This is achieved through a modified rejection sampling scheme.

For each draft token xi sampled from q(x), we compute the acceptance probability:

Acceptance criterion: Accept token x with probability min(1, p(x)/q(x)), where p(x) is the target model probability and q(x) is the draft model probability. When rejected, resample from the residual distribution: norm(max(0, p(x) - q(x))).

The residual distribution ensures that the combined accept-or-resample procedure yields exactly p(x). Formally:

P(output = x) = q(x) · min(1, p(x)/q(x)) + (1 - Σ_x q(x)·min(1, p(x)/q(x))) · max(0, p(x)-q(x)) / Σ_x max(0, p(x)-q(x)) = p(x)

The acceptance rate α for a single token is:

α = Σ_x min(p(x), q(x)) = 1 - ½ · TV(p, q)

where TV(p,q) is the total variation distance between the two distributions. When the draft model closely matches the target model (small TV distance), the acceptance rate is high.

High Acceptance (α ≈ 0.85)

Draft model closely matches target. Typical for well-distilled models on common text patterns. Expected tokens per iteration: ~5.7 with γ=6.

Low Acceptance (α ≈ 0.5)

Draft model diverges from target. Occurs on domain-specific text or reasoning-heavy tasks. Expected tokens per iteration: ~2.0 with γ=6.

Speedup Analysis & Math

Let c = Tdraft/Ttarget be the cost ratio of one draft forward pass to one target forward pass (typically c ≈ 0.01–0.05 for a 68M draft with a 70B target).

The expected number of tokens per iteration with speculation length γ is:

E[tokens] = (1 - α^(γ+1)) / (1 - α)

The wallclock cost per iteration is γ·c·Ttarget + Ttarget = Ttarget(γc + 1). The expected speedup is therefore:

Speedup = E[tokens] / (γc + 1) = (1 - α^(γ+1)) / ((1 - α)(γc + 1))

For α = 0.8, γ = 5, c = 0.02:

  • E[tokens] = (1 - 0.8⁶)/(1 - 0.8) = (1 - 0.262)/0.2 = 3.69
  • Cost = 5 × 0.02 + 1 = 1.10 target-passes
  • Speedup = 3.69 / 1.10 = 3.35×

In practice, reported speedups range from 2.0–3.5× depending on task distribution, model pair quality, and hardware utilization.

Optimal γ: The optimal speculation length depends on α and c. Too small wastes the verification pass; too large wastes draft compute on tokens that will be rejected. For α=0.8, the optimal γ is typically 5–8. Some systems adapt γ dynamically based on recent acceptance rates.

Implementation

Below is the core speculative decoding loop. The key operations are: (1) draft generation, (2) parallel verification, (3) acceptance with rejection sampling.

import torch
from torch.distributions import Categorical

def speculative_decode(target_model, draft_model, prefix, gamma=5, max_tokens=256):
    """Speculative decoding: exact target distribution, ~3x speedup."""
    generated = prefix.clone()
    n_generated = 0

    while n_generated < max_tokens:
        # Phase 1: Draft model generates gamma candidate tokens
        draft_tokens = []
        draft_probs = []
        draft_input = generated.clone()

        for _ in range(gamma):
            logits = draft_model(draft_input)[:, -1, :]
            q = torch.softmax(logits, dim=-1)
            token = Categorical(q).sample()
            draft_tokens.append(token)
            draft_probs.append(q)
            draft_input = torch.cat([draft_input, token.unsqueeze(1)], dim=1)

        # Phase 2: Target model verifies all tokens in one forward pass
        candidate_seq = torch.cat([generated] + [t.unsqueeze(1) for t in draft_tokens], dim=1)
        target_logits = target_model(candidate_seq)
        # Extract p(x) at each draft position
        n = generated.shape[1]
        target_probs = [torch.softmax(target_logits[:, n+i, :], dim=-1) for i in range(gamma + 1)]

        # Phase 3: Accept/reject with modified rejection sampling
        accepted = 0
        for i in range(gamma):
            token = draft_tokens[i]
            p_x = target_probs[i].gather(1, token.unsqueeze(1)).squeeze()
            q_x = draft_probs[i].gather(1, token.unsqueeze(1)).squeeze()
            acceptance_prob = torch.min(torch.ones_like(p_x), p_x / q_x)

            if torch.rand(1).item() < acceptance_prob.item():
                generated = torch.cat([generated, token.unsqueeze(1)], dim=1)
                accepted += 1
            else:
                # Resample from residual distribution: norm(max(0, p - q))
                residual = torch.clamp(target_probs[i] - draft_probs[i], min=0)
                residual = residual / residual.sum(dim=-1, keepdim=True)
                new_token = Categorical(residual).sample()
                generated = torch.cat([generated, new_token.unsqueeze(1)], dim=1)
                accepted += 1  # resampled token still counts
                break

        if accepted == gamma:
            # All draft tokens accepted — bonus token from target at position gamma
            bonus = Categorical(target_probs[gamma]).sample()
            generated = torch.cat([generated, bonus.unsqueeze(1)], dim=1)
            accepted += 1

        n_generated += accepted

    return generated

Variants: Medusa, EAGLE & Beyond

The original speculative decoding requires a separate draft model. Recent variants eliminate this requirement or improve upon the basic scheme.

Medusa (Cai et al., 2024)

Adds multiple parallel prediction heads to the target model itself. Each head i predicts the token at position t+i. Uses a tree-structured attention mask to verify multiple candidate continuations simultaneously. Achieves 2.2–3.6× speedup with minimal additional parameters (~0.6% of model size). No separate draft model needed.

EAGLE (Li et al., 2024)

Uses a lightweight auto-regression head that operates on the feature level rather than the token level. Predicts the next hidden state using the current hidden state + token embedding. Achieves 2.5–3.8× speedup, outperforming Medusa on most benchmarks. Tree-structured verification for better acceptance.

SpecInfer (Miao et al., 2023)

Uses multiple small draft models in parallel, generating a token tree of candidates. The target model verifies the entire tree in a single pass using tree attention. Higher acceptance rates through diversity. Integrated into FlexFlow serving framework. Reports 1.5–2.8× speedup.

Staged Speculative (Spector et al., 2023)

Cascades multiple draft models of increasing size (e.g., 68M → 1B → 7B → 70B). Each stage speculates for the next one. Allows deeper speculation trees with better acceptance rates. Particularly effective when the cost ratio between adjacent stages is small.

Self-speculative decoding: Recent approaches like Draft & Verify (Zhang et al., 2023) use early exit from the target model itself as the draft. By exiting after layer 8 of a 32-layer model, you get a cheap approximation without any additional model. This eliminates the draft model entirely but requires architectural modifications.

Practical Considerations

Deploying speculative decoding in production requires careful engineering beyond the core algorithm.

Draft Model Selection

The ideal draft model has high acceptance rate (close distribution to target) and low latency (small parameter count). Common pairings:

  • LLaMA-70B + LLaMA-68M: c ≈ 0.01, α ≈ 0.7–0.85 depending on domain
  • GPT-4 + GPT-3.5: High acceptance rate but larger cost ratio c ≈ 0.1
  • CodeLlama-34B + CodeLlama-7B: Strong alignment on code tasks, α ≈ 0.8

KV Cache Management

Both draft and target models maintain KV caches. On rejection, the target model's KV cache must be rolled back to exclude rejected tokens. Efficient implementations pre-allocate the cache for the maximum speculation length and simply adjust the valid-length pointer.

Batching Challenges

Speculative decoding complicates continuous batching because different sequences accept different numbers of tokens per iteration. Solutions include: (1) padding to the max accepted length, (2) per-request speculation length, or (3) running speculative decoding only for latency-critical requests while using standard decoding for throughput-oriented batches.

When NOT to Use Speculative Decoding

  • High-throughput serving: When the GPU is already compute-saturated from large batches, speculative decoding provides no benefit since the bottleneck shifts from memory-bandwidth to compute.
  • Very low acceptance rate domains: If α < 0.5 consistently, the overhead of draft generation may negate any speedup.
  • Prefill-dominated workloads: Long input contexts where prefill time dominates make the per-token decode speedup less impactful.
Production numbers: Google DeepMind reported 2.46× speedup on PaLM 2 using speculative decoding in production. Hugging Face's implementation of Medusa on Vicuna-33B achieves 2.3× speedup. vLLM's speculative decoding with LLaMA-70B + LLaMA-7B achieves approximately 1.8–2.5× speedup depending on the request distribution.
