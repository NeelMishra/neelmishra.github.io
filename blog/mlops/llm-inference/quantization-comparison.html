Why Quantize LLMs?

A Llama-2 70B model in FP16 requires 140 GB of VRAM — two A100-80GB GPUs just to load weights. Quantization compresses weights (and optionally activations) from 16-bit to 8-bit, 4-bit, or even 2-bit representations, directly reducing memory footprint and improving inference throughput by reducing memory bandwidth demand.

Key Insight: LLM inference is almost always memory-bandwidth bound, not compute bound. During autoregressive decoding, each token generation reads the entire weight matrix but performs relatively few FLOPs per byte. Quantizing from FP16 to INT4 cuts memory traffic by 4×, yielding near-linear speedups on bandwidth-limited hardware.

The memory bandwidth equation for decode-phase throughput:

// Tokens per second ≈ Memory Bandwidth / Bytes per Token
// A100 80GB: 2 TB/s bandwidth
// Llama-2 70B FP16: 140 GB weights → ~14.3 tok/s
// Llama-2 70B INT4:  35 GB weights → ~57.1 tok/s (theoretical)

def estimate_decode_throughput(model_bytes, bandwidth_bytes_per_sec):
    # Each token reads all weights once (simplified)
    tokens_per_sec = bandwidth_bytes_per_sec / model_bytes
    return tokens_per_sec

# Real-world: ~70-85% of theoretical due to
# KV cache reads, activation compute, kernel overhead

Quantization Taxonomy

Quantization methods differ along three axes: when quantization happens (post-training vs. quantization-aware training), what gets quantized (weights only vs. weights + activations), and how the quantization grid is calibrated.

Weight-Only Quantization

GPTQ, AWQ, GGUF

Only model weights are stored in low precision. During inference, weights are dequantized to FP16 on-the-fly before matrix multiplication. Activations remain in FP16/BF16.

Advantage: Simpler calibration, no activation outlier handling needed. Works with any hardware that supports FP16 matmul.

Weight + Activation Quantization

SmoothQuant, FP8 (W8A8)

Both weights and activations are quantized. Enables use of INT8/FP8 tensor cores for actual low-precision compute, not just compressed storage.

Advantage: True compute speedup from tensor cores. Up to 2× faster matmul on H100 FP8 vs FP16.

Post-Training Quantization (PTQ)

Applied after training completes. Uses a small calibration dataset (128-512 samples) to determine optimal quantization parameters. No retraining required.

All methods in this post are PTQ.

Quantization-Aware Training (QAT)

Simulates quantization during fine-tuning. Produces higher quality at aggressive bit widths (2-3 bit) but requires full training infrastructure.

Examples: QuIP#, AQLM, BitNet. Not covered here.

Method Deep Dive

GPTQ — Optimal Brain Quantizer for GPTs

GPTQ applies layer-wise quantization using an approximate second-order method derived from Optimal Brain Surgeon. For each layer, it quantizes weights one column at a time, using the inverse Hessian to optimally redistribute quantization error to remaining unquantized columns.

# GPTQ Algorithm (simplified)
for layer in model.layers:
    W = layer.weight                     # [out_features, in_features]
    H = compute_hessian(layer, calibration_data)  # X^T X
    H_inv = cholesky_inverse(H + λI)    # Damped inverse Hessian

    for col_group in column_groups(W, block_size=128):
        for col in col_group:
            q = quantize(W[:, col])       # Round to nearest grid point
            error = W[:, col] - q
            # Redistribute error to remaining columns
            W[:, col+1:] += outer(error, H_inv[col, col+1:]) / H_inv[col, col]
            W[:, col] = q

Strengths: Mature ecosystem (AutoGPTQ, ExLlama), wide model availability on HuggingFace. Handles 4-bit and 3-bit well. Weaknesses: Quantization is slow (hours for 70B), column-order dependent, group_size sensitivity.

AWQ — Activation-Aware Weight Quantization

AWQ observes that a small fraction (~1%) of weight channels are disproportionately important because they correspond to large activation magnitudes. Instead of uniform quantization, AWQ applies per-channel scaling that protects these salient channels before quantization.

# AWQ: scale salient channels before quantization
for layer in model.layers:
    act_scales = collect_activation_scales(layer, calib_data)
    # Channels with large activations get larger scale → smaller quant error
    s = search_optimal_scale(layer.weight, act_scales, bit_width=4)
    # Apply: W_scaled = W * diag(s), X_scaled = X * diag(1/s)
    # Mathematical equivalence: Y = X @ W = (X/s) @ (W*s)
    layer.weight = quantize(layer.weight * s)
    layer.input_scale = 1.0 / s  # Applied at runtime to activations

Strengths: Faster quantization than GPTQ (~minutes), slightly better quality at 4-bit (preserves salient channels), hardware-friendly. Weaknesses: Requires activation statistics from calibration.

GGUF — GGML Universal Format

GGUF (used by llama.cpp) is a file format and runtime that supports CPU inference with optional GPU offloading. It uses block quantization with various type codes (Q4_0, Q4_K_M, Q5_K_M, Q8_0, etc.) where "K" variants use k-quant with importance-based mixed precision across blocks.

K-Quants: The "K_M" (medium) and "K_S" (small) variants assign different bit widths to different tensor types. Attention layers get higher precision (Q6_K) while FFN layers get lower precision (Q4_K), preserving quality where it matters most.

SmoothQuant — Migrating Difficulty from Activations to Weights

LLM activations contain outlier channels with magnitudes 100× larger than the median, making activation quantization extremely difficult. SmoothQuant applies a mathematically equivalent per-channel scaling that "smooths" activation outliers into weights, which are easier to quantize.

# SmoothQuant: Y = (X diag(s)^-1) @ (diag(s) W) = X @ W
for layer in model.layers:
    act_max = per_channel_absmax(activations)   # shape: [C_in]
    wt_max  = per_channel_absmax(layer.weight)   # shape: [C_in]

    # Migration strength α controls the trade-off (typically 0.5)
    s = act_max.pow(α) / wt_max.pow(1 - α)

    # Now both X_smooth and W_smooth have manageable ranges
    layer.weight = quantize_int8(layer.weight * diag(s))

FP8 (E4M3 / E5M2)

H100 GPUs natively support FP8 tensor cores. E4M3 (4-bit exponent, 3-bit mantissa) provides the dynamic range for weights, while E5M2 (5-bit exponent, 2-bit mantissa) suits gradients. For inference, W8A8-FP8 with per-tensor scaling achieves near-FP16 quality with 2× compute throughput.

Quality vs Compression Tradeoff

The relationship between bit width and perplexity degradation is non-linear. Quality drops gradually from 16→8→4 bits, then sharply at 3→2 bits:

The 4-bit Sweet Spot: For most production deployments, 4-bit quantization (AWQ or GPTQ with group_size=128) offers the best balance — ~3.7× compression with less than 0.2 perplexity degradation on large models. Going to 3-bit saves only ~22% more memory but costs 0.3-0.5 perplexity points.

Hardware Requirements

GPTQ / AWQ (4-bit)

Requires: NVIDIA GPU with FP16 support (Ampere+)

Runtime: CUDA kernels (Marlin, ExLlama v2)

Quantization: 1 GPU, 2-6 hours (GPTQ) or 10-30 min (AWQ) for 70B

Best for: GPU-only cloud deployments

GGUF (llama.cpp)

Requires: CPU (AVX2/NEON) with optional GPU offload

Runtime: Custom C/C++ with CUDA/Metal/Vulkan backends

Quantization: Minutes for any model size

Best for: Edge/desktop, Apple Silicon, mixed CPU+GPU

SmoothQuant (INT8)

Requires: INT8 tensor cores (Turing T4+, Ampere, Hopper)

Runtime: TensorRT-LLM, FasterTransformer

Quantization: ~30 minutes for 70B with calibration

Best for: Batch processing, high throughput scenarios

FP8 (E4M3)

Requires: H100/H200 (native FP8 tensor cores)

Runtime: TensorRT-LLM, vLLM with FP8 support

Quantization: Per-tensor scaling, fast calibration

Best for: Maximum quality at 8-bit, Hopper deployments

# Quick start: Quantize a model with AWQ using AutoAWQ
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "meta-llama/Llama-2-70b-hf"
quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,
    "version": "GEMM"    # Use GEMV for batch_size=1 decode
}

model = AutoAWQForCausalLM.from_pretrained(model_path)
tokenizer = AutoTokenizer.from_pretrained(model_path)
model.quantize(tokenizer, quant_config=quant_config)
model.save_quantized("llama-2-70b-awq")

Use Case Recommendations

🏢 Production API (Cloud)

Recommended: AWQ 4-bit (Marlin kernels) on A100/H100 with vLLM

Best throughput-per-dollar. AWQ + Marlin kernels achieve near-FP16 quality with 3-4× higher throughput. For H100, consider FP8 for maximum quality retention.

💻 Local/Desktop Inference

Recommended: GGUF Q4_K_M via llama.cpp or Ollama

Best ecosystem for consumer hardware. K-quant variants intelligently allocate bits. Works on Apple Silicon (Metal), AMD (Vulkan/ROCm), and Intel (AVX-512).

📊 Batch Processing

Recommended: FP8 or SmoothQuant INT8 on TensorRT-LLM

W8A8 quantization enables actual INT8/FP8 tensor core compute, providing 2× throughput gain at large batch sizes where compute becomes the bottleneck.

🎯 Maximum Quality

Recommended: FP8 E4M3 on H100, or AWQ 4-bit with group_size=32

FP8 is nearly lossless (<0.01 PPL increase). If 4-bit is needed, smaller group_size (32 vs 128) reduces quantization error at the cost of ~15% more memory.

Common Pitfall: Don't assume quantization quality is model-size independent. A method that works well on 70B may degrade significantly on 7B at the same bit width. Always benchmark perplexity and task-specific accuracy on your target model size before deploying to production.
