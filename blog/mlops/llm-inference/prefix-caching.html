Prefix Caching: Reusing KV Cache

MLOps Series LLM Inference & Serving

The KV Cache Cost Problem

Every LLM request begins with a prefill phase that processes the entire prompt through all transformer layers, computing Key and Value tensors at each layer. For a Llama-2 70B model with a 2,000-token system prompt, this prefill takes ~800ms on an A100 and consumes significant compute. In a chatbot scenario where every request shares the same system prompt, this identical computation is repeated for every single request.

# KV cache memory per token per layer (Llama-2 70B)
# K: [num_kv_heads × head_dim] = [8 × 128] = 1024 values
# V: [num_kv_heads × head_dim] = [8 × 128] = 1024 values
# Per token per layer: 2 × 1024 × 2 bytes (FP16) = 4 KB
# Per token all layers: 4 KB × 80 layers = 320 KB
# 2000-token system prompt: 320 KB × 2000 = 640 MB

num_kv_heads = 8      # GQA groups for Llama-2 70B
head_dim = 128
num_layers = 80
bytes_per_param = 2   # FP16

kv_per_token = 2 * num_kv_heads * head_dim * bytes_per_param * num_layers
# = 2 × 8 × 128 × 2 × 80 = 327,680 bytes ≈ 320 KB per token

system_prompt_tokens = 2000
system_prompt_kv = kv_per_token * system_prompt_tokens
# = 320 KB × 2000 = 640 MB of KV cache for the system prompt alone
The Redundancy: If you serve 100 concurrent requests with the same system prompt, you're computing 640 MB × 100 = 64 GB of identical KV cache data. Prefix caching computes it once and shares the cached KV tensors across all requests, saving both compute (prefill latency) and memory.

Prefix Caching Concept

Prefix caching exploits the autoregressive property of transformers: the KV cache for tokens 0 through N depends only on those tokens, regardless of what follows. If two requests share the same first N tokens, their KV caches for those positions are identical and can be reused.

Without Prefix Caching

Request 1: [system_prompt + "What is ML?"]

→ Prefill all 2,010 tokens from scratch: ~810ms

Request 2: [system_prompt + "Explain CNNs"]

→ Prefill all 2,012 tokens from scratch: ~812ms

Total prefill compute: ~1,622ms

With Prefix Caching

Request 1: [system_prompt + "What is ML?"]

→ Prefill all 2,010 tokens (cache miss): ~810ms

Request 2: [system_prompt + "Explain CNNs"]

→ Reuse 2,000 cached tokens + prefill 12 new: ~5ms

Total prefill compute: ~815ms (2× speedup)

The savings compound dramatically at scale. With 1,000 requests sharing a system prompt, prefix caching saves ~799,000ms of cumulative prefill time. But the real challenge is efficiently matching shared prefixes across diverse request streams. This is where RadixAttention comes in.

RadixAttention & Radix Trees

RadixAttention (Zheng et al., 2023, SGLang) uses a radix tree (compact trie) to index all cached KV sequences. Each edge in the tree represents a sequence of token IDs, and each node points to the corresponding KV cache pages in GPU memory. Prefix matching becomes a tree traversal — O(n) in the length of the query prefix.

The radix tree stores compressed edges: instead of one node per token (as in a naive trie), long shared sequences are represented as a single edge. This keeps the tree compact even with thousands of cached sequences. When a new request arrives, the tree is traversed to find the longest matching prefix.

# Radix tree node for KV cache prefix matching
class RadixNode:
    def __init__(self):
        self.children = {}       # first_token → (edge_tokens, child_node)
        self.kv_page_ids = []    # Pages storing KV cache for this prefix
        self.ref_count = 0       # Number of active sequences using this prefix
        self.last_access = 0     # For LRU eviction

class RadixTree:
    def __init__(self):
        self.root = RadixNode()

    def match_prefix(self, token_ids):
        """Find longest cached prefix. Returns (matched_length, kv_pages)."""
        node = self.root
        matched = 0
        matched_pages = []

        while matched < len(token_ids):
            first_tok = token_ids[matched]
            if first_tok not in node.children:
                break  # No further match

            edge_tokens, child = node.children[first_tok]
            # Compare edge tokens with query tokens
            match_len = 0
            for i, etok in enumerate(edge_tokens):
                if matched + i >= len(token_ids):
                    break
                if token_ids[matched + i] != etok:
                    break
                match_len += 1

            if match_len == 0:
                break

            matched += match_len
            matched_pages.extend(child.kv_page_ids[:match_len // page_size])
            child.ref_count += 1
            child.last_access = time()

            if match_len < len(edge_tokens):
                break  # Partial edge match
            node = child

        return matched, matched_pages

Automatic Prefix Matching

Automatic prefix caching (APC) transparently detects shared prefixes without requiring users to explicitly declare them. The system hashes token sequences at page-granularity (typically 256 tokens per page) and uses a hash table for O(1) page lookup.

# vLLM-style automatic prefix caching using content-hash
class AutoPrefixCache:
    def __init__(self, page_size=256):
        self.page_size = page_size
        self.hash_to_page = {}   # hash(token_block) → gpu_page_id
        self.page_ref_count = {} # gpu_page_id → reference count

    def compute_block_hash(self, token_ids, block_idx):
        """Hash includes all preceding tokens for correctness.
        Block N's KV depends on tokens 0...(N+1)*page_size."""
        prefix_end = (block_idx + 1) * self.page_size
        return hash(tuple(token_ids[:prefix_end]))

    def allocate_with_cache(self, token_ids):
        """Allocate KV pages, reusing cached pages where possible."""
        num_blocks = ceil(len(token_ids) / self.page_size)
        pages = []
        cache_hit_blocks = 0

        for i in range(num_blocks):
            block_hash = self.compute_block_hash(token_ids, i)

            if block_hash in self.hash_to_page:
                # Cache hit: reuse existing KV page
                page_id = self.hash_to_page[block_hash]
                self.page_ref_count[page_id] += 1
                pages.append(page_id)
                cache_hit_blocks += 1
            else:
                # Cache miss: allocate new page, will be filled during prefill
                page_id = self.allocate_new_page()
                self.hash_to_page[block_hash] = page_id
                self.page_ref_count[page_id] = 1
                pages.append(page_id)

        return pages, cache_hit_blocks
Critical detail: The hash for block N must include all tokens from 0 to (N+1)×page_size, not just the tokens in that block. This is because KV values at position P depend on all preceding tokens (causal attention). Two blocks with the same local tokens but different preceding context produce different KV values. The full-prefix hash ensures correctness.

Production Cache Hit Rates

Cache hit rates vary dramatically by workload. The key factor is prefix diversity — how many distinct prefixes exist and how frequently they recur.

Workload Typical Prefix Length Cache Hit Rate TTFT Reduction Memory Savings
Chatbot (shared system prompt)1,000-4,000 tokens85-98%60-95%~40% KV memory
RAG (shared retrieval context)500-2,000 tokens20-60%15-50%~15% KV memory
Multi-turn chat (conversation history)200-8,000 tokens70-90%50-85%~30% KV memory
Code completion (file context)1,000-6,000 tokens40-75%30-65%~25% KV memory
Unique prompts (no sharing)0 tokens0-5%~0%~0%

High-Hit Workload: Chatbot

All requests share an identical system prompt (e.g., "You are a helpful assistant..."). With a 2,000-token system prompt and 100 concurrent requests, prefix caching saves:

Compute: 99 × 800ms prefill = ~79.2 seconds of GPU time saved per batch

Memory: 99 × 640 MB avoided = ~63 GB of redundant KV cache eliminated

Low-Hit Workload: RAG

RAG pipelines retrieve different documents per query, so prefix overlap is limited to the system instruction (typically short). Caching helps only when multiple users query the same document chunk within the cache TTL.

Optimization: Prepend shared instructions before per-query context to maximize cacheable prefix length.

vLLM Automatic Prefix Caching

vLLM implements automatic prefix caching (APC) that works transparently with PagedAttention. Enable it with a single flag:

# Enable automatic prefix caching in vLLM
# CLI: vllm serve meta-llama/Llama-2-7b-hf --enable-prefix-caching

from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-2-7b-hf",
    enable_prefix_caching=True,    # This is all you need
    gpu_memory_utilization=0.9,
    max_model_len=8192,
)

# These two requests share the system prompt prefix
# vLLM automatically detects and caches the shared KV blocks
system = "You are a legal analysis assistant. Always cite relevant case law..."

prompts = [
    system + "\n\nAnalyze this contract clause...",
    system + "\n\nIs this NDA enforceable in California?",
    system + "\n\nWhat are the liability limitations here?",
]

# First request: cache miss, full prefill (~800ms for 2K system prompt)
# Second request: cache hit on system prompt, prefill only user query (~5ms)
# Third request: cache hit again (~5ms)
outputs = llm.generate(prompts, SamplingParams(temperature=0.3))

How vLLM APC Works Internally

# Simplified vLLM APC flow
def schedule_request(request):
    token_ids = tokenize(request.prompt)
    num_blocks = ceil(len(token_ids) / block_size)

    allocated_pages = []
    prefill_start_pos = 0

    for block_idx in range(num_blocks):
        # Hash includes full prefix up to this block
        content_hash = hash_token_block(
            token_ids[: (block_idx + 1) * block_size]
        )

        cached_page = block_manager.lookup_cached(content_hash)
        if cached_page is not None:
            # Reuse cached KV page — no prefill computation needed
            allocated_pages.append(cached_page)
            prefill_start_pos = (block_idx + 1) * block_size
        else:
            # Allocate fresh page — will be computed during prefill
            new_page = block_manager.allocate()
            allocated_pages.append(new_page)
            # Register in cache for future requests
            block_manager.register_cached(content_hash, new_page)

    # Only prefill tokens after the cached prefix
    request.prefill_tokens = token_ids[prefill_start_pos:]
    request.page_table = allocated_pages
Block granularity: vLLM caches at the block level (default 16 tokens per block). This means prefix matching is quantized to block boundaries — a 2,001-token shared prefix caches 125 full blocks (2,000 tokens), and the 1 remaining token starts a new block that may not match. For maximum hit rates, keep shared prefixes aligned to block boundaries.

Memory Savings & Design Tradeoffs

Memory Savings Calculation

Scenario: 64 concurrent requests, 2,048-token system prompt, Llama-2 7B

KV per token: 2 × 32 × 128 × 2 bytes × 32 layers = 1 MB per token (all layers)

System prompt KV: 2,048 × 1 MB = 2 GB per request

Without caching: 64 × 2 GB = 128 GB

With caching: 2 GB (shared) + 64 × ~50 KB (unique suffixes) ≈ 2.003 GB

Savings: ~98.4% (126 GB)

TTFT Improvement

Scenario: Same as left, A100-80GB

Prefill throughput: ~2,500 tokens/second (Llama-7B, batch=1)

Without caching: 2,048 / 2,500 = ~820ms TTFT

With caching (hit): 50 / 2,500 = ~20ms TTFT

41× TTFT reduction

This is the difference between "sluggish" and "instant" in user perception.

Design Tradeoffs

Cache Eviction Strategy

LRU (Least Recently Used): Default in vLLM. Simple, works well for steady-state workloads. Risk: popular prefixes evicted during traffic spikes.

LFU (Least Frequently Used): Better for skewed distributions (few hot prefixes, many cold). More memory overhead for frequency counters.

Reference-counted: Pages are only evictable when ref_count drops to 0 (no active sequences use them). Prevents evicting in-use pages.

Cache Size vs. Throughput

Larger cache → more prefixes retained → higher hit rate → better TTFT. But cache consumes GPU memory that could serve more concurrent requests.

Optimal split: Profile your workload's prefix distribution. If 90% of requests share 5 system prompts, a small cache suffices. If prefix diversity is high (RAG), allocate more memory to cache.

Rule of thumb: Reserve 10-20% of GPU memory for prefix cache in chatbot workloads.

Correctness Warning: Prefix caching assumes causal (left-to-right) attention. If your model uses bidirectional attention (e.g., encoder models, prefix-tuned models with bidirectional prefix), KV values at position P depend on all tokens, not just preceding ones. Prefix caching is invalid for such architectures — it will produce silently incorrect outputs.
Key Takeaway: Prefix caching is the single highest-impact optimization for chatbot and multi-turn conversation workloads. With a shared system prompt, it eliminates 60-95% of prefill compute, reduces TTFT by 10-40×, and cuts KV cache memory by 30-98%. It's essentially free — vLLM's --enable-prefix-caching flag has negligible overhead and should be enabled by default for any workload with repeated prefixes.
