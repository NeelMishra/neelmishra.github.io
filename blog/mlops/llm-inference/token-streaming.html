Token Streaming: SSE, WebSocket, gRPC

MLOps Series LLM Inference & Serving

Why Stream Tokens?

LLM inference generates tokens sequentially at 30–100 tokens/second. Without streaming, a 500-token response takes 5–17 seconds before the user sees anything. Streaming delivers tokens as they are generated, reducing time-to-first-token (TTFT) perceived latency from seconds to milliseconds.

Latency Perception: Studies show users perceive streaming responses as 3–5× faster than equivalent batch responses, even when total generation time is identical. The progressive rendering creates a sense of responsiveness that dramatically improves UX.

Three protocols dominate the LLM streaming landscape, each with distinct tradeoffs in complexity, browser support, and bidirectionality:

Protocol Comparison

Latency Overhead

SSE: ~0.1ms per event (text framing only)

WebSocket: ~0.05ms per frame (minimal 2-byte header)

gRPC: ~0.2ms per message (protobuf serialization + HTTP/2 framing)

All are negligible relative to the 10-30ms per-token generation time.

Connection Limits

SSE: 6 per domain (HTTP/1.1), unlimited with HTTP/2

WebSocket: ~200 per browser tab (browser-dependent)

gRPC: Multiplexed over single HTTP/2 connection

For SSE on HTTP/1.1, each chat tab consumes one of 6 slots — problematic for multi-tab usage.

SSE Implementation

Server-Sent Events use a simple text-based protocol over HTTP. The server sends lines prefixed with data:, separated by double newlines. The EventSource API handles reconnection automatically.

Server (Python / FastAPI)

from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import json, asyncio

app = FastAPI()

async def generate_tokens(prompt: str):
    """Yield tokens from the LLM engine one at a time."""
    engine = get_llm_engine()
    request_id = generate_request_id()
    engine.add_request(request_id, prompt, sampling_params)

    async for output in engine.generate(request_id):
        token_text = output.outputs[0].text
        delta = token_text[len(prev_text):]
        prev_text = token_text

        # SSE format: "data: {json}\n\n"
        chunk = {
            "id": request_id,
            "object": "chat.completion.chunk",
            "choices": [{"delta": {"content": delta}}]
        }
        yield f"data: {json.dumps(chunk)}\n\n"

    yield "data: [DONE]\n\n"

@app.post("/v1/chat/completions")
async def chat_completions(request: ChatRequest):
    if request.stream:
        return StreamingResponse(
            generate_tokens(request.messages),
            media_type="text/event-stream",
            headers={
                "Cache-Control": "no-cache",
                "X-Accel-Buffering": "no",  # Disable nginx buffering
            }
        )

Client (JavaScript)

// Using fetch() for POST-based SSE (EventSource only supports GET)
async function streamChat(messages) {
  const response = await fetch("/v1/chat/completions", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ messages, stream: true }),
  });

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = "";

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split("\n");
    buffer = lines.pop();  // Keep incomplete line in buffer

    for (const line of lines) {
      if (line.startsWith("data: ")) {
        const data = line.slice(6);
        if (data === "[DONE]") return;
        const chunk = JSON.parse(data);
        const token = chunk.choices[0]?.delta?.content;
        if (token) appendToken(token);
      }
    }
  }
}
Proxy Buffering: Nginx, CloudFlare, and many CDNs buffer responses by default. For SSE to work, you must disable buffering: X-Accel-Buffering: no (nginx), proxy_buffering off, or use chunked transfer encoding. Without this, tokens arrive in batches rather than individually.

WebSocket Implementation

WebSocket provides full-duplex communication, enabling the client to cancel generation mid-stream, send follow-up messages without a new connection, and receive binary data. Ideal for real-time interactive applications.

Server (Python / FastAPI)

from fastapi import FastAPI, WebSocket, WebSocketDisconnect
import json, asyncio

app = FastAPI()

@app.websocket("/ws/chat")
async def websocket_chat(ws: WebSocket):
    await ws.accept()
    generation_task = None

    try:
        while True:
            msg = json.loads(await ws.receive_text())

            if msg["type"] == "cancel":
                if generation_task:
                    generation_task.cancel()  # Cancel ongoing generation
                    await ws.send_json({"type": "cancelled"})
                continue

            if msg["type"] == "generate":
                async def stream_generation():
                    async for token in generate_tokens(msg["messages"]):
                        await ws.send_json({
                            "type": "token",
                            "content": token
                        })
                    await ws.send_json({"type": "done"})

                generation_task = asyncio.create_task(stream_generation())

    except WebSocketDisconnect:
        if generation_task:
            generation_task.cancel()  # Clean up on disconnect

Client (JavaScript)

class LLMWebSocket {
  constructor(url) {
    this.ws = new WebSocket(url);
    this.onToken = null;
    this.ws.onmessage = (event) => {
      const msg = JSON.parse(event.data);
      if (msg.type === "token" && this.onToken)
        this.onToken(msg.content);
    };
  }

  generate(messages) {
    this.ws.send(JSON.stringify({
      type: "generate", messages
    }));
  }

  cancel() {
    this.ws.send(JSON.stringify({ type: "cancel" }));
  }
}

gRPC Streaming

gRPC uses HTTP/2 multiplexed streams with Protocol Buffers for typed, efficient serialization. Server-side streaming is natural for LLM inference; bidirectional streaming enables advanced patterns like speculative decoding feedback loops.

Proto Definition

// llm_service.proto
syntax = "proto3";

service LLMService {
  // Server-streaming: one request, many token responses
  rpc GenerateStream(GenerateRequest)
      returns (stream TokenResponse);

  // Bidirectional: client can send cancellation mid-stream
  rpc ChatStream(stream ChatMessage)
      returns (stream TokenResponse);
}

message GenerateRequest {
  repeated Message messages = 1;
  SamplingParams params = 2;
}

message TokenResponse {
  string token = 1;
  bool is_finished = 2;
  UsageStats usage = 3;
}

Server (Python)

import grpc
from concurrent import futures

class LLMServicer(llm_pb2_grpc.LLMServiceServicer):
    async def GenerateStream(self, request, context):
        engine = get_llm_engine()

        async for output in engine.generate(request.messages):
            if context.cancelled():
                return  # gRPC native cancellation

            yield llm_pb2.TokenResponse(
                token=output.delta,
                is_finished=output.finished,
                usage=llm_pb2.UsageStats(
                    prompt_tokens=output.prompt_tokens,
                    completion_tokens=output.completion_tokens
                )
            )
gRPC Cancellation: gRPC propagates cancellation across service boundaries automatically via HTTP/2 RST_STREAM. When a client cancels, the server's context.cancelled() returns True immediately, enabling clean resource cleanup — a major advantage over SSE where cancellation requires closing the HTTP connection.

OpenAI-Compatible Streaming Format

The de facto standard for LLM API streaming follows the OpenAI chat completions format. All major serving frameworks (vLLM, TGI, Ollama) support this format, making it the safest choice for interoperability.

// Each SSE event contains a JSON chunk:
data: {"id":"chatcmpl-abc123","object":"chat.completion.chunk","created":1694268190,"model":"gpt-4","choices":[{"index":0,"delta":{"role":"assistant"},"finish_reason":null}]}

data: {"id":"chatcmpl-abc123","object":"chat.completion.chunk","created":1694268190,"model":"gpt-4","choices":[{"index":0,"delta":{"content":"Hello"},"finish_reason":null}]}

data: {"id":"chatcmpl-abc123","object":"chat.completion.chunk","created":1694268190,"model":"gpt-4","choices":[{"index":0,"delta":{"content":"!"},"finish_reason":"stop"}]}

data: [DONE]

// Key fields in each chunk:
// - First chunk: delta contains "role" (no content)
// - Middle chunks: delta contains "content" (token text)
// - Last chunk: finish_reason = "stop"|"length"|"content_filter"
// - Terminal: literal string "[DONE]" (not JSON)

Python Client (OpenAI SDK)

from openai import OpenAI

# Works with vLLM, TGI, or any OpenAI-compatible server
client = OpenAI(base_url="http://localhost:8000/v1", api_key="dummy")

stream = client.chat.completions.create(
    model="meta-llama/Llama-2-70b-chat-hf",
    messages=[{"role": "user", "content": "Explain quantization"}],
    stream=True,
    stream_options={"include_usage": True}  # Get token counts in final chunk
)

for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)

Backpressure & Error Recovery

When clients consume tokens slower than the server generates them, backpressure builds up. Each protocol handles this differently:

SSE Backpressure

TCP flow control handles backpressure implicitly. If the client's receive buffer fills up, the kernel stops ACKing TCP segments, which causes the server's write() call to block or the send buffer to fill. The server's async generator naturally pauses.

Risk: Server-side buffering in async frameworks (uvicorn) can mask backpressure, leading to OOM. Use StreamingResponse with bounded internal queues.

WebSocket Backpressure

Similar TCP-level flow control, but WebSocket frames can queue in the server's send buffer. Monitor ws.client_state and implement explicit backpressure:

Pattern: Check transport.get_write_buffer_size() before sending. If buffer exceeds threshold, pause generation and resume when buffer drains.

Error Recovery Pattern

# Robust SSE client with retry and partial response recovery
class ResilientStreamClient:
    def __init__(self, base_url, max_retries=3):
        self.base_url = base_url
        self.max_retries = max_retries

    async def stream_with_recovery(self, messages):
        accumulated_text = ""
        retries = 0

        while retries < self.max_retries:
            try:
                async for token in self._stream_request(messages):
                    accumulated_text += token
                    yield token
                return  # Success: exit retry loop

            except (ConnectionError, asyncio.TimeoutError) as e:
                retries += 1
                wait = min(2 ** retries, 30)  # Exponential backoff, max 30s
                await asyncio.sleep(wait)

                # Append partial response as assistant message
                # and ask model to continue from where it left off
                if accumulated_text:
                    messages = messages + [
                        {"role": "assistant", "content": accumulated_text},
                        {"role": "user", "content": "Continue."}
                    ]
Critical: Never trust the client to handle [DONE] termination alone. Always set a server-side timeout (e.g., 120s) per request and an AbortController on the client with a matching timeout. Without this, network interruptions can leave zombie connections consuming server resources and GPU memory for abandoned KV caches.
