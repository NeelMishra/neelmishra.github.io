Continuous Batching: Maximizing GPU Utilization

MLOps Series LLM Inference & Serving

The Batching Problem

As we established in the fundamentals post, LLM decode is memory-bandwidth-bound. A single decode step for a 70B model reads 140 GB of weights from HBM to generate one token. Batching amortizes this weight read across multiple sequences: with batch size B, the same 140 GB weight read produces B tokens instead of one.

But LLM requests have a fundamental challenge: sequences finish at different times. One request might generate 10 tokens, another 500. Any batching strategy must handle this heterogeneity efficiently.

# The core problem: variable output lengths
# Imagine 4 concurrent requests:
request_A: prompt=100 tokens, output=20 tokens    # finishes quickly
request_B: prompt=200 tokens, output=500 tokens   # runs for a long time
request_C: prompt=50 tokens,  output=150 tokens
request_D: prompt=300 tokens, output=30 tokens     # finishes quickly

# How do we batch these efficiently?
# Static batching: wait for slowest (B: 500 tokens) → GPU idle 70% of time
# Continuous batching: swap in new requests as others finish → near-100% util

Static Batching

The simplest approach: collect B requests, run them all to completion, then start the next batch. This is how most frameworks (HuggingFace Transformers, early vLLM competitors) originally worked.

def static_batching(request_queue, model, batch_size=8):
    while request_queue:
        # Step 1: Collect a full batch
        batch = []
        for _ in range(batch_size):
            if request_queue:
                batch.append(request_queue.pop())

        # Step 2: Pad all sequences to max length
        max_prompt_len = max(len(r.prompt) for r in batch)
        padded_prompts = pad_sequences(batch, max_prompt_len)

        # Step 3: Run prefill for entire batch
        kv_caches = model.prefill(padded_prompts)

        # Step 4: Decode until ALL sequences hit EOS or max_len
        for step in range(max_output_len):
            logits, kv_caches = model.decode_step(batch, kv_caches)
            for seq in batch:
                if not seq.finished:
                    token = sample(logits[seq.idx])
                    seq.append_token(token)
                # Finished sequences still occupy GPU memory!
                # Their KV cache slots are wasted until batch completes

        # Step 5: Return results, start next batch
        yield [seq.output for seq in batch]
Waste in static batching: If one sequence generates 500 tokens and another generates 20, the short sequence's GPU slot (weights computation, KV cache memory) sits idle for 480 decode steps. With diverse output lengths, static batching achieves only 20–40% GPU utilization on average. New requests wait in queue even when GPU capacity is available.

Continuous Batching

Continuous batching (also called in-flight batching) operates at the iteration level instead of the request level. After each decode step, the scheduler checks which sequences have completed, evicts them, and immediately admits new requests from the queue into the freed slots.

def continuous_batching_scheduler(request_queue, model, max_batch=64):
    active_sequences = {}  # slot_id → sequence state
    
    while request_queue or active_sequences:
        # ──── Phase 1: Evict completed sequences ────
        completed = [sid for sid, seq in active_sequences.items()
                     if seq.finished]
        for sid in completed:
            yield_result(active_sequences.pop(sid))
            free_kv_cache(sid)  # immediately reclaim memory

        # ──── Phase 2: Admit new sequences into free slots ────
        while len(active_sequences) < max_batch and request_queue:
            new_req = request_queue.pop()
            slot_id = allocate_kv_cache()
            # Prefill happens inline, mixed with ongoing decode
            kv = model.prefill(new_req.prompt)
            active_sequences[slot_id] = SequenceState(new_req, kv)

        # ──── Phase 3: One decode step for ALL active sequences ────
        if active_sequences:
            tokens = model.decode_step_batched(active_sequences)
            for sid, token in tokens.items():
                active_sequences[sid].append_token(token)
        
        # Loop back → completed sequences freed, new ones admitted
        # GPU is ALWAYS working on a full (or near-full) batch!
The key insight: Continuous batching decouples request lifetime from batch lifetime. A request that generates 10 tokens occupies a slot for 10 iterations, then immediately frees it for a new request. The GPU stays at maximum batch occupancy virtually all the time, instead of waiting for the slowest request in the batch.

  • Memory: Freed as soon as sequence completes
  • Throughput: 2–3× higher than static

Iteration-Level Scheduling

The heart of continuous batching is the iteration-level scheduler. Instead of making scheduling decisions at the request level (when to start/stop a batch), decisions happen at every single decode iteration—typically every 30–70ms for large models.

class IterationScheduler:
    def __init__(self, max_batch_size, max_kv_memory):
        self.active = {}           # running sequences
        self.waiting = deque()     # queue of pending requests
        self.max_batch = max_batch_size
        self.kv_budget = max_kv_memory  # in bytes
        self.kv_used = 0

    def schedule_iteration(self):
        """Called once per decode step (~30-70ms)."""
        
        # 1. Evict finished sequences
        for sid in list(self.active):
            if self.active[sid].is_done():
                freed = self.active[sid].kv_size
                self.kv_used -= freed
                del self.active[sid]
        
        # 2. Admit new sequences (respecting memory budget)
        while (self.waiting
               and len(self.active) < self.max_batch
               and self.can_fit(self.waiting[0])):
            req = self.waiting.popleft()
            seq = self.run_prefill(req)   # prefill inline
            self.active[seq.id] = seq
            self.kv_used += seq.kv_size
        
        # 3. Build decode batch from active sequences
        return list(self.active.values())
    
    def can_fit(self, request):
        # Estimate KV cache needed for this request
        est_kv = self.estimate_kv(request.prompt_len, request.max_output)
        return self.kv_used + est_kv <= self.kv_budget
Preemption support: Advanced schedulers can also preempt running sequences. If a high-priority request arrives and no slots are free, the scheduler can swap a low-priority sequence's KV cache to CPU memory (or recompute it later), admit the high-priority request, and resume the preempted sequence when capacity is available. vLLM implements this with its swap/recompute mechanisms.

The Orca Paper

Continuous batching was formalized in the Orca paper (Yu et al., OSDI 2022). Orca introduced two key concepts that fundamentally changed LLM serving:

1. Iteration-level scheduling: As described above—schedule at the granularity of individual decode steps, not entire request batches.

2. Selective batching: Not all sequences in a batch need the same operations. Some might be in prefill (processing long prompts) while others are in decode (generating tokens). Orca handles this by selectively applying the appropriate operation to each sequence within the batch.

# Orca's selective batching: mixed prefill + decode in one iteration
def selective_batch_step(prefill_seqs, decode_seqs, model):
    # Some sequences are doing prefill (new arrivals)
    # Others are doing decode (ongoing generation)
    
    # Prefill sequences: process full prompt tokens
    for seq in prefill_seqs:
        seq.kv_cache = model.prefill(seq.prompt_tokens)
        seq.phase = 'decode'
    
    # Decode sequences: process one token each
    # Batch all decode sequences together for efficiency
    if decode_seqs:
        input_tokens = [seq.last_token for seq in decode_seqs]
        kv_caches = [seq.kv_cache for seq in decode_seqs]
        
        # Batched decode: one forward pass for all decode sequences
        new_tokens = model.batched_decode(input_tokens, kv_caches)
        
        for seq, token in zip(decode_seqs, new_tokens):
            seq.append(token)

# Key insight: prefill and decode have very different compute profiles
# Chunked prefill (Sarathi, DeepSpeed-FastGen) interleaves prefill
# chunks with decode tokens to balance compute within each iteration
Chunked prefill: A refinement introduced by Sarathi (Agrawal et al., 2023) and adopted by vLLM. Instead of processing an entire long prompt in one iteration (which would stall decode tokens), the prompt is broken into chunks (e.g., 512 tokens). Each iteration processes one chunk of prefill alongside ongoing decode tokens. This prevents prefill from causing latency spikes for in-flight decode sequences.

Production Impact

Continuous batching is now standard in every major LLM serving framework. The throughput improvements are substantial and well-documented:

📊 Throughput Numbers

  • vLLM benchmark (LLaMA-13B): 14–24× throughput vs HuggingFace Transformers with continuous batching + PagedAttention
  • TensorRT-LLM: Reports 2–3× throughput gain from in-flight batching alone
  • Orca paper: 36.9× throughput improvement over FasterTransformer for GPT-2
  • Typical real-world: 2–8× throughput improvement depending on workload

🏗️ Framework Support

  • vLLM: Continuous batching + PagedAttention (default)
  • TensorRT-LLM: In-flight batching (GptManager)
  • TGI (HuggingFace): Continuous batching with Flash Attention
  • DeepSpeed-FastGen: SplitFuse (chunked prefill)
  • SGLang: RadixAttention with continuous batching
  • LMDeploy: Persistent batch scheduling
# Why the throughput gain is so large:
# Static batching with 4 slots, requests averaging 200 output tokens:
# - Longest request: 500 tokens → batch takes 500 steps
# - Total tokens generated: 4 × 200 = 800 (avg)
# - Effective throughput: 800 / 500 = 1.6 tokens/step
# - GPU serving wasted slots: (500-200)/500 = 60% idle time per slot

# Continuous batching with 4 slots, same workload:
# - Slots always full: as one finishes, next starts immediately
# - Total tokens generated per 500 steps: ~4 × 500 = 2000
# - Effective throughput: 2000 / 500 = 4.0 tokens/step
# - 2.5× improvement from batching strategy alone!

# Combined with PagedAttention (eliminates memory waste):
# - Can fit MORE concurrent sequences → larger effective batch
# - Typical combined improvement: 5-15× over naive static serving
The bottom line: Continuous batching is not optional for production LLM serving. It's the difference between serving 10 requests/sec and 50+ requests/sec on the same hardware. Combined with PagedAttention (which we cover in the next post), it enables serving thousands of concurrent users from a single GPU node. If you're running model.generate() in a loop, you're leaving 60–80% of your GPU performance on the table.
