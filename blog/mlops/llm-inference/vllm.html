vLLM: High-Throughput LLM Serving

MLOps Series LLM Inference & Serving

Why vLLM?

Autoregressive LLM inference is fundamentally memory-bound. A 70B-parameter model in FP16 requires 140 GB of GPU memory just for weights—before a single KV-cache entry is allocated. Naive serving wastes 60–80% of allocated KV-cache memory due to fragmentation and over-reservation, leaving expensive GPU cycles idle while memory sits unused.

vLLM, developed at UC Berkeley's Sky Computing Lab, solves this with a radical insight: treat GPU KV-cache memory like an operating system manages virtual memory. The result is PagedAttention, which eliminates fragmentation and enables near-optimal memory utilization.

Key insight: Traditional serving frameworks pre-allocate a contiguous KV-cache per sequence based on the maximum possible length. A 2048-token max length reserves 2048 slots even for a 50-token prompt, wasting ~97% of memory. PagedAttention allocates on demand, block by block.

Naive KV-Cache

  • Contiguous allocation per sequence
  • Reserved for max_seq_len upfront
  • Internal fragmentation: 60–80%
  • Cannot share across sequences
  • Limits batch size severely

vLLM PagedAttention

  • Non-contiguous block allocation
  • Allocates on demand per token
  • Waste < 4% (last-block only)
  • Copy-on-write sharing (beam search)
  • 2–4× larger batch sizes

PagedAttention Deep Dive

PagedAttention partitions each sequence's KV-cache into fixed-size blocks (default: 16 tokens per block). A block table maps each sequence's logical blocks to physical blocks in GPU memory, exactly like a page table in virtual memory systems.

Each physical block stores KV tensors for exactly block_size tokens. For a model with 80 layers and 64 heads of dimension 128, each FP16 block of 16 tokens uses:

2 (K+V) × 80 layers × 64 heads × 128 dim × 16 tokens × 2 bytes = ~40 MB per block

The attention kernel is modified to fetch KV data from non-contiguous physical blocks using the block table as a lookup. The overhead is negligible because GPU memory access is already random for attention computations.

Copy-on-Write for Beam Search: When multiple beams share a common prefix, their block table entries point to the same physical blocks. A block is only copied when one beam diverges—reducing memory by up to 55% for beam search with width 4.

Continuous Batching

Static batching pads all sequences to the longest one and waits for the entire batch to finish. With 32 sequences and a mean length of 100 but max of 2000, this wastes ~95% of compute. vLLM implements iteration-level scheduling:

  1. Each decode step, the scheduler evaluates which sequences have finished (hit EOS or max length).
  2. Finished sequences are evicted immediately, freeing their KV-cache blocks.
  3. Waiting requests are admitted into the running batch if enough blocks are available.
  4. The engine executes a single forward pass for all running sequences, regardless of their individual progress.

This means GPU utilization stays near 100% throughout serving—there is no idle waiting for the longest sequence to finish.

# vLLM scheduler pseudocode (simplified)
class Scheduler:
    def schedule(self):
        # 1. Check running sequences for completion
        finished = [s for s in self.running if s.is_finished()]
        for seq in finished:
            self.free_blocks(seq)
            self.running.remove(seq)

        # 2. Admit new requests from waiting queue
        while self.waiting and self.can_allocate(self.waiting[0]):
            seq = self.waiting.popleft()
            self.allocate_blocks(seq)
            self.running.append(seq)

        # 3. If memory pressure, preempt lowest-priority seqs
        while self.num_free_blocks() < self.watermark:
            victim = self.running.pop()  # FCFS priority
            self.swap_out(victim)  # move KV to CPU

        return SchedulerOutputs(self.running)

When GPU memory is exhausted, vLLM uses preemption via two strategies: swapping (KV-cache to CPU RAM) or recomputation (discard KV-cache and recompute later). Swapping is default for sequences that have generated many tokens; recomputation is preferred for short sequences.

Architecture & Parallelism

vLLM's architecture separates the engine into a frontend (API server, scheduler) and a backend (model runner, GPU workers). This separation enables clean scaling:

Tensor Parallelism (TP): vLLM uses Megatron-style tensor parallelism. Each attention head is assigned to a single GPU. For the FFN layer, the first linear is column-parallel and the second is row-parallel, requiring a single AllReduce per layer. A Llama-70B model with TP=4 on 4×A100-80GB uses ~35 GB per GPU for weights, leaving ~45 GB per GPU for KV-cache.

Pipeline Parallelism (PP): Available since vLLM 0.3+, PP splits layers across GPUs. Combined with TP, you can serve a 70B model on 8×A10G (24GB each): TP=4 × PP=2 gives each GPU ~17.5 GB of weights from a fraction of the layers.

Deployment & Configuration

vLLM provides both a Python API and an OpenAI-compatible HTTP server. The most common deployment is via Docker:

# Launch vLLM with Docker (production-ready)
docker run --runtime nvidia --gpus all \
    -v ~/.cache/huggingface:/root/.cache/huggingface \
    -p 8000:8000 \
    vllm/vllm-openai:latest \
    --model meta-llama/Llama-3.1-70B-Instruct \
    --tensor-parallel-size 4 \
    --gpu-memory-utilization 0.92 \
    --max-model-len 8192 \
    --max-num-seqs 256 \
    --enforce-eager  # disable CUDA graphs for debugging

Key configuration parameters:

Flag Default Description
--tensor-parallel-size1Number of GPUs for tensor parallelism
--gpu-memory-utilization0.90Fraction of GPU memory for KV-cache + weights
--max-model-lenmodel defaultMaximum sequence length (prompt + output)
--max-num-seqs256Maximum concurrent sequences in a batch
--quantizationNoneawq, gptq, squeezellm, fp8
--enable-prefix-cachingFalseAutomatic prefix caching (APC) for shared system prompts
Warning: Setting --gpu-memory-utilization above 0.95 can cause OOM errors during CUDA graph capture. The default 0.90 leaves headroom for activation memory. For production, 0.90–0.92 is optimal.

API Usage & Client Code

vLLM's server exposes an OpenAI-compatible API, making it a drop-in replacement for the OpenAI Python client:

from openai import OpenAI

# Point to vLLM server
client = OpenAI(base_url="http://localhost:8000/v1", api_key="dummy")

# Completion with streaming
stream = client.chat.completions.create(
    model="meta-llama/Llama-3.1-70B-Instruct",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Explain PagedAttention."}
    ],
    max_tokens=512,
    temperature=0.7,
    stream=True
)

for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")

For offline batch inference (e.g., evaluation workloads), use the Python API directly:

from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",
    tensor_parallel_size=1,
    gpu_memory_utilization=0.90
)
params = SamplingParams(temperature=0.0, max_tokens=256)

prompts = ["What is KV-cache?", "Explain tensor parallelism."]
outputs = llm.generate(prompts, params)

for out in outputs:
    print(out.outputs[0].text)

Benchmarks & Performance

vLLM's improvements come from two axes: higher throughput (more tokens/sec via better batching and memory utilization) and lower waste (more sequences fit in GPU memory simultaneously).

Llama-2-7B (1×A100-80GB)

  • HF Transformers: ~8 req/s
  • vLLM: ~24 req/s (3× faster)
  • Input: 128 tokens, Output: 128 tokens
  • Batch size: 64 concurrent requests

Llama-2-13B (1×A100-80GB)

  • HF Transformers: ~4.5 req/s
  • vLLM: ~15 req/s (3.3× faster)
  • Input: 256 tokens, Output: 256 tokens
  • Memory util: 92% vs 35%

Llama-2-70B (4×A100-80GB, TP=4)

  • TGI: ~6 req/s
  • vLLM: ~14 req/s (2.3× faster)
  • ShareGPT dataset (variable lengths)
  • Continuous batching advantage shines

High-Concurrency (Llama-7B)

  • 256 concurrent requests
  • vLLM throughput: 2,400 tok/s
  • HF generate: 100 tok/s (24× gap)
  • vLLM keeps all 256 in-flight; HF serializes
Prefix Caching (APC): When many requests share a system prompt, --enable-prefix-caching reuses KV-cache blocks for common prefixes. For chatbot workloads with a 1000-token system prompt, this reduces TTFT by 40–60% and increases throughput by 1.5–2×.

vLLM also supports speculative decoding, chunked prefill, and multi-LoRA serving—making it the most feature-complete open-source LLM serving engine as of 2024. Its OpenAI-compatible API means zero code changes when migrating from OpenAI's hosted models to self-hosted inference.
