MLOps Series LLM Inference & Serving

Why Disaggregate Prefill and Decode?

Traditional LLM serving colocates both the prefill (prompt processing) and decode (token generation) phases on the same GPU. This seems natural — one model, one machine — but it creates a fundamental resource mismatch. Prefill is compute-bound, saturating tensor cores with massive matrix multiplications across all input tokens simultaneously. Decode is memory-bandwidth-bound, reading the entire model weight matrix for each single token generated autoregressively.

When both phases share a GPU, you face an impossible optimization problem: tuning for prefill throughput (large batches, high compute utilization) actively hurts decode latency (small sequential operations that need fast memory access). The GPU is either underutilizing compute during decode or creating head-of-line blocking during prefill.

Core insight: Prefill achieves 60-80% MFU (Model FLOPS Utilization) on compute-optimized hardware, while decode rarely exceeds 5% MFU but needs 2-3 TB/s memory bandwidth. Disaggregation lets you match each phase to its optimal hardware profile.

Prefill vs Decode: The Computational Asymmetry

Understanding the arithmetic intensity difference is key. For a Llama-2 70B model with 8192 hidden dimensions and 64 attention heads:

Prefill phase processes all N input tokens in parallel. The QKV projection for one layer performs matrix multiplications of shape [N, 8192] × [8192, 24576]. With N=2048 tokens, that's ~103 billion FLOPs per layer, across 80 layers — roughly 8.2 TFLOPs total. An H100 with 989 TFLOPS FP16 can do this in ~8.3ms. The arithmetic intensity is 2×N×d / (N+d) ≈ 2×d ≈ 16384 ops/byte — deeply compute-bound.

Decode phase generates one token at a time. The same QKV projection becomes [1, 8192] × [8192, 24576] — only ~402 million FLOPs per layer but still reading the same 24576×8192×2 = 384 MB weight matrix. Arithmetic intensity drops to ~2 ops/byte — squarely memory-bandwidth-bound. Even with batching 64 requests, intensity only reaches ~128 ops/byte.

⚡ Prefill Phase

  • Bound by: Compute (TFLOPS)
  • Arithmetic intensity: ~16K ops/byte
  • Ideal hardware: H100 SXM (989 TFLOPS FP16)
  • GPU utilization: 60-80% MFU achievable
  • Parallelism: All input tokens processed simultaneously
  • Duration: 50-500ms (proportional to prompt length)

🔄 Decode Phase

  • Bound by: Memory bandwidth (TB/s)
  • Arithmetic intensity: ~2 ops/byte (batch=1)
  • Ideal hardware: A10G / L4 (high BW/$ ratio)
  • GPU utilization: 2-5% MFU typical
  • Parallelism: One token at a time per sequence
  • Duration: 20-50ms per token × hundreds of tokens

Disaggregated Serving Architecture

In a disaggregated system, dedicated prefill nodes handle prompt processing while separate decode nodes handle autoregressive generation. A routing layer directs incoming requests to available prefill nodes, then migrates the computed KV cache to a decode node for token generation.

The flow is: (1) Router receives prompt → (2) Assigns to prefill node with lowest queue depth → (3) Prefill node computes full KV cache for the prompt → (4) KV cache is transferred to a decode node → (5) Decode node generates tokens autoregressively, streaming back to client.

KV Cache Transfer: The Critical Path

The KV cache transfer between prefill and decode nodes is the key engineering challenge. For Llama-2 70B with 2048 input tokens, the KV cache size per request is:

# KV cache size calculation for Llama-2 70B
num_layers     = 80
num_kv_heads   = 8          # GQA: 8 KV heads (not 64)
head_dim       = 128
seq_len        = 2048
dtype_bytes    = 2          # FP16

# Per-layer: 2 (K+V) × num_kv_heads × seq_len × head_dim × dtype
per_layer = 2 * 8 * 2048 * 128 * 2    # = 8,388,608 bytes = 8 MB
total_kv  = per_layer * 80              # = 640 MB per request

# Transfer time at different bandwidths:
# 100 Gbps (InfiniBand HDR):  ~51 ms
# 200 Gbps (InfiniBand NDR):  ~25 ms
# 400 Gbps (NVLink / NDR×2):  ~12 ms

# Compare to prefill compute time: ~80ms on 4×H100
# Transfer overhead: 15-60% of prefill time
Transfer overhead matters: If KV transfer takes 50ms and prefill takes 80ms, you're adding 62% latency overhead to TTFT. For latency-sensitive applications, you need ≥200 Gbps interconnect or the disaggregation penalty outweighs the benefit.

Strategies to reduce transfer overhead:

  • KV cache compression: Quantize KV cache to INT8 or INT4 during transfer (2-4× reduction), dequantize on decode node. Research shows <1% quality loss with INT8 KV cache.
  • Pipelined transfer: Start transferring KV cache layer-by-layer as prefill completes each layer, overlapping compute and network. Reduces effective transfer latency by 40-60%.
  • Proximity placement: Co-locate prefill and decode nodes on the same rack with NVLink bridges or NVSwitch for sub-10ms transfer.
  • Selective transfer: Only transfer the KV cache for the attention-heavy layers; recompute shallow layers on the decode node.

Splitwise & DistServe Approaches

Splitwise (UW, 2024) was among the first to formalize disaggregated inference. Key contributions:

  • Profiled the compute/memory asymmetry across 8 popular LLMs and showed 2-3× cost reduction potential
  • Designed a mixed-pool scheduler that assigns prefill to high-compute nodes (H100) and decode to cost-efficient nodes (A10G)
  • Demonstrated 1.4× higher throughput at the same SLO compared to colocated serving on Llama-2 70B
  • Showed that even with network transfer overhead, disaggregation wins when prompt lengths exceed ~512 tokens

DistServe (PKU, 2024) extended this with a goodput-optimized placement algorithm:

  • Formulated GPU placement as a mixed-integer linear program (MILP) optimizing for goodput — requests meeting both TTFT and TPOT SLOs
  • Introduced prefill-decode parallelism: pipeline KV transfer with the next prefill computation
  • Achieved 2.0× higher goodput than vLLM colocated serving under strict SLOs (TTFT < 200ms, TPOT < 50ms)
  • Showed 4.48× goodput improvement over colocated serving on a Llama-70B chatbot workload

Splitwise

  • Focus: Hardware heterogeneity
  • Scheduler: Queue-depth based routing
  • Transfer: Full KV cache migration
  • Result: 1.4× throughput, 30% cost reduction
  • Best for: Mixed GPU fleets

DistServe

  • Focus: SLO-aware goodput optimization
  • Scheduler: MILP-based placement
  • Transfer: Pipelined layer-by-layer
  • Result: 2-4.48× goodput improvement
  • Best for: Strict latency SLO workloads

Cost Savings & Latency Tradeoffs

The economics of disaggregation are compelling. Consider serving Llama-2 70B at 1000 requests/minute with average 1024-token prompts and 512-token completions:

Colocated (Baseline)

  • 8× H100 SXM nodes (4 GPUs each)
  • Cost: 8 × $30/hr = $240/hr
  • Throughput: ~125 req/min per node
  • P50 TTFT: 180ms
  • P50 TPOT: 35ms
  • GPU utilization: 45% compute, 60% memory BW

Disaggregated

  • 3× H100 SXM (prefill) + 6× A10G (decode)
  • Cost: 3×$30 + 6×$4 = $114/hr
  • Throughput: ~333 prefill/min + ~167 decode/min
  • P50 TTFT: 120ms (faster — dedicated prefill)
  • P50 TPOT: 40ms (slightly slower — cheaper HW)
  • GPU utilization: 75% compute, 80% memory BW
52% cost reduction ($240/hr → $114/hr) while maintaining comparable latency. The key: H100 compute is wasted during decode, and A10G memory bandwidth is sufficient for autoregressive generation at 40ms/token.

The tradeoff space depends on workload characteristics:

  • Long prompts, short outputs (summarization): Disaggregation saves 40-50%. Prefill dominates, so specializing compute pays off significantly.
  • Short prompts, long outputs (creative writing): Savings drop to 15-25%. Decode dominates, and KV transfer overhead is amortized less.
  • Balanced prompts/outputs (chat): Savings of 30-40%. Best overall tradeoff for disaggregation.
  • Very short prompts (<128 tokens): Disaggregation may hurt — transfer overhead exceeds prefill compute time.

Implementation Patterns

A production disaggregated serving system requires careful orchestration. Here's a reference architecture using vLLM with a custom router:

# Disaggregated inference router
class DisaggregatedRouter:
    def __init__(self, prefill_pool, decode_pool):
        self.prefill_pool = prefill_pool    # List[PrefillNode]
        self.decode_pool  = decode_pool     # List[DecodeNode]
        self.kv_store     = DistributedKVStore(backend="rdma")

    async def serve_request(self, prompt: str, params: SamplingParams):
        # 1. Route to least-loaded prefill node
        prefill_node = min(self.prefill_pool, key=lambda n: n.queue_depth)

        # 2. Run prefill — returns KV cache reference
        kv_ref = await prefill_node.prefill(prompt)

        # 3. Select decode node by available KV cache memory
        decode_node = min(self.decode_pool,
            key=lambda n: n.active_sequences / n.max_sequences)

        # 4. Transfer KV cache via RDMA (pipelined, layer-by-layer)
        await self.kv_store.transfer(
            src=prefill_node, dst=decode_node,
            kv_ref=kv_ref, pipeline=True
        )

        # 5. Start decode — streams tokens back
        async for token in decode_node.decode(kv_ref, params):
            yield token

        # 6. Free KV cache on decode node when done
        decode_node.release_kv(kv_ref)

Key implementation considerations for production deployment:

  • Fault tolerance: If a decode node fails mid-generation, the KV cache must be re-created. Keep prefill nodes stateless so any node can re-prefill. Store prompts in the router until generation completes.
  • Load balancing: Prefill queue depth is the best signal for routing. Decode nodes should be balanced by active sequence count relative to their KV cache capacity.
  • Scaling ratio: Typical ratio is 1 prefill node : 2-4 decode nodes, depending on average output length. Monitor and adjust dynamically.
  • KV cache memory management: Decode nodes need aggressive PagedAttention or similar block-level memory management. KV cache fragmentation is the primary bottleneck at scale.
Production tip: Start with colocated serving and profile your workload. Disaggregate only when (a) prompt lengths average >512 tokens, (b) you have heterogeneous GPU inventory, and (c) your interconnect supports ≥100 Gbps. The operational complexity is significant — make sure the savings justify it.
