Text Generation Inference (TGI)

MLOps Series LLM Inference & Serving

TGI Overview

Text Generation Inference (TGI) is HuggingFace's production-grade inference server, written primarily in Rust with Python model implementations. It powers HuggingFace's Inference Endpoints and the Inference API, handling millions of requests daily across thousands of model deployments.

TGI was designed with a specific philosophy: take the best optimizations from the research community (Flash Attention, continuous batching, quantization) and package them in a turnkey Docker container that works out of the box with any HuggingFace model.

Core Features

  • Flash Attention 2 (Dao-AILab)
  • Continuous batching with token streaming
  • Paged Attention for KV-cache management
  • Tensor parallelism with NCCL
  • GPTQ / AWQ / EETQ / bitsandbytes quantization
  • Speculative decoding (draft model)

Operational Features

  • A/B testing via watermarking
  • Token-level streaming via SSE
  • Structured output (grammar constraints)
  • Built-in Prometheus metrics
  • Safetensors weight loading
  • LoRA adapter hot-swapping
Model Support: TGI supports Llama, Mistral, Mixtral, Falcon, StarCoder, GPT-NeoX, Phi, Gemma, Cohere Command-R, DBRX, and more. The architecture auto-detection loads the correct implementation from the model's config.json.

Architecture Deep Dive

TGI uses a two-process architecture: a Rust-based router handles HTTP/gRPC traffic, validation, batching logic, and token streaming, while a Python model server runs the actual GPU computations. They communicate via gRPC internally.

The Rust router performs input validation (length checks, stop sequence parsing), tokenization using HuggingFace tokenizers (Rust-native, ~10× faster than Python), and manages the request queue. When the batch scheduler determines capacity, it sends a batch to the Python model server via gRPC, which returns generated tokens one at a time for streaming.

Flash Attention & Paged KV-Cache

TGI integrates Flash Attention 2 (Tri Dao, 2023), which restructures the attention computation to be IO-aware. Standard attention materializes the full N×N attention matrix in HBM, using O(N²) memory. Flash Attention computes attention in tiles that fit in SRAM (on-chip), reducing HBM reads/writes by ~5–10×.

Standard Attention

  • Q × K^T → S (N×N in HBM)
  • softmax(S) → P (N×N in HBM)
  • P × V → O (read N×N from HBM)
  • Memory: O(N²)
  • HBM accesses: O(N² × d)

Flash Attention 2

  • Tile Q,K,V into SRAM blocks
  • Compute attention per-tile with online softmax
  • Never materialize full N×N matrix
  • Memory: O(N)
  • HBM accesses: O(N² × d / M), M=SRAM size

TGI also implements Paged Attention (same concept as vLLM) for KV-cache management. The KV-cache is divided into fixed-size blocks, and a block table maps logical token positions to physical memory locations. This eliminates the internal fragmentation that occurs with contiguous KV-cache allocation.

# TGI's attention computation (simplified)
# Uses Flash Attention for prefill, Paged Attention for decode

def attention_forward(query, key, value, kv_cache, block_tables, is_prefill):
    if is_prefill:
        # Prefill: process full prompt with Flash Attention 2
        # All tokens attend to each other — compute-bound
        output = flash_attn_varlen_func(
            query, key, value,
            cu_seqlens_q, cu_seqlens_k,
            max_seqlen_q, max_seqlen_k,
            causal=True
        )
    else:
        # Decode: single new token attends to all cached KV
        # Memory-bound — Paged Attention fetches from block table
        output = paged_attention_v2(
            query, kv_cache,
            block_tables, context_lens,
            block_size=16
        )
    return output
Prefill vs Decode: The prefill phase (processing the prompt) is compute-bound — Flash Attention's tiling maximizes FLOPs/byte. The decode phase (generating tokens one by one) is memory-bound — Paged Attention minimizes wasted memory so more sequences fit in a batch, improving throughput via batching.

Continuous Batching & Scheduling

TGI implements continuous batching (also called "in-flight batching") where new requests can join a running batch at any decode step. The Rust router manages two key parameters:

  • --max-batch-prefill-tokens (default: 4096): Maximum number of tokens that can be prefilled in a single batch. This bounds prefill latency — a large prefill batch delays in-progress decode steps. Setting this to 4096 means at most ~4 prompts of 1024 tokens can prefill simultaneously.
  • --max-batch-total-tokens: Total budget for KV-cache tokens across all sequences in the batch. On an A100-80GB with Llama-70B (TP=2, FP16), ~20 GB is available for KV-cache, supporting roughly 16,000 total tokens across all sequences.
  • --waiting-served-ratio (default: 1.2): Controls when to interrupt decoding to admit prefill. A ratio of 1.2 means TGI prefers prefilling waiting requests once the ratio of waiting to served exceeds 1.2.
# TGI batch scheduling (conceptual)
class BatchScheduler:
    def next_batch(self):
        # Running decode sequences
        decode_seqs = self.get_running()
        decode_tokens = sum(s.current_len for s in decode_seqs)

        # Budget remaining for new prefills
        budget = self.max_batch_total_tokens - decode_tokens

        # Admit waiting requests that fit
        prefill_batch = []
        prefill_tokens = 0
        for req in self.waiting_queue:
            if prefill_tokens + req.input_len > self.max_batch_prefill_tokens:
                break
            if decode_tokens + prefill_tokens + req.input_len > budget:
                break
            prefill_batch.append(req)
            prefill_tokens += req.input_len

        return CombinedBatch(decode_seqs, prefill_batch)

TGI batches prefill and decode together in a single forward pass using variable-length attention (via flash_attn_varlen_func). Prefill sequences process their full input, while decode sequences process a single token each. This is more efficient than separating prefill and decode into distinct passes.

Quantization Support

TGI supports multiple quantization backends to reduce memory and improve throughput:

GPTQ (4-bit)

  • Post-training quantization via calibration
  • INT4 weights, FP16 activations
  • Llama-70B: 140 GB → ~35 GB
  • Fits on 1× A100-80GB
  • ~5% quality loss on benchmarks

AWQ (4-bit)

  • Activation-aware weight quantization
  • Protects salient weights (top 1%)
  • Slightly better quality than GPTQ
  • Same memory savings (~4× reduction)
  • Faster kernel via FP16 accumulation

EETQ (INT8)

  • Per-channel INT8 quantization
  • Llama-70B: 140 GB → ~70 GB
  • Minimal quality loss (< 1%)
  • Uses CUTLASS INT8 GEMM kernels
  • Good for latency-sensitive workloads

bitsandbytes (NF4)

  • NormalFloat4 quantization
  • Double quantization of scales
  • Popular for QLoRA fine-tuning
  • Higher overhead per token
  • Best for memory-constrained GPUs
# Launch TGI with GPTQ quantized model
docker run --gpus all -p 8080:80 \
    -v /data:/data \
    ghcr.io/huggingface/text-generation-inference:latest \
    --model-id TheBloke/Llama-2-70B-Chat-GPTQ \
    --quantize gptq \
    --num-shard 2 \
    --max-input-length 2048 \
    --max-total-tokens 4096

Deployment & Configuration

TGI is distributed as a Docker image. For production, the key is tuning batch parameters to match your hardware and workload:

# Production deployment: Llama-3.1-70B on 4×A100-80GB
docker run --gpus all -p 8080:80 \
    -v /models:/data \
    -e HUGGING_FACE_HUB_TOKEN=$HF_TOKEN \
    ghcr.io/huggingface/text-generation-inference:2.0 \
    --model-id meta-llama/Llama-3.1-70B-Instruct \
    --num-shard 4 \
    --max-input-length 4096 \
    --max-total-tokens 8192 \
    --max-batch-prefill-tokens 4096 \
    --max-batch-total-tokens 32000 \
    --max-concurrent-requests 128 \
    --dtype float16 \
    --port 80
Parameter Default Guidance
--num-shard1Set to number of GPUs for tensor parallelism
--max-input-length1024Max tokens in a single prompt
--max-total-tokens2048Max prompt + generation per request
--max-batch-prefill-tokens4096Caps prefill latency; lower = faster TTFT
--max-batch-total-tokensautoTotal KV-cache budget; auto-calculated from GPU mem
--max-concurrent-requests128Reject requests beyond this queue depth
Tuning tip: If TTFT (time to first token) is too high, lower --max-batch-prefill-tokens. This limits how many prefill tokens run in one forward pass, reducing the delay before decode steps resume. Typical production value: 2048–4096 for chat workloads.

HTTP & gRPC APIs

TGI exposes a REST API for text generation with built-in SSE streaming:

# Streaming request with curl
curl http://localhost:8080/generate_stream \
    -X POST \
    -H "Content-Type: application/json" \
    -d '{
        "inputs": "Explain continuous batching in LLM inference",
        "parameters": {
            "max_new_tokens": 256,
            "temperature": 0.7,
            "top_p": 0.95,
            "repetition_penalty": 1.1,
            "do_sample": true,
            "stop": ["\n\n"]
        }
    }'

# Response: Server-Sent Events stream
# data:{"token":{"id":1,"text":"Con","logprob":-0.12,...},...}
# data:{"token":{"id":2,"text":"tinuous","logprob":-0.08,...},...}

For Python clients, HuggingFace provides a typed client library:

from huggingface_hub import InferenceClient

client = InferenceClient(model="http://localhost:8080")

# Non-streaming
response = client.text_generation(
    prompt="What is Flash Attention?",
    max_new_tokens=256,
    temperature=0.7
)
print(response)

# Streaming
for token in client.text_generation(
    prompt="Explain paged attention",
    max_new_tokens=512,
    stream=True
):
    print(token, end="")

TGI also provides a /metrics endpoint exposing Prometheus metrics including:

  • tgi_request_duration_seconds — end-to-end latency histogram
  • tgi_batch_current_size — current batch size gauge
  • tgi_queue_size — waiting queue depth
  • tgi_request_generated_tokens_total — total tokens generated
  • tgi_request_inference_duration_seconds — pure GPU time
Watermarking: TGI supports A/B watermarking via the watermark parameter. It embeds a statistical signal in the token distribution (green/red list approach from Kirchenbauer et al., 2023) that can be detected with high confidence without affecting text quality perceptibly.

TGI's Rust router, Flash Attention integration, and first-class HuggingFace Hub support make it the easiest path from a HuggingFace model to production serving. For teams already in the HuggingFace ecosystem, TGI offers the shortest time-to-deployment with strong performance defaults.
