Load Balancing for LLM Services

MLOps Series LLM Inference & Serving

Why Standard Load Balancing Fails for LLMs

Traditional load balancers (round-robin, least-connections, random) assume requests are roughly equivalent in cost. For web APIs, this works — most requests complete in 10-200ms with similar resource consumption. LLM inference shatters this assumption in every dimension:

  • Request duration varies 100×: A short completion (32 tokens, 128-token prompt) finishes in ~200ms. A long generation (2048 tokens, 4096-token prompt) takes 30+ seconds. Round-robin distributes request count evenly but leaves one replica overloaded with long requests while another idles.
  • Request resource cost varies 50×: A 128-token prompt uses ~10MB of KV cache memory. A 32K-token prompt uses ~2.5GB. The short request leaves 99% of GPU memory available; the long one can prevent any new requests from being accepted.
  • Streaming responses are long-lived connections: LLM responses stream tokens over 5-60 seconds via SSE. Least-connections routing counts each stream equally, but a stream with 2000 remaining tokens occupies the GPU 20× longer than one with 100 remaining tokens.
  • Prefill and decode have different resource profiles: During prefill, the GPU is compute-saturated. During decode, it's memory-bandwidth-bound. A replica with 10 requests all in decode phase has fundamentally different capacity than one with 10 requests in prefill.

Request-Aware Routing

Request-aware routing considers the cost of each request — estimated token count, expected generation length, and current replica load — to make intelligent routing decisions. The router inspects the request before forwarding it.

# Request-aware LLM load balancer
class LLMLoadBalancer:
    def __init__(self, replicas: List[Replica]):
        self.replicas = replicas

    def estimate_cost(self, request) -> float:
        # Estimate token count from request (tokenizer or heuristic)
        prompt_tokens = len(request.prompt) // 4  # rough: ~4 chars/token
        max_output    = request.max_tokens or 512
        # Weighted cost: prefill is bursty, decode is sustained
        return prompt_tokens * 1.0 + max_output * 2.5

    def get_replica_load(self, replica) -> float:
        # Composite load score from multiple signals
        queue_score  = replica.queue_depth / replica.max_queue
        kv_score     = replica.kv_cache_usage          # 0.0-1.0
        active_score = replica.active_tokens / replica.max_tokens
        return 0.4 * queue_score + 0.35 * kv_score + 0.25 * active_score

    def route(self, request) -> Replica:
        cost = self.estimate_cost(request)
        # Select replica with lowest effective load after adding this request
        return min(self.replicas,
            key=lambda r: self.get_replica_load(r) + cost / r.capacity)
Key insight: Request-aware routing reduces P99 TTFT by 3-5× compared to round-robin in production workloads with high request variance. The improvement comes from avoiding the "long-request-on-loaded-replica" pathology that dominates tail latency.

The cost estimation doesn't need to be perfect — even a rough heuristic (prompt length × 1 + max_tokens × 2.5) dramatically outperforms round-robin. The max_tokens parameter in the request is the best predictor of total request cost, since decode time dominates for most workloads.

Prefix Locality Routing

Prefix locality is a powerful optimization unique to LLM serving: route requests that share the same system prompt or prefix to the same replica, so the KV cache for the shared prefix is already computed and cached. This eliminates redundant prefill computation.

# Prefix-locality-aware router
class PrefixLocalityRouter:
    def __init__(self, replicas, num_prefix_buckets=64):
        self.replicas = replicas
        self.prefix_map = {}   # prefix_hash → replica_id

    def hash_prefix(self, prompt: str, prefix_len: int = 512) -> str:
        # Hash the first N characters as the "prefix key"
        # Captures system prompt + few-shot examples
        prefix = prompt[:prefix_len]
        return hashlib.md5(prefix.encode()).hexdigest()

    def route(self, request) -> Replica:
        prefix_hash = self.hash_prefix(request.prompt)

        # Check if we have a preferred replica for this prefix
        if prefix_hash in self.prefix_map:
            preferred = self.prefix_map[prefix_hash]
            # Use preferred replica if not overloaded (<80% capacity)
            if preferred.kv_cache_usage < 0.80:
                return preferred

        # Fallback: route to least-loaded replica
        target = min(self.replicas, key=lambda r: r.queue_depth)
        self.prefix_map[prefix_hash] = target
        return target

The impact of prefix locality depends on workload characteristics:

High Impact Workloads

  • RAG pipelines: Shared system prompt + retrieval template (2K-4K tokens). Cache hit rate: 80-95%. TTFT reduction: 60-80%.
  • Tool-use agents: Fixed tool definitions + formatting instructions (~1K tokens). Cache hit rate: 90%+. Prefill savings: 50-70%.
  • Multi-tenant SaaS: Each tenant has a unique system prompt. Route all traffic for tenant A to replica 1, tenant B to replica 2, etc.

Low Impact Workloads

  • Open-ended chat: Each conversation has a unique context. No shared prefix beyond a short system prompt. Cache hit rate: <10%.
  • Code completion: Prefix is the code context, which changes with every keystroke. Prefix caching is ineffective.
  • Single-turn diverse queries: No prefix commonality. Random routing performs equally well.
Quantified impact: For a RAG workload with a 2048-token system prompt and 85% cache hit rate, prefix locality routing reduces average TTFT from 350ms to 80ms (4.4× improvement) and reduces total GPU compute by 40%. The router overhead is negligible (<1ms per request for hash computation and lookup).

Shortest-Queue Routing

Shortest-queue (or least-outstanding-requests) routing is the simplest effective strategy for LLMs. Unlike round-robin, it naturally adapts to heterogeneous request costs — replicas processing long requests accumulate fewer completions and thus receive fewer new requests.

However, naive shortest-queue has a flaw: it only counts requests, not tokens. A replica with 5 short requests (128 tokens each, total 640 tokens) appears equally loaded as one with 5 long requests (4096 tokens each, total 20,480 tokens). The token-weighted variant fixes this:

# Token-weighted shortest-queue routing
def token_weighted_route(replicas, request):
    def effective_load(replica):
        # Sum of estimated remaining tokens across all active sequences
        remaining = sum(
            seq.max_tokens - seq.generated_tokens
            for seq in replica.active_sequences
        )
        # Add queued requests' estimated total tokens
        queued = sum(
            req.prompt_tokens + req.max_tokens
            for req in replica.queue
        )
        return (remaining + queued) / replica.max_throughput_tokens

    return min(replicas, key=effective_load)

# Comparison on production traffic (Llama-2 70B, 6 replicas):
# Round-robin:        P50 TTFT = 220ms, P99 TTFT = 4,200ms
# Shortest-queue:     P50 TTFT = 180ms, P99 TTFT = 1,800ms
# Token-weighted SQ:  P50 TTFT = 160ms, P99 TTFT =   650ms

Token-weighted shortest-queue reduces P99 TTFT by 6.5× compared to round-robin. The P50 improvement is modest (27%), but the tail latency improvement is dramatic because it eliminates the worst-case scenario of stacking long requests on one replica.

Session Affinity for Multi-Turn Conversations

Multi-turn conversations (chatbots, agents) benefit from session affinity — routing all turns of a conversation to the same replica. This enables two optimizations:

  • KV cache reuse across turns: The KV cache from the previous turn is still in GPU memory. The next turn only needs to compute KV for the new user message and assistant response, not the entire conversation history. For a 10-turn conversation with 8K context, this saves 70-90% of prefill compute on turns 2+.
  • Reduced memory allocation: The existing KV cache memory blocks are already allocated. No new PagedAttention block allocation needed for the historical context.
# Session-affinity router with fallback
class SessionAffinityRouter:
    def __init__(self, replicas):
        self.replicas = replicas
        self.sessions = {}      # session_id → (replica_id, last_active)
        self.ttl = 300           # 5 min session TTL

    def route(self, request) -> Replica:
        session_id = request.headers.get("X-Session-ID")

        if session_id and session_id in self.sessions:
            replica_id, last_active = self.sessions[session_id]
            if time.time() - last_active < self.ttl:
                replica = self.get_replica(replica_id)
                if replica.is_healthy and replica.kv_cache_usage < 0.90:
                    self.sessions[session_id] = (replica_id, time.time())
                    return replica
                # Fallback: replica unhealthy or overloaded

        # New session or expired: use token-weighted shortest-queue
        target = token_weighted_route(self.replicas, request)
        if session_id:
            self.sessions[session_id] = (target.id, time.time())
        return target
Affinity vs balance tradeoff: Strong session affinity can create load imbalance if some sessions are much heavier than others (e.g., one user sends 50 messages while others send 2). Always include a load-based override: break affinity when the preferred replica's KV cache exceeds 90% utilization. The cost of re-prefilling the conversation is lower than the cost of SLO breaches from overload.

Health Checks & Model Readiness Probes

LLM serving requires more nuanced health checks than standard web services. A replica can be "alive" (process running) but not "ready" (model not loaded) or "overloaded" (accepting requests but with degraded latency).

Liveness Probe

Checks if the process is alive. Failure triggers pod restart.

livenessProbe:
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 180
  periodSeconds: 30
  failureThreshold: 3

initialDelaySeconds: 180 — critical for LLMs. Model loading takes 1-3 minutes. Without this, Kubernetes restarts the pod before the model is loaded, creating an infinite restart loop.

Readiness Probe

Checks if the model is loaded and ready to serve. Failure removes pod from service endpoints.

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8000
  initialDelaySeconds: 60
  periodSeconds: 10
  successThreshold: 1

The /health/ready endpoint should return 200 only after the model weights are fully loaded into GPU memory and a warm-up inference has completed successfully.

Beyond basic liveness/readiness, implement a load-aware health check that returns degraded status when the replica is overloaded:

# Custom health endpoint for LLM serving
@app.get("/health/detailed")
def detailed_health():
    status = {
        "model_loaded": engine.model_loaded,
        "gpu_healthy":  check_gpu_errors() == 0,
        "queue_depth":  engine.waiting_queue_size,
        "kv_cache_pct": engine.kv_cache_utilization,
        "active_seqs":  engine.num_running_sequences,
        "capacity":     engine.max_sequences,
    }
    # Return 200 if healthy, 429 if overloaded, 503 if unhealthy
    if not status["model_loaded"] or not status["gpu_healthy"]:
        return JSONResponse(status, status_code=503)
    if status["kv_cache_pct"] > 0.90 or status["queue_depth"] > 50:
        return JSONResponse(status, status_code=429)
    return JSONResponse(status, status_code=200)

Envoy / Istio Configuration for LLM Workloads

Envoy is the most common data-plane proxy for LLM serving (used directly or via Istio service mesh). Key configurations for LLM workloads differ significantly from standard web traffic:

# Envoy cluster config for LLM backend
clusters:
- name: llm_backend
  type: STRICT_DNS
  lb_policy: LEAST_REQUEST    # NOT round_robin
  least_request_lb_config:
    choice_count: 5            # Sample 5 replicas, pick least loaded
  load_assignment:
    endpoints:
    - lb_endpoints:
      - endpoint:
          address:
            socket_address:
              address: vllm-service
              port_value: 8000

  # Critical: increase timeouts for long generations
  connect_timeout: 5s
  typed_extension_protocol_options:
    envoy.extensions.upstreams.http.v3.HttpProtocolOptions:
      common_http_protocol_options:
        idle_timeout: 300s       # 5 min for streaming responses
      explicit_http_config:
        http2_protocol_options:
          max_concurrent_streams: 100

  # Health checking
  health_checks:
  - timeout: 10s
    interval: 15s
    healthy_threshold: 1
    unhealthy_threshold: 3
    http_health_check:
      path: /health/ready

  # Circuit breaking for GPU memory protection
  circuit_breakers:
    thresholds:
    - max_connections: 200
      max_pending_requests: 100  # Queue limit per replica
      max_requests: 150          # Max concurrent requests
      max_retries: 2

Critical Envoy settings for LLM workloads:

  • idle_timeout: 300s — Default is 60s, which will kill streaming responses that take longer. A 2048-token generation at 30 tok/s takes 68 seconds, and many models generate longer outputs.
  • LEAST_REQUEST with choice_count: 5 — "Power of two choices" extended to 5 choices. Samples 5 random backends and picks the one with fewest active requests. Good approximation of shortest-queue without requiring global state.
  • Circuit breakers: max_pending_requests: 100 — Prevents KV cache OOM by limiting the queue depth per replica. When the circuit breaker trips, Envoy returns 503 immediately rather than queuing indefinitely. The client can retry on a different replica.
  • max_retries: 2 — Keep retries low. LLM requests are expensive — retrying a 4096-token prefill that failed due to OOM will likely fail again. Better to shed load gracefully.
Production architecture: For most LLM deployments, use Envoy's LEAST_REQUEST as the baseline, add a custom external processing filter (ext_proc) for prefix-locality routing when KV cache hit rates justify the complexity, and rely on circuit breakers as the safety net against GPU memory exhaustion. This combination handles 90%+ of production LLM load balancing needs.
