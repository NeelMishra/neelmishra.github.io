MLOps Series LLM Inference & Serving

What Gets Cached

In a standard multi-head attention layer, the input hidden state h is projected into three tensors: Query (Q), Key (K), and Value (V). During autoregressive generation, Q for the current token attends over K and V from all previous tokens. Without caching, this means recomputing K and V for the entire sequence history at every decode step.

The KV cache stores the Key and Value tensors for every token, at every layer, for every attention head. The Query tensor is not cached—it's only needed for the current token's attention computation and is discarded immediately.

# What a single attention layer computes per token
def attention_layer(hidden, kv_cache_layer, layer_idx):
    # Project current token's hidden state
    q = hidden @ W_q[layer_idx]   # [batch, 1, n_heads × head_dim]
    k = hidden @ W_k[layer_idx]   # [batch, 1, n_kv_heads × head_dim]
    v = hidden @ W_v[layer_idx]   # [batch, 1, n_kv_heads × head_dim]

    # Reshape to multi-head format
    q = q.view(batch, 1, n_heads, head_dim)
    k = k.view(batch, 1, n_kv_heads, head_dim)
    v = v.view(batch, 1, n_kv_heads, head_dim)

    # Append new K, V to cache (this is the caching step)
    kv_cache_layer['k'] = torch.cat([kv_cache_layer['k'], k], dim=1)
    kv_cache_layer['v'] = torch.cat([kv_cache_layer['v'], v], dim=1)

    # Attend: Q (1 token) × K^T (all cached tokens)
    all_k = kv_cache_layer['k']   # [batch, seq_so_far, n_kv_heads, head_dim]
    all_v = kv_cache_layer['v']

    attn_weights = (q @ all_k.transpose(-1, -2)) / sqrt(head_dim)
    attn_out = softmax(attn_weights) @ all_v
    return attn_out @ W_o[layer_idx]
Per-layer, per-head storage: Each of the L layers independently maintains its own K and V cache. For a model with 80 layers (like LLaMA-70B), there are 80 separate K tensors and 80 separate V tensors in the cache. Each K/V tensor has shape [batch, seq_len, n_kv_heads, head_dim].

The Memory Formula

The KV cache memory for a single sequence is determined by a straightforward formula:

# KV Cache Memory Formula
kv_memory = 2 × num_layers × num_kv_heads × head_dim × seq_len × bytes_per_element
#           ↑       ↑            ↑            ↑         ↑          ↑
#        K and V   depth of    heads per     dim per   context   FP16=2,
#        tensors   network     KV group      head     length    FP8=1

# For a batch of B sequences:
total_kv_memory = kv_memory × batch_size

# ──────────────────────────────────────────────────────────────────────
# Concrete examples (FP16, batch=1):
# ──────────────────────────────────────────────────────────────────────

# LLaMA-2 7B: L=32, n_kv_heads=32, head_dim=128, seq=4096
kv_7b = 2 × 32 × 32 × 128 × 4096 × 2   # = 2.15 GB

# LLaMA-2 13B: L=40, n_kv_heads=40, head_dim=128, seq=4096
kv_13b = 2 × 40 × 40 × 128 × 4096 × 2  # = 3.36 GB

# LLaMA-2 70B: L=80, n_kv_heads=8 (GQA!), head_dim=128, seq=4096
kv_70b = 2 × 80 × 8 × 128 × 4096 × 2   # = 1.34 GB
# Note: 70B uses GQA with only 8 KV heads (vs 64 query heads)!
# Without GQA it would be: 2 × 80 × 64 × 128 × 4096 × 2 = 10.7 GB

# GPT-4 class (estimated): L=120, n_kv_heads=16, head_dim=128, seq=128K
kv_gpt4 = 2 × 120 × 16 × 128 × 131072 × 2  # = 62.9 GB per sequence!
The hidden cost: A 70B model in FP16 needs 140 GB of GPU memory for weights alone. But at batch size 64 with 4096 context, the KV cache adds another 85.9 GB (1.34 GB × 64). At 128K context length, a single sequence's KV cache for a 120-layer model can exceed 60 GB—nearly an entire A100's memory. This is why KV cache management is the central challenge of LLM serving.

Scaling with Sequence Length

The KV cache grows linearly with sequence length, but this linear growth has quadratic consequences for the total memory budget. When combined with batching, KV cache becomes the dominant memory consumer—often exceeding model weights.

def kv_cache_memory_gb(n_layers, n_kv_heads, head_dim, seq_len, batch, dtype_bytes=2):
    """Calculate KV cache memory in GB."""
    bytes_total = 2 * n_layers * n_kv_heads * head_dim * seq_len * batch * dtype_bytes
    return bytes_total / (1024**3)

# LLaMA-2 70B (GQA: 8 KV heads) at various sequence lengths
for seq in [512, 2048, 4096, 8192, 32768, 131072]:
    mem = kv_cache_memory_gb(80, 8, 128, seq, batch=1)
    print(f"seq={seq:>6d}: {mem:.2f} GB/seq")

# Output:
# seq=   512: 0.16 GB/seq
# seq=  2048: 0.66 GB/seq
# seq=  4096: 1.31 GB/seq
# seq=  8192: 2.62 GB/seq
# seq= 32768: 10.49 GB/seq
# seq=131072: 41.94 GB/seq  ← nearly fills an entire A100!
Memory budget crunch: An 8×A100 node has 640 GB total GPU memory. LLaMA-70B weights (FP16) consume 140 GB. That leaves 500 GB for KV cache. At 4096 context length, you can serve ~381 concurrent sequences (500 GB / 1.31 GB). At 32K context, only ~47. At 128K, only ~11. This is why long-context serving requires aggressive KV cache optimization.

MQA and GQA: Shrinking the Cache

The most architecturally impactful KV cache optimization is reducing the number of KV heads. Standard Multi-Head Attention (MHA) uses n_heads separate KV heads. Multi-Query Attention (MQA) and Grouped-Query Attention (GQA) share KV heads across multiple query heads.

Multi-Head Attention (MHA)

  • KV heads: Same as query heads (e.g., 64)
  • Cache size: Baseline (1×)
  • Quality: Best attention quality
  • Models: GPT-3, LLaMA-1, OPT
  • 70B cache (seq=4096): 10.7 GB/seq

Grouped-Query Attention (GQA)

  • KV heads: Fewer (e.g., 8 for 70B)
  • Cache size: Reduced by n_heads/n_kv_heads (8×)
  • Quality: Near-MHA quality with proper training
  • Models: LLaMA-2 70B, Mistral, Gemma
  • 70B cache (seq=4096): 1.34 GB/seq
# GQA: each KV head is shared across a group of query heads
# LLaMA-2 70B: 64 query heads, 8 KV heads → group_size = 8

def gqa_attention(q, k, v, n_q_heads=64, n_kv_heads=8):
    group_size = n_q_heads // n_kv_heads   # = 8
    
    # q: [batch, seq, 64, head_dim]
    # k: [batch, seq,  8, head_dim]  ← 8× smaller!
    # v: [batch, seq,  8, head_dim]  ← 8× smaller!
    
    # Repeat K,V to match query heads (or use grouped GEMM)
    k_expanded = k.repeat_interleave(group_size, dim=2)  # [b,s,64,d]
    v_expanded = v.repeat_interleave(group_size, dim=2)
    
    # Standard attention from here
    attn = (q @ k_expanded.transpose(-1, -2)) / sqrt(head_dim)
    return softmax(attn) @ v_expanded

# Memory savings: 64→8 KV heads = 8× reduction in KV cache
# 10.7 GB → 1.34 GB per sequence at 4096 context

Cache Compression Techniques

Beyond architectural choices like GQA, several post-training techniques can further reduce KV cache memory:

# 1. KV Cache Quantization (INT8 or INT4)
# Reduces bytes_per_element from 2 (FP16) to 1 (INT8) or 0.5 (INT4)
# Typically per-channel or per-token quantization

kv_fp16 = 1.34   # GB for LLaMA-70B, seq=4096
kv_int8 = 0.67   # 2× reduction, minimal quality loss
kv_int4 = 0.34   # 4× reduction, some quality degradation

# 2. Sliding Window Attention (Mistral, StreamingLLM)
# Only cache last W tokens (e.g., W=4096)
# Attention sink: keep first few tokens + sliding window

def sliding_window_cache(kv_cache, window_size=4096, n_sink=4):
    if kv_cache.seq_len() > window_size + n_sink:
        # Keep sink tokens (positions 0..n_sink-1)
        sink = kv_cache[:, :n_sink, :, :]
        # Keep recent window
        recent = kv_cache[:, -window_size:, :, :]
        kv_cache = torch.cat([sink, recent], dim=1)
    return kv_cache

# 3. Token Eviction (H2O, ScissorHands)
# Evict tokens with lowest cumulative attention scores
# Keep "heavy hitter" tokens that receive high attention

def h2o_eviction(kv_cache, attn_scores, budget=2048):
    # Accumulate attention scores across all heads and layers
    cumulative = attn_scores.sum(dim=(0, 1))   # sum over layers, heads
    # Keep top-k tokens by cumulative attention
    _, keep_idx = cumulative.topk(budget)
    return kv_cache[:, sorted(keep_idx), :, :]
Combining techniques: In practice, production systems stack multiple optimizations. A system might use GQA (8× reduction) + INT8 quantization (2× reduction) + sliding window (bounded growth) to achieve 16× or more memory savings over naive MHA with FP16 cache. vLLM supports FP8 KV cache, and TensorRT-LLM supports INT8 KV cache quantization out of the box.

Memory Pressure in Production

In production serving, GPU memory is partitioned between model weights, KV cache, and activations. The KV cache is the only component that grows dynamically during inference, making it the primary source of memory pressure and out-of-memory errors.

# Memory budget for an 8×A100-80GB node serving LLaMA-70B
total_gpu_memory  = 8 × 80             # = 640 GB
model_weights     = 140                 # 70B params × 2 bytes (FP16)
activation_buffer = 10                  # overhead for intermediate activations
framework_overhead = 10                 # CUDA contexts, PyTorch allocator
kv_budget = 640 - 140 - 10 - 10        # = 480 GB for KV cache

# Max concurrent sequences at different context lengths:
# seq=2048:  480 / 0.66  = 727 sequences
# seq=4096:  480 / 1.31  = 366 sequences
# seq=8192:  480 / 2.62  = 183 sequences
# seq=32768: 480 / 10.49 =  45 sequences

# The serving system must:
# 1. Pre-allocate or dynamically manage KV cache memory
# 2. Reject/queue requests when cache budget is exhausted
# 3. Free cache immediately when sequences complete
# 4. Handle variable-length sequences efficiently (→ PagedAttention)
The fragmentation problem: Naive KV cache allocation pre-allocates memory for the maximum sequence length. If a request only uses 500 tokens of a 4096-token allocation, 87% of that memory is wasted. Across hundreds of sequences, this internal fragmentation can waste 60–80% of available KV cache memory. This is the exact problem that PagedAttention (vLLM) solves by allocating cache in small, fixed-size pages.

The KV cache is the single most important data structure in LLM serving. Every major optimization—from GQA to quantization to PagedAttention to speculative decoding—is fundamentally about managing KV cache memory more efficiently. Understanding its structure, size, and growth dynamics is essential for anyone working on LLM inference systems.
