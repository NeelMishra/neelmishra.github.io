LLM Inference: Prefill, Decode, and KV Cache

MLOps Series LLM Inference & Serving

Autoregressive Generation

Large Language Models generate text one token at a time. Each new token is conditioned on all preceding tokens—the prompt plus every token generated so far. For a model like LLaMA-2 70B with a 4096-token context window, this means that generating a 512-token response to a 512-token prompt requires 512 sequential forward passes through a 70-billion-parameter network.

This autoregressive loop creates a fundamental tension: the prompt can be processed in parallel (all tokens are known), but generation is inherently sequential. This asymmetry splits inference into two distinct phases with radically different computational profiles.

The Prefill Phase

During prefill, the model processes the entire input prompt in a single forward pass. All n prompt tokens are fed through the transformer stack simultaneously, allowing full parallelism across the sequence dimension. This phase is compute-bound—the GPU's FLOPS are the bottleneck, not memory bandwidth.

For a transformer with L layers, hidden dimension d, and prompt length n, the prefill FLOPs scale as:

# Approximate FLOPs for prefill (attention + FFN per layer)
# Each layer: 4 projections (Q,K,V,O) + attention matmul + 2 FFN layers
prefill_flops = L × (
    4 × n × d²           # Q, K, V, O projections
  + 2 × n² × d           # attention: QK^T and attn×V
  + 2 × n × d × 4d       # FFN (up + down projections, 4× expansion)
)

# For LLaMA-2 70B: L=80, d=8192, prompt n=2048
# ≈ 80 × (4×2048×8192² + 2×2048²×8192 + 2×2048×8192×32768)
# ≈ 80 × (5.5e11 + 6.9e10 + 1.1e12) ≈ 1.3e14 FLOPs
Key insight: Prefill processes all prompt tokens in parallel. On an A100 (312 TFLOPS FP16), a 2048-token prompt through LLaMA-2 70B takes roughly 130 TFLOPS / 312 TFLOPS ≈ 0.4 seconds—limited by compute, not memory. The quadratic attention term (2n²d per layer) starts dominating only at very long sequences (>8K tokens).

The critical output of prefill is not just the first token's logits—it's the KV cache. Each layer computes key and value tensors for every prompt token, and these are stored in GPU memory for reuse during decode.

The Decode Phase

Once the first token is generated, inference enters the decode phase. Each step processes exactly one new token—the token just generated. The model computes Q, K, V only for this single token, then attends over the entire KV cache (all previous keys and values).

This creates a fundamentally different computational profile. Each decode step performs a tiny matrix-vector multiplication (not matrix-matrix), making it memory-bandwidth-bound:

# Decode step FLOPs (single token, n_past = past context length)
decode_flops_per_step = L × (
    4 × 1 × d²           # Q, K, V, O projections (same weights, 1 token)
  + 2 × 1 × n_past × d   # attention over cached keys/values
  + 2 × 1 × d × 4d       # FFN
)

# For LLaMA-2 70B decode step with n_past=2048:
# ≈ 80 × (2.7e8 + 3.4e7 + 5.4e8) ≈ 6.7e10 FLOPs per token
# That's ~2000× fewer FLOPs than prefill!

# But: must still load all 70B parameters from HBM → compute units
# 140 GB (FP16) / 2 TB/s (A100 HBM) ≈ 70 ms per token minimum
The decode bottleneck: Each decode step only does ~67 GFLOPs of compute but must read 140 GB of model weights from HBM. On an A100, this means you're using roughly 67 GFLOPS / 312 TFLOPS = 0.02% of peak compute. The GPU is almost entirely idle, waiting for data.

⚡ Prefill Phase

  • Input: All prompt tokens (n tokens)
  • Operations: Matrix × Matrix (GEMM)
  • Bottleneck: Compute-bound (GPU FLOPS)
  • Parallelism: High (all tokens in parallel)
  • Arithmetic Intensity: ~100–1000 FLOPs/byte
  • GPU Utilization: 60–80% of peak FLOPS

🐢 Decode Phase

  • Input: Single new token
  • Operations: Matrix × Vector (GEMV)
  • Bottleneck: Memory-bandwidth-bound
  • Parallelism: None across time steps
  • Arithmetic Intensity: ~1–2 FLOPs/byte
  • GPU Utilization: <5% of peak FLOPS

KV Cache: Avoiding Redundant Compute

Without caching, generating token t requires recomputing K and V for all tokens 1..t−1 at every layer. This means O(n²) total compute across all decode steps. The KV cache stores these intermediate activations so each decode step only computes K and V for the new token, reducing total decode compute from O(n²) per step to O(n).

def generate_with_kv_cache(model, prompt_ids, max_new_tokens):
    # Phase 1: Prefill — process full prompt, initialize KV cache
    logits, kv_cache = model.forward(prompt_ids, kv_cache=None)
    next_token = sample(logits[:, -1, :])   # sample from last position
    output = [next_token]

    # Phase 2: Decode — one token at a time, reuse KV cache
    for _ in range(max_new_tokens - 1):
        # Only feed the NEW token; KV cache has all history
        logits, kv_cache = model.forward(
            next_token.unsqueeze(0),          # shape: [1, 1]
            kv_cache=kv_cache                 # reuse cached K, V
        )
        next_token = sample(logits[:, -1, :])
        output.append(next_token)
        if next_token == eos_token:
            break
    return output

Arithmetic Intensity Analysis

The roofline model explains why prefill and decode behave so differently. Arithmetic intensity = FLOPs / bytes transferred. When intensity exceeds the machine's compute-to-bandwidth ratio, you're compute-bound; below it, you're memory-bound.

# A100 80GB SXM specs
peak_flops   = 312e12    # 312 TFLOPS FP16
hbm_bandwidth = 2.0e12   # 2 TB/s
ridge_point  = 312 / 2.0 # = 156 FLOPs/byte

# Prefill: matrix-matrix multiply, intensity ≈ d/2 ≈ 4096 FLOPs/byte
# → 4096 >> 156, so COMPUTE-BOUND ✓

# Decode: matrix-vector multiply, intensity ≈ 2 FLOPs/byte
# → 2 << 156, so MEMORY-BOUND ✓

# This is why decode generates ~14 tokens/sec on A100 for 70B models
# while prefill can process ~5000 tokens/sec
The roofline gap: During decode, a 70B model in FP16 needs 140 GB of weight reads per token. At 2 TB/s bandwidth, that's a minimum 70ms per token (~14 tok/s). No amount of extra FLOPS helps—you need more bandwidth (e.g., H100's 3.35 TB/s gives ~24 tok/s) or fewer bytes (quantization).

Practical Implications

Understanding the prefill/decode split drives every major optimization in LLM serving:

Optimizing Prefill

  • Flash Attention: Fuses attention into one kernel, reducing HBM reads from O(n²) to O(n)
  • Tensor Parallelism: Splits weight matrices across GPUs to increase aggregate FLOPS
  • Chunked Prefill: Breaks long prompts into chunks to overlap with decode requests
  • Prefix Caching: Reuses KV cache from common system prompts across requests

Optimizing Decode

  • Quantization: INT8/INT4 weights reduce bytes read per token by 2–4×
  • Batching: Amortize weight reads across B sequences (B tokens computed per weight read)
  • Speculative Decoding: Draft model proposes N tokens, verified in parallel by large model
  • KV Cache Compression: Quantize or evict cache entries to reduce memory pressure

The single most impactful optimization for decode throughput is batching. If you batch 32 sequences together, each weight read from HBM is amortized over 32 tokens, increasing arithmetic intensity by 32×. This is why serving systems like vLLM, TensorRT-LLM, and TGI focus heavily on maximizing batch size through techniques like continuous batching and PagedAttention.

# Impact of batching on decode throughput
# Without batching (batch=1):
weight_read = 140e9         # 140 GB for 70B model in FP16
tokens_per_read = 1
intensity = 2               # FLOPs/byte → memory-bound
throughput = 14              # tokens/sec

# With batching (batch=32):
weight_read = 140e9         # same weights read once
tokens_per_read = 32        # but 32 tokens computed
intensity = 64              # 32× improvement
throughput = 448             # tokens/sec total (14 tok/s per seq)

# With batch=128: intensity=256 → approaching compute-bound!
# But: KV cache memory = 128 × 5.2 GB = 665 GB → needs multiple GPUs
The memory wall: Batching improves throughput but KV cache memory grows linearly with batch size. For LLaMA-70B at 4096 context length, each sequence's KV cache is ~5.2 GB in FP16. A batch of 128 sequences needs 665 GB just for the cache—more than 8× A100 80GB GPUs. This is why PagedAttention and KV cache quantization are critical.
