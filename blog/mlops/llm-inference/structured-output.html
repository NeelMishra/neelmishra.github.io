Structured Output: JSON Mode, Constrained Decoding

MLOps Series LLM Inference & Serving

Getting reliable structured output (JSON, XML, SQL) from LLMs is one of the most critical challenges in production systems. Prompt engineering alone fails 5-15% of the time even with strong models. Constrained decoding guarantees 100% valid output by modifying the sampling process at each token step, using finite state machines to enforce grammar rules. This post covers the theory, implementation, and performance trade-offs of this approach.

Why Structured Output?

In production pipelines, LLM outputs feed into downstream code — JSON parsers, API handlers, database writers. A malformed response doesn't just degrade quality; it crashes the pipeline. The failure modes are subtle:

Prompt-Based (Unreliable)

  • Add "respond in JSON" to the prompt
  • Include few-shot examples of valid JSON
  • Retry on parse failure (adds latency)
  • Failure rate: 2-15% depending on model and schema complexity
  • Trailing text after JSON closing brace
  • Missing required fields, wrong types

Constrained Decoding (Guaranteed)

  • Modify logits at each decoding step
  • Only allow tokens valid per grammar state
  • Output is valid by construction
  • Failure rate: 0% for grammar compliance
  • Schema-level enforcement (types, required fields)
  • Works with any model, any schema

FSM-Based Constrained Decoding

The core idea: compile a JSONSchema (or any formal grammar) into a finite state machine (FSM), where each state represents a position in the grammar. At each decoding step, the FSM determines which tokens are valid transitions from the current state. Invalid tokens get their logits set to -∞.

The FSM approach works in three phases:

  1. Compile time: Convert JSONSchema → regular expressions → deterministic finite automaton (DFA). This is done once per schema and cached.
  2. Index time: For each DFA state, precompute which tokens in the vocabulary are valid transitions. This creates a mapping: state → Set[token_ids].
  3. Decode time: At each step, look up the current DFA state, apply the token mask to logits, then sample normally.

Token Masking Mechanics

The critical challenge is mapping grammar states to vocabulary tokens. A single vocabulary token might correspond to multiple characters (e.g., the token " name" contains a space and letters). The FSM must track all possible DFA state transitions for every token.

## Token masking: precompute valid tokens per FSM state
def build_token_mask_index(dfa, tokenizer):
    """For each DFA state, find all vocabulary tokens that produce
    a valid transition (possibly through multiple DFA states)."""
    vocab = tokenizer.get_vocab()
    mask_index = {}  # state → set of valid token_ids

    for state in dfa.states:
        valid_tokens = set()
        for token_id, token_str in vocab.items():
            # Walk the DFA through each character in the token
            current = state
            valid = True
            for char in token_str:
                if char in dfa.transitions[current]:
                    current = dfa.transitions[current][char]
                else:
                    valid = False
                    break
            if valid:
                valid_tokens.add(token_id)
        mask_index[state] = valid_tokens

    return mask_index  # O(|states| × |vocab|) — precomputed once

## At decode time — nearly zero overhead
def apply_mask(logits, current_state, mask_index):
    valid_tokens = mask_index[current_state]
    mask = full_like(logits, float('-inf'))
    mask[list(valid_tokens)] = 0.0
    return logits + mask
Precomputation cost: For a typical JSONSchema with ~50 DFA states and a 32K vocabulary, the index contains ~1.6M entries. Building it takes 1-5 seconds. But once built, the per-token overhead is just a single lookup + mask application — adding <0.1ms per token to decoding.

Outlines Library

Outlines is the reference implementation for grammar-constrained generation. It compiles regular expressions and JSONSchemas into an interleaved automaton that works with any tokenizer.

import outlines

## 1. JSON output from a schema
from pydantic import BaseModel

class UserProfile(BaseModel):
    name: str
    age: int
    email: str
    interests: list[str]
    is_active: bool

model = outlines.models.transformers("mistralai/Mistral-7B-v0.1")
generator = outlines.generate.json(model, UserProfile)

result = generator("Create a user profile for a software engineer named Alex.")
# result is a guaranteed-valid UserProfile instance
# {"name": "Alex", "age": 32, "email": "alex@dev.io",
#  "interests": ["rust", "distributed-systems"], "is_active": true}

## 2. Regex-constrained output
phone_pattern = r"\(\d{3}\) \d{3}-\d{4}"
phone_gen = outlines.generate.regex(model, phone_pattern)
phone = phone_gen("What is the phone number for the office?")
# Guaranteed format: (555) 123-4567

## 3. Choice (enum) output
sentiment = outlines.generate.choice(model, ["positive", "negative", "neutral"])
label = sentiment("The movie was absolutely fantastic!")
# "positive" — guaranteed to be one of the three options

Under the hood, Outlines performs these steps:

  1. Schema → Regex: Pydantic model is converted to a regex pattern that matches all valid JSON conforming to the schema.
  2. Regex → DFA: The regex is compiled into a minimal DFA using the interegular library.
  3. DFA → Token Mask Index: For each DFA state, valid vocabulary tokens are precomputed.
  4. Decode loop: Standard autoregressive decoding with mask applied at each step.

vLLM Guided Decoding

vLLM integrates constrained decoding natively through its guided_decoding API, using either Outlines or lm-format-enforcer as backends.

from openai import OpenAI

client = OpenAI(base_url="http://localhost:8000/v1", api_key="na")

## Method 1: response_format with json_schema (OpenAI-compatible)
response = client.chat.completions.create(
    model="mistralai/Mistral-7B-Instruct-v0.3",
    messages=[{"role": "user", "content": "Extract entities from: 'Apple CEO Tim Cook announced...'"}],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "entities",
            "schema": {
                "type": "object",
                "properties": {
                    "entities": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "name": {"type": "string"},
                                "type": {"type": "string",
                                        "enum": ["person", "org", "location"]}
                            },
                            "required": ["name", "type"]
                        }
                    }
                },
                "required": ["entities"]
            }
        }
    }
)

## Method 2: extra_body for regex-guided (vLLM-specific)
response = client.chat.completions.create(
    model="mistralai/Mistral-7B-Instruct-v0.3",
    messages=[{"role": "user", "content": "Generate an ISO date."}],
    extra_body={"guided_regex": r"\d{4}-\d{2}-\d{2}"}
)
vLLM backend choice: Set --guided-decoding-backend outlines (default) or lm-format-enforcer. Outlines precomputes the full mask index upfront (higher first-request latency, faster subsequent tokens). lm-format-enforcer computes masks lazily (lower startup, slightly higher per-token cost). For schemas used repeatedly, Outlines is faster; for one-off schemas, lm-format-enforcer may be preferable.

Performance Impact

Constrained decoding is not free. The overhead comes from three sources:

Compilation Overhead

  • Schema → DFA: 10-500ms depending on complexity
  • DFA → Mask Index: 1-5s for 32K vocab, ~50 states
  • Cached after first use — amortized to zero
  • Complex schemas (deeply nested, many enums) → larger DFAs

Per-Token Overhead

  • Mask lookup + apply: 0.05-0.2ms per token
  • Relative to model forward pass: <1% overhead
  • Reduced sampling entropy: often generates fewer tokens (good!)
  • Net effect: often faster than unconstrained + retry
## Benchmark: constrained vs unconstrained JSON generation
## Model: Mistral-7B on A100, schema: 8-field Pydantic model

## Unconstrained + retry:
##   Avg tokens generated: 145 (verbose, extra whitespace)
##   Parse failure rate: 8.2%
##   Avg latency (including retries): 2.3s
##   Effective throughput: 42 req/s

## Constrained (Outlines):
##   Avg tokens generated: 89 (compact, valid JSON)
##   Parse failure rate: 0.0%
##   Avg latency: 1.1s
##   Effective throughput: 68 req/s
##   First-request overhead: +1.8s (DFA compilation, cached after)

## Net result: constrained is ~60% faster in steady state
Pitfall — quality degradation: Constraining output too aggressively can hurt quality. If the schema is very restrictive (e.g., enum with only 3 options for a nuanced field), the model might be forced into a locally valid but globally nonsensical path. Always validate semantic quality, not just structural validity.

Practical Patterns

Battle-tested patterns for production structured output:

## Pattern 1: Schema versioning with fallback
from pydantic import BaseModel, Field
from typing import Optional, Literal

class ExtractionV2(BaseModel):
    """Versioned schema — allows graceful migration."""
    schema_version: Literal["2.0"] = "2.0"
    confidence: float = Field(ge=0.0, le=1.0)
    entities: list[dict]
    reasoning: Optional[str] = None  # free-form thinking

## Pattern 2: Two-pass generation
## First pass: unconstrained chain-of-thought
## Second pass: constrained extraction from the reasoning
def two_pass_extract(prompt, schema):
    # Pass 1: Let the model think freely
    reasoning = llm.generate(
        f"Think step by step about: {prompt}",
        max_tokens=500
    )
    # Pass 2: Extract structured data with constraints
    result = constrained_generate(
        f"Based on this analysis:\n{reasoning}\n\nExtract:",
        schema=schema
    )
    return result

## Pattern 3: Streaming with partial validation
## Parse JSON incrementally as tokens arrive
import ijson  # incremental JSON parser

async def stream_json(response_stream):
    buffer = ""
    async for chunk in response_stream:
        buffer += chunk.text
        # ijson can parse partial JSON and yield complete sub-objects
        for prefix, event, value in ijson.parse(buffer):
            if event == 'map_key':
                handle_field(prefix, value)
Production recommendation: Use constrained decoding for all structured output in production. The ~0% failure rate eliminates retry logic, simplifies error handling, reduces latency, and often produces more compact output. Combine with schema versioning and the two-pass pattern for complex extraction tasks where reasoning quality matters.
