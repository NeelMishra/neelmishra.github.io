The Fragmentation Problem

Traditional LLM serving systems pre-allocate KV cache as contiguous memory blocks for each sequence. Since the output length is unknown at request time, systems must allocate for the maximum possible sequence length. This causes two devastating forms of memory waste:

# The contiguous allocation problem
# System allocates for max_seq_len=2048 per sequence

# Scenario: 3 sequences on a GPU with 10 GB KV cache budget
# Per-sequence allocation: 2048 positions × 327 KB/position = 654 MB

seq_A: allocated=654 MB, actually_used=120 MB (184 tokens)   # 82% wasted
seq_B: allocated=654 MB, actually_used=654 MB (2048 tokens)  # 0% wasted
seq_C: allocated=654 MB, actually_used=45 MB  (69 tokens)    # 93% wasted

total_allocated = 1962 MB
total_used      = 819 MB
# Internal fragmentation: 58% of allocated KV cache is wasted!

# External fragmentation: after seq_A finishes, its 654 MB slot
# may not be reusable if a new request needs a different-sized block
# Free memory gets split into unusable fragments over time
Real-world waste: The vLLM paper (Kwon et al., SOSP 2023) measured that existing systems waste 60–80% of KV cache memory due to fragmentation. For a server with 8×A100-80GB GPUs serving LLaMA-13B, this means ~300 GB of GPU memory is wasted on empty pre-allocated KV cache slots. PagedAttention recovers nearly all of this waste.

There are three sources of memory waste in traditional KV cache management:

  • Internal fragmentation: Pre-allocated space beyond what a sequence actually uses (most sequences don't reach max length)
  • External fragmentation: Free memory broken into non-contiguous chunks too small to be useful
  • Reservation waste: Memory reserved for future tokens that may never be generated (early EOS)

Virtual Memory Analogy

PagedAttention borrows the core idea from operating system virtual memory. In an OS, each process sees a contiguous virtual address space, but the physical memory is divided into fixed-size pages that can be scattered anywhere in RAM. A page table maps virtual pages to physical frames.

Paged KV Cache

In PagedAttention, KV cache memory is divided into fixed-size blocks (pages). Each block holds KV data for a fixed number of tokens (typically 16). Blocks are allocated from a free pool on demand—one at a time—as the sequence generates more tokens.

# PagedAttention data structures
BLOCK_SIZE = 16   # tokens per block (typical: 16 or 32)

class KVBlock:
    """One physical block of KV cache in GPU memory."""
    # Stores K and V for BLOCK_SIZE tokens, for ALL layers and heads
    # Shape per tensor: [num_layers, num_kv_heads, BLOCK_SIZE, head_dim]
    # Memory: 2 × 80 × 8 × 16 × 128 × 2 bytes = 5.24 MB per block (70B)
    k: Tensor  # [layers, kv_heads, block_size, head_dim]
    v: Tensor
    ref_count: int = 0   # for copy-on-write sharing

class BlockAllocator:
    """Manages a pool of physical KV blocks on GPU."""
    def __init__(self, num_blocks):
        self.free_blocks = deque(range(num_blocks))
        self.blocks = [KVBlock() for _ in range(num_blocks)]
    
    def allocate(self) -> int:
        if not self.free_blocks:
            raise OutOfMemoryError("No free KV blocks")
        return self.free_blocks.popleft()
    
    def free(self, block_id):
        self.blocks[block_id].ref_count -= 1
        if self.blocks[block_id].ref_count == 0:
            self.free_blocks.append(block_id)

class SequenceState:
    """Per-sequence metadata: logical → physical block mapping."""
    block_table: List[int]  # block_table[i] = physical block ID for logical block i
    num_tokens: int          # current sequence length

    def append_token(self, allocator):
        # Check if current last block is full
        if self.num_tokens % BLOCK_SIZE == 0:
            # Need a new block!
            new_block = allocator.allocate()
            self.block_table.append(new_block)
        self.num_tokens += 1
On-demand allocation: A sequence that generates 50 tokens only uses ceil(50/16) = 4 blocks. At 5.24 MB per block (LLaMA-70B), that's 21 MB—versus 654 MB for a pre-allocated 2048-token contiguous cache. The last block may have some internal waste (up to BLOCK_SIZE−1 = 15 tokens), but this is negligible: worst case is <5% waste with block size 16, compared to 60–80% with contiguous allocation.

Block Tables and Attention Kernel

The block table is the core data structure enabling PagedAttention. It's a simple array per sequence: block_table[i] maps logical block i to a physical block ID. The custom attention kernel uses this mapping to gather KV data from scattered physical blocks during the attention computation.

# PagedAttention kernel (simplified)
def paged_attention(query, key_cache, value_cache, block_tables, context_lens):
    """
    query:        [batch, 1, num_heads, head_dim]     (current token)
    key_cache:    [num_blocks, kv_heads, block_size, head_dim]  (all physical blocks)
    value_cache:  [num_blocks, kv_heads, block_size, head_dim]
    block_tables: [batch, max_blocks_per_seq]          (logical → physical mapping)
    context_lens: [batch]                              (current sequence lengths)
    """
    output = torch.zeros_like(query)
    
    for b in range(batch_size):                  # parallel in GPU
        seq_len = context_lens[b]
        num_blocks = ceil(seq_len / BLOCK_SIZE)
        
        for block_idx in range(num_blocks):        # parallel in GPU
            # Look up physical block from block table
            phys_block = block_tables[b][block_idx]
            
            # Fetch K, V from the physical block
            k_block = key_cache[phys_block]        # [kv_heads, block_size, head_dim]
            v_block = value_cache[phys_block]
            
            # Compute partial attention scores for this block
            scores = query[b] @ k_block.transpose(-1, -2)  # [heads, 1, block_size]
            
            # Accumulate using online softmax (log-sum-exp trick)
            # Avoids materializing the full attention matrix
            output[b] = online_softmax_update(output[b], scores, v_block)
    
    return output

# The kernel iterates over blocks, not individual tokens
# Each block is fetched via one pointer lookup → minimal overhead
# Block-level computation enables efficient GPU parallelism
Overhead is minimal: The indirection through block tables adds only ~5% overhead compared to contiguous attention. Each block table lookup is a single integer index. The blocks are stored contiguously within each block (16 consecutive tokens), so GPU memory access is still coalesced within a block. The gain in memory efficiency (2–4× more sequences served) vastly outweighs this small overhead.

Memory Efficiency Gains

🔴 Contiguous Allocation

  • Pre-allocate: max_seq_len per sequence
  • Internal fragmentation: 60–80% (typical)
  • External fragmentation: Grows over time
  • Memory utilization: 20–38%
  • Max batch size (70B, 480GB KV budget, seq=4096): ~146 sequences (allocated) but only ~55 effectively used

🟢 PagedAttention

  • Allocate: One block at a time, on demand
  • Internal fragmentation: <4% (last block only)
  • External fragmentation: Zero (any free block fits)
  • Memory utilization: >96%
  • Max batch size (70B, 480GB KV budget, seq=4096): ~366 sequences (nearly 100% of theoretical max)
# Memory efficiency comparison (LLaMA-70B GQA, 480 GB KV budget)

# Contiguous allocation (pessimistic: allocate for max_seq_len=4096)
per_seq_allocated = 2 × 80 × 8 × 128 × 4096 × 2       # = 1.34 GB
max_seqs_contiguous = 480 / 1.34                         # = 358 sequences
# But average utilization only 35% → effective sequences ≈ 125

# PagedAttention (allocate only for actual tokens used, avg 1500 tok)
per_seq_actual = 2 × 80 × 8 × 128 × 1500 × 2           # = 0.49 GB
max_seqs_paged = 480 / 0.49                              # = 979 sequences!
# Plus ~4% waste in last blocks → ~940 effective sequences

# Result: 2.6× more concurrent sequences with PagedAttention
# At same average seq length, 7.5× more effective sequences

# vLLM benchmark result (LLaMA-13B, A100-40GB):
# - Contiguous: max 8 concurrent sequences
# - PagedAttention: max 20+ concurrent sequences (2.5×)
# - Combined with continuous batching: 14-24× throughput improvement

Advanced: Sharing and Swapping

PagedAttention's block-based architecture enables two powerful advanced features that are impossible with contiguous allocation:

1. Copy-on-Write Block Sharing (Parallel Sampling / Beam Search):

# Parallel sampling: generate N completions for one prompt
# With contiguous allocation: N copies of the prompt's KV cache
# With PagedAttention: N sequences SHARE the same physical blocks

def fork_sequence(parent_seq, allocator):
    """Create a new sequence sharing the parent's KV cache blocks."""
    child = SequenceState()
    # Share all physical blocks (just copy the block table)
    child.block_table = parent_seq.block_table.copy()
    # Increment ref counts
    for phys_id in child.block_table:
        allocator.blocks[phys_id].ref_count += 1
    return child

# When child diverges (generates different tokens):
def copy_on_write(seq, block_idx, allocator):
    """Copy block only when modified, if shared."""
    phys_id = seq.block_table[block_idx]
    if allocator.blocks[phys_id].ref_count > 1:
        new_block = allocator.allocate()
        gpu_memcpy(new_block, phys_id)      # copy data
        allocator.blocks[phys_id].ref_count -= 1
        seq.block_table[block_idx] = new_block  # point to copy

# Memory savings for parallel sampling (N=4, prompt=2048 tokens):
# Contiguous: 4 × 0.66 GB = 2.64 GB
# PagedAttention: 0.66 GB shared + 4 × (divergent blocks only)
# If avg divergence is 100 tokens: 0.66 + 4 × 0.03 = 0.78 GB (3.4× savings)

2. KV Cache Swapping (Preemption Support):

# When GPU memory is full but a high-priority request arrives:
# PagedAttention can swap blocks to CPU memory

class BlockSwapper:
    def swap_out(self, sequence, gpu_allocator, cpu_allocator):
        """Move sequence's KV blocks from GPU to CPU."""
        cpu_block_table = []
        for gpu_block_id in sequence.block_table:
            cpu_block = cpu_allocator.allocate()
            async_d2h_copy(cpu_block, gpu_block_id)  # GPU → CPU async
            gpu_allocator.free(gpu_block_id)
            cpu_block_table.append(cpu_block)
        sequence.block_table = cpu_block_table
        sequence.on_gpu = False
    
    def swap_in(self, sequence, gpu_allocator, cpu_allocator):
        """Move sequence's KV blocks from CPU back to GPU."""
        gpu_block_table = []
        for cpu_block_id in sequence.block_table:
            gpu_block = gpu_allocator.allocate()
            async_h2d_copy(gpu_block, cpu_block_id)  # CPU → GPU async
            cpu_allocator.free(cpu_block_id)
            gpu_block_table.append(gpu_block)
        sequence.block_table = gpu_block_table
        sequence.on_gpu = True

# Swap granularity is per-block → can partially swap sequences
# PCIe 4.0 x16: 32 GB/s → swapping 500 MB takes ~16ms
# Better than recomputing from scratch for long sequences!
vLLM in production: vLLM combines PagedAttention with continuous batching, prefix caching, tensor parallelism, and speculative decoding into a production-ready serving engine. As of 2024, it's the most widely deployed open-source LLM serving system, used by organizations ranging from startups to large tech companies. The key insight—treating KV cache memory like OS virtual memory—eliminated the largest source of waste in LLM serving and enabled 2–4× more sequences per GPU.

PagedAttention is now implemented in virtually every major serving framework: vLLM (original), TensorRT-LLM (NVIDIA), TGI (HuggingFace), and SGLang. It's the foundation that makes large-scale LLM serving economically viable.
