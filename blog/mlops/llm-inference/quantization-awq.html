AWQ: Activation-Aware Quantization

MLOps Series LLM Inference & Serving

The Core Observation

AWQ (Lin et al., 2023) starts from a surprisingly simple observation: not all weights are equally important. In a typical transformer linear layer, skipping the quantization of just 1% of weight channels (keeping them in FP16) can reduce quantization error by an order of magnitude — far more than the 1% parameter overhead would suggest.

The key question is: which 1%? AWQ discovers that the salient weight channels correspond to channels with large activation magnitudes. A weight column wj is "salient" not because |wj| is large, but because |xj| (the corresponding input activation) is consistently large across calibration data.

Why activations, not weights? The quantization error for output y is: Δy = (ΔW)·x. Even if ΔW is small for a given column, if the corresponding activation xj is large, the output error is amplified. Conversely, large ΔW on a channel with near-zero activations has negligible impact. AWQ protects the channels where errors are most amplified.

However, a mixed-precision approach (keeping 1% in FP16) creates hardware inefficiency — GPUs cannot efficiently process a layer with mixed INT4/FP16 columns. AWQ's breakthrough is achieving the same protection through a mathematical scaling trick that keeps all weights at the same bit-width.

Salient Weight Detection

Salient channels are identified by computing the average activation magnitude per input channel across the calibration dataset:

s_j ∝ (E[|x_j|])^α where α is a search parameter, typically α ∈ [0.5, 0.7]

The exponent α controls the aggressiveness of protection: α = 0 gives uniform scaling (no protection), α = 1 gives full proportional scaling. The optimal α is found via grid search on a small calibration set.

Per-Channel Scaling Transform

The mathematical foundation of AWQ is the scaling equivalence. For a linear layer y = Wx, we can introduce a diagonal scaling matrix S = diag(s₁, ..., s_d):

y = Wx = (W·S)·(S⁻¹·x) = W'·x'

This transformation is mathematically exact — the output y is unchanged. But the quantization properties of W' = W·S differ from W. By choosing s_j > 1 for salient channels, we scale up those weight columns, which:

  1. Increases the magnitude of salient weights, giving them more quantization bins (less relative error).
  2. Decreases the corresponding activations x'_j = x_j/s_j, but this is fine because activations are computed in full precision (FP16) before being multiplied by the quantized weights.
  3. Trades off quantization precision: non-salient channels get slightly less resolution (scaled down), but their low activation magnitudes mean the resulting output error is minimal.
Implementation detail: The scaling is "folded" into the weights before quantization and into the preceding layer's output (or LayerNorm parameters). At inference time, there is zero overhead — the INT4 weights already incorporate the scaling, and the preceding layer absorbs the inverse scaling. No runtime division is needed.

For the preceding LayerNorm with parameters γ (scale) and β (bias), the fusion is:

γ' = γ / s,    β' = β / s,    W' = W · diag(s)

Searching for Optimal Scales

AWQ performs a per-channel grid search for the optimal scaling exponent α per group of channels. The objective is to minimize the MSE between the FP16 output and the quantized output:

def awq_find_scales(weight, activations, group_size=128, n_grid=20):
    """Find optimal per-channel scaling factors for AWQ."""
    d_out, d_in = weight.shape
    scales = torch.ones(d_in, device=weight.device)
    
    # Compute average activation magnitude per channel
    act_mean = activations.abs().mean(dim=0)  # shape: [d_in]
    
    for g in range(0, d_in, group_size):
        g_end = min(g + group_size, d_in)
        w_group = weight[:, g:g_end]          # [d_out, group_size]
        a_group = act_mean[g:g_end]           # [group_size]
        x_group = activations[:, g:g_end]     # [n_cal, group_size]
        
        # FP16 reference output for this group
        y_ref = x_group @ w_group.T           # [n_cal, d_out]
        
        best_error = float("inf")
        best_scale = torch.ones(g_end - g)
        
        # Grid search over alpha ∈ [0, 1]
        for alpha in torch.linspace(0, 1, n_grid):
            s = a_group.pow(alpha)
            s = s / s.mean()                  # normalize to preserve magnitude
            s = s.clamp(min=1e-4)
            
            # Apply scaling: scale up weights, scale down activations
            w_scaled = w_group * s.unsqueeze(0)
            x_scaled = x_group / s.unsqueeze(0)
            
            # Quantize the scaled weights to INT4
            w_q = pseudo_quantize(w_scaled, bits=4)
            
            # Compute quantized output
            y_q = x_scaled @ w_q.T
            
            # MSE loss between FP16 and quantized output
            error = (y_ref - y_q).pow(2).mean()
            
            if error < best_error:
                best_error = error
                best_scale = s.clone()
        
        scales[g:g_end] = best_scale
    
    return scales

def pseudo_quantize(w, bits=4, group_size=128):
    """Simulate INT4 quantization (round-to-nearest)."""
    max_val = (2 ** (bits - 1)) - 1
    for g in range(0, w.shape[1], group_size):
        g_end = min(g + group_size, w.shape[1])
        w_g = w[:, g:g_end]
        scale = w_g.abs().max(dim=1, keepdim=True).values / max_val
        w[:, g:g_end] = (w_g / scale).round().clamp(-max_val, max_val) * scale
    return w
Speed advantage: AWQ quantization takes only ~10 minutes for a 70B model on a single GPU, compared to ~4 hours for GPTQ. This is because AWQ performs a simple grid search per channel group, while GPTQ requires sequential Hessian-based weight updates across all columns.

AWQ vs GPTQ

AWQ

  • Method: Activation-aware per-channel scaling → round-to-nearest
  • Quantization time: ~10 minutes (70B)
  • Calibration data: 128 samples
  • Quality (LLaMA-7B INT4): PPL 5.78 on WikiText-2
  • Quality (LLaMA-70B INT4): PPL 3.55 on WikiText-2
  • Kernel efficiency: Excellent (uniform INT4, no reordering)
  • Fine-tuning friendly: Yes (weights are simply scaled)

GPTQ

  • Method: Hessian-based optimal weight update → column-by-column
  • Quantization time: ~4 hours (70B)
  • Calibration data: 128 samples
  • Quality (LLaMA-7B INT4): PPL 5.85 on WikiText-2
  • Quality (LLaMA-70B INT4): PPL 3.60 on WikiText-2
  • Kernel efficiency: Good (act_order hurts fused kernels)
  • Fine-tuning friendly: No (weight updates are non-trivial to reverse)

AWQ generally achieves better or equal perplexity compared to GPTQ at the same bit-width, while being significantly faster to quantize. The key difference is in methodology: GPTQ performs complex weight surgery using second-order information, while AWQ leverages a simpler scaling insight combined with efficient search.

Implementation with AutoAWQ

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_name = "meta-llama/Llama-2-70b-hf"
quant_path = "./llama-70b-awq-int4"

# Load model (requires FP16 model in memory during quantization)
model = AutoAWQForCausalLM.from_pretrained(model_name, device_map="auto")
tokenizer = AutoTokenizer.from_pretrained(model_name)

# Configure AWQ quantization
quant_config = {
    "zero_point": True,         # asymmetric quantization
    "q_group_size": 128,        # group size for quantization
    "w_bit": 4,                  # 4-bit weight quantization
    "version": "GEMM",          # GEMM kernel (vs GEMV for batch=1)
}

# Quantize — ~10 minutes for 70B on single A100
model.quantize(tokenizer, quant_config=quant_config)

# Save: ~35GB INT4 model
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)

# Load for inference (fits on single 48GB GPU)
model = AutoAWQForCausalLM.from_quantized(
    quant_path,
    fuse_layers=True,          # fuse QKV + MLP for faster inference
    device_map="auto",
)
# Generate
tokens = tokenizer("The meaning of life is", return_tensors="pt").to("cuda")
output = model.generate(**tokens, max_new_tokens=128)
print(tokenizer.decode(output[0], skip_special_tokens=True))

Hardware Efficiency & Deployment

AWQ is designed with hardware efficiency as a first-class concern. The uniform INT4 format enables highly optimized CUDA kernels:

  • GEMM kernels: For batched inference (batch size > 1), AWQ uses INT4×FP16 GEMM kernels. Throughput improvement: 1.45× over FP16 GEMM on A100 at batch size 8.
  • GEMV kernels: For single-request inference (batch size = 1), AWQ provides specialized GEMV kernels that are 3.2–3.5× faster than FP16, directly proportional to the 4× memory reduction.
  • Fused operations: AutoAWQ fuses QKV projections and MLP gate/up projections into single kernel launches, reducing kernel launch overhead by ~40%.

Throughput Comparison (A100, LLaMA-70B)

  • FP16: 18 tokens/s (2× A100)
  • AWQ INT4: 42 tokens/s (1× A100)
  • GPTQ INT4: 38 tokens/s (1× A100)
  • AWQ INT4 + vLLM: 1200 tokens/s (1× A100, batch=32)

Memory Usage (LLaMA-70B)

  • FP16: 140 GB (2× A100 80GB)
  • AWQ INT4: 35 GB (1× A100 80GB)
  • AWQ INT4 + KV cache (4K ctx): ~52 GB
  • AWQ INT4 + KV cache (8K ctx): ~68 GB
TinyChat integration: The AWQ authors also released TinyChat, a lightweight inference engine that runs AWQ models on edge devices. LLaMA-2-7B INT4 achieves 30 tokens/s on a Jetson Orin (mobile GPU with 32GB unified memory) and 18 tokens/s on a Raspberry Pi 5 (with 8GB RAM, using CPU-only execution). This demonstrates AWQ's versatility across the hardware spectrum.

AWQ has been adopted as the default quantization method in several production systems including vLLM, TGI (Text Generation Inference), and NVIDIA's TensorRT-LLM. Its combination of superior quality, fast quantization, and hardware-friendly format makes it the recommended choice for most INT4 deployment scenarios.
