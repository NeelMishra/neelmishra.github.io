Autoscaling LLM Services

MLOps Series LLM Inference & Serving

Why LLM Autoscaling Is Different

Standard web application autoscaling (CPU utilization → add pods) fails catastrophically for LLM workloads. The fundamental issues:

  • Scale-up latency is minutes, not seconds. Loading a 70B model takes 30-120 seconds even from local NVMe. During this time, all queued requests are blocked or failing. By contrast, a typical web pod starts in 2-5 seconds.
  • GPU utilization is a poor signal. A GPU can show 90% utilization while batch size is 1 (inefficient) or 30% utilization while efficiently processing a batch of 64 at memory bandwidth limits. The relationship between GPU util and capacity is non-linear.
  • Requests have wildly variable cost. A 128-token prompt + 32-token output costs ~100× less compute than a 4096-token prompt + 2048-token output. Standard request-count-based scaling ignores this variance entirely.
  • KV cache memory is the real bottleneck. Even if compute is available, the GPU can run out of KV cache memory and reject new requests. Memory pressure must be a first-class scaling signal.

Custom Metrics for LLM Autoscaling

The right metrics determine scaling accuracy. Here are the four essential metrics for LLM serving, ranked by importance:

1. Request queue depth — the most responsive signal. Measures pending requests waiting for GPU capacity. When queue depth exceeds a threshold, scale up. Unlike throughput metrics, queue depth reacts instantly to traffic spikes.

2. KV cache utilization — measures what percentage of GPU memory allocated for KV cache is in use. When KV cache utilization exceeds 80%, the system cannot accept new long-context requests. This is often the binding constraint before compute saturates.

3. Generation throughput (tokens/second) — actual output token rate across all active sequences. Useful for capacity planning but lags behind demand changes. Best used with a target-based policy: if per-pod throughput drops below target, add replicas.

4. Time-to-First-Token (TTFT) P95 — direct SLO metric. If TTFT exceeds the SLO threshold, it's a definitive signal that capacity is insufficient. However, it's a lagging indicator — by the time P95 TTFT degrades, many users are already affected.

# Prometheus metrics exposed by vLLM / TGI serving endpoints
# These are scraped by Prometheus and made available to HPA via adapter

# Queue depth — instant signal
vllm_num_requests_waiting                   # gauge

# KV cache utilization — memory pressure signal
vllm_gpu_cache_usage_perc                    # gauge, 0.0-1.0

# Throughput — capacity signal
vllm_generation_tokens_total                  # counter (rate it)
rate(vllm_generation_tokens_total[1m])        # tokens/sec

# Latency — SLO signal
vllm_time_to_first_token_seconds             # histogram
histogram_quantile(0.95,
  rate(vllm_time_to_first_token_seconds_bucket[5m]))

# Batch size — efficiency signal
vllm_num_requests_running                     # gauge
Best practice: Use queue depth as the primary scaling trigger (fast response), KV cache utilization as a safety valve (prevents OOM), and TTFT P95 as an SLO-based backstop. Combine them with a max() policy — scale up if any metric exceeds its threshold.

GPU-Aware HPA Configuration

Here's a production-ready HPA configuration that combines multiple custom metrics for LLM serving:

# HPA with custom LLM metrics
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: llm-serving-hpa
  namespace: inference
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: vllm-llama70b
  minReplicas: 2          # Always keep 2 warm replicas
  maxReplicas: 12         # Bounded by GPU node pool size
  metrics:
    # Primary: scale on queue depth per pod
    - type: Pods
      pods:
        metric:
          name: vllm_num_requests_waiting
        target:
          type: AverageValue
          averageValue: "10"   # Scale when avg queue > 10
    # Secondary: scale on KV cache pressure
    - type: Pods
      pods:
        metric:
          name: vllm_gpu_cache_usage_perc
        target:
          type: AverageValue
          averageValue: "0.80" # Scale when KV cache > 80%
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Pods
          value: 2          # Add max 2 pods per 60s
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # 5 min cooldown
      policies:
        - type: Pods
          value: 1          # Remove max 1 pod per 5min
          periodSeconds: 300
Asymmetric scaling is critical: Scale up aggressively (60s window, +2 pods) but scale down conservatively (300s window, -1 pod). GPU pods are expensive to start (model loading = 30-120s) and cheap to keep warm relative to the user experience impact of being under-provisioned.

Scale-to-Zero with KEDA

For development, staging, or low-traffic production endpoints, scale-to-zero eliminates GPU costs during idle periods. KEDA (Kubernetes Event-Driven Autoscaler) provides this capability with external metric triggers.

# KEDA ScaledObject for LLM scale-to-zero
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: llm-serving-keda
  namespace: inference
spec:
  scaleTargetRef:
    name: vllm-llama70b
  minReplicaCount: 0       # Enable scale-to-zero
  maxReplicaCount: 8
  idleReplicaCount: 0      # Zero pods when no traffic
  cooldownPeriod: 600      # 10 min before scaling to zero
  triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus:9090
        metricName: llm_request_rate
        query: |
          sum(rate(
            vllm_request_success_total{deployment="vllm-llama70b"}[2m]
          ))
        threshold: "0.1"    # Scale from 0→1 at >0.1 req/s
  advanced:
    horizontalPodAutoscalerConfig:
      behavior:
        scaleUp:
          stabilizationWindowSeconds: 0  # Instant scale-up from zero

The cold start challenge with scale-to-zero is severe for LLMs. When the first request arrives after an idle period, the pod must: (1) be scheduled on a GPU node (5-15s), (2) pull the container image if not cached (10-60s), (3) load the model into GPU memory (30-120s). Total cold start: 45-195 seconds — unacceptable for synchronous requests.

Cold Start Mitigation Strategies

Cold start is the central challenge of LLM autoscaling. Here are production-tested mitigation strategies, from simple to sophisticated:

🔥 Keep-Warm (Simplest)

Maintain minReplicas: 1 at all times. Cost: one GPU pod 24/7. For an A100 at $2.21/hr, that's $1,613/month — but guarantees P1 latency for the first request.

When to use: Production endpoints where cold start is unacceptable and cost is secondary.

📦 Model Pre-Loading (Fastest Start)

Pre-bake the model into the container image or use a DaemonSet to cache model files on local NVMe. Eliminates download time, reducing cold start to 30-60s (GPU load only).

When to use: Multi-model deployments where you can predict which models will be needed.

🔮 Predictive Pre-Scaling

Analyze historical traffic patterns (time-of-day, day-of-week) and scale up before demand arrives. A CronJob triggers scale-up 15 minutes before predicted peaks.

# Predictive pre-scale CronJob
apiVersion: batch/v1
kind: CronJob
metadata:
  name: llm-prescale-morning
spec:
  # Scale to 4 replicas at 8:45 AM
  schedule: "45 8 * * 1-5"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: scaler
            image: bitnami/kubectl
            command:
            - kubectl
            - scale
            - deployment/vllm-llama70b
            - --replicas=4

🏊 Warm Pool (Most Sophisticated)

Maintain a pool of "warm" pods with the model loaded but not serving traffic. When HPA triggers scale-up, promote a warm pod to active in <5 seconds instead of 60-120s cold start.

# Warm pool controller (concept)
class WarmPoolController:
  def reconcile(self):
    active = get_active_pods()
    warm   = get_warm_pods()
    # Always keep 2 warm pods ready
    if len(warm) < 2:
      create_warm_pod()
    # Promote warm → active on demand
    if needs_scaleup():
      pod = warm.pop()
      pod.add_to_service()
      create_warm_pod()  # replenish
Production recommendation: Use keep-warm (minReplicas=2) for production, predictive pre-scaling for known traffic patterns, and warm pools for large fleets (10+ replicas) where the cost of 2 idle warm pods is small relative to total fleet cost.

Scaling Policies: Target-Based vs Step-Based

Kubernetes HPA supports two scaling policy types. For LLM workloads, the choice matters:

🎯 Target-Based (Recommended)

desiredReplicas = ceil(currentMetricValue / targetValue × currentReplicas)

  • Smooth, proportional scaling
  • Self-correcting (converges to target)
  • Works well with queue depth target
  • Example: target queue depth = 10 per pod. If total queue = 45 with 3 pods → scale to 5

📊 Step-Based (Emergency Scaling)

Discrete steps triggered by metric thresholds.

  • Faster response to sudden spikes
  • Better for binary signals (SLO breach)
  • Risk of over-provisioning (stepping too high)
  • Example: if TTFT P95 > 5s → immediately add 4 pods

In practice, combine both: use target-based for steady-state scaling and step-based as an emergency override when SLOs are breached.

Kubernetes GPU Scheduling

GPU scheduling in Kubernetes adds complexity beyond standard CPU pod scheduling. Key considerations:

# Pod spec for multi-GPU LLM serving
apiVersion: v1
kind: Pod
metadata:
  name: vllm-llama70b-0
spec:
  containers:
  - name: vllm
    image: vllm/vllm-openai:latest
    resources:
      limits:
        nvidia.com/gpu: 4     # Request 4 GPUs for tensor parallelism
      requests:
        nvidia.com/gpu: 4
        memory: "64Gi"
        cpu: "16"
    env:
    - name: NVIDIA_VISIBLE_DEVICES
      value: all
    readinessProbe:           # Model loaded and ready
      httpGet:
        path: /health
        port: 8000
      initialDelaySeconds: 120  # Model load takes ~2 min
      periodSeconds: 10
  nodeSelector:
    nvidia.com/gpu.product: A100-SXM4-80GB
  tolerations:
  - key: nvidia.com/gpu
    operator: Exists
    effect: NoSchedule

Critical GPU scheduling pitfalls:

  • GPU topology awareness: For tensor parallelism, all GPUs must be on the same node with NVLink interconnect. Kubernetes doesn't guarantee GPU topology by default — use the NVIDIA GPU Operator's gpu-feature-discovery to label nodes with GPU topology and set affinity rules.
  • Node pool isolation: Separate GPU node pools by GPU type (A100 vs H100) and size (4-GPU vs 8-GPU nodes). Use node selectors or node affinity to ensure pods land on the right hardware.
  • Cluster Autoscaler / Karpenter: Configure GPU-specific node pools with appropriate scaling limits. Set scale-down-delay-after-add: 10m for GPU node pools (longer than default) to avoid thrashing on expensive GPU nodes.
  • Preemption priority: Set high PriorityClass for serving workloads so they preempt batch jobs when GPU capacity is needed. This acts as a natural overflow mechanism.
Architecture summary: The two-level scaling model works best: HPA scales pods based on LLM-specific metrics (queue depth, KV cache) → Cluster Autoscaler provisions GPU nodes when pods are unschedulable. Each level has different time constants (pods: seconds, nodes: minutes), and both must be tuned for the cold start budget of your SLO.
