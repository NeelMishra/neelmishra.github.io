Multi-Model Serving: LoRA Adapters

MLOps Series LLM Inference & Serving

LoRA Recap & Serving Challenge

Low-Rank Adaptation (LoRA) fine-tunes LLMs by injecting trainable rank-decomposition matrices into frozen pre-trained weights. Instead of updating the full weight matrix W ∈ ℝd×k, LoRA learns ΔW = BA where B ∈ ℝd×r and A ∈ ℝr×k with rank r ≪ min(d, k). Typically r = 8–64, making adapter weights 0.1–1% of the base model size.

# LoRA forward pass for a single linear layer
# Base: h = Wx    LoRA: h = Wx + BAx (with scaling α/r)

class LoRALinear(nn.Module):
    def __init__(self, base_linear, rank=16, alpha=32):
        self.base = base_linear          # Frozen: [d, k]
        self.lora_A = nn.Linear(k, rank, bias=False)  # [r, k]
        self.lora_B = nn.Linear(rank, d, bias=False)  # [d, r]
        self.scaling = alpha / rank

    def forward(self, x):
        base_out = self.base(x)                        # [batch, d]
        lora_out = self.lora_B(self.lora_A(x))         # [batch, d]
        return base_out + self.scaling * lora_out

The serving challenge: an organization may have hundreds of LoRA adapters — one per customer, task, or language — all sharing the same base model (e.g., Llama-2 70B). Naively, you'd need a separate GPU deployment per adapter. With multi-adapter serving, you load the base model once and dynamically apply different adapters per request.

Scale example: A Llama-2 7B base model is 13.5 GB in FP16. A rank-16 LoRA adapter for all attention layers is ~40 MB. Serving 100 adapters naively requires 100 × 13.5 GB = 1,350 GB. With shared base + adapter routing: 13.5 GB + 100 × 40 MB = 17.5 GB — a 77× memory reduction.

Adapter Routing Architecture

In a multi-adapter serving system, each incoming request specifies which adapter to use. The router directs the request to the correct adapter weights while sharing the base model computation across all requests in the same batch.

S-LoRA: Unified Paging

S-LoRA (Sheng et al., 2023) introduces Unified Paging — a memory management system that treats both KV cache pages and adapter weight pages uniformly. This enables serving thousands of LoRA adapters concurrently by swapping adapter weights between GPU and CPU memory on demand.

Naive Multi-LoRA

Pre-load all adapter weights into GPU memory. Limited by GPU capacity — an A100-80GB can hold ~1,600 rank-16 adapters for Llama-7B alongside the base model and KV cache.

Problem: With 10,000+ adapters, most don't fit in GPU memory. Cold start latency when loading from CPU/disk.

S-LoRA Unified Paging

Adapter weights are paged into GPU memory on demand using the same page table mechanism as KV cache (from PagedAttention). Hot adapters stay on GPU; cold adapters are evicted to CPU memory with LRU policy.

Result: Serve 10,000+ adapters with near-zero overhead for popular ones.

# S-LoRA: Unified Paging pseudo-code
class UnifiedPageManager:
    def __init__(self, gpu_pages, cpu_pages, page_size=256):
        self.gpu_pool = PagePool(gpu_pages, device="cuda")
        self.cpu_pool = PagePool(cpu_pages, device="cpu")
        self.page_table = {}   # Maps (adapter_id, layer, type) → page_list
        self.lru_cache = LRUCache()

    def get_adapter_pages(self, adapter_id, layer_idx):
        key = (adapter_id, layer_idx)
        if key in self.gpu_pool:
            self.lru_cache.touch(key)       # Update LRU
            return self.page_table[key]

        # Page fault: load from CPU → GPU
        cpu_pages = self.cpu_pool.get(key)
        gpu_pages = self.gpu_pool.allocate(len(cpu_pages))

        if gpu_pages is None:
            # Evict least-recently-used adapter from GPU
            victim = self.lru_cache.evict()
            self.swap_out(victim)           # GPU → CPU (async)
            gpu_pages = self.gpu_pool.allocate(len(cpu_pages))

        self.swap_in(cpu_pages, gpu_pages)  # CPU → GPU (async)
        self.page_table[key] = gpu_pages
        return gpu_pages

Custom CUDA Kernels for Batched LoRA

S-LoRA implements custom CUDA kernels that efficiently batch LoRA computations across requests with different adapters. The key insight: instead of separate GEMM calls per adapter, S-LoRA uses a segmented gather-GEMM-scatter pattern.

# Batched LoRA GEMM: different adapters in same batch
# For each request i with adapter a_i:
#   h_lora_i = B_{a_i} @ (A_{a_i} @ x_i)

# Naive: N separate small GEMMs (poor GPU utilization)
# S-LoRA: Fused segmented GEMM

def batched_lora_forward(x_batch, adapter_ids, A_pool, B_pool):
    # x_batch: [total_tokens, hidden_dim]
    # adapter_ids: [total_tokens] — which adapter for each token

    # Step 1: Gather — reorder tokens by adapter ID
    sorted_indices = argsort(adapter_ids)
    x_sorted = x_batch[sorted_indices]

    # Step 2: Segmented GEMM — one GEMM per adapter segment
    segments = compute_segments(adapter_ids[sorted_indices])
    lora_out = segmented_gemm(x_sorted, A_pool, B_pool, segments)

    # Step 3: Scatter — reorder back to original positions
    output = zeros_like(x_batch)
    output[sorted_indices] = lora_out
    return output

vLLM LoRA Support

vLLM natively supports multi-LoRA serving with a simple API. You register adapters at startup or dynamically, and clients specify the adapter per request via the model field in OpenAI-compatible requests.

# Launch vLLM with LoRA support
# vllm serve meta-llama/Llama-2-7b-hf \
#   --enable-lora \
#   --lora-modules customer-A=./adapters/customer_a \
#                  customer-B=./adapters/customer_b \
#   --max-loras 8 \
#   --max-lora-rank 64

# Python API
from vllm import LLM, SamplingParams
from vllm.lora.request import LoRARequest

llm = LLM(
    model="meta-llama/Llama-2-7b-hf",
    enable_lora=True,
    max_loras=8,          # Max concurrent adapters in GPU memory
    max_lora_rank=64,     # Max rank across all adapters
    max_cpu_loras=100,    # Additional adapters in CPU for S-LoRA swapping
)

# Each request specifies its adapter
outputs = llm.generate(
    ["Summarize this contract..."],
    SamplingParams(temperature=0.7, max_tokens=512),
    lora_request=LoRARequest(
        lora_name="legal-adapter",
        lora_int_id=1,
        lora_local_path="./adapters/legal_v2"
    )
)

Dynamic Adapter Loading

In production, adapters are added and updated frequently. A robust system must support loading new adapters without server restart, hot-swapping updated adapter versions, and graceful degradation when adapters are unavailable.

# Dynamic adapter management service
class AdapterManager:
    def __init__(self, llm_engine, adapter_store_path):
        self.engine = llm_engine
        self.store = adapter_store_path
        self.loaded = {}       # adapter_name → (version, lora_int_id)
        self.next_id = 1

    async def register_adapter(self, name, path, version):
        """Load a new adapter or hot-swap an existing one."""
        if name in self.loaded:
            old_version, old_id = self.loaded[name]
            if old_version == version:
                return  # Already loaded, skip
            # Wait for in-flight requests using old version to drain
            await self.drain_requests(name, timeout=30)
            self.engine.remove_lora(old_id)

        lora_id = self.next_id
        self.next_id += 1
        self.engine.add_lora(LoRARequest(name, lora_id, path))
        self.loaded[name] = (version, lora_id)

    async def unload_adapter(self, name):
        """Gracefully remove an adapter."""
        if name not in self.loaded:
            return
        await self.drain_requests(name, timeout=60)
        _, lora_id = self.loaded.pop(name)
        self.engine.remove_lora(lora_id)
Hot-swap pitfall: Never remove an adapter while requests are actively using it. Implement request draining with a timeout — redirect new requests to the updated adapter while allowing in-flight requests on the old version to complete. vLLM handles this internally when using max_cpu_loras with S-LoRA-style paging.

Production Patterns

🏢 Multi-Tenant SaaS

Pattern: One base model, one LoRA per customer. Route by API key → customer → adapter name.

Sizing: For 500 customers with rank-16 adapters on Llama-7B: 13.5 GB base + 20 GB adapters. Fits on 1× A100-80GB with S-LoRA paging for cold adapters.

Latency: Hot adapters add <1ms. Cold adapter page-in adds 2-5ms (from CPU) or 50-200ms (from disk).

🔬 A/B Testing Adapters

Pattern: Multiple adapter versions coexist. Traffic routing splits requests between adapter_v1 and adapter_v2 based on experiment config.

Implementation: Register both versions with distinct names. Use a feature flag service to select the adapter per request. Log adapter version in metrics for comparison.

🌍 Language Specialization

Pattern: Base multilingual model + language-specific LoRA adapters. Auto-detect language from input, route to appropriate adapter.

Benefit: Each adapter improves fluency in its language without degrading the base model's capabilities. 20 language adapters add only ~800 MB total.

📋 Task-Specific Routing

Pattern: One base model with adapters for summarization, translation, code generation, etc. A lightweight classifier routes requests to the appropriate adapter.

Optimization: Batch requests by adapter ID to maximize GPU GEMM efficiency. S-LoRA's segmented GEMM handles mixed-adapter batches, but same-adapter batches are still 10-30% faster.

Key Takeaway: Multi-LoRA serving transforms the economics of LLM deployment. Instead of N separate model deployments for N use cases, you deploy one base model and swap adapters per-request. This reduces GPU costs by 10-100× while maintaining task-specific quality, and enables use cases like per-customer fine-tuning that would otherwise be prohibitively expensive.
