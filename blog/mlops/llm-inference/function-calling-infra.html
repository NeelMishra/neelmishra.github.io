Function Calling Infrastructure

MLOps Series LLM Inference & Serving

Function calling (tool use) transforms LLMs from text generators into action-taking agents. But behind the clean API surface — tools=[...] in the OpenAI API — lies significant serving infrastructure: special token handling, constrained decoding for tool call arguments, multi-turn context management, and streaming complications. This post covers the end-to-end infrastructure that makes function calling work at scale.

How Function Calling Works

At the inference level, function calling is not a separate model capability — it's structured output with special tokens. The model has been fine-tuned to recognize tool definitions in the system prompt and emit structured tool call blocks when it decides a tool is needed.

The lifecycle of a function call:

  1. Tool definitions are serialized into the system prompt as structured text (JSON schemas describing each function's name, parameters, and descriptions).
  2. The model generates text and, when it decides to invoke a tool, emits a special token sequence that marks the start of a tool call.
  3. The serving infrastructure detects the tool call token, switches to constrained decoding to ensure valid JSON arguments, and stops generation at the end-of-tool-call token.
  4. The tool call is returned to the client with finish_reason: "tool_calls".
  5. The client executes the function and sends the result back as a tool message.
  6. The model continues generation with the tool result injected into context.

Special Tokens & Formats

Different model families use different special tokens to delimit tool calls. The serving infrastructure must recognize these and handle them correctly.

Llama 3.1 / 3.2 Format

# Special tokens:
<|python_tag|>  # code interpreter
<|eom_id|>     # end of tool call

# Tool call output format:
<|start_header_id|>assistant<|end_header_id|>

<|python_tag|>
{"name": "get_weather",
 "parameters": {"city": "SF"}}
<|eom_id|>

Mistral / Mixtral Format

# Special tokens:
[TOOL_CALLS]  # start of tool calls
[TOOL_RESULTS]  # tool result marker

# Tool call output format:
[TOOL_CALLS]
[{"name": "get_weather",
  "arguments": {"city": "SF"},
  "id": "call_abc123"}]

# Tool result injection:
[TOOL_RESULTS]
{"call_id":"call_abc123",
 "content":"72°F, sunny"}
[/TOOL_RESULTS]
## OpenAI API format — the industry standard
## This is what clients send/receive regardless of the underlying model format

## Request: define available tools
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get current weather for a city",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string"},
                    "units": {"type": "string", "enum": ["celsius", "fahrenheit"]}
                },
                "required": ["city"]
            }
        }
    }
]

## Response: model decides to call a tool
{
    "choices": [{
        "message": {
            "role": "assistant",
            "content": null,
            "tool_calls": [{
                "id": "call_abc123",
                "type": "function",
                "function": {
                    "name": "get_weather",
                    "arguments": "{\"city\": \"San Francisco\", \"units\": \"fahrenheit\"}"
                }
            }]
        },
        "finish_reason": "tool_calls"
    }]
}
Key serving infrastructure role: The inference server must translate between the model-specific token format (Llama's <|python_tag|>, Mistral's [TOOL_CALLS]) and the standardized OpenAI API format. This is done in the chat template processor, typically via Jinja2 templates that ship with each model. vLLM, TGI, and SGLang all handle this translation layer.

Serving Flow

The serving infrastructure handles several non-trivial tasks during function calling:

## Serving infrastructure for function calls
class ToolCallHandler:
    def __init__(self, model_config):
        self.tool_start_token = model_config.tool_start_token
        self.tool_end_token = model_config.tool_end_token
        # Optional: constrained decoding for tool args
        self.arg_decoder = ConstrainedDecoder()

    def process_generation(self, token_ids, tool_schemas):
        """Called after each generated token."""
        decoded = self.tokenizer.decode(token_ids)

        if self.detected_tool_start(token_ids):
            # Switch to constrained decoding mode
            # Build a JSON schema from the tool definitions
            tool_call_schema = self.build_tool_call_schema(tool_schemas)
            self.arg_decoder.activate(tool_call_schema)
            return {"state": "tool_call", "constrained": True}

        if self.detected_tool_end(token_ids):
            # Parse the completed tool call
            tool_json = self.extract_tool_json(decoded)
            self.arg_decoder.deactivate()
            return {
                "state": "tool_call_complete",
                "tool_call": tool_json,
                "finish_reason": "tool_calls"
            }

        return {"state": "generating"}

    def inject_tool_result(self, conversation, tool_call_id, result):
        """Add tool result back into the conversation context."""
        conversation.append({
            "role": "tool",
            "tool_call_id": tool_call_id,
            "content": str(result)
        })
        # Re-tokenize and continue generation
        # The KV cache from the previous turn can be reused!
        return self.resume_generation(conversation)
KV cache reuse across tool calls: When the tool result comes back, the server can reuse the KV cache from the first generation turn. It only needs to process the new tool result tokens through prefill, not the entire conversation history. This is why prefix caching is essential for tool-use workloads — a 3-turn tool interaction reuses most of the KV cache from turn 1.

Parallel Function Calls

Modern models can emit multiple tool calls in a single generation turn. The model generates multiple tool call blocks sequentially, and the client can execute them in parallel:

## Parallel tool call handling
import asyncio
from openai import AsyncOpenAI

client = AsyncOpenAI(base_url="http://localhost:8000/v1")

async def handle_tool_calls(response):
    tool_calls = response.choices[0].message.tool_calls
    if not tool_calls:
        return response.choices[0].message.content

    # Execute all tool calls in parallel
    tasks = []
    for tc in tool_calls:
        func = TOOL_REGISTRY[tc.function.name]
        args = json.loads(tc.function.arguments)
        tasks.append(execute_tool(tc.id, func, args))

    results = await asyncio.gather(*tasks)

    # Send all results back in one request
    messages = conversation + [response.choices[0].message]
    for tool_call_id, result in results:
        messages.append({
            "role": "tool",
            "tool_call_id": tool_call_id,
            "content": json.dumps(result)
        })

    # Resume generation with all tool results
    continuation = await client.chat.completions.create(
        model="meta-llama/Llama-3.1-70B-Instruct",
        messages=messages
    )
    return await handle_tool_calls(continuation)  # recursive for multi-hop

## Example: model calls get_weather("NYC") AND get_weather("SF") in parallel
## Client executes both API calls concurrently → ~1 RTT instead of 2
Parallel calls reduce latency significantly. If a model needs data from 3 independent APIs (each taking ~200ms), sequential execution takes ~600ms for tool execution. Parallel execution takes ~200ms. Combined with the model generation time (~500ms per turn), this cuts total latency from ~2.1s to ~1.3s for a typical tool-use interaction.

Streaming with Tools

Streaming function calls is one of the trickiest parts of the infrastructure. The challenge: the client doesn't know if the model will emit text or a tool call until it starts generating. And tool call arguments arrive token by token.

## Streaming tool calls — partial argument accumulation
async def stream_with_tools(client, messages, tools):
    stream = await client.chat.completions.create(
        model="meta-llama/Llama-3.1-70B-Instruct",
        messages=messages,
        tools=tools,
        stream=True
    )

    tool_calls_buffer = {}  # id → accumulated arguments
    text_buffer = ""

    async for chunk in stream:
        delta = chunk.choices[0].delta

        # Case 1: Regular text content
        if delta.content:
            text_buffer += delta.content
            yield_text(delta.content)

        # Case 2: Tool call chunks
        if delta.tool_calls:
            for tc_chunk in delta.tool_calls:
                idx = tc_chunk.index
                if idx not in tool_calls_buffer:
                    tool_calls_buffer[idx] = {
                        "id": tc_chunk.id,
                        "name": tc_chunk.function.name or "",
                        "arguments": ""
                    }
                if tc_chunk.function.name:
                    tool_calls_buffer[idx]["name"] += tc_chunk.function.name
                if tc_chunk.function.arguments:
                    tool_calls_buffer[idx]["arguments"] += tc_chunk.function.arguments

        # Case 3: Stream ended
        if chunk.choices[0].finish_reason == "tool_calls":
            # All tool calls fully accumulated — execute them
            for tc in tool_calls_buffer.values():
                yield_tool_call(tc)

## Chunk 2: {"function": {"arguments": 'ty": "'}}
## Chunk 3: {"function": {"arguments": 'San Fra'}}
## Chunk 4: {"function": {"arguments": 'ncisco"}'}}

Latency Optimization

Function calling adds significant latency due to multiple model invocations and external API calls. Here are the key optimization strategies:

Server-Side Optimizations

  • Prefix caching: Reuse KV cache from system prompt + tool definitions across all requests (saves 200-500ms per turn)
  • Constrained decoding: Enforce valid tool call JSON → fewer retries, shorter output
  • Speculative decoding: Use small model draft for tool call args (highly predictable tokens)
  • Batched tool results: Process all tool results in one prefill pass

Client-Side Optimizations

  • Parallel execution: Run independent tool calls concurrently
  • Speculative execution: Pre-fetch likely tool results before model decides
  • Tool result caching: Cache deterministic tool results (e.g., math operations)
  • Timeout + fallback: Return partial results if tool takes >2s
## Latency breakdown for typical tool-use interaction
## Model: Llama-3.1-70B on 8×H100, 4K context

## Turn 1: User query → tool call decision
##   Prefill (system + tools + user msg, ~2K tokens):  120ms
##   Decode tool call (~50 tokens, 30 tok/s):            ~1.7s
##   Subtotal:                                           ~1.8s

## Tool execution (external API):
##   Network RTT + processing:                           ~200ms

## Turn 2: Tool result → final response
##   Prefill (tool result, ~100 tokens, KV cache reused): 15ms
##   Decode response (~100 tokens, 30 tok/s):             ~3.3s
##   Subtotal:                                            ~3.3s

## Total end-to-end: ~5.3s
## With optimizations (prefix cache + parallel tools): ~3.8s
Production tip: Tool definitions expand the system prompt significantly (each tool adds ~200-500 tokens). With 20+ tools, the system prompt alone can be 10K+ tokens. Prefix caching is not optional here — it turns a 500ms prefill into a 5ms cache hit for every request after the first. Always enable --enable-prefix-caching in vLLM for tool-use workloads.

Key takeaway: Function calling infrastructure bridges the gap between LLM text generation and programmatic tool execution. The critical components are: chat template translation (model-specific → OpenAI format), constrained decoding for tool arguments, KV cache reuse across multi-turn tool interactions, and parallel tool execution on the client side. Getting this right can reduce end-to-end tool-use latency by 30-40%.
