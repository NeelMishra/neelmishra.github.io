TensorRT-LLM: NVIDIA's Runtime

MLOps Series LLM Inference & Serving

TensorRT-LLM Overview

TensorRT-LLM is NVIDIA's purpose-built library for compiling and serving LLMs on NVIDIA GPUs. Unlike PyTorch-based frameworks (vLLM, TGI), TensorRT-LLM compiles models into optimized TensorRT engines—static computation graphs with fused CUDA kernels that extract maximum performance from the hardware.

The performance gap is real: on H100 GPUs, TensorRT-LLM achieves up to 2× higher throughput than vLLM for Llama-70B at the same latency target, primarily through aggressive kernel fusion, FP8 quantization (H100 Transformer Engine), and CUDA graph execution.

Advantages

  • Highest throughput on NVIDIA hardware
  • FP8 on H100 (near-FP16 quality, 2× speed)
  • Deep kernel fusion (GemmPlugin, etc.)
  • In-flight batching with C++ runtime
  • Triton Inference Server integration
  • Pipeline + Tensor parallelism

Trade-offs

  • Long compilation time (30 min – 2 hrs)
  • Engine is GPU-architecture specific
  • Less model coverage than vLLM/TGI
  • More complex setup & debugging
  • NVIDIA-only (no AMD/Intel support)
  • Python API less mature than CLI tools
When to choose TensorRT-LLM: When you need maximum tokens/sec/$, are running on NVIDIA GPUs (especially H100/H200), are willing to invest in build-time complexity for runtime performance, and your model is supported by TensorRT-LLM's model definitions.

Compilation Pipeline

TensorRT-LLM's build process transforms a HuggingFace/checkpoint model into an optimized TensorRT engine through a multi-stage pipeline:

# Step 1: Convert HuggingFace checkpoint to TensorRT-LLM format
python convert_checkpoint.py \
    --model_dir /models/Llama-3.1-70B-Instruct \
    --output_dir /engines/llama70b/ckpt \
    --dtype float16 \
    --tp_size 4 \
    --pp_size 1

# Step 2: Build TensorRT engine
trtllm-build \
    --checkpoint_dir /engines/llama70b/ckpt \
    --output_dir /engines/llama70b/engine \
    --gemm_plugin float16 \
    --gpt_attention_plugin float16 \
    --max_batch_size 64 \
    --max_input_len 4096 \
    --max_seq_len 8192 \
    --workers 4  # parallel build across GPUs

# Engine size: ~35GB per GPU (70B / TP4)
# Build time: ~45 min on 4×A100
Important: The engine is compiled for a specific max_batch_size and max_seq_len. If you need to handle 128-token sequences and later need 8192, you must rebuild. Plan your max dimensions carefully—overbuild wastes memory, underbuild limits flexibility.

Graph Optimizations

The TensorRT builder applies several optimization passes that are impossible in eager-mode PyTorch:

1. Layer Fusion: Adjacent operations are merged into a single CUDA kernel, eliminating intermediate memory reads/writes. Key fusions include:

  • QKV Fusion: Three separate linear projections (Q, K, V) merge into a single GEMM with 3× wider output
  • MLP Fusion: Gate + Up projections in Llama-style SwiGLU fuse into one kernel
  • Residual + LayerNorm: The add-then-normalize pattern becomes a single fused kernel
  • Attention + Softmax + Dropout: Entire attention score computation fuses (similar to Flash Attention)

2. Kernel Auto-Tuning: TensorRT profiles hundreds of kernel implementations for each operation (different tile sizes, shared memory configurations, thread block dimensions) and selects the fastest for the exact tensor shapes and GPU architecture. This is why build takes long—it's benchmarking.

3. Memory Optimization: Activation tensors that don't overlap in lifetime share the same physical memory, reducing peak memory by 20–40% compared to PyTorch's allocation strategy.

GemmPlugin (Recommended)

  • Custom GEMM kernels for LLM shapes
  • Handles variable sequence lengths
  • Supports FP16, BF16, FP8, INT8, INT4
  • 10–30% faster than cuBLAS for small batches
  • Enable: --gemm_plugin float16

GPT Attention Plugin

  • Fused multi-head attention kernel
  • Integrates RoPE in-kernel
  • Paged KV-cache with block tables
  • Flash Attention equivalent performance
  • Enable: --gpt_attention_plugin float16

FP8 / INT8 / INT4 Quantization

TensorRT-LLM offers the most comprehensive quantization stack for NVIDIA hardware:

FP8 (H100/H200 only): The H100's Transformer Engine supports native FP8 computation with per-tensor dynamic scaling. FP8 halves memory vs FP16 while achieving <0.5% quality loss on most benchmarks. The key insight is using E4M3 format for weights and E5M2 for gradients/activations.

# Quantize to FP8 using NVIDIA's quantization toolkit
from tensorrt_llm.quantization import quantize

# Calibration with representative dataset (512 samples recommended)
quantize(
    model_dir="/models/Llama-3.1-70B-Instruct",
    output_dir="/models/Llama-3.1-70B-FP8",
    dtype="float16",
    qformat="fp8",
    calib_size=512,
    calib_dataset="cnn_dailymail",
    tp_size=4
)

# Build engine with FP8
# trtllm-build --checkpoint_dir /models/Llama-3.1-70B-FP8 ...
# Result: 70B model → ~70GB total (17.5GB/GPU with TP=4)
# vs FP16: 140GB total (35GB/GPU with TP=4)

INT4-AWQ / INT4-GPTQ: 4-bit weight-only quantization achieves a ~4× memory reduction. TensorRT-LLM uses custom W4A16 GEMM kernels that dequantize INT4 weights to FP16 on-the-fly during matrix multiplication, keeping activations in full precision.

FP8 (Recommended for H100)

  • Memory: 2× reduction (70B → 70GB)
  • Quality: < 0.5% degradation
  • Speed: ~1.8× vs FP16 (Tensor Cores)
  • Calibration: 512 samples, ~10 min
  • Best for: latency-sensitive production

INT4-AWQ (Best for A100)

  • Memory: 4× reduction (70B → 35GB)
  • Quality: 2–5% degradation
  • Speed: ~1.3× vs FP16 (dequant overhead)
  • Calibration: 128 samples, ~20 min
  • Best for: fitting large models on fewer GPUs

SmoothQuant (INT8): Migrates quantization difficulty from activations to weights by applying a mathematically equivalent per-channel scaling. This enables INT8 for both weights and activations (W8A8), using INT8 Tensor Cores for ~1.5× speedup on A100.

In-Flight Batching

TensorRT-LLM's C++ runtime implements in-flight batching (continuous batching) with several enhancements over Python-based schedulers:

  • Zero-overhead scheduling: The C++ executor schedules the next batch with sub-microsecond overhead, compared to ~100μs in Python runtimes. At 50 tok/s decode speed, this matters.
  • Paged KV-cache: Same block-table approach as vLLM, but with tighter integration into fused attention kernels.
  • Chunked context (prefill chunking): Long prompts are split into chunks of max_num_tokens and processed across multiple iterations, preventing a single long-prompt prefill from blocking all decode steps.
# Using the TensorRT-LLM Python API for serving
import tensorrt_llm
from tensorrt_llm.executor import Executor, Request, SamplingConfig

executor = Executor(
    model_path="/engines/llama70b/engine",
    executor_config=tensorrt_llm.ExecutorConfig(
        max_beam_width=1,
        batching_type=tensorrt_llm.BatchingType.INFLIGHT,
        kv_cache_config=tensorrt_llm.KvCacheConfig(
            free_gpu_memory_fraction=0.85
        )
    )
)

# Submit requests asynchronously
request = Request(
    input_token_ids=[1, 15043, 29892, 825, 338],  # "Hello, what is"
    sampling_config=SamplingConfig(
        max_new_tokens=256,
        temperature=0.7,
        top_p=0.9
    )
)

request_id = executor.enqueue_request(request)

# Poll for results (or use callback)
while True:
    responses = executor.await_responses(timeout=1.0)
    for resp in responses:
        if resp.has_error():
            print(f"Error: {resp.error_msg}")
        else:
            print(resp.output_token_ids)
            if resp.is_final:
                break

Multi-GPU Parallelism

TensorRT-LLM supports both tensor and pipeline parallelism, configured at build time:

Tensor Parallelism (TP): Splits each layer across GPUs. Attention heads are partitioned (e.g., 80 heads ÷ 4 GPUs = 20 heads/GPU). FFN uses column-parallel first linear and row-parallel second linear. Requires 2 AllReduce per layer (one for attention, one for FFN). Best within a node (NVLink bandwidth: 900 GB/s on H100).

Pipeline Parallelism (PP): Assigns layer groups to different GPUs. GPU 0 runs layers 0–19, GPU 1 runs layers 20–39, etc. Requires only point-to-point communication between adjacent stages. Uses micro-batching to keep all stages busy—the batch is split into micro-batches that pipeline through stages.

Tensor Parallel (TP=4)

  • All GPUs see all layers
  • Communication: AllReduce per layer
  • Bandwidth: 2 × hidden_size × 2 bytes × 2
  • Latency: low (single AllReduce)
  • Best: intra-node (NVLink)

Pipeline Parallel (PP=2)

  • Each GPU sees a subset of layers
  • Communication: P2P between stages
  • Bandwidth: batch_size × hidden_size × 2 bytes
  • Latency: pipeline bubble overhead
  • Best: inter-node or mixed configs
# Build a 70B model with TP=4, PP=2 across 8 GPUs
python convert_checkpoint.py \
    --model_dir /models/Llama-3.1-70B-Instruct \
    --output_dir /engines/llama70b_tp4pp2/ckpt \
    --dtype float16 \
    --tp_size 4 \
    --pp_size 2

trtllm-build \
    --checkpoint_dir /engines/llama70b_tp4pp2/ckpt \
    --output_dir /engines/llama70b_tp4pp2/engine \
    --gemm_plugin float16 \
    --gpt_attention_plugin float16 \
    --max_batch_size 128 \
    --max_input_len 4096 \
    --max_seq_len 8192

# Launch with mpirun
mpirun -n 8 --allow-run-as-root \
    python run.py \
    --engine_dir /engines/llama70b_tp4pp2/engine \
    --max_output_len 512
TP × PP rule of thumb: Use TP within a node (up to 8 for DGX H100), PP across nodes. For 70B on 2 nodes of 4×A100: TP=4, PP=2. For 70B on 1 node of 8×H100: TP=8, PP=1. Minimize PP to reduce pipeline bubble overhead.

Triton Inference Server Integration

For production deployments, TensorRT-LLM engines are served via NVIDIA Triton Inference Server, which provides load balancing, model versioning, dynamic batching, and multi-model serving:

# Triton model repository structure
model_repository/
├── llama70b/
│   ├── config.pbtxt
│   └── 1/
│       └── # empty (engines loaded from engine_dir)
├── preprocessing/
│   ├── config.pbtxt
│   └── 1/
│       └── model.py  # tokenization
└── postprocessing/
    ├── config.pbtxt
    └── 1/
        └── model.py  # detokenization

# Launch Triton with TRT-LLM backend
docker run --gpus all --rm -p 8000:8000 -p 8001:8001 \
    -v /model_repository:/models \
    -v /engines:/engines \
    nvcr.io/nvidia/tritonserver:24.05-trtllm-python-py3 \
    tritonserver --model-repository=/models \
    --model-control-mode=explicit \
    --load-model=llama70b

Triton adds features that TensorRT-LLM's native runtime lacks:

  • Ensemble models: Chain preprocessing → LLM → postprocessing as a single logical model
  • Model versioning: Blue-green deployments with instant rollback
  • Metrics: Prometheus metrics for request latency, throughput, queue depth, GPU utilization
  • Multi-model: Serve multiple LLMs on the same GPU with time-slicing
Operational complexity: The TensorRT-LLM + Triton stack has the highest setup complexity among LLM serving frameworks. Budget 1–3 days for initial setup and benchmarking. However, for large-scale production on NVIDIA hardware, the 30–50% throughput gain over vLLM/TGI often justifies the investment.

TensorRT-LLM represents the ceiling of what's achievable on NVIDIA GPUs. Its ahead-of-time compilation, hardware-specific kernel tuning, and FP8 support on H100 make it the go-to choice when every token/sec/$ matters. The trade-off is operational complexity and build-time rigidity—every configuration change triggers a 30+ minute rebuild.
