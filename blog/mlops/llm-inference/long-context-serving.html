Serving Long-Context LLMs

MLOps Series LLM Inference & Serving

Long-context models like Llama-3.1 (128K), Claude (200K), and Gemini (1M+) have transformed what LLMs can process. But serving these models in production is an entirely different beast — the KV cache alone for a 128K-token request on a 70B model can exceed 160 GB. This post dives into the systems-level techniques that make long-context serving feasible: chunked prefill, ring attention, sparse attention, and practical memory management.

KV Cache Memory Math

Every transformer layer stores key and value tensors for each token that has been processed. For a model with L layers, H attention heads, and head dimension d, the KV cache per token is:

## KV cache size calculation
kv_bytes_per_token = 2 × L × H_kv × d × bytes_per_param

## Llama-2 70B: L=80, H_kv=8 (GQA), d=128, FP16 (2 bytes)
kv_per_token = 2 × 80 × 8 × 128 × 2 = 327,680 bytes ≈ 320 KB

## For 128K tokens:
total_kv = 320 KB × 131,072 = ~40 GB per request

## With 4 concurrent requests:
total_kv_4req = 40 GB × 4 = 160 GB just for KV cache

## Compare: model weights (FP16) ≈ 140 GB
## KV cache can exceed the model itself!
Critical insight: At 128K context length, the KV cache for a single Llama-70B request (40 GB in FP16) approaches 30% of the model weights themselves. With just 4 concurrent users, KV cache alone requires more memory than the model. This is why naive long-context serving is impossible on current hardware without optimization.

Key parameters that determine KV cache size:

Model Architecture

  • GQA ratio: Llama-2 70B uses 8 KV heads vs 64 query heads (8× reduction)
  • MQA: Falcon uses 1 KV head (64× reduction) but lower quality
  • Layer count: 80 layers in 70B vs 32 in 7B

Quantization

  • FP16 → FP8: 2× reduction, minimal quality loss
  • FP16 → INT4: 4× reduction, noticeable degradation past 32K tokens
  • KV cache quantization is independent of weight quantization

Chunked Prefill

The prefill phase processes the entire prompt in one forward pass. For a 128K-token prompt, this single forward pass can take 30+ seconds on a single GPU, during which no other requests can be served. Chunked prefill breaks the prompt into smaller chunks (e.g., 2048 or 4096 tokens) and processes them across multiple iterations, interleaving decode steps for other requests.

## vLLM chunked prefill configuration
from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-3.1-70B-Instruct",
    tensor_parallel_size=4,
    # Enable chunked prefill
    enable_chunked_prefill=True,
    # Max tokens per prefill chunk — smaller = better latency for others
    max_num_batched_tokens=4096,
    # Max number of sequences in a batch
    max_num_seqs=128,
    # KV cache dtype — FP8 halves memory
    kv_cache_dtype="fp8_e5m2",
)

## Chunk size trade-offs:
## 2048: Best ITL for concurrent decodes, slower total prefill
## 4096: Good balance (vLLM default)
## 8192: Faster prefill, higher ITL variance
Why chunked prefill matters: Without it, a single 128K-token request monopolizes the GPU for the entire prefill duration (~30s on an A100). Every other request sees TTFT spike to 30+ seconds. With chunked prefill (chunk size 4096), ongoing requests maintain ~50ms inter-token latency while the long prompt is processed incrementally.

Ring Attention

Ring attention distributes the sequence across multiple devices in a ring topology. Each device holds a contiguous block of the sequence and computes local attention. The KV blocks are then passed around the ring so that each device eventually attends to all positions — but never needs to store the full KV cache locally.

How it works:

  1. Partition the sequence of length S into P devices: each holds S/P tokens.
  2. Each device computes Q×KT for its local KV block and stores partial softmax statistics (log-sum-exp).
  3. KV blocks rotate around the ring — each device sends its KV block to the next device and receives from the previous.
  4. After P-1 rotations, every device has attended to all positions. Final attention output is computed using the online softmax correction (log-sum-exp trick).
## Ring attention — each device processes one block
def ring_attention(Q_local, K_local, V_local, ring_comm):
    """Each device has Q for its block, and iterates over all K,V blocks."""
    P = ring_comm.world_size
    running_numerator = zeros_like(Q_local @ V_local[0])
    running_denominator = zeros_like(Q_local.shape[0])
    running_max = full(Q_local.shape[0], float('-inf'))

    K_buf, V_buf = K_local, V_local

    for step in range(P):
        # Overlap communication with computation
        if step < P - 1:
            K_next, V_next = ring_comm.async_send_recv(K_buf, V_buf)

        # Local attention: Q_local × K_buf^T
        scores = Q_local @ K_buf.T / sqrt(d_head)
        # Apply causal mask if needed
        scores = apply_causal_mask(scores, step, ring_comm.rank)

        # Online softmax update (numerically stable)
        block_max = scores.max(dim=-1)
        new_max = maximum(running_max, block_max)
        correction = exp(running_max - new_max)
        new_weights = exp(scores - new_max.unsqueeze(-1))

        running_numerator = correction.unsqueeze(-1) * running_numerator + new_weights @ V_buf
        running_denominator = correction * running_denominator + new_weights.sum(dim=-1)
        running_max = new_max

        if step < P - 1:
            K_buf, V_buf = K_next.wait(), V_next.wait()

    return running_numerator / running_denominator.unsqueeze(-1)
Scaling: Ring attention achieves near-linear scaling. With 8 GPUs, each device holds 16K tokens of a 128K sequence. KV cache per device drops from 40 GB to ~5 GB. Communication overhead is hidden behind computation via double-buffering, adding only ~5-8% overhead in practice.

Sliding Window & Sparse Attention

Full quadratic attention is often unnecessary. Many production models use structured sparsity to reduce both computation and KV cache requirements:

Sliding Window Attention

Each token attends to the last W tokens (e.g., W=4096 in Mistral). Memory scales as O(W) instead of O(S). Used in Mistral-7B, Mixtral-8x7B.

  • KV cache: Fixed at W × kv_per_token regardless of sequence length
  • Mistral 7B: W=4096 → only 1.3 GB KV per request
  • Limitation: Cannot recall information beyond the window

Sparse Attention (e.g., BigBird, LongFormer)

Combines local windows with global tokens and random connections. O(S) complexity instead of O(S²).

  • Global tokens: First few tokens attend to all positions (anchors)
  • Local window: Each token attends to ±W neighbors
  • Random: Small set of random connections per token

Hybrid approaches in modern models like Llama-3.1 use full attention in some layers and sliding window in others. For example, every 4th layer might use full attention while others use a 4096-token window. This balances long-range recall with memory efficiency.

## Sliding window KV cache eviction in serving
class SlidingWindowKVCache:
    def __init__(self, window_size=4096, num_layers=32):
        self.window = window_size
        # Circular buffer per layer
        self.k_cache = [CircularBuffer(window_size) for _ in range(num_layers)]
        self.v_cache = [CircularBuffer(window_size) for _ in range(num_layers)]

    def append(self, layer_idx, new_k, new_v):
        # Old entries are overwritten automatically
        self.k_cache[layer_idx].push(new_k)
        self.v_cache[layer_idx].push(new_v)
        # Memory stays constant at window_size × kv_per_token × num_layers

Memory-Efficient Attention Implementations

Even with architectural changes, the attention computation itself must be memory-efficient. Key implementations:

FlashAttention-2

  • Fuses Q×KT, softmax, and ×V into a single kernel
  • Never materializes the S×S attention matrix in HBM
  • Uses tiling: processes attention in blocks that fit in SRAM (192 KB on A100)
  • Memory: O(S) instead of O(S²) — critical for 128K+ sequences
  • Speed: 2-4× faster than standard attention at 8K+ sequence lengths

PagedAttention (vLLM)

  • Manages KV cache like virtual memory with fixed-size blocks (16 tokens/block)
  • Eliminates fragmentation from variable-length sequences
  • Enables KV cache sharing across requests (prompt caching)
  • Memory waste: Reduced from 60-80% to <4% on average
  • Supports copy-on-write for beam search and shared prefixes
## FlashAttention-2 conceptual tiling
def flash_attention_forward(Q, K, V, block_size=128):
    """Process attention in SRAM-sized tiles."""
    S, d = Q.shape
    O = zeros(S, d)          # output accumulator
    l = zeros(S)              # log-sum-exp accumulator
    m = full(S, float('-inf'))  # running max

    for j in range(0, S, block_size):      # iterate over K,V blocks
        K_j = K[j:j+block_size]
        V_j = V[j:j+block_size]
        for i in range(0, S, block_size):  # iterate over Q blocks
            Q_i = Q[i:i+block_size]
            # All of this happens in SRAM — no HBM reads for S×S matrix
            S_ij = Q_i @ K_j.T / sqrt(d)
            m_new = maximum(m[i:i+block_size], S_ij.max(dim=-1))
            P_ij = exp(S_ij - m_new.unsqueeze(-1))
            # Online softmax correction
            correction = exp(m[i:i+block_size] - m_new)
            l[i:i+block_size] = correction * l[i:i+block_size] + P_ij.sum(dim=-1)
            O[i:i+block_size] = correction.unsqueeze(-1) * O[i:i+block_size] + P_ij @ V_j
            m[i:i+block_size] = m_new

    return O / l.unsqueeze(-1)

Deployment Strategies

Putting it all together — practical configurations for serving long-context models in production:

Small Scale (1-2 GPUs)

  • Model: Llama-3.1-8B, 32K context
  • Hardware: 1× A100 80GB
  • KV cache quantization: FP8 (halves KV memory)
  • Chunked prefill: 4096 tokens/chunk
  • Max concurrent requests: ~8-12
  • Effective throughput: ~500 tok/s

Production Scale (8+ GPUs)

  • Model: Llama-3.1-70B, 128K context
  • Hardware: 8× H100 80GB (TP=8)
  • Ring attention for sequences >64K
  • KV cache: FP8 + PagedAttention
  • Prefix caching for system prompts
  • Max concurrent long-context: 4-6 requests

## Key monitoring metrics:
## - gpu_cache_usage_perc: KV cache utilization (alert at >90%)
## - num_preemptions: requests evicted due to memory pressure
## - avg_prompt_throughput_toks_per_s: prefill speed
## - avg_generation_throughput_toks_per_s: decode speed
Prefix caching is critical for long-context deployments. If many requests share a common system prompt (e.g., 4K tokens of instructions), prefix caching stores this KV cache once and reuses it across all requests. For a 4K shared prefix with 100 requests, this saves 100 × 4K × 320 KB ≈ 122 GB of redundant KV computation and memory.

Key takeaway: Serving long-context models is primarily a memory management problem. The combination of chunked prefill (latency isolation), KV cache quantization (2× memory reduction), PagedAttention (eliminate fragmentation), and prefix caching (deduplicate shared context) makes 128K-token serving practical on 8× H100 nodes. Beyond 128K, ring attention or model-specific architectural changes (sliding window + sparse) become essential.
