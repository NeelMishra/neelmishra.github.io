Scaling Laws: Chinchilla, Kaplan

MLOps Series Training Infrastructure

Power Laws in Deep Learning

One of the most remarkable empirical findings in deep learning is that model performance follows power laws as a function of three variables: model parameters (N), training data (D), and compute budget (C). When plotted on log-log axes, loss decreases as a straight line — indicating predictable, smooth scaling over many orders of magnitude.

The key insight: these power laws have no inflection points. There is no "wall" where scaling stops working. Doubling compute always yields a predictable improvement in loss. This observation is what drives the race toward ever-larger models and clusters.

The Three Variables

  • N (Parameters): The total number of trainable parameters in the model. More parameters = more capacity to model complex distributions.
  • D (Data/Tokens): The total number of training tokens. More data = better coverage of the target distribution.
  • C (Compute): Total training FLOPs, approximately C ≈ 6ND for a standard transformer forward+backward pass. This is the binding constraint in practice.
Why C ≈ 6ND? Each token's forward pass through N parameters costs ~2N FLOPs (one multiply + one add per parameter). The backward pass costs ~2× the forward (computing gradients for both weights and activations). Total: 2N + 4N = 6N FLOPs per token, multiplied by D tokens.

Kaplan Scaling Laws

The original scaling laws paper by Kaplan et al. (OpenAI, January 2020) established three key power-law relationships for autoregressive language models:

The Three Laws

  • L(N) ∝ N−0.076: Loss scales as a power law of model size, with exponent ~0.076. A 10× increase in parameters yields ~0.076 × log₁₀(10) ≈ 17% reduction in loss.
  • L(D) ∝ D−0.095: Loss scales with data, exponent ~0.095. Data is slightly more efficient than parameters per unit increase.
  • L(C) ∝ C−0.050: When compute is allocated optimally between N and D, loss scales with total FLOPs at exponent ~0.050.

Kaplan's Compute-Optimal Allocation

Kaplan et al. found that given a fixed compute budget, most of it should go to increasing model size rather than training on more data. Specifically:

  • Optimal allocation: N ∝ C0.73 and D ∝ C0.27
  • Implication: large models trained on relatively little data (compared to their size) are compute-optimal
  • This led to training practices like GPT-3 (175B parameters, 300B tokens — a ratio of ~1.7 tokens per parameter)
# Kaplan scaling law formulas
# Loss as function of model size (parameters)
def loss_vs_params(N):
    # N in billions of parameters
    L_inf = 1.69   # irreducible loss (entropy of natural language)
    a = 8.8e3      # scaling coefficient (fitted)
    alpha = 0.076  # scaling exponent
    return (a / N) ** alpha + L_inf

# Compute-optimal allocation under Kaplan
def kaplan_optimal(C_flops):
    # Given compute budget C, find optimal N and D
    N_opt = 1.3e9 * (C_flops / 1e18) ** 0.73   # parameters
    D_opt = C_flops / (6 * N_opt)                # tokens (from C ≈ 6ND)
    return N_opt, D_opt

# Example: 1e23 FLOPs budget (roughly GPT-3 scale)
N, D = kaplan_optimal(1e23)
# → N ≈ 175B params, D ≈ 95B tokens
# GPT-3 actually used 300B tokens — slightly over-trained by Kaplan's rule
Important caveat: Kaplan's experiments used a fixed learning rate schedule across all model sizes, which systematically disadvantaged smaller models that converge faster. This methodological issue led Chinchilla to reach very different conclusions about the optimal N-to-D ratio.

Chinchilla Scaling Laws

Hoffmann et al. (DeepMind, March 2022) revisited the scaling laws with a more careful experimental setup and reached a fundamentally different conclusion: models should be trained on far more data than Kaplan suggested.

Key Findings

  • Optimal allocation: N ∝ C0.50 and D ∝ C0.50 — parameters and data should scale equally with compute.
  • Rule of thumb: train on ~20 tokens per parameter (vs Kaplan's ~1.7 tokens/parameter).
  • Chinchilla (70B params, 1.4T tokens) matched the performance of Gopher (280B params, 300B tokens) using the same compute budget.

Kaplan (OpenAI, 2020)

  • N ∝ C0.73, D ∝ C0.27
  • ~1.7 tokens per parameter
  • Bigger model, less data
  • GPT-3: 175B params, 300B tokens
  • Fixed LR schedule across sizes
  • Optimizes training compute only

Chinchilla (DeepMind, 2022)

  • N ∝ C0.50, D ∝ C0.50
  • ~20 tokens per parameter
  • Balanced model size and data
  • Chinchilla: 70B params, 1.4T tokens
  • Per-model optimal LR tuning
  • Optimizes training compute only

Why Chinchilla Differs

The main methodological improvements in Chinchilla:

  1. Proper LR tuning: Each model size gets its own optimal learning rate and cosine schedule, rather than sharing a schedule. This revealed that small models were under-performing due to suboptimal hyperparameters in Kaplan's setup.
  2. Three independent approaches: Chinchilla used three different experimental methodologies (fixed-N varying-D, fixed-C varying-N/D, and parametric loss fitting) — all converged to similar conclusions.
  3. Wider range of model sizes: Models from 70M to 16B parameters were tested, giving more reliable estimates of scaling exponents.
The Chinchilla tax: If you've already trained a model under Kaplan rules (big model, little data), you can't fix it by training longer — the learning rate schedule has already decayed. You need to restart from scratch with a new schedule. This realization cost many organizations significant compute budgets.

Compute-Optimal Training

Given a fixed compute budget C (in FLOPs), compute-optimal training finds the (N, D) pair that minimizes the final loss. Under Chinchilla scaling:

# Chinchilla compute-optimal planning
def chinchilla_optimal(C_flops):
    """Given compute budget C, find optimal model size and data."""
    # Under Chinchilla: N* ∝ C^0.5, D* ∝ C^0.5
    # With D ≈ 20 × N, and C ≈ 6ND:
    # C = 6 × N × 20N = 120N²
    # N* = sqrt(C / 120)

    N_opt = (C_flops / 120) ** 0.5         # optimal parameters
    D_opt = C_flops / (6 * N_opt)          # optimal tokens
    ratio = D_opt / N_opt                   # should be ~20

    return {
        "params": N_opt,
        "tokens": D_opt,
        "ratio": ratio,
    }

# Practical examples at different compute scales
budgets = {
    "Small (1e19)":  1e19,   # ~1 A100 for 1 day
    "Medium (1e21)": 1e21,   # ~100 A100s for 1 day
    "Large (1e23)":  1e23,   # ~1000 A100s for 2 weeks
    "Frontier (1e25)": 1e25, # ~10000 H100s for months
}

for name, C in budgets.items():
    result = chinchilla_optimal(C)
    print(f"{name}:")
    print(f"  Params: {result['params']:.1e}")
    print(f"  Tokens: {result['tokens']:.1e}")
    print(f"  Ratio:  {result['ratio']:.0f} tokens/param")

# Output:
# Small (1e19):   Params: 2.9e+08 (~300M), Tokens: 5.7e+09 (~6B)
# Medium (1e21):  Params: 2.9e+09 (~3B),   Tokens: 5.7e+10 (~57B)
# Large (1e23):   Params: 2.9e+10 (~29B),  Tokens: 5.7e+11 (~570B)
# Frontier (1e25): Params: 2.9e+11 (~290B), Tokens: 5.7e+12 (~5.7T)

Practical Constraints

  • Memory limits: A 290B parameter model requires ~580 GB in fp16 for weights alone. With optimizer states (Adam: 2× for momentum and variance), activations, and gradients, you need significant model parallelism even on 80 GB GPUs.
  • Data availability: 5.7T tokens of high-quality text is extremely difficult to source. Common Crawl after deduplication and filtering yields ~1–3T tokens. This is a real constraint for frontier scaling.
  • Wall-clock time: More tokens = longer training time. A 5.7T token run at 1M tokens/sec still takes 66 days. Cluster reliability becomes critical.

Beyond Chinchilla

Chinchilla optimizes for training compute. But in production, inference compute often dominates total cost. This insight has led to a post-Chinchilla paradigm: over-train smaller models for inference efficiency.

The LLaMA Approach

Meta's LLaMA (February 2023) deliberately violated Chinchilla scaling by training smaller models on much more data:

  • LLaMA-7B: 7B parameters trained on 1T tokens (ratio: ~143 tokens/param, vs Chinchilla's ~20)
  • LLaMA-13B: 13B parameters on 1T tokens (~77 tokens/param)
  • LLaMA-65B: 65B parameters on 1.4T tokens (~22 tokens/param — close to Chinchilla)

Why? Because inference cost is proportional to N (model size), but training cost is proportional to N×D. By over-training a 7B model, Meta spent more on training but got a model that's 10× cheaper per inference query compared to a Chinchilla-optimal 70B model with similar performance.

Chinchilla-Optimal (Training-First)

  • Minimizes training FLOP cost
  • ~20 tokens per parameter
  • Larger models, trained less
  • Best when training once, inferring rarely
  • Research-oriented philosophy

Over-Training (Inference-First)

  • Minimizes total lifecycle cost
  • 50–200+ tokens per parameter
  • Smaller models, trained longer
  • Best for high-volume inference products
  • Production-oriented philosophy

Data-Constrained Scaling

When unique data is limited, practitioners face a choice: train for fewer tokens (risking under-training) or repeat data (risking memorization). Muennighoff et al. (2023) studied this tradeoff:

  • Up to 4 epochs: Repeating data works with negligible degradation. Each epoch beyond the first contributes roughly as much as ~60% of unique data.
  • Beyond 4 epochs: Returns diminish rapidly. The model starts memorizing and validation loss plateaus.
  • Mixing strategies: Code data tolerates more repetition than natural language. Domain-specific data (medical, legal) is more sensitive to repetition.
Data quality > quantity: Recent work (Phi, TinyStories, DCLM) shows that high-quality filtered data can shift scaling curves upward. A 1.3B model trained on curated data can match a 7B model trained on unfiltered web data. Filtering effectively gives you "free" tokens per parameter.

Infrastructure Implications

Scaling laws are not just academic curiosities — they directly inform infrastructure decisions. Every GPU cluster sizing, training time estimate, and cost budget should be grounded in scaling law predictions.

Cluster Sizing from Scaling Laws

# Infrastructure planning using scaling laws
def plan_training_run(target_loss, model_family="dense_transformer"):
    """Plan infrastructure from a target validation loss."""

    # Step 1: Estimate required compute from loss target
    # L(C) = a * C^(-alpha) + L_inf
    # C = (a / (L_target - L_inf))^(1/alpha)
    a = 3.1e8           # fitted for this model family
    alpha = 0.050
    L_inf = 1.69
    C_required = (a / (target_loss - L_inf)) ** (1 / alpha)

    # Step 2: Find optimal N, D from Chinchilla
    N_opt = (C_required / 120) ** 0.5
    D_opt = C_required / (6 * N_opt)

    # Step 3: Estimate GPU requirements
    gpu_flops = 312e12          # H100 bf16 peak FLOPS
    mfu = 0.45                  # model FLOP utilization (typical)
    effective_flops = gpu_flops * mfu
    gpu_seconds = C_required / effective_flops
    target_days = 30            # finish in 1 month
    num_gpus = gpu_seconds / (target_days * 86400)

    # Step 4: Memory estimation for model parallelism
    bytes_per_param = 2              # bf16
    optimizer_multiplier = 12        # Adam states + gradients in fp32
    total_memory_gb = N_opt * optimizer_multiplier / 1e9
    gpu_memory_gb = 80               # H100 80GB
    min_mp_degree = ceil(total_memory_gb / gpu_memory_gb)

    return {
        "compute_flops": C_required,
        "params": N_opt,
        "tokens": D_opt,
        "num_gpus": ceil(num_gpus),
        "training_days": target_days,
        "model_parallel_degree": min_mp_degree,
        "estimated_cost": ceil(num_gpus) * target_days * 24 * 2.50,  # $/GPU-hr
    }

Cost Estimation Framework

  • GPU-hours = C / (GPU_FLOPS × MFU): With H100 at 312 TFLOPS bf16 and 45% MFU, effective throughput is ~140 TFLOPS per GPU.
  • Cloud cost: H100 on-demand: $2.50–3.50/GPU-hr. Reserved instances: $1.50–2.00/GPU-hr. Spot/preemptible: $0.80–1.50/GPU-hr (with interruption risk).
  • Scaling tax: Communication overhead reduces effective FLOPS by 10–30% as you scale from 1 to 1000+ GPUs. Factor this into estimates.
  • Failed runs: Budget 2–3× the compute for a single successful run. Hyperparameter sweeps, diverged runs, and hardware failures are inevitable.
Rough cost estimates (2024 cloud pricing): 1e21 FLOPs ≈ $500–1K (small model). 1e23 FLOPs ≈ $50–100K (medium). 1e24 FLOPs ≈ $500K–1M (large). 1e25 FLOPs ≈ $5–10M (frontier). These assume reserved H100 pricing and ~40% MFU.

Practical Application

Scaling laws are most useful for two things: (1) predicting the performance of a large run from small-scale experiments, and (2) deciding how to allocate a fixed budget between model size and data.

Predicting Final Loss from Small Runs

import numpy as np
from scipy.optimize import curve_fit

# Step 1: Run small-scale experiments
# Train 4-6 models at different scales (e.g., 10M to 1B params)
# Each trained to Chinchilla-optimal tokens
experiments = [
    {"N": 1e7,  "D": 2e8,  "loss": 3.45},   # 10M params, 200M tokens
    {"N": 5e7,  "D": 1e9,  "loss": 3.12},   # 50M params, 1B tokens
    {"N": 2e8,  "D": 4e9,  "loss": 2.85},   # 200M params, 4B tokens
    {"N": 5e8,  "D": 1e10, "loss": 2.68},   # 500M params, 10B tokens
    {"N": 1e9,  "D": 2e10, "loss": 2.55},   # 1B params, 20B tokens
]

# Step 2: Fit the scaling law L(C) = a * C^(-alpha) + L_inf
def scaling_law(C, a, alpha, L_inf):
    return a * C ** (-alpha) + L_inf

C_values = np.array([6 * e["N"] * e["D"] for e in experiments])
L_values = np.array([e["loss"] for e in experiments])

params, _ = curve_fit(scaling_law, C_values, L_values, p0=[1e3, 0.05, 1.5])
a_fit, alpha_fit, L_inf_fit = params
print(f"Fitted: a={a_fit:.2e}, alpha={alpha_fit:.4f}, L_inf={L_inf_fit:.3f}")

# Step 3: Extrapolate to target compute
C_target = 1e23  # 100× more compute than largest experiment
predicted_loss = scaling_law(C_target, a_fit, alpha_fit, L_inf_fit)
print(f"Predicted loss at {C_target:.0e} FLOPs: {predicted_loss:.3f}")

Extrapolation Confidence

  • 1 order of magnitude: Predictions are typically within 5% of actual. Very reliable for planning.
  • 2 orders of magnitude: Predictions are within 10–15%. Useful for budgeting but allow safety margin.
  • 3+ orders of magnitude: Increasingly uncertain. Emergent capabilities may appear that violate smooth scaling. Use as rough guidance only.

Common Pitfalls in Applying Scaling Laws

  • Architecture changes: Scaling laws are fitted to a specific architecture family. Switching from dense to MoE, or changing attention mechanisms, invalidates the fit — you need new small-scale experiments.
  • Data distribution shifts: Laws fitted on web data don't transfer to code, math, or domain-specific text. Each data mixture needs its own calibration runs.
  • Downstream task performance: Scaling laws predict pre-training loss, not downstream task accuracy. Task performance can exhibit emergent jumps that don't follow smooth power laws.
  • Hyperparameter sensitivity: Batch size, learning rate schedule, and optimizer choice can shift the scaling curve. Keep these consistent between calibration and target runs.
Interview takeaway: Scaling laws convert ML research from an empirical art into something closer to engineering. Before committing $10M to a frontier training run, you can spend $10K on calibration experiments and predict the final loss to within 10%. This changes the economics of large-scale AI development and is a core skill for ML infrastructure engineers.
