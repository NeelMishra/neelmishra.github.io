GPU Memory Management: OOM Debugging

MLOps Series Training Infrastructure

Anatomy of GPU Memory Usage

When you call model.to("cuda") and start training, GPU memory is consumed by four major categories. Understanding the relative size of each is the first step to solving OOM errors.

For a model with P parameters in FP32, the baseline memory requirement is:

  • Parameters: 4P bytes (FP32) or 2P bytes (FP16/BF16)
  • Gradients: Same size as parameters
  • Adam optimizer states: 2 × parameter size (first & second moments)
  • Activations: Depends on batch size, sequence length, and model architecture

A 7B parameter model in FP32 requires at minimum: 7B × 4 bytes × 4 (params + grads + 2 optimizer states) = 112 GB just for static state—before a single activation is stored.

The 4× Rule: With Adam in FP32, you need roughly 4× the model parameter memory just for training state. This is why a 7B model doesn't fit on a single 80 GB GPU without mixed precision or optimizer tricks.

Memory Profiling Tools

Before optimizing, you need to measure. PyTorch provides excellent built-in tools for GPU memory introspection.

# Quick memory snapshot
print(torch.cuda.memory_summary(device="cuda:0"))

# Key metrics to watch:
allocated   = torch.cuda.memory_allocated()     # currently used by tensors
reserved    = torch.cuda.memory_reserved()      # held by caching allocator
max_alloc   = torch.cuda.max_memory_allocated()  # peak usage (the OOM trigger)

# Reset peak stats between experiments
torch.cuda.reset_peak_memory_stats()

# Memory snapshot for detailed analysis (PyTorch 2.1+)
torch.cuda.memory._record_memory_history(max_entries=100000)
# ... run your training step ...
torch.cuda.memory._dump_snapshot("memory_snapshot.pickle")
torch.cuda.memory._record_memory_history(enabled=None)

The memory snapshot can be visualized with PyTorch's Memory Visualizer (torch.cuda.memory._snapshot()) which shows allocation timelines, identifies the largest tensors, and traces them back to the Python source code that created them.

Important: memory_reserved is always ≥ memory_allocated. The difference is fragmentation—memory held by PyTorch's caching allocator but not currently used by tensors. If fragmentation is high, try torch.cuda.empty_cache() (frees reserved memory back to CUDA) or PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True.

OOM Debugging Strategies

When you hit CUDA out of memory, resist the urge to immediately halve batch size. Instead, follow a systematic debugging approach:

Step 1: Identify the peak. Use max_memory_allocated() to find peak usage. Run a single training step and check where the peak occurs—usually during the backward pass when both activations and gradients coexist.

# OOM debugging helper
def debug_memory(tag=""):
    alloc = torch.cuda.memory_allocated() / 1e9
    peak  = torch.cuda.max_memory_allocated() / 1e9
    print(f"[{tag}] Allocated: {alloc:.2f} GB | Peak: {peak:.2f} GB")

debug_memory("after model load")
outputs = model(inputs)
debug_memory("after forward")
loss.backward()
debug_memory("after backward")   # usually the peak
optimizer.step()
debug_memory("after optimizer step")

Step 2: Check for leaks. Memory should return to baseline after each step. If it grows monotonically, you have a leak—common causes:

  • Appending loss tensors to a list without .item() (keeps entire computation graph alive)
  • Storing tensors in global scope or module attributes
  • Not detaching tensors used for logging: use loss.detach().item()

Step 3: Reduce strategically. Apply techniques in this order of impact:

  1. Mixed precision (saves ~50% memory)
  2. Gradient checkpointing (saves activation memory at ~33% compute cost)
  3. Reduce batch size (linear memory savings)
  4. Gradient accumulation (maintain effective batch size with smaller micro-batches)
  5. CPU offloading for optimizer states
Pro Tip: Use CUDA_LAUNCH_BLOCKING=1 to get accurate stack traces for CUDA OOM errors. Without it, the error may point to the wrong line because CUDA operations are asynchronous.

Gradient Checkpointing for Memory

During the backward pass, PyTorch needs the activations from the forward pass to compute gradients. Normally, all activations are stored in memory. Gradient checkpointing trades compute for memory: instead of storing all activations, it stores only a subset (checkpoints) and recomputes the rest during backward.

Memory savings formula: For a model with L layers, standard training stores O(L) activations. With checkpointing every √L layers, memory drops to O(√L) at the cost of one extra forward pass (~33% compute overhead).

from torch.utils.checkpoint import checkpoint_sequential

class CheckpointedModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = nn.ModuleList([
            TransformerBlock(d_model=4096) for _ in range(32)
        ])

    def forward(self, x):
        # Checkpoint every 4 layers (8 segments for 32 layers)
        return checkpoint_sequential(self.layers, segments=8, input=x)

# Or per-layer checkpointing with use_reentrant=False (recommended)
from torch.utils.checkpoint import checkpoint

class TransformerBlock(nn.Module):
    def forward(self, x):
        return checkpoint(self._forward_impl, x, use_reentrant=False)

    def _forward_impl(self, x):
        x = x + self.attention(self.norm1(x))
        x = x + self.ffn(self.norm2(x))
        return x
Warning: Use use_reentrant=False (the new default in PyTorch 2.4+). The legacy use_reentrant=True has subtle bugs with torch.autograd.grad() and doesn't support all autograd features.

Mixed Precision for Memory Savings

Mixed precision training stores model weights, activations, and gradients in FP16 or BF16 (2 bytes) instead of FP32 (4 bytes), roughly halving memory usage while maintaining training quality.

scaler = torch.amp.GradScaler()  # only needed for FP16, not BF16

for inputs, targets in train_loader:
    optimizer.zero_grad(set_to_none=True)

    # Autocast: forward pass in FP16/BF16
    with torch.amp.autocast(device_type="cuda", dtype=torch.bfloat16):
        outputs = model(inputs)
        loss = criterion(outputs, targets)

    # BF16: just call backward directly (no scaler needed)
    loss.backward()
    optimizer.step()

    # FP16 alternative: use GradScaler to prevent underflow
    # scaler.scale(loss).backward()
    # scaler.step(optimizer)
    # scaler.update()

FP16

  • Range: ±65,504
  • Requires GradScaler for loss scaling
  • Risk of overflow/underflow in gradients
  • Widely supported (V100+)

BF16

  • Range: same as FP32 (±3.4×10³⁸)
  • No GradScaler needed — same exponent range as FP32
  • Less precision (7-bit mantissa vs 10-bit)
  • Ampere+ (A100, H100)
Recommendation: If your hardware supports BF16 (A100+), prefer it over FP16. BF16 has the same dynamic range as FP32, eliminating the need for loss scaling and reducing the risk of training instability.

Advanced Techniques

When mixed precision and gradient checkpointing aren't enough, these advanced techniques can push the boundary further:

1. CPU Offloading: Move optimizer states to CPU RAM. With ZeRO-Offload (DeepSpeed), optimizer steps run on CPU while forward/backward stay on GPU. Trades PCIe bandwidth for GPU memory.

# DeepSpeed ZeRO-Offload config
{
    "zero_optimization": {
        "stage": 2,
        "offload_optimizer": {
            "device": "cpu",
            "pin_memory": true
        }
    }
}

2. 8-bit Optimizers: bitsandbytes provides 8-bit Adam that stores optimizer states in INT8, reducing optimizer memory by 75% with minimal quality loss.

import bitsandbytes as bnb

optimizer = bnb.optim.Adam8bit(
    model.parameters(),
    lr=1e-4,
    betas=(0.9, 0.999)
)
# Drop-in replacement — same API as torch.optim.Adam
# Optimizer states: 2 bytes/param instead of 8 bytes/param

3. Activation Offloading: Move activations to CPU during forward pass, fetch them back during backward. Effective when PCIe bandwidth isn't the bottleneck (large models with slow backward).

4. Memory-Efficient Attention: FlashAttention computes attention without materializing the full N×N attention matrix, reducing memory from O(N²) to O(N). This is now the default in PyTorch 2.0+ via torch.nn.functional.scaled_dot_product_attention.

Memory Budget Calculator

Use these formulas to estimate GPU memory requirements before running a single experiment:

# Memory estimation formulas
def estimate_training_memory(
    num_params_billions,
    batch_size,
    seq_len,
    hidden_dim,
    num_layers,
    precision="bf16",   # "fp32", "bf16", "fp16"
    optimizer="adam",    # "adam", "sgd", "adam8bit"
    gradient_checkpointing=False
):
    P = num_params_billions * 1e9
    bytes_per_param = 2 if precision in ("bf16", "fp16") else 4

    # Static memory
    param_mem   = P * bytes_per_param
    grad_mem    = P * bytes_per_param
    if optimizer == "adam":
        optim_mem = P * 8                  # FP32 momentum + variance
    elif optimizer == "adam8bit":
        optim_mem = P * 2                  # INT8 states
    else:
        optim_mem = P * 4                  # SGD momentum

    # Activation memory (rough estimate for transformers)
    act_per_layer = batch_size * seq_len * hidden_dim * bytes_per_param * 12
    if gradient_checkpointing:
        act_mem = act_per_layer * (num_layers ** 0.5)  # √L layers stored
    else:
        act_mem = act_per_layer * num_layers

    total_gb = (param_mem + grad_mem + optim_mem + act_mem) / 1e9
    return total_gb

Quick Reference Table (approximate):

Model Size → Min GPU Memory

  • 1B params: ~20 GB (BF16 + Adam)
  • 7B params: ~70 GB (BF16 + Adam) — fits 1× A100 80GB with checkpointing
  • 13B params: ~130 GB — needs 2× A100 or ZeRO
  • 70B params: ~700 GB — needs 8+ GPUs with ZeRO Stage 3

Savings per Technique

  • BF16 mixed precision: ~50% savings on params/grads/activations
  • Gradient checkpointing: ~60% savings on activations
  • 8-bit Adam: ~75% savings on optimizer states
  • FlashAttention: O(N) vs O(N²) attention memory
Interview Formula: For a transformer with P parameters in BF16 with Adam: total ≈ 2P (params) + 2P (grads) + 8P (Adam FP32 states) + activations = 12P bytes + activations. For P = 7B, that's 84 GB + activations—right at the limit of an 80 GB A100.
