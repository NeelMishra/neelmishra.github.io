Pipeline Parallelism: GPipe and PipeDream

MLOps Series Training Infrastructure

Pipeline Parallelism Concept

Pipeline parallelism partitions a model's layers into stages, each assigned to a different device. A mini-batch flows through the stages sequentially — GPU 0 runs the first group of layers, sends activations to GPU 1 for the next group, and so on. The backward pass reverses this flow, sending gradients from the last stage back to the first.

The Sequential Dependency Problem

Stage k cannot begin its forward pass for micro-batch i until stage k-1 finishes and sends activations. Similarly, stage k cannot begin backward until stage k+1 sends back gradients. This creates idle time — the dreaded pipeline bubble.

The Cost: With p pipeline stages and 1 micro-batch, each GPU is idle for (p-1)/p of the total time. For 4 stages, that's 75% idle — clearly unacceptable. The solution: micro-batching.

GPipe

GPipe (Google, 2019) splits a mini-batch into m micro-batches and pipelines them through stages. All forward passes complete before any backward passes begin (synchronous schedule). Gradients from all micro-batches are accumulated and applied in a single optimizer step.

GPipe Characteristics

  • Synchronous: All forwards complete before any backward. Equivalent to gradient accumulation — no weight staleness.
  • Memory cost: Must store activations for all m micro-batches across all stages (for backward pass). Mitigated with activation recomputation.
  • Bubble ratio: (p - 1) / (m + p - 1) where p = stages, m = micro-batches.
Key GPipe Design Choice: By keeping the schedule synchronous, GPipe guarantees the same convergence semantics as standard gradient accumulation. The cost is higher peak memory since activations from all micro-batches must be kept until the backward pass.

PipeDream (1F1B)

PipeDream (Microsoft, 2019) introduces the 1F1B (one-forward-one-backward) schedule. After the pipeline fills (warmup phase), each GPU alternates: do one forward, then one backward. This dramatically reduces peak memory since each stage only needs to buffer a bounded number of activations.

1F1B Schedule

# 1F1B schedule for 4 stages (steady state after warmup)
# Each cell is one time step
#
# S0: F1  F2  F3  F4  B1  F5  B2  F6  B3  F7  B4  ...
# S1:     F1  F2  F3  B1  F4  B2  F5  B3  F6  B4  ...
# S2:         F1  F2  B1  F3  B2  F4  B3  F5  B4  ...
# S3:             F1  B1  F2  B2  F3  B3  F4  B4  ...
#
# Warmup: first p micro-batches fill the pipeline (forward only)
# Steady state: alternate 1F, 1B per stage
# Cooldown: drain remaining backwards

Memory Advantage

In GPipe, stage 0 must store activations for all m micro-batches. In 1F1B, stage k stores at most p - k activations at any time (where p is the number of stages). This is because each stage starts its backward as soon as the downstream stage sends gradients — activations are freed earlier.

GPipe

  • All-forward-then-all-backward
  • Peak activations: m × a per stage
  • No weight staleness
  • Simple gradient synchronization
  • Higher memory, same convergence

PipeDream 1F1B

  • Interleaved forward/backward
  • Peak activations: p × a (stage 0)
  • Original PipeDream has weight staleness
  • PipeDream-Flush avoids staleness
  • Lower memory, similar bubble ratio
Weight Staleness: In the original PipeDream, different micro-batches may use different weight versions because backward updates happen mid-pipeline. PipeDream-Flush (a.k.a. PipeDream-2BW) fixes this by flushing the pipeline at the end of each batch — all micro-batches use the same weights, matching GPipe's convergence guarantees.

Interleaved Schedules

The interleaved schedule (introduced in Megatron-LM v2) assigns multiple virtual stages to each GPU. Instead of each GPU owning one contiguous chunk of layers, each GPU owns several non-contiguous chunks. This reduces the bubble ratio by a factor of v (number of virtual stages per GPU).

How It Works

# Standard PP: 32 layers, 4 GPUs
# GPU 0: layers 0-7,  GPU 1: layers 8-15
# GPU 2: layers 16-23, GPU 3: layers 24-31

# Interleaved PP (v=2 virtual stages per GPU):
# GPU 0: layers 0-3 + layers 16-19  (stages 0, 4)
# GPU 1: layers 4-7 + layers 20-23  (stages 1, 5)
# GPU 2: layers 8-11 + layers 24-27 (stages 2, 6)
# GPU 3: layers 12-15 + layers 28-31 (stages 3, 7)

# Bubble ratio reduction:
# Standard: (p-1) / (m+p-1)
# Interleaved: (p-1) / (v×m + p-1)
# With v=2, it's as if we doubled the micro-batch count!

Trade-off: More Communication

Interleaving requires more point-to-point communication. In standard PP, activations flow GPU 0 → 1 → 2 → 3. With interleaving, they flow 0 → 1 → 2 → 3 → 0 → 1 → 2 → 3 (through all virtual stages). The total communication volume increases by a factor of v. However, since PP uses only point-to-point send/recv (not collective), this is acceptable when bandwidth is sufficient.

Megatron-LM Default: Megatron-LM uses interleaved scheduling by default with v=number_of_layers/(pipeline_parallel_size × layers_per_virtual_stage). Typical settings: v=2–4 virtual stages per GPU, combined with tensor parallelism within each stage.

Bubble Overhead Analysis

The fundamental formula governing pipeline efficiency is:

# Pipeline bubble ratio
bubble_ratio = (p - 1) / (m + p - 1)

# where:
#   p = number of pipeline stages
#   m = number of micro-batches

# Examples (p = 4 stages):
#   m = 1:   bubble = 3/4 = 75.0%
#   m = 4:   bubble = 3/7 = 42.9%
#   m = 8:   bubble = 3/11 = 27.3%
#   m = 16:  bubble = 3/19 = 15.8%
#   m = 32:  bubble = 3/35 =  8.6%
#   m = 64:  bubble = 3/67 =  4.5%

# Rule of thumb: m ≥ 4×p for bubble < 20%

# Interleaved schedule (v virtual stages per GPU):
bubble_ratio_interleaved = (p - 1) / (v * m + p - 1)
# With v=2, m=16, p=4: bubble = 3/35 = 8.6% (same as m=32 standard)

Effective Throughput

The throughput of a pipeline-parallel system is:

# Effective throughput per step
throughput = (1 - bubble_ratio) × p × single_gpu_throughput

# Scaling efficiency
pipeline_efficiency = 1 - bubble_ratio = m / (m + p - 1)

# To achieve >90% efficiency: m > 9 × (p - 1)
# For p=8: m > 63, so m=64 micro-batches
Memory Constraint on m: Increasing micro-batches reduces bubble overhead but each micro-batch's activations must be stored. With gradient accumulation, the total effective batch size = m × micro_batch_size × dp_size. This must align with the desired global batch size for training stability.

Implementation Patterns

PyTorch torch.distributed.pipelining

import torch
from torch.distributed.pipelining import SplitPoint, pipeline, ScheduleGPipe

# Step 1: Define split points (where to cut the model)
split_spec = {
    "layers.8": SplitPoint.BEGINNING,   # Stage 1 starts at layer 8
    "layers.16": SplitPoint.BEGINNING,  # Stage 2 starts at layer 16
    "layers.24": SplitPoint.BEGINNING,  # Stage 3 starts at layer 24
}

# Step 2: Create pipeline from the model
pipe = pipeline(
    module=model,
    mb_args=(micro_batch_input,),
    split_spec=split_spec,
)

# Step 3: Create a schedule (GPipe or 1F1B)
schedule = ScheduleGPipe(
    pipe,
    n_microbatches=8,
    loss_fn=loss_fn,
)

# Step 4: Run one training step
if rank == 3:  # Last stage computes loss
    losses = schedule.step(target=labels)
else:
    schedule.step()

Manual Stage Splitting

class PipelineStage(torch.nn.Module):
    def __init__(self, layers, stage_id, num_stages):
        super().__init__()
        self.layers = torch.nn.ModuleList(layers)
        self.stage_id = stage_id
        self.is_first = (stage_id == 0)
        self.is_last = (stage_id == num_stages - 1)

    def forward(self, x):
        for layer in self.layers:
            x = layer(x)
        return x

# Partition the model's layers evenly
def partition_model(model, num_stages, rank):
    all_layers = list(model.layers)
    chunk_size = len(all_layers) // num_stages
    start = rank * chunk_size
    end = start + chunk_size if rank < num_stages - 1 else len(all_layers)
    stage = PipelineStage(all_layers[start:end], rank, num_stages)
    return stage.to(f"cuda:{rank}")

Trade-offs & Best Practices

Key Trade-offs

More Micro-batches (↑m)

  • ✅ Lower bubble ratio
  • ✅ Better GPU utilization
  • ❌ More activation memory
  • ❌ Larger effective batch size
  • ❌ Smaller micro-batch per GPU → less compute efficiency per kernel

More Pipeline Stages (↑p)

  • ✅ Fewer layers per GPU → less memory
  • ✅ Enables very large models
  • ❌ Higher bubble ratio (need more m to compensate)
  • ❌ More communication rounds
  • ❌ Load balancing harder (first/last stages differ)

Best Practices

  1. Set m ≥ 4p: Aim for bubble overhead below 20%. Many frameworks default to even higher ratios.
  2. Use activation recomputation: Reduces peak memory from O(m × layers × hidden²) to O(m × layers × hidden). Essential for GPipe.
  3. Balance stages carefully: The first and last stages often have extra ops (embedding, LM head). Use profiling to ensure even compute time per stage.
  4. Combine with tensor parallelism: Use TP within each pipeline stage to reduce per-stage memory further. Standard practice in Megatron-LM.
  5. Prefer 1F1B over GPipe for memory-constrained scenarios. Most modern frameworks (Megatron-LM, DeepSpeed) use 1F1B by default.
  6. Consider interleaving: If communication bandwidth allows, interleaved scheduling reduces bubbles without increasing batch size.
Interview Answer Template: "Pipeline parallelism partitions model layers across GPUs. The main challenge is pipeline bubbles — idle time. GPipe uses micro-batching to reduce bubbles from (p-1)/p to (p-1)/(m+p-1). PipeDream's 1F1B schedule reduces memory by interleaving forward and backward passes. Megatron-LM adds interleaved stages that further reduce bubbles by factor v. The rule of thumb is m ≥ 4p for acceptable efficiency, combined with TP within stages for large models."
