TPU Training: Differences from GPU

MLOps Series Training Infrastructure

TPU Architecture

Google's Tensor Processing Units are ASICs designed from the ground up for matrix-heavy deep-learning workloads. Unlike GPUs, which evolved from graphics pipelines, TPUs eliminate virtually all general-purpose circuitry and dedicate transistor area to two things: systolic-array matrix multiply units (MXUs) and high-bandwidth memory (HBM).

Key architectural details across TPU generations:

  • TPU v2: 45 TFLOPS bf16, 8 GB HBM, 2D torus ICI. First version available to external users via Cloud TPU.
  • TPU v3: 123 TFLOPS bf16, 16 GB HBM, liquid-cooled. Doubled the MXU throughput with improved clock speeds.
  • TPU v4: 275 TFLOPS bf16, 32 GB HBM2e, 3D torus ICI with 4,096 chips per pod. Used to train PaLM.
  • TPU v5e: Cost-optimized for inference and medium-scale training, 256-chip pods.
  • TPU v5p: 459 TFLOPS bf16, 95 GB HBM, designed for frontier model training at 8,960 chips per pod.
Interview note: The MXU is a 128×128 systolic array that performs matrix multiplications in bf16 with fp32 accumulation. This is the single most important thing to understand about TPU hardware — the entire software stack (XLA, JAX) is designed to keep the MXU fed.

XLA Compilation

XLA (Accelerated Linear Algebra) is the compiler that transforms high-level tensor operations into optimized TPU machine code. Unlike GPU's eager execution (CUDA kernels launch immediately), XLA uses a trace-compile-execute paradigm.

How XLA Works

  1. Tracing: The framework (JAX or PyTorch/XLA) records operations into an HLO (High-Level Operations) graph instead of executing them immediately.
  2. Optimization: XLA applies fusion, constant folding, dead code elimination, layout assignment, and memory scheduling.
  3. Code Generation: The optimized HLO is lowered to TPU machine code targeting the MXU, vector unit, and memory hierarchy.
  4. Execution: The compiled binary runs on TPU hardware.
# XLA compilation pipeline (conceptual)

# Step 1: Python-level operations are traced, not executed
def training_step(params, batch):
    logits = model_forward(params, batch["input_ids"])  # traced
    loss = cross_entropy(logits, batch["labels"])        # traced
    grads = compute_gradients(loss, params)               # traced
    return loss, grads                                     # graph captured

# Step 2: XLA optimizations on the HLO graph
#   - Op fusion: matmul + bias + relu → single fused kernel
#   - Layout optimization: assign memory layouts for MXU
#   - Padding: shapes padded to multiples of 128 for MXU
#   - Memory scheduling: overlap compute and data movement

# Step 3: Compiled once, executed many times
compiled_step = xla_compile(training_step)
for batch in dataloader:
    loss, grads = compiled_step(params, batch)  # fast execution

Key Optimization Passes

  • Operator Fusion: Combines chains of element-wise ops (add, multiply, activation) into single kernels, eliminating intermediate memory reads/writes.
  • Padding Insertion: Tensor dimensions are padded to multiples of 8 or 128 to align with the MXU's systolic array. A dimension of 127 becomes 128; a dimension of 129 becomes 256.
  • Layout Assignment: Decides row-major vs. column-major ordering to maximize MXU throughput. Transposes are inserted where needed.
  • Memory Scheduling: Overlaps HBM reads for the next operation with current MXU computation (double-buffering).
Recompilation trap: XLA caches compiled programs keyed by input shapes. If your input shapes change every step (e.g., variable-length sequences without padding), XLA recompiles each time — destroying performance. Always pad to fixed bucket sizes.

TPU vs GPU

The choice between TPU and GPU is nuanced. TPUs are not strictly "better" — they excel at specific workload profiles while GPUs offer broader flexibility.

🟢 TPU Strengths

  • Dense matmul throughput: 128×128 systolic array delivers exceptional throughput for large, regular matrix operations
  • bf16 native: Brain floating point is the native format — no mixed-precision complexity
  • ICI interconnect: 3D torus with >100 Gbps per link enables tight coupling between chips without host CPU involvement
  • Cost at scale: TPU pods often cheaper per FLOP for large-scale training on GCP
  • JAX ecosystem: First-class support with SPMD, pjit, and mesh abstractions

🔵 GPU Strengths

  • Flexibility: CUDA supports arbitrary kernels — custom ops, sparse operations, variable control flow
  • Ecosystem maturity: PyTorch, CUDA libraries (cuBLAS, cuDNN, NCCL, Triton) are battle-tested
  • Dynamic shapes: Eager execution handles variable-length inputs without recompilation
  • Multi-cloud: Available on AWS, Azure, GCP, Oracle, Lambda Labs — not vendor-locked
  • Sparse & irregular ops: Tensor cores handle diverse workloads beyond dense matmul

When TPUs Win

Large transformer pre-training with fixed sequence lengths, massive batch sizes, regular compute graphs. Models like PaLM (540B), Gemini, and T5 were all trained on TPU pods.

When GPUs Win

Research with frequent iteration, models requiring custom CUDA kernels (FlashAttention was GPU-first), variable-length inputs, reinforcement learning with irregular control flow, or multi-cloud requirements.

Cost heuristic: For a fixed GCP budget on large-scale transformer training, TPU v4 pods often deliver 1.3–1.7× more throughput-per-dollar than equivalent A100 clusters, primarily because ICI eliminates the InfiniBand bottleneck.

PyTorch/XLA

PyTorch/XLA bridges PyTorch's eager API with XLA's compiled execution. The key abstraction is the lazy tensor: operations are recorded into a graph and only executed when results are needed (at mark_step() boundaries or when a value is explicitly requested).

import torch
import torch_xla.core.xla_model as xm
import torch_xla.distributed.parallel_loader as pl
import torch_xla.distributed.xla_multiprocessing as xmp

def train_loop(index):
    device = xm.xla_device()              # get TPU core device
    model = MyModel().to(device)           # move model to TPU
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

    train_loader = pl.MpDeviceLoader(dataloader, device)

    for epoch in range(num_epochs):
        for batch in train_loader:
            optimizer.zero_grad()
            loss = model(batch)               # lazy: builds graph
            loss.backward()                    # lazy: extends graph
            xm.optimizer_step(optimizer)     # calls mark_step()
            # ↑ mark_step triggers XLA compilation + execution

# Launch on all TPU cores
xmp.spawn(train_loop, args=(), nprocs=8)  # 8 cores per TPU v4 host

Common Pitfalls

  • Calling .item() or print(tensor): Forces immediate graph execution and blocks the pipeline. Never do this in the hot loop — accumulate values and print at step boundaries.
  • Dynamic tensor shapes: Every unique shape triggers recompilation. Pad sequences to fixed bucket sizes (e.g., 128, 256, 512, 1024).
  • Python control flow on tensor values: if loss > threshold forces evaluation. Use torch.where() instead.
  • Missing mark_step(): Without it, the graph grows unboundedly and eventually OOMs. Use xm.optimizer_step() which calls it automatically.
  • CPU-TPU transfer: Minimize host-device transfers. Use MpDeviceLoader to prefetch batches to TPU asynchronously.
Debugging tip: Set XLA_IR_DEBUG=1 and XLA_HLO_DEBUG=1 to see the computation graph. Use xm.get_metrics_report() to check for unexpected recompilations — look for CompileTime and ExecuteTime counters.

JAX on TPU

JAX was designed TPU-first. Its function-transformation model (jit, grad, vmap, pmap) maps naturally onto XLA's trace-compile-execute paradigm. There is no eager mode to tempt you — everything is compiled by default.

import jax
import jax.numpy as jnp
from jax import jit, grad, vmap, pmap
from jax.sharding import Mesh, PartitionSpec as P, NamedSharding

# Define model + loss as pure functions
def loss_fn(params, batch):
    logits = model_apply(params, batch["input_ids"])
    return cross_entropy(logits, batch["labels"])

# Single-device: jit + grad
@jit
def train_step(params, opt_state, batch):
    loss, grads = jax.value_and_grad(loss_fn)(params, batch)
    updates, opt_state = optimizer.update(grads, opt_state, params)
    params = optax.apply_updates(params, updates)
    return params, opt_state, loss

# Multi-device with pmap (data parallelism)
@pmap
def train_step_dp(params, opt_state, batch):
    loss, grads = jax.value_and_grad(loss_fn)(params, batch)
    grads = jax.lax.pmean(grads, axis_name="devices")  # all-reduce
    updates, opt_state = optimizer.update(grads, opt_state, params)
    params = optax.apply_updates(params, updates)
    return params, opt_state, loss

# Advanced: SPMD with named sharding (TPU pod)
devices = jax.devices()  # all TPU cores
mesh = Mesh(devices, axis_names=("data", "model"))

def shard_params(params):
    # Shard large matrices across model axis, replicate biases
    return jax.tree_map(
        lambda p: jax.device_put(p, NamedSharding(mesh,
            P("model", None) if p.ndim == 2 else P())),
        params
    )

JAX Transformations Explained

  • jit: Traces the function, compiles via XLA, caches the compiled artifact. Subsequent calls with the same input shapes skip compilation.
  • grad: Automatic differentiation via source transformation. Composes with jit — gradients are computed inside the same XLA program.
  • vmap: Vectorizing map — automatically batches a function written for single examples. XLA fuses the batch dimension into efficient batched kernels.
  • pmap: Parallel map across devices. Each device executes the same function on a different data shard. Collectives like pmean and psum handle cross-device communication.
SPMD vs pmap: Modern JAX code prefers jax.jit with named sharding over pmap. SPMD lets the XLA compiler decide communication patterns (e.g., all-gather, reduce-scatter) rather than requiring explicit collective calls. This becomes essential for model parallelism on TPU pods.

TPU Pods

A TPU pod is a cluster of TPU chips connected via the Inter-Chip Interconnect (ICI) — a high-bandwidth, low-latency network that operates independently of the host CPU network. This is the fundamental advantage of TPU pods over GPU clusters: collective operations happen at hardware speed without going through host NICs.

Pod Topology

  • TPU v4 pod: 4,096 chips arranged in a 3D torus (e.g., 16×16×16). Each chip has 6 ICI links (±x, ±y, ±z) at ~100 Gbps each.
  • Pod slice: A contiguous sub-section of the full pod. You can request slices of 8, 16, 32, ... up to 4,096 chips.
  • Multi-host: Each host machine connects to 4 TPU chips. A 256-chip slice has 64 host VMs. Your training script runs on each host, and ICI handles inter-chip communication.

Parallelism Strategies on Pods

  • Data parallelism: Each chip processes a different micro-batch. All-reduce over ICI aggregates gradients. Simple and effective up to thousands of chips.
  • Model parallelism (tensor): Large weight matrices are sharded across chips. The MXU on each chip computes a partial matmul, and ICI all-gathers the results. GSPMD automates this.
  • Pipeline parallelism: Different layers reside on different groups of chips. Micro-batches are pipelined across stages. Less common on TPUs because ICI is fast enough for tensor parallelism.

Debugging TPU Training

TPU debugging differs fundamentally from GPU debugging because of lazy execution. You cannot simply print tensor values — by the time print() runs, the tensor might not have been computed yet.

Profiling with TensorBoard

import torch_xla.debug.profiler as xp

# Start profiler server
server = xp.start_server(9012)

# Profile a training step
with xp.Trace("train_step"):
    loss = model(batch)
    loss.backward()
    xm.optimizer_step(optimizer)

# Then capture with: tensorboard --logdir . --port 6006
# Navigate to "Profile" tab → Trace Viewer
# Look for: MXU utilization, memory timeline, compilation events

Avoiding Recompilation

  • Fixed shapes: Pad all inputs to a set of predetermined bucket sizes. For NLP: [128, 256, 512, 1024, 2048].
  • Avoid Python conditionals on tensor values: if tensor.item() > 0 forces evaluation and creates a new graph branch.
  • Monitor compilations: Check xm.get_metrics_report() for CompileTime — if it's growing over time, you have a recompilation issue.
  • Use mark_step() at regular intervals: Once per training step is ideal. More frequent calls mean smaller graphs (faster compile, more overhead). Less frequent means larger graphs (longer compile, less overhead).

Dynamic Shape Handling

# BAD: variable sequence length causes recompilation every step
for batch in dataloader:
    tokens = batch["input_ids"]  # shape varies: (B, 73), (B, 159), ...
    loss = model(tokens)          # XLA recompiles for each new shape!

# GOOD: pad to fixed buckets
BUCKETS = [128, 256, 512, 1024]

def pad_to_bucket(tokens):
    seq_len = tokens.shape[1]
    target = next(b for b in BUCKETS if b >= seq_len)
    pad_len = target - seq_len
    return F.pad(tokens, (0, pad_len), value=0)

for batch in dataloader:
    tokens = pad_to_bucket(batch["input_ids"])  # now limited shapes
    loss = model(tokens)                          # XLA reuses compiled cache
XLA metrics cheatsheet: CompileTime — total time spent compiling (should plateau). ExecuteTime — time running compiled programs. TransferToDeviceTime — host→TPU data transfer. InboundData / OutboundData — bytes transferred. If CompileTime keeps growing, you have a recompilation bug.

Common Debugging Environment Variables

  • XLA_IR_DEBUG=1 — Print the intermediate representation graph.
  • XLA_HLO_DEBUG=1 — Print the HLO graph sent to XLA compiler.
  • PT_XLA_DEBUG=1 — Enable PyTorch/XLA debug logging for graph compilation events.
  • TPU_MIN_LOG_LEVEL=0 — Verbose TPU runtime logs.
  • XLA_FLAGS="--xla_dump_to=/path" — Dump HLO graphs to disk for offline analysis.
