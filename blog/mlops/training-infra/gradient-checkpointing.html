Gradient Checkpointing: Compute vs Memory

MLOps Series Training Infrastructure

The Memory Problem

During the forward pass, every intermediate activation must be stored for use in the backward pass. For a model with n layers, each producing activations of size a, total activation memory is O(n × a). This quickly dominates GPU memory for deep models.

Memory breakdown for a transformer layer (hidden dim h, sequence length s, batch size b):

  • Attention Q, K, V projections: 3 × b × s × h (often stored as 3 separate tensors)
  • Attention scores: b × num_heads × s × s (quadratic in sequence length!)
  • Attention output: b × s × h
  • FFN intermediate: b × s × 4h (common expansion ratio)
  • Layer norm inputs: 2 × b × s × h

For a 7B parameter model with 32 layers, batch size 8, sequence length 2048, and hidden dim 4096 in FP16: activation memory can reach ~60 GB — well beyond a single 80 GB A100's capacity after accounting for model weights (14 GB) and optimizer states (28 GB with Adam).

Key insight: Model parameters and optimizer states scale with model size. Activations scale with model size × batch size × sequence length. For LLMs with long sequences, activations are almost always the memory bottleneck.

How Checkpointing Works

Gradient checkpointing (also called activation recomputation or rematerialization) trades compute for memory: instead of storing all activations, we save only a subset and recompute the rest during backward.

The process: (1) During forward, only save activations at checkpoint boundaries. (2) During backward, when gradients need non-checkpointed activations, re-run the forward pass from the nearest checkpoint to recompute them. (3) Free recomputed activations after use.

Terminology note: "Gradient checkpointing" is the PyTorch/HuggingFace term. Megatron-LM calls it "activation recomputation." DeepSpeed uses "activation checkpointing." They all refer to the same technique.

Memory Savings Formula

For a model with n layers, the optimal checkpoint placement uses √n evenly-spaced checkpoints, achieving O(√n) memory instead of O(n).

Derivation: Suppose we place k checkpoints, dividing the model into k segments of n/k layers each. We store k checkpoint activations plus the activations for one segment being recomputed during backward:

Memory = k + n/k

Minimizing with respect to k (d/dk = 1 − n/k² = 0): the minimum occurs at k = √n, giving memory = 2√n.

Memory Comparison

  • n = 24 layers (GPT-2 medium): Standard: 24 → Checkpointed: ~10 (2.4× savings)
  • n = 48 layers (GPT-3 6.7B): Standard: 48 → Checkpointed: ~14 (3.4× savings)
  • n = 96 layers (GPT-3 175B): Standard: 96 → Checkpointed: ~20 (4.8× savings)

Compute Overhead

  • √n strategy: One extra forward pass per segment during backward
  • Worst case: ~33% compute overhead (one full extra forward pass)
  • Typical: 20–30% overhead in practice due to caching and overlap

Full recomputation is the extreme case where only the model input is checkpointed: all layers are recomputed during backward, giving O(1) activation memory but doubling compute (100% overhead). This is what gradient_checkpointing_enable() in HuggingFace does by default for every layer.

Segment Checkpointing

Rather than checkpointing every individual layer, segment checkpointing groups layers into contiguous segments. The entire segment is treated as a single checkpointed function: only the segment input is saved, and all internal activations are recomputed during backward.

Choosing segment boundaries: In transformer models, natural segment boundaries are at transformer block (layer) boundaries, since these have the smallest activation tensors (just the hidden states). Placing checkpoints mid-layer would require storing intermediate attention scores or FFN activations, which defeats the purpose.

For a 32-layer transformer with √32 ≈ 6 segments of ~5 layers each:

  • Checkpoint activations saved at layers: 0, 5, 10, 15, 20, 25
  • During backward for layers 20–24, recompute forward from checkpoint at layer 20
  • Peak memory: 6 checkpoint activations + 5 recomputed activations for current segment
Megatron-LM approach: Megatron uses two recomputation strategies: "full" (recompute everything per layer — stores nothing) and "selective" (only recompute the attention softmax and dropout, which are the most memory-heavy per-layer activations). Selective recomputation gives ~70% of the memory savings for ~5% compute overhead.

PyTorch Implementation

PyTorch provides torch.utils.checkpoint.checkpoint for wrapping individual functions and checkpoint_sequential for sequential models.

# Per-layer checkpointing in a transformer
import torch
from torch.utils.checkpoint import checkpoint

class CheckpointedTransformer(torch.nn.Module):
    def __init__(self, num_layers, hidden_dim, num_heads):
        super().__init__()
        self.layers = torch.nn.ModuleList([
            TransformerBlock(hidden_dim, num_heads)
            for _ in range(num_layers)
        ])
        self.use_checkpoint = True

    def forward(self, x):
        for layer in self.layers:
            if self.use_checkpoint and self.training:
                # use_reentrant=False is the modern recommended API
                x = checkpoint(layer, x, use_reentrant=False)
            else:
                x = layer(x)
        return x
use_reentrant=False: Always set this flag. The legacy use_reentrant=True (default in older PyTorch) has subtle bugs: it doesn't work with torch.autograd.detect_anomaly(), doesn't support non-tensor inputs, and has issues with torch.compile. PyTorch 2.x recommends use_reentrant=False.

For sequential models, checkpoint_sequential is more convenient:

# Sequential checkpointing — divide into segments
from torch.utils.checkpoint import checkpoint_sequential

class SequentialTransformer(torch.nn.Module):
    def __init__(self, num_layers, hidden_dim, num_heads):
        super().__init__()
        self.layers = torch.nn.Sequential(*[
            TransformerBlock(hidden_dim, num_heads)
            for _ in range(num_layers)
        ])
        self.num_segments = int(num_layers ** 0.5)  # √n segments

    def forward(self, x):
        if self.training:
            return checkpoint_sequential(
                self.layers,
                segments=self.num_segments,
                input=x,
                use_reentrant=False
            )
        return self.layers(x)

HuggingFace integration: For pre-built models, simply call model.gradient_checkpointing_enable(). This wraps each transformer layer with checkpoint automatically. Pass gradient_checkpointing_kwargs={"use_reentrant": False} for the modern API.

Selective Checkpointing

Not all activations are equal. Selective checkpointing recomputes only the most memory-expensive operations while keeping cheaper ones stored. This provides a better Pareto trade-off than uniform checkpointing.

Cost-benefit analysis for a transformer layer:

Expensive to Store (Checkpoint These)

  • Attention scores: O(b × h × s²) — quadratic in seq_len
  • Softmax output: Same size as attention scores
  • Dropout masks: Same size as their input tensor
  • FFN intermediate: O(b × s × 4d) — 4× hidden dim

Cheap to Store (Keep These)

  • Layer norm output: O(b × s × d) — small relative to above
  • Residual connections: O(b × s × d) — must keep for addition
  • Q, K, V projections: O(b × s × d) — cheap to store, expensive to recompute
# Selective checkpointing: only recompute attention core
class SelectiveCheckpointAttention(torch.nn.Module):
    def forward(self, hidden_states):
        # Q, K, V projections: STORE these (cheap to store, expensive to recompute)
        q = self.q_proj(hidden_states)
        k = self.k_proj(hidden_states)
        v = self.v_proj(hidden_states)

        # Attention core: RECOMPUTE this (expensive to store due to O(s²))
        if self.training and self.use_selective_checkpoint:
            attn_out = checkpoint(self._attention_core, q, k, v,
                                  use_reentrant=False)
        else:
            attn_out = self._attention_core(q, k, v)

        return self.out_proj(attn_out)

    def _attention_core(self, q, k, v):
        # This is the O(s²) memory operation
        scores = torch.matmul(q, k.transpose(-2, -1)) / self.scale
        attn_weights = torch.nn.functional.softmax(scores, dim=-1)
        attn_weights = self.dropout(attn_weights)
        return torch.matmul(attn_weights, v)
FlashAttention interaction: FlashAttention already avoids materializing the full O(s²) attention matrix, making selective checkpointing of the attention core redundant. When using FlashAttention, checkpoint the FFN instead — it becomes the largest activation consumer at O(b × s × 4d).

Performance Impact

Gradient checkpointing introduces a fundamental compute-memory trade-off. Understanding the actual overhead helps make informed decisions about when and where to apply it.

Theoretical overhead: Full recomputation (every layer checkpointed) performs one additional forward pass during backward, increasing total compute by ~33% (forward pass is roughly 1/3 of forward + backward). In practice, the overhead varies:

  • 20–25% for typical transformer training with selective checkpointing
  • 30–35% for full per-layer checkpointing
  • 50–60% in rare worst-case scenarios with very deep models and no overlap

Why the overhead is often less than 33%: (1) The backward pass itself is typically 2× the cost of forward (due to gradient computation), so the extra forward is 1/(1+2) = 33%. (2) Recomputed activations are often already warm in GPU L2 cache. (3) Recomputation can overlap with gradient communication in distributed training.

When to Use Checkpointing

  • Training doesn't fit in memory at desired batch size
  • Long sequence lengths (where attention activations dominate)
  • Large models (50+ layers)
  • Prefer over reducing batch size (larger batches → better convergence)

When to Avoid

  • Model already fits comfortably (no memory pressure)
  • Throughput is the primary concern (inference, fast iteration)
  • Using FlashAttention already eliminated the biggest activation hog
  • Can scale to more GPUs instead (pure parallelism is faster)
Optimal strategy for LLM training: Use selective checkpointing (attention core or FFN) + FlashAttention + gradient accumulation together. This combination typically allows 2–4× larger batch sizes per GPU with only 10–15% throughput reduction — a far better trade-off than reducing batch size or using more GPUs.

Profiling tip: Use torch.cuda.memory_stats() to measure peak activation memory with and without checkpointing. The ratio directly quantifies your memory savings. Compare allocated_bytes.all.peak across configurations.
