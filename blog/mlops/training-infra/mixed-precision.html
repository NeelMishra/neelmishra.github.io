Mixed Precision: FP16, BF16, FP8

MLOps Series Training Infrastructure

Number Formats Explained

Every floating-point number is stored as three fields: a sign bit, an exponent (dynamic range), and a mantissa (precision). The trade-off between exponent and mantissa bits defines each format's character.

Key properties — FP32 is the baseline with 8 exponent and 23 mantissa bits. FP16 trades range for memory (max representable value ≈ 65,504). BF16 keeps the FP32 exponent range (max ≈ 3.39 × 10³⁸) but sacrifices precision. TF32 is an NVIDIA Ampere internal format used in tensor cores. FP8 variants push the frontier: E4M3 for forward activations (more precision), E5M2 for gradients (more range).

Interview tip: Knowing the exact bit layout is critical. BF16's 8-bit exponent matching FP32 is why it rarely needs loss scaling — it can represent the same dynamic range, just with less precision per value.

Why Mixed Precision

Mixed precision training stores and computes certain operations in half-precision (16-bit) while keeping master weights and critical accumulations in FP32. The benefits are threefold:

  • 2× memory reduction — Activations and gradients in FP16/BF16 halve memory consumption, enabling larger batch sizes or models.
  • Tensor core utilization — NVIDIA tensor cores perform FP16/BF16 matrix multiplications at 2–8× the throughput of FP32 CUDA cores.
  • Bandwidth savings — Halving data width doubles effective memory bandwidth, often the true bottleneck in transformer training.

The "mixed" in mixed precision refers to maintaining a master copy of weights in FP32. Gradients and activations flow in reduced precision, but the optimizer applies updates to the FP32 master weights. This prevents the accumulation of small updates from being lost due to limited mantissa bits.

Common misconception: Mixed precision does NOT mean "train everything in FP16." Without FP32 master weights, small gradient updates (e.g., 1e-7) vanish entirely in FP16, causing training to stall or diverge.

Throughput gains on modern hardware — A100 tensor cores deliver 312 TFLOPS in TF32 vs 156 TFLOPS in FP32, and 624 TFLOPS in FP16/BF16. H100 pushes FP8 to nearly 2000 TFLOPS.

Loss Scaling

FP16 has a limited dynamic range with a minimum positive normal value of ~6.1 × 10⁻⁵. Many gradient values during backpropagation fall below this threshold and underflow to zero. Loss scaling shifts gradients into the representable range.

The idea is simple: multiply the loss by a large constant S before backward(), which scales all gradients by S, then unscale (divide by S) before the optimizer step.

# Static loss scaling — fixed scale factor
def train_step_static(model, data, target, optimizer, loss_fn, scale=1024):
    optimizer.zero_grad()
    output = model(data)
    loss = loss_fn(output, target)

    # Scale loss before backward to prevent gradient underflow
    scaled_loss = loss * scale
    scaled_loss.backward()

    # Unscale gradients before optimizer step
    for param in model.parameters():
        if param.grad is not None:
            param.grad.data /= scale

    optimizer.step()
    return loss.item()

Dynamic loss scaling attempts to find the maximum usable scale factor automatically. PyTorch's GradScaler starts with a large scale (e.g., 2¹⁶) and halves it whenever an inf/nan gradient is detected. It doubles the scale periodically when training is stable.

# Dynamic loss scaling with GradScaler
from torch.cuda.amp import GradScaler, autocast

scaler = GradScaler(init_scale=2**16, growth_interval=2000)

for data, target in dataloader:
    optimizer.zero_grad()
    with autocast(dtype=torch.float16):
        output = model(data)
        loss = loss_fn(output, target)

    scaler.scale(loss).backward()    # Scaled backward
    scaler.unscale_(optimizer)        # Unscale for clipping
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    scaler.step(optimizer)            # Skips step if inf/nan
    scaler.update()                   # Adjust scale factor
Key detail: When scaler.step() detects inf/nan gradients, it skips the optimizer step entirely and reduces the scale factor. This means the effective learning rate can be slightly impacted — monitor scaler.get_scale() to ensure it's not collapsing.

BF16 vs FP16

Both formats use 16 bits, but their trade-offs are fundamentally different. This choice is one of the most frequently asked questions in ML infrastructure interviews.

FP16 (IEEE 754 Half)

  • Exponent: 5 bits → max ≈ 65,504
  • Mantissa: 10 bits → higher precision
  • Loss scaling: Required to avoid underflow
  • Hardware: Volta+ tensor cores (V100, A100, H100)
  • Use case: Inference, legacy training pipelines
  • Risk: Overflow in activations, underflow in gradients

BF16 (Brain Float)

  • Exponent: 8 bits → max ≈ 3.39 × 10³⁸ (same as FP32)
  • Mantissa: 7 bits → lower precision
  • Loss scaling: Rarely needed
  • Hardware: Ampere+ tensor cores (A100, H100)
  • Use case: LLM pretraining, modern training
  • Risk: Precision loss in small differences (e.g., softmax)

When to choose BF16: For LLM training and any new project on Ampere+ hardware, BF16 is the default choice. Its FP32-matching exponent range eliminates the need for loss scaling entirely, simplifying the training loop and reducing the chance of numerical instability.

When FP16 still wins: FP16 has more precision per value (10 vs 7 mantissa bits), so tasks sensitive to small numerical differences — like certain GAN training or fine-grained regression — may benefit. Also, FP16 is available on older hardware (V100).

Pitfall: BF16 can cause issues in operations that depend on precise small differences. A common example: computing softmax attention scores where the difference between logits is small. The solution is to keep softmax computation in FP32 via autocast rules.

torch.autocast

PyTorch's autocast context manager automatically selects which operations run in reduced precision. This is the recommended API for mixed-precision training as of PyTorch 2.x.

Operations are categorized into three lists:

  • FP16/BF16-safe — matmul, conv, linear, BMM → cast to reduced precision
  • FP32-required — softmax, layer norm, cross-entropy loss, log, exp → cast to FP32
  • Promote — add, cat → run in the higher precision of their inputs
# Modern mixed-precision training loop (PyTorch 2.x)
import torch
from torch.amp import autocast, GradScaler

model = MyTransformer().cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)
scaler = GradScaler("cuda")        # Only needed for FP16

for epoch in range(num_epochs):
    for batch in dataloader:
        inputs, labels = batch["input_ids"].cuda(), batch["labels"].cuda()
        optimizer.zero_grad(set_to_none=True)  # More memory-efficient

        with autocast(device_type="cuda", dtype=torch.bfloat16):
            logits = model(inputs)
            loss = cross_entropy(logits, labels)  # Runs in FP32 automatically

        # For BF16: no scaler needed, just backward + step
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()

Master weights: The optimizer always stores and updates FP32 copies. When using autocast, the model parameters remain in FP32 in memory — autocast creates temporary lower-precision copies for the forward pass. This is distinct from model-level casting (model.half()), which actually changes parameter storage.

Performance note: Use set_to_none=True in zero_grad() to avoid an unnecessary memset to zero. This replaces gradients with None, saving memory bandwidth and a small allocation overhead.

FP8 Training

FP8 (8-bit floating point) represents the next frontier in mixed precision, enabled by NVIDIA's H100 (Hopper) architecture and the Transformer Engine library. FP8 delivers up to 2× throughput over FP16/BF16 on tensor cores.

Two FP8 formats serve different roles:

  • E4M3 (4 exponent, 3 mantissa) — Used for forward-pass activations and weights. More precision for representing model parameters.
  • E5M2 (5 exponent, 2 mantissa) — Used for gradients during backward pass. More dynamic range to handle the wider gradient distribution.

Per-tensor scaling is essential for FP8. Unlike FP16 where a single loss-scale factor works globally, FP8's limited range (max 448 for E4M3) requires each tensor to have its own scale factor to maximize the representable range. The Transformer Engine maintains these scales automatically.

Delayed scaling is a key optimization: instead of computing the scale factor from the current tensor (requiring an extra pass), NVIDIDA's approach uses the amax history — the maximum absolute value from the previous several iterations — to set the scale. This avoids extra kernel launches and works well because tensor statistics change slowly during training.

# FP8 training with Transformer Engine
import transformer_engine.pytorch as te

# Replace standard layers with TE equivalents
model = te.TransformerLayer(
    hidden_size=4096,
    ffn_hidden_size=16384,
    num_attention_heads=32,
    # TE handles FP8 casting internally
)

# Enable FP8 training context
with te.fp8_autocast(enabled=True, fp8_recipe=te.recipe.DelayedScaling(
    margin=0,
    interval=1,
    fp8_format=te.recipe.Format.HYBRID,  # E4M3 fwd, E5M2 bwd
    amax_history_len=16,
    amax_compute_algo="max",
)):
    output = model(input_tensor)
    loss = loss_fn(output, labels)
    loss.backward()
H100 FP8 performance: H100 SXM delivers ~1,979 TFLOPS in FP8, nearly 2× the ~989 TFLOPS in FP16/BF16. This makes FP8 the highest-throughput format for training on Hopper GPUs.

Best Practices

Mixed precision is powerful but introduces subtle failure modes. Here is a systematic guide to avoiding them:

1. Always monitor loss scale. If using FP16 + GradScaler, log scaler.get_scale(). A continuously decreasing scale factor indicates persistent numerical instability — your gradients are frequently producing inf/nan values.

2. Keep critical ops in FP32. Even with autocast, verify that loss computation, softmax, and normalization layers run in FP32. You can check with torch.autocast debug mode or manually inspecting tensor dtypes during a forward pass.

3. Gradient clipping before unscaling. When using GradScaler, call scaler.unscale_(optimizer) before clip_grad_norm_. If you clip scaled gradients, your effective max norm is scale-dependent and unstable.

4. Watch for overflow in reductions. Operations that sum many FP16 values (e.g., large batch mean, global norm) can overflow. Use FP32 for reduction operations even if inputs are FP16.

5. Validate with FP32 baseline. Always establish an FP32 training baseline. If mixed precision diverges where FP32 converges, the issue is numerical — not a bug in your model architecture.

Debugging checklist: Training loss goes to NaN? (1) Check if loss scale collapsed to 1 or 0. (2) Look for overflow in attention logits — common with large sequence lengths. (3) Verify all layer norms are in FP32. (4) Try BF16 instead of FP16 to rule out range issues. (5) Enable torch.autograd.detect_anomaly() to find the exact operation.

6. FP8 readiness. Before moving to FP8, ensure your model is stable in BF16. FP8 amplifies any existing numerical issues. Start with FP8 for inference (simpler), then move to training once you've validated convergence matches BF16.

Format Selection Cheat Sheet

  • V100: FP16 + GradScaler (only option)
  • A100: BF16 preferred, TF32 for matmul
  • H100: BF16 for stability, FP8 for throughput

Memory Impact (1B param model)

  • FP32 everywhere: ~4 GB weights + ~16 GB activations
  • Mixed FP16: ~2 GB weights + ~8 GB activations + 4 GB master
  • Mixed FP8: ~1 GB weights + ~4 GB activations + 4 GB master
