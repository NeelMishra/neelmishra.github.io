MLOps Series Training Infrastructure

GPU Scheduling Overview

A single A100 GPU costs $1–2/hour in the cloud and $15,000+ to buy. A 1,000-GPU cluster represents millions of dollars in hardware. The scheduler's job is to maximize the useful utilization of this hardware while providing fair access to multiple teams and projects.

In practice, most GPU clusters operate at 40–60% utilization without good scheduling. The gap comes from fragmentation (jobs requesting specific GPU counts that leave remainders), queuing delays, and idle GPUs during checkpointing or data loading phases.

Key scheduling objectives, often in tension:

  • Throughput: Maximize total GPU-hours of useful work per day
  • Fairness: Give each team their proportional share of resources
  • Latency: Minimize wait time from job submission to start
  • Priority: Urgent production training gets resources before exploratory experiments
  • Efficiency: Avoid GPU fragmentation and idle resources
Interview Context: "How would you schedule 50 teams sharing a 2,000-GPU cluster?" is a common ML infra interview question. The answer involves fair-share scheduling, priority queues, preemption, and gang scheduling for distributed jobs.

Slurm for ML

Slurm (Simple Linux Utility for Resource Management) is the dominant scheduler for HPC and ML clusters, used by most academic labs and many industry clusters (Meta, NVIDIA, etc.).

#!/bin/bash
#SBATCH --job-name=llm-pretrain
#SBATCH --partition=gpu
#SBATCH --nodes=8
#SBATCH --ntasks-per-node=1
#SBATCH --gres=gpu:a100:8          # 8 GPUs per node
#SBATCH --cpus-per-task=96          # all CPUs for data loading
#SBATCH --mem=0                     # all memory on node
#SBATCH --time=72:00:00             # 3-day wall-clock limit
#SBATCH --output=logs/%j.out
#SBATCH --error=logs/%j.err
#SBATCH --exclusive                 # no sharing the node

# Set up distributed training environment
export MASTER_ADDR=$(scontrol show hostnames $SLURM_JOB_NODELIST | head -n 1)
export MASTER_PORT=29500
export WORLD_SIZE=$(($SLURM_NNODES * 8))

# Launch with srun (one task per node, torchrun handles per-GPU processes)
srun torchrun \
    --nnodes=$SLURM_NNODES \
    --nproc_per_node=8 \
    --rdzv_backend=c10d \
    --rdzv_endpoint=$MASTER_ADDR:$MASTER_PORT \
    train.py \
    --config configs/7b.yaml \
    --checkpoint_dir /shared_fs/checkpoints/$SLURM_JOB_ID

Job Arrays for Hyperparameter Sweeps

#SBATCH --array=0-19                # 20 jobs
#SBATCH --gres=gpu:a100:1           # 1 GPU each

# Each array task picks a different hyperparameter
LR_VALUES=(1e-4 2e-4 5e-4 1e-3 2e-3)
BS_VALUES=(32 64 128 256)

LR_IDX=$(($SLURM_ARRAY_TASK_ID / 4))
BS_IDX=$(($SLURM_ARRAY_TASK_ID % 4))

python train.py \
    --lr ${LR_VALUES[$LR_IDX]} \
    --batch_size ${BS_VALUES[$BS_IDX]}

Job Dependencies

# Chain jobs: preprocess → train → evaluate
JOB1=$(sbatch preprocess.sh | awk '{print $4}')
JOB2=$(sbatch --dependency=afterok:$JOB1 train.sh | awk '{print $4}')
sbatch --dependency=afterok:$JOB2 evaluate.sh
Slurm Pro Tip: Use --exclusive for distributed training to prevent other jobs from landing on your nodes and competing for GPU memory/PCIe bandwidth. Use --mem=0 to claim all node memory.

Kubernetes GPU Scheduling

Kubernetes with the NVIDIA device plugin enables GPU scheduling in containerized environments. This is the standard for cloud-native ML platforms (GKE, EKS, AKS).

# Pod spec requesting GPUs
apiVersion: v1
kind: Pod
metadata:
  name: training-worker
spec:
  containers:
  - name: trainer
    image: nvcr.io/nvidia/pytorch:24.03-py3
    resources:
      requests:
        nvidia.com/gpu: 8       # request 8 GPUs
        memory: "512Gi"
        cpu: "96"
      limits:
        nvidia.com/gpu: 8       # must match requests for GPUs
    volumeMounts:
    - name: shared-data
      mountPath: /data
    - name: checkpoints
      mountPath: /checkpoints
  nodeSelector:
    accelerator: nvidia-a100
    topology: nvlink            # ensure NVLink connectivity
  tolerations:
  - key: nvidia.com/gpu
    operator: Exists
    effect: NoSchedule

GPU Sharing Strategies

Exclusive (Default)

  • One GPU per container
  • Full GPU memory available
  • Simple, no contention
  • Best for training workloads

MIG (Multi-Instance GPU)

  • A100/H100 partitioned into slices
  • Hardware-level isolation
  • Up to 7 instances per GPU
  • Best for inference, small experiments

Time-Slicing

  • NVIDIA device plugin time-shares GPUs
  • No memory isolation
  • Risk of OOM between containers
  • Best for development/debugging

MPS (Multi-Process Service)

  • Concurrent kernel execution
  • Shared GPU context
  • Good for small-batch inference
  • Not recommended for training

Fair-Share Policies

In a multi-tenant cluster, fair-share scheduling ensures each team gets their proportional allocation while allowing opportunistic use of idle resources.

Quota System

Assign each team a guaranteed quota (minimum GPUs) and a burst limit (maximum GPUs). Teams can burst above their quota when the cluster has spare capacity, but are preempted back to their quota when higher-priority work arrives.

# Slurm fair-share configuration (/etc/slurm/slurm.conf)
PriorityType=priority/multifactor
PriorityWeightFairshare=10000
PriorityWeightAge=1000
PriorityWeightJobSize=500
PriorityDecayHalfLife=14-0     # 14-day decay window
PreemptType=preempt/partition_prio
PreemptMode=REQUEUE

# Account setup for teams
sacctmgr add account team_nlp set FairShare=40    # 40% share
sacctmgr add account team_vision set FairShare=30  # 30% share
sacctmgr add account team_rl set FairShare=20      # 20% share
sacctmgr add account team_intern set FairShare=10   # 10% share

# Set max concurrent GPU limits per team
sacctmgr modify account team_nlp set GrpTRES=gres/gpu=400
sacctmgr modify account team_vision set GrpTRES=gres/gpu=300
Billing Integration: Track GPU-hours per team for internal chargeback. Slurm's sacct provides per-job resource usage. Weight billing by GPU type (A100 hours ≠ V100 hours) and priority tier (preemptible jobs cost less).

Gang Scheduling

Gang scheduling (or coscheduling) ensures all workers of a distributed training job start simultaneously. Without it, partial allocation creates deadlocks: Job A gets 6 of its 8 requested nodes and waits for the last 2, while Job B gets 2 nodes and waits for 6 more—neither can run.

This is the hardest scheduling problem in ML infrastructure. Standard Kubernetes doesn't support gang scheduling natively; you need an extension.

Volcano (K8s)

  • CNCF project for batch scheduling
  • Native gang scheduling with PodGroup
  • Queue management, fair-share
  • Used by many production ML platforms

Kueue (K8s)

  • Official K8s SIG-Scheduling project
  • Job queuing and admission control
  • ResourceFlavor for GPU types
  • ClusterQueue for quotas
# Volcano PodGroup for gang scheduling
apiVersion: scheduling.volcano.sh/v1beta1
kind: PodGroup
metadata:
  name: llm-training-pg
spec:
  minMember: 8               # all 8 pods must be schedulable
  minResources:
    nvidia.com/gpu: 64        # total: 8 pods × 8 GPUs
  queue: high-priority
  priorityClassName: production-training

---
# Pod template referencing the PodGroup
apiVersion: v1
kind: Pod
metadata:
  annotations:
    scheduling.volcano.sh/group-name: llm-training-pg
spec:
  schedulerName: volcano      # use Volcano scheduler
  containers:
  - name: worker
    resources:
      limits:
        nvidia.com/gpu: 8
Deadlock Scenario: Without gang scheduling, two 4-node jobs on an 6-node cluster can deadlock: each gets 3 nodes, neither has enough to run. The scheduler must either (a) run one fully and queue the other, or (b) preempt one to free resources. Gang scheduling enforces (a).

Custom Schedulers

When Slurm or Kubernetes don't meet your needs, organizations build custom scheduling layers. Common reasons:

  • Topology awareness: Schedule jobs to minimize network hops (same rack, same switch, same NVLink domain)
  • Bin packing with fragmentation control: Standard schedulers don't optimize for GPU cluster fragmentation
  • Cost optimization: Dynamic spot/on-demand mixing based on current pricing
  • Multi-cluster federation: Route jobs across multiple clusters or cloud regions
class TopologyAwareScheduler:
    """Schedule distributed jobs minimizing network diameter."""

    def __init__(self, cluster_topology):
        self.topology = cluster_topology  # tree: cluster→rack→node→GPU
        self.allocations = {}

    def schedule(self, job):
        num_nodes = job.num_nodes
        num_gpus_per_node = job.gpus_per_node

        # Strategy: find the tightest subtree that fits the job
        # Prefer: same-switch > same-rack > cross-rack

            free_nodes = [n for n in rack.nodes

            if len(free_nodes) >= num_nodes:
                candidates.append((rack, free_nodes[:num_nodes]))

Cluster Management

Running a GPU cluster is an operational challenge beyond just scheduling. Hardware fails, firmware needs updates, and capacity must be planned months ahead.

Node Health Monitoring

Run continuous health checks on every node: GPU memory tests (ECC errors), NVLink bandwidth tests, InfiniBand connectivity tests, and filesystem access tests. Mark unhealthy nodes as draining so no new jobs land on them.

# Automated GPU health check script (runs on each node via cron)

def gpu_health_check():
    results = []
    for gpu_id in range(get_gpu_count()):
        health = {
            "gpu_id": gpu_id,
            "ecc_errors": get_ecc_errors(gpu_id),
            "memory_test": run_memtest(gpu_id),
            "nvlink_ok": check_nvlink(gpu_id),
            "temp_ok": get_temperature(gpu_id) < 85,
            "pcie_bandwidth": test_pcie(gpu_id),
        }
        results.append(health)

    if any(not r["memory_test"] or r["ecc_errors"] > 0 for r in results):
        drain_node()        # remove from scheduler
        alert_ops_team()    # page on-call
        open_ticket()       # auto-create hardware ticket

    return results

GPU Failure Handling

At scale (1,000+ GPUs), GPU failures are a daily occurrence. Plan for a 1–3% failure rate monthly. Maintain a spare pool (5–10% of cluster) for hot-swap replacement. When a GPU fails mid-job:

  1. The training framework detects the failure (NCCL timeout)
  2. The job checkpoints on surviving workers (if using elastic training)
  3. The scheduler drains the failed node and assigns a replacement
  4. The job restarts from the checkpoint on the new topology

Maintenance Windows

Schedule regular maintenance windows for firmware updates, driver upgrades, and hardware inspections. Use rolling updates: drain one rack at a time, update, verify, then move to the next. Avoid draining the entire cluster simultaneously.

Capacity Planning Rule: For a 1,000-GPU cluster expect ~10–30 GPU failures per month, plus 2–5 node-level failures (PSU, motherboard, NIC). Budget 5–10% spare capacity and 1 FTE for hardware operations per 500 GPUs.

On-Prem Cluster

  • Full control over hardware and network
  • High upfront CapEx ($200K+ per node)
  • Need dedicated ops team
  • Best for: sustained high utilization, custom topology

Cloud GPU Cluster

  • Elastic scaling, no hardware ops
  • Higher per-GPU-hour cost
  • Limited topology control
  • Best for: variable demand, fast ramp-up
