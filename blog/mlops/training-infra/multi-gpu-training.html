Multi-GPU: DataParallel vs DDP

MLOps Series Training Infrastructure

Why Multi-GPU Training

Single-GPU training hits hard limits: memory caps the model size, and compute caps the throughput. An A100 80GB can train ~7B parameters with mixed precision—but modern LLMs range from 7B to 400B+. Even for models that fit on one GPU, training speed matters: halving training time by using 2 GPUs directly translates to faster iteration cycles and lower cloud costs.

Multi-GPU training strategies fall into three categories:

  • Data Parallelism: Replicate model on each GPU, split data across GPUs, synchronize gradients
  • Model Parallelism: Split model layers across GPUs (pipeline or tensor parallelism)
  • Hybrid Parallelism: Combine data + model parallelism (the approach used for 100B+ models)

This post focuses on data parallelism—the most common and often sufficient strategy for models up to ~10B parameters.

Decision Framework: If your model fits on one GPU with mixed precision → use DDP. If it doesn't fit → use FSDP or DeepSpeed ZeRO. If it's 100B+ → you need tensor + pipeline + data parallelism combined.

nn.DataParallel (DP)

nn.DataParallel is PyTorch's simplest multi-GPU wrapper. It requires only one line of code to use—and that simplicity comes with severe performance limitations.

How DP Works: (1) Scatter input batch across GPUs, (2) replicate model to each GPU, (3) forward pass in parallel, (4) gather outputs on GPU 0, (5) compute loss and backward on GPU 0, (6) update parameters on GPU 0 and broadcast.

Critical Limitations:

  • Python GIL: DP uses threads for parallelism. The GIL serializes Python-level operations, limiting true parallelism.
  • GPU 0 bottleneck: All loss computation, backward pass, and parameter updates happen on GPU 0. It uses ~2× the memory of other GPUs.
  • Per-iteration broadcast: Model weights are replicated to all GPUs every forward pass—a massive bandwidth waste.
Verdict: nn.DataParallel is effectively deprecated for serious training. Use DistributedDataParallel instead. DP exists only for quick prototyping with minimal code changes.

DistributedDataParallel (DDP)

DDP launches one process per GPU, each with its own Python interpreter (no GIL!). Each process maintains a full copy of the model. After the backward pass, gradients are synchronized via AllReduce—not gathered on a single GPU.

Why DDP is Superior:

  • No GIL: Each GPU has its own process with its own Python interpreter
  • Balanced memory: Every GPU holds the same data—no GPU 0 memory spike
  • Overlapped communication: Gradient AllReduce begins while backward is still running (bucket-based)
  • NCCL backend: Uses NVIDIA's highly optimized collective communication library

DP vs DDP Comparison

nn.DataParallel (DP)

  • Parallelism: Multi-threaded (GIL-limited)
  • Processes: 1 process, N GPUs
  • Gradient sync: Gather on GPU 0 → scatter
  • Memory balance: GPU 0 uses ~2× memory
  • Code change: 1 line (model = nn.DataParallel(model))
  • Multi-node: ❌ Single machine only
  • Scaling: ~60-70% efficiency at 4 GPUs

DistributedDataParallel (DDP)

  • Parallelism: Multi-process (no GIL)
  • Processes: N processes, N GPUs
  • Gradient sync: AllReduce (bucket-based, overlapped)
  • Memory balance: Equal across all GPUs
  • Code change: ~20 lines (init, sampler, torchrun)
  • Multi-node: ✅ Scales across machines
  • Scaling: ~90-95% efficiency at 8 GPUs
Bottom Line: Always use DDP. The extra 20 lines of setup code give you 30-50% better throughput, balanced memory, and the ability to scale to multiple nodes.

DDP Setup Guide

A complete DDP training script requires four changes from single-GPU code:

import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler

def setup():
    # torchrun sets these env vars automatically
    dist.init_process_group(backend="nccl")
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)
    return local_rank

def cleanup():
    dist.destroy_process_group()

def main():
    local_rank = setup()

    # 1. Create model on local GPU
    model = MyModel().to(local_rank)

    # 2. Wrap with DDP
    model = DDP(model, device_ids=[local_rank])

    # 3. Use DistributedSampler (ensures no data overlap)
    sampler = DistributedSampler(dataset, shuffle=True)
    loader  = DataLoader(dataset, batch_size=64, sampler=sampler)

    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)  # 4. Shuffle per epoch
        for inputs, targets in loader:
            inputs  = inputs.to(local_rank)
            targets = targets.to(local_rank)
            loss = train_step(model, inputs, targets, optimizer)

    cleanup()

Launch command:

# Single node, 4 GPUs
$ torchrun --nproc_per_node=4 train.py

# Multi-node (2 nodes × 4 GPUs)
$ torchrun --nnodes=2 --nproc_per_node=4 \
    --rdzv_id=job1 --rdzv_backend=c10d \
    --rdzv_endpoint=master_ip:29500 train.py
Key Detail: sampler.set_epoch(epoch) is critical—without it, every epoch uses the same data order, harming convergence. The sampler uses the epoch number as a seed for shuffling.

Common Pitfalls

DDP is robust but has sharp edges. These are the issues that consume hours of debugging:

1. Unused Parameters: If any parameter doesn't receive a gradient during backward, DDP hangs waiting for its AllReduce bucket. Fix: DDP(model, find_unused_parameters=True) or restructure the model so all parameters participate in every forward pass.

2. BatchNorm Synchronization: Standard BatchNorm computes stats per-GPU. With small per-GPU batch sizes, stats are noisy. Use torch.nn.SyncBatchNorm.convert_sync_batchnorm(model) to synchronize stats across GPUs.

3. Uneven Dataset Sizes: If the dataset isn't evenly divisible by world_size, some processes get one fewer batch. The slower processes hang at AllReduce. Fix: DistributedSampler(dataset, drop_last=True) or pad the dataset.

4. Saving Checkpoints: Only save on rank 0 to avoid file corruption from concurrent writes:

if dist.get_rank() == 0:
    torch.save({
        "model": model.module.state_dict(),  # .module unwraps DDP
        "optimizer": optimizer.state_dict(),
        "epoch": epoch
    }, "checkpoint.pt")
dist.barrier()  # wait for rank 0 to finish saving

5. Hanging Processes: Usually caused by (a) unused parameters, (b) different control flow across ranks, or (c) one rank crashing silently. Set NCCL_ASYNC_ERROR_HANDLING=1 and use timeout in init_process_group to detect hangs.

Debug Tip: Set NCCL_DEBUG=INFO to see detailed NCCL communication logs. For hanging issues, NCCL_DEBUG=WARN + TORCH_DISTRIBUTED_DEBUG=DETAIL will identify which parameter bucket is stuck.

Scaling Efficiency

Ideal (linear) scaling means 2× GPUs = 2× throughput. In practice, communication overhead reduces this. Measuring and understanding scaling efficiency is critical for cost optimization.

Linear Scaling Rule (Goyal et al., 2017): When you scale from N to kN GPUs (kN× batch size), multiply the learning rate by k. Use a warmup period of 5–10 epochs to stabilize training at the higher learning rate.

# Scaling efficiency measurement
def measure_scaling_efficiency(throughput_1gpu, throughput_Ngpu, N):
    ideal_throughput = throughput_1gpu * N
    efficiency = throughput_Ngpu / ideal_throughput
    return efficiency  # 1.0 = perfect linear scaling

# Example: ResNet-50 on A100s
# 1 GPU:  1,200 img/sec
# 8 GPUs: 9,100 img/sec
# Efficiency = 9100 / (1200 × 8) = 94.8%

Factors affecting efficiency:

  • Compute/communication ratio: Larger models and larger batch sizes amortize communication better
  • Interconnect bandwidth: NVLink (900 GB/s) >> PCIe 5.0 (64 GB/s). PCIe-only nodes see 20-30% lower efficiency
  • Gradient compression: Techniques like PowerSGD reduce communication volume at the cost of slight accuracy loss
Cost Calculation: If 8× GPUs gives you 7.5× throughput (93.7% efficiency), you're paying 8× the GPU cost for 7.5× the speed. The effective cost per sample is 8/7.5 = 1.07× a single GPU—a 7% cost premium for 7.5× faster training. Almost always worth it for production.
