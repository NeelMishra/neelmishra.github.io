PyTorch FSDP: Fully Sharded Data Parallel

MLOps Series Training Infrastructure

FSDP Overview

Fully Sharded Data Parallel (FSDP) is PyTorch's native implementation of parameter sharding for distributed training. It provides functionality equivalent to DeepSpeed ZeRO Stage 3 — full sharding of parameters, gradients, and optimizer states — but as a first-class PyTorch API built on top of torch.distributed.

How FSDP Works

FSDP wraps each module (or group of modules) in an FSDP unit. During forward, FSDP performs an AllGather to reconstruct full parameters, runs the computation, then discards the unsharded parameters. During backward, it AllGathers again, computes gradients, and uses ReduceScatter to distribute gradient shards.

Key Difference from DDP: In DDP, every GPU holds the full model at all times. In FSDP, each GPU only stores its shard — the full model exists temporarily during compute and is immediately discarded. This is semantically identical to ZeRO-3 but implemented natively in PyTorch.

Sharding Strategies

FSDP offers three sharding strategies that trade off memory savings against communication overhead — directly analogous to ZeRO stages.

FULL_SHARD

Equivalent to ZeRO-3. Shards parameters, gradients, and optimizer states. Maximum memory savings.

  • Memory: 16Ψ/N per GPU
  • Communication: AllGather + ReduceScatter per FSDP unit
  • Use when: Model doesn't fit with other strategies

SHARD_GRAD_OP

Equivalent to ZeRO-2. Shards gradients and optimizer states. Parameters stay replicated after forward.

  • Memory: 2Ψ + 14Ψ/N per GPU
  • Communication: ReduceScatter only (no forward AllGather)
  • Use when: Params fit but optimizer doesn't

NO_SHARD

Equivalent to standard DDP. No sharding — all parameters replicated. Useful as a baseline or for small models.

  • Memory: 16Ψ per GPU
  • Communication: AllReduce for gradients only
  • Use when: Model fits easily on each GPU

HYBRID_SHARD

Full sharding within a node, replication across nodes. Reduces cross-node communication while still saving intra-node memory.

  • Memory: Between FULL_SHARD and NO_SHARD
  • Communication: Intra-node AllGather + Inter-node AllReduce
  • Use when: High-bandwidth intra-node, low-bandwidth inter-node
from torch.distributed.fsdp import (
    FullyShardedDataParallel as FSDP,
    ShardingStrategy,
)

# Choose sharding strategy
model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # ZeRO-3 equivalent
    # sharding_strategy=ShardingStrategy.SHARD_GRAD_OP,  # ZeRO-2
    # sharding_strategy=ShardingStrategy.HYBRID_SHARD,   # Hybrid
)

Wrapping Policies

FSDP operates on "FSDP units" — each wrapped module is independently sharded. The wrapping policy determines how the model is partitioned into these units, directly affecting memory efficiency and communication granularity.

Size-Based Auto Wrapping

from torch.distributed.fsdp.wrap import size_based_auto_wrap_policy
import functools

# Wrap any module with more than 100M parameters as its own FSDP unit
auto_wrap_policy = functools.partial(
    size_based_auto_wrap_policy,
    min_num_params=100_000_000   # 100M params threshold
)

model = FSDP(
    model,
    auto_wrap_policy=auto_wrap_policy,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
)

Transformer Auto Wrapping

from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy

# Wrap each transformer block as an FSDP unit
# This is the most common policy for LLM training
auto_wrap_policy = functools.partial(
    transformer_auto_wrap_policy,
    transformer_layer_cls={
        TransformerDecoderLayer,  # your model's block class
    }
)

model = FSDP(
    model,
    auto_wrap_policy=auto_wrap_policy,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
)

# Result: each TransformerDecoderLayer is wrapped independently
# FSDP(model)
#   ├─ FSDP(embed_tokens)
#   ├─ FSDP(layer_0)     ← each block is an FSDP unit
#   ├─ FSDP(layer_1)
#   ├─ ...
#   ├─ FSDP(layer_31)
#   └─ FSDP(lm_head)

Custom Wrapping Policy

from torch.distributed.fsdp.wrap import _module_wrap_policy

# Wrap specific module types
def custom_policy(module, recurse, nonwrapped_numel):
    # Wrap attention and MLP blocks separately for finer granularity
    if recurse:
        return True   # always recurse into children
    # Wrap modules with > 50M params OR specific types
    return (
        nonwrapped_numel >= 50_000_000
        or isinstance(module, (SelfAttention, FeedForward))
    )
Best Practice: Use transformer_auto_wrap_policy for LLMs — it wraps each transformer block, which provides the right balance of memory savings and communication efficiency. Too fine-grained wrapping (e.g., wrapping every linear layer) increases AllGather frequency without benefit.

Mixed Precision with FSDP

FSDP has first-class mixed precision support through the MixedPrecision policy. Unlike basic AMP, FSDP's mixed precision controls the dtype at three levels: parameters, reduction operations, and buffer tensors.

from torch.distributed.fsdp import MixedPrecision
import torch

# BF16 mixed precision (recommended for modern hardware)
bf16_policy = MixedPrecision(
    param_dtype=torch.bfloat16,     # AllGather'd params in BF16
    reduce_dtype=torch.bfloat16,    # ReduceScatter in BF16
    buffer_dtype=torch.bfloat16,    # Buffers (e.g., BatchNorm stats) in BF16
)

# FP16 with FP32 reduction (more stable for some models)
fp16_policy = MixedPrecision(
    param_dtype=torch.float16,      # Compute in FP16
    reduce_dtype=torch.float32,     # Reduce in FP32 (avoids gradient underflow)
    buffer_dtype=torch.float16,
)

model = FSDP(
    model,
    mixed_precision=bf16_policy,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
    auto_wrap_policy=auto_wrap_policy,
)
BF16 vs FP16: BF16 has the same exponent range as FP32 (8 bits) so it doesn't need loss scaling. FP16 has only 5 exponent bits and requires a GradScaler to avoid gradient underflow. With FSDP, prefer BF16 on Ampere+ GPUs (A100, H100) — it simplifies the training loop and avoids loss scaling bugs.

Activation Checkpointing

Activation checkpointing (gradient checkpointing) trades compute for memory by recomputing activations during the backward pass instead of storing them. This is critical for training large models with FSDP since activations often dominate memory for long sequences.

from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    checkpoint_wrapper,
    apply_activation_checkpointing,
    CheckpointImpl,
)

# Method 1: Apply to specific layer types (preferred)
non_reentrant_wrapper = functools.partial(
    checkpoint_wrapper,
    checkpoint_impl=CheckpointImpl.NO_REENTRANT,  # required for FSDP
)

apply_activation_checkpointing(
    model,
    checkpoint_wrapper_fn=non_reentrant_wrapper,
    check_fn=lambda submodule: isinstance(
        submodule, TransformerDecoderLayer
    ),
)

# Method 2: Manual wrapping during model construction
for i, layer in enumerate(model.layers):
    model.layers[i] = checkpoint_wrapper(
        layer,
        checkpoint_impl=CheckpointImpl.NO_REENTRANT,
    )

Memory Savings

# Without activation checkpointing (32-layer model, seq_len=2048, hidden=4096)
# Activations per layer ≈ batch × seq × hidden × ~12 (for attention + MLP)
# Total: 32 × batch × 2048 × 4096 × 12 × 2 bytes ≈ enormous

# With checkpointing: only store input to each checkpointed block
# Memory: 32 × batch × 2048 × 4096 × 2 bytes (just the inputs)
# ~10-12× activation memory reduction
# Cost: ~33% more compute (one extra forward per layer during backward)
Important: Always use CheckpointImpl.NO_REENTRANT with FSDP. The reentrant implementation has known issues with FSDP's parameter management and can cause hangs or incorrect gradients.

Complete Setup Guide

Here's a complete FSDP training script that combines all the components:

import os, functools, torch
import torch.distributed as dist
from torch.distributed.fsdp import (
    FullyShardedDataParallel as FSDP,
    ShardingStrategy,
    MixedPrecision,
    BackwardPrefetch,
    CPUOffload,
)
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy

def setup_fsdp_training(rank, world_size):
    # 1. Initialize process group
    dist.init_process_group("nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)

    # 2. Create model on meta device (avoid OOM during init)
    with torch.device("meta"):
        model = MyTransformer(num_layers=32, hidden=4096, heads=32)

    # 3. Define policies
    wrap_policy = functools.partial(
        transformer_auto_wrap_policy,
        transformer_layer_cls={TransformerBlock}
    )
    mixed_prec = MixedPrecision(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.bfloat16,
        buffer_dtype=torch.bfloat16,
    )

    # 4. Wrap with FSDP
    model = FSDP(
        model,
        sharding_strategy=ShardingStrategy.FULL_SHARD,
        auto_wrap_policy=wrap_policy,
        mixed_precision=mixed_prec,
        backward_prefetch=BackwardPrefetch.BACKWARD_PRE,  # prefetch next unit
        device_id=rank,
        limit_all_gathers=True,          # limit concurrent AllGathers
        use_orig_params=True,             # needed for torch.compile
    )

    # 5. Apply activation checkpointing
    apply_activation_checkpointing(
        model,
        checkpoint_wrapper_fn=functools.partial(
            checkpoint_wrapper,
            checkpoint_impl=CheckpointImpl.NO_REENTRANT,
        ),
        check_fn=lambda m: isinstance(m, TransformerBlock),
    )

    # 6. Optimizer (after FSDP wrapping)
    optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)

    # 7. Training loop
    model.train()
    for epoch in range(num_epochs):
        for batch in dataloader:
            batch = batch.to(rank)
            loss = model(batch)
            loss.backward()
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            optimizer.step()
            optimizer.zero_grad()

    # 8. Save checkpoint (gather full state dict)
    from torch.distributed.fsdp import FullStateDictConfig, StateDictType
    save_policy = FullStateDictConfig(offload_to_cpu=True, rank0_only=True)
    with FSDP.state_dict_type(model, StateDictType.FULL_STATE_DICT, save_policy):
        state = model.state_dict()
        if rank == 0:
            torch.save(state, "checkpoint.pt")
Checkpoint Tips: For large models, use StateDictType.SHARDED_STATE_DICT with torch.distributed.checkpoint to save/load sharded checkpoints in parallel — much faster than gathering to rank 0.

FSDP2 & DTensor Integration

FSDP2 (available in PyTorch 2.4+) is a complete rewrite of FSDP using the composable APIs and DTensor as the underlying sharding primitive. It addresses several limitations of FSDP1.

Key Improvements

FSDP1 (Legacy)

  • Flat parameter groups — all params in FSDP unit concatenated
  • Limited composability with tensor parallelism
  • Module-level wrapping only
  • use_orig_params=True needed for torch.compile
  • Separate API from other parallelisms

FSDP2 (New)

  • Per-parameter sharding via DTensor
  • Native composition with TP, PP, and CP
  • Parameter-level granularity
  • torch.compile friendly by default
  • Unified torch.distributed API

FSDP2 Usage

from torch.distributed._composable.fsdp import fully_shard, MixedPrecisionPolicy

# FSDP2: per-module sharding with composable API
mp_policy = MixedPrecisionPolicy(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.float32,
)

# Apply sharding to individual layers (composable)
for layer in model.layers:
    fully_shard(layer, mp_policy=mp_policy)   # shard each block

# Shard the root module
fully_shard(model, mp_policy=mp_policy)

# Now each parameter is a DTensor with Shard placement
# model.layers[0].attn.weight → DTensor(Shard(0), device_mesh)

# Compose with tensor parallelism
from torch.distributed.tensor.parallel import parallelize_module

# First apply TP, then FSDP2 — they compose naturally via DTensor
for layer in model.layers:
    parallelize_module(layer, tp_mesh, tp_plan)  # TP first
    fully_shard(layer, mesh=dp_mesh)              # then FSDP
Migration Note: FSDP2 is not a drop-in replacement for FSDP1 — it uses a different API (fully_shard() vs FSDP()). However, the conceptual model is the same (AllGather before compute, ReduceScatter after). If you're starting a new project with PyTorch 2.4+, prefer FSDP2 for better composability and performance.
