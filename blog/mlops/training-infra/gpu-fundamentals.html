GPU vs CPU Architecture

CPUs and GPUs represent fundamentally different approaches to computation. A CPU is optimized for latency—executing a single thread of complex, branching logic as fast as possible. A GPU is optimized for throughput—executing thousands of simple, identical operations simultaneously.

CPU

  • Cores: 8–128 (high-performance)
  • Design: Large caches, branch prediction, out-of-order execution
  • Strength: Low-latency serial tasks
  • Memory: ~50 GB/s DDR5 bandwidth
  • Best for: Control flow, I/O, OS tasks

GPU

  • Cores: 10,000–18,000+ CUDA cores
  • Design: SIMT execution, minimal cache per core
  • Strength: Massive data parallelism
  • Memory: ~3.35 TB/s HBM3e (H100)
  • Best for: Matrix ops, convolutions, element-wise

For ML workloads, the key insight is that neural network operations—matrix multiplications, convolutions, element-wise activations—are embarrassingly parallel. A single matrix multiply of shapes [4096 × 4096] × [4096 × 4096] involves ~137 billion FLOPs, all independent. This maps perfectly to the GPU's thousands of parallel execution units.

Interview Insight: "Why are GPUs good for ML?" is deceptively simple. The strong answer covers SIMT execution, memory bandwidth (not just core count), and tensor cores for mixed-precision matrix math.

CUDA Cores & Streaming Multiprocessors

A GPU is organized into Streaming Multiprocessors (SMs). Each SM is an independent processing unit containing CUDA cores, tensor cores, load/store units, special function units (SFUs), and a register file. The H100 has 132 SMs.

Warps: The GPU schedules threads in groups of 32 called warps. All threads in a warp execute the same instruction simultaneously (SIMT). When threads diverge (e.g., different branches of an if), both paths are executed serially—this is warp divergence, a major performance killer.

Occupancy: The ratio of active warps to maximum warps an SM can host. Higher occupancy helps hide memory latency (while one warp waits for data, another can execute). The H100 supports up to 64 warps (2048 threads) per SM.

Key Number: An H100 SXM has 132 SMs × 128 CUDA cores = 16,896 CUDA cores. With 4 tensor cores per SM, that's 528 tensor cores total.

Tensor Cores

Tensor cores are specialized matrix multiply-accumulate (MMA) units that perform a fused D = A × B + C operation on small matrix tiles in a single clock cycle. They are the reason modern GPUs achieve such extraordinary FLOPS for ML workloads.

Each tensor core computes a 4×4×4 MMA per cycle. The hardware tiles larger matrices by decomposing them into these small blocks. Different precisions are supported with dramatically different throughput:

Precision Formats

  • FP32: Full precision, CUDA cores only (19.5 TFLOPS on H100)
  • TF32: 19-bit (10-bit mantissa), tensor core default for torch.matmul
  • FP16/BF16: Half precision, 2× throughput of TF32
  • FP8: Hopper-generation, 2× throughput of FP16

H100 SXM TFLOPS

  • FP32 (CUDA): 67 TFLOPS
  • TF32 (Tensor): 989 TFLOPS
  • FP16/BF16 (Tensor): 1,979 TFLOPS
  • FP8 (Tensor): 3,958 TFLOPS
  • INT8 (Tensor): 3,958 TOPS
TF32 Gotcha: Since Ampere, PyTorch uses TF32 by default for torch.matmul on tensor cores. This is a silent precision reduction. Disable with torch.backends.cuda.matmul.allow_tf32 = False if you need exact FP32 reproducibility.

Memory Hierarchy

GPU performance is often memory-bound, not compute-bound. Understanding the memory hierarchy and its bandwidth at each level is critical for optimization.

The arithmetic intensity (FLOPs per byte) of an operation determines whether it is compute-bound or memory-bound. Matrix multiplications have high arithmetic intensity (O(n³) FLOPs, O(n²) memory) and are usually compute-bound. Element-wise operations (ReLU, LayerNorm) have O(n) FLOPs and O(n) memory accesses—they are memory-bound.

This is why kernel fusion (combining multiple operations into a single GPU kernel) is so impactful: it avoids round-trips to HBM between operations. FlashAttention, for example, fuses the entire attention computation into a single kernel, reducing HBM reads by 5–20×.

Rule of Thumb: If your operation's arithmetic intensity is below the GPU's compute-to-bandwidth ratio (~200 FLOPs/byte on H100 for FP16), it's memory-bound, and you should focus on reducing memory traffic, not increasing FLOPS.

GPU Generations

Knowing the generational leaps helps you choose hardware and understand framework defaults.

V100 (Volta, 2017)

  • 5,120 CUDA cores, 640 tensor cores
  • 16/32 GB HBM2 @ 900 GB/s
  • 125 TFLOPS FP16 tensor
  • First tensor cores (FP16 MMA)
  • NVLink 2.0: 300 GB/s

A100 (Ampere, 2020)

  • 6,912 CUDA cores, 432 tensor cores
  • 40/80 GB HBM2e @ 2 TB/s
  • 312 TFLOPS FP16 tensor
  • Introduced TF32, sparsity support
  • NVLink 3.0: 600 GB/s

H100 (Hopper, 2022)

  • 16,896 CUDA cores, 528 tensor cores
  • 80 GB HBM3e @ 3.35 TB/s
  • 1,979 TFLOPS FP16/BF16 tensor
  • FP8 support, Transformer Engine
  • NVLink 4.0: 900 GB/s

B100/B200 (Blackwell, 2024)

  • Two-die design, 192 GB HBM3e
  • Up to 4.5 TB/s memory bandwidth
  • ~2× FP8 TFLOPS vs H100
  • 5th-gen NVLink: 1.8 TB/s
  • Native FP4 support

CUDA Programming Model

While most ML engineers don't write raw CUDA, understanding the programming model helps debug performance issues and interpret profiler output.

CUDA organizes computation into a 3-level hierarchy: Grid → Block → Thread. A kernel launch specifies the grid dimensions (how many blocks) and block dimensions (how many threads per block).

// CUDA kernel: vector addition
__global__ void vecAdd(float* A, float* B, float* C, int N) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < N) {
        C[idx] = A[idx] + B[idx];
    }
}

// Launch: 256 threads per block, ceil(N/256) blocks
int blockSize = 256;
int numBlocks = (N + blockSize - 1) / blockSize;
vecAdd<<<numBlocks, blockSize>>>(d_A, d_B, d_C, N);

Key rules: Threads within a block can communicate via shared memory and synchronize with __syncthreads(). Threads across blocks cannot synchronize—they are independent. Block size is typically 128 or 256; values must be multiples of 32 (warp size).

PyTorch Connection: When you call torch.matmul(A, B), PyTorch dispatches a cuBLAS GEMM kernel that was carefully tuned to maximize tensor core utilization, shared memory tiling, and occupancy. Custom CUDA kernels (like FlashAttention) exist where cuBLAS's general approach leaves performance on the table.

Profiling GPU Workloads

Profiling is the difference between guessing and knowing where your bottleneck is. The three essential profiling tools:

1. nvidia-smi — The quick health check. Shows GPU utilization, memory usage, temperature, and power draw. Use nvidia-smi dmon for continuous monitoring.

# Quick snapshot
$ nvidia-smi

# Continuous monitoring (1-second interval)
$ nvidia-smi dmon -s pucvmet -d 1

# Show per-process GPU memory
$ nvidia-smi --query-compute-apps=pid,process_name,used_memory --format=csv

2. torch.profiler — PyTorch's built-in profiler. Records CPU/GPU operations, memory allocation, and CUDA kernel execution times. Generates Chrome trace files and TensorBoard visualizations.

with torch.profiler.profile(
    activities=[
        torch.profiler.ProfilerActivity.CPU,
        torch.profiler.ProfilerActivity.CUDA,
    ],
    schedule=torch.profiler.schedule(wait=1, warmup=1, active=3),
    on_trace_ready=torch.profiler.tensorboard_trace_handler("./log"),
    record_shapes=True,
    with_stack=True,
) as prof:
    for step, (inputs, targets) in enumerate(loader):
        train_step(inputs, targets)
        prof.step()

3. Nsight Systems / Nsight Compute — NVIDIA's professional profiling tools. Nsight Systems provides a system-wide timeline showing CPU, GPU, and communication events. Nsight Compute drills into individual kernel performance with metrics like achieved occupancy, memory throughput, and instruction mix.

Common Mistake: Low "GPU Utilization" in nvidia-smi doesn't always mean the GPU is idle—it means the SM occupancy averaged over time is low. Profile with torch.profiler to see if your bottleneck is data loading (CPU-bound) or small kernel launches (launch overhead).
