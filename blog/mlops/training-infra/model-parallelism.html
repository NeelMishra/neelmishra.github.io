Model Parallelism: Tensor and Pipeline

MLOps Series Training Infrastructure

Why Model Parallelism

Data parallelism replicates the entire model on each GPU and splits the batch. This works when the model fits in a single GPU's memory. But modern LLMs blow past that limit: a 70B parameter model in FP16 requires ~140 GB just for parameters — far exceeding the 80 GB of an A100. Model parallelism splits the model itself across devices so training can proceed even when no single accelerator can hold the full model state.

Memory Breakdown for a Transformer

For a model with Ψ parameters trained with Adam in mixed precision, the memory per GPU under data parallelism is:

# Per-GPU memory in data parallelism (mixed precision + Adam)
parameters:       2Ψ bytes   # FP16 weights
gradients:        2Ψ bytes   # FP16 gradients
optimizer_states: 12Ψ bytes  # FP32 copy + momentum + variance (Adam)
activations:      variable      # depends on batch size, sequence length, hidden dim
# Total ≈ 16Ψ + activations
# For 70B params: 16 × 70B = 1,120 GB just for model state!
Interview Tip: When asked "Why not just use data parallelism for 70B models?", the answer is memory — you cannot fit 16Ψ bytes on one GPU. Model parallelism is about distributing model state, not data.

Tensor Parallelism

Tensor parallelism (TP) splits individual weight matrices across GPUs. Rather than placing different layers on different devices, TP partitions the computation within a single layer. The seminal approach is Megatron-LM style column-parallel and row-parallel linear layers.

Column-Parallel Linear Layer

For a linear layer Y = XA, column parallelism splits A along its columns across N GPUs. Each GPU computes a partial output with its shard of A, then results are concatenated (or used directly by the next row-parallel layer).

Row-Parallel Linear Layer

Row parallelism splits A along rows. Each GPU holds a portion of the input and computes a partial sum. An AllReduce combines partial results into the full output.

Key Insight: In Megatron-LM, the MLP uses column-parallel for the first linear and row-parallel for the second. The self-attention heads are naturally split across GPUs (each GPU owns a subset of heads). This means only two AllReduce operations per transformer layer — one in the MLP and one in the attention.

Communication Pattern

Tensor parallelism requires AllReduce within each layer (forward and backward). This means it is latency-sensitive and works best when GPUs are connected via high-bandwidth interconnects like NVLink (600 GB/s on A100) rather than cross-node PCIe or InfiniBand.

Pipeline Parallelism Overview

Pipeline parallelism (PP) assigns different groups of layers to different GPUs. GPU 0 runs layers 0–15, GPU 1 runs layers 16–31, and so on. The forward pass flows sequentially through stages, and the backward pass flows in reverse.

Naive Pipeline: The Bubble Problem

With naive pipelining, only one GPU is active at a time. While GPU 0 computes forward for micro-batch 1, GPUs 1–3 are idle. This creates pipeline bubbles — wasted GPU time.

# Naive pipeline schedule for 4 stages, 1 micro-batch
# Time ─────────────────────────────────────────────►
# GPU 0: [  F  ] [       ] [       ] [  B  ]
# GPU 1: [     ] [  F   ] [  B   ] [     ]
# GPU 2: [     ] [      ] [F / B ] [     ]
# GPU 3: [     ] [      ] [ F ][B] [     ]
#
# Bubble ratio ≈ (p-1) / p  where p = number of stages
# For p=4: 75% of GPU time is wasted!

The solution is micro-batching: split each mini-batch into m micro-batches and pipeline them through the stages. This shrinks the bubble ratio to (p-1) / (m + p - 1). See the dedicated Pipeline Parallelism deep dive for GPipe and PipeDream schedules.

Hybrid Parallelism (3D)

Real-world large model training combines all three strategies into 3D parallelism: Data Parallelism (DP) × Tensor Parallelism (TP) × Pipeline Parallelism (PP). Megatron-Turing NLG (530B) used TP=8, PP=35, DP=48 across 280 DGX A100 nodes.

Placement Strategy

The key principle: tensor parallelism within a node, pipeline and data parallelism across nodes. TP requires high bandwidth (AllReduce every layer) while PP only needs point-to-point communication between stages. DP communicates gradient AllReduce once per step.

# Typical 3D parallelism placement for 64 GPUs
# 8 nodes × 8 GPUs per node

tp_size = 8        # Intra-node (NVLink, 600 GB/s per A100 pair)
pp_size = 4        # Cross-node (InfiniBand, 200 Gb/s)
dp_size = 2        # Cross-node (InfiniBand)

# Total GPUs = 8 × 4 × 2 = 64
# Each TP group = 1 node (8 GPUs with NVLink)
# Each PP group spans 4 nodes (point-to-point between stages)
# DP groups span remaining 2 copies (AllReduce for gradients)

Implementation with PyTorch

PyTorch provides several primitives for model parallelism. Below is a practical implementation showing tensor parallelism for a linear layer and a device map for pipeline placement.

Manual Tensor Parallelism

import torch
import torch.distributed as dist

class ColumnParallelLinear(torch.nn.Module):
    def __init__(self, in_features, out_features, tp_group):
        super().__init__()
        self.tp_world = dist.get_world_size(group=tp_group)
        self.tp_rank = dist.get_rank(group=tp_group)
        # Each GPU holds out_features / tp_world columns
        self.local_out = out_features // self.tp_world
        self.weight = torch.nn.Parameter(
            torch.empty(self.local_out, in_features)
        )
        torch.nn.init.kaiming_uniform_(self.weight)

    def forward(self, x):
        # x: [batch, seq, hidden] — replicated on all TP ranks
        local_out = torch.nn.functional.linear(x, self.weight)
        # local_out: [batch, seq, local_out_features]
        # Next layer (RowParallel) consumes this directly
        return local_out

class RowParallelLinear(torch.nn.Module):
    def __init__(self, in_features, out_features, tp_group):
        super().__init__()
        self.tp_group = tp_group
        self.local_in = in_features // dist.get_world_size(group=tp_group)
        self.weight = torch.nn.Parameter(
            torch.empty(out_features, self.local_in)
        )

    def forward(self, x):
        local_out = torch.nn.functional.linear(x, self.weight)
        # AllReduce to sum partial results across TP group
        dist.all_reduce(local_out, group=self.tp_group)
        return local_out

Pipeline Placement with device_map

from accelerate import infer_auto_device_map, dispatch_model

# Automatically partition layers across GPUs based on memory
device_map = infer_auto_device_map(
    model,
    max_memory={0: "20GB", 1: "20GB", 2: "20GB", 3: "20GB"},
    no_split_module_classes=["TransformerBlock"]  # Don't split within a block
)

# device_map example: {'embed': 0, 'layers.0-7': 0, 'layers.8-15': 1,
#                      'layers.16-23': 2, 'layers.24-31': 3, 'lm_head': 3}

model = dispatch_model(model, device_map=device_map)
Warning: device_map with Accelerate gives you naive pipeline parallelism without micro-batching — no schedule overlap. For production training use Megatron-LM or DeepSpeed's pipeline engine which implements proper 1F1B or interleaved schedules.

Communication Analysis

Understanding communication costs is critical for choosing parallelism strategies and hardware topology.

Communication Primitives

Primitive Used By Volume Pattern
AllReduce TP (per layer), DP (per step) 2(N-1)/N × tensor_size Collective (all-to-all)
Send/Recv PP (between stages) activation_size Point-to-point
AllGather FSDP, ZeRO-3 (N-1)/N × tensor_size Collective
ReduceScatter FSDP, ZeRO-2/3 (N-1)/N × tensor_size Collective

Bandwidth Requirements

# Tensor Parallelism communication per transformer layer (forward + backward)
# 2 AllReduce ops per layer (attention + MLP), each 2× in backward
tp_comm_per_layer = 4 * allreduce_volume
allreduce_volume  = 2 * (tp_size - 1) / tp_size * hidden_size * batch * seq

# Pipeline Parallelism communication per micro-batch
pp_comm_per_microbatch = 2 * hidden_size * micro_batch * seq  # Send + Recv

# Data Parallelism communication per step (once after all micro-batches)
dp_comm_per_step = 2 * (dp_size - 1) / dp_size * total_params * 2  # FP16 grads
Optimal Placement Rule: Place tensor-parallel groups on GPUs connected by NVLink (intra-node, 600+ GB/s). Place pipeline stages on adjacent nodes connected by InfiniBand (200 Gb/s). Data-parallel communication is less frequent and more tolerant of lower bandwidth.

Decision Framework

Choosing the right parallelism strategy depends on model size, available hardware, and training objectives.

Tensor Parallelism

  • Best for: Very wide layers, attention heads
  • Requires: NVLink / high-BW interconnect
  • Typical TP size: 2, 4, or 8 (within one node)
  • Pro: Balanced memory, no pipeline bubbles
  • Con: AllReduce every layer = latency sensitive
  • Use when: Model fits in 1 node with TP splitting

Pipeline Parallelism

  • Best for: Deep models with many layers
  • Requires: Only point-to-point links (IB is fine)
  • Typical PP size: 4–64 stages
  • Pro: Low bandwidth requirement, cross-node friendly
  • Con: Pipeline bubbles, memory imbalance at edges
  • Use when: Model spans multiple nodes

Hybrid 3D Parallelism

  • Best for: 100B+ parameter models
  • Strategy: TP within node → PP across nodes → DP for throughput
  • Pro: Scales to thousands of GPUs
  • Con: Complex configuration and debugging
  • Use when: Neither TP nor PP alone suffices

Data Parallelism Only

  • Best for: Model fits in one GPU
  • Requires: Any interconnect
  • Pro: Simplest to implement, near-linear scaling
  • Con: Cannot train models larger than GPU memory
  • Use when: Model + optimizer state < GPU memory
Interview Decision Tree: (1) Does the model fit on one GPU? → Use DP. (2) Does it fit on one node with TP? → Use TP + DP. (3) Too large for one node? → Use TP (intra-node) + PP (inter-node) + DP. (4) Still OOM? → Add ZeRO-3/FSDP for parameter sharding.
