Gradient Accumulation: Virtual Batch Sizes

MLOps Series Training Infrastructure

Why Gradient Accumulation

Large-batch training is essential for modern deep learning — larger batches provide more stable gradient estimates, enabling higher learning rates and faster convergence. Research on LLMs frequently uses effective batch sizes of 1M+ tokens. However, a single GPU often cannot fit such batches in memory.

The core insight: gradient accumulation exploits the mathematical property that the gradient of a sum equals the sum of gradients. Instead of computing gradients on one large batch, we compute them on K smaller micro-batches and sum the results before applying the optimizer update.

Mathematically, for a loss function L averaged over a batch of size N:

∇θ L(B) = (1/N) Σi=1..N ∇θ ℓ(xi) = (1/K) Σk=1..K [ (1/m) Σj∈B_k ∇θ ℓ(xj) ]

where K is the accumulation steps and m = N/K is the micro-batch size. The result is mathematically identical to processing the full batch at once.

Interview insight: Gradient accumulation is mathematically exact for most loss functions (MSE, cross-entropy with mean reduction). However, it is NOT exact for losses with batch-level statistics (e.g., contrastive loss with in-batch negatives), where the effective negative pool size shrinks per micro-batch.

How It Works

The accumulation cycle has three phases: accumulate (forward + backward without optimizer step), sync (average accumulated gradients), and step (apply optimizer update and zero gradients).

The critical detail: loss.backward() accumulates gradients by default in PyTorch — calling it multiple times adds to param.grad rather than replacing it. We only call optimizer.zero_grad() after the optimizer step, not between micro-batches.

Implementation

Here is the standard PyTorch implementation. The key is the division by accumulation_steps to maintain correct gradient magnitude.

# Standard gradient accumulation in PyTorch
def train_with_accumulation(model, dataloader, optimizer, loss_fn,
                            accumulation_steps=4):
    model.train()
    optimizer.zero_grad(set_to_none=True)

    for step, (inputs, targets) in enumerate(dataloader):
        inputs, targets = inputs.cuda(), targets.cuda()

        # Forward pass
        outputs = model(inputs)
        loss = loss_fn(outputs, targets)

        # Normalize loss by accumulation steps
        # This ensures gradient magnitude matches a true large batch
        loss = loss / accumulation_steps
        loss.backward()  # Gradients accumulate in param.grad

        # Only step when we've accumulated enough
        if (step + 1) % accumulation_steps == 0:
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            optimizer.step()
            optimizer.zero_grad(set_to_none=True)
Critical detail: You must divide the loss by accumulation_steps before calling backward(). If you use reduction='mean' in your loss function (the default), each micro-batch loss is already averaged over that micro-batch. Dividing by K then gives you the correct average over the full effective batch. Forgetting this step effectively multiplies your learning rate by K.

For mixed precision with gradient accumulation:

# Gradient accumulation + mixed precision
scaler = torch.amp.GradScaler("cuda")

for step, (inputs, targets) in enumerate(dataloader):
    with torch.amp.autocast(device_type="cuda", dtype=torch.float16):
        outputs = model(inputs.cuda())
        loss = loss_fn(outputs, targets.cuda()) / accumulation_steps

    scaler.scale(loss).backward()

    if (step + 1) % accumulation_steps == 0:
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        scaler.step(optimizer)
        scaler.update()
        optimizer.zero_grad(set_to_none=True)

Effective Batch Size

The effective batch size determines the actual number of samples contributing to each optimizer step. This is the number that matters for learning rate tuning, warmup schedules, and convergence behavior.

effective_batch_size = micro_batch_size × accumulation_steps × num_gpus

Example A: Single GPU

  • micro_batch_size = 8
  • accumulation_steps = 16
  • num_gpus = 1
  • Effective: 8 × 16 × 1 = 128

Example B: Multi-GPU

  • micro_batch_size = 8
  • accumulation_steps = 4
  • num_gpus = 8
  • Effective: 8 × 4 × 8 = 256

Choosing the right decomposition: Prefer maximizing micro_batch_size first (better GPU utilization), then num_gpus (parallel computation), and use accumulation_steps as the final multiplier (sequential, slowest). Each accumulation step adds wall-clock time proportional to one forward-backward pass.

Token-based batching (LLMs): For language models, effective batch size is measured in tokens, not sequences: effective_tokens = micro_bs × seq_len × accum_steps × num_gpus. GPT-3 used 3.2M tokens per batch; LLaMA used 4M tokens.

Learning rate scaling: When increasing effective batch size, the linear scaling rule suggests scaling LR proportionally: if you double the batch size, double the learning rate. For very large batches (>8K), use the square root scaling rule or LARS/LAMB optimizers to maintain stability.

Interaction with DDP

In Distributed Data Parallel (DDP), gradients are synchronized across GPUs via AllReduce after every backward() call. During gradient accumulation, this synchronization is wasteful for intermediate steps — we only need the AllReduce on the final accumulation step.

PyTorch provides the no_sync() context manager to skip gradient synchronization:

# Efficient gradient accumulation with DDP
from contextlib import nullcontext

def train_ddp_accumulation(model, dataloader, optimizer, loss_fn,
                           accumulation_steps=4):
    for step, (inputs, targets) in enumerate(dataloader):
        # Skip AllReduce on intermediate accumulation steps
        is_accumulating = (step + 1) % accumulation_steps != 0
        context = model.no_sync() if is_accumulating else nullcontext()

        with context:
            outputs = model(inputs.cuda())
            loss = loss_fn(outputs, targets.cuda()) / accumulation_steps
            loss.backward()  # AllReduce only on sync step

        if not is_accumulating:
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            optimizer.step()
            optimizer.zero_grad(set_to_none=True)
Performance impact: Without no_sync(), DDP performs an AllReduce at every backward pass. For accumulation_steps=8, you'd do 8 AllReduce operations instead of 1 per optimizer step — a potential 8× increase in communication overhead. The no_sync() optimization is essential for training at scale.

How no_sync works internally: DDP registers a backward hook that triggers AllReduce when gradients are ready. no_sync() temporarily disables these hooks. On the final accumulation step (without no_sync()), the hooks fire and AllReduce synchronizes the accumulated gradients across all ranks.

Batch Normalization Issues

Batch normalization computes statistics (mean and variance) over the current micro-batch. With gradient accumulation, each micro-batch is small, leading to noisy statistics that degrade model quality.

The problem: With a micro-batch of 4 samples, the BatchNorm running mean and variance are computed from just 4 examples — far too few for stable statistics. The forward pass uses these noisy statistics, and the running averages updated during training become unreliable.

Problem: Small Micro-batch BN

  • Noisy mean/variance estimates
  • Running stats diverge from true population stats
  • Eval mode performance degrades
  • Training becomes unstable with micro_bs < 16

Solutions

  • GroupNorm / LayerNorm: Compute stats per-sample, independent of batch
  • SyncBatchNorm: Aggregate stats across GPUs before normalization
  • Ghost BatchNorm: Use virtual mini-batches within the micro-batch
  • Freeze BN: Use pre-computed running stats, skip BN gradient updates
Critical insight: BatchNorm's running stats are updated per forward pass, NOT per optimizer step. With accumulation_steps=8, the running stats are updated 8× per optimizer step but with 8× smaller batches. This can cause train/eval distribution mismatch. Prefer LayerNorm for transformer architectures where this is a non-issue.

For vision models that rely on BatchNorm (ResNet, EfficientNet), the recommended approach is SyncBatchNorm combined with gradient accumulation:

# Convert BatchNorm to SyncBatchNorm for multi-GPU training
model = ResNet50()
model = torch.nn.SyncBatchNorm.convert_sync_batchnorm(model)
model = model.cuda()
model = torch.nn.parallel.DistributedDataParallel(model)

# SyncBatchNorm computes mean/var across all GPUs
# Effective BN batch = micro_bs × num_gpus
# For 8 GPUs with micro_bs=4, BN sees 32 samples

Learning Rate Scheduling

A subtle but important detail: learning rate schedulers must be aligned with optimizer steps, not micro-batch iterations. Miscounting steps is one of the most common bugs when adding gradient accumulation to existing code.

Steps vs iterations: An "iteration" is a single forward-backward pass on one micro-batch. An optimizer "step" happens every accumulation_steps iterations. The scheduler should advance once per optimizer step.

# Correct LR scheduling with gradient accumulation
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)
num_training_steps = len(dataloader) // accumulation_steps * num_epochs
scheduler = get_cosine_schedule_with_warmup(
    optimizer,
    num_warmup_steps=1000,         # In optimizer steps, not iterations
    num_training_steps=num_training_steps,
)

for step, (inputs, targets) in enumerate(dataloader):
    loss = loss_fn(model(inputs.cuda()), targets.cuda()) / accumulation_steps
    loss.backward()

    if (step + 1) % accumulation_steps == 0:
        optimizer.step()
        scheduler.step()      # Advance scheduler ONCE per optimizer step
        optimizer.zero_grad(set_to_none=True)
Warmup alignment: Warmup steps should be specified in optimizer steps. If you want 1000 warmup steps with accumulation_steps=4, that's 4000 micro-batch iterations of warmup. HuggingFace Trainer handles this automatically; custom loops must be explicit.

Logging alignment: Training metrics (loss, learning rate, gradient norm) should also be logged per optimizer step, not per iteration. Logging per iteration produces K repeated data points at the same effective training progress, making loss curves misleading.

Common Bugs

  • Scheduler advances per iteration → LR decays K× too fast
  • Warmup counted in iterations → model under-warms
  • Forgetting loss division → effective LR multiplied by K
  • Missing no_sync() in DDP → K× communication overhead

Verification Checklist

  • ✓ Loss curves match non-accumulated baseline
  • ✓ LR schedule shape matches expected curve
  • ✓ Gradient norm is comparable across configs
  • ✓ Total optimizer steps = total_iters / accum_steps
