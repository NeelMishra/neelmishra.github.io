Checkpointing: Frequency, Async, Elastic

MLOps Series Training Infrastructure

Why Checkpointing Matters

Checkpointing is the foundation of reliable training infrastructure. Without checkpoints, a single hardware failure, OOM event, or preemption wipes out hours or days of compute. For a 70B parameter model training on 256 GPUs at $40/hr per node, a lost 12-hour segment costs $15,000+ in wasted compute.

Checkpoints serve three critical purposes:

  • Crash Recovery: Resume training from the last saved state after hardware failures, preemptions, or software bugs
  • Experiment Resumption: Pause and resume runs for cluster maintenance, hyperparameter adjustments, or debugging
  • Model Selection: Save multiple checkpoints to pick the best model (lowest validation loss, highest downstream metric) after training completes
Interview Context: "How do you ensure a 2-week training run isn't lost to a single GPU failure?" touches on checkpointing, distributed resilience, and monitoring—three pillars of production ML infrastructure.

What's in a Checkpoint

A complete checkpoint contains everything needed to resume training from the exact state where it was saved. Missing any component causes subtle bugs—training may resume but converge to a worse result.

def save_full_checkpoint(model, optimizer, scheduler, scaler, step, epoch, config, path):
    checkpoint = {
        # Model weights — the primary artifact
        "model_state_dict": model.state_dict(),

        # Optimizer state: momentum buffers, Adam running averages
        # Often 2-3× the size of model weights for Adam
        "optimizer_state_dict": optimizer.state_dict(),

        # LR scheduler state (current step, warmup phase, etc.)
        "scheduler_state_dict": scheduler.state_dict(),

        # GradScaler state for mixed-precision training
        "scaler_state_dict": scaler.state_dict(),

        # RNG states for exact reproducibility
        "rng_states": {
            "python": random.getstate(),
            "numpy": np.random.get_state(),
            "torch": torch.get_rng_state(),
            "cuda": torch.cuda.get_rng_state_all(),
        },

        # Training progress metadata
        "global_step": step,
        "epoch": epoch,
        "best_val_loss": best_val_loss,
        "config": config,
        "timestamp": time.time(),
    }
    torch.save(checkpoint, path)
Common Mistake: Forgetting to save optimizer state. Without it, Adam's running mean/variance estimates reset to zero on resume, causing a transient loss spike and potentially divergence. This is especially damaging late in training when these estimates are well-tuned.

Frequency Strategies

How often to checkpoint depends on your compute cost, storage budget, and preemption risk. There is no single correct answer—it's an optimization problem.

Time-Based

Checkpoint every N minutes. Simple, predictable storage usage. Best for spot instances where preemption risk is constant over time.

Step-Based

Checkpoint every N training steps. Most common approach. Easy to reason about training progress but wall-clock time between checkpoints varies.

Loss-Based

Save when validation loss improves. Efficient storage but risky—long gaps without a checkpoint if loss plateaus. Combine with a step-based fallback.

Keep-Last-K

Maintain only the K most recent checkpoints. Prevents storage explosion. Typically K=3–5, plus a "best" checkpoint based on validation metric.

Asynchronous Checkpointing

Synchronous checkpointing stalls training while tensors are serialized and written to disk. For a 70B model, this can take 2–5 minutes per checkpoint. Asynchronous checkpointing eliminates this stall by offloading the I/O to a background thread.

import torch, threading, copy
from concurrent.futures import ThreadPoolExecutor

class AsyncCheckpointer:
    def __init__(self, max_concurrent=1):
        self.executor = ThreadPoolExecutor(max_workers=max_concurrent)
        self.pending_future = None

    def save(self, state_dict, path):
        # Wait for any previous checkpoint to finish
        if self.pending_future and not self.pending_future.done():
            self.pending_future.result()  # block if still writing

        # Stage 1: Copy tensors to CPU pinned memory (fast, ~1s)
        # This is the only part that blocks the training loop
        cpu_state = {}
        for key, tensor in state_dict.items():
            if isinstance(tensor, torch.Tensor):
                cpu_state[key] = tensor.to("cpu", non_blocking=True).clone()
            else:
                cpu_state[key] = copy.deepcopy(tensor)

        torch.cuda.synchronize()  # ensure D2H copies complete

        # Stage 2: Serialize + write to disk in background thread
        self.pending_future = self.executor.submit(
            torch.save, cpu_state, path
        )

    def finalize(self):
        # Call at training end to ensure last checkpoint is saved
        if self.pending_future:
            self.pending_future.result()
Key Insight: The GPU→CPU copy (Stage 1) takes ~1s even for large models thanks to pinned memory and async DMA. The serialization + disk write (Stage 2) takes minutes but runs in the background while training continues on GPU.

PyTorch 2.3+ includes torch.distributed.checkpoint.async_save, which implements this pattern with additional distributed coordination.

Distributed Checkpointing

When using FSDP or DeepSpeed, model parameters are sharded across GPUs. Each rank holds only a fraction of the full model. This fundamentally changes checkpointing.

Approach 1: Gather-then-Save (Full State Dict)

All shards are gathered to rank 0, which saves a single complete checkpoint. Simple to load but creates a memory bottleneck on rank 0 and doesn't scale beyond ~30B parameters.

Approach 2: Sharded Checkpoint (Distributed Save)

Each rank saves its own shard independently. No gathering, no memory bottleneck, and linear write throughput. The checkpoint is a directory of shard files.

import torch.distributed.checkpoint as dcp
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import StateDictType

# Save sharded checkpoint — each rank writes its own shard
with FSDP.state_dict_type(model, StateDictType.SHARDED_STATE_DICT):
    state_dict = {
        "model": model.state_dict(),
        "optimizer": FSDP.optim_state_dict(model, optimizer),
    }
    dcp.save(state_dict, checkpoint_id=f"ckpt/step_{step}")

# Load with potentially different world_size (resharding)
with FSDP.state_dict_type(model, StateDictType.SHARDED_STATE_DICT):
    state_dict = {
        "model": model.state_dict(),
        "optimizer": FSDP.optim_state_dict(model, optimizer),
    }
    dcp.load(state_dict, checkpoint_id=f"ckpt/step_{step}")
    model.load_state_dict(state_dict["model"])
    # Optimizer state is automatically resharded to match new topology

Full State Dict

  • Single file, easy to manage
  • Memory bottleneck on rank 0
  • Doesn't scale past ~30B
  • No resharding needed on load

Sharded State Dict

  • Directory of shard files
  • No memory bottleneck
  • Scales to any model size
  • Supports elastic resharding on load

Elastic Checkpoints

Elastic checkpoints handle topology changes—loading a checkpoint saved on N GPUs into a training run on M GPUs. This is essential for elastic training on spot instances, where the number of available nodes changes between restarts.

The challenge: optimizer state (Adam moments) is sharded differently on 8 GPUs vs 16 GPUs. The checkpoint format must encode enough metadata to reshape these tensors at load time.

# DTensor-based resharding (PyTorch 2.x)
# The checkpoint stores tensor metadata (shape, sharding spec)
# On load, the runtime figures out how to redistribute shards

from torch.distributed.checkpoint import FileSystemReader

def load_elastic_checkpoint(model, optimizer, ckpt_dir):
    # Create empty state dicts with current topology's sharding
    model_sd = model.state_dict()   # sharded for current world_size
    optim_sd = FSDP.optim_state_dict(model, optimizer)

    # DCP reads checkpoint metadata, reshards tensors to match
    # current topology, and fills in the state dicts
    dcp.load(
        {"model": model_sd, "optimizer": optim_sd},
        checkpoint_id=ckpt_dir,
    )

    # Apply resharded state
    model.load_state_dict(model_sd)
    load_optim_state_dict(model, optimizer, optim_sd)
DeepSpeed Elastic Checkpoints: DeepSpeed uses a "universal checkpoint" format that stores tensors in a topology-independent layout. On load, the runtime re-partitions for the target parallelism configuration (TP, PP, DP degrees can all change).

What Can Change Between Save and Load

  • Data Parallelism degree: Most common change—different number of nodes. Fully supported by DCP and DeepSpeed.
  • Tensor Parallelism degree: Supported by DeepSpeed universal checkpoints and Megatron-LM's resharding tools.
  • Pipeline Parallelism depth: Requires re-mapping layers to pipeline stages. More complex but supported.
  • Precision change: Load fp32 checkpoint into bf16 run or vice versa. Usually straightforward with casting.

Storage Optimization

A 70B model produces ~200 GB per checkpoint. With keep-last-5, that's 1 TB of checkpoint storage. Training for weeks generates tens of TB. Storage optimization is critical.

Compression

Neural network weights have low entropy and compress well. gzip typically achieves 40–60% reduction; zstd is faster with similar ratios. Trade compute for storage.

Incremental Checkpoints

Save only tensors that changed since the last checkpoint. Model weights barely change between steps, but optimizer moments drift significantly. In practice, incremental saves reduce I/O by 30–50%.

Deduplication

When using keep-last-K, many tensors are identical across consecutive checkpoints (especially frozen layers). Content-addressable storage deduplicates these automatically.

import hashlib, torch

class DeduplicatingCheckpointer:
    def __init__(self, storage_dir):
        self.storage_dir = storage_dir
        self.manifest = {}  # maps tensor_name → content_hash

    def save_tensor(self, name, tensor):
        # Hash the tensor content
        content_hash = hashlib.sha256(
            tensor.numpy().tobytes()
        ).hexdigest()[:16]

        blob_path = f"{self.storage_dir}/blobs/{content_hash}.pt"
        if not exists(blob_path):
            torch.save(tensor, blob_path)  # only write if new

        self.manifest[name] = content_hash

    def save_checkpoint(self, state_dict, step):
        for name, tensor in state_dict.items():
            self.save_tensor(name, tensor)
        # Save lightweight manifest (just name → hash mappings)
        save_json(self.manifest, f"{self.storage_dir}/step_{step}.json")
Cloud Storage Integration: Use S3/GCS multi-part upload for large checkpoints. Each shard can upload in parallel, achieving 5–10 GB/s aggregate throughput. Set lifecycle policies to auto-delete old checkpoints after N days.
