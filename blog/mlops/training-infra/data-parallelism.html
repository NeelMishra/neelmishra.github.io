Data Parallelism: Scaling Across GPUs

MLOps Series Training Infrastructure

Data Parallelism Concept

Data parallelism is the most straightforward multi-GPU strategy: replicate the model on every GPU, split the data batch, and synchronize gradients. Each GPU processes a different subset of the batch (micro-batch), computes local gradients, and then all GPUs collectively average their gradients before the optimizer step.

The mathematical guarantee is simple: since all replicas start with the same weights and apply the same averaged gradient update, they remain synchronized without explicitly broadcasting weights.

Key Insight: Data parallelism is mathematically equivalent to training on a single GPU with a larger batch size. The only approximation is when batch normalization statistics differ per GPU (solved by SyncBatchNorm).

Gradient Synchronization

After each backward pass, every GPU has computed gradients using only its micro-batch. These local gradients must be averaged across all GPUs before the optimizer step. Why averaging and not summing?

Consider the loss over the global batch B split into N micro-batches:

# Mathematical equivalence
L_global = (1/B) × Σᵢ loss(xᵢ)                # loss over full batch

# Split into N micro-batches of size B/N:
L_global = (1/N) × Σₖ [ (1/(B/N)) × Σⱼ loss(xⱼ) ]  # average of micro-batch losses

# Therefore:
∇L_global = (1/N) × Σₖ ∇L_micro_k             # average of micro-batch gradients

This is exactly what AllReduce computes: the sum of all local gradient tensors divided by the number of participants. After AllReduce, every GPU has the identical averaged gradient tensor, so they all apply the identical optimizer update and stay in sync.

Subtle Bug: If you compute loss with reduction='sum' instead of reduction='mean', your effective learning rate scales with the number of GPUs. This causes training to diverge when adding GPUs. Always verify your loss reduction mode matches your gradient averaging strategy.

Gradient Accumulation extends this further: if you can only fit micro-batch size M on each GPU but want effective batch size B, accumulate gradients over B/(M×N) micro-batches before synchronizing:

accumulation_steps = effective_batch_size // (micro_batch_size * world_size)

for step, (inputs, targets) in enumerate(loader):
    with model.no_sync() if (step + 1) % accumulation_steps != 0 else nullcontext():
        loss = model(inputs, targets) / accumulation_steps
        loss.backward()

    if (step + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()
Performance Tip: Use model.no_sync() during accumulation steps to skip AllReduce for intermediate micro-batches. This reduces communication by accumulation_steps - 1× per effective batch.

AllReduce Algorithms

AllReduce is the collective operation that sums (or averages) a tensor across all processes and distributes the result to every process. The algorithm choice dramatically affects communication time.

Naive AllReduce: Each GPU sends its gradient to GPU 0, which sums them and broadcasts back. Cost: O(N × data_size). The master GPU is a bottleneck.

Ring AllReduce: GPUs are arranged in a logical ring. Data is split into N chunks. In N-1 steps, chunks are passed around the ring, each GPU adding its local contribution. After 2(N-1) steps, every GPU has the full sum. Cost: O(data_size), independent of N.

Tree AllReduce: Uses a binary tree topology. Reduce phase aggregates up the tree to root, then broadcast phase distributes down. Cost: O(data_size × log(N)). Better for small messages with many nodes.

Ring AllReduce

  • Time: 2(N-1) × (data/N) / bandwidth
  • Bandwidth-optimal — scales well with N
  • Latency: 2(N-1) × latency_per_step
  • Best for: large tensors, moderate N
  • Used by NCCL for most operations

Tree AllReduce

  • Time: 2 × log(N) × data / bandwidth
  • Not bandwidth-optimal
  • Latency: 2 × log(N) × latency_per_step
  • Best for: small tensors, many nodes
  • Used for small collective operations
NCCL Insight: NVIDIA's NCCL library automatically selects the optimal algorithm based on message size and topology. For most DDP training, ring AllReduce is used for gradient tensors (large). NCCL also leverages NVLink/NVSwitch topology to overlap inter-node and intra-node communication.

Batch Distribution

How data is distributed across GPUs matters for both correctness and training dynamics.

from torch.utils.data.distributed import DistributedSampler

sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,  # total number of GPUs
    rank=rank,                # this GPU's rank
    shuffle=True,
    drop_last=True           # drop remainder to ensure even splits
)

loader = DataLoader(
    dataset,
    batch_size=micro_batch_size,  # per-GPU batch size
    sampler=sampler,
    num_workers=4,
    pin_memory=True
)

# Critical: update sampler epoch for proper shuffling
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)
    for batch in loader:
        train_step(batch)

drop_last: When the dataset size isn't evenly divisible by world_size × batch_size, the last incomplete batch causes problems. Without drop_last=True, some ranks get fewer batches, causing a hang at AllReduce. With drop_last=True, a small amount of data is unused each epoch—acceptable for large datasets.

Padding alternative: For small datasets where dropping data is unacceptable, the sampler can pad with duplicate indices to ensure even distribution. However, this biases the gradient toward duplicated samples.

Data Leakage Risk: With drop_last=False and padding, DistributedSampler wraps around and duplicates samples from the beginning of the dataset. If used for validation, this inflates metrics for the duplicated samples. Always use drop_last=True for validation or compute metrics only on rank 0 with the full dataset.

Learning Rate Scaling

When you scale from 1 GPU to N GPUs, the effective batch size increases by N×. Larger batch sizes require learning rate adjustment to maintain training dynamics.

Linear Scaling Rule (Goyal et al., 2017): Multiply the base learning rate by N (the number of GPUs, i.e., the batch size multiplier). Intuition: with N× more samples per gradient step, the gradient variance decreases by N×, so you can take N× larger steps.

# Linear scaling with warmup
base_lr = 0.1             # learning rate for batch_size = 256
world_size = 8
scaled_lr = base_lr * world_size  # 0.8 for effective batch = 2048

# Warmup: ramp from base_lr to scaled_lr over first 5 epochs
# Without warmup, the large LR causes divergence early in training
warmup_epochs = 5
for epoch in range(warmup_epochs):
    lr = base_lr + (scaled_lr - base_lr) * epoch / warmup_epochs
    for param_group in optimizer.param_groups:
        param_group["lr"] = lr

The linear rule works well up to moderate batch sizes (~8K for ImageNet). Beyond that, training quality degrades. Two optimizers address this:

LARS (Layer-wise Adaptive Rate Scaling)

  • Scales LR per layer based on weight norm / gradient norm ratio
  • Enables training with batch sizes up to 32K
  • Works well for CNNs (ResNet)
  • Key formula: lr_layer = η × ||w|| / ||∇w||

LAMB (Layer-wise Adaptive Moments for Batch training)

  • Extends LARS to Adam optimizer
  • Enables batch sizes up to 64K+ for BERT pre-training
  • Better for transformers and NLP models
  • Combines Adam's adaptive moments with layer-wise scaling
Practical Guidance: For most training runs with ≤8 GPUs, linear scaling + warmup is sufficient. Only reach for LARS/LAMB when scaling to 32+ GPUs or batch sizes above ~8K.

Communication-Computation Overlap

The key to DDP's efficiency is overlapping gradient communication with backward computation. Instead of waiting for the entire backward pass to finish before communicating, DDP starts AllReduce on the first gradient buckets while later layers are still computing.

How DDP buckets work: PyTorch DDP groups parameters into buckets (default 25 MB). When all gradients in a bucket are ready, AllReduce starts immediately for that bucket—while backward continues computing gradients for earlier layers.

# Timeline for overlapped DDP backward:
#
# Layer order:  [Layer N] → [Layer N-1] → ... → [Layer 1]
#
# GPU compute:  |--backward LayerN--|--backward LayerN-1--|--..--|
# NCCL comms:           |---AllReduce bucket_N---|---bucket_N-1---|
#                       ↑ starts as soon as bucket_N grads ready
#
# Without overlap: total = T_backward + T_allreduce
# With overlap:    total ≈ max(T_backward, T_allreduce)

# Tune bucket size (default 25MB is usually good)
model = DDP(
    model,
    device_ids=[local_rank],
    bucket_cap_mb=25,          # group gradients into 25MB buckets
    gradient_as_bucket_view=True  # reduce memory copy overhead
)

Tuning bucket size: Smaller buckets start communication sooner (better overlap) but incur more kernel launch overhead. Larger buckets are more efficient per-byte but delay the start. 25 MB is a good default; increase for very large models, decrease for very fast networks (NVSwitch).

When Overlap Fails: If backward is much faster than communication (small model, slow interconnect), overlap doesn't help—you're communication-bound. Profile with torch.profiler to see if backward and AllReduce actually overlap in the timeline.

Benchmarking Scaling

Proper benchmarking separates actual GPU scaling issues from data loading bottlenecks and measurement artifacts.

import time

def benchmark_throughput(model, loader, device, num_steps=100, warmup=10):
    model.train()
    # Warmup: CUDA lazy initialization, JIT compilation
    for i, (x, y) in enumerate(loader):
        if i >= warmup: break
        train_step(model, x.to(device), y.to(device))

    torch.cuda.synchronize()
    start = time.perf_counter()

    total_samples = 0
    for i, (x, y) in enumerate(loader):
        if i >= num_steps: break
        train_step(model, x.to(device), y.to(device))
        total_samples += x.size(0)

    torch.cuda.synchronize()  # critical: wait for all GPU work
    elapsed = time.perf_counter() - start

    samples_per_sec = total_samples / elapsed
    return samples_per_sec

Key metrics to track:

  • Throughput: samples/second (or tokens/second for LLMs)
  • Scaling efficiency: throughput_N / (throughput_1 × N)
  • GPU utilization: SM occupancy from nvidia-smi
  • Communication fraction: time in AllReduce / total step time

Amdahl's Law sets the theoretical limit on scaling. If fraction s of the workload is serial (non-parallelizable), the maximum speedup with N GPUs is:

# Amdahl's Law
Speedup(N) = 1 / (s + (1 - s) / N)

# Example: 5% serial fraction
# N=8:   Speedup = 1 / (0.05 + 0.95/8) = 5.9×  (74% efficiency)
# N=64:  Speedup = 1 / (0.05 + 0.95/64) = 14.7× (23% efficiency)
# N=∞:   Speedup = 1 / 0.05 = 20× (hard ceiling)

# In DDP, "serial" includes: data loading, AllReduce latency,
# optimizer step, gradient accumulation overhead, Python dispatch

Typical Scaling Numbers

  • 2 GPUs (NVLink): 95-98% efficiency
  • 4 GPUs (NVLink): 90-95% efficiency
  • 8 GPUs (NVLink): 88-94% efficiency
  • 16 GPUs (2 nodes): 80-90% efficiency
  • 64 GPUs (8 nodes): 70-85% efficiency

Improving Efficiency

  • Increase batch size (better compute/comm ratio)
  • Use NVLink instead of PCIe
  • Enable gradient bucketing overlap
  • Use gradient compression (PowerSGD)
  • Ensure data loading isn't the bottleneck
Benchmarking Checklist: (1) Always use torch.cuda.synchronize() before timing—GPU operations are async. (2) Include warmup steps. (3) Use synthetic data to isolate GPU scaling from data loading. (4) Report both throughput and scaling efficiency.
