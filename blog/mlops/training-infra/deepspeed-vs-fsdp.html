DeepSpeed vs FSDP

MLOps Series Training Infrastructure

Overview

Both DeepSpeed ZeRO and PyTorch FSDP solve the same fundamental problem: eliminating memory redundancy in data-parallel training so that models too large for a single GPU can be trained across many GPUs. Despite their shared goal, they differ significantly in design philosophy, API surface, and ecosystem integration.

DeepSpeed is a standalone optimization library from Microsoft Research that wraps PyTorch. It is config-driven — you specify ZeRO stage, offloading, and other options in a JSON file. FSDP is PyTorch's built-in solution, using an API-driven approach where sharding, wrapping, and mixed precision are configured in Python code.

Interview Context: When asked "DeepSpeed or FSDP?", the answer isn't one-size-fits-all. DeepSpeed has more features (offloading, MoE, inference). FSDP has better PyTorch integration (torch.compile, DTensor, composability). The choice depends on your model, team, and infrastructure.

Feature Comparison

Key Takeaway: DeepSpeed is a broader toolkit (training + inference + MoE + sparse). FSDP is a focused training primitive that composes well with the rest of the PyTorch ecosystem. DeepSpeed wins on features; FSDP wins on integration.

Ease of Use

DeepSpeed: Config-Driven

// ds_config.json
{
  "zero_optimization": {
    "stage": 2,
    "overlap_comm": true,
    "reduce_bucket_size": 5e8
  },
  "fp16": {"enabled": true},
  "train_batch_size": 128,
  "gradient_accumulation_steps": 4
}

# Python: minimal code changes
model_engine, opt, _, _ = \
  deepspeed.initialize(
    model=model,
    config="ds_config.json"
  )
  • ✅ Non-engineers can tune via JSON
  • ✅ One-line initialization
  • ❌ Opaque — hard to debug internal state
  • ❌ Config sprawl with many options

FSDP: API-Driven

# Python: explicit configuration
model = FSDP(
  model,
  sharding_strategy=
    ShardingStrategy.FULL_SHARD,
  mixed_precision=MixedPrecision(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.bfloat16,
  ),
  auto_wrap_policy=partial(
    transformer_auto_wrap_policy,
    transformer_layer_cls=
      {TransformerBlock}
  ),
)
  • ✅ Transparent — debuggable Python
  • ✅ Composes with PyTorch ecosystem
  • ❌ More code to write
  • ❌ Steeper learning curve for beginners

Trend: The PyTorch ecosystem is converging on FSDP2 + DTensor as the standard distributed training primitive. New PyTorch-native tools (torchtune, torchtitan) use FSDP exclusively. However, DeepSpeed's inference engine and MoE support keep it relevant for specific use cases.

Migration Guide

Switching between DeepSpeed and FSDP is increasingly common as teams evaluate both options. Here are the key mappings:

DeepSpeed → FSDP Mapping

# DeepSpeed ZeRO Stage      →    FSDP Sharding Strategy
# ─────────────────────────────────────────────────────
# "stage": 0                →    ShardingStrategy.NO_SHARD (DDP)
# "stage": 1                →    No direct equivalent (use SHARD_GRAD_OP)
# "stage": 2                →    ShardingStrategy.SHARD_GRAD_OP
# "stage": 3                →    ShardingStrategy.FULL_SHARD
# ZeRO++ (hpZ)              →    ShardingStrategy.HYBRID_SHARD

# DeepSpeed Config Key      →    FSDP Equivalent
# ─────────────────────────────────────────────────────
# "fp16.enabled": true      →    MixedPrecision(param_dtype=torch.float16)
# "bf16.enabled": true      →    MixedPrecision(param_dtype=torch.bfloat16)
# "overlap_comm": true      →    backward_prefetch=BACKWARD_PRE
# "offload_optimizer"       →    CPUOffload(offload_params=True)
# "offload_param"           →    CPUOffload(offload_params=True)
# "reduce_bucket_size"      →    No direct equivalent (auto-tuned)

Migration Code Example

Before: DeepSpeed

import deepspeed

engine, opt, _, _ = deepspeed.initialize(
    model=model,
    config="ds_config.json",
    model_parameters=model.parameters()
)

for batch in loader:
    loss = engine(batch)
    engine.backward(loss)
    engine.step()

After: FSDP

from torch.distributed.fsdp import FSDP

model = FSDP(model,
  sharding_strategy=FULL_SHARD,
  mixed_precision=bf16_policy,
  auto_wrap_policy=wrap_policy)
opt = torch.optim.AdamW(model.parameters())

for batch in loader:
    loss = model(batch)
    loss.backward()
    opt.step(); opt.zero_grad()
Migration Pitfalls:
  • Checkpoints are not compatible. You must convert DeepSpeed checkpoints to standard PyTorch state dicts before loading into FSDP (and vice versa).
  • FSDP has no ZeRO-1 equivalent. If you use ZeRO-1 (optimizer-only sharding), the closest is SHARD_GRAD_OP which also shards gradients.
  • Custom learning rate schedulers that rely on DeepSpeed's scheduler interface need to be rewritten for PyTorch's native schedulers.

When to Use Which

Decision Matrix

Scenario Recommendation Reason
New project, PyTorch 2.4+ FSDP2 Best torch.compile support, composable with TP/PP
Need NVMe offloading DeepSpeed ZeRO-Infinity is unique to DeepSpeed
MoE (Mixture of Experts) model DeepSpeed Native MoE layer with expert parallelism
HuggingFace Trainer user Either Both equally supported via Trainer/Accelerate
Azure ML infrastructure DeepSpeed Deep Azure integration, Microsoft support
Need both training + inference DeepSpeed DeepSpeed-Inference for optimized serving
Combining FSDP + TP + PP FSDP2 DTensor-based composition is native
Single-GPU, very large model DeepSpeed ZeRO-Offload designed for this case
Maximal torch.compile speedup FSDP2 Designed for graph-based compilation

Summary Recommendation

Choose DeepSpeed When

  • You need CPU/NVMe offloading
  • Training MoE architectures
  • Team already has DeepSpeed expertise
  • Using Azure ML infrastructure
  • Need inference optimization too
  • Existing codebase uses DeepSpeed

Choose FSDP When

  • Starting a new PyTorch project
  • Want torch.compile integration
  • Need TP + PP composition
  • Prefer native PyTorch (fewer deps)
  • Using torchtune or torchtitan
  • Want future-proof (FSDP2 momentum)
Interview Answer: "Both DeepSpeed and FSDP achieve comparable throughput for the same sharding strategy. DeepSpeed offers more features — NVMe offloading, MoE, inference engine — and is config-driven. FSDP is PyTorch-native, composes with torch.compile and tensor parallelism via DTensor, and is the direction the ecosystem is heading. I'd choose FSDP2 for new projects unless I specifically need DeepSpeed-only features like NVMe offloading or MoE support."
