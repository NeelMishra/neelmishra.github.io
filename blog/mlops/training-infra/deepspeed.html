DeepSpeed: ZeRO Stages 1–3

MLOps Series Training Infrastructure

ZeRO Overview

ZeRO (Zero Redundancy Optimizer) eliminates the memory redundancy inherent in standard data parallelism. In DDP, every GPU holds a complete copy of model parameters, gradients, and optimizer states. For a model with Ψ parameters trained with Adam in mixed precision, each GPU stores:

# Memory per GPU in standard DDP (mixed precision + Adam)
FP16 parameters:    2Ψ bytes
FP16 gradients:     2Ψ bytes
FP32 master weights: 4Ψ bytes  # optimizer maintains FP32 copy
FP32 momentum:      4Ψ bytes  # Adam first moment
FP32 variance:      4Ψ bytes  # Adam second moment
─────────────────────────────────
Total:              16Ψ bytes  # per GPU — ALL identical!

# For 7B params: 16 × 7B = 112 GB per GPU
# With 64 GPUs: 64 × 112 GB = 7,168 GB total — 63 copies are redundant!

ZeRO partitions these redundant states across GPUs. Each stage eliminates more redundancy at the cost of additional communication.

Key Insight: ZeRO doesn't change the mathematical semantics of training. It produces identical results to DDP — the same gradients, the same parameter updates. It only changes where data is stored and when it is communicated.

ZeRO Stage 1: Optimizer State Partitioning

Stage 1 partitions only the optimizer states (master weights, momentum, variance) across N GPUs. Each GPU stores 1/N of the optimizer states. Parameters and gradients remain replicated.

Memory Savings

# ZeRO Stage 1 memory per GPU
FP16 parameters:     2Ψ bytes     # still replicated
FP16 gradients:      2Ψ bytes     # still replicated
FP32 optimizer:      12Ψ / N bytes # partitioned!
─────────────────────────────────
Total:               4Ψ + 12Ψ/N bytes

# For 7B params, N=64 GPUs:
# Standard DDP: 112 GB
# ZeRO-1: 4×7B + 12×7B/64 = 28 GB + 1.3 GB ≈ 29.3 GB  (3.8× reduction!)

Communication

After the backward pass computes gradients, an AllReduce averages gradients across GPUs (same as DDP). Then each GPU updates only its 1/N partition of optimizer states and parameters. Finally, an AllGather broadcasts the updated parameters so every GPU has the full model for the next forward pass.

Communication overhead vs DDP: ZeRO-1 adds one AllGather (for updated parameters) compared to standard DDP. The AllReduce for gradients is the same. Total communication ≈ 1.5× DDP.

ZeRO Stage 2: Gradient Partitioning

Stage 2 partitions both optimizer states and gradients. Instead of AllReduce (which leaves a full copy of gradients on each GPU), ZeRO-2 uses ReduceScatter — each GPU only keeps 1/N of the reduced gradients (the partition it's responsible for).

Memory Savings

# ZeRO Stage 2 memory per GPU
FP16 parameters:     2Ψ bytes     # still replicated
FP16 gradients:      2Ψ / N bytes # partitioned!
FP32 optimizer:      12Ψ / N bytes # partitioned
─────────────────────────────────
Total:               2Ψ + 14Ψ/N bytes

# For 7B params, N=64 GPUs:
# ZeRO-2: 2×7B + 14×7B/64 = 14 GB + 1.53 GB ≈ 15.5 GB
# vs DDP: 112 GB → 7.2× reduction!

Communication Pattern

The key change: ReduceScatter replaces AllReduce for gradients. This has the same communication volume as AllReduce (both are 2(N-1)/N × tensor_size) but each GPU retains only its partition. After the optimizer step, an AllGather broadcasts updated parameters.

# ZeRO-2 training step
for micro_batch in micro_batches:
    loss = forward(micro_batch)
    loss.backward()                     # compute local gradients

# ReduceScatter: each GPU gets 1/N of averaged gradients
reduce_scatter(gradients)               # replaces AllReduce

# Each GPU updates only its partition
optimizer_step(local_grad_partition)     # update 1/N params

# AllGather: broadcast updated params to all GPUs
all_gather(parameters)

ZeRO Stage 3: Parameter Partitioning

Stage 3 partitions everything — parameters, gradients, and optimizer states. No GPU holds a full copy of the model. Parameters are gathered on-demand before each forward/backward computation and discarded immediately after.

Memory Savings

# ZeRO Stage 3 memory per GPU
FP16 parameters:     2Ψ / N bytes # partitioned!
FP16 gradients:      2Ψ / N bytes # partitioned
FP32 optimizer:      12Ψ / N bytes # partitioned
─────────────────────────────────
Total:               16Ψ / N bytes

# For 7B params, N=64 GPUs:
# ZeRO-3: 16×7B/64 = 1.75 GB per GPU
# vs DDP: 112 GB → 64× reduction!
# This enables training models that are 64× larger on the same hardware!

Communication Pattern

ZeRO-3 requires AllGather before each forward and backward layer computation to reconstruct full parameters temporarily, plus ReduceScatter after backward. Total communication volume is 3× DDP.

# ZeRO-3 forward/backward for one layer
def zero3_layer_forward(layer, x, group):
    # Gather full params from all GPUs (temporary)
    full_weight = all_gather(layer.weight_partition, group=group)
    output = torch.nn.functional.linear(x, full_weight)
    del full_weight  # free immediately — don't keep!
    return output

def zero3_layer_backward(layer, grad_output, group):
    # Gather params again for gradient computation
    full_weight = all_gather(layer.weight_partition, group=group)
    grad_input = grad_output @ full_weight
    grad_weight = grad_output.T @ input_activation
    del full_weight  # free immediately
    # ReduceScatter gradients — each GPU keeps its partition
    local_grad = reduce_scatter(grad_weight, group=group)
    return grad_input, local_grad
Communication Cost: ZeRO-3 has 1.5× the communication volume of DDP per layer (AllGather forward + AllGather backward + ReduceScatter). However, this is overlapped with computation using prefetching — the AllGather for layer k+1 starts while layer k computes. In practice, the overhead is 10–20% on high-bandwidth clusters.

DeepSpeed Configuration

DeepSpeed is configured via a JSON file. Here are the key configurations for each ZeRO stage:

ZeRO Stage 2 Config (Most Common)

// ds_config.json — ZeRO Stage 2
{
  "train_batch_size": 256,
  "train_micro_batch_size_per_gpu": 4,
  "gradient_accumulation_steps": 8,

  "zero_optimization": {
    "stage": 2,
    "allgather_partitions": true,
    "allgather_bucket_size": 5e8,
    "overlap_comm": true,          // overlap communication with compute
    "reduce_scatter": true,
    "reduce_bucket_size": 5e8,
    "contiguous_gradients": true
  },

  "fp16": {
    "enabled": true,
    "loss_scale": 0,              // dynamic loss scaling
    "initial_scale_power": 16
  },

  "optimizer": {
    "type": "AdamW",
    "params": {
      "lr": 3e-4,
      "betas": [0.9, 0.999],
      "weight_decay": 0.01
    }
  }
}

ZeRO Stage 3 Config

}

Training Script Integration

import deepspeed

# Initialize DeepSpeed engine
model_engine, optimizer, _, scheduler = deepspeed.initialize(
    model=model,
    model_parameters=model.parameters(),
    config="ds_config.json"
)

# Training loop — DeepSpeed handles all ZeRO mechanics
for batch in dataloader:
    loss = model_engine(batch)
    model_engine.backward(loss)      # handles gradient partitioning
    model_engine.step()             # handles optimizer partitioning + AllGather
Pro Tip: Set "overlap_comm": true to overlap AllGather/ReduceScatter with computation. This can hide 50–80% of communication overhead on high-bandwidth interconnects.

ZeRO-Offload & ZeRO-Infinity

When even ZeRO-3 isn't enough, DeepSpeed can offload state to CPU RAM or NVMe storage.

ZeRO-Offload (CPU)

  • Offloads optimizer states + gradients to CPU RAM
  • Adam step computed on CPU (with DeepSpeed CPU Adam)
  • Enables 10× larger models on a single GPU
  • Throughput: ~60–70% of GPU-only ZeRO-2
  • Best for: single-GPU or small cluster training

ZeRO-Infinity (NVMe)

  • Offloads everything to NVMe SSD via CPU
  • Theoretically unlimited model size
  • Uses memory-mapped files + async I/O
  • Throughput: ~30–40% of GPU-only ZeRO-3
  • Best for: research on models that don't fit in GPU+CPU memory

Offloading Communication Flow

# ZeRO-Offload data flow per training step
#
# 1. GPU: Forward pass (params gathered via AllGather)
# 2. GPU: Backward pass (compute gradients)
# 3. GPU → CPU: Transfer gradients to CPU (PCIe)
# 4. CPU: Adam optimizer step (DeepSpeedCPUAdam, AVX-optimized)
# 5. CPU → GPU: Transfer updated params back (PCIe)
#
# PCIe bandwidth: ~32 GB/s (PCIe 4.0 x16)
# For 7B model: 14 GB of data transferred each way → ~0.44s per step
# Overlapped with GPU computation for next micro-batch
When NOT to Offload: If you have enough GPUs to fit the model with ZeRO-3, don't offload. CPU/NVMe offloading introduces PCIe bottlenecks that reduce throughput by 30–70%. Only offload when you literally cannot fit the model in aggregate GPU memory.

Performance Benchmarks

Below are representative throughput numbers for a 7B parameter model on 8× A100-80GB GPUs with NVLink + InfiniBand.

Configuration Memory/GPU Throughput Comm Overhead
DDP (baseline) ~72 GB 100% (reference) 1.0×
ZeRO Stage 1 ~25 GB ~97% 1.1×

Interview Recommendation: "Start with ZeRO Stage 2 — it gives the best memory-to-throughput ratio. Move to Stage 3 only when the model doesn't fit with Stage 2. Use offloading as a last resort when aggregate GPU memory isn't sufficient. Always enable overlap_comm to hide communication latency."
