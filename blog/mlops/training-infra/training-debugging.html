Debugging Training: NaN, Divergence

MLOps Series Training Infrastructure

Common Failure Modes

Training failures fall into distinct categories, each with characteristic signatures. Recognizing the pattern is the first step toward a fix. Below is a visual guide to the most common failure modes as observed in loss curves.

Failure Mode Summary

  • NaN loss: Loss becomes NaN or Inf, training halts. Usually caused by numerical overflow in fp16, division by zero, or log of zero/negative values.
  • Divergence: Loss increases monotonically. Learning rate too high, or gradient explosion in deep/recurrent networks.
  • Loss plateau: Loss stalls at a high value. Dead neurons (ReLU dying), vanishing gradients, learning rate too low, or incorrect normalization.
  • Loss spikes: Intermittent jumps in loss that may or may not recover. Corrupted data batches, learning rate schedule issues, or numerical instability in specific layers.
  • Mode collapse (GANs): Generator produces identical outputs. Discriminator too strong, or training imbalance between G and D.

NaN Debugging

NaN values propagate silently through computation graphs — a single NaN in any tensor will eventually corrupt the entire model. The key is to find the first operation that produces NaN.

import torch

# Method 1: torch.autograd.detect_anomaly (slow but thorough)
with torch.autograd.detect_anomaly():
    output = model(inputs)
    loss = criterion(output, targets)
    loss.backward()   # will raise RuntimeError at first NaN
    # Error msg includes the op that produced NaN and its traceback

# Method 2: Register forward hooks to catch NaN layer-by-layer
def nan_hook(module, input, output):
    if not isinstance(output, tuple):
        outputs = [output]
    else:
        outputs = output
    for i, out in enumerate(outputs):
        if isinstance(out, torch.Tensor):
            nan_mask = torch.isnan(out)
            if nan_mask.any():
                print(f"NaN in {module.__class__.__name__} output {i}")
                print(f"  Shape: {out.shape}, NaN count: {nan_mask.sum()}")
                raise RuntimeError(f"NaN detected in {module}")

for name, module in model.named_modules():
    module.register_forward_hook(nan_hook)

# Method 3: Gradient-level NaN detection
def grad_nan_hook(grad):
    if torch.isnan(grad).any():
        print(f"NaN gradient detected! Shape: {grad.shape}")
        return torch.zeros_like(grad)  # optionally zero out
    return grad

for name, param in model.named_parameters():
    param.register_hook(grad_nan_hook)

Common NaN Sources

Numerical Operations

  • log(0) → -Inf → NaN on next op
  • 0 / 0 → NaN directly
  • exp(large) → Inf in fp16 (>65504)
  • sqrt(negative) from numerical drift
  • Softmax overflow: subtract max first

Training Dynamics

  • Learning rate too high → gradient explosion
  • Missing gradient clipping on deep nets
  • fp16 underflow (grads too small) without loss scaling
  • Uninitialized parameters (random garbage)
  • Corrupted data (NaN in input features)
fp16 trap: The fp16 dynamic range is only [6e-8, 65504]. Values outside this range become zero (underflow) or Inf (overflow). Always use a loss scaler with fp16 — it multiplies the loss by a large factor before backward, then divides gradients afterward, keeping values in the representable range.

Gradient Issues

Gradient pathologies are the root cause of most training failures. Monitoring gradient norms gives you an early warning system before the loss curve shows obvious problems.

import torch

def monitor_gradients(model, step, writer):
    """Log gradient statistics for each layer every N steps."""
    total_norm = 0.0
    for name, param in model.named_parameters():
        if param.grad is not None:
            grad_norm = param.grad.data.norm(2).item()
            total_norm += grad_norm ** 2

            # Per-layer monitoring
            writer.add_scalar(f"grad_norm/{name}", grad_norm, step)

            # Detect pathological values
            if grad_norm > 100:
                print(f"⚠ Large gradient in {name}: {grad_norm:.2f}")
            if grad_norm < 1e-7:
                print(f"⚠ Vanishing gradient in {name}: {grad_norm:.2e}")

    total_norm = total_norm ** 0.5
    writer.add_scalar("grad_norm/total", total_norm, step)
    return total_norm

# Gradient clipping strategies
# 1. Global norm clipping (most common, used by GPT/LLaMA)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# 2. Value clipping (clip each element independently)
torch.nn.utils.clip_grad_value_(model.parameters(), clip_value=0.5)

# 3. Adaptive clipping (AGC — used in NFNets)
def adaptive_grad_clip(parameters, clip_factor=0.01, eps=1e-3):
    for p in parameters:
        if p.grad is None:
            continue
        p_norm = p.data.norm(2).clamp_(min=eps)
        g_norm = p.grad.data.norm(2)
        max_norm = p_norm * clip_factor
        clip_coef = max_norm / g_norm.clamp_(min=1e-6)
        if clip_coef < 1:
            p.grad.data.mul_(clip_coef)

Vanishing vs Exploding Gradients

Vanishing Gradients

Symptoms: Early layers learn nothing, loss stalls, gradient norms < 1e-6 in early layers.

Causes: Deep networks without residual connections, saturating activations (sigmoid/tanh), poor initialization.

Fixes: Skip connections (ResNet), LayerNorm/BatchNorm, GELU/SiLU activations, careful init (Kaiming/Xavier), gradient checkpointing to avoid floating-point accumulation errors.

Exploding Gradients

Symptoms: Loss becomes NaN/Inf, gradient norms > 1000, parameter values diverge rapidly.

Causes: Learning rate too high, missing normalization, deep recurrent networks, unstable attention patterns.

Fixes: Gradient clipping (max_norm=1.0), lower learning rate, warmup schedule, pre-norm transformer architecture, weight decay.

Loss Spike Analysis

Loss spikes — sudden jumps in the loss value — are among the trickiest issues to debug because they're intermittent. At scale (thousands of GPUs, billions of tokens), they become almost inevitable. The key is determining whether they're benign (recoverable) or catastrophic (requiring rollback).

Root Cause Analysis

  • Data corruption: A batch contains NaN values, extremely long sequences, mislabeled examples, or encoding errors. Log batch statistics (min, max, mean, NaN count) when a spike occurs.
  • Learning rate schedule: A sudden LR increase (e.g., warmup restart, cosine cycle peak) combined with a particularly hard batch can cause a temporary spike. Check if spikes correlate with the LR schedule.
  • Numerical instability: Specific attention patterns (e.g., attending to padding tokens) can create softmax instability. Layer norm with very small variance can amplify noise.
  • Gradient accumulation bugs: If gradient accumulation count is wrong, effective batch size changes, causing an implicit learning rate change.
# Automated loss spike detection and logging
class SpikeDetector:
    def __init__(self, window=100, threshold=3.0):
        self.losses = []
        self.window = window
        self.threshold = threshold  # standard deviations

    def check(self, step, loss, batch_meta):
        self.losses.append(loss)
        if len(self.losses) < self.window:
            return False

        recent = self.losses[-self.window:]
        mean = sum(recent) / len(recent)
        std = (sum((x - mean) ** 2 for x in recent) / len(recent)) ** 0.5

        if loss > mean + self.threshold * std:
            log_spike(
                step=step,
                loss=loss,
                mean=mean,
                std=std,
                batch_ids=batch_meta["sample_ids"],
                grad_norm=batch_meta["grad_norm"],
                lr=batch_meta["lr"],
            )
            return True
        return False
Production practice: Teams training frontier LLMs (Google, Meta, Anthropic) maintain spike budgets — if the cumulative number of spikes exceeds a threshold, the run is rolled back to the last clean checkpoint. Typical policy: roll back if loss hasn't recovered within 100 steps of a spike.

Learning Rate Debugging

The learning rate is the single most impactful hyperparameter. A wrong LR can cause divergence (too high), painfully slow convergence (too low), or stalled training (warmup too short/long).

LR Finder (Smith 2018)

# LR range test: sweep LR from very small to very large
def lr_finder(model, dataloader, optimizer, start_lr=1e-7, end_lr=10, steps=200):
    lrs, losses = [], []
    lr_mult = (end_lr / start_lr) ** (1 / steps)
    lr = start_lr
    best_loss = float("inf")

    for step, batch in zip(range(steps), dataloader):
        # Set learning rate
        for pg in optimizer.param_groups:
            pg["lr"] = lr

        loss = train_step(model, batch, optimizer)
        lrs.append(lr)
        losses.append(loss)

        # Stop if loss diverges (4× the best loss)
        if loss > 4 * best_loss:
            break
        best_loss = min(best_loss, loss)
        lr *= lr_mult

    # Plot lrs vs losses (log scale on x-axis)
    # Optimal LR: ~1/10 of the LR where loss starts increasing
    return lrs, losses

Warmup Importance

Learning rate warmup is critical for transformers. Without it, the Adam optimizer's variance estimates are inaccurate in the first few hundred steps, leading to erratic updates.

  • Linear warmup: LR increases linearly from 0 to peak LR over N steps. Standard for most transformer training (GPT, BERT, LLaMA).
  • Typical warmup duration: 0.1–1% of total steps for large runs, 5–10% for small/medium runs.
  • Without warmup symptoms: Loss spikes or diverges in the first 100–500 steps, then sometimes recovers but with worse final performance.

Common LR Schedules and Debugging

Cosine Annealing

LR decays following a cosine curve from peak to minimum (usually 0 or 0.1× peak). Standard for LLM pre-training.

Debug tip: If loss plateaus early, the peak LR might be too low. If it diverges mid-training, peak LR is too high or min LR is not low enough.

WSD (Warmup-Stable-Decay)

Warmup → constant LR → decay. Used by MiniCPM and some recent models. Allows extending training without committing to total step count upfront.

Debug tip: If using WSD but loss stalls during stable phase, try a brief cooldown period.

Distributed Training Debugging

Distributed training introduces an entirely new class of bugs: synchronization issues, rank-specific failures, NCCL hangs, and non-determinism. These are among the hardest bugs to debug because they only manifest at scale.

Rank-Specific Failures

import torch.distributed as dist

# Log per-rank information for debugging
def debug_distributed(loss, grad_norm, step):
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    # Gather loss from all ranks
    all_losses = [torch.zeros(1, device="cuda") for _ in range(world_size)]
    dist.all_gather(all_losses, torch.tensor([loss], device="cuda"))

    if rank == 0:
        losses = [l.item() for l in all_losses]
        max_diff = max(losses) - min(losses)
        if max_diff > 0.1 * sum(losses) / len(losses):
            print(f"⚠ Step {step}: Loss divergence across ranks!")
            print(f"  Losses: {losses}")
            print(f"  Max diff: {max_diff:.4f}")

# Check for NCCL hangs with timeout
import os
os.environ["NCCL_ASYNC_ERROR_HANDLING"] = "1"
os.environ["NCCL_TIMEOUT"] = "600"  # 10 minute timeout
os.environ["TORCH_NCCL_BLOCKING_WAIT"] = "1"

# For debugging, enable NCCL trace
os.environ["NCCL_DEBUG"] = "INFO"    # WARNING: very verbose
os.environ["NCCL_DEBUG_SUBSYS"] = "INIT,COLL"  # limit to subset

Common Distributed Bugs

  • NCCL hang: One rank crashes or stalls, others wait forever at the next collective. Set NCCL_TIMEOUT and NCCL_ASYNC_ERROR_HANDLING=1 to convert hangs into errors.
  • Gradient desynchronization: Ranks compute different gradients (different data, different dropout masks without synced RNG, or a bug in sharding). Periodically compare parameter checksums across ranks.
  • Uneven data distribution: One rank gets fewer batches and finishes its epoch early, causing a deadlock at the next all-reduce. Use DistributedSampler(drop_last=True).
  • Mixed-precision sync issues: All-reduce in fp16 vs fp32 can cause silent divergence. Ensure all ranks use the same precision for collective operations.
Reproducibility: Set seeds on every rank: torch.manual_seed(seed + rank) for data shuffling, torch.manual_seed(seed) (same seed) for model initialization. Use torch.backends.cudnn.deterministic = True for exact reproducibility, but expect a 10–15% slowdown.

Systematic Debugging Approach

When training fails, resist the urge to change hyperparameters randomly. Follow a systematic approach: isolate, reproduce, diagnose, fix, and verify.

The Binary Search Method

When you don't know where the bug is, bisect your way to it:

  1. Overfit a single batch: If the model can't overfit 1 batch, there's a fundamental bug in the model or loss function.
  2. Scale to 10 batches: Can it memorize a small dataset? If not, check data pipeline, augmentation, or label errors.
  3. Scale to 100 batches: Does it generalize slightly? If it overfits these but not the full dataset, data diversity or augmentation issues.
  4. Full dataset: Normal hyperparameter tuning territory.

Ablation Studies

  • Remove one component at a time: dropout, augmentation, specific layers, normalization.
  • Replace complex components with simple baselines: attention → MLP, custom loss → cross-entropy.
  • Reduce model size until it works, then scale back up to find the breaking point.
🔍 Debugging Checklist (bookmark this):
  1. Data: Verify inputs are normalized, labels are correct, no NaN in features, shapes are consistent.
  2. Model: Check parameter initialization, verify forward pass shapes, ensure loss function matches task.
  3. Optimization: Start with a known-good LR (3e-4 for Adam), enable gradient clipping, add warmup.
  4. Numerics: Enable anomaly detection, check for overflow/underflow, verify mixed-precision setup.
  5. Scale: First make it work on 1 GPU with small data, then scale up. Never debug at full scale.
  6. Reproducibility: Set all seeds, use deterministic algorithms, log everything.
  7. Monitoring: Log gradient norms, parameter norms, activation statistics, loss per rank.

import torch

# 1. Fixed seed for reproducibility
torch.manual_seed(42)

# 2. Minimal model (smallest version that shows the bug)
model = SmallModel(hidden=64, layers=2).cuda()

# 3. Synthetic data (eliminate data pipeline as variable)
def make_batch(batch_size=4, seq_len=32):
    return {
        "input_ids": torch.randint(0, 1000, (batch_size, seq_len)).cuda(),
        "labels": torch.randint(0, 1000, (batch_size, seq_len)).cuda(),
    }

# 4. Train loop with full monitoring
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)
for step in range(100):
    batch = make_batch()
    loss = model(batch)
    loss.backward()
    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    optimizer.step()
    optimizer.zero_grad()
    print(f"Step {step}: loss={loss.item():.4f}, grad_norm={grad_norm:.4f}")

# If bug doesn't reproduce → data pipeline or scale issue
