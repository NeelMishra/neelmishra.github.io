Training Monitoring: Loss, GPU Utilization

MLOps Series Training Infrastructure

Key Training Metrics

Effective training monitoring begins with the right metrics. Without visibility into what's happening inside a multi-day, multi-GPU run, you're flying blind—problems compound silently until they manifest as wasted compute or a diverged model.

Loss Curves

The training loss is your primary convergence signal. Log both per-step loss (noisy but immediate) and a smoothed moving average (EMA with α=0.99). Validation loss at regular intervals gives the generalization signal. Watch for:

  • Loss spikes: Sudden jumps indicate data quality issues, learning rate problems, or numerical instability
  • Loss plateaus: May indicate learning rate too low, exhausted data diversity, or architectural bottleneck
  • Train-val divergence: Growing gap signals overfitting

Learning Rate

Log the effective learning rate at each step, especially with complex schedules (warmup + cosine decay + restarts). A common bug is the scheduler resetting on resume, causing the LR to jump back to warmup values.

Gradient Norms

The L2 norm of gradients reveals training health. Stable norms (within a 2–3× range) indicate healthy training. Exploding norms precede loss spikes; vanishing norms indicate dead layers or saturated activations.

Throughput

Measure in domain-specific units: tokens/second for language models, samples/second for vision models, steps/second for general training. Throughput drops indicate data pipeline bottlenecks, GPU contention, or network issues in distributed training.

Interview Tip: When asked "How do you know if training is going well?", answer with this hierarchy: (1) loss is decreasing, (2) gradient norms are stable, (3) throughput matches expectations, (4) GPU utilization is high.

GPU Utilization Monitoring

GPU utilization is the most commonly misunderstood training metric. The nvidia-smi "GPU-Util" number is SM (Streaming Multiprocessor) activity—the percentage of time at least one kernel is running on the GPU. This can be 100% even if you're using the GPU inefficiently.

import subprocess, torch, time

def get_gpu_metrics(device_id=0):
    # Query nvidia-smi for detailed GPU stats
    result = subprocess.run(
        ["nvidia-smi", "--query-gpu=utilization.gpu,utilization.memory,"
         "memory.used,memory.total,temperature.gpu,power.draw",
         "--format=csv,noheader,nounits", f"--id={device_id}"],
        capture_output=True, text=True
    )
    gpu_util, mem_util, mem_used, mem_total, temp, power = \
        [float(x.strip()) for x in result.stdout.split(",")]

    return {
        "gpu_utilization_pct": gpu_util,      # SM activity %
        "memory_utilization_pct": mem_util,    # memory controller activity %
        "memory_used_gb": mem_used / 1024,
        "memory_total_gb": mem_total / 1024,
        "temperature_c": temp,
        "power_draw_w": power,
    }

def monitor_training_loop(model, dataloader, optimizer):
    for step, batch in enumerate(dataloader):
        t0 = time.perf_counter()

        loss = train_step(model, batch, optimizer)

        step_time = time.perf_counter() - t0
        tokens_per_sec = batch["input_ids"].numel() / step_time

        if step % 10 == 0:
            gpu = get_gpu_metrics()
            log_metrics({
                "loss": loss.item(),
                "throughput_tokens_sec": tokens_per_sec,
                "step_time_ms": step_time * 1000,
                **gpu
            })

Key distinctions to understand:

GPU Utilization (SM Activity)

  • % of time a GPU kernel is active
  • 100% doesn't mean efficient
  • A memory-bound kernel shows 100% but low compute throughput

Memory Utilization

  • % of time the memory controller is active
  • High values indicate memory-bound workloads
  • Different from "memory used" (allocation)
Common Pitfall: Seeing 99% GPU utilization and assuming the training is efficient. In reality, the GPU may be idle between all_reduce calls in distributed training, and the 99% reflects only the compute phases. Use MFU (next section) for true efficiency measurement.

Model FLOPs Utilization (MFU)

MFU is the gold-standard efficiency metric for large-scale training, introduced in the PaLM paper. It measures what fraction of the GPU's theoretical peak FLOPS your training actually achieves.

MFU = Actual FLOPS achieved / Theoretical peak FLOPS

def compute_mfu(model_params_B, batch_size, seq_len, step_time_s, num_gpus, gpu_type="A100"):
    # Theoretical peak FLOPS per GPU (bf16/fp16)
    peak_flops = {
        "A100": 312e12,   # 312 TFLOPS bf16
        "H100": 989e12,   # 989 TFLOPS bf16 (with sparsity: 1978 TFLOPS)
        "H200": 989e12,   # Same compute as H100, more memory
    }

    # Approximate FLOPS per token for transformer forward+backward
    # ~6 * model_params per token (2× forward, 4× backward)
    flops_per_token = 6 * model_params_B * 1e9
    tokens_per_step = batch_size * seq_len

    # Actual FLOPS achieved per second
    actual_flops = (flops_per_token * tokens_per_step) / step_time_s

    # MFU as fraction of total cluster peak
    total_peak = peak_flops[gpu_type] * num_gpus
    mfu = actual_flops / total_peak

    return mfu  # e.g., 0.55 means 55% MFU

# Example: 7B model, bs=1024, seq=2048, 0.8s/step, 64 A100s
# mfu = (6 * 7e9 * 1024 * 2048) / 0.8 / (312e12 * 64) ≈ 0.56

What does good MFU look like?

MFU Range Assessment Likely Issue
> 55%ExcellentState-of-the-art, well-optimized
45–55%GoodNormal for most setups
30–45%SuboptimalCommunication overhead, data pipeline stalls
< 30%PoorSevere bottleneck—profile immediately
Why MFU rarely exceeds 60%: Even perfectly optimized code loses FLOPS to: (1) all_reduce communication between GPUs, (2) attention computation isn't pure matmul, (3) non-matmul ops (LayerNorm, softmax, activations), (4) pipeline bubbles. The 6N approximation is also an overcount.

Weights & Biases Integration

W&B (Weights & Biases) is the de facto standard for ML experiment tracking. It provides metric logging, hyperparameter sweeps, run comparison, and artifact versioning.

import wandb, torch

def setup_wandb(config):
    wandb.init(
        project="llm-pretraining",
        name=f"{config.model_name}-{config.run_id}",
        config={
            "model_params": config.num_params,
            "learning_rate": config.lr,
            "batch_size": config.global_batch_size,
            "seq_length": config.seq_len,
            "optimizer": config.optimizer_name,
            "precision": config.precision,
            "num_gpus": config.world_size,
            "parallelism": config.parallel_strategy,
        },
        tags=["pretraining", config.model_size, config.dataset_name],
        group=config.experiment_group,  # group related runs
    )

def log_training_step(step, loss, lr, grad_norm, throughput, gpu_metrics):
    wandb.log({
        # Core training metrics
        "train/loss": loss,
        "train/learning_rate": lr,
        "train/grad_norm": grad_norm,
        "train/throughput_tokens_sec": throughput,
        "train/mfu": compute_mfu(throughput),

        # GPU health metrics
        "gpu/utilization": gpu_metrics["gpu_util"],
        "gpu/memory_used_gb": gpu_metrics["mem_used"],
        "gpu/temperature": gpu_metrics["temperature"],
        "gpu/power_watts": gpu_metrics["power"],

        # System metrics (auto-logged by wandb agent)
        # CPU utilization, disk I/O, network I/O tracked automatically
    }, step=step)

def log_validation(step, val_loss, val_perplexity):
    wandb.log({
        "val/loss": val_loss,
        "val/perplexity": val_perplexity,
    }, step=step)
    # Log model checkpoint as artifact for versioning
    artifact = wandb.Artifact(f"model-ckpt-{step}", type="model")
    artifact.add_dir(f"checkpoints/step_{step}")
    wandb.log_artifact(artifact)
Pro Tip: Use wandb.define_metric("val/*", step_metric="train/global_step") to align validation metrics with training steps on the x-axis, even though validation runs less frequently.

Alerting & Anomaly Detection

Monitoring is only useful if anomalies trigger timely alerts. For multi-day training runs, a 6-hour detection delay means 6 hours of wasted compute.

Loss Spike Detection

Trigger an alert when the loss exceeds median(last_1000_steps) + 5 * MAD (median absolute deviation). MAD is more robust than standard deviation against the inherent noise in per-step loss.

GPU Failure Alerts

Monitor for: (1) sudden drop in GPU utilization to 0%, (2) ECC memory errors reported by nvidia-smi, (3) GPU temperature exceeding throttle threshold (83°C for A100). Any of these likely indicates hardware failure that will hang the training.

Training Stall Detection

If no training steps complete for N minutes (typically 5–10 minutes), trigger an alert. Common causes: NCCL timeout from a dead node, data loader stuck waiting for I/O, or a deadlocked collective operation.

class TrainingAlerter:
    def __init__(self, loss_window=1000, spike_threshold=5.0, stall_timeout=600):
        self.loss_history = []
        self.spike_threshold = spike_threshold
        self.stall_timeout = stall_timeout
        self.last_step_time = time.time()

    def check_loss_spike(self, loss):
        self.loss_history.append(loss)
        if len(self.loss_history) < 100:
            return False
        median = np.median(self.loss_history[-1000:])
        mad = np.median(np.abs(
            np.array(self.loss_history[-1000:]) - median
        ))
        return loss > median + self.spike_threshold * mad

    def check_stall(self):
        elapsed = time.time() - self.last_step_time
        return elapsed > self.stall_timeout

    def on_step(self, step, loss, gpu_metrics):
        self.last_step_time = time.time()
        if self.check_loss_spike(loss):
            send_alert(f"Loss spike at step {step}: {loss:.4f}")
        if gpu_metrics["temperature"] > 83:
            send_alert(f"GPU thermal throttle: {gpu_metrics['temperature']}°C")
Warning: Don't auto-stop training on loss spikes. Many spikes are transient (bad data batches, numerical noise) and the model self-corrects within 50–100 steps. Alert the on-call engineer, but let training continue unless the spike persists for >500 steps.

Profiling Deep Dives

When monitoring reveals a bottleneck (low MFU, slow steps), profiling tools help pinpoint the cause. The goal is to build a trace timeline showing exactly where time is spent.

from torch.profiler import profile, ProfilerActivity, schedule

# Profile 3 steps after 5 warmup steps
with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=schedule(wait=2, warmup=3, active=3, repeat=1),
    on_trace_ready=torch.profiler.tensorboard_trace_handler("./profiler_logs"),
    record_shapes=True,
    profile_memory=True,
    with_stack=True,
) as prof:
    for step, batch in enumerate(dataloader):
        loss = train_step(model, batch, optimizer)
        prof.step()
        if step >= 10:
            break

# Print top CUDA operations by time
print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=15))

Common bottlenecks revealed by profiling:

  • Data loading gaps: GPU idle between steps → increase DataLoader num_workers or use prefetching
  • Large all_reduce blocks: Communication dominates → overlap compute and communication, use gradient compression
  • Memory-bound kernels: Low SM throughput → use Flash Attention, fused kernels
  • Frequent CUDA synchronization: Python-side .item() or logging calls → batch metric collection

Production Dashboards

A well-designed dashboard turns raw metrics into actionable insights. For production training runs, build a hierarchical dashboard with three levels of detail.

Level 1: Overview (Glance)

Single screen showing run health: current loss, ETA to completion, aggregate GPU utilization, and any active alerts. This is what the on-call engineer checks every few hours.

Level 2: Training Detail

Loss curve, learning rate schedule, gradient norms, throughput over time, and validation metrics. This is for the ML engineer optimizing convergence.

Level 3: Infrastructure

Per-GPU utilization, memory usage, network bandwidth, data pipeline throughput, checkpoint save latency. This is for diagnosing infrastructure bottlenecks.

Grafana Setup Tip: Use Prometheus for GPU metrics (via dcgm-exporter), push training metrics via a StatsD sidecar, and query W&B for experiment metadata. Template variables for run_id and gpu_id let you drill down without building separate dashboards.

Key Dashboard Panels

Training Health

  • Loss curve (train + val)
  • Learning rate schedule
  • Gradient norm distribution
  • Tokens/sec throughput
  • MFU over time

Infrastructure Health

  • GPU memory waterfall
  • NCCL bandwidth (GB/s)
  • Data loader queue depth
  • Checkpoint save duration
