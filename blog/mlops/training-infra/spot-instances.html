Training on Spot Instances: Fault Tolerance

MLOps Series Training Infrastructure

Spot/Preemptible Instances

Cloud providers sell unused compute capacity at steep discounts—60–90% cheaper than on-demand pricing. AWS calls them Spot Instances, GCP calls them Preemptible VMs (or Spot VMs), and Azure offers Spot VMs. The catch: the provider can reclaim these instances with as little as 30 seconds to 2 minutes of warning.

For ML training, this creates a tension. GPU hours are the dominant cost center for large-scale training, so spot pricing is enormously attractive. But a preemption event mid-epoch can waste hours of compute if the job isn't designed to handle interruptions gracefully.

Interview Insight: A common question is "How would you reduce training costs by 70% without compromising convergence?" The answer nearly always involves spot instances plus a robust checkpointing and restart strategy.

Key economics at typical 2024 pricing:

Instance On-Demand ($/hr) Spot ($/hr) Savings
AWS p4d.24xlarge (8×A100)$32.77$11.50~65%
GCP a2-highgpu-8g (8×A100)$29.39$8.82~70%
Azure ND96amsr_A100 (8×A100)$27.20$8.16~70%

Preemption rates vary by instance type, region, and time of day. GPU instances tend to have lower preemption rates (5–15%) than CPU instances because the spare-capacity pool is smaller but demand is more predictable. Even so, for a multi-day training run, the probability of experiencing at least one preemption approaches certainty.

Checkpointing for Spot

The core defense against preemption is frequent checkpointing. Every checkpoint you save bounds the maximum amount of work you can lose. If you checkpoint every 10 minutes, a preemption costs at most 10 minutes of compute.

Cloud providers emit preemption signals before termination. AWS sends a 2-minute warning via instance metadata; GCP provides 30 seconds. Your training loop should poll for this signal and trigger an emergency checkpoint.

import signal, requests, torch

class SpotCheckpointManager:
    def __init__(self, model, optimizer, save_dir, interval_steps=500):
        self.model = model
        self.optimizer = optimizer
        self.save_dir = save_dir
        self.interval = interval_steps
        self.global_step = 0
        # Register SIGTERM handler for graceful preemption
        signal.signal(signal.SIGTERM, self._emergency_save)

    def _check_spot_termination(self):
        # AWS: poll instance metadata for spot termination notice
        try:
            resp = requests.get(
                "http://169.254.169.254/latest/meta-data/spot/termination-time",
                timeout=0.1
            )
            return resp.status_code == 200
        except:
            return False

    def _emergency_save(self, signum=None, frame=None):
        # Save checkpoint immediately on preemption signal
        self.save_checkpoint(tag="emergency")

    def save_checkpoint(self, tag=None):
        tag = tag or f"step_{self.global_step}"
        path = f"{self.save_dir}/ckpt-{tag}.pt"
        torch.save({
            "model": self.model.state_dict(),
            "optimizer": self.optimizer.state_dict(),
            "step": self.global_step,
        }, path)

    def step(self):
        self.global_step += 1
        if self.global_step % self.interval == 0:
            self.save_checkpoint()
        if self._check_spot_termination():
            self._emergency_save()
Warning: AWS gives 2 minutes; GCP gives only 30 seconds. Your checkpoint save must complete within that window. For large models (70B+), async checkpointing or sharded saves are essential to meet this deadline.

Elastic Training

Elastic training allows the number of workers to change dynamically during a training run—scaling down on preemption and back up when new instances become available. PyTorch's TorchElastic (via torchrun) is the standard framework for this.

The key abstraction is the rendezvous protocol: when the set of workers changes, surviving workers coordinate to form a new process group, re-shard the data, and resume from the last checkpoint.

# Launch elastic training with torchrun
# min 2 workers, max 8, will adapt to available nodes
$ torchrun \
    --nnodes=2:8 \
    --nproc_per_node=4 \
    --rdzv_backend=c10d \
    --rdzv_endpoint=master:29400 \
    --max_restarts=5 \
    train.py

# Inside train.py — elastic-aware training loop
import torch.distributed as dist

def train_elastic(args):
    dist.init_process_group(backend="nccl")
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    model = build_model(args)
    model = torch.nn.parallel.DistributedDataParallel(model, device_ids=[rank])

    # Load latest checkpoint (handles topology changes)
    ckpt = load_latest_checkpoint(args.ckpt_dir)
    if ckpt:
        model.load_state_dict(ckpt["model"])
        start_step = ckpt["step"]

    # DataLoader adapts to current world_size
    sampler = torch.utils.data.DistributedSampler(
        dataset, num_replicas=world_size, rank=rank
    )

    for epoch in range(start_epoch, max_epochs):
        sampler.set_epoch(epoch)
        for batch in dataloader:
            loss = train_step(model, batch)
            # Checkpoint frequently for preemption resilience
            if step % 200 == 0:
                save_checkpoint(model, optimizer, step)
Rendezvous in Detail: The c10d rendezvous backend uses a key-value store (etcd or the master node) to track live workers. When a worker fails, remaining workers detect the failure via NCCL timeout, trigger a new rendezvous, and reform the process group—all automatically with torchrun.

Fault Tolerance Patterns

Beyond checkpointing and elastic scaling, production spot training requires multiple layers of fault tolerance:

Automatic Restart

Wrap your training job in a restart loop. On preemption, the orchestrator (Kubernetes, Slurm, or a custom controller) should automatically request new instances and restart the job from the last checkpoint.

# Kubernetes Job spec with restart policy
apiVersion: batch/v1
kind: Job
metadata:
  name: training-job
spec:
  backoffLimit: 10          # retry up to 10 times
  template:
    spec:
      restartPolicy: OnFailure
      nodeSelector:
        cloud.google.com/gke-spot: "true"
      containers:
      - name: trainer
        resources:
          limits:
            nvidia.com/gpu: 8
        env:
        - name: CHECKPOINT_DIR
          value: "gs://bucket/checkpoints/"

Health Checks & Heartbeats

Each worker periodically sends heartbeat signals to a coordinator. If a heartbeat is missed (e.g., due to silent GPU failure), the coordinator initiates a checkpoint + restart cycle before the training run hangs indefinitely on a stuck all_reduce.

Redundant Workers

Some frameworks (e.g., Bagua, Legate) support redundant gradient computation: extra workers compute the same gradients so that if one is preempted, the training step completes without rollback. This trades extra compute for reduced preemption impact.

Pitfall: Silent data corruption from flaky GPUs is harder to detect than preemption. Implement gradient checksum validation on critical training runs—compare gradient norms across workers and flag outliers.

Cost Optimization

The goal isn't 100% spot—it's finding the mix that minimizes total cost while bounding wall-clock time.

🟢 All Spot

  • Lowest $/GPU-hour
  • High preemption risk
  • Unpredictable wall-clock time
  • Best for: non-urgent experiments, hyperparameter sweeps

🔵 Spot + On-Demand Mix

  • On-demand nodes anchor the training
  • Spot workers provide elastic extra throughput
  • Predictable minimum progress
  • Best for: production training with deadline

Availability Zone Diversification

Spot preemptions often correlate within an AZ. Spreading workers across multiple AZs (or even regions) reduces the probability of a full-cluster preemption. The tradeoff is higher inter-node latency for gradient synchronization.

Instance Selection Strategy

Use instance flexibility—allow your orchestrator to choose from multiple equivalent instance types (e.g., p4d, p4de, p5) to maximize the chance of getting capacity. AWS Fleet API and GCP Instance Groups support this.

Strategy: Capacity-Optimized

Select instances from the pool with the most available capacity. Lower preemption rate but potentially higher price.

Strategy: Lowest-Price

Select the cheapest instances available. Higher preemption rate but minimum cost when running. Good for short, frequently-checkpointed jobs.

Best Practices

Checkpoint Frequency Tuning

The optimal checkpoint interval balances wasted work (longer interval = more lost on preemption) vs checkpoint overhead (shorter interval = more I/O stalls). A good rule of thumb:

# Optimal checkpoint interval formula
# C = checkpoint save time, P = preemption probability per hour
# Minimize: E[wasted_time] = interval/2 * P + C / interval

optimal_interval = sqrt(2 * C / P)

# Example: C = 60s save time, P = 0.1 preemptions/hour
# optimal_interval = sqrt(2 * 60 / 0.1) = ~35 minutes

Grace Period Handling

Always register a SIGTERM handler. On receiving the preemption signal: (1) stop the training loop immediately, (2) save a checkpoint synchronously, (3) flush all metrics/logs, (4) exit cleanly. Do not attempt to finish the current epoch.

Persistent Storage

Never store checkpoints on instance-local storage (ephemeral SSDs). Use networked storage: S3/GCS/Azure Blob for checkpoints, or a shared filesystem (EFS, Filestore, Azure Files) for faster writes. If using local NVMe for speed, async-replicate to durable storage.

Production Checklist:
  • ✅ SIGTERM handler triggers emergency checkpoint
  • ✅ Checkpoints saved to durable networked storage
  • ✅ Orchestrator auto-restarts from latest checkpoint
  • ✅ Elastic training handles worker count changes
  • ✅ Multi-AZ or multi-instance-type for capacity diversity
  • ✅ Monitoring alerts on preemption events and restart failures
  • ✅ Cost dashboard tracking spot savings vs checkpoint overhead
