Testing ML Pipelines: Mocking, Fixtures, Contracts

MLOps Series CI/CD for ML

Testing ML pipelines is harder than testing regular software. Models are non-deterministic, training data is massive, and pipeline components depend on external services (GPUs, model registries, feature stores). This post covers practical strategies to make ML pipelines testable, reliable, and fast to validate in CI.

Testing Challenges

ML pipelines present unique testing challenges that don't exist in traditional software:

  • Non-determinism — model training with random initialization produces different weights each run, making exact output assertions impossible.
  • Data dependency — tests need realistic data, but production datasets are too large for CI and may contain sensitive information.
  • Expensive resources — training tests require GPUs, large memory, and significant compute time.
  • External dependencies — pipelines talk to model registries, feature stores, cloud storage, and monitoring services.
  • Slow feedback — a full training run can take hours, making traditional TDD cycles impractical.
Flaky tests warning: ML pipeline tests are inherently prone to flakiness due to floating-point non-determinism, random initialization, and race conditions in distributed training. Use tolerance-based assertions (e.g., accuracy within ±0.02) rather than exact equality, and seed random number generators in test fixtures.

Mocking ML Services

Mock objects replace external ML services with controlled test doubles. This isolates the component under test and eliminates dependency on external infrastructure.

class MockModelServer: # Test double for a model inference server def __init__(self, responses=None, latency_ms=5): self.responses = responses or {} self.latency_ms = latency_ms self.call_log = [] def predict(self, features): self.call_log.append(features) sleep(self.latency_ms / 1000) # Return a pre-defined response based on input hash key = hash_features(features) if key in self.responses: return self.responses[key] # Default: return a valid but generic prediction return {"score": 0.5, "label": "default", "confidence": 0.8} def assert_called_with(self, expected_features): assert expected_features in self.call_log class MockModelRegistry: # Test double for a model registry (MLflow, etc.) def __init__(self): self.models = {} def register_model(self, name, version, artifact_path): self.models[(name, version)] = artifact_path def load_model(self, name, version): if (name, version) not in self.models: raise ModelNotFoundError(f"{name}:{version}") return FakeModel(self.models[(name, version)])

Mocks are essential for unit tests but dangerous if overused — they can mask real integration issues. Use mocks for unit tests, real services for integration tests.

Test Fixtures

Fixtures provide reproducible test data and environment setup. For ML pipelines, fixtures must handle data generation, model artifacts, and configuration:

import pytest import numpy as np @pytest.fixture def seed_rng(): # Ensure deterministic behavior in tests np.random.seed(42) yield np.random.seed(None) @pytest.fixture def sample_training_data(seed_rng): # Generate small but realistic training data n_samples = 100 features = np.random.randn(n_samples, 10) labels = (features[:, 0] > 0).astype(int) # Deterministic labels return {"features": features, "labels": labels} @pytest.fixture def trained_model(sample_training_data): # Provide a pre-trained model for serving tests model = SimpleClassifier() model.fit(sample_training_data["features"], sample_training_data["labels"]) return model @pytest.fixture def mock_feature_store(): # In-memory feature store for testing store = InMemoryFeatureStore() store.add_feature("user_age", [25, 30, 35, 40, 45]) store.add_feature("user_spend", [100, 250, 500, 150, 300]) return store def test_pipeline_produces_valid_predictions(trained_model, mock_feature_store): pipeline = InferencePipeline(model=trained_model, features=mock_feature_store) result = pipeline.predict(user_id="test_user_1") assert 0 <= result["score"] <= 1 assert result["label"] in ["positive", "negative"]
Data fixtures tip: Store fixture data as small CSV or parquet files in your test directory, not generated programmatically. This makes it easy to inspect, update, and version-control the exact data your tests use.

Contract Testing

Contract tests verify the interface between pipeline components without running the full pipeline. Each component publishes a contract (expected input/output schema) and consumer tests validate against it.

class PipelineContract: # Define the contract between pipeline stages def __init__(self, name, input_schema, output_schema): self.name = name self.input_schema = input_schema self.output_schema = output_schema def validate_input(self, data): for field, dtype in self.input_schema.items(): assert field in data, f"Missing input field: {field}" assert isinstance(data[field], dtype), f"Wrong type for {field}" def validate_output(self, data): for field, dtype in self.output_schema.items(): assert field in data, f"Missing output field: {field}" assert isinstance(data[field], dtype), f"Wrong type for {field}" # Define contracts for each pipeline stage feature_contract = PipelineContract( name="feature_engineering", input_schema={"user_id": str, "timestamp": int}, output_schema={"features": list, "user_id": str} ) model_contract = PipelineContract( name="model_inference", input_schema={"features": list, "user_id": str}, output_schema={"score": float, "label": str, "model_version": str} ) def test_feature_output_matches_model_input(): # Contract test: feature output must satisfy model input sample_output = run_feature_engineering({"user_id": "u1", "timestamp": 1234}) feature_contract.validate_output(sample_output) model_contract.validate_input(sample_output) # Cross-contract check

Integration Patterns

Integration tests verify that pipeline components work together correctly. For ML pipelines, this means testing the full data flow from input to prediction:

class PipelineIntegrationTest: def setup(self): # Spin up lightweight local services self.feature_store = LocalFeatureStore() self.model_server = LocalModelServer(model_path="fixtures/model.pkl") self.pipeline = InferencePipeline( features=self.feature_store, model=self.model_server ) def test_end_to_end_prediction(self): # Load test data into the local feature store self.feature_store.load_fixture("fixtures/test_features.csv") # Run the full pipeline result = self.pipeline.predict(user_id="integration_test_user") # Validate output structure and bounds assert "score" in result assert 0 <= result["score"] <= 1 assert result["model_version"] == "v2.1" def test_pipeline_handles_missing_features(self): # Feature store returns partial data self.feature_store.set_missing(["user_spend"]) result = self.pipeline.predict(user_id="incomplete_user") # Pipeline should gracefully handle missing data assert result["score"] is not None assert result["warnings"].includes("missing_features") def test_pipeline_latency_under_slo(self): start = time_now() for _ in range(100): self.pipeline.predict(user_id="perf_test_user") elapsed = time_now() - start avg_latency = elapsed / 100 assert avg_latency < 0.2 # Under 200ms SLO

Unit Testing (Pipelines)

  • Tests individual functions and transformations
  • Mocks all external dependencies
  • Runs in seconds, no GPU needed
  • Catches logic bugs in data processing
  • Cannot catch integration mismatches

Integration Testing (Pipelines)

  • Tests multiple components together
  • Uses local service doubles or testcontainers
  • Runs in minutes, may need small GPU
  • Catches schema mismatches and data flow bugs
  • More expensive but higher confidence

CI Test Strategies

ML pipeline testing in CI requires a tiered approach that balances speed with confidence:

# .github/workflows/ml-pipeline-tests.yaml (conceptual) stages: fast_checks: # < 2 minutes - lint_pipeline_config() # Validate YAML/config syntax - run_unit_tests() # All unit tests with mocks - run_contract_tests() # Cross-component schema validation - check_data_schema() # Validate fixture data schemas integration_checks: # < 15 minutes - run_integration_tests() # Local service doubles - train_smoke_test() # Train on tiny dataset, verify output - serving_smoke_test() # Load model, run 10 predictions full_validation: # < 2 hours, nightly - train_on_sample_data() # 10% of prod data - evaluate_model_quality() # Check accuracy, precision, recall - run_load_tests() # Verify latency under load - run_data_validation() # Great Expectations / Pandera checks
CI optimization: Run fast checks on every PR, integration checks on merge to main, and full validation nightly. Cache model artifacts and test data between runs. Use test parallelism for independent test suites.

The key insight is that you don't need to run a full training cycle on every commit. A well-structured test pyramid — unit tests at the base, contract tests in the middle, integration tests above, and full E2E runs at the top — gives you fast feedback on most changes while still catching subtle issues in periodic full runs.

Test Maintenance

  • Version fixtures — update test data when the data schema evolves.
  • Track flaky tests — log every non-deterministic test failure and fix the root cause (usually missing random seeds or timing issues).
  • Review test coverage — ensure every pipeline stage has at least unit tests and contract tests.
  • Automate fixture generation — use scripts to generate fresh test data from production samples (anonymized) periodically.
