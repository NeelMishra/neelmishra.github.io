Feature Flags for ML: Gradual Rollouts

MLOps Series CI/CD for ML

Deploying a new ML model to 100% of traffic on day one is a gamble. Feature flags let you control exactly who sees the new model, ramp up traffic gradually, and roll back instantly if metrics degrade — all without redeploying code.

Why Feature Flags for ML

Traditional deployment treats a model release as a binary event: the old model is gone, the new model serves everything. Feature flags introduce a control layer between deployment and activation, providing several advantages for ML systems:

  • Decoupled deployment and release — deploy the model artifact without activating it in production.
  • Gradual exposure — ramp from 1% to 100% over hours or days while monitoring metrics.
  • Instant rollback — flip a flag instead of redeploying the previous model binary.
  • Targeted testing — expose a model only to internal users or a specific cohort before broader release.
  • Multi-model experimentation — run multiple model variants simultaneously under flag control.
Key insight: Feature flags shift risk from deployment time to runtime. You can deploy with confidence because activation is a separate, reversible decision.

Flag Architecture

An ML feature flag system evaluates a context (user ID, request metadata, environment) against flag rules and returns a model variant. The architecture has three layers:

  1. Flag store — persists flag definitions, rules, and targeting configuration.
  2. Evaluator — takes a context and flag key, evaluates rules, and returns a variant.
  3. SDK / client — embedded in the serving layer, calls the evaluator on each request.
class FlagEvaluator: def __init__(self, flag_store): self.store = flag_store self.cache = {} def evaluate(self, flag_key, context): # Fetch flag definition from store (with caching) flag = self.cache.get(flag_key) or self.store.get_flag(flag_key) # Check kill switch first if not flag.enabled: return flag.default_variant # Evaluate targeting rules in priority order for rule in flag.rules: if rule.matches(context): return rule.select_variant(context) # Fall back to percentage rollout return self.percentage_rollout(flag, context)

The evaluator is designed for low latency — flag definitions are cached locally and refreshed on an interval or via streaming updates. Evaluation should add less than 1ms to the request path.

Percentage Rollouts

Percentage rollouts use consistent hashing to assign users deterministically to a variant. The same user always gets the same model, ensuring a stable experience and valid metric comparisons.

import hashlib def percentage_rollout(flag, context): # Deterministic hash based on user_id + flag_key hash_input = f"{context.user_id}:{flag.key}" hash_val = int(hashlib.sha256(hash_input.encode()).hexdigest(), 16) bucket = hash_val % 100 # 0-99 # Walk the variant allocation buckets cumulative = 0 for variant, pct in flag.rollout_percentages.items(): cumulative += pct if bucket < cumulative: return variant return flag.default_variant # Example rollout configuration # rollout_percentages = {"model_v2": 10, "model_v1": 90} # Gradually increase model_v2 from 10 → 25 → 50 → 100

A typical ramp schedule might look like: 1% for 1 hour, 5% for 4 hours, 25% for 1 day, 50% for 2 days, then 100%. At each step, validate latency p99, error rate, and business metrics before proceeding.

Watch out: Changing rollout percentages mid-experiment can shift users between variants. Always use consistent hashing so existing users stay in the same bucket as you ramp up.

User Segmentation

Beyond percentage rollouts, you often want to target specific user segments with different models. Segmentation rules evaluate user attributes against predefined criteria:

class SegmentRule: def __init__(self, attribute, operator, values, variant): self.attribute = attribute # e.g., "plan", "country", "user_id" self.operator = operator # "in", "not_in", "gt", "regex" self.values = values self.variant = variant def matches(self, context): user_val = getattr(context, self.attribute, None) if self.operator == "in": return user_val in self.values if self.operator == "not_in": return user_val not in self.values if self.operator == "gt": return user_val > self.values[0] return False # Target internal testers with the new model internal_rule = SegmentRule("email_domain", "in", ["@company.com"], "model_v2") # Target premium users with a high-accuracy model premium_rule = SegmentRule("plan", "in", ["enterprise", "pro"], "model_v2_premium")

Segmentation is evaluated before percentage rollouts. This lets you guarantee that internal users always see the new model while external users are gradually ramped.

LaunchDarkly Integration

LaunchDarkly provides a managed feature flag platform with SDKs for every major language. Integrating it with an ML serving layer is straightforward:

import ldclient from ldclient.config import Config # Initialize the LaunchDarkly client once at startup ldclient.set_config(Config("sdk-key-production")) client = ldclient.get() def get_model_variant(user_id, user_attrs): # Build the evaluation context context = { "key": user_id, "custom": { "plan": user_attrs.get("plan", "free"), "country": user_attrs.get("country", "US"), "signup_date": user_attrs.get("signup_date"), } } # Evaluate the flag — returns the model variant string variant = client.variation("ml-recommendation-model", context, "model_v1") return variant def serve_prediction(request): variant = get_model_variant(request.user_id, request.user_attrs) model = load_model(variant) return model.predict(request.features)

Feature Flags

  • Controlled by targeting rules and rollout %
  • Deterministic per-user assignment
  • Instant toggle — no redeployment
  • Best for gradual rollouts and kill switches
  • No built-in statistical significance

A/B Testing

  • Random assignment for statistical validity
  • Requires sample size calculation upfront
  • Runs for a fixed duration to reach significance
  • Best for measuring causal impact
  • Built-in hypothesis testing and p-values

In practice, teams often combine both: use feature flags for the deployment mechanics and layer an experimentation platform on top for statistical analysis of the same traffic split.

Custom Flag System

For teams that want full control (or cannot use a third-party service), building a lightweight flag system is feasible. The core components are a flag store, an evaluator, and a sync mechanism:

class FlagStore: # Store flags in a database or config file def __init__(self, db_connection): self.db = db_connection self.local_cache = {} self.last_sync = 0 def get_flag(self, key): if key in self.local_cache: return self.local_cache[key] flag = self.db.query("SELECT * FROM flags WHERE key = ?", key) self.local_cache[key] = flag return flag def sync(self): # Poll for changes every 30 seconds updated = self.db.query( "SELECT * FROM flags WHERE updated_at > ?", self.last_sync ) for flag in updated: self.local_cache[flag.key] = flag self.last_sync = time_now() class MLFlagSystem: def __init__(self, store): self.store = store self.evaluator = FlagEvaluator(store) def get_model(self, flag_key, user_context): variant = self.evaluator.evaluate(flag_key, user_context) return load_model_by_variant(variant) def update_rollout(self, flag_key, new_percentages): # Safely update rollout without downtime self.store.db.execute( "UPDATE flags SET rollout = ? WHERE key = ?", json_encode(new_percentages), flag_key ) self.store.sync()
Technical debt warning: Feature flags accumulate quickly. Set an expiration date on every flag and schedule cleanup sprints. Stale flags with complex targeting rules become a maintenance nightmare — especially in ML systems where model variants may reference deleted artifacts.

A custom system gives you full audit logging, integration with your ML metadata store, and the ability to tie flag changes to model registry events. The trade-off is maintaining the infrastructure yourself.

Flag Lifecycle Best Practices

  • Create — define the flag with a clear owner and expiration date.
  • Ramp — gradually increase the new variant's percentage while monitoring.
  • Complete — once at 100%, mark the flag for cleanup.
  • Remove — delete the flag and all conditional code paths within one sprint.

Feature flags for ML transform model deployment from a high-stakes event into a controlled, observable process. Whether you choose a managed platform like LaunchDarkly or build your own, the pattern is the same: decouple deployment from activation, ramp gradually, and always have an instant off-switch.
