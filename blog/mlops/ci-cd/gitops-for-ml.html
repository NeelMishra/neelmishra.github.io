GitOps for ML: IaC for ML Pipelines

MLOps Series CI/CD for ML

GitOps applies the core Git workflow — pull requests, code review, merge — to infrastructure and application deployment. For ML systems, this means your model serving configuration, pipeline definitions, and infrastructure are all version-controlled, auditable, and reproducible through Git as the single source of truth.

GitOps Principles

GitOps for ML follows four core principles, adapted from the broader GitOps framework:

  1. Declarative configuration — ML infrastructure, pipelines, and model serving are defined as YAML or code, not clicked through a UI.
  2. Git as single source of truth — the desired state of every ML deployment lives in a Git repository.
  3. Automated reconciliation — an agent (ArgoCD, Flux) continuously compares desired state in Git with actual state in the cluster and corrects drift.
  4. Pull-based deployment — instead of CI pushing changes to a cluster, the cluster pulls its own configuration from Git.
Why pull-based? Push-based CI/CD requires your CI system to have cluster credentials. Pull-based GitOps keeps credentials inside the cluster — the agent pulls from Git, reducing the attack surface.

ArgoCD for ML

ArgoCD is a Kubernetes-native GitOps controller. For ML, you define an ArgoCD Application resource that points to your model serving manifests in Git:

# argocd-ml-app.yaml — ArgoCD Application for ML serving apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: ml-recommendation-service namespace: argocd spec: project: ml-production source: repoURL: https://github.com/org/ml-deployments.git targetRevision: main path: services/recommendation/overlays/production destination: server: https://kubernetes.default.svc namespace: ml-serving syncPolicy: automated: prune: true # Remove resources not in Git selfHeal: true # Fix drift automatically syncOptions: - CreateNamespace=true - ApplyOutOfSyncOnly=true

When you merge a PR that changes the model version in your Kustomize overlay, ArgoCD detects the change and automatically rolls out the new model. If the sync fails, ArgoCD reports the error and can auto-rollback.

Flux for ML

Flux is another popular GitOps toolkit, more modular than ArgoCD. It uses custom resources for source tracking, Kustomize, and Helm:

# flux-ml-helmrelease.yaml — Flux HelmRelease for ML serving apiVersion: helm.toolkit.fluxcd.io/v2beta1 kind: HelmRelease metadata: name: ml-serving namespace: ml-production spec: interval: 5m chart: spec: chart: ml-serving-chart version: "2.1.0" sourceRef: kind: HelmRepository name: internal-charts values: model: name: recommendation-v3 registry: s3://ml-models/recommendation/v3 replicas: 4 resources: requests: memory: "4Gi" nvidia.com/gpu: "1" canary: enabled: true weight: 10 # 10% canary traffic

Flux's modular design lets you compose sources, Kustomizations, and Helm releases. For ML, you can have separate Flux resources for the model artifact source, the serving infrastructure, and the monitoring stack.

Declarative ML Infra

GitOps works best when your ML infrastructure is fully declarative. This means defining custom resources (CRDs) that describe your ML deployments in Kubernetes-native terms:

# ml-model-deployment-crd.yaml — Custom Resource Definition apiVersion: ml.example.com/v1 kind: MLModelDeployment metadata: name: recommendation-model namespace: ml-serving spec: modelRef: registry: s3://ml-models name: recommendation version: v3.2.1 serving: framework: triton # triton, seldon, kserve replicas: 4 autoscaling: minReplicas: 2 maxReplicas: 10 targetGPUUtilization: 70 rollout: strategy: canary canaryWeight: 10 analysisTemplate: ml-metrics-check monitoring: metricsEndpoint: /metrics healthEndpoint: /health alertRules: - name: accuracy-drop threshold: 0.85 action: rollback
Drift detection: In ML systems, infrastructure drift is especially dangerous. A manually scaled-up deployment or a hotfixed model version that bypasses Git creates invisible divergence between Git state and cluster state. Enable self-heal in your GitOps agent and alert on every drift event.

Config-Driven Pipelines

Beyond serving infrastructure, your ML training pipelines can also be GitOps-managed. Store pipeline configurations as YAML and let the GitOps agent reconcile them:

# pipeline-config.yaml — declarative pipeline definition apiVersion: ml.example.com/v1 kind: MLPipeline metadata: name: recommendation-training spec: schedule: "0 2 * * 1" # Weekly Monday 2 AM dataSource: type: bigquery query: SELECT * FROM events WHERE date > CURRENT_DATE - 30 training: framework: pytorch image: ml-training:v2.1 hyperparameters: learning_rate: 0.001 batch_size: 256 epochs: 50 resources: gpus: 4 memory: "32Gi" validation: minAccuracy: 0.87 maxLatencyMs: 200 promotion: autoPromote: false # Require PR approval before deploy

Config-driven pipelines mean that changing a hyperparameter is a Git commit, reviewed and approved like any code change. This provides full audit trails and the ability to revert any configuration change.

Push-Based Deployment

  • CI system pushes changes to the cluster
  • CI needs cluster credentials
  • Immediate deployment on CI success
  • No continuous drift detection
  • Examples: Jenkins, GitHub Actions deploy step

Pull-Based Deployment (GitOps)

  • Cluster agent pulls desired state from Git
  • Credentials stay inside the cluster
  • Continuous reconciliation loop
  • Automatic drift detection and self-heal
  • Examples: ArgoCD, Flux, Rancher Fleet

Reconciliation Loop

The reconciliation loop is the heartbeat of GitOps. Every N seconds, the agent compares the desired state in Git with the actual state in the cluster and takes corrective action:

class ReconciliationLoop: def __init__(self, git_source, cluster_client, interval=300): self.git = git_source self.cluster = cluster_client self.interval = interval # 5 minutes def run(self): while True: desired = self.git.get_manifests() actual = self.cluster.get_current_state() diff = self.compute_diff(desired, actual) if diff.has_changes: log(f"Drift detected: {diff.summary}") self.apply_changes(diff) self.emit_event("reconciled", diff) else: log("In sync — no drift detected") sleep(self.interval) def compute_diff(self, desired, actual): # Compare resources by kind, name, namespace changes = [] for resource in desired: current = actual.find(resource.kind, resource.name) if not current: changes.append(Change("create", resource)) elif current.spec != resource.spec: changes.append(Change("update", resource)) return Diff(changes)
ML-specific reconciliation: For ML systems, reconciliation isn't just about Kubernetes resources. You may also need to reconcile model versions in the model registry, feature store schemas, and monitoring alert rules. Extend your GitOps approach to cover the full ML stack.

GitOps for ML transforms infrastructure management from ad-hoc scripts into a disciplined, auditable workflow. Every change goes through a PR, every deployment is traceable to a commit, and drift is automatically detected and corrected. Start with serving infrastructure and expand to training pipelines as your GitOps maturity grows.
