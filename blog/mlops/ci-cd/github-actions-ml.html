GitHub Actions for ML: CI/CD Workflows

MLOps Series CI/CD for ML

Workflow Basics

GitHub Actions is a natural fit for ML CI/CD because it integrates directly with your code repository, supports custom runners with GPU hardware, and provides artifact storage for models and datasets. The workflow file lives alongside your code, making your CI/CD pipeline version-controlled and reviewable.

An ML workflow differs from a typical software workflow in three key ways: it needs large artifacts (datasets, model weights), it requires specialized hardware (GPUs), and its "tests" are threshold-based rather than pass/fail.

# .github/workflows/ml-ci.yaml — basic ML CI workflow name: ML CI Pipeline on: push: branches: [main] pull_request: branches: [main] jobs: lint-and-test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Set up Python uses: actions/setup-python@v5 with: python-version: "3.11" - name: Install dependencies run: pip install -r requirements.txt - name: Lint run: ruff check src/ tests/ - name: Unit tests run: pytest tests/unit/ -v --tb=short - name: Data validation tests run: pytest tests/data/ -v --tb=short
Tip: Split your workflow into separate jobs for linting, unit tests, data tests, and training. This gives you parallel execution and clearer failure signals — you'll immediately know whether a failure is a code issue, data issue, or model issue.

Training Workflows

Training workflows are triggered either by code changes (a new model architecture) or data changes (new training data available). They are the most expensive part of your CI/CD pipeline, often requiring GPU hardware and running for hours.

# .github/workflows/train.yaml — training workflow name: Model Training on: workflow_dispatch: inputs: dataset_version: description: "Dataset version to train on" required: true default: "latest" experiment_name: description: "MLflow experiment name" required: true schedule: - cron: "0 2 * * 1" # Weekly retraining on Monday 2 AM jobs: train: runs-on: self-hosted-gpu timeout-minutes: 360 steps: - uses: actions/checkout@v4 - name: Pull training data run: dvc pull data/ - name: Train model run: | python train.py \ --dataset-version ${{ inputs.dataset_version }} \ --experiment ${{ inputs.experiment_name }} \ --output models/ - name: Upload model artifact uses: actions/upload-artifact@v4 with: name: trained-model path: models/ retention-days: 30

Training workflows should log everything: hyperparameters, data versions, environment details, training curves, and final metrics. This metadata is essential for debugging, comparison, and audit trails.

Evaluation Gates

Evaluation gates are the core of ML CI/CD. They automatically compare a candidate model against the production baseline and block deployment if the candidate doesn't meet performance thresholds. This is the ML equivalent of a failing test suite blocking a merge.

# .github/workflows/evaluate.yaml — evaluation gate name: Model Evaluation Gate on: workflow_run: workflows: ["Model Training"] types: [completed] jobs: evaluate: runs-on: self-hosted-gpu if: ${{ github.event.workflow_run.conclusion == 'success' }} steps: - uses: actions/checkout@v4 - name: Download candidate model uses: actions/download-artifact@v4 with: name: trained-model run-id: ${{ github.event.workflow_run.id }} - name: Run evaluation suite run: | python evaluate.py \ --candidate models/candidate.pt \ --baseline models/production.pt \ --test-data data/test/ \ --output evaluation_report.json - name: Check evaluation gate run: python scripts/check_gate.py evaluation_report.json

The gate-checking script is where thresholds are enforced:

# scripts/check_gate.py import json, sys def check_gate(report_path): with open(report_path) as f: report = json.load(f) candidate = report["candidate_metrics"] baseline = report["baseline_metrics"] gates = { "accuracy": {"min": 0.90, "regression_tolerance": 0.01}, "f1_score": {"min": 0.85, "regression_tolerance": 0.02}, "latency_p99_ms": {"max": 100}, } failures = [] for metric, thresholds in gates.items(): val = candidate[metric] if "min" in thresholds and val < thresholds["min"]: failures.append(f"{metric}={val:.3f} below min {thresholds['min']}") if "max" in thresholds and val > thresholds["max"]: failures.append(f"{metric}={val:.1f} above max {thresholds['max']}") if "regression_tolerance" in thresholds: base_val = baseline[metric] if val < base_val - thresholds["regression_tolerance"]: failures.append(f"{metric} regressed: {val:.3f} vs {base_val:.3f}") if failures: print("GATE FAILED:") for f in failures: print(f" - {f}") sys.exit(1) print("GATE PASSED: Model approved for deployment") check_gate(sys.argv[1])

Deployment Automation

Once a model passes the evaluation gate, deployment automation takes over. This typically involves registering the model, building a serving container, deploying to staging, running smoke tests, and then promoting to production via a canary or blue-green strategy.

# .github/workflows/deploy.yaml — deployment automation name: Model Deployment on: workflow_run: workflows: ["Model Evaluation Gate"] types: [completed] jobs: deploy-staging: runs-on: ubuntu-latest if: ${{ github.event.workflow_run.conclusion == 'success' }} steps: - uses: actions/checkout@v4 - name: Build serving container run: | docker build -t ml-serve:${{ github.sha }} . docker push registry.example.com/ml-serve:${{ github.sha }} - name: Deploy to staging run: kubectl set image deploy/ml-serve ml-serve=registry.example.com/ml-serve:${{ github.sha }} - name: Run smoke tests run: pytest tests/smoke/ --endpoint=https://staging.example.com deploy-production: needs: deploy-staging runs-on: ubuntu-latest environment: production steps: - name: Canary deployment (10% traffic) run: | kubectl set image deploy/ml-serve-canary \ ml-serve=registry.example.com/ml-serve:${{ github.sha }} - name: Monitor canary (15 minutes) run: python scripts/monitor_canary.py --duration=900 - name: Full rollout run: | kubectl set image deploy/ml-serve \ ml-serve=registry.example.com/ml-serve:${{ github.sha }}
GPU runners and cost: GPU-enabled GitHub Actions runners are expensive. Use them only for jobs that truly need GPU (training, GPU inference tests). Run linting, unit tests, and data validation on standard runners. Consider spot instances for training to reduce costs by 60-80%.

Caching Strategies

ML workflows deal with large artifacts: datasets, model weights, pip packages, and Docker layers. Without caching, every workflow run downloads gigabytes of data. Strategic caching can reduce workflow time from hours to minutes.

# Caching strategies in GitHub Actions jobs: train: runs-on: self-hosted-gpu steps: - uses: actions/checkout@v4 # Cache pip dependencies - name: Cache pip uses: actions/cache@v4 with: path: ~/.cache/pip key: pip-${{ hashFiles('requirements.txt') }} restore-keys: pip- # Cache pre-trained model weights (HuggingFace, etc.) - name: Cache model weights uses: actions/cache@v4 with: path: ~/.cache/huggingface key: hf-${{ hashFiles('model_config.yaml') }} restore-keys: hf- # Cache DVC data - name: Cache training data uses: actions/cache@v4 with: path: .dvc/cache key: dvc-${{ hashFiles('data/*.dvc') }} restore-keys: dvc-
Cache key strategy: Use file hashes (like hashFiles('requirements.txt')) as cache keys so the cache invalidates automatically when dependencies change. Always provide restore-keys as fallback to get partial cache hits.

Self-Hosted Runners

For ML workloads, self-hosted runners are often essential. GitHub-hosted runners don't have GPUs, have limited disk space, and may not have the compute needed for training. Self-hosted runners let you control hardware, pre-install heavy dependencies, and persist caches across runs.

GitHub-Hosted Runners

  • Zero maintenance
  • Clean environment every run
  • No GPU support
  • 7 GB RAM / 14 GB disk standard
  • Pay per minute
  • Great for linting, unit tests, docs

Self-Hosted Runners

  • Full hardware control (GPUs, RAM)
  • Persistent environment and caches
  • Custom pre-installed dependencies
  • Higher throughput for large artifacts
  • Requires ops maintenance
  • Essential for training and GPU inference
# Self-hosted runner setup with GPU labels # .github/workflows/gpu-training.yaml name: GPU Training on: workflow_dispatch: jobs: train-on-gpu: runs-on: [self-hosted, gpu, linux] steps: - uses: actions/checkout@v4 - name: Verify GPU run: nvidia-smi - name: Train run: python train.py --gpus=all --mixed-precision - name: Upload metrics run: python scripts/upload_metrics.py lint-and-test: runs-on: ubuntu-latest # Standard runner for cheap tasks steps: - uses: actions/checkout@v4 - name: Lint and unit test run: | pip install -r requirements-dev.txt ruff check src/ pytest tests/unit/ -x

A well-designed ML CI/CD system uses both runner types: GitHub-hosted for cheap, fast jobs (linting, unit tests, docs) and self-hosted for expensive, hardware-dependent jobs (training, GPU evaluation, load tests). Route each job to the right runner to optimize both cost and speed.
