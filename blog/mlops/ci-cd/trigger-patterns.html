Training Triggers: Schedule, Data, Performance

MLOps Series CI/CD for ML

When should you retrain a model? The answer is rarely "on a fixed schedule" alone. Modern ML systems use a combination of time-based, data-driven, and performance-driven triggers to initiate retraining at the right moment — not too early (wasting compute) and not too late (serving stale predictions).

Trigger Overview

A training trigger is a condition or event that initiates the ML training pipeline. Triggers fall into three categories:

  • Schedule-based — retrain on a fixed cadence (daily, weekly) regardless of conditions.
  • Data-driven — retrain when the training data changes significantly.
  • Performance-driven — retrain when model metrics degrade below acceptable thresholds.

Each category has trade-offs between freshness, compute cost, and complexity. Most production systems use a composite approach that combines two or more trigger types.

Schedule Triggers

The simplest trigger: retrain on a fixed schedule. This is the "cron job" approach. It's easy to implement and reason about but can waste compute if data hasn't changed, or miss urgent retraining if data shifts between scheduled runs.

class ScheduleTrigger: def __init__(self, cron_expression, pipeline): self.cron = cron_expression # e.g., "0 2 * * 1" = Monday 2 AM self.pipeline = pipeline self.last_run = None def check(self): if self.cron_matches_now(): # Prevent double-firing within the same window if self.last_run and self.within_window(self.last_run): return False self.last_run = now() self.pipeline.trigger(reason="scheduled") return True return False def cron_matches_now(self): return cron_parse(self.cron).matches(now()) # Common schedules for ML retraining # Daily: "0 2 * * *" — retrain every night at 2 AM # Weekly: "0 2 * * 1" — retrain every Monday at 2 AM # Monthly: "0 2 1 * *" — retrain on the 1st of each month
Tip: Schedule triggers work well as a baseline guarantee. Even if your data-driven triggers are perfect, a weekly scheduled retrain ensures you never go too long without refreshing the model.

Data-Change Triggers

Data-change triggers monitor the training data source and fire when significant new data arrives or when the data distribution shifts. This avoids retraining on stale data and ensures the model stays fresh.

import hashlib class DataChangeTrigger: def __init__(self, data_source, pipeline, threshold=0.05): self.data_source = data_source self.pipeline = pipeline self.threshold = threshold # 5% change threshold self.last_fingerprint = None self.last_row_count = 0 def compute_fingerprint(self): # Hash the schema + sample of data for fast comparison schema = self.data_source.get_schema() sample = self.data_source.sample(n=10000) content = f"{schema}:{sample.to_csv()}" return hashlib.sha256(content.encode()).hexdigest() def check(self): current_count = self.data_source.row_count() current_fp = self.compute_fingerprint() # Check if data volume changed significantly volume_change = abs(current_count - self.last_row_count) / max(self.last_row_count, 1) # Check if data content changed (fingerprint differs) content_changed = current_fp != self.last_fingerprint if volume_change > self.threshold or content_changed: self.last_fingerprint = current_fp self.last_row_count = current_count self.pipeline.trigger(reason=f"data_change: vol={volume_change:.1%}") return True return False

Data fingerprinting catches both volume changes (many new rows) and distribution shifts (same volume but different content). For large datasets, use stratified sampling and statistical tests (KS test, PSI) instead of full-data hashing.

Performance Triggers

Performance triggers monitor production model metrics and fire when quality drops below a threshold. This is the most reactive approach — it catches degradation regardless of cause.

class PerformanceTrigger: def __init__(self, monitor, pipeline, thresholds): self.monitor = monitor self.pipeline = pipeline self.thresholds = thresholds # {"accuracy": 0.85, "latency_p99": 500} def check(self): metrics = self.monitor.get_rolling_metrics(window="1h") violations = [] for metric_name, min_value in self.thresholds.items(): current = metrics.get(metric_name, 0) if metric_name in ["accuracy", "precision", "recall"]: # Higher is better — trigger if below threshold if current < min_value: violations.append(f"{metric_name}={current:.3f} < {min_value}") else: # Lower is better (latency, error rate) — trigger if above if current > min_value: violations.append(f"{metric_name}={current:.1f} > {min_value}") if violations: self.pipeline.trigger(reason=f"perf_degradation: {', '.join(violations)}") return True return False
Beware trigger storms: If a performance trigger fires, retraining starts, but the new model also underperforms, you can enter an infinite retrain loop. Always implement rate limiting — e.g., maximum one retrain per 6 hours — and require the new model to pass validation before deployment.

Composite Triggers

Real systems rarely rely on a single trigger type. Composite triggers combine multiple signals with configurable logic (AND, OR, priority) to make smarter retrain decisions.

class CompositeTrigger: def __init__(self, triggers, mode="any", rate_limit_hours=6): self.triggers = triggers # List of trigger instances self.mode = mode # "any" = OR, "all" = AND self.rate_limit = rate_limit_hours self.last_fire = None def check(self): # Rate limiting: don't fire too often if self.last_fire and hours_since(self.last_fire) < self.rate_limit: return False results = {t.name: t.check() for t in self.triggers} should_fire = ( any(results.values()) if self.mode == "any" else all(results.values()) ) if should_fire: self.last_fire = now() fired = [n for n, v in results.items() if v] log(f"Composite trigger fired by: {fired}") return True return False # Example: retrain if data changed OR performance degraded, # but no more than once every 6 hours trigger = CompositeTrigger( triggers=[schedule_trigger, data_trigger, perf_trigger], mode="any", rate_limit_hours=6 )

Proactive Triggers

  • Fire before performance degrades
  • Based on data changes or schedules
  • Higher compute cost (may retrain unnecessarily)
  • Better freshness guarantee
  • Examples: cron, data fingerprint, data volume

Reactive Triggers

  • Fire after performance degrades
  • Based on metric monitoring
  • Lower compute cost (only retrain when needed)
  • Users experience degradation before fix
  • Examples: accuracy drop, latency spike, drift alert

Trigger Infrastructure

Building reliable trigger infrastructure requires several supporting components beyond the trigger logic itself:

class TriggerInfrastructure: def __init__(self): self.event_bus = EventBus() # Kafka, SQS, or Pub/Sub self.dedup_store = RedisStore() # Prevent duplicate triggers self.audit_log = AuditLogger() # Record all trigger decisions self.circuit_breaker = CircuitBreaker(max_failures=3) def process_event(self, event): # Deduplication: skip if we've seen this event recently event_key = f"{event.type}:{event.source}:{event.timestamp // 3600}" if self.dedup_store.exists(event_key): return # Circuit breaker: stop firing if pipeline keeps failing if self.circuit_breaker.is_open(): log_warning("Circuit breaker open — suppressing trigger") return # Process and log the trigger decision self.dedup_store.set(event_key, ttl=3600) self.audit_log.record(event, decision="fired") self.event_bus.publish("retrain_requested", event)

Key infrastructure concerns include deduplication (the same trigger firing twice should not start two training runs), rate limiting (cap retraining frequency), circuit breaking (stop retrain loops on persistent failures), and audit logging (record every trigger decision for debugging).

Start simple: Begin with a schedule trigger and add data-change and performance triggers incrementally. A weekly cron retrain with manual overrides covers 80% of use cases. Add sophistication only when you have evidence it's needed.
