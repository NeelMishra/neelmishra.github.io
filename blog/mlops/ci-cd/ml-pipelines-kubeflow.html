Kubeflow Pipelines: End-to-End Workflows

MLOps Series CI/CD for ML

Kubeflow Pipelines is a Kubernetes-native platform for building, deploying, and managing ML workflows as directed acyclic graphs (DAGs). Each step runs in its own container, giving you isolation, reproducibility, and scalability. This post walks through the core concepts and practical patterns for building production pipelines with Kubeflow.

Kubeflow Overview

Kubeflow is an open-source ML platform designed to make deploying ML workflows on Kubernetes straightforward. Kubeflow Pipelines is one of its core components, providing:

  • A Python SDK for defining pipelines as code
  • A UI dashboard for visualizing runs, comparing metrics, and browsing artifacts
  • An execution engine (Argo Workflows or Tekton) that schedules containers on Kubernetes
  • A metadata store (ML Metadata / MLMD) for tracking lineage and artifacts

Kubeflow Pipelines is cloud-agnostic — it runs on any Kubernetes cluster (GKE, EKS, AKS, on-prem), which makes it a popular choice for teams that need portability or run in multi-cloud environments.

Kubeflow on different K8s providers: While Kubeflow is cloud-agnostic, the installation complexity varies. GKE has first-party support via the Kubeflow distribution. On EKS and AKS, you'll need to handle storage classes, IAM mappings, and ingress configuration yourself. Consider managed alternatives (Vertex AI, SageMaker) if operational overhead is a concern.

Pipeline SDK

The Kubeflow Pipelines SDK (v2, based on the KFP package) uses Python decorators to define components and pipelines. Each component is a self-contained function that runs inside a container.

# Kubeflow Pipelines v2 SDK — pipeline definition from kfp import dsl from kfp import compiler @dsl.component(base_image="python:3.10", packages_to_install=["pandas", "scikit-learn"]) def preprocess(input_path: str, output_path: dsl.OutputPath(str)): import pandas as pd df = pd.read_csv(input_path) df = df.dropna().reset_index(drop=True) df.to_csv(output_path, index=False) @dsl.component(base_image="python:3.10", packages_to_install=["scikit-learn", "joblib"]) def train(data_path: str, model_path: dsl.OutputPath(str)): import joblib from sklearn.ensemble import RandomForestClassifier # Load data, split, and train model = RandomForestClassifier(n_estimators=100) model.fit(X_train, y_train) joblib.dump(model, model_path) @dsl.component(base_image="python:3.10") def evaluate(model_path: str, test_data: str) -> float: # Load model, compute accuracy on test data return accuracy_score @dsl.pipeline(name="ml-training-pipeline", description="End-to-end training") def training_pipeline(input_data: str): prep_task = preprocess(input_path=input_data) train_task = train(data_path=prep_task.outputs["output_path"]) eval_task = evaluate( model_path=train_task.outputs["model_path"], test_data=prep_task.outputs["output_path"], ) # Compile to YAML for submission compiler.Compiler().compile(training_pipeline, "pipeline.yaml")

Custom Components

When lightweight Python components aren't enough — for example, when you need GPU access, specific system libraries, or pre-built ML frameworks — you create custom container components with their own Dockerfiles.

# Custom container component with GPU @dsl.container_component def train_gpu_model(training_data: dsl.Input[dsl.Dataset], model_out: dsl.Output[dsl.Model]): return dsl.ContainerSpec( image="gcr.io/my-project/gpu-trainer:latest", command=["python", "train.py"], args=[ "--data", training_data.path, "--output", model_out.path, "--epochs", "50", ], ) # Setting resource requirements @dsl.pipeline(name="gpu-training") def gpu_pipeline(data_uri: str): train_task = train_gpu_model(training_data=data_uri) train_task.set_gpu_limit(1) train_task.set_memory_limit("16Gi") train_task.set_cpu_limit("4")

Best practices for custom components:

  • Pin container image tags — never use :latest in production pipelines
  • Keep components small and focused — one responsibility per component
  • Use Kubeflow's input/output artifact types (dsl.Dataset, dsl.Model, dsl.Metrics) for type safety

Caching

Kubeflow Pipelines supports step-level caching. If a component's inputs (parameters and input artifacts) match a previous execution, the cached output is reused instead of re-running the container. This dramatically speeds up iterative development.

# Caching behaviour in KFP v2 @dsl.pipeline(name="cached-pipeline") def pipeline_with_caching(data: str, hparams: str): # This step will be cached if data hasn't changed prep = preprocess(input_path=data) prep.set_caching_options(enable_caching=True) # Training re-runs when hyperparameters change model = train(data_path=prep.outputs["output_path"]) model.set_caching_options(enable_caching=True) # Evaluation always runs fresh score = evaluate(model_path=model.outputs["model_path"]) score.set_caching_options(enable_caching=False)
Warning: Caching is based on input fingerprinting. If your component reads external state (e.g., a database table that changes), caching may serve stale results. Disable caching for components with external side effects.

Artifact Tracking

Kubeflow integrates with ML Metadata (MLMD) to track every artifact produced by pipeline runs — datasets, models, metrics, and visualizations. This provides full lineage: for any model in production, you can trace back to the exact data, code, and parameters that produced it.

# Logging metrics and artifacts in a component @dsl.component(base_image="python:3.10") def evaluate_and_log( model_path: dsl.Input[dsl.Model], test_data: dsl.Input[dsl.Dataset], metrics_out: dsl.Output[dsl.Metrics], ): # Compute evaluation metrics accuracy = compute_accuracy(model_path.path, test_data.path) f1 = compute_f1(model_path.path, test_data.path) # Log to MLMD via the output artifact metrics_out.log_metric("accuracy", accuracy) metrics_out.log_metric("f1_score", f1) metrics_out.log_metric("eval_samples", num_samples)

KFP v1 SDK

  • Components defined via YAML or lightweight functions
  • Argo Workflows as the only backend
  • Artifacts tracked via file paths
  • Limited type checking
  • Widely deployed but now in maintenance mode

KFP v2 SDK

  • Python decorators (@dsl.component, @dsl.pipeline)
  • Supports Argo and Tekton backends
  • First-class artifact types with MLMD integration
  • Strong typing with input/output annotations
  • Actively developed — recommended for new projects

Deployment

Deploying a Kubeflow pipeline involves compiling the Python definition to a portable YAML/JSON spec, then submitting it to the KFP backend via the API or CLI.

# Submitting a pipeline run programmatically from kfp.client import Client client = Client(host="https://kfp.example.com") # Upload the compiled pipeline pipeline = client.upload_pipeline( pipeline_package_path="pipeline.yaml", pipeline_name="ml-training-v2", ) # Create a recurring run (scheduled execution) experiment = client.create_experiment(name="weekly-retraining") client.create_recurring_run( experiment_id=experiment.id, job_name="weekly-retrain-job", pipeline_id=pipeline.id, cron_expression="0 3 * * 0", # Sunday 3 AM params={"input_data": "gs://bucket/data/latest"}, )

Production deployment considerations:

  • Namespace isolation — run pipelines in dedicated Kubernetes namespaces with RBAC controls
  • Resource quotas — set CPU/memory/GPU limits to prevent runaway training jobs from affecting cluster stability
  • Pipeline versioning — maintain semantic versions for pipeline definitions alongside model versions
  • Monitoring — use Prometheus and Grafana to track pipeline execution metrics, step durations, and failure rates
