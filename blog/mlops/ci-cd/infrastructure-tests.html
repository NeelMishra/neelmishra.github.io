Infrastructure Tests for ML Pipelines

MLOps Series CI/CD for ML

Container Tests

ML models run inside containers in production. A container that builds locally but fails in CI, or works on x86 but crashes on ARM, can block deployments for days. Container tests verify that your Docker images build correctly, contain the right dependencies, and can load the model successfully.

# Dockerfile validation tests def test_docker_build(): result = run_command("docker build -t ml-serve:test .") assert result.exit_code == 0, f"Build failed: {result.stderr}" def test_container_has_dependencies(): result = run_in_container("ml-serve:test", "pip list --format=json") packages = json_loads(result.stdout) names = [p["name"] for p in packages] assert "torch" in names assert "numpy" in names assert "fastapi" in names def test_container_model_loading(): result = run_in_container( "ml-serve:test", "python -c 'from model import load; load(\"model.pt\")'" ) assert result.exit_code == 0 assert "error" not in result.stderr.lower() def test_container_image_size(): # Prevent image bloat — large images slow down deployment size_mb = get_image_size("ml-serve:test") assert size_mb < 4000, f"Image size {size_mb}MB exceeds 4GB limit"
Tip: Use multi-stage Docker builds to separate the training environment from the serving environment. Your serving image should contain only inference dependencies — no training frameworks, no Jupyter, no development tools.

Here is an example CI pipeline step for container validation:

# .github/workflows/container-tests.yaml (CI config) name: Container Tests on: [push, pull_request] jobs: build-and-test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Build container run: docker build -t ml-serve:${{ github.sha }} . - name: Test dependencies run: docker run ml-serve:${{ github.sha }} pip check - name: Test model loading run: docker run ml-serve:${{ github.sha }} python -c "import model" - name: Check image size run: | SIZE=$(docker image inspect ml-serve:${{ github.sha }} --format='{{.Size}}') if [ "$SIZE" -gt 4000000000 ]; then echo "Too large"; exit 1; fi

GPU Availability

GPU-dependent pipelines must validate that GPUs are available before starting expensive training or inference jobs. Nothing is worse than a two-hour training job that silently falls back to CPU because CUDA wasn't properly configured.

# GPU availability checks import subprocess def test_cuda_available(): import torch assert torch.cuda.is_available(), "CUDA not available" def test_gpu_count(expected_gpus=1): import torch count = torch.cuda.device_count() assert count >= expected_gpus, \ f"Expected {expected_gpus} GPUs, found {count}" def test_gpu_memory_sufficient(min_gb=8): import torch for i in range(torch.cuda.device_count()): mem = torch.cuda.get_device_properties(i).total_mem mem_gb = mem / (1024 ** 3) assert mem_gb >= min_gb, \ f"GPU {i} has {mem_gb:.1f}GB, need {min_gb}GB" def test_nvidia_driver_version(): result = subprocess.run( ["nvidia-smi", "--query-gpu=driver_version", "--format=csv,noheader"], capture_output=True, text=True ) version = result.stdout.strip() major = int(version.split(".")[0]) assert major >= 525, f"Driver {version} too old, need >= 525"
Infrastructure drift: GPU drivers, CUDA versions, and library versions can drift between your development environment, CI runners, and production servers. Pin exact versions in your container and validate them in tests. A CUDA 12.x training run won't load on a CUDA 11.x serving container.

Endpoint Health

After deploying a model, you need automated health checks that verify the endpoint is alive, responsive, and serving the correct model version. These run as post-deployment verification and feed into your rollback automation.

# Endpoint health verification import requests def test_health_endpoint(base_url): resp = requests.get(f"{base_url}/health", timeout=5) assert resp.status_code == 200 body = resp.json() assert body["status"] == "healthy" assert "model_version" in body def test_readiness_probe(base_url): # Readiness means model is loaded and ready to serve resp = requests.get(f"{base_url}/ready", timeout=10) assert resp.status_code == 200 def test_prediction_returns_valid_response(base_url): payload = {"features": [1.0, 2.0, 3.0, 4.0]} resp = requests.post(f"{base_url}/predict", json=payload, timeout=5) assert resp.status_code == 200 body = resp.json() assert "prediction" in body assert "latency_ms" in body def test_model_version_matches_expected(base_url, expected_version): resp = requests.get(f"{base_url}/health", timeout=5) actual = resp.json()["model_version"] assert actual == expected_version, \ f"Expected model {expected_version}, got {actual}"

Smoke Tests

Smoke tests are lightweight, fast checks that run immediately after deployment. They don't validate correctness deeply — they verify that the system is up, accepts requests, and doesn't crash. Think of them as the "can it breathe?" test before running the full diagnostic suite.

# Post-deployment smoke tests def smoke_test_suite(endpoint_url): tests = { "health": lambda: http_get(f"{endpoint_url}/health"), "single_pred":lambda: http_post(f"{endpoint_url}/predict", json=SAMPLE_INPUT), "batch_pred": lambda: http_post(f"{endpoint_url}/batch", json=SAMPLE_BATCH), "metrics": lambda: http_get(f"{endpoint_url}/metrics"), } results = {} for name, test_fn in tests.items(): try: resp = test_fn() results[name] = resp.status_code == 200 except Exception as e: results[name] = False return results def test_smoke_all_pass(endpoint_url): results = smoke_test_suite(endpoint_url) failed = [k for k, v in results.items() if not v] assert len(failed) == 0, f"Smoke tests failed: {failed}"

Load Tests

Load tests verify that your ML serving infrastructure can handle production traffic levels. ML endpoints are particularly susceptible to load issues because model inference is computationally expensive and memory-intensive.

# Load testing with Locust from locust import HttpUser, task, between class MLEndpointUser(HttpUser): wait_time = between(0.5, 2.0) @task(3) def single_prediction(self): self.client.post("/predict", json={ "features": [1.0, 2.0, 3.0, 4.0] }) @task(1) def batch_prediction(self): batch = [{"features": random_features()} for _ in range(50)] self.client.post("/batch", json=batch) @task(1) def health_check(self): self.client.get("/health") # Assertions on load test results def test_load_results(stats): assert stats.p99_latency_ms < 500, "P99 latency exceeds 500ms" assert stats.error_rate < 0.01, "Error rate exceeds 1%" assert stats.rps >= 100, "Throughput below 100 RPS"

Chaos Engineering

Chaos engineering for ML systems tests resilience to infrastructure failures: what happens when a GPU goes down mid-inference, when the model registry is unreachable, or when a serving replica crashes under load? These tests are run in staging environments to build confidence in production resilience.

# Chaos engineering experiments for ML infrastructure class MLChaosExperiments: def test_gpu_failure_fallback(self, cluster): # Simulate GPU becoming unavailable cluster.disable_gpu(node="serving-node-1") wait(30) # System should route to other GPU nodes resp = http_post(f"{cluster.url}/predict", json=SAMPLE) assert resp.status_code == 200 cluster.enable_gpu(node="serving-node-1") def test_replica_crash_recovery(self, cluster): # Kill a serving replica cluster.kill_pod("ml-serve-abc123") wait(60) # Kubernetes should restart and system should recover resp = http_get(f"{cluster.url}/health") assert resp.status_code == 200 def test_model_registry_outage(self, cluster): # Block access to model registry cluster.block_network(target="registry.internal") # Serving should continue with cached model resp = http_post(f"{cluster.url}/predict", json=SAMPLE) assert resp.status_code == 200 cluster.unblock_network(target="registry.internal")
Start small: Don't jump straight to killing production nodes. Begin chaos experiments in staging with a single failure mode. Verify your monitoring detects the failure and your automation responds correctly. Gradually increase scope as confidence grows.
