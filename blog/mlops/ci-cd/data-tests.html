Data Tests: Validating Training and Serving Data

MLOps Series CI/CD for ML

Why Data Tests Matter

In ML systems, data is code. A subtle shift in a feature distribution can degrade model accuracy far more than a code bug. Yet most teams invest heavily in code testing and almost nothing in data testing. This is the single highest-leverage improvement you can make to ML reliability.

Data issues are insidious because they don't raise exceptions. A column that silently changes from integers to strings, a date field that shifts time zones, or a categorical variable that gains new categories — these pass through your pipeline without errors but produce garbage predictions.

Training-serving skew: The most dangerous data bug in ML. If your serving pipeline preprocesses data differently from your training pipeline, your model will underperform in production even if it aced all offline metrics. Always test that both paths produce identical outputs for identical inputs.

Schema Tests

Schema tests are the first line of defense. They verify that every column exists, has the correct data type, and falls within expected value ranges. Think of schema tests as type checking for your data.

# Schema validation with Pandera import pandera as pa schema = pa.DataFrameSchema({ "user_id": pa.Column(int, nullable=False, unique=True), "age": pa.Column(int, pa.Check.in_range(0, 120)), "income": pa.Column(float, pa.Check.ge(0)), "category": pa.Column(str, pa.Check.isin(["A", "B", "C"])), "timestamp": pa.Column(pa.DateTime), }) def test_schema_compliance(new_batch): validated = schema.validate(new_batch) assert len(validated) == len(new_batch)
Tip: Version your schemas alongside your code. When a legitimate schema change occurs (e.g., a new feature column), update the schema definition in the same PR that updates the pipeline code.

Distribution Tests

Schema tests catch structural breaks; distribution tests catch statistical drift. A column can have the right type and range but contain wildly different values than what the model was trained on. Distribution tests compare incoming data against a known baseline.

# Distribution drift detection import numpy as np from scipy import stats def population_stability_index(baseline, current, bins=10): # PSI measures distribution shift between two datasets base_hist, edges = np.histogram(baseline, bins=bins, density=True) curr_hist, _ = np.histogram(current, bins=edges, density=True) base_hist = np.clip(base_hist, 1e-6, None) curr_hist = np.clip(curr_hist, 1e-6, None) psi = np.sum((curr_hist - base_hist) * np.log(curr_hist / base_hist)) return psi def test_no_significant_drift(baseline_data, new_batch): for col in baseline_data.columns: psi = population_stability_index(baseline_data[col], new_batch[col]) assert psi < 0.25, f"Drift in {col}: PSI={psi:.3f}" def test_ks_statistic(baseline_data, new_batch): for col in ["age", "income", "score"]: stat, pvalue = stats.ks_2samp(baseline_data[col], new_batch[col]) assert pvalue > 0.01, f"KS test failed for {col}: p={pvalue:.4f}"

Freshness Tests

Stale data is a silent killer. If your daily training pipeline runs on yesterday's data, you expect the most recent timestamp to be within 24 hours. If a data source goes down and your pipeline ingests week-old data, the model will be trained on outdated patterns.

from datetime import datetime, timedelta def test_data_freshness(dataset, max_age_hours=24): latest_ts = dataset["timestamp"].max() age = datetime.now() - latest_ts assert age < timedelta(hours=max_age_hours), \ f"Data is {age.total_seconds()/3600:.1f}h old, limit is {max_age_hours}h" def test_no_future_timestamps(dataset): now = datetime.now() future_rows = dataset[dataset["timestamp"] > now] assert len(future_rows) == 0, \ f"Found {len(future_rows)} rows with future timestamps" def test_temporal_coverage(dataset, expected_days=30): # Ensure data covers the expected time range date_range = dataset["timestamp"].max() - dataset["timestamp"].min() assert date_range.days >= expected_days, \ f"Only {date_range.days} days of data, expected {expected_days}"

Completeness Tests

Completeness tests ensure that your dataset has sufficient data across all important dimensions. A dataset might pass schema and distribution checks but be missing entire segments — for example, no data from a specific region, or zero samples for a minority class.

def test_null_rate_within_bounds(dataset, max_null_rate=0.05): for col in dataset.columns: null_rate = dataset[col].isnull().mean() assert null_rate <= max_null_rate, \ f"Column {col} has {null_rate:.1%} nulls (limit: {max_null_rate:.1%})" def test_minimum_class_count(dataset, label_col, min_per_class=100): counts = dataset[label_col].value_counts() for cls, count in counts.items(): assert count >= min_per_class, \ f"Class '{cls}' has only {count} samples (min: {min_per_class})" def test_row_count_in_range(dataset, min_rows=10000, max_rows=10000000): n = len(dataset) assert min_rows <= n <= max_rows, \ f"Dataset has {n} rows, expected [{min_rows}, {max_rows}]"

Batch Validation

  • Run after full dataset is assembled
  • Can compute global statistics
  • Slower but more thorough
  • Ideal for training pipelines
  • Tools: Great Expectations, Pandera

Streaming Validation

  • Run on individual records/micro-batches
  • Only per-record checks (no global stats)
  • Sub-second latency required
  • Ideal for serving pipelines
  • Tools: JSON Schema, Pydantic, Protobuf

Great Expectations & Pandera

Great Expectations (GE) is the most widely adopted data validation framework. It provides a rich library of built-in expectations, data documentation, and integration with orchestrators like Airflow. Pandera offers a more Pythonic, pandas-native approach that works well for smaller-scale validation.

# Great Expectations suite import great_expectations as ge def build_expectation_suite(): suite = ge.ExpectationSuite("training_data_suite") suite.add(ge.expect_column_to_exist("user_id")) suite.add(ge.expect_column_values_to_not_be_null("user_id")) suite.add(ge.expect_column_values_to_be_between( "age", min_value=0, max_value=120 )) suite.add(ge.expect_column_mean_to_be_between( "income", min_value=20000, max_value=200000 )) suite.add(ge.expect_table_row_count_to_be_between( min_value=10000, max_value=5000000 )) return suite def validate_batch(dataframe, suite): result = ge.validate(dataframe, suite) if not result.success: failed = [r for r in result.results if not r.success] raise DataValidationError(failed) return result
Best practice: Run Great Expectations as a pipeline step, not just in tests. Generate data documentation (Data Docs) automatically so that stakeholders can browse validation results in a web UI without reading test logs.
