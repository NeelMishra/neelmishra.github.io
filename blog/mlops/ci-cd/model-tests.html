Model Tests: Performance, Behavioral, Invariance

MLOps Series CI/CD for ML

Threshold Tests

Threshold tests are the most basic model validation: does the model meet minimum performance requirements? These are the gates that prevent a degraded model from reaching production. Every metric that matters to your business should have a hard threshold.

# Threshold tests for model evaluation def test_accuracy_threshold(model, test_data): predictions = model.predict(test_data.features) accuracy = compute_accuracy(predictions, test_data.labels) assert accuracy >= 0.92, f"Accuracy {accuracy:.3f} below threshold 0.92" def test_precision_recall(model, test_data): predictions = model.predict(test_data.features) precision = compute_precision(predictions, test_data.labels) recall = compute_recall(predictions, test_data.labels) assert precision >= 0.88, f"Precision {precision:.3f} too low" assert recall >= 0.85, f"Recall {recall:.3f} too low" def test_beats_production(candidate, production, test_data): # Candidate must not be worse than production model cand_score = evaluate(candidate, test_data) prod_score = evaluate(production, test_data) assert cand_score.f1 >= prod_score.f1 - 0.01, \ f"Candidate F1 {cand_score.f1:.3f} regressed vs production {prod_score.f1:.3f}"
Choosing thresholds: Don't pick thresholds arbitrarily. Base them on the current production model's performance, business requirements, and acceptable degradation margins. A 1% F1 tolerance is common for allowing minor variance without blocking deployment.

Slice-Based Evaluation

Aggregate metrics hide problems. A model with 95% overall accuracy might have 60% accuracy on a critical minority segment. Slice-based evaluation breaks performance down by meaningful subgroups — demographics, geographies, time periods, edge cases — to catch these hidden failures.

# Slice-based evaluation def evaluate_slices(model, test_data, slice_definitions): results = {} for slice_name, slice_fn in slice_definitions.items(): subset = slice_fn(test_data) if len(subset) < 30: continue # skip slices too small for reliable metrics preds = model.predict(subset.features) results[slice_name] = compute_metrics(preds, subset.labels) return results slices = { "age_under_25": lambda d: d[d["age"] < 25], "age_25_to_50": lambda d: d[(d["age"] >= 25) & (d["age"] < 50)], "age_over_50": lambda d: d[d["age"] >= 50], "high_income": lambda d: d[d["income"] > 100000], "low_income": lambda d: d[d["income"] <= 30000], "region_na": lambda d: d[d["region"] == "NA"], "region_eu": lambda d: d[d["region"] == "EU"], } def test_no_slice_regression(model, test_data, min_f1=0.80): results = evaluate_slices(model, test_data, slices) for name, metrics in results.items(): assert metrics.f1 >= min_f1, \ f"Slice '{name}' F1={metrics.f1:.3f} below {min_f1}"

Behavioral Tests

Behavioral tests validate what the model does rather than measuring aggregate statistics. They specify expected behavior for specific inputs, similar to unit tests for traditional software. If a sentiment model sees "This is amazing!", it should predict positive. Period.

# Behavioral tests — Minimum Functionality Tests (MFTs) def test_obvious_positive_sentiment(model): texts = [ "This product is absolutely wonderful!", "I love this so much, best purchase ever.", "Fantastic quality, highly recommend.", ] for text in texts: pred = model.predict(text) assert pred == "positive", f"Failed on: {text}" def test_obvious_negative_sentiment(model): texts = [ "Terrible experience, complete waste of money.", "I hate this product, it broke immediately.", "Worst purchase I have ever made.", ] for text in texts: pred = model.predict(text) assert pred == "negative", f"Failed on: {text}" def test_negation_handling(model): # Model must handle negation correctly assert model.predict("This is not good") == "negative" assert model.predict("This is not bad") == "positive"
Warning: Behavioral tests are not a substitute for metric-based evaluation. They complement it. A model can pass all behavioral tests but still have poor recall on rare cases that weren't anticipated. Use both approaches.

Invariance Tests

Invariance tests verify that the model's prediction doesn't change when irrelevant aspects of the input are modified. If swapping a person's name or changing capitalization flips the prediction, the model has learned a spurious correlation.

# Invariance tests (INV) — prediction shouldn't change def test_name_invariance(model): template = "The service from {name} was excellent." names = ["Alice", "Bob", "Priya", "Mohamed", "Yuki", "Carlos"] predictions = [model.predict(template.format(name=n)) for n in names] assert len(set(predictions)) == 1, \ f"Predictions vary by name: {dict(zip(names, predictions))}" def test_typo_robustness(model): original = "This restaurant has great food and service." typos = [ "This resturant has great food and service.", "This restaurant has gret food and service.", "This restaurant has great food and servce.", ] base_pred = model.predict(original) for t in typos: assert model.predict(t) == base_pred, f"Typo changed prediction: {t}" def test_location_invariance(model): # Sentiment shouldn't depend on city name template = "The hotel in {city} was amazing." cities = ["Paris", "Tokyo", "Lagos", "Mumbai", "Berlin"] preds = [model.predict(template.format(city=c)) for c in cities] assert len(set(preds)) == 1

CheckList Methodology

The CheckList paper (Ribeiro et al., 2020) introduced a systematic framework for behavioral testing of NLP models. It defines three test types that together provide comprehensive coverage of model behavior:

Offline Metrics

  • Aggregate accuracy, F1, AUC
  • Computed on held-out test sets
  • Can hide per-slice failures
  • Easy to game with overfitting
  • Standard but insufficient alone

Behavioral Tests

  • Specific input → expected output
  • Tests model capabilities directly
  • Catches targeted failure modes
  • Hard to game — tests what matters
  • Complements metrics for full coverage

MFT — Minimum Functionality Tests: Simple, unambiguous examples where the expected output is clear. "This movie is amazing" → positive. These are sanity checks: if the model can't get these right, it's fundamentally broken.

INV — Invariance Tests: Perturbations that should not change the output. Swapping names, introducing typos, changing irrelevant context. Failures here indicate the model relies on spurious features.

DIR — Directional Expectation Tests: Perturbations that should change the output in a known direction. Adding "but it was terrible" to a positive review should push sentiment toward negative. The exact prediction doesn't matter, but the direction must be correct.

# Directional Expectation Test (DIR) def test_directional_negation(model): base = "The food was good." modified = "The food was good, but the service was terrible." base_score = model.predict_proba(base)["positive"] mod_score = model.predict_proba(modified)["positive"] assert mod_score < base_score, \ f"Adding negative clause didn't decrease positive score: {base_score:.3f} → {mod_score:.3f}" def test_directional_intensifier(model): base = "This product is good." intensified = "This product is incredibly good." base_score = model.predict_proba(base)["positive"] int_score = model.predict_proba(intensified)["positive"] assert int_score >= base_score
MFT tip: Minimum Functionality Tests should cover the core capabilities your model was built for. If it's a sentiment model, MFTs should include obvious positive, obvious negative, neutral, mixed sentiment, and negation. Think of them as your model's "unit tests."

Regression Tests

Regression tests ensure that model updates don't break previously correct behavior. Every time you find a bug in production — a misclassified example, a failure on a specific input — add it to your regression test suite. Over time, this suite becomes your model's institutional memory.

# Regression test suite — grows over time REGRESSION_CASES = [ # (input, expected_output, ticket) — linked to the bug report ("I didn't not like it", "positive", "BUG-1234"), ("The price was $0.00", "neutral", "BUG-1301"), ("gr8 product!!!", "positive", "BUG-1415"), ("Product arrived 3 weeks late", "negative", "BUG-1502"), ] def test_regression_suite(model): failures = [] for text, expected, ticket in REGRESSION_CASES: pred = model.predict(text) if pred != expected: failures.append((ticket, text, expected, pred)) assert len(failures) == 0, \ f"Regressions found: {failures}" def test_golden_set_stability(model, golden_set): # Golden set: curated examples with verified correct labels preds = [model.predict(x) for x in golden_set.inputs] match_rate = sum(p == g for p, g in zip(preds, golden_set.labels)) / len(preds) assert match_rate >= 0.98, \ f"Golden set match rate {match_rate:.1%} below 98%"
