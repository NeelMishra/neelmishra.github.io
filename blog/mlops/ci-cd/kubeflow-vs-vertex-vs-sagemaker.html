Kubeflow vs Vertex AI vs SageMaker

MLOps Series CI/CD for ML

Choosing an ML pipeline platform is one of the most consequential infrastructure decisions an ML team makes. Kubeflow Pipelines, Vertex AI Pipelines, and SageMaker Pipelines each represent a different trade-off between flexibility, operational overhead, and vendor commitment. This post provides a structured comparison to help you decide.

Comparison Overview

At a high level, the three platforms sit on a spectrum from fully open-source and self-managed (Kubeflow) to fully managed and cloud-specific (Vertex AI, SageMaker).

  • Kubeflow Pipelines — open-source, Kubernetes-native, cloud-agnostic. Maximum flexibility and portability, but you own the infrastructure.
  • Vertex AI Pipelines — Google Cloud's managed pipeline service. Uses the KFP SDK, so definitions are portable, but execution is GCP-only.
  • SageMaker Pipelines — Amazon's native pipeline service. Proprietary SDK with deep SageMaker integration, tightest coupling to AWS.

Vendor Lock-In

Vendor lock-in is the primary strategic consideration. Each platform locks you in at different layers:

  • Kubeflow — no vendor lock-in at the pipeline layer. Runs on any Kubernetes cluster. However, you may still depend on cloud services for storage, compute, and monitoring.
  • Vertex AI — uses the standard KFP v2 SDK, so pipeline definitions are portable. But Google Cloud Pipeline Components (GCPC) tie steps to GCP services. Migration requires replacing GCPC steps.
  • SageMaker — proprietary SDK with SageMaker-specific abstractions (Estimators, ProcessingStep, etc.). The highest lock-in of the three. Migration requires rewriting pipeline definitions.
Migration strategy: If portability matters, write your core ML logic (training, evaluation) as plain Python functions and wrap them in platform-specific pipeline code. This way, migrating between platforms only requires rewriting the orchestration layer, not the ML code itself.

Equivalent pipeline definitions across all three platforms:

# Kubeflow Pipelines (KFP v2) from kfp import dsl @dsl.component(base_image="python:3.10") def train(data: str) -> str: # platform-independent training logic return train_model(data) @dsl.pipeline(name="kubeflow-pipeline") def kf_pipeline(data: str): train(data=data)
# Vertex AI Pipelines (same KFP SDK, GCP submission) from google.cloud import aiplatform aiplatform.init(project="my-project", location="us-central1") job = aiplatform.PipelineJob( display_name="vertex-pipeline", template_path="pipeline.json", # same compiled KFP pipeline ) job.run()
# SageMaker Pipelines (proprietary SDK) from sagemaker.workflow.pipeline import Pipeline from sagemaker.workflow.steps import TrainingStep # Requires SageMaker-specific Estimator and step abstractions train_step = TrainingStep(name="Train", estimator=xgb_estimator, inputs=inputs) pipeline = Pipeline(name="sagemaker-pipeline", steps=[train_step]) pipeline.upsert(role_arn=role)

Flexibility

Flexibility encompasses custom components, non-ML steps, hybrid workloads, and escape hatches when the platform's abstractions don't fit.

  • Kubeflow — maximum flexibility. Any Docker container can be a pipeline step. You control networking, storage, secrets, and scheduling at the Kubernetes level.
  • Vertex AI — high flexibility for ML workloads. Custom container components are supported, but you're constrained to GCP networking and storage. Non-ML steps require workarounds.
  • SageMaker — strong flexibility within the SageMaker ecosystem (custom containers, processing scripts, BYOC). But integrating non-SageMaker services requires Lambda steps or callback steps, adding complexity.

Open-Source (Kubeflow)

  • Full control over infrastructure
  • No vendor lock-in at orchestration layer
  • Community-driven development
  • Higher operational overhead
  • Requires K8s expertise on the team

Managed (Vertex / SageMaker)

  • Zero infrastructure management
  • Deep cloud service integration
  • Vendor-backed support and SLAs
  • Higher per-unit cost
  • Constrained to provider's ecosystem

Cost Analysis

Cost is notoriously hard to compare across platforms because pricing models differ. Here's a framework for thinking about it:

Kubeflow — you pay for the underlying Kubernetes cluster (nodes, storage, networking) plus any cloud services you integrate. No per-pipeline surcharge. Cost is lower at scale but requires dedicated infrastructure expertise.

Vertex AI — charges per pipeline run ($0.03 per run) plus compute costs for custom jobs, training jobs, and endpoints. Caching reduces redundant compute. For teams already on GCP, the marginal cost is modest.

SageMaker — charges for processing and training instance hours. Spot instances can reduce training costs by up to 90%. Model hosting costs are separate. The most expensive option for small-scale experimentation, but competitive at production scale.

Warning: Don't compare list prices alone. Factor in the cost of engineering time for infrastructure management (Kubeflow), the cost of vendor lock-in (Vertex AI, SageMaker), and the cost of migration if you switch platforms later.

Decision Framework

Use the following decision framework to guide your platform choice:

Choose Kubeflow if:

  • You need multi-cloud or on-premises deployment
  • You have strong Kubernetes expertise on the team
  • You need maximum control over the execution environment
  • You want to avoid vendor lock-in at the orchestration layer
  • You're building a platform team that will abstract pipeline infrastructure for other teams

Choose Vertex AI if:

  • You're primarily on Google Cloud
  • You want serverless pipeline execution without managing Kubernetes
  • You need deep integration with BigQuery, Cloud Storage, and Vertex AI services
  • You value the KFP SDK for future portability
  • Your team is small and cannot dedicate resources to infrastructure management

Choose SageMaker if:

  • You're primarily on AWS
  • You need built-in model registry with approval workflows
  • You want tight integration with S3, Glue, Athena, and other AWS data services
  • You need managed spot training for cost optimization
  • Your team is already familiar with the SageMaker ecosystem
# Decision helper — programmatic platform selection def recommend_platform(requirements): if requirements.multi_cloud or requirements.on_prem: return "Kubeflow Pipelines" if requirements.primary_cloud == "GCP": if requirements.k8s_expertise and requirements.max_control: return "Kubeflow on GKE" return "Vertex AI Pipelines" if requirements.primary_cloud == "AWS": if requirements.k8s_expertise and requirements.portability_priority: return "Kubeflow on EKS" return "SageMaker Pipelines" # Azure or other — Kubeflow is the safest bet return "Kubeflow Pipelines"
Final thought: The "best" platform is the one your team can operate reliably. A well-run SageMaker pipeline is better than a poorly maintained Kubeflow cluster. Start with the platform that matches your team's existing skills and cloud commitments, and invest in portability patterns (containerised ML logic, KFP-compatible definitions) to keep your options open.
