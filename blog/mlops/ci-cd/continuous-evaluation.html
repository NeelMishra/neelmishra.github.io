Continuous Evaluation: Production Quality

MLOps Series CI/CD for ML

Offline evaluation tells you how a model might perform. Continuous evaluation tells you how it actually performs — in production, on real traffic, right now. This post covers the infrastructure, patterns, and automation needed to maintain model quality after deployment.

Online Metrics

Online metrics measure model quality on live production data. Unlike offline metrics computed on held-out test sets, online metrics reflect actual user interactions, feedback signals, and real-world distributions.

Common online metrics include:

  • Prediction latency — p50, p95, p99 response times
  • Throughput — predictions per second the model serves
  • Accuracy proxies — click-through rate, conversion rate, or rejection rate as implicit feedback
  • Delayed ground truth — when labels arrive later (e.g., fraud confirmed after investigation)
# Online metric collector class OnlineMetricCollector: def __init__(self, metric_store): self.store = metric_store def record_prediction(self, model_id, request, prediction, latency_ms): self.store.emit({ "model_id": model_id, "timestamp": utcnow(), "latency_ms": latency_ms, "prediction": prediction, "features_hash": hash_features(request), }) def record_outcome(self, prediction_id, ground_truth): # Join prediction with delayed label self.store.update(prediction_id, { "ground_truth": ground_truth, "correct": self.compute_correctness(prediction_id, ground_truth), })
Statistical significance matters: A 1% accuracy improvement on 50 samples is noise. Always compute confidence intervals and run proper hypothesis tests before concluding that one model outperforms another in production.

Shadow Evaluation

Shadow mode (also called dark launching) runs a candidate model alongside the production champion. Both models receive the same requests, but only the champion's predictions are served to users. The challenger's outputs are logged and evaluated offline.

Shadow evaluation is the safest way to test a new model on production traffic — users experience zero risk because the shadow model's predictions never reach them.

# Shadow mode evaluation proxy class ShadowEvaluationProxy: def __init__(self, champion, challenger, logger): self.champion = champion self.challenger = challenger self.logger = logger def predict(self, request): # Champion prediction is served to the user champ_pred = self.champion.predict(request) # Challenger runs in shadow — output is logged, not served try: challenger_pred = self.challenger.predict(request) self.logger.log_shadow({ "request_id": request.id, "champion": champ_pred, "challenger": challenger_pred, "agreement": champ_pred == challenger_pred, }) except Exception as e: self.logger.log_error("shadow_failure", str(e)) return champ_pred # only champion is served

Champion / Challenger

Once shadow evaluation confirms the challenger is viable, you can move to live A/B testing. In the champion/challenger pattern, production traffic is split between the existing model (champion) and the candidate (challenger) at a configurable ratio.

# A/B traffic router for champion/challenger import random class ABRouter: def __init__(self, champion, challenger, challenger_pct=0.10): self.champion = champion self.challenger = challenger self.split = challenger_pct def route(self, request): if random.random() < self.split: pred = self.challenger.predict(request) return {"model": "challenger", "prediction": pred} pred = self.champion.predict(request) return {"model": "champion", "prediction": pred} def promote_challenger(self, analysis_result): if analysis_result.is_significant and analysis_result.challenger_wins: self.champion = self.challenger self.challenger = None return True return False

Offline Evaluation

  • Fast feedback — runs on stored datasets
  • Reproducible — same data, same results
  • No production risk
  • Cannot capture real-world distribution shifts
  • May miss feature pipeline bugs

Online Evaluation

  • Reflects actual production conditions
  • Captures end-to-end system behaviour
  • Includes real user feedback signals
  • Requires sufficient traffic for significance
  • Takes longer to reach conclusions

Quality Gates

Quality gates are automated checkpoints that prevent bad models from reaching production. They compare a candidate model's metrics against thresholds and the current champion, blocking promotion if criteria aren't met.

# Automated quality gate class QualityGate: def __init__(self, thresholds): self.thresholds = thresholds # e.g. {"accuracy": 0.92, "latency_p99": 200} def check(self, candidate_metrics, champion_metrics): failures = [] for metric, threshold in self.thresholds.items(): if candidate_metrics[metric] < threshold: failures.append({ "metric": metric, "required": threshold, "actual": candidate_metrics[metric], }) # Also check regression against champion for metric in ["accuracy", "f1_score"]: if candidate_metrics[metric] < champion_metrics[metric] * 0.98: failures.append({ "metric": metric, "reason": "regression vs champion" }) return len(failures) == 0, failures

Evaluation Infrastructure

Building continuous evaluation requires several infrastructure components working in concert:

  • Prediction logger — captures every prediction with request features, model version, and timestamp
  • Label joiner — matches predictions with delayed ground truth using prediction IDs
  • Metric calculator — computes accuracy, precision, recall, and business metrics on sliding windows
  • Dashboard — visualizes metric trends, drift indicators, and model comparisons over time
  • Experiment tracker — logs A/B test configurations, traffic splits, and statistical test results

The label joiner is often the hardest part. In many domains (fraud detection, ad conversion, medical diagnosis), the true label arrives hours, days, or weeks after the prediction. The infrastructure must handle this temporal gap gracefully.

Warning: Don't forget to account for selection bias in online evaluation. If the model's predictions influence which outcomes are observed (e.g., a fraud model blocks transactions), the evaluation metric can be misleading.

Alerting

Continuous evaluation is only useful if degradation triggers a response. Alerting bridges the gap between metric collection and human or automated action.

# Multi-tier alerting configuration class ModelAlertManager: tiers = { "info": { "condition": lambda m: m.accuracy < 0.95, "action": "log_to_dashboard", }, "warning": { "condition": lambda m: m.accuracy < 0.90, "action": "notify_slack_channel", }, "critical": { "condition": lambda m: m.accuracy < 0.85, "action": "page_oncall_and_trigger_rollback", }, } def evaluate(self, current_metrics): for tier, config in self.tiers.items(): if config["condition"](current_metrics): self.fire_alert(tier, config["action"], current_metrics) return tier return "healthy"

Effective alerting follows these principles:

  • Multi-tier severity — distinguish between informational, warning, and critical alerts to avoid alert fatigue
  • Anomaly-based thresholds — set dynamic thresholds based on historical metric distributions rather than hard-coded values
  • Automated remediation — critical alerts should trigger rollback or traffic rerouting without waiting for human intervention
  • Context-rich notifications — alerts should include the metric value, the threshold, the model version, and a link to the relevant dashboard
