ML Pipeline Design Patterns

MLOps Series CI/CD for ML

ML pipelines are more than linear sequences of steps. Real-world pipelines branch, parallelize, retry, wait for human approval, and handle partial failures. Understanding these design patterns lets you build pipelines that are both powerful and maintainable.

Fan-Out / Fan-In

The fan-out/fan-in pattern splits a pipeline into parallel branches and merges results downstream. This is essential for training multiple model variants, running parallel feature engineering, or evaluating models across different data slices simultaneously.

import concurrent.futures class FanOutFanIn: def __init__(self, tasks, merge_fn, max_workers=4): self.tasks = tasks # List of callable tasks self.merge_fn = merge_fn # Function to combine results self.max_workers = max_workers def execute(self, shared_input): results = {} # Fan-out: run all tasks in parallel with concurrent.futures.ThreadPoolExecutor(self.max_workers) as pool: futures = { pool.submit(task, shared_input): name for name, task in self.tasks.items() } for future in concurrent.futures.as_completed(futures): name = futures[future] results[name] = future.result() # Fan-in: merge all results return self.merge_fn(results) # Usage: train 3 models in parallel, pick the best pipeline = FanOutFanIn( tasks={"xgboost": train_xgb, "rf": train_rf, "nn": train_nn}, merge_fn=lambda r: max(r.values(), key=lambda m: m.score) )
Idempotency matters: Each branch in a fan-out must be idempotent. If a branch fails and is retried, it should produce the same result without side effects. Write outputs to unique paths keyed by run ID and branch name.

Conditional Execution

Not every pipeline step needs to run every time. Conditional execution skips steps based on runtime conditions — saving compute, avoiding unnecessary retraining, or branching logic based on data characteristics.

class ConditionalStep: def __init__(self, name, condition_fn, action_fn, skip_value=None): self.name = name self.condition = condition_fn self.action = action_fn self.skip_value = skip_value def execute(self, context): if self.condition(context): log(f"Running step: {self.name}") return self.action(context) log(f"Skipping step: {self.name}") return self.skip_value # Only retrain if data has changed significantly retrain_step = ConditionalStep( name="retrain_model", condition_fn=lambda ctx: ctx.data_drift_score > 0.1, action_fn=lambda ctx: train_model(ctx.dataset), skip_value="model_unchanged" ) # Only run GPU evaluation if model complexity exceeds threshold gpu_eval = ConditionalStep( name="gpu_evaluation", condition_fn=lambda ctx: ctx.model.param_count > 1_000_000, action_fn=lambda ctx: evaluate_on_gpu(ctx.model), )

Retry Patterns

ML pipelines interact with external services (cloud storage, GPU clusters, API endpoints) that can have transient failures. Retry patterns add resilience without manual intervention.

import time, random class RetryWithBackoff: def __init__(self, max_retries=3, base_delay=1.0, max_delay=60.0): self.max_retries = max_retries self.base_delay = base_delay self.max_delay = max_delay def execute(self, fn, *args): for attempt in range(self.max_retries + 1): try: return fn(*args) except TransientError as e: if attempt == self.max_retries: raise # Exhausted all retries # Exponential backoff with jitter delay = min( self.base_delay * (2 ** attempt) + random.uniform(0, 1), self.max_delay ) log(f"Retry {attempt+1}/{self.max_retries} in {delay:.1f}s: {e}") time.sleep(delay) # Wrap flaky operations with retry retry = RetryWithBackoff(max_retries=3, base_delay=2.0) model_artifact = retry.execute(download_model, "s3://bucket/model.tar.gz")

Human-in-the-Loop

Some pipeline transitions require human judgment — approving a model for production, reviewing data quality reports, or signing off on compliance checks. Approval gates pause the pipeline until a human acts.

class ApprovalGate: def __init__(self, name, approvers, timeout_hours=24): self.name = name self.approvers = approvers self.timeout = timeout_hours def request_approval(self, context): # Create a review request with model metrics request = { "gate": self.name, "model_version": context.model_version, "metrics": context.eval_metrics, "approvers": self.approvers, "expires_at": now_plus_hours(self.timeout), } self.notify_approvers(request) return self.wait_for_decision(request) def wait_for_decision(self, request): # Poll or subscribe to approval events while not is_expired(request): decision = self.check_approval_status(request) if decision == "approved": return True if decision == "rejected": raise ApprovalRejected(request) sleep(60) raise ApprovalTimeout(request)
Avoid bottlenecks: Approval gates can stall your pipeline for hours or days. Set reasonable timeouts, auto-approve in non-production environments, and track time-spent-in-gate as a metric.

Error Handling

ML pipelines must distinguish between recoverable errors (transient network issues), non-recoverable errors (corrupt data), and partial failures (some branches succeed, others fail).

class PipelineErrorHandler: def handle(self, step_name, error, context): if isinstance(error, TransientError): # Retriable — schedule for retry return Action("retry", delay=self.backoff(context.attempt)) if isinstance(error, DataCorruptionError): # Non-recoverable — fail the pipeline, alert on-call self.alert_oncall(step_name, error) return Action("fail") if isinstance(error, PartialFailure): # Some branches failed — continue with successful ones log_warning(f"{step_name}: partial failure, continuing with {error.succeeded}") return Action("continue", data=error.succeeded) # Unknown error — fail safely return Action("fail")

Composition Patterns

Complex pipelines are built by composing simpler patterns. Two dominant approaches exist:

Orchestration

  • Central coordinator controls execution order
  • Explicit workflow definition (DAG)
  • Easy to reason about and debug
  • Single point of failure at the orchestrator
  • Examples: Airflow, Kubeflow, Prefect

Choreography

  • Steps react to events independently
  • No central coordinator — event-driven
  • Highly decoupled and scalable
  • Harder to trace end-to-end execution
  • Examples: Kafka, AWS EventBridge, Pub/Sub
# Composing patterns into a full pipeline class MLTrainingPipeline: def run(self, config): # Step 1: Load and validate data data = RetryWithBackoff().execute(load_data, config.source) # Step 2: Conditional feature engineering features = ConditionalStep( "feature_engineering", condition_fn=lambda c: c.needs_refresh, action_fn=lambda c: compute_features(data) ).execute(config) # Step 3: Fan-out training across model types models = FanOutFanIn( tasks=config.model_configs, merge_fn=select_best_model ).execute(features) # Step 4: Human approval gate for production ApprovalGate("prod-deploy", ["ml-lead@co.com"]).request_approval(models) # Step 5: Deploy with error handling try: deploy_model(models.best) except Exception as e: PipelineErrorHandler().handle("deploy", e, config)

The key to maintainable ML pipelines is treating each pattern as a composable building block. Keep individual steps small, idempotent, and independently testable. The pipeline orchestrator's job is to wire them together and handle the failure modes between them.
