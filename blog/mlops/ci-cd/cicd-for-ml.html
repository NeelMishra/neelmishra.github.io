CI/CD for ML: Beyond Software Pipelines

MLOps Series CI/CD for ML

Why ML CI/CD Is Harder

Traditional CI/CD pipelines deal with a single artifact: code. You write it, test it, build it, and deploy it. Machine learning pipelines, however, produce three coupled artifacts — data, model, and code — each with its own lifecycle, validation requirements, and failure modes.

A code change in a traditional service is deterministic: the same input produces the same output. ML models are stochastic: retrain with the same data and you may get different weights. This fundamentally changes how we think about testing, versioning, and rollback.

Key insight: In ML systems, a "passing build" doesn't just mean the code compiles — it means the data is valid, the model meets performance thresholds, and the serving infrastructure can handle the model's resource requirements.

Consider the blast radius of a single failure. A bug in traditional software usually manifests immediately as an error. A bug in ML data processing may silently degrade model accuracy for weeks before anyone notices. This demands a fundamentally different approach to continuous integration.

Three Pipeline Types

Google's MLOps whitepaper identifies three levels of ML maturity, but at the core every ML system must manage three distinct pipelines that interleave and depend on each other.

Traditional CI/CD

  • Single artifact (code)
  • Deterministic builds
  • Fast feedback loops (seconds)
  • Rollback = previous binary
  • Tests are pass/fail

ML CI/CD

  • Three artifacts (data + model + code)
  • Stochastic training
  • Slow feedback loops (hours/days)
  • Rollback = previous model + data snapshot
  • Tests are threshold-based

Each pipeline type has unique triggers, validation steps, and deployment patterns. A change in data may trigger model retraining, which may require infrastructure scaling, which may need code changes to the serving layer. Understanding these dependencies is the first step toward robust ML CI/CD.

Data Pipeline

The data pipeline handles ingestion, validation, transformation, and feature engineering. It is often the most fragile part of the ML system because upstream data sources change without warning.

# Data pipeline configuration class DataPipeline: def __init__(self, sources, schema, freshness_sla): self.sources = sources self.schema = schema self.freshness_sla = freshness_sla def validate(self, dataframe): # Schema validation assert check_schema(dataframe, self.schema) # Distribution drift detection drift_score = compute_psi(dataframe, self.baseline) if drift_score > 0.25: alert("Distribution drift detected", drift_score) # Freshness check assert data_age(dataframe) < self.freshness_sla def transform(self, raw_data): features = apply_feature_engineering(raw_data) return normalize(features)
Warning: Never skip data validation even when "nothing changed." Upstream schemas drift silently — a vendor adds a column, a partner changes date formats, or a service starts sending nulls where it didn't before.

Model Pipeline

The model pipeline covers training, evaluation, registration, and promotion. Unlike code artifacts, models must pass performance gates before deployment — you wouldn't ship a model that's worse than what's already in production.

# Model pipeline with evaluation gates class ModelPipeline: def train(self, train_data, config): model = build_model(config.architecture) model.fit(train_data, epochs=config.epochs) return model def evaluate(self, model, test_data, thresholds): metrics = compute_metrics(model, test_data) # Gate: must beat production model prod_metrics = get_production_metrics() for metric, threshold in thresholds.items(): assert metrics[metric] >= prod_metrics[metric] - threshold def register(self, model, metrics): log_to_registry(model, metrics, timestamp=now()) return create_model_version(model)

Model pipelines are inherently slower than code pipelines. Training can take hours or days, which means your CI feedback loop is fundamentally different. This is why many teams adopt an asynchronous evaluation pattern: merge the training code, trigger training, and gate deployment on the resulting metrics.

Code Pipeline

The code pipeline is the most familiar — linting, unit tests, integration tests, and deployment of the serving infrastructure. But even here, ML adds complexity: you need to test that your serving code correctly loads and invokes the model, handles edge cases in input preprocessing, and meets latency requirements.

# Code pipeline tests for ML serving def test_model_loading(): model = load_model("models/v2.3.1") assert model is not None assert model.input_shape == (None, 224, 224, 3) def test_preprocessing_consistency(): # Serving preprocessing must match training train_output = training_preprocess(sample_image) serve_output = serving_preprocess(sample_image) assert np_allclose(train_output, serve_output, atol=1e-6) def test_latency_budget(): start = time() result = predict(sample_input) elapsed = time() - start assert elapsed < 0.100 # 100ms budget

ML CI/CD Maturity Levels

Google's MLOps maturity model defines three levels that describe how automated and integrated your ML CI/CD processes are:

Level 0 — Manual Process: Data scientists train models in notebooks, hand off serialized files to engineers, and deployment is a manual, error-prone process. There's no pipeline, no automation, and no monitoring.

Level 1 — ML Pipeline Automation: Training is automated end-to-end. Data validation, feature engineering, training, and evaluation run as a pipeline triggered by data changes or schedules. But CI/CD for the pipeline code itself is still manual.

Level 2 — CI/CD Pipeline Automation: Both the ML pipeline and its code are under CI/CD. Code changes trigger automated tests, pipeline validation, and deployment. This is the target state for production ML systems.

Practical advice: Most teams should aim for Level 1 first. The jump from Level 0 to Level 2 is too large. Automate training and evaluation, then layer on code CI/CD once the pipeline is stable.
# Maturity level check — a simple self-assessment def assess_maturity(team): checks = { "level_0": [ team.trains_in_notebooks, team.manual_deployment, ], "level_1": [ team.has_training_pipeline, team.auto_data_validation, team.auto_evaluation_gates, ], "level_2": [ team.pipeline_code_tested, team.auto_pipeline_deployment, team.monitoring_and_rollback, ], } for level, criteria in checks.items(): if all(criteria): current = level return current
