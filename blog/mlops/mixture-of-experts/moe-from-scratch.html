Mixture of Experts from Scratch

Build the mental model and a minimal PyTorch MoE layer, then understand exactly where sparsity saves compute and where it creates systems complexity.

April 2026 · 34 min read MLOps · Mixture of Experts · Part 1

Why Mixture of Experts exists

A dense Transformer spends the same feed-forward compute on every token. That is simple, stable, and GPU-friendly, but it ties model capacity to per-token FLOPs. If you double the FFN width, every token pays the price. A Mixture-of-Experts (MoE) layer breaks that link: it keeps many FFN blocks available, then activates only a small subset for each token.

Dense FFN

One shared MLP handles all tokens. Every token uses all parameters in the layer.

MoE FFN

A router picks k experts out of E. Every token uses only the selected experts.

Tradeoff

More parameters and specialization, but routing, load balance, communication, and serving get harder.

The practical promise is straightforward: train a model with the knowledge capacity of a very large dense network while keeping the active compute closer to a smaller model. The cost is that the model is no longer a uniform matrix-multiply pipeline; tokens branch to different experts.

From dense FFN to sparse experts

A standard Transformer block contains attention followed by an FFN. In an MoE Transformer, attention usually remains dense and shared. The FFN is replaced by a bank of experts:

Dense block: token -> attention -> one FFN -> residual
MoE block:   token -> attention -> router -> top-k FFNs -> weighted sum -> residual

Each expert is often architecturally identical to the dense FFN: a projection up, activation, projection down. The router is much smaller: usually a single linear layer mapping the token hidden state to expert logits.

The MoE equation

For token representation x, expert functions E_i, router logits r(x), and selected experts S(x), the output is:

MoE(x) = ∑i ∈ S(x) softmax(r(x))i · Ei(x)

Top-1 routing chooses one expert and is cheap. Top-2 routing combines two expert outputs and is often more stable because a token can share gradient signal across multiple experts.

Implementation invariant: the output shape of an MoE FFN must exactly match the dense FFN shape: (batch, seq, d_model). All routing complexity is internal to the layer.

Implementation from scratch

This minimal PyTorch implementation favors clarity over speed. It shows the full algorithm: flatten tokens, compute router probabilities, select top-k experts, run each expert on the tokens assigned to it, and scatter weighted outputs back to the original order.

import torch
import torch.nn as nn
import torch.nn.functional as F

class Expert(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.GELU(),
            nn.Linear(d_ff, d_model),
        )

    def forward(self, x):
        return self.net(x)

class NaiveMoE(nn.Module):
    def __init__(self, d_model, d_ff, num_experts=8, top_k=2):
        super().__init__()
        self.num_experts = num_experts
        self.top_k = top_k
        self.router = nn.Linear(d_model, num_experts, bias=False)
        self.experts = nn.ModuleList([Expert(d_model, d_ff) for _ in range(num_experts)])

    def forward(self, x):
        batch, seq, d_model = x.shape
        tokens = x.reshape(batch * seq, d_model)

        logits = self.router(tokens)                  # (T, E)
        probs = F.softmax(logits, dim=-1)
        top_prob, top_idx = torch.topk(probs, self.top_k, dim=-1)
        top_prob = top_prob / top_prob.sum(dim=-1, keepdim=True)

        out = torch.zeros_like(tokens)
        for expert_id, expert in enumerate(self.experts):
            token_pos, slot = torch.where(top_idx == expert_id)
            if token_pos.numel() == 0:
                continue
            expert_in = tokens[token_pos]
            expert_out = expert(expert_in)
            weight = top_prob[token_pos, slot].unsqueeze(-1)
            out.index_add_(0, token_pos, expert_out * weight)

        return out.reshape(batch, seq, d_model), logits

Shape trace

TensorShapeMeaning
x(B, S, D)Input hidden states
tokens(T, D)Flattened tokens, where T = B × S
logits(T, E)Router scores for each expert
top_idx(T, k)Selected expert ids per token
out(T, D)Weighted sum of selected expert outputs

Debug checklist

  • Verify every token receives at least one expert output.
  • Check that top-k probabilities are renormalized after selection.
  • Log expert token counts; uniform routing is not guaranteed without auxiliary losses.
  • Start with small tensors and compare a loop implementation against a vectorized one.
  • Keep router logits in FP32 when training in mixed precision if instability appears.
