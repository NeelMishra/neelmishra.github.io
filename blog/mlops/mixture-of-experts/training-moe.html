Training MoE Models from Scratch

Turn an MoE layer into a trainable model: losses, optimizer groups, mixed precision, stability guardrails, and a compact training loop.

April 2026 · 35 min read MLOps · Mixture of Experts · Part 4

Loss composition

An MoE model optimizes the normal task loss plus routing regularizers. The task loss teaches the model what to predict. The auxiliary losses teach the router how to use experts without creating hot spots.

total_loss = task_loss
       + aux_coef * load_balance_loss
       + z_coef * router_z_loss

Start with small coefficients. A common debugging pattern is to train a tiny dense baseline, replace only one FFN with MoE, then confirm that task loss still decreases while expert counts remain non-zero.

Optimizer groups

Router parameters often benefit from separate treatment. They are small, sensitive, and directly control load balance. You may want lower learning rate, no weight decay, or FP32 precision for the router.

def make_optimizer(model, base_lr=3e-4, router_lr=1e-4):
router_params, other_params = [], []
for name, param in model.named_parameters():
    if not param.requires_grad:
        continue
    if "router" in name:
        router_params.append(param)
    else:
        other_params.append(param)

return torch.optim.AdamW([
    {"params": other_params, "lr": base_lr, "weight_decay": 0.1},
    {"params": router_params, "lr": router_lr, "weight_decay": 0.0},
], betas=(0.9, 0.95))

Training loop

A clean MoE layer should return both output and routing statistics. The model can aggregate auxiliary losses from every MoE block.

for step, batch in enumerate(loader):
optimizer.zero_grad(set_to_none=True)

logits, moe_stats = model(batch["input_ids"])
task_loss = F.cross_entropy(
    logits[:, :-1].reshape(-1, logits.size(-1)),
    batch["input_ids"][:, 1:].reshape(-1),
)

aux = sum(s["load_balance_loss"] for s in moe_stats) / len(moe_stats)
z = sum(s["router_z_loss"] for s in moe_stats) / len(moe_stats)
loss = task_loss + 0.01 * aux + 0.001 * z

loss.backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
optimizer.step()
Sanity test: overfit a tiny dataset before scaling. If a small MoE cannot memorize a few batches, distributed training will only hide the bug behind noise.

Stability guardrails

  • Warm up router noise: noise helps exploration early but can hurt late convergence.
  • Clip gradients: sparse expert updates can be spiky.
  • Monitor dropped tokens: persistent overflow means the capacity factor or load-balance coefficient is wrong.
  • Keep router math stable: compute softmax logits in FP32 under mixed precision.
  • Avoid instant large expert counts: scale from 4 to 8 to 16 experts while watching routing metrics.

Evaluation

Evaluate both model quality and routing health. Two checkpoints can have similar validation loss but very different serving properties if one uses experts evenly and the other overloads a subset.

Experiment plan

For a from-scratch project, run controlled ablations:

  1. Dense baseline with identical attention and embedding stack.
  2. MoE with top-1 routing and no auxiliary loss.
  3. MoE with top-2 routing and load-balancing loss.
  4. Capacity factor sweep: 1.0, 1.25, 1.5, 2.0.
  5. Router z-loss on/off under BF16 mixed precision.
