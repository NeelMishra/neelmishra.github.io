MoE Inference and Serving Systems

Sparse activation lowers per-token compute, but serving still has to place experts, route requests, batch tokens, and avoid hot expert bottlenecks.

April 2026 · 32 min read MLOps · Mixture of Experts · Part 5

The serving paradox

MoE inference activates only a few experts per token, but the serving system must have access to all experts because routing is input-dependent. That creates the MoE serving paradox: FLOPs are sparse, memory is not.

Rule of thumb: MoE reduces compute per token more reliably than it reduces deployment memory. Plan capacity for total parameters, not just active parameters.

Expert placement

There are three common placement strategies:

StrategyBest forRisk
Replicate all experts on every GPUSmall MoE, low latencyWastes memory
Shard experts across GPUsLarge MoERequires token all-to-all
Hybrid hot expert replicationSkewed routing trafficPlacement policy complexity

When experts are sharded, each GPU owns a subset of experts. Tokens must be exchanged so each expert receives its assigned tokens, then outputs must be sent back to the original sequence owner.

Batching and hot experts

Continuous batching improves dense LLM serving by merging decode steps across requests. MoE adds a second batching dimension: tokens must also be grouped by expert. If one expert is consistently hot, it becomes the latency bottleneck for the whole batch.

def route_decode_step(tokens, router):
weights, experts, logits = top_k_router(tokens, router, k=2)
groups = defaultdict(list)
for token_id, expert_choices in enumerate(experts.tolist()):
    for slot, expert_id in enumerate(expert_choices):
        groups[expert_id].append((token_id, slot))
return groups, weights

Serving systems often cap per-expert batch sizes, replicate hot experts, or bias routing away from overloaded experts when quality allows.

Latency model

A useful first-order latency model is:

latency = router + dispatch + max(expert_compute) + combine + communication

The max(expert_compute) term matters: a batch completes when the slowest expert group completes. Perfect average utilization is not enough; tail expert load controls latency.

Quantization

MoE models are strong candidates for weight-only quantization because inactive expert weights still consume memory. Quantizing experts from BF16 to INT8 or INT4 can reduce residency cost substantially. Keep router precision higher if routing quality regresses.

  • Quantize expert FFN weights first; they dominate parameter count.
  • Keep shared attention and router in BF16 during early experiments.
  • Measure per-expert activation ranges; experts may have different distributions.
  • Validate routing-sensitive tasks after quantization, not only perplexity.

Operational checklist

  • Expose expert utilization, per-expert latency, and token overflow metrics.
  • Plan checkpoint loading for many expert shards; cold start can dominate deploy time.
  • Test adversarial prompts that route disproportionately to one expert.
  • Use canary deployments because router behavior can shift traffic shape.
  • Benchmark prefill and decode separately; MoE bottlenecks differ across phases.
  • Document expert placement so on-call engineers can reason about GPU failures.
