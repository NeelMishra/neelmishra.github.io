Routers, Top-k Gating, and Load Balancing

Routers decide which experts learn. This post builds top-k gating, capacity limits, auxiliary losses, z-loss, and practical telemetry for routing health.

April 2026 · 38 min read MLOps · Mixture of Experts · Part 2

The router job

The router is the small network that controls the large network. Its output determines which experts receive tokens, which experts get gradients, and which experts specialize. A bad router can collapse the entire MoE layer into one busy expert and many idle experts.

Failure mode: routing collapse is self-reinforcing. If one expert starts winning, it receives more updates, improves faster, and becomes even more attractive to the router.

A production MoE implementation therefore treats routing as both an ML problem and a systems scheduling problem. It must select useful experts while keeping GPU work balanced.

Top-k gating

Top-k gating computes a categorical distribution over experts, keeps only the largest k entries, and renormalizes the selected probabilities. During training, small router noise can improve exploration.

def top_k_router(hidden, router, k, noise_std=0.0):
logits = router(hidden.float())
if noise_std > 0:
    logits = logits + torch.randn_like(logits) * noise_std

scores = torch.softmax(logits, dim=-1)
weights, indices = torch.topk(scores, k, dim=-1)
weights = weights / weights.sum(dim=-1, keepdim=True).clamp_min(1e-9)
return weights.to(hidden.dtype), indices, logits

Top-1 routing maximizes sparsity. Top-2 routing gives smoother optimization because the second expert still receives gradient signal. Top-k above 2 is usually expensive unless the model is small or the serving latency budget is generous.

Expert capacity

Experts run most efficiently when each expert receives a bounded number of tokens. Capacity is the maximum tokens an expert processes in one MoE layer call:

capacity = ceil(capacity_factor × tokens × top_k / num_experts)

If more tokens choose an expert than capacity allows, implementations either drop overflow tokens, reroute them to backup experts, or process them with slower dynamic shapes. Dropping is common in early systems because it preserves static buffers.

def build_expert_slots(top_idx, num_experts, capacity):
# top_idx: (tokens, k)
counters = torch.zeros(num_experts, dtype=torch.long, device=top_idx.device)
slots = torch.full_like(top_idx, -1)
keep = torch.zeros_like(top_idx, dtype=torch.bool)

for token in range(top_idx.shape[0]):
    for choice in range(top_idx.shape[1]):
        expert = top_idx[token, choice]
        slot = counters[expert]
        if slot < capacity:
            slots[token, choice] = slot
            keep[token, choice] = True
            counters[expert] += 1
return slots, keep, counters

Load-balancing loss

The standard auxiliary loss compares two distributions: how often experts are selected and how much router probability mass they receive. If either distribution is skewed, the loss increases.

def load_balancing_loss(router_probs, expert_indices, num_experts):
# router_probs: (tokens, experts)
# expert_indices: (tokens, k)
tokens = router_probs.shape[0]
density_proxy = router_probs.mean(dim=0)                 # probability mass

selected = torch.zeros_like(router_probs)
selected.scatter_add_(1, expert_indices, torch.ones_like(expert_indices, dtype=router_probs.dtype))
density = selected.mean(dim=0) / expert_indices.shape[1] # assignment fraction

return (density * density_proxy).sum() * num_experts

The coefficient is usually small. If it is too high, routing becomes uniform even when specialization would help. If it is too low, hot experts dominate.

Router z-loss

The router z-loss penalizes large log-sum-exp values. It keeps logits numerically tame, which matters in BF16/FP16 training where softmax saturation can silently destroy gradient quality.

def router_z_loss(logits):
z = torch.logsumexp(logits.float(), dim=-1)
return torch.mean(z ** 2)

Routing telemetry

Track routing health from the first training step. Useful dashboards include:

  • Tokens per expert: catches hot experts and idle experts.
  • Overflow rate: percentage of selected routes dropped by capacity.
  • Router entropy: low entropy means confident routing; extremely low early entropy hints at collapse.
  • Expert gradient norms: detects dead experts that receive no meaningful updates.
  • Auxiliary loss trend: confirms balancing pressure is active but not dominating task loss.
