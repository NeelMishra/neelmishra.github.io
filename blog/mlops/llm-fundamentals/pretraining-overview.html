Pre-Training LLMs: Data, Compute & Curriculum

July 2025 LLM Fundamentals

Pre-training is the most expensive and least-forgiving phase of building a large language model. A single run of Llama 3 405B consumed 30.84 million GPU-hours on 16,384 H100 GPUs over roughly 54 days. Getting the data mix, compute budget, and learning schedule right before you press "go" is not optional—mistakes compound into months of wasted compute. This post covers the full pipeline from raw web crawls to a trained base model, with concrete numbers you would need to reason about in a systems design interview at Anthropic or OpenAI.

1 · Pre-Training Data Sources

Modern frontier models train on multi-trillion-token corpora assembled from diverse sources. The quality, diversity, and licensing profile of each source directly shapes model capability.

SourceTypical Token ShareWhy It Matters
Common Crawl~50-60%Broadest coverage of internet text. Raw dumps are ~250 billion pages/month. After filtering, yields trillions of unique tokens spanning almost every domain.
Books (BookCorpus, Gutenberg, etc.)~10-15%Long-range coherence, narrative structure, formal prose. Critical for reasoning chains that span thousands of tokens.
Code (GitHub, StackOverflow)~10-15%Dramatically improves reasoning, chain-of-thought, and structured output. Code is highly deduplicated compared to web text.
Wikipedia + Reference~3-5%Factual grounding, encyclopedic knowledge, multi-lingual coverage. Small in volume but high information density.
Scientific Papers (arXiv, PubMed)~3-5%Technical reasoning, mathematical notation, domain-specific vocabulary.
Curated / Synthetic~5-10%Instruction-style data, math problems, multi-turn dialogue. Increasingly used even in pre-training to improve downstream alignment.
Interview insight: The Chinchilla paper (Hoffmann et al., 2022) showed that most LLMs were severely under-trained relative to their parameter count. Llama 3 trained a 70B model on 15 trillion tokens—over 30× the Chinchilla-optimal ratio—because inference cost scales with parameters, not training tokens. This is why data quality and quantity dominate modern pre-training strategy.

2 · Data Processing Pipeline

Raw web data is unusable without heavy processing. The pipeline below runs at petabyte scale and typically takes weeks on hundreds of CPU nodes.

Stage-by-Stage Breakdown

1. Language Identification & Extraction: Raw HTML is stripped to text via tools like trafilatura or resiliparse. A fastText classifier identifies the language. Non-target languages and garbled text are dropped. This alone removes 60-70% of raw data.

2. Quality Filtering: A lightweight classifier (often logistic regression or small fastText model) trained on "high-quality" vs "low-quality" examples scores each document. Documents below a threshold are discarded. Heuristic rules also filter: minimum word count, maximum symbol ratio, repetition checks (e.g., a page that repeats "buy now" 50 times).

3. Deduplication: Near-duplicate removal is essential. Without it, the model memorises popular pages verbatim. Two common approaches:

Exact Dedup (URL/Hash)

  • SHA-256 of normalised text
  • O(N) with hash table
  • Misses near-duplicates (boilerplate variants)
  • Cheap, run first

Fuzzy Dedup (MinHash LSH)

  • Jaccard similarity on n-gram shingles
  • O(N log N) with LSH bands
  • Catches near-duplicates across domains
  • Expensive, run second on surviving docs

4. Toxicity & PII Removal: Classifier-based toxicity scoring (e.g., Perspective API or custom models) flags harmful content. Regex + NER models detect and redact personal information (emails, phone numbers, SSNs). This is both a safety and legal requirement.

# Simplified data processing pipeline
import hashlib
from datasketch import MinHash, MinHashLSH

def process_document(doc: str) -> dict | None:
    # Stage 1: Language filter
    lang, confidence = fasttext_detect(doc)
    if lang != target_lang or confidence < 0.65:
        return None

    # Stage 2: Quality score
    quality = quality_classifier(doc)
    if quality < QUALITY_THRESHOLD:
        return None

    # Stage 3: Exact dedup
    doc_hash = hashlib.sha256(doc.encode()).hexdigest()
    if doc_hash in seen_hashes:
        return None
    seen_hashes.add(doc_hash)

    # Stage 4: Toxicity filter
    toxicity = toxicity_score(doc)
    if toxicity > TOXICITY_THRESHOLD:
        return None

    # Stage 5: PII redaction
    doc = redact_pii(doc)

    return {"text": doc, "hash": doc_hash, "quality": quality}


def fuzzy_dedup_batch(docs: list, threshold=0.8):
    # MinHash LSH for near-duplicate detection
    lsh = MinHashLSH(threshold=threshold, num_perm=128)
    unique = []
    for i, doc in enumerate(docs):
        m = MinHash(num_perm=128)
        for shingle in ngrams(doc["text"], n=5):
            m.update(shingle.encode())
        if not lsh.query(m):
            lsh.insert(f"doc_{i}", m)
            unique.append(doc)
    return unique
Data quality > data quantity. The Phi series (Microsoft) demonstrated that a carefully curated 1.4T-token corpus of "textbook-quality" data can train a 2.7B model that outperforms models 5× larger trained on noisier data. Every filtering decision has downstream capability implications.

3 · Compute Requirements & Scaling Laws

The foundational formula for estimating total training compute is deceptively simple:

C ≈ 6 × N × D

Where C is total floating-point operations (FLOPS), N is the number of model parameters, and D is the number of training tokens. The factor of 6 comes from 2× for the forward pass (multiply-accumulate) and 4× for the backward pass (gradient computation + parameter update).

The Chinchilla Scaling Laws

Hoffmann et al. (2022) established that for a given compute budget C, the loss-optimal allocation is:

Nopt ∝ C0.50   &   Dopt ∝ C0.50

This means parameters and tokens should scale equally with compute. The practical takeaway: a 10× bigger compute budget should yield roughly a 3.16× bigger model trained on 3.16× more tokens.

# Compute budget estimation
def estimate_training_flops(n_params: int, n_tokens: int) -> float:
    # C ≈ 6 * N * D (Kaplan et al., 2020)
    return 6 * n_params * n_tokens

def estimate_gpu_hours(flops: float, gpu_tflops: float, mfu: float) -> float:
    # MFU = Model FLOPS Utilization (typically 0.30-0.55)
    # gpu_tflops: peak BF16 TFLOPS (e.g., H100 = 989 TFLOPS)
    effective_tflops = gpu_tflops * mfu
    seconds = flops / (effective_tflops * 1e12)
    return seconds / 3600

# Example: Llama 3 405B
flops = estimate_training_flops(405e9, 15.6e12)  # ≈ 3.8e25
gpu_hrs = estimate_gpu_hours(flops, gpu_tflops=989, mfu=0.38)
print(f"Single-GPU hours: {gpu_hrs:,.0f}")   # ≈ 28.3M GPU-hours
print(f"With 16384 GPUs: {gpu_hrs/16384/24:.0f} days")  # ≈ 72 days

def chinchilla_optimal(compute_budget_flops: float):
    # Equal scaling: N_opt ∝ C^0.5, D_opt ∝ C^0.5
    # With 6ND = C → N ≈ (C/6)^0.5 tokens ≈ N
    n_opt = (compute_budget_flops / 6) ** 0.5
    d_opt = n_opt  # ~20 tokens per parameter
    return n_opt, d_opt
Model FLOPS Utilization (MFU) is the ratio of achieved FLOPS to hardware peak. State-of-the-art training achieves 35-55% MFU. The gap comes from communication overhead (AllReduce, pipeline bubbles), memory bandwidth limits, and kernel inefficiencies. Llama 3 reported ~38-43% MFU on H100 clusters. Improving MFU by just 5% on a 16k-GPU run saves millions of dollars.

4 · Training Schedule & Curriculum

The curriculum—the order and mix of data shown during training—and the learning rate schedule together determine convergence quality.

Learning Rate Schedule

The near-universal recipe for LLM pre-training:

Warmup (0.1-0.5% of total steps): Linearly increase learning rate from ~0 to peak η. This stabilises early training when gradients are noisy and the model's weights are randomly initialised. Typical warmup: 2,000 steps for large models.

Cosine Decay (remaining steps): Smoothly anneal the learning rate following a cosine schedule down to ηmin (usually 10% of peak). This prevents sharp loss spikes and allows fine-grained convergence in the final phase.

import math

def cosine_lr_schedule(step, total_steps, warmup_steps, lr_max, lr_min):
    # Phase 1: Linear warmup
    if step < warmup_steps:
        return lr_max * (step / warmup_steps)

    # Phase 2: Cosine decay
    progress = (step - warmup_steps) / (total_steps - warmup_steps)
    cosine_decay = 0.5 * (1 + math.cos(math.pi * progress))
    return lr_min + (lr_max - lr_min) * cosine_decay

# Llama 3 405B schedule
lr_max = 8e-5
lr_min = 8e-6   # 10% of peak
warmup_steps = 2000
total_steps = 300_000  # ~15.6T tokens / (16M tokens/step)

Data Curriculum

Modern pre-training increasingly uses phased curricula rather than a fixed data mix:

Phase% of TrainingData Mix ChangeRationale
Phase 1: Foundation0-80%Standard web + book + code mixBuild broad language understanding
Phase 2: Upsampling80-95%Increase code, math, reasoning dataSharpen specific capabilities
Phase 3: Annealing95-100%High-quality curated + synthetic data, aggressive LR decayLlama 3 showed a 10% boost on benchmarks from annealing alone
Annealing matters. Llama 3 performed data annealing in the final 5% of training: they upsampled high-quality sources, turned off data parallelism for certain domains, and aggressively decayed the learning rate. This final phase alone improved GSM8K (math) by ~10% and MBPP (code) by ~6%.

Batch Size Schedule

Batch size is often ramped during training. Llama 3 started with 4M tokens/batch and gradually increased to 16M tokens/batch. The reasoning: early in training, gradient noise is beneficial (acts as regularisation), but later, larger batches produce more stable gradients for fine-tuning the loss landscape.

5 · Ops Challenges of Multi-Month Runs

Running 16,384 GPUs for 54+ days continuously is an engineering feat. Here are the challenges an MLOps/infra team faces:

Hardware Failures

At the scale of Llama 3 training, Meta reported 466 job interruptions during the 54-day run, of which 47 were planned and 419 were unexpected. That is roughly 8 unexpected failures per day. Sources include:

  • GPU failures: Memory ECC errors, thermal throttling, NVLink failures
  • Network failures: InfiniBand link flaps, switch firmware bugs, cable degradation
  • Host failures: Kernel panics, NVMe failures, PSU faults
  • Software issues: NCCL hangs, CUDA OOM from fragmentation, silent data corruption
Ops reality check: Meta's Llama 3 paper reveals that <3% of 419 unexpected interruptions were due to confirmed GPU hardware faults. The majority were caused by software bugs, network issues, and cascading failures from a single node affecting the entire training job. Robust health-checking and fast node replacement are more important than buying more reliable GPUs.

Checkpointing Strategy

A checkpoint for a 405B model includes optimizer states and is ~4-5TB. The checkpointing strategy must balance:

# Checkpointing strategy for large-scale training
class CheckpointManager:
    def __init__(self):
        self.ckpt_interval = 500         # Save every 500 steps (~30 min)
        self.async_save = True           # Don't block training
        self.keep_last_n = 5             # Rolling window of recent ckpts
        self.milestone_interval = 5000   # Permanent saves every 5000 steps
        self.storage_backend = "S3"      # Fast parallel upload

    def save(self, step, model, optimizer):
        # Each rank saves its shard independently
        shard = get_local_shard(model, optimizer)
        path = f"s3://training-ckpts/step_{step}/rank_{get_rank()}"

        # Async I/O: overlap save with next training step
        if self.async_save:
            future = async_upload(shard, path)
            self.pending_saves.append(future)
        else:
            blocking_upload(shard, path)

        # Garbage collection for rolling window
        if step % self.ckpt_interval == 0:
            cleanup_old_checkpoints(self.keep_last_n)

Monitoring & Observability

Key metrics the ops team watches during a training run:

  • Training loss curve: Sudden spikes indicate data issues or hardware failures
  • Gradient norm: Exploding gradients require intervention (clipping, LR reduction)
  • MFU (Model FLOPS Utilization): Drops indicate stragglers or communication bottlenecks
  • GPU utilization & memory: Per-GPU monitoring for early fault detection
  • Network throughput: Inter-node bandwidth for detecting degraded links
  • Loss spikes: May require rollback to a previous checkpoint and data inspection

Loss Spike Recovery

Loss spikes are a notorious problem in large-scale training. The typical recovery procedure:

  1. Detect spike via automated monitoring (loss > 2σ above running average)
  2. Automatically halt training
  3. Roll back to the last known-good checkpoint (typically 100-500 steps back)
  4. Skip or remove the offending data batch (suspected data quality issue)
  5. Optionally reduce learning rate by 10-20% for the next few thousand steps
  6. Resume training and monitor convergence
# Automated loss spike detection and recovery
class LossSpikeDetector:
    def __init__(self, window=100, sigma_threshold=2.5):
        self.window = window
        self.sigma_threshold = sigma_threshold
        self.loss_history = []

    def check(self, current_loss, step):
        self.loss_history.append(current_loss)
        if len(self.loss_history) < self.window:
            return False

        recent = self.loss_history[-self.window:]
        mean = sum(recent) / len(recent)
        std = (
            sum((x - mean) ** 2 for x in recent) / len(recent)
        ) ** 0.5

        if current_loss > mean + self.sigma_threshold * std:
            log_alert(f"Loss spike at step {step}: {current_loss:.4f}")
            return True
        return False

Key Takeaways for Interviews

Data Questions

  • C ≈ 6ND — know this formula cold
  • Web data dominates but needs aggressive filtering
  • Dedup is critical: near-dupes cause memorisation
  • Modern models train well beyond Chinchilla-optimal to reduce inference cost
  • Data quality > data quantity (Phi series proof)

Ops Questions

  • 16K-GPU runs see ~8 failures/day
  • Checkpointing every 500 steps with async I/O
  • MFU of 35-55% is state-of-the-art
  • Loss spikes require automated detection + rollback
  • Most failures are software/network, not GPU hardware
The meta lesson: Pre-training an LLM is equal parts machine learning and distributed systems engineering. The data team, the training framework team, and the infrastructure/ops team must work in tight coordination for months. A 5% improvement in MFU or a 10% improvement in data quality can save tens of millions of dollars at frontier scale.
