Instruction Tuning: SFT Data, Formatting & Best Practices

MLOps Series LLM Fundamentals

What Is Instruction Tuning?

Pre-trained language models learn to predict the next token from massive text corpora. While they acquire broad knowledge, they do not inherently know how to follow human instructions. Instruction tuning—often called supervised fine-tuning (SFT)—bridges that gap by training the model on curated (instruction, response) pairs so it learns to produce helpful, structured answers to user requests.

SFT sits in the middle of the modern LLM training pipeline:

The key insight: SFT teaches the model what format to use and how to behave, while RLHF/DPO later refines the quality and safety of those behaviors. In practice, a high-quality SFT stage is often the single biggest lever for model usefulness.

Interview insight: Pre-training gives the model knowledge; SFT gives the model behavior. A well-instruction-tuned model with 7B parameters often outperforms a raw 70B base model on user-facing tasks.

SFT vs Few-Shot Prompting

Few-shot prompting places examples directly in the context window at inference time. SFT permanently bakes the instruction-following behavior into the model weights. SFT is preferred when you need consistent behavior, lower latency (no prompt overhead), or domain-specific formatting that few-shot cannot reliably replicate.

SFT Data Format

Instruction-tuning datasets follow a structured schema. The canonical format popularized by the Stanford Alpaca project uses three fields:

Alpaca Format (3-field)

{
  "instruction": "Summarize the following article.",
  "input": "The Federal Reserve announced...",
  "output": "The Fed raised rates by 25bp..."
}

instruction: the task description.
input: optional context (can be empty).
output: the ideal model response.

Conversational Format (multi-turn)

{
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user",   "content": "Explain gradient descent."},
    {"role": "assistant", "content": "Gradient descent is..."}
  ]
}

This is the modern standard used by OpenAI, Anthropic, and HuggingFace TRL. Each sample is a full conversation.

Loss Masking

During SFT training, the loss is typically computed only on the assistant tokens. User and system tokens are masked so the model is not penalized for predicting those. This is critical—without loss masking, the model wastes capacity learning to reproduce system prompts and user queries rather than focusing on generating good responses.

Key concept: In the conversational format, each sample can contain multiple turns. The model learns from all assistant turns in the conversation, enabling multi-turn coherence.

Chat Templates

A chat template converts structured messages (system, user, assistant) into a single string the tokenizer can process. Different model families use different template formats. Using the wrong template causes severe quality degradation because the model cannot parse its own role boundaries.

ChatML (OpenAI / Qwen / Mistral-Instruct)

ChatML uses explicit <|im_start|> and <|im_end|> delimiters with the role name on the first line: 

# ChatML template
<|im_start|>system
You are a helpful coding assistant.<|im_end|>
<|im_start|>user
Write a Python function to reverse a string.<|im_end|>
<|im_start|>assistant
def reverse_string(s: str) -> str:
    return s[::-1]<|im_end|>

Llama 3 Format (Meta)

Llama 3 uses <|begin_of_text|>, <|start_header_id|>, <|end_header_id|>, and <|eot_id|> tokens: 

# Llama 3 chat template
<|begin_of_text|><|start_header_id|>system<|end_header_id|>

You are a helpful coding assistant.<|eot_id|>
<|start_header_id|>user<|end_header_id|>

Write a Python function to reverse a string.<|eot_id|>
<|start_header_id|>assistant<|end_header_id|>

def reverse_string(s: str) -> str:
    return s[::-1]<|eot_id|>

Anthropic Messages API Format

Anthropic (Claude) uses a structured JSON API rather than a raw text template. The system prompt is a top-level field, and the conversation alternates between user and assistant roles: 

# Anthropic API request structure
{
  "model": "claude-sonnet-4-20250514",
  "max_tokens": 1024,
  "system": "You are a helpful coding assistant.",
  "messages": [
    {"role": "user", "content": "Write a Python function to reverse a string."},
    {"role": "assistant", "content": "Here is a concise implementation..."}
  ]
}
Warning: Using the wrong chat template at inference time is one of the most common failure modes. If you fine-tune with ChatML but serve with Llama format, the model will produce incoherent outputs. Always match the template to the model family.

Jinja2 Templates in HuggingFace

HuggingFace tokenizers store chat templates as Jinja2 strings in tokenizer_config.json. The apply_chat_template() method renders messages into the model’s expected format automatically: 

from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3-8B-Instruct")

messages = [
    {"role": "system", "content": "You are helpful."},
    {"role": "user",   "content": "What is SFT?"},
]

prompt = tokenizer.apply_chat_template(messages, tokenize=False)
print(prompt)
# Output: <|begin_of_text|><|start_header_id|>system...

Data Quality vs Quantity

One of the most important findings in instruction-tuning research is that data quality far outweighs data quantity. The LIMA paper (Zhou et al., 2023) demonstrated that fine-tuning LLaMA-65B on just 1,000 carefully curated examples produced results competitive with GPT-4 on many benchmarks.

✅ High-Quality Data Signals

  • Written or verified by domain experts
  • Diverse task coverage (not repetitive)
  • Responses are detailed, correct, and well-structured
  • Consistent formatting and tone
  • Appropriate length (neither truncated nor padded)

❌ Low-Quality Data Signals

  • Auto-generated without human review
  • Duplicate or near-duplicate examples
  • Factually incorrect or hallucinated responses
  • Inconsistent style (mixing formal/informal)
  • Responses that are too short or generic

Data Curation Strategies

Practical approaches for building a high-quality SFT dataset:

  1. Seed with expert-written examples — Start with 500–2,000 gold-standard samples that define the target behavior precisely.
  2. Use a strong teacher model — Generate candidate responses with GPT-4 or Claude, then have humans verify and edit rather than write from scratch.
  3. Deduplication — Remove near-duplicates using embedding similarity (cosine > 0.95) to maximize diversity.
  4. Difficulty balancing — Include a mix of easy, medium, and hard tasks. Models that only see easy examples struggle with complex reasoning.
  5. Negative examples — Include examples of what not to do (e.g., refusals for harmful prompts) to teach safety boundaries.
Rule of thumb: 1,000 high-quality examples > 100,000 noisy examples. Start small and iterate. Measure on a held-out eval set before scaling data collection.

Formatting SFT Data in Python

Below is a practical pipeline for converting raw data into the conversational format expected by modern SFT frameworks.

Converting Alpaca to Conversational Format

import json

def alpaca_to_messages(example):
    """Convert Alpaca-format dict to messages list."""
    messages = []

    # Add system prompt
    messages.append({"role": "system", "content": "You are a helpful assistant."})

    # Build user message: instruction + optional input
    user_content = example["instruction"]
    if example.get("input", "").strip():
        user_content += f"\n\nInput: {example['input']}"
    messages.append({"role": "user", "content": user_content})

    # Add assistant response
    messages.append({"role": "assistant", "content": example["output"]})

    return {"messages": messages}


# Load and convert dataset
with open("alpaca_data.json") as f:
    raw_data = json.load(f)

converted = [alpaca_to_messages(ex) for ex in raw_data]
print(f"Converted {len(converted)} examples")

Deduplication with Embeddings

import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")

def deduplicate(examples, threshold=0.95):
    """Remove near-duplicate examples using cosine similarity."""
    # Extract instruction text for embedding
    texts = [ex["messages"][1]["content"] for ex in examples]
    embeddings = model.encode(texts, normalize_embeddings=True)

    # Compute pairwise cosine similarity
    sim_matrix = embeddings @ embeddings.T

    keep = set(range(len(examples)))
    for i in range(len(examples)):
        if i not in keep:
            continue
        for j in range(i + 1, len(examples)):
            if sim_matrix[i][j] > threshold:
                keep.discard(j)

    return [examples[i] for i in sorted(keep)]

clean_data = deduplicate(converted)
print(f"Kept {len(clean_data)}/{len(converted)} after dedup")

Saving to HuggingFace Datasets Format

from datasets import Dataset

# Create HuggingFace Dataset from list of dicts
dataset = Dataset.from_list(clean_data)

# Train/test split
split = dataset.train_test_split(test_size=0.05, seed=42)
print(f"Train: {len(split['train'])}, Eval: {len(split['test'])}")

# Push to Hub (optional)
split.push_to_hub("my-org/my-sft-dataset", private=True)

Training with TRL SFTTrainer

HuggingFace’s TRL (Transformer Reinforcement Learning) library provides the SFTTrainer class, which wraps the standard Trainer with SFT-specific features: chat template application, loss masking on non-assistant tokens, packing, and LoRA integration.

Full SFTTrainer Example

from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from trl import SFTTrainer, SFTConfig
from peft import LoraConfig
from datasets import load_dataset

# ── 1. Load model with 4-bit quantization ──
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype="bfloat16",
)

model_id = "meta-llama/Llama-3-8B"
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    quantization_config=bnb_config,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained(model_id)
tokenizer.pad_token = tokenizer.eos_token

# ── 2. Load dataset ──
dataset = load_dataset("my-org/my-sft-dataset")

# ── 3. LoRA configuration ──
peft_config = LoraConfig(
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    task_type="CAUSAL_LM",
)

# ── 4. SFT training configuration ──
sft_config = SFTConfig(
    output_dir="./sft-llama3-8b",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,        # effective batch = 16
    learning_rate=2e-4,
    lr_scheduler_type="cosine",
    warmup_ratio=0.1,
    max_seq_length=2048,
    packing=True,                             # pack short examples together
    logging_steps=10,
    save_strategy="steps",
    save_steps=200,
    bf16=True,
    gradient_checkpointing=True,
)

# ── 5. Train ──
trainer = SFTTrainer(
    model=model,
    args=sft_config,
    train_dataset=dataset["train"],
    eval_dataset=dataset["test"],
    processing_class=tokenizer,
    peft_config=peft_config,
)

trainer.train()
trainer.save_model("./sft-llama3-8b-final")
Key hyperparameters explained:
  • packing=True — concatenates short examples into single sequences to avoid wasted padding tokens. Can yield 2–3× throughput improvements.
  • max_seq_length — truncates or packs up to this length. Set based on your data distribution (check the 95th percentile token count).
  • learning_rate=2e-4 — standard for LoRA SFT. Full fine-tuning typically uses 1e-5 to 5e-5.
  • gradient_checkpointing — trades compute for memory by recomputing activations during the backward pass.

Common Pitfalls & Best Practices

Overfitting on Small Datasets

SFT datasets are often small (1K–100K examples). With standard fine-tuning, models can memorize training examples within a single epoch. Mitigations:

  • Use LoRA or QLoRA to reduce trainable parameters
  • Train for 1–3 epochs maximum; monitor eval loss for early stopping
  • Use dropout (0.05–0.1) in LoRA layers
  • Apply weight decay (0.01–0.1)

Catastrophic Forgetting

Aggressive SFT can cause the model to forget pre-trained knowledge. Strategies to mitigate:

  • Low learning rate for full fine-tuning (1e-5 to 5e-5)
  • LoRA inherently limits forgetting by keeping most weights frozen
  • Replay buffer — mix a small percentage (5–10%) of general-purpose instruction data with your domain-specific data
Common mistake: Training on only domain-specific data without any general instruction-following examples. The model becomes a narrow specialist that cannot handle basic conversational patterns. Always include some general-purpose data in your mix.

Evaluation Best Practices

SFT evaluation should be multi-faceted:

Automated Metrics

  • Eval loss — primary training signal; monitor for overfitting
  • Benchmark suites — MMLU, HumanEval, MT-Bench for general ability
  • LLM-as-judge — use GPT-4 or Claude to rate response quality on a rubric

Human Evaluation

  • A/B testing — compare SFT model vs baseline side-by-side
  • Rubric scoring — helpfulness, accuracy, safety, formatting
  • Red-teaming — adversarial testing for safety failures

SFT Checklist

Before launching an SFT training run, verify:

  1. ✅ Chat template matches the base model family
  2. ✅ Loss is masked on non-assistant tokens
  3. ✅ Data is deduplicated and quality-reviewed
  4. ✅ Eval set is held out and representative
  5. ✅ Packing is enabled (or padding is efficient)
  6. ✅ Learning rate is appropriate for LoRA vs full fine-tuning
  7. ✅ Some general-purpose data is mixed in to prevent forgetting
  8. ✅ Gradient checkpointing is on for memory efficiency
Interview takeaway: Instruction tuning is the most accessible and impactful way to customize an LLM for a specific use case. The key decisions are: (1) which base model to start from, (2) how to curate high-quality data, (3) which chat template to use, and (4) whether to use LoRA or full fine-tuning. Getting these four decisions right matters far more than scaling to millions of examples.
