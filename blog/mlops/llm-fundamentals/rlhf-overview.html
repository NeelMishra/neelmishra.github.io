RLHF: Reinforcement Learning from Human Feedback

MLOps Series LLM Fundamentals

Pre-trained language models are excellent next-token predictors, but that objective doesn't directly optimise for helpfulness, harmlessness, or honesty. RLHF — Reinforcement Learning from Human Feedback — bridges this gap by using human preferences to guide model behaviour after pre-training. It is the technique behind InstructGPT, ChatGPT, Claude, and Gemini.

Why RLHF?

The core insight is that evaluation is easier than generation. A human annotator can compare two completions and say "A is better than B" far more reliably and cheaply than writing a perfect completion from scratch. RLHF exploits this asymmetry: collect cheap comparison labels, train a reward model, then optimise the LLM against it.

Supervised Fine-Tuning Alone

  • Requires high-quality demonstrations
  • Expensive: expert-written completions
  • Bounded by demonstrator skill
  • Can't express "avoid this" easily

SFT + RLHF

  • Comparisons are 3–5× cheaper to collect
  • Can surpass demonstrator performance
  • Captures nuanced human preferences
  • Naturally encodes "avoid this" via low reward
Interview Insight: InstructGPT (Ouyang et al., 2022) showed that a 1.3B RLHF model was preferred over the 175B base GPT-3 by human evaluators, demonstrating that alignment quality often matters more than raw scale.

The Three-Stage RLHF Pipeline

The canonical RLHF pipeline proceeds in three sequential phases. Each stage feeds into the next and requires its own data, compute budget, and evaluation strategy.

Stage 1 — Supervised Fine-Tuning (SFT)

Start from a pre-trained base model and fine-tune on high-quality demonstration data using standard cross-entropy loss. This produces πSFT, the starting policy for RL. Typical datasets: 10K–100K examples, 1–3 epochs, learning rate ~1e-5 with cosine decay.

Stage 2 — Reward Model Training

Train a scalar reward model from human comparison data (detailed in the next section). The RM learns to assign higher scores to human-preferred completions.

Stage 3 — PPO Optimisation

Use proximal policy optimisation to update the SFT policy against the learned reward signal, with a KL divergence constraint to prevent catastrophic drift from πSFT.

Reward Model Training

The reward model is the linchpin of RLHF. It converts ordinal human preferences into a differentiable scalar signal that PPO can optimise.

Preference Data Collection

For each prompt x, sample two completions yw (preferred) and yl (rejected) from the SFT policy. A human annotator labels which is better, producing a dataset of triples: (x, y_w, y_l). InstructGPT used ~33K prompts with each prompt yielding 4–9 ranked completions, generating >300K comparison pairs.

Bradley-Terry Loss

The reward model rθ(x, y) is trained with the Bradley-Terry pairwise loss, which models the probability that yw is preferred: 

# Bradley-Terry loss for reward model training
# r_theta: reward model, maps (prompt, completion) → scalar

def reward_model_loss(r_theta, x, y_w, y_l):
    # Score both completions
    r_w = r_theta(x, y_w)          # scalar reward for preferred
    r_l = r_theta(x, y_l)          # scalar reward for rejected

    # Bradley-Terry: maximise log σ(r_w - r_l)
    loss = -log_sigmoid(r_w - r_l)
    return loss.mean()
Key Detail: The reward model is typically initialised from the SFT model with the final unembedding layer replaced by a linear projection to a single scalar. This means the RM is the same size as the policy model — for a 70B policy, you need a 70B reward model (~140 GB in bf16).

Calibration & Label Quality

Inter-annotator agreement on preference data is typically 70–80%. To mitigate noise, teams use majority voting (3–5 annotators per comparison), filter out ties, and apply label smoothing. Anthropic's Constitutional AI (CAI) augments human labels with AI-generated preference data from a "critique → revision" pipeline.

PPO for LLM Fine-Tuning

Proximal Policy Optimisation (Schulman et al., 2017) is adapted for language models by treating text generation as a sequential decision process: each token is an action, the prompt is the initial state, and the reward is given at the end of the sequence.

The PPO-RLHF Training Loop

# PPO training loop for RLHF
# Four models in memory: policy, reference, reward, value

for batch in prompt_dataloader:
    # 1. Generate completions from current policy
    completions = policy.generate(batch, max_new_tokens=512)

    # 2. Score with frozen reward model
    rewards = reward_model(batch, completions)     # shape: (B,)

    # 3. Compute KL penalty against reference policy
    log_probs = policy.log_prob(completions)
    ref_log_probs = ref_policy.log_prob(completions)
    kl_penalty = log_probs - ref_log_probs           # per-token KL

    # 4. Adjusted reward = reward - β * KL
    adjusted_reward = rewards - beta * kl_penalty.sum(dim=-1)

    # 5. Compute advantages via GAE (value function)
    values = value_head(completions)
    advantages = compute_gae(adjusted_reward, values, gamma=1.0, lam=0.95)

    # 6. PPO clipped surrogate loss
    ratio = exp(log_probs - old_log_probs)
    clipped = clip(ratio, 1 - epsilon, 1 + epsilon)
    policy_loss = -min(ratio * advantages, clipped * advantages).mean()

    # 7. Value function loss
    value_loss = mse(values, returns)

    # 8. Update policy + value head (reward + reference frozen)
    loss = policy_loss + 0.5 * value_loss
    loss.backward()
    optimizer.step()
Why PPO over REINFORCE? PPO's clipped surrogate objective limits the policy update to a trust region (ε ≈ 0.2), preventing catastrophically large steps. Vanilla REINFORCE has high variance and no such safeguard, making training very unstable for billion-parameter models.

KL Divergence Penalty

The KL penalty is arguably the most critical component of RLHF stability. Without it, the policy will rapidly mode-collapse toward outputs that exploit reward model weaknesses rather than genuinely improving quality.

What the KL Term Does

The per-token KL divergence between the current policy πθ and the frozen reference policy πref (usually the SFT checkpoint) acts as a regulariser: 

# Per-token KL divergence
# π_θ = current policy, π_ref = frozen SFT model

def kl_per_token(logits_policy, logits_ref):
    log_p = log_softmax(logits_policy, dim=-1)
    log_q = log_softmax(logits_ref, dim=-1)
    p = log_p.exp()
    return (p * (log_p - log_q)).sum(dim=-1)   # shape: (B, T)

# Full RLHF objective: maximise reward, minimise drift
# J(θ) = E[r(x,y)] - β · E[KL(π_θ ∥ π_ref)]

objective = rewards.mean() - beta * kl_per_token.sum(dim=-1).mean()

β Too Low (< 0.01)

  • Policy drifts far from reference
  • Reward hacking becomes likely
  • Generated text degrades in diversity
  • Training may become unstable

β Too High (> 0.5)

  • Policy barely moves from SFT
  • RL has negligible effect
  • Wasted compute — nearly same as SFT
  • Reward scores plateau immediately
Adaptive KL: Some implementations (e.g., TRL library) use an adaptive β that increases when KL exceeds a target (e.g., 6 nats) and decreases when KL is below the target. This auto-tunes the exploration–exploitation balance.

Infrastructure & Memory Requirements

RLHF is 4× more memory-intensive than standard fine-tuning because it requires four full-sized models simultaneously in GPU memory. This is the primary reason RLHF is expensive and technically challenging at scale.

The Four Models

Model Trainable? Memory (70B, bf16) Role
Policy (πθ)✅ Yes~140 GB (+ optim states)The model being optimised
Reference (πref)❌ Frozen~140 GBKL divergence anchor (SFT checkpoint)
Reward Model (rθ)❌ Frozen~140 GBScores completions
Value Head (V)✅ Yes~140 GB (or shared)Baseline for advantage estimation
Memory Reality Check: For a 70B model, naïve RLHF needs ~560 GB of bf16 weights alone — before optimizer states, activations, or KV cache. With Adam optimizer states for trainable models, peak memory can exceed 1 TB. This requires multi-node setups with 8–16× A100/H100 80GB GPUs using DeepSpeed ZeRO-3 or FSDP.

Memory Optimisations

  • LoRA on policy: Freeze policy weights, train only low-rank adapters (~0.1% params). Reduces trainable memory from ~420 GB (weights + Adam) to ~2–4 GB.
  • Shared backbone: Policy, reference, and value head share the same backbone with different LoRA adapters or heads — cuts memory by ~60%.
  • Offloading: Move reference and reward model to CPU; bring to GPU only for scoring. Adds latency but halves GPU memory.
  • Quantisation: Load frozen models (reference, reward) in 4-bit or 8-bit (QLoRA). A 70B model in 4-bit ≈ 35 GB.
  • Gradient checkpointing: Trade compute for memory on activation storage — typically 2× slower, ~60% less activation memory.

Challenges & Failure Modes

RLHF is powerful but fragile. Understanding its failure modes is essential for both practitioners and interview discussions.

1. Reward Hacking (Goodhart's Law)

The policy learns to exploit reward model weaknesses rather than genuinely improving. Classic examples: generating verbose but vacuous answers (longer = higher reward), repeating phrases the RM associates with quality, or producing "sycophantic" responses that agree with user premises regardless of accuracy.

Real-world example: Early ChatGPT-style models would generate excessively long answers with lots of bullet points because the reward model associated length and formatting with quality. Mitigation: add a length penalty, use reward model ensembles, or apply reward clipping.

2. Training Instability

PPO for LLMs can suffer from reward score collapse, where the policy suddenly produces degenerate text (repetitive tokens, empty outputs). Common causes:

  • Learning rate too high — policy overshoots the trust region
  • KL penalty too low — unconstrained drift
  • Value function poorly calibrated — advantage estimates become noisy
  • Reward model has sharp discontinuities that the policy exploits

3. Reward Model Limitations

  • Distribution shift: The RM was trained on SFT-policy outputs, but PPO pushes the policy into new regions the RM hasn't seen
  • Overconfidence: RM assigns extreme scores to out-of-distribution inputs
  • Annotation noise: 70–80% annotator agreement means 20–30% of labels are wrong
  • Positional bias: Annotators systematically prefer the first or second option

4. Alignment Tax

RLHF can degrade raw task performance (e.g., coding benchmarks, factual recall) while improving helpfulness. This "alignment tax" requires careful monitoring of diverse evaluation benchmarks during training, not just the reward score.

Mitigation Strategies

# Reward model ensembles to reduce hacking
def ensemble_reward(prompt, completion, reward_models):
    scores = [rm(prompt, completion) for rm in reward_models]
    # Use minimum or conservative estimate
    return min(scores)   # or: mean(scores) - std(scores)

# Reward clipping to prevent exploitation
def clipped_reward(score, low=-4.0, high=4.0):
    return clip(score, low, high)

# Rejection sampling as cheaper alternative to PPO
def best_of_n(policy, reward_model, prompt, n=16):
    candidates = [policy.generate(prompt) for _ in range(n)]
    scores = [reward_model(prompt, c) for c in candidates]
    return candidates[argmax(scores)]

DPO & RLHF Alternatives

The complexity of PPO-based RLHF has driven research into simpler alignment methods. The most prominent is Direct Preference Optimisation (DPO).

PPO-based RLHF

  • 4 models in memory simultaneously
  • Separate reward model training
  • Complex: RL loop + GAE + clipping
  • Hyperparameter-sensitive (β, ε, lr)
  • Can surpass SFT performance ceiling
  • Battle-tested at scale (ChatGPT, Claude)

Direct Preference Optimisation (DPO)

  • 2 models only (policy + reference)
  • No separate reward model needed
  • Simple: supervised-style loss on pairs
  • Fewer hyperparameters (just β)
  • Implicit reward = log-ratio of policies
  • Growing adoption (Zephyr, Mixtral-Instruct)
# DPO loss — no reward model, no RL loop
def dpo_loss(policy, ref_policy, x, y_w, y_l, beta=0.1):
    # Log-probability ratios
    log_ratio_w = policy.log_prob(y_w | x) - ref_policy.log_prob(y_w | x)
    log_ratio_l = policy.log_prob(y_l | x) - ref_policy.log_prob(y_l | x)

    # DPO objective: push preferred up, rejected down
    loss = -log_sigmoid(beta * (log_ratio_w - log_ratio_l))
    return loss.mean()
Interview Talking Point: DPO shows that the RLHF objective can be analytically reparameterised to eliminate the reward model. The optimal policy under the KL-constrained reward maximisation objective is: π*(y|x) ∝ πref(y|x) · exp(r(x,y) / β), which means the reward is implicitly defined by the log-ratio of the two policies. DPO directly optimises this closed-form solution.

Other Alternatives

  • RLAIF (Constitutional AI): Replace human labellers with an AI critic that evaluates outputs against a constitution of principles. Used by Anthropic for Claude.
  • Best-of-N / Rejection Sampling: Generate N candidates, score with RM, return the best. No training needed but N× inference cost at serving time.
  • REINFORCE Leave-One-Out (RLOO): Simpler RL algorithm that uses leave-one-out baselines. Less variance than vanilla REINFORCE, simpler than PPO.
  • KTO (Kahneman-Tversky Optimisation): Requires only binary feedback (good/bad) rather than pairwise comparisons, inspired by prospect theory.
  • IPO (Identity Preference Optimisation): Addresses DPO's overfitting issues with a regularised objective.
The Landscape in 2024–25: The field is moving toward simpler methods. DPO and its variants (IPO, KTO, ORPO) dominate open-source alignment, while frontier labs (OpenAI, Anthropic, Google) still use PPO-style RLHF — likely because PPO's flexibility matters more at extreme scale and for multi-turn, tool-use, and reasoning tasks.
