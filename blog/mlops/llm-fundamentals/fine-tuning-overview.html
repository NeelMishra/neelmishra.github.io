Fine-Tuning LLMs: Full, Freeze & Parameter-Efficient

MLOps Series LLM Fundamentals

Why Fine-Tune?

Pre-trained LLMs like GPT-4, LLaMA, and Mistral learn broad language understanding from massive corpora. However, production deployments often require domain-specific behaviour—medical terminology, legal document parsing, code generation in niche frameworks, or brand-aligned tone. Fine-tuning adapts a pre-trained model to your distribution by continuing gradient-based optimization on a curated dataset.

The three primary strategies sit on a spectrum of capacity vs. efficiency:

Key insight: Fine-tuning is not about teaching the model new facts—it is about adjusting the model’s behaviour, format, and domain distribution. For factual knowledge, prefer Retrieval-Augmented Generation (RAG).

Full Fine-Tuning

Full fine-tuning updates every parameter in the model. For a 7B-parameter LLaMA-2 model, that means all 7 billion weights receive gradient updates on each training step. This gives maximum expressiveness—the model can reshape any layer’s representations to fit your data.

How It Works

Starting from pre-trained weights θ0, we minimize the task loss over dataset D:

θ* = argminθ ∑(x,y)∈D L(fθ(x), y)

All layers—embedding, attention (Q/K/V projections and output), feed-forward networks (FFN), layer norms, and the language model head—participate in backpropagation.

Memory Requirements

Full fine-tuning is memory-intensive. For a 7B model in fp32:

  • Model weights: 7B × 4 bytes = 28 GB
  • Gradients: 7B × 4 bytes = 28 GB
  • Optimizer states (Adam): 7B × 8 bytes (m + v) = 56 GB
  • Activations: varies with batch size and sequence length; typically 8–20+ GB
  • Total: ~120–140 GB GPU VRAM for fp32

With mixed-precision (bf16 weights + fp32 optimizer), this drops to ~70–80 GB—still requiring multi-GPU setups (e.g., 2× A100 80 GB with FSDP or DeepSpeed ZeRO-3).

Catastrophic forgetting: Full fine-tuning can overwrite general capabilities if your dataset is small or narrow. Mitigate with low learning rates (1e-5 to 5e-6), short training runs (1–3 epochs), and mixing in general-domain data.

Freeze-Layer Tuning

Freeze-layer tuning (also called partial fine-tuning) keeps the lower transformer layers frozen and only updates the top N layers plus the LM head. The intuition: lower layers capture universal linguistic features (syntax, morphology), while upper layers encode more task-specific semantics.

Implementation

In practice, you iterate through named parameters and set requires_grad = False for layers below your threshold. For a 32-layer model, freezing the bottom 24 layers means only 8 layers (25% of the transformer) receive gradients. 

# Freeze bottom layers of a 32-layer LLaMA model
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-hf")

# Freeze embedding + layers 0-23
for name, param in model.named_parameters():
    if "embed" in name:
        param.requires_grad = False
    elif "layers." in name:
        layer_idx = int(name.split("layers.")[1].split(".")[0])
        if layer_idx < 24:
            param.requires_grad = False

# Verify: count trainable params
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
total = sum(p.numel() for p in model.parameters())
print(f"Trainable: {trainable/1e6:.1f}M / {total/1e6:.1f}M ({100*trainable/total:.1f}%)")
# Output: Trainable: ~1750M / 6738M (26.0%)

Trade-offs

  • Memory savings: Gradients and optimizer states are only needed for unfrozen layers. Freezing 75% of layers saves ~50–60% of optimizer memory.
  • Speed: Backward pass is faster since frozen layers skip gradient computation (though forward pass cost is unchanged).
  • Capacity: Less expressive than full FT. Works well when the domain shift is moderate.

PEFT: Parameter-Efficient Fine-Tuning

PEFT methods inject or modify a tiny number of parameters (< 2% of the model) while keeping the entire base model frozen. The key insight is that the weight updates ΔW during fine-tuning live in a low-rank subspace—we don’t need to update all dimensions.

LoRA (Low-Rank Adaptation)

LoRA decomposes the weight update into two low-rank matrices: ΔW = BA, where B ∈ ℜd×r and A ∈ ℜr×k. Instead of updating a d×k weight matrix (e.g., 4096×4096 = 16.7M parameters), we train two small matrices with rank r (typically 8–64), yielding (d+k)×r parameters.

For r=16 on a 4096×4096 matrix: (4096+4096)×16 = 131K parameters vs 16.7M—a 128× reduction.

LoRA at inference: The adapter weights can be merged back into the base weights (W' = W + BA), adding zero latency at serving time. This is a major advantage over other PEFT methods.

Adapter Layers

Adapters insert small bottleneck modules between existing transformer layers. Each adapter has a down-projection (d → r), a non-linearity (ReLU/GELU), and an up-projection (r → d), plus a residual connection. Typical bottleneck size: r=64, yielding ~0.5–3% additional parameters.

Unlike LoRA, adapters add inference latency because they introduce new sequential computation in the forward pass. However, they can be more expressive than LoRA due to the non-linearity.

Prefix Tuning / Prompt Tuning

Prefix tuning prepends learnable continuous vectors (virtual tokens) to every attention layer’s key and value matrices. For a model with L layers and prefix length p, the trainable parameters = L × 2 × p × dmodel.

Prompt tuning (simpler variant) only prepends learnable embeddings at the input layer. It’s extremely parameter-efficient (~0.01% of model) but less expressive—works best for very large models (10B+) where the pre-trained representations are already strong.

LoRA Strengths

  • Zero inference overhead after merging
  • Can target specific modules (Q, K, V, FFN)
  • Easy to swap adapters for multi-task serving
  • Well-supported: HuggingFace PEFT, LitGPT

Adapter Strengths

  • Non-linear bottleneck adds expressiveness
  • Works well for classification tasks
  • Composable: stack adapters for multi-task
  • Battle-tested in NLP since 2019

Prefix Tuning Strengths

  • Extremely few trainable parameters
  • Works at every layer (deep context)
  • Good for conditional generation tasks
  • No modification to model weights

Prompt Tuning Strengths

  • Simplest PEFT method to implement
  • Input-level only: ~0.01% params
  • Scales well with model size (>10B)
  • Easy multi-tenant: one model, many prompts

When to Use Each Approach

Selecting a fine-tuning strategy depends on your data volume, compute budget, domain gap, and serving constraints.

Use Full Fine-Tuning When

  • Large domain-specific dataset (>100K examples)
  • Significant distribution shift from pre-training
  • You need maximum task performance
  • Sufficient compute: multi-GPU with 80+ GB VRAM per GPU
  • Example: training a specialized medical LLM

Use Freeze-Layer Tuning When

  • Medium dataset (10K–100K examples)
  • Moderate domain shift
  • Limited multi-GPU resources
  • Want a middle ground between FT and PEFT
  • Example: adapting a general chatbot to customer support

Use LoRA/QLoRA When

  • Small to medium dataset (1K–50K examples)
  • Single-GPU constraint (e.g., 1× A100 or even RTX 4090)
  • Need to serve multiple adapters on one base model
  • Want fast iteration with low memory
  • Example: instruction-tuning for a specific coding task

Use Prefix/Prompt Tuning When

  • Very large base model (>10B) with strong pre-trained representations
  • Multi-tenant deployment: many tasks, one model
  • Minimal compute budget
  • Task is close to pre-training distribution
  • Example: tone-shifting for different brand voices
Interview tip: In production, QLoRA (LoRA + 4-bit quantization) is the most popular choice for fine-tuning 7B–70B models. It enables training a 65B model on a single 48 GB GPU with minimal quality loss. Know the paper: Dettmers et al. (2023).

Memory & Compute Comparison

The following table compares memory and compute requirements for fine-tuning a 7B parameter model (e.g., LLaMA-2-7B) with a context length of 2048 tokens and batch size of 4.

Full FT (bf16) 7B (100%) ~70–80 GB 1× (baseline) 2× A100 80GB
Full FT + DeepSpeed ZeRO-3 7B (100%) ~25 GB/GPU 0.7–0.9× 4× A100 40GB
Freeze Top-8 Layers ~1.75B (26%) ~40–50 GB 1.3–1.5× 1× A100 80GB

HuggingFace Trainer Example

Below is a complete example using HuggingFace’s transformers and peft libraries to fine-tune LLaMA-2-7B with QLoRA on a custom instruction-following dataset. 

# QLoRA Fine-Tuning with HuggingFace Trainer
import torch
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    TrainingArguments,
    Trainer,
    BitsAndBytesConfig,
    DataCollatorForLanguageModeling,
)
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training
from datasets import load_dataset

# 1. Quantization config (4-bit NF4 for QLoRA)
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,  # nested quantization
)

# 2. Load model in 4-bit
model_id = "meta-llama/Llama-2-7b-hf"
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    quantization_config=bnb_config,
    device_map="auto",
    trust_remote_code=True,
)
tokenizer = AutoTokenizer.from_pretrained(model_id)
tokenizer.pad_token = tokenizer.eos_token

# 3. Prepare model for k-bit training
model = prepare_model_for_kbit_training(model)

# 4. LoRA configuration
lora_config = LoraConfig(
    r=16,                        # rank
    lora_alpha=32,               # scaling factor (alpha/r)
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 20,971,520 || all: 6,758,404,096 || 0.31%

# 5. Load and tokenize dataset
dataset = load_dataset("json", data_files="instruction_data.jsonl")

def tokenize(example):
    prompt = f"### Instruction:\n{example['instruction']}\n\n### Response:\n{example['output']}"
    tokens = tokenizer(prompt, truncation=True, max_length=2048, padding="max_length")
    tokens["labels"] = tokens["input_ids"].copy()
    return tokens

tokenized = dataset["train"].map(tokenize, remove_columns=dataset["train"].column_names)

# 6. Training arguments
training_args = TrainingArguments(
    output_dir="./qlora-llama2-7b",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,       # effective batch = 16
    learning_rate=2e-4,
    lr_scheduler_type="cosine",
    warmup_ratio=0.05,
    bf16=True,
    logging_steps=10,
    save_strategy="steps",
    save_steps=200,
    optim="paged_adamw_8bit",           # memory-efficient optimizer
    gradient_checkpointing=True,        # trade compute for memory
    report_to="wandb",
)

# 7. Initialize Trainer and train
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized,
    data_collator=DataCollatorForLanguageModeling(tokenizer, mlm=False),
)
trainer.train()

# 8. Save adapter weights (only ~80 MB for LoRA)
model.save_pretrained("./qlora-llama2-7b/adapter")

# 9. Merge adapter into base model for deployment
merged = model.merge_and_unload()
merged.save_pretrained("./qlora-llama2-7b/merged")
Key hyperparameters for LoRA:
  • r (rank): 8–64. Higher rank = more capacity but more memory. Start with 16.
  • lora_alpha: Typically 2×r. Controls the scaling of ΔW.
  • target_modules: At minimum Q/V projections; including all attention + FFN gives best results.
  • lora_dropout: 0.05–0.1 for regularization on small datasets.

Production Tips & Pitfalls

1. Evaluation Strategy

Always evaluate on a held-out set that mirrors your production distribution. Track both task-specific metrics (accuracy, F1, ROUGE) and general capability benchmarks (MMLU, HellaSwag) to detect catastrophic forgetting.

2. Data Quality Over Quantity

For instruction tuning, 1,000 high-quality examples often outperform 100,000 noisy ones. The LIMA paper (Zhou et al., 2023) showed that 1,000 carefully curated examples can produce a strong instruction-following model.

3. Multi-Adapter Serving

With LoRA, you can serve multiple task-specific adapters from one base model. Libraries like vLLM and LoRAX support dynamic adapter loading, enabling multi-tenant deployments where each customer gets a personalized adapter on shared infrastructure. 

# Multi-adapter serving with PEFT
from peft import PeftModel

# Load base model once
base_model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-hf")

# Load different adapters on demand
legal_model = PeftModel.from_pretrained(base_model, "./adapters/legal-lora")
medical_model = PeftModel.from_pretrained(base_model, "./adapters/medical-lora")
code_model = PeftModel.from_pretrained(base_model, "./adapters/code-lora")

# Each adapter is ~80 MB vs 14 GB for the base model
# Total: 14 GB + (3 × 80 MB) = ~14.25 GB for 3 specialized models

4. Common Pitfalls

  • Learning rate too high: LoRA is sensitive to LR. Use 1e-4 to 3e-4 for LoRA; 1e-5 to 5e-6 for full FT.
  • Forgetting to set pad token: Many LLMs don’t have a pad token by default. Set tokenizer.pad_token = tokenizer.eos_token.
  • Label masking: For instruction tuning, mask the prompt tokens in labels so the model only learns to predict the response.
  • Evaluation on training format: Always evaluate using the same template/format used during fine-tuning.
  • Overtraining LoRA: With small datasets, 1–3 epochs is usually sufficient. More epochs risk overfitting the adapter.
Cost reality check: Fine-tuning LLaMA-2-7B with QLoRA on 10K examples for 3 epochs takes ~2–4 hours on a single A100 40GB, costing roughly $4–8 on cloud providers. Full fine-tuning the same model takes ~8–12 hours on 4× A100s, costing $60–120. Always start with LoRA/QLoRA and only move to full FT if quality demands it.
