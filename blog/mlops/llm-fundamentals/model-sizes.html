Model Sizes: Parameters, Memory & Compute Requirements

MLOps Series LLM Fundamentals

Before you can serve or fine-tune a large language model, you need to answer one deceptively simple question: how much hardware does this model actually need? The answer depends on the number of parameters, the numerical precision, the batch size, the sequence length, and whether you're training or running inference. This post gives you the formulas and reference tables to plan GPU resources with confidence.

Parameter Counting in Transformers

A standard decoder-only transformer (GPT-style) has L identical layers plus an embedding table and a final projection head. Each layer contains a self-attention block and a feed-forward network (FFN). Here is how the parameter count breaks down: 

def count_transformer_params(
    vocab_size, d_model, n_layers, d_ff, n_heads,
    use_swiglu=True, tied_embeddings=False
):
    # Embedding: token + (optional) output projection
    embedding = vocab_size * d_model
    output_proj = 0 if tied_embeddings else vocab_size * d_model

    # Per-layer self-attention: Q, K, V, O projections
    attn_per_layer = 4 * d_model * d_model  # + 4*d_model bias (often omitted)

    # Per-layer FFN (SwiGLU uses 3 matrices instead of 2)
    ffn_multiplier = 3 if use_swiglu else 2
    ffn_per_layer = ffn_multiplier * d_model * d_ff

    # LayerNorm: 2 per layer (pre-attn, pre-ffn) × 2 params each
    norm_per_layer = 4 * d_model  # RMSNorm uses only d_model per norm

    total_per_layer = attn_per_layer + ffn_per_layer + norm_per_layer
    total = embedding + output_proj + n_layers * total_per_layer

    return total

# LLaMA-2 7B
params = count_transformer_params(32000, 4096, 32, 11008, 32)
print(f"Parameters: {params / 1e9:.2f}B")  # ≈ 6.74B
Key insight: The FFN dominates parameter count. In LLaMA-2 7B, each layer's FFN has ~135M parameters vs ~67M for attention. At scale, roughly 65% of parameters live in FFN layers.

Precision Formats & Bytes Per Parameter

Every parameter is stored as a floating-point (or integer) number. The choice of precision directly determines memory consumption. Here are the formats used in modern LLM workloads:

FP32 — 4 bytes

Range: ±3.4 × 10³⁸

Mantissa: 23 bits (7 decimal digits)

Use case: Master weights in mixed-precision training, optimizer states (Adam).

A 7B model in FP32 = 28 GB

FP16 — 2 bytes

Range: ±65,504

Mantissa: 10 bits (3.3 decimal digits)

Use case: Forward/backward pass in mixed-precision training. Risk of overflow.

A 7B model in FP16 = 14 GB

BF16 — 2 bytes

Range: ±3.4 × 10³⁸ (same as FP32)

Mantissa: 7 bits (2.4 decimal digits)

Use case: Preferred for LLM training on A100/H100. Same range as FP32, avoids overflow.

A 7B model in BF16 = 14 GB

INT8 / INT4 — 1 / 0.5 bytes

Range: -128 to 127 (INT8) / -8 to 7 (INT4)

Mantissa: N/A (integer)

Use case: Post-training quantization for inference (GPTQ, AWQ, GGUF).

7B model: INT8 = 7 GB, INT4 = 3.5 GB

BF16 vs FP16: BF16 has the same exponent range as FP32 (8 bits) but fewer mantissa bits. This means it rarely overflows during training — which is why Google designed it and why it's the default on A100/H100 GPUs. FP16 can cause loss spikes due to overflow if not carefully managed with loss scaling.

Memory Requirements for Training

Training memory is far more than just "model weights." With Adam optimizer and mixed-precision training, you need to account for four distinct memory pools:

The Training Memory Formula

# Mixed-precision training with AdamW (the standard recipe)
# Let P = number of parameters

model_weights   = P * 2     # BF16/FP16 weights for forward/backward
master_weights  = P * 4     # FP32 copy maintained by optimizer
gradients       = P * 2     # BF16/FP16 gradients
optimizer_adam   = P * 8    # FP32 first moment (4B) + second moment (4B)

# Total for parameters + optimizer
param_memory    = P * 16    # bytes (2 + 4 + 2 + 8)

# Activations (depends on batch, seq_len, layers, hidden_dim)
# Rule of thumb: activations ≈ 2 × param_memory at batch_size=1, seq=2048
# Activation checkpointing reduces this by ~60% at cost of 33% more compute

# Example: 7B model
P = 7e9
total_param_bytes = P * 16  # = 112 GB
print(f"Parameter memory: {total_param_bytes / (1024**3):.1f} GB")
# Plus activations: ~70-140 GB depending on batch size
# Total: 180-250 GB → need multi-GPU setup
Component Precision Bytes / Param 7B Model 70B Model
Model weights BF16 2 14 GB 140 GB
Master weights FP32 4 28 GB 280 GB
Gradients BF16 2 14 GB 140 GB
Adam states (m + v) FP32 8 56 GB 560 GB
Subtotal (params) — 16 112 GB 1,120 GB
Activations (est.) BF16 varies ~70 GB ~700 GB
Total estimate — — ~182 GB ~1,820 GB
Why you can't train a 7B model on a single GPU: Even an A100 80 GB can only hold the BF16 weights (14 GB) and some activations. The full training state (112 GB for params + optimizer) requires at least 2× A100 80 GB with ZeRO Stage 3 (DeepSpeed) or FSDP sharding across GPUs.

Activation Memory

Activations are the intermediate tensors saved during the forward pass for use in the backward pass. For a transformer layer with batch size B, sequence length S, hidden dimension d, and attention heads h: 

# Per-layer activation memory (without checkpointing)
def activation_memory_per_layer(B, S, d, h):
    # Self-attention activations
    qkv_proj    = 3 * B * S * d * 2     # Q, K, V projections (BF16)
    attn_scores = B * h * S * S * 2     # attention matrix (BF16)
    attn_output = B * S * d * 2         # output projection input

    # FFN activations
    ffn_hidden  = B * S * d * 4 * 2     # SwiGLU: 2 up-projections (simplified)

    # LayerNorm inputs + residual connections
    norms       = 2 * B * S * d * 2

    return qkv_proj + attn_scores + attn_output + ffn_hidden + norms

# LLaMA-2 7B: B=1, S=2048, d=4096, h=32
per_layer = activation_memory_per_layer(1, 2048, 4096, 32)
total_act = per_layer * 32  # 32 layers
print(f"Activations: {total_act / (1024**3):.1f} GB")  # ~4.5 GB at B=1
# Scales linearly with batch size!
Activation checkpointing: Instead of storing all activations, you can recompute them during the backward pass. This trades ~33% more compute time for a ~60% reduction in activation memory. Most large-scale training runs use this technique.

Memory Requirements for Inference

Inference is much cheaper than training because you don't store gradients, optimizer states, or full activation checkpoints. The two main memory consumers are:

Model Weights

The parameters loaded once into GPU memory. Size depends purely on precision:

  • FP16/BF16: 2 bytes × P
  • INT8 (GPTQ/AWQ): 1 byte × P
  • INT4 (GPTQ/GGUF): 0.5 bytes × P

KV Cache

Grows with each generated token. Stores key/value vectors for all previous tokens:

  • Per token: 2 × L × d × 2 bytes (BF16)
  • Scales with batch size and sequence length
  • Often the bottleneck for long-context serving
def inference_memory(
    params_B,         # parameters in billions
    bytes_per_param,  # 2 for BF16, 1 for INT8, 0.5 for INT4
    n_layers,
    d_model,
    n_kv_heads,       # GQA: may be fewer than n_heads
    d_head,           # = d_model / n_heads typically
    seq_len,
    batch_size,
    kv_precision=2    # bytes per element in KV cache
):
    # Model weights
    weight_mem = params_B * 1e9 * bytes_per_param

    # KV cache: 2 (K+V) × layers × kv_heads × d_head × seq × batch × precision
    kv_cache = 2 * n_layers * n_kv_heads * d_head * seq_len * batch_size * kv_precision

    total = weight_mem + kv_cache
    return {
        "weights_gb": weight_mem / (1024**3),
        "kv_cache_gb": kv_cache / (1024**3),
        "total_gb": total / (1024**3)
    }

# LLaMA-2 7B, BF16, seq=4096, batch=1
mem = inference_memory(7, 2, 32, 4096, 32, 128, 4096, 1)
print(f"Weights: {mem['weights_gb']:.1f} GB, KV: {mem['kv_cache_gb']:.2f} GB")
# Weights: 13.0 GB, KV: 2.0 GB → fits on one A100 40GB

# LLaMA-2 70B, INT4, seq=4096, batch=8
mem = inference_memory(70, 0.5, 80, 8192, 8, 128, 4096, 8)
print(f"Weights: {mem['weights_gb']:.1f} GB, KV: {mem['kv_cache_gb']:.1f} GB")
# Weights: 32.6 GB, KV: 2.5 GB → fits on one A100 80GB (INT4)
GQA saves KV cache memory: Grouped-Query Attention (used in LLaMA-2 70B, Mistral) uses fewer KV heads than query heads. LLaMA-2 70B uses 8 KV heads instead of 64 query heads — an 8× reduction in KV cache size. This is critical for serving many concurrent requests.

FLOPS Estimation for Training & Inference

FLOPS (Floating Point Operations) determine how long a training run or inference request takes. The key formulas come from the Chinchilla and Kaplan et al. scaling papers:

Training FLOPS

# The "6ND" rule: total training FLOPS ≈ 6 × N × D
# N = number of parameters
# D = number of training tokens
# Factor of 6: 2 for forward (multiply-add), 4 for backward (2× forward)

def training_flops(params_B, tokens_T):
    """Estimate total training FLOPS."""
    N = params_B * 1e9
    D = tokens_T * 1e12
    return 6 * N * D

def training_time_days(total_flops, num_gpus, gpu_tflops, utilization=0.4):
    """Estimate wall-clock training time.
    utilization: typically 30-50% (MFU) for large-scale training."""
    effective_tflops = num_gpus * gpu_tflops * utilization
    seconds = total_flops / (effective_tflops * 1e12)
    return seconds / 86400

# Example: Training LLaMA-2 70B on 2T tokens
flops = training_flops(70, 2)
print(f"Total FLOPS: {flops:.1e}")           # 8.4e23 FLOPS

days = training_time_days(flops, 2048, 989)  # 2048× H100 @ 989 BF16 TFLOPS
print(f"Training time: {days:.0f} days")      # ~12 days at 40% MFU

Inference FLOPS

# Per-token inference is approximately 2N FLOPS (forward pass only)
# But: prefill (processing prompt) vs decode (generating) differ

def inference_flops(params_B, prompt_tokens, output_tokens):
    N = params_B * 1e9
    # Prefill: processes all prompt tokens in parallel (compute-bound)
    prefill = 2 * N * prompt_tokens
    # Decode: generates one token at a time (memory-bound)
    decode = 2 * N * output_tokens
    return {"prefill": prefill, "decode": decode, "total": prefill + decode}

# LLaMA-2 70B: 2048 prompt tokens, 512 output tokens
flops = inference_flops(70, 2048, 512)
print(f"Prefill: {flops['prefill']:.1e} FLOPS")  # 2.87e14
print(f"Decode:  {flops['decode']:.1e} FLOPS")   # 7.17e13

# Decode is memory-bandwidth-bound, not compute-bound!
# Arithmetic intensity = FLOPS / bytes_loaded ≈ 1 for autoregressive decode
# This is why batching and quantization matter so much for serving
MFU (Model FLOPS Utilization): Real-world training achieves 30-50% of a GPU's peak FLOPS. The rest is lost to communication overhead, memory operations, and pipeline bubbles. Google's PaLM achieved ~46% MFU, which is considered excellent. Always use realistic MFU values (0.3-0.5) when estimating training time.

GPU Requirements Calculator

This calculator combines everything above into a single function you can use for capacity planning. It covers both training and inference scenarios: 

def gpu_requirements(
    params_B,
    mode="inference",          # "training" or "inference"
    precision="bf16",           # "fp32", "bf16", "fp16", "int8", "int4"
    batch_size=1,
    seq_len=4096,
    gpu_mem_gb=80,              # per-GPU memory (e.g., A100=80, H100=80)
    n_layers=None,              # for KV cache calc
    d_model=None,
    n_kv_heads=None,
    d_head=128
):
    bytes_map = {"fp32": 4, "bf16": 2, "fp16": 2, "int8": 1, "int4": 0.5}
    bpp = bytes_map[precision]
    P = params_B * 1e9

    if mode == "training":
        # Always train in mixed precision: BF16 weights + FP32 optimizer
        param_mem = P * 16              # weights + master + grads + adam
        # Rough activation estimate
        act_mem = P * 2 * batch_size * (seq_len / 2048)
        total_bytes = param_mem + act_mem
    else:
        # Inference
        weight_mem = P * bpp
        kv_mem = 0
        if n_layers and d_model and n_kv_heads:
            kv_mem = 2 * n_layers * n_kv_heads * d_head * seq_len * batch_size * 2
        total_bytes = weight_mem + kv_mem

    total_gb = total_bytes / (1024**3)
    overhead = 1.15  # 15% framework overhead
    adjusted_gb = total_gb * overhead
    min_gpus = max(1, -(-int(adjusted_gb) // gpu_mem_gb))  # ceiling division

    print(f"{'='*50}")
    print(f"  Model: {params_B}B | Mode: {mode} | Precision: {precision}")
    print(f"  Raw memory: {total_gb:.1f} GB")
    print(f"  With overhead: {adjusted_gb:.1f} GB")
    print(f"  Min GPUs ({gpu_mem_gb}GB each): {min_gpus}")
    print(f"{'='*50}")
    return {"total_gb": adjusted_gb, "min_gpus": min_gpus}

# ─── Example usage ───

# Training LLaMA-2 7B on A100 80GB
gpu_requirements(7, "training", "bf16", batch_size=4, gpu_mem_gb=80)
# → Raw: ~168 GB, With overhead: ~193 GB → Min 3× A100 80GB

# Inference LLaMA-2 70B in INT4 on A100 80GB
gpu_requirements(70, "inference", "int4", batch_size=8, seq_len=4096,
    gpu_mem_gb=80, n_layers=80, d_model=8192, n_kv_heads=8)
# → Weights: 32.6 GB + KV: 2.5 GB → fits on 1× A100 80GB!

# Inference LLaMA-3.1 405B in BF16 on H100 80GB
gpu_requirements(405, "inference", "bf16", batch_size=1, seq_len=8192,
    gpu_mem_gb=80, n_layers=126, d_model=16384, n_kv_heads=8)
# → Weights: 753 GB + KV: 2 GB → Min 10× H100 80GB

Key Rules of Thumb

The 2× rule: For BF16 inference, you need roughly 2× the parameter count in GB of GPU memory (e.g., 7B → ~14 GB). For INT4, divide by 4: 0.5× params in GB.
The 18× rule for training: With mixed-precision AdamW, you need approximately 18× the parameter count in GB (16× for params/optimizer + ~2× for activations at modest batch sizes). Use activation checkpointing and ZeRO/FSDP to shard across GPUs.
Don't forget KV cache at scale: Serving 100 concurrent users with 4K context on a 70B model can require 50+ GB just for KV cache — potentially more than the model weights themselves (if quantized). Use PagedAttention (vLLM) to manage KV cache efficiently.

Parallelism Strategies

Interview tip: When asked "How would you serve a 70B model?", structure your answer: (1) Calculate memory — 70B × 2 bytes = 140 GB in BF16, (2) Choose parallelism — 2× A100 80GB with tensor parallelism, (3) Consider quantization — INT4 (35 GB) fits on one GPU but with quality trade-offs, (4) Account for KV cache and batching — use vLLM with PagedAttention, (5) Estimate throughput and cost.
