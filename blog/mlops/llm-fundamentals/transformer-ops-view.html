Transformer Architecture: An Ops Engineer's Perspective

MLOps Series LLM Fundamentals

Why Ops Engineers Should Understand Transformers

You don't need to derive back-propagation gradients to run LLMs in production. But you do need to understand the architecture well enough to answer questions like:

  • How many GPUs do I need to serve a 70B-parameter model?
  • Why does doubling the context window quadruple my KV-cache memory?
  • What changes when we switch from FP16 to INT8 quantization?
  • Why is batch size 32 OOM-ing when batch size 16 fits comfortably?

Every one of these maps directly to a specific component of the transformer. This post gives you the mental model to reason about infrastructure from first principles.

Ops Rule of Thumb: If you can estimate the parameter count, bytes-per-parameter, and activation memory, you can predict GPU requirements within 20% for any model — without running a single benchmark.

Architecture Overview

The transformer, introduced in "Attention Is All You Need" (Vaswani et al., 2017), consists of stacked layers of two primary sub-blocks: Multi-Head Self-Attention (MHA) and Feed-Forward Networks (FFN). Modern decoder-only LLMs (GPT, LLaMA, Claude) drop the encoder and use only the decoder stack with causal masking.

Each decoder layer contains the same pattern: LayerNorm → Attention → Residual Add → LayerNorm → FFN → Residual Add. This is repeated num_layers times. The final output passes through a linear head that maps hidden states back to vocabulary logits.

Key insight for ops: The FFN block holds roughly ⅔ of total parameters per layer (8·h² vs 4·h² for attention, in standard architectures). But during inference with long sequences, the attention mechanism dominates memory due to the KV-cache growing linearly with sequence length.

Key Parameters & Their Infra Impact

Every transformer is defined by a handful of hyperparameters. Understanding what each one controls — and how it scales — is critical for capacity planning.

Parameter Symbol Typical Range Infrastructure Impact
num_layers L 32 – 128 Scales total params linearly; determines pipeline-parallel stages
hidden_dim h 4096 – 12288 Params scale as h²; dominant factor in model size
num_heads nh 32 – 96 Each head has dim h/nh; affects tensor-parallel sharding granularity
vocab_size V 32K – 128K Embedding table: V × h params; large V increases memory on first/last GPU
context_length s 4K – 128K+ KV-cache memory scales as O(batch × layers × s × h); attention FLOPs scale O(s²)
intermediate_dim dff 4h (often 8/3·h with SwiGLU) FFN weight size: 2 × h × dff (or 3 × h × dff for gated variants)
num_kv_heads nkv 1 – nh GQA/MQA reduces KV-cache by factor nh/nkv; huge inference savings
Watch out: The hidden_dim has a quadratic effect on model size. Going from h=4096 to h=8192 doesn't double the model — it 4× the parameters in attention and FFN weight matrices. This is the single biggest driver of GPU memory requirements.

Compute & FLOPs Estimation

For capacity planning, you need to estimate both the total FLOPs for a forward pass and the throughput of your hardware. The standard approximation for a forward pass through a transformer:

# FLOPs estimation for a single forward pass
# Rule of thumb: ~2 FLOPs per parameter per token

def estimate_forward_flops(num_params, seq_len, batch_size):
    # Each parameter participates in one multiply-add = 2 FLOPs
    flops_per_token = 2 * num_params
    total_flops = flops_per_token * seq_len * batch_size
    return total_flops

# Example: LLaMA-2 70B, seq_len=4096, batch_size=1
flops = estimate_forward_flops(70e9, 4096, 1)
print(f"Forward pass: {flops / 1e12:.1f} TFLOPs")
# → Forward pass: 573.4 TFLOPs

# A100 GPU: 312 TFLOPS (FP16 Tensor Cores)
# Theoretical min latency: 573.4 / 312 ≈ 1.84 seconds
# Real-world (50-70% utilization): ~2.6-3.7 seconds per full prefill

But attention has an additional quadratic cost in sequence length that the "2N" rule doesn't capture:

def attention_flops(num_layers, num_heads, head_dim, seq_len, batch_size):
    # QK^T: seq_len × seq_len × head_dim per head
    # Softmax × V: seq_len × seq_len × head_dim per head
    qk_flops = 2 * batch_size * num_heads * seq_len * seq_len * head_dim
    sv_flops = 2 * batch_size * num_heads * seq_len * seq_len * head_dim
    total = (qk_flops + sv_flops) * num_layers
    return total

# At seq_len = 4K:  attention ≈ 5% of total FLOPs
# At seq_len = 32K: attention ≈ 30% of total FLOPs
# At seq_len = 128K: attention ≈ 70%+ of total FLOPs
# This is why long-context inference is so expensive!

A100-80GB (FP16)

  • 312 TFLOPS peak (Tensor Cores)
  • ~200 TFLOPS sustained
  • 80 GB HBM2e, 2 TB/s bandwidth
  • Sweet spot: 7B–13B models single-GPU

H100-80GB (FP16)

  • 990 TFLOPS peak (Tensor Cores)
  • ~600 TFLOPS sustained
  • 80 GB HBM3, 3.35 TB/s bandwidth
  • Sweet spot: 13B–70B with tensor parallelism
Bandwidth vs. Compute: During autoregressive decoding (generating one token at a time), LLMs are memory-bandwidth bound, not compute bound. You read all model weights from HBM for each token but only do one multiply per weight. This is why the arithmetic intensity is so low (~1 FLOP/byte) and GPU utilization drops to 1-5% during generation.

How Architecture Drives Infrastructure

Three architecture-level decisions have outsized infrastructure impact:

1. Model Size → GPU Count

The number of GPUs required is a direct function of parameter count and precision:

def min_gpus_for_inference(num_params_B, precision_bytes, gpu_mem_GB, overhead=0.15):
    # overhead accounts for KV-cache, activations, CUDA context (~15%)
    model_mem = num_params_B * 1e9 * precision_bytes / 1e9  # in GB
    usable_mem = gpu_mem_GB * (1 - overhead)
    import math
    return math.ceil(model_mem / usable_mem)

# LLaMA-2 70B in FP16 on A100-80GB
print(min_gpus_for_inference(70, 2, 80))   # → 3 GPUs minimum
# LLaMA-2 70B in INT4 on A100-80GB
print(min_gpus_for_inference(70, 0.5, 80))  # → 1 GPU (fits!)
# GPT-4 class (est. ~1.8T MoE, ~280B active) in FP16
print(min_gpus_for_inference(1800, 2, 80))  # → 53 GPUs

2. Context Length → KV-Cache Memory

The KV-cache stores previously computed key and value tensors for each layer. It grows linearly with both sequence length and batch size:

def kv_cache_memory_gb(num_layers, num_kv_heads, head_dim,
                        seq_len, batch_size, precision_bytes=2):
    # K and V each: batch × kv_heads × seq_len × head_dim
    kv_per_layer = 2 * batch_size * num_kv_heads * seq_len * head_dim
    total_elements = kv_per_layer * num_layers
    return total_elements * precision_bytes / (1024**3)

# LLaMA-2 70B: 80 layers, 8 KV heads, head_dim=128
# Batch=1, seq_len=4K
print(f"{kv_cache_memory_gb(80, 8, 128, 4096, 1):.2f} GB")  # → 0.63 GB
# Batch=1, seq_len=128K (long context)
print(f"{kv_cache_memory_gb(80, 8, 128, 131072, 1):.2f} GB")  # → 20 GB !
# Batch=32, seq_len=4K
print(f"{kv_cache_memory_gb(80, 8, 128, 4096, 32):.2f} GB")  # → 20 GB
# Without GQA (64 KV heads instead of 8): 8× more!
print(f"{kv_cache_memory_gb(80, 64, 128, 4096, 1):.2f} GB")  # → 5 GB
GQA is an ops lifesaver. Grouped Query Attention (using fewer KV heads than Q heads) reduces KV-cache by a factor of n_heads/n_kv_heads. LLaMA-2 70B uses 8 KV heads vs. 64 Q heads → 8× KV-cache reduction. This is the single biggest reason GQA is standard in modern models.

3. Batch Size → Throughput vs. Latency Trade-off

Increasing batch size improves GPU utilization (better arithmetic intensity) but increases latency and memory. For inference serving:

  • Small batch (1–4): Low latency, low GPU utilization (~5-15%), memory-bandwidth bound
  • Medium batch (8–32): Balanced; typical for real-time serving with continuous batching
  • Large batch (64+): High throughput; good for offline processing; may need more GPUs for KV-cache
# Total inference memory = model weights + KV-cache + activations
def total_inference_memory_gb(num_params_B, prec_bytes,
                              L, n_kv, d_head, seq_len, batch,
                              kv_prec=2):
    weights = num_params_B * 1e9 * prec_bytes / 1e9
    kv = kv_cache_memory_gb(L, n_kv, d_head, seq_len, batch, kv_prec)
    activations = 0.5 * batch  # rough estimate in GB
    return weights + kv + activations

# LLaMA-2 70B, FP16, batch=16, seq_len=4K
mem = total_inference_memory_gb(70, 2, 80, 8, 128, 4096, 16)
print(f"Total: {mem:.1f} GB → need {math.ceil(mem / 68):.0f}× A100-80GB")
# → Total: ~158 GB → need 3× A100-80GB

Sizing Models in Practice

Here's a complete function to compute model parameter counts from architecture configs — the kind of thing you'd use during capacity planning:

def count_transformer_params(
    num_layers,       # L: number of decoder layers
    hidden_dim,       # h: model dimension
    num_heads,        # n_h: query heads
    num_kv_heads,     # n_kv: key/value heads (GQA)
    vocab_size,       # V: vocabulary size
    intermediate_dim, # d_ff: FFN intermediate dimension
    tie_embeddings=False
):
    head_dim = hidden_dim // num_heads

    # === Embedding layer ===
    embed = vocab_size * hidden_dim

    # === Per-layer attention ===
    q_proj = hidden_dim * (num_heads * head_dim)        # = h²
    k_proj = hidden_dim * (num_kv_heads * head_dim)     # = h × (n_kv × d_h)
    v_proj = hidden_dim * (num_kv_heads * head_dim)
    o_proj = (num_heads * head_dim) * hidden_dim        # = h²
    attn_per_layer = q_proj + k_proj + v_proj + o_proj

    # === Per-layer FFN (SwiGLU) ===
    gate_proj = hidden_dim * intermediate_dim
    up_proj   = hidden_dim * intermediate_dim
    down_proj = intermediate_dim * hidden_dim
    ffn_per_layer = gate_proj + up_proj + down_proj

    # === Per-layer norms (RMSNorm: just a scale vector) ===
    norm_per_layer = 2 * hidden_dim  # pre-attn + pre-FFN

    # === Total ===
    per_layer = attn_per_layer + ffn_per_layer + norm_per_layer
    all_layers = per_layer * num_layers
    final_norm = hidden_dim  # final RMSNorm
    lm_head = 0 if tie_embeddings else hidden_dim * vocab_size

    total = embed + all_layers + final_norm + lm_head
    return {
        "embedding": embed,
        "attn_per_layer": attn_per_layer,
        "ffn_per_layer": ffn_per_layer,
        "norm_per_layer": norm_per_layer,
        "total_per_layer": per_layer,
        "all_layers": all_layers,
        "lm_head": lm_head,
        "total": total,
    }

# === Verify against known model sizes ===
# LLaMA-2 7B
p = count_transformer_params(32, 4096, 32, 32, 32000, 11008)
print(f"LLaMA-2 7B:  {p['total']/1e9:.2f}B")   # → ~6.74B

# LLaMA-2 70B
p = count_transformer_params(80, 8192, 64, 8, 32000, 28672)
print(f"LLaMA-2 70B: {p['total']/1e9:.2f}B")   # → ~68.98B

# LLaMA-3 8B
p = count_transformer_params(32, 4096, 32, 8, 128256, 14336)
print(f"LLaMA-3 8B:  {p['total']/1e9:.2f}B")   # → ~8.03B
Interview tip: When asked "how would you estimate the GPU requirements for model X?", walk through this sequence: (1) Count parameters from architecture config, (2) Multiply by bytes-per-param for your precision, (3) Add KV-cache for target batch × sequence length, (4) Add ~15% overhead for CUDA context and activations, (5) Divide by usable GPU memory. This systematic approach demonstrates you understand the architecture at an operational level.

Summary: What to Remember

  • FFN dominates parameters (~⅔ of each layer) — this determines model download size and weight memory
  • Attention dominates runtime memory for long sequences due to KV-cache growing with O(batch × seq_len × layers)
  • hidden_dim has quadratic impact on model size — doubling h roughly 4× parameters
  • GQA is essential for practical inference — reduces KV-cache by n_heads/n_kv_heads
  • Quantization is your biggest lever — FP16→INT4 cuts memory 4×, often with <1% quality loss
  • Decoding is bandwidth-bound — more FLOPS doesn't help; you need more memory bandwidth
  • Know your hardware: A100 (80 GB, 2 TB/s), H100 (80 GB, 3.35 TB/s), H200 (141 GB, 4.8 TB/s)
