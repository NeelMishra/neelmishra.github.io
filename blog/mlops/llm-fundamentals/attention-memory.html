Attention Memory Costs: Why Context Length Matters for Infra

MLOps Series LLM Fundamentals

Context length is the single most impactful parameter for LLM infrastructure costs. Moving from 4K to 128K tokens doesn't just use 32× more memory — the attention mechanism's quadratic scaling means the actual memory and compute costs can grow by 1,024× for the attention matrices alone. This post breaks down exactly why, how the KV cache dominates GPU memory during inference, and what this means if you're building or operating serving infrastructure.

Self-Attention: The O(n²) Problem

In standard multi-head self-attention, every token attends to every other token. For a sequence of length n, this creates an attention score matrix of shape [n × n] per head. With h heads, you store h × n × n float values just for the attention weights.

The core computation is:

Attention(Q, K, V) = softmax( Q KT / √dk ) · V

The Q KT product produces the [n × n] matrix. This is where the quadratic cost lives — both in FLOPs and in memory. 

# Self-attention: the quadratic bottleneck
def self_attention(Q, K, V, d_k):
    # Q, K, V: [batch, heads, seq_len, head_dim]
    # This matmul creates an [n × n] matrix — O(n²) memory
    scores = matmul(Q, K.transpose(-2, -1)) / sqrt(d_k)
    weights = softmax(scores, dim=-1)   # [batch, heads, n, n]
    output = matmul(weights, V)          # [batch, heads, n, d_k]
    return output

Let's put concrete numbers on this. For a single forward pass through one attention layer:

Sequence Length Attention Matrix Size (per head) Memory (32 heads, FP16) Relative Cost
2,0484M entries~256 MB1×
8,19267M entries~4 GB16×
32,7681.07B entries~64 GB256×
131,07217.2B entries~1,024 GB4,096×
Key insight: At 128K context, the raw attention matrices alone would need over 1 TB of memory per layer — far exceeding any single GPU. This is why FlashAttention (which tiles the computation and never materializes the full matrix) is essential for long-context models.

KV Cache Mechanics

During autoregressive inference (generating tokens one at a time), you don't recompute attention from scratch for every new token. Instead, you cache the Key and Value projections from all previous tokens — the KV cache.

For each new token generated, you only compute a fresh Q vector, then attend over the entire cached K and V. This turns the per-step cost from O(n²) to O(n) — but the cache itself grows linearly with every token generated. 

# KV cache during autoregressive generation
class KVCache:
    def __init__(self, num_layers, num_heads, head_dim, max_seq_len, dtype):
        # Pre-allocate cache for K and V across all layers
        self.k_cache = zeros(num_layers, max_seq_len, num_heads, head_dim, dtype=dtype)
        self.v_cache = zeros(num_layers, max_seq_len, num_heads, head_dim, dtype=dtype)
        self.seq_pos = 0

    def update(self, layer_idx, new_k, new_v):
        # Append new K, V at current position
        self.k_cache[layer_idx, self.seq_pos] = new_k
        self.v_cache[layer_idx, self.seq_pos] = new_v
        self.seq_pos += 1

    def get_kv(self, layer_idx):
        # Return all cached K, V up to current position
        return (
            self.k_cache[layer_idx, :self.seq_pos],
            self.v_cache[layer_idx, :self.seq_pos]
        )
Why this matters for infra: The KV cache is the dominant memory consumer during inference. Model weights are static and shared across requests, but KV cache is per-request and grows with every token. At 128K context, the KV cache for a single request on a 70B model can consume 40+ GB — more than an entire A100-40GB GPU.

The KV cache stores two tensors (K and V) at every layer. For models using Grouped Query Attention (GQA) — like Llama 2 70B (8 KV heads vs 64 query heads) — the cache is reduced by the grouping ratio, but it still scales linearly with sequence length.

The Memory Formula

The KV cache memory for a single request can be computed precisely. Here is the formula:

KV Cache = 2 × num_layers × 2 × hidden_dim × seq_len × batch_size × bytes_per_param

Breaking down each factor:

  • First 2: We store both K and V tensors
  • num_layers: Each transformer layer has its own KV cache
  • Second 2: Convention factor — some formulations fold this into hidden_dim as 2 × num_kv_heads × head_dim; here we separate it for clarity
  • hidden_dim: The model's hidden dimension (e.g., 8192 for Llama 2 70B)
  • seq_len: Current sequence length (prompt + generated tokens)
  • batch_size: Number of concurrent sequences
  • bytes_per_param: 2 for FP16/BF16, 1 for INT8, 0.5 for INT4
def kv_cache_memory_gb(
    num_layers,      # e.g., 80 for Llama 2 70B
    hidden_dim,      # e.g., 8192
    seq_len,         # e.g., 4096
    batch_size=1,    # concurrent requests
    bytes_per_param=2,  # FP16 = 2 bytes
    gqa_ratio=1      # 1 = MHA, 8 for Llama 2 70B (64/8)
):
    effective_dim = hidden_dim / gqa_ratio
    kv_bytes = 2 * num_layers * effective_dim * seq_len * batch_size * bytes_per_param
    return kv_bytes / (1024 ** 3)

# Example: Llama 2 70B (GQA with 8 KV heads, head_dim=128)
# effective_dim = 8192 / 8 = 1024
print(kv_cache_memory_gb(80, 8192, 4096, batch_size=1, gqa_ratio=8))
# → ~0.625 GB per request at 4K context

print(kv_cache_memory_gb(80, 8192, 131072, batch_size=1, gqa_ratio=8))
# → ~20 GB per request at 128K context

print(kv_cache_memory_gb(80, 8192, 131072, batch_size=8, gqa_ratio=8))
# → ~160 GB for 8 concurrent 128K requests

The batch size trap: A single 128K request on a 70B model uses ~20 GB of KV cache. Serving 32 concurrent users at 128K would require 640 GB just for KV cache — that's 8× A100-80GB GPUs dedicated entirely to caching, before you even count model weights.

Impact on GPU Memory & Throughput

GPU memory during inference is split between three consumers:

  1. Model weights — fixed, shared across all requests (e.g., 140 GB for 70B in FP16)
  2. KV cache — per-request, grows with sequence length
  3. Activation memory — temporary, relatively small during inference

The critical constraint: Total GPU Memory ≥ Weights + KV_Cache(batch × seq_len) + Activations. Since weights are fixed and activations are small, the KV cache determines how many concurrent requests (and at what context length) you can serve. 

# GPU memory budget calculation
def max_batch_size(
    gpu_memory_gb,
    model_weight_gb,
    kv_per_request_gb,
    activation_overhead_gb=2.0
):
    available = gpu_memory_gb - model_weight_gb - activation_overhead_gb
    if available <= 0:
        return 0
    return int(available / kv_per_request_gb)

# Llama 2 70B on 8× A100-80GB (640 GB total)
# Model weights in FP16: ~140 GB
print(max_batch_size(640, 140, 0.625))   # 4K ctx → 796 concurrent requests
print(max_batch_size(640, 140, 5.0))     # 32K ctx → 99 concurrent requests
print(max_batch_size(640, 140, 20.0))    # 128K ctx → 24 concurrent requests
Throughput cliff: Going from 4K → 128K context cuts your maximum concurrent batch from ~800 to ~24 on the same hardware. That's a 33× reduction in serving throughput. This is why long-context APIs are more expensive — you need proportionally more GPUs.

This creates a direct relationship between context length and cost-per-token:

Short Context (≤4K)

  • KV cache fits easily in GPU memory
  • Hundreds of concurrent requests per node
  • High GPU utilization (compute-bound)
  • Low cost per token

Long Context (32K–200K)

  • KV cache dominates GPU memory
  • Tens of concurrent requests per node
  • Low GPU utilization (memory-bound)
  • High cost per token

Why Providers Care About This

Companies like Anthropic and OpenAI operate serving clusters with thousands of GPUs. The KV cache memory problem directly dictates their infrastructure costs, pricing, and architectural choices.

Scheduling and Bin-Packing

Serving requests of mixed context lengths is a bin-packing problem. A 128K request occupies GPU memory that could serve 30+ short-context requests. Serving systems like vLLM use PagedAttention — borrowing virtual memory concepts from OS design — to allocate KV cache in non-contiguous blocks, reducing fragmentation from ~60% to under 4%. 

# PagedAttention: KV cache as virtual memory pages
# Instead of pre-allocating max_seq_len per request,
# allocate in small fixed-size blocks on demand

class PagedKVCache:
    BLOCK_SIZE = 16  # tokens per block

    def __init__(self, total_gpu_blocks):
        self.block_pool = list(range(total_gpu_blocks))
        self.page_table = {}  # request_id → [block_ids]

    def allocate(self, request_id, num_tokens):
        num_blocks = ceil(num_tokens / self.BLOCK_SIZE)
        blocks = [self.block_pool.pop() for _ in range(num_blocks)]
        self.page_table[request_id] = blocks

    def free(self, request_id):
        self.block_pool.extend(self.page_table.pop(request_id))

Pricing Asymmetry: Input vs. Output Tokens

Most API providers charge differently for input and output tokens. This reflects the KV cache economics:

  • Input (prefill): Processed in parallel, high compute utilization, KV cache built once — cheaper per token
  • Output (decode): Generated sequentially, one token at a time, KV cache read every step — more expensive due to memory bandwidth bottleneck

Why Context Length Tiers Exist

Claude's pricing for 200K context and GPT-4 Turbo's 128K option aren't just product features — they're infrastructure decisions. Longer context means:

  • Fewer concurrent requests per GPU → lower throughput → higher cost per token
  • More tensor parallelism needed to fit KV cache → more inter-GPU communication
  • Longer time-to-first-token (TTFT) due to larger prefill computation
  • Higher tail latency from memory bandwidth saturation
Interview insight: When asked "why do long-context requests cost more?", the answer isn't just "more tokens." It's that the KV cache memory pressure reduces batch sizes, underutilizes GPU compute, and shifts the bottleneck from compute to memory bandwidth — fundamentally changing the economics of serving.

Mitigation Strategies

The industry has developed several approaches to tame the attention memory problem:

1. Grouped Query Attention (GQA)

Instead of separate K, V heads for each query head, GQA groups multiple query heads to share a single KV head. Llama 2 70B uses 64 query heads with only 8 KV heads — an 8× reduction in KV cache size with minimal quality loss.

2. Multi-Query Attention (MQA)

The extreme version: all query heads share a single K and single V head. Used in PaLM and Falcon. Reduces KV cache by the number of heads (e.g., 32×) but can degrade quality on complex reasoning tasks.

3. FlashAttention

Doesn't reduce the theoretical compute — still O(n²) FLOPs — but avoids materializing the full [n × n] attention matrix in HBM. By tiling the computation and keeping intermediate results in SRAM (on-chip memory), it reduces memory usage from O(n²) to O(n) for the attention computation itself.

4. KV Cache Quantization

Quantize cached K and V values to INT8 or INT4, cutting memory by 2–4× with ~1% quality loss. Critical for production serving — most inference engines now support this. 

# KV cache quantization impact
def kv_cache_with_quantization(base_gb, quant_bits=8):
    # FP16 = 16 bits baseline
    ratio = 16 / quant_bits
    return base_gb / ratio

base = 20.0  # 128K context, 70B model, FP16
print(f"FP16: {base:.1f} GB")                                     # 20.0 GB
print(f"INT8: {kv_cache_with_quantization(base, 8):.1f} GB")      # 10.0 GB
print(f"INT4: {kv_cache_with_quantization(base, 4):.1f} GB")      #  5.0 GB

5. Sliding Window / Sparse Attention

Mistral uses a sliding window of 4,096 tokens — each token only attends to its nearest 4K neighbors. This caps the KV cache at a fixed size regardless of total context length, with periodic "sink" tokens that attend globally to preserve long-range information.

6. Prefix Caching

When multiple requests share the same system prompt (common in API serving), the KV cache for the shared prefix can be computed once and reused. For a 2K-token system prompt served to 1,000 users, this saves ~600 GB of redundant KV computation.

Interview Takeaways

When discussing attention memory in an infrastructure interview, anchor your answers in these concrete facts:

The numbers to know:
  • Self-attention is O(n²) in sequence length for compute and memory
  • KV cache grows linearly with sequence length per request, but it's per-request
  • A 70B model at 128K context uses ~20 GB of KV cache per request (GQA, FP16)
  • Going from 4K → 128K reduces concurrent batch capacity by ~33×
  • FlashAttention reduces attention memory from O(n²) to O(n) but not FLOPs
  • GQA reduces KV cache by the grouping ratio (8× for Llama 2 70B)
  • KV cache quantization to INT4 gives another 4× reduction

Key questions to be prepared for:

  • "Why is 128K context more expensive to serve than 4K?" — Memory-bound regime: KV cache dominates GPU memory, reducing batch sizes and shifting the bottleneck from compute to memory bandwidth.
  • "How would you estimate GPU requirements for serving a 70B model at 100K context?" — Use the KV cache formula, add model weights, account for GQA ratio and quantization, then divide by available GPU memory to get minimum node count.
  • "What's the difference between prefill and decode phases?" — Prefill is compute-bound (parallel attention over full prompt), decode is memory-bandwidth-bound (sequential generation reading entire KV cache per token).
  • "How does PagedAttention help?" — Eliminates KV cache memory fragmentation by allocating in fixed-size blocks, improving GPU memory utilization from ~40% to ~96%.

Understanding these memory constraints is foundational for any MLOps or infrastructure role. Every decision in LLM serving — from hardware provisioning to request scheduling to pricing strategy — flows from the simple fact that attention doesn't scale for free.
