    Logging for ML: Structured, Request/Response

    MLOps SeriesMonitoring & Observability

    Introduction

    Logging in machine learning systems is fundamentally different from traditional application logging. Every prediction request carries rich contextual data — feature vectors, model versions, latency breakdowns, and confidence scores — that must be captured in a structured, queryable format. Without disciplined logging, debugging model regressions, auditing predictions, and monitoring data drift become nearly impossible.

    Traditional print-statement logging breaks down in ML because the data is high-dimensional, the volume is enormous, and downstream consumers (monitoring dashboards, retraining pipelines, audit systems) need machine-readable formats. This post covers how to design structured logging for ML services, capture request/response pairs efficiently, apply sampling strategies to control costs, and set up storage and analysis pipelines.

    Key principle: Every ML log entry should answer three questions — what was the input, what did the model produce, and what was the system state at the time of prediction.

    Structured Log Schemas for ML

    Structured logging replaces free-form text with JSON objects that have a consistent schema. For ML systems, the schema must capture model metadata, feature summaries, prediction outputs, and system telemetry in a single record.

    A well-designed ML log schema includes these core fields:

    • request_id — Unique identifier for tracing the prediction end-to-end
    • timestamp — ISO 8601 timestamp with timezone information
    • model_name and model_version — Exact model artifact that served the prediction
    • features — Summary or hash of input features (not raw PII)
    • prediction — Model output including class probabilities or regression values
    • latency_ms — End-to-end prediction latency
    • status — Success, fallback, or error state
    import json import logging import hashlib from datetime import datetime, timezone from dataclasses import dataclass, asdict from typing import Any, Dict, Optional # Define structured log schema as a dataclass @dataclass class MLLogEntry: request_id: str timestamp: str model_name: str model_version: str feature_hash: str prediction: Dict[str, Any] latency_ms: float status: str metadata: Optional[Dict[str, Any]] = None class StructuredMLLogger: def __init__(self, service_name: str): self.logger = logging.getLogger(service_name) self.logger.setLevel(logging.INFO) # JSON formatter for structured output handler = logging.StreamHandler() handler.setFormatter( logging.Formatter('%(message)s') ) self.logger.addHandler(handler) def compute_feature_hash(self, features: Dict) -> str: # Hash features to avoid logging PII feature_str = json.dumps(features, sort_keys=True) return hashlib.sha256(feature_str.encode()).hexdigest()[:16] def log_prediction(self, entry: MLLogEntry): self.logger.info(json.dumps(asdict(entry)))
    Warning: Never log raw feature values that contain PII (names, emails, addresses). Always hash or redact sensitive fields before writing to logs.

    Request/Response Logging

    Request/response logging captures the complete lifecycle of a prediction — from the raw API input through feature extraction, model inference, and final response. This is critical for debugging, A/B test analysis, and regulatory compliance.

    The logging pipeline should capture data at multiple stages:

    Request Phase

    • Raw API payload (sanitized)
    • Parsed feature vector
    • Feature store lookup results
    • Pre-processing transformations
    • Request metadata (headers, auth)

    Response Phase

    • Raw model output (logits/probs)
    • Post-processing results
    • Business logic overrides
    • Final API response payload
    • Latency breakdown per stage
    import time import uuid from functools import wraps from typing import Callable class RequestResponseLogger: def __init__(self, ml_logger: StructuredMLLogger): self.ml_logger = ml_logger def log_request(self, func: Callable) -> Callable: # Decorator to log full request/response cycle @wraps(func) def wrapper(*args, **kwargs): request_id = str(uuid.uuid4()) start_time = time.perf_counter() status = "success" prediction = {} try: result = func(*args, **kwargs) prediction = result return result except Exception as e: status = f"error: {type(e).__name__}" raise finally: latency = (time.perf_counter() - start_time) * 1000 features = kwargs.get("features", {}) entry = MLLogEntry( request_id=request_id, timestamp=datetime.now(timezone.utc).isoformat(), model_name=kwargs.get("model_name", "unknown"), model_version=kwargs.get("model_version", "unknown"), feature_hash=self.ml_logger.compute_feature_hash(features), prediction=prediction, latency_ms=round(latency, 2), status=status, ) self.ml_logger.log_prediction(entry) return wrapper # Usage with a prediction endpoint logger = StructuredMLLogger("recommendation-service") rr_logger = RequestResponseLogger(logger) @rr_logger.log_request def predict(features: Dict, model_name: str, model_version: str): # Model inference happens here scores = model.predict(features) return {"scores": scores.tolist(), "top_k": scores.argsort()[-5:].tolist()}

    Sampling Strategies

    At production scale, logging every prediction is often cost-prohibitive. A recommendation system handling 50,000 requests per second would generate terabytes of logs daily if every request were captured. Sampling strategies let you reduce volume while preserving statistical validity and ensuring edge cases are captured.

    Common Sampling Approaches

    Uniform Random Sampling

    Log a fixed percentage of all requests. Simple and unbiased but may miss rare events. Typical rates: 1-10% for high-traffic services.

    Stratified Sampling

    Sample at different rates per segment (model version, user cohort, prediction class). Ensures minority classes are represented.

    Error-Biased Sampling

    Log 100% of errors, timeouts, and low-confidence predictions. Sample normally for successful predictions. Captures the most debugging value.

    Adaptive Sampling

    Dynamically adjust sampling rate based on system load and anomaly signals. Increase sampling when drift is detected.

    import random import hashlib from typing import Dict class MLLogSampler: def __init__(self, base_rate: float = 0.05): self.base_rate = base_rate self.error_rate = 1.0 # Always log errors self.low_confidence_rate = 0.5 # 50% for uncertain predictions self.confidence_threshold = 0.6 def should_log(self, prediction: Dict, status: str) -> bool: # Always log errors and exceptions if status != "success": return random.random() < self.error_rate # Higher sampling rate for low-confidence predictions confidence = prediction.get("confidence", 1.0) if confidence < self.confidence_threshold: return random.random() < self.low_confidence_rate # Deterministic sampling by request_id for consistency return random.random() < self.base_rate def deterministic_sample(self, request_id: str, rate: float) -> bool: # Same request_id always gets same decision hash_val = int(hashlib.md5(request_id.encode()).hexdigest(), 16) return (hash_val % 10000) < int(rate * 10000)
    Tip: Use deterministic sampling (hash-based) when you need consistent behavior across distributed services — the same request ID should be sampled or skipped everywhere.

    Storage and Analysis

    ML logs generate massive volumes that require purpose-built storage and analysis infrastructure. The storage tier must support both real-time queries (debugging a live issue) and batch analysis (computing weekly drift reports).

    Storage Architecture

    A typical ML logging pipeline flows through three tiers:

    1. Hot tier (0–24 hours): In-memory or fast SSD storage for real-time queries. Elasticsearch or ClickHouse work well here.
    2. Warm tier (1–30 days): Columnar storage like Apache Parquet on object storage (S3/GCS). Queryable via Athena, BigQuery, or Spark.
    3. Cold tier (30+ days): Compressed archival storage for compliance. Cheap object storage with lifecycle policies.
    import json from datetime import datetime from typing import List class LogStorageRouter: # Route logs to appropriate storage tier def __init__(self, hot_client, warm_client, cold_client): self.hot = hot_client self.warm = warm_client self.cold = cold_client def route_batch(self, logs: List[Dict]): for log in logs: # Always write to hot tier first self.hot.write(log) # Write errors and anomalies to warm tier immediately if log["status"] != "success" or log.get("anomaly_flag"): self.warm.write(log) def compact_to_parquet(self, date_partition: str): # Batch job: compact hot tier logs to Parquet logs = self.hot.query( date=date_partition, format="json" ) parquet_buffer = self.convert_to_parquet(logs) self.warm.upload( path=f"logs/{date_partition}/predictions.parquet", data=parquet_buffer ) def convert_to_parquet(self, logs: List[Dict]): import pyarrow as pa import pyarrow.parquet as pq table = pa.Table.from_pylist(logs) return pq.write_table(table)

    Implementation Guide

    Putting it all together, here is a complete implementation that combines structured logging, request/response capture, sampling, and storage routing into a production-ready ML logging system.

    import json import time import uuid import logging from typing import Dict, Any, Optional from dataclasses import dataclass, field class ProductionMLLogger: # Complete ML logging system with sampling and routing def __init__(self, config: Dict): self.service_name = config["service_name"] self.sampler = MLLogSampler( base_rate=config.get("sample_rate", 0.05) ) self.logger = StructuredMLLogger(self.service_name) self.buffer = [] self.buffer_size = config.get("buffer_size", 100) def log_inference(self, request: Dict, response: Dict, model_name: str, model_version: str): request_id = str(uuid.uuid4()) status = response.get("status", "success") # Apply sampling strategy if not self.sampler.should_log(response, status): return entry = MLLogEntry( request_id=request_id, timestamp=datetime.now(timezone.utc).isoformat(), model_name=model_name, model_version=model_version, feature_hash=self.logger.compute_feature_hash(request), prediction=response, latency_ms=response.get("latency_ms", 0), status=status, metadata={ "feature_count": len(request), "model_type": response.get("model_type"), } ) self.buffer.append(entry) # Flush buffer when full if len(self.buffer) >= self.buffer_size: self.flush() def flush(self): for entry in self.buffer: self.logger.log_prediction(entry) self.buffer = []

    Best Practices

    After deploying structured ML logging across multiple production services, several patterns consistently prove their value:

    • Separate ML logs from application logs. Model prediction data has different retention, access patterns, and consumers than standard application logs. Use dedicated log streams.
    • Include model lineage in every log entry. Model name, version, training data hash, and experiment ID make it possible to trace any prediction back to its exact origin.
    • Use correlation IDs across services. A single user request often touches multiple ML models. Propagate a shared correlation ID so you can reconstruct the full prediction path.
    • Log feature distributions, not just values. Periodic summary statistics (mean, variance, percentiles) of feature distributions are more useful for drift detection than individual values.
    • Set up automated log schema validation. Reject malformed log entries at write time rather than discovering schema drift during analysis.
    • Implement log-level controls per model. Allow operators to increase logging verbosity for a specific model version without affecting other models in the system.
    Remember: Good ML logging is an investment that pays off during every incident, every model retraining, and every compliance audit. The cost of structured logging is always less than the cost of debugging blind.
