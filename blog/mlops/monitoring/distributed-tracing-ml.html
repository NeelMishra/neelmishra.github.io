    Distributed Tracing for ML Pipelines

    MLOps SeriesMonitoring & Observability

    Introduction

    A single ML prediction often traverses multiple services — an API gateway, a feature store, one or more model servers, a post-processing layer, and a caching tier. When latency spikes or errors occur, pinpointing the root cause without distributed tracing is like debugging a distributed system with print statements: technically possible, but agonizingly slow.

    Distributed tracing gives you end-to-end visibility into every prediction request as it flows through your ML pipeline. Each service creates spans that record timing, metadata, and error information, linked together by a shared trace ID. This post covers how to instrument ML services with OpenTelemetry, propagate trace context across pipeline stages, and visualize traces with Jaeger.

    Why ML tracing is special: Unlike typical web services, ML pipelines have variable-latency stages (feature computation, model inference, post-processing) where performance depends on input data characteristics, not just system load.

    Why Tracing Matters for ML Systems

    ML systems introduce unique observability challenges that make distributed tracing especially valuable:

    • Multi-model orchestration: Ensemble predictions, cascading models, and routing logic create complex call graphs that are invisible without tracing.
    • Feature computation latency: Feature store lookups, real-time feature engineering, and data transformations often dominate end-to-end latency but are opaque to API-level monitoring.
    • Model version routing: A/B tests, canary deployments, and shadow mode run multiple model versions simultaneously. Traces reveal which version handled each request.
    • Async pipeline stages: Batch pre-computation, cache warming, and asynchronous post-processing create timing relationships that only tracing can capture.

    Without Tracing

    • Latency spikes are visible but unexplained
    • Error attribution requires log correlation
    • Feature store timeouts are invisible
    • Model version impact is hard to isolate

    With Tracing

    • Per-stage latency breakdown in every request
    • Errors pinpointed to exact service and span
    • Feature store latency as a first-class metric
    • Model version tagged on every inference span

    OpenTelemetry Setup for ML Services

    OpenTelemetry (OTel) is the industry standard for distributed tracing. It provides vendor-neutral APIs for creating spans, propagating context, and exporting telemetry data. For ML services, we instrument four key stages: request ingestion, feature computation, model inference, and response assembly.

    from opentelemetry import trace from opentelemetry.sdk.trace import TracerProvider from opentelemetry.sdk.trace.export import BatchSpanProcessor from opentelemetry.exporter.jaeger.thrift import JaegerExporter from opentelemetry.sdk.resources import Resource def setup_tracing(service_name: str, jaeger_host: str = "localhost"): # Configure the tracer provider with service metadata resource = Resource.create({ "service.name": service_name, "service.version": "1.2.0", "deployment.environment": "production", }) provider = TracerProvider(resource=resource) # Export spans to Jaeger via Thrift protocol jaeger_exporter = JaegerExporter( agent_host_name=jaeger_host, agent_port=6831, ) processor = BatchSpanProcessor(jaeger_exporter) provider.add_span_processor(processor) # Set global tracer provider trace.set_tracer_provider(provider) return trace.get_tracer(service_name) # Initialize tracer for the ML service tracer = setup_tracing("ml-inference-service")

    With the tracer configured, we can now instrument each stage of the ML pipeline:

    from opentelemetry import trace from opentelemetry.trace import StatusCode from typing import Dict, Any class TracedMLService: def __init__(self, tracer, model, feature_store): self.tracer = tracer self.model = model self.feature_store = feature_store def predict(self, request: Dict[str, Any]) -> Dict: with self.tracer.start_as_current_span("ml.predict") as span: span.set_attribute("ml.request_id", request["id"]) # Stage 1: Feature retrieval features = self.get_features(request) # Stage 2: Model inference prediction = self.run_inference(features) # Stage 3: Post-processing result = self.post_process(prediction) span.set_attribute("ml.model_name", self.model.name) span.set_attribute("ml.confidence", result["confidence"]) return result def get_features(self, request: Dict) -> Dict: with self.tracer.start_as_current_span("ml.feature_retrieval") as span: span.set_attribute("ml.feature_store", "redis") features = self.feature_store.get(request["user_id"]) span.set_attribute("ml.feature_count", len(features)) return features def run_inference(self, features: Dict) -> Dict: with self.tracer.start_as_current_span("ml.inference") as span: span.set_attribute("ml.model_version", self.model.version) span.set_attribute("ml.model_framework", "pytorch") try: result = self.model.predict(features) span.set_status(StatusCode.OK) return result except Exception as e: span.set_status(StatusCode.ERROR, str(e)) span.record_exception(e) raise def post_process(self, prediction: Dict) -> Dict: with self.tracer.start_as_current_span("ml.post_process") as span: # Apply business rules, thresholding, formatting result = self.apply_threshold(prediction) span.set_attribute("ml.post_process_applied", True) return result

    Trace Context Propagation

    In a microservices architecture, ML predictions often cross service boundaries — an API gateway calls a feature service, which calls a model server, which calls a post-processing service. Trace context propagation ensures all these spans are linked into a single trace.

    OpenTelemetry supports automatic context propagation via HTTP headers (W3C TraceContext format) and manual propagation for non-HTTP transports like message queues and gRPC.

    from opentelemetry.context import attach, detach from opentelemetry.propagate import inject, extract from opentelemetry import context as otel_context import requests class TracedHTTPClient: # Propagate trace context in HTTP requests def __init__(self, tracer): self.tracer = tracer self.session = requests.Session() def call_model_service(self, endpoint: str, payload: Dict): with self.tracer.start_as_current_span("http.model_call") as span: headers = {} # Inject trace context into HTTP headers inject(headers) span.set_attribute("http.url", endpoint) span.set_attribute("http.method", "POST") response = self.session.post( endpoint, json=payload, headers=headers, timeout=5.0 ) span.set_attribute("http.status_code", response.status_code) return response.json() class TracedKafkaProducer: # Propagate trace context through Kafka messages def __init__(self, tracer, producer): self.tracer = tracer self.producer = producer def send_prediction(self, topic: str, prediction: Dict): with self.tracer.start_as_current_span("kafka.produce") as span: headers = {} inject(headers) # Attach trace headers to Kafka message kafka_headers = [ (k, v.encode()) for k, v in headers.items() ] self.producer.send( topic, value=prediction, headers=kafka_headers, ) span.set_attribute("messaging.destination", topic)
    Common pitfall: Async workers (Celery, Kafka consumers) lose trace context by default. You must manually extract and re-attach context when processing messages in background workers.

    Jaeger Integration

    Jaeger is a popular open-source distributed tracing backend that stores, indexes, and visualizes traces. For ML systems, Jaeger's trace comparison feature is particularly valuable — you can compare traces from different model versions or feature store configurations to identify performance regressions.

    Deployment Architecture

    A production Jaeger deployment for ML services typically includes:

    1. Jaeger Agent: Lightweight sidecar that receives spans via UDP and batches them to the collector. Deploy alongside each ML service pod.
    2. Jaeger Collector: Validates, indexes, and stores spans. Supports Kafka as an intermediate buffer for high-throughput ML services.
    3. Elasticsearch/Cassandra: Backend storage for span data. Elasticsearch is preferred for ML workloads due to its flexible query capabilities.
    4. Jaeger Query + UI: Web interface for searching, viewing, and comparing traces.

    Implementation Patterns

    Beyond basic instrumentation, ML pipelines benefit from several advanced tracing patterns:

    Custom Span Attributes for ML

    Standard HTTP tracing attributes are insufficient for ML. Define custom semantic conventions that capture model-specific metadata:

    # Custom ML span attributes convention ML_ATTRIBUTES = { "ml.model.name": "recommendation-v2", "ml.model.version": "2.3.1", "ml.model.framework": "pytorch", "ml.feature.count": 128, "ml.feature.source": "redis-feature-store", "ml.inference.batch_size": 1, "ml.inference.device": "gpu:0", "ml.prediction.confidence": 0.92, "ml.prediction.class": "positive", "ml.experiment.id": "exp-2024-001", } def add_ml_attributes(span, model, prediction): # Attach ML-specific attributes to the active span span.set_attribute("ml.model.name", model.name) span.set_attribute("ml.model.version", model.version) span.set_attribute("ml.prediction.confidence", float(prediction["confidence"])) span.set_attribute("ml.prediction.latency_ms", prediction["latency_ms"])

    Trace-Based Sampling for ML

    from opentelemetry.sdk.trace.sampling import Sampler, Decision, SamplingResult from opentelemetry.trace import SpanKind from typing import Sequence class MLAwareSampler(Sampler): # Custom sampler that prioritizes slow and error traces def __init__(self, base_rate: float = 0.1): self.base_rate = base_rate def should_sample(self, parent_context, trace_id, name, kind, attributes, links): # Always sample error spans if attributes and attributes.get("error"): return SamplingResult(Decision.RECORD_AND_SAMPLE) # Always sample spans with low model confidence confidence = attributes.get("ml.prediction.confidence", 1.0) \ if attributes else 1.0 if confidence < 0.5: return SamplingResult(Decision.RECORD_AND_SAMPLE) # Probabilistic sampling for normal traces if (trace_id % 10000) < int(self.base_rate * 10000): return SamplingResult(Decision.RECORD_AND_SAMPLE) return SamplingResult(Decision.DROP) def get_description(self): return f"MLAwareSampler(rate={self.base_rate})"
    Performance tip: Use BatchSpanProcessor instead of SimpleSpanProcessor in production. Batching reduces the overhead per prediction from ~100μs to ~5μs by buffering and sending spans asynchronously.

    Best Practices

    Effective distributed tracing for ML systems requires attention to several key areas:

    • Name spans semantically. Use ml.inference, ml.feature_retrieval, ml.post_process — not generic names like step1 or process. Good span names make traces self-documenting.
    • Record model version on every inference span. This is the single most important attribute for debugging model regressions. When latency or accuracy degrades, filtering by model version instantly narrows the search space.
    • Set appropriate span kinds. Use SpanKind.CLIENT for outgoing calls, SpanKind.SERVER for incoming requests, and SpanKind.INTERNAL for in-process stages like feature transformation.
    • Add events for significant state changes. Use span.add_event() to mark cache hits/misses, fallback activations, and threshold crossings within a span.
    • Implement tail-based sampling. Head-based sampling decides at trace start whether to sample, potentially discarding interesting traces. Tail-based sampling waits until the trace completes, ensuring slow or error traces are always captured.
    • Monitor tracing overhead. Instrument the instrumentation — track the CPU and memory overhead of your tracing setup. For low-latency ML services (sub-10ms), even microseconds of overhead matter.
    • Correlate traces with ML metrics. Link trace IDs to model performance metrics (accuracy, drift scores) so you can drill down from a dashboard anomaly to specific problematic requests.
    Getting started: Begin by instrumenting the model inference span alone. Once that's working end-to-end in Jaeger, expand to feature retrieval, post-processing, and cross-service propagation.
