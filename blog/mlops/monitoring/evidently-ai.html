    Evidently AI: Open Source ML Monitoring

    MLOps SeriesMonitoring & Observability

    Introduction

    Machine learning models in production are not static software artifacts. Their performance degrades silently as the data they consume shifts away from the distribution they were trained on. Generic application monitoring tools track latency and error rates, but they are blind to the statistical health of your model. A prediction endpoint can return HTTP 200 on every request while quietly producing garbage outputs because the input distribution has drifted, a feature pipeline has introduced nulls, or an upstream schema change has corrupted a categorical column.

    Evidently AI is an open-source Python library purpose-built for ML monitoring. It provides first-class primitives for detecting data drift, evaluating model performance, and validating data quality — all operating directly on pandas DataFrames. Unlike heavyweight platforms that require dedicated infrastructure, Evidently can run as a simple Python function inside your existing serving code, Airflow DAGs, or CI/CD pipelines.

    The library ships three core abstractions: Reports for interactive visual analysis, Test Suites for automated pass/fail validation, and a Monitoring UI for tracking metric evolution over time. Together they cover the full lifecycle — from ad-hoc exploration in a notebook to production alerting in a scheduled pipeline.

    Why not just Prometheus + Grafana? Traditional monitoring watches system metrics (CPU, memory, latency). ML monitoring must watch statistical metrics: feature distributions, prediction drift, target leakage, and class balance. Evidently bridges this gap.

    Evidently Overview & Architecture

    Evidently's architecture revolves around three pillars. Reports compute a set of metrics over a reference dataset and a current dataset, then render rich HTML visualizations. Test Suites wrap those same metrics in pass/fail conditions so you can automate data validation. The Monitoring UI (powered by a lightweight workspace server) stores snapshots of reports over time and renders trend dashboards.

    All three pillars operate on standard pandas DataFrames. You point Evidently at two DataFrames — a reference (typically your training or validation set) and a current (recent production data) — and it computes statistical tests, distance metrics, and aggregations automatically. No special data formats, no external databases, no schema definitions required.

    The engine accepts individual Metric objects (e.g., ColumnDriftMetric, DatasetSummaryMetric) or Metric Presets that bundle related metrics into a single convenience call. This composability means you can start with a preset like DataDriftPreset() and later add fine-grained column-level metrics as your monitoring matures.

    # Install evidently # pip install evidently import pandas as pd from evidently.report import Report from evidently.metric_preset import DataDriftPreset # Load reference and current data ref_df = pd.read_csv("reference_data.csv") cur_df = pd.read_csv("production_batch_2024_06.csv") # Quick sanity check print(f"Reference shape: {ref_df.shape}") print(f"Current shape: {cur_df.shape}") # Build and run a drift report report = Report(metrics=[DataDriftPreset()]) report.run(reference_data=ref_df, current_data=cur_df) report.save_html("quick_drift_check.html")

    Data Reports

    Reports are Evidently's primary exploration tool. Each report computes one or more metrics over your reference and current datasets, then produces interactive HTML with charts, tables, and statistical summaries. You can compose a report from individual metrics or use presets that bundle common analyses.

    The three most common report types are Data Drift (detecting distributional shift across features), Data Quality (missing values, duplicates, correlations), and Model Performance (accuracy, precision, recall, ROC curves, calibration). Each can be generated with a few lines of code and saved as standalone HTML files for sharing with stakeholders or archiving.

    Data Drift Report

    A data drift report compares the statistical distribution of each feature between reference and current data. Evidently automatically selects an appropriate statistical test based on column type and cardinality — for example, the Kolmogorov-Smirnov test for numerical features and the chi-squared test for categorical ones. You can override these defaults per column if needed.

    from evidently.report import Report from evidently.metric_preset import DataDriftPreset from evidently.metrics import ( ColumnDriftMetric, DataDriftTable, DatasetDriftMetric, ) # Fine-grained drift report with individual column metrics drift_report = Report(metrics=[ # Dataset-level drift summary DatasetDriftMetric(), # Per-column drift table DataDriftTable(), # Track specific critical features individually ColumnDriftMetric(column_name="age"), ColumnDriftMetric(column_name="income"), ColumnDriftMetric(column_name="credit_score"), ColumnDriftMetric(column_name="loan_amount"), ]) drift_report.run(reference_data=ref_df, current_data=cur_df) # Save as interactive HTML for stakeholders drift_report.save_html("reports/data_drift_june.html") # Save as JSON for programmatic consumption drift_report.save_json("reports/data_drift_june.json") # Extract results as a Python dictionary result = drift_report.as_dict() dataset_drift = result["metrics"][0]["result"]["dataset_drift"] print(f"Dataset drift detected: {dataset_drift}")

    Data Quality & Model Performance Reports

    Beyond drift, Evidently can profile your data quality — checking for missing values, constant columns, duplicated rows, and out-of-range values. The model performance report covers classification and regression metrics including confusion matrices, ROC-AUC, and error distributions.

    from evidently.metric_preset import ( DataQualityPreset, ClassificationPreset, RegressionPreset, ) # Data quality report — no reference needed for basic profiling quality_report = Report(metrics=[DataQualityPreset()]) quality_report.run(reference_data=ref_df, current_data=cur_df) quality_report.save_html("reports/quality_report.html") # Classification model performance (requires target + prediction columns) cls_report = Report(metrics=[ClassificationPreset()]) cls_report.run( reference_data=ref_df, current_data=cur_df, column_mapping=ColumnMapping( target="label", prediction="prediction", ), ) cls_report.save_html("reports/classification_perf.html") # Regression model performance reg_report = Report(metrics=[RegressionPreset()]) reg_report.run( reference_data=ref_df, current_data=cur_df, column_mapping=ColumnMapping( target="price", prediction="predicted_price", ), ) reg_report.save_html("reports/regression_perf.html")
    Tip: Use ColumnMapping to tell Evidently which columns are targets, predictions, numerical features, and categorical features. Without it, Evidently infers types automatically — which works well for most cases but can misclassify integer-encoded categoricals.

    Test Suites

    While reports are designed for visual exploration, Test Suites are designed for automation. A test suite runs a collection of checks against your data and returns a structured pass/fail result. This makes them ideal for CI/CD gates, scheduled validation jobs, and real-time data quality checks in serving pipelines.

    Each test wraps a metric with a condition. Evidently ships dozens of built-in tests covering drift, missing values, statistical properties, and model performance. You can also define custom conditions using lambdas or threshold values. When a test fails, Evidently provides a detailed explanation of what went wrong, making debugging straightforward.

    from evidently.test_suite import TestSuite from evidently.test_preset import DataDriftTestPreset from evidently.tests import ( TestColumnDrift, TestShareOfMissingValues, TestMeanInNSigmas, TestNumberOfRows, TestNumberOfColumns, TestShareOfOutRangeValues, ) # Build a comprehensive test suite suite = TestSuite(tests=[ # Structural checks TestNumberOfRows(gte=1000), TestNumberOfColumns(eq=15), # Missing value thresholds TestShareOfMissingValues(column_name="age", lte=0.02), TestShareOfMissingValues(column_name="income", lte=0.05), # Statistical stability TestMeanInNSigmas(column_name="age", n=3), TestMeanInNSigmas(column_name="income", n=3), # Drift on critical features TestColumnDrift(column_name="credit_score"), TestColumnDrift(column_name="loan_amount"), # Out-of-range detection TestShareOfOutRangeValues(column_name="age", lte=0.01), ]) suite.run(reference_data=ref_df, current_data=cur_df) # Inspect results programmatically result = suite.as_dict() for test in result["tests"]: status = test["status"] name = test["name"] if status == "FAIL": print(f"FAILED: {name} — {test['description']}") # Save HTML report of test results suite.save_html("test_results/validation_june.html")

    Using Test Suites in CI/CD

    The real power of test suites emerges when you integrate them into automated pipelines. A common pattern is to run a test suite against each new data batch before it enters your feature store or model training pipeline. If any test fails, the pipeline halts and alerts the data engineering team.

    import sys from evidently.test_suite import TestSuite from evidently.test_preset import DataDriftTestPreset, DataQualityTestPreset def validate_data_batch(ref_path, batch_path): """Run data validation as a CI/CD gate.""" ref_df = pd.read_parquet(ref_path) batch_df = pd.read_parquet(batch_path) suite = TestSuite(tests=[ DataDriftTestPreset(), DataQualityTestPreset(), ]) suite.run(reference_data=ref_df, current_data=batch_df) # Save report artifact regardless of outcome suite.save_html("artifacts/validation_report.html") # Exit with non-zero code on failure result = suite.as_dict() failed = [t for t in result["tests"] if t["status"] == "FAIL"] if failed: print(f"Validation FAILED: {len(failed)} tests failed") sys.exit(1) print("All validation tests passed.") if __name__ == "__main__": validate_data_batch( ref_path="data/reference.parquet", batch_path="data/latest_batch.parquet", )
    Warning: Test suites with many per-column tests on wide datasets (100+ features) can be slow. Consider running drift tests only on your most critical features and using the DataDriftTestPreset with a stattest_threshold parameter to control sensitivity.

    Dashboard Setup

    Reports and test suites give you point-in-time snapshots. The Evidently Monitoring UI lets you track how those metrics evolve over time. It works through three concepts: a Workspace (a directory on disk or a remote store), a Project (a logical grouping for one model), and Snapshots (timestamped report or test suite results logged to the project).

    You create a workspace, add a project for each model you want to monitor, and then log snapshots on a schedule — for example, after every prediction batch, every hour, or every day. The UI renders line charts, bar charts, and heatmaps so you can see trends like creeping drift or gradual accuracy degradation.

    from evidently.ui.workspace import Workspace, WorkspaceBase from evidently.ui.dashboards import ( DashboardConfig, PanelValue, ReportFilter, CounterAgg, ) from evidently.report import Report from evidently.metric_preset import DataDriftPreset, DataQualityPreset import datetime # Step 1: Create a workspace and project ws = Workspace.create("monitoring_workspace") project = ws.create_project("Loan Default Model") project.description = "Production monitoring for the loan default classifier." # Step 2: Configure dashboard panels project.dashboard = DashboardConfig( name="Loan Model Dashboard", panels=[ PanelValue( title="Dataset Drift (share of drifted features)", filter=ReportFilter(metadata_values={}, tag_values=[]), field_path="DatasetDriftMetric", legend="Drift Share", agg=CounterAgg.LAST, ), PanelValue( title="Missing Values Rate", filter=ReportFilter(metadata_values={}, tag_values=[]), field_path="DatasetMissingValuesMetric", legend="Missing %", agg=CounterAgg.LAST, ), ], ) project.save()

    Logging Snapshots Over Time

    Once the project is configured, you log snapshots on each batch or at regular intervals. Each snapshot contains the full report results and a timestamp. The dashboard aggregates these into time-series panels automatically.

    import glob import pandas as pd from evidently.report import Report from evidently.metric_preset import DataDriftPreset from evidently.ui.workspace import Workspace ws = Workspace("monitoring_workspace") project = ws.search_project("Loan Default Model")[0] ref_df = pd.read_parquet("data/reference.parquet") # Iterate over daily production batches and log snapshots batch_files = sorted(glob.glob("data/production/batch_*.parquet")) for batch_file in batch_files: cur_df = pd.read_parquet(batch_file) report = Report(metrics=[DataDriftPreset()]) report.run(reference_data=ref_df, current_data=cur_df) # Log the snapshot to the project ws.add_report(project.id, report) print(f"Logged snapshot for {batch_file}") # Launch the monitoring UI server # evidently ui --workspace monitoring_workspace
    Storage: By default, snapshots are stored as JSON files in the workspace directory. For production use at scale, consider pointing the workspace at cloud object storage (S3, GCS) and using a database backend for metadata.

    Integration Patterns

    Evidently's value multiplies when it is embedded into your production infrastructure rather than run ad-hoc. The three most common integration patterns are: (1) serving middleware that runs checks on every prediction batch, (2) orchestrator tasks inside Airflow or Prefect DAGs, and (3) metric export to Grafana/Prometheus for unified observability.

    FastAPI Middleware

    The simplest real-time integration is a FastAPI endpoint that collects recent predictions into a buffer and periodically runs Evidently checks. When drift is detected, you can log an alert, return a warning header, or trigger a retraining pipeline.

    from fastapi import FastAPI, BackgroundTasks import pandas as pd from evidently.report import Report from evidently.metric_preset import DataDriftPreset import logging app = FastAPI() request_buffer = [] ref_df = pd.read_parquet("data/reference.parquet") BUFFER_SIZE = 500 logger = logging.getLogger("ml_monitor") def run_drift_check(buffer_data): """Background task: run Evidently drift report.""" cur_df = pd.DataFrame(buffer_data) report = Report(metrics=[DataDriftPreset()]) report.run(reference_data=ref_df, current_data=cur_df) result = report.as_dict() drift_detected = result["metrics"][0]["result"]["dataset_drift"] if drift_detected: logger.warning("DATA DRIFT DETECTED — triggering alert") # send_alert_to_slack(result) # trigger_retraining_pipeline() report.save_html("monitoring/latest_drift_check.html") @app.post("/predict") async def predict(features: dict, bg: BackgroundTasks): prediction = model.predict(features) request_buffer.append(features) # Trigger drift check when buffer fills up if len(request_buffer) >= BUFFER_SIZE: bg.add_task(run_drift_check, request_buffer.copy()) request_buffer.clear() return {"prediction": prediction}

    Integration Comparison

    🔄 Serving Middleware

    When: Real-time or near-real-time monitoring on every prediction batch.

    Pros: Immediate drift detection, tight coupling with serving logic, can trigger alerts in seconds.

    Cons: Adds latency to the serving path (mitigate with background tasks), buffer management complexity.

    Tools: FastAPI BackgroundTasks, Flask after_request hooks, or async workers.

    📅 Orchestrator DAGs

    When: Batch monitoring on a schedule (hourly, daily, weekly).

    Pros: Clean separation of concerns, built-in retry/alerting, no serving overhead.

    Cons: Delayed detection (only catches drift at next scheduled run), requires orchestrator infrastructure.

    Tools: Airflow, Prefect, Dagster, or cron jobs with the Evidently CLI.

    📊 Grafana Export

    When: You want ML metrics in your existing observability stack.

    Pros: Unified dashboards, reuse existing alerting rules, familiar UI for ops teams.

    Cons: Requires metric translation layer, loses rich HTML report visualizations.

    Tools: Prometheus client, StatsD, or Evidently's built-in Prometheus collector.

    🧪 CI/CD Gates

    When: Validating training data or model artifacts before deployment.

    Pros: Prevents bad models from reaching production, integrates with existing CI workflows.

    Cons: Only checks at deploy time, not runtime. Requires curated reference datasets.

    Tools: GitHub Actions, GitLab CI, Jenkins — run test suites as pipeline steps.

    Airflow DAG Integration

    For batch monitoring, an Airflow DAG that runs Evidently checks after each data ingestion is a robust pattern. The DAG pulls the latest batch from your data warehouse, runs a test suite, and logs the snapshot to the Evidently workspace. If tests fail, the DAG triggers an alert task.

    from airflow import DAG from airflow.operators.python import PythonOperator from datetime import datetime, timedelta def run_evidently_checks(**kwargs): import pandas as pd from evidently.test_suite import TestSuite from evidently.test_preset import DataDriftTestPreset ref_df = pd.read_parquet("s3://ml-data/reference.parquet") batch_df = pd.read_parquet(kwargs["batch_path"]) suite = TestSuite(tests=[DataDriftTestPreset()]) suite.run(reference_data=ref_df, current_data=batch_df) result = suite.as_dict() failed = [t for t in result["tests"] if t["status"] == "FAIL"] if failed: raise ValueError(f"{len(failed)} data quality tests failed") dag = DAG( "ml_data_validation", schedule_interval="@daily", start_date=datetime(2024, 1, 1), catchup=False, default_args={"retries": 2, "retry_delay": timedelta(minutes=5)}, ) validate_task = PythonOperator( task_id="validate_production_batch", python_callable=run_evidently_checks, op_kwargs={"batch_path": "s3://ml-data/daily/{{ ds }}.parquet"}, dag=dag, )

    Best Practices

    Deploying Evidently effectively requires more than just writing code. The following recommendations come from real-world experience running ML monitoring in production systems handling millions of predictions per day.

    Choosing the Right Metrics

    Not every feature deserves a drift test. Focus your monitoring budget on high-importance features — those with the highest feature importance scores in your model — and known-volatile features like date-derived or external-API-sourced columns. Monitoring every column on a 200-feature model will generate noise and false alarms. Start with your top 10–20 features and expand incrementally.

    Rule of thumb: If a feature's drift wouldn't change your model's predictions by more than 1%, consider monitoring it at a lower frequency (weekly instead of daily) or excluding it from automated alerts.

    Setting Test Thresholds

    Evidently's default statistical test thresholds are calibrated for general use, but your domain may require tuning. Financial models may need tighter thresholds (lower p-values, smaller drift windows), while recommendation systems can tolerate more variation. Run your test suite against historical data to establish a false-positive baseline before deploying to production.

    Common mistake: Setting thresholds too tight on launch day. This leads to alert fatigue and teams ignoring legitimate warnings. Start with loose thresholds and tighten them over 2–4 weeks as you learn your system's natural variation.

    Managing Report Storage

    HTML reports are convenient for sharing but consume significant storage over time. A single report for a 100-feature dataset can be 2–5 MB. At daily cadence, that adds up to 1–2 GB per year per model. Consider a tiered strategy: keep the last 30 days of HTML reports for quick access, archive JSON results to object storage for long-term analysis, and configure automatic cleanup of old HTML files.

    Scaling for High-Traffic Models

    For models serving thousands of requests per second, running a full Evidently report on every request is impractical. Instead, use sampling — collect every Nth request or a random 1–5% sample into a buffer, then run Evidently on the aggregated batch. This gives you statistical confidence without adding per-request overhead. Combine this with the FastAPI background task pattern shown in the integration section.

    Production checklist:
    ✅ Monitor top 10–20 features by importance
    ✅ Run test suites on every data batch before model training
    ✅ Log snapshots to a persistent workspace for trend analysis
    ✅ Set up alerts (Slack, PagerDuty) for critical test failures
    ✅ Review and retune thresholds quarterly
    ✅ Archive JSON results; auto-delete HTML reports older than 30 days
    Do not rely solely on data drift as a proxy for model degradation. Drift detection tells you the input distribution has changed — it does not guarantee the model's predictions are wrong. Always pair drift monitoring with model performance monitoring using ground-truth labels when available, even if labels arrive with a delay.
