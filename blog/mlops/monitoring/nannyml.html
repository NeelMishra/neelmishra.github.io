    NannyML: Performance Without Ground Truth

    MLOps SeriesMonitoring & Observability

    Introduction

    One of the hardest problems in production ML is knowing whether your model is still performing well when you don't have ground truth labels. In many real-world scenarios—fraud detection, loan approvals, medical diagnoses—labels arrive days, weeks, or even months after predictions are made. During that gap, traditional accuracy metrics are simply unavailable.

    NannyML is an open-source Python library specifically designed for this challenge. Rather than relying on ground truth labels, it estimates model performance using the model's own prediction probabilities and calibration characteristics. This lets teams detect silent model degradation long before labels arrive.

    Key Insight: NannyML doesn't replace traditional monitoring—it fills the critical gap between prediction time and label availability. Think of it as an early warning system that buys you time to investigate and remediate issues.

    The library provides two core estimation algorithms: Confidence-Based Performance Estimation (CBPE) for classification tasks and Direct Loss Estimation (DLE) for regression tasks. Both operate on the principle that a well-calibrated model's confidence scores carry meaningful information about expected performance.

    NannyML Overview

    NannyML provides a comprehensive suite of post-deployment monitoring tools organized around three pillars: performance estimation, data drift detection, and data quality monitoring. The library is designed to work with both classification and regression models.

    The core philosophy is that model monitoring should not be blocked by the absence of ground truth. Instead, NannyML leverages statistical relationships learned during the reference period—when labels are available—to estimate performance on new, unlabeled production data.

    # Install NannyML # pip install nannyml import nannyml as nml import pandas as pd # Load reference data (where labels are available) reference_df = pd.read_csv("reference_data.csv") # Load analysis data (production data without labels) analysis_df = pd.read_csv("production_data.csv") # Inspect the data structure print(reference_df.columns.tolist()) # ['feature_1', 'feature_2', ..., 'prediction', 'predicted_proba', 'target']

    NannyML expects a reference dataset where both predictions and ground truth are available, plus an analysis dataset from production where only predictions exist. The library learns the relationship between model confidence and actual performance from the reference set, then applies this learned mapping to estimate performance in production.

    Classification Support

    Binary and multiclass classification with CBPE. Supports metrics like ROC AUC, F1, accuracy, precision, and recall estimation without ground truth labels.

    Regression Support

    Regression tasks via DLE. Estimates MAE, MSE, RMSE, and MAPE by learning the relationship between features and prediction errors during the reference period.

    Confidence-Based Performance Estimation (CBPE)

    CBPE is the core algorithm for classification performance estimation. The fundamental insight behind CBPE is that a well-calibrated model's predicted probabilities directly encode information about expected performance. If a model predicts a 90% probability for class A and is well-calibrated, then roughly 90% of such predictions should indeed be class A.

    The algorithm works in three stages. First, it calibrates the model's predicted probabilities using the reference dataset. Second, it constructs a confusion matrix from the calibrated probabilities. Third, it derives performance metrics from this estimated confusion matrix.

    Calibration Matters: CBPE requires well-calibrated probabilities. If your model outputs poorly calibrated scores, NannyML applies isotonic regression or Platt scaling during the reference period to correct them before estimation.
    # CBPE for binary classification import nannyml as nml # Initialize the CBPE estimator estimator = nml.CBPE( y_pred_proba="predicted_probability", y_pred="prediction", y_true="target", timestamp_column_name="timestamp", metrics=["roc_auc", "f1", "accuracy"], chunk_size=5000, problem_type="classification_binary" ) # Fit on reference data where labels exist estimator.fit(reference_df) # Estimate performance on unlabeled production data results = estimator.estimate(analysis_df) # Access the estimated metrics for metric in results.metrics: print(f"{metric.display_name}: {metric.estimated_values()}") # Check if any alerts were raised if metric.alerts(): print(f" ALERT: Performance drop detected!")

    Under the hood, CBPE constructs the confusion matrix elements using expected values derived from calibrated probabilities. For each prediction with calibrated probability p, the expected true positive contribution is p, and the expected false positive contribution is (1 - p). By summing these expectations across all predictions in a chunk, CBPE reconstructs the full confusion matrix without ever seeing a single label.

    The estimation is bounded by confidence intervals computed via bootstrapping during the reference period. When the estimated metric falls outside these bounds, NannyML raises an alert indicating potential performance degradation.

    Direct Loss Estimation (DLE)

    For regression problems, CBPE doesn't apply because there are no class probabilities to work with. Instead, NannyML uses Direct Loss Estimation (DLE), which takes a fundamentally different approach. DLE trains a secondary model—a "nanny model"—to predict the error of the primary model based on the input features.

    The intuition is straightforward: if the primary model tends to make larger errors on certain feature combinations, the nanny model can learn this pattern from the reference data and then estimate the expected error on new production data without seeing the actual target values.

    # DLE for regression performance estimation import nannyml as nml # Initialize the DLE estimator dle_estimator = nml.DLE( feature_column_names=["feature_1", "feature_2", "feature_3"], y_pred="prediction", y_true="target", timestamp_column_name="timestamp", metrics=["mae", "rmse", "mape"], chunk_size=5000 ) # Fit on the reference dataset dle_estimator.fit(reference_df) # Estimate regression performance without labels dle_results = dle_estimator.estimate(analysis_df) # Plot the estimated MAE over time figure = dle_results.filter( period="analysis", metrics=["mae"] ).plot() figure.show()
    Limitation: DLE assumes that the relationship between features and prediction errors remains stable. If the underlying data distribution shifts dramatically, the nanny model's error estimates may become unreliable. Always combine DLE with univariate drift detection for robust monitoring.

    DLE internally uses a LightGBM model by default for the error estimation, though this can be configured. The nanny model is trained on the absolute or squared residuals from the reference period, learning to predict how much error the primary model will produce for a given input vector.

    Implementation Pipeline

    Building a complete NannyML monitoring pipeline involves several stages: data preparation, estimator configuration, scheduled estimation runs, and alert integration. Here's a production-ready implementation pattern.

    import nannyml as nml import pandas as pd from datetime import datetime, timedelta class NannyMLMonitor: def __init__(self, reference_path, config): self.reference_df = pd.read_csv(reference_path) self.config = config self.estimators = {} self._initialize_estimators() def _initialize_estimators(self): # Set up CBPE for classification metrics self.estimators["cbpe"] = nml.CBPE( y_pred_proba=self.config["proba_col"], y_pred=self.config["pred_col"], y_true=self.config["target_col"], timestamp_column_name=self.config["ts_col"], metrics=self.config["metrics"], chunk_size=self.config.get("chunk_size", 5000), problem_type="classification_binary" ) self.estimators["cbpe"].fit(self.reference_df) # Set up univariate drift detector self.estimators["drift"] = nml.UnivariateDriftCalculator( column_names=self.config["feature_cols"], timestamp_column_name=self.config["ts_col"], chunk_size=self.config.get("chunk_size", 5000) ) self.estimators["drift"].fit(self.reference_df) def run_estimation(self, production_df): # Run performance estimation perf_results = self.estimators["cbpe"].estimate(production_df) drift_results = self.estimators["drift"].calculate(production_df) alerts = [] for metric in perf_results.metrics: if any(metric.alerts()): alerts.append({ "type": "performance", "metric": metric.display_name, "values": metric.estimated_values() }) return {"performance": perf_results, "drift": drift_results, "alerts": alerts}

    The monitor class encapsulates both performance estimation and drift detection. In production, this would be triggered on a schedule—hourly or daily—processing new batches of prediction data and emitting alerts when estimated performance drops below configured thresholds.

    Univariate Drift Detection

    Beyond performance estimation, NannyML provides robust univariate drift detection that complements CBPE and DLE. The library supports multiple statistical tests for detecting distribution shifts in individual features.

    For continuous features, NannyML uses the Kolmogorov-Smirnov test and the Wasserstein distance. For categorical features, it applies the chi-squared test and Jensen-Shannon divergence. Each method captures different aspects of distributional change.

    # Univariate drift detection with NannyML import nannyml as nml # Configure drift calculator with multiple methods drift_calc = nml.UnivariateDriftCalculator( column_names=["age", "income", "credit_score", "category"], timestamp_column_name="timestamp", chunk_size=5000, continuous_methods=["kolmogorov_smirnov", "wasserstein"], categorical_methods=["chi2", "jensen_shannon"] ) # Fit on reference data drift_calc.fit(reference_df) # Calculate drift on production data drift_results = drift_calc.calculate(analysis_df) # Check for drift alerts per feature for col in drift_results.column_names: for method in drift_results.methods_for_column(col): if any(method.alerts()): print(f"Drift detected in {col} via {method.display_name}")

    Combining drift detection with performance estimation creates a powerful monitoring story. If performance estimation shows degradation and drift detection identifies which features shifted, you have an immediate starting point for root cause analysis.

    Comparison with Alternatives

    NannyML occupies a unique niche in the ML monitoring ecosystem. While tools like Evidently and whylogs excel at drift detection and data quality monitoring, they require ground truth for performance metrics. NannyML's performance estimation capability is its primary differentiator.

    NannyML Strengths

    • Performance estimation without labels
    • Statistical confidence intervals
    • Built-in calibration correction
    • Both classification and regression

    NannyML Limitations

    • Requires well-calibrated probabilities
    • Estimation degrades under extreme drift
    • No real-time streaming support
    • Smaller community than alternatives
    Recommendation: Use NannyML as part of a layered monitoring strategy. Pair it with Evidently for rich drift reports and whylogs for lightweight profiling. NannyML fills the label-gap blind spot that other tools cannot address.

    When labels eventually arrive, always validate NannyML's estimates against realized performance. This feedback loop helps you understand the estimation accuracy for your specific use case and tune chunk sizes or calibration methods accordingly. Over time, you'll develop confidence in the estimates and know exactly when to trust them versus when to wait for ground truth.
