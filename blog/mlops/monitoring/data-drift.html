    Data Drift Detection: Statistical Methods

    MLOps SeriesMonitoring & Observability

    Introduction

    Data drift — also called covariate shift — occurs when the statistical distribution of input features changes between training time and inference time. It is one of the most common causes of silent model degradation in production. Unlike software bugs, data drift produces no exceptions or error codes. The model continues to make predictions; they simply become less reliable.

    Detecting data drift requires statistical methods that can quantify how much a distribution has shifted. In this post, we cover three widely-used techniques — Population Stability Index (PSI), the Kolmogorov-Smirnov test, and Jensen-Shannon divergence — with complete Python implementations you can adapt for your own monitoring pipelines.

    What Is Data Drift?

    In formal terms, data drift means that Ptrain(X) ≠ Pproduction(X), where X represents the feature vector. The joint distribution of input features in production has diverged from the distribution observed during training.

    Data drift can be caused by many factors:

    • Seasonal effects — e-commerce features shift during holidays, weather data changes with seasons
    • Population changes — a product expands to a new market with different demographics
    • Upstream pipeline changes — a feature engineering step is modified, a data source changes its schema
    • Sensor degradation — IoT devices lose calibration over time, producing subtly different readings
    • Adversarial adaptation — fraud patterns evolve to bypass detection models
    Important distinction: Data drift refers to changes in the input distribution P(X). This is different from concept drift, which refers to changes in the conditional distribution P(Y|X) — the relationship between inputs and the target.

    Data drift is especially valuable to monitor because it can be detected without ground truth labels. You only need to compare the current input distribution to the training distribution, making it the fastest signal available for detecting potential model degradation.

    Population Stability Index (PSI)

    The Population Stability Index was originally developed in the credit risk industry to monitor the stability of scorecard models. It quantifies how much a distribution has shifted by comparing the proportion of observations in each bin between a reference (training) distribution and the current (production) distribution.

    The PSI formula for a single feature is:

    PSI = Σ (Pi - Qi) × ln(Pi / Qi)

    Where Pi is the proportion of observations in bin i of the production distribution, and Qi is the proportion in the training distribution.

    # Population Stability Index calculation import numpy as np def compute_psi(reference, current, n_bins=10, eps=1e-4): """ Compute Population Stability Index between two distributions. Parameters: reference: array-like, training distribution current: array-like, production distribution n_bins: number of bins for discretisation eps: small constant to avoid division by zero Returns: psi_value: float, the PSI score """ # Create bins from the reference distribution breakpoints = np.linspace( min(np.min(reference), np.min(current)), max(np.max(reference), np.max(current)), n_bins + 1 ) # Compute proportions in each bin ref_counts = np.histogram(reference, bins=breakpoints)[0] cur_counts = np.histogram(current, bins=breakpoints)[0] ref_proportions = ref_counts / len(reference) + eps cur_proportions = cur_counts / len(current) + eps # PSI formula: sum of (P_i - Q_i) * ln(P_i / Q_i) psi_value = np.sum( (cur_proportions - ref_proportions) * np.log(cur_proportions / ref_proportions) ) return psi_value # Interpretation thresholds def interpret_psi(psi): if psi < 0.1: return "No significant drift" elif psi < 0.2: return "Moderate drift — investigate" else: return "Significant drift — action required"
    PSI thresholds are guidelines, not rules. The commonly used thresholds (0.1 for moderate, 0.2 for significant) were developed for credit scoring. Your domain may require tighter or looser thresholds depending on the sensitivity of your model to input distribution changes.

    Strengths of PSI: Easy to interpret, works well for continuous features, the binning approach is robust to outliers, and it gives a single scalar value per feature that is easy to track over time.

    Weaknesses of PSI: Sensitive to the number of bins chosen, does not work directly with high-cardinality categorical features, and the threshold interpretation is somewhat arbitrary outside the credit risk domain.

    Kolmogorov-Smirnov Test

    The Kolmogorov-Smirnov (KS) test is a non-parametric test that quantifies the maximum distance between two cumulative distribution functions (CDFs). Unlike PSI, it does not require binning and provides both a test statistic and a p-value, giving you a built-in significance test.

    The KS statistic is defined as:

    D = max|Fref(x) - Fcur(x)|

    Where Fref and Fcur are the empirical CDFs of the reference and current distributions.

    # Kolmogorov-Smirnov test for drift detection from scipy import stats import numpy as np def ks_drift_test(reference, current, alpha=0.05): """ Perform two-sample KS test for data drift. Parameters: reference: array-like, training distribution current: array-like, production distribution alpha: significance level for the test Returns: dict with statistic, p_value, and drift_detected """ ks_stat, p_value = stats.ks_2samp(reference, current) return { "statistic": ks_stat, "p_value": p_value, "drift_detected": p_value < alpha, "severity": classify_ks(ks_stat) } def classify_ks(ks_stat): """Classify the severity of drift based on KS statistic.""" if ks_stat < 0.1: return "negligible" elif ks_stat < 0.2: return "small" elif ks_stat < 0.35: return "moderate" else: return "large" # Example usage across all features def batch_ks_test(ref_df, cur_df, features, alpha=0.05): """Run KS tests across all features and return results.""" results = {} for feat in features: results[feat] = ks_drift_test( ref_df[feat].values, cur_df[feat].values, alpha=alpha ) return results

    Strengths of KS test: Non-parametric (no distribution assumptions), no binning required, provides a formal statistical test with p-value, and is well understood in the statistics community.

    Weaknesses of KS test: With large sample sizes (common in production), the test becomes overly sensitive — detecting statistically significant but practically irrelevant shifts. It also only captures the maximum divergence point, potentially missing distributed shifts across the distribution.

    Jensen-Shannon Divergence

    Jensen-Shannon (JS) divergence is an information-theoretic measure that quantifies the similarity between two probability distributions. It is derived from the Kullback-Leibler divergence but is symmetric and always finite, making it more suitable for practical monitoring applications.

    The JS divergence is defined as:

    JSD(P || Q) = ½ KL(P || M) + ½ KL(Q || M), where M = ½(P + Q)

    The JS divergence is bounded between 0 (identical distributions) and 1 (when using log base 2), making it naturally normalised and easy to interpret as a percentage of maximum divergence.

    # Jensen-Shannon divergence for drift detection import numpy as np from scipy.spatial.distance import jensenshannon from scipy.stats import entropy def compute_js_divergence(reference, current, n_bins=50): """ Compute Jensen-Shannon divergence between two distributions. Parameters: reference: array-like, training distribution current: array-like, production distribution n_bins: number of bins for histogram approximation Returns: js_divergence: float between 0 and 1 """ # Create shared bin edges from combined data combined = np.concatenate([reference, current]) bin_edges = np.histogram_bin_edges(combined, bins=n_bins) # Compute normalised histograms ref_hist, _ = np.histogram(reference, bins=bin_edges, density=True) cur_hist, _ = np.histogram(current, bins=bin_edges, density=True) # Add small epsilon to avoid zero probabilities ref_hist = ref_hist + 1e-10 cur_hist = cur_hist + 1e-10 # Normalise to probability distributions ref_prob = ref_hist / ref_hist.sum() cur_prob = cur_hist / cur_hist.sum() # JS divergence (squared distance from scipy) js_div = jensenshannon(ref_prob, cur_prob) ** 2 return js_div def js_manual(p, q): """Manual JS divergence using KL divergence components.""" m = 0.5 * (p + q) return 0.5 * entropy(p, m) + 0.5 * entropy(q, m)

    Strengths of JS divergence: Symmetric (unlike KL divergence), bounded between 0 and 1, information-theoretically grounded, and works well for both continuous and discrete distributions after binning.

    Weaknesses of JS divergence: Requires binning for continuous distributions, sensitive to bin width choice, and the thresholds for "significant drift" must be calibrated per feature and domain.

    Putting It All Together: A Drift Monitoring Pipeline

    In practice, you rarely use a single drift detection method in isolation. A robust monitoring pipeline combines multiple methods and aggregates their signals to reduce false positives while maintaining sensitivity to real drift.

    # Complete drift monitoring pipeline import numpy as np import pandas as pd from dataclasses import dataclass from typing import Dict, List @dataclass class DriftReport: feature: str psi: float ks_stat: float ks_pvalue: float js_divergence: float overall_status: str class DriftMonitor: def __init__(self, reference_data, features, thresholds=None): self.reference = reference_data self.features = features self.thresholds = thresholds or { "psi": 0.15, "ks_pvalue": 0.01, "js": 0.1 } def analyse(self, current_data) -> List[DriftReport]: """Run all drift tests and produce a unified report.""" reports = [] for feat in self.features: ref = self.reference[feat].values cur = current_data[feat].values psi = compute_psi(ref, cur) ks = ks_drift_test(ref, cur) js = compute_js_divergence(ref, cur) # Determine overall status by majority vote signals = [ psi > self.thresholds["psi"], ks["p_value"] < self.thresholds["ks_pvalue"], js > self.thresholds["js"] ] drift_count = sum(signals) if drift_count >= 2: status = "DRIFT_DETECTED" elif drift_count == 1: status = "WARNING" else: status = "OK" reports.append(DriftReport( feature=feat, psi=psi, ks_stat=ks["statistic"], ks_pvalue=ks["p_value"], js_divergence=js, overall_status=status )) return reports
    Majority voting reduces false positives. A single method might flag drift due to sensitivity or bin-width artefacts. Requiring two of three methods to agree gives you higher confidence that the drift is real and actionable.

    Choosing the Right Method

    Each drift detection method has distinct trade-offs. The best choice depends on your data characteristics, monitoring requirements, and operational constraints.

    PSI

    • Best for: Continuous features with moderate cardinality
    • Interpretability: High — single scalar with established thresholds
    • Limitations: Bin-dependent, not ideal for categorical features
    • Use when: You need a quick, interpretable drift score per feature

    KS Test

    • Best for: Continuous features of any shape
    • Interpretability: High — p-value has standard statistical meaning
    • Limitations: Over-sensitive with large samples
    • Use when: You want a formal hypothesis test for drift

    JS Divergence

    • Best for: Both continuous and discrete distributions
    • Interpretability: Moderate — bounded 0–1 but thresholds are domain-specific
    • Limitations: Requires binning for continuous data
    • Use when: You need a symmetric, bounded divergence measure

    Ensemble (Recommended)

    • Best for: Production monitoring with low false-positive tolerance
    • Interpretability: High — majority vote is easy to explain
    • Limitations: Slightly more compute per feature
    • Use when: Drift alerts trigger costly actions like retraining

    Regardless of which method you choose, the key operational considerations remain the same: establish clear baselines from your training data, set thresholds that balance sensitivity with actionability, and always monitor trends rather than reacting to individual point-in-time values.
