    Feature Drift Monitoring and Alerting

    MLOps SeriesMonitoring & Observability

    Introduction

    Feature-level drift monitoring is the operational backbone of any ML observability system. While aggregate drift scores tell you something has changed, per-feature monitoring tells you what changed and helps you diagnose why. When a model starts degrading in production, the first question is always: "Which feature drifted?" The second question is: "Does that feature actually matter for predictions?"

    This post covers the complete feature drift monitoring pipeline — from tracking individual feature distributions, to setting meaningful thresholds, to routing alerts to the right teams, to weighting drift signals by feature importance so you focus on what matters most.

    A well-designed feature monitoring system transforms ML debugging from "the model is broken, good luck" into "feature X drifted by Y%, which has Z impact on model performance, and the upstream owner is team A."

    Per-Feature Monitoring

    Monitoring features individually — rather than only tracking aggregate drift — gives you the granularity needed for root-cause analysis. Each feature has its own distribution baseline, drift characteristics, and acceptable range of variation.

    For each feature in your model, you should track:

    • Statistical summary — mean, median, standard deviation, min, max, and key quantiles (p5, p25, p75, p95)
    • Distribution shape — histogram or KDE comparison against the training baseline
    • Drift score — PSI, KS statistic, or JS divergence computed against the reference distribution
    • Missing rate — the percentage of null or missing values, compared to the training-time missing rate
    • Cardinality — for categorical features, track the number of unique values and detect new or disappeared categories
    # Per-feature monitoring with comprehensive statistics import numpy as np import pandas as pd from dataclasses import dataclass, field from typing import Dict, List, Optional from scipy import stats @dataclass class FeatureProfile: """Statistical profile of a single feature.""" name: str mean: float std: float median: float min_val: float max_val: float quantiles: Dict[str, float] missing_rate: float unique_count: int class FeatureMonitor: """Track per-feature statistics and detect drift.""" def __init__(self, reference_df, features): self.features = features self.baselines = {} for feat in features: self.baselines[feat] = self._profile( reference_df[feat], feat ) def _profile(self, series, name): """Compute statistical profile for a feature.""" clean = series.dropna() return FeatureProfile( name=name, mean=float(clean.mean()), std=float(clean.std()), median=float(clean.median()), min_val=float(clean.min()), max_val=float(clean.max()), quantiles={ "p5": float(clean.quantile(0.05)), "p25": float(clean.quantile(0.25)), "p75": float(clean.quantile(0.75)), "p95": float(clean.quantile(0.95)), }, missing_rate=float(series.isna().mean()), unique_count=int(series.nunique()) ) def compute_drift(self, current_df): """Compute drift metrics for all monitored features.""" results = {} for feat in self.features: baseline = self.baselines[feat] current = self._profile(current_df[feat], feat) ref_vals = current_df[feat].dropna().values # KS test for distribution comparison ks_stat, ks_p = stats.ks_2samp( ref_vals, current_df[feat].dropna().values ) results[feat] = { "baseline": baseline, "current": current, "ks_statistic": ks_stat, "ks_pvalue": ks_p, "mean_shift": abs( current.mean - baseline.mean ) / (baseline.std + 1e-10), "missing_rate_change": ( current.missing_rate - baseline.missing_rate ) } return results
    Practical tip: Store feature profiles as versioned artefacts alongside your model. When you retrain, generate a new baseline profile. When investigating drift, compare the current profile against the exact baseline the deployed model was trained on — not just the most recent training run.

    Threshold Setting

    Setting drift thresholds is more art than science. Thresholds that are too tight produce alert fatigue; thresholds that are too loose miss real drift until it is too late. The key insight is that thresholds should be feature-specific and empirically calibrated.

    A principled approach to threshold setting involves three steps:

    1. Measure natural variation. Compute drift scores across historical time windows where you know the model was performing well. This gives you the feature's natural variability baseline.
    2. Set thresholds at a multiple of natural variation. For example, alert when drift exceeds 3× the standard deviation of historical drift scores for that feature.
    3. Calibrate against performance impact. Retrospectively analyse periods of known degradation and determine which drift levels corresponded to meaningful performance drops.
    # Adaptive threshold calculation from historical drift import numpy as np from typing import Dict, Tuple class AdaptiveThresholds: """Compute per-feature drift thresholds from historical data.""" def __init__(self, sensitivity=3.0, min_samples=30): self.sensitivity = sensitivity # std deviations self.min_samples = min_samples self.history: Dict[str, list] = {} def record(self, feature, drift_score): """Record a historical drift observation.""" if feature not in self.history: self.history[feature] = [] self.history[feature].append(drift_score) def get_threshold(self, feature) -> Tuple[float, bool]: """ Compute the adaptive threshold for a feature. Returns (threshold, is_calibrated). """ if feature not in self.history: return 0.2, False # default fallback scores = np.array(self.history[feature]) if len(scores) < self.min_samples: return 0.2, False # not enough history mean_drift = np.mean(scores) std_drift = np.std(scores) threshold = mean_drift + self.sensitivity * std_drift return max(threshold, 0.05), True def get_all_thresholds(self) -> Dict: """Return thresholds for all tracked features.""" return { feat: self.get_threshold(feat) for feat in self.history }

    Static Thresholds

    • Same threshold for every feature (e.g., PSI > 0.2)
    • Simple to implement and explain
    • Ignores feature-specific variability
    • High false-positive rate for volatile features
    • May miss drift in naturally stable features

    Adaptive Thresholds

    • Per-feature thresholds based on historical variation
    • Requires historical drift data to calibrate
    • Automatically adjusts for feature volatility
    • Lower false-positive rate
    • Better sensitivity for stable features
    Avoid alert fatigue. When you first deploy drift monitoring, start with relaxed thresholds and tighten them gradually as you learn which alerts are actionable. A monitoring system that produces 50 false alerts per day will be ignored, defeating its purpose entirely.

    Alert Routing

    Not all drift alerts are equal, and not all alerts should go to the same team. Effective alert routing maps drift signals to the right responders based on the likely root cause.

    A practical alert routing strategy considers three dimensions:

    • Severity — Is this a warning (single feature, moderate drift) or critical (multiple features, severe drift)?
    • Source — Does the drifted feature come from an upstream pipeline, a third-party API, or user input?
    • Impact — Is the drifted feature among the model's top-importance features, or is it a minor contributor?
    # Alert routing based on drift severity and source import logging from typing import Dict, List from enum import Enum class Severity(Enum): INFO = "info" WARNING = "warning" CRITICAL = "critical" class AlertRouter: """Route drift alerts to appropriate teams and channels.""" def __init__(self, routing_config): self.config = routing_config self.logger = logging.getLogger("alert_router") def classify_severity(self, drift_results, importance_map): """Determine alert severity from drift results.""" drifted = [ f for f, r in drift_results.items() if r["overall_status"] == "DRIFT_DETECTED" ] if not drifted: return Severity.INFO # Check if any high-importance features drifted high_importance = [ f for f in drifted if importance_map.get(f, 0) > 0.1 ] if len(high_importance) >= 2 or len(drifted) >= 5: return Severity.CRITICAL elif len(high_importance) >= 1 or len(drifted) >= 3: return Severity.WARNING return Severity.INFO def route(self, feature, severity, drift_info): """Send alert to the appropriate channel.""" owner = self.config["feature_owners"].get( feature, "ml-platform" ) channel = self.config["severity_channels"][severity.value] alert_payload = { "feature": feature, "severity": severity.value, "owner": owner, "channel": channel, "drift_score": drift_info["drift_score"], "threshold": drift_info["threshold"], "recommendation": self._recommend(severity) } self.logger.info( "Routing alert for %s to %s via %s", feature, owner, channel ) return alert_payload def _recommend(self, severity): """Generate a recommended action based on severity.""" recommendations = { Severity.INFO: "Monitor — no action required", Severity.WARNING: "Investigate upstream data sources", Severity.CRITICAL: "Consider model rollback or retraining" } return recommendations[severity]

    Feature Importance Weighting

    Not all features contribute equally to a model's predictions. A 20% drift in a feature with 0.01 importance is noise; a 5% drift in a feature with 0.3 importance could be catastrophic. Weighting drift signals by feature importance dramatically reduces false positives and focuses attention on the features that actually matter.

    # Importance-weighted drift scoring import numpy as np from typing import Dict class ImportanceWeightedDrift: """Weight drift scores by feature importance for prioritisation.""" def __init__(self, importance_map: Dict[str, float]): # Normalise importance scores to sum to 1 total = sum(importance_map.values()) + 1e-10 self.importance = { k: v / total for k, v in importance_map.items() } def weighted_score(self, drift_scores: Dict[str, float]) -> float: """Compute a single weighted drift score across all features.""" weighted = 0.0 for feature, score in drift_scores.items(): weight = self.importance.get(feature, 0.0) weighted += score * weight return weighted def prioritise( self, drift_scores: Dict[str, float] ) -> list: """Rank features by importance-weighted drift impact.""" impacts = [] for feature, score in drift_scores.items(): weight = self.importance.get(feature, 0.0) impacts.append({ "feature": feature, "drift_score": score, "importance": weight, "impact": score * weight }) impacts.sort(key=lambda x: x["impact"], reverse=True) return impacts def should_alert( self, drift_scores, threshold=0.05 ) -> bool: """Alert only when weighted drift exceeds threshold.""" return self.weighted_score(drift_scores) > threshold

    Feature importance can be derived from several sources:

    • Model-intrinsic importance — tree-based models provide feature importance scores directly (Gini importance, permutation importance)
    • SHAP values — compute the mean absolute SHAP value per feature across a reference dataset for model-agnostic importance
    • Permutation importance — measure the performance drop when each feature is randomly shuffled, giving an empirical measure of dependency
    • Domain knowledge — for some features, business context tells you they are critical regardless of what the model says
    Refresh importance scores when you retrain. Feature importance can change between model versions. A feature that was unimportant in v1 might become critical in v2 if the model architecture or training data changed. Always recompute importance when updating your drift thresholds.

    Implementation: Complete Feature Monitoring Pipeline

    Here is a complete pipeline that brings together per-feature monitoring, adaptive thresholds, importance weighting, and alert routing into a production-ready system.

    # Complete feature drift monitoring pipeline import logging from datetime import datetime from typing import Dict, List class FeatureDriftPipeline: """End-to-end feature drift monitoring and alerting.""" def __init__( self, feature_monitor, thresholds, importance_scorer, alert_router, config ): self.monitor = feature_monitor self.thresholds = thresholds self.scorer = importance_scorer self.router = alert_router self.config = config self.logger = logging.getLogger("feature_pipeline") def run(self, current_data) -> Dict: """Execute the full monitoring pipeline.""" timestamp = datetime.utcnow().isoformat() self.logger.info("Running drift check at %s", timestamp) # Step 1: Compute per-feature drift drift_results = self.monitor.compute_drift(current_data) # Step 2: Apply adaptive thresholds flagged = {} for feat, result in drift_results.items(): threshold, calibrated = self.thresholds.get_threshold(feat) ks = result["ks_statistic"] if ks > threshold: flagged[feat] = { "drift_score": ks, "threshold": threshold, "calibrated": calibrated } # Record for future threshold updates self.thresholds.record(feat, ks) # Step 3: Prioritise by importance drift_scores = {f: r["ks_statistic"] for f, r in drift_results.items()} prioritised = self.scorer.prioritise(drift_scores) weighted_total = self.scorer.weighted_score(drift_scores) # Step 4: Route alerts for flagged features alerts = [] severity = self.router.classify_severity( drift_results, self.scorer.importance ) for feat, info in flagged.items(): alert = self.router.route(feat, severity, info) alerts.append(alert) return { "timestamp": timestamp, "features_checked": len(drift_results), "features_flagged": len(flagged), "weighted_drift": weighted_total, "severity": severity.value, "top_features": prioritised[:5], "alerts_sent": len(alerts) }

    Best Practices

    After deploying feature drift monitoring across multiple production systems, here are the patterns that consistently deliver the best results:

    1. Start with your top 10 features by importance. You do not need to monitor every feature from day one. Focus on the features that matter most to your model and expand coverage over time. The top 10 features by importance typically account for 80%+ of model behaviour.
    2. Use multiple drift methods and vote. No single statistical test is perfect. Run PSI, KS, and JS divergence on each feature and require at least two of three to flag drift before alerting. This dramatically reduces false positives.
    3. Separate data quality from data drift. Missing values, type errors, and schema violations are data quality issues — not drift. Monitor these separately with strict, deterministic checks. Drift monitoring is for statistical shifts in valid data.
    4. Version your baselines alongside your models. When you retrain a model, generate and store a new feature baseline. When investigating drift, always compare against the baseline of the currently deployed model, not the most recent training run.
    5. Set different monitoring frequencies for different features. High-importance or high-volatility features may need hourly checks. Low-importance, stable features can be checked daily or even weekly. This saves compute and reduces noise.

    Do

    • Weight drift by feature importance
    • Use adaptive, per-feature thresholds
    • Map features to upstream data owners
    • Log every drift computation for audit
    • Alert on sustained trends, not single spikes
    • Recompute baselines when models are retrained

    Don't

    • Use the same threshold for every feature
    • Alert on every minor fluctuation
    • Ignore low-drift features permanently
    • Treat missing values as drift
    • Forget to update importance weights on retrain
    • Monitor aggregate metrics without per-feature detail
    The goal of feature monitoring is actionability. Every alert should tell the responder exactly which feature drifted, by how much, whether it matters, and who owns the upstream data source. If an alert does not provide this context, it will be ignored — and rightfully so.
