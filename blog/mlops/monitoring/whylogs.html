    whylogs: Data Logging and Profiling

    MLOps SeriesMonitoring & Observability

    Introduction

    Machine learning models are only as reliable as the data they consume. In production systems processing millions of predictions daily, data quality issues — silent schema changes, feature drift, missing values, or upstream pipeline failures — can degrade model performance long before anyone notices. Traditional monitoring that stores raw data quickly becomes expensive and impractical at scale.

    whylogs is an open-source data logging library created by WhyLabs that solves this problem with a fundamentally different approach: instead of storing or sampling raw data, whylogs computes lightweight statistical profiles that capture the essential shape of your data in a fraction of the storage. These profiles are mergeable, meaning you can combine hourly profiles into daily summaries without losing fidelity, and they never contain personally identifiable information.

    Whether you are monitoring a batch training pipeline, a real-time serving endpoint, or a streaming data ingestion system, whylogs provides a consistent API to profile data at every stage. This post covers the core concepts, practical integration patterns, constraint-based validation, and how whylogs compares to alternative monitoring tools in the MLOps ecosystem.

    Why profiles instead of raw data? A whylogs profile for a million-row DataFrame is typically under 100 KB. Storing the raw data would require hundreds of megabytes. Profiles are also privacy-safe since no individual data points are retained.

    whylogs Core Concepts

    At its heart, whylogs revolves around four key abstractions: profiles, sessions, loggers, and the merge operation. Understanding these concepts is essential for designing effective data monitoring pipelines.

    A profile is a lightweight statistical summary of a dataset. For each column (feature), whylogs tracks approximate distributions using data sketches (based on Apache DataSketches), along with counts for nulls, inferred types, cardinality estimates, and basic statistics like mean, standard deviation, min, and max. Profiles are the fundamental unit of data quality tracking.

    A session manages the lifecycle of logging operations. You create a session, log data through it, and close it to finalize profiles. Sessions can be configured to write profiles to local disk, S3, or the WhyLabs platform.

    A logger is the component that accepts data and builds profiles incrementally. You can log individual dictionaries, pandas DataFrames, or batches of records. The logger maintains a running profile that grows in statistical accuracy without growing in memory.

    The merge operation is what makes whylogs truly powerful for production systems. Two profiles computed over different data partitions can be merged into a single profile that is statistically equivalent to having profiled the combined dataset. This enables map-reduce style profiling across distributed systems and time-based roll-ups from minute-level to hour-level to day-level summaries.

    import whylogs as why from whylogs.core import DatasetProfileView # Create a basic profile from a dictionary data = {"feature_a": [1.0, 2.5, 3.1], "feature_b": ["cat", "dog", "cat"]} result = why.log(data) profile_view = result.view() # Profiles are mergeable across time windows morning_result = why.log(morning_data) afternoon_result = why.log(afternoon_data) morning_profile = morning_result.view() afternoon_profile = afternoon_result.view() # Merge into a single daily profile daily_profile = morning_profile.merge(afternoon_profile) # Serialize for storage — tiny binary format daily_profile.write("daily_profile.bin") # Reload later for comparison or visualization loaded = DatasetProfileView.read("daily_profile.bin")

    Data Profiling in Practice

    Profiling is the foundation of data observability with whylogs. When you call why.log(), whylogs computes a rich set of statistics for every column in your dataset. For numeric columns, this includes mean, standard deviation, min, max, quantile estimates, and a histogram approximation using the KLL sketch algorithm. For categorical columns, whylogs tracks frequent items, cardinality estimates (using HyperLogLog), and type inference counts.

    The profiling operation is designed to be fast and memory-efficient. Profiling a million-row DataFrame typically takes seconds, and the resulting profile is a compact binary blob that can be written to disk, uploaded to a monitoring platform, or merged with other profiles. This makes it feasible to profile every batch, every API request window, and every training run without worrying about storage costs.

    Below is a comprehensive example showing how to profile a pandas DataFrame, inspect individual column statistics, and work with profile summaries. Note how the profile captures distribution information, missing value counts, and inferred data types all in one pass.

    import whylogs as why import pandas as pd import numpy as np # Generate sample prediction data np.random.seed(42) df = pd.DataFrame({ "user_id": range(10000), "prediction_score": np.random.normal(0.75, 0.15, 10000), "category": np.random.choice(["A", "B", "C", None], 10000), "latency_ms": np.random.exponential(50, 10000), "timestamp": pd.date_range("2024-01-01", periods=10000, freq="T") }) # Profile the entire DataFrame in one call result = why.log(df) profile_view = result.view() # Convert to a pandas summary for inspection summary_df = profile_view.to_pandas() print(summary_df.head(20)) # Access column-level statistics col_view = profile_view.get_column("prediction_score") dist = col_view.get_metric("distribution") counts = col_view.get_metric("counts") print(f"Mean: {dist.mean}") print(f"Stddev: {dist.stddev}") print(f"Null count: {counts.null.value}") print(f"Total count: {counts.n.value}")

    For real-time or streaming scenarios, you can log data incrementally. Each call to the logger appends statistics to the running profile without needing to keep raw data in memory. This is particularly useful for model serving endpoints where predictions arrive one at a time.

    import whylogs as why from datetime import datetime, timedelta # Rolling logger for streaming data schema = why.DatasetSchema() logger = why.logger(mode="rolling", interval=timedelta(hours=1), schema=schema) logger.append_writer("local", base_dir="profiles/") # Log individual records as they arrive for record in incoming_stream(): logger.log(record) # The rolling logger automatically rotates profiles every hour # and writes them to the configured destination logger.close()
    Tip: Use rolling loggers in production to automatically rotate profiles at fixed intervals. This gives you time-series data quality tracking without managing profile lifecycle manually.

    Constraints & Validation

    Profiling tells you what your data looks like. Constraints tell you whether what you see is acceptable. The whylogs constraints system lets you define declarative validation rules that are checked against profiles — not raw data. This means validation is as fast and lightweight as profiling itself.

    Constraints are defined using a builder pattern. You specify which column to constrain, what condition must hold (value ranges, no missing values, distribution bounds, type expectations), and optionally a severity level. Multiple constraints are grouped into a ConstraintsSuite that can be evaluated against any profile.

    This approach is particularly powerful as an automated quality gate in CI/CD pipelines. Before deploying a new model or after an ETL job completes, you can profile the output data and validate it against a suite of constraints. If any constraint fails, the pipeline halts and alerts the responsible team.

    from whylogs.core.constraints import ConstraintsBuilder from whylogs.core.constraints.factories import ( greater_than_number, smaller_than_number, no_missing_values, mean_between_range, stddev_between_range, null_percentage_below, n_values_below ) # Build a constraints suite for model prediction data builder = ConstraintsBuilder(dataset_profile_view=profile_view) # Prediction scores must be between 0 and 1 builder.add_constraint(greater_than_number( column_name="prediction_score", number=0.0 )) builder.add_constraint(smaller_than_number( column_name="prediction_score", number=1.0 )) # Mean prediction should stay in expected range builder.add_constraint(mean_between_range( column_name="prediction_score", lower=0.5, upper=0.95 )) # Standard deviation should not spike builder.add_constraint(stddev_between_range( column_name="prediction_score", lower=0.01, upper=0.3 )) # No more than 5% missing values in category builder.add_constraint(null_percentage_below( column_name="category", number=0.05 )) # Latency must always be positive builder.add_constraint(greater_than_number( column_name="latency_ms", number=0.0 )) # Build and run the suite constraints = builder.build() report = constraints.generate_constraints_report() for name, passed in report: status = "PASS" if passed else "FAIL" print(f" [{status}] {name}") # Use as a pipeline gate if not constraints.validate(): raise ValueError("Data quality constraints failed — blocking deployment")
    Warning: Constraints are evaluated against profile statistics, not individual rows. A constraint like greater_than_number(0.0) checks whether the tracked minimum in the profile exceeds zero. Outliers below the threshold will trigger a failure for the entire profile.

    You can also serialize constraints suites and load them in different environments. This allows you to define constraints once during model development and enforce them automatically in production without code changes.

    import json # Serialize constraints to JSON for deployment constraints_json = constraints.to_json() with open("constraints.json", "w") as f: json.dump(constraints_json, f) # In production, load and validate with open("constraints.json", "r") as f: loaded_constraints = json.load(f) # Apply to a new profile from today's data today_result = why.log(todays_dataframe) today_view = today_result.view() builder = ConstraintsBuilder(dataset_profile_view=today_view) # ... re-add constraints from loaded config ... if not builder.build().validate(): send_alert("Data quality check failed for today's batch")

    Model Serving Integration

    One of whylogs' strongest use cases is monitoring data quality at the model serving layer. Every prediction request carries input features and produces output scores — both of which should be continuously profiled. By embedding whylogs into your serving middleware, you get real-time visibility into feature distributions, prediction drift, and data anomalies without adding significant latency.

    The pattern is straightforward: intercept incoming requests, log the feature dictionary to whylogs, run the model prediction, then log the output. A rolling logger handles profile rotation and persistence. Below is a complete FastAPI integration that demonstrates this approach.

    import whylogs as why from fastapi import FastAPI, Request from datetime import timedelta from contextlib import asynccontextmanager import threading import time # Global rolling logger with hourly rotation logger = None @asynccontextmanager async def lifespan(app): global logger logger = why.logger( mode="rolling", interval=timedelta(minutes=30), schema=why.DatasetSchema() ) logger.append_writer("whylabs") yield logger.close() app = FastAPI(lifespan=lifespan) @app.post("/predict") async def predict(request: Request): payload = await request.json() features = payload["features"] # Log input features to whylogs logger.log(features) # Run model inference prediction = model.predict(features) # Log output alongside inputs for drift detection logger.log({"prediction": prediction, **features}) return {"prediction": prediction} @app.get("/health") async def health(): return {"status": "healthy", "profiler": "active"}

    For Flask-based services, the integration pattern is similar but uses the WSGI middleware approach. The key difference is managing the logger lifecycle through Flask's application context.

    FastAPI / Async Integration

    Use the lifespan context manager to initialize and clean up the rolling logger. Log features inside async route handlers. Best for high-throughput async services with non-blocking IO patterns.

    Pros: Native async support, clean lifecycle management, integrates well with Pydantic validation.

    Flask / Sync Integration

    Initialize the logger in before_first_request and tear down in atexit. Log features inside synchronous route handlers. Best for existing Flask services that need minimal refactoring.

    Pros: Simple to add to legacy services, wide ecosystem of extensions, familiar middleware pattern.

    Sidecar / Agent Pattern

    Run whylogs as a separate process that reads from a shared message queue (Redis, Kafka). The serving endpoint publishes feature dictionaries; the sidecar profiles them asynchronously. Zero latency impact on predictions.

    Pros: Complete decoupling, no serving latency impact, can handle very high throughput.

    Batch Collection Pattern

    Buffer prediction requests in memory and profile them in periodic batch intervals using a background thread. Reduces per-request overhead at the cost of slightly delayed monitoring visibility.

    Pros: Lower per-request cost, works well with existing batch infrastructure, simple implementation.

    Performance note: Logging a single dictionary with whylogs adds approximately 0.1–0.5 ms of latency per request. For latency-critical services processing thousands of requests per second, consider the sidecar or batch collection patterns to move profiling off the critical path.

    Visualization & Profile Comparison

    Understanding data quality trends requires more than raw statistics — you need visual comparisons. whylogs includes a built-in visualization module through the NotebookProfileVisualizer that renders interactive HTML reports comparing two profiles side by side. This is invaluable for detecting drift between training and serving data, or between consecutive time windows.

    The visualizer generates distribution overlay plots, difference summaries, and statistical comparison tables. You can use it in Jupyter notebooks during development or generate standalone HTML reports for dashboards and automated review workflows.

    from whylogs.viz import NotebookProfileVisualizer # Profile training data (reference) and production data (target) ref_result = why.log(training_df) target_result = why.log(production_df) ref_view = ref_result.view() target_view = target_result.view() # Initialize the visualizer with both profiles viz = NotebookProfileVisualizer() viz.set_profiles(target_profile_view=target_view, reference_profile_view=ref_view) # Distribution comparison for a specific feature viz.feature_statistics(feature_name="prediction_score") # Overlay histograms for visual drift detection viz.double_histogram(feature_name="prediction_score") # Summary drift report across all features viz.summary_drift_report() # Export as standalone HTML for dashboards viz.write( rendered_html=viz.summary_drift_report(), preferred_path="drift_report.html" )

    Comparison with Alternatives

    whylogs occupies a specific niche in the data quality and monitoring landscape. Understanding where it excels and where alternatives might be better suited helps you make an informed tooling decision. The primary alternatives are Evidently AI, Great Expectations, and custom monitoring solutions built on top of libraries like pandas and scipy.

    Each tool has a different philosophy. whylogs prioritizes lightweight, mergeable profiles suitable for real-time and high-volume use cases. Evidently AI focuses on pre-built drift detection reports and dashboards. Great Expectations emphasizes declarative data validation with rich test suites. Custom solutions offer maximum flexibility but require significant engineering effort to maintain.

    whylogs

    Approach: Statistical profiling via data sketches. Profiles are mergeable, privacy-safe, and extremely lightweight (~100 KB per profile).

    Best for: High-volume production monitoring, real-time serving, streaming pipelines, and scenarios where storage and privacy constraints are paramount.

    Limitations: Visualization requires additional setup. Constraint system is less expressive than Great Expectations. Smaller community compared to alternatives.

    Evidently AI

    Approach: Pre-built drift detection reports and dashboards. Generates rich HTML reports comparing reference and production datasets.

    Best for: Quick drift analysis during development, generating stakeholder-friendly reports, teams that want opinionated defaults for monitoring.

    Limitations: Operates on raw data (heavier at scale), reports can be slow for large datasets, less suitable for real-time monitoring without additional infrastructure.

    Great Expectations

    Approach: Declarative data validation with a comprehensive library of expectations (tests). Strong documentation ecosystem with built-in data docs.

    Best for: Data pipeline validation, ETL quality gates, teams that need extensive test coverage with clear documentation and audit trails.

    Limitations: Heavier setup and configuration. Runs against raw data, making it expensive at scale. Not designed for real-time serving use cases.

    Custom Solutions

    Approach: Hand-built monitoring using pandas, scipy, and custom dashboards. Maximum flexibility to define exactly what metrics matter.

    Best for: Unique monitoring requirements, teams with strong engineering resources, situations where existing tools don't cover specific domain needs.

    Limitations: High maintenance burden, no built-in merging or drift detection, reinventing the wheel for common patterns, inconsistent across teams.

    Recommendation: Start with whylogs if you need production-grade monitoring at scale with minimal overhead. Use it alongside Evidently for richer ad-hoc analysis during model development. Add Great Expectations for upstream ETL pipeline validation where comprehensive data testing is more important than lightweight profiling.
    Common mistake: Do not try to replace all three tools with one. whylogs excels at continuous lightweight profiling in production; Great Expectations excels at thorough batch validation during ETL; Evidently excels at interactive drift investigation. A mature MLOps stack often uses two or three of these tools in complementary roles.

    When choosing between these tools, consider your primary use case: if you process millions of predictions per day and need sub-millisecond profiling overhead, whylogs is the clear winner. If you need rich, interactive dashboards for periodic model reviews, Evidently's pre-built reports are hard to beat. If your biggest pain point is ETL pipeline reliability and you need hundreds of validation rules, Great Expectations' comprehensive expectation library is the right choice.

    For most production ML systems, the ideal stack combines whylogs for continuous real-time profiling with one of the batch-oriented tools for periodic deep analysis. The profiles generated by whylogs can feed into custom alerting systems, while Evidently or Great Expectations handle the detailed investigation when alerts fire.
