Prompt Routing: Model Selection by Query

MLOps Series Prompt Engineering

Not every user query demands the most powerful (and most expensive) model in your fleet. A simple "What time is it in Tokyo?" doesn't need GPT-4o, while "Derive the Bellman equation from first principles" probably does. Prompt routing is the discipline of automatically dispatching each incoming query to the most cost-effective model that can still produce an acceptable response. Done well, it slashes inference costs by 40-70% with negligible quality loss.

Why Route Prompts

The core insight behind prompt routing is a simple economic observation: model capability and cost are highly correlated, but query difficulty is not uniformly distributed. In most production workloads, the vast majority of queries are routine — simple look-ups, greetings, format conversions — while only a small tail requires deep reasoning.

The 80/20 rule of LLM traffic: In typical production systems, roughly 70-80% of queries can be handled by a model that costs 10-50× less than the frontier model. Routing exploits this distribution to achieve massive cost savings.

Without routing, teams face a binary choice: use the best model for everything (expensive, slow) or use a cheap model for everything (quality drops on hard queries). Routing gives you the best of both worlds — premium quality where it matters, economy everywhere else.

🚫 No Routing

All queries → GPT-4o

100k queries/day × $2.50/1M tokens ≈ $750/day

Simple queries waste 10-30× more budget than necessary. Latency is uniformly high.

✅ With Routing

75% → mini, 25% → GPT-4o

75k × $0.25 + 25k × $2.50 ≈ $81/day

89% cost reduction. Simple queries return 3-5× faster. Hard queries still get full power.

Complexity Classification

The heart of any router is a complexity classifier — a function that estimates how difficult a query is before it reaches any LLM. There are three main approaches, each trading off accuracy against latency and cost of the classifier itself.

Rule-Based Classification

The simplest approach uses heuristics: token count, keyword presence, regex patterns, and structural features. Despite its simplicity, rule-based routing handles the obvious cases surprisingly well.

# Rule-based complexity classifier import re from dataclasses import dataclass from enum import Enum class Complexity(Enum): SIMPLE = "simple" MODERATE = "moderate" COMPLEX = "complex" COMPLEX_KEYWORDS = {"derive", "prove", "analyze", "compare and contrast", "multi-step", "trade-offs", "optimize"} CODE_PATTERNS = [r"write .* function", r"implement .* algorithm", r"debug .* error", r"refactor"] def classify_rule_based(query: str) -> Complexity: tokens = query.split() lower = query.lower() # Heuristic 1: very short queries are usually simple if len(tokens) < 8: return Complexity.SIMPLE # Heuristic 2: complex keyword detection if any(kw in lower for kw in COMPLEX_KEYWORDS): return Complexity.COMPLEX # Heuristic 3: code generation patterns if any(re.search(p, lower) for p in CODE_PATTERNS): return Complexity.MODERATE # Heuristic 4: long queries often carry more context if len(tokens) > 60: return Complexity.MODERATE return Complexity.SIMPLE

Embedding-Based Classification

Train a lightweight classifier (logistic regression, small MLP) on top of query embeddings. The embedding model captures semantic nuance that rules miss — like distinguishing "explain photosynthesis" (textbook recall → simple) from "explain the quantum yield anomalies in C4 photosynthesis" (research-grade → complex).

# Embedding-based complexity classifier import numpy as np from sklearn.linear_model import LogisticRegression import pickle class EmbeddingClassifier: def __init__(self, model_path: str, embed_client): self.embed_client = embed_client with open(model_path, "rb") as f: self.clf: LogisticRegression = pickle.load(f) def classify(self, query: str) -> tuple[str, float]: # Get embedding from a fast, small model emb = self.embed_client.embed(query, model="text-embedding-3-small") vec = np.array(emb).reshape(1, -1) label = self.clf.predict(vec)[0] proba = self.clf.predict_proba(vec).max() return label, proba # ("complex", 0.91)
Training tip: Label 1-2k production queries by which model tier answered them successfully. Use human annotators or a strong LLM-as-judge to create ground truth. Even a simple logistic regression on embeddings reaches ~85% routing accuracy with this approach.

LLM-Based Classification

Use a small, fast LLM (like GPT-4o-mini or Haiku) to judge query complexity before routing to the target model. This is the most accurate approach but adds latency and cost for the classifier call itself.

# LLM-as-classifier for complexity routing ROUTER_PROMPT = """Classify the following user query into exactly one complexity level. Reply with ONLY the label. - SIMPLE: factual lookups, greetings, translations, formatting - MODERATE: summarization, code generation, explanations - COMPLEX: multi-step reasoning, proofs, novel analysis, debates Query: {query} Label:""" async def classify_with_llm(query: str, client) -> str: resp = await client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": ROUTER_PROMPT.format(query=query)}], temperature=0, max_tokens=5, ) return resp.choices[0].message.content.strip().upper()

Model Cascading

Cascading is a complementary strategy to classification: instead of predicting difficulty before calling any model, you try the cheapest model first and only escalate if the response quality is insufficient. The cascade checks a confidence signal after each tier.

# Model cascading with confidence checks from dataclasses import dataclass @dataclass class Tier: model: str conf_threshold: float cost_per_1m: float TIERS = [ Tier("claude-3-haiku", conf_threshold=0.85, cost_per_1m=0.08), Tier("gpt-4o", conf_threshold=0.80, cost_per_1m=2.50), Tier("o1", conf_threshold=0.0, cost_per_1m=15.0), ] async def cascade(query: str, client, judge) -> dict: for tier in TIERS: response = await client.generate(tier.model, query) # Confidence = P(response is adequate) from a judge confidence = await judge.score(query, response) if confidence >= tier.conf_threshold: return { "response": response, "model": tier.model, "confidence": confidence, "cost_tier": tier.cost_per_1m, } # Final tier always returns (threshold = 0.0) return {"response": response, "model": TIERS[-1].model}
Cascading trade-off: Cascade adds latency for queries that escalate — the user waits for Tier 1's response plus the confidence check plus Tier 2's response. For latency-sensitive applications, prefer upfront classification. Cascading shines when you can do the confidence check cheaply (e.g., logprob-based) and most queries resolve at Tier 1.

Router Architectures

Several open-source and commercial systems have emerged that package routing into reusable components. Understanding their design helps you pick the right approach — or build your own.

🔀 RouteLLM

Approach: Trains a classifier on preference data from Chatbot Arena to predict which model will be preferred for a given query.

Pros: Open-source, pluggable, strong empirical results. Reduces cost by ~50% at <3% quality loss on MT-Bench.

Cons: Requires preference training data. Classifier itself adds ~10ms latency.

🪐 Martian Router

Approach: Commercial meta-model that predicts per-model quality scores, then selects the cheapest model meeting an SLA threshold.

Pros: No training needed — works out of the box. Handles 100+ models. Built-in fallback.

Cons: Proprietary. Adds API hop. Less customizable than self-hosted classifiers.

# Semantic routing with embeddings (à la semantic-router) import numpy as np from dataclasses import dataclass, field @dataclass class Route: name: str model: str utterances: list[str] # seed examples centroid: np.ndarray = field(default=None, repr=False) class SemanticRouter: def __init__(self, routes: list[Route], embed_fn): self.routes = routes self.embed_fn = embed_fn self._build_centroids() def _build_centroids(self): for route in self.routes: embeddings = [self.embed_fn(u) for u in route.utterances] route.centroid = np.mean(embeddings, axis=0) def route(self, query: str) -> Route: q_emb = self.embed_fn(query) best, best_sim = None, -1 for r in self.routes: sim = np.dot(q_emb, r.centroid) / ( np.linalg.norm(q_emb) * np.linalg.norm(r.centroid)) if sim > best_sim: best, best_sim = r, sim return best # Usage routes = [ Route("simple", "haiku", ["hi", "what time is it", "translate hello"]), Route("moderate", "gpt-4o", ["summarize this paper", "write a function"]), Route("complex", "o1", ["prove the central limit theorem", "design a distributed consensus algorithm"]), ] router = SemanticRouter(routes, embed_fn=get_embedding)
Semantic routing is composable: You can define routes not just by complexity, but by domain (medical, legal, code), language, or task type (classification, generation, chat). Each route maps to a different model or prompt template. This turns the router into a full request orchestrator.

Cost Optimization Math

The objective of routing is to minimize total inference cost while maintaining a quality floor. This is a constrained optimization problem that can be formalized.

# Cost optimization as a linear program from scipy.optimize import linprog import numpy as np # Models: [haiku, gpt-4o-mini, gpt-4o, o1] costs = np.array([0.08, 0.25, 2.50, 15.0]) # $/1M tokens quality = np.array([0.62, 0.75, 0.91, 0.97]) # benchmark score latency = np.array([0.3, 0.5, 1.2, 4.5]) # seconds p50 Q_MIN = 0.88 # minimum acceptable quality L_MAX = 1.5 # maximum average latency (seconds) # minimize c^T p # subject to: -q^T p ≤ -Q_MIN, l^T p ≤ L_MAX, p_i ∈ [0,1] result = linprog( c=costs, A_ub=np.array([[-q for q in quality], [l for l in latency]]), b_ub=np.array([-Q_MIN, L_MAX]), A_eq=np.ones((1, 4)), b_eq=np.array([1.0]), bounds=[(0, 1)] * 4, method="highs" ) optimal_mix = result.x print("Optimal routing mix:", dict(zip( ["haiku", "mini", "gpt4o", "o1"], [f"{p:.1%}" for p in optimal_mix] ))) print(f"Min cost: ${result.fun:.2f}/1M tokens")

Pareto Efficiency

A routing configuration is Pareto-efficient if you cannot improve quality without increasing cost, and vice versa. The set of all Pareto-efficient configurations forms the Pareto frontier.

SLA-aware routing: In production, you don't optimize for the cheapest configuration globally — you pick the cheapest point on the Pareto frontier that sits above your quality SLA line. Different product tiers (free, pro, enterprise) can target different points on the same frontier.

Building a Production Router

A production-grade router goes beyond a classifier — it needs monitoring, feedback loops, graceful degradation, and the ability to retrain on fresh data. Here's a reference architecture that brings it all together.

# Production router with monitoring and fallback import time, logging from dataclasses import dataclass, field from typing import Optional logger = logging.getLogger("router") @dataclass class RoutingDecision: model: str confidence: float classifier_latency_ms: float reason: str @dataclass class RouterConfig: default_model: str = "gpt-4o-mini" fallback_model: str = "gpt-4o" confidence_floor: float = 0.70 timeout_ms: float = 50 # classifier budget shadow_pct: float = 0.05 # % of traffic for A/B eval class ProductionRouter: def __init__(self, classifier, config: RouterConfig): self.clf = classifier self.cfg = config def route(self, query: str) -> RoutingDecision: t0 = time.perf_counter() try: label, conf = self.clf.classify(query) latency = (time.perf_counter() - t0) * 1000 # If classifier is too slow, use default if latency > self.cfg.timeout_ms: logger.warning(f"Classifier timeout: {latency:.0f}ms") return RoutingDecision( self.cfg.default_model, 0.5, latency, "timeout") # Low-confidence → fallback to stronger model if conf < self.cfg.confidence_floor: return RoutingDecision( self.cfg.fallback_model, conf, latency, "low_conf") model = self._label_to_model(label) return RoutingDecision(model, conf, latency, "classified") except Exception as e: logger.error(f"Router error: {e}") return RoutingDecision( self.cfg.default_model, 0.0, 0, "error_fallback") def _label_to_model(self, label: str) -> str: mapping = { "simple": "claude-3-haiku", "moderate": "gpt-4o-mini", "complex": "gpt-4o", } return mapping.get(label, self.cfg.default_model)
Never let the router become a single point of failure. Always set a sensible default model that the system falls back to if the classifier errors out, times out, or returns a low-confidence prediction. A degraded routing decision is always better than a 500 error.

Monitoring & Retraining

A router that isn't monitored will silently degrade. Query distributions shift, new models get released, and pricing changes. Track these metrics continuously:

📊 Key Metrics

  • Routing distribution: % per model tier
  • Quality by tier: LLM-judge scores segmented by route
  • Escalation rate: how often Tier 1 falls through
  • Cost per query: actual blended cost trend

🔄 Retraining Triggers

  • Quality drift: Tier 1 quality drops below SLA
  • Distribution shift: query types change >15%
  • New model onboarding: a cheaper model becomes available
  • Scheduled: weekly re-calibration on logged data
# Routing telemetry collector from collections import defaultdict import json, time class RouterTelemetry: def __init__(self): self.counts = defaultdict(int) self.quality_scores = defaultdict(list) self.latencies = [] def log(self, decision: RoutingDecision, quality: float): self.counts[decision.model] += 1 self.quality_scores[decision.model].append(quality) self.latencies.append(decision.classifier_latency_ms) def report(self) -> dict: total = sum(self.counts.values()) return { "routing_pct": { m: f"{c/total:.1%}" for m, c in self.counts.items() }, "avg_quality": { m: f"{sum(s)/len(s):.3f}" for m, s in self.quality_scores.items() }, "p50_latency_ms": sorted(self.latencies)[ len(self.latencies) // 2], }
Key takeaway: Prompt routing transforms model selection from a static configuration choice into a dynamic, data-driven optimization. Start simple — even rule-based routing on token count saves money. Graduate to embedding classifiers as you accumulate labeled data, and add cascading for the long tail of hard queries. Monitor everything, retrain regularly, and always have a fallback.
