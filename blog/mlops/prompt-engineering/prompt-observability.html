Prompt Observability: Logging and Analytics

MLOps Series Prompt Engineering

Deploying prompts into production is only half the battle. Without observability, you are flying blind — unable to diagnose regressions, track costs, or prove that a new prompt version actually improves user outcomes. This post covers the full observability stack: structured logging, token economics, quality scoring, controlled experiments, and the platforms that tie it all together.

Why Observe LLM Interactions

Traditional software emits deterministic outputs for given inputs. LLMs do not. The same prompt can produce wildly different responses across runs, models, and parameter settings. Observability gives you the instrumentation to debug failures, control costs, and assure quality at scale.

Key insight: LLM observability is not optional — it is the only way to turn a prompt prototype into a production system you can reason about. Without it, every deployment is a gamble.

The observability stack for LLMs mirrors traditional APM (Application Performance Monitoring) but adds LLM-specific dimensions: prompt versions, token counts, model identifiers, and semantic quality scores that have no equivalent in classical web services.

Structured Logging for LLMs

Unstructured print statements won't scale. Every LLM call should emit a structured log record containing the request, response, metadata, and a trace ID that links it to the broader application request.

# Structured LLM call logger with trace propagation import uuid, time, json, logging logger = logging.getLogger("llm_observability") class LLMCallLogger: def __init__(self, service_name: str): self.service = service_name def log_call(self, prompt, response, model, params, trace_id=None): record = { "trace_id": trace_id or str(uuid.uuid4()), "span_id": str(uuid.uuid4()), "timestamp": time.time(), "service": self.service, "model": model, "prompt_hash": hash_prompt(prompt), # version fingerprint "messages": prompt, # full message array "completion": response.choices[0].message.content, "finish_reason": response.choices[0].finish_reason, "tokens": { "input": response.usage.prompt_tokens, "output": response.usage.completion_tokens, "total": response.usage.total_tokens, }, "latency_ms": None, # filled by wrapper "params": params, # temperature, top_p, etc. } logger.info(json.dumps(record)) return record
Trace IDs matter: A single user action may trigger multiple LLM calls (chain-of-thought → retrieval → summarisation). Without a shared trace ID, correlating these calls during debugging is nearly impossible.

The prompt_hash field is critical for version tracking. By hashing the system prompt template (excluding dynamic variables), you can group logs by prompt version and detect when a template change causes a quality regression.

# Prompt hashing for version tracking import hashlib, re def hash_prompt(messages: list) -> str: # Extract system template, strip variable placeholders system_msgs = [m["content"] for m in messages if m["role"] == "system"] template = " ".join(system_msgs) normalised = re.sub(r"\{\{.*?\}\}", "{{VAR}}", template) return hashlib.sha256(normalised.encode()).hexdigest()[:12]

Token Usage Tracking

Token costs can escalate fast — a single verbose system prompt replicated across thousands of requests compounds into significant spend. Tracking input and output tokens per call lets you attribute costs, identify inefficiencies, and set budget guardrails.

# Token usage tracker with cost attribution and budget alerts from dataclasses import dataclass, field from collections import defaultdict PRICING = { # USD per 1K tokens "gpt-4o": {"input": 0.005, "output": 0.015}, "gpt-4o-mini": {"input": 0.00015, "output": 0.0006}, "claude-3.5": {"input": 0.003, "output": 0.015}, } class TokenTracker: def __init__(self, daily_budget_usd=50.0): self.daily_budget = daily_budget_usd self.usage = defaultdict(lambda: {"input": 0, "output": 0, "cost": 0.0}) def record(self, model, input_tok, output_tok, feature="default"): prices = PRICING.get(model, {"input": 0.01, "output": 0.03}) cost = (input_tok * prices["input"] + output_tok * prices["output"]) / 1000 self.usage[feature]["input"] += input_tok self.usage[feature]["output"] += output_tok self.usage[feature]["cost"] += cost self._check_budget() return cost def _check_budget(self): total = sum(v["cost"] for v in self.usage.values()) if total > self.daily_budget * 0.8: alert(f"⚠️ 80% budget consumed: ${total:.2f}/{self.daily_budget}") if total > self.daily_budget: alert(f"🚨 BUDGET EXCEEDED: ${total:.2f}") raise BudgetExceededError(total)
Hidden cost trap: RAG pipelines often stuff thousands of context tokens into every call. A single poorly-chunked retrieval step can 10× your input token count without improving output quality. Always log input vs. output tokens separately.

Group token spend by feature (search, summarisation, chat) and by customer tier to understand where your budget actually goes. Most teams discover that 80% of cost comes from fewer than 20% of features.

Quality Metrics & Scoring

Token counts tell you how much you spend, but not whether the output is good. Quality metrics bridge this gap. The four pillars of LLM quality are latency, relevance, faithfulness, and user satisfaction.

⏱️ Latency

What: Time-to-first-token (TTFT) and total generation time.

Why: Users abandon after 3–5s. Streaming helps perceived speed, but TTFT is the true bottleneck.

Target: TTFT < 500ms, total < 3s for interactive use.

🎯 Relevance

What: Does the response address the user's actual question?

Why: Off-topic or tangential answers erode trust even when factually correct.

Method: LLM-as-judge scoring or embedding similarity to gold references.

📋 Faithfulness

What: Is the response grounded in the provided context (no hallucinations)?

Why: Critical for RAG systems where the model must not fabricate beyond retrieved docs.

Method: NLI-based entailment checks or claim-level verification.

👍 User Feedback

What: Explicit signals — thumbs up/down, regeneration clicks, copy events.

Why: The ultimate ground truth. Automated metrics are proxies; user actions are real.

Method: Log implicit signals alongside explicit ratings.

# Automated quality scorer using LLM-as-judge from openai import OpenAI client = OpenAI() def score_response(question: str, answer: str, context: str) -> dict: # LLM-as-judge evaluates relevance and faithfulness eval_prompt = f"""Score this LLM response on two dimensions. Question: {question} Context provided: {context} Response: {answer} Rate each 1-5 and explain briefly: 1. Relevance — does the response address the question? 2. Faithfulness — is every claim supported by the context? Return JSON: {{"relevance": n, "faithfulness": n, "reasoning": "..."}}""" result = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": eval_prompt}], temperature=0.0, response_format={"type": "json_object"}, ) return json.loads(result.choices[0].message.content)
LLM-as-judge caveat: The judge model has its own biases — it may prefer verbose responses or penalise unconventional phrasing. Always calibrate judge scores against human annotations on a held-out set before trusting them in production.

A/B Analysis for Prompts

When you have two prompt variants, how do you decide which one is better? Gut feeling doesn't cut it in production. You need a statistically rigorous experiment framework that accounts for the high variance inherent in LLM outputs.

# Prompt A/B experiment framework import random, hashlib from scipy import stats class PromptExperiment: def __init__(self, name, control_prompt, variant_prompt, split=0.5): self.name = name self.prompts = {"control": control_prompt, "variant": variant_prompt} self.split = split self.scores = {"control": [], "variant": []} def assign_group(self, user_id: str) -> str: # Deterministic bucketing — same user always gets same variant h = int(hashlib.md5(f"{self.name}:{user_id}".encode()).hexdigest(), 16) return "variant" if (h % 100) / 100 < self.split else "control" def record_score(self, group: str, score: float): self.scores[group].append(score) def analyse(self) -> dict: ctrl, var = self.scores["control"], self.scores["variant"] if len(ctrl) < 30 or len(var) < 30: return {"status": "insufficient_data"} # Mann-Whitney U — non-parametric, works for ordinal scores stat, p_value = stats.mannwhitneyu(ctrl, var, alternative="two-sided") effect = (sum(var) / len(var)) - (sum(ctrl) / len(ctrl)) return { "control_mean": round(sum(ctrl) / len(ctrl), 3), "variant_mean": round(sum(var) / len(var), 3), "effect_size": round(effect, 3), "p_value": round(p_value, 4), "significant": p_value < 0.05, "winner": "variant" if effect > 0 and p_value < 0.05 else "control", "n_control": len(ctrl), "n_variant": len(var), }
Why Mann-Whitney over t-test? LLM quality scores (1–5 Likert scales) are ordinal, not continuous. Mann-Whitney U makes no assumptions about the distribution shape, making it more robust for the heavy-tailed score distributions typical of LLM evaluations.

For teams that want faster decisions, Bayesian A/B testing offers an alternative: instead of waiting for a fixed sample size, you continuously update a posterior distribution and stop when the probability of one variant being better exceeds a threshold (e.g., 95%).

# Bayesian A/B stopping rule (Beta-Binomial model) import numpy as np def bayesian_ab(successes_a, trials_a, successes_b, trials_b, threshold=0.95): # Beta posterior: Beta(alpha=successes+1, beta=failures+1) samples_a = np.random.beta(successes_a + 1, trials_a - successes_a + 1, size=50000) samples_b = np.random.beta(successes_b + 1, trials_b - successes_b + 1, size=50000) prob_b_wins = (samples_b > samples_a).mean() return { "prob_b_wins": round(prob_b_wins, 4), "can_stop": prob_b_wins > threshold or prob_b_wins < (1 - threshold), "recommend": "B" if prob_b_wins > threshold else "A", }

Observability Platforms

You don't need to build your own observability stack from scratch. Several purpose-built platforms provide logging, tracing, evaluation, and analytics for LLM applications out of the box.

🔗 LangSmith

By: LangChain team

Strengths: Deep LangChain integration, trace visualisation for chains/agents, dataset management, automated evaluation runs.

Best for: Teams already using LangChain or LangGraph.

📊 Weights & Biases Prompts

By: W&B (Weave)

Strengths: Experiment tracking lineage, rich comparison UI, integrates with ML training pipelines, model versioning.

Best for: ML teams who already use W&B for training.

☁️ Helicone

By: Helicone AI

Strengths: One-line proxy integration (change base URL), real-time cost dashboard, rate limiting, caching layer built in.

Best for: Fast setup without code changes — ideal for startups.

🔥 Phoenix (Arize)

By: Arize AI

Strengths: Open-source, OpenTelemetry-native tracing, embedding drift detection, local-first with optional cloud.

Best for: Teams wanting vendor-neutral, open-source observability.

# Example: Helicone integration — one-line proxy swap from openai import OpenAI # Before: direct OpenAI call # client = OpenAI() # After: route through Helicone proxy for automatic logging client = OpenAI( base_url="https://oai.helicone.ai/v1", default_headers={ "Helicone-Auth": f"Bearer {HELICONE_API_KEY}", "Helicone-Property-Feature": "search", # cost attribution tag "Helicone-Property-Env": "production", } ) # All calls now auto-logged with latency, tokens, cost, and custom tags response = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": "Summarise this document..."}], )
# Example: Phoenix (Arize) with OpenTelemetry tracing from phoenix.otel import register from openinference.instrumentation.openai import OpenAIInstrumentor # Configure OpenTelemetry exporter to Phoenix tracer_provider = register( project_name="my-llm-app", endpoint="http://localhost:6006/v1/traces", # local Phoenix server ) # Auto-instrument all OpenAI calls OpenAIInstrumentor().instrument(tracer_provider=tracer_provider) # Every client.chat.completions.create() now emits OTEL spans # with prompt, completion, tokens, latency, model, and params
Vendor lock-in tip: Prefer platforms that export to OpenTelemetry format. OTEL is the industry standard for distributed tracing, and adopting it means you can switch backends (Phoenix → Jaeger → Datadog) without re-instrumenting your application code.

Start at Level 1 — even basic logging gives you debugging superpowers compared to nothing. Progress through the maturity levels as your LLM usage grows. Most production teams should target Level 3 within six months of their first deployment: structured logging, cost attribution, quality scoring, and controlled experiments are the foundation of reliable LLM operations.
