Prompt Compression: LLMLingua, Context Distillation

MLOps Series Prompt Engineering

Modern LLM applications frequently exceed context-window limits — RAG pipelines retrieve dozens of passages, agents accumulate long tool-call histories, and multi-turn conversations grow unbounded. Prompt compression is the family of techniques that reduce token count while preserving the information the model actually needs. Done well, it cuts cost and latency without harming answer quality. Done poorly, it strips away the very tokens the model relies on.

Why Compress Prompts

Before diving into specific algorithms, it is worth understanding the four forces that make prompt compression a practical necessity in production systems.

The cost equation: At $10 per million input tokens (GPT-4-class pricing), a RAG pipeline serving 1 M requests/day with 4 000-token prompts spends $40 000/day on input alone. A 2× compression halves that to $20 000.

Token limits. Every model has a finite context window. Even 128 K-token models struggle when you need to pack hundreds of retrieved documents. Compression lets you fit more relevant material inside the budget.

Latency. Prefill time scales roughly linearly with input length. Halving prompt tokens can shave 30-50 % off time-to-first-token in long-context scenarios.

The lost-in-the-middle problem. Research by Liu et al. (2023) showed that LLMs attend more strongly to the beginning and end of long contexts, often ignoring information in the middle. Compression that selectively retains high-signal tokens can actually improve accuracy by removing distracting filler.

# Quick token-count estimator for budgeting compression import tiktoken def estimate_savings(prompt: str, compression_ratio: float = 0.4): enc = tiktoken.encoding_for_model("gpt-4o") original = len(enc.encode(prompt)) compressed = int(original * compression_ratio) saved = original - compressed cost_saved = saved * 10 / 1_000_000 # $10/M tokens return { "original_tokens": original, "compressed_tokens": compressed, "tokens_saved": saved, "cost_saved_usd": round(cost_saved, 6), }

LLMLingua: Perplexity-Based Compression

LLMLingua (Jiang et al., 2023) is the foundational framework for prompt compression. Its key insight: tokens that a small language model finds predictable (low perplexity) contribute little new information and can be safely dropped. The algorithm has three stages.

Core idea: Use a small, well-aligned LM (e.g., LLaMA-7B) to score every token in the prompt by perplexity. Keep the high-perplexity (surprising, informative) tokens; drop the low-perplexity (predictable, redundant) ones.

1. Budget Controller. The total token budget is split across prompt segments — demonstrations, instructions, and the question — proportionally to their information density. This prevents the compressor from gutting a critical segment to save tokens in a verbose one.

2. Iterative Token-Level Pruning. Within each segment, LLMLingua computes per-token perplexity using the small LM and iteratively removes the lowest-perplexity tokens until the segment budget is met. Tokens are removed in rounds rather than all at once, because removing one token changes the perplexity landscape for its neighbours.

3. Distribution Alignment. Because the small scoring LM and the target LLM (e.g., GPT-4) have different vocabularies and learned distributions, LLMLingua applies an alignment step that adjusts perplexity thresholds so that tokens the target model finds important are preserved, not just those the scoring model finds surprising.

# LLMLingua-style perplexity-based token pruning (simplified) import torch from transformers import AutoModelForCausalLM, AutoTokenizer class LLMLinguaCompressor: def __init__(self, model_name="meta-llama/Llama-2-7b-hf"): self.tokenizer = AutoTokenizer.from_pretrained(model_name) self.model = AutoModelForCausalLM.from_pretrained( model_name, torch_dtype=torch.float16, device_map="auto" ) self.model.eval() def compute_token_perplexity(self, text: str) -> list: """Return per-token negative log-likelihood (higher = more surprising).""" ids = self.tokenizer.encode(text, return_tensors="pt").to(self.model.device) with torch.no_grad(): out = self.model(ids, labels=ids) shift_logits = out.logits[:, :-1, :] shift_labels = ids[:, 1:] loss_fn = torch.nn.CrossEntropyLoss(reduction="none") nll = loss_fn( shift_logits.view(-1, shift_logits.size(-1)), shift_labels.view(-1), ) return nll.cpu().tolist() def compress(self, text: str, ratio: float = 0.5) -> str: """Keep the top `ratio` fraction of highest-perplexity tokens.""" tokens = self.tokenizer.encode(text) nlls = self.compute_token_perplexity(text) # First token has no NLL score — always keep it scored = list(zip(tokens[1:], nlls)) keep_n = max(1, int(len(scored) * ratio)) ranked = sorted(scored, key=lambda x: x[1], reverse=True)[:keep_n] # Restore original order keep_set = {id(pair) for pair in ranked} kept_ids = [tokens[0]] + [ t for t, n in scored if id((t, n)) in keep_set ] return self.tokenizer.decode(kept_ids, skip_special_tokens=True)
Alignment matters: Without the distribution-alignment step, a LLaMA scorer might preserve tokens that LLaMA finds surprising but GPT-4 considers redundant (and vice versa). Always pair the scorer with alignment when cross-model compressing.

LongLLMLingua: Question-Aware Compression

LongLLMLingua extends the original framework with two critical improvements designed for retrieval-augmented settings where dozens of documents compete for context-window space.

1. Question-Aware Document Reordering. Instead of compressing documents in their retrieval order, LongLLMLingua first scores each document by its contrastive perplexity — how much more surprising each document is given the question versus without it. High-contrast documents are placed at the start and end of the context (where LLMs attend most), directly mitigating the lost-in-the-middle problem.

2. Conditional Compression. Token-level pruning is conditioned on the question: a token's importance is measured not by standalone perplexity but by how much it reduces perplexity of the question. Tokens that help the model "understand" the question are preserved even if they look redundant in isolation.

LLMLingua

  • Unconditional perplexity scoring
  • Budget splits across segments
  • No document reordering
  • Best for single-doc compression
  • ~2× compression, <5 % quality loss

LongLLMLingua

  • Question-conditioned perplexity
  • Contrastive document ranking
  • Position-aware reordering
  • Best for multi-doc RAG contexts
  • ~4× compression, <5 % quality loss
# Question-aware contrastive scoring (LongLLMLingua concept) def contrastive_score(compressor, doc: str, question: str) -> float: """Higher score → doc is more relevant to the question.""" ppl_unconditional = compressor.compute_token_perplexity(doc) ppl_conditional = compressor.compute_token_perplexity(question + " " + doc) # Contrastive: how much does the question change doc perplexity? unconditional_mean = sum(ppl_unconditional) / len(ppl_unconditional) conditional_mean = sum(ppl_conditional) / len(ppl_conditional) return unconditional_mean - conditional_mean def reorder_docs(compressor, docs: list, question: str) -> list: """Place highest-contrast docs at start and end (U-shape attention).""" scored = [ (doc, contrastive_score(compressor, doc, question)) for doc in docs ] ranked = sorted(scored, key=lambda x: x[1], reverse=True) # U-shape: top docs at edges, lowest-scored in the middle reordered = [] for i, (doc, score) in enumerate(ranked): if i % 2 == 0: reordered.append(doc) # front else: reordered.insert(0, doc) # back (reversed at end) return reordered
Practical impact: LongLLMLingua achieves up to 4× compression on NaturalQuestions with less than 1 % accuracy drop, while also recovering performance lost by the naive long-context ordering.

Context Distillation

Context distillation is an orthogonal strategy: instead of pruning tokens at inference time, you train a model (or a set of soft prompts) to internalize the knowledge that would otherwise live in the prompt. The result is a model that no longer needs the context at all.

Knowledge distillation variant. A teacher model receives the full prompt and generates outputs. A smaller student model is fine-tuned to replicate those outputs without the prompt context. The student effectively "memorizes" what the prompt taught the teacher.

Soft-prompt distillation. Instead of training an entire model, you train a small set of continuous embedding vectors (soft prompts) that, when prepended to the input, reproduce the effect of the full context. This reduces thousands of discrete tokens to 10-50 continuous vectors.

# Context distillation: train student to mimic teacher's prompted outputs import torch from transformers import AutoModelForCausalLM, AutoTokenizer, Trainer, TrainingArguments def distill_context( teacher_name: str, student_name: str, system_prompt: str, train_queries: list, ): """Train student to produce teacher's context-conditioned answers.""" teacher_tok = AutoTokenizer.from_pretrained(teacher_name) teacher = AutoModelForCausalLM.from_pretrained(teacher_name) student_tok = AutoTokenizer.from_pretrained(student_name) student = AutoModelForCausalLM.from_pretrained(student_name) # Step 1: Generate teacher outputs WITH the full system prompt distill_pairs = [] for query in train_queries: prompted_input = system_prompt + "\n\n" + query ids = teacher_tok.encode(prompted_input, return_tensors="pt") out = teacher.generate(ids, max_new_tokens=256) answer = teacher_tok.decode(out[0], skip_special_tokens=True) distill_pairs.append({"input": query, "output": answer}) # Step 2: Fine-tune student on (query → answer) WITHOUT system prompt # Student internalizes the context during training # ... standard Trainer loop with distill_pairs as dataset ... return student

Token-Level Compression

  • Applied at inference time
  • No training required
  • Works with any black-box LLM
  • Compression ratio is tunable per call
  • Slight quality loss on each request

Context Distillation

  • Applied at training time
  • Requires fine-tuning pipeline
  • Needs model weight access
  • Eliminates prompt entirely at inference
  • Quality locked in at training time
Trade-off: Context distillation is the most aggressive compression — 100 % token removal — but it's static. If the context changes (new documents, updated policies), you must retrain. Use it for stable, high-volume prompts where the context rarely changes.

Summarization-Based Approaches

A simpler family of compression methods uses an LLM itself to summarize long contexts before passing them to the final generation step. While less precise than perplexity scoring, summarization-based approaches are easy to implement and require no small scoring model.

Recursive Summarization. Long context is split into chunks; each chunk is summarized; the summaries are concatenated and summarized again, recursively, until the total fits within the token budget. This is the simplest approach but introduces compounding summarization errors at each level.

Map-Reduce. Each chunk is independently summarized (the "map" step), then all summaries are combined and a final summary is produced (the "reduce" step). Unlike recursive summarization, map-reduce is easily parallelizable and limits error propagation to two levels.

Hierarchical Summarization. Documents are organized into a tree. Leaf nodes contain raw text; parent nodes contain summaries of their children. At query time, you traverse from root to the most relevant leaves, carrying compressed context at each level. This gives you adaptive depth — more detail where it matters.

# Map-reduce summarization pipeline from openai import OpenAI from concurrent.futures import ThreadPoolExecutor client = OpenAI() def summarize_chunk(chunk: str, max_tokens: int = 200) -> str: """Map step: independently summarize one chunk.""" resp = client.chat.completions.create( model="gpt-4o-mini", messages=[ {"role": "system", "content": "Summarize concisely. Preserve key facts and numbers."}, {"role": "user", "content": chunk}, ], max_tokens=max_tokens, temperature=0.0, ) return resp.choices[0].message.content def map_reduce_compress(chunks: list, final_budget: int = 500) -> str: """Parallel map, then single reduce.""" # Map — parallelise across chunks with ThreadPoolExecutor(max_workers=8) as pool: summaries = list(pool.map(summarize_chunk, chunks)) # Reduce — combine summaries into one combined = "\n\n".join(summaries) final = summarize_chunk(combined, max_tokens=final_budget) return final # Usage docs = ["... long document 1 ...", "... long document 2 ...", "..."] compressed_context = map_reduce_compress(docs) print(f"Compressed to {len(compressed_context.split())} words")
When to use what: Summarization is best for soft compression where exact wording doesn't matter. Perplexity-based methods are better when precise token-level preservation is needed (e.g., code snippets, structured data, named entities).

Benchmarking Compression Quality

Compression is useless if it destroys the information the downstream task needs. Evaluating compression quality requires both intrinsic metrics (how faithful is the compressed text?) and extrinsic metrics (does the final task still work?).

Faithfulness metrics. Measure whether the compressed prompt retains the same facts as the original. Common approaches include NLI-based entailment scores (does the original entail the compressed version?), token-level F1 overlap, and semantic similarity via embedding cosine distance.

Downstream task performance. The gold standard: run the same benchmark with original and compressed prompts and compare scores. For QA tasks, measure exact match and F1; for summarization, use ROUGE; for code generation, use pass@k.

Compression–quality Pareto frontier. Plot compression ratio on the x-axis and task performance on the y-axis. The best method is the one closest to the top-right corner — high compression with minimal quality loss. Different methods dominate at different compression ratios.

# Evaluate compression quality on a QA benchmark import numpy as np from sklearn.metrics.pairwise import cosine_similarity from sentence_transformers import SentenceTransformer class CompressionEvaluator: def __init__(self): self.embedder = SentenceTransformer("all-MiniLM-L6-v2") def semantic_similarity(self, original: str, compressed: str) -> float: """Embedding cosine similarity between original and compressed.""" embs = self.embedder.encode([original, compressed]) return float(cosine_similarity([embs[0]], [embs[1]])[0][0]) def token_f1(self, original: str, compressed: str) -> float: """Token-level F1 overlap.""" orig_tokens = set(original.lower().split()) comp_tokens = set(compressed.lower().split()) if not orig_tokens or not comp_tokens: return 0.0 precision = len(orig_tokens & comp_tokens) / len(comp_tokens) recall = len(orig_tokens & comp_tokens) / len(orig_tokens) if precision + recall == 0: return 0.0 return 2 * precision * recall / (precision + recall) def compression_ratio(self, original: str, compressed: str) -> float: return 1.0 - len(compressed.split()) / len(original.split()) def evaluate_batch(self, pairs: list) -> dict: """Evaluate a batch of (original, compressed) pairs.""" sims, f1s, ratios = [], [], [] for orig, comp in pairs: sims.append(self.semantic_similarity(orig, comp)) f1s.append(self.token_f1(orig, comp)) ratios.append(self.compression_ratio(orig, comp)) return { "mean_semantic_sim": round(np.mean(sims), 4), "mean_token_f1": round(np.mean(f1s), 4), "mean_compression": round(np.mean(ratios), 4), } # Usage evaluator = CompressionEvaluator() results = evaluator.evaluate_batch([ ("The capital of France is Paris, a city known for the Eiffel Tower.", "Capital France Paris Eiffel Tower."), ]) print(results) # → {"mean_semantic_sim": 0.8731, "mean_token_f1": 0.5714, "mean_compression": 0.5455}
Key takeaway: No single compression method dominates everywhere. LongLLMLingua leads on multi-document RAG benchmarks, LLMLingua is simplest for single-document cases, and summarization works best when you need human-readable compressed text. Always benchmark on your specific task before committing to a method.
