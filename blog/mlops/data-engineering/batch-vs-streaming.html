Batch vs Streaming Data for ML

MLOps Series Data Engineering

Introduction

Every production ML system consumes data, but how that data arrives—and when it is processed—fundamentally shapes model freshness, infrastructure cost, and operational complexity. The batch-vs-streaming decision is not a one-time architectural choice; it is a spectrum that most mature teams navigate differently for different features, models, and business domains.

At one extreme, a nightly Spark job aggregates the previous day's transactions into features for a credit-scoring model. At the other, a Flink pipeline computes click-through-rate features in sub-second windows for a real-time recommendation engine. Between these poles lie hybrid architectures —Lambda and Kappa—that try to give teams the best of both worlds without doubling operational burden.

This post walks through the fundamentals of each paradigm, the two dominant hybrid architectures, the latency-throughput tradeoff surface, and a practical framework for choosing the right approach for your ML workload.

Prerequisites: Familiarity with distributed data systems (MapReduce, message brokers) and basic feature engineering concepts. We assume you have read the earlier posts in this series on data pipelines and feature stores.

Batch Processing Fundamentals

Batch processing operates on bounded datasets—finite collections of records accumulated over a fixed time window (hourly, daily, weekly). The processing engine reads the entire dataset, applies transformations, and writes the result to a sink before the next window arrives.

Core Properties

  • High throughput: Engines like Apache Spark, Hive, and Presto are optimised for scanning terabytes. Columnar formats (Parquet, ORC) and predicate pushdown make full-table aggregations efficient.
  • Deterministic replay: Because the input is bounded and immutable, re-running a batch job on the same input always produces the same output—critical for auditability and debugging.
  • Resource efficiency: Clusters can be spun up, used, and torn down (ephemeral compute). You pay only for the wall-clock time of the job, not for 24/7 infrastructure.
  • Simple exactly-once semantics: With atomic write-and-commit patterns (e.g., Spark's saveAsTable with Hive ACID or Delta Lake transactions), exactly-once is straightforward.

Typical Batch Pipeline

# Daily feature pipeline – Apache Spark (PySpark)
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder.appName("daily_features").getOrCreate()

# 1. Read yesterday's raw events from the data lake
events = spark.read.parquet(f"s3://lake/events/dt={yesterday}")

# 2. Compute aggregate features per user
user_features = events.groupBy("user_id").agg(
    F.count("event_id").alias("event_count_1d"),
    F.sum("purchase_amount").alias("spend_1d"),
    F.approx_count_distinct("product_id").alias("unique_products_1d"),
    F.avg("session_duration_s").alias("avg_session_s_1d"),
)

# 3. Write to the feature store (offline table)
user_features.write.mode("overwrite").format("delta") \
    .saveAsTable("ml.features.user_daily")

When Batch Excels for ML

Batch processing is the workhorse behind most training pipelines. Training sets are inherently bounded—you freeze a snapshot, compute features, join labels, split, and train. Batch also suits any feature whose business definition is inherently periodic: "average order value over the last 30 days", "monthly active users", or "quarterly churn probability".

The key limitation is staleness. If your model depends on features that change faster than your batch cadence, predictions degrade between refreshes. For a fraud-detection model, a feature like transactions_last_5_min cannot wait for tomorrow's Spark job.

Stream Processing Fundamentals

Stream processing operates on unbounded datasets—an infinite, continuously arriving sequence of events. Engines like Apache Flink, Apache Kafka Streams, and Google Dataflow process each event (or micro-batch) as it arrives, maintaining state across the stream.

Core Properties

  • Low latency: End-to-end latency from event ingestion to materialised feature can be milliseconds to low seconds, depending on windowing and checkpointing configuration.
  • Continuous computation: The pipeline is always running. There is no "job start" or "job end"—state is maintained across event arrivals via embedded RocksDB or remote state backends.
  • Windowing semantics: Tumbling, sliding, session, and global windows let you define aggregation boundaries over event time or processing time.
  • Complex exactly-once: Achieving exactly-once in a distributed streaming system requires coordinated checkpointing (Flink's Chandy-Lamport barriers), idempotent sinks, and transactional producers (Kafka transactions).

Typical Streaming Pipeline

# Real-time feature pipeline – Apache Flink (PyFlink)
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment

env = StreamExecutionEnvironment.get_execution_environment()
t_env = StreamTableEnvironment.create(env)

# 1. Consume from Kafka topic in real-time
t_env.execute_sql("""
    CREATE TABLE events (
        user_id   STRING,
        event_id  STRING,
        amount    DOUBLE,
        event_ts  TIMESTAMP(3),
        WATERMARK FOR event_ts AS event_ts - INTERVAL '5' SECOND
    ) WITH (
        'connector' = 'kafka',
        'topic'     = 'user-events',
        'format'    = 'json'
    )
""")

# 2. Sliding-window aggregation (5-min window, 1-min slide)
t_env.execute_sql("""
    INSERT INTO feature_sink
    SELECT
        user_id,
        COUNT(event_id)       AS event_count_5m,
        SUM(amount)            AS spend_5m,
        HOP_END(event_ts, INTERVAL '1' MINUTE, INTERVAL '5' MINUTE) AS window_end
    FROM TABLE(
        HOP(TABLE events, DESCRIPTOR(event_ts), INTERVAL '1' MINUTE, INTERVAL '5' MINUTE)
    )
    GROUP BY user_id, HOP_END(event_ts, INTERVAL '1' MINUTE, INTERVAL '5' MINUTE)
""")

When Streaming Excels for ML

Streaming is essential when prediction freshness is directly tied to business value. Real-time fraud detection, dynamic pricing, session-based recommendations, and anomaly detection on sensor data all require features computed over the last seconds or minutes. Streaming also enables continuous model monitoring—computing distribution drift metrics on live prediction traffic without waiting for a batch job.

The key cost is operational complexity. A streaming pipeline is a long-running, stateful distributed system. Failures require careful recovery from checkpoints, schema evolution must be handled without downtime, and resource provisioning must handle traffic spikes (back-pressure management).

Lambda Architecture

Proposed by Nathan Marz, the Lambda Architecture addresses the batch-streaming tension by running both in parallel. It consists of three layers:

The Three Layers

🗄️ Batch Layer

Stores the immutable, append-only master dataset (the system of record). Periodically recomputes batch views —complete, correct aggregations over the entire history. Typically runs on Hadoop/Spark reading from HDFS or a data lake.

Strength: Batch views are always eventually correct because they reprocess the full dataset from scratch.

⚡ Speed Layer

Processes only recent events that the batch layer hasn't incorporated yet. Produces real-time views that compensate for the batch layer's latency. Typically runs on Storm, Flink, or Kafka Streams.

Strength: Provides low-latency approximate results that are automatically superseded when the next batch run completes.

Serving Layer

The serving layer merges batch views and real-time views at query time. When a model requests a feature, the serving layer retrieves the last batch-computed value and overlays any incremental updates from the speed layer. Once the next batch run completes, the speed layer's state for that window is discarded.

Lambda Pros & Cons

✅ Advantages

  • Batch layer guarantees eventual correctness via full recomputation.
  • Speed layer provides low-latency results for time-sensitive features.
  • Fault-tolerant: if the speed layer fails, batch views are still valid.
  • Well-understood; battle-tested at LinkedIn, Twitter, and many large-scale systems.

❌ Disadvantages

  • Dual codebase: You maintain the same business logic in two different frameworks (e.g., Spark + Flink).
  • Operational overhead: Two pipelines to monitor, debug, and keep in sync.
  • Merge complexity: The serving layer must reconcile batch and speed views—edge cases around late-arriving data and out-of-order events can be subtle.
# Lambda: merging batch + speed views at serving time
def get_feature(user_id, feature_name):
    # 1. Read the last batch-computed value
    batch_val = batch_store.get(user_id, feature_name)
    batch_ts  = batch_store.get_timestamp(user_id, feature_name)

    # 2. Read incremental updates from the speed layer
    speed_delta = speed_store.get_since(user_id, feature_name, since=batch_ts)

    # 3. Merge: strategy depends on the feature type
    if feature_name.endswith("_count"):
        return batch_val + speed_delta          # additive merge
    elif feature_name.endswith("_latest"):
        return speed_delta if speed_delta else batch_val  # last-write-wins
    else:
        return weighted_merge(batch_val, speed_delta)

Kappa Architecture

Proposed by Jay Kreps (co-creator of Apache Kafka), the Kappa Architecture eliminates the batch layer entirely. The core insight: if your streaming engine can reprocess historical data from a replayable log (e.g., Kafka with long retention), then the batch layer is redundant.

How It Works

  1. Immutable log: All raw events are appended to a durable, replayable log (Kafka, Pulsar, Kinesis) with retention measured in days, weeks, or indefinitely (tiered storage).
  2. Single processing path: A single streaming application computes both real-time views and historical reprocessing. To "recompute from scratch", you deploy a new version of the streaming job that reads from the beginning of the log.
  3. Swap on completion: Once the new version catches up to the head of the log, traffic is switched from the old view to the new one. The old job is decommissioned.

Kappa Pros & Cons

✅ Advantages

  • Single codebase: One framework, one set of tests, one deployment pipeline.
  • Simpler operations: No merge logic between batch and speed layers.
  • Natural fit for event-sourced systems where the log is the source of truth.

❌ Disadvantages

  • Reprocessing cost: Replaying months of Kafka data through a streaming job is slower and more expensive than a Spark full-scan of Parquet files.
  • Log retention limits: Infinite retention is possible but expensive. Tiered storage mitigates this but adds operational complexity.
  • Not all computations are stream-friendly: Complex joins over large historical windows or ML training loops don't map cleanly to a streaming operator graph.
# Kappa: reprocessing via new consumer group
def reprocess_features(job_config, new_version):
    # 1. Deploy new version reading from earliest offset
    new_job = deploy_flink_job(
        jar=job_config.jar,
        consumer_group=f"features-v{new_version}",
        start_offset="earliest",
        sink_table=f"features_v{new_version}",
    )

    # 2. Wait until new job catches up to live traffic
    while not is_caught_up(new_job):
        sleep(60)
        log(f"Lag: {get_consumer_lag(new_job)} events")

    # 3. Atomic swap: point serving layer to new table
    serving_layer.swap(
        old_table=f"features_v{new_version - 1}",
        new_table=f"features_v{new_version}",
    )

    # 4. Drain and decommission old job
    cancel_job(f"features-v{new_version - 1}")
Kappa ≠ "streaming only". Many teams adopt Kappa for their online feature pipelines but still run batch Spark jobs for training data generation, backfills, and ad-hoc analytics. The architecture choice governs the serving path, not every workload in the organisation.

Latency vs Throughput Tradeoffs

The fundamental tradeoff in data processing is between latency (how quickly a single event is reflected in features) and throughput (how many events per second the system can handle at a given cost). Batching amortises per-event overhead, so larger batches increase throughput at the expense of latency. Streaming minimises latency but pays a per-event coordination tax.

The Tradeoff Spectrum

Quantifying the Tradeoff

Processing Mode Typical Latency Throughput Cost Efficiency
Event-at-a-time 1–50 ms 10K–100K eps Low (high per-event overhead)
Micro-batch (Spark Streaming) 500 ms–5 s 100K–1M eps Medium
Mini-batch (custom) 5–60 s 500K–5M eps Medium-High
Full batch (Spark/Hive) Minutes–hours Millions eps (amortised) High (ephemeral clusters)

Micro-Batching: The Middle Ground

Apache Spark Structured Streaming popularised micro-batching: accumulate events for a short interval (100 ms–10 s), then process them as a small batch. This gives sub-minute latency with batch-like throughput and exactly-once semantics via write-ahead logs. The trade-off is a latency floor determined by the trigger interval—you cannot go below it without switching to a true continuous engine like Flink. 

# Micro-batch with Spark Structured Streaming
stream_df = spark.readStream \
    .format("kafka") \
    .option("subscribe", "user-events") \
    .load()

features = stream_df \
    .withWatermark("event_ts", "10 seconds") \
    .groupBy(
        window("event_ts", "5 minutes", "1 minute"),
        "user_id"
    ).agg(
        F.count("*").alias("event_count_5m"),
        F.sum("amount").alias("spend_5m"),
    )

# Trigger every 2 seconds → ~2s latency floor
features.writeStream \
    .format("delta") \
    .trigger(processingTime="2 seconds") \
    .option("checkpointLocation", "s3://checkpoints/features") \
    .start()
Rule of thumb: If your acceptable feature staleness is >5 minutes, batch is almost always cheaper and simpler. If it's <1 second, you need true streaming. Between 1 second and 5 minutes, micro-batching offers the best cost-latency ratio for most workloads.

Feature Computation: Batch vs Streaming

In ML systems, the batch-vs-streaming question becomes concrete when building feature pipelines. Different features have different freshness requirements, and most production systems compute features using a mix of both paradigms.

Feature Taxonomy by Freshness

🐢 Batch Features

  • User lifetime value (computed daily)
  • 30-day rolling purchase average
  • Historical click-through rate
  • Demographic attributes (updated weekly)
  • Embedding vectors from periodic model runs
  • Aggregate statistics for normalisation

🐇 Streaming Features

  • Transactions in the last 5 minutes
  • Current session page-view count
  • Real-time device fingerprint signals
  • Live inventory levels
  • Seconds since last interaction
  • Sliding-window click-through rate

The Train/Serve Skew Problem

One of the most insidious bugs in ML systems is training-serving skew—features computed differently at training time (batch) vs serving time (streaming). If your training pipeline uses a Spark SQL expression and your serving pipeline uses a Flink UDF, subtle differences in null handling, timestamp rounding, or aggregation order can degrade model performance silently. 

# Anti-pattern: different logic in training vs serving

# Training (PySpark) – uses floor division
def avg_spend_7d_spark(df):
    return df.groupBy("user_id").agg(
        (F.sum("amount") / F.count("amount")).alias("avg_spend_7d")
    )

# Serving (Flink SQL) – uses true division, handles NULL differently
# SELECT user_id, SUM(amount) / NULLIF(COUNT(amount), 0) AS avg_spend_7d
# → Different results when count is 0!

# Solution: shared feature definition (e.g., Feast, Tecton)
from feast import Feature, FeatureView, Field
from feast.types import Float64

user_features = FeatureView(
    name="user_spending",
    entities=["user"],
    schema=[
        Field(name="avg_spend_7d", dtype=Float64),
        Field(name="txn_count_7d", dtype=Float64),
    ],
    # Single definition → Feast generates both batch and stream pipelines
    source=kafka_source,
    batch_source=redshift_source,
)

Strategies to Avoid Skew

  • Feature stores (Feast, Tecton, Feathr): Define features once; the store generates consistent batch and streaming computation.
  • Shared transformation libraries: Write feature logic in a language-agnostic format (SQL, DSL) and compile to both Spark and Flink targets.
  • Log-and-wait: At serving time, log the raw inputs and the computed features. Later, recompute features in batch from the same inputs and compare—any divergence signals skew.
  • Online-offline consistency tests: Periodically sample serving requests, recompute features via the batch path, and assert equality within a tolerance.

Choosing the Right Approach for ML

There is no universal "best" architecture. The right choice depends on your latency requirements, team expertise, data volume, and operational maturity. Use the following decision framework:

Decision Matrix

Signal → Batch → Streaming → Hybrid
Feature staleness tolerance > 1 hour < 1 minute Mix of both
Team streaming expertise Low High Medium
Data volume TB+ / periodic scans Continuous high-velocity events Both patterns present
Reprocessing frequency Rare, full recompute OK Frequent, incremental preferred Periodic + incremental
Infra budget Optimise cost (ephemeral) Always-on clusters OK Tiered budget allocation

Common Patterns in Production ML

  1. Batch-only (most common starting point): Daily or hourly Spark jobs compute all features. Models are retrained on a schedule. This is sufficient for the vast majority of ML use cases and should be your default unless latency requirements force otherwise.
  2. Batch + streaming features (Lambda-style): Slow-changing features (user profile, historical aggregates) are batch-computed. A handful of latency-sensitive features (session signals, recent activity) are streamed. The model consumes both at inference time via a feature store.
  3. Stream-first (Kappa-style): All features are computed via streaming. Batch reprocessing is handled by replaying the log. This works well when the team has deep Flink/Kafka expertise and the system is already event-driven.
  4. Near-real-time micro-batch: Spark Structured Streaming with triggers of 1–30 seconds provides a pragmatic middle ground for teams that want sub-minute features without operating a full Flink cluster. 
# Decision helper: choose processing mode per feature
def choose_processing_mode(feature):
    if feature.staleness_tolerance > timedelta(hours=1):
        return "BATCH"            # cheapest, simplest

    if feature.staleness_tolerance < timedelta(seconds=10):
        return "STREAMING"        # true event-at-a-time (Flink)

    if feature.staleness_tolerance < timedelta(minutes=5):
        return "MICRO_BATCH"      # Spark Structured Streaming

    return "BATCH"              # default: batch until proven otherwise


# Apply to all features in the registry
for feat in feature_registry.all():
    mode = choose_processing_mode(feat)
    print(f"{feat.name:30s} → {mode}")
    feat.set_processing_mode(mode)

Anti-Patterns to Avoid

  • "Stream everything" without latency requirements: Streaming infrastructure is expensive and complex. If nobody needs sub-minute features, don't build sub-minute pipelines.
  • Ignoring backfill: Any streaming feature must have a batch backfill path for training data generation. Without it, you can't train on historical data.
  • Mismatched windowing: Training on 24-hour batch windows but serving on 5-minute streaming windows creates skew. Align windowing semantics across the train/serve boundary.
  • No monitoring: Streaming pipelines fail silently. Monitor consumer lag, checkpoint duration, and feature freshness SLOs. Set alerts on staleness, not just throughput.
Start simple, evolve deliberately. Begin with batch. Add streaming only for features where you can measure the business impact of lower latency. Every streaming feature you add is a long-running distributed system you commit to operating—forever.
Watch for hidden costs: Streaming infrastructure requires always-on compute, persistent state backends, and 24/7 on-call. A single Flink cluster with RocksDB state can cost 3–5× more than an equivalent ephemeral Spark job—and that's before counting the operational burden.
