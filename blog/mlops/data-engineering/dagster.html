Dagster: Software-Defined Data Assets

MLOps Series Data Engineering

Dagster Philosophy

Most orchestration tools — Airflow, Luigi, Prefect — model pipelines as task graphs: directed acyclic graphs of imperative steps. You tell the system what to do. Dagster flips this paradigm: instead of defining tasks, you define the data assets you want to exist and how they depend on each other. The orchestrator derives the execution plan from those declarations.

This "asset-centric" model draws a clear line between what the pipeline produces and how it produces it. The same asset definition can be materialized to a local Parquet file during development and to a Snowflake table in production — simply by swapping an IO manager.

Core insight: In a traditional orchestrator you think "run task A, then task B." In Dagster you think "I need asset X, which depends on asset Y." The execution plan is a consequence of the dependency graph, not the other way around.

Key pillars of the Dagster philosophy:

  • Declarative data assets — describe what should exist, not the imperative steps to produce it.
  • Type-aware — every input and output can carry a Dagster type with runtime validation.
  • Testable by design — assets and ops are plain Python functions; you can unit-test them without a running orchestrator.
  • Separation of business logic and infrastructure — IO managers, resources, and executors are injected, not hard-coded.
  • Built-in observability — Dagit provides a real-time asset lineage graph, run timeline, and structured event log out of the box.

Software-Defined Assets

A software-defined asset (SDA) is a Python function decorated with @asset that declares: "this piece of data should exist, and here is how to compute it." Dagster infers the dependency graph from function parameters that share the names of other assets. 

from dagster import asset, AssetIn
import pandas as pd

# A root asset — no upstream dependencies
@asset
def raw_events() -> pd.DataFrame:
    """Ingest raw click-stream events from S3."""
    return pd.read_parquet("s3://lake/raw/events/")

# Dagster infers that cleaned_events depends on raw_events
@asset
def cleaned_events(raw_events: pd.DataFrame) -> pd.DataFrame:
    """Drop nulls, deduplicate, cast timestamps."""
    df = raw_events.dropna(subset=["user_id", "event_ts"])
    df = df.drop_duplicates(subset=["event_id"])
    df["event_ts"] = pd.to_datetime(df["event_ts"])
    return df

# Explicit key mapping when names differ
@asset(ins={"events": AssetIn(key="cleaned_events")})
def daily_active_users(events: pd.DataFrame) -> pd.DataFrame:
    """Aggregate unique users per day."""
    return (
        events.groupby(events.event_ts.dt.date)
              .agg(dau=("user_id", "nunique"))
              .reset_index()
    )

When you load these three assets into Dagit, the UI automatically renders a lineage graph: raw_events → cleaned_events → daily_active_users. You can materialize any subset of the graph, and Dagster will topologically sort the required upstream assets.

Naming matters. Asset function names become the canonical asset keys. If you rename a function, Dagster treats it as a new asset — the old one becomes an orphan. Use the key parameter to decouple the Python name from the logical key when refactoring.

Asset Groups and Partitions

Large data platforms may define hundreds of assets. Dagster provides asset groups for logical organization and partitions for efficient incremental processing: 

from dagster import asset, DailyPartitionsDefinition

partitions = DailyPartitionsDefinition(start_date="2024-01-01")

@asset(partitions_def=partitions, group_name="analytics")
def daily_revenue(context, cleaned_events: pd.DataFrame) -> pd.DataFrame:
    """Revenue aggregated for a single partition date."""
    partition_date = context.partition_key
    day_events = cleaned_events[
        cleaned_events["event_ts"].dt.date == pd.Timestamp(partition_date).date()
    ]
    return day_events.groupby("product_id").agg(
        revenue=("amount", "sum")
    ).reset_index()

Partitioned assets allow you to rematerialize only the slices that changed, avoiding full-table recomputation — critical for datasets with billions of rows.

Ops and Graphs

Before the asset-centric API, Dagster's primary abstractions were ops (units of computation) and graphs (compositions of ops). These still exist and are useful when you need fine-grained control over execution — e.g., branching logic, fan-out / fan-in patterns, or dynamic mapping. 

from dagster import op, graph, In, Out

@op(out={"valid": Out(), "invalid": Out()})
def validate_records(context, records: pd.DataFrame):
    """Split records into valid and invalid sets."""
    mask = records["email"].str.contains("@")
    yield Output(records[mask], output_name="valid")
    yield Output(records[~mask], output_name="invalid")

@op
def persist_valid(context, valid: pd.DataFrame):
    valid.to_parquet("s3://warehouse/valid_records/")

@op
def quarantine_invalid(context, invalid: pd.DataFrame):
    invalid.to_parquet("s3://quarantine/invalid_records/")

@graph
def ingest_pipeline():
    valid, invalid = validate_records(extract_records())
    persist_valid(valid)
    quarantine_invalid(invalid)

# Convert graph → job for execution
ingest_job = ingest_pipeline.to_job(
    resource_defs={"io_manager": fs_io_manager}
)

A graph wires ops together at definition time; calling .to_job() binds it to concrete resources and config, producing an executable job.

Assets (declarative)

  • Define what data should exist
  • Dependencies inferred from function signatures
  • Lineage graph is the primary mental model
  • Best for ETL / ELT, feature stores, analytics

Ops + Graphs (imperative)

  • Define steps to execute
  • Dependencies wired explicitly inside @graph
  • Execution plan is the primary mental model
  • Best for branching, fan-out, and non-data workloads

In practice, most modern Dagster codebases use assets for the bulk of their pipelines and drop down to ops/graphs only when they need conditional branching or dynamic task mapping.

IO Managers

IO managers are Dagster's abstraction for reading and writing data between steps. They decouple business logic from storage concerns: your asset function returns a DataFrame, and the IO manager decides whether to persist it as Parquet on S3, a table in BigQuery, or an in-memory object for testing. 

from dagster import IOManager, io_manager, InputContext, OutputContext
import pandas as pd

class SnowflakeIOManager(IOManager):
    """Persist DataFrames to Snowflake tables."""

    def __init__(self, conn_str: str):
        self.conn_str = conn_str

    def handle_output(self, context: OutputContext, obj: pd.DataFrame):
        table = context.asset_key.path[-1]   # e.g. "daily_active_users"
        obj.to_sql(table, self.conn_str, if_exists="replace")
        context.log.info(f"Wrote {len(obj)} rows → {table}")

    def load_input(self, context: InputContext) -> pd.DataFrame:
        table = context.asset_key.path[-1]
        return pd.read_sql_table(table, self.conn_str)

@io_manager(config_schema={"conn_str": str})
def snowflake_io_manager(context):
    return SnowflakeIOManager(context.resource_config["conn_str"])

Dagster ships with several built-in IO managers:

  • FilesystemIOManager — pickles objects to the local filesystem (default).
  • S3PickleIOManager — pickles to S3; useful for distributed execution.
  • dagster-snowflake, dagster-bigquery, dagster-duckdb — warehouse-native managers that map assets to tables automatically.
Why IO managers matter for ML: During local development you can use the filesystem IO manager for fast iteration. In CI you swap in an in-memory IO manager for deterministic tests. In production you use a warehouse IO manager — all without changing a single line of business logic.

Per-Asset IO Manager Override

You can assign different IO managers to different assets. For example, raw data may land in S3 while aggregated tables go to Snowflake: 

from dagster import Definitions, asset

@asset(io_manager_key="s3_io")
def raw_events() -> pd.DataFrame: ...

@asset(io_manager_key="snowflake_io")
def daily_active_users(cleaned_events): ...

defs = Definitions(
    assets=[raw_events, daily_active_users],
    resources={
        "s3_io":       s3_pickle_io_manager,
        "snowflake_io": snowflake_io_manager,
    },
)

Resources & Configuration

Resources represent external services or shared state that your ops and assets need — database connections, API clients, ML experiment trackers. They follow a dependency-injection pattern: you declare what a function needs, and Dagster provides it at runtime. 

from dagster import resource, asset, ConfigurableResource
from pydantic import Field

class MLflowResource(ConfigurableResource):
    """Wraps the MLflow tracking client."""
    tracking_uri: str = Field(description="MLflow server URL")

    def log_metric(self, key: str, value: float):
        import mlflow
        mlflow.set_tracking_uri(self.tracking_uri)
        mlflow.log_metric(key, value)

@asset
def trained_model(context, features: pd.DataFrame, mlflow: MLflowResource):
    """Train model, log metrics to MLflow."""
    model = train(features)
    mlflow.log_metric("rmse", model.rmse)
    return model

The new ConfigurableResource API (Dagster 1.3+) uses Pydantic models for configuration, giving you type-safe, validated config with IDE autocompletion.

Environment-Specific Configuration

A common pattern is to bind different resource implementations per environment: 

import os
from dagster import Definitions

if os.getenv("ENV") == "production":
    resources = {
        "io_manager": snowflake_io_manager,
        "mlflow": MLflowResource(tracking_uri="https://mlflow.prod.internal"),
    }
else:
    resources = {
        "io_manager": fs_io_manager,
        "mlflow": MLflowResource(tracking_uri="http://localhost:5000"),
    }

defs = Definitions(assets=[...], resources=resources)

Dagit UI & Observability

Dagit (now called the Dagster UI) is a rich web application that serves as the control plane for your data platform. It provides three main views:

  • Global Asset Lineage — a full-graph view of every asset, its upstream dependencies, materialization status, and freshness. Click any node to see run history and metadata.
  • Run Timeline — a Gantt-chart view of each step's wall-clock duration, making bottlenecks immediately visible.
  • Structured Event Log — every log message, expectation check, and output type is captured as a structured event, enabling powerful filtering.

Asset Observations and Metadata

Beyond materialization, assets can emit observations — lightweight metadata snapshots that do not trigger a recomputation. This is useful for recording row counts, schema drifts, or freshness timestamps: 

from dagster import asset, Output, MetadataValue

@asset
def cleaned_events(raw_events: pd.DataFrame) -> Output[pd.DataFrame]:
    df = raw_events.dropna().drop_duplicates()
    return Output(
        df,
        metadata={
            "row_count":   MetadataValue.int(len(df)),
            "null_pct":    MetadataValue.float(raw_events.isnull().mean().mean()),
            "schema":      MetadataValue.md(df.dtypes.to_markdown()),
        },
    )

This metadata is rendered inline in Dagit, giving data engineers a real-time dashboard of data quality without external tooling.

Schedules and Sensors

Dagster supports two complementary mechanisms for triggering materialization runs:

Schedules

Cron-based triggers. Define a cadence and Dagster launches runs on that interval. Ideal for nightly ETL, hourly rollups, or weekly model retraining. 

from dagster import (
    ScheduleDefinition,
    define_asset_job,
)

analytics_job = define_asset_job(
    "analytics_job",
    selection=[daily_revenue, daily_active_users],
)

daily_schedule = ScheduleDefinition(
    job=analytics_job,
    cron_schedule="0 6 * * *",  # 06:00 UTC
)

Sensors

Event-driven triggers. A sensor function polls an external system and yields RunRequest objects when a condition is met — e.g., a new file appears in S3. 

from dagster import sensor, RunRequest

@sensor(job=ingest_job, minimum_interval_seconds=60)
def new_file_sensor(context):
    new_files = list_new_s3_files("s3://lake/raw/")
    for f in new_files:
        yield RunRequest(
            run_key=f.key,
            run_config={"ops": {
                "extract": {
                    "config": {"path": f.key}
                }
            }},
        )
Freshness policies (Dagster 1.1+) let you define how stale an asset is allowed to be. When an asset exceeds its freshness window, Dagster can automatically trigger upstream materializations — a powerful alternative to cron for reactive pipelines.

End-to-End ML Pipeline Example

Let's wire everything together into a production ML pipeline: ingest data, engineer features, train a model, evaluate it, and gate deployment on quality checks. 

from dagster import asset, AssetIn, Output, MetadataValue
from sklearn.ensemble import GradientBoostingRegressor
from sklearn.model_selection import train_test_split
from sklearn.metrics import mean_squared_error
import numpy as np, pandas as pd

# ── 1. Feature Engineering ──
@asset(group_name="ml")
def feature_table(cleaned_events: pd.DataFrame) -> pd.DataFrame:
    """Build features: session length, page count, recency."""
    features = cleaned_events.groupby("user_id").agg(
        session_count  = ("session_id", "nunique"),
        total_pages    = ("page_url",    "count"),
        last_seen      = ("event_ts",   "max"),
    ).reset_index()
    features["recency_days"] = (pd.Timestamp.now() - features["last_seen"]).dt.days
    return features

# ── 2. Train / Evaluate ──
@asset(group_name="ml")
def trained_model(context, feature_table: pd.DataFrame, mlflow: MLflowResource):
    X = feature_table[["session_count", "total_pages", "recency_days"]]
    y = feature_table["ltv"]
    X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

    model = GradientBoostingRegressor(n_estimators=200, max_depth=5)
    model.fit(X_train, y_train)

    preds = model.predict(X_test)
    rmse  = np.sqrt(mean_squared_error(y_test, preds))

    mlflow.log_metric("rmse", rmse)
    context.log.info(f"RMSE = {rmse:.4f}")

    return Output(
        model,
        metadata={
            "rmse":         MetadataValue.float(rmse),
            "n_estimators": MetadataValue.int(200),
            "train_rows":   MetadataValue.int(len(X_train)),
        },
    )

# ── 3. Quality Gate ──
@asset(group_name="ml")
def deployment_gate(context, trained_model) -> dict:
    """Only promote if RMSE < threshold."""
    rmse = trained_model.rmse_  # stored from training
    approved = rmse < 0.15
    context.log.info(f"Gate {'PASSED' if approved else 'FAILED'}: RMSE={rmse:.4f}")
    return {"approved": approved, "rmse": rmse}

The asset lineage for this pipeline looks like:

raw_events → cleaned_events → feature_table → trained_model → deployment_gate

With partitions on the date dimension, you can retrain weekly while keeping daily feature refreshes — and Dagster tracks which model version was trained on which partition of features.

Dagster vs Alternatives

How does Dagster compare to the other major orchestrators in the modern data stack?

Feature Dagster Airflow Prefect
Core abstraction Assets Tasks (DAGs) Tasks (Flows)
IO management Built-in IO managers XCom (limited) Result handlers
Lineage tracking Native asset graph Dataset (v2.4+) Limited
Testing First-class, in-process Requires running scheduler Good (local runs)
Type system Rich Dagster types None Pydantic models
Partitioning Native, multi-dimensional Manual catchup Basic
Airflow is not going away. If your team already has hundreds of production DAGs in Airflow, a wholesale migration to Dagster is rarely justified. Consider Dagster for net-new workloads or greenfield projects, and use dagster-airflow to bridge the two.

Best Practices

After running Dagster in production across multiple data platforms, the following patterns consistently improve maintainability and reliability:

  1. One asset per logical table. Map each asset to a single destination table or file. Avoid mega-assets that produce multiple outputs — they defeat the purpose of granular lineage.
  2. Use IO managers everywhere. Never call df.to_parquet() directly inside an asset. Let the IO manager handle persistence so you can swap storage without touching business logic.
  3. Emit metadata liberally. Row counts, schema snapshots, freshness timestamps, and null percentages cost almost nothing to log but save hours during incident triage.
  4. Partition by time early. Even if your data fits in memory today, time-partitioned assets allow incremental rematerialization and back-filling without reprocessing the entire history.
  5. Test assets with materialize_to_memory. Dagster provides a built-in test utility that materializes assets in-process without any external infrastructure: 
from dagster import materialize_to_memory

def test_daily_active_users():
    result = materialize_to_memory(
        [raw_events, cleaned_events, daily_active_users]
    )
    assert result.success
    dau_df = result.output_for_node("daily_active_users")
    assert "dau" in dau_df.columns
    assert len(dau_df) > 0
  1. Use Definitions as the single entry point. Dagster 1.3+ encourages a single Definitions object that registers all assets, resources, schedules, and sensors in one place — making the deployment unit explicit: 
from dagster import Definitions, load_assets_from_modules
import assets.analytics, assets.ml

defs = Definitions(
    assets=load_assets_from_modules([assets.analytics, assets.ml]),
    resources={
        "io_manager": snowflake_io_manager,
        "mlflow":     mlflow_resource,
    },
    schedules=[daily_schedule],
    sensors=[new_file_sensor],
)

By following these patterns, you get a data platform where every asset is testable in isolation, observable in Dagit, and deployable to any environment — from a laptop to a Kubernetes cluster — without rewriting a single line of pipeline logic.
