Data Collection Patterns for ML at Scale

MLOps Series Data Engineering

Introduction

The performance ceiling of any machine-learning system is determined less by the model architecture and more by the quality, volume, and diversity of its training data. Yet data collection is routinely under-engineered — treated as a one-off script rather than a production-grade pipeline. In this post we survey the six dominant data-collection patterns used by ML teams operating at scale and distil the architectural decisions behind each one.

Each pattern has a distinct cost profile, latency envelope, and regulatory surface. Choosing the right mix — and wiring the patterns into a unified data lake — is the first real engineering decision of any ML project.

Rule of thumb: If your model needs < 10 k labelled samples, manual curation may suffice. Beyond that, you need at least one automated collection pipeline backed by scheduling, monitoring, and schema validation.

Web Scraping at Scale

Web scraping is the backbone of many NLP and computer-vision datasets. The two dominant Python frameworks — Scrapy and Beautiful Soup — serve complementary roles. Scrapy provides an asynchronous, spider-based crawl engine with built-in middleware for retries, rate-limiting, and feed exports. Beautiful Soup excels at ad-hoc extraction from irregular HTML and pairs naturally with requests or httpx.

Scrapy Spider — Product Catalog

import scrapy
from scrapy.loader import ItemLoader

class ProductSpider(scrapy.Spider):
    name = "products"
    start_urls = ["https://example.com/catalog"]
    custom_settings = {
        "CONCURRENT_REQUESTS": 16,
        "DOWNLOAD_DELAY": 0.25,       # politeness delay per domain
        "AUTOTHROTTLE_ENABLED": True,
    }

    def parse(self, response):
        for card in response.css("div.product-card"):
            loader = ItemLoader(selector=card)
            loader.add_css("title", "h2::text")
            loader.add_css("price", "span.price::text")
            loader.add_css("image_url", "img::attr(src)")
            yield loader.load_item()

        next_page = response.css("a.next-page::attr(href)").get()
        if next_page:
            yield response.follow(next_page, self.parse)

Beautiful Soup — Quick Extraction

import httpx
from bs4 import BeautifulSoup

async def fetch_articles(urls: list[str]) -> list[dict]:
    results = []
    async with httpx.AsyncClient(timeout=30) as client:
        for url in urls:
            resp = await client.get(url)
            soup = BeautifulSoup(resp.text, "lxml")
            title = soup.select_one("h1")
            body  = soup.select_one("article")
            results.append({
                "url": url,
                "title": title.get_text(strip=True) if title else None,
                "body": body.get_text(separator="\n", strip=True) if body else None,
            })
    return results

✅ Scrapy Strengths

  • Built-in concurrency & auto-throttle
  • Middleware pipeline (proxies, user-agent rotation)
  • Feed exports to S3, GCS, or local JSON/CSV
  • Robust retry & error-handling out of the box

✅ Beautiful Soup Strengths

  • Lenient parser handles broken HTML gracefully
  • Minimal boilerplate — ideal for prototyping
  • Composable with httpx for async I/O
  • Wide community & extensive documentation
Legal note: Always check a site's robots.txt and terms of service before scraping. Respect Crawl-delay directives and avoid scraping personal data unless you have a lawful basis under GDPR / CCPA.

API Ingestion Pipelines

Many valuable datasets live behind REST or GraphQL APIs — social media feeds, financial tickers, weather services, government open-data portals. A production ingestion pipeline must handle pagination, rate-limit back-off, incremental checkpointing, and schema evolution. 

import httpx, time, json
from pathlib import Path

class APICollector:
    """Generic paginated API collector with exponential back-off."""

    def __init__(self, base_url: str, token: str, sink: Path):
        self.base_url = base_url
        self.headers  = {"Authorization": f"Bearer {token}"}
        self.sink     = sink
        self.client   = httpx.Client(timeout=60, headers=self.headers)

    def collect(self, endpoint: str, params: dict | None = None):
        url     = f"{self.base_url}/{endpoint}"
        cursor  = None
        retries = 0

        while True:
            if cursor:
                params = params or {}
                params["cursor"] = cursor

            resp = self.client.get(url, params=params)

            if resp.status_code == 429:          # rate-limited
                wait = min(2 ** retries, 120)
                time.sleep(wait)
                retries += 1
                continue

            resp.raise_for_status()
            data = resp.json()
            self._write_batch(data["items"])

            cursor = data.get("next_cursor")
            if not cursor:
                break
            retries = 0   # reset on success

    def _write_batch(self, items: list[dict]):
        with open(self.sink, "a") as f:
            for item in items:
                f.write(json.dumps(item) + "\n")

Key architectural choices for API pipelines:

  • Idempotent checkpoints — persist the last cursor so a crashed job can resume without re-fetching.
  • Schema registry — validate each batch against a JSON Schema or Avro definition before landing it in the lake.
  • Backfill vs. incremental — on first run pull the full history; on subsequent runs only fetch updated_since records.

User Event Logs

Clickstream and in-app event data is the richest signal for recommender systems, search ranking, and personalisation models. The canonical architecture pipes events through a message broker (Kafka, Kinesis, Pub/Sub) into a columnar store (Parquet on object storage or a data-warehouse table). 

import json, datetime
from kafka import KafkaConsumer

def consume_events(topic: str, group: str, output_path: str):
    """Consume user events from Kafka and write newline-delimited JSON."""
    consumer = KafkaConsumer(
        topic,
        group_id=group,
        bootstrap_servers=["kafka-broker:9092"],
        value_deserializer=lambda m: json.loads(m.decode("utf-8")),
        auto_offset_reset="earliest",
        enable_auto_commit=False,
    )

    batch, batch_size = [], 500
    for msg in consumer:
        event = msg.value
        event["_ingested_at"] = datetime.datetime.utcnow().isoformat()
        batch.append(event)

        if len(batch) >= batch_size:
            flush_to_parquet(batch, output_path)
            consumer.commit()
            batch = []

    # flush remaining
    if batch:
        flush_to_parquet(batch, output_path)
        consumer.commit()

A well-designed event schema is critical. Every event should carry at minimum: event_type, timestamp, user_id (or anonymous session ID), device_context, and an arbitrary properties map. This keeps the schema flexible enough for new features while still enabling efficient columnar storage.

Privacy first: Hash or pseudonymise user identifiers at the collection boundary, not downstream. Apply retention policies (e.g., 90-day TTL) to raw event tables and archive only aggregated, de-identified datasets for model training.

IoT & Sensor Data

Autonomous vehicles, industrial-equipment monitoring, and wearable-health devices all generate high-frequency time-series data. The collection challenge is twofold: handling throughput (millions of readings per second) and maintaining temporal ordering for downstream models that rely on sequential patterns. 

import asyncio, json
from aiomqtt import Client as MQTTClient

async def ingest_sensor_stream(broker: str, topic: str):
    """Subscribe to an MQTT topic and buffer readings."""
    buffer: list[dict] = []

    async with MQTTClient(broker) as client:
        await client.subscribe(topic)
        async for message in client.messages:
            payload = json.loads(message.payload)
            buffer.append({
                "sensor_id": payload["id"],
                "ts": payload["timestamp"],
                "readings": payload["values"],
            })

            if len(buffer) >= 1000:
                await write_to_timescaledb(buffer)
                buffer = []

asyncio.run(ingest_sensor_stream("mqtt://iot-broker", "factory/line-3/#"))

Edge-side collection

  • Pre-aggregate on the device to reduce bandwidth
  • Use local ring buffers to survive network outages
  • Transmit compressed Protobuf payloads over MQTT / AMQP

Cloud-side landing

  • Ingest into a time-series DB (TimescaleDB, InfluxDB)
  • Partition by sensor_id and time bucket
  • Export daily snapshots to Parquet for batch ML training

For ML training it is common to window the raw stream into fixed intervals (e.g., 5-second or 1-minute aggregates) and compute statistical features (mean, variance, FFT peaks) before feeding the result into a model. The collection layer should preserve raw readings; feature engineering belongs in the transformation stage.

Crowdsourcing & Labeling

Supervised learning requires labels, and labels require human judgement. Crowdsourcing platforms — Amazon Mechanical Turk, Labelbox, Scale AI, Toloka — allow you to distribute annotation tasks across a large workforce. The engineering challenge is designing tasks that yield consistent, high-quality labels while controlling cost. 

from dataclasses import dataclass
from typing import Optional

@dataclass
class AnnotationTask:
    item_id: str
    data_url: str
    instructions: str
    redundancy: int = 3          # annotators per item
    gold_answer: Optional[str] = None  # for quality checks

def aggregate_labels(annotations: list[dict]) -> str:
    """Majority-vote aggregation with gold-question filtering."""
    from collections import Counter

    # Filter out annotators who failed gold questions
    trusted = [
        a for a in annotations
        if a["gold_accuracy"] >= 0.8
    ]

    votes = Counter(a["label"] for a in trusted)
    winner, count = votes.most_common(1)[0]

    # Flag items with low agreement for expert review
    agreement = count / len(trusted) if trusted else 0
    if agreement < 0.6:
        raise LowAgreementError(f"Agreement {agreement:.0%} — route to expert")

    return winner

Best practices for crowdsource pipelines:

  • Gold questions — mix known-answer items into every batch to measure annotator quality in real time.
  • Redundancy — assign each item to 3–5 annotators and aggregate via majority vote or Dawid-Skene.
  • Clear guidelines — ambiguity in instructions is the single largest source of label noise. Provide examples for every edge case.
  • Iterative refinement — after each pilot batch, review disagreements, update guidelines, and re-run.

Data Partnerships & Licensing

Not all data can — or should — be collected in-house. Data partnerships with vendors, academic institutions, and government agencies can shortcut months of collection work. The engineering concerns shift from extraction to integration: schema mapping, delivery cadence, and licence compliance.

Common delivery patterns

Push-based (vendor sends)

  • SFTP drops on a schedule (daily / weekly)
  • Webhook callbacks on new data events
  • Shared cloud-storage bucket with notifications

Pull-based (you fetch)

  • Vendor REST API with API-key auth
  • Snowflake / BigQuery data-share (zero-copy)
  • AWS Data Exchange or marketplace subscriptions
import hashlib, pathlib, json

def validate_delivery(manifest_path: str, data_dir: str) -> bool:
    """Verify a vendor data drop against its SHA-256 manifest."""
    with open(manifest_path) as f:
        manifest = json.load(f)

    for entry in manifest["files"]:
        path = pathlib.Path(data_dir) / entry["name"]
        if not path.exists():
            raise FileNotFoundError(f"Missing file: {path}")

        sha = hashlib.sha256(path.read_bytes()).hexdigest()
        if sha != entry["sha256"]:
            raise ValueError(f"Checksum mismatch for {path}")

    return True

Licence management is non-trivial. Track every dataset with a metadata record containing: source, licence type (CC-BY, proprietary, government open-data), permitted uses (research only, commercial OK, no redistribution), expiry date, and PII classification. Automate licence-expiry alerts so stale data is removed from training pipelines before it becomes a compliance risk.

Watch out: "Free for research" does not mean "free for production." Many academic datasets carry non-commercial clauses that prohibit use in deployed ML products. Always review the licence with legal counsel before training on partnership data.

Best Practices & Governance

Regardless of the collection pattern, every production pipeline shares a common set of requirements. The checklist below captures the governance and reliability fundamentals.

Universal pipeline checklist

  • Schema validation — Reject malformed records at the ingestion boundary, not in the training loop.
  • Idempotency — Every job should be safe to re-run. Use deterministic file names and upsert semantics.
  • Observability — Emit metrics for record count, error rate, latency p99, and data freshness. Alert on drift.
  • Lineage tracking — Tag every record with source_id, collected_at, and pipeline_version so you can trace any training sample back to its origin.
  • PII handling — Classify fields at collection time. Hash direct identifiers; redact or mask quasi-identifiers.
  • Retention policies — Define TTLs per data category. Automate deletion to stay compliant with GDPR, CCPA, and sector-specific regulations.
from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class CollectionManifest:
    """Immutable record describing one collection run."""
    pipeline: str
    source: str
    run_id: str
    started_at: datetime
    finished_at: datetime | None = None
    record_count: int = 0
    error_count: int = 0
    schema_version: str = "1.0"
    tags: dict = field(default_factory=dict)

    def success_rate(self) -> float:
        total = self.record_count + self.error_count
        return self.record_count / total if total else 0.0

A healthy data-collection platform is not one particular tool — it is the combination of clear ownership, automated quality gates, and a metadata catalog that lets every team member answer the question: "Where did this data come from, and am I allowed to use it?"
