Data Lineage: Source to Model

MLOps Series Data Engineering

What Is Data Lineage?

Data lineage is the end-to-end record of how data flows from its origin through every transformation, aggregation, and join until it reaches a downstream consumer — a dashboard, a feature store, or a trained model. It answers three deceptively simple questions: where did this data come from?, what happened to it along the way?, and who or what depends on it?

In traditional data warehousing, lineage was a nice-to-have for auditors. In ML systems it is a hard requirement. A single stale join key or silently changed upstream schema can invalidate months of training data and silently degrade model performance. Without lineage, debugging a model quality regression becomes a manual archaeological expedition through hundreds of Airflow DAGs, Spark jobs, and dbt models.

Lineage metadata is typically represented as a directed acyclic graph (DAG) where nodes are datasets, columns, or jobs, and edges encode producer–consumer or derivation relationships. The richness of the graph determines how useful it is: a table-level lineage graph tells you which tables feed a model, while a column-level graph tells you exactly which columns are involved — and can automatically propagate sensitivity labels, SLA requirements, and freshness constraints.

Key insight: Lineage is not just metadata — it is operational infrastructure. Teams that treat lineage as a first-class signal can build automated alerting for upstream breakages, automatic impact analysis for schema migrations, and root-cause tracing for model quality regressions.

Column-Level vs Table-Level Lineage

Lineage granularity is the first design decision. Table-level lineage records that model_features depends on raw_events and user_profiles. Column-level lineage refines this: it records that model_features.avg_session_duration is derived from raw_events.timestamp and raw_events.session_id via a window function.

Table-Level Lineage

  • Lightweight to capture — needs only job I/O metadata
  • Sufficient for impact analysis of table drops or renames
  • Supported by most orchestrators (Airflow, Prefect) out of the box
  • Cannot trace PII propagation through individual columns
  • Graphs are compact — hundreds of nodes at enterprise scale

Column-Level Lineage

  • Requires SQL parsing or runtime instrumentation
  • Enables PII tagging propagation and GDPR deletion tracing
  • Critical for feature-store debugging (which raw field feeds which feature)
  • Graphs can explode — millions of edges in a large warehouse
  • Tools: sqllineage, sqlglot, Marquez column-level API

In practice, most teams start with table-level lineage and selectively enable column-level tracking for regulated or high-value pipelines. The compute and storage cost of full column-level lineage across every Spark job is non-trivial, so a tiered strategy is sensible.

Column-Level Extraction with sqlglot

import sqlglot
from sqlglot.lineage import lineage

# Parse a SQL statement and extract column-level lineage
sql = """
  SELECT u.user_id,
         AVG(e.duration) AS avg_duration
  FROM   events e
  JOIN   users  u ON e.uid = u.user_id
  GROUP BY u.user_id
"""

# sqlglot traces each output column back to its source columns
for node in lineage(
    column="avg_duration",
    sql=sql,
    schema={"events": {"uid": "INT", "duration": "FLOAT"},
            "users":  {"user_id": "INT"}}
):
    print(f"  {node.source.sql()} -> {node.expression.sql()}")

The sqlglot library parses SQL into an AST and resolves column references through joins, sub-queries, CTEs, and window functions. It supports most SQL dialects (BigQuery, Spark SQL, Snowflake, Postgres) making it a practical choice for heterogeneous warehouse environments.

Lineage Capture Methods

There are three fundamental approaches to capturing lineage, each with different trade-offs in accuracy, coverage, and engineering effort.

1. Static SQL Parsing

Parse SQL queries at rest — from version-controlled dbt models, Airflow SQL templates, or query logs — and extract read/write dependencies without executing anything. Tools like sqllineage and sqlglot operate in this mode. The approach is fast, deterministic, and works on historical queries, but it cannot capture dynamic SQL (EXECUTE IMMEDIATE), UDFs written in Python/Java, or Spark DataFrame transformations that never materialise as SQL.

2. Runtime Instrumentation

Hook into the execution engine and capture lineage as a side effect of query execution. Spark's QueryExecutionListener and the OpenLineage Spark integration follow this pattern: they intercept the optimised logical plan after Catalyst has resolved all references and emit a lineage event with exact input/output datasets and column mappings. 

# Spark OpenLineage integration — automatic lineage capture
# Set these in spark-defaults.conf or via SparkConf

spark.extraListeners = io.openlineage.spark.agent.OpenLineageSparkListener
spark.openlineage.transport.type = http
spark.openlineage.transport.url  = http://marquez:5000/api/v1/lineage
spark.openlineage.namespace      = production

# Every Spark job now automatically emits START, COMPLETE, and FAIL
# events with full input/output dataset facets to Marquez.

Runtime instrumentation is the gold standard for accuracy — it captures exactly what the engine executed, including dynamically generated SQL and programmatic DataFrame operations. The downside is that it requires engine-specific integrations and adds a small overhead to each job.

3. Manual / Declarative Annotation

Engineers explicitly declare lineage relationships in configuration files, dbt meta tags, or custom YAML manifests. This is the simplest approach and works well for non-SQL pipelines (Python scripts, shell jobs, ML training code) where automated extraction is impossible. The obvious drawback is that manual annotations drift from reality over time unless enforced by CI checks. 

# dbt model with manually declared lineage metadata
# models/features/user_features.yml

version: 2
models:
  - name: user_features
    description: "Aggregated user features for churn model"
    meta:
      lineage:
        upstream:
          - raw.events
          - raw.user_profiles
          - staging.session_aggregates
        owner: ml-platform-team
        sla_hours: 6
    columns:
      - name: avg_session_duration
        description: "Mean session length in seconds"
        meta:
          derived_from:
            - raw.events.timestamp
            - raw.events.session_id
Watch out: Relying solely on manual annotations creates a false sense of security. Use automated capture as the source of truth and reserve manual annotations for supplementary context (ownership, SLA, business glossary terms) that parsers cannot infer.

Open-Source Tooling

The lineage tooling landscape has matured rapidly. Three projects dominate the open-source ecosystem, each with a distinct philosophy.

Apache Atlas

Originally built at Hortonworks for Hadoop governance, Atlas provides a type system, classification engine, and REST/Kafka-based ingestion API. It models lineage as typed entities (hive_table, spark_process) linked by process edges. Atlas excels in Hadoop-native environments but its Java-heavy architecture and HBase/Solr dependencies make it heavyweight for cloud-native stacks.

Marquez (LF AI & Data)

Marquez is a metadata service built around the OpenLineage standard. It stores jobs, datasets, and run-level lineage events in Postgres and exposes a clean REST API. Its lightweight architecture (single JAR + Postgres) makes it easy to deploy alongside Airflow, Spark, and dbt. Marquez recently added column-level lineage support, making it viable for regulated environments.

OpenLineage (The Standard)

OpenLineage is not a tool but a specification — a JSON schema for lineage events that any producer can emit and any consumer can ingest. It defines three core event types: START, COMPLETE, and FAIL, each carrying input/output dataset facets with optional column-level detail. Integrations exist for Airflow, Spark, dbt, Flink, and Great Expectations. 

# OpenLineage event structure (simplified)
{
  "eventType": "COMPLETE",
  "eventTime": "2024-11-15T08:30:00Z",
  "job": {
    "namespace": "production",
    "name": "etl.build_user_features"
  },
  "inputs": [
    {"namespace": "warehouse", "name": "raw.events",
     "facets": {"columnLineage": {"fields": {
       "avg_duration": {"inputFields": [
         {"namespace":"warehouse","name":"raw.events","field":"duration"}
       ]}
     }}}}
  ],
  "outputs": [
    {"namespace": "warehouse", "name": "features.user_features"}
  ]
}

Atlas vs Marquez

  • Atlas: Rich type system, classification / tagging engine, Kafka ingestion
  • Atlas: Heavy — requires HBase, Solr, Kafka, Zookeeper
  • Marquez: Lightweight — single JAR + Postgres
  • Marquez: OpenLineage-native — first-class Airflow/Spark/dbt support
  • Marquez: Column-level lineage via dataset facets

When to Choose What

  • Hadoop-heavy org with Ranger/Knox → Atlas
  • Cloud-native stack (Airflow + Spark on K8s) → Marquez
  • Multi-engine, multi-cloud → adopt OpenLineage as the wire format
  • Existing Datahub deployment → use Datahub's lineage backend
  • Need column-level + PII propagation → Marquez or Datahub

Building a Lineage Graph

Regardless of tooling, the core abstraction is a directed graph where datasets and jobs alternate as node types: a job reads from input datasets and writes to output datasets. This bipartite structure naturally prevents cycles and mirrors the producer–consumer reality of data pipelines.

Below is a Python implementation that builds an in-memory lineage graph from OpenLineage events and supports both forward (impact analysis) and backward (root-cause) traversal. 

from collections import defaultdict
from typing import Dict, List, Set

class LineageGraph:
    """Bipartite DAG: dataset ↔ job nodes with directional edges."""

    def __init__(self):
        self.downstream: Dict[str, List[str]] = defaultdict(list)
        self.upstream:   Dict[str, List[str]] = defaultdict(list)
        self.node_type:  Dict[str, str]       = {}  # "dataset" | "job"

    def add_job_run(self, job: str, inputs: List[str], outputs: List[str]):
        """Register a single job execution with its I/O datasets."""
        self.node_type[job] = "job"
        for ds in inputs:
            self.node_type[ds] = "dataset"
            self.downstream[ds].append(job)
            self.upstream[job].append(ds)
        for ds in outputs:
            self.node_type[ds] = "dataset"
            self.downstream[job].append(ds)
            self.upstream[ds].append(job)

    def impact(self, node: str) -> Set[str]:
        """Forward traversal — everything downstream of this node."""
        visited: Set[str] = set()
        stack = [node]
        while stack:
            n = stack.pop()
            if n in visited:
                continue
            visited.add(n)
            stack.extend(self.downstream[n])
        return visited - {node}

    def root_cause(self, node: str) -> Set[str]:
        """Backward traversal — all ancestors of this node."""
        visited: Set[str] = set()
        stack = [node]
        while stack:
            n = stack.pop()
            if n in visited:
                continue
            visited.add(n)
            stack.extend(self.upstream[n])
        return visited - {node}

# --- Usage ---
g = LineageGraph()
g.add_job_run("etl.clean_events",
    inputs=["raw.clickstream", "raw.user_profiles"],
    outputs=["staging.clean_events"])
g.add_job_run("etl.build_features",
    inputs=["staging.clean_events"],
    outputs=["features.user_features"])
g.add_job_run("ml.train_churn_model",
    inputs=["features.user_features"],
    outputs=["models.churn_v3"])

# Impact analysis: what breaks if raw.clickstream schema changes?
print(g.impact("raw.clickstream"))
# {'etl.clean_events', 'staging.clean_events', 'etl.build_features',
#  'features.user_features', 'ml.train_churn_model', 'models.churn_v3'}

# Root cause: what upstream data feeds models.churn_v3?
print(g.root_cause("models.churn_v3"))
# {'ml.train_churn_model', 'features.user_features', 'etl.build_features',
#  'staging.clean_events', 'etl.clean_events', 'raw.clickstream',
#  'raw.user_profiles'}

Lineage for ML Debugging

Model quality regressions are notoriously hard to debug because the root cause is rarely in the model code itself — it is almost always upstream: a schema change, a broken join, a delayed partition, or a silently changed feature definition. Lineage turns this needle-in-a-haystack problem into a systematic traversal.

Debugging Workflow

When a monitoring alert fires (e.g., prediction drift or accuracy drop), the on-call engineer follows a lineage-driven workflow:

  1. Identify the model version — look up the model artifact in the registry and retrieve its training run ID.
  2. Backward traverse — walk the lineage graph from the model back through the feature table, the feature-building job, and the staging tables to the raw sources.
  3. Check freshness and schema — at each node, verify that the data arrived on time and the schema matches what the downstream job expects. Lineage metadata often includes facets for row count, byte size, and schema hash.
  4. Diff against the last good run — compare the current lineage subgraph against the last successful training run. Any new edge, missing input, or changed schema hash is a candidate root cause. 
def diagnose_regression(graph: LineageGraph, model: str, last_good_run: dict):
    """Walk backward from a degraded model and flag changes."""
    ancestors = graph.root_cause(model)
    issues = []

    for node in ancestors:
        meta = fetch_metadata(node)  # from Marquez / catalog API

        # Check freshness — was the data delivered on time?
        if meta.last_updated < last_good_run["timestamp"]:
            issues.append(f"STALE: {node} not updated since {meta.last_updated}")

        # Check schema drift — did columns change?
        if meta.schema_hash != last_good_run.get(f"{node}.schema_hash"):
            issues.append(f"SCHEMA_DRIFT: {node} schema changed")

        # Check volume anomaly — did row count drop unexpectedly?
        if meta.row_count < last_good_run.get(f"{node}.row_count", 0) * 0.5:
            issues.append(f"VOLUME_DROP: {node} has {meta.row_count} rows (expected ~{last_good_run[f'{node}.row_count']})")

    return issues
Pro tip: Integrate lineage-based diagnostics directly into your model monitoring pipeline. When prediction drift exceeds a threshold, automatically trigger a backward traversal and attach the flagged issues to the PagerDuty or Slack alert. This cuts mean-time-to-resolution from hours to minutes.

Lineage for Compliance & Governance

Regulatory frameworks — GDPR, CCPA, HIPAA, the EU AI Act — impose requirements that are impossible to meet without lineage. Right to deletion (GDPR Article 17) requires that you can trace every copy and derivative of a user's data and confirm its erasure. Model explainability (EU AI Act) requires proving which data was used to train a model and demonstrating that prohibited attributes were not used directly or indirectly.

PII Propagation Tracking

Column-level lineage enables automatic PII label propagation: if raw.users.email is tagged as PII, every downstream column derived from it inherits the tag. This eliminates the need for manual PII audits and catches cases where PII leaks through joins or string concatenation that engineers did not anticipate. 

def propagate_pii_labels(graph: LineageGraph, catalog: dict):
    """Forward-propagate PII tags through column-level lineage."""
    pii_sources = [
        col for col, meta in catalog.items()
        if meta.get("pii") is True
    ]

    for source in pii_sources:
        downstream = graph.impact(source)
        for col in downstream:
            if graph.node_type.get(col) == "column":
                catalog[col]["pii"] = True
                catalog[col]["pii_source"] = source
                print(f"  Tagged {col} as PII (derived from {source})")

    return catalog

Deletion Certification

When a GDPR deletion request arrives, the compliance system uses lineage to enumerate every dataset and derived column that contains data attributable to the user. It then triggers deletion or anonymisation jobs for each node and records the completion status. The lineage graph serves as the deletion certificate — auditable proof that all derivatives were addressed.

Model Training Provenance

The EU AI Act requires that high-risk AI systems document the data used for training, validation, and testing. By linking the model registry entry to its lineage subgraph, you get a machine-readable provenance record: every raw table, every transformation, every feature, and every intermediate dataset that contributed to the model. This provenance record can be exported as a Model Card addendum or a compliance report for auditors.

Compliance trap: Table-level lineage is not sufficient for GDPR deletion certification. If raw.users feeds a feature table through a join, table-level lineage tells you the feature table is affected, but it does not tell you which columns carry PII. You need column-level lineage to generate a precise deletion plan.

Putting It All Together

Data lineage is the connective tissue of a mature ML platform. It bridges the gap between the data engineer who builds the pipeline, the ML engineer who consumes the features, the compliance officer who needs an audit trail, and the on-call engineer who is paged at 3 AM because the model's accuracy dropped. Investing in lineage early — ideally by adopting OpenLineage as a standard and Marquez or Datahub as a backend — pays compound dividends as the platform scales.
