Apache Kafka for Real-Time ML

MLOps Series Data Engineering

Introduction

Batch prediction pipelines that run once a day are fine for recommendation emails, but they fall flat when you need fraud scores in <100 ms or dynamic pricing that adapts to demand every second. Apache Kafka has emerged as the de-facto backbone for event-driven architectures precisely because it solves the hard problems — durable, ordered, high-throughput message delivery — while staying horizontally scalable.

In this post we will build up from Kafka's core primitives (brokers, topics, partitions) through the Python producer / consumer API, into Kafka Streams for real-time feature engineering, and finally wire everything into an end-to-end ML prediction pipeline that scores events as they arrive.

Prerequisites: Basic familiarity with Python and machine learning inference concepts. We use kafka-python for the client examples and assume a local Kafka cluster (Docker Compose is the easiest way to spin one up).

Kafka Architecture: Brokers, Topics & Partitions

A Kafka cluster is a set of one or more brokers — JVM processes that persist messages to disk and serve read / write requests from clients. Every broker can handle hundreds of thousands of messages per second and store terabytes of data.

Topics

A topic is the logical channel to which producers write and from which consumers read. Think of it as a durable, append-only log. Topics are identified by name — for example user-clicks, model-predictions, or feature-vectors.

Partitions

Each topic is split into one or more partitions. A partition is an ordered, immutable sequence of records. Kafka guarantees ordering within a single partition but not across partitions. Partitions are the unit of parallelism — more partitions allow more consumers to read concurrently.

When a producer sends a record, a partitioner decides which partition it lands in. The default strategy hashes the message key modulo the partition count: 

# Conceptual partitioning logic inside the Kafka producer
def partition(key, num_partitions):
    if key is None:
        return random_partition(num_partitions)
    return murmur2_hash(key) % num_partitions

Replication

Every partition can be replicated across N brokers (controlled by the replication.factor setting). One replica is the leader (handles all reads and writes) and the rest are followers that stay in sync. If the leader crashes, ZooKeeper (or KRaft in newer versions) elects a follower as the new leader — giving you fault tolerance with zero data loss when acks=all.

Partition count is hard to change. Increasing partitions later can break key-based ordering guarantees because the hash-to-partition mapping changes. Plan your partition count carefully up front — a common heuristic is max(throughput_MB/s ÷ 10, consumer_count).

Offsets and the Commit Log

Every record within a partition is assigned a monotonically increasing 64-bit offset. Consumers track their position by committing offsets back to Kafka (stored in an internal __consumer_offsets topic). This design decouples reading from writing — consumers can rewind to any offset and replay history, which is invaluable for back-filling feature stores or debugging ML pipelines.

Producers & Consumers in Python

The kafka-python library provides a Pythonic interface to the Kafka protocol. It supports all producer delivery semantics (acks=0, acks=1, acks=all) and consumer group coordination out of the box.

Writing a Producer

A producer serialises your data (JSON, Avro, Protobuf — your choice), assigns a key, and sends the record asynchronously. The send() call returns a Future so you can handle success / failure callbacks. 

from kafka import KafkaProducer
import json

# Create a producer that serialises values as JSON
producer = KafkaProducer(
    bootstrap_servers=["localhost:9092"],
    value_serializer=lambda v: json.dumps(v).encode("utf-8"),
    key_serializer=lambda k: k.encode("utf-8") if k else None,
    acks="all",           # wait for all replicas
    retries=3,
    linger_ms=5,            # micro-batch for throughput
)

# Publish a user-click event
event = {
    "user_id": "u-12345",
    "item_id": "item-789",
    "action": "click",
    "timestamp": "2024-07-01T14:22:01Z",
}
future = producer.send("user-clicks", key="u-12345", value=event)
record_meta = future.get(timeout=10)   # block until ack
print(f"Sent to {record_meta.topic}:{record_meta.partition} @ offset {record_meta.offset}")

Setting acks="all" ensures the record is persisted to every in-sync replica before the future resolves — the strongest durability guarantee Kafka offers. The linger_ms=5 setting lets the producer batch records for up to 5 ms, trading a tiny latency bump for significantly higher throughput.

Writing a Consumer

Consumers poll for new records in a loop. Each poll returns a batch of ConsumerRecord objects. The consumer commits offsets either automatically (default) or manually for at-least-once semantics: 

from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    "user-clicks",
    bootstrap_servers=["localhost:9092"],
    group_id="click-processor",
    auto_offset_reset="earliest",
    enable_auto_commit=False,   # manual commit for safety
    value_deserializer=lambda m: json.loads(m.decode("utf-8")),
)

for msg in consumer:
    payload = msg.value
    print(f"[{msg.partition}:{msg.offset}] user={payload['user_id']}")

    # Process the event (e.g., update feature store)
    process_click(payload)

    # Commit offset only after successful processing
    consumer.commit()
At-least-once vs exactly-once: Manual offset commits give you at-least-once delivery. If your consumer crashes after processing but before committing, the record is redelivered on restart. For ML feature updates this is usually acceptable — idempotent writes to a feature store make duplicates harmless.

Consumer Groups & Scalable Ingestion

A consumer group is a set of consumers that cooperatively read from a topic. Kafka assigns each partition to exactly one consumer in the group, so records are processed in parallel without duplication. If a consumer dies, its partitions are rebalanced to the remaining members.

This is the key scalability primitive for ML workloads: you can spin up as many consumers as there are partitions. If the topic user-clicks has 12 partitions, you can run 12 consumer instances to achieve 12× throughput. 

# Scaling consumers — each instance uses the same group_id
# Kafka automatically assigns partitions to each consumer

# Instance 1 (gets partitions 0-3)
consumer_1 = KafkaConsumer("user-clicks", group_id="ml-feature-updater", ...)

# Instance 2 (gets partitions 4-7)
consumer_2 = KafkaConsumer("user-clicks", group_id="ml-feature-updater", ...)

# Instance 3 (gets partitions 8-11)
consumer_3 = KafkaConsumer("user-clicks", group_id="ml-feature-updater", ...)

Multiple Consumer Groups

Different consumer groups read independently — each group maintains its own offsets. This lets you build fan-out architectures where the same stream of events feeds multiple ML systems:

  • Group A — updates the online feature store in Redis for real-time inference.
  • Group B — writes raw events to S3 / Parquet for offline training data.
  • Group C — streams events to a monitoring dashboard for data-quality checks.

Single Consumer Group

Each partition assigned to one consumer. Good for exactly-once-per-group processing. Throughput scales up to the partition count.

  • Partition-level ordering preserved
  • Automatic rebalancing on failure
  • Max parallelism = partition count

Multiple Consumer Groups

Each group reads all records independently. Perfect for fan-out to multiple downstream systems.

  • Independent offset tracking per group
  • No cross-group coordination
  • Same data feeds training, serving, and monitoring

Kafka Streams for Real-Time Feature Engineering

Raw events are rarely suitable as direct model inputs. You need to compute aggregations (click counts in the last 5 minutes), joins (enrich clicks with user profile data), and transformations (normalisation, hashing) in real time. Kafka Streams is a lightweight, library-based stream processing framework that runs inside your JVM application — no separate cluster needed.

Core Abstractions

  • KStream — an unbounded stream of key-value records (event stream). Each record is an independent fact.
  • KTable — a changelog stream where each key has a latest value (like a materialised view). Updates with the same key overwrite previous values.
  • GlobalKTable — a fully replicated KTable on every instance. Ideal for small lookup tables (user profiles, item metadata) that you join against high-volume streams.

Windowed Aggregations for Feature Computation

Windowed aggregations are the bread and butter of real-time feature engineering. Kafka Streams supports tumbling, hopping, sliding, and session windows. Below is a conceptual Python-flavoured pseudocode (Kafka Streams is Java-native but faust / quixstreams provide Python alternatives with similar semantics): 

import quixstreams as qx

# Define a Quix Streams application (Python Kafka Streams alternative)
app = qx.Application(broker_address="localhost:9092")

input_topic = app.topic("user-clicks", value_deserializer="json")
output_topic = app.topic("click-features", value_serializer="json")

sdf = app.dataframe(topic=input_topic)

# Tumbling window: count clicks per user in 5-minute windows
sdf = sdf.group_by("user_id")
sdf = sdf.tumbling_window(duration_ms=300_000)  # 5 minutes
sdf = sdf.count()

# Emit the windowed count as a feature vector
sdf = sdf.apply(
    lambda row: {
        "user_id": row["user_id"],
        "click_count_5m": row["count"],
        "window_start": row["window_start"],
    }
)

sdf.to_topic(output_topic)
app.run()

Stream-Table Joins for Enrichment

A common pattern is to join a high-volume event stream (clicks) with a slowly-changing dimension table (user profiles) to produce enriched feature vectors: 

# Pseudocode: join click stream with user-profile table
# user_profiles is a KTable / GlobalKTable
# clicks is a KStream

def enrich_click(click, profile):
    return {
        "user_id": click["user_id"],
        "item_id": click["item_id"],
        "user_segment": profile["segment"],
        "account_age_days": profile["account_age_days"],
        "click_count_5m": click["click_count_5m"],
    }

enriched = clicks.join(
    user_profiles,
    key="user_id",
    how="left",
    value_joiner=enrich_click,
)
State stores: Kafka Streams maintains local RocksDB state stores for aggregations and joins. These are backed by changelog topics, so state is fully recoverable after crashes. For ML feature computation this means your windowed counts survive restarts without data loss.

Kafka Connect: Bridging Data Systems

Kafka Connect is a framework for streaming data between Kafka and external systems without writing code. It uses pre-built connectors — source connectors pull data into Kafka, sink connectors push data out of Kafka.

Connectors for ML Pipelines

  • JDBC Source Connector — streams database change events (CDC) from PostgreSQL / MySQL into Kafka. Great for capturing label updates or new training samples.
  • S3 Sink Connector — writes Kafka records to S3 as Parquet or JSON files. Use this to build offline training datasets from your event stream.
  • Elasticsearch Sink Connector — indexes predictions and model metadata for search and dashboarding.
  • Debezium (CDC) — captures row-level changes from databases and publishes them as Kafka events. Essential for keeping feature stores synchronised with source-of-truth databases.
# Example: S3 Sink Connector configuration (JSON)
{
    "name": "s3-training-data-sink",
    "config": {
        "connector.class": "io.confluent.connect.s3.S3SinkConnector",
        "tasks.max": "4",
        "topics": "enriched-click-features",
        "s3.bucket.name": "ml-training-data",
        "s3.region": "us-east-1",
        "flush.size": "10000",
        "rotate.interval.ms": "600000",
        "format.class": "io.confluent.connect.s3.format.parquet.ParquetFormat",
        "partitioner.class": "io.confluent.connect.storage.partitioner.TimeBasedPartitioner",
        "path.format": "'year'=YYYY/'month'=MM/'day'=dd/'hour'=HH",

        "timezone": "UTC"
    }
}
Schema management matters. Use the Confluent Schema Registry with Avro or Protobuf to enforce contracts between producers and consumers. Without schemas, a single upstream field rename can silently break every downstream ML pipeline.

Real-Time ML Prediction Pipeline with Kafka

Now let us wire everything together into a complete, production-grade pipeline. The architecture follows a three-stage pattern: Ingest → Enrich → Score.

Pipeline Architecture

  1. Ingest: Raw events (clicks, transactions, sensor readings) land on a Kafka topic via producers or Kafka Connect source connectors.
  2. Enrich: A Kafka Streams / Faust application computes real-time features (windowed aggregations, stream-table joins) and writes enriched feature vectors to a new topic.
  3. Score: A scoring consumer reads enriched features, calls the ML model, and publishes predictions to an output topic. Downstream systems (API gateways, alerting) consume predictions.

The Scoring Consumer

The scoring consumer loads a pre-trained model (scikit-learn, ONNX, or a lightweight TensorFlow SavedModel), reads feature vectors from Kafka, runs inference, and publishes the prediction: 

import json
import numpy as np
import onnxruntime as ort
from kafka import KafkaConsumer, KafkaProducer

# Load the ONNX model once at startup
session = ort.InferenceSession("fraud_model.onnx")
input_name = session.get_inputs()[0].name

# Kafka consumer: read enriched feature vectors
consumer = KafkaConsumer(
    "enriched-click-features",
    bootstrap_servers=["localhost:9092"],
    group_id="fraud-scorer",
    value_deserializer=lambda m: json.loads(m.decode("utf-8")),
    enable_auto_commit=False,
)

# Kafka producer: publish predictions
producer = KafkaProducer(
    bootstrap_servers=["localhost:9092"],
    value_serializer=lambda v: json.dumps(v).encode("utf-8"),
    acks="all",
)

for msg in consumer:
    features = msg.value
    # Build the input tensor from the feature vector
    x = np.array([[
        features["click_count_5m"],
        features["account_age_days"],
        features.get("avg_transaction_amount", 0.0),
        features.get("distinct_ips_24h", 1),
    ]], dtype=np.float32)

    # Run inference
    prediction = session.run(None, {input_name: x})[0]
    fraud_score = float(prediction[0][1])

    # Publish the prediction
    result = {
        "user_id": features["user_id"],
        "fraud_score": fraud_score,
        "is_fraud": fraud_score > 0.85,
        "model_version": "v2.3.1",
    }
    producer.send("fraud-predictions", key=features["user_id"], value=result)
    consumer.commit()

Scaling the Scoring Layer

Because the scoring consumer is a standard Kafka consumer group, you can scale it horizontally by running more instances — each instance picks up a subset of partitions and scores independently. For GPU-accelerated inference, wrap the model call in a micro-batching layer that collects records for a few milliseconds before calling the model in a single batch: 

import time, threading, queue

batch_queue = queue.Queue()
BATCH_SIZE = 32
BATCH_TIMEOUT_MS = 10

def batch_scorer():
    """Background thread that collects records and scores in batches."""
    while True:
        batch = []
        deadline = time.time() + BATCH_TIMEOUT_MS / 1000
        while len(batch) < BATCH_SIZE and time.time() < deadline:
            try:
                item = batch_queue.get(timeout=0.001)
                batch.append(item)
            except queue.Empty:
                continue
        if batch:
            # Build a single numpy array for the whole batch
            x = np.stack([item["features"] for item in batch])
            predictions = session.run(None, {input_name: x})[0]
            for i, item in enumerate(batch):
                item["callback"](predictions[i])

scorer_thread = threading.Thread(target=batch_scorer, daemon=True)
scorer_thread.start()
Latency budgets: In a fraud detection pipeline, the end-to-end latency budget is often <200 ms. Profile each stage: Kafka produce (~2 ms), partition routing (~1 ms), Streams aggregation (~10-50 ms), model inference (~5-30 ms), prediction publish (~2 ms). The biggest knobs are batch size and model complexity.

Production Concerns & Best Practices

Deploying Kafka-backed ML systems in production requires attention to several cross-cutting concerns beyond the happy path.

Schema Evolution

Feature vectors evolve as data scientists add or remove features. Use the Confluent Schema Registry with backward / forward compatibility modes to ensure that old consumers can read new messages and vice versa. Avro and Protobuf both handle schema evolution gracefully; JSON does not.

Exactly-Once Semantics (EOS)

Kafka supports idempotent producers (enable.idempotence=true) and transactional APIs to achieve exactly-once processing across read-process-write cycles. For ML scoring pipelines where duplicate predictions are costly (e.g., triggering duplicate alerts), enable EOS: 

# Enable exactly-once semantics in the producer
producer = KafkaProducer(
    bootstrap_servers=["localhost:9092"],
    acks="all",
    enable_idempotence=True,
    transactional_id="fraud-scorer-tx-01",
)
producer.init_transactions()

for msg in consumer:
    producer.begin_transaction()
    try:
        prediction = score(msg.value)
        producer.send("fraud-predictions", value=prediction)
        producer.send_offsets_to_transaction(
            consumer.position(msg.partition),
            consumer.group_id,
        )
        producer.commit_transaction()
    except Exception:
        producer.abort_transaction()

Monitoring & Observability

Key metrics to track for a Kafka-powered ML pipeline:

  • Consumer lag — the difference between the latest offset and the consumer's committed offset. Rising lag means your scoring layer cannot keep up with the event rate.
  • End-to-end latency — timestamp at which the event was produced vs. timestamp at which the prediction was published. This is your real-time SLA.
  • Prediction throughput — predictions per second per consumer instance. Use this to plan capacity.
  • Model staleness — time since the model artefact was last updated. Stale models lead to silent accuracy degradation.

Dead Letter Queues

Records that fail deserialization or cause model inference errors should not block the consumer. Route them to a dead letter topic for later inspection: 

def safe_score(msg, producer):
    try:
        features = msg.value
        prediction = run_inference(features)
        producer.send("fraud-predictions", value=prediction)
    except Exception as e:
        # Route to dead letter queue for investigation
        error_record = {
            "original_topic": msg.topic,
            "partition": msg.partition,
            "offset": msg.offset,
            "error": str(e),
            "payload": msg.value,
        }
        producer.send("dlq-fraud-scorer", value=error_record)
        logger.error(f"DLQ routed: {e}")

Batch ML (Traditional)

  • Predictions computed on a schedule (hourly / daily)
  • High latency, simple architecture
  • Uses Spark / Airflow for orchestration
  • Training and serving share the same data warehouse
  • Best for: recommendations, churn prediction, reporting

Real-Time ML (Kafka)

  • Predictions computed per event (<200 ms)
  • Low latency, event-driven architecture
  • Uses Kafka Streams / Faust for feature engineering
  • Separate online (Kafka) and offline (S3) data paths
  • Best for: fraud detection, dynamic pricing, anomaly alerts

Capacity Planning Rules of Thumb

Use these rough guidelines when sizing your Kafka cluster for ML workloads:

  • Partitions: Aim for 1 partition per 50-100 MB/s of throughput per topic. Over-partitioning increases broker metadata overhead and rebalance time.
  • Retention: Keep at least 7 days of retention on feature topics so you can replay and back-fill feature stores after a bug fix.
  • Replication factor: Always use replication.factor=3 in production to survive broker failures without data loss.
  • Consumer instances: Never exceed the partition count — extra consumers sit idle. Right-size by monitoring consumer lag and scaling up when lag grows.
Key takeaway: Kafka gives you a durable, replayable, horizontally scalable event bus. Pair it with a stream processing library for real-time feature engineering and a lightweight model serving layer, and you have the backbone for any real-time ML system — from fraud detection to personalisation to autonomous decision-making.
