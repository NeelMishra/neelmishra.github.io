Schema Evolution: Managing Data Contracts

MLOps Series Data Engineering

Why Schemas Matter

In any production data system, multiple teams produce and consume data simultaneously. A feature engineering pipeline reads events written by a backend service; an analytics dashboard queries tables populated by an ETL job; a model training pipeline consumes features stored in a feature store. Every one of these hand-offs is a contract: producers promise to deliver data in a specific shape, and consumers rely on that shape to function correctly.

Without explicit schemas, these contracts are implicit — buried in code, column names, and tribal knowledge. When a producer adds a field, renames a column, or changes a type, downstream consumers break silently. In ML systems the failure mode is even worse: instead of a crash, the model might train on corrupted features and produce subtly wrong predictions that go undetected for days.

Schemas make contracts explicit and machine-enforceable. They enable:

  • Validation at ingestion: reject malformed records before they enter the pipeline.
  • Compatibility checks at deploy time: verify that a new producer version won't break existing consumers.
  • Automated documentation: the schema is the documentation — always in sync with reality.
  • Code generation: strongly-typed client libraries derived directly from the schema.
Rule of thumb: If more than one team reads your data, you need a schema. If more than one service writes to the same topic, you need a schema and a compatibility policy.

Compatibility Types

Schema evolution is the ability to change a schema over time without breaking producers or consumers that use older or newer versions. The key question is: which direction of change is safe? The answer depends on your compatibility mode.

Backward Compatible

A new schema can read data written by the old schema. Consumers are upgraded first; they must handle data produced by both old and new producers.

Safe changes:

  • Add a field with a default value
  • Remove a field (new reader ignores it)

Unsafe changes:

  • Add a required field without a default
  • Change a field type

Forward Compatible

An old schema can read data written by the new schema. Producers are upgraded first; old consumers must gracefully ignore unknown fields.

Safe changes:

  • Add a field (old reader ignores it)
  • Remove a field that has a default

Unsafe changes:

  • Remove a required field that old readers depend on
  • Rename a field

Full Compatible

Both backward and forward compatible. The strictest mode — safe for any upgrade order. This is the recommended default for production Kafka topics and shared data lakes.

Safe changes: add or remove optional fields with defaults.

None (Breaking)

No compatibility guarantees. Every change is potentially breaking. Only acceptable in development environments or when all consumers and producers are deployed atomically.

When used: rapid prototyping, single-service internal topics.

Transitive vs. non-transitive: Non-transitive compatibility only checks the latest two schema versions against each other. Transitive compatibility checks all historical versions. Always prefer FULL_TRANSITIVE in production — it prevents edge cases where v3 is compatible with v2 but not with v1.

Avro Schema Evolution

Apache Avro is the most common schema format in the Kafka ecosystem. Avro schemas are written in JSON, and the binary encoding is compact because field names are not stored in the payload — they come from the schema itself. This also means that the reader must have the writer's schema (or a compatible one) to deserialize data.

Avro's resolution rules define how a reader schema maps to a writer schema. Fields present in both are matched by name; missing fields use default values; extra fields are ignored.

Version 1 — Initial User Event Schema

{
  "type": "record",
  "name": "UserEvent",
  "namespace": "com.example.events",
  "fields": [
    {"name": "user_id",    "type": "string"},
    {"name": "event_type", "type": "string"},
    {"name": "timestamp",  "type": "long"}       // epoch millis
  ]
}

Version 2 — Add an Optional Field (Backward + Forward Compatible)

{
  "type": "record",
  "name": "UserEvent",
  "namespace": "com.example.events",
  "fields": [
    {"name": "user_id",    "type": "string"},
    {"name": "event_type", "type": "string"},
    {"name": "timestamp",  "type": "long"},
    {"name": "device_os",  "type": ["null", "string"], "default": null}
    // ↑ union type with null default — safe to add
  ]
}

Because device_os has a default of null, old readers (using v1) simply ignore the field, and new readers (using v2) populate null when reading old data. This change is fully compatible.

Version 3 — Dangerous: Removing a Field Without a Default

// BREAKING — removing 'event_type' which has no default in v1
{
  "type": "record",
  "name": "UserEvent",
  "fields": [
    {"name": "user_id",   "type": "string"},
    {"name": "timestamp", "type": "long"},
    {"name": "device_os", "type": ["null", "string"], "default": null}
  ]
}
// Old reader expects 'event_type' — Avro resolution fails!
Avro rule: You can only remove a field if the reader's schema has a default for it. Since v1 readers have no default for event_type, removing it in v3 breaks backward compatibility.

Protobuf Schema Evolution

Protocol Buffers (Protobuf) take a different approach to evolution. Each field is assigned a numeric tag, and the wire format uses tags — not names — to identify fields. This makes Protobuf inherently forward-compatible: unknown tags are skipped by old parsers.

Version 1 — Initial Message

syntax = "proto3";

message UserEvent {
  string  user_id    = 1;
  string  event_type = 2;
  int64   timestamp  = 3;  // epoch millis
}

Version 2 — Add a New Field

syntax = "proto3";

message UserEvent {
  string  user_id    = 1;
  string  event_type = 2;
  int64   timestamp  = 3;
  string  device_os  = 4;  // new — old readers skip tag 4
}

Old readers compiled against v1 silently ignore field 4. New readers see an empty string for device_os when reading old data (proto3 default). This is safe in both directions.

Protobuf Evolution Rules

Change Safe? Notes
Add a field with a new tag ✅ Yes Old readers skip unknown tags
Remove a field (keep tag reserved) ✅ Yes Use reserved keyword to prevent tag reuse
Reuse a field tag number ❌ No Old readers interpret bytes with the wrong type
Change field type ⚠️ Depends Some wire-compatible promotions allowed (int32 → int64)
Rename a field ✅ Yes Wire format uses tags, not names

Reserving Removed Fields

message UserEvent {
  reserved 2;                // tag 2 was 'event_type' — never reuse
  reserved "event_type";     // prevent name reuse too

  string  user_id     = 1;
  int64   timestamp   = 3;
  string  device_os   = 4;
  string  action_type = 5;   // replacement for event_type
}

Schema Registry

A Schema Registry is a centralized service that stores and versions schemas, and enforces compatibility rules at write time. The most widely used implementation is the Confluent Schema Registry, which supports Avro, Protobuf, and JSON Schema.

How It Works

  1. Producer registers a schema with the registry before sending data. The registry assigns a schema ID.
  2. Compatibility check: The registry compares the new schema against existing versions using the configured compatibility mode (BACKWARD, FORWARD, FULL, FULL_TRANSITIVE, etc.).
  3. Rejection on incompatibility: If the new schema violates the compatibility contract, registration fails and the producer cannot deploy.
  4. Consumer fetches the schema by ID from the registry to deserialize incoming messages.

Registering & Checking Compatibility via REST API

# Register a new schema version for subject 'user-events-value'
curl -X POST http://schema-registry:8081/subjects/user-events-value/versions \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d '{
    "schema": "{\"type\":\"record\",\"name\":\"UserEvent\",\"fields\":[...]}"
  }'
# Response: {"id": 42}

# Check compatibility before deploying
curl -X POST http://schema-registry:8081/compatibility/subjects/user-events-value/versions/latest \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d '{
    "schema": "{\"type\":\"record\",\"name\":\"UserEvent\",\"fields\":[...]}"
  }'
# Response: {"is_compatible": true}

# Set compatibility level for a subject
curl -X PUT http://schema-registry:8081/config/user-events-value \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d '{"compatibility": "FULL_TRANSITIVE"}'

Python Producer with Schema Registry

from confluent_kafka import SerializingProducer
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroSerializer

# 1. Connect to the registry
registry = SchemaRegistryClient({"url": "http://schema-registry:8081"})

# 2. Define the Avro schema string
schema_str = """
{
  "type": "record",
  "name": "UserEvent",
  "fields": [
    {"name": "user_id",    "type": "string"},
    {"name": "event_type", "type": "string"},
    {"name": "timestamp",  "type": "long"},
    {"name": "device_os",  "type": ["null", "string"], "default": null}
  ]
}
"""

# 3. Build the serializer — auto-registers the schema
avro_ser = AvroSerializer(registry, schema_str)

# 4. Configure the producer
producer = SerializingProducer({
    "bootstrap.servers": "kafka:9092",
    "value.serializer": avro_ser
})

# 5. Produce a message
producer.produce(
    topic="user-events",
    value={
        "user_id": "u-12345",
        "event_type": "page_view",
        "timestamp": 1717027200000,
        "device_os": "iOS 17"
    }
)
producer.flush()
CI/CD integration: Add a schema compatibility check to your deployment pipeline. Before deploying a new producer version, call the registry's compatibility endpoint. If it returns is_compatible: false, fail the build. This shifts schema-breaking bugs from runtime to deploy time.

Data Contracts

A data contract goes beyond a schema. While a schema defines the structure of data (field names, types, nullability), a data contract also specifies:

  • Semantics: What does each field mean? What units is timestamp in?
  • SLAs: Freshness guarantees, delivery latency, completeness thresholds.
  • Quality rules: user_id must match ^u-[a-z0-9]{5,}$; timestamp must be within 24h of now.
  • Ownership: Which team owns this data? Who approves schema changes?
  • Lifecycle: Deprecation timeline, migration path for breaking changes.

Data Contract as Code

# data_contracts/user_events.yaml
apiVersion: datacontract/v1
kind: DataContract
metadata:
  name: user-events
  owner: platform-team
  version: 2.1.0

schema:
  type: avro
  registry: http://schema-registry:8081
  subject: user-events-value
  compatibility: FULL_TRANSITIVE

quality:
  rules:
    - field: user_id
      check: regex('^u-[a-z0-9]{5,}$')
    - field: timestamp
      check: freshness('24h')
    - field: event_type
      check: not_null()
  completeness: 99.5%

sla:
  freshness: 5 minutes
  availability: 99.9%

lifecycle:
  deprecation_notice: 90 days
  migration_guide: docs/migration/user-events-v2-to-v3.md

Enforcing Contracts in Python

import re
from datetime import datetime, timedelta

class UserEventContract:
    """Validates a single UserEvent record against the data contract."""

    USER_ID_PATTERN = re.compile(r'^u-[a-z0-9]{5,}$')
    VALID_EVENTS = {"page_view", "click", "purchase", "signup"}
    MAX_AGE = timedelta(hours=24)

    def validate(self, record: dict) -> list[str]:
        errors = []

        # user_id format check
        if not self.USER_ID_PATTERN.match(record.get("user_id", "")):
            errors.append("user_id must match '^u-[a-z0-9]{5,}$'")

        # event_type not-null and valid enum
        if record.get("event_type") not in self.VALID_EVENTS:
            errors.append(f"event_type must be one of {self.VALID_EVENTS}")

        # timestamp freshness
        ts = record.get("timestamp", 0)
        event_time = datetime.fromtimestamp(ts / 1000)
        if datetime.now() - event_time > self.MAX_AGE:
            errors.append("timestamp older than 24h — stale record")

        return errors

# Usage in a pipeline stage
contract = UserEventContract()
for record in incoming_batch:
    violations = contract.validate(record)
    if violations:
        dead_letter_queue.send(record, violations)
    else:
        downstream_topic.send(record)

Impact on ML Pipelines

Schema evolution has outsized consequences in ML systems because models are stateful artifacts trained on a specific data shape. A schema change that is perfectly safe for a streaming consumer can still break a model's feature extraction, training, or serving pipeline.

Failure Modes

Schema Change Impact on ML Severity
Add optional field Feature extraction code doesn't use it — safe unless model retraining picks it up automatically Low
Remove a field used as a feature Feature pipeline crashes or fills nulls, degrading model accuracy Critical
Change field type (e.g., string → int) Feature encoding breaks; model may train on garbage values Critical
Change enum values One-hot encoding dimensionality changes; serving model expects different categories High
Semantic drift (same field, different meaning) Silent accuracy degradation — the hardest bug to detect Critical

Defensive Feature Engineering

from pyspark.sql import DataFrame
from pyspark.sql import functions as F

def extract_features(events: DataFrame) -> DataFrame:
    """Schema-aware feature extraction with fallbacks."""

    # Explicitly declare expected columns and their defaults
    expected = {
        "user_id":    ("string", None),
        "event_type": ("string", "unknown"),
        "timestamp":  ("long",   0),
        "device_os":  ("string", "unknown"),   # added in v2
    }

    # Add missing columns with defaults — survives schema lag
    for col_name, (col_type, default) in expected.items():
        if col_name not in events.columns:
            events = events.withColumn(col_name, F.lit(default).cast(col_type))

    # Type-safe casting — prevents silent type mismatches
    for col_name, (col_type, _) in expected.items():
        events = events.withColumn(col_name, F.col(col_name).cast(col_type))

    # Validate contract invariants
    valid = events.filter(
        F.col("user_id").isNotNull() &
        F.col("timestamp") > 0
    )

    invalid_count = events.count() - valid.count()
    if invalid_count > 0:
        log_metric("schema.contract_violations", invalid_count)

    return valid
Golden rule for ML: Never let a schema change silently alter the feature vector. Pin your feature extraction to an explicit list of expected columns, types, and defaults. Log and alert on any deviation.

Putting It All Together

A robust schema evolution strategy combines format-level rules (Avro, Protobuf), infrastructure-level enforcement (Schema Registry), and organizational-level agreements (data contracts). Here is a recommended workflow for production ML systems:

  1. Define the schema in version control. Store .avsc or .proto files alongside the producing service's code. Every schema change goes through code review.
  2. Set FULL_TRANSITIVE compatibility on all production subjects in the Schema Registry. This is the strictest mode and prevents the widest class of breaking changes.
  3. Add a CI gate that calls the Schema Registry compatibility endpoint. Block merges that introduce incompatible schemas.
  4. Attach a data contract that specifies SLAs, quality rules, and ownership. Enforce quality rules in the first stage of your pipeline (e.g., a Flink job or Spark streaming stage) and route violations to a dead-letter queue.
  5. Pin feature extraction schemas. In your ML pipeline, explicitly declare which schema version you expect. If a new version arrives, require a deliberate update to the feature extraction code — never auto-adopt new fields.
  6. Monitor schema drift. Track metrics like schema.version_in_use, contract.violation_rate, and feature.null_rate. Alert when these deviate from baseline.

Avro Strengths

  • Compact binary encoding (no field names on wire)
  • Rich schema resolution rules
  • First-class Kafka & Schema Registry support
  • Schema stored with data (self-describing files)

Protobuf Strengths

  • Faster serialization / deserialization
  • Tag-based evolution is inherently forward-compatible
  • Strong multi-language codegen (gRPC ecosystem)
  • Widely used in microservice communication

Neither format is universally superior. Use Avro when your primary transport is Kafka and you need rich schema-resolution semantics. Use Protobuf when you need high-performance RPC, strong cross-language support, or are already in the gRPC ecosystem. Many organizations use both: Protobuf for service-to-service calls and Avro for event streaming.

Key takeaway: Schema evolution is not just a serialization concern — it is a sociotechnical problem. The best tooling in the world won't help if teams deploy schema changes without checking compatibility or notifying downstream consumers. Invest in data contracts, CI gates, and a culture of schema ownership alongside the technical infrastructure.
