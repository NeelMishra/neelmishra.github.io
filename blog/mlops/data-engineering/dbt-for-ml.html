dbt for ML Feature Transforms

MLOps Series Data Engineering

Introduction

Machine learning models are only as good as the features they consume, and most feature logic lives in messy, undocumented SQL scripts scattered across notebooks. dbt (data build tool) brings software engineering discipline—version control, testing, documentation, and dependency management—to SQL-based transformations, making it an ideal backbone for feature engineering pipelines.

Unlike heavyweight orchestration frameworks, dbt works inside the warehouse. It compiles Jinja-templated SQL into pure SQL and executes it on engines like BigQuery, Snowflake, Redshift, or Databricks. This means your feature transforms run at warehouse scale with zero data movement.

This post walks through every dbt concept an ML engineer needs: models, refs, sources, tests, snapshots, incremental materialisation, and patterns for building a production-grade feature layer entirely in SQL.

Prerequisites: Basic SQL fluency and familiarity with at least one cloud data warehouse. Experience with feature stores (e.g., Feast, Tecton) is helpful but not required—dbt can serve as the transform layer that feeds any downstream store.

dbt Fundamentals

A dbt project is a directory of .sql and .yml files. Each SQL file is a model—a SELECT statement that dbt materialises as a table or view. Models reference each other with the ref() function, forming a DAG that dbt resolves at compile time.

Project Structure

-- Typical dbt project layout for ML features
my_feature_project/
├── dbt_project.yml
├── models/
│   ├── staging/          -- 1:1 maps to raw sources
│   │   ├── stg_events.sql
│   │   └── stg_users.sql
│   ├── intermediate/     -- shared business logic
│   │   └── int_user_sessions.sql
│   └── features/         -- final feature tables
│       ├── feat_user_engagement.sql
│       └── feat_user_lifetime.sql
├── tests/
│   └── assert_no_null_user_ids.sql
├── snapshots/
│   └── snap_user_profiles.sql
└── macros/
    └── rolling_agg.sql

Sources

Sources are the raw tables your warehouse already has. Declaring them in a YAML file gives you lineage tracking, freshness checks, and a single place to rename upstream tables without editing dozens of models. 

# models/staging/_sources.yml
version: 2

sources:
  - name: raw_app
    database: analytics_prod
    schema: raw
    freshness:
      warn_after: {count: 12, period: hour}
      error_after: {count: 24, period: hour}
    loaded_at_field: _loaded_ts
    tables:
      - name: events
      - name: users
      - name: transactions

Models and ref()

Every model is a SELECT. The ref() function resolves model names to fully-qualified table names, automatically building the dependency graph. 

-- models/staging/stg_events.sql
SELECT
    event_id,
    user_id,
    event_type,
    TIMESTAMP(event_ts)   AS event_at,
    properties
FROM {{ source('raw_app', 'events') }}
WHERE event_ts IS NOT NULL
-- models/intermediate/int_user_sessions.sql
WITH events AS (
    SELECT * FROM {{ ref('stg_events') }}
),

session_boundaries AS (
    SELECT
        user_id,
        event_at,
        event_type,
        CASE WHEN TIMESTAMP_DIFF(
            event_at,
            LAG(event_at) OVER (
                PARTITION BY user_id ORDER BY event_at
            ),
            MINUTE
        ) > 30 THEN 1 ELSE 0 END AS new_session_flag
    FROM events
)

SELECT
    user_id,
    event_at,
    event_type,
    SUM(new_session_flag) OVER (
        PARTITION BY user_id ORDER BY event_at
    ) AS session_id
FROM session_boundaries

Materialisation Strategies

view

Default materialisation. Creates a database view. Zero storage cost but re-executes on every query. Ideal for staging models that are read infrequently or used only by downstream models.

table

Full table rebuild on every dbt run. Guarantees a clean snapshot of the logic. Best for feature tables that are small or where rebuild is fast (< a few minutes).

incremental

Appends or merges only new/changed rows. Essential for large event-driven feature tables (billions of rows). Covered in depth in the Incremental Models section.

ephemeral

Compiled as a CTE and inlined into downstream models—never materialised in the warehouse. Useful for lightweight helper transforms that don't need their own table.

Testing & Data Quality

Data bugs silently corrupt features and poison models downstream. dbt's testing framework catches issues at build time, before features reach your training pipeline.

Schema Tests

Schema tests are declared in YAML and run automatically with dbt test. dbt ships four built-in tests; you can compose them freely. 

# models/features/_features.yml
version: 2

models:
  - name: feat_user_engagement
    description: "Rolling engagement features per user"
    columns:
      - name: user_id
        tests:
          - unique
          - not_null
      - name: sessions_7d
        tests:
          - not_null
      - name: avg_session_duration_7d
        tests:
          - not_null
          - accepted_values:
              values: []   # placeholder—use dbt_utils for range
      - name: signup_date
        tests:
          - not_null
          - relationships:
              to: ref('stg_users')
              field: user_id

Custom Data Tests

Any SQL query that returns rows is a failing test. This lets you encode domain invariants that go beyond null/unique checks. 

-- tests/assert_feature_freshness.sql
-- Fail if the most recent feature row is older than 2 days
SELECT COUNT(*) AS violations
FROM (
    SELECT MAX(computed_at) AS latest
    FROM {{ ref('feat_user_engagement') }}
)
WHERE latest < CURRENT_TIMESTAMP() - INTERVAL 2 DAY
-- tests/assert_no_future_events.sql
-- No event should have a timestamp in the future
SELECT *
FROM {{ ref('stg_events') }}
WHERE event_at > CURRENT_TIMESTAMP()
Warning: Schema tests run after the model is materialised. If a test fails, the bad table already exists in the warehouse. Use dbt build (not dbt run + dbt test separately) so that downstream models are not built on top of a failing upstream model.

dbt_utils & dbt_expectations

Community packages extend the built-in tests. dbt_utils adds expression_is_true, recency, at_least_one, and more. dbt_expectations ports Great Expectations-style checks (distribution tests, regex validation, cross-column comparisons) directly into YAML-configured dbt tests.

Snapshots & Slowly Changing Dimensions

User profiles, subscription tiers, credit scores—these attributes change over time, and your training data needs to reflect what the value was at prediction time, not what it is today. Training on current values when the model saw historical values is a textbook case of data leakage.

dbt snapshots implement Type-2 Slowly Changing Dimensions (SCD2). On each run, dbt compares the current source row to the previously captured version and, if anything changed, closes the old record and inserts a new one with updated validity timestamps. 

-- snapshots/snap_user_profiles.sql
{% snapshot snap_user_profiles %}

{{
    config(
        target_schema  = 'snapshots',
        unique_key     = 'user_id',
        strategy       = 'timestamp',
        updated_at     = 'updated_at',
        invalidate_hard_deletes = True
    )
}}

SELECT
    user_id,
    subscription_tier,
    credit_score_band,
    country,
    updated_at
FROM {{ source('raw_app', 'users') }}

{% endsnapshot %}

After several runs the snapshot table contains a full change history: 

-- Querying the snapshot for point-in-time features
SELECT
    s.user_id,
    s.subscription_tier,
    s.credit_score_band
FROM {{ ref('snap_user_profiles') }} s
WHERE s.dbt_valid_from <= @prediction_ts
  AND (s.dbt_valid_to > @prediction_ts OR s.dbt_valid_to IS NULL)
Tip: Use the timestamp strategy when your source has a reliable updated_at column. Fall back to the check strategy (which hashes specified columns) when the source lacks a trustworthy modification timestamp.

Feature Engineering in SQL

SQL is remarkably expressive for feature engineering. Window functions, CTEs, and CASE expressions cover the vast majority of tabular feature patterns without leaving the warehouse.

Rolling Aggregations

Rolling windows are the bread and butter of behavioural features. The pattern below computes 7-day and 30-day aggregations in a single pass. 

-- models/features/feat_user_engagement.sql
{{
    config(
        materialized = 'incremental',
        unique_key   = 'user_id'
    )
}}

WITH daily_stats AS (
    SELECT
        user_id,
        DATE(event_at) AS event_date,
        COUNT(*)                                        AS events_today,
        COUNT(DISTINCT session_id)                       AS sessions_today,
        SUM(CASE WHEN event_type = 'purchase'
                THEN 1 ELSE 0 END)                      AS purchases_today,
        AVG(session_duration_sec)                        AS avg_session_sec
    FROM {{ ref('int_user_sessions') }}
    {% if is_incremental() %}
    WHERE event_at >= (SELECT MAX(computed_at) - INTERVAL 31 DAY
                       FROM {{ this }})
    {% endif %}
    GROUP BY user_id, DATE(event_at)
),

rolling AS (
    SELECT
        user_id,
        event_date,
        -- 7-day rolling windows
        SUM(events_today) OVER (PARTITION BY user_id
            ORDER BY event_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        )                                               AS events_7d,
        SUM(sessions_today) OVER (PARTITION BY user_id
            ORDER BY event_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        )                                               AS sessions_7d,
        AVG(avg_session_sec) OVER (PARTITION BY user_id
            ORDER BY event_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        )                                               AS avg_session_duration_7d,
        -- 30-day rolling windows
        SUM(events_today) OVER (PARTITION BY user_id
            ORDER BY event_date ROWS BETWEEN 29 PRECEDING AND CURRENT ROW
        )                                               AS events_30d,
        SUM(purchases_today) OVER (PARTITION BY user_id
            ORDER BY event_date ROWS BETWEEN 29 PRECEDING AND CURRENT ROW
        )                                               AS purchases_30d
    FROM daily_stats
)

SELECT
    user_id,
    events_7d,
    sessions_7d,
    avg_session_duration_7d,
    events_30d,
    purchases_30d,
    SAFE_DIVIDE(purchases_30d, sessions_7d)           AS purchase_rate_30d,
    CURRENT_TIMESTAMP()                                AS computed_at
FROM rolling
WHERE event_date = CURRENT_DATE()

Reusable Macros

When the same rolling pattern appears across many feature models, extract it into a Jinja macro. 

-- macros/rolling_agg.sql
{% macro rolling_agg(agg_func, column, partition, order, window_size) %}
    {{ agg_func }}({{ column }}) OVER (
        PARTITION BY {{ partition }}
        ORDER BY {{ order }}
        ROWS BETWEEN {{ window_size - 1 }} PRECEDING AND CURRENT ROW
    )
{% endmacro %}

Use it in any model:

SELECT
    user_id,
    event_date,
    {{ rolling_agg('SUM', 'events_today', 'user_id', 'event_date', 7) }}  AS events_7d,
    {{ rolling_agg('AVG', 'avg_session_sec', 'user_id', 'event_date', 7) }}  AS avg_session_7d
FROM daily_stats

Categorical Encoding

-- One-hot encoding via CASE (warehouse-native)
SELECT
    user_id,
    CASE WHEN device_type = 'mobile'  THEN 1 ELSE 0 END AS is_mobile,
    CASE WHEN device_type = 'desktop' THEN 1 ELSE 0 END AS is_desktop,
    CASE WHEN device_type = 'tablet'  THEN 1 ELSE 0 END AS is_tablet,
    -- Frequency encoding via a subquery / window
    COUNT(*) OVER (PARTITION BY country)
        / CAST(COUNT(*) OVER () AS FLOAT64)       AS country_freq_enc
FROM {{ ref('stg_users') }}

Time-Since Features

SELECT
    user_id,
    TIMESTAMP_DIFF(
        CURRENT_TIMESTAMP(),
        MAX(event_at),
        HOUR
    )                                   AS hours_since_last_event,
    TIMESTAMP_DIFF(
        CURRENT_TIMESTAMP(),
        MIN(event_at),
        DAY
    )                                   AS account_age_days
FROM {{ ref('stg_events') }}
GROUP BY user_id

Incremental Models

For tables with billions of rows, rebuilding from scratch on every run is wasteful. Incremental models process only the new or updated rows, cutting compute costs dramatically.

How Incremental Works

On the first run an incremental model behaves exactly like a table materialisation—full build. On subsequent runs, dbt wraps your SELECT in a merge (or insert-overwrite) that touches only rows matching the incremental predicate. 

-- models/features/feat_daily_revenue.sql
{{
    config(
        materialized   = 'incremental',
        unique_key     = ['user_id', 'event_date'],
        incremental_strategy = 'merge',
        on_schema_change     = 'sync_all_columns'
    )
}}

WITH src AS (
    SELECT
        user_id,
        DATE(event_at)          AS event_date,
        SUM(amount_usd)          AS daily_revenue,
        COUNT(*)                 AS txn_count,
        AVG(amount_usd)          AS avg_order_value,
        MAX(amount_usd)          AS max_order_value
    FROM {{ ref('stg_transactions') }}

    {% if is_incremental() %}
    -- only process rows since last run (with overlap buffer)
    WHERE event_at >= (SELECT MAX(event_date) - INTERVAL 3 DAY
                       FROM {{ this }})
    {% endif %}

    GROUP BY user_id, DATE(event_at)
)

SELECT
    user_id,
    event_date,
    daily_revenue,
    txn_count,
    avg_order_value,
    max_order_value,
    CURRENT_TIMESTAMP()     AS computed_at
FROM src
Late-arriving data: Always add a lookback buffer (the - INTERVAL 3 DAY above). Without it, rows that arrive late will never be processed, silently under-counting features.

Incremental Strategy Comparison

merge

Uses MERGE INTO (or equivalent). Upserts rows by unique_key. Supports updates to existing rows. Available on BigQuery, Snowflake, Databricks.

delete+insert

Deletes matching rows first, then inserts. Safer when you need to re-aggregate partitions. Default for Redshift and Postgres.

insert_overwrite

Replaces entire partitions. The most efficient approach when features are partition-aligned (e.g., by date). Native on BigQuery and Hive.

append

Pure insert, no deduplication. Only suitable for immutable event logs where duplicates are handled downstream.

dbt + ML Workflow Integration

dbt transforms data inside the warehouse. The ML pipeline consumes that data for training and inference. Connecting the two cleanly is the last mile of feature engineering.

Exposures

dbt exposures document downstream consumers—dashboards, ML models, APIs—giving you lineage from raw source to production model.

exposures:

    depends_on:
      - ref('feat_user_engagement')
      - ref('feat_daily_revenue')

    owner:
      name: ML Team

Point-in-Time Correct Training Sets

Combining feature tables with snapshot history lets you build training sets that are free of future-data leakage. 

-- models/training/train_churn_dataset.sql
WITH labels AS (
    SELECT
        user_id,
        label_date,
        churned   -- 1 if user churned within 30 days of label_date
    FROM {{ ref('churn_labels') }}
),

pit_profiles AS (
    -- Snapshot lookup: what was the profile AT label_date?
    SELECT
        s.user_id,
        l.label_date,
        s.subscription_tier,
        s.credit_score_band
    FROM {{ ref('snap_user_profiles') }} s
    JOIN labels l
      ON  s.user_id = l.user_id
      AND s.dbt_valid_from <= l.label_date
      AND (s.dbt_valid_to > l.label_date OR s.dbt_valid_to IS NULL)
),

engagement AS (
    SELECT * FROM {{ ref('feat_user_engagement') }}
)

SELECT
    l.user_id,
    l.label_date,
    l.churned,
    -- Engagement features
    e.events_7d,
    e.sessions_7d,
    e.avg_session_duration_7d,
    e.events_30d,
    e.purchases_30d,
    e.purchase_rate_30d,
    -- Profile features (point-in-time)
    p.subscription_tier,
    p.credit_score_band
FROM labels l
LEFT JOIN engagement e
  ON l.user_id = e.user_id
LEFT JOIN pit_profiles p
  ON l.user_id = p.user_id
  AND l.label_date = p.label_date

Orchestration

In production, an orchestrator (Airflow, Dagster, Prefect) triggers dbt build on a schedule, then triggers the training or inference job once dbt succeeds. The typical DAG looks like: 

# Airflow DAG (simplified)
dbt_build  = BashOperator(
    task_id  = "dbt_build",
    bash_cmd = "cd /opt/dbt && dbt build --select tag:features"
)

dbt_test   = BashOperator(
    task_id  = "dbt_test",
    bash_cmd = "cd /opt/dbt && dbt test --select tag:features"
)

train_model = PythonOperator(
    task_id   = "train_model",
    python_fn = train_churn_model
)

dbt_build >> dbt_test >> train_model

  ────────         ────────         ─────────────       ──────────
  raw.events  ──▶  stg_events  ──▶  int_sessions  ──▶  feat_engagement
  raw.users   ──▶  stg_users   ──▶  int_profiles  ──▶  feat_lifetime
  raw.txns    ──▶  stg_txns    ──────────────────────▶  feat_revenue

  Snapshots                        Training Sets
  ──────────                       ──────────────
  snap_user_profiles  ─────────────▶  train_churn_dataset

Best Practices Checklist

  • Staging models are views. They do only renaming, casting, and simple filtering—no business logic.
  • Feature models are incremental. They process only new data, with a lookback buffer for late arrivals.
  • Every feature column has a schema test. At minimum: not_null and unique on the primary key.
  • Snapshots cover mutable dimensions. Any attribute that can change and is used as a feature must be snapshotted.
  • Use tags for selective runs. Tag feature models with tag: features so orchestrators can run only the ML subset of a larger dbt project.
  • Document with descriptions. Every model and column should have a description in the YAML schema file. dbt auto-generates a searchable docs site via dbt docs generate.
  • Pin package versions. In packages.yml, always specify exact versions for dbt_utils and other packages to ensure reproducible builds.
  • Use CI with dbt build --defer. In pull requests, build only changed models against production state to catch regressions without full rebuilds.
Key takeaway: dbt turns ad-hoc feature SQL into a tested, versioned, documented DAG. Combined with snapshots for point-in-time correctness and incremental models for scale, it provides the foundation for reliable, reproducible ML feature pipelines—without leaving the warehouse.
