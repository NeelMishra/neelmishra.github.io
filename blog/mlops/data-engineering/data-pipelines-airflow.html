ML Data Pipelines with Apache Airflow

MLOps Series Data Engineering

Airflow Architecture

Apache Airflow is an open-source workflow orchestration platform that lets you programmatically author, schedule, and monitor complex data pipelines. Unlike cron-based approaches, Airflow models your pipeline as a Directed Acyclic Graph (DAG) of tasks, giving you dependency management, retries, back-fills, and observability out of the box.

At its core, Airflow consists of four primary components:

  • Scheduler — Parses DAG files, determines which tasks are ready to run based on dependencies and schedule intervals, and submits them to the executor.
  • Executor — Defines how tasks run. The LocalExecutor spawns processes on the same machine; the CeleryExecutor distributes work across a Celery cluster; the KubernetesExecutor spins up a pod per task.
  • Web Server — A Flask-based UI for monitoring DAG runs, inspecting task logs, triggering manual runs, and managing connections and variables.
  • Metadata Database — A PostgreSQL (or MySQL) instance that stores DAG definitions, task states, XCom values, connection credentials, and audit history.
Production tip: Always use PostgreSQL as the metadata database (not SQLite) and either the CeleryExecutor or KubernetesExecutor in production. SQLite and the SequentialExecutor are only suitable for local development.

When the scheduler detects that a task's upstream dependencies have all succeeded and its schedule interval has elapsed, it creates a TaskInstance record in the metadata database and hands it to the executor. The executor runs the task, reports success or failure, and the scheduler evaluates downstream dependencies. This decoupled architecture lets Airflow scale horizontally — you can add more Celery workers or let Kubernetes auto-scale pods without touching the scheduler. 

# Typical production Airflow deployment (docker-compose excerpt)
services:
  webserver:
    image: apache/airflow:2.8.1
    command: webserver
    ports: ["8080:8080"]

      AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: postgresql+psycopg2://airflow:airflow@postgres/airflow

  worker:

    image: postgres:15

      POSTGRES_USER: airflow
      POSTGRES_PASSWORD: airflow

DAG Definition

A DAG in Airflow is a Python file that defines a collection of tasks and their execution order. Airflow 2.x introduced the TaskFlow API (@dag and @task decorators) which dramatically simplifies DAG authoring compared to the classic operator-based style. Both approaches produce identical execution graphs; choose whichever fits your team's preferences. 

# Classic operator-based DAG definition
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime, timedelta

default_args = {
    "owner": "ml-team",
    "retries": 2,
    "retry_delay": timedelta(minutes=5),
    "email_on_failure": True,
}

with DAG(
    dag_id="ml_training_pipeline",
    default_args=default_args,
    description="End-to-end ML training DAG",
    schedule_interval="0 6 * * *",       # daily at 06:00 UTC
    start_date=datetime(2024, 1, 1),
    catchup=False,
    tags=["ml", "training"],
) as dag:

    extract = PythonOperator(
        task_id="extract_data",
        python_callable=extract_from_warehouse,
    )

    validate = PythonOperator(
        task_id="validate_schema",
        python_callable=run_great_expectations,
    )

    transform = PythonOperator(
        task_id="feature_engineering",
        python_callable=build_features,
    )

    train = PythonOperator(
        task_id="train_model",
        python_callable=train_xgboost,
    )

    evaluate = PythonOperator(
        task_id="evaluate_model",
        python_callable=compute_metrics,
    )

    # Define dependencies — the execution order
    extract >> validate >> transform >> train >> evaluate

The >> (bitshift) operator defines upstream dependencies. Airflow also supports <<, lists for fan-out ([a, b] >> c), and the chain() helper for complex graphs. Key DAG parameters to understand:

  • schedule_interval — A cron expression, timedelta, or dataset trigger that controls how often the DAG runs.
  • catchup — When True, Airflow back-fills all missed runs between start_date and now. Set to False for ML pipelines unless you need historical reprocessing.
  • max_active_runs — Limits concurrent DAG runs. Critical for GPU-bound training pipelines where parallelism is constrained by hardware.
  • tags — Metadata labels that make DAGs searchable in the Airflow UI.

TaskFlow API (Airflow 2.x)

# Modern TaskFlow API — cleaner, XCom-native
from airflow.decorators import dag, task
from datetime import datetime

@dag(
    dag_id="ml_training_taskflow",
    schedule="0 6 * * *",
    start_date=datetime(2024, 1, 1),
    catchup=False,
)
def ml_pipeline():

    @task()
    def extract():
        # Return value is auto-serialised as XCom
        return {"path": "s3://bucket/raw/2024-01-15.parquet"}

    @task()
    def validate(data_ref):
        run_great_expectations(data_ref["path"])
        return data_ref

    @task()
    def transform(data_ref):
        features = build_features(data_ref["path"])
        return {"features_path": features}

    @task()
    def train(features_ref):
        model = train_xgboost(features_ref["features_path"])
        return {"model_uri": model.uri}

    @task()
    def evaluate(model_ref):
        compute_metrics(model_ref["model_uri"])

    # Chaining return values auto-creates dependencies
    raw = extract()
    validated = validate(raw)
    features = transform(validated)
    model = train(features)
    evaluate(model)

ml_pipeline()
TaskFlow vs Classic: The TaskFlow API automatically handles XCom serialisation — return values from one task flow directly as function arguments to the next. Prefer TaskFlow for new DAGs; use the classic API when you need operators that don't map cleanly to Python functions (e.g., KubernetesPodOperator).

Operators Deep Dive

An Operator is a template for a single task. Airflow ships with dozens of built-in operators and the community maintains hundreds more via provider packages. For ML pipelines, the most important operators fall into three categories:

PythonOperator

The workhorse of ML DAGs. It executes an arbitrary Python callable, passing the Airflow context dictionary (which contains the execution date, task instance, DAG run info, and more) via **kwargs or the op_kwargs parameter. 

from airflow.operators.python import PythonOperator

def train_model(**kwargs):
    # Access execution context
    execution_date = kwargs["ds"]           # "2024-01-15"
    ti = kwargs["ti"]                       # TaskInstance object

    # Pull XCom from upstream task
    features_path = ti.xcom_pull(task_ids="feature_engineering")

    # Train and push model artifact URI
    model = xgb.train(params, dtrain)
    model_uri = save_to_registry(model, execution_date)
    ti.xcom_push(key="model_uri", value=model_uri)

train_task = PythonOperator(
    task_id="train_model",
    python_callable=train_model,
    provide_context=True,
)

BashOperator

Executes shell commands. Useful for invoking CLI tools, running dbt transformations, or triggering external processes. 

from airflow.operators.bash import BashOperator

dbt_run = BashOperator(
    task_id="dbt_transform",
    bash_command="cd /opt/dbt && dbt run --select tag:features",
    env={"DBT_PROFILE": "production"},
)

data_quality_check = BashOperator(
    task_id="run_great_expectations",
    bash_command="great_expectations checkpoint run ml_features",
)

Custom Operators

When built-in operators don't fit your workflow, create custom ones by subclassing BaseOperator. This is common for ML pipelines that need to interface with ML platforms, feature stores, or custom infrastructure. 

from airflow.models import BaseOperator
from airflow.utils.decorators import apply_defaults

class MLflowTrainOperator(BaseOperator):
    """Trains a model and logs to MLflow."""

    template_fields = ("experiment_name", "params")

    @apply_defaults
    def __init__(self, experiment_name, model_type, params, **kwargs):
        super().__init__(**kwargs)
        self.experiment_name = experiment_name
        self.model_type = model_type
        self.params = params

    def execute(self, context):
        import mlflow
        mlflow.set_experiment(self.experiment_name)
        with mlflow.start_run():
            mlflow.log_params(self.params)
            model = train(self.model_type, self.params)
            metrics = evaluate(model)
            mlflow.log_metrics(metrics)
            model_uri = mlflow.log_model(model, "model")
        return model_uri  # auto-pushed as XCom

KubernetesPodOperator

Runs a task in an isolated Kubernetes pod. Perfect for tasks that need custom Docker images (e.g., GPU training with CUDA) or strict resource limits.

SparkSubmitOperator

Submits PySpark jobs to a cluster. Ideal for large-scale feature engineering over terabytes of data where Pandas-based transforms would OOM.

Sensors

A Sensor is a special type of operator that waits for an external condition to be met before allowing downstream tasks to execute. Sensors are essential for ML pipelines because data arrives on unpredictable schedules — you can't train a model until today's feature table has been fully materialized.

Airflow provides two sensor modes: poke (the sensor occupies a worker slot and checks repeatedly at a fixed interval) and reschedule (the sensor releases its worker slot between checks, freeing resources). Always use reschedule mode for sensors with long expected wait times. 

from airflow.sensors.s3_key_sensor import S3KeySensor
from airflow.sensors.external_task import ExternalTaskSensor
from airflow.sensors.sql import SqlSensor

# Wait for raw data file to land in S3
wait_for_data = S3KeySensor(
    task_id="wait_for_raw_data",
    bucket_name="ml-data-lake",
    bucket_key="raw/events/{{ ds }}/part-*.parquet",
    aws_conn_id="aws_default",
    mode="reschedule",        # frees worker slot between pokes
    poke_interval=300,          # check every 5 minutes
    timeout=3600 * 6,           # fail after 6 hours
)

# Wait for upstream DAG to complete (cross-DAG dependency)
wait_for_features = ExternalTaskSensor(
    task_id="wait_for_feature_pipeline",
    external_dag_id="feature_engineering_daily",
    external_task_id="write_feature_table",
    mode="reschedule",
    timeout=7200,
)

# Wait for row count threshold in a database table
wait_for_labels = SqlSensor(
    task_id="wait_for_labels",
    conn_id="postgres_warehouse",
    sql="SELECT COUNT(*) > 10000 FROM labels WHERE dt = '{{ ds }}'",
    mode="reschedule",
    poke_interval=600,
)
Sensor deadlocks: If all your worker slots are occupied by sensors in poke mode, no other tasks can run — your entire cluster stalls. This is the #1 operational anti-pattern in Airflow. Mitigations: use reschedule mode, set smart_sensor (deprecated in 2.x), or use Deferrable Operators (Airflow 2.2+) which hand off waiting to the triggerer process.

Deferrable Operators (Async Sensors)

Airflow 2.2 introduced deferrable operators which solve the resource waste problem. Instead of occupying a worker slot, the task defers itself to a lightweight triggerer process that uses asyncio to monitor thousands of conditions concurrently: 

from airflow.providers.amazon.aws.sensors.s3 import S3KeySensorAsync

# Deferrable version — uses triggerer, zero worker slots consumed
wait_for_data_async = S3KeySensorAsync(
    task_id="wait_for_raw_data",
    bucket_name="ml-data-lake",
    bucket_key="raw/events/{{ ds }}/part-*.parquet",
    aws_conn_id="aws_default",
    poke_interval=300,
)

XComs & Data Passing

XCom (short for "cross-communication") is Airflow's mechanism for passing small pieces of data between tasks. When a task's execute() method returns a value, Airflow automatically serialises it as JSON and stores it in the metadata database. Downstream tasks can pull this value using xcom_pull().

This is essential for ML pipelines where you need to pass artifact references, metric dictionaries, or configuration between stages: 

# Pushing XComs (two styles)

# Style 1: Return value (auto-pushed with key "return_value")
def extract_data(**kwargs):
    path = download_from_warehouse(kwargs["ds"])
    return {"data_path": path, "row_count": 150_000}

# Style 2: Explicit push (multiple keys from one task)
def evaluate_model(**kwargs):
    ti = kwargs["ti"]
    metrics = compute_all_metrics()
    ti.xcom_push(key="accuracy", value=metrics["accuracy"])
    ti.xcom_push(key="f1_score", value=metrics["f1"])
    ti.xcom_push(key="auc_roc",  value=metrics["auc"])

# Pulling XComs
def register_model(**kwargs):
    ti = kwargs["ti"]
    model_uri = ti.xcom_pull(task_ids="train_model")
    accuracy  = ti.xcom_pull(task_ids="evaluate_model", key="accuracy")

    if accuracy > 0.95:
        promote_to_production(model_uri)
    else:
        log.warning("Accuracy below threshold, skipping promotion")
XCom size limits: XComs are stored in the metadata database — don't push large datasets through them. The default limit is ~48 KB (MySQL) or ~1 GB (PostgreSQL), but pushing even a few MB per task run creates bloat. Instead, push references (S3 URIs, table names, file paths) and let downstream tasks fetch the actual data.

Custom XCom Backends

For ML pipelines that need to pass larger artefacts (serialised DataFrames, model weights, embeddings), Airflow supports pluggable XCom backends. You can configure S3, GCS, or even a custom object store as the XCom storage layer: 

# airflow.cfg
[core]
xcom_backend = airflow.providers.amazon.aws.xcom.s3_xcom_backend.S3XComBackend

[aws]
xcom_s3_bucket = my-airflow-xcoms
xcom_s3_key_prefix = xcoms/

ML Pipeline Patterns

Beyond basic linear DAGs, production ML systems require sophisticated orchestration patterns. Here are the most important ones and how to implement them in Airflow:

Pattern 1: Branching (Conditional Training)

Use BranchPythonOperator to conditionally execute different training paths based on data characteristics or business rules. For example, retrain only if data drift exceeds a threshold: 

from airflow.operators.python import BranchPythonOperator

def check_drift(**kwargs):
    drift_score = kwargs["ti"].xcom_pull(task_ids="compute_drift")
    if drift_score > 0.15:
        return "retrain_model"       # task_id to execute
    return "skip_training"          # skip branch

branch = BranchPythonOperator(
    task_id="drift_gate",
    python_callable=check_drift,
)

compute_drift >> branch >> [retrain_model, skip_training]

Pattern 2: Dynamic Task Mapping (Airflow 2.3+)

Train multiple models in parallel with expand(). This is the Airflow equivalent of a distributed hyperparameter sweep: 

@task()
def generate_configs():
    return [
        {"lr": 0.01, "depth": 6, "estimators": 100},
        {"lr": 0.05, "depth": 8, "estimators": 200},
        {"lr": 0.1,  "depth": 4, "estimators": 500},
    ]

@task()
def train_variant(config):
    model = xgb.train(config)
    return {"config": config, "auc": evaluate(model)}

@task()
def select_best(results):
    best = max(results, key=lambda r: r["auc"])
    return best

# Dynamic fan-out: Airflow creates N parallel tasks at runtime
configs = generate_configs()
results = train_variant.expand(config=configs)  # maps over list
select_best(results)

Pattern 3: Dataset-Triggered DAGs (Airflow 2.4+)

Instead of time-based scheduling, trigger DAGs when upstream datasets are updated. This creates event-driven ML pipelines: 

from airflow.datasets import Dataset

# Define datasets as first-class objects
features_dataset = Dataset("s3://ml-bucket/features/daily")

# Producer DAG: marks the dataset as updated on completion
@dag(schedule="0 4 * * *")
def feature_pipeline():
    @task(outlets=[features_dataset])
    def write_features():
        compute_and_write_features()

# Consumer DAG: runs automatically when features_dataset is updated
@dag(schedule=[features_dataset])
def training_pipeline():
    @task()
    def train():
        train_model_on_latest_features()

Pattern 4: Task Groups (Sub-DAGs Done Right)

Organise related tasks into collapsible groups in the Airflow UI without the overhead and pitfalls of sub-DAGs: 

from airflow.utils.task_group import TaskGroup

with DAG("ml_pipeline_grouped", ...) as dag:
    with TaskGroup("data_prep") as prep:
        extract = PythonOperator(task_id="extract", ...)
        validate = PythonOperator(task_id="validate", ...)
        transform = PythonOperator(task_id="transform", ...)
        extract >> validate >> transform

    with TaskGroup("model_ops") as model:
        train = PythonOperator(task_id="train", ...)
        evaluate = PythonOperator(task_id="evaluate", ...)
        register = PythonOperator(task_id="register", ...)
        train >> evaluate >> register

    prep >> model  # connect groups as a single dependency

Scheduling & Monitoring

Getting the schedule right is critical for ML pipelines. Unlike web services, ML workflows have complex timing dependencies — you can't train until today's features are ready, and features depend on upstream data landing. Airflow provides several scheduling strategies:

Strategy When to Use
cron expression Fixed daily / hourly schedules. Use for batch training that runs at the same time every day.
timedelta Interval-based scheduling (timedelta(hours=6)). Use for retraining every N hours regardless of wall-clock time.
Dataset triggers Event-driven. The DAG runs when an upstream dataset is updated. Best for decoupled pipeline architectures.
@once / None Manual or programmatic triggering only. Use for ad-hoc model experimentation or one-time migrations.

Monitoring Best Practices

Airflow exposes a rich set of metrics via StatsD (or Prometheus with the statsd_exporter). For ML pipelines, monitor these key signals:

  • DAG run duration — Detect training time regressions that may indicate data volume spikes or infrastructure issues.
  • Task failure rate — Alert on repeated failures, especially in data extraction and validation tasks.
  • Sensor timeout rate — High sensor timeouts signal upstream data delivery problems.
  • XCom payload size — Track to prevent metadata database bloat.
  • Scheduler heartbeat — If the scheduler goes down, nothing runs.
# airflow.cfg — enable Prometheus-compatible metrics
[metrics]
statsd_on = True
statsd_host = statsd-exporter
statsd_port = 9125
statsd_prefix = airflow

# SLA configuration on individual tasks
train_task = PythonOperator(
    task_id="train_model",
    python_callable=train_model,
    sla=timedelta(hours=2),  # alert if training exceeds 2 hours
)

# Callback for custom alerting
def sla_miss_callback(dag, task_list, blocking_task_list, slas, blocking_tis):
    send_slack_alert(
        channel="#ml-alerts",
        message=f"SLA miss: {[t.task_id for t in task_list]}",
    )

Alerting Pipeline

Configure failure callbacks at both the DAG and task level to route alerts to Slack, PagerDuty, or email: 

def on_failure(context):
    task_id = context["task_instance"].task_id
    dag_id  = context["dag"].dag_id
    log_url = context["task_instance"].log_url
    send_slack_alert(
        channel="#ml-pipeline-alerts",
        message=f":x: Task {task_id} in {dag_id} failed. Logs: {log_url}",
    )

with DAG(
    dag_id="ml_training_pipeline",
    default_args={"on_failure_callback": on_failure},
    ...
) as dag:
    ...
Idempotency is non-negotiable. Every task in your ML pipeline must be idempotent — re-running it with the same inputs should produce the same outputs without side effects. This enables safe retries, back-fills, and manual re-runs. Design tasks to overwrite (not append) partitions, use deterministic random seeds, and always parameterise by execution date.

Apache Airflow has become the de facto orchestration standard for ML data pipelines. Its Python-native DAG definitions, extensible operator ecosystem, and first-class support for scheduling, retries, and observability make it the natural backbone for any MLOps platform. Combined with sensors for event-driven triggering, XComs for lightweight data passing, and dynamic task mapping for parallel experimentation, Airflow provides the orchestration primitives needed to move ML workloads from notebooks to production with confidence.
