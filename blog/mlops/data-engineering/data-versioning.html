Data Versioning with DVC

MLOps Series · July 2025 · 22 min read · Data Engineering

Machine-learning projects depend on code and data, yet most teams only version control their code. When a dataset changes—new labels, dropped columns, updated splits—reproducing earlier results becomes guesswork. DVC (Data Version Control) extends Git to track large files, datasets, and ML models without bloating the repository. Every data state gets a content-addressable hash committed alongside your code, so git checkout v1.2 also restores the exact data that produced that release.

Why Version Data?

Code versioning alone creates a dangerous illusion of reproducibility. The model weights you trained last month depend on the training set that existed at that moment. Without data versioning you cannot:

  • Reproduce experiments — re-running the same script on a silently-changed dataset yields different metrics.
  • Roll back regressions — if a data-quality bug tanks accuracy, you need the previous snapshot instantly.
  • Audit lineage — compliance and debugging both require a full chain from raw ingestion to model artifact.
  • Collaborate safely — two engineers adding features to the same dataset risk silent overwrites.

Without Data Versioning

  • Datasets live on shared NFS or a random S3 prefix
  • No link between code commit and data snapshot
  • train.csv overwritten in place
  • Reproducing last month's model is impossible
  • Data bugs discovered weeks after introduction

With DVC

  • Every dataset version content-addressed (SHA-256)
  • .dvc metafiles committed alongside code
  • git log shows data changes with code changes
  • dvc checkout restores exact data for any commit
  • Pipeline DAG makes lineage auditable end-to-end
Key insight: DVC stores lightweight pointer files (.dvc) in Git while the actual data lives in a remote—S3, GCS, Azure Blob, SSH, or even a local directory. This keeps your Git repo small while making every data version retrievable by hash.

DVC Init & Tracking

DVC layers on top of an existing Git repository. Initialization creates a .dvc/ directory (analogous to .git/) that holds cache configuration and internal state. From there, you track files or directories to bring them under version control.

Initializing a project

# Inside an existing git repo
$ pip install dvc[s3]          # install with S3 support
$ dvc init                     # creates .dvc/ directory
$ git add .dvc .dvcignore
$ git commit -m "Initialize DVC"

Tracking a dataset

When you run dvc add, DVC computes the MD5 hash of the file, moves the data into a local cache (.dvc/cache), and creates a .dvc metafile that records the hash, size, and path.

# Track a large training dataset
$ dvc add data/train.parquet

# DVC creates data/train.parquet.dvc (pointer file)
# and adds data/train.parquet to .gitignore

$ cat data/train.parquet.dvc
# outs:
#   - md5: a1b2c3d4e5f6...
#     size: 1073741824
#     path: train.parquet

$ git add data/train.parquet.dvc data/.gitignore
$ git commit -m "Track training data v1"

Tracking entire directories

DVC can track directories as single units. It hashes every file inside and stores a manifest (.dir file) in the cache. This is efficient for image datasets with thousands of small files.

$ dvc add data/images/         # track 50k images as one unit
$ git add data/images.dvc data/.gitignore
$ git commit -m "Track image dataset"

# Later, after adding new images:
$ dvc add data/images/
$ git diff data/images.dvc     # shows hash change
$ git commit -am "Update image dataset - added 2k samples"
Warning: Never git add the actual data files—only the .dvc pointer files. DVC automatically appends tracked files to .gitignore, but double-check when working with nested directories or glob patterns.

Remote Storage

The local DVC cache is just a staging area. To share data across machines or persist it beyond a single laptop, you configure a remote—a storage backend that mirrors the content-addressable cache. DVC supports many backends natively.

Configuring remotes

# Amazon S3
$ dvc remote add -d myremote s3://my-bucket/dvc-store
$ dvc remote modify myremote region us-east-1

# Google Cloud Storage
$ dvc remote add -d gcs-remote gs://my-bucket/dvc-store
$ dvc remote modify gcs-remote credentialpath /path/to/keyfile.json

# Azure Blob Storage
$ dvc remote add -d azure-remote azure://my-container/dvc-store

# SSH (self-hosted)
$ dvc remote add -d ssh-remote ssh://user@host/path/to/dvc-store

# Local directory (testing / air-gapped environments)
$ dvc remote add -d local-remote /mnt/shared/dvc-cache

Push and pull

Once a remote is configured, dvc push uploads cached data and dvc pull downloads it. These operations are content-addressed—only new or changed chunks are transferred.

# Push all tracked data to remote
$ dvc push

# On another machine: clone repo, then pull data
$ git clone https://github.com/team/ml-project.git
$ cd ml-project
$ dvc pull                     # downloads data matching current commit

# Push only a specific file
$ dvc push data/train.parquet.dvc

# Pull data for a specific Git tag
$ git checkout v1.0
$ dvc checkout                 # sync workspace to match .dvc pointers
$ dvc pull                     # fetch from remote if not in local cache

The remote configuration is saved in .dvc/config, which you commit to Git. Credentials can be stored in .dvc/config.local (gitignored) or injected via environment variables in CI.

Tip: Use dvc gc -c to garbage-collect unused data from the cache and remote. Pass --all-branches --all-tags to keep data referenced by any branch or tag, removing only truly orphaned files.

DVC Pipelines

Beyond file tracking, DVC models your ML workflow as a directed acyclic graph (DAG) of stages. Each stage declares its dependencies, commands, and outputs. DVC only re-runs stages whose inputs have changed—giving you incremental, reproducible builds for data processing.

dvc.yaml structure

# dvc.yaml — defines the pipeline DAG
stages:
  prepare:
    cmd: python src/prepare.py --input data/raw --output data/prepared
    deps:
      - src/prepare.py
      - data/raw
    outs:
      - data/prepared

  featurize:
    cmd: python src/featurize.py --input data/prepared --output data/features
    deps:
      - src/featurize.py
      - data/prepared
    outs:
      - data/features

  train:
    cmd: python src/train.py --features data/features --model models/model.pkl
    deps:
      - src/train.py
      - data/features
    params:
      - params.yaml:
          - train.epochs
          - train.lr
          - train.batch_size
    outs:
      - models/model.pkl
    metrics:
      - metrics/train.json:
          cache: false

  evaluate:
    cmd: python src/evaluate.py --model models/model.pkl --test data/features/test
    deps:
      - src/evaluate.py
      - models/model.pkl
      - data/features/test
    metrics:
      - metrics/eval.json:
          cache: false
    plots:
      - metrics/roc.csv:
          x: fpr
          y: tpr

Running the pipeline

# Reproduce the full pipeline (only changed stages re-run)
$ dvc repro

# Reproduce a single stage and its upstream deps
$ dvc repro train

# Visualize the DAG
$ dvc dag
          +----------+
          | prepare  |
          +----------+
               *
               *
          +-----------+
          | featurize |
          +-----------+
               *
               *
          +---------+
          |  train  |
          +---------+
               *
               *
         +----------+
         | evaluate |
         +----------+

# Force re-run everything regardless of cache
$ dvc repro --force

DVC writes a dvc.lock file that records the exact hashes of every dependency, parameter, and output for each stage. This lock file is committed to Git and ensures bit-for-bit reproducibility.

Metrics & Experiments

DVC integrates tightly with experiment tracking. The metrics and params fields in dvc.yaml let you compare runs across Git branches, tags, or DVC’s built-in experiment mechanism.

Defining parameters

# params.yaml
train:
  epochs: 50
  lr: 0.001
  batch_size: 64
  dropout: 0.3

featurize:
  max_features: 200
  ngrams: 2

Comparing metrics

# Show current metrics
$ dvc metrics show
Path               epoch   loss     accuracy   f1
metrics/train.json 50      0.0342   0.9721     0.9685

# Diff metrics between current workspace and a Git ref
$ dvc metrics diff v1.0
Path               Metric     Old      New      Change
metrics/train.json accuracy   0.9580   0.9721   0.0141
metrics/train.json f1         0.9512   0.9685   0.0173

# Diff parameters
$ dvc params diff v1.0
Path        Param              Old    New
params.yaml train.lr           0.01   0.001
params.yaml train.epochs       30     50
params.yaml train.dropout      0.2    0.3

Running experiments

DVC experiments let you queue and compare parameter sweeps without creating Git branches manually. Each experiment is stored as a hidden Git ref.

# Run an experiment with modified params
$ dvc exp run --set-param train.lr=0.0005 --set-param train.epochs=100

# Queue multiple experiments
$ dvc exp run --queue --set-param train.lr=0.01
$ dvc exp run --queue --set-param train.lr=0.005
$ dvc exp run --queue --set-param train.lr=0.001
$ dvc exp run --run-all --jobs 3    # execute queue with 3 parallel workers

# List and compare experiments
$ dvc exp show --sort-by metrics/train.json:accuracy --sort-order desc

$ dvc exp branch exp-abc123 best-model
$ git checkout best-model
Experiment storage: DVC experiments are lightweight Git refs (stored under refs/exps/). They don’t pollute your branch list but are fully recoverable. Use dvc exp gc to clean up old experiments.

Python API

DVC provides a Python API through the dvc.api module, enabling programmatic access to tracked files, parameters, and metrics. This is especially useful inside training scripts, notebooks, and orchestrators like Airflow or Prefect.

Reading tracked files

import dvc.api
import pandas as pd

# Open a DVC-tracked file as a file-like object
with dvc.api.open(
    "data/train.parquet",
    repo="https://github.com/team/ml-project",
    rev="v1.0",
) as f:
    df = pd.read_parquet(f)
    print(f"Loaded {len(df)} rows from v1.0")

# Get the cloud URL for a tracked file (useful for Spark / distributed reads)
url = dvc.api.get_url(
    "data/train.parquet",
    repo="https://github.com/team/ml-project",
    rev="main",
)
print(url)  # s3://my-bucket/dvc-store/a1/b2c3d4e5f6...

Reading parameters

import dvc.api

# Load all params from params.yaml
params = dvc.api.params_show()
print(params)
# {'train': {'epochs': 50, 'lr': 0.001, 'batch_size': 64, 'dropout': 0.3},
#  'featurize': {'max_features': 200, 'ngrams': 2}}

# Load params for a specific stage
train_params = dvc.api.params_show(stages="train")
lr = train_params["train"]["lr"]

# Load params from a specific Git revision
old_params = dvc.api.params_show(rev="v1.0")

# Use inside a training script
def train_model():
    params = dvc.api.params_show(stages="train")
    cfg = params["train"]

    model = build_model(dropout=cfg["dropout"])
    optimizer = Adam(lr=cfg["lr"])

    for epoch in range(cfg["epochs"]):
        train_one_epoch(model, optimizer, batch_size=cfg["batch_size"])

    return model

Reading metrics and artifacts

import json
import dvc.api

# Read metrics from a specific revision
with dvc.api.open("metrics/eval.json", rev="v2.0") as f:
    metrics = json.load(f)
    print(f"v2.0 accuracy: {metrics['accuracy']}")

# Iterate over experiments programmatically
import subprocess, json
result = subprocess.run(
    ["dvc", "exp", "show", "--json"],
    capture_output=True, text=True
)
experiments = json.loads(result.stdout)
for exp_name, exp_data in experiments.items():
    print(f"{exp_name}: {exp_data}")

CI/CD Integration

DVC pipelines fit naturally into CI/CD. A typical pattern is: CI checks out code, pulls data from the remote, runs dvc repro, and compares metrics against the base branch.

# .github/workflows/ml-pipeline.yml
name: ML Pipeline
on: [push, pull_request]

jobs:
  train:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install dvc[s3]
      - run: dvc pull
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      - run: dvc repro
      - run: dvc push
      - name: Compare metrics
        run: |
          dvc metrics diff --md main > metrics-report.md
          cat metrics-report.md >> $GITHUB_STEP_SUMMARY
CML integration: Combine DVC with CML (Continuous Machine Learning) to post metric diffs, plots, and model cards as pull-request comments automatically.

For large datasets, use dvc pull --run-cache to skip re-execution of stages whose outputs already exist in the remote cache. This dramatically reduces CI time for pipelines with expensive preprocessing steps.

Best Practices

After operating DVC across dozens of ML projects, the following patterns consistently reduce friction and improve reproducibility.

1. Separate raw and processed data

Track raw ingestion outputs with dvc add and derived datasets as pipeline outputs in dvc.yaml. This ensures raw data is immutable while processed data is always reproducible from the pipeline.

# Raw data: tracked directly (immutable snapshots)
$ dvc add data/raw/events_2025_07.parquet

# Processed data: produced by pipeline stages (reproducible)
# Defined in dvc.yaml as stage outputs — never dvc add these

2. Use params.yaml as single source of truth

Avoid hardcoding hyperparameters in scripts. Load them via dvc.api.params_show() so that DVC can detect changes and invalidate the correct stages.

3. Pin DVC version in requirements

# requirements.txt
dvc[s3]==3.55.2
dvc-s3==3.2.0

4. Structure your project consistently

# Recommended layout
├── data/
│   ├── raw/              # dvc add (immutable)
│   ├── prepared/         # pipeline output
│   └── features/         # pipeline output
├── models/               # pipeline output
├── metrics/              # pipeline metrics (cache: false)
├── src/
│   ├── prepare.py
│   ├── featurize.py
│   ├── train.py
│   └── evaluate.py
├── params.yaml
├── dvc.yaml
├── dvc.lock
└── .dvc/
    └── config

5. Tag meaningful data versions

# After a major data update, tag both Git and push DVC
$ dvc add data/raw/
$ git add data/raw.dvc
$ git commit -m "Data v2.0: added Q2 2025 annotations"
$ git tag data-v2.0
$ dvc push
$ git push --tags

6. Garbage-collect regularly

# Remove cache entries not referenced by any branch or tag
$ dvc gc -c --all-branches --all-tags

# Clean up old experiments
$ dvc exp gc --workspace

✅ Do

  • Commit dvc.lock with every pipeline run
  • Use --set-param for experiment sweeps
  • Set cache: false on small metric files
  • Automate dvc repro in CI
  • Tag releases with git tag + dvc push

❌ Don’t

  • Commit large data files directly to Git
  • Edit dvc.lock by hand
  • Store credentials in .dvc/config (use config.local)
  • Skip dvc push—your teammates need the data
  • Use dvc add on pipeline outputs

Data versioning is a force multiplier for ML teams. DVC bridges the gap between Git’s code versioning and the reality that models depend on data just as much as code. By adopting content-addressed storage, declarative pipelines, and lightweight experiments, you get reproducibility, auditability, and collaboration without sacrificing velocity.
