Synthetic Data Generation for ML

MLOps Series · July 2025 · 28 min read · Data Engineering

Real-world ML projects are perpetually data-hungry. Labels are expensive, edge cases are rare, and privacy regulations like GDPR and HIPAA restrict how production data can be used for model development. Synthetic data generation offers a path forward: algorithmically producing datasets that preserve the statistical properties of real data without exposing any individual record. Gartner predicts that by 2030, synthetic data will completely overshadow real data in AI models. This post dives into the three dominant paradigms—rule-based, GAN-based, and VAE-based generation—together with privacy guarantees and rigorous evaluation metrics.

Why Synthetic Data?

Synthetic data is not a workaround; for many teams it is the primary data strategy. The motivations span four broad categories:

  • Privacy compliance — Share datasets across teams, geographies, and third-party vendors without risking PII leakage. Synthetic records have no 1-to-1 mapping to real individuals.
  • Data scarcity — Rare events (fraud, manufacturing defects, medical conditions) may constitute <0.1% of production data. Synthetic oversampling corrects class imbalance without naive duplication.
  • Cost reduction — A single labelled medical image can cost $50–$100. Generating realistic synthetic variants reduces annotation budgets by an order of magnitude.
  • Faster iteration — Teams can prototype models on synthetic data days before a real dataset clears legal review.
  • Fairness testing — Controlled generation lets you stress-test models across demographic subgroups that are underrepresented in organic data.

Real Data Challenges

  • Collection requires user consent & legal review
  • Labelling is expensive and error-prone
  • Long-tail events severely underrepresented
  • Cannot share across regulatory boundaries
  • Data drift requires continuous re-collection

Synthetic Data Advantages

  • Generated on demand with exact specifications
  • Labels are free—they come from the generation process
  • Rare classes can be oversampled arbitrarily
  • No PII means frictionless data sharing
  • Distribution shifts can be simulated proactively
Key insight: Synthetic data does not replace real data—it augments it. The strongest results come from mixing real and synthetic samples during training, using real data exclusively for final validation.

Rule-Based Generation

The simplest and most controllable approach encodes domain knowledge directly into generator functions. Libraries like Faker provide locale-aware primitives for names, addresses, dates, and financial data. For domain-specific schemas, you build custom generators that enforce business rules—value ranges, inter-column dependencies, and temporal ordering.

Faker for Tabular Data

from faker import Faker
import pandas as pd
import random

fake = Faker()
Faker.seed(42)
random.seed(42)

def generate_customer_records(n: int) -> pd.DataFrame:
    """Generate n synthetic customer records with realistic constraints."""
    records = []
    for _ in range(n):
        age = random.randint(18, 85)
        # Income correlates with age (peaks at 45-55)
        base_income = 25000 + 1500 * min(age, 55) - 800 * max(0, age - 55)
        income = max(15000, base_income + random.gauss(0, 8000))

        records.append({
            "customer_id": fake.uuid4(),
            "name": fake.name(),
            "email": fake.email(),
            "age": age,
            "annual_income": round(income, 2),
            "signup_date": fake.date_between(
                start_date="-5y", end_date="today"
            ),
            "is_churned": random.random() < (0.05 + 0.01 * max(0, age - 60)),
        })

    return pd.DataFrame(records)

df = generate_customer_records(10_000)
print(df.describe())

Custom Generators with Inter-Column Dependencies

Real datasets have rich correlations. A rule-based generator must encode these explicitly. Below, we generate healthcare claims where diagnosis codes determine procedure codes and cost ranges:

import numpy as np
from datetime import datetime, timedelta

# Domain knowledge encoded as lookup tables
DIAGNOSIS_PROFILES = {
    "E11": {  # Type 2 diabetes
        "procedures": ["99213", "99214", "82947", "83036"],
        "cost_range": (150, 1200),
        "avg_visits_year": 6,
    },
    "I10": {  # Essential hypertension
        "procedures": ["99213", "93000", "80053"],
        "cost_range": (100, 800),
        "avg_visits_year": 4,
    },
    "J06": {  # Acute upper respiratory infection
        "procedures": ["99212", "87880"],
        "cost_range": (75, 350),
        "avg_visits_year": 2,
    },
}

def generate_claims(n_patients: int, year: int) -> list[dict]:
    claims = []
    for pid in range(n_patients):
        # Each patient gets 1-3 diagnoses
        dx_codes = np.random.choice(
            list(DIAGNOSIS_PROFILES.keys()),
            size=np.random.randint(1, 4), replace=False
        )
        for dx in dx_codes:
            profile = DIAGNOSIS_PROFILES[dx]
            n_visits = np.random.poisson(profile["avg_visits_year"])
            for _ in range(max(1, n_visits)):
                lo, hi = profile["cost_range"]
                claims.append({
                    "patient_id": f"P{pid:06d}",
                    "diagnosis": dx,
                    "procedure": np.random.choice(profile["procedures"]),
                    "cost": round(np.random.uniform(lo, hi), 2),
                    "date": datetime(year, 1, 1) + timedelta(
                        days=np.random.randint(0, 365)
                    ),
                })
    return claims
Limitation: Rule-based generators cannot capture correlations you haven’t explicitly programmed. As schema complexity grows, the maintenance burden becomes unsustainable. For datasets with 50+ columns and non-obvious joint distributions, consider learning-based methods.

GAN-Based Synthesis

Generative Adversarial Networks learn the joint distribution of a real dataset through an adversarial game between a generator G and a discriminator D. The generator produces synthetic rows; the discriminator tries to distinguish real from fake. At equilibrium, G produces samples indistinguishable from the training data.

Architecture: CTGAN

CTGAN (Conditional Tabular GAN) adapts the GAN framework specifically for tabular data with mixed types. It addresses three challenges that make vanilla GANs fail on tables:

  • Mode-specific normalization — Continuous columns are modelled as mixtures of Gaussians via a variational Gaussian mixture model (VGM). Each value is encoded as a one-hot mode indicator plus a normalized scalar within that mode.
  • Conditional generation — A training-by-sampling strategy ensures the generator sees each category of each discrete column evenly, preventing mode collapse on imbalanced categorical features.
  • Fully-connected architecture — Both G and D use residual layers with batch normalization (G) and dropout + leaky ReLU (D).
# CTGAN with the SDV (Synthetic Data Vault) library
from sdv.single_table import CTGANSynthesizer
from sdv.metadata import SingleTableMetadata
import pandas as pd

# Load real data
real_df = pd.read_csv("transactions.csv")

# Infer metadata (column types, ranges, formats)
metadata = SingleTableMetadata()
metadata.detect_from_dataframe(real_df)
metadata.update_column(
    column_name="amount",
    sdtype="numerical",
    computer_representation="Float",
)
metadata.update_column(
    column_name="merchant_category",
    sdtype="categorical",
)

# Configure and train the CTGAN synthesizer
synthesizer = CTGANSynthesizer(
    metadata,
    epochs=500,
    batch_size=500,
    generator_dim=(256, 256),
    discriminator_dim=(256, 256),
    generator_lr=2e-4,
    discriminator_lr=2e-4,
    discriminator_steps=1,
    verbose=True,
)

synthesizer.fit(real_df)

# Generate synthetic samples
synthetic_df = synthesizer.sample(num_rows=50_000)

# Conditional sampling: generate only high-value transactions
from sdv.sampling import Condition
high_value = Condition(
    num_rows=5_000,
    column_values={"merchant_category": "luxury"}
)
conditional_df = synthesizer.sample_from_conditions([high_value])

print(f"Real shape: {real_df.shape}")
print(f"Synthetic shape: {synthetic_df.shape}")

Training Dynamics and Pitfalls

GAN training is notoriously unstable for tabular data. Common failure modes include:

  • Mode collapse — The generator produces only a few distinct row patterns. Monitor the number of unique values per column in synthetic vs. real data.
  • Training divergence — The discriminator loss collapses to zero while generator loss explodes. Use gradient penalty (WGAN-GP) to stabilize.
  • Categorical leakage — The generator memorizes rare categorical combinations verbatim. Apply minimum frequency thresholds during preprocessing.

VAE-Based Approaches

Variational Autoencoders offer an alternative generative paradigm. A VAE learns a smooth, continuous latent space by encoding each input row into a distribution (mean μ and variance σ²) rather than a point. The decoder reconstructs rows from samples drawn from this latent distribution. The loss function balances reconstruction accuracy against KL divergence, which keeps the latent space regularized.

Why VAEs for Tabular Data?

  • Stable training — No adversarial game means no mode collapse or training divergence. The ELBO loss provides a single, well-behaved objective.
  • Interpretable latent space — Interpolation between latent vectors produces smooth transitions between record types—useful for generating boundary cases.
  • Density estimation — The encoder gives an approximate likelihood for each row, enabling outlier detection on the same model used for generation.
# TVAE (Tabular VAE) via SDV
from sdv.single_table import TVAESynthesizer

tvae = TVAESynthesizer(
    metadata,
    epochs=300,
    batch_size=500,
    embedding_dim=128,
    compress_dims=(128, 128),
    decompress_dims=(128, 128),
    l2scale=1e-5,
    loss_factor=2,      # weight of reconstruction vs KL
)

tvae.fit(real_df)
synthetic_vae = tvae.sample(num_rows=50_000)
print(synthetic_vae.head())

Custom VAE with PyTorch

For fine-grained control over architecture and loss terms, implement the VAE directly. The key design choices for tabular data are: (1) separate encoding heads for continuous vs. categorical columns, (2) Gumbel-Softmax for differentiable categorical sampling, and (3) beta-annealing to prevent posterior collapse.

import torch
import torch.nn as nn
import torch.nn.functional as F

class TabularVAE(nn.Module):
    def __init__(self, input_dim, latent_dim=64, hidden=256):
        super().__init__()
        # Encoder
        self.enc = nn.Sequential(
            nn.Linear(input_dim, hidden), nn.ReLU(), nn.BatchNorm1d(hidden),
            nn.Linear(hidden, hidden),    nn.ReLU(), nn.BatchNorm1d(hidden),
        )
        self.mu_head = nn.Linear(hidden, latent_dim)
        self.logvar_head = nn.Linear(hidden, latent_dim)

        # Decoder
        self.dec = nn.Sequential(
            nn.Linear(latent_dim, hidden), nn.ReLU(), nn.BatchNorm1d(hidden),
            nn.Linear(hidden, hidden),     nn.ReLU(), nn.BatchNorm1d(hidden),
            nn.Linear(hidden, input_dim),  nn.Sigmoid(),
        )

    def encode(self, x):
        h = self.enc(x)
        return self.mu_head(h), self.logvar_head(h)

    def reparameterize(self, mu, logvar):
        std = torch.exp(0.5 * logvar)
        eps = torch.randn_like(std)
        return mu + eps * std

    def decode(self, z):
        return self.dec(z)

    def forward(self, x):
        mu, logvar = self.encode(x)
        z = self.reparameterize(mu, logvar)
        x_recon = self.decode(z)
        return x_recon, mu, logvar

def vae_loss(x_recon, x, mu, logvar, beta=1.0):
    # Reconstruction: binary cross-entropy for normalized inputs
    recon = F.binary_cross_entropy(x_recon, x, reduction="sum")
    # KL divergence: closed-form for Gaussian prior
    kl = -0.5 * torch.sum(1 + logvar - mu.pow(2) - logvar.exp())
    return recon + beta * kl
TVAE vs CTGAN in practice: Benchmarks on the SDV suite show TVAE trains 3–5× faster and produces competitive or superior statistical fidelity on datasets with <50 columns. CTGAN tends to win on high-cardinality categorical columns where mode-specific normalization matters most.

Privacy & Differential Privacy

Generating synthetic data does not automatically guarantee privacy. A powerful generative model can memorize and reproduce training records verbatim. Differential privacy (DP) provides a mathematical framework to bound the influence of any single training record on the generator’s output.

DP Guarantees for Generative Models

A randomized mechanism M satisfies (ε, δ)-differential privacy if for any two datasets D and D′ differing by one record, and any set of outputs S:

Pr[M(D) ∈ S] ≤ eε · Pr[M(D′) ∈ S] + δ

Lower ε means stronger privacy. For generative models, DP is enforced by adding calibrated Gaussian noise to gradients during training (DP-SGD). The Opacus library integrates with PyTorch to provide per-sample gradient clipping and noise injection.

# DP-SGD training with Opacus
from opacus import PrivacyEngine
import torch

model = TabularVAE(input_dim=30, latent_dim=64)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
dataloader = get_dataloader(real_data, batch_size=256)

# Wrap model, optimizer, and dataloader with Opacus
privacy_engine = PrivacyEngine()
model, optimizer, dataloader = privacy_engine.make_private_with_epsilon(
    module=model,
    optimizer=optimizer,
    data_loader=dataloader,
    epochs=50,
    target_epsilon=3.0,     # privacy budget
    target_delta=1e-5,      # should be < 1/n
    max_grad_norm=1.0,      # per-sample gradient clip
)

# Training loop — Opacus handles noise injection automatically
for epoch in range(50):
    for batch in dataloader:
        optimizer.zero_grad()
        x_recon, mu, logvar = model(batch)
        loss = vae_loss(x_recon, batch, mu, logvar)
        loss.backward()
        optimizer.step()

    epsilon = privacy_engine.get_epsilon(delta=1e-5)
    print(f"Epoch {epoch}: loss={loss.item():.4f}, ε={epsilon:.2f}")

Privacy-Utility Trade-off

Stronger privacy (ε < 1) requires more noise, degrading the statistical fidelity of synthetic outputs. The practical strategy is:

  • ε ≤ 1 — Strong privacy, noticeable utility loss. Use for public dataset releases and regulatory submissions.
  • 1 < ε ≤ 10 — Moderate privacy. Suitable for internal analytics and feature engineering where re-identification risk is lower.
  • ε > 10 — Weak privacy but near-original utility. Appropriate for development and testing environments with access controls.
Memorization risk: Even without DP, always run a membership inference attack (MIA) test before releasing synthetic data. If a classifier can distinguish “was this record in the training set?” above chance, the generator has memorized too much. The sdmetrics library includes a NewRowSynthesis metric that checks for exact and near-duplicate matches.

Evaluation Metrics

Synthetic data quality is multi-dimensional. No single metric suffices. A rigorous evaluation covers three axes: statistical similarity (does the synthetic data look like the real data?), downstream utility (does a model trained on synthetic data perform well on real test data?), and privacy (can we link synthetic records back to real individuals?).

Statistical Similarity

  • Column-wise distributions — KS test (continuous) and chi-squared test (categorical) per column. Report the average p-value across all columns.
  • Pairwise correlations — Compare the correlation matrix of real vs. synthetic data using Frobenius norm or column-wise Pearson difference.
  • Joint distribution — Train a classifier to distinguish real from synthetic. The closer the ROC-AUC is to 0.5, the better—the model cannot tell them apart.
from sdmetrics.single_table import QualityReport

# Run the full SDMetrics quality report
report = QualityReport()
report.generate(real_df, synthetic_df, metadata.to_dict())

# Overall quality score (0 to 1)
print(f"Overall quality: {report.get_score():.3f}")

# Column-level shape similarity
column_shapes = report.get_details(property_name="Column Shapes")
print(column_shapes.sort_values("Score").head(10))

# Pairwise correlation similarity
column_pairs = report.get_details(property_name="Column Pair Trends")
print(column_pairs.sort_values("Score").head(10))

Downstream Utility (Train-on-Synthetic, Test-on-Real)

from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import train_test_split
from sklearn.metrics import f1_score

# Split real data into train/test
real_train, real_test = train_test_split(real_df, test_size=0.2, random_state=42)

# Model A: trained on REAL data (baseline)
model_real = GradientBoostingClassifier(n_estimators=200, random_state=42)

End-to-End Pipeline

A production synthetic data pipeline ties together data profiling, generation, evaluation, and versioning. Below is a complete pipeline that reads real data, trains a CTGAN, evaluates quality and privacy, and only publishes the synthetic dataset if all thresholds pass.

import json
import sys
from pathlib import Path
from sdv.single_table import CTGANSynthesizer
from sdv.metadata import SingleTableMetadata
from sdmetrics.single_table import QualityReport, DiagnosticReport
import pandas as pd

# ── Configuration ──
CONFIG = {
    "real_data_path": "data/real/transactions.parquet",
    "output_dir": "data/synthetic/",
    "num_synthetic_rows": 100_000,
    "ctgan_epochs": 500,
    "quality_threshold": 0.85,
    "privacy_threshold": 0.95,
}

# ── Step 1: Load and profile ──
real_df = pd.read_parquet(CONFIG["real_data_path"])
metadata = SingleTableMetadata()
metadata.detect_from_dataframe(real_df)
print(f"Loaded {len(real_df)} real rows, {len(real_df.columns)} columns")

# ── Step 2: Train generator ──
synth = CTGANSynthesizer(metadata, epochs=CONFIG["ctgan_epochs"])
synth.fit(real_df)

# ── Step 3: Generate ──
synthetic_df = synth.sample(num_rows=CONFIG["num_synthetic_rows"])

# ── Step 4: Evaluate quality ──
quality = QualityReport()
quality.generate(real_df, synthetic_df, metadata.to_dict())
quality_score = quality.get_score()

# ── Step 5: Evaluate privacy ──
diagnostic = DiagnosticReport()
diagnostic.generate(real_df, synthetic_df, metadata.to_dict())
privacy_score = diagnostic.get_score()

# ── Step 6: Gate and publish ──
results = {
    "quality_score": round(quality_score, 4),
    "privacy_score": round(privacy_score, 4),
    "num_real_rows": len(real_df),
    "num_synthetic_rows": len(synthetic_df),
    "passed": quality_score >= CONFIG["quality_threshold"]
              and privacy_score >= CONFIG["privacy_threshold"],
}

output_dir = Path(CONFIG["output_dir"])
output_dir.mkdir(parents=True, exist_ok=True)

if results["passed"]:
    synthetic_df.to_parquet(output_dir / "transactions_synthetic.parquet")
    print("✓ Synthetic data published")
else:
    print("✗ Quality/privacy gates failed — data NOT published")
    sys.exit(1)

# Save evaluation report
with open(output_dir / "eval_report.json", "w") as f:
    json.dump(results, f, indent=2)
print(json.dumps(results, indent=2))

Best Practices

After building synthetic data pipelines across healthcare, fintech, and e-commerce domains, the following guidelines consistently improve results:

1. Start rule-based, graduate to learned models

For schemas with <15 columns and well-understood business logic, rule-based generators are faster to build, easier to debug, and produce perfectly valid records. Switch to CTGAN/TVAE only when column interactions exceed what you can encode manually.

2. Always validate on real holdout data

Never evaluate synthetic data quality against the same real data used for training the generator. Hold out 20% of real data exclusively for evaluation to detect overfitting.

3. Version your generators alongside data

Serialize trained synthesizer objects and commit them with DVC. This ensures reproducibility: any team member can regenerate the exact synthetic dataset from a given model checkpoint.

4. Monitor distribution drift

When real data evolves, retrain generators periodically. Compare column distributions between consecutive synthetic releases to catch model staleness.

5. Layer privacy defences

Use differential privacy during training and run post-hoc privacy audits (membership inference, nearest-neighbor distance) before every release. Defence in depth is essential when dealing with sensitive data.

✅ Do

  • Profile real data thoroughly before choosing a generator
  • Use conditional sampling to balance rare classes
  • Run statistical and utility evaluations on every batch
  • Enforce privacy gates in CI/CD before publishing
  • Document generation parameters in metadata

❌ Don’t

  • Assume synthetic data is automatically private
  • Train generators on data you haven’t cleaned
  • Use synthetic data for final model validation
  • Ignore column correlations in rule-based generators
  • Ship synthetic datasets without a quality report

Synthetic data generation has matured from a research curiosity to a production-grade capability. By combining rule-based generators for controlled scenarios, GANs and VAEs for learning complex distributions, differential privacy for formal guarantees, and rigorous evaluation metrics, ML teams can break through data bottlenecks while maintaining the trust of regulators and users alike.
