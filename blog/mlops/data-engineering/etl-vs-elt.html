ETL vs ELT for ML Workloads

MLOps Series Data Engineering

How you move and transform data has a profound effect on the velocity, cost, and reliability of your ML system. The Extract-Transform-Load (ETL) and Extract-Load-Transform (ELT) paradigms each carry distinct trade-offs—especially when your downstream consumer is a training pipeline rather than a BI dashboard. This post breaks down both patterns, maps them to ML-specific requirements, and shows how the modern data stack reshapes the decision.

The Traditional ETL Pattern

In ETL, data is extracted from source systems, transformed in an intermediate processing layer (often an ETL server or Spark cluster), and then loaded into a target warehouse or data store. The transformation happens before the data lands in the destination.

This paradigm dates back to the era when storage was expensive and compute was centralized. It made sense to cleanse, denormalize, and aggregate data before writing it to disk. For decades, ETL was the default for enterprise data warehousing—tools like Informatica, Talend, and SSIS dominated the space.

Key characteristics of ETL:

  • Schema-on-write — data must conform to the target schema before loading
  • Transformation is coupled to ingestion — changes to business logic require modifying the pipeline itself
  • Raw data is often discarded — only the transformed version persists in the warehouse
  • Lower storage costs — only clean, aggregated data occupies warehouse space
  • Higher pipeline complexity — orchestration must manage extract, transform, and load as a single unit
# Classic ETL pipeline (Spark-based)
class TraditionalETL:
    def extract(self, sources: List[DataSource]) -> List[DataFrame]:
        raw_frames = []
        for src in sources:
            df = spark.read.format(src.format).load(src.path)
            raw_frames.append(df)
        return raw_frames

    def transform(self, frames: List[DataFrame]) -> DataFrame:
        # All business logic lives here — before loading
        joined = frames[0].join(frames[1], on="user_id")
        cleaned = joined.filter(col("age") > 0)
        enriched = cleaned.withColumn(
            "age_bucket", when(col("age") < 25, "young")
                .when(col("age") < 50, "mid")
                .otherwise("senior")
        )
        return enriched

    def load(self, df: DataFrame, target: str):
        df.write.mode("overwrite").parquet(target)

The critical drawback for ML teams: when you discard raw data during transformation, you lose the ability to retroactively engineer new features. If a data scientist discovers that a raw field (e.g., user-agent string, raw click coordinates) is predictive, the ETL pipeline must be rebuilt to preserve that field—and historical data may be unrecoverable.

The Modern ELT Pattern

ELT flips the order: raw data is extracted and loaded directly into a cloud warehouse or lakehouse, and transformations happen inside the destination using its native compute engine. This became viable when cloud warehouses (Snowflake, BigQuery, Redshift) decoupled storage from compute, making it cheap to store raw data and powerful to transform it in-place.

Key characteristics of ELT:

  • Schema-on-read — raw data lands as-is; structure is imposed at query time
  • Transformation is decoupled from ingestion — you can iterate on transformations without re-ingesting
  • Full data lineage preserved — raw source data remains accessible for new features or audits
  • Higher storage costs — raw data persists alongside transformed views
  • Simpler ingestion pipelines — extract and load can be handled by managed connectors
# Modern ELT: dbt transformation inside the warehouse
-- models/staging/stg_user_events.sql
WITH raw_events AS (
    SELECT *
    FROM {{ source('raw', 'user_events') }}
    WHERE _loaded_at > CURRENT_DATE() - INTERVAL '7 days'
),
cleaned AS (
    SELECT
        user_id,
        event_type,
        PARSE_JSON(event_payload) AS payload,
        event_timestamp,
        ROW_NUMBER() OVER(
            PARTITION BY user_id, event_type
            ORDER BY event_timestamp DESC
        ) AS row_num
    FROM raw_events
    WHERE user_id IS NOT NULL
)
SELECT * FROM cleaned WHERE row_num = 1
Why ELT wins for ML: ML feature engineering is inherently iterative. A data scientist might test dozens of feature transformations before settling on a final set. ELT lets you re-transform the same raw data without re-ingesting—turning a multi-day pipeline change into a SQL model edit and a dbt run.

ETL vs ELT for ML Workloads

The core tension for ML is between data freshness, feature iteration speed, and compute cost. Let's compare along the dimensions that matter most for training pipelines.

ETL for ML

  • Feature engineering: Baked into the pipeline; changes require code deploys
  • Raw data access: Typically discarded post-transform
  • Iteration speed: Slow — pipeline rebuild + backfill for new features
  • Compute model: External cluster (Spark, Beam) bears transformation cost
  • Data quality: Enforced at ingestion time — bad data never enters the warehouse

ELT for ML

  • Feature engineering: SQL/dbt models; iterate without touching ingestion
  • Raw data access: Always available for retroactive feature discovery
  • Iteration speed: Fast — edit SQL, run transformation, retrain
  • Compute model: Warehouse compute (pay-per-query or reserved slots)
  • Data quality: Validated post-load via dbt tests, Great Expectations

Where ETL Excels

  • Streaming features: Real-time feature computation from Kafka/Kinesis
  • Complex transformations: Image resizing, audio feature extraction, NLP tokenization
  • Privacy compliance: PII scrubbing before data lands in shared storage
  • Resource-constrained targets: Edge devices, embedded databases

Where ELT Excels

  • Tabular ML: Structured data where SQL can express most feature logic
  • Exploratory analysis: Data scientists need ad-hoc access to raw fields
  • Multi-team environments: Different teams derive different features from the same raw data
  • Audit requirements: Regulations require raw data retention and lineage

A practical heuristic: if your feature engineering can be expressed in SQL, ELT is almost always the better choice. If it requires arbitrary code (Python, C++, GPU compute), you need an ETL or hybrid pattern.

# Feature engineering complexity spectrum
# SQL-expressible (ELT-friendly):
#   aggregations, window functions, joins, CASE expressions,
#   date arithmetic, string parsing, JSON extraction

# Code-required (ETL-necessary):
#   image augmentation, tokenization, embedding generation,
#   custom distance metrics, graph feature computation,
#   signal processing (FFT, spectrograms)

def classify_feature_pipeline(features: List[Feature]) -> str:
    sql_expressible = all(
        f.transform_type in {"agg", "window", "join", "cast", "case"}
        for f in features
    )
    if sql_expressible:
        return "ELT"
    elif any(f.requires_gpu for f in features):
        return "ETL_with_GPU_cluster"
    else:
        return "Hybrid"

The Modern Data Stack

The modern data stack has standardized around a set of tools that make ELT the default for most analytics and ML workloads. Understanding where each tool fits in the pipeline is essential for designing ML data infrastructure.

Ingestion Layer: Managed Connectors

Fivetran and Airbyte handle the "EL" part—extracting from hundreds of sources (databases, SaaS APIs, event streams) and loading raw data into your warehouse. The key difference:

Fivetran

  • Fully managed SaaS — zero infrastructure to maintain
  • 300+ pre-built connectors with automatic schema migration
  • Usage-based pricing (monthly active rows)
  • Best for: teams that prioritize reliability over customization
  • Limitation: custom connectors are limited; closed-source

Airbyte

  • Open-source core with managed cloud option
  • 350+ connectors; easy to build custom ones via CDK
  • Self-hostable — critical for data sovereignty requirements
  • Best for: teams needing custom connectors or on-prem deployment
  • Limitation: self-hosted requires operational investment

Warehouse Layer: Compute Engines

The warehouse is where ELT transformations execute. For ML, the warehouse often serves double duty as both the feature store source and the training data source.

Snowflake

  • Separated storage and compute with virtual warehouses
  • Time travel (up to 90 days) enables reproducible training datasets
  • Snowpark allows Python/Scala UDFs for non-SQL transforms
  • Data sharing for cross-org feature pipelines
  • Credit-based pricing; idle warehouses cost nothing

BigQuery

  • Serverless — no cluster sizing or warehouse management
  • Built-in ML with BigQuery ML (train models in SQL)
  • Slot-based or on-demand pricing; flat-rate for predictable costs
  • Native integration with Vertex AI for training export
  • Excellent for petabyte-scale feature computation

Transformation Layer: dbt

dbt (data build tool) has become the standard for the "T" in ELT. It treats SQL transformations as software—version-controlled, tested, documented, and modular. For ML teams, dbt provides:

  • Modular feature definitions: Each feature is a dbt model that can be independently tested and versioned
  • Data quality tests: Built-in and custom tests (not_null, unique, accepted_values, custom SQL assertions)
  • Documentation as code: Feature descriptions live alongside transformation logic
  • Incremental models: Process only new data for large tables, reducing compute costs by 10–100×
  • Lineage graphs: Visualize the full dependency chain from raw source to training-ready feature
-- dbt model: feature_user_engagement.sql
-- Computes rolling engagement features for ML training
{{ config(
    materialized='incremental',
    unique_key='user_id',
    on_schema_change='sync_all_columns'
) }}

WITH events AS (
    SELECT * FROM {{ ref('stg_user_events') }}
    {% if is_incremental() %}
    WHERE event_timestamp > (SELECT MAX(updated_at) FROM {{ this }})
    {% endif %}
),
features AS (
    SELECT
        user_id,
        COUNT(*) AS total_events_7d,
        COUNT(DISTINCT event_type) AS unique_event_types_7d,
        AVG(DATEDIFF('second', prev_event_ts, event_timestamp))
            AS avg_inter_event_seconds,
        MAX(event_timestamp) AS last_active_at,
        SUM(CASE WHEN event_type = 'purchase' THEN 1 ELSE 0 END)
            AS purchase_count_7d,
        CURRENT_TIMESTAMP() AS updated_at
    FROM events
    GROUP BY user_id
)
SELECT * FROM features
Watch out: dbt's incremental models can introduce subtle training-serving skew. If your incremental logic processes only the last N days but your training pipeline expects a full historical snapshot, the resulting features will differ from what batch training produces. Always validate incremental outputs against full-refresh baselines during development.

Cost & Performance Considerations

The ETL vs ELT decision has direct financial implications. Understanding cost drivers helps you choose the pattern that scales sustainably as data volume and model count grow.

ETL Cost Model

In ETL, you pay for a dedicated compute cluster (Spark on EMR/Dataproc, or a managed ETL service) that runs transformations before loading. The cost equation:

# ETL monthly cost breakdown
etl_cost = (
    cluster_instance_hours * price_per_hour    # Spark/EMR compute
    + source_api_calls * api_cost              # extraction costs
    + warehouse_storage_tb * storage_price     # only transformed data
    + orchestration_cost                       # Airflow/Step Functions
)

# Example: 500 GB daily ingestion, 4-hour Spark job
# EMR (8x r5.2xlarge): ~$18/hr × 4hr × 30 days = $2,160/mo
# S3 storage (transformed): ~$23/TB × 2 TB = $46/mo
# Total compute-heavy, storage-light

ELT Cost Model

In ELT, you pay for warehouse compute (queries/slots) and raw + transformed storage. The cost equation shifts toward storage:

# ELT monthly cost breakdown
elt_cost = (
    warehouse_compute                          # BigQuery on-demand or Snowflake credits
    + ingestion_tool_cost                      # Fivetran/Airbyte subscription
    + storage_tb * storage_price               # raw + staging + transformed
)

# Example: same 500 GB daily, BigQuery on-demand
# Ingestion (Fivetran): ~$1,500/mo for volume tier
# BQ on-demand: ~$5/TB scanned × ~60 TB/mo = $300/mo
# BQ storage: $0.02/GB × 15 TB raw = $300/mo
# Total storage-heavier, no cluster management

ML-Specific Cost Amplifiers

ML workloads introduce cost dynamics that differ from analytics:

  • Feature iteration multiplier: Each experiment may trigger a full feature recomputation. In ETL, this means a new Spark job. In ELT, it's an incremental dbt run—typically 5–20× cheaper per iteration.
  • Training data snapshots: ML requires point-in-time-correct datasets. Snowflake's Time Travel or BigQuery's snapshot decorators provide this natively in ELT; ETL pipelines must build custom snapshotting.
  • Multi-model divergence: Different models may need different feature sets from the same raw data. ELT lets you create multiple transformation paths without re-extracting; ETL often requires separate pipelines.

ETL Cost Profile

  • High fixed compute cost (always-on clusters)
  • Low storage cost (only transformed data)
  • Linear cost scaling with pipeline count
  • Expensive feature iteration (new Spark jobs)
  • Operational overhead: cluster tuning, autoscaling

ELT Cost Profile

  • Variable compute cost (pay-per-query)
  • Higher storage cost (raw data persists)
  • Sub-linear scaling with transformation reuse
  • Cheap feature iteration (SQL model edits)
  • Lower ops overhead: managed warehouse

Hybrid Approaches

In practice, most production ML systems use a hybrid pattern—ELT for structured data and SQL-expressible features, with ETL components for unstructured data and compute-intensive transformations. The key is drawing the boundary clearly.

The Medallion Architecture

Popularized by Databricks, the medallion (Bronze → Silver → Gold) architecture is inherently hybrid. The Bronze-to-Silver transition is ELT (raw to cleaned inside the lakehouse), while Silver-to-Gold may involve ETL-style external compute for ML-specific transformations.

# Medallion architecture for ML
# Bronze: raw ingestion (ELT — Fivetran/Airbyte → warehouse)
# Silver: cleaned + deduplicated (ELT — dbt models)
# Gold:   ML-ready features (hybrid — dbt + Python UDFs)

class MedallionPipeline:
    def bronze_to_silver(self):
        # Pure SQL transformations via dbt
        dbt_run(select="staging")  # dedup, type casting, null handling

    def silver_to_gold_tabular(self):
        # SQL-expressible features stay in dbt
        dbt_run(select="features")  # aggregations, windows, joins

    def silver_to_gold_unstructured(self):
        # Non-SQL transforms use external compute
        images = spark.read.parquet("silver/images")
        embeddings = images.map(generate_clip_embedding)
        embeddings.write.parquet("gold/image_embeddings")

Feature Platform Pattern

Another hybrid approach places a feature platform (Feast, Tecton, or a custom solution) as the boundary between ELT and ETL. The platform consumes both:

  • Batch features from dbt models (ELT path) — materialized on a schedule
  • Streaming features from Flink/Spark Streaming (ETL path) — computed in real-time
  • On-demand features computed at inference time from request context
# Feature platform as the hybrid boundary
from feast import FeatureStore, Entity, FeatureView, Field
from feast.types import Float64, Int64

store = FeatureStore(repo_path="feature_repo/")

# Batch features — sourced from dbt/BigQuery (ELT)
user_engagement = FeatureView(
    name="user_engagement_features",
    entities=[user_entity],
    schema=[
        Field(name="total_events_7d", dtype=Int64),
        Field(name="avg_session_duration", dtype=Float64),
        Field(name="purchase_count_7d", dtype=Int64),
    ],
    source=bigquery_source,  # reads from dbt-materialized table
    ttl=timedelta(days=1),
)

# Streaming features — sourced from Kafka (ETL)
realtime_signals = FeatureView(
    name="realtime_user_signals",
    entities=[user_entity],
    schema=[
        Field(name="clicks_last_5min", dtype=Int64),
        Field(name="cart_value_current", dtype=Float64),
    ],
    source=kafka_source,  # real-time stream processor
    ttl=timedelta(minutes=10),
)
Architecture insight: The feature platform pattern cleanly separates the computation of features (ETL or ELT) from the serving of features (online store for inference, offline store for training). This lets you choose the right data pattern per feature without coupling your serving infrastructure to your transformation choices.

Implementation Patterns

Let's look at concrete implementation patterns for connecting ELT/ETL pipelines to ML training workflows.

Pattern 1: dbt → Warehouse → Training

The simplest ELT-to-ML pattern. dbt models materialize feature tables in the warehouse, and the training pipeline reads directly via a warehouse connector.

# Training pipeline reads from dbt-materialized features
import snowflake.connector
import pandas as pd

def load_training_data(snapshot_date: str) -> pd.DataFrame:
    conn = snowflake.connector.connect(**creds)
    query = f"""
        SELECT f.*, l.label
        FROM gold.user_features f
        JOIN gold.labels l ON f.user_id = l.user_id
        WHERE f.snapshot_date = '{snapshot_date}'
    """
    df = pd.read_sql(query, conn)
    conn.close()
    return df

# Orchestration: Airflow DAG
# Task 1: dbt run --select features  (materializes feature tables)
# Task 2: load_training_data()        (reads from warehouse)
# Task 3: train_model(df)             (trains on the snapshot)
# Task 4: evaluate_and_register()     (validates + pushes to registry)

Pattern 2: Spark ETL → Object Store → Training

For unstructured or compute-heavy features, Spark processes raw data and writes Parquet/TFRecord files to object storage. Training reads directly from the files.

# ETL pipeline for image features
from pyspark.sql import SparkSession
from pyspark.sql.functions import udf

spark = SparkSession.builder.getOrCreate()

# Extract: read raw images from S3
raw_images = spark.read.format("binaryFile").load("s3://raw/images/")

# Transform: resize + extract embeddings (GPU UDF)
@udf(returnType=ArrayType(FloatType()))
def extract_embedding(image_bytes):
    img = decode_and_resize(image_bytes, size=(224, 224))
    return model.encode(img).tolist()

features = raw_images.withColumn(
    "embedding", extract_embedding(col("content"))
)

# Load: write to training-ready format
features.select("path", "embedding").write.parquet(
    "s3://gold/image_embeddings/",
    mode="overwrite",
    partitionBy=["date"]
)

Pattern 3: Orchestrating Hybrid Pipelines

Use Airflow or Dagster to orchestrate both ELT and ETL components within a single DAG, ensuring correct execution order and data dependencies.

# Airflow DAG: hybrid ELT + ETL pipeline
from airflow import DAG
from airflow.operators.bash import BashOperator
from airflow.providers.apache.spark import SparkSubmitOperator

with DAG("ml_feature_pipeline", schedule="@daily") as dag:

    # ELT path: structured features via dbt
    dbt_features = BashOperator(
        task_id="dbt_run_features",
        bash_command="dbt run --select tag:ml_features"
    )

    # ETL path: unstructured features via Spark
    spark_embeddings = SparkSubmitOperator(
        task_id="compute_embeddings",
        application="jobs/image_embeddings.py",
        conf={"spark.executor.instances": "8"}
    )

    # Join point: combine and validate all features
    validate = BashOperator(
        task_id="validate_features",
        bash_command="python validate_feature_completeness.py"
    )

    # Training: consume combined feature set
    train = BashOperator(
        task_id="train_model",
        bash_command="python train.py --features-date={{ ds }}"
    )

    [dbt_features, spark_embeddings] >> validate >> train

Decision Framework

Use this framework to decide between ETL, ELT, or hybrid for your ML workload. Start from your data characteristics and follow the decision path.

Choose Pure ELT When

  • Data is primarily structured (tabular, JSON, CSV)
  • Feature engineering is expressible in SQL
  • Team has strong SQL skills but limited Spark/infrastructure experience
  • Iteration speed is the top priority
  • Budget favors pay-per-query over dedicated clusters
  • Multiple teams need different views of the same data

Choose Pure ETL When

  • Data is unstructured (images, audio, video, point clouds)
  • Transformations require GPU compute or custom libraries
  • Real-time feature computation from streaming sources
  • Strict PII/compliance requires scrubbing before storage
  • Target system has limited compute (edge, embedded)
  • Pipeline is stable and rarely changes
Rule of thumb: Start with ELT for all new ML projects. The ability to iterate on features without re-ingesting data pays for the extra storage cost within weeks. Add ETL components only when you hit a concrete limitation—unstructured data, real-time requirements, or transformations that cannot be expressed in SQL. 
# Decision matrix scorer
def recommend_pattern(workload: MLWorkload) -> str:
    score = 0
    # Factors favoring ELT (positive score)
    if workload.data_type == "structured":        score += 2
    if workload.feature_sql_expressible:          score += 3
    if workload.iteration_frequency == "daily":   score += 2
    if workload.team_sql_proficiency > 0.7:       score += 1
    if workload.multi_team_access:                score += 2

    # Factors favoring ETL (negative score)
    if workload.data_type == "unstructured":      score -= 3
    if workload.requires_gpu_transform:           score -= 3
    if workload.latency_requirement_ms < 100:     score -= 2
    if workload.pii_scrub_before_storage:         score -= 2
    if workload.pipeline_change_frequency == "rare": score -= 1

    if score >= 4:    return "Pure ELT"
    elif score <= -4: return "Pure ETL"
    else:             return "Hybrid (ELT base + ETL extensions)"

The boundary between ETL and ELT continues to blur as warehouses gain Python UDF support (Snowpark, BigQuery remote functions) and lakehouse formats (Iceberg, Delta) bring warehouse-like capabilities to object storage. The trajectory is clear: push computation to where the data lives, and bring code to data rather than data to code. For ML teams, this means ELT as the foundation, with surgical ETL components for what SQL cannot express.
