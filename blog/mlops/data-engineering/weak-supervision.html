Weak Supervision & Programmatic Labeling (Snorkel)

MLOps Series Data Engineering

The Data Programming Paradigm

Supervised machine learning is bottlenecked not by model capacity but by labeled data. Hand-labeling is expensive, slow, and brittle — it encodes domain knowledge implicitly inside opaque annotation guidelines that are difficult to version, audit, or reuse. Data programming flips the script: instead of labeling individual data points, engineers write labeling functions (LFs) — small, noisy heuristic programs — and a generative model learns to combine them.

Formally, given an unlabeled dataset X = {x₁, …, xₙ} and a set of m labeling functions λ₁, …, λₘ, each function produces a label matrix Λ ∈ {−1, 0, 1}ⁿˣᵐ where 0 indicates abstain. The data programming pipeline then estimates the accuracies of each LF and their pairwise correlations without any ground-truth labels, producing probabilistic training labels that a downstream discriminative model can learn from.

This paradigm delivers three critical advantages over hand-labeling:

  • Speed: Domain experts express knowledge as code in hours, not weeks of annotation.
  • Auditability: Labeling logic lives in version-controlled functions, not spreadsheets.
  • Adaptability: When the task drifts, you update a function — not re-label 50k examples.
Key insight: Data programming treats labeling as a modeling problem, not an annotation task. The supervision signal comes from the structure and agreement patterns among noisy sources, not from any single authoritative label.

Snorkel Labeling Functions

A labeling function is any Python callable that accepts a data point and returns a label or ABSTAIN. Snorkel provides the @labeling_function() decorator to register them. LFs can encode keyword heuristics, pattern matches, distant supervision from knowledge bases, pre-trained model outputs, or even crowd-worker votes. 

# Define label constants
from snorkel.labeling import labeling_function, LFAnalysis
from snorkel.labeling import PandasLFApplier

SPAM = 1
HAM  = 0
ABSTAIN = -1

# Heuristic LF — keyword-based
@labeling_function()
def lf_contains_free(x):
    return SPAM if "free" in x.text.lower() else ABSTAIN

# Pattern LF — regex-based
import re
@labeling_function()
def lf_regex_url(x):
    return SPAM if re.search(r"https?://\S+", x.text) else ABSTAIN

# Distant supervision LF — external knowledge base
@labeling_function()
def lf_known_spammer(x):
    return SPAM if x.sender in KNOWN_SPAM_DOMAINS else ABSTAIN

# Pre-trained model LF — transfer learning
@labeling_function()
def lf_sentiment_model(x):
    score = pretrained_clf.predict_proba([x.text])[0][1]
    if score > 0.85:
        return SPAM
    elif score < 0.15:
        return HAM
    return ABSTAIN

The critical design principle: each LF should trade off coverage for precision. An LF that fires on every example but is only 55% accurate adds noise. An LF that fires on 5% of examples but is 95% accurate provides clean signal. The label model handles the rest.

LF Design Taxonomy

🔑 Keyword / Pattern

Simple string or regex matches. Fast to write, high precision on narrow slices. Example: flagging emails containing "click here now" as spam.

Coverage: Low–Medium
Precision: High

📚 Distant Supervision

Leverages external knowledge bases (Wikidata, curated lists, ontologies). Maps KB entries to data points. Example: matching entity mentions against a medical terminology database.

Coverage: Medium
Precision: Medium–High

🤖 Pre-trained Models

Wraps a noisy classifier (zero-shot, few-shot, or a legacy model) as an LF. Thresholding the confidence score lets you control the precision/coverage trade-off.

Coverage: High
Precision: Medium

👥 Crowd Workers

Each annotator becomes a labeling function. The label model estimates per-annotator accuracy and correlations, automatically down-weighting unreliable annotators.

Coverage: High
Precision: Variable

LF Analysis & Diagnostics

Before training the label model, you must profile your LFs. Snorkel's LFAnalysis computes per-LF statistics — polarity, coverage, overlap, and conflict rates — on the label matrix. 

# Apply LFs to the dataset
lfs = [lf_contains_free, lf_regex_url, lf_known_spammer, lf_sentiment_model]
applier = PandasLFApplier(lfs=lfs)
L_train = applier.apply(df=df_train)

# Analyze coverage, overlap, and conflicts
analysis = LFAnalysis(L=L_train, lfs=lfs).lf_summary()
print(analysis)
#              Polarity  Coverage  Overlaps  Conflicts
# lf_contains_free  [1]     0.18      0.12       0.04
# lf_regex_url      [1]     0.09      0.06       0.01
# lf_known_spammer  [1]     0.22      0.14       0.03
# lf_sentiment_model [0,1]  0.61      0.38       0.09

Coverage measures the fraction of data points an LF labels (doesn't abstain on). Overlap counts how often two or more LFs label the same point — essential for the label model to estimate correlations. Conflicts indicate disagreements: the label model needs some conflict to learn which LFs to trust.

The Label Model

The label model is the mathematical core of data programming. It takes the label matrix Λ and learns a generative model over the LFs' outputs — estimating each LF's accuracy, propensity to abstain, and pairwise correlations — all without ground-truth labels.

Generative Model Formulation

The label model assumes that each labeling function λⱼ votes conditionally on the true (latent) label Y. Let μⱼ = P(λⱼ = Y) be the accuracy of LF j. The model factorizes the joint distribution:

P(Λ, Y) = P(Y) ∏ⱼ P(λⱼ | Y)   (independent LF assumption)

When LFs are correlated — which they almost always are in practice — the model extends to a factor graph that includes pairwise dependency factors between correlated LFs:

P(Λ, Y) ∝ exp( ∑ⱼ θⱼ · 𝟙[λⱼ = Y] + ∑_{j,k} θⱼₖ · 𝟙[λⱼ = λₖ] )

The parameters θ are learned by maximizing the marginal log-likelihood of the observed label matrix (marginalizing out the unobserved true label Y). Snorkel uses matrix completion and a triplet method based on observable moments of the label matrix to solve this efficiently, avoiding expensive EM iterations. 

from snorkel.labeling.model import LabelModel

# Fit the label model on the label matrix
label_model = LabelModel(cardinality=2, verbose=True)
label_model.fit(
    L_train=L_train,
    n_epochs=500,
    lr=0.001,
    log_freq=100,
    seed=42
)

# Produce probabilistic labels
probs_train = label_model.predict_proba(L=L_train)
# probs_train.shape → (n, 2)  — soft labels per class

# Hard labels with tie-breaking policy
preds_train = label_model.predict(L=L_train, tie_break_policy="abstain")
Common pitfall: If all your LFs have the same polarity (e.g., they only vote SPAM, never HAM), the label model cannot estimate class balance or LF accuracies. You need at least some LFs voting for each class — or use Snorkel's class_balance parameter to inject a prior.

Why Not Just Majority Vote?

Majority voting treats all LFs equally and ignores correlations. If three LFs are derived from the same underlying signal (e.g., all check for URL patterns), majority vote triple-counts that evidence. The label model learns these correlations and down-weights redundant LFs accordingly. In benchmarks, the label model consistently outperforms majority vote by 5–15 F1 points, with the gap widening as LF correlation increases.

Noise-Aware Learning

The label model outputs probabilistic (soft) labels — not hard 0/1 assignments. A noise-aware discriminative model is trained on these soft labels using a modified loss function that accounts for label uncertainty.

Soft-Label Cross-Entropy

Instead of the standard cross-entropy L = −log P(y | x), the noise-aware loss uses the full probability distribution from the label model:

L = − ∑_k P̃(Y=k | Λ) · log P_θ(Y=k | x)

where P̃(Y=k | Λ) is the label model's posterior probability for class k and P_θ(Y=k | x) is the end model's prediction. This is equivalent to using the label model's posteriors as soft targets — a form of knowledge distillation from the generative label model to the discriminative end model. 

import torch
import torch.nn as nn

class NoisyLabelLoss(nn.Module):
    """Cross-entropy with soft probabilistic targets."""
    def forward(self, logits, soft_labels):
        # logits: (batch, num_classes)
        # soft_labels: (batch, num_classes) from label model
        log_probs = torch.nn.functional.log_softmax(logits, dim=-1)
        loss = -(soft_labels * log_probs).sum(dim=-1).mean()
        return loss

# Training loop with noise-aware loss
criterion = NoisyLabelLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-4)

for epoch in range(num_epochs):
    for batch_x, batch_soft_y in dataloader:
        logits = model(batch_x)
        loss = criterion(logits, batch_soft_y)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

Why Noise-Aware Training Matters

The end model learns generalizable features from the raw input x — it doesn't have access to the LFs at inference time. This is crucial: the end model can generalize to data points that no LF covers. Empirically, a well-trained end model often outperforms the label model itself because it leverages the full feature space of the input, not just the narrow heuristics encoded in the LFs.

Additionally, by propagating uncertainty through soft labels, the end model learns to be less confident on ambiguous examples — producing better-calibrated probabilities. Filtering out low-confidence label model predictions before training (e.g., |P̃(Y=1) − 0.5| < 0.1) can further improve end model quality.

Combining Multiple Weak Sources

Real-world weak supervision pipelines combine heterogeneous signal sources. The power of the data programming abstraction is that all sources become labeling functions, regardless of their origin. Here we examine strategies for source integration and dependency modeling.

Source Integration Architecture

# Heterogeneous LFs: combine keywords, models, KBs, crowd
lfs = [
    # Keyword heuristics (written by domain expert)
    lf_keyword_urgent,
    lf_keyword_promotion,
    lf_keyword_invoice,

    # Regex patterns
    lf_phone_number_pattern,
    lf_currency_pattern,

    # Distant supervision from external DB
    lf_known_sender_whitelist,
    lf_known_sender_blacklist,

    # Pre-trained model outputs
    lf_bert_zero_shot,
    lf_fasttext_classifier,

    # Crowd worker votes (one LF per worker)
    lf_worker_alice,
    lf_worker_bob,
    lf_worker_charlie,
]

# Apply and build the label matrix
applier = PandasLFApplier(lfs=lfs)
L_train = applier.apply(df=df_train)
# L_train.shape → (n_samples, 12)

Modeling Dependencies

When LFs share underlying signals, treating them as independent inflates their combined vote. Snorkel handles this through dependency-aware label models. You specify a dependency graph indicating which LFs are correlated, and the label model includes pairwise factors in its factor graph. 

# Specify LF dependencies as (index_i, index_j) pairs
# E.g., keyword LFs 0,1,2 share similar signals
deps = [
    (0, 1),  # lf_keyword_urgent & lf_keyword_promotion
    (0, 2),  # lf_keyword_urgent & lf_keyword_invoice
    (7, 8),  # lf_bert_zero_shot & lf_fasttext_classifier
]

label_model = LabelModel(cardinality=2, verbose=True)
label_model.fit(
    L_train=L_train,
    n_epochs=500,
    lr=0.001,
    seed=42,
    deps=deps   # inject dependency structure
)
Tip — Automatic dependency detection: Snorkel also supports structure learning via the DependencySelector, which estimates pairwise dependencies from the label matrix itself. Use this when you have >1000 data points and >5 LFs for reliable estimation.

Handling Multi-Class Scenarios

Data programming extends naturally to multi-class settings. Each LF can return any class label or abstain. The label model generalizes from the binary case by learning a per-class accuracy parameter for each LF, and the factor graph includes multi-class potential functions. Set cardinality=K when constructing the label model.

Practical Snorkel Pipeline

Let's walk through a complete end-to-end Snorkel pipeline for a binary text classification task: classifying customer support tickets as Billing vs. Technical. 

import pandas as pd
from snorkel.labeling import labeling_function, PandasLFApplier, LFAnalysis
from snorkel.labeling.model import LabelModel
from sklearn.model_selection import train_test_split
from sklearn.linear_model import LogisticRegression
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics import classification_report

BILLING   = 0
TECHNICAL = 1
ABSTAIN   = -1

# ── Step 1: Load unlabeled data ─────────────────────
df_train = pd.read_csv("tickets_unlabeled.csv")
df_test  = pd.read_csv("tickets_test_labeled.csv")

# ── Step 2: Define labeling functions ────────────────
@labeling_function()
def lf_refund(x):
    return BILLING if "refund" in x.text.lower() else ABSTAIN

@labeling_function()
def lf_charge(x):
    return BILLING if "charge" in x.text.lower() else ABSTAIN

@labeling_function()
def lf_crash(x):
    return TECHNICAL if "crash" in x.text.lower() else ABSTAIN

@labeling_function()
def lf_error_code(x):
    import re
    return TECHNICAL if re.search(r"error\s?\d{3,}", x.text.lower()) else ABSTAIN

@labeling_function()
def lf_payment_method(x):
    terms = ["credit card", "paypal", "invoice", "billing"]
    return BILLING if any(t in x.text.lower() for t in terms) else ABSTAIN

@labeling_function()
def lf_install_issue(x):
    terms = ["install", "setup", "configure", "dependency"]
    return TECHNICAL if any(t in x.text.lower() for t in terms) else ABSTAIN

# ── Step 3: Apply LFs and analyze ────────────────────
lfs = [lf_refund, lf_charge, lf_crash, lf_error_code,
       lf_payment_method, lf_install_issue]

applier = PandasLFApplier(lfs=lfs)
L_train = applier.apply(df=df_train)
L_test  = applier.apply(df=df_test)

print(LFAnalysis(L=L_train, lfs=lfs).lf_summary())

# ── Step 4: Train label model ────────────────────────
label_model = LabelModel(cardinality=2, verbose=True)
label_model.fit(L_train=L_train, n_epochs=500, lr=0.001, seed=42)

# Evaluate label model against gold test set
lm_acc = label_model.score(
    L=L_test,
    Y=df_test.label.values,
    metrics=["accuracy", "f1"]
)
print(f"Label Model — Acc: {lm_acc['accuracy']:.3f}, F1: {lm_acc['f1']:.3f}")

# ── Step 5: Filter & extract soft labels ────────────
probs_train = label_model.predict_proba(L=L_train)
df_filtered = df_train[L_train.sum(axis=1) != -1 * L_train.shape[1]].reset_index(drop=True)
probs_filtered = probs_train[L_train.sum(axis=1) != -1 * L_train.shape[1]]

# ── Step 6: Train end model (noise-aware) ───────────
vectorizer = TfidfVectorizer(max_features=5000)
X_train = vectorizer.fit_transform(df_filtered.text)
X_test  = vectorizer.transform(df_test.text)

# Use hard labels from soft predictions for sklearn
y_train_soft = probs_filtered.argmax(axis=1)

end_model = LogisticRegression(max_iter=1000)
end_model.fit(X_train, y_train_soft)

print(classification_report(df_test.label, end_model.predict(X_test)))
Production tip: In practice, replace the LogisticRegression end model with a fine-tuned transformer (BERT, DeBERTa) for best results. Use the label model's soft probabilities directly as training targets via the noise-aware cross-entropy loss shown in Section 4. This typically yields an additional 3–8 F1 point improvement over hard-label training.

Advanced Patterns & Pitfalls

Slicing Functions for Error Analysis

Snorkel introduces slicing functions (SFs) — analogous to LFs but for evaluation. An SF identifies a data subset (e.g., "short texts under 10 tokens") so you can monitor per-slice performance and catch regressions on critical subpopulations. 

from snorkel.slicing import slicing_function, PandasSFApplier

@slicing_function()
def sf_short_text(x):
    return len(x.text.split()) < 10

@slicing_function()
def sf_has_attachment(x):
    return x.has_attachment == True

sf_applier = PandasSFApplier([sf_short_text, sf_has_attachment])
S_test = sf_applier.apply(df_test)

Common Pitfalls

⚠️ Correlated LFs Without Dependencies

If you copy-paste LFs with slight variations (e.g., "free" vs. "FREE shipping"), they'll be highly correlated. Without declaring dependencies, the label model overestimates their combined signal. Always check the overlap and conflict columns in LFAnalysis.

⚠️ Single-Polarity LFs

If all LFs only vote for the positive class, the label model degenerates. Ensure at least 2–3 LFs vote for each class. If impossible, set class_balance=[0.5, 0.5] or use a small labeled dev set to calibrate the model.

⚠️ Overfitting the Dev Set

It's tempting to tune LF thresholds against a labeled dev set until metrics peak. This leaks the dev set's label distribution into your LFs, reducing their generalizability. Use the dev set to validate, not to design LFs.

⚠️ Ignoring Abstain Rates

An LF with 95% accuracy but 2% coverage contributes almost nothing. Conversely, a 60%-accurate LF with 80% coverage can dominate your label matrix. Balance the portfolio of LFs across the accuracy-coverage spectrum.

Iterative LF Development Workflow

Building a strong LF set is an iterative process. A recommended workflow:

  1. Start broad: Write 3–5 high-coverage LFs (model-based, distant supervision).
  2. Analyze errors: Run the label model, examine misclassified examples on the dev set.
  3. Add targeted LFs: Write narrow, high-precision LFs to fix the top error modes.
  4. Re-run analysis: Check that new LFs reduce conflicts and improve coverage.
  5. Repeat: Iterate until dev set metrics plateau (typically 10–20 LFs suffice).

Evaluation Strategy

Evaluation in a weak supervision pipeline differs fundamentally from standard supervised learning. You must evaluate three separate components: the LFs, the label model, and the end model.

Three-Layer Evaluation

# Layer 1: LF Quality (no labels needed)
# — Coverage, overlap, conflict from LFAnalysis
print(LFAnalysis(L_train, lfs).lf_summary())

# Layer 2: Label Model Quality (requires small gold set)
lm_metrics = label_model.score(
    L=L_dev,
    Y=y_dev_gold,
    metrics=["accuracy", "f1", "precision", "recall"]
)
print(f"Label Model → {lm_metrics}")

# Layer 3: End Model Quality (same gold set)
end_preds = end_model.predict(X_dev)
print(classification_report(y_dev_gold, end_preds))

# ── Key comparison ──────────────────────────────────
# If end_model >> label_model  → good generalization
# If end_model ≈ label_model   → end model learning LF heuristics only
# If end_model << label_model  → training issue (check features, lr)

The gold standard practice is to invest in a small, high-quality labeled dev/test set (200–500 examples) for evaluation, even though you don't need labeled data for training. This is the "calibration investment" that makes weak supervision workflows trustworthy in production.

Benchmarking insight: Across published Snorkel benchmarks (WRENCH, various NLP tasks), data programming achieves 85–95% of fully-supervised performance using zero hand-labeled training data. The remaining gap typically closes when combining weak supervision with a small amount of active learning on the most uncertain examples — a hybrid strategy gaining traction in production ML systems.

When to Use Weak Supervision

Weak supervision is most effective when: (a) you have abundant unlabeled data, (b) domain experts can express labeling heuristics but cannot label thousands of examples, (c) the task evolves over time requiring frequent relabeling, and (d) you need an auditable, version-controlled labeling pipeline. It is less effective when the classification boundary is too complex for any simple heuristic to approximate — in such cases, consider combining weak supervision with active learning.
