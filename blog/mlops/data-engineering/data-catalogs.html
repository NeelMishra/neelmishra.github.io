Data Catalogs and Discovery

MLOps Series Data Engineering

Why Data Catalogs Matter

Modern organizations generate data across hundreds of systems—data warehouses, object stores, streaming platforms, feature stores, and third-party APIs. Without a centralized catalog, engineers spend an estimated 30–40% of their time simply finding, understanding, and validating datasets before they can use them. A data catalog is the knowledge layer that sits above your storage and compute infrastructure, answering three fundamental questions: What data exists? Where does it live? Can I trust it?

Beyond basic discovery, catalogs solve deeper organizational problems. They enforce data governance by tracking ownership, access policies, and PII classification. They provide institutional memory—when an engineer leaves, their tribal knowledge about table schemas, known data quality issues, and upstream dependencies survives in the catalog's metadata. They also bridge the gap between data producers and consumers, reducing the coordination overhead that slows down analytics and ML teams.

The core metadata that a catalog manages falls into several categories:

  • Technical metadata — schemas, column types, partitioning strategies, storage format (Parquet, ORC, Delta)
  • Operational metadata — freshness timestamps, row counts, job run history, SLA compliance
  • Business metadata — descriptions, tags, glossary terms, ownership, domain classification
  • Social metadata — usage statistics, popular queries, user bookmarks, peer endorsements
  • Lineage metadata — upstream sources, downstream consumers, transformation logic
Key Insight: The most successful catalog deployments treat metadata as a product, not a byproduct. They assign metadata owners, define freshness SLAs for catalog entries, and build feedback loops where consumers can flag stale or incorrect descriptions. Without this cultural investment, even the best catalog tooling becomes a graveyard of outdated entries.

Amundsen Architecture & Usage

Amundsen, originally developed at Lyft and now a Linux Foundation AI & Data project, was one of the first open-source data catalogs purpose-built for data discovery. Its architecture cleanly separates concerns into three microservices plus a frontend application:

Metadata Service is the system of record. It exposes a RESTful API backed by a graph database—Neo4j by default, with Apache Atlas as an alternative. The graph model naturally represents relationships: tables belong to databases, columns belong to tables, users own tables, dashboards reference tables. This makes traversal queries (e.g., "find all tables downstream of this Kafka topic") efficient.

Search Service wraps Elasticsearch and provides full-text search with ranking. Amundsen's ranking algorithm boosts results based on usage frequency—tables that many people query rank higher than obscure staging tables. This "popularity signal" is one of Amundsen's most impactful features for discovery.

Databuilder is the ingestion framework. It uses an Extract-Transform-Load pattern where each extractor connects to a source system (Hive metastore, Postgres information_schema, Airflow task metadata) and produces standardized metadata records:

# Amundsen Databuilder — ingesting Hive table metadata
from databuilder.extractor.hive_table_metadata_extractor import HiveTableMetadataExtractor
from databuilder.loader.file_system_neo4j_csv_loader import FsNeo4jCSVLoader
from databuilder.publisher.neo4j_csv_publisher import Neo4jCsvPublisher
from databuilder.task.task import DefaultTask

extractor = HiveTableMetadataExtractor()
loader    = FsNeo4jCSVLoader()
publisher = Neo4jCsvPublisher()

# Configure the extraction job
job_config = ConfigFactory.from_dict({
    # Hive metastore connection
    'extractor.hive_table_metadata.conn_name': 'hive_metastore',
    'extractor.hive_table_metadata.where_clause_suffix':
        "WHERE schema_name NOT IN ('tmp', 'staging')",
    # Neo4j publish endpoint
    'publisher.neo4j.node_files_directory': '/tmp/amundsen/nodes',
    'publisher.neo4j.relation_files_directory': '/tmp/amundsen/rels',
    'publisher.neo4j.neo4j_endpoint': 'bolt://neo4j:7687',
})

task = DefaultTask(extractor, loader)
job  = DefaultJob(conf=job_config, task=task, publisher=publisher)
job.launch()

Amundsen's power comes from its graph-first approach. Because metadata lives in a property graph, you can answer complex relationship queries that would require expensive JOINs in a relational model. For example, impact analysis—"if I drop this column, which dashboards break?"—becomes a single graph traversal.

Operational Challenge: Neo4j is powerful but operationally heavy. At scale (millions of metadata nodes), you need careful tuning of heap sizes, page cache, and query timeouts. Many teams hit performance walls around 5–10 million nodes and switch to Atlas or a custom store. Plan your graph database capacity early.

DataHub Features & Design

DataHub, originally built at LinkedIn, takes a fundamentally different architectural approach from Amundsen. Instead of a graph database, DataHub is built on an event-driven metadata platform powered by Apache Kafka and a generalized metadata store (GMS). Every metadata change is an event, enabling real-time reactivity, time-travel over metadata history, and seamless integration with streaming architectures.

DataHub's core abstraction is the Metadata Change Event (MCE). When an ingestion source discovers a new table or a user adds a tag, the system emits an MCE onto Kafka. Downstream consumers—search indexers, graph builders, analytics engines—subscribe to these events and update their views independently. This decoupled architecture scales horizontally and allows teams to add custom metadata consumers without modifying the core platform.

# DataHub — emitting a metadata change event via Python SDK
from datahub.emitter.rest_emitter import DatahubRestEmitter
from datahub.metadata.schema_classes import (
    DatasetPropertiesClass, MetadataChangeProposalWrapper
)

emitter = DatahubRestEmitter(gms_server="http://datahub-gms:8080")

# Create a metadata change proposal for a dataset
mcp = MetadataChangeProposalWrapper(
    entityUrn="urn:li:dataset:(urn:li:dataPlatform:snowflake,analytics.user_events,PROD)",
    aspect=DatasetPropertiesClass(
        name="user_events",
        description="Clickstream events from the web app, partitioned daily by event_date",
        customProperties={
            "pii_columns": "user_id,email",
            "refresh_cadence": "hourly",
            "owner_team": "data-platform"
        }
    )
)
emitter.emit(mcp)
emitter.flush()

Key differentiators of DataHub include:

  • Timeline API — query the state of any metadata entity at any point in time, enabling audit trails and debugging data-related incidents
  • Fine-grained access control — role-based policies at the dataset, column, and tag level with LDAP/OIDC integration
  • Domain-driven organization — group datasets into business domains (e.g., "Payments," "User Growth") for clearer ownership
  • Data quality integration — native support for Great Expectations and dbt test results as metadata assertions
  • GraphQL API — a flexible query interface that lets frontend teams build custom catalog UIs without backend changes

DataHub's ingestion framework supports over 50 source connectors—Snowflake, BigQuery, Redshift, Kafka, Airflow, dbt, Spark, Looker, Tableau, and more. Each connector extracts technical metadata, lineage, and usage statistics in a single ingestion run:

# DataHub ingestion recipe — YAML configuration
source:
  type: snowflake
  config:
    account_id: "org-account.us-east-1"
    username: "${SNOWFLAKE_USER}"
    password: "${SNOWFLAKE_PASS}"
    include_table_lineage: true
    include_usage_stats:  true
    include_column_lineage: true
    database_pattern:
      allow: ["ANALYTICS", "ML_FEATURES"]
      deny:  ["SCRATCH_.*"]

sink:
  type: datahub-rest
  config:
    server: "http://datahub-gms:8080"

OpenMetadata

OpenMetadata positions itself as the open standard for metadata, with a schema-first design built on a well-defined JSON Schema specification. Unlike Amundsen's graph-centric model or DataHub's event-driven architecture, OpenMetadata uses a relational store (MySQL or Postgres) as its primary backend, making it operationally simpler for teams already running relational databases.

The platform's defining philosophy is that metadata schemas should be API-first and language-agnostic. Every entity type—Table, Dashboard, Pipeline, ML Model, Topic—has a formal JSON Schema definition. This enables automatic SDK generation, validation, and interoperability across tools. Third-party systems can produce or consume metadata without vendor lock-in.

OpenMetadata includes several features that differentiate it from the earlier generation of catalogs:

  • Native data quality engine — built-in test framework for profiling and validating data without external tools like Great Expectations
  • Collaboration layer — threaded conversations on any metadata entity, task assignments, and announcement banners
  • Glossary management — hierarchical business glossary with term-to-asset linking and approval workflows
  • Automations — event-driven rules that trigger actions (e.g., auto-tag PII columns, notify owners on schema changes)
  • Role-based access control — granular policies with support for teams, roles, and data-level permissions
# OpenMetadata — defining a data quality test via API
POST /api/v1/dataQuality/testCases

{
  "name": "user_events_freshness_check",
  "entityLink": "<#E::table::analytics.user_events>",
  "testDefinition": "tableFreshness",
  "parameterValues": [
    { "name": "maxIntervalInHours", "value": "2" }
  ],
  "testSuite": "analytics_critical_tests"
}

# The test checks if user_events was updated within 2 hours
# Results are stored as metadata and surfaced in the catalog UI
Why OpenMetadata is Gaining Traction: Teams that struggled with Neo4j operations in Amundsen or Kafka complexity in DataHub find OpenMetadata's MySQL/Postgres backend refreshingly simple. The built-in data quality and collaboration features reduce the number of tools in the stack. However, the relational backend may limit the complexity of graph traversal queries compared to Neo4j.

Search & Discovery Workflows

The ultimate test of a data catalog is whether users can find the right dataset in under 60 seconds. Search quality depends on three pillars: indexing strategy, ranking algorithm, and contextual signals. All three major catalogs use Elasticsearch as their search backend, but they differ significantly in how they rank results.

A typical discovery workflow involves:

  1. Keyword search — user types "user events" and gets a ranked list of matching tables, dashboards, and pipelines
  2. Faceted filtering — narrow by database, schema, tag, owner, data platform, or freshness
  3. Entity detail inspection — view schema, sample data, column statistics, and descriptions
  4. Lineage exploration — trace upstream sources and downstream consumers
  5. Social validation — check usage stats, recent queries, and peer endorsements
  6. Bookmark or subscribe — save the dataset and get notified on schema changes
# Search ranking pseudocode — combining relevance signals
function rank_search_results(query, candidates):
    for candidate in candidates:
        # Base relevance from Elasticsearch BM25
        text_score = elasticsearch_bm25(query, candidate.indexed_fields)

        # Popularity boost — log-scaled usage count
        usage_boost = log(1 + candidate.weekly_query_count) * 0.3

        # Freshness boost — penalize stale datasets
        hours_since_update = now() - candidate.last_updated
        freshness_boost = max(0, 1.0 - hours_since_update / 168) * 0.2

        # Quality boost — datasets with descriptions and owners rank higher
        completeness = metadata_completeness_score(candidate)  # 0.0 to 1.0
        quality_boost = completeness * 0.15

        # Personalization — boost datasets from user's domain
        domain_boost = 0.2 if candidate.domain == user.primary_domain else 0

        candidate.final_score = text_score + usage_boost + freshness_boost + quality_boost + domain_boost

    return sorted(candidates, key=lambda c: c.final_score, reverse=True)

Advanced discovery features that distinguish production-grade catalogs:

  • Column-level search — find datasets by column name or column description, not just table name
  • Query-based discovery — surface datasets that frequently appear in SQL queries matching certain patterns
  • Schema similarity — recommend tables with similar column structures to the one you're viewing
  • Automated tagging — ML classifiers that detect PII columns, suggest business glossary terms, and infer data domains
Common Pitfall: Many teams index only table names and descriptions, ignoring column-level metadata. This cripples discovery for analytics engineers who search for specific fields like "customer_ltv" or "event_timestamp." Always index column names, descriptions, and tags alongside table-level metadata.

Data Lineage Tracking

Data lineage answers the question: "Where did this data come from, and where does it go?" It is the connective tissue of a data catalog, enabling impact analysis, root cause debugging, compliance auditing, and trust assessment. Without lineage, a catalog is just a searchable schema browser.

Lineage operates at multiple granularity levels:

Table-Level Lineage

Tracks which tables feed into which other tables. Easiest to capture—often derived from SQL parsing or orchestrator DAGs.

  • Low implementation cost
  • Sufficient for impact analysis
  • Cannot trace individual column flows
  • Captured from Airflow, dbt, Spark jobs

Column-Level Lineage

Maps specific columns through transformations. Requires SQL parsing or instrumented compute engines.

  • Critical for PII tracking and GDPR compliance
  • Enables precise impact analysis
  • Significantly harder to capture accurately
  • SQL dialects and UDFs complicate parsing

Lineage capture mechanisms vary by source system. The three primary approaches are:

1. SQL Parsing — Analyze SQL queries to extract source and target tables and columns. Tools like sqllineage, sqlglot, and DataHub's SQL parser handle ANSI SQL well but struggle with vendor-specific syntax, dynamic SQL, and complex CTEs.

2. Orchestrator Integration — Extract lineage from DAG definitions in Airflow, Prefect, or Dagster. Each task's inputs and outputs define edges in the lineage graph. This captures the pipeline structure but misses transformations happening inside tasks.

3. Compute Engine Instrumentation — Hook into Spark's query plan, dbt's manifest, or BigQuery's audit logs to capture lineage at execution time. This is the most accurate method because it reflects what actually ran, not what was declared.

# Extracting lineage from a dbt manifest.json
import json

def extract_dbt_lineage(manifest_path):
    with open(manifest_path) as f:
        manifest = json.load(f)

    lineage_edges = []
    for node_id, node in manifest["nodes"].items():
        if node["resource_type"] not in ("model", "snapshot"):
            continue

        target = node["relation_name"]       # e.g., analytics.dim_users
        sources = node.get("depends_on", {}).get("nodes", [])

        for src in sources:
            src_node = manifest["nodes"].get(src) or manifest["sources"].get(src)
            if src_node:
                lineage_edges.append({
                    "upstream": src_node["relation_name"],
                    "downstream": target,
                    "transform_type": "dbt_model",
                    "sql": node.get("compiled_sql", "")[:500]
                })

    return lineage_edges

# Result: a list of directed edges forming the lineage DAG
# Each edge carries the transformation SQL for auditability

Lineage visualization is equally important. Production catalogs render lineage as interactive directed graphs where users can click through multiple hops, filter by entity type, and highlight the critical path from a source to a target. DataHub and OpenMetadata both provide column-level lineage views where you can trace a single column through multiple transformation stages.

Integrating Catalogs with ML Workflows

For ML teams, data catalogs serve a specialized purpose beyond general analytics discovery. They become the bridge between feature engineering, experiment tracking, and model governance. The catalog answers ML-specific questions: "Which features were used to train model v2.3?" "Has the training data distribution shifted since last month?" "Which models will break if this upstream table schema changes?"

Key integration patterns for ML workflows:

Feature Store ↔ Catalog Sync: Feature stores like Feast or Tecton maintain feature definitions and serving infrastructure, but they lack the discovery and governance capabilities of a catalog. Bidirectional sync ensures that feature definitions appear in the catalog as searchable entities with lineage back to their source tables:

# Syncing Feast feature definitions into DataHub
from feast import FeatureStore
from datahub.emitter.rest_emitter import DatahubRestEmitter

store = FeatureStore(repo_path="./feature_repo")
emitter = DatahubRestEmitter(gms_server="http://datahub-gms:8080")

for fv in store.list_feature_views():
    # Register each feature view as a dataset in the catalog
    emit_feature_view_metadata(
        emitter=emitter,
        feature_view=fv,
        platform="feast",
        # Link back to the source table for lineage
        source_urn=build_dataset_urn(fv.batch_source),
        tags=["ml-feature", fv.name],
        owner="ml-platform-team"
    )

# Now ML engineers can discover features alongside raw tables
# Lineage shows: raw_table → feature_view → training_dataset → model

Experiment Tracking Integration: Connect MLflow or Weights & Biases to the catalog so that each experiment run records which catalog datasets it consumed. This creates a complete provenance chain from raw data through features to model artifacts:

  • Training datasets are tagged with the experiment IDs that consumed them
  • Model artifacts in the registry link back to their training data lineage
  • Data quality issues surface as alerts on dependent model endpoints

Schema Change Impact on Models: When an upstream table schema changes—a column is renamed, a type changes, or a column is dropped—the catalog's lineage graph can automatically identify all downstream ML pipelines and models that may be affected. This proactive impact analysis prevents silent model failures in production.

ML Governance Pattern: Mature ML organizations use the catalog as the single source of truth for model-to-data lineage. Before any model is promoted to production, an automated check verifies that all training data sources are cataloged, have passing quality tests, and comply with data retention policies. This "catalog-gated deployment" pattern catches data issues before they reach production.

Choosing the Right Catalog

Selecting a data catalog depends on your organization's scale, existing infrastructure, and primary use case. Below is a feature comparison across the three major open-source options:

Amundsen

  • Backend: Neo4j / Atlas + Elasticsearch
  • Strengths: Mature graph model, strong at Lyft-scale discovery, active community
  • Weaknesses: Neo4j ops overhead, limited built-in governance
  • Best for: Teams needing deep graph queries and relationship exploration

DataHub

  • Backend: Kafka + GMS + Elasticsearch + Neo4j
  • Strengths: Event-driven, 50+ connectors, timeline API, fine-grained RBAC
  • Weaknesses: Kafka dependency adds infra complexity
  • Best for: Large orgs with existing Kafka infra and streaming metadata needs

OpenMetadata

  • Backend: MySQL/Postgres + Elasticsearch
  • Strengths: Simple ops, built-in data quality, collaboration features, JSON Schema standard
  • Weaknesses: Younger project, relational backend limits complex graph queries
  • Best for: Teams wanting an all-in-one platform with minimal infra overhead

Decision Criteria

  • Team size <20: OpenMetadata (simplest to operate)
  • Already have Kafka: DataHub (leverage existing infra)
  • Need deep graph queries: Amundsen (Neo4j power)
  • Compliance-heavy: DataHub or OpenMetadata (stronger RBAC)

Regardless of which catalog you choose, success depends on three non-technical factors:

  1. Executive sponsorship — catalog adoption requires organizational buy-in, not just engineering effort
  2. Metadata ownership model — every dataset needs an assigned owner who keeps descriptions and tags current
  3. Integration with existing workflows — the catalog must live where engineers already work (Slack, IDE, notebook, CI/CD)
Final Thought: A data catalog is not a one-time deployment—it is a living system that requires continuous investment. The best catalogs evolve from passive metadata repositories into active knowledge hubs that shape how teams discover, trust, and govern their data. Start with automated ingestion, add lineage, then layer on quality checks and governance. Each layer compounds the value of the one before it.
