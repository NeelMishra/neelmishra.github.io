Data Validation with Great Expectations

MLOps Series Data Engineering

Why Validate Data?

Models are only as reliable as the data that feeds them. In production ML systems, data is the most common source of silent failures — upstream schema changes, missing columns, shifted distributions, duplicated rows, and timezone bugs can all degrade model performance without raising a single exception. By the time someone notices, the damage has already propagated through predictions, dashboards, and business decisions.

Data validation creates a contract between your data producers and consumers. Instead of hoping the data is correct, you assert it. Every batch, every micro-batch, every API response is tested against a suite of declarative rules before it enters your pipeline. Think of it as unit tests for data.

The consequences of skipping validation are well-documented. Google's seminal paper "Data Management Challenges in Production Machine Learning" (2017) reports that data issues account for the majority of production ML incidents. Without automated checks, teams resort to manual spot-checks — a practice that scales poorly and catches problems only after they have already caused harm.

Rule of thumb: If your model retrains automatically, your data must be validated automatically. Manual validation gates are acceptable only when a human explicitly approves every retraining run.

What Should You Validate?

  • Schema — column names, data types, nullable constraints
  • Completeness — null ratios, missing-value thresholds
  • Uniqueness — primary-key uniqueness, deduplication
  • Distributions — mean, standard deviation, quantiles, KL divergence from a reference
  • Freshness — maximum allowed age of the most recent record
  • Relationships — referential integrity between tables, foreign-key constraints
  • Business rules — domain-specific invariants (e.g., price > 0, age ∈ [0, 150])

Core Concepts in Great Expectations

Great Expectations (GE) is the most widely adopted open-source framework for data validation in the Python ecosystem. It introduces a layered architecture with four key abstractions:

Data Context

The DataContext is the entry point to everything in GE. It manages configuration, stores expectation suites, connects to data sources, and orchestrates validation runs. Under the hood it is backed by a YAML file (great_expectations.yml) that lives in your project root. 

# Initialise a new Great Expectations project
# This creates the great_expectations/ directory tree
$ pip install great_expectations

$ great_expectations init

# Or programmatically:
import great_expectations as gx

context = gx.get_context()          # loads ./great_expectations/great_expectations.yml
print(context.root_directory)       # /home/user/project/great_expectations

Datasources & Data Assets

A Datasource tells GE how to connect to your data (Pandas, Spark, SQL). A Data Asset tells it which table or file pattern to read. Together they form the bridge between your storage layer and the validation engine. 

# Add a Pandas datasource backed by a local directory of CSVs
datasource = context.sources.add_pandas_filesystem(
    name="local_csv",
    base_directory="data/",
)

# Register a data asset with a regex for partitioned files
asset = datasource.add_csv_asset(
    name="user_events",
    batching_regex=r"user_events_(?P<year>\d{4})-(?P<month>\d{2})\.csv",
)

# For SQL datasources:
sql_ds = context.sources.add_sql(
    name="warehouse",
    connection_string="postgresql+psycopg2://user:pass@host:5432/analytics",
)
sql_asset = sql_ds.add_table_asset(name="orders", table_name="public.orders")

Batch Requests

A Batch Request selects a specific slice of a data asset — for instance, the partition for January 2025. GE uses batch requests to materialise data lazily, so you can validate terabyte-scale datasets without loading everything into memory. 

batch_request = asset.build_batch_request(
    options={"year": "2025", "month": "01"}
)

# Preview the batch to confirm it loaded correctly
batches = asset.get_batch_list_from_batch_request(batch_request)
print(f"Loaded {len(batches)} batch(es), first has {batches[0].data.dataframe.shape[0]} rows")

Expectation Suites

An Expectation is a single declarative assertion about your data — "column age should have values between 0 and 150" or "column email should never be null." An Expectation Suite bundles related expectations into a reusable, version-controlled collection.

GE ships with over 300 built-in expectations spanning column-level, table-level, and multi-column checks. They follow a consistent naming convention: expect_column_values_to_<predicate>, expect_table_row_count_to_<predicate>, etc. 

# Create a new expectation suite
suite = context.add_expectation_suite(
    expectation_suite_name="user_events.warning"
)

# Get a validator — the object that runs expectations against data
validator = context.get_validator(
    batch_request=batch_request,
    expectation_suite_name="user_events.warning",
)

# ── Schema checks ──
validator.expect_table_columns_to_match_ordered_list(
    column_list=["user_id", "event_type", "timestamp", "payload"]
)

# ── Null checks ──
validator.expect_column_values_to_not_be_null(column="user_id")
validator.expect_column_values_to_not_be_null(column="timestamp")

# ── Type checks ──
validator.expect_column_values_to_be_of_type(column="user_id", type_="int64")

# ── Range checks ──
validator.expect_column_values_to_be_between(
    column="timestamp",
    min_value="2020-01-01",
    max_value="2026-12-31",
    parse_strings_as_datetimes=True,
)

# ── Uniqueness checks ──
validator.expect_compound_columns_to_be_unique(
    column_list=["user_id", "timestamp"]
)

# ── Distribution checks ──
validator.expect_column_mean_to_be_between(
    column="payload", min_value=50, max_value=500
)

# ── Categorical checks ──
validator.expect_column_values_to_be_in_set(
    column="event_type",
    value_set=["click", "purchase", "signup", "pageview"],
)

# ── Row count checks ──
validator.expect_table_row_count_to_be_between(min_value=1000, max_value=10_000_000)

# Persist the suite to disk
validator.save_expectation_suite(discard_failed_expectations=False)
Warning vs. Error suites: A common pattern is to maintain two suites per dataset — a .warning suite with soft thresholds (alert but don't block) and a .critical suite with hard constraints (fail the pipeline). This lets you catch drift early while still halting on truly broken data.

Profiling: Bootstrap Suites from Data

Manually writing hundreds of expectations is tedious. GE's Data Assistant (formerly the profiler) can analyse a reference batch and auto-generate a baseline suite with statistically derived thresholds. 

# Auto-profile a reference batch
result = context.assistants.onboarding.run(
    batch_request=batch_request,
    exclude_column_names=["payload"],  # skip free-text columns
)

# The result contains an auto-generated suite
profiled_suite = result.get_expectation_suite(
    expectation_suite_name="user_events.profiled"
)
context.add_or_update_expectation_suite(expectation_suite=profiled_suite)
print(f"Generated {len(profiled_suite.expectations)} expectations")

Checkpoints & Actions

A Checkpoint ties together a batch request, an expectation suite, and a list of actions to execute after validation completes. Checkpoints are the primary unit of orchestration — you trigger a checkpoint from Airflow, CI, or a cron job, and it handles everything else. 

# Define a checkpoint with multiple validation configs
checkpoint = context.add_or_update_checkpoint(
    name="user_events_checkpoint",
    validations=[
        {
            "batch_request": batch_request,
            "expectation_suite_name": "user_events.critical",
        },
        {
            "batch_request": batch_request,
            "expectation_suite_name": "user_events.warning",
        },
    ],
    action_list=[
        # 1. Persist validation results as JSON
        {
            "name": "store_validation_result",
            "action": {"class_name": "StoreValidationResultAction"},
        },
        # 2. Rebuild the Data Docs static site
        {
            "name": "update_data_docs",
            "action": {"class_name": "UpdateDataDocsAction"},
        },
        # 3. Send a Slack alert on failure
        {
            "name": "slack_on_failure",
            "action": {
                "class_name": "SlackNotificationAction",
                "slack_webhook": "${SLACK_WEBHOOK_URL}",
                "notify_on": "failure",
                "renderer": {
                    "class_name": "SlackRenderer",
                },
            },
        },
    ],
)

# Run the checkpoint
result = checkpoint.run()
print(f"Checkpoint success: {result.success}")

# Programmatically inspect per-suite results
for vr in result.run_results.values():
    stats = vr["validation_result"].statistics
    print(f"  Suite: {stats['expectation_suite_name']}")
    print(f"  Evaluated: {stats['evaluated_expectations']}")
    print(f"  Successful: {stats['successful_expectations']}")
    print(f"  Failed: {stats['unsuccessful_expectations']}")

Built-in Actions

  • StoreValidationResultAction — saves the JSON result to a configured store (local, S3, GCS, Azure Blob)
  • UpdateDataDocsAction — rebuilds the human-readable Data Docs site
  • SlackNotificationAction — posts a summary to Slack on success or failure
  • EmailAction — sends an email via SMTP
  • OpsgenieAlertAction — creates an Opsgenie incident
  • PagerdutyAlertAction — triggers a PagerDuty event
  • SNSNotificationAction — publishes to an AWS SNS topic

Writing Custom Expectations

The 300+ built-in expectations cover the vast majority of use cases, but real-world datasets often require domain-specific validations. GE provides a clean extension API: subclass ColumnMapExpectation (for row-level checks) or ColumnAggregateExpectation (for aggregate checks) and implement the core logic. 

from great_expectations.expectations.expectation import ColumnMapExpectation
from great_expectations.execution_engine import PandasExecutionEngine
from great_expectations.expectations.metrics import (
    ColumnMapMetricProvider,
    column_condition_partial,
)


class ColumnValuesLuhnValid(ColumnMapMetricProvider):
    """Metric: checks if each value passes the Luhn algorithm (credit cards)."""
    condition_metric_name = "column_values.luhn_valid"

    @column_condition_partial(engine=PandasExecutionEngine)
    def _pandas(cls, column, **kwargs):
        def luhn_check(num_str):
            digits = [int(d) for d in str(num_str) if d.isdigit()]
            odd_digits = digits[-1::-2]
            even_digits = digits[-2::-2]
            total = sum(odd_digits)
            for d in even_digits:
                total += sum(divmod(d * 2, 10))
            return total % 10 == 0
        return column.apply(luhn_check)


class ExpectColumnValuesToBeLuhnValid(ColumnMapExpectation):
    """Expect each value in the column to be a valid Luhn number."""
    map_metric = "column_values.luhn_valid"
    success_keys = ("mostly",)
    default_kwarg_values = {"mostly": 1.0}

    library_metadata = {
        "maturity": "experimental",
        "tags": ["finance", "credit-card"],
    }

# Usage — the expectation is now available on any validator
validator.expect_column_values_to_be_luhn_valid(
    column="card_number", mostly=0.99
)

Custom expectations can also be published as standalone Python packages and shared across teams via an internal PyPI registry. GE automatically discovers any expectation class installed in the Python environment.

Testing your expectations: Always write unit tests for custom expectations using pytest. GE provides test fixtures via great_expectations.self_check.util that let you run expectations against in-memory DataFrames without needing a full Data Context.

Integration with Airflow & CI

Data validation is most valuable when it runs automatically as part of your existing infrastructure. GE integrates cleanly with pipeline orchestrators, CI systems, and cloud platforms.

Airflow Integration

The great_expectations_provider package provides a native Airflow operator that wraps checkpoint execution. Use it as a gate task between your ingestion and transformation steps. 

# Airflow DAG with a Great Expectations validation gate
from airflow import DAG
from airflow.operators.python import PythonOperator
from great_expectations_airflow.operators.great_expectations import (
    GreatExpectationsOperator,
)
from datetime import datetime

with DAG(
    dag_id="ml_training_pipeline",
    start_date=datetime(2025, 1, 1),
    schedule_interval="@daily",
    catchup=False,
) as dag:

    ingest = PythonOperator(
        task_id="ingest_data",
        python_callable=ingest_from_warehouse,
    )

    validate = GreatExpectationsOperator(
        task_id="validate_training_data",
        data_context_root_dir="/opt/airflow/great_expectations",
        checkpoint_name="user_events_checkpoint",
        fail_task_on_validation_failure=True,   # blocks downstream tasks
        return_json_dict=True,
    )

    train = PythonOperator(
        task_id="train_model",
        python_callable=train_and_register,
    )

    ingest >> validate >> train  # validate gates the training step

CI/CD Integration

Run checkpoints in your CI pipeline to catch data contract violations on every pull request that touches SQL queries, feature engineering code, or data config files. 

# .github/workflows/data-validation.yml
name: Data Validation
on:
  pull_request:
    paths:
      - "sql/**"
      - "features/**"
      - "great_expectations/**"

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install great_expectations
      - run: |
          python -c "
          import great_expectations as gx
          context = gx.get_context()
          result = context.run_checkpoint('user_events_checkpoint')
          assert result.success, 'Data validation failed!'
          "
      - name: Upload Data Docs
        uses: actions/upload-artifact@v4
        with:
          name: data-docs
          path: great_expectations/uncommitted/data_docs/

dbt Integration

If you use dbt for transformations, GE can validate source tables before dbt runs and validate output tables after. The dbt-expectations package also ports many GE expectations into dbt's native test framework.

Data Docs & Reporting

One of GE's strongest features is Data Docs — an auto-generated static HTML site that renders every expectation suite, every validation run, and every profiling result as rich, browsable documentation. It serves as a living data catalog that is always in sync with your actual validation code. 

# Configure Data Docs to publish to S3 (great_expectations.yml)
data_docs_sites:
  s3_site:
    class_name: SiteBuilder
    store_backend:
      class_name: TupleS3StoreBackend
      bucket: my-data-docs-bucket
      prefix: data_docs/
    site_index_builder:
      class_name: DefaultSiteIndexBuilder

# Build and open Data Docs from Python
context.build_data_docs()
context.open_data_docs()  # opens in browser

# Or from the CLI:
$ great_expectations docs build
$ great_expectations docs list

What Data Docs Show

  • Expectation suites — every expectation rendered with human-readable descriptions, thresholds, and examples
  • Validation results — pass/fail status, observed values, unexpected counts, and sample failing rows
  • Profiling reports — column-level statistics, distributions, and auto-generated expectations
  • Run history — time-series of validation results showing how data quality changes over time

For teams with multiple data domains, you can configure separate Data Docs sites — one per domain, each deployed to its own S3 prefix or GCS bucket. Access control is handled at the cloud storage level.

GE vs Alternatives

Great Expectations is not the only data validation framework. Here is how it compares to the most common alternatives across key dimensions.

Great Expectations

  • 300+ built-in expectations
  • Rich Data Docs UI
  • Native Pandas, Spark, SQL engines
  • Checkpoint-based orchestration
  • Profiler for auto-generating suites
  • Steeper learning curve
  • Heavier dependency footprint

Pandera

  • Pythonic, type-hint-based schemas
  • Lightweight — single pip install
  • Deep Pandas / Polars integration
  • Excellent for DataFrame-level checks
  • No built-in reporting UI
  • No native SQL support
  • Better for small–medium datasets

dbt Tests

  • First-class SQL support
  • Integrated with dbt workflows
  • Declarative YAML definitions
  • dbt-expectations package adds GE tests
  • Limited to SQL / warehouse data
  • No DataFrame-level checks
  • No standalone Data Docs

Soda Core

  • SodaCL — human-readable YAML checks
  • Multi-engine: SQL, Spark, Pandas
  • Soda Cloud for dashboards & alerts
  • Anomaly detection built-in
  • Smaller open-source community
  • Some features require paid Soda Cloud
  • Fewer built-in checks than GE
Which should you pick? Use Great Expectations when you need multi-engine support (Pandas + Spark + SQL), rich auto-generated documentation, and deep integration with orchestrators. Use Pandera for lightweight, code-first validation in pure Python pipelines. Use dbt tests if your validation lives entirely in the warehouse. Use Soda if you prefer YAML-driven checks with built-in anomaly detection.

Decision Flowchart

Ask yourself these questions in order:

  1. Is all your data in a SQL warehouse? → Start with dbt tests; add GE or Soda for advanced checks.
  2. Are you working with DataFrames in Python? → Pandera for lightweight checks; GE if you need orchestration.
  3. Do you need multi-engine validation (SQL + Spark + files)? → Great Expectations or Soda Core.
  4. Do you need auto-generated documentation and profiling? → Great Expectations.
  5. Do you need anomaly detection out of the box? → Soda Core with Soda Cloud.

In practice, many mature teams combine tools — for example, using dbt tests for SQL-layer checks and Great Expectations for feature engineering validation in Spark. The key is to have some automated validation at every boundary where data crosses between systems.
