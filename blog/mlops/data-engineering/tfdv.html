TensorFlow Data Validation

MLOps Series Data Engineering

Introduction & the TFX Ecosystem

Machine-learning models are only as good as the data they consume. Yet most production ML failures are data failures — missing columns, unexpected null rates, silently changed encodings, or distribution shifts that creep in between training and serving. TensorFlow Data Validation (TFDV) is the library purpose-built to catch these problems before they propagate downstream.

TFDV is a core component of TFX (TensorFlow Extended), Google's end-to-end platform for deploying production ML pipelines. Within TFX, TFDV powers two standard pipeline components:

  • StatisticsGen — computes descriptive statistics over each dataset split.
  • SchemaGen — infers a schema from those statistics.
  • ExampleValidator — validates new data against the schema and detects anomalies, skew, and drift.

Even outside TFX, TFDV can be used as a standalone Python library. It runs on Apache Beam, which means the same code scales from a laptop to a Dataflow or Spark cluster with zero changes. 

# Install TFDV (works in any Python 3.9+ environment)
$ pip install tensorflow-data-validation

# Verify installation
import tensorflow_data_validation as tfdv
print(tfdv.__version__)
Why not just use Pandas profiling? TFDV is designed for production pipelines. It generates compact, serialised protocol-buffer statistics that integrate with schema enforcement, anomaly detection, and serving-time validation — none of which Pandas-based profilers provide out of the box.

Generating Statistics

The first step in any TFDV workflow is computing dataset statistics. TFDV produces a DatasetFeatureStatisticsList proto that captures per-feature distributions, missing-value rates, domain cardinality, quantiles, and more — for both numerical and categorical features. 

import tensorflow_data_validation as tfdv

# Generate statistics from a CSV file
train_stats = tfdv.generate_statistics_from_csv(
    data_location="data/train.csv",
    delimiter=",",
    output_path="stats/train_stats.pb"
)

# Generate statistics from a TFRecord file
train_stats = tfdv.generate_statistics_from_tfrecord(
    data_location="data/train.tfrecord",
    output_path="stats/train_stats.pb"
)

# Visualise in a Jupyter notebook
tfdv.visualize_statistics(train_stats)

The visualize_statistics call renders an interactive Facets Overview widget inside Jupyter, showing histograms, quantile plots, and missing-value bars for every feature. You can also compare two datasets side by side — invaluable for spotting train/eval divergence early. 

# Compare training and evaluation statistics
eval_stats = tfdv.generate_statistics_from_csv(
    data_location="data/eval.csv"
)

# Side-by-side visualisation
tfdv.visualize_statistics(
    lhs_statistics=train_stats,
    rhs_statistics=eval_stats,
    lhs_name="Training",
    rhs_name="Evaluation"
)

Statistics on Beam

Under the hood, generate_statistics_from_csv launches a local Apache Beam pipeline. For large datasets, you can pass a pipeline_options argument to run on Dataflow, Flink, or Spark: 

from apache_beam.options.pipeline_options import PipelineOptions

options = PipelineOptions([
    "--runner=DataflowRunner",
    "--project=my-gcp-project",
    "--region=us-central1",
    "--temp_location=gs://my-bucket/tmp"
])

train_stats = tfdv.generate_statistics_from_csv(
    data_location="gs://my-bucket/data/train-*.csv",
    output_path="gs://my-bucket/stats/train.pb",
    pipeline_options=options
)

Schema Inference

A schema is a contract that describes what valid data looks like. TFDV can automatically infer a schema from computed statistics. The schema captures:

  • Feature presence — required vs. optional, minimum fraction of non-missing values.
  • Feature type — INT, FLOAT, BYTES (string).
  • Domain constraints — allowed categorical values, numerical ranges, string patterns.
  • Feature shape — fixed-length vs. variable-length, expected valency.
  • Semantic types — natural-language text, image bytes, etc.
# Infer schema from training statistics
schema = tfdv.infer_schema(statistics=train_stats)

# Display the schema in a notebook
tfdv.display_schema(schema=schema)

# Save the schema for version control
tfdv.write_schema_text(schema, "schema/v1/schema.pbtxt")

The inferred schema is a starting point. In practice you will curate it — tightening constraints, adding domain-specific rules, and committing it to source control so that it becomes the authoritative contract between data producers and model consumers.

Editing the Schema Programmatically

# Mark a feature as required (min_fraction = 1.0)
tfdv.get_feature(schema, "user_age").presence.min_fraction = 1.0

# Restrict a categorical feature to a fixed domain
tfdv.set_domain(schema, "country",
    tfdv.schema_pb2.StringDomain(
        name="country",
        value=["US", "UK", "DE", "FR", "IN"]
    )
)

# Set a valid numerical range
tfdv.set_domain(schema, "transaction_amount",
    tfdv.schema_pb2.FloatDomain(
        name="transaction_amount",
        min=0.0,
        max=50000.0
    )
)

# Add an environment for serving (no label feature)
schema.default_environment.append("TRAINING")
schema.default_environment.append("SERVING")
tfdv.get_feature(schema, "label").not_in_environment.append("SERVING")
Never ship the raw inferred schema to production. The auto-generated schema reflects whatever your training data happens to contain today — including any existing bugs. Always review and tighten constraints before promoting the schema to your validation pipeline.

Anomaly Detection

Once you have a curated schema, TFDV can validate any new dataset against it and surface anomalies — deviations from the expected contract. Anomaly types include:

  • Missing features — a column that the schema expects is absent.
  • New features — a column appears that the schema doesn't know about.
  • Type mismatch — an INT feature now contains FLOAT values.
  • Out-of-domain values — a categorical feature contains labels not in the allowed list.
  • Out-of-range values — numerical values exceed the specified min/max.
  • Missing-value rate spikes — nulls exceed the tolerated threshold.
  • Unexpected string patterns — values that violate regex constraints.
# Validate new serving data against the curated schema
serving_stats = tfdv.generate_statistics_from_csv(
    data_location="data/serving_batch_20240601.csv"
)

anomalies = tfdv.validate_statistics(
    statistics=serving_stats,
    schema=schema,
    environment="SERVING"
)

# Display anomalies as a table
tfdv.display_anomalies(anomalies)

The returned Anomalies proto contains a per-feature breakdown. Each entry describes the anomaly type, a short reason string, and a longer description explaining exactly what went wrong. In a TFX pipeline the ExampleValidator component runs this check automatically and can gate downstream components.

Handling Known Deviations

Sometimes an anomaly is expected — for instance, your serving environment legitimately lacks the label column. TFDV supports schema environments to handle this: 

# The label is expected in TRAINING but not SERVING
tfdv.get_feature(schema, "label").not_in_environment.append("SERVING")

# Validate again — the "missing label" anomaly is now suppressed
anomalies = tfdv.validate_statistics(
    statistics=serving_stats,
    schema=schema,
    environment="SERVING"
)

# Check programmatically if any anomalies remain
if anomalies.anomaly_info:
    print("DATA QUALITY GATE FAILED")
    for feature, info in anomalies.anomaly_info.items():
        print(f"  {feature}: {info.short_description}")
    raise RuntimeError("Anomalies detected — aborting pipeline")
else:
    print("All checks passed ✓")

Without TFDV

  • Schema violations silently reach training
  • Missing columns cause cryptic TensorFlow errors
  • New categorical values degrade model accuracy
  • Null spikes discovered only after model ships
  • No systematic record of data expectations

With TFDV

  • Schema contract enforced before training begins
  • Clear, actionable anomaly reports per feature
  • Out-of-vocabulary values caught at ingestion
  • Quality gates halt the pipeline on null spikes
  • Schema is version-controlled and auditable

Training-Serving Skew & Drift Detection

Two of the most insidious production-ML problems are training-serving skew and data drift. They are related but distinct:

  • Training-serving skew — the distribution of a feature at serving time differs significantly from the distribution seen during training. This often happens when the feature-engineering code path diverges between training and serving.
  • Data drift — the distribution of a feature changes over time within the same environment (e.g., this week's training data vs. last week's training data). This signals that the world is changing and the model may need retraining.

TFDV uses the L-infinity distance (Chebyshev distance) between distributions as its default divergence metric. For a categorical feature, the L-infinity distance is the maximum absolute difference in probability between corresponding categories: 

# L-infinity distance for categorical distributions
# d = max_i | P_train(x_i) - P_serve(x_i) |
#
# Example:
#   Train: {US: 0.45, UK: 0.30, DE: 0.25}
#   Serve: {US: 0.60, UK: 0.25, DE: 0.15}
#   d = max(|0.45-0.60|, |0.30-0.25|, |0.25-0.15|) = 0.15

Configuring Skew & Drift Comparators

You set thresholds on the schema features. If the measured distance exceeds the threshold, TFDV raises an anomaly: 

# Set a skew comparator (training vs. serving)
tfdv.get_feature(schema, "country").skew_comparator.infinity_norm.threshold = 0.1

# Set a drift comparator (consecutive training spans)
tfdv.get_feature(schema, "country").drift_comparator.infinity_norm.threshold = 0.05

# For numerical features, use Jensen-Shannon divergence
tfdv.get_feature(schema, "transaction_amount"
).skew_comparator.jensen_shannon_divergence.threshold = 0.02

# Validate with both training and serving stats
anomalies = tfdv.validate_statistics(
    statistics=serving_stats,
    schema=schema,
    environment="SERVING",
    serving_statistics=train_stats   # baseline for skew
)

tfdv.display_anomalies(anomalies)

Drift Detection Between Training Spans

For drift detection, you compare statistics from two consecutive time periods of training data. This is especially important for models trained on streaming or daily-batch data: 

# Compare last week's training data with this week's
prev_stats = tfdv.generate_statistics_from_csv("data/train_week_23.csv")
curr_stats = tfdv.generate_statistics_from_csv("data/train_week_24.csv")

# Validate current stats against schema with drift checks
anomalies = tfdv.validate_statistics(
    statistics=curr_stats,
    schema=schema,
    environment="TRAINING",
    previous_statistics=prev_stats   # baseline for drift
)

if anomalies.drift_skew_info:
    print("Drift detected in features:")
    for info in anomalies.drift_skew_info:
        print(f"  {info.path.step[0]}: {info.drift_measurements}")
Choosing thresholds: Start with generous thresholds (e.g., 0.1 for L-infinity) and tighten them over time as you learn what "normal" variance looks like for each feature. A threshold that is too tight will produce noisy alerts; too loose and you'll miss real problems.

TFX Pipeline Integration

In a full TFX pipeline, TFDV functionality is split across three standard components that run in sequence. Together, they form the data-validation gate that protects every downstream step — from feature engineering to model training and evaluation. 

from tfx.components import (
    CsvExampleGen,
    StatisticsGen,
    SchemaGen,
    ExampleValidator,
    Transform,
    Trainer,
    Evaluator,
    Pusher,
)

# 1. Ingest data
example_gen = CsvExampleGen(input_base="data/")

# 2. Compute statistics
statistics_gen = StatisticsGen(
    examples=example_gen.outputs["examples"]
)

# 3. Infer or import a curated schema
schema_gen = SchemaGen(
    statistics=statistics_gen.outputs["statistics"],
    infer_feature_shape=True
)

# 4. Validate data — anomalies, skew, drift
example_validator = ExampleValidator(
    statistics=statistics_gen.outputs["statistics"],
    schema=schema_gen.outputs["schema"]
)

# 5. Continue only if validation passes
transform = Transform(
    examples=example_gen.outputs["examples"],
    schema=schema_gen.outputs["schema"],
    module_file="transform_module.py"
)

Using a Curated Schema (ImportSchemaGen)

In production you typically do not re-infer the schema on every run. Instead, you version-control a curated .pbtxt file and import it: 

from tfx.components import ImportSchemaGen

# Import the curated schema from version control
schema_importer = ImportSchemaGen(
    schema_file="schema/v3/schema.pbtxt"
)

# Wire it into ExampleValidator
example_validator = ExampleValidator(
    statistics=statistics_gen.outputs["statistics"],
    schema=schema_importer.outputs["schema"]
)

Custom Validation Logic

For validation rules that go beyond what the schema proto supports, you can write a custom Beam PTransform that consumes the statistics proto and emits pass/fail signals: 

import apache_beam as beam
from tensorflow_metadata.proto.v0 import statistics_pb2

class CheckRowCount(beam.DoFn):
    """Fail if the dataset has fewer than min_rows examples."""
    def __init__(self, min_rows):
        self.min_rows = min_rows

    def process(self, stats_proto):
        for dataset in stats_proto.datasets:
            if dataset.num_examples < self.min_rows:
                raise ValueError(
                    f"Expected >= {self.min_rows} rows, "
                    f"got {dataset.num_examples}"
                )

Best Practices

After deploying TFDV across dozens of production pipelines, a set of recurring patterns emerges. Following these guidelines will save your team significant debugging time.

1. Version Your Schema in Git

Treat the schema the same way you treat code — review it in PRs, tag releases, and keep a changelog. When an anomaly fires, you need to know whether the data changed or the schema changed.

2. Separate Environments

Define at least two environments — TRAINING and SERVING. The serving environment will typically exclude the label column, and may have relaxed presence constraints for features that are populated asynchronously.

3. Set Skew and Drift Thresholds on Critical Features

Don't try to monitor every feature with tight thresholds. Focus on features with high feature importance in your model. Start loose, collect baseline metrics for a few weeks, then tighten based on observed variance. 

# Focus skew/drift monitoring on top features
critical_features = ["user_age", "country", "device_type", "purchase_amount"]

for feat_name in critical_features:
    feat = tfdv.get_feature(schema, feat_name)
    feat.skew_comparator.infinity_norm.threshold = 0.1
    feat.drift_comparator.infinity_norm.threshold = 0.05

4. Gate Your Pipeline on Validation Results

In TFX, the ExampleValidator produces an anomalies artifact. Wire a conditional so that downstream components (Transform, Trainer) only execute when validation passes. This prevents wasting compute on training with bad data.

5. Log Statistics to a Time-Series Store

Persist the DatasetFeatureStatisticsList proto from every pipeline run. Over time, this gives you a rich historical view of how each feature's distribution evolves — making it much easier to set drift thresholds and diagnose production issues.

6. Combine with Great Expectations for Non-TFX Pipelines

If your organization doesn't use TFX, you can still use TFDV for statistics and schema inference, then export expectations to Great Expectations for integration with Airflow, Dagster, or Prefect. The two tools complement each other: TFDV excels at ML-specific checks (skew, drift), while Great Expectations covers broader data-quality rules.

Common pitfall: Running TFDV validation only in training pipelines. You should also validate serving-time data — either in the serving pipeline itself or via a shadow validation job that samples production traffic. Training-time validation alone cannot catch training-serving skew.

TFDV is a foundational building block for reliable ML systems. By investing in schema curation and automated validation early, you shift data-quality problems left — catching them at ingestion rather than discovering them through degraded model performance in production. Combined with the rest of the TFX ecosystem, it provides a battle-tested path from notebook experiments to production-grade ML pipelines.
