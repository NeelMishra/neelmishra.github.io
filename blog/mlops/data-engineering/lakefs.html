LakeFS: Git-like Branching for Data Lakes

MLOps Series · Data Engineering

What Is LakeFS?

LakeFS is an open-source platform that layers Git-like version control on top of object storage (S3, GCS, Azure Blob). Instead of versioning code, it versions data — every Parquet file, every Delta partition, every raw CSV landing in your lake gets the same branch/commit/merge semantics developers already know.

The core problem it solves: data lakes are mutable shared state. A single bad ETL write can corrupt production tables with no rollback path. LakeFS turns your lake into an immutable, auditable, branch-isolated environment where you can experiment freely and promote changes atomically.

Architecture Overview

LakeFS sits as a lightweight metadata layer between your compute engines (Spark, Trino, dbt) and the underlying object store. It does not copy data. Instead, it maintains a metadata tree (a Merkle-like structure called a graveler) that maps logical paths to physical object addresses.

// LakeFS Architecture — simplified data flow

Client (Spark / Trino / CLI)
    │
    ▼
LakeFS API Server        ← stateless, horizontally scalable
    │
    ├── Auth              ← RBAC, OIDC, LDAP integration
    ├── Graveler          ← metadata tree engine (branches, commits)
    │       │
    │       └── KV Store  ← PostgreSQL or DynamoDB (metadata persistence)
    │
    └── Block Adapter    ← S3 / GCS / Azure (actual data I/O)
Zero-copy design: Creating a branch in LakeFS is an O(1) metadata pointer operation, not a data duplication. A branch with 100 TB of data costs the same as a branch with 1 KB — only the metadata pointer differs.

The API server exposes three access interfaces:

  • S3-compatible Gateway — existing S3 SDK calls work unchanged; just swap the endpoint
  • OpenAPI REST API — full programmatic control over branches, commits, diffs
  • lakectl CLI — developer-friendly command line for interactive workflows

Repository Concept

A LakeFS repository maps to a single storage namespace (e.g., s3://my-lake/repo1). Inside it, you get branches, commits, and tags — all scoped to that namespace. Each repository has a default branch (typically main) that serves as the production-ready view of the data.

# Create a new repository via lakectl
lakectl repo create \
  lakefs://my-analytics \
  s3://company-datalake/analytics \
  --default-branch main

# List existing repositories
lakectl repo list
# ┌───────────────────┬──────────────────────────────────┬─────────┐
# │ Repository        │ Storage Namespace                 │ Branch  │
# ├───────────────────┼──────────────────────────────────┼─────────┤
# │ my-analytics      │ s3://company-datalake/analytics  │ main    │
# │ ml-features       │ s3://company-datalake/features   │ main    │
# └───────────────────┴──────────────────────────────────┴─────────┘

The Branching Model

Branching is the foundation of the LakeFS workflow. Like Git, branches are cheap pointers — creating one takes milliseconds regardless of data size. This unlocks patterns impossible with traditional data lakes: isolated ETL testing, safe schema migrations, feature branch experimentation for ML pipelines.

Creating and Managing Branches

# Create a feature branch from main
lakectl branch create \
  lakefs://my-analytics/feature/new-schema \
  --source lakefs://my-analytics/main

# List all branches
lakectl branch list lakefs://my-analytics
# main
# feature/new-schema
# staging/etl-v2

# Delete a branch (metadata only — no data loss)
lakectl branch delete lakefs://my-analytics/feature/new-schema

# Revert uncommitted changes on a branch
lakectl branch revert lakefs://my-analytics/staging/etl-v2

Branch Isolation Guarantees

Each branch presents an independent, consistent snapshot of the entire repository. Writes on feature/new-schema are completely invisible to readers on main. This is enforced at the metadata level — there is no file-level locking, no race conditions between branches.

The isolation model supports several data engineering patterns:

  • Environment branches — dev, staging, main mirror deployment stages
  • ETL branches — each pipeline run writes to an ephemeral branch, merges on success
  • Experiment branches — ML engineers test feature engineering without polluting production tables
  • Hotfix branches — urgent schema corrections applied in isolation, merged after validation
Caution: LakeFS branches do not provide ACID transactions across multiple object writes within a single commit. If your ETL writes 500 Parquet files and fails at file 300, the uncommitted branch will have partial state. Always wrap your pipeline: write to branch → validate → commit. Never commit partial writes.

Branch Naming Conventions

Adopt a structured naming convention that integrates with your orchestration tooling:

# Recommended branch naming patterns
etl/{pipeline_name}/{run_id}       # e.g., etl/user-events/2024-01-15-001
experiment/{user}/{description}     # e.g., experiment/alice/new-feature-store
staging/{release_version}           # e.g., staging/v2.3.0
hotfix/{ticket_id}                  # e.g., hotfix/DATA-1234

Commits, History & Merges

A LakeFS commit is an immutable, content-addressed snapshot of the repository at a point in time. Once committed, data cannot be altered — only new commits can be appended. This provides a full audit trail and enables instant rollback to any historical state.

Committing Changes

# Stage and commit changes on a branch
lakectl commit lakefs://my-analytics/feature/new-schema \
  --message "Add user_events partitioned by date" \
  --meta pipeline_id=etl-v2 \
  --meta run_date=2024-01-15

# View commit log
lakectl log lakefs://my-analytics/main
# commit abc123def4 (HEAD)
# Author: etl-service
# Date:   2024-01-15T08:30:00Z
# Message: Daily user_events ingestion
#
# commit 789fed012a
# Author: alice
# Date:   2024-01-14T16:22:00Z
# Message: Fix schema drift in payment_transactions

Diffing Between References

LakeFS computes diffs at the object level — added, removed, or modified files between any two references (branches, commits, tags).

# Diff between a feature branch and main
lakectl diff lakefs://my-analytics/feature/new-schema lakefs://my-analytics/main
# Added:    data/user_events/dt=2024-01-15/part-00000.parquet
# Added:    data/user_events/dt=2024-01-15/part-00001.parquet
# Modified: _metadata/schema.json
# Deleted:  data/legacy/old_events.csv

# Diff between two commits
lakectl diff lakefs://my-analytics/abc123def4 lakefs://my-analytics/789fed012a

Merging Strategies

LakeFS supports merging one branch into another. The merge operation is metadata-only — it updates pointers, not data. If there are no conflicts (same file modified on both branches), the merge completes instantly.

# Merge feature branch into main
lakectl merge \
  lakefs://my-analytics/feature/new-schema \
  lakefs://my-analytics/main \
  --message "Merge new user_events schema"

# If conflicts exist, LakeFS will report them:
# CONFLICT: data/user_events/_metadata/schema.json
# Resolve by choosing source or destination version

Conflict resolution in LakeFS is simpler than in Git because conflicts are at the whole-file level, not line-level. If the same object path was modified on both branches, you choose one version or the other. There is no three-way merge of file contents.

Tags and Rollback

# Create a tag for a known-good state
lakectl tag create lakefs://my-analytics/v2.3.0-release abc123def4

# Rollback main to a previous commit
lakectl branch revert lakefs://my-analytics/main \
  --commit 789fed012a \
  --message "Revert: bad schema migration"

# Instant recovery — main now reflects the state at 789fed012a
Rollback is O(1): Since commits are immutable snapshots, reverting does not move or delete data. It simply repoints the branch HEAD to an older commit. The "bad" data still exists in the object store (for audit) until garbage collection runs.

Spark + LakeFS Integration

The most powerful integration point is the S3 Gateway. LakeFS exposes an S3-compatible API, meaning any tool that reads/writes S3 can work with LakeFS branches without code changes. You simply swap the endpoint and encode the branch name in the path.

S3 Gateway URL Convention

# Standard S3 path
s3://my-bucket/data/user_events/

# LakeFS equivalent — branch encoded in path
s3://my-analytics/main/data/user_events/
#     ^^^^^^^^^^^    ^^^^
#     repository     branch

# Reading from a feature branch
s3://my-analytics/feature/new-schema/data/user_events/

# Reading from a specific commit
s3://my-analytics/abc123def4/data/user_events/

Spark Configuration

# spark-defaults.conf — point Spark at LakeFS S3 Gateway

spark.hadoop.fs.s3a.endpoint           https://lakefs.company.internal:8000
spark.hadoop.fs.s3a.access.key         AKIAIOSFODNN7LAKEFS
spark.hadoop.fs.s3a.secret.key         wJalrXUtnFEMI/K7MDENG/bPxRfiCYLAKEFS
spark.hadoop.fs.s3a.path.style.access  true
spark.hadoop.fs.s3a.impl               org.apache.hadoop.fs.s3a.S3AFileSystem

# For Delta Lake — ensure log store uses the same endpoint
spark.delta.logStore.class             org.apache.spark.sql.delta.storage.S3SingleDriverLogStore

PySpark Example: Branch-Isolated ETL

from pyspark.sql import SparkSession
import requests

# ── 1. Create an isolated ETL branch via LakeFS API ──
lakefs_api = "https://lakefs.company.internal/api/v1"
headers    = {"Authorization": "Bearer "}

requests.post(
    f"{lakefs_api}/repositories/my-analytics/branches",
    json={"name": "etl/daily-ingest/2024-01-15", "source": "main"},
    headers=headers
)

# ── 2. Spark reads from main, writes to the ETL branch ──
spark = SparkSession.builder \
    .appName("daily-ingest") \
    .config("spark.hadoop.fs.s3a.endpoint", "https://lakefs.company.internal:8000") \
    .getOrCreate()

# Read production data from main branch
raw_df = spark.read.parquet(
    "s3a://my-analytics/main/data/raw/user_events/"
)

# Transform
cleaned_df = raw_df \
    .filter("event_timestamp IS NOT NULL") \
    .withColumn("dt", to_date("event_timestamp")) \
    .dropDuplicates(["event_id"])

# Write to the isolated ETL branch — main is untouched
cleaned_df.write \
    .partitionBy("dt") \
    .mode("overwrite") \
    .parquet("s3a://my-analytics/etl/daily-ingest/2024-01-15/data/curated/user_events/")

# ── 3. Validate, then commit and merge ──
requests.post(
    f"{lakefs_api}/repositories/my-analytics/branches/etl%2Fdaily-ingest%2F2024-01-15/commits",
    json={"message": "Daily user_events ingestion — 2024-01-15"},
    headers=headers
)

requests.post(
    f"{lakefs_api}/repositories/my-analytics/refs/etl%2Fdaily-ingest%2F2024-01-15/merge/main",
    json={"message": "Merge daily ingest into main"},
    headers=headers
)
Why this matters: If the Spark job fails mid-write, the main branch is untouched. No partial data, no corruption. You simply delete the ETL branch and retry. This is the fundamental guarantee that LakeFS provides over raw S3 writes.

Hadoop FileSystem Client (lakefs:// protocol)

For tighter integration, LakeFS also provides a native Hadoop FileSystem implementation that uses the lakefs:// URI scheme directly, avoiding the S3 Gateway entirely:

# Then read/write with lakefs:// URIs

CI/CD for Data Quality

LakeFS provides hooks — pre-commit and pre-merge actions that execute validation logic before data is promoted. This is the data equivalent of CI/CD: automated quality gates that block bad data from reaching production.

Hook Types

  • pre-commit — runs before a commit is finalized on any branch
  • pre-merge — runs before a merge into a target branch
  • post-commit — triggers after a successful commit (notifications, downstream pipelines)
  • post-merge — triggers after a successful merge

Hook Configuration (actions YAML)

# _lakefs_actions/quality_gates.yaml
name: data-quality-checks
on:
  pre-merge:
    branches:
      - main
      - staging/*

hooks:
  - id: schema-validation
    type: webhook
    properties:
      url: https://data-quality.internal/validate-schema
      timeout: 60s
      query_params:
        min_rows: "1000"
        max_null_pct: "0.05"

  - id: great-expectations-suite
    type: webhook
    properties:
      url: https://data-quality.internal/run-ge-suite
      timeout: 300s
      query_params:
        suite_name: "production_expectations"

  - id: freshness-check
    type: webhook
    properties:
      url: https://data-quality.internal/check-freshness
      timeout: 30s

Automated ETL Pipeline Pattern

Combining branches, hooks, and merge gives you a fully automated data promotion pipeline:

// End-to-end ETL pipeline with LakeFS quality gates

STEP 1: create_branch("etl/daily/{run_id}", source="main")
           // Isolated workspace — zero-copy from main

STEP 2: run_spark_etl(read_from="main", write_to="etl/daily/{run_id}")
           // Transform and write data to the isolated branch

STEP 3: commit(branch="etl/daily/{run_id}", msg="Daily ingest {date}")
           // pre-commit hook fires → schema validation
           // If hook fails → commit is rejected, branch has uncommitted state

STEP 4: merge(source="etl/daily/{run_id}", target="main")
           // pre-merge hook fires → Great Expectations suite
           // If hook fails → merge is blocked, main is safe

STEP 5: delete_branch("etl/daily/{run_id}")
           // Cleanup — metadata only

STEP 6: garbage_collect()
           // Reclaim storage from unreferenced objects (periodic)
Garbage collection matters: Deleted branches and old commits still reference physical objects. Run lakectl gc periodically (or the server-side GC job) to reclaim storage. Without GC, your object store will grow monotonically even as you delete branches and overwrite files.

LakeFS vs DVC

Both LakeFS and DVC (Data Version Control) aim to version data, but they operate at fundamentally different layers and serve different primary audiences.

🌊 LakeFS

  • Layer: Infrastructure — sits between compute and storage
  • Scope: Entire data lake, multi-TB to PB scale
  • Branching: Zero-copy, O(1) metadata-only branches
  • Integration: S3 Gateway — transparent to any S3-compatible tool
  • Data format: Format-agnostic — Parquet, Delta, Iceberg, CSV, images
  • Collaboration: Multi-user, RBAC, concurrent branch writes
  • CI/CD: Built-in hooks (pre-commit, pre-merge webhooks)
  • Best for: Data platform teams, lake-wide governance

📦 DVC

  • Layer: Client-side — Git extension for data files
  • Scope: Project-level, typically per-repo datasets
  • Branching: Inherits Git branches; data stored externally with .dvc pointers
  • Integration: CLI-driven; requires dvc pull/push to sync data
  • Data format: Format-agnostic, but best for discrete file versioning
  • Collaboration: Through Git; no native multi-user concurrency
  • CI/CD: Via standard Git CI (GitHub Actions, GitLab CI)
  • Best for: ML engineers versioning training datasets alongside code

When to Choose Which

// Decision matrix

IF you need to version an entire data lake (multi-TB+)
   AND multiple teams write concurrently
   AND you want transparent integration with Spark/Trino/dbt
   → Choose LakeFS

IF you need to version specific datasets alongside ML code
   AND your workflow is Git-centric (branches, PRs)
   AND dataset size is moderate (GBs, not PBs)
   → Choose DVC

IF you need both lake-level governance AND project-level tracking
   → Use both: LakeFS for the lake, DVC for experiment artifacts
They're complementary, not competing: LakeFS versions the lake infrastructure. DVC versions ML project artifacts. In a mature MLOps stack, you might use LakeFS to manage the feature store and raw data lake, while DVC tracks the specific dataset snapshots and model files within each training repository.

Production Deployment Patterns

Deploying LakeFS at scale requires thoughtful architecture around high availability, metadata storage, and operational runbooks.

High Availability Setup

# Production HA deployment — Kubernetes

apiVersion: apps/v1
kind: Deployment
metadata:
  name: lakefs-server
spec:
  replicas: 3                     # Stateless — scale horizontally
  template:
    spec:
      containers:
        - name: lakefs
          image: treeverse/lakefs:1.x
          env:
            - name: LAKEFS_DATABASE_TYPE
              value: "postgres"
            - name: LAKEFS_DATABASE_POSTGRES_CONNECTION_STRING
              value: "postgres://lakefs:***@pg-cluster:5432/lakefs"
            - name: LAKEFS_BLOCKSTORE_TYPE
              value: "s3"
            - name: LAKEFS_BLOCKSTORE_S3_REGION
              value: "us-east-1"
          ports:
            - containerPort: 8000          # API + S3 Gateway
          resources:
            requests:
              cpu: "500m"
              memory: "1Gi"
            limits:
              cpu: "2"
              memory: "4Gi"

Metadata Store Considerations

The metadata backend is the critical path for all LakeFS operations. Choose based on scale:

PostgreSQL

  • Best for: up to ~50M objects per repository
  • Simple operations (backup, restore, monitoring)
  • Use managed services (RDS, Cloud SQL) for HA
  • Tune max_connections and shared_buffers for LakeFS load

DynamoDB

  • Best for: 50M+ objects, massive scale deployments
  • Auto-scaling, serverless pricing model
  • Higher operational complexity for cross-region replication
  • Use on-demand capacity for unpredictable workloads

Operational Runbook Essentials

# Essential operational commands

# 1. Health check
curl -s https://lakefs.company.internal/api/v1/healthcheck | jq .

# 2. Garbage collection — reclaim unreferenced objects
lakectl gc set-rules lakefs://my-analytics \
  --default-retention-days 30 \
  --branch main --retention-days 90

lakectl gc run lakefs://my-analytics

# 3. Export — materialize a branch to plain S3 (for tools that can't use LakeFS)
lakectl fs export \
  --source lakefs://my-analytics/main/data/curated/ \
  --dest s3://export-bucket/curated/ \
  --parallelism 50

# 4. Audit — who did what, when
lakectl log lakefs://my-analytics/main \
  --amount 100 \
  --after 2024-01-01T00:00:00Z
Monitoring tip: Expose LakeFS Prometheus metrics (/metrics endpoint) and build dashboards for: API latency by operation type, S3 Gateway throughput, active branches count, and metadata store query latency. Alert on commit/merge failures — they often signal upstream ETL issues.
