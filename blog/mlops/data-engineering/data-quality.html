Data Quality Metrics and Monitoring

MLOps Series Data Engineering

In production ML systems, model performance is bounded by data quality. A model trained on incomplete, stale, or inconsistent data will silently degrade—often with no error message, just worse predictions. Data quality monitoring is the immune system of your ML platform: it catches corruption before it reaches your models.

This post covers the six core data quality dimensions, how to measure each one quantitatively, how to build automated monitoring pipelines, and how degraded data quality propagates into model failures.

Data Quality Dimensions

Data quality is not a single number. It decomposes into orthogonal dimensions, each capturing a different failure mode. Understanding these dimensions lets you build targeted checks rather than vague "data looks wrong" alerts.

1. Completeness

Completeness measures the proportion of expected data that is actually present. A dataset is incomplete when rows are missing entirely (volume completeness) or when individual fields within present rows are null (column completeness). In an e-commerce pipeline, completeness failure might mean missing transaction records from a specific region due to an ingestion timeout.

  • Row-level: Does every expected entity have a record? Compare against a reference count or expected arrival cadence.
  • Column-level: What fraction of non-null values exist for each column? A null_ratio above threshold triggers an alert.
  • Temporal: Are there gaps in time-series data? Missing hourly partitions in event logs indicate ingestion failures.

2. Accuracy

Accuracy measures whether values correctly represent the real-world entities they describe. Inaccurate data passes schema validation but contains wrong values—a user's age recorded as 350, or a GPS coordinate placed in the ocean. Accuracy is the hardest dimension to measure because it requires a ground-truth reference.

  • Cross-reference validation: Compare values against authoritative external sources.
  • Statistical plausibility: Flag values outside learned distributions (z-score > 3).
  • Business rule checks: order_total == sum(line_items), end_date >= start_date.

3. Consistency

Consistency ensures that the same entity is represented identically across all systems and tables. When your user table says "active" but the billing system says "churned," downstream joins produce contradictory features. Consistency failures are especially dangerous in feature stores where features are assembled from multiple sources.

4. Freshness

Freshness measures how recently the data was updated. A recommendation model using week-old click data will serve stale suggestions. Freshness is measured as the difference between the current wall-clock time and the maximum event timestamp (or partition timestamp) in the dataset.

5. Timeliness

Timeliness is distinct from freshness: it measures whether data arrives within its expected SLA window. Data can be fresh (recent events) but untimely (arrived 3 hours late, missing the training window). Timeliness failures break scheduled pipelines even when the data itself is correct.

6. Uniqueness

Uniqueness ensures there are no unintended duplicates. Duplicate records inflate training distributions, bias aggregate statistics, and cause data leakage between train/test splits. Uniqueness is measured by checking primary key cardinality against row count and by computing exact or approximate duplicate rates across composite keys.

Key Insight: These six dimensions are orthogonal. Data can score perfectly on completeness (no nulls) while failing accuracy (wrong values everywhere). A robust monitoring system checks all dimensions independently.

Measuring Each Dimension

Each dimension must be reduced to a quantitative metric that can be tracked over time, compared against thresholds, and used to trigger alerts. Below are the formulas and implementation patterns for production measurement.

# Core data quality metric computations
import pandas as pd
import numpy as np
from datetime import datetime, timedelta

def completeness_score(df: pd.DataFrame) -> dict:
    """Column-level and row-level completeness metrics."""
    total_rows = len(df)
    column_completeness = {}
    for col in df.columns:
        non_null = df[col].notna().sum()
        column_completeness[col] = non_null / total_rows
    overall = np.mean(list(column_completeness.values()))
    return {"column": column_completeness, "overall": overall}

def accuracy_score(df: pd.DataFrame, rules: list) -> dict:
    """Evaluate accuracy via business rule predicates."""
    results = {}
    for rule_name, predicate in rules:
        passing = df.apply(predicate, axis=1).sum()
        results[rule_name] = passing / len(df)
    return results

def freshness_seconds(df: pd.DataFrame, ts_col: str) -> float:
    """Seconds since most recent record timestamp."""
    latest = pd.to_datetime(df[ts_col]).max()
    return (datetime.utcnow() - latest).total_seconds()

def uniqueness_score(df: pd.DataFrame, key_cols: list) -> float:
    """Ratio of unique keys to total rows (1.0 = no duplicates)."""
    unique_keys = df.drop_duplicates(subset=key_cols).shape[0]
    return unique_keys / len(df)

def consistency_check(df_a: pd.DataFrame, df_b: pd.DataFrame,
                       join_key: str, check_col: str) -> float:
    """Fraction of joined rows where check_col matches across sources."""
    merged = df_a.merge(df_b, on=join_key, suffixes=("_a", "_b"))
    matches = (merged[f"{check_col}_a"] == merged[f"{check_col}_b"]).sum()
    return matches / len(merged)

For timeliness, track the delta between the pipeline's scheduled run time and the actual data availability timestamp. Store this as a time-series metric:

def timeliness_metric(scheduled_time: datetime,
                       actual_arrival: datetime) -> dict:
    """Measure SLA adherence for data arrival."""
    delay = (actual_arrival - scheduled_time).total_seconds()
    sla_met = delay <= 0  # arrived on or before schedule
    return {
        "delay_seconds": max(0, delay),
        "sla_met": sla_met,
        "scheduled": scheduled_time.isoformat(),
        "arrived": actual_arrival.isoformat()
    }
Warning: Avoid computing accuracy metrics solely from the dataset itself. Without an external reference or domain constraints, you are measuring internal consistency, not actual accuracy. Always pair statistical outlier detection with business rule validation.

Automated Monitoring

Manual data checks do not scale. Production systems ingest millions of records per hour across hundreds of tables. You need automated monitoring that runs as part of the data pipeline, not as an afterthought.

Pipeline-Integrated Checks

The most reliable approach embeds quality checks directly into your data pipeline as gate stages. If a check fails, the pipeline halts before bad data reaches downstream consumers. This is the "shift-left" philosophy applied to data engineering.

# Great Expectations-style pipeline integration
import great_expectations as gx

def validate_incoming_batch(df: pd.DataFrame, suite_name: str) -> bool:
    """Run expectation suite against incoming data batch."""
    context = gx.get_context()
    suite = context.get_expectation_suite(suite_name)
    validator = context.get_validator(
        batch_request=gx.RuntimeBatchRequest(
            datasource_name="runtime_ds",
            data_connector_name="default_connector",
            data_asset_name="batch",
            runtime_parameters={"batch_data": df},
            batch_identifiers={"run_id": "pipeline_run_001"}
        ),
        expectation_suite=suite
    )

    results = validator.validate()
    success_pct = results.statistics["success_percent"]

    if success_pct < 95.0:
        send_alert(f"Quality gate failed: {success_pct}% pass rate")
        return False
    return True

Statistical Drift Detection

Beyond rule-based checks, monitor for distribution drift. Incoming data should have similar statistical properties to historical baselines. Significant drift indicates an upstream schema change, a broken ETL, or a genuine shift in the data-generating process.

from scipy import stats

def detect_drift(reference: pd.Series, current: pd.Series,
                  threshold: float = 0.05) -> dict:
    """KS test for distribution drift between reference and current."""
    stat, p_value = stats.ks_2samp(reference, current)
    drifted = p_value < threshold

    # Also compute PSI for categorical stability
    def psi(expected, actual, bins=10):
        exp_pct = np.histogram(expected, bins=bins)[0] / len(expected)
        act_pct = np.histogram(actual, bins=bins)[0] / len(actual)
        exp_pct = np.clip(exp_pct, 1e-6, None)
        act_pct = np.clip(act_pct, 1e-6, None)
        return np.sum((act_pct - exp_pct) * np.log(act_pct / exp_pct))

    psi_value = psi(reference.values, current.values)
    return {
        "ks_statistic": stat,
        "p_value": p_value,
        "drifted": drifted,
        "psi": psi_value,
        "psi_alert": psi_value > 0.2  # PSI > 0.2 = significant drift
    }
Best Practice: Run quality checks at three points: (1) immediately after ingestion, (2) after transformation, and (3) before serving to downstream consumers. This "defense in depth" approach catches failures at the layer closest to the root cause.

Alerting Strategies

Monitoring without actionable alerting is just logging. A good alerting strategy minimizes alert fatigue while ensuring genuine data quality incidents are caught before they impact production models.

Static Thresholds

How: Fixed bounds (e.g., null_rate < 5%)

  • Simple to implement and understand
  • Works for metrics with stable baselines
  • Prone to false positives during seasonal shifts
  • Requires manual threshold tuning per metric

Adaptive Thresholds

How: Rolling statistics (mean ± 3σ over 30-day window)

  • Automatically adjusts to seasonal patterns
  • Reduces false positives significantly
  • Requires sufficient history for stable baselines
  • May miss slow, gradual degradation

Alert Severity Tiers

Not all data quality failures are equal. Structure your alerts into tiers so critical failures page on-call engineers immediately while minor anomalies are batched into daily digests.

# Tiered alerting configuration
class AlertSeverity:
    CRITICAL = "critical"  # Pages on-call, blocks pipeline
    WARNING  = "warning"   # Slack notification, continues pipeline
    INFO     = "info"      # Logged to dashboard, daily digest

alert_rules = {
    "completeness_overall": {
        "critical": lambda v: v < 0.80,  # >20% data missing
        "warning":  lambda v: v < 0.95,  # >5% data missing
    },
    "freshness_seconds": {
        "critical": lambda v: v > 7200,  # >2 hours stale
        "warning":  lambda v: v > 3600,  # >1 hour stale
    },
    "uniqueness_score": {
        "critical": lambda v: v < 0.90,  # >10% duplicates
        "warning":  lambda v: v < 0.99,  # >1% duplicates
    },
    "drift_psi": {
        "critical": lambda v: v > 0.25,  # severe distribution shift
        "warning":  lambda v: v > 0.10,  # moderate drift
    },
}

def evaluate_alerts(metrics: dict) -> list:
    """Evaluate all metrics against alert rules and return fired alerts."""
    fired = []
    for metric_name, thresholds in alert_rules.items():
        value = metrics.get(metric_name)
        if value is None:
            continue
        for severity in [AlertSeverity.CRITICAL, AlertSeverity.WARNING]:
            if thresholds[severity](value):
                fired.append({
                    "metric": metric_name,
                    "value": value,
                    "severity": severity,
                })
                break  # highest severity wins
    return fired
Warning: Over-alerting is worse than under-alerting. If your team receives more than 5 data quality alerts per day on average, they will start ignoring them. Tune thresholds aggressively and use anomaly-based detection to reduce noise.

Data Quality SLAs

A Data Quality SLA (Service Level Agreement) formalizes the contract between data producers and consumers. Without SLAs, "data quality" is a vague aspiration. With SLAs, it becomes a measurable commitment with defined consequences for violations.

Defining SLA Components

Every SLA should specify four elements: the metric being measured, the threshold that constitutes compliance, the measurement window, and the escalation path when the SLA is breached.

# Data Quality SLA definition as code
data_quality_slas = {
    "user_events_table": {
        "completeness": {
            "threshold": 0.99,         # 99% non-null rate
            "window": "1h",            # measured per hourly partition
            "escalation": "page_oncall" # immediate page
        },
        "freshness": {
            "threshold_seconds": 1800,  # max 30 min staleness
            "window": "continuous",
            "escalation": "slack_channel"
        },
        "uniqueness": {
            "threshold": 0.999,         # 0.1% max duplicate rate
            "window": "1d",
            "escalation": "daily_report"
        },
        "timeliness": {
            "threshold_seconds": 900,   # data must arrive within 15 min
            "window": "per_batch",
            "escalation": "page_oncall"
        }
    },
    "product_catalog": {
        "accuracy": {
            "threshold": 0.995,
            "window": "1d",
            "escalation": "slack_channel"
        },
        "consistency": {
            "threshold": 0.998,         # cross-system agreement
            "window": "6h",
            "escalation": "page_oncall"
        }
    }
}

SLA Tracking Dashboard

Publish SLA compliance metrics to a central dashboard. Track the SLA compliance rate (percentage of measurement windows where the SLA was met) and the mean time to recovery (MTTR) after SLA breaches. These meta-metrics tell you whether your data quality posture is improving or degrading over time.

  • SLA Compliance Rate: windows_passed / total_windows × 100. Target ≥ 99.5% for critical tables.
  • MTTR: Average time from SLA breach detection to resolution. Target < 30 minutes for critical SLAs.
  • SLA Burn Rate: How fast you are consuming your error budget. A burn rate > 1.0 means you will exhaust the budget before the period ends.
Best Practice: Define SLAs collaboratively between data producers (engineering) and data consumers (ML teams, analysts). One-sided SLAs either set unrealistic targets or leave critical quality gaps uncovered.

Impact on ML Model Performance

Data quality failures do not produce error messages in model training—they produce silently worse models. Understanding how each quality dimension maps to specific model failure modes is essential for building resilient ML systems.

Failure Mode Mapping

  • Completeness → Bias: Missing data is rarely random (MNAR). If records from low-income users are disproportionately missing, the model learns a biased representation. Imputation strategies (mean, median, KNN) introduce their own biases.
  • Accuracy → Noisy Labels: Incorrect labels directly increase the irreducible error. A 5% label noise rate can degrade accuracy by 10–15% on sensitive tasks like fraud detection.
  • Consistency → Feature Skew: When training features come from source A but serving features come from source B, and they disagree, you get training-serving skew—the number one cause of silent production failures.
  • Freshness → Concept Drift: Stale training data means your model learns yesterday's patterns. In fast-moving domains (ads, finance), even a few hours of staleness degrades relevance.
  • Timeliness → Missing Features: Late-arriving data causes features to be computed from incomplete windows, producing systematically different values at training vs. serving time.
  • Uniqueness → Distribution Inflation: Duplicated records over-represent certain patterns, biasing the model toward those examples. In recommendation systems, this causes filter bubble effects.

Quantifying the Impact

# Simulate impact of data quality degradation on model metrics
from sklearn.metrics import accuracy_score, f1_score
import numpy as np

def simulate_label_noise(y_true: np.ndarray,
                          noise_rate: float) -> np.ndarray:
    """Flip a fraction of labels to simulate accuracy degradation."""
    y_noisy = y_true.copy()
    n_flip = int(len(y_true) * noise_rate)
    flip_idx = np.random.choice(len(y_true), n_flip, replace=False)
    unique_labels = np.unique(y_true)
    for idx in flip_idx:
        wrong = [l for l in unique_labels if l != y_true[idx]]
        y_noisy[idx] = np.random.choice(wrong)
    return y_noisy

def measure_degradation(model, X_test, y_test, noise_rates):
    """Train with increasing noise, measure performance drop."""
    results = []
    for rate in noise_rates:
        y_noisy = simulate_label_noise(y_test, rate)
        acc = accuracy_score(y_test, model.predict(X_test))
        f1 = f1_score(y_test, model.predict(X_test), average="macro")
        results.append({"noise_rate": rate, "accuracy": acc, "f1": f1})
    return results

# Typical results:
# noise_rate | accuracy | f1_score
# 0.00       | 0.94     | 0.93
# 0.05       | 0.89     | 0.87
# 0.10       | 0.83     | 0.80
# 0.20       | 0.72     | 0.68
Warning: Training-serving skew caused by consistency failures is notoriously hard to detect. Your model metrics in offline evaluation look fine—the problem only manifests in production. Always compute feature distributions at serving time and compare against training-time distributions.

Frameworks & Tooling

You don't need to build a data quality platform from scratch. The ecosystem offers mature tools, each with different strengths. Choose based on your data stack, scale, and team expertise.

Great Expectations

  • Python-native, integrates with Pandas/Spark/SQL
  • Declarative expectation suites (JSON/YAML)
  • Built-in data docs for human-readable reports
  • Open-source with commercial GX Cloud option
  • Best for: teams with Python-centric pipelines

dbt Tests

  • SQL-native quality checks inside dbt models
  • Schema tests (not_null, unique, accepted_values)
  • Custom tests via SQL queries returning failures
  • Tightly coupled with transformation layer
  • Best for: SQL-first analytics engineering teams

Monte Carlo / Bigeye

  • Automated anomaly detection (no rules to write)
  • ML-based threshold setting and drift detection
  • End-to-end lineage and impact analysis
  • SaaS model, minimal setup required
  • Best for: organizations wanting low-maintenance observability

Soda Core

  • YAML-based check definitions (SodaCL language)
  • Supports SQL databases, Spark, and file formats
  • Built-in anomaly detection and schema evolution
  • Open-source core with Soda Cloud for collaboration
  • Best for: teams wanting config-driven checks across diverse sources
# Soda Core check definition (checks.yml)
checks for user_events:
  - row_count > 0
  - missing_count(user_id) = 0
  - duplicate_count(event_id) = 0
  - freshness(event_timestamp) < 1h
  - avg(session_duration) between 30 and 3600
  - schema:
      fail:
        when required column missing:
          [user_id, event_id, event_timestamp, event_type]
        when wrong column type:
          event_timestamp: timestamp

Production Checklist

Use this checklist when setting up data quality monitoring for a new table or pipeline. Each item maps to a specific dimension and catches a common production failure.

✅ Do

  • Define SLAs for every table consumed by ML models
  • Run quality checks inside the pipeline (not as a separate job)
  • Monitor all six dimensions independently
  • Track SLA compliance rate and MTTR as meta-metrics
  • Alert on distribution drift, not just null rates
  • Version your expectation suites alongside code
  • Compare training-time vs. serving-time feature distributions
  • Log quality metrics to your experiment tracker

❌ Don't

  • Rely on manual spot-checks for production data
  • Use the same thresholds for all tables
  • Ignore freshness because "the pipeline always runs"
  • Alert on every metric individually (aggregate first)
  • Skip quality checks during backfills
  • Assume upstream schema changes will be communicated
  • Treat data quality as a one-time setup task
  • Deploy models without a data quality gate in the pipeline
Final Thought: Data quality is not a feature you ship once—it is a continuous practice. The best ML teams treat data quality metrics with the same rigor as model metrics. When your data quality dashboard is as monitored as your model accuracy dashboard, your ML system is production-grade.
