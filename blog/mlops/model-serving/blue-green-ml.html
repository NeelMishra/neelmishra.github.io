Blue-Green Deployments for ML

MLOps Series Model Serving & Inference

Blue-Green Concept

Blue-green deployment maintains two identical environments — "blue" (current) and "green" (new). The new model is deployed to the idle environment, validated, and traffic is switched instantly. If anything goes wrong, switch back in seconds.

Key advantage: Zero-downtime switching. Unlike canary (gradual rollout), blue-green switches 100% of traffic instantly. The old environment stays running as an instant rollback target.

Architecture Patterns

DNS-Level Switch

  • Change DNS to point at green env
  • Simplest to implement
  • ❌ DNS propagation delay (minutes)
  • ❌ Not suitable for instant rollback

Load Balancer Switch

  • Change target group in ALB/NLB
  • Instant traffic switch
  • ✅ Sub-second rollback
  • ✅ Health check before switch

Blue-Green for ML Models

# ML-specific blue-green deployment script
class BlueGreenDeployer:
    def deploy_new_model(self, model_uri, model_version):
        # 1. Deploy to inactive environment
        inactive = self.get_inactive_env()
        self.deploy_model(inactive, model_uri)

        # 2. Wait for model to load and warm up
        self.wait_for_ready(inactive, timeout=300)
        self.run_warmup_requests(inactive, n=100)

        # 3. Run validation suite against green
        results = self.run_validation(inactive, test_suite)
        if not results.passed:
            self.alert(f"Validation failed: {results.failures}")
            return False

        # 4. Compare prediction distributions
        drift = self.compare_predictions(active_env, inactive, sample_data)
        if drift.psi > 0.2:
            self.alert(f"High prediction drift: PSI={drift.psi:.3f}")
            return False

        # 5. Switch traffic
        self.switch_traffic(to=inactive)
        self.log_deployment(model_version, inactive.name)
        return True

The Switch Mechanism

# AWS ALB target group swap
import boto3

elbv2 = boto3.client("elbv2")

def switch_traffic(listener_arn, new_target_group_arn):
    elbv2.modify_listener(
        ListenerArn=listener_arn,
        DefaultActions=[{
            "Type": "forward",
            "TargetGroupArn": new_target_group_arn
        }]
    )

# Kubernetes service swap
def k8s_switch(service_name, new_selector):
    # Update service selector to point at green deployment
    kubectl patch svc {service_name} \
      -p '{{"spec":{{"selector":{{"version":"green"}}}}}}'

Rollback Strategy

# Automated rollback with health monitoring
def monitor_after_switch(deployer, duration_minutes=30):
    for minute in range(duration_minutes):
        metrics = deployer.get_current_metrics()

        if metrics.error_rate > 0.02:  # >2% errors
            deployer.rollback()
            deployer.alert("ROLLBACK: High error rate")
            return "rolled_back"

        if metrics.p99_latency > 200:  # >200ms
            deployer.rollback()
            deployer.alert("ROLLBACK: High latency")
            return "rolled_back"

        time.sleep(60)

    # All clear — decommission old env
    deployer.decommission_old_env()
    return "success"
Cost warning: Blue-green requires 2× the infrastructure during deployment (both environments running). For GPU-intensive ML models, this can be expensive. Consider running the standby environment on cheaper spot instances or scale it down when idle.

Blue-Green vs Canary

Blue-Green

  • ✅ Zero-downtime, instant switch
  • ✅ Simple rollback
  • ✅ Full pre-deployment validation
  • ❌ 2× infrastructure cost
  • ❌ No gradual exposure
  • Best for: batch model updates, high-confidence deployments

Canary

  • ✅ Gradual risk reduction
  • ✅ Statistical validation at each step
  • ✅ Lower infrastructure overhead
  • ❌ Slower rollout
  • ❌ More complex monitoring
  • Best for: user-facing models, high-uncertainty changes
Combined approach: Deploy via blue-green (instant switch after validation), then run a canary analysis (comparing metrics post-switch with pre-switch baseline). Rollback instantly if canary analysis fails.
