Quantization Fundamentals

Quantization reduces the numerical precision of model weights and activations from 32-bit floating point to lower bit-widths (INT8, INT4, or even binary). This shrinks model size, reduces memory bandwidth, and accelerates inference — often with minimal accuracy loss.

Why quantize? Memory bandwidth is the bottleneck for most inference workloads. Smaller numbers move faster through memory → faster inference. INT8 operations are also 2-4× faster than FP32 on modern GPUs.

Precision Types: FP32 to INT4

FP32 (32-bit float)

  • Training default
  • 8 bits exponent, 23 bits mantissa
  • Full dynamic range
  • Baseline for accuracy comparison

FP16 / BF16 (16-bit)

  • 2× speedup, 2× less memory
  • BF16: same exponent range as FP32
  • Negligible accuracy loss
  • Native on A100, H100, TPUs

INT8 (8-bit integer)

  • 4× smaller, 2-4× faster
  • Requires calibration data
  • 0.1-0.5% typical accuracy drop
  • Best cost/accuracy trade-off

INT4 / NF4 (4-bit)

  • 8× smaller than FP32
  • Critical for LLM deployment
  • GPTQ, AWQ, GGML formats
  • 1-3% accuracy drop typical

Post-Training Quantization (PTQ)

PTQ quantizes a pre-trained model without retraining. It's the fastest path to a quantized model — just provide calibration data and run the quantizer.

# PyTorch dynamic quantization (no calibration needed)
import torch

model = torch.load("model.pt")
quantized = torch.quantization.quantize_dynamic(
    model,
    {torch.nn.Linear},  # Layers to quantize
    dtype=torch.qint8
)
torch.save(quantized, "model_int8.pt")
# PyTorch static quantization (with calibration)
import torch

model = torch.load("model.pt")
model.eval()

# Prepare for quantization
model.qconfig = torch.quantization.get_default_qconfig("x86")
prepared = torch.quantization.prepare(model)

# Run calibration (representative data)
with torch.no_grad():
    for batch in calibration_loader:
        prepared(batch)

# Convert to quantized model
quantized = torch.quantization.convert(prepared)

LLM Quantization (GPTQ / AWQ)

# GPTQ quantization with auto-gptq
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig

config = BaseQuantizeConfig(
    bits=4,
    group_size=128,
    desc_act=False
)

model = AutoGPTQForCausalLM.from_pretrained("meta-llama/Llama-2-7b", config)
model.quantize(calibration_dataset)
model.save_quantized("llama-2-7b-gptq")

Quantization-Aware Training (QAT)

QAT simulates quantization during training, letting the model learn to compensate for quantization error. It produces better accuracy than PTQ, especially at lower bit-widths.

# QAT with PyTorch
model.train()
model.qconfig = torch.quantization.get_default_qat_qconfig("x86")
prepared = torch.quantization.prepare_qat(model)

# Train with fake quantization nodes
for epoch in range(fine_tune_epochs):
    for batch, labels in train_loader:
        output = prepared(batch)
        loss = criterion(output, labels)
        loss.backward()
        optimizer.step()

# Convert to actual quantized model
prepared.eval()
quantized = torch.quantization.convert(prepared)
When to use QAT vs PTQ: Use PTQ first — it's fast and often sufficient. Switch to QAT only if PTQ accuracy drop exceeds your tolerance (typically > 1%). QAT requires 10-20% of original training time.

Mixed Precision Inference

Mixed precision keeps sensitive layers in higher precision (FP16/FP32) while quantizing the rest. This preserves accuracy in critical paths while still gaining speedup.

# Mixed precision with TensorRT
import tensorrt as trt

builder = trt.Builder(logger)
config = builder.create_builder_config()

# Enable INT8 with FP16 fallback
config.set_flag(trt.BuilderFlag.INT8)
config.set_flag(trt.BuilderFlag.FP16)

# Set specific layers to higher precision
for layer in network:
    if layer.name in sensitive_layers:
        layer.precision = trt.float16
    else:
        layer.precision = trt.int8

Quantization Tools

Framework Tools

  • PyTorch: torch.quantization, torch.ao
  • TensorFlow: TF Lite converter
  • NVIDIA: TensorRT quantization
  • Intel: Neural Compressor

LLM-Specific

  • GPTQ: 4-bit weight quantization
  • AWQ: Activation-aware quantization
  • GGML/GGUF: CPU-friendly quantization
  • bitsandbytes: 8/4-bit for HuggingFace

Best Practices

  • ✅ Always benchmark accuracy on a held-out test set after quantization
  • ✅ Use representative calibration data (100-500 samples is usually enough)
  • ✅ Start with INT8 PTQ — it works for 90% of models
  • ✅ Keep first and last layers in FP16 for better accuracy
  • ✅ Monitor inference outputs for distributional shift post-quantization
Production impact: Quantization is the single highest-ROI optimization for inference cost. A properly quantized INT8 model typically runs 2-4× faster, uses 4× less memory, and costs 2-4× less to serve — with negligible accuracy loss.
