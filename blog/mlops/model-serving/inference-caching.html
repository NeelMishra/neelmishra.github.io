Inference Caching: Redis, Embeddings, Semantic Dedup

MLOps Series Model Serving & Inference

Why Cache Inference?

ML inference is expensive — GPU compute, memory, and energy. Many production workloads have high input repetition (search queries, product recommendations, embeddings). Caching predictions eliminates redundant computation and dramatically reduces latency and cost.

Real-world impact: At scale, 30-70% of inference requests can be cache hits. An embedding service with 50% hit rate cuts GPU cost in half and reduces p50 latency from 20ms to <1ms for cached requests.

Exact Match Caching

The simplest caching strategy: hash the input, check if we've seen it before, return the cached prediction. Works when inputs repeat exactly.

# Redis-based exact match cache
import redis, json, hashlib

cache = redis.Redis(host="redis", port=6379, db=0)
TTL = 3600  # 1 hour cache

def cached_predict(model, features):
    # Create deterministic cache key
    key = "pred:" + hashlib.sha256(
        json.dumps(features, sort_keys=True).encode()
    ).hexdigest()

    # Check cache
    cached = cache.get(key)
    if cached:
        return json.loads(cached)  # Cache hit!

    # Cache miss — run inference
    prediction = model.predict(features)
    cache.setex(key, TTL, json.dumps(prediction))
    return prediction

✅ Good for Exact Cache

  • Product embeddings (finite catalog)
  • Categorical feature predictions
  • Repeated search queries
  • Static content classification

❌ Poor for Exact Cache

  • Continuous numerical features
  • High-cardinality user features
  • Time-sensitive predictions
  • Unique image/audio inputs

Semantic Caching

Semantic caching returns cached results for inputs that are semantically similar — not just identical. This dramatically improves hit rates for text-based models.

# Semantic cache using embedding similarity
import numpy as np
from redis.commands.search.field import VectorField

class SemanticCache:
    def __init__(self, embedder, threshold=0.95):
        self.embedder = embedder
        self.threshold = threshold
        self.cache = {}  # embedding → prediction
        self.embeddings = []  # for similarity search

    def get(self, query):
        query_emb = self.embedder.encode(query)

        # Find nearest cached embedding
        if self.embeddings:
            similarities = np.dot(self.embeddings, query_emb)
            best_idx = np.argmax(similarities)
            if similarities[best_idx] >= self.threshold:
                return self.cache[best_idx]  # Semantic hit!

        return None  # Cache miss

    def put(self, query, prediction):
        emb = self.embedder.encode(query)
        idx = len(self.embeddings)
        self.embeddings.append(emb)
        self.cache[idx] = prediction
Threshold tuning: Too low (0.85) returns irrelevant cached results → bad predictions. Too high (0.99) rarely matches → low hit rate. Start at 0.95 and validate by comparing cached vs fresh predictions on a sample.

Embedding Cache

Embedding computation is one of the most cacheable ML operations — the same text/image always maps to the same vector.

# Multi-layer embedding cache
class EmbeddingCache:
    def __init__(self, model, redis_client, local_lru_size=10000):
        self.model = model
        self.redis = redis_client
        self.local_cache = LRUCache(maxsize=local_lru_size)

    def encode(self, text):
        # Layer 1: In-process LRU cache (~0.001ms)
        if text in self.local_cache:
            return self.local_cache[text]

        # Layer 2: Redis cache (~1ms)
        key = f"emb:{hash(text)}"
        cached = self.redis.get(key)
        if cached:
            emb = np.frombuffer(cached, dtype=np.float32)
            self.local_cache[text] = emb
            return emb

        # Layer 3: Model inference (~20ms)
        emb = self.model.encode(text)
        self.redis.setex(key, 86400, emb.tobytes())
        self.local_cache[text] = emb
        return emb

Cache Invalidation

# Cache invalidation strategies for ML

# 1. TTL-based (simplest)
cache.setex(key, TTL_SECONDS, prediction)  # Auto-expire

# 2. Model version-based (invalidate on model update)
def make_key(model_version, features):
    return f"v{model_version}:{hash(features)}"
# New model version → new keys → old cache ignored

# 3. Feature staleness (for time-sensitive features)
def is_stale(cached_entry, max_age_seconds=300):
    return time.time() - cached_entry.timestamp > max_age_seconds

# 4. Active invalidation (model retrain triggers flush)
def on_model_deploy(new_model_version):
    cache.flushdb()  # Nuclear option
    # Or selective: delete keys matching old version
Stale prediction risk: Cached predictions may become incorrect when the underlying data changes (user preferences, market conditions). Set TTLs based on feature volatility — seconds for real-time features, hours for stable features.

Cache hierarchy: L1 (in-process LRU, ~0.001ms) → L2 (Redis, ~1ms) → L3 (model inference, ~20-100ms). This three-layer approach handles hot items locally while distributing the long tail across Redis.
