Model Mesh: Serving Thousands of Models

MLOps Series Model Serving & Inference

The Thousands-of-Models Problem

Most ML platforms serve not one model but thousands — per-customer models, per-region variants, per-product classifiers. Traditional serving (one pod per model) doesn't scale: 10,000 models × 1 pod each = 10,000 pods. ModelMesh solves this with intelligent model packing.

Traditional: 1 Pod per Model

  • 10,000 models = 10,000 pods
  • Most pods idle (low traffic)
  • Massive resource waste
  • ~$500K/month at scale

ModelMesh: Shared Pods

  • 10,000 models on ~50 pods
  • Models loaded/unloaded on demand
  • LRU eviction for memory
  • ~$10K/month — 50× cheaper

ModelMesh Architecture

ModelMesh is a distributed model cache built by IBM, now part of KServe. It intelligently distributes models across a pool of serving pods, loading frequently-used models into memory and evicting idle ones.

How it works: ModelMesh maintains a distributed model-to-pod mapping. When a request arrives for a model, the router checks if it's loaded. If loaded, routes to the right pod. If not, triggers loading (from S3/GCS) and evicts an LRU model to make room.

Model Loading & Unloading

# ModelMesh loading strategies (conceptual)
class ModelMeshScheduler:
    def route_request(self, model_id):
        # Check if model is loaded somewhere
        pod = self.model_to_pod.get(model_id)

        if pod and pod.is_healthy:
            return pod  # Fast path: model already loaded

        # Model not loaded — find best pod
        target_pod = self.find_best_pod(model_id)

        # Load model (may need to evict)
        if target_pod.memory_available < model_size:
            victim = target_pod.lru_model()
            target_pod.unload(victim)
        target_pod.load(model_id)

        self.model_to_pod[model_id] = target_pod
        return target_pod

    def find_best_pod(self, model_id):
        # Prefer pods with: most free memory, fewest loads in progress
        return min(self.pods, key=lambda p:
            p.loading_count * 10 - p.free_memory_mb
        )

Memory Management

# Memory budgets per pod
pod_memory: 16Gi
# Each model: ~50-500MB loaded
# Fits: 30-300 models per pod depending on size

# LRU eviction with priority classes
class EvictionPolicy:
    def select_victim(self, loaded_models):
        # Never evict priority models (always loaded)
        candidates = [m for m in loaded_models if not m.pinned]

        # Evict least recently used, considering:
        # - Last access time
        # - Model size (prefer evicting large models)
        # - Load cost (prefer evicting cheap-to-reload models)
        return min(candidates, key=lambda m:
            m.last_access * 0.7 + m.load_time_ms * 0.3
        )

KServe Integration

# Deploy ModelMesh with KServe
# 1. Install ModelMesh serving
kubectl apply -f modelmesh-serving.yaml

# 2. Create a ServingRuntime (model server type)
apiVersion: serving.kserve.io/v1alpha1
kind: ServingRuntime
metadata:
  name: sklearn-runtime
spec:
  supportedModelFormats:
  - name: sklearn
    version: "1"
    autoSelect: true
  multiModel: true
  containers:
  - name: mlserver
    image: seldonio/mlserver:latest
    resources:
      limits:
        memory: "4Gi"

# 3. Register models (lightweight — just metadata)
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: customer-model-42
  annotations:
    serving.kserve.io/deploymentMode: ModelMesh
spec:
  predictor:
    model:
      modelFormat: { name: sklearn }
      storageUri: "s3://models/customer-42/v1"
Cold load latency: The first request to an unloaded model will be slow (seconds) while the model is fetched from object storage and loaded. For latency-sensitive models, pin them in memory or use pre-warming.

When to Use ModelMesh

  • Multi-tenant platforms: Per-customer model customization
  • Large model catalogs: Hundreds to thousands of models
  • Long-tail traffic: Most models get infrequent requests
  • Cost optimization: Can't afford 1 pod per model
Not for: If you have fewer than 50 models or all models get consistent high traffic, standard per-model deployments with autoscaling are simpler and provide better isolation.
