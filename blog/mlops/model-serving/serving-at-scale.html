Serving at Scale: Google, Meta, Netflix

MLOps Series Model Serving & Inference

Google: TFX & Vertex AI

Google serves ML at a scale few organizations approach — billions of predictions per second across Search, Ads, YouTube, Gmail, Maps, and more. Their approach emphasizes standardization and custom hardware.

Architecture Highlights

  • TF Serving — battle-tested since 2016
  • TPU inference — custom hardware for large models
  • Spanner ML — in-database predictions
  • Pathways — multi-model orchestration

Scale Numbers

  • Billions of predictions/second
  • Thousands of models in production
  • p99 latency < 10ms for Search ranking
  • Custom TPU pods for inference
# Google's serving philosophy (simplified)
class GoogleServingPrinciples:
    # 1. Standardize the serving API
    protocol = "TF Serving API"  # One API for all models

    # 2. Custom hardware for cost efficiency
    default_hardware = "TPU v4"  # 2-3× cheaper than GPU for their workloads

    # 3. Batch + online hybrid
    # Pre-compute what you can, real-time for the rest
    serving_pattern = "pre-scored candidates + real-time re-ranking"

    # 4. Continuous evaluation
    # Every model has online metrics compared to holdout
    evaluation = "always-on A/B with holdout groups"
Google's key insight: Pre-compute everything possible offline. Their search ranking uses batch-computed candidate scores that are re-ranked in real-time. This hybrid approach handles massive scale while keeping online latency low.

Meta: PyTorch Ecosystem

Meta serves ML across Facebook, Instagram, WhatsApp, and their ads platform. They built PyTorch and invest heavily in inference optimization.

# Meta's inference stack
class MetaServingStack:
    framework = "PyTorch"
    serving = "Caffe2 / TorchServe / custom"
    optimization = [
        "torch.compile with Inductor",
        "INT8 quantization everywhere",
        "Model pruning (50-80% sparsity)",
        "FBGEMM for CPU inference",
    ]
    hardware = "Custom MTIA chips + NVIDIA GPUs"

    # Recommendation model specifics
    embedding_tables = "Terabytes of embeddings"  # Distributed across memory
    batch_sizes = "Thousands"  # Massive batch inference for Ads

    # Key principle: CPU for cheap, GPU for big
    # Small models (fraud, spam) → CPU clusters
    # Large models (ranking, generative) → GPU/MTIA

Meta's Innovations

  • DLRM — Deep Learning Recommendation Model architecture
  • FBGEMM — Optimized CPU inference kernels
  • MTIA — Custom inference accelerator
  • torch.compile — Compiler-driven optimization

Lessons from Meta

  • Embedding tables are the bottleneck for recommenders
  • CPU inference is underrated for smaller models
  • Quantization is non-negotiable at scale
  • Custom silicon pays off at extreme scale

Netflix: Recommendations

Netflix serves personalized recommendations to 200M+ members. Their ML serving architecture is a masterclass in hybrid batch-online patterns.

# Netflix recommendation serving (simplified)
class NetflixRecommendations:
    # Three-layer architecture

    # Layer 1: Offline (nightly batch)
    offline_models = [
        "candidate_generation",    # ~1000 candidates per member
        "collaborative_filtering", # User-item similarity
        "content_embeddings",      # Title representations
    ]
    offline_store = "Cassandra"  # Pre-computed results

    # Layer 2: Nearline (streaming, minutes-fresh)
    nearline_models = [
        "session_personalization",  # Recent viewing context
        "trending_signals",         # What's popular now
    ]
    nearline_compute = "Apache Flink"

    # Layer 3: Online (real-time, per-request)
    online_models = [
        "ranking",         # Final re-ranking of candidates
        "artwork_selection",# Personalized title images
        "row_selection",    # Which rows to show
    ]
    online_latency = "p99 < 150ms for full page"
Netflix's secret: They don't try to run everything in real-time. Candidate generation (expensive) runs nightly. Session context (fast-changing) runs nearline. Final ranking (latency-sensitive) runs online. Each layer uses the right serving pattern for its requirements.

Common Patterns Across All Three

  • Hybrid serving: Batch for heavy compute, online for real-time personalization
  • Feature stores: Centralized feature computation shared across models
  • Aggressive caching: Pre-computed embeddings, candidate scores, feature vectors
  • Quantization everywhere: INT8 is the default for production, not FP32
  • Continuous evaluation: Always-on A/B testing with holdout groups
  • Custom infrastructure: At extreme scale, custom hardware/software pays off

Lessons Learned

✅ What Works

  • Pre-compute everything you can
  • Use the simplest serving pattern that meets SLAs
  • Invest in monitoring before scaling
  • Standardize on one serving protocol
  • Automate model deployment end-to-end

❌ Common Mistakes

  • Over-engineering serving before product-market fit
  • Ignoring preprocessing/feature costs
  • No rollback plan for new models
  • Optimizing inference while ignoring network latency
  • Building custom when open-source works

Practical Takeaways

# Scaling maturity levels
def choose_architecture(scale):
    if scale.qps < 100:
        return "Single FastAPI server + Redis cache"

    if scale.qps < 10_000:
        return "TorchServe/BentoML + K8s HPA + feature store"

    if scale.qps < 100_000:
        return "KServe + Triton + batch/online hybrid"

    if scale.qps < 1_000_000:
        return "Custom serving + extensive pre-computation + caching"

    return "You're Google/Meta. Build custom everything."
The most important lesson: Start simple. Every company in this post started with simple serving and evolved complexity as scale demanded it. Netflix's first recommendation system was a simple batch pipeline. Google's first serving was a single binary. Complexity should be earned, not assumed.
Don't copy Big Tech architecture: Google's TPU clusters, Meta's custom silicon, and Netflix's multi-layer serving were built for their specific scale and economics. At 1% of their scale, simpler architectures with off-the-shelf tools are both more cost-effective and easier to maintain.
