Canary Deployment Concept

A canary deployment gradually shifts traffic from the current model to a new version, monitoring metrics at each step. If metrics degrade, traffic automatically rolls back. This is safer than big-bang deployments for ML models where prediction quality is hard to verify offline.

Why canary for ML? Unlike software bugs, model regressions may be subtle — slightly worse recommendations, marginal increase in false positives. Canary deployments detect these statistical degradations before they affect all users.

Traffic Splitting

# Canary traffic splitting strategies

# 1. Random percentage split
def random_split(request, canary_pct):
    return "canary" if random() < canary_pct else "stable"

# 2. Sticky user assignment (preferred for ML)
def sticky_split(user_id, canary_pct, experiment_salt):
    bucket = hash(f"{user_id}:{experiment_salt}") % 10000
    return "canary" if bucket < canary_pct * 100 else "stable"

# 3. Feature-based routing (test on specific segments)
def segment_split(request, canary_segments):
    if request.region in canary_segments:
        return "canary"
    return "stable"

Metric-Based Promotion

# Automated canary promotion logic
class CanaryPromoter:
    stages = [0.01, 0.05, 0.10, 0.25, 0.50, 1.0]

    def evaluate_stage(self, metrics):
        checks = {
            "error_rate": metrics.canary_error <= metrics.stable_error * 1.05,
            "latency_p99": metrics.canary_p99 <= metrics.stable_p99 * 1.10,
            "prediction_drift": metrics.psi_score < 0.1,
            "business_metric": metrics.canary_ctr >= metrics.stable_ctr * 0.98,
        }

        if all(checks.values()):
            return "PROMOTE"
        elif any(not v for k, v in checks.items()
                 if k in ["error_rate", "latency_p99"]):
            return "ROLLBACK"
        return "HOLD"  # Need more data

Automated Rollback

# Rollback triggers
rollback_conditions:
  - metric: error_rate_5xx
    threshold: 0.01       # >1% errors
    window: 5m
    action: immediate_rollback

  - metric: p99_latency_ms
    threshold: 200        # >200ms p99
    window: 10m
    action: immediate_rollback

  - metric: prediction_distribution_psi
    threshold: 0.2        # Significant distribution shift
    window: 30m
    action: pause_and_alert

  - metric: business_conversion
    threshold: -0.05      # >5% drop in conversions
    window: 60m
    action: gradual_rollback
Fast rollback is critical: Your rollback should take seconds, not minutes. Pre-warm the stable model, keep it running alongside the canary, and use traffic routing (not pod replacement) for instant switches.

Implementation with Istio

# Istio VirtualService for canary traffic splitting
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: model-canary
spec:
  hosts: ["model.ml.internal"]
  http:
  - route:
    - destination:
        host: model-stable
        port: { number: 8080 }
      weight: 90
    - destination:
        host: model-canary
        port: { number: 8080 }
      weight: 10
# Argo Rollouts for progressive delivery
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: model-rollout
spec:
  strategy:
    canary:
      steps:
      - setWeight: 5
      - pause: { duration: 30m }
      - setWeight: 20
      - pause: { duration: 30m }
      - setWeight: 50
      - pause: { duration: 1h }
      analysis:
        templates:
        - templateName: model-quality-check
        args:
        - name: service-name
          value: model-canary

ML-Specific Considerations

  • Prediction distribution monitoring: Track PSI/KL divergence between canary and stable predictions
  • Delayed feedback: For models where ground truth arrives later (fraud, churn), use proxy metrics initially
  • Feature store consistency: Ensure both models read from the same feature store version
  • Warm-up period: New models may have cold caches — exclude the first few minutes from analysis
  • Segment analysis: Check canary performance across user segments, not just overall
Canary vs A/B testing: Canary is a deployment safety mechanism (detect breakage). A/B testing is a scientific experiment (measure improvement). Use canary for every deployment; A/B test only when measuring business impact.
