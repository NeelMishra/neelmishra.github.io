ML API Design Principles

ML APIs differ from typical CRUD APIs. The request carries high-dimensional input data, the response includes probabilistic outputs, and latency budgets are often strict. Good API design makes your model accessible, debuggable, and evolvable.

Core principles: (1) Make inputs and outputs explicit and typed. (2) Include model metadata in responses. (3) Support both single and batch requests. (4) Return confidence scores, not just labels. (5) Version everything.

REST API Design

REST is the default choice for most ML APIs — universal client support, easy debugging with curl, and familiar to every developer.

# Well-designed ML REST API with FastAPI
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field
from typing import List, Optional
import time

app = FastAPI(title="Fraud Detection API", version="2.0")

class PredictRequest(BaseModel):
    features: List[float] = Field(..., min_length=10, max_length=10)
    request_id: Optional[str] = None

class PredictResponse(BaseModel):
    prediction: str
    confidence: float
    model_version: str
    latency_ms: float
    request_id: Optional[str]

@app.post("/v2/predict", response_model=PredictResponse)
async def predict(req: PredictRequest):
    start = time.perf_counter()
    score = model.predict([req.features])[0]
    elapsed = (time.perf_counter() - start) * 1000

    return PredictResponse(
        prediction="fraud" if score > 0.5 else "legitimate",
        confidence=round(score, 4),
        model_version="2.1.0",
        latency_ms=round(elapsed, 2),
        request_id=req.request_id
    )

gRPC for ML

gRPC excels for service-to-service ML calls where latency matters. Binary serialization (Protocol Buffers) is 2-10× faster than JSON for large tensor payloads.

REST Advantages

  • Universal browser support
  • Easy debugging (curl, Postman)
  • Human-readable JSON
  • Simpler client integration

gRPC Advantages

  • 2-5× lower latency
  • Strict schema enforcement
  • Bidirectional streaming
  • Auto-generated clients
# inference.proto — gRPC schema for ML
syntax = "proto3";
package ml.inference;

service InferenceService {
  rpc Predict(PredictRequest) returns (PredictResponse);
  rpc StreamPredict(stream PredictRequest) returns (stream PredictResponse);
  rpc BatchPredict(BatchPredictRequest) returns (BatchPredictResponse);
}

message PredictRequest {
  string model_name = 1;
  string model_version = 2;
  repeated float features = 3;
  string request_id = 4;
}

message PredictResponse {
  string prediction = 1;
  float confidence = 2;
  string model_version = 3;
  float latency_ms = 4;
}

GraphQL for ML

GraphQL makes sense when clients need different subsets of model output — some want just the label, others need full probability distributions, feature importances, or explanations.

# GraphQL schema for ML API
type Prediction {
  label: String!
  confidence: Float!
  probabilities: [ClassProbability!]
  explanation: Explanation
  modelVersion: String!
  latencyMs: Float!
}

type ClassProbability {
  className: String!
  probability: Float!
}

type Explanation {
  featureImportances: [FeatureScore!]
  method: String!
}

type Query {
  predict(features: [Float!]!): Prediction!
}

# Client queries only what they need:
# query { predict(features: [1.0, 2.0]) { label confidence } }
# query { predict(features: [1.0, 2.0]) { label explanation { featureImportances { ... } } } }

API Versioning

# Strategy 1: URL versioning (most common)
POST /v1/predict   # Old model, old schema
POST /v2/predict   # New model, new schema

# Strategy 2: Header versioning
POST /predict
X-Model-Version: 2.1.0

# Strategy 3: Model name in URL (KServe/Triton style)
POST /v2/models/fraud-detector/versions/3/infer

# Strategy 4: Query parameter
POST /predict?model_version=2.1.0
Breaking changes: Changing input/output schema requires a new API version. Changing the model (same schema, better predictions) does NOT — use model versioning instead.

  "error_code": "INPUT_OUT_OF_DISTRIBUTION",
  "message": "Feature 'age' value -5 is outside expected range [0, 120]",
  "prediction": null  # Don't return a prediction for bad input
}

Batch Endpoints

# Synchronous batch endpoint
@app.post("/v2/batch-predict")
async def batch_predict(requests: List[PredictRequest]):
    if len(requests) > 1000:
        raise HTTPException(413, "Batch size exceeds 1000")
    results = model.predict_batch([r.features for r in requests])
    return [{"prediction": r} for r in results]

# Async batch endpoint (for large batches)
@app.post("/v2/async-batch")
async def async_batch(requests: List[PredictRequest]):
    job_id = submit_batch_job(requests)
    return {"job_id": job_id, "status_url": f"/v2/jobs/{job_id}"}

@app.get("/v2/jobs/{job_id}")
async def get_job_status(job_id: str):
    job = get_job(job_id)
    return {"status": job.status, "results": job.results if job.done else None}
