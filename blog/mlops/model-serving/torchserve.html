TorchServe: Serving PyTorch in Production

MLOps Series Model Serving & Inference

TorchServe Architecture

TorchServe is the official model serving framework for PyTorch, developed jointly by AWS and Meta. It provides a lightweight, scalable serving solution with out-of-the-box support for multi-model serving, model versioning, and A/B testing.

Key design: TorchServe uses separate Java frontend (Netty-based) for HTTP handling and Python backend workers for inference. This separation lets the frontend handle thousands of connections while workers focus on compute.

Model Archiver (MAR)

The Model Archive (.mar) is TorchServe's packaging format. It bundles the model weights, handler code, and any extra files into a single deployable artifact.

# Create a MAR archive
torch-model-archiver \
  --model-name resnet50 \
  --version 1.0 \
  --serialized-file model_weights.pt \
  --handler image_classifier \
  --extra-files index_to_name.json \
  --export-path model_store/

# Directory structure inside .mar:
#   resnet50/
#   ├── MAR-INF/
#   │   └── MANIFEST.json
#   ├── model_weights.pt
#   ├── image_classifier.py
#   └── index_to_name.json

Custom Handlers

Handlers define the full inference pipeline: preprocessing, inference, and postprocessing. TorchServe provides built-in handlers for common tasks and lets you write custom ones.

Built-in Handlers

  • image_classifier
  • image_segmenter
  • object_detector
  • text_classifier

Custom Handler Methods

  • initialize() — load model
  • preprocess() — transform input
  • inference() — run model
  • postprocess() — format output
# Custom TorchServe handler
from ts.torch_handler.base_handler import BaseHandler
import torch, json

class FraudHandler(BaseHandler):
    def initialize(self, context):
        self.manifest = context.manifest
        model_dir = context.system_properties["model_dir"]
        self.model = torch.jit.load(f"{model_dir}/model.pt")
        self.model.eval()
        self.device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
        self.model.to(self.device)

    def preprocess(self, data):
        inputs = []
        for row in data:
            body = row.get("body") or row
            features = json.loads(body)
            tensor = torch.tensor(features["values"], dtype=torch.float32)
            inputs.append(tensor)
        return torch.stack(inputs).to(self.device)

    def inference(self, inputs):
        with torch.no_grad():
            outputs = self.model(inputs)
        return outputs

    def postprocess(self, outputs):
        probs = torch.sigmoid(outputs).tolist()
        return [{"fraud_probability": p[0]} for p in probs]

Server Configuration

TorchServe is configured via a config.properties file. Key settings control worker count, batching, timeouts, and device assignment.

# config.properties
inference_address=http://0.0.0.0:8080
management_address=http://0.0.0.0:8081
metrics_address=http://0.0.0.0:8082

# Model loading
model_store=model_store
load_models=all

# Worker configuration
default_workers_per_model=4
job_queue_size=1000

# Batching — key for GPU utilization
batch_size=8
max_batch_delay=100  # ms to wait for batch to fill

# Timeouts
default_response_timeout=120

# GPU pinning
number_of_gpu=2
Batching trade-off: Higher max_batch_delay means better GPU utilization but higher p50 latency. For real-time serving, keep it under 50ms. For throughput-optimized workloads, 200-500ms is acceptable.

Metrics & Monitoring

TorchServe exposes metrics in Prometheus format on port 8082. Key metrics to monitor include:

System Metrics

  • ts_inference_requests_total
  • ts_queue_latency_microseconds
  • ts_inference_latency_microseconds
  • GPUUtilization
  • GPUMemoryUtilization

Custom Metrics

  • Add via metrics.yaml
  • Counter, gauge, histogram types
  • Per-model metric dimensions
  • Integrate with CloudWatch / Datadog
# Custom metrics in handler
from ts.metrics.metric import Metric

class MyHandler(BaseHandler):
    def postprocess(self, outputs):
        # Track prediction distribution
        for output in outputs:
            if output > 0.5:
                self.context.metrics.add_counter(
                    "high_confidence_predictions", 1
                )
        return outputs.tolist()

Advanced Features

Model Versioning

# Register multiple versions
curl -X POST "http://localhost:8081/models?url=resnet_v1.mar&model_name=resnet"
curl -X POST "http://localhost:8081/models?url=resnet_v2.mar&model_name=resnet"

# Set default version
curl -X PUT "http://localhost:8081/models/resnet/2.0/set-default"

# Query specific version
curl "http://localhost:8080/predictions/resnet/1.0" -d @input.json

TorchScript & torch.compile

# Export with TorchScript for production
model = MyModel()
scripted = torch.jit.script(model)
scripted.save("model.pt")

# Or use torch.compile for PyTorch 2.x
compiled_model = torch.compile(model, backend="inductor")
# TorchServe supports compiled models natively

Production Deployment

Docker Deployment

# Dockerfile for TorchServe
FROM pytorch/torchserve:latest-gpu

COPY model_store /home/model-server/model-store
COPY config.properties /home/model-server/config.properties

CMD ["torchserve", \
     "--start", \
     "--model-store", "/home/model-server/model-store", \
     "--ts-config", "/home/model-server/config.properties"]

Kubernetes with HPA

# k8s deployment excerpt
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: torchserve
        image: myrepo/torchserve:v1
        resources:
          limits:
            nvidia.com/gpu: 1
        ports:
        - containerPort: 8080
        - containerPort: 8081
        readinessProbe:
          httpGet:
            path: /ping
            port: 8080
          initialDelaySeconds: 30
Production checklist: Enable gRPC for lower latency, configure dynamic batching, set up Prometheus scraping, use readiness probes for rolling updates, and pin GPU memory to avoid OOM kills.
