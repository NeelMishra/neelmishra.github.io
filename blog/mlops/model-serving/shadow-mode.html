Shadow Mode: Validate Before You Ship

MLOps Series Model Serving & Inference

Shadow Mode Concept

Shadow mode (also called dark launching or traffic mirroring) runs a new model alongside the production model. Both receive the same requests, but only the production model's predictions are served. The shadow model's predictions are logged and compared — zero risk to users.

Zero-risk validation: Shadow mode is the safest deployment pattern because shadow predictions never reach users. It's ideal for high-stakes models (fraud, healthcare, autonomous driving) where even a brief canary exposure is risky.

Shadow Architecture

# Shadow mode proxy
import asyncio, httpx

class ShadowProxy:
    def __init__(self, production_url, shadow_url):
        self.production = production_url
        self.shadow = shadow_url
        self.client = httpx.AsyncClient()

    async def handle_request(self, request_data):
        # Run both in parallel, return only production
        prod_task = self.client.post(self.production, json=request_data)
        shadow_task = self.client.post(self.shadow, json=request_data)

        prod_resp, shadow_resp = await asyncio.gather(
            prod_task, shadow_task, return_exceptions=True
        )

        # Log comparison (async, non-blocking)
        asyncio.create_task(self.log_comparison(
            request_data, prod_resp, shadow_resp
        ))

        # Only return production prediction to client
        return prod_resp.json()

    async def log_comparison(self, request, prod, shadow):
        log_to_warehouse({
            "request": request,
            "production_prediction": prod.json() if not isinstance(prod, Exception) else None,
            "shadow_prediction": shadow.json() if not isinstance(shadow, Exception) else None,
            "shadow_error": str(shadow) if isinstance(shadow, Exception) else None,
        })

Comparison Metrics

Agreement Metrics

  • Prediction agreement rate — how often they agree
  • Rank correlation — for ranking models
  • Distribution divergence — PSI / KL divergence
  • Confidence calibration — reliability curves

Operational Metrics

  • Error rate — shadow model failures
  • Latency comparison — faster or slower?
  • Memory usage — resource footprint
  • Edge cases — where do they disagree?

Dual-Write Patterns

# Dual-write: log both predictions for later analysis
class DualWritePredictor:
    def predict(self, features):
        # Production prediction (served to user)
        prod_result = self.prod_model.predict(features)

        # Shadow prediction (logged only)
        try:
            shadow_result = self.shadow_model.predict(features)
        except Exception as e:
            shadow_result = {"error": str(e)}

        # Async log to comparison table
        self.log_dual_prediction(features, prod_result, shadow_result)

        # Only return production result
        return prod_result

Istio Traffic Mirroring

# Istio mirror traffic to shadow service
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: model-shadow
spec:
  hosts: ["model.ml.internal"]
  http:
  - route:
    - destination:
        host: model-production
    mirror:
      host: model-shadow
    mirrorPercentage:
      value: 100.0  # Mirror all traffic

When to Use Shadow Mode

  • High-stakes models: Fraud, healthcare, safety-critical where any regression is costly
  • Major model changes: New architecture, new features, or significant retraining
  • New model types: First deployment of a model type (e.g., first NLP model)
  • Compliance requirements: When regulations require pre-deployment validation
Cost consideration: Shadow mode doubles your compute cost. For GPU models, run shadow on a smaller replica count or sample only a percentage of traffic. The insight-to-cost ratio is usually excellent.
