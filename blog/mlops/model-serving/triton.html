NVIDIA Triton: Multi-Framework Serving

MLOps Series Model Serving & Inference

Triton Overview

NVIDIA Triton Inference Server is the industry standard for high-performance, multi-framework model serving. It supports TensorFlow, PyTorch, ONNX, TensorRT, OpenVINO, and custom Python backends — all from a single server process.

Why Triton? It's the only server that supports all major frameworks simultaneously. You can serve a TensorRT model, an ONNX model, and a Python pipeline all from one process, with shared GPU resources and unified metrics.

Supported Backends

Framework Backends

  • TensorRT — Optimized NVIDIA inference
  • PyTorch — TorchScript / torch.compile
  • TensorFlow — SavedModel & GraphDef
  • ONNX Runtime — Cross-framework
  • OpenVINO — Intel hardware

Special Backends

  • Python — Custom preprocessing
  • DALI — GPU data preprocessing
  • FIL — Forest Inference (XGBoost)
  • vLLM — LLM serving backend
  • Custom C++ — Roll your own

Model Repository

Triton organizes models in a structured repository. Each model directory contains a config file and numbered version subdirectories.

# Model repository structure
model_repository/
├── resnet50/
│   ├── config.pbtxt
│   ├── 1/
│   │   └── model.plan       # TensorRT engine
│   └── 2/
│       └── model.plan
├── bert_tokenizer/
│   ├── config.pbtxt
│   └── 1/
│       └── model.py          # Python backend
└── ensemble_pipeline/
    ├── config.pbtxt
    └── 1/                     # Empty for ensemble
# config.pbtxt for a TensorRT model
name: "resnet50"
platform: "tensorrt_plan"
max_batch_size: 32

input [
  {
    name: "input"
    data_type: TYPE_FP32
    dims: [ 3, 224, 224 ]
  }
]
output [
  {
    name: "output"
    data_type: TYPE_FP32
    dims: [ 1000 ]
  }
]

instance_group [
  { count: 2  kind: KIND_GPU  gpus: [ 0 ] }
]

dynamic_batching {
  preferred_batch_size: [ 8, 16, 32 ]
  max_queue_delay_microseconds: 5000
}

Dynamic Batching

Triton's dynamic batcher combines individual requests into larger batches for GPU efficiency, respecting latency constraints.

# Dynamic batching configuration
dynamic_batching {
  preferred_batch_size: [ 4, 8, 16 ]
  max_queue_delay_microseconds: 10000   # Wait 10ms for more requests
  preserve_ordering: true
  priority_levels: 2
  default_priority_level: 1
  default_queue_policy {
    timeout_action: REJECT
    default_timeout_microseconds: 50000
    allow_timeout_override: true
  }
}

# Sequence batching for stateful models (e.g., RNNs)
sequence_batching {
  max_sequence_idle_microseconds: 5000000
  control_input [
    { name: "START"  control [ { kind: CONTROL_SEQUENCE_START } ] }
    { name: "END"    control [ { kind: CONTROL_SEQUENCE_END   } ] }
  ]
}
Batch size vs latency: Larger batches improve throughput but increase latency for early-arriving requests. Profile with perf_analyzer to find the optimal trade-off for your SLA.

Ensemble Models

Triton ensembles chain multiple models into a DAG pipeline — preprocessing → model → postprocessing — without intermediate network hops.

# config.pbtxt for ensemble
name: "ensemble_pipeline"
platform: "ensemble"
max_batch_size: 16

input  [ { name: "raw_image"  data_type: TYPE_UINT8  dims: [ -1, -1, 3 ] } ]
output [ { name: "class_label" data_type: TYPE_STRING dims: [ 1 ] } ]

ensemble_scheduling {
  step [
    {
      model_name: "image_preprocess"
      model_version: 1
      input_map  { key: "raw"    value: "raw_image" }
      output_map { key: "tensor" value: "preprocessed" }
    },
    {
      model_name: "resnet50"
      model_version: -1  # latest
      input_map  { key: "input"  value: "preprocessed" }
      output_map { key: "output" value: "logits" }
    },
    {
      model_name: "postprocess"
      model_version: 1
      input_map  { key: "logits" value: "logits" }
      output_map { key: "label"  value: "class_label" }
    }
  ]
}

Performance Tuning

Model Analyzer

# Profile with perf_analyzer
perf_analyzer \
  -m resnet50 \
  -u localhost:8001 \
  --concurrency-range 1:16:2 \
  --measurement-interval 5000 \
  -b 1

  --model-repository /models \
  --profile-models resnet50 \
  --triton-launch-mode docker \
  --output-model-repository-path /optimized

Instance Groups & Resource Allocation

# Multi-GPU with rate limiting
instance_group [
  { count: 2  kind: KIND_GPU  gpus: [ 0 ]  rate_limiter { resources [ { name: "R1"  count: 4 } ] } },
  { count: 1  kind: KIND_GPU  gpus: [ 1 ] }
]

# CPU fallback for overflow
instance_group [
  { count: 2  kind: KIND_GPU  gpus: [ 0 ] },
  { count: 4  kind: KIND_CPU }
]

Production Deployment

# Docker launch with all options
docker run --gpus all -d \
  --shm-size=1g --ulimit memlock=-1 \
  -p 8000:8000 -p 8001:8001 -p 8002:8002 \
  -v /path/to/models:/models \
  nvcr.io/nvidia/tritonserver:24.01-py3 \
  tritonserver \
    --model-repository=/models \
    --strict-model-config=false \
    --log-verbose=1 \
    --metrics-port=8002
Key ports: 8000 (HTTP), 8001 (gRPC), 8002 (Prometheus metrics). Always use --shm-size=1g or higher for models with large tensors — the default 64MB is too small.
