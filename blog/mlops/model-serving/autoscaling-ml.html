ML Scaling Challenges

ML inference services have unique scaling characteristics: GPU-bound compute, high memory per replica (model weights), long startup times (model loading), and bursty traffic patterns. Standard web app scaling doesn't work.

Web App Scaling

  • CPU-bound, fast startup
  • Small memory per replica
  • Scale on CPU/RPS
  • Seconds to start new pod

ML Inference Scaling

  • GPU-bound, slow startup
  • GBs of memory (model weights)
  • Scale on GPU util / queue depth
  • 30s-5min to start (model loading)

Horizontal Pod Autoscaler (HPA)

# HPA for ML inference service
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: model-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: model-server
  minReplicas: 2
  maxReplicas: 20
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Pods
        value: 4
        periodSeconds: 60    # Add up to 4 pods per minute
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 25
        periodSeconds: 120   # Scale down 25% every 2 min
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
Aggressive scale-down is dangerous for ML: Each new pod needs 30s-5min to load the model. Scale down too fast and you won't have capacity to handle the next traffic spike. Use long stabilization windows (300s+).

Vertical Pod Autoscaler (VPA)

# VPA for right-sizing ML pods
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: model-server-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: model-server
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: model-server
      minAllowed:
        cpu: "1"
        memory: "2Gi"
      maxAllowed:
        cpu: "8"
        memory: "16Gi"
      controlledResources: ["cpu", "memory"]

Custom Metrics Scaling

# Scale on inference queue depth (Prometheus metric)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  metrics:
  - type: Pods
    pods:
      metric:
        name: inference_queue_depth
      target:
        type: AverageValue
        averageValue: "5"   # Scale when >5 requests queued per pod

  # Or scale on p99 latency
  - type: Pods
    pods:
      metric:
        name: inference_latency_p99_ms
      target:
        type: AverageValue
        averageValue: "80"  # Scale up when p99 > 80ms

GPU-Aware Scaling

# Scale on GPU utilization via DCGM exporter
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  metrics:
  - type: Pods
    pods:
      metric:
        name: DCGM_FI_DEV_GPU_UTIL  # NVIDIA GPU utilization
      target:
        type: AverageValue
        averageValue: "75"   # Scale when GPU > 75%

# GPU memory-based scaling
  - type: Pods
    pods:
      metric:
        name: DCGM_FI_DEV_FB_USED
      target:
        type: AverageValue
        averageValue: "12000"  # MB, scale when using >12GB

Scale-to-Zero

Scale-to-zero saves cost for intermittently used models. The trade-off is cold start latency when the first request arrives.

# KServe/Knative scale-to-zero config
metadata:
  annotations:
    autoscaling.knative.dev/min-scale: "0"
    autoscaling.knative.dev/max-scale: "10"
    autoscaling.knative.dev/scale-to-zero-pod-retention-period: "10m"
    autoscaling.knative.dev/target: "5"     # 5 concurrent requests/pod

# Strategies to mitigate cold start:
# 1. Keep model in shared memory / tmpfs
# 2. Use model caching layer (ModelMesh)
# 3. Pre-warm on schedule for predictable traffic
# 4. Use smaller/quantized models (faster load)

Scaling Strategies

🟢 Low Traffic Models

  • Scale-to-zero with Knative
  • Shared GPU with MIG/MPS
  • ModelMesh for density
  • Accept cold start latency

🔴 High Traffic Models

  • HPA on GPU util + queue depth
  • Predictive scaling (time-based)
  • Over-provision for burst headroom
  • Multiple GPU types (A100 + T4)
Cost optimization formula: Use spot/preemptible instances for burst capacity (with graceful draining), reserved instances for baseline load, and scale-to-zero for long-tail models. This can reduce GPU costs by 40-60%.
