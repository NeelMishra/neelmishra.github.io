TensorFlow Serving: Production Model Server

MLOps Series Model Serving & Inference

SavedModel Format

TensorFlow Serving consumes models in TensorFlow's SavedModel format. This is a language-neutral, self-contained serialization that includes the computation graph, weights, and serving signatures.

# Export a SavedModel
import tensorflow as tf

model = tf.keras.Sequential([
    tf.keras.layers.Dense(128, activation="relu"),
    tf.keras.layers.Dense(10, activation="softmax")
])
model.compile(optimizer="adam", loss="sparse_categorical_crossentropy")
model.fit(train_x, train_y, epochs=5)

# Save with explicit signature
tf.saved_model.save(model, "models/classifier/1")

Directory Structure

# TF Serving model repository layout
models/
└── classifier/
    ├── 1/                # Version 1
    │   ├── saved_model.pb
    │   └── variables/
    │       ├── variables.data-00000-of-00001
    │       └── variables.index
    └── 2/                # Version 2 (auto-detected)
        ├── saved_model.pb
        └── variables/
Auto-versioning: TF Serving automatically detects new version directories (numbered folders). When you drop a new version folder, TF Serving loads it and starts routing traffic — no restart required.

REST & gRPC APIs

TF Serving provides both REST and gRPC endpoints. gRPC is preferred for production due to lower latency and efficient binary serialization via Protocol Buffers.

REST API

  • Port 8501 by default
  • JSON request/response
  • Easy to debug with curl
  • Higher overhead per request

gRPC API

  • Port 8500 by default
  • Protobuf binary serialization
  • ~2-5× lower latency than REST
  • Streaming support
# REST API call
curl -X POST http://localhost:8501/v1/models/classifier:predict \
  -H "Content-Type: application/json" \
  -d '{"instances": [[1.0, 2.0, 3.0, 4.0]]}'

# Response
{
  "predictions": [[0.1, 0.05, 0.7, 0.02, ...]]
}
# gRPC client in Python
import grpc
from tensorflow_serving.apis import predict_pb2, prediction_service_pb2_grpc
import tensorflow as tf

channel = grpc.insecure_channel("localhost:8500")
stub = prediction_service_pb2_grpc.PredictionServiceStub(channel)

request = predict_pb2.PredictRequest()
request.model_spec.name = "classifier"
request.model_spec.signature_name = "serving_default"
request.inputs["input_1"].CopyFrom(
    tf.make_tensor_proto(input_data, shape=[1, 4])
)

response = stub.Predict(request, timeout=5.0)

Request Batching

TF Serving's built-in batching groups individual requests into a batch tensor for GPU efficiency. This is configured via a batching parameters file.

# batching_config.txt
max_batch_size { value: 32 }
batch_timeout_micros { value: 10000 }  # 10ms
num_batch_threads { value: 4 }
max_enqueued_batches { value: 100 }
pad_variable_length_inputs: true
# Start TF Serving with batching
tensorflow_model_server \
  --model_name=classifier \
  --model_base_path=/models/classifier \
  --enable_batching=true \
  --batching_parameters_file=batching_config.txt
Latency impact: The batch_timeout_micros adds latency to every request (worst case). Set it based on your p99 latency budget minus the inference time. For a 100ms budget with 60ms inference, use ≤30ms timeout.

Model Versioning

TF Serving manages multiple model versions simultaneously, enabling canary deployments and safe rollbacks.

# model_config.txt — serve multiple models and versions
model_config_list {
  config {
    name: "classifier"
    base_path: "/models/classifier"
    model_platform: "tensorflow"
    model_version_policy {
      specific { versions: 1  versions: 2 }
    }
    version_labels {
      key: "stable"
      value: 1
    }
    version_labels {
      key: "canary"
      value: 2
    }
  }
  config {
    name: "recommender"
    base_path: "/models/recommender"
    model_platform: "tensorflow"
  }
}

Model Warmup

First requests after model load are slow due to memory allocation and JIT compilation. Warmup sends synthetic requests during loading to amortize this cost.

# Generate warmup data
import tensorflow as tf
from tensorflow_serving.apis import predict_pb2, prediction_service_pb2

with tf.io.TFRecordWriter("models/classifier/1/assets.extra/tf_serving_warmup_requests") as writer:
    request = predict_pb2.PredictRequest()
    request.model_spec.name = "classifier"
    request.inputs["input_1"].CopyFrom(
        tf.make_tensor_proto([[0.0]*4], shape=[1, 4])
    )
    log = prediction_service_pb2.PredictionLog(predict_log=
        prediction_service_pb2.PredictLog(request=request))
    for _ in range(100):
        writer.write(log.SerializeToString())

Docker Deployment

# Docker run with GPU support
docker run -d --gpus all \
  -p 8500:8500 -p 8501:8501 \
  -v /path/to/models:/models \
  -e MODEL_NAME=classifier \
  tensorflow/serving:latest-gpu

# Docker Compose for multi-model
version: "3.8"
services:
  tf-serving:
    image: tensorflow/serving:latest-gpu
    ports:
      - "8500:8500"
      - "8501:8501"
    volumes:
      - ./models:/models
      - ./model_config.txt:/config/model_config.txt
    command: --model_config_file=/config/model_config.txt
    deploy:
      resources:
        reservations:
          devices:
            - capabilities: [gpu]

Monitoring & Debugging

TF Serving exposes model status and metadata endpoints for health checks and debugging.

# Check model status
curl http://localhost:8501/v1/models/classifier

# Get model metadata (signature info)
curl http://localhost:8501/v1/models/classifier/metadata

# Response includes input/output tensor names and shapes
{
  "model_spec": { "name": "classifier", "version": "2" },
  "metadata": {
    "signature_def": {
      "serving_default": {
        "inputs": { "input_1": { "dtype": "DT_FLOAT", "shape": [-1, 4] } },
        "outputs": { "dense_1": { "dtype": "DT_FLOAT", "shape": [-1, 10] } }
      }
    }
  }
}
Production tip: Use the /v1/models/{model}/metadata endpoint in your CI/CD pipeline to validate that a newly deployed model has the expected input/output signatures before routing traffic.
