A/B Testing ML Models

MLOps Series Model Serving & Inference

Why A/B Test Models?

Offline metrics (AUC, F1, RMSE) don't fully predict online performance. A model with higher offline accuracy may actually decrease business metrics due to latency, edge cases, or user behavior changes. A/B testing is the gold standard for validating ML models in production.

Offline vs online gap: Google researchers found that improvements in offline metrics translate to online improvements only 30-50% of the time. Always A/B test before fully deploying a new model.

    significance_level: float = 0.05
    power: float = 0.80

# Traffic routing
def route_request(user_id, config):
    # Deterministic hashing for consistent assignment
    bucket = hash(f"{user_id}:{config.name}") % 100
    if bucket < config.traffic_split * 100:
        return config.treatment_model
    return config.control_model

Sample Size Calculation

# Calculate minimum sample size
from scipy import stats
import numpy as np

def min_sample_size(baseline_rate, mde, alpha=0.05, power=0.80):
    """
    baseline_rate: current metric (e.g., 0.85 catch rate)
    mde: minimum detectable effect (e.g., 0.02 = 2% lift)
    """
    z_alpha = stats.norm.ppf(1 - alpha / 2)
    z_beta = stats.norm.ppf(power)
    p1, p2 = baseline_rate, baseline_rate + mde
    pooled = (p1 + p2) / 2
    n = ((z_alpha * np.sqrt(2 * pooled * (1-pooled)) +
          z_beta * np.sqrt(p1*(1-p1) + p2*(1-p2))) / mde) ** 2
    return int(np.ceil(n))

# Example: detect 2% lift in 85% baseline catch rate
n = min_sample_size(0.85, 0.02)
print(f"Need {n:,} samples per variant")
# → ~3,600 samples per variant

Metric Selection

Primary Metrics

  • The business outcome you're optimizing
  • One metric drives the decision
  • Examples: revenue, CTR, catch rate
  • Must be measurable within test duration

Secondary Metrics

  • Supporting context and diagnostics
  • Don't override primary metric
  • Model-specific: confidence, coverage
  • User-specific: engagement, satisfaction

Guardrail Metrics

Guardrail metrics are "do no harm" constraints. Even if the primary metric improves, failing a guardrail blocks deployment.

# Guardrail check
def check_guardrails(control_metrics, treatment_metrics, thresholds):
    failures = []
    for metric, threshold in thresholds.items():
        delta = treatment_metrics[metric] - control_metrics[metric]
        if threshold["direction"] == "lower_is_better":
            if delta > threshold["max_increase"]:
                failures.append(f"{metric}: +{delta:.3f} exceeds {threshold['max_increase']}")
        elif threshold["direction"] == "upper_bounded":
            if treatment_metrics[metric] > threshold["max_value"]:
                failures.append(f"{metric}: {treatment_metrics[metric]:.1f} > {threshold['max_value']}")
    return failures

guardrails = {
    "false_positive_rate": {"direction": "lower_is_better", "max_increase": 0.005},
    "p99_latency_ms": {"direction": "upper_bounded", "max_value": 100},
}

Statistical Analysis

# Analyze A/B test results
from scipy import stats

def analyze_ab_test(control_data, treatment_data, alpha=0.05):
    # Two-sample proportion test
    control_rate = control_data.mean()
    treatment_rate = treatment_data.mean()
    lift = (treatment_rate - control_rate) / control_rate

    stat, p_value = stats.proportions_ztest(
        [treatment_data.sum(), control_data.sum()],
        [len(treatment_data), len(control_data)]
    )

    significant = p_value < alpha

    return {
        "control_rate": control_rate,
        "treatment_rate": treatment_rate,
        "lift": f"{lift:.2%}",
        "p_value": p_value,
        "significant": significant,
        "decision": "SHIP" if significant and lift > 0 else "NO SHIP"
    }
Don't peek! Checking results before reaching the planned sample size inflates your false positive rate. Use sequential testing methods (like CUPED or always-valid p-values) if you need early stopping.

Common Pitfalls

  • Novelty/primacy effects: Users may engage differently with a new model simply because it's new. Run tests for at least 2 weeks.
  • Network effects: In social/marketplace models, treatment users affect control users. Use cluster randomization.
  • Multiple comparisons: Testing many metrics inflates false positives. Use Bonferroni correction or FDR control.
  • Simpson's paradox: Overall results may differ from segment-level results. Always check key segments.
  • Survivorship bias: Only measuring users who completed the flow ignores those who dropped off.
ML-specific A/B testing challenges: Model predictions influence user behavior (e.g., a better recommender changes what users click), creating feedback loops. Consider using interleaving experiments for ranking models — they require 10× fewer samples than traditional A/B tests.
