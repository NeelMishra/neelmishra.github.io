Serving Patterns: Batch, Online, Streaming, Embedded

MLOps Series Model Serving & Inference

Serving Patterns Overview

How you serve a model is as important as how you train it. The serving pattern determines latency, throughput, infrastructure cost, and how quickly predictions reach end users. There are four dominant patterns in production ML systems.

Key insight: Most production systems use a combination of patterns — batch for offline recommendations, online for real-time scoring, and streaming for near-real-time fraud detection.

Batch Serving

Batch serving runs predictions on large datasets at scheduled intervals. This is the simplest and most common pattern — think nightly recommendation updates, weekly churn scores, or daily fraud risk assessments.

Architecture

A typical batch pipeline reads features from a data warehouse, runs the model on a compute cluster (Spark, Dask, or a GPU node), and writes predictions back to a store for downstream consumption.

# Batch inference with Spark
from pyspark.sql import SparkSession
import mlflow.pyfunc

spark = SparkSession.builder.appName("batch-inference").getOrCreate()

# Load model from registry
model = mlflow.pyfunc.load_model("models:/fraud-detector/Production")

# Read features from data warehouse
features_df = spark.read.parquet("s3://warehouse/features/daily/")

# Apply predictions in parallel across the cluster
predictions = features_df.toPandas()
results = model.predict(predictions)

# Write back to serving store
output_df = spark.createDataFrame(results)
output_df.write.parquet("s3://predictions/fraud/latest/")

When to Use Batch

  • Latency tolerance: Predictions can be minutes to hours old
  • High volume: Millions to billions of predictions
  • Cost efficiency: Spot/preemptible instances for compute
  • Simplicity: No serving infrastructure to maintain
Watch out: Batch predictions become stale. If your model's features change rapidly (e.g., real-time user behavior), batch predictions may be inaccurate by the time they're served.

Online (Real-Time) Serving

Online serving exposes a model behind an API endpoint. Clients send a request with features, the model runs inference, and the prediction is returned — all within milliseconds.

Architecture

# FastAPI model serving endpoint
from fastapi import FastAPI
import torch

app = FastAPI()
model = torch.load("model.pt")
model.eval()

@app.post("/predict")
async def predict(request: PredictRequest):
    features = torch.tensor(request.features)
    with torch.no_grad():
        prediction = model(features)
    return {"prediction": prediction.tolist()}

Key Considerations

  • Latency budget: Typically p99 < 100ms for user-facing
  • Autoscaling: HPA on GPU utilization or request queue depth
  • Health checks: Liveness + readiness probes for model loading
  • Request batching: Group requests for GPU efficiency

Streaming Serving

Streaming serving processes events as they arrive through a message queue. This bridges the gap between batch (high latency) and online (synchronous request-response) patterns.

Architecture

Events flow through Kafka/Kinesis, a consumer picks them up, runs inference, and publishes predictions back to another topic or directly to a serving store.

# Streaming inference with Kafka consumer
from kafka import KafkaConsumer, KafkaProducer
import json, torch

consumer = KafkaConsumer("events", bootstrap_servers="kafka:9092")
producer = KafkaProducer(bootstrap_servers="kafka:9092")
model = torch.load("model.pt")

for message in consumer:
    event = json.loads(message.value)
    features = extract_features(event)

    with torch.no_grad():
        prediction = model(features)

    result = {"event_id": event["id"], "score": prediction.item()}
    producer.send("predictions", json.dumps(result).encode())

Use Cases

  • Fraud detection: Score transactions as they happen
  • IoT anomaly detection: Process sensor data continuously
  • Content moderation: Flag content in near-real-time
  • Personalization: Update recommendations based on recent activity
Streaming + batch = lambda architecture: Many teams run batch for complete reprocessing and streaming for low-latency updates. The batch layer corrects any streaming errors.

Embedded Serving

Embedded serving runs the model directly on the client device — mobile phones, IoT sensors, browsers, or edge hardware. No network call needed.

Deployment Targets

📱 Mobile (TFLite / Core ML)

  • On-device image classification
  • Keyboard next-word prediction
  • Face detection in camera apps
  • Offline-first experiences

🌐 Browser (ONNX.js / TF.js)

  • Client-side spam filtering
  • Real-time video effects
  • Privacy-preserving inference
  • Zero server cost

🔧 Edge (NVIDIA Jetson / Coral)

  • Factory quality inspection
  • Autonomous vehicle perception
  • Security camera analytics
  • Low-latency robotics

🗄️ In-Database (SQL ML)

  • BigQuery ML, Redshift ML
  • Predictions alongside queries
  • No data movement needed
  • Simplest integration
# TensorFlow Lite on-device inference
import tensorflow as tf

# Load quantized model for mobile
interpreter = tf.lite.Interpreter(model_path="model.tflite")
interpreter.allocate_tensors()

input_details = interpreter.get_input_details()

Pattern Comparison Matrix

⏱ Latency

  • Batch: Minutes to hours
  • Online: Milliseconds
  • Streaming: Seconds to minutes
  • Embedded: Milliseconds (no network)

📈 Throughput

  • Batch: Very high (millions)
  • Online: Medium (autoscale)
  • Streaming: High (partitioned)
  • Embedded: Per-device limited

💰 Infrastructure Cost

  • Batch: Low (spot instances)
  • Online: High (always-on GPUs)
  • Streaming: Medium (Kafka + compute)
  • Embedded: Zero server cost

🔧 Complexity

  • Batch: Low
  • Online: High (scaling, monitoring)
  • Streaming: High (exactly-once, ordering)
  • Embedded: Medium (model optimization)

Choosing the Right Pattern

Use this decision framework to pick the right serving pattern for your use case:

# Decision tree for serving pattern selection
def choose_pattern(requirements):
    if requirements.latency_tolerance > minutes(5):
        if requirements.volume > 1_000_000:
            return "BATCH"  # Spark / Dask / Ray
        return "BATCH"  # Simple scheduled job

    if requirements.needs_offline:
        return "EMBEDDED"  # TFLite / Core ML / ONNX

    if requirements.event_driven:
        return "STREAMING"  # Kafka + Flink consumer

    return "ONLINE"  # REST / gRPC model server
Hybrid is normal: Netflix uses batch for nightly pre-computed recommendations, online serving for real-time ranking, and streaming for session-based personalization — all for the same model family.

Final Checklist

  • ✅ Define your latency SLA before choosing a pattern
  • ✅ Consider data freshness requirements
  • ✅ Factor in team expertise and operational burden
  • ✅ Plan for hybrid architectures from the start
  • ✅ Build observability into every serving path
