Why Batching Matters

GPUs achieve peak efficiency when processing many inputs in parallel. A single inference request uses a tiny fraction of GPU compute. Batching groups multiple requests into one GPU operation, dramatically improving throughput without proportionally increasing latency.

The batching paradox: Individual request latency increases slightly (waiting for the batch), but throughput increases dramatically. The sweet spot depends on your latency SLA and traffic pattern.

Static vs Dynamic Batching

Static Batching

  • Fixed batch size (e.g., always 32)
  • Wait until batch is full
  • Simple to implement
  • ❌ High latency at low traffic
  • ❌ Wasted GPU at varying loads

Dynamic Batching

  • Variable batch size (1 to max)
  • Timeout-based: send partial batch
  • Adapts to traffic patterns
  • ✅ Low latency at low traffic
  • ✅ High throughput at high traffic

Adaptive Batching

Adaptive batching goes beyond dynamic batching by adjusting batch parameters in real-time based on observed latency and throughput metrics.

# Adaptive batching algorithm
class AdaptiveBatcher:
    def __init__(self, max_batch=64, latency_target_ms=50):
        self.max_batch = max_batch
        self.latency_target = latency_target_ms
        self.current_timeout = 10  # ms, adaptive
        self.queue = []

    def add_request(self, request):
        self.queue.append(request)
        if len(self.queue) >= self.max_batch:
            return self.flush()

    def flush(self):
        batch = self.queue[:self.max_batch]
        self.queue = self.queue[self.max_batch:]
        return batch

    def adjust_timeout(self, observed_latency_ms):
        # Increase timeout if latency is under budget (fill bigger batches)
        if observed_latency_ms < self.latency_target * 0.7:
            self.current_timeout = min(self.current_timeout * 1.2, 100)
        # Decrease timeout if latency is near budget
        elif observed_latency_ms > self.latency_target * 0.9:
            self.current_timeout = max(self.current_timeout * 0.8, 1)

Client-Side Batching

# Client-side batch collector
import asyncio
from collections import deque

class ClientBatcher:
    def __init__(self, endpoint, batch_size=32, timeout=0.05):
        self.endpoint = endpoint
        self.batch_size = batch_size
        self.timeout = timeout
        self.queue = deque()
        self.futures = {}

    async def predict(self, features):
        future = asyncio.Future()
        self.queue.append((features, future))
        if len(self.queue) >= self.batch_size:
            await self._flush()
        return await future

    async def _flush(self):
        batch = [self.queue.popleft() for _ in range(min(len(self.queue), self.batch_size))]
        features = [b[0] for b in batch]
        results = await self._call_endpoint(features)
        for (_, future), result in zip(batch, results):
            future.set_result(result)

Configuration Tuning

# Finding optimal batch size empirically
def benchmark_batch_sizes(model, input_shape):
    results = {}
    for bs in [1, 2, 4, 8, 16, 32, 64, 128]:
        inputs = torch.randn(bs, *input_shape).cuda()
        # Warmup
        for _ in range(10):
            model(inputs)
        torch.cuda.synchronize()

        start = time.perf_counter()
        for _ in range(100):
            model(inputs)
        torch.cuda.synchronize()
        elapsed = time.perf_counter() - start

        results[bs] = {
            "throughput": bs * 100 / elapsed,
            "latency_ms": elapsed / 100 * 1000,
            "gpu_util": get_gpu_util()
        }
    return results
Memory constraint: Larger batches need more GPU memory. Profile memory usage with torch.cuda.max_memory_allocated(). If a batch size causes OOM, the server crashes — always set max_batch_size conservatively.

Common Pitfalls

  • Variable-length inputs: Padding sequences to max length wastes compute. Use bucketing to group similar-length inputs.
  • Head-of-line blocking: One slow request in a batch delays all others. Set per-request timeouts.
  • Over-batching: Beyond a certain point, larger batches increase memory without improving throughput. Find the saturation point.
  • Ignoring preprocessing: If preprocessing is CPU-bound, batching the GPU won't help. Batch the entire pipeline.
Rule of thumb: Start with max_batch_size = GPU_memory / (2 × single_inference_memory) and max_delay = latency_SLA × 0.3. Tune from there based on actual traffic patterns.
