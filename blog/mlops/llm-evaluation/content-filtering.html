Content Filtering: Input/Output Moderation

MLOps Series LLM Evaluation & Safety

Large language models can generate remarkably useful text — but without guardrails they can also produce toxic content, leak private data, or be manipulated into ignoring safety instructions. Content filtering is the practice of inspecting both user inputs and model outputs in real time, blocking or transforming anything that violates your safety policy before it reaches the end user.

This post covers the full moderation stack: regex-based filters, ML classifiers, LLM-based moderation, prompt-injection detection, PII scrubbing, and how to combine them into a defense-in-depth architecture that balances safety, latency, and false-positive rates in production.

The Content Moderation Landscape

Content moderation for LLM applications differs fundamentally from traditional web moderation. In a classic social-media pipeline you moderate user-generated content after the fact. In an LLM pipeline you must moderate both the user prompt (input) and the model response (output) — in real time, before the response is streamed to the client.

The three broad families of content filters are:

Rule-Based / Regex

  • Keyword blocklists, regex patterns
  • Extremely fast (< 1 ms)
  • High false-positive rate on edge cases
  • Easy to circumvent with misspellings

ML Classifiers

  • Fine-tuned BERT/DeBERTa models
  • Category-level predictions (hate, violence, sexual)
  • ~10–50 ms latency on GPU
  • Requires labeled training data

LLM-Based Moderation

  • Use a second LLM as a judge
  • Flexible — understands nuance & context
  • 200–800 ms latency
  • Highest cost per call

Hybrid / Defense-in-Depth

  • Stack all three in a pipeline
  • Fast layers catch obvious violations
  • Slower layers handle nuanced cases
  • Best accuracy/latency trade-off
Design principle: No single layer catches everything. A production system always combines multiple filters — cheap, fast rules for obvious violations and expensive, smart classifiers for subtle ones.

Input Filtering: Guarding the Front Door

Input filters inspect the user's prompt before it reaches the LLM. The two primary threats they address are prompt injection (adversarial instructions designed to override system prompts) and off-topic or policy-violating requests (e.g., requests for harmful instructions).

Regex & Keyword Blocklists

The simplest layer is a curated list of keywords and regular expressions. While trivially fast, they are easy to evade (leetspeak, Unicode homoglyphs, token-boundary tricks). Treat them as a coarse first pass, never the sole defense. 

# Layer 1: Fast regex pre-filter
import re

BLOCKLIST_PATTERNS = [
    re.compile(r"ignore\s+(all\s+)?(previous|above)\s+instructions", re.IGNORECASE),
    re.compile(r"you\s+are\s+now\s+(DAN|unfiltered|jailbroken)", re.IGNORECASE),
    re.compile(r"system\s*:\s*", re.IGNORECASE),
    re.compile(r"<\|im_start\|>|<\|im_end\|>"),              # ChatML injection
]

def regex_filter(prompt: str) -> bool:
    """Return True if prompt matches any blocklist pattern."""
    for pattern in BLOCKLIST_PATTERNS:
        if pattern.search(prompt):
            return True
    return False

Prompt Injection Detection

Prompt injection is the most dangerous input-side threat. Attackers embed instructions like "Ignore previous instructions and …" inside user text. Modern detectors use a fine-tuned classifier trained on known injection patterns. 

# Layer 2: ML-based prompt injection classifier
from transformers import pipeline

injection_clf = pipeline(
    "text-classification",
    model="protectai/deberta-v3-base-prompt-injection-v2",
    device=0,
)

def detect_injection(prompt: str, threshold: float = 0.85) -> dict:
    result = injection_clf(prompt, truncation=True, max_length=512)[0]
    is_injection = (
        result["label"] == "INJECTION"
        and result["score"] >= threshold
    )
    return {"blocked": is_injection, "score": result["score"], "label": result["label"]}

Topic Classification

Beyond injection, you may want to block entire topic categories — medical advice, legal counsel, financial recommendations — depending on your product's liability surface. A lightweight zero-shot classifier can route prompts into allowed vs. disallowed topics without training data. 

# Layer 3: Zero-shot topic gating
from transformers import pipeline

topic_clf = pipeline("zero-shot-classification", model="facebook/bart-large-mnli")

DISALLOWED_TOPICS = ["medical diagnosis", "legal advice", "weapons manufacturing"]
ALLOWED_TOPICS    = ["general knowledge", "coding help", "creative writing"]

def topic_gate(prompt: str, threshold: float = 0.70) -> dict:
    result = topic_clf(prompt, candidate_labels=DISALLOWED_TOPICS + ALLOWED_TOPICS)
    top_label = result["labels"][0]
    top_score = result["scores"][0]
    blocked = top_label in DISALLOWED_TOPICS and top_score >= threshold
    return {"blocked": blocked, "topic": top_label, "confidence": top_score}
Watch out: Zero-shot classifiers are sensitive to how you phrase the candidate labels. "Medical diagnosis" and "health advice" can yield very different scores for the same prompt. Always validate label wording against a test set before deploying.

Output Filtering: Screening Responses

Even if the input passes all checks, the model can still produce unsafe content — hallucinated PII, toxic language, or policy-violating instructions it learned from pre-training data. Output filters operate on the model's response before it is returned to the user.

Toxicity Classifiers

Toxicity classifiers assign a score (0–1) across categories like hate speech, harassment, self-harm, sexual content, and violence. They are typically fine-tuned BERT-family models that run in 10–30 ms on a GPU. 

# Output toxicity screening with a multi-label classifier
from transformers import pipeline

toxicity_clf = pipeline(
    "text-classification",
    model="unitary/toxic-bert",
    top_k=None,          # return all category scores
    device=0,
)

TOXICITY_THRESHOLDS = {
    "toxic": 0.7, "severe_toxic": 0.5,
    "obscene": 0.7, "threat": 0.5,
    "insult": 0.7, "identity_hate": 0.5,
}

def screen_toxicity(text: str) -> dict:
    scores = {r["label"]: r["score"] for r in toxicity_clf(text)[0]}
    violations = [
        cat for cat, thresh in TOXICITY_THRESHOLDS.items()
        if scores.get(cat, 0) >= thresh
    ]
    return {"blocked": len(violations) > 0, "violations": violations, "scores": scores}

PII Detection & Redaction

LLMs can accidentally surface personal information — email addresses, phone numbers, SSNs, credit card numbers — either from their training data or from context supplied via RAG. A PII layer detects and redacts these entities before the response reaches the user. 

# PII detection using regex + NER hybrid
import re
from presidio_analyzer import AnalyzerEngine
from presidio_anonymizer import AnonymizerEngine

analyzer  = AnalyzerEngine()
anonymizer = AnonymizerEngine()

def redact_pii(text: str, language: str = "en") -> dict:
    # Detect PII entities (email, phone, SSN, credit card, etc.)
    results = analyzer.analyze(
        text=text,
        language=language,
        entities=["EMAIL_ADDRESS", "PHONE_NUMBER", "US_SSN",
                  "CREDIT_CARD", "PERSON", "LOCATION"],
    )
    # Anonymize detected entities
    anonymized = anonymizer.anonymize(text=text, analyzer_results=results)
    return {
        "clean_text": anonymized.text,
        "entities_found": [
            {"type": r.entity_type, "score": r.score, "start": r.start, "end": r.end}
            for r in results
        ],
    }
Microsoft Presidio is the de-facto open-source PII engine. It combines regex recognizers, spaCy NER, and custom recognizers in a single pipeline. For high-throughput production use, deploy it behind a FastAPI service with batch support.

Multi-Layer Defense Architecture

The defense-in-depth principle is simple: stack multiple independent filters so that an adversary must bypass every layer simultaneously. Each layer is optimized for a different threat class and latency budget.

The key design decisions for a multi-layer pipeline:

  1. Order by latency — cheapest filters run first. If Layer 1 blocks, you save the cost of Layers 2–4.
  2. Short-circuit on block — return a canned refusal message the instant any layer triggers. Do not continue processing.
  3. Run expensive layers asynchronously or via sampling — the LLM-as-judge layer is too slow for every request. Run it on 5–10 % of traffic and flag violations for human review.
  4. Separate input and output pipelines — input filters focus on injection & topic gating; output filters focus on toxicity & PII. They are different threat models.
# Multi-layer content filter orchestrator
import asyncio
from dataclasses import dataclass

@dataclass
class FilterResult:
    blocked: bool
    layer: str
    reason: str = ""

async def filter_input(prompt: str) -> FilterResult:
    # Layer 1: regex (sync, < 1 ms)
    if regex_filter(prompt):
        return FilterResult(blocked=True, layer="regex", reason="blocklist_match")

    # Layer 2: injection classifier (10-20 ms)
    inj = await asyncio.to_thread(detect_injection, prompt)
    if inj["blocked"]:
        return FilterResult(blocked=True, layer="injection_clf", reason="prompt_injection")

    # Layer 3: topic gating (20-40 ms)
    topic = await asyncio.to_thread(topic_gate, prompt)
    if topic["blocked"]:
        return FilterResult(blocked=True, layer="topic_gate", reason=topic["topic"])

    return FilterResult(blocked=False, layer="none")

async def filter_output(response: str) -> FilterResult:
    # Layer 1: toxicity classifier
    tox = await asyncio.to_thread(screen_toxicity, response)
    if tox["blocked"]:
        return FilterResult(blocked=True, layer="toxicity", reason=str(tox["violations"]))

    # Layer 2: PII redaction (always applied, never blocks — just redacts)
    pii = await asyncio.to_thread(redact_pii, response)

    return FilterResult(blocked=False, layer="none")

OpenAI Moderation API & Alternatives

If you don't want to host your own classifiers, managed moderation APIs provide a turnkey solution. The most widely used is the OpenAI Moderation endpoint, which is free for all API customers and returns category scores across 11 harm dimensions. 

# Using the OpenAI Moderation API
from openai import OpenAI

client = OpenAI()

def openai_moderate(text: str) -> dict:
    response = client.moderations.create(
        model="omni-moderation-latest",
        input=text,
    )
    result = response.results[0]

    # Check if any category is flagged
    flagged_cats = [
        cat for cat, flagged in result.categories.model_dump().items()
        if flagged
    ]
    return {
        "flagged": result.flagged,
        "categories": flagged_cats,
        "scores": result.category_scores.model_dump(),
    }

# Example usage
result = openai_moderate("I want to hurt someone.")
# {'flagged': True, 'categories': ['violence'], 'scores': {'violence': 0.92, ...}}

The OpenAI Moderation API covers these categories:

Content Categories

  • hate — targeting protected groups
  • harassment — hostile or abusive language
  • self-harm — encouraging or depicting self-injury
  • sexual — explicit sexual content
  • violence — graphic depictions of violence

Subcategories & Modifiers

  • hate/threatening — hate speech with intent to cause harm
  • self-harm/intent — expressing desire to self-harm
  • self-harm/instructions — instructions for self-harm
  • sexual/minors — sexual content involving minors
  • violence/graphic — extremely graphic violence

Alternative Moderation Services

Several alternatives exist depending on your requirements:

Managed APIs

  • Azure AI Content Safety — multi-severity (0–7) scores, image + text
  • AWS Comprehend — toxicity detection + PII as a service
  • Perspective API (Google/Jigsaw) — free, research-oriented

Self-Hosted Models

  • Llama Guard (Meta) — LLM-based safety classifier
  • NeMo Guardrails (NVIDIA) — programmable rail system
  • Presidio (Microsoft) — PII-specific, rule + NER
Vendor lock-in risk: If you route all moderation through a single provider, a policy change on their end can break your product overnight. Keep at least one self-hosted fallback layer for critical safety categories.

Category Taxonomies & Policy Design

A taxonomy defines what you filter and how severely you respond. Getting the taxonomy right is as important as getting the model right — an overly broad taxonomy blocks legitimate users, while a narrow one leaves gaps.

Designing Your Category Hierarchy

Start with a two-level hierarchy: broad categories (violence, sexual, PII) and sub-categories that map to specific policy actions. Each sub-category has a severity level and an action. 

# Category taxonomy configuration
POLICY = {
    "violence": {
        "graphic_violence":     {"severity": "high",   "action": "block"},
        "fantasy_violence":      {"severity": "low",    "action": "allow"},
        "violence_instructions": {"severity": "critical","action": "block_and_log"},
    },
    "pii": {
        "email":        {"severity": "medium", "action": "redact"},
        "phone":        {"severity": "medium", "action": "redact"},
        "ssn":          {"severity": "critical","action": "block_and_log"},
        "credit_card":  {"severity": "critical","action": "block_and_log"},
    },
    "prompt_injection": {
        "direct_injection":   {"severity": "critical","action": "block_and_log"},
        "indirect_injection": {"severity": "high",   "action": "block"},
    },
}

def enforce_policy(category: str, sub_category: str, score: float) -> str:
    rule = POLICY.get(category, {}).get(sub_category)
    if rule is None:
        return "allow"
    return rule["action"]
Tip: Store your policy as a versioned JSON/YAML config — not hardcoded. This lets safety teams update thresholds and actions without code deployments, and gives you an audit trail of policy changes over time.

Latency Budgets & False Positive Management

Every moderation layer adds latency. In a streaming chat product where users expect first-token latency under 500 ms, you cannot afford to run a 300 ms LLM judge on every request synchronously. Designing within a latency budget is critical.

Strategies to Stay Within Budget

  • Parallel execution — run independent classifiers concurrently (injection + topic classifiers in parallel rather than sequentially).
  • Async sampling for expensive layers — run LLM-as-judge on 5–10 % of requests asynchronously; log violations for human review rather than blocking in real time.
  • Model distillation — distill a large judge model into a smaller, faster student classifier that can run synchronously.
  • Caching — cache moderation results for identical or near-identical prompts using a semantic hash.

False Positive Management

Blocking a legitimate user request is costly — it degrades trust, drives churn, and generates support tickets. False positive rate (FPR) is the single most important metric for a production content filter.

Reducing False Positives

  • Raise confidence thresholds (e.g., 0.85 instead of 0.5)
  • Require two independent layers to agree before blocking
  • Use allowlists for known-safe patterns (e.g., medical professionals discussing symptoms)
  • Log soft-blocks for human review rather than hard-blocking

Monitoring & Feedback Loops

  • Track block rate per category over time — sudden spikes indicate misclassification
  • Implement a user appeal flow — "I think this was blocked incorrectly"
  • Feed appeal data back into classifier fine-tuning
  • A/B test threshold changes on a shadow pipeline before deploying
The 1 % rule: In production, aim for a false positive rate below 1 %. At 10,000 requests/day, a 5 % FPR means 500 legitimate users blocked daily — enough to tank your product reputation. Instrument dashboards to track FPR in real time.

Production Deployment Patterns

Deploying content filters in production involves more than model inference. You need to handle streaming responses, graceful degradation when a filter service goes down, structured logging for audit trails, and configurable policies that safety teams can update without code changes.

Streaming-Compatible Output Filtering

Most LLM APIs stream tokens via SSE. You cannot run a classifier on the full response until streaming is complete. The common patterns are:

  1. Buffer-and-release — collect the full response, filter, then stream to the client (adds latency equal to full generation time).
  2. Sliding-window — run the classifier on every N tokens with a sliding window; block mid-stream if a violation is detected.
  3. Post-stream audit — stream to the client immediately, but run the filter asynchronously. If flagged, retroactively remove the response and notify the user.
# Sliding-window output filter for streaming responses
from collections import deque

class StreamingOutputFilter:
    def __init__(self, window_size: int = 50, stride: int = 25):
        self.window_size = window_size
        self.stride = stride
        self.buffer = deque(maxlen=window_size)
        self.token_count = 0

    def ingest_token(self, token: str) -> dict:
        self.buffer.append(token)
        self.token_count += 1

        # Check at every stride interval
        if self.token_count % self.stride == 0 and len(self.buffer) >= self.window_size:
            window_text = " ".join(self.buffer)
            tox = screen_toxicity(window_text)
            if tox["blocked"]:
                return {"action": "stop_stream", "reason": tox["violations"]}

        return {"action": "continue"}

Graceful Degradation

When a moderation service is unreachable, you must decide: fail-open (allow the request through unfiltered) or fail-closed (block all requests until the service recovers). Most safety-critical applications choose fail-closed for output filters and fail-open with logging for input filters. 

# Circuit breaker for moderation service
import time

class ModerationCircuitBreaker:
    def __init__(self, failure_threshold: int = 5, reset_timeout: int = 60):
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.reset_timeout = reset_timeout
        self.last_failure_time = 0
        self.state = "closed"           # closed = healthy, open = tripped

    def call(self, func, text: str, fail_mode: str = "closed") -> dict:
        if self.state == "open":
            if time.time() - self.last_failure_time > self.reset_timeout:
                self.state = "half-open"  # allow one probe request
            else:
                return self._default_response(fail_mode)

        try:
            result = func(text)
            self.failure_count = 0
            self.state = "closed"
            return result
        except Exception:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.state = "open"
            return self._default_response(fail_mode)

    def _default_response(self, fail_mode: str) -> dict:
        if fail_mode == "closed":
            return {"blocked": True, "reason": "moderation_unavailable"}
        return {"blocked": False, "reason": "fail_open_degraded"}

Structured Logging & Audit Trail

Every moderation decision — allow, block, redact — must be logged with full context for compliance and debugging. Include the request ID, user ID, timestamp, which layer triggered, the category, the confidence score, and the action taken. 

# Structured moderation log entry
import json, time, uuid

def log_moderation_event(
    request_id: str,
    direction: str,         # "input" or "output"
    layer: str,
    category: str,
    score: float,
    action: str,            # "allow", "block", "redact"
    text_hash: str,         # SHA-256 of the text (never log raw text)
):
    event = {
        "event_id": str(uuid.uuid4()),
        "timestamp": time.time(),
        "request_id": request_id,
        "direction": direction,
        "layer": layer,
        "category": category,
        "score": round(score, 4),
        "action": action,
        "text_hash": text_hash,
    }
    # Write to structured logging pipeline (e.g., JSON lines to Kafka)
    print(json.dumps(event))
Compliance note: Many jurisdictions (GDPR, CCPA) require that you can explain why a user's request was blocked. Structured logs with category, score, and layer information make this audit trail trivial to produce.

Putting It All Together

A production-grade content filtering system is not a single model — it is an orchestration layer that combines fast rules, ML classifiers, PII engines, and optional LLM judges behind circuit breakers, with every decision logged. The key takeaways:

  • Defense in depth — no single layer is sufficient. Stack regex, ML, and LLM layers.
  • Latency-aware design — order filters by cost, short-circuit on block, sample expensive layers.
  • Policy as config — decouple category taxonomies and thresholds from code.
  • False positive obsession — track FPR per category, build appeal flows, A/B test threshold changes.
  • Fail-safe defaults — use circuit breakers; decide fail-open vs. fail-closed per layer.
  • Audit everything — structured logs with request ID, category, score, and action for every decision.
