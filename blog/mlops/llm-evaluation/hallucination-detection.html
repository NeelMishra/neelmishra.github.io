Hallucination Detection and Mitigation

MLOps Series LLM Evaluation & Safety

Large Language Models generate fluent, confident-sounding text — but they routinely fabricate facts, invent citations, and contradict their own source material. These hallucinations are arguably the single biggest obstacle to deploying LLMs in production systems where accuracy matters. This post dissects why hallucinations happen, how to detect them algorithmically, and what mitigation strategies actually work at scale.

What Are LLM Hallucinations?

A hallucination occurs when a language model generates content that is nonsensical, unfaithful to the source, or factually incorrect — yet presents it with the same confidence as accurate information. Unlike traditional software bugs, hallucinations are an emergent property of how autoregressive models work: they predict the next most probable token, not the next most truthful one.

Why Do LLMs Hallucinate?

  • Training data noise: Pre-training corpora contain contradictions, outdated facts, and fabricated content that the model absorbs indiscriminately.
  • Exposure bias: During training the model always sees ground-truth prefixes, but at inference it conditions on its own (possibly wrong) outputs — errors compound.
  • Knowledge gaps: When the model lacks information, the softmax distribution still assigns probability mass to plausible-sounding continuations.
  • Compression artifacts: Billions of facts compressed into fixed-size parameters leads to lossy recall, especially for long-tail knowledge.
  • Instruction tuning pressure: RLHF and instruction tuning reward helpfulness, which can incentivise the model to fabricate answers rather than admit ignorance.
Real-world impact: In 2023, a lawyer submitted a legal brief containing six fabricated case citations generated by ChatGPT. Healthcare chatbots have provided incorrect dosage information. Financial models have hallucinated earnings figures. Hallucination is not an edge case — it is a systemic risk.

Hallucination Taxonomy

Not all hallucinations are created equal. A useful taxonomy helps you choose the right detection strategy for each failure mode.

Intrinsic Hallucination

The generated output contradicts the provided source material. For example, a summariser states "revenue fell 12%" when the source document says "revenue grew 12%".

Detection: Easier — compare output against source with NLI or entailment checking.

Extrinsic Hallucination

The output contains claims that cannot be verified from the source — neither supported nor contradicted. The model invents plausible-sounding but ungrounded facts.

Detection: Harder — requires external knowledge bases or multi-sample consistency checks.

Closed-Domain Hallucination

Occurs in tasks with a defined reference (summarisation, QA over documents). The ground truth is available for comparison.

Examples: RAG pipelines returning answers not in retrieved chunks, document summaries adding fabricated details.

Open-Domain Hallucination

Occurs in free-form generation where no single reference exists. The model must rely on parametric knowledge alone.

Examples: Chatbot inventing biographical facts, code assistant hallucinating non-existent API methods.

Key insight: Closed-domain hallucination is more tractable because you have a reference to compare against. Open-domain hallucination requires sampling-based or retrieval-augmented approaches for detection.

Detection Methods

Hallucination detection methods fall into three broad families: NLI-based entailment checking, self-consistency via multi-sampling, and knowledge-grounded verification. Each has different trade-offs in latency, cost, and coverage.

1. NLI-Based Detection

Natural Language Inference (NLI) models classify whether a hypothesis is entailed by, contradicts, or is neutral to a premise. By treating the source as the premise and each generated claim as a hypothesis, you get a per-claim faithfulness score.

# NLI-based hallucination detection from transformers import pipeline nli = pipeline("text-classification", model="cross-encoder/nli-deberta-v3-large") def detect_hallucination_nli(source: str, claims: list[str]) -> list[dict]: """Score each claim against the source document.""" results = [] for claim in claims: pair = f"{source} [SEP] {claim}" scores = nli(pair) label = scores[0]["label"] # ENTAILMENT | CONTRADICTION | NEUTRAL conf = scores[0]["score"] results.append({ "claim": claim, "label": label, "confidence": round(conf, 3), "hallucinated": label != "ENTAILMENT" }) return results # Example usage source = "Apple reported Q3 2024 revenue of $85.8 billion, up 5% year-over-year." claims = [ "Apple's Q3 2024 revenue was $85.8 billion.", # ✓ Entailed "Revenue declined 3% compared to Q3 2023.", # ✗ Contradiction "Tim Cook announced a new product line.", # ? Neutral (extrinsic) ] for r in detect_hallucination_nli(source, claims): print(f"{r['claim'][:50]:<50} {r['label']:<15} hall={r['hallucinated']}")

2. Self-Consistency Checking

Sample multiple completions at non-zero temperature and measure agreement. If the model's own responses contradict each other, at least some of them must be hallucinated. This works for open-domain scenarios where no reference document exists.

# Self-consistency hallucination detection import numpy as np from openai import OpenAI from sentence_transformers import SentenceTransformer client = OpenAI() embedder = SentenceTransformer("all-MiniLM-L6-v2") def self_consistency_check(prompt: str, n_samples: int = 5, temp: float = 0.7) -> dict: """Sample multiple responses and measure semantic agreement.""" responses = [] for _ in range(n_samples): resp = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=temp, max_tokens=300 ) responses.append(resp.choices[0].message.content) # Compute pairwise cosine similarities embeddings = embedder.encode(responses) sim_matrix = np.inner(embeddings, embeddings) norms = np.linalg.norm(embeddings, axis=1, keepdims=True) sim_matrix = sim_matrix / (norms @ norms.T) # Average off-diagonal similarity = consistency score n = len(responses) mask = ~np.eye(n, dtype=bool) consistency = float(sim_matrix[mask].mean()) return { "consistency_score": round(consistency, 3), "likely_hallucinating": consistency < 0.75, "n_samples": n_samples, "responses": responses } result = self_consistency_check("What year was the Treaty of Tordesillas signed?") print(f"Consistency: {result['consistency_score']} Hallucinating: {result['likely_hallucinating']}")

3. Knowledge-Grounded Verification

Retrieve relevant documents from a trusted knowledge base (Wikipedia, internal docs, verified databases) and cross-check generated claims against them. This is the foundation of retrieval-augmented verification — essentially RAG applied to the output rather than the input.

Trade-off summary: NLI detection is fast and cheap but needs a reference document. Self-consistency is reference-free but requires multiple API calls (≥3×). Knowledge grounding offers the best accuracy but adds retrieval latency and infrastructure.

SelfCheckGPT & Related Tools

SelfCheckGPT (Manakul et al., 2023) formalises the self-consistency approach. The core idea: if an LLM knows a fact, it will consistently reproduce it across stochastic samples. If it's hallucinating, sampled responses will diverge.

SelfCheckGPT supports multiple scoring backends:

  • BERTScore: Computes token-level semantic similarity between the original sentence and each sample.
  • NLI: Uses an entailment model to check if samples support each sentence in the original response.
  • N-gram: Measures lexical overlap — fast but less robust to paraphrasing.
  • LLM-as-judge: Uses another LLM to evaluate whether the original sentence is supported by the samples.

Related Hallucination Detection Tools

🔍 HHEM (Vectara)

Hughes Hallucination Evaluation Model. A cross-encoder fine-tuned specifically for hallucination detection. Provides a single factual consistency score for a source–summary pair.

🔍 FActScore

Decomposes generation into atomic facts, then verifies each against a knowledge source. Fine-grained but computationally expensive. Ideal for biographical and factoid-heavy text.

🔍 RAGAS Faithfulness

Part of the RAGAS framework for RAG evaluation. Extracts claims from the answer and checks each against retrieved context chunks using NLI. Production-ready and easy to integrate.

🔍 ChainPoll

Runs multiple LLM-as-judge evaluations and aggregates via majority vote. Reduces single-judge variance. Works well as a complement to embedding-based methods.

Mitigation Strategies

Detection tells you when hallucination occurs. Mitigation strategies aim to prevent it from happening in the first place — or at least reduce its frequency and severity.

1. Retrieval-Augmented Generation (RAG)

Ground every generation step in retrieved evidence. By providing relevant documents in the context window, you shift the model from parametric recall (unreliable) to contextual reading (verifiable). Effective RAG requires high-quality retrieval — garbage in, hallucination out.

2. Citation Generation

Force the model to cite specific passages for each claim. This makes hallucinations immediately auditable. Systems like Perplexity and Bing Chat use inline citations to provide verifiability.

# Citation-enforced generation prompt system_prompt = """You are a research assistant. For EVERY factual claim, include an inline citation in the format [Source: chunk_id]. If you cannot find a supporting source, write [UNVERIFIED]. NEVER fabricate a citation.""" def generate_with_citations(query: str, context_chunks: list[dict]) -> str: # Format retrieved chunks with IDs context = "\n".join( f"[{c['id']}] {c['text']}" for c in context_chunks ) messages = [ {"role": "system", "content": system_prompt}, {"role": "user", "content": f"Context:\n{context}\n\nQuestion: {query}"} ] resp = client.chat.completions.create( model="gpt-4o", messages=messages, temperature=0.0 ) return resp.choices[0].message.content

3. Constrained Decoding

Restrict the model's output space to valid tokens. Techniques include structured output formats (JSON mode), grammar-constrained generation (e.g., Outlines, Guidance), and entity-linking during decoding.

4. Chain-of-Verification (CoVe)

The model generates an initial response, then generates follow-up verification questions about its own claims, answers those questions independently, and revises the original response based on any inconsistencies found.

5. Fine-Tuning for Honesty

Train the model to say "I don't know" when uncertain. Direct Preference Optimization (DPO) with preference pairs that reward honest uncertainty over confident fabrication. This fights the RLHF-induced pressure to always produce an answer.

No silver bullet: No single mitigation technique eliminates hallucination. Production systems layer multiple strategies — RAG for grounding, citations for auditability, NLI for automated detection, and human review for high-stakes outputs.

Confidence Calibration

A well-calibrated model assigns high confidence to correct predictions and low confidence to incorrect ones. LLMs are notoriously overconfident — they produce hallucinations with the same fluency and assertiveness as correct answers.

Measuring Calibration

The Expected Calibration Error (ECE) bins predictions by confidence and measures the gap between predicted probability and actual accuracy in each bin. A perfectly calibrated model has ECE = 0.

# Token-level confidence extraction def get_token_confidence(prompt: str) -> list[dict]: """Extract per-token log probabilities as a proxy for confidence.""" resp = client.chat.completions.create( model="gpt-4o", messages=[{"role": "user", "content": prompt}], temperature=0.0, logprobs=True, top_logprobs=5 ) tokens = resp.choices[0].logprobs.content return [ {"token": t.token, "logprob": t.logprob, "prob": round(2.718 ** t.logprob, 4)} # e^logprob for t in tokens ] # Low-confidence tokens often correlate with hallucinated spans tokens = get_token_confidence("What is the capital of Burkina Faso?") for t in tokens: flag = " ⚠️" if t["prob"] < 0.5 else "" print(f"{t['token']:<15} p={t['prob']}{flag}")
Practical tip: Token-level log probabilities can serve as lightweight hallucination detectors. Flag spans where average token probability drops below a threshold. This is much cheaper than multi-sample methods but less reliable for semantic hallucinations.

Hallucination Benchmarks

Standardised benchmarks let you compare hallucination rates across models and evaluate your detection methods. The two most widely used benchmarks are:

📊 TruthfulQA

817 questions designed to elicit imitative falsehoods — answers that a human might give based on common misconceptions. Models are scored on both truthfulness and informativeness.

Key insight: Larger models are often less truthful on TruthfulQA because they better learn human misconceptions from training data.

📊 HaluEval

A large-scale benchmark with 35,000 samples spanning QA, dialogue, and summarisation. Each sample includes both hallucinated and correct responses, allowing binary classification evaluation.

Key insight: Even GPT-4 struggles on HaluEval's adversarially constructed examples, detecting only ~60-70% of hallucinations.

Other Notable Benchmarks

  • FaithDial: Evaluates faithfulness in knowledge-grounded dialogue systems.
  • FEVER: Fact Extraction and VERification — 185K claims classified as SUPPORTED, REFUTED, or NOT ENOUGH INFO against Wikipedia.
  • BEGIN: Benchmarking Natural Language Generation with reference-based and reference-free hallucination metrics.
  • FactCC / DAE: Specialised benchmarks for factual consistency in summarisation.
Evaluation tip: Always evaluate hallucination detection on multiple benchmarks. A detector that excels on TruthfulQA (open-domain) may perform poorly on FactCC (summarisation), and vice versa. Domain-specific evaluation is essential.

Production Hallucination Monitoring

Detecting hallucinations in a benchmark is one thing. Catching them in a live system with thousands of requests per minute is another entirely. Production monitoring requires automated pipelines that balance accuracy, latency, and cost.

Architecture Principles

  • Async verification: Run hallucination checks asynchronously to avoid adding latency to the user-facing response path. Log results and alert on threshold breaches.
  • Tiered detection: Use a fast, cheap detector (logprob thresholding) for 100% of traffic and an expensive detector (NLI + multi-sample) for flagged or sampled responses.
  • Claim-level tracking: Decompose each response into atomic claims and track hallucination rates per claim type, per topic, and per model version.
  • Feedback loops: Route flagged hallucinations to human reviewers. Use their annotations to fine-tune your detection models and improve the LLM's honesty over time.

Key Metrics to Track

📈 Hallucination Rate

Percentage of responses containing ≥1 hallucinated claim. Track as a time series, broken down by model version, prompt template, and topic domain.

📈 Faithfulness Score

Average NLI entailment score between generated claims and source documents. Target > 0.9 for production RAG systems.

📈 Detection Precision / Recall

How accurately your automated detector catches real hallucinations vs. false alarms. Calibrate thresholds using human-labelled samples.

📈 Time-to-Detection

How quickly hallucinations are caught after generation. Async pipelines should flag within seconds; batch analysis within minutes.

Operational insight: Hallucination rates tend to spike after model updates, prompt template changes, or shifts in user query distribution. Set up automated regression tests that run your hallucination benchmark suite against every new model or prompt deployment.

Hallucination remains an active area of research with no complete solution. The practical approach for production systems is defence in depth: ground with retrieval, verify with NLI, monitor with automated pipelines, and always keep a human in the loop for high-stakes decisions. As detection methods mature and models improve their calibration, we'll see hallucination rates decline — but the need for robust verification infrastructure will persist.
