Human Evaluation: Protocols and Scaling

MLOps Series LLM Evaluation & Safety

Why Human Evaluation Matters

Automated metrics like BLEU, ROUGE, and perplexity capture surface-level text similarity but fail to measure fluency, factual correctness, helpfulness, and safety—the qualities users actually care about. Human evaluation remains the gold standard for assessing open-ended LLM outputs where no single correct answer exists.

Key insight: Chatbot Arena showed that Elo ratings from real user preferences diverge significantly from automated benchmark rankings. GPT-4 leads human preferences by a wider margin than its MMLU score alone would suggest.

Human evaluation is essential when you need to assess subjective quality (tone, creativity), safety alignment (refusal behavior, bias detection), and task-specific correctness (medical, legal, or code review outputs).

Rating Scale Design

The choice of rating scale directly impacts annotator consistency and the statistical power of your evaluation. Three primary paradigms dominate LLM evaluation.

Likert Scales (Absolute Rating)

Annotators assign a score on a fixed scale (e.g., 1–5) for each response along specific dimensions. Common dimensions include fluency, relevance, coherence, and factual accuracy.

Pitfall: 5-point scales suffer from central tendency bias—annotators cluster around 3. Consider 4-point scales (no neutral) to force clear judgments, or binary scales for high-throughput tasks.

Pairwise Comparison

Annotators see two model outputs side-by-side and choose which is better (A, B, or Tie). This is cognitively easier than absolute scoring and produces more reliable results. LMSYS Chatbot Arena is the most prominent example of pairwise evaluation at scale.

A/B Testing (Online)

Real users interact with different model variants and implicit signals (engagement, completion rate, thumbs-up/down) serve as evaluations. This captures real-world preferences but requires production traffic and careful experiment design.

✅ Likert Pros

  • Multi-dimensional scoring
  • Fine-grained signal per sample
  • Good for regression testing

✅ Pairwise Pros

  • Lower cognitive load
  • Higher inter-annotator agreement
  • Naturally produces Elo/Bradley-Terry rankings

Designing Annotation Guidelines

Clear, unambiguous guidelines are the single most important factor in evaluation quality. A well-written guideline document should include:

  1. Task definition — Exactly what the annotator is evaluating and why.
  2. Scoring rubric — Concrete descriptions for each score level, with examples.
  3. Edge case catalog — How to handle partial correctness, refusals, code outputs, multi-turn context.
  4. Gold examples — Pre-labeled samples that establish ground truth and anchor calibration.
  5. Counter-examples — Common mistakes and how the scoring should differ.
Best practice: Run a calibration session with 3–5 annotators before launching. Have everyone label the same 20 samples, then discuss disagreements live. This typically improves agreement by 15–25% on subsequent batches.

Guidelines should be versioned alongside the evaluation data. When you update a rubric, note what changed and re-calibrate annotators on the delta. Track guideline version in your annotation metadata so you can stratify results.

Inter-Annotator Agreement

Agreement metrics quantify whether annotators interpret guidelines consistently. Low agreement signals ambiguous guidelines, insufficient training, or inherently subjective tasks.

Cohen’s Kappa (κ)

Measures agreement between two annotators, correcting for chance. Values range from −1 to 1:

  • < 0.20 — Poor agreement
  • 0.21–0.40 — Fair
  • 0.41–0.60 — Moderate
  • 0.61–0.80 — Substantial
  • 0.81–1.00 — Almost perfect
import numpy as np from sklearn.metrics import cohen_kappa_score # Ratings from two annotators on 100 samples (1-5 Likert) annotator_a = [4, 3, 5, 2, 4, 3, 5, 4, 2, 3] # ... 100 total annotator_b = [4, 3, 4, 2, 5, 3, 5, 3, 2, 4] # ... 100 total # Unweighted kappa (nominal categories) kappa = cohen_kappa_score(annotator_a, annotator_b) print(f"Cohen's kappa: {kappa:.3f}") # Weighted kappa (ordinal scales — penalizes large disagreements more) kappa_w = cohen_kappa_score(annotator_a, annotator_b, weights="quadratic") print(f"Weighted kappa: {kappa_w:.3f}")

Krippendorff’s Alpha (α)

Generalizes to any number of annotators, handles missing data, and supports multiple measurement levels (nominal, ordinal, interval, ratio). Preferred for production annotation pipelines.

import krippendorff # Reliability data: rows = annotators, columns = items # Use np.nan for missing annotations reliability_data = [ [4, 3, 5, 2, 4, 3], # annotator 1 [4, 3, 4, 2, 5, 3], # annotator 2 [3, 3, 5, 1, 4, 3], # annotator 3 ] alpha = krippendorff.alpha( reliability_data=reliability_data, level_of_measurement="ordinal" ) print(f"Krippendorff's alpha: {alpha:.3f}") # Interpretation: # α ≥ 0.80 → reliable annotations # 0.67 ≤ α < 0.80 → tentatively acceptable # α < 0.67 → guidelines need revision
Common mistake: Using raw percent agreement instead of kappa/alpha. If two annotators both default to “4” on 60% of items, percent agreement is high but kappa reveals the agreement is largely due to chance.

Handling Disagreements

When annotators disagree, you have several resolution strategies:

  • Majority vote — Add a third annotator and take the mode. Simple but expensive.
  • Adjudication — An expert reviews disputed items. Higher quality but bottleneck-prone.
  • Soft labels — Keep the distribution of ratings as a label (e.g., [0.2, 0.6, 0.2]). Better for training reward models.
  • Discussion rounds — Annotators discuss and converge. Best for guideline refinement, not scalable for production.

Annotation Platforms

Choosing the right platform depends on your data sensitivity, volume, annotator expertise needs, and budget.

Scale AI

  • Managed workforce with quality SLAs
  • RLHF-specific pipelines (comparison, ranking)
  • SOC 2 compliant; enterprise-grade security
  • Higher cost ($15–50+ per hour per annotator)

Surge AI

  • Specialized NLP workforce
  • Pre-vetted annotators for LLM tasks
  • Custom task interfaces
  • Focus on data quality over volume

Labelbox

  • Strong collaboration & workflow tools
  • Model-assisted labeling (pre-annotation)
  • Built-in agreement metrics
  • Bring-your-own-workforce option

Open-Source (Label Studio, Argilla)

  • Full data control — on-prem deployment
  • Free for any volume
  • Community plugins and ML backends
  • Requires managing your own annotators

Quality Control & Calibration

Even skilled annotators drift over time. A robust QC pipeline keeps quality consistent across thousands of annotations.

Calibration Sessions

Before each batch, annotators label 10–20 gold items with known answers. Their accuracy on these items determines whether they proceed. Target ≥85% gold accuracy for most tasks.

Honeypot Injection

Mix 5–10% gold items into production batches. If an annotator's honeypot accuracy drops below threshold, flag their batch for review or exclude them from the pool.

def compute_annotator_quality(annotations, gold_labels): """Flag annotators below quality threshold.""" annotator_scores = {} for annotator_id, items in annotations.items(): gold_items = [i for i in items if i["item_id"] in gold_labels] if not gold_items: continue correct = sum( 1 for g in gold_items if g["label"] == gold_labels[g["item_id"]] ) accuracy = correct / len(gold_items) annotator_scores[annotator_id] = { "accuracy": accuracy, "flagged": accuracy < 0.85, "total_gold": len(gold_items) } return annotator_scores

Time-Based Outlier Detection

Annotators who complete tasks significantly faster than the median are likely not reading carefully. Track median time per item and flag those below 50% of the median for review.

Production tip: Combine honeypot accuracy, time monitoring, and agreement scores into a single annotator trust score. Weight final labels by trust score rather than treating all annotators equally.

def estimate_eval_cost( n_samples, time_per_sample_min, annotators_per_sample, hourly_rate, platform_markup=1.3 ): """Estimate total human evaluation cost.""" total_annotations = n_samples * annotators_per_sample total_hours = (total_annotations * time_per_sample_min) / 60 base_cost = total_hours * hourly_rate total_cost = base_cost * platform_markup # platform overhead return { "total_annotations": total_annotations, "total_hours": round(total_hours, 1), "base_cost": round(base_cost, 2), "total_with_markup": round(total_cost, 2) } # Example: 1000 samples, 2 min each, 3 annotators, $20/hr cost = estimate_eval_cost(1000, 2, 3, 20) print(cost) # {'total_annotations': 3000, 'total_hours': 100.0, # 'base_cost': 2000.00, 'total_with_markup': 2600.00}
Budget trap: Teams often budget only for initial labeling and forget the cost of calibration rounds, guideline iterations, adjudication of disagreements, and re-labeling after rubric changes. Plan for 1.5–2× the raw annotation cost.

Scaling Human Evaluation

Moving from ad-hoc annotation to a repeatable evaluation pipeline requires process engineering.

Stratified Sampling

You rarely need to evaluate every output. Use stratified sampling across prompt categories, difficulty levels, and model variants to get statistically meaningful results with fewer annotations.

from sklearn.model_selection import StratifiedShuffleSplit def sample_for_evaluation(outputs, categories, n_per_category=50): """Sample n items per category for balanced evaluation.""" sampled = [] for cat in set(categories): cat_items = [o for o, c in zip(outputs, categories) if c == cat] n = min(n_per_category, len(cat_items)) sampled.extend(random.sample(cat_items, n)) return sampled # Evaluate 50 samples per category instead of all 10,000 eval_set = sample_for_evaluation(model_outputs, prompt_categories)

Annotation Batching & Parallelism

Split large evaluation jobs into small batches (50–100 items). Assign overlapping subsets for agreement measurement. This lets you detect quality drift within a batch run and scale across many annotators simultaneously.

Continuous Evaluation Pipelines

Integrate human evaluation into your CI/CD-like workflow:

  1. Model produces outputs on a fixed evaluation prompt set.
  2. Outputs are auto-routed to the annotation platform via API.
  3. Annotators label the batch; agreement metrics are computed.
  4. Results are logged to your experiment tracker (W&B, MLflow).
  5. Dashboards show quality trends over model versions.
Scale rule of thumb: For pairwise comparisons, ~400 comparisons give tight 95% confidence intervals on win rates. For Likert ratings, ~200 samples per condition with 3 annotators each are typically sufficient for statistical significance.

Hybrid Human + Auto Approaches

The most cost-effective production systems combine automated evaluation with targeted human review. This hybrid approach maximizes coverage while controlling costs.

LLM-as-Judge for Triage

Use a strong LLM (e.g., GPT-4) as a first-pass evaluator. Route outputs where the judge has low confidence or that touch sensitive topics to human annotators. This typically reduces human annotation volume by 70–90% while maintaining quality on the hard cases.

def hybrid_evaluation_pipeline(outputs, auto_threshold=0.9): """Route outputs through auto → LLM judge → human review.""" results = [] human_queue = [] for output in outputs: # Stage 1: fast automated checks auto_score = run_auto_metrics(output) # BLEU, toxicity, etc. if auto_score["confidence"] >= auto_threshold: results.append({"output": output, "eval": auto_score, "source": "auto"}) continue # Stage 2: LLM-as-judge for uncertain cases judge_result = llm_judge_evaluate(output) if judge_result["confidence"] >= 0.8: results.append({"output": output, "eval": judge_result, "source": "llm_judge"}) else: # Stage 3: route to human annotators human_queue.append(output) # Send human_queue to annotation platform submit_to_platform(human_queue, platform="scale_ai") return results, human_queue

Active Learning for Annotation Efficiency

Prioritize human annotation on samples where the model is most uncertain or where automated metrics disagree with each other. This active learning strategy maximizes the information gained per annotation dollar.

⚡ Hybrid Pipeline

Calibrating the judge: Periodically run the same samples through both LLM-as-judge and human annotators. Compute agreement (kappa/alpha) between the judge and humans. If agreement drops below 0.7, retune your judge prompt or switch models.

Human evaluation isn’t going away—it’s becoming more targeted. The key is building pipelines where humans focus on the high-value decisions that only they can make, while automated systems handle the rest.
