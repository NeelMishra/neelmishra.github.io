LLM Evaluation Pipelines: CI/CD for Quality

MLOps Series LLM Evaluation & Safety

Eval Pipelines Overview

Manual model evaluation doesn't scale. Every prompt template change, fine-tune iteration, or retrieval config update can silently degrade output quality. An evaluation pipeline embeds quality checks directly into your development workflow, catching regressions before they reach users.

The core loop: change → trigger → evaluate → gate → deploy. Every component in this pipeline is automatable and measurable.

  • Deterministic evals — exact-match, BLEU, ROUGE, code-pass-rate — run every time.
  • LLM-as-judge evals — coherence, helpfulness, safety — run on a sampled subset for cost control.
  • Human-in-the-loop evals — triggered weekly or before major releases, results feed back into automated benchmarks.
Key principle: Treat evaluation datasets as code. Version them alongside prompts and model configs in the same repository so every commit has a reproducible evaluation snapshot.

CI/CD Integration

The most practical entry point is GitHub Actions or GitLab CI. The workflow triggers on code changes that affect model behavior: prompt templates, retrieval configs, fine-tuned model references, or evaluation datasets themselves.

GitHub Actions Workflow

# .github/workflows/llm-eval.yml
name: LLM Evaluation Pipeline
on:
  pull_request:
    paths:
      - 'prompts/**'
      - 'configs/model*.yaml'
      - 'evals/**'
  push:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'  # Weekly full eval Monday 6 AM
  workflow_dispatch:        # Manual trigger for ad-hoc runs

env:
  EVAL_CACHE_DIR: .eval-cache
  WANDB_PROJECT: llm-eval-pipeline

jobs:
  eval-deterministic:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements-eval.txt
      - name: Run deterministic evals
        run: |
          python -m evals.run \
            --suite deterministic \
            --output results/deterministic.json \
            --cache-dir $EVAL_CACHE_DIR
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
      - uses: actions/upload-artifact@v4
        with:
          name: deterministic-results
          path: results/

  eval-llm-judge:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements-eval.txt
      - name: Run LLM-as-judge evals
        run: |
          python -m evals.run \
            --suite llm_judge \
            --sample-rate 0.3 \
            --output results/llm_judge.json
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

  gate-check:
    needs: [eval-deterministic, eval-llm-judge]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v4
      - name: Aggregate and gate
        run: python -m evals.gate --config evals/gate_config.yaml

Trigger Strategies

🔀 On Pull Request

Run lightweight evals on every PR that touches prompt templates, model configs, or eval datasets. Fast feedback loop — aim for <5 minute wall-clock time. Use sampling (30% of eval set) and result caching to stay within budget.

🚀 On Merge to Main

Run the full deterministic eval suite on every merge. This is your quality gate before deployment. No sampling — evaluate every test case. Results become the baseline for future PR comparisons.

📅 Scheduled (Cron)

Weekly full evaluation including expensive LLM-as-judge runs and human-eval aggregation. Catches API-provider-side regressions (model updates, rate limit changes) that code changes wouldn't trigger.

🔧 Manual Dispatch

On-demand full evaluation for pre-release validation or debugging. Accepts parameters: model override, eval suite selection, sample rate. Essential for A/B testing new model versions.

Cost trap: Running LLM-as-judge evals on every PR against the full dataset can cost $50-200 per run. Always set a --sample-rate for PR-level evals and reserve full runs for merge and scheduled triggers.

Evaluation Gating Criteria

A gate is a binary pass/fail decision that blocks or allows deployment. Effective gates combine absolute thresholds (minimum acceptable quality) with regression detection (comparison against baseline).

Gate Configuration

# evals/gate_config.yaml
gates:
  exact_match:
    metric: accuracy
    min_threshold: 0.85          # Absolute floor
    max_regression: 0.02        # Max drop vs baseline
    blocking: true

  llm_judge_coherence:
    metric: coherence_score
    min_threshold: 0.78
    max_regression: 0.05
    blocking: true

  safety_score:
    metric: safety_pass_rate
    min_threshold: 0.99          # Near-zero tolerance
    max_regression: 0.001
    blocking: true

  latency_p95:
    metric: p95_latency_ms
    max_threshold: 3000
    max_regression_pct: 10    # 10% increase allowed
    blocking: false            # Warning only

baseline:
  source: wandb               # Pull baseline from W&B
  project: llm-eval-pipeline
  tag: production-baseline

Gate Evaluation Script

# evals/gate.py
import json, yaml, sys

def load_results(results_dir):
    """Aggregate results from all eval suite outputs."""
    combined = {}
    for f in Path(results_dir).glob("*.json"):
        with open(f) as fh:
            combined.update(json.load(fh))
    return combined

def check_gates(results, config, baseline):
    failures = []
    warnings = []
    for name, gate in config["gates"].items():
        value = results.get(gate["metric"])
        if value is None:
            failures.append(f"{name}: metric missing")
            continue

        # Absolute threshold check
        if "min_threshold" in gate and value < gate["min_threshold"]:
            msg = f"{name}: {value:.3f} < {gate['min_threshold']}"
            (failures if gate["blocking"] else warnings).append(msg)

        # Regression check against baseline
        base_val = baseline.get(gate["metric"])
        if base_val and "max_regression" in gate:
            drop = base_val - value
            if drop > gate["max_regression"]:
                msg = f"{name}: regression {drop:.3f}"
                (failures if gate["blocking"] else warnings).append(msg)

    return failures, warnings

# Main: exit 1 on any blocking failure
results = load_results("results/")
config  = yaml.safe_load(open("evals/gate_config.yaml"))
baseline = fetch_baseline(config["baseline"])
fails, warns = check_gates(results, config, baseline)
for w in warns: print(f"⚠️  {w}")
for f in fails: print(f"❌ {f}")
if fails:
    print(f"Gate FAILED: {len(fails)} blocking failures")
    sys.exit(1)
print("✅ All gates passed")
Baseline strategy: After every successful production deployment, tag the current eval results as the new baseline in W&B. This ensures the regression window always compares against what's actually running, not a stale snapshot.

Result Storage & Tracking

Raw eval results are ephemeral CI artifacts. For trend analysis, regression detection, and auditing, you need persistent storage with queryable metadata.

Weights & Biases Integration

import wandb

def log_eval_results(results, run_meta):
    """Log evaluation results to W&B for tracking and comparison."""
    run = wandb.init(
        project="llm-eval-pipeline",
        config={
            "model": run_meta["model_id"],
            "commit": run_meta["git_sha"],
            "trigger": run_meta["trigger_type"],  # pr, merge, scheduled
            "eval_suite": run_meta["suite"],
        },
        tags=[run_meta["trigger_type"], run_meta["suite"]],
    )

    # Log scalar metrics
    wandb.log({
        "accuracy": results["accuracy"],
        "coherence_score": results["coherence_score"],
        "safety_pass_rate": results["safety_pass_rate"],
        "p95_latency_ms": results["p95_latency_ms"],
        "eval_cost_usd": results["total_cost"],
    })

    # Log per-example results as a W&B Table
    table = wandb.Table(
        columns=["input", "expected", "actual", "score", "pass"]
    )
    for ex in results["examples"]:
        table.add_data(ex["input"], ex["expected"],
                       ex["actual"], ex["score"], ex["pass"])
    wandb.log({"eval_examples": table})

    # Tag production baseline on successful deploy
    if run_meta["trigger_type"] == "merge":
        run.tags += ("production-baseline",)

    run.finish()

Result Caching for Cost Control

import hashlib, json

def get_cache_key(prompt, model_id, eval_config):
    """Deterministic cache key for an eval input."""
    payload = json.dumps({
        "prompt": prompt,
        "model": model_id,
        "temperature": eval_config["temperature"],
        "max_tokens": eval_config["max_tokens"],
    }, sort_keys=True)
    return hashlib.sha256(payload.encode()).hexdigest()

def eval_with_cache(prompt, model_id, config, cache_dir):
    key = get_cache_key(prompt, model_id, config)
    cache_path = Path(cache_dir) / f"{key}.json"
    if cache_path.exists():
        return json.loads(cache_path.read_text())
    result = call_llm(prompt, model_id, config)
    cache_path.write_text(json.dumps(result))
    return result
Cache invalidation: Only cache results when temperature=0. Stochastic outputs produce different results each call — caching them gives misleading consistency. Invalidate the cache when the model version or API endpoint changes.

Dashboards & Visualization

Dashboards transform raw eval data into actionable insights. Two systems cover most needs: Weights & Biases for ML-specific analysis and Grafana for operational monitoring and alerting.

Grafana Alert Configuration

# Prometheus alert rules for eval pipeline
groups:
  - name: llm-eval-alerts
    rules:
      - alert: EvalAccuracyDrop
        expr: |
          llm_eval_accuracy{suite="deterministic"}
            < 0.85
        for: 0m   # Fire immediately on any run
        labels:
          severity: critical
        annotations:
          summary: "Eval accuracy below threshold"
          runbook: "https://wiki/runbooks/eval-accuracy"

      - alert: EvalSafetyRegression
        expr: |
          llm_eval_safety_pass_rate{suite="safety"}
            < 0.99
        for: 0m
        labels:
          severity: page    # Wake someone up

      - alert: EvalPipelineStale
        expr: |
          time() - llm_eval_last_run_timestamp
            > 172800       # No eval in 48 hours
        labels:
          severity: warning
Dashboard layering: Use W&B for deep-dive analysis during development — comparing runs, inspecting failure modes, sharing with the ML team. Use Grafana for production-facing alerts that integrate with your existing on-call (PagerDuty, Opsgenie) and SLO infrastructure.

Scaling Eval Pipelines

As your eval suite grows from 100 to 10,000+ test cases across multiple models, naive sequential execution becomes a bottleneck. Parallel execution, cost management, and intelligent scheduling are essential.

Parallel Evaluation with Matrix Strategy

# GitHub Actions matrix for parallel eval suites
jobs:
  eval-parallel:
    strategy:
      matrix:
        suite: [deterministic, coherence, safety, latency]
        model: [gpt-4o, claude-3.5-sonnet]
      fail-fast: false  # Run all even if one fails
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          python -m evals.run \
            --suite ${{ matrix.suite }} \
            --model ${{ matrix.model }} \
            --output results/${{ matrix.suite }}_${{ matrix.model }}.json

  aggregate:
    needs: [eval-parallel]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
      - run: python -m evals.aggregate --input results/ --output summary.json
      - run: python -m evals.gate --config evals/gate_config.yaml

Result Aggregation Script

# evals/aggregate.py
import json
from pathlib import Path
from collections import defaultdict

def aggregate_results(results_dir):
    """Merge parallel eval outputs into a single report."""
    by_suite = defaultdict(list)
    by_model = defaultdict(list)

    for f in Path(results_dir).glob("*.json"):
        data = json.loads(f.read_text())
        by_suite[data["suite"]].append(data)
        by_model[data["model"]].append(data)

    summary = {
        "total_examples": sum(d["count"] for d in by_suite.values()
                               for d2 in d),
        "total_cost_usd": sum(d["cost"] for d in by_suite.values()
                               for d2 in d),
        "suites": {},
        "models": {},
    }

    for suite, runs in by_suite.items():
        scores = [r["score"] for r in runs]
        summary["suites"][suite] = {
            "mean_score": sum(scores) / len(scores),
            "min_score": min(scores),
            "run_count": len(runs),
        }

    return summary

Cost Management Strategies

💰 Budget Controls

  • Set per-run cost limits ($5 for PR, $50 for full)
  • Track cumulative daily/weekly spend
  • Auto-downgrade to cheaper model if budget exceeded
  • Alert on cost anomalies (>2× average)

🧠 Smart Scheduling

  • Cache deterministic eval outputs (temperature=0)
  • Skip unchanged prompt/model combos
  • Run expensive suites only on merge/schedule
  • Use smaller judge models for PR-level checks
Production pattern: The most cost-effective pipeline uses a three-tier strategy — fast deterministic evals on every PR (~$0.50), full deterministic + sampled LLM-judge on merge (~$15), and comprehensive scheduled runs weekly (~$150). Total monthly cost for a team: $500-800 with proper caching.
