Eval Data Curation: High-Quality Test Sets

MLOps Series LLM Evaluation & Safety

Evaluation is only as trustworthy as the data it runs on. A high-quality eval dataset separates genuine model capability from overfitting, contamination artifacts, and benchmark gaming. Yet most practitioners underinvest in eval data curation—spending months on training data pipelines but only hours assembling test sets. The consequences are severe: contaminated benchmarks report inflated scores, narrow test distributions miss real-world failure modes, and poorly annotated labels introduce irreducible noise into every metric. This post covers the full lifecycle of eval data curation—from sampling strategies and difficulty calibration through contamination detection, diversity engineering, human review workflows, and long-term dataset maintenance.

Why Data Quality Matters

Every evaluation metric is a function of two inputs: the model’s predictions and the ground-truth labels. If the labels are noisy, even a perfect model will appear imperfect—and a flawed model can appear strong if its errors correlate with label noise. Research by Northcutt et al. (2021) found that major benchmarks like ImageNet, Amazon Reviews, and IMDB contain 3–10% label errors. In LLM evals, the problem is compounded by subjectivity: what constitutes a “good” summary or a “helpful” response depends on annotator judgment, creating an inherent noise floor.

Beyond labels, the distribution of eval data determines what you actually measure. A math benchmark dominated by arithmetic will tell you little about algebraic reasoning. A coding eval that only tests Python masks weaknesses in less-common languages. The eval dataset is itself a specification of what you care about—and building that specification carefully is a design decision, not a data collection afterthought.

Rule of Thumb: If your eval dataset has fewer than 500 examples, confidence intervals on accuracy are ±3–5%. Below 200 examples, the noise overwhelms the signal. For reliable comparisons between models, target at least 1,000 items per eval dimension—and always report confidence intervals alongside point estimates.

Data quality failures propagate non-linearly. A 5% mislabeling rate does not merely add 5% noise—it can systematically bias evaluations if certain categories are more error-prone. For instance, if annotators consistently mislabel sarcasm as toxic, then models that handle sarcasm well will be unfairly penalized. Understanding and controlling these failure modes requires deliberate dataset engineering, not just more data.

Dataset Design Principles

Sampling Strategies

The sampling strategy determines the composition and representativeness of your eval set. Naive uniform random sampling from a source corpus produces eval sets that mirror the corpus distribution—which typically overrepresents common, easy cases and underrepresents rare, hard ones. For evaluation purposes, this is rarely what you want. Three alternatives produce more informative benchmarks:

Stratified Sampling

Partition the source into strata (by topic, difficulty, format, language) and sample proportionally or equally from each stratum. This guarantees minimum coverage of each category. Use proportional allocation when you need a representative distribution; use equal allocation when you need reliable per-stratum estimates.

Adversarial Sampling

Actively seek examples that target known model weaknesses. Use error analysis from a baseline model to identify failure patterns, then mine the source for similar examples. This produces harder, more diagnostic evals but can overrepresent edge cases if not balanced with representative items.

Importance Sampling

Weight sampling toward items that are more informative for distinguishing between models. Items where strong models succeed and weak models fail are more discriminative than items where all models agree. Item Response Theory (IRT) can estimate item difficulty and discrimination parameters.

Temporal Sampling

For knowledge-dependent evals, sample across time periods to test temporal freshness. A model trained on data through 2023 should be evaluated on both pre-2023 knowledge (recall) and post-2023 events (knowledge cutoff awareness). Include time-stamped items explicitly.

Difficulty Calibration

A well-calibrated eval set spans a range of difficulty levels, producing a smooth score distribution rather than clustering near 0% or 100%. If the baseline model achieves 95%+ accuracy, the eval lacks headroom to differentiate improvements; if it achieves <20%, most items are uninformative noise. Target a baseline accuracy of 40–70% for maximum discrimination.

Difficulty calibration requires iterative piloting: assemble candidate items, run them against 2–3 reference models, compute per-item accuracy, then filter or re-weight. Items that all models get right (trivial) or all models get wrong (ambiguous or broken) contribute little information and should be replaced with items in the discriminative range.

Warning: Calibrating difficulty to a specific model risks overfitting the eval to that model’s error patterns. Always calibrate against a diverse set of models—at minimum, one open-source and one closed-source model of different sizes—so that the difficulty distribution reflects genuine task complexity rather than a single architecture’s blind spots.

Adversarial Example Construction

Adversarial examples probe robustness by introducing controlled perturbations that should not change the correct answer. Common techniques include paraphrasing questions while preserving semantics, swapping entity names to test for memorization versus reasoning, adding irrelevant context (distractor injection), and negating premises to verify logical sensitivity. Each adversarial category should comprise 10–20% of the total eval set, tagged with metadata so you can report robustness metrics separately from core accuracy.

Contamination Detection & Prevention

Data contamination occurs when eval examples (or close paraphrases) appear in the training corpus, inflating benchmark scores without reflecting genuine capability. This is the single biggest threat to eval integrity in the LLM era. GPT-4’s technical report flagged contamination rates of 1–25% across major benchmarks. When 10% of an eval set is contaminated, a model that memorizes those items gets a 5–10 point accuracy boost that has nothing to do with generalization.

N-gram Overlap Detection

The simplest contamination check computes n-gram overlap between eval items and the training corpus. A 13-gram match (13 consecutive tokens in common) is the threshold used by GPT-4 and LLaMA for flagging potential contamination. Shorter n-grams produce excessive false positives from common phrases; longer n-grams miss paraphrased leakage.

# N-gram contamination detection for eval datasets import hashlib from collections import defaultdict from typing import List, Set, Tuple class NgramContaminationDetector: def __init__(self, n: int = 13, hash_bits: int = 64): self.n = n self.hash_bits = hash_bits self.train_ngrams: Set[str] = set() def _tokenize(self, text: str) -> List[str]: # Lowercase + whitespace tokenization for consistency return text.lower().split() def _extract_ngrams(self, tokens: List[str]) -> List[str]: # Slide a window of size n across the token list return [ hashlib.md5( " ".join(tokens[i:i+self.n]).encode() ).hexdigest() for i in range(len(tokens) - self.n + 1) ] def index_training_data(self, documents: List[str]): # Build n-gram set from the full training corpus for doc in documents: tokens = self._tokenize(doc) for h in self._extract_ngrams(tokens): self.train_ngrams.add(h) def check_contamination(self, eval_item: str) -> Tuple[bool, float]: # Returns (is_contaminated, overlap_ratio) tokens = self._tokenize(eval_item) item_ngrams = self._extract_ngrams(tokens) if not item_ngrams: return False, 0.0 matches = sum(1 for h in item_ngrams if h in self.train_ngrams) ratio = matches / len(item_ngrams) # Flag if >20% of n-grams appear in training data return ratio > 0.2, ratio def audit_eval_set(self, eval_items: List[str]) -> dict: # Produce a contamination report for the full eval set results = {"total": len(eval_items), "contaminated": 0, "details": []} for idx, item in enumerate(eval_items): flagged, ratio = self.check_contamination(item) if flagged: results["contaminated"] += 1 results["details"].append({"index": idx, "overlap": ratio}) results["rate"] = results["contaminated"] / results["total"] return results

Embedding-Based Detection

N-gram methods miss semantically equivalent paraphrases. Embedding-based contamination detection encodes both training documents and eval items into dense vectors (using a sentence encoder like E5 or GTE), then flags eval items whose nearest training neighbor falls below a cosine-distance threshold (typically 0.15–0.25). This catches reformulations and translations that preserve meaning while changing surface form. The trade-off is computational cost: building a FAISS index over billions of training chunks requires significant infrastructure.

Warning: Embedding similarity can produce false positives when eval items test common knowledge (e.g., “What is the capital of France?”). Pair embedding checks with manual review of flagged items, and distinguish between factual overlap (acceptable) and verbatim leakage (disqualifying). Report contamination metrics both with and without common-knowledge items.

Prevention Strategies

Detection alone is insufficient—prevention is cheaper and more reliable. Key strategies include: (1) creating eval items after the training data cutoff date so they cannot appear in the corpus; (2) using private eval sets that are never published online; (3) generating synthetic eval items from templates with randomized parameters; (4) watermarking eval items with unique identifiers to trace leakage. Organizations like LMSYS (Chatbot Arena) use live human preference data that is inherently post-training, making contamination structurally impossible.

Diversity and Coverage

A diverse eval dataset ensures that performance scores reflect broad capability rather than narrow specialization. Diversity must be engineered across multiple axes simultaneously: topic domain, task format, difficulty level, demographic representation, linguistic register, and input length. Failing to balance any single axis creates blind spots that real-world deployment will expose.

Demographic Balance

Language models exhibit measurable performance disparities across demographic groups. An eval set that only tests English-language, Western-centric content will miss failures in other contexts. For fairness-sensitive applications, eval sets should include balanced representation of gender, ethnicity, age, and geographic context in the scenarios, names, and cultural references used in prompts. Use disaggregated evaluation—reporting metrics separately per demographic slice—to surface disparities that aggregate scores conceal.

Coverage Analysis

# Analyze eval dataset coverage across key dimensions import json from collections import Counter from typing import Dict, List class CoverageAnalyzer: DIMENSIONS = ["topic", "difficulty", "format", "language", "demographic"] def __init__(self, items: List[Dict]): self.items = items def compute_distribution(self, dim: str) -> Dict[str, float]: # Returns percentage distribution for a dimension counts = Counter(item.get(dim, "unknown") for item in self.items) total = sum(counts.values()) return {k: round(v / total * 100, 1) for k, v in counts.most_common()} def flag_imbalances(self, threshold: float = 0.05) -> List[str]: # Flag dimensions where any category is below threshold warnings = [] for dim in self.DIMENSIONS: dist = self.compute_distribution(dim) for cat, pct in dist.items(): if pct < threshold * 100: warnings.append(f"Under-represented: {dim}={cat} ({pct}%)") return warnings def generate_report(self) -> str: # Full coverage report as structured JSON report = {"total_items": len(self.items)} for dim in self.DIMENSIONS: report[dim] = self.compute_distribution(dim) report["warnings"] = self.flag_imbalances() return json.dumps(report, indent=2)

Run coverage analysis before finalizing the eval set. If STEM topics comprise 60% of your eval but only 30% of real queries, your benchmark overweights STEM performance. Conversely, if “hard” items are only 5%, the eval cannot distinguish between models at the frontier. Target roughly equal representation across difficulty tiers, and use real query-log distributions to weight topic proportions when measuring aggregate performance.

Best Practice: Compute cross-dimensional coverage: not just “how many coding items” and “how many hard items,” but “how many hard coding items.” A dataset can appear balanced on each dimension independently while having empty cells in the cross-product. Build a coverage matrix and fill sparse cells first.

Human Review Workflows

Human annotation is the gold-standard source of eval labels, but unmanaged annotation produces noisy, inconsistent data. Annotation quality control is a system design problem—it requires clear guidelines, calibration exercises, inter-annotator agreement monitoring, adjudication protocols, and feedback loops. Skipping any stage degrades label quality in predictable ways.

Annotation Quality Control

Guideline Design

Write detailed annotation guidelines with 15+ worked examples covering edge cases. Include both positive and negative examples for each label. Guidelines should be versioned and updated as ambiguities surface. Expect 3–5 revision cycles before guidelines stabilize.

Calibration Rounds

Before production annotation begins, run 2–3 calibration rounds where all annotators label the same 50 items, then discuss disagreements. This surfaces guideline ambiguities and aligns mental models. Track agreement improvement across rounds—proceed to production only when Cohen’s kappa exceeds 0.7.

Multi-Rater Redundancy

Assign each item to 3–5 independent annotators and take the majority label. For subjective tasks (helpfulness, safety), use 5+ raters. Track per-item agreement: items with zero or slim majority should be flagged for adjudication, not forced into a label by the majority rule.

Gold & Trap Items

Embed pre-labeled gold items (5–10% of the batch) with known correct answers. Monitor annotator accuracy on gold items in real-time. Flag annotators who fall below 85% gold accuracy for re-calibration or removal. This catches inattentive or adversarial annotators before they corrupt large batches.

Adjudication and Escalation

Items where annotators disagree reveal genuine ambiguity—and ambiguous items are the most informative part of your eval set. Rather than discarding them, route high-disagreement items to an expert adjudicator who provides a final label and writes a rationale. Store the rationale alongside the label: it becomes part of the dataset documentation and helps future annotators handle similar cases. For truly ambiguous items, consider multi-label ground truth (e.g., both “helpful” and “somewhat helpful” are acceptable) or a soft-label distribution reflecting annotator proportions.

Annotation Economics: Budget 2–5 minutes per item for complex evaluations (summarization quality, safety classification). At $15–25/hour for skilled annotators, a 2,000-item eval set with 3-way redundancy costs $3,000–$8,000 and takes 2–4 weeks. Plan for this timeline and budget from the start—annotation is rarely the bottleneck that gets cut last, but it should be.

Versioning and Maintenance

Eval datasets are living artifacts that degrade over time. Models train on data that includes old benchmarks; the world changes and eval items become outdated; annotation standards evolve. Without explicit versioning and maintenance practices, eval scores become meaningless within 1–2 years of a dataset’s release.

Dataset Cards

Every eval dataset should ship with a dataset card (following the Hugging Face template or the Datasheets for Datasets framework) that documents: the intended use, construction methodology, sampling strategy, contamination checks performed, annotator demographics and guidelines, known limitations, and version history. The card is not optional documentation—it is part of the dataset. Without it, downstream consumers cannot interpret scores or replicate results.

Temporal Freshness

Knowledge-dependent eval items have an expiration date. A question about “the current U.S. president” becomes ambiguous after an election. A coding eval that tests deprecated APIs measures memorization of obsolete patterns. Schedule quarterly reviews of time-sensitive items: retire stale items, add fresh replacements, and log the changes in the version history. Maintain a temporal tag on every item indicating whether it tests stable knowledge (math, logic) or time-dependent knowledge (current events, API syntax).

Warning: Never silently update an eval dataset in-place. If you modify even one item, increment the version number and document the change. Scores reported against v1.0 are not comparable to scores against v1.1. Use semantic versioning: patch versions for label corrections, minor versions for item additions/removals, major versions for methodology changes.

Maintenance Checklist

A systematic maintenance workflow prevents eval drift:

  • Monthly: Run automated contamination checks against newly released training corpora and fine-tuning datasets. Flag and quarantine newly contaminated items.
  • Quarterly: Review time-sensitive items for staleness. Refresh 10–15% of the dataset with new items that test recent knowledge and emerging capabilities.
  • Per-release: Re-run inter-annotator agreement on a random 10% sample. If agreement drops below the original threshold, investigate and update guidelines.
  • Annually: Conduct a full dataset audit: coverage analysis, contamination sweep, difficulty re-calibration, and annotator re-training. Publish an updated dataset card.

Version control for eval datasets should use content-addressable storage (hash the full dataset to produce a version fingerprint) alongside semantic version tags. Store eval datasets in dedicated repositories separate from code, with access controls that prevent accidental inclusion in training pipelines. Tools like DVC (Data Version Control) or LakeFS integrate dataset versioning with Git workflows, making it straightforward to track which eval version produced which scores.

Key Takeaway: High-quality eval datasets require the same engineering rigor as training data pipelines—sampling design, contamination prevention, diversity auditing, annotation QC, and continuous maintenance. The eval is the specification of what you care about: invest in it accordingly, or accept that your benchmarks measure noise rather than capability.
