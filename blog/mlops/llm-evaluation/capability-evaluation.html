Capability Eval: Reasoning, Coding, Domain

MLOps Series LLM Evaluation & Safety

A single accuracy number tells you almost nothing about an LLM. A model that scores 82% on a general benchmark might excel at factual recall while completely failing at multi-step reasoning. Capability-specific evaluation breaks performance into distinct dimensions — reasoning, coding, domain expertise, instruction following — and measures each independently. This gives you a capability profile rather than a single number, enabling targeted improvement and informed model selection.

Capability-Specific Evaluation

Monolithic benchmarks like MMLU or HellaSwag collapse many distinct skills into one score. Capability-specific evaluation disaggregates performance into axes that map to real deployment needs:

  • Reasoning: Chain-of-thought, logical deduction, mathematical proof, commonsense inference
  • Coding: Code generation, bug fixing, test writing, code explanation, refactoring
  • Domain knowledge: Medical, legal, financial, scientific accuracy
  • Instruction following: Format compliance, constraint adherence, length control
  • Safety & alignment: Refusal on harmful prompts, honesty, calibration
Why disaggregate? Two models with identical MMLU scores can have wildly different capability profiles. Model A may score 90% on reasoning but 60% on coding, while Model B does the reverse. Your use case determines which profile is better.

A well-designed evaluation framework scores each axis independently, then composes the results into a multi-dimensional profile. This approach surfaces capability gaps that aggregate scores hide.

Designing Multi-Dimensional Scoring Rubrics

Each capability axis needs a rubric with clearly defined levels. A rubric turns subjective judgment into reproducible scoring:

Reasoning Evaluation

Reasoning evaluation is the hardest to get right because reasoning quality is about the process, not just the final answer. A model that arrives at the correct answer through flawed reasoning is fragile — it will break on slight variations.

Chain-of-Thought Evaluation

Chain-of-thought (CoT) prompts ask the model to show its work. Evaluation checks both the final answer and the intermediate steps:

def evaluate_chain_of_thought(model, problem, expected_answer, expected_steps): # Generate with explicit CoT prompting response = model.generate( prompt=f"Solve step by step:\n{problem}", max_tokens=1024 ) # Score final answer (binary) final_correct = extract_final_answer(response) == expected_answer # Score reasoning chain (rubric-based) steps = parse_reasoning_steps(response) step_scores = [] for step, expected in zip(steps, expected_steps): score = grade_reasoning_step(step, expected) # 0-5 rubric step_scores.append(score) return { "final_correct": final_correct, "step_accuracy": mean(step_scores), "reasoning_completeness": len(steps) / len(expected_steps), "logical_coherence": check_logical_flow(steps) }

Multi-Step Reasoning

Multi-step problems test whether the model can maintain state and chain inferences across multiple hops. Key failure modes to evaluate:

  • State tracking errors: Forgetting intermediate results from earlier steps
  • Premise abandonment: Dropping constraints partway through the solution
  • Shortcut reasoning: Jumping to the answer without valid intermediate steps
  • Error propagation: One wrong step corrupting all subsequent reasoning

Logical Consistency

Logical consistency tests present the same problem in different framings. A model that answers "yes" to "Is A > B?" but also "yes" to "Is B > A?" has a consistency failure. Standard tests include negation consistency, commutativity checks, and contrapositive reasoning.

Reasoning evaluation pitfall: Many benchmarks only check the final answer. A model can score 80% on GSM8K by pattern-matching common problem structures without genuine multi-step reasoning. Always evaluate the reasoning trace, not just the output.

Coding Evaluation

Coding evaluation benefits from automatic verification — generated code either passes tests or it doesn't. But pass rates alone miss important quality dimensions like readability, efficiency, and robustness.

Standard Benchmarks

HumanEval

164 problems — function-level Python code generation from docstrings. Measures pass@k: probability that at least 1 of k samples passes all unit tests.

Limitations: Small dataset, mostly simple algorithms, docstrings leak solution structure. Saturating — top models score 90%+.

MBPP (Mostly Basic Python)

974 problems — broader coverage of programming tasks, from string manipulation to data structures. Also uses pass@k.

Limitations: Still function-level. No multi-file projects, no dependency management, no real-world constraints.

SWE-bench

2,294 real GitHub issues — model must read issue descriptions, navigate repos, and produce working patches. Tests end-to-end software engineering.

Strengths: Real-world complexity, multi-file edits, test verification against the project's own test suite.

Custom Coding Evals

Build domain-specific coding benchmarks from your team's actual tasks: API integration, data pipeline construction, configuration generation.

Key metric: Task completion rate with automated test verification + human review for code quality.

Beyond Pass Rates

A comprehensive coding evaluation scores multiple dimensions:

class CodingEvaluator: def evaluate(self, generated_code, test_suite, reference_solution): results = {} # Functional correctness results["pass_rate"] = self.run_tests(generated_code, test_suite) # Code quality metrics results["cyclomatic_complexity"] = compute_complexity(generated_code) results["style_score"] = run_linter(generated_code) # pylint/flake8 # Efficiency comparison gen_time = self.benchmark_runtime(generated_code) ref_time = self.benchmark_runtime(reference_solution) results["efficiency_ratio"] = ref_time / gen_time # Robustness: edge cases and error handling edge_tests = self.generate_edge_cases(test_suite) results["edge_case_pass_rate"] = self.run_tests(generated_code, edge_tests) # Security: check for common vulnerabilities results["security_issues"] = run_bandit(generated_code) return results

Domain-Specific Evaluation

Domain evaluation tests whether the model has absorbed specialized knowledge and can apply it correctly. Off-the-shelf benchmarks rarely cover your specific domain, so you'll typically need to build custom evaluation sets with expert reviewers.

Medical Domain

Medical evaluation requires domain experts to verify clinical accuracy. Standard benchmarks include MedQA (USMLE-style), PubMedQA (research literature), and MedMCQA (Indian medical exams). Beyond multiple-choice, evaluate:

  • Differential diagnosis quality: Does the model consider relevant alternative diagnoses?
  • Contraindication awareness: Does it flag dangerous drug interactions?
  • Evidence grounding: Can it cite clinical guidelines, not just generate plausible text?
  • Uncertainty calibration: Does it express appropriate uncertainty on ambiguous cases?

Legal Domain

Legal evaluation tests statutory interpretation, case law application, and jurisdiction-specific rules. Key benchmarks include LegalBench (diverse legal reasoning tasks) and custom sets built from actual legal workflows:

  • Contract analysis: Identifying risks, missing clauses, non-standard terms
  • Regulatory compliance: Mapping requirements to specific regulations
  • Precedent reasoning: Correctly applying prior rulings to new fact patterns

Finance Domain

Financial evaluation combines numerical reasoning with domain knowledge. Test areas include financial statement analysis, risk assessment, regulatory interpretation (SEC/GAAP), and market analysis. Critical to check for:

  • Numerical precision: Financial calculations must be exact, not approximate
  • Temporal awareness: Regulations and market conditions change; is the model applying current rules?
  • Disclaimer generation: Does it properly qualify financial advice?
Expert evaluation protocol: For domain evals, recruit 3+ domain experts who independently score model outputs on a shared rubric. Compute inter-annotator agreement (Cohen's κ) to validate rubric clarity. If κ < 0.6, the rubric needs refinement before scores are meaningful.

Building Custom Domain Benchmarks

def build_domain_benchmark(domain, expert_panel, num_questions=200): # Step 1: Collect real-world tasks from domain experts raw_tasks = collect_expert_tasks(expert_panel, domain) # Step 2: Stratify by difficulty and subtopic stratified = stratify_tasks(raw_tasks, difficulty_levels=["basic", "intermediate", "expert"], subtopics=get_domain_taxonomy(domain) ) # Step 3: Create gold-standard answers with citations for task in stratified: task.gold_answer = expert_consensus_answer(expert_panel, task) task.rubric = create_scoring_rubric(task, levels=5) task.citations = gather_authoritative_sources(task) # Step 4: Validate with held-out experts validation_scores = validate_with_held_out_experts(stratified) assert validation_scores.cohens_kappa > 0.7, "Rubric needs refinement" return DomainBenchmark( tasks=stratified[:num_questions], rubric_version="1.0", domain=domain )

Capability Profiles & Radar Charts

A capability profile collapses per-axis scores into a visual fingerprint. Radar charts (spider diagrams) are the standard visualization — each spoke represents a capability axis, and the area covered indicates overall strength.

In this radar chart, Model A (blue) excels at reasoning and domain knowledge but has weaker coding. Model B (green) shows strong coding and factuality but weaker safety alignment. For a coding-assistant use case, Model B wins; for a medical advice tool, Model A is safer.

Generating Profiles Programmatically

class CapabilityProfile: AXES = ["reasoning", "coding", "domain", "safety", "instruction", "factuality"] def __init__(self, model_name): self.model_name = model_name self.scores = {} def evaluate_all(self, eval_suites): for axis in self.AXES: suite = eval_suites[axis] results = suite.run(self.model_name) self.scores[axis] = { "mean": results.mean_score(), "std": results.std_dev(), "n": results.num_examples(), "ci_95": results.confidence_interval(0.95) } def compare(self, other): # Per-axis comparison with statistical significance comparison = {} for axis in self.AXES: delta = self.scores[axis]["mean"] - other.scores[axis]["mean"] significant = welch_t_test(self.scores[axis], other.scores[axis]) comparison[axis] = {"delta": delta, "significant": significant} return comparison
Don't compare without confidence intervals. A model scoring 78.2% vs 77.8% on reasoning is not meaningfully different unless you've computed confidence intervals and performed a significance test. Always report ± CI alongside point estimates.

Tracking Improvement Over Time

Model capabilities change with fine-tuning, RLHF iterations, data updates, and prompt engineering. A tracking system records every evaluation run and produces time-series views of each capability axis.

The chart shows capability improvements across model versions. Note that coding (green) saw the biggest jump from v1.1 → v2.0 — likely the release that added code-specific training data. Domain knowledge (orange) improves slowly, reflecting the difficulty of acquiring specialized data.

Regression Detection

Improving one capability often degrades another. A tracking system must flag capability regressions — cases where a model update improves reasoning by 5% but silently drops coding by 3%. Set per-axis regression thresholds:

def check_regressions(current_profile, previous_profile, thresholds): # Flag any axis that dropped beyond the allowed threshold regressions = [] for axis in CapabilityProfile.AXES: delta = current_profile.scores[axis]["mean"] - \ previous_profile.scores[axis]["mean"] if delta < -thresholds[axis]: regressions.append({ "axis": axis, "delta": delta, "threshold": thresholds[axis], "severity": "critical" if delta < -2 * thresholds[axis] else "warning" }) return regressions # Example: allow 1% drop in safety, 2% in others thresholds = {"reasoning": 0.02, "coding": 0.02, "domain": 0.02, "safety": 0.01, "instruction": 0.02, "factuality": 0.02}

Building an Evaluation Harness

An evaluation harness orchestrates benchmark execution, result collection, scoring, and reporting. A well-designed harness is modular — you plug in new benchmarks without changing the runner.

class EvalHarness: def __init__(self, model, benchmarks, output_dir): self.model = model self.benchmarks = benchmarks # dict: axis → BenchmarkSuite self.output_dir = output_dir def run_full_evaluation(self): profile = CapabilityProfile(self.model.name) for axis, suite in self.benchmarks.items(): print(f"Evaluating {axis}...") results = suite.run(self.model) profile.scores[axis] = results.aggregate() # Save per-example results for analysis results.save_detailed( f"{self.output_dir}/{axis}_detailed.jsonl" ) # Generate report report = generate_report(profile) report.save_html(f"{self.output_dir}/report.html") report.save_json(f"{self.output_dir}/report.json") # Check for regressions against previous run prev = load_previous_profile(self.output_dir) if prev: regressions = check_regressions(profile, prev, thresholds) if regressions: alert_team(regressions) return profile
Automate in CI/CD. Run your evaluation harness on every model checkpoint and fine-tuning iteration. Store results in a structured format (JSONL) so you can query historical trends. Treat capability regressions like failing tests — they block deployment.

Capability Gap Analysis & Roadmaps

Once you have capability profiles, the next step is identifying gaps between your model's current capabilities and your deployment requirements, then building a targeted improvement roadmap.

Gap Identification

Define minimum capability thresholds for each axis based on your use case. Any axis below its threshold is a gap that must be closed before deployment:

🔴 Critical Gaps

Capability is more than 15% below the required threshold. Deployment is blocked. Requires targeted data collection, fine-tuning, or model swap.

Example: Medical model scores 52% on drug interaction accuracy vs. 80% required threshold.

🟠 Moderate Gaps

Capability is 5–15% below threshold. Deployment possible with guardrails (human review, restricted scope). Improvement planned for next iteration.

Example: Coding model scores 72% on SWE-bench vs. 80% target.

🟢 On Target

Capability meets or exceeds the threshold. Monitor for regressions but no immediate action needed.

Example: Instruction following scores 91% vs. 85% required.

🔵 Over-Invested

Capability far exceeds the threshold. Consider reallocating training resources to gap areas rather than further improving this axis.

Example: Factual recall at 96% vs. 80% threshold — shift focus to reasoning.

Building an Improvement Roadmap

For each identified gap, define a concrete improvement plan:

  1. Root cause analysis: Examine failure cases on the gap axis. Are errors concentrated in specific subtopics? Is it a data gap or a model architecture limitation?
  2. Data intervention: Collect targeted training data for the weak subtopics. Use active learning to prioritize examples the model currently gets wrong.
  3. Training strategy: Fine-tune with emphasis on the gap axis, but run full capability profiles after each iteration to catch regressions on other axes.
  4. Re-evaluation cadence: Set a schedule — weekly for active development, monthly for production models — and automate the harness execution.
Capability-accuracy tradeoff: Aggressively fine-tuning on one capability (e.g., coding) with large amounts of domain-specific data can degrade general capabilities. Always evaluate the full profile, not just the axis you're trying to improve. Use data mixing ratios to balance improvement against regression.

Key takeaway: Single-number evaluation is obsolete for LLMs. Build capability profiles across reasoning, coding, and domain axes. Track these profiles over time, flag regressions automatically, and use gap analysis to drive targeted improvements. The goal is not a higher number on one leaderboard — it's a model whose capability profile matches your deployment needs.
