Factuality: Grounding, Verification, Attribution

MLOps Series LLM Evaluation & Safety

Large language models generate fluent, convincing text—but fluency is not truth. LLMs hallucinate facts, conflate entities, fabricate citations, and present speculation as certainty. In high-stakes domains like medicine, law, and finance, a single wrong fact can erode user trust or cause real harm. Factuality evaluation is the discipline of measuring, verifying, and improving the truthfulness of model outputs, and it is rapidly becoming a cornerstone of responsible LLM deployment.

This post covers the full factuality stack: grounding model outputs in trusted knowledge sources, decomposing claims for fine-grained verification, attributing answers to evidence, scoring confidence, and assembling these components into production-grade fact-checking systems.

The Factuality Challenge

Hallucination in LLMs is not a bug to be patched—it is an inherent property of autoregressive generation. The model predicts the next token based on statistical patterns, not grounded reasoning over a world model. This means every output is a probabilistic guess, and some guesses will be confidently wrong.

The factuality problem manifests in several distinct failure modes:

  • Entity conflation — merging attributes of different entities ("Albert Einstein won the Nobel Prize in Physics for general relativity" mixes the correct prize with the wrong reason).
  • Temporal drift — stating outdated facts as current ("The UK is part of the EU").
  • Fabricated citations — inventing plausible-looking references that don't exist.
  • Numerical hallucination — generating precise but incorrect statistics or dates.
  • Hedging failure — presenting uncertain claims without qualifiers.
Warning: Factuality is not the same as truthfulness. A model can be factual (stating verifiable claims correctly) while being misleading through selective omission. Full trustworthiness requires factuality plus completeness, relevance, and calibration.

Measuring factuality requires decomposing model outputs into atomic, verifiable claims and checking each one against trusted evidence. This is the core idea behind FActScore (Factual Precision in Atomicity Score), which we will examine in the verification section.

Grounding Techniques

Grounding anchors model generation to external, trusted knowledge sources. Instead of relying solely on parametric memory (weights learned during pretraining), grounded systems retrieve evidence at inference time and condition generation on that evidence. The two dominant paradigms are retrieval-augmented generation (RAG) and knowledge-graph grounding.

Retrieval-Augmented Generation (RAG)

RAG injects relevant documents into the model's context window before generation. The retriever fetches top-k passages from a corpus (Wikipedia, internal docs, domain knowledge bases), and the generator produces an answer conditioned on both the query and retrieved evidence.

Dense Retrieval

Encode queries and passages into shared embedding space (e.g., Contriever, E5, BGE). Fast approximate nearest-neighbor search via FAISS or ScaNN. Best for semantic similarity matching.

Sparse Retrieval

BM25, SPLADE, or TF-IDF based. Excels at exact term matching and rare-entity lookup. Often combined with dense retrieval in hybrid pipelines for best recall.

Knowledge Graph Grounding

For structured domains, knowledge graphs (Wikidata, domain-specific ontologies) provide explicit entity-relation-entity triples that can be used as hard constraints on generation. KG grounding is especially powerful for numerical facts, entity relationships, and temporal data where unstructured text retrieval may return conflicting passages. 

# RAG grounding with evidence scoring

from sentence_transformers import SentenceTransformer
import numpy as np

class GroundedGenerator:
    def __init__(self, retriever_model, llm, corpus_embeddings, corpus_texts):
        self.encoder = SentenceTransformer(retriever_model)
        self.llm = llm
        self.corpus_emb = corpus_embeddings      # precomputed (N, d) matrix
        self.corpus_txt = corpus_texts

    def retrieve(self, query, top_k=5):
        q_emb = self.encoder.encode([query])
        scores = q_emb @ self.corpus_emb.T       # cosine similarity
        top_idx = np.argsort(scores[0])[-top_k:][::-1]
        return [(self.corpus_txt[i], float(scores[0][i])) for i in top_idx]

    def generate(self, query):
        evidence = self.retrieve(query)
        context = "\n".join([f"[{s:.2f}] {t}" for t, s in evidence])
        prompt = f"""Answer based ONLY on the evidence below.
If the evidence is insufficient, say "I don't have enough information."

Evidence:
{context}

Question: {query}
Answer:"""
        return self.llm.generate(prompt), evidence
Key insight: Grounding shifts the factuality burden from parametric memory to the retrieval pipeline. This makes factuality measurable and auditable—if the answer is wrong, you can trace it to either retrieval failure (wrong documents) or generation failure (ignoring the documents).

Verification Pipelines

Verification is the process of checking whether claims in an LLM's output are supported by evidence. Modern verification pipelines follow a three-stage architecture: claim decomposition, evidence retrieval, and entailment classification. This is the machinery behind FActScore and related evaluation frameworks.

Stage 1 — Claim Decomposition

Complex sentences are broken into atomic, independently verifiable claims. For example, "Marie Curie was born in Warsaw in 1867 and won two Nobel Prizes" decomposes into three atomic facts: (1) Marie Curie was born in Warsaw, (2) She was born in 1867, (3) She won two Nobel Prizes. 

# Claim decomposition using an LLM

import json

DECOMPOSE_PROMPT = """Break the following text into a list of atomic, independently
verifiable factual claims. Each claim should contain exactly one fact.
Return a JSON list of strings.

Text: {text}
Claims:"""

def decompose_claims(text, llm):
    """Split a passage into atomic factual claims."""
    response = llm.generate(DECOMPOSE_PROMPT.format(text=text))
    claims = json.loads(response)
    return [c.strip() for c in claims if len(c.strip()) > 10]

# Example
text = "Albert Einstein published special relativity in 1905 while working at a patent office in Bern."
claims = decompose_claims(text, llm)
# Output: [
#   "Albert Einstein published special relativity in 1905.",
#   "Albert Einstein worked at a patent office.",
#   "The patent office was in Bern."
# ]

Stage 2 — Evidence Retrieval

For each atomic claim, the system retrieves relevant passages from a trusted corpus (Wikipedia, knowledge base, domain documentation). The retrieval quality directly bounds the verification accuracy—a claim cannot be verified if the relevant evidence is not retrieved.

Stage 3 — Entailment Classification (NLI)

Each (claim, evidence) pair is classified using a natural language inference model into one of three categories: supported, refuted, or not enough info. The NLI classifier is typically a fine-tuned model (DeBERTa, RoBERTa) trained on FEVER, MNLI, or domain-specific verification datasets. 

# Full NLI-based claim verification pipeline

from transformers import pipeline
from dataclasses import dataclass
from typing import List

@dataclass
class VerificationResult:
    claim: str
    label: str            # "supported" | "refuted" | "nei"
    confidence: float
    evidence: str

class FactVerifier:
    def __init__(self, nli_model="cross-encoder/nli-deberta-v3-large"):
        self.nli = pipeline("text-classification", model=nli_model)
        self.label_map = {"ENTAILMENT": "supported",
                          "CONTRADICTION": "refuted",
                          "NEUTRAL": "nei"}

    def verify_claim(self, claim: str, evidence: str) -> VerificationResult:
        """Classify whether evidence supports, refutes, or is neutral to claim."""
        result = self.nli(f"{evidence} [SEP] {claim}")[0]
        return VerificationResult(
            claim=claim,
            label=self.label_map.get(result["label"], "nei"),
            confidence=result["score"],
            evidence=evidence
        )

    def verify_all(self, claims: List[str], retriever) -> List[VerificationResult]:
        """Full pipeline: retrieve evidence, then verify each claim."""
        results = []
        for claim in claims:
            passages = retriever.retrieve(claim, top_k=3)
            best_evidence = passages[0][0]         # highest-scored passage text
            results.append(self.verify_claim(claim, best_evidence))
        return results

# ---------- Computing FActScore ----------
def factscore(results: List[VerificationResult]) -> float:
    """FActScore = fraction of atomic claims that are supported."""
    supported = sum(1 for r in results if r.label == "supported")
    return supported / max(len(results), 1)
FActScore explained: FActScore (Min et al., 2023) measures the percentage of atomic facts in a generated biography that are supported by a reference source (typically Wikipedia). A model with FActScore 0.85 means 85% of its atomic claims are verifiably correct. This metric has become the de facto standard for fine-grained factuality evaluation.

Source Attribution

Verification tells you whether a claim is correct; attribution tells you where the supporting evidence comes from. High-quality attribution lets users audit model outputs, builds trust, and is increasingly a regulatory requirement (e.g., the EU AI Act's transparency mandates).

Inline Citations

The most user-facing form of attribution is inline citation—inserting reference markers (e.g., [1], [2]) directly into generated text. This requires the model to track which evidence passage supports which part of its answer and to cite accurately.

Post-hoc Attribution

Generate the answer first, then match each sentence to the most relevant retrieved passage via NLI or semantic similarity. Simpler to implement, but citations can feel disconnected from the generation process.

Inline Generation Attribution

Train or prompt the model to emit citations during generation (e.g., "The Eiffel Tower is 330m tall [doc3]."). More natural but requires careful prompt engineering or fine-tuning to avoid hallucinated citations.

Provenance Tracking

Beyond user-facing citations, production systems need end-to-end provenance: which corpus was queried, which retriever version was used, which passages were returned, and how the generator consumed them. This is critical for debugging factual errors and for continuous improvement of the retrieval pipeline. 

# Attribution quality scoring

from dataclasses import dataclass, field
from typing import List, Dict

@dataclass
class Attribution:
    sentence: str
    cited_doc_id: str
    evidence_text: str
    nli_label: str          # entailment, contradiction, neutral
    nli_score: float

def attribution_quality(attributions: List[Attribution]) -> Dict[str, float]:
    """Compute attribution quality metrics (AIS-style)."""
    total = len(attributions)
    if total == 0:
        return {"precision": 0.0, "citation_rate": 0.0}

    # Citation precision: fraction of citations that actually entail the claim
    correct_cites = sum(1 for a in attributions if a.nli_label == "entailment")

    # Citation rate: fraction of sentences that have any citation
    cited = sum(1 for a in attributions if a.cited_doc_id != "")

    return {
        "precision": correct_cites / max(cited, 1),
        "citation_rate": cited / total,
        "avg_nli_score": sum(a.nli_score for a in attributions) / total
    }
AIS metric: Attributable to Identified Sources (AIS) is a binary human evaluation metric: for each statement in the output, a human judge decides whether it is fully supported by the cited source. Automated AIS approximations use NLI models, reaching ~85% agreement with human annotators on well-structured outputs.

Confidence Scoring

Not all claims should be treated equally. A well-calibrated factuality system assigns confidence scores to each claim, enabling downstream consumers to filter, flag, or escalate low-confidence outputs. Confidence scoring bridges the gap between binary verification (supported/not) and the nuanced reality of partial evidence.

Sources of Confidence Signal

  • NLI entailment score — the probability assigned by the NLI model to the "entailment" class. Higher scores indicate stronger evidential support.
  • Retrieval score — cosine similarity between the claim and the best-matching evidence passage. Low retrieval scores suggest the claim is out of scope of the knowledge base.
  • Token-level uncertainty — the model's own generation probabilities. Claims generated with high average log-probability are more likely to be parametrically grounded.
  • Consensus across sources — if multiple independent passages support a claim, confidence increases. Contradictory evidence signals lower confidence.
# Composite confidence scoring

import numpy as np

def composite_confidence(
    nli_score: float,
    retrieval_score: float,
    gen_log_prob: float,
    n_supporting: int,
    n_contradicting: int,
    weights: dict = None
) -> float:
    """Weighted composite confidence for a single claim."""
    w = weights or {"nli": 0.4, "retrieval": 0.2, "gen": 0.15, "consensus": 0.25}

    # Normalize generation log-prob to [0, 1]
    gen_conf = 1.0 / (1.0 + np.exp(-gen_log_prob - 2.0))

    # Consensus: penalize contradictions heavily
    total_sources = n_supporting + n_contradicting
    consensus = n_supporting / max(total_sources, 1) if total_sources > 0 else 0.5

    score = (w["nli"]       * nli_score +
             w["retrieval"]  * retrieval_score +
             w["gen"]        * gen_conf +
             w["consensus"]  * consensus)

    return round(float(np.clip(score, 0.0, 1.0)), 4)

# Usage example
conf = composite_confidence(
    nli_score=0.92, retrieval_score=0.85,
    gen_log_prob=-1.2, n_supporting=3, n_contradicting=0
)
print(f"Confidence: {conf}")  # ~0.87
Calibration matters: Raw confidence scores are only useful if they are well-calibrated—a score of 0.9 should mean the claim is correct ~90% of the time. In practice, NLI models and LLMs are poorly calibrated out-of-the-box. Apply Platt scaling or isotonic regression on a held-out verification dataset before using confidence scores for thresholding.

Production Factuality Systems

Building a factuality system for production requires integrating all the components above into a coherent, scalable pipeline. The architecture must handle real-time verification for interactive applications and batch evaluation for model development and monitoring.

Real-Time vs Batch Verification

Real-Time (Inline)

Runs verification during generation, adding 200–500ms latency. Best for chat interfaces and customer-facing apps. Use lightweight NLI models (MiniLM-based) and cache frequent claims for speed.

Batch (Offline)

Runs verification post-hoc on logged outputs. No latency impact. Ideal for model evaluation, regression testing, and compliance audits. Use full-size NLI models for maximum accuracy.

Automated Fact-Checking at Scale

Production fact-checking pipelines process thousands of outputs per hour. Key engineering considerations include:

  • Claim caching — Many outputs contain repeated or near-duplicate claims. A semantic deduplication layer (embedding similarity > 0.95) avoids redundant verification.
  • Tiered verification — High-confidence claims (NLI score > 0.95) pass without further review. Medium-confidence claims (0.7–0.95) get additional evidence retrieval. Low-confidence claims (< 0.7) are flagged for human review.
  • Knowledge base freshness — Schedule regular corpus updates and reindex embeddings. Stale knowledge bases are the #1 cause of false "refuted" verdicts on correct, timely claims.
  • Adversarial robustness — Users may craft prompts to bypass verification. Red-team your pipeline with adversarial inputs that exploit retrieval gaps or NLI blind spots.

Building Trustworthy Outputs

The ultimate goal of a factuality system is not just to catch errors, but to produce outputs that users can trust. This means combining grounding, verification, attribution, and confidence into a unified user experience:

  1. Generate with RAG grounding to maximize baseline factuality.
  2. Verify each claim against retrieved evidence using NLI.
  3. Attribute every claim to its source with inline citations.
  4. Score confidence and surface uncertainty to the user.
  5. Flag low-confidence or unsupported claims visually (color-coding, warning icons).
  6. Log everything for audit, debugging, and continuous improvement.
The bottom line: Factuality is not a single metric—it is a system property that emerges from grounding, verification, attribution, and calibration working together. The most impactful investment is usually in retrieval quality: a system that retrieves the right evidence and verifies against it will outperform a system with a better LLM but weaker grounding. Start with FActScore as your north-star metric, build a claim-level verification pipeline, and iterate on retrieval before tuning the generator.
