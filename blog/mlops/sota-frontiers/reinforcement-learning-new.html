Beyond RLHF: RLAIF, KTO, ORPO, GRPO

MLOps Series SOTA & Frontiers

RLHF Recap & Limitations

Reinforcement Learning from Human Feedback (RLHF) remains the foundational alignment technique that powered ChatGPT and Claude. The classic pipeline has three stages: (1) supervised fine-tuning on high-quality demonstrations, (2) training a reward model on human preference pairs, and (3) optimising the policy with PPO against that reward model while constraining divergence from the SFT checkpoint with a KL penalty.

Despite its success, RLHF carries significant operational and theoretical costs that motivated the wave of alternatives we cover in this post:

  • Reward model bottleneck — the RM requires tens of thousands of human comparisons, is expensive to collect, and introduces a proxy objective gap: the model optimises the RM score, not true human preference.
  • PPO instability — PPO requires four models in memory simultaneously (policy, reference, reward model, value head), careful hyperparameter tuning of clip ratios, GAE λ, and KL coefficients, and is notoriously brittle at scale.
  • Reward hacking — policies learn to exploit reward model idiosyncrasies (verbosity bias, sycophancy) rather than genuinely improving quality.
  • Human annotation cost — scaling to new domains, languages, or modalities requires fresh annotation rounds costing $50K–$500K+ per domain.
Key Insight: Every method in this post exists to address one or more of the above limitations — either by removing the reward model, replacing human annotators, simplifying the optimisation, or reducing memory overhead. 
# Classic RLHF pipeline — 4 models in GPU memory
def rlhf_training_step(prompt, policy, ref_policy, reward_model, value_head):
    response = policy.generate(prompt)
    reward   = reward_model.score(prompt, response)
    kl_pen   = kl_divergence(policy, ref_policy, prompt, response)
    advantage = gae(reward - β * kl_pen, value_head)

    # PPO clipped objective
    ratio = policy(response|prompt) / ref_policy(response|prompt)
    loss  = -min(ratio * advantage,
                clip(ratio, 1-ε, 1+ε) * advantage)
    return loss  # + value loss + entropy bonus

Alignment Method Taxonomy

The post-RLHF alignment landscape can be organised by two axes: whether a separate reward model is trained and whether human annotations are required. The taxonomy below maps every method covered in this post.

Reading the tree: Methods on the left require a learned reward model; methods on the right collapse the RM into the policy loss itself. Within the RM-free branch, some still need a reference model (DPO, KTO) while others eliminate even that (ORPO, SimPO).

DPO: Bypassing the Reward Model

Direct Preference Optimization (Rafailov et al., 2023) was the first major simplification of RLHF. The core insight: the optimal policy under a KL-constrained reward-maximisation objective can be expressed in closed form, so you can directly optimise a classification loss on preference pairs without ever training a reward model or running PPO.

The DPO loss operates on preference pairs (x, yw, yl) — a prompt, a chosen (winning) response, and a rejected (losing) response:

# DPO — only 2 models in memory (policy + frozen reference)
def dpo_loss(policy, ref_policy, prompt, y_win, y_lose, β=0.1):
    # Log-probability ratios under policy vs reference
    log_ratio_w = logprob(policy, y_win|prompt) - logprob(ref_policy, y_win|prompt)
    log_ratio_l = logprob(policy, y_lose|prompt) - logprob(ref_policy, y_lose|prompt)

    # Bradley-Terry preference model in closed form
    loss = -log_sigmoid(β * (log_ratio_w - log_ratio_l))
    return loss.mean()

DPO Advantages

  • No reward model training — saves 30-50% of RLHF compute
  • No PPO — eliminates clip ratios, GAE, value heads
  • Stable gradients — simple binary cross-entropy
  • 2 models in memory vs 4 for RLHF

DPO Limitations

  • Requires pairwise preference data — expensive to collect
  • Still needs a frozen reference model (memory cost)
  • Suffers from "chosen reward margin collapse" in long training
  • Sensitive to data quality — noisy pairs degrade performance sharply

RLAIF: AI Feedback at Scale

Reinforcement Learning from AI Feedback (Lee et al., 2023; Bai et al., 2022) keeps the RLHF pipeline intact but replaces human annotators with a strong LLM (e.g., GPT-4, Claude) that judges which response is better. The key motivation: human annotation costs $2–$5 per comparison, scales linearly, and introduces inter-annotator disagreement. An AI judge costs ~$0.01 per comparison and is perfectly consistent.

The RLAIF pipeline works as follows:

  1. Generate candidate pairs — sample two responses from the policy for each prompt.
  2. AI judge scores — prompt a strong model with a rubric (helpfulness, harmlessness, honesty) and collect preference labels.
  3. Train reward model — standard Bradley-Terry RM on AI-labeled data.
  4. PPO optimisation — identical to RLHF from this point.
# RLAIF — replace human annotator with LLM judge
def rlaif_label(judge_model, prompt, response_a, response_b):
    rubric = format_rubric(
        criteria=["helpfulness", "harmlessness", "honesty"],
        scale="pairwise"
    )
    judge_prompt = f"""Given this prompt and two responses,
which response is better according to: {rubric}

Prompt: {prompt}
Response A: {response_a}
Response B: {response_b}

Output ONLY 'A' or 'B'."""

    verdict = judge_model.generate(judge_prompt)
    return ("A", "B") if verdict == "A" else ("B", "A")
Position Bias Warning: LLM judges exhibit systematic position bias — they tend to prefer whichever response appears first. Mitigate by running each comparison twice with swapped order and taking the majority vote. This doubles cost but halves bias.

Google's Constitutional AI (Bai et al., 2022) demonstrated that RLAIF can match or exceed human-feedback RLHF on safety benchmarks. Subsequent work from Google DeepMind showed RLAIF reaching 96-98% agreement with human labels on summarisation and helpfulness tasks. The critical requirement: the judge model must be significantly stronger than the policy being trained, otherwise you amplify the policy's own biases.

KTO: Binary Signal Alignment

Kahneman-Tversky Optimization (Ethayarajh et al., 2024) is built on a striking observation: you don't actually need pairwise preferences. Most production feedback is binary — a user clicks thumbs-up or thumbs-down on a single response. Forcing this into pairwise format (matching thumbs-up responses with thumbs-down responses for the same prompt) introduces arbitrary pairings that don't reflect real preference.

KTO's theoretical foundation comes from prospect theory (Kahneman & Tversky, 1979): humans weight losses more heavily than equivalent gains. KTO incorporates this asymmetry directly into the loss function. A "bad" response is penalised more than an equally "good" response is rewarded.

# KTO — binary (desirable/undesirable) signal, no pairing needed
def kto_loss(policy, ref_policy, prompt, response, is_desirable, β=0.1):
    log_ratio = logprob(policy, response|prompt) - logprob(ref_policy, response|prompt)
    kl_term   = kl_estimate(policy, ref_policy)  # running average

    if is_desirable:
        # Value function for gains (σ is sigmoid)
        loss = 1 - σ(β * (log_ratio - kl_term))
    else:
        # Value function for losses — weighted more heavily (loss aversion)
        loss = 1 - σ(β * (kl_term - log_ratio))

    return loss

Why KTO Matters

  • Works with unpaired binary feedback — thumbs up/down
  • No need to match chosen/rejected for the same prompt
  • Matches or exceeds DPO on 1B–30B models (Ethayarajh et al., 2024)
  • Loss aversion baked in — naturally avoids harmful outputs

KTO Trade-offs

  • Still requires a reference model (memory overhead)
  • KL estimate uses a running average — can be noisy early in training
  • Needs balanced desirable/undesirable data (~60/40 to 50/50)
  • Less studied at 70B+ scale compared to DPO/RLHF
Production Insight: KTO is ideal when you have user-facing products collecting implicit feedback (upvotes, regeneration requests, copy-to-clipboard actions). You don't need the expensive annotation step of asking "which is better?"

ORPO: No Reference Model Needed

Odds Ratio Preference Optimization (Hong et al., 2024) takes simplification further by eliminating the reference model entirely. ORPO's key insight: the SFT loss already teaches the model what to generate; you just need an additional term that teaches it to prefer chosen over rejected responses, and the odds ratio provides exactly this signal.

ORPO merges SFT and alignment into a single training phase. The loss is a weighted sum of the negative log-likelihood (NLL) on the chosen response and a log-odds penalty on the rejected response:

# ORPO — single model, no reference needed, SFT + alignment in one pass
def orpo_loss(policy, prompt, y_win, y_lose, λ=1.0):
    # Standard SFT loss on chosen response
    nll_loss = -logprob(policy, y_win | prompt).mean()

    # Odds ratio: P(y)/(1-P(y))
    odds_w = exp(logprob(policy, y_win|prompt)) / (1 - exp(logprob(policy, y_win|prompt)))
    odds_l = exp(logprob(policy, y_lose|prompt)) / (1 - exp(logprob(policy, y_lose|prompt)))

    # Preference term: log-sigmoid of log odds ratio
    pref_loss = -log_sigmoid(log(odds_w / odds_l))

    return nll_loss + λ * pref_loss
Memory Savings: ORPO uses exactly one model in GPU memory — no reference copy, no reward model, no value head. For a 7B model, this means ~14 GB instead of ~28 GB (DPO) or ~56 GB (RLHF). This makes alignment feasible on consumer GPUs.

ORPO achieves competitive results with Llama-2 7B and Mistral 7B, matching DPO on AlpacaEval and MT-Bench while using half the memory. The merged SFT + alignment training also converges faster — typically 1 epoch vs 3 for separate SFT + DPO stages.

Caveat: Because ORPO couples SFT and alignment, you cannot independently tune either stage. If your SFT data and preference data have different quality levels, this coupling can hurt. DPO with separate SFT remains more flexible for production pipelines where data quality varies.

GRPO: Group Relative Policy Optimization

Group Relative Policy Optimization (Shao et al., 2024) is the alignment algorithm behind DeepSeek-R1, the open-source reasoning model that matched OpenAI's o1 on key benchmarks. GRPO's distinguishing insight: instead of learning a value function (expensive, unstable), estimate the baseline from group statistics of sampled outputs.

For each prompt, GRPO samples a group of G responses (typically G=8–64), scores each with a rule-based or model-based reward, then normalises rewards within the group to compute advantages. This eliminates the value head entirely while providing stable variance-reduced gradient estimates:

# GRPO — group sampling eliminates the value network
def grpo_step(policy, ref_policy, prompt, G=16, β=0.04, ε=0.2):
    # Sample G responses for the same prompt
    responses = [policy.generate(prompt) for _ in range(G)]
    rewards   = [reward_fn(prompt, r) for r in responses]

    # Group normalisation → advantage (no value model needed)
    μ, σ = mean(rewards), std(rewards)
    advantages = [(r - μ) / (σ + 1e-8) for r in rewards]

    # PPO-style clipped objective per response
    loss = 0
    for resp, adv in zip(responses, advantages):
        ratio = exp(logprob(policy, resp) - logprob(ref_policy, resp))
        clipped = clip(ratio, 1-ε, 1+ε) * adv
        loss += -min(ratio * adv, clipped)

    # KL regularisation against reference
    kl = β * kl_divergence(policy, ref_policy)
    return loss / G + kl

Why GRPO works for reasoning: DeepSeek-R1 uses rule-based rewards (correct final answer = +1, format compliance = +0.5) that are trivially scalable. Combined with group sampling, this gives clean training signal without any human annotations or learned reward models.

GRPO is particularly effective for tasks with verifiable outcomes — mathematics, coding, and formal logic — where reward functions can be exact and cheap. For open-ended creative tasks, GRPO requires either an LLM judge (back to RLAIF territory) or human-defined heuristic rewards.

SimPO & IPO

SimPO — Simple Preference Optimization

SimPO (Meng et al., 2024) builds on DPO with two modifications: (1) it uses length-normalised log-probabilities as the implicit reward — preventing the model from gaming scores by generating longer responses — and (2) it removes the reference model entirely by using the average log-probability as the reward signal.

# SimPO — reference-free, length-normalised DPO variant
def simpo_loss(policy, prompt, y_win, y_lose, β=2.0, γ=0.5):
    # Length-normalised log-prob as implicit reward
    r_w = logprob(policy, y_win|prompt) / len(y_win)
    r_l = logprob(policy, y_lose|prompt) / len(y_lose)

    # Margin γ ensures minimum reward gap
    loss = -log_sigmoid(β * (r_w - r_l) - γ)
    return loss.mean()
Length Normalisation Matters: Without it, DPO and similar methods develop a verbosity bias — longer responses accumulate higher total log-probability. SimPO's per-token normalisation directly addresses this, producing more concise aligned models.

IPO — Identity Preference Optimization

IPO (Azar et al., 2024) addresses a theoretical flaw in DPO: the Bradley-Terry preference model assumes preferences are deterministic — given the true rewards, the better response is always preferred. In practice, human preferences are noisy and stochastic. IPO replaces the log-sigmoid with a simpler squared loss that doesn't make deterministic assumptions:

# IPO — robust to noisy/stochastic preferences
def ipo_loss(policy, ref_policy, prompt, y_win, y_lose, τ=0.5):
    log_ratio_w = logprob(policy, y_win|prompt) - logprob(ref_policy, y_win|prompt)
    log_ratio_l = logprob(policy, y_lose|prompt) - logprob(ref_policy, y_lose|prompt)

    # Squared loss — no log-sigmoid, no Bradley-Terry assumption
    loss = (log_ratio_w - log_ratio_l - 1 / (2 * τ)) ** 2
    return loss.mean()

SimPO Highlights

  • No reference model — same memory as ORPO
  • Length normalisation eliminates verbosity bias
  • Margin term γ prevents reward collapse
  • State-of-the-art on AlpacaEval 2, Arena-Hard

IPO Highlights

  • Robust to noisy or contradictory preference labels
  • Theoretically grounded — regularised minimax objective
  • Drop-in replacement for DPO (same data format)
  • Performs best when annotation quality is mixed

When to Use Which Method

Choosing an alignment method depends on four practical factors: your data format, GPU budget, task type, and annotation quality. Use the decision framework below:

🧪 Research / Exploration

Recommendation: DPO or SimPO

DPO is the best-understood baseline with extensive ablation studies. SimPO if you're GPU-constrained and want reference-free training. Both are simple to implement and debug.

🏭 Production with User Feedback

Recommendation: KTO

When you have thumbs-up/down logs from real users, KTO directly consumes binary signals. No need to construct artificial preference pairs. Combine with RLAIF for the judge-labeled portion.

💻 Math / Code / Verifiable Tasks

Recommendation: GRPO

Rule-based rewards (correct answer, passing tests) make GRPO's group sampling extremely effective. This is how DeepSeek-R1 was trained. Scale G=16–64 for best results.

🚀 Consumer GPU / Resource-Limited

Recommendation: ORPO or SimPO

Single-model methods fit 7B parameters on a single 24 GB GPU. ORPO merges SFT + alignment. SimPO provides better control over verbosity. Both eliminate the reference model entirely.

📊 Noisy / Crowd-sourced Annotations

Recommendation: IPO

When your preference labels come from heterogeneous annotators (MTurk, crowd-sourcing), IPO's squared loss is robust to contradictory preferences. Drop-in replacement for DPO with the same data format.

🌍 Scaling to New Domains Cheaply

Recommendation: RLAIF + DPO

Use an LLM judge to generate preference pairs in the new domain, then train with DPO. Costs ~100× less than human annotation. Validate a sample with human eval to check judge quality.

Common Pitfall: Don't chase the newest method blindly. DPO remains the strongest default for most scenarios. Switch to alternatives only when you have a specific constraint (binary data → KTO, no reference GPU budget → ORPO/SimPO, verifiable tasks → GRPO). The marginal quality difference between methods is often smaller than the impact of data quality.

The alignment landscape is evolving rapidly — new methods appear monthly. The core trend is clear: simpler objectives, less infrastructure, cheaper data. RLHF's four-model pipeline is giving way to single-model closed-form losses that are easier to implement, debug, and scale. The limiting factor has shifted from algorithmic sophistication to data quality and curation.
