Vision-Language Architectures

Multimodal models fuse visual and textual understanding within a single system. The core challenge is bridging two radically different representation spaces: pixels (continuous, 2-D spatial grids) and tokens (discrete, sequential symbols). Three dominant architectural paradigms have emerged to solve this.

The pipeline above illustrates the modular approach used by most open-source vision-language models: a frozen or fine-tuned vision encoder produces patch embeddings, a projection layer aligns dimensionality, and the LLM backbone processes the interleaved visual + textual token sequence.

Three paradigms: (1) Modular fusion — separate vision encoder + projection + LLM (LLaVA, Qwen-VL). (2) Native multimodal — single model trained on all modalities from scratch (Gemini). (3) Cross-attention injection — visual features injected via cross-attention layers into the LLM (Flamingo, Idefics).

Vision Encoders & Projection Layers

The vision encoder is the first bottleneck in any multimodal system. It determines how much spatial detail, color fidelity, and semantic richness the LLM actually sees.

ViT (Vision Transformer)

  • Splits image into fixed 14×14 or 16×16 patches
  • Each patch → linear embedding + positional encoding
  • Standard self-attention across all patches
  • Used in CLIP ViT-L/14 (used by LLaVA, many open-source VLMs)
  • Output: sequence of 256 or 576 patch tokens

SigLIP

  • Sigmoid loss instead of softmax contrastive loss
  • Enables batch-size-independent pairwise training
  • Better calibrated zero-shot classification
  • Used in PaLI-X, Gemma-based VLMs
  • SigLIP-SO400M: 400M params, 384×384 native resolution

Projection Layer Variants

The projection layer bridges the vision encoder's embedding dimension (e.g., 1024 for ViT-L) to the LLM's hidden dimension (e.g., 4096 for LLaMA-7B). The design choice dramatically affects training cost and quality.

# Projection layer variants in multimodal models # 1. Simple linear projection (LLaVA v1) class LinearProjection(nn.Module): def __init__(self, visual_dim, llm_dim): super().__init__() self.proj = nn.Linear(visual_dim, llm_dim) def forward(self, visual_features): return self.proj(visual_features) # [B, N_patches, llm_dim] # 2. Two-layer MLP with GELU (LLaVA v1.5+) class MLPProjection(nn.Module): def __init__(self, visual_dim, llm_dim): super().__init__() self.fc1 = nn.Linear(visual_dim, llm_dim) self.act = nn.GELU() self.fc2 = nn.Linear(llm_dim, llm_dim) def forward(self, x): return self.fc2(self.act(self.fc1(x))) # 3. Q-Former / Perceiver resampler (BLIP-2, Flamingo) # Uses learnable query tokens + cross-attention # Compresses N_patches → N_queries (e.g., 576 → 32) class PerceiverResampler(nn.Module): def __init__(self, n_queries, visual_dim, llm_dim): super().__init__() self.queries = nn.Parameter(torch.randn(n_queries, llm_dim)) self.cross_attn = nn.MultiheadAttention(llm_dim, num_heads=16) self.proj = nn.Linear(visual_dim, llm_dim) def forward(self, visual_features): kv = self.proj(visual_features) # [B, N_patches, llm_dim] q = self.queries.unsqueeze(0).expand(kv.size(0), -1, -1) out, _ = self.cross_attn(q, kv, kv) return out # [B, N_queries, llm_dim] — much shorter!
Trade-off: Simple linear projections preserve all patch tokens (high fidelity but expensive at inference — 576 tokens per image). Perceiver resamplers compress to 32–64 tokens (cheaper inference but lossy on fine-grained spatial detail). Choose based on your task: document OCR demands high token counts; image classification can use few.

Image Tokenization Strategies

How an image is converted into tokens fundamentally shapes what the model can "see." Different strategies trade off between spatial resolution, token budget, and computational cost.

Fixed-Resolution Tokenization

The simplest approach: resize every input image to a fixed size (e.g., 224×224 or 336×336) regardless of original aspect ratio. This produces a fixed number of tokens per image. CLIP ViT-L/14 at 336px yields (336/14)² = 576 patch tokens.

Dynamic Resolution & Tiling

Modern VLMs like LLaVA-NeXT, InternVL2, and Qwen-VL support variable resolution by tiling the image into sub-images. A 1344×672 image might be split into a 4×2 grid of 336×336 tiles, each producing 576 tokens, for a total of ~4,608 visual tokens plus a low-res global thumbnail.

# Dynamic tiling strategy (simplified LLaVA-NeXT approach) def tile_image(image, tile_size=336, max_tiles=6): w, h = image.size # Find best grid that fits within max_tiles aspect = w / h best_grid = find_optimal_grid(aspect, max_tiles) n_cols, n_rows = best_grid # Resize image to fit the grid exactly target_w = n_cols * tile_size target_h = n_rows * tile_size resized = image.resize((target_w, target_h)) # Split into tiles tiles = [] for r in range(n_rows): for c in range(n_cols): crop = resized.crop(( c * tile_size, r * tile_size, (c+1) * tile_size, (r+1) * tile_size )) tiles.append(crop) # Add low-res global thumbnail thumbnail = image.resize((tile_size, tile_size)) tiles.insert(0, thumbnail) return tiles # Each tile → 576 patch tokens via ViT
Token counts explode with resolution. A single high-res image (e.g., 2048×2048) with 4×4 tiling produces 16 × 576 = 9,216 visual tokens, plus a thumbnail tile. This is equivalent to ~3,000 words of text in context-window cost. Budget carefully.

GPT-4V Architecture Insights

OpenAI's GPT-4V (GPT-4 with Vision, now part of GPT-4o and GPT-4 Turbo) was the first commercially dominant multimodal LLM. While the architecture is not fully public, several operational details are confirmed or strongly inferred from API behavior and research literature.

Known Operational Details

  • Resolution handling: Images are resized into tiles. Low-detail mode uses a single 512×512 tile (85 tokens). High-detail mode tiles into 512×512 patches covering the image, each costing 170 tokens, plus a base 85 tokens.
  • Token formula (high detail): tokens = 85 + 170 × ceil(width/512) × ceil(height/512)
  • Max image dimension: Images are first fit within 2048×2048, then the shortest side scaled to 768px before tiling.
  • Multiple images: Supported — each image independently tokenized. Total cost is additive.
# GPT-4V token cost estimator import math def gpt4v_image_tokens(width, height, detail="high"): if detail == "low": return 85 # Step 1: Fit within 2048×2048 max_dim = max(width, height) if max_dim > 2048: scale = 2048 / max_dim width = int(width * scale) height = int(height * scale) # Step 2: Scale shortest side to 768px min_dim = min(width, height) if min_dim > 768: scale = 768 / min_dim width = int(width * scale) height = int(height * scale) # Step 3: Count 512×512 tiles tiles_x = math.ceil(width / 512) tiles_y = math.ceil(height / 512) return 85 + 170 * tiles_x * tiles_y # Examples print(gpt4v_image_tokens(1024, 768)) # 85 + 170×2×2 = 765 print(gpt4v_image_tokens(4096, 4096)) # scaled → 768×768 → 85 + 170×2×2 = 765 print(gpt4v_image_tokens(512, 512)) # 85 + 170×1×1 = 255

GPT-4V Strengths

  • Exceptional OCR and document understanding
  • Strong spatial reasoning and counting
  • Robust multi-image comparison
  • Seamless function calling with vision
  • Broad world knowledge grounds visual understanding

GPT-4V Limitations

  • No video input (frame-by-frame only)
  • Token cost scales sharply with resolution
  • No fine-tuning available for vision tasks
  • Closed-source — no architectural customization
  • Safety filters can be overly conservative on benign images

Gemini: Native Multimodal

Google's Gemini family takes a fundamentally different approach: rather than bolting a vision encoder onto an existing LLM, Gemini is natively multimodal — trained from scratch on interleaved text, image, audio, and video data. This eliminates the projection-layer bottleneck and allows richer cross-modal reasoning.

Architecture Highlights

  • Unified tokenizer: Images, audio, and video are tokenized into the same sequence space as text. No separate "vision encoder" module in the classical sense.
  • Multi-resolution: Gemini 1.5 Pro processes images at native resolution up to very high token counts, with a 1M+ token context window accommodating dozens of high-res images or hours of video.
  • Video natively: Up to 1 hour of video can be processed directly — not as sampled frames but as a continuous input stream with temporal understanding.
  • Audio input: Gemini 1.5 Pro accepts raw audio alongside text and images. Enables multimodal reasoning across all three modalities simultaneously.
Native vs. modular: Native multimodal training means the model learns cross-modal representations during pretraining itself, avoiding the "information bottleneck" of a frozen projection layer. The trade-off is vastly larger training cost — Gemini's multimodal pretraining dataset includes trillions of image-text-audio-video tokens.

Gemini Token Pricing (Operational Impact)

# Gemini image token estimation # Gemini 1.5 Pro: images cost 258 tokens each (standard resolution) # Higher res images scale — but Google abstracts the tiling internally # Video: ~263 tokens per second of video # 1 minute of video ≈ 15,780 tokens # 1 hour of video ≈ 946,800 tokens (fits in 1M context window) gemini_costs = { "image_standard": 258, # tokens per image "video_per_second": 263, # tokens per second "audio_per_second": 32, # tokens per second of audio } def estimate_gemini_cost(n_images=0, video_seconds=0, audio_seconds=0): img_tokens = n_images * gemini_costs["image_standard"] vid_tokens = video_seconds * gemini_costs["video_per_second"] aud_tokens = audio_seconds * gemini_costs["audio_per_second"] return img_tokens + vid_tokens + aud_tokens # 10 images + 30s video + 60s audio total = estimate_gemini_cost(n_images=10, video_seconds=30, audio_seconds=60) print(f"Total tokens: {total}") # 2,580 + 7,890 + 1,920 = 12,390

Claude Vision Capabilities

Anthropic's Claude 3 family (Haiku, Sonnet, Opus) introduced vision as a core capability. Claude's vision system processes images inline within the conversation, supporting up to 20 images per request.

How Claude Processes Images

  • Token estimation: Images are internally converted to tokens based on dimensions. Roughly, tokens ≈ (width × height) / 750. A 1000×1000 image ≈ 1,334 tokens.
  • Max resolution: Images beyond 1568px on any side are downscaled. Optimal input range is 1000–1500px on the longest side.
  • Base64 or URL: Images can be sent as base64-encoded data or via URL reference in the API.
  • Multiple images: Claude can compare, contrast, and reason across multiple images in a single turn.

Claude Vision Strengths

  • Excellent chart and graph interpretation
  • Strong text extraction from documents
  • Detailed image description and analysis
  • Good spatial understanding and layout reasoning
  • Can process up to 20 images per request

Operational Considerations

  • No video or audio input (image-only)
  • No bounding box / object detection output
  • Image caching reduces cost on repeated images
  • Token cost proportional to image dimensions
  • Pre-resize images to optimal range for cost savings
# Sending images to Claude API import anthropic, base64 client = anthropic.Anthropic() # Method 1: Base64 encoded with open("chart.png", "rb") as f: img_data = base64.b64encode(f.read()).decode() response = client.messages.create( model="claude-sonnet-4-20250514", max_tokens=1024, messages=[{ "role": "user", "content": [ {"type": "image", "source": { "type": "base64", "media_type": "image/png", "data": img_data }}, {"type": "text", "text": "Describe this chart in detail."} ] }] ) # Method 2: URL reference (for supported sources) response = client.messages.create( model="claude-sonnet-4-20250514", max_tokens=1024, messages=[{ "role": "user", "content": [ {"type": "image", "source": { "type": "url", "url": "https://example.com/chart.png" }}, {"type": "text", "text": "What trends do you see?"} ] }] )

Serving Multimodal Models

Serving multimodal models introduces unique infrastructure challenges beyond standard LLM serving. The image preprocessing pipeline, variable token lengths, and GPU memory dynamics all differ substantially from text-only workloads.

Image Preprocessing Pipeline

Before any image reaches the vision encoder, it must be preprocessed. This CPU-bound step can become a bottleneck if not handled carefully.

# Production image preprocessing pipeline import torch from PIL import Image from torchvision import transforms class MultimodalPreprocessor: def __init__(self, target_size=336, max_tiles=6): self.target_size = target_size self.max_tiles = max_tiles self.transform = transforms.Compose([ transforms.ToTensor(), transforms.Normalize( mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225] ) ]) def validate_image(self, image_bytes): # Security: check format, dimensions, file size img = Image.open(io.BytesIO(image_bytes)) if img.format not in ("PNG", "JPEG", "WEBP", "GIF"): raise ValueError("Unsupported image format") if img.width * img.height > 25_000_000: raise ValueError("Image exceeds maximum pixel count") return img.convert("RGB") def estimate_tokens(self, width, height): # Estimate token cost before processing tiles = min( (width // self.target_size + 1) * (height // self.target_size + 1), self.max_tiles ) patches_per_tile = (self.target_size // 14) ** 2 return tiles * patches_per_tile + patches_per_tile # +thumbnail def preprocess(self, image_bytes, token_budget=4096): img = self.validate_image(image_bytes) est = self.estimate_tokens(img.width, img.height) # Downscale if exceeds token budget if est > token_budget: scale = (token_budget / est) ** 0.5 img = img.resize((int(img.width * scale), int(img.height * scale))) tiles = tile_image(img, self.target_size, self.max_tiles) return [self.transform(t) for t in tiles]

Token Budget Management

In production, you must enforce token budgets across the full multimodal request: text tokens + image tokens + output tokens ≤ context window. A single high-res image can consume 5,000+ tokens — as much as a lengthy document.

Common failure mode: A user sends 10 high-res images (50,000+ visual tokens) with a complex prompt (2,000 text tokens) to a model with a 128K context window. While it fits, the inference cost is 10× a text-only request and KV-cache memory usage can OOM your GPU. Always enforce per-request image count and resolution limits.

Serving Architecture Patterns

Coupled Serving

  • Vision encoder + LLM on same GPU
  • Simpler deployment, no network hops
  • Vision encoder idle during text generation
  • GPU memory shared — limits batch size
  • Suitable for: low-throughput, single-model setups

Disaggregated Serving

  • Vision encoder on separate GPU / CPU cluster
  • Visual embeddings sent to LLM serving cluster
  • Each component scales independently
  • Vision encoder can batch many images efficiently
  • Suitable for: high-throughput, multi-tenant platforms

Video & Audio Understanding

Video and audio extend multimodal models beyond static images but introduce massive token-count and latency challenges.

Video Processing Strategies

  • Frame sampling: Extract N frames uniformly (e.g., 1 FPS) and process each as an independent image. Simple but loses temporal continuity. Used by GPT-4V workarounds and many open-source VLMs.
  • Native video encoding: Gemini 1.5 Pro processes video as a continuous stream with temporal tokens. Preserves motion, action, and temporal relationships.
  • Video tokenizers: Models like VideoLLaMA and Video-ChatGPT use spatial-temporal pooling to compress video frames into fewer tokens while retaining motion information.
# Frame sampling for video → VLM pipeline import cv2 def sample_video_frames(video_path, fps=1, max_frames=32): cap = cv2.VideoCapture(video_path) original_fps = cap.get(cv2.CAP_PROP_FPS) frame_interval = int(original_fps / fps) frames = [] frame_idx = 0 while cap.isOpened() and len(frames) < max_frames: ret, frame = cap.read() if not ret: break if frame_idx % frame_interval == 0: # Convert BGR → RGB, resize for vision encoder rgb = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB) pil_img = Image.fromarray(rgb) frames.append(pil_img) frame_idx += 1 cap.release() return frames # List of PIL Images → feed to VLM

Audio Models

Audio multimodality is advancing rapidly. Gemini processes raw audio natively. OpenAI's GPT-4o handles real-time audio. Open-source alternatives include Whisper (ASR) + LLM pipeline, Qwen-Audio, and SALMONN. The operational pattern mirrors vision: an audio encoder (e.g., Whisper encoder, HuBERT) produces embeddings that are projected into the LLM's token space.

Audio token rates: Gemini charges ~32 tokens/second for audio. Whisper-based pipelines transcribe audio to text first (no audio tokens, but lossy on tone/music). For applications needing understanding of audio content (music, environmental sounds, speaker emotion), native audio models are essential.

Cost & Token Economics

Multimodal inference is substantially more expensive than text-only inference. Understanding token economics is critical for budgeting and architecture decisions.

API Cost Comparison (per image)

  • GPT-4o: 85 tokens (low-detail) to 765 tokens (high-detail, 1024×768)
  • Gemini 1.5 Pro: 258 tokens per standard image
  • Claude 3.5 Sonnet: ~1,334 tokens for 1000×1000 image
  • Open-source (LLaVA): 576 fixed tokens per 336×336 image

Cost Optimization Strategies

  • Use low-detail mode when high fidelity is unnecessary
  • Pre-resize images to optimal dimensions before API call
  • Implement image caching (Claude supports prompt caching)
  • Batch similar images together for throughput
  • Consider open-source VLMs for high-volume, lower-stakes tasks

Self-Hosted Multimodal Cost Model

# Cost estimation for self-hosted multimodal serving class MultimodalCostEstimator: def __init__(self, gpu_cost_per_hour, model_params_b, dtype_bytes=2): self.gpu_cost_hr = gpu_cost_per_hour self.model_memory_gb = model_params_b * dtype_bytes # A100 80GB ≈ $2/hr on cloud self.gpus_needed = max(1, int(self.model_memory_gb / 70) + 1) def cost_per_request(self, text_tokens, image_tokens, time_seconds): total_tokens = text_tokens + image_tokens cost = (time_seconds / 3600) * self.gpu_cost_hr * self.gpus_needed return { "gpu_cost_usd": round(cost, 6), "total_tokens": total_tokens, "cost_per_1k_tokens": round(cost / total_tokens * 1000, 6), "image_token_fraction": round(image_tokens / total_tokens, 2) } # Example: LLaVA-34B on A100 80GB estimator = MultimodalCostEstimator(gpu_cost_per_hour=2.0, model_params_b=34) result = estimator.cost_per_request( text_tokens=500, image_tokens=4608, # 8 tiles × 576 tokens time_seconds=3.5 ) print(result) # {'gpu_cost_usd': 0.001944, 'total_tokens': 5108, # 'cost_per_1k_tokens': 0.000381, 'image_token_fraction': 0.90}
The 90% rule: In many multimodal requests, image tokens constitute 80–95% of total token count. This means image preprocessing and resolution choices have a far greater impact on cost than prompt engineering. Optimizing image resolution is the single highest-leverage cost lever for multimodal workloads.

Decision Framework: Choosing a Multimodal Provider

Use API Providers When

  • Low to moderate volume (<10K images/day)
  • Need best-in-class quality (OCR, reasoning)
  • No GPU infrastructure available
  • Rapid prototyping and iteration
  • Multi-modality needed (video, audio, images)

Self-Host When

  • High volume (>100K images/day)
  • Data privacy or compliance requirements
  • Custom fine-tuning needed for domain
  • Predictable, fixed-cost infrastructure preferred
  • Latency-sensitive real-time applications
