Synthetic Data: Self-Play, Distillation

MLOps Series SOTA & Frontiers

Why Synthetic Data Matters

The performance of large language models is fundamentally gated by data quality and diversity. As we exhaust publicly available human-written text, the community faces an impending data wall — by some estimates, high-quality internet text will be fully consumed by 2026–2028. Synthetic data offers an escape route: use existing strong models to generate new training examples, creating a virtuous cycle where models improve by training on data they (or stronger peers) produce.

Synthetic data isn't merely a stop-gap. Research consistently shows that curated synthetic data can outperform raw web crawls token-for-token. Microsoft's Phi series demonstrated that 1.3B-parameter models trained on carefully generated "textbook-quality" data can rival models 10× their size trained on generic web corpora. The key insight: data quality dominates data quantity once you control for token budget.

Core thesis: A 1B model trained on 50B high-quality synthetic tokens can match a 7B model trained on 1T web tokens on many benchmarks. This is the "small model, great data" paradigm that Phi, Orca, and TinyLlama validated.

Three pillars define the synthetic data landscape: (1) generation — how we produce data, (2) filtering — how we ensure quality and remove contamination, and (3) self-improvement — how models bootstrap from their own outputs to get better over successive rounds.

Synthetic Data Pipeline

The canonical synthetic data pipeline forms a closed loop. A generator model produces candidate examples, a filter (reward model, rule-based heuristic, or stronger LLM judge) scores them, the surviving examples enter the training set, the fine-tuned model is evaluated, and the cycle repeats. Each iteration tightens the quality distribution.

Textbooks Are All You Need (Phi Series)

Microsoft Research's Phi-1 (2023) proved that a 1.3B model could achieve 50.6% on HumanEval — surpassing many 10B+ models — by training almost exclusively on synthetic "textbook-quality" code data. The approach was deceptively simple: use GPT-3.5 to generate ~1B tokens of self-contained coding exercises and explanations, then combine with ~6B tokens of filtered web data selected for educational quality.

# Phi-1 data generation prompt template (simplified)
SYSTEM = """You are a computer science professor writing a textbook.
Write a self-contained Python exercise that teaches {topic}.
Include: problem statement, step-by-step solution, explanation.
The code must be correct and runnable."""

# Topics sampled from curriculum:
topics = [
    "recursion with memoization",
    "binary search edge cases",
    "graph BFS with visited set",
    "dynamic programming on strings",
    ...  # ~1500 topics covering CS fundamentals
]

for topic in topics:
    for variation in range(200):  # multiple exercises per topic
        prompt = SYSTEM.format(topic=topic)
        response = gpt35_generate(prompt, temperature=0.7)
        if passes_quality_filter(response):
            dataset.append(response)

The Phi-2 (2.7B) extended this to general reasoning by mixing synthetic textbook data with web data filtered through a "textbook-quality" classifier. It matched Llama-2-70B on several reasoning benchmarks. Phi-3 (3.8B) pushed further with heavily filtered web data + synthetic data, achieving GPT-3.5-level performance. The Phi-3 data recipe used a multi-stage pipeline:

🔬 Phi-1 (1.3B)

Data: ~7B tokens (1B synthetic + 6B filtered web)

HumanEval: 50.6%

Key insight: Textbook-quality synthetic data alone can teach strong coding ability

🧠 Phi-2 (2.7B)

Data: ~250B tokens (synthetic + filtered web)

MMLU: 56.7% (matches Llama-2-70B)

Key insight: Extend textbook approach to general knowledge with quality-filtered web mix

⚡ Phi-3-mini (3.8B)

Data: ~4.8T tokens (heavily filtered + synthetic)

MMLU: 69.0% (GPT-3.5 level)

Key insight: Aggressive web filtering + synthetic data + longer training enables frontier-class small models

📊 Takeaway

Pattern: Each Phi generation increases the ratio of synthetic to raw web data

Quality classifiers trained on Phi-1's synthetic data are reused to filter web data for Phi-2, creating a bootstrapping flywheel

Self-Instruct & Evol-Instruct

Self-Instruct (Wang et al., 2022) pioneered the idea of a model generating its own instruction-following training data. The recipe: start with a small seed set (~175 human-written instructions), prompt the model to generate new instructions, generate input-output pairs for each instruction, and filter for quality. The process generates thousands of diverse (instruction, input, output) triples from a minimal seed.

# Self-Instruct pipeline
def self_instruct(model, seed_tasks, n_generate=50000):
    task_pool = list(seed_tasks)  # start with 175 seed tasks
    generated = []

    while len(generated) < n_generate:
        # Step 1: Sample 8 seed tasks as few-shot examples
        examples = random.sample(task_pool, k=8)

        # Step 2: Generate new instruction
        new_instruction = model.generate(
            prompt=format_instruction_prompt(examples)
        )

        # Step 3: Classify as classification or generation task
        task_type = model.classify(new_instruction)

        # Step 4: Generate input (if needed) and output
        if task_type == "classification":
            # Output-first: generate output, then input
            output = model.generate(instruction=new_instruction)
            input_ = model.generate_input(new_instruction, output)
        else:
            # Input-first: generate input, then output
            input_ = model.generate_input(new_instruction)
            output = model.generate(new_instruction, input_)

        # Step 5: Filter — ROUGE-L < 0.7 against existing tasks
        if is_novel(new_instruction, task_pool, threshold=0.7):
            task_pool.append(new_instruction)
            generated.append((new_instruction, input_, output))

    return generated
Self-Instruct produced Alpaca: Stanford used Self-Instruct with GPT-3.5 (text-davinci-003) to generate 52K instruction-following examples for $500. Fine-tuning LLaMA-7B on this data produced Alpaca, which showed surprisingly strong instruction-following ability — demonstrating that the data recipe matters more than model scale for alignment.

Evol-Instruct (WizardLM)

Evol-Instruct (Xu et al., 2023) addresses Self-Instruct's tendency to produce simple, repetitive instructions. Instead of generating new instructions from scratch, Evol-Instruct takes existing instructions and evolves them through complexity-increasing transformations. The evolution operators work in two directions:

# Evol-Instruct: Evolve instructions to increase complexity
DEPTH_EVOLVERS = [
    "Add constraints",           # "Sort a list" → "Sort without built-ins, O(n log n)"
    "Deepen reasoning",          # "What is X?" → "Compare X and Y with tradeoffs"
    "Concretize",                # "Write code" → "Write a Redis-backed rate limiter"
    "Increase complexity",        # Add edge cases, multi-step requirements
    "Add reasoning steps",        # Require chain-of-thought or proofs
]
BREADTH_EVOLVERS = [
    "Generate related topic",     # Same difficulty, different domain
    "Mutate to new task type",    # Classification → generation
]

def evol_instruct(seed_instructions, model, rounds=4):
    pool = list(seed_instructions)
    for round in range(rounds):
        evolved = []
        for inst in pool:
            evolver = random.choice(DEPTH_EVOLVERS + BREADTH_EVOLVERS)
            new_inst = model.generate(
                prompt=f"Evolve this instruction using '{evolver}':\n{inst}"
            )
            if quality_check(new_inst):  # not too similar, answerable
                evolved.append(new_inst)
        pool.extend(evolved)
    return pool
WizardLM results: Evol-Instruct on 70K Alpaca instructions → WizardLM-7B surpassed ChatGPT on complex instruction-following (Vicuna bench) by 6.2%. WizardCoder applied the same technique to code — generating progressively harder coding problems — and achieved 57.3% on HumanEval (7B params), surpassing models 5× larger.

Distillation from Frontier Models

The most direct form of synthetic data generation is knowledge distillation: prompting a powerful "teacher" model (GPT-4, Claude, Gemini) to generate training data for a smaller "student" model. This transfers capabilities — reasoning patterns, instruction following, factual knowledge — without sharing model weights. The student can't exceed the teacher's ceiling, but it can match the teacher on specific domains at a fraction of the inference cost.

# Distillation data generation pipeline
def generate_distillation_data(teacher, prompts, strategy="cot"):
    dataset = []
    for prompt in prompts:
        if strategy == "cot":
            # Chain-of-thought distillation (Orca-style)
            response = teacher.generate(
                system="Think step by step. Explain your reasoning.",
                user=prompt
            )
        elif strategy == "multi_turn":
            # Generate realistic conversation data
            response = teacher.generate_conversation(
                scenario=prompt, turns=random.randint(3, 8)
            )
        elif strategy == "verification":
            # Generate answer + self-verification
            answer = teacher.generate(prompt)
            verification = teacher.generate(
                f"Check this answer for correctness:\n{answer}"
            )
            response = combine(answer, verification)

        dataset.append({"prompt": prompt, "response": response})
    return dataset

Quality Filtering & Decontamination

Raw synthetic data is noisy. Models hallucinate, generate incorrect code, produce repetitive outputs, and occasionally leak benchmark answers verbatim. Quality filtering is the single most impactful step in any synthetic data pipeline — the difference between training on filtered vs. unfiltered synthetic data can be 5–15% on downstream benchmarks.

# Multi-stage quality filtering pipeline
def quality_filter_pipeline(examples):
    filtered = []
    for ex in examples:
        # Stage 1: Rule-based filters
        if len(ex["response"]) < 50:              continue  # too short
        if detect_repetition(ex["response"]) > 0.4: continue  # n-gram repetition
        if contains_refusal(ex["response"]):         continue  # "I cannot..."
        if detect_language(ex["response"]) != "en":  continue  # wrong language

        # Stage 2: Reward model scoring
        score = reward_model.score(ex["prompt"], ex["response"])
        if score < 0.6:                             continue  # low quality

        # Stage 3: Decontamination — check against benchmark datasets
        if ngram_overlap(ex, benchmark_data, n=13) > 0.8:
            continue  # likely benchmark contamination

        # Stage 4: Embedding-based deduplication
        embedding = encoder.encode(ex["response"])
        if cosine_sim(embedding, existing_embeddings) > 0.95:
            continue  # near-duplicate

        filtered.append({**ex, "quality_score": score})
    return filtered

Reward Model as Quality Filter

The most effective filtering approach uses a reward model (RM) — the same type trained for RLHF — to score each synthetic example. The RM provides a scalar quality estimate that captures instruction-following, correctness, helpfulness, and safety simultaneously. Best-of-N sampling generates N responses per prompt and keeps only the top-scoring one:

# Best-of-N sampling with reward model filtering
def best_of_n(model, reward_model, prompt, n=16):
    candidates = []
    for _ in range(n):
        response = model.generate(prompt, temperature=0.8)
        score = reward_model.score(prompt, response)
        candidates.append((response, score))

    # Keep the highest-scoring response
    best = max(candidates, key=lambda x: x[1])
    return best

# At N=16, best-of-N typically improves quality by 15-30%
# but costs 16× the compute per example
Decontamination matters: GPT-4 memorizes benchmark examples. If you distill from GPT-4 without decontamination, your model's benchmark scores will be inflated. The standard approach: compute 13-gram overlap between each synthetic example and all known benchmarks (MMLU, HumanEval, GSM8K, etc.). Remove any example with >80% overlap. Also apply embedding-based matching for paraphrased contamination.

Self-Play (SPIN) & Improvement Loops

Self-Play Fine-Tuning (SPIN) (Chen et al., 2024) introduces a game-theoretic framework for self-improvement. Instead of requiring a stronger teacher model, SPIN lets a model improve by playing against itself. The core idea: in each round, the model from the previous iteration generates responses (the "opponent"), and the current model learns to distinguish these from human-written ground truth (the "main player").

# SPIN: Self-Play Fine-Tuning
def spin_training(model, human_data, rounds=3):
    for t in range(rounds):
        # Opponent: previous iteration generates responses
        synthetic_responses = []
        for (prompt, human_response) in human_data:
            opponent_response = model.generate(prompt)
            synthetic_responses.append({
                "prompt": prompt,
                "chosen": human_response,       # real human data
                "rejected": opponent_response,  # model's own output
            })

        # Main player: train to prefer human over self-generated
        model = dpo_train(model, synthetic_responses)

        # Convergence: when model output ≈ human data
        if win_rate(model, human_data) > 0.95:
            break

    return model

# SPIN results on Zephyr-7B:
# Round 0 → 1: MT-Bench 5.94 → 6.78 (+14%)
# Round 1 → 2: 6.78 → 7.02 (+3.5%)
# Round 2 → 3: 7.02 → 7.10 (+1.1%) — diminishing returns
Why SPIN works: The key theoretical insight is that SPIN converges to a fixed point where the model's distribution matches the human data distribution. At this fixed point, the model can no longer distinguish its own outputs from human outputs. Each round narrows the gap, with the largest gains in round 1 (typically 10–15%) and diminishing returns thereafter.

Other Self-Improvement Methods

SPIN is one instance of a broader family of self-improvement loops. Related approaches include:

🔄 STaR (Self-Taught Reasoner)

Model generates chain-of-thought rationales for training problems. Keep rationales that lead to correct answers. Fine-tune on the (rationale, answer) pairs. Iterate.

Result: GSM8K accuracy +12% over 3 iterations without any external model.

🪞 ReST (Reinforced Self-Training)

Generate → Filter (keep above threshold) → Train. Unlike SPIN, uses an external reward signal (correctness checker or RM) rather than human data.

Result: Used by Google for PaLM-2 math improvement: MATH +8.6%, GSM8K +6.4%.

🎯 RAFT (Reward-rAnked Fine-Tuning)

Generate K responses per prompt, reward-rank them, fine-tune only on top responses. Simple but effective — essentially best-of-N as a training strategy.

Result: Consistent 3–8% gains across helpfulness, truthfulness, safety evals.

🧬 Iterative DPO

Run DPO → generate new pairs from updated model → run DPO again. Each round shifts the preference boundary. Online DPO variant generates on-policy data.

Result: 2–3 rounds typically saturate gains. Beyond that, overfitting to reward model biases emerges.

Constitutional AI Data

Constitutional AI (CAI) (Bai et al., 2022) from Anthropic generates safety-aligned training data without human feedback on harmful content. The approach uses a set of written principles (the "constitution") to guide a model in self-critiquing and revising its own harmful outputs. This produces (harmful prompt, revised safe response) pairs that can be used for SFT and RLHF without exposing human annotators to toxic content.

# Constitutional AI data generation pipeline
CONSTITUTION = [
    "Choose the response that is least harmful or toxic",
    "Choose the response that is most helpful while being safe",
    "Choose the response that best respects user autonomy",
    "Choose the response that is most honest",
    ...  # ~16 principles
]

def generate_cai_data(model, red_team_prompts):
    sft_pairs = []
    for prompt in red_team_prompts:
        # Step 1: Generate initial (potentially harmful) response
        initial = model.generate(prompt)

        # Step 2: Self-critique using a constitutional principle
        principle = random.choice(CONSTITUTION)
        critique = model.generate(
            f"Critique this response using the principle: '{principle}'\n"
            f"Response: {initial}"
        )

        # Step 3: Revise based on critique
        revision = model.generate(
            f"Revise the response based on this critique:\n"
            f"Critique: {critique}\nOriginal: {initial}"
        )

        sft_pairs.append({"prompt": prompt, "response": revision})

    # Step 4: Use revision pairs for RLAIF (RL from AI Feedback)
    preference_data = create_preferences(sft_pairs, model, CONSTITUTION)
    return sft_pairs, preference_data
CAI's impact: Constitutional AI enabled Anthropic to scale safety training without a linear increase in human red-teaming effort. The constitution acts as a compressed form of human values — 16 principles replace thousands of individual human preference labels. Models trained with CAI show comparable safety to RLHF-trained models while requiring 10× fewer human annotations for the safety component.

The CAI framework has been adopted broadly: RLAIF (RL from AI Feedback) generalizes this by replacing all human preferences with AI-generated preferences, and Self-Alignment methods (Sun et al., 2023) use a similar principle-driven self-refinement loop to align models from scratch using only a set of written rules and the model's own generations.

Data Contamination Risks

Synthetic data amplifies contamination risk. When teacher models have memorized benchmark datasets, distillation transfers that memorization to student models, creating artificially inflated scores. This is the benchmark contamination problem — and it's becoming the central challenge in evaluating models trained on synthetic data.

Real-world example: Phi-1 was initially criticized for potential HumanEval contamination, since GPT-3.5 (its teacher) had likely seen HumanEval problems during pre-training. Microsoft addressed this by (1) testing on CodeContests (post-GPT-3.5 training cutoff), (2) n-gram decontamination, and (3) showing consistent gains on held-out problems. But the concern remains: how do you prove your synthetic data isn't contaminated when the teacher's training data is opaque?

Contamination Detection & Mitigation

# Comprehensive decontamination pipeline
def decontaminate(synthetic_data, benchmarks):
    clean_data = []
    for example in synthetic_data:
        contaminated = False

        # Method 1: Exact n-gram matching (n=13, standard)
        for bench in benchmarks:
            if ngram_overlap(example, bench, n=13) > 0.8:
                contaminated = True
                break

        # Method 2: Embedding similarity (catches paraphrases)
        if not contaminated:
            emb = embed(example["response"])
            for bench_emb in benchmark_embeddings:
                if cosine_sim(emb, bench_emb) > 0.92:
                    contaminated = True
                    break

        # Method 3: Canary string detection
        if not contaminated:
            if contains_canary(example["response"], canary_strings):
                contaminated = True

        if not contaminated:
            clean_data.append(example)

    print(f"Removed {len(synthetic_data) - len(clean_data)} contaminated")
    return clean_data

Best practices for contamination-aware synthetic data pipelines: (1) Always run n-gram decontamination against all benchmarks you plan to evaluate on. (2) Use embedding-based detection for paraphrase contamination. (3) Include held-out test sets that were created after your teacher model's training cutoff. (4) Report both contaminated and decontaminated scores. (5) Prefer live evaluation (Chatbot Arena, LMSYS) over static benchmarks for final comparisons.

Scaling Synthetic Data

Scaling synthetic data follows different laws than scaling natural data. While Chinchilla scaling laws suggest optimal compute allocation between model size and data quantity for web data, synthetic data introduces a third axis: data quality. The effective token count of high-quality synthetic data is 5–50× higher than raw web tokens on a per-token basis.

📈 Scaling Dimension 1: Volume

More synthetic examples generally help up to a saturation point. For instruction-tuning: diminishing returns beyond ~100K high-quality examples for 7B models. For pre-training: the ceiling is much higher (trillions of tokens), but diversity becomes the bottleneck.

🎯 Scaling Dimension 2: Diversity

Generating 1M paraphrases of the same 100 instructions is worse than 10K diverse instructions. Evol-Instruct, topic trees, and domain-specific prompt templates are all diversity-scaling strategies.

🏆 Scaling Dimension 3: Quality

Best-of-N with N=64 produces dramatically better data than N=1, but costs 64×. Reward-model filtering, multi-judge consensus, and execution-based verification (for code) all trade compute for quality.

🔄 Scaling Dimension 4: Iterations

Self-improvement loops (SPIN, STaR, iterative DPO) provide diminishing returns: round 1 gives 10–15%, round 2 gives 3–5%, round 3 gives 1–2%. Model collapse risk increases with each iteration.

The Model Collapse Problem

A critical risk when scaling synthetic data: model collapse (Shumailov et al., 2023). When models are trained on data generated by previous models recursively, the tails of the distribution erode over generations. Rare but important knowledge — minority viewpoints, niche facts, unusual phrasings — progressively disappear, causing the model to converge to a narrow, homogeneous output distribution.

# Model collapse illustration
# Generation 0: Human data (rich, diverse, long-tailed)
# Generation 1: Model trained on Gen 0 → generates data for Gen 2
# Generation 2: Slightly less diverse, tail events underrepresented
# ...
# Generation N: Converged to mode of distribution, lost rare patterns

def mitigate_collapse(synthetic_data, human_data, ratio=0.1):
    # KEY MITIGATION: Always mix in original human data
    combined = synthetic_data + random.sample(
        human_data,
        k=int(len(synthetic_data) * ratio)
    )
    random.shuffle(combined)
    return combined

# Additional mitigations:
# 1. Never train solely on synthetic data — always mix 5-20% human data
# 2. Use diverse teacher models (ensemble distillation)
# 3. Monitor distribution metrics (entropy, n-gram diversity) each round
# 4. Apply nucleus sampling (top-p) to maintain output diversity
# 5. Cap iteration depth — rarely go beyond 3 self-improvement rounds
The practical recipe (2024): The most successful synthetic data pipelines combine: (1) a strong open teacher model (Llama-3, Mixtral, Qwen-2) to avoid ToS issues, (2) Evol-Instruct-style diversity amplification, (3) reward-model filtering with best-of-N (N=8–16), (4) rigorous decontamination, (5) 10–20% human data mixed in to prevent collapse, and (6) 1–2 rounds of self-play refinement (SPIN/iterative DPO). This recipe consistently produces 7B models that compete with 70B models on targeted tasks.
