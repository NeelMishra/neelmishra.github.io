Mixture of Experts: Architecture and Ops

MLOps Series SOTA & Frontiers

MoE Architecture Overview

Mixture of Experts (MoE) models replace every dense feed-forward layer (or a subset) with N parallel expert networks and a lightweight gating router that selects a sparse subset of experts per token. This decouples total parameter count from per-token compute: Mixtral 8×7B has 46.7B total parameters but only activates ~12.9B per forward pass (2 of 8 experts per layer).

The core equation for an MoE layer output given input token x:

# MoE layer forward pass # G(x) = Softmax(W_g · x) → router logits # TopK selects k experts with highest gate values # y = Σ_{i ∈ TopK} G(x)_i · E_i(x) import torch import torch.nn as nn class MoELayer(nn.Module): def __init__(self, d_model, num_experts, top_k, d_ff): super().__init__() self.num_experts = num_experts self.top_k = top_k self.gate = nn.Linear(d_model, num_experts, bias=False) self.experts = nn.ModuleList([ nn.Sequential( nn.Linear(d_model, d_ff), nn.GELU(), nn.Linear(d_ff, d_model) ) for _ in range(num_experts) ]) def forward(self, x): # x: (batch, seq_len, d_model) gate_logits = self.gate(x) # (B, S, num_experts) weights, indices = torch.topk( torch.softmax(gate_logits, dim=-1), self.top_k, dim=-1 ) weights = weights / weights.sum(dim=-1, keepdim=True) out = torch.zeros_like(x) for i in range(self.top_k): expert_idx = indices[..., i] w = weights[..., i].unsqueeze(-1) for e in range(self.num_experts): mask = (expert_idx == e) if mask.any(): out[mask] += w[mask] * self.experts[e](x[mask]) return out
Key insight: MoE achieves 4–6× more parameters at roughly the same FLOPs as a dense model. Switch Transformer (2022) scaled to 1.6T parameters while matching the per-step cost of a T5-Base model.

Routing Mechanisms & Gating

The router (gating network) is the most critical design choice. Different architectures use fundamentally different strategies:

Top-K Gating (Standard)

The original Shazeer et al. (2017) approach uses top-2 gating: each token is dispatched to its two highest-scoring experts, with outputs combined via normalized gate weights. This remains the default in Mixtral 8×7B and GShard.

Top-1 / Switch Routing

Switch Transformer (Fedus et al., 2022) simplified to top-1: each token goes to exactly one expert. This halves communication cost and improves throughput at the expense of some quality. The key finding: top-1 works if you add auxiliary load-balancing loss and use large enough expert capacity.

Shared + Routed Experts (DeepSeek-MoE)

DeepSeek-MoE (Dai et al., 2024) introduces shared experts that process every token alongside routed experts. With 2 shared + 6 routed (top-2 of 64 fine-grained experts), this avoids the "representation collapse" problem where important general knowledge must be duplicated across experts.

# DeepSeek-MoE style: shared + routed experts class DeepSeekMoE(nn.Module): def __init__(self, d_model, n_shared, n_routed, top_k, d_ff): super().__init__() self.shared = nn.ModuleList([ FFN(d_model, d_ff) for _ in range(n_shared) ]) self.routed = nn.ModuleList([ FFN(d_model, d_ff // 4) for _ in range(n_routed) ]) self.gate = nn.Linear(d_model, n_routed, bias=False) self.top_k = top_k def forward(self, x): # All tokens pass through shared experts shared_out = sum(e(x) for e in self.shared) # Route to top-k of fine-grained experts scores = torch.softmax(self.gate(x), dim=-1) w, idx = torch.topk(scores, self.top_k, dim=-1) routed_out = sparse_dispatch(x, self.routed, w, idx) return shared_out + routed_out

Expert Choice Routing

Zhou et al. (2022) flipped the paradigm: instead of tokens choosing experts, experts choose their top-k tokens. Each expert picks a fixed number of tokens, guaranteeing perfect load balance. The downside: some tokens may be processed by zero experts (dropped) or many experts (over-computed), requiring a secondary padding mechanism.

Capacity Factors & Load Balancing

Without intervention, routers collapse to sending all tokens to 1–2 "popular" experts, wasting the rest. Two mechanisms prevent this:

Capacity Factor (CF)

Each expert has a buffer size = CF × (tokens_in_batch / num_experts). Switch Transformer uses CF = 1.25 during training and CF = 2.0 at inference for headroom. Tokens exceeding capacity are dropped and pass through via a residual connection.

# Capacity factor computation def compute_expert_capacity(num_tokens, num_experts, capacity_factor): # Perfect balance: each expert gets tokens/experts tokens tokens_per_expert = num_tokens / num_experts capacity = int(capacity_factor * tokens_per_expert) return capacity # Example: batch of 2048 tokens, 8 experts, CF=1.25 # capacity = 1.25 * (2048 / 8) = 320 slots per expert # With top-1: at most 320 tokens routed to any single expert # Overflow tokens use residual skip (identity)

Auxiliary Load Balancing Loss

Added to the main loss to penalize uneven routing. The standard formulation from Switch Transformer:

# L_balance = α · N · Σ_i (f_i · P_i) # f_i = fraction of tokens routed to expert i # P_i = mean router probability assigned to expert i # α = balancing coefficient (typically 0.01) def load_balancing_loss(gate_probs, expert_indices, num_experts, alpha=0.01): # gate_probs: (B*S, num_experts) — softmax router output # expert_indices: (B*S,) — chosen expert per token f = torch.zeros(num_experts, device=gate_probs.device) for i in range(num_experts): f[i] = (expert_indices == i).float().mean() P = gate_probs.mean(dim=0) # mean prob per expert return alpha * num_experts * (f * P).sum()
Warning: Setting α too high collapses routing to uniform (every expert gets equal traffic, defeating specialization). Too low and you get expert collapse. Switch Transformer found α = 0.01 optimal. DeepSeek-V2 uses α = 0.001 with their finer-grained experts.

Training Instability & Fixes

MoE models are notoriously harder to train than dense equivalents. Key failure modes:

1. Router Z-Loss

Router logits can grow unboundedly, causing numerical instability in softmax. Zoph et al. (ST-MoE, 2022) introduced the router z-loss that penalizes large logits:

# Router z-loss: penalizes large gate logits # L_z = (1/B) · Σ (log Σ exp(z_i))² def router_z_loss(gate_logits, coeff=0.001): # gate_logits: (B*S, num_experts) log_sum_exp = torch.logsumexp(gate_logits, dim=-1) return coeff * (log_sum_exp ** 2).mean()

2. Expert Dropout During Training

Randomly dropping 1–2 experts per step forces the router to spread probability mass. Mixtral uses this implicitly by having a small expert count (8) with top-2, where the loss landscape naturally distributes load.

3. Lower Precision Sensitivity

MoE routers are extremely sensitive to numerical precision. Switch Transformer found that training the router in FP32 while the rest of the model uses BF16 eliminates most instabilities. This selective precision is now standard practice.

Production recipe: Router in FP32 + z-loss (0.001) + load-balancing loss (0.01) + jitter noise (uniform ±0.01 on router input). This combination from ST-MoE-32B trains stably across 2048 TPU-v4 chips.

4. Expert Parallelism Communication

During training, all-to-all communication shuffles tokens between devices. For a model with E experts on E devices, each step requires:

  • Dispatch all-to-all: send each token to the device hosting its selected expert(s)
  • Combine all-to-all: gather expert outputs back to original device
  • Total communication volume per layer: 2 × batch_size × seq_len × d_model × sizeof(dtype)

Serving Challenges

MoE inference has unique operational challenges that make deployment significantly harder than dense models:

Memory Footprint

All experts must be in memory even though only k are active. Mixtral 8×7B needs ~93GB in FP16 (vs ~26GB for a 13B dense model with equivalent active parameters). This means 2× A100-80GB GPUs minimum, despite only using 13B parameters per token.

Expert Parallelism vs Tensor Parallelism

Expert Parallelism (EP)

  • Each GPU holds a subset of experts
  • Requires all-to-all for token dispatch
  • Scales with number of experts
  • Latency: dominated by routing + communication

Tensor Parallelism (TP)

  • Each GPU holds a shard of every expert
  • Requires all-reduce per expert computation
  • Scales with hidden dimension
  • Latency: dominated by all-reduce volume

Batch Size Sensitivity

Small batch sizes cause severe GPU underutilization because experts receive very few tokens. With batch size 1 and top-2 of 8 experts, each expert processes 0–2 tokens per layer — far below the thousands needed to saturate a GPU. Solutions:

  • Continuous batching with large queues to maximize batch occupancy
  • Expert buffering: accumulate tokens across multiple microbatches before dispatching
  • Expert offloading: keep inactive experts on CPU/SSD, load on-demand (DeepSeek-V2 uses this for 160 experts)
Latency trap: MoE models often have higher latency than smaller dense models despite lower FLOPs. The all-to-all communication and load imbalance at inference can add 2–5ms per layer. For Mixtral on 2× A100s with EP=2, expect ~40ms per token vs ~25ms for a 13B dense model on 1× A100.

Quantization Challenges

Experts develop different weight distributions, making per-tensor quantization lossy. Best practices:

  • Use per-expert quantization scales (each expert gets its own INT4/INT8 calibration)
  • Keep the router in FP16 — quantizing the gating network degrades routing accuracy catastrophically
  • GPTQ/AWQ work well per-expert; Mixtral INT4 (3.5 bits effective) fits in a single A100-80GB

Interview Essentials

Key points for system design and architecture interviews at frontier labs:

Q: Why not just scale dense models?

Dense scaling hits a wall: 8× more compute for 2× quality (Chinchilla scaling). MoE breaks this by increasing model capacity (parameters) without proportionally increasing FLOPs. A 46.7B MoE matches or beats a 70B dense model at 3× less inference compute.

Q: How do you handle expert load imbalance in production?

# Production monitoring for expert utilization def monitor_expert_balance(routing_decisions, num_experts): # Track per-expert token counts over a window counts = torch.bincount(routing_decisions, minlength=num_experts) utilization = counts.float() / counts.sum() # Ideal: 1/num_experts for each ideal = 1.0 / num_experts imbalance = (utilization - ideal).abs().max().item() # Alert if worst-case imbalance exceeds 3× ideal if imbalance > 2.0 * ideal: alert(f"Expert imbalance: {imbalance:.3f}, max_util={utilization.max():.3f}") return utilization

Q: What's the memory vs compute tradeoff?

MoE models trade memory bandwidth for compute efficiency. You store 4–8× more weights but only read k/N of them per token. This makes MoE latency memory-bound at small batch sizes (loading expert weights) and compute-bound at large batch sizes (enough tokens to saturate each expert's matmul).

Q: How would you deploy Mixtral 8×7B cost-effectively?

  • Quantize per-expert to INT4: Reduces from 93GB to ~25GB, fits single A100-80GB
  • Use expert parallelism across 2 GPUs for full precision: 4 experts per GPU, all-to-all over NVLink
  • Continuous batching with batch sizes ≥ 64 to amortize routing overhead
  • Speculative decoding with a small dense draft model to reduce per-token latency
