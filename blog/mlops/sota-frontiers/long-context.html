Long Context: RoPE Scaling, ALiBi, Ring Attention

MLOps Series SOTA & Frontiers

Positional Encoding Foundations

Transformers are permutation-invariant by design—without positional information, a self-attention layer produces the same output regardless of token order. Every modern LLM therefore injects position signals, but the choice of how dictates the model's ability to extrapolate beyond its training context length.

The original "Attention Is All You Need" paper used sinusoidal positional encodings: fixed sine and cosine functions of different frequencies added directly to token embeddings. Learned absolute position embeddings (used by GPT-2/3) replaced these but introduced the same hard ceiling—positions beyond the trained maximum have no meaningful representation.

The core problem: Absolute position embeddings (sinusoidal or learned) create a fixed vocabulary of positions. A model trained with max_position = 2048 has literally never seen position 2049 during training. At inference, those positions produce out-of-distribution embeddings that degrade attention patterns catastrophically.

This limitation drove the field toward relative position encodings—methods that encode the distance between tokens rather than their absolute index. The two dominant families today are RoPE (Rotary Position Embedding) and ALiBi (Attention with Linear Biases), each with fundamentally different philosophies on how to represent position.

The SVG above illustrates the core challenge. RoPE encodes position via rotation at different frequencies per dimension. Within the training window, all frequency bands are well-calibrated. Beyond it, high-frequency dimensions (which complete many rotations within the training range) encounter angular positions never seen during training, producing chaotic attention scores.

RoPE Internals

Rotary Position Embedding (RoPE), introduced in RoFormer (Su et al., 2021), encodes position by rotating query and key vectors in 2D subspaces. For a head dimension d, RoPE pairs up consecutive dimensions (2i, 2i+1) and applies a rotation by angle m · θ_i, where m is the token position and θ_i is the frequency for dimension pair i.

# RoPE: Rotary Position Embedding core computation
import torch

def compute_rope_freqs(dim, max_pos, base=10000):
    # θ_i = base^(-2i/d) for i = 0, 1, ..., d/2 - 1
    freqs = 1.0 / (base ** (torch.arange(0, dim, 2).float() / dim))
    # positions: [0, 1, 2, ..., max_pos-1]
    positions = torch.arange(max_pos).float()
    # outer product → angles[m, i] = m * θ_i
    angles = torch.outer(positions, freqs)  # [max_pos, d/2]
    return torch.polar(torch.ones_like(angles), angles)

def apply_rope(x, freqs_cis):
    # x: [batch, seq_len, n_heads, head_dim]
    # Reshape to complex: pair (2i, 2i+1) → complex number
    x_complex = torch.view_as_complex(
        x.float().reshape(*x.shape[:-1], -1, 2)
    )
    # Rotate by element-wise multiplication with unit complex numbers
    x_rotated = x_complex * freqs_cis
    # Convert back to real
    return torch.view_as_real(x_rotated).flatten(-2)

The key mathematical property: when computing q_m · k_n (the dot product between query at position m and key at position n), the rotations compose such that the result depends only on the relative distance m - n. This gives RoPE its relative positional encoding behavior while being applied as an absolute rotation.

Why the base matters: The base frequency (default 10,000) controls how quickly each dimension pair rotates. Dimension pair 0 rotates fastest (θ₀ = 1.0), while the last pair rotates slowest (θ_{d/2-1} ≈ 0.0001). Increasing the base stretches all frequencies, making rotations slower—this is the foundation of NTK-aware scaling.

RoPE is used by virtually all modern open-weight LLMs: LLaMA 1/2/3, Mistral, Qwen, DeepSeek, Gemma, and Phi. Its simplicity (no learned parameters, applied only to Q and K) and clean mathematical properties make it the default choice. The challenge is extending it beyond the training window.

RoPE Scaling Methods

When a RoPE-based model is trained with context length L_train and needs to operate at L_target > L_train, the high-frequency dimensions see rotation angles never encountered during training. Multiple scaling strategies have emerged to address this:

Position Interpolation (PI)

The simplest approach: linearly scale down all positions by a factor s = L_train / L_target. Position 8192 in a model trained on 4096 tokens becomes position 4096 (mapped back into the training range). This works but degrades resolution—tokens that were 1 position apart now appear 0.5 positions apart.

# Position Interpolation (Chen et al., 2023)
def position_interpolation(positions, L_train, L_target):
    scale = L_train / L_target  # e.g. 4096/16384 = 0.25
    return positions * scale      # compress all positions into [0, L_train)

# Problem: position 100 and 101 now differ by only 0.25
# → reduced ability to distinguish nearby tokens

NTK-Aware Scaling

Neural Tangent Kernel-aware scaling changes the RoPE base frequency instead of scaling positions. By increasing the base from 10,000 to a higher value, all rotation frequencies are reduced proportionally. The insight: high-frequency dimensions (which cause extrapolation failures) are slowed the most, while low-frequency dimensions (already well-behaved) are minimally affected.

# NTK-Aware Scaling (Reddit user "bloc97", 2023)
def ntk_aware_rope(dim, max_pos, base=10000, scale=4):
    # Increase base frequency proportionally
    new_base = base * scale ** (dim / (dim - 2))
    # Example: dim=128, scale=4 → new_base ≈ 160,000
    freqs = 1.0 / (new_base ** (torch.arange(0, dim, 2).float() / dim))
    positions = torch.arange(max_pos).float()
    return torch.outer(positions, freqs)

# Key insight: no position compression → nearby tokens stay distinguishable
# High-freq dims slow down most, low-freq dims barely change

YaRN (Yet Another RoPE ExtensioN)

YaRN combines the best of PI and NTK-aware scaling with a dimension-dependent strategy. It divides frequency dimensions into three bands and applies different treatments:

Low Frequencies

No modification. These dimensions rotate so slowly they never complete a full cycle within the training window. They naturally extrapolate well and should be left alone.

High Frequencies

Position Interpolation. These dimensions complete many rotations during training. Scaling positions keeps them within the trained range at the cost of some resolution loss.

Medium Frequencies

Smooth interpolation between the two extremes using a ramp function. This avoids the sharp boundary that NTK-aware scaling creates.

Attention Scaling

Temperature correction. YaRN multiplies attention logits by a scale factor to compensate for the entropy increase from longer sequences. Without this, attention becomes too diffuse.

# YaRN: dimension-dependent scaling (Peng et al., 2023)
def yarn_rope(dim, max_pos, base=10000, scale=4, beta_fast=32, beta_slow=1):
    freqs = 1.0 / (base ** (torch.arange(0, dim, 2).float() / dim))

    # Classify each frequency dimension into low/mid/high
    low_freq_wavelen = max_pos / beta_slow  # wavelength threshold
    high_freq_wavelen = max_pos / beta_fast
    wavelens = 2 * math.pi / freqs

    # Ramp function: 0 for high-freq, 1 for low-freq, smooth in between
    ramp = (wavelens - high_freq_wavelen) / (low_freq_wavelen - high_freq_wavelen)
    ramp = ramp.clamp(0, 1)

    # Blend: interpolated_freq for high-freq dims, original for low-freq
    scaled_freqs = freqs / scale               # position interpolation
    blended = (1 - ramp) * scaled_freqs + ramp * freqs
    return blended

Dynamic NTK Scaling

Rather than choosing a fixed extension ratio, Dynamic NTK increases the RoPE base on-the-fly as the sequence grows. When the current sequence length is within the training window, no scaling is applied. Once it exceeds it, the base is adjusted proportionally to the overshoot. This is the default strategy in many vLLM deployments.

Fine-tuning still helps: All RoPE scaling methods work best with a short fine-tuning phase on longer sequences (typically 1000–2000 steps). Position Interpolation without fine-tuning degrades significantly; NTK-aware and YaRN degrade more gracefully but still benefit from adaptation. LLaMA 3.1 was fine-tuned with a 128K context after initial pretraining at 8K.

ALiBi: Linear Bias Approach

Attention with Linear Biases (ALiBi), introduced by Press et al. (2022), takes a radically different approach: no positional encoding at all. Instead, ALiBi adds a static, linear bias to the attention scores that penalizes distant tokens. No learned parameters, no rotations—just a slope that decays attention as distance increases.

# ALiBi: Attention with Linear Biases
def alibi_attention(Q, K, V, n_heads):
    # Standard attention scores
    attn = Q @ K.transpose(-1, -2) / sqrt(d_k)

    # ALiBi bias: -m * |i - j| for head m
    # Slopes are head-specific, geometrically spaced
    slopes = get_alibi_slopes(n_heads)
    # slopes for 8 heads: [1/2, 1/4, 1/8, ..., 1/256]
    # slopes for 16 heads: [1/2, 1/2^(3/2), 1/4, ..., 1/2^8]

    seq_len = Q.size(-2)
    # Distance matrix: position i attends to position j
    positions = torch.arange(seq_len)
    bias = -slopes[:, None, None] * (positions[None, :, None] - positions[None, None, :]).abs()

    # Add bias directly to attention scores (before softmax)
    attn = attn + bias
    return softmax(attn) @ V

def get_alibi_slopes(n_heads):
    # Geometric sequence: 2^(-8/n_heads), 2^(-16/n_heads), ...
    ratio = 2 ** (-8.0 / n_heads)
    return torch.tensor([ratio ** (i + 1) for i in range(n_heads)])
ALiBi's extrapolation advantage: Since the bias is a simple linear function of distance, it naturally extends to any sequence length. A token 100K positions away simply gets a larger penalty—there's no "unseen position" problem. ALiBi-trained models (like BLOOM and MPT) showed strong length generalization, often extrapolating 2–6× beyond training length without modification.

✅ ALiBi Strengths

  • Zero-cost extrapolation—no scaling needed
  • No added parameters or computation
  • Simple implementation
  • Different slopes let heads specialize (local vs. global attention)

⚠️ ALiBi Limitations

  • Linear decay may be too aggressive for tasks needing long-range retrieval
  • Cannot learn position-dependent patterns (e.g., code indentation)
  • Less expressive than RoPE at short/medium ranges
  • Largely superseded by RoPE + scaling in most frontier models

In practice, the field has converged on RoPE as the dominant positional encoding. ALiBi's linear decay was too rigid—it couldn't represent the nuanced positional patterns that RoPE captures through its multi-frequency rotations. However, ALiBi's core insight (that simple biases can encode position) influenced later work on attention sinks and sliding window attention in Mistral.

Ring Attention for Distributed Long Context

Even with perfect positional encoding, long-context models face a hard constraint: KV cache memory. A single 1M-token sequence with a 70B model would require hundreds of GB of KV cache—far beyond any single GPU. Ring Attention (Liu et al., 2023) solves this by distributing the sequence across multiple devices in a ring topology.

How Ring Attention Works

The input sequence of length L is split into N chunks across N devices. Each device holds its own chunk's Q, K, V tensors. The computation proceeds in N rounds:

  1. Round 0: Each device computes local attention (its Q against its own K, V).
  2. Round 1: Each device sends its K, V to the next device in the ring and receives K, V from the previous device. It computes attention of its Q against the received K, V.
  3. Rounds 2 to N−1: Continue passing K, V around the ring. After N−1 rounds, every device has attended to every chunk's K, V.
  4. Combine: Use the online softmax trick (from Flash Attention) to merge partial attention outputs without storing full attention matrices.
# Ring Attention: distributed long-context attention
def ring_attention(Q_local, K_local, V_local, n_devices):
    # Each device holds a chunk of the sequence
    # Q_local: [batch, chunk_len, heads, dim]

    # Initialize running numerator/denominator for online softmax
    out = torch.zeros_like(Q_local)
    lse = torch.full((..., 1), float('-inf'))  # log-sum-exp

    K_recv, V_recv = K_local, V_local

    for round_idx in range(n_devices):
        # Overlap: start async send/recv for NEXT round
        if round_idx < n_devices - 1:
            K_next, V_next = async_ring_send_recv(K_recv, V_recv)

        # Compute partial attention: local Q × received K,V
        partial_out, partial_lse = flash_attention_fwd(
            Q_local, K_recv, V_recv
        )

        # Online softmax merge (numerically stable)
        out, lse = merge_attention_outputs(
            out, lse, partial_out, partial_lse
        )

        # Wait for next K,V to arrive
        if round_idx < n_devices - 1:
            K_recv, V_recv = K_next.wait(), V_next.wait()

    return out
Communication hiding: Ring Attention overlaps computation with communication. While device i computes attention against the current K,V chunk, it simultaneously sends that chunk to device i+1 and receives the next chunk from device i−1. If the chunk computation time ≥ transfer time, communication is fully hidden and the wall-clock time scales linearly with devices.

Ring Attention enables context lengths that scale linearly with the number of devices. With 8 GPUs, you can handle 8× the context of a single GPU. Google's 1M-token Gemini models and Anthropic's 200K Claude models both employ variants of distributed attention for processing extremely long sequences.

Context Window Landscape

The context window race has accelerated dramatically. Here is the current state of production context windows and the techniques each model family uses:

LLaMA 3.1 (Meta)

128K tokens. RoPE with increased base frequency (500K). Pre-trained at 8K, then extended via gradual fine-tuning: 8K → 32K → 128K with progressive training steps. Uses GQA (8 KV heads) to manage KV cache.

Claude 3.5 (Anthropic)

200K tokens. Position encoding details undisclosed. Achieves near-perfect retrieval across the full window (needle-in-a-haystack). Likely uses distributed attention internally for long sequences.

Gemini 1.5 Pro (Google)

1M+ tokens (up to 2M in preview). Uses a MoE architecture with efficient attention. Ring Attention or similar distributed attention for cross-device sequence sharding. Demonstrated video and audio processing spanning hours.

GPT-4 Turbo (OpenAI)

128K tokens. Details undisclosed. Known to struggle with retrieval in the middle of very long contexts ("lost in the middle" phenomenon). Likely uses sliding window + global attention hybrid.

"Lost in the middle" problem: Many models with large context windows show a U-shaped retrieval curve—they find information well at the beginning and end of the context but struggle with facts placed in the middle third. This is likely a training distribution artifact: during pre-training, documents rarely exceed a few thousand tokens, so the model has limited exposure to relevant information at position 50K.

KV Cache Memory at Scale

Long context fundamentally transforms the memory economics of LLM serving. The KV cache grows linearly with sequence length, and at 100K+ tokens, it often dominates total GPU memory—dwarfing the model weights themselves.

# KV cache memory formula:
# memory = 2 × layers × kv_heads × head_dim × seq_len × bytes_per_elem

# ────────────────────────────────────────────────────────────
# LLaMA 3.1 70B (GQA: 8 KV heads, 80 layers, head_dim=128)
# ────────────────────────────────────────────────────────────
kv_4k   = 2 * 80 * 8 * 128 * 4_096 * 2    # =  1.34 GB
kv_32k  = 2 * 80 * 8 * 128 * 32_768 * 2   # = 10.74 GB
kv_128k = 2 * 80 * 8 * 128 * 131_072 * 2  # = 42.95 GB

# With batch_size=8 at 128K context:
total   = 42.95 * 8                        # = 343.6 GB (!)

# ────────────────────────────────────────────────────────────
# For comparison — model weights (FP16):
# LLaMA 70B ≈ 140 GB
# At 128K×8 batch, KV cache is 2.5× the model size!
# ────────────────────────────────────────────────────────────

# Gemini-scale: hypothetical 1M context, 64 layers, 16 KV heads
kv_1m = 2 * 64 * 16 * 128 * 1_000_000 * 2  # = 524 GB per sequence
GQA is critical for long context: Grouped-Query Attention reduces KV heads from n_heads (e.g., 64) to a much smaller number (e.g., 8). For LLaMA 70B, GQA reduces KV cache by 8×. Without GQA, the 128K KV cache would be ~344 GB per sequence instead of 43 GB. Every modern long-context model uses GQA or MQA.

This memory pressure has driven multiple innovations in KV cache management for long-context models:

  • Paged Attention (vLLM): Manages KV cache in non-contiguous pages, eliminating memory fragmentation that wastes 60–80% of GPU memory with naive allocation.
  • KV Cache Quantization: Storing K and V in INT8 or even INT4 halves or quarters memory. FP8 KV cache is default in many vLLM deployments.
  • KV Cache Eviction: Policies like H₂O (Heavy Hitter Oracle) keep only the most "important" KV entries, discarding tokens that receive low attention scores.
  • Sliding Window + Sink Tokens: Mistral uses a fixed-size sliding window (4K) for local attention plus a small set of "sink" tokens (initial tokens that anchor attention), achieving infinite-length generation with bounded memory.
  • KV Cache Offloading: Move older KV entries to CPU memory or NVMe, fetching them back on demand. InfiniGen and similar systems prefetch predicted KV cache blocks.

Serving Long-Context Models

Serving models at 100K+ context lengths introduces unique engineering challenges that fundamentally differ from short-context serving:

Prefill Bottleneck

The prefill phase (processing the input prompt) is compute-bound and scales quadratically with sequence length for standard attention. A 128K-token prompt takes ~1,000× longer to prefill than a 4K prompt (not 32×). Flash Attention reduces the constant factor but doesn't change the O(n²) scaling. This means time-to-first-token (TTFT) can be 30+ seconds for very long prompts.

Disaggregated Prefill/Decode

Long-context serving benefits enormously from separating prefill and decode onto different hardware. Prefill is compute-heavy (benefits from dense GPU clusters), while decode is memory-bandwidth-bound (benefits from high-bandwidth GPUs with large memory). Systems like DistServe and Splitwise route these phases to specialized pools.

# Typical long-context serving architecture

# Prefill cluster: optimized for compute throughput
# - Dense A100/H100 nodes
# - Process 128K prompt in parallel using tensor parallelism
# - Generate KV cache, stream it to decode cluster

# Decode cluster: optimized for memory bandwidth
# - Nodes with large memory (8×H100 80GB)
# - Receive KV cache from prefill
# - Generate tokens one-by-one, memory-bandwidth bound

# KV cache transfer between clusters:
# 128K context, LLaMA 70B → ~43 GB KV cache
# Over NVLink/IB at 400 Gbps → ~0.86 seconds transfer
# Over PCIe Gen5 at 64 GBps → ~0.67 seconds transfer

Scheduling and Batching

Long-context requests are hostile to naive batching. A single 128K request consumes the KV cache memory that could serve 32 × 4K requests. Sophisticated schedulers must balance:

  • Priority queuing: Separate queues for short and long requests to prevent head-of-line blocking.
  • Preemption: Ability to pause a long-context decode (offloading its KV cache to CPU) to serve urgent short requests, then resume.
  • Prefix caching: System prompts and shared context can be cached and reused across requests, amortizing the 128K prefill cost over many users.
  • Chunked prefill: Split long prefills into chunks interleaved with decode steps from other requests, preventing latency spikes.
Cost reality: Serving a 128K-context request costs roughly 30–50× more than a 4K request (combining prefill compute, KV cache memory, and decode bandwidth). This is why API pricing for long-context models typically charges per-token for both input and output, with input tokens being significant cost drivers at long context lengths.

Interview Cheat Sheet

🎯 Key Questions

  • "How does RoPE differ from absolute positional encoding?" — Encodes relative distance via rotation; extends via base frequency scaling.
  • "Why not just train on longer sequences?" — O(n²) attention cost makes long pre-training prohibitively expensive; extension methods achieve 90%+ quality at <1% training cost.
  • "What limits context length in practice?" — KV cache memory, not positional encoding. A 1M-token KV cache can exceed 500 GB.

📊 Numbers to Know

  • RoPE default base: 10,000 (LLaMA 3.1 uses 500,000)
  • KV cache at 128K (70B): ~43 GB per sequence
  • Prefill time 128K: 10–30 seconds on 8×H100
  • Ring Attention scales context ∝ number of GPUs
  • ALiBi slopes: geometric series 2^(-8/n_heads)
