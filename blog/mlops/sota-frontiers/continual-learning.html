Continual Learning: Avoiding Catastrophic Forgetting

MLOps Series SOTA & Frontiers

The Catastrophic Forgetting Problem

Neural networks learn by adjusting weights to minimise loss on the current training data. When a model trained on Task A is subsequently fine-tuned on Task B, the weights that encoded Task A knowledge are overwritten — often dramatically. This phenomenon is called catastrophic forgetting, and it is the central challenge of continual (or lifelong) learning.

The curve above illustrates the core issue: as Task B training begins (dashed line), Task A accuracy plunges while Task B accuracy climbs. An ideal continual learner would maintain high performance on all previously seen tasks while assimilating new knowledge. The strategies below attack this problem from complementary angles: regularisation, replay, architecture expansion, and parameter isolation.

Stability–Plasticity Dilemma: Too much stability → the model cannot learn new tasks. Too much plasticity → old knowledge is overwritten. Every continual learning method navigates this trade-off.

Elastic Weight Consolidation (EWC)

Elastic Weight Consolidation (Kirkpatrick et al., 2017) adds a quadratic penalty that discourages changes to weights deemed important for previous tasks. Importance is estimated via the diagonal of the Fisher information matrix F, computed from the old task's data.

The EWC loss for task B is:

# EWC loss formulation L_total = L_B(θ) + (λ / 2) · Σ_i F_i · (θ_i − θ*_A,i)² # θ*_A = optimal weights after Task A # F_i = Fisher importance of weight i (diagonal approx) # λ = regularisation strength (typically 400–5000)

Weights with high Fisher values are "anchored" near their Task A optima; weights that barely affected Task A loss are free to move for Task B.

import torch def compute_fisher(model, dataloader, criterion): """Estimate diagonal Fisher information from a task's data.""" fisher = {n: torch.zeros_like(p) for n, p in model.named_parameters()} model.eval() for x, y in dataloader: model.zero_grad() loss = criterion(model(x), y) loss.backward() for n, p in model.named_parameters(): fisher[n] += p.grad.data ** 2 # Average over dataset for n in fisher: fisher[n] /= len(dataloader) return fisher def ewc_loss(model, fisher, old_params, lam=1000): """Quadratic penalty weighted by Fisher importance.""" loss = 0 for n, p in model.named_parameters(): loss += (fisher[n] * (p - old_params[n]) ** 2).sum() return (lam / 2) * loss
Limitation: EWC uses a diagonal Fisher approximation — it ignores correlations between weights. For large models, even storing the full diagonal can be expensive, and the approximation degrades with many sequential tasks (Fisher values accumulate noise).

Variants of EWC

Online EWC

Maintains a running Fisher estimate instead of storing one per task. Uses exponential decay: F = α·F_new + (1−α)·F_old. Memory cost is O(|θ|) regardless of task count.

Synaptic Intelligence (SI)

Tracks importance during training by accumulating the contribution of each weight to loss reduction over the optimisation path. No separate Fisher pass required.

Experience Replay

The simplest and often most effective continual learning technique: keep a small replay buffer of examples from previous tasks and interleave them with new task data during training. This directly combats forgetting by refreshing old task gradients.

import random class ReplayBuffer: def __init__(self, max_size=5000): self.buffer = [] self.max_size = max_size def add_task_data(self, dataset, num_samples=500): """Reservoir-sample examples from a completed task.""" samples = random.sample(dataset, min(num_samples, len(dataset))) self.buffer.extend(samples) if len(self.buffer) > self.max_size: self.buffer = random.sample(self.buffer, self.max_size) def sample_batch(self, batch_size=32): return random.sample(self.buffer, min(batch_size, len(self.buffer))) # Training loop with replay for batch in new_task_loader: replay = buffer.sample_batch(batch_size=16) combined = concat(batch, replay) loss = criterion(model(combined.x), combined.y) loss.backward() optimizer.step()

Replay Variants

Dark Experience Replay (DER)

Stores not just inputs and labels, but also the model's logits at storage time. The replay loss includes a knowledge-distillation term that preserves the old model's soft predictions.

Generative Replay

Train a generative model (VAE / GAN) on previous tasks and synthesise pseudo-examples on the fly. Eliminates the need to store raw data — important when data cannot be retained for privacy reasons.

Gradient Episodic Memory (GEM)

Uses stored examples to define constraints: the update for the new task must not increase loss on any replay example. Solved via a small QP (quadratic program) at each step.

Averaged GEM (A-GEM)

Relaxes GEM's per-example constraints to a single average constraint, reducing per-step cost from O(k·d) to O(d) while retaining most of the forgetting protection.

Rule of thumb: A replay buffer holding just 1–5% of previous data often recovers 80–90% of the full joint-training performance. Replay is the single strongest baseline in nearly every continual learning benchmark.

Progressive Neural Networks

Progressive Neural Networks (Rusu et al., 2016) avoid forgetting entirely by freezing all parameters of previous task columns and adding a new column (a fresh sub-network) for each new task. Lateral connections allow the new column to leverage features from all frozen columns.

Because old columns are never modified, there is zero forgetting. However, model size grows linearly with the number of tasks, making this approach impractical for long task sequences without pruning or compression.

Scalability concern: After 50 tasks, the model has 50× the original parameters. Progressive nets are best suited for short task sequences (≤ 10) or when combined with pruning to reclaim capacity.

PackNet & Pruning Methods

PackNet (Mallya & Lazebnik, 2018) achieves continual learning within a fixed-capacity network. After training on a task, the least important weights are pruned, and the surviving weights are frozen. The pruned (freed) capacity is then available for the next task.

# PackNet lifecycle for 3 tasks for task_id in [1, 2, 3]: # 1. Train on current task (only free weights are trainable) train(model, task_data[task_id], mask=free_mask) # 2. Prune: keep top-k% of free weights by magnitude importance = abs(model.free_weights) threshold = percentile(importance, prune_ratio) task_mask = importance >= threshold # 3. Freeze surviving weights (assign to this task) frozen_masks[task_id] = task_mask free_mask = free_mask & ~task_mask # remaining capacity # At inference: activate the mask for the desired task model.apply_mask(frozen_masks[target_task])

With typical pruning ratios of 50–75% per task, PackNet can accommodate 3–5 tasks in a single network with minimal interference. The key limitation is finite capacity — eventually the free parameter pool is exhausted.

SupSup (Supermasks in Superposition)

Learns a binary supermask per task over a randomly initialised (untrained) network. At inference, supermasks are superimposed, allowing the network to handle many tasks with just mask storage overhead.

Piggyback

Learns a real-valued mask on top of a frozen pre-trained backbone. Each task adds only a mask (~1 bit per weight), keeping total storage nearly constant as tasks accumulate.

Adapter-Based Continual Learning

Modern parameter-efficient fine-tuning methods — LoRA, adapters, prefix tuning — naturally lend themselves to continual learning. The shared backbone stays frozen, and each task gets its own lightweight module. This is essentially a parameter-efficient version of progressive nets.

from peft import get_peft_model, LoraConfig # Shared frozen backbone base_model = load_pretrained("llama-3-8b") base_model.freeze() # Task-specific LoRA adapters adapters = {} for task in tasks: config = LoraConfig(r=16, lora_alpha=32, target_modules=["q_proj", "v_proj"]) adapters[task.name] = get_peft_model(base_model, config) train(adapters[task.name], task.data) # Inference: load the adapter for the desired task model = base_model.load_adapter(adapters["summarisation"])

Each LoRA adapter adds only 0.1–1% of the base parameters. Storing 100 adapters is cheaper than storing a single full fine-tune. The challenge shifts to task routing: given an input, which adapter should be activated?

Adapter Fusion: O-LoRA merges knowledge across adapters by constraining new LoRA updates to be orthogonal to previous adapter subspaces, enabling a single merged adapter that serves all tasks without explicit routing.

Routing Strategies

Task-ID Routing

If the task identity is known at inference, simply index into the adapter dictionary. Simple but requires metadata at serving time.

Learned Router

Train a small classifier on input embeddings to predict the correct adapter. Works in task-agnostic settings but adds inference latency and can mis-route on ambiguous inputs.

Continual Pre-Training for LLMs

Foundation models face a continual learning challenge at the pre-training level: as new corpora become available (updated web crawls, domain-specific data), the model should absorb new knowledge without forgetting existing capabilities like language understanding, reasoning, and safety alignment.

Real-world example: Naively continuing to pre-train GPT on a medical corpus can degrade its coding ability by 10–30%. Continual pre-training strategies mitigate this regression while efficiently injecting domain knowledge.

Key Techniques

  • Learning-rate rewarm: Restart the cosine schedule from a small learning rate (1/10th of original peak) to gently update weights without large, destructive gradient steps.
  • Replay mixing: Interleave 5–20% of the original pre-training distribution alongside the new domain corpus.
  • Progressive data introduction: Start with data similar to the original distribution and gradually increase the proportion of novel domain data.
  • CL-regularisation: Apply lightweight EWC or L2 penalty toward the original checkpoint weights to anchor general capabilities.
# Continual pre-training recipe for domain adaptation config = { # Learning rate: 1/10 of original peak, cosine decay "lr_peak": 1e-5, "lr_schedule": "cosine_with_warmup", "warmup_steps": 500, # Data mixing: 85% new domain, 15% original distribution "domain_ratio": 0.85, "replay_ratio": 0.15, # Regularisation "weight_decay": 0.01, "ewc_lambda": 100, # light EWC anchor # Training duration "total_tokens": "10B", "batch_size": 512, }

Data Mixing Strategies

The composition of the training batch is often more important than the choice of continual learning algorithm. Thoughtful data mixing can prevent forgetting with minimal overhead.

Fixed Ratio

A constant mix (e.g., 80% new / 20% replay) throughout training. Simple to implement but may under-replay early (lots of old knowledge to protect) or over-replay late (new task nearly learned).

Curriculum Mixing

Start with heavy replay (50/50), then linearly decrease replay as the model stabilises on the new task. Adaptive to training dynamics.

Loss-Proportional Mixing

Sample more from the distribution (old or new) where the model has higher loss. Automatically allocates capacity to struggling tasks.

DoReMi / Data Selection

Learn optimal data mixing weights using a small proxy model. The proxy model's per-domain loss guides the sampling distribution for the main model, achieving near-optimal allocation.

Practical tip: Always validate your mixing strategy on a held-out set from every previous task. A 2% regression on Task 1 is easy to miss if you only track aggregate metrics.

Evaluation Protocols

Evaluating continual learning methods requires metrics beyond simple accuracy. The community uses standardised protocols that capture both forward and backward transfer effects.

Core Metrics

# After training on T tasks, build the accuracy matrix R # R[i][j] = accuracy on task j after training on task i # Average Accuracy after all T tasks A_T = (1/T) · Σ_j R[T][j] # Backward Transfer (forgetting) BWT = (1/(T-1)) · Σ_j=1..T-1 (R[T][j] − R[j][j]) # Negative BWT = forgetting; Positive BWT = backward improvement # Forward Transfer FWT = (1/(T-1)) · Σ_j=2..T (R[j-1][j] − baseline[j]) # Measures how learning prior tasks helps future ones

Standard Benchmarks

  • Split CIFAR-100: CIFAR-100 divided into 10 or 20 sequential tasks of 5–10 classes each. Widely used for vision CL.
  • Split miniImageNet: Similar splitting protocol on a more challenging dataset.
  • Permuted MNIST: Each task applies a fixed random pixel permutation, testing domain shift without class overlap.
  • CL-Bench (NLP): A sequence of NLP tasks (sentiment, NER, QA, summarisation) used to evaluate language model CL.
  • TRACE (LLM): A benchmark for evaluating continual pre-training of LLMs across temporal knowledge shifts.
Beware task-boundary assumptions: Most benchmarks assume clear task boundaries. In production, tasks often overlap or arrive as a continuous stream. Task-free (or boundary-agnostic) continual learning is an active research frontier.

Practical Approaches & Recipes

In production systems, practitioners typically combine multiple strategies rather than relying on a single method. Below is a decision framework and a concrete recipe for LLM continual fine-tuning.

Decision Framework

# Choosing a continual learning strategy if can_store_previous_data: if memory_budget > 5%: → Experience Replay + EWC # best overall else: → Experience Replay (small buffer) # still strong elif privacy_constraints: if can_train_generator: → Generative Replay # synthesise pseudodata else: → EWC / SI (regularisation only) # no data stored elif task_count_small (< 10): → Progressive Nets or Adapter-per-task elif llm_continual_pretraining: → Replay Mixing + LR rewarm + light EWC

Production Recipe: LLM Continual Fine-Tuning

from transformers import Trainer, TrainingArguments from peft import LoraConfig, get_peft_model # Step 1 — Adapter isolation: new LoRA per task base = load_model("llama-3-8b-instruct") lora_cfg = LoraConfig(r=32, lora_alpha=64, target_modules=["q_proj","k_proj","v_proj","o_proj"]) model = get_peft_model(base, lora_cfg) # Step 2 — Data mixing: 80% new task, 20% replay from previous tasks train_data = interleave_datasets( [new_task_ds, replay_buffer.to_dataset()], probabilities=[0.8, 0.2], stopping_strategy="first_exhausted" ) # Step 3 — Conservative LR + cosine decay args = TrainingArguments( learning_rate=2e-5, # 5-10× lower than initial fine-tune lr_scheduler_type="cosine", warmup_ratio=0.05, num_train_epochs=3, per_device_train_batch_size=8, gradient_accumulation_steps=4, ) # Step 4 — Train with EWC-style regularisation callback trainer = Trainer( model=model, args=args, train_dataset=train_data, callbacks=[EWCCallback(fisher=prev_fisher, old_params=prev_params, lam=50)], ) trainer.train() # Step 5 — Update replay buffer with new task examples replay_buffer.add_task_data(new_task_ds, num_samples=1000) # Step 6 — Evaluate on ALL previous tasks for task_name, eval_ds in all_eval_sets.items(): metrics = trainer.evaluate(eval_ds) log_metric(f"eval/{task_name}_accuracy", metrics["accuracy"])
Key takeaway: Continual learning in production is rarely about choosing one method — it's about layering defences: adapter isolation limits which parameters change, replay ensures old knowledge stays in the gradient landscape, and regularisation anchors critical weights. Together these strategies let models evolve continuously without catastrophic regressions.
