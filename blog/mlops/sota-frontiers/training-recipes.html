Training Recipes: Open Source Playbooks

MLOps Series SOTA & Frontiers

The Multi-Stage Training Pipeline

Every major open-source LLM follows a remarkably similar multi-stage training pipeline. While each lab adds proprietary twists, the skeleton is consistent: massive-scale pretraining on raw web text, followed by supervised fine-tuning (SFT) on curated instruction data, then alignment via RLHF or DPO, and finally rigorous evaluation. Understanding this pipeline is critical because each stage has distinct failure modes, costs, and data requirements.

The cost distribution is heavily skewed: pretraining accounts for 80–95% of the total compute budget. Post-training (SFT + RLHF) is comparatively cheap but has an outsized impact on model quality — a well-tuned post-training recipe can make a mediocre pretrained base competitive with a stronger one.

Why open-source recipes matter: Before Llama 2's release, the community had to reverse-engineer training details from sparse papers and API behavior. Meta's decision to publish comprehensive training reports set a new standard. Now, labs like DeepSeek and Qwen routinely share hyperparameters, data mix ratios, and training curves — accelerating the entire field.

Llama 2/3 Training Recipe

Meta's Llama family is the most well-documented open-source LLM series, providing detailed technical reports at each major release. The progression from Llama 1 to Llama 3.1 405B showcases how training recipes evolve at the frontier.

Llama 2 (July 2023)

Architecture: Standard dense Transformer with pre-norm (RMSNorm), SwiGLU activations, rotary positional embeddings (RoPE), and grouped-query attention (GQA) for 34B and 70B variants. The 7B model uses standard multi-head attention.

# Llama 2 architecture configuration (70B)

    "num_kv_heads":     8,        # GQA: 8 KV heads for 64 query heads
    "ffn_dim":          28672,    # SwiGLU: 8/3 × hidden_dim rounded
    "vocab_size":       32000,
    "max_seq_len":      4096,
    "total_params":     "~70B",
}

# Training hyperparameters
llama2_training = {

Data: 2 trillion tokens from a custom mix of publicly available web data. Key details: no Meta user data, aggressive deduplication, upsampled knowledge-heavy sources (Wikipedia, books, scientific papers). The data mix shifted significantly from Llama 1 — more code and reasoning-heavy content.

Llama 3 / 3.1 (April–July 2024)

Llama 3 represented a massive scale-up in data and compute while keeping the architecture relatively simple. Meta's philosophy was clear: scale data, not architecture complexity.

llama3_405b = {

    "num_heads":        128,
    "num_kv_heads":     8,         # GQA ratio of 16:1
    "ffn_dim":          53248,

    "max_seq_len":      131072,    # 128K context (extended in post-training)
    "total_params":     "~405B",
    "rope_theta":       500000,    # Increased from 10000 for longer ctx
}

# Key training decisions in Llama 3
llama3_training = {
    "tokens_trained":    "15.6T tokens",   # 7.5× Llama 2
    "batch_ramp":        "4M → 8M → 16M tokens",  # Progressive
    "peak_lr":           8e-5,
    "min_lr":            8e-7,       # 1/100 of peak (not 1/10)
    "training_gpus":     16384,      # H100-80GB GPUs

}
Llama 3's data bet: Meta trained the 8B model on 15T tokens — far beyond the compute-optimal "Chinchilla" ratio which would suggest ~200B tokens for 8B parameters. This "over-training" strategy produces smaller models that punch above their weight at inference time, because inference cost scales with model size, not training data.

Post-training for Llama 3: Meta's post-training was multi-round and increasingly sophisticated:

  1. Round 1 SFT: Human-written demonstrations of instruction following, safety, and helpfulness. ~27K high-quality examples.
  2. Round 1 RLHF: Reward model trained on human preference data; PPO applied to the SFT checkpoint.
  3. Rejection sampling: Generate many completions per prompt, score with the reward model, and fine-tune on the top-k responses. This bootstraps training data quality for subsequent rounds.
  4. Rounds 2–6: Iterative DPO with fresh preference data from the latest model checkpoint. Each round focuses on identified weaknesses (code, math, safety edge cases).
  5. Tool use training: Dedicated SFT on tool-calling patterns (search, code execution, calculator).

Mistral & Mixtral Details

Mistral AI's approach contrasts sharply with Meta's brute-force scaling. Their models emphasize architectural efficiency — doing more with fewer active parameters — and they release far less training detail, making the available information especially valuable.

Mistral 7B (September 2023)

The model that proved a well-trained 7B could match Llama 2 13B. Key architectural choices:

  • Sliding Window Attention (SWA): Each layer attends only to the last W=4096 tokens (not the full sequence). Because attention is applied across layers, the effective receptive field grows to num_layers × W — providing long-context capability with O(W) memory per layer instead of O(seq_len).
  • GQA with 8 KV heads: Same as Llama 2 70B but in a 7B model — aggressive KV compression for fast inference.
  • Rolling buffer cache: KV-cache wraps around at window size W, capping memory at a fixed bound regardless of sequence length.
# Sliding Window Attention — Mistral's efficiency trick
def sliding_window_attention(Q, K, V, window_size=4096):
    seq_len = Q.shape[1]
    # Build attention mask: each token attends to [max(0, i-W), i]
    mask = torch.zeros(seq_len, seq_len, dtype=torch.bool)
    for i in range(seq_len):
        start = max(0, i - window_size)
        mask[i, start:i+1] = True

    scores = Q @ K.transpose(-2, -1) / math.sqrt(d_k)
    scores.masked_fill_(~mask, float('-inf'))
    attn = softmax(scores, dim=-1)
    return attn @ V

# Effective receptive field across 32 layers:
# Layer 1: 4096 tokens, Layer 2: 8192 tokens, ...
# Layer 32: 32 × 4096 = 131,072 tokens (theoretical)

Mixtral 8×7B (December 2023)

Mixtral brought Mixture-of-Experts to the open-source frontier. The architecture uses 8 expert FFN blocks per layer, routing each token to the top-2 experts — giving 47B total parameters but only ~13B active per token.

Experts: 8 per layer, top-2 routing

Context: 32K (SWA)

Performance: Matches Llama 2 70B on most benchmarks at 3× faster inference

Mixtral 8×22B

Experts: 8 per layer, top-2 routing

Context: 64K

Performance: Competitive with GPT-4-turbo on code and reasoning tasks

Mistral's transparency gap: Unlike Meta and DeepSeek, Mistral has never published a detailed training report. We know they use web-scale data with heavy code and multilingual content, but specific data volumes, mix ratios, and training schedules remain undisclosed. This limits reproducibility but their architectural innovations are well-documented.

DeepSeek-V2/V3 Training Innovations

DeepSeek has produced the most detailed open training reports since Llama, and their engineering innovations — especially around efficient attention and MoE training — have pushed the frontier of what's achievable per dollar. Their V2 and V3 models introduced techniques that other labs are now racing to adopt.

DeepSeek-V2 (May 2024)

Multi-head Latent Attention (MLA): The headline innovation. Instead of caching full key-value pairs per head, MLA compresses the KV representation into a low-rank latent space, reducing KV-cache memory by 93.3% compared to standard MHA.

# Multi-head Latent Attention (MLA) — DeepSeek-V2's core innovation
class MultiHeadLatentAttention:
    def __init__(self, d_model=5120, n_heads=128, d_compress=512):
        # Compress KV into low-rank latent vector
        self.kv_compress = Linear(d_model, d_compress)     # Down-project
        self.k_decompress = Linear(d_compress, n_heads * d_k)  # Up-project K
        self.v_decompress = Linear(d_compress, n_heads * d_v)  # Up-project V
        self.q_proj = Linear(d_model, n_heads * d_k)

    def forward(self, x):
        q = self.q_proj(x)
        # Only cache this compressed latent — 512 dims vs 128 × 128 = 16384
        c_kv = self.kv_compress(x)     # shape: (batch, seq, 512)
        k = self.k_decompress(c_kv)    # expand at attention time
        v = self.v_decompress(c_kv)
        return scaled_dot_product_attention(q, k, v)

# KV cache savings: 512 floats vs 16384 floats per token → 32× reduction
# Combined with RoPE absorbed into latent: 93.3% total cache reduction

DeepSeek-V3 (December 2024)

DeepSeek-V3 scaled to 671B total parameters (37B active) and trained on 14.8T tokens for an estimated $5.576M — a fraction of comparable models' costs. Key innovations:

  • Multi-Token Prediction (MTP): Each position predicts not just the next token but the next 2 tokens. This provides denser gradient signals and improves sample efficiency by ~10%. The auxiliary prediction heads are discarded at inference (or optionally used for speculative decoding).
  • Auxiliary-loss-free load balancing: Traditional MoE uses auxiliary losses to prevent expert collapse. DeepSeek-V3 introduces a bias term per expert that is dynamically adjusted based on recent routing statistics — achieving balanced routing without distorting the main training objective.
  • FP8 mixed-precision training: Pioneered large-scale FP8 training, using FP8 for forward pass matrix multiplications while keeping gradients and optimizer states in higher precision. This nearly doubles training throughput on H800 GPUs.
# DeepSeek-V3 training configuration
deepseek_v3 = {

    "routed_experts":     8,              # Top-8 of 256
    "shared_experts":     1,              # Always active
    "tokens_trained":     "14.8T",
    "peak_lr":            2.2e-4,
    "min_lr":             2.2e-5,
    "batch_size_tokens":  "60M → stepped increases",
    "hardware":           "2048 H800 GPUs",
    "training_time":      "~2 months",
    "estimated_cost":     "$5.576M",        # Remarkably low
    "precision":          "FP8 forward, BF16 backward",

# Multi-Token Prediction head
class MTPHead:
    def __init__(self, d_model, vocab_size, n_future=2):
        self.heads = [Linear(d_model, vocab_size) for _ in range(n_future)]

    def loss(self, hidden_states, target_ids):
        total_loss = 0
        for k, head in enumerate(self.heads):
            # Predict token at position i+k+1 from hidden state at i
            logits = head(hidden_states[:, :-(k+1)])
            targets = target_ids[:, (k+1):]
            total_loss += cross_entropy(logits, targets)
        return total_loss / len(self.heads)
The $5.576M question: DeepSeek-V3's reported training cost is so low that many researchers were skeptical. The key enablers were (1) FP8 training halving compute requirements, (2) fine-grained 256-expert MoE keeping active parameters small, (3) efficient pipeline parallelism with near-zero bubble overhead, and (4) access to H800 GPUs at Chinese domestic pricing. At US cloud prices, the same run would cost roughly $20–30M.

Qwen Series Training

Alibaba's Qwen series has quietly become one of the most capable open-weight model families, especially for multilingual and coding tasks. The Qwen2.5 release (September 2024) included models from 0.5B to 72B, and the training reports reveal a meticulous approach to data curation and multi-stage training.

Qwen's differentiators:

  • Massive vocabulary (152K tokens): Optimized for CJK languages where each character is a token. This reduces sequence lengths by 15–30% for Chinese text compared to Llama's tokenizer, directly lowering inference cost.
  • Synthetic data bootstrap: Qwen2 was used to generate synthetic training data (especially for math and code) that trained Qwen2.5 — a self-improvement loop that Meta also adopted for Llama 3.1.
  • Staged context extension: Pretraining at 4K context, then extending to 32K, then 128K through continued training with modified RoPE (YaRN) and progressively longer sequences.
  • Comprehensive model lineup: 0.5B, 1.5B, 3B, 7B, 14B, 32B, 72B — covering every deployment tier. Smaller models are distilled from larger ones, not trained independently.
Qwen's code advantage: Qwen2.5-Coder-32B matches GPT-4o on HumanEval and MBPP, making it the strongest open-weight coding model. The recipe: (1) 5.5T tokens of code-heavy pretraining, (2) 400K synthetic coding problems generated by Qwen2-72B, (3) iterative DPO on code preference pairs with execution-verified correctness. This formula — strong base + synthetic data + execution feedback — has become the standard recipe for code models.

Data Mix Strategies

The data mix — the ratio of different data sources during pretraining — is arguably the most important (and least reproducible) aspect of any training recipe. Small changes in data proportions can dramatically shift model capabilities.

# Trend: synthetic data went from 0% to 10%+ of the mix

Data mix principles that emerge from published recipes:

  • Code is the universal booster: Every successful model trained in 2024 uses 15–25% code. Code improves not just coding benchmarks but also reasoning, instruction following, and structured output — likely because code requires precise logical chains and syntax adherence.
  • Quality > quantity for knowledge: Wikipedia and books are heavily upsampled (repeated 2–5× during training) despite being <10% of raw data. Their high quality per token makes each repetition valuable.
  • Dynamic mixing during training: Llama 3 and DeepSeek-V3 both adjust data mix ratios during training — increasing code and math proportions in later stages to boost those specific capabilities. This "curriculum learning" approach assumes the model first needs broad language understanding before specializing.
  • Deduplication is critical: Both MinHash and exact-match deduplication remove 30–60% of raw web data. Training on duplicated data causes memorization, reduces generalization, and wastes compute.
The data mix is the real secret sauce. When labs report "we trained on X trillion tokens of web data," the actual performance depends on how they filtered, weighted, and ordered that data. Two models trained on the same token count from Common Crawl can differ by 10+ points on MMLU depending on quality filtering alone. This is the least reproducible part of any recipe.

Learning Rate Schedules

Learning rate scheduling may seem like a minor detail, but it has a surprisingly large impact on final model quality. The community has converged on a few proven schedules, but there is active research on alternatives.

# Learning rate schedules used by major open-source models

import math

def cosine_schedule(step, warmup, total, peak_lr, min_lr):
    # Used by Llama 2, Mistral, most models
    if step < warmup:
        return peak_lr * step / warmup          # Linear warmup
    progress = (step - warmup) / (total - warmup)
    return min_lr + 0.5 * (peak_lr - min_lr) * (1 + math.cos(math.pi * progress))

def warmup_stable_decay(step, warmup, stable_end, total, peak_lr, min_lr):
    # WSD schedule — used by DeepSeek-V3, MiniCPM
    # Warmup → constant LR → rapid cooldown
    if step < warmup:
        return peak_lr * step / warmup
    elif step < stable_end:
        return peak_lr                            # Constant phase
    else:
        decay_progress = (step - stable_end) / (total - stable_end)
        return min_lr + (peak_lr - min_lr) * (1 - decay_progress)

def llama3_schedule(step, warmup, total, peak_lr, min_lr):
    # Llama 3: cosine decay to 1/100 of peak (not 1/10)
    assert min_lr == peak_lr / 100  # Llama 3 decays more aggressively
    return cosine_schedule(step, warmup, total, peak_lr, min_lr)

# Key insight: WSD allows "anytime" checkpointing — you can fork
# the training run at any point during the stable phase and start
# a rapid cooldown to get a usable model. Cosine schedule requires
# committing to the full schedule upfront.

Emerging schedule innovations:

  • WSD (Warmup-Stable-Decay): DeepSeek-V3's choice. The long constant-LR phase means you can branch the run at any point and do a short cooldown to produce a checkpoint — enabling rapid experimentation without restarting from scratch.
  • Progressive batch size ramp: Llama 3 starts with a small batch (4M tokens) and increases to 16M over training. This acts as an implicit LR schedule — effective LR scales with 1/batch_size, so ramping batch size creates a natural decay.
  • Two-phase training: Some recipes (StableLM, Yi) use a high LR phase on diverse data, then a low LR phase on curated high-quality data — similar to pretraining + continued pretraining, but as a single run.
Practitioner tip: If you're fine-tuning an open model (not pretraining from scratch), use a peak LR that is 5–20× lower than the original pretraining LR, with a cosine schedule. For a Llama 3 8B base, this means peak LR around 1e-5 to 5e-5. Too high and you get catastrophic forgetting; too low and fine-tuning has no effect.

Context Length Extension

Modern models ship with 128K+ context windows, but they don't train at that length from the start — it would be prohibitively expensive (attention cost scales quadratically). Instead, context is extended in post-pretraining stages using modified positional encodings and progressively longer training sequences.

# Context extension techniques used by open-source models

class RoPEExtension:
    # RoPE frequency scaling — the foundation of most extensions

    def base_rope(self, position, dim, theta=10000):
        freq = 1.0 / (theta ** (torch.arange(0, dim, 2) / dim))
        return position * freq

    def linear_scaling(self, position, dim, scale=4.0):
        # Simply divide position by scale factor
        # Used in early Code Llama (4K → 16K)
        return self.base_rope(position / scale, dim)

    def ntk_scaling(self, position, dim, theta=10000, alpha=4.0):
        # Increase base theta instead of scaling positions
        # Preserves high-frequency components better
        new_theta = theta * alpha ** (dim / (dim - 2))
        return self.base_rope(position, dim, theta=new_theta)

    def yarn(self, position, dim, scale=4.0, theta=10000):
        # YaRN (Qwen2.5): interpolate between NTK and linear
        # Low frequencies: NTK-aware, High frequencies: linear
        # Best of both worlds — preserves local attention patterns
        freq = self.base_rope(position, dim, theta)
        # Apply frequency-dependent interpolation weights
        return freq  # Simplified — actual YaRN has per-dim blending

# Llama 3.1 approach: increase RoPE theta from 10K to 500K
# and train on long sequences (up to 128K) during post-training
# This is the simplest effective method — just change one number

Multi-stage context extension (Llama 3.1 recipe):

  1. Stage 1 (pretraining): Train at 8K context on 15T+ tokens. This is where 95% of compute goes.
  2. Stage 2 (long-context continued pretraining): ~800B additional tokens with gradually increasing sequence lengths (8K → 16K → 32K → 128K). RoPE theta increased to 500K. This stage is ~5% of total pretraining compute.
  3. Stage 3 (long-context SFT): Fine-tune on a mix of short and long instruction-response pairs. Critical: include short-context data to prevent degrading short-context performance ("context length tax").
The context length tax: Naively extending context length degrades short-context performance by 2–5% on standard benchmarks. The fix is to always include a majority of short-context training data during extension. Llama 3.1 uses approximately 70% short (<8K) and 30% long (8K–128K) sequences during Stage 2. DeepSeek-V3 uses a similar ratio.

Reproducibility & Cost Estimates

The promise of "open-source AI" includes reproducibility — the ability to independently verify and build on published results. In practice, reproducibility of frontier LLM training varies dramatically by lab and by component.

What's Reproducible

  • Architecture: All major models publish full architecture details. You can rebuild Llama 3 or DeepSeek-V3 from the papers.
  • Hyperparameters: LR schedules, batch sizes, optimizer settings — mostly published.
  • Post-training methods: SFT + RLHF/DPO pipeline is well-understood and achievable at modest scale.
  • Evaluation: Standard benchmarks are public and deterministic (mostly).

What's NOT Reproducible

  • Pretraining data: Exact data mix, filtering criteria, and quality classifiers are proprietary even for "open" models.
  • Data ordering: The sequence in which data is presented matters. No lab publishes their data shuffling seeds.
  • Internal tooling: Custom CUDA kernels, distributed training frameworks, and data pipelines are rarely released.
  • Compute budget: Few researchers have access to 16K+ GPUs for months.

The reproducibility gap and what it means for practitioners:

  • Don't pretrain — fine-tune. Unless you have $5M+ and a team of infrastructure engineers, start from an open base model. The post-training stages (SFT + DPO) are where individual practitioners and small teams can make the biggest impact, and they cost 100–1000× less than pretraining.
  • The real moat is data. Architecture innovations (MLA, MoE routing) are quickly replicated. Training infrastructure advantages persist for ~6 months. But proprietary data — curated, filtered, deduped, and mixed in the right proportions — remains the most durable advantage.
  • Open weights ≠ open source. Llama 3.1 and DeepSeek-V3 release model weights but not training data, code, or infrastructure. True open-source efforts like OLMo (AI2) and Pythia (EleutherAI) release everything — but trail the frontier by 6–12 months.
  • Cost is dropping fast. DeepSeek-V3's training cost ($5.6M for a GPT-4-class model) would have been inconceivable in 2023. FP8 training, better parallelism strategies, and cheaper GPUs will continue this trend.
The 2025 playbook for practitioners: Pick a strong open base model (Llama 3.1, Qwen2.5, or DeepSeek-V3), curate domain-specific data, apply SFT + DPO with execution-based verification where possible (especially for code and math), and evaluate rigorously on your target task — not generic benchmarks. This recipe, costing $1K–$50K in compute, routinely produces models that beat generic frontier models on specific domains.
