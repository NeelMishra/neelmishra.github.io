Inference Innovations: Medusa, Lookahead

MLOps Series SOTA & Frontiers

Speculative Decoding Recap

Standard autoregressive decoding generates one token per forward pass through the full model. For a 70B-parameter LLM on a single GPU, each forward pass takes roughly 30–50 ms — meaning generation is fundamentally memory-bandwidth-bound, not compute-bound. The GPU's arithmetic units sit largely idle while waiting for weights to stream from HBM.

Speculative decoding (Leviathan et al., 2022; Chen et al., 2023) addresses this by using a small draft model to propose K candidate tokens, then the large target model verifies all K tokens in a single forward pass. Because verification is parallel (the target model processes K tokens simultaneously via the KV cache), accepted tokens are generated at near-zero marginal cost.

# Classic speculative decoding
def speculative_decode(target, draft, prompt, K=5):
    tokens = prompt
    while not done(tokens):
        # Draft K candidate tokens autoregressively
        candidates = []
        for i in range(K):
            t = draft.sample(tokens + candidates)
            candidates.append(t)

        # Verify all K candidates in one target forward pass
        target_probs = target.forward(tokens + candidates)

        # Accept/reject using modified rejection sampling
        n_accepted = verify_and_accept(target_probs, draft_probs, candidates)
        tokens.extend(candidates[:n_accepted])

        # Sample one more token from corrected distribution
        tokens.append(sample_residual(target_probs[n_accepted]))
    return tokens
Key guarantee: Speculative decoding produces exactly the same output distribution as standard autoregressive sampling from the target model. The speedup comes from parallelism, not approximation. Typical speedups are 2–3× depending on draft-target alignment.

The fundamental limitation of classic speculative decoding is the need for a separate draft model — one that must be loaded into GPU memory, tuned for alignment with the target, and maintained alongside the target model in production. The innovations below eliminate or reimagine this requirement.

Medusa: Multi-Head Prediction

Medusa (Cai et al., 2024) replaces the external draft model with additional prediction heads attached directly to the target model's final hidden states. Each Medusa head predicts a token at a different future position — head 1 predicts position t+1 (same as the original LM head), head 2 predicts t+2, head 3 predicts t+3, and so on.

Each head is a lightweight MLP (typically a single hidden layer with a SiLU activation) operating on the same hidden representation that the original LM head uses. The parameter overhead is minimal — roughly 0.5–2% of total model parameters for 3–5 heads.

# Medusa head architecture
class MedusaHead(nn.Module):
    def __init__(self, hidden_dim, vocab_size):
        self.linear1 = nn.Linear(hidden_dim, hidden_dim)
        self.act = nn.SiLU()
        self.linear2 = nn.Linear(hidden_dim, vocab_size)

    def forward(self, hidden_states):
        # Residual connection through the head
        x = self.act(self.linear1(hidden_states))
        return self.linear2(hidden_states + x)

# During inference with K Medusa heads
def medusa_propose(model, heads, hidden, top_k=10):
    # Original LM head gives t+1 candidates
    p0 = model.lm_head.forward(hidden)
    proposals = [top_k_tokens(p0, top_k)]

    # Each Medusa head gives t+2, t+3, ... candidates
    for head in heads:
        pk = head.forward(hidden)
        proposals.append(top_k_tokens(pk, top_k))

    # Build candidate tree from Cartesian product
    return build_tree_candidates(proposals)
Training cost: Medusa heads require fine-tuning on representative data. Medusa-1 freezes the base model and trains only the heads (~hours on 8×A100s for a 7B model). This avoids catastrophic forgetting but caps head accuracy at ~60–70% per position.

The crucial insight is that Medusa heads produce multiple candidates per position. With top-10 from each of K=4 heads, the Cartesian product yields up to 10⁴ = 10,000 candidate sequences. Obviously verifying all is impractical, so Medusa uses tree attention to verify a pruned tree of the most likely candidates in a single forward pass.

Medusa-2: Tree Attention & Self-Distillation

Medusa-2 introduces two key improvements over the original: tree-structured attention masks for more efficient verification, and self-distillation training that jointly trains the heads with the backbone.

Tree-Based Attention

Instead of verifying a flat sequence of draft tokens, Medusa-2 organizes candidates into a tree where each node represents a token prediction and branches represent alternative choices. A custom tree attention mask ensures each candidate token attends only to its ancestors in the tree, not to sibling branches.

# Tree attention mask construction
def build_tree_mask(tree_indices, seq_len):
    # tree_indices: list of (position, parent_index) pairs
    n_candidates = len(tree_indices)
    mask = zeros(n_candidates, seq_len + n_candidates)

    for i, (pos, parent) in enumerate(tree_indices):
        # Attend to all prior context tokens
        mask[i, :seq_len] = 1
        # Attend to ancestors in the tree
        ancestor = parent
        while ancestor >= 0:
            mask[i, seq_len + ancestor] = 1
            ancestor = tree_indices[ancestor][1]
        mask[i, seq_len + i] = 1  # attend to self
    return mask

# Verified in ONE forward pass using tree attention

Self-Distillation

Medusa-2 jointly trains the base model and Medusa heads, using the base model's own predictions as soft targets for the heads. This self-distillation approach improves head accuracy to 70–80% per position while preserving the base model's capabilities. The loss combines standard next-token cross-entropy with KL divergence between head predictions and the base LM head's distribution.

Medusa-2 speedups: On Vicuna-7B, Medusa-2 achieves 2.3–3.6× speedup over vanilla decoding across various tasks, with no degradation in output quality. The tree attention overhead is ~5% of a standard forward pass.

Lookahead Decoding: Jacobi Iteration

Lookahead Decoding (Fu et al., 2024) takes a completely different approach — it requires no draft model, no additional parameters, and no fine-tuning. Instead, it reformulates autoregressive generation as a nonlinear system of equations and applies Jacobi fixed-point iteration to solve multiple positions simultaneously.

The core idea: in autoregressive decoding, position t+k depends on positions t+1 through t+k-1. Jacobi iteration initializes all future positions with guesses, then iteratively refines each position based on the previous iteration's values for the other positions. If the guess for position t+1 happens to be correct, position t+2 gets a correct input and may also converge — creating a chain reaction.

# Lookahead Decoding with Jacobi iteration
def lookahead_decode(model, prompt, W=5, N=7):
    # W = lookahead window size, N = n-gram pool depth
    tokens = prompt
    ngram_pool = {}  # collected n-grams from past iterations
    window = random_init(W)  # initial guesses for W future tokens

    while not done(tokens):
        # Parallel forward pass: compute logits for all W positions
        logits = model.forward(tokens + window)  # one pass!

        # Jacobi update: each position uses prev iteration's context
        new_window = []
        for i in range(W):
            new_window.append(argmax(logits[len(tokens) + i]))

        # Collect n-grams from the window trajectory
        update_ngram_pool(ngram_pool, window, new_window)

        # Check for verified tokens (converged positions)
        n_verified = count_converged(window, new_window)

        # Also check n-gram pool for matching continuations
        pool_match = lookup_ngram(ngram_pool, new_window[0], N)
        n_accepted = max(n_verified, verify_pool_match(model, pool_match))

        tokens.extend(new_window[:n_accepted])
        window = new_window[n_accepted:] + random_init(n_accepted)
    return tokens

The N-gram Pool

A key innovation in Lookahead Decoding is the n-gram pool — a cache of token sequences observed during previous Jacobi iterations. Even when a Jacobi step doesn't converge, the intermediate trajectories contain useful n-gram patterns. When a newly generated token matches the start of a cached n-gram, the continuation can be used as a speculative draft — and verified in the same forward pass.

No extra parameters: Lookahead Decoding achieves 1.5–2.3× speedup on LLaMA-2-Chat models with zero fine-tuning and zero additional memory for draft model weights. The only overhead is the n-gram pool (typically <10 MB) and the wider attention window.
Limitation: Lookahead Decoding's speedup degrades with higher temperatures and more creative sampling. Jacobi convergence depends on predictable patterns — the more stochastic the generation, the fewer positions converge per step.

EAGLE: Feature-Level Auto-Regression

EAGLE (Li et al., 2024) — Extrapolation Algorithm for Greater Language-model Efficiency — drafts at the feature level rather than the token level. Instead of predicting tokens and feeding them back, EAGLE auto-regressively predicts the next hidden state based on the current hidden state and the sampled token's embedding.

The draft model is a lightweight single-layer transformer that takes as input the concatenation of the current hidden state and the token embedding, and predicts the next hidden state. This predicted hidden state is then passed through the original LM head to get token probabilities.

# EAGLE feature-level drafting
class EAGLEDrafter(nn.Module):
    def __init__(self, hidden_dim):
        self.fc = nn.Linear(hidden_dim * 2, hidden_dim)
        self.transformer_layer = TransformerBlock(hidden_dim)

    def forward(self, hidden_state, token_embedding):
        # Concatenate hidden state + token embedding
        x = concat(hidden_state, token_embedding)
        x = self.fc(x)
        # Predict next hidden state
        return self.transformer_layer(x)

def eagle_speculate(model, drafter, hidden, token, K=6):
    draft_tokens = []
    h = hidden
    for _ in range(K):
        emb = model.embed(token)
        h = drafter.forward(h, emb)       # predict next hidden state
        logits = model.lm_head.forward(h) # reuse original LM head
        token = sample(logits)
        draft_tokens.append(token)
    return draft_tokens

Why Feature-Level Works Better

  • Richer signal: Hidden states contain full contextual information; token IDs are lossy discrete bottlenecks.
  • Better alignment: Draft hidden states are closer to what the target model would produce, leading to higher acceptance rates (~75–80%).
  • LM head reuse: The draft model doesn't need its own output projection — it uses the target's LM head.

EAGLE-2 Improvements

  • Context-aware dynamic tree: Adjusts speculation depth based on draft confidence — wider trees when confident, shallower when uncertain.
  • 3.5–4.0× speedup on MT-bench and code generation tasks with Vicuna-7B and LLaMA-2-Chat-70B.
  • Lossless: Maintains exact output distribution via standard speculative rejection sampling.

Staged Speculative Decoding

Staged Speculative Decoding (Spector & Re, 2023) extends the two-model paradigm into a multi-stage pipeline. Instead of one draft model and one target model, it chains multiple models of increasing size: a tiny model proposes, a medium model refines, and the large model verifies.

# Staged speculative decoding (3-stage example)
def staged_speculate(tiny, medium, large, prompt):
    tokens = prompt
    while not done(tokens):
        # Stage 1: Tiny model (68M) proposes 12 candidates
        drafts_1 = tiny.generate(tokens, n=12)

        # Stage 2: Medium model (1.3B) filters to 6 candidates
        probs_2 = medium.forward(tokens + drafts_1)
        drafts_2, n_kept = verify_and_filter(probs_2, drafts_1, keep=6)

        # Stage 3: Large model (70B) final verification
        probs_3 = large.forward(tokens + drafts_2)
        accepted = verify_final(probs_3, drafts_2)

        tokens.extend(accepted)
    return tokens

The staged approach is especially effective in heterogeneous hardware environments. The tiny model can run on CPU while the medium model runs on a secondary GPU, and the large model occupies the primary GPU. Each stage filters out low-probability candidates, so the large model only verifies high-quality drafts.

Latency stacking: In a well-tuned 3-stage pipeline, the tiny model adds ~2 ms, the medium model ~8 ms, and verification on the large model costs the same ~40 ms as a single-token forward pass. Total: ~50 ms for 4–6 accepted tokens vs 40 ms × 5 = 200 ms for standard decoding.

Method Comparison

Each inference innovation makes different trade-offs between speedup, memory overhead, integration complexity, and output quality. The following comparison synthesizes reported results from the original papers, normalized to similar hardware (A100 GPUs) and model sizes (7B and 70B targets).

Classic Speculative Decoding

Speedup: 2.0–2.5× (7B target), 2.5–3.0× (70B target)

Memory overhead: Full draft model (~1–7B parameters)

Quality: Lossless (exact distribution match)

Integration: Need aligned draft model; moderate complexity

Medusa / Medusa-2

Speedup: 2.3–3.6× (depends on task predictability)

Memory overhead: 0.5–2% additional parameters

Quality: Medusa-1 typical sampling only; Medusa-2 lossless

Integration: Requires head training; tree attention kernel

Lookahead Decoding

Speedup: 1.5–2.3× (degrades with high temperature)

Memory overhead: N-gram pool only (~few MB)

Quality: Lossless for greedy; approximate for sampling

Integration: No training needed; drop-in compatible

EAGLE / EAGLE-2

Speedup: 2.8–4.0× (highest reported among lossless methods)

Memory overhead: ~1 transformer layer (~0.5–1% of model)

Quality: Lossless (exact distribution match)

Integration: Requires drafter training on hidden states

Staged Speculative

Speedup: 2.5–3.5× (with 3-stage pipeline)

Memory overhead: Multiple draft models

Quality: Lossless (standard rejection sampling)

Integration: Complex pipeline orchestration; heterogeneous hardware

Key Insight

The best method depends on your constraints: EAGLE leads on raw speedup, Lookahead wins on zero setup cost, Medusa offers a middle ground of fast integration with good speedup, and Staged Speculative shines on multi-device setups.

Production Deployment Considerations

Moving these innovations from research benchmarks to production serving introduces challenges that papers rarely address. Here are the critical considerations:

Batch Compatibility

Speculative methods generate variable numbers of tokens per step. In continuous batching systems (vLLM, TensorRT-LLM), this creates sequence-length mismatches across requests in the same batch. Solutions include padding to max speculation depth, independent speculation per request, or synchronizing speculation lengths across the batch.

# Batched speculative decoding challenge
def batched_speculative_step(model, batch_requests):
    # Problem: each request may accept different # of tokens
    # Request A: proposed 5, accepted 4
    # Request B: proposed 5, accepted 1
    # → KV cache lengths now diverge within the batch

    # Solution 1: Pad to max speculation length
    max_K = max(r.speculation_depth for r in batch_requests)
    pad_all_to(batch_requests, max_K)  # wastes some compute

    # Solution 2: Independent speculation (vLLM approach)
    # Run draft + verify per-request, batch only verification
    drafts = [draft_for(r) for r in batch_requests]
    verified = model.batched_verify(drafts)  # single GPU kernel
    return verified

KV Cache Management

Speculative methods require careful KV cache management. Rejected draft tokens must have their KV cache entries rolled back. With tree-structured speculation (Medusa, EAGLE-2), the cache must support non-contiguous rollback — keeping accepted branches while discarding rejected ones.

Memory planning: Tree speculation can temporarily inflate KV cache usage by 3–10× during the verification step (64 candidates × sequence length). Ensure your GPU memory budget accounts for peak speculation overhead, not just steady-state cache size.

Choosing a Method for Production

Decision framework:
  • Cannot retrain? → Lookahead Decoding (zero training, drop-in)
  • Have aligned draft model? → Classic speculative or Staged
  • Can train heads (~hours)? → Medusa-2 for best effort/speedup ratio
  • Want maximum speedup, can invest in training? → EAGLE-2
  • Multi-GPU heterogeneous setup? → Staged Speculative Decoding

Framework Support

As of 2024, the major inference frameworks have varying levels of support:

  • vLLM: Native speculative decoding with draft models; Medusa support via community extensions; EAGLE integration in progress.
  • TensorRT-LLM: Speculative decoding built in; Medusa and EAGLE via custom plugins; best raw performance on NVIDIA hardware.
  • Hugging Face TGI: Medusa support integrated; speculative decoding with any compatible draft model.
  • SGLang: EAGLE-2 integrated natively; Radix attention pairs well with tree speculation for KV cache reuse.

Inference innovation is one of the highest-leverage areas in LLM deployment. A well-implemented speculative decoding scheme can deliver 2–4× speedup at zero quality cost, effectively cutting your serving bill in half. As models grow larger and serving costs dominate ML budgets, these techniques transition from "nice to have" to essential infrastructure.
