Model Merging: SLERP, TIES, DARE

MLOps Series SOTA & Frontiers

Why Merge Models?

Fine-tuning a foundation model produces a specialist. But what if you want one model that excels at coding and creative writing and math reasoning? Training on all tasks jointly is expensive and often leads to catastrophic forgetting. Model merging sidesteps this by combining the weight tensors of independently fine-tuned checkpoints into a single model — no additional training required.

Key Insight: Merging works because independently fine-tuned models from the same base share a similar loss landscape. The fine-tuned weights live in overlapping basins of attraction, so interpolating between them often lands in a region that is good for both tasks.

The open-source community has embraced merging aggressively — many of the top entries on the Open LLM Leaderboard are "Frankenmerges" rather than direct fine-tunes. Let's understand the algorithms that power this.

LERP & SLERP

Linear Interpolation (LERP)

The simplest merge: take a weighted average of every corresponding parameter. For two models θA and θB with blending factor α ∈ [0, 1]: 

# Linear interpolation (LERP)
def lerp(theta_a, theta_b, alpha):
    return (1 - alpha) * theta_a + alpha * theta_b

LERP is fast and intuitive — α = 0.5 means "equal blend." However, it assumes the interpolation path through weight space is straight, which ignores the geometry of the parameter manifold.

Spherical Linear Interpolation (SLERP)

SLERP treats weight vectors as points on a hypersphere and interpolates along the great-circle arc rather than the chord. This preserves vector magnitude and angular relationships. 

# Spherical Linear Interpolation (SLERP)
import numpy as np

def slerp(v0, v1, t):
    # Normalize input vectors
    v0_n = v0 / np.linalg.norm(v0)
    v1_n = v1 / np.linalg.norm(v1)

    # Angle between the two vectors
    omega = np.arccos(np.clip(np.dot(v0_n, v1_n), -1.0, 1.0))
    sin_omega = np.sin(omega)

    # Degenerate case — vectors nearly parallel
    if sin_omega < 1e-6:
        return lerp(v0, v1, t)

    # Spherical interpolation along the great-circle arc
    s0 = np.sin((1.0 - t) * omega) / sin_omega
    s1 = np.sin(t * omega) / sin_omega
    return s0 * v0 + s1 * v1

LERP

  • Element-wise weighted average
  • Can shrink vector norms at α ≈ 0.5
  • O(n) — trivially fast
  • Prone to "representation collapse" on deep layers

SLERP

  • Interpolates on the hypersphere
  • Preserves magnitude / angular structure
  • Slightly more expensive (norm + arccos)
  • Default choice for 2-model merges in mergekit
Limitation: Both LERP and SLERP only merge two models at a time. For N > 2 models you must either chain pairwise merges (order-dependent!) or use algorithms designed for multi-model merging like TIES or DARE.

Task Arithmetic

Ilharco et al. (2023) introduced task vectors: the difference between a fine-tuned model and the base model. 

# Task vector = fine-tuned weights − base weights
tau_A = theta_A - theta_base   # "code skill" direction
tau_B = theta_B - theta_base   # "writing skill" direction

# Merge: add scaled task vectors back to base
theta_merged = theta_base + lambda_A * tau_A + lambda_B * tau_B

This is elegant: each task vector encodes what the fine-tuning changed, and you simply add those changes together. You can even negate a task vector to remove a capability (e.g., reducing toxicity). Task arithmetic is the conceptual backbone for both TIES and DARE.

Analogy: Think of task vectors like Git diffs. The base model is the main branch. Each fine-tune is a feature branch. Merging adds the diffs back to main. Conflicts arise when two diffs modify the same parameter in opposite directions.

TIES-Merging

TIES (Trim, Elect Sign, Merge) by Yadav et al. (2023) addresses the key problem of task arithmetic: interference between task vectors whose parameters disagree in sign.

The Three Steps

← click a step
TIES pipeline: Trim low-magnitude deltas → Elect a consensus sign per parameter → Merge the surviving entries via averaging. 
def ties_merge(base, task_vectors, top_k=0.2, lam=1.0):
    # Step 1 — Trim: keep only top-k% magnitude entries
    trimmed = []
    for tv in task_vectors:
        threshold = np.percentile(np.abs(tv), (1 - top_k) * 100)
        mask = np.abs(tv) >= threshold
        trimmed.append(tv * mask)

    # Step 2 — Elect Sign: majority vote per parameter
    sign_sum = sum(np.sign(t) for t in trimmed)
    elected_sign = np.sign(sign_sum)

    # Zero out entries that disagree with elected sign
    aligned = []
    for t in trimmed:
        agree = (np.sign(t) == elected_sign)
        aligned.append(t * agree)

    # Step 3 — Disjoint Merge: average surviving entries
    counts = sum((a != 0).astype(float) for a in aligned)
    counts = np.maximum(counts, 1)
    merged_tv = sum(aligned) / counts

    return base + lam * merged_tv
Why TIES works: By trimming noise and resolving sign conflicts before averaging, TIES keeps the meaningful deltas from each model while preventing destructive interference. On standard benchmarks it consistently outperforms naive task arithmetic.

DARE

DARE (Drop And REscale) by Yu et al. (2024) takes an entirely different approach to interference: rather than trimming by magnitude, it randomly drops a fraction of task-vector entries and rescales the survivors to preserve expected magnitude. 

def dare(base, task_vectors, drop_rate=0.9, lam=1.0):
    # For each task vector, randomly drop entries
    sparsified = []
    for tv in task_vectors:
        mask = np.random.binomial(1, 1 - drop_rate, tv.shape)
        # Rescale surviving entries to maintain expectation
        sparse_tv = (tv * mask) / (1 - drop_rate)
        sparsified.append(sparse_tv)

    # Average the sparsified task vectors
    merged_tv = sum(sparsified) / len(sparsified)

    return base + lam * merged_tv

The intuition mirrors dropout in training: by randomly removing parameters, DARE reduces the chance of conflicting entries colliding. The rescale factor 1 / (1 - p) ensures the merged model's parameter norms stay close to the individual fine-tunes.

TIES

  • Deterministic trimming (top-k magnitude)
  • Explicit sign election — resolves conflicts
  • Better when tasks share many parameter directions
  • Hyperparams: top_k, λ

DARE

  • Stochastic dropout of task-vector entries
  • Rescale to preserve expected magnitude
  • Better when tasks are largely orthogonal
  • Hyperparams: drop_rate, λ
DARE + TIES hybrid: In practice, many practitioners combine both — first apply DARE dropout, then run TIES sign election on the surviving entries. Mergekit supports this as the dare_ties method.

Mergekit in Practice

mergekit is the de-facto open-source toolkit for model merging. It supports all major algorithms and runs efficiently on CPU (no GPU required for merging). 

# Install mergekit
pip install -U mergekit

# Example: SLERP merge of two Mistral fine-tunes
mergekit-yaml-config slerp_config.yaml:

merge_method: slerp
slices:
  - sources:
      - model: mistral-code-7b
        layer_range: [0, 32]
      - model: mistral-writing-7b
        layer_range: [0, 32]
base_model: mistralai/Mistral-7B-v0.1
parameters:
  t:
    - filter: self_attn
      value: 0.5
    - filter: mlp
      value: 0.3
    - value: 0.4            # default for other layers
dtype: bfloat16
# Example: TIES merge of three models
merge_method: ties
slices:
  - sources:
      - model: model-code
        layer_range: [0, 32]
      - model: model-math
        layer_range: [0, 32]
      - model: model-chat
        layer_range: [0, 32]
base_model: meta-llama/Llama-2-7b-hf
parameters:
  weight: 1.0
  density: 0.5              # top-k = 50%
dtype: float16
# Run the merge
mergekit-yaml slerp_config.yaml ./merged-output \
  --copy-tokenizer --lazy-unpickle --low-cpu-memory
Per-layer blending: Mergekit lets you set different interpolation weights per layer type (attention vs MLP vs embeddings). A common recipe is to lean more toward the "chat" model in attention layers and the "task" model in MLP layers.

When Merging Works vs Fails

✅ Merging Works When

  • Same base model — all checkpoints must share the same pre-trained initialization
  • Complementary tasks — code + chat + math modify mostly different parameters
  • Similar fine-tuning recipes — same LoRA rank, learning rate, and epoch count
  • Moderate fine-tuning — short fine-tunes stay near the base loss basin
  • Careful α / λ tuning — even a small grid search over blending factors helps

❌ Merging Fails When

  • Different base models — merging Llama with Mistral produces garbage
  • Conflicting objectives — safety-tuned + uncensored model = unpredictable
  • Heavy fine-tuning — many epochs push checkpoints far from the base basin
  • Tokenizer mismatch — different vocabularies make embedding merging meaningless
  • Architecture mismatch — different hidden dims or layer counts cannot be merged
The Frankenmodel Trap: Just because a merged model posts high scores on Open LLM Leaderboard benchmarks doesn't mean it's useful. Merges often overfit to multiple-choice evaluation formats while degrading in open-ended generation. Always test on your downstream tasks, not just public benchmarks.

The "mode connectivity" hypothesis explains the success condition: if two checkpoints are connected by a low-loss path in weight space, any point along that path is a good model. Fine-tunes from the same base typically satisfy this — independent pre-training runs do not.

Evaluation & Leaderboards

After merging, you need to verify the result hasn't regressed. Here's a practical evaluation pipeline: 

# Step 1 — Quick sanity: perplexity on a held-out set
lm_eval --model hf --model_args pretrained=./merged-output \
  --tasks wikitext --batch_size 8

# Step 2 — Multi-benchmark sweep (Open LLM Leaderboard style)
lm_eval --model hf --model_args pretrained=./merged-output \
  --tasks arc_challenge,hellaswag,mmlu,truthfulqa_mc2,winogrande,gsm8k \
  --batch_size auto --num_fewshot 5

# Step 3 — Task-specific eval (your actual use case)
python eval_my_task.py --model ./merged-output --dataset ./test_set.jsonl
Open LLM Leaderboard merges: As of 2024, merged models routinely appear in the top-20 of Hugging Face's Open LLM Leaderboard. Many use SLERP or DARE-TIES with 3-5 source models. The MergeKit community tracks successful recipes in shared YAML configs on Hugging Face Hub.

A robust evaluation checklist:

  • Perplexity check — should not increase > 5% vs the best individual model
  • Task-specific accuracy — test each original task (code, math, chat) individually
  • Generation quality — manual spot-check of open-ended outputs for coherence
  • Safety scan — verify no unintended behaviors emerged from conflicting training data
  • Latency benchmark — merged model should have identical inference cost as any single source

# Not interpolation — literally stacks layers from different models
# Creates a LARGER model (e.g., 2×7B = 14B equivalent)
merge_method: passthrough
slices:
  - sources:
      - model: model-a

Pro tip: Start with SLERP for two models, DARE-TIES for three or more. Use density between 0.3–0.7 for DARE. Always test α / t values at 0.3, 0.5, and 0.7 before going to a finer grid.

Interview Prep

Q: What is model merging and why does it work?
Model merging combines the weights of multiple fine-tuned models into a single model without additional training. It works because models fine-tuned from the same base share a similar loss landscape — their weight spaces are connected by low-loss paths (mode connectivity), so interpolating between them yields a model that performs well on all tasks.
Q: Explain the difference between SLERP and LERP for model merging.
LERP (linear interpolation) computes a straight-line weighted average: θ = (1-α)·θ_A + α·θ_B. SLERP (spherical interpolation) treats weight vectors as points on a hypersphere and interpolates along the great-circle arc. SLERP preserves vector magnitudes and angular relationships, making it more stable for deep-layer weights where LERP can cause norm collapse.
Q: How does TIES-Merging handle parameter conflicts?
TIES uses three steps: (1) Trim — zero out low-magnitude entries to remove noise; (2) Elect Sign — for each parameter, take a majority vote on the sign across task vectors, zeroing entries that disagree; (3) Merge — average the surviving sign-consistent entries. This prevents destructive interference from conflicting gradients.
Q: What is DARE and how does it differ from TIES?
DARE (Drop And REscale) randomly drops a fraction of task-vector entries (like dropout) then rescales survivors by 1/(1-p) to preserve expected magnitude. Unlike TIES's deterministic magnitude-based trimming, DARE uses stochastic sparsification. DARE works better when task vectors are largely orthogonal; TIES excels when tasks share parameter directions.
Q: What is task arithmetic? Why might you negate a task vector?
A task vector is τ = θ_finetuned − θ_base. Task arithmetic merges by adding scaled task vectors back to the base: θ_merged = θ_base + Σ λ_i·τ_i. Negating a task vector (subtracting it) removes that capability — e.g., subtracting a "toxicity" task vector can reduce harmful outputs without explicit safety training.
Q: When does model merging fail?
Merging fails when: (1) source models have different base architectures or tokenizers; (2) fine-tuning was too aggressive, pushing models far from the shared loss basin; (3) tasks have conflicting objectives (safety vs uncensored); (4) evaluation only checks benchmarks without testing real downstream quality — the "Frankenmodel trap" of leaderboard gaming.
