MLOps Series SOTA & Frontiers

Chain-of-Thought: Prompting vs Training

Chain-of-thought (CoT) reasoning appeared first as a prompting technique (Wei et al., 2022): prepend "Let's think step by step" and the model produces intermediate reasoning steps that dramatically improve accuracy on math and logic problems. But prompting only elicits latent capabilities — it doesn't create new ones.

The paradigm shift in 2024 was moving CoT from inference-time prompting to training-time optimization. Instead of hoping the model reasons well, you train it to produce high-quality reasoning traces using reinforcement learning. The model learns when to think harder, how to decompose problems, and when to backtrack.

CoT Prompting

How: Few-shot examples with reasoning steps, or "think step by step" zero-shot.

Pros: Zero additional training cost. Works with any instruction-tuned model.

Cons: Quality ceiling bounded by base model capabilities. No learned error correction. Inconsistent reasoning depth.

Accuracy lift: +10–25% on GSM8K for GPT-4-class models.

CoT Training (RL-based)

How: RL with reward models scoring intermediate reasoning steps, not just final answers.

Pros: Model learns to self-correct, backtrack, verify. Reasoning quality improves with more thinking tokens.

Cons: Expensive training. Requires high-quality reasoning traces or reward models. Harder to control inference cost.

Accuracy lift: +30–50% on MATH, +20–40% on GPQA over non-reasoning models.

# CoT prompting vs trained reasoning — behavior difference

# Prompting CoT: model follows template
prompt = """Solve: What is 23 × 47?
Let's think step by step:
1) 23 × 40 = 920
2) 23 × 7 = 161
3) 920 + 161 = 1081"""

# Trained CoT (o1-style): model generates internal reasoning
# <thinking>
#   I need to find 23 × 47. Let me break this down.
#   23 × 47 = 23 × (50 - 3) = 1150 - 69 = 1081
#   Wait, let me verify: 23 × 50 = 1150, 23 × 3 = 69
#   1150 - 69 = 1081 ✓
# </thinking>
# The answer is 1081.
Key insight: Trained reasoning models don't just produce longer outputs — they produce adaptive outputs. Easy questions get short thinking traces (2–10 tokens); hard problems trigger extended reasoning chains (1000+ tokens) with verification loops. This adaptive compute is the core innovation.

OpenAI o1 & o1-pro Architecture

OpenAI's o1 (September 2024) was the first widely deployed reasoning model. While the full architecture is proprietary, key insights can be inferred from public benchmarks, API behavior, and research context.

Training pipeline (inferred): o1 starts from a GPT-4-class base model, then undergoes large-scale reinforcement learning where the reward signal scores reasoning chains, not just final answers. The RL objective encourages the model to produce extended <thinking> blocks that decompose, verify, and backtrack before emitting a final answer.

# Simplified o1-style training loop
def train_reasoning_model(base_model, reward_model, problems):
    policy = copy(base_model)
    for epoch in range(num_epochs):
        for problem in problems:
            # Sample multiple reasoning traces
            traces = [policy.generate(problem) for _ in range(K)]

            # Score each trace with process reward model
            rewards = [reward_model.score_trace(t) for t in traces]

            # PPO/GRPO update — reinforce good reasoning paths
            loss = compute_rl_loss(traces, rewards, policy, base_model)
            policy.update(loss)
    return policy

Key architectural features:

  • Hidden thinking tokens: o1 generates a <thinking> block that is processed but hidden from the user. The API returns only the final answer plus a summary of reasoning. You pay for thinking tokens but don't see them directly.
  • Adaptive compute: Simple questions consume ~100 thinking tokens; competition math problems can use 10,000+. The model learned to allocate thinking proportional to difficulty.
  • Self-verification loops: The thinking trace often includes explicit "let me check" steps where the model re-derives or cross-validates its intermediate results.
  • Backtracking: Unlike standard autoregressive generation, the reasoning trace may include "wait, that's wrong" corrections, suggesting the RL training rewarded models that catch and fix their own errors.
o1-pro vs o1: o1-pro uses more test-time compute per query (longer thinking, possibly multiple internal samples with majority voting). On MATH, o1 scores ~94% while o1-pro reaches ~96%. The marginal improvement costs roughly 3–5× more in thinking tokens — a direct illustration of diminishing returns on test-time compute scaling.

DeepSeek-R1: Open Reasoning

DeepSeek-R1 (January 2025) demonstrated that reasoning capabilities comparable to o1 can be achieved with open weights and transparent methodology. The key contribution: pure RL training on a base model produces emergent reasoning — no supervised fine-tuning on curated reasoning traces is required.

DeepSeek-R1-Zero: The initial experiment applied Group Relative Policy Optimization (GRPO) directly to DeepSeek-V3-Base without any SFT. The model spontaneously learned to produce long reasoning chains, self-verify, and even allocate more thinking to harder problems — all emergent from RL rewards on correctness alone.

# GRPO — Group Relative Policy Optimization (DeepSeek-R1)
def grpo_step(policy, problem, group_size=64):
    # Sample a GROUP of responses for the same problem
    responses = [policy.generate(problem) for _ in range(group_size)]

    # Score each response (binary: correct/incorrect)
    rewards = [check_answer(r, problem.answer) for r in responses]

    # Normalize rewards WITHIN the group (no value network needed)
    mean_r = mean(rewards)
    std_r = std(rewards)
    advantages = [(r - mean_r) / (std_r + 1e-8) for r in rewards]

    # Policy gradient with group-relative advantages
    loss = -sum(adv * log_prob(resp, policy)
               for adv, resp in zip(advantages, responses))
    return loss
Why GRPO matters: Traditional PPO requires training a separate value/critic network that's often as large as the policy — doubling GPU memory. GRPO eliminates the critic by normalizing rewards within each sample group. This made it feasible to RL-train a 671B-parameter MoE model.

DeepSeek-R1 (full): The production model adds a small amount of SFT on curated long-CoT examples before RL to improve formatting, readability, and language mixing. The pipeline:

  1. Cold start SFT: ~thousands of curated long-CoT examples to teach output format
  2. RL with GRPO: Large-scale RL on math, code, science, and logic problems
  3. Rejection sampling SFT: Use the RL-trained model to generate solutions, keep correct ones, retrain
  4. Final RL round: Another GRPO pass for helpfulness and safety alignment

Process vs Outcome Reward Models

The reward signal is the most critical component in training reasoning models. Two paradigms dominate: Outcome Reward Models (ORM) that score only the final answer, and Process Reward Models (PRM) that score each intermediate reasoning step.

Outcome Reward Models (ORM)

Signal: Binary — was the final answer correct?

Pros: Cheap labels (just check the answer). Scales to any domain with verifiable answers.

Cons: Sparse signal — a long chain with 20 steps gets one bit of reward. Vulnerable to reward hacking (right answer, wrong reasoning).

Used by: DeepSeek-R1-Zero, many code generation systems.

Process Reward Models (PRM)

Signal: Per-step score — is each reasoning step valid?

Pros: Dense signal enables faster learning. Catches errors early in the chain. More robust reasoning.

Cons: Expensive annotation — human labelers must judge each step. Hard to scale beyond math.

Used by: OpenAI's "Let's Verify Step by Step" (PRM800K), likely o1.

# ORM vs PRM scoring comparison
def orm_score(trace, ground_truth):
    final_answer = extract_answer(trace)
    return 1.0 if final_answer == ground_truth else 0.0

def prm_score(trace, prm_model):
    steps = split_into_steps(trace)
    step_scores = []
    for i, step in enumerate(steps):
        context = steps[:i+1]
        score = prm_model.score_step(context)  # P(step is correct | context)
        step_scores.append(score)
    # Overall score = product of step scores (any bad step tanks it)
    return prod(step_scores)

# Best-of-N selection with PRM (used at inference)
def best_of_n_prm(model, problem, prm, n=64):
    candidates = [model.generate(problem) for _ in range(n)]
    scores = [prm_score(c, prm) for c in candidates]
    return candidates[argmax(scores)]
PRM800K result: OpenAI's "Let's Verify Step by Step" paper showed that best-of-N selection with PRM scoring solved 78.2% of MATH problems, compared to 72.4% with ORM scoring — using the same base model and the same N=100. The gap widens as N increases because PRM better distinguishes correct reasoning from lucky guesses.

MCTS for Reasoning

Monte Carlo Tree Search (MCTS), the algorithm behind AlphaGo, can be adapted for LLM reasoning. Instead of searching over board positions, you search over reasoning steps. Each node is a partial reasoning chain; children represent possible next steps; and a value model (PRM) estimates the probability of reaching a correct answer from each state.

# MCTS for LLM reasoning (simplified)
class ReasoningNode:
    def __init__(self, state, parent=None):
        self.state = state        # partial reasoning chain
        self.parent = parent
        self.children = []
        self.visits = 0
        self.value = 0.0          # estimated P(correct | state)

def mcts_reasoning(problem, llm, prm, simulations=800):
    root = ReasoningNode(state=problem)

    for _ in range(simulations):
        node = select(root)          # UCB1 to balance explore/exploit
        child = expand(node, llm)     # LLM generates next reasoning step
        value = evaluate(child, prm)  # PRM scores partial chain
        backpropagate(child, value)    # update ancestor statistics

    # Return highest-value complete chain
    return best_chain(root)

def select(node):
    while node.children:
        node = max(node.children,
            key=lambda c: c.value/c.visits + 1.4*sqrt(log(node.visits)/c.visits))
    return node
Practical MCTS cost: With 800 simulations and an average of 5 steps per chain, MCTS requires ~4000 LLM forward passes per problem. At $15/M tokens for o1-class models, solving one hard math problem could cost $0.50–$2.00. This is why MCTS reasoning is reserved for high-value problems (competitive math, formal verification, code generation).

Test-Time Compute Scaling Laws

Traditional scaling laws (Chinchilla, Kaplan) focus on training-time compute: more FLOPs during training → better models. Reasoning models introduce a second scaling axis: test-time compute. The key finding (Snell et al., 2024): for many tasks, spending 10× more compute at inference is more efficient than spending 10× more compute at training.

# Test-time compute scaling strategies
def scale_test_time_compute(model, problem, budget):
    if budget == "low":
        # Single pass — standard inference
        return model.generate(problem)

    elif budget == "medium":
        # Best-of-N with self-consistency
        answers = [model.generate(problem) for _ in range(32)]
        return majority_vote(answers)

    elif budget == "high":
        # Extended thinking + verification
        return model.generate(problem, max_thinking_tokens=50000)

    elif budget == "maximum":
        # MCTS + PRM-guided search
        return mcts_reasoning(problem, model, prm, simulations=1600)

The compute-optimal frontier: For a fixed total budget (training + inference), there exists an optimal split. Small models with heavy test-time compute can match large models with standard inference:

  • A 7B model with 256× test-time compute can match a 70B model with 1× compute on MATH benchmarks
  • A 14B reasoning model (DeepSeek-R1-Distill-Qwen-14B) outperforms GPT-4o on math and code tasks
  • The crossover point depends on query volume: high-volume APIs favor larger models; low-volume high-value tasks favor smaller models with more thinking
Scaling law formula (simplified): If training-time performance scales as L ∝ C_train^(-α) with α ≈ 0.05, test-time compute scales as L ∝ C_test^(-β) with β ≈ 0.1–0.3 for reasoning tasks. The higher exponent means test-time compute is more efficient — but only on tasks that benefit from extended reasoning (math, code, logic), not on knowledge retrieval or creative writing.

Verification & Self-Consistency

A critical advantage of reasoning models is their ability to verify their own outputs. This takes several forms, each with different cost-accuracy tradeoffs:

Self-consistency (Wang et al., 2022): Sample N independent reasoning chains, extract the final answer from each, and return the majority vote. This is the simplest verification strategy and provides surprisingly strong improvements:

# Self-consistency with majority voting
def self_consistency(model, problem, n_samples=40):
    answers = []
    for _ in range(n_samples):
        trace = model.generate(problem, temperature=0.7)
        answer = extract_final_answer(trace)
        answers.append(answer)

    # Majority vote — most common answer wins
    counter = Counter(answers)
    best_answer = counter.most_common(1)[0][0]
    confidence = counter[best_answer] / n_samples

    return best_answer, confidence

# Weighted self-consistency (using PRM scores)
def weighted_consistency(model, problem, prm, n_samples=40):
    traces = [model.generate(problem, temperature=0.7)
              for _ in range(n_samples)]
    scores = [prm_score(t, prm) for t in traces]

    # Group by answer, sum PRM scores per group
    answer_scores = defaultdict(float)
    for trace, score in zip(traces, scores):
        answer = extract_final_answer(trace)
        answer_scores[answer] += score

    return max(answer_scores, key=answer_scores.get)
Verification asymmetry: For many problems, verifying an answer is cheaper than generating it. Checking a proof takes less effort than discovering one. Reasoning models exploit this by generating multiple candidate solutions and then using verification (PRM, code execution, symbolic checks) to select the best one. This is why test-time compute scaling works: more samples → more chances for a correct solution + better selection.

Reasoning Benchmarks

Reasoning models are evaluated on benchmarks specifically designed to require multi-step logical, mathematical, or scientific reasoning — tasks where simple pattern matching fails.

MATH (Hendrycks et al.)

12,500 competition math problems across 7 subjects (algebra, number theory, geometry, etc.) at 5 difficulty levels. Gold standard for mathematical reasoning.

Scores: GPT-4 ~52%, o1 ~94%, DeepSeek-R1 ~97% (MATH-500 subset)

GPQA (Rein et al.)

Graduate-level science questions written by domain experts. Questions are designed so non-experts (even with internet access) score ~34%; domain PhDs score ~65%.

Scores: GPT-4 ~38%, o1 ~78%, DeepSeek-R1 ~71%

ARC-AGI (Chollet)

Abstraction and reasoning tasks requiring novel pattern recognition. Each task has unique rules that must be inferred from 2–3 examples. Designed to test genuine generalization.

Scores: GPT-4o ~5%, o1 ~25%, o3 (high) ~88%. Humans ~85%. Extremely challenging for current models.

AIME (AMC)

American Invitational Mathematics Examination — 15 problems from the USA math olympiad pipeline. Integer answers 0–999. Tests deep mathematical problem-solving.

Scores: GPT-4 ~12%, o1 ~74%, DeepSeek-R1 ~80% (2024 problems)

Benchmark saturation warning: MATH and GSM8K are approaching ceiling performance. When top models score 97%+, the benchmark no longer discriminates between systems. The field is rapidly moving to harder benchmarks: FrontierMath (unsolved problems), EpochAI's newly proposed benchmarks, and ARC-AGI-2. Always check when a benchmark was designed relative to the models being tested.

Cost & Deployment Implications

Reasoning models fundamentally change the economics of LLM deployment. The cost model shifts from fixed per-query to variable per-query, with thinking tokens creating unpredictable bills.

# Cost model comparison
def estimate_cost(model_type, query_difficulty):
    if model_type == "standard":
        # GPT-4o: ~$5/M input, ~$15/M output
        input_tokens = 500
        output_tokens = 200
        cost = (500 * 5 + 200 * 15) / 1e6  # $0.0055 per query

    elif model_type == "reasoning":
        # o1: ~$15/M input, ~$60/M output (including thinking)
        input_tokens = 500
        if query_difficulty == "easy":
            thinking_tokens = 200       # quick reasoning
            output_tokens = 150
        elif query_difficulty == "hard":
            thinking_tokens = 15000     # extended reasoning
            output_tokens = 500
        total_output = thinking_tokens + output_tokens
        cost = (500 * 15 + total_output * 60) / 1e6

    return cost
# Easy query: standard=$0.006, reasoning=$0.028 (5× more)
# Hard query: standard=$0.006, reasoning=$0.938 (156× more!)

Deployment strategies for reasoning models:

  • Difficulty routing: Use a lightweight classifier to route easy queries to standard models and hard queries to reasoning models. This reduces average cost by 60–80% compared to routing everything to a reasoning model.
  • Token budgets: Set max_thinking_tokens per API call to cap worst-case cost. o1's API exposes reasoning_effort (low/medium/high) as a simpler control.
  • Distillation: DeepSeek showed that distilling R1's reasoning traces into smaller models (1.5B–70B) captures much of the reasoning capability at a fraction of inference cost. DeepSeek-R1-Distill-Qwen-32B matches o1-mini on many benchmarks.
  • Caching reasoning: For recurring problem types, cache successful reasoning patterns to avoid repeated thinking on structurally similar queries.
The new cost equation: Total inference cost = base_cost + thinking_cost × difficulty_factor. For production systems, monitor the distribution of thinking tokens, not just the average. A long tail of expensive queries can dominate your bill. Set up alerts when thinking token usage exceeds your 95th percentile threshold.

Reasoning models represent a fundamental shift in how we think about LLM capabilities. Instead of scaling only at training time, we now have a powerful second axis — test-time compute — that lets smaller, cheaper models match larger ones on hard problems. The field is evolving rapidly: expect tighter integration of search algorithms, better reward models, and more efficient reasoning distillation. For MLOps practitioners, the key takeaway is that inference is no longer a fixed cost — it's a variable you must actively manage, monitor, and optimize.
