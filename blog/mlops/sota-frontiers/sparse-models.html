Sparse Models: Activation Sparsity, Conditional Compute

MLOps Series SOTA & Frontiers

Activation Sparsity in FFN Layers

A striking empirical observation: in transformer feed-forward networks (FFN), a large fraction of neurons produce near-zero activations for any given input token. In ReLU-based models, 90–95% of intermediate activations are exactly zero. Even in models using GELU or SwiGLU, 70–85% of activations fall below negligible thresholds. This activation sparsity is distinct from weight sparsity (pruning) — the weights are dense, but the computation graph is dynamically sparse at runtime.

The core idea behind sparse inference: if most FFN neurons contribute nothing to a given token, we can skip their computation entirely and still recover the same (or nearly the same) output. This turns a dense O(d·d_ff) matmul into a sparse operation touching only k active neurons out of d_ff total.

Key insight: Activation sparsity is a free lunch that already exists in trained models. Unlike weight pruning, it requires no retraining — we just need efficient mechanisms to predict which neurons will be active and skip the rest.

ReLU vs SwiGLU Sparsity Patterns

The choice of activation function fundamentally determines natural sparsity levels. ReLU produces exact zeros for all negative pre-activations, yielding 90–97% sparsity in FFN intermediate layers of OPT-66B and similar models. Modern architectures have shifted to SwiGLU (LLaMA, Mistral, Gemma) and GELU (GPT, BLOOM), which never produce exact zeros but still exhibit heavy-tailed activation distributions.

🔷 ReLU Sparsity

Mechanism: max(0, x) → exact zeros

Typical sparsity: 90–97%

Advantage: Trivially exploitable — zero activations mean zero FLOPs with no approximation error

Models: OPT, early GPT variants, ReluLLaMA

Catch: ReLU degrades model quality slightly vs SwiGLU at same param count

🔶 SwiGLU / GELU Sparsity

Mechanism: Smooth activations → near-zero but non-zero

Typical sparsity: 70–90% (with threshold ε < 0.01)

Advantage: Better model quality, standard in SOTA LLMs

Models: LLaMA-2/3, Mistral, Gemma, Qwen

Catch: Requires thresholding or top-k selection; small approximation error

Recent work on ReluLLaMA (Zhang et al., 2024) shows that replacing SwiGLU with ReLU in LLaMA-2 and continuing pre-training for a small number of tokens recovers model quality while boosting natural sparsity to 90%+. This enables direct exploitation of ReLU zeros without any predictor overhead.

# Measuring activation sparsity across different activations import torch import torch.nn.functional as F def measure_sparsity(hidden_states, activation_fn, threshold=0.01): """Compare sparsity of different activation functions.""" if activation_fn == "relu": activated = F.relu(hidden_states) sparsity = (activated == 0).float().mean() elif activation_fn == "gelu": activated = F.gelu(hidden_states) sparsity = (activated.abs() < threshold).float().mean() elif activation_fn == "swiglu": gate = hidden_states[..., :hidden_states.shape[-1]//2] up = hidden_states[..., hidden_states.shape[-1]//2:] activated = F.silu(gate) * up sparsity = (activated.abs() < threshold).float().mean() return sparsity.item() # Empirical results on LLaMA-2-7B (SwiGLU): ~85% near-zero # Empirical results on OPT-66B (ReLU): ~95% exact-zero # ReluLLaMA-7B after conversion: ~92% exact-zero
Approximation error: For SwiGLU/GELU models, zeroing out near-zero activations introduces a small error. At 80% sparsity, perplexity typically increases by <0.5%; at 90%, it can increase by 1–3%. Top-k selection (keeping the k largest) is more robust than flat thresholding.

Top-K Activation Selection

Rather than relying on natural sparsity (which varies by token and layer), top-k activation enforces a fixed sparsity budget: for each token, keep only the k largest activations in the FFN intermediate layer and zero out the rest. This gives a predictable FLOPs reduction of (1 − k/d_ff) and enables hardware-friendly sparse kernels.

The Top-K FFN Forward Pass

# Top-K sparse FFN: only compute output projection for k active neurons import torch import torch.nn as nn class TopKSparseFFN(nn.Module): def __init__(self, d_model, d_ff, k): super().__init__() self.w_up = nn.Linear(d_model, d_ff, bias=False) self.w_gate = nn.Linear(d_model, d_ff, bias=False) self.w_down = nn.Linear(d_ff, d_model, bias=False) self.k = k def forward(self, x): # Step 1: compute full intermediate (unavoidable) gate = torch.sigmoid(self.w_gate(x)) up = self.w_up(x) intermediate = gate * up # SwiGLU-style # Step 2: select top-k activations topk_vals, topk_idx = torch.topk( intermediate.abs(), self.k, dim=-1 ) mask = torch.zeros_like(intermediate) mask.scatter_(-1, topk_idx, 1.0) sparse_intermediate = intermediate * mask # Step 3: down-projection (only k columns active) return self.w_down(sparse_intermediate)

The naive implementation above still computes the full up-projection. The real speedup comes from predictive approaches that determine which k neurons will be active before the expensive matmul, allowing us to load only the corresponding weight columns.

Sparsity budgets in practice: LLaMA-2-7B has d_ff = 11008. At k = 1024 (~9.3% active), the FFN compute drops from ~90M FLOPs to ~8.4M FLOPs per token per layer. Across 32 layers, that's a theoretical 10× reduction in FFN compute — FFN layers account for ~67% of total model FLOPs.

Predictive Sparsity: DejaVu & PowerInfer

The key challenge: how do we know which neurons will be active without computing the full intermediate? Two landmark systems solve this with lightweight predictors.

DejaVu (Liu et al., 2023)

DejaVu trains a small MLP predictor (2-layer, ~1% of FFN size) per layer that takes the attention output as input and predicts which FFN neurons will have top-k activations. The predictor is trained offline on calibration data with a binary cross-entropy loss against ground-truth activation masks.

# DejaVu: lightweight sparsity predictor class SparsityPredictor(nn.Module): """Small MLP that predicts top-k neuron indices.""" def __init__(self, d_model, d_ff, hidden=256): super().__init__() self.net = nn.Sequential( nn.Linear(d_model, hidden), nn.ReLU(), nn.Linear(hidden, d_ff), # logits per neuron ) def forward(self, attn_out): # Predict which neurons will be active logits = self.net(attn_out) # Select top-k predicted neurons _, pred_idx = torch.topk(logits, k=1024, dim=-1) return pred_idx # Prediction accuracy: ~95% overlap with oracle top-k # Predictor overhead: <2% of total FFN compute # Net speedup: 2–6× on OPT-175B with <0.1% quality loss

PowerInfer (Song et al., 2023)

PowerInfer takes a different approach: instead of predicting sparsity at runtime, it profiles activation patterns offline and partitions neurons into "hot" (frequently activated) and "cold" (rarely activated) sets. Hot neurons are permanently placed on GPU; cold neurons stay in CPU DRAM. At inference time, a lightweight classifier routes each token's computation.

🟢 Hot Neurons (GPU)

~10% of FFN neurons

Activated for >50% of tokens

Preloaded into GPU VRAM permanently

Zero transfer overhead at inference

🔵 Cold Neurons (CPU)

~90% of FFN neurons

Activated for <5% of tokens each

Computed on CPU when activated

Async transfer overlaps with GPU compute

PowerInfer results: On a single RTX 4090 (24GB), PowerInfer runs LLaMA-2-70B at 11.7 tokens/sec — an 11.6× speedup over llama.cpp CPU offloading. For Falcon-40B, it achieves 8.0 tok/s vs 0.8 tok/s baseline. The key enabler: power-law activation distributions mean a tiny subset of neurons handles most of the work.

Mixture-of-Depths & Early Exit

Activation sparsity operates within each layer. Conditional compute across layers goes further: not every token needs every layer. Two strategies exploit this insight.

Mixture-of-Depths (Raposo et al., 2024)

MoD introduces a per-layer router that decides whether each token should be processed by a layer or skip it via a residual connection. The router is trained jointly with the model using a capacity factor C that limits how many tokens each layer processes (e.g., C = 0.5 means only 50% of tokens are actively processed per layer).

# Mixture-of-Depths: per-token, per-layer routing class MoDLayer(nn.Module): def __init__(self, d_model, capacity=0.5): super().__init__() self.router = nn.Linear(d_model, 1) self.attn = MultiHeadAttention(d_model) self.ffn = FeedForward(d_model) self.capacity = capacity def forward(self, x): # x: (batch, seq_len, d_model) scores = self.router(x).squeeze(-1) # (B, S) k = int(x.shape[1] * self.capacity) _, top_idx = torch.topk(scores, k, dim=-1) # Only process top-k tokens through the layer selected = x.gather(1, top_idx.unsqueeze(-1).expand(-1,-1,x.shape[-1])) processed = self.ffn(self.attn(selected)) # Scatter back; skipped tokens pass through unchanged out = x.clone() out.scatter_(1, top_idx.unsqueeze(-1).expand_as(processed), processed) return out

Early Exit Strategies

Early exit takes a complementary approach: tokens that the model is already confident about can exit the network at intermediate layers without passing through all N layers. A small classifier head at each layer predicts whether the current representation is "good enough."

🔷 Mixture-of-Depths

Granularity: Per-token, per-layer

Decision: Router selects which tokens to process

Savings: 30–50% FLOPs with isoFLOP-optimal quality

Training: Joint training with capacity constraint

Benefit: Uniform batch size at each layer (good for hardware)

🔶 Early Exit

Granularity: Per-token, exits entire remaining network

Decision: Confidence threshold on auxiliary head

Savings: 20–40% FLOPs on "easy" tokens

Training: Auxiliary heads trained post-hoc or jointly

Challenge: Ragged batches hurt GPU utilization

Practical challenge: Both MoD and early exit create variable-length computation per token, which complicates batched inference. Current frameworks (vLLM, TGI) assume all tokens traverse all layers. Production deployment requires custom schedulers that handle heterogeneous compute paths.

Structured vs Unstructured Sparsity

How sparsity is organized in the weight/activation matrices determines whether hardware can actually exploit it for speedups.

Unstructured Sparsity

Individual elements are zeroed without pattern constraints. Achieves the best quality-per-sparsity ratio because the pruning/selection algorithm has maximum freedom. However, hardware hates it: scattered memory accesses, no vectorization, GPU cores sit idle waiting for irregular data patterns.

Structured Sparsity (N:M)

NVIDIA's Ampere and Hopper architectures support 2:4 structured sparsity: in every group of 4 contiguous elements, exactly 2 must be zero. This yields a guaranteed 50% sparsity with hardware-accelerated sparse tensor core operations, achieving near 2× throughput improvement on A100/H100 GPUs.

# Structured 2:4 sparsity enforcement def apply_2_4_sparsity(weight): """Enforce 2:4 structured sparsity on a weight matrix. For every 4 contiguous elements, keep the 2 largest.""" w = weight.clone() # Reshape into groups of 4 orig_shape = w.shape w_grouped = w.view(-1, 4) # Find top-2 in each group _, top2_idx = torch.topk(w_grouped.abs(), 2, dim=-1) # Create mask: keep only top-2 per group of 4 mask = torch.zeros_like(w_grouped) mask.scatter_(1, top2_idx, 1.0) sparse_w = (w_grouped * mask).view(orig_shape) return sparse_w # Usage with NVIDIA's sparse tensor core API # sparse_w = apply_2_4_sparsity(model.ffn.w_up.weight) # Speedup: 1.5–1.9× on A100 sparse tensor cores # Quality: <1% accuracy drop with fine-tuning recovery

Unstructured Sparsity

Pattern: Arbitrary zero locations

Achievable sparsity: 90%+ with quality

Hardware support: Poor — requires sparse formats (CSR/COO), overhead often eats the theoretical gain

Use case: Research, CPU inference, specialized accelerators

Structured Sparsity (2:4)

Pattern: 2 zeros per 4 elements

Fixed sparsity: Exactly 50%

Hardware support: Native on A100/H100/B100 sparse tensor cores — near-zero overhead

Use case: Production GPU inference, complements quantization

Combining techniques: Structured 2:4 weight sparsity + activation sparsity are orthogonal. Apply 2:4 pruning to weights (50% FLOPs saved) and top-k activation selection (additional 5–10× reduction in active neurons). Together: up to 90–95% compute reduction in FFN layers with minimal quality impact.

Hardware Support & Sparse Tensor Cores

Exploiting sparsity at the hardware level requires architectural support that goes beyond general-purpose CUDA cores.

NVIDIA Sparse Tensor Cores

Starting with Ampere (A100), NVIDIA GPUs include sparse tensor cores that natively execute 2:4 structured sparse matrix multiplications. The sparse matrix is stored in a compressed format alongside a 2-bit index metadata array. The tensor core skips zero-element multiplications and achieves up to 2× throughput over dense operations with identical latency characteristics.

Hardware Sparsity Roadmap

A100 (Ampere)

2:4 structured sparsity

FP16/BF16/INT8/TF32 sparse ops

~2× speedup on eligible matmuls

H100 (Hopper)

2:4 + improved FP8 sparse support

TMA + sparse = overlapped data movement

~2× on FP8, combined with quantization for 4× gains

Beyond NVIDIA, emerging architectures target broader sparsity support. Cerebras WSE-3 natively handles unstructured sparsity via its dataflow architecture — each PE skips zero-valued computations without index overhead. Graphcore IPUs support block-sparse operations with minimal overhead in their BSP execution model.

The sparsity gap: Theoretical speedup from sparsity is often 5–10× higher than achieved speedup. Reasons: (1) memory bandwidth dominates over compute in LLM inference, so reducing FLOPs helps less than expected; (2) irregular access patterns cause cache misses; (3) sparse format overhead (indices, metadata) eats into savings. The real wins come from approaches like PowerInfer that avoid loading inactive weights entirely.

Key Takeaways

  • Activation sparsity is ubiquitous: 70–95% of FFN neurons produce negligible output per token. This is free computational waste to eliminate.
  • ReLU enables exact sparsity but SwiGLU/GELU models can still achieve 80–90% effective sparsity via top-k selection with small quality tradeoffs.
  • Predictive sparsity (DejaVu, PowerInfer) avoids computing the full intermediate by using lightweight predictors or offline profiling to identify active neurons before the matmul.
  • Conditional compute across layers (MoD, early exit) offers orthogonal savings by routing tokens around entire layers.
  • Hardware matters: Structured 2:4 sparsity gets native GPU acceleration; unstructured sparsity needs specialized systems. The gap between theoretical and practical speedup remains large.
  • Combine everything: Sparsity + quantization + speculative decoding = multiplicative speedups. This is the path to running 100B+ models on consumer hardware.
