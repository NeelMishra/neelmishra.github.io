Scaling Laws Revisited: Chinchilla, DeepSeek

MLOps Series SOTA & Frontiers

Kaplan Scaling Laws

The foundational work on neural scaling laws by Kaplan et al. (2020) at OpenAI established that language model loss follows smooth power-law relationships with three variables: model size N (parameters), dataset size D (tokens), and compute budget C (FLOPs). The key equation is:

# Kaplan power-law decomposition
# L(N, D) ≈ [(N_c / N)^α_N + (D_c / D)^α_D + L_∞]
# where L_∞ is the irreducible loss (entropy of natural language)

def kaplan_loss(N, D, N_c=8.8e13, D_c=5.4e13, alpha_N=0.076, alpha_D=0.095, L_inf=1.69):
    """Estimate cross-entropy loss from model size and data."""
    loss = (N_c / N) ** alpha_N + (D_c / D) ** alpha_D + L_inf
    return loss

# Kaplan's key finding: for a fixed compute C, scale model size FASTER than data
# Optimal allocation: N ∝ C^0.73, D ∝ C^0.27
# → "make the model as large as possible, use relatively less data"
Kaplan's prescription: When doubling compute, invest ~73% of the budget increase into a larger model and only ~27% into more data. This led to a generation of models that were parameter-heavy but data-starved — GPT-3 175B was trained on only 300B tokens (~1.7 tokens per parameter), well below what later research proved optimal.

Kaplan's laws were remarkably predictive across 7 orders of magnitude of compute, but they contained a critical methodological flaw: training runs were not continued to convergence. Each model was trained with a fixed schedule, meaning larger models received disproportionately fewer gradient steps relative to their capacity. This biased the findings toward larger models appearing more efficient than they actually were.

The compute estimate C ≈ 6·N·D (FLOPs) comes from the forward and backward pass through each parameter for each token. This approximation holds for dense transformer models and remains the standard budgeting formula across the industry:

# Compute budget estimation
def estimate_flops(num_params, num_tokens):
    """6ND rule: 2 for forward, 4 for backward (gradient + activation)."""
    return 6 * num_params * num_tokens

# Example: GPT-3 175B trained on 300B tokens
gpt3_flops = estimate_flops(175e9, 300e9)   # ≈ 3.15e23 FLOPs
# At A100 peak (312 TFLOPS BF16), that's ~1,010,000 GPU-hours
# With ~40% MFU: ~2,525,000 GPU-hours = ~290 GPU-years

Chinchilla: Compute-Optimal Training

Hoffmann et al. (2022) at DeepMind upended the Kaplan orthodoxy with the Chinchilla paper. By training over 400 models from 70M to 16B parameters, each to multiple data horizons, they showed that model size and data should scale equally with compute:

# Chinchilla optimal allocation
# For a fixed compute budget C:
#   N_opt ∝ C^0.50  (model parameters)
#   D_opt ∝ C^0.50  (training tokens)
# → Approximately 20 tokens per parameter

def chinchilla_optimal(compute_budget_flops):
    """Given a FLOPs budget, return optimal (params, tokens)."""
    # C ≈ 6·N·D and N ≈ D/20 → N = sqrt(C / 120)
    N_opt = (compute_budget_flops / 120) ** 0.5
    D_opt = 20 * N_opt
    return N_opt, D_opt

# Example: 10^24 FLOPs budget
n, d = chinchilla_optimal(1e24)
print(f"Optimal: {n/1e9:.1f}B params, {d/1e9:.0f}B tokens")
# → Optimal: 91.3B params, 1826B tokens

# Compare to Kaplan allocation for same budget:
# Kaplan:     ~200B params, ~833B tokens (2.4:1 ratio)
# Chinchilla: ~91B params,  ~1826B tokens (1:20 ratio)
Why Kaplan was wrong: Kaplan's experiments used a fixed learning rate schedule across all model sizes, without tuning the schedule length to the dataset. Larger models appeared more sample-efficient because they hadn't been trained long enough to saturate. Chinchilla's approach — training each model to its own compute-optimal point — revealed the true symmetric scaling.

The landmark result: Chinchilla (70B params, 1.4T tokens) outperformed Gopher (280B params, 300B tokens) on virtually every benchmark despite using the same training compute (~5×10²³ FLOPs). A 4× smaller model, trained on 4.7× more data, was strictly better. This single result forced the entire field to rethink data collection and model sizing.

The Over-Training Paradigm

Chinchilla defined compute-optimal training, but production deployments care about inference cost too. A 70B model costs 4× more per token to serve than a 13B model. This led to the over-training paradigm: intentionally training smaller models on far more data than the Chinchilla ratio suggests, accepting higher training cost for lower inference cost.

Meta's LLaMA family is the canonical example. LLaMA-1 7B was trained on 1T tokens — that's ~143 tokens per parameter, roughly 7× the Chinchilla-optimal ratio. LLaMA-2 7B pushed further to 2T tokens (~286 tok/param). LLaMA-3 8B used 15T tokens (~1,875 tok/param), nearly 94× the Chinchilla ratio.

# Over-training ratio analysis
def over_training_ratio(model_params, train_tokens, chinchilla_ratio=20):
    """How many multiples of Chinchilla-optimal data was used."""
    chinchilla_tokens = chinchilla_ratio * model_params
    return train_tokens / chinchilla_tokens

models = {
    "GPT-3 175B":      (175e9, 300e9),    # pre-Chinchilla
    "Chinchilla 70B":  (70e9, 1.4e12),    # compute-optimal
    "LLaMA-1 7B":      (7e9, 1e12),       # 7.1× over-trained
    "LLaMA-2 7B":      (7e9, 2e12),       # 14.3× over-trained
    "LLaMA-3 8B":      (8e9, 15e12),      # 93.8× over-trained
    "Mistral 7B":      (7e9, 8e12),       # 57.1× over-trained
    "Gemma-2 9B":      (9e9, 8e12),       # 44.4× over-trained
}

for name, (n, d) in models.items():
    ratio = over_training_ratio(n, d)
    tok_per_param = d / n
    print(f"{name:<20} {tok_per_param:>8.0f} tok/param  {ratio:>6.1f}× Chinchilla")

# Output:
# GPT-3 175B                2 tok/param     0.1× Chinchilla
# Chinchilla 70B           20 tok/param     1.0× Chinchilla
# LLaMA-1 7B              143 tok/param     7.1× Chinchilla
# LLaMA-2 7B              286 tok/param    14.3× Chinchilla
# LLaMA-3 8B             1875 tok/param    93.8× Chinchilla
# Mistral 7B             1143 tok/param    57.1× Chinchilla
# Gemma-2 9B              889 tok/param    44.4× Chinchilla
Does over-training help? Yes — but with diminishing returns. Empirically, loss scales as a power law in data even past the Chinchilla point, but the exponent flattens. LLaMA-3 technical report shows the 8B model's loss curve was still decreasing at 15T tokens, but the improvement from 10T→15T tokens was ~3× smaller per token than from 1T→5T. There's no cliff, just diminishing marginal value.

The economic logic is straightforward. Training a model is a one-time cost; inference runs continuously. For a model deployed at scale (millions of requests/day), every parameter in the model costs money on every forward pass. Over-training shifts cost from the recurring inference line to the one-time training line:

# Total cost of ownership: training + inference
def total_cost(N, D, requests_per_day, deploy_days, gpu_cost_hr=2.50):
    """Simplified TCO for training + inference."""
    # Training cost: C = 6ND FLOPs
    train_flops = 6 * N * D
    train_gpu_hrs = train_flops / (312e12 * 0.4 * 3600)  # A100 BF16, 40% MFU
    train_cost = train_gpu_hrs * gpu_cost_hr

    # Inference cost: ~2N FLOPs per token (forward pass only)
    avg_tokens_per_req = 500
    daily_flops = 2 * N * avg_tokens_per_req * requests_per_day
    infer_gpu_hrs_day = daily_flops / (312e12 * 0.6 * 3600)  # higher util
    infer_cost = infer_gpu_hrs_day * gpu_cost_hr * deploy_days

    return train_cost, infer_cost

# Compare: 70B model (Chinchilla) vs 8B model (over-trained)
# Both targeting similar quality on key benchmarks
t70, i70 = total_cost(70e9, 1.4e12, 1e6, 365)
t8, i8   = total_cost(8e9, 15e12, 1e6, 365)
# 70B: train ~$580K, inference ~$4.1M/yr  → TCO $4.7M
#  8B: train ~$710K, inference ~$470K/yr   → TCO $1.2M

DeepSeek Scaling Analysis

DeepSeek's series of papers (2023–2024) contributed important refinements to scaling law theory, particularly around Mixture-of-Experts (MoE) scaling and data quality interactions. Their work addresses a gap in the original Chinchilla analysis, which only studied dense models on English-dominated web text.

DeepSeek's key findings include:

  • MoE scaling correction: For MoE models, the effective parameter count for scaling laws is closer to N_active × (1 + 0.3 × log(N_total / N_active)), not simply N_active. The routing mechanism provides a bonus that scales logarithmically with the expert multiplier.
  • Data quality multiplier: High-quality data (code, math, curated web) effectively shifts the loss curve down. DeepSeek estimates that 1T tokens of curated data ≈ 3–5T tokens of unfiltered CommonCrawl for downstream task performance.
  • Multi-stage training: Scaling laws derived from single-stage pretraining don't fully predict performance after SFT and RLHF. DeepSeek-V2 demonstrated that a 21B active-parameter MoE could match dense 70B models after alignment, even though raw pretraining loss was higher.
# DeepSeek MoE effective parameter estimation
def moe_effective_params(active_params, total_params):
    """Estimate effective params for scaling law predictions.
    MoE routing provides a logarithmic bonus over active params."""
    import math
    expert_ratio = total_params / active_params
    bonus = 0.3 * math.log(expert_ratio)
    return active_params * (1 + bonus)

# DeepSeek-V2: 21B active out of 236B total
eff = moe_effective_params(21e9, 236e9)
print(f"Effective params: {eff/1e9:.1f}B")  # ~36.2B

# DeepSeek-V3: 37B active out of 671B total
eff_v3 = moe_effective_params(37e9, 671e9)
print(f"Effective params: {eff_v3/1e9:.1f}B")  # ~66.0B
# → Comparable to dense ~66B, which matches empirical benchmark results
DeepSeek-V3 economics: DeepSeek-V3 (671B total, 37B active) was reportedly trained for ~$5.5M using 2048 H800 GPUs over ~2 months. The MoE architecture reduced active compute per token to roughly that of a 37B dense model, while the routing mechanism captures knowledge distributed across 671B parameters. At inference, only 37B parameters are activated per token, making serving costs comparable to a dense 40B model.

DeepSeek's scaling analysis also challenged the assumption that scaling laws are architecture-invariant. Their experiments showed that MoE models with fine-grained experts (256 experts, top-8 routing in DeepSeek-V3) have a steeper scaling curve than coarse-grained MoE (8 experts, top-2), suggesting that routing granularity is itself a scaling dimension.

Inference-Aware Scaling

The traditional scaling law framework optimizes for a single objective: minimising training loss per FLOP. But in production, the objective is minimising total cost of ownership (TCO) per unit of quality. This requires factoring in inference cost, which depends on model size but not training data size. Sardana & Frankle (2023) formalised this as inference-aware scaling laws.

# Inference-aware optimal model size
# The key insight: if you'll serve R tokens of inference for every
# training token, the optimal model shrinks and data grows.

def inference_aware_optimal(compute_budget, inference_ratio):
    """
    compute_budget: total FLOPs for training
    inference_ratio: total inference tokens / training tokens
    Returns (optimal_params, optimal_train_tokens)
    """
    # Chinchilla baseline: N* = sqrt(C/120)
    N_chinchilla = (compute_budget / 120) ** 0.5

    # Inference penalty shrinks optimal model
    # Sardana & Frankle: N_opt ∝ N_chinchilla / (1 + R)^β
    # where β ≈ 0.42 empirically
    beta = 0.42
    N_opt = N_chinchilla / (1 + inference_ratio) ** beta
    D_opt = compute_budget / (6 * N_opt)
    return N_opt, D_opt

# High-volume API: 1000× more inference than training tokens
n, d = inference_aware_optimal(1e24, 1000)
print(f"Params: {n/1e9:.1f}B, Tokens: {d/1e12:.1f}T")
# → ~5.8B params, ~28.7T tokens (massively over-trained)

# Low-volume internal tool: 10× inference/training ratio
n2, d2 = inference_aware_optimal(1e24, 10)
print(f"Params: {n2/1e9:.1f}B, Tokens: {d2/1e12:.1f}T")
# → ~42.5B params, ~3.9T tokens (mild over-training)

Compute-Optimal (Chinchilla)

Minimises loss per training FLOP

Best for: Research, one-off evals

~20 tokens per parameter

Produces larger model, higher serving cost

Inference-Optimal

Minimises loss per total FLOP (train + serve)

Best for: Production APIs, chatbots

100–2000+ tokens per parameter

Smaller model, much cheaper to serve

The practical implication is a spectrum: the higher your expected inference volume relative to training, the smaller your model should be (and the more data you should train on). This explains why frontier API providers (OpenAI, Anthropic, Google) offer both a large "flagship" model and a smaller distilled model — they face different inference-to-training ratios across customer segments.

The Emergent Abilities Debate

Wei et al. (2022) observed that certain capabilities — multi-step arithmetic, chain-of-thought reasoning, word unscrambling — appear to emerge abruptly as models scale, rather than improving gradually. This was termed emergent abilities: capabilities not present in smaller models that appear in larger ones without being explicitly trained.

Schaeffer et al. (2023) challenged this framing, arguing that "emergence" is largely a measurement artifact. Their key insight: when performance is measured with discontinuous metrics (e.g., exact-match accuracy), smooth underlying improvements appear as sudden jumps. Switching to continuous metrics (e.g., token-level log-likelihood, Brier score) reveals the improvement was gradual all along.

Metric matters: Consider 3-digit addition. A model that predicts "456" for an answer of "457" scores 0% on exact-match but has clearly learned most of the task. As models scale, they go from "totally wrong" to "off by a digit" to "correct" — a smooth progression that exact-match collapses into a step function. Plotting per-digit accuracy instead of whole-answer accuracy eliminates the apparent emergence.

The implications for infrastructure are significant:

  • If abilities emerge smoothly: Scaling laws can reliably predict when a capability will become useful. This enables confident compute budgeting — you can forecast the model size needed for your target quality level.
  • If abilities truly emerge: Scaling laws predict loss but not capabilities. You might invest in scaling and miss the threshold, or find unexpected capabilities at a scale you didn't plan for. This makes infrastructure planning harder and favors building flexible, right-sizable platforms.
  • Practical consensus: Most practitioners now treat capabilities as smooth on the right metric but acknowledge that some task-level performance does have phase-transition-like behavior (particularly tasks requiring multiple sequential reasoning steps). The safe approach is to build evaluation suites with continuous metrics and plan infrastructure with safety margins.
# Evaluating "emergence" with proper metrics
def evaluate_arithmetic(model, problems):
    """Compare discontinuous vs continuous metrics."""
    exact_match = 0
    digit_accuracy = 0
    log_likelihood = 0
    total_digits = 0

    for problem, answer in problems:
        pred = model.generate(problem)
        # Discontinuous: all-or-nothing
        if pred == answer:
            exact_match += 1

        # Continuous: per-digit accuracy
        for p_digit, a_digit in zip(pred, answer):
            if p_digit == a_digit:
                digit_accuracy += 1
            total_digits += 1

        # Continuous: log-likelihood of correct answer
        log_likelihood += model.log_prob(answer, context=problem)

    return {
        "exact_match": exact_match / len(problems),
        "digit_accuracy": digit_accuracy / total_digits,
        "avg_log_likelihood": log_likelihood / len(problems),
    }

Practical Compute Budgeting

Bringing together the theoretical frameworks into a practical decision process requires balancing multiple scaling regimes. Below is a decision framework for choosing model size and training data given a fixed budget and deployment scenario.

Estimate your inference-to-training ratio Formula: R = (avg_requests/day × avg_tokens/request × deployment_days) / training_tokens. For a popular API serving 1M req/day at 500 tok/req over 1 year: R = 1M × 500 × 365 / D_train. Even with D_train = 5T, R ≈ 36.5. For internal tools R may be <1.
Set model size using inference-aware formula Rule of thumb: If R > 100, use a model ≤ 10B params. If 10 < R < 100, use 10–40B params. If R < 10, Chinchilla-optimal sizing is fine. Always validate with pilot training runs on 1–5% of target compute to confirm loss predictions.
Plan your data pipeline capacity Critical: Over-training demands massive data. LLaMA-3 8B used 15T tokens of carefully deduplicated, quality-filtered text. Budget 30–50% of training engineering effort on data pipelines: deduplication, quality scoring, domain mixing, curriculum design. Data is the binding constraint in the over-training regime. 
# Complete scaling decision framework
def plan_training_run(
    budget_flops,
    inference_ratio,
    data_quality="curated",     # "raw", "filtered", "curated"
    architecture="dense",        # "dense", "moe"
    moe_expert_ratio=1,          # total_params / active_params
):
    """Recommend model config given constraints."""
    # Step 1: Chinchilla baseline
    N_chin = (budget_flops / 120) ** 0.5
    D_chin = 20 * N_chin

    # Step 2: Inference-aware adjustment
    beta = 0.42
    N_opt = N_chin / (1 + inference_ratio) ** beta
    D_opt = budget_flops / (6 * N_opt)

    # Step 3: Data quality multiplier
    quality_mult = {"raw": 1.0, "filtered": 2.0, "curated": 4.0}
    effective_D = D_opt * quality_mult[data_quality]

    # Step 4: MoE adjustment (if applicable)
    if architecture == "moe":
        N_total = N_opt * moe_expert_ratio
        N_active = N_opt
    else:
        N_total = N_opt
        N_active = N_opt

    over_train_ratio = D_opt / D_chin

    return {
        "total_params": N_total,
        "active_params": N_active,
        "train_tokens": D_opt,
        "effective_tokens": effective_D,
        "over_training_ratio": over_train_ratio,
        "chinchilla_params": N_chin,
        "tok_per_param": D_opt / N_active,
    }

# Scenario: $10M budget, high-volume API, curated data, MoE
# ~10^24 FLOPs budget (rough: $10M / $2.50/GPU-hr × 450 TFLOPS × 3600)
plan = plan_training_run(
    budget_flops=1e24,
    inference_ratio=500,
    data_quality="curated",
    architecture="moe",
    moe_expert_ratio=8,
)
print(f"Active: {plan['active_params']/1e9:.1f}B")
print(f"Total (MoE): {plan['total_params']/1e9:.1f}B")
print(f"Tokens: {plan['train_tokens']/1e12:.1f}T")
print(f"Over-training: {plan['over_training_ratio']:.1f}× Chinchilla")
print(f"Tok/param: {plan['tok_per_param']:.0f}")
Key takeaway: Scaling laws have evolved from a single-variable power law (Kaplan) to a multi-objective optimisation problem. The "right" model size depends on your data quality, inference volume, architecture choice (dense vs MoE), and deployment timeline. Chinchilla showed that data matters as much as parameters; the over-training paradigm showed that inference economics can dominate; and DeepSeek showed that architecture (MoE granularity) adds another dimension to the scaling frontier.
What to watch: Test-time compute scaling (e.g., chain-of-thought, tree search at inference) introduces yet another dimension. A smaller model that "thinks longer" can match a larger model's one-shot answer. OpenAI's o1 and DeepSeek-R1 represent early steps toward inference-time scaling laws — where spending more FLOPs at inference (not training) improves results. This may eventually dominate the training-time scaling picture.
