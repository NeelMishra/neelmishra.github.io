Benchmark Methodology

A fair comparison requires a controlled setup. Our benchmarks use the ann-benchmarks framework with the following parameters: 1M vectors of dimension 1536 (matching OpenAI text-embedding-3-small output), cosine similarity metric, recall@10 target of ≥0.95, and a single-node deployment for each database on an r6i.2xlarge EC2 instance (8 vCPU, 64 GB RAM, gp3 SSD).

Each database was configured with its recommended HNSW parameters: M=16, ef_construction=200, ef_search=128 (tuned to hit the 0.95 recall target). For Milvus, we additionally tested IVF-PQ as a memory-optimised alternative. Chroma was tested in persistent mode with its default HNSW settings. Indexing was timed end-to-end from the first upsert call to the last flush confirmation.

Disclaimer: Benchmarks are highly sensitive to hardware, dataset distribution, and configuration tuning. Treat these as directional rather than absolute.

Cost Analysis

Cost modelling depends on vector count, query volume, and whether you use managed or self-hosted infrastructure. Below is a simplified monthly cost estimate for a 5M-vector index serving 100 QPS.

Managed Services

  • Pinecone s2: ~$280/mo (pod-based) or usage-based on serverless
  • Weaviate Cloud: ~$220/mo (performance tier, 5M vectors)
  • Qdrant Cloud: ~$195/mo (4 GB RAM node, 5M vectors)
  • Zilliz Cloud: ~$250/mo (performance CU, 5M vectors)

Self-Hosted (AWS)

  • EC2 r6i.xlarge: ~$145/mo (on-demand) + EBS
  • EKS overhead: ~$73/mo (control plane)
  • Total infra: ~$230/mo before ops labour
  • Break-even: Self-hosted wins past ~20M vectors
Key insight: At ≤5M vectors and ≤200 QPS, managed services are almost always cheaper than self-hosted when you factor in engineering time for upgrades, monitoring, and incident response.

Scalability

Scalability encompasses three axes: vector count (how many vectors fit in a single node before you must shard), query throughput (horizontal read scaling), and write throughput (ingestion rate during bulk loads or streaming updates).

For billion-scale deployments, Milvus and Qdrant are the strongest contenders. Milvus's disaggregated architecture lets you scale query nodes independently from data nodes, while Qdrant's quantisation and memory-mapped segments push single-node capacity higher than competitors with similar RAM.

Recommendations

There is no universally "best" vector database — the right choice depends on your constraints. Here is a decision framework based on common scenarios:

# Decision tree (pseudo-code) if vectors < 100_000 and team == "prototyping": return "Chroma" # zero-config, pip install, in-process elif need_zero_ops and budget_ok: return "Pinecone Serverless" # managed, pay-per-use elif need_hybrid_search: return "Weaviate" # best BM25 + dense fusion elif vectors > 50_000_000 and need_gpu_index: return "Milvus" # GPU-IVF, disaggregated arch elif latency_critical and filtered_search: return "Qdrant" # Rust speed, in-graph filtering else: evaluate("Weaviate", "Qdrant") # both are strong defaults
Pro tip: Abstract your vector DB behind a thin interface (upsert, query, delete). This lets you swap backends without rewriting application logic — critical when the ecosystem is evolving this fast.

Abstraction Layer Example

from abc import ABC, abstractmethod from typing import Any class VectorStore(ABC): """Thin abstraction over any vector database.""" @abstractmethod def upsert(self, ids: list[str], vectors: list[list[float]], metadatas: list[dict]) -> None: ... @abstractmethod def query(self, vector: list[float], top_k: int = 10, filters: dict | None = None) -> list[dict]: ... @abstractmethod def delete(self, ids: list[str]) -> None: ... class QdrantStore(VectorStore): def __init__(self, url, collection): from qdrant_client import QdrantClient self.client = QdrantClient(url=url) self.collection = collection def upsert(self, ids, vectors, metadatas): from qdrant_client.models import PointStruct points = [ PointStruct(id=i, vector=v, payload=m) for i, v, m in zip(ids, vectors, metadatas) ] self.client.upsert(self.collection, points=points) def query(self, vector, top_k=10, filters=None): hits = self.client.query_points( self.collection, query=vector, limit=top_k ) return [{"id": h.id, "score": h.score, **h.payload} for h in hits.points] def delete(self, ids): self.client.delete(self.collection, points_selector=ids)
