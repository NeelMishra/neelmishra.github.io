BM25 + Dense Hybrid: Best of Both Worlds

MLOps Series RAG Systems

Dense retrievers capture semantic similarity. BM25 captures exact lexical matching. Neither alone is optimal for every query. Hybrid retrieval fuses both signals, and in practice it outperforms each individual method on virtually every benchmark. This post provides a complete, production-grade implementation of hybrid retrieval with two fusion strategies: Reciprocal Rank Fusion (RRF) and weighted score combination.

Why Hybrid?

Consider two failure modes of single-method retrieval:

Dense Fails Here

Query: "error code 0x80070005"
Dense embeddings compress the hex code into a general "error" region. BM25 matches the exact string.

BM25 Fails Here

Query: "how to speed up my machine learning training"
Document: "Techniques for accelerating neural network convergence". Zero keyword overlap, but semantically relevant.

Empirical evidence from the BEIR benchmark (a diverse collection of 18 retrieval datasets) consistently shows hybrid retrieval improving NDCG@10 by 3–8 points over the best single retriever. The gains are most pronounced on out-of-domain datasets where neither method alone generalizes well.

Key insight: Hybrid retrieval is not just a marginal improvement — it fundamentally broadens the types of queries your RAG system can handle. It should be the default retrieval strategy.

Reciprocal Rank Fusion (RRF)

RRF was introduced by Cormack et al. (2009). The idea is elegant: instead of combining raw scores (which may have incompatible scales), combine rankings. Each retriever produces a ranked list. For each document d, we compute:

RRF(d) = Σr∈retrievers 1 / (k + rankr(d))

Where k is a constant (typically 60) that dampens the influence of high-ranking items. Documents are then sorted by their RRF score in descending order.

RRF has three critical advantages:

  • Scale-invariant: Works regardless of whether one retriever returns cosine similarities in [0, 1] and another returns BM25 scores in [0, 50].
  • No hyperparameter tuning: k=60 works well across domains. Unlike weighted combination, you don't need a dev set to tune α.
  • Robust to outliers: A single retriever scoring a document anomalously high doesn't dominate the fused result.
# Full RRF implementation from typing import List, Dict, Tuple def reciprocal_rank_fusion( ranked_lists: List[List[int]], k: int = 60, top_n: int = 10 ) -> List[Tuple[int, float]]: """Fuse multiple ranked lists using RRF. Args: ranked_lists: List of ranked doc ID lists (best first). k: RRF constant. Higher k = more weight to lower ranks. top_n: Number of results to return. Returns: List of (doc_id, rrf_score) tuples, sorted desc. """ rrf_scores: Dict[int, float] = {} for ranked in ranked_lists: for rank, doc_id in enumerate(ranked): if doc_id not in rrf_scores: rrf_scores[doc_id] = 0.0 rrf_scores[doc_id] += 1.0 / (k + rank + 1) sorted_docs = sorted( rrf_scores.items(), key=lambda x: x[1], reverse=True ) return sorted_docs[:top_n]

Weighted Score Combination

Weighted combination operates on scores rather than ranks. It requires normalizing scores from each retriever to a common scale, then computing: final(d) = α·norm_dense(d) + (1−α)·norm_sparse(d).

The normalization step is critical. Common approaches:

  • Min-Max: norm(s) = (s − min) / (max − min). Simple but sensitive to outliers.
  • Z-Score: norm(s) = (s − μ) / σ. Better for normally-distributed scores.
  • Percentile rank: Map each score to its percentile. Most robust but requires sorting.
# Weighted combination with min-max normalization import numpy as np def min_max_normalize(scores: np.ndarray) -> np.ndarray: s_min, s_max = scores.min(), scores.max() if s_max - s_min < 1e-9: return np.zeros_like(scores) return (scores - s_min) / (s_max - s_min) def weighted_hybrid( dense_scores: np.ndarray, sparse_scores: np.ndarray, alpha: float = 0.7, top_k: int = 10 ) -> np.ndarray: """Combine dense and sparse scores with alpha weighting.""" d_norm = min_max_normalize(dense_scores) s_norm = min_max_normalize(sparse_scores) combined = alpha * d_norm + (1 - alpha) * s_norm return np.argsort(combined)[::-1][:top_k]
Alpha tuning required: The optimal α depends on your domain. Technical documentation with precise terminology benefits from higher sparse weight (α ≈ 0.4). Conversational Q&A benefits from higher dense weight (α ≈ 0.7). Always tune on a held-out validation set.

Production Implementation

Here is a complete, production-ready hybrid retriever class that supports both RRF and weighted combination:

from sentence_transformers import SentenceTransformer from rank_bm25 import BM25Okapi import numpy as np from typing import List, Optional class HybridRetriever: def __init__(self, model_name: str = "all-MiniLM-L6-v2"): self.encoder = SentenceTransformer(model_name) self.corpus: List[str] = [] self.doc_embeddings: Optional[np.ndarray] = None self.bm25: Optional[BM25Okapi] = None def index(self, documents: List[str]): """Index documents for both dense and sparse retrieval.""" self.corpus = documents self.doc_embeddings = self.encoder.encode( documents, normalize_embeddings=True, show_progress_bar=True ) tokenized = [doc.lower().split() for doc in documents] self.bm25 = BM25Okapi(tokenized) def search_rrf(self, query: str, top_k: int = 10, rrf_k: int = 60): dense_ranking = self._dense_rank(query) sparse_ranking = self._sparse_rank(query) return reciprocal_rank_fusion( [dense_ranking, sparse_ranking], k=rrf_k, top_n=top_k ) def search_weighted(self, query: str, alpha=0.7, top_k=10): dense_s = self._dense_scores(query) sparse_s = self._sparse_scores(query) ids = weighted_hybrid(dense_s, sparse_s, alpha, top_k) return [(i, self.corpus[i]) for i in ids] def _dense_scores(self, query): q = self.encoder.encode([query], normalize_embeddings=True) return np.dot(self.doc_embeddings, q.T).flatten() def _dense_rank(self, query): return np.argsort(self._dense_scores(query))[::-1].tolist() def _sparse_scores(self, query): return self.bm25.get_scores(query.lower().split()) def _sparse_rank(self, query): return np.argsort(self._sparse_scores(query))[::-1].tolist()

# Evaluation code: compute recall@k def recall_at_k(predicted_ids, ground_truth_ids, k=10): """Compute recall@k for a single query.""" pred_set = set(predicted_ids[:k]) gt_set = set(ground_truth_ids) return len(pred_set & gt_set) / len(gt_set) def ndcg_at_k(predicted_ids, relevance_map, k=10): """Compute NDCG@k using a relevance map {doc_id: score}.""" import math dcg = sum( relevance_map.get(doc_id, 0) / math.log2(i + 2) for i, doc_id in enumerate(predicted_ids[:k]) ) ideal = sorted(relevance_map.values(), reverse=True)[:k] idcg = sum(r / math.log2(i + 2) for i, r in enumerate(ideal)) return dcg / idcg if idcg > 0 else 0.0

Production Tips

Deploying hybrid retrieval at scale introduces additional considerations beyond algorithm correctness:

1. Parallel Retrieval

Run dense and sparse retrievals concurrently using async I/O or thread pools. Both are independent and I/O-bound, so concurrent execution typically halves the latency compared to sequential.

import asyncio from concurrent.futures import ThreadPoolExecutor executor = ThreadPoolExecutor(max_workers=2) async def parallel_retrieve(query, retriever): loop = asyncio.get_event_loop() dense_task = loop.run_in_executor( executor, retriever._dense_rank, query ) sparse_task = loop.run_in_executor( executor, retriever._sparse_rank, query ) dense_r, sparse_r = await asyncio.gather(dense_task, sparse_task) return reciprocal_rank_fusion([dense_r, sparse_r])

2. Caching BM25 Scores

BM25 scores are deterministic for a fixed corpus. Cache document-frequency statistics and pre-compute IDF values at index time. Only per-query TF computation happens at search time.

3. Index Versioning

Version your BM25 tokenizer configuration alongside your embedding model. A mismatch (e.g., different lowercasing or stemming) will silently degrade sparse retrieval quality.

4. Monitoring

Log both dense and sparse scores for each query in production. This lets you diagnose whether quality regressions come from the embedding model (dense score distribution shifts) or corpus changes (BM25 IDF shifts).

Start with RRF. It requires no tuning and is robust. Only switch to weighted combination if you have a labeled validation set and can afford the tuning cycle. In our experience, RRF matches or exceeds weighted combination 80% of the time.
