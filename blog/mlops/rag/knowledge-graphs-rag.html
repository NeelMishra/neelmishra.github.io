Knowledge Graphs + RAG: GraphRAG

MLOps Series RAG Systems

Why GraphRAG

Standard vector-based RAG retrieves chunks that are semantically close to the query, but it has no understanding of how entities in those chunks relate to one another. When a user asks "How does Company X's acquisition of Company Y affect product Z?", vector search may return passages mentioning each entity individually, but it cannot reason about the causal chain connecting them.

Knowledge graphs solve this by representing documents as a network of typed entities and relationships. Instead of retrieving flat text, GraphRAG traverses the graph to collect a connected sub-graph that structurally answers the question. This is especially powerful for multi-hop reasoning, where the answer depends on information scattered across several documents.

Key insight: GraphRAG does not replace vector retrieval — it augments it. Vectors find semantically relevant regions; the graph connects them through explicit entity relationships.

Empirically, GraphRAG pipelines show 15–30% improvements on multi-hop QA benchmarks (HotpotQA, MuSiQue) compared to vanilla RAG, because they resolve entity coreferences and follow relational paths that embedding similarity alone cannot capture.

Entity Extraction

The first stage of any GraphRAG pipeline is extracting entities and relationships from raw text. There are three main approaches: NER-based extraction using spaCy or similar, LLM-based extraction via structured prompts, and hybrid methods that combine both. LLM-based extraction is currently dominant because it handles open-domain entity types without pre-training on a fixed schema.

import openai, json def extract_entities(text: str, model: str = "gpt-4o-mini") -> dict: """Extract entities and relationships using structured LLM output.""" system = ( "Extract all named entities and relationships from the text. " "Return JSON with keys 'entities' (list of {name, type, description}) " "and 'relationships' (list of {source, target, relation, weight})." ) resp = openai.chat.completions.create( model=model, response_format={"type": "json_object"}, messages=[ {"role": "system", "content": system}, {"role": "user", "content": text}, ], ) return json.loads(resp.choices[0].message.content)

A critical design choice is the entity schema. A completely open schema allows maximum flexibility but produces noisy, inconsistent graphs. Constraining entity types to a domain-specific ontology (e.g., Person, Organization, Product, Event) dramatically improves graph quality. The sweet spot is 8–15 entity types for most enterprise use cases.

Watch out: LLM extraction is non-deterministic. Running the same text twice may yield different entity names or relationship labels. Always apply entity resolution (deduplication) downstream.

Graph Construction

Once entities and relationships are extracted from each chunk, they must be merged into a unified knowledge graph. This involves three sub-tasks: entity resolution (merging "OpenAI" and "Open AI" into one node), relationship deduplication, and community detection for downstream summarization.

import networkx as nx from collections import defaultdict class KnowledgeGraph: def __init__(self): self.G = nx.DiGraph() self._entity_map = {} # normalized name -> canonical name def resolve_entity(self, name: str) -> str: """Fuzzy-match entity names to canonical forms.""" normalized = name.lower().strip() if normalized in self._entity_map: return self._entity_map[normalized] # Check Levenshtein distance against existing entities for existing in self._entity_map: if levenshtein(normalized, existing) <= 2: self._entity_map[normalized] = self._entity_map[existing] return self._entity_map[existing] self._entity_map[normalized] = name return name def add_extraction(self, extraction: dict, source_chunk: str): for ent in extraction["entities"]: canonical = self.resolve_entity(ent["name"]) self.G.add_node(canonical, type=ent["type"], desc=ent["description"], source=source_chunk) for rel in extraction["relationships"]: src = self.resolve_entity(rel["source"]) tgt = self.resolve_entity(rel["target"]) self.G.add_edge(src, tgt, relation=rel["relation"], weight=rel.get("weight", 1.0))

Entity resolution is the hardest sub-task. Production systems typically combine exact string matching, Levenshtein distance, and embedding-based similarity to merge nodes. A graph with 50,000 entities may contain 10–15% duplicates without resolution, severely degrading retrieval quality.

Graph-Guided Retrieval

Given a query, graph-guided retrieval works in three stages. First, entities mentioned in the query are identified and mapped to graph nodes. Second, the local neighborhood of those nodes is traversed — typically 1–3 hops — to collect relevant edges and connected entities. Third, the sub-graph is serialized into natural language context and injected into the LLM prompt alongside the original query.

def graph_retrieve(kg: KnowledgeGraph, query_entities: list, hops: int = 2) -> str: """Retrieve sub-graph context for a list of query entities.""" visited = set() triples = [] for entity in query_entities: canonical = kg.resolve_entity(entity) if canonical not in kg.G: continue # BFS traversal up to N hops queue = [(canonical, 0)] while queue: node, depth = queue.pop(0) if node in visited or depth > hops: continue visited.add(node) for _, neighbor, data in kg.G.edges(node, data=True): triples.append(f"{node} --[{data['relation']}]--> {neighbor}") queue.append((neighbor, depth + 1)) return "\n".join(triples)

The hop parameter controls the trade-off between recall and noise. One hop retrieves immediate neighbors — good for factoid questions. Two hops captures bridge entities — essential for multi-hop reasoning. Three or more hops often introduces too much noise unless the graph is very sparse.

Hybrid retrieval: The best production systems merge graph sub-graph context with top-k vector retrieval results, using the graph to re-rank and connect the vector-retrieved passages.

Microsoft GraphRAG

Microsoft's GraphRAG paper (2024) introduced a specific two-level indexing strategy. First, entities and relationships are extracted from all chunks and merged into a global graph. Then, the Leiden algorithm is used for hierarchical community detection, grouping densely connected entities into communities at multiple resolutions.

Each community receives an LLM-generated summary that describes the entities it contains and their key relationships. At query time, there are two retrieval modes:

Local Search

Starts from query entities, expands through the graph, and retrieves entity descriptions, relationship triples, and associated text chunks. Best for specific, entity-centric questions like "What products does Company X sell?"

Global Search

Aggregates community summaries using a map-reduce pattern. Each community summary is scored for relevance, and top summaries are combined into the final context. Best for broad, thematic questions like "What are the main trends in the dataset?"

Community detection using the Leiden algorithm produces a hierarchy. Level 0 communities are fine-grained (5–20 entities), while higher levels aggregate into coarser groups. This enables the system to match the granularity of the query to the right summary level.

import graspologic.partition as gp def detect_communities(G: nx.Graph, resolution: float = 1.0) -> dict: """Run Leiden community detection and return node-to-community mapping.""" undirected = G.to_undirected() partitions = gp.hierarchical_leiden( undirected, max_cluster_size=30, random_seed=42 ) # Build community -> [entities] mapping at each level communities = {} for level, partition in enumerate(partitions): communities[level] = {} for node, comm_id in partition.items(): communities[level].setdefault(comm_id, []).append(node) return communities def summarize_community(entities: list, kg: KnowledgeGraph) -> str: """Generate LLM summary for a community of entities.""" sub = kg.G.subgraph(entities) triples = [f"{u} -[{d['relation']}]-> {v}" for u, v, d in sub.edges(data=True)] prompt = f"Summarize these entities and relationships:\n{chr(10).join(triples)}" return llm_call(prompt)

Full Implementation

Putting it all together, a complete GraphRAG pipeline involves: (1) chunking documents, (2) extracting entities per chunk, (3) building and resolving the graph, (4) detecting communities and generating summaries, and (5) routing queries to local or global search based on query type.

class GraphRAGPipeline: def __init__(self, docs: list[str]): self.kg = KnowledgeGraph() self.chunks = [] self.community_summaries = {} def index(self, docs: list[str]): # Phase 1: chunk and extract for doc in docs: chunks = chunk_document(doc, size=600, overlap=100) for chunk in chunks: self.chunks.append(chunk) extraction = extract_entities(chunk) self.kg.add_extraction(extraction, chunk) # Phase 2: community detection + summarization communities = detect_communities(self.kg.G) for level, comms in communities.items(): for comm_id, entities in comms.items(): summary = summarize_community(entities, self.kg) self.community_summaries[(level, comm_id)] = summary def query(self, question: str, mode: str = "local") -> str: if mode == "local": entities = extract_query_entities(question) context = graph_retrieve(self.kg, entities, hops=2) else: # Global: map-reduce over community summaries scored = [] for key, summary in self.community_summaries.items(): score = relevance_score(question, summary) scored.append((score, summary)) scored.sort(reverse=True) context = "\n\n".join(s for _, s in scored[:10]) return generate_answer(question, context)

In production, the indexing phase is the most expensive step — entity extraction alone can cost $5–15 per million tokens with GPT-4o-mini. However, the graph only needs to be rebuilt when documents change, and incremental updates (adding new chunks without re-extracting everything) are straightforward with a well-designed entity resolution layer.

Storage: For production graphs exceeding 100K nodes, use Neo4j or Amazon Neptune instead of in-memory NetworkX. These databases support native graph traversal, indexing, and Cypher/SPARQL queries that scale to millions of relationships.

GraphRAG represents a fundamental shift in how we think about retrieval — from "find similar text" to "navigate structured knowledge." As LLM extraction quality improves and graph databases become more accessible, expect GraphRAG patterns to become standard in enterprise RAG deployments.
