Advanced RAG: Self-RAG, CRAG, Adaptive

MLOps Series RAG Systems

Naive RAG — retrieve top-k, concatenate, generate — breaks down quickly in production. Documents can be irrelevant, contradictory, or simply insufficient. Advanced RAG techniques add reflection, correction, and adaptive decision-making to the retrieval loop so the model knows when to retrieve, what to keep, and whether the final answer is grounded.

Naive RAG Limitations

Standard retrieve-then-generate pipelines suffer from several fundamental problems that become critical at scale:

  • Retrieval noise: Top-k results often include tangentially related passages that dilute the context window and cause hallucination.
  • No retrieval gating: The system always retrieves, even for queries the LLM can answer from parametric knowledge alone, wasting latency and adding noise.
  • No self-verification: Once the model generates, there is no mechanism to check whether the output is actually supported by the retrieved context.
  • Single-hop limitation: Complex questions requiring multi-step reasoning across documents cannot be answered in one retrieval pass.
Production reality: In benchmarks, naive RAG can hallucinate on 15–30% of queries where retrieved context is partially relevant but insufficient. Advanced techniques reduce this to under 5%.

Self-RAG: Self-Reflective Retrieval-Augmented Generation

Self-RAG (Asai et al., 2023) trains a single LM to generate special reflection tokens that decide retrieval and assess output quality on-the-fly. The model learns four types of reflection tokens:

  • [Retrieve]: Should I retrieve? Values: yes, no, continue.
  • [ISREL]: Is the retrieved passage relevant to the query? Values: relevant, irrelevant.
  • [ISSUP]: Is the generated response supported by the passage? Values: fully supported, partially supported, no support.
  • [ISUSE]: Is the response useful to the user? Utility grade from 1 to 5.

During inference, Self-RAG generates multiple candidate continuations in parallel. Each continuation is scored by the reflection tokens and the best-scoring segment is selected. This tree-search over generations lets the model self-correct without external reward models.

class SelfRAG: def generate(self, query: str, documents: list[str]) -> str: # Step 1: Decide whether retrieval is needed retrieve_token = self.predict_retrieve_token(query) if retrieve_token == "yes": passages = self.retrieve(query, top_k=5) # Step 2: Filter by relevance reflection relevant = [p for p in passages if self.predict_isrel(query, p) == "relevant"] else: relevant = [] # Step 3: Generate candidates per passage candidates = [] for passage in relevant: response = self.generate_with_context(query, passage) # Step 4: Score with ISSUP and ISUSE tokens support = self.predict_issup(passage, response) utility = self.predict_isuse(query, response) candidates.append((response, support, utility)) # Step 5: Select best candidate by composite score return self.rank_and_select(candidates)
Key insight: Self-RAG does not require a separate retriever-critic model. The same language model learns both generation and reflection during fine-tuning, keeping the system simple and fast.

Corrective RAG (CRAG)

Corrective RAG (Yan et al., 2024) introduces a lightweight retrieval evaluator that grades documents as Correct, Ambiguous, or Incorrect before they reach the generator. When documents are rated incorrect, CRAG triggers a web search fallback to supplement or replace the retrieved context.

The CRAG pipeline operates in three correction modes:

  1. Correct: At least one document passes the evaluator threshold. Apply knowledge refinement to strip irrelevant sentences and pass refined context to the generator.
  2. Incorrect: All documents fail. Trigger web search, extract relevant snippets, and replace the original context entirely.
  3. Ambiguous: Mixed results. Combine refined internal documents with web search results for maximum coverage.
class CorrectiveRAG: def __init__(self, retriever, evaluator, generator, web_searcher): self.retriever = retriever self.evaluator = evaluator # Lightweight T5 classifier self.generator = generator self.web_searcher = web_searcher def process(self, query: str) -> str: docs = self.retriever.retrieve(query, top_k=10) # Grade each document grades = [self.evaluator.grade(query, doc) for doc in docs] correct = [d for d, g in zip(docs, grades) if g == "correct"] incorrect = [d for d, g in zip(docs, grades) if g == "incorrect"] if len(correct) > 0 and len(incorrect) == 0: # All correct: refine and generate refined = self.knowledge_refinement(correct) return self.generator.generate(query, refined) elif len(correct) == 0: # All incorrect: fall back to web search web_results = self.web_searcher.search(query) return self.generator.generate(query, web_results) else: # Ambiguous: combine both sources refined = self.knowledge_refinement(correct) web_results = self.web_searcher.search(query) return self.generator.generate(query, refined + web_results)

Adaptive Retrieval

Not every query requires retrieval. Adaptive retrieval systems learn to classify queries into those that benefit from external knowledge and those the model can answer from its parametric memory alone. This saves latency and avoids retrieval noise on straightforward questions.

There are two dominant approaches to adaptive retrieval:

Classifier-Based Gating

A lightweight classifier (logistic regression or small transformer) predicts whether retrieval will improve the answer. Trained on query–answer pairs with and without retrieval, measuring answer quality delta.

  • Fast inference (~1ms overhead)
  • Requires labeled training data
  • Binary decision: retrieve or skip

Confidence-Based Gating

Generate an initial answer, measure the model’s confidence via token probabilities. If confidence is below a threshold, trigger retrieval and regenerate. Uses the LLM itself as the gating signal.

  • No external classifier needed
  • Higher latency (two-pass generation)
  • Calibration-dependent accuracy
class AdaptiveRetriever: def __init__(self, llm, retriever, confidence_threshold=0.7): self.llm = llm self.retriever = retriever self.threshold = confidence_threshold def answer(self, query: str) -> str: # First pass: attempt without retrieval response, logprobs = self.llm.generate_with_logprobs(query) avg_confidence = mean([exp(lp) for lp in logprobs]) if avg_confidence >= self.threshold: return response # Model is confident; skip retrieval # Second pass: retrieve and regenerate docs = self.retriever.retrieve(query, top_k=5) context = "\n".join([d.text for d in docs]) return self.llm.generate(f"{context}\n\nQuestion: {query}")

FLARE: Forward-Looking Active Retrieval

FLARE (Jiang et al., 2023) takes a generative-first approach: the model generates a sentence at a time and monitors its own confidence. When token probability drops below a threshold mid-generation, FLARE pauses, formulates a retrieval query from the low-confidence span, retrieves relevant passages, and regenerates that sentence with the new context.

FLARE’s sentence-level granularity is particularly effective for long-form generation (reports, summaries) where different parts of the output require different source documents. Unlike single-pass RAG, FLARE can retrieve different documents for different sentences.

class FLARE: def generate(self, query: str, threshold: float = 0.5) -> str: output_sentences = [] context = query while not self.is_complete(context): # Generate next sentence with token probabilities sentence, token_probs = self.llm.generate_sentence(context) min_prob = min(token_probs) if min_prob < threshold: # Low confidence: use generated text as retrieval query retrieval_query = self.extract_query(sentence, token_probs) docs = self.retriever.retrieve(retrieval_query) # Regenerate sentence with retrieved context sentence, _ = self.llm.generate_sentence( context + "\n" + docs[0].text ) output_sentences.append(sentence) context += " " + sentence return " ".join(output_sentences)

Implementation: Unified Advanced RAG

In practice, you can compose these techniques into a unified pipeline. The following implementation combines adaptive gating, CRAG-style correction, and Self-RAG reflection in a single orchestrator:

from dataclasses import dataclass from enum import Enum class RetrievalDecision(Enum): SKIP = "skip" RETRIEVE = "retrieve" WEB_SEARCH = "web_search" @dataclass class GradedDocument: text: str relevance_score: float is_supported: bool class AdvancedRAGPipeline: def __init__(self, llm, retriever, evaluator, web_searcher): self.llm = llm self.retriever = retriever self.evaluator = evaluator self.web_searcher = web_searcher def run(self, query: str) -> str: # Phase 1: Adaptive gating decision = self.decide_retrieval(query) if decision == RetrievalDecision.SKIP: return self.llm.generate(query) # Phase 2: Retrieve and grade (CRAG-style) docs = self.retriever.retrieve(query, top_k=10) graded = self.grade_documents(query, docs) correct = [g for g in graded if g.relevance_score > 0.7] if not correct: web_docs = self.web_searcher.search(query) context = "\n".join([d.text for d in web_docs]) else: context = "\n".join([g.text for g in correct]) # Phase 3: Generate with self-reflection response = self.llm.generate(f"{context}\n\n{query}") support_score = self.evaluator.check_support(context, response) if support_score < 0.6: # Regenerate with stricter grounding prompt response = self.llm.generate( f"Answer ONLY using the context below.\n{context}\n\n{query}" ) return response
Takeaway: Advanced RAG is not a single technique but a family of strategies. Start with adaptive retrieval gating (lowest effort, highest ROI), then add CRAG correction, and finally Self-RAG reflection as your accuracy requirements increase.
