RAG at Scale: Caching, Streaming, Multi-Tenant

MLOps Series RAG Systems

Scaling Challenges

Moving a RAG prototype to production exposes bottlenecks invisible at small scale. A single query involves an embedding call (~50 ms), a vector search (~20 ms), context assembly (~5 ms), and an LLM generation call (~2–8 s). At 100 queries per second, the LLM becomes the dominant bottleneck: you need roughly 200–800 concurrent connections to an LLM provider, and each connection consumes memory for the full context window.

Beyond latency, production RAG must address: cost (LLM calls at $3–15 per million output tokens), data freshness (indexing lag between document updates and searchability), tenant isolation (ensuring customer A never sees customer B's documents), and observability (tracing why a specific answer was generated).

Cost trap: At 1,000 QPS with an average of 4K context tokens per query, you burn through ~345 billion tokens per day. Without caching, this costs $1M+/month on GPT-4o pricing.

Semantic Caching

Semantic caching stores LLM responses keyed by the semantic meaning of the query, not its exact text. Two queries like "What is our refund policy?" and "How do I get a refund?" are semantically equivalent and should return the same cached response. This is achieved by embedding queries and looking up the cache using cosine similarity with a threshold (typically 0.95–0.98).

import redis, numpy as np, json, hashlib class SemanticCache: def __init__(self, redis_url: str, embed_fn, threshold: float = 0.96): self.r = redis.Redis.from_url(redis_url) self.embed = embed_fn self.threshold = threshold self.index_key = "rag:cache:index" def _cosine_sim(self, a: np.ndarray, b: np.ndarray) -> float: return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))) def get(self, query: str, tenant_id: str) -> str | None: """Look up cached response by semantic similarity.""" q_emb = self.embed(query) # Scan tenant-scoped cache entries pattern = f"rag:cache:{tenant_id}:*" best_score, best_key = 0.0, None for key in self.r.scan_iter(match=pattern): entry = json.loads(self.r.get(key)) cached_emb = np.array(entry["embedding"]) sim = self._cosine_sim(q_emb, cached_emb) if sim > best_score: best_score, best_key = sim, key if best_score >= self.threshold and best_key: entry = json.loads(self.r.get(best_key)) return entry["response"] return None def put(self, query: str, response: str, tenant_id: str, ttl: int = 3600): q_emb = self.embed(query) key = f"rag:cache:{tenant_id}:{hashlib.sha256(query.encode()).hexdigest()[:16]}" entry = {"query": query, "response": response, "embedding": q_emb.tolist()} self.r.setex(key, ttl, json.dumps(entry))

In production, the linear scan over cache entries is replaced by Redis Vector Search (RediSearch) or a dedicated ANN index. With 100K cached entries, ANN lookup takes <5 ms versus >500 ms for a brute-force scan. Cache hit rates of 30–60% are common in customer support RAG deployments, cutting LLM costs proportionally.

Invalidation: Cache entries must be invalidated when the underlying documents change. Tag each cache entry with the document IDs used in retrieval, and flush tagged entries on document update.

Streaming Responses

Users expect sub-second time-to-first-token (TTFT). Streaming the LLM response while it generates — rather than waiting for the full completion — reduces perceived latency from 3–8 seconds to under 500 ms. The architecture involves: an SSE (Server-Sent Events) connection from the client, a streaming LLM call in the backend, and a proxy that forwards tokens as they arrive.

from fastapi import FastAPI from fastapi.responses import StreamingResponse import openai, asyncio app = FastAPI() async def stream_rag_response(query: str, context: str): """Yield tokens as SSE events from the LLM stream.""" stream = await openai.chat.completions.create( model="gpt-4o", messages=[ {"role": "system", "content": f"Answer using this context:\n{context}"}, {"role": "user", "content": query}, ], stream=True, ) async for chunk in stream: delta = chunk.choices[0].delta.content if delta: yield f"data: {delta}\n\n" yield "data: [DONE]\n\n" @app.get("/query") async def query_endpoint(q: str, tenant: str): # 1. Check cache cached = cache.get(q, tenant) if cached: return {"answer": cached, "cached": True} # 2. Retrieve context context = await retrieve(q, tenant) # 3. Stream response return StreamingResponse( stream_rag_response(q, context), media_type="text/event-stream" )

A subtlety: when streaming, you cannot cache the response until it completes. The pattern is to accumulate tokens in a buffer while streaming, then write the full response to the cache after the stream ends. This adds a small async post-processing step.

Multi-Tenant Architecture

Multi-tenancy in RAG means multiple customers share the same infrastructure while seeing only their own documents. There are three isolation models:

Namespace Isolation

All tenants share one vector index, but each document is tagged with a tenant_id metadata field. Queries are filtered at search time using metadata filters. Cheapest to operate, but filter performance degrades with many tenants.

Collection Isolation

Each tenant gets a dedicated collection (index) in the vector database. Better search performance and stronger isolation, but higher memory overhead — each collection has its own HNSW graph. Works well up to ~1,000 tenants.

class TenantRouter: """Route queries to tenant-specific vector collections.""" def __init__(self, qdrant_client): self.client = qdrant_client self._tenant_collections = {} def ensure_collection(self, tenant_id: str): if tenant_id not in self._tenant_collections: collection_name = f"rag_{tenant_id}" self.client.create_collection( collection_name=collection_name, vectors_config={"size": 1536, "distance": "Cosine"}, ) self._tenant_collections[tenant_id] = collection_name def search(self, query_vec: list, tenant_id: str, top_k: int = 10): self.ensure_collection(tenant_id) return self.client.search( collection_name=self._tenant_collections[tenant_id], query_vector=query_vec, limit=top_k, )
Security critical: Never rely solely on application-level filtering for tenant isolation. Always validate the tenant_id at the API gateway level and include it in all database queries as a non-optional parameter.

Load Balancing

RAG workloads are bursty — a support team may generate 500 queries in a minute after a product incident. Load balancing must handle this by distributing requests across a pool of RAG workers, each with its own LLM connection pool.

The preferred strategy is weighted least-connections: route each new query to the worker with the fewest in-flight LLM requests. This naturally balances uneven processing times caused by variable context lengths (a 2K token context generates faster than a 12K token context).

import asyncio from dataclasses import dataclass, field @dataclass class Worker: url: str in_flight: int = 0 max_concurrent: int = 50 class RAGLoadBalancer: def __init__(self, workers: list[Worker]): self.workers = workers self._lock = asyncio.Lock() async def get_worker(self) -> Worker: async with self._lock: available = [w for w in self.workers if w.in_flight < w.max_concurrent] if not available: raise RuntimeError("All workers saturated") best = min(available, key=lambda w: w.in_flight) best.in_flight += 1 return best async def release_worker(self, worker: Worker): async with self._lock: worker.in_flight -= 1

Production Architecture

A battle-tested production RAG stack combines all the above layers into a coherent system. Here is a reference architecture that handles 1,000+ QPS with sub-2-second P95 latency:

Ingestion pipeline: Documents → chunker → embedding model → vector DB (Qdrant/Pinecone) + metadata store (PostgreSQL). Runs as an async job queue (Celery/Temporal) so uploads never block queries.

Query pipeline: API Gateway (validates tenant, rate-limits) → Semantic Cache (Redis + ANN) → Load Balancer → RAG Worker (embed query → vector search → rerank → assemble context → stream LLM response) → Logging/Tracing (OpenTelemetry → Datadog).

# docker-compose.yml (simplified) # services: # gateway: nginx + lua for auth/rate-limit # cache: redis-stack with vector search module # worker-1..N: FastAPI RAG workers (auto-scaled by CPU) # qdrant: vector database (3-node cluster) # postgres: metadata, tenants, document registry # temporal: ingestion workflow orchestrator # otel: OpenTelemetry collector → Datadog class ProductionRAG: def __init__(self): self.cache = SemanticCache("redis://cache:6379", embed_fn) self.router = TenantRouter(qdrant_client) self.lb = RAGLoadBalancer(workers) self.reranker = CrossEncoderReranker("BAAI/bge-reranker-v2-m3") async def handle_query(self, query: str, tenant_id: str): # Step 1: Cache check cached = self.cache.get(query, tenant_id) if cached: return {"answer": cached, "latency_ms": 5} # Step 2: Retrieve + rerank q_vec = embed(query) results = self.router.search(q_vec, tenant_id, top_k=20) reranked = self.reranker.rerank(query, results, top_k=5) # Step 3: Generate via load-balanced worker worker = await self.lb.get_worker() try: answer = await stream_to_worker(worker, query, reranked) finally: await self.lb.release_worker(worker) # Step 4: Cache the result self.cache.put(query, answer, tenant_id) return {"answer": answer}
Monitoring essentials: Track cache hit rate, P50/P95/P99 latency, retrieval recall@5, LLM token usage per query, and per-tenant usage. Alert on cache hit rate drops (indicates stale cache or query distribution shift).

Horizontal scaling is straightforward: add more RAG workers behind the load balancer. The vector DB scales independently (Qdrant supports sharded clusters). The semantic cache scales by adding Redis replicas. The key architectural principle is that every component is stateless except the databases, making auto-scaling trivial with Kubernetes HPA.
