RAG Evaluation: Faithfulness, Relevance, RAGAS

MLOps Series RAG Systems

Building a RAG pipeline is straightforward; knowing whether it actually works is not. Without rigorous evaluation, teams ship systems that hallucinate, retrieve irrelevant context, or give correct answers for wrong reasons. This post covers the metrics, frameworks, and implementation patterns for evaluating RAG end-to-end.

Why Evaluate RAG?

RAG evaluation is fundamentally harder than evaluating a standalone LLM because you must assess two independent components (retriever and generator) and their interaction. A correct answer might come from irrelevant context (lucky generation), and a wrong answer might occur despite perfect retrieval (generation failure).

The four core evaluation dimensions map to different parts of the pipeline:

  • Context Precision: Are the retrieved documents actually relevant? (Retriever quality)
  • Context Recall: Did we retrieve all the necessary information? (Retriever coverage)
  • Faithfulness: Is the answer grounded in the retrieved context? (Generation grounding)
  • Answer Relevancy: Does the answer address the user’s question? (End-to-end quality)

Faithfulness: Measuring Hallucination

Faithfulness quantifies whether every claim in the generated answer can be traced back to the retrieved context. It is the single most important metric for production RAG systems because unfaithful answers erode user trust.

The standard approach to computing faithfulness involves three steps:

  1. Claim extraction: Decompose the generated answer into atomic claims (individual factual statements).
  2. Entailment verification: For each claim, check whether the retrieved context entails (supports) that claim using an NLI model or LLM judge.
  3. Score aggregation: Faithfulness = (number of supported claims) / (total number of claims).
from openai import OpenAI class FaithfulnessEvaluator: def __init__(self, model="gpt-4o"): self.client = OpenAI() self.model = model def extract_claims(self, answer: str) -> list[str]: # Decompose answer into atomic factual claims response = self.client.chat.completions.create( model=self.model, messages=[{ "role": "user", "content": f"""Extract all atomic factual claims from this text. Return one claim per line. Text: {answer}""" }] ) claims = response.choices[0].message.content.strip().split("\n") return [c.strip() for c in claims if c.strip()] def verify_claim(self, claim: str, context: str) -> bool: # Check if context entails this claim response = self.client.chat.completions.create( model=self.model, messages=[{ "role": "user", "content": f"""Does the context support this claim? Answer YES or NO. Context: {context} Claim: {claim}""" }] ) return "YES" in response.choices[0].message.content.upper() def score(self, answer: str, context: str) -> float: claims = self.extract_claims(answer) if not claims: return 1.0 supported = sum(1 for c in claims if self.verify_claim(c, context)) return supported / len(claims)
LLM-as-judge pitfall: Using the same model for generation and evaluation creates bias. Use a different model (or NLI classifier) for faithfulness scoring, or ensemble multiple judges and take the conservative estimate.

Relevance Metrics: Precision, Recall, Answer Relevancy

Retrieval quality metrics measure how well the retriever serves the generator. They are adapted from information retrieval but applied in the RAG context where ground truth is the set of passages needed to answer a question.

Context Precision

Measures the proportion of retrieved documents that are actually relevant. Formally: Context Precision = |Relevant ∩ Retrieved| / |Retrieved|. High precision means less noise in the context window. Computed by having an LLM judge rate each retrieved passage as relevant or not, then averaging.

Context Recall

Measures the proportion of ground-truth relevant passages that were successfully retrieved: Context Recall = |Relevant ∩ Retrieved| / |Relevant|. Low recall means the generator is missing critical information. This metric requires a ground-truth answer or set of reference passages to compute.

Answer Relevancy

Measures whether the generated answer addresses the original question. Unlike faithfulness (which checks grounding), answer relevancy checks completeness and topicality. Computed by generating synthetic questions from the answer and measuring cosine similarity with the original question — a relevant answer should reproduce similar questions.

import numpy as np from sentence_transformers import SentenceTransformer class RelevanceMetrics: def __init__(self, embedder_name="all-MiniLM-L6-v2"): self.embedder = SentenceTransformer(embedder_name) self.llm = OpenAI() def context_precision(self, query: str, contexts: list[str]) -> float: # Judge each context passage as relevant or not relevant_count = 0 for ctx in contexts: resp = self.llm.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": f"Is this passage relevant to the query?\n" f"Query: {query}\nPassage: {ctx}\n" f"Answer YES or NO."}] ) if "YES" in resp.choices[0].message.content.upper(): relevant_count += 1 return relevant_count / len(contexts) if contexts else 0.0 def answer_relevancy(self, query: str, answer: str, n_gen=3) -> float: # Generate synthetic questions from the answer synthetic_qs = [] for _ in range(n_gen): resp = self.llm.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": f"Generate a question that this answer responds to:\n{answer}"}] ) synthetic_qs.append(resp.choices[0].message.content) # Compute similarity between original and synthetic questions q_emb = self.embedder.encode([query]) s_embs = self.embedder.encode(synthetic_qs) similarities = np.dot(s_embs, q_emb.T).flatten() return float(np.mean(similarities))

RAGAS Framework

RAGAS (Retrieval Augmented Generation Assessment) is the most widely adopted open-source framework for RAG evaluation. It provides automated, reference-free metrics that use LLMs as judges, eliminating the need for expensive human annotation.

RAGAS computes four core metrics out of the box:

  • Faithfulness (0–1): Claim-level grounding verification against context
  • Answer Relevancy (0–1): Synthetic question similarity scoring
  • Context Precision (0–1): Ranked relevance of retrieved passages
  • Context Recall (0–1): Coverage of ground-truth information (requires reference answer)
from ragas import evaluate from ragas.metrics import ( faithfulness, answer_relevancy, context_precision, context_recall ) from datasets import Dataset # Prepare evaluation dataset eval_data = { "question": [ "What is the capital of France?", "How does photosynthesis work?", ], "answer": [ "The capital of France is Paris.", "Plants convert CO2 and water into glucose using sunlight.", ], "contexts": [ ["Paris is the capital and largest city of France."], ["Photosynthesis converts carbon dioxide and water to glucose."], ], "ground_truth": [ "Paris is the capital of France.", "Photosynthesis uses sunlight to convert CO2 and water to glucose and oxygen.", ], } dataset = Dataset.from_dict(eval_data) # Run evaluation results = evaluate( dataset, metrics=[faithfulness, answer_relevancy, context_precision, context_recall], ) print(results) # {'faithfulness': 0.95, 'answer_relevancy': 0.88, # 'context_precision': 1.0, 'context_recall': 0.85}
RAGAS tip: Run evaluation on at least 50–100 diverse queries to get stable metric estimates. Single-query scores have high variance due to LLM judge stochasticity.

Framework Comparison

RAGAS

  • Reference-free metrics (no ground truth needed for most)
  • LLM-as-judge approach
  • Excellent integration with LangChain & LlamaIndex
  • Active open-source community

DeepEval

  • 14+ metrics including bias and toxicity
  • Pytest integration for CI/CD
  • Built-in dataset synthesis
  • Cloud dashboard for tracking

TruLens

  • Real-time monitoring with feedback functions
  • Chain-of-thought evaluation traces
  • Provider-agnostic (OpenAI, Hugging Face, Bedrock)
  • Production dashboard

Custom LLM Judge

  • Full control over evaluation criteria
  • Domain-specific rubrics
  • No framework dependency
  • Highest flexibility, most engineering effort

Custom Evaluation Pipeline

Framework metrics are a starting point, but production systems need domain-specific evaluation. Common customizations include:

  • Domain-specific faithfulness: In medical or legal domains, even partial hallucination is unacceptable. Use stricter claim verification with domain ontologies.
  • Citation accuracy: Does the answer correctly attribute statements to specific source documents?
  • Latency-adjusted quality: A slightly less accurate answer in 200ms may be better than a perfect answer in 5 seconds.
  • Retrieval diversity: Are we retrieving from multiple sources, or always returning the same few documents?
from dataclasses import dataclass @dataclass class EvalResult: faithfulness: float relevancy: float context_precision: float citation_accuracy: float latency_ms: float class CustomRAGEvaluator: def __init__(self, faithfulness_eval, relevance_eval, citation_eval): self.faithfulness_eval = faithfulness_eval self.relevance_eval = relevance_eval self.citation_eval = citation_eval def evaluate_single(self, query, answer, contexts, sources) -> EvalResult: import time start = time.time() faith = self.faithfulness_eval.score(answer, "\n".join(contexts)) relev = self.relevance_eval.answer_relevancy(query, answer) ctx_p = self.relevance_eval.context_precision(query, contexts) cite = self.citation_eval.verify_citations(answer, sources) elapsed = (time.time() - start) * 1000 return EvalResult(faith, relev, ctx_p, cite, elapsed) def evaluate_batch(self, test_cases: list) -> dict: results = [self.evaluate_single(**tc) for tc in test_cases] return { "avg_faithfulness": mean([r.faithfulness for r in results]), "avg_relevancy": mean([r.relevancy for r in results]), "avg_context_precision": mean([r.context_precision for r in results]), "avg_citation_accuracy": mean([r.citation_accuracy for r in results]), "p95_latency_ms": sorted([r.latency_ms for r in results])[ int(0.95 * len(results)) ], }

Implementation: CI/CD Evaluation Pipeline

The real power of RAG evaluation is running it automatically on every code change, embedding model update, or prompt modification. Integrate evaluation into your CI/CD pipeline to catch regressions before they reach production.

import json from pathlib import Path class RAGEvalPipeline: # Minimum thresholds for CI/CD gating THRESHOLDS = { "faithfulness": 0.85, "answer_relevancy": 0.80, "context_precision": 0.75, "context_recall": 0.70, } def __init__(self, rag_pipeline, evaluator, test_suite_path: str): self.rag = rag_pipeline self.evaluator = evaluator self.test_suite = self.load_test_suite(test_suite_path) def load_test_suite(self, path: str) -> list[dict]: with open(path) as f: return json.load(f) def run_evaluation(self) -> dict: # Generate answers for all test queries for case in self.test_suite: result = self.rag.query(case["question"]) case["answer"] = result.answer case["contexts"] = result.contexts # Compute metrics metrics = self.evaluator.evaluate_batch(self.test_suite) # Gate: fail CI if below thresholds failures = [] for metric, threshold in self.THRESHOLDS.items(): if metrics.get(metric, 0) < threshold: failures.append(f"{metric}: {metrics[metric]:.3f} < {threshold}") if failures: raise AssertionError(f"RAG quality gate failed:\n" + "\n".join(failures)) return metrics
Best practice: Maintain a golden test suite of 50–200 question-answer pairs with verified ground truth. Update it quarterly as your domain evolves. This is your regression safety net.
