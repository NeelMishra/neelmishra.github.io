Query Transform: HyDE, Multi-Query, Step-Back

MLOps Series RAG Systems

The quality of a RAG system is fundamentally limited by retrieval quality, and retrieval quality starts with the query. Users write ambiguous, incomplete, or poorly-phrased queries. Query transformation techniques rewrite the user's original query into one or more forms that are more effective for retrieval. This post covers four techniques — HyDE, Multi-Query, Step-Back Prompting, and Query Decomposition — with complete Python implementations.

Why Transform Queries?

There is a fundamental asymmetry between queries and documents. Documents are written to explain concepts in detail. Queries are short, often vague, and may use different vocabulary than the documents they need to find. This query-document mismatch is the root cause of many retrieval failures.

Consider these query failure modes that transformation solves:

  • Vocabulary mismatch: User says "fix slow RAG" but documents say "optimize retrieval latency"
  • Under-specification: "How does caching work?" — caching in what context?
  • Multi-hop: "Is HNSW faster than IVF for my 10M vector dataset?" — requires retrieving info about both algorithms plus dataset sizing guidelines
  • Abstraction mismatch: User asks about a specific error but the relevant document discusses the general concept
When to use query transformation: Not every query needs transformation. Short, precise keyword queries often work fine with BM25. Use transformation for ambiguous, multi-faceted, or complex queries — typically detected by query length, question marks, or low first-stage retrieval confidence.

HyDE — Hypothetical Document Embeddings

HyDE (Gao et al., 2022) is a counterintuitive but effective technique. Instead of embedding the query directly, you ask an LLM to generate a hypothetical answer to the query, then embed that hypothetical document and use it for retrieval. The intuition: the generated document is in the "language of documents" (detailed, explanatory) rather than the "language of queries" (short, vague), so it naturally aligns better with real documents in embedding space.

The process: (1) user query → (2) LLM generates hypothetical answer → (3) embed the hypothetical answer → (4) retrieve real documents similar to the hypothetical.

# HyDE implementation from openai import OpenAI from sentence_transformers import SentenceTransformer import numpy as np client = OpenAI() encoder = SentenceTransformer("all-MiniLM-L6-v2") HYDE_PROMPT = """Write a detailed paragraph that would answer the following question. Do not include any preamble. Just write the answer as if it were from a textbook. Question: {query} Answer:""" def generate_hypothetical_doc(query: str) -> str: """Generate a hypothetical document using an LLM.""" response = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": HYDE_PROMPT.format(query=query)}], temperature=0.7, max_tokens=256, ) return response.choices[0].message.content def hyde_retrieve(query: str, doc_embeddings, documents, top_k=10): """Retrieve using HyDE: embed a hypothetical answer, not the query.""" hypo_doc = generate_hypothetical_doc(query) hypo_emb = encoder.encode([hypo_doc], normalize_embeddings=True) scores = np.dot(doc_embeddings, hypo_emb.T).flatten() top_ids = np.argsort(scores)[::-1][:top_k] return [(documents[i], float(scores[i])) for i in top_ids]
HyDE limitations: (1) Adds LLM latency (~200-500ms). (2) If the LLM hallucinates, the hypothetical document may pull retrieval in the wrong direction. (3) Works best for factual questions; less effective for navigational queries like "show me the API docs for X".

Multi-Query Retrieval

Multi-Query generates multiple reformulations of the original query, retrieves documents for each reformulation, and unions the results. Different phrasings of the same question surface different relevant documents, improving overall recall.

For example, the query "How to optimize vector search performance?" might be expanded into:

  • "What are the best practices for fast approximate nearest neighbor search?"
  • "How to tune HNSW and IVF index parameters for low latency?"
  • "Techniques to improve vector database query throughput"
# Multi-Query implementation from openai import OpenAI from typing import List client = OpenAI() MULTI_QUERY_PROMPT = """You are an AI assistant helping to improve document retrieval. Given the user's question, generate 3 alternative versions of the question that capture different aspects or use different vocabulary. Return each question on a separate line, numbered 1-3. Original question: {query} Alternative questions:""" def generate_multi_queries(query: str, n: int = 3) -> List[str]: """Generate n alternative query formulations.""" response = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": MULTI_QUERY_PROMPT.format(query=query)}], temperature=0.8, max_tokens=200, ) text = response.choices[0].message.content # Parse numbered lines queries = [] for line in text.strip().split("\n"): cleaned = line.strip().lstrip("0123456789.)").strip() if cleaned: queries.append(cleaned) return queries[:n] def multi_query_retrieve(query, retriever, top_k=10): """Retrieve using multiple query formulations and union results.""" queries = [query] + generate_multi_queries(query) all_results = {} for q in queries: results = retriever.search(q, top_k=top_k) for doc_id, score in results: if doc_id not in all_results: all_results[doc_id] = score else: all_results[doc_id] = max(all_results[doc_id], score) # Return top-k by best score across all queries sorted_results = sorted(all_results.items(), key=lambda x: x[1], reverse=True) return sorted_results[:top_k]

Multi-Query works especially well with RRF fusion: instead of taking the max score across queries, produce a ranked list from each query and apply reciprocal rank fusion. This is more robust to score scale differences between queries.

Step-Back Prompting

Step-Back Prompting (Zheng et al., 2023) abstracts the specific question into a more general one before retrieval. The insight is that specific questions often need context from general documents, and direct keyword matching on the specific question misses these foundational documents.

Example: "Why does my HNSW index with M=16 have low recall at ef_search=32?" → Step-back: "How do HNSW parameters M and ef_search affect search recall?"

# Step-Back Prompting implementation from openai import OpenAI client = OpenAI() STEP_BACK_PROMPT = """You are an expert at reformulating questions. Given a specific question, generate a more general "step-back" question that captures the broader concept needed to answer the original question. The step-back question should be about the underlying principle or concept, not about the specific instance. Specific question: {query} Step-back question:""" def step_back_query(query: str) -> str: """Generate a more abstract step-back question.""" response = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": STEP_BACK_PROMPT.format(query=query)}], temperature=0.3, max_tokens=100, ) return response.choices[0].message.content.strip() def step_back_retrieve(query, retriever, top_k=10): """Retrieve using both original and step-back queries.""" abstract_query = step_back_query(query) # Retrieve for both queries original_results = retriever.search(query, top_k=top_k) abstract_results = retriever.search(abstract_query, top_k=top_k) # Merge with RRF original_ranking = [doc_id for doc_id, _ in original_results] abstract_ranking = [doc_id for doc_id, _ in abstract_results] return reciprocal_rank_fusion( [original_ranking, abstract_ranking], k=60, top_n=top_k )
When step-back shines: Debugging questions ("Why does X fail?"), troubleshooting queries, and questions about specific configurations. The step-back question retrieves the foundational documentation that explains why things work the way they do.

Query Decomposition

Query Decomposition breaks a complex multi-part question into atomic sub-questions, retrieves documents for each, and synthesizes the final answer from all retrieved context. This is essential for questions that require information from multiple documents.

Example: "Compare HNSW and IVF-PQ for a 50M vector dataset with 768 dimensions and a 200ms latency budget" decomposes into:

  • Q1: "What is the memory footprint of HNSW for 50M vectors at 768 dimensions?"
  • Q2: "What is the memory footprint of IVF-PQ for 50M vectors at 768 dimensions?"
  • Q3: "What is the query latency of HNSW at 50M scale?"
  • Q4: "What is the query latency of IVF-PQ at 50M scale?"
# Query Decomposition implementation from openai import OpenAI from typing import List client = OpenAI() DECOMPOSE_PROMPT = """Break down the following complex question into 2-5 simpler sub-questions that, when answered together, fully address the original question. Each sub-question should be self-contained and answerable independently. Return each sub-question on a separate line, numbered. Complex question: {query} Sub-questions:""" def decompose_query(query: str) -> List[str]: """Decompose a complex query into simpler sub-queries.""" response = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": DECOMPOSE_PROMPT.format(query=query)}], temperature=0.3, max_tokens=300, ) text = response.choices[0].message.content sub_queries = [] for line in text.strip().split("\n"): cleaned = line.strip().lstrip("0123456789.)").strip() if cleaned: sub_queries.append(cleaned) return sub_queries def decomposed_retrieve(query, retriever, top_k_per_sub=5): """Retrieve for each sub-query and merge results.""" sub_queries = decompose_query(query) all_rankings = [] for sq in sub_queries: results = retriever.search(sq, top_k=top_k_per_sub) ranking = [doc_id for doc_id, _ in results] all_rankings.append(ranking) # Fuse all sub-query rankings return reciprocal_rank_fusion(all_rankings, k=60, top_n=10)

Unified Query Transformation Pipeline

In production, you may want to select the transformation strategy dynamically based on query characteristics. Here is a unified pipeline that routes queries to the appropriate strategy:

from enum import Enum from typing import List, Tuple class TransformStrategy(Enum): NONE = "none" HYDE = "hyde" MULTI_QUERY = "multi_query" STEP_BACK = "step_back" DECOMPOSE = "decompose" class QueryTransformPipeline: def __init__(self, retriever, reranker=None): self.retriever = retriever self.reranker = reranker def classify_query(self, query: str) -> TransformStrategy: """Heuristic-based strategy selection.""" words = query.split() if len(words) <= 3: return TransformStrategy.NONE if any(w in query.lower() for w in ["compare", "versus", "vs", "difference"]): return TransformStrategy.DECOMPOSE if any(w in query.lower() for w in ["why", "debug", "error", "fail"]): return TransformStrategy.STEP_BACK if len(words) > 10: return TransformStrategy.MULTI_QUERY return TransformStrategy.HYDE def retrieve(self, query: str, strategy: TransformStrategy = None, top_k=10): if strategy is None: strategy = self.classify_query(query) if strategy == TransformStrategy.NONE: results = self.retriever.search(query, top_k=top_k) elif strategy == TransformStrategy.HYDE: results = hyde_retrieve(query, self.retriever.doc_embeddings, self.retriever.corpus, top_k) elif strategy == TransformStrategy.MULTI_QUERY: results = multi_query_retrieve(query, self.retriever, top_k) elif strategy == TransformStrategy.STEP_BACK: results = step_back_retrieve(query, self.retriever, top_k) elif strategy == TransformStrategy.DECOMPOSE: results = decomposed_retrieve(query, self.retriever, top_k) # Optional: apply reranker on top if self.reranker: docs = [self.retriever.corpus[r[0]] for r in results] results = self.reranker.rerank(query, docs, top_k) return results

Best For Short / Vague Queries

HyDE: Generates a detailed hypothetical answer that aligns with document embedding space. Use when the query is too short to retrieve effectively.

Best For Broad Queries

Multi-Query: Generates diverse reformulations to surface different relevant documents. Use for broad topics with many potential angles.

Best For Debugging / Why

Step-Back: Abstracts to the general principle. Use when the user asks about a specific instance but needs foundational context.

Best For Multi-Part Questions

Decomposition: Breaks complex questions into atomic sub-queries. Use for comparison questions or multi-faceted requests.

Cost-aware deployment: Every transformation technique adds an LLM call (~200-500ms, ~$0.001-0.005 per query). In latency-critical applications, consider caching transformed queries for common patterns, or running transformation asynchronously while the base retrieval executes. Monitor transformation quality — if the LLM produces poor rewrites, the entire retrieval degrades.

Query transformation is a surprisingly high-leverage improvement for RAG systems. In our experience, adding HyDE or Multi-Query to an existing pipeline improves answer quality by 10-20% on ambiguous queries, at the cost of one additional LLM call. The key is to apply transformation selectively — not every query needs it, and the overhead is only justified when the base retrieval would otherwise fail.
