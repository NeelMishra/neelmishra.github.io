    Containerizing ML Models: Docker

    MLOps Series Model Registry

    Why Containerize ML Models?

    The "it works on my machine" problem is amplified in ML. A model depends not only on code but on specific versions of Python, CUDA, cuDNN, NumPy, PyTorch, and a constellation of other packages — each with their own compatibility matrix. Containers solve this by packaging the entire runtime environment into a portable, reproducible unit.

    • Reproducibility — The exact same container image runs identically in development, CI/CD, staging, and production. No more debugging environment differences.
    • Dependency isolation — Model A can use PyTorch 2.1 with CUDA 12.1 while Model B uses TensorFlow 2.15 with CUDA 11.8, running side by side on the same host.
    • Portability — Deploy to any orchestrator: Kubernetes, ECS, Cloud Run, or bare metal. The container is the universal deployment unit.
    • Consistent environments — Training and serving use the same base image, eliminating the risk of training-serving skew from mismatched package versions.
    Key insight: A Docker image is an immutable artifact — the perfect complement to model versioning. Tag your image with the model version (fraud-model:v2.1.0) and you have a single deployable unit that includes model weights, serving code, and the complete runtime environment.

    Dockerfile Basics for ML

    A Dockerfile for ML serving follows the same principles as any Dockerfile, but the choice of base image and dependency installation strategy is critical for both image size and runtime correctness.

    Choosing a Base Image

    The base image decision sets your image's floor size and capability ceiling:

    • python:3.11-slim — ~120 MB. Good for CPU-only models (scikit-learn, XGBoost, small PyTorch CPU).
    • python:3.11 — ~920 MB. Includes build tools. Use when packages need compilation (numpy, scipy from source).
    • nvidia/cuda:12.1.1-runtime-ubuntu22.04 — ~3.5 GB. For GPU inference. Includes CUDA runtime but not development tools.
    • nvidia/cuda:12.1.1-devel-ubuntu22.04 — ~5.8 GB. For GPU training. Includes CUDA compiler and headers.
    # Basic Dockerfile for a scikit-learn model served with FastAPI FROM python:3.11-slim # Set working directory WORKDIR /app # Install system dependencies (if needed for packages like scipy) RUN apt-get update && \ apt-get install -y --no-install-recommends libgomp1 && \ rm -rf /var/lib/apt/lists/* # Copy and install Python dependencies first (layer caching) COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # Copy application code and model COPY app/ ./app/ COPY models/fraud_detector.joblib ./models/ # Expose port and set entrypoint EXPOSE 8080 ENTRYPOINT ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8080"]
    Warning: Always copy requirements.txt and install dependencies before copying your application code. Docker caches each layer — if you change your code but not your dependencies, the pip install layer is reused from cache, saving minutes per build.

    Multi-Stage Builds

    Multi-stage builds separate the build environment (where you compile packages and download dependencies) from the runtime environment (the lean image that runs in production). This technique can reduce image sizes by 50-80%.

    The pattern: use a full Python image to install everything, then copy only the installed packages and your code into a slim runtime image.

    # ─── Stage 1: Builder ──────────────────────────────────────── FROM python:3.11 AS builder WORKDIR /build # Install build-time dependencies COPY requirements.txt . RUN pip install --no-cache-dir --prefix=/install -r requirements.txt # ─── Stage 2: Runtime ──────────────────────────────────────── FROM python:3.11-slim AS runtime WORKDIR /app # Copy only the installed packages from builder stage COPY --from=builder /install /usr/local # Install minimal system libraries RUN apt-get update && \ apt-get install -y --no-install-recommends libgomp1 && \ rm -rf /var/lib/apt/lists/* # Copy application code and model artifact COPY app/ ./app/ COPY models/ ./models/ # Non-root user for security RUN useradd -m appuser USER appuser EXPOSE 8080 ENTRYPOINT ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8080"]

    Single-Stage Build

    Image size: ~2.4 GB

    Includes compilers, headers, pip cache, build artifacts. Larger attack surface.

    Multi-Stage Build

    Image size: ~650 MB

    Only runtime libraries, installed packages, and app code. Minimal attack surface.

    Tip: For even smaller images, consider distroless base images from Google (gcr.io/distroless/python3). They contain only the language runtime — no shell, no package manager — reducing the attack surface to near zero.

    GPU Support

    GPU-accelerated inference requires the NVIDIA Container Toolkit (formerly nvidia-docker) on the host machine and a CUDA-compatible base image. The container does not bundle GPU drivers — it uses the host's drivers via the toolkit.

    CUDA Base Images

    NVIDIA provides three image variants:

    • base — CUDA runtime only. Smallest image. Use when your framework bundles its own CUDA libraries.
    • runtime — Includes cuBLAS, cuDNN, and other runtime libraries. Most common choice for inference.
    • devel — Includes compilers, headers, and development tools. Needed for building custom CUDA kernels.
    # GPU-enabled Dockerfile for PyTorch inference # ─── Stage 1: Builder ──────────────────────────────────────── FROM nvidia/cuda:12.1.1-devel-ubuntu22.04 AS builder RUN apt-get update && apt-get install -y python3 python3-pip COPY requirements-gpu.txt . RUN pip3 install --no-cache-dir --prefix=/install -r requirements-gpu.txt # ─── Stage 2: Runtime ──────────────────────────────────────── FROM nvidia/cuda:12.1.1-runtime-ubuntu22.04 AS runtime RUN apt-get update && \ apt-get install -y --no-install-recommends python3 python3-pip && \ rm -rf /var/lib/apt/lists/* COPY --from=builder /install /usr/local WORKDIR /app COPY app/ ./app/ COPY models/ ./models/ ENV NVIDIA_VISIBLE_DEVICES=all ENV NVIDIA_DRIVER_CAPABILITIES=compute,utility EXPOSE 8080 ENTRYPOINT ["python3", "-m", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8080"]

    Runtime Flags

    To give a container GPU access, pass the --gpus flag at runtime:

    # Run with all available GPUs $ docker run --gpus all -p 8080:8080 fraud-model:v2.1.0-gpu # Run with a specific GPU $ docker run --gpus '"device=0"' -p 8080:8080 fraud-model:v2.1.0-gpu # Kubernetes: use nvidia.com/gpu resource requests # resources: # limits: # nvidia.com/gpu: 1
    Warning: CUDA version in your container image must be compatible with the GPU driver on the host. CUDA 12.1 requires driver ≥ 530. Always document the minimum driver version in your model card and Dockerfile comments.

    Image Optimization

    ML Docker images can balloon to 10+ GB without careful optimization. Every gigabyte slows down CI/CD pipelines, container pulls, and auto-scaling cold starts. These techniques keep images lean:

    .dockerignore

    Exclude files that should never enter the build context:

    # .dockerignore — keep these OUT of the Docker build context .git/ __pycache__/ *.pyc data/raw/ notebooks/ tests/ .env *.log wandb/ mlruns/ .vscode/ .idea/

    Layer Optimization Strategies

    • pip install --no-cache-dir — Prevents pip from caching downloaded wheels inside the image. Saves 200-500 MB for ML stacks.
    • Combine RUN commands — Each RUN creates a layer. Combine related commands with && and clean up in the same layer to avoid retaining intermediate files.
    • Order layers by change frequency — Put rarely-changing layers (system deps, Python deps) before frequently-changing layers (app code). This maximizes cache hits.
    • Remove build artifacts — Delete apt lists, pip caches, and temporary files in the same RUN command that created them.
    # Optimized RUN commands — combine & clean in one layer RUN apt-get update && \ apt-get install -y --no-install-recommends \ libgomp1 \ curl && \ rm -rf /var/lib/apt/lists/* && \ pip install --no-cache-dir -r requirements.txt && \ # Remove any stale bytecode find /usr/local -name '*.pyc' -delete && \ find /usr/local -name '__pycache__' -type d -exec rm -rf {} +
    Tip: Use docker image history <image> to inspect layer sizes and identify the biggest contributors. Often, a single pip install layer accounts for 70%+ of the image.

    Serving from Containers

    A production ML serving container goes beyond just loading a model and exposing a predict endpoint. It needs health checks, graceful shutdown, proper concurrency handling, and structured logging.

    from fastapi import FastAPI, HTTPException from pydantic import BaseModel import joblib import numpy as np import signal import sys import logging logging.basicConfig(level=logging.INFO, format="%(asctime)s %(levelname)s %(message)s") logger = logging.getLogger(__name__) app = FastAPI(title="Fraud Detector", version="2.1.0") # Load model at startup (once, not per request) model = joblib.load("models/fraud_detector.joblib") logger.info("Model loaded successfully") class PredictRequest(BaseModel): features: list[float] class PredictResponse(BaseModel): prediction: int probability: float model_version: str = "v2.1.0" @app.get("/health") def health_check(): # Kubernetes liveness/readiness probe endpoint return {"status": "healthy", "model_loaded": model is not None} @app.post("/predict", response_model=PredictResponse) def predict(request: PredictRequest): try: X = np.array(request.features).reshape(1, -1) pred = int(model.predict(X)[0]) proba = float(model.predict_proba(X)[0][1]) return PredictResponse(prediction=pred, probability=proba) except Exception as e: logger.error(f"Prediction failed: {e}") raise HTTPException(status_code=422, detail=str(e)) # Graceful shutdown handler def shutdown_handler(signum, frame): logger.info("Received shutdown signal, draining requests...") sys.exit(0) signal.signal(signal.SIGTERM, shutdown_handler)

    Running with Gunicorn

    For production, run Uvicorn workers behind Gunicorn for process-level concurrency:

    # Production entrypoint with Gunicorn + Uvicorn workers ENTRYPOINT ["gunicorn", "app.main:app", \ "--worker-class", "uvicorn.workers.UvicornWorker", \ "--workers", "4", \ "--bind", "0.0.0.0:8080", \ "--timeout", "120", \ "--graceful-timeout", "30", \ "--access-logfile", "-"]
    Tip: Set the number of Gunicorn workers to 2 × CPU_CORES + 1 for CPU-bound inference. For GPU inference, use a single worker with async request handling to avoid GPU memory contention.

    Best Practices

    These battle-tested practices keep your ML containers secure, maintainable, and production-ready:

    Security

    • Non-root user — Always create and switch to a non-root user. Running as root inside a container is a security anti-pattern that many container security scanners will flag.
    • Pin all versions — Pin your base image tag (python:3.11.7-slim, not python:3.11-slim), pin every package in requirements.txt, and pin system packages. Unpinned versions cause non-reproducible builds.
    • Security scanning — Run docker scout, Trivy, or Snyk on every image before pushing to registry. Automate this in CI/CD as a blocking gate.

    CI/CD Pipeline

    # CI/CD pipeline pseudo-code for ML container builds def build_and_push_pipeline(model_name, version, registry_url): tag = f"{registry_url}/{model_name}:{version}" latest = f"{registry_url}/{model_name}:latest" # Step 1: Build with cache run(f"docker build --cache-from {latest} -t {tag} -t {latest} .") # Step 2: Security scan (fail on HIGH/CRITICAL) result = run(f"docker scout cves {tag} --only-severity high,critical") if result.returncode != 0: raise ValueError("Security vulnerabilities detected — blocking push") # Step 3: Smoke test — start container and hit health endpoint run(f"docker run -d --name test -p 8080:8080 {tag}") wait_for_healthy("http://localhost:8080/health", timeout=30) run("docker stop test && docker rm test") # Step 4: Push to container registry run(f"docker push {tag}") run(f"docker push {latest}") # Step 5: Update model registry with image reference update_registry(model_name, version, container_image=tag)

    Summary Checklist

    1. Use multi-stage builds to minimize image size
    2. Pin all base images and dependency versions
    3. Run as a non-root user
    4. Include health check endpoints
    5. Implement graceful shutdown (handle SIGTERM)
    6. Use .dockerignore to exclude unnecessary files
    7. Scan for security vulnerabilities in CI/CD
    8. Tag images with model version for traceability
    9. Log structured JSON for observability platforms
    10. Document GPU driver requirements in Dockerfile comments
    Warning: Never embed secrets (API keys, database credentials, cloud tokens) in your Docker image. Use environment variables, mounted secret files, or a secrets manager (AWS Secrets Manager, HashiCorp Vault) at runtime. Secrets in layers persist in the image history even if deleted in later layers.
