    Model Versioning Strategies

    MLOps Series Model Registry

    Why Version Models?

    In traditional software, version control is non-negotiable. Yet many ML teams still overwrite model files, lose track of which checkpoint produced last week's predictions, and struggle to reproduce results when something breaks in production. Model versioning brings the same discipline to machine learning that Git brought to code.

    There are four core motivations for rigorous model versioning:

    • Reproducibility — Given a version identifier, you can recreate the exact model binary, its training data snapshot, and the code that produced it.
    • Rollback — When a new deployment degrades performance, you need a one-command path back to the last known-good version.
    • Auditing — Regulated industries (finance, healthcare) require an immutable audit trail of every model that touched production data.
    • Compliance — Standards like the EU AI Act mandate traceability from prediction back to training provenance.
    Key insight: A model version is not just a file — it is a snapshot of code + data + environment + hyperparameters frozen at a point in time. If any of those change, a new version must be created.

    Semantic Versioning for ML

    Semantic Versioning (SemVer) uses a MAJOR.MINOR.PATCH scheme. Adapting it for ML requires redefining what each component means in the context of models rather than APIs:

    • MAJOR — Breaking changes: input/output schema modification, feature column removal, target variable redefinition, or architecture overhaul that requires downstream consumers to update.
    • MINOR — Backward-compatible improvements: retrained on new data with better metrics, new features added without removing old ones, hyperparameter tuning that improves accuracy.
    • PATCH — Trivial fixes: configuration tweaks, logging changes, metadata corrections, no retraining involved.

    🔴 MAJOR bump (v1.x → v2.0)

    • Input features renamed or removed
    • Output format changed (e.g., single score → multi-label)
    • Model architecture swap (XGBoost → Transformer)
    • Target variable redefined

    🟢 MINOR bump (v2.0 → v2.1)

    • Retrained on newer data slice
    • Hyperparameter optimization applied
    • New features added alongside existing ones
    • Accuracy improved without schema change
    class MLSemVer: # Semantic versioning adapted for ML models def __init__(self, major=1, minor=0, patch=0): self.major = major self.minor = minor self.patch = patch def bump_major(self, reason="schema_change"): # Breaking change: schema, architecture, or target redefinition self.major += 1 self.minor = 0 self.patch = 0 return self.__str__() def bump_minor(self, reason="retrained"): # Backward-compatible improvement self.minor += 1 self.patch = 0 return self.__str__() def bump_patch(self, reason="config_fix"): # Trivial fix, no retraining self.patch += 1 return self.__str__() def __str__(self): return f"v{self.major}.{self.minor}.{self.patch}"
    Warning: SemVer alone does not capture data drift or distribution shifts. A model retrained on fundamentally different data might deserve a MAJOR bump even if the schema is unchanged, because downstream consumers may see dramatically different behavior.

    Version Naming Conventions

    Beyond SemVer, teams use several naming strategies to uniquely identify model versions. The choice depends on your workflow, tooling, and traceability needs.

    Common Schemes

    • Timestamp-based — fraud-detector-20240315T142030Z — sortable, human-readable, but no semantic meaning.
    • Git hash suffix — fraud-detector-v2.1.0-a3f8c2d — ties the model to the exact code commit that produced it.
    • Auto-increment — fraud-detector-42 — simple, monotonic, works well with registry tools like MLflow that auto-assign run IDs.
    • Composite — fraud-detector-v2.1.0-20240315-a3f8c2d — combines SemVer, date, and commit hash for maximum traceability.
    import datetime import subprocess def generate_version_tag(model_name, semver, scheme="composite"): # Generate a version tag using different naming conventions timestamp = datetime.datetime.utcnow().strftime("%Y%m%dT%H%M%SZ") git_hash = subprocess.check_output( ["git", "rev-parse", "--short", "HEAD"] ).decode().strip() if scheme == "timestamp": return f"{model_name}-{timestamp}" if scheme == "git_hash": return f"{model_name}-{semver}-{git_hash}" if scheme == "auto_increment": next_id = get_next_run_id(model_name) return f"{model_name}-{next_id}" if scheme == "composite": date_part = timestamp[:8] return f"{model_name}-{semver}-{date_part}-{git_hash}" # Example outputs: # "fraud-detector-20240315T142030Z" # "fraud-detector-v2.1.0-a3f8c2d" # "fraud-detector-42" # "fraud-detector-v2.1.0-20240315-a3f8c2d"
    Recommendation: Use the composite scheme for production models. It gives you semantic meaning (SemVer), temporal ordering (date), and exact code traceability (git hash) in one identifier.

    Lineage Tracking

    Lineage tracking connects every model version to the full chain of inputs that produced it: data version, code commit, training configuration, and hyperparameters. Without lineage, a version number is just a label with no substance.

    from dataclasses import dataclass, field from typing import Dict, Optional # A lineage record ties every model version to its provenance @dataclass class ModelLineage: model_name: str model_version: str data_version: str # DVC hash or dataset version tag code_commit: str # git SHA that produced the model training_config: Dict # full hyperparameter dictionary framework: str # e.g., "pytorch-2.1.0" parent_version: Optional[str] = None # which version this was derived from def to_metadata(self) -> Dict: return { "model_name": self.model_name, "model_version": self.model_version, "data_version": self.data_version, "code_commit": self.code_commit, "training_config": self.training_config, "framework": self.framework, "parent_version": self.parent_version, } def validate_lineage(self) -> bool: # Verify all provenance fields are populated required = [self.model_name, self.model_version, self.data_version, self.code_commit] return all(required) and len(self.training_config) > 0
    Tip: Store lineage metadata inside the registry alongside the model artifact. This way the lineage is never separated from the model, even if your tracking server goes down.

    Version Metadata

    Each registered model version should carry rich metadata that enables comparison, debugging, and governance. Think of metadata as the model's passport — it tells you everything about where the model came from and how it performs.

    Essential Metadata Fields

    • Performance metrics — accuracy, precision, recall, F1, AUC-ROC, RMSE (task-dependent)
    • Training parameters — learning rate, batch size, epochs, optimizer, regularization
    • Environment info — Python version, package versions, CUDA version, hardware (GPU type, RAM)
    • Data statistics — number of training samples, feature count, class distribution, data hash
    • Timing — training duration, inference latency (p50, p95, p99)
    • Signatures — input/output schema (column names, types, shapes)
    def collect_version_metadata(model, X_test, y_test, train_params, start_time): import time, platform, pkg_resources, hashlib, json import numpy as np predictions = model.predict(X_test) proba = model.predict_proba(X_test)[:, 1] # Compute performance metrics metrics = { "accuracy": accuracy_score(y_test, predictions), "precision": precision_score(y_test, predictions), "recall": recall_score(y_test, predictions), "f1": f1_score(y_test, predictions), "auc_roc": roc_auc_score(y_test, proba), } # Capture environment details environment = { "python_version": platform.python_version(), "os": platform.platform(), "packages": { p.key: p.version for p in pkg_resources.working_set }, } # Measure inference latency latencies = [] for _ in range(100): t0 = time.perf_counter() model.predict(X_test[:1]) latencies.append(time.perf_counter() - t0) return { "metrics": metrics, "parameters": train_params, "environment": environment, "data_stats": { "n_samples": len(X_test), "n_features": X_test.shape[1], }, "latency_ms": { "p50": round(np.percentile(latencies, 50) * 1000, 2), "p95": round(np.percentile(latencies, 95) * 1000, 2), "p99": round(np.percentile(latencies, 99) * 1000, 2), }, "training_duration_sec": time.time() - start_time, }

    Branching Strategies

    Just like code has branches, model registries often maintain multiple active versions serving different purposes simultaneously. Understanding branching strategies lets you safely experiment without disrupting production.

    Champion / Challenger

    The champion is the current production model. A challenger is a candidate version being evaluated in shadow mode — it receives the same traffic but its predictions are only logged, not served. When the challenger consistently outperforms the champion on live data, it gets promoted.

    A/B Versions

    Two or more versions serve live traffic simultaneously, each to a random fraction of users. This is essential for measuring real-world impact (e.g., click-through rate, conversion) rather than relying solely on offline metrics.

    Canary Versions

    A new version receives a tiny slice of traffic (e.g., 2%) while the old version handles the rest. If error rates or latency spike, the canary is automatically rolled back. Canary deployments are the safest way to introduce model changes.

    Champion / Challenger

    Challenger runs in shadow mode. Zero risk to users. Requires logging infrastructure to compare predictions.

    A/B Testing

    Both versions serve live traffic. Measures real business impact. Requires routing logic and statistical significance tracking.

    Warning: Never run A/B tests without a pre-defined stopping criterion and minimum sample size. Peeking at results early and making decisions leads to false positives and unreliable conclusions.

    Best Practices

    Model versioning is only as strong as the discipline around it. These best practices ensure your versioning strategy remains effective as your team and model portfolio grow.

    The Golden Rules

    1. Immutable versions — Once a version is registered, it must never be modified. Any change, no matter how small, creates a new version. This is the foundation of reproducibility.
    2. Never overwrite — Deleting or overwriting a version breaks audit trails and lineage chains. Mark deprecated versions as archived instead.
    3. Clear promotion paths — Define explicit stages: Development → Staging → Production → Archived. Every promotion requires passing validation gates (metric thresholds, approval sign-off).
    4. Automate version bumping — Integrate version computation into your CI/CD pipeline. Detect schema changes automatically and bump MAJOR; detect retraining and bump MINOR.
    5. Tag everything — Use labels like experiment:pricing-v3, team:fraud, dataset:q1-2024 to make versions searchable and filterable.
    def promote_model(registry, model_name, version, target_stage): # Promotion pipeline with validation gates model_info = registry.get_model_version(model_name, version) # Gate 1: Metric thresholds metrics = model_info.get_metrics() if metrics["auc_roc"] < 0.90: raise ValueError(f"AUC {metrics['auc_roc']} below threshold 0.90") # Gate 2: Lineage completeness lineage = model_info.get_lineage() if not lineage.validate_lineage(): raise ValueError("Incomplete lineage — cannot promote") # Gate 3: Approval (for Production stage) if target_stage == "Production": require_approval(model_name, version, approvers=["ml-lead"]) # Execute promotion registry.transition_model_version_stage( name=model_name, version=version, stage=target_stage, archive_existing_versions=True, ) log_event(f"Promoted {model_name} v{version} → {target_stage}")
    Tip: Combine automated gates (metric thresholds, lineage checks) with human approval for production promotions. Full automation is fine for Development → Staging, but the Staging → Production step benefits from a human-in-the-loop review.
