    ONNX: The Universal Model Format

    MLOps Series Model Registry

    What is ONNX?

    ONNX—Open Neural Network Exchange—is an open-source format for representing machine learning models as portable computation graphs. Originally launched in 2017 by Facebook and Microsoft, ONNX is now governed by the Linux Foundation and maintained by a broad consortium including AWS, NVIDIA, Intel, Huawei, and AMD.

    The core problem ONNX solves is framework lock-in. Without a standard interchange format, a model trained in PyTorch could only be served by PyTorch-compatible infrastructure. ONNX decouples training from inference: you train with whatever framework fits your research workflow, export to ONNX, and deploy on any runtime that speaks ONNX—ONNX Runtime, TensorRT, OpenVINO, CoreML, or dozens of others.

    ONNX is defined as a Protocol Buffer schema. The specification covers two domains: ONNX for deep learning / neural network operators and ONNX-ML for traditional machine learning operators (tree ensembles, SVMs, scalers, one-hot encoders). This dual-domain design means ONNX can represent everything from a ResNet-50 to an XGBoost classifier.

    Ecosystem scale: As of 2024, ONNX supports 200+ operators across 20 opset versions, with converters available for PyTorch, TensorFlow, scikit-learn, XGBoost, LightGBM, Keras, and SparkML.

    The ONNX Graph IR

    At its core, an ONNX model is a directed acyclic graph (DAG). The graph has three key components: nodes, tensors, and initializers.

    Nodes represent operations—Conv, MatMul, Relu, Softmax, BatchNormalization. Each node has an op_type, a list of input tensor names, a list of output tensor names, and a set of attributes (e.g., kernel size, padding mode, axis). Nodes do not hold data; they only describe computation.

    Tensors are the edges of the graph, carrying multi-dimensional arrays between nodes. Each tensor has a name, element type (float32, float16, int8, etc.), and a shape that can include symbolic dimensions for dynamic sizing.

    Initializers are constant tensors—model weights and biases. They are stored directly in the model file (or in external data files for models exceeding 2 GB). At runtime, initializers are loaded into memory before the first inference call.

    import onnx from onnx import TensorProto, helper # Build a minimal ONNX graph by hand: Y = ReLU(X @ W + B) X = helper.make_tensor_value_info("X", TensorProto.FLOAT, [None, 4]) Y = helper.make_tensor_value_info("Y", TensorProto.FLOAT, [None, 2]) # Weight initializer import numpy as np W_data = np.random.randn(4, 2).astype(np.float32) W = helper.make_tensor("W", TensorProto.FLOAT, [4, 2], W_data.flatten()) B_data = np.zeros(2, dtype=np.float32) B = helper.make_tensor("B", TensorProto.FLOAT, [2], B_data) # Nodes matmul = helper.make_node("MatMul", ["X", "W"], ["XW"]) add = helper.make_node("Add", ["XW", "B"], ["pre_relu"]) relu = helper.make_node("Relu", ["pre_relu"], ["Y"]) # Assemble graph and model graph = helper.make_graph([matmul, add, relu], "linear_relu", [X], [Y], initializer=[W, B]) model = helper.make_model(graph, opset_imports=[helper.make_opsetid("", 17)]) onnx.checker.check_model(model) onnx.save(model, "linear_relu.onnx")

    This hand-built example illustrates the graph's anatomy: three nodes (MatMul → Add → Relu) connected by named tensors, with weights and biases supplied as initializers.

    Opsets and Versioning

    An opset is a versioned collection of operator specifications. Each operator defines its name, inputs, outputs, attributes, type constraints, and semantic behavior. When a runtime loads an ONNX model, it checks the declared opset version and uses the corresponding operator implementations.

    ONNX uses monotonically increasing opset versions. Opset 1 was the initial release; opset 20 (ONNX 1.15, released late 2023) is the latest stable version. Each new opset may add new operators, extend existing ones with new attributes, or deprecate old behaviors—but it never removes operators that were previously defined.

    The opset version is declared per-model in the opset_import field. A model can import multiple opsets: the default ONNX domain plus custom domains for vendor-specific operators.

    ⚠ Version mismatch: If you export a model with opset 18 but deploy on a runtime that only supports up to opset 15, inference will fail. Always check runtime compatibility before choosing your opset. You can downgrade opsets using onnx.version_converter.
    import onnx from onnx import version_converter # Load a model exported with opset 17 model = onnx.load("model_opset17.onnx") # Downgrade to opset 13 for older runtime compatibility converted = version_converter.convert_version(model, 13) onnx.save(converted, "model_opset13.onnx") # Inspect the opset version of a model for opset in model.opset_import: print(f"Domain: '{opset.domain or 'default'}', Version: {opset.version}")

    Converting Models to ONNX

    The most common workflow is to train a model in PyTorch, TensorFlow, or scikit-learn and then export it to ONNX. Each framework has its own conversion pathway.

    PyTorch → ONNX

    PyTorch's built-in torch.onnx.export() traces the model with a dummy input and records the operations into an ONNX graph. You must specify input/output names and, for variable-length dimensions, dynamic axes.

    import torch import torchvision.models as models model = models.resnet18(pretrained=True) model.eval() dummy = torch.randn(1, 3, 224, 224) torch.onnx.export( model, dummy, "resnet18.onnx", input_names=["image"], output_names=["class_logits"], dynamic_axes={"image": {0: "batch_size"}, "class_logits": {0: "batch_size"}}, opset_version=17, do_constant_folding=True # optimize constant expressions )

    TensorFlow → ONNX

    The tf2onnx package converts SavedModel or Keras models to ONNX. It can be invoked from the command line or as a Python API.

    # Command line conversion # python -m tf2onnx.convert --saved-model ./my_tf_model --output model.onnx --opset 17 # Python API import tf2onnx import tensorflow as tf model = tf.keras.models.load_model("my_keras_model.h5") spec = (tf.TensorSpec((None, 784), tf.float32, name="input"),) onnx_model, _ = tf2onnx.convert.from_keras(model, input_signature=spec, opset=17) onnx.save(onnx_model, "keras_model.onnx")

    scikit-learn → ONNX

    from skl2onnx import convert_sklearn from skl2onnx.common.data_types import FloatTensorType from sklearn.ensemble import GradientBoostingClassifier model = GradientBoostingClassifier(n_estimators=200) model.fit(X_train, y_train) # Define input type: batch of 4-feature float vectors initial_type = [("input", FloatTensorType([None, 4]))] onnx_model = convert_sklearn(model, initial_types=initial_type, target_opset=17) onnx.save(onnx_model, "gb_classifier.onnx")

    ONNX Runtime

    ONNX Runtime (ORT) is Microsoft's high-performance inference engine for ONNX models. It supports multiple execution providers (EPs) that dispatch operations to different hardware backends: CPU (default), CUDA (NVIDIA GPUs), TensorRT (NVIDIA optimized), DirectML (Windows GPUs), OpenVINO (Intel), NNAPI (Android), and CoreML (Apple).

    ORT applies graph-level optimizations before execution: constant folding, dead-code elimination, operator fusion (e.g., fusing Conv + BatchNorm + Relu into a single kernel), and memory planning to minimize allocations during inference.

    import onnxruntime as ort import numpy as np # Create session with GPU acceleration providers = ["CUDAExecutionProvider", "CPUExecutionProvider"] session = ort.InferenceSession("resnet18.onnx", providers=providers) # Inspect inputs and outputs for inp in session.get_inputs(): print(f"Input: {inp.name}, Shape: {inp.shape}, Type: {inp.type}") for out in session.get_outputs(): print(f"Output: {out.name}, Shape: {out.shape}") # Run inference image = np.random.randn(1, 3, 224, 224).astype(np.float32) results = session.run(["class_logits"], {"image": image}) logits = results[0] # shape: (1, 1000) # Session options for optimization opts = ort.SessionOptions() opts.graph_optimization_level = ort.GraphOptimizationLevel.ORT_ENABLE_ALL opts.intra_op_num_threads = 4 opts.execution_mode = ort.ExecutionMode.ORT_PARALLEL fast_session = ort.InferenceSession("resnet18.onnx", opts, providers=providers)
    Performance tip: Enable ORT_ENABLE_ALL optimizations and set intra_op_num_threads to the number of physical cores (not logical). For CUDA, ensure your ONNX Runtime build matches your CUDA toolkit version.

    Validating ONNX Models

    Before deploying an ONNX model, you should validate its structure, verify shape inference, and optionally visualize the graph. ONNX provides built-in tools for all three.

    import onnx # 1. Structural validation — checks graph consistency model = onnx.load("model.onnx") onnx.checker.check_model(model) print("✓ Model passes structural validation") # 2. Shape inference — propagates shapes through the graph from onnx import shape_inference inferred = shape_inference.infer_shapes(model) # Inspect inferred shapes on intermediate tensors for vi in inferred.graph.value_info: dims = [d.dim_value if d.dim_value else d.dim_param for d in vi.type.tensor_type.shape.dim] print(f" {vi.name}: {dims}") # 3. Model metadata print(f"IR version: {model.ir_version}") print(f"Opset: {model.opset_import[0].version}") print(f"Graph name: {model.graph.name}") print(f"Nodes: {len(model.graph.node)}") print(f"Initializers:{len(model.graph.initializer)}")

    For visual inspection, tools like Netron (netron.app) render the ONNX graph as an interactive diagram, showing each node's op type, attributes, and tensor shapes. This is invaluable for debugging conversion issues where operators may have been fused or split differently than expected.

    Limitations and Gotchas

    ONNX is powerful but not without sharp edges. Understanding these limitations will save you hours of debugging.

    Dynamic shapes. While ONNX supports symbolic dimensions (e.g., batch size = "N"), some runtimes and optimization passes struggle with fully dynamic shapes. If your model accepts variable-length sequences, test carefully with the target runtime. You may need to pad inputs or specify shape ranges via runtime session options.

    Custom operations. If your PyTorch model uses a custom C++ extension or a non-standard TensorFlow op, it likely has no ONNX equivalent. You must either (a) register a custom ONNX operator and implement it in your target runtime, or (b) refactor the model to use only standard operations before export.

    ⚠ Silent correctness bugs: Conversion can succeed without errors but produce subtly different numerical results. Always compare ONNX model outputs against the original framework's outputs on identical inputs—check both absolute and relative tolerances. A tolerance of 1e-5 for float32 is a reasonable starting point.

    Large models. ONNX models larger than 2 GB cannot be stored in a single protobuf file (Protocol Buffers has a 2 GB limit). Use onnx.save with save_as_external_data=True to split weights into external files alongside the main .onnx graph file.

    import onnx # Save large model with external data model = onnx.load("large_model.onnx") onnx.save( model, "large_model_ext.onnx", save_as_external_data=True, all_tensors_to_one_file=True, location="weights.bin", size_threshold=1024 # tensors > 1KB go external )

    Opset coverage gaps. Not every framework operation maps cleanly to an ONNX operator. Complex operations like multi-head attention may be decomposed into many smaller ONNX nodes, potentially losing runtime-specific optimizations. Check the converter's output graph carefully and benchmark before committing to ONNX for latency-sensitive workloads.
