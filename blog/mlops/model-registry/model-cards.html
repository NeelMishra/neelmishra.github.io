    Model Cards: Documentation for Responsible ML

    MLOps Series Model Registry

    What Are Model Cards?

    Model cards are standardized documentation artifacts that accompany trained machine learning models. Introduced by Margaret Mitchell et al. at Google in their 2019 paper "Model Cards for Model Reporting," they serve as a transparency mechanism — a concise yet comprehensive report that communicates what a model does, how it performs, and where it should (and should not) be used.

    Think of a model card as a nutrition label for ML. Just as food labels inform consumers about ingredients and allergens, model cards inform stakeholders — engineers, product managers, regulators, end users — about a model's capabilities, limitations, and ethical considerations.

    Why it matters: Model cards are becoming a regulatory expectation. The EU AI Act requires documentation of high-risk AI systems covering intended purpose, limitations, and performance — which maps directly to model card sections.

    Model Card Template

    The standard model card template contains eight sections. Each section serves a distinct audience: engineers care about metrics and training data, product managers focus on intended use, and compliance teams prioritize ethical considerations.

    1. Model Details — Name, version, type (classification/regression/etc.), owner, date, framework, license.
    2. Intended Use — Primary use cases, intended users, and explicitly out-of-scope uses.
    3. Factors — Relevant demographic or environmental factors that affect performance (age groups, geographies, device types).
    4. Metrics — Which metrics were chosen, why, and how they were computed (including confidence intervals).
    5. Evaluation Data — Description of the evaluation dataset, preprocessing, and any known biases in the data.
    6. Training Data — Description of the training dataset, size, features, time range, and collection methodology.
    7. Ethical Considerations — Bias analysis, sensitive attributes, fairness metrics, and societal impact assessment.
    8. Caveats and Recommendations — Known failure modes, edge cases, deployment recommendations, and monitoring guidance.

    Documenting Intended Use

    The intended use section is arguably the most important part of a model card. It defines the boundary between responsible deployment and misuse. A well-written intended use section answers three questions:

    Primary Use Cases

    Describe the specific tasks the model is designed for. Be precise: instead of "fraud detection," write "scoring individual credit card transactions in real-time for US-issued Visa and Mastercard cards with transaction amounts between $1 and $10,000."

    Out-of-Scope Uses

    Explicitly state what the model should not be used for. This protects both users and your organization. Common out-of-scope uses include:

    • Applying a US-trained model to non-US markets without validation
    • Using a classification model's scores as calibrated probabilities
    • Deploying on data modalities the model was not trained on
    • Using as the sole decision-maker in high-stakes scenarios without human review

    User Personas

    Identify who will interact with the model's predictions and their technical sophistication:

    👩‍💻 ML Engineers

    Integrate model via API. Need input/output schemas, latency SLAs, and batch inference documentation.

    👨‍💼 Business Analysts

    Consume prediction outputs in dashboards. Need plain-language explanations of scores and thresholds.

    Warning: Vague intended use sections like "general-purpose NLP" provide no protection. If a model is misapplied and causes harm, a poorly documented intended use section means your organization cannot demonstrate due diligence.

    Metrics and Evaluation

    The metrics section goes beyond reporting a single accuracy number. A responsible model card presents performance across multiple dimensions: aggregate metrics, per-slice breakdowns, fairness metrics, and statistical confidence intervals.

    Performance Across Slices

    Aggregate metrics can mask disparate performance. A model with 95% overall accuracy might have 99% accuracy for one demographic group and 82% for another. Always report metrics broken down by relevant factors.

    def compute_sliced_metrics(y_true, y_pred, sensitive_attr, slices): # Compute metrics per demographic or factor slice results = {} for slice_name, mask in slices.items(): y_t = y_true[mask] y_p = y_pred[mask] results[slice_name] = { "n_samples": int(sum(mask)), "accuracy": accuracy_score(y_t, y_p), "precision": precision_score(y_t, y_p), "recall": recall_score(y_t, y_p), "fpr": false_positive_rate(y_t, y_p), } return results # Example: slice by age group slices = { "age_18_30": df["age"].between(18, 30), "age_31_50": df["age"].between(31, 50), "age_51_plus": df["age"] >= 51, } sliced = compute_sliced_metrics(y_true, y_pred, "age", slices)

    Fairness Metrics

    Beyond accuracy parity, report metrics like equalized odds (equal true positive and false positive rates across groups), demographic parity (equal positive prediction rates), and calibration (predicted probabilities match observed frequencies per group).

    def compute_fairness_metrics(y_true, y_pred, protected_groups): # Compute fairness metrics across protected groups group_metrics = {} for group_name, mask in protected_groups.items(): tpr = recall_score(y_true[mask], y_pred[mask]) fpr = false_positive_rate(y_true[mask], y_pred[mask]) pos_rate = y_pred[mask].mean() group_metrics[group_name] = { "tpr": tpr, "fpr": fpr, "positive_rate": pos_rate, } # Check equalized odds: max TPR difference < threshold tprs = [m["tpr"] for m in group_metrics.values()] equalized_odds_gap = max(tprs) - min(tprs) return { "per_group": group_metrics, "equalized_odds_gap": equalized_odds_gap, "passes_threshold": equalized_odds_gap < 0.05, }
    Tip: Always report confidence intervals alongside point estimates. A model with AUC = 0.92 ± 0.03 tells a very different story than AUC = 0.92 ± 0.15. Use bootstrap resampling with at least 1,000 iterations.

    Ethical Considerations

    The ethical considerations section is where you confront the real-world impact of your model. This is not a checkbox exercise — it requires genuine analysis of how the model's predictions could affect individuals and communities.

    Bias Analysis

    Document any known biases in the training data and their potential downstream effects. If historical data reflects systemic discrimination (e.g., lending decisions, hiring patterns), the model may learn and perpetuate those biases.

    Sensitive Attributes

    List which attributes were considered sensitive (race, gender, age, disability, religion) and how they were handled during training and evaluation. Options include:

    • Exclusion — Removed from features (but proxies may still leak information)
    • Fairness constraints — Applied during training to enforce equalized odds
    • Post-hoc calibration — Adjusted thresholds per group to equalize error rates

    Societal Impact

    Consider second-order effects. A highly accurate fraud model that disproportionately flags transactions from certain zip codes can effectively create financial exclusion zones — even if "zip code" is not explicitly a feature.

    Warning: "We removed sensitive attributes from the feature set" is not sufficient bias mitigation. Correlated features (zip code, school name, employer) can serve as proxies. Always test for disparate impact even after removing protected attributes.

    Automating Model Cards

    Manual model card creation does not scale. When your team trains dozens of models per week, automation is essential. The goal is to generate model cards from training metadata with minimal human effort, reserving human input for sections that require judgment (ethical considerations, caveats).

    Generation Pipeline

    The automation pipeline extracts metadata from your training run (metrics, parameters, data stats), fills in template sections, and produces a structured document. Sections requiring human input are flagged as [REQUIRES REVIEW].

    import json from datetime import datetime class ModelCardGenerator: def __init__(self, template_path="model_card_template.md"): self.template = open(template_path).read() def generate(self, run_metadata, manual_sections=None): # Auto-populate from training metadata card = { "model_details": { "name": run_metadata["model_name"], "version": run_metadata["version"], "framework": run_metadata["framework"], "date": datetime.utcnow().isoformat(), "owner": run_metadata.get("owner", "[REQUIRES REVIEW]"), }, "metrics": run_metadata.get("metrics", {}), "training_data": { "n_samples": run_metadata.get("n_train_samples"), "features": run_metadata.get("feature_names", []), "date_range": run_metadata.get("data_date_range"), }, "evaluation_data": { "n_samples": run_metadata.get("n_eval_samples"), "sliced_metrics": run_metadata.get("sliced_metrics", {}), }, } # Merge in human-authored sections if manual_sections: card.update(manual_sections) # Flag empty required sections for section in ["ethical_considerations", "intended_use", "caveats"]: if section not in card: card[section] = "[REQUIRES REVIEW]" return card def to_markdown(self, card): # Render card as Markdown for human review lines = [f"# Model Card: {card['model_details']['name']}"] for section, content in card.items(): lines.append(f"\n## {section.replace('_', ' ').title()}") if isinstance(content, dict): for k, v in content.items(): lines.append(f"- **{k}**: {v}") else: lines.append(str(content)) return "\n".join(lines)

    CI/CD Integration

    Add model card generation as a step in your training pipeline. After every registered model version, the pipeline auto-generates a card draft and opens a pull request for human review of the ethical and caveats sections.

    Tip: Use Google's model-card-toolkit for a production-grade solution. It integrates with TFX and ML Metadata, supports HTML rendering, and provides a JSON schema for structured storage.

    Let's walk through a complete model card for a transaction fraud classifier. This example demonstrates how each section comes together to create a transparent, actionable document.

    Key takeaway: A good model card is a living document. Update it when the model is retrained, when new limitations are discovered, or when deployment context changes. Automate what you can, and invest human effort where judgment matters most.
