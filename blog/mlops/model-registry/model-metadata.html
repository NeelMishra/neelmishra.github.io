    Model Metadata: What to Track

    MLOps Series Model Registry

    Why Track Metadata?

    Every ML model is the product of dozens of decisions: the training data selected, the features engineered, the hyperparameters tuned, the hardware used, and the evaluation criteria applied. Without structured metadata, these decisions live only in the memory of whoever ran the training job. When that person leaves the team—or simply forgets—the knowledge is lost.

    Comprehensive metadata tracking enables five critical capabilities:

    • Reproducibility — recreate any model by replaying its recorded configuration and data
    • Comparison — systematically compare models across experiments to identify what actually drove improvements
    • Debugging — when a model fails in production, metadata reveals what changed between the good and bad versions
    • Compliance — regulatory audits require documented evidence of how models were built and validated
    • Auditability — trace any prediction back to the exact model version, training data, and code that produced it
    Rule of Thumb: If you can't answer the question "Why does this model behave differently from the previous version?" in under five minutes, your metadata tracking is insufficient.

    Performance Metrics

    Performance metrics are the most obvious metadata to track, but most teams only record aggregate numbers. Production-grade metadata goes further: it captures per-class metrics, per-slice performance, confidence distributions, and calibration curves.

    from sklearn.metrics import ( accuracy_score, precision_recall_fscore_support, roc_auc_score, confusion_matrix ) import numpy as np def compute_comprehensive_metrics(y_true, y_pred, y_prob, class_names): """Compute aggregate and per-class metrics for model metadata.""" precision, recall, f1, support = precision_recall_fscore_support( y_true, y_pred, average=None ) metrics = { "aggregate": { "accuracy": float(accuracy_score(y_true, y_pred)), "macro_f1": float(np.mean(f1)), "weighted_f1": float(np.average(f1, weights=support)), "auc_roc": float(roc_auc_score(y_true, y_prob, multi_class="ovr")), }, "per_class": { name: { "precision": float(precision[i]), "recall": float(recall[i]), "f1": float(f1[i]), "support": int(support[i]) } for i, name in enumerate(class_names) }, "confusion_matrix": confusion_matrix(y_true, y_pred).tolist() } return metrics

    Per-slice metrics are essential for fairness and robustness. A model with 95% overall accuracy might have 70% accuracy on an underrepresented demographic group. Track performance across every meaningful data slice:

    def compute_slice_metrics(df, y_true_col, y_pred_col, slice_columns): """Compute metrics across data slices for fairness analysis.""" slice_results = {} for col in slice_columns: slice_results[col] = {} for value in df[col].unique(): mask = df[col] == value subset_true = df.loc[mask, y_true_col] subset_pred = df.loc[mask, y_pred_col] slice_results[col][str(value)] = { "accuracy": float(accuracy_score(subset_true, subset_pred)), "count": int(mask.sum()), "positive_rate": float(subset_pred.mean()) } return slice_results
    Warning: Aggregate metrics hide failure modes. A fraud detection model with 99.5% accuracy might catch only 40% of actual fraud because the class is extremely rare. Always track per-class and per-slice metrics alongside aggregates.

    Training Configuration

    Training configuration encompasses every parameter that affects how the model learns. This includes hyperparameters, optimizer settings, learning rate schedules, regularization, data augmentation, and early stopping criteria.

    def capture_training_config(model, optimizer, scheduler, args): """Capture full training configuration as structured metadata.""" config = { "model_architecture": { "type": model.__class__.__name__, "num_parameters": sum( p.numel() for p in model.parameters() ), "trainable_parameters": sum( p.numel() for p in model.parameters() if p.requires_grad ), }, "optimizer": { "type": optimizer.__class__.__name__, "lr": optimizer.defaults["lr"], "weight_decay": optimizer.defaults.get("weight_decay", 0), "betas": optimizer.defaults.get("betas", None), }, "scheduler": { "type": scheduler.__class__.__name__ if scheduler else None, }, "training": { "epochs": args.epochs, "batch_size": args.batch_size, "gradient_accumulation_steps": args.grad_accum, "max_grad_norm": args.max_grad_norm, "early_stopping_patience": args.patience, "seed": args.seed, }, "regularization": { "dropout": args.dropout, "label_smoothing": args.label_smoothing, } } return config
    Tip: Use a configuration management library like Hydra or OmegaConf to define training configs as YAML files. This makes the entire configuration automatically serializable and diff-able between runs.

    Data Versions

    Data versioning metadata answers the question: "What data did this model see during training?" This goes beyond just a dataset name—it includes the content hash, split ratios, preprocessing steps applied, and the exact feature columns used.

    import hashlib, json def capture_data_metadata(train_df, val_df, test_df, feature_cols, target_col): """Capture comprehensive data version metadata.""" def df_hash(df): return hashlib.sha256( df.to_csv(index=False).encode() ).hexdigest() total = len(train_df) + len(val_df) + len(test_df) metadata = { "splits": { "train": {"rows": len(train_df), "hash": df_hash(train_df)}, "val": {"rows": len(val_df), "hash": df_hash(val_df)}, "test": {"rows": len(test_df), "hash": df_hash(test_df)}, }, "split_ratios": { "train": round(len(train_df) / total, 3), "val": round(len(val_df) / total, 3), "test": round(len(test_df) / total, 3), }, "features": feature_cols, "target": target_col, "num_features": len(feature_cols), "class_distribution": train_df[target_col].value_counts().to_dict() } return metadata

    A robust data versioning strategy should capture these elements:

    • Content hash — SHA-256 of the dataset content, not the file path or timestamp
    • Split ratios and strategy — random, stratified, temporal, or group-based splitting
    • Preprocessing pipeline — ordered list of transformations (imputation, scaling, encoding) with parameters
    • Feature list — exact columns used, including any engineered features
    • Class distribution — label balance per split (critical for imbalanced datasets)
    • Data source — origin database, table, query, or file path with access timestamp

    Environment Information

    The same code and data can produce different models on different hardware or software stacks due to floating-point non-determinism, library version differences, and GPU driver behavior. Capturing environment metadata is essential for reproducing results.

    import platform, subprocess, os def capture_environment_metadata(): """Capture the full training environment specification.""" env = { "python": { "version": platform.python_version(), "implementation": platform.python_implementation(), }, "os": { "system": platform.system(), "release": platform.release(), "machine": platform.machine(), }, "hardware": { "cpu_count": os.cpu_count(), }, "packages": {} } # Capture key ML library versions for lib in ["torch", "tensorflow", "numpy", "sklearn", "pandas"]: try: mod = __import__(lib) env["packages"][lib] = mod.__version__ except ImportError: pass # Capture GPU information if available try: import torch if torch.cuda.is_available(): env["gpu"] = { "count": torch.cuda.device_count(), "name": torch.cuda.get_device_name(0), "memory_gb": round( torch.cuda.get_device_properties(0).total_mem / 1e9, 1 ), "cuda_version": torch.version.cuda, "cudnn_version": str(torch.backends.cudnn.version()), } except ImportError: pass return env

    Minimum Environment Metadata

    • Python version
    • Key library versions (torch, numpy, etc.)
    • OS and architecture
    • pip freeze output

    Production-Grade Metadata

    • All of the minimum, plus:
    • CUDA / cuDNN versions
    • GPU model and VRAM
    • Docker image hash
    • CPU count and RAM
    • NVIDIA driver version

    Metadata Schema

    A standardized schema ensures every model in your registry carries consistent, queryable metadata. Without a schema, metadata drifts across teams: one team logs "lr", another logs "learning_rate", and a third logs "base_lr". Define a schema once and enforce it at registration time.

    # model_metadata_schema.json { "$schema": "https://json-schema.org/draft/2020-12/schema", "title": "ModelMetadata", "type": "object", "required": ["model_name", "version", "metrics", "training_config", "data"], "properties": { "model_name": {"type": "string"}, "version": {"type": "string", "pattern": "^v\\d+\\.\\d+\\.\\d+$"}, "created_at": {"type": "string", "format": "date-time"}, "metrics": { "type": "object", "required": ["accuracy", "f1"], "properties": { "accuracy": {"type": "number", "minimum": 0, "maximum": 1}, "f1": {"type": "number", "minimum": 0, "maximum": 1}, "auc_roc": {"type": "number", "minimum": 0, "maximum": 1}, "precision": {"type": "number"}, "recall": {"type": "number"} } }, "training_config": { "type": "object", "required": ["learning_rate", "epochs", "batch_size"], "properties": { "learning_rate": {"type": "number"}, "epochs": {"type": "integer"}, "batch_size": {"type": "integer"}, "optimizer": {"type": "string"} } }, "data": { "type": "object", "required": ["dataset_hash", "num_samples"], "properties": { "dataset_hash": {"type": "string"}, "num_samples": {"type": "integer"}, "features": {"type": "array", "items": {"type": "string"}} } }, "environment": {"type": "object"} } }

    Enforce the schema at registration time using validation:

    import jsonschema, json def validate_model_metadata(metadata, schema_path): """Validate metadata against the standard schema before registration.""" with open(schema_path) as f: schema = json.load(f) try: jsonschema.validate(instance=metadata, schema=schema) print("Metadata validation passed") return True except jsonschema.ValidationError as e: print(f"Metadata validation FAILED: {e.message}") print(f" Path: {'.'.join(str(p) for p in e.path)}") raise
    Tip: Start with a minimal required schema and extend it over time. It's better to enforce a small set of required fields consistently than to define an exhaustive schema that teams ignore because it's too burdensome.

    Querying Metadata

    Metadata is only valuable if it's queryable. The ability to search models by performance, filter by dataset version, and compare configurations across runs transforms a model registry from a dumb storage layer into a knowledge base.

    class MetadataQueryEngine: """Query model metadata across the registry.""" def __init__(self, registry_client): self.registry = registry_client def find_best_model(self, model_name, metric, stage=None): """Find the model version with the highest value for a metric.""" versions = self.registry.get_all_versions(model_name) if stage: versions = [v for v in versions if v.stage == stage] best = max(versions, key=lambda v: v.metadata["metrics"].get(metric, 0)) return best def compare_versions(self, model_name, v1, v2): """Compare metadata between two model versions.""" meta1 = self.registry.get_metadata(model_name, v1) meta2 = self.registry.get_metadata(model_name, v2) diff = {} for key in set(meta1.keys()) | set(meta2.keys()): if meta1.get(key) != meta2.get(key): diff[key] = {"v1": meta1.get(key), "v2": meta2.get(key)} return diff def find_models_by_dataset(self, dataset_hash): """Find all model versions trained on a specific dataset.""" results = [] for model in self.registry.list_models(): for version in self.registry.get_all_versions(model.name): data_meta = version.metadata.get("data", {}) if data_meta.get("dataset_hash") == dataset_hash: results.append(version) return results def find_models_above_threshold(self, metric, threshold): """Find all models exceeding a metric threshold.""" results = [] for model in self.registry.list_models(): for v in self.registry.get_all_versions(model.name): score = v.metadata.get("metrics", {}).get(metric, 0) if score >= threshold: results.append({"model": model.name, "version": v.version, metric: score}) return sorted(results, key=lambda x: x[metric], reverse=True)
    Warning: Naive iteration over all models and versions (as shown above) doesn't scale. In production registries with thousands of models, use indexed database queries. MLflow supports mlflow.search_runs() with filter expressions; build similar indexing into custom registries.

    Common metadata queries every team should support:

    • "Show me all models trained on dataset v2.3 with F1 > 0.90"
    • "Compare hyperparameters between v2.0 and v2.1 of the fraud detector"
    • "Which models are affected if we retract dataset batch #47?"
    • "List all production models trained with PyTorch < 2.0 (security vulnerability)"
    • "Show the best model per team for Q1 2024 by AUC"
