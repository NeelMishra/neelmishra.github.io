    TorchScript and torch.compile

    MLOps Series Model Registry

    Why TorchScript?

    PyTorch's eager execution model makes development intuitive—you write standard Python, set breakpoints, and inspect tensors at any point. But this flexibility comes at a cost: eager execution relies on the Python interpreter, which means you cannot deploy PyTorch models without a Python runtime, and you miss out on whole-graph optimizations that require seeing the full computation ahead of time.

    TorchScript bridges this gap. It is a statically-typed subset of Python that PyTorch's JIT compiler can analyze, optimize, and execute independently of the Python runtime. A TorchScript model is a self-contained artifact containing both the computation graph and the learned weights, loadable in C++ via libtorch, on mobile via PyTorch Mobile, or in any environment where Python is unavailable or undesirable.

    The key benefits of TorchScript are: (1) Python-free deployment—no GIL, no interpreter overhead, (2) graph-level optimization—operator fusion, constant propagation, dead-code elimination, and (3) portability—a single .pt file runs on servers, mobile devices, and embedded systems.

    Note: With the introduction of torch.compile in PyTorch 2.0, TorchScript is no longer the only compilation path. We cover both approaches in this post so you can make an informed choice for your deployment scenario.

    Tracing with torch.jit.trace

    Tracing is the simpler of the two TorchScript mechanisms. You provide a model and a sample input; PyTorch executes the model in eager mode while recording every tensor operation into a static graph. The result is a ScriptModule that replays those exact operations for any future input.

    import torch import torch.nn as nn class ConvBlock(nn.Module): def __init__(self, in_ch, out_ch): super().__init__() self.conv = nn.Conv2d(in_ch, out_ch, 3, padding=1) self.bn = nn.BatchNorm2d(out_ch) def forward(self, x): return torch.relu(self.bn(self.conv(x))) class SimpleCNN(nn.Module): def __init__(self): super().__init__() self.block1 = ConvBlock(3, 32) self.block2 = ConvBlock(32, 64) self.pool = nn.AdaptiveAvgPool2d(1) self.fc = nn.Linear(64, 10) def forward(self, x): x = self.block1(x) x = self.block2(x) x = self.pool(x).flatten(1) return self.fc(x) model = SimpleCNN() model.eval() # Trace with a sample input sample = torch.randn(1, 3, 32, 32) traced = torch.jit.trace(model, sample) # Inspect the traced graph print(traced.graph) # Save the traced model traced.save("simple_cnn_traced.pt")

    Tracing works perfectly for models that are pure functions of their inputs—no data-dependent branching, no loops with variable trip counts, no Python-side randomness that affects the computation path.

    ⚠ Silent data-dependent branch loss: If your model's forward() contains if x.sum() > 0, tracing will record only the branch taken for the sample input. The other branch is permanently lost. PyTorch emits a warning, but it is easy to miss in a noisy training log.

    Scripting with torch.jit.script

    Scripting takes a fundamentally different approach: instead of running the model and recording operations, it parses the Python source code of the forward() method and compiles it into TorchScript IR. This means control flow, loops, and conditionals are preserved in the compiled graph.

    import torch import torch.nn as nn class DynamicNet(nn.Module): def __init__(self, num_layers: int): super().__init__() self.layers = nn.ModuleList( [nn.Linear(64, 64) for _ in range(num_layers)] ) self.output = nn.Linear(64, 10) def forward(self, x: torch.Tensor) -> torch.Tensor: # Data-dependent early exit — scripting preserves both branches if x.abs().max() < 1e-6: return torch.zeros(x.size(0), 10, device=x.device) for layer in self.layers: x = torch.relu(layer(x)) return self.output(x) model = DynamicNet(num_layers=4) model.eval() # Script — parses source code, preserves control flow scripted = torch.jit.script(model) print(scripted.code) # view the TorchScript source scripted.save("dynamic_net_scripted.pt") # Load without original class definition loaded = torch.jit.load("dynamic_net_scripted.pt") output = loaded(torch.randn(4, 64))

    Scripting requires that all code within forward() is compatible with the TorchScript Python subset. This subset supports: basic types (int, float, bool, str, Tensor), lists, tuples, dicts with uniform types, optional types, for-loops, while-loops, if/else, and most torch.* operations. It does not support arbitrary Python objects, generators, comprehensions with side effects, or dynamic attribute access.

    Type annotations matter: TorchScript's compiler uses type annotations to resolve types statically. Always annotate function signatures and local variables when they cannot be inferred. Missing annotations are the #1 source of scripting errors.

    Tracing vs Scripting

    The choice between tracing and scripting depends on your model's characteristics. Here is a side-by-side comparison:

    torch.jit.trace

    • Mechanism: Records ops from a sample execution
    • Control flow: ❌ Not captured; only one branch recorded
    • Type annotations: Not required
    • Python subset: No restrictions on source code
    • Debugging: Easy—just run the model normally
    • Use when: Model is a pure function with no branching
    • Examples: ResNet, VGG, standard CNNs, transformers without dynamic shapes

    torch.jit.script

    • Mechanism: Parses Python source into TorchScript IR
    • Control flow: ✅ Fully preserved (if/else, loops)
    • Type annotations: Required for all function signatures
    • Python subset: Must conform to TorchScript subset
    • Debugging: Harder—compiler errors can be cryptic
    • Use when: Model has data-dependent control flow
    • Examples: RNNs with early stopping, dynamic routing, tree models

    You can also combine both approaches: trace the feed-forward backbone and script only the modules that contain control flow. Use @torch.jit.export to expose additional methods beyond forward().

    torch.compile and TorchDynamo

    PyTorch 2.0 introduced torch.compile, a fundamentally new compilation paradigm that does not require TorchScript's static analysis. Instead, it uses TorchDynamo—a Python-level JIT that captures computation graphs from eager Python execution using CPython's frame evaluation API.

    The architecture works in three stages: (1) TorchDynamo intercepts Python bytecode execution, identifies sequences of PyTorch operations, and captures them as FX graphs; (2) AOTAutograd traces both the forward and backward passes to produce a joint graph; (3) a backend compiler (Inductor by default) compiles the graph into optimized machine code.

    import torch class TransformerBlock(torch.nn.Module): def __init__(self, d_model=512, nhead=8): super().__init__() self.attn = torch.nn.MultiheadAttention(d_model, nhead, batch_first=True) self.ffn = torch.nn.Sequential( torch.nn.Linear(d_model, 2048), torch.nn.GELU(), torch.nn.Linear(2048, d_model), ) self.norm1 = torch.nn.LayerNorm(d_model) self.norm2 = torch.nn.LayerNorm(d_model) def forward(self, x): attn_out, _ = self.attn(x, x, x) x = self.norm1(x + attn_out) x = self.norm2(x + self.ffn(x)) return x model = TransformerBlock().cuda() # One-line compilation — default backend is "inductor" compiled = torch.compile(model) # First call triggers compilation (slow), subsequent calls are fast x = torch.randn(8, 128, 512, device="cuda") out = compiled(x) # compiled kernel execution # Advanced: specify backend and mode compiled_max = torch.compile(model, backend="inductor", mode="max-autotune") # mode="max-autotune" tries more kernel configurations for best perf

    Unlike TorchScript, torch.compile handles graph breaks gracefully. When it encounters code it cannot compile (e.g., a call to a non-torch library), it falls back to eager Python for that segment and resumes compilation after. This means virtually any PyTorch model can benefit from torch.compile without modification.

    torch.compile vs TorchScript: For training speedups and GPU optimization, torch.compile is the modern choice. For Python-free deployment in C++ or mobile, TorchScript (or ONNX export) is still necessary—torch.compile does not produce a portable artifact.

    Optimization Techniques

    Both TorchScript and torch.compile enable several optimization passes that improve inference performance beyond what eager execution can achieve.

    Operator fusion combines multiple sequential operations into a single kernel call, reducing memory bandwidth usage and kernel launch overhead. For example, Conv → BatchNorm → ReLU can be fused into a single CUDA kernel. TorchScript's optimizer does this automatically, and Inductor (torch.compile's default backend) is even more aggressive.

    Constant propagation evaluates expressions involving constant tensors at compile time, removing them from the runtime graph. If a batch normalization layer's running mean and variance are frozen (eval mode), the normalization can be folded into the preceding convolution's weights.

    import torch # Freeze a traced model — folds constants into the graph model = SimpleCNN() model.eval() traced = torch.jit.trace(model, torch.randn(1, 3, 32, 32)) frozen = torch.jit.freeze(traced) # Run optimization passes torch.jit.optimize_for_inference(frozen) frozen.save("optimized_cnn.pt") # Quantization with torch.compile from torch.ao.quantization import quantize_dynamic quantized = quantize_dynamic(model, {torch.nn.Linear}, dtype=torch.qint8) compiled_quant = torch.compile(quantized) out = compiled_quant(torch.randn(1, 3, 32, 32))
    ⚠ Eval mode is critical: Always call model.eval() before tracing, scripting, or compiling for inference. Without it, batch normalization uses batch statistics instead of running statistics, dropout remains active, and optimization passes that assume frozen parameters will produce incorrect results.

    Memory planning analyzes tensor lifetimes in the graph and reuses memory allocations. In eager mode, each intermediate tensor gets its own allocation. With graph-level visibility, the compiler can overlap allocations for tensors whose lifetimes do not overlap, significantly reducing peak memory usage.

    Deployment with TorchScript

    The primary deployment advantage of TorchScript is the ability to load and run models in environments without Python. The C++ API mirrors the Python API closely.

    // C++ inference with libtorch // #include <torch/script.h> // Load the serialized TorchScript module // torch::jit::script::Module module = torch::jit::load("model.pt"); // module.eval(); // Create input tensor // auto input = torch::randn({1, 3, 224, 224}); // std::vector<torch::jit::IValue> inputs = {input}; // Run inference // auto output = module.forward(inputs).toTensor(); // std::cout << output.sizes() << std::endl;

    The C++ libtorch runtime provides the same tensor operations and autograd engine as PyTorch Python, but without the Python interpreter overhead. This is critical for latency-sensitive applications like real-time video processing, autonomous vehicles, and high-frequency trading.

    For mobile deployment, PyTorch provides the torch.utils.mobile_optimizer module that applies mobile-specific optimizations on top of TorchScript:

    from torch.utils.mobile_optimizer import optimize_for_mobile model = SimpleCNN() model.eval() traced = torch.jit.trace(model, torch.randn(1, 3, 32, 32)) # Apply mobile-specific optimizations optimized = optimize_for_mobile(traced) optimized._save_for_lite_interpreter("mobile_model.ptl") # The .ptl file can be loaded on iOS / Android # using PyTorch Mobile's Java / Objective-C API
    Deployment decision matrix: Use TorchScript for C++ or mobile deployment where no Python is available. Use torch.compile for GPU training speedups and GPU-based serving where Python is acceptable. Use ONNX export for cross-platform or non-PyTorch serving runtimes.

    For serving infrastructure, TorchScript models integrate with NVIDIA Triton Inference Server (via the PyTorch backend), TorchServe (Amazon's model serving framework), and custom gRPC/REST endpoints using libtorch. The key advantage over raw PyTorch checkpoints is that TorchScript artifacts are versioned, self-contained, and reproducible—essential properties for production model registries.
