    What is a Model Registry?

    A model registry is a centralized repository that stores, versions, and manages machine learning model artifacts throughout their lifecycle. Think of it as a specialized package manager—like npm or PyPI—but designed specifically for ML models, with first-class support for versioning, stage transitions, lineage tracking, and deployment automation.

    Without a registry, teams resort to ad-hoc solutions: models saved to shared drives with cryptic filenames (model_final_v2_FINAL_USE_THIS.pkl), no record of which training run produced which artifact, and no systematic way to promote a model from staging to production. A model registry eliminates this chaos by providing a single source of truth.

    The core capabilities every model registry must provide are: (1) artifact storage —persisting serialized model files with their dependencies, (2) versioning—tracking multiple versions of the same model with auto-incrementing identifiers, (3) lifecycle management—moving models through stages (staging, production, archived), and (4) metadata—linking each version to its training run, metrics, parameters, and data lineage.

    Registry ≠ Experiment tracker: An experiment tracker records training runs (hyperparameters, metrics, plots). A model registry manages the output of those runs—the deployable artifacts. MLflow uniquely provides both in a single platform.

    MLflow Registry Overview

    MLflow's Model Registry is one of four core components in the MLflow platform, alongside Tracking, Projects, and Models. It builds directly on top of MLflow Tracking: when you log a model during a training run, you can register it in the registry with a single API call, and the registry automatically links the registered model version back to the originating run.

    The registry supports model flavors—MLflow's abstraction for different serialization formats. A single registered model can have a python_function flavor (generic Python callable), a pytorch flavor (native PyTorch checkpoint), an onnx flavor, a sklearn flavor, and so on. Each flavor knows how to save and load the model in its native format, enabling framework-agnostic serving.

    The architecture consists of: (1) a backend store (SQL database) that persists model metadata, versions, and stage transitions, (2) an artifact store (filesystem, S3, GCS, Azure Blob) that holds the actual model files, and (3) a REST API / UI for managing models programmatically or visually.

    Registering Models

    There are multiple ways to register a model in MLflow. The most common is to log a model during a training run and then register it, either in a single step or as two separate operations.

    import mlflow import mlflow.sklearn from sklearn.ensemble import RandomForestClassifier # Method 1: Register during logging (single step) with mlflow.start_run() as run: model = RandomForestClassifier(n_estimators=100, max_depth=10) model.fit(X_train, y_train) # Log metrics mlflow.log_metric("accuracy", 0.95) mlflow.log_metric("f1_score", 0.93) # Log and register in one call mlflow.sklearn.log_model( model, "model", registered_model_name="fraud-detection-rf" )
    # Method 2: Register after logging (two steps) with mlflow.start_run() as run: mlflow.sklearn.log_model(model, "model") run_id = run.info.run_id # Register the logged model using its URI model_uri = f"runs:/{run_id}/model" result = mlflow.register_model(model_uri, "fraud-detection-rf") print(f"Version: {result.version}") # auto-incremented
    # Method 3: Auto-logging — MLflow logs the model automatically mlflow.autolog() with mlflow.start_run(): model = RandomForestClassifier(n_estimators=100) model.fit(X_train, y_train) # Model is automatically logged — register after # Model URIs follow the pattern: # runs:/{run_id}/{artifact_path} # models:/{model_name}/{version} # models:/{model_name}/{stage}
    Model URIs are powerful. You can reference a model by run ID (runs:/abc123/model), by version number (models:/fraud-detection-rf/3), or by stage (models:/fraud-detection-rf/Production). This enables deployment scripts that always pull the current production model without hardcoding a version.

    Model Lifecycle Stages

    Every model version in the MLflow registry exists in one of four lifecycle stages: None, Staging, Production, and Archived. These stages provide a structured workflow for promoting models from development to serving and eventually retirement.

    Stage 1 of 4
    None — the initial stage when a model version is first registered. The model exists in the registry but has not been evaluated or approved for any environment.

    Stage transitions are explicit API calls, not automatic. This ensures human oversight (or deliberate automation) at every promotion step. The transition is recorded in the registry's audit log, including the user, timestamp, and optional description.

    from mlflow.tracking import MlflowClient client = MlflowClient() # Transition version 3 from None to Staging client.transition_model_version_stage( name="fraud-detection-rf", version=3, stage="Staging", archive_existing_versions=False ) # After validation, promote to Production client.transition_model_version_stage( name="fraud-detection-rf", version=3, stage="Production", archive_existing_versions=True # auto-archive the old production version )
    ⚠ Deprecation note: The stage-based lifecycle (None/Staging/Production/Archived) is being superseded by model aliases in MLflow 2.x. Aliases like "champion" and "challenger" offer more flexibility. However, stage-based transitions remain widely used and fully supported.

    Model Versions

    Every time you register a model under the same name, MLflow automatically increments the version number. Version 1 is the first registration; subsequent registrations produce version 2, 3, and so on. Each version is an immutable snapshot—you cannot overwrite a version's artifact.

    from mlflow.tracking import MlflowClient client = MlflowClient() # List all versions of a model versions = client.search_model_versions("name='fraud-detection-rf'") for v in versions: print(f"Version {v.version} | Stage: {v.current_stage} | " f"Run: {v.run_id} | Status: {v.status}") # Get details of a specific version mv = client.get_model_version("fraud-detection-rf", version=3) print(f"Description: {mv.description}") print(f"Created: {mv.creation_timestamp}") print(f"Source: {mv.source}") # Add a description to a version client.update_model_version( name="fraud-detection-rf", version=3, description="Trained on Q4 data with feature engineering v2. AUC=0.97" ) # Compare metrics between versions via their linked runs for v in [2, 3]: mv = client.get_model_version("fraud-detection-rf", version=v) run = client.get_run(mv.run_id) auc = run.data.metrics.get("auc", "N/A") print(f"Version {v}: AUC = {auc}")

    The version lineage is key to reproducibility. Because each version links back to a specific MLflow run, you can always trace from a production model back to its exact training code, hyperparameters, training data hash, and metric history.

    Version-based References

    • models:/fraud-detection-rf/3 — exact version
    • Immutable, deterministic, ideal for rollbacks
    • Use in CI/CD pipelines and deployment configs

    Stage-based References

    • models:/fraud-detection-rf/Production — current production
    • Dynamic—resolves to whichever version holds the stage
    • Use in serving endpoints for auto-promotion

    Webhooks and Automation

    MLflow supports webhooks that fire HTTP callbacks when registry events occur—model creation, version registration, stage transitions, and more. This enables fully automated CI/CD pipelines triggered by model promotions.

    A typical automated workflow looks like this: (1) a data scientist registers a new model version, (2) a webhook fires on MODEL_VERSION_CREATED, triggering a CI pipeline, (3) the pipeline runs validation tests (accuracy checks, bias audits, latency benchmarks), (4) if all tests pass, the pipeline calls the MLflow API to transition the model to Staging, (5) another webhook fires on MODEL_VERSION_TRANSITIONED_TO_STAGING, triggering a canary deployment, (6) after monitoring confirms no degradation, a final API call promotes to Production.

    # Databricks MLflow webhook example (Databricks-hosted MLflow) from databricks_registry_webhooks import RegistryWebhooksClient, HttpUrlSpec client = RegistryWebhooksClient() # Create a webhook for stage transitions webhook = client.create_webhook( model_name="fraud-detection-rf", events=["MODEL_VERSION_TRANSITIONED_STAGE"], http_url_spec=HttpUrlSpec( url="https://ci.example.com/hooks/mlflow-stage-change", secret="webhook-secret-token" ), description="Trigger CI/CD on stage transition" ) # For open-source MLflow, use the REST API directly: # POST /api/2.0/mlflow/registry-webhooks/create # { # "model_name": "fraud-detection-rf", # "events": ["MODEL_VERSION_TRANSITIONED_STAGE"], # "http_url_spec": { # "url": "https://ci.example.com/hooks/mlflow", # "enable_ssl_verification": true # } # }
    GitOps pattern: Many teams use webhooks to update a Git repository's deployment manifest (e.g., a Kubernetes ConfigMap referencing the model URI). ArgoCD or Flux then detects the change and rolls out the new model version—treating model deployments exactly like code deployments.

    Python API Deep Dive

    The MlflowClient class provides the full programmatic interface to the Model Registry. Here is a comprehensive reference for the most important operations.

    from mlflow.tracking import MlflowClient client = MlflowClient(tracking_uri="http://mlflow.example.com:5000") # ── Create a registered model ── client.create_registered_model( name="churn-predictor", tags={"team": "growth", "framework": "xgboost"}, description="Customer churn prediction model for the growth team" ) # ── Register a version from a run ── result = client.create_model_version( name="churn-predictor", source="runs:/abc123def456/model", run_id="abc123def456", description="XGBoost with 500 trees, tuned with Optuna" ) print(f"Created version {result.version}") # ── Tag a version with metadata ── client.set_model_version_tag( name="churn-predictor", version=result.version, key="validation_auc", value="0.94" ) # ── Promote through stages ── client.transition_model_version_stage( name="churn-predictor", version=result.version, stage="Staging" ) # ── Load a model for inference ── import mlflow.pyfunc # By version model_v2 = mlflow.pyfunc.load_model("models:/churn-predictor/2") # By stage prod_model = mlflow.pyfunc.load_model("models:/churn-predictor/Production") # Run inference predictions = prod_model.predict(X_test) # ── Search across all registered models ── models = client.search_registered_models( filter_string="name LIKE '%churn%'", order_by=["name ASC"] ) for m in models: print(f"{m.name}: {len(m.latest_versions)} versions") # ── Delete a version (soft delete) ── client.delete_model_version(name="churn-predictor", version=1) # ── Delete an entire registered model ── # client.delete_registered_model(name="churn-predictor")
    ⚠ Deletion is permanent. Deleting a model version removes it from the registry. For audit compliance, prefer transitioning to Archived instead of deleting. Only delete models that were registered by mistake or contain sensitive data that must be purged.

    For production systems, wrap these API calls in your CI/CD pipeline scripts. A typical pattern is a "model promotion" GitHub Action that runs on a schedule or webhook trigger: it queries the Staging models, runs validation, and promotes passing models to Production automatically. This closes the loop between the model registry and your deployment infrastructure.
