    Model Signing and Integrity

    MLOps Series Model Registry

    Why Sign Models?

    Machine learning models are executable code disguised as data files. When you load a pickled PyTorch model or a TensorFlow SavedModel, you're deserializing arbitrary objects that can execute code on your machine. This makes ML models a prime attack vector for supply chain compromises—and yet most teams treat model files with far less scrutiny than they give to source code or container images.

    Real-World Threat: In 2023, researchers demonstrated that malicious pickle files hosted on Hugging Face Hub could execute arbitrary code during model loading. Without integrity verification, any model downloaded from an external source is a potential trojan horse.

    Model signing solves four critical problems:

    • Tamper detection — verify that no bits have been altered since the model was produced by a trusted training pipeline
    • Provenance attestation — cryptographically prove who produced the model and when
    • Compliance requirements — EU AI Act and FDA SaMD regulations require auditability of deployed model artifacts
    • Trust boundary enforcement — only allow models signed by authorized pipelines to be promoted to production

    Think of model signing as the ML equivalent of code signing for executables. Without it, you have no cryptographic guarantee that the model served in production is the same model your team trained and validated.

    Hash Verification

    The simplest form of integrity checking is hash verification. A cryptographic hash function like SHA-256 produces a fixed-size digest from the model file. Any change to the file—even flipping a single bit—produces a completely different hash. By computing the hash at training time and verifying it before loading, you detect accidental corruption and intentional tampering.

    import hashlib import os def compute_sha256(file_path, chunk_size=8192): """Compute SHA-256 hash of a file in streaming fashion.""" sha256 = hashlib.sha256() with open(file_path, "rb") as f: while True: chunk = f.read(chunk_size) if not chunk: break sha256.update(chunk) return sha256.hexdigest() def verify_model_hash(model_path, expected_hash): """Verify a model file against its expected SHA-256 hash.""" actual_hash = compute_sha256(model_path) if actual_hash != expected_hash: raise ValueError( f"Hash mismatch for {model_path}!\n" f" Expected: {expected_hash}\n" f" Got: {actual_hash}" ) return True # Usage during model registration model_hash = compute_sha256("model.pt") print(f"Model SHA-256: {model_hash}") # Store hash alongside model in registry metadata registry.log_artifact( model_path="model.pt", metadata={"sha256": model_hash} )
    Tip: Always compute hashes in streaming mode using chunked reads. Loading an entire 10GB model into memory just for hashing is wasteful. A chunk size of 8KB to 64KB balances I/O efficiency with memory use.

    For multi-file models (e.g., TensorFlow SavedModel directories), compute a manifest hash: hash each file individually, sort the hashes deterministically, then hash the concatenated list. This gives you a single digest representing the entire model directory.

    def compute_directory_hash(dir_path): """Compute a deterministic hash for an entire model directory.""" file_hashes = [] for root, dirs, files in os.walk(dir_path): dirs.sort() # Ensure deterministic traversal for fname in sorted(files): fpath = os.path.join(root, fname) rel = os.path.relpath(fpath, dir_path) fhash = compute_sha256(fpath) file_hashes.append(f"{rel}:{fhash}") manifest = "\n".join(file_hashes) return hashlib.sha256(manifest.encode()).hexdigest()

    Digital Signatures

    Hashes detect tampering but don't prove who produced the model. An attacker could replace both the model and its hash. Digital signatures solve this by using asymmetric cryptography: the training pipeline signs the model hash with a private key, and anyone can verify the signature using the corresponding public key.

    Hash Verification

    • Detects corruption and tampering
    • No identity or provenance
    • Hash can be replaced by attacker
    • Simple to implement

    Digital Signatures

    • Detects corruption and tampering
    • Proves identity of signer
    • Cannot be forged without private key
    • Requires key management infrastructure

    The following example uses the cryptography library with Ed25519 keys, which provide fast, secure signatures with small key sizes:

    from cryptography.hazmat.primitives.asymmetric.ed25519 import ( Ed25519PrivateKey, Ed25519PublicKey ) from cryptography.hazmat.primitives import serialization import json, base64 def generate_signing_keys(private_path, public_path): """Generate an Ed25519 key pair for model signing.""" private_key = Ed25519PrivateKey.generate() public_key = private_key.public_key() with open(private_path, "wb") as f: f.write(private_key.private_bytes( serialization.Encoding.PEM, serialization.PrivateFormat.PKCS8, serialization.BestAvailableEncryption(b"passphrase") )) with open(public_path, "wb") as f: f.write(public_key.public_bytes( serialization.Encoding.PEM, serialization.PublicFormat.SubjectPublicKeyInfo )) def sign_model(model_path, private_key_path, passphrase): """Sign a model file and return a signature envelope.""" model_hash = compute_sha256(model_path) with open(private_key_path, "rb") as f: private_key = serialization.load_pem_private_key( f.read(), password=passphrase ) signature = private_key.sign(model_hash.encode()) envelope = { "model_hash": model_hash, "algorithm": "Ed25519", "signature": base64.b64encode(signature).decode(), } return envelope
    def verify_model_signature(model_path, envelope, public_key_path): """Verify a model's digital signature against a public key.""" with open(public_key_path, "rb") as f: public_key = serialization.load_pem_public_key(f.read()) actual_hash = compute_sha256(model_path) if actual_hash != envelope["model_hash"]: raise ValueError("Model file has been tampered with") signature = base64.b64decode(envelope["signature"]) try: public_key.verify(signature, actual_hash.encode()) print("Signature verification PASSED") return True except Exception: raise ValueError("Signature verification FAILED")

    Sigstore and Model Transparency

    Managing your own PKI (Public Key Infrastructure) is operationally expensive. Sigstore offers a radically simpler approach: keyless signing backed by transparency logs. Instead of managing long-lived private keys, developers authenticate via OIDC (OpenID Connect) and receive short-lived certificates from the Fulcio certificate authority. Every signature is recorded in Rekor, a tamper-evident transparency log.

    The Sigstore ecosystem includes three core components:

    • Fulcio — a certificate authority that issues short-lived signing certificates tied to OIDC identities
    • Rekor — an immutable transparency log where all signatures are publicly recorded
    • Cosign — a CLI tool for signing and verifying container images, which can be extended to model artifacts
    Why Keyless? Traditional code signing requires storing private keys securely, rotating them periodically, and revoking them if compromised. Sigstore eliminates this by binding signatures to ephemeral certificates and recording everything in a transparency log. If a signing event didn't appear in Rekor, it didn't happen.

    For containerized models, you can sign the OCI image directly with cosign:

    # Sign a model container image with Sigstore (keyless) $ cosign sign --yes registry.example.com/models/fraud-detector:v2.1 # Verify the signature $ cosign verify \ --certificate-identity=pipeline@myorg.iam.gserviceaccount.com \ --certificate-oidc-issuer=https://accounts.google.com \ registry.example.com/models/fraud-detector:v2.1 # Attach model metadata as a signed attestation $ cosign attest --predicate model-card.json \ --type custom \ registry.example.com/models/fraud-detector:v2.1

    For non-containerized model files, you can use sigstore-python to sign arbitrary blobs:

    from sigstore.sign import SigningContext from sigstore.verify import Verifier, VerificationMaterials # Sign a model file using ambient OIDC credentials with SigningContext.production() as ctx: signer = ctx.signer() with open("model.onnx", "rb") as f: result = signer.sign(f.read()) result.write("model.onnx.sigstore")

    Supply Chain Security

    Model signing is one piece of a broader supply chain security strategy. The SLSA (Supply-chain Levels for Software Artifacts) framework, originally designed for software, maps directly to ML workflows. SLSA defines four levels of increasing assurance:

    SLSA Level 1

    Documentation: The build process is documented. For ML, this means recording the training script, data source, and hyperparameters in a structured provenance document.

    SLSA Level 2

    Hosted build: Training runs on a managed build service (not a developer laptop). CI/CD systems like GitHub Actions generate signed provenance automatically.

    SLSA Level 3

    Hardened builds: The training environment is isolated and tamper-resistant. No human has shell access to the training node during execution.

    SLSA Level 4

    Two-party review: All changes to training code require review from two authorized individuals. Hermetic builds ensure reproducibility.

    SBOM (Software Bill of Materials) for models extends traditional SBOMs to include ML-specific components: training data licenses, pre-trained base model provenance, and fine-tuning dataset origins. An ML-SBOM might look like:

    # ML-SBOM example (CycloneDX format excerpt) { "bomFormat": "CycloneDX", "components": [ {"type": "machine-learning-model", "name": "fraud-detector-v2.1", "hashes": [{"alg": "SHA-256", "content": "a1b2c3..."}]}, {"type": "data", "name": "transactions-2024-q1", "licenses": [{"id": "Apache-2.0"}]}, {"type": "library", "name": "torch", "version": "2.2.0"}, {"type": "machine-learning-model", "name": "bert-base-uncased", "description": "Pre-trained base model from HuggingFace"} ] }

    Implementation Guide

    Here is an end-to-end workflow for integrating model signing into your ML pipeline. The process has four stages: key generation, signing during training, distribution through the registry, and verification at deployment time.

    import json, time, os class ModelSigner: """End-to-end model signing and verification workflow.""" def __init__(self, private_key_path=None, public_key_path=None): self.private_key_path = private_key_path self.public_key_path = public_key_path def sign_and_register(self, model_path, registry_client, model_name, version): """Sign a model and register it with its signature envelope.""" # Step 1: Compute file hash model_hash = compute_sha256(model_path) # Step 2: Create digital signature envelope = sign_model( model_path, self.private_key_path, b"passphrase" ) # Step 3: Add provenance metadata envelope["signed_at"] = time.time() envelope["signer_id"] = os.getenv("CI_PIPELINE_ID", "local") envelope["model_name"] = model_name envelope["version"] = version # Step 4: Upload model + signature to registry registry_client.upload_model(model_path, model_name, version) registry_client.attach_metadata( model_name, version, "signature_envelope", json.dumps(envelope) ) return envelope def verify_and_load(self, model_path, envelope_json): """Verify model signature before loading into memory.""" envelope = json.loads(envelope_json) verify_model_signature( model_path, envelope, self.public_key_path ) print(f"Model signed by: {envelope['signer_id']}") print(f"Signed at: {envelope['signed_at']}") return load_model(model_path)
    Warning: Never store private signing keys in your Git repository or model registry. Use a secrets manager (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault) and inject keys into the CI/CD environment at runtime.

    Best Practices

    Effective model signing requires more than just generating keys and computing hashes. Here are the operational practices that separate toy implementations from production-grade signing workflows:

    • Key rotation: Rotate signing keys on a fixed schedule (e.g., every 90 days). Old models remain verifiable with archived public keys, while new models use current keys.
    • Separation of duties: The key that signs training models should not be the same key that signs deployment approvals. Use separate key pairs for each pipeline stage.
    • Air-gapped signing: For high-assurance environments, perform signing on hardware security modules (HSMs) that never expose the private key to software.
    • Verification gates: Add mandatory signature verification as a deployment gate. No model reaches production without a valid signature from an authorized pipeline.
    • Transparency logging: Record every signing event to an append-only log. If you can't use Sigstore's Rekor, maintain an internal transparency log.
    # CI/CD integration: GitHub Actions step for model signing # .github/workflows/train-and-sign.yml (excerpt) # # - name: Sign model artifact # run: | # python sign_model.py \ # --model outputs/model.pt \ # --key-vault-url ${{ secrets.KEY_VAULT_URL }} \ # --output outputs/model.pt.sig # # - name: Verify before push # run: | # python verify_model.py \ # --model outputs/model.pt \ # --signature outputs/model.pt.sig \ # --public-key keys/pipeline-signing.pub def ci_signing_workflow(model_path, vault_client, registry): """Automated signing workflow for CI/CD pipelines.""" # Fetch signing key from vault (never stored on disk) private_key_pem = vault_client.get_secret("model-signing-key") # Sign model in memory signer = ModelSigner() envelope = signer.sign_with_key_bytes(model_path, private_key_pem) # Push model + signature to registry registry.push(model_path, envelope) # Log signing event for audit trail audit_log.record({ "event": "model_signed", "pipeline_id": os.getenv("GITHUB_RUN_ID"), "model_hash": envelope["model_hash"], "timestamp": time.time() })
    Summary: Start with SHA-256 hashing as your baseline integrity check. Add Ed25519 digital signatures for provenance. Adopt Sigstore for keyless signing at scale. Wrap everything in a SLSA-compliant pipeline to build a complete supply chain security posture for your ML models.
