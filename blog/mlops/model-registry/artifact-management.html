    Artifact Management: S3, GCS, Model Stores

    MLOps Series Model Registry

    What Are ML Artifacts?

    An ML artifact is any file or data object produced or consumed during the machine learning lifecycle. While "model artifact" often refers to the serialized model binary (a .pkl, .pt, or .onnx file), the full set of artifacts is much broader:

    • Model binaries — Serialized model weights and architecture (PyTorch .pt, TensorFlow SavedModel, ONNX, scikit-learn .joblib)
    • Datasets — Training, validation, and test data snapshots (Parquet, CSV, TFRecord)
    • Configurations — Hyperparameter files, feature engineering configs, preprocessing pipelines
    • Logs — Training logs, TensorBoard event files, metric histories
    • Visualizations — Confusion matrices, ROC curves, feature importance plots
    • Environment specs — requirements.txt, conda.yaml, Dockerfiles
    Key principle: Every artifact produced during training should be stored, versioned, and linked to the model version that produced it. Orphaned artifacts are a liability — they consume storage without providing traceability.

    Storage Backends

    Choosing the right storage backend depends on your cloud provider, team size, compliance requirements, and budget. Here is a comparison of the major options:

    ☁️ Amazon S3

    • Industry standard for object storage
    • Native versioning, lifecycle policies
    • Tiered storage (Glacier for archival)
    • IAM-based access control

    ☁️ Google Cloud Storage

    • Tight integration with Vertex AI
    • Object versioning, retention policies
    • Nearline/Coldline/Archive tiers
    • Uniform bucket-level IAM

    ☁️ Azure Blob Storage

    • Integrates with Azure ML workspace
    • Hot/Cool/Archive tiers
    • Blob versioning, soft delete
    • RBAC via Azure Active Directory

    💻 Local / NFS

    • No cloud dependency, low latency
    • Simple for small teams / prototyping
    • Manual backup and versioning needed
    • Scalability and durability concerns
    Warning: Local filesystem storage works for experiments but is a liability in production. A single disk failure can wipe out months of model artifacts. Always use durable cloud storage or replicated NFS for anything beyond prototyping.

    Versioned Artifacts

    Artifact versioning ensures you can always retrieve the exact artifact used in a specific training run or deployment. There are three primary approaches to versioning large ML files:

    Object Versioning in S3/GCS

    Cloud storage providers support native object versioning. When enabled, every overwrite creates a new version with a unique version ID. This is transparent to the uploader and provides point-in-time recovery.

    import boto3 def upload_versioned_artifact(bucket, key, local_path): # Upload to a versioning-enabled S3 bucket s3 = boto3.client("s3") # Upload returns the VersionId automatically response = s3.upload_file( Filename=local_path, Bucket=bucket, Key=key, ExtraArgs={"Metadata": {"content-hash": compute_sha256(local_path)}}, ) # Retrieve the version ID for this upload head = s3.head_object(Bucket=bucket, Key=key) version_id = head["VersionId"] print(f"Uploaded {key} → version {version_id}") return version_id def download_specific_version(bucket, key, version_id, dest_path): # Download a specific version of an artifact s3 = boto3.client("s3") s3.download_file( Bucket=bucket, Key=key, Filename=dest_path, ExtraArgs={"VersionId": version_id}, ) print(f"Downloaded {key} v{version_id} → {dest_path}")

    DVC for Large File Versioning

    DVC (Data Version Control) tracks large files via lightweight pointer files in Git while storing actual data in remote storage. This gives you Git-like versioning semantics for multi-gigabyte datasets and models.

    # DVC workflow for artifact versioning # Initialize DVC in your repo $ dvc init $ dvc remote add -d myremote s3://ml-artifacts/dvc-store # Track a large model file $ dvc add models/fraud_detector.pt # This creates models/fraud_detector.pt.dvc (pointer file) # and adds the actual file to .gitignore # Commit the pointer to Git $ git add models/fraud_detector.pt.dvc .gitignore $ git commit -m "Register fraud detector v2.1.0" # Push the artifact to remote storage $ dvc push # Later: retrieve a specific version $ git checkout v2.0.0 -- models/fraud_detector.pt.dvc $ dvc pull
    Tip: DVC uses content-addressable storage (SHA-256 hashes). If two training runs produce identical model files, DVC stores only one copy — saving significant storage costs for iterative experiments.

    Handling Large Files

    Large language models and deep learning checkpoints routinely exceed 10 GB. Standard tools struggle with files this large, so you need specialized strategies for upload, download, and storage.

    Git LFS

    Git Large File Storage replaces large files in your repository with text pointers while storing the file contents on a separate server. It integrates with Git seamlessly but has limitations for very large files (GitHub LFS has a 2 GB per-file limit).

    Chunked Uploads

    For multi-gigabyte artifacts, use multipart uploads provided by S3 and GCS. This splits the file into chunks (typically 100 MB each), uploads them in parallel, and assembles them server-side.

    import boto3 from boto3.s3.transfer import TransferConfig def upload_large_artifact(bucket, key, local_path): # Configure multipart upload for large files config = TransferConfig( multipart_threshold=100 * 1024 * 1024, # 100 MB multipart_chunksize=100 * 1024 * 1024, # 100 MB per part max_concurrency=10, # parallel uploads use_threads=True, ) s3 = boto3.client("s3") s3.upload_file( Filename=local_path, Bucket=bucket, Key=key, Config=config, Callback=ProgressCallback(local_path), ) class ProgressCallback: def __init__(self, filename): import os self.total = os.path.getsize(filename) self.uploaded = 0 def __call__(self, bytes_transferred): self.uploaded += bytes_transferred pct = (self.uploaded / self.total) * 100 print(f"\rUpload progress: {pct:.1f}%", end="")
    Warning: Never store large model files directly in Git. A 5 GB model file permanently inflates your repository history, making clones painfully slow. Use Git LFS, DVC, or direct cloud storage instead.

    Artifact Stores

    Dedicated artifact stores provide a higher-level abstraction than raw cloud storage. They handle versioning, metadata association, access control, and lineage tracking out of the box.

    MLflow Artifact Store

    MLflow's artifact store integrates with S3, GCS, Azure Blob, and HDFS. Artifacts logged during an MLflow run are automatically associated with that run's metrics, parameters, and tags.

    import mlflow # Configure remote artifact store mlflow.set_tracking_uri("http://mlflow-server:5000") with mlflow.start_run(run_name="fraud-detector-v2.1") as run: # Log training parameters mlflow.log_params({"lr": 0.001, "epochs": 50, "batch_size": 256}) # Log performance metrics mlflow.log_metrics({"auc_roc": 0.943, "f1": 0.79}) # Log model artifact (auto-uploads to artifact store) mlflow.sklearn.log_model(model, "model") # Log additional artifacts mlflow.log_artifact("confusion_matrix.png") mlflow.log_artifact("feature_importance.json") mlflow.log_artifact("training_config.yaml") # All artifacts stored at: # s3://ml-artifacts/mlflow/{run_id}/artifacts/

    Organizing Artifacts

    A well-organized artifact store is searchable, self-documenting, and intuitive for new team members. Without conventions, artifact stores devolve into chaotic dump directories.

    Recommended Directory Structure

    # Hierarchical artifact organization in cloud storage s3://ml-artifacts/ ├── models/ │ ├── fraud-detector/ │ │ ├── v2.1.0/ │ │ │ ├── model.pt │ │ │ ├── config.yaml │ │ │ ├── metrics.json │ │ │ ├── model_card.md │ │ │ └── lineage.json │ │ └── v2.0.0/ │ │ └── ... │ └── churn-predictor/ │ └── ... ├── datasets/ │ ├── transactions/ │ │ ├── v3.2/ │ │ │ ├── train.parquet │ │ │ ├── test.parquet │ │ │ └── manifest.json │ │ └── ... │ └── ... └── experiments/ ├── exp-2024-03-15-hp-sweep/ │ ├── run-001/ │ ├── run-002/ │ └── summary.json └── ...

    Tagging and Searchability

    Use cloud storage tags (S3 object tags, GCS labels) to make artifacts searchable without parsing directory paths. Essential tags include: model_name, version, stage, team, experiment_id, and data_version.

    Tip: Maintain a manifest.json in each version directory that lists all artifacts with their SHA-256 hashes, sizes, and descriptions. This acts as a local index and enables integrity verification.

    Lifecycle Management

    ML artifacts accumulate rapidly. A single hyperparameter sweep can generate hundreds of model checkpoints, each several gigabytes. Without lifecycle management, storage costs balloon and useful artifacts become buried under noise.

    Retention Policies

    • Production models — Retain indefinitely (or minimum 7 years for regulated industries)
    • Staging models — Retain for 90 days after demotion
    • Experiment artifacts — Retain for 30 days, then archive or delete
    • Failed runs — Auto-delete after 7 days (keep only logs for debugging)
    import boto3 from datetime import datetime, timedelta def enforce_retention_policy(bucket, prefix, max_age_days, dry_run=True): # Clean up artifacts older than retention period s3 = boto3.client("s3") cutoff = datetime.utcnow() - timedelta(days=max_age_days) paginator = s3.get_paginator("list_objects_v2") to_delete = [] for page in paginator.paginate(Bucket=bucket, Prefix=prefix): for obj in page.get("Contents", []): if obj["LastModified"].replace(tzinfo=None) < cutoff: to_delete.append({"Key": obj["Key"]}) if dry_run: print(f"Would delete {len(to_delete)} objects from {prefix}") return to_delete # Batch delete (max 1000 per request) for i in range(0, len(to_delete), 1000): batch = to_delete[i:i+1000] s3.delete_objects( Bucket=bucket, Delete={"Objects": batch}, ) print(f"Deleted {len(to_delete)} expired artifacts")

    Cost Optimization

    Use S3 Intelligent-Tiering or lifecycle rules to automatically move infrequently accessed artifacts to cheaper storage tiers. A model checkpoint that has not been accessed in 90 days should be in Glacier, not Standard storage.

    Warning: Always run retention cleanup in dry_run mode first. A misconfigured retention policy can permanently delete production model artifacts. Test thoroughly, review the deletion list, and keep deletion logs for audit purposes.
