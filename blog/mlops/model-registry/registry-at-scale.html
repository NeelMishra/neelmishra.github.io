    Running a Model Registry at Scale

    MLOps Series Model Registry

    Scaling Challenges

    A model registry that works for a single team with ten models breaks down spectacularly when an organization grows to fifty teams managing hundreds of models. The challenges are not primarily technical—storage and compute scale linearly—but organizational: governance, discovery, consistency, and trust.

    The scaling pain points most organizations encounter:

    • Naming collisions — two teams independently create a model called "fraud-detector" with incompatible schemas
    • Inconsistent metadata — each team tracks different fields, making cross-team comparison impossible
    • Promotion chaos — no clear process for who can move a model to production, leading to either bottlenecks or unchecked deployments
    • Discovery failure — teams retrain models that already exist because they can't find them in the registry
    • Compliance gaps — audit requests take weeks because metadata is scattered across teams with no central authority
    Warning: Most organizations hit registry scaling problems at around 20-30 models across 3+ teams. Don't wait until you have hundreds of models to implement governance—retrofitting is far more painful than building it in from the start.

    Multi-Team Governance

    Multi-team governance starts with namespaces: logical partitions that give each team ownership over their models while maintaining global discoverability. A namespace is typically aligned with a team, business unit, or product area.

    class RegistryNamespace: """Namespace-based governance for multi-team registries.""" def __init__(self, name, owners, metadata_schema): self.name = name self.owners = owners # List of team members with admin access self.metadata_schema = metadata_schema self.approval_policy = None def set_approval_policy(self, policy): """Define who can approve stage transitions.""" self.approval_policy = policy def register_model(self, model_name, version, metadata, user): """Register a model within this namespace.""" full_name = f"{self.name}/{model_name}" # Validate metadata against namespace schema validate_model_metadata(metadata, self.metadata_schema) # Check user permissions if not self._user_can_register(user): raise PermissionError( f"User {user} not authorized for namespace {self.name}" ) return self.registry.create_version(full_name, version, metadata) def request_promotion(self, model_name, version, target_stage, requester): """Create a promotion request requiring approval.""" request = { "model": f"{self.name}/{model_name}", "version": version, "target_stage": target_stage, "requester": requester, "required_approvers": self.approval_policy.get_approvers(target_stage), "status": "pending" } return self.registry.create_promotion_request(request)

    The approval workflow should scale with risk. Promoting a model to a staging environment might require one peer review, while production promotion requires sign-off from both the owning team's lead and a central ML platform reviewer:

    Low-Risk Promotions

    • None → Development: Automatic on registration
    • Development → Staging: 1 peer review from same team

    High-Risk Promotions

    • Staging → Production: Team lead + ML platform review
    • Production rollback: Any team member (speed over ceremony)

    Access Control

    Role-Based Access Control (RBAC) for model registries maps organizational roles to specific registry permissions. The key insight is that model registry RBAC has more dimensions than typical application RBAC: you need to control access by namespace, by model, by stage, and by operation type.

    class RegistryRBAC: """Role-based access control for model registries.""" ROLES = { "viewer": {"read_metadata", "list_models", "search"}, "contributor": { "read_metadata", "list_models", "search", "register_model", "download_model", "update_metadata" }, "approver": { "read_metadata", "list_models", "search", "register_model", "download_model", "update_metadata", "approve_promotion", "transition_stage" }, "admin": { "read_metadata", "list_models", "search", "register_model", "download_model", "update_metadata", "approve_promotion", "transition_stage", "delete_model", "manage_namespace", "manage_roles" }, } def __init__(self): self.assignments = {} # user -> {namespace: role} self.audit_log = [] def check_permission(self, user, namespace, action): """Check if a user has permission for an action in a namespace.""" role = self.assignments.get(user, {}).get(namespace) if not role: self._log_denied(user, namespace, action) return False allowed = action in self.ROLES.get(role, set()) if not allowed: self._log_denied(user, namespace, action) return allowed def _log_denied(self, user, namespace, action): self.audit_log.append({ "event": "access_denied", "user": user, "namespace": namespace, "action": action, "timestamp": datetime.utcnow().isoformat() })
    Tip: Implement stage-gated access in addition to role-based access. Even within a team, only CI/CD service accounts should be able to write to the Production stage. Human users can register models and request promotions, but automated pipelines perform the actual transitions after all gates pass.

    Every access event—reads, writes, promotions, and denials—should be recorded in an immutable audit log. This log is your primary compliance artifact and is essential for investigating security incidents.

    Automation

    At scale, manual model registration and promotion don't work. Teams forget to register models, metadata is incomplete, and promotion bottlenecks form around a few approvers. Automation solves this by making the registry a natural output of your training and CI/CD pipelines.

    class AutomatedRegistrationPipeline: """Automatically register and validate models from training pipelines.""" def __init__(self, registry, validator, notifier): self.registry = registry self.validator = validator self.notifier = notifier def on_training_complete(self, run_context): """Hook called when a training run completes.""" model_path = run_context["model_path"] metadata = run_context["metadata"] # Gate 1: Metadata schema validation if not self.validator.validate_schema(metadata): self.notifier.alert("Registration blocked: invalid metadata") return False # Gate 2: Minimum performance threshold if metadata["metrics"]["f1"] < 0.80: self.notifier.alert("Registration blocked: F1 below threshold") return False # Gate 3: Model integrity check if not self.validator.verify_signature(model_path): self.notifier.alert("Registration blocked: signature invalid") return False # Gate 4: Bias and fairness checks if not self.validator.check_fairness(metadata): self.notifier.alert("Registration blocked: fairness check failed") return False # All gates passed: register the model version = self.registry.register( model_path, metadata, stage="development" ) self.notifier.notify(f"Model {metadata['name']} v{version} registered") return True def auto_promote(self, model_name, version, target_stage): """Automatically promote if all validation gates pass.""" gates = self.registry.get_promotion_gates(target_stage) results = {} for gate in gates: results[gate.name] = gate.evaluate(model_name, version) if all(results.values()): self.registry.transition_stage(model_name, version, target_stage) return {"promoted": True, "gates": results} return {"promoted": False, "gates": results}
    Warning: Automated promotion to production should always include a human approval step, even if all automated gates pass. Full automation is appropriate for Development → Staging, but Staging → Production should require at least one human sign-off.

    CI/CD Integration

    Integrating the model registry with CI/CD pipelines turns model registration from an afterthought into a first-class part of the development workflow. Here's a GitHub Actions workflow that trains, validates, signs, and registers a model:

    # .github/workflows/train-register.yml # name: Train and Register Model # # on: # push: # branches: [main] # paths: ['models/**', 'data/**'] # # jobs: # train-and-register: # runs-on: ubuntu-latest # steps: # - uses: actions/checkout@v4 # - name: Train model # run: python train.py --config config/prod.yaml # - name: Run validation suite # run: python validate.py --model outputs/model.pt # - name: Sign model artifact # run: python sign.py --model outputs/model.pt # - name: Register in model registry # run: python register.py --model outputs/model.pt # env: # MLFLOW_TRACKING_URI: ${{ secrets.MLFLOW_URI }} # REGISTRY_TOKEN: ${{ secrets.REGISTRY_TOKEN }}

    The registration script integrates all the pieces—metadata collection, schema validation, signing, and upload:

    import os, json, sys def ci_register_model(model_path, config_path): """CI/CD model registration with full validation.""" # Collect all metadata metadata = { "model_name": os.getenv("MODEL_NAME"), "version": os.getenv("MODEL_VERSION"), "metrics": json.load(open("outputs/metrics.json")), "training_config": json.load(open(config_path)), "data": json.load(open("outputs/data_metadata.json")), "environment": capture_environment_metadata(), "ci": { "run_id": os.getenv("GITHUB_RUN_ID"), "commit": os.getenv("GITHUB_SHA"), "branch": os.getenv("GITHUB_REF_NAME"), "actor": os.getenv("GITHUB_ACTOR"), } } # Validate schema validate_model_metadata(metadata, "schemas/model_metadata.json") # Register with MLflow import mlflow mlflow.set_tracking_uri(os.getenv("MLFLOW_TRACKING_URI")) with mlflow.start_run() as run: mlflow.log_params(metadata["training_config"]) mlflow.log_metrics(metadata["metrics"]) mlflow.log_dict(metadata, "full_metadata.json") mlflow.pytorch.log_model(model_path, "model") result = mlflow.register_model( f"runs:/{run.info.run_id}/model", metadata["model_name"] ) print(f"Registered: {result.name} v{result.version}") return result
    Tip: Use GitHub Actions environments with required reviewers for production deployments. The staging environment can auto-deploy, while the production environment requires manual approval from designated reviewers.

    Model Discovery

    A model registry at scale is only useful if people can find what they need. Model discovery encompasses search, tagging, documentation, and catalog features that make the registry navigable by teams who didn't create the models.

    class ModelCatalog: """Searchable model catalog for cross-team discovery.""" def __init__(self, registry, search_index): self.registry = registry self.search = search_index def index_model(self, model_name, version, metadata): """Index a model for full-text and faceted search.""" doc = { "name": model_name, "version": version, "description": metadata.get("description", ""), "tags": metadata.get("tags", []), "task_type": metadata.get("task_type"), # classification, regression, etc. "domain": metadata.get("domain"), # fraud, nlp, vision, etc. "owner_team": metadata.get("owner_team"), "stage": metadata.get("stage"), "metrics": metadata.get("metrics", {}), "framework": metadata.get("framework"), "input_schema": metadata.get("input_schema"), "output_schema": metadata.get("output_schema"), } self.search.index_document(doc) def search_models(self, query, filters=None): """Search models with text query and optional faceted filters.""" search_params = {"query": query} if filters: if "domain" in filters: search_params["domain"] = filters["domain"] if "stage" in filters: search_params["stage"] = filters["stage"] if "min_f1" in filters: search_params["metrics.f1__gte"] = filters["min_f1"] return self.search.query(search_params) def get_popular_models(self, limit=20): """Get most downloaded/used models across the organization.""" return self.search.query( {"sort": "download_count", "order": "desc", "limit": limit} )

    Essential discovery features for a scaled registry:

    • Full-text search across model names, descriptions, and tags
    • Faceted filtering by domain, task type, framework, stage, and team
    • Input/output schema browsing so consumers know how to use a model
    • Usage statistics showing download counts and active deployments
    • Model cards attached to every production model with human-readable documentation

    Case Study: Enterprise Registry

    Consider a financial services company with four ML teams: Fraud Detection, Credit Scoring, Customer Analytics, and Risk Modeling. Together they manage 120+ models, with 15-20 models in active production at any time. Here's how they structured their registry:

    Architecture Decisions

    • MLflow backend with PostgreSQL metadata store
    • S3-compatible object storage for model artifacts
    • Elasticsearch for model catalog search
    • GitHub Actions for CI/CD integration
    • OIDC authentication via corporate SSO

    Governance Structure

    • 4 namespaces (one per team)
    • Shared metadata schema with team-specific extensions
    • Mandatory model cards for production models
    • Quarterly access reviews by compliance
    • Automated bias testing in the promotion pipeline
    # Enterprise registry configuration example class EnterpriseRegistryConfig: NAMESPACES = { "fraud": { "owners": ["fraud-ml-leads"], "schema": "schemas/fraud_model.json", "promotion_policy": "two_approver", "required_fairness_slices": ["region", "account_age"], }, "credit": { "owners": ["credit-ml-leads"], "schema": "schemas/credit_model.json", "promotion_policy": "two_approver_plus_compliance", "required_fairness_slices": ["age_group", "income_bracket"], }, "analytics": { "owners": ["analytics-ml-leads"], "schema": "schemas/base_model.json", "promotion_policy": "single_approver", }, "risk": { "owners": ["risk-ml-leads"], "schema": "schemas/risk_model.json", "promotion_policy": "two_approver_plus_compliance", "required_fairness_slices": ["geography", "product_type"], }, } GLOBAL_GATES = [ "metadata_schema_valid", "model_signature_verified", "minimum_f1_threshold", "no_data_leakage_detected", ] PRODUCTION_GATES = GLOBAL_GATES + [ "model_card_complete", "fairness_slices_passed", "latency_budget_met", "rollback_tested", ]

    Key outcomes after one year of operating this registry:

    • Model reuse increased 40% — teams discovered and fine-tuned existing models instead of training from scratch
    • Audit preparation dropped from 3 weeks to 2 days — all lineage and metadata centrally available
    • Production incidents from bad models dropped 70% — automated gates caught issues before deployment
    • Time-to-production for new models decreased 50% — standardized pipeline eliminated manual steps
    Key Takeaway: Scaling a model registry is fundamentally an organizational challenge, not a technical one. The technology choices matter less than having clear ownership, consistent schemas, automated gates, and a culture that treats the registry as the single source of truth for deployed models.
