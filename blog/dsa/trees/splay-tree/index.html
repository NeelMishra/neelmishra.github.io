    Splay Trees

    April 2026 · 25 min read

    Introduction

    A splay tree is a self-adjusting binary search tree invented by Sleator and Tarjan in 1985. Unlike AVL or Red-Black trees, a splay tree stores no balance information at all—no heights, no colors, no balance factors. Instead, every time you access a node, you splay it: move it to the root through a specific sequence of rotations.

    This sounds reckless. Any single operation can take O(n) time if the tree is badly shaped. But the magic is in the amortized analysis: any sequence of m operations on an n-node splay tree takes O(m log n) total time. Each operation is O(log n) amortized.

    Why care? Splay trees are the foundation for Link-Cut Trees, one of the most powerful data structures in competitive programming (used in dynamic connectivity, maximum flow, and more). Master splay trees first, and Link-Cut Trees become a natural extension.

    Key properties that make splay trees special:

    • No extra storage — each node is just (key, left, right, parent). No height, color, or priority field.
    • Adaptive — frequently accessed elements migrate toward the root, giving better than O(log n) for skewed access patterns.
    • Split/Merge — natural support for split and merge operations, making them ideal for implicit-key problems.
    • Simple code — the entire implementation fits in ~60 lines of C++.

    Splaying: The Core Operation

    Splaying moves a node x to the root by repeatedly applying one of three cases based on the relationship between x, its parent p, and its grandparent g (if it exists). Each case involves one or two rotations.

    Case 1: Zig (x is child of root)

    When x's parent p is the root, we perform a single rotation. This is the base case and happens at most once per splay (as the last step).

    Case 2: Zig-Zig (x and p are same-side children)

    When x and p are both left children (or both right children), we rotate g first, then rotate p. This is the critical case that distinguishes splay trees from naive “rotate to root”.

    Case 3: Zig-Zag (x and p are different-side children)

    When x is a right child of p, and p is a left child of g (or vice versa), we rotate p first, then g. This is like an AVL double rotation.

    Why Zig-Zig, Not Two Single Rotations?

    This is the deepest insight behind splay trees. Consider a degenerate chain 1→2→3→4→5 and suppose we want to splay node 1 to the root. We have two options:

    Approach 1: “Rotate to root” — repeatedly do single rotations (zig) moving x up one level at a time. After splaying 1 to the root, the tree becomes 1←5←4←3←2. We’ve just reversed the chain! It’s still degenerate. The next access at the bottom costs O(n) again, and repeating this gives Θ(n) per operation on average.

    Approach 2: Zig-Zig (splay) — for every pair of edges on the path, we rotate the grandparent first, then the parent. After splaying 1 to the root, the remaining nodes are distributed much more evenly. The tree roughly halves in depth.

    The key insight: Zig-zig rotates the grandparent first. This doesn’t just move x up—it also compresses the entire path, improving the tree for future operations. This “paying it forward” is what makes the amortized analysis work.

    Operations

    Every splay tree operation is built on top of splay(x). The pattern is always: do the BST thing, then splay.

    Find(key)

    1. Walk down the tree as in a standard BST search.
    2. If found, splay the node to the root.
    3. If not found, splay the last visited node (the would-be parent) to the root.

    After find(key), the root is either the node with the given key (if it exists) or the node closest to it. This is extremely useful for split operations.

    Split and Merge

    These are the workhorses for competitive programming splay tree implementations.

    Split(tree, key) → (left, right):

    1. Splay the largest element ≤ key to the root.
    2. Detach the right subtree. Now root has no right child.
    3. Return (root, detached right subtree).

    Merge(left, right) (requires all keys in left < all keys in right):

    1. Splay the maximum element in left to its root.
    2. Now left’s root has no right child (it’s the max).
    3. Set left.root.right = right.root.

    Insert(key)

    1. Split the tree by key into (L, R).
    2. Create a new node with the given key.
    3. Make L its left child and R its right child.

    Alternatively: do a standard BST insert, then splay the new node to the root.

    Delete(key)

    1. Splay the node with the given key to the root.
    2. Now the root has key = target. Remove it.
    3. Merge the left and right subtrees.

    Amortized Analysis Sketch

    The amortized analysis of splay trees uses a potential function. The intuition is elegant: we define a “bank account” for the tree, and show that expensive operations must be preceded by cheap ones that build up credit.

    Setup: For each node x, define its rank as r(x) = log2(size of subtree rooted at x). The potential of the tree is Φ = Σ r(x) for all nodes x.

    Key observations:

    • A balanced tree has potential Θ(n log n).
    • A degenerate chain has potential Θ(n log n) as well (somewhat surprisingly!).
    • The amortized cost of a splay is at most 3(r(root) − r(x)) + 1 = O(log n).

    Why does zig-zig work? In the zig-zig case, the potential drop from the grandparent’s subtree shrinking is large enough to “pay” for both rotations. This is precisely why we rotate the grandparent first—it causes a bigger structural change that releases more potential.

    Intuition: When splay takes O(n) time (accessing a deep node), it compresses the path so dramatically that the potential drops by Ω(n). The “cost” was already deposited by previous cheap operations. You can think of it as: every node you pass through on the way down “paid in advance” to be closer to the root.

    The full proof is a case analysis on zig, zig-zig, and zig-zag. Each case telescopes, and the total amortized cost is O(log n). For details, see Sleator and Tarjan’s original 1985 paper, or Tarjan’s “Data Structures and Network Algorithms.”

    Splay Tree as Implicit Array

    Just like a treap can be used as an “implicit array” (where keys are positions, not values), a splay tree can do the same. This is one of the most powerful applications in competitive programming.

    The idea: Instead of storing explicit keys, each node stores a size field. The in-order position of a node determines its index in the array. To access position k:

    // Find the k-th element (0-indexed) via subtree sizes
Node* kth(Node* t, int k) {
    push(t);  // propagate lazy tags if any
    int left_size = sz(t->ch[0]);
    if (k == left_size) { splay(t); return t; }
    if (k < left_size) return kth(t->ch[0], k);
    return kth(t->ch[1], k - left_size - 1);
}

    Operations on the implicit array:

    Array OperationSplay Tree TranslationAmortized
    Insert at position iSplit at i, make new root, mergeO(log n)
    Delete at position iSplay i-th, merge childrenO(log n)
    Reverse range [l, r]Split out [l,r], flip lazy flag, merge backO(log n)
    Cyclic shift rangeSplit, rearrange, mergeO(log n)
    Range sum/min/maxSplit out range, read aggregate, mergeO(log n)

    The reverse operation is the classic example: to reverse a[l..r], split out the subtree for that range, toggle a “reversed” lazy flag on its root, and merge it back. The push function swaps left/right children when it encounters the flag.

    void push(Node* t) {
    if (!t || !t->rev) return;
    swap(t->ch[0], t->ch[1]);
    if (t->ch[0]) t->ch[0]->rev ^= 1;
    if (t->ch[1]) t->ch[1]->rev ^= 1;
    t->rev = 0;
}

    C++ Implementation

    Here is a contest-ready splay tree with split/merge, suitable for both explicit-key and implicit-key (array) problems. This implementation uses the “parent pointer” style for splaying.

    struct Node {
    int key, sz;
    bool rev;
    Node *ch[2], *par;
    Node(int k) : key(k), sz(1), rev(0), par(0) { ch[0]=ch[1]=0; }
};

int sz(Node* t) { return t ? t->sz : 0; }

void push(Node* t) {
    if (!t || !t->rev) return;
    swap(t->ch[0], t->ch[1]);
    if (t->ch[0]) t->ch[0]->rev ^= 1;
    if (t->ch[1]) t->ch[1]->rev ^= 1;
    t->rev = 0;
}

void pull(Node* t) {
    if (t) t->sz = 1 + sz(t->ch[0]) + sz(t->ch[1]);
}

// dir=0: left child, dir=1: right child
bool isRoot(Node* t) { return !t->par; }
int dir(Node* t) { return t->par->ch[1] == t; }

void setChild(Node* p, int d, Node* c) {
    if (p) p->ch[d] = c;
    if (c) c->par = p;
}

// Single rotation: rotate t up
void rotate(Node* t) {
    Node *p = t->par, *g = p->par;
    int d = dir(t), gd = g ? dir(p) : 0;
    setChild(p, d, t->ch[d^1]);
    setChild(t, d^1, p);
    setChild(g, gd, t);
    pull(p); pull(t);
}

// Splay t to root (or to child of 'above' if given)
void splay(Node* t) {
    if (!t) return;
    // Push all ancestors first
    static Node* stk[200000];
    int top = 0;
    for (Node* u = t; u; u = u->par) stk[top++] = u;
    while (top) push(stk[--top]);

    while (!isRoot(t)) {
        Node* p = t->par;
        if (!isRoot(p)) {
            // Zig-zig or zig-zag
            if (dir(t) == dir(p)) rotate(p); // zig-zig: rotate parent first
            else rotate(t);                   // zig-zag: rotate t first
        }
        rotate(t); // final rotation
    }
}

// Split: all keys ≤ key go to left, rest to right
pair<Node*,Node*> split(Node* t, int key) {
    if (!t) return {0, 0};
    // Find the rightmost node with key ≤ key
    Node* cur = t;
    Node* last = 0;
    while (cur) {
        push(cur);
        last = cur;
        if (key < cur->key) cur = cur->ch[0];
        else cur = cur->ch[1];
    }
    splay(last);
    if (last->key <= key) {
        Node* r = last->ch[1];
        if (r) r->par = 0;
        last->ch[1] = 0;
        pull(last);
        return {last, r};
    } else {
        Node* l = last->ch[0];
        if (l) l->par = 0;
        last->ch[0] = 0;
        pull(last);
        return {l, last};
    }
}

// Merge: all keys in l must be < all keys in r
Node* merge(Node* l, Node* r) {
    if (!l) return r;
    if (!r) return l;
    // Splay rightmost of l
    Node* cur = l;
    while (cur->ch[1]) { push(cur); cur = cur->ch[1]; }
    splay(cur);
    setChild(cur, 1, r);
    pull(cur);
    return cur;
}

// Insert a new key
Node* insert(Node* t, int key) {
    auto [l, r] = split(t, key - 1);
    Node* nd = new Node(key);
    setChild(nd, 0, l);
    setChild(nd, 1, r);
    pull(nd);
    return nd;
}

// Delete a key (assumes key exists)
Node* erase(Node* t, int key) {
    // Find and splay the target
    Node* cur = t;
    while (cur) {
        push(cur);
        if (key == cur->key) break;
        cur = cur->ch[key > cur->key];
    }
    if (!cur) return t;
    splay(cur);
    Node *l = cur->ch[0], *r = cur->ch[1];
    if (l) l->par = 0;
    if (r) r->par = 0;
    delete cur;
    return merge(l, r);
}

    For implicit arrays, replace key comparisons with sz(cur->ch[0]) comparisons, and the split function takes a position instead of a key. The rest is identical.

    Contest tip: In practice, use a global node pool (Node pool[MAXN]; int pool_ptr = 0;) instead of new/delete for better cache performance and no memory leaks.

    Splay Tree vs Treap

    Both splay trees and treaps support split/merge and can be used as implicit arrays. Here’s when to choose which:

    AspectSplay TreeTreap
    Balance guaranteeAmortized O(log n)Expected O(log n)
    Worst-case single opO(n)O(n) (very unlikely)
    Extra storageParent pointer onlyRandom priority per node
    Adaptive cachingYes — recent access near rootNo
    Required forLink-Cut TreesNot applicable
    Constant factorSlightly higher (parent pointers)Slightly lower
    DeterministicYesNo (randomized)
    Implementation~70 lines~50 lines
    PersistenceHard (splaying mutates)Easy (path-copy)
    Rule of thumb: Use treap for most problems (simpler, persistence-friendly). Use splay tree when you need Link-Cut Trees, or when the access pattern is highly skewed (splay’s adaptivity gives a real speedup).

    Splay trees are not the simplest balanced BST, and they’re not the fastest for uniform access patterns. But their self-adjusting nature, clean split/merge interface, and role as the backbone of Link-Cut Trees make them an essential tool in the competitive programmer’s arsenal. Once you understand the zig-zig insight, the rest follows naturally.
