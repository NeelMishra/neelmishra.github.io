    Mo’s Algorithm on Trees

    July 2025 · 22 min read

    Introduction

    Mo’s Algorithm is a powerful offline technique originally designed for range queries on arrays. By cleverly ordering the queries and maintaining a running answer, it achieves O((n + q)√n) time complexity without any persistent data structure.

    But what if our data lives on a tree instead of an array? Problems like “count distinct values on the path from u to v” or “find the sum of values in the subtree of u” arise constantly in competitive programming. With the right transformations, we can flatten the tree into an array and apply Mo’s algorithm directly.

    In this post we cover two flavours:

    • Subtree queries — using the standard Euler tour [tin[u], tout[u]].
    • Path queries — using a modified Euler tour where each node appears twice and an XOR-toggle trick decides which nodes are “active”.
    Prerequisites: You should be comfortable with DFS, Euler tours, LCA (Lowest Common Ancestor), and the basic idea of Mo’s algorithm on arrays. We include a quick recap below.

    Recap: Mo’s Algorithm on Arrays

    Given an array a[0..n-1] and q offline queries [l, r], Mo’s algorithm:

    1. Divides the array into blocks of size B ≈ √n.
    2. Sorts queries by (l / B, r) — primary key is the block of l, secondary key is r (with alternating order for even/odd blocks as an optimisation).
    3. Maintains two pointers curL, curR representing the current window. For each query in the sorted order it expands or contracts the window one element at a time using add(x) and remove(x).

    Because of the sort order, the left pointer moves at most O(q · B) steps total and the right pointer moves at most O(n · n/B) steps total. Setting B = √n balances these to O((n + q)√n).

    function mosAlgorithm(a, queries):
    B = ceil(√n) sort queries by (l / B, r) // alternating R for even/odd blocks curL = 0, curR = -1, answer = 0 for each query (l, r, idx): while curR < r: curR++; add(a[curR]) while curL > l: curL--; add(a[curL]) while curR > r: remove(a[curR]); curR-- while curL < l: remove(a[curL]); curL++ ans[idx] = answer

    The key insight for trees: if we can flatten the tree into an array so that interesting subsets (subtrees or paths) become contiguous ranges, we can reuse Mo’s machinery unchanged.

    Euler Tour Flattening for Subtree Queries

    The classic Euler tour assigns each node u two timestamps: tin[u] (when DFS enters) and tout[u] (when DFS leaves). Every node in the subtree of u has its tin value in the range [tin[u], tout[u]]. This is exactly a contiguous range!

    So for subtree queries we:

    1. Build the Euler tour array euler[i] = node whose tin equals i.
    2. Convert each query “subtree of u” to the range query [tin[u], tout[u]].
    3. Apply standard Mo’s on the Euler tour array.

    This is the simplest application of Mo’s on trees. The Euler tour guarantees that every subtree corresponds to a contiguous range, so we can directly apply the standard Mo’s with no modifications to the add/remove logic.

    Building the Euler Tour

    int timer = 0;
vector<int> tin(n), tout(n), euler_order;

void dfs(int u, int parent) {
    tin[u] = timer++;
    euler_order.push_back(u);
    for (int v : adj[u]) {
        if (v != parent) dfs(v, u);
    }
    tout[u] = timer - 1;
}

// Query "subtree of u" becomes range [tin[u], tout[u]]

    Path Queries — The XOR Toggle Trick

    Path queries are harder because a path from u to v does not correspond to a contiguous range in the standard Euler tour. The solution uses a modified Euler tour where each node appears twice: once at entry (tin) and once at exit (tout).

    The Modified Euler Tour

    We create an array of length 2n where position tin[u] stores the entry of node u and position tout[u] stores its exit. The key insight is the XOR / toggle trick:

    • Maintain a boolean array active[]. When we encounter node u in the range, we toggle it: if inactive, add it; if active, remove it.
    • A node that appears twice inside the range cancels out (toggled on, then off). A node that appears once remains active.
    • For the path u → v (assuming tin[u] ≤ tin[v]):
      • If u is an ancestor of v: use range [tin[u], tin[v]].
      • Otherwise: use range [tout[u], tin[v]] and manually add LCA(u, v).

    Why Does the Toggle Trick Work?

    Consider a node w that lies in the range [tout[u], tin[v]]:

    • If w is on the path u → v, exactly one of tin[w] or tout[w] lies in the range. It gets toggled once → active.
    • If w is NOT on the path, either both tin[w] and tout[w] are in the range (cancel out) or neither is. It gets toggled an even number of times → inactive.

    The LCA itself is special: both its tin and tout lie outside the range when u is not an ancestor of v, so we add it manually.

    Block Size & Query Sorting

    For the path variant the Euler tour has length 2n, so the optimal block size is:

    int BLOCK = max(1, (int)ceil(sqrt(2.0 * n)));

    Queries are sorted by:

    1. Primary key: L / BLOCK (block number of the left endpoint).
    2. Secondary key: R — but with alternating direction. If the block number is even, sort R ascending; if odd, sort R descending. This reduces the total pointer movement by avoiding worst-case zig-zag patterns.
    bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    return (ba & 1) ? (a.r > b.r) : (a.r < b.r);
}

    Add & Remove: Toggle-Based Approach

    For path Mo’s, add and remove do the same thing — they toggle the node. This is because expanding the window adds a new occurrence, and contracting it removes one.

    bool active[MAXN];
    int cnt[MAXN]; // frequency of each value
    int cur_ans = 0;

    function toggle(u):
    if active[u]: // removing node u cnt[val[u]]-- if cnt[val[u]] == 0: cur_ans-- else: // adding node u if cnt[val[u]] == 0: cur_ans++ cnt[val[u]]++ active[u] = !active[u]

    When processing Mo’s, we toggle the node at each position we pass through. The identity of the node at Euler tour position i is flat[i].

    Key Insight: The toggle-based approach means add(i) and remove(i) are identical: toggle(flat[i]). This makes the implementation elegant and avoids subtle bugs with expand vs. contract logic.

    C++ Implementation

    Subtree Queries

    #include <bits/stdc++.h>
using namespace std;

const int MAXN = 200005;
int n, q, val[MAXN], cnt[MAXN], cur_ans;
int tin[MAXN], tout[MAXN], euler[MAXN], timer_val;
vector<int> adj[MAXN];
int ans[MAXN];
int BLOCK;

struct Query {
    int l, r, idx;
};

void dfs(int u, int p) {
    tin[u] = timer_val;
    euler[timer_val++] = u;
    for (int v : adj[u])
        if (v != p) dfs(v, u);
    tout[u] = timer_val - 1;
}

void add(int pos) {
    int v = val[euler[pos]];
    if (cnt[v]++ == 0) cur_ans++;
}

void remove(int pos) {
    int v = val[euler[pos]];
    if (--cnt[v] == 0) cur_ans--;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    cin >> n >> q;
    for (int i = 1; i <= n; i++) cin >> val[i];
    for (int i = 0; i < n - 1; i++) {
        int u, v; cin >> u >> v;
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    dfs(1, 0);
    BLOCK = max(1, (int)ceil(sqrt((double)n)));

    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int u; cin >> u;
        queries[i] = {tin[u], tout[u], i};
    }

    sort(queries.begin(), queries.end(), [](const Query& a, const Query& b) {
        int ba = a.l / BLOCK, bb = b.l / BLOCK;
        if (ba != bb) return ba < bb;
        return (ba & 1) ? (a.r > b.r) : (a.r < b.r);
    });

    int curL = 0, curR = -1;
    for (auto& qr : queries) {
        while (curR < qr.r) add(++curR);
        while (curL > qr.l) add(--curL);
        while (curR > qr.r) remove(curR--);
        while (curL < qr.l) remove(curL++);
        ans[qr.idx] = cur_ans;
    }

    for (int i = 0; i < q; i++)
        cout << ans[i] << "\n";
}

    Path Queries (Full Implementation)

    #include <bits/stdc++.h>
using namespace std;

const int MAXN = 200005;
const int LOG  = 18;

int n, q, val[MAXN], cnt[MAXN], cur_ans;
int tin[MAXN], tout[MAXN], flat[2 * MAXN], timer_val;
bool active[MAXN];
vector<int> adj[MAXN];
int depth[MAXN], up[MAXN][LOG];
int ans[MAXN];
int BLOCK;

struct Query {
    int l, r, lca, idx;
};

// ---- LCA with binary lifting ----
void dfs(int u, int p, int d) {
    depth[u] = d;
    up[u][0] = p;
    for (int j = 1; j < LOG; j++)
        up[u][j] = up[up[u][j-1]][j-1];

    tin[u] = timer_val;
    flat[timer_val++] = u;

    for (int v : adj[u])
        if (v != p) dfs(v, u, d + 1);

    tout[u] = timer_val;
    flat[timer_val++] = u;
}

int lca(int u, int v) {
    if (depth[u] < depth[v]) swap(u, v);
    int diff = depth[u] - depth[v];
    for (int j = 0; j < LOG; j++)
        if ((diff >> j) & 1) u = up[u][j];
    if (u == v) return u;
    for (int j = LOG - 1; j >= 0; j--)
        if (up[u][j] != up[v][j]) {
            u = up[u][j];
            v = up[v][j];
        }
    return up[u][0];
}

// ---- Toggle ----
void toggle(int u) {
    if (active[u]) {
        cnt[val[u]]--;
        if (cnt[val[u]] == 0) cur_ans--;
    } else {
        if (cnt[val[u]] == 0) cur_ans++;
        cnt[val[u]]++;
    }
    active[u] = !active[u];
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    cin >> n >> q;
    for (int i = 1; i <= n; i++) cin >> val[i];
    for (int i = 0; i < n - 1; i++) {
        int u, v; cin >> u >> v;
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    dfs(1, 0, 0);
    BLOCK = max(1, (int)ceil(sqrt(2.0 * n)));

    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int u, v; cin >> u >> v;
        if (tin[u] > tin[v]) swap(u, v);
        int w = lca(u, v);
        if (w == u) {
            queries[i] = {tin[u], tin[v], -1, i};
        } else {
            queries[i] = {tout[u], tin[v], w, i};
        }
    }

    sort(queries.begin(), queries.end(), [](const Query& a, const Query& b) {
        int ba = a.l / BLOCK, bb = b.l / BLOCK;
        if (ba != bb) return ba < bb;
        return (ba & 1) ? (a.r > b.r) : (a.r < b.r);
    });

    int curL = 0, curR = -1;
    for (auto& qr : queries) {
        while (curR < qr.r) toggle(flat[++curR]);
        while (curL > qr.l) toggle(flat[--curL]);
        while (curR > qr.r) toggle(flat[curR--]);
        while (curL < qr.l) toggle(flat[curL++]);

        if (qr.lca != -1) toggle(qr.lca);
        ans[qr.idx] = cur_ans;
        if (qr.lca != -1) toggle(qr.lca);
    }

    for (int i = 0; i < q; i++)
        cout << ans[i] << "\n";
}

    Complexity Analysis

    Let the Euler tour length be N (for subtree: N = n; for paths: N = 2n). With block size B:

    Left Pointer Movement

    Within a single block, consecutive queries have l values in the same block (size B). The left pointer moves at most B per query → O(q · B) total.

    Right Pointer Movement

    Within a fixed block, the right pointer is monotonic (increasing or decreasing for alternating blocks). There are N / B blocks, and in each block R traverses at most N positions → O(N · N/B) total.

    Optimal Block Size

    Total cost = O(q · B + N² / B). Minimised when B = N / √q. For q ≈ n, this gives B ≈ √N and total O((N + q)√N).

    VariantTour LengthBlock SizeComplexity
    Subtreen√nO((n + q)√n)
    Path2n√(2n)O((n + q)√n)

    The LCA computation adds O(n log n) preprocessing and O(log n) per query, dominated by the Mo’s term.

    When to Use Mo’s on Trees

    Mo’s algorithm on trees is the right tool when:

    • Offline: All queries are known in advance (no online dependency between queries).
    • No updates: The tree values don’t change. (For updates, consider Mo’s with rollback or block decomposition with updates.)
    • Aggregate queries: You need something like count of distinct values, frequency sums, XOR of values, etc. The add/remove must run in O(1) or O(log n).
    • Subtree or path structure: The queries ask about subtrees or paths specifically.

    Pro tip: If the problem says “offline” and asks for distinct counts on paths, Mo’s on Trees is almost certainly the intended approach. Look for constraints like n, q ≤ 105 which fit the O(n√n) time.
