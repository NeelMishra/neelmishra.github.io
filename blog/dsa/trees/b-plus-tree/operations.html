    B+ Tree Operations

    March 2026 · 22 min read

    In the previous post we saw why B+ trees dominate database indexing. Now let's implement the operations. The key difference from B-tree operations: leaf splits copy the key up, internal splits move the key up.

    Insertion

    Step 1: Find the Right Leaf

    Walk down from the root using the internal node keys as guides. Always go all the way to a leaf.

    Step 2: Insert into Leaf

    Insert the key in sorted position within the leaf. If the leaf still has room, done.

    Step 3: Leaf Split (if full)

    When a leaf overflows, split it into two leaf nodes. The key difference from B-tree splits:

    • In a B-tree split, the median key moves up (removed from the original node).
    • In a B+ tree leaf split, the median key is copied up to the parent. It stays in the right leaf because all data must be in leaves.

    Internal Node Split

    If inserting a key into the parent causes the parent to overflow, it splits exactly like a B-tree: the median key moves up (not copied). Internal nodes are just routing guides with no data, so the key doesn't need to stay.

    Deletion

    Deletion always targets a leaf (since data only lives in leaves). The tricky parts are handling underflow and deciding whether to update internal routing keys.

    Simple Case: Leaf Has Room

    Remove the key. If the deleted key appears in an internal node, you can optionally leave it there (it still works as a valid separator) or replace it with the new leftmost key of the right subtree.

    Underflow: Borrow or Merge

    Same as B-tree underflow handling:

    • Borrow from sibling: If an adjacent leaf sibling has more than the minimum keys, redistribute. Update the parent separator key.
    • Merge with sibling: Combine two leaves and remove the separator from the parent. Update the parent's leaf links.
    A subtle difference from B-tree merge: In a B-tree merge, you pull a parent key down into the merged node (it's data). In a B+ tree, you just remove the separator from the parent because the merged leaf already contains all the data keys.

    Complexity

    OperationTimeDisk I/O
    Point lookupO(logm n)O(logm n)
    Range query (k results)O(logm n + k)O(logm n + k/m)
    InsertO(logm n)O(logm n)
    DeleteO(logm n)O(logm n)
    Full scanO(n)O(n/m)

    The range query cost is the standout: once you reach the first matching leaf, you just follow the linked list. Each leaf page contains m keys, so scanning k results costs only k/m disk reads.
