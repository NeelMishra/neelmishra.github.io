    B+ Trees: The Database Index Standard

    March 2026 · 20 min read

    In the B-tree series, we learned how multi-way trees keep trees shallow for disk-based storage. B+ trees take that idea further with two critical modifications that make them the dominant index structure in virtually every relational database:

    1. All data lives in the leaves. Internal nodes are a pure routing index.
    2. Leaves are linked together in a doubly-linked list for fast sequential access.

    B-Tree vs B+ Tree

    Why B+ Trees Dominate Databases

    1. Higher Fanout

    Internal nodes in a B+ tree contain only keys and child pointers. No data pointers, no row payloads. This means you can fit more keys per page, which means more children per node, which means a shorter tree.

    Imagine 4 KB pages. In a B-tree, each entry is (key + data pointer + child pointer) = maybe 24 bytes, so ~170 entries per page. In a B+ tree internal node, each entry is (key + child pointer) = 16 bytes, so ~250 entries. The B+ tree is shallower for the same data.

    2. Range Queries

    This is the killer feature. SELECT * FROM users WHERE age BETWEEN 25 AND 35 translates to: find the leaf containing 25, then follow the linked leaf chain until you pass 35. No going back up the tree. Pure sequential I/O.

    3. Sequential Scans

    Full table scans (SELECT * with no WHERE) just follow the leaf chain from start to end. Purely sequential disk reads, which modern storage (both HDD and SSD) handles efficiently with prefetching.

    4. Consistent Search Depth

    In a B-tree, some keys are found in internal nodes (shorter path) and others in leaves (longer path). Search time varies. In a B+ tree, every search goes to a leaf, so search time is always exactly h levels. More predictable for query planning.

    Structure Details

    A B+ tree of order m has:

    • Internal nodes: keys + child pointers only. Each node has at most m children and m-1 keys. At least ⌈m/2⌉ children.
    • Leaf nodes: keys + data pointers (or actual row data). Linked together in order. Same fill rules.
    • Key duplication: Internal node keys are copies of leaf keys, used purely for routing. Deleting a key from a leaf does not necessarily require removing it from internal nodes.
    • All data in leaves: A full in-order traversal reads only the leaf chain.
    struct BPlusLeaf {
    int keys[MAX_KEYS];
    DataPtr data[MAX_KEYS];    // actual data or row pointers
    BPlusLeaf* next;           // linked list to next leaf
    int n;                     // current key count
};

struct BPlusInternal {
    int keys[MAX_KEYS];
    void* children[MAX_KEYS+1]; // child pointers (internal or leaf)
    int n;
    bool isLeaf;
};

    Operations Overview

    Search

    Same as B-tree search, except we always go all the way down to a leaf. Internal nodes only tell us which child to follow; we don't stop at internal nodes even if we match a key.

    Insert

    Same split-on-full logic as B-trees, with one difference: when a leaf splits, the median key is copied up (not moved), so it stays in the new right leaf too. When an internal node splits, the median key is moved up (same as B-tree).

    Delete

    Remove the key from the leaf. If the leaf underflows, borrow from a sibling or merge. Internal node keys that are copies of deleted keys can optionally remain as routing guides, since they only need to separate ranges.
