    B+ Tree Patterns & Real-World Usage

    March 2026 · 18 min read

    We've built the B+ tree and its operations. Now let's see how real databases actually use them, understand the difference between clustered and non-clustered indices, and cover the patterns that come up in interviews.

    Clustered vs Non-Clustered Indices

    Clustered Index

    A clustered index determines the physical order of data on disk. The leaf nodes of the B+ tree contain the actual row data. There can be only one clustered index per table (you can only sort data one way on disk).

    Non-Clustered (Secondary) Index

    A non-clustered index is a separate B+ tree where leaf nodes contain pointers (row IDs or primary keys) back to the actual data. You can have many secondary indices on a table.

    PropertyClusteredNon-Clustered
    Leaf dataActual row dataPointers (PKs or row IDs)
    Count per tableExactly 1Many
    Range scansVery fast (sequential I/O)May need random I/O for row lookup
    Insert costMay need page splitsSame, plus update all affected secondary indices

    How Real Databases Use B+ Trees

    MySQL / InnoDB

    • Every table is stored as a clustered B+ tree on the primary key.
    • If you don't define a PK, InnoDB creates a hidden 6-byte row ID as the clustered key.
    • Secondary indices store the primary key as their data pointer (not a physical row address).
    • Page size: 16 KB (configurable).

    PostgreSQL

    • Tables are stored as heaps (unsorted by any index). No clustered index by default.
    • CREATE INDEX creates a separate B+ tree that points to heap tuple IDs (ctid).
    • CLUSTER command reorders the heap to match an index, but it's a one-time operation (not maintained on writes).
    • Page size: 8 KB.

    SQLite

    • Tables with an INTEGER PRIMARY KEY use B-trees (rowid tables).
    • Indices use B+ trees (called "index B-trees" in SQLite docs).
    • WITHOUT ROWID tables use a clustered B+ tree on the PK.
    • Page size: 4 KB default.

    The Buffer Pool

    Databases don't read from disk on every B+ tree access. They maintain a buffer pool (in-memory page cache). Hot pages (like root and top-level internal nodes) stay in memory permanently. Only leaf pages and rarely-accessed internal pages cause actual disk I/O.

    For a B+ tree with a billion rows and 200 keys per page:

    • Root: 1 page (always in memory)
    • Level 1: ~200 pages (usually in memory)
    • Level 2: ~40,000 pages (often in memory)
    • Leaves: ~5,000,000 pages (read from disk as needed)

    So in practice, a point query on a billion-row table typically requires only 1-2 actual disk reads (the leaf page and maybe one level above), not the theoretical 4-5.

    Interview Patterns

    Pattern 1: "How does a database index work?"

    Answer: A B+ tree where internal nodes hold keys for routing and leaf nodes hold the actual data (or pointers to data). Keys are sorted within each node. Leaves are linked for range scans. Each node fits in one disk page (4-16 KB).

    Key operations: point lookup (O(log n) disk reads), range scan (O(log n + k/page_size)), insert/delete (O(log n) with potential splits/merges).

    Pattern 2: "Clustered vs non-clustered index?"

    Clustered: determines physical order of data. Leaf = actual row. One per table. Best for range queries on the indexed column.

    Non-clustered: separate structure with pointers. Multiple allowed per table. Point lookup may require a "bookmark lookup" back to the clustered index. Covering index avoids this by including all needed columns.

    Pattern 3: "Why not use a hash index instead?"

    Hash indices give O(1) for exact lookups, but they cannot do range queries. WHERE x BETWEEN 10 AND 20 requires a full table scan with a hash index. B+ trees give O(log n) for point lookups and O(log n + k) for range queries. Since most real workloads involve both, B+ trees are the default choice.

    Some databases (PostgreSQL, Memory engine in MySQL) offer hash indices as an option for workloads that are purely point lookups.

    Pattern 4: "What happens when you create an index on a large table?"

    The database scans the entire table, sorts the values of the indexed column, and builds a new B+ tree from bottom up (bulk loading). This is an O(n log n) operation that can take a long time and significant I/O for large tables. It also uses temporary disk space for the sort.

    During bulk loading, leaves are filled sequentially (often to ~90% full) and internal nodes are created bottom-up, which is much faster than inserting one key at a time.

    Pattern 5: "Why do composite indices care about column order?"

    A composite index on (last_name, first_name) sorts by last_name first, then by first_name within each last_name. This means:

    • WHERE last_name = 'Smith' → can use the index (prefix match)
    • WHERE last_name = 'Smith' AND first_name = 'John' → can use the index fully
    • WHERE first_name = 'John' → cannot use the index (not a prefix)

    This is the "leftmost prefix" rule. It follows directly from how B+ tree keys are sorted: compound keys are compared left to right.
