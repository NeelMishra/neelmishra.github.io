    Trie Patterns & Interview Questions

    March 2026 · 18 min read

    Tries appear in interviews more than you might expect. They're the go-to when you hear "prefix," "autocomplete," "dictionary," or "word search." Let's cover the patterns.

    Pattern 1: Implement a Trie (LeetCode 208)

    The most common trie interview question. Implement three methods: insert, search, and startsWith.

    class Trie {
    struct Node {
        Node* ch[26] = {};
        bool end = false;
    };
    Node* root;
public:
    Trie() : root(new Node()) {}

    void insert(const string& word) {
        Node* cur = root;
        for (char c : word) {
            if (!cur->ch[c-'a']) cur->ch[c-'a'] = new Node();
            cur = cur->ch[c-'a'];
        }
        cur->end = true;
    }

    bool search(const string& word) {
        Node* n = find(word);
        return n && n->end;
    }

    bool startsWith(const string& prefix) {
        return find(prefix) != nullptr;
    }

private:
    Node* find(const string& s) {
        Node* cur = root;
        for (char c : s) {
            if (!cur->ch[c-'a']) return nullptr;
            cur = cur->ch[c-'a'];
        }
        return cur;
    }
};
    Interview tip: Extract a find() helper that both search and startsWith use. The only difference is whether you check isEnd afterward. Clean code impresses interviewers.

    Pattern 2: Word Search II (LeetCode 212)

    Given a 2D board of characters and a list of words, find all words that can be formed by connecting adjacent cells.

    The brute force: for each word, do a DFS on the board. With k words and an m×n board, that's O(k · m · n · 4L).

    The trie approach: insert all words into a trie, then DFS the board once while walking the trie simultaneously. When you reach a trie dead-end, prune. When you reach an end-of-word node, record the word.

    void dfs(vector<vector<char>>& board, TrieNode* node, int r, int c,
         string& path, vector<string>& result) {
    char ch = board[r][c];
    if (ch == '#' || !node->children[ch - 'a']) return;  // visited or no trie path

    node = node->children[ch - 'a'];
    if (node->isEnd) {
        result.push_back(path + ch);
        node->isEnd = false;  // avoid duplicates
    }

    board[r][c] = '#';  // mark visited
    path.push_back(ch);
    // explore 4 directions
    for (auto [dr, dc] : dirs)
        if (inBounds(r+dr, c+dc))
            dfs(board, node, r+dr, c+dc, path, result);
    path.pop_back();
    board[r][c] = ch;  // unmark
}

    Pattern 3: Search with Wildcards (LeetCode 211)

    Support search("b.d") where . matches any character. On a dot, try all 26 children recursively.

    bool searchWild(TrieNode* node, const string& word, int i) {
    if (i == word.size()) return node->isEnd;
    if (word[i] == '.') {
        for (int c = 0; c < 26; c++)
            if (node->children[c] && searchWild(node->children[c], word, i+1))
                return true;
        return false;
    }
    if (!node->children[word[i]-'a']) return false;
    return searchWild(node->children[word[i]-'a'], word, i+1);
}

    Pattern 4: Word Break (LeetCode 139)

    Given a string s and a dictionary of words, determine if s can be segmented into dictionary words.

    Build a trie from the dictionary. Use DP where dp[i] = true if s[0..i-1] can be segmented. For each position i where dp[i] is true, walk the trie from s[i] onwards to find all dictionary words starting at position i.

    bool wordBreak(string s, vector<string>& wordDict) {
    Trie trie;
    for (auto& w : wordDict) trie.insert(w);

    int n = s.size();
    vector<bool> dp(n + 1, false);
    dp[0] = true;

    for (int i = 0; i < n; i++) {
        if (!dp[i]) continue;
        TrieNode* cur = trie.root;
        for (int j = i; j < n; j++) {
            if (!cur->children[s[j]-'a']) break;
            cur = cur->children[s[j]-'a'];
            if (cur->isEnd) dp[j+1] = true;
        }
    }
    return dp[n];
}
    Why a trie here? Without a trie, checking "does s[i..j] exist in dictionary?" is O(j-i) per check with a hash set. With a trie, we walk s[i..j] incrementally, checking every prefix in one pass. This avoids redundant substring creation.

    Pattern 5: Maximum XOR of Two Numbers (LeetCode 421)

    Given an array of integers, find the maximum XOR of any two elements.

    Insert all numbers into a bitwise trie (32 levels, one per bit). For each number, greedily traverse the trie choosing the opposite bit at each level to maximize XOR.

    Time: O(32n) = O(n). Much better than the O(n2) brute force.

    Pattern 6: Autocomplete with Ranking

    Real autocomplete doesn't return all matches, it returns the top k most relevant. Modifications:

    • Store a frequency/score at each end-of-word node.
    • At each internal node, cache the top-k words in its subtree (precomputed on insert).
    • On prefix search, return the cached top-k from the prefix node in O(p + k).

    This is how Google Search autocomplete works at a simplified level.

    When to Reach for a Trie

    Signal in problemWhy trie fits
    "Find all words starting with..."Prefix search is the trie's superpower
    "Dictionary of words" + search/matchTrie shares prefixes, avoids redundant comparisons
    "Longest prefix match"Walk the trie greedily; IP routing, LPM problems
    "Maximum XOR"Bitwise trie, greedy opposite-bit traversal
    "Word search on a grid"Trie prunes DFS branches efficiently
    "Can string be segmented into dictionary words"Trie walks substrings incrementally
    "Count distinct substrings"Suffix trie (or suffix array) approach
    Trie vs hash set rule of thumb: If you only need exact lookup, use a hash set. If you need prefix operations (search, count, autocomplete, longest match), use a trie. If you need substring operations, consider a suffix trie or suffix array.
