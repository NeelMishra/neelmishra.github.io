    Trie Variants

    March 2026 · 20 min read

    The standard trie from the previous posts is clean and fast, but it wastes memory: each node allocates 26 child pointers even if most are null. For large datasets, this memory overhead matters. Let's explore variants that fix this.

    Compressed Trie (Radix Tree / Patricia Tree)

    The core problem with a standard trie: long chains of single-child nodes are wasteful. If "application" is the only word starting with "applic", we waste 6 nodes that each have exactly one child.

    A compressed trie (also called a radix tree or Patricia tree) solves this by collapsing single-child chains into a single edge labeled with the whole substring.

    The key optimization: edges store substrings, not single characters. A node only exists at branching points (where the path forks) and at end-of-word markers.

    When to Use

    • Large alphabets (Unicode strings have 1M+ possible characters per node)
    • Sparse tries with many long, non-branching chains
    • IP routing tables (Linux kernel uses a compressed trie variant called LC-trie)

    Ternary Search Tree (TST)

    A ternary search tree is a trie-BST hybrid. Each node stores one character and has three children:

    • Left: characters less than this node's character
    • Middle: characters equal (the "matched" path, moves to the next character)
    • Right: characters greater than this node's character

    Advantages over standard tries:

    • Space: Each node has exactly 3 pointers, not 26 (or 256). Huge savings.
    • No alphabet restriction: Works with any character set without allocating huge arrays.
    • Balanced: If built from sorted keys (like a balanced BST), search is O(m + log n).

    Disadvantage: search is slower than a standard trie (O(m + log n) vs O(m)) because we do BST-style comparisons at each level.

    Suffix Trie

    A suffix trie stores all suffixes of a string. For "banana", it stores: banana, anana, nana, ana, na, a. This allows O(m) substring search (not just prefix search).

    "Does pattern P appear anywhere in text T?" becomes "does P exist as a prefix of any suffix of T?" which is a standard trie prefix search.

    Space problem: A suffix trie for a string of length n has O(n2) nodes. That's too much for long strings. The solution is the suffix tree, a compressed suffix trie that uses O(n) space, or the even more practical suffix array.

    Bitwise Trie (for Integers)

    Tries aren't just for strings. A bitwise trie stores integers by their binary representation, one bit per level. This is the foundation of:

    • IP routing: Longest prefix match on 32-bit IP addresses. Each level checks one bit.
    • XOR problems: Find maximum XOR pair in an array. Insert all numbers into a bitwise trie, then for each number greedily choose the opposite bit at each level.
    • Van Emde Boas trees: Conceptually related to bitwise tries with clever caching.
    // Maximum XOR: insert all numbers, then query each
struct BitTrieNode {
    BitTrieNode* children[2];  // bit 0 and bit 1
};

void insert(BitTrieNode* root, int num) {
    BitTrieNode* cur = root;
    for (int i = 31; i >= 0; i--) {
        int bit = (num >> i) & 1;
        if (!cur->children[bit])
            cur->children[bit] = new BitTrieNode();
        cur = cur->children[bit];
    }
}

int maxXOR(BitTrieNode* root, int num) {
    BitTrieNode* cur = root;
    int result = 0;
    for (int i = 31; i >= 0; i--) {
        int bit = (num >> i) & 1;
        int want = 1 - bit;  // want opposite bit for max XOR
        if (cur->children[want]) {
            result |= (1 << i);
            cur = cur->children[want];
        } else {
            cur = cur->children[bit];
        }
    }
    return result;
}
