    Red-Black Trees: Properties & Intuition

    March 2026 · 22 min read

    Red-black trees are the most widely deployed self-balancing BST in production software. std::map in C++, TreeMap in Java, the Linux kernel's process scheduler, and the Completely Fair Scheduler all use red-black trees. If you've used a sorted dictionary in any major language, you've used a red-black tree.

    Yet they have a reputation for being confusing. The coloring rules seem arbitrary, the fix-up cases feel like memorization, and the connection to "balance" is murky. This series fixes that by building your intuition before the mechanics.

    The Five Properties

    A red-black tree is a BST where every node is colored red or black, and the following five properties hold:

    1. Every node is either red or black.
    2. The root is black.
    3. Every leaf (NIL) is black. (We use sentinel NIL nodes, not actual null pointers.)
    4. Red nodes cannot have red children. (No two consecutive reds on any path.)
    5. Every path from a node to its descendant NIL leaves has the same number of black nodes (called the black-height).

    Why Do These Rules Guarantee O(log n)?

    Property 5 (equal black-height on all paths) is the balance guarantee. Let's see why.

    Consider the longest possible path from root to a leaf. It alternates red and black (because of property 4: no consecutive reds). So it has at most 2 · bh nodes, where bh is the black-height.

    The shortest possible path is all black nodes: exactly bh nodes.

    The height bound: The longest path is at most twice the shortest path. This means the tree height h ≤ 2 · log2(n+1). Not as tight as AVL (1.44 log n), but still O(log n).

    The 2-3-4 Tree Connection

    Red-black trees seem arbitrary until you see that they're just binary encodings of 2-3-4 trees (B-trees of order 4). Every 2-3-4 tree maps to exactly one red-black tree, and vice versa.

    2-3-4 NodeRed-Black Encoding
    2-node (1 key, 2 children)One black node
    3-node (2 keys, 3 children)Black node with one red child
    4-node (3 keys, 4 children)Black node with two red children

    This mapping explains everything:

    • Property 4 (no red-red): A 4-node has at most 3 keys. A red child of a red node would make a "5-node," which doesn't exist in 2-3-4 trees.
    • Property 5 (equal black-height): In a 2-3-4 tree, all leaves are at the same depth. Since black nodes correspond to 2-3-4 levels, every path has the same number of black nodes.
    • Recoloring = splitting: When we recolor a black node with two red children (flip all three colors), we're splitting a 4-node in the 2-3-4 view.

    The Red-Black Node

    enum Color { RED, BLACK };

struct RBNode {
    int key;
    Color color;
    RBNode* left;
    RBNode* right;
    RBNode* parent;   // needed for fix-up traversals

    RBNode(int k) : key(k), color(RED), left(NIL), right(NIL), parent(nullptr) {}
};

// Sentinel NIL node (shared by all leaf pointers)
RBNode* NIL = new RBNode(0);
NIL->color = BLACK;

    Two things differ from AVL nodes:

    • Color instead of height: One bit of information (vs an integer for AVL).
    • Parent pointer: Red-black fix-ups need to access uncle/grandparent nodes, which requires walking up. AVL trees do this via recursion.

    Red-Black vs AVL: The Trade-off

    PropertyRed-BlackAVL
    Max height2 log n (looser)1.44 log n (tighter)
    SearchSlightly slower (taller)Faster
    Insert rotations≤ 2≤ 2
    Delete rotations≤ 3Up to O(log n)
    Recolorings (insert)O(log n)N/A
    Memory per node1 bit (color)1 int (height)
    Used instd::map, Java TreeMap, LinuxDatabases, read-heavy apps
    The practical difference: Red-black trees do at most 3 rotations per delete (the rest is cheap recoloring). AVL trees may need O(log n) rotations per delete. For write-heavy workloads, this matters. For read-heavy, AVL's shorter tree wins.
