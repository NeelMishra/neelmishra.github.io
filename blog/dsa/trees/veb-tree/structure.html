    The Recursive Structure

    March 2026 · 22 min read

    In the previous post, we saw the key idea: split the universe into √u clusters, each of size √u, and maintain a summary that tracks which clusters are non-empty. Now let's formalize this into an actual data structure.

    Proto-vEB: The Simplified Version

    Before the full Van Emde Boas tree, let's build a simpler version called proto-vEB. It has the right recursive structure but is missing the min/max optimization that makes operations fast.

    The Definition

    A proto-vEB(u) node for universe size u has:

    • u: the universe size
    • cluster[0 .. √u - 1]: an array of √u child proto-vEB(√u) structures, one per cluster
    • summary: a single proto-vEB(√u) that tracks which clusters are non-empty

    Base case: when u = 2, just store a simple 2-bit array. No more recursion needed.

    Mapping Keys to Clusters

    For any key x in {0, ..., u-1}, we compute:

    high(x) = floor(x / sqrt(u))    // which cluster
low(x)  = x mod sqrt(u)          // position within cluster
index(i, j) = i * sqrt(u) + j    // reconstruct key from cluster i, position j

    These three functions are the address translation layer of the entire structure. Every operation uses them to navigate from the top level down into the right cluster.

    The Problem with Proto-vEB

    Proto-vEB has the right structure, but its operations are too slow. Consider insert: we need to insert into the cluster and update the summary. That's two recursive calls at each level:

    proto-vEB-insert(V, x):
    if V.u == 2:
        V.A[x] = 1
        return
    proto-vEB-insert(V.cluster[high(x)], low(x))   // recurse into cluster
    proto-vEB-insert(V.summary, high(x))            // recurse into summary

    The recurrence is T(u) = 2T(√u) + O(1). This solves to T(u) = O(log u), not O(log log u). Two recursive calls kill us; we're no better than a balanced BST.

    The Min/Max Optimization

    The full Van Emde Boas tree fixes this with one brilliant trick: cache the minimum and maximum directly in each node, and critically, do NOT store the min inside any cluster.

    A full vEB(u) node has:

    • u: universe size
    • min: the smallest element (stored directly, not in any cluster)
    • max: the largest element (stored in a cluster AND here for O(1) access)
    • summary: a vEB(√u) tracking which clusters are non-empty
    • cluster[0..√u-1]: array of √u child vEB(√u) structures

    Why Not Store Min in a Cluster?

    This is the single most important design choice in the entire data structure. By keeping min outside the clusters, we get a huge benefit for insert:

    When we insert into an empty vEB node, we just set min = max = x. No recursive call at all. No cluster update. No summary update. Pure O(1).

    And when we insert into a non-empty node, we might swap x with the current min (to keep the smaller value as min), then insert the other value into the appropriate cluster. If that cluster was previously empty, we need to update the summary. But here's the trick: inserting into an empty cluster is O(1) (just set its min), so the summary update is the only recursive call. We went from two recursive calls to one.

    The recurrence is now T(u) = T(√u) + O(1). Either we recurse into the cluster (and the summary insert is O(1) because that cluster becoming non-empty means it was empty, so its min-set is O(1)), or we don't recurse at all. One path, not two.

    The Node in Code

    struct vEB {
    int u;               // universe size
    int min, max;        // cached min and max (-1 if empty)
    vEB* summary;        // which clusters are non-empty
    vector<vEB*> cluster; // sqrt(u) child structures

    vEB(int u) : u(u), min(-1), max(-1), summary(nullptr) {
        if (u > 2) {
            int upper = ceil(sqrt(u));
            int lower = floor(sqrt(u));
            summary = new vEB(upper);
            cluster.resize(upper);
            for (int i = 0; i < upper; i++)
                cluster[i] = new vEB(lower);
        }
    }
};

    For u = 2, we don't create children. min and max alone handle two elements. For larger u, we allocate √u clusters and one summary, each managing a universe of size √u.

    Base Case: u = 2

    When the universe is just {0, 1}, the structure is trivial: min and max directly encode which of the two values are present. No clusters, no summary, no recursion.

    Space: O(u)

    How much memory does a vEB tree use? A vEB(u) node creates √u clusters and 1 summary, each of size vEB(√u). The recurrence for space S(u) is:

    S(u) = (sqrt(u) + 1) * S(sqrt(u)) + O(sqrt(u))

    This solves to S(u) = O(u). The tree uses space proportional to the universe size, not the number of elements stored. For u = 216 = 65,536, that's manageable. For u = 232, it's 4GB of nodes, which is large but can be resolved with hashing (the "RS-vEB" variant, beyond our scope).
