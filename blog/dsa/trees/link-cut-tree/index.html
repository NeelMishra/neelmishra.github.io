    Link-Cut Trees

    July 2026 · 35 min read

    The Dynamic Forest Problem

    Imagine you have a forest of rooted trees. You need to support three operations efficiently:

    • Link(u, v): Add an edge between node u and node v (merging two trees).
    • Cut(u, v): Remove the edge between u and v (splitting a tree into two).
    • Path query(u, v): Compute an aggregate (sum, max, min, xor) along the path from u to v.

    With a static tree, Heavy-Light Decomposition handles path queries in O(log2 n). But what if the tree itself is changing? Edges appear and disappear. HLD falls apart because it relies on a fixed tree structure decomposed up front.

    The Link-Cut Tree, introduced by Sleator and Tarjan in 1983, solves all three operations in amortized O(log n) time. It's the gold standard data structure for dynamic forests in competitive programming.

    Why not Euler Tour Trees? Euler tour trees also support dynamic forests, but they can only handle subtree aggregates, not path aggregates. Link-Cut Trees handle path queries natively, making them the go-to structure when you need sums, maxes, or XORs along root-to-node paths.

    Preferred Paths & Splay Trees

    The key insight behind Link-Cut Trees is a decomposition of the tree into preferred paths. At any moment, each node has at most one preferred child. The chain of preferred edges from a node downward forms a preferred path.

    Each preferred path is stored in a splay tree, keyed by depth (nodes deeper in the original tree appear further right in the splay tree). The splay trees are connected by path-parent pointers: the root of each splay tree has a pointer to the parent of the topmost node on that preferred path in the original tree.

    The critical invariant: the in-order traversal of each splay tree gives the nodes in increasing depth order within the original tree. So the leftmost node in a splay tree is the shallowest (closest to root), and the rightmost is the deepest.

    There are two types of edges in this representation:

    • Splay edges (solid): connect nodes within the same preferred path. These are bidirectional — both parent and child know about each other in the splay tree.
    • Path-parent edges (dashed orange): connect the root of one splay tree to a node in another splay tree. These are one-directional — the child knows its parent, but the parent does not know about this child. This is the key distinction.
    Path-parent vs. splay parent: If node->parent exists and node->parent->left != node && node->parent->right != node, then this is a path-parent pointer. Otherwise it's a normal splay tree edge. This one check is how the implementation distinguishes the two.

    The Splay Tree Refresher

    Before diving into the Link-Cut Tree operations, recall how splay trees work. A splay tree is a self-adjusting BST where every access operation brings the accessed node to the root via a sequence of rotations:

    • Zig: Single rotation when the node is a child of the root.
    • Zig-zig: Two same-direction rotations (both left-left or both right-right).
    • Zig-zag: Two opposite-direction rotations (left-right or right-left).

    In a Link-Cut Tree, the splay operation is modified slightly: we only splay within a single auxiliary splay tree (we stop at the splay tree root, not when we hit a path-parent edge). The implementation checks isRoot(x) — whether x is the root of its splay tree — to know when to stop.

    bool isRoot(int x) {
    return ch[fa[x]][0] != x && ch[fa[x]][1] != x;
}

void rotate(int x) {
    int y = fa[x], z = fa[y], k = (ch[y][1] == x);
    if (!isRoot(y)) ch[z][ch[z][1] == y] = x;
    fa[x] = z;
    ch[y][k] = ch[x][!k]; if (ch[x][!k]) fa[ch[x][!k]] = y;
    ch[x][!k] = y; fa[y] = x;
    pushUp(y); pushUp(x);
}

void splay(int x) {
    pushDownAll(x);  // push lazy tags from root to x
    while (!isRoot(x)) {
        int y = fa[x], z = fa[y];
        if (!isRoot(y))
            rotate((ch[z][0]==y) == (ch[y][0]==x) ? y : x);
        rotate(x);
    }
    pushUp(x);
}

    The isRoot check is elegant: if x's parent doesn't claim x as its left or right child, then the edge from x to its parent is a path-parent edge, meaning x is a splay tree root.

    The Access Operation

    Access is the fundamental operation of Link-Cut Trees. access(v) restructures the preferred paths so that the entire path from v up to the root of its tree becomes a single preferred path, with v at the bottom. After access(v), v and the root are in the same splay tree.

    The algorithm is beautifully simple:

    void access(int x) {
    int last = 0;
    for (int y = x; y; y = fa[y]) {
        splay(y);       // bring y to root of its splay tree
        ch[y][1] = last; // replace right child (deeper part) with last
        pushUp(y);       // recalculate aggregates
        last = y;
    }
    splay(x);  // bring x to root of entire path
}

    What's happening step by step:

    1. Start at x. Walk up via parent pointers (including path-parent pointers).
    2. At each splay tree root encountered, splay it to the root of that splay tree.
    3. Cut its old right subtree (detaching the old preferred path below) and attach last (the splay tree we just came from) as the new right subtree.
    4. This effectively switches the preferred child, making the path from this node to x the new preferred path.
    5. Finally, splay x to the root of the combined splay tree.

    Cut & Link

    Link(u, v)

    link(u, v) makes u a child of v (u must be the root of its tree). The implementation is simple: make u the root of its represented tree, access v, then set v as the parent of u. Since u is a root, it has no path-parent pointer, so this just sets one.

    void link(int x, int y) {
    makeRoot(x);  // make x root of its tree
    access(y);    // bring y to splay root
    fa[x] = y;    // x's parent = y (path-parent edge)
}

    Cut(u, v)

    cut(u, v) removes the edge between u and v. We make u the root, access v, and then v's left child in the splay tree is u (since u is the shallowest node on the path from root=u to v). Detach it:

    void cut(int x, int y) {
    makeRoot(x);
    access(y);     // now x is leftmost in y's splay tree
    ch[y][0] = 0;  // detach x from y
    fa[x] = 0;     // x has no parent now
    pushUp(y);
}
    Safety check: In competitive programming, you often know the edge exists. But if you need to verify, after makeRoot(x); access(y); check that ch[y][0] == x && fa[x] == y && !ch[x][1] (x is the immediate left child of y with no right subtree, meaning they are directly connected).

    Path Queries

    This is where Link-Cut Trees really shine. To query an aggregate on the path from u to v:

    1. makeRoot(u) — re-root the tree at u.
    2. access(v) — now the entire u→v path is in one splay tree.
    3. The answer is just the aggregate stored at the root of that splay tree (v after access).
    // Query path aggregate from x to y
int pathQuery(int x, int y) {
    makeRoot(x);
    access(y);
    return val[y];  // val[y] holds the aggregate of y's splay subtree
}

// Update a single node's value
void update(int x, int w) {
    access(x);       // bring x to splay root
    self[x] = w;     // update x's own value
    pushUp(x);       // recalculate subtree aggregate
}

    The pushUp function maintains the aggregate. For example, for path sum and path max:

    void pushUp(int x) {
    sum[x] = sum[ch[x][0]] + sum[ch[x][1]] + self[x];
    mx[x] = max({mx[ch[x][0]], mx[ch[x][1]], self[x]});
}

    Since the splay tree contains exactly the nodes on the path from u to v after makeRoot(u); access(v);, the aggregate at the root covers the entire path. This is the O(log n) path query — no HLD needed, no segment tree on top, no log2 factor.

    Make Root

    makeRoot(v) re-roots the represented tree at v. This is what enables arbitrary path queries (not just root-to-v). The implementation:

    void makeRoot(int x) {
    access(x);   // path from old root to x is now one splay tree
    reverse(x);  // reverse the splay tree → x is now the root
}

    After access(x), the splay tree contains the path from the tree's root down to x, keyed by depth. Reversing this splay tree (swapping left/right children throughout) makes x the shallowest node, effectively re-rooting the represented tree at x.

    The reversal is done lazily with a flip tag:

    void reverse(int x) {
    rev[x] ^= 1;
}

void pushDown(int x) {
    if (rev[x]) {
        swap(ch[x][0], ch[x][1]);
        if (ch[x][0]) rev[ch[x][0]] ^= 1;
        if (ch[x][1]) rev[ch[x][1]] ^= 1;
        rev[x] = 0;
    }
}
    Why does reversal re-root the tree? The splay tree is keyed by depth. Root is leftmost (shallowest), x is rightmost (deepest). After reversal, x becomes leftmost — it's now the shallowest node on the path, meaning the root. The parent-child relationships along this path are all flipped, which is exactly what re-rooting does.

    Find Root

    findRoot(v) returns the root of the tree containing v. This is essential for connectivity queries — two nodes are connected if and only if they have the same root.

    int findRoot(int x) {
    access(x);
    pushDown(x);
    while (ch[x][0]) {
        x = ch[x][0];
        pushDown(x);
    }
    splay(x);  // amortize: bring root to splay root
    return x;
}

    After access(x), the splay tree contains the path from the tree root to x. The tree root is the leftmost (shallowest) node in this splay tree. We walk left until we can't anymore — that's the root.

    The final splay(x) is important: without it, repeated findRoot calls could degenerate to O(n) per call. With it, the amortized cost stays O(log n).

    Connectivity check is then trivial:

    bool connected(int x, int y) {
    return findRoot(x) == findRoot(y);
}

    C++ Implementation

    Here's a full, contest-ready Link-Cut Tree with path sum and path max. Copy-paste friendly for competitive programming:

    struct LinkCutTree {
    static const int MAXN = 100005;
    int ch[MAXN][2], fa[MAXN];
    int self_val[MAXN]; // node's own value
    int sum[MAXN];      // subtree sum (of splay tree = path aggregate)
    int mx[MAXN];       // subtree max
    bool rev[MAXN];     // lazy reversal tag
    int sz[MAXN];       // subtree size

    bool isRoot(int x) {
        return ch[fa[x]][0] != x && ch[fa[x]][1] != x;
    }

    void pushUp(int x) {
        if (!x) return;
        sum[x] = sum[ch[x][0]] + sum[ch[x][1]] + self_val[x];
        mx[x] = self_val[x];
        if (ch[x][0]) mx[x] = max(mx[x], mx[ch[x][0]]);
        if (ch[x][1]) mx[x] = max(mx[x], mx[ch[x][1]]);
        sz[x] = sz[ch[x][0]] + sz[ch[x][1]] + 1;
    }

    void pushDown(int x) {
        if (x && rev[x]) {
            swap(ch[x][0], ch[x][1]);
            if (ch[x][0]) rev[ch[x][0]] ^= 1;
            if (ch[x][1]) rev[ch[x][1]] ^= 1;
            rev[x] = 0;
        }
    }

    // Push down from splay root to x
    void pushDownAll(int x) {
        // collect ancestors in splay tree
        static int stk[MAXN];
        int top = 0;
        stk[++top] = x;
        for (int i = x; !isRoot(i); i = fa[i])
            stk[++top] = fa[i];
        while (top) pushDown(stk[top--]);
    }

    void rotate(int x) {
        int y = fa[x], z = fa[y];
        int k = (ch[y][1] == x);
        if (!isRoot(y)) ch[z][ch[z][1] == y] = x;
        fa[x] = z;
        ch[y][k] = ch[x][!k];
        if (ch[x][!k]) fa[ch[x][!k]] = y;
        ch[x][!k] = y;
        fa[y] = x;
        pushUp(y);
        pushUp(x);
    }

    void splay(int x) {
        pushDownAll(x);
        while (!isRoot(x)) {
            int y = fa[x], z = fa[y];
            if (!isRoot(y))
                rotate((ch[z][0] == y) == (ch[y][0] == x) ? y : x);
            rotate(x);
        }
        pushUp(x);
    }

    void access(int x) {
        int last = 0;
        for (int y = x; y; y = fa[y]) {
            splay(y);
            ch[y][1] = last;
            pushUp(y);
            last = y;
        }
        splay(x);
    }

    void makeRoot(int x) {
        access(x);
        rev[x] ^= 1;
    }

    int findRoot(int x) {
        access(x);
        pushDown(x);
        while (ch[x][0]) {
            x = ch[x][0];
            pushDown(x);
        }
        splay(x);
        return x;
    }

    void link(int x, int y) {
        makeRoot(x);
        if (findRoot(y) != x) // ensure they're in different trees
            fa[x] = y;
    }

    void cut(int x, int y) {
        makeRoot(x);
        access(y);
        if (ch[y][0] == x && !ch[x][1]) {
            ch[y][0] = 0;
            fa[x] = 0;
            pushUp(y);
        }
    }

    bool connected(int x, int y) {
        return findRoot(x) == findRoot(y);
    }

    // Path sum from x to y
    int pathSum(int x, int y) {
        makeRoot(x);
        access(y);
        return sum[y];
    }

    // Path max from x to y
    int pathMax(int x, int y) {
        makeRoot(x);
        access(y);
        return mx[y];
    }

    // Update node x's value to w
    void update(int x, int w) {
        access(x);
        self_val[x] = w;
        pushUp(x);
    }

    // Initialize node x with value w
    void init(int x, int w) {
        ch[x][0] = ch[x][1] = fa[x] = rev[x] = 0;
        self_val[x] = sum[x] = mx[x] = w;
        sz[x] = 1;
    }
} lct;

    Usage in main:

    int main() {
    int n, q;
    scanf("%d%d", &n, &q);
    for (int i = 1; i <= n; i++) {
        int w; scanf("%d", &w);
        lct.init(i, w);
    }
    while (q--) {
        int op, x, y;
        scanf("%d%d%d", &op, &x, &y);
        if (op == 0)      printf("%d\n", lct.pathSum(x, y));  // XOR in some problems
        else if (op == 1) lct.link(x, y);
        else if (op == 2) lct.cut(x, y);
        else              lct.update(x, y);  // set val[x] = y
    }
}
    Contest tip: Many problems use XOR instead of sum. Just change pushUp to sum[x] = sum[ch[x][0]] ^ sum[ch[x][1]] ^ self_val[x];. The structure stays identical.

    Complexity Analysis

    Every Link-Cut Tree operation boils down to a constant number of access + splay calls. The key result:

    OperationTime (Amortized)Notes
    access(v)O(log n)Core operation, dominates all others
    link(u, v)O(log n)makeRoot + access + O(1)
    cut(u, v)O(log n)makeRoot + access + O(1)
    pathQuery(u, v)O(log n)makeRoot + access + O(1)
    makeRoot(v)O(log n)access + O(1) reversal
    findRoot(v)O(log n)access + walk left + splay
    connected(u, v)O(log n)Two findRoot calls

    Why O(log n)?

    The proof relies on the access lemma from splay tree analysis. Define the size of a node x as the number of nodes in x's virtual subtree (the entire subtree in the represented tree, not just the splay tree). The potential function is:

    Φ = Σ log(size(v))

    Each access operation has amortized cost O(log n) because:

    • Each splay within the access costs O(log n) amortized (standard splay tree analysis).
    • Each preferred path change during access decreases the potential, paying for the cost of that step.
    • The number of preferred path changes over all accesses is bounded — each change is "charged" against the potential drop.

    The detailed proof is in the Sleator-Tarjan paper. For competitive programming, the key takeaway: m operations on n nodes cost O(m log n) total.

    Comparison with HLD: HLD gives O(log2 n) per path query on a static tree (log n for decomposition × log n for segment tree). Link-Cut Trees give O(log n) amortized and support dynamic edges. The trade-off: HLD has smaller constants and is simpler to implement for static trees.

    Applications

    1. Dynamic Connectivity

    Maintain a forest under link/cut operations and answer "are u and v connected?" queries. This is the most basic application. Use connected(u, v) which calls findRoot on both.

    2. Maximum Flow Augmentation

    In Dinic's algorithm with Link-Cut Trees, augmenting along a path takes O(log n) instead of O(path length). The bottleneck edge is the minimum on the path, which can be tracked with a path min aggregate. This gives an O(V E log V) max-flow algorithm that outperforms plain Dinic's on some graph types.

    3. Dynamic Minimum Spanning Tree

    To maintain an MST under edge insertions: when adding edge (u, v, w), if u and v are already connected, find the maximum weight edge on the u→v path. If w is smaller, cut the max edge and link (u, v, w). This requires storing edge weights as virtual nodes.

    // Dynamic MST: edge (u, v, w) represented as node n+edge_id
// link(u, node_for_edge), link(node_for_edge, v)
// pathMax gives the heaviest edge on the path
void addEdge(int u, int v, int w, int edgeNode) {
    lct.init(edgeNode, w);
    if (lct.connected(u, v)) {
        int maxEdge = lct.pathMax(u, v);
        if (maxEdge <= w) return; // new edge is not better
        // cut the max edge (need to track which node has max)
        // and link the new one
    }
    lct.link(u, edgeNode);
    lct.link(edgeNode, v);
}

    4. Online Bridge Finding

    Maintain a graph under edge additions and count bridges (edges whose removal disconnects the graph). When a new edge (u, v) is added: if u and v are already connected, the entire u→v path becomes part of a 2-edge-connected component — no edges on this path are bridges anymore.

    5. Subtree Queries (with Extensions)

    With additional bookkeeping (tracking the sum of values in path-parent children), Link-Cut Trees can also support subtree aggregates. This is trickier and less common in contests.

    Summary

    Link-Cut Trees are one of the most powerful data structures in competitive programming. Here's the mental model:

    1. Represented tree decomposed into preferred paths, each stored in a splay tree keyed by depth.
    2. access(v) makes root→v a single preferred path by walking up and switching preferred children.
    3. makeRoot(v) re-roots via access + reversal, enabling arbitrary path queries.
    4. link/cut add/remove edges by manipulating parent pointers after access.
    5. All operations O(log n) amortized thanks to splay tree potential analysis.

    The implementation is ~80 lines of C++. Once you have the template memorized, Link-Cut Trees become a versatile weapon for any problem involving dynamic forests with path queries.
