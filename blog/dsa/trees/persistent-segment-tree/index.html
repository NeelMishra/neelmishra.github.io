    Persistent Segment Trees

    July 2025 · 35 min read

    What if you could update a data structure without destroying the previous version? Persistent segment trees give you exactly that: every update creates a new version of the tree, while the old versions remain fully queryable. This lets you "time travel" through the history of your data.

    The idea sounds expensive, but thanks to a beautiful trick called path copying, each new version only costs O(log n) extra nodes instead of duplicating the entire tree. This post covers the theory, the implementation, and the classic competitive programming application: k-th smallest element in any sub-array.

    Prerequisites: You should be comfortable with segment trees (build, point update, range query). If you need a refresher, start with the Introduction to Segment Trees post.

    Why Persistent Data Structures?

    In the standard segment tree, when you perform an update like arr[3] = 10, the old values are overwritten. The tree only reflects the latest state. But many problems need access to historical versions:

    • K-th smallest in a range: Given an array, answer queries "what is the k-th smallest element in arr[l..r]?" This is the most famous application.
    • Undo operations: Roll back to any previous state in O(log n).
    • Offline queries with sorting: Process elements in sorted order and query any prefix of processed elements.
    • Functional updates: In problems where each version branches from a different past version (tree of versions).

    The Naive Idea (and Why It Fails)

    The brute-force approach: make a full copy of the segment tree before each update. If the tree has O(n) nodes and you perform n updates, that's O(n2) total space, completely impractical for n = 105.

    The Key Observation

    When you perform a point update in a segment tree, how many nodes actually change? Only the nodes on the path from the leaf to the root, exactly O(log n) nodes. Every other node in the tree remains identical. So why copy them?

    Core insight: Instead of copying the entire tree, only create new copies of the O(log n) nodes that change. The new version's unchanged subtrees simply point to (share) the old version's nodes. This is called path copying.

    The Path Copying Idea

    Let's make this concrete. Consider a segment tree built on arr = [1, 2, 3, 4] with sum queries. The tree has 7 nodes:

    Now suppose we update arr[1] = 5 (changing the value from 2 to 5). In a normal segment tree, we'd modify nodes on the path: leaf [1], then [0-1], then [0-3]. But in a persistent segment tree, we create new copies of these 3 nodes and leave the originals untouched:

    Shared / old nodes New nodes (this version) Dashed line = pointer to shared node

    Key observations:

    • Version 0 (root v0 with value 10) is still fully intact. You can query it as if nothing happened.
    • Version 1 (root v1 with value 13) has its own root, but shares the entire right subtree [2-3] and the left leaf [0] with version 0.
    • Only 3 new nodes were created (one per level = O(log n)), not 7.
    • Both versions coexist in memory simultaneously.

    Why Pointers, Not Array Indexing

    Normal segment trees use 2*i and 2*i+1 for children because nodes are stored in a flat array. But persistent segment trees can't do this, new nodes don't fit neatly into a [1..4n] array because multiple versions share subtrees. Instead, each node explicitly stores left and right child pointers (or indices into a node pool):

    struct Node {
    int left, right;  // indices of children in the node pool
    int val;           // aggregate value (e.g., sum, count)
};
Node pool[MAXNODES];  // pre-allocated node pool
int poolPtr = 0;      // next free slot

    Each version is identified by a root index: roots[0], roots[1], etc. To query version k, start from roots[k] and traverse using the node's left/right pointers, exactly like a normal segment tree query.

    Building the Initial Tree (Version 0)

    Building the initial persistent segment tree is almost identical to building a normal one, except we use the node pool instead of array indexing:

    int newNode(int v = 0, int l = 0, int r = 0) {
    pool[poolPtr] = {l, r, v};
    return poolPtr++;
}

// Build version 0 from array arr[lo..hi]
int build(int lo, int hi) {
    if (lo == hi) {
        return newNode(arr[lo]);  // leaf node
    }
    int mid = (lo + hi) / 2;
    int left  = build(lo, mid);
    int right = build(mid + 1, hi);
    return newNode(pool[left].val + pool[right].val, left, right);
}

    This creates the same tree structure as a normal segment tree, but stored in our node pool. The returned index becomes roots[0].

    Node count for build: Building version 0 creates exactly 2n - 1 nodes (same as a normal segment tree). The persistent overhead comes from updates, not the initial build.

    Point Update (Creating a New Version)

    This is where persistence shines. To create version i+1 from version i with update arr[pos] = val:

    1. Start from version i's root.
    2. Walk down the tree toward position pos.
    3. At each level, create a new node that copies the current node's value.
    4. The child on the path to pos points to the next new node. The other child points to the old node (shared).
    5. Recalculate the aggregate on the way back up.
    // Create a new version by updating position pos
// prev = root of the previous version
int update(int prev, int lo, int hi, int pos, int val) {
    if (lo == hi) {
        return newNode(val);  // new leaf with updated value
    }
    int mid = (lo + hi) / 2;
    int L = pool[prev].left;
    int R = pool[prev].right;
    if (pos <= mid) {
        L = update(L, lo, mid, pos, val);       // recurse left, share right
    } else {
        R = update(R, mid + 1, hi, pos, val);   // recurse right, share left
    }
    return newNode(pool[L].val + pool[R].val, L, R);
}

// Usage:
// roots[i+1] = update(roots[i], 0, n-1, pos, val);

    Each call to update creates exactly O(log n) new nodes (one per level) and returns the new root. The old root roots[i] is unmodified.

    Range Query on Any Version

    Querying version k is exactly the same as querying a normal segment tree. The only difference is you start from roots[k] and follow the node pool pointers:

    // Query sum of arr[l..r] on version k
int query(int node, int lo, int hi, int l, int r) {
    if (l > hi || r < lo) return 0;     // out of range
    if (l <= lo && hi <= r) {
        return pool[node].val;           // fully covered
    }
    int mid = (lo + hi) / 2;
    return query(pool[node].left,  lo, mid, l, r) +
           query(pool[node].right, mid + 1, hi, l, r);
}

// Usage: query version k for range [l, r]
// query(roots[k], 0, n-1, l, r);

    Since the tree structure is identical (only the root entry point differs), the query visits the same O(log n) nodes as a normal segment tree query. There is zero performance overhead for persistence on queries.

    The power of persistence: You now have O(n) versions of the segment tree, each queryable in O(log n) time, and the total memory is O(n log n) instead of O(n2). This is what makes persistent segment trees so useful.

    K-th Smallest in a Range

    This is the classic problem that made persistent segment trees famous. It appears as SPOJ MKTHNUM and in many contest problems.

    Problem Statement

    Given an array of n integers and q queries, each query asks: What is the k-th smallest element in the sub-array arr[l..r]?

    Example: arr = [5, 1, 3, 2, 4], query (l=1, r=3, k=2) → the sub-array is [1, 3, 2], sorted = [1, 2, 3], so the 2nd smallest is 2.

    The Idea: Frequency Segment Trees on Sorted Values

    The approach uses a segment tree over values (not indices), where each node stores the count of elements falling in its value range. Here's the plan:

    1. Coordinate compress the array values to the range [0, m-1] where m is the number of distinct values.
    2. Build a persistent segment tree of frequencies. Start with version 0 (all zeros), an empty frequency tree.
    3. Process elements left to right: version i is obtained by inserting arr[i-1] into version i-1. "Inserting" means incrementing the count at the compressed position of arr[i-1].
    4. Now roots[i] represents the frequency of all elements in arr[0..i-1].
    5. To query the sub-array arr[l..r]: the frequency tree for this range is roots[r+1] minus roots[l] (element-wise subtraction). We don't actually subtract, we walk both trees simultaneously.

    How the Query Works

    To find the k-th smallest in arr[l..r], walk down the persistent segment tree from the roots of version r+1 and version l simultaneously:

    // Walk down both trees to find k-th smallest
int kthSmallest(int vl, int vr, int lo, int hi, int k) {
    if (lo == hi) return lo;  // found the k-th value (compressed)
    int mid = (lo + hi) / 2;
    // Count of elements in left half for range [l..r]
    int leftCount = pool[pool[vr].left].val - pool[pool[vl].left].val;
    if (k <= leftCount) {
        return kthSmallest(pool[vl].left, pool[vr].left, lo, mid, k);
    } else {
        return kthSmallest(pool[vl].right, pool[vr].right, mid+1, hi, k - leftCount);
    }
}

    At each node, we compute leftCount = number of elements from arr[l..r] that fall in the left half of the value range. If k ≤ leftCount, the answer is in the left half. Otherwise, go right and search for the (k - leftCount)-th element.

    Step-by-Step Example

    Let's trace through arr = [5, 1, 3, 2, 4]. After coordinate compression, distinct sorted values are [1, 2, 3, 4, 5] mapping to indices [0, 1, 2, 3, 4].

    We build versions by inserting elements left to right:

    • Version 0: Empty tree (all counts = 0).
    • Version 1: Insert arr[0]=5 → increment position 4. Counts: [0,0,0,0,1].
    • Version 2: Insert arr[1]=1 → increment position 0. Counts: [1,0,0,0,1].
    • Version 3: Insert arr[2]=3 → increment position 2. Counts: [1,0,1,0,1].
    • Version 4: Insert arr[3]=2 → increment position 1. Counts: [1,1,1,0,1].
    • Version 5: Insert arr[4]=4 → increment position 3. Counts: [1,1,1,1,1].

    Query: k=2, l=1, r=3 (sub-array [1, 3, 2]). We use roots[4] (arr[0..3]) and roots[1] (arr[0..0]).

    • Root: total count = 4 − 1 = 3 elements. Value range [0-4].
    • Left child [0-2]: count = 3 − 0 = 3. Since k=2 ≤ 3, go left.
    • Left child [0-1]: count = 1 − 0 = 1. Since k=2 > 1, go right, k becomes 2−1=1.
    • Right child [1-1]: we reached a leaf. Compressed index 1 maps to value 2. ✓

    Memory Management

    Node Pool

    Since we can't use a flat array with 2*i / 2*i+1 indexing, persistent segment trees use a pre-allocated node pool. Each "new node" call simply takes the next available slot:

    const int MAXN = 200005;
const int MAXNODES = MAXN * 40;  // generous upper bound

struct Node {
    int left, right, val;
} pool[MAXNODES];
int poolPtr = 0;

int newNode(int v = 0, int l = 0, int r = 0) {
    pool[poolPtr] = {l, r, v};
    return poolPtr++;
}

    Estimating Pool Size

    How large should the node pool be?

    • Build: Creates 2n - 1 nodes (same as normal segment tree).
    • Each update: Creates O(log n) new nodes. Specifically, ⌈log2 n⌉ + 1 nodes per update.
    • Total for n updates after build: (2n - 1) + n · (⌈log2 n⌉ + 1).

    For n = 105: 2 × 105 + 105 × 18 ≈ 2 × 106 nodes. In practice, allocating n × 20 to n × 40 is safe. Each node is 12 bytes (three ints), so 2 × 106 nodes ≈ 24 MB, well within typical memory limits.

    Memory tip: If memory is tight, you can use short or pack the struct more tightly. But for most contest problems with n ≤ 2 × 105, a pool of n × 40 nodes with int fields works fine.

    No Garbage Collection Needed

    Unlike many persistent data structures, we never need to free old nodes. Every version might be queried, so all nodes must remain alive. The pool only grows, never shrinks. This simplicity is a major advantage in competitive programming.

    C++ Implementation

    Here's a complete, contest-ready implementation for the k-th smallest element in a range problem:

    #include <bits/stdc++.h>
using namespace std;

const int MAXN = 200005;
const int MAXNODES = MAXN * 20;

struct Node {
    int left, right, val;
} pool[MAXNODES];
int poolPtr = 0;
int roots[MAXN];

int newNode(int v = 0, int l = 0, int r = 0) {
    pool[poolPtr] = {l, r, v};
    return poolPtr++;
}

// Build an empty tree over value range [lo, hi]
int build(int lo, int hi) {
    if (lo == hi) return newNode(0);
    int mid = (lo + hi) / 2;
    int L = build(lo, mid);
    int R = build(mid + 1, hi);
    return newNode(pool[L].val + pool[R].val, L, R);
}

// Insert: increment count at position pos
// Returns new root for this version
int update(int prev, int lo, int hi, int pos) {
    if (lo == hi) {
        return newNode(pool[prev].val + 1);
    }
    int mid = (lo + hi) / 2;
    int L = pool[prev].left;
    int R = pool[prev].right;
    if (pos <= mid) {
        L = update(L, lo, mid, pos);
    } else {
        R = update(R, mid + 1, hi, pos);
    }
    return newNode(pool[L].val + pool[R].val, L, R);
}

// Query: find the k-th smallest in the range
// vl = roots[l], vr = roots[r+1]
int query(int vl, int vr, int lo, int hi, int k) {
    if (lo == hi) return lo;
    int mid = (lo + hi) / 2;
    int leftCount = pool[pool[vr].left].val
                  - pool[pool[vl].left].val;
    if (k <= leftCount) {
        return query(pool[vl].left, pool[vr].left,
                     lo, mid, k);
    } else {
        return query(pool[vl].right, pool[vr].right,
                     mid + 1, hi, k - leftCount);
    }
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;

    vector<int> arr(n);
    for (int i = 0; i < n; i++) cin >> arr[i];

    // Coordinate compression
    vector<int> sorted_vals = arr;
    sort(sorted_vals.begin(), sorted_vals.end());
    sorted_vals.erase(
        unique(sorted_vals.begin(), sorted_vals.end()),
        sorted_vals.end()
    );
    int m = sorted_vals.size();

    auto compress = [&](int x) {
        return lower_bound(sorted_vals.begin(),
                           sorted_vals.end(), x)
               - sorted_vals.begin();
    };

    // Build persistent segment tree
    roots[0] = build(0, m - 1);
    for (int i = 0; i < n; i++) {
        roots[i + 1] = update(roots[i], 0, m - 1,
                              compress(arr[i]));
    }

    // Answer queries
    while (q--) {
        int l, r, k;
        cin >> l >> r >> k;
        l--; r--;  // convert to 0-indexed
        int idx = query(roots[l], roots[r + 1],
                        0, m - 1, k);
        cout << sorted_vals[idx] << "\n";
    }
    return 0;
}
    Contest note: The input format varies by problem. SPOJ MKTHNUM uses 1-indexed l, r. Codeforces problems may differ. Always read the input specification carefully.

    Complexity Analysis

    OperationTimeSpace (new nodes)
    Build (version 0)O(n)O(n) nodes
    Point update (new version)O(log n)O(log n) nodes
    Range query (any version)O(log n)O(1) (no allocation)
    K-th smallest queryO(log n)O(1) (no allocation)

    Total Complexity for K-th Smallest Problem

    • Time: O(n log n) to build all versions + O(q log n) for queries = O((n + q) log n).
    • Space: O(n) for version 0 + O(n log n) for n updates = O(n log n) total nodes.

    For n = q = 105, that's about 2 × 106 nodes and 2 × 106 operations, easily within time and memory limits.

    Why Not Just Sort Each Query?

    ApproachTimeSpaceOnline?
    Sort each query rangeO(q · n log n)O(n)Yes
    Merge sort treeO(n log2 n + q log3 n)O(n log n)Yes
    Mo's algorithm + BITO((n + q) √n log n)O(n)No (offline)
    Persistent segment treeO((n + q) log n)O(n log n)Yes

    Persistent segment trees give the best time complexity and support online queries (each query can be answered as it arrives, without seeing future queries).

    Practice Problems

    Medium

    SPOJ MKTHNUM — K-th Number

    The classic problem: find the k-th smallest number in a sub-array. Direct application of the persistent segment tree approach described above.

    Medium

    CSES — Distinct Values Queries

    Count the number of distinct values in a range. Build a persistent structure tracking last occurrence of each value. Can also be solved with Mo's algorithm, but persistent seg tree gives O(n log n).

    Medium

    CF 813D — Team Training

    Requires answering range frequency queries. Build a persistent segment tree over values and use the subtraction trick to count elements in a value range within a sub-array.

    Hard

    CF 893F — Subtree Minimum Query

    Persistent segment tree on an Euler tour. Each version corresponds to a depth level. Requires combining DFS ordering with persistence for subtree queries at bounded depth.

    Hard

    CF 840D — Destiny

    Find the element that occurs more than n/k times in a range. Use a persistent segment tree on values. At each level, if the count in the left child exceeds the threshold, recurse left; similarly for right.

    Hard

    SPOJ COT — Count on a Tree

    K-th smallest on a tree path. Build persistent segment trees along the tree using DFS: each node's version is derived from its parent's version. Query uses LCA: root[u] + root[v] - root[lca] - root[parent(lca)].

    Medium

    CF 484E — Sign on the Trail

    Find the maximum XOR with a given value over a range. Persistent trie variant, which is conceptually similar: each version adds one element, and queries use two versions to isolate a range.

    Summary

    • Persistent segment trees preserve all historical versions of the tree using path copying: each update creates O(log n) new nodes instead of duplicating the whole tree.
    • Each version has its own root pointer. The unchanged subtrees are shared across versions.
    • Uses a node pool with explicit left/right pointers instead of 2*i / 2*i+1 array indexing.
    • Build: O(n) time and space. Update: O(log n) time and space per version. Query: O(log n) time.
    • Total memory for n versions: O(n log n).
    • The killer application is k-th smallest in a range: build persistent frequency trees, then use the subtraction trick roots[r+1] - roots[l] to get frequencies for any sub-array.
    • Persistent segment trees are online (answer queries as they arrive) and give O((n+q) log n) total time for the k-th smallest problem.
