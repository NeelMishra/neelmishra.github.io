    Level-Order (BFS) Serialization

    March 2026 · 10 min read

    If you've ever pasted a tree into LeetCode's input box, you've used level-order serialization. The format [1,2,3,null,null,null,5] is so widespread that most developers think of it as "the" tree format. It writes nodes top-to-bottom, left-to-right, one level at a time.

    The Idea

    Use a queue (BFS) to visit nodes level by level. For each node:

    • If it's non-null, write its value and enqueue its children (even if they're null).
    • If it's null, write null and don't enqueue anything.

    The result is the tree "flattened by level": first the root, then all depth-1 nodes left-to-right, then all depth-2 nodes, and so on.

    Why Level-Order Is Human-Friendly

    Consider the preorder encoding of the same tree: "1 2 4 # # # 3 # 5 # #". To mentally reconstruct the tree, you need to simulate the recursive descent. It's like reading assembly language.

    Level-order is different. You can literally draw the tree from the array:

    Position 0: root = 1
    Position 1: left child of 1 = 2
    Position 2: right child of 1 = 3
    Position 3: left child of 2 = 4
    Position 4: right child of 2 = null
    Position 5: left child of 3 = null
    Position 6: right child of 3 = 5

    The parent-child relationship follows the familiar array-heap formula: for a node at index i, its children are at 2i + 1 and 2i + 2... but only if there are no null gaps. Once nulls appear, the indexing shifts. That's why the queue-based approach is necessary rather than simple index math.

    Serialization Code

    std::string serialize(TreeNode* root) {
    if (!root) return "[]";
    std::vector<std::string> result;
    std::queue<TreeNode*> q;
    q.push(root);

    while (!q.empty()) {
        TreeNode* node = q.front(); q.pop();
        if (node) {
            result.push_back(std::to_string(node->val));
            q.push(node->left);
            q.push(node->right);
        } else {
            result.push_back("null");
        }
    }

    // Trim trailing "null" entries
    while (!result.empty() && result.back() == "null")
        result.pop_back();

    // Build output string
    std::string out = "[";
    for (int i = 0; i < result.size(); i++) {
        if (i > 0) out += ",";
        out += result[i];
    }
    return out + "]";
}

    Why Trim Trailing Nulls?

    Without trimming, a tree with a single left child would serialize as [1,2,null,null,null]. The last three nulls are the children of node 2 (two nulls) and the right child position of a non-existent right subtree. They carry no information. Trimming gives [1,2], which is more compact and equally unambiguous, because any missing positions are implicitly null.

    Deserialization Code

    Deserialization uses a queue to assign children to nodes as we read tokens left to right:

    TreeNode* deserialize(const std::string& data) {
    if (data == "[]" || data.empty()) return nullptr;

    // Parse tokens (strip brackets, split by comma)
    std::vector<std::string> tokens = parse(data);

    TreeNode* root = new TreeNode(std::stoi(tokens[0]));
    std::queue<TreeNode*> q;
    q.push(root);
    int i = 1;

    while (!q.empty() && i < tokens.size()) {
        TreeNode* parent = q.front(); q.pop();

        // Left child
        if (i < tokens.size() && tokens[i] != "null") {
            parent->left = new TreeNode(std::stoi(tokens[i]));
            q.push(parent->left);
        }
        i++;

        // Right child
        if (i < tokens.size() && tokens[i] != "null") {
            parent->right = new TreeNode(std::stoi(tokens[i]));
            q.push(parent->right);
        }
        i++;
    }
    return root;
}

    The key insight: the queue holds nodes that still need children assigned. For each such node, we read exactly two tokens (left child, right child). Non-null children get enqueued for their own child assignment later.

    The Sparse Tree Problem

    Level-order has a weakness: sparse trees waste a lot of space. Consider a right-skewed tree with 5 nodes (1→2→3→4→5):

    Level-order: [1, null, 2, null, 3, null, 4, null, 5]
Preorder:    "1 # 2 # 3 # 4 # 5 # #"

    For a balanced tree with n nodes, both approaches use O(n) space. But for a skewed tree of height h, level-order can use O(2h) space in the worst case (if you don't trim), because you're encoding the "shape" of a complete tree up to level h. Trimming helps but doesn't fully solve it.

    Preorder vs Level-Order

    PropertyPreorder + NullsLevel-Order (BFS)
    Token countAlways 2n + 1Variable; up to 2h+1 - 1 before trim
    Human readabilityHard to readEasy to draw
    StreamabilityYes (start building immediately)No (need full level before placing children)
    Sparse tree overheadMinimalSignificant without trimming
    ImplementationRecursive (simple)Queue-based (slightly more code)
    Interview standardDFS-focused interviewsLeetCode format

    Special Case: Complete Binary Trees

    For complete binary trees (every level full except possibly the last, filled left-to-right), level-order serialization has a beautiful property: no null markers needed at all. The shape is entirely determined by the node count.

    // For a complete binary tree with n nodes:
// Parent of node i: (i - 1) / 2
// Left child: 2*i + 1
// Right child: 2*i + 2
// This is exactly how a binary heap is stored in an array!

    This is why binary heaps use arrays instead of pointer-based trees: the array is the level-order serialization, and it's maximally compact.

    Extension: N-ary Tree BFS Serialization

    For N-ary trees, level-order needs a way to delimit children groups. One approach: use a null token as a "end of children" marker.

    // N-ary tree: root=1, children=[2,3,4], 2 has children [5,6]
// BFS with child-group delimiters:
// [1, null, 2, 3, 4, null, 5, 6, null, null, null]
//    ^sep   ^children of 1  ^sep  ^children of 2...

    Alternatively, encode the child count before each node's children: [1,3,2,3,4,2,5,6,0,0,0] where the number after each value tells how many children follow.

    Complexity Analysis

    Time: O(n) for both serialize and deserialize. Each node processed exactly once.

    Space (output): O(n) for balanced trees, up to O(2h) for degenerate trees before trimming.

    Space (auxiliary): O(w) for the queue, where w is the maximum width. For a complete tree, w = n/2.
