    Serialization & Deserialization of Trees

    March 2026 · 8 min read

    A tree lives in memory as a web of pointers: each node points to its children, maybe its parent. You can traverse it, query it, mutate it. But you cannot save it to a file, send it over a network, or log it to a database. Not directly. Pointers are addresses in one process's virtual memory, meaningless to anything else.

    Serialization bridges the gap. It converts the tree into a flat, portable sequence of bytes (or characters) that can be stored, transmitted, and later reconstructed into the exact same tree. Deserialization is the reverse: given the flat data, rebuild the original tree structure, every node, every edge, every null child, exactly as before.

    Why This Matters

    Serialization is everywhere in real software. Every time you:

    • Parse JSON or XML, you're deserializing a tree structure. {"name": "Alice", "children": [...]} is a serialized N-ary tree.
    • Store an AST (Abstract Syntax Tree) from a compiler, you serialize the parse tree so later passes can read it without re-parsing the source.
    • Persist an index in a database, the B-tree or B+ tree gets serialized into disk pages. Reading a page back means deserializing nodes.
    • Send data in distributed systems, protocol buffers, Thrift, and FlatBuffers all handle tree-shaped data by serializing it into flat bytes.
    • Cache computed results, if a function returns a tree, you serialize and store it so the next call can skip the computation.

    And in interviews, it appears directly as "Serialize and Deserialize a Binary Tree" (LeetCode 297), one of the most frequently asked hard problems.

    There are two fundamental strategies to solve this:

    Strategy 1: One traversal + null markers. Record the traversal and explicitly mark where null children appear. The null markers encode the shape. This is the most common approach.

    Strategy 2: Two traversals, no markers. Use two different traversals (preorder + inorder, or postorder + inorder). Their combination uniquely determines the tree. More compact, but with constraints on values.

    What Makes a Good Serialization?

    Not all encodings are equal. When choosing a serialization scheme, think about:

    PropertyWhy It Matters
    CorrectnessThe deserialized tree must be structurally identical to the original. No exceptions.
    CompactnessHow many bytes per node? Trailing nulls waste space for sparse trees.
    SimplicityCan you write the serialize/deserialize pair without bugs under interview pressure?
    StreamabilityCan you begin deserializing before the full string is received? DFS approaches allow this; BFS does not.
    Human readabilityLevel-order is easy to visualize (LeetCode uses it). Preorder is harder to read.
    GeneralityDoes it handle duplicates? Negative values? N-ary trees? BSTs?

    A Mental Model

    Think of a tree as a sentence, and serialization as writing it down. The question is: what writing system do you use?

    Preorder + null markers is like writing a recipe step by step: "Start at the root. Go left. Go left again. Dead end. Back up. Dead end. Back up..." It records the exact walk through the tree. Every null marker is a terminal, a place where you stop and backtrack.

    Level-order (BFS) is like taking a group photo at each floor. "First row: the root. Second row: its children. Third row: its grandchildren..." It's the format humans find easiest to visualize, which is why LeetCode uses [1,2,3,null,null,null,5].

    Two-traversal encoding is like a lock and key. Preorder alone or inorder alone is ambiguous, but together they uniquely identify the tree, each traversal constrains what the other can mean.

    The Key Invariant

    Every correct serialization scheme guarantees one thing:

    Invariant: deserialize(serialize(tree)) === tree for all valid trees. The round-trip must produce an identical tree. If two different trees can produce the same serialized string, the scheme is broken.

    This invariant is what separates serialization from mere traversal. A traversal outputs values; a serialization outputs an encoding that is injective (one-to-one) over the space of all trees.

    Complexity at a Glance

    ApproachSerializeDeserializeSpace
    Preorder + null markersO(n)O(n)O(n) string + O(h) stack
    Level-order (BFS)O(n)O(n)O(n) string + O(w) queue
    Preorder + InorderO(n)O(n)O(n) per array + O(n) map
    BST-optimized (bounds)O(n)O(n)O(n) string + O(h) stack

    Where n = number of nodes, h = height, w = maximum width of any level.
