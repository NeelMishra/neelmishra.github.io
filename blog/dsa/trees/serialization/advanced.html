    Advanced Serialization Techniques

    March 2026 · 12 min read

    The preorder and BFS approaches handle general binary trees. But in practice, you often know more about your tree: it might be a BST, it might be N-ary, or you might need extreme compactness. Each constraint opens the door to a more specialized encoding.

    Two-Traversal Encoding (No Null Markers)

    A classic result in tree theory: preorder + inorder uniquely determines a binary tree. No null markers needed. The idea: preorder tells you the root, inorder tells you what's in the left vs right subtree.

    How It Works

    Given preorder [1, 2, 4, 3, 5] and inorder [4, 2, 1, 3, 5]:

    // Step 1: Preorder[0] = 1 is the root
    // Step 2: In inorder, 1 is at index 2
    // Left subtree inorder: [4, 2] (indices 0..1)
    // Right subtree inorder: [3, 5] (indices 3..4)
    // Step 3: Left subtree has 2 nodes
    // Left subtree preorder: [2, 4] (next 2 in preorder)
    // Right subtree preorder: [3, 5] (remaining)
    // Step 4: Recurse on each subtree

    The Code

    TreeNode* build(std::vector<int>& preorder, int preL, int preR,
                std::vector<int>& inorder, int inL, int inR,
                std::unordered_map<int,int>& inMap) {
    if (preL > preR) return nullptr;

    int rootVal = preorder[preL];
    int rootIdx = inMap[rootVal];     // O(1) lookup in inorder
    int leftSize = rootIdx - inL;

    TreeNode* root = new TreeNode(rootVal);
    root->left  = build(preorder, preL+1, preL+leftSize,
                        inorder, inL, rootIdx-1, inMap);
    root->right = build(preorder, preL+leftSize+1, preR,
                        inorder, rootIdx+1, inR, inMap);
    return root;
}

    Limitations

    • No duplicate values allowed. The inorder index lookup must be unique. If two nodes have the same value, we can't determine which one is the root's left vs right child.
    • Two arrays required. We transmit roughly 2n integers instead of 2n+1 tokens, so the space savings are marginal.
    • O(n) hash map for the inorder index lookup. Without it, the algorithm degrades to O(n²).

    Other valid pairs: Postorder + Inorder also works (root is last in postorder). Preorder + Postorder works only for full binary trees (every node has 0 or 2 children).

    BST-Optimized Serialization

    If the tree is a Binary Search Tree, we can serialize with preorder alone, no null markers. The BST property tells us how to split the tokens.

    Why BSTs Don't Need Null Markers

    In a BST, all values in the left subtree are less than the root, and all values in the right subtree are greater. So given the preorder sequence [5, 3, 1, 4, 8, 7, 9], we know:

    • Root = 5
    • Left subtree values = everything < 5 that follows: [3, 1, 4]
    • Right subtree values = everything ≥ 5 that follows: [8, 7, 9]

    No ambiguity. The BST property acts as an implicit delimiter.

    Deserialization with Bounds

    TreeNode* deserializeBST(std::vector<int>& preorder, int& i,
                         int minVal, int maxVal) {
    if (i >= preorder.size()) return nullptr;
    int val = preorder[i];
    if (val < minVal || val > maxVal) return nullptr;

    TreeNode* node = new TreeNode(val);
    i++;
    node->left  = deserializeBST(preorder, i, minVal, val);
    node->right = deserializeBST(preorder, i, val, maxVal);
    return node;
}

// Usage: int i = 0; root = deserializeBST(pre, i, INT_MIN, INT_MAX);

    Each recursive call narrows the valid range. If the next value falls outside the range, it belongs to an ancestor's subtree, so we return null. This is equivalent to what null markers do, but using the BST invariant instead of explicit sentinels.

    Space savings: $n$ tokens instead of $2n + 1$. For large trees, this is significant.

    N-ary Tree Serialization

    All our approaches so far assumed binary trees (each node has at most 2 children). N-ary trees, where a node can have any number of children, require a different strategy.

    Approach 1: Child-Count Prefix

    Before each node's children, write how many children it has:

    // Tree: 1 has children [2, 3, 4], 3 has children [5, 6]
void serialize(NaryNode* root, std::ostringstream& out) {
    if (!root) return;
    out << root->val << " " << root->children.size() << " ";
    for (auto* child : root->children)
        serialize(child, out);
}
// Output: "1 3 2 0 3 2 5 0 6 0 4 0"
//          ^ ^   ^   ^ ^       ^
//          1 has 3 children; 2 has 0; 3 has 2; etc.

    Approach 2: Sentinel End-of-Children

    Use a special marker (like ]) to signal the end of a node's children list:

    void serialize(NaryNode* root, std::ostringstream& out) {
    if (!root) { out << "# "; return; }
    out << root->val << " ";
    for (auto* child : root->children)
        serialize(child, out);
    out << "] ";   // end of children marker
}
// Output: "1 2 ] 3 5 ] 6 ] ] 4 ] ]"

    Approach 3: Left-Child Right-Sibling (Binary Encoding)

    Transform the N-ary tree into a binary tree using the left-child right-sibling representation, then serialize the binary tree with any standard approach.

    Compact Binary Encoding

    For production systems (databases, network protocols), string-based serialization is wasteful. Consider a tree with 32-bit integer values:

    • String format: Each value takes up to 11 characters (e.g., "-2147483648") plus a delimiter. Total: ~12 bytes per node.
    • Binary format: 4 bytes per value + 1 bit per null marker. Total: ~4.125 bytes per node. A 3x improvement.

    Structure Bitmap Encoding

    Write the tree structure as a bitmap (1 = node exists, 0 = null) using BFS order, followed by the values array:

    // Tree: 1(2(4,null),3(null,5))
// Structure bitmap (BFS): 1 1 1 1 0 0 1 0 0 0 0
// Values (BFS order):     [1, 2, 3, 4, 5]
// Total: 11 bits + 5 * 4 bytes = ~22 bytes
// vs string "[1,2,3,4,null,null,5]" = 21 characters

// For large trees with many nodes, binary is far more compact.

    Succinct Tree Encoding

    Information theory tells us that a binary tree with $n$ nodes requires at least $\log_2 C_n \approx 2n - O(\log n)$ bits to encode its structure, where $C_n$ is the $n$-th Catalan number. The balanced parentheses encoding achieves this bound:

    // DFS: write '(' when entering a node, ')' when leaving
// Tree: 1(2(4,,),3(,5(,)))
// Parentheses: ( ( ( ) ) ) ( ( ) )
// As bits:     1 1 1 0 0 0 1 1 0 0   = 2n bits for n=5 nodes

    This is called a succinct encoding: it uses space within a constant factor of the information-theoretic minimum. Succinct trees support navigation queries (parent, left-child, right-child) in O(1) time using rank/select operations on the bit vector.

    Real-World Serialization

    JSON and XML

    JSON objects are N-ary trees serialized with the sentinel approach: { starts a node, } ends it. Arrays use [ and ]. Parsing JSON is literally tree deserialization.

    // JSON is an N-ary tree:
{
  "name": "Alice",          // leaf
  "scores": [95, 87, 92],   // subtree (array)
  "address": {              // subtree (object)
    "city": "NYC",
    "zip": "10001"
  }
}

    Database B-Trees

    Databases serialize B-tree nodes into fixed-size disk pages (typically 4KB or 16KB). Each page stores: key count, keys array, child page pointers, and values/row pointers. Reading a page from disk = deserializing a B-tree node.

    Protocol Buffers

    Google's protobuf uses a tag-length-value encoding. Nested messages (tree children) are length-prefixed, so the deserializer knows exactly how many bytes to consume for each subtree, analogous to our null-marker approach.

    Abstract Syntax Trees

    Compilers serialize ASTs to disk for incremental compilation. Clang's serialized AST format uses a data stream where each node is encoded with its kind, properties, and child offsets. This allows random access to subtrees without deserializing the entire tree.
