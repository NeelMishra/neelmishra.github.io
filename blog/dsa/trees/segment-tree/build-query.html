    Building & Querying Segment Trees

    March 2026 · 22 min read

    In the introduction, we learned what a segment tree is and why it exists. Now we'll see exactly how it's built and how range queries work, with full code, step-by-step traces, and interactive animations.

    Building the Segment Tree

    The Idea

    Building a segment tree is a classic divide and conquer algorithm:

    1. Base case: If the range has one element (start == end), it's a leaf. Store the array value directly.
    2. Recursive case: Split the range at mid = (start + end) / 2. Recursively build the left child for [start, mid] and the right child for [mid+1, end]. Then set this node's value to left + right (merge).

    The recursion naturally produces a bottom-up construction: leaves are filled first, then their parents, all the way up to the root.

    The Pseudocode

    function build(arr, tree, node, start, end):
    if start == end: // leaf: single element
    tree[node] = arr[start]
    return
    mid = (start + end) / 2
    build(arr, tree, 2*node, start, mid) // left child
    build(arr, tree, 2*node+1, mid+1, end) // right child
    tree[node] = tree[2*node] + tree[2*node+1] // merge

    Why 2*node and 2*node+1?

    This is the same indexing scheme as a binary heap. If the current node is at index node in the array:

    • Left child: 2 * node
    • Right child: 2 * node + 1

    Starting from node 1 (root), the entire tree maps cleanly into a flat array. No pointers needed.

    Build Complexity

    Time: O(n), Each of the 2n-1 nodes is visited exactly once. The merge at each internal node is O(1). Total: O(n).

    Space: O(4n) for the tree array. O(log n) for the recursion stack.

    C++ Implementation: Build

    void build(const vector<int>& arr, vector<int>& tree,
           int node, int start, int end) {
    if (start == end) {
        // Leaf node: store the array element
        tree[node] = arr[start];
        return;
    }
    int mid = (start + end) / 2;

    // Recursively build children
    build(arr, tree, 2 * node,     start, mid);
    build(arr, tree, 2 * node + 1, mid + 1, end);

    // Internal node: merge children
    tree[node] = tree[2 * node] + tree[2 * node + 1];
}

// Usage:
// vector<int> arr = {1, 3, 5, 7};
// vector<int> tree(4 * arr.size(), 0);
// build(arr, tree, 1, 0, arr.size() - 1);

    Iterative Build (Bottom-Up Alternative)

    The recursive version is intuitive but has function call overhead and uses O(log n) stack space. Here's an iterative bottom-up approach that builds the same tree with a different array layout:

    Key layout difference: The iterative approach uses a tree of size 2n (not 4n). Leaves sit at indices [n, 2n−1] and internal nodes at [1, n−1]. The root is still at index 1. This requires n to be a power of 2 (pad with identity values if not). 
    void build_iterative(const vector<int>& arr, vector<int>& tree, int n) {
    // Step 1: Place leaves at positions [n, 2n-1]
    for (int i = 0; i < n; i++)
        tree[n + i] = arr[i];

    // Step 2: Build internal nodes bottom-up
    for (int i = n - 1; i >= 1; i--)
        tree[i] = tree[2 * i] + tree[2 * i + 1];
}

// Usage:
// int n = arr.size();  // pad to next power of 2 if needed
// vector<int> tree(2 * n, 0);
// build_iterative(arr, tree, n);

    Why this works: By placing leaves at [n, 2n−1], the parent–child relationship tree[i] = tree[2i] + tree[2i+1] holds automatically. Iterating from n−1 down to 1 guarantees both children are filled before we compute any parent.

    Range Queries

    Now the powerful part. Given the built tree, we want to compute sum(l, r) in O(log n).

    The Three Cases

    At each recursive call, we're at a node that covers range [start, end] and we want the sum of [l, r]. There are exactly three possibilities:

    Case 1: Total Overlap (the range is fully inside the query)

    If l ≤ start AND end ≤ r, then the node's entire range is inside our query range. Return the node's value directly. No need to recurse further, this is the "shortcut" that makes segment trees fast.

    Case 2: No Overlap (the range is completely outside the query)

    If start > r OR end < l, there's zero overlap. Return the identity element (0 for sum, ∞ for min, −∞ for max). This entire subtree is irrelevant.

    Case 3: Partial Overlap

    The node's range straddles the query boundary. We can't use this node's value directly because it covers elements we don't want. Recurse into both children and merge the results.

    Why only O(log n) nodes? At each level of the tree, at most two nodes can have partial overlap (one touching the left boundary of [l,r], one touching the right boundary). All other nodes at that level are either fully inside (Case 1, we take them) or fully outside (Case 2, we skip). Since the tree has O(log n) levels, we visit at most O(2 · log n) = O(log n) partial-overlap nodes + some fully-inside nodes.

    The Pseudocode

    function query(tree, node, start, end, l, r):
    if r < start or end < l: // Case 2: no overlap
    return 0
    if l ≤ start and end ≤ r: // Case 1: total overlap
    return tree[node]
    // Case 3: partial overlap — recurse both
    mid = (start + end) / 2
    leftSum = query(tree, 2*node, start, mid, l, r)
    rightSum = query(tree, 2*node+1, mid+1, end, l, r)
    return leftSum + rightSum

    Walkthrough: query(1, 3) on [1, 3, 5, 7]

    We want the sum of arr[1] + arr[2] + arr[3] = 3 + 5 + 7 = 15.

    C++ Implementation: Query

    int query(const vector<int>& tree, int node,
          int start, int end, int l, int r) {
    // Case 2: No overlap
    if (r < start || end < l)
        return 0;

    // Case 1: Total overlap
    if (l <= start && end <= r)
        return tree[node];

    // Case 3: Partial overlap
    int mid = (start + end) / 2;
    int leftSum  = query(tree, 2 * node,     start, mid,     l, r);
    int rightSum = query(tree, 2 * node + 1, mid + 1, end,   l, r);
    return leftSum + rightSum;
}

// Usage:
// int result = query(tree, 1, 0, n - 1, l, r);

    Iterative Query (Two-Pointer Alternative)

    The iterative query uses a two-pointer bottom-up technique. Instead of recursing top-down and checking three cases, we start at the leaves and walk upward, collecting boundary values:

    int query_iterative(const vector<int>& tree, int n, int l, int r) {
    int res = 0;
    // Move l and r to leaf positions, make r exclusive
    for (l += n, r += n + 1; l < r; l >>= 1, r >>= 1) {
        if (l & 1)        // l is a right child → include it, move right
            res += tree[l++];
        if (r & 1)        // r-1 is inside range → include it
            res += tree[--r];
    }
    return res;
}

// Usage (0-indexed, inclusive range):
// int result = query_iterative(tree, n, l, r);

    How the Two-Pointer Query Works

    We maintain two pointers l (left boundary) and r (right boundary, exclusive), starting at the leaf level and walking them upward:

    1. If l is odd (a right child): its parent covers elements outside our range on the left. So we include tree[l] individually and advance l one step right to “detach” from that parent.
    2. If r is odd (a right child): the node at r−1 is the last node still inside our range. We decrement r and include tree[r].
    3. Move to parents: l >>= 1; r >>= 1.
    4. Repeat until l ≥ r (pointers have met or crossed).
    Why “is odd”? In the iterative tree layout, a node at index i is a right child if i is odd (since left child = 2×parent = always even, right child = 2×parent+1 = always odd). Being a right child means its parent’s range extends to the left beyond our query — so we must include this node alone, not its parent.

    Query Complexity: Why O(log n)?

    Let's prove this more carefully. The claim is that a range query visits at most O(log n) nodes.

    The Counting Argument

    Consider any level of the tree. There are at most 2level nodes at that level. For a query [l, r]:

    • At most 2 nodes at each level have partial overlap (one touching the left boundary l, one touching the right boundary r). These are the only nodes where we recurse into both children.
    • All other nodes visited at that level are total-overlap (we stop) or no-overlap (we stop).

    Since the tree has ⌈log2 n⌉ + 1 levels, we recurse into both children at most O(log n) times. Each such split adds one node to the "frontier." The total nodes visited is bounded by 4 · log n (more precisely, at most 4 · ⌈log2 n⌉ - 5 for n ≥ 3).

    Visual intuition: Think of the query range [l, r] as a "band" sweeping across the tree from left to right. At the top levels, this band might span multiple nodes. But as we go deeper, the band gets split and eventually matches node boundaries exactly. The O(log n) bound comes from the fact that the band can "straddle" at most one boundary per level.

    Complete Implementation

    class SegmentTree {
    int n;
    vector<int> tree;

    void build(const vector<int>& arr, int node, int s, int e) {
        if (s == e) {
            tree[node] = arr[s];
            return;
        }
        int mid = (s + e) / 2;
        build(arr, 2 * node,     s, mid);
        build(arr, 2 * node + 1, mid + 1, e);
        tree[node] = tree[2 * node] + tree[2 * node + 1];
    }

    int query(int node, int s, int e, int l, int r) {
        if (r < s || e < l) return 0;
        if (l <= s && e <= r) return tree[node];
        int mid = (s + e) / 2;
        return query(2 * node, s, mid, l, r)
             + query(2 * node + 1, mid + 1, e, l, r);
    }

public:
    SegmentTree(const vector<int>& arr)
        : n(arr.size()), tree(4 * n, 0) {
        build(arr, 1, 0, n - 1);
    }

    int query(int l, int r) {
        return query(1, 0, n - 1, l, r);
    }
};

    Complete Iterative Implementation

    Here's the full iterative segment tree, using the bottom-up layout with leaves at [n, 2n−1]. This version also includes point updates (covered in the next post) for completeness:

    class IterativeSegTree {
    int n;
    vector<int> tree;

public:
    IterativeSegTree(const vector<int>& arr) {
        n = arr.size();
        tree.assign(2 * n, 0);
        // Copy leaves
        for (int i = 0; i < n; i++)
            tree[n + i] = arr[i];
        // Build internal nodes
        for (int i = n - 1; i >= 1; i--)
            tree[i] = tree[2 * i] + tree[2 * i + 1];
    }

    // Point update: set arr[pos] += val (or = val, adjust as needed)
    void update(int pos, int val) {
        pos += n;              // go to leaf
        tree[pos] += val;
        // Walk up to root, updating parents
        for (pos >>= 1; pos >= 1; pos >>= 1)
            tree[pos] = tree[2 * pos] + tree[2 * pos + 1];
    }

    // Range query: sum of [l, r] (0-indexed, inclusive)
    int query(int l, int r) {
        int res = 0;
        for (l += n, r += n + 1; l < r; l >>= 1, r >>= 1) {
            if (l & 1) res += tree[l++];
            if (r & 1) res += tree[--r];
        }
        return res;
    }
};
    Which should you use?
    • Learning & interviews: Use the recursive version. It maps directly to the divide-and-conquer intuition and is easier to extend to lazy propagation.
    • Competitive programming: Use the iterative version when you need speed and don't need lazy propagation. It's 2–3× faster due to zero recursion overhead.
    • Lazy propagation: The recursive version is strongly preferred. Iterative lazy propagation exists but is significantly more complex (requires explicit top-down push before queries).

    Common Mistakes

    • Off-by-one in mid: The split must be mid = (start + end) / 2, with left covering [start, mid] and right covering [mid+1, end]. Getting this wrong causes missing or double-counted elements.
    • Wrong identity value: For sum, return 0 on no-overlap. For min, return INT_MAX. For max, return INT_MIN. Using the wrong identity silently produces wrong answers.
    • Insufficient array size: Always allocate 4 * n. Allocating 2 * n causes out-of-bounds for non-power-of-2 sizes.
    • Mixing 0-indexed and 1-indexed: The tree is 1-indexed (root at 1), but the array can be 0-indexed. Be consistent.

    Summary

    • Build: Recursive divide-and-conquer or iterative bottom-up. Leaves store array values, internals merge children. O(n) time.
    • Query: Recursive (3-case top-down) or iterative (two-pointer bottom-up). O(log n) time.
    • At most 2 partial-overlap nodes per level, so the total work is bounded by the tree height.
    • The recursive tree uses 1-based indexing with 4n space. The iterative tree packs leaves at [n, 2n−1] with 2n space.
    • Use recursive for learning, interviews, and lazy propagation. Use iterative for speed-critical competitive programming.
