    Segment Tree Patterns & Practice

    March 2026 · 16 min read

    You've learned the fundamentals, building & querying, updates, and lazy propagation. Now let's apply segment trees to real problems, explore variations beyond sums, and build a mental toolkit for recognizing when to use them.

    Pattern 1: Range Sum with Point Updates

    This is the "hello world" of segment trees. Given an array, answer sum(l, r) queries and handle update(i, val) operations.

    Classic Problems

    • Range Sum Query, Mutable (LeetCode 307): Direct application.
    • Count of Smaller Numbers After Self (LeetCode 315): Build segment tree over value range, process array right-to-left.
    • Dynamic Range Sum Queries (CSES): Point update + range sum — the cleanest pure seg tree problem.
    • Count Inversions: Same idea, segment tree on values, count elements already seen that are greater.
    Template: Range Sum + Point Update 
    class SegTree {
    int n;
    vector<long long> t;
    void build(vector<int>& a, int v, int s, int e) {
        if (s == e) { t[v] = a[s]; return; }
        int m = (s+e)/2;
        build(a, 2*v, s, m); build(a, 2*v+1, m+1, e);
        t[v] = t[2*v] + t[2*v+1];
    }
    void upd(int v, int s, int e, int i, long long val) {
        if (s == e) { t[v] = val; return; }
        int m = (s+e)/2;
        if (i <= m) upd(2*v, s, m, i, val);
        else upd(2*v+1, m+1, e, i, val);
        t[v] = t[2*v] + t[2*v+1];
    }
    long long qry(int v, int s, int e, int l, int r) {
        if (r < s || e < l) return 0;
        if (l <= s && e <= r) return t[v];
        int m = (s+e)/2;
        return qry(2*v, s, m, l, r) + qry(2*v+1, m+1, e, l, r);
    }
public:
    SegTree(vector<int>& a) : n(a.size()), t(4*n) { build(a,1,0,n-1); }
    void update(int i, int v) { upd(1,0,n-1,i,v); }
    long long query(int l, int r) { return qry(1,0,n-1,l,r); }
};

    Pattern 2: Range Min/Max Queries

    Replace sum with min (or max). The only changes:

    • Merge: min(left, right) instead of left + right
    • Identity: INT_MAX instead of 0

    Classic Problems

    • Dynamic Range Minimum Queries (CSES): The textbook RMQ application with point updates.
    • Sliding Window Maximum (LeetCode 239): Can solve with segment tree on the array, query max of sliding window. (Deque is optimal, but seg tree works and generalizes.)
    • Salary Queries (CSES): Update values, query “how many in range [a,b]?” — coordinate compress + seg tree on values.
    Template: Range Min + Point Update 
    class MinSegTree {
    int n;
    vector<int> t;
    void build(vector<int>& a, int v, int s, int e) {
        if (s == e) { t[v] = a[s]; return; }
        int m = (s+e)/2;
        build(a, 2*v, s, m); build(a, 2*v+1, m+1, e);
        t[v] = min(t[2*v], t[2*v+1]);
    }
    void upd(int v, int s, int e, int i, int val) {
        if (s == e) { t[v] = val; return; }
        int m = (s+e)/2;
        if (i <= m) upd(2*v, s, m, i, val);
        else upd(2*v+1, m+1, e, i, val);
        t[v] = min(t[2*v], t[2*v+1]);
    }
    int qry(int v, int s, int e, int l, int r) {
        if (r < s || e < l) return INT_MAX;
        if (l <= s && e <= r) return t[v];
        int m = (s+e)/2;
        return min(qry(2*v, s, m, l, r), qry(2*v+1, m+1, e, l, r));
    }
public:
    MinSegTree(vector<int>& a) : n(a.size()), t(4*n) { build(a,1,0,n-1); }
    void update(int i, int v) { upd(1,0,n-1,i,v); }
    int query(int l, int r) { return qry(1,0,n-1,l,r); }
};

    Pattern 3: Range Update + Range Query (Lazy)

    When both updates and queries operate on ranges. This requires lazy propagation.

    Classic Problems

    • Range Updates and Sums (CSES): Add a value to a range or set a range to a value, then query range sum. The core lazy propagation problem — requires careful handling of two lazy types.
    • Range Update Queries (CSES): Add a value to a range, query single elements. Simpler lazy intro.
    • Circular RMQ (Codeforces 52C): Range add + range min on a circular array. Good test of lazy propagation with modular indices.
    • Painting the Fence: Range color assignments, then query how many segments have a certain color.

    Pattern 4: Custom Merge Functions

    Sometimes the "aggregate" stored at each node is more complex than a single number. As long as two children can be merged in O(1), the segment tree works.

    Examples

    Node StoresMerge LogicUse Case
    {sum, max, min}Component-wise combineMultiple stats in one pass
    {prefix_sum, suffix_sum, total, max_subarray}Kadane-style mergeMax subarray sum with updates
    {count_zeros, count_ones}AdditionCount occurrences with flips
    {matrix 2×2}Matrix multiplicationFibonacci range queries

    Example: Maximum Subarray Sum (LeetCode 53 variant with updates)

    Each node stores four values:

    • total: sum of the range
    • prefix: maximum prefix sum
    • suffix: maximum suffix sum
    • best: maximum subarray sum within the range

    Merge is:

    Node merge(Node left, Node right) {
    return {
        .total  = left.total + right.total,
        .prefix = max(left.prefix, left.total + right.prefix),
        .suffix = max(right.suffix, right.total + left.suffix),
        .best   = max({left.best, right.best,
                       left.suffix + right.prefix})
    };
}

    This is one of the most elegant segment tree applications. It handles point updates in O(log n) and answers "what's the maximum subarray sum in [l, r]?" in O(log n).

    Pattern 5: Coordinate Compression + Segment Tree

    When values are large (up to 109) but there are few distinct values (up to 105), compress values to indices:

    1. Collect all values, sort and deduplicate.
    2. Map each value to its rank (0, 1, 2, ...).
    3. Build a segment tree over ranks, not original values.

    Used in: counting inversions, merge sort tree, order-statistics queries.

    Advanced Variations (Brief Overview)

    Persistent Segment Tree

    Every update creates a new "version" of the tree, reusing unchanged nodes. Gives access to any historical version. Uses O(n log n) space total for n updates. Classic application: kth smallest in any subarray.

    2D Segment Tree

    A segment tree of segment trees. The outer tree splits rows; each node stores an inner tree over columns. Supports 2D range queries (e.g., "sum of sub-matrix") in O(log2 n).

    Segment Tree Beats

    Handles operations like "clamp all elements in [l, r] to be ≤ v" (range-min assignment). Uses "break" and "tag" conditions to achieve amortized O(n log2 n) for a sequence of operations.

    Merge Sort Tree

    Each node stores the sorted list of elements in its range. Queries like "how many elements in [l, r] are ≤ k" can be answered in O(log2 n) using binary search at each visited node.

    Pattern Recognition Cheat Sheet

    If the problem says...Think...
    "range sum/min/max with updates"Basic segment tree
    "add value to range" + "query range"Segment tree with lazy propagation
    "count elements < k in range"Merge sort tree or persistent seg tree
    "kth smallest in range"Persistent segment tree
    "maximum subarray in range with updates"Custom merge segment tree (pattern 4)
    "2D sub-matrix queries"2D segment tree or BIT
    "offline queries on ranges"Consider sweep + segment tree
    "range query, no updates"Sparse table or prefix sums (simpler!)

    Common Pitfalls in Competitive Programming

    • Using segment tree when not needed: If there are no updates, prefix sums or sparse tables are simpler and faster.
    • Forgetting 1-based vs 0-based: Tree is 1-indexed, but the array can be either. Convert carefully at the public interface.
    • Overflow: Sum of 105 elements each up to 109 exceeds int. Use long long.
    • Wrong lazy push-down: For non-additive operations (set, min-assignment), the push-down logic differs. Think carefully about how lazy tags compose.
    • Allocating 2n instead of 4n: For non-power-of-2 sizes, 2n isn't enough. Always use 4n.

    Segment Tree vs. Fenwick Tree (BIT)

    Segment TreeFenwick Tree (BIT)
    Range queryO(log n)O(log n)
    Point updateO(log n)O(log n)
    Range updateO(log n) with lazyO(log n) for add (with trick)
    Min/Max query✔✘ (only prefix min/max)
    Custom merge✔✘
    Code length~30-50 lines~15-20 lines
    Memory4nn
    Constant factorLargerSmaller (faster in practice)

    Rule of thumb: Use Fenwick for range-sum + point-update problems (it's faster and shorter). Use segment tree for everything else (min/max, custom merges, range updates, lazy propagation).

    Practice Problems

    Curated problems from beginner to advanced. Each has a brief description, then a spoiler with progressive hints — try to solve it yourself before revealing each hint.

    Beginner: Core Segment Tree

    CSES — Dynamic Range Sum Queries

    Given an array, handle point updates and range sum queries. The quintessential first segment tree problem.

    Hints

    Hint 1: Build a segment tree where each node stores the sum of all elements in its range.

    Hint 2: For update(i, val): walk from root to the leaf at index i, set the leaf’s value, then recalculate every ancestor on the way back up.

    Hint 3: For query(l, r): decompose [l, r] into O(log n) disjoint node ranges and return their sum. Use the three-case recursion (no overlap → 0, total overlap → return, partial → recurse both children).

    CSES — Dynamic Range Minimum Queries

    Same as above but query the minimum in a range instead of the sum. Tests that you can swap the merge function.

    Hints

    Hint 1: Each node stores the minimum value in its range. The merge function is min(left, right) instead of left + right.

    Hint 2: The identity element changes: return INT_MAX (or ∞) for no-overlap base cases instead of 0.

    Hint 3: Update is identical to the sum version — change the leaf, then recalculate each parent as min(tree[2*v], tree[2*v+1]).

    LeetCode 307 — Range Sum Query Mutable

    Implement a class with update(index, val) and sumRange(left, right). Good for validating your template against LeetCode’s judge.

    Hints

    Hint 1: Build a standard sum segment tree in the constructor from the input array.

    Hint 2: update(index, val) sets the value at the leaf (not adds!). Walk to the leaf, set it to val, recalculate ancestors.

    Hint 3: sumRange(left, right) is a direct range sum query. Be careful with 0-indexing.

    CSES — Xor Queries

    Query the XOR of elements in a range. Can be solved with prefix XOR, but building a segment tree with XOR merge is great practice.

    Hints

    Hint 1: XOR is associative and commutative, so it works perfectly as a segment tree merge function.

    Hint 2: The identity element for XOR is 0 (a ⊕ 0 = a), so return 0 for no-overlap cases.

    Hint 3: Alternative without a tree: prefix XOR array where pre[i] = a[0] ⊕ a[1] ⊕ ... ⊕ a[i]. Then xor(l, r) = pre[r] ⊕ pre[l-1]. But the seg tree approach handles point updates too.

    Intermediate: Lazy Propagation

    CSES — Range Update Queries

    Add a value to all elements in a range, then query single element values. The gentlest introduction to lazy propagation.

    Hints

    Hint 1: Since queries are point queries (single index), you don’t necessarily need a full lazy seg tree. But it’s great practice to implement one.

    Hint 2: Lazy approach: range add with lazy tags. For a point query, just walk from root to leaf, pushing down lazy tags as you go.

    Hint 3: Simpler alternative: use a difference array. diff[l] += val, diff[r+1] -= val, then prefix sum gives the actual values. But this doesn’t handle interleaved updates and queries — for that, use a BIT with range-update-point-query trick or a lazy seg tree.

    CSES — Range Updates and Sums

    Support both “add d to range” and “set range to v” operations, with range sum queries. The hardest lazy propagation problem at this level.

    Hints

    Hint 1: You need two lazy tags per node: one for pending “set” and one for pending “add.” Use a flag to distinguish whether a set is pending.

    Hint 2: Tag interaction: if a “set” arrives at a node with a pending “add,” the add is discarded (overwritten). If an “add” arrives at a node with a pending “set,” the add modifies the set value.

    Hint 3: In push-down, apply “set” first (overwrite child’s value and clear its add tag), then apply “add” on top. Keep the order consistent to avoid bugs.

    Hint 4: For “set to v”: tree[node] = v × range_size. For “add d”: tree[node] += d × range_size.

    Codeforces 52C — Circular RMQ

    Range add and range min queries on a circular array. Tests lazy propagation plus handling wrap-around indices.

    Hints

    Hint 1: If l ≤ r, it’s a normal range operation. If l > r, split into two operations: [l, n−1] and [0, r].

    Hint 2: Use lazy propagation with range-add + range-min. Remember: for min, adding d to every element shifts the min by exactly d (no × range_size).

    Hint 3: Push-down: tree[child] += lazy[parent] and lazy[child] += lazy[parent]. Merge: min(left, right). Identity: LLONG_MAX.

    Codeforces 242E — XOR on Segment

    Range XOR update + range sum query. A clever modeling problem that requires thinking about bit decomposition.

    Hints

    Hint 1: XOR flips bits. Think about each bit position independently.

    Hint 2: Maintain 20 segment trees (one per bit). For bit k, each node stores the count of elements in its range that have bit k set.

    Hint 3: XOR update with value v: for each bit k where v has bit k set, flip the count: new_count = range_size − old_count. Use a lazy tag to track pending flips (0 or 1).

    Hint 4: To answer sum queries, reconstruct: sum = Σ(count[k] × 2k) across all 20 bit trees for the queried range.

    Intermediate: Counting & Inversions

    LeetCode 315 — Count of Smaller Numbers After Self

    For each element, count how many elements to its right are smaller. The classic “inversion counting with segment tree” pattern.

    Hints

    Hint 1: Process the array from right to left. Maintain a segment tree over the value range (not index range).

    Hint 2: Coordinate compress the values first (map to 0..n−1). The tree tracks how many of each value have been inserted so far.

    Hint 3: For each element with compressed value v: answer = query(0, v−1) (count of smaller values already seen), then update(v, +1).

    LeetCode 493 — Reverse Pairs

    Count pairs where i < j and nums[i] > 2 × nums[j]. Requires coordinate compression and careful ordering.

    Hints

    Hint 1: Coordinate compress both the original values AND their doubled values into one sorted set.

    Hint 2: Process from left to right. For each nums[j], query: “how many values already in the tree are > 2 × nums[j]?” Then insert nums[j].

    Hint 3: “Count of values > x” = query(compressed_index_of(x) + 1, max_index). Or equivalently, total inserted − query(0, compressed_index_of(x)).

    CSES — Salary Queries

    Employees have salaries that change. Query: how many employees have salary in [a, b]? Needs coordinate compression since salaries can be up to 109.

    Hints

    Hint 1: Collect all salary values (initial + all update values + all query bounds) upfront and coordinate compress them.

    Hint 2: Build a segment tree over the compressed value range. Each position stores the count of employees with that salary.

    Hint 3: For salary change (old → new): update(compressed(old), −1) then update(compressed(new), +1). For query [a, b]: query(compressed(a), compressed(b)).

    CSES — Nested Ranges Count

    Given n ranges, for each range count (a) how many other ranges it contains and (b) how many ranges contain it.

    Hints

    Hint 1: Sort ranges by left endpoint ascending. Break ties by right endpoint descending (larger right = might contain others).

    Hint 2: After sorting, range A contains range B if A comes before B in sorted order and A.right ≥ B.right. Sweep left-to-right, using a seg tree on compressed right endpoints.

    Hint 3: For “contains count”: when processing range i, answer = query(0, right_i) (how many rights ≤ right_i already inserted at positions sorted before i). Then update(right_i, +1).

    Hint 4: For “contained by count”: reverse the sweep direction (right to left) or use a second pass with adjusted logic.

    Advanced: Custom Merge & Complex Queries

    CSES — Subarray Sum Queries

    After point updates, query the maximum subarray sum of the entire array. Requires storing 4 fields per node.

    Hints

    Hint 1: Each node stores 4 values: total (sum), prefix (max prefix sum), suffix (max suffix sum), best (max subarray sum in range).

    Hint 2: Merge: best = max(left.best, right.best, left.suffix + right.prefix). The last case spans the split point.

    Hint 3: Also: prefix = max(left.prefix, left.total + right.prefix) and suffix = max(right.suffix, right.total + left.suffix) and total = left.total + right.total.

    Hint 4: Leaf node: all four fields equal max(0, a[i]) if you want non-negative answer, or just a[i] if empty subarrays aren’t allowed. Check the problem statement carefully.

    Codeforces 380C — Sereja and Brackets

    Query the length of the longest valid (balanced) parentheses subsequence in a range. A classic custom-merge problem.

    Hints

    Hint 1: Each node stores two values: open (unmatched opening brackets) and close (unmatched closing brackets) after greedily matching within the range.

    Hint 2: Merge: some unmatched ‘(’ from the left child match with unmatched ‘)’ from the right child. matched = min(left.open, right.close).

    Hint 3: After merging: new.open = left.open + right.open − matched, new.close = left.close + right.close − matched. The answer for a query = total characters − remaining unmatched = (r − l + 1) − new.open − new.close.

    Codeforces 474F — Ant Colony

    For a range query, find the GCD and count how many elements don’t equal the GCD (those ants “leave”).

    Hints

    Hint 1: Each node stores two fields: {gcd, count} where count = number of elements in the range that equal the gcd.

    Hint 2: Merge: if left.gcd == right.gcd, combine counts. Otherwise compute g = gcd(left.gcd, right.gcd). If g == left.gcd, count = left.count (right side values are all multiples, none equal g). Similarly for the other side. If g is smaller than both, count = 0.

    Hint 3: Answer for query [l, r] = (r − l + 1) − count. The elements NOT equal to the range GCD are the ones that leave.

    Library Checker — Point Set Range Composite

    Each array element is a linear function f(x) = ax + b. Query the composition of functions over a range. Non-commutative merge!

    Hints

    Hint 1: Composition: if left = (a₁, b₁) and right = (a₂, b₂), then (left ∘ right)(x) = a₁(a₂x + b₂) + b₁ = (a₁·a₂)x + (a₁·b₂ + b₁).

    Hint 2: This is NOT commutative! merge(A, B) ≠ merge(B, A). Your segment tree must apply left-to-right consistently: left child first, then right child.

    Hint 3: The identity function is (a=1, b=0). Return this for no-overlap base cases.

    Hint 4: All arithmetic is modular (mod 998244353). Use modular multiplication carefully to avoid overflow.

    Advanced: Persistent & Offline

    SPOJ — MKTHNUM

    Find the kth smallest element in any subarray [l, r]. The canonical persistent segment tree problem.

    Hints

    Hint 1: Build a persistent segment tree over sorted values. Process array elements left to right, creating a new version for each position.

    Hint 2: Version i contains the first i elements. To answer a query on [l, r], use version[r] − version[l−1] to get the count distribution in [l, r].

    Hint 3: Binary search on the persistent tree: at each node, check if the left subtree’s count ≥ k. If yes, go left. Otherwise subtract left count from k and go right.

    CSES — Distinct Values Queries

    Count the number of distinct values in a subarray [l, r]. Best solved offline with a sweep.

    Hints

    Hint 1: Sort queries by their right endpoint. Process the array left to right, maintaining a segment tree.

    Hint 2: For each value, track its last occurrence. When processing position i: if value a[i] was last seen at position j, do update(j, −1) then update(i, +1). This ensures each value is counted only at its rightmost occurrence.

    Hint 3: Answer for query [l, r] = query(l, r) at the time when we’ve processed up to index r. Each position with a 1 represents a distinct value in the suffix.

    Codeforces 600E — Lomsat gelral

    For each vertex in a rooted tree, find the sum of the most frequent colors in its subtree. Uses DSU on tree (small-to-large merging).

    Hints

    Hint 1: For each subtree you need the color frequency distribution. Brute-force per subtree is O(n²). The key trick is “DSU on tree” (also called small-to-large merging).

    Hint 2: Process children: keep the “heavy” child’s data (don’t reset it), and re-add all elements from light children. This gives O(n log n) total work.

    Hint 3: Maintain a global frequency array and a variable tracking the current max frequency and its sum. When adding an element, update frequency, check if it creates a new max or ties the current max.

    More practice: The USACO Guide — Segment Tree (Platinum) and CP-Algorithms: Segment Tree pages maintain comprehensive problem lists with difficulty tags. For lazy propagation specifically, see this classic Codeforces tutorial.

    Summary

    • Pattern 1: Range sum + point update, the basic template. Drop-in for many counting/inversion problems.
    • Pattern 2: Range min/max, change merge to min/max, identity to ∞/-∞.
    • Pattern 3: Range update + range query, requires lazy propagation.
    • Pattern 4: Custom merges, store complex state per node, merge in O(1).
    • Pattern 5: Coordinate compression, map large values to small ranks.
    • When there are no updates, prefer prefix sums or sparse tables.
    • When only sum + point update is needed, Fenwick tree is faster and simpler.
    • Segment trees shine with min/max queries, range updates, and complex merge logic.
