    Heavy-Light Decomposition

    July 2025 · 18 min read

    Introduction

    Suppose you have a tree with n nodes, each holding a value, and you need to answer thousands of queries of the form: "What is the sum (or max) of values on the path from node u to node v?"—with updates mixed in. A naive approach walks the path in O(n) per query. Heavy-Light Decomposition (HLD) reduces this to O(log² n) per query.

    The core idea is elegant: decompose the tree into a set of chains (paths), so that any root-to-leaf path crosses at most O(log n) chains. Then flatten the chains into a contiguous array and overlay a segment tree on top. Each path query decomposes into O(log n) contiguous segments, each answered in O(log n) by the segment tree.

    Key insight: By always choosing the child with the largest subtree (the “heavy” child), we guarantee that every time we step onto a “light” edge, the subtree size at least halves. So any path crosses at most O(log n) light edges, hence O(log n) chains.

    When to Use HLD

    • Path queries: sum, max, min, xor on paths between arbitrary nodes
    • Path updates: add a value to every node on a path
    • Subtree queries/updates (bonus: subtrees are also contiguous in the flattened array)
    • Any problem requiring combining a segment tree with tree-path operations

    Heavy vs Light Edges

    Root the tree at any node (typically node 0 or 1). For each internal node, compute the subtree size. The child with the largest subtree is the heavy child; the edge to it is a heavy edge. All other edges from that node are light edges. If there is a tie, pick any one.

    Heavy edge Light edge (dashed) Orange labels = subtree sizes

    Consider node 1: children are 2 (sz=6) and 3 (sz=3). Since 6 > 3, the edge 1→2 is heavy. Node 2: children 4 (sz=3) and 5 (sz=1), so 2→4 is heavy. Node 4: children 8 (sz=1) and 9 (sz=1) — tie, we pick 8. Node 3: children 6 (sz=2) and 7 (sz=1), so 3→6 is heavy.

    Light-edge property: If (u, v) is a light edge with u being the parent, then sz[v] ≤ sz[u] / 2. This is because v is not the heaviest child, so another child has at least as many descendants. This is the key to the O(log n) chain bound.

    Chain Decomposition

    Consecutive heavy edges form heavy paths (chains). Each node belongs to exactly one chain. A chain starts either at the root or at a node reached by a light edge from its parent, and extends downward along heavy edges.

    For our example tree, the chains are:

    • Chain 0: 1 → 2 → 4 → 8 (the heaviest path from root)
    • Chain 1: 5 (single node, reached by light edge from 2)
    • Chain 2: 3 → 6 → 10 (heavy path from light child of 1)
    • Chain 3: 7 (single node)
    • Chain 4: 9 (single node)
    Chain 0: 1→2→4→8 Chain 1: 5 Chain 2: 3→6→10 Chain 3: 7 Chain 4: 9

    Chain Head

    Each chain has a head: the topmost node in the chain (the one closest to the root). For Chain 0 the head is node 1, for Chain 2 the head is node 3. The head[v] array stores, for each node v, the head of the chain it belongs to. This is crucial for path queries: when two nodes are on different chains, we climb the one with the deeper head.

    Building the HLD (Interactive)

    The decomposition is built with two DFS passes. The first computes subtree sizes, parents, and depths. The second assigns chains: always recurse into the heavy child first (so the chain is contiguous), then handle light children (each starts a new chain).

    Flattening with DFS

    During the second DFS (chain assignment), we also assign a position pos[v] to each node — a global counter that increments as we visit nodes. Because we always visit the heavy child first, all nodes in the same chain get consecutive positions.

    This is the magic that lets a segment tree work: a chain [head, ..., tail] maps to a contiguous range [pos[head], pos[tail]] in the position array.

    The Position Array

    For our example, the DFS visits nodes in this order (heavy child first at each step):

    Node12489536107
    pos[]0123456789
    chain 0 0 0 0 4 1 2 2 2 3
    head[]1111953337

    Notice how Chain 0 occupies positions [0,3], Chain 4 is [4,4], Chain 1 is [5,5], Chain 2 is [6,8], and Chain 3 is [9,9]. Each chain is a contiguous block!

    Subtree Queries (Bonus)

    Because DFS visits all descendants of a node before returning, the entire subtree of node v occupies positions [pos[v], pos[v] + sz[v] - 1]. This means subtree queries are also just range queries on the segment tree — no extra work needed!

    Path Queries

    To query the path from u to v, we repeatedly climb the node whose chain head is deeper, querying the segment from that node up to its chain head, then jumping to the parent of the head. When both nodes are on the same chain, we do one final segment query.

    function pathQuery(u, v):
    result = identity while head[u] ≠ head[v]: // climb the deeper chain head if depth[head[u]] < depth[head[v]]: swap(u, v) result = combine(result, segQuery(pos[head[u]], pos[u])) u = parent[head[u]] // now on same chain if depth[u] > depth[v]: swap(u, v) result = combine(result, segQuery(pos[u], pos[v])) return result

    Combining with Segment Tree

    The segment tree is built on the flattened position array. When building it, seg[pos[v]] = value[v] for each node v. Then:

    • Point update on node v: update position pos[v] in the segment tree.
    • Path query (u, v): decompose into O(log n) chain segments, each queried as a range in the segment tree.
    • Path update (u, v): same decomposition, but apply range update instead of query.
    • Subtree query on v: query range [pos[v], pos[v] + sz[v] - 1].

    Code Structure

    // Build: place node values into position array
for (int v = 1; v <= n; v++)
    flat[pos[v]] = val[v];
seg.build(flat, n);

// Path query
int pathQuery(int u, int v) {
    int res = 0;  // identity for sum
    while (head[u] != head[v]) {
        if (depth[head[u]] < depth[head[v]]) swap(u, v);
        res += seg.query(pos[head[u]], pos[u]);
        u = par[head[u]];
    }
    if (depth[u] > depth[v]) swap(u, v);
    res += seg.query(pos[u], pos[v]);
    return res;
}

// Subtree query
int subtreeQuery(int v) {
    return seg.query(pos[v], pos[v] + sz[v] - 1);
}
    Time complexity: Each path query makes O(log n) segment tree queries, each taking O(log n) time, giving O(log² n) per operation. With lazy propagation, path/subtree updates are also O(log² n).

    Edge vs Vertex HLD

    The standard HLD described above is vertex-based: each node holds a value, and we query/update node values on paths. But many problems have edge weights instead.

    The Edge-to-Vertex Trick

    For each edge (u, v) where u is the parent of v, assign the edge’s weight to node v (the child). Since every non-root node has exactly one parent edge, this mapping is bijective (except the root, which has no parent edge).

    The only change in the query algorithm is: when u and v are finally on the same chain, query [pos[u]+1, pos[v]] instead of [pos[u], pos[v]] — because we must exclude the LCA node (whose stored value belongs to the edge above it, not below).

    // Edge-weight path query (u and v on same chain, depth[u] <= depth[v])
// Exclude LCA's value since it represents the edge ABOVE the LCA
res += seg.query(pos[u] + 1, pos[v]);
    Common pitfall: When u == v in the final step of an edge-weight query, the range [pos[u]+1, pos[v]] is empty — this is correct since there are no edges on a path of length 0. Make sure your segment tree handles empty ranges gracefully.

    Complete C++ Implementation

    Here is a contest-ready implementation supporting path sum queries and point updates. It uses a simple segment tree with the HLD decomposition.

    #include <bits/stdc++.h>
using namespace std;

const int MAXN = 200005;
int n, q;
int val[MAXN];
vector<int> adj[MAXN];

// HLD arrays
int sz[MAXN], dep[MAXN], par[MAXN], heavy[MAXN];
int head_node[MAXN], pos_idx[MAXN], cur_pos;

// Segment tree
long long seg[4 * MAXN];

void seg_build(int v, int l, int r, int a[]) {
    if (l == r) { seg[v] = a[l]; return; }
    int m = (l + r) / 2;
    seg_build(2*v, l, m, a);
    seg_build(2*v+1, m+1, r, a);
    seg[v] = seg[2*v] + seg[2*v+1];
}

void seg_update(int v, int l, int r, int pos, int val) {
    if (l == r) { seg[v] = val; return; }
    int m = (l + r) / 2;
    if (pos <= m) seg_update(2*v, l, m, pos, val);
    else seg_update(2*v+1, m+1, r, pos, val);
    seg[v] = seg[2*v] + seg[2*v+1];
}

long long seg_query(int v, int l, int r, int ql, int qr) {
    if (ql > qr || l > qr || r < ql) return 0;
    if (ql <= l && r <= qr) return seg[v];
    int m = (l + r) / 2;
    return seg_query(2*v, l, m, ql, qr)
         + seg_query(2*v+1, m+1, r, ql, qr);
}

// DFS 1: compute sz, dep, par, heavy child
void dfs_sz(int v, int p, int d) {
    par[v] = p; dep[v] = d; sz[v] = 1;
    int mx = 0; heavy[v] = -1;
    for (int u : adj[v]) {
        if (u == p) continue;
        dfs_sz(u, v, d + 1);
        sz[v] += sz[u];
        if (sz[u] > mx) { mx = sz[u]; heavy[v] = u; }
    }
}

// DFS 2: assign chains and positions
void dfs_hld(int v, int h) {
    head_node[v] = h;
    pos_idx[v] = cur_pos++;
    if (heavy[v] != -1)
        dfs_hld(heavy[v], h);  // same chain
    for (int u : adj[v]) {
        if (u == par[v] || u == heavy[v]) continue;
        dfs_hld(u, u);  // new chain, u is its own head
    }
}

long long path_query(int u, int v) {
    long long res = 0;
    while (head_node[u] != head_node[v]) {
        if (dep[head_node[u]] < dep[head_node[v]]) swap(u, v);
        res += seg_query(1, 0, n-1, pos_idx[head_node[u]], pos_idx[u]);
        u = par[head_node[u]];
    }
    if (dep[u] > dep[v]) swap(u, v);
    res += seg_query(1, 0, n-1, pos_idx[u], pos_idx[v]);
    return res;
}

void point_update(int v, int new_val) {
    seg_update(1, 0, n-1, pos_idx[v], new_val);
}

int flat[MAXN];

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    scanf("%d %d", &n, &q);
    for (int i = 1; i <= n; i++) scanf("%d", &val[i]);
    for (int i = 0; i < n-1; i++) {
        int u, v; scanf("%d %d", &u, &v);
        adj[u].push_back(v);
        adj[v].push_back(u);
    }
    cur_pos = 0;
    dfs_sz(1, 0, 0);
    dfs_hld(1, 1);
    for (int i = 1; i <= n; i++) flat[pos_idx[i]] = val[i];
    seg_build(1, 0, n-1, flat);

    while (q--) {
        int type; scanf("%d", &type);
        if (type == 1) {
            int v, x; scanf("%d %d", &v, &x);
            point_update(v, x);
        } else {
            int u, v; scanf("%d %d", &u, &v);
            printf("%lld\n", path_query(u, v));
        }
    }
}
    Note: For iterative DFS (needed when n > 105 to avoid stack overflow), replace the recursive dfs_sz and dfs_hld with explicit stack versions. The logic remains identical.

    Complexity Analysis

    Why O(log n) Chains on Any Path

    Consider any path from a node v up to the root. Each time the path crosses a light edge from child c to parent p, we know that sz[c] ≤ sz[p] / 2 (because c is not the heavy child). Since the subtree size at least doubles each time we go up a light edge, and the maximum subtree size is n, we can cross at most log2(n) light edges.

    Each light edge is a chain boundary, so the path crosses at most O(log n) chains. A path between any two nodes u and v goes up from u to LCA and down to v, so it crosses at most 2 · O(log n) = O(log n) chains total.

    Total Complexity

    OperationTimeExplanation
    Build HLDO(n)Two DFS passes
    Build Segment TreeO(n)Standard build on flat array
    Path QueryO(log² n)O(log n) chains × O(log n) per segment query
    Path UpdateO(log² n)Same decomposition with lazy propagation
    Point UpdateO(log n)Single segment tree update
    Subtree QueryO(log n)Single contiguous range
    Can we do better? Yes! Using Euler Tour + BIT/Fenwick tree we can handle some operations in O(log n). Also, Link-Cut Trees achieve amortized O(log n) per path operation, but HLD’s O(log² n) is usually fast enough and much simpler to implement.

    Space Complexity

    HLD requires O(n) for the decomposition arrays (sz, dep, par, heavy, head, pos) plus O(n) for the segment tree — total O(n).

    Summary

    Heavy-Light Decomposition transforms a tree into a set of chains that let you use a flat segment tree for path operations. The key steps are:

    1. Compute subtree sizes to identify heavy children.
    2. Decompose into chains along heavy edges.
    3. Flatten with DFS (heavy child first) so chains are contiguous.
    4. Build a segment tree on the flat array.
    5. Answer path queries by climbing chain heads (O(log n) jumps) and querying segments.

    The result: O(n) preprocessing, O(log² n) per query/update. It’s one of the most versatile tree decomposition techniques in competitive programming, and once you’ve written it a few times, the template becomes second nature.
