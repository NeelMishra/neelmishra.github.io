    Li Chao Tree

    July 2025 · 25 min read

    The Convex Hull Trick Problem

    Here's a problem that appears constantly in competitive programming and DP optimization:

    Problem: You have a set of lines y = m·x + b. You need to support two operations:
    1. Insert a new line y = m·x + b into the set.
    2. Query the minimum (or maximum) value among all lines at a given x-coordinate.

    Naively, each query scans all lines in O(n). The classical Convex Hull Trick (CHT) solves this in O(log n) per query, but requires lines inserted in sorted slope order, or uses complex data structures like std::set with careful bookkeeping.

    The Li Chao Tree (named after competitive programmer Li Chao) offers a beautifully simple alternative: it's a segment tree over x-coordinates where each node stores one line. Insertion and query are both O(log n), lines can be inserted in any order, and the implementation is remarkably short.

    Let's visualize the problem. Given four lines, we want to find the lower envelope, the minimum value at any queried x:

    Structure: A Segment Tree Over X-Coordinates

    The Li Chao tree is a segment tree over the range of possible x-values, not over an array of data. If queries come with x ∈ [0, 8], we build a segment tree whose leaves correspond to integer x-coordinates 0, 1, 2, …, 8.

    Each node in the tree stores exactly one line, the line that is "best" (gives the minimum value) at the midpoint of that node's x-range.

    Key properties:

    • Each node stores at most one line.
    • The stored line is the winner at the midpoint of the node's range.
    • A line that loses at the midpoint but might win in a sub-range is pushed down to the appropriate child.
    • To query at x, we walk from root to the leaf containing x, evaluating each stored line, and return the minimum.

    Insertion: The Core Algorithm

    Inserting a new line into a Li Chao tree node covering range [lo, hi] works as follows:

    function insert(node, newLine, lo, hi):
    mid = (lo + hi) / 2 curLine = node.line // currently stored line (may be null) // Compare at midpoint leftBetter = newLine(lo) < curLine(lo) midBetter = newLine(mid) < curLine(mid) if midBetter: swap(node.line, newLine) // new line wins at mid, keep it if lo == hi: return // leaf node // The loser might still win in one half if leftBetter != midBetter: insert(node.left, newLine, lo, mid) // loser might win on left else: insert(node.right, newLine, mid+1, hi) // loser might win on right

    The logic is elegant: after comparing the new line with the stored line at the midpoint, we keep the winner at mid in the current node. The loser is then recursed into the half where it might still beat the winner. Since the lines are linear (they can cross at most once), the loser can only be better on one side.

    Why Only One Recursive Call?

    Two lines can intersect at most once (they're linear). After comparing at the midpoint:

    • The winner at mid stays in the current node.
    • The loser can only be better on one side of mid, the side where it was winning before the intersection point.
    • So we recurse into exactly one child, never both.

    This guarantees O(log C) per insertion, where C is the range of x-coordinates.

    Query: Finding the Minimum at x

    Querying is even simpler than insertion. To find the minimum at a specific x:

    function query(node, x, lo, hi):
    result = node.line(x) // evaluate stored line at x if lo == hi: return result mid = (lo + hi) / 2 if x <= mid: return min(result, query(node.left, x, lo, mid)) else: return min(result, query(node.right, x, mid+1, hi))

    We walk from the root to the leaf containing x. At each node, we evaluate the stored line at x and keep track of the minimum. The path has length O(log C), so the query is O(log C).

    Key insight: Even though a node's stored line is the best at the midpoint, it may not be the best at x. But by walking the entire root-to-leaf path and taking the minimum of all stored lines, we're guaranteed to find the globally optimal line at x.

    Why It Works: The Li Chao Invariant

    The correctness of the Li Chao tree rests on a single invariant:

    Li Chao Invariant: For every node with range [lo, hi] and midpoint mid, the stored line is the best line at mid among all lines that were ever inserted into this node (i.e., passed through this node during insertion).

    Proof Sketch

    Claim: After all insertions, querying at any x returns the global minimum over all inserted lines.

    Proof by induction on tree depth:

    1. Base case (leaf): A leaf [x, x] stores the best line at x among all lines that reached it. Since every line that could be best at x must pass through this leaf (the root-to-leaf path covers all ranges containing x), the leaf's stored line is correct.
    2. Inductive step: Consider a node [lo, hi] with midpoint mid. Let L* be the globally best line at some query point x ∈ [lo, hi]. There are two cases:
      • L* is stored at this node → we find it directly.
      • L* was pushed to a child. This can only happen if L* lost at mid to the stored line. Since L* wins at x but loses at mid, the intersection point is between x and mid, so L* was correctly pushed to the child containing x. By induction, the child subtree returns the correct answer for x.
    3. The query takes the minimum of the stored line at this node and the recursive result, so it correctly finds L*.

    Complexity: Each insertion visits at most one node per level → O(log C). Each query visits exactly one node per level → O(log C). Space is O(C) for the static version, or O(n log C) for the dynamic version.

    Dynamic Li Chao Tree

    The static version requires allocating an array of size 4C. If the x-coordinates span [−109, 109], that's way too much memory. The solution: dynamic/implicit nodes.

    Instead of pre-allocating all nodes, create them on demand. Each node stores pointers (indices) to its children, initially null. When we need to recurse into a child, we create the node only then.

    struct Line {
    long long m, b;
    long long eval(long long x) { return m * x + b; }
};

struct Node {
    Line line;
    int left, right;  // child indices, -1 = null
    bool hasLine;
};

const int MAXNODES = 2000000;  // adjust based on problem
Node pool[MAXNODES];
int poolSize = 0;

int newNode() {
    pool[poolSize] = {{0, 0}, -1, -1, false};
    return poolSize++;
}

const long long INF = 1e18;
const long long LO = -1000000000LL, HI = 1000000000LL;

void insert(int v, Line seg, long long lo, long long hi) {
    if (!pool[v].hasLine) {
        pool[v].line = seg;
        pool[v].hasLine = true;
        return;
    }
    long long mid = lo + (hi - lo) / 2;
    bool leftBetter = seg.eval(lo) < pool[v].line.eval(lo);
    bool midBetter  = seg.eval(mid) < pool[v].line.eval(mid);
    if (midBetter) swap(pool[v].line, seg);
    if (lo == hi) return;
    if (leftBetter != midBetter) {
        if (pool[v].left == -1) pool[v].left = newNode();
        insert(pool[v].left, seg, lo, mid);
    } else {
        if (pool[v].right == -1) pool[v].right = newNode();
        insert(pool[v].right, seg, mid + 1, hi);
    }
}

long long query(int v, long long x, long long lo, long long hi) {
    if (v == -1) return INF;
    long long res = pool[v].hasLine ? pool[v].line.eval(x) : INF;
    if (lo == hi) return res;
    long long mid = lo + (hi - lo) / 2;
    if (x <= mid)
        return min(res, query(pool[v].left, x, lo, mid));
    else
        return min(res, query(pool[v].right, x, mid + 1, hi));
}

    This uses O(n · log C) nodes total since each of the n insertions creates at most log C nodes. For n = 105 lines and C = 2 × 109, that's about 105 × 31 ≈ 3 × 106 nodes, very manageable.

    Tip: In contests, allocate a pool of nodes (e.g., 2×106) and use an index-based approach instead of pointers. This avoids memory allocation overhead and is cache-friendlier.

    Handling Line Segments

    Sometimes lines aren't infinite: you need to add a segment of a line valid only for x ∈ [x1, x2]. The Li Chao tree handles this elegantly by modifying insertion to only recurse into nodes whose ranges overlap with the segment.

    void insertSegment(int v, Line seg, long long lo, long long hi,
                    long long segL, long long segR) {
    if (lo > segR || hi < segL) return;
    if (segL <= lo && hi <= segR) {
        // segment covers entire node range: normal insert
        insertLine(v, seg, lo, hi);
        return;
    }
    long long mid = lo + (hi - lo) / 2;
    if (pool[v].left == -1) pool[v].left = newNode();
    if (pool[v].right == -1) pool[v].right = newNode();
    insertSegment(pool[v].left, seg, lo, mid, segL, segR);
    insertSegment(pool[v].right, seg, mid + 1, hi, segL, segR);
}

    This is analogous to a segment tree range update: we decompose the segment [x1, x2] into O(log C) nodes and perform a normal Li Chao insertion at each. Total complexity: O(log2 C) per segment insertion.

    DP Optimization with Li Chao Tree

    The most important application of the Li Chao tree (and CHT in general) is DP optimization. Many DP recurrences have the form:

    dp[i] = min over j < i of { dp[j] + cost(j, i) }

    When cost(j, i) can be decomposed as a(j) · x(i) + b(j), the transition becomes a line query: each previous state j defines a line with slope a(j) and intercept b(j), and we query at x = x(i).

    Example: Minimum Cost to Split Array

    Consider the problem: given an array of n elements with weights w[i], split it into contiguous groups. The cost of a group [l, r] is (prefix[r] − prefix[l−1])2 + C. Minimize total cost.

    // dp[i] = min over j of { dp[j] + (prefix[i] - prefix[j])^2 + C }
// Expand: dp[j] + prefix[i]^2 - 2*prefix[i]*prefix[j] + prefix[j]^2 + C
// Group by j: (dp[j] + prefix[j]^2) + (-2*prefix[j]) * prefix[i] + (prefix[i]^2 + C)
//              ^^^^^ intercept b(j)    ^^^^^ slope m(j)    ^^^^^^^^^ constant for i

// So for each j, add line: m = -2*prefix[j], b = dp[j] + prefix[j]^2
// Query at x = prefix[i], then add prefix[i]^2 + C

    The naive DP is O(n2). With the Li Chao tree, we insert one line per state and query once per state, bringing it down to O(n log C).

    int root = newNode();
long long prefix[MAXN], dp[MAXN];
prefix[0] = 0;
for (int i = 1; i <= n; i++) prefix[i] = prefix[i-1] + w[i];

dp[0] = 0;
// Insert line for j=0: slope=0, intercept=0
insert(root, {0, 0}, LO, HI);

for (int i = 1; i <= n; i++) {
    long long val = query(root, prefix[i], LO, HI);
    dp[i] = val + prefix[i] * prefix[i] + C;
    // Insert line for j=i
    insert(root, {-2 * prefix[i], dp[i] + prefix[i] * prefix[i]}, LO, HI);
}
    When can you use this? The DP cost must decompose into f(j) · g(i) + h(j) where f and h depend only on j, and g depends only on i. Both f(j) and g(i) should be monotonic for classical CHT, but the Li Chao tree works even without monotonicity.

    C++ Implementation

    Static Li Chao Tree (Fixed Range)

    Use this when x-coordinates are small integers, e.g., x ∈ [0, 106]:

    struct Line {
    long long m = 0, b = 0;
    bool active = false;
    long long eval(long long x) const { return m * x + b; }
};

struct LiChaoTree {
    int n;
    vector<Line> tree;

    LiChaoTree(int _n) : n(_n), tree(4 * _n) {}

    void insert(int v, int lo, int hi, Line seg) {
        if (!tree[v].active) {
            tree[v] = seg;
            tree[v].active = true;
            return;
        }
        int mid = (lo + hi) / 2;
        bool leftBetter = seg.eval(lo) < tree[v].eval(lo);
        bool midBetter  = seg.eval(mid) < tree[v].eval(mid);
        if (midBetter) swap(tree[v], seg);
        if (lo == hi) return;
        if (leftBetter != midBetter)
            insert(2*v, lo, mid, seg);
        else
            insert(2*v+1, mid+1, hi, seg);
    }

    void insert(Line seg) { insert(1, 0, n - 1, seg); }

    long long query(int v, int lo, int hi, long long x) {
        long long res = tree[v].active ? tree[v].eval(x) : LLONG_MAX;
        if (lo == hi) return res;
        int mid = (lo + hi) / 2;
        if (x <= mid)
            return min(res, query(2*v, lo, mid, x));
        else
            return min(res, query(2*v+1, mid+1, hi, x));
    }

    long long query(long long x) { return query(1, 0, n - 1, x); }
};

    Dynamic Li Chao Tree (Large Range)

    Use this when x-coordinates span a huge range like [−109, 109]:

    struct DynLiChao {
    struct Node {
        Line line;
        int left = -1, right = -1;
        bool hasLine = false;
    };

    vector<Node> pool;
    long long lo, hi;

    DynLiChao(long long _lo, long long _hi) : lo(_lo), hi(_hi) {
        pool.reserve(2000000);
        pool.push_back({});  // root = 0
    }

    int newNode() {
        pool.push_back({});
        return (int)pool.size() - 1;
    }

    void insert(int v, long long lo, long long hi, Line seg) {
        if (!pool[v].hasLine) {
            pool[v].line = seg;
            pool[v].hasLine = true;
            return;
        }
        long long mid = lo + (hi - lo) / 2;
        bool leftBetter = seg.eval(lo) < pool[v].line.eval(lo);
        bool midBetter  = seg.eval(mid) < pool[v].line.eval(mid);
        if (midBetter) swap(pool[v].line, seg);
        if (lo == hi) return;
        if (leftBetter != midBetter) {
            if (pool[v].left == -1) pool[v].left = newNode();
            insert(pool[v].left, lo, mid, seg);
        } else {
            if (pool[v].right == -1) pool[v].right = newNode();
            insert(pool[v].right, mid + 1, hi, seg);
        }
    }

    void insert(Line seg) { insert(0, lo, hi, seg); }

    long long query(int v, long long lo, long long hi, long long x) {
        if (v == -1) return LLONG_MAX;
        long long res = pool[v].hasLine ? pool[v].line.eval(x) : LLONG_MAX;
        if (lo == hi) return res;
        long long mid = lo + (hi - lo) / 2;
        if (x <= mid)
            return min(res, query(pool[v].left, lo, mid, x));
        else
            return min(res, query(pool[v].right, mid + 1, hi, x));
    }

    long long query(long long x) { return query(0, lo, hi, x); }
};

    Usage Example

    // Static version (x in [0, 1000000])
LiChaoTree lct(1000001);
lct.insert({3, 5});    // y = 3x + 5
lct.insert({-1, 20});  // y = -x + 20
lct.insert({1, 8});    // y = x + 8
cout << lct.query(4);  // min at x=4

// Dynamic version (x in [-1e9, 1e9])

dlct.insert({3, 5});

dlct.insert({1, 8});

    Li Chao Tree vs. Convex Hull Trick

    Both solve the same problem, but they have different trade-offs:

    FeatureLi Chao TreeClassical CHT
    Insert orderAny orderSorted by slope
    Insert timeO(log C)O(1) amortized
    Query orderAny orderSorted or O(log n) binary search
    Query timeO(log C)O(log n) or O(1) amortized
    Line segmentsO(log2 C)Complex / not standard
    Dynamic (online)Yes, triviallyNeeds std::set based approach
    SpaceO(C) or O(n log C)O(n)
    Code complexity~30 lines~50-80 lines
    Constant factorModerateSmall
    PersistabilityNatural (segment tree)Difficult

    When to Use Which

    • Use Li Chao Tree when:
      • Lines/queries arrive in arbitrary order (online).
      • You need to handle line segments, not just full lines.
      • You want short, bug-free code in a contest.
      • You might need persistence (e.g., for a rollback operation).
    • Use Classical CHT when:
      • Slopes and queries are in sorted order (can use the stack-based O(1) amortized version).
      • You need the tightest possible constant factor.
      • Memory is tight and you can't afford O(n log C) nodes.
    Contest advice: In most cases, the Li Chao tree is the safer choice. It's shorter to code, harder to mess up, and handles the general case. Only switch to CHT when you're confident slopes are sorted and you need the speed.

    OperationStaticDynamicLine Segments

    Summary

    • The Li Chao tree is a segment tree over x-coordinates where each node stores the line that's best at its midpoint.
    • Insertion compares the new line with the stored line at the midpoint, keeps the winner, and recurses into exactly one child with the loser. O(log C) per insert.
    • Query walks from root to leaf, evaluating each stored line, and returns the minimum. O(log C) per query.
    • The dynamic version uses implicit nodes for huge x-ranges, using O(n log C) space.
    • Line segments are handled by decomposing the segment range into O(log C) nodes, costing O(log2 C) per segment.
    • The primary application is DP optimization: when the transition cost decomposes into a(j) · x(i) + b(j) form.
    • Compared to the classical CHT, the Li Chao tree is simpler to code, handles arbitrary insertion/query order, and naturally supports line segments and persistence.
