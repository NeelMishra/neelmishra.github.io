    B-Trees: Multi-Way Search Trees

    March 2026 · 18 min read

    Every tree we've studied so far (BST, AVL, red-black) is a binary tree: each node has at most 2 children. That works beautifully when the entire tree fits in RAM. But what happens when your data lives on a hard disk?

    A disk read fetches a whole block (typically 4-16 KB) in one operation. Reading a single key from a binary tree node wastes almost the entire block. If your tree has a million keys and height 20, you need 20 disk reads per search. Each read takes ~10ms on a spinning disk, so a search takes 200ms. That's unacceptable for a database.

    The Disk Problem

    The issue isn't the number of comparisons (log n is fine). The issue is the number of disk reads. Each tree level requires one disk access. With a binary tree of n = 106 keys, height ≈ 20 = 20 disk reads per query.

    The solution: instead of 2 children per node, allow hundreds or thousands of children per node. Pack each node full of keys so that one disk read gives us lots of information about where to go next. This dramatically reduces the tree height.

    The B-Tree Idea

    A B-tree of order m (also called minimum degree t in some textbooks) is a balanced search tree where:

    • Each node can hold up to m - 1 keys (and m children).
    • Each internal node (except root) has at least ⌈m/2⌉ children.
    • All leaves are at the same depth.
    • Keys within a node are sorted.

    For a database with 4 KB disk pages and 8-byte keys + 8-byte pointers, you can fit roughly m = 250 children per node. With n = 109 keys, the tree height is log250(109) ≈ 4. Four disk reads to search a billion records.

    Formal Properties

    For a B-tree of minimum degree t (order m = 2t):

    PropertyConstraint
    Keys per nodet-1 to 2t-1 (root can have as few as 1)
    Children per internal nodet to 2t
    All leavesSame depth
    Keys within a nodeSorted in ascending order
    Subtree orderingchild[i] contains keys between key[i-1] and key[i]
    The "minimum degree" naming: Different textbooks use different conventions. CLRS uses t (minimum degree), where each node has t-1 to 2t-1 keys. Knuth uses m (order), where each node has at most m children. They're related: m = 2t. This series uses t.

    Search: How It Works

    Searching a B-tree is a natural generalization of BST search:

    1. Start at the root.
    2. Within the current node, binary search (or linear scan) the sorted keys to find where the search key fits.
    3. If you find the key, done.
    4. If not, follow the appropriate child pointer and repeat.
    BTreeNode* search(BTreeNode* node, int key) {
    int i = 0;
    while (i < node->n && key > node->keys[i])
        i++;
    if (i < node->n && key == node->keys[i])
        return node;           // found
    if (node->leaf)
        return nullptr;        // not found
    return search(node->children[i], key);  // recurse
}

    Time per search: O(t · logt n). The t factor comes from scanning keys within a node (could be binary search for O(log t · logt n)). The logt n factor is the tree height. What matters for databases is the number of disk reads = O(logt n), which is tiny for large t.

    Height Analysis

    For a B-tree with n keys and minimum degree t:

    h ≤ log_t((n+1)/2)

    Key insight: In practice, the root node is kept in memory. So for a B-tree with t = 500, searching a billion keys requires only 3 disk reads. This is why every major database (PostgreSQL, MySQL, SQLite, Oracle) uses B-trees or B+ trees for their indices.
