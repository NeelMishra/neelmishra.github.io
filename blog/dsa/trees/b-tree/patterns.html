    B-Tree Patterns & Analysis

    March 2026 · 16 min read

    We've covered B-tree fundamentals and operations. Now let's look at the bigger picture: where B-trees actually live in production systems, how they compare to their variants, and the interview patterns you should know.

    B-Tree Variants

    VariantKey DifferenceUsed In
    B-treeData stored in all nodes (internal + leaf)General-purpose, some DBs
    B+ treeData only in leaves; leaves linkedMost relational DBs, file systems
    B* treeNodes must be 2/3 full (not 1/2)Some file systems (HFS+)
    2-3 treeB-tree with t=2 (2 or 3 children)Teaching, theoretical analysis
    2-3-4 treeB-tree with t=2 (isomorphic to red-black tree)Understanding red-black trees
    B+ trees vs B-trees: The B+ tree series covers B+ trees in depth. The short version: B+ trees store all data in the leaves and link leaves together, making range queries (SELECT * WHERE x BETWEEN 10 AND 50) dramatically faster. This is why virtually every database index is a B+ tree, not a plain B-tree.

    Where B-Trees Live

    Databases

    Every major relational database uses B-trees (almost always B+ trees) for its primary index structure:

    • PostgreSQL: B-tree is the default index type. CREATE INDEX creates a B+ tree.
    • MySQL/InnoDB: All tables are stored as a B+ tree (clustered index). Secondary indices are separate B+ trees.
    • SQLite: Uses B-trees for tables and B+ trees for indices.
    • Oracle, SQL Server: B+ tree indices everywhere.

    File Systems

    • NTFS (Windows): Master File Table uses B+ trees.
    • HFS+ (macOS): Uses B* trees (tighter fill requirement).
    • ext4 (Linux): Uses extents (not directly B-tree), but the directory index is an HTree (hash tree variant of B-tree).
    • Btrfs: Named after B-trees. Uses copy-on-write B-trees throughout.

    Key-Value Stores

    • MongoDB: Default index type is B-tree (WiredTiger engine uses B+ trees internally).
    • CouchDB: Append-only B+ trees for its storage.

    2-3-4 Trees and Red-Black Trees

    A fascinating connection: a 2-3-4 tree (B-tree with t=2) is structurally equivalent to a red-black tree. Every 2-3-4 tree can be converted to a red-black tree and vice versa. The mapping:

    • A 2-node (1 key, 2 children) → a black node
    • A 3-node (2 keys, 3 children) → a black node with one red child
    • A 4-node (3 keys, 4 children) → a black node with two red children

    This is why red-black tree rotations are often taught as "split the 4-node and push the middle up," which is exactly B-tree splitting. Understanding B-trees makes red-black trees less mysterious.

    Interview Patterns

    Pattern 1: "Why do databases use B-trees instead of binary trees?"

    The answer is disk I/O. Binary trees have O(log2 n) height, meaning log2 n disk reads. B-trees have O(logt n) height with t typically 100-1000. For a billion records, that's 30 vs 3 disk reads. Disk reads dominate latency (~10ms each for HDD), so minimizing them is critical.

    Also mention: B-tree nodes are sized to match disk pages (4-16 KB), maximizing useful data per I/O.

    Pattern 2: "B-tree vs B+ tree?"

    B-trees store data in all nodes. B+ trees store data only in leaves и link leaves together. B+ trees are better for:

    • Range queries: follow the leaf linked list instead of traversing up and down. Huge win for SELECT ... WHERE x BETWEEN a AND b.
    • Higher fanout: internal nodes have no data pointers, so they fit more keys, making the tree shorter.
    • Sequential scanning: leaf chain = fast full table scan.

    B-trees are better when you frequently look up single keys and want to find them in internal nodes (shorter search path on average).

    Pattern 3: "What is the minimum fill factor of a B-tree?"

    Each non-root node must have at least t-1 keys (equivalently, at least t children). This means each node is at least half full. The root can have as few as 1 key.

    For B* trees, the minimum is 2/3 full, which wastes less space.

    Pattern 4: "How does a B-tree grow taller?"

    A B-tree only grows taller when the root splits. When the root is full and we need to insert, we create a new root that contains the median key, with the two halves as children. This is the only way the tree gets taller, which is why all leaves always stay at the same depth.

    Similarly, the tree only shrinks when the root has 0 keys and 1 child (after a merge); we make the child the new root.
