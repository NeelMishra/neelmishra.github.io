    B-Tree Operations

    March 2026 · 24 min read

    In the previous post we saw the structure and search. Now let's tackle the harder operations: insertion (which may split nodes) and deletion (which may merge or redistribute). Both maintain the invariant that all leaves stay at the same depth.

    Insertion

    Inserting into a B-tree always happens at a leaf. We search down to find the right leaf, then insert the key in sorted position. If the leaf now has too many keys (more than 2t-1), we split it.

    Splitting a Node

    When a node has 2t-1 keys and we need to add one more, we split it into two nodes of t-1 keys each, and push the median key up to the parent.

    If the parent also becomes full after receiving the median, it splits too. This can cascade up to the root. If the root splits, a new root is created, and the tree height increases by 1. This is the only way a B-tree grows taller.

    Proactive Splitting (Top-Down)

    A clever optimization: on the way down during insertion, whenever we encounter a full node (2t-1 keys), we split it before descending into it. This guarantees the parent always has room for the median when a child splits. No need to cascade back up.

    void insert(BTree& tree, int key) {
    if (root is full) {
        // Split root: tree grows taller
        newRoot = new Node();
        newRoot->children[0] = root;
        splitChild(newRoot, 0, root);
        root = newRoot;
    }
    insertNonFull(root, key);
}

void insertNonFull(Node* node, int key) {
    if (node is leaf) {
        insert key in sorted position;
        return;
    }
    find child index i where key belongs;
    if (child[i] is full) {
        splitChild(node, i, child[i]);
        if (key > node->keys[i]) i++;
    }
    insertNonFull(child[i], key);
}

    Deletion

    Deletion is more complex than insertion. There are three cases depending on where the key lives.

    Case 1: Key is in a leaf

    Just remove it. If the leaf now has fewer than t-1 keys, we need to fix it (borrow from a sibling or merge).

    Case 2: Key is in an internal node

    Replace the key with its in-order predecessor (max key in the left child subtree) or in-order successor (min key in the right child subtree). Then recursively delete that replacement key from the child.

    Case 3: Fixing underflow

    When a child has fewer than t keys before we descend into it during deletion:

    • Borrow from sibling: If an adjacent sibling has ≥ t keys, rotate a key through the parent.
    • Merge with sibling: If both siblings have exactly t-1 keys, merge the child with a sibling and pull a key down from the parent.

    Complexity Summary

    OperationTime (comparisons)Disk reads
    SearchO(t · logt n)O(logt n)
    InsertO(t · logt n)O(logt n)
    DeleteO(t · logt n)O(logt n)
    Disk I/O is king: The number of comparisons is actually higher than a binary tree (O(log n) vs O(t logt n)). But what matters for databases is disk reads, and B-trees minimize those by packing many keys into each node. Comparing keys within a node uses fast in-memory operations.
