    Merge Sort Tree

    July 2025 · 22 min read

    What Is a Merge Sort Tree?

    A Merge Sort Tree is a segment tree where each node stores the sorted array of all elements in the range it covers. Instead of storing a single aggregate value (sum, min, max), every internal node holds a sorted list obtained by merging the sorted lists of its two children — exactly the merge step of merge sort.

    This structure allows you to answer powerful range queries that ordinary segment trees cannot handle efficiently:

    • Count of elements $\leq x$ in any subarray $[l, r]$
    • $k$-th smallest element in any subarray $[l, r]$
    • Count of elements in $[x, y]$ within $[l, r]$
    • Number of inversions within any subrange
    Core Idea: Build a segment tree normally, but instead of storing one number per node, store the entire sorted subarray. Queries use binary search at each visited node, leveraging the sorted order for fast counting.

    The trade-off is space: we use $O(n \log n)$ memory instead of $O(n)$. But the payoff is enormous — problems that would otherwise require persistent segment trees or complex offline algorithms become straightforward online queries.

    Building the Tree

    The construction mirrors merge sort exactly. Start with leaves (single elements), then merge upward:

    Leaves

    Each leaf node stores a single-element array: [arr[i]]. A single element is trivially sorted.

    Merge Children

    Each internal node merges the sorted arrays of its left and right children (the merge step from merge sort) to produce its own sorted array.

    Root

    The root contains the entire array, sorted. Every intermediate node contains the sorted version of the subarray it covers.

    Space Complexity

    Each element appears in exactly $\lceil \log_2 n \rceil + 1$ nodes (once at its leaf, once at each ancestor up to the root). Since there are $n$ elements, total storage is $O(n \log n)$.

    Time Complexity

    Building takes $O(n \log n)$ — the same as merge sort. At each level of the tree, we merge all elements once, and there are $O(\log n)$ levels.

    Each node shows its range and the sorted array it stores. Notice how leaves hold single values and parents merge their children's sorted lists:

    Build Code

    vector<int> tree[4 * MAXN]; // each node stores a sorted vector

void build(int node, int start, int end, int arr[]) {
    if (start == end) {
        tree[node].push_back(arr[start]);
        return;
    }
    int mid = (start + end) / 2;
    build(2 * node, start, mid, arr);
    build(2 * node + 1, mid + 1, end, arr);
    // merge the two sorted children
    merge(tree[2 * node].begin(), tree[2 * node].end(),
          tree[2 * node + 1].begin(), tree[2 * node + 1].end(),
          back_inserter(tree[node]));
}

    Count of Elements ≤ x in Range [l, r]

    This is the fundamental query. Given a range $[l, r]$ and a value $x$, count how many elements in $\text{arr}[l \ldots r]$ are $\leq x$.

    Algorithm

    1. Traverse the segment tree exactly as a standard range query.
    2. When a node's range is fully inside $[l, r]$: perform upper_bound(x) on the node's sorted array. The iterator position gives the count of elements $\leq x$.
    3. When a node's range partially overlaps $[l, r]$: recurse into both children and sum the results.
    4. When a node's range is completely outside $[l, r]$: return 0.

    At each fully-contained node, the binary search takes $O(\log n)$ time. We visit $O(\log n)$ nodes in total, giving $O(\log^2 n)$ per query.

    int countLeq(int node, int start, int end, int l, int r, int x) {
    if (r < start || end < l) return 0;          // outside
    if (l <= start && end <= r) {                // fully inside
        return upper_bound(tree[node].begin(),
                           tree[node].end(), x)
               - tree[node].begin();
    }
    int mid = (start + end) / 2;
    return countLeq(2*node, start, mid, l, r, x)
         + countLeq(2*node+1, mid+1, end, l, r, x);
}

    Why $O(\log^2 n)$?

    The segment tree traversal visits $O(\log n)$ nodes. At each fully-contained node, we run upper_bound which is $O(\log(\text{node size}))$. The node size can be up to $n$, so each binary search is $O(\log n)$. Multiplying: $O(\log n) \times O(\log n) = O(\log^2 n)$.

    K-th Smallest Element in Range [l, r]

    Given a range $[l, r]$ and an integer $k$, find the $k$-th smallest element. This is the classic range order statistics problem.

    Algorithm: Binary Search on the Answer

    The key insight: if we can count how many elements $\leq x$ exist in $[l, r]$, we can binary search on $x$ to find the smallest value where the count reaches $k$.

    Binary Search on Value

    Binary search over the sorted array at the root (which contains all elements). For a candidate value $x$, ask: "How many elements in $[l, r]$ are $\leq x$?" If the count is $\geq k$, search left (smaller values). Otherwise, search right (larger values).

    1. Let sorted = tree[1] (root's sorted array = entire array sorted).
    2. Binary search on indices of sorted: lo = 0, hi = n - 1.
    3. For mid: let x = sorted[mid]. Count elements $\leq x$ in $[l, r]$.
    4. If count $\geq k$: answer could be x or smaller → hi = mid.
    5. If count $< k$: answer must be larger → lo = mid + 1.
    6. When lo == hi: sorted[lo] is the $k$-th smallest.
    int kthSmallest(int l, int r, int k, int n) {
    int lo = 0, hi = (int)tree[1].size() - 1;
    while (lo < hi) {
        int mid = (lo + hi) / 2;
        int cnt = countLeq(1, 0, n - 1, l, r, tree[1][mid]);
        if (cnt >= k) hi = mid;
        else            lo = mid + 1;
    }
    return tree[1][lo];
}

    Complexity

    The outer binary search runs $O(\log n)$ iterations (searching over $n$ distinct candidates). Each iteration calls countLeq which is $O(\log^2 n)$. Total: $O(\log^3 n)$.

    Fractional Cascading Optimization

    The $O(\log^2 n)$ count query can be improved to $O(\log n)$ using fractional cascading — an elegant technique that eliminates the binary search at each node by propagating position information from parent to children.

    The Problem with Repeated Binary Searches

    In the standard approach, when we visit a node and binary search for $x$, then visit its child, we binary search for $x$ again. But the child's sorted array is a subset of the parent's — we're doing redundant work.

    The Idea

    For each element in a node's sorted array, precompute its position in both children's sorted arrays. Then, given the position in the parent, we can jump directly to the correct position in each child in $O(1)$.

    Augmented Build

    For each node, store two auxiliary arrays: left_pos[i] = number of elements among the first $i$ elements of this node that went to the left child; right_pos[i] = same for the right child.

    Cascaded Query

    Binary search once at the root to find the position of $x$. Then, as you recurse, use the precomputed left_pos and right_pos arrays to determine the position in each child in $O(1)$. No more binary searches at lower levels.

    Complexity Improvement

    QueryWithout FCWith FC
    Count $\leq x$ in $[l, r]$$O(\log^2 n)$$O(\log n)$
    $k$-th smallest$O(\log^3 n)$$O(\log^2 n)$

    Implementation Sketch

    // During build, also compute position mappings
void build(int node, int start, int end, int arr[]) {
    if (start == end) {
        tree[node].push_back(arr[start]);
        return;
    }
    int mid = (start + end) / 2;
    build(2*node, start, mid, arr);
    build(2*node+1, mid+1, end, arr);
    merge(tree[2*node].begin(), tree[2*node].end(),
          tree[2*node+1].begin(), tree[2*node+1].end(),
          back_inserter(tree[node]));

    // Build position arrays
    int sz = tree[node].size();
    leftPos[node].resize(sz + 1, 0);
    for (int i = 0; i < sz; i++) {
        leftPos[node][i+1] = leftPos[node][i];
        // Check if tree[node][i] came from left child
        if (leftPos[node][i+1] < (int)tree[2*node].size()
            && tree[2*node][leftPos[node][i+1]] == tree[node][i])
            leftPos[node][i+1]++;
    }
}

// Query: binary search only at root, then cascade
int countLeqFC(int node, int start, int end,
               int l, int r, int pos) {
    if (r < start || end < l) return 0;
    if (l <= start && end <= r) return pos;
    int mid = (start + end) / 2;
    int lpos = leftPos[node][pos];      // O(1) lookup
    int rpos = pos - lpos;              // O(1) lookup
    return countLeqFC(2*node, start, mid, l, r, lpos)
         + countLeqFC(2*node+1, mid+1, end, l, r, rpos);
}
    Trade-off: Fractional cascading adds implementation complexity and extra memory (the position arrays). For most competitive programming problems, the $O(\log^2 n)$ approach is fast enough. Use FC when you need to squeeze out that extra log factor — typically when $n, q \geq 10^5$ and TLE is tight.

    Merge Sort Tree vs Persistent Segment Tree

    Both structures solve range order statistics. Here's how they compare:

    Aspect Merge Sort Tree Persistent Segment Tree
    Space $O(n \log n)$ $O(n \log n)$
    Build time $O(n \log n)$ $O(n \log n)$
    Count $\leq x$ $O(\log^2 n)$ $O(\log n)$
    $k$-th smallest $O(\log^3 n)$ or $O(\log^2 n)$ with FC $O(\log n)$
    Updates Difficult (rebuild sorted arrays) $O(\log n)$ new version per update
    Implementation Simple — just vectors + upper_bound Complex — pointer-based, node allocation
    Memory constant Small (contiguous vectors) Larger (node pointers, allocator)
    Cache performance Good (sorted vectors are contiguous) Poor (pointer chasing)
    Debugging Easy (print sorted arrays) Hard (tree of pointers)
    When to choose which:
    • Merge Sort Tree when you need a quick, correct solution and $O(\log^2 n)$ is fast enough. Great for count queries, simpler problems, or when you want easy debugging.
    • Persistent Segment Tree when you need $O(\log n)$ per query, point updates between queries, or the problem has tight time limits with $n, q \geq 2 \times 10^5$.

    C++ Implementation (Contest-Ready)

    Here's a complete, copy-paste-ready implementation covering build, count, and $k$-th smallest:

    #include <bits/stdc++.h>
using namespace std;

const int MAXN = 100005;
vector<int> tree[4 * MAXN];
int arr[MAXN];

void build(int node, int start, int end) {
    if (start == end) {
        tree[node].push_back(arr[start]);
        return;
    }
    int mid = (start + end) / 2;
    build(2 * node, start, mid);
    build(2 * node + 1, mid + 1, end);
    merge(tree[2*node].begin(), tree[2*node].end(),
          tree[2*node+1].begin(), tree[2*node+1].end(),
          back_inserter(tree[node]));
}

// Count elements <= x in [l, r]
int countLeq(int node, int start, int end,
             int l, int r, int x) {
    if (r < start || end < l) return 0;
    if (l <= start && end <= r)
        return (int)(upper_bound(tree[node].begin(),
                                  tree[node].end(), x)
                     - tree[node].begin());
    int mid = (start + end) / 2;
    return countLeq(2*node, start, mid, l, r, x)
         + countLeq(2*node+1, mid+1, end, l, r, x);
}

// Count elements in [x, y] within [l, r]
int countRange(int node, int start, int end,
               int l, int r, int x, int y) {
    return countLeq(node, start, end, l, r, y)
         - countLeq(node, start, end, l, r, x - 1);
}

// K-th smallest in [l, r]
int kthSmallest(int l, int r, int k, int n) {
    int lo = 0, hi = (int)tree[1].size() - 1;
    while (lo < hi) {
        int mid = (lo + hi) / 2;
        int cnt = countLeq(1, 0, n-1, l, r, tree[1][mid]);
        if (cnt >= k) hi = mid;
        else            lo = mid + 1;
    }
    return tree[1][lo];
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;
    for (int i = 0; i < n; i++) cin >> arr[i];

    build(1, 0, n - 1);

    while (q--) {
        int type;
        cin >> type;
        if (type == 1) {
            // Count elements <= x in [l, r]
            int l, r, x;
            cin >> l >> r >> x;
            cout << countLeq(1, 0, n-1, l, r, x) << "\n";
        } else {
            // K-th smallest in [l, r]
            int l, r, k;
            cin >> l >> r >> k;
            cout << kthSmallest(l, r, k, n) << "\n";
        }
    }
    return 0;
}
    Tip: If you need count of elements $\geq x$, compute (r - l + 1) - countLeq(l, r, x - 1). For elements strictly $< x$, use lower_bound instead of upper_bound.

    Applications

    1. Range Order Statistics

    The most direct use: answer "$k$-th smallest in $[l, r]$" queries online. Unlike offline approaches (e.g., persistent segment tree built on sorted order), the merge sort tree works directly on the original array.

    2. Counting Inversions in a Range

    An inversion is a pair $(i, j)$ where $i < j$ but $\text{arr}[i] > \text{arr}[j]$. To count inversions in a subarray $[l, r]$:

    • For each index $i$ from $l$ to $r$, count elements in $[i+1, r]$ that are less than $\text{arr}[i]$.
    • This is countLeq(i+1, r, arr[i]-1) — directly answered by the merge sort tree.

    Total: $O(n \log^2 n)$ for the full range, or per-query for arbitrary ranges.

    3. Count of Distinct Values in Range

    Not directly solvable with a basic merge sort tree, but with a modified build that stores (value, first_occurrence) pairs, you can count distinct values by counting elements whose first occurrence is within $[l, r]$.

    4. Range Frequency Queries

    Count how many times a specific value $v$ appears in $[l, r]$: compute countLeq(l, r, v) - countLeq(l, r, v-1). This gives exact frequency in $O(\log^2 n)$.

    5. Quantile Queries for Streaming / Competitive

    Find the median of any subarray: use $k$-th smallest with $k = \lfloor(r - l + 2)/2\rfloor$. Useful in problems asking for medians of sliding windows or arbitrary subarrays.

    6. Threshold Counting

    Problems of the form "count subarrays where more than half the elements are $\leq x$" can be decomposed into merge sort tree queries combined with two-pointer or binary search on subarray boundaries.

    Common Pitfalls & Edge Cases

    • Duplicate elements: upper_bound correctly handles duplicates. If using lower_bound, remember it gives the first position $\geq x$, not $\leq x$.
    • 0-indexed vs 1-indexed: Be consistent. The implementation above is 0-indexed for the array and 1-indexed for the segment tree.
    • Memory: For $n = 10^5$, the tree uses ~$n \log n \approx 1.7 \times 10^6$ integers ≈ 6.8 MB. For $n = 2 \times 10^5$, it's ~14 MB. Watch the memory limit.
    • Build before query: All elements must be inserted before any query. Merge sort trees don't support efficient online insertions.
    • K-th smallest with duplicates: The binary search on the root's sorted array handles duplicates correctly because upper_bound ensures we count all copies.

    Key Takeaways

    • A Merge Sort Tree is a segment tree where each node stores the sorted subarray of its range.
    • Build time and space are both $O(n \log n)$ — the same as merge sort.
    • Count of elements $\leq x$ in $[l, r]$: $O(\log^2 n)$ via binary search at each visited node.
    • $k$-th smallest in $[l, r]$: $O(\log^3 n)$ by binary searching on the answer value.
    • Fractional cascading reduces queries by one $\log$ factor, at the cost of implementation complexity.
    • Compared to persistent segment trees, merge sort trees are simpler to implement and debug, with slightly worse asymptotic complexity.
    • The structure is static — efficient updates are not supported. Use persistent structures if you need dynamic updates.
    • In competitive programming, the merge sort tree is the go-to when you need order statistics queries and want clean, fast-to-write code.
