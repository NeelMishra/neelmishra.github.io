    Palindromic Tree (Eertree)

    July 2026 · 22 min read

    How many distinct palindromic substrings does a string have? What are they? What is the longest palindromic suffix at each position? Can you factor a string into the minimum number of palindromes?

    All of these questions can be answered by a single, elegant data structure: the Palindromic Tree, also called the Eertree (introduced by Mikhail Rubinchik and Arseny Shur in 2015). It stores every distinct palindromic substring of a string using at most n + 2 nodes, and builds itself online in O(n) time.

    That's remarkable. A string of length n can have O(n2) substrings, yet it has at most n + 1 distinct palindromic substrings (plus two special root nodes). The eertree captures all of them in a compact, automaton-like structure with edges and suffix links — similar in spirit to a suffix automaton, but specialized for palindromes.

    Why "Eertree"? The name comes from "palindromE trEE" — it's a palindrome itself. The structure is also called a palindromic automaton in some competitive programming communities.

    Structure

    The palindromic tree has two special root nodes and up to n additional nodes, one for each distinct palindromic substring:

    • Odd-length root (length = −1): An imaginary node that serves as the parent for all length-1 palindromes. Its length of −1 is a clever trick — when we extend it by adding a character on both sides, we get a string of length −1 + 2 = 1.
    • Even-length root (length = 0): Represents the empty string. Parent for all even-length palindromes that aren't built by extending a shorter even palindrome.
    • Regular nodes: Each represents a distinct palindromic substring. Stores its length, a suffix link, and a map of edge transitions.

    Edges

    An edge labeled with character c from node u to node v means: the palindrome represented by v is obtained by prepending and appending c to the palindrome represented by u. If u represents palindrome P, then v represents cPc.

    Suffix Links

    Each node has a suffix link pointing to its longest proper palindromic suffix. This is analogous to the suffix links in a suffix automaton or failure links in Aho-Corasick, but for palindromes. The suffix link of both roots points to the odd-length root.

    Key properties of the palindromic tree:

    • At most n + 2 nodes for a string of length n (n distinct palindromic substrings + 2 roots).
    • Each node has at most |Σ| outgoing edges (one per character in the alphabet).
    • Every node except the roots has exactly one incoming edge and one suffix link.
    • The tree is built online: characters are processed one at a time, left to right.

    Online Construction

    The construction algorithm processes the string character by character. At each step, we maintain a pointer to the node representing the longest palindromic suffix of the string processed so far. When we add a new character s[i]:

    1. Find the right parent: Starting from the current longest palindromic suffix node, follow suffix links until we find a node u such that s[i] matches the character just before u's occurrence. Formally, we need s[i] == s[i - len(u) - 1].
    2. Check if edge exists: If node u already has an edge labeled s[i], the palindrome s[i] + u + s[i] already exists — just follow the edge and update the current pointer.
    3. Create new node: If no such edge exists, create a new node v with length len(u) + 2, and add an edge from u to v labeled s[i].
    4. Set suffix link: To find the suffix link for the new node v, continue following suffix links from u to find the next node w where s[i] == s[i - len(w) - 1], then v's suffix link points to the node reached by following w's edge labeled s[i].

    Eertree Construction — "abacaba"

    Suffix Links in Detail

    Suffix links are the backbone of the palindromic tree — they enable the O(n) construction. Each node's suffix link points to the node representing its longest proper palindromic suffix.

    For example, if a node represents "abacaba", its suffix link points to the node for "aba" (the longest proper palindromic suffix of "abacaba"). The suffix link of "aba" points to "a", and "a"'s suffix link points to the odd root.

    During construction, suffix links serve a critical role: when adding character s[i], we follow suffix links from the current node until we find a palindrome that can be extended by s[i]. The condition is that the character just before the palindrome in the original string matches s[i].

    The amortized O(n) time comes from the fact that the total length of the longest palindromic suffix can increase by at most 1 per character added, but can decrease arbitrarily. Across the entire string, the suffix link traversals telescope, giving linear total work.

    Counting Palindromes

    Number of Distinct Palindromic Substrings

    This is the simplest application: the number of nodes in the eertree (minus the two roots) equals the number of distinct palindromic substrings. Just build the tree and count.

    // After building the eertree:
int distinct_palindromes = tree.size - 2; // subtract two roots
    Upper bound: Any string of length n has at most n + 1 distinct palindromic substrings (including the empty string). This bound is tight — the string "abcdef..." with all distinct characters achieves n palindromes (each single character) plus the empty string.

    Frequency of Each Palindrome

    During construction, each time we land on a node (either by creating it or by following an existing edge), we increment that node's count field. After construction, we propagate counts down the suffix links in reverse topological order:

    // Propagate counts: process nodes in reverse order of creation
for (int i = tree.size - 1; i >= 2; i--) {
    tree.nodes[tree.nodes[i].suf_link].count += tree.nodes[i].count;
}

    After propagation, count[v] gives the total number of occurrences of palindrome v in the string. This works because every occurrence of a palindrome P also contains an occurrence of every palindromic suffix of P.

    Counting Palindromes — "abba"

    Palindromic Factorization

    Given a string s, the palindromic factorization problem asks: what is the minimum number of palindromes that s can be split into? For example:

    • "abacaba" → 1 (the whole string is a palindrome)
    • "abacc" → 2 ("aba" + "cc")
    • "abcd" → 4 ("a" + "b" + "c" + "d")

    The eertree enables an O(n log n) or even O(n) solution using a technique called the series decomposition of suffix links. Here's the high-level approach:

    DP Formulation

    Let dp[i] = minimum number of palindromes to factorize s[0..i-1]. We want dp[n].

    For each position i, consider every palindromic suffix s[j..i-1]. Then dp[i] = min(dp[j] + 1) over all such j. The eertree gives us all palindromic suffixes via the suffix link chain, but naively iterating the chain can be O(n) per position.

    Series Links (Quick Links)

    To achieve O(n) total, we use series links. A series is a maximal chain of nodes where each node's length differs from its suffix link's length by the same amount. Define:

    • diff[v] = len[v] - len[suf_link[v]]
    • series_link[v] = the highest ancestor of v in the suffix link chain that has the same diff value.

    We maintain a helper value g[v] for each node in the current series. When processing position i:

    // For each position i, after finding the node v = longest palindromic suffix:
while (v is not a root) {
    g[v] = dp[i - len[series_link[v]] - diff[v]];
    if (diff[v] == diff[suf_link[v]])
        g[v] = min(g[v], g[suf_link[v]]);
    dp[i] = min(dp[i], g[v] + 1);
    v = series_link[v];  // jump to next series
}

    Each position only visits O(log n) series (amortized), giving O(n log n) total, or O(n) with additional tricks.

    C++ Implementation

    Here is a clean, contest-ready palindromic tree implementation. It supports:

    • Online character-by-character construction
    • Counting distinct palindromic substrings
    • Frequency counting with suffix link propagation
    • Series links for palindromic factorization
    struct PalindromicTree {
    struct Node {
        int len, suf_link, cnt;
        int series_link, diff;
        int g; // helper for factorization DP
        map<char, int> edges;
    };

    vector<Node> t;
    string s;
    int last; // node for longest palindromic suffix of s

    PalindromicTree() {
        // Node 0: odd root (len = -1)
        t.push_back({-1, 0, 0, 0, 0, 0, {}});
        // Node 1: even root (len = 0)
        t.push_back({0, 0, 0, 0, 0, 0, {}});
        last = 1; // start at even root
    }

    int getSuffix(int v, int i) {
        // Follow suffix links until s[i] == s[i - len(v) - 1]
        while (i - t[v].len - 1 < 0 || s[i - t[v].len - 1] != s[i])
            v = t[v].suf_link;
        return v;
    }

    void addChar(char c) {
        s += c;
        int i = (int)s.size() - 1;

        // 1. Find node whose palindrome can be extended by c
        int cur = getSuffix(last, i);

        if (t[cur].edges.count(c)) {
            // Palindrome already exists
            last = t[cur].edges[c];
            t[last].cnt++;
            return;
        }

        // 2. Create new node
        int now = t.size();
        t.push_back({t[cur].len + 2, 0, 1, 0, 0, 0, {}});
        t[cur].edges[c] = now;

        // 3. Set suffix link
        if (t[now].len == 1) {
            t[now].suf_link = 1; // single char -> even root
        } else {
            int q = getSuffix(t[cur].suf_link, i);
            t[now].suf_link = t[q].edges[c];
        }

        // 4. Set series link and diff
        t[now].diff = t[now].len - t[t[now].suf_link].len;
        if (t[now].diff == t[t[now].suf_link].diff)
            t[now].series_link = t[t[now].suf_link].series_link;
        else
            t[now].series_link = t[now].suf_link;

        last = now;
    }

    // Count distinct palindromic substrings
    int countDistinct() {
        return (int)t.size() - 2;
    }

    // Propagate frequency counts via suffix links
    void propagateCounts() {
        for (int i = (int)t.size() - 1; i >= 2; i--)
            t[t[i].suf_link].cnt += t[i].cnt;
    }

    // Minimum palindrome factorization
    // Returns dp[n] = min palindromes to split s[0..n-1]
    vector<int> factorize() {
        int n = s.size();
        vector<int> dp(n + 1, n + 1);
        dp[0] = 0;

        // Rebuild with factorization DP
        PalindromicTree pt;
        for (int i = 0; i < n; i++) {
            pt.addChar(s[i]);
            // Walk series links from last node
            for (int v = pt.last; v > 1; v = pt.t[v].series_link) {
                pt.t[v].g = dp[i + 1 - pt.t[pt.t[v].series_link].len
                            - pt.t[v].diff];
                if (pt.t[v].diff == pt.t[pt.t[v].suf_link].diff)
                    pt.t[v].g = min(pt.t[v].g,
                                    pt.t[pt.t[v].suf_link].g);
                dp[i + 1] = min(dp[i + 1], pt.t[v].g + 1);
            }
        }
        return dp;
    }
};
    Contest tip: Replace map<char, int> with int edges[26] (initialized to −1) for lowercase-only problems. This avoids map overhead and is significantly faster.

    A faster array-based version for competitive programming:

    struct Eertree {
    static const int MAXN = 300005;

    struct Node {
        int len, suf, cnt, diff, slink, g;
        int to[26];
        Node() { memset(to, -1, sizeof(to)); cnt = 0; g = 0; }
    } t[MAXN];

    int sz, last;
    char s[MAXN];
    int n;

    void init() {
        sz = 2; last = 1; n = 0;
        t[0].len = -1; t[0].suf = 0; // odd root
        t[1].len = 0;  t[1].suf = 0; // even root
    }

    int getLink(int v, int i) {
        while (i - t[v].len - 1 < 0 || s[i - t[v].len - 1] != s[i])
            v = t[v].suf;
        return v;
    }

    void add(char c) {
        s[n] = c;
        int cur = getLink(last, n);
        int ch = c - 'a';

        if (t[cur].to[ch] != -1) {
            last = t[cur].to[ch];
            t[last].cnt++;
            n++;
            return;
        }

        int now = sz++;
        t[now] = Node();
        t[now].len = t[cur].len + 2;
        t[now].cnt = 1;
        t[cur].to[ch] = now;

        if (t[now].len == 1) {
            t[now].suf = 1;
        } else {
            int q = getLink(t[cur].suf, n);
            t[now].suf = t[q].to[ch];
        }

        t[now].diff = t[now].len - t[t[now].suf].len;
        t[now].slink = (t[now].diff == t[t[now].suf].diff)
                        ? t[t[now].suf].slink : t[now].suf;

        last = now;
        n++;
    }
};

    Complexity Analysis

    Time: O(n)

    Each character addition involves two getSuffix calls (one for finding the parent, one for the suffix link). Although each call may follow multiple suffix links, the amortized total across all n characters is O(n).

    Proof sketch: Let Φ = length of the longest palindromic suffix. When we add a character, Φ can increase by at most 1 (if the longest palindromic suffix grew by 1). Each suffix link hop decreases Φ by at least 1. Since Φ ≤ n and increases by at most n total, the total number of suffix link hops is at most 2n = O(n).

    Space: O(n · |Σ|)

    We store at most n + 2 nodes, each with up to |Σ| edge pointers. With the array implementation, this is O(26n) = O(n) for lowercase English. With the map implementation, the space is O(n) since each node has at most one incoming edge, giving at most n edges total.

    Applications

    1. Counting Distinct Palindromic Substrings

    The most direct application. Build the eertree and return size - 2. This is a classic competitive programming problem that previously required hashing or suffix structures.

    2. Number of Palindromic Substrings (with Repetitions)

    After count propagation, sum up all count[v] for non-root nodes. This gives the total number of palindromic substrings including duplicates.

    3. Longest Palindromic Substring

    Track the maximum len value seen during construction. The eertree gives you this naturally without needing Manacher's algorithm, though both run in O(n).

    4. Minimum Palindromic Factorization

    As described in the factorization section, the eertree with series links gives an efficient DP for splitting a string into the fewest palindromes.

    5. Palindromic Rich Strings

    A string is palindromic rich (or just "rich") if it has the maximum possible number of distinct palindromic substrings: n + 1. The eertree can identify rich strings instantly by checking if every character addition creates a new node.

    6. Building on Two Strings

    You can merge two eertrees or build a "generalized" eertree for problems involving palindromic substrings common to two strings, similar to generalized suffix trees.

    7. Online Queries

    Since the eertree is built online, it naturally supports queries like "how many distinct palindromes exist in the first i characters?" — just track the count as you build.

    Eertree vs. Manacher's Algorithm

    Both solve palindrome-related problems in O(n), but they capture different information:

    FeatureManacher'sEertree
    What it computesPalindrome radii at each centerAll distinct palindromic substrings
    Distinct palindrome countRequires post-processingDirect (size − 2)
    Frequency of each palindrome✗ Not directly✓ Via count propagation

    Rule of thumb: If you only need the longest palindrome or palindrome radii, use Manacher's. For anything involving distinct palindromic substrings, frequencies, or palindromic factorization, reach for the eertree.

    Summary

    The palindromic tree (eertree) is one of the most elegant string data structures in competitive programming:

    • It captures every distinct palindromic substring in at most n + 2 nodes.
    • It builds online in O(n) time and space.
    • Suffix links enable frequency counting, palindromic factorization, and more.
    • Series links unlock efficient DP for minimum palindromic factorization.
    • It strictly subsumes Manacher's algorithm for problems about distinct palindromes.

    When you see a problem about palindromic substrings — counting them, finding their frequencies, factorizing strings into palindromes — the eertree should be your first thought.
