    Centroid Decomposition

    April 2026 · 35 min read

    Centroid decomposition is one of the most elegant and powerful techniques in competitive programming. It takes a tree—an inherently irregular, unbalanced structure—and imposes a divide-and-conquer framework on it that guarantees O(log n) depth. The result: problems that seem to require O(n²) can be solved in O(n log n).

    If you've ever struggled with "count all paths in a tree satisfying property X," centroid decomposition is almost certainly the right tool. This post builds the technique from scratch with interactive animations, full C++ code, and classic contest problems.

    Introduction: Divide and Conquer on Trees

    What is a centroid?

    Given a tree with n nodes, a centroid is a node whose removal results in no remaining connected component having more than n/2 nodes. Think of it as the "balance point" of the tree.

    For arrays, divide and conquer is straightforward: pick the middle element, recurse on both halves. For trees, there's no obvious "middle." A tree could be a long chain (very unbalanced) or a star graph. The centroid gives us a universal way to split any tree into roughly equal parts.

    Why does this matter?

    Consider this problem: Count the number of pairs of nodes at distance exactly k in a tree. A brute-force approach computes all-pairs distances in O(n²). But with centroid decomposition:

    1. Find the centroid c of the tree.
    2. Count all paths that pass through c (paths between nodes in different subtrees of c).
    3. Remove c and recursively solve each remaining component.

    Because each component has at most n/2 nodes, the recursion depth is O(log n). At each level of recursion, we do O(n) total work. The result: O(n log n) total.

    Key insight: Every path in the tree either passes through the centroid, or lies entirely within one of the subtrees formed by removing the centroid. This partitions all paths cleanly.

    Finding the Centroid

    The algorithm

    Finding the centroid is a two-step process:

    1. Compute subtree sizes via a DFS rooted at any node.
    2. Walk toward the heavy child: Starting at the root, move to the child whose subtree has the most nodes. Stop when no child's subtree exceeds n/2.

    More precisely, for a node v with subtree size sz[v], the "component above" v has size n - sz[v]. Node v is the centroid when:

    max(n − sz[v], max over children c of sz[c]) ≤ n/2

    Proof: a centroid always exists

    Claim: Every tree with n ≥ 1 nodes has at least one centroid.

    Proof: Start at any node. If it satisfies the centroid condition, we're done. Otherwise, there exists some child c whose subtree has more than n/2 nodes. Move to c. When we do this, the component "above" us shrinks (since we moved away from it), but more importantly, c's subtree (which is now the part of the tree below us) gets smaller—it loses the subtree of the sibling we left behind. Since each move strictly decreases the maximum component size, and the tree is finite, we must eventually reach a node where all components have size ≤ n/2. ■

    Uniqueness: A tree has at most two centroids. If there are two, they must be adjacent. (If a tree has an even number of nodes, both endpoints of a "balanced edge" may qualify.)

    Implementation

    int sz[MAXN];
bool removed[MAXN];
vector<int> adj[MAXN];

void calc_sz(int v, int p) {
    sz[v] = 1;
    for (int u : adj[v]) {
        if (u != p && !removed[u]) {
            calc_sz(u, v);
            sz[v] += sz[u];
        }
    }
}

int find_centroid(int v, int p, int tree_sz) {
    for (int u : adj[v]) {
        if (u != p && !removed[u] && sz[u] > tree_sz / 2)
            return find_centroid(u, v, tree_sz);
    }
    return v;
}

    Time: O(n) for a single call—we visit each node at most twice (once for sizes, once for the centroid walk).

    Building the Centroid Tree

    The full centroid decomposition recursively finds and removes centroids to build a new tree called the centroid tree (or centroid decomposition tree):

    1. Find the centroid c of the current tree.
    2. Mark c as removed.
    3. For each connected component formed by removing c, recursively find the centroid c' of that component.
    4. Make c' a child of c in the centroid tree.

    Properties of the centroid tree

    • Depth O(log n): Each level of the centroid tree removes a centroid whose subtree had at most half the nodes of its parent's subtree. So the depth is at most log₂ n.
    • Same node set: The centroid tree has the same nodes as the original tree, but different edges.
    • Path through ancestor: For any two nodes u and v in the original tree, their path passes through their LCA in the centroid tree. This is the key property that makes queries efficient.

    Implementation

    int centroid_parent[MAXN];

void build_centroid_tree(int v, int p) {
    calc_sz(v, -1);
    int c = find_centroid(v, -1, sz[v]);
    centroid_parent[c] = p;
    removed[c] = true;

    for (int u : adj[c]) {
        if (!removed[u]) {
            build_centroid_tree(u, c);
        }
    }
}

    Call build_centroid_tree(0, -1) where 0 is any starting node and -1 indicates the root of the centroid tree has no parent.

    Why O(log n) depth?

    Consider a node v at depth d in the centroid tree. At each ancestor of v, the component size at least halves. Starting from n, after d halvings, the component containing v has at most n / 2d nodes. Since a component must have at least 1 node, we get n / 2d ≥ 1, which means d ≤ log₂ n.

    Counting Paths Through the Centroid

    The centroid decomposition framework for path problems follows a consistent pattern:

    function solve(v): // 1. Find centroid of current component c = find_centroid(v) // 2. Count/process paths passing through c for each subtree T of c: gather distances from c to all nodes in T combine with previously gathered distances // 3. Remove c and recurse on each component removed[c] = true for each neighbor u of c: if not removed[u]: solve(u)

    The crucial step is Step 2: processing paths that pass through the centroid. The idea is:

    • Any path through c consists of a segment from some node u to c, and a segment from c to some node v, where u and v are in different subtrees of c.
    • So we process subtrees one by one. For each subtree, we gather information (e.g., distances), combine it with what we've accumulated from previous subtrees, then add the current subtree's data to the accumulator.

    This "process subtrees one at a time, merge with running total" avoids double-counting paths within a single subtree.

    Example: counting pairs at distance exactly k

    int ans = 0;

void count_paths(int v, int tree_sz) {
    calc_sz(v, -1);
    int c = find_centroid(v, -1, sz[v]);
    removed[c] = true;

    // cnt[d] = number of nodes at distance d from c (from previous subtrees)
    map<int, int> cnt;
    cnt[0] = 1;  // centroid itself at distance 0

    for (int u : adj[c]) {
        if (removed[u]) continue;

        // Gather distances in this subtree
        vector<int> dists;
        function<void(int, int, int)> gather = [&](int v, int p, int d) {
            dists.push_back(d);
            for (int w : adj[v])
                if (w != p && !removed[w])
                    gather(w, v, d + 1);
        };
        gather(u, c, 1);

        // Count valid pairs: pair current subtree with previous subtrees
        for (int d : dists)
            if (cnt.count(k - d))
                ans += cnt[k - d];

        // Add current subtree's distances to accumulator
        for (int d : dists)
            cnt[d]++;
    }

    // Recurse on each component
    for (int u : adj[c])
        if (!removed[u])
            count_paths(u, sz[u]);
}
    Why map? Using a map<int,int> is convenient but O(n log n) per level. For maximum performance, use an array cnt[] of size up to n and reset it after each centroid (avoid memset—keep track of which indices were modified).

    Distance Queries

    Centroid decomposition provides an elegant solution for distance-related queries on trees. The key observation:

    For any two nodes u, v: their path in the original tree passes through their LCA in the centroid tree. So dist(u, v) = dist(u, lca) + dist(v, lca) where lca is their LCA in the centroid tree.

    Closest marked node query

    A classic application: given a tree where nodes can be "marked" one at a time, answer queries of the form "what is the distance to the closest marked node from node v?"

    Approach:

    1. Build the centroid decomposition. For each node, precompute its distance to every ancestor in the centroid tree (there are at most O(log n) ancestors).
    2. For each node v in the centroid tree, maintain best[v] = minimum distance from v to any marked node in v's "centroid subtree."
    3. Mark update: When marking node u, walk up the centroid tree from u to the root. At each ancestor a, update best[a] = min(best[a], dist(u, a)).
    4. Query: To find the closest marked node to v, walk up the centroid tree from v. At each ancestor a, the candidate answer is best[a] + dist(v, a). Return the minimum.

    Both operations are O(log n) since the centroid tree has depth O(log n).

    Implementation

    const int INF = 1e9;
int best[MAXN];
int ct_par[MAXN];        // centroid tree parent
int dist_to_ct[MAXN][20]; // dist_to_ct[v][j] = dist from v to j-th centroid ancestor
int ct_depth[MAXN];

void init() {
    fill(best, best + MAXN, INF);
}

void mark(int u) {
    int v = u, j = 0;
    while (v != -1) {
        best[v] = min(best[v], dist_to_ct[u][j]);
        v = ct_par[v];
        j++;
    }
}

int query(int u) {
    int res = INF;
    int v = u, j = 0;
    while (v != -1) {
        res = min(res, best[v] + dist_to_ct[u][j]);
        v = ct_par[v];
        j++;
    }
    return res;
}

    Precomputing distances: During the centroid decomposition, when we process centroid c, we BFS/DFS from c to all nodes in its component. For each such node v, we record dist_to_ct[v][depth of c in centroid tree] = dist(v, c).

    Classic Problem: Paths of Length K

    Problem: Given a tree with n nodes and a value k, count the number of pairs (u, v) such that the path from u to v has exactly k edges.

    This is the quintessential centroid decomposition problem. Let's walk through the full approach:

    Step 1: Find centroid

    For the current component, find the centroid c.

    Step 2: Count paths through c

    A path of length k through c consists of:

    • A path of length d1 from node u to c, and
    • A path of length d2 from c to node v, where d1 + d2 = k.
    • u and v must be in different subtrees of c (or one of them is c itself).

    We process subtrees of c one at a time:

    1. Initialize a frequency array cnt[] where cnt[d] = number of nodes at distance d from c seen so far. Set cnt[0] = 1 (for c itself).
    2. For each subtree, DFS to collect all distances. For each distance d, add cnt[k - d] to the answer (if k - d ≥ 0).
    3. After counting, add the current subtree's distances to cnt[].

    Step 3: Recurse

    Remove c and recurse on each remaining component.

    Optimized implementation

    int cnt[MAXN]; // cnt[d] = # nodes at distance d from centroid (accumulated)
int ans = 0;
int k;
vector<int> modified; // track modified indices for cleanup

void gather(int v, int p, int d, vector<int>& dists) {
    if (d > k) return; // prune: no need to go deeper than k
    dists.push_back(d);
    for (int u : adj[v])
        if (u != p && !removed[u])
            gather(u, v, d + 1, dists);
}

void solve(int v) {
    calc_sz(v, -1);
    int c = find_centroid(v, -1, sz[v]);
    removed[c] = true;

    modified.clear();
    cnt[0] = 1;
    modified.push_back(0);

    for (int u : adj[c]) {
        if (removed[u]) continue;

        vector<int> dists;
        gather(u, c, 1, dists);

        // Count: pair with previously seen distances
        for (int d : dists)
            if (k - d >= 0)
                ans += cnt[k - d];

        // Accumulate
        for (int d : dists) {
            if (cnt[d] == 0) modified.push_back(d);
            cnt[d]++;
        }
    }

    // Clean up cnt[] (only modified entries)
    for (int d : modified)
        cnt[d] = 0;

    // Recurse on each component
    for (int u : adj[c])
        if (!removed[u])
            solve(u);
}
    Performance tip: Pruning if (d > k) return; in the gather function is critical. Without it, you might explore unnecessarily deep subtrees. Also, cleaning up cnt[] by tracking modified indices avoids an O(n) memset at each level.

    Complete C++ Implementation

    Here's a full, contest-ready centroid decomposition template. It builds the centroid tree and precomputes distances for efficient path queries.

    #include <bits/stdc++.h>
using namespace std;

const int MAXN = 200005;
const int LOG = 18;
const int INF = 1e9;

vector<int> adj[MAXN];
int sz[MAXN];
bool removed[MAXN];
int ct_par[MAXN];       // parent in centroid tree
int ct_depth[MAXN];     // depth in centroid tree
vector<pair<int,int>> ct_anc[MAXN]; // (ancestor in CT, dist to it)

void calc_sz(int v, int p) {
    sz[v] = 1;
    for (int u : adj[v])
        if (u != p && !removed[u]) {
            calc_sz(u, v);
            sz[v] += sz[u];
        }
}

int find_centroid(int v, int p, int tree_sz) {
    for (int u : adj[v])
        if (u != p && !removed[u] && sz[u] > tree_sz / 2)
            return find_centroid(u, v, tree_sz);
    return v;
}

// BFS from centroid c to all nodes in its component
void bfs_distances(int c) {
    queue<pair<int,int>> q;
    q.push({c, 0});
    vector<bool> vis(MAXN, false);
    vis[c] = true;
    while (!q.empty()) {
        auto [v, d] = q.front(); q.pop();
        ct_anc[v].push_back({c, d});
        for (int u : adj[v]) {
            if (!removed[u] && !vis[u]) {
                vis[u] = true;
                q.push({u, d + 1});
            }
        }
    }
}

void build(int v, int p, int depth) {
    calc_sz(v, -1);
    int c = find_centroid(v, -1, sz[v]);
    ct_par[c] = p;
    ct_depth[c] = depth;
    removed[c] = true;

    bfs_distances(c);

    for (int u : adj[c])
        if (!removed[u])
            build(u, c, depth + 1);
}

// === Application: closest marked node ===
int best[MAXN];

void init(int n) {
    fill(best, best + n, INF);
    memset(removed, 0, sizeof(removed));
    build(0, -1, 0);
}

void mark(int u) {
    for (auto [anc, d] : ct_anc[u])
        best[anc] = min(best[anc], d);
}

int query(int u) {
    int res = INF;
    for (auto [anc, d] : ct_anc[u])
        res = min(res, best[anc] + d);
    return res == INF ? -1 : res;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;
    for (int i = 0; i < n - 1; i++) {
        int u, v;
        cin >> u >> v;
        u--; v--;
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    init(n);

    while (q--) {
        int type, v;
        cin >> type >> v;
        v--;
        if (type == 1) mark(v);
        else cout << query(v) << "\n";
    }
    return 0;
}
    Template flexibility: This template stores ct_anc[v] as a vector of (centroid ancestor, distance) pairs for each node. To adapt for different problems, change what you store in best[] and how you combine in query().

    Complexity Analysis

    Why O(n log n) total work?

    The key claim is that the total work across all levels of the centroid decomposition is O(n log n). Here's why:

    • Consider a single node v. It participates in the "gather distances" step for every centroid that is an ancestor of v in the centroid tree.
    • The centroid tree has depth O(log n), so v participates in at most O(log n) centroid computations.
    • In each computation, v contributes O(1) work (it's visited once by DFS/BFS).
    • Total: each of the n nodes contributes O(log n) work = O(n log n).

    Another way to see it: at each "level" of the recursion, all components together cover all n nodes (minus removed centroids). There are O(log n) levels. So the total work is O(n) per level × O(log n) levels = O(n log n).

    OperationTimeSpace
    Build centroid treeO(n log n)O(n log n) for ancestor distances
    Mark a nodeO(log n)O(1)
    Query closest markedO(log n)O(1)
    Count paths of length kO(n log n)O(n)
    Find centroid (single)O(n)O(n)

    Space complexity

    The main space cost is storing distances from each node to its centroid ancestors: n nodes × O(log n) ancestors = O(n log n). In practice with n = 2 × 105, this is about 2 × 105 × 18 ≈ 3.6 × 106 entries—very manageable.
