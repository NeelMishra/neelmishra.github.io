    bitset Performance & When to Use It

    June 2026 · 22 min read

    The first two posts covered what std::bitset does. This one covers why it is fast, what it costs in memory, the one constraint that rules it out, and the situations where it is the right tool. The recurring theme is that a bitset processes 64 bits per machine instruction, which turns many linear scans into operations that are effectively dozens of times faster without changing the asymptotic class.

    Why bitset Is Fast

    A bitset packs its N bits into machine words of width w (typically $w = 64$). Every whole-bitset operation, such as &, |, ^, a shift, or count(), processes one full word at a time, so it runs in

    $$O\!\left(\frac{N}{w}\right)$$

    word steps rather than $O(N)$ bit steps. The work is still linear in N, but the constant factor shrinks by a factor of w. For $N = 64$ a whole-bitset AND is a single instruction; for $N = 100{,}000$ it is about 1,563 word operations instead of 100,000 bit tests. Modern CPUs also expose a popcount instruction, which count() uses directly.

    Same big-O, smaller constant. A bitset does not beat an $O(N)$ algorithm asymptotically; it divides the constant by the word width. That 64x factor is often the difference between a solution that fits in the time limit and one that does not.

    Memory Footprint

    A bitset uses one bit per element, rounded up to a whole number of words. That is roughly N / 8 bytes, eight times denser than an array of bool (where each bool takes a full byte) and far denser than a node-based set of integers.

    ContainerBytes for N booleansSize known at
    std::bitset<N>~N / 8Compile time
    std::vector<bool>~N / 8 (also bit-packed)Run time
    bool[N] / std::array<bool, N>NCompile time
    std::set<int>~32 * (number of present elements)Run time

    For a few thousand to a few million flags, the bitset version often fits entirely in cache where the byte-per-bool version would not, which compounds the speed advantage beyond the raw instruction count.

    The Compile-Time Size Constraint

    The single rule that decides whether std::bitset is usable: N must be a compile-time constant. A length read from input cannot size a bitset.

    int n;
std::cin >> n;
// std::bitset<n> bad;   // does NOT compile: n is not constexpr

std::vector<bool> ok(n);            // runtime size, bit-packed
// boost::dynamic_bitset<> faster(n);  // runtime size, full bitset interface

    When the maximum size is bounded and known, the usual trick is to declare a bitset at that maximum and ignore the unused tail. When the size is genuinely unbounded at compile time, reach for std::vector<bool> for storage, or boost::dynamic_bitset when the full word-parallel operator set is needed at a runtime width.

    Use Case: Sieve of Eratosthenes

    A prime sieve is a natural fit: one bit per number, flipped as composites are found. The bitset version is compact and cache-friendly.

    #include <bitset>

const int LIMIT = 1'000'000;
std::bitset<LIMIT + 1> composite;   // 0 = prime candidate, 1 = composite

void sieve() {
    composite.set(0);
    composite.set(1);
    for (int p = 2; (long long)p * p <= LIMIT; ++p) {
        if (!composite[p]) {
            for (int m = p * p; m <= LIMIT; m += p) {
                composite.set(m);
            }
        }
    }
}
// composite[k] == false  means k is prime

    One million bits is about 125 KB, small enough to stay resident while the inner loop streams through it.

    Use Case: Subset-Sum and Reachability DP

    When a DP tracks which sums (or which states) are reachable, a bitset holds the whole reachable set and a single shift advances it by one item. Adding an item of weight w is one instruction over the entire set.

    // Which subset sums are achievable from the given weights?
std::bitset<100001> reachable;
reachable.set(0);                      // sum 0 is always reachable
for (int w : weights) {
    reachable |= (reachable << w);      // add w to every reachable sum
}
// reachable[s] == true  means sum s can be formed

    This collapses the classic $O(n \cdot S)$ subset-sum table into $O(n \cdot S / w)$ word operations. The same shift-and-or pattern drives reachability and several string-matching DPs; the dedicated Bitset Optimization post develops these in depth.

    Use Case: Fast Set Membership

    For a dense universe of small integer keys, a bitset is a direct presence array: set(x) to add, test(x) to query, both in constant time and one bit of storage each. It beats a hash set on both speed and memory whenever the key range is bounded and reasonably dense.

    Pitfalls Checklist

    • Size must be constexpr. A runtime length needs std::vector<bool> or boost::dynamic_bitset.
    • Default is all zeros. A fresh bitset is empty, not uninitialized; rely on that rather than clearing it manually.
    • Conversions can throw. to_ulong() and to_ullong() raise std::overflow_error when the value will not fit.
    • Indexing is least-significant-first while printing is most-significant-first; see Part 2.
    • operator[] is unchecked. Use test(), set(pos), or reset(pos) when a position might be out of range.

    When to Reach for bitset

    • The number of bits is known at compile time, or has a fixed upper bound.
    • Whole-set operations (AND, OR, XOR, shifts, popcount) run in an inner loop.
    • The universe of keys is dense and a one-bit-per-element layout is acceptable.
    • A linear scan needs a constant-factor speedup of roughly the word width.

    When the width is only known at runtime, or the data is sparse enough that a hash set wins, prefer those instead.

    Summary

    • Bitset operations run in $O(N/w)$ word steps, dividing the constant factor by the machine word width.
    • Storage is about N / 8 bytes, dense enough to stay in cache where a bool array would not.
    • N must be a compile-time constant; otherwise use std::vector<bool> or boost::dynamic_bitset.
    • Sieves, subset-sum and reachability DP, and dense membership tests are the canonical wins.
