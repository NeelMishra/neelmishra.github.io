    Q6: Detecting Multiplication Overflow

    June 2026 · 13 min read

    Addition overflow grows the value by at most one bit, so a wider type buys you a lot of room. Multiplication is meaner: two 32-bit operands can need up to 64 bits, and two 64-bit operands up to 128. This final problem is the canonical defensive pattern — a running product you must keep from overflowing, detected before each multiply.

    The Problem

    Given an array of positive integers and a cap LIMIT (for instance INT_MAX), return the product of all elements, but saturate: if at any point the running product would exceed LIMIT, return LIMIT. With LIMIT = INT_MAX this is exactly "compute the product, or report 32-bit overflow." The array can be long and the values large, so a single naive multiply can blow past the type.

    The Tempting (Broken) Approach

    // multiply first, then "check"
int prod = 1;
for (int x : a) prod *= x;          // overflows mid-loop (signed UB)
if (prod < 0) return INT_MAX;       // sign flip is NOT a reliable signal
    Why this fails: signed overflow is undefined, so the loop may be miscompiled outright. Even ignoring that, a "negative product means overflow" heuristic is wrong — an overflowed product can land back on a positive value (overflow by a full $2^{32}$), and a genuinely negative product is legitimate when operands are negative. The sign of a corrupted value tells you nothing.

    Method 1: The Division Pre-Check

    For positive prod and x, the product exceeds LIMIT exactly when

    $$prod \cdot x > \text{LIMIT} \iff prod > \left\lfloor \frac{\text{LIMIT}}{x} \right\rfloor.$$

    The equivalence is exact over the positive integers because prod is an integer: $prod \le \lfloor \text{LIMIT}/x \rfloor \iff prod \le \text{LIMIT}/x \iff prod \cdot x \le \text{LIMIT}$. The right side LIMIT / x is always in range (it only shrinks LIMIT), so the check itself never overflows — the same "divide the bound" move from Q2.

    #include <climits>
#include <vector>
using namespace std;

int cappedProduct(const vector<int>& a, int LIMIT) {
    int prod = 1;
    for (int x : a) {
        if (x == 0) return 0;                 // a zero zeroes the whole product
        if (prod > LIMIT / x) return LIMIT;   // prod*x would exceed LIMIT -> saturate
        prod *= x;                            // proven safe: prod*x <= LIMIT
    }
    return prod;
}

    The x == 0 guard does double duty: it gives the mathematically correct zero product and protects the division LIMIT / x from a divide-by-zero. With LIMIT = INT_MAX, this detects 32-bit multiplication overflow using nothing wider than int.

    Method 2: Compiler Builtins

    When operands may be negative, the division rule needs sign gymnastics; the checked-arithmetic builtins handle every sign correctly and compile to a single multiply plus an overflow-flag branch.

    int prod = 1;
for (int x : a) {
    int next;
    if (__builtin_mul_overflow(prod, x, &next))  // true if it overflowed int
        return INT_MAX;
    prod = next;                                  // the safe product
}
// GCC/Clang; MSVC offers similar via <intsafe.h> / SafeInt.

    Method 3: Compute Wider, Then Range-Check

    The most readable option when a wider type exists: multiply in long long (for 32-bit operands) and compare against the bound. Remember to widen an operand, not the result.

    long long prod = 1;
for (int x : a) {
    prod *= x;                       // 64-bit room: 32x32 cannot overflow long long
    if (prod > LIMIT) return LIMIT;  // single, exact range check
}
return (int)prod;

    For 64-bit operands the same idea steps up to __int128:

    __int128 wide = (__int128)x * y;
if (wide > LLONG_MAX || wide < LLONG_MIN) { /* would not fit in long long */ }

    Dry Run: a = [100000, 100000, 7], LIMIT = INT_MAX

    INT_MAX = 2147483647. The true product 100000 * 100000 = 10^{10} already exceeds it.

    prod = 1
x = 100000:  LIMIT/x = 21474;  prod(1)      > 21474?  no   -> prod = 100000
x = 100000:  LIMIT/x = 21474;  prod(100000) > 21474?  YES  -> return INT_MAX

    The overflow is caught at the second multiply — prod never actually holds the out-of-range value, and the trailing 7 is never examined.

    Complexity

    $O(n)$ over the array with $O(1)$ extra space for all three methods. The pre-check adds one division per element; the builtin adds one predicated branch; widening adds one comparison.

    Edge Cases

    CaseHandling
    A zero elementReturn 0 immediately — also avoids LIMIT / 0.
    x == 1 repeatedlyLIMIT / 1 == LIMIT; prod > LIMIT is false, product unchanged. Correct.
    Negative operandsThe division rule is sign-sensitive; prefer __builtin_mul_overflow or __int128.
    Single huge element > LIMITFirst iteration: prod(1) > LIMIT / x may be false, but the element itself can exceed LIMIT; add an explicit if (x > LIMIT) return LIMIT; if elements may individually exceed the cap.
    Empty arrayEmpty product is 1 — the loop returns the initial prod.
    Series takeaway: every problem here had the same cure — reason about the bound before the dangerous operation. Rearrange so the test stays in range (prod > LIMIT / x, lo + (hi - lo) / 2), let the hardware tell you (__builtin_*_overflow), or compute where overflow cannot happen (long long, __int128, the negative domain). Never inspect a value that has already overflowed.
