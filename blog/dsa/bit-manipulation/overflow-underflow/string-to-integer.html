    Q3: Clamping atoi (String to Integer)

    June 2026 · 12 min read

    Parse a leading integer out of a string and return it as a 32-bit signed int. If the parsed value spills past the range, clamp it to INT_MIN or INT_MAX instead of returning 0. The parsing rules are fiddly but mechanical; the interesting part is that the input string can encode a number with hundreds of digits, so you must clamp the accumulator mid-parse — and the clamp has a sharp corner exactly at INT_MIN.

    The Problem

    LeetCode 8. Read and discard leading spaces; read an optional +/-; read consecutive digits and stop at the first non-digit (or end). Convert those digits to an integer. If it exceeds the 32-bit signed range, clamp to [-2^{31},\, 2^{31}-1]. If no digits are present, return 0.

    The Overflow Trap

    The string "-91283472332" denotes a value far below INT_MIN. A natural instinct is to accumulate the magnitude as a positive int and negate at the end — but that breaks precisely at the boundary:

    // magnitude-then-negate: subtly broken
int mag = 0;
for (digit d : digits) mag = mag * 10 + d;   // overflows for INT_MIN's magnitude
return sign * mag;
    The corner case: the magnitude of INT_MIN is $2^{31} = 2147483648$, which is larger than INT_MAX = 2147483647. So even a correct-looking input like "-2147483648" overflows a positive int accumulator before you ever apply the sign. Magnitude-first cannot represent its own valid minimum.

    Build the Signed Value Directly

    The robust fix — the same shape as Q2 — is to fold the sign into each digit and accumulate the signed result, checking both bounds before each step. Now there is no asymmetric magnitude to overflow: a negative running total is compared against INT_MIN directly.

    With $M = \text{INT\_MAX}/10 = 214748364$ (last digit 7) and $m = \text{INT\_MIN}/10 = -214748364$ (last digit -8):

    • Positive side: if res > M, or res == M and the next signed digit > 7 → return INT_MAX.
    • Negative side: if res < m, or res == m and the next signed digit < -8 → return INT_MIN.

    Because we clamp the moment the bound is crossed, every subsequent digit can be ignored — the answer is already pinned.

    Reference Implementation

    #include <climits>
#include <string>
#include <cctype>
using namespace std;

int myAtoi(string s) {
    int i = 0, n = (int)s.size();

    while (i < n && s[i] == ' ') i++;          // 1) skip spaces

    int sign = 1;                              // 2) optional sign
    if (i < n && (s[i] == '+' || s[i] == '-')) {
        if (s[i] == '-') sign = -1;
        i++;
    }

    int res = 0;                               // 3) accumulate SIGNED value
    while (i < n && isdigit((unsigned char)s[i])) {
        int pop = sign * (s[i] - '0');         // signed digit: -9..9
        i++;
        if (res > INT_MAX / 10 || (res == INT_MAX / 10 && pop > 7))
            return INT_MAX;                    // clamp high
        if (res < INT_MIN / 10 || (res == INT_MIN / 10 && pop < -8))
            return INT_MIN;                    // clamp low
        res = res * 10 + pop;                  // safe now
    }
    return res;
}

    Casting to unsigned char before isdigit avoids UB when a byte is negative — a small but real correctness detail in C++ character classification.

    Dry Run: "-91283472332"

    Sign is -1; the true value is below INT_MIN, so we expect INT_MIN = -2147483648. The digits are 9 1 2 8 3 4 7 2 3 3 2.

    m = INT_MIN/10 = -214748364

res (before)     pop    action
          0       -9    res = -9
         -9       -1    res = -91
        -91       -2    res = -912
       -912       -8    res = -9128
      -9128       -3    res = -91283
     -91283       -4    res = -912834
    -912834       -7    res = -9128347
   -9128347       -2    res = -91283472
  -91283472       -3    res = -912834723
 -912834723       -3    -912834723 < m(-214748364)  ->  return INT_MIN

    The clamp triggers the instant res < m: once the running total reaches -912834723, the next check fires and returns INT_MIN. Every intermediate stayed inside int, and the trailing digits after the clamp point never matter.

    Complexity

    $O(n)$ over the input length (we stop at the first non-digit or at clamp), $O(1)$ extra space, and never a 64-bit type.

    Edge Cases

    InputOutputReason
    " -42"-42Leading spaces skipped, sign read.
    "4193 with words"4193Stop at first non-digit.
    "words and 987"0First non-space is a letter; no number.
    "+-12"0Two signs; the second is a non-digit, parsing ends.
    "2147483648"2147483647One past INT_MAX → clamp high.
    "-2147483648"-2147483648Exactly INT_MIN; signed accumulation represents it (magnitude-first could not).
    "-2147483649"-2147483648One below INT_MIN → clamp low.
    Takeaway: when a result can be either sign and you need the full range, never accumulate a magnitude and negate — the negative range is one unit wider, so the magnitude of INT_MIN does not fit the positive type. Accumulate the signed value and clamp against both bounds.
