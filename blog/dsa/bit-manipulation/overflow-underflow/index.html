    Integer Overflow & Underflow

    June 2026 · 18 min read

    Every integer in a program lives in a fixed number of bits. The moment an arithmetic result needs a bit that does not exist, the value silently changes meaning. That single fact is behind a huge share of "impossible" bugs: a binary search that crashes only on large inputs, a sum that turns negative, a loop that never ends. This page builds the mental model from the bits up, then the rest of the series is six problems where overflow or underflow is the puzzle.

    What Overflow and Underflow Mean

    A signed 32-bit int holds values in $[-2^{31},\, 2^{31}-1] = [-2147483648,\, 2147483647]$. An unsigned int holds $[0,\, 2^{32}-1]$. Overflow is when a result is too large for the type; underflow (for integers) is when a result drops below the minimum — most commonly an unsigned value going below 0 and wrapping to a huge number.

    Naming note: in floating point, "underflow" means a number too close to zero to represent. For integers the word is used loosely for "wrapped below the minimum." This series is about integers, so underflow here means the low-end wrap.

    Two's Complement and the Asymmetric Range

    Signed integers use two's complement: the top bit carries weight $-2^{W-1}$ and the rest carry positive powers. For width $W$,

    $$\text{value}(b) = -b_{W-1}\,2^{W-1} + \sum_{i=0}^{W-2} b_i\,2^{i}.$$

    This makes the range asymmetric: there is one more negative value than positive, because zero takes a slot on the non-negative side. So $|{\,\text{INT\_MIN}\,}| = 2^{31}$ but the largest positive is only $2^{31}-1$.

    INT_MAX =  2147483647  = 0x7fffffff
INT_MIN = -2147483648  = 0x80000000
-INT_MIN does NOT fit in int  (it would be 2147483648)

    That last line is the root of an entire class of bugs: negating, or taking the absolute value of, INT_MIN overflows. We exploit and defend against this in Q4 (Divide Two Integers).

    Signed UB vs Unsigned Modular Wraparound

    The two integer families behave completely differently when they exceed their range, and conflating them is the most expensive mistake in this topic.

    Signed (int, long long)Unsigned (unsigned, size_t)
    On overflowUndefined behavior in C++Well-defined: wraps modulo $2^{W}$
    Compiler mayAssume it never happens and optimize it awayNothing surprising; result is exact mod $2^W$
    Detect after?No — the overflowed value is not trustworthyYes — e.g. a + b < a means it wrapped
     
    unsigned u = 0xffffffffu;
u + 1;          // == 0, defined, wraps mod 2^32

int s = INT_MAX;
s + 1;          // undefined behavior, NOT "INT_MIN"

    Because signed overflow is UB, you must detect it before performing the operation. Because unsigned wrap is defined, you may detect it after — but the wrap itself can still be a logic bug, as Q5 shows.

    Detecting Overflow Safely

    There are three reliable strategies. Pick by context.

    1. Algebraic pre-checks

    Rearrange the dangerous operation so the check itself never overflows.

    // Addition: would a + b exceed INT_MAX?  (a, b >= 0)
bool addOverflows(int a, int b) { return a > INT_MAX - b; }

// Multiplication: would a * b exceed INT_MAX?  (a, b > 0)
bool mulOverflows(int a, int b) { return a > INT_MAX / b; }

// Safe midpoint (never overflows for lo <= hi)
int mid = lo + (hi - lo) / 2;

    The key idea: INT_MAX - b and INT_MAX / b are both in range, so the comparison is safe even though a + b or a * b would not be.

    2. Compiler builtins

    GCC and Clang provide overflow-checked arithmetic that returns a flag and writes the wrapped result. These are exact and branch-free on most CPUs.

    int r;
if (__builtin_add_overflow(a, b, &r)) { /* overflowed */ }
if (__builtin_mul_overflow(a, b, &r)) { /* overflowed */ }
// also __builtin_sub_overflow, and ..._overflow_p for the flag only
// C++20: <numeric> has std::add_sat / std::mul_sat (saturating)

    3. Compute in a wider type

    Do the math where it cannot overflow, then range-check before narrowing.

    long long prod = (long long)a * b;          // 32-bit * 32-bit fits in 64
if (prod > INT_MAX || prod < INT_MIN) { /* would not fit in int */ }

// For 64-bit operands, step up to __int128 (GCC/Clang):
__int128 wide = (__int128)x * y;
    Watch the cast position. (long long)(a * b) is useless — the multiply already happened in int and may have overflown. You must widen an operand: (long long)a * b.
