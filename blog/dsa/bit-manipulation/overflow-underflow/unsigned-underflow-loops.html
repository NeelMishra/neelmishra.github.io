    Q5: The Unsigned Underflow Trap

    June 2026 · 12 min read

    The previous problems were about overflow at the top of the range. This one is its mirror at the bottom: subtracting past zero in an unsigned type. Because std::size(), array indices, and size_t are unsigned, this is the most common overflow bug you will actually ship — and it never throws; it just produces a number near four billion.

    The Mental Model

    Unsigned arithmetic is exact modulo $2^{W}$ and has no negative values. Subtracting below zero wraps to the top:

    $$0 - 1 \equiv 2^{W} - 1 \pmod{2^{W}}, \qquad \text{so } \texttt{0u - 1} = \texttt{4294967295}.$$

    Two consequences poison loops and size math:

    • An unsigned value is never negative, so i >= 0 is a tautology — always true.
    • a - b with a < b is a huge number, not a negative one.

    Bug 1: The Descending Loop That Never Ends

    // INFINITE LOOP
for (size_t i = n - 1; i >= 0; --i) {
    use(v[i]);
}

    When i finally reaches 0, the condition i >= 0 is still true (unsigned), so --i runs once more, wrapping i to SIZE_MAX. The next v[i] reads astronomically out of bounds. And if n == 0 to begin with, n - 1 is already SIZE_MAX on the first iteration.

    The Fix: the i-- > 0 idiom

    The clean reverse-iteration idiom tests then decrements, so the loop stops at the right moment without ever going below zero:

    for (size_t i = n; i-- > 0; ) {   // i takes n-1, n-2, ..., 1, 0 then stops
    use(v[i]);
}

    On the iteration where i == 0: the condition reads the current value 0 > 0 → false, so the body does not run, but i was already post-decremented to SIZE_MAX — harmlessly, because we never use it. The alternative is simply to use a signed counter: for (int i = (int)n - 1; i >= 0; --i).

    Bug 2: Size Arithmetic That Explodes

    Consider building the windows of a fixed size k over an array of length n: there are n - k + 1 of them. Reserve that many slots:

    vector<int> windows;
windows.reserve(nums.size() - k + 1);   // nums.size() is size_t (unsigned)
    What happens when k > n? nums.size() - k underflows to roughly $2^{64}$. The reserve then tries to allocate billions of elements and throws std::bad_alloc (or the OS kills the process). The math "number of windows = n - k + 1" is correct over the integers but catastrophic over size_t when the count should have been zero.

    The Fix: guard before subtracting

    if (k == 0 || k > nums.size()) return {};       // no valid windows
size_t count = nums.size() - k + 1;             // now guaranteed >= 1
windows.reserve(count);

    The rule: before any unsigned subtraction, prove the minuend is at least the subtrahend, or do the comparison in a signed/wider type: (long long)nums.size() - k.

    Bug 3: The Mixed Signed/Unsigned Comparison

    When you compare a signed and an unsigned value of the same width, the signed one is converted to unsigned — turning negatives into giants.

    int n = -1;
if (n < nums.size()) {     // nums.size() is unsigned!
    // n is converted to a huge unsigned value, so this is FALSE,
    // even though -1 < (any size) is true over the integers.
}

    This silently breaks bounds checks like if (idx < v.size()) when idx can legitimately be -1 (a common "not found" sentinel). Compile with -Wsign-compare (in -Wall) to catch these.

    Dry Run: the wrap in slow motion

    Take n = 2, the broken loop for (size_t i = n - 1; i >= 0; --i):

    i = 1   1 >= 0 true   use(v[1])   --i -> 0
i = 0   0 >= 0 true   use(v[0])   --i -> 18446744073709551615   (0 - 1 wraps)
i = SIZE_MAX   true   use(v[SIZE_MAX])   <-- out-of-bounds read, UB

    And the size bug with n = 3, k = 5: nums.size() - k = 3 - 5 = 18446744073709551614, so reserve requests ~18 quintillion elements.

    Defensive Checklist

    SmellSafe replacement
    for (size_t i = n-1; i >= 0; --i)for (size_t i = n; i-- > 0; ) or signed counter
    a.size() - bguard b <= a.size() first

    Edge Cases

    • Empty container: v.size() - 1 is SIZE_MAX, the single most common source of this bug; always handle n == 0 first.
    • k == n: exactly one window; n - k + 1 == 1 — correct, so the guard must be k > n, not k >= n.
    • Difference of unsigned distances: abs(a - b) on size_t is meaningless after a wrap; compute a > b ? a - b : b - a.
    Takeaway: unsigned underflow is well-defined and therefore silent — no UB, no crash at the subtraction, just a value near $2^{W}$ that detonates later as a bad index or a giant allocation. Guard every unsigned subtraction, and prefer signed counters for anything that counts down.
