    Q4: Divide Two Integers Without Multiply/Divide

    June 2026 · 14 min read

    Divide two 32-bit signed integers without using *, /, or %, truncating toward zero. This problem is a double feature: the algorithm is binary long division built from shifts and subtractions (pure bit manipulation), and the only thing that can break it is a single overflow case — the most-negative integer.

    The Problem

    LeetCode 29. Given dividend and divisor, return dividend / divisor with the fractional part dropped (truncate toward zero, so 7 / -3 == -2). The result must fit in a 32-bit signed int; if it would not, return INT_MAX.

    The One Overflow Case

    The quotient range almost matches the input range — with exactly one exception. The largest-magnitude quotient is

    $$\frac{\text{INT\_MIN}}{-1} = \frac{-2^{31}}{-1} = 2^{31} = 2147483648 > \text{INT\_MAX}.$$

    Every other dividend/divisor pair yields a quotient that fits. So the entire overflow handling collapses to one guarded line:

    if (dividend == INT_MIN && divisor == -1) return INT_MAX;

    Why You Cannot Just Take Absolute Values

    The cleanest mental model is "divide the magnitudes, then fix the sign." But computing abs(dividend) in int is exactly the trap from the overview: abs(INT_MIN) is undefined, because $|{\text{INT\_MIN}}| = 2^{31}$ does not fit in a signed 32-bit int.

    Two ways out. Either (a) widen the magnitudes to a 64-bit type where $2^{31}$ is trivially representable, or (b) never go positive at all — do the whole division in the negative domain, which is one unit wider and can hold every value's image. We use (a) for the main solution and sketch (b) after.

    Binary Long Division by Shifting

    To divide without /, subtract the largest shifted copy of the divisor that still fits, highest power first. b << k is $b \cdot 2^k$, so the quotient bit at position k is set exactly when $a \ge b \cdot 2^k$. Checking it as (a >> k) >= b keeps both sides small and avoids overflowing b << k.

    #include <climits>
#include <cstdlib>

int divide(int dividend, int divisor) {
    if (dividend == INT_MIN && divisor == -1) return INT_MAX;   // only overflow

    long long a = llabs((long long)dividend);   // widen BEFORE abs: safe
    long long b = llabs((long long)divisor);
    bool neg = (dividend < 0) != (divisor < 0); // XOR of signs

    long long q = 0;
    for (int k = 31; k >= 0; --k) {
        if ((a >> k) >= b) {     // does b * 2^k fit into the remaining a?
            a -= (b << k);       // subtract it off
            q += (1LL << k);     // record that quotient bit
        }
    }
    return (int)(neg ? -q : q);
}

    The cast to long long happens before llabs, so the absolute value is taken on a 64-bit operand and INT_MIN is no longer special. After the special-cased pair is removed, q is at most $2^{31}$ only when neg is true (giving INT_MIN), and at most INT_MAX when positive — both fit on narrowing back to int.

    Dry Run: divide(10, 3)

    a = 10, b = 3, neg = false. Scan k from high to low; only the productive steps are shown.

    k = 1:  a>>1 = 5  >= 3   ->  a -= 3<<1 = 6   (a: 10 -> 4),   q += 2   (q = 2)
k = 0:  a>>0 = 4  >= 3   ->  a -= 3<<0 = 3   (a:  4 -> 1),   q += 1   (q = 3)
result = +q = 3      (remainder a = 1 is discarded)

    And the boundary: divide(INT_MIN, 2) gives a = 2147483648, b = 2; the k = 30 step sets q = 2^30 = 1073741824, neg = true, so the answer is -1073741824 — no overflow because the magnitude lived in 64 bits.

    Sidebar: The Pure 32-bit Version

    If you are barred from 64-bit types entirely, flip the strategy: push both operands into the non-positive domain and build the quotient as a negative count. Every 32-bit value has a negative image (the range is asymmetric in favor of negatives), so nothing overflows. Negate only at the very end, and only when the result is positive — positive results are at most INT_MAX, so that negation is safe.

    // 32-bit only: a, b held as <= 0; quotient accumulated as <= 0
if (dividend == INT_MIN && divisor == -1) return INT_MAX;   // still the one overflow
int a = dividend > 0 ? -dividend : dividend;   // a <= 0  (INT_MIN stays put)
int b = divisor  > 0 ? -divisor  : divisor;    // b <  0
int q = 0;                                      // q <= 0
while (a <= b) {                                 // while |a| >= |b|
    int val = b, mult = -1;
    while (val >= (INT_MIN - val) && (val + val) >= a) {  // safe doubling
        val += val; mult += mult;
    }
    a -= val; q += mult;
}
// q is in [INT_MIN, 0]; positive answers fit after negation
return ((dividend < 0) == (divisor < 0)) ? -q : q;

    The doubling guard val >= INT_MIN - val is itself overflow-safe: INT_MIN - val is in range because val is negative.

    Complexity

    $O(\log |\text{dividend}|) = O(32)$ shift-subtract steps, $O(1)$ space. No multiplication, division, or modulo — only shifts, comparisons, additions, and subtractions.

    Edge Cases

    CallResultNote
    divide(INT_MIN, -1)INT_MAXThe sole overflow; must be clamped.
    divide(INT_MIN, 1)INT_MINFits; the magnitude 2^31 only exists in the 64-bit step or the negative domain.
    divide(INT_MIN, 2)-1073741824Needs abs on a 64-bit value, not int.
    divide(7, -3)-2Truncates toward zero, not toward $-\infty$.
    divide(0, 5)0Loop never sets a quotient bit.
    Takeaway: the asymmetry of two's complement creates exactly one un-representable quotient (INT_MIN / -1) and makes abs(INT_MIN) undefined. Widen before taking magnitudes, or stay in the negative half where every value has a home.
