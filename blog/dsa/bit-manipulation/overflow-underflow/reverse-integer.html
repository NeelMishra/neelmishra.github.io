    Q2: Reverse Integer Without Overflowing

    June 2026 · 13 min read

    Reverse the digits of a signed 32-bit integer. If the reversed value falls outside $[-2^{31},\, 2^{31}-1]$, return 0. The reversing is trivial; the entire difficulty is that you are not allowed to use a 64-bit type, so you must decide "does this overflow?" using only 32-bit arithmetic — and you must decide it before the overflow happens.

    The Problem

    Given int x (e.g. 123 -> 321, -123 -> -321, 120 -> 21), return the reversed integer, or 0 on overflow. Assume the environment cannot store 64-bit integers. LeetCode 7.

    The Tempting (Broken) Approach

    Build the result, then check if it overflowed:

    int reverse(int x) {
    int rev = 0;
    while (x != 0) {
        rev = rev * 10 + x % 10;   // (!) may overflow HERE
        x /= 10;
    }
    if (rev > INT_MAX || rev < INT_MIN) return 0;  // useless check
    return rev;
}
    Why this is wrong: rev is an int, so rev > INT_MAX is always false — an int can never exceed INT_MAX. The overflow already corrupted rev on the line above, and because signed overflow is UB the value you are inspecting is meaningless. You cannot detect signed overflow after it occurs.

    The Idea: Predict, Don't React

    Before executing rev = rev * 10 + pop (where pop = x % 10), ask whether that operation would exceed the range. The new value overflows the positive bound exactly when

    $$rev \cdot 10 + pop > \text{INT\_MAX} = 2147483647.$$

    Divide the reasoning by 10 so every comparison stays in 32-bit range. Let $M = \lfloor \text{INT\_MAX} / 10 \rfloor = 214748364$ and note INT_MAX ends in 7:

    • If rev > M: then rev * 10 alone is already past INT_MAX. Overflow.
    • If rev == M and pop > 7: the high part fits, but the last digit pushes it over. Overflow.

    Symmetrically for the negative side, with $m = \lceil \text{INT\_MIN}/10 \rceil = -214748364$ and INT_MIN ending in 8 (so the trailing digit of a negative pop can be as low as -8):

    • If rev < m: underflow.
    • If rev == m and pop < -8: underflow.
    Negatives just work. In C++, / and % truncate toward zero, so for x = -123 you get x % 10 == -3 and x / 10 == -12. The reversal builds a negative rev directly — no separate sign handling, and no abs(x) (which would itself overflow on INT_MIN; see Q4).

    Dry Run: x = 1534236469

    The reverse of 1534236469 would be 9646324351, which is greater than INT_MAX = 2147483647, so the answer is 0. Watch the guard catch it before corruption:

    M = 214748364,  INT_MAX last digit = 7

step           rev (before)   pop   check
push 9              0          9     ok
push 6              9          6     ok
push 4             96          4     ok
push 6            964          6     ok
push 3           9646          3     ok
push 2          96463          2     ok
push 4         964632          4     ok
push 3        9646324          3     ok
push 5       96463243          5     ok
push 1      964632435          1     rev(=964632435) > M(=214748364) -> return 0

    The last digit (1) is never even multiplied in; the guard fires one step early and returns 0, with every intermediate safely inside int.

    Reference Implementation

    #include <climits>

int reverse(int x) {
    int rev = 0;
    while (x != 0) {
        int pop = x % 10;     // -9..9, sign follows x (trunc toward zero)
        x /= 10;

        // positive overflow guard (before doing rev*10 + pop)
        if (rev > INT_MAX / 10 || (rev == INT_MAX / 10 && pop > 7))
            return 0;
        // negative overflow guard
        if (rev < INT_MIN / 10 || (rev == INT_MIN / 10 && pop < -8))
            return 0;

        rev = rev * 10 + pop;  // now guaranteed not to overflow
    }
    return rev;
}

    INT_MAX / 10 == 214748364 and INT_MIN / 10 == -214748364 are computed at compile time and are themselves in range, so the whole function never leaves 32-bit arithmetic.

    Complexity

    $O(\log_{10} x)$ time — one iteration per digit, at most 10 — and $O(1)$ space. No 64-bit type, no string conversion.

    Edge Cases

    InputOutputWhy
    00Loop body never runs.
    12021Trailing zeros vanish; leading zeros are not stored.
    15342364690Reverse exceeds INT_MAX; guard returns 0.
    -2147483648 (INT_MIN)0Reverse 8463847412 underflows; note pop stays in -8..0, no abs needed.
    -123-321Truncation toward zero keeps pop negative; sign is automatic.
    Pattern to keep: when a type cannot be widened, fold the dangerous operation through division so the comparison runs in the safe range — rev > MAX/10 stands in for the unsafe rev*10 > MAX. The same "divide the bound" trick reappears in Q6.
