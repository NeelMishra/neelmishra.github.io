    Q1: The Binary Search Midpoint Bug

    June 2026 · 12 min read

    This is the most famous overflow bug in all of computing — it sat undetected in the JDK's Arrays.binarySearch for nine years. It is the right place to start because the broken code looks obviously correct.

    The Problem

    You are binary-searching for the smallest integer answer x in a range [lo, hi] such that a monotonic predicate f(x) is true. This "binary search on the answer" pattern is everywhere: minimum capacity, smallest feasible time, Koko eating bananas, allocating books. Crucially, lo and hi are values, not array indices, so they can be as large as the constraints allow — up to $10^9$ or beyond.

    int firstTrue(int lo, int hi, function<bool(int)> f) {
    while (lo < hi) {
        int mid = (lo + hi) / 2;   // looks fine...
        if (f(mid)) hi = mid;
        else        lo = mid + 1;
    }
    return lo;
}

    Why It Breaks

    Suppose the search has narrowed to lo = 1500000000 and hi = 2000000000. Both fit comfortably in an int ($\le 2^{31}-1 = 2147483647$). But their sum does not:

    $$lo + hi = 3.5 \times 10^{9} > 2147483647 = \text{INT\_MAX}.$$

    The addition is evaluated in int before the division, so it overflows. Signed overflow is undefined behavior — in practice it usually wraps to a large negative number, and mid becomes nonsense.

      lo = 1500000000  = 0b0101_1001_0110_1110_1011_0010_1000_0000
+ hi = 2000000000  = 0b0111_0111_0011_0101_1001_0100_0000_0000
-----------------------------------------------------------------
  sum = 3500000000  needs 32 value bits, but bit 31 is the sign bit
      -> the carry lands in the sign bit
      -> reinterpreted as -794967296   (UB; typical two's-complement result)

mid = -794967296 / 2 = -397483648     // out of range, f(mid) is garbage

    A negative mid can make f(mid) throw, index out of bounds, or loop forever. The bug never shows up in small tests, which is exactly why it survives code review.

    Three Correct Fixes

    Fix 1: lo + (hi - lo) / 2 (the standard one)

    Rewrite the midpoint so no intermediate exceeds the range. Since lo <= hi, the difference hi - lo is non-negative and at most hi <= INT_MAX, so it is always in range. Adding back lo lands between lo and hi, also in range.

    int mid = lo + (hi - lo) / 2;

    This is the canonical fix and the one to memorize. It rounds toward lo (floor), which is what the lo < hi / hi = mid shape needs to make progress.

    Subtle: this fix assumes lo <= hi. If lo and hi can straddle zero with one near INT_MIN and the other near INT_MAX, then hi - lo itself can overflow. For ordinary searches where lo <= hi and both are within a sane range that is impossible; if you truly need full-range signed endpoints, use Fix 2.

    Fix 2: unsigned average

    Unsigned addition is defined to wrap modulo $2^{32}$, and crucially the sum plus carry is preserved across the shift. Cast both operands to unsigned, add (the carry goes into bit 32 conceptually, but the low 32 bits plus a logical right shift recover the true average), then shift:

    int mid = (int)(((unsigned)lo + (unsigned)hi) >> 1);

    Here the "overflow" bit is not lost — a logical right shift by 1 divides the 33-bit true sum by 2, pulling the carry back down into bit 31. This even works when lo and hi are arbitrary signed values whose true average is in range. It is the trick the fixed JDK used ((low + high) >>> 1 in Java's unsigned shift).

    Fix 3: the bitwise average

    A pure bit identity computes $\lfloor (a+b)/2 \rfloor$ with no overflow and no division:

    int mid = (lo & hi) + ((lo ^ hi) >> 1);

    The intuition is full-adder algebra: a + b = (a ^ b) + 2 (a & b). The XOR holds the "sum without carries" and the AND holds the "carries" that are worth 2 each. Dividing by two gives (a & b) + (a ^ b) / 2, and shifting the XOR term right by one is that division. No term ever exceeds the operands, so nothing overflows. (For signed inputs use an arithmetic right shift, which C++ guarantees to round toward $-\infty$ for the implementation-defined-but-universal two's-complement behavior; this yields the floor average.)

    Dry Run

    Take lo = 2000000000, hi = 2000000002. The correct midpoint is 2000000001.

    FormulaIntermediateResult
    (lo + hi) / 24000000002 overflows intgarbage / UB
    lo + (hi - lo) / 22000000000 + 2/22000000001 ✓
    ((unsigned)lo + hi) >> 14000000002u >> 12000000001 ✓
    (lo & hi) + ((lo ^ hi) >> 1)2000000000 + 2/22000000001 ✓

    Complexity

    All four midpoint formulas are $O(1)$; the binary search remains $O(\log(\text{range}))$ predicate evaluations. The fixes change correctness, not asymptotics.

    Edge Cases

    • Adjacent endpoints hi = lo + 1: floor midpoint gives lo, so the else lo = mid + 1 branch is what guarantees progress — never use a ceiling midpoint with this loop shape or it can spin forever.
    • Negative ranges: searching over [-2e9-clamped, ...]? Fix 1's hi - lo can overflow if endpoints straddle zero at full magnitude; prefer Fix 2.
    • 64-bit answers: if the answer can exceed INT_MAX (e.g. products), make lo, hi, mid all long long and apply the same fix — widening the type does not remove the need for a safe midpoint, it just moves the cliff to $2^{63}$.

    Reference Implementation

    #include <functional>
using namespace std;

int firstTrue(int lo, int hi, const function<bool(int)>& f) {
    // invariant: answer in [lo, hi]; f is false..false,true..true
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;     // overflow-safe floor midpoint
        if (f(mid)) hi = mid;             // mid works -> answer is <= mid
        else        lo = mid + 1;         // mid fails -> answer is >  mid
    }
    return lo;                            // lo == hi == smallest true x
}
    Takeaway: the danger is never the division, it is the hidden addition. Any time you average, sum, or accumulate two near-maximal signed values, the intermediate is where overflow strikes — rearrange so no intermediate leaves the range.
