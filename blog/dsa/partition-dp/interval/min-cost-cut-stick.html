    Minimum Cost to Cut a Stick

    June 2026 · 28 min read

    A wooden stick of length $L$ must be cut at a given set of positions. Each cut costs the length of the stick segment being cut at that moment, and the order of cuts is yours to choose. Minimise the total cost. This problem closes the Partition DP series by showing that interval DP can fix the first operation, not just the last, whenever the operation's cost is order independent.

    The Problem

    Given stick length $n$ and an array cuts of positions strictly inside $[0, n]$, you may perform the cuts in any order. A cut splits one segment into two; its cost equals the current segment's length. Different orders give different totals because each cut's cost depends on how long the surrounding segment still is.

    Example: $n = 7$, $\text{cuts} = [1, 3, 4, 5]$. Cutting left to right costs $7 + 6 + 4 + 3 = 20$, but a smarter order costs only $16$. We want that minimum.

    Why Order Changes the Cost

    Suppose the stick is length $7$ and you must cut at $3$ and $5$. If you cut at $3$ first you pay $7$, leaving pieces $[0,3]$ and $[3,7]$; then cutting at $5$ pays $4$ (the length of $[3,7]$). Total $11$. If instead you cut at $5$ first you pay $7$, leaving $[0,5]$ and $[5,7]$; then cutting at $3$ pays $5$. Total $12$. Same two cuts, different totals. The cost of a cut is the length of whatever segment it currently sits in, and earlier cuts shrink that segment.

    The Setup: Sort and Add Sentinels

    Sort the cut positions and bookend them with the two ends of the stick, $0$ and $n$. Build $c = [0, \text{sorted cuts}, n]$. Now think in terms of segments delimited by two of these positions. A subproblem is "fully cut the stick segment from position $c_i$ to position $c_j$, given that no cut strictly between them has happened yet". The boundaries $c_i$ and $c_j$ are fixed walls (either real cuts already made, or the stick ends).

    Why sentinels help. Adding $0$ and $n$ as the outer positions means every real cut is interior to some segment $[c_i, c_j]$, and the cost of a cut inside it is simply $c_j - c_i$, the segment length, with no edge cases. This mirrors the wall padding you saw in Burst Balloons.

    The Recurrence: Fix the First Cut

    Let $dp[i][j]$ be the minimum cost to perform every cut strictly between positions $c_i$ and $c_j$. If there is no cut strictly between them, $dp[i][j] = 0$. Otherwise, consider the cut $k$ you make first in this segment, with $i < k < j$. That first cut costs the whole current segment length $c_j - c_i$ (nothing has shrunk it yet), and it splits the work into two independent sub-segments $[c_i, c_k]$ and $[c_k, c_j]$:

    $$dp[i][j] = \min_{i < k < j} \Big(\, dp[i][k] + dp[k][j] + (c_j - c_i) \,\Big),$$

    with $dp[i][j] = 0$ when $j \le i + 1$. The answer is $dp[0][\,m-1\,]$ where $m$ is the size of the padded array $c$.

    First works here, unlike Burst Balloons. Why can we fix the first cut and still get independent halves? Because the cost of that first cut, the segment length $c_j - c_i$, depends only on the fixed walls, not on the order of the remaining cuts. After it, the two sides never interact again: a cut on the left can never change the length of a segment on the right, since the wall at $c_k$ now separates them permanently. Contrast Burst Balloons, where bursting first leaves a hole through which the halves keep influencing each other, forcing the "last" reframing.

    The Code

    int minCost(int n, vector<int>& cuts) {
    sort(cuts.begin(), cuts.end());
    vector<int> c;
    c.push_back(0);
    for (int x : cuts) c.push_back(x);
    c.push_back(n);
    int m = c.size();

    vector<vector<int>> dp(m, vector<int>(m, 0));
    for (int gap = 2; gap < m; ++gap) {            // segment spans at least one interior cut
        for (int i = 0; i + gap < m; ++i) {
            int j = i + gap;
            int best = INT_MAX;
            for (int k = i + 1; k < j; ++k)           // k = first cut made in [c_i, c_j]
                best = min(best, dp[i][k] + dp[k][j] + (c[j] - c[i]));
            dp[i][j] = best;
        }
    }
    return dp[0][m - 1];
}

    If $M$ is the number of cuts, the padded array has $M + 2$ positions and the algorithm runs in $O(M^3)$ time and $O(M^2)$ space, independent of the stick length $n$. Only the relative positions of cuts matter, which is why sorting is the right first move.

    ▶ Cut-a-Stick Interval DP — n = 7, cuts = [1, 3, 4, 5]

    Padded positions c = [0, 1, 3, 4, 5, 7]. We fill dp[i][j] over segments by increasing gap. Each interior cut k is tried as the first cut, paying the segment length c[j]-c[i] plus the two solved halves. Step to build up to dp[0][5].

    First or Last: The Unifying Principle

    Across the three interval posts you have now fixed three different things:

    • Matrix Chain: the last multiplication. Its cost $p_{i-1} p_k p_j$ depends only on the two halves' outer dimensions.
    • Burst Balloons: the last burst. Fixing first would leave a coupling hole; fixing last walls the halves off.
    • Cut a Stick: the first cut. Its cost (segment length) is fixed by the walls, and after it the halves are permanently separated.

    The single rule behind all three: fix the one operation whose cost is independent of the order of the others, and that splits the interval into independent sub-intervals. Sometimes that operation is the first, sometimes the last; the discipline is to find the one that decouples the halves.

    Pitfalls

    • Forgetting to sort. The recurrence assumes $c$ is increasing so that $c_j - c_i$ is a real segment length and the split positions are ordered. Unsorted cuts break everything.
    • Omitting the sentinels. Without $0$ and $n$ in the array, the outermost cuts have no wall to measure against and the segment-length term is wrong.
    • Charging the wrong length. The cost added is the current full segment length $c_j - c_i$, paid once per segment regardless of which interior cut you pick first. A frequent bug is charging $c_k - c_i$ or $c_j - c_k$ instead.

    Wrapping Up the Series

    You now have both halves of partition DP. The front family handles problems where each piece's cost is self-contained, with a one-index dp[i] over prefixes. The interval family handles problems where combining pieces costs something tied to a split point, with a two-index dp[i][j] filled by increasing length. When you meet a new "cut the sequence" problem, the first question is always: does a piece's cost stand alone (front) or depend on a boundary between pieces (interval)? That single distinction routes you to the right template.
