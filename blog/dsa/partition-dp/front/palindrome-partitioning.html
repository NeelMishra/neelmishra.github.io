    Palindrome Partitioning II

    June 2026 · 28 min read

    Given a string $s$, partition it so every piece is a palindrome, using the minimum number of cuts. For s = "aab" the answer is $1$: cut into aa | b. This is the textbook front partition problem, and it is the cleanest place to see how a precomputation table turns a slow recursion into a tidy quadratic DP.

    Where a Beginner Starts: Brute Force

    The naive recursion tries every place to end the first palindromic piece, then recurses on the rest:

    int solve(const string& s, int start) {
    if (start == (int)s.size()) return -1;   // -1 so the last piece adds 0 cuts
    int best = INT_MAX;
    for (int end = start; end < (int)s.size(); ++end) {
        if (isPalindrome(s, start, end))
            best = min(best, 1 + solve(s, end + 1));
    }
    return best;
}

    The -1 base case is a small trick: a partition into $p$ pieces needs $p-1$ cuts, so seeding the terminal with $-1$ makes the "$1 +$" per piece land on the right total. Without memoization this is exponential, and recomputing isPalindrome from scratch makes it worse. Both problems have clean fixes.

    Step 1: Precompute the Palindrome Table

    Define a boolean table $\text{pal}[l][r]$ that is true when $s[l..r]$ is a palindrome. The recurrence is the standard expand-from-the-ends idea:

    $$\text{pal}[l][r] = \big(s[l] = s[r]\big) \;\wedge\; \big(r - l < 2 \;\lor\; \text{pal}[l+1][r-1]\big).$$

    A substring is a palindrome when its outer characters match and its inside is already a palindrome (or is empty / single, the $r - l < 2$ case). Fill it by increasing length, or by decreasing $l$ and increasing $r$, so the inner state $\text{pal}[l+1][r-1]$ is ready:

    int n = s.size();
vector<vector<char>> pal(n, vector<char>(n, 0));
for (int l = n - 1; l >= 0; --l)
    for (int r = l; r < n; ++r)
        pal[l][r] = (s[l] == s[r]) && (r - l < 2 || pal[l + 1][r - 1]);

    This costs $O(n^2)$ time and space and answers any "is $s[l..r]$ a palindrome" query in $O(1)$. With it, the partition DP becomes trivial.

    Step 2: The Front Partition Min-Cut DP

    Reuse the linear template from Part 1. Let $dp[i]$ be the minimum cuts needed for the prefix of length $i$ (the first $i$ characters). The last piece is a palindrome $[j, i-1]$, and the prefix before it needs $dp[j]$ cuts, plus one cut to separate them:

    $$dp[i] = \min_{0 \le j \le i-1,\; \text{pal}[j][i-1]} \big(\, dp[j] + 1 \,\big), \qquad dp[0] = 0.$$

    One subtlety: $dp[i]$ as defined counts pieces, not cuts, if we are not careful. The clean fix is to let $dp[i]$ count cuts and treat the whole prefix being a palindrome as the special case $dp[i] = 0$ (zero cuts). Concretely:

    vector<int> dp(n + 1);
dp[0] = 0;
for (int i = 1; i <= n; ++i) {
    dp[i] = INT_MAX;
    for (int j = 0; j < i; ++j) {
        if (!pal[j][i - 1]) continue;       // last piece [j, i-1] must be a palindrome
        int prev = (j == 0) ? -1 : dp[j];   // j==0 means the whole prefix is one palindrome
        dp[i] = min(dp[i], prev + 1);
    }
}
return dp[n];

    When $j = 0$, the entire prefix $[0, i-1]$ is itself a palindrome, so it needs zero cuts; the prev = -1 then makes prev + 1 = 0. For any other $j$, we pay $dp[j]$ cuts for the front plus one cut to detach the last palindrome. The answer is $dp[n]$, and the whole thing runs in $O(n^2)$.

    Why front partition fits perfectly. The cost of a piece, "is it a palindrome", depends only on that piece's own characters, never on its neighbours. That is the independence condition that makes a one-index state correct. The palindrome table is just the $O(1)$ cost oracle the template asked for.

    ▶ Min-Cut DP — s = "ababbbabbababa"

    We fill dp[i] = minimum cuts for the prefix of length i. At each i we scan every start j of the last piece and keep it only when s[j..i-1] is a palindrome. Step through to watch each palindromic last piece compete.

    Recovering the Actual Cuts

    Track the argmin. Store from[i] = j, the start of the last palindromic piece that achieved $dp[i]$. To print the partition, walk from $i = n$ backward: emit $s[\text{from}[i] .. i-1]$, jump to $i = \text{from}[i]$, repeat until $i = 0$, and reverse. For "aab" you recover ["aa", "b"].

    An $O(n^2)$-Time, $O(n)$-Space Refinement

    The palindrome table costs $O(n^2)$ memory, which is usually fine, but you can avoid it with a different palindrome generation: expand around each of the $2n-1$ centres, and the instant you discover $s[l..r]$ is a palindrome, relax $dp[r+1] = \min(dp[r+1], dp[l] + 1)$. This interleaves palindrome discovery with the DP and keeps only the 1D dp array. It is the same recurrence, just fed by an online palindrome stream instead of a precomputed table.

    vector<int> dp(n + 1, INT_MAX);
dp[0] = 0;
for (int c = 0; c < n; ++c) {
    // odd-length centres, then even-length centres
    for (int d = 0; d <= 1; ++d) {
        int l = c, r = c + d;
        while (l >= 0 && r < n && s[l] == s[r]) {
            int prev = (l == 0) ? -1 : dp[l];
            if (prev != INT_MAX) dp[r + 1] = min(dp[r + 1], prev + 1);
            --l; ++r;
        }
    }
}
return dp[n];
