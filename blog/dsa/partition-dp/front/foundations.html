    Front Partition: The Linear Template

    June 2026 · 26 min read

    Front partition is the gentler half of partition DP. The state is a single index, the recurrence fixes the last piece, and the fill order is just "left to right". Master this one template and you immediately own Word Break, Palindrome Partitioning, Partition Array for Maximum Sum, Split Array Largest Sum, and a long tail of "cut the array into groups" problems.

    The Setup

    You are given a sequence of $n$ elements, $a_0, a_1, \dots, a_{n-1}$. A partition places cuts so the sequence becomes a list of contiguous pieces. Each piece $[l, r]$ has a cost $\text{cost}(l, r)$, and the cost of a partition is the sum of its pieces' costs (sometimes a max, or a count). You want the best partition.

    The defining feature of the front family: a piece's cost depends only on its own boundaries. Nothing outside $[l, r]$ changes $\text{cost}(l, r)$. That independence is exactly what lets a one-index state work.

    Deriving the Recurrence

    Define

    $$dp[i] = \text{best cost to partition the prefix } a_0, \dots, a_{i-1}.$$

    We use a length-based prefix: $dp[i]$ covers the first $i$ elements, so $dp[0]$ is the empty prefix and the final answer is $dp[n]$. This convention avoids most off-by-one pain because "prefix of length $i$" and "next piece starts at index $i$" line up cleanly.

    Now the key move. In any optimal partition of the first $i$ elements, look at the last piece. It must end at index $i-1$ and begin at some index $j$, where $0 \le j \le i-1$. Everything before it is a partition of the first $j$ elements, which is an optimal subproblem. Therefore:

    $$dp[i] = \min_{0 \le j \le i-1} \Big(\, dp[j] + \text{cost}(j, i-1) \,\Big), \qquad dp[0] = 0.$$
    Read the indices carefully. $dp[j]$ handles elements $0 \dots j-1$. The last piece is the closed range $[j, i-1]$. They meet exactly at the boundary between index $j-1$ and index $j$, with no gap and no overlap. Whenever you adapt this template, re-derive this boundary on paper, it is where bugs hide.

    The Template in C++

    Every front partition solution is this loop with a problem-specific cost:

    // dp[i] = best cost for the first i elements
vector<long long> dp(n + 1, LLONG_MAX);
dp[0] = 0;
for (int i = 1; i <= n; ++i) {
    for (int j = i - 1; j >= 0; --j) {     // last piece is [j, i-1]
        if (dp[j] == LLONG_MAX) continue;  // prefix j unreachable
        long long c = cost(j, i - 1);
        if (c == INVALID) continue;        // this piece is not allowed
        dp[i] = min(dp[i], dp[j] + c);
    }
}
return dp[n];

    Three knobs turn this skeleton into a specific problem:

    • cost(l, r) is the cost of making $[l, r]$ a single piece. For Palindrome Partitioning II it is $0$ if $[l, r]$ is a palindrome and "not allowed" otherwise (you want the fewest cuts, so each allowed piece adds $1$). For Partition Array for Maximum Sum it is $\text{max}(a_l..a_r) \times (r-l+1)$.
    • The combiner is usually min or max or a boolean OR (reachability, as in Word Break).
    • A length cap often bounds the inner loop: if pieces can be at most $L$ long, iterate $j$ only from $i-1$ down to $\max(0, i-L)$, giving $O(nL)$ instead of $O(n^2)$.

    A Boolean Instance: Word Break

    Word Break asks: can string $s$ be split into a sequence of dictionary words? This is front partition with a boolean combiner. Let $dp[i]$ be "is the prefix of length $i$ segmentable?". Then

    $$dp[i] = \bigvee_{0 \le j < i} \Big(\, dp[j] \;\wedge\; s[j..i-1] \in \text{dict} \,\Big), \qquad dp[0] = \text{true}.$$

    The animation below fills exactly this table for s = "applepenapple" with dictionary {apple, pen}. Each index $i$ becomes reachable the moment some earlier reachable index $j$ is followed by a dictionary word ending at $i-1$.

    Green boundaries are reachable prefixes. An arrow from boundary $j$ to boundary $i$ means the substring $s[j..i-1]$ is a dictionary word, so reaching $j$ lets us reach $i$. Step through to see the front partition propagate.

    Adding a Piece-Count Dimension

    Many problems fix the number of pieces to exactly $k$ (Split Array Largest Sum, "paint $k$ partitions", and friends). Add one dimension:

    $$dp[i][p] = \text{best cost to split the first } i \text{ elements into exactly } p \text{ pieces}.$$ $$dp[i][p] = \min_{p-1 \le j \le i-1} \Big(\, dp[j][p-1] + \text{cost}(j, i-1) \,\Big).$$

    The base case is $dp[0][0] = 0$ and everything else infinite. The lower bound $j \ge p-1$ just says you cannot form $p-1$ non-empty pieces out of fewer than $p-1$ elements. Time becomes $O(n^2 k)$, which is fine for the usual constraints. When the objective is "minimise the maximum piece sum", the combiner is $\min$ of a $\max$:

    dp[i][p] = min over j of  max( dp[j][p-1], sum(j, i-1) );

    Reconstructing the Cuts

    If you need the actual partition, store the argmin. Keep from[i] = j, the chosen start of the last piece for the best $dp[i]$. Then walk backward from $i = n$: the last piece is $[\text{from}[n], n-1]$, the one before ends at $\text{from}[n]-1$, and so on until you reach $0$. Reverse the collected pieces to print them left to right.

    Complexity

    VariantStateTime
    Unbounded piecesdp[i]$O(n^2 \cdot C)$, $C$ = cost eval
    Piece length $\le L$dp[i]$O(n \cdot L \cdot C)$
    Exactly $k$ piecesdp[i][p]$O(n^2 \cdot k)$

    Often $C$ collapses to $O(1)$ by precomputing: prefix sums for range sums, a palindrome table for palindrome checks, a running max inside the inner loop for max-of-piece costs. Precomputation is what turns a quadratic-with-a-fat-constant solution into a clean one.
