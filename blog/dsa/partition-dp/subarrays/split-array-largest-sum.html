    Split Array Largest Sum

    June 2026 · 29 min read

    Split an array of non-negative integers into exactly $k$ non-empty contiguous subarrays, minimising the largest subarray sum among the pieces. For $a = [7, 2, 5, 10, 8]$ and $k = 2$ the answer is $18$, achieved by $[7, 2, 5]\;|\;[10, 8]$. This is the cleanest min-max partition problem, and it admits two very different solutions: the partition DP from Part 1, and a binary search on the answer that is often the intended one.

    Solution 1: Partition DP

    This is the min-largest-block instance of the K-subarray template verbatim. With $dp[i][p]$ = the smallest achievable largest-block sum for the first $i$ elements split into $p$ blocks:

    $$dp[i][p] = \min_{p-1 \le j \le i-1} \max\big(\, dp[j][p-1],\; \text{sum}(j, i-1) \,\big), \qquad dp[0][0] = 0.$$ 
    int splitArray(vector<int>& a, int k) {
    int n = a.size();
    vector<long long> pre(n + 1, 0);
    for (int i = 0; i < n; ++i) pre[i + 1] = pre[i] + a[i];
    auto sum = [&](int l, int r) { return pre[r + 1] - pre[l]; };

    const long long INF = LLONG_MAX / 4;
    vector<vector<long long>> dp(n + 1, vector<long long>(k + 1, INF));
    dp[0][0] = 0;
    for (int p = 1; p <= k; ++p)
        for (int i = p; i <= n; ++i)
            for (int j = p - 1; j < i; ++j) {
                if (dp[j][p - 1] == INF) continue;
                dp[i][p] = min(dp[i][p], max(dp[j][p - 1], sum(j, i - 1)));
            }
    return (int)dp[n][k];
}

    Time $O(n^2 k)$, space $O(nk)$. Correct and general, but for large $n$ the quadratic factor hurts. The structure of this particular objective lets us do better.

    Solution 2: Binary Search on the Answer

    Flip the question. Instead of "what is the minimum largest sum", ask the decision version: "can we split into at most $k$ subarrays each with sum $\le X$?" Call this $\text{feasible}(X)$. Two facts make binary search work:

    • Monotonicity. If a cap $X$ is feasible, any larger cap is also feasible (more slack can only help). So $\text{feasible}$ is a step function: false for small $X$, true from some threshold onward. The answer is exactly that threshold.
    • A cheap feasibility test. Greedily grow the current block; whenever adding the next element would exceed $X$, start a new block. This uses the fewest blocks possible for cap $X$, so it is feasible iff that minimal block count is $\le k$.

    The search range is $[\max(a),\ \sum a]$. The lower end is forced because every element must fit in some block (a block sum is at least the largest single element). The upper end is the whole array as one block.

    int splitArray(vector<int>& a, int k) {
    long long lo = 0, hi = 0;
    for (int x : a) { lo = max(lo, (long long)x); hi += x; }

    auto feasible = [&](long long cap) {
        int blocks = 1; long long cur = 0;
        for (int x : a) {
            if (cur + x > cap) { ++blocks; cur = x; }  // start a new block
            else cur += x;
        }
        return blocks <= k;
    };

    while (lo < hi) {
        long long mid = lo + (hi - lo) / 2;
        if (feasible(mid)) hi = mid;       // mid works, try smaller
        else lo = mid + 1;                 // mid too tight, raise the cap
    }
    return (int)lo;
}

    Each feasibility test is $O(n)$, and binary search runs $O(\log(\sum a))$ times, for $O(n \log(\sum a))$ total. For $n = 1000$ and values up to $10^6$ that is a few thousand operations versus the DP's millions.

    Why greedy is optimal for the test. For a fixed cap $X$, packing each block as full as possible never increases the number of blocks: any solution that closed a block early can absorb the next element instead without exceeding $X$, never needing more blocks. So the greedy block count is the minimum, and comparing it to $k$ is an exact feasibility check.

    ▶ Binary Search on the Answer — a = [7, 2, 5, 10, 8], k = 2

    Search range starts at [max = 10, sum = 32]. Each step tests the midpoint cap: the greedy pass packs blocks left to right, counts how many it needs, and the cap is feasible when that count is at most k = 2. Watch the window shrink to the answer 18.

    DP vs Binary Search

    Partition DPBinary search
    Time$O(n^2 k)$$O(n \log \sum a)$
    Space$O(nk)$$O(1)$
    Needs monotone feasibilitynoyes
    Generalises to sum-of-scoresyesno

    Binary search wins on speed and memory whenever the objective is a min-max (or max-min) with a monotone feasibility test: Painter's Partition, Book Allocation, Koko Eating Bananas, and "minimum capacity to ship in $D$ days" are all the same shape. The DP is the right tool the moment the objective is additive over blocks, like the sum of averages in Part 3, where no monotone cap test exists.

    The Min-Max Partition Family

    • Painter's Partition / Book Allocation. Identical to Split Array Largest Sum with relabelled story (boards to paint, pages to read). Same binary search.
    • Capacity To Ship Packages Within D Days. Blocks are days; the cap is daily capacity. The greedy test packs packages until the day is full.
    • Minimum Number of Days / machines for a quota. Often the dual: binary search on the resource, greedy-check the count.

    Recognising "minimise the maximum over a contiguous partition" is the cue to reach for binary search on the answer first, and fall back to the DP only if feasibility is not monotone.

    Pitfalls

    • Wrong search bounds. The low end must be $\max(a)$, not $0$ or $1$; otherwise a single oversized element makes every small cap infeasible and the search can underflow the true answer logic. The high end is $\sum a$.
    • Counting blocks off by one. Start blocks = 1 (the first block is open before you add anything), and increment only when you open a new one.
    • Using binary search when the objective is additive. Sum-of-averages or sum-of-costs has no monotone cap test. Reaching for binary search there gives wrong answers; use the DP.
