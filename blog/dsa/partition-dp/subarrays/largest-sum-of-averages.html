    Largest Sum of Averages

    June 2026 · 26 min read

    Partition an array into at most $k$ non-empty contiguous groups, and score a partition by the sum of the groups' averages. Maximise that score. For $a = [9, 1, 2, 3, 9]$ and $k = 3$ the best is $[9]\,|\,[1,2,3]\,|\,[9]$, scoring $9 + 2 + 9 = 20$. This is LeetCode 813, and it is the perfect foil to Split Array Largest Sum: change the objective from a min-max to an additive sum, and the binary search trick evaporates, leaving the partition DP as the clean solution.

    Why Binary Search Does Not Apply

    Split Array Largest Sum worked with binary search because its objective was a maximum over blocks, which gives a monotone feasibility test: "can every block stay under cap $X$". Sum of averages has no such cap. The score is a sum across all blocks, so there is no single threshold to binary search on, and no greedy left-to-right packing is optimal, an early group's average trades off against later ones in a way the greedy cannot see. We need to consider all split positions, which is exactly what DP does.

    The diagnostic. Min-max / max-min objectives over a contiguous partition invite binary search on the answer. Additive objectives (sum of averages, sum of costs, total error) demand a partition DP. Identifying which kind of objective you have is the first decision on any partition problem.

    The Recurrence

    Define $dp[i][p]$ = the maximum sum of averages for the first $i$ elements split into exactly $p$ groups. Fix the last group $[j, i-1]$; the prefix of $j$ elements forms $p-1$ groups:

    $$dp[i][p] = \max_{p-1 \le j \le i-1} \Big(\, dp[j][p-1] + \text{avg}(j, i-1) \,\Big), \qquad dp[0][0] = 0,$$

    where $\text{avg}(l, r) = \dfrac{\text{sum}(l, r)}{r - l + 1}$. Because the problem says at most $k$ groups, the answer is the best over all group counts:

    $$\text{answer} = \max_{1 \le p \le k} dp[n][p].$$

    Using more groups can only help (splitting a group can never lower the total of averages by much, and never below keeping it whole when beneficial), so in practice the maximum lands at $p = k$, but taking the max over $p$ is always safe and costs nothing.

    The Average Oracle

    Prefix sums give $O(1)$ averages, just as in Part 1:

    vector<double> pre(n + 1, 0.0);
for (int i = 0; i < n; ++i) pre[i + 1] = pre[i] + a[i];
auto avg = [&](int l, int r) { return (pre[r + 1] - pre[l]) / (r - l + 1); };

    The Code

    double largestSumOfAverages(vector<int>& a, int k) {
    int n = a.size();
    vector<double> pre(n + 1, 0.0);
    for (int i = 0; i < n; ++i) pre[i + 1] = pre[i] + a[i];
    auto avg = [&](int l, int r) { return (pre[r + 1] - pre[l]) / (r - l + 1); };

    // dp[i][p] = max sum of averages, first i elements in exactly p groups
    vector<vector<double>> dp(n + 1, vector<double>(k + 1, -1e18));
    dp[0][0] = 0.0;
    double best = 0.0;
    for (int p = 1; p <= k; ++p)
        for (int i = p; i <= n; ++i) {
            for (int j = p - 1; j < i; ++j)
                if (dp[j][p - 1] > -1e17)
                    dp[i][p] = max(dp[i][p], dp[j][p - 1] + avg(j, i - 1));
            if (i == n) best = max(best, dp[n][p]);
        }
    return best;
}

    Time $O(n^2 k)$, space $O(nk)$. The only differences from the Split Array code are the combine operator ($+$ instead of $\max$), the opt direction ($\max$ instead of $\min$), and the cost oracle (average instead of sum), exactly the knobs the template told us to expect.

    ▶ Maximising Sum of Averages — a = [9, 1, 2, 3, 9], k = 3

    We fill dp[i][p] = best sum of averages for the first i elements in p groups. Each step fixes the last group [j, i-1] and adds its average to the solved prefix dp[j][p-1]. The answer is the best of dp[5][1..3]. Step through to watch it reach 20.0.

    Reconstructing the Groups

    As before, store from[i][p] = j for the argmax. Start at $(n, p^\star)$ where $p^\star = \arg\max_p dp[n][p]$, emit group $[\,\text{from}[i][p],\, i-1\,]$, jump to $(\text{from}[i][p], p-1)$, repeat, reverse. For the example this recovers $[9]\,|\,[1,2,3]\,|\,[9]$.

    Why Isolating Extremes Helps

    The optimal partition for $[9, 1, 2, 3, 9]$ puts each $9$ in its own singleton group. Intuitively, averaging dilutes large values: lumping a $9$ with small neighbours drags its contribution down, while a singleton group contributes the full $9$. So the optimiser tends to isolate large elements and pool small ones, and the DP discovers this automatically by trying every split. This is a useful sanity check when debugging: if your DP is not isolating obvious extremes, a boundary index is probably wrong.

    The Additive-Objective Family

    • Minimise total partition cost. Replace avg with any per-block cost (variance, squared error, a lookup table) and flip to $\min$. The structure is unchanged.
    • Partition for minimum total error (1D k-means style). Block cost is the sum of squared deviations from the block mean; the same $dp[i][p]$ finds the optimal $k$ contiguous clusters.
    • Maximum / minimum sum of block functions in general. As long as a block's value depends only on its own elements and the objective is additive, this DP applies.

    Pitfalls

    • Reaching for binary search. The additive objective has no monotone cap. Binary search gives wrong answers here; use the DP.
    • Forgetting "at most $k$". Take the max over $p = 1 \dots k$ of $dp[n][p]$, not just $dp[n][k]$, unless you have proven the optimum uses all $k$ groups.
    • Integer division for averages. Averages are real numbers. Use floating point (or exact rationals) for the cost; integer division silently truncates and corrupts the comparison.

    Wrapping Up the Subarray Series

    Three posts, one template. Part 1 built the $dp[i][p]$ skeleton for cutting an array into $k$ contiguous subarrays. Part 2 showed that a min-max objective also yields to binary search on the answer. This post showed that an additive objective rules binary search out and leaves the DP as the clean path. When you meet a new "cut the array into groups" problem, ask first whether the objective is a max/min over blocks (binary search candidate) or a sum over blocks (DP), then plug the cost into the template.
