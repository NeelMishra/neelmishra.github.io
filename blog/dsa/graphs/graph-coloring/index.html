    Graph Coloring: Greedy, Bipartite & Backtracking

    June 2026 · 25 min read

    Imagine you have a map of countries and you need to color each country so that no two neighboring countries share the same color. How many colors do you need? This classic puzzle is the heart of graph coloring — one of the most studied problems in combinatorics and computer science. In this post we will explore graph coloring from fundamentals to competitive-programming applications, with step-by-step visual walkthroughs, an interactive animation, and production-quality C++ code.

    What Is Graph Coloring?

    Graph coloring assigns labels (called “colors”) to vertices of a graph such that no two adjacent vertices share the same color. This deceptively simple constraint appears everywhere in real life:

    • Map coloring: The famous Four Color Theorem (proved in 1976) states that every planar map can be colored with at most 4 colors so that no two adjacent regions share a color.
    • Exam scheduling: Students taking overlapping courses form edges; colors represent time slots. A proper coloring ensures no student has two exams simultaneously.
    • Register allocation: In compilers, variables that are “live” at the same time form edges; colors represent CPU registers. Minimizing colors minimizes register spills.
    • Frequency assignment: Cell towers that interfere form edges; colors represent frequencies. A proper coloring avoids signal interference.
    • Sudoku: Each cell is a vertex, cells in the same row/column/box are adjacent, and the 9 digits are colors.

    Consider this 7-node undirected graph. We will use it throughout this post to demonstrate coloring algorithms:

    Edges: 0–1, 0–2, 1–2, 1–3, 2–4, 3–4, 3–5, 4–6, 5–6. Nodes 0, 1, 2 form a triangle, so we need at least 3 colors.

    Formal Definition

    Proper coloring: A function c : V → {1, 2, …, k} such that for every edge (u, v) ∈ E, c(u) ≠ c(v).

    Key terminology:

    • k-colorable: A graph is k-colorable if a proper coloring exists using at most k colors.
    • Chromatic number χ(G): The smallest k such that G is k-colorable. Finding χ(G) is NP-hard in general.
    • k-chromatic: A graph is k-chromatic if χ(G) = k (it needs exactly k colors, no fewer).
    • Independent set: A set of vertices with no edges between them. Each color class in a proper coloring is an independent set.
    • Clique number ω(G): The size of the largest clique (complete subgraph). Since every vertex in a clique needs a different color, ω(G) ≤ χ(G).

    For our example graph, vertices {0, 1, 2} form a triangle (3-clique), so ω(G) = 3. We’ll see that χ(G) = 3 as well — the greedy algorithm finds an optimal coloring in this case.

    Bounds: ω(G) ≤ χ(G) ≤ Δ(G) + 1, where Δ(G) is the maximum degree. Brooks’ theorem tightens this: χ(G) ≤ Δ(G) unless G is a complete graph or an odd cycle.

    Greedy Coloring Algorithm

    The simplest coloring approach: process vertices in some order, and assign each vertex the smallest color not used by its already-colored neighbors.

    function greedyColor(V, adj):
    color = [-1] * V
    for v = 0 to V-1:
    usedByNeighbors = {color[u] : u ∈ adj[v], color[u] ≠ -1}
    color[v] = smallest non-negative integer ∉ usedByNeighbors
    return color

    Properties:

    • Always produces a valid coloring (no two adjacent vertices share a color).
    • Uses at most Δ(G) + 1 colors (since a vertex has at most Δ neighbors, at most Δ colors are blocked).
    • The result depends on vertex ordering. A bad ordering can use more colors than necessary; an optimal ordering always gives χ(G) colors.
    • Time complexity: O(V + E) with adjacency lists.
    Key insight: Greedy coloring is not guaranteed to find the optimal coloring. However, it is fast, simple, and “good enough” for many competitive programming problems. For exact chromatic number, we need backtracking or other exponential methods.

    Why Ordering Matters

    If we had processed nodes in a different order, greedy might use more colors. For instance, an adversarial ordering on certain graphs can force greedy to use Δ + 1 colors even when χ(G) is much smaller. The greedy algorithm guarantees at most Δ+1 colors, but finding the ordering that minimizes colors is itself NP-hard.

    Practical tip: In competitive programming, the largest-degree-first ordering often produces good results. Sort vertices by degree descending before running greedy. This is the Welsh-Powell heuristic.

    Bipartite Checking as 2-Coloring

    A graph is bipartite if and only if it is 2-colorable (chromatic number ≤ 2). Equivalently, a graph is bipartite iff it contains no odd-length cycle. We can check this with a simple BFS or DFS 2-coloring:

    function isBipartite(V, adj):
    color = [-1] * V
    for v = 0 to V-1:
    if color[v] == -1:
      color[v] = 0
      queue = [v]
      while queue not empty:
        u = queue.dequeue()
        for w ∈ adj[u]:
          if color[w] == -1:
            color[w] = 1 - color[u]
            queue.enqueue(w)
          else if color[w] == color[u]:
            return false // odd cycle found
    return true

    When 2-Coloring Fails: Odd Cycles

    If the graph contains an odd cycle, BFS will find a conflict: two adjacent nodes assigned the same color. The simplest example is a triangle:

    Node A gets Green, B gets Orange. Node C is adjacent to both — it needs a color ≠ Green and ≠ Orange. With only 2 colors, this is impossible. The triangle (3-cycle) is the smallest non-bipartite graph.

    Backtracking for Optimal Coloring (m-Coloring)

    The m-coloring problem asks: can the graph be properly colored with exactly m colors? This is a classic backtracking problem.

    function canColor(node, m, adj, color):
    if node == V: return true // all nodes colored
    for c = 0 to m-1:
    if isSafe(node, c, adj, color):
      color[node] = c
      if canColor(node+1, m, adj, color):
        return true
      color[node] = -1 // backtrack
    return false

    The isSafe function checks if assigning color c to node conflicts with any already-colored neighbor.

    Pruning Strategies

    • Forward checking: Before assigning a color, verify that every uncolored neighbor still has at least one available color. If any neighbor’s domain becomes empty, prune this branch immediately.
    • Degree ordering (MRV): Color the vertex with the fewest remaining color options first. This “most constrained first” heuristic prunes the search tree early.
    • Color symmetry breaking: For the first vertex, only try color 0 (all other initial colors are symmetric). For the second vertex, try at most colors 0 and 1.
    Complexity: Worst case is O(mV), but pruning makes it practical for small graphs. To find χ(G), binary-search on m: try m = 1, 2, 3, … until canColor returns true.

    Finding χ(G): Run the backtracking algorithm with m = 1, then m = 2, then m = 3, etc. The first m that succeeds is the chromatic number. For our 7-node graph, m = 1 and m = 2 fail (triangle exists), and m = 3 succeeds, confirming χ(G) = 3.

    Chromatic Numbers of Special Graph Classes

    Some graph families have known chromatic numbers, which can save you from running any algorithm:

    Graph Classχ(G)Reason
    Empty graph (no edges)1No adjacency constraints.
    Tree / Forest2Trees are bipartite (no cycles).
    Even cycle C2k2Bipartite — alternate two colors.
    Odd cycle C2k+13Cannot 2-color; 3 suffice.
    Complete graph KnnEvery pair is adjacent — all different.
    Complete bipartite Km,n2Bipartite by definition.
    Planar graph≤ 4Four Color Theorem.
    Interval graphω(G)Interval graphs are perfect: χ = ω.
    Bipartite graph≤ 2Definition of bipartiteness.
    Wheel Wn3 or 43 if n is odd, 4 if n is even.
    Petersen graph3Classic result; 3-chromatic.
    Perfect graphs: A graph is perfect if χ(H) = ω(H) for every induced subgraph H. By the Strong Perfect Graph Theorem, a graph is perfect iff it contains no odd hole or odd antihole as an induced subgraph. Interval graphs, chordal graphs, and bipartite graphs are all perfect.

    Edge Coloring

    While vertex coloring assigns colors to vertices, edge coloring assigns colors to edges such that no two edges sharing a vertex have the same color. The minimum number of colors needed is the chromatic index χ′(G).

    Vizing’s Theorem (1964): For any simple graph, Δ(G) ≤ χ′(G) ≤ Δ(G) + 1, where Δ is the maximum vertex degree.

    This means the chromatic index is either Δ or Δ + 1. Graphs where χ′ = Δ are called Class 1; those where χ′ = Δ + 1 are Class 2.

    • Bipartite graphs are always Class 1 (König’s theorem: χ′ = Δ).
    • Petersen graph is Class 2 (Δ = 3, χ′ = 4).
    • Complete graphs: Kn is Class 1 when n is even, Class 2 when n is odd.

    Relationship to Vertex Coloring

    Edge coloring of G is equivalent to vertex coloring of the line graph L(G), where each edge of G becomes a vertex of L(G), and two vertices in L(G) are adjacent iff the corresponding edges in G share an endpoint. This connection is sometimes useful in competitive programming to transform edge-coloring problems into vertex-coloring problems.

    C++ Implementations

    Greedy Coloring

    #include <bits/stdc++.h>
using namespace std;

// Greedy vertex coloring - O(V + E)
vector<int> greedyColoring(int V, vector<vector<int>>& adj) {
    vector<int> color(V, -1);

    for (int u = 0; u < V; u++) {
        // Collect colors used by neighbors
        set<int> blocked;
        for (int v : adj[u]) {
            if (color[v] != -1)
                blocked.insert(color[v]);
        }
        // Assign smallest available color
        int c = 0;
        while (blocked.count(c)) c++;
        color[u] = c;
    }
    return color;
}

int main() {
    int V = 7;
    vector<vector<int>> adj(V);
    auto addEdge = [&](int u, int v) {
        adj[u].push_back(v);
        adj[v].push_back(u);
    };
    addEdge(0,1); addEdge(0,2); addEdge(1,2);
    addEdge(1,3); addEdge(2,4); addEdge(3,4);
    addEdge(3,5); addEdge(4,6); addEdge(5,6);

    auto color = greedyColoring(V, adj);
    int numColors = *max_element(color.begin(), color.end()) + 1;
    cout << "Colors used: " << numColors << endl;
    for (int i = 0; i < V; i++)
        cout << "Node " << i << " -> Color " << color[i] << endl;
    return 0;
}

    Bipartite Check (2-Coloring via BFS)

    #include <bits/stdc++.h>
using namespace std;

bool isBipartite(int V, vector<vector<int>>& adj) {
    vector<int> color(V, -1);

    for (int start = 0; start < V; start++) {
        if (color[start] != -1) continue;
        queue<int> q;
        color[start] = 0;
        q.push(start);

        while (!q.empty()) {
            int u = q.front(); q.pop();
            for (int v : adj[u]) {
                if (color[v] == -1) {
                    color[v] = 1 - color[u];
                    q.push(v);
                } else if (color[v] == color[u]) {
                    return false; // odd cycle detected
                }
            }
        }
    }
    return true;
}

int main() {
    int V = 6;
    vector<vector<int>> adj(V);
    auto addEdge = [&](int u, int v) {
        adj[u].push_back(v);
        adj[v].push_back(u);
    };
    addEdge(0,1); addEdge(0,3); addEdge(1,2);
    addEdge(2,5); addEdge(3,4); addEdge(4,5);

    cout << (isBipartite(V, adj) ? "Bipartite" : "Not Bipartite") << endl;
    return 0;
}

    Output: Bipartite

    m-Coloring Backtracking

    #include <bits/stdc++.h>
using namespace std;

int V;
vector<vector<int>> adj;

bool isSafe(int node, int c, vector<int>& color) {
    for (int v : adj[node])
        if (color[v] == c) return false;
    return true;
}

bool solve(int node, int m, vector<int>& color) {
    if (node == V) return true;

    for (int c = 0; c < m; c++) {
        if (isSafe(node, c, color)) {
            color[node] = c;
            if (solve(node + 1, m, color))
                return true;
            color[node] = -1; // backtrack
        }
    }
    return false;
}

int chromaticNumber() {
    for (int m = 1; m <= V; m++) {
        vector<int> color(V, -1);
        if (solve(0, m, color)) return m;
    }
    return V;
}

int main() {
    V = 7;
    adj.assign(V, {});
    auto addEdge = [&](int u, int v) {
        adj[u].push_back(v);
        adj[v].push_back(u);
    };
    addEdge(0,1); addEdge(0,2); addEdge(1,2);
    addEdge(1,3); addEdge(2,4); addEdge(3,4);
    addEdge(3,5); addEdge(4,6); addEdge(5,6);

    cout << "Chromatic number: " << chromaticNumber() << endl;
    return 0;
}

    Output: Chromatic number: 3

    Implementation note: The backtracking approach is exponential and only suitable for small graphs (V ≤ 20). For competitive programming, greedy coloring and bipartite checks cover most problems. Use backtracking only when exact χ(G) is required on small inputs.

    Chromatic Polynomial

    The chromatic polynomial P(G, k) counts the number of proper k-colorings of graph G. For example:

    • Empty graph (n vertices, no edges): P(G, k) = kn.
    • Complete graph Kn: P(Kn, k) = k(k−1)(k−2)…(k−n+1) = k(n) (falling factorial).
    • Tree on n vertices: P(T, k) = k(k−1)n−1.
    • Cycle Cn: P(Cn, k) = (k−1)n + (−1)n(k−1).

    The chromatic polynomial satisfies the deletion-contraction recurrence: P(G, k) = P(G−e, k) − P(G/e, k), where G−e deletes edge e and G/e contracts it. The chromatic number is the smallest positive integer k where P(G, k) > 0.

    Contest relevance: Chromatic polynomials rarely appear directly in CP, but the deletion-contraction technique is conceptually useful for understanding graph coloring proofs and for solving small instances by hand.

    Applications in Competitive Programming

    Graph coloring problems in CP are usually disguised. The key skill is recognizing when a problem reduces to graph coloring. Here are common patterns:

    Exam / Job Scheduling

    Students share exams, jobs share resources. Build a conflict graph where an edge means “cannot be scheduled together.” The chromatic number gives the minimum number of time slots / machines needed. Often reduces to bipartite matching or interval coloring.

    Frequency Assignment

    Cell towers that interfere are connected by edges. Assign frequencies (colors) to minimize the number used. Constraint: adjacent towers get different frequencies. This is classic graph coloring on an interference graph.

    Register Allocation

    In compiler optimization, variables alive simultaneously conflict. Build an interference graph. The chromatic number tells you the minimum registers needed. If χ exceeds available registers, some variables must be “spilled” to memory.

    Map / Region Coloring

    Adjacent regions on a map must have different colors. The dual graph of the map is planar, so by the Four Color Theorem, 4 colors always suffice. Problems may ask you to find a valid 3-coloring or prove 4 suffice.

    Modeling Tips

    1. Identify conflicts: What pairs of items cannot share the same “resource”? Each pair becomes an edge.
    2. Check for bipartiteness first: If the conflict graph is bipartite, 2 resources suffice. This is O(V + E) and should always be checked first.
    3. Interval graphs: If conflicts arise from overlapping intervals (time slots, ranges), the graph is an interval graph. Greedy coloring sorted by start time gives χ = ω (optimal).
    4. Planar graphs: If the graph is planar, you know χ ≤ 4. Problems may give you a planar graph implicitly (e.g., map regions).
    5. k-coloring decision: If the problem asks “is k colors enough?” for small k (2 or 3), use BFS (k=2) or backtracking (k=3). For general k, it is NP-complete.

    Practice Problems

    Is Graph Bipartite? (LeetCode 785)

    Given an adjacency list, determine if the graph is bipartite. Direct application of BFS 2-coloring. Handle disconnected components.

    Possible Bipartition (LeetCode 886)

    Given N people and a list of mutual dislikes, determine if we can split everyone into two groups such that no two people in the same group dislike each other. This is bipartite checking on the “dislike graph.”

    Flower Planting With No Adjacent (LeetCode 1042)

    N gardens, each connected to at most 3 others. Plant one of 4 flower types in each garden so no two adjacent gardens have the same type. Since max degree ≤ 3, greedy coloring with 4 colors always works.

    m-Coloring Problem (GFG / SPOJ)

    Given a graph and integer m, determine if the graph can be colored with at most m colors. Classic backtracking problem — implement the m-coloring solver with pruning.

    Graph Coloring (Codeforces 664D)

    Color edges of a graph with two colors such that the counts differ by at most 1, and no vertex has all incident edges of one color. Requires creative edge-coloring approach using Eulerian path decomposition.

    Chromatic Number (SPOJ CHROMA)

    Find the chromatic number of a small graph. Use backtracking with pruning. Good practice for the m-coloring solver.

    Interval Graph Coloring / Activity Selection

    Given a set of intervals, find the minimum number of “rooms” so that overlapping intervals are in different rooms. This is coloring an interval graph — greedy by start time gives the optimal answer equal to the maximum overlap.

    Summary

    Let’s recap the key takeaways:

    1. Graph coloring assigns colors to vertices so that no two adjacent vertices share a color. The minimum colors needed is the chromatic number χ(G).
    2. Greedy coloring is simple and fast — O(V + E) — but may not find the optimal number. It always uses at most Δ + 1 colors.
    3. 2-coloring (bipartite check) via BFS/DFS runs in O(V + E) and is the most common graph-coloring technique in CP.
    4. m-coloring backtracking finds exact solutions but is exponential. Use it only for small graphs or when exact χ is needed.
    5. Special graph classes (trees, cycles, planar, interval, perfect) have known chromatic numbers — recognizing them avoids unnecessary computation.
    6. Edge coloring is related via Vizing’s theorem: χ′ is either Δ or Δ + 1.
    7. Most CP coloring problems reduce to bipartite checking, greedy with smart ordering, or recognizing special structure.
    Practice tip: When you see a problem about assigning labels, slots, or resources with conflict constraints, think graph coloring. Start by checking if the conflict graph is bipartite (2-colorable). If not, check for special structure (planar, interval, etc.). Use greedy for fast approximate solutions and backtracking only when exact optimality is required.
