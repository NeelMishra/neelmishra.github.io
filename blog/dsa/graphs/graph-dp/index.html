Graph DP: Dynamic Programming on Graphs

July 2026 · 35 min read

Dynamic programming and graphs are two of the most powerful paradigms in algorithm design. When a graph has no cycles — that is, it’s a Directed Acyclic Graph (DAG) — we can combine both paradigms into a single elegant technique: DP on Graphs. By processing vertices in topological order, we guarantee that every predecessor of a node is already solved before we process that node, exactly the “overlapping subproblems with optimal substructure” setup that DP requires. In this post we cover shortest paths, longest paths, counting paths, classic LeetCode problems, and a decision framework for recognizing Graph DP problems in the wild.

Why DP on Graphs?

Traditional DP operates on a linear or grid-shaped state space. But many problems have a dependency graph that isn’t neatly linear. Consider:

  • Task scheduling where each task depends on multiple predecessors
  • Finding the minimum-cost path through a network of one-way roads
  • Counting the number of distinct routes from a start city to a destination

The key insight is that a DAG has a topological ordering. This ordering gives us the evaluation order for our DP recurrence. If we process nodes in topological order, then when we reach node v, all nodes u with edges u→v have already been processed — their DP values are final. This is exactly the guarantee DP needs.

Core principle: A DAG’s topological ordering is a natural DP evaluation order. No cycles means no circular dependencies in the recurrence. Every DP-on-DAG problem follows the same template: (1) topological sort, (2) process in that order, (3) for each node, combine answers from predecessors/successors.

Compared to general shortest-path algorithms, DP on a DAG is strictly faster:

Dijkstra’s Algorithm

O((V+E) log V) — requires non-negative weights, uses a priority queue.

DAG DP (Topological Relaxation)

O(V+E) — works with any edge weights (including negative), no heap needed.

The DAG structure also lets us solve problems that are NP-hard on general graphs. Longest path on a general graph is NP-hard, but on a DAG it’s solvable in O(V+E). Similarly, counting all paths is #P-hard in general but linear on a DAG.

See Topological Sort for the underlying algorithm that powers all Graph DP solutions.

Shortest Path in a DAG

The most natural application of Graph DP: find the shortest path from a source vertex to all other vertices in a weighted DAG. The algorithm is beautifully simple:

  1. Compute a topological ordering of the DAG.
  2. Initialize dist[source] = 0, all others = ∞.
  3. Process nodes in topological order. For each node u, relax all outgoing edges: for each edge u→v with weight w, set dist[v] = min(dist[v], dist[u] + w).

Because we process in topological order, when we visit u, dist[u] is already optimal — every path to u has already been considered. This makes the relaxation safe.

Why it works with negative weights: Unlike Dijkstra, DAG relaxation doesn’t rely on a greedy “closest first” invariant. The topological order guarantees correctness regardless of edge signs. See Bellman-Ford for the general case on graphs with cycles.

C++ Implementation

// Shortest path from source in a DAG – O(V+E)
#include <bits/stdc++.h>
using namespace std;

void shortestPathDAG(int V, vector<vector<pair<int,int>>>& adj, int src) {
    // Step 1: Topological sort via Kahn's
    vector<int> indeg(V, 0);
    for (int u = 0; u < V; u++)
        for (auto& [v, w] : adj[u]) indeg[v]++;

    queue<int> q;
    for (int i = 0; i < V; i++)
        if (indeg[i] == 0) q.push(i);

    vector<int> topo;
    while (!q.empty()) {
        int u = q.front(); q.pop();
        topo.push_back(u);
        for (auto& [v, w] : adj[u])
            if (--indeg[v] == 0) q.push(v);
    }

    // Step 2: Relax in topological order
    vector<int> dist(V, INT_MAX);
    dist[src] = 0;

    for (int u : topo) {
        if (dist[u] == INT_MAX) continue;
        for (auto& [v, w] : adj[u])
            dist[v] = min(dist[v], dist[u] + w);
    }

    for (int i = 0; i < V; i++)
        cout << i << ": " << (dist[i] == INT_MAX ? -1 : dist[i]) << "\n";
}

The algorithm runs in O(V + E) — one pass for topological sort, one pass to relax edges. No priority queue needed, and negative weights are handled correctly.

Longest Path in a DAG

On general graphs, longest path is NP-hard. But on a DAG, it’s a simple twist on shortest path: replace min with max and initialize distances to −∞ (or negate all weights and run shortest path).

This problem arises in critical path analysis — finding the longest chain of dependent tasks determines the minimum total project duration. It’s also the key idea behind the longest increasing subsequence when modeled as a DAG.

Algorithm

  1. Compute a topological ordering.
  2. Initialize dist[source] = 0, all others = −∞.
  3. Process nodes in topological order. For each edge u→v with weight w: dist[v] = max(dist[v], dist[u] + w).
Critical path: In project scheduling, each task has a duration and dependency edges. The longest path from the start node to the end node gives the minimum project completion time. Any task on this critical path cannot be delayed without delaying the entire project.

C++ Implementation

// Longest path from source in a DAG – O(V+E)
#include <bits/stdc++.h>
using namespace std;

void longestPathDAG(int V, vector<vector<pair<int,int>>>& adj, int src) {
    // Topological sort (Kahn's)
    vector<int> indeg(V, 0);
    for (int u = 0; u < V; u++)
        for (auto& [v, w] : adj[u]) indeg[v]++;

    queue<int> q;
    for (int i = 0; i < V; i++)
        if (indeg[i] == 0) q.push(i);

    vector<int> topo;
    while (!q.empty()) {
        int u = q.front(); q.pop();
        topo.push_back(u);
        for (auto& [v, w] : adj[u])
            if (--indeg[v] == 0) q.push(v);
    }

    // Relax with max instead of min
    vector<int> dist(V, INT_MIN);
    dist[src] = 0;

    for (int u : topo) {
        if (dist[u] == INT_MIN) continue;
        for (auto& [v, w] : adj[u])
            dist[v] = max(dist[v], dist[u] + w);
    }

    for (int i = 0; i < V; i++)
        cout << i << ": " << (dist[i] == INT_MIN ? -1 : dist[i]) << "\n";
}

Same O(V+E) complexity. The only change from shortest path is max instead of min and INT_MIN instead of INT_MAX for initialization.

Watch out: If you need the longest path between two specific nodes in a general graph (with cycles), that’s NP-hard. Always check whether your graph is a DAG first!

Counting Paths in a DAG

How many distinct paths exist from node s to node t in a DAG? On a general graph this is #P-hard, but on a DAG the answer is a simple DP:

paths[v] = Σ paths[u] for all edges u→v (if processing in topological order), with paths[s] = 1.

Alternatively, we can think in the reverse direction: paths[u] = Σ paths[v] for all edges u→v, with paths[t] = 1, processing in reverse topological order.

Example

Consider a DAG with edges: S→A, S→B, A→C, A→B, B→C. How many paths from S to C?

  • S→A→C
  • S→A→B→C
  • S→B→C

Answer: 3 paths. The DP computes: paths[S]=1, paths[A]=1 (from S), paths[B]=1+1=2 (from S and A), paths[C]=1+2=3 (from A and B).

C++ Implementation

// Count distinct paths from src to dst in a DAG – O(V+E)
#include <bits/stdc++.h>
using namespace std;

long long countPathsDAG(int V, vector<vector<int>>& adj, int src, int dst) {
    // Topological sort
    vector<int> indeg(V, 0);
    for (int u = 0; u < V; u++)
        for (int v : adj[u]) indeg[v]++;

    queue<int> q;
    for (int i = 0; i < V; i++)
        if (indeg[i] == 0) q.push(i);

    vector<int> topo;
    while (!q.empty()) {
        int u = q.front(); q.pop();
        topo.push_back(u);
        for (int v : adj[u])
            if (--indeg[v] == 0) q.push(v);
    }

    // DP: paths[v] = sum of paths[u] for all u → v
    vector<long long> paths(V, 0);
    paths[src] = 1;

    for (int u : topo)
        for (int v : adj[u])
            paths[v] += paths[u];

    return paths[dst];
}
Modular counting: In competitive programming, you’re often asked for the count modulo 109+7. Simply add % MOD to the accumulation step: paths[v] = (paths[v] + paths[u]) % MOD.

Cheapest Flights Within K Stops (LC 787)

This classic LeetCode problem combines graph traversal with DP thinking. You’re given n cities, flights with prices, a source, destination, and at most k intermediate stops. Find the cheapest price.

This is not a pure DAG problem — the flight graph may have cycles. But we can model it as DP over “layers”: dp[t][v] = cheapest cost to reach v using at most t edges.

Approach: Modified Bellman-Ford

Run at most k+1 rounds of Bellman-Ford relaxation. In each round, relax all edges using the distances from the previous round (not the current one, to avoid using more stops than allowed).

Key insight: By copying the dist array before each round, we ensure each relaxation uses at most one more edge than the previous round. This turns the general graph into a “layered” DP. See Bellman-Ford for the general shortest-path algorithm.

C++ Implementation

// LC 787: Cheapest Flights Within K Stops
class Solution {
public:
    int findCheapestPrice(int n, vector<vector<int>>& flights,
                          int src, int dst, int k) {
        const int INF = 1e9;
        vector<int> dist(n, INF);
        dist[src] = 0;

        // k stops = k+1 edges
        for (int i = 0; i <= k; i++) {
            vector<int> prev = dist;  // snapshot from last round
            for (auto& f : flights) {
                int u = f[0], v = f[1], w = f[2];
                if (prev[u] < INF)
                    dist[v] = min(dist[v], prev[u] + w);
            }
        }
        return dist[dst] >= INF ? -1 : dist[dst];
    }
};

Time: O(k · E). Space: O(V). This is essentially Bellman-Ford limited to k+1 iterations, with the critical optimization of snapshotting the previous round’s distances.

Alternative: BFS with Level Constraint

You can also solve this with a BFS/Dijkstra approach that tracks (cost, node, stops_remaining) as state. This expands the state space but may be faster in practice for sparse graphs.

Course Schedule Variations

The “Course Schedule” family of LeetCode problems are quintessential Graph DP applications. They all reduce to operations on a prerequisite DAG.

LC 207: Can Finish?

Detect if the prerequisite graph has a cycle. If it’s a DAG, all courses can be finished. Use Kahn’s algorithm — if the topological sort includes all nodes, there’s no cycle.

LC 210: Course Order

Return a valid topological ordering of the courses. If a cycle exists, return empty. This is a direct application of topological sort.

LC 1136: Parallel Courses

Minimum semesters to complete all courses, taking as many as possible in parallel. This is the longest path in the DAG (BFS layers = minimum semesters).

LC 630: Course Schedule III

Maximize courses taken given deadlines and durations. Uses a greedy approach with a heap, not strictly Graph DP, but builds on the ordering intuition.

Parallel Courses (LC 1136): Minimum Semesters

The key insight: courses with no prerequisites can be taken in semester 1. After removing those, courses whose prerequisites are all completed can be taken in semester 2, and so on. This is BFS level-by-level — the number of levels equals the answer.

Equivalently, this computes the longest path in the DAG (measured in number of edges). Each BFS level increments the “semester counter.”

C++ Implementation

// LC 1136: Parallel Courses – minimum semesters
class Solution {
public:
    int minimumSemesters(int n, vector<vector<int>>& relations) {
        vector<vector<int>> adj(n + 1);
        vector<int> indeg(n + 1, 0);

        for (auto& r : relations) {
            adj[r[0]].push_back(r[1]);
            indeg[r[1]]++;
        }

        queue<int> q;
        for (int i = 1; i <= n; i++)
            if (indeg[i] == 0) q.push(i);

        int semesters = 0, completed = 0;

        while (!q.empty()) {
            int sz = q.size();
            semesters++;
            for (int i = 0; i < sz; i++) {
                int u = q.front(); q.pop();
                completed++;
                for (int v : adj[u])
                    if (--indeg[v] == 0) q.push(v);
            }
        }

        return completed == n ? semesters : -1;  // -1 if cycle
    }
};

Time: O(V + E). The BFS processes each node and edge exactly once. The number of BFS levels is the answer.

Connection to Graph DP: This is longest-path DP where each edge has weight 1. The BFS layer number when a node is first reached equals the length of the longest path from any source to that node. See Topological Sort for the Kahn’s algorithm foundation.

Alien Dictionary (LC 269)

Given a sorted list of words in an alien language, derive the character ordering. This is a classic “build a graph, then topological sort” problem.

Algorithm

  1. Build the graph: Compare adjacent words character by character. The first differing character pair (a, b) gives us the edge a→b (a comes before b in the alien alphabet).
  2. Detect invalid input: If a longer word is a prefix of a shorter word that appears after it (e.g., “abc” before “ab”), the input is invalid.
  3. Topological sort: Run Kahn’s algorithm on the character graph. If the result includes all characters, it’s a valid ordering. If not, there’s a contradiction (cycle).
Edge cases to watch:
  • Duplicate words — skip comparison, no edge.
  • Prefix conflict — “abcd” appearing before “abc” is invalid.
  • Characters that appear in words but have no ordering constraints — they can be placed anywhere in the result.
  • Multiple valid orderings may exist — any topological order is acceptable.

C++ Implementation

// LC 269: Alien Dictionary
class Solution {
public:
    string alienOrder(vector<string>& words) {
        unordered_map<char, unordered_set<char>> adj;
        unordered_map<char, int> indeg;

        // Initialize all characters
        for (auto& w : words)
            for (char c : w)
                if (!indeg.count(c)) indeg[c] = 0;

        // Build edges from adjacent word pairs
        for (int i = 0; i + 1 < words.size(); i++) {
            string& w1 = words[i], &w2 = words[i + 1];
            int len = min(w1.size(), w2.size());
            // Prefix conflict check
            if (w1.size() > w2.size() &&
                w1.substr(0, len) == w2.substr(0, len))
                return "";

            for (int j = 0; j < len; j++) {
                if (w1[j] != w2[j]) {
                    if (!adj[w1[j]].count(w2[j])) {
                        adj[w1[j]].insert(w2[j]);
                        indeg[w2[j]]++;
                    }
                    break;
                }
            }
        }

        // Kahn's topological sort
        queue<char> q;
        for (auto& [c, d] : indeg)
            if (d == 0) q.push(c);

        string result;
        while (!q.empty()) {
            char c = q.front(); q.pop();
            result += c;
            for (char nb : adj[c])
                if (--indeg[nb] == 0) q.push(nb);
        }

        return result.size() == indeg.size() ? result : "";
    }
};

Time: O(C) where C is the total number of characters across all words (for building the graph). The topological sort is O(U + E) where U is the number of unique characters and E is the number of ordering edges. In practice E ≤ |words|−1.

Recognizing Graph DP Problems

In contests and interviews, the hardest part is often recognizing that a problem is solvable with Graph DP. Here are the key signals:

Signal 1: DAG Structure

The problem involves dependencies, prerequisites, or a clear ordering where the state transitions form a directed acyclic graph. Look for phrases like:

  • “Task A must be completed before Task B”
  • “Course prerequisites”
  • “Words are sorted in alien order”
  • Any problem where the state space is naturally acyclic

Signal 2: Optimization Along Paths

The problem asks for:

  • Minimum cost / shortest path — classic DAG relaxation
  • Maximum cost / longest path — critical path, longest chain
  • Number of ways / counting paths — DP accumulation

Signal 3: The Graph Can Be Made Acyclic

Sometimes the problem graph has cycles, but you can introduce a “layer” dimension to make it a DAG. Example: Cheapest Flights Within K Stops — the original flight graph has cycles, but the (node, stops_used) state space is acyclic because stops_used only increases.

Decision Framework

Is the graph a DAG?

If yes, proceed directly with topological sort + DP. If no, check if you can make it a DAG (add layer/time dimension, remove back edges, etc.).

What is the DP recurrence?

Shortest: dp[v] = min over predecessors u of (dp[u] + w(u,v))
Longest: dp[v] = max over predecessors u of (dp[u] + w(u,v))
Count: dp[v] = sum over predecessors u of dp[u]

What is the base case?

Usually dp[source] = 0 (for distance) or dp[source] = 1 (for counting). Nodes unreachable from the source keep their default value (∞, −∞, or 0).

Quick test: If removing the “graph” framing, the problem looks like DP with states that have a natural ordering (no circular dependencies), it’s almost certainly Graph DP. The graph just makes the state transitions explicit as edges.

Complexity Analysis

All DAG DP algorithms share the same clean complexity:

Problem Time Space
Shortest Path in DAG O(V + E) O(V + E)
Longest Path in DAG O(V + E) O(V + E)
Counting Paths in DAG O(V + E) O(V + E)
Cheapest Flights (K stops) O(K · E) O(V)
Parallel Courses (min semesters) O(V + E) O(V + E)
Alien Dictionary O(C) O(U + E)

Comparison with Non-DAG Alternatives

DAG Shortest Path

O(V + E)
No heap, works with negative weights. Requires DAG structure.

Dijkstra

O((V + E) log V)
Needs a priority queue. Non-negative weights only. Works on general graphs.

DAG Longest Path

O(V + E)
Simple max-relaxation. Only possible on DAGs.

General Longest Path

NP-hard
No known polynomial algorithm for general graphs.

Bottom line: Whenever you can confirm the graph is a DAG, exploit it! DAG DP gives you O(V+E) algorithms for problems that are much harder on general graphs. The topological ordering is the secret weapon — it eliminates the need for priority queues, repeated relaxation, or exponential search.

Summary

  1. DAG + topological order = DP evaluation order. No cycles means no circular dependencies in the recurrence.
  2. Shortest path in a DAG uses min-relaxation in topological order — O(V+E) with any edge weights.
  3. Longest path in a DAG uses max-relaxation — solves a problem that’s NP-hard on general graphs.
  4. Counting paths uses additive DP — solvable in linear time on DAGs.
  5. Non-DAG problems (Cheapest Flights, Course Schedule) can often be reduced to Graph DP by introducing layers or detecting DAG structure.
  6. Always look for DAG structure — it unlocks the most efficient solutions. When in doubt, check for cycles first.
Practice tip: Start with the three pure DAG problems (shortest, longest, count paths) to build intuition. Then tackle LC 787, LC 1136, and LC 269 — they each add a twist that deepens your understanding of how graph structure enables DP.
