Grid Graphs: BFS & DFS on 2D Matrices

DSA Series Graphs > Grid Graphs

A 2D grid is one of the most common graph representations in competitive programming and coding interviews. Instead of an explicit adjacency list, the grid itself is the graph: each cell is a node, and adjacent cells are connected by implicit edges. This post covers every major grid-graph pattern you need to know, from flood fill to 0-1 BFS, with interactive animations and C++ implementations.

The Grid as a Graph

Any R×C matrix maps naturally to a graph with R·C nodes. Cell (r, c) is a node, and its neighbors are the cells you can move to—typically the 4 cardinal directions (up, down, left, right).

The Direction Arrays Pattern

The most important idiom for grid problems is the direction array. Instead of writing four separate if-statements for each direction, you iterate over a direction array:

// 4-directional movement: up, down, left, right
int dx[] = {-1, 1, 0, 0};
int dy[] = {0, 0, -1, 1};

// 8-directional (includes diagonals)
int dx[] = {-1, -1, -1, 0, 0, 1, 1, 1};
int dy[] = {0, -1, 1, -1, 1, 0, -1, 1};

// Visiting all neighbors of cell (r, c):
for (int d = 0; d < 4; d++) {
    int nr = r + dx[d], nc = c + dy[d];
    if (nr >= 0 && nr < R && nc >= 0 && nc < C) {
        // (nr, nc) is a valid neighbor
    }
}
Convention: In this post, dx changes the row and dy changes the column. Some authors swap this. Pick one convention and stick to it. The direction array pattern is used in every grid problem—memorize it.

Boundary Checks

Before accessing grid[nr][nc], always verify that nr and nc are in bounds. Out-of-bounds access is the #1 source of runtime errors in grid problems. The check nr >= 0 && nr < R && nc >= 0 && nc < C should be automatic muscle memory.

Coordinate Encoding

Sometimes you need to store a cell in a set or use it as a map key. Two common approaches:

// 1. Encode (r, c) as a single integer
int id = r * C + c;    // unique for 0 ≤ r < R, 0 ≤ c < C
int r = id / C, c = id % C;  // decode back

// 2. Use pair or struct
set<pair<int,int>> visited;
visited.insert({r, c});

The integer encoding is faster and uses less memory. Prefer it when the grid fits in R*C < 107.

Flood Fill & Connected Components

The most fundamental grid-graph problem: given a grid of 1s (land) and 0s (water), count the number of islands. Each island is a connected component of land cells. This is LeetCode 200 — Number of Islands.

The Idea

Scan every cell. When you find an unvisited land cell, start a DFS (or BFS) from it, marking all reachable land cells as visited. Each time you initiate a new traversal, you have found a new island.

DFS Solution — Number of Islands (LC 200)

class Solution {
    int R, C;
    int dx[4] = {-1, 1, 0, 0};
    int dy[4] = {0, 0, -1, 1};

    void dfs(vector<vector<char>>& grid, int r, int c) {
        grid[r][c] = '0';  // mark visited by sinking the land
        for (int d = 0; d < 4; d++) {
            int nr = r + dx[d], nc = c + dy[d];
            if (nr >= 0 && nr < R && nc >= 0 && nc < C
                && grid[nr][nc] == '1') {
                dfs(grid, nr, nc);
            }
        }
    }

public:
    int numIslands(vector<vector<char>>& grid) {
        R = grid.size(); C = grid[0].size();
        int count = 0;
        for (int r = 0; r < R; r++)
            for (int c = 0; c < C; c++)
                if (grid[r][c] == '1') {
                    count++;
                    dfs(grid, r, c);
                }
        return count;
    }
};
Trick: Instead of a separate visited[][] array, we modify the grid in-place by setting visited land cells to '0'. This saves O(R·C) space. Only do this if the problem allows modifying the input.

When to Use BFS vs DFS for Flood Fill

Both work. DFS is shorter to code (recursive). BFS is safer for very large grids because it avoids deep recursion stack overflow (e.g., a 1000×1000 grid of all 1s would create a recursion depth of 106). For interview/CP purposes, DFS is standard unless the grid is extremely large.

BFS on Grids: Shortest Path

BFS guarantees the shortest path in an unweighted graph. On a grid, this means the minimum number of moves to get from one cell to another. This is the core idea behind problems like LC 1091 — Shortest Path in Binary Matrix.

The key insight: when BFS reaches a cell for the first time, it has found the shortest path to that cell. This is because BFS explores cells layer by layer—all cells at distance d are processed before any cell at distance d+1. (See the BFS page for a full proof.)

BFS Shortest Path on Grid — C++ Implementation

int shortestPath(vector<vector<int>>& grid) {
    int R = grid.size(), C = grid[0].size();
    if (grid[0][0] == 1 || grid[R-1][C-1] == 1) return -1;

    int dx[] = {-1, 1, 0, 0};
    int dy[] = {0, 0, -1, 1};
    vector<vector<int>> dist(R, vector<int>(C, -1));
    queue<pair<int,int>> q;

    dist[0][0] = 0;
    q.push({0, 0});

    while (!q.empty()) {
        auto [r, c] = q.front(); q.pop();
        if (r == R-1 && c == C-1) return dist[r][c];

        for (int d = 0; d < 4; d++) {
            int nr = r + dx[d], nc = c + dy[d];
            if (nr >= 0 && nr < R && nc >= 0 && nc < C
                && grid[nr][nc] == 0 && dist[nr][nc] == -1) {
                dist[nr][nc] = dist[r][c] + 1;
                q.push({nr, nc});
            }
        }
    }
    return -1; // target unreachable
}
Key detail: Using dist[nr][nc] == -1 doubles as the visited check, just like in standard BFS. No separate visited[][] array needed.

Early Termination

Notice we check if (r == R-1 && c == C-1) right after dequeuing. BFS guarantees that the first time we dequeue the target, we have the shortest distance. We can return immediately without processing the rest of the queue.

Multi-Source BFS

Standard BFS starts from one cell. Multi-source BFS starts from multiple cells simultaneously by pushing all of them into the queue at the beginning. This is the key technique for problems like LC 994 — Rotting Oranges and LC 286 — Walls and Gates. (See also the dedicated multi-source BFS page for more detail.)

The Idea

Enqueue all source cells at distance 0. BFS then radiates outward from all sources at once, finding the minimum distance from any source to every other cell.

Rotting Oranges (LC 994) — C++ Implementation

int orangesRotting(vector<vector<int>>& grid) {
    int R = grid.size(), C = grid[0].size();
    int dx[] = {-1, 1, 0, 0}, dy[] = {0, 0, -1, 1};
    queue<pair<int,int>> q;
    int fresh = 0;

    // Push ALL rotten oranges into the queue (multi-source)
    for (int r = 0; r < R; r++)
        for (int c = 0; c < C; c++) {
            if (grid[r][c] == 2) q.push({r, c});
            if (grid[r][c] == 1) fresh++;
        }

    int minutes = 0;
    while (!q.empty() && fresh > 0) {
        int sz = q.size();
        for (int i = 0; i < sz; i++) {
            auto [r, c] = q.front(); q.pop();
            for (int d = 0; d < 4; d++) {
                int nr = r + dx[d], nc = c + dy[d];
                if (nr >= 0 && nr < R && nc >= 0 && nc < C
                    && grid[nr][nc] == 1) {
                    grid[nr][nc] = 2;
                    fresh--;
                    q.push({nr, nc});
                }
            }
        }
        minutes++;
    }
    return fresh == 0 ? minutes : -1;
}
Pattern: Multi-source BFS = enqueue all sources at level 0, then run standard BFS. The level-by-level loop (int sz = q.size()) counts the number of rounds (minutes, turns, etc.).

Walls and Gates (LC 286)

Same idea: push all gates (value 0) into the queue. BFS fills each empty room with its distance to the nearest gate. This replaces O(R·C) individual BFS runs with a single multi-source BFS, bringing the total time from O((R·C)2) down to O(R·C).

Boundary DFS/BFS

Some problems ask you to identify cells based on their connectivity to the grid boundary. The trick: instead of starting from interior cells, start from the boundary and work inward. This is the key to LC 130 — Surrounded Regions and LC 417 — Pacific Atlantic Water Flow.

Surrounded Regions (LC 130)

The problem: flip all 'O' regions to 'X' unless they touch the boundary. Direct approach (start from each 'O' and check if it reaches the boundary) is complicated. The reverse approach is elegant:

  1. Mark safe cells: DFS/BFS from every 'O' on the boundary. Mark these cells as safe (e.g., change them to 'S').
  2. Flip remaining: Any 'O' not marked safe is surrounded. Flip it to 'X'.
  3. Restore safe cells: Change 'S' back to 'O'.
void solve(vector<vector<char>>& board) {
    int R = board.size(), C = board[0].size();
    int dx[] = {-1, 1, 0, 0}, dy[] = {0, 0, -1, 1};

    // DFS to mark boundary-connected O's as safe
    function<void(int,int)> dfs = [&](int r, int c) {
        if (r < 0 || r >= R || c < 0 || c >= C) return;
        if (board[r][c] != 'O') return;
        board[r][c] = 'S'; // mark safe
        for (int d = 0; d < 4; d++)
            dfs(r + dx[d], c + dy[d]);
    };

    // Step 1: DFS from all boundary O's
    for (int r = 0; r < R; r++) {
        dfs(r, 0);
        dfs(r, C - 1);
    }
    for (int c = 0; c < C; c++) {
        dfs(0, c);
        dfs(R - 1, c);
    }

    // Step 2 & 3: Flip remaining O's, restore safe cells
    for (int r = 0; r < R; r++)
        for (int c = 0; c < C; c++) {
            if (board[r][c] == 'O') board[r][c] = 'X';
            if (board[r][c] == 'S') board[r][c] = 'O';
        }
}
Pattern: When a problem says “connected to the boundary,” start your traversal from the boundary. This “reverse thinking” pattern appears in many grid problems. Pacific Atlantic Water Flow (LC 417) uses the same approach: BFS inward from each ocean boundary instead of outward from each cell.

0-1 BFS on Grids

Standard BFS works on unweighted graphs (all edges cost 1). What if some edges cost 0 and some cost 1? You could use Dijkstra with a priority queue, but there is a faster approach: 0-1 BFS using a deque.

The Idea

When exploring neighbors:

  • If the edge weight is 0, push the neighbor to the front of the deque (it should be processed at the same level).
  • If the edge weight is 1, push to the back (normal BFS behavior).

This maintains the invariant that the deque is sorted by distance, just like a priority queue would—but in O(1) per operation instead of O(log n).

Example: Minimum Cost to Make at Least One Valid Path (LC 1368)

A grid has arrows in each cell pointing in a direction. Moving in the arrow’s direction costs 0. Moving in any other direction costs 1 (you “change” the arrow). Find the minimum cost to reach (R-1, C-1) from (0,0).

int minCost(vector<vector<int>>& grid) {
    int R = grid.size(), C = grid[0].size();
    // Directions: 1=right, 2=left, 3=down, 4=up
    int dx[] = {0, 0, 1, -1};
    int dy[] = {1, -1, 0, 0};

    vector<vector<int>> dist(R, vector<int>(C, INT_MAX));
    deque<pair<int,int>> dq;
    dist[0][0] = 0;
    dq.push_back({0, 0});

    while (!dq.empty()) {
        auto [r, c] = dq.front(); dq.pop_front();

        for (int d = 0; d < 4; d++) {
            int nr = r + dx[d], nc = c + dy[d];
            if (nr < 0 || nr >= R || nc < 0 || nc >= C) continue;

            // Cost 0 if moving in the arrow's direction, else 1
            int w = (grid[r][c] == d + 1) ? 0 : 1;
            if (dist[r][c] + w < dist[nr][nc]) {
                dist[nr][nc] = dist[r][c] + w;
                if (w == 0) dq.push_front({nr, nc});
                else        dq.push_back({nr, nc});
            }
        }
    }
    return dist[R-1][C-1];
}

Dijkstra on Grid

  • Time: O(R·C · log(R·C))
  • Uses priority queue
  • Works for any non-negative weights

0-1 BFS on Grid

  • Time: O(R·C)
  • Uses deque
  • Only works for weights in {0, 1}
When to use 0-1 BFS: Whenever grid edge weights are only 0 or 1. Common scenarios: breaking walls (cost 1) vs free movement (cost 0), changing directions, toggling switches. If weights can be anything > 1, use Dijkstra (see the Dijkstra page).

Grid Problem Tips

After solving dozens of grid problems, these patterns become second nature. Here is a reference cheat sheet.

1. Visited Matrix vs Modifying Input

Separate visited[][]

  • Preserves original grid
  • Uses O(R·C) extra space
  • Needed if you read the grid again later

Modify grid in-place

  • Saves memory
  • Destructive (can’t reuse grid)
  • Common in interview-style problems

A third option: use dist[r][c] == -1 as the “unvisited” sentinel. This combines the visited check with distance tracking, which is the standard approach for BFS shortest path.

2. 4-Direction vs 8-Direction

Read the problem carefully. “Adjacent” usually means 4-directional in LeetCode problems. “Connected” in “Number of Islands” is 4-dir. But LC 1091 (Shortest Path in Binary Matrix) explicitly uses 8-directional movement. Always check the problem constraints.

// 4-directional
int dx[] = {-1, 1, 0, 0};
int dy[] = {0, 0, -1, 1};

// 8-directional
int dx[] = {-1, -1, -1, 0, 0, 1, 1, 1};
int dy[] = {0, -1, 1, -1, 1, 0, -1, 1};

3. Coordinate Encoding for Sets/Maps

When you need to store cells in a set or use them as map keys:

// Fast: encode as single integer
int id = r * C + c;

// Or use pair with set
set<pair<int,int>> visited;
visited.insert({r, c});

// Or use unordered_set with a custom hash
auto hash = [](const pair<int,int>& p) {
    return hash<long long>{}((long long)p.first << 32 | p.second);
};

4. Common Pitfalls

  • Off-by-one on boundaries: Always check nr >= 0 && nr < R && nc >= 0 && nc < C before accessing grid[nr][nc].
  • Marking visited too late in BFS: Mark when enqueuing, not when dequeuing. Otherwise the same cell enters the queue multiple times.
  • Recursion depth for DFS: A 1000×1000 grid can have DFS recursion depth up to 106, causing stack overflow. Use iterative DFS (explicit stack) or BFS for large grids.
  • Forgetting to handle disconnected components: If the grid can have multiple components, you need an outer loop over all cells, not just one BFS/DFS call.
  • Wrong direction array: Make sure dx/dy are consistent. A common bug is using the same offset for both arrays.

5. Decision Framework

Complexity Analysis

For an R×C grid:

Time Complexity

Every grid traversal (BFS, DFS, multi-source BFS, 0-1 BFS) visits each cell at most once and checks each cell’s 4 neighbors. This gives:

  • Time: O(R · C)

Each cell is enqueued/dequeued (or pushed/popped) at most once. Each cell has at most 4 neighbors, so the total work across all neighbor checks is 4 · R · C = O(R · C).

For problems that run a separate traversal from each cell (e.g., naive approach to Walls and Gates), the time is O((R·C)2). Multi-source BFS reduces this to O(R·C) by starting from all sources simultaneously.

Space Complexity

  • Visited array / dist array: O(R · C)
  • BFS queue: At most O(R · C) entries in the worst case (all cells enqueued), but typically O(min(R, C)) for the BFS frontier.
  • DFS stack: O(R · C) in the worst case (spiral path through entire grid). This is why large grids can cause stack overflow with recursive DFS.
Practical note: For competitive programming constraints of R, C ≤ 1000, the grid has up to 106 cells. BFS and 0-1 BFS handle this comfortably. Dijkstra on a grid is O(R·C · log(R·C)) ≈ 2×107, which is also fine. DFS with 106 recursion depth may need stack size adjustments on some systems.
