Implicit Graphs & State-Space Search

DSA Series Graphs > Implicit Graphs

What is an Implicit Graph?

Most graph problems hand you an adjacency list or edge list up front. You know the nodes, you know the edges, and you run BFS or DFS on them. But a large class of problems never gives you a graph at all. Instead, the graph is implicit: you discover it as you explore.

In an implicit graph:

  • Nodes are states — a word, a board configuration, a combination lock reading, a position in a maze with collected keys.
  • Edges are valid transitions — changing one letter, turning one dial, sliding one tile, making one move.
  • The graph is generated on-the-fly during BFS/DFS. You never build the full adjacency list. You compute neighbors of the current state when you need them.

Explicit Graph

Given: adj[0] = {1, 3}

You iterate over pre-built neighbor lists. The graph exists in memory before traversal begins.

Implicit Graph

Given: current state "hit"

You compute neighbors by trying all single-character changes and checking which are valid dictionary words. The graph is never fully materialized.

Classic examples of implicit graphs:

  • Word transformations: each word is a node; edges connect words differing by exactly one letter.
  • Lock combinations: each 4-digit state is a node; turning any dial up or down is an edge.
  • Board/puzzle states: each arrangement of tiles is a node; sliding a tile is an edge.
  • Grid with keys and doors: state = (position, set of keys held); picking up a key or moving is an edge.
Key insight: the BFS/DFS algorithm is identical to what you already know (see the BFS page). The only difference is how you get neighbors. Instead of looking up adj[node], you generate valid next states from the current state.

The state space can be enormous. A 4-digit lock has 10,000 states. A 15-puzzle has over 1012 states. The art of these problems lies in encoding states efficiently, pruning invalid states, and choosing the right search strategy.

Word Ladder (BFS on Transformations)

LeetCode 127. Given two words (beginWord and endWord) and a dictionary, find the length of the shortest transformation sequence from beginWord to endWord, such that only one letter can be changed at a time and each intermediate word must exist in the dictionary.

This is a textbook implicit graph BFS:

  • Nodes: words in the dictionary (plus the start word).
  • Edges: two words are connected if they differ by exactly one character.
  • Goal: shortest path from beginWord to endWord = minimum BFS levels + 1.

The neighbor generation is the key part. For each word, try changing each position to every letter a–z. If the resulting word is in the dictionary and hasn’t been visited, it’s a valid neighbor.

C++ Solution

int ladderLength(string beginWord, string endWord,
                    vector<string>& wordList) {
    unordered_set<string> dict(wordList.begin(), wordList.end());
    if (!dict.count(endWord)) return 0;

    queue<string> q;
    q.push(beginWord);
    dict.erase(beginWord);
    int level = 1;

    while (!q.empty()) {
        int sz = q.size();
        for (int i = 0; i < sz; i++) {
            string word = q.front(); q.pop();
            if (word == endWord) return level;

            for (int j = 0; j < word.size(); j++) {
                char orig = word[j];
                for (char c = 'a'; c <= 'z'; c++) {
                    word[j] = c;
                    if (dict.count(word)) {
                        q.push(word);
                        dict.erase(word); // erase = mark visited
                    }
                }
                word[j] = orig;
            }
        }
        level++;
    }
    return 0; // no path
}
Visited trick: instead of a separate visited set, we erase words from the dictionary as we enqueue them. This serves the same purpose and avoids duplicate storage.

Time: O(M² × N) where M = word length, N = dictionary size. For each word, we try 26 × M character substitutions, and each hash set lookup is O(M). Space: O(M × N) for the queue and set.

Open the Lock

LeetCode 752. A lock has 4 circular wheels, each with digits 0–9. Each move turns one wheel one slot up or down (9 wraps to 0 and vice versa). Given a list of deadend combinations to avoid and a target, find the minimum number of turns to reach the target from "0000".

This maps directly to an implicit graph:

  • Nodes: all 10,000 possible 4-digit combinations.
  • Edges: each combination connects to 8 neighbors (4 wheels × 2 directions).
  • Forbidden nodes: deadends are removed from the search space.
  • BFS from "0000" finds the shortest path to the target.

Generating neighbors for a state like "0273": for each of the 4 positions, turn the digit up or down by 1 (mod 10). That gives up to 8 new states: "1273", "9273", "0373", "0173", "0283", "0263", "0274", "0272".

C++ Solution

int openLock(vector<string>& deadends, string target) {
    unordered_set<string> dead(deadends.begin(), deadends.end());
    if (dead.count("0000")) return -1;

    queue<string> q;
    unordered_set<string> visited;
    q.push("0000");
    visited.insert("0000");
    int turns = 0;

    while (!q.empty()) {
        int sz = q.size();
        for (int i = 0; i < sz; i++) {
            string cur = q.front(); q.pop();
            if (cur == target) return turns;

            for (int j = 0; j < 4; j++) {
                for (int d : {-1, 1}) {
                    string next = cur;
                    next[j] = (cur[j] - '0' + d + 10) % 10 + '0';
                    if (!visited.count(next) && !dead.count(next)) {
                        visited.insert(next);
                        q.push(next);
                    }
                }
            }
        }
        turns++;
    }
    return -1;
}
State encoding: here the state is just a string of 4 digits. For more complex problems, you might encode the state as an integer or a hash. The choice of encoding directly affects performance.

Time: O(104 × 4 × 2) = O(80,000) in the worst case. Space: O(104) for visited set and queue.

Sliding Puzzle

LeetCode 773. On a 2×3 board, tiles numbered 1–5 and one blank (0) can slide. Find the minimum number of moves to reach the goal state [[1,2,3],[4,5,0]].

The implicit graph structure:

  • Nodes: each unique board configuration (6! = 720 possible states).
  • Edges: swapping the blank with an adjacent tile.
  • State encoding: flatten the 2×3 board to a string, e.g., "123450" for the goal. This makes hashing and comparison trivial.

The blank at position i in the flattened array can swap with specific positions. For a 2×3 grid:

// Adjacency for each position in flattened 2x3 grid:
// Position:  0  1  2  3  4  5
// Grid:     [0][1][2]
//           [3][4][5]
vector<vector<int>> swaps = {
    {1, 3},       // pos 0 can swap with 1, 3
    {0, 2, 4},    // pos 1 can swap with 0, 2, 4
    {1, 5},       // pos 2 can swap with 1, 5
    {0, 4},       // pos 3 can swap with 0, 4
    {1, 3, 5},    // pos 4 can swap with 1, 3, 5
    {2, 4}        // pos 5 can swap with 2, 4
};

C++ Solution

int slidingPuzzle(vector<vector<int>>& board) {
    string start;
    for (auto& row : board)
        for (int x : row) start += to_string(x);
    string goal = "123450";
    if (start == goal) return 0;

    vector<vector<int>> swaps = {
        {1,3}, {0,2,4}, {1,5}, {0,4}, {1,3,5}, {2,4}
    };

    queue<string> q;
    unordered_set<string> visited;
    q.push(start);
    visited.insert(start);
    int moves = 0;

    while (!q.empty()) {
        moves++;
        int sz = q.size();
        for (int i = 0; i < sz; i++) {
            string cur = q.front(); q.pop();
            int z = cur.find('0'); // blank position

            for (int nb : swaps[z]) {
                string next = cur;
                swap(next[z], next[nb]);
                if (next == goal) return moves;
                if (!visited.count(next)) {
                    visited.insert(next);
                    q.push(next);
                }
            }
        }
    }
    return -1;
}
Why strings? Converting the board to a string makes it hashable and comparable in O(1) average time via unordered_set. Storing the full 2D array as a state would require custom hashing. For small boards (2×3 = 6 chars), a string is efficient and clean.

Time: O(6! × 6) = O(4,320). We visit at most 720 states, and each state generates up to ~3 neighbors. Space: O(6!) for the visited set.

Bidirectional BFS

Standard BFS from the start explores all states within distance d. If the branching factor is b, that’s O(bd) states. Bidirectional BFS runs two BFS simultaneously — one from the start, one from the end — and stops when they meet in the middle.

The complexity drops from O(bd) to O(bd/2). For a branching factor of 26 (like word ladder) and depth 10, that’s 2610 ≈ 1014 vs 2 × 265 ≈ 24 million — a massive improvement.

Algorithm:

  1. Maintain two sets: forward (starts with beginWord) and backward (starts with endWord).
  2. Each iteration, expand the smaller set (this balances the search).
  3. For each word in the current set, generate neighbors. If a neighbor is in the other set, the two searches have met.
  4. If a neighbor is new, add it to a nextLevel set.
  5. Replace the current set with nextLevel and increment the level counter.

C++ Bidirectional BFS for Word Ladder

int ladderLengthBidir(string beginWord, string endWord,
                        vector<string>& wordList) {
    unordered_set<string> dict(wordList.begin(), wordList.end());
    if (!dict.count(endWord)) return 0;

    unordered_set<string> fwd = {beginWord};
    unordered_set<string> bwd = {endWord};
    dict.erase(beginWord);
    dict.erase(endWord);
    int level = 1;

    while (!fwd.empty() && !bwd.empty()) {
        level++;
        // always expand the smaller frontier
        if (fwd.size() > bwd.size()) swap(fwd, bwd);

        unordered_set<string> next;
        for (auto& word : fwd) {
            string w = word;
            for (int j = 0; j < w.size(); j++) {
                char orig = w[j];
                for (char c = 'a'; c <= 'z'; c++) {
                    w[j] = c;
                    if (bwd.count(w)) return level;
                    if (dict.count(w)) {
                        next.insert(w);
                        dict.erase(w);
                    }
                }
                w[j] = orig;
            }
        }
        fwd = next;
    }
    return 0;
}
Always expand the smaller set. This keeps both frontiers roughly the same size and maximizes the benefit of meeting in the middle. Without this optimization, one side can grow much faster, negating the advantage.

BFS vs DFS for State Spaces

Both BFS and DFS can explore implicit graphs. The choice depends on the problem:

BFS — Use When:

  • Shortest path / minimum moves is required. BFS guarantees the first time you reach a state, it’s via the shortest path.
  • The search depth is bounded and moderate. BFS explores level by level, so if the answer is at depth d, it checks all states up to that depth.
  • You need all states at a given distance (e.g., “all words reachable in 3 transformations”).

DFS — Use When:

  • Existence is the question: “Can I reach state X?” without needing the shortest path.
  • Memory is a concern. DFS uses O(depth) memory vs BFS’s O(branchingdepth).
  • The state space has very deep paths with few branches (e.g., Sudoku backtracking).
  • You want to enumerate all solutions (DFS with backtracking).

Iterative Deepening DFS (IDDFS)

IDDFS combines the best of both: BFS’s shortest-path guarantee with DFS’s memory efficiency. It repeatedly runs depth-limited DFS with increasing depth limits: first depth 0, then 1, then 2, and so on.

  • Time: O(bd) — same as BFS. The repeated work is only a constant factor overhead.
  • Space: O(d) — same as DFS. Only one path is stored at a time.
  • Guarantee: finds the shortest solution, just like BFS.
bool dls(State cur, State target, int limit) {
    if (cur == target) return true;
    if (limit == 0) return false;
    for (State next : getNeighbors(cur)) {
        if (dls(next, target, limit - 1))
            return true;
    }
    return false;
}

int iddfs(State start, State target) {
    for (int depth = 0; ; depth++) {
        if (dls(start, target, depth))
            return depth;
    }
}
When to use IDDFS: when BFS runs out of memory. For problems like the 15-puzzle (1012 states), BFS is infeasible. IDDFS with a good heuristic (IDA*) is the standard approach.

Recognizing Implicit Graph Problems

Not every problem says “graph” in the description. Here are the signals that tell you an implicit graph is hiding underneath:

Signal in Problem Statement Implicit Graph Structure Algorithm
“Minimum number of moves/steps”Each state is a node; each move is an edgeBFS
“Transform X into Y”States connected by valid transformationsBFS (shortest) or DFS (existence)
“Can you reach state Z?”Reachability in state graphBFS or DFS
“Shortest sequence of operations”Operation = edge, configuration = nodeBFS
“All distinct configurations”Enumerate reachable statesBFS or DFS
Puzzle with moves (Rubik, 8-puzzle, etc.)Board state = node, legal move = edgeBFS / IDDFS / A*
Grid with keys/doors/portalsState = (position, inventory); transitions change position or inventoryBFS on compound state

Common Patterns

Pattern 1: Single-Edit BFS. States are strings or arrays. Each step modifies one element (one character, one digit, one tile). Examples: Word Ladder, Open the Lock, Sliding Puzzle.

Pattern 2: Compound State BFS. The state is more than just a position. It includes collected items, toggled switches, or visited nodes. Example: “Shortest path in a grid collecting all keys” — state = (row, col, bitmask of keys).

Pattern 3: Forbidden States. Some states are invalid (deadends, obstacles, visited-before). Handle by putting them in a set and skipping during neighbor generation. Example: Open the Lock deadends.

Recipe for any implicit graph problem:
1. Define the state (what information fully describes the current situation).
2. Define transitions (how can one state change to another in one step).
3. Define the goal condition.
4. Choose BFS (shortest) or DFS (existence/enumeration).
5. Implement state encoding for efficient hashing/comparison.

Complexity Analysis

Analyzing the complexity of state-space search differs from standard graph traversal because you must reason about the size of the state space rather than a given V and E.

State Space Size

  • Word Ladder: |V| = N (dictionary size), |E| = O(N × M × 26) where M is word length.
  • Open the Lock: |V| = 104 = 10,000 states, |E| = 8 per state = 80,000 total.
  • Sliding Puzzle (2×3): |V| = 6! = 720, |E| ≈ 2–3 per state.
  • 15-Puzzle (4×4): |V| = 16!/2 ≈ 1013 reachable states. BFS is infeasible.
  • Rubik’s Cube: |V| ≈ 4.3 × 1019. Requires specialized search (IDA*, pattern databases).

Branching Factor

The branching factor b is the average number of neighbors per state. It determines how fast the search frontier grows:

Problem Branching Factor States at Depth d
Grid (4-connected)~4O(4d)
Open the Lock8O(8d)
Word Ladder (M=3)up to 78O(78d)
Rubik’s Cube~18O(18d)

Memory: The Real Bottleneck

For BFS, memory usage is O(bd) because the entire frontier (and visited set) must be stored. This is often the limiting factor, not time. Strategies for large state spaces:

  • Bidirectional BFS: reduces frontier from O(bd) to O(bd/2).
  • IDDFS: O(d) memory, same time as BFS.
  • A* / IDA*: heuristic-guided search. Explores fewer states by prioritizing those closer to the goal.
  • Compact state encoding: use integers or bitfields instead of strings/arrays. A 2×3 sliding puzzle state fits in a single 32-bit integer.
Rule of thumb: BFS works well when the state space has ≤ 106–107 states. Beyond that, consider bidirectional BFS, IDDFS, or heuristic search. Always estimate the state space size before coding.
