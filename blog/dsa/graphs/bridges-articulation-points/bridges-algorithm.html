Tarjan's Bridge Detection Algorithm

DSA Series Graphs > Bridges & Articulation Points

What is a Bridge?

A bridge (also called a cut edge) in an undirected graph is an edge whose removal increases the number of connected components. In simpler terms, if you remove a bridge, the graph falls apart into two or more disconnected pieces.

Graph with a Bridge

Edge 0-2 is a bridge. Remove it and nodes {0,1} disconnect from {2,3,4}.

Graph Without Bridges

Adding edge 1-4 creates a cycle. Now no single edge removal disconnects the graph.

Key Insight: An edge is a bridge if and only if it does not lie on any cycle. If an edge belongs to a cycle, removing it still leaves an alternate path between its endpoints.

The Naive Approach

The brute-force method is straightforward: for every edge (u, v) in the graph, temporarily remove it and run BFS/DFS to check whether the graph remains connected.

for each edge (u, v) in graph: remove (u, v) if isDisconnected(): // O(V + E) BFS/DFS report (u, v) as bridge restore (u, v)
AspectComplexity
Edges checkedO(E)
Connectivity check per edgeO(V + E)
TotalO(E × (V + E))

For a graph with 100,000 edges, that is around 1010 operations. Far too slow. Tarjan's algorithm solves this in a single DFS pass: O(V + E).

DFS Trees and Back Edges

When you run DFS on an undirected graph, every edge falls into exactly one of two categories:

  • Tree edge: The edge used to discover a new (unvisited) node. These edges form a spanning tree.
  • Back edge: An edge to an already-visited node (not the parent). These create cycles in the graph.
Why only these two? In an undirected graph, there are no forward edges or cross edges during DFS. Every non-tree edge connects a node to one of its ancestors, making it a back edge.

Visualizing the DFS Tree

Consider a graph with 6 nodes. The DFS starting from node 0 produces a spanning tree (solid green) with back edges (dashed purple):

Crucial observation: A tree edge (u, v) where u is the parent can only be a bridge if the subtree rooted at v has no back edge reaching u or any ancestor of u. If such a back edge exists, removing (u, v) still leaves a path from v back to u through that back edge.

Discovery Time and Low Values

disc[v] (Discovery Time)

A counter incremented each time DFS visits a new node. disc[v] records when node v was first discovered. Nodes discovered earlier have smaller disc values.

low[v] (Low-Link Value)

low[v] is the minimum discovery time reachable from the subtree rooted at v, using tree edges going down and at most one back edge going up. Formally:

low[v] = min( disc[v], // itself disc[w] for each back-edge (v, w), // direct back edges low[c] for each child c of v // propagated from children )
Common mistake: Some implementations use low[w] instead of disc[w] for back edges. For bridges this still works correctly, but for articulation points it can give wrong results. We use disc[w] for back edges to keep the algorithm correct for both.

How low values propagate

Consider a chain 0 → 1 → 2 → 3 with a back edge 3 → 0. When DFS reaches node 3 and sees the back edge to 0 (disc=0), it sets low[3] = 0. As the recursion unwinds, each parent takes the min of its own low and the child's low, so the value 0 propagates up through nodes 2 and 1.

Back edge 3→0 causes low=0 to propagate to all nodes. No bridges in this chain.

The Bridge Condition

After computing disc and low arrays via DFS, the bridge detection rule is remarkably simple:

A tree edge (u, v) where u is the parent of v is a bridge if and only if:

low[v] > disc[u]

Why does this work?

low[v] > disc[u] means the subtree rooted at v cannot reach node u or any of u's ancestors through back edges. The only connection from v's subtree to the rest of the graph is through the edge (u, v). Remove it and the subtree becomes disconnected.

Conversely, if low[v] ≤ disc[u], then some node in v's subtree has a back edge reaching u or an ancestor. Even without (u, v), there is still a path.

Bridge: low[v] > disc[u]

No back edges from v's subtree reaching u. Subtree is isolated without (u,v).

Not a Bridge: low[v] ≤ disc[u]

Back edge provides an alternate path. Removing (u,v) does not disconnect.

Strict inequality matters: For bridges we use low[v] > disc[u] (strictly greater). For articulation points, the condition is low[v] ≥ disc[u] (greater or equal). Mixing these up is one of the most common implementation errors.

Complete Algorithm

bridges(graph): timer = 0 disc = [-1] * V // -1 means unvisited low = [0] * V for each node u in 0..V-1: if disc[u] == -1: dfs(u, -1) dfs(u, parent): disc[u] = low[u] = timer++ for each neighbor v of u: if v == parent: continue // skip the edge back to parent if disc[v] == -1: // v is unvisited: tree edge dfs(v, u) low[u] = min(low[u], low[v]) if low[v] > disc[u]: // BRIDGE FOUND report (u, v) as bridge else: // v is visited: back edge low[u] = min(low[u], disc[v])

C++ Implementation

#include <vector> using namespace std; int timer_val = 0; vector<int> disc, low; vector<vector<int>> adj; vector<pair<int,int>> bridges; void dfs(int u, int parent) { disc[u] = low[u] = timer_val++; for (int v : adj[u]) { if (v == parent) { parent = -1; // handle multigraph: skip only once continue; } if (disc[v] == -1) { dfs(v, u); low[u] = min(low[u], low[v]); if (low[v] > disc[u]) bridges.push_back({u, v}); } else { low[u] = min(low[u], disc[v]); } } } void findBridges(int n) { disc.assign(n, -1); low.assign(n, 0); bridges.clear(); timer_val = 0; for (int i = 0; i < n; i++) if (disc[i] == -1) dfs(i, -1); }
Multigraph note: The parent = -1 trick ensures that if there are multiple edges between u and v, we only skip one of them. If we skip all, we might miss that duplicate edges prevent a bridge.

Edge Cases and Pitfalls

1. Parallel (Multiple) Edges

If nodes u and v are connected by two edges, neither is a bridge (removing one still leaves the other). The standard parent-skip logic skips ALL edges to the parent, which is wrong for multigraphs.

Fix: Track the edge index or use the parent = -1 trick shown in the implementation. After skipping one occurrence, set parent to -1 so subsequent edges to the same node are treated as back edges.

2. Self Loops

A self-loop (u, u) is never a bridge. Since v == parent when v == u, the standard algorithm naturally skips it. No special handling needed.

3. Disconnected Graphs

The outer loop (for each node u: if disc[u] == -1: dfs(u, -1)) handles disconnected components. Each component is processed independently.

4. Single Node / Single Edge

  • Single node, no edges: No bridges. The algorithm does nothing meaningful.
  • Two nodes, one edge: That edge is a bridge. disc[0]=0, disc[1]=1, low[1]=1. Since low[1]=1 > disc[0]=0, the edge is correctly identified.

5. A Tree (No Cycles)

Every edge in a tree is a bridge. Since trees have no back edges, low[v] = disc[v] for every node v. For every parent-child pair, low[child] = disc[child] > disc[parent], so every edge is flagged.

Time and Space Complexity

ResourceComplexityExplanation
TimeO(V + E)Single DFS traversal. Each node visited once, each edge examined twice (once from each endpoint).
Space: disc[] and low[]O(V)Two integer arrays of size V.
Space: Recursion stackO(V) worst caseFor a path graph (chain), the DFS recursion depth equals V.
Space: Adjacency listO(V + E)Standard graph storage.
Total SpaceO(V + E)Dominated by the adjacency list.

Why is it O(V + E) and not O(V × E)?

The naive approach checks connectivity after removing each edge, leading to O(E) work per edge. Tarjan's algorithm avoids this by encoding reachability information in the low-link values during a single DFS. The low values capture whether a subtree can "escape" via back edges, so we check the bridge condition in O(1) per edge as we backtrack.

Stack Overflow Risk

For very large graphs (V > 105), the recursive version may overflow the call stack. Solutions:

  • Increase stack size (e.g., ulimit -s unlimited on Linux, or create a thread with larger stack)
  • Convert to iterative DFS using an explicit stack
