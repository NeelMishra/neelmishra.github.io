    Bridges vs Articulation Points: The Algorithm Comparison

    April 2026 · 25 min read

    Bridge detection and articulation point detection share the same DFS framework, the same disc[] and low[] arrays, and can run in the same single pass. Yet they answer fundamentally different questions and their detection conditions differ by a single character: > vs >=. This post puts the two algorithms side by side, shows exactly where and why they diverge, and builds to a combined C++ implementation.

    Same Framework, Different Questions

    Both algorithms protect against network failure, but they guard against different kinds of failure:

    Bridge (Edge Failure)

    An edge whose removal disconnects the graph. Think of a single cable between two offices — cut it and the offices can’t communicate.

    Question: “Which connections are irreplaceable?”

    Articulation Point (Vertex Failure)

    A vertex whose removal disconnects the graph. Think of a router that links two subnetworks — lose it and they’re isolated.

    Question: “Which nodes are single points of failure?”

    We’ll use this undirected graph throughout. It has triangles (cycles), bridges, and a leaf — the perfect test bed.

    The Shared Foundation: DFS Tree & disc[]/low[]

    Both algorithms start with identical steps: run DFS, compute discovery times (disc[]) and low-link values (low[]). Only the detection condition differs.

    How disc[] and low[] are computed

    disc[u] = the order in which u is first visited. low[u] = the lowest disc[] reachable from u’s subtree via tree + at most one back edge.

    Key point: disc[] and low[] are computed identically for both bridges and APs. The arrays don’t change — only what you check changes.

    Bridge Detection: The Strict Inequality

    For a tree edge (u → v) where u is the parent:

    // Bridge condition
    if (low[v] > disc[u]):
    edge (u, v) is a bridge

    Meaning: v’s entire subtree cannot reach u or anything above u. The edge (u,v) is the only connection between the subtree and the rest of the graph.

    Checking each tree edge

    low[1]=0 > disc[0]=0?   0 > 0?   NO

    Node 1’s subtree can reach node 0 (via 2→0 back edge). Not a bridge.

    low[2]=0 > disc[1]=1?   0 > 1?   NO

    Node 2 can reach node 0 (above node 1). Not a bridge.

    low[3]=3 > disc[2]=2?   3 > 2?   YES — BRIDGE!

    Node 3’s subtree cannot reach node 2 or above. Removing edge 2-3 disconnects {3,4,5,6} from {0,1,2}.

    Bridge found: (2, 3)

    low[4]=3 > disc[3]=3?   3 > 3?   NO

    Node 4’s subtree reaches node 3 exactly (via 5→3). Not a bridge — the strict inequality rejects the equal case.

    low[5]=3 > disc[4]=4?   3 > 4?   NO

    Node 5 reaches above node 4 (back to 3). Not a bridge.

    low[6]=6 > disc[4]=4?   6 > 4?   YES — BRIDGE!

    Node 6 is a leaf with no back edges. Removing 4-6 isolates it.

    Bridge found: (4, 6)
    Result: Bridges are (2,3) and (4,6). Notice edge 3→4 was NOT a bridge because low[4]=3 == disc[3]=3 — the strict > rejects it.

    Articulation Point: The Non-Strict Inequality + Root Rule

    For a non-root vertex u with child v in the DFS tree:

    // AP condition (non-root)
    if (low[v] >= disc[u]):
    u is an articulation point

    // AP condition (root)
    if (parent == -1 and childCount >= 2):
    u is an articulation point

    Checking each vertex

    Root with 1 DFS-tree child (only node 1).   NOT an AP

    Root needs 2+ children. Node 0 has only 1 child in the DFS tree.

    Child 2: low[2]=0 >= disc[1]=1?   0 >= 1?   NO

    Node 2’s subtree reaches above node 1 (to node 0). Removing 1 doesn’t disconnect anything.

    Child 3: low[3]=3 >= disc[2]=2?   3 >= 2?   YES — AP!

    Node 3’s subtree cannot reach above node 2. Removing node 2 isolates {3,4,5,6} from {0,1}.

    Articulation Point: node 2

    Child 4: low[4]=3 >= disc[3]=3?   3 >= 3?   YES — but wait...

    The condition fires! But is node 3 really an AP? Node 4’s subtree can reach node 3 itself (via 5→3), but if we remove node 3, that back edge is gone too. However, node 5 connects to node 3, and removing 3 disconnects 4,5 from the rest... Actually, let’s think carefully.

    Node 3 has neighbors: 2, 4, 5. In the DFS tree, child = 4. Back edge 5→3 comes from 5, which is in 4’s subtree. If we remove node 3: nodes 4, 5, 6 lose their only connection to 0, 1, 2 (since edge 2-3 was the only link). Yes, node 3 IS an AP.

    Articulation Point: node 3

    Child 5: low[5]=3 >= disc[4]=4?   3 >= 4?   NO

    Child 6: low[6]=6 >= disc[4]=4?   6 >= 4?   YES — AP!

    Node 6 is trapped below node 4. Removing node 4 isolates node 6.

    Articulation Point: node 4

    Leaf nodes have no children in DFS tree. Cannot be APs

    A leaf vertex can never be an AP because removing it doesn’t disconnect any other nodes from each other.

    Result: Articulation points are 2, 3, 4. Notice node 3 was caught because low[4]=3 == disc[3]=3 — the non-strict >= accepts the equal case, which the bridge check rejected!

    The Critical Difference: > vs >=

    This is the single most important distinction between the two algorithms. Let’s zoom into the case that only the AP algorithm catches.

    The low[v] == disc[u] Case

    When low[v] == disc[u], it means v’s subtree has a back edge that reaches exactly u (not above u). Let’s see why this matters differently:

    For Bridges: NOT a bridge

    v can reach u through another path (the back edge). Even if we remove the tree edge (u,v), we can still go v → ... → back edge → u. The graph stays connected. The edge is redundant.

    For APs: IS an AP

    The back edge reaches u itself. But if we remove vertex u, that back edge is destroyed too (since u is an endpoint). Now v’s subtree has no way to reach anything above u. The removal of u disconnects the graph.

    Visual: The Exact Scenario

    Look at edge 3→4 in our graph. low[4]=3, disc[3]=3. The back edge 5→3 reaches node 3 exactly.

    • Bridge check: 3 > 3? NO. Edge 3-4 is NOT a bridge. ✓ Correct — removing edge 3-4, we can still go 4→5→3.
    • AP check: 3 >= 3? YES. Node 3 IS an AP. ✓ Correct — removing node 3 destroys the back edge too, trapping {4,5,6}.

    The Truth Table

    ConditionBridge?AP?Intuition
    low[v] > disc[u] ✓ YES ✓ YES Subtree is completely isolated below u. Both detect it.
    low[v] == disc[u] ✗ NO ✓ YES Back edge reaches u. Edge survives (bridge: no), but removing vertex u kills the back edge too (AP: yes).
    low[v] < disc[u] ✗ NO ✗ NO Subtree connects above u. Neither is a failure point.
    The one-character difference: Bridges use > (strict). APs use >= (non-strict). Confusing them is the #1 bug in interview implementations.

    The Root Special Case (APs Only)

    Bridges never need a root special case. APs do. Here’s why:

    Why bridges skip the root

    A bridge is an edge property. The root’s tree edges are checked just like any other edge — the low[v] > disc[root] condition works fine. No special handling needed.

    Why APs need it

    An AP is a vertex property. The non-root condition low[v] >= disc[u] works because it checks whether v’s subtree can escape above u via u’s ancestors. But the root has no ancestors — there’s no “above” to escape to. Instead, we check: if the root has two or more children in the DFS tree, those children are in separate subtrees, and the only path between them goes through the root. Remove the root → subtrees disconnect.

    Root is NOT an AP

    Root R has 1 DFS-tree child. Even though it has 2 neighbors, DFS only descends into 1 child (the other is visited via the back edge). Not an AP.

    Root IS an AP

    Root R has 2 DFS-tree children. A and B are in separate subtrees with no cross-connection. Remove R → A and B are disconnected. AP!

    Remember: Count children in the DFS tree, NOT the degree of the vertex in the original graph. A root with 10 neighbors but only 1 DFS-tree child is NOT an AP.

    Combined C++ Implementation

    The beauty of this shared framework: one DFS pass finds both bridges and articulation points simultaneously.

    #include <bits/stdc++.h>
using namespace std;

class BridgeAndAP {
    int n, timer = 0;
    vector<vector<int>> adj;
    vector<int> disc, low;
    vector<bool> visited, isAP;
    vector<pair<int,int>> bridges;

    void dfs(int u, int parent) {
        visited[u] = true;
        disc[u] = low[u] = timer++;
        int childCount = 0;

        for (int v : adj[u]) {
            if (!visited[v]) {
                childCount++;
                dfs(v, u);
                low[u] = min(low[u], low[v]);

                // BRIDGE: strict >
                if (low[v] > disc[u])
                    bridges.push_back({u, v});

                // AP (non-root): non-strict >=
                if (parent != -1 && low[v] >= disc[u])
                    isAP[u] = true;

            } else if (v != parent) {
                low[u] = min(low[u], disc[v]);
            }
        }

        // AP (root): 2+ DFS-tree children
        if (parent == -1 && childCount >= 2)
            isAP[u] = true;
    }

public:
    BridgeAndAP(int n) : n(n), adj(n), disc(n), low(n),
                          visited(n, false), isAP(n, false) {}

    void addEdge(int u, int v) {
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    void solve() {
        for (int i = 0; i < n; i++)
            if (!visited[i]) dfs(i, -1);

        cout << "Bridges: ";
        for (auto [u, v] : bridges)
            cout << "(" << u << "," << v << ") ";
        cout << "\nArticulation Points: ";
        for (int i = 0; i < n; i++)
            if (isAP[i]) cout << i << " ";
        cout << endl;
    }
};

int main() {
    BridgeAndAP g(7);
    g.addEdge(0,1); g.addEdge(1,2); g.addEdge(2,0);
    g.addEdge(2,3); g.addEdge(3,4); g.addEdge(4,5);
    g.addEdge(5,3); g.addEdge(4,6);
    g.solve();
    // Output:
    // Bridges: (2,3) (4,6)
    // Articulation Points: 2 3 4
}
    Three lines are the only difference between bridge and AP detection within the same DFS:
    Line 1: low[v] > disc[u] → bridge
    Line 2: low[v] >= disc[u] → AP (non-root)
    Line 3: childCount >= 2 → AP (root)

    Edge Cases & Gotchas

    1. Multigraphs (Parallel Edges)

    If two nodes are connected by multiple edges, the bridge condition changes. The edge is a bridge only if it’s the sole connection. Track edge IDs instead of just v != parent. APs are unaffected — vertex removal still removes all incident edges.

    2. Self-Loops

    Affect neither. A self-loop doesn’t help connectivity between different nodes.

    3. Leaf Nodes

    A leaf can never be an AP (removing it disconnects nothing). But the edge to a leaf is always a bridge.

    4. Are Bridge Endpoints Always APs?

    Common misconception: “If (u,v) is a bridge, then both u and v are APs.” FALSE!

    If v is a leaf (degree 1), then removing v doesn’t disconnect anything — v is not an AP. Only u is. In our example, edge 4-6 is a bridge, node 4 is an AP, but node 6 (leaf) is NOT an AP.

    Correct rule: If (u,v) is a bridge and v is NOT a leaf, then both u and v are APs. If v IS a leaf, only u is an AP (unless u is the only node, but then there’s no bridge).

    When to Use Which

    Use Bridge Detection When

    • Finding critical connections (LeetCode 1192)
    • Network cable redundancy analysis
    • Road network: which roads, if destroyed, split the city?
    • 2-edge-connected component decomposition

    Use AP Detection When

    • Finding critical routers/servers
    • Social network: whose departure fragments the group?
    • Biconnected component decomposition
    • Block-cut tree construction

    Use both together for comprehensive vulnerability analysis. The combined algorithm costs the same as running either one alone.

    Interview Tips & Common Mistakes

    Mistake 1: Swapping > and >=

    Using > for APs misses the low[v] == disc[u] case. Using >= for bridges falsely reports non-bridges.

    Mistake 2: Forgetting the root special case

    The non-root AP condition never fires for the root (since parent == -1 guard blocks it). You MUST add the separate childCount >= 2 check. Many interviewees forget this and miss root APs.

    Mistake 3: Counting degree instead of DFS-tree children

    The root check counts children in the DFS tree, not the vertex degree. A root with 5 neighbors might have only 1 DFS-tree child (the rest are visited via back edges).

    Mistake 4: Multigraph parent check

    With parallel edges, v != parent incorrectly skips back edges. Use edge indices instead: skip the specific edge you came from, not all edges to parent.

    Quick Reference (put this on a sticky note):
    low[v] > disc[u] → Bridge (edge u-v)
    low[v] >= disc[u] → AP (vertex u, if non-root)
    Root + 2 children → AP (vertex root)

    Summary

    • Same foundation: Both use DFS, disc[], low[] — computed identically.
    • Different objects: Bridges are edges; APs are vertices.
    • Different conditions: > (strict) for bridges vs >= (non-strict) for APs.
    • Extra rule: APs need a root special case; bridges don’t.
    • The equal case: When low[v] == disc[u], the edge survives (not a bridge) but the vertex is critical (is an AP).
    • Combine them: One DFS, three extra lines, both results. O(V+E).
