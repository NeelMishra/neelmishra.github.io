    Kosaraju’s Algorithm: Finding Strongly Connected Components

    April 2026 · 28 min read

    In a directed graph, some groups of vertices are so tightly connected that you can reach every vertex in the group from every other vertex in the same group. These groups are called Strongly Connected Components (SCCs). Kosaraju’s Algorithm finds all of them in linear time using two elegant passes of DFS. In this post we will dissect the algorithm, walk through a visual example step by step, build an interactive animation, write production-quality C++ code, and prove why it works.

    What Are Strongly Connected Components?

    A Strongly Connected Component of a directed graph is a maximal set of vertices such that there is a directed path from every vertex in the set to every other vertex in the set. “Maximal” means you cannot add any more vertex without breaking this property.

    Formal definition: A set S ⊆ V is an SCC if (1) for every pair u, v ∈ S there exist directed paths u →…→ v and v →…→ u, and (2) no proper superset of S also has this property.

    Consider this directed graph with 8 nodes. It contains three SCCs, highlighted with dashed outlines:

    SCC 1: {0, 1, 2} SCC 2: {3, 4} SCC 3: {5, 6, 7} Cross-SCC edge

    Within each colored group, you can follow directed edges to get from any node to any other node in the same group. For instance, in the green group: 0→1→2→0 forms a cycle, so every pair is mutually reachable. But node 0 can reach node 3 (via 1→3), while node 3 cannot reach node 0 — so they belong to different SCCs.

    Why Do SCCs Matter?

    SCCs appear everywhere:

    • Web connectivity: Groups of web pages where you can navigate from any page to any other page within the group via hyperlinks.
    • Social networks: Clusters of users who can all reach each other through directed follow/friendship chains.
    • Compiler optimization: Cycles in function call graphs or dependency graphs represent mutual recursion or circular dependencies.
    • 2-SAT problems: The satisfiability of a 2-SAT formula is determined by whether any variable and its negation belong to the same SCC in the implication graph.
    • Ecosystem modeling: In food webs, SCCs identify groups of species with mutual predator-prey relationships.

    Any time you need to decompose a directed graph into its “tightly connected clusters,” you need an SCC algorithm.

    Prerequisites

    Before diving into Kosaraju’s algorithm, make sure you’re comfortable with these concepts:

    DFS & Finish Times

    Depth-First Search explores a graph by going as deep as possible before backtracking. A critical concept for Kosaraju’s algorithm is the finish time of each node — the moment DFS finishes processing a node (i.e., all its descendants have been fully explored and DFS backtracks from it).

    Key insight: If you push each node onto a stack when it finishes (not when it’s first visited), the stack captures a topological-like ordering that is essential for the second pass of Kosaraju’s algorithm.

    In our example, running DFS from node 0 produces this finish order (earliest to latest): 3, 7, 6, 5, 4, 2, 1, 0. Pushed onto a stack, the top is 0 (latest finish) and the bottom is 3 (earliest finish).

    Graph Transpose

    The transpose (or reverse) of a directed graph G is a new graph GT with the same vertices but every edge reversed. If G has edge u→v, then GT has edge v→u.

    Key property: G and GT have exactly the same SCCs. Reversing all edges doesn’t break the mutual reachability within any SCC — if you could go from u to v and back in G, you can still do so in GT (the paths just swap directions).

    Condensation Graph (DAG of SCCs)

    If you collapse each SCC into a single “super-node,” the resulting graph is always a DAG (Directed Acyclic Graph). This is called the condensation of G. It captures the high-level structure of how SCCs connect to each other. We’ll explore this in detail later in the post.

    Kosaraju’s Algorithm — The Idea

    Kosaraju’s algorithm (also called the Kosaraju-Sharir algorithm) finds all SCCs using two passes of DFS. The key insight is beautifully simple:

    Core intuition: The first DFS pass computes a finishing order that guarantees the second DFS pass (on the transposed graph) will discover one complete SCC per DFS tree.

    Here’s why this works intuitively:

    1. Pass 1 discovers the “reachability frontier” of each node. Nodes that finish later in DFS can reach more of the graph. In particular, a node in an SCC that has outgoing cross-SCC edges will finish later than nodes in the “downstream” SCCs.
    2. Transposing the graph reverses the direction of cross-SCC edges. So if SCC A had an edge to SCC B in the original graph, now SCC B has an edge to SCC A in the transpose.
    3. Pass 2 processes nodes in reverse finish order (latest finish first). Starting DFS from the node with the latest finish time on the transposed graph, DFS can only reach nodes in the same SCC — because cross-SCC edges now point the wrong way (toward already-processed SCCs).

    Think of it like this: Pass 1 determines the “pecking order” of SCCs, and Pass 2 uses that order to peel off one SCC at a time, starting from the “source” SCCs of the condensation DAG.

    The Three Steps

    1

    DFS on G — Build Finish Stack

    Run DFS on the original graph. As each node finishes, push it onto a stack.

    2

    Transpose G → GT

    Reverse every edge in the graph.

    3

    DFS on GT in Stack Order

    Pop nodes from the stack. For each unvisited node, run DFS on GT — all nodes visited form one SCC.

    function kosaraju(V, adj):
    stack = []
    visited = [false] * V

    // Pass 1: DFS on original graph, record finish order
    for v = 0 to V-1:
    if not visited[v]:
      dfs1(v, adj, visited, stack)

    // Step 2: Build transposed graph
    adj_t = transpose(adj)

    // Pass 2: DFS on G^T in reverse finish order
    visited = [false] * V
    sccs = []
    while stack is not empty:
    v = stack.pop()
    if not visited[v]:
      component = []
      dfs2(v, adj_t, visited, component)
      sccs.append(component)
    return sccs

    Detailed Visual Walkthrough

    Let’s trace Kosaraju’s algorithm on our example graph with 8 nodes and 3 SCCs. We’ll track the DFS call stack, finish stack, and node states at each stage.

    PASS 1 — DFS on Original Graph

    Begin DFS from Node 0

    All nodes start unvisited. We call dfs(0). Node 0 is marked as visiting. Its first neighbor is 1, so we recurse into dfs(1). Node 1’s first neighbor is 2, so we recurse into dfs(2).

    Call stack: [dfs(0), dfs(1), dfs(2)]

    Finish stack: []

    DFS(2): Back Edge to 0, Then Explore 4

    Node 2 checks neighbor 0 — already visiting (back edge, confirms a cycle in SCC 1). Next neighbor is 4, so we recurse into dfs(4). Node 4’s first neighbor is 3, so we recurse into dfs(3).

    Call stack: [dfs(0), dfs(1), dfs(2), dfs(4), dfs(3)]

    Finish stack: []

    DFS(3): Dead End — First Node Finishes

    Node 3’s only neighbor is 4, which is already visiting. Dead end. Node 3 finishes and is pushed onto the finish stack. We return to dfs(4).

    Call stack: [dfs(0), dfs(1), dfs(2), dfs(4)]

    Finish stack: [3]
    Node 3 finished (finish time = 1)

    DFS(4) → DFS(5) → DFS(6) → DFS(7): Deep Chain

    Back in dfs(4), next neighbor is 5. We recurse: 4→5→6→7. Node 7’s only neighbor (5) is already visiting. Node 7 finishes, then 6, then 5. All three are pushed onto the finish stack.

    Call stack: [dfs(0), dfs(1), dfs(2), dfs(4)]

    Finish stack: [3, 7, 6, 5]
    Nodes 7, 6, 5 finished

    Unwinding: Nodes 4, 2, 1, 0 Finish

    Node 4 has no more unvisited neighbors — it finishes. Back in dfs(2): no more unvisited neighbors — finishes. Back in dfs(1): neighbor 3 is visited — finishes. Back in dfs(0): finishes. All nodes are done.

    Call stack: []

    Finish stack: [3, 7, 6, 5, 4, 2, 1, 0]  (top → 0)
    Pass 1 complete! All 8 nodes finished.
    STEP 2 — Transpose the Graph

    Reverse All Edges

    Every directed edge is reversed. Notice how the internal SCC edges still form cycles (reversing a cycle gives a cycle), but the cross-SCC edges now point in the opposite direction: 3→1 instead of 1→3, 4→2 instead of 2→4, and 5→4 instead of 4→5.

    Finish stack: [3, 7, 6, 5, 4, 2, 1, 0]  (top → 0)
    PASS 2 — DFS on Transposed Graph

    Pop 0 — DFS Finds SCC 1

    Pop 0 from the stack (highest finish time). Run DFS from 0 on GT: 0→2 (transposed edge from original 2→0), 2→1 (from 1→2), 1→0 (already visited). DFS finishes. All visited nodes form SCC 1 = {0, 1, 2}.

    Remaining stack: [3, 7, 6, 5, 4, 2, 1]  (pop 1, 2 — already visited)
    SCC 1 found: {0, 1, 2}

    Pop 4 — DFS Finds SCC 2

    Pop 1, 2 (already visited). Pop 4. DFS from 4 on GT: 4→3 (from 3→4), 3→1 (from 1→3, but 1 is already visited!). DFS stops. SCC 2 = {3, 4}.

    Remaining stack: [3, 7, 6, 5]  (pop 5 next)
    SCC 2 found: {3, 4}

    Pop 5 — DFS Finds SCC 3

    Pop 5. DFS from 5 on GT: 5→7 (from 7→5), 7→6 (from 6→7), 6→5 (from 5→6, already visited). SCC 3 = {5, 6, 7}. Pop remaining 6, 7, 3 — all visited. Done!

    Remaining stack: [] — empty!
    SCC 3 found: {5, 6, 7} — Algorithm complete!

    C++ Implementation

    Here is a clean, well-commented C++ implementation of Kosaraju’s algorithm:

    #include <bits/stdc++.h>
using namespace std;

class KosarajuSCC {
    int V;
    vector<vector<int>> adj;    // original graph
    vector<vector<int>> adj_t;  // transposed graph

    // Pass 1: DFS on original graph, push to stack on finish
    void dfs1(int u, vector<bool>& visited, stack<int>& stk) {
        visited[u] = true;
        for (int v : adj[u]) {
            if (!visited[v])
                dfs1(v, visited, stk);
        }
        stk.push(u);  // push when node finishes
    }

    // Pass 2: DFS on transposed graph, collect component
    void dfs2(int u, vector<bool>& visited, vector<int>& component) {
        visited[u] = true;
        component.push_back(u);
        for (int v : adj_t[u]) {
            if (!visited[v])
                dfs2(v, visited, component);
        }
    }

public:
    KosarajuSCC(int V) : V(V), adj(V), adj_t(V) {}

    void addEdge(int u, int v) {
        adj[u].push_back(v);
        adj_t[v].push_back(u);  // build transpose simultaneously
    }

    vector<vector<int>> findSCCs() {
        // Step 1: Fill the stack with finish order
        vector<bool> visited(V, false);
        stack<int> stk;
        for (int i = 0; i < V; i++) {
            if (!visited[i])
                dfs1(i, visited, stk);
        }

        // Step 2: Process nodes in reverse finish order on G^T
        fill(visited.begin(), visited.end(), false);
        vector<vector<int>> sccs;

        while (!stk.empty()) {
            int u = stk.top();
            stk.pop();
            if (!visited[u]) {
                vector<int> component;
                dfs2(u, visited, component);
                sccs.push_back(component);
            }
        }
        return sccs;
    }
};

int main() {
    KosarajuSCC g(8);

    // SCC 1: {0, 1, 2}
    g.addEdge(0, 1);
    g.addEdge(1, 2);
    g.addEdge(2, 0);

    // SCC 2: {3, 4}
    g.addEdge(3, 4);
    g.addEdge(4, 3);

    // SCC 3: {5, 6, 7}
    g.addEdge(5, 6);
    g.addEdge(6, 7);
    g.addEdge(7, 5);

    // Cross-SCC edges
    g.addEdge(1, 3);
    g.addEdge(2, 4);
    g.addEdge(4, 5);

    auto sccs = g.findSCCs();
    cout << "Number of SCCs: " << sccs.size() << endl;
    for (int i = 0; i < sccs.size(); i++) {
        cout << "SCC " << i+1 << ": ";
        for (int v : sccs[i]) cout << v << " ";
        cout << endl;
    }
    return 0;
}
    Implementation note: We build the transposed graph at the same time as the original graph in addEdge(). This avoids a separate O(V+E) transpose pass and keeps the code clean.

    Complexity Analysis

    Time Complexity: O(V + E)

    Let’s break down each step:

    • Pass 1 (DFS on G): Visits every vertex and edge exactly once — O(V + E).
    • Transpose: If built simultaneously with the original graph (as in our implementation), this is O(1) extra. If built separately, it’s O(V + E).
    • Pass 2 (DFS on GT): Again visits every vertex and edge exactly once — O(V + E).

    Total: O(V + E) — linear in the size of the graph. This is optimal because you need to examine every vertex and edge at least once to determine SCCs.

    Space Complexity: O(V + E)

    • Adjacency lists for G and GT: O(V + E) each.
    • Visited arrays: O(V).
    • Finish stack: O(V).
    • Recursion stack (worst case): O(V).

    Total: O(V + E).

    Comparison with Tarjan’s Algorithm

    Kosaraju’s

    • Two DFS passes
    • Needs transposed graph (2× memory for edges)
    • Conceptually simpler
    • Easier to implement correctly
    • Naturally produces topological order of SCCs

    Tarjan’s

    • Single DFS pass
    • No transposed graph needed
    • Uses low-link values & stack
    • Slightly more complex logic
    • Better for very large graphs (less memory)

    Both algorithms are O(V + E). In practice, Kosaraju’s is often preferred for interviews and competitive programming due to its simplicity, while Tarjan’s is preferred in memory-constrained environments.

    Why Does It Work? (Correctness Proof Intuition)

    The correctness of Kosaraju’s algorithm rests on a key lemma about finish times and the structure of the condensation DAG.

    Key Lemma

    Lemma: If there is an edge from SCC C to SCC C′ in the condensation DAG, then the maximum finish time in C is greater than the maximum finish time in C′.

    Proof sketch: Consider an edge u→v where u∈C and v∈C′. During DFS Pass 1, there are two cases:

    1. DFS reaches C before C′: DFS from some node in C will eventually cross into C′ via u→v. All of C′ will finish before the DFS in C finishes (since we must backtrack through C after C′ is done). So the last node in C finishes after the last node in C′.
    2. DFS reaches C′ before C: Since C′ has no edge back to C (the condensation is a DAG), DFS on C′ cannot enter C. All of C′ finishes before C is even started. Again, max finish time of C > max finish time of C′.

    Why Reverse Finish Order on GT Works

    By the lemma, the SCC with the globally highest finish time is a source in the condensation DAG (no SCC has an edge pointing to it). When we start DFS from a node in this source SCC on GT:

    • Within the SCC, all nodes are mutually reachable even in GT (since SCCs are the same in G and GT).
    • Cross-SCC edges in GT point into this source SCC, not out of it. So DFS cannot escape to other SCCs.

    After this SCC is identified and marked, the next highest-finish-time unvisited node belongs to the next source SCC (in the condensation DAG minus the already-found SCCs). The same argument applies recursively.

    This is why the algorithm peels off SCCs one at a time in reverse topological order of the condensation DAG. Each DFS on GT is perfectly contained within exactly one SCC.

    The Condensation Graph

    The condensation of a directed graph G is formed by collapsing each SCC into a single super-node and keeping only the cross-SCC edges (removing duplicates). The result is always a DAG.

    For our example graph, the condensation looks like this:

    Notice the condensation is a simple chain: SCC 1 → SCC 2 → SCC 3. There are no cycles (by definition of a condensation). The cross-SCC edges 1→3, 2→4 both map to the single edge SCC 1→SCC 2, and edge 4→5 maps to SCC 2→SCC 3.

    Why Is the Condensation Useful?

    • Simplification: Reduces a complex directed graph to a manageable DAG while preserving high-level connectivity.
    • Reachability: Answering “can u reach v?” reduces to checking reachability in the condensation DAG (after identifying which SCCs u and v belong to).
    • Topological ordering: The condensation can be topologically sorted, enabling efficient processing of the original graph’s macro-structure.
    • 2-SAT: The condensation is directly used to extract the solution once SCCs are found.

    Applications

    2-SAT Solver

    Build the implication graph for a 2-SAT formula. Find SCCs. If any variable and its negation are in the same SCC, the formula is unsatisfiable. Otherwise, extract a valid assignment from the condensation’s topological order.

    Social Network Analysis

    In a directed social graph (follows, not friendships), SCCs represent tightly-knit communities where every member can reach every other member through a chain of follows.

    Compiler Optimization

    In call graphs, SCCs represent groups of mutually recursive functions. Compilers can optimize these groups together (e.g., inlining decisions, register allocation).

    Web Crawling & PageRank

    The “bow-tie structure” of the web is based on SCCs. The largest SCC of the web graph is the “core” of the web, reachable from itself. PageRank and similar algorithms leverage SCC structure.

    More Applications

    • Model checking: Verifying temporal logic properties of state machines requires SCC decomposition.
    • Circuit analysis: Finding feedback loops in digital circuits.
    • Biological networks: Identifying feedback cycles in gene regulatory networks or metabolic pathways.
    • Database dependency analysis: Detecting circular foreign key dependencies.

    Common Interview Problems

    Here are popular problems that use SCC concepts:

    Critical Connections in a Network (LeetCode 1192)

    While this problem is more about bridges than SCCs directly, understanding SCC decomposition helps identify which edges, if removed, would disconnect parts of the graph. SCCs help identify the “redundant” edges within strongly connected subgraphs.

    Satisfiability of Equality & Inequality Equations

    In 2-SAT variants, finding SCCs determines whether a consistent assignment exists. Given implications like “if x then y,” build the implication graph and check SCCs for contradictions.

    Minimum Number of Edges to Make Graph Strongly Connected

    Find SCCs, build the condensation DAG, count sources (in-degree 0) and sinks (out-degree 0). The answer is max(sources, sinks), since you need to add edges to eliminate all sources and sinks in the condensation.

    Redundant Connection II (LeetCode 685)

    Find the edge that, when removed, makes a directed graph into a rooted tree. SCC detection helps identify the problematic cycle.

    Summary

    Let’s recap the key takeaways:

    1. Strongly Connected Components are maximal sets of vertices with mutual reachability in directed graphs.
    2. Kosaraju’s Algorithm finds all SCCs in O(V + E) using two DFS passes:
      • Pass 1: DFS on G, build finish-order stack
      • Transpose G to get GT
      • Pass 2: DFS on GT in reverse finish order
    3. The algorithm works because processing in reverse finish order on GT confines each DFS to exactly one SCC.
    4. The condensation graph (DAG of SCCs) captures the macro-structure and is useful for reachability queries, 2-SAT, and more.
    5. Kosaraju’s is simpler to implement than Tarjan’s but requires extra space for the transposed graph.
    Practice tip: Implement Kosaraju’s from scratch at least 3 times before a contest. The two-DFS pattern becomes second nature with practice. Focus on understanding why finish order matters — that insight transfers to many other graph problems.
