    Concatenate Array With Reverse

    Weekly Contest 501 · Q1 LeetCode #3925 Easy · 3 pts View on LeetCode ↗
    Topics arrays simulation
    Tags array-construction two-pointer single-pass ad-hoc mirror

    Problem Statement

    You are given an integer array nums of length n.

    Construct a new array ans of length 2 * n such that the first n elements are the same as nums, and the next n elements are the elements of nums in reverse order.

    Formally, for 0 <= i <= n - 1:

    • ans[i] = nums[i]
    • ans[i + n] = nums[n - i - 1]

    Return the integer array ans.

    Example 1:

    Input:  nums = [1,2,3]
Output: [1,2,3,3,2,1]

    Example 2:

    Input:  nums = [1]
Output: [1,1]
Explanation: The array remains the same when reversed.

    Constraints:

    • 1 <= nums.length <= 100
    • 1 <= nums[i] <= 100

    Source: LeetCode · Statement © LeetCode.

    Intuition

    The output is fully specified by the input — no decision-making, no search. Position $i$ in the second half is just $\mathrm{nums}[n-1-i]$, the mirror of position $i$ in the first half. So the entire problem is "allocate $2n$ slots and fill them in one pass."

    The trick worth noticing: each iteration $i$ writes both a forward slot ($\mathrm{ans}[i]$) and a mirror slot ($\mathrm{ans}[i + n]$). One loop, $n$ iterations, no temporary buffers. That's optimal — you can't construct $2n$ outputs in fewer than $\Theta(n)$ steps.

    Brute Force

    Build the prefix by copying nums, then build a reversed temporary and append it. Same asymptotic cost as the optimal but uses an extra buffer and an extra library call:

    class Solution {
public:
    vector<int> concatWithReverse(vector<int>& nums) {
        vector<int> ans = nums;
        vector<int> rev = nums;
        reverse(rev.begin(), rev.end());
        ans.insert(ans.end(), rev.begin(), rev.end());
        return ans;
    }
};

    Time $O(n)$, space $O(n)$ extra (the temporary reversed copy) on top of the output.

    Optimal Approach

    Preallocate ans at the final size, then in a single pass write both halves simultaneously:

    class Solution {
public:
    vector<int> concatWithReverse(vector<int>& nums) {
        int n = nums.size();
        vector<int> ans(2 * n);
        for (int i = 0; i < n; ++i) {
            ans[i]     = nums[i];
            ans[i + n] = nums[n - 1 - i];
        }
        return ans;
    }
};

    Time $O(n)$, space $O(n)$ (the output itself, no extra buffer).

    Why preallocate? Constructing with vector<int> ans(2 * n) performs a single zero-initialized allocation. Repeated push_back would trigger logarithmically many reallocations — correct, but slower in practice.

    Dry Run

    Take nums = [1, 2, 3] (so $n = 3$, output length $6$). One pass:

    ians[i] = nums[i]ans[i + 3] = nums[2 - i]ans so far
    01nums[2] = 3[1, _, _, 3, _, _]
    12nums[1] = 2[1, 2, _, 3, 2, _]
    23nums[0] = 1[1, 2, 3, 3, 2, 1]

    Final result: [1, 2, 3, 3, 2, 1].

    For nums = [1] ($n = 1$): the loop runs once with $i = 0$, writing ans[0] = 1 and ans[1] = nums[0] = 1. Output [1, 1].

    Complexity Recap

    ApproachTimeSpaceNotes
    Copy + reverse + appendO(n)O(n) extraUses a temporary reversed buffer plus a vector::insert call.
    Preallocate & two-write loopO(n)O(1) extraSingle pass, single allocation, writes both halves per iteration.

    Edge Cases & Pitfalls

    • $n = 1$: the loop runs once. Output is [nums[0], nums[0]] — a palindrome of length 2.
    • Index off-by-one: the mirror index is n - 1 - i, not n - i. The latter is out of bounds when $i = 0$.
    • Output length: always exactly $2n$. vector<int> ans(2 * n) is the cleanest spelling; avoid ans.reserve(2*n) + push_back which inflates the size dynamically.
    • Aliasing: we read from nums and write into ans (distinct objects), so no overlap concerns — even an in-place trick over a single buffer would be sound because the read at index n - 1 - i happens before any write to that index in the doubled buffer.
    • Palindromic nums: the result is also a palindrome of length $2n$. The algorithm doesn’t care — the rule is structural, not value-based.
