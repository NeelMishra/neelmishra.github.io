    Minimum Cost to Buy Apples II

    Weekly Contest 501 · Q4 LeetCode #3928 Hard · 6 pts View on LeetCode ↗
    Topics graphs shortest-path dijkstra
    Tags dijkstra multi-source-shortest-path two-metric-graph undirected forward-return-decomposition priority-queue long-long-arithmetic overflow-safety

    Problem Statement

    You are given an integer n and an array prices of length n, where prices[i] is the price of apples at shop i.

    You are also given a 2D array roads, where roads[i] = [u, v, cost, tax] represents a bidirectional road:

    • u and v are the shops connected by the road.
    • cost is the cost of traveling the road without carrying apples.
    • tax is the multiplier applied to cost when traveling with apples.

    For each shop i you can either:

    • Buy locally for prices[i], or
    • Travel empty to any shop j, buy for prices[j], and return to i paying cost × tax on each road used for the return trip. The forward and return paths may differ.

    Return an integer array ans where ans[i] is the minimum total cost to buy apples starting from shop i.

    Example 1:

    Input:  n = 2, prices = [8,3], roads = [[0,1,1,2]]
Output: [6,3]
Explanation: From shop 0 buy at shop 1: forward 1, return 1*2 = 2, plus price 3 → 6.
From shop 1 buying locally for 3 is cheapest.

    Example 2:

    Input:  n = 3, prices = [9,4,6], roads = [[0,1,1,3],[1,2,4,2]]
Output: [8,4,6]

    Example 3:

    Input:  n = 3, prices = [10,11,1], roads = [[0,2,1,3],[1,2,3,4],[0,1,5,2]]
Output: [5,11,1]

    Constraints:

    • 1 <= n <= 1000; prices.length == n; 1 <= prices[i] <= 109
    • 0 <= roads.length <= min(n · (n - 1) / 2, 2000)
    • 0 <= u, v <= n - 1, u != v, no repeated edges
    • 1 <= cost <= 109, 1 <= tax <= 100

    Intuition

    Decompose by buyer shop. The decision for each starting shop $i$ is: pick some $j$ (possibly $j = i$), pay prices[j] there, traverse some path $i \rightsquigarrow j$ empty, and some (possibly different) path $j \rightsquigarrow i$ carrying apples. The forward leg uses raw cost; the return leg uses cost × tax. For each leg independently we want the cheapest possible path — that’s just a shortest-path problem under the relevant edge weight.

    Two key simplifications:

    1. Two independent metrics. Build two graphs on the same vertex set: graph $G_c$ with edge weight cost, graph $G_t$ with edge weight cost × tax. Then:

      $$\mathrm{ans}[i] = \min_{0 \le j < n} \Big( d_c(i, j) + d_t(i, j) + \mathrm{prices}[j] \Big)$$

      where $d_c$ and $d_t$ are the shortest-path distances in the two graphs (and $d_c(i,i) = d_t(i,i) = 0$, recovering the "buy locally" option).
    2. Forward and return are independent. No constraint ties the two paths together, so optimizing them separately is exact — not a relaxation.

    Why undirected paths suffice. The graph is undirected; $d_c(j, i) = d_c(i, j)$ and similarly for $d_t$. So running Dijkstra from each $i$ gives all distances from $i$, which equals distances to $i$ for the return leg.

    Complexity budget. $n \le 1000$, $|E| \le 2000$. Two Dijkstras per source × $n$ sources = $2n$ Dijkstras, each $O((n + |E|) \log n) \approx 3 \times 10^4$. Total $\sim 6 \times 10^7$ priority-queue ops — well under a typical 1–2 s limit. Floyd–Warshall would be $O(n^3) = 10^9$ — too slow.

    Brute Force

    The naive textbook all-pairs shortest path: Floyd–Warshall on both graphs, then a double loop to combine:

    class Solution {
public:
    vector<int> minCost(int n, vector<int>& prices,
                         vector<vector<int>>& roads) {
        const long long INF = LLONG_MAX / 4;
        vector<vector<long long>> dc(n, vector<long long>(n, INF));
        vector<vector<long long>> dt(n, vector<long long>(n, INF));
        for (int i = 0; i < n; ++i) dc[i][i] = dt[i][i] = 0;
        for (auto& r : roads) {
            int u = r[0], v = r[1];
            long long c = r[2], tx = r[3];
            dc[u][v] = dc[v][u] = min(dc[u][v], c);
            dt[u][v] = dt[v][u] = min(dt[u][v], c * tx);
        }
        for (int k = 0; k < n; ++k)
            for (int i = 0; i < n; ++i)
                for (int j = 0; j < n; ++j) {
                    if (dc[i][k] + dc[k][j] < dc[i][j]) dc[i][j] = dc[i][k] + dc[k][j];
                    if (dt[i][k] + dt[k][j] < dt[i][j]) dt[i][j] = dt[i][k] + dt[k][j];
                }
        vector<int> ans(n);
        for (int i = 0; i < n; ++i) {
            long long best = prices[i];
            for (int j = 0; j < n; ++j) {
                if (dc[i][j] >= INF || dt[i][j] >= INF) continue;
                best = min(best, dc[i][j] + dt[i][j] + (long long)prices[j]);
            }
            ans[i] = (int)best;
        }
        return ans;
    }
};

    Time $O(n^3)$, space $O(n^2)$. For $n = 1000$ that’s $10^9$ inner-loop ops — tight at best, more often TLE in C++. Conceptually clean, operationally too slow.

    Optimal Approach

    Replace Floyd–Warshall with $n$ runs of Dijkstra per metric. Build two adjacency lists, then for each source $i$ run Dijkstra in $G_c$ and $G_t$ separately and combine over all $j$:

    class Solution {
public:
    vector<int> minCost(int n, vector<int>& prices,
                         vector<vector<int>>& roads) {
        vector<vector<pair<int, long long>>> adjC(n), adjT(n);
        for (auto& r : roads) {
            int u = r[0], v = r[1];
            long long c = r[2], tx = r[3];
            adjC[u].push_back({v, c});
            adjC[v].push_back({u, c});
            adjT[u].push_back({v, c * tx});
            adjT[v].push_back({u, c * tx});
        }
        const long long INF = LLONG_MAX / 4;
        auto dijkstra = [&](const auto& adj, int src) {
            vector<long long> d(n, INF);
            d[src] = 0;
            priority_queue<pair<long long, int>,
                           vector<pair<long long, int>>,
                           greater<>> pq;
            pq.push({0, src});
            while (!pq.empty()) {
                auto [du, u] = pq.top(); pq.pop();
                if (du > d[u]) continue;
                for (auto& [v, w] : adj[u]) {
                    long long nd = du + w;
                    if (nd < d[v]) { d[v] = nd; pq.push({nd, v}); }
                }
            }
            return d;
        };
        vector<int> ans(n);
        for (int i = 0; i < n; ++i) {
            auto dc = dijkstra(adjC, i);
            auto dt = dijkstra(adjT, i);
            long long best = prices[i];
            for (int j = 0; j < n; ++j) {
                if (dc[j] >= INF || dt[j] >= INF) continue;
                long long cand = dc[j] + dt[j] + (long long)prices[j];
                if (cand < best) best = cand;
            }
            ans[i] = (int)best;
        }
        return ans;
    }
};

    Time $O(n \cdot (n + |E|) \log n)$, space $O(n + |E|)$ per Dijkstra plus the adjacency lists.

    Why the answer fits in int. $\mathrm{ans}[i] \le \mathrm{prices}[i] \le 10^9$ because "buy locally" is always a feasible choice that we initialize best with. So even though intermediate sums (forward + return) can reach $\sim 4 \times 10^{14}$, the winning candidate is always $\le 10^9$ and the cast back to int is safe. The intermediate arithmetic must still be long long — that’s where the overflow trap is.

    Lazy decrease-key. The if (du > d[u]) continue; skips stale entries pulled from the heap after a better $d[u]$ was committed. This keeps the heap correct without an explicit decrease-key operation.

    Dry Run

    Take Example 2: n = 3, prices = [9,4,6], roads = [[0,1,1,3],[1,2,4,2]].

    Build adjacencies. $G_c$ edges: $0\!-\!1$ (weight 1), $1\!-\!2$ (weight 4). $G_t$ edges: $0\!-\!1$ (weight $1 \cdot 3 = 3$), $1\!-\!2$ (weight $4 \cdot 2 = 8$).

    source $i$$d_c(i,\cdot)$$d_t(i,\cdot)$combine $d_c + d_t + prices[j]$$\mathrm{ans}[i]$
    0[0, 1, 5][0, 3, 11]j=0: 0+0+9=9, j=1: 1+3+4=8, j=2: 5+11+6=228
    1[1, 0, 4][3, 0, 8]j=0: 1+3+9=13, j=1: 0+0+4=4, j=2: 4+8+6=184
    2[5, 4, 0][11, 8, 0]j=0: 5+11+9=25, j=1: 4+8+4=16, j=2: 0+0+6=66

    Result: [8, 4, 6] — matches the expected output.

    Sanity check on Example 3: from shop 0 the cheapest route is forward via the direct edge $0\!-\!2$ (cost 1) and return via the same edge (cost $1 \cdot 3 = 3$), buying at shop 2 for 1 → total $1 + 3 + 1 = 5$. That beats prices[0] = 10 and beats routing through shop 1.

    Complexity Recap

    ApproachTimeSpaceNotes
    Floyd–Warshall on both metricsO(n3)O(n2)$10^9$ updates for $n = 1000$ — usually TLE.
    $2n$ Dijkstras (priority queue)O(n(n + E) log n)O(n + E)$\sim 6 \times 10^7$ ops — comfortably AC.
    $2n$ Dijkstras with array-based keyO(n · n2)O(n + E)Slightly worse constants; still passes for these bounds.

    Edge Cases & Pitfalls

    • Disconnected graph: shops in different components can’t trade. Use a sentinel INF and skip any $j$ with $d_c(i,j) = \infty$ or $d_t(i,j) = \infty$. The "buy locally" baseline keeps ans[i] well-defined.
    • Empty roads: every shop is isolated. Each Dijkstra returns just {src: 0, others: INF}; the only viable $j$ is $i$ itself, so ans[i] = prices[i].
    • Overflow on weight multiplication: a single $\mathrm{cost} \cdot \mathrm{tax}$ is up to $10^9 \cdot 100 = 10^{11}$ — already past int. Multiply as long long before storing in the adjacency list.
    • Overflow on accumulated distance: with up to $|E|$ edges per path, $d_t$ can climb to $\sim 2000 \cdot 10^{11} = 2 \cdot 10^{14}$. Use long long everywhere; LLONG_MAX / 4 as INF leaves plenty of headroom for one extra addition without wraparound.
    • $j = i$ case: $d_c(i,i) = d_t(i,i) = 0$, so the combine term is exactly prices[i] — the "buy locally" branch is captured automatically and the explicit initial best = prices[i] is just a no-op safety net.
    • Multiple edges between the same pair: the constraints forbid it ("no repeated edges"), but if they didn’t, Dijkstra would still be correct — the adjacency list would just hold both and relaxations would pick the cheaper.
    • Self-loops: forbidden ($u \neq v$). Self-loops with positive weight would be harmless to Dijkstra but pollute the graph — the input guarantee makes this moot.
    • Stale heap entries: standard priority_queue doesn’t support decrease-key. Skipping with du > d[u] is the canonical fix; each vertex is finalized at most $O(\deg + 1)$ times in the heap.
    • Tax = 1: $G_c$ and $G_t$ have identical edge weights and identical distance maps, but the algorithm doesn’t need to special-case this — you can either run Dijkstra twice or reuse the result, only a constant-factor saving.
    • Why not multi-source Dijkstra over a virtual "shop" node? Tempting, but the two metrics ($d_c$ for forward, $d_t$ for return) prevent a clean reduction into a single graph — you’d need a product/layered graph, which adds complexity without saving asymptotics on this $n$.

public:
    vector<int> minCost(int n, vector<int>& prices,
                         vector<vector<int>>& roads) {
        vector<vector<pair<int, long long>>> adjC(n), adjT(n);
        for (auto& r : roads) {
            int u = r[0], v = r[1];
            long long c = r[2], tx = r[3];
            adjC[u].push_back({v, c});
            adjC[v].push_back({u, c});
            adjT[u].push_back({v, c * tx});
            adjT[v].push_back({u, c * tx});
        }
        const long long INF = LLONG_MAX / 4;
        auto dijkstra = [&](const auto& adj, int src) {
