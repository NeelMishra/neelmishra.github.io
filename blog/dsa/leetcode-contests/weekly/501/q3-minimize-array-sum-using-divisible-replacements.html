    Minimize Array Sum Using Divisible Replacements

    Weekly Contest 501 · Q3 LeetCode #3927 Medium · 5 pts View on LeetCode ↗
    Topics arrays math number-theory
    Tags divisibility divisor-enumeration transitivity-argument greedy smallest-divisor-in-set long-long-sum presence-array

    Problem Statement

    You are given an integer array nums. You can perform the following operation any number of times:

    • Choose two indices a and b such that nums[a] % nums[b] == 0.
    • Replace nums[a] with nums[b].

    Return the minimum possible sum of the array after performing any number of operations.

    Example 1:

    Input:  nums = [3,6,2]
Output: 7
Explanation: 6 % 2 == 0, replace nums[1] = 6 with nums[2] = 2.
Array: [3, 2, 2]. Sum = 7.

    Example 2:

    Input:  nums = [4,2,8,3]
Output: 9
Explanation: Both 4 and 8 are divisible by 2, replace them with 2.
Array: [2, 2, 2, 3]. Sum = 9.

    Example 3:

    Input:  nums = [7,5,9]
Output: 21
Explanation: No pair satisfies the divisibility condition. Sum stays 21.

    Constraints:

    • 1 <= nums.length <= 105
    • 1 <= nums[i] <= 105

    Source: LeetCode · Statement © LeetCode.

    Intuition

    Key insight — the value set never grows. Every operation replaces nums[a] with the current value at some other index. So the multiset of values present in nums at any moment is a subset of the values that were ever present, which is a subset of the original values. Call this initial set $S$.

    Each position chooses independently. Two positions don’t interfere with each other: once you decide what final value each $\mathrm{nums}[i]$ should take, you can apply the necessary single replacement (using any index b currently holding the target value — the originals are still in the array). So the problem decomposes per index: what is the smallest value that index $i$ can hold?

    Transitivity collapses chains. A chain of replacements $x \to v_1 \to v_2 \to \cdots \to y$ requires $x \% v_1 = 0$, $v_1 \% v_2 = 0$, …, $v_{k-1} \% y = 0$. Divisibility composes: $v_1 \mid x$ and $v_2 \mid v_1 \Rightarrow v_2 \mid x$. By induction, $y \mid x$. So any chain from $x$ ending at $y$ implies $x \% y = 0$ directly — the intermediate values are unnecessary. Plus, since the originals stay in the array (we only modify other indices), $y \in S$ is a sufficient and necessary condition.

    Conclusion: the minimum value each $\mathrm{nums}[i]$ can become is

    $$\mathrm{best}(x) \ = \ \min\{\, y \in S \ : \ x \bmod y = 0 \,\}$$

    and the answer is $\sum_{i} \mathrm{best}(\mathrm{nums}[i])$.

    Since $\max \mathrm{nums}[i] \le 10^5$, divisors of $x$ can be enumerated in $O(\sqrt{x})$, and presence in $S$ tested in $O(1)$ with a boolean array. Total $O(n \sqrt{\max})$, about $3 \times 10^7$ for the constraints — safely AC.

    Brute Force

    For each index $i$ scan every other index $j$, keep the smallest $\mathrm{nums}[j]$ that divides $\mathrm{nums}[i]$:

    class Solution {
public:
    long long minArraySum(vector<int>& nums) {
        int n = nums.size();
        long long sum = 0;
        for (int i = 0; i < n; ++i) {
            int best = nums[i];
            for (int j = 0; j < n; ++j)
                if (nums[i] % nums[j] == 0 && nums[j] < best)
                    best = nums[j];
            sum += best;
        }
        return sum;
    }
};

    Time $O(n^2)$, space $O(1)$. For $n = 10^5$ that’s $10^{10}$ modulus operations — TLE.

    Optimal Approach

    Flip the inner loop: instead of scanning indices, enumerate divisors of each $x$. Pre-mark which values appear in the array using a presence array has[] indexed by value (cheap since $\mathrm{nums}[i] \le 10^5$). For each $x$, walk $d = 1, 2, \ldots, \lfloor\sqrt{x}\rfloor$ and check both $d$ and $x/d$ as candidates:

    class Solution {
public:
    long long minArraySum(vector<int>& nums) {
        int maxV = 0;
        for (int x : nums) maxV = max(maxV, x);

        vector<bool> has(maxV + 1, false);
        for (int x : nums) has[x] = true;

        long long sum = 0;
        for (int x : nums) {
            int best = x;             // x itself is always a divisor of x
            for (int d = 1; (long long)d * d <= x; ++d) {
                if (x % d == 0) {
                    if (has[d]      && d      < best) best = d;
                    int q = x / d;
                    if (has[q]      && q      < best) best = q;
                }
            }
            sum += best;
        }
        return sum;
    }
};

    Time $O(n \sqrt{\max\,\mathrm{nums}})$, space $O(\max\,\mathrm{nums})$ for the presence array. For $n = 10^5$ and $\max = 10^5$, that’s $\sim 3 \times 10^7$ — comfortably under the limit.

    Why best = x as the initial sentinel? Every $x$ divides itself, and every $x$ is in $S$ by construction, so $x$ is always a feasible answer. We only update if we find a strictly smaller divisor present in $S$.

    Two-for-one divisor enumeration. Divisors of $x$ come in pairs $(d, x/d)$ with $d \le \sqrt{x}$, so a single sweep $d = 1 \ldots \sqrt{x}$ covers all of them. When $x$ is a perfect square the pair degenerates to $(\sqrt{x}, \sqrt{x})$; checking both branches still gives the right answer because the comparison is monotonic.

    Dry Run

    Take nums = [4, 2, 8, 3] (Example 2). Set $S = \{2, 3, 4, 8\}$, has[2] = has[3] = has[4] = has[8] = true. Process each element:

    xdivisors scanned (d, x/d)divisors ∩ Sbest
    4(1, 4), (2, 2){2, 4}2
    2(1, 2){2}2
    8(1, 8), (2, 4){2, 4, 8}2
    3(1, 3){3}3

    Sum = $2 + 2 + 2 + 3 = 9$. Matches the expected output.

    For [7, 5, 9]: divisors of 7 are {1, 7}, intersect $S = \{5,7,9\}$ → {7}; same for 5 and 9. No element shrinks, sum = $21$.

    Complexity Recap

    ApproachTimeSpaceNotes
    Brute pairwise scanO(n2)O(1)$\sim 10^{10}$ modulus ops for $n = 10^5$ — TLE.
    Sieve over multiplesO(maxV log maxV)O(maxV)For each $y$ in $S$, mark its multiples. Alternative, similar order of magnitude.
    Divisor enumeration + presence arrayO(n · √maxV)O(maxV)$\sim 3 \times 10^7$ ops — the implementation shown.

    Edge Cases & Pitfalls

    • $1 \in \mathrm{nums}$: 1 divides every integer, and 1 is in $S$, so every $x$ collapses to 1. The answer is exactly $n$.
    • All values pairwise coprime: the only divisor of $x$ in $S$ is $x$ itself. Nothing shrinks; answer equals the original sum.
    • Sum overflow: $n \cdot \max \le 10^5 \cdot 10^5 = 10^{10}$ exceeds int. The return type is long long — honor it.
    • "Two indices" pedantry: some readings of "two indices a and b" forbid $a = b$. It doesn’t matter: even if forbidden, since the smallest divisor $y$ of $x$ that’s in $S$ comes from some index $b \ne a$ (because either $y < x$, in which case $y$ sits at a different position by definition, or $y = x$ which is the no-op case). The algorithm produces the same sum either way.
    • Perfect squares: when $x = k^2$, the divisor loop visits $d = k$ with $x/d = k$. We may update best twice with the same value — harmless.
    • Initial best = x: guarantees the inner loop always has a fallback. Without it (e.g. best = INT_MAX), you might emit an out-of-set value if you mis-handle the divisor loop bounds.
    • Off-by-one in loop bound: use (long long)d * d <= x to avoid overflow when $d \approx \sqrt{10^5} \approx 316$. (Even int works for $x \le 10^5$, but the cast is defensive and free.)
    • Presence array vs hash set: a vector<bool> over $[0, \max]$ is cache-friendlier than unordered_set<int> here, since indices are dense and bounded by $10^5$.
