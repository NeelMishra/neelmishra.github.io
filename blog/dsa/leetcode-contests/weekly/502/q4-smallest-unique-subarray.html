    Smallest Unique Subarray

    Weekly Contest 502 · Q4 LeetCode #? Hard · 6 pts View on LeetCode ↗
    Topics strings binary-search hashing
    Tags rolling-hash double-hash polynomial-hash monotonicity binary-search subarray uniqueness string-hashing

    Problem Statement

    You are given an integer array nums.

    Find the minimum length of a subarray that is not identical to any other subarray in nums.

    A subarray is a contiguous non-empty sequence of elements within an array. Two subarrays are considered identical if they have the same length and the same elements in corresponding positions.

    Example 1:

    Input:  nums = [3,3,3]
Output: 3
Explanation: [3] appears 3 times, [3,3] appears 2 times, [3,3,3] appears once.

    Example 2:

    Input:  nums = [2,1,2,3,3]
Output: 1
Explanation: [1] appears exactly once.

    Example 3:

    Input:  nums = [1,1,2,2,1]
Output: 2
Explanation: All four length-2 subarrays are distinct, so some length-2 is unique.

    Constraints:

    • 1 <= nums.length <= 105
    • 1 <= nums[i] <= 105

    Source: LeetCode · Statement © LeetCode.

    Intuition

    Key insight — uniqueness is upward-closed in length. If some length-$L$ subarray $W$ is unique, then for any window $W'$ of length $L + 1$ that contains $W$, $W'$ is also unique: any duplicate $W''$ of $W'$ would have its corresponding length-$L$ sub-window equal to $W$, contradicting uniqueness. So if length $L$ "works", every length $\ge L$ also works.

    This monotonicity is exactly the binary-search predicate. Search the smallest feasible $L$ in $[1, n]$. The upper bound is trivially feasible: the single length-$n$ subarray is unique.

    For the per-$L$ check we need to know whether any length-$L$ window appears exactly once. Storing each window as a vector<int> in a map is too slow ($O(n L)$ key comparisons). The fix is polynomial rolling hash with $O(n)$ preprocessing and $O(1)$ per window. To make hash collisions astronomically unlikely on adversarial inputs we use double hashing: two independent base/modulus pairs, combined into a 64-bit key.

    Total complexity: $O(n \log n)$ check loop · $O(1)$ per query → $\approx 1.7 \times 10^6$ ops for $n = 10^5$.

    Brute Force

    Try every length $L$ from $1$ upward; for each, count occurrences of each length-$L$ subarray via direct comparison.

    class Solution {
public:
    int minUniqueSubarrayLength(vector<int>& a) {
        int n = a.size();
        for (int L = 1; L <= n; ++L) {
            for (int i = 0; i + L <= n; ++i) {
                int c = 0;
                for (int j = 0; j + L <= n; ++j) {
                    bool eq = true;
                    for (int t = 0; t < L; ++t)
                        if (a[i + t] != a[j + t]) { eq = false; break; }
                    if (eq) ++c;
                }
                if (c == 1) return L;
            }
        }
        return n;
    }
};

    Time $O(n^3)$ — fine for the random cross-check oracle, but $\sim 10^{15}$ for $n = 10^5$. TLE.

    Optimal Approach

    Precompute two polynomial prefix hashes over nums with different bases and moduli, plus the corresponding power tables:

    $$h_b(i) = \sum_{j=0}^{i-1} (\mathrm{nums}[j] + 1) \cdot B^{i-1-j} \pmod{M}$$

    so that the hash of nums[l..r-1] is $h_b(r) - h_b(l) \cdot B^{r - l} \pmod M$. The +1 shift avoids representing a zero value as the empty contribution. We compute this for $(B_1, M_1) = (131,\ 10^9 + 7)$ and $(B_2, M_2) = (137,\ 998244353)$, then combine into a single 64-bit key.

    Predicate check(L): hash every length-$L$ window, count by key in an unordered_map, return true iff some key has count 1. $O(n)$ per call.

    Driver: binary-search the smallest $L \in [1, n]$ for which check(L) returns true. $O(\log n)$ calls.

    class Solution {
public:
    int minUniqueSubarrayLength(vector<int>& nums) {
        int n = nums.size();
        const long long M1 = 1000000007LL, B1 = 131;
        const long long M2 = 998244353LL,  B2 = 137;
        vector<long long> ph1(n + 1, 0), ph2(n + 1, 0),
                           pw1(n + 1, 1), pw2(n + 1, 1);
        for (int i = 0; i < n; ++i) {
            ph1[i + 1] = (ph1[i] * B1 + nums[i] + 1) % M1;
            ph2[i + 1] = (ph2[i] * B2 + nums[i] + 1) % M2;
            pw1[i + 1] = pw1[i] * B1 % M1;
            pw2[i + 1] = pw2[i] * B2 % M2;
        }
        auto getH = [&](int l, int r) -> unsigned long long {
            long long h1 = ((ph1[r] - ph1[l] * pw1[r - l]) % M1 + M1 * M1) % M1;
            long long h2 = ((ph2[r] - ph2[l] * pw2[r - l]) % M2 + M2 * M2) % M2;
            return ((unsigned long long)h1 << 32) ^ (unsigned long long)h2;
        };
        auto check = [&](int L) -> bool {
            unordered_map<unsigned long long, int> cnt;
            cnt.reserve(n * 2);
            for (int i = 0; i + L <= n; ++i) cnt[getH(i, i + L)]++;
            for (auto& [k, v] : cnt) if (v == 1) return true;
            return false;
        };
        int lo = 1, hi = n;     // length n is always unique
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (check(mid)) hi = mid; else lo = mid + 1;
        }
        return lo;
    }
};

    Time $O(n \log n)$, space $O(n)$.

    Collision safety: with two independent hashes modulo $\sim 10^9$ each, the probability that two distinct length-$L$ windows collide on both hashes is $\approx 10^{-18}$. For $n^2 \le 10^{10}$ pairs, total collision probability $\approx 10^{-8}$ — negligible.

    Dry Run

    Take nums = [1, 1, 2, 2, 1] (Example 3, expected answer 2). Binary search on $L \in [1, 5]$:

    steplohimidlength-mid windowsany unique?action
    1153[1,1,2], [1,2,2], [2,2,1]yes (all three distinct, each appears once)hi = 3
    2132[1,1], [1,2], [2,2], [2,1]yes (all four distinct)hi = 2
    3121{1:3, 2:2}nolo = 2

    Loop ends with lo == hi == 2 → return 2.

    For nums = [3, 3, 3]: length 1 has [3] appearing 3 times (no), length 2 has [3,3] twice (no), length 3 has [3,3,3] once (yes). Binary search narrows to 3.

    Edge Cases & Pitfalls

    • $n = 1$: the single length-1 subarray is unique. Binary search terminates immediately at lo = hi = 1.
    • All identical values: e.g. [1,1,1,1,1]. The only unique subarray is the full array; return $n$. The binary search exhausts smaller $L$ and lands on $n$.
    • All distinct values: every length-1 subarray is unique, return 1.
    • Single-hash collisions: if you use only one hash, an adversary can craft a collision and your code returns a wrong (too small) $L$. Double hashing makes this practically impossible.
    • Hashing zeros: if values could be 0, ensure the polynomial uses nums[i] + 1 (or some non-zero shift) so that [0] and [0, 0] don’t hash the same. For this problem $\mathrm{nums}[i] \ge 1$ so the shift is defensive.
    • Negative modular subtraction: (ph[r] - ph[l] * pw[r-l]) % M can be negative in C++. Add M * M (a large multiple of M that’s safe inside long long) and take % M again.
    • Combining two hashes: shift one left by 32 bits and XOR with the other. Both are at most $\sim 10^9 < 2^{30}$ so the bit ranges don’t overlap meaningfully.
    • Monotonicity proof: if you ever consider skipping the binary search, remember the monotonicity argument — uniqueness can’t suddenly disappear at a larger length, so binary search is sound.
