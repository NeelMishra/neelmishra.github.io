    Smallest Stable Index II

    Weekly Contest 498 · Q2 LeetCode #3904 View on LeetCode ↗ Reference (doocs) ↗
    Hard mode of Q1. Same statement; the only difference is the constraint n ≤ 105, which forces a linear-time solution. If you read Q1's editorial, the optimal approach below is identical.

    Problem Statement

    You are given an integer array nums of length n and an integer k.

    For each index i, define its instability score as max(nums[0..i]) - min(nums[i..n - 1]).

    In other words:

    • max(nums[0..i]) is the largest value among the elements from index 0 to index i.
    • min(nums[i..n - 1]) is the smallest value among the elements from index i to index n - 1.

    An index i is called stable if its instability score is less than or equal to k.

    Return the smallest stable index. If no such index exists, return -1.

    Example 1:

    Input:  nums = [5,0,1,4], k = 3
Output: 3

    Example 2:

    Input:  nums = [3,2,1], k = 1
Output: -1

    Example 3:

    Input:  nums = [0], k = 0
Output: 0

    Constraints:

    • 1 <= nums.length <= 105
    • 0 <= nums[i] <= 109
    • 0 <= k <= 109

    Source: LeetCode · Statement © LeetCode.

    Intuition

    The score is a difference of two monotone-in-i quantities: the prefix maximum is non-decreasing, and the suffix minimum is also non-decreasing. So we never need to "reopen" a window — every left-side update only adds a candidate to the prefix max, and every right-side update only removes a candidate from the suffix min. Both sides admit O(1) incremental updates, which collapses the search to a single linear scan after one preprocessing pass.

    The key insight that makes this Q2-not-Q1 is that with n up to 105, the O(n²) recompute-everything approach times out — you must commit to the incremental update.

    Brute Force

    Recompute the prefix max and suffix min from scratch at every i. Will TLE for n = 105, but useful as a baseline reference.

    class Solution {
public:
    int firstStableIndex(vector<int>& nums, int k) {
        int n = nums.size();
        for (int i = 0; i < n; ++i) {
            int mx = *max_element(nums.begin(), nums.begin() + i + 1);
            int mn = *min_element(nums.begin() + i, nums.end());
            if (mx - mn <= k) return i;
        }
        return -1;
    }
};

    Time O(n²), space O(1). With n = 105, that's ~1010 operations — too slow.

    Optimal Approach

    Run a backward sweep storing suffMin[i] = min(nums[i..n-1]), then a forward sweep that maintains prefMax and tests prefMax − suffMin[i] ≤ k.

    class Solution {
public:
    int firstStableIndex(vector<int>& nums, int k) {
        int n = nums.size();
        vector<int> suffMin(n);
        suffMin[n - 1] = nums[n - 1];
        for (int i = n - 2; i >= 0; --i)
            suffMin[i] = min(suffMin[i + 1], nums[i]);
        int prefMax = 0;
        for (int i = 0; i < n; ++i) {
            prefMax = max(prefMax, nums[i]);
            if (prefMax - suffMin[i] <= k) return i;
        }
        return -1;
    }
};

    Why it works. Both endpoints of the score are monotone in i, so each update is a single max or a precomputed lookup. We touch each index at most twice — once in each pass.

    Time O(n), space O(n).

    Dry Run

    Take Example 1: nums = [5,0,1,4], k = 3. The backward sweep yields suffMin = [0, 0, 1, 4]. Then:

    1. i = 0: prefMax = 5, suffMin = 0, score 5 — not stable.
    2. i = 1: prefMax = 5, suffMin = 0, score 5 — not stable.
    3. i = 2: prefMax = 5, suffMin = 1, score 4 — not stable.
    4. i = 3: prefMax = 5, suffMin = 4, score 1 — stable. Return 3.

    Complexity Recap

    ApproachTimeSpaceNotes
    Brute forceO(n²)O(1)TLE at n = 105.
    Optimal (prefix max + suffix min)O(n)O(n)Two linear passes; passes the 105 bound easily.

    Edge Cases & Pitfalls

    • Index i appears in both windows. The prefix is [0..i] and the suffix is [i..n-1]; they share the element at i. Off-by-one slicing changes results at the boundary.
    • Don't sort. The order of nums matters — sorting changes which element is in which window. Beginners sometimes try to "simplify" with sort and get wrong answers.
    • Integer width. Values fit in int, but if you accumulate or change to a different problem, beware 109 − 0 staying within int by ~one bit.
    • Strict early exit. Return on the first stable index; do not continue searching for a "more stable" index — the problem asks for the smallest, not the minimum-score, index.
    • n = 1. Always returns 0 (score is 0).

    C++ Code

    class Solution {
public:
    int firstStableIndex(vector<int>& nums, int k) {
        int n = (int)nums.size();
        vector<int> suffMin(n);
        suffMin[n - 1] = nums[n - 1];
        for (int i = n - 2; i >= 0; --i)
            suffMin[i] = min(suffMin[i + 1], nums[i]);
        int prefMax = nums[0];
        for (int i = 0; i < n; ++i) {
            prefMax = max(prefMax, nums[i]);
            if (prefMax - suffMin[i] <= k) return i;
        }
        return -1;
    }
};
