    Maximum Sum of Alternating Subsequence With Distance at Least K

    Problem Statement

    You are given an integer array nums of length n and an integer k.

    Pick a subsequence with indices 0 <= i1 < i2 < ... < im < n such that:

    • For every 1 <= t < m, it+1 - it >= k.
    • The selected values form a strictly alternating sequence; either nums[i1] < nums[i2] > nums[i3] < ..., or nums[i1] > nums[i2] < nums[i3] > ....

    A subsequence of length 1 is also considered strictly alternating. The score of a valid subsequence is the sum of its selected values.

    Return an integer denoting the maximum possible score of a valid subsequence.

    Example 1:

    Input:  nums = [5,4,2], k = 2
Output: 7
Explanation: Indices [0, 2] -> values [5, 2]. Distance 2 - 0 = 2 >= k.
5 > 2 is strictly alternating. Score = 5 + 2 = 7.

    Example 2:

    Input:  nums = [3,5,4,2,4], k = 1
Output: 14
Explanation: Indices [0, 1, 3, 4] -> values [3, 5, 2, 4].
3 < 5 > 2 < 4 is strictly alternating. Score = 3 + 5 + 2 + 4 = 14.

    Example 3:

    Input:  nums = [5], k = 1
Output: 5
Explanation: A length-1 subsequence is trivially alternating.

    Constraints:

    • 1 <= n == nums.length <= 105
    • 1 <= nums[i] <= 105
    • 1 <= k <= n

    Intuition

    Standard alternating-subsequence DP: for each index i, track two states — hi[i] = best sum of a valid subseq ending at i where nums[i] plays the role of a "peak" (its predecessor in the chosen subseq has a strictly smaller value), and lo[i] = same but as a "valley" (predecessor strictly larger). Each state has a free base case where i starts a length-1 subseq.

    The gap constraint (consecutive chosen indices differ by at least k) only changes when a previous state becomes available. While processing index i we may extend from any j with j <= i - k. That looks like a sliding window over indices, but we still need the best lo[j] with nums[j] < nums[i] (and symmetrically hi[j] with nums[j] > nums[i]) — range-max queries over a value dimension.

    Values are bounded by 105, so two Fenwick trees keyed by value give us prefix-max / suffix-max queries in O(log V). Inserts are deferred: when the sweep reaches index i, we first insert j = i - k into both BITs (if j >= 0), which keeps every queryable state on the legal side of the gap constraint.

    Brute Force

    Direct O(n2) DP — for every i, scan all j <= i - k and update both states.

    class Solution {
public:
    long long maxAlternatingSum(vector<int>& nums, int k) {
        int n = nums.size();
        vector<long long> hi(n), lo(n);
        long long ans = 0;
        for (int i = 0; i < n; ++i) {
            hi[i] = lo[i] = nums[i];
            for (int j = 0; j <= i - k; ++j) {
                if (nums[j] < nums[i]) hi[i] = max(hi[i], lo[j] + nums[i]);
                if (nums[j] > nums[i]) lo[i] = max(lo[i], hi[j] + nums[i]);
            }
            ans = max({ans, hi[i], lo[i]});
        }
        return ans;
    }
};

    Time O(n2), space O(n). With n = 105 this is ~1010 ops — too slow.

    Optimal Approach

    Replace the inner scan with two value-indexed Fenwick trees (BITs) supporting point updates and prefix-max queries:

    • lowBIT stores lo[j] at value nums[j]. Querying the prefix max up to nums[i] - 1 returns the best valley lo[j] with strictly smaller value — the predecessor we want when extending into a peak at i.
    • highBIT stores hi[j] at the reversed value index V - nums[j] + 1. Querying its prefix max up to V - nums[i] answers "best peak hi[j] with value strictly greater than nums[i]" — the predecessor when extending into a valley at i.

    The gap constraint is enforced by delayed insertion: state for index j is pushed into the BITs only when the sweep reaches j + k. Initialising the BITs with zeros lets the base case "start a length-1 subseq at i" fall out naturally — an empty query returns 0, so hi[i] = nums[i] + 0.

    class Solution {
public:
    long long maxAlternatingSum(vector<int>& nums, int k) {
        int n = nums.size();
        const int V = 100001;
        struct BIT {
            int n; vector<long long> t;
            BIT(int n) : n(n), t(n + 2, 0) {}
            void update(int i, long long v) {
                for (; i <= n; i += i & -i) if (t[i] < v) t[i] = v;
            }
            long long query(int i) {
                long long r = 0;
                for (; i > 0; i -= i & -i) if (t[i] > r) r = t[i];
                return r;
            }
        };
        BIT lowBIT(V), highBIT(V);
        vector<long long> lo(n), hi(n);
        long long ans = 0;
        for (int i = 0; i < n; ++i) {
            int j = i - k;
            if (j >= 0) {
                lowBIT.update(nums[j], lo[j]);
                highBIT.update(V - nums[j] + 1, hi[j]);
            }
            long long bestLow  = nums[i] > 1 ? lowBIT.query(nums[i] - 1) : 0;
            long long bestHigh = nums[i] < V ? highBIT.query(V - nums[i])  : 0;
            hi[i] = (long long)nums[i] + bestLow;
            lo[i] = (long long)nums[i] + bestHigh;
            ans = max({ans, hi[i], lo[i]});
        }
        return ans;
    }
};

    Time O(n log V), space O(V).

    Why it works: the alternating-subseq DP is correct on its own (peaks extend valleys with strictly smaller value, valleys extend peaks with strictly larger value). The BITs are just an accelerator: at the moment we query for index i, every inserted state corresponds to a j <= i - k, so the gap constraint is automatic. Strict inequality is enforced by querying nums[i] - 1 (and the symmetric reversed bound), never nums[i] itself.

    Edge Cases & Pitfalls

    • Length-1 subsequence is legal: the BIT is initialised with zeros, so the base case hi[i] = lo[i] = nums[i] falls out automatically.
    • long long required: the score can reach n · max(nums) = 105 · 105 = 1010. Use long long in BIT cells and DP arrays.
    • k = n: no j ever satisfies j <= i - k for i < n except the inserts that fire only at i = k, k+1, ...; effectively only single-element subseqs are queryable, so the answer is max(nums). The code handles this without a special case.
    • Strict alternation: querying nums[i] - 1 (and the reversed analogue) is essential. Querying nums[i] would let equal values chain, breaking the "strictly <" / "strictly >" requirement.
    • Boundary values: if nums[i] == 1 there is no smaller value to query — skip lowBIT; if nums[i] == V there is no larger value — skip highBIT (default to 0).
    • Reversed-index off-by-one: "value > nums[i]" maps to reversed indices in [1, V - nums[i]]; using V - nums[i] + 1 would wrongly include equal-valued states.
    • Delayed insertion order: the insert for index j = i - k must happen before the query for index i; reversing it would either leak future state or miss legal predecessors.
