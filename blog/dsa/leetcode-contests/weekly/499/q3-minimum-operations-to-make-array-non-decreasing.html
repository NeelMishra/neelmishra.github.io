    Minimum Operations to Make Array Non Decreasing

    Problem Statement

    You are given an integer array nums of length n.

    In one operation, you may choose any subarray nums[l..r] and increase each element in that subarray by x, where x is any positive integer.

    Return the minimum possible sum of the values of x across all operations required to make the array non-decreasing.

    An array is non-decreasing if nums[i] <= nums[i + 1] for all 0 <= i < n - 1.

    Example 1:

    Input:  nums = [3,3,2,1]
Output: 2
Explanation: Add x=1 to nums[2..3] -> [3,3,3,2]; add x=1 to nums[3..3] -> [3,3,3,3].
Total sum of x values = 2.

    Example 2:

    Input:  nums = [5,1,2,3]
Output: 4
Explanation: Add x=4 to nums[1..3] -> [5,5,6,7]. Total sum of x values = 4.

    Constraints:

    • 1 <= n == nums.length <= 105
    • 1 <= nums[i] <= 109

    Source: LeetCode · Statement © LeetCode.

    Intuition

    Look only at adjacent pairs. Whenever nums[i-1] > nums[i] there is a "drop" of size d = nums[i-1] - nums[i]. Any sequence of operations must contribute a total of at least d more increase to indices >= i than to indices < i, so each operation costs at least the sum of the drops.

    Conversely, we can always achieve exactly that bound: for every drop, fire one operation on the suffix subarray [i, n-1] with x = d. Lifting a whole suffix uniformly never breaks any later non-decreasing pair (the relative order on the right is preserved), so each drop can be handled in isolation. The optimum therefore collapses to Σ max(0, nums[i-1] - nums[i]).

    The naive trap is to stop at "increase the value at index i" without realising the operation is on a contiguous subarray — which is exactly the freedom that makes a single suffix lift admissible.

    Brute Force

    Simulate the suffix-lift greedy unit by unit: each time we see a drop, lift the suffix [i, n-1] by 1 and pay 1.

    class Solution {
public:
    long long minOperations(vector<int>& nums) {
        long long ans = 0;
        for (int i = 1; i < (int)nums.size(); ++i) {
            while (nums[i] < nums[i - 1]) {
                for (int j = i; j < (int)nums.size(); ++j) ++nums[j];
                ++ans;
            }
        }
        return ans;
    }
};

    Time O(n · Σdrops) — up to O(n · 1014) on adversarial input. Useful only as an existence proof of the greedy.

    Optimal Approach

    Sum the positive drops directly — one pass, one accumulator.

    class Solution {
public:
    long long minOperations(vector<int>& nums) {
        long long ans = 0;
        for (int i = 1; i < (int)nums.size(); ++i)
            ans += max(nums[i - 1] - nums[i], 0);
        return ans;
    }
};

    Time O(n), space O(1).

    Why it works: a single suffix-lift achieves each drop without harming downstream pairs, and any feasible plan must cumulatively lift the suffix by at least each drop, so the lower bound matches the upper bound exactly.

    Edge Cases & Pitfalls

    • long long required: n can be 105 and each drop up to 109, so the answer can reach ~1014. Accumulating in int overflows.
    • Already non-decreasing: all max(...) terms are zero, return 0 (don't perform any unnecessary operations).
    • Single element (n == 1): the loop is empty and the answer is 0.
    • Strictly decreasing input: stress test — every adjacent pair contributes; this is where the long long bound is tight.
    • Negative diff must be clamped: nums[i-1] - nums[i] can be negative; without the max(..., 0) you would subtract from the answer.
    • Don't be tempted to "increase a single index": the operation is on a contiguous subarray; the proof only goes through because lifting the entire suffix is admissible.
