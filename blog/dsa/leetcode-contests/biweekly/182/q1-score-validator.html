    Score Validator

    Biweekly Contest 182 · Q1 Easy · 3 pts Tags: Simulation, Array

    Problem Statement

    You are given a string array events. Initially score = 0 and counter = 0. Each element of events is one of:

    EventEffect on scoreEffect on counter
    "0" … "6" (only 0,1,2,3,4,6)add the integer valueunchanged
    "W"unchanged+1
    "WD"+1unchanged
    "NB"+1unchanged

    Process the array left to right. Stop as soon as either:

    1. All elements have been processed, or
    2. The counter reaches 10.

    Return [score, counter] at the moment processing stops.

    Examples

    events = ["1","4","W","6","WD"] → [12, 1]
    events = ["WD","NB","0","4","4"] → [10, 0]
    events = ["W","W","W","W","W","W","W","W","W","W","W"] → [0, 10] (stops at the 10th "W"; the 11th is ignored).

    Intuition

    The problem is just cricket scoring, lightly disguised. Once you see that, every rule snaps into place:

    • "0".."6" — runs scored by the batter, added to the team total.
    • "W" — wicket: a batter is out. No runs, but the wicket counter goes up. An innings ends at 10 wickets, which is exactly the stop condition.
    • "WD" — wide: an illegal delivery, the batting side gets a penalty run. No wicket implication for our purposes.
    • "NB" — no-ball: same idea, +1 penalty run, no wicket.

    So the counter in the problem is the wicket count, and the cap of 10 is just "all out, innings over." There is no clever data structure here — we walk the array once, dispatch on each token, and break the moment wickets reach 10.

    Pattern name: linear scan with an early-exit predicate. Same shape as "process events until X happens" problems — the trick is recognising the stop condition checks after the increment, not before.

    Brute Force

    For a problem this small, "brute force" is essentially the same as the optimal: a single pass with a chain of comparisons. There is no faster asymptotic algorithm to upgrade to. The only "non-brute" element is making sure we do not over-process events past the wicket cap. A naive implementation that loops over the entire array unconditionally still gives the right answer in Examples 1 and 2 — but fails Example 3, where the 11th "W" would push the counter to 11 instead of stopping at 10.

    function scoreValidator_naive(events):
    score = 0; counter = 0
    for e in events:                # BUG: no early stop
        if e == "W":  counter += 1
        else if e in {"WD", "NB"}: score += 1
        else:           score += int(e)
    return [score, counter]

    This passes Examples 1 and 2 but returns [0, 11] on Example 3 — the failure mode is exactly the missing stop. The fix is one line.

    Optimal Approach

    Walk left to right. After every increment to counter, check whether it has reached 10; if so, break. The order matters: the problem says "stop processing when the counter becomes 10," which means the event that caused the 10th wicket is itself counted — we increment first, then break.

    function scoreValidator(events):
    score = 0; counter = 0
    for e in events:
        if e == "W":
            counter += 1
            if counter == 10: break   # stop AFTER this increment
        else if e == "WD" or e == "NB":
            score += 1
        else:
            score += int(e)
    return [score, counter]

    Two equivalent placements of the break check work:

    • Inside the W branch (above) — only that branch can change the counter, so checking only there is sufficient and slightly cleaner.
    • At the end of the loop body — if (counter == 10) break;. Equally correct, just one extra comparison per iteration.

    Pre-checking with if (counter >= 10) break; at the top of the loop is also correct here, because once the counter hits 10 it can never reach 11 (we'd stop first). Functionally all three are the same; the post-increment check is the one that maps most directly onto the problem wording.

    Where I Got Stuck

    Friction point: the order of "increment counter" vs "check for break."

    My first instinct was to write the loop as:

    for e in events:
    if counter == 10: break      # pre-check
    ... process e ...

    That's also correct in this problem (because the moment counter hits 10, no event afterwards can reduce it, so the next iteration's pre-check breaks). But it conflates two situations that I usually want to keep distinct:

    • Pre-check ("don't even start an iteration if we're already in the stop state").
    • Post-check ("we just changed state — did we trigger the stop?").

    The problem statement says "stop when the counter becomes 10," which is post-check phrasing. Mapping the wording onto a post-check break removes the mental gymnastics about whether the boundary case is fence-post-correct.

    Lesson I want to keep: when a problem uses verbs like becomes, reaches, turns into, write the break after the mutation that could trigger it. When the wording is while, as long as, until, write a pre-check. They look almost identical and are often interchangeable on easy problems — but they diverge on harder ones (e.g. when the same iteration both increments and decrements the trigger value).

    Complexity

    • Time: O(n) where n = events.length. Each event is touched at most once. With the cap, we sometimes do strictly less than n work.
    • Space: O(1) extra. Two integer accumulators.

    With n ≤ 1000 the absolute cost is trivial — this question is testing reading comprehension and careful boundary handling, not algorithmic skill.

    Edge Cases & Pitfalls

    • Forgetting the stop condition: the single most common WA on this problem. The naive "just iterate the whole array" passes 2/3 examples and fails the third — classic case where Example 3 is constructed specifically to catch the bug.
    • Off-by-one on the break: stopping before the increment that would push counter to 10 returns [0, 9] on Example 3. Stop after.
    • Treating "W" as a string equal to "WD" by prefix: in C++ if you only test e[0] == 'W' you collapse W, WD, and any future "W..." token into one branch. Use full-string equality.
    • "5": the problem deliberately omits 5 from the digit set (5 runs is impossible in cricket). Don't pre-build a digit set that includes it; better, just call e[0] - '0' on the digit branch and trust the constraint that the input is well-formed. If you want to be defensive, you can assert the value is in {0,1,2,3,4,6}.
    • Empty array: the constraints say events.length ≥ 1, so we don't need to handle it — but the loop already returns [0, 0] for an empty input, which is the right answer anyway.
    • Counter beyond 10: impossible if the break is correct. If you ever see an output with counter > 10, the break is misplaced.

    C++ Code

    class Solution {
public:
    vector<int> scoreValidator(vector<string>& events) {
        int score = 0, counter = 0;
        for (const string& e : events) {
            if (e == "W") {
                ++counter;
                if (counter == 10) break;   // post-increment check
            } else if (e == "WD" || e == "NB") {
                score += 1;
            } else {
                // e is one of "0","1","2","3","4","6"
                score += e[0] - '0';
            }
        }
        return {score, counter};
    }
};

    A couple of small style choices worth noting:

    • const string& e in the range-for avoids copying each event string. Cheap optimisation, free correctness benefit (no accidental mutation).
    • e[0] - '0' is faster and clearer than stoi(e) for single-digit tokens.
    • Returning a brace-initialised {score, counter} implicitly constructs the vector<int> — one less line of boilerplate.

    One-Line Takeaway

    "Stop when X becomes Y" means increment, then check — not check, then increment. On Easy problems they often agree; on Mediums they don't, and that's where this small habit pays off.
