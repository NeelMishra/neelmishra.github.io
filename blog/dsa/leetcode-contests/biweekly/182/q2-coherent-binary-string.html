    Coherent Binary String

    Biweekly Contest 182 · Q2 Medium Tags: String, Greedy, Counting

    Problem Statement

    Given a binary string s, return the minimum number of flips (0⇔1) so that the resulting string is coherent — meaning it contains neither "011" nor "110" as a subsequence.

    Constraint: 1 ≤ |s| ≤ 105.

    Examples:

    • s = "1010" → 1 (e.g. flip s[0] to get "0010")
    • s = "0110" → 1 (flip s[1] to get "0010")
    • s = "1000" → 0 (already coherent)

    Intuition — What Does “Coherent” Even Mean?

    Read the two forbidden patterns slowly. They're both about where 0's sit relative to 1's:

    • "011" as a subsequence exists iff some 0 has at least two 1's after it.
    • "110" as a subsequence exists iff some 0 has at least two 1's before it.

    Combine the two and the constraint becomes very restrictive. For every 0 in a coherent string:

    ones_before(0) ≤ 1   AND   ones_after(0) ≤ 1

    Let K = total number of 1's in the string. Now ask: how many 1's can possibly fit?

    • K = 0: no 1's, nothing to forbid. The string is all 0's. ✓ coherent.
    • K = 1: only one 1 in the whole string, so no 0 can have two 1's on either side. ✓ coherent regardless of where that 1 sits.
    • K = 2: for any 0, ones_before + ones_after = 2. Both must be ≤ 1, so both equal 1. That means every 0 lies strictly between the two 1's. The string therefore looks like 1[0]*1 — starts with 1, ends with 1, only 0's between.
    • K ≥ 3: for any 0, ones_before + ones_after = K ≥ 3, so at least one side has ≥ 2 ones — instant violation. The only way out is to have no 0's at all, i.e. the string is all 1's.
    Structural lemma. A binary string is coherent if and only if it matches one of exactly four shapes.

    A — All zeros

    0*
    e.g. "", "0", "000"

    B — All ones

    1+
    e.g. "1", "111", "11111"

    C — Exactly one 1

    0* · 1 · 0*
    e.g. "100", "010", "0010"

    D — Exactly two 1's wrapping zeros

    1 · 0* · 1
    e.g. "11", "101", "10001"

    Once we have this list, the problem reduces to: for each candidate shape, compute the number of flips needed to transform s into that shape, and take the minimum.

    Brute Force

    Try every flip mask. With n ≤ 105 there are up to 2105 masks — obviously infeasible. Even checking subsequence-freedom for each candidate is non-trivial. The brute force is useful only as a correctness oracle for tiny n (which is what I used to validate the solution: 2000 random tests against an exhaustive enumeration on n ≤ 10).

    function brute(s):
    n = len(s)
    best = n
    for mask in range(1 << n):
        t = apply_flips(s, mask)
        if is_coherent(t):
            best = min(best, popcount(mask))
    return best

    Optimal Approach — Enumerate the 4 Shapes

    Let n = |s|, ones = count of '1' in s, zeros = n − ones.

    For each shape, the target string is fully determined (or has a single free choice we can resolve optimally). The flip cost is just the number of positions where target differs from s:

    CaseTarget shapeFlip cost
    Aall 0ones
    Ball 1zeros
    Cexactly one 1ones − 1 if ones ≥ 1, else 1
    D1 [0]* 1 (only if n ≥ 2)[s[0]=='0'] + [s[n−1]=='0'] + (ones in s[1..n−2])

    A few notes on why each cost formula is right:

    • Case A: turn every 1 into 0 → exactly ones flips.
    • Case B: symmetric — zeros flips.
    • Case C: we need exactly one 1. If we already have at least one, keep any of them and flip the other ones − 1 to 0 (no need to touch the 0's). If we have no 1's at all, flip a single 0 into a 1, costing 1.
    • Case D: the target has '1' at positions 0 and n−1 and '0' everywhere else. The cost decomposes cleanly into three independent terms: pay 1 if the first character isn't already '1', pay 1 if the last isn't, and pay 1 for every interior '1' that needs to become '0'. The interior-ones count is ones − [s[0]=='1'] − [s[n−1]=='1'].

    Return min(A, B, C, D). That's the entire algorithm. Counting ones is a single linear pass; everything else is O(1).

    Where I Got Stuck

    Friction point #1: the seductive but wrong "make it monotonic" idea.

    My first instinct on seeing “avoid 011 and 110 as subsequences” was to think: just split into a left half of all 0's and a right half of all 1's (or vice versa) — like a sorted binary string. Then surely both forbidden patterns are gone. That feels right and even has a clean O(n) prefix-sum implementation. It's also completely incorrect.

    Counter-example: take "0011". It is fully sorted. But pick the characters at positions 0, 2, 3 — that's '0','1','1', which is exactly the forbidden subsequence "011". So "0011" is not coherent at all.

    The trap is conflating substring with subsequence. A monotonic string has no "011" as a contiguous substring — but a subsequence skips characters, and "0011" has plenty of room to skip into "011". The same idea kills any approach based on splitting the string into two homogeneous halves: as long as the left half has at least one 0 and the right half has at least two 1's, you have a forbidden subsequence.

    Friction point #2: trusting the structural lemma over a generic DP.

    After abandoning the monotonic idea, the next instinct is a DP over the state of the longest matched prefix of each forbidden pattern (a finite-automaton DP, O(n · states)). That works and generalises to any list of forbidden subsequences — but it's overkill here, and on a contest clock it's easy to misimplement the transitions.

    The real unlock was sitting with the question “what does it actually mean for a string to avoid both?” long enough to derive the per-zero condition ones_before ≤ 1 AND ones_after ≤ 1. Once that's in front of you, the case analysis on K writes itself, and the problem collapses from “cleverness needed” to “just compare with four fixed targets.”

    Lesson I want to keep: when a problem forbids a small set of subsequence patterns, characterise the legal strings structurally before reaching for a DP. Subsequence constraints often force the legal set to be much smaller than it looks — sometimes just a handful of shapes — and once you list those shapes, the rest is bookkeeping.

    Complexity

    • Time: O(n). One pass to count ones; cases A–D are each O(1) given that count plus s[0] and s[n−1].
    • Space: O(1) extra.

    Edge Cases & Pitfalls

    • n = 1: Case D needs at least two characters — skip it (or set its cost to +∞). Cases A, B, C still work; the answer is always 0 for a single character.
    • ones = 0: the string is already all 0's, so Case A wins with cost 0. The Case-C formula needs the else 1 branch (flip a single 0 to 1), which never gets selected here but should still be valid output.
    • ones = n (all 1's): Case B wins with cost 0. Case D with target "1[0]*1" would cost n − 2 (flip all interior 1's to 0), so D is correctly dominated.
    • Off-by-one in Case D's “interior ones” count: the formula ones − [s[0]=='1'] − [s[n−1]=='1'] double-counts only when n = 1 — which we exclude anyway. For n = 2, interior is empty, so the count is ones − [s[0]=='1'] − [s[1]=='1'], which is 0 whenever both ends are '1' (e.g. "11") — correct.
    • "0011" trap: the most common wrong intuition for this problem. If your solution outputs 0 for "0011", your check is treating the patterns as substrings instead of subsequences. The correct answer is 1 (e.g. flip s[2] to get "0001", which has only one 1 — Case C).
    • Subsequence vs substring under contest pressure: the problem statement explicitly defines “subsequence” in the body. Skipping that paragraph leads straight to a wrong solution.

    C++ Code

    class Solution {
public:
    int minModifications(string s) {
        int n = (int)s.size();
        int ones = 0;
        for (char c : s) if (c == '1') ++ones;
        int zeros = n - ones;

        // Case A — all zeros
        int A = ones;
        // Case B — all ones
        int B = zeros;
        // Case C — exactly one '1' somewhere
        int C = (ones >= 1) ? (ones - 1) : 1;
        // Case D — "1 [0]* 1" (needs n >= 2)
        int D = INT_MAX;
        if (n >= 2) {
            int firstFlip = (s.front() == '0') ? 1 : 0;
            int lastFlip  = (s.back()  == '0') ? 1 : 0;
            int interiorOnes = ones
                              - (s.front() == '1' ? 1 : 0)
                              - (s.back()  == '1' ? 1 : 0);
            D = firstFlip + lastFlip + interiorOnes;
        }
        return min({A, B, C, D});
    }
};

    Compact, O(n), and easy to reason about. The four cases are independent of each other, so adding a fifth or removing one is trivial — useful if the problem variant changes the forbidden patterns.

    A Note on Generalisation

    For a problem that forbids an arbitrary set of subsequence patterns (not just 011 and 110), the case-enumeration trick stops working — the legal strings stop having a tidy closed-form description. The fallback is the aho-corasick-on-subsequences DP: state = (longest matched prefix of each forbidden pattern), transitions on each character of the result, minimise total flips. That's O(n · product-of-pattern-lengths) and is the right hammer when the structural lemma route runs out.

    For this specific problem, both work and give the same answer — but the case enumeration is faster to write under contest pressure and harder to get wrong.

    One-Line Takeaway

    “Forbid pattern P as a subsequence” usually shrinks the legal-string set far more than it first appears. Before reaching for a DP, ask what shapes are still legal? — sometimes the answer is “four of them,” and the problem is over.
