    Foundations: The Template, the Tight Flag, and Why They Work

    April 2026 · 30 min read

    Part 1 motivated the problem. This post builds the machinery. We're going to write the digit DP template from scratch, justify every line, prove the memoization rule, and walk through the full execution on $N = 357$ digit by digit. By the end of this post you will not be memorizing a template — you will be able to derive one for any new problem.

    The Canonical Problem

    Let's pin down the reference problem so the template has something concrete to compute.

    Reference problem. Given $N \leq 10^{18}$, count the integers $x$ with $0 \leq x \leq N$ such that $x$ contains no digit 4. Return the count.

    We'll extend the template to more problems in Part 3. But every mechanism we need — position tracking, the tight flag, leading zeros, memoization — shows up already here.

    Deriving the Recursion

    Think of $N$ as a fixed string of decimal digits, $d_0 d_1 \ldots d_{D-1}$ with $d_0$ most significant. We're going to enumerate numbers $x$ with at most $D$ digits and $x \leq N$ by choosing each digit of $x$ left-to-right. Define:

    $$\mathrm{solve}(pos, tight) = \#\{\text{valid completions starting from position } pos\text{ with constraint } tight\}.$$

    where:

    • pos $\in \{0, 1, \ldots, D\}$ is the next digit to place.
    • tight $\in \{0, 1\}$: $1$ if every digit placed so far equals the corresponding digit of $N$, $0$ otherwise.

    The base case is when we've placed all $D$ digits: we've built one valid number, so we return $1$.

    The recursion at (pos, tight) loops over the digit $d$ to place at position $pos$. The upper limit for $d$ is:

    • $d_{pos}$ (the digit of $N$ at this position) if $tight = 1$;
    • $9$ if $tight = 0$.

    After picking $d$, the new tight flag is $tight \land (d = d_{pos})$. That's the critical line. Read it carefully: the constraint persists only if we were previously tight and we picked the exact ceiling digit. Any smaller digit breaks the tie forever.

    string D;                 // decimal digits of N, MSB first
long long memo[20][2];

long long solve(int pos, int tight) {
    if (pos == (int)D.size()) return 1;
    if (!tight && memo[pos][tight] != -1) return memo[pos][tight];

    int hi = tight ? (D[pos] - '0') : 9;
    long long ans = 0;
    for (int d = 0; d <= hi; ++d) {
        if (d == 4) continue;                            // problem-specific filter
        ans += solve(pos + 1, tight && (d == hi));
    }
    if (!tight) memo[pos][tight] = ans;                  // memoize only free states
    return ans;
}

long long countNo4(long long N) {
    D = to_string(N);
    memset(memo, -1, sizeof memo);
    return solve(0, 1);
}

    That's the whole thing. Every digit DP you will ever write follows this skeleton, with extra state fields alongside tight.

    Why the Tight Update Rule Is Exactly tight && (d == hi)

    Let $x = d_0 d_1 \ldots d_{D-1}$ be the number we're building and $N = n_0 n_1 \ldots n_{D-1}$ be the bound. The constraint $x \leq N$ is, digit-wise, lexicographic: $x \leq N$ iff either $x = N$ or there exists an index $j$ with $d_0 = n_0, \ldots, d_{j-1} = n_{j-1}$ and $d_j < n_j$ (with no constraint on positions after $j$).

    So if we've placed $d_0, \ldots, d_{pos-1}$:

    • If all of them equal $n_0, \ldots, n_{pos-1}$ (i.e. tight = 1), then we need $d_{pos} \leq n_{pos}$. If $d_{pos} = n_{pos}$ exactly, the invariant continues; if $d_{pos} < n_{pos}$, the rest of the number can be anything.
    • If there's already an index $j < pos$ where $d_j < n_j$ (i.e. tight = 0), then the lexicographic condition is already satisfied, independently of the digits we pick from here on. No cap applies to $d_{pos}$.

    This is why tight is binary and why its transition rule is exactly the conjunction above. It is not a heuristic; it is the logical statement of lexicographic comparison.

    Why We Can Only Memoize Free States

    Here is the trap that bites every first-time digit DP writer: the value of solve(pos, tight=1) depends on the digits of $N$ from position $pos$ onward. Two different upper bounds $N$ and $N'$ that share the first $pos$ digits but differ afterwards will produce different values for the same state label (pos, 1). Memoizing it with a shared table would silently poison later calls.

    By contrast, solve(pos, tight=0) does not see $N$ at all — once tight = 0, the transition always uses hi = 9 and the forbidden-digit filter, never consulting $D[pos]$. So its return value depends only on the problem-specific state and pos. Those are the states we can safely cache across different $N$ within the same program run.

    Consequence: in the problem $f(R) - f(L-1)$, we solve $f(R)$ first with the memo empty, then solve $f(L-1)$ and keep the memo. Every (pos, tight=0, \ldots) cached during the first call is immediately reusable in the second. That's free speedup. Tight states visit at most once per call anyway (there's exactly one tight path per $N$), so not caching them costs nothing.

    Leading Zeros: When They Matter

    In the no-digit-4 problem, leading zeros are harmless: a leading zero is not a "4", so it never violates the property. But consider this variation:

    Variation. Count integers $x \in [1, N]$ such that no two adjacent digits of $x$ are equal.

    If we naively encode the state as (pos, tight, last_digit) and start with last_digit = 0, we'll incorrectly reject numbers starting with $0$ at position $1$ (because "adjacent to the virtual leading zero"). But a virtual leading zero is not a real digit of the number — $42$ is a two-digit number, not a nineteen-digit number.

    The fix: add a leading_zero flag. While leading_zero = 1, we haven't placed a "real" first digit yet; placing a $0$ keeps us in that state, and placing any nonzero $d$ flips us out of it. Only after we exit does last_digit start to matter.

    long long solve(int pos, int tight, int lz, int last) {
    if (pos == (int)D.size()) return 1;                  // counts the number built (incl. 0 if lz still 1)
    if (!tight && !lz && memo[pos][last] != -1) return memo[pos][last];

    int hi = tight ? (D[pos] - '0') : 9;
    long long ans = 0;
    for (int d = 0; d <= hi; ++d) {
        int nlz  = lz && (d == 0);
        int nlast = nlz ? -1 : d;
        // adjacency constraint only applies once we've started the number
        if (!lz && d == last) continue;
        ans += solve(pos + 1, tight && (d == hi), nlz, nlast);
    }
    if (!tight && !lz) memo[pos][last] = ans;
    return ans;
}

    Two subtle points:

    1. We only memoize when both tight == 0 and lz == 0. While in leading-zero mode, the effective number of digits remaining is not a function of pos alone — it depends on where the real number started. (You can memoize under lz with a slightly different state, but the simpler rule is: don't.)
    2. We terminate at pos == D.size() returning $1$. This counts the number $0$ as a "valid" integer (the sequence of all leading zeros terminated at the end). For problems on $[1, N]$, subtract $1$ at the end. For problems on $[0, N]$, this is correct as-is.

    Three Common Ways to Write the Loop

    Different authors factor the recursion differently. They are all equivalent; pick the one your brain likes.

    Style A: Pass tight as a boolean argument (what we use above)

    Pros: the cleanest mapping from the mathematical definition. Cons: the memo key has to include tight, even though we skip caching when it's $1$.

    Style B: Drop out of the tight branch after the loop

    // pseudo: compute the "free" subtree once, then add the tight edge
long long ans = 0;
int top = D[pos] - '0';
// all digits strictly less than the cap -> free
for (int d = 0; d < top; ++d) if (d != 4) ans += freeSolve(pos + 1);
// digit equal to the cap -> still tight
if (top != 4) ans += solve(pos + 1, /*tight=*/1);

    Here freeSolve is a separate recursion that ignores $N$ entirely — pure DP on $(pos)$ plus problem state. This style makes the "free subtree is independent of $N$" property visible in the code, which helps when debugging.

    Style C: Iterative digit-by-digit with accumulators

    Iterate $pos = 0, 1, \ldots, D-1$ maintaining running totals for the free-subtree contributions at each step. Looks like a hand-built version of Style B. It avoids recursion, which matters if you're calling the routine inside another DP. Commonly seen in competitive programming templates.

    From Counting to Summing

    A frequent follow-up: not just how many numbers satisfy the property, but the sum of them. The template lifts cleanly: instead of returning count, return a pair (count, sum). When you pick a digit $d$ at position $pos$, the contribution of $d$ to the final number is $d \cdot 10^{D - 1 - pos}$, multiplied by the count of completions below.

    struct R { long long cnt, sum; };

R solve(int pos, int tight) {
    if (pos == (int)D.size()) return {1, 0};
    if (!tight && memo[pos].cnt != -1) return memo[pos];

    int hi = tight ? (D[pos] - '0') : 9;
    long long place = pow10[D.size() - 1 - pos];
    R ans{0, 0};
    for (int d = 0; d <= hi; ++d) {
        if (d == 4) continue;
        R sub = solve(pos + 1, tight && (d == hi));
        ans.cnt += sub.cnt;
        ans.sum += sub.sum + d * place * sub.cnt;
    }
    if (!tight) memo[pos] = ans;
    return ans;
}

    Every aggregate over digits (sum, sum of squares, products modulo $p$, etc.) can be computed by the same trick: carry a tuple through the recursion and combine at each transition.

    Pitfalls Checklist

    • Off-by-one on $[L, R]$. Always compute $f(R) - f(L - 1)$, not $f(R) - f(L)$.
    • Counting zero. If your base case returns $1$, you are counting the number $0$. Subtract it out if the problem says $[1, N]$.
    • Forgetting leading_zero. Only needed when the constraint discriminates between a leading zero and a real digit $0$. If in doubt, add the flag — the extra state factor is $2$, negligible.
    • Memoizing tight states. Don't, unless you clear the memo between calls.
    • Integer overflow. For $N \approx 10^{18}$ the count itself might exceed $10^{18}$ (e.g. counting subsets of digit positions). Use __int128 or modular arithmetic if the problem demands it.
    • Digit array direction. Consistency: if you store digits MSB-first, pos = 0 is the leading digit. If LSB-first, pos = 0 is the units place. Pick one and stick with it everywhere — mixing is a guaranteed bug.

    Reusable Template

    Here's the template we'll reach for in the rest of the series. You should be able to plug any digit-level property into the isValidDigit filter and the extra state fields.

    // Generic digit DP template (count version)
// Customize: extra state fields (sum, rem, mask, ...), the digit filter, and the base case.

struct DigitDP {
    string D;
    // memo[pos][state] — only populated when tight == 0 && lz == 0
    vector<vector<long long>> memo;

    long long solve(int pos, int tight, int lz, int state) {
        if (pos == (int)D.size()) return isAccepting(lz, state) ? 1 : 0;
        if (!tight && !lz && memo[pos][state] != -1) return memo[pos][state];

        int hi = tight ? (D[pos] - '0') : 9;
        long long ans = 0;
        for (int d = 0; d <= hi; ++d) {
            if (!passesFilter(d, lz, state)) continue;
            int ns   = nextState(d, lz, state);
            int nlz  = lz && (d == 0);
            int nt   = tight && (d == hi);
            ans += solve(pos + 1, nt, nlz, ns);
        }
        if (!tight && !lz) memo[pos][state] = ans;
        return ans;
    }

    long long countUpTo(long long N) {
        D = to_string(N);
        memo.assign(D.size(), vector<long long>(STATE_MAX, -1));
        return solve(0, 1, 1, INITIAL_STATE);
    }

    // Plug in these four hooks:
    bool isAccepting(int lz, int state);     // terminal acceptance predicate
    bool passesFilter(int d, int lz, int s); // digit-level validity
    int  nextState(int d, int lz, int s);    // state transition
    // ... STATE_MAX, INITIAL_STATE defined per problem
};
