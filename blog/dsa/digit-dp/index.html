    Digit DP: The Complete Guide

    April 2026 · 22 min read

    Suppose someone asks you: how many integers between $1$ and $10^{18}$ contain the digit 7 an even number of times and are divisible by $13$? You cannot loop from $1$ to $10^{18}$ — that is a quintillion iterations. You also cannot reasonably answer this with any closed-form combinatorial identity. This is where digit DP lives.

    Digit DP is a family of dynamic programming techniques where the state is defined by the position of a digit in the number you are constructing, rather than by an array index or a graph node. It lets you count — or sum, or take a min over — all numbers in an astronomically large range $[L, R]$ that satisfy some digit-level property, usually in $O(\log_{10}(R) \cdot S)$ time where $S$ is the size of the auxiliary state.

    What this series covers. Part 1 (this post) motivates the technique and introduces the prefix-count trick. Part 2 develops the core template and state design vocabulary — pos, tight, leading zero, and memoization rules. Part 3 works through half a dozen classic problems end-to-end. Part 4 pushes into advanced territory — modular state, digit products, XOR constraints, binary digit DP, and counting the $K$-th valid number.

    When Should You Reach For Digit DP?

    A problem is a digit DP candidate when it has all three of these flavors:

    1. The answer depends on integers in some interval $[L, R]$, where $R$ is too large to enumerate — typically $R \leq 10^{18}$.
    2. The condition on each integer is expressible digit-by-digit: things like "sum of digits is $s$", "contains digit $d$ exactly $k$ times", "no two adjacent digits are equal", "divisible by $m$", "the digits, read as a sequence, are non-decreasing".
    3. You need to count (or sum, or find the $K$-th, or take min/max over) all such integers.

    If any of those three are missing, the right tool is probably different. Pure combinatorics works when the constraint decouples cleanly. Meet-in-the-middle works when the digits are independent but $R$ is not too huge. Digit DP earns its keep precisely when the constraint couples digits together (e.g., via a running sum or modular accumulator) and you need tight answers modulo some prime.

    The Range-to-Prefix Reduction

    Almost every digit DP problem is solved via the same first move. Define:

    $$f(N) = \#\{\,x : 0 \leq x \leq N \text{ and } x \text{ satisfies } P\,\}.$$

    Then:

    $$\#\{\,x \in [L, R] : P(x)\,\} = f(R) - f(L - 1).$$

    You only ever have to solve the prefix version. That one-sided bound is what makes the state tractable: you are building a number digit-by-digit from the most significant end, and at each step you only need to know whether you are still tied to the upper bound or already strictly below it.

    Why prefix, not range, directly? A range constraint would force you to track two tight flags simultaneously (against $L$ and against $R$). Doing that is legal but doubles the state and complicates corner cases around leading zeros. The subtraction trick reduces any range problem to two independent prefix problems, which is cleaner and strictly easier to code.

    A First Example: No Digit Four

    Let's count integers in $[1, N]$ that contain no digit 4. For small $N$ the brute force is trivial:

    long long bruteCount(long long N) {
    long long cnt = 0;
    for (long long x = 1; x <= N; ++x) {
        long long y = x; bool ok = true;
        while (y) { if (y % 10 == 4) { ok = false; break; } y /= 10; }
        if (ok) ++cnt;
    }
    return cnt;
}

    This is $O(N \log N)$ and falls over for $N = 10^{18}$. The digit DP version runs in $O(D \cdot 2)$ where $D = \lfloor \log_{10} N \rfloor + 1$ — about $19 \cdot 2 = 38$ states, regardless of how large $N$ is. A preview of the shape of the recursion (Part 2 will derive it carefully):

    string digits;          // decimal digits of N, most significant first
long long memo[20][2];

long long solve(int pos, int tight) {
    if (pos == (int)digits.size()) return 1;  // one valid number
    if (!tight && memo[pos][tight] != -1) return memo[pos][tight];

    int hi = tight ? (digits[pos] - '0') : 9;
    long long ans = 0;
    for (int d = 0; d <= hi; ++d) {
        if (d == 4) continue;                 // skip forbidden digit
        ans += solve(pos + 1, tight && (d == hi));
    }
    if (!tight) memo[pos][tight] = ans;
    return ans;
}

    Notice two things. First, the memo is only written when tight == 0 — we'll justify this rule in Part 2, but intuitively, once you go below the upper bound, the remaining subproblem no longer depends on $N$ at all, so the answer is reusable. Second, the state is tiny: just (pos, tight). Every digit DP problem is a variation on this skeleton with more fields tacked onto the state.

    Anatomy of a Digit DP State

    Every digit DP state has the same conceptual shape:

    state = (pos, tight, leading_zero, ...problem-specific fields...)
    • pos — the index of the digit you are about to choose, counted from the most significant end. Ranges from $0$ to $D$ (inclusive on $D$, which is the terminal state).
    • tight — a boolean. 1 means every digit you have placed so far exactly matches the corresponding digit of $N$; your next digit is capped at N[pos]. 0 means you have already placed a strictly smaller digit somewhere, so all following digits are unconstrained (0–9).
    • leading_zero — a boolean indicating whether the number is still "pre-number" (all zeros placed so far). This matters when the problem distinguishes between a digit appearing and the implicit padding zero: for example, the number $42$ has two digits, not nineteen.
    • Problem-specific fields — the running sum of digits mod $k$, a count of some specific digit, a bitmask of which digits have appeared, the last digit placed (for "no equal adjacent" constraints), and so on. These are what distinguish one digit DP problem from another.

    Why the Tight Flag Is Load-Bearing

    Consider $N = 357$. When you are choosing the most significant digit (hundreds place), you may pick 0, 1, 2, 3. If you pick 0, 1, 2, the remaining two digits can be anything $00$–$99$: the subproblem forgot about $N$ entirely. If you pick 3, though, the next digit is capped at 5, and you still care about $N$.

    The tight flag captures exactly that binary distinction: are we still tied to the ceiling, or have we broken free? Once free, the subproblem is independent of $N$ and can be memoized once and reused across every call in which tight is false at this position. That is the source of the exponential speedup.

    What Gets Memoized, What Doesn't

    The rule of thumb: memoize only the states where tight == 0 and leading_zero == 0. States with tight == 1 encode information about the specific upper bound $N$ passed in, so they are (typically) visited at most once per call anyway, and caching them would be incorrect if you reuse the table across different $N$. Pragmatically you can also memoize tight states as long as you clear the table between calls.

    You will see different codebases take different conventions here. The safest and most widely-taught convention — and what we'll use throughout this series — is: memoize only non-tight, non-leading-zero states. That keeps the semantics "this state's answer does not depend on $N$" clean and prevents an entire class of subtle bugs.

    Counting, Summing, and Beyond

    Once you can count, you can compute almost any digit-level aggregate. Three common variations:

    • Sum of $f(x)$ over all $x \in [1, N]$: carry a pair (count, sum) through the recursion. The sum of the whole subtree is count * (prefix contribution) + subsum.
    • $K$-th number with property: walk digit by digit, and at each step count how many valid numbers start with each candidate digit. Choose the smallest digit for which that cumulative count is $\geq K$; subtract and recurse.
    • Min / max over valid numbers: a trivial greedy walk — pick the smallest / largest digit for which the remaining subtree is non-empty. Digit DP handles the emptiness test.

    All three reuse the same core count function. Part 4 builds each one.

    In every row, the "$\times 10$" for the digit loop inside the transition is implicit. The takeaway: digit DP is almost always fast. The challenge is designing the state.

    Common Misconceptions

    • "Digit DP is just DP over an array of digits." Only structurally. Ordinary array DP has a fixed upper bound on every state field. Digit DP has a dynamic upper bound — the tight flag — and handling that is the whole skill.
    • "I can memoize everything, even tight states." You can, if you clear the memo between calls with different $N$. But if you solve $f(R) - f(L-1)$ in one program run with a single memo table, you must not cache tight states.
    • "Leading zeros don't matter." They matter whenever the property cares about digit identity. For "no digit 4", leading zeros are irrelevant. For "no two adjacent digits equal", they matter — a leading zero should not count as "the previous digit is 0" when you place the first real digit.
