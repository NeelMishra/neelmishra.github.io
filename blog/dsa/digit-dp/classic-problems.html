    Six Classic Digit DP Problems, Worked End to End

    April 2026 · 32 min read

    We have a template. Time to use it. This post works through six named problems that cover the state-design patterns you'll meet 90% of the time: additive accumulators, modular accumulators, counting occurrences, bitmask accumulators, lexicographic adjacency, and binary-representation digit DP. Each section states the problem, shows the state design, gives full code, and notes a pitfall.

    Assume throughout that we have a helper long long g(long long N) returning the count on $[0, N]$, and the range answer is g(R) - g(L-1).

    Problem 1: Numbers With Digit Sum Exactly $S$

    Given $N$ and $S$, count $x \in [0, N]$ whose decimal digits sum to exactly $S$.

    State. $(pos, tight, lz, sum)$. The $lz$ field actually doesn't matter here — a leading zero contributes $0$ to the sum, same as a real zero digit — so we can drop it and start with $sum = 0$.

    Transition. At each digit $d$, new state has $sum \to sum + d$. If $sum + d > S$, prune (optional, for speed).

    Acceptance. At $pos = D$, return $[sum == S]$.

    int D, S;
string digits;
long long memo[20][200];   // sum ≤ 9 × 20 = 180

long long solve(int pos, int tight, int sum) {
    if (sum > S) return 0;
    if (pos == D) return sum == S ? 1 : 0;
    if (!tight && memo[pos][sum] != -1) return memo[pos][sum];

    int hi = tight ? (digits[pos] - '0') : 9;
    long long ans = 0;
    for (int d = 0; d <= hi; ++d)
        ans += solve(pos + 1, tight && (d == hi), sum + d);
    if (!tight) memo[pos][sum] = ans;
    return ans;
}
long long g(long long N, int targetSum) {
    digits = to_string(N); D = digits.size(); S = targetSum;
    memset(memo, -1, sizeof memo);
    return solve(0, 1, 0);
}

    Complexity: $O(D \cdot 9D \cdot 10) = O(D^2)$ states with a $10$-way branch. Under 4000 basic operations even for $N = 10^{18}$.

    ▶ Memo Table Fill — Count Numbers $\leq 99$ With Digit Sum = 5

    Step through filling the $(pos, sum)$ memo table. Only free states (tight = 0) get cached. Watch how a single free-state fill is reused by every subsequent call that lands on the same state.

    Problem 2: Numbers Divisible by $K$ in $[L, R]$

    Count $x \in [L, R]$ with $x \equiv 0 \pmod{K}$.

    Yes, the answer is just $\lfloor R/K \rfloor - \lfloor (L-1)/K \rfloor$. But the digit DP version is the essential building block for more interesting problems like "divisible by $K$ and digit sum $= S$" or "divisible by the sum of its own digits" (Harshad / Niven numbers), where no closed form exists.

    State. $(pos, tight, rem)$. The transition is $rem \to (rem \cdot 10 + d) \bmod K$. Acceptance: $rem == 0$.

    int D, K; string digits;
long long memo[20][100];

long long solve(int pos, int tight, int rem) {
    if (pos == D) return rem == 0 ? 1 : 0;
    if (!tight && memo[pos][rem] != -1) return memo[pos][rem];

    int hi = tight ? (digits[pos] - '0') : 9;
    long long ans = 0;
    for (int d = 0; d <= hi; ++d)
        ans += solve(pos + 1, tight && (d == hi), (rem * 10 + d) % K);
    if (!tight) memo[pos][rem] = ans;
    return ans;
}

    Pitfall. The state includes only free cached values. When you call g(R) then g(L-1) back to back with $K$ unchanged, the memo is fully reusable and the second call is nearly free. If $K$ changes between calls, you must clear the memo, because rem is modulo $K$ and cached answers under a different modulus are garbage.

    Problem 3: Occurrences of Digit $d^*$ in $[1, N]$

    How many times does the digit $d^*$ (a fixed digit, say $7$) appear across the decimal representations of all integers from $1$ to $N$?

    This is a sum problem, not a count problem — we count occurrences, not numbers. The fix: carry a pair (count_of_numbers, sum_of_occurrences) as described in Part 2. Each placed digit equal to $d^*$ contributes $1$ occurrence to every completion of the suffix, times the count of those completions.

    int D, target;
string digits;
struct R { long long cnt, occ; };
R memo[20]; bool seen[20];

R solve(int pos, int tight) {
    if (pos == D) return {1, 0};
    if (!tight && seen[pos]) return memo[pos];

    int hi = tight ? (digits[pos] - '0') : 9;
    R ans{0, 0};
    for (int d = 0; d <= hi; ++d) {
        R sub = solve(pos + 1, tight && (d == hi));
        ans.cnt += sub.cnt;
        ans.occ += sub.occ + (d == target ? sub.cnt : 0);
    }
    if (!tight) { memo[pos] = ans; seen[pos] = true; }
    return ans;
}

    Leading zeros. If $d^* = 0$, you must use a leading_zero flag so that padding zeros in short numbers don't get counted as real digits. For any nonzero $d^*$, no flag needed.

    Problem 4: Magic Numbers (Digit Bitmask)

    Given a subset $M \subseteq \{0, 1, \ldots, 9\}$ of allowed digits and a required subset $R \subseteq M$. Count $x \in [0, N]$ whose digits are all in $M$ and that uses every digit of $R$ at least once.

    State. $(pos, tight, lz, mask)$, where $mask$ is the 10-bit set of digits seen so far (ignoring leading zeros). Transition: $mask \to mask \cup \{d\}$ when $d \in M$ and we're not in leading-zero mode.

    Acceptance. At $pos = D$: $R \subseteq mask$ and the number is nonempty (i.e. $lz = 0$, or we explicitly allow $x = 0$).

    int D; string digits;
int M, R;                            // bitmasks
long long memo[20][1024];           // indexed by (pos, mask) when free & not lz

long long solve(int pos, int tight, int lz, int mask) {
    if (pos == D) {
        if (lz) return 0;
        return (mask & R) == R ? 1 : 0;
    }
    if (!tight && !lz && memo[pos][mask] != -1) return memo[pos][mask];

    int hi = tight ? (digits[pos] - '0') : 9;
    long long ans = 0;
    for (int d = 0; d <= hi; ++d) {
        if (!((M >> d) & 1)) continue;
        int nlz = lz && (d == 0);
        int nmask = nlz ? 0 : (mask | (1 << d));
        ans += solve(pos + 1, tight && (d == hi), nlz, nmask);
    }
    if (!tight && !lz) memo[pos][mask] = ans;
    return ans;
}

    Observation. The state size is $D \times 2^{10} = 20 \cdot 1024 = 20480$ entries. Even with a 10-way branch that's $\sim 200\text{K}$ work — trivial.

    Problem 5: No Two Adjacent Digits Equal

    Count $x \in [1, N]$ such that no two adjacent digits are equal.

    Classical leading-zero gotcha. The state is $(pos, tight, lz, last)$. In leading-zero mode, $last$ is undefined; we pass a sentinel like $-1$ (or $10$) and skip the equality check whenever $lz = 1$.

    int D; string digits;
long long memo[20][11];   // last ∈ {0..9, 10=unused}

long long solve(int pos, int tight, int lz, int last) {
    if (pos == D) return lz ? 0 : 1;   // reject the all-zeros build
    int key = lz ? 10 : last;
    if (!tight && !lz && memo[pos][key] != -1) return memo[pos][key];

    int hi = tight ? (digits[pos] - '0') : 9;
    long long ans = 0;
    for (int d = 0; d <= hi; ++d) {
        if (!lz && d == last) continue;            // adjacency check
        int nlz = lz && (d == 0);
        int nlast = nlz ? 10 : d;
        ans += solve(pos + 1, tight && (d == hi), nlz, nlast);
    }
    if (!tight && !lz) memo[pos][key] = ans;
    return ans;
}

    Pitfall. If you forget the lz guard on the equality check, you'll reject any number starting with a digit other than the virtual leading zero — wait, actually the opposite: you'll accept nothing starting with $0$. The symptom is missing valid numbers. Always stress-test digit DP against brute force for $N \leq 10^4$.

    Problem 6: Binary Digit DP — Count Integers $\leq N$ With Popcount $= k$

    Given $N$ (up to $10^{18}$) and $k$, count $x \in [0, N]$ whose binary representation has exactly $k$ set bits.

    Digit DP doesn't have to be decimal. Any positional number system works; the most common alternative is base $2$. The template is identical with hi ∈ {0, 1} and $D = \lceil \log_2 N \rceil + 1$ (typically 60 or 64).

    int D, k; vector<int> bits;   // MSB first
long long memo[64][64];

long long solve(int pos, int tight, int ones) {
    if (pos == D) return ones == k ? 1 : 0;
    if (!tight && memo[pos][ones] != -1) return memo[pos][ones];

    int hi = tight ? bits[pos] : 1;
    long long ans = 0;
    for (int b = 0; b <= hi; ++b)
        ans += solve(pos + 1, tight && (b == hi), ones + b);
    if (!tight) memo[pos][ones] = ans;
    return ans;
}
long long countPopcount(long long N, int k_) {
    bits.clear();
    for (int i = 62; i >= 0; --i) bits.push_back((N >> i) & 1);
    D = bits.size(); k = k_;
    memset(memo, -1, sizeof memo);
    return solve(0, 1, 0);
}

    Closed-form cross-check. This particular problem actually has a $O(\log N)$ closed form using C(pos, k - ones) at each bit where $N$ has a $1$. Writing both and stress-testing them against each other is a great exercise — it reveals off-by-ones in either.

    Recap: Four State-Design Idioms

    • Additive accumulator (digit sum, sum of squares): state field ranges over $[0, 9D]$.
    • Modular accumulator (divisibility): state field ranges over $[0, K-1]$. Update: $r \to (10r + d) \bmod K$.
    • Bitmask accumulator (which digits appeared): state field ranges over $[0, 2^{10})$.
    • Last-digit memory (adjacency, monotonicity): state field ranges over $\{-1, 0, 1, \ldots, 9\}$.

    Almost every digit DP problem is one of these four, sometimes two of them composed. Start by asking: what does the problem care about at the boundary between the digits I've placed and the digits I haven't? Whatever the answer is, that's your state field.
