    Greedy on Arrays

    March 2026 · 25 min read

    A greedy algorithm builds a solution one piece at a time, always choosing the option that looks best right now. It never reconsiders a past decision. This sounds reckless, and for many problems it is. But for a specific class of problems, greedy gives you the optimal answer in $O(n)$ or $O(n \log n)$ time, far faster than brute force or dynamic programming. This post explains what makes a problem greedy-friendly, how to prove correctness with the exchange argument, and works through three classic array problems: Jump Game, Gas Station, and Activity Selection.

    What Does Greedy Mean?

    In a greedy algorithm, you make a locally optimal choice at each step, hoping that these local choices lead to a globally optimal solution. There is no backtracking. Once you pick an option, you commit to it and move on.

    Define the Choice

    At each step, identify the set of candidates. Pick the one that maximizes (or minimizes) some local criterion.

    Commit

    Include the chosen candidate in the solution. Do not revisit this decision.

    Reduce

    The problem shrinks. Repeat from Step 1 on the remaining subproblem.

    Compare this with dynamic programming, where you explore all choices and combine subproblem results. Greedy is faster because it commits immediately, but it only works when the problem has the right structure.

    When Does Greedy Work?

    A greedy algorithm produces an optimal solution when the problem has two properties:

    1. Greedy Choice Property

    A globally optimal solution can be assembled by making locally optimal choices. In other words, the best local decision at each step does not disqualify you from reaching the overall best answer.

    2. Optimal Substructure

    After making a greedy choice, the remaining problem is a smaller instance of the same type. The optimal solution to the whole problem contains optimal solutions to its subproblems.

    Both properties are needed. Optimal substructure alone is not enough (dynamic programming also uses it). The greedy choice property is what lets you commit to one choice without exploring alternatives.

    The Exchange Argument

    How do you prove that a greedy choice is correct? The most common technique is the exchange argument. The idea is simple but powerful:

    1. Assume there exists an optimal solution $O$ that does not include the greedy choice $g$.
    2. Show that you can swap some element in $O$ with $g$ to produce a solution $O'$ that is at least as good as $O$.
    3. Since $O'$ is also optimal and includes $g$, there always exists an optimal solution that starts with the greedy choice.
    4. Apply the argument inductively to every subsequent step.

    This is how you turn intuition ("sort by end time seems right") into a formal proof. We will use this argument for the Activity Selection problem below.

    Greedy vs Dynamic Programming

    Aspect Greedy Dynamic Programming
    Choices One best choice per step All choices explored
    Backtracking Never Implicit (via subproblem cache)
    Correctness Requires greedy choice property Always correct if recurrence is right
    Time Usually $O(n)$ or $O(n \log n)$ Usually $O(n^2)$ or $O(n \times W)$
    Space Usually $O(1)$ Usually $O(n)$ or more
    Warning: If a problem does not have the greedy choice property, a greedy approach will give wrong answers. The 0/1 Knapsack problem is a classic example where greedy (by value/weight ratio) fails but DP succeeds.

    Jump Game (LeetCode 55)

    You are given an array of non-negative integers. Each element represents the maximum jump length from that position. Starting at index 0, determine if you can reach the last index.

    Example: [2, 3, 1, 1, 4]. From index 0, you can jump up to 2 positions. From index 1, up to 3 positions. The question: can you reach index 4?

    The Greedy Insight

    Maintain a variable maxReach that tracks the farthest index reachable so far. Scan from left to right. At each index $i$:

    • If $i > \text{maxReach}$, you cannot reach index $i$. The answer is false.
    • Otherwise, update $\text{maxReach} = \max(\text{maxReach},\; i + \text{arr}[i])$.
    • If $\text{maxReach} \geq n - 1$, the answer is true.

    You never backtrack. You simply extend your reachable frontier as far as possible. This is the greedy choice: at each step, expand the reachable range to the maximum.

    Step-Through Animation

    Array: [2, 3, 1, 1, 4]. Watch how maxReach expands as we scan left to right. Green cells are reachable. The yellow cell is the one currently being processed.

    Press Step to begin.

    Code

    bool canJump(vector<int>& nums) {
    int n = nums.size();
    int maxReach = 0;

    for (int i = 0; i < n; i++) {
        if (i > maxReach) return false;
        maxReach = max(maxReach, i + nums[i]);
        if (maxReach >= n - 1) return true;
    }
    return true;
}

    Time: $O(n)$. Single pass. Space: $O(1)$.

    Why Greedy Works Here

    The greedy choice property holds because reachability is monotonic. If you can reach index $i$, you can also reach every index before $i$. Extending maxReach to the largest possible value can never hurt: it only opens up more positions. There is no scenario where jumping shorter would let you reach farther overall.

    When It Fails: Unreachable Case

    Consider [3, 2, 1, 0, 4]. The element at index 3 is 0, which means you are stuck there. Let us trace it:

    • $i = 0$: maxReach = max(0, 0+3) = 3
    • $i = 1$: maxReach = max(3, 1+2) = 3
    • $i = 2$: maxReach = max(3, 2+1) = 3
    • $i = 3$: maxReach = max(3, 3+0) = 3
    • $i = 4$: 4 > 3, so we cannot reach index 4. Return false.

    Gas Station (LeetCode 134)

    There are $n$ gas stations along a circular route. Station $i$ has gas[i] units of fuel. Traveling from station $i$ to station $i+1$ costs cost[i] units. Starting with an empty tank, find the station index where you can begin a complete circuit, or return $-1$ if no such station exists.

    Key Observations

    1. If the total gas is less than the total cost, no solution exists. You simply cannot complete the circuit regardless of where you start.
    2. If the total gas is at least the total cost, a solution always exists and is unique.
    3. The greedy insight: if starting from station $s$ you run out of fuel at station $f$, then no station between $s$ and $f$ (inclusive) can be a valid start either. So you skip all of them and try $f + 1$.

    Circular Array Visual

    Five stations on a circular route. Each station shows net = gas[i] - cost[i]. Green means surplus, red means deficit. Watch the algorithm find the valid starting station.

    Press Step to begin.

    Code

    int canCompleteCircuit(vector<int>& gas, vector<int>& cost) {
    int n = gas.size();
    int totalSurplus = 0;
    int currentSurplus = 0;
    int start = 0;

    for (int i = 0; i < n; i++) {
        int net = gas[i] - cost[i];
        totalSurplus += net;
        currentSurplus += net;

        if (currentSurplus < 0) {
            // Cannot start from 'start' through 'i'
            start = i + 1;
            currentSurplus = 0;
        }
    }

    return (totalSurplus >= 0) ? start : -1;
}

    Time: $O(n)$. Single pass. Space: $O(1)$.

    Why Greedy Works Here

    If you start at station $s$ and run out at station $f$, then for every station $m$ between $s$ and $f$, arriving at $m$ from $s$ means you had non-negative fuel at $m$. Starting fresh at $m$ with zero fuel is even worse (you lose the surplus accumulated from $s$ to $m$). So no station in $[s, f]$ can work. This lets you skip ahead to $f + 1$ safely.

    Uniqueness: When the total gas equals or exceeds the total cost, the starting station is unique. This is because the prefix sum of net gains has exactly one global minimum, and the valid start is the station right after that minimum.

    Activity Selection

    Given $n$ activities, each with a start time and end time, select the maximum number of non-overlapping activities. Two activities overlap if one starts before the other ends.

    The Greedy Strategy

    1. Sort all activities by their end time in ascending order.
    2. Pick the first activity (it ends earliest).
    3. For each subsequent activity, if its start time is greater than or equal to the end time of the last selected activity, select it.
    4. Otherwise, skip it.

    Why Sort by End Time?

    Choosing the activity that ends earliest leaves the most room for future activities. This is the greedy choice. We can prove it with the exchange argument:

    Suppose an optimal solution $O$ does not include the activity $a$ with the earliest end time. Let $b$ be the first activity in $O$. Since $a$ ends no later than $b$, we can replace $b$ with $a$ in $O$. This does not cause any new overlaps (since $a$ ends at most when $b$ does), so the new solution $O'$ is still valid and has the same size. Therefore, there exists an optimal solution that includes the earliest-ending activity.

    Visual: Activity Selection

    Six activities shown as horizontal bars. The algorithm sorts by end time, then greedily selects non-overlapping activities. Green bars are selected. Red dashed bars are rejected due to overlap.

    Press Step to begin.

    Code

    int activitySelection(vector<pair<int,int>>& activities) {
    // Sort by end time
    sort(activities.begin(), activities.end(),
         [](auto& a, auto& b){ return a.second < b.second; });

    int count = 1;
    int lastEnd = activities[0].second;

    for (int i = 1; i < (int)activities.size(); i++) {
        if (activities[i].first >= lastEnd) {
            count++;
            lastEnd = activities[i].second;
        }
    }
    return count;
}

    Time: $O(n \log n)$ for sorting. Space: $O(1)$ extra (assuming in-place sort).

    Common Mistakes with Greedy

    Mistake 1: No proof of correctness. "It seems greedy should work" is not a proof. Always verify the greedy choice property, even informally. Try small counterexamples.
    Mistake 2: Wrong sorting criterion. In activity selection, sorting by start time or by duration gives wrong answers. The criterion matters.
    Mistake 3: Applying greedy to DP problems. The 0/1 Knapsack, Longest Common Subsequence, and Edit Distance do not have the greedy choice property. Greedy will give suboptimal results.

    Key Takeaways

    • Greedy algorithms make the locally optimal choice at each step without backtracking.
    • They work when the problem has both the greedy choice property and optimal substructure.
    • The exchange argument is the standard way to prove correctness: show you can swap any non-greedy choice with the greedy one without worsening the solution.
    • Jump Game uses greedy reachability expansion in $O(n)$ time.
    • Gas Station uses a single pass with prefix surplus tracking to find the unique starting point.
    • Activity Selection sorts by end time and greedily picks non-overlapping activities.
    • Always verify the greedy choice property. If it does not hold, use dynamic programming instead.
