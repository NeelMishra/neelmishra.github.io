    Greedy Patterns

    March 2026 · 30 min read

    The previous post covered what greedy means and worked through three foundational problems. This post presents five more greedy patterns that appear frequently in interviews and contests. Each problem follows the same structure: identify the greedy criterion, sort (if needed), scan and commit. Every pattern includes an SVG visual and complete C++ code.

    1. Assign Cookies (LeetCode 455)

    You have a set of children, each with a greed factor $g[i]$ (minimum cookie size they will accept). You have a set of cookies, each with size $s[j]$. A child is satisfied if they get a cookie with size at least $g[i]$. Maximize the number of satisfied children. Each child gets at most one cookie, and each cookie goes to at most one child.

    Greedy Strategy

    1. Sort both arrays in ascending order.
    2. Use two pointers: one for children, one for cookies.
    3. For the current child, try the smallest available cookie. If it satisfies them, assign it and move both pointers. If not, discard the cookie (it is too small for anyone remaining) and try the next one.

    The greedy choice: always give the smallest sufficient cookie to the least greedy child. This preserves larger cookies for greedier children.

    Visual

    Children greed: [1, 2, 3]. Cookies: [1, 1, 3]. Both sorted. Watch the two-pointer matching.

    Press Step to begin.

    Code

    int findContentChildren(vector<int>& g, vector<int>& s) {
    sort(g.begin(), g.end());
    sort(s.begin(), s.end());

    int child = 0, cookie = 0;
    while (child < (int)g.size() && cookie < (int)s.size()) {
        if (s[cookie] >= g[child]) {
            child++;  // child satisfied
        }
        cookie++;     // cookie used or discarded
    }
    return child;
}

    Time: $O(n \log n + m \log m)$ for sorting. Space: $O(1)$ extra.

    2. Minimum Platforms

    Given arrival and departure times of $n$ trains at a station, find the minimum number of platforms required so that no train has to wait. If a train arrives at the same time another departs, they can share a platform.

    Greedy Strategy

    1. Sort arrival times and departure times independently.
    2. Use two pointers. If the next event is an arrival, a train comes in: increment the platform count. If the next event is a departure, a train leaves: decrement the count.
    3. Track the maximum platform count at any point.

    The insight: by sorting events chronologically, you process them in the order they happen. The peak overlap is the answer.

    Visual

    Arrivals: [900, 940, 950, 1100, 1500, 1800]. Departures: [910, 1200, 1120, 1130, 1900, 2000]. Watch the platform count rise and fall as trains arrive and depart.

    Press Step to begin.

    Code

    int minPlatforms(vector<int>& arrival, vector<int>& departure) {
    int n = arrival.size();
    sort(arrival.begin(), arrival.end());
    sort(departure.begin(), departure.end());

    int platforms = 0, maxPlatforms = 0;
    int i = 0, j = 0;

    while (i < n) {
        if (arrival[i] <= departure[j]) {
            platforms++;
            i++;
        } else {
            platforms--;
            j++;
        }
        maxPlatforms = max(maxPlatforms, platforms);
    }
    return maxPlatforms;
}

    Time: $O(n \log n)$. Space: $O(1)$ extra.

    Why independent sorting? We do not need to pair arrivals with their corresponding departures. We only need to know the chronological order of all events. Sorting them independently achieves this in $O(n \log n)$ without extra bookkeeping.

    3. Job Sequencing with Deadlines

    You have $n$ jobs. Each job has a deadline $d[i]$ (must be completed by this time slot) and a profit $p[i]$ (earned if completed). Each job takes exactly one time unit. You can do at most one job per time slot. Maximize total profit.

    Greedy Strategy

    1. Sort jobs by profit in descending order.
    2. For each job, try to schedule it in the latest available slot before its deadline. Scheduling late leaves earlier slots free for other jobs.
    3. If no slot is available, skip the job.

    Visual

    Jobs (sorted by profit): J1(profit=100, deadline=2), J2(profit=80, deadline=1), J3(profit=60, deadline=2), J4(profit=40, deadline=1). Time slots: [1, 2]. Watch the greedy scheduler fill slots from highest profit down.

    Press Step to begin.

    Code

    int jobSequencing(vector<pair<int,int>>& jobs) {
    // jobs[i] = {deadline, profit}
    // Sort by profit descending
    sort(jobs.begin(), jobs.end(),
         [](auto& a, auto& b){ return a.second > b.second; });

    int maxDeadline = 0;
    for (auto& j : jobs) maxDeadline = max(maxDeadline, j.first);

    vector<bool> slot(maxDeadline + 1, false);
    int totalProfit = 0;

    for (auto& j : jobs) {
        // Try latest available slot
        for (int t = j.first; t >= 1; t--) {
            if (!slot[t]) {
                slot[t] = true;
                totalProfit += j.second;
                break;
            }
        }
    }
    return totalProfit;
}

    Time: $O(n \log n + n \times d)$ where $d$ is the max deadline. Can be improved to $O(n \log n)$ with a disjoint set. Space: $O(d)$.

    4. Candy Distribution (LeetCode 135)

    There are $n$ children standing in a line. Each child has a rating. You must distribute candies such that:

    • Every child gets at least one candy.
    • A child with a higher rating than their neighbor gets more candies than that neighbor.

    Minimize the total number of candies.

    Greedy Strategy: Two Passes

    1. Initialize every child with 1 candy.
    2. Left pass: Scan left to right. If rating[i] > rating[i-1], set candy[i] = candy[i-1] + 1.
    3. Right pass: Scan right to left. If rating[i] > rating[i+1], set candy[i] = max(candy[i], candy[i+1] + 1).

    The left pass ensures every child with a higher rating than their left neighbor has more candies. The right pass does the same for the right neighbor. Taking the maximum in the right pass preserves the left-pass constraint.

    Visual

    Ratings: [1, 0, 2, 1, 3]. Watch the two passes build the candy array.

    Press Step to begin. Left pass first, then right pass.

    Code

    int candy(vector<int>& ratings) {
    int n = ratings.size();
    vector<int> candies(n, 1);

    // Left pass
    for (int i = 1; i < n; i++) {
        if (ratings[i] > ratings[i - 1]) {
            candies[i] = candies[i - 1] + 1;
        }
    }

    // Right pass
    for (int i = n - 2; i >= 0; i--) {
        if (ratings[i] > ratings[i + 1]) {
            candies[i] = max(candies[i], candies[i + 1] + 1);
        }
    }

    int total = 0;
    for (int c : candies) total += c;
    return total;
}

    Time: $O(n)$. Two linear passes. Space: $O(n)$ for the candy array.

    Why two passes? A single left-to-right pass cannot see future neighbors. The right-to-left pass corrects for the right neighbor constraint. The max in the right pass ensures we do not undo the left-pass work.

    5. Minimum Number of Coins

    Given an amount and a set of coin denominations, find the minimum number of coins to make change. Assume the coin system is canonical (e.g., standard currency denominations like 1, 5, 10, 25), where greedy works correctly.

    Greedy Strategy

    1. Sort denominations in descending order.
    2. For each denomination, use as many of that coin as possible (integer division).
    3. Update the remaining amount.
    Important: Greedy works for canonical coin systems (US coins, Indian currency, Euro coins). For arbitrary denominations like {1, 3, 4}, greedy fails. Example: amount = 6. Greedy picks 4+1+1 (3 coins), but optimal is 3+3 (2 coins). Use DP for non-canonical systems.

    Visual

    Denominations: [100, 50, 20, 10, 5, 2, 1] (Indian currency). Amount: 186. Watch greedy pick the largest coin first.

    Press Step to begin.

    Code

    int minCoins(int amount, vector<int>& coins) {
    // coins sorted descending for canonical systems
    sort(coins.begin(), coins.end(), greater<int>());

    int count = 0;
    for (int coin : coins) {
        if (amount == 0) break;
        int use = amount / coin;
        count += use;
        amount -= use * coin;
    }
    return (amount == 0) ? count : -1;
}

    Time: $O(d)$ where $d$ is the number of denominations. Space: $O(1)$.

    Pattern Summary

    Problem Greedy Criterion Sort By Time
    Assign Cookies Smallest sufficient cookie to least greedy child Both arrays ascending $O(n \log n)$
    Minimum Platforms Process events chronologically Arrivals and departures independently $O(n \log n)$
    Job Sequencing Highest profit job in latest available slot Profit descending $O(n \log n)$
    Candy Distribution Two passes: left neighbor, then right neighbor No sort needed $O(n)$
    Minimum Coins Largest denomination first Denomination descending $O(d)$

    Key Takeaways

    • Most greedy array problems follow the same pattern: sort by the right criterion, then scan linearly and commit.
    • The sorting criterion is the hardest part. Get it wrong and the algorithm produces incorrect results.
    • Assign Cookies and Minimum Platforms use two-pointer techniques after sorting.
    • Job Sequencing combines greedy ordering with slot-based scheduling.
    • Candy Distribution is a rare greedy problem that requires two passes instead of one.
    • Greedy coin change only works for canonical denominations. Always check before assuming greedy applies.
