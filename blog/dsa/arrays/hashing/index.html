    Hashing Fundamentals

    March 2026 · 25 min read

    Searching for a value in an unsorted array takes $O(n)$. Searching in a sorted array takes $O(\log n)$ with binary search. A hash table does it in $O(1)$ on average. This single idea, mapping keys to positions through a mathematical function, is the most practical data structure in all of programming. This post covers how hash tables work from the ground up: hash functions, collision handling with chaining and open addressing, the load factor, and how C++ exposes all of this through unordered_map and unordered_set.

    Motivation: The Two Sum Problem

    Before diving into theory, consider this classic problem: given an array of integers and a target sum, find two elements that add up to the target.

    Brute force: Check every pair. That is $O(n^2)$, far too slow for large inputs.

    Key insight: For each element x, you need to know whether target - x exists somewhere in the array. If you could answer that question in $O(1)$, the entire problem becomes $O(n)$.

    A hash table provides exactly that. As you scan the array left to right, insert each value into a hash table. Before inserting x, check if target - x is already in the table. If it is, you have found your pair.

    std::vector<int> twoSum(std::vector<int>& nums, int target) {
    std::unordered_map<int, int> seen;  // value -> index
    for (int i = 0; i < nums.size(); i++) {
        int complement = target - nums[i];
        if (seen.count(complement))
            return {seen[complement], i};
        seen[nums[i]] = i;
    }
    return {};  // no pair found
}

    This runs in $O(n)$ time and $O(n)$ space. The hash table turns a quadratic search into a linear one. But how does it achieve $O(1)$ lookups? That is the subject of this post.

    What Is a Hash Table?

    A hash table is an array of fixed size $m$ (the number of buckets). To store a key-value pair, you compute a hash function on the key to get an integer, then take that integer modulo $m$ to find which bucket the pair belongs in.

    bucket_index = hash(key) % m

    The core operations are:

    • Insert(key, value): Compute the bucket index, store the pair there.
    • Lookup(key): Compute the bucket index, search that bucket for the key.
    • Delete(key): Compute the bucket index, remove the pair from that bucket.

    If the hash function distributes keys evenly and the table is not too full, each bucket holds zero or one element, making all three operations $O(1)$.

    Hash Functions

    A hash function takes an input of arbitrary type and produces a fixed-size integer. A good hash function has two properties:

    1. Deterministic: The same key always produces the same hash value.
    2. Uniform distribution: Different keys spread evenly across the output range, minimizing collisions.

    Simple example: modular hashing

    For integer keys, the simplest hash function is:

    $$h(k) = k \bmod m$$

    where $m$ is the table size. If $m$ is a prime number, this tends to distribute keys more evenly. For example, with $m = 7$:

    KeyHash = key mod 7Bucket
    1010 mod 7 = 33
    2222 mod 7 = 11
    3131 mod 7 = 33
    44 mod 7 = 44
    1515 mod 7 = 11
    2828 mod 7 = 00

    Notice that keys 10 and 31 both map to bucket 3, and keys 22 and 15 both map to bucket 1. These are collisions, and handling them is the central challenge in hash table design.

    Hash functions for strings

    For strings, a common approach is polynomial rolling hash:

    $$h(s) = \left(\sum_{i=0}^{n-1} s[i] \cdot p^i\right) \bmod m$$

    where $p$ is a small prime (like 31 or 37) and $m$ is a large prime. Each character contributes to the hash weighted by its position, which means "abc" and "bac" produce different values.

    In practice: You almost never write your own hash function. C++ provides std::hash for built-in types, and you only need custom hash functions for user-defined types or composite keys.

    Collision Handling

    When two different keys map to the same bucket, we have a collision. There are two main strategies to resolve collisions.

    Strategy 1: Chaining (Separate Chaining)

    Each bucket stores a linked list (or any dynamic collection). When a collision occurs, the new element is appended to the list at that bucket.

    • Insert: Hash the key, append to the list at that bucket. $O(1)$.
    • Lookup: Hash the key, traverse the list at that bucket looking for the key. $O(k)$ where $k$ is the chain length.
    • Delete: Hash the key, find and remove it from the list. $O(k)$.

    If the hash function is good and the load factor is kept low, chains are short and operations are effectively $O(1)$. In the worst case (all keys hash to one bucket), it degrades to $O(n)$.

    Chaining Walkthrough

    Let us manually trace insertions into a hash table of size 7 with separate chaining.

    Insert 10

    hash(10) = 10 mod 7 = 3. Bucket 3 is empty, so 10 goes directly into bucket 3.

    Insert 22

    hash(22) = 22 mod 7 = 1. Bucket 1 is empty, so 22 goes directly into bucket 1.

    Insert 31 (collision)

    hash(31) = 31 mod 7 = 3. Bucket 3 already contains 10. With chaining, 31 is appended to the chain: bucket 3 now holds [10, 31].

    Keys 10 and 31 both hash to bucket 3, forming a chain of length 2.

    Insert 15 (another collision)

    hash(15) = 15 mod 7 = 1. Bucket 1 already contains 22. Key 15 is appended: bucket 1 now holds [22, 15].

    Two collisions so far. Both at buckets that already had occupants.

    Strategy 2: Open Addressing

    Instead of storing chains, open addressing keeps everything inside the array itself. When a collision occurs, the algorithm probes for the next available slot using a probing sequence.

    Linear probing

    The simplest probing strategy. If bucket $h$ is occupied, try $h+1$, then $h+2$, and so on (wrapping around).

    function insert(table, key, value): idx = hash(key) % m while table[idx] is occupied: idx = (idx + 1) % m table[idx] = (key, value)

    Linear probing suffers from primary clustering: occupied slots tend to clump together, making future insertions slower as they must walk through long runs of filled slots.

    Quadratic probing

    Instead of stepping by 1 each time, step by $1, 4, 9, 16, \ldots$ (squares). The probe sequence is:

    $$h_i = (h + i^2) \bmod m$$

    This reduces primary clustering but can still suffer from secondary clustering where keys that hash to the same bucket follow the same probe sequence.

    Double hashing

    Use a second hash function to determine the step size:

    $$h_i = (h_1(k) + i \cdot h_2(k)) \bmod m$$

    This eliminates both primary and secondary clustering, giving the most uniform distribution of all open addressing methods.

    MethodProbe sequenceClusteringCache performance
    Linear probingh, h+1, h+2, ...Primary clusteringExcellent (sequential memory)
    Quadratic probingh, h+1, h+4, h+9, ...Secondary clusteringGood
    Double hashingh1, h1+h2, h1+2*h2, ...MinimalPoor (random jumps)

    Chaining vs Open Addressing

    PropertyChainingOpen Addressing
    StorageArray of linked listsSingle flat array
    Load factor can exceed 1.0Yes (chains grow)No (table must have empty slots)
    Cache performancePoor (pointer chasing)Good (data is contiguous)
    DeletionSimple (remove from list)Needs tombstones or special handling
    Worst case per operation$O(n)$ if all keys collide$O(n)$ if table is nearly full
    Used by C++ STLYes (unordered_map)No (by default)
    C++ choice: The C++ standard library uses separate chaining for unordered_map and unordered_set. This is why these containers allow a load factor greater than 1.0 and why deletion is straightforward.

    Load Factor

    The load factor $\alpha$ is the ratio of the number of stored elements to the number of buckets:

    $$\alpha = \frac{n}{m}$$

    where $n$ is the number of elements and $m$ is the number of buckets. The load factor directly controls performance:

    • For chaining, the average chain length is $\alpha$. So the expected time per operation is $O(1 + \alpha)$.
    • For open addressing, the expected number of probes is approximately $\frac{1}{1 - \alpha}$. As $\alpha$ approaches 1.0, performance degrades sharply.

    When the load factor exceeds a threshold (typically 0.7 to 1.0), the hash table rehashes: it allocates a larger array (usually double the size), recomputes the hash of every element, and inserts them into the new array. Rehashing is $O(n)$ but happens infrequently, so the amortized cost per insertion remains $O(1)$.

    Load factor in C++ unordered_map

    The default maximum load factor in std::unordered_map is 1.0. You can query and adjust it:

    std::unordered_map<int, int> mp;
mp.max_load_factor();     // returns 1.0 by default
mp.load_factor();         // current n / bucket_count
mp.bucket_count();        // number of buckets
mp.max_load_factor(0.5);  // force rehash at 50% fill
    Performance trap: When you know the approximate number of elements in advance, call mp.reserve(n) to pre-allocate enough buckets. This avoids repeated rehashing during bulk insertions, which can be the difference between a solution passing or getting TLE in competitive programming.

    C++ unordered_map and unordered_set

    The C++ standard library provides two hash-based containers for everyday use.

    unordered_map

    A hash table that stores key-value pairs. Keys are unique. Average $O(1)$ for insert, lookup, and delete.

    #include <unordered_map>
#include <string>

std::unordered_map<std::string, int> ages;

// Insert
ages["Alice"] = 30;
ages.insert({"Bob", 25});
ages.emplace("Carol", 28);

// Lookup
if (ages.count("Alice"))             // 1 if found, 0 otherwise
    int a = ages["Alice"];           // returns 30
auto it = ages.find("Bob");          // iterator to element
if (it != ages.end())
    int b = it->second;             // 25

// Delete
ages.erase("Carol");

// Iterate
for (auto& [name, age] : ages)
    std::cout << name << ": " << age << "\n";

    unordered_set

    A hash table that stores unique keys with no associated values. It is the hash-based equivalent of std::set.

    #include <unordered_set>

std::unordered_set<int> seen;

seen.insert(10);
seen.insert(20);
seen.insert(10);    // duplicate, ignored

if (seen.count(20))
    std::cout << "20 is in the set\n";

seen.erase(10);
std::cout << seen.size();   // 1

    When to use which

    ContainerOrdered?Average lookupUse when
    unordered_mapNo$O(1)$Need fast key-value lookups, order does not matter
    unordered_setNo$O(1)$Need fast membership checks, no values needed
    mapYes (sorted)$O(\log n)$Need keys in sorted order, or range queries
    setYes (sorted)$O(\log n)$Need sorted unique elements
    Worst-case pitfall: unordered_map has $O(n)$ worst-case per operation because adversarial inputs can force all keys into one bucket. In competitive programming, if you get TLE with unordered_map, try a custom hash or switch to map. A common trick is to XOR with a random constant: 
    struct custom_hash {
    size_t operator()(long long x) const {
        x ^= x >> 16;
        x *= 0x45d9f3b;
        x ^= x >> 16;
        return x;
    }
};
std::unordered_map<long long, int, custom_hash> mp;

    The worst case occurs when all keys collide into the same bucket. With a good hash function and a reasonable load factor, this is extremely unlikely in practice.

    Lookup Animation

    After building the hash table with chaining from the earlier animation, let us look up a key. To find key 31:

    1. Compute hash(31) = 31 mod 7 = 3.
    2. Go to bucket 3. The chain contains [10, 31].
    3. Compare with 10. No match.
    4. Compare with 31. Match found.

    The lookup traversed two elements in the chain. If the chain had 100 elements, it would take 100 comparisons in the worst case for that bucket. This is why keeping chains short (low load factor) matters.

    Key Takeaways

    • A hash table maps keys to array indices via a hash function, giving $O(1)$ average-case operations.
    • Collisions are unavoidable. Chaining stores colliding elements in linked lists. Open addressing finds the next empty slot.
    • The load factor $\alpha = n/m$ controls performance. Keep it below 1.0 (or below 0.7 for open addressing).
    • C++ provides unordered_map (key-value) and unordered_set (keys only), both using chaining internally.
    • Use reserve() when you know the number of elements to avoid repeated rehashing.
    • For competitive programming, consider a custom hash to defend against adversarial test cases.
