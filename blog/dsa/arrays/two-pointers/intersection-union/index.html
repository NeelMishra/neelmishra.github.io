    Intersection & Union of Arrays

    March 2026 · 28 min read

    Introduction

    Finding common or combined elements between two arrays is a fundamental operation that appears everywhere: database joins, search engine index merging, recommendation systems, and of course coding interviews. The two classic operations are intersection (elements present in both arrays) and union (all distinct elements from both arrays combined).

    There are two main families of solutions: sorting + two pointers and hash-based approaches. Each has its strengths, and understanding both lets you pick the right tool for the constraints at hand.

    This post covers five problems in depth, with step-through animations, C++ implementations, and a detailed comparison of the two approaches.

    Prerequisites: Familiarity with sorting, basic hash maps (unordered_map, unordered_set), and the two pointer technique on sorted arrays.

    1. Intersection of Two Arrays (LeetCode 349)

    Problem: Given two integer arrays nums1 and nums2, return an array of their intersection. Each element in the result must be unique, and you may return the result in any order.

    Approach A: Hash Set

    Insert all elements of nums1 into an unordered_set. Then iterate through nums2; if an element exists in the set, add it to the result and remove it from the set (to avoid duplicates).

    vector<int> intersection(vector<int>& nums1, vector<int>& nums2) {
    unordered_set<int> s(nums1.begin(), nums1.end());
    vector<int> result;
    for (int x : nums2) {
        if (s.erase(x)) {       // erase returns 1 if found
            result.push_back(x);
        }
    }
    return result;
}

    Hash Set Approach

    Time: $O(n + m)$ average, where $n$ and $m$ are the sizes of the two arrays.

    Space: $O(\min(n, m))$ for the hash set (build it from the smaller array for efficiency).

    Approach B: Sort + Two Pointers

    Sort both arrays. Walk two pointers i and j starting at the beginning of each array. At each step:

    • If nums1[i] == nums2[j]: add to result (if not a duplicate of the last added), advance both pointers.
    • If nums1[i] < nums2[j]: advance i (the smaller side needs to catch up).
    • If nums1[i] > nums2[j]: advance j.
    vector<int> intersection(vector<int>& nums1, vector<int>& nums2) {
    sort(nums1.begin(), nums1.end());
    sort(nums2.begin(), nums2.end());
    vector<int> result;
    int i = 0, j = 0;
    while (i < (int)nums1.size() && j < (int)nums2.size()) {
        if (nums1[i] == nums2[j]) {
            if (result.empty() || result.back() != nums1[i])
                result.push_back(nums1[i]);
            i++; j++;
        } else if (nums1[i] < nums2[j]) {
            i++;
        } else {
            j++;
        }
    }
    return result;
}

    Sort + Two Pointers Approach

    Time: $O(n \log n + m \log m)$ for sorting, then $O(n + m)$ for the scan.

    Space: $O(1)$ extra (ignoring sort internals and result storage).

    2. Intersection with Duplicates (LeetCode 350)

    Problem: Given two arrays, return their intersection including duplicates. Each element in the result should appear as many times as it shows in both arrays.

    Example: nums1 = [1,2,2,1], nums2 = [2,2] gives [2,2].

    Approach A: Sort + Two Pointers

    Sort both arrays. Walk two pointers. When values match, always add (no dedup needed since we want duplicates). When they differ, advance the smaller one.

    vector<int> intersect(vector<int>& nums1, vector<int>& nums2) {
    sort(nums1.begin(), nums1.end());
    sort(nums2.begin(), nums2.end());
    vector<int> result;
    int i = 0, j = 0;
    while (i < (int)nums1.size() && j < (int)nums2.size()) {
        if (nums1[i] == nums2[j]) {
            result.push_back(nums1[i]);
            i++; j++;
        } else if (nums1[i] < nums2[j]) {
            i++;
        } else {
            j++;
        }
    }
    return result;
}

    349 vs 350

    In problem 349, we skip duplicates in the result. In 350, every matching pair is recorded. The two-pointer code is almost identical; the only change is removing the dedup check.

    Approach B: Frequency Map

    Count frequencies of nums1 in a map. For each element in nums2, if count is positive, add to result and decrement.

    vector<int> intersect(vector<int>& nums1, vector<int>& nums2) {
    unordered_map<int, int> freq;
    for (int x : nums1) freq[x]++;
    vector<int> result;
    for (int x : nums2) {
        if (freq[x] > 0) {
            result.push_back(x);
            freq[x]--;
        }
    }
    return result;
}

    LeetCode 350 Follow-ups

    Q: What if nums1 is sorted? Use binary search for each element of nums2.

    Q: What if both are sorted? Two-pointer is optimal at $O(n + m)$.

    Q: What if nums2 is stored on disk and memory is limited? Load chunks of nums2, process against a frequency map of nums1.

    3. Union of Two Sorted Arrays

    Problem: Given two sorted arrays, return a single sorted array of all unique elements present in either array.

    This is the merge step of merge-sort, but we skip duplicates. Two pointers walk forward, always picking the smaller element and skipping any value equal to the last written output.

    vector<int> sortedUnion(const vector<int>& a, const vector<int>& b) {
    vector<int> result;
    int i = 0, j = 0;
    while (i < (int)a.size() && j < (int)b.size()) {
        // Skip duplicates in a
        if (i > 0 && a[i] == a[i - 1]) { i++; continue; }
        // Skip duplicates in b
        if (j > 0 && b[j] == b[j - 1]) { j++; continue; }

        if (a[i] < b[j]) {
            result.push_back(a[i]);
            i++;
        } else if (a[i] > b[j]) {
            result.push_back(b[j]);
            j++;
        } else {
            result.push_back(a[i]);  // equal: take once
            i++; j++;
        }
    }
    // Drain remaining from a
    while (i < (int)a.size()) {
        if (result.empty() || result.back() != a[i])
            result.push_back(a[i]);
        i++;
    }
    // Drain remaining from b
    while (j < (int)b.size()) {
        if (result.empty() || result.back() != b[j])
            result.push_back(b[j]);
        j++;
    }
    return result;
}

    Union of Two Sorted Arrays

    Time: $O(n + m)$ since each pointer moves forward at each step.

    Space: $O(1)$ extra beyond the output array.

    Why not just merge and deduplicate? You could merge both arrays into one and then remove consecutive duplicates. But that takes $O(n + m)$ extra space for the merged temporary. The two-pointer approach does it in a single pass with no temporary array.

    4. Find Common Characters (LeetCode 1002)

    Problem: Given a list of strings, find all characters that appear in every string (including duplicates). Return a list of these characters.

    Example: ["bella", "label", "roller"] returns ["e", "l", "l"].

    Approach: Frequency Intersection

    This is a generalized intersection problem. Instead of intersecting two arrays, we intersect frequency tables across multiple strings.

    1. Compute a frequency array (size 26) for the first string. Call this minFreq.
    2. For each subsequent string, compute its frequency array. For each character, update minFreq[c] = min(minFreq[c], freq[c]).
    3. At the end, minFreq[c] holds the minimum count of character c across all strings. Emit each character that many times.
    vector<string> commonChars(vector<string>& words) {
    int minFreq[26];
    // Initialize from first word
    fill(minFreq, minFreq + 26, 0);
    for (char c : words[0]) minFreq[c - 'a']++;

    for (int w = 1; w < (int)words.size(); w++) {
        int freq[26] = {};
        for (char c : words[w]) freq[c - 'a']++;
        for (int i = 0; i < 26; i++)
            minFreq[i] = min(minFreq[i], freq[i]);
    }

    vector<string> result;
    for (int i = 0; i < 26; i++) {
        for (int j = 0; j < minFreq[i]; j++)
            result.push_back(string(1, 'a' + i));
    }
    return result;
}

    Common Characters

    Time: $O(n \cdot L)$ where $n$ is the number of words and $L$ is average word length.

    Space: $O(1)$ since frequency arrays are always size 26.

    5. Minimum Difference Between Largest and Smallest Value

    Problem: Given an array of size $n$, you can remove exactly $k$ elements. Find the minimum possible difference between the largest and smallest value in the remaining array.

    A classic variant: LeetCode 1509, "Minimum Difference Between Largest and Smallest Value in Three Moves" ($k = 3$).

    Sorted Two-Pointer Approach

    Sort the array. After removing $k$ elements optimally, we are left with a contiguous window of $n - k$ elements (because removing from the extremes is always optimal). Slide a window of size $n - k$ across the sorted array and track the minimum arr[right] - arr[left].

    int minDifference(vector<int>& nums, int k) {
    int n = nums.size();
    if (n <= k + 1) return 0;  // can remove all but one
    sort(nums.begin(), nums.end());
    int windowSize = n - k;
    int minDiff = INT_MAX;
    for (int i = 0; i + windowSize - 1 < n; i++) {
        minDiff = min(minDiff, nums[i + windowSize - 1] - nums[i]);
    }
    return minDiff;
}

    Why a Contiguous Window?

    In a sorted array, the smallest range containing exactly $n - k$ elements must be a contiguous subarray. Removing elements from the interior would only increase the range (or leave it unchanged), never decrease it. So the optimal strategy always removes from the ends.

    Minimum Difference

    Time: $O(n \log n)$ for sorting, $O(n)$ for the scan.

    Space: $O(1)$ extra.

    Worked Example

    Array: [9, 4, 1, 7, 3, 6, 2], $k = 2$. After sorting: [1, 2, 3, 4, 6, 7, 9]. Window size = 5.

    6. Two Pointers vs Hash-Based Intersection

    Both approaches solve the intersection problem, but they excel in different scenarios. Here is a detailed comparison.

    Criterion Sort + Two Pointers Hash Set / Map
    Time complexity $O(n \log n + m \log m)$ $O(n + m)$ average
    Space complexity $O(1)$ extra $O(\min(n, m))$
    Input already sorted? $O(n + m)$ total; optimal Still $O(n + m)$; sorting advantage wasted
    Worst-case time Always $O(n \log n + m \log m)$ $O(n \cdot m)$ if hash collisions degrade
    Cache performance Excellent (sequential access) Poor (random access in hash table)
    Preserves order? Sorted order (may differ from input) Insertion order of matches
    External / streaming data Works if both streams are sorted Works if one fits in memory
    Best for Large sorted data, memory-constrained Small unsorted data, average-case speed
    Rule of thumb: If the arrays are already sorted (or sorting is acceptable), prefer two pointers for its guaranteed $O(n + m)$ scan and $O(1)$ space. If the arrays are unsorted and you need average-case speed on small to medium data, a hash set is simpler to code and slightly faster in practice.
    Watch out: Hash-based approaches have worst-case $O(n \cdot m)$ time due to hash collisions. In competitive programming or adversarial inputs, the two-pointer approach is safer. In C++, unordered_set can be attacked with collision-inducing inputs; consider using a custom hash or just sorting.

    7. Follow-Up: What if One Array is Much Larger?

    Suppose nums1 has $n$ elements and nums2 has $m$ elements, with $n \ll m$, and both are sorted. The two-pointer approach takes $O(n + m)$, which is dominated by the larger array. Can we do better?

    Binary Search Per Element

    For each element in the smaller array, binary search for it in the larger array. This takes $O(n \log m)$, which is faster than $O(n + m)$ when $n$ is very small relative to $m$.

    vector<int> intersectAsymmetric(const vector<int>& small,
                                  const vector<int>& big) {
    vector<int> result;
    int lastFound = -1;  // track position in big to handle dups
    for (int i = 0; i < (int)small.size(); i++) {
        // Skip duplicates in small (for problem 349)
        if (i > 0 && small[i] == small[i - 1]) continue;

        // Binary search in big starting from lastFound+1
        int lo = lastFound + 1, hi = (int)big.size() - 1;
        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (big[mid] < small[i]) lo = mid + 1;
            else hi = mid - 1;
        }
        if (lo < (int)big.size() && big[lo] == small[i]) {
            result.push_back(small[i]);
            lastFound = lo;
        }
    }
    return result;
}

    Binary Search Intersection

    Time: $O(n \log m)$ where $n$ is the smaller array size.

    Space: $O(1)$ extra.

    When Does Binary Search Win?

    Compare the two approaches as a function of the size ratio:

    $n$ (small)$m$ (large)Two Pointers $O(n+m)$Binary Search $O(n \log m)$Winner
    101,000,0001,000,010~200Binary Search
    1,0001,000,0001,001,000~20,000Binary Search
    100,0001,000,0001,100,000~2,000,000Two Pointers
    500,0001,000,0001,500,000~10,000,000Two Pointers

    The crossover point is roughly when $n \log m \approx n + m$, i.e., when $n \approx \frac{m}{\log m}$. Below that ratio, binary search is faster.

    Practical guideline: If the smaller array has fewer than $\frac{m}{\log m}$ elements, use binary search. Otherwise, use the standard two-pointer merge.

    Galloping (Exponential) Search

    An even more refined approach: for each element in the smaller array, start with exponential search (doubling the step size) in the larger array, then binary search within the found range. This achieves $O(n \log \frac{m}{n})$ time, which is optimal for the sorted intersection problem.

    // Find first index >= target in arr[lo..hi]
int lowerBound(const vector<int>& arr, int lo, int hi, int target) {
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (arr[mid] < target) lo = mid + 1;
        else hi = mid - 1;
    }
    return lo;
}

vector<int> intersectGallop(const vector<int>& small,
                             const vector<int>& big) {
    vector<int> result;
    int pos = 0;
    for (int i = 0; i < (int)small.size(); i++) {
        if (i > 0 && small[i] == small[i - 1]) continue;

        // Exponential search from pos
        int step = 1;
        while (pos + step < (int)big.size()
               && big[pos + step] < small[i])
            step *= 2;

        int lo = pos + step / 2;
        int hi = min(pos + step, (int)big.size() - 1);
        int idx = lowerBound(big, lo, hi, small[i]);

        if (idx < (int)big.size() && big[idx] == small[i]) {
            result.push_back(small[i]);
            pos = idx + 1;
        } else {
            pos = idx;
        }
    }
    return result;
}

    Galloping Search Intersection

    Time: $O\!\left(n \log \frac{m}{n}\right)$, which smoothly interpolates between $O(n \log m)$ (when $n = 1$) and $O(n + m)$ (when $n \approx m$).

    Space: $O(1)$ extra.

    Summary

    • Intersection (unique): Sort + two pointers with dedup, or hash set. Both $O(n + m)$ scan after preprocessing.
    • Intersection (with duplicates): Same two-pointer logic without the dedup check, or a frequency map.
    • Union (sorted): Merge-like two-pointer pass, skipping consecutive duplicates. $O(n + m)$ time, $O(1)$ extra space.
    • Common characters: Frequency intersection via element-wise minimum across all frequency arrays.
    • Minimum difference: Sort, then slide a fixed-size window to find the smallest range.
    • Two pointers vs hashing: Two pointers wins on sorted data, worst-case guarantees, and memory. Hashing wins on unsorted data in the average case.
    • Asymmetric sizes: When one array is much smaller, binary search per element ($O(n \log m)$) or galloping search ($O(n \log \frac{m}{n})$) outperforms a linear scan.
