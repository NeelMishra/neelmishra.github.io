    Two Sum & Pair Patterns

    March 2026 · 30 min read

    Introduction

    The "Two Sum" family of problems is one of the most common interview topics. The core question is simple: given an array, find a pair of elements satisfying some condition (exact sum, bounded sum, exact difference, etc.). While hash maps offer one approach, the two pointer technique on a sorted array provides elegant, space-efficient solutions that generalize across many variants.

    This post covers six distinct pair-finding patterns in depth. For each one, we walk through the intuition, provide a formal correctness argument, analyze complexity, and build interactive step-through visualizations so you can watch the pointers move.

    Prerequisites: Familiarity with sorting algorithms and basic two pointer concepts. If you have not read the Two Pointers Overview, start there first.

    Problems Covered

    2. Two Sum Less Than K

    1. Two Sum on Sorted Array

    Problem Statement

    Given a 1-indexed array of integers sorted in non-decreasing order, find two numbers that add up to a specific target. Return their indices. There is exactly one solution, and you may not use the same element twice.

    Example: numbers = [2, 7, 11, 15], target = 9. Answer: indices [1, 2] because 2 + 7 = 9.

    Brute Force

    Check every pair $(i, j)$ where $i < j$. If numbers[i] + numbers[j] == target, return the pair. This runs in $O(n^2)$ time and completely ignores the fact that the array is sorted.

    // Brute force: O(n^2)
vector<int> twoSum(vector<int>& nums, int target) {
    int n = nums.size();
    for (int i = 0; i < n; i++) {
        for (int j = i + 1; j < n; j++) {
            if (nums[i] + nums[j] == target) {
                return {i + 1, j + 1};
            }
        }
    }
    return {};
}

    Two Pointer Approach

    Place one pointer left at the start and one pointer right at the end. Compute the sum:

    • If sum == target, we found the answer.
    • If sum < target, we need a larger sum. Move left forward (the smallest remaining value increases).
    • If sum > target, we need a smaller sum. Move right backward (the largest remaining value decreases).

    Initialize Pointers

    Set left = 0 and right = n - 1. The search space is the entire array.

    Compute Sum

    Calculate sum = nums[left] + nums[right].

    Compare and Move

    If sum equals target, return. If sum is too small, increment left. If sum is too large, decrement right.

    Repeat

    Continue until the pointers meet. Since the array is sorted and a solution exists, the pointers will always converge to it.

    // Two Pointer: O(n) time, O(1) space
vector<int> twoSum(vector<int>& nums, int target) {
    int left = 0, right = nums.size() - 1;
    while (left < right) {
        int sum = nums[left] + nums[right];
        if (sum == target) {
            return {left + 1, right + 1};
        } else if (sum < target) {
            left++;
        } else {
            right--;
        }
    }
    return {};  // no solution found
}

    Why It Works: Proof of Correctness

    We can prove correctness using a search space reduction argument. Consider all pairs $(i, j)$ where $0 \le i < j \le n-1$. At each step, we eliminate either an entire row or an entire column of this matrix without discarding the answer.

    When nums[left] + nums[right] < target: pairing nums[left] with any value at index $\le$ right will also be too small (since those values are smaller). So we can safely discard left and all pairs containing it. Move left forward.

    When nums[left] + nums[right] > target: pairing nums[right] with any value at index $\ge$ left will also be too large. Discard right. Move right backward.

    Since each step eliminates at least one index from consideration, and we never eliminate the answer pair, the algorithm converges in at most $n - 1$ steps.

    Invariant: At every point in the loop, the answer pair $(i^*, j^*)$ satisfies left $\le i^*$ and $j^* \le$ right. This invariant holds because we only advance left past indices that cannot be part of the solution, and only retreat right past indices that cannot be part of the solution.

    Array [1, 3, 5, 7, 9, 11], target = 10. Watch the left (blue) and right (red) pointers converge to the solution pair (1, 9).

    Complexity

    ApproachTimeSpace
    Brute Force$O(n^2)$$O(1)$
    Hash Map (unsorted)$O(n)$$O(n)$
    Two Pointers (sorted)$O(n)$$O(1)$

    Edge Cases

    • Array of size 2: Only one pair exists. The algorithm checks it immediately.
    • Negative numbers: Works without modification since the array is still sorted.
    • Duplicate values: The algorithm finds the correct pair even when values repeat, because it uses indices, not values.
    • Large values: Use long long for the sum to avoid integer overflow when values can be up to $10^9$.
    Common Mistake: Forgetting that the problem uses 1-based indexing. The two pointer logic uses 0-based indices internally; convert when returning the answer.

    2. Two Sum Less Than K

    Problem Statement

    Given an array of integers and an integer $K$, find the pair $(a, b)$ with the maximum sum such that $a + b < K$. If no such pair exists, return $-1$.

    Example: arr = [34, 23, 1, 24, 75, 33, 54, 8], K = 60. After sorting: [1, 8, 23, 24, 33, 34, 54, 75]. The best pair is 24 + 33 = 57 < 60.

    Approach

    Sort the array first. Then use two pointers from opposite ends:

    • If nums[left] + nums[right] < K, this is a valid pair. Record it if it improves the best answer. Move left forward to try a larger sum.
    • If nums[left] + nums[right] >= K, the sum is too large. Move right backward.
    int twoSumLessThanK(vector<int>& arr, int K) {
    sort(arr.begin(), arr.end());
    int left = 0, right = arr.size() - 1;
    int best = -1;

    while (left < right) {
        int sum = arr[left] + arr[right];
        if (sum < K) {
            best = max(best, sum);
            left++;
        } else {
            right--;
        }
    }
    return best;
}

    Why Moving Left Forward Works

    When we find a valid sum (sum < K), we want to see if we can get even closer to $K$ without exceeding it. The only way to increase the sum is to increase the smaller element. Moving left forward replaces the smaller value with a (potentially) larger one. If we moved right backward instead, the sum would decrease, which cannot improve our answer since we already recorded this sum.

    When the sum is $\ge K$, moving right backward is the only way to reduce it. We certainly cannot move left forward, as that would make the sum even larger.

    Key Difference from Classic Two Sum: Here we do not stop at the first valid pair. We must explore all candidates to find the maximum valid sum. But each pointer still moves in one direction, so the total work is $O(n)$ after sorting.

    Complexity

    ApproachTimeSpace
    Brute Force$O(n^2)$$O(1)$
    Two Pointers$O(n \log n)$$O(1)$

    The $O(n \log n)$ comes from sorting. The two pointer scan itself is $O(n)$.

    Edge Cases

    • All pairs exceed K: Return $-1$. The algorithm naturally handles this since best stays at $-1$.
    • Multiple pairs with same max sum: We only need the sum value, not the indices, so duplicates do not matter.
    • Array of size less than 2: No pair exists; return $-1$ immediately.

    3. Count Pairs with Given Sum

    Problem Statement

    Given a sorted array and a target sum, count the total number of pairs $(i, j)$ with $i < j$ such that arr[i] + arr[j] == target.

    Example: arr = [1, 2, 3, 4, 5, 5, 6], target = 7. Valid pairs: (1,6), (2,5), (2,5), (3,4). Count = 4.

    Approach

    Start with two pointers at opposite ends. When a match is found, there are two sub-cases:

    Left and Right Values Are Different

    Count how many copies of arr[left] exist at the left boundary and how many copies of arr[right] exist at the right boundary. The number of pairs from this combination is countLeft * countRight. Skip both groups.

    Left and Right Values Are Equal

    All elements between left and right (inclusive) have the same value. The number of pairs is "choose 2" from that count: $\binom{m}{2} = \frac{m(m-1)}{2}$ where $m$ = number of equal elements.

    int countPairs(vector<int>& arr, int target) {
    int left = 0, right = arr.size() - 1;
    int count = 0;

    while (left < right) {
        int sum = arr[left] + arr[right];
        if (sum < target) {
            left++;
        } else if (sum > target) {
            right--;
        } else {
            // sum == target
            if (arr[left] == arr[right]) {
                // All values in [left..right] are equal
                int m = right - left + 1;
                count += m * (m - 1) / 2;
                break;
            }
            // Count duplicates on left side
            int leftVal = arr[left];
            int cntL = 0;
            while (left < right && arr[left] == leftVal) {
                cntL++;
                left++;
            }
            // Count duplicates on right side
            int rightVal = arr[right];
            int cntR = 0;
            while (left <= right && arr[right] == rightVal) {
                cntR++;
                right--;
            }
            count += cntL * cntR;
        }
    }
    return count;
}

    Why the Multiplication Works

    If there are 3 copies of value $a$ on the left and 2 copies of value $b$ on the right, and $a + b$ = target, then each of the 3 left copies can pair with each of the 2 right copies. By the multiplication principle, that gives $3 \times 2 = 6$ distinct pairs.

    When arr[left] == arr[right], every element in the range has the same value. We need to count the number of ways to pick 2 elements from $m$ elements, which is $\binom{m}{2}$.

    Complexity

    ApproachTimeSpace
    Brute Force$O(n^2)$$O(1)$
    Two Pointers$O(n)$$O(1)$
    Hash Map$O(n)$$O(n)$
    Common Mistake: Treating each match as a single pair and moving both pointers by one. This misses pairs formed by duplicate elements. Always count the duplicates on both sides before advancing.

    Edge Cases

    • All elements identical: Falls into Case B. If the array is [5,5,5,5] and target is 10, the count is $\binom{4}{2} = 6$.
    • No valid pairs: Returns 0. The loop ends when pointers cross without finding a match.
    • Large duplicate groups: The inner while loops skip entire groups, maintaining linear time overall.

    4. Two Sum with Duplicates

    Problem Statement

    Given an array that may contain duplicates, find all unique pairs that sum to a target. Return distinct value pairs only (no duplicate pairs in the output).

    Example: arr = [1, 2, 2, 3, 4, 4, 5], target = 6. Unique pairs: (1,5), (2,4). The pair (2,4) appears multiple times in the raw data but is listed only once.

    Approach

    Sort the array and use two pointers. The key addition: after finding a match, skip all duplicates of both arr[left] and arr[right] before continuing.

    vector<pair<int,int>> uniquePairSum(vector<int>& arr, int target) {
    sort(arr.begin(), arr.end());
    vector<pair<int,int>> result;
    int left = 0, right = arr.size() - 1;

    while (left < right) {
        int sum = arr[left] + arr[right];
        if (sum == target) {
            result.push_back({arr[left], arr[right]});
            // Skip duplicates on the left
            int lv = arr[left];
            while (left < right && arr[left] == lv) left++;
            // Skip duplicates on the right
            int rv = arr[right];
            while (left < right && arr[right] == rv) right--;
        } else if (sum < target) {
            left++;
        } else {
            right--;
        }
    }
    return result;
}

    The Deduplication Mechanism

    After recording a pair $(a, b)$, we advance left past all copies of $a$ and retreat right past all copies of $b$. This guarantees the next match (if any) involves a different value on at least one side. Since the array is sorted, all copies of a value are contiguous, so skipping duplicates is a simple while loop.

    This is more efficient than using a set to track seen pairs, as it requires $O(1)$ extra space (beyond the result vector).

    Why Not Just Use a Set? A hash set works, but it adds $O(n)$ space. The two pointer approach with in-place duplicate skipping achieves the same result in $O(1)$ auxiliary space.

    Complexity

    • Time: $O(n \log n)$ for sorting, $O(n)$ for the two pointer pass. Total: $O(n \log n)$.
    • Space: $O(1)$ auxiliary (not counting the output).

    5. Pair with Given Difference

    Problem Statement

    Given a sorted array of distinct integers and a non-negative integer $d$, find if there exists a pair $(a, b)$ with $a > b$ such that $a - b = d$.

    Example: arr = [1, 3, 5, 8, 12], d = 7. The pair is $(8, 1)$ since $8 - 1 = 7$.

    Key Insight: Same-Direction Pointers

    Unlike the sum problems, the difference problem uses same-direction two pointers. Both pointers start at the left and move rightward. This is because subtracting two values and trying to hit a target has a different monotonic structure.

    Let i be the slow pointer and j be the fast pointer, with $j > i$:

    • If arr[j] - arr[i] == d, we found the pair.
    • If arr[j] - arr[i] < d, the difference is too small. Increase it by moving j forward (larger value).
    • If arr[j] - arr[i] > d, the difference is too large. Decrease it by moving i forward (larger subtrahend).
    bool findPairWithDiff(vector<int>& arr, int d) {
    int n = arr.size();
    int i = 0, j = 1;

    while (i < n && j < n) {
        if (i == j) {
            j++;  // ensure different elements
            continue;
        }
        int diff = arr[j] - arr[i];
        if (diff == d) {
            return true;
        } else if (diff < d) {
            j++;
        } else {
            i++;
        }
    }
    return false;
}

    Why Same-Direction Works

    Consider what happens when we fix i and increase j. Since the array is sorted, arr[j] increases, so the difference arr[j] - arr[i] increases. Once it exceeds $d$, no further value of j (with this i) will give exactly $d$. So we advance i to reduce the difference.

    Conversely, if the difference is too small, moving i forward would only make it smaller (since arr[i] increases while arr[j] stays). So we advance j.

    Important: Both pointers always move forward. This is not the opposite-ends pattern. The time complexity is $O(n)$ because each pointer advances at most $n$ times.

    Opposite-Ends vs Same-Direction

    AspectSum ProblemsDifference Problems
    Pointer initOpposite endsBoth at left
    MovementConverge toward centerMove right together
    When value too smallMove left forwardMove fast pointer forward
    When value too largeMove right backwardMove slow pointer forward
    Time$O(n)$$O(n)$

    Edge Cases

    • d = 0 with distinct elements: No pair exists (need two equal elements). The condition i != j prevents using the same element.
    • d = 0 with duplicates: Works correctly. The pair will be two identical values at different indices.
    • Negative values in array: The algorithm works unchanged since the array is sorted regardless of sign.
    Common Mistake: Using opposite-ends pointers for the difference problem. This does not work because moving left forward and right backward both reduce the difference, making it impossible to control the search direction.

    6. K-Closest Pairs (Smallest Sum Pairs)

    Problem Statement

    Given two sorted arrays A and B, find the $K$ pairs $(a, b)$ with the smallest sum, where $a$ comes from A and $b$ comes from B.

    Example: A = [1, 3, 5], B = [2, 4, 6], K = 3. The three smallest-sum pairs are: $(1,2)=3$, $(1,4)=5$, $(3,2)=5$.

    Min-Heap with Two Pointer Idea

    This problem cannot be solved with a pure two pointer scan because we are pairing elements across two arrays. However, the two pointer intuition informs the heap-based approach:

    • Start with the smallest possible pair: (A[0], B[0]).
    • For each pair (A[i], B[j]) extracted from the heap, the next candidates are (A[i+1], B[j]) and (A[i], B[j+1]).
    • Use a min-heap ordered by sum. Extract K times.

    To avoid duplicates in the heap, a common optimization: for each A[i], only pair it with B[0] initially. When (A[i], B[j]) is popped, push (A[i], B[j+1]).

    vector<pair<int,int>> kSmallestPairs(vector<int>& A,
                                       vector<int>& B,
                                       int K) {
    vector<pair<int,int>> result;
    if (A.empty() || B.empty() || K == 0) return result;

    // Min-heap: (sum, index_in_A, index_in_B)
    auto cmp = [](tuple<int,int,int>& a,
                  tuple<int,int,int>& b) {
        return get<0>(a) > get<0>(b);
    };
    priority_queue<tuple<int,int,int>,
                   vector<tuple<int,int,int>>,
                   decltype(cmp)> pq(cmp);

    // Seed: pair each A[i] with B[0]
    for (int i = 0; i < min((int)A.size(), K); i++) {
        pq.push({A[i] + B[0], i, 0});
    }

    while (K > 0 && !pq.empty()) {
        auto [sum, ai, bi] = pq.top();
        pq.pop();
        result.push_back({A[ai], B[bi]});
        K--;
        if (bi + 1 < (int)B.size()) {
            pq.push({A[ai] + B[bi + 1], ai, bi + 1});
        }
    }
    return result;
}

    Why This Works

    The key insight is that pairs are generated in non-decreasing order of sum. If (A[i], B[j]) has sum $s$, then (A[i], B[j+1]) has sum $\ge s$ (since B is sorted). By seeding the heap with all (A[i], B[0]) pairs and expanding only in the B direction, we guarantee completeness without revisiting pairs.

    The heap acts as a "frontier" of the smallest unexplored pairs. At each step, we pop the global minimum and expand it.

    Complexity

    ApproachTimeSpace
    Brute Force (all pairs, sort)$O(mn \log(mn))$$O(mn)$
    Min-Heap$O(K \log(\min(m, K)))$$O(\min(m, K))$

    Where $m = |A|$ and $n = |B|$. The heap never holds more than $\min(m, K)$ elements.

    Edge Cases

    • K larger than m * n: Return all possible pairs. The heap will naturally drain.
    • One array empty: No pairs exist. Return empty.
    • Duplicate sums: Multiple pairs may have the same sum. The heap handles them correctly in order.
    Common Mistake: Seeding the heap with all $m \times n$ pairs. This defeats the purpose and uses $O(mn)$ space. Only seed with the first row (A[i], B[0]).

    Pattern Comparison

    Problem Pointer Style Time Space Key Idea
    Two Sum Sorted Opposite ends $O(n)$ $O(1)$ Sum too small: move left. Sum too big: move right.
    Two Sum < K Opposite ends $O(n \log n)$ $O(1)$ Track best valid sum while converging.
    Count Pairs Opposite ends $O(n)$ $O(1)$ Count duplicate groups with multiplication.
    Unique Pairs Opposite ends $O(n \log n)$ $O(1)$ Skip all duplicates after each match.
    Pair Difference Same direction $O(n)$ $O(1)$ Both pointers move right. Different monotonic structure.
    K-Closest Pairs Heap + expansion $O(K \log K)$ $O(K)$ Expand frontier in sorted order using min-heap.

    Interview Tips

    Clarify: Is the Array Sorted?

    If the array is sorted, two pointers give $O(n)$ time with $O(1)$ space. If not, you have two options: sort it first (adding $O(n \log n)$), or use a hash map ($O(n)$ time but $O(n)$ space). Ask the interviewer which trade-off they prefer.

    Watch for Duplicate Handling

    Many candidates lose points by not handling duplicates. Always ask: "Are there duplicate values? Do we want unique pairs or all pairs?" This changes the algorithm significantly (skip vs count).

    Sum vs Difference: Different Pointer Patterns

    Sum problems use opposite-ends pointers. Difference problems use same-direction pointers. Mixing these up is a frequent error. Pause and think about the monotonic property before coding.

    Overflow Protection

    When values can be large (up to $10^9$ or $2^{31}-1$), the sum of two values can overflow a 32-bit integer. Cast to long long before adding: (long long)a + b.

    State Your Invariant

    In an interview, explicitly state the loop invariant: "At every iteration, the answer lies within [left, right]." This demonstrates understanding and helps the interviewer follow your proof of correctness.

    Common Mistakes

    • Off-by-one on pointer initialization: Starting right at n instead of n - 1 causes an out-of-bounds access. Always double check.
    • Infinite loops with duplicates: When arr[left] == arr[right] and you forget to break or handle the equal-values case, the pointers may never cross.
    • Not sorting first: Two pointers on an unsorted array produces incorrect results. The monotonic property depends entirely on sorted order.
    • Using the same element twice: If the problem says "distinct indices," ensure left < right (not left <= right).
    • Returning values instead of indices: Some problems ask for indices, others for values. Read carefully.

    Key Takeaways

    • The two pointer technique on a sorted array solves pair-finding problems in $O(n)$ time and $O(1)$ space.
    • Sum problems use opposite-ends pointers. The decision at each step reduces the search space by one row or column.
    • Difference problems use same-direction pointers. Both pointers move forward, maintaining a different monotonic property.
    • Counting pairs requires careful handling of duplicates: multiply the counts of matching groups, or use $\binom{m}{2}$ when all values are equal.
    • For unique pairs, skip over all duplicates after recording each match.
    • K-closest pairs across two arrays use a min-heap with a frontier expansion strategy.
    • Always clarify: sorted or unsorted? Return indices or values? Allow duplicates or not?
