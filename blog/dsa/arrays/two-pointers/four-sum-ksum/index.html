    Four Sum & K-Sum Generalization

    March 2026 · 20 min read

    Three Sum asked us to find triplets that sum to a target. Four Sum extends the same idea to quadruplets. And once you see the pattern, you can generalize it to K-Sum for any value of K. This post covers the complete picture: from the concrete 4Sum problem to the recursive K-Sum framework, along with hash map variants and interview strategies.

    1. Four Sum (LeetCode 18)

    Given an array nums and a target value, find all unique quadruplets [nums[a], nums[b], nums[c], nums[d]] such that:

    • a, b, c, d are distinct indices.
    • nums[a] + nums[b] + nums[c] + nums[d] == target.
    • The solution set must not contain duplicate quadruplets.

    Approach: Two Nested Loops + Two Pointers

    The strategy is a direct extension of 3Sum. Sort the array first. Then use two nested loops to fix the first two elements (i and j). For each pair, use two pointers (left and right) on the remaining subarray to find the other two elements.

    Sort the Array

    Sorting enables two things: the two-pointer technique requires sorted order, and duplicate skipping becomes a simple comparison with the previous element.

    First Loop (i = 0 to n-4)

    The outer loop fixes the first element. Skip duplicates: if nums[i] == nums[i-1] and i > 0, continue to the next iteration. This prevents duplicate quadruplets where the first element is the same.

    Second Loop (j = i+1 to n-3)

    The second loop fixes the second element. Again skip duplicates: if nums[j] == nums[j-1] and j > i+1, continue. The condition j > i+1 is critical. Without it, you would skip valid pairs.

    Two Pointers (left = j+1, right = n-1)

    Compute sum = nums[i] + nums[j] + nums[left] + nums[right]. If sum equals target, record the quadruplet and advance both pointers while skipping duplicates. If sum is less than target, increment left. If sum exceeds target, decrement right.

    Implementation

    vector<vector<int>> fourSum(vector<int>& nums, int target) {
    vector<vector<int>> result;
    int n = nums.size();
    if (n < 4) return result;
    sort(nums.begin(), nums.end());

    for (int i = 0; i < n - 3; i++) {
        // Skip duplicates for i
        if (i > 0 && nums[i] == nums[i - 1]) continue;

        // Pruning: smallest possible sum with current i
        // Use long long to avoid overflow
        long long minSum = (long long)nums[i] + nums[i+1]
                         + nums[i+2] + nums[i+3];
        if (minSum > target) break;

        // Pruning: largest possible sum with current i
        long long maxSum = (long long)nums[i] + nums[n-3]
                         + nums[n-2] + nums[n-1];
        if (maxSum < target) continue;

        for (int j = i + 1; j < n - 2; j++) {
            // Skip duplicates for j
            if (j > i + 1 && nums[j] == nums[j - 1]) continue;

            // Pruning for j
            long long minJ = (long long)nums[i] + nums[j]
                           + nums[j+1] + nums[j+2];
            if (minJ > target) break;

            long long maxJ = (long long)nums[i] + nums[j]
                           + nums[n-2] + nums[n-1];
            if (maxJ < target) continue;

            int left = j + 1, right = n - 1;
            while (left < right) {
                long long sum = (long long)nums[i] + nums[j]
                              + nums[left] + nums[right];
                if (sum == target) {
                    result.push_back({nums[i], nums[j],
                                      nums[left], nums[right]});
                    // Skip duplicates for left
                    while (left < right
                           && nums[left] == nums[left + 1]) left++;
                    // Skip duplicates for right
                    while (left < right
                           && nums[right] == nums[right - 1]) right--;
                    left++;
                    right--;
                } else if (sum < target) {
                    left++;
                } else {
                    right--;
                }
            }
        }
    }
    return result;
}

    Overflow Warning: When the target or array values are large, summing four integers can overflow a 32-bit int. Always cast to long long before adding. This is a common interview pitfall that catches many candidates.

    Duplicate Skipping in Detail

    Duplicate skipping happens at every level. This is the part most candidates get wrong. Here is exactly where each skip occurs:

    PointerConditionReason
    ii > 0 && nums[i] == nums[i-1]Prevents quadruplets with the same first element
    jj > i+1 && nums[j] == nums[j-1]Prevents quadruplets with the same second element (given i)
    leftAfter a match: skip while nums[left] == nums[left+1]Prevents recording the same third element
    rightAfter a match: skip while nums[right] == nums[right-1]Prevents recording the same fourth element

    Why j > i+1? Consider array [-1, -1, 0, 2] with target 0. When i=0 and j=1, both point to -1. That is a valid pairing. The condition j > i+1 ensures we only skip j when it repeats as the second element for the same i, not when it is the very first j value after i.

    Pruning Optimizations

    Pruning can significantly speed up 4Sum in practice, even though the worst-case complexity remains the same.

    Early Termination (Min Sum)

    For a fixed i, the smallest possible sum uses the four smallest available elements: nums[i] + nums[i+1] + nums[i+2] + nums[i+3]. If this exceeds target, no valid quadruplet exists for any larger i, so we can break out of the outer loop entirely.

    Skip Hopeless i (Max Sum)

    The largest possible sum with current i is nums[i] + nums[n-3] + nums[n-2] + nums[n-1]. If this is less than target, the current i is too small. Skip to the next i with continue.

    Same Pruning for j

    Apply identical min-sum and max-sum pruning inside the inner loop for j. This avoids scanning the two-pointer range when no solution is possible with the current (i, j) pair.

    Complexity Analysis

    AspectValue
    TimeO(n3). Two nested loops O(n2) times the two-pointer scan O(n).
    SpaceO(1) extra (excluding the output). Sorting is in-place for most implementations.
    Best case with pruningMuch faster in practice. Pruning skips entire ranges of i and j.

    2. 4Sum II (LeetCode 454)

    This problem is fundamentally different from 4Sum. You are given four separate arrays A, B, C, D and must count the number of tuples (i, j, k, l) such that A[i] + B[j] + C[k] + D[l] == 0.

    Because the elements come from separate arrays, sorting one array does not help with the others. The two-pointer approach does not apply here.

    Hash Map Approach

    Split the four arrays into two groups: (A, B) and (C, D). Compute all pairwise sums from A and B, storing each sum and its frequency in a hash map. Then iterate over all pairwise sums from C and D, looking up the negation in the map.

    int fourSumCount(vector<int>& A, vector<int>& B,
                 vector<int>& C, vector<int>& D) {
    unordered_map<int, int> sumAB;
    for (int a : A)
        for (int b : B)
            sumAB[a + b]++;

    int count = 0;
    for (int c : C)
        for (int d : D) {
            auto it = sumAB.find(-(c + d));
            if (it != sumAB.end())
                count += it->second;
        }
    return count;
}

    Meet in the Middle

    This is the "meet in the middle" technique. By splitting the problem into two halves, we reduce O(n4) brute force to O(n2) time using O(n2) space for the hash map.

    3. Generalized K-Sum

    The 4Sum solution suggests a pattern: fix one element, reduce to 3Sum. 3Sum fixes one element, reduces to 2Sum. 2Sum uses two pointers. This recursive structure works for any K.

    Recursive Reduction

    To solve K-Sum:

    1. Sort the array (once, at the top level).
    2. If K equals 2, solve with two pointers.
    3. Otherwise, iterate through the array: fix one element, recursively solve (K-1)-Sum on the remaining subarray with an adjusted target.
    4. At each level, skip duplicates the same way.

    The Invariant: At every recursive level, the subarray from start to n-1 is already sorted. Each level fixes one element and calls the next level with start+1. The base case (K=2) runs two pointers from start to n-1.

    K-Sum Implementation

    vector<vector<int>> kSum(vector<int>& nums, long long target,
                         int start, int k) {
    vector<vector<int>> result;
    int n = nums.size();

    // Edge cases
    if (start >= n) return result;

    // Average value check for early termination
    long long avgVal = target / k;
    if (nums[start] > avgVal || nums.back() < avgVal)
        return result;

    // Base case: 2Sum with two pointers
    if (k == 2) {
        int lo = start, hi = n - 1;
        while (lo < hi) {
            long long sum = (long long)nums[lo] + nums[hi];
            if (sum == target) {
                result.push_back({nums[lo], nums[hi]});
                while (lo < hi && nums[lo] == nums[lo+1]) lo++;
                while (lo < hi && nums[hi] == nums[hi-1]) hi--;
                lo++; hi--;
            } else if (sum < target) {
                lo++;
            } else {
                hi--;
            }
        }
        return result;
    }

    // Recursive case: fix one element, solve (k-1)-Sum
    for (int i = start; i < n - k + 1; i++) {
        // Skip duplicates
        if (i > start && nums[i] == nums[i - 1]) continue;

        // Pruning
        long long minS = (long long)nums[i];
        for (int p = 1; p < k; p++) minS += nums[i + p];
        if (minS > target) break;

        long long maxS = (long long)nums[i];
        for (int p = 1; p < k; p++) maxS += nums[n - p];
        if (maxS < target) continue;

        auto sub = kSum(nums, target - nums[i], i + 1, k - 1);
        for (auto& v : sub) {
            v.insert(v.begin(), nums[i]);
            result.push_back(move(v));
        }
    }
    return result;
}

// Entry point
vector<vector<int>> fourSum(vector<int>& nums, int target) {
    sort(nums.begin(), nums.end());
    return kSum(nums, (long long)target, 0, 4);
}

    4. Complexity Analysis of K-Sum

    Understanding the time complexity of K-Sum requires thinking about the recursive structure.

    Time Complexity

    At each recursive level, we iterate through the array once and call the next level. The recursion depth is K-2 (since the base case at K=2 uses a linear two-pointer scan).

    • 2Sum: O(n) using two pointers on a sorted array.
    • 3Sum: One loop over n elements, each calling 2Sum. Total: O(n2).
    • 4Sum: One loop calling 3Sum. Total: O(n3).
    • K-Sum: Follows the pattern. Total: O(nK-1).

    The O(n log n) sorting cost is dominated by O(nK-1) for K ≥ 3. For K = 2, sorting is the bottleneck unless the array is pre-sorted.

    T(K, n) = n * T(K-1, n), T(2, n) = O(n)

    Unrolling: T(K, n) = n * n * ... * n * O(n) = O(nK-1). There are K-2 levels of recursion, each contributing a factor of n, and the base case contributes O(n).

    Space Complexity

    The recursion depth is O(K). Each level stores O(1) variables. The call stack uses O(K) space. The output can have up to O(nK-1) tuples in the worst case, but we typically exclude output from space analysis.

    KTimeSpace (excl. output)Common Name
    2O(n)O(1)Two Sum (sorted)
    3O(n2)O(1)Three Sum
    4O(n3)O(1)Four Sum
    KO(nK-1)O(K)K-Sum

    Can we do better than O(nK-1)? For general K-Sum, it is conjectured that O(nceil(K/2)) is achievable for even K using meet-in-the-middle (the 4Sum II approach generalizes). However, this requires O(nK/2) space and does not easily produce the actual tuples. For odd K, the lower bound is less clear. In interviews, the O(nK-1) approach with two pointers at the base is the expected solution.

    5. Count Quadruplets (LeetCode 1995)

    Given an array nums, count the number of tuples (a, b, c, d) where a < b < c < d and nums[a] + nums[b] + nums[c] == nums[d].

    At first glance, this seems like a 4Sum variant. But notice that the constraint requires index ordering and the target is the fourth element itself, not a fixed value.

    Brute Force: O(n4)

    int countQuadruplets(vector<int>& nums) {
    int n = nums.size(), count = 0;
    for (int a = 0; a < n; a++)
        for (int b = a + 1; b < n; b++)
            for (int c = b + 1; c < n; c++)
                for (int d = c + 1; d < n; d++)
                    if (nums[a]+nums[b]+nums[c] == nums[d])
                        count++;
    return count;
}

    Optimized: Reverse Scan with Hash Map

    Rewrite the condition as nums[d] - nums[c] == nums[a] + nums[b]. Iterate c from right to left. For each c, first add nums[c+1] through nums[n-1] minus nums[c] differences to a map (these represent valid d candidates). Then check all pairs (a, b) with a < b < c against the map.

    int countQuadruplets(vector<int>& nums) {
    int n = nums.size(), count = 0;
    unordered_map<int, int> diff;
    // Start c from n-2 (need at least one d after c)
    for (int c = n - 2; c >= 2; c--) {
        // Add differences nums[d] - nums[c] for d = c+1..n-1
        // We incrementally add the new d = c+1 as c decreases
        for (int d = c + 1; d < n; d++)
            diff[nums[d] - nums[c]]++;
        // Actually, we should rebuild or update carefully.
        // Simpler correct approach: rebuild each time, or use
        // the incremental version below.
    }

    // Cleaner O(n^2) approach using running map:
    count = 0;
    diff.clear();
    for (int c = n - 2; c >= 1; c--) {
        // d values are indices > c. Add nums[c+1] to the map.
        // Key insight: we process c from right to left.
        // For d = c+1, the "target" for nums[a]+nums[b] is nums[d]-nums[c].
        // But c changes, so store raw nums[d] values and query differently.
    }
    // The cleanest version:
    return countQuadSimple(nums);
}

// Clean O(n^3) with hash map
int countQuadSimple(vector<int>& nums) {
    int n = nums.size(), count = 0;
    for (int c = 2; c < n; c++) {
        unordered_map<int, int> pairSum;
        for (int a = 0; a < c; a++)
            for (int b = a + 1; b < c; b++)
                pairSum[nums[a] + nums[b]]++;
        for (int d = c + 1; d < n; d++) {
            int need = nums[d] - nums[c];
            if (pairSum.count(need))
                count += pairSum[need];
        }
    }
    return count;
}

    A cleaner O(n2) approach exists using running sums:

    int countQuadruplets(vector<int>& nums) {
    int n = nums.size(), count = 0;
    // diff[val] = number of times val appears as nums[d] - nums[c]
    // where d > c. We process b from right to left.
    unordered_map<int,int> freq;
    for (int b = n - 3; b >= 1; b--) {
        // Add pairs (b+1 as c, everything after as d)
        int c = b + 1;
        for (int d = c + 1; d < n; d++)
            freq[nums[d] - nums[c]]++;
        // Now check all a < b: does nums[a] + nums[b] exist in freq?
        for (int a = 0; a < b; a++)
            count += freq[nums[a] + nums[b]];
    }
    return count;
}

    O(n2) Time, O(n2) Space

    The outer loop runs O(n) times. The inner loop adding to the map runs O(n) per iteration. The inner loop checking pairs runs O(n) per iteration. Total: O(n2). The map stores at most O(n2) entries across all iterations.

    Edge Cases and Pitfalls

    Integer Overflow

    The most common bug in 4Sum implementations is integer overflow. When summing four values that can each be up to 109, the sum can reach 4 * 109, which exceeds the range of 32-bit integers (max ~2.1 * 109).

    Always use long long for the sum computation. Cast at least one operand: (long long)nums[i] + nums[j] + nums[left] + nums[right]. The cast propagates to the entire expression.

    Empty and Short Arrays

    If the array has fewer than K elements, immediately return an empty result. Check n < 4 at the start for 4Sum, or n < k for K-Sum.

    All Zeros or All Same Elements

    Array: [0, 0, 0, 0, 0], target = 0. Without proper duplicate skipping, you could generate many duplicate [0, 0, 0, 0] tuples. The skip logic ensures only one copy appears.

    Negative Targets

    Be careful with pruning when the target is negative. The min-sum and max-sum pruning works correctly regardless of sign because the array is sorted and the comparisons are straightforward. But do not assume the target is positive when writing conditions.

    Target Outside Array Range

    The average-value check in the K-Sum code (target / k) catches cases where the target is unreachable. If the smallest element exceeds the average or the largest element falls below it, no solution exists.

    6. Interview Strategy

    When you encounter a sum-finding problem in an interview, follow this decision process:

    Identify the Structure

    How many elements are being summed? Is the target fixed or derived from the array? Are elements from the same array or separate arrays? Answers determine which variant you face.

    Choose the Framework

    Same array, find actual tuples: use the recursive K-Sum with sorted array and two pointers. Separate arrays, count tuples: use meet-in-the-middle with hash maps. Target derived from array elements: look for algebraic reformulation.

    Handle Duplicates

    Always clarify whether duplicates matter. If the problem says "unique" tuples, you need skip logic at every level. If counting, hash map frequencies handle it naturally.

    Mention Overflow

    Proactively mention overflow risks. Using long long for sums shows awareness of production code quality. This is a signal interviewers look for.

    Discuss Pruning

    After presenting the core solution, mention pruning as an optimization. The min-sum/max-sum checks are easy to add and can dramatically reduce runtime on certain inputs. This shows depth of understanding.
