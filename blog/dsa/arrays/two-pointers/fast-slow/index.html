    Fast & Slow Pointers

    March 2026 · 25 min read

    The fast and slow pointer technique (also called the tortoise and hare algorithm) uses two pointers that traverse a sequence at different speeds. The slow pointer advances one step at a time, while the fast pointer advances two steps. This speed difference creates powerful properties that let us detect cycles, find midpoints, and solve problems that seem unrelated to pointer manipulation.

    This post covers the technique in depth: the mathematical proof behind Floyd's cycle detection, six classic problems, and interactive SVG animations to build visual intuition.

    1. Floyd's Cycle Detection (LeetCode 141)

    Given a linked list, determine whether it contains a cycle. A cycle exists when some node's next pointer points back to an earlier node, creating an infinite loop.

    The Tortoise and Hare Idea

    Place two pointers at the head. The slow pointer moves one node per step, and the fast pointer moves two nodes per step. If there is no cycle, the fast pointer will reach the end (nullptr). If there is a cycle, both pointers enter the loop, and the fast pointer will eventually catch up to the slow pointer from behind.

    Why must they meet? Once both are inside the cycle of length $C$, the gap between fast and slow shrinks by exactly 1 each step. Since the gap starts as some value between 1 and $C - 1$, they must meet within $C$ steps.

    Think of a Circular Track

    Imagine two runners on a circular track. One runs at speed 1, the other at speed 2. The faster runner laps the slower one. On a track of length $C$, the relative speed is $2 - 1 = 1$, so the gap closes by 1 each unit of time. After $C$ steps at most, the gap becomes 0 and they overlap.

    Animation: Cycle Detection

    The linked list below has nodes 1 through 8, with node 8 pointing back to node 3 (forming a cycle of length 6). Step through to watch the slow (blue) and fast (red) pointers advance until they meet.

    Step 0: Both at node 1

    C++ Solution

    // LeetCode 141: Linked List Cycle
bool hasCycle(ListNode* head) {
    ListNode* slow = head;
    ListNode* fast = head;
    while (fast != nullptr && fast->next != nullptr) {
        slow = slow->next;
        fast = fast->next->next;
        if (slow == fast) return true;
    }
    return false;
}

    Time and Space

    Time: $O(n)$. If there is no cycle, fast reaches the end in $n/2$ steps. If there is a cycle, after slow enters the loop (at most $n$ steps total), they meet within $C \leq n$ more steps.

    Space: $O(1)$. Only two pointers.

    2. Find the Cycle Entry (LeetCode 142)

    Once we know a cycle exists, the next question is: which node is the start of the cycle? Floyd's algorithm has a beautiful second phase that answers this in $O(1)$ extra space.

    The Algorithm

    1. Phase 1: Run slow and fast until they meet inside the cycle.
    2. Phase 2: Reset one pointer to the head. Move both pointers one step at a time. The node where they meet again is the cycle entry.

    Why Does Phase 2 Work?

    Let us define the distances. Let $F$ be the number of nodes from the head to the cycle entry. Let $C$ be the cycle length. When slow and fast first meet:

    • Slow has traveled $F + a$ steps, where $a$ is the distance from the cycle entry to the meeting point (measured inside the cycle).
    • Fast has traveled $F + a + kC$ steps for some integer $k \geq 1$, since fast went around the cycle at least once.
    • Fast moves at twice the speed of slow, so $2(F + a) = F + a + kC$.

    Simplifying: $F + a = kC$, which gives us $F = kC - a$.

    This means: the distance from the head to the cycle entry ($F$) equals the distance from the meeting point to the cycle entry going forward ($C - a$), plus some whole number of full cycle loops ($(k - 1)C$). So if one pointer starts at the head and another starts at the meeting point, advancing both by one step at a time, they will meet exactly at the cycle entry.

    Distance Proof Summary

    $$2(F + a) = F + a + kC$$

    $$F + a = kC$$

    $$F = kC - a = (k - 1)C + (C - a)$$

    After $F$ steps from the head, the first pointer reaches the entry. After $F$ steps from the meeting point, the second pointer has traveled $(k-1)$ full cycles plus $(C - a)$ steps, which also lands on the entry.

    Animation: Finding the Cycle Entry

    After the meeting point is found (purple node), we reset one pointer to the head (green) and advance both by one. They meet at the cycle entry (yellow).

    Phase 2 start: p1 at head (1), p2 at meeting point (5)

    C++ Solution

    // LeetCode 142: Linked List Cycle II
ListNode* detectCycle(ListNode* head) {
    ListNode* slow = head;
    ListNode* fast = head;
    // Phase 1: find meeting point
    while (fast != nullptr && fast->next != nullptr) {
        slow = slow->next;
        fast = fast->next->next;
        if (slow == fast) {
            // Phase 2: find entry
            ListNode* entry = head;
            while (entry != slow) {
                entry = entry->next;
                slow = slow->next;
            }
            return entry;
        }
    }
    return nullptr;
}

    Time and Space

    Time: $O(n)$. Phase 1 takes $O(n)$ and Phase 2 takes at most $O(n)$ additional steps.

    Space: $O(1)$. No extra data structures.

    3. Happy Number (LeetCode 202)

    A happy number is defined by the process: starting with any positive integer, replace the number by the sum of the squares of its digits. Repeat until the number equals 1 (it is happy) or it loops endlessly in a cycle (it is not happy).

    For example, 19 is happy: $19 \to 82 \to 68 \to 100 \to 1$.

    But 2 is not happy: $2 \to 4 \to 16 \to 37 \to 58 \to 89 \to 145 \to 42 \to 20 \to 4 \to \ldots$ (cycle detected).

    The Key Insight: The sequence of digit-square sums forms an implicit linked list. Each number maps to exactly one next number (the sum of squares of its digits). If 1 is never reached, the sequence must eventually repeat a value (since the values are bounded). This is cycle detection in disguise.

    Why Are Values Bounded?

    For any number with $d$ digits, the maximum digit-square sum is $81d$ (since $9^2 = 81$). For a 3-digit number, the max is 243. For a 4-digit number, 324. Numbers above 243 always decrease. So every sequence eventually enters the range [1, 243] and stays there, guaranteeing either convergence to 1 or a cycle.

    Animation: Happy Number Sequence

    Follow the sequence starting at 19. Each node shows a number, and the arrow points to the sum of squares of its digits. The slow pointer (blue) advances one step, and the fast pointer (red) advances two steps.

    Step 0: slow=19, fast=19

    C++ Solution

    // LeetCode 202: Happy Number
int digitSquareSum(int n) {
    int sum = 0;
    while (n > 0) {
        int d = n % 10;
        sum += d * d;
        n /= 10;
    }
    return sum;
}

bool isHappy(int n) {
    int slow = n, fast = n;
    do {
        slow = digitSquareSum(slow);
        fast = digitSquareSum(digitSquareSum(fast));
    } while (slow != fast);
    return slow == 1;
}

    Why do-while?

    We use a do-while loop because slow and fast start at the same value. A regular while (slow != fast) would exit immediately. The do-while ensures at least one iteration before checking the condition.

    4. Find the Duplicate Number (LeetCode 287)

    Given an array nums of $n + 1$ integers where each integer is in the range $[1, n]$, there is exactly one repeated number. Find it without modifying the array and using $O(1)$ extra space.

    The Array as a Linked List

    Treat each array index as a node and nums[i] as the "next pointer." Starting from index 0, follow the chain: $0 \to nums[0] \to nums[nums[0]] \to \ldots$

    Since there are $n + 1$ values mapping into the range $[1, n]$, by the pigeonhole principle at least two indices point to the same value. This creates a cycle in the functional graph. The duplicate number is the entry point of that cycle.

    Why not start from any index? We must start from index 0 because 0 is not in the range $[1, n]$, so no node points to 0. This means index 0 is guaranteed to be outside the cycle, like the "tail" of a linked list. Starting from inside the cycle would skip the entry detection.

    Array: [1, 3, 4, 2, 2]

    Index chain from 0: $0 \to 1 \to 3 \to 2 \to 4 \to 2 \to 4 \to 2 \to \ldots$
    The cycle is $2 \to 4 \to 2$. The entry point is index 2, so the duplicate value is nums[2] = 4... wait, let us trace more carefully.

    $0 \to nums[0]=1 \to nums[1]=3 \to nums[3]=2 \to nums[2]=4 \to nums[4]=2 \to nums[2]=4 \to \ldots$

    Cycle: $2 \to 4 \to 2$. Entry node is 2. The duplicate is 2 (the value that two indices point to).

    Visualization: Array as Functional Graph

    C++ Solution

    // LeetCode 287: Find the Duplicate Number
int findDuplicate(vector<int>& nums) {
    int slow = nums[0], fast = nums[0];
    // Phase 1: find meeting point
    do {
        slow = nums[slow];
        fast = nums[nums[fast]];
    } while (slow != fast);
    // Phase 2: find cycle entry
    slow = nums[0];
    while (slow != fast) {
        slow = nums[slow];
        fast = nums[fast];
    }
    return slow;
}
    Constraints matter. This solution works only because the problem guarantees values in $[1, n]$ with exactly one duplicate. Without that guarantee, you might get out-of-bounds access or no cycle at all.

    5. Middle of the Linked List (LeetCode 876)

    Given a singly linked list, return the middle node. If there are two middle nodes (even length), return the second one.

    The Fast/Slow Trick

    When the fast pointer reaches the end, the slow pointer is at the middle. Fast moves at 2x speed, so when fast has traversed the full length, slow has traversed half.

    • Odd length: fast stops at the last node. Slow is exactly at the middle.
    • Even length: fast goes past the last node (becomes nullptr). Slow is at the second middle node.

    Visualization: Finding the Middle

    C++ Solution

    // LeetCode 876: Middle of the Linked List
ListNode* middleNode(ListNode* head) {
    ListNode* slow = head;
    ListNode* fast = head;
    while (fast != nullptr && fast->next != nullptr) {
        slow = slow->next;
        fast = fast->next->next;
    }
    return slow;
}

    Even vs Odd Length

    For a list [1, 2, 3, 4, 5, 6], the function returns node 4 (the second middle). This matches the LeetCode requirement. If you need the first middle, check fast->next != nullptr && fast->next->next != nullptr instead.

    6. Reorder List (LeetCode 143)

    Given a singly linked list $L_0 \to L_1 \to \ldots \to L_{n-1} \to L_n$, reorder it to $L_0 \to L_n \to L_1 \to L_{n-1} \to L_2 \to L_{n-2} \to \ldots$ in-place.

    The Three-Step Strategy

    This problem combines three fundamental linked list operations:

    1. Find the middle using slow/fast pointers.
    2. Reverse the second half of the list.
    3. Merge the two halves by alternating nodes.

    Visualization: Three Steps

    C++ Solution

    // LeetCode 143: Reorder List
void reorderList(ListNode* head) {
    if (!head || !head->next) return;

    // Step 1: Find middle
    ListNode* slow = head;
    ListNode* fast = head;
    while (fast->next && fast->next->next) {
        slow = slow->next;
        fast = fast->next->next;
    }

    // Step 2: Reverse second half
    ListNode* prev = nullptr;
    ListNode* curr = slow->next;
    slow->next = nullptr;     // cut the list
    while (curr) {
        ListNode* next = curr->next;
        curr->next = prev;
        prev = curr;
        curr = next;
    }

    // Step 3: Merge two halves
    ListNode* first = head;
    ListNode* second = prev;
    while (second) {
        ListNode* tmp1 = first->next;
        ListNode* tmp2 = second->next;
        first->next = second;
        second->next = tmp1;
        first = tmp1;
        second = tmp2;
    }
}

    Time and Space

    Time: $O(n)$. Each step (find middle, reverse, merge) is a single pass.

    Space: $O(1)$. Everything is done in-place with pointer manipulation.

    7. Mathematical Proof of Floyd's Algorithm

    Let us prove rigorously why Floyd's algorithm works. This section derives the result from first principles.

    Setup and Notation

    • $H$ = head of the list
    • $E$ = cycle entry node
    • $M$ = meeting point of slow and fast
    • $F$ = distance from $H$ to $E$ (number of edges in the tail)
    • $a$ = distance from $E$ to $M$ (measured forward along the cycle)
    • $C$ = total length of the cycle

    Phase 1: Proving They Meet

    Theorem: If a cycle exists, slow and fast must meet.

    Once both pointers are inside the cycle, consider the gap $g$ between fast and slow, measured as the number of steps fast needs to reach slow going forward around the cycle.

    Each step: slow moves +1, fast moves +2. The gap changes by $+2 - 1 = +1$ in the forward direction, which means the gap decreases by 1 when measured from fast to slow in the forward direction (equivalently, the "catching up" distance decreases by 1 each step).

    Since the gap is a positive integer less than $C$, it reaches 0 after at most $C - 1$ steps. When the gap is 0, the pointers are on the same node.

    Phase 1: What Are the Distances?

    Relating slow and fast distances

    When they meet at $M$:

    $$d_{\text{slow}} = F + a$$

    (Slow traveled $F$ steps to enter the cycle, then $a$ steps to reach $M$.)

    $$d_{\text{fast}} = F + a + kC \quad \text{for some integer } k \geq 1$$

    (Fast also reached $M$ but completed $k$ extra full loops around the cycle.)

    Since fast moves at twice the speed: $d_{\text{fast}} = 2 \cdot d_{\text{slow}}$.

    $$2(F + a) = F + a + kC$$

    $$F + a = kC$$

    $$\boxed{F = kC - a}$$

    Phase 2: Why the Entry is Found

    Theorem: Starting one pointer at H and another at M, both moving at speed 1, they meet at E.

    From the equation $F = kC - a$, rewrite as:

    $$F = (k - 1)C + (C - a)$$

    Pointer from $H$: after $F$ steps, it arrives at $E$ by definition.

    Pointer from $M$: after $F$ steps, it has traveled $(k - 1)$ full cycles (returning to $M$ each time) plus $(C - a)$ additional steps. Since $M$ is $a$ steps past $E$, going $(C - a)$ more steps from $M$ lands exactly at $E$.

    Both pointers are at $E$ after exactly $F$ steps. They meet at the cycle entry.

    The Beautiful Result

    The distance from the head to the cycle entry equals the distance from the meeting point to the cycle entry (going forward), modulo the cycle length. This is not a coincidence. It is a direct consequence of the 2:1 speed ratio.

    What If k > 1?

    When the tail is long relative to the cycle, fast may loop around multiple times before slow even enters the cycle. The proof still holds because the extra $(k - 1)$ full loops add multiples of $C$ to the distance, which do not change the landing position within the cycle.

    What If F = 0?

    If the cycle starts at the head (no tail), then $F = 0$, which gives $a = kC$. This means the meeting point is at the cycle entry itself, and Phase 2 immediately returns the head.

    Summary and Comparison

    Problem Core Idea Time Space Key Insight
    Linked List Cycle (141) Fast meets slow $O(n)$ $O(1)$ Gap shrinks by 1 each step
    Cycle Entry (142) Reset + equal speed $O(n)$ $O(1)$ $F = kC - a$
    Happy Number (202) Implicit cycle detection $O(\log n)$ $O(1)$ Digit-square sequence is bounded
    Find Duplicate (287) Array as functional graph $O(n)$ $O(1)$ Pigeonhole creates a cycle
    Middle of List (876) Fast at 2x reaches end $O(n)$ $O(1)$ Slow covers half the distance
    Reorder List (143) Middle + reverse + merge $O(n)$ $O(1)$ Combines three techniques

    When to Reach for Fast/Slow Pointers

    Use fast/slow when you see:

    1. Cycle detection: Anything that might loop (linked lists, sequences, functional graphs).

    2. Finding a midpoint: Lists, arrays, or any sequential structure where you need the halfway point without knowing the length.

    3. Implicit linked lists: Arrays where values serve as next pointers (like LeetCode 287).

    4. Bounded sequences: Numerical sequences that must eventually cycle due to finiteness (like happy numbers).

    Common Mistake: Forgetting to check fast != nullptr && fast->next != nullptr before advancing fast. If the list has no cycle, accessing fast->next->next on a null pointer causes a segfault.

    Practice Problems

    1. Palindrome Linked List (LeetCode 234): Find middle, reverse second half, compare both halves.
    2. Circular Array Loop (LeetCode 457): Floyd's on array indices with direction constraints.
    3. Sort List (LeetCode 148): Find middle (fast/slow), then merge sort.
    4. Remove Nth Node From End (LeetCode 19): Two pointers with an $n$-step gap.
    5. Intersection of Two Linked Lists (LeetCode 160): Two pointers across both lists, meeting at intersection.
