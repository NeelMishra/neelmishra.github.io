    Remove Duplicates & Elements

    March 2026 · 22 min read

    A family of problems asks you to remove certain elements from an array or linked list in place, returning the new length. They all share the same backbone: a slow (write) pointer that marks where the next valid element should go, and a fast (read) pointer that scans every element. Mastering one makes the rest easy.

    Prerequisites: This page builds on the Two Pointers Overview and the Same-Direction Pointers page. If you have not read them yet, start there for the general technique and motivation.

    1. Remove Duplicates from Sorted Array (LeetCode 26)

    Problem: Given a sorted integer array nums, remove the duplicates in place so each unique element appears only once. Return the count of unique elements k. The first k elements of nums must hold the unique values in order.

    Constraints: 1 ≤ nums.length ≤ 3 × 104. Array is sorted in non-decreasing order.

    Intuition

    Because the array is sorted, duplicates always sit next to each other. We keep a slow pointer at the last position of the unique portion. A fast pointer scans forward. Every time fast finds a value different from nums[slow], we advance slow and copy the new value there.

    The Write Pointer Invariant

    At every iteration, nums[0..slow] contains unique elements in sorted order. Everything beyond slow is either unprocessed or leftover data that will be overwritten.

    Step-by-Step Walkthrough

    Consider nums = [1, 1, 2, 2, 3, 4, 4]. We start with slow = 0 and fast = 1.

    fast = 1: nums[1] = 1, same as nums[slow] = 1

    Duplicate found. Skip. Do not advance slow.

    fast = 2: nums[2] = 2, different from nums[slow] = 1

    New unique value. Advance slow to 1, write nums[1] = 2.

    fast = 3: nums[3] = 2, same as nums[slow] = 2

    Duplicate. Skip.

    fast = 4: nums[4] = 3, different from nums[slow] = 2

    Advance slow to 2, write nums[2] = 3.

    fast = 5: nums[5] = 4, different from nums[slow] = 3

    Advance slow to 3, write nums[3] = 4.

    fast = 6: nums[6] = 4, same as nums[slow] = 4

    Duplicate. Skip. Loop ends. Return slow + 1 = 4.

    Result: [1, 2, 3, 4, ...] with k = 4

    Interactive Animation

    Step through the algorithm on [1, 1, 2, 2, 3, 4, 4]. The green zone is the unique (write) portion. The orange pointer is fast (read).

    Remove Duplicates (LC 26)

    slow=0, fast=1. Press Next to begin.

    C++ Solution

    int removeDuplicates(vector<int>& nums) {
    if (nums.empty()) return 0;
    int slow = 0;
    for (int fast = 1; fast < (int)nums.size(); fast++) {
        if (nums[fast] != nums[slow]) {
            slow++;
            nums[slow] = nums[fast];
        }
    }
    return slow + 1;
}
    Complexity: Time O(n), one pass. Space O(1), only two integer pointers.

    2. Remove Duplicates II (LeetCode 80)

    Problem: Given a sorted array, remove duplicates in place so each unique element appears at most twice. Return the new length.

    Example: [1, 1, 1, 2, 2, 3] becomes [1, 1, 2, 2, 3, ...] with k = 5.

    How the Logic Changes

    In LC 26, the slow pointer compared against one element back. Here, we compare against two elements back. If nums[fast] != nums[slow - 1], the value has appeared at most once in the write zone, so we can write it again. If it equals nums[slow - 1], we already have two copies and must skip.

    The Generalized Rule

    To allow at most k duplicates, compare nums[fast] with nums[slow - (k-1)]. For k=1 (LC 26), compare with nums[slow]. For k=2 (LC 80), compare with nums[slow - 1].

    Walkthrough

    Array: [1, 1, 1, 2, 2, 3]. We start with slow = 1 (first two elements are always valid) and fast = 2.

    fast = 2: nums[2] = 1, compare with nums[slow - 1] = nums[0] = 1

    Equal. Already have two 1s. Skip.

    fast = 3: nums[3] = 2, compare with nums[slow - 1] = nums[0] = 1

    Different. Advance slow to 2, write nums[2] = 2.

    fast = 4: nums[4] = 2, compare with nums[slow - 1] = nums[1] = 1

    Different. Advance slow to 3, write nums[3] = 2.

    fast = 5: nums[5] = 3, compare with nums[slow - 1] = nums[2] = 2

    Different. Advance slow to 4, write nums[4] = 3. Loop ends.

    Result: [1, 1, 2, 2, 3, ...] with k = 5

    Interactive Animation

    Step through the algorithm on [1, 1, 1, 2, 2, 3]. Green cells are the "kept" zone (up to 2 copies allowed). Orange pointer is fast.

    Remove Duplicates II (LC 80)

    slow=1, fast=2. Press Next to begin.

    C++ Solution

    int removeDuplicates(vector<int>& nums) {
    int n = nums.size();
    if (n <= 2) return n;
    int slow = 1;
    for (int fast = 2; fast < n; fast++) {
        if (nums[fast] != nums[slow - 1]) {
            slow++;
            nums[slow] = nums[fast];
        }
    }
    return slow + 1;
}
    Generalised version: To allow at most K duplicates, initialise slow = K - 1, start fast = K, and compare nums[fast] != nums[slow - (K - 1)].

    3. Remove Element (LeetCode 27)

    Problem: Given an unsorted array nums and a value val, remove all instances of val in place. Return the new length. The order of remaining elements may change.

    Example: nums = [3, 2, 2, 3], val = 3, result = [2, 2, ...] with k = 2.

    Approach A: Same-Direction (Order Preserved)

    Use the standard read/write template. The write pointer copies every element that is not equal to val.

    int removeElement(vector<int>& nums, int val) {
    int write = 0;
    for (int read = 0; read < (int)nums.size(); read++) {
        if (nums[read] != val) {
            nums[write] = nums[read];
            write++;
        }
    }
    return write;
}

    This preserves relative order. Every element is visited exactly once. Time O(n), space O(1).

    Approach B: Swap-with-End (Fewer Writes)

    When the value to remove is rare, the same-direction approach performs many unnecessary copies. A smarter idea: when we hit val, swap it with the last element and shrink the array from the end.

    int removeElement(vector<int>& nums, int val) {
    int left = 0, right = (int)nums.size() - 1;
    while (left <= right) {
        if (nums[left] == val) {
            nums[left] = nums[right];
            right--;
        } else {
            left++;
        }
    }
    return left;
}

    When to Pick Which Approach

    Approach A does exactly n reads and at most n writes. Approach B does at most n reads and at most count(val) writes, but does not preserve order. If the number of elements to remove is small, Approach B saves writes.

    4. Remove Linked List Elements (LeetCode 203)

    Problem: Given the head of a linked list and an integer val, remove all nodes with node->val == val. Return the new head.

    Technique: Dummy Head + Two Pointers

    Linked list removal is different from array removal. Instead of overwriting, we re-link pointers. A dummy head node simplifies edge cases (what if the head itself must be removed?).

    We maintain a prev pointer (the last kept node) and a curr pointer (the node being examined). If curr->val == val, we skip it by setting prev->next = curr->next. Otherwise, we advance prev.

    Visual: Pointer Re-linking

    Removing value 6 from 1 -> 6 -> 2 -> 6 -> 3:

    Linked List Removal (LC 203)

    prev=dummy, curr=node(1). Press Next to step.

    C++ Solution

    ListNode* removeElements(ListNode* head, int val) {
    ListNode dummy(0);
    dummy.next = head;
    ListNode* prev = &dummy;
    ListNode* curr = head;
    while (curr != nullptr) {
        if (curr->val == val) {
            prev->next = curr->next;   // bypass
        } else {
            prev = curr;                // advance prev
        }
        curr = curr->next;             // always advance curr
    }
    return dummy.next;
}
    Common mistake: Forgetting the dummy node. Without it, removing the head element requires special-case logic that is easy to get wrong.

    5. Comparison of All Removal Patterns

    Problem Input Condition Pointer Roles Time Space
    LC 26: Remove Dups Sorted array Keep unique only slow = last unique, fast = scanner O(n) O(1)
    LC 80: Remove Dups II Sorted array Keep at most 2 slow = write pos, compare slow-1 O(n) O(1)
    LC 27: Remove Element (A) Any array Remove specific value write = next slot, read = scanner O(n) O(1)
    LC 27: Remove Element (B) Any array Remove specific value left = scanner, right = end shrink O(n) O(1)
    LC 203: Remove List Elements Linked list Remove specific value prev = last kept, curr = scanner O(n) O(1)

    Structural Similarities

    Every problem follows the same skeleton:

    1. One pointer reads (scans all elements linearly).
    2. Another pointer writes (tracks the boundary of valid output).
    3. A condition decides whether the current element is kept or skipped.
    4. When kept: copy/link to the write position, advance the write pointer.
    5. When skipped: advance only the read pointer.

    6. When to Use Which Approach

    Same-Direction Read/Write

    Use when: Order must be preserved, or the input is sorted and you rely on adjacency for duplicate detection.

    Examples: LC 26, LC 80, LC 27 (variant A), Move Zeroes.

    Swap-with-End

    Use when: Order does not matter and the target value is rare. Minimises writes.

    Examples: LC 27 (variant B), partitioning around a pivot.

    Dummy Head + Prev/Curr

    Use when: Working with linked lists. The dummy head eliminates special cases for head removal.

    Examples: LC 203, LC 82 (Remove Dups from Sorted List II), LC 19 (Remove Nth from End).

    Decision flow: Is the data a linked list? Use dummy head. Is it an array? Does order matter? If yes, use same-direction read/write. If no, consider swap-with-end for fewer writes.

    7. Edge Cases and Common Mistakes

    Edge Cases

    • Empty array / null head: Always check for empty input first. Return 0 or nullptr immediately.
    • All elements are duplicates: e.g., [5, 5, 5, 5]. LC 26 returns 1, LC 80 returns 2, LC 27 with val=5 returns 0.
    • No duplicates at all: The write pointer mirrors the read pointer. The array is unchanged.
    • Single element: Always valid. Return 1 (or return head for linked lists).
    • Value to remove is not present (LC 27): The write pointer reaches the end. Return original length.
    • Head node must be removed (LC 203): Without a dummy node this is a special case. With a dummy node it is handled automatically.

    Common Mistakes

    Mistake 1: Off-by-one in the return value. In LC 26, the answer is slow + 1 (length), not slow (index). In LC 80, same thing.
    Mistake 2: Comparing nums[fast] with nums[fast-1] instead of nums[slow]. This only detects adjacent duplicates, not duplicates relative to the write zone. It gives wrong results when multiple groups overlap.
    Mistake 3: Advancing prev when curr is removed (LC 203). After bypassing curr, prev must stay put because the next node has not been examined yet. Only advance prev when the current node is kept.
    Mistake 4: Forgetting that swap-with-end does not advance left. After swapping nums[left] with nums[right], do not increment left. The swapped-in value must be re-examined.

    Testing Checklist

    Test CaseWhy It Matters
    Empty array / null listBoundary: no elements to process
    [x] (single element)Boundary: trivially valid
    All same valuesMaximum removal, write pointer barely moves
    All unique valuesNo removal, write pointer mirrors read
    Target at start / endTests head removal (lists) and boundary handling
    Target not presentArray should remain unchanged
    Alternating duplicates: [1,2,1,2,1]Only valid for LC 27; sorted array precludes this

    Summary

    All in-place removal problems reduce to the same idea: one pointer decides what to keep, another pointer scans. The differences are in the keep condition and the data structure:

    • LC 26: keep if different from the last written value.
    • LC 80: keep if different from the value two positions back in the write zone.
    • LC 27: keep if not equal to the target value (two variants).
    • LC 203: keep if node value is not the target (re-link pointers instead of copying).

    Once you recognise this skeleton, each variant becomes a matter of adjusting the condition. Practice all four to build the muscle memory.
