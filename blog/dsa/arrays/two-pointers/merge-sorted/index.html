    Merging Sorted Sequences

    March 2026 · 22 min read

    Why Merging Matters

    Merging two sorted sequences into one sorted sequence is one of the most fundamental operations in computer science. It is the backbone of merge sort, powers database join operations, and appears constantly in interview problems. The core idea is simple: since both inputs are already sorted, you only need to compare the front elements of each sequence and pick the smaller one.

    This gives us O(n + m) time to merge sequences of length n and m. No element is visited more than once. The two-pointer approach is the natural fit: one pointer walks through the first sequence, the other walks through the second, and you advance whichever pointer is at the smaller value.

    Core Principle: When both inputs are sorted, a single linear scan with two pointers produces a sorted output. Each pointer moves forward only, so the total work is proportional to the combined length.

    In this post, we cover six problems that use this merge primitive in increasingly sophisticated ways, from the basic in-place array merge to the binary search approach for finding the median of two sorted arrays.

    1. Merge Sorted Array (LeetCode 88)

    Problem Statement

    You are given two integer arrays nums1 and nums2, sorted in non-decreasing order. nums1 has a length of m + n, where the first m elements are the real values and the last n elements are zeros (placeholders). Merge nums2 into nums1 in-place so that nums1 is sorted.

    Why Merge Backwards?

    The naive approach is to merge from the front. But if you insert a small element at position 0, you have to shift everything to the right. That turns an O(n + m) merge into O((n + m)2) due to repeated shifting.

    The trick is to fill nums1 from the back. The last n slots are empty placeholders, so writing there never overwrites any real data that has not been processed yet. You compare the largest unprocessed elements from both arrays and place the winner at the current back position.

    Backwards Filling Avoids Overwrites

    When you place the largest remaining element at the tail, you consume one placeholder slot. The element you placed was either from the tail of nums1 (which just freed its own slot) or from nums2 (which goes into a placeholder). Either way, no real element is overwritten.

    C++ Implementation

    void merge(vector<int>& nums1, int m,
           vector<int>& nums2, int n) {
    int i = m - 1;   // last real element in nums1
    int j = n - 1;   // last element in nums2
    int k = m + n - 1; // last position in nums1

    while (i >= 0 && j >= 0) {
        if (nums1[i] > nums2[j]) {
            nums1[k--] = nums1[i--];
        } else {
            nums1[k--] = nums2[j--];
        }
    }
    // Copy remaining elements from nums2
    while (j >= 0) {
        nums1[k--] = nums2[j--];
    }
    // No need to copy remaining nums1 elements;
    // they are already in place.
}

    Time and Space

    Time: O(m + n). Each element is placed exactly once.

    Space: O(1). The merge happens in-place inside nums1.

    Step-Through: Backward Merge

    Watch the three pointers (i, j, k) fill nums1 from right to left. nums1 = [1, 3, 5, 0, 0, 0] (m=3), nums2 = [2, 4, 6] (n=3).

    Step 0 / 6: Initial state. i=2(5), j=2(6), k=5.

    Edge Cases

    • nums2 is empty (n = 0): No work needed. nums1 is already sorted.
    • nums1 real part is empty (m = 0): Just copy all of nums2 into nums1.
    • All of nums2 are larger: The main loop places all nums2 elements at the back. The remaining nums1 elements stay in place.
    • All of nums2 are smaller: The main loop places all nums1 elements at the back first, then the cleanup loop copies nums2 to the front.

    2. Merge Two Sorted Lists (LeetCode 21)

    Problem Statement

    Given the heads of two sorted linked lists, merge them into one sorted linked list by splicing the nodes. Return the head of the merged list.

    Dummy Node Approach

    The standard technique uses a dummy head node. You create a temporary node that acts as the starting point of the merged list. A tail pointer tracks where the next node should be appended. At each step, compare the values at the heads of the two lists, detach the smaller node, and attach it after tail.

    The dummy node avoids special-casing the first insertion. Without it, you would need extra logic to decide which list provides the head of the result.

    C++ Implementation

    ListNode* mergeTwoLists(ListNode* l1, ListNode* l2) {
    ListNode dummy(0);
    ListNode* tail = &dummy;

    while (l1 && l2) {
        if (l1->val <= l2->val) {
            tail->next = l1;
            l1 = l1->next;
        } else {
            tail->next = l2;
            l2 = l2->next;
        }
        tail = tail->next;
    }
    tail->next = l1 ? l1 : l2;
    return dummy.next;
}

    Time and Space

    Time: O(n + m) where n and m are the lengths of the two lists.

    Space: O(1). We reuse existing nodes; no new nodes are allocated.

    Step-Through: Linked List Merge

    Lists: L1 = 1 → 3 → 5, L2 = 2 → 4 → 6. The merged list builds from the dummy node.

    Step 0 / 6: Compare L1(1) vs L2(2).
    Common Mistake: Forgetting to handle the remaining tail. After the main loop, one list may still have nodes. Attach the entire remaining sublist in O(1) rather than looping through it.

    3. Merge k Sorted Lists (LeetCode 23)

    Problem Statement

    Given an array of k linked lists, each sorted in ascending order, merge all lists into one sorted linked list.

    Two Approaches

    There are two main strategies. Both use the two-pointer merge as a building block.

    Approach A: Pairwise (Divide and Conquer)

    Pair up the lists and merge each pair. This halves the number of lists. Repeat until one list remains. This is exactly how merge sort works at the top level.

    ListNode* mergeKLists(vector<ListNode*>& lists) {
    if (lists.empty()) return nullptr;
    int interval = 1;
    while (interval < (int)lists.size()) {
        for (int i = 0; i + interval < (int)lists.size();
             i += interval * 2) {
            lists[i] = mergeTwoLists(lists[i],
                                     lists[i + interval]);
        }
        interval *= 2;
    }
    return lists[0];
}

    Approach B: Priority Queue (Min-Heap)

    Push the head of each list into a min-heap. Pop the smallest, append it to the result, and push the popped node's next (if it exists). The heap always contains at most k elements.

    ListNode* mergeKLists(vector<ListNode*>& lists) {
    auto cmp = [](ListNode* a, ListNode* b) {
        return a->val > b->val;
    };
    priority_queue<ListNode*, vector<ListNode*>,
                   decltype(cmp)> pq(cmp);

    for (auto* l : lists)
        if (l) pq.push(l);

    ListNode dummy(0);
    ListNode* tail = &dummy;
    while (!pq.empty()) {
        ListNode* node = pq.top(); pq.pop();
        tail->next = node;
        tail = tail->next;
        if (node->next) pq.push(node->next);
    }
    return dummy.next;
}

    Complexity Comparison

    Let N be the total number of nodes across all k lists.

    ApproachTimeSpaceWhen to Prefer
    Pairwise Merge$O(N \log k)$$O(1)$When memory is tight
    Priority Queue$O(N \log k)$$O(k)$When code simplicity matters
    Merge One-by-One$O(Nk)$$O(1)$Never (too slow)
    Why $O(N \log k)$? Pairwise merge performs $\log k$ rounds. In each round, every node participates in exactly one merge, contributing O(N) work per round. The heap approach does $N$ insertions and extractions, each costing $O(\log k)$.

    4. Sort List (LeetCode 148)

    Problem Statement

    Given the head of a linked list, sort it in ascending order in O(n log n) time and O(1) space (ignoring recursion stack).

    Merge Sort on a Linked List

    Merge sort is the natural choice for linked lists because splitting at the middle is O(n) and merging is O(n), giving O(n log n) total. The approach has two phases:

    Split: Find the Middle with Fast/Slow Pointers

    Use two pointers: slow moves one step at a time, fast moves two steps. When fast reaches the end, slow is at the middle. Cut the list in two by setting slow->next = nullptr.

    Merge: Combine Two Sorted Halves

    Recursively sort each half, then merge them using the same two-pointer merge from LeetCode 21. The base case is a list with zero or one node.

    C++ Implementation

    ListNode* sortList(ListNode* head) {
    if (!head || !head->next) return head;

    // Find middle using slow/fast
    ListNode* slow = head;
    ListNode* fast = head->next;
    while (fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
    }

    ListNode* mid = slow->next;
    slow->next = nullptr;  // cut

    ListNode* left  = sortList(head);
    ListNode* right = sortList(mid);
    return mergeTwoLists(left, right);
}

    Time and Space

    Time: $O(n \log n)$. Each level does O(n) work across all merges, and there are $\log n$ levels.

    Space: $O(\log n)$ for the recursion stack. The bottom-up iterative version achieves true O(1) space.

    Visual: Split and Merge

    This diagram shows how a 6-node list is recursively split and then merged back together.

    Pitfall: When finding the middle, use fast = head->next (not fast = head) to ensure the left half is never longer than the right. Otherwise, a two-node list splits into one node and one node, but you may accidentally get a two-node left half with an empty right half, causing infinite recursion.

    5. Median of Two Sorted Arrays (LeetCode 4)

    Problem Statement

    Given two sorted arrays nums1 of size m and nums2 of size n, find the median of the combined sorted array. The overall run time complexity must be O(log(m + n)).

    Starting from Two-Pointer Intuition

    The simplest approach is to merge the two arrays using two pointers and stop once you reach the middle. This takes O(m + n) time and clearly works. But the problem demands O(log(m + n)), so we need binary search.

    The key observation: finding the median means partitioning the combined array into a left half and a right half of equal size such that every element on the left is less than or equal to every element on the right.

    Partition, Do Not Merge

    If we choose i elements from nums1 for the left half, we must choose half - i from nums2 (where half = (m + n + 1) / 2). Binary search on i finds the valid partition in O(log(min(m, n))) time.

    Partition Validity

    Let i be the partition index in nums1 and j = half - i in nums2. The partition is valid when:

    • nums1[i - 1] <= nums2[j] (left part of nums1 does not exceed right part of nums2)
    • nums2[j - 1] <= nums1[i] (left part of nums2 does not exceed right part of nums1)

    If the first condition fails, i is too large (move left). If the second fails, i is too small (move right).

    C++ Implementation

    double findMedianSortedArrays(vector<int>& nums1,
                               vector<int>& nums2) {
    // Ensure binary search runs on the shorter array
    if (nums1.size() > nums2.size())
        swap(nums1, nums2);

    int m = nums1.size(), n = nums2.size();
    int half = (m + n + 1) / 2;
    int lo = 0, hi = m;

    while (lo <= hi) {
        int i = (lo + hi) / 2;
        int j = half - i;

        int left1  = (i > 0) ? nums1[i - 1] : INT_MIN;
        int right1 = (i < m) ? nums1[i]     : INT_MAX;
        int left2  = (j > 0) ? nums2[j - 1] : INT_MIN;
        int right2 = (j < n) ? nums2[j]     : INT_MAX;

        if (left1 <= right2 && left2 <= right1) {
            if ((m + n) % 2 == 1)
                return max(left1, left2);
            return (max(left1, left2)
                  + min(right1, right2)) / 2.0;
        } else if (left1 > right2) {
            hi = i - 1;
        } else {
            lo = i + 1;
        }
    }
    return 0.0; // unreachable for valid input
}

    Time and Space

    Time: $O(\log(\min(m, n)))$. Binary search on the shorter array.

    Space: O(1). Only a few variables.

    Visual: Binary Search Partitions

    nums1 = [1, 3, 8, 9], nums2 = [2, 4, 5, 7, 10, 11]. We binary search on nums1 (shorter). Step through to see how the partition converges.

    Step 1 / 3: Binary search on i. half = 5.
    Connection to merging: The O(m + n) two-pointer merge approach finds the median by literally building the merged array up to position (m + n) / 2. The binary search approach skips the merge entirely by finding where the partition should go. Both rely on the same sorted property but the binary search exploits it more aggressively.

    6. Sorted Array Merge Patterns

    When Is Merging Better Than Re-sorting?

    If you have two already-sorted sequences of length m and n, you have two options to combine them:

    • Merge: O(m + n) time, preserving the existing order.
    • Concatenate and sort: O((m + n) log(m + n)) time.

    Merging is always faster when the inputs are already sorted. The overhead of a general-purpose sort is wasted work because it has to rediscover an order that already exists.

    ScenarioBetter ApproachReason
    Both arrays sortedMergeO(m + n) beats O((m + n) log(m + n))
    One sorted, one unsortedSort then mergeSort the unsorted one, then merge in linear time
    Both unsortedConcatenate and sortNo sorted order to exploit
    k sorted streams arriving onlineHeap-based mergeMaintains sorted order with O(log k) per element
    Nearly sorted (few inversions)Insertion sort or mergeInsertion sort is O(n + d) where d is inversions

    Pattern Recognition Checklist

    When to Think About Merging

    1. The input has two or more sorted sequences.

    2. The problem asks for a combined sorted result or a property of the combined sorted result (like the median or kth element).

    3. The problem involves intervals that are sorted by start time.

    4. You need to maintain sorted order while streaming data.

    Merge Template

    Here is a general-purpose merge template for two sorted arrays. Adapt it to your specific problem by changing what happens when elements are compared.

    // General merge template for sorted arrays a and b
vector<int> mergeSorted(const vector<int>& a,
                         const vector<int>& b) {
    vector<int> result;
    result.reserve(a.size() + b.size());
    int i = 0, j = 0;

    while (i < (int)a.size() && j < (int)b.size()) {
        if (a[i] <= b[j]) {
            result.push_back(a[i++]);
        } else {
            result.push_back(b[j++]);
        }
    }
    while (i < (int)a.size()) result.push_back(a[i++]);
    while (j < (int)b.size()) result.push_back(b[j++]);
    return result;
}
    Stability: The template above is stable. When a[i] == b[j], it picks a[i] first. This preserves the relative order of equal elements from list a before list b. Stability matters in problems like counting inversions during merge sort.

    Summary

    All six problems in this post share a common foundation: exploiting the sorted property with two pointers to achieve linear (or better) time complexity.

    The Merge Mindset

    When you see sorted sequences, think merge before sort. The sorted property is free information. Merging uses that information in O(n) time. Re-sorting discards it and pays O(n log n) to rediscover it. Build the habit of asking: can I merge instead of sort?
