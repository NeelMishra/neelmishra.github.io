    Move Zeroes & Segregation

    March 2026 · 22 min read

    Introduction

    Many array problems ask you to rearrange elements so that one group comes before another. Move all zeroes to the end. Put even numbers first. Separate 0s from 1s. Sort three distinct values. These are all segregation problems, and they share a common backbone: the two-pointer technique.

    In this post we will work through five classic segregation problems, starting with the simplest (Move Zeroes) and building up to Sort Colors. Along the way you will see two core pointer strategies emerge: read/write pointers (same direction) and opposite-end pointers (converging from both sides). By the end, you will have a mental framework for choosing the right approach for any segregation task.

    Prerequisites: Basic understanding of arrays, swaps, and the idea behind two pointers. Familiarity with C++ syntax.

    1. Move Zeroes (LeetCode 283)

    Problem Statement

    Given an integer array nums, move all 0s to the end while maintaining the relative order of the non-zero elements. You must do this in-place without making a copy of the array.

    Example: [0, 1, 0, 3, 12] becomes [1, 3, 12, 0, 0].

    This is one of the most popular easy-level interview questions. It appears simple, but the follow-up ("Can you minimize the number of operations?") leads to interesting design choices.

    Initial Array Visualization

    Approach 1: Snowball (Write Pointer)

    The idea is simple. Maintain a write pointer that tracks the next position where a non-zero value should be placed. Walk through the array with a read pointer. Every time you find a non-zero element, copy it to the write position and advance write. After scanning the entire array, fill every position from write onward with zero.

    This is called the "snowball" approach because the block of zeroes between write and read grows like a snowball rolling downhill. The snowball gets bigger each time read passes over a zero.

    Initialize write = 0

    The write pointer starts at index 0. It represents the boundary: everything to its left is finalized non-zero values.

    Scan with read pointer

    For each index read from 0 to n-1: if nums[read] != 0, copy nums[read] to nums[write] and increment write.

    Fill remaining with zeroes

    After the loop, every index from write to n-1 should be set to 0.

    // Approach 1: Snowball / Write-pointer
void moveZeroes(vector<int>& nums) {
    int write = 0;
    for (int read = 0; read < (int)nums.size(); ++read) {
        if (nums[read] != 0) {
            nums[write] = nums[read];
            ++write;
        }
    }
    // Fill the rest with zeroes
    for (int i = write; i < (int)nums.size(); ++i) {
        nums[i] = 0;
    }
}

    Time: O(n). Space: O(1). Every element is written at most twice (once during the copy, possibly once during the zero-fill).

    Approach 2: Two-Pointer Swap

    Can we do better? The snowball approach may overwrite a position with zero even if it was already zero. The swap approach avoids unnecessary writes. When read finds a non-zero element, we swap nums[read] and nums[write], then advance write. This places the non-zero value at the front and moves the zero backward in one operation.

    // Approach 2: Swap-based
void moveZeroes(vector<int>& nums) {
    int write = 0;
    for (int read = 0; read < (int)nums.size(); ++read) {
        if (nums[read] != 0) {
            swap(nums[read], nums[write]);
            ++write;
        }
    }
}

    This is cleaner and performs fewer total writes when zeroes are sparse. The relative order of non-zero elements is preserved because both pointers move left to right and write never overtakes read.

    Why does the swap preserve order? The write pointer always points at a zero (or coincides with read). Non-zero values are placed at the write position in the exact order they are encountered by the read pointer.

    Edge case: If the array contains no zeroes, write and read stay in sync. Every element is swapped with itself (a no-op in practice). The algorithm handles this gracefully.

    2. Sort Array By Parity (LeetCode 905)

    Problem Statement

    Given an integer array nums, move all even integers to the front and all odd integers to the back. Any order within the even or odd groups is acceptable.

    Example: [3, 1, 2, 4] can become [2, 4, 3, 1] or [4, 2, 1, 3].

    Key Insight: Opposite-End Pointers

    Since relative order does not matter, we can use opposite-end pointers. Place a left pointer at the start and a right pointer at the end. Advance left while it points to an even number. Retreat right while it points to an odd number. When both stop, swap and continue.

    This is fundamentally different from Move Zeroes. Move Zeroes required maintaining relative order, so we needed same-direction pointers. Sort By Parity has no ordering constraint, so the faster opposite-end approach works.

    Opposite-End Segregation

    Use this when you need to partition into two groups and relative order within each group does not matter. One pointer scans from the left for misplaced items, the other scans from the right. Swap when both find a mismatch.

    // Sort Array By Parity: opposite-end pointers
vector<int> sortArrayByParity(vector<int>& nums) {
    int left = 0, right = (int)nums.size() - 1;
    while (left < right) {
        // Advance left past even numbers
        while (left < right && nums[left] % 2 == 0) ++left;
        // Retreat right past odd numbers
        while (left < right && nums[right] % 2 == 1) --right;
        if (left < right) {
            swap(nums[left], nums[right]);
            ++left;
            --right;
        }
    }
    return nums;
}

    Time: O(n). Space: O(1). Each element is visited at most once by each pointer.

    3. Sort Array By Parity II (LeetCode 922)

    Problem Statement

    Given an array where half the elements are even and half are odd, rearrange it so that nums[i] is even when i is even, and odd when i is odd.

    Example: [4, 2, 5, 7] becomes [4, 5, 2, 7].

    Approach: Two Independent Pointers

    Use one pointer even that walks through even indices (0, 2, 4, ...) and one pointer odd that walks through odd indices (1, 3, 5, ...). When even finds an odd value at an even position, and odd finds an even value at an odd position, swap them. Both pointers only move forward through their respective index sets.

    Why This Works

    If an even index holds an odd value, there must be a corresponding odd index holding an even value (since exactly half the values are even). By scanning both tracks simultaneously, we always find a valid swap partner.

    // Sort Array By Parity II
vector<int> sortArrayByParityII(vector<int>& nums) {
    int n = (int)nums.size();
    int even = 0, odd = 1;
    while (even < n && odd < n) {
        // Find misplaced value at an even index
        while (even < n && nums[even] % 2 == 0) even += 2;
        // Find misplaced value at an odd index
        while (odd < n && nums[odd] % 2 == 1) odd += 2;
        if (even < n && odd < n) {
            swap(nums[even], nums[odd]);
            even += 2;
            odd += 2;
        }
    }
    return nums;
}

    Time: O(n). Space: O(1). Each pointer visits at most n/2 indices.

    4. Segregate 0s and 1s

    Problem Statement

    Given a binary array (containing only 0s and 1s), segregate all 0s to the left and all 1s to the right. This is a common interview warm-up that tests your understanding of basic partitioning.

    Example: [0, 1, 0, 1, 1, 0] becomes [0, 0, 0, 1, 1, 1].

    Two Approaches

    A. Same-Direction (Read/Write)

    Use a write pointer for 0s. Scan left to right. When you see a 0, swap it into the write position and advance write. This preserves relative order if needed.

    B. Opposite-End Pointers

    Left pointer advances past 0s, right pointer retreats past 1s. When both stop, swap. This does not preserve relative order but is conceptually simpler.

    // Segregate 0s and 1s: opposite-end approach
void segregate01(vector<int>& arr) {
    int left = 0, right = (int)arr.size() - 1;
    while (left < right) {
        while (left < right && arr[left] == 0) ++left;
        while (left < right && arr[right] == 1) --right;
        if (left < right) {
            swap(arr[left], arr[right]);
            ++left;
            --right;
        }
    }
}
    // Segregate 0s and 1s: same-direction (order-preserving)
void segregate01Stable(vector<int>& arr) {
    int write = 0;
    for (int read = 0; read < (int)arr.size(); ++read) {
        if (arr[read] == 0) {
            swap(arr[read], arr[write]);
            ++write;
        }
    }
}

    Time: O(n) for both approaches. Space: O(1).

    Note: For binary arrays, you can also simply count the number of 0s, then overwrite the first count positions with 0 and the rest with 1. This is O(n) time and O(1) space but involves two passes. The two-pointer approach does it in one pass.

    5. Sort Colors (LeetCode 75)

    Problem Statement

    Given an array containing only 0s, 1s, and 2s, sort it in-place so that all 0s come first, then 1s, then 2s. Do it in one pass with constant extra space.

    Example: [2, 0, 2, 1, 1, 0] becomes [0, 0, 1, 1, 2, 2].

    This is a direct application of the Dutch National Flag algorithm by Edsger Dijkstra. While there is a dedicated page on that topic in this series, we cover it here from the two-pointer (actually three-pointer) perspective.

    Approach 1: Two-Pass Segregation

    The simplest approach partitions twice. First pass: use two pointers to move all 0s to the front. Second pass: starting from where the 0s end, move all 1s next. The 2s naturally end up at the back.

    // Sort Colors: two-pass approach
void sortColors(vector<int>& nums) {
    int n = (int)nums.size();
    // Pass 1: segregate 0s to front
    int write = 0;
    for (int i = 0; i < n; ++i) {
        if (nums[i] == 0) {
            swap(nums[i], nums[write]);
            ++write;
        }
    }
    // Pass 2: segregate 1s after the 0s
    for (int i = write; i < n; ++i) {
        if (nums[i] == 1) {
            swap(nums[i], nums[write]);
            ++write;
        }
    }
}

    This is O(n) time and O(1) space, but it uses two passes. The interviewer usually wants a single-pass solution.

    Approach 2: One-Pass (Three Pointers)

    Use three pointers: low, mid, and high.

    • low tracks the boundary for 0s (everything before low is 0).
    • high tracks the boundary for 2s (everything after high is 2).
    • mid is the scanning pointer that processes each element.

    Three-Way Partition Invariant

    At every step: indices [0..low-1] contain 0s, indices [low..mid-1] contain 1s, indices [mid..high] are unprocessed, and indices [high+1..n-1] contain 2s.

    // Sort Colors: one-pass (Dutch National Flag)
void sortColors(vector<int>& nums) {
    int low = 0, mid = 0, high = (int)nums.size() - 1;
    while (mid <= high) {
        if (nums[mid] == 0) {
            swap(nums[low], nums[mid]);
            ++low;
            ++mid;
        } else if (nums[mid] == 2) {
            swap(nums[mid], nums[high]);
            --high;
            // Do NOT advance mid: the swapped element is unprocessed
        } else {
            // nums[mid] == 1, already in correct region
            ++mid;
        }
    }
}
    Common mistake: Incrementing mid after swapping with high. The element swapped from high has not been examined yet, so mid must stay in place.

    Why the One-Pass Solution Works

    The three pointers carve the array into four regions. The mid pointer scans the unprocessed region and routes each element to the correct zone. When mid passes high, the unprocessed region is empty and the array is sorted.

    The key subtlety: when we swap nums[mid] with nums[high], the value coming from high could be 0, 1, or 2. We must re-examine it (hence no mid++). But when we swap with nums[low], the value coming back is always a 1 (because everything between low and mid-1 is 1), so we can safely advance mid.

    6. Comparison of Segregation Patterns

    All five problems share the two-pointer backbone, but the choice between same-direction and opposite-end pointers depends on two factors: (1) whether relative order must be preserved, and (2) how many groups you are partitioning into.

    Problem Groups Order Preserved? Pointer Style Passes
    Move Zeroes 2 (non-zero, zero) Yes Same-direction (read/write) 1
    Sort By Parity 2 (even, odd) No Opposite-end 1
    Sort By Parity II 2 (even/odd interleaved) N/A (positional) Two independent tracks 1
    Segregate 0s and 1s 2 Depends on approach Either style works 1
    Sort Colors 3 Yes (values are sorted) Three pointers (DNF) 1 (one-pass) or 2

    Decision Flowchart

    Choosing the Right Segregation Strategy

    When to Use Read/Write Pointers

    Read/write pointers (same direction) are the right choice when:

    • You must preserve relative order of elements within a group.
    • The problem says "move X to end" or "remove X in-place" while keeping everything else stable.
    • You are building the result from left to right in a single scan.

    The pattern: write is always at or behind read. The region [0..write-1] is the finalized output. The region [write..read-1] contains elements that will be overwritten or are already accounted for. The region [read..n-1] is unexplored.

    When to Use Opposite-End Pointers

    Opposite-end pointers are the right choice when:

    • Relative order within groups does not matter.
    • You are partitioning into exactly two groups (or a pivot-based partition like in quicksort).
    • Each pointer can skip over "correctly placed" elements, converging toward the middle.

    The pattern: left advances past correctly placed elements on the left side, right retreats past correctly placed elements on the right side. When both stop, swap and continue. Terminates when they cross.

    Complexity Summary

    Problem Time Space Min Swaps
    Move Zeroes (snowball) O(n) O(1) Writes: n + (n - nonzero_count)
    Move Zeroes (swap) O(n) O(1) Swaps: count of non-zeroes
    Sort By Parity O(n) O(1) Swaps: count of misplaced pairs
    Sort By Parity II O(n) O(1) Swaps: count of misplaced pairs
    Segregate 0s/1s O(n) O(1) Swaps: min(count_0s_right, count_1s_left)
    Sort Colors (one-pass) O(n) O(1) Swaps: at most n

    Practice Tips

    Draw the Pointer Regions

    Before writing code, sketch the array and label the regions each pointer defines. For Move Zeroes: [finalized non-zeroes | snowball of zeroes | unexplored]. For Sort Colors: [0s | 1s | unprocessed | 2s]. This invariant drives the logic.

    Watch for the "Do Not Advance" Trap

    In Sort Colors, after swapping with the high pointer, the scanning pointer must not advance because the incoming element is unexamined. This is the most common bug. If your regions overlap unexpectedly, check whether you are advancing a pointer too early.

    Test with Edge Cases

    Always test with: (a) an already-sorted array, (b) a reverse-sorted array, (c) all elements the same, (d) array of length 0 or 1. These edge cases catch off-by-one errors in pointer initialization and termination.

    Connect to Quicksort Partition

    The Lomuto partition scheme uses same-direction pointers (like Move Zeroes). The Hoare partition scheme uses opposite-end pointers (like Sort By Parity). Understanding segregation patterns deepens your understanding of quicksort itself.

    Summary

    Segregation problems ask you to rearrange an array so that elements satisfying some criterion are grouped together. The two-pointer technique handles all of these efficiently in O(n) time and O(1) space.

    • Move Zeroes: Same-direction pointers. The snowball approach copies then fills; the swap approach is cleaner.
    • Sort By Parity: Opposite-end pointers. No order constraint means we can converge from both sides.
    • Sort By Parity II: Two independent track pointers (even indices and odd indices), swapping misplaced pairs.
    • Segregate 0s and 1s: Either pointer style works. Opposite-end is simpler when order does not matter.
    • Sort Colors: Three pointers (Dutch National Flag). One pass, but requires care not to skip unexamined elements.

    The unifying theme: define clear regions with your pointers, maintain an invariant about what each region contains, and process elements one at a time until the unprocessed region is empty.
