    Container With Most Water

    March 2026 · 18 min read

    Problem Statement (LeetCode 11)

    You are given an integer array height of length n. There are n vertical lines drawn such that the two endpoints of the i-th line are (i, 0) and (i, height[i]).

    Find two lines that, together with the x-axis, form a container that holds the most water. Return the maximum amount of water a container can store.

    The container cannot be tilted. Water is limited by the shorter of the two chosen lines.

    Formal definition: Given an array height[0..n-1], find indices i < j that maximise min(height[i], height[j]) * (j - i).

    Visual Explanation

    Below is an example with height = [1, 8, 6, 2, 5, 4, 8, 3, 7]. Each vertical bar represents one element. The shaded blue region between indices 1 and 8 holds the most water: min(8, 7) * (8 - 1) = 49.

    The green bars (indices 1 and 8) form the optimal container. Water level is limited by the shorter bar (height 7). Width is 8 - 1 = 7, so area = 7 * 7 = 49.

    Constraints

    • n == height.length
    • 2 <= n <= 105
    • 0 <= height[i] <= 104

    Brute Force: Check Every Pair

    The most straightforward approach: try every pair of lines (i, j) and compute the area. Keep track of the maximum.

    int maxArea(vector<int>& height) {
    int n = height.size();
    int best = 0;
    for (int i = 0; i < n; i++) {
        for (int j = i + 1; j < n; j++) {
            int w = j - i;
            int h = min(height[i], height[j]);
            best = max(best, w * h);
        }
    }
    return best;
}
    MetricValue
    Time ComplexityO(n2)
    Space ComplexityO(1)

    With n = 105, this means roughly 5 billion operations. Far too slow for the given constraints. We need a way to skip pairs that cannot possibly improve the answer.

    Two Pointer Strategy

    Place a pointer at each end of the array. At each step, compute the area formed by the two lines, then move the pointer pointing to the shorter line inward. Continue until the pointers meet.

    int maxArea(vector<int>& height) {
    int left = 0, right = height.size() - 1;
    int best = 0;
    while (left < right) {
        int w = right - left;
        int h = min(height[left], height[right]);
        best = max(best, w * h);
        if (height[left] < height[right])
            left++;
        else
            right--;
    }
    return best;
}
    MetricValue
    Time ComplexityO(n)
    Space ComplexityO(1)

    Why Move the Shorter Line?

    This is the core insight. Consider a state where left and right point to two lines, and height[left] < height[right]. The current area is:

    area = height[left] * (right - left)

    What happens if we move the right pointer inward instead? The width (right - left) decreases by 1. The water level is still bounded by height[left] because the left side is the shorter one. Even if the new right line is taller, the water level cannot exceed height[left]. So the area can only decrease or stay the same (with strictly less width and the same or lower height).

    Therefore, no pair involving the current left pointer and any index less than the current right can beat the current area. We can safely discard the shorter side and move it inward.

    The shorter line is the bottleneck

    When height[left] < height[right], keeping left fixed and moving right inward can only reduce width while the height cap remains height[left]. The area can never improve. Moving left inward gives a chance of finding a taller line that raises the height cap.

    Formal Proof of Correctness

    The critical question: how do we know we never skip the optimal pair? This section gives a rigorous argument.

    Theorem

    The two-pointer algorithm considers the optimal pair (i*, j*) at some point during execution, so it correctly finds the maximum area.

    Proof by Contradiction

    Suppose the optimal pair is (i*, j*) with i* < j*, and suppose the algorithm never evaluates this pair. The algorithm starts with left = 0, right = n - 1. Since it skipped (i*, j*), at some step either:

    1. left moved past i* (i.e., left became i* + 1) while right > j*, or
    2. right moved past j* (i.e., right became j* - 1) while left < i*.

    Consider case 1. At the moment left = i* and right = r for some r > j*, the algorithm moved left forward. This means height[i*] < height[r] (or they were equal and we moved right, but let us handle the general case). The area at (i*, r) is:

    min(height[i*], height[r]) * (r - i*) = height[i*] * (r - i*)

    Since r > j*, we have r - i* > j* - i*. And the height is height[i*] in both cases (because height[i*] is the shorter side). So the area at (i*, r) is:

    height[i*] * (r - i*) ≥ height[i*] * (j* - i*) ≥ min(height[i*], height[j*]) * (j* - i*)

    This means the pair (i*, r) has area at least as large as the optimal pair (i*, j*). But we assumed (i*, j*) was optimal, so the area at (i*, r) equals the optimal area. The algorithm did evaluate (i*, r) and recorded its area. Since this area equals the optimal, the algorithm returns the correct answer. Contradiction with the assumption that the algorithm misses the optimum.

    Case 2 is symmetric. This completes the proof.

    Intuition summary: When we discard the shorter side, we are only discarding pairs that are provably no better than a pair we have already seen. The wider pairing with the same short side always produces at least the same area.

    Variations and Related Problems

    Container With Most Water vs Trapping Rain Water

    These two problems look similar but are fundamentally different. Container With Most Water picks exactly two lines and measures a single rectangular region. Trapping Rain Water considers water trapped above every bar between the walls.

    AspectContainer With Most WaterTrapping Rain Water
    What we chooseTwo lines (boundaries of one container)All bars together (water above each bar)
    Water shapeSingle rectangle between two chosen linesIrregular shape filling gaps between bars
    Formulamin(h[i], h[j]) * (j - i)Sum of min(leftMax[i], rightMax[i]) - h[i] for each i
    Bars in betweenIgnored (water goes over them)Bars block water, creating individual pools
    Two-pointer approachMove shorter side inwardMove side with smaller max inward, accumulate water
    Time / SpaceO(n) / O(1)O(n) / O(1) with two pointers, O(n) with prefix arrays
    Common confusion: In Container With Most Water, the bars between the two chosen lines do NOT block the water. The container is formed by only the two chosen lines and the x-axis. In Trapping Rain Water, every bar matters because water fills the gaps above shorter bars.

    Volume of Water in 3D

    A natural extension is the 3D version: given a 2D grid of heights (an elevation map), compute how much water it can trap. This is LeetCode 407 (Trapping Rain Water II). The two-pointer technique does not directly apply. Instead, you use a priority queue (min-heap) starting from the boundary cells and processing inward, always expanding from the lowest boundary cell.

    Key differences from the 1D problem:

    • Water can flow in all four directions, not just left/right.
    • The boundary is a 2D perimeter, not just two endpoints.
    • Time complexity: O(m * n * log(m * n)) where m and n are grid dimensions.

    Common Mistakes and Edge Cases

    Moving the taller side instead of the shorter side

    If you move the taller side, you lose potential containers. The taller side might pair with a future line to create a larger area. Always move the shorter side.

    Handling equal heights incorrectly

    When height[left] == height[right], it does not matter which pointer you move. Both sides are equally limiting. Moving either one is correct. Some implementations move both pointers inward simultaneously. All three approaches produce the correct answer.

    Confusing this with Trapping Rain Water

    The water in this problem forms a single rectangle between two chosen lines. Bars in between do not block the water. If you try to account for intermediate bars, you are solving the wrong problem.

    All bars have the same height

    When all heights are equal, the optimal pair is the outermost one (indices 0 and n-1), giving area height[0] * (n - 1). The two-pointer algorithm naturally picks this on the first step.

    Strictly increasing or decreasing array

    For a sorted array like [1, 2, 3, 4, 5], the optimal container is not necessarily at the ends. For example, (0, 4) gives area 1 * 4 = 4, but (1, 4) gives 2 * 3 = 6. The two-pointer approach handles this correctly by moving the smaller side inward.

    Array of length 2

    The minimum valid input. There is only one possible container. The algorithm computes it in a single iteration.

    Zero-height bars

    If height[i] = 0, any container involving that bar has area 0. The two-pointer algorithm handles this naturally since min(0, anything) = 0, and the zero-height pointer gets moved immediately.

    Time and Space Analysis

    Time: O(n)

    Each iteration of the while loop moves exactly one pointer inward (either left increments or right decrements). The pointers start n - 1 apart and meet when left == right. Therefore, the loop runs exactly n - 1 times.

    Each iteration does O(1) work: one comparison, one min, one max, one addition or subtraction. Total: O(n).

    Space: O(1)

    We use only a constant number of variables: left, right, best, w, h. No auxiliary data structures are needed.

    Optimality: You must read every element at least once (any element could be part of the optimal pair), so O(n) time is optimal for this problem. The O(1) space is also optimal since the output is a single integer.

    How to Recognise This Pattern

    Look for these signals:

    • You are optimising over pairs of indices in an array.
    • The objective function involves both the distance between indices and the values at those indices.
    • There is a monotonic trade-off: moving one pointer in a direction predictably affects the objective.
    • The brute force is O(n2) and the constraint is n up to 105, hinting at an O(n) solution.

    General Template

    // Opposite-end two-pointer template
int left = 0, right = n - 1;
int result = initial_value;
while (left < right) {
    // Compute current candidate
    int candidate = compute(left, right);
    result = better(result, candidate);
    // Move the pointer that limits improvement
    if (should_move_left(left, right))
        left++;
    else
        right--;
}
return result;

    The challenge for each problem is identifying which pointer limits improvement and proving that discarding that side is safe.

    Summary

    Container With Most Water

    Start with pointers at both ends. Compute the area. Move the shorter side inward. The shorter side is the bottleneck: keeping it fixed while reducing width can never improve the area. This gives an O(n) time, O(1) space solution that provably never misses the optimal pair.
