    Binary Search Fundamentals

    March 2026 · 30 min read

    Binary search is the most important algorithm you will ever learn. It transforms a linear scan into a logarithmic one by exploiting a single property: order. If the data is sorted, you can eliminate half the candidates with every comparison. This post covers the core idea, the mathematics behind $O(\log n)$, a full step-through animation, iterative and recursive code, the loop invariant that proves correctness, common off-by-one pitfalls, and the three essential templates you will use in every binary search problem: exact match, lower bound, and upper bound.

    The Core Idea

    Suppose you have a sorted array of $n$ elements and you want to find whether a target value exists. A naive linear scan checks every element one by one: $O(n)$. Binary search does something smarter.

    1. Look at the middle element.
    2. If it equals the target, you are done.
    3. If the target is smaller, discard the right half. The target cannot be there because the array is sorted.
    4. If the target is larger, discard the left half.
    5. Repeat on the remaining half.

    Each step eliminates exactly half of the remaining elements. After $k$ steps, only $\frac{n}{2^k}$ elements remain. The search ends when one element is left:

    $$\frac{n}{2^k} = 1 \implies k = \log_2 n$$

    So binary search runs in $O(\log n)$ time. For an array of one million elements, that is at most 20 comparisons instead of one million.

    Prerequisite: The array must be sorted. Binary search on an unsorted array gives meaningless results. If you need to search repeatedly, sort once in $O(n \log n)$ and then binary search each query in $O(\log n)$.

    Why $O(\log n)$?

    At each level of recursion, we do $O(1)$ work (one comparison) and recurse on half the input. By the Master Theorem (Case 2 with $a=1, b=2, f(n)=O(1)$), this resolves to $T(n) = O(\log n)$.

    Alternatively, think of it as a decision tree. Each comparison gives a binary outcome (go left or go right), and the tree has depth $\log_2 n$. The number of comparisons is the depth of the path from root to leaf.

    Concrete numbers

    Array size $n$ Linear scan (worst) Binary search (worst)
    16164
    1,0241,02410
    1,000,0001,000,00020
    $10^9$$10^9$30
    $10^{18}$$10^{18}$60

    The jump from $10^{18}$ operations to 60 comparisons is staggering. This is the power of logarithmic time.

    Manual Walkthrough

    Let us trace binary search on the array [3, 7, 11, 18, 24, 37, 42, 55, 68, 91] searching for target = 37.

    Initialize pointers

    lo = 0, hi = 9, mid = (0 + 9) / 2 = 4.

    Compare arr[4] = 24 with target 37. Since 24 < 37, the target is in the right half.

    Update: lo = mid + 1 = 5.

    24 < 37, so we discard indices 0 through 4.

    Narrow to right half

    lo = 5, hi = 9, mid = (5 + 9) / 2 = 7.

    Compare arr[7] = 55 with target 37. Since 55 > 37, the target is in the left half.

    Update: hi = mid - 1 = 6.

    55 > 37, so we discard indices 7 through 9.

    Two elements remain

    lo = 5, hi = 6, mid = (5 + 6) / 2 = 5.

    Compare arr[5] = 37 with target 37. Match found!

    Found target 37 at index 5 in just 3 comparisons out of 10 elements.

    Template 1: Exact Match

    This is the standard binary search. The search space uses inclusive bounds on both ends: lo and hi both point to valid indices. The loop condition is lo <= hi and we return as soon as we find the target.

    Exact Match Template

    int binarySearch(vector<int>& arr, int target) {
    int lo = 0, hi = (int)arr.size() - 1;

    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;

        if (arr[mid] == target) {
            return mid;
        } else if (arr[mid] < target) {
            lo = mid + 1;
        } else {
            hi = mid - 1;
        }
    }

    return -1;  // not found
}

    Line-by-line analysis

    • lo = 0, hi = n - 1: The search space is the entire array, inclusive on both ends.
    • while (lo <= hi): We continue as long as at least one element remains. When lo > hi, the window is empty and the target does not exist.
    • mid = lo + (hi - lo) / 2: Equivalent to (lo + hi) / 2 but avoids integer overflow when lo + hi exceeds INT_MAX. This is a real bug that existed in the Java standard library for years.
    • arr[mid] < target: Target is larger, so everything at index mid and below is irrelevant. Set lo = mid + 1.
    • arr[mid] > target: Target is smaller, so everything at index mid and above is irrelevant. Set hi = mid - 1.

    Recursive Binary Search

    The recursive form mirrors the mathematical definition more closely but uses $O(\log n)$ stack space.

    int binarySearchRec(vector<int>& arr, int target, int lo, int hi) {
    if (lo > hi) return -1;

    int mid = lo + (hi - lo) / 2;

    if (arr[mid] == target) return mid;
    if (arr[mid] < target)
        return binarySearchRec(arr, target, mid + 1, hi);
    return binarySearchRec(arr, target, lo, mid - 1);
}

// Call: binarySearchRec(arr, target, 0, arr.size() - 1);

    Both versions have identical time complexity: $O(\log n)$. The iterative version uses $O(1)$ auxiliary space while the recursive version uses $O(\log n)$ due to the call stack. Prefer iterative in practice.

    The Loop Invariant

    A correct binary search maintains this invariant at the start of every iteration:

    Invariant: If the target exists in the array, it is within arr[lo..hi] (inclusive).

    Proving correctness requires showing three things:

    1. Initialization: Before the loop, lo = 0 and hi = n - 1, which covers the entire array. The invariant holds.
    2. Maintenance: If arr[mid] < target, then all elements at indices lo through mid are less than or equal to arr[mid] (sorted order), so they cannot be the target. Setting lo = mid + 1 preserves the invariant. The symmetric argument holds when arr[mid] > target.
    3. Termination: When lo > hi, the search space is empty. Combined with the invariant, this means the target is not in the array. Returning -1 is correct.

    The invariant also guarantees termination: each iteration either finds the target (done) or strictly shrinks the window by at least one element (lo increases or hi decreases). A window of size $n$ can shrink at most $n$ times.

    Off-by-One Pitfalls

    Binary search is famously easy to get wrong. Knuth noted that while the idea was published in 1946, the first correct implementation did not appear until 1962. Here are the most common mistakes.

    Pitfall 1: Using < instead of <=

    Bug: 
    while (lo < hi) {  // WRONG for exact match template
    ...
}

    This terminates when lo == hi, skipping the last element. If the target is at that position, it will be missed.

    Fix: 
    while (lo <= hi) {  // Correct: checks the last remaining element
    ...
}

    Pitfall 2: Not moving past mid

    Bug: 
    lo = mid;      // WRONG: infinite loop when lo == mid
hi = mid;     // WRONG: infinite loop when hi == mid

    When two elements remain (lo + 1 == hi), mid equals lo. Setting lo = mid does not shrink the window, causing an infinite loop.

    Fix: 
    lo = mid + 1;   // Always move past mid
hi = mid - 1;  // Always move past mid

    Pitfall 3: Integer overflow in mid calculation

    Bug: 
    int mid = (lo + hi) / 2;  // Overflows if lo + hi > INT_MAX
    Fix: 
    int mid = lo + (hi - lo) / 2;  // Overflow-safe

    Alternatively, in languages with unsigned right shift: int mid = (lo + hi) >>> 1;

    Pitfall 4: Signed vs unsigned comparison

    Bug: 
    int hi = arr.size() - 1;  // If arr is empty, size() returns 0
                           // and size() - 1 wraps to a huge number
                           // because size() returns size_t (unsigned)
    Fix: 
    int hi = (int)arr.size() - 1;  // Cast to signed first

    Or check for emptiness before searching.

    Template 2: Lower Bound

    Lower bound finds the first position where the target could be inserted to keep the array sorted. Equivalently, it finds the index of the first element that is greater than or equal to the target. If the target exists, lower bound points to its first occurrence. If the target does not exist, it points to where it would go.

    This is the most versatile binary search template. Most binary search problems reduce to a lower bound query.

    Lower Bound Template

    // Returns the index of the first element >= target.
// If all elements are less than target, returns n.
int lowerBound(vector<int>& arr, int target) {
    int lo = 0, hi = (int)arr.size();  // note: hi = n, not n-1

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (arr[mid] < target) {
            lo = mid + 1;
        } else {
            hi = mid;   // mid could be the answer
        }
    }

    return lo;  // lo == hi == insertion point
}

    Key differences from exact match

    • hi = n (not n - 1). The answer might be "past the end" if all elements are smaller than the target.
    • while (lo < hi) (not <=). The loop converges when lo == hi, which is the answer.
    • When arr[mid] >= target, we set hi = mid (not mid - 1) because mid itself might be the first valid position.
    • There is no early return. We always let the loop converge.

    Lower bound invariant

    Invariant: At every point, arr[0..lo-1] are all strictly less than target, and arr[hi..n-1] are all greater than or equal to target. When lo == hi, the boundary is determined.

    Using lower bound for exact match

    You can implement exact match using lower bound:

    int idx = lowerBound(arr, target);
if (idx < (int)arr.size() && arr[idx] == target) {
    // found at index idx
} else {
    // not found
}

    Counting occurrences

    If the array has duplicates, the number of occurrences of a value equals upperBound(target) - lowerBound(target).

    Template 3: Upper Bound

    Upper bound finds the first position where an element is strictly greater than the target. It is the insertion point that would place the target after all existing copies.

    Upper Bound Template

    // Returns the index of the first element > target.
// If all elements are <= target, returns n.
int upperBound(vector<int>& arr, int target) {
    int lo = 0, hi = (int)arr.size();

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (arr[mid] <= target) {
            lo = mid + 1;
        } else {
            hi = mid;
        }
    }

    return lo;
}

    The only difference from lower bound is the comparison: <= instead of <. This one-character change shifts the boundary from "first >= target" to "first > target".

    Comparing the Three Templates

    Property Exact Match Lower Bound Upper Bound
    Initial hi n - 1 n n
    Loop condition lo <= hi lo < hi lo < hi
    Shrink left lo = mid + 1 lo = mid + 1 lo = mid + 1
    Shrink right hi = mid - 1 hi = mid hi = mid
    Early return? Yes No No
    Returns Index or -1 First >= target First > target
    Best for Single lookup First occurrence, insert position Last occurrence + 1, range count

    Decision guide

    When you encounter a binary search problem, ask yourself: "What exactly do I need to find?"

    • Does the target exist? Use exact match.
    • First occurrence of target? Use lower bound, then check arr[lo] == target.
    • Last occurrence of target? Use upperBound(target) - 1, then verify.
    • Count of target? upperBound(target) - lowerBound(target).
    • Insert position? Lower bound gives you exactly this.
    • Monotonic predicate on answer range? Use lower bound on the answer space (covered in the Advanced post).

    Binary Search in the C++ STL

    The C++ Standard Library provides all three operations:

    #include <algorithm>
#include <vector>

vector<int> arr = {3, 7, 11, 18, 24, 37, 37, 42, 55, 68};

// Exact match (returns bool, not index)
bool found = binary_search(arr.begin(), arr.end(), 37);

// Lower bound: iterator to first element >= 37
auto lb = lower_bound(arr.begin(), arr.end(), 37);
int lbIdx = lb - arr.begin();  // 5

// Upper bound: iterator to first element > 37
auto ub = upper_bound(arr.begin(), arr.end(), 37);
int ubIdx = ub - arr.begin();  // 7

// Count of 37s
int count = ubIdx - lbIdx;     // 2

// Equal range: pair of (lower_bound, upper_bound)
auto [first, last] = equal_range(arr.begin(), arr.end(), 37);
// first points to index 5, last points to index 7

    In Python, the bisect module provides equivalent functionality:

    import bisect

arr = [3, 7, 11, 18, 24, 37, 37, 42, 55, 68]

lb = bisect.bisect_left(arr, 37)   # 5 (lower bound)
ub = bisect.bisect_right(arr, 37)  # 7 (upper bound)
count = ub - lb                     # 2

    Key Takeaways

    • Binary search halves the search space at every step, giving $O(\log n)$ time.
    • The array (or search space) must have a monotonic property.
    • Use lo + (hi - lo) / 2 to avoid overflow.
    • The exact match template uses lo <= hi with lo = mid + 1 and hi = mid - 1.
    • The lower/upper bound templates use lo < hi with hi = mid (no minus one) and converge when lo == hi.
    • Lower bound is the most versatile: it answers "first >= target" and doubles as insert position.
    • Upper bound is identical to lower bound except it uses <= instead of <.
    • Always verify your loop terminates by ensuring the search space shrinks every iteration.
