    Advanced Binary Search

    March 2026 · 35 min read

    In the fundamentals post we covered binary search on sorted arrays with three templates. This post moves beyond searching for a value in a sorted array. We cover three families of problems that all reduce to binary search but look nothing like the classic version: search on the answer (where you binary search the result space), rotated sorted arrays (where the sort order is broken at one point), and peak element (where there is no global sort at all). Each section includes code, analysis, and interactive SVG animations.

    Search on the Answer

    Sometimes you are not searching for an element in an array. Instead, the answer itself lies in a numeric range, and you need to find the optimal value. The key insight: if you can write a function feasible(x) that returns true for all values above (or below) some threshold and false otherwise, then the function is monotonic and you can binary search on the answer.

    Pattern: Define a predicate feasible(x). If the predicate is monotonic (once it becomes true, it stays true), binary search finds the boundary.

    Framework

    // Binary search on the answer: find minimum x such that feasible(x) is true
int lo = MIN_POSSIBLE, hi = MAX_POSSIBLE;

while (lo < hi) {
    int mid = lo + (hi - lo) / 2;

    if (feasible(mid)) {
        hi = mid;       // mid works; try smaller
    } else {
        lo = mid + 1;   // mid does not work; need larger
    }
}
// lo == hi == minimum feasible answer

    This is exactly the lower bound template from the fundamentals post, applied to an abstract answer space instead of an array.

    Example 1: Minimize the Maximum (Capacity to Ship Packages)

    Problem: You have an array of package weights and $D$ days to ship them all. Packages must be shipped in order. Each day, you load consecutive packages onto a ship until the total weight would exceed the ship's capacity. Find the minimum ship capacity that lets you ship everything in $D$ days.

    Why binary search works: If a capacity of $C$ is enough to ship in $D$ days, then any capacity $C' > C$ is also enough. The feasibility predicate is monotonic.

    bool canShip(vector<int>& weights, int days, int capacity) {
    int currentLoad = 0, daysNeeded = 1;

    for (int w : weights) {
        if (currentLoad + w > capacity) {
            daysNeeded++;
            currentLoad = 0;
        }
        currentLoad += w;
    }

    return daysNeeded <= days;
}

int shipWithinDays(vector<int>& weights, int days) {
    int lo = *max_element(weights.begin(), weights.end());
    int hi = accumulate(weights.begin(), weights.end(), 0);

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (canShip(weights, days, mid)) {
            hi = mid;
        } else {
            lo = mid + 1;
        }
    }

    return lo;
}

    Bounds reasoning

    • lo = max(weights): The ship must be able to carry at least the heaviest package.
    • hi = sum(weights): If the ship can carry everything, one day is enough.
    • The search space is $[\max, \text{sum}]$. Binary search finds the smallest capacity where canShip returns true.

    Complexity: $O(n \log S)$ where $S = \text{sum} - \text{max}$ is the range of the answer space.

    Example 2: Capacity Planning (Split Array Largest Sum)

    Problem: Split an array into $k$ non-empty contiguous subarrays to minimize the largest subarray sum.

    This is the same structure as the shipping problem with a different story. The predicate checks: "Given a maximum subarray sum of $x$, can we split the array into at most $k$ parts?"

    bool canSplit(vector<int>& nums, int k, int maxSum) {
    int parts = 1, currentSum = 0;

    for (int x : nums) {
        if (currentSum + x > maxSum) {
            parts++;
            currentSum = 0;
        }
        currentSum += x;
    }

    return parts <= k;
}

int splitArray(vector<int>& nums, int k) {
    int lo = *max_element(nums.begin(), nums.end());
    int hi = accumulate(nums.begin(), nums.end(), 0);

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (canSplit(nums, k, mid)) hi = mid;
        else lo = mid + 1;
    }

    return lo;
}
    Insight: "Minimize the maximum" and "maximize the minimum" are both search-on-answer problems. The direction of the binary search flips (minimize the max = find the smallest feasible; maximize the min = find the largest feasible).

    Rotated Sorted Array

    A rotated sorted array is a sorted array that has been cycled: some suffix has been moved to the front. For example, [4, 5, 6, 7, 0, 1, 2] is [0, 1, 2, 4, 5, 6, 7] rotated at index 4. The array is no longer globally sorted, but each "half" around the rotation point is individually sorted.

    Finding the Pivot (Minimum Element)

    The pivot is the index of the smallest element. In a rotated sorted array, it is the point where the value drops. Finding the pivot is a binary search problem: at each step, check which half is sorted and which half contains the drop.

    Find Minimum in Rotated Sorted Array

    int findMin(vector<int>& nums) {
    int lo = 0, hi = (int)nums.size() - 1;

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (nums[mid] > nums[hi]) {
            // mid is in the left (larger) portion
            // pivot is in (mid, hi]
            lo = mid + 1;
        } else {
            // mid is in the right (smaller) portion
            // pivot is in [lo, mid]
            hi = mid;
        }
    }

    return nums[lo];  // lo == hi == pivot index
}

    Why compare with nums[hi]?

    • If nums[mid] > nums[hi], the rotation break must be between mid and hi. The left portion is sorted but contains larger values. So lo = mid + 1.
    • If nums[mid] <= nums[hi], the right portion from mid to hi is sorted and mid could be the minimum. So hi = mid.

    Complexity: $O(\log n)$ time, $O(1)$ space.

    Searching in a Rotated Sorted Array

    Once you know the structure, searching becomes possible in a single pass. The key observation: when you pick mid, at least one of the two halves (lo to mid, or mid to hi) must be sorted. Check that sorted half to decide which side the target is on.

    Search in Rotated Sorted Array

    int search(vector<int>& nums, int target) {
    int lo = 0, hi = (int)nums.size() - 1;

    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;

        if (nums[mid] == target) return mid;

        if (nums[lo] <= nums[mid]) {
            // Left half [lo..mid] is sorted
            if (target >= nums[lo] && target < nums[mid]) {
                hi = mid - 1;
            } else {
                lo = mid + 1;
            }
        } else {
            // Right half [mid..hi] is sorted
            if (target > nums[mid] && target <= nums[hi]) {
                lo = mid + 1;
            } else {
                hi = mid - 1;
            }
        }
    }

    return -1;
}

    Step-by-step logic

    1. Compute mid. If it is the target, return.
    2. Check if the left half [lo..mid] is sorted by comparing nums[lo] with nums[mid].
    3. If the left half is sorted, check if the target falls within [nums[lo], nums[mid]). If yes, search left. Otherwise, search right.
    4. If the left half is not sorted, the right half must be sorted. Apply the symmetric check.

    Handling duplicates

    If the array can contain duplicates, the worst case degrades to $O(n)$. When nums[lo] == nums[mid] == nums[hi], you cannot determine which half is sorted. The standard fix is to shrink from both ends:

    // When nums[lo] == nums[mid] == nums[hi]:
lo++;
hi--;
// Then continue the loop.

    This handles the degenerate case but means worst-case $O(n)$ for arrays like [1, 1, 1, 1, 0, 1, 1].

    Peak Element

    A peak element is an element that is strictly greater than its neighbors. In an array where nums[-1] = nums[n] = negative infinity, at least one peak always exists.

    Why binary search works: You do not need the array to be sorted. You just need the guarantee that going "uphill" always leads to a peak. If nums[mid] < nums[mid + 1], there must be a peak somewhere to the right (the values are increasing, and eventually they must decrease or we hit the boundary). This gives a monotonic property to binary search on.

    Find Peak Element

    int findPeakElement(vector<int>& nums) {
    int lo = 0, hi = (int)nums.size() - 1;

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (nums[mid] < nums[mid + 1]) {
            // Ascending: peak is to the right
            lo = mid + 1;
        } else {
            // Descending or equal: peak is at mid or to the left
            hi = mid;
        }
    }

    return lo;  // lo == hi == a peak index
}

    Why this is correct

    • If nums[mid] < nums[mid + 1], the slope is going up. A peak must exist in [mid + 1, hi] because either the values keep increasing until hitting the boundary (which is a peak since nums[n] = -inf) or they decrease at some point (creating a peak).
    • If nums[mid] >= nums[mid + 1], the slope is going down or flat. A peak must exist in [lo, mid] by the same argument in the other direction.

    The key insight: mid + 1 is always a valid index because the loop condition lo < hi guarantees mid < hi.

    Peak element visual

    The array [1, 3, 7, 6, 2, 5, 4] has peaks at index 2 (value 7) and index 5 (value 5). Binary search will find one of them.

    Peak element walkthrough

    lo = 0, hi = 6, mid = 3

    nums[3] = 6, nums[4] = 2. Since 6 > 2, the slope is descending. A peak is at mid or to the left. Set hi = 3.

    lo = 0, hi = 3, mid = 1

    nums[1] = 3, nums[2] = 7. Since 3 < 7, the slope is ascending. A peak is to the right. Set lo = 2.

    lo = 2, hi = 3, mid = 2

    nums[2] = 7, nums[3] = 6. Since 7 > 6, descending. Set hi = 2.

    Now lo == hi == 2. Peak found at index 2 (value 7).

    Problem Recognition Guide

    How do you recognize when a problem uses one of these advanced binary search patterns?

    Pattern Clues in the problem What you binary search on
    Search on answer "Minimize the maximum", "maximize the minimum", find the optimal capacity/speed/threshold The answer value itself
    Rotated array "Sorted and rotated", "circular sorted", array with exactly one break in sorted order Array indices; check which half is sorted
    Peak element "Find a local maximum", "any peak", guaranteed boundary conditions (edges are negative infinity) Array indices; follow the ascending slope

    Key Takeaways

    • Binary search does not require a sorted array. It requires a monotonic predicate.
    • Search on answer: define feasible(x), identify the range, binary search for the boundary. Works for "minimize the maximum" and "maximize the minimum".
    • Rotated sorted array: one half is always sorted. Use that sorted half to determine which side the target is on.
    • Compare with nums[hi] (not nums[lo]) for finding the pivot. Compare nums[lo] with nums[mid] for determining which half is sorted when searching.
    • Peak element: follow the ascending slope. The existence of a peak is guaranteed when boundaries are negative infinity.
    • Duplicates in rotated arrays can force $O(n)$ worst case. Handle it by shrinking from both ends.
