    Binary Search: Competition Patterns

    March 2026 · 40 min read

    This post covers five binary search patterns that appear frequently in competitive programming and hard interview problems. Each one looks different on the surface, but they all share the same core idea: identify a monotonic property and binary search on it. We cover binary search on real numbers (continuous domains), aggressive cows (maximize the minimum), painters partition (minimize the maximum with order constraints), kth smallest in a sorted matrix, and the median of two sorted arrays.

    Binary Search on Real Numbers

    All the binary search we have seen so far operates on integers. But binary search works on continuous domains too. When the answer is a real number, replace the termination condition with a precision check: stop when hi - lo < epsilon, or run a fixed number of iterations (typically 100 is enough for double precision).

    Real-Valued Binary Search Template

    // Find x such that f(x) = target, where f is monotonic.
// Example: square root of n.
double binarySearchReal(double lo, double hi, int iterations) {
    for (int i = 0; i < iterations; i++) {
        double mid = lo + (hi - lo) / 2.0;

        if (feasible(mid)) {
            hi = mid;
        } else {
            lo = mid;  // no +1 for reals!
        }
    }

    return lo;  // or hi; they are within epsilon of each other
}

    Why fixed iterations instead of epsilon?

    Using while (hi - lo > eps) can be dangerous. If lo and hi are very large (say $10^{18}$), floating-point precision means hi - lo might never get smaller than eps. A fixed number of iterations (100) always converges: after 100 iterations, the interval shrinks by a factor of $2^{100} \approx 10^{30}$, which is more precision than a 64-bit double can represent.

    Example: Square root

    double mySqrt(double n) {
    double lo = 0, hi = max(1.0, n);

    for (int i = 0; i < 100; i++) {
        double mid = lo + (hi - lo) / 2.0;
        if (mid * mid >= n) {
            hi = mid;
        } else {
            lo = mid;
        }
    }

    return lo;
}

    Example: Nth root

    double nthRoot(double x, int n) {
    double lo = 0, hi = max(1.0, x);

    for (int i = 0; i < 100; i++) {
        double mid = lo + (hi - lo) / 2.0;
        if (pow(mid, n) >= x) {
            hi = mid;
        } else {
            lo = mid;
        }
    }

    return lo;
}
    Caution: For real-valued binary search, never use lo = mid + 1 or hi = mid - 1. Adding or subtracting 1 from a double makes no sense for precision. Just use lo = mid and hi = mid. The halving at each step guarantees convergence.

    Aggressive Cows

    Problem Statement

    Medium/Hard

    You have $n$ stalls at positions $x_1, x_2, \ldots, x_n$ and $c$ cows. Place the cows in stalls such that the minimum distance between any two cows is maximized. Return this maximum possible minimum distance.

    This is the classic "maximize the minimum" binary search problem. It is the dual of the "minimize the maximum" pattern from the advanced post.

    Approach

    Binary search on the answer: the minimum distance $d$. For a given $d$, check if you can place all $c$ cows such that every pair is at least $d$ apart. This predicate is monotonic: if placing cows with minimum gap $d$ is possible, then placing them with a smaller gap $d' < d$ is also possible.

    bool canPlace(vector<int>& stalls, int cows, int minDist) {
    int count = 1;
    int lastPos = stalls[0];

    for (int i = 1; i < (int)stalls.size(); i++) {
        if (stalls[i] - lastPos >= minDist) {
            count++;
            lastPos = stalls[i];
            if (count == cows) return true;
        }
    }

    return false;
}

int aggressiveCows(vector<int>& stalls, int cows) {
    sort(stalls.begin(), stalls.end());

    int lo = 1;
    int hi = stalls.back() - stalls.front();

    while (lo < hi) {
        int mid = lo + (hi - lo + 1) / 2;  // round up!

        if (canPlace(stalls, cows, mid)) {
            lo = mid;       // mid works, try larger
        } else {
            hi = mid - 1;   // mid too large
        }
    }

    return lo;
}

    Why round up the mid?

    We are maximizing, so the structure is different from the lower bound template. When canPlace succeeds, we move lo = mid (not mid + 1, since mid might be the answer). To prevent an infinite loop when lo + 1 == hi, we use mid = lo + (hi - lo + 1) / 2 which rounds up. This ensures mid > lo always.

    Rule of thumb: When you write lo = mid, use mid = lo + (hi - lo + 1) / 2 (round up). When you write hi = mid, use mid = lo + (hi - lo) / 2 (round down). This prevents infinite loops.

    Complexity: $O(n \log D)$ where $D$ is the range of stall positions. The greedy check is $O(n)$ per binary search iteration.

    Painters Partition

    Problem Statement

    Hard

    You have $n$ boards of lengths $a_1, a_2, \ldots, a_n$ and $k$ painters. Each painter paints consecutive boards. All painters paint at the same speed (1 unit per unit time). Find the minimum time to paint all boards. This is the time taken by the painter who paints the most.

    This is "minimize the maximum subarray sum" with $k$ partitions. It is structurally identical to the split array problem from the advanced post, but worded differently to test whether you recognize the pattern.

    The connection

    • Boards = array elements. Painters = number of partitions. Time = maximum subarray sum.
    • Binary search on the answer (maximum time allowed per painter). The predicate: "Can $k$ painters finish if each paints at most $T$ units?"
    bool canFinish(vector<int>& boards, int k, long long maxTime) {
    int painters = 1;
    long long current = 0;

    for (int b : boards) {
        if (current + b > maxTime) {
            painters++;
            current = 0;
        }
        current += b;
    }

    return painters <= k;
}

long long paintersPartition(vector<int>& boards, int k) {
    long long lo = *max_element(boards.begin(), boards.end());
    long long hi = accumulate(boards.begin(), boards.end(), 0LL);

    while (lo < hi) {
        long long mid = lo + (hi - lo) / 2;
        if (canFinish(boards, k, mid)) {
            hi = mid;
        } else {
            lo = mid + 1;
        }
    }

    return lo;
}

    Complexity: $O(n \log S)$ where $S$ is the sum of board lengths.

    Pattern family: Capacity to Ship Packages, Split Array Largest Sum, Painters Partition, Allocate Minimum Pages, and Aggressive Cows are all the same problem with different stories. Learn to recognize the structure: you have items, you have groups, you are optimizing the worst-case group metric.

    Kth Smallest Element in a Sorted Matrix

    Problem Statement

    Medium

    Given an $n \times n$ matrix where each row and each column is sorted in ascending order, find the $k$th smallest element.

    You might think to flatten the matrix and sort it, but that gives $O(n^2 \log n)$. Binary search gives $O(n \log(\text{max} - \text{min}))$.

    Approach: binary search on the value

    The answer is some value in the range [matrix[0][0], matrix[n-1][n-1]]. For a candidate value $x$, count how many elements in the matrix are less than or equal to $x$. If the count is at least $k$, then $x$ is large enough (or exactly right). If the count is less than $k$, then $x$ is too small.

    Counting efficiently

    Start from the bottom-left corner. At position $(r, c)$:

    • If matrix[r][c] <= x, all elements above in column $c$ are also $\leq x$. Add $r + 1$ to the count and move right.
    • If matrix[r][c] > x, move up.

    This traversal visits at most $2n$ cells, so the count is $O(n)$.

    int countLessOrEqual(vector<vector<int>>& matrix, int x) {
    int n = matrix.size();
    int count = 0;
    int r = n - 1, c = 0;  // start bottom-left

    while (r >= 0 && c < n) {
        if (matrix[r][c] <= x) {
            count += r + 1;  // entire column above is also <= x
            c++;
        } else {
            r--;
        }
    }

    return count;
}

int kthSmallest(vector<vector<int>>& matrix, int k) {
    int n = matrix.size();
    int lo = matrix[0][0];
    int hi = matrix[n - 1][n - 1];

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (countLessOrEqual(matrix, mid) >= k) {
            hi = mid;
        } else {
            lo = mid + 1;
        }
    }

    return lo;  // guaranteed to be an actual matrix element
}

    Why does this return an actual element?

    The binary search converges to the smallest value $x$ such that at least $k$ elements are $\leq x$. Since we binary search on integers and the predicate jumps from false to true at an actual matrix value, lo converges to a value present in the matrix.

    Complexity: $O(n \log(\text{max} - \text{min}))$. Each iteration counts in $O(n)$, and the range has at most $O(\log V)$ binary search steps where $V$ is the value range.

    Median of Two Sorted Arrays

    Problem Statement

    Hard

    Given two sorted arrays nums1 (length $m$) and nums2 (length $n$), find the median of the combined sorted array in $O(\log(\min(m, n)))$ time.

    This is one of the hardest binary search problems. The key idea: instead of merging the arrays, binary search for the correct partition that divides the combined elements into two equal halves.

    The partition approach

    We want to split the combined $m + n$ elements into a left half and a right half of equal size (or differing by one). If we choose $i$ elements from nums1 and $j$ elements from nums2 for the left half, then $i + j = \lfloor(m + n + 1) / 2\rfloor$. Once we fix $i$, we get $j$ automatically.

    The condition for a valid partition

    A partition at (i, j) is valid when all elements in the left half are less than or equal to all elements in the right half. Since both arrays are sorted, this reduces to two cross-checks:

    • nums1[i - 1] <= nums2[j] (left end of nums1 does not exceed right start of nums2)
    • nums2[j - 1] <= nums1[i] (left end of nums2 does not exceed right start of nums1)
    double findMedianSortedArrays(vector<int>& nums1, vector<int>& nums2) {
    // Ensure nums1 is shorter for O(log(min(m,n)))
    if (nums1.size() > nums2.size()) swap(nums1, nums2);

    int m = nums1.size(), n = nums2.size();
    int lo = 0, hi = m;
    int half = (m + n + 1) / 2;

    while (lo <= hi) {
        int i = lo + (hi - lo) / 2;  // elements from nums1
        int j = half - i;             // elements from nums2

        int left1  = (i > 0) ? nums1[i - 1] : INT_MIN;
        int right1 = (i < m) ? nums1[i]     : INT_MAX;
        int left2  = (j > 0) ? nums2[j - 1] : INT_MIN;
        int right2 = (j < n) ? nums2[j]     : INT_MAX;

        if (left1 <= right2 && left2 <= right1) {
            // Valid partition found
            if ((m + n) % 2 == 1) {
                return max(left1, left2);
            }
            return (max(left1, left2) + min(right1, right2)) / 2.0;
        } else if (left1 > right2) {
            hi = i - 1;  // too many from nums1
        } else {
            lo = i + 1;  // too few from nums1
        }
    }

    return 0;  // unreachable for valid input
}

    Walking through the logic

    Setup

    Ensure nums1 is the shorter array. We binary search on $i$ (how many elements to take from nums1), ranging from 0 to $m$. The corresponding $j$ from nums2 is half - i.

    Validate the partition

    Compare the four boundary elements: left1 (last element in nums1's left half), right1 (first element in nums1's right half), left2, right2. If left1 <= right2 and left2 <= right1, we have a valid partition.

    Adjust the search

    If left1 > right2, we took too many elements from nums1. Move hi left. If left2 > right1, we took too few from nums1. Move lo right.

    Step-through animation

    Median of Two Sorted Arrays

    Complexity: $O(\log(\min(m, n)))$ time, $O(1)$ space. This is optimal since we only binary search on the shorter array.

    Decision Framework

    When you see a problem that might use binary search, work through these questions:

    Question If yes
    Is the input sorted? Classic binary search (exact match, lower/upper bound)
    Is the input sorted but rotated? Rotated array search
    Does the problem ask to minimize a maximum or maximize a minimum? Binary search on the answer
    Is there a monotonic predicate on a numeric range? Binary search on the answer (general)
    Does the answer need real-number precision? Binary search on reals with fixed iterations
    Is the problem about the kth element in a structured collection? Binary search on value + efficient counting
    Does the problem involve two sorted sequences? Binary search on partition

    Complexity cheat sheet

    Problem Binary search on Check cost Total
    Standard search Index [0, n) $O(1)$ $O(\log n)$
    Ship packages / split array Answer [max, sum] $O(n)$ $O(n \log S)$
    Aggressive cows Answer [1, maxGap] $O(n)$ $O(n \log D)$
    Kth in sorted matrix Value [min, max] $O(n)$ $O(n \log V)$
    Median of two arrays Partition [0, m] $O(1)$ $O(\log \min(m,n))$

    Key Takeaways

    • Binary search on real numbers: use a fixed number of iterations (100) instead of epsilon comparison. Never use mid + 1 or mid - 1.
    • Aggressive cows: "maximize the minimum" = binary search on the answer with a greedy feasibility check. Use mid = lo + (hi - lo + 1) / 2 to avoid infinite loops when lo = mid.
    • Painters partition: structurally identical to "minimize the maximum subarray sum". The predicate is a greedy scan that counts partitions.
    • Kth smallest in sorted matrix: binary search on the value, not the index. Count elements less than or equal to candidate using the staircase traversal in $O(n)$.
    • Median of two sorted arrays: binary search on the partition of the shorter array. Check cross-boundary conditions to validate.
    • Every competition binary search follows the same skeleton: define the search space, define the monotonic predicate, and run lower bound (or upper bound).
