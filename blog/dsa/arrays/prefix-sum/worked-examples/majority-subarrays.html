    Counting Majority-Element Subarrays in O(n)

    June 2026 · 22 min read

    This is the first worked example in the Prefix Sum series, and it is a small masterpiece of a problem because it chains three ideas the rest of the series builds toward: a sign transform that turns a counting condition into a sum, the subarray-to-prefix-pair reduction, and a special ±1 structure that collapses an O(n log n) counting problem all the way down to O(n). The task (LeetCode 3737) is to count the subarrays in which a given target is the majority element.

    The Problem

    Given an integer array nums and an integer target, return how many subarrays of nums have target as their majority element. The majority element of a subarray is the value that appears strictly more than half of the time in that subarray.

    Examples. nums = [1,2,2,3], target = 2 gives 5 (the subarrays [2], [2], [2,2], [1,2,2], [2,2,3]). nums = [1,1,1,1], target = 1 gives 10 (every subarray). nums = [1,2,3], target = 4 gives 0. Constraints are small here (n ≤ 1000), so even O(n²) passes, which makes the O(n) solution a pure exercise in elegance.

    The ±1 Transform: Majority Becomes Sum > 0

    The word "majority" hides a clean inequality. Inside any subarray, let c be the count of target and o be the count of everything else, so the length is c + o. target is the majority exactly when

    c > (c + o) / 2   ⇔   2c > c + o   ⇔   c > o   ⇔   c - o > 0.

    So the only thing that matters is the difference between how many targets and non-targets a subarray contains. That difference is a sum if each element is mapped to a weight:

    w[i] = +1  if nums[i] == target
w[i] = -1  if nums[i] != target

    Then c - o over a subarray is just the sum of w over that subarray. The whole problem becomes: count the subarrays of w whose sum is strictly positive. The actual values in nums no longer matter; only the +1 / -1 pattern does.

    From Subarray Sum to a Prefix-Pair Inequality

    This is the move the Prefix Sum series is named for. Define the prefix-sum array of the weights:

    P[0] = 0
P[k] = w[0] + w[1] + ... + w[k-1]     (k = 1..n)

    The sum of the subarray nums[l..r] equals P[r+1] - P[l]. Requiring that sum to be positive is the same as requiring

    P[r+1] - P[l] > 0   ⇔   P[l] < P[r+1].

    Every subarray corresponds to a pair of prefix indices (l, r+1) with l < r+1. So the answer is exactly the number of index pairs (i, j) with i < j and P[i] < P[j]. The problem has turned into the classic count-ordered-pairs pattern (the mirror image of counting inversions). The animation below performs the transform on [1,2,2,3] and steps through every subarray, confirming the count of 5.

    ▶ Majority Subarray = Positive-Sum Subarray

    For nums = [1,2,2,3], target = 2: each element maps to +1 (target, green) or -1. Stepping through every subarray, the balance c - o = P[r+1] - P[l] decides whether target is the majority.

    Brute Force: O(n²)

    Counting positive-sum subarrays directly is a textbook double loop. Fix the left end, extend the right end, and keep a running balance, incrementing the answer whenever it goes positive. No prefix array is even needed for this version.

    long long brute(vector<int>& nums, int target) {
    int n = nums.size();
    long long ans = 0;
    for (int l = 0; l < n; ++l) {
        int balance = 0;                 // (#target) - (#non-target) in nums[l..r]
        for (int r = l; r < n; ++r) {
            balance += (nums[r] == target) ? 1 : -1;
            if (balance > 0) ++ans;      // target is the majority of nums[l..r]
        }
    }
    return ans;
}

    With n ≤ 1000 this is roughly a million operations and passes comfortably. The interesting question is whether the structure of the problem allows something faster, and it does.

    The ±1 Gift: an O(n) Sweep

    The general problem "count pairs i < j with P[i] < P[j]" needs a Fenwick tree or a merge sort and costs O(n log n) (see the Fenwick-tree treatment of this exact pattern). But here the prefix sums are not arbitrary: because every weight is +1 or -1, consecutive prefix sums differ by exactly one. That single fact removes the logarithm.

    Sweep the prefix sums left to right while maintaining two running quantities against the multiset of prefix sums seen so far:

    • freq[v] — how many already-seen prefix sums equal the value v.
    • cntLess — how many already-seen prefix sums are strictly less than the current prefix value cur.

    When the next weight moves cur by one step, cntLess can be patched in O(1) instead of recomputed, because only one value enters or leaves the "strictly less than" boundary:

    step +1 (cur -> cur+1):   values < cur+1  =  values < cur  +  freq[cur]
                          so  cntLess += freq[cur]

step -1 (cur -> cur-1):   values < cur-1  =  values < cur  -  freq[cur-1]
                          so  cntLess -= freq[cur-1]

    After the patch, cntLess is precisely the number of earlier prefix sums below the new cur, which is the number of valid left endpoints for a subarray ending here. Adding cntLess to the answer at every step, then inserting the new prefix sum, solves the whole thing in one pass. Because prefix sums range over [-n, n], the frequency array needs 2n + 1 buckets and an offset of n so that a value of -n lands at index 0. The animation traces the sweep on the same example, watching cntLess and the answer build up to 5.

    ▶ The O(n) Sweep: Patching cntLess in Place

    Prefix sums for [1,2,2,3], target = 2 form a ±1 walk: 0, -1, 0, 1, 0. Each step moves cur by one and patches cntLess using a single frequency bucket, then adds it to the answer.

    Code being executed (the highlighted line runs on this step):

    freq[idx(0)] = 1; // P[0] = 0 already seen
    cur = 0, cntLess = 0, ans = 0;
    for (int x : nums) {
    if (x == target) { // weight +1
    cntLess += freq[idx(cur)];
    ++cur;
    } else { // weight -1
    cntLess -= freq[idx(cur - 1)];
    --cur;
    }
    ans += cntLess;
    ++freq[idx(cur)];
    }
    return ans;

    The Code

    The full O(n) solution is barely longer than the brute force. It was checked against the brute force on a couple of thousand random arrays and matches every time.

    long long countMajoritySubarrays(vector<int>& nums, int target) {
    int n = nums.size();
    vector<long long> freq(2 * n + 1, 0);     // prefix sums live in [-n, n]
    auto idx = [&](int v) { return v + n; };  // offset so -n maps to 0

    freq[idx(0)] = 1;        // P[0] = 0 is already seen
    int cur = 0;             // current prefix sum
    long long cntLess = 0;   // # earlier prefix sums strictly < cur
    long long ans = 0;

    for (int x : nums) {
        if (x == target) {              // weight +1
            cntLess += freq[idx(cur)];
            ++cur;
        } else {                        // weight -1
            cntLess -= freq[idx(cur - 1)];
            --cur;
        }
        ans += cntLess;                 // valid left endpoints for this right end
        ++freq[idx(cur)];               // insert the new prefix sum
    }
    return ans;
}

    Each iteration touches one bucket and updates two counters, so the loop is genuinely linear, with no hidden logarithmic factor.

    Complexity

    ApproachTimeSpaceIdea
    Brute forceO(n²)O(1)Every subarray, running balance.
    Fenwick / merge sortO(n log n)O(n)Generic count of pairs P[i] < P[j].
    ±1 sweepO(n)O(n)Adjacent prefix sums differ by one; patch cntLess in O(1).

    The frequency array uses 2n + 1 buckets, so the space is O(n). That is the one tradeoff for dropping the logarithm: a small integer domain (here [-n, n]) is what makes bucket counting legal in the first place. Without the bounded ±1 values, prefix sums could be arbitrary and a frequency array would be impossible.

    The General Pattern, and When the Logarithm Comes Back

    It is worth being precise about exactly which gift the ±1 structure gave, because the surrounding patterns appear constantly:

    • Count pairs P[i] < P[j] for arbitrary values. This is the count-ordered-pairs (equivalently, count-inversions) problem. With unbounded prefix sums it needs a Fenwick tree / BIT over compressed values or a merge sort, both O(n log n). The Fenwick-tree post solves this very reduction for the "positive-sum subarray" case. The ±1 steps are exactly what let this problem skip that structure.
    • Count subarrays with sum equal to (or at most) k. The same prefix-sum-pair framing, but the comparison becomes P[j] - P[i] = k or ≤ k. Equality is handled by a hash map of prefix-sum frequencies in O(n); range conditions fall back to a Fenwick tree.
    • Boyer–Moore majority vote. Worth contrasting because it shares the word "majority" but solves a different question: it finds the single element that is a majority of the whole array in O(n) time and O(1) space. It does not count subarrays, and it cannot be swapped in here. The connection is only that both reframe "majority" as a +1 / -1 balance.

    Practice and Related

    • This problem (LeetCode 3737) — implement both the O(n²) and O(n) versions and confirm they agree.
    • Counting positive-sum subarrays with a Fenwick tree — the general O(n log n) version of the same reduction, for when the weights are not ±1.
    • Subarray Sum Equals K — the hash-map-of-prefix-sums cousin in the Prefix Sum patterns post.
    • Count of Range Sum / Reverse Pairs — the same count-ordered-pairs machinery with a banded or scaled comparison.

    The throughline is the one worth memorizing: a counting condition becomes a sign-weighted sum, a subarray becomes a pair of prefix indices, and the structure of the values decides whether the final count costs O(n), O(n log n), or something in between.
