    Merge Intervals Overview

    March 2026 · 20 min read

    What Are Intervals?

    An interval is a pair of numbers $[start, end]$ that represents a continuous range. You encounter intervals everywhere: a meeting from 9:00 to 10:30, a reservation from day 5 to day 8, a highlighted region on a timeline from position 2 to position 6.

    In coding problems, an interval is typically stored as a two-element array or a struct with two fields. The convention is that the start is always less than or equal to the end.

    // C++ representation
vector<vector<int>> intervals = {{1,3}, {2,6}, {8,10}, {15,18}};

# Python representation
intervals = [[1,3], [2,6], [8,10], [15,18]]

    On a number line, each interval is a horizontal bar spanning from its start to its end. Here are the four intervals above drawn on a number line:

    Notice that [1,3] and [2,6] overlap because 2 is between 1 and 3. The other two intervals are separate from each other and from the first two.

    When Do Two Intervals Overlap?

    Two intervals $[a, b]$ and $[c, d]$ overlap if and only if one starts before the other ends. Formally:

    $$a \le d \;\text{ and }\; c \le b$$

    Equivalently, they do not overlap if one ends before the other starts:

    $$b < c \;\text{ or }\; d < a$$

    When the intervals are sorted by start time, the check simplifies. If $a \le c$ (the first interval starts no later), then they overlap if and only if $c \le b$ (the second starts before the first ends).

    Sorted Overlap Rule: If intervals are sorted by start, the current interval overlaps the previous one if and only if current.start ≤ previous.end.

    The Merge Algorithm

    The classic merge intervals problem asks: given a list of intervals, merge all overlapping intervals and return the list of non-overlapping intervals that cover the same ranges.

    The algorithm has two phases:

    Sort by Start

    Sort all intervals by their start value. If two intervals have the same start, sort by end (though this is not strictly necessary for correctness). Sorting ensures that when we scan left to right, any interval that could overlap the current one comes immediately after it.

    Scan and Merge

    Initialize a result list with the first interval. For each subsequent interval:

    • If the current interval's start is less than or equal to the last merged interval's end, they overlap. Extend the last merged interval's end to max(last.end, current.end).
    • Otherwise, there is no overlap. Push the current interval onto the result list as a new entry.

    That is the entire algorithm. It works because sorting guarantees that once an interval does not overlap the last merged interval, no future interval can overlap it either (their starts are even larger).

    Walkthrough Example

    Let us trace through the algorithm on the input [[1,3],[2,6],[8,10],[15,18]].

    Sort

    The intervals are already sorted by start: [1,3], [2,6], [8,10], [15,18].

    Initialize result with [1,3]

    Result: [[1,3]]. The last merged interval is [1,3].

    Process [2,6]

    Compare: start 2 is less than or equal to last end 3, so they overlap. Merge by updating end to max(3,6) = 6. Result: [[1,6]].

    Process [8,10]

    Compare: start 8 is greater than last end 6, so no overlap. Push [8,10]. Result: [[1,6],[8,10]].

    Process [15,18]

    Compare: start 15 is greater than last end 10, so no overlap. Push [15,18]. Result: [[1,6],[8,10],[15,18]].

    Code

    C++

    vector<vector<int>> merge(vector<vector<int>>& intervals) {
    sort(intervals.begin(), intervals.end());
    vector<vector<int>> merged;

    for (auto& iv : intervals) {
        // If merged is empty or no overlap, push new interval
        if (merged.empty() || merged.back()[1] < iv[0]) {
            merged.push_back(iv);
        } else {
            // Overlap: extend the end of the last merged interval
            merged.back()[1] = max(merged.back()[1], iv[1]);
        }
    }
    return merged;
}

    Python

    def merge(intervals):
    intervals.sort()
    merged = []

    for start, end in intervals:
        # If merged is empty or no overlap, add new interval
        if not merged or merged[-1][1] < start:
            merged.append([start, end])
        else:
            # Overlap: extend the end of the last merged interval
            merged[-1][1] = max(merged[-1][1], end)

    return merged

    Java

    public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    List<int[]> merged = new ArrayList<>();

    for (int[] iv : intervals) {
        if (merged.isEmpty() || merged.get(merged.size() - 1)[1] < iv[0]) {
            merged.add(iv);
        } else {
            merged.get(merged.size() - 1)[1] =
                Math.max(merged.get(merged.size() - 1)[1], iv[1]);
        }
    }
    return merged.toArray(new int[merged.size()][]);
}

    Complexity Analysis

    AspectValueExplanation
    Time $O(n \log n)$ Sorting dominates. The scan is $O(n)$.
    Space $O(n)$ The merged list can contain up to $n$ intervals (if none overlap).
    Best case time $O(n \log n)$ Sorting is always required, even if all intervals merge into one.
    Why Not $O(n)$? Without sorting, you would need to compare every pair of intervals, which takes $O(n^2)$. Sorting is the key insight that makes the scan linear.

    Edge Cases

    • Single interval: Return it as-is. No merging needed.
    • All intervals overlap: Everything merges into one interval. Example: [[1,4],[2,5],[3,6]] becomes [[1,6]].
    • No intervals overlap: The output is the same as the sorted input.
    • Nested intervals: [2,3] inside [1,5]. The merge step handles this because max(5,3) = 5, so the larger interval is preserved.
    • Touching intervals: [1,3] and [3,5]. These are considered overlapping (3 equals 3) and merge to [1,5]. If touching intervals should not merge, change the condition to strict less-than.
    • Empty input: Return an empty list.

    Why Sorting Makes This Work

    Without sorting, you could have intervals in any order: [8,10], [1,3], [15,18], [2,6]. Trying to merge them in this order would be messy because [2,6] comes after [15,18] but should merge with [1,3].

    Sorting by start guarantees a critical property: if interval $i$ does not overlap the last merged interval, then no interval $j > i$ can overlap it either, because $j$ has a start at least as large as $i$'s start. This means one pass through the sorted array is sufficient.

    Common Variations

    The merge intervals pattern appears in many forms. The next post covers these patterns in detail, but here is a quick overview:

    ProblemKey Difference
    Insert Interval Insert one new interval into a sorted non-overlapping list, merging as needed.
    Non-overlapping Intervals Find the minimum number of intervals to remove so that none overlap.
    Meeting Rooms I Determine if a person can attend all meetings (check for any overlap).
    Meeting Rooms II Find the minimum number of conference rooms needed (maximum overlap depth).
    Interval Intersection Find all intervals common to two sorted lists.
    Minimum Arrows Find the minimum number of points that hit all intervals (similar to greedy scheduling).

    Key Takeaways

    • An interval $[start, end]$ is a continuous range on the number line.
    • Two sorted intervals overlap when the second starts before or at the first's end.
    • The merge algorithm is: sort by start, scan left to right, extend or push.
    • Time complexity is $O(n \log n)$ from sorting. The scan itself is $O(n)$.
    • Nested intervals are handled naturally because max preserves the larger end.
    • Sorting is the key insight. It turns a potentially $O(n^2)$ problem into $O(n \log n)$.
