    Stock Problems

    March 2026 · 30 min read

    The "Best Time to Buy and Sell Stock" family of problems is one of the most popular interview topics. Starting from a simple single-transaction problem, the series progressively builds into dynamic programming with state machines. Each variant adds one constraint, and understanding how that constraint changes the solution is the key to mastering this family.

    We will cover six variants, each building on the ideas from the previous one:

    1. Stock I – one transaction allowed
    2. Stock II – unlimited transactions
    3. Stock III – at most 2 transactions
    4. Stock IV – at most $k$ transactions
    5. With Cooldown – must wait one day after selling
    6. With Transaction Fee – pay a fee on each sale
    Common Setup: You are given an array prices[0..n-1] where prices[i] is the stock price on day $i$. You want to maximize profit. You must buy before you sell, and you cannot hold more than one share at a time.

    Visualizing Stock Prices

    Before we dive into algorithms, let us look at the price array we will use as our running example: [7, 1, 5, 3, 6, 4].

    The optimal single transaction is: buy on Day 1 at price 1, sell on Day 4 at price 6, for a profit of 5. But as we will see, different constraints change the optimal strategy entirely.

    Stock I: One Transaction

    LeetCode 121

    You are allowed to complete at most one transaction (one buy and one sell). Find the maximum profit.

    Track the Minimum Price So Far

    As you scan left to right, maintain the smallest price you have seen so far. At each day, the best profit you could get by selling on that day is prices[i] - min_price. Track the maximum of this value across all days.

    Why This Works

    For any selling day $i$, the best buying day is the day with the lowest price among days $0, 1, \ldots, i{-}1$. By maintaining a running minimum, we compute this in $O(1)$ per day instead of scanning backwards.

    Step-Through Animation

    Watch how min_price and max_profit update as we scan through [7, 1, 5, 3, 6, 4].

    min_price: - max_profit: - current profit: -
    Press Step to begin. Prices: [7, 1, 5, 3, 6, 4].

    C++ Code

    int maxProfit(vector<int>& prices) {
    int minPrice = INT_MAX;
    int maxProfit = 0;
    for (int price : prices) {
        minPrice = min(minPrice, price);
        maxProfit = max(maxProfit, price - minPrice);
    }
    return maxProfit;
}
    MetricValue
    Time$O(n)$
    Space$O(1)$

    Stock II: Unlimited Transactions

    LeetCode 122

    You can complete as many transactions as you like (buy and sell multiple times). You must sell before you buy again.

    Collect Every Upward Slope

    If the price goes up from day $i$ to day $i{+}1$, you should buy on day $i$ and sell on day $i{+}1$. Equivalently, add up every positive difference prices[i+1] - prices[i]. This collects all the profit from every upward movement.

    Why Greedy Works

    Any profitable multi-day transaction can be decomposed into consecutive single-day transactions. Holding from day 2 to day 5 with profit $p[5]-p[2]$ is the same as $(p[3]-p[2]) + (p[4]-p[3]) + (p[5]-p[4])$. The positive differences are exactly the days where the price increased. Negative differences contribute nothing, so we skip them.

    SVG: Collecting Upward Slopes

    Green segments are profitable (upward slopes). Dashed red segments are losses we skip.

    C++ Code

    int maxProfit(vector<int>& prices) {
    int profit = 0;
    for (int i = 1; i < prices.size(); i++) {
        if (prices[i] > prices[i - 1]) {
            profit += prices[i] - prices[i - 1];
        }
    }
    return profit;
}
    MetricValue
    Time$O(n)$
    Space$O(1)$

    The DP State Machine Framework

    Starting from Stock III onward, the problems are best understood through a state machine. On each day, you are in one of several states, and you can transition between them. This framework unifies all the remaining variants.

    Two Core States: At any point in time, you are either not holding stock (rest) or holding stock (hold). Transitions happen when you buy (rest to hold) or sell (hold to rest).

    State Machine Diagram

    The recurrences for any day $i$:

    $$\text{rest}[i] = \max(\text{rest}[i{-}1],\ \text{hold}[i{-}1] + \text{price}[i])$$ $$\text{hold}[i] = \max(\text{hold}[i{-}1],\ \text{rest}[i{-}1] - \text{price}[i])$$

    The first equation says: on day $i$ with no stock, I either did nothing (stayed in rest) or I sold today (was holding, gained price[i]). The second says: on day $i$ holding stock, I either did nothing (kept holding) or I bought today (was resting, paid price[i]).

    This framework lets us handle all variants by modifying either the transitions or the states.

    Stock III: At Most 2 Transactions

    LeetCode 123

    You can complete at most two buy-sell transactions. You must sell before buying again.

    Track States for Each Transaction

    We need four variables: the profit after the first buy, first sell, second buy, and second sell. Process each day and update these four values from right to left to avoid using the same day's updated value.

    The Four States

    • buy1: maximum profit after buying the first stock (negative or zero)
    • sell1: maximum profit after selling the first stock
    • buy2: maximum profit after buying the second stock (uses sell1 as capital)
    • sell2: maximum profit after selling the second stock

    State Transitions Visualization

    C++ Code

    int maxProfit(vector<int>& prices) {
    int buy1 = INT_MIN, sell1 = 0;
    int buy2 = INT_MIN, sell2 = 0;
    for (int price : prices) {
        buy1  = max(buy1,  -price);
        sell1 = max(sell1, buy1 + price);
        buy2  = max(buy2,  sell1 - price);
        sell2 = max(sell2, buy2 + price);
    }
    return sell2;
}
    Why update in order? Updating buy1, sell1, buy2, sell2 in sequence within the same loop iteration is safe because using the just-updated sell1 for buy2 on the same day is equivalent to buying and selling on the same day (zero profit), which affects nothing.
    MetricValue
    Time$O(n)$
    Space$O(1)$

    Stock IV: At Most k Transactions

    LeetCode 188

    Generalize Stock III: you can complete at most $k$ transactions.

    DP with Transaction Count

    Maintain arrays buy[j] and sell[j] for $j = 0, 1, \ldots, k$. For each day, update from $j=1$ to $j=k$. This is the natural extension of Stock III where we had $k=2$.

    Important Optimization

    If $k \ge n/2$ (where $n$ is the number of days), you can make as many transactions as profitable ones exist. This reduces to the Stock II problem (unlimited transactions). This optimization avoids creating huge arrays when $k$ is large.

    C++ Code

    int maxProfit(int k, vector<int>& prices) {
    int n = prices.size();
    if (n == 0) return 0;

    // If k is large enough, reduce to Stock II
    if (k >= n / 2) {
        int profit = 0;
        for (int i = 1; i < n; i++) {
            if (prices[i] > prices[i - 1])
                profit += prices[i] - prices[i - 1];
        }
        return profit;
    }

    // DP: buy[j] and sell[j] for j-th transaction
    vector<int> buy(k + 1, INT_MIN);
    vector<int> sell(k + 1, 0);

    for (int price : prices) {
        for (int j = 1; j <= k; j++) {
            buy[j]  = max(buy[j],  sell[j - 1] - price);
            sell[j] = max(sell[j], buy[j] + price);
        }
    }
    return sell[k];
}
    MetricValue
    Time$O(n \cdot k)$
    Space$O(k)$

    How Stock IV Generalizes the Family

    With Cooldown

    LeetCode 309

    Unlimited transactions, but after you sell on day $i$, you cannot buy on day $i{+}1$. You must wait one day (cooldown).

    Add a Cooldown State

    The state machine now has three states: rest (can buy), hold (has stock), and cooldown (just sold, must wait). After selling, you transition to cooldown. After cooldown, you transition to rest.

    State Machine with Cooldown

    Recurrences

    $$\text{rest}[i] = \max(\text{rest}[i{-}1],\ \text{cool}[i{-}1])$$ $$\text{hold}[i] = \max(\text{hold}[i{-}1],\ \text{rest}[i{-}1] - \text{price}[i])$$ $$\text{cool}[i] = \text{hold}[i{-}1] + \text{price}[i]$$

    The only difference from the basic framework: you can only buy from the rest state (not from cooldown), and selling takes you to the cooldown state instead of directly back to rest.

    C++ Code

    int maxProfit(vector<int>& prices) {
    int n = prices.size();
    if (n < 2) return 0;

    int rest = 0;           // not holding, free to buy
    int hold = -prices[0];  // holding stock
    int cool = 0;           // just sold, in cooldown

    for (int i = 1; i < n; i++) {
        int prevRest = rest;
        int prevHold = hold;
        int prevCool = cool;

        rest = max(prevRest, prevCool);
        hold = max(prevHold, prevRest - prices[i]);
        cool = prevHold + prices[i];
    }
    return max(rest, cool);
}
    MetricValue
    Time$O(n)$
    Space$O(1)$

    With Transaction Fee

    LeetCode 714

    Unlimited transactions, but you must pay a fee for each transaction. This means small upward slopes might not be worth trading on.

    Subtract the Fee on Each Sale

    The state machine is identical to Stock II, but we subtract the fee when we sell. This naturally filters out transactions where the profit does not exceed the fee.

    Why the Fee Changes Strategy

    In Stock II with prices [1, 3, 2, 8] and fee=2, the greedy "collect every uphill" approach gives $(3-1) + (8-2) = 8$. But each transaction costs 2, so the actual profit is $(3-1-2) + (8-2-2) = 4$. The optimal is a single transaction: buy at 1, sell at 8, profit $= 8-1-2 = 5$. The fee makes it better to hold through a small dip rather than sell and rebuy.

    Recurrences

    $$\text{rest}[i] = \max(\text{rest}[i{-}1],\ \text{hold}[i{-}1] + \text{price}[i] - \text{fee})$$ $$\text{hold}[i] = \max(\text{hold}[i{-}1],\ \text{rest}[i{-}1] - \text{price}[i])$$

    Identical to the basic state machine, except we subtract the fee when transitioning from hold to rest (selling).

    C++ Code

    int maxProfit(vector<int>& prices, int fee) {
    int n = prices.size();
    int rest = 0;
    int hold = -prices[0];

    for (int i = 1; i < n; i++) {
        int prevRest = rest;
        rest = max(rest, hold + prices[i] - fee);
        hold = max(hold, prevRest - prices[i]);
    }
    return rest;
}
    MetricValue
    Time$O(n)$
    Space$O(1)$

    Comparison of All Variants

    Problem Constraint Core Idea Time Space
    Stock I 1 transaction Track min price so far $O(n)$ $O(1)$
    Stock II Unlimited Sum all positive differences $O(n)$ $O(1)$
    Stock III At most 2 4 state variables (buy1, sell1, buy2, sell2) $O(n)$ $O(1)$
    Stock IV At most k DP with k buy/sell pairs $O(nk)$ $O(k)$
    Cooldown 1-day wait after sell 3-state machine (rest, hold, cool) $O(n)$ $O(1)$
    Fee Fee per transaction Subtract fee on sell transition $O(n)$ $O(1)$

    Interview Tips

    Recognizing the Variant

    When you see a stock problem in an interview, identify the constraint first:

    • How many transactions are allowed? (1, 2, k, or unlimited)
    • Is there a cooldown period?
    • Is there a transaction fee?

    Once you know the constraint, the state machine framework gives you the solution structure immediately.

    Common Mistakes

    • Stock I: Forgetting that you must buy before you sell (cannot sell on day 0).
    • Stock III: Initializing buy1 and buy2 to 0 instead of negative infinity (or INT_MIN).
    • Stock IV: Not handling the case where $k \ge n/2$ (leads to memory or time issues).
    • Cooldown: Using the updated rest value (instead of previous rest) when computing hold.
    • Fee: Subtracting the fee from both buy and sell (should only be one).

    Building Block Approach

    In an interview, start with Stock I. Show the interviewer you understand the simple case. Then if the problem is a harder variant, explain how Stock I generalizes into the state machine framework. This demonstrates structured thinking and makes the transition to the correct solution natural.

    Summary

    • Stock I is a simple one-pass tracking the minimum price seen so far.
    • Stock II collects every upward slope using a greedy approach.
    • From Stock III onward, the DP state machine framework is the cleanest approach: define states (rest, hold, and optionally cooldown), and transitions (buy, sell, wait).
    • Stock IV generalizes to $k$ transactions by maintaining $k$ buy/sell state pairs.
    • Cooldown adds a third state to enforce the one-day wait after selling.
    • Transaction fee modifies the sell transition by subtracting the fee, naturally filtering unprofitable trades.
    • The state machine diagram is the single most useful tool for these problems. Draw it first, then the recurrences write themselves.
