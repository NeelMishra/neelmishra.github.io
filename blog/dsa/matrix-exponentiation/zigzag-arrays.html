    ZigZag Arrays II: Counting Sequences with Matrix Exponentiation

    June 2026 · 28 min read

    The first post showed that a linear recurrence is secretly a matrix, so the $n$-th term can be found in $O(\log n)$. That was a warm-up. The real power of the technique shows up when the recurrence is not handed to you, and you have to discover it from a counting problem. LeetCode 3700, Number of ZigZag Arrays II, is a perfect example: it asks you to count sequences of length up to $10^9$, which is far too many to enumerate, yet the answer falls out of a fixed-size matrix raised to a huge power.

    The Problem

    You are given three integers $n$, $l$, and $r$. A ZigZag array of length $n$ satisfies all three rules:

    • Every element lies in the range $[l, r]$.
    • No two adjacent elements are equal.
    • No three consecutive elements are strictly increasing or strictly decreasing.

    Return the number of valid ZigZag arrays, modulo $10^9 + 7$. The constraints are what make this hard:

    $$3 \le n \le 10^9, \qquad 1 \le l < r \le 75.$$

    So $n$ is astronomically large, but the number of distinct values, $m = r - l + 1$, is at most $75$. That gap between a giant length and a tiny alphabet is the signature of a matrix exponentiation problem.

    Two worked examples. For $n = 3,\ l = 4,\ r = 5$ the answer is $2$: only $[4,5,4]$ and $[5,4,5]$. For $n = 3,\ l = 1,\ r = 3$ the answer is $10$. We will reproduce both with the animation below and again with the final code.

    The Key Insight: ZigZag Means Alternating Slopes

    The three rules look fussy, but together they say something very simple. Because no two neighbours are equal, every adjacent pair is either a strict up step ($a_i < a_{i+1}$) or a strict down step ($a_i > a_{i+1}$). Because no three in a row are monotonic, you can never take two ups in a row or two downs in a row. So the sequence of step directions has no two equal neighbours either: it must strictly alternate, up, down, up, down, or down, up, down, up.

    That is exactly a zigzag. The only freedom is the direction of the first step; after that, every step is forced to flip. The two starting choices give two mirror-image families, a fact we will exploit later.

    Step through every length-3 candidate over $[1,3]$ below. The animation marks each as ZigZag or not, and the valid count lands on $10$, matching the second example.

    ▶ What Counts as a ZigZag? (all length-3 arrays over [1,3])

    Each candidate $[a,b,c]$ is drawn as a slope. It is rejected for equal neighbours or for three-in-a-row monotonic, and accepted only when the slope flips. Valid arrays collect on the right.

    A DP on (Value, Direction)

    To grow a ZigZag array one element at a time, the only things that constrain the next element are the current last value and which direction the next step must take. The full prefix does not matter. That gives a compact state. Index the values $0,1,\dots,m-1$ (subtract $l$), and define, for an array of length $i$:

    • $\mathrm{up}_i[v]$ = number of ZigZag arrays of length $i$ that end at value $v$ whose last step was an up step. The next step from here must go down.
    • $\mathrm{down}_i[v]$ = number ending at value $v$ whose last step was a down step. The next step must go up.

    The base case is length $2$, the shortest array that already has a defined "last step." An array $[a, b]$ with $a < b$ is an up-ending array, and there are exactly as many of them ending at $v$ as there are values below $v$:

    $$\mathrm{up}_2[v] = \#\{a : a < v\} = v, \qquad \mathrm{down}_2[v] = \#\{a : a > v\} = (m-1) - v.$$

    The answer for length $n$ is the grand total over both directions and all ending values:

    $$\text{answer} = \sum_{v=0}^{m-1}\big(\mathrm{up}_n[v] + \mathrm{down}_n[v]\big).$$

    The Transition Is a Prefix Sum

    Now extend a length-$i$ array by one element. If the array currently ends with an up step at value $v$, the next element $w$ must be a down step, so $w < v$, and the new array ends with a down step at $w$. Summing over all the up-ending arrays that can drop to $w$:

    $$\mathrm{down}_{i+1}[w] = \sum_{v > w} \mathrm{up}_i[v] \quad(\text{a suffix sum of } \mathrm{up}_i).$$

    Symmetrically, a down-ending array at value $v$ must rise to some $w > v$, becoming an up-ending array at $w$:

    $$\mathrm{up}_{i+1}[w] = \sum_{v < w} \mathrm{down}_i[v] \quad(\text{a prefix sum of } \mathrm{down}_i).$$

    This is the crucial observation: each new entry is a fixed linear combination (a prefix or suffix sum) of the old entries. There are no products, no conditionals on the data, just sums with coefficients $0$ or $1$. A fixed linear map from one state vector to the next is precisely a matrix. Watch one full step run, computing every new entry from highlighted prefixes and suffixes.

    ▶ One ZigZag Step = Prefix and Suffix Sums

    With $m=4$ and the length-2 base state, each step index $w$ fills $\mathrm{down}[w]$ from the highlighted suffix of up and $\mathrm{up}[w]$ from the highlighted prefix of down. Every output is a fixed sum of inputs.

    From Recurrence to Matrix

    Stack the two halves into a single state vector of length $2m$:

    $$\mathbf{s}_i = \big[\,\underbrace{\mathrm{up}_i[0],\dots,\mathrm{up}_i[m-1]}_{m},\ \underbrace{\mathrm{down}_i[0],\dots,\mathrm{down}_i[m-1]}_{m}\,\big]^\top.$$

    The two prefix-sum rules define a single $2m \times 2m$ transition matrix $M$ with $\mathbf{s}_{i+1} = M\,\mathbf{s}_i$. Its entries are all $0$ or $1$, arranged in two triangular blocks: the up → down block (a suffix sum) and the down → up block (a prefix sum). The diagonal blocks are empty, because an up-ending array never extends to another up-ending array in one step.

    Because the same map applies at every length, unrolling from the base case at length $2$ gives the whole answer as a single matrix power:

    $$\mathbf{s}_n = M^{\,n-2}\,\mathbf{s}_2, \qquad \text{answer} = \sum_{i=0}^{2m-1} (\mathbf{s}_n)_i.$$

    The index $n$, which can be a billion, now appears only as an exponent. That is the entire trick, and it is the same move as in Part 1.

    Raising It to the Power n

    Computing $M^{\,n-2}$ by multiplying $M$ in a loop would cost $O(n)$ matrix multiplications, defeating the purpose. Instead use binary exponentiation: square the matrix and fold in the squares that line up with the $1$-bits of the exponent, in $O(\log n)$ multiplications. Part 1 derives and animates this in full; here we just reuse it.

    One matrix multiply on a $2m \times 2m$ matrix costs $O\big((2m)^3\big)$, and we do $O(\log n)$ of them, so the total is

    $$O\big((2m)^3 \log n\big) = O\big((r-l)^3 \log n\big).$$

    With $m \le 75$ and $\log n \le 30$, that is a few hundred million cheap modular multiply-adds in the worst case, comfortably fast.

    The Code

    The same multiply and matrixPower helpers from Part 1 carry over unchanged. The only new work is building $M$ and the base vector $\mathbf{s}_2$, then summing the result. This is a direct, runnable LeetCode solution.

    const long long MOD = 1e9 + 7;
typedef vector<vector<long long>> Matrix;

Matrix multiply(const Matrix& A, const Matrix& B) {
    int n = A.size(), p = B.size(), q = B[0].size();
    Matrix C(n, vector<long long>(q, 0));
    for (int i = 0; i < n; ++i)
        for (int k = 0; k < p; ++k)
            if (A[i][k])
                for (int j = 0; j < q; ++j)
                    C[i][j] = (C[i][j] + A[i][k] * B[k][j]) % MOD;
    return C;
}

Matrix identity(int n) {
    Matrix I(n, vector<long long>(n, 0));
    for (int i = 0; i < n; ++i) I[i][i] = 1;
    return I;
}

Matrix matrixPower(Matrix base, long long exp) {
    Matrix result = identity(base.size());
    while (exp > 0) {
        if (exp & 1) result = multiply(result, base);
        base = multiply(base, base);
        exp >>= 1;
    }
    return result;
}

class Solution {
public:
    int zigZagArrays(int n, int l, int r) {
        int m = r - l + 1;            // number of distinct values
        int S = 2 * m;                // state = up[0..m-1] then down[0..m-1]
        // index helpers: up(v) = v,  down(v) = m + v
        Matrix M(S, vector<long long>(S, 0));
        for (int w = 0; w < m; ++w) {
            for (int v = 0; v < w; ++v)     M[w][m + v] = 1;   // new up[w]   = sum_{v<w} down[v]
            for (int v = w + 1; v < m; ++v) M[m + w][v]   = 1;   // new down[w] = sum_{v>w} up[v]
        }
        Matrix s(S, vector<long long>(1, 0));                    // base: length-2 arrays
        for (int v = 0; v < m; ++v) {
            s[v][0]     = v;            // up[v]   = number of values below v
            s[m + v][0] = m - 1 - v;    // down[v] = number of values above v
        }
        Matrix sn = multiply(matrixPower(M, n - 2), s);         // state at length n
        long long ans = 0;
        for (int i = 0; i < S; ++i) ans = (ans + sn[i][0]) % MOD;
        return (int)ans;
    }
};

    Running it on the two examples returns $2$ and $10$, as promised. Validated against a brute-force enumerator on a few hundred small random cases, it matches every time.

    Halving the Matrix by Symmetry

    Reflecting every value $v \mapsto (m-1) - v$ turns an up step into a down step and vice versa, so it pairs each up-ending array with a unique down-ending one:

    $$\mathrm{up}_i[v] = \mathrm{down}_i[\,m-1-v\,].$$

    The two halves of the state carry identical information. Keep only $\mathrm{down}$, and the suffix sum over $\mathrm{up}$ rewrites as a prefix sum over $\mathrm{down}$:

    $$\mathrm{down}_{i+1}[w] = \sum_{v > w} \mathrm{up}_i[v] = \sum_{v > w}\mathrm{down}_i[\,m-1-v\,] = \sum_{u \,\le\, m-2-w} \mathrm{down}_i[u].$$

    That is an $m \times m$ transition, half the size, so each multiply is $8\times$ cheaper. Seeding a length-$1$ all-ones vector and applying it $n-1$ times gives $\mathrm{down}_n$; doubling the total accounts for the mirror half:

    class Solution {
public:
    int zigZagArrays(int n, int l, int r) {
        int m = r - l + 1;
        Matrix M(m, vector<long long>(m, 0));
        for (int w = 0; w < m; ++w)
            for (int u = 0; u + w <= m - 2; ++u)   // down[i+1][w] = sum_{u <= m-2-w} down[i][u]
                M[w][u] = 1;
        Matrix start(1, vector<long long>(m, 1));   // length-1 seed, all ones
        Matrix res = multiply(start, matrixPower(M, n - 1));
        long long ans = 0;
        for (int j = 0; j < m; ++j) ans = (ans + res[0][j]) % MOD;
        return (int)(ans * 2 % MOD);               // x2 for the mirrored half
    }
};

    This is the version that comfortably fits the worst-case constraints, at a clean $O\big((r-l)^3 \log n\big)$ time and $O\big((r-l)^2\big)$ space. The explicit two-block version above is easier to reason about; this one is what you submit.

    Complexity and Edge Cases

    • Time: $O\big((r-l)^3 \log n\big)$. The matrix is $m \times m$ (or $2m \times 2m$ unreduced), and binary exponentiation does $O(\log n)$ multiplies.
    • Space: $O\big((r-l)^2\big)$ for the matrices.
    • Modulus: every multiply-add reduces modulo $10^9 + 7$; use 64-bit accumulators so $A[i][k]\cdot B[k][j]$ does not overflow before the reduction.
    • Small $n$: the constraints guarantee $n \ge 3$, so $n - 2 \ge 1$ and the base length-$2$ state is always valid. If you ever adapt this to allow $n = 1$ or $n = 2$, handle those directly (the answers are $m$ and $m(m-1)$).
    • Single value: the constraint $l < r$ guarantees $m \ge 2$, so a non-trivial zigzag always exists; with $m = 1$ the answer would be $0$ for $n \ge 2$.
