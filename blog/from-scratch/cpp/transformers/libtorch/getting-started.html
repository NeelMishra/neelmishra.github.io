    Installing LibTorch and Your First Tensor Program

    From Scratch Series From Scratch · C++ · LibTorch Deep Dives

    What Is LibTorch?

    LibTorch is the official C++ distribution of PyTorch. It ships the same tensor library, autograd engine, and neural network modules that the Python package exposes, but as native C++ headers and shared libraries. There is no Python interpreter in the loop, so the same torch::Tensor operations run directly from a compiled binary.

    This post covers the three things needed before any of the later LibTorch deep dives make sense: downloading the prebuilt library, pointing a minimal CMake project at it, and compiling a first program that adds two tensors.

    What this guide assumes: a working C++ toolchain (GCC, Clang, or MSVC), cmake version 3.18 or newer, and an unzip utility. No prior LibTorch experience is required.

    Step 1: Download LibTorch

    The prebuilt archives are published on the PyTorch "Get Started" page at https://pytorch.org/get-started/locally/. The selector on that page builds a direct download link from a few choices:

    SelectorWhat to choose
    PyTorch BuildStable for production work; Preview (Nightly) only if a brand new feature is needed.
    Your OSLinux, Mac, or Windows.
    PackageLibTorch. This is the key choice that switches the page from the Python wheel to the C++ archive.
    LanguageC++ / Java.
    Compute PlatformCPU for a CPU-only build, or a CUDA version such as CUDA 12.1 for GPU support (Linux and Windows only).

    Once those are selected, the page shows one or more download links. A few platform-specific details are worth knowing:

    • Linux ABI variants: there are two archives, cxx11 ABI and Pre-cxx11 ABI. Pick cxx11 ABI for any modern GCC or Clang. The pre-cxx11 build exists only for old toolchains that still use the legacy std::string ABI.
    • Windows variants: separate Release and Debug archives are provided. The one chosen must match the configuration used to build the project; mixing them causes link or runtime errors.
    • macOS: the build is CPU only (with MPS acceleration on Apple Silicon). There is no CUDA option on Mac.

    On Linux the link can be fetched directly. The exact file name encodes the version and compute platform, so substitute the values shown on the page (the version number changes over time):

    # CPU-only build (Linux, cxx11 ABI) wget https://download.pytorch.org/libtorch/cpu/libtorch-cxx11-abi-shared-with-deps-2.5.1%2Bcpu.zip # CUDA 12.1 build (Linux, cxx11 ABI) wget https://download.pytorch.org/libtorch/cu121/libtorch-cxx11-abi-shared-with-deps-2.5.1%2Bcu121.zip
    CPU vs CUDA: the CUDA archive only works if a matching NVIDIA driver and CUDA runtime are present. When unsure, start with the CPU build. Tensor addition and most of the from-scratch examples in this series run identically on CPU, just slower for large tensors.

    Step 2: Extract and Inspect

    "Installing" LibTorch means unzipping the archive somewhere stable. There is no system installer; the extracted folder is self-contained.

    unzip libtorch-cxx11-abi-shared-with-deps-2.5.1+cpu.zip # This creates a folder named: libtorch/

    Move that folder to a permanent location, for example a home directory or a shared tools path:

    mv libtorch ~/libtorch

    Inside, the layout is what every later build step relies on:

    libtorch/ bin/ # helper executables include/ # headers: torch/torch.h, ATen/, c10/, ... lib/ # shared libraries: libtorch.so, libc10.so, ... share/cmake/Torch/TorchConfig.cmake # the file CMake's find_package(Torch) looks for

    The two directories that matter most are include/ (the headers the compiler reads) and lib/ (the shared libraries the linker and the running program need). The share/cmake/Torch/ directory is what makes the CMake integration in the next step effortless.

    Remember the root path. The path to use later is the top-level libtorch/ folder (for example ~/libtorch), not its include/ or lib/ subfolders. Pointing a build at the wrong level is the most common setup mistake.

    Step 3: A Minimal CMake Project

    LibTorch is built around CMake, so a tiny CMakeLists.txt is the cleanest way to compile against it. The project folder needs just two files:

    first-tensor/ CMakeLists.txt main.cpp

    Here is the complete CMakeLists.txt:

    cmake_minimum_required(VERSION 3.18) project(first_tensor LANGUAGES CXX) set(CMAKE_CXX_STANDARD 17) set(CMAKE_CXX_STANDARD_REQUIRED ON) find_package(Torch REQUIRED) set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} ${TORCH_CXX_FLAGS}") add_executable(first_tensor main.cpp) target_link_libraries(first_tensor PRIVATE "${TORCH_LIBRARIES}")
    Not covered here on purpose. This post keeps CMake to the minimum needed to build. For a line-by-line explanation of find_package, add_executable, target_link_libraries, CMAKE_PREFIX_PATH, and the LibTorch example in full, read the dedicated guide: CMake for C++ Projects.

    Step 4: Your First Program (Tensor Addition)

    The "hello world" of a tensor library is adding two vectors. A tensor is just an n-dimensional array, and a one-dimensional tensor is a vector. Create main.cpp:

    #include <torch/torch.h> #include <iostream> int main() { // Two 1-D tensors (vectors) of four elements each torch::Tensor a = torch::tensor({1.0, 2.0, 3.0, 4.0}); torch::Tensor b = torch::tensor({10.0, 20.0, 30.0, 40.0}); // Element-wise addition: c[i] = a[i] + b[i] torch::Tensor c = a + b; std::cout << "a = " << a << std::endl; std::cout << "b = " << b << std::endl; std::cout << "a + b = " << c << std::endl; return 0; }

    Two small but important details:

    • torch::tensor({1.0, 2.0, 3.0, 4.0}) builds a tensor from a brace-enclosed list. Floating-point literals produce a Float tensor; integer literals like {1, 2, 3, 4} would produce a Long tensor instead.
    • a + b is element-wise. Both tensors share the shape {4}, so the result also has shape {4}. If shapes differ, LibTorch applies broadcasting rules borrowed directly from NumPy and PyTorch.

    To confirm the build also sees the GPU (only meaningful with a CUDA archive), one extra line helps:

    std::cout << "CUDA available: " << torch::cuda::is_available() << std::endl;

    Step 5: Build and Run

    From inside the first-tensor/ folder, configure the project and tell CMake where LibTorch lives through CMAKE_PREFIX_PATH, then build and run:

    cmake -S . -B build -DCMAKE_PREFIX_PATH=~/libtorch cmake --build build ./build/first_tensor

    CMAKE_PREFIX_PATH points at the LibTorch root extracted in Step 2. That is how find_package(Torch REQUIRED) locates TorchConfig.cmake and fills in the include paths, libraries, and compiler flags automatically.

    The expected output prints each tensor with a small type footer that LibTorch appends:

    a = 1 2 3 4 [ CPUFloatType{4} ] b = 10 20 30 40 [ CPUFloatType{4} ] a + b = 11 22 33 44 [ CPUFloatType{4} ]

    The [ CPUFloatType{4} ] line confirms three facts at once: the tensor lives on the CPU, holds 32-bit floats, and has shape {4}. Seeing 11, 22, 33, 44 means the headers, libraries, and linker flags all lined up correctly.

    Runtime library errors on Linux? If the program builds but fails to start with a message about a missing libtorch.so or libc10.so, the loader cannot find the shared libraries at runtime. Point it at the lib/ folder: export LD_LIBRARY_PATH=~/libtorch/lib:$LD_LIBRARY_PATH, then run again. The full set of build and runtime path details is covered in the CMake guide.
