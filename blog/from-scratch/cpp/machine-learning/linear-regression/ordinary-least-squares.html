    Ordinary Least Squares in C++

    July 2026 · 24 min read

    Ordinary least squares (OLS) is the baseline linear model: it minimizes squared error and, for a well-posed problem, has a closed-form solution. This post derives that solution from the model definition, then builds it as a complete, dependency-free C++ program: input validation, a partial-pivot linear solver with singularity detection, the OLS fit itself, two direct extensions (weighted least squares and polynomial basis expansion), and the evaluation metrics used throughout the rest of the series. Every function below is shown exactly once, in the order it appears in the source file, so reading top to bottom and pasting the snippets in sequence reproduces the full program compiled at the end.

    The Model and the Design Matrix

    A linear model predicts a target as a weighted sum of features plus an intercept:

    $$\hat y = w_0 + w_1 x_1 + w_2 x_2 + \dots + w_p x_p$$

    To keep the intercept inside the same machinery as the other weights, every feature row is extended with a leading constant 1, turning the row into $\mathbf{x} = [1, x_1, \dots, x_p]$ and the prediction into a single dot product $\hat y = \mathbf{x}^\top \mathbf{w}$. Stacking $n$ such rows gives the design matrix $X \in \mathbb{R}^{n \times (p+1)}$, and stacking the targets gives $\mathbf{y} \in \mathbb{R}^n$. The whole dataset's predictions are then $\hat{\mathbf y} = X\mathbf{w}$, one matrix-vector product.

    This convention, an explicit leading-1 column instead of a separately tracked intercept term, is what lets one function handle both the intercept and the slopes with the same loop, and it is used consistently across every variant in this series.

    Linear in the parameters, not necessarily in the features. "Linear regression" means the prediction is linear in $\mathbf w$. The columns of $X$ can themselves be nonlinear transforms of the raw inputs, such as $x^2$ or $\log x$; the basis-expansion section below builds exactly that.

    Deriving the Normal Equations

    OLS chooses $\mathbf w$ to minimize the mean squared error over the training rows:

    $$J(\mathbf w) = \frac{1}{2n} \lVert \mathbf y - X\mathbf w \rVert_2^2 = \frac{1}{2n} \sum_{i=1}^n \left(y_i - \mathbf x_i^\top \mathbf w\right)^2$$

    The factor $\tfrac12$ is a convenience that cancels in the derivative. Differentiating with respect to the vector $\mathbf w$ and using the chain rule:

    $$\nabla J(\mathbf w) = -\frac{1}{n} X^\top (\mathbf y - X\mathbf w)$$

    $J$ is convex (it is a sum of squares), so any point where the gradient vanishes is a global minimum. Setting $\nabla J(\mathbf w) = 0$ and rearranging gives the normal equations:

    $$X^\top X\, \mathbf w = X^\top \mathbf y$$

    When $X^\top X$ is invertible, the closed-form solution is $\mathbf w = (X^\top X)^{-1} X^\top \mathbf y$. In code, the matrix is never inverted explicitly; a linear solver is used instead, both because it is cheaper and because it avoids amplifying rounding error further than necessary (see the numerical stability section below). Geometrically, $X\mathbf w$ is the orthogonal projection of $\mathbf y$ onto the column space of $X$, which is exactly what $X^\top(\mathbf y - X\mathbf w) = 0$ says: the residual is orthogonal to every column of $X$. The dedicated Ordinary Least Squares series develops the full statistical story, while Least Squares Geometry focuses on the projection view.

    Validating the Input

    A solver that silently accepts malformed input produces a number that looks like an answer but means nothing. Every fitting function in this file starts by checking that X and y agree in row count, that no row is a different length from the others (a ragged design matrix usually means a bug upstream), and that every value is finite, since a single NaN propagates through every subsequent sum:

    struct DataError : runtime_error {
    explicit DataError(const string& msg) : runtime_error(msg) {}
};

static void validate_dataset(const Matrix& X, const Vector& y) {
    if (X.empty()) throw DataError("X has zero rows");
    if (X.size() != y.size())
        throw DataError("row count mismatch: X has " + to_string(X.size()) +
                         " rows, y has " + to_string(y.size()));
    size_t d = X[0].size();
    if (d == 0) throw DataError("X rows have zero columns");
    for (size_t i = 0; i < X.size(); ++i) {
        if (X[i].size() != d)
            throw DataError("ragged row " + to_string(i) + ": expected " +
                             to_string(d) + " columns, got " + to_string(X[i].size()));
        for (double v : X[i])
            if (!isfinite(v)) throw DataError("non-finite feature at row " + to_string(i));
        if (!isfinite(y[i])) throw DataError("non-finite target at row " + to_string(i));
    }
}

    A fourth check, whether there are enough distinct rows to determine every parameter, is easiest to express once the design matrix size is known, so it lives inside fit_ols below rather than here.

    Building the Design Matrix

    Two small helpers build design matrices: one that only prepends the intercept column, and one that expands a single feature into a polynomial basis (used later in the basis-expansion section):

    // Prepend a leading 1.0 to every row for the intercept term.
static Matrix add_intercept(const Matrix& X) {
    Matrix D(X.size(), Vector(X[0].size() + 1, 1.0));
    for (size_t i = 0; i < X.size(); ++i)
        for (size_t j = 0; j < X[i].size(); ++j)
            D[i][j + 1] = X[i][j];
    return D;
}

// Expand a single feature column into [1, x, x^2, ..., x^degree].
// This is the standard way to fit curves with a linear-in-parameters model.
static Matrix polynomial_design(const Vector& x, int degree) {
    Matrix D(x.size(), Vector(degree + 1, 1.0));
    for (size_t i = 0; i < x.size(); ++i) {
        double p = 1.0;
        for (int j = 0; j <= degree; ++j) {
            D[i][j] = p;
            p *= x[i];
        }
    }
    return D;
}

    fit_ols and fit_wls below call add_intercept implicitly (by building the augmented row inline); polynomial_design is used explicitly in the worked example to fit a curve.

    A Partial-Pivot Linear Solver

    The normal equations are a small, dense, symmetric linear system: $d \times d$ where $d = p+1$ is the parameter count, independent of how many rows $n$ the dataset has. Gauss-Jordan elimination with partial pivoting solves it directly. At each column, the solver swaps in the row with the largest magnitude entry in that column before eliminating, which keeps the divisor as large as possible and controls rounding error. If the largest available pivot is still smaller than a relative tolerance, the matrix is treated as singular rather than solved with a near-zero divisor, which would blow up the solution silently:

    // Gaussian elimination with partial pivoting on an augmented [A | b] system.
// Throws if a column has no usable pivot above a relative tolerance, which
// signals a singular or near-singular normal-equation matrix (collinear
// or duplicated features, or fewer distinct rows than columns).
static Vector solve_linear(Matrix A, Vector b) {
    int n = (int)b.size();
    if ((int)A.size() != n) throw DataError("solve_linear: A is not square-compatible with b");

    // Scale for a relative singularity tolerance.
    double scale = 0.0;
    for (auto& row : A) for (double v : row) scale = max(scale, fabs(v));
    if (scale == 0.0) scale = 1.0;
    const double tol = 1e-10 * scale;

    for (int col = 0; col < n; ++col) {
        int piv = col;
        for (int r = col + 1; r < n; ++r)
            if (fabs(A[r][col]) > fabs(A[piv][col])) piv = r;
        if (fabs(A[piv][col]) < tol)
            throw DataError("solve_linear: matrix is singular or near-singular at column " +
                             to_string(col) + " (check for duplicated/collinear features)");
        swap(A[col], A[piv]);
        swap(b[col], b[piv]);
        double div = A[col][col];
        for (int c = col; c < n; ++c) A[col][c] /= div;
        b[col] /= div;
        for (int r = 0; r < n; ++r) if (r != col) {
            double m = A[r][col];
            if (m == 0.0) continue;
            for (int c = col; c < n; ++c) A[r][c] -= m * A[col][c];
            b[r] -= m * b[col];
        }
    }
    return b;
}

    The tolerance is scaled by the largest entry in the matrix rather than fixed at, say, 1e-10 outright, because the raw magnitude of $X^\top X$ depends entirely on the units of the input features: a feature measured in millimeters produces a very differently scaled matrix than the same feature measured in kilometers.

    Fitting OLS

    fit_ols accumulates $X^\top X$ and $X^\top \mathbf y$ directly from the rows, one at a time, rather than materializing the design matrix and multiplying it out. This is both less memory and reads directly off the normal-equations derivation: each row contributes its own outer product to the running sum.

    // Builds X^T X and X^T y directly (never materializing the full design
// matrix product), then solves the normal equations. O(n*d^2 + d^3).
static Vector fit_ols(const Matrix& X, const Vector& y) {
    validate_dataset(X, y);
    int n = (int)X.size();
    int d = (int)X[0].size() + 1;
    Matrix xtx(d, Vector(d, 0.0));
    Vector xty(d, 0.0);
    for (int i = 0; i < n; ++i) {
        Vector row(d, 1.0);
        for (int j = 1; j < d; ++j) row[j] = X[i][j - 1];
        for (int a = 0; a < d; ++a) {
            xty[a] += row[a] * y[i];
            for (int b = 0; b < d; ++b) xtx[a][b] += row[a] * row[b];
        }
    }
    if (n < d)
        throw DataError("fit_ols: " + to_string(n) + " rows is fewer than " +
                         to_string(d) + " parameters; the system is underdetermined");
    return solve_linear(xtx, xty);
}

    The row-count check comes after the accumulation only because it is cheap to state once $d$ is known; logically it belongs with the other validation. With fewer distinct rows than parameters, $X^\top X$ is guaranteed to be singular (its rank cannot exceed $n$), so solve_linear would reject it anyway, but raising a specifically worded error here is more useful for debugging than a generic singularity message.

    Weighted Least Squares

    Ordinary least squares implicitly assumes every row is equally informative. That is often false: some observations come from noisier sensors, smaller sample sizes, or less trustworthy sources. Weighted least squares (WLS) minimizes a weighted sum of squared errors,

    $$J_w(\mathbf w) = \frac{1}{2n}\sum_{i=1}^n \text{weight}_i \left(y_i - \mathbf x_i^\top \mathbf w\right)^2$$

    which leads by the same derivation to weighted normal equations $X^\top W X\, \mathbf w = X^\top W \mathbf y$, where $W$ is the diagonal matrix of weights. In code this only changes what gets accumulated into xtx and xty: each row's outer product is scaled by its own weight before being added in.

    // Minimizes sum_i weight[i] * (y_i - x_i . w)^2. Setting weight[i] small
// down-weights a noisy or low-confidence row instead of deleting it.
static Vector fit_wls(const Matrix& X, const Vector& y, const Vector& weight) {
    validate_dataset(X, y);
    if (weight.size() != y.size()) throw DataError("fit_wls: weight size mismatch");
    for (double w : weight) {
        if (!isfinite(w) || w < 0.0) throw DataError("fit_wls: weights must be finite and non-negative");
    }
    int n = (int)X.size();
    int d = (int)X[0].size() + 1;
    Matrix xtx(d, Vector(d, 0.0));
    Vector xty(d, 0.0);
    for (int i = 0; i < n; ++i) {
        Vector row(d, 1.0);
        for (int j = 1; j < d; ++j) row[j] = X[i][j - 1];
        for (int a = 0; a < d; ++a) {
            xty[a] += weight[i] * row[a] * y[i];
            for (int b = 0; b < d; ++b) xtx[a][b] += weight[i] * row[a] * row[b];
        }
    }
    return solve_linear(xtx, xty);
}

    WLS is the right tool when the per-row noise variance is known or can be estimated (weight $\propto 1/\text{variance}$), when some rows are duplicated aggregates of others (weight $\propto$ count), or as a pragmatic way to soften the influence of a handful of suspicious rows without deleting them outright. It is not a substitute for a robust loss; a single very large error still dominates the squared-error term unless its weight is specifically reduced, which usually requires already knowing which rows are suspect. The robust regression post handles the case where the noisy rows are not known in advance.

    Polynomial and Basis Expansion, in Depth

    polynomial_design, shown above, turns one feature $x$ into $[1, x, x^2, \dots, x^k]$. Fitting OLS on that expanded matrix finds the best-fit degree-$k$ polynomial in $x$, while remaining exactly linear regression in the expanded parameters: the objective and the normal equations are unchanged, only the column set is richer. The same idea generalizes past polynomials to any fixed, precomputed transform: indicator columns for categories, $\log x$ or $\sqrt x$ for right-skewed features, radial basis functions centered at fixed points, or interaction terms $x_1 x_2$. All of these are "basis expansion": choose the columns before fitting, and OLS treats them exactly like any other feature.

    Two things get worse as the basis grows richer, and both compound with each other. First, the expanded columns are almost always correlated (adjacent powers of the same variable move together), which is exactly the kind of near-singularity solve_linear is built to detect. Second, a rich enough basis will fit training noise as well as signal, degrading test performance even as the training error keeps improving. Centering and scaling $x$ before expanding it (so a large raw magnitude is not raised to increasingly enormous powers) reduces the first problem; regularization or a validation-based choice of $k$ addresses the second. The regularized regression post picks this up directly, since penalizing the expanded coefficients is the standard fix once the basis is rich enough to overfit.

    Prediction and Evaluation Metrics

    Prediction is the same dot product used in the derivation. Three metrics summarize how good a set of predictions is, and they disagree on purpose: RMSE weights large errors more than small ones (because it squares them before averaging), MAE weights every unit of error the same regardless of size, and R2 reports the fraction of the target's variance explained relative to always predicting the mean.

    static double predict(const Vector& w, const Vector& x) {
    double yhat = w[0];
    for (size_t j = 0; j < x.size(); ++j) yhat += w[j + 1] * x[j];
    return yhat;
}

static Vector predict_all(const Vector& w, const Matrix& X) {
    Vector out(X.size());
    for (size_t i = 0; i < X.size(); ++i) out[i] = predict(w, X[i]);
    return out;
}

static double rmse(const Vector& yhat, const Vector& y) {
    double s = 0.0;
    for (size_t i = 0; i < y.size(); ++i) s += (yhat[i] - y[i]) * (yhat[i] - y[i]);
    return sqrt(s / y.size());
}

static double mae(const Vector& yhat, const Vector& y) {
    double s = 0.0;
    for (size_t i = 0; i < y.size(); ++i) s += fabs(yhat[i] - y[i]);
    return s / y.size();
}

static double r_squared(const Vector& yhat, const Vector& y) {
    double mean = accumulate(y.begin(), y.end(), 0.0) / y.size();
    double ss_res = 0.0, ss_tot = 0.0;
    for (size_t i = 0; i < y.size(); ++i) {
        ss_res += (y[i] - yhat[i]) * (y[i] - yhat[i]);
        ss_tot += (y[i] - mean) * (y[i] - mean);
    }
    return ss_tot > 0.0 ? 1.0 - ss_res / ss_tot : 0.0;
}

    R2 is computed here against whatever mean is passed in; computed on the training set it can only decrease as features are removed, which is why comparing R2 across models with different numbers of parameters usually calls for an adjusted variant instead. All three metrics should be reported on held-out data, not training data, since training error is a biased (optimistic) estimate of how the model performs on new rows.

    Worked Example: Compiling and Running

    The dataset is a small synthetic housing table: house size in units of 100 square feet, bedroom count, and price in thousands of dollars. It is reused, unchanged, in the gradient descent and regularization posts so the three fitting methods can be compared directly on the same numbers. The last four rows are held out as a test set.

    // ---------- worked example ----------

int main() {
    // Housing-style toy dataset: size in 100 sqft units, bedroom count -> price in $1000s.
    Matrix X = {
        {20.5, 4}, {25.9, 5}, {23.5, 5}, {12.5, 5}, {14.0, 4}, {25.5, 4},
        {8.1, 3}, {24.4, 5}, {23.9, 3}, {17.4, 2}, {14.1, 5}, {13.6, 2},
        {13.1, 5}, {16.9, 4}, {18.1, 2}, {19.1, 2}
    };
    Vector y = {228.0, 277.8, 241.1, 174.4, 177.6, 265.2, 109.5, 264.2,
                232.4, 173.0, 199.2, 144.5, 183.0, 206.8, 180.1, 191.4};

    // Fixed 75/25 split. In production, shuffle with a seeded RNG before
    // splitting, and never let the split depend on the target column.
    int n_train = 12;
    Matrix X_train(X.begin(), X.begin() + n_train), X_test(X.begin() + n_train, X.end());
    Vector y_train(y.begin(), y.begin() + n_train), y_test(y.begin() + n_train, y.end());

    Matrix D = add_intercept(X_train);
    cout << fixed << setprecision(3);
    cout << "Design matrix row 0 (with intercept column): [";
    for (size_t j = 0; j < D[0].size(); ++j) cout << D[0][j] << (j + 1 < D[0].size() ? ", " : "");
    cout << "]\n\n";

    Vector w = fit_ols(X_train, y_train);
    cout << "OLS coefficients (intercept, size, rooms):\n";
    cout << "  w0 = " << w[0] << "\n  w1 = " << w[1] << "\n  w2 = " << w[2] << "\n";

    Vector pred_train = predict_all(w, X_train);
    Vector pred_test = predict_all(w, X_test);
    cout << "train RMSE = " << rmse(pred_train, y_train)
         << "  train R2 = " << r_squared(pred_train, y_train) << "\n";
    cout << "test  RMSE = " << rmse(pred_test, y_test)
         << "  test  MAE = " << mae(pred_test, y_test)
         << "  test  R2 = " << r_squared(pred_test, y_test) << "\n";

    // Weighted least squares: down-weight row 6 (the smallest, noisiest house)
    // to a quarter of the influence of a normal row.
    Vector weight(n_train, 1.0);
    weight[6] = 0.25;
    Vector w_wls = fit_wls(X_train, y_train, weight);
    cout << "\nWLS coefficients with row 6 down-weighted:\n";
    cout << "  w0 = " << w_wls[0] << "\n  w1 = " << w_wls[1] << "\n  w2 = " << w_wls[2] << "\n";

    // Polynomial regression: fit price as a cubic in size alone, ignoring rooms,
    // to show basis expansion turning a curved relationship into a linear-in-parameters fit.
    Vector size_only(n_train);
    for (int i = 0; i < n_train; ++i) size_only[i] = X_train[i][0];
    Matrix P = polynomial_design(size_only, 3);
    // polynomial_design already includes the leading 1 column, so fit_ols's own
    // intercept column would duplicate it; build normal equations directly here.
    int d = 4;
    Matrix xtx(d, Vector(d, 0.0));
    Vector xty(d, 0.0);
    for (int i = 0; i < n_train; ++i)
        for (int a = 0; a < d; ++a) {
            xty[a] += P[i][a] * y_train[i];
            for (int b = 0; b < d; ++b) xtx[a][b] += P[i][a] * P[i][b];
        }
    Vector wp = solve_linear(xtx, xty);
    cout << "\nCubic-in-size coefficients (c0..c3): ";
    for (size_t i = 0; i < wp.size(); ++i)
        cout << (i ? " " : "") << wp[i];
    cout << "\n";

    // Validation demo: a singular system (duplicated column) is rejected, not silently solved.
    try {
        Matrix Xbad = {{1, 1}, {2, 2}, {3, 3}};
        Vector ybad = {1, 2, 3};
        fit_ols(Xbad, ybad);
    } catch (const DataError& e) {
        cout << "\nCaught expected error on collinear input: " << e.what() << "\n";
    }
    return 0;
}

    Concatenating every snippet above, in order, with #include <bits/stdc++.h>, using namespace std;, and using Vector = vector<double>; using Matrix = vector<Vector>; at the top, reproduces the full program. It compiles cleanly with g++ -std=c++17 -O2 -Wall -Wextra ols.cpp -o ols and prints:

    Design matrix row 0 (with intercept column): [1.000, 20.500, 4.000] OLS coefficients (intercept, size, rooms): w0 = 16.519 w1 = 7.395 w2 = 13.545 train RMSE = 6.765 train R2 = 0.982 test RMSE = 6.654 test MAE = 5.548 test R2 = 0.589 WLS coefficients with row 6 down-weighted: w0 = 21.350 w1 = 7.203 w2 = 13.389 Cubic-in-size coefficients (c0..c3): -140.343 48.626 -2.586 0.051 Caught expected error on collinear input: solve_linear: matrix is singular or near-singular at column 2 (check for duplicated/collinear features)

    Every extra house's size adds roughly 7.4 to its price (in thousands), and every extra bedroom adds roughly 13.5, holding the other feature fixed. Down-weighting row 6, the smallest and cheapest house in the training set, shifts the intercept up and the size coefficient down slightly, exactly the kind of small, localized change WLS is meant to produce. The cubic-in-size fit shows basis expansion working (four coefficients for one input feature), and the final block confirms that a duplicated-column system is rejected with a clear message instead of returning a numerically meaningless result.

    Numerical Stability and Production Guidance

    Forming $X^\top X$ squares the condition number of $X$: if $X$ has condition number $\kappa$, then $X^\top X$ has condition number roughly $\kappa^2$. For well-scaled, weakly correlated features this rarely matters. It matters a great deal once features are on very different scales (raw square footage next to a 0/1 indicator) or strongly correlated (two measurements of nearly the same thing), because $\kappa^2$ can turn a merely large condition number into one that consumes most of the solver's usable precision.

    • Center and scale features before solving. Standardizing each column to zero mean and unit variance keeps the entries of $X^\top X$ on a comparable scale and improves the conditioning of the system being solved, independent of whether the model is fit by a direct solver or by gradient descent.
    • Prefer QR or SVD in production code. Both factor $X$ directly and never form $X^\top X$ explicitly, so they do not pay the squared-condition-number penalty. LAPACK-backed libraries (Eigen, LAPACK itself, or a numerical library's lstsq) implement this correctly; the Gaussian-elimination solver here is intentionally simple and educational, appropriate for small, well-scaled problems, not a replacement for a production linear algebra library on ill-conditioned or very large systems.
    • Treat "singular" as information, not just an error to catch. A rejected solve almost always means two columns are duplicated or nearly so (collinearity), or that there are fewer distinct rows than parameters. Both are worth fixing at the data level (drop or combine the redundant column, collect more rows) in addition to, or instead of, adding regularization.
    • Regularization is the standard fallback when collinearity is real and unavoidable rather than a data bug: Ridge regression adds a small positive value to the diagonal of $X^\top X$ before solving, which guarantees invertibility and improves conditioning at the cost of a small, controlled bias. That is the subject of the next post.
