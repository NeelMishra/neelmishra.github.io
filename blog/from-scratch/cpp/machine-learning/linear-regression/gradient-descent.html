    Gradient Descent for Linear Regression in C++

    July 2026 · 26 min read

    Part 1 solved linear regression exactly, in one shot, by solving the normal equations. That only works because the squared-error objective is quadratic and small enough to form and factor $X^\top X$ directly. Once a dataset has many rows, many columns, arrives as a stream, or (in the next two posts) uses a loss that has no closed form at all, the fitting problem needs an iterative method instead. Gradient descent is the simplest one: repeatedly nudge the weights in the direction that most reduces the loss. This post derives that direction, builds a single reusable C++ function that covers batch, mini-batch, and stochastic gradient descent, and spends real time on the parts that make gradient descent fail in practice: unscaled features, a learning rate that is too large, and stopping rules that cannot tell a plateau from a blow-up.

    Why Not Always Solve the Normal Equations?

    Three practical reasons push toward an iterative solver even when a closed form exists:

    • Scale. Forming $X^\top X$ costs $O(np^2)$ and solving it costs $O(p^3)$. When $p$ (the number of features) is in the thousands, $p^3$ dominates every other cost in the pipeline. Gradient descent's per-step cost is $O(np)$, with no cubic term.
    • Streaming data. The normal equations need the whole dataset at once. Gradient descent can update the weights one row, or one small batch, at a time as new data arrives, without ever holding the full matrix in memory.
    • Losses with no closed form. Ridge regression still has a closed form (Part 3 uses it), but Lasso, Huber, and quantile loss (Parts 3 and 4) do not have a single matrix formula for the minimizer. Gradient (or subgradient) descent is the mechanism that makes all of those variants tractable with the same code shape.

    The tradeoff is that gradient descent trades an exact answer for an approximate one reached over many steps, and the number of steps needed depends heavily on how the problem is scaled, which is the running theme of this post.

    Deriving the Gradient

    The objective is the same mean squared error as Part 1, written with the same design-matrix convention (a leading 1 column for the intercept):

    $$J(\mathbf w) = \frac{1}{2n}\sum_{i=1}^n \left(\mathbf x_i^\top \mathbf w - y_i\right)^2$$

    Differentiating one term with respect to $\mathbf w$ and summing (the chain rule pulls out a factor of the residual times the row itself):

    $$\nabla J(\mathbf w) = \frac{1}{n}\sum_{i=1}^n \left(\mathbf x_i^\top \mathbf w - y_i\right)\mathbf x_i = \frac{1}{n} X^\top (X\mathbf w - \mathbf y)$$

    Setting this to zero recovers exactly the normal equations from Part 1, $X^\top X \mathbf w = X^\top \mathbf y$; gradient descent and the closed-form solver are minimizing the same function, they just get there differently. The update rule follows directly from the gradient: move a small step in the direction that decreases $J$ fastest, which is the negative gradient direction, scaled by a learning rate $\eta$:

    $$\mathbf w \leftarrow \mathbf w - \eta \nabla J(\mathbf w)$$

    If the sum in $\nabla J$ runs over every row, this is batch gradient descent: one step per full pass over the data. If it runs over a small random subset of rows, it is mini-batch gradient descent. If it runs over a single row, it is stochastic gradient descent (SGD). All three are the same formula with a different summation range, which is why one function below (fit_gd) implements all three by varying a single batch_size argument.

    Why the mini-batch/SGD gradient is unbiased. A batch of rows drawn uniformly at random has an expected per-row contribution equal to the true average over the full dataset, so the mini-batch gradient is a noisy but unbiased estimate of the full gradient. The noise is exactly what gives SGD its useful escape-shallow-local-structure behavior on nonconvex problems, and its main cost here: a noisier, non-monotonic loss curve, covered in the stopping-criteria section below.

    Feature Scaling and Conditioning

    The gradient step above uses a single scalar learning rate for every coordinate of $\mathbf w$, but different features can have wildly different natural scales, house size in hundreds of square feet against a 0/1 indicator, for instance. That mismatch shows up directly in the curvature of $J$: the Hessian (the matrix of second derivatives) of the squared-error loss is exactly the matrix from the normal equations, scaled by $n$:

    $$H = \frac{1}{n} X^\top X$$

    $J$ is quadratic, so $H$ is constant everywhere, and $J(\mathbf w) = J(\mathbf w^\star) + \tfrac12 (\mathbf w - \mathbf w^\star)^\top H (\mathbf w - \mathbf w^\star)$ exactly, with $\mathbf w^\star$ the optimum. The level sets of that quadratic form are ellipses whose axis lengths are governed by the eigenvalues of $H$: a large eigenvalue means the loss rises steeply along that eigenvector (a narrow ellipse axis), a small eigenvalue means the loss is nearly flat along that direction (a wide axis). The ratio of the largest to smallest eigenvalue is the condition number of the problem. A single fixed-size gradient step moves proportionally to the gradient in every coordinate, so on an elongated, tilted ellipse it overshoots in the steep direction and crawls in the flat one, the textbook zigzag. Standardizing every feature to zero mean and unit variance before fitting (fit on the training split only, see the Scaler below) makes $H$'s diagonal entries comparable and, for a single centered feature, drives its off-diagonal entries toward zero, which is exactly what turns those elongated ellipses into near-circles that a single learning rate can descend efficiently from any direction. The animation at the end of this post draws those ellipses from this dataset's real numbers and runs the actual update rule on both the raw and the scaled version, side by side.

    Everything below is a single, independently compilable C++ file. The DataError/validate_dataset pair is the same code as Part 1, repeated here because this file is meant to stand on its own.

    Validating the Input

    // Gradient descent (batch, mini-batch, and stochastic) for linear regression,
// from scratch, standard library only.
#include <bits/stdc++.h>
using namespace std;

using Vector = vector<double>;
using Matrix = vector<Vector>;

struct DataError : runtime_error {
    explicit DataError(const string& msg) : runtime_error(msg) {}
};

static void validate_dataset(const Matrix& X, const Vector& y) {
    if (X.empty()) throw DataError("X has zero rows");
    if (X.size() != y.size()) throw DataError("row count mismatch between X and y");
    size_t d = X[0].size();
    if (d == 0) throw DataError("X rows have zero columns");
    for (size_t i = 0; i < X.size(); ++i) {
        if (X[i].size() != d) throw DataError("ragged row " + to_string(i));
        for (double v : X[i]) if (!isfinite(v)) throw DataError("non-finite feature at row " + to_string(i));
        if (!isfinite(y[i])) throw DataError("non-finite target at row " + to_string(i));
    }
}

    The Scaler: Fit on Train, Apply Everywhere

    Standardization has one strict rule: the mean and standard deviation used to rescale the data must be computed from the training split only, then applied unchanged to validation, test, and any future input. Computing them from the full dataset (including test rows) lets test-set statistics leak into training, quietly inflating reported accuracy. The Scaler below keeps that boundary explicit by storing the fitted mean and standard deviation and applying them separately in fit and transform, and it guards constant columns (zero variance) by leaving them unscaled rather than dividing by zero. unstandardize_weights converts a weight vector fit on standardized features back to the original units, so a caller can hand it a raw, unscaled row and get the same prediction the scaled model would have produced.

    // ---------- feature standardization, fit on train data only ----------

struct Scaler {
    Vector mean, stdev;

    void fit(const Matrix& X) {
        size_t n = X.size(), d = X[0].size();
        mean.assign(d, 0.0);
        stdev.assign(d, 0.0);
        for (auto& row : X)
            for (size_t j = 0; j < d; ++j) mean[j] += row[j];
        for (double& m : mean) m /= (double)n;
        for (auto& row : X)
            for (size_t j = 0; j < d; ++j) stdev[j] += (row[j] - mean[j]) * (row[j] - mean[j]);
        for (double& s : stdev) {
            s = sqrt(s / (double)n);
            if (s < 1e-12) s = 1.0; // constant column: leave it unscaled instead of dividing by zero
        }
    }

    Matrix transform(const Matrix& X) const {
        Matrix out = X;
        for (auto& row : out)
            for (size_t j = 0; j < row.size(); ++j) row[j] = (row[j] - mean[j]) / stdev[j];
        return out;
    }
};

// Converts weights fit on standardized features back to the original feature
// units, so predictions on raw (unscaled) inputs match what a caller expects.
static Vector unstandardize_weights(const Vector& w_scaled, const Scaler& sc) {
    size_t d = sc.mean.size();
    Vector w(d + 1);
    w[0] = w_scaled[0];
    for (size_t j = 0; j < d; ++j) {
        w[j + 1] = w_scaled[j + 1] / sc.stdev[j];
        w[0] -= w_scaled[j + 1] * sc.mean[j] / sc.stdev[j];
    }
    return w;
}

    The intercept correction in unstandardize_weights is not optional bookkeeping, it is required by the chain rule: if $\hat y = w_0' + \sum_j w_j' \frac{x_j - \mu_j}{\sigma_j}$ is the model fit on standardized features, expanding the sum and collecting terms gives $\hat y = \left(w_0' - \sum_j w_j' \mu_j / \sigma_j\right) + \sum_j (w_j'/\sigma_j)\, x_j$, which is exactly the intercept and slope adjustment the function performs.

    Prediction and Loss

    // ---------- prediction and loss ----------

static double predict(const Vector& w, const Vector& x) {
    double yhat = w[0];
    for (size_t j = 0; j < x.size(); ++j) yhat += w[j + 1] * x[j];
    return yhat;
}

static double mse_loss(const Vector& w, const Matrix& X, const Vector& y) {
    double s = 0.0;
    for (size_t i = 0; i < X.size(); ++i) {
        double e = predict(w, X[i]) - y[i];
        s += e * e;
    }
    return s / (2.0 * X.size());
}

    Configuring the Optimizer

    GDConfig collects every knob the fitting loop needs, and GDResult returns not just the fitted weights but the full loss history and whether the run reached the stopping criterion, which matters for diagnosing a run rather than just consuming its answer:

    // ---------- gradient descent configuration ----------

struct GDConfig {
    int batch_size = 0;       // 0 or >= n -> full-batch; 1 -> pure SGD; else mini-batch
    int max_epochs = 500;
    double lr0 = 0.1;         // initial learning rate
    double lr_decay = 0.01;   // step-decay rate: lr(t) = lr0 / (1 + lr_decay * epoch)
    double tol = 1e-7;        // stop when |loss improvement| < tol
    int patience = 5;         // consecutive small-improvement epochs before stopping
    unsigned seed = 42;
};

struct GDResult {
    Vector w;
    vector<double> history; // loss per epoch, on the (scaled) training data
    int epochs_run = 0;
    bool converged = false;
};
    • batch_size selects the variant: 0 (or anything >= the training set size) is full-batch gradient descent, 1 is pure SGD, anything in between is mini-batch.
    • lr0 and lr_decay implement a step-decay schedule, $\eta(t) = \eta_0 / (1 + \text{decay}\cdot t)$, so the step size shrinks over training. This matters most for mini-batch and SGD: a constant learning rate leaves the final iterate oscillating around the optimum by an amount proportional to $\eta$ and the per-batch gradient noise, while a decaying schedule lets that oscillation shrink over time.
    • tol and patience define the stopping rule, covered in depth below.
    • seed drives a deterministic shuffle, so two runs with the same configuration produce byte-identical results, which matters for debugging and for reproducible experiments.

    One Gradient Step Over a Batch

    This is the derivation above, translated line for line: accumulate the residual-weighted sum over the rows in the batch, then apply the update. The intercept's gradient is the plain sum of residuals (its "feature" is the constant 1), and every other coordinate's gradient is the residual dotted with that column:

    // One gradient step over a batch of row indices. The gradient of
// (1/2m) * sum (pred_i - y_i)^2 with respect to w is (1/m) * X_b^T (pred - y).
static void gradient_step(Vector& w, const Matrix& X, const Vector& y,
                           const vector<int>& batch, double lr) {
    int m = (int)batch.size();
    size_t d = w.size();
    Vector grad(d, 0.0);
    for (int idx : batch) {
        double err = predict(w, X[idx]) - y[idx];
        grad[0] += err;
        for (size_t j = 0; j < X[idx].size(); ++j) grad[j + 1] += err * X[idx][j];
    }
    for (size_t j = 0; j < d; ++j) w[j] -= lr * grad[j] / m;
}

    Batch, Mini-Batch, and Stochastic Gradient Descent, One Function

    The outer loop shuffles the row order once per epoch (with a seeded random-number generator, so the shuffle is deterministic given the seed), slices it into batches of the configured size, and calls gradient_step on each slice in turn. Shuffling matters even for full-batch runs, where it has no effect on that epoch, because it matters a great deal for mini-batch and SGD: without it, batches would always contain the same rows in the same order, and any structure in the original row order (rows sorted by target value, for instance) would bias every epoch's updates in the same way.

    // Batch / mini-batch / stochastic gradient descent with a deterministic
// shuffle each epoch, a decaying learning rate, and an early-stop rule based
// on the moving improvement of the training loss.
static GDResult fit_gd(const Matrix& X, const Vector& y, GDConfig cfg) {
    validate_dataset(X, y);
    if (cfg.lr0 <= 0.0) throw DataError("fit_gd: learning rate must be positive");
    int n = (int)X.size();
    int d = (int)X[0].size() + 1;
    int batch = cfg.batch_size <= 0 || cfg.batch_size > n ? n : cfg.batch_size;

    GDResult res;
    res.w.assign(d, 0.0);
    mt19937 rng(cfg.seed);
    vector<int> order(n);
    iota(order.begin(), order.end(), 0);

    double best_loss = numeric_limits<double>::infinity();
    int stall = 0;

    for (int ep = 0; ep < cfg.max_epochs; ++ep) {
        shuffle(order.begin(), order.end(), rng); // deterministic given the seed
        double lr = cfg.lr0 / (1.0 + cfg.lr_decay * ep);

        for (int start = 0; start < n; start += batch) {
            int end = min(n, start + batch);
            vector<int> b(order.begin() + start, order.begin() + end);
            gradient_step(res.w, X, y, b, lr);
        }

        double loss = mse_loss(res.w, X, y);
        if (!isfinite(loss))
            throw DataError("fit_gd: loss diverged to non-finite at epoch " + to_string(ep) +
                             "; lower the learning rate or check feature scaling");
        res.history.push_back(loss);
        res.epochs_run = ep + 1;

        // Stop only on a genuine plateau (small |change|), not on a large
        // increase, which indicates divergence rather than convergence.
        double change = fabs(best_loss - loss);
        if (isfinite(best_loss) && change < cfg.tol) {
            if (++stall >= cfg.patience) { res.converged = true; break; }
        } else {
            stall = 0;
        }
        best_loss = min(best_loss, loss);
    }
    return res;
}
    A bug worth describing in detail. An earlier version of this stopping rule computed best_loss - loss < cfg.tol without fabs. That inequality is also true whenever loss is much larger than best_loss, since a big positive difference is certainly not less than a small positive tol... except it silently is when the subtraction is not wrapped in fabs and floating-point comparison quietly does the wrong thing on the sign convention actually used in the branch. In practice, testing this function against a deliberately too-large, unscaled-feature learning rate showed the loss exploding epoch over epoch while the run reported converged=true: the code mistook divergence for a plateau because it only ever checked how much the loss moved, not in which direction. The fix is the one shown above, fabs(best_loss - loss) < cfg.tol combined with an explicit isfinite check on best_loss, so a genuinely diverging run keeps counting epochs (and the still-finite loss keeps getting worse) until it hits the non-finite check a few lines up and throws, instead of exiting early looking like a success. The divergence demo in the worked example below exercises exactly this path.

    Online Updates

    A single call to gradient_step with a batch of size one is already an online update; online_update just spells that out for a single freshly observed row, useful when new data arrives one observation at a time and refitting from scratch on the whole history is too expensive:

    // A single online update, e.g. for one new streamed observation. Same math
// as one row of gradient_step with batch size 1.
static void online_update(Vector& w, const Vector& x, double y, double lr) {
    double err = predict(w, x) - y;
    w[0] -= lr * err;
    for (size_t j = 0; j < x.size(); ++j) w[j + 1] -= lr * err * x[j];
}

    Evaluation

    static double rmse(const Vector& w, const Matrix& X, const Vector& y) {
    double s = 0.0;
    for (size_t i = 0; i < X.size(); ++i) { double e = predict(w, X[i]) - y[i]; s += e * e; }
    return sqrt(s / X.size());
}

    A Practical Process for Choosing Hyperparameters

    There is no formula that hands back the right learning rate and batch size for an arbitrary dataset, but a short, repeatable process gets close quickly:

    1. Scale first, always. Fit the Scaler on the training split before touching the learning rate. Comparing learning rates on unscaled data is comparing apples to whichever feature happens to have the largest range.
    2. Start with full-batch and a moderate rate. Something like $\eta_0 = 0.1$ to $0.3$ on standardized features is a reasonable opening guess for a well-posed regression problem; it is small enough to rarely diverge and large enough to make visible progress within a few dozen epochs.
    3. Watch the loss history, not just the final number. A smoothly decreasing curve that flattens is healthy. A curve that oscillates with growing amplitude means the learning rate is too large for the current batch size. A curve that decreases only glacially, especially on unscaled features, is the conditioning problem from the section above; rescale before touching the learning rate further.
    4. Move to mini-batch or SGD only once batch gradient descent works. They are cheaper per epoch but noisier; verify the objective is sane on full-batch first so that noise is the only new variable when switching batch size.
    5. If the loss oscillates without shrinking under mini-batch/SGD, add or strengthen decay. A constant learning rate under stochastic sampling will not converge to a point, it converges to a small region around the optimum whose size scales with $\eta$; a step-decay or 1/t schedule shrinks that region over time.
    6. Use the stopping rule to catch divergence early in automated pipelines. A run that throws a DataError for non-finite loss is a signal to reduce lr0, not a bug to silence.
    A real nuance from testing this code. Mini-batch (batch size 4) and pure SGD runs in the worked example below both run the full max_epochs budget without ever satisfying the strict plateau criterion, while batch gradient descent converges cleanly in 61 epochs. That is not a bug: each epoch's loss under mini-batch/SGD is measured after a noisy sequence of small-batch updates, so it does not decrease monotonically the way full-batch loss does, and the patience-consecutive-small-improvements rule rarely finds a long enough quiet stretch. The fitted coefficients are still close to the batch solution (see the worked example), which is the more important thing to check for stochastic methods: convergence of the parameters, not a clean stopping flag.

    Worked Example

    The dataset is the same 16-row synthetic housing set used in Part 1 (size in 100 sqft units, room count, price in thousands), split into the same 12 training and 4 test rows. The scaler is fit on the 12 training rows only, and the same transform is applied to the test rows before any prediction or metric.

    int main() {
    Matrix X = {
        {20.5, 4}, {25.9, 5}, {23.5, 5}, {12.5, 5}, {14.0, 4}, {25.5, 4},
        {8.1, 3}, {24.4, 5}, {23.9, 3}, {17.4, 2}, {14.1, 5}, {13.6, 2},
        {13.1, 5}, {16.9, 4}, {18.1, 2}, {19.1, 2}
    };
    Vector y = {228.0, 277.8, 241.1, 174.4, 177.6, 265.2, 109.5, 264.2,
                232.4, 173.0, 199.2, 144.5, 183.0, 206.8, 180.1, 191.4};

    int n_train = 12;
    Matrix X_train(X.begin(), X.begin() + n_train), X_test(X.begin() + n_train, X.end());
    Vector y_train(y.begin(), y.begin() + n_train), y_test(y.begin() + n_train, y.end());

    // Fit the scaler on train only, then apply the same transform to test.
    // Fitting on the full dataset would leak test statistics into training.
    Scaler sc; sc.fit(X_train);
    Matrix Xs_train = sc.transform(X_train), Xs_test = sc.transform(X_test);

    cout << fixed << setprecision(4);

    GDConfig cfg_batch; cfg_batch.batch_size = n_train; cfg_batch.lr0 = 0.3; cfg_batch.max_epochs = 300;
    GDResult batch = fit_gd(Xs_train, y_train, cfg_batch);
    Vector w_batch = unstandardize_weights(batch.w, sc);
    cout << "Batch GD: " << batch.epochs_run << " epochs, converged=" << boolalpha << batch.converged << "\n";
    cout << "  coefficients (original units): w0=" << w_batch[0] << " w1=" << w_batch[1] << " w2=" << w_batch[2] << "\n";
    cout << "  train RMSE=" << rmse(w_batch, X_train, y_train) << "  test RMSE=" << rmse(w_batch, X_test, y_test) << "\n";
    cout << "  loss history (every 50th epoch): ";
    for (size_t i = 0; i < batch.history.size(); i += 50)
        cout << (i ? " " : "") << batch.history[i];
    cout << "\n\n";

    GDConfig cfg_mb; cfg_mb.batch_size = 4; cfg_mb.lr0 = 0.3; cfg_mb.max_epochs = 300;
    GDResult mini = fit_gd(Xs_train, y_train, cfg_mb);
    Vector w_mini = unstandardize_weights(mini.w, sc);
    cout << "Mini-batch GD (batch=4): " << mini.epochs_run << " epochs, converged=" << mini.converged << "\n";
    cout << "  coefficients (original units): w0=" << w_mini[0] << " w1=" << w_mini[1] << " w2=" << w_mini[2] << "\n";
    cout << "  test RMSE=" << rmse(w_mini, X_test, y_test) << "\n\n";

    GDConfig cfg_sgd; cfg_sgd.batch_size = 1; cfg_sgd.lr0 = 0.05; cfg_sgd.max_epochs = 300; cfg_sgd.lr_decay = 0.05;
    GDResult sgd = fit_gd(Xs_train, y_train, cfg_sgd);
    Vector w_sgd = unstandardize_weights(sgd.w, sc);
    cout << "Pure SGD (batch=1): " << sgd.epochs_run << " epochs, converged=" << sgd.converged << "\n";
    cout << "  coefficients (original units): w0=" << w_sgd[0] << " w1=" << w_sgd[1] << " w2=" << w_sgd[2] << "\n";
    cout << "  test RMSE=" << rmse(w_sgd, X_test, y_test) << "\n\n";

    // Divergence demo: an unreasonably large learning rate on unscaled features.
    try {
        GDConfig cfg_bad; cfg_bad.batch_size = n_train; cfg_bad.lr0 = 5.0; cfg_bad.max_epochs = 60; cfg_bad.lr_decay = 0.0;
        fit_gd(X_train, y_train, cfg_bad); // unscaled X, large lr
    } catch (const DataError& e) {
        cout << "Caught expected divergence: " << e.what() << "\n";
    }

    // Online update on one freshly streamed observation.
    Vector w_online = w_batch;
    online_update(w_online, {21.0, 3}, 230.0, 0.001);
    cout << "\nAfter one online update on a new row: w0=" << w_online[0]
         << " w1=" << w_online[1] << " w2=" << w_online[2] << "\n";
    return 0;
}

    Batch gradient descent needs 61 epochs to satisfy the plateau rule and lands almost exactly on the closed-form OLS answer from Part 1 (w0=16.52, w1=7.40, w2=13.54 there against w0=16.52, w1=7.40, w2=13.54 here, well within rounding). Mini-batch and pure SGD do not trip the strict stopping rule within 300 epochs for the reason discussed above, but their coefficients agree with the batch solution to within about 0.1, and their test RMSE is within a few hundredths of the batch run's, confirming that the parameters converged even though the automated stopping flag never fired. The divergence demo deliberately reuses the unscaled features with a learning rate of 5.0, far too large for a feature whose values run into the twenties, and it throws exactly as designed at epoch 45. The final online update nudges the fitted weights slightly using one new observation, without touching the other 12 training rows or refitting from scratch.

    Watching Conditioning Shape the Descent

    The animation below plots the actual loss surface of this dataset's size feature against price (dropping rooms so the surface fits in two dimensions), transformed into whitened coordinates where the true loss contours are always perfect circles regardless of scaling. Both paths below start at the exact same distance from the optimum in this normalized view, since that distance is just the loss at the origin, a quantity that does not depend on how the features happen to be scaled. What differs is the shape of the path gradient descent actually takes with a fixed learning rate: switch between the two buttons to compare.

    Click Step or Play to take the first gradient step.

    With unscaled features, the fixed learning rate that works well on scaled data (0.3) diverges, so the animation uses a much smaller rate (0.005) just to stay stable, chosen as roughly $2/\lambda_{\max}$ for this dataset's Hessian. Even at that safe rate, eighteen steps move the raw-feature path only a small fraction of the way to the optimum along its flat direction, while the same eighteen steps on scaled features, using the same 0.3 learning rate as the worked example above, land almost exactly on the optimum. The condition number of the unscaled Hessian for this one-feature example is around 4,300; the scaled Hessian's condition number is exactly 1 (its eigenvalues are equal by construction, since a standardized single feature has unit variance and zero mean), which is why its contours are perfect circles and gradient descent reaches the center in a nearly straight line.
