    Linear Regression from Scratch in C++

    July 2026 · Series overview

    Every linear regression variant predicts the same thing: a weighted sum of features plus an intercept. What changes from one variant to the next is how the weights are chosen: which objective is minimized, what penalty is added, and whether the fit is solved in one shot or approached iteratively. This series builds that whole family in dependency-free, standard-library C++, starting from the plain closed-form solution and working outward to regularized and robust variants. It is the implementation companion to the conceptual Linear Regression: Foundations series; that series explains the statistics and geometry in depth, this one turns the same ideas into code you can compile, test, and extend.

    This page is the map. It defines the shared model and data abstractions every part reuses, gives a decision table for picking a variant, lays out the implementation roadmap, and covers three things that are easy to get wrong regardless of which variant you pick: splitting data, scaling features, and avoiding leakage. It ends with an animation that fits the same eleven points with an underfit line, an overfit high-degree polynomial, and two Ridge-regularized versions of that same polynomial, so you can see bias, variance, and regularization strength trade off against each other directly.

    The Model and the Data

    Every variant in this series predicts a target $y$ from a feature row $x \in \mathbb{R}^p$ with the same functional form:

    $$\hat{y} = w_0 + w_1 x_1 + w_2 x_2 + \cdots + w_p x_p = w_0 + x^{\top} w_{1:p}$$

    Stack $n$ rows into a design matrix $X \in \mathbb{R}^{n \times (p+1)}$ whose first column is all ones (for the intercept) and whose remaining columns are the features, and stack the targets into $y \in \mathbb{R}^n$. The prediction for the whole dataset is $\hat{y} = Xw$. Every part of this series works with exactly this pair, Matrix X and Vector y, represented in code as:

    using Vector = std::vector<double>;
using Matrix = std::vector<Vector>;  // Matrix[row][col], row-major, includes the intercept column

    A DataError exception type and a validate_dataset function (checking row-count agreement, finite values, and a minimum row count before attempting to solve anything) appear near the top of every part's C++ file, because every variant needs the same basic sanity checks before it can safely fit. What differs, part to part, is only the objective function used to choose $w$ and, sometimes, the shape of $X$ itself (adding polynomial columns, or reweighting rows).

    Implementation Roadmap

    Part 1: Ordinary Least Squares

    Derive the normal equations, build a validated design matrix, solve with a partial-pivot Gaussian elimination that detects near-singular systems, then extend to weighted least squares and polynomial basis expansion. Complete, compilable program with RMSE, MAE, and R-squared.

    Part 2: Gradient Descent

    Derive the gradient of squared error, implement batch, mini-batch, and stochastic descent with a deterministic shuffle, train-only feature standardization, a learning-rate schedule, and a correct convergence check. Covers conditioning, why scaling matters, and online updates.

    Part 3: Regularized Regression

    Ridge in closed form, and a full coordinate-descent solver for Lasso and Elastic Net, not just a soft-threshold helper. Standardization, converting coefficients back to original units, warm-started paths, and cross-validation with the one-standard-error rule.

    Part 4: Robust and Quantile Regression

    Huber and pinball (quantile) losses with correctly signed gradients, a constant-learning-rate subgradient descent with best-iterate tracking, MAD-based scale estimation, multi-quantile fitting with a crossing repair step, and outlier-influence comparisons against OLS.

    Related: Linear Regression Assumptions and Diagnostics

    From the conceptual series: residual plots, leverage, Cook's distance, variance inflation factor, heteroscedasticity and autocorrelation checks, and repair strategies. Read this alongside any part above when a fit looks wrong and you need to work out why.

    Splitting, Scaling, and Leakage

    These three rules apply to every variant in the table above, so they are stated once here rather than repeated four times.

    Split before you look. Decide on a train/validation/test split (or a cross-validation scheme) before computing anything from the data, including summary statistics you might use "just to understand the data". A held-out test set that has influenced any modeling decision, even indirectly, no longer measures generalization; it measures how well you memorized the test set.

    Fit scaling on the training set only. Every part in this series that scales features (gradient descent, regularized regression, robust regression) fits the mean and standard deviation on the training rows, then applies that same fitted transform to validation and test rows. Computing the mean and standard deviation over the full dataset before splitting leaks test-set information into the training statistics, an easy mistake because the code still runs and the numbers still look reasonable.

    Leakage is a broader trap than just scaling. The same principle applies to polynomial degree selection, regularization strength, quantile bandwidth, or any other choice made "by looking at what works." If a choice is tuned by checking performance on data, that data can no longer be used to report an unbiased final performance number. The usual fix is a three-way split: train to fit parameters, validation to tune hyperparameters, and test touched exactly once at the end to report a number.

    A practical checklist. Before trusting any fit in this series: (1) was the split made before any data-dependent decision, (2) was every scaler or basis expansion fit only on the training rows, (3) was the test set used more than once, and (4) does the reported metric come from the untouched test set rather than from whichever split happened to look best?

    Evaluation Metrics

    Three metrics recur throughout the series, each answering a different question:

    MetricFormulaWhat it answers
    RMSE$\sqrt{\frac{1}{n}\sum_i (y_i - \hat{y}_i)^2}$Typical error size, in the target's original units, with large errors weighted more heavily (because of the square).
    MAE$\frac{1}{n}\sum_i |y_i - \hat{y}_i|$Typical error size, in the target's original units, with every row weighted equally regardless of error size. More robust to outliers than RMSE; the gap between MAE and RMSE is itself a diagnostic for outlier influence.
    R-squared$1 - \frac{\sum_i (y_i-\hat{y}_i)^2}{\sum_i (y_i-\bar{y})^2}$Fraction of target variance explained relative to always predicting the mean. Useful for comparing models on the same dataset; not directly comparable across different datasets or targets.

    Part 4 adds one more, the pinball (quantile) loss, because RMSE and MAE both implicitly target the conditional mean and are the wrong yardstick for judging a quantile fit; a $\tau=0.9$ fit is doing its job if 90% of points fall below it, which RMSE cannot tell you. Each detail page reports the metrics that are actually meaningful for its variant rather than repeating all of them by default.

    Watching Bias, Variance, and Regularization Interact

    The animation below fits the same eleven noisy points (drawn from a smooth curve, not a straight line) with four different models, computed live in JavaScript using the same normal-equations approach as Part 1 and the same ridge penalty as Part 3: a degree-1 line (too simple to capture the curve), a degree-9 polynomial with no regularization (flexible enough to chase the noise between points), the same degree-9 polynomial with a small ridge penalty (a sweet spot), and the same polynomial with a large ridge penalty (shrunk back toward too simple). Step through to see training error and wiggliness move in opposite directions as the model gets more flexible, and watch a small, well-chosen regularization strength recover most of the fit quality without the wiggle.

    The direction-change count in the info line above counts how many times the fitted curve switches from rising to falling or back, sampled densely across the x-range. The true underlying curve genuinely bends twice (it is half a sine wave), so a good fit should show roughly that many direction changes, not many more. The unregularized degree-9 fit has almost the lowest training error of the four (it is nearly interpolating the noisy points) but achieves that by wiggling far more than the signal actually does; a validation set, not training error, is what would catch this in practice. The small-ridge version gives up a little training accuracy to cut the wiggle back down to match the signal, which is exactly the trade Ridge is built to make.
