    Smart Pointers — Ownership Without the Pain

    March 2026 · 11 min read

    The Problem with Raw Pointers

    Raw pointers (new / delete) are the single biggest source of memory bugs in C++: leaks, double-frees, dangling references. Smart pointers automate ownership and cleanup via RAII.

    // Bad: who deletes this?
Widget* w = new Widget();
process(w);
// Did process() take ownership? Is w still valid?
// Forgot delete → leak. Double delete → crash.

    std::unique_ptr — Exclusive Ownership

    unique_ptr owns a resource exclusively. It cannot be copied, only moved. When it goes out of scope, it deletes the managed object.

    #include <memory>

auto w = std::make_unique<Widget>(42);

process(*w);              // pass by reference
take_ownership(std::move(w));  // transfers ownership
// w is now nullptr

    unique_ptr for Arrays

    auto arr = std::make_unique<int[]>(100);
arr[0] = 42;
// automatically calls delete[] when arr goes out of scope

    Custom Deleters

    For resources that aren't freed with delete (file handles, C library objects):

    auto file_closer = [](FILE* f) { if (f) fclose(f); };
std::unique_ptr<FILE, decltype(file_closer)> fp(
    fopen("data.txt", "r"), file_closer
);
    Rule: Default to unique_ptr. It has zero overhead compared to a raw pointer. Only reach for shared_ptr when you genuinely need shared ownership.

    std::shared_ptr — Shared Ownership

    shared_ptr uses reference counting. The last shared_ptr pointing to an object destroys it. Copying increments the count; destruction decrements it.

    auto a = std::make_shared<Widget>(42);
auto b = a;  // ref count = 2

b.reset();   // ref count = 1
// Widget destroyed when 'a' goes out of scope

    The Cost of shared_ptr

    • Control block — heap-allocated metadata (ref count, weak count, deleter). make_shared merges the control block and object into one allocation.
    • Atomic ref counting — thread-safe increment/decrement, which has a small but measurable overhead.
    • Size — two pointers (object + control block) vs one for unique_ptr.

    std::weak_ptr — Breaking Cycles

    weak_ptr observes a shared_ptr's resource without incrementing the ref count. Used to break circular references:

    struct Node {
    std::shared_ptr<Node> next;
    std::weak_ptr<Node> prev;  // weak to break cycle
};

auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;
b->prev = a;   // if this were shared_ptr, neither node would be freed

    Accessing a weak_ptr

    std::weak_ptr<Widget> weak = some_shared;

if (auto locked = weak.lock()) {
    // locked is a valid shared_ptr
    locked->doWork();
} else {
    // object has been destroyed
}

    Common Pitfalls

    Creating shared_ptr from raw pointer twice

    Widget* raw = new Widget();
std::shared_ptr<Widget> a(raw);
std::shared_ptr<Widget> b(raw);  // ⚠️ Two control blocks → double free!

// Fix: always use make_shared, or copy the shared_ptr
auto a = std::make_shared<Widget>();
auto b = a;  // shares the same control block

    Sharing this

    If a class needs to create shared_ptr<this>, inherit from std::enable_shared_from_this:

    class Server : public std::enable_shared_from_this<Server> {
public:
    std::shared_ptr<Server> getPtr() {
        return shared_from_this();
    }
};

    Circular References

    Two shared_ptrs pointing at each other keep both alive forever. Break the cycle with weak_ptr on at least one side.

    Which Smart Pointer?

    • Exclusive ownership, no sharing → unique_ptr (default choice)
    • Multiple owners, shared lifetime → shared_ptr
    • Observing without owning (cache, back-pointers) → weak_ptr
    • Non-owning access within a single scope → raw pointer or reference

    Prefer make_unique / make_shared

    • Exception-safe (no leak if constructor throws between new and smart pointer construction).
    • make_shared fuses object + control block into one allocation.
    • Cleaner syntax: auto p = std::make_unique<T>(args...).

    Summary

    • Default to unique_ptr — zero overhead, exclusive ownership.
    • Use shared_ptr only when shared ownership is genuinely needed.
    • Use weak_ptr to observe shared resources and break reference cycles.
    • Always use make_unique / make_shared.
    • Never create two shared_ptrs from the same raw pointer.
