    Precision Handling in Geometry

    July 2025 · 16 min read

    When integer arithmetic is not an option — for example, when the problem involves circle intersections, trigonometry, or explicit float output — you must handle floating-point precision carefully. A single mishandled comparison can cascade into a completely wrong answer.

    This post covers the essential precision techniques every competitive programmer should know: epsilon-based comparisons, error propagation, when long double helps, and coordinate tricks that reduce precision pressure.

    Why Precision Matters

    IEEE 754 double has a 52-bit mantissa, giving about 15–16 significant decimal digits. The machine epsilon for double is approximately 2.22 × 10−16 — the smallest value such that 1.0 + ε ≠ 1.0 in floating point.

    In geometry, precision errors manifest as:

    • False negatives: Two lines that should intersect are reported as parallel.
    • False positives: A point slightly outside a polygon is classified as inside.
    • Inconsistent predicates: Point A is left of line L, B is right of L, but the segment AB doesn’t intersect L.
    • Infinite loops: Convex hull algorithms that depend on consistent orientation may loop.

    The diagram shows the three-way classification used in epsilon comparisons. Values within ±EPS of zero are treated as zero. Values clearly below −EPS are negative, and values above +EPS are positive.

    Absolute vs Relative EPS

    There are two standard ways to compare floating-point numbers:

    Absolute epsilon comparison:

    const double EPS = 1e-9;

int sign(double x) {
    if (x > EPS) return 1;
    if (x < -EPS) return -1;
    return 0;
}

bool eq(double a, double b) { return abs(a - b) < EPS; }
bool lt(double a, double b) { return a < b - EPS; }
bool le(double a, double b) { return a < b + EPS; }

    Relative epsilon comparison:

    bool relEq(double a, double b, double relEps = 1e-9) {
    double diff = abs(a - b);
    double maxAB = max(abs(a), abs(b));
    if (maxAB < 1e-15) return diff < relEps;  // both near zero
    return diff / maxAB < relEps;
}

    When to use which:

    • Absolute EPS works well when values are bounded (e.g., coordinates ≤ 104, areas ≤ 108). This covers most CP problems.
    • Relative EPS is necessary when values span many orders of magnitude. Rarely needed in CP but common in industrial geometry.
    • In competitive programming, absolute EPS = 1e-9 is the most common choice. Use 1e-7 for problems with explicit float output and generous tolerance.

    EPS Propagation

    Errors grow with each arithmetic operation. Understanding how they propagate helps you choose the right EPS and detect when precision is too tight.

    • Addition / Subtraction: Absolute error roughly adds. If a has error δa and b has error δb, then a ± b has error δa + δb.
    • Multiplication: Relative error roughly adds. The absolute error of a × b is approximately |a| · δb + |b| · δa.
    • Division: Similar to multiplication but the denominator’s error is amplified when the denominator is small.
    • Square root: sqrt(x) has half the relative error of x. Square roots are actually good for precision.

    Example: Computing the cross product of two vectors with coordinates around 106: 

    // cross = a.x * b.y - a.y * b.x
// Each product ≈ 1e12, machine epsilon ≈ 1e-16
// Rounding error per product ≈ 1e12 * 1e-16 = 1e-4
// Subtraction error ≈ 2e-4
// So cross products of large coordinates have error ~1e-4

    This means using EPS = 1e-9 for the cross product comparison is too tight! You would need EPS ≈ 1e-3 or, much better, use integers.

    Long Double

    C++ offers long double, which on most x86 systems provides 80-bit extended precision (63-bit mantissa, ≈18–19 decimal digits). This gives you roughly 3–4 extra digits compared to double. 

    // Use long double for extra precision
typedef long double ld;
const ld EPS = 1e-12L;

ld cross(ld ax, ld ay, ld bx, ld by) {
    return ax * by - ay * bx;
}

int sign(ld x) {
    if (x > EPS) return 1;
    if (x < -EPS) return -1;
    return 0;
}

    When to use long double:

    • When the problem has tight precision requirements and you cannot use integers.
    • When coordinates are large (106+) and you need cross product comparisons.
    • When chaining many operations (e.g., computing circumcircles, Voronoi diagrams).

    Caveats:

    • On MSVC (Windows), long double is the same as double (64-bit). You only get the benefit on GCC/Linux.
    • Use scanf("%Lf", &x) and printf("%.15Lf", x) for I/O. cin / cout may not handle long double correctly on all compilers.
    • long double is slower than double. For tight time limits, prefer integer techniques.

    Coordinate Techniques

    Several coordinate tricks reduce precision pressure without changing algorithms:

    1. Coordinate compression / translation:

    // Translate so that one key point is at the origin
// This reduces the magnitude of coordinates
void translate(vector<pair<double,double>> &pts) {
    double cx = pts[0].first, cy = pts[0].second;
    for (auto &[x, y] : pts) {
        x -= cx;
        y -= cy;
    }
}

    By translating points near the origin, products have smaller magnitude and thus smaller absolute error.

    2. Scaling to avoid small values:

    // If coordinates are very small (e.g., 1e-6 to 1e-3),
// scale them up to use more mantissa bits
void scaleUp(vector<pair<double,double>> &pts, double factor) {
    for (auto &[x, y] : pts) {
        x *= factor;
        y *= factor;
    }
}

    3. Represent fractions as pairs (numerator, denominator):

    // Exact intersection point as rational numbers
struct Frac {
    long long num, den;
    Frac(long long n, long long d) : num(n), den(d) {
        if (den < 0) { num = -num; den = -den; }
    }
    bool operator<(const Frac &o) const {
        return (__int128)num * o.den < (__int128)o.num * den;
    }
    bool operator==(const Frac &o) const {
        return (__int128)num * o.den == (__int128)o.num * den;
    }
};

    This avoids division entirely. Intersection coordinates are stored as exact rationals, and comparisons use cross-multiplication. This is a powerful technique for sweep line algorithms where precise event ordering is critical.

    Best Practices

    A summary of precision best practices for competitive programming geometry:

    1. Use integers first. If the problem can be solved with integer arithmetic, do so. This eliminates all precision concerns.
    2. Choose EPS based on error analysis. Don’t blindly use 1e-9. Estimate the expected error magnitude from coordinate ranges and operation count.
    3. Minimize operation count. Fewer floating-point operations means less accumulated error. Simplify formulas algebraically before coding.
    4. Avoid subtracting nearly equal values. Rearrange formulas to subtract small from large rather than large from large.
    5. Use long double as a safety net when integer arithmetic is impossible and precision is tight.
    6. Test with adversarial cases. Points that are nearly collinear, nearly coincident, or have coordinates at the extremes of the allowed range.
    7. Read the problem’s output tolerance. If it says “absolute or relative error ≤ 10−6”, you have room. If it says “exact answer”, use integers or rationals.

    Common EPS choices in practice:

    • 1e-9 — default for most problems, coordinates ≤ 104
    • 1e-7 — when coordinates up to 106
    • 1e-6 — generous tolerance (some Codeforces geometry problems)
    • 1e-12 — with long double and moderate coordinates

    A complete precision-safe geometry template: 

    const double EPS = 1e-9;

int sign(double x) { return (x > EPS) - (x < -EPS); }

struct Pt {
    double x, y;
    Pt operator+(Pt o) const { return {x+o.x, y+o.y}; }
    Pt operator-(Pt o) const { return {x-o.x, y-o.y}; }
    Pt operator*(double t) const { return {x*t, y*t}; }
    double dot(Pt o) const { return x*o.x + y*o.y; }
    double cross(Pt o) const { return x*o.y - y*o.x; }
    double norm() const { return sqrt(x*x + y*y); }
    double norm2() const { return x*x + y*y; }
    bool operator<(Pt o) const {
        int cx = sign(x - o.x);
        return cx ? cx < 0 : sign(y - o.y) < 0;
    }
    bool operator==(Pt o) const {
        return sign(x - o.x) == 0 && sign(y - o.y) == 0;
    }
};

int orient(Pt a, Pt b, Pt c) {
    return sign((b - a).cross(c - a));
}

bool onSegment(Pt p, Pt a, Pt b) {
    return sign((p - a).cross(b - a)) == 0
        && sign((p - a).dot(p - b)) <= 0;
}

    This template wraps all comparisons through the sign function, ensuring consistent epsilon handling throughout. The point comparison operators use sign so that points within EPS of each other are treated as equal, which is essential for algorithms like sweep line that depend on point ordering.

    Debugging precision issues: When you get Wrong Answer on a geometry problem and suspect precision, try these steps in order:

    1. Switch from double to long double.
    2. Increase EPS by a factor of 100 (e.g., 1e-9 to 1e-7).
    3. Decrease EPS by a factor of 100.
    4. Replace float comparisons with integer arithmetic where possible.
    5. Print intermediate values and check for unexpected magnitudes.
