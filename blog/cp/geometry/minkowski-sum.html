    Minkowski Sum of Polygons

    July 2025 · 15 min read

    The Minkowski sum is a fundamental operation that combines two shapes by “sweeping” one along the boundary of the other. In competitive programming, it appears in collision detection, robot motion planning, and polygon offsetting problems.

    Definition

    Given two sets A and B in the plane, their Minkowski sum is:

    A ⊕ B = { a + b : a ∈ A, b ∈ B }

    Here a + b means component-wise vector addition: (aₓ + bₓ, aᵧ + bᵧ). Intuitively, place a copy of B at every point of A, and the union of all these copies is A ⊕ B.

    Key facts:

    • The Minkowski sum of two convex polygons with n and m vertices is a convex polygon with at most n + m vertices.
    • If either polygon is non-convex, the result can have up to O(n²m²) vertices.
    • The operation is commutative: A ⊕ B = B ⊕ A.
    • The Minkowski sum of a polygon with a circle of radius r gives the polygon “inflated” by r.

    Geometric Intuition

    Think of A as a room and -B (the reflection of B through the origin) as a robot. The Minkowski sum A ⊕ (-B) gives all positions where the robot’s reference point can be placed such that the robot intersects A.

    Equivalently, two convex polygons P and Q intersect if and only if the origin lies inside P ⊕ (-Q). This is the foundation of the GJK algorithm used in physics engines.

    Construction Algorithm for Convex Polygons

    For two convex polygons, the Minkowski sum can be computed in O(n + m) time by merging their edge vectors sorted by polar angle:

    1. Reorder both polygons to start at their bottommost (then leftmost) vertex, in counter-clockwise order.
    2. Compute the edge vectors for each polygon.
    3. Merge the two sorted sequences of edge vectors by polar angle (like merge in merge-sort).
    4. Build the result polygon by starting at the sum of the two starting vertices and appending edge vectors in merged order.
    // Minkowski sum of two convex polygons
#include <bits/stdc++.h>
using namespace std;

typedef long long ll;
struct Pt {
    ll x, y;
    Pt operator+(const Pt& o) const { return {x+o.x, y+o.y}; }
    Pt operator-(const Pt& o) const { return {x-o.x, y-o.y}; }
    ll cross(const Pt& o) const { return x*o.y - y*o.x; }
};

// Reorder polygon so that it starts at the bottom-most point (CCW)
void reorder(vector<Pt>& P) {
    int n = P.size();
    int pos = 0;
    for (int i = 1; i < n; i++)
        if (P[i].y < P[pos].y || (P[i].y == P[pos].y && P[i].x < P[pos].x))
            pos = i;
    rotate(P.begin(), P.begin() + pos, P.end());
}

vector<Pt> minkowskiSum(vector<Pt> A, vector<Pt> B) {
    reorder(A); reorder(B);
    int n = A.size(), m = B.size();
    // Compute edge vectors
    vector<Pt> eA(n), eB(m);
    for (int i = 0; i < n; i++) eA[i] = A[(i+1)%n] - A[i];
    for (int i = 0; i < m; i++) eB[i] = B[(i+1)%m] - B[i];

    vector<Pt> result;
    result.push_back(A[0] + B[0]);
    int i = 0, j = 0;
    while (i < n || j < m) {
        Pt edge;
        if (i < n && (j >= m || eA[i].cross(eB[j]) > 0))
            edge = eA[i++];
        else if (j < m && (i >= n || eA[i].cross(eB[j]) < 0))
            edge = eB[j++];
        else {
            edge = eA[i] + eB[j];
            i++; j++;
        }
        result.push_back(result.back() + edge);
    }
    result.pop_back(); // last point == first point
    return result;
}

    Complexity: O(n + m) after both polygons are already in counter-clockwise order. The merge step is linear because each edge vector is consumed exactly once.

    Non-Convex Polygons

    When one or both polygons are non-convex, the Minkowski sum becomes significantly more complex:

    • The boundary can have O(n²m²) segments in the worst case.
    • The result may not be a simple polygon — it can have self-intersections that need to be resolved.
    • A common approach is to decompose each polygon into convex parts, compute all pairwise Minkowski sums, and take their union.

    For competitive programming, most problems restrict to convex polygons. If you encounter non-convex inputs, consider:

    1. Convex decomposition: Decompose into O(n) convex polygons using ear clipping or Hertel-Mehlhorn.
    2. Compute Minkowski sums of each pair of convex pieces.
    3. Compute the union of all resulting convex polygons.

    This yields O(n²m²) complexity in the worst case, which is acceptable only for small inputs. 

    // Convex decomposition approach (outline)
vector<vector<Pt>> decomposeConvex(vector<Pt>& P);

vector<Pt> minkowskiNonConvex(vector<Pt>& A, vector<Pt>& B) {
    auto partsA = decomposeConvex(A);
    auto partsB = decomposeConvex(B);
    vector<vector<Pt>> sums;
    for (auto& pa : partsA)
        for (auto& pb : partsB)
            sums.push_back(minkowskiSum(pa, pb));
    // return union of all polygons in sums
    // (polygon union is itself a non-trivial problem)
    return {}; // placeholder
}

    Applications

    The Minkowski sum has diverse applications:

    • Collision detection: Two convex objects P and Q (with Q translated by vector t) overlap if and only if t lies inside P ⊕ (-Q). This reduces collision to a point-in-polygon test.
    • Robot motion planning: The configuration space obstacle for a translating convex robot B among convex obstacles Oᵢ is the Minkowski sum Oᵢ ⊕ (-B). The robot's reference point must avoid these expanded obstacles.
    • Polygon offsetting: Expanding a polygon by radius r is the Minkowski sum with a disk of radius r. For a convex polygon, this rounds the corners.
    • Morphing: The Minkowski sum can interpolate between two shapes: (1-t)A ⊕ tB gives a smooth transition.

    A classic competitive programming problem: “Given two convex polygons, can one be translated to avoid the other?” reduces to checking if the origin is outside the Minkowski sum of one polygon with the negation of the other. 

    // Collision detection via Minkowski sum
bool collides(vector<Pt>& P, vector<Pt>& Q, Pt translation) {
    // Negate Q
    vector<Pt> negQ(Q.size());
    for (int i = 0; i < (int)Q.size(); i++)
        negQ[i] = {-Q[i].x, -Q[i].y};
    // Reverse winding for negated polygon
    reverse(negQ.begin(), negQ.end());

    auto msum = minkowskiSum(P, negQ);

    // Check if 'translation' is inside msum
    int n = msum.size();
    for (int i = 0; i < n; i++) {
        Pt edge = msum[(i+1)%n] - msum[i];
        Pt toPoint = translation - msum[i];
        if (edge.cross(toPoint) < 0) return false;
    }
    return true;
}

    Implementation Tips

    Practical tips for Minkowski sum in contests:

    • Use integer coordinates: The Minkowski sum of two integer-coordinate polygons has integer coordinates. Avoid floating point entirely.
    • CCW ordering is crucial: Both polygons must be in counter-clockwise order. A wrong winding direction gives an incorrect result.
    • Handle collinear edges: When two edge vectors have the same angle (cross product = 0), advance both pointers and add the combined edge. Forgetting this case is a common bug.
    • Bottom-most starting vertex: The algorithm requires starting from the lowest vertex. Ties broken by leftmost. If you start elsewhere, the merge step breaks.
    • Degenerate inputs: A “polygon” with only 1 or 2 vertices (a point or segment) works with the same algorithm — just treat them as degenerate convex polygons. 
    // Verifying correct winding order
ll polygonArea2(vector<Pt>& P) {
    ll area = 0;
    int n = P.size();
    for (int i = 0; i < n; i++)
        area += P[i].cross(P[(i+1)%n]);
    return area; // positive if CCW
}

void ensureCCW(vector<Pt>& P) {
    if (polygonArea2(P) < 0)
        reverse(P.begin(), P.end());
}

    Minkowski Difference

    The Minkowski difference is closely related:

    A ⊖ B = A ⊕ (-B) = { a - b : a ∈ A, b ∈ B }

    This is simply the Minkowski sum of A with the reflection of B through the origin. The key property:

    • A and B intersect if and only if the origin is inside A ⊖ B.
    • The minimum distance between A and B (when they don't intersect) equals the distance from the origin to the boundary of A ⊖ B.
    • The penetration depth (when they do intersect) equals the distance from the origin to the boundary of A ⊖ B in the opposite direction.
    // Minimum distance between two convex polygons via Minkowski difference
double minDistConvex(vector<Pt>& A, vector<Pt>& B) {
    // Compute A ⊖ B = A ⊕ (-B)
    vector<Pt> negB(B.size());
    for (int i = 0; i < (int)B.size(); i++)
        negB[i] = {-B[i].x, -B[i].y};
    reverse(negB.begin(), negB.end());
    ensureCCW(negB);

    auto diff = minkowskiSum(A, negB);

    // Check if origin is inside diff
    int n = diff.size();
    bool inside = true;
    for (int i = 0; i < n; i++) {
        Pt edge = diff[(i+1)%n] - diff[i];
        Pt toOrigin = Pt{0,0} - diff[i];
        if (edge.cross(toOrigin) < 0) { inside = false; break; }
    }
    if (inside) return 0.0; // polygons overlap

    // Find minimum distance from origin to boundary of diff
    double minD = 1e18;
    for (int i = 0; i < n; i++) {
        Pt a = diff[i], b = diff[(i+1)%n];
        // Point-to-segment distance from origin to segment (a, b)
        Pt ab = b - a;
        double t = (double)(Pt{0,0} - a).x * ab.x + (Pt{0,0} - a).y * ab.y;
        t /= (double)(ab.x * ab.x + ab.y * ab.y);
        t = max(0.0, min(1.0, t));
        double px = a.x + t * ab.x, py = a.y + t * ab.y;
        minD = min(minD, sqrt(px*px + py*py));
    }
    return minD;
}

    Competition context: Problems asking for the minimum distance between two convex shapes, or whether two moving convex objects ever collide, often reduce to Minkowski difference computations.

    Practice Problems and Patterns

    Here are common problem types that use Minkowski sums:

    1. “Does translating polygon Q by vector t cause it to overlap with polygon P?” — Check if t is inside P ⊕ (-Q).
    2. “Find all translations where a robot R can fit inside a room C.” — The valid positions form C ⊖ R (Minkowski difference with the room as the “positive” shape).
    3. “Expand polygon P by distance r.” — Compute the Minkowski sum of P with a regular polygon approximating a circle of radius r. For exact results with convex P, offset each edge outward by r and compute the new vertices.
    4. “Two convex polygons are moving with constant velocities. Do they ever collide?” — In the relative frame, one polygon is stationary and the other moves linearly. Check if the moving polygon's path (a line segment) intersects the Minkowski sum. 
    // Check if a line segment intersects a convex polygon
// (used for moving-object collision after Minkowski sum)
bool segmentIntersectsConvex(Pt a, Pt b, vector<Pt>& poly) {
    int n = poly.size();
    // Check if either endpoint is inside
    auto inside = [&](Pt p) -> bool {
        for (int i = 0; i < n; i++) {
            Pt edge = poly[(i+1)%n] - poly[i];
            Pt toP = p - poly[i];
            if (edge.cross(toP) < 0) return false;
        }
        return true;
    };
    if (inside(a) || inside(b)) return true;
    // Check if segment crosses any edge
    for (int i = 0; i < n; i++) {
        Pt c = poly[i], d = poly[(i+1)%n];
        Pt ab = b - a, cd = d - c, ac = c - a;
        ll denom = ab.cross(cd);
        if (denom == 0) continue;
        double t = (double)ac.cross(cd) / denom;
        double s = (double)ac.cross(ab) / denom;
        if (t >= 0 && t <= 1 && s >= 0 && s <= 1) return true;
    }
    return false;
}

    Summary

    The Minkowski sum combines two shapes by vector addition of all point pairs. For convex polygons, it can be computed in O(n + m) time by merging edge vectors sorted by polar angle. The Minkowski difference reduces collision detection to a point-in-polygon test. The operation is central to robot motion planning, polygon offsetting, and moving-object collision detection. In competitive programming, stick to the convex case and use integer arithmetic for robust results.
