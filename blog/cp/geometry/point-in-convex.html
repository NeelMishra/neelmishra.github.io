    Point in Convex Polygon: Binary Search Approach

    July 2025 · 15 min read

    Motivation

    In Part 6, we solved point-in-polygon in O(n) for arbitrary polygons. But when the polygon is convex, we can exploit its structure to answer queries in O(log n) after O(n) preprocessing.

    This is critical when you have many queries against the same polygon — a common pattern in competitive programming. With Q queries on a convex polygon of size n, we go from O(Q·n) to O(Q·log n).

    When to Use This

    • The polygon is guaranteed to be convex (or you've computed the convex hull).
    • You have multiple queries against the same polygon.
    • Constraints like n, Q ≤ 10⁵ make O(n) per query too slow.

    Triangle Fan Decomposition

    Fix one vertex of the convex polygon — say v0. Draw diagonals from v0 to all non-adjacent vertices. This divides the polygon into n−2 triangles:

    • Triangle 0: (v0, v1, v2)
    • Triangle 1: (v0, v2, v3)
    • Triangle k: (v0, vk+1, vk+2)

    The key insight: these triangles form angular wedges around v0, and because the polygon is convex, the wedges are sorted by angle. This means we can binary search for the wedge containing our query point.

    The convex polygon is decomposed into a fan of triangles from v₀. The query point q lies in the highlighted wedge (v₀, v₃, v₄).

    Binary Search on Wedges

    The algorithm proceeds in three steps:

    Step 1: Preliminary Checks

    Before binary searching, verify that the query point is within the angular range of the polygon as seen from v0:

    • Check that q is on the left side of edge (v0, v1).
    • Check that q is on the right side of edge (v0, vn−1).

    If either check fails, the point is outside.

    Step 2: Find the Wedge

    Binary search on the diagonals (v0, vk) to find the wedge that contains the query point's angle:

    // cross product of vectors (b-a) and (c-a)
ll cross(Point a, Point b, Point c) {
    return (ll)(b.x - a.x) * (c.y - a.y)
         - (ll)(b.y - a.y) * (c.x - a.x);
}

// Binary search for the wedge index
// Returns k such that q is between rays v0->v[k] and v0->v[k+1]
int find_wedge(Point q, vector<Point>& poly) {
    int n = poly.size();
    int lo = 1, hi = n - 1;
    while (hi - lo > 1) {
        int mid = (lo + hi) / 2;
        if (cross(poly[0], poly[mid], q) >= 0)
            lo = mid;
        else
            hi = mid;
    }
    return lo;
}

    Step 3: Triangle Containment

    Once we find wedge k, check if q lies inside triangle (v0, vk, vk+1) using a cross product test:

    // Point in triangle test using cross products
bool in_triangle(Point a, Point b, Point c, Point q) {
    ll d1 = cross(a, b, q);
    ll d2 = cross(b, c, q);
    ll d3 = cross(c, a, q);
    bool has_neg = (d1 < 0) || (d2 < 0) || (d3 < 0);
    bool has_pos = (d1 > 0) || (d2 > 0) || (d3 > 0);
    return !(has_neg && has_pos);
}
    Boundary handling: The cross product being exactly 0 means the point is on the triangle edge. Whether this counts as "inside" depends on the problem — handle accordingly.

    Complete Implementation

    Here is the full contest-ready solution. The polygon must be convex and given in counter-clockwise order.

    #include <bits/stdc++.h>
using namespace std;

typedef long long ll;
struct Point { ll x, y; };

ll cross(Point O, Point A, Point B) {
    return (A.x - O.x) * (ll)(B.y - O.y)
         - (A.y - O.y) * (ll)(B.x - O.x);
}

// Returns: 1 = inside, 0 = boundary, -1 = outside
// Polygon must be convex, counter-clockwise, n >= 3
int point_in_convex(Point q, vector<Point>& poly) {
    int n = poly.size();
    // Check if q is on the correct side of first and last edges from v0
    ll c1 = cross(poly[0], poly[1], q);
    ll cn = cross(poly[0], poly[n - 1], q);

    if (c1 < 0 || cn > 0) return -1;  // outside angular range

    // Special case: q on edge (v0, v1) or (v0, v_{n-1})
    if (c1 == 0) {
        // q is on ray v0->v1
        if (q.x < min(poly[0].x, poly[1].x) ||
            q.x > max(poly[0].x, poly[1].x) ||
            q.y < min(poly[0].y, poly[1].y) ||
            q.y > max(poly[0].y, poly[1].y))
            return -1;
        return 0;
    }
    if (cn == 0) {
        if (q.x < min(poly[0].x, poly[n-1].x) ||
            q.x > max(poly[0].x, poly[n-1].x) ||
            q.y < min(poly[0].y, poly[n-1].y) ||
            q.y > max(poly[0].y, poly[n-1].y))
            return -1;
        return 0;
    }

    // Binary search for the wedge
    int lo = 1, hi = n - 1;
    while (hi - lo > 1) {
        int mid = (lo + hi) / 2;
        if (cross(poly[0], poly[mid], q) >= 0)
            lo = mid;
        else
            hi = mid;
    }

    // Check if q is inside triangle (poly[0], poly[lo], poly[lo+1])
    ll c = cross(poly[lo], poly[lo + 1], q);
    if (c > 0) return 1;   // strictly inside
    if (c == 0) return 0;   // on the edge
    return -1;              // outside
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;
    vector<Point> poly(n);
    for (int i = 0; i < n; i++) cin >> poly[i].x >> poly[i].y;

    while (q--) {
        Point p;
        cin >> p.x >> p.y;
        int res = point_in_convex(p, poly);
        if (res == 1)       cout << "INSIDE\n";
        else if (res == 0)  cout << "BOUNDARY\n";
        else                cout << "OUTSIDE\n";
    }
}

    Complexity Analysis

    • Preprocessing: O(n) — just read the polygon (it must already be convex and sorted).
    • Per query: O(log n) — one binary search + constant cross product checks.
    • Space: O(n) for storing the polygon.

    The binary search approach vastly outperforms linear scan for large convex polygons.

    Applications & Extensions

    Common Applications

    • Convex hull + queries: Compute the convex hull of a point set, then answer containment queries in O(log n) each.
    • Half-plane intersection: After computing the intersection polygon (always convex), use this for fast point queries.
    • Dynamic convex hull: Maintain a convex hull with insertions and answer containment queries online.

    Handling Clockwise Polygons

    If your polygon is in clockwise order, simply reverse the sign of all cross product comparisons, or reverse the polygon first:

    // Check orientation: if clockwise, reverse to make CCW
ll area2 = 0;
for (int i = 0; i < n; i++)
    area2 += cross({0,0}, poly[i], poly[(i+1)%n]);
if (area2 < 0) reverse(poly.begin(), poly.end());

    What If the Polygon Isn't Convex?

    This algorithm only works for convex polygons. For general polygons:

    • Use the O(n) ray casting or winding number from Part 6.
    • Decompose into convex parts and test each (but this is rarely worth it in competitions).

    Practice Problems

    • CSES — Point in Polygon: Works with both O(n) and O(log n) approaches.
    • Codeforces 166B — Polygons: Check if all vertices of polygon B are inside convex polygon A.
    • Codeforces 87E — Mogohu-Ree: Convex polygon containment with large Q.

    Competition Tips

    • Always normalize to CCW order before applying the algorithm.
    • Watch for the edge case n = 3 (just one triangle, no binary search needed).
    • Remember: if coordinates are integers up to 10⁹, the cross product can reach ~10¹⁸. Use long long.
    • The CSES problem "Polygon Lattice Points" also benefits from this technique for fast boundary detection.

    Step-by-Step Walkthrough

    Let's trace through a concrete example to solidify the algorithm.

    Example Polygon

    Consider a convex polygon with vertices (in CCW order):

    // v0=(0,0), v1=(4,0), v2=(6,3), v3=(4,6), v4=(1,5)
// Query point q=(3,2)

// Step 1: Check angular range
// cross(v0, v1, q) = cross((0,0), (4,0), (3,2))
//   = (4-0)*(2-0) - (0-0)*(3-0) = 8  > 0 ✓
// cross(v0, v4, q) = cross((0,0), (1,5), (3,2))
//   = (1-0)*(2-0) - (5-0)*(3-0) = 2 - 15 = -13  < 0 ✓

// Step 2: Binary search on wedges
// lo=1, hi=4
// mid=2: cross(v0, v2, q) = cross((0,0), (6,3), (3,2))
//   = 6*2 - 3*3 = 3  >= 0 → lo=2
// mid=3: cross(v0, v3, q) = cross((0,0), (4,6), (3,2))
//   = 4*2 - 6*3 = -10  < 0 → hi=3
// Now lo=2, hi=3, hi-lo=1 → done, wedge k=2

// Step 3: Check triangle (v0, v2, v3) = (0,0), (6,3), (4,6)
// cross(v2, v3, q) = (4-6)*(2-3) - (6-3)*(3-6)
//   = (-2)*(-1) - (3)*(-3) = 2 + 9 = 11  > 0 → INSIDE ✓

    Visualization of the Search

    Binary search identifies wedge (v₀, v₂, v₃) as the one containing q. A final cross product confirms q is inside the triangle.

    Handling Degenerate Cases

    • Point exactly on v₀: The boundary check (c1 == 0 or cn == 0) catches this.
    • Point on a diagonal: The final cross product is zero → report BOUNDARY.
    • Point beyond the polygon's angular range: The preliminary c1/cn checks immediately return OUTSIDE.
    • All vertices collinear (degenerate polygon): n ≥ 3 is required. Validate input.
    Key takeaway: For convex polygons with multiple queries, always prefer the O(log n) binary search approach. The implementation is only ~30 lines and the speedup is dramatic.
