    Point in Polygon: Ray Casting & Winding Number

    July 2025 · 18 min read

    Problem Statement

    Given a simple polygon P with n vertices and a query point q, determine whether q lies inside, outside, or on the boundary of P. This is one of the most fundamental queries in computational geometry and appears frequently in competitive programming.

    Two classical approaches solve this problem in O(n) per query for arbitrary simple polygons:

    • Ray Casting (also called the crossing number test)
    • Winding Number

    Both work for non-convex polygons, but they differ in how they handle edge cases and what mathematical invariant they exploit.

    A ray cast from q₁ crosses the polygon boundary an odd number of times, confirming it is inside.

    Ray Casting Algorithm

    The idea is beautifully simple: shoot a ray from q in any direction (typically the positive x-direction) and count how many times it crosses the polygon boundary.

    • Odd crossings → point is inside
    • Even crossings → point is outside

    This works because every time the ray enters the polygon it must eventually exit, adding two crossings. A point inside starts with one "entry" crossing that is never paired.

    Algorithm Steps

    1. Initialize a crossing counter to 0.
    2. For each edge (vi, vi+1) of the polygon:
    3. Check if the horizontal ray from q intersects this edge.
    4. An intersection occurs when q.y is between vi.y and vi+1.y, and the x-coordinate of the intersection is to the right of q.x.
    5. If odd crossings, return INSIDE; otherwise OUTSIDE.

    Handling the Ray-Edge Intersection

    For a horizontal ray going right from point q = (qx, qy), and edge from A = (ax, ay) to B = (bx, by):

    The ray intersects the edge if qy is strictly between ay and by (using a half-open interval to handle vertices), and the x-coordinate of the intersection point is greater than qx.

    // Ray casting: returns true if point (qx,qy) is inside polygon
// Polygon given as vector of points, 0-indexed, closed automatically
bool ray_cast(double qx, double qy,
              const vector<pair<double,double>>& poly) {
    int n = poly.size();
    bool inside = false;
    for (int i = 0, j = n - 1; i < n; j = i++) {
        double xi = poly[i].first, yi = poly[i].second;
        double xj = poly[j].first, yj = poly[j].second;
        // Half-open interval on y: include lower endpoint, exclude upper
        if ((yi > qy) != (yj > qy)) {
            // x-coordinate of intersection
            double x_int = xi + (qy - yi) / (yj - yi) * (xj - xi);
            if (qx < x_int) {
                inside = !inside;
            }
        }
    }
    return inside;
}
    Half-open interval trick: By using (yi > qy) != (yj > qy), we count a vertex exactly once when the ray passes through it. This elegantly avoids double-counting at polygon vertices.

    Complexity

    O(n) per query, where n is the number of polygon vertices. No preprocessing required.

    Winding Number Method

    The winding number counts how many times the polygon winds around the query point. For a simple polygon:

    • Winding number ≠ 0 → point is inside
    • Winding number = 0 → point is outside

    Unlike ray casting, the winding number is well-defined for self-intersecting polygons too, making it more robust in certain scenarios.

    The winding number counts how many times the polygon boundary wraps around the query point.

    Computing the Winding Number

    We iterate over edges and track upward/downward crossings of the horizontal line through q. An upward crossing increments the winding number; a downward crossing decrements it.

    The key test uses the cross product to determine which side of the edge the point lies on:

    // Returns the winding number of point q with respect to polygon
int winding_number(double qx, double qy,
                   const vector<pair<double,double>>& poly) {
    int n = poly.size();
    int wn = 0;
    for (int i = 0; i < n; i++) {
        double x1 = poly[i].first,       y1 = poly[i].second;
        double x2 = poly[(i+1)%n].first, y2 = poly[(i+1)%n].second;
        if (y1 <= qy) {
            if (y2 > qy) {
                // Upward crossing
                double cross = (x2 - x1) * (qy - y1)
                             - (qx - x1) * (y2 - y1);
                if (cross > 0) wn++;  // point is left of edge
            }
        } else {
            if (y2 <= qy) {
                // Downward crossing
                double cross = (x2 - x1) * (qy - y1)
                             - (qx - x1) * (y2 - y1);
                if (cross < 0) wn--;  // point is right of edge
            }
        }
    }
    return wn;
}

// Usage: if (winding_number(qx, qy, poly) != 0) => inside

    Why Cross Product?

    The cross product (B−A) × (Q−A) tells us which side of the directed edge AB the point Q lies on:

    • Positive → Q is to the left
    • Negative → Q is to the right
    • Zero → Q is on the edge

    Edge Cases & Boundary Detection

    In competitive programming, problems often ask you to distinguish between inside, outside, and on the boundary. Neither basic algorithm naturally handles the boundary case, so we add a separate check.

    Point on Segment Test

    A point q lies on segment AB if:

    1. The cross product (B−A) × (Q−A) = 0 (collinear)
    2. The dot product (Q−A) · (B−A) is between 0 and |B−A|² (between endpoints)
    typedef long long ll;
struct Point { ll x, y; };

ll cross(Point O, Point A, Point B) {
    return (A.x - O.x) * (ll)(B.y - O.y)
         - (A.y - O.y) * (ll)(B.x - O.x);
}

bool on_segment(Point p, Point a, Point b) {
    if (cross(a, b, p) != 0) return false;
    return min(a.x, b.x) <= p.x && p.x <= max(a.x, b.x)
        && min(a.y, b.y) <= p.y && p.y <= max(a.y, b.y);
}

    Common Pitfalls

    • Ray through a vertex: The half-open interval trick in ray casting handles this. Without it, a ray grazing a vertex may be counted twice.
    • Ray along an edge: If the ray is collinear with an edge, the half-open convention ensures zero crossings for that edge. The separate boundary check catches this case.
    • Floating-point precision: Use integer arithmetic when coordinates are integers. For the cross product test, long long avoids precision issues entirely.
    • Degenerate polygons: Polygons with zero-length edges or repeated vertices can break both algorithms. Preprocess to remove degeneracies.

    When a ray passes exactly through a vertex, careful counting rules prevent errors.

    Comparison of Approaches

    For competitive programming, ray casting is used more often due to its simplicity. The winding number is preferred in production software and when dealing with complex polygon topologies.

    Full Implementation

    Here is a complete, contest-ready implementation that handles all three cases: inside, outside, and boundary.

    #include <bits/stdc++.h>
using namespace std;

typedef long long ll;
struct Point { ll x, y; };

ll cross(Point O, Point A, Point B) {
    return (A.x - O.x) * (B.y - O.y)
         - (A.y - O.y) * (B.x - O.x);
}

bool on_segment(Point p, Point a, Point b) {
    if (cross(a, b, p) != 0) return false;
    return min(a.x, b.x) <= p.x && p.x <= max(a.x, b.x)
        && min(a.y, b.y) <= p.y && p.y <= max(a.y, b.y);
}

// Returns: 1 = inside, 0 = boundary, -1 = outside
int point_in_polygon(Point q, vector<Point>& poly) {
    int n = poly.size();
    // Boundary check
    for (int i = 0; i < n; i++) {
        if (on_segment(q, poly[i], poly[(i + 1) % n]))
            return 0;
    }
    // Ray casting
    int crossings = 0;
    for (int i = 0, j = n - 1; i < n; j = i++) {
        ll yi = poly[i].y, yj = poly[j].y;
        if ((yi > q.y) != (yj > q.y)) {
            // Compute x-intersection using cross product to stay in integers
            // We check: q.x < poly[i].x + (q.y - yi) * (xj - xi) / (yj - yi)
            // Rearranged to avoid division:
            ll dx = poly[j].x - poly[i].x;
            ll dy = yj - yi;
            ll t = (q.y - yi);
            // Check sign: (q.x - poly[i].x) * dy < t * dx
            // Need to be careful with sign of dy
            ll lhs = (q.x - poly[i].x) * dy;
            ll rhs = t * dx;
            if (dy > 0) {
                if (lhs < rhs) crossings++;
            } else {
                if (lhs > rhs) crossings++;
            }
        }
    }
    return (crossings & 1) ? 1 : -1;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;
    vector<Point> poly(n);
    for (int i = 0; i < n; i++) cin >> poly[i].x >> poly[i].y;

    while (q--) {
        Point p;
        cin >> p.x >> p.y;
        int res = point_in_polygon(p, poly);
        if (res == 1)       cout << "INSIDE\n";
        else if (res == 0)  cout << "BOUNDARY\n";
        else                cout << "OUTSIDE\n";
    }
}

    Key Implementation Notes

    • We use integer arithmetic throughout to avoid floating-point errors. The cross product and the ray intersection test are rearranged to use only multiplication and comparison.
    • The boundary check is done first as a separate pass. This keeps the ray casting logic clean.
    • The polygon is assumed to be given in order (clockwise or counter-clockwise). No sorting needed.

    Practice Problems & Tips

    Competition Tips

    • Always check boundary first. Many problems treat boundary points differently from interior points.
    • Prefer integer coordinates. If the problem gives integers, keep everything in long long.
    • For multiple queries on a convex polygon, use the O(log n) binary search method (covered in Part 7) instead.
    • Some problems give polygons with collinear vertices. Both algorithms still work, but make sure your on_segment test handles zero-length edges gracefully.

    Classic Problems

    • CSES — Point in Polygon: The standard problem. Test your implementation here.
    • Codeforces 166B — Polygons: Check if one polygon is inside another.
    • UVa 11030 — Predator II: Multiple polygon containment queries.

    Optimizing for Multiple Queries

    When you have Q queries on the same polygon, each approach costs O(Q·n) total. For large inputs this can be a bottleneck. Several strategies help:

    Grid-Based Acceleration

    Overlay a grid on the bounding box of the polygon. For each cell, precompute whether it is entirely inside, entirely outside, or on the boundary. Queries falling in "all-inside" or "all-outside" cells are answered in O(1).

    // Grid acceleration sketch
struct Grid {
    int rows, cols;
    double x0, y0, cellW, cellH;
    vector<vector<int>> status; // 1=inside, -1=outside, 0=mixed

    int query(double qx, double qy, vector<Point>& poly) {
        int r = (int)((qy - y0) / cellH);
        int c = (int)((qx - x0) / cellW);
        if (r < 0 || r >= rows || c < 0 || c >= cols)
            return -1; // outside bounding box
        if (status[r][c] != 0) return status[r][c];
        // Fall back to exact test for mixed cells
        return point_in_polygon({(ll)qx, (ll)qy}, poly);
    }
};

    Convex Decomposition

    Decompose the polygon into convex parts. For each part, use the O(log n) binary search approach from Part 7. This gives O(k · log n) per query, where k is the number of convex pieces.

    When to Use Which

    • Single query: Ray casting — simplest, O(n).
    • Few queries, general polygon: Ray casting with boundary check.
    • Many queries, convex polygon: Binary search — O(log n) per query.
    • Many queries, general polygon: Grid acceleration or convex decomposition.
    • Self-intersecting polygon: Winding number is the only correct choice.

    Grid acceleration: cells fully inside or outside are answered in O(1); mixed cells fall back to exact testing.

    Complexity Summary: Both ray casting and winding number run in O(n) per query with no preprocessing. For convex polygons, O(log n) is possible — see the next post in this series.
