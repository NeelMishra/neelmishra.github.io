    Integer Geometry: Avoiding Floats

    July 2025 · 15 min read

    One of the most important lessons in competitive programming geometry is: avoid floating point whenever possible. Integer arithmetic is exact, deterministic, and fast. Many geometry operations — orientation tests, area comparisons, convex hull construction — can be performed entirely with integers.

    In this post we explore the core techniques for integer-only geometry, common pitfalls when contestants reach for double prematurely, and practical recipes for the most frequently needed operations.

    The Floating Point Problem

    IEEE 754 double provides approximately 15–16 significant decimal digits. This sounds like a lot, but geometry computations can amplify errors dramatically:

    • Subtraction of similar values causes catastrophic cancellation.
    • Chained operations accumulate rounding errors.
    • Comparisons with zero (e.g., orientation tests) become unreliable.
    • Non-associativity: (a + b) + c ≠ a + (b + c) in floating point.

    Consider testing whether three points are collinear. With floating point, the cross product might be 1e-14 instead of exactly zero. Is that collinear? You are forced into epsilon comparisons, and the choice of epsilon can make or break your solution.

    The left grid shows integer geometry: three collinear points produce a cross product of exactly zero. On the right, floating-point arithmetic introduces uncertainty halos around each point, making the collinearity test unreliable.

    Integer Cross Product

    The 2D cross product (or the z-component of the 3D cross product of 2D vectors) is the foundation of integer geometry: 

    long long cross(long long ax, long long ay, long long bx, long long by) {
    return ax * by - ay * bx;
}

// Orientation of triangle (O, A, B):
// positive = counter-clockwise, negative = clockwise, zero = collinear
long long orient(long long ox, long long oy,
                 long long ax, long long ay,
                 long long bx, long long by) {
    return cross(ax - ox, ay - oy, bx - ox, by - oy);
}

    Overflow analysis: If coordinates are at most C, then each difference is at most 2C, and the cross product is at most 4C². For C = 109, this is 4 × 1018, which fits in long long (max ≈ 9.2 × 1018) — but just barely. For C = 109 you’re safe. For C > 2 × 109, consider __int128.

    The key insight: orientation tests never need floating point. The sign of the cross product tells you left/right/collinear exactly.

    Distance Comparison

    A frequent mistake is computing sqrt(dx*dx + dy*dy) to compare distances. You almost never need the actual distance — you need to compare distances. Since sqrt is monotonic, comparing squared distances gives the same result: 

    // Compare dist(A,B) vs dist(C,D) without sqrt
long long dist2(long long x1, long long y1, long long x2, long long y2) {
    long long dx = x1 - x2, dy = y1 - y2;
    return dx * dx + dy * dy;
}

// Returns negative if AB < CD, zero if equal, positive if AB > CD
long long cmpDist(long long ax, long long ay, long long bx, long long by,
                  long long cx, long long cy, long long dx, long long dy) {
    return dist2(ax, ay, bx, by) - dist2(cx, cy, dx, dy);
}

    Overflow check: With coordinates up to 109, dx is up to 2 × 109, and dx*dx is up to 4 × 1018. The sum of two squares fits in long long. The difference of two such sums also fits (at most 8 × 1018).

    Other common operations that can stay integer:

    • Twice the signed area: cross(B-A, C-A) gives exactly twice the signed area of triangle ABC.
    • Point on segment: Use orientation + range checks, all integer.
    • Segment intersection test: Four orientation tests, no division needed.
    • Polygon area: The shoelace formula gives twice the area as an integer.

    Integer Convex Hull

    Convex hull algorithms (Graham scan, Andrew’s monotone chain) only need orientation tests and coordinate comparisons — both are exact with integers. 

    struct Pt {
    long long x, y;
    bool operator<(const Pt &o) const {
        return x < o.x || (x == o.x && y < o.y);
    }
};

long long cross(Pt O, Pt A, Pt B) {
    return (A.x - O.x) * (long long)(B.y - O.y)
         - (A.y - O.y) * (long long)(B.x - O.x);
}

vector<Pt> convexHull(vector<Pt> pts) {
    int n = pts.size(), k = 0;
    if (n < 3) return pts;
    sort(pts.begin(), pts.end());
    vector<Pt> H(2 * n);

    // Lower hull
    for (int i = 0; i < n; i++) {
        while (k >= 2 && cross(H[k-2], H[k-1], pts[i]) <= 0) k--;
        H[k++] = pts[i];
    }
    // Upper hull
    for (int i = n - 2, t = k + 1; i >= 0; i--) {
        while (k >= t && cross(H[k-2], H[k-1], pts[i]) <= 0) k--;
        H[k++] = pts[i];
    }
    H.resize(k - 1);
    return H;
}

    This implementation is completely float-free. The sort uses integer comparisons, the cross product is integer, and no division or square root is ever performed. This is the gold standard for CP convex hull implementations.

    Techniques and Tricks

    Here are additional integer geometry recipes for competitive programming:

    1. Point in triangle (integer):

    bool inTriangle(Pt A, Pt B, Pt C, Pt P) {
    long long d1 = cross(A, B, P);
    long long d2 = cross(B, C, P);
    long long d3 = cross(C, A, P);
    bool hasNeg = (d1 < 0) || (d2 < 0) || (d3 < 0);
    bool hasPos = (d1 > 0) || (d2 > 0) || (d3 > 0);
    return !(hasNeg && hasPos);
}

    2. Segment intersection test (integer):

    bool onSegment(Pt p, Pt a, Pt b) {
    return min(a.x, b.x) <= p.x && p.x <= max(a.x, b.x)
        && min(a.y, b.y) <= p.y && p.y <= max(a.y, b.y);
}

bool segmentsIntersect(Pt a, Pt b, Pt c, Pt d) {
    long long d1 = cross(c, d, a), d2 = cross(c, d, b);
    long long d3 = cross(a, b, c), d4 = cross(a, b, d);
    if (((d1 > 0 && d2 < 0) || (d1 < 0 && d2 > 0)) &&
        ((d3 > 0 && d4 < 0) || (d3 < 0 && d4 > 0)))
        return true;
    if (d1 == 0 && onSegment(a, c, d)) return true;
    if (d2 == 0 && onSegment(b, c, d)) return true;
    if (d3 == 0 && onSegment(c, a, b)) return true;
    if (d4 == 0 && onSegment(d, a, b)) return true;
    return false;
}

    3. Integer angle comparison: Instead of computing atan2, compare angles by quadrant and cross product sign.

    int quadrant(long long x, long long y) {
    if (x > 0 && y >= 0) return 0;
    if (x <= 0 && y > 0) return 1;
    if (x < 0 && y <= 0) return 2;
    return 3;
}

bool angleLess(Pt a, Pt b) {
    int qa = quadrant(a.x, a.y), qb = quadrant(b.x, b.y);
    if (qa != qb) return qa < qb;
    long long cr = a.x * (long long)b.y - a.y * (long long)b.x;
    return cr > 0;
}

    Common Pitfalls

    Even experienced contestants fall into these traps:

    • Overflow on multiplication: Always use long long for products of coordinates. A common bug is writing int a = x1 * y2 where the product exceeds 231.
    • Forgetting the 2× factor: The cross product gives twice the signed area. If the problem asks for the actual area, divide at the end (or compare 2*area values directly).
    • Using double for sorting: Sorting points by angle using atan2 can produce different orderings for nearly equal angles. Use integer angle comparison instead.
    • Mixing int and long long: In C++, (int)(1000000000) * (int)(1000000000) overflows before the assignment to long long. Cast at least one operand first.
    • Assuming sqrt is needed: 90% of distance operations in CP only require comparisons, which work fine with squared distances.

    Rule of thumb: When writing geometry code for a contest, start by asking “Can I solve this with integers only?” If the answer is yes (and it usually is), you have eliminated an entire class of bugs.

    __int128 for extreme ranges: When coordinates can be up to 1018 (rare but it happens), even long long overflows on cross products. GCC supports __int128 which handles values up to approximately 1.7 × 1038: 

    typedef __int128 lll;

lll cross128(long long ax, long long ay, long long bx, long long by) {
    return (lll)ax * by - (lll)ay * bx;
}

// Note: __int128 has no built-in I/O in C++.
// Read as long long, compute in __int128, output the sign.

    4. Polygon area (Shoelace, integer):

    // Returns TWICE the signed area to keep everything integer
long long twiceArea(vector<Pt> &poly) {
    long long area = 0;
    int n = poly.size();
    for (int i = 0; i < n; i++) {
        int j = (i + 1) % n;
        area += poly[i].x * (long long)poly[j].y;
        area -= poly[j].x * (long long)poly[i].y;
    }
    return area; // positive if CCW, negative if CW
}

// To get the actual area: abs(twiceArea(poly)) / 2.0
// But for comparisons, just compare abs(twiceArea) values!

    5. Integer GCD for lattice problems:

    // Number of lattice points on segment from (x1,y1) to (x2,y2)
// (including endpoints) = gcd(|dx|, |dy|) + 1
long long latticeOnSegment(long long x1, long long y1,
                           long long x2, long long y2) {
    return __gcd(abs(x2 - x1), abs(y2 - y1)) + 1;
}

// Pick's theorem: A = I + B/2 - 1
// where A = area, I = interior lattice points, B = boundary points
// Since 2A = twiceArea and B = sum of gcd(|dx|,|dy|) over edges:
// I = (twiceArea - B + 2) / 2

    6. Comparing slopes without division:

    // Instead of computing dy/dx as a float, compare slopes by
// cross-multiplication:
// slope(A,B) < slope(C,D)  iff
// (B.y-A.y)*(D.x-C.x) < (D.y-C.y)*(B.x-A.x)
// (assuming positive dx for both)
bool slopeLess(Pt a, Pt b, Pt c, Pt d) {
    long long lhs = (b.y - a.y) * (long long)(d.x - c.x);
    long long rhs = (d.y - c.y) * (long long)(b.x - a.x);
    return lhs < rhs;
}

    Summary: The integer geometry toolkit — cross product, squared distance, twice-area, slope comparison via cross-multiplication, and lattice-point counting — handles the vast majority of competitive programming geometry without ever touching a double. Master these techniques and you will avoid hundreds of hours of precision debugging over your competitive programming career.

    When you must use floats: If the problem explicitly asks for a floating-point answer (e.g., “print the circumradius with 9 decimal places”), do all intermediate computation in integers as long as possible, and convert to double only at the very end. This minimizes accumulated error.
