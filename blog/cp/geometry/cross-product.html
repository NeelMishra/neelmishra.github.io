The Cross Product: Orientation and Area

July 2025 · 11 min read

If there's one operation you must master for computational geometry, it's the 2-D cross product. It tells you orientation, gives you area, and underpins nearly every algorithm in this series. This post is a deep dive.

Cross Product Definition

Given two 2-D vectors u = (ux, uy) and v = (vx, vy), the 2-D cross product is the scalar:

u × v = ux · vy − uy · vx

Geometrically this equals |u| · |v| · sin(θ), where θ is the signed angle from u to v. 

// Scalar 2-D cross product
long long cross(Pt u, Pt v) {
    return (long long)u.x * v.y - (long long)u.y * v.x;
}

// Cross product of vectors OA and OB (useful for orientation)
long long cross(Pt o, Pt a, Pt b) {
    return cross(a - o, b - o);
}

Why long long? If coordinates are up to 109, the product of two coordinates can reach 1018, which overflows int but fits in long long. Always cast before multiplying.

Orientation Test

Given three points A, B, C, the sign of cross(A, B, C) tells us how C relates to the directed line from A to B:

  • Positive (+): C is to the left of AB ⇒ the triplet is counter-clockwise (CCW).
  • Negative (−): C is to the right ⇒ clockwise (CW).
  • Zero: A, B, C are collinear.
// Returns +1 (CCW), -1 (CW), 0 (collinear)
int orient(Pt a, Pt b, Pt c) {
    long long v = cross(a, b, c);
    if (v > 0) return +1;
    if (v < 0) return -1;
    return 0;
}

This test runs in O(1) and uses only integer multiplication and subtraction—no division, no square roots, no trigonometry. It is the workhorse of computational geometry.

Signed Area of a Triangle

The cross product also gives us the signed area of the triangle formed by three points:

signed_area(A, B, C) = cross(A, B, C) / 2

  • Positive if the vertices are in CCW order.
  • Negative if CW.
  • Zero if degenerate (collinear).
// Twice the signed area (avoids division for integers)
long long twiceArea(Pt a, Pt b, Pt c) {
    return cross(a, b, c);
}

// Signed area as a double
double signedArea(Pt a, Pt b, Pt c) {
    return cross(a, b, c) / 2.0;
}

// Unsigned area
double area(Pt a, Pt b, Pt c) {
    return fabs(cross(a, b, c)) / 2.0;
}

Competition tip: Work with twice the area when you can. If the problem only asks you to compare areas or check if area is zero, you avoid the division by 2 and keep everything in integers.

Collinearity Check

Three points are collinear if and only if cross(A, B, C) == 0. For floating-point coordinates, use an EPS check. 

bool collinear(Pt a, Pt b, Pt c) {
    return cross(a, b, c) == 0; // integer version
}

bool collinearEps(Point a, Point b, Point c) {
    return fabs(cross(b - a, c - a)) < EPS;
}

A related check: is point P on segment AB?

bool onSegment(Pt p, Pt a, Pt b) {
    if (cross(a, b, p) != 0) return false;
    return min(a.x, b.x) <= p.x && p.x <= max(a.x, b.x)
        && min(a.y, b.y) <= p.y && p.y <= max(a.y, b.y);
}

The collinearity check via cross product is the first filter. The bounding-box check is the second. Both are O(1).

Applications in CP

The cross product appears everywhere. Here are its major uses:

1. Convex Hull

Both Graham Scan and Andrew's Monotone Chain use the orientation test to decide whether to include or remove a point from the hull. We'll cover this in detail in Part 4.

2. Polygon Area (Shoelace Formula)

The shoelace formula is simply a sum of cross products over consecutive edges: 

double polygonArea(vector<Point> &poly) {
    double area = 0;
    int n = poly.size();
    for (int i = 0; i < n; i++) {
        int j = (i + 1) % n;
        area += cross(poly[i], poly[j]);
    }
    return fabs(area) / 2.0;
}

Detailed in Part 5.

3. Segment Intersection

As seen in Part 2, the orientation test determines whether two segments cross.

4. Point-in-Polygon

The winding number algorithm uses the cross product to determine whether a point is left or right of each edge.

5. Sorting by Polar Angle

Instead of computing atan2, sort points by comparing their cross products. This is faster and avoids floating-point errors: 

// Sort by polar angle around origin
sort(pts.begin(), pts.end(), [](Pt &a, Pt &b) {
    int ha = (a.y > 0 || (a.y == 0 && a.x > 0)) ? 0 : 1;
    int hb = (b.y > 0 || (b.y == 0 && b.x > 0)) ? 0 : 1;
    if (ha != hb) return ha < hb;
    long long c = (long long)a.x * b.y - (long long)a.y * b.x;
    return c > 0;
});

Common Pitfalls

Here are mistakes competitors frequently make with the cross product:

1. Integer overflow

If coordinates are up to 109, the cross product can reach 2 × 1018. Always use long long and cast before the multiplication: 

// WRONG — overflows with int
int cross_bad(int ax, int ay, int bx, int by) {
    return ax * by - ay * bx;
}

// CORRECT
long long cross_ok(long long ax, long long ay, long long bx, long long by) {
    return ax * by - ay * bx;
}

2. Confusing CW and CCW

The sign convention depends on the coordinate system. In the standard math convention (y-axis up), positive cross product means CCW. In screen coordinates (y-axis down), the meaning is reversed. Always verify with a simple example.

3. Forgetting the collinear case

When the cross product is zero, the points are collinear. Many algorithms need special handling here: convex hull must decide whether to keep collinear boundary points, segment intersection must check bounding boxes, etc.

4. EPS with floating-point cross product

When using double, never compare the cross product to exactly zero. Use an epsilon threshold: 

int orient_eps(Point a, Point b, Point c) {
    double v = (b.x - a.x) * (c.y - a.y) - (b.y - a.y) * (c.x - a.x);
    if (v > EPS) return +1;
    if (v < -EPS) return -1;
    return 0;
}
