    3D Convex Hull

    July 2025 · 18 min read

    The convex hull in two dimensions is a cornerstone of computational geometry, but many real-world and competitive programming problems extend into three dimensions. The 3D convex hull is the smallest convex polyhedron enclosing a set of points in space. While conceptually a natural extension of 2D, the algorithms and data structures are significantly more involved.

    In this post we explore the incremental construction algorithm, the horizon edge technique for stitching new faces, the DCEL data structure for representing the resulting polyhedron, and the key complexity results that make this practical in contests.

    3D Convex Hull Definition

    Given a set P of n points in 3D space, the 3D convex hull is the smallest convex polyhedron CH(P) such that every point in P lies on or inside the polyhedron. Equivalently, it is the intersection of all half-spaces containing P.

    The boundary of the 3D convex hull consists of:

    • Vertices — a subset of the original points.
    • Edges — line segments connecting pairs of hull vertices.
    • Faces — convex polygons (typically triangulated) bounded by edges.

    By the Upper Bound Theorem, a convex hull of n points in 3D has at most O(n) vertices, O(n) edges, and O(n) faces. More precisely, the number of faces is at most 2n − 4 (when all faces are triangles) by Euler’s formula.

    The diagram above shows a 3D convex hull of five points (A–E) projected onto a 2D plane. The red interior point lies inside the hull. Notice the triangulated faces and the wireframe edges connecting hull vertices.

    Incremental Algorithm

    The most practical 3D convex hull algorithm for competitive programming is the incremental algorithm. The idea is straightforward:

    1. Start with an initial tetrahedron formed by four non-coplanar points.
    2. For each remaining point p:
      • If p is inside the current hull, skip it.
      • Otherwise, find all faces visible from p.
      • Remove visible faces, creating a “hole” in the hull.
      • Stitch new triangular faces from p to each edge on the hole boundary (the horizon).

    A face with outward normal n and a vertex v is visible from point p if: 

    dot(n, p - v) > 0

    Here dot is the dot product. We compute the outward normal of a triangle (A, B, C) as the cross product (B − A) × (C − A), oriented so it points away from the interior of the hull.

    The initial tetrahedron is built by choosing four non-coplanar points. A simple strategy: pick the point with the smallest x-coordinate, then the one farthest from it, then the one farthest from the line they form, then the one farthest from the plane they define. 

    struct Point3 {
    long long x, y, z;
    Point3 operator-(const Point3 &o) const {
        return {x - o.x, y - o.y, z - o.z};
    }
};

Point3 cross3(Point3 a, Point3 b) {
    return {a.y*b.z - a.z*b.y,
            a.z*b.x - a.x*b.z,
            a.x*b.y - a.y*b.x};
}

long long dot3(Point3 a, Point3 b) {
    return a.x*b.x + a.y*b.y + a.z*b.z;
}

// Returns positive if p is above the plane of triangle (a, b, c)
// with outward normal cross3(b-a, c-a).
long long orient(Point3 a, Point3 b, Point3 c, Point3 p) {
    return dot3(cross3(b - a, c - a), p - a);
}

    Horizon Edges

    The most delicate part of the incremental algorithm is finding the horizon — the boundary between visible and invisible faces. When we remove all visible faces, the boundary edges of the resulting hole form a closed polygon on the hull surface.

    An edge is a horizon edge if exactly one of its two adjacent faces is visible from the new point. We walk along the boundary collecting these edges in order, then create a new triangular face from each horizon edge to the new point.

    In the diagram, the red faces are visible from the green point p. The thick yellow edges form the horizon. The dashed green lines indicate new faces that will be created connecting p to the horizon.

    A key invariant: after stitching, every horizon edge is shared between exactly one old (invisible) face and one new face. This preserves the manifold property of the hull.

    Implementation

    Below is a compact 3D convex hull implementation suitable for competitive programming. We represent each face as a triangle with indices into the point array. For each face we store the three vertex indices and, for each edge, the index of the adjacent face. 

    #include <bits/stdc++.h>
using namespace std;

struct P3 {
    long long x, y, z;
    P3 operator-(const P3 &o) const { return {x-o.x, y-o.y, z-o.z}; }
};

P3 cross3(P3 a, P3 b) {
    return {a.y*b.z - a.z*b.y, a.z*b.x - a.x*b.z, a.x*b.y - a.y*b.x};
}
long long dot3(P3 a, P3 b) { return a.x*b.x + a.y*b.y + a.z*b.z; }

struct Face { int v[3]; };

long long orient(P3 &a, P3 &b, P3 &c, P3 &p) {
    return dot3(cross3(b - a, c - a), p - a);
}

vector<Face> convexHull3D(vector<P3> &pts) {
    int n = pts.size();
    // Shuffle to avoid adversarial input
    mt19937 rng(42);
    shuffle(pts.begin(), pts.end(), rng);

    vector<Face> faces;
    // Build initial tetrahedron
    // Find 4 non-coplanar points
    int a = 0, b = -1, c = -1, d = -1;
    for (int i = 1; i < n; i++) {
        if (pts[i].x != pts[a].x || pts[i].y != pts[a].y || pts[i].z != pts[a].z)
            { b = i; break; }
    }
    if (b < 0) return faces;
    for (int i = b + 1; i < n; i++) {
        P3 cr = cross3(pts[b] - pts[a], pts[i] - pts[a]);
        if (cr.x || cr.y || cr.z) { c = i; break; }
    }
    if (c < 0) return faces;
    for (int i = 0; i < n; i++) {
        if (i == a || i == b || i == c) continue;
        if (orient(pts[a], pts[b], pts[c], pts[i]) != 0)
            { d = i; break; }
    }
    if (d < 0) return faces;

    // Ensure consistent orientation
    if (orient(pts[a], pts[b], pts[c], pts[d]) > 0) swap(b, c);
    faces.push_back({a, b, c});
    faces.push_back({a, c, b}); // we need all 4 faces
    faces[1] = {a, c, d};
    faces.push_back({a, d, b});
    faces.push_back({b, d, c});

    // Incremental insertion
    // For each face, track if it is "alive"
    vector<bool> dead;
    auto addFaces = [&]() {
        dead.assign(faces.size(), false);
    };
    addFaces();

    for (int i = 0; i < n; i++) {
        if (i == a || i == b || i == c || i == d) continue;
        // Find visible faces
        vector<bool> vis(faces.size(), false);
        bool any = false;
        for (int f = 0; f < (int)faces.size(); f++) {
            if (dead[f]) continue;
            if (orient(pts[faces[f].v[0]], pts[faces[f].v[1]],
                       pts[faces[f].v[2]], pts[i]) > 0) {
                vis[f] = true;
                any = true;
            }
        }
        if (!any) continue;

        // Collect horizon edges
        // An edge (u, v) is horizon if face containing (u,v) is visible
        // and the adjacent face sharing that edge is not.
        map<pair<int,int>, int> edgeFace;
        for (int f = 0; f < (int)faces.size(); f++) {
            if (dead[f]) continue;
            for (int j = 0; j < 3; j++) {
                int u = faces[f].v[j], v = faces[f].v[(j+1)%3];
                edgeFace[{u, v}] = f;
            }
        }

        vector<pair<int,int>> horizon;
        for (int f = 0; f < (int)faces.size(); f++) {
            if (!vis[f] || dead[f]) continue;
            for (int j = 0; j < 3; j++) {
                int u = faces[f].v[j], v = faces[f].v[(j+1)%3];
                auto it = edgeFace.find({v, u});
                if (it != edgeFace.end() && !vis[it->second]) {
                    horizon.push_back({u, v});
                }
            }
        }

        // Remove visible faces
        for (int f = 0; f < (int)faces.size(); f++) {
            if (vis[f]) dead[f] = true;
        }

        // Stitch new faces
        for (auto &[u, v] : horizon) {
            faces.push_back({v, u, i});
            dead.push_back(false);
        }
    }

    // Compact
    vector<Face> result;
    for (int f = 0; f < (int)faces.size(); f++)
        if (!dead[f]) result.push_back(faces[f]);
    return result;
}

    Important notes:

    • We shuffle the points with a fixed seed to avoid worst-case behavior from adversarial input. The randomized incremental algorithm has expected O(n log n) complexity.
    • Each face stores three vertex indices. Edges are implicit as consecutive vertex pairs.
    • The dead vector avoids expensive deletion from the face list. We compact at the end.

    DCEL Data Structure

    For more advanced applications, the Doubly-Connected Edge List (DCEL) provides efficient traversal of the hull topology. Each edge is stored as a pair of half-edges, each belonging to one adjacent face.

    A DCEL half-edge stores:

    • origin — the vertex where this half-edge starts.
    • twin — the half-edge in the opposite direction on the adjacent face.
    • next — the next half-edge around the same face.
    • prev — the previous half-edge around the same face.
    • face — the face to the left of this half-edge.
    struct HalfEdge {
    int origin;
    int twin, next, prev;
    int face;
};

struct DCELFace {
    int edge; // any half-edge on this face
};

struct DCEL {
    vector<HalfEdge> edges;
    vector<DCELFace> faces;
    vector<int> vertexEdge; // one outgoing half-edge per vertex

    // Walk around a face
    vector<int> faceVertices(int f) {
        vector<int> verts;
        int start = faces[f].edge, e = start;
        do {
            verts.push_back(edges[e].origin);
            e = edges[e].next;
        } while (e != start);
        return verts;
    }
};

    In competitive programming, full DCEL implementations are rarely needed. The simpler face-based representation usually suffices. However, DCEL becomes essential when you need to walk along the hull surface, find adjacent faces efficiently, or answer point-location queries on the hull.

    Complexity and Applications

    The randomized incremental 3D convex hull algorithm runs in expected O(n log n) time. The worst-case for the deterministic version is O(n²), but randomization makes pathological cases astronomically unlikely.

    Complexity breakdown:

    • Space: O(n) faces, edges, and vertices on the hull.
    • Insertion: Expected O(log n) amortized per point (randomized).
    • Conflict graph: An optimized version maintains a conflict graph between points and faces for O(n log n) total. In contests, the simpler O(n²) version often passes for n ≤ 10&sup4;.

    Applications in competitive programming:

    1. Volume computation: The volume of a 3D convex hull can be computed as the sum of signed tetrahedron volumes from the origin to each face.
    2. Surface area: Sum the areas of all triangular faces.
    3. Point-in-hull queries: Test whether a query point is inside the hull by checking that it is on the interior side of every face.
    4. Smallest enclosing box / sphere: Often a preprocessing step.
    5. 3D farthest pair: The farthest pair of points must be hull vertices. Use rotating calipers in 3D. 
    // Volume of 3D convex hull (6x actual volume to stay in integers)
long long hullVolume6(vector<P3> &pts, vector<Face> &faces) {
    long long vol = 0;
    for (auto &f : faces) {
        P3 &a = pts[f.v[0]], &b = pts[f.v[1]], &c = pts[f.v[2]];
        vol += dot3(a, cross3(b, c));
    }
    return abs(vol);
}

    Practical tips for contests:

    • Always use long long for 3D cross products — coordinates up to 106 yield cross product values up to 1018.
    • Watch for degenerate cases: coplanar points, collinear points, duplicate points. Add small perturbation or handle them explicitly.
    • The simple O(n²) implementation usually works for n ≤ 5000 within typical 2-second time limits.
    • If you only need the hull faces (not the full DCEL), the compact face-list representation is sufficient and much easier to code.
