    Number Theoretic Transform

    July 2026 · 35 min read

    From FFT to NTT

    The Fast Fourier Transform (FFT) multiplies polynomials in O(n log n) by evaluating at complex roots of unity, performing pointwise multiplication, and interpolating. However, FFT operates over floating-point numbers, introducing rounding errors that corrupt results when coefficients exceed roughly 1015.

    The Number Theoretic Transform (NTT) is the exact analogue of FFT over Zp for a suitable prime p. Instead of complex n-th roots of unity e2πi/n, we use primitive n-th roots of unity modulo p.

    Core Idea. If p is prime and n | (p − 1), then Zp× contains a primitive n-th root of unity g(p−1)/n, where g is a primitive root mod p. This element plays the exact role of e2πi/n in the DFT.

    The key algebraic fact making NTT work: the DFT matrix and its inverse are well-defined over any field containing a primitive n-th root of unity. The complex numbers are one such field; Zp for appropriate p is another.

    Why Exact Arithmetic Matters

    • Large coefficients: Convolutions where result coefficients reach 1018 or higher cannot use floating-point FFT reliably.
    • Modular results: When the problem asks for the answer mod p, NTT gives exact results with no rounding step.
    • Polynomial operations: Polynomial division, power series inversion, and composition all require exact arithmetic to be numerically stable.

    Primitive Roots of Unity mod p

    Existence Conditions

    For NTT of size n (where n is a power of 2), we need a prime p such that n | (p − 1). This ensures the multiplicative group Zp× (which has order p − 1) contains an element of order exactly n.

    Theorem. Let p be prime and n | (p − 1). If g is a primitive root modulo p, then ω = g(p−1)/n is a primitive n-th root of unity in Zp. That is, ωn ≡ 1 (mod p) and ωk ¬≡ 1 (mod p) for 0 < k < n.

    Primes of the Form c · 2k + 1

    For NTT with power-of-two sizes, we need p − 1 to be divisible by large powers of 2. Primes of the form c · 2k + 1 are ideal:

    • 998244353 = 119 · 223 + 1 — supports NTT up to size 223 ≈ 8.4 × 106
    • 7340033 = 7 · 220 + 1 — supports NTT up to size 220 ≈ 106
    • 469762049 = 7 · 226 + 1 — supports up to 226
    • 985661441 = 235 · 222 + 1

    Finding the Primitive Root

    To find a primitive root g modulo p, test small candidates g = 2, 3, 5, … by verifying g(p−1)/q ¬≡ 1 (mod p) for each prime factor q of p − 1:

    long long modpow(long long base, long long exp, long long mod) {
    long long result = 1;
    base %= mod;
    while (exp > 0) {
        if (exp & 1) result = (__int128)result * base % mod;
        base = (__int128)base * base % mod;
        exp >>= 1;
    }
    return result;
}

long long findPrimitiveRoot(long long p) {
    long long phi = p - 1;
    // Factor phi
    vector<long long> factors;
    long long tmp = phi;
    for (long long d = 2; d * d <= tmp; d++) {
        if (tmp % d == 0) {
            factors.push_back(d);
            while (tmp % d == 0) tmp /= d;
        }
    }
    if (tmp > 1) factors.push_back(tmp);

    for (long long g = 2; ; g++) {
        bool ok = true;
        for (long long q : factors) {
            if (modpow(g, phi / q, p) == 1) {
                ok = false; break;
            }
        }
        if (ok) return g;
    }
}
    Fact. For p = 998244353, the smallest primitive root is g = 3. The primitive 223-th root of unity is 3(998244353−1)/223 = 3119 mod 998244353 = 15311432.

    The NTT Algorithm

    Cooley–Tukey Butterfly

    The iterative radix-2 decimation-in-time NTT is structurally identical to FFT. We process log2(n) stages, each performing n/2 butterfly operations. The butterfly for NTT mod p is:

    // Butterfly operation at stage with twiddle factor w let u = a[j] let v = a[j + half] * w mod p a[j] = (u + v) mod p a[j + half] = (u - v + p) mod p

    Bit-Reversal Permutation

    Before the butterfly stages, we reorder the input by bit-reversing the indices. For an array of size n = 2k, element at position i goes to position bitreverse(i, k). This ensures the recursive subdivision aligns correctly for the iterative algorithm.

    function bitReverse(a[], n): for i = 1 to n-1: j ← bitReversed(i, log2(n)) if i < j: swap(a[i], a[j])

    Forward and Inverse NTT

    The forward NTT evaluates polynomial A(x) at the powers of ω: A(ω0), A(ω1), …, A(ωn−1).

    The inverse NTT uses ω−1 as the twiddle factor and divides by n at the end. Since p is prime, n−1 mod p exists (compute via modpow(n, p−2, p)).

    Inverse NTT. INTT(y) = (1/n) · NTTω−1(y). The factor 1/n = n−1 mod p is applied after the transform.

    Choosing the Right Prime

    Why 998244353?

    This prime has become the de facto standard in competitive programming for several reasons:

    • Large 2-adic valuation: 998244353 − 1 = 119 · 223, so NTT supports sizes up to 223 = 8,388,608.
    • Fits in 32 bits: 998244353 < 230, so products of two residues fit in 64-bit integers without overflow.
    • Small primitive root: g = 3 is the primitive root, easy to remember.
    • Codeforces standard: Thousands of problems use this modulus specifically because NTT works with it.

    Implementation

    Full C++ NTT

    const long long MOD = 998244353;
const long long G   = 3;  // primitive root

long long power(long long a, long long b, long long mod) {
    long long res = 1;
    a %= mod;
    while (b > 0) {
        if (b & 1) res = res * a % mod;
        a = a * a % mod;
        b >>= 1;
    }
    return res;
}

void ntt(vector<long long>& a, bool inverse) {
    int n = a.size();
    // Bit-reversal permutation
    for (int i = 1, j = 0; i < n; i++) {
        int bit = n >> 1;
        for (; j & bit; bit >>= 1) j ^= bit;
        j ^= bit;
        if (i < j) swap(a[i], a[j]);
    }
    // Butterfly stages
    for (int len = 2; len <= n; len <<= 1) {
        long long w = inverse
            ? power(G, MOD - 1 - (MOD - 1) / len, MOD)
            : power(G, (MOD - 1) / len, MOD);
        for (int i = 0; i < n; i += len) {
            long long wn = 1;
            for (int j = 0; j < len / 2; j++) {
                long long u = a[i + j];
                long long v = a[i + j + len / 2] * wn % MOD;
                a[i + j]             = (u + v) % MOD;
                a[i + j + len / 2] = (u - v + MOD) % MOD;
                wn = wn * w % MOD;
            }
        }
    }
    if (inverse) {
        long long inv_n = power(n, MOD - 2, MOD);
        for (auto& x : a) x = x * inv_n % MOD;
    }
}

    Polynomial Multiplication

    vector<long long> multiply(vector<long long> a, vector<long long> b) {
    int result_size = a.size() + b.size() - 1;
    int n = 1;
    while (n < result_size) n <<= 1;
    a.resize(n); b.resize(n);

    ntt(a, false);
    ntt(b, false);
    for (int i = 0; i < n; i++)
        a[i] = a[i] * b[i] % MOD;
    ntt(a, true);

    a.resize(result_size);
    return a;
}
    Correctness Proof Sketch. The NTT evaluates both polynomials at the n-th roots of unity, multiplies pointwise (which corresponds to polynomial multiplication by the convolution theorem), and the inverse NTT recovers the product coefficients. Since all arithmetic is exact mod p, there is no rounding error.

    NTT Convolution Tricks

    Arbitrary Modulus via 3-NTT (CRT Approach)

    When the problem modulus m is not NTT-friendly, perform NTT with three different NTT-friendly primes p1, p2, p3, then reconstruct the true coefficients via CRT.

    // Multiply mod arbitrary m using three NTT primes choose p1 = 998244353, p2 = 985661441, p3 = 469762049 // Product p1*p2*p3 > 10^26, enough for most problems c1 = multiply_mod_p(a, b, p1) c2 = multiply_mod_p(a, b, p2) c3 = multiply_mod_p(a, b, p3) for each coefficient index i: // Garner's algorithm to reconstruct c[i] from c1[i], c2[i], c3[i] x = garnerCRT(c1[i], c2[i], c3[i], p1, p2, p3) result[i] = x mod m

    This technique handles moduli like 109 + 7 where direct NTT is impossible. The overhead is 3× the NTT cost.

    Coefficient Splitting

    An alternative to 3-NTT: split each coefficient into high and low 15-bit halves, perform 3 NTTs (for low×low, low×high + high×low, and high×high), and combine. This uses only one NTT-friendly prime:

    // Split each a[i] into a_lo[i] and a_hi[i] where
// a[i] = a_hi[i] * (1 << 15) + a_lo[i]
// Then a*b = (a_hi*b_hi) * 2^30 + (a_hi*b_lo + a_lo*b_hi) * 2^15 + a_lo*b_lo
// Each sub-product has coefficients < n * (2^15)^2 < 10^15 for n < 10^6
// which fits in a single NTT mod 998244353

    Online / Relaxed Multiplication

    For power series where later coefficients depend on earlier ones (e.g., polynomial inversion), the “divide and conquer NTT” technique computes convolutions incrementally in O(n log2 n) total.

    Complexity & Practice

    Time and Space Complexity

    NTT (single)

    Time: O(n log n)

    Space: O(n)

    About 2–3× faster than FFT due to integer arithmetic and no sin/cos.

    Polynomial Multiply

    Time: O(n log n) where n is the sum of degrees

    Three NTT calls: forward A, forward B, inverse product.

    3-NTT (arb. mod)

    Time: O(n log n) with 3× constant

    Nine NTT calls total plus CRT reconstruction.

    Poly Inversion

    Time: O(n log n) via Newton’s method

    Computes first n terms of 1/A(x) mod xn.

    NTT vs FFT: When to Use Which

    • Use NTT when the answer is required mod a prime (especially 998244353), or when coefficient magnitudes exceed 1015.
    • Use FFT when coefficients are small (e.g., polynomial multiplication where results fit in 64-bit integers) and no modular arithmetic is needed, or when the modulus isn’t NTT-friendly and you want simplicity over the 3-NTT approach.
    • Use NTT with CRT for non-NTT-friendly moduli when precision is critical.

    Common Contest Problems

    • Polynomial multiplication mod 998244353: Direct NTT application. (CF “E” and “F” problems frequently.)
    • Counting paths/sequences via generating functions: Express the answer as a polynomial product or power, compute via NTT.
    • Subset sum convolution: Uses Möbius/Zeta transforms related to NTT over the Boolean lattice.
    • Multi-point evaluation and interpolation: Subquadratic algorithms built on top of NTT for polynomial operations.
    Practice Tip. Template your NTT code and test it on CF 1096G (“Lucky Tickets”) or CF 993E (“Nikita and Order Statistics”). Both are clean NTT applications that verify your implementation.

    Optimization Tips

    • Precompute twiddle factors: Instead of computing wj inside the inner loop, precompute the array of roots once per NTT call.
    • Montgomery multiplication: For extreme speed, replace modular reduction with Montgomery form. This avoids expensive division operations.
    • Cache-friendly access: The iterative NTT already has reasonable cache behavior; for very large transforms, consider 4-step NTT that processes data in cache-sized blocks.
    • Avoid unnecessary resizing: Pre-allocate arrays of size 2k once and reuse across multiple NTT calls.
