    FFT Implementation

    July 2026 · 25 min read

    1 — Why Iterative FFT?

    The recursive FFT from Part 3 is elegant but has two practical drawbacks:

    • Memory allocation: Each recursion level creates two new vectors of size N/2, leading to O(N log N) total allocations. Memory allocation is slow and fragments the heap.
    • Function call overhead: O(N) recursive calls add stack-frame overhead. For N = 2²⁰ ≈ 10⁶, that’s a million function calls.

    The iterative (bottom-up) FFT works entirely in-place on a single array. It has the same O(N log N) time complexity but runs 2–5× faster in practice due to better cache behavior and zero allocations.

    Rule of thumb. Always use iterative FFT in competitive programming submissions. The recursive version is only useful for understanding the algorithm.

    Overview of the iterative approach

    1. Apply a bit-reversal permutation to the input array.
    2. Perform butterfly merges bottom-up: start with pairs, then groups of 4, 8, …, up to N.
    3. If computing the inverse, divide every element by N at the end.

    2 — Bit-Reversal Permutation

    2.1 Where does it come from?

    In the recursive FFT, the first split separates even-indexed and odd-indexed coefficients. At the next level, the even-indexed elements are split again, and so on. After log₂ N levels of splitting, element a[i] ends up at position bitrev(i), where bitrev reverses the binary representation of i (using log₂ N bits).

    2.2 Derivation

    Consider N = 8. The recursive decomposition proceeds:

    Level 0:  [0, 1, 2, 3, 4, 5, 6, 7]
Level 1:  [0, 2, 4, 6] [1, 3, 5, 7]       (evens | odds)
Level 2:  [0, 4] [2, 6] [1, 5] [3, 7]     (evens of evens | odds of evens | …)
Level 3:  [0] [4] [2] [6] [1] [5] [3] [7] (base cases)

    The final ordering [0, 4, 2, 6, 1, 5, 3, 7] is exactly the bit-reversal permutation:

    Index (binary) → Reversed → Decimal
  0  (000)     →   000    →   0
  1  (001)     →   100    →   4
  2  (010)     →   010    →   2
  3  (011)     →   110    →   6
  4  (100)     →   001    →   1
  5  (101)     →   101    →   5
  6  (110)     →   011    →   3
  7  (111)     →   111    →   7

    Proof (by induction on log₂ N).

    Base case: N = 2. Even/odd split gives [a₀, a₁]. Bit-reversal of {0, 1} in 1 bit is {0, 1}. ✓

    Inductive step: Assume the claim holds for size N/2. The even-odd split maps index i (0-indexed, using b = log₂ N bits) to:

    • If i is even (LSB = 0): position in even sub-array is i/2, which by induction will be placed at bitrev(i/2, b − 1) within the first half.
    • If i is odd (LSB = 1): position in odd sub-array is (i − 1)/2, placed at bitrev((i−1)/2, b − 1) within the second half, i.e., offset by N/2.

    In both cases, the final position is bitrev(i, b): the LSB of i becomes the MSB, and the remaining bits are reversed by the induction hypothesis. ∎

    2.3 Computing bit-reversal

    // O(N) bit-reversal permutation in-place
void bitReverse(std::vector<cd>& a) {
    int n = (int)a.size(), lg = 0;
    while ((1 << lg) < n) ++lg;
    for (int i = 0; i < n; ++i) {
        int j = 0;
        for (int b = 0; b < lg; ++b)
            if (i & (1 << b))
                j |= 1 << (lg - 1 - b);
        if (i < j) std::swap(a[i], a[j]);
    }
}

    An even faster method builds the bit-reversed index incrementally:

    void bitReverseFast(std::vector<cd>& a) {
    int n = (int)a.size();
    for (int i = 1, j = 0; i < n; ++i) {
        int bit = n >> 1;
        for (; j & bit; bit >>= 1)
            j ^= bit;
        j ^= bit;
        if (i < j) std::swap(a[i], a[j]);
    }
}

    3 — Butterfly Operations

    3.1 Structure

    After bit-reversal, the iterative FFT merges sub-arrays bottom-up. At each stage, the array is divided into blocks of size len (starting at 2, doubling each stage). Within each block, we combine the first half (even part) and second half (odd part) using the butterfly:

    for each block of size len:
    ω = e^{2πi/len}   (or e^{−2πi/len} for inverse)
    w = 1
    for j = 0 to len/2 − 1:
        u = a[start + j]             // even element
        t = w · a[start + j + len/2] // odd element × twiddle
        a[start + j]           = u + t
        a[start + j + len/2]   = u − t
        w = w · ω

    3.2 Why it works

    At stage s (where len = 2s), each block contains a sub-DFT of size len. The butterfly combines two sub-DFTs of size len/2 into one of size len, which is exactly the Cooley-Tukey merge step. Since we process blocks from smallest (len = 2) to largest (len = N), we build the full DFT from the base cases up.

    3.3 Twiddle factor precision

    The twiddle factor ω^j is accumulated by repeated multiplication: w *= wn. Over many iterations this accumulates error. For N ≤ 2²³ (≈ 8 × 10⁶) and double precision (53-bit mantissa), the accumulated error stays well within 10⁻⁹, which is more than enough for rounding to integers.

    4 — Complete Iterative FFT in C++

    #include <bits/stdc++.h>
using cd = std::complex<double>;
const double PI = acos(-1.0);

void fft(std::vector<cd>& a, bool invert) {
    int n = (int)a.size();
    if (n == 1) return;

    // Bit-reversal permutation
    for (int i = 1, j = 0; i < n; ++i) {
        int bit = n >> 1;
        for (; j & bit; bit >>= 1)
            j ^= bit;
        j ^= bit;
        if (i < j) std::swap(a[i], a[j]);
    }

    // Bottom-up butterfly merges
    for (int len = 2; len <= n; len <<= 1) {
        double angle = 2.0 * PI / len * (invert ? -1 : 1);
        cd wn(cos(angle), sin(angle));
        for (int i = 0; i < n; i += len) {
            cd w(1);
            for (int j = 0; j < len / 2; ++j) {
                cd u = a[i + j];
                cd t = w * a[i + j + len / 2];
                a[i + j]           = u + t;
                a[i + j + len / 2] = u - t;
                w *= wn;
            }
        }
    }

    if (invert) {
        for (auto& x : a)
            x /= n;
    }
}
    Template note. This is the standard competitive programming FFT. Copy it into your template library. The only parameter that changes between DFT and IDFT is the sign of the angle and the final division.

    Complexity analysis

    • Bit-reversal: O(N) swaps, each O(1).
    • Butterfly stages: log₂ N stages, each doing N/2 butterflies → O(N log N) total.
    • Space: O(1) auxiliary (everything is in-place).

    5 — End-to-End Polynomial Multiplication

    Here is the complete function for multiplying two integer polynomials:

    std::vector<long long> polyMul(const std::vector<int>& a,
                                const std::vector<int>& b) {
    std::vector<cd> fa(a.begin(), a.end());
    std::vector<cd> fb(b.begin(), b.end());

    // Round up to next power of 2
    int n = 1;
    while (n < (int)(a.size() + b.size()))
        n <<= 1;
    fa.resize(n);
    fb.resize(n);

    fft(fa, false);  // Forward DFT
    fft(fb, false);

    // Pointwise multiplication
    for (int i = 0; i < n; ++i)
        fa[i] *= fb[i];

    fft(fa, true);   // Inverse DFT

    // Round to nearest integer
    std::vector<long long> result(n);
    for (int i = 0; i < n; ++i)
        result[i] = llround(fa[i].real());

    // Remove trailing zeros
    while (result.size() > 1 && result.back() == 0)
        result.pop_back();
    return result;
}

    Usage example

    int main() {
    // Multiply (1 + 2x + 3x^2) by (4 + 5x)
    auto c = polyMul({1, 2, 3}, {4, 5});
    // c = {4, 13, 22, 15}
    // = 4 + 13x + 22x^2 + 15x^3
    for (auto v : c) std::cout << v << ' ';
    // Output: 4 13 22 15
}

    6 — Floating-Point Precision

    6.1 Sources of error

    There are three main sources of floating-point error in FFT-based multiplication:

    1. Twiddle factor computation: cos and sin are computed to ~15 significant digits (double precision). Multiplying by imprecise twiddle factors introduces small errors at each butterfly.
    2. Accumulation through butterflies: After log₂ N stages, errors accumulate. The worst-case error in any single output coefficient is approximately O(N · ε_mach · max|aᵢ|), where ε_mach ≈ 10⁻¹⁶ for doubles.
    3. Catastrophic cancellation: When we subtract nearly equal complex numbers in the butterfly (u − t), we can lose significant digits. This is rare in practice but theoretically possible.

    6.2 When to round

    Always round, never truncate. After the inverse FFT, use llround(fa[i].real()), not (long long)fa[i].real(). Truncation toward zero can turn 6.9999999997 into 6 instead of 7.

    6.3 Safe limits

    Coefficient magnitude Max safe N Notes
    ≤ 10³~2²³ (8M)Comfortable margin
    ≤ 10⁶~2²⁰ (1M)Usually fine
    ≤ 10⁹~2¹⁸ (256K)Marginal; consider NTT
    ≤ 10¹⁸—Must use NTT or split trick

    6.4 The split trick for large coefficients

    If coefficients are large (up to 10⁹) and N is large (up to 10⁶), split each coefficient into two parts:

    aᵢ = aᵢ_hi · BASE + aᵢ_lo     where BASE = √(max coeff) ≈ 10⁴ or 2¹⁵

    Then A(x) = A_hi(x) · BASE + A_lo(x), and

    A · B = A_hi · B_hi · BASE² + (A_hi · B_lo + A_lo · B_hi) · BASE + A_lo · B_lo

    Each sub-product involves coefficients ≤ BASE ≈ 10⁴, so floating-point FFT is accurate. This costs 3–4 FFTs instead of 2, but avoids precision issues entirely.

    7 — Common Pitfalls & Tips

    Pitfall 1: Not padding to a power of 2. The iterative FFT requires N to be a power of 2. If you forget to pad, the bit-reversal permutation produces garbage.
    Pitfall 2: Off-by-one in output size. The product of a degree-m and degree-n polynomial has degree m + n, so the output has m + n + 1 coefficients. Pad N ≥ m + n + 1, not N ≥ max(m, n).
    Tip: Precompute twiddle factors. For problems with many FFT calls, precompute cos and sin values in a lookup table. This avoids repeated trigonometric function calls and can give a 20–30% speedup.
    Tip: Use __builtin_ctz for the log. Instead of computing log₂ N with a loop, use __builtin_ctz(n) (count trailing zeros). Since N is a power of 2, this directly gives log₂ N.
