    Polynomial Hashing

    July 2026 · 25 min read

    1. The Polynomial Hash Function

    A polynomial hash (also called a rolling hash) maps a string s = s0s1…sn−1 to an integer via:

    H(s) = ∑i=0..n−1 si · pi mod m

    where p is the base (a prime, typically larger than the alphabet size) and m is the modulus (a large prime). Here si is the numeric value of the i-th character (e.g., 'a' → 1, 'b' → 2, …).

    Why polynomial? The hash evaluates the polynomial f(p) = s0 + s1p + s2p² + … at point p modulo m. Different strings correspond to different polynomials, and by the Schwartz-Zippel lemma, two distinct polynomials of degree < n agree at a random point with probability < n/m. This is the theoretical foundation for the collision bound.

    The key advantage of this hash is its rolling property: we can compute the hash of any substring in O(1) after O(n) preprocessing using prefix hashes.

    2. Collision Probability & the Birthday Bound

    2.1 Single Pair Collision

    For two distinct strings s ≠ t of length ≤ L, treating the base p as a random element of Z/mZ, the probability they hash to the same value is at most L/m (by Schwartz-Zippel, since the difference polynomial has degree < L and at most L roots mod m). For m ≈ 109 and L ≈ 106, this is about 10−3.

    2.2 Birthday Bound

    In practice we compare many pairs of substrings. If we make Q hash comparisons, the expected number of collisions is about Q · L/m. By the birthday paradox, if we have K distinct hash values chosen uniformly from {0, …, m−1}, the probability of at least one collision is approximately:

    P(collision) ≈ 1 − e−K(K−1)/(2m) ≈ K²/(2m)

    Proof sketch: The probability that all K values are distinct is ∏i=1..K−1(1 − i/m). Taking logarithms:

    ln P(all distinct) = ∑i=1..K−1 ln(1 − i/m) ≈ −∑i=1..K−1 i/m = −K(K−1)/(2m)

    So P(all distinct) ≈ e−K²/(2m) and the collision probability is approximately 1 minus this. For K = 105 and m = 109, we get P ≈ 1 − e−5×109/2×109 ≈ 1 − e−5 ≈ 0.99, which is almost certain! This motivates the need for either a larger modulus or double hashing.

    Takeaway: A single mod ≈ 109 is not safe when comparing > 104 substrings. Either use a 64-bit modulus or double hashing.

    3. Prefix Hashes & O(1) Substring Queries

    Precompute prefix hashes h[i] = H(s[0..i−1]) and powers pw[i] = pi mod m:

    • h[0] = 0
    • h[i] = (h[i−1] + s[i−1] · pw[i−1]) mod m

    Then the hash of the substring s[l..r−1] (0-indexed, length r−l) is:

    H(s[l..r−1]) = (h[r] − h[l]) · pw−1[l] mod m

    Alternatively, to avoid computing modular inverses, we can normalize differently. Define forward hashes where h[i] = s[0]·pi−1 + s[1]·pi−2 + … + s[i−1]. Then:

    H(s[l..r−1]) = h[r] − h[l] · pw[r−l] mod m

    This avoids modular inverse entirely and is the preferred approach in contests.

    4. Rabin-Karp String Matching

    The classic application of polynomial hashing is the Rabin-Karp algorithm for pattern matching: given a text T of length n and a pattern P of length m, find all occurrences of P in T.

    Algorithm:

    1. Compute H(P).
    2. Build prefix hashes of T.
    3. For each position i in [0, n−m], compute H(T[i..i+m−1]) in O(1) and compare with H(P).
    4. On a hash match, optionally verify character-by-character to eliminate false positives.

    Complexity: O(n + m) expected time (with high probability if using a good hash). Worst case O(nm) if we verify every match, but with double hashing the probability of any false positive is negligible. 

    #include <bits/stdc++.h>
using namespace std;

// Rabin-Karp with single hash
vector<int> rabin_karp(const string& text, const string& pattern) {
    const long long MOD = 1e9 + 7;
    const long long BASE = 31;
    int n = text.size(), m = pattern.size();
    if (m > n) return {};

    // Precompute powers
    vector<long long> pw(n + 1);
    pw[0] = 1;
    for (int i = 1; i <= n; i++)
        pw[i] = pw[i - 1] * BASE % MOD;

    // Forward prefix hashes of text
    // h[i] = text[0]*p^{i-1} + text[1]*p^{i-2} + ... + text[i-1]
    vector<long long> h(n + 1, 0);
    for (int i = 0; i < n; i++)
        h[i + 1] = (h[i] * BASE + (text[i] - 'a' + 1)) % MOD;

    // Hash of pattern
    long long hp = 0;
    for (int i = 0; i < m; i++)
        hp = (hp * BASE + (pattern[i] - 'a' + 1)) % MOD;

    // Slide over text
    vector<int> matches;
    for (int i = 0; i + m <= n; i++) {
        // Hash of text[i..i+m-1] = h[i+m] - h[i]*p^m
        long long ht = (h[i + m] - h[i] * pw[m] % MOD + MOD * 2) % MOD;
        if (ht == hp) {
            // Optional: verify to eliminate false positives
            if (text.substr(i, m) == pattern)
                matches.push_back(i);
        }
    }
    return matches;
}

int main() {
    string text = "abcabcabc";
    string pattern = "abc";
    auto res = rabin_karp(text, pattern);
    for (int pos : res)
        cout << "Match at position " << pos << "\n";
    return 0;
}

    5. Anti-Hash Tests & Adversarial Attacks

    On Codeforces (and similar platforms with "hacking"), adversaries can craft inputs that cause collisions for a specific single-mod hash. The standard attack exploits the fact that if we know the base p and modulus m, we can construct strings s ≠ t with H(s) = H(t).

    5.1 How the Attack Works

    Given base p and modulus m, find two short strings s1, s2 of the same length with H(s1) = H(s2) mod m. This can be done by birthday attack: generate ∼√m random strings and look for a collision. With m ≈ 109, this requires only ∼3×104 strings.

    Once we have a collision pair (s1, s2), we can substitute s1 ↔ s2 independently at multiple positions to create exponentially many distinct strings that all hash to the same value.

    5.2 Defenses

    • Random base: Choose p randomly at runtime using a random seed. The attacker cannot predict which base you use, so pre-computed collision pairs don't apply.
    • Double hashing: Use two independent (base, modulus) pairs. The probability that a collision occurs in both hashes simultaneously is ≈ 1/(m1 · m2), which is ≈ 10−18 for two 109-primes.
    • Large modulus: Use a single modulus ≥ 261 (e.g., the Mersenne prime 261 − 1). The birthday bound gives collision probability ≈ K² / 262, which is negligible for K ≤ 106.
    Best practice: Always use at least double hashing in contests where hacking is possible. Randomizing the base adds another layer of safety. The performance overhead is minimal (2× more multiplications).

    6. Double Hashing Implementation

    The following is a complete, contest-ready implementation of a double polynomial hash with O(n) preprocessing and O(1) substring hash queries. 

    #include <bits/stdc++.h>
using namespace std;

struct DoubleHash {
    static const long long MOD1 = 1e9 + 7, MOD2 = 1e9 + 9;
    long long BASE1, BASE2;
    int n;
    vector<long long> h1, h2, pw1, pw2;

    DoubleHash() {}

    DoubleHash(const string& s) {
        // Randomize bases for anti-hack safety
        mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
        BASE1 = uniform_int_distribution<long long>(256, MOD1 - 1)(rng);
        BASE2 = uniform_int_distribution<long long>(256, MOD2 - 1)(rng);

        n = (int)s.size();
        h1.resize(n + 1); h2.resize(n + 1);
        pw1.resize(n + 1); pw2.resize(n + 1);

        pw1[0] = pw2[0] = 1;
        h1[0] = h2[0] = 0;

        for (int i = 0; i < n; i++) {
            int c = s[i] - 'a' + 1;
            h1[i + 1] = (h1[i] * BASE1 + c) % MOD1;
            h2[i + 1] = (h2[i] * BASE2 + c) % MOD2;
            pw1[i + 1] = pw1[i] * BASE1 % MOD1;
            pw2[i + 1] = pw2[i] * BASE2 % MOD2;
        }
    }

    // Hash of s[l..r-1] (0-indexed, length = r - l)
    pair<long long, long long> get(int l, int r) const {
        long long v1 = (h1[r] - h1[l] * pw1[r - l] % MOD1 + MOD1 * 2) % MOD1;
        long long v2 = (h2[r] - h2[l] * pw2[r - l] % MOD2 + MOD2 * 2) % MOD2;
        return {v1, v2};
    }

    bool equal(int l1, int r1, int l2, int r2) const {
        return get(l1, r1) == get(l2, r2);
    }
};

int main() {
    string s = "abcabcabc";
    DoubleHash dh(s);

    // Check if s[0..2] == s[3..5]  ("abc" == "abc")
    cout << "s[0..2] == s[3..5]? " << dh.equal(0, 3, 3, 6) << "\n";

    // Check if s[0..2] == s[1..3]  ("abc" == "bca")
    cout << "s[0..2] == s[1..3]? " << dh.equal(0, 3, 1, 4) << "\n";

    return 0;
}

    6.1 Using the Mersenne Prime 261 − 1

    An alternative to double hashing is using a single very large prime. The Mersenne prime M = 261 − 1 allows fast modular arithmetic using only bitwise operations and additions (no native 128-bit types needed): 

    // Fast modular arithmetic with Mersenne prime 2^61 - 1
struct MersenneHash {
    static const long long M = (1LL << 61) - 1;
    long long BASE;
    int n;
    vector<long long> h, pw;

    static long long mod_m(long long a) {
        // Reduce a mod (2^61 - 1) assuming 0 <= a < 2^62
        a = (a >> 61) + (a & M);
        if (a >= M) a -= M;
        return a;
    }

    static long long mul_mod(long long a, long long b) {
        // Compute a*b mod M using 128-bit intermediate
        __int128 res = (__int128)a * b;
        return mod_m((long long)(res >> 61) + (long long)(res & M));
    }

    MersenneHash(const string& s) {
        mt19937_64 rng(chrono::steady_clock::now().time_since_epoch().count());
        BASE = rng() % (M - 256) + 256;

        n = (int)s.size();
        h.resize(n + 1);
        pw.resize(n + 1);
        h[0] = 0; pw[0] = 1;

        for (int i = 0; i < n; i++) {
            int c = s[i] - 'a' + 1;
            h[i + 1] = mod_m(mul_mod(h[i], BASE) + c);
            pw[i + 1] = mul_mod(pw[i], BASE);
        }
    }

    long long get(int l, int r) const {
        return mod_m(h[r] - mul_mod(h[l], pw[r - l]) + M * 2);
    }
};

    This single hash has collision probability < n / 261 ≈ 10−13 per comparison, making it comparable in safety to double hashing with 109-primes, but with less code and fewer operations.

    7. Choosing Base & Modulus

    Base p
    • Must be > alphabet size (e.g., > 26 for lowercase letters).
    • Typically a prime in range [31, 1000].
    • Randomize at runtime for anti-hack safety.
    • Avoid p = 0 or 1 (degenerate hashes).
    Modulus m
    • Must be prime (ensures every nonzero element has an inverse).
    • Common choices: 109+7, 109+9, or 261−1.
    • Larger m → fewer collisions but potentially slower arithmetic.
    • For safety, use two mods or one Mersenne prime.

    A practical rule of thumb: if you compare at most K substrings, use a modulus m >> K² to keep collision probability below 1%. With double hashing, the effective modulus is m1·m2 ≈ 1018, which is safe for K up to about 108.

    8. Common Applications & Patterns

    8.1 Longest Common Substring

    Binary search on the length L. For each candidate L, compute the set of all substring hashes of length L from both strings and check for intersection. Time: O(n log n) with hashing.

    8.2 Number of Distinct Substrings

    Insert all O(n²) substring hashes into a hash set. The set size is the number of distinct substrings. With suffix arrays this is O(n), but the hashing approach is simpler to implement and sufficient for n ≤ 5000.

    8.3 Palindrome Checking

    Build prefix hashes for both s and its reverse. A substring s[l..r] is a palindrome iff H(s[l..r]) = H(rev(s)[n−1−r..n−1−l]). This gives O(1) palindrome checks after O(n) preprocessing.

    8.4 Polynomial Equivalence (Connection to Algebra)

    Polynomial hashing is literally polynomial evaluation at a point. Two strings are "equal" iff the polynomials ∑ sixi and ∑ tixi agree at x = p. By the Schwartz-Zippel lemma (from algebraic complexity theory), this happens with probability at most deg/|S| over a random evaluation point from set S. This is why polynomial hashing works — it's a probabilistic polynomial identity test.

    9. Summary & Best Practices

    • Polynomial hashing maps strings to integers via f(p) = ∑ sipi mod m.
    • Prefix hashes enable O(1) substring hash queries after O(n) preprocessing.
    • Single mod ≈ 109 is not safe for > 104 comparisons due to the birthday bound.
    • Use double hashing (two independent mods) or the Mersenne prime 261−1 for safety.
    • Randomize the base at runtime to defend against adversarial inputs.
    • Rabin-Karp string matching is the textbook application: O(n+m) expected time.
    Key takeaway. Polynomial hashing is one of the most versatile tools in competitive programming. It reduces string comparison to integer comparison, enabling O(1) substring equality checks. The theoretical foundation is the Schwartz-Zippel lemma from algebraic complexity. Always use double hashing or a large Mersenne prime in contests with hacking to avoid adversarial collisions.
