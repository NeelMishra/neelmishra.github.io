Output-Sensitive Algorithms — Enumeration & Output-Dependent Complexity

July 2026 · 25 min read

What Is Output-Sensitive?

Most algorithm complexity bounds are stated purely in terms of the input size n. An output-sensitive algorithm has a running time that also depends on the size of the output k (or sometimes denoted h). The complexity is typically written as O(f(n, k)) where k is the number of elements in the result.

Why does this matter? Consider computing the convex hull of n points in the plane. Graham scan runs in O(n log n) regardless of whether the hull has 3 vertices or n vertices. But the gift-wrapping algorithm (Jarvis march) runs in O(nh) where h is the number of hull vertices. When h is small — say O(log n) — gift wrapping is asymptotically faster than Graham scan.

Output-sensitive algorithms shine in two broad categories:

  • Geometric algorithms — convex hulls, Voronoi diagrams, line arrangement intersections, where the output can be much smaller than the input.
  • Enumeration algorithms — listing all maximal cliques, spanning trees, permutations, subsets, partitions, etc., where the output can be exponentially larger than the input and we want constant or polynomial amortized cost per output item.
Key insight: When a problem asks you to enumerate or output all of something, the lower bound is Ω(k) where k is the output size — you can’t do better than reading what you print. The goal is to match this lower bound (up to polynomial factors in n) so that the cost per output item is as small as possible.

Formally, we say an algorithm is optimal output-sensitive if its running time is O(poly(n) + k) where k is the output size. For enumeration problems, the gold standard is constant amortized time (CAT) per output item — total time O(k) after O(poly(n)) preprocessing.

Convex Hull — A Case Study

The convex hull is the canonical example where output-sensitivity yields real speedups. Let’s compare the major algorithms:

Graham Scan — O(n log n)

Sort points by polar angle, then sweep. The running time is dominated by sorting and is independent of h (the number of hull vertices). Always O(n log n).

Jarvis March — O(nh)

Start from the lowest point, repeatedly find the point making the smallest polar angle with the current edge. Each of the h iterations scans all n points. When h = O(log n), this beats Graham scan.

The Chan’s algorithm achieves the best of both worlds: O(n log h). It uses a clever parameter-guessing technique — try h = 2, 4, 8, 16, … and run a limited Jarvis march using Graham-scan hulls of groups of size h. The total work across all guesses is geometric, so the overall cost is O(n log h).

Gift Wrapping (Jarvis March) Implementation

Here is a clean C++ implementation of Jarvis march that runs in O(nh): 

#include <bits/stdc++.h>
using namespace std;

typedef long long ll;
struct Point {
    ll x, y;
};

// Cross product of vectors OA and OB
ll cross(const Point& O, const Point& A, const Point& B) {
    return (A.x - O.x) * (B.y - O.y) - (A.y - O.y) * (B.x - O.x);
}

ll dist2(const Point& A, const Point& B) {
    return (A.x - B.x) * (A.x - B.x) + (A.y - B.y) * (A.y - B.y);
}

// Jarvis March: O(nh) convex hull
vector<Point> jarvisMarch(vector<Point>& pts) {
    int n = pts.size();
    if (n < 3) return pts;

    // Find leftmost point (lowest y to break ties)
    int start = 0;
    for (int i = 1; i < n; i++) {
        if (pts[i].x < pts[start].x ||
            (pts[i].x == pts[start].x && pts[i].y < pts[start].y))
            start = i;
    }

    vector<Point> hull;
    int cur = start;
    do {
        hull.push_back(pts[cur]);
        int next = 0;
        for (int i = 1; i < n; i++) {
            if (i == cur) continue;
            ll cp = cross(pts[cur], pts[next], pts[i]);
            if (next == cur || cp > 0 ||
                (cp == 0 && dist2(pts[cur], pts[i]) > dist2(pts[cur], pts[next]))) {
                next = i;
            }
        }
        cur = next;
    } while (cur != start);

    return hull;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    vector<Point> pts(n);
    for (auto& p : pts) cin >> p.x >> p.y;

    auto hull = jarvisMarch(pts);
    cout << hull.size() << "\n";
    for (auto& p : hull)
        cout << p.x << " " << p.y << "\n";
    return 0;
}
When to prefer Jarvis march in CP: If you know (or suspect) the hull has very few vertices — e.g., points are generated uniformly in a disk (expected h = O(n1/3)) or the problem guarantees small h — Jarvis march with O(nh) can outperform Graham’s O(n log n). Chan’s algorithm is rarely implemented in contests due to its complexity, but it’s worth knowing theoretically.

Enumerating Maximal Cliques — Bron-Kerbosch

A maximal clique is a clique that cannot be extended by adding any adjacent vertex. The number of maximal cliques in a graph on n vertices can be up to 3n/3 (the Moon-Moser bound). The Bron-Kerbosch algorithm enumerates all maximal cliques with worst-case time O(3n/3), which is optimal since that’s the maximum number of maximal cliques.

The Algorithm

Bron-Kerbosch maintains three sets:

  • R — the current growing clique
  • P — candidate vertices that can extend R
  • X — vertices already processed (used to avoid duplicates)

When P and X are both empty, R is a maximal clique. The pivot optimization chooses a vertex u from P ∪ X that maximizes |P ∩ N(u)|, which prunes branches where we would discover non-maximal cliques. 

#include <bits/stdc++.h>
using namespace std;

const int MAXN = 200;
bitset<MAXN> adj[MAXN];
int n, m;
vector<vector<int>> allCliques;

// Bron-Kerbosch with pivot
void bronKerbosch(bitset<MAXN>& R, bitset<MAXN>& P, bitset<MAXN>& X) {
    if (P.none() && X.none()) {
        // R is a maximal clique
        vector<int> clique;
        for (int i = 0; i < n; i++)
            if (R[i]) clique.push_back(i);
        allCliques.push_back(clique);
        return;
    }

    // Choose pivot: vertex in P | X with most neighbors in P
    int pivot = -1, best = -1;
    bitset<MAXN> PX = P | X;
    for (int u = 0; u < n; u++) {
        if (!PX[u]) continue;
        int cnt = (int)(P & adj[u]).count();
        if (cnt > best) {
            best = cnt;
            pivot = u;
        }
    }

    // Iterate over P \ N(pivot)
    bitset<MAXN> candidates = P & ~adj[pivot];
    for (int v = 0; v < n; v++) {
        if (!candidates[v]) continue;

        bitset<MAXN> newR = R;  newR.set(v);
        bitset<MAXN> newP = P & adj[v];
        bitset<MAXN> newX = X & adj[v];

        bronKerbosch(newR, newP, newX);

        P.reset(v);
        X.set(v);
    }
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    cin >> n >> m;
    for (int i = 0; i < m; i++) {
        int u, v;
        cin >> u >> v;
        adj[u].set(v);
        adj[v].set(u);
    }

    bitset<MAXN> R, P, X;
    for (int i = 0; i < n; i++) P.set(i);

    bronKerbosch(R, P, X);

    cout << "Number of maximal cliques: " << allCliques.size() << "\n";
    for (auto& c : allCliques) {
        for (int v : c) cout << v << " ";
        cout << "\n";
    }
    return 0;
}
Complexity analysis: With the pivot optimization, Bron-Kerbosch runs in O(3n/3) time. The key insight is that each recursive call either adds a vertex to R or prunes a branch. The branching factor is bounded by the Moon-Moser graphs that achieve the maximum number of maximal cliques. In practice, for sparse graphs, the algorithm is dramatically faster.

Degeneracy Ordering Variant

For sparse graphs, the degeneracy ordering variant achieves O(d · n · 3d/3) where d is the graph’s degeneracy (maximum minimum-degree over all subgraphs). Since d is often much smaller than n, this is the preferred variant for competitive programming on sparse graphs.

The idea: process vertices in degeneracy order. For each vertex v, run Bron-Kerbosch on the subgraph induced by v and its later neighbors. Since each vertex has at most d later neighbors, each subproblem is small.

Spanning Tree Enumeration

Given a connected graph G with n vertices and m edges, the number of spanning trees can be computed via Kirchhoff’s theorem (determinant of any cofactor of the Laplacian matrix). But sometimes we need to enumerate all spanning trees explicitly — for example, to find the one minimizing some secondary objective.

The number of spanning trees can be exponential (a complete graph Kn has nn−2 spanning trees by Cayley’s formula). An output-sensitive algorithm should enumerate all τ spanning trees in O(poly(n, m) · τ) time — ideally O(n + m + nτ) or better.

Algorithm Sketch

The classic approach by Gabow and Myers (1978) enumerates all spanning trees with O(n + m + nτ) total time and O(n + m) space. The idea is:

  1. Start with an arbitrary spanning tree T.
  2. For each non-tree edge e, swapping e into T creates a unique fundamental cycle. Removing any tree edge on that cycle gives a new spanning tree.
  3. Use a DFS-based search through the “exchange graph” to visit every tree exactly once, using efficient data structures to find exchange edges in amortized O(1).

The core operation is an edge swap: given a spanning tree T and a non-tree edge (u, v), adding (u, v) creates a fundamental cycle. Removing any tree edge on that cycle yields a new spanning tree. A DFS over the “swap graph” visits every tree exactly once.

Practical note: A full implementation of Gabow-Myers is complex. In CP, if you only need to count spanning trees, use Kirchhoff’s theorem (Laplacian determinant). If explicit enumeration is needed, the output is usually bounded small enough that a careful swap-based DFS with O(n) per tree suffices.

Reverse Search Technique

Reverse search, introduced by Avis and Fukuda (1996), is a powerful general framework for output-sensitive enumeration. The core idea is strikingly simple:

  1. Define a local search function f that, given any feasible solution, returns an “improved” (or canonical) neighbor. This function must converge: repeated application from any solution eventually reaches a unique root solution.
  2. The function f defines an implicit rooted tree (the “reverse search tree”) over all feasible solutions, where the parent of solution S is f(S).
  3. Enumerate all solutions by traversing this tree via DFS, without ever storing the entire tree.

The beauty of reverse search is that it requires only O(poly(n)) space (the DFS stack) regardless of the number of solutions, and the amortized time per solution is O(poly(n)) — the cost of evaluating f and listing neighbors at each node.

Example: Enumerating Vertices of a Polytope

Given a convex polytope defined by m inequalities in d dimensions, reverse search can enumerate all v vertices using only O(d · m) space and O(d · m · v) total time. The local search function is a simplex pivot that decreases a generic objective — this defines a tree rooted at the optimal vertex.

Example: Enumerating Triangulations

Given n points in the plane, the number of triangulations can be exponential. Reverse search enumerates all of them by defining f as the “flip toward lexicographic minimum” — among all edge flips that produce a valid triangulation, choose the one that lexicographically decreases the edge set. The root is the lexicographically smallest triangulation (the Delaunay triangulation under generic conditions).

Why reverse search matters for CP: Many constructive problems ask you to enumerate or count configurations. Even if you don’t implement full reverse search, the mental model — define a parent function that creates an implicit tree over the solution space — is extremely useful for designing complete enumeration without duplicates.

Generating Combinatorial Objects

A major branch of output-sensitive algorithms deals with generating all objects of a given combinatorial class: permutations, subsets, partitions, binary strings, etc. The gold standard is CAT — Constant Amortized Time per object, meaning total time O(k) for k objects (after preprocessing).

Generating All Permutations — Heap’s Algorithm

Heap’s algorithm generates all n! permutations of an array using only adjacent swaps. It achieves CAT — each permutation differs from the previous one by exactly one swap, and the total number of swaps is n! − 1. 

#include <bits/stdc++.h>
using namespace std;

int n;
int a[10];
long long count_perms = 0;

// Process one permutation (here we just count; replace with your logic)
void process() {
    count_perms++;
    // To print: for (int i = 0; i < n; i++) cout << a[i] << " \n"[i==n-1];
}

// Heap's algorithm: generates all n! permutations
void heapPermute(int k) {
    if (k == 1) {
        process();
        return;
    }
    heapPermute(k - 1);
    for (int i = 0; i < k - 1; i++) {
        if (k % 2 == 0)
            swap(a[i], a[k - 1]);
        else
            swap(a[0], a[k - 1]);
        heapPermute(k - 1);
    }
}

int main() {
    cin >> n;
    for (int i = 0; i < n; i++) a[i] = i + 1;
    heapPermute(n);
    cout << "Total permutations: " << count_perms << "\n";
    return 0;
}

Generating All Subsets — Gray Code

The binary reflected Gray code generates all 2n subsets such that consecutive subsets differ by exactly one element. This is the subset analog of Heap’s algorithm and is CAT.

The key observation is that the position of the bit to flip for the i-th transition is the position of the lowest set bit of i (i.e., i & (-i) gives the mask, and __builtin_ctz(i) gives the index). This yields an extremely simple iterative implementation: 

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    int total = 1 << n;

    // Gray code enumeration of all 2^n subsets
    int gray = 0;
    for (int i = 0; i < total; i++) {
        // Process current subset represented by 'gray'
        // Example: print the subset
        cout << "{";
        bool first = true;
        for (int j = 0; j < n; j++) {
            if (gray & (1 << j)) {
                if (!first) cout << ", ";
                cout << j;
                first = false;
            }
        }
        cout << "}\n";

        // Advance to next Gray code
        if (i + 1 < total) {
            int bit = (i + 1) & -(i + 1); // lowest set bit of (i+1)
            gray ^= bit;
        }
    }
    return 0;
}

Generating Integer Partitions

An integer partition of n is a way of writing n as a sum of positive integers (order doesn’t matter). The number of partitions p(n) grows like eπ√(2n/3) / (4n√3). Generating all partitions in anti-lexicographic order can be done with amortized O(1) time per partition using Kelleher and O’Sullivan’s algorithm: 

#include <bits/stdc++.h>
using namespace std;

int n;
long long count_parts = 0;

void process(vector<int>& parts, int len) {
    count_parts++;
    // Uncomment to print:
    // for (int i = 0; i < len; i++) cout << parts[i] << " \n"[i==len-1];
}

// Generate all partitions of n in anti-lexicographic order
// Amortized O(1) per partition
void generatePartitions(int n) {
    vector<int> a(n + 1);
    int k = 1;
    a[1] = n;
    while (k != 0) {
        int x = a[k - 1] + 1;
        int y = a[k] - 1;
        k--;
        while (x <= y) {
            a[k] = x;
            y -= x;
            k++;
        }
        a[k] = x + y;
        process(a, k + 1);
    }
}

int main() {
    cin >> n;
    generatePartitions(n);
    cout << "Total partitions of " << n << ": " << count_parts << "\n";
    return 0;
}
CAT algorithms in CP: While contests rarely ask you to enumerate all permutations (n! is too large for n > 12), these techniques become crucial when the problem constrains the output. For instance, “list all partitions of n into distinct parts” or “enumerate all subsets summing to S” are feasible when the output is small. Understanding CAT algorithms helps you design solutions that scale with output size rather than input size.

CP Applications & Problems

Output-sensitive thinking shows up in competitive programming more often than you might expect. Here are the key patterns:

Pattern 1: “Print All X” Problems

When a problem says “print all solutions,” your first question should be: how large can the output be? If the output can be exponential, you need an output-sensitive approach. If the problem guarantees the output fits in the time limit, then the key is to ensure your algorithm’s cost is proportional to what you print.

Pattern 2: Pruned Search Trees

Many CP problems involve backtracking where the running time depends on the number of valid solutions rather than the entire search space. Recognizing this means you can confidently submit solutions that look exponential but are actually fast because the problem’s structure limits the output.

Pattern 3: Incremental Construction

Some problems ask you to build the output incrementally: “construct a sequence with property P” where the sequence length is the output. If your per-element cost is O(poly(n)), the total cost scales with output size.

Pattern 4: Convex Hull Trick and Geometry

In geometry problems, the output (hull vertices, intersection points, etc.) is often much smaller than the input. Using output-sensitive algorithms like Jarvis march or output-sensitive line segment intersection (Balaban’s algorithm: O(n log n + k)) can be the difference between TLE and AC.

Example: Enumerate All Bridges in a Graph

Finding all bridges in a graph with Tarjan’s algorithm is O(n + m) regardless of the number of bridges. This is output-sensitive in the sense that it doesn’t get slower when there are many bridges — but it’s also never slower than linear. A truly output-sensitive variant would be useful if we only wanted to count bridges without listing them, but since the listing cost is O(n + m + k) ≤ O(n + m), the distinction doesn’t matter here.

Example: All Pairs Shortest Paths When Output Is Small

Johnson’s algorithm computes all-pairs shortest paths in O(n2 log n + nm). But if you only need shortest paths between k specific pairs, you can run k instances of Dijkstra for O(k(n log n + m)), which is output-sensitive in k. When k « n2, this is much faster.

Takeaway: Output-sensitive algorithms are not a single technique but a mindset. When analyzing your solution’s complexity, always ask: does the running time depend on the output size? If the output is small, can I exploit that? If the output is large, is my per-item cost minimal? This thinking will help you choose the right algorithm and convince yourself (and the judge) that your solution is fast enough.
