    Invariants & Monovariants

    July 2026 · 25 min read

    Some of the most elegant arguments in competitive programming rely on finding a hidden quantity that never changes under operations (an invariant) or one that always moves in one direction (a monovariant). This post builds the theory from scratch and applies it to contest problems.

    Invariants — Quantities That Never Change

    An invariant is a property of the system state that remains constant across every legal operation. If the initial state has invariant value V and a target state requires a different value V′, then the target is unreachable — no matter how many operations you perform.

    Parity as an Invariant

    The most common invariant is parity (odd/even). Consider an array of integers where the only allowed operation is: pick two elements and add 1 to both. Each operation increases the total sum by exactly 2, so the parity of the sum never changes. If the initial sum is odd, you can never make all elements equal when that would require an even sum. 

    // Check whether we can make all elements equal
// Operation: pick any two indices, add 1 to both
bool canMakeEqual(vector<int>& a, int target) {
    int n = a.size();
    long long currentSum = 0;
    for (int x : a) currentSum += x;
    long long targetSum = (long long)target * n;

    // Each operation adds exactly 2 to the sum
    // so (targetSum - currentSum) must be non-negative and even
    long long diff = targetSum - currentSum;
    if (diff < 0 || diff % 2 != 0) return false;

    // Also each element needs a[i] <= target
    for (int x : a) {
        if (x > target) return false;
    }
    // Additional check: we need enough "pairs" of increments
    // Each operation increments two elements, so we need
    // sum of (target - a[i]) to be even (already checked) and
    // no single element needs more than half the total increments
    long long maxNeed = 0;
    for (int x : a) maxNeed = max(maxNeed, (long long)(target - x));
    return maxNeed <= diff / 2;
}

    The Mutilated Chessboard

    A classic example: remove two diagonally opposite corners of an 8×8 chessboard. Can the remaining 62 squares be tiled by 31 dominoes (each covering two adjacent squares)? The answer is no. Color the board in the standard checkerboard pattern. Each domino always covers exactly one black and one white square. After removing two opposite corners (which share the same color), we have 30 squares of one color and 32 of the other. An invariant of every partial tiling is that the number of covered black squares equals the number of covered white squares. Since 30 ≠ 32, a complete tiling is impossible.

    Strategy for finding invariants. Look at what each operation changes, then ask: is there a function f(state) that is preserved? Common candidates: parity, sum modulo k, XOR of all elements, number of inversions mod 2, and multiset-level properties like the GCD of all elements.

    XOR Invariant

    In Nim and many other games, the XOR of pile sizes is an invariant of the “losing position” class. But XOR invariants appear in constructive problems too. Suppose we have an array and the operation is: pick three distinct indices and XOR each of those elements with some value v. Then the XOR of the entire array changes by v ^ v ^ v = v. But if the operation is pick two elements and XOR each with v, the total XOR changes by v ^ v = 0 — the total XOR is invariant. 

    // Given operations that XOR pairs of elements with v,
// check if a target configuration is reachable
bool isReachable(vector<int>& initial, vector<int>& target) {
    int xorInit = 0, xorTarget = 0;
    for (int x : initial)  xorInit  ^= x;
    for (int x : target)   xorTarget ^= x;
    // Invariant: total XOR is preserved
    return xorInit == xorTarget;
}

    Monovariants — Quantities That Only Move One Way

    A monovariant is a function f(state) that strictly increases (or strictly decreases) with every operation, and is bounded. Monovariants give you two things at once:

    • Termination: the process must eventually stop, because a bounded integer-valued function cannot increase (or decrease) forever.
    • Bound on steps: if f starts at a, ends at at most b, and increases by at least 1 each step, the process takes at most b − a steps.

    The Euclidean Algorithm as a Monovariant

    The Euclidean algorithm replaces (a, b) with (b, a mod b). The monovariant a + b strictly decreases each step (since a mod b < a) and is bounded below by 0, guaranteeing termination. 

    // The Euclidean algorithm: a + b is a monovariant (strictly decreasing)
int gcd(int a, int b) {
    // Monovariant: a + b strictly decreases each step
    // Proof: if a >= b, then new pair is (b, a%b)
    //   new sum = b + a%b < b + a = a + b  (since a%b < a when b > 0)
    // Bounded below by gcd(a,b) + 0, so it terminates.
    while (b) {
        a %= b;
        swap(a, b);
    }
    return a;
}
    Monovariant recipe. (1) Define a function f on the state space. (2) Show f strictly increases or decreases after each operation. (3) Show f is bounded. Conclude: the process terminates in at most O(range of f) steps.

    The Extremal Principle

    The extremal principle is a proof technique that works hand-in-hand with monovariants. The idea: among all states reachable by the process, consider the one that minimizes (or maximizes) the monovariant. Since the monovariant is strictly decreasing, the extremal state must be a fixed point — no further operation is applicable. Then you argue that this fixed point has the desired property.

    Example: Convex Hull Shrinking

    Given n points in the plane, remove any point strictly inside the convex hull of the rest. The monovariant is the point count (strictly decreasing, bounded below). The extremal state — when no more removals are possible — is exactly the set of convex hull vertices, regardless of removal order.

    Example: Tournament Edge Flipping

    In a tournament on n vertices, pick any vertex with out-degree 0 and reverse all its edges. The monovariant: the number of edges going from lower to higher index. It strictly increases and is bounded by n(n−1)/2, so the process terminates at a state where every vertex has positive out-degree.

    Potential Functions & Amortized Analysis

    A potential function Φ assigns a non-negative real number to each state. The amortized cost of an operation is its actual cost plus the change in potential: ĉi = ci + Φ(Si) − Φ(Si−1). If Φ never goes negative, the total amortized cost is an upper bound on the total actual cost. This is exactly the monovariant idea applied to algorithm analysis.

    Classic Example: Chip-Firing on Graphs

    Place chips on the vertices of a graph. A vertex v can fire if it has at least deg(v) chips: it sends one chip along each edge. Key observations:

    • The total number of chips is invariant (firing just redistributes).
    • Let Φ = ∑ cv2 where cv is the chip count at vertex v. When a vertex with a large number of chips fires, it sends chips to neighbors, reducing the “imbalance.” Under the right conditions, Φ is a monovariant (strictly decreasing), proving termination.
    • The final configuration is independent of firing order (the “abelian” property of chip-firing).
    // Chip-firing simulation on a graph
// Returns the final stable configuration
vector<int> chipFiring(int n, vector<vector<int>>& adj, vector<int> chips) {
    // Invariant: total number of chips is constant
    // Monovariant: sum of squares decreases (or max chip count decreases)
    // until no vertex can fire
    bool fired = true;
    while (fired) {
        fired = false;
        for (int v = 0; v < n; v++) {
            int deg = adj[v].size();
            if (chips[v] >= deg) {
                // Fire vertex v: send one chip along each edge
                chips[v] -= deg;
                for (int u : adj[v]) {
                    chips[u]++;
                }
                fired = true;
            }
        }
    }
    return chips;
}
    Chip-firing termination. If the total number of chips is less than the number of edges, the process is guaranteed to terminate. The monovariant is the sum ∑ cv², which strictly decreases when a “heavy” vertex fires. The abelian property means the final state is unique.

    Classic Problem: Water Pouring

    You have two jugs with capacities a and b liters (both integers). Allowed operations: fill a jug completely, empty a jug completely, or pour from one jug into the other until the source is empty or the destination is full. Can you measure exactly t liters?

    By Bézout’s identity, the measurable quantities are exactly the multiples of gcd(a, b) up to max(a, b). The invariant: every reachable state has each jug containing a multiple of gcd(a, b) liters. 

    // BFS to find all reachable water states with two jugs
// Returns whether we can measure exactly target liters
#include <bits/stdc++.h>
using namespace std;

bool canMeasure(int a, int b, int target) {
    if (target > max(a, b)) return false;
    // Invariant: all reachable amounts are multiples of gcd(a, b)
    if (target % __gcd(a, b) != 0) return false;

    // BFS over states (x, y) where x in [0,a], y in [0,b]
    set<pair<int,int>> visited;
    queue<pair<int,int>> q;
    q.push({0, 0});
    visited.insert({0, 0});

    while (!q.empty()) {
        auto [x, y] = q.front(); q.pop();
        if (x == target || y == target) return true;

        // Generate all possible next states
        vector<pair<int,int>> nextStates = {
            {a, y}, {x, b},       // fill jug 1 or 2
            {0, y}, {x, 0},       // empty jug 1 or 2
            // pour jug 1 -> jug 2
            {x - min(x, b - y), y + min(x, b - y)},
            // pour jug 2 -> jug 1
            {x + min(y, a - x), y - min(y, a - x)}
        };

        for (auto& s : nextStates) {
            if (!visited.count(s)) {
                visited.insert(s);
                q.push(s);
            }
        }
    }
    return false;
}

int main() {
    int a, b, t;
    cin >> a >> b >> t;
    cout << (canMeasure(a, b, t) ? "YES" : "NO") << "\n";
    return 0;
}

    Competitive Programming Applications

    Invariant and monovariant arguments show up constantly in constructive and ad-hoc problems. Here are the main patterns.

    Pattern 1: Impossibility via Invariant

    If you need to prove that a target state is unreachable, find a function f such that every operation preserves f but f(initial) ≠ f(target).

    Example (CF 1365C): You have an array of 0s and 1s. The operation is to swap two adjacent elements. Can you sort the array? The invariant is the count of 1s (swaps don’t change it). If the count of 1s matches, the array is always sortable. But in harder variants where operations are restricted (e.g., swap only if both are different), parity of the number of inversions becomes the relevant invariant.

    Pattern 2: Termination via Monovariant

    To prove that a constructive algorithm halts, find a non-negative integer-valued function that strictly decreases with each step.

    Example: Replace any pair (a, b) in an array where a > b and they are adjacent with (a − b, b). The monovariant is the sum of all elements, which strictly decreases with each step and is bounded below by n · 1 (assuming positive elements). So the process terminates.

    Pattern 3: Constructive Solution via Greedy + Monovariant

    Many constructive problems ask you to build a sequence of operations. The strategy: define a monovariant that measures “distance to the goal,” then greedily pick the operation that improves it the most. 

    // Problem: Given an array, make it sorted using adjacent swaps.
// Monovariant: inversion count, strictly decreases with each swap.
// Strategy: bubble sort — each swap fixes one inversion.

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> a(n);
    for (int& x : a) cin >> x;

    vector<pair<int,int>> ops;
    bool sorted = false;
    while (!sorted) {
        sorted = true;
        for (int i = 0; i + 1 < n; i++) {
            if (a[i] > a[i + 1]) {
                swap(a[i], a[i + 1]);
                ops.push_back({i, i + 1});
                sorted = false;
            }
        }
    }

    cout << ops.size() << "\n";
    for (auto [i, j] : ops) {
        cout << "swap " << i + 1 << " " << j + 1 << "\n";
    }
    return 0;
}

    Pattern 4: Bounding the Number of Steps

    When a problem asks “what is the maximum number of operations before the process stops?”, find a monovariant and compute its range.

    Example: Start with the array [n, n−1, …, 1]. The operation: pick any adjacent pair that is out of order and swap them (bubble sort step). The monovariant is the inversion count. The initial inversion count is n(n−1)/2. Each swap reduces it by exactly 1. So the process terminates in exactly n(n−1)/2 steps.

    Advanced Techniques

    Multi-Dimensional Monovariants

    Sometimes no single scalar works, but a tuple of values does. Define a lexicographic order on tuples and show the tuple strictly decreases. For example, in the simplex algorithm, the tuple (objective value, basis vector) under Bland’s rule is a lexicographic monovariant, proving termination even in degenerate cases. 

    // Multi-dimensional monovariant example:
// Process: given (a, b, c) with a >= b >= c > 0,
// replace with (a - b, b, c) sorted in decreasing order.
// Monovariant: (a, b, c) in lexicographic order, strictly decreasing.
// This is essentially a multi-number GCD computation.

#include <bits/stdc++.h>
using namespace std;

int main() {
    long long a, b, c;
    cin >> a >> b >> c;

    int steps = 0;
    // Sort in decreasing order
    vector<long long> v = {a, b, c};
    sort(v.rbegin(), v.rend());

    while (v[0] != v[1] || v[1] != v[2]) {
        // Replace largest with (largest - second largest)
        // Optimization: use modulo to skip repeated subtractions
        if (v[1] == 0) break;
        long long times = (v[0] - 1) / v[1]; // at least 1
        if (times == 0) times = 1;
        v[0] -= v[1] * times;
        steps += times;

        // Re-sort
        sort(v.rbegin(), v.rend());

        // The tuple (v[0], v[1], v[2]) is lexicographically
        // strictly smaller than before each iteration
    }

    cout << "Final state: " << v[0] << " " << v[1] << " " << v[2] << "\n";
    cout << "Steps (approx): " << steps << "\n";
    return 0;
}

    Energy Functions for Grid Problems

    In grid-based problems, define an “energy” function such as the sum of Manhattan distances from each piece to its target. Each legal move reduces the energy by at least 1, giving both termination and a step bound. The 15-puzzle combines both ideas: an invariant (permutation parity + blank row) partitions reachable states, while Manhattan distance serves as an admissible A* heuristic (a monovariant that never overestimates).

    Balanced Operations and Modular Invariants

    When the operation changes the state by a fixed amount modulo some m, the state modulo m is an invariant. This is especially useful when m = 3 (many contest problems use mod-3 invariants because contestants only check parity). 

    // Modular invariant: replace any pair (a, b) with a + b.
// Invariant: total sum mod 3 is preserved.
// Final single value mod 3 = initial sum mod 3.

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> a(n);
    int totalMod3 = 0;
    for (int i = 0; i < n; i++) {
        cin >> a[i];
        totalMod3 = (totalMod3 + a[i] % 3 + 3) % 3;
    }

    cout << "Final value mod 3: " << totalMod3 << "\n";

    // Simulate to verify
    while (a.size() > 1) {
        int x = a.back(); a.pop_back();
        int y = a.back(); a.pop_back();
        a.push_back(x + y);
    }
    cout << "Actual final value: " << a[0] << "\n";
    cout << "Actual mod 3: " << ((a[0] % 3) + 3) % 3 << "\n";
    return 0;
}

    Complete Contest Solutions

    Solution 1: Array Equalization (Invariant + Construction)

    Problem: Given an array of n integers, the operation is: pick any two elements and replace them with their absolute difference and their sum (i.e., (a, b) → (|a−b|, a+b)). Can you make all elements equal?

    Invariant: Each operation preserves the total sum modulo 2. We have |a−b| + (a+b) = 2·max(a,b), so the new sum is always even if and only if max(a,b) has the right parity. More usefully, notice that a+b and |a−b| have the same parity. So parity of each element is the same after the operation. The invariant: the GCD of all elements divides every reachable value. 

    // Can we equalize? Check the GCD-based invariant.
// Operation: (a, b) -> (|a-b|, a+b)
// Invariant: gcd of all elements is preserved
#include <bits/stdc++.h>
using namespace std;

int main() {
    int t;
    cin >> t;
    while (t--) {
        int n;
        cin >> n;
        vector<int> a(n);
        int g = 0;
        for (int& x : a) {
            cin >> x;
            g = __gcd(g, x);
        }
        // All elements can be reduced to multiples of g.
        // If all a[i] / g have the same parity, answer is YES.
        bool allSame = true;
        int par = (a[0] / g) % 2;
        for (int x : a) {
            if ((x / g) % 2 != par) { allSame = false; break; }
        }
        cout << (allSame ? "YES" : "NO") << "\n";
    }
    return 0;
}

    Solution 2: Reduce to GCD (Monovariant Construction)

    Problem: Given an array of positive integers, the operation is: pick any two elements ai > aj and replace ai with ai − aj. What is the minimum possible maximum element?

    Answer: gcd of the entire array. The operation is exactly the step in the extended Euclidean algorithm. The GCD is an invariant (subtracting preserves divisibility), and the maximum element is a monovariant (strictly decreasing). The process terminates when all elements equal gcd. 

    // Reduce array to all-GCD using subtraction operations
// Outputs the sequence of operations
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<long long> a(n);
    for (auto& x : a) cin >> x;

    // Monovariant: max element, strictly decreasing
    // Invariant: GCD of all elements
    long long g = 0;
    for (long long x : a) g = __gcd(g, x);

    cout << "Target (GCD): " << g << "\n";

    // Simulate: always pick max and subtract the min from it
    // Use modulo for efficiency (like Euclidean algorithm)
    int ops = 0;
    while (true) {
        long long mx = *max_element(a.begin(), a.end());
        long long mn = *min_element(a.begin(), a.end());
        if (mx == mn) break;

        // Find index of max
        int imax = max_element(a.begin(), a.end()) - a.begin();

        // Use modulo to speed up repeated subtraction
        if (a[imax] % mn == 0) {
            a[imax] = mn; // would become 0 then replaced by mn conceptually
        } else {
            a[imax] = a[imax] % mn;
        }
        ops++;
    }

    cout << "Operations: " << ops << "\n";
    cout << "Final array: ";
    for (long long x : a) cout << x << " ";
    cout << "\n";
    return 0;
}

    Solution 3: Token Passing on a Cycle (Monovariant + Invariant)

    Problem: n people stand in a circle, person i has ai tokens. Each second, everyone with ≥1 token simultaneously passes one token clockwise. Does the process stabilize?

    Invariant: Total tokens S = ∑ai is constant. Monovariant: The variance Φ = ∑(ai − S/n)² is non-increasing. The stable state has each person holding ⌊S/n⌋ or ⌈S/n⌉ tokens. 

    // Token passing on a cycle until stabilization
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<long long> a(n);
    long long total = 0;
    for (int i = 0; i < n; i++) {
        cin >> a[i];
        total += a[i];
    }

    // Invariant: sum = total (constant)
    // Monovariant: variance decreases until stable
    int steps = 0;
    while (true) {
        // Check stability: is the state a fixed point?
        vector<long long> next(n);
        bool changed = false;
        for (int i = 0; i < n; i++) {
            long long give = (a[i] > 0) ? 1 : 0;
            int prev = (i - 1 + n) % n;
            long long receive = (a[prev] > 0) ? 1 : 0;
            next[i] = a[i] - give + receive;
            if (next[i] != a[i]) changed = true;
        }
        if (!changed) break;
        a = next;
        steps++;
        if (steps > 1000000) break; // safety cap
    }

    cout << "Stabilized after " << steps << " steps\n";
    for (long long x : a) cout << x << " ";
    cout << "\n";

    // Verify invariant
    long long check = 0;
    for (long long x : a) check += x;
    assert(check == total && "Invariant violated!");
    return 0;
}
    Summary of techniques.
    • Invariant: a quantity preserved by every operation. Use it to prove impossibility (f(start) ≠ f(target) ⇒ unreachable).
    • Monovariant: a quantity that strictly changes in one direction and is bounded. Use it to prove termination and bound the number of steps.
    • Extremal principle: consider the state that optimizes the monovariant — it must be a fixed point with special structure.
    • Potential function: the continuous/amortized version of a monovariant, used for runtime analysis.
    When stuck on a constructive problem, ask: “What quantity is preserved? What quantity always goes down?” These two questions unlock a surprising number of contest problems.
