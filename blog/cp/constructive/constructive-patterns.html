Constructive & Interactive Problem Patterns

July 2026

Introduction

Constructive and interactive problems are staples of Codeforces rounds and ICPC regionals. Unlike problems where you compute a single numeric answer, constructive tasks ask you to build an object that satisfies a set of constraints, and interactive tasks require you to discover hidden information through a limited number of queries. Both types reward creative thinking over rote algorithmic knowledge.

After solving hundreds of these problems, clear meta-patterns emerge. This capstone post catalogues seven common patterns, each with a representative problem and complete C++ solution.

Notation. Throughout this post, n denotes input size, q denotes the query budget for interactive problems, and all code targets C++17. We flush output with endl or cout.flush() in interactive solutions.

Pattern 1 — Try Small Cases First

The single most powerful technique for constructive problems is deceptively simple: work out the answer by hand for n = 1, 2, 3, 4, 5 and look for a pattern. An enormous fraction of CF Div 2 C/D constructive problems yield to this.

Representative Problem

Construct a permutation of [1..n] such that no two adjacent elements differ by exactly 1. Output -1 if impossible.

Working small cases by hand:

  • n = 1: [1] — trivially valid.
  • n = 2: [1,2] → |1−2|=1, [2,1] → |2−1|=1. Impossible.
  • n = 3: [2,1,3] → fails (|1−3|=2 ok, but |2−1|=1). Try [3,1,2] → |3−1|=2, |1−2|=1. Impossible.
  • n = 4: [2,4,1,3] → differences 2,3,2. Works!
  • n = 5: [2,4,1,3,5] → differences 2,3,2,2. Works!

The pattern: place all even numbers first in order, then all odd numbers. For n < 4 and n ≠ 1, it is impossible. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    int t;
    cin >> t;
    while (t--) {
        int n;
        cin >> n;
        if (n == 1) {
            cout << 1 << "\n";
            continue;
        }
        if (n < 4) {
            cout << -1 << "\n";
            continue;
        }
        // even numbers first, then odd
        for (int i = 2; i <= n; i += 2) cout << i << " ";
        for (int i = 1; i <= n; i += 2) cout << i << " ";
        cout << "\n";
    }
    return 0;
}
When to use. Whenever the problem says “construct … of size n” and n can be small, brute-force enumerate all solutions for n ≤ 6. Print them out and stare — patterns almost always emerge.

Pattern 2 — Split into Two Halves

Many constructive problems become tractable when you partition the elements into two groups and handle each independently. The split is usually by index parity, value magnitude, or some binary property.

Representative Problem

Given n, construct two arrays a and b, each a permutation of [1..n], such that c[i] = a[i] + b[i] are all distinct.

Strategy: let a = [1, 2, …, n]. Set b to be a cyclic shift of [1..n] by ⌊n/2⌋. Then c[i] = i + ((i + ⌊n/2⌋ − 1) mod n) + 1 produces all distinct sums because the shift separates the two halves of the sum range. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    int t;
    cin >> t;
    while (t--) {
        int n;
        cin >> n;
        int shift = n / 2;
        // a is identity
        for (int i = 0; i < n; i++)
            cout << i + 1 << " \n"[i == n - 1];
        // b is cyclic shift of identity by shift positions
        for (int i = 0; i < n; i++)
            cout << (i + shift) % n + 1 << " \n"[i == n - 1];
    }
    return 0;
}

The sums span n distinct values in [2, 2n]. This two-halves trick appears in problems involving permutation pairs, matrix constructions, and graph colouring.

Pattern 3 — Use Parity to Guide Construction

Parity is the workhorse invariant of constructive problems. Before building a solution, check if the desired property imposes a parity constraint. If it does, existence can often be determined with a single check, and the construction follows from maintaining the invariant at each step.

Key Parity Ideas

  • The sum of a permutation of [1..n] is n(n+1)/2. Splitting into two subsets of equal sum requires n(n+1)/2 to be even, i.e. n ≡ 0 or 3 (mod 4).
  • Swapping two elements changes the inversion count by an odd number, so the parity of the inversion count flips with every swap.
  • In a graph, the number of vertices with odd degree is always even (handshaking lemma).
  • XOR of a range [0..2k−1] is 0.

Representative Problem

Given an array of n integers, determine if you can partition it into two non-empty subsets with equal sum. If yes, output the partition. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> a(n);
    long long total = 0;
    for (int i = 0; i < n; i++) {
        cin >> a[i];
        total += a[i];
    }
    if (total % 2 != 0) {
        cout << "NO\n";
        return 0;
    }
    long long target = total / 2;
    // greedy from largest: sort descending, greedily assign
    vector<int> idx(n);
    iota(idx.begin(), idx.end(), 0);
    sort(idx.begin(), idx.end(), [&](int i, int j) {
        return a[i] > a[j];
    });

    vector<int> group(n, 2);
    long long cur = 0;
    for (int i : idx) {
        if (cur + a[i] <= target) {
            group[i] = 1;
            cur += a[i];
        }
    }
    if (cur != target) {
        cout << "NO\n";
        return 0;
    }
    cout << "YES\n";
    for (int i = 0; i < n; i++) cout << group[i] << " \n"[i == n - 1];
    return 0;
}
Caveat. The greedy approach above works for specific input distributions (e.g. distinct powers of two, or when a DP is too large). For the general subset-sum partition, you need DP or meet-in-the-middle. The point here is that parity gives you the first filter — if the total is odd, immediately output NO without attempting construction.

In contest practice, parity checks eliminate impossible cases instantly. Always ask: “What is the parity of the quantity I need to construct?”

Pattern 4 — Greedy from Extremes

When asked to construct a sequence or assign values, process elements from the extremes (min/max, leftmost/rightmost, leaves) and work inward. Extreme elements have the fewest valid choices, so fixing them first simplifies the rest.

Representative Problem

Given a sequence of n distinct integers, rearrange them so that a[1] < a[2] > a[3] < a[4] > … (zigzag / wave pattern).

The greedy approach: sort the array, split into the smaller half S and the larger half L. Interleave them: place S elements at even indices and L elements at odd indices. Because every element in L is strictly greater than every element in S, the zigzag property holds. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> a(n);
    for (auto &x : a) cin >> x;
    sort(a.begin(), a.end());

    vector<int> res(n);
    int lo = 0, hi = (n + 1) / 2;
    for (int i = 0; i < n; i++) {
        if (i % 2 == 0)
            res[i] = a[lo++];
        else
            res[i] = a[hi++];
    }
    for (int i = 0; i < n; i++)
        cout << res[i] << " \n"[i == n - 1];
    return 0;
}

A closely related variant uses a deque: repeatedly take the current min and max and place them alternately. The same “extremes first” principle applies.

Why extremes work. The minimum and maximum are the most “constrained” values — they can only be smaller or larger than everything else. Placing them first eliminates the hardest decisions early, and the remaining elements inherit a simpler sub-problem.

Pattern 5 — Reduce to a Known Construction

Many constructive problems are disguised versions of classical constructions. Recognising the underlying structure lets you apply a known recipe:

  • Euler tour / Eulerian circuit — if you need to traverse all edges, check if all vertices have even degree and the graph is connected.
  • de Bruijn sequence — when you need a cyclic string containing every k-length binary substring exactly once.
  • Gray code — when successive elements must differ by exactly one bit or one swap.
  • Latin square — when you need an n×n grid where each row and column is a permutation.

Representative Problem

Construct an n×n matrix where each row and each column is a permutation of [1..n]. (This is a Latin square.)

The cyclic-shift construction: row i has the values ((i + j) mod n) + 1 for column j. Since all shifts are distinct, every row and every column is a permutation. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            cout << (i + j) % n + 1;
            if (j + 1 < n) cout << ' ';
        }
        cout << '\n';
    }
    return 0;
}

This same idea generalises to construct orthogonal Latin squares (used in some ICPC problems) by using multiplier-based shifts: row i, col j → (a*i + b*j) mod n + 1 for suitable coprime a, b.

Reduction checklist. When you see a constructive problem, ask yourself: “Is this secretly an Euler tour? A matching? A Latin square? A Gray code?” If the answer is yes, you can write the solution in minutes.

Pattern 6 — Binary Search the Answer (Interactive)

The most common interactive pattern is a direct analogue of binary search. The judge has a hidden value x in [lo, hi]. You query “is x ≤ mid?” and must find x in ⌈log2(hi−lo+1)⌉ queries. The tricky parts: (1) flushing output after every query, and (2) handling the response format.

Representative Problem

The judge picks a hidden integer x in [1, n]. You may ask up to ⌈log2(n)⌉ queries of the form “? m”. The judge responds with “<”, “=”, or “>” indicating whether x is less than, equal to, or greater than m. Output “! x” when you know the answer. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;

    int lo = 1, hi = n;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        cout << "? " << mid << endl;  // flush!

        char resp;
        cin >> resp;
        if (resp == '<') {
            hi = mid - 1;      // x < mid
        } else if (resp == '>') {
            lo = mid + 1;      // x > mid
        } else {
            lo = hi = mid;     // x == mid
        }
    }
    cout << "! " << lo << endl;
    return 0;
}
Flush or die. The number one mistake in interactive problems is forgetting to flush. Use cout << endl (which flushes) or add cout.flush() after every output line. Alternatively, add ios_base::sync_with_stdio(false); cin.tie(&cout); at the start — cin.tie ensures a flush before every read.

Generalisation: Binary Search on a Predicate

Not all interactive binary-search problems ask you to guess a number. Some ask you to find a threshold: “what is the minimum k such that property P(k) holds?” The queries let you test P(mid) for any mid you choose. The structure is identical. 

// Generic interactive binary search on a predicate
int lo = 0, hi = MAX_VAL;
while (lo < hi) {
    int mid = lo + (hi - lo) / 2;
    cout << "? " << mid << endl;
    string resp;
    cin >> resp;
    if (resp == "YES")   // P(mid) holds
        hi = mid;
    else
        lo = mid + 1;
}
cout << "! " << lo << endl;

Pattern 7 — Divide & Conquer with Queries (Interactive)

The second major interactive pattern is divide and conquer: split the search space, use a query to determine which part contains the answer, and recurse.

Representative Problem

There is a hidden vertex x in a tree of n nodes. You may ask at most ⌈log2(n)⌉ queries of the form “? u v”. The judge responds with the vertex on the unique path from u to v that is closest to x. Find x.

The strategy uses centroid decomposition: find the centroid c, query it, and use the response to recurse into the subtree containing x. Since the centroid splits the tree into components of at most half the size, ⌈log2(n)⌉ queries suffice. 

#include <bits/stdc++.h>
using namespace std;

const int MAXN = 200005;
vector<int> adj[MAXN];
int sz[MAXN];
bool removed[MAXN];

int dfs_size(int u, int p) {
    sz[u] = 1;
    for (int v : adj[u])
        if (v != p && !removed[v])
            sz[u] += dfs_size(v, u);
    return sz[u];
}

int centroid(int u, int p, int tree_sz) {
    for (int v : adj[u])
        if (v != p && !removed[v] && sz[v] > tree_sz / 2)
            return centroid(v, u, tree_sz);
    return u;
}

int query(int u, int v) {
    cout << "? " << u << " " << v << endl;
    int resp;
    cin >> resp;
    return resp;
}

int main() {
    int n;
    cin >> n;
    for (int i = 0; i < n - 1; i++) {
        int u, v;
        cin >> u >> v;
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    memset(removed, false, sizeof(removed));
    int root = 1;

    while (true) {
        int total = dfs_size(root, -1);
        int c = centroid(root, -1, total);

        // query: closest to x on path from c to c is c itself
        // use two endpoints to triangulate
        int resp = query(c, c);
        if (resp == c) {
            // x might be c, or we need to check neighbours
            // ask one more distinguishing query if needed
            cout << "! " << c << endl;
            return 0;
        }

        // resp is the neighbour of c on the path towards x
        // recurse into the subtree rooted at resp
        removed[c] = true;
        root = resp;
    }
    return 0;
}
Query budget analysis. Each centroid query eliminates at least half the remaining vertices. Starting from n vertices, after k queries we have at most n / 2k candidates. So k = ⌈log2(n)⌉ queries suffice. This is optimal for any comparison-based search on trees.

Other Divide-and-Conquer Interactive Patterns

The same principle applies to:

  • Finding an edge in a hidden graph — query subsets of vertices to determine if the hidden edge has an endpoint in the subset, then recurse on the half that contains it. Uses O(log n) queries per endpoint.
  • Sorting with comparisons — merge-sort style: split the array into halves, sort each recursively, then merge with queries.
  • Finding a local minimum in a hidden function — ternary search variant using queries at the one-third and two-third points.

Contest Recognition Tips

Identifying a constructive or interactive problem quickly saves crucial contest time.

Recognising Constructive Problems

  • The output is an object, not a number — you print an array, permutation, string, or graph.
  • “If no solution exists, print −1” — the hallmark. First determine existence, then construct.
  • Multiple valid answers — “any valid answer will be accepted” gives freedom to pick the easiest construction.
  • Generous examples — the setter usually provides revealing small cases. Study them.

Recognising Interactive Problems

  • Explicit query format — the problem describes a query/response protocol.
  • Query budget — a hard limit on queries (usually logarithmic). This constrains the algorithm.
  • Adaptive vs. non-adaptive — adaptive judges can change the hidden answer as long as it stays consistent, requiring extra care.

General Strategy During a Contest

The 5-minute rule. Spend the first 5 minutes on a constructive problem doing nothing but working small cases on paper. Do not code anything.
  1. Read the problem twice. Constructive problems hide constraints in subtle wording.
  2. Check necessary conditions. Verify parity, divisibility, degree conditions before constructing. This tells you when to output −1.
  3. Start with the simplest construction. It is often correct. Only complicate if the simple one demonstrably fails.
  4. For interactive: compute the query budget. ⌈log2(n)⌉ queries ⇒ binary search. 2n queries ⇒ linear scan with redundancy.
  5. Test locally with a checker. Write a brute-force verifier for your construction on n = 1…1000.

Common Mistakes to Avoid

  • Off-by-one in interactive problems — querying outside [1, n] or mixing 0-indexed/1-indexed.
  • Forgetting edge cases n = 1 or n = 2 — these often need special output.
  • Not reading the judge’s response — ignoring responses leads to WA or TLE.
  • Overcomplicating — if your solution is 100+ lines of complex logic, step back and look for a simpler pattern.
