Adaptive Adversary Arguments — Lower Bounds & Worst-Case Analysis

July 2026

Introduction

How many comparisons does it really take to find the largest element in an array of n distinct numbers? You might intuitively say n − 1, but can you prove that no algorithm can do better? This is the domain of adversary arguments — a powerful technique for proving lower bounds on the number of operations any algorithm must perform to solve a problem.

The core idea is elegant: instead of analyzing a specific algorithm, we construct an adversary — a malicious oracle that answers each query in the worst possible way (while remaining consistent with some valid input). If the adversary can force every algorithm to ask at least k queries, then k is a lower bound for the problem.

In competitive programming, adversary arguments appear in two guises:

  • Theoretical tool: proving that your algorithm is optimal (e.g., ⌈log2 n⌉ comparisons for binary search).
  • Practical reality: in interactive problems, the judge literally is an adaptive adversary — it can choose its answers on the fly to maximize the number of queries you need.
Key insight: An adversary argument doesn’t analyze any particular algorithm. It shows that for every possible algorithm, there exists an input that forces it to use at least a certain number of operations. This is a universal lower bound.

The Adversary Method

Formally, an adversary argument proceeds as follows:

  1. We have a problem P that an algorithm solves by asking queries (comparisons, membership tests, etc.).
  2. We define an adversary strategy — a rule for answering each query. The adversary maintains a set S of inputs that are consistent with all answers given so far.
  3. For each query, the adversary answers in whichever way keeps S as large as possible (or, more precisely, keeps the problem as “unresolved” as possible).
  4. We argue that after fewer than k queries, the set S still contains inputs that require different outputs — so the algorithm cannot be correct with fewer than k queries.

The adversary is adaptive because its answers depend on the algorithm’s previous queries. It is consistent because at all times there exists at least one input matching every answer it has given. This consistency requirement is what separates a valid adversary argument from a hand-wave.

Information-Theoretic vs. Adversary Lower Bounds

A simpler lower-bound technique is information-theoretic: if there are N possible outputs and each query has at most b outcomes, then we need at least ⌈logb N⌉ queries. Adversary arguments are strictly stronger — they can prove bounds that information-theoretic arguments cannot, because they exploit the structure of the problem rather than just counting outcomes.

Information-Theoretic

  • Counts possible outputs
  • Easy to apply
  • Sometimes loose
  • E.g., sorting: log2(n!) = Θ(n log n)

Adversary Argument

  • Constructs worst-case input on the fly
  • Harder to design, but tighter
  • Can prove bounds information theory misses
  • E.g., second-largest: n + ⌈log2 n⌉ − 2

Searching Lower Bound

Let’s start with the simplest example: searching for a target value in a sorted array of n elements using comparisons.

Theorem: ⌈log2 n⌉ Comparisons Are Necessary

Adversary strategy: The adversary maintains an interval [lo, hi] of positions where the target could be. Initially lo = 0, hi = n − 1. When the algorithm probes position mid:

  • If mid < (lo + hi) / 2, the adversary says “target is to the right” (set lo = mid + 1).
  • If mid ≥ (lo + hi) / 2, the adversary says “target is to the left” (set hi = mid − 1).
  • In both cases the adversary picks the answer that keeps the larger half of the remaining interval.

After k queries, the remaining interval has size at least ⌈n / 2k⌉. For the algorithm to uniquely identify the target, this must equal 1, requiring k ≥ ⌈log2 n⌉.

Here is a simulation that demonstrates the adversary in action: 

#include <bits/stdc++.h>
using namespace std;

// Simulate an adversary for binary search.
// The adversary always forces the algorithm towards the larger half.
struct SearchAdversary {
    int lo, hi;
    int queries_used = 0;

    SearchAdversary(int n) : lo(0), hi(n - 1) {}

    // Returns "left" or "right" — always picks the larger remaining half.
    // answer: -1 means target < mid, +1 means target > mid, 0 means found.
    int query(int mid) {
        queries_used++;
        if (lo == hi) {
            // forced to reveal
            return 0;
        }
        int left_size = mid - lo;
        int right_size = hi - mid;
        if (right_size >= left_size) {
            // say target is to the right
            lo = mid + 1;
            return 1;
        } else {
            // say target is to the left
            hi = mid - 1;
            return -1;
        }
    }

    bool resolved() const { return lo == hi; }
};

int main() {
    for (int n : {8, 16, 100, 1000000}) {
        SearchAdversary adv(n);
        // Optimal algorithm: always probe the midpoint
        while (!adv.resolved()) {
            int mid = (adv.lo + adv.hi) / 2;
            adv.query(mid);
        }
        int lower_bound = (int)ceil(log2((double)n));
        cout << "n=" << n
             << "  queries=" << adv.queries_used
             << "  ceil(log2(n))=" << lower_bound << "\n";
    }
    // Output shows queries == ceil(log2(n)) every time.
    return 0;
}
Why this matters: Binary search is optimal among comparison-based search algorithms. No clever trick — ternary search, interpolation on unknown distributions, or any other scheme — can beat ⌈log2 n⌉ in the worst case when the adversary is adaptive.

Finding the Maximum

Problem: Given n distinct elements, find the maximum using pairwise comparisons. We claim n − 1 comparisons are both sufficient and necessary.

Upper Bound: n − 1 Suffices

A simple linear scan maintains a running maximum, comparing each new element against it. This uses exactly n − 1 comparisons.

Lower Bound: Adversary Proof

Adversary strategy: Call an element undefeated if it has never lost a comparison. The adversary’s rule is:

  • When two undefeated elements are compared, the adversary can answer either way — say the one with the larger label wins.
  • When an undefeated element is compared against a defeated one, the undefeated one wins.
  • When two defeated elements are compared, the adversary answers consistently with some fixed total order.

Key observation: Each comparison defeats at most one previously-undefeated element. Initially all n elements are undefeated. To identify the maximum, at most one element can remain undefeated. Therefore we need at least n − 1 comparisons to defeat the other n − 1 elements. ∎ 

#include <bits/stdc++.h>
using namespace std;

// Adversary for finding-maximum: tracks which elements are undefeated.
struct MaxAdversary {
    int n;
    vector<bool> defeated;
    int comparisons = 0;
    int undefeated_count;

    MaxAdversary(int n) : n(n), defeated(n, false), undefeated_count(n) {}

    // Compare elements i and j. Returns the "winner" (larger element).
    // Adversary ensures the undefeated element wins when possible.
    int compare(int i, int j) {
        comparisons++;
        bool di = defeated[i], dj = defeated[j];
        int winner, loser;

        if (!di && !dj) {
            // Both undefeated: adversary picks arbitrarily (say larger index wins)
            winner = max(i, j);
            loser = min(i, j);
        } else if (!di) {
            winner = i; loser = j;
        } else if (!dj) {
            winner = j; loser = i;
        } else {
            // Both defeated: consistent ordering (larger index wins for simplicity)
            winner = max(i, j);
            loser = min(i, j);
        }

        if (!defeated[loser]) {
            defeated[loser] = true;
            undefeated_count--;
        }
        return winner;
    }
};

int main() {
    int n = 10;
    MaxAdversary adv(n);

    // Any algorithm that finds the max: simple tournament
    int champion = 0;
    for (int i = 1; i < n; i++) {
        champion = adv.compare(champion, i);
    }
    cout << "n=" << n
         << "  comparisons=" << adv.comparisons
         << "  undefeated=" << adv.undefeated_count << "\n";
    // Output: n=10  comparisons=9  undefeated=1
    return 0;
}

Finding the Second Largest

This is where adversary arguments really shine. The naive approach uses 2n − 3 comparisons (find the max in n − 1, then find the max of the rest in n − 2). Can we do better?

Theorem (Kislitsyn, 1964)

Finding the second-largest element among n distinct elements requires at least n + ⌈log2 n⌉ − 2 comparisons. This bound is tight — a tournament method achieves it.

Adversary Argument Sketch

The adversary assigns weights to elements. Let w(x) be the weight of element x, initially 1 for all elements. When the algorithm compares x and y:

  • The adversary makes the element with the larger weight win.
  • The winner’s weight becomes w(x) + w(y) (it absorbs the loser’s potential).

After the algorithm terminates:

  • The maximum must have beaten all others directly or transitively — its weight is n.
  • The second-largest must be one of the elements that lost directly to the maximum. Let k be the number of such elements.
  • The maximum made exactly k comparisons (one per direct opponent), so its weight is at most 2k (it at most doubles each time). Therefore 2k ≥ n, giving k ≥ ⌈log2 n⌉.
  • Finding the max costs n − 1 comparisons, then finding the second-largest among k direct losers costs k − 1 more.
  • Total: (n − 1) + (⌈log2 n⌉ − 1) = n + ⌈log2 n⌉ − 2. ∎

Optimal Tournament Algorithm

The algorithm that achieves this bound is a knockout tournament: pair up elements, compare each pair, advance winners, repeat. The maximum is found in n − 1 comparisons. Then the second-largest is the best among the ⌈log2 n⌉ elements that lost directly to the champion. 

#include <bits/stdc++.h>
using namespace std;

// Tournament-based algorithm to find the 2nd largest in
// n + ceil(log2(n)) - 2 comparisons.
int comparisons = 0;

bool cmp(int a, int b) {
    comparisons++;
    return a > b;
}

pair<int,int> find_top2(vector<int>& a) {
    int n = a.size();
    // losers[i] = list of elements that lost directly to a[i]
    map<int, vector<int>> losers;

    // Phase 1: Knockout tournament to find the maximum
    vector<int> alive(a.begin(), a.end());
    for (int &x : alive) losers[x] = {};

    while (alive.size() > 1) {
        vector<int> next_round;
        for (int i = 0; i + 1 < (int)alive.size(); i += 2) {
            int w, l;
            if (cmp(alive[i], alive[i+1])) {
                w = alive[i]; l = alive[i+1];
            } else {
                w = alive[i+1]; l = alive[i];
            }
            losers[w].push_back(l);
            next_round.push_back(w);
        }
        if (alive.size() % 2 == 1) {
            next_round.push_back(alive.back());
        }
        alive = next_round;
    }

    int champion = alive[0];

    // Phase 2: Find the max among elements that lost to the champion
    vector<int>& candidates = losers[champion];
    int runner_up = candidates[0];
    for (int i = 1; i < (int)candidates.size(); i++) {
        if (cmp(candidates[i], runner_up)) {
            runner_up = candidates[i];
        }
    }

    return {champion, runner_up};
}

int main() {
    // Test with n = 16
    vector<int> a;
    for (int i = 1; i <= 16; i++) a.push_back(i);

    // Shuffle so the algorithm doesn't benefit from sorted order
    mt19937 rng(42);
    shuffle(a.begin(), a.end(), rng);

    comparisons = 0;
    auto [mx, second] = find_top2(a);

    int expected = 16 + (int)ceil(log2(16.0)) - 2;  // 16 + 4 - 2 = 18
    cout << "max=" << mx << " second=" << second << "\n";
    cout << "comparisons=" << comparisons
         << " optimal=" << expected << "\n";
    // Output: max=16 second=15
    //         comparisons=18 optimal=18
    return 0;
}
Competitive programming takeaway: When a problem asks you to minimize the number of queries to find something, the adversary argument tells you the theoretical limit. If your algorithm matches it, you know it is optimal and no further optimization is possible.

Sorting Lower Bound

The most famous lower bound in computer science: comparison-based sorting requires Ω(n log n) comparisons. While this is usually proved via the information-theoretic argument (there are n! permutations, each comparison halves the space, so ⌈log2(n!)⌉ comparisons are needed), the adversary argument gives a more constructive proof and can yield tighter constants.

Adversary for Sorting

The adversary maintains a partial order — a directed acyclic graph (DAG) on the elements that is consistent with all comparisons made so far. For each comparison query “is a < b?”, the adversary:

  1. Checks whether a < b or a > b is already forced by the transitive closure of existing comparisons.
  2. If neither is forced, the adversary picks the answer that keeps the number of consistent total orders (linear extensions) as large as possible.

Each comparison can at most halve the number of consistent total orders (since it has only two outcomes). Starting from n! total orders, we need at least ⌈log2(n!)⌉ comparisons before only one remains.

The adversary perspective makes it clear why merge sort is nearly optimal: it asks comparisons that are maximally informative, roughly halving the set of compatible permutations each time.

Problem Lower Bound Best Algorithm Gap
Search (sorted) ⌈log2 n⌉ Binary search Tight
Maximum n − 1 Linear scan Tight
Second largest n + ⌈log2 n⌉ − 2 Tournament Tight
Sorting ⌈log2(n!)⌉ Merge sort Tight (to leading term)
Median 2n − o(n) Median of medians (5n) Open (constant factor)

Interactive Problems & Adaptive Judges

In competitive programming, interactive problems are the living embodiment of the adversary argument. The judge answers your queries in real time, and an adaptive judge may choose answers on the fly to maximize the number of queries you need — exactly like the adversaries we’ve been constructing.

Fixed vs. Adaptive Judges

Fixed Judge

  • Chooses the hidden input before your queries
  • Answers are deterministic given the input
  • Easier: average-case tricks may work
  • Most CF interactive problems

Adaptive Judge

  • Decides answers during interaction
  • Maintains consistency with a set of valid inputs
  • Harder: must handle worst-case
  • Problems that say “the interactor is adaptive”
Warning: When a problem says the judge is adaptive, your algorithm must work against any consistent sequence of answers, not just some fixed hidden input. Strategies that rely on “the answer is already determined” will fail.

Example: Guessing a Number (Adaptive Lies)

Consider this classic interactive problem variant: you must find a hidden number in [1, n], but the judge is allowed to lie at most k times. After each query “is the number ≤ x?”, the judge answers YES or NO, but may lie up to k total times.

The adversary strategy: maintain a set of (number, lies_remaining) states. Each query partitions this set, and the adversary picks the larger partition. The algorithm needs to reduce this set to a single valid state.

The optimal query count is ⌈log2 C(n + k, k)⌉, where the state space size is bounded by binomial coefficients. For k = 0 this reduces to ⌈log2 n⌉ as expected.

Example: Adaptive Coin Weighing

You have n coins, and exactly one is fake (lighter). You can use a balance scale that returns “left heavier”, “right heavier”, or “equal”. An adaptive adversary picks which coin is fake based on your queries (consistent with all previous answers). You need ⌈log3 n⌉ queries. 

#include <bits/stdc++.h>
using namespace std;

// Interactive coin-weighing strategy against an adaptive adversary.
// n coins, one is fake (lighter). Balance scale: 3 outcomes.
// Strategy: divide into 3 groups as evenly as possible.

void solve() {
    int n;
    cin >> n;

    // candidates[i] = true means coin i could still be fake
    vector<bool> candidates(n, true);
    int remaining = n;

    while (remaining > 1) {
        // Collect candidate indices
        vector<int> cands;
        for (int i = 0; i < n; i++)
            if (candidates[i]) cands.push_back(i);

        int sz = cands.size();
        int group_size = sz / 3;
        int extra = sz % 3;

        // Divide into three groups as evenly as possible
        // A: [0, group_size + (extra > 0))
        // B: [a_end, a_end + group_size + (extra > 1))
        // C: the rest (not weighed)
        int a_sz = group_size + (extra > 0 ? 1 : 0);
        int b_sz = group_size + (extra > 1 ? 1 : 0);

        vector<int> A(cands.begin(), cands.begin() + a_sz);
        vector<int> B(cands.begin() + a_sz, cands.begin() + a_sz + b_sz);
        // C is the rest

        // Query: weigh A against B
        cout << "? " << A.size() << " " << B.size();
        for (int x : A) cout << " " << x;
        for (int x : B) cout << " " << x;
        cout << endl;

        int result;
        cin >> result;
        // result: 1 = A lighter, -1 = B lighter, 0 = equal

        if (result == 1) {
            // Fake is in A (lighter side)
            for (int i = 0; i < n; i++)
                if (candidates[i]) {
                    bool in_A = false;
                    for (int x : A) if (x == i) in_A = true;
                    if (!in_A) candidates[i] = false;
                }
        } else if (result == -1) {
            // Fake is in B
            for (int i = 0; i < n; i++)
                if (candidates[i]) {
                    bool in_B = false;
                    for (int x : B) if (x == i) in_B = true;
                    if (!in_B) candidates[i] = false;
                }
        } else {
            // Equal: fake is in C (not weighed)
            for (int x : A) candidates[x] = false;
            for (int x : B) candidates[x] = false;
        }

        remaining = 0;
        for (int i = 0; i < n; i++) remaining += candidates[i];
    }

    // Output the fake coin
    for (int i = 0; i < n; i++) {
        if (candidates[i]) {
            cout << "! " << i << endl;
            return;
        }
    }
}

int main() {
    solve();
    return 0;
}

Connection to Game Theory — Minimax

The adversary argument is, at its core, a two-player zero-sum game:

  • Player 1 (Algorithm): Wants to minimize the number of queries to solve the problem.
  • Player 2 (Adversary): Wants to maximize the number of queries by choosing worst-case answers.

By the minimax theorem (von Neumann, 1928):

maxadversary minalgorithm queries = minalgorithm maxadversary queries

In other words, the optimal algorithm’s worst-case cost equals the adversary’s best guarantee. When we exhibit both an algorithm and an adversary that match, we have proved the exact complexity of the problem.

Yao’s Minimax Principle

Yao’s principle (1977) extends this to randomized algorithms: the expected cost of the best randomized algorithm on the worst-case input equals the expected cost of the best deterministic algorithm on the worst-case distribution of inputs. This lets us prove lower bounds for randomized algorithms using adversary arguments:

  1. Choose a “hard” probability distribution D over inputs.
  2. Show that every deterministic algorithm has high expected cost on D.
  3. By Yao’s principle, every randomized algorithm must have at least that cost in the worst case.
Practical impact: When an interactive problem permits randomization (e.g., “your solution may use random bits”), the adversary cannot be fully adaptive against a randomized strategy. This is why some interactive problems become easier with randomization — the adversary’s power is reduced because it cannot predict your random choices.

Game Tree and Alpha-Beta Analogy

The interaction between algorithm and adversary can be modeled as a game tree. Each node where the algorithm queries is a “min” node (algorithm minimizes queries); each node where the adversary answers is a “max” node (adversary maximizes queries). The value of this game tree is the exact worst-case query complexity.

This is identical in structure to the minimax game trees used in alpha-beta pruning for chess engines. The depth of the game tree equals the query complexity; the branching factor at each level depends on the number of possible queries (algorithm’s turn) or answers (adversary’s turn).

Practical Strategies for CP

How do you apply adversary thinking when solving competitive programming problems? Here is a systematic approach:

Step 1: Identify the Query Model

What questions can your algorithm ask? Comparisons? Membership tests? Arbitrary function evaluations? The query model determines what the adversary controls.

Step 2: Think Like the Adversary

Before designing your algorithm, ask: “If I were the adversary, how would I answer to make life difficult?” This reveals the worst case and helps you design an algorithm that handles it.

Step 3: Design for Information Gain

Each query should maximally reduce the adversary’s freedom. Formally, choose queries that most evenly partition the set of remaining consistent inputs, no matter how the adversary answers.

Step 4: Verify Optimality

Use the adversary argument to prove your algorithm is optimal. If the adversary can force k queries and your algorithm uses k queries, you are done.

Complete Example: Interactive Sorting with Limited Queries

Suppose you are given n = 5 elements and must sort them using at most 7 comparisons. The information-theoretic bound is ⌈log2(5!)⌉ = ⌈log2(120)⌉ = 7, so this is tight. Here is a strategy based on merge-insertion sort (Ford–Johnson algorithm): 

#include <bits/stdc++.h>
using namespace std;

// Sort 5 elements in exactly 7 comparisons.
// This matches the adversary lower bound of ceil(log2(120)) = 7.
//
// Strategy (merge-insertion / Ford-Johnson for n=5):
// 1. Compare a,b and c,d (2 comparisons). WLOG a<b and c<d.
// 2. Compare b,d (1 comparison). WLOG b<d, so chain: a<b<d, c<d.
// 3. Insert e into the chain {a, b, d} using 2 comparisons.
// 4. Insert c into the known chain using 2 comparisons.
// Total: 2 + 1 + 2 + 2 = 7 comparisons.

int queries = 0;

// Simulated comparator (replace with cin/cout for interactive judge)
bool less_than(int a, int b) {
    queries++;
    return a < b;
}

// Insert val into sorted array at correct position using binary insertion.
// arr must be sorted. Uses ceil(log2(len+1)) comparisons.
void binary_insert(vector<int>& arr, int val, int max_pos) {
    // Insert val into arr[0..max_pos-1] region (val could go up to position max_pos)
    int lo = 0, hi = min((int)arr.size(), max_pos);
    while (lo < hi) {
        int mid = (lo + hi) / 2;
        if (less_than(val, arr[mid]))
            hi = mid;
        else
            lo = mid + 1;
    }
    arr.insert(arr.begin() + lo, val);
}

vector<int> sort5(int a, int b, int c, int d, int e) {
    // Step 1: Compare pairs
    if (!less_than(a, b)) swap(a, b);  // now a < b  (1 cmp)
    if (!less_than(c, d)) swap(c, d);  // now c < d  (2 cmp)

    // Step 2: Compare the larger of each pair
    if (!less_than(b, d)) {            // (3 cmp)
        swap(a, c);
        swap(b, d);
    }
    // Now: a < b < d, c < d

    // Build chain from known order: [a, b, d]
    vector<int> chain = {a, b, d};

    // Step 3: Insert e into chain (at most 2 comparisons for 3 elements)
    // Binary search in [a, b, d] for e: ceil(log2(4)) = 2
    binary_insert(chain, e, 3);  // 2 comparisons (4-5 total)

    // Step 4: Insert c into chain.
    // We know c < d, so c goes into positions before d.
    // Find d's position, and binary-insert c into [0..pos_of_d)
    int pos_d = find(chain.begin(), chain.end(), d) - chain.begin();
    binary_insert(chain, c, pos_d);  // 2 comparisons (6-7 total)

    return chain;
}

int main() {
    int a = 3, b = 1, c = 4, d = 5, e = 2;
    queries = 0;

    vector<int> sorted = sort5(a, b, c, d, e);

    cout << "Sorted: ";
    for (int x : sorted) cout << x << " ";
    cout << "\nComparisons: " << queries << "\n";
    // Output: Sorted: 1 2 3 4 5
    //         Comparisons: 7
    return 0;
}
Why 7 and not 8? The adversary argument shows any algorithm needs at least 7 comparisons to sort 5 elements. Merge-insertion sort achieves this exactly. Standard merge sort would use 8 comparisons on 5 elements — which is suboptimal! The adversary argument guides us to the better algorithm.

Common Patterns in Interactive CP Problems

When facing an interactive problem with an adaptive judge, keep these patterns in mind:

  • Binary search still works: Even against an adaptive adversary, binary search is optimal for searching. The adversary cannot “trick” you if you always halve the remaining space.
  • Randomization helps: If you can make random queries, the adversary cannot predict your strategy. This reduces the effective lower bound.
  • Invariant tracking: Maintain what the adversary is still free to choose. If you can show this set shrinks by a constant factor each query, you have an O(log n) algorithm.
  • Witness arguments: Sometimes you need not just the answer but a witness (proof). The adversary must be unable to deny the witness — this often requires extra queries.

Anti-Adversary Technique: Deterministic vs. Randomized

Against a truly adaptive adversary, a deterministic algorithm’s behavior is fully predictable. The adversary can simulate the algorithm and choose the worst-case answer at each step. A randomized algorithm breaks this: the adversary must commit to a strategy without knowing the algorithm’s future random choices. This is why many interactive problems that seem hard deterministically become tractable with randomization. 

#include <bits/stdc++.h>
using namespace std;

// Example: Interactive problem — find a "1" in a binary array of length n
// where the adaptive judge has placed exactly one "1".
// Deterministic: adversary forces n-1 queries (always answer "0" until forced).
// Randomized: expected n/2 queries by querying random unvisited positions.

mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

void solve_randomized(int n) {
    vector<int> positions(n);
    iota(positions.begin(), positions.end(), 0);
    shuffle(positions.begin(), positions.end(), rng);

    for (int i = 0; i < n; i++) {
        cout << "? " << positions[i] << endl;
        int response;
        cin >> response;
        if (response == 1) {
            cout << "! " << positions[i] << endl;
            return;
        }
    }
}

void solve_deterministic(int n) {
    // Against an adaptive adversary, this always takes n-1 queries.
    // The adversary answers "0" for every position until only one remains.
    for (int i = 0; i < n; i++) {
        cout << "? " << i << endl;
        int response;
        cin >> response;
        if (response == 1) {
            cout << "! " << i << endl;
            return;
        }
    }
    // If we get here, the last position must be the answer
    cout << "! " << (n - 1) << endl;
}

int main() {
    int n;
    cin >> n;

    // Use randomized strategy against adaptive adversary
    solve_randomized(n);
    return 0;
}

  • Ulam’s Liar Game (CF 672D): The quintessential adversary-with-lies problem. Find a number when the judge can lie k times. Requires Berlekamp’s counting argument.
Summary: The adversary argument is both a proof technique and a problem-solving mindset. It teaches you to ask: “What is the worst case, and can I handle it?” When you design algorithms that work against the strongest possible adversary, you are guaranteed correctness and optimality — exactly what competitive programming demands.
