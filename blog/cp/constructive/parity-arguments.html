    Parity Arguments

    July 2026

    Parity — the distinction between even and odd — is one of the most powerful and elegant tools in a competitive programmer’s arsenal. Many constructive problems that look intractable become trivial once you notice an invariant that tracks whether something is even or odd. Conversely, parity arguments are the most common way to prove that a construction is impossible.

    In this post we’ll cover the core ideas: impossibility proofs via parity invariants, the famous checkerboard coloring argument, permutation parity (even vs. odd permutations), XOR-based reasoning, and how these show up in Codeforces-style constructive problems. Every section includes concrete C++ implementations you can use in contests.

    Parity Invariants & Impossibility Proofs

    The fundamental idea is simple: find a quantity Q that is preserved modulo 2 by every allowed operation. If the initial value of Q (mod 2) differs from the target value, the target is unreachable — no matter how many operations you perform.

    The Classic Coin-Flip Problem

    You have n coins in a row, all showing heads. In one move you must flip exactly k consecutive coins. Can you make all coins show tails?

    Let Q = the number of heads. Initially Q = n. Each move changes Q by some amount, but the key observation is that each flip toggles exactly k coins, so Q changes by an amount whose parity is the same as k’s parity. If k is even, every move changes Q by an even number, so Q mod 2 is invariant. If n is odd and k is even, Q starts odd and can never reach 0 (which is even) — impossible.

    Recipe for a parity impossibility proof: (1) Define a quantity Q. (2) Show every legal operation preserves Q mod 2. (3) Show Qstart and Qtarget differ mod 2. Therefore the target is unreachable.

    Sum-Parity Invariant

    A very common pattern: you have an array and each operation adds or subtracts values in a way that preserves the parity of the total sum. For example, if every operation picks two elements and adds 1 to both, the sum changes by 2, so the sum’s parity is invariant. 

    // Check if it's possible to make all elements equal
// when each operation adds 1 to two distinct elements.
// If target*n and current sum have different parity, impossible.
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> a(n);
    long long sum = 0;
    for (int &x : a) { cin >> x; sum += x; }

    int mx = *max_element(a.begin(), a.end());
    // We need target >= mx, and target*n - sum must be even
    // (each op increases sum by 2)
    long long target = mx;
    long long deficit = (long long)target * n - sum;
    if (deficit % 2 != 0) {
        target++;  // try next value
        deficit = (long long)target * n - sum;
    }
    // Now deficit is even; number of operations = deficit / 2
    // but we also need each a[i] <= target
    cout << (deficit >= 0 ? target : -1) << "\n";
    return 0;
}

    Checkerboard Coloring Arguments

    The mutilated chessboard problem is perhaps the most famous parity argument in all of mathematics: remove two diagonally opposite corners from an 8×8 chessboard. Can you tile the remaining 62 squares with 31 dominoes (each covering exactly 2 squares)?

    The answer is no, and the proof is beautiful. Color the board like a chessboard: alternating black and white. Each domino, no matter how placed, covers exactly one black square and one white square. So any valid tiling must cover an equal number of black and white squares. But two diagonally opposite corners have the same color, so after removal we have 30 squares of one color and 32 of the other. Tiling is impossible.

    Checkerboard coloring assigns color (i + j) % 2 to the cell at row i, column j. Any unit move (up/down/left/right) changes the color. This is an incredibly useful invariant for grid problems.

    Generalised Domino Tiling Check

    Given an n × m grid with some cells blocked, can the remaining cells be tiled by dominoes? A necessary condition is that the number of free black cells equals the number of free white cells (under checkerboard coloring). This isn’t sufficient in general (you also need a connected matching), but it’s the first filter and catches most “impossible” cases. 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    int n, m;
    cin >> n >> m;
    vector<string> grid(n);
    for (auto &row : grid) cin >> row;

    int black = 0, white = 0;
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < m; j++) {
            if (grid[i][j] == '.') {  // free cell
                if ((i + j) % 2 == 0) white++;
                else black++;
            }
        }
    }

    if (black != white) {
        cout << "NO\n";  // parity obstruction
    } else {
        // Further checks needed (e.g., max matching),
        // but parity is necessary.
        cout << "MAYBE\n";
    }
    return 0;
}

    Knight’s Tour & Color Parity

    A knight on a chessboard always moves from a white square to a black square or vice versa (because a knight move changes both coordinates by odd amounts, so (i+j) changes parity). This means a knight’s path alternates colors. If you want a closed knight’s tour on an n × n board, you need an equal number of black and white squares — so n must be even. (The converse isn’t automatic, but parity is the first necessary condition.)

    Grid Reachability

    Many problems ask: starting at (r1, c1), can you reach (r2, c2) in exactly k steps, moving one cell per step (up/down/left/right)? The Manhattan distance d = |r1 − r2| + |c1 − c2| must satisfy d ≤ k and d ≡ k (mod 2). The second condition is a parity argument: each step changes (r + c) by ±1, so after k steps the parity of (r + c) has changed by k mod 2.

    Parity of Permutations

    Every permutation can be decomposed into transpositions (swaps of two elements). The number of transpositions in any decomposition always has the same parity. A permutation is even if it can be written as an even number of transpositions, and odd otherwise. This is called the sign or signature of the permutation.

    The sign can be computed by counting inversions: pairs (i, j) with i < j but π(i) > π(j). The permutation is even iff the number of inversions is even. Equivalently, decompose the permutation into disjoint cycles; the sign is (−1)n − c where c is the number of cycles.

    Permutation parity rule: A single swap changes the sign. So if you start with the identity (even) and need to reach a target permutation using adjacent swaps, the minimum number of swaps (= number of inversions) has a fixed parity. You can never reach the target in a number of swaps with the wrong parity.

    Computing Permutation Parity

    #include <bits/stdc++.h>
using namespace std;

// Returns 0 if even, 1 if odd permutation.
// Uses cycle decomposition: sign = (n - #cycles) mod 2
int permutation_parity(vector<int>& p) {
    int n = p.size();
    vector<bool> visited(n, false);
    int cycles = 0;
    for (int i = 0; i < n; i++) {
        if (!visited[i]) {
            cycles++;
            int j = i;
            while (!visited[j]) {
                visited[j] = true;
                j = p[j];
            }
        }
    }
    return (n - cycles) % 2;
}

int main() {
    int n;
    cin >> n;
    vector<int> p(n);
    for (int &x : p) { cin >> x; x--; }  // 0-indexed

    cout << (permutation_parity(p) == 0 ? "EVEN" : "ODD") << "\n";
    return 0;
}

    The 15-Puzzle Connection

    The classic 15-puzzle (4×4 sliding tiles) is solvable from a given configuration if and only if the permutation parity of the tiles plus the row of the blank (counting from the bottom) is even. This is a direct application of permutation parity: each slide is a transposition of the blank with an adjacent tile, and the blank’s row parity tracks the overall sign. 

    #include <bits/stdc++.h>
using namespace std;

// Check if a 15-puzzle configuration is solvable.
// Input: 16 values (1..15 and 0 for blank), row-major order.
bool is_solvable_15(vector<int>& board) {
    // Count inversions among non-zero tiles
    int inv = 0;
    vector<int> tiles;
    int blank_row = -1;
    for (int i = 0; i < 16; i++) {
        if (board[i] == 0) {
            blank_row = i / 4;
        } else {
            tiles.push_back(board[i]);
        }
    }
    for (int i = 0; i < (int)tiles.size(); i++)
        for (int j = i + 1; j < (int)tiles.size(); j++)
            if (tiles[i] > tiles[j]) inv++;

    // blank_row from bottom (0-indexed): 3 - blank_row
    int blank_from_bottom = 3 - blank_row;
    // Solvable iff (inversions + blank_from_bottom) is even
    return (inv + blank_from_bottom) % 2 == 0;
}

int main() {
    vector<int> board(16);
    for (int &x : board) cin >> x;
    cout << (is_solvable_15(board) ? "SOLVABLE" : "UNSOLVABLE") << "\n";
    return 0;
}

    XOR-Based Parity Arguments

    XOR is the arithmetic of parity. The XOR of a set of numbers captures, bit by bit, the parity of how many numbers have a 1 in each position. This makes XOR the natural tool for parity-based reasoning in many problems.

    Nim & Sprague–Grundy

    The XOR (nim-sum) of pile sizes determines who wins in Nim — the first player wins iff the XOR is nonzero. This is essentially a multi-bit parity argument: each bit position is an independent parity game.

    XOR of a Range

    A useful identity: XOR of 0, 1, 2, …, n follows a period-4 pattern:

    • n mod 4 = 0 ⇒ XOR = n
    • n mod 4 = 1 ⇒ XOR = 1
    • n mod 4 = 2 ⇒ XOR = n + 1
    • n mod 4 = 3 ⇒ XOR = 0

    This lets you compute XOR(l, …, r) = XOR(0...r) ⊕ XOR(0...l−1) in O(1). 

    // XOR of [0, 1, ..., n]
long long xor_upto(long long n) {
    if (n < 0) return 0;
    switch (n % 4) {
        case 0: return n;
        case 1: return 1;
        case 2: return n + 1;
        case 3: return 0;
    }
    return 0;  // unreachable
}

// XOR of [l, l+1, ..., r]
long long xor_range(long long l, long long r) {
    return xor_upto(r) ^ xor_upto(l - 1);
}

    XOR Cancellation for Missing Element

    Given an array of n−1 distinct integers from 1 to n, find the missing one. XOR all elements with XOR(1...n); duplicates cancel and the missing element remains. This is a parity argument on each bit.

    XOR Invariant in Operations

    Some problems define operations that preserve the XOR of the array. For example, “replace two elements a, b with a XOR b and a AND b.” Does this preserve XOR? Let’s check: a ⊕ b ⊕ (a XOR b) ⊕ (a AND b). Since a ⊕ b = (a XOR b), we need to verify if the total XOR is preserved — it depends on the specific operation. Tracking the XOR invariant is the key first step.

    XOR as a parity detector: When a problem involves toggling, flipping, or pairing elements, try computing the XOR of the relevant values. If the operation preserves the total XOR, you have an invariant.

    Even/Odd Invariants in Operations

    Beyond simple sums, many operations preserve some modular quantity. The trick is to find the right one.

    The Chocolate Bar Problem

    You have an n×m chocolate bar. Each move, you break one piece into two along a grid line. How many moves to reduce it to n·m individual squares? Each break increases the piece count by exactly 1. You start with 1 piece and need n·m pieces, so you need exactly n·m − 1 breaks. The parity of the answer is (nm − 1) mod 2.

    Lamp Toggling

    You have n lamps, all off. Each operation toggles a contiguous range [l, r]. Can you turn on exactly the lamps in a target set S? Consider the “difference array” view: let di = lamp[i] XOR lamp[i−1]. Each range toggle flips exactly two positions in the difference array (dl and dr+1). So the parity of the number of 1s in d is invariant. If the target difference array has an odd number of 1s, it’s unreachable.

    Operation: Add 1 to a Subset

    Many Codeforces problems let you “add 1 to all elements of a subset of size k.” Each operation increases the total sum by k. If k is odd, the sum’s parity flips each operation, so after an even number of operations the sum parity is unchanged. If k is even, the sum parity never changes. Either way, you get a parity constraint on whether a target sum is achievable.

    Solving Constructive Problems with Parity

    Let’s work through complete Codeforces-style problems where parity is the central insight.

    Problem 1: Make All Elements Equal

    Given an array of n integers, in one operation you choose two indices i ≠ j and set ai := ai + 1, aj := aj − 1. Can you make all elements equal?

    Each operation preserves the sum. So we need sum mod n = 0 (so the target value is sum/n). But there’s a subtler parity issue: each operation changes two elements by ±1, so each element’s deviation from the target changes parity. Since we change two elements at a time, the total number of odd deviations changes by 0 or ±2 — its parity is invariant. Initially, the number of elements with odd deviation from target must be even. 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t;
    cin >> t;
    while (t--) {
        int n;
        cin >> n;
        vector<int> a(n);
        long long sum = 0;
        for (int &x : a) { cin >> x; sum += x; }

        if (sum % n != 0) {
            cout << "NO\n";
            continue;
        }
        long long target = sum / n;
        int odd_dev = 0;
        for (int x : a)
            if ((x - target) % 2 != 0) odd_dev++;

        // odd_dev must be even (invariant preserved by ops)
        // Actually, since each op changes exactly 2 elements,
        // and the sum is preserved, divisibility is sufficient.
        // The parity of odd_dev is always even when sum % n == 0
        // and ops are +1/-1 pairs. So just check divisibility.
        cout << "YES\n";
    }
    return 0;
}

    Problem 2: Swap to Sort — Parity Constraint

    Given a permutation, you can swap adjacent elements. Sort it using the minimum number of swaps. Then determine: can you sort it using exactly k swaps (where k ≥ min_swaps)?

    The minimum number of swaps is the inversion count. Each additional “wasted” swap (swap and then swap back) adds exactly 2 swaps. So you can achieve any k where k ≥ inv_count and k ≡ inv_count (mod 2). 

    #include <bits/stdc++.h>
using namespace std;

// Merge sort inversion count
long long merge_count(vector<int>& arr, int l, int r) {
    if (r - l <= 1) return 0;
    int mid = (l + r) / 2;
    long long cnt = merge_count(arr, l, mid) + merge_count(arr, mid, r);
    vector<int> tmp;
    int i = l, j = mid;
    while (i < mid && j < r) {
        if (arr[i] <= arr[j]) {
            tmp.push_back(arr[i++]);
        } else {
            tmp.push_back(arr[j++]);
            cnt += mid - i;  // all remaining in left half are inversions
        }
    }
    while (i < mid) tmp.push_back(arr[i++]);
    while (j < r) tmp.push_back(arr[j++]);
    for (int x = l; x < r; x++) arr[x] = tmp[x - l];
    return cnt;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    long long k;
    cin >> n >> k;
    vector<int> a(n);
    for (int &x : a) cin >> x;

    long long inv = merge_count(a, 0, n);

    if (k >= inv && (k - inv) % 2 == 0)
        cout << "YES\n";
    else
        cout << "NO\n";
    return 0;
}

    Problem 3: Construct Array with Given XOR

    Construct an array of n positive integers whose XOR is exactly x. If n = 1, just output x (if x > 0). For n ≥ 3, use n−3 ones (XOR of an even count of 1s is 0, odd count is 1), then carefully choose the last three elements. The parity of n−3 determines whether the remaining XOR from the ones is 0 or 1. 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t;
    cin >> t;
    while (t--) {
        long long n, x;
        cin >> n >> x;

        if (n == 1) {
            cout << x << "\n";
            continue;
        }
        if (n == 2) {
            // a ^ b = x: use a = x ^ big, b = big
            long long big = 1LL << 20;
            if ((x ^ big) == 0) big = 1LL << 21;
            cout << (x ^ big) << " " << big << "\n";
            continue;
        }

        // n >= 3: fill first n-3 with 1s, track running XOR
        vector<long long> ans(n, 1);
        long long cur_xor = (n - 3) % 2;  // XOR of (n-3) ones

         // Need last 3 elements to XOR to (x ^ cur_xor)
         long long need = x ^ cur_xor;
         long long big = 1LL << 20;
         long long c = need ^ big ^ 1;
        // Ensure all three are positive and distinct enough
        if (c <= 0 || c == 1 || c == big) {
            big = 1LL << 21;
            c = need ^ big ^ 1;
        }
        ans[n - 3] = 1;
        ans[n - 2] = big;
        ans[n - 1] = c;

        for (int i = 0; i < n; i++) {
            cout << ans[i] << " \n"[i == n - 1];
        }
    }
    return 0;
}
    Contest tip: When a problem says “determine if a construction exists,” your first instinct should be to check parity. Compute the parity of the starting state, determine what the operation does to it, and see if the target is reachable. This alone solves (or eliminates) a huge fraction of constructive problems.

    Advanced Parity Techniques

    Multi-Dimensional Parity

    Sometimes a single parity invariant isn’t enough. For 2D grids, you might need to track parities of row sums and column sums independently. You have a binary grid and can toggle all cells in any row or any column. Can you reach a target grid? Cell (i,j) ends up flipped iff row_toggle[i] XOR col_toggle[j] = 1. So the diff grid must be rank-1 over GF(2): 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    int n, m;
    cin >> n >> m;
    vector<vector<int>> init(n, vector<int>(m)), target(n, vector<int>(m));
    for (auto &row : init) for (int &x : row) cin >> x;
    for (auto &row : target) for (int &x : row) cin >> x;

    // diff[i][j] = init[i][j] ^ target[i][j]
    vector<vector<int>> diff(n, vector<int>(m));
    for (int i = 0; i < n; i++)
        for (int j = 0; j < m; j++)
            diff[i][j] = init[i][j] ^ target[i][j];

    // diff[i][j] must equal r[i] XOR c[j] for some r[], c[].
    // Fix r[0] = 0, derive c[j] = diff[0][j], r[i] = diff[i][0] ^ diff[0][0].
    vector<int> r(n), c(m);
    for (int j = 0; j < m; j++) c[j] = diff[0][j];
    for (int i = 0; i < n; i++) r[i] = diff[i][0] ^ diff[0][0];

    bool ok = true;
    for (int i = 0; i < n; i++)
        for (int j = 0; j < m; j++)
            if (diff[i][j] != (r[i] ^ c[j]))
                ok = false;

    cout << (ok ? "YES" : "NO") << "\n";
    return 0;
}

    Parity & Bipartite Graphs

    A graph is bipartite if and only if it has no odd-length cycles. This is secretly a parity argument: assign each node a color based on its distance (mod 2) from a root. If you find an edge connecting two nodes of the same color, you’ve found an odd cycle. BFS/DFS coloring checks bipartiteness by tracking this distance parity.

    The Key Takeaways

    Whenever you encounter a constructive or impossibility problem:

    1. Check the sum parity — does the operation change the total sum by an even or odd amount?
    2. Check element-wise parity — does the operation preserve the parity of individual elements?
    3. Check the XOR — is the total XOR of the array an invariant?
    4. Use coloring — for grid problems, checkerboard coloring often reveals hidden parity constraints.
    5. Count inversions — for permutation problems, the parity of the inversion count is an invariant under single swaps.

    Parity arguments are the first line of defense in constructive problems. They’re fast to check, easy to implement, and surprisingly often they’re the only obstruction. Master these patterns and you’ll crack a huge class of competition problems that others find baffling.
