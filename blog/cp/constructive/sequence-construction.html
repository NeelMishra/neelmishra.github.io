    Sequence Construction

    July 2026 · 25 min read

    Constructive problems are the bread and butter of Codeforces rounds — and among them, sequence construction stands out as the most common variant. You are given a set of constraints (a target sum, a required number of inversions, a specific MEX value, etc.) and must produce a concrete sequence that satisfies all of them. There is rarely a unique answer; the challenge is finding any valid one efficiently.

    This post covers the core techniques: building permutations with prescribed properties, constructing arrays driven by MEX constraints, hitting target sums under tight bounds, and always preferring the lexicographically smallest solution when asked.

    Permutations with a Given Inversion Count

    An inversion in a permutation p of [1..n] is a pair (i, j) with i < j and p[i] > p[j]. The identity permutation has 0 inversions; the reverse permutation has n(n−1)/2. Given n and a target inversion count k, we want to build a permutation with exactly k inversions.

    Key Insight

    Start from the identity [1, 2, …, n]. Reversing a suffix of length m adds exactly m(m−1)/2 inversions. More usefully: placing element i at position j < i (shifting others right) adds exactly i − j inversions. We can greedily consume the inversion budget from the last position backward.

    Greedy Algorithm

    Process positions n, n−1, …, 1. At position i, we can add at most i−1 inversions by placing the largest remaining value at that position. Take min(k, i−1) inversions from position i, subtract from k, and continue.

    #include <bits/stdc++.h>
using namespace std;

// Build a permutation of [1..n] with exactly k inversions.
// Returns empty vector if k > n*(n-1)/2.
vector<int> perm_with_inversions(int n, long long k) {
    if (k > (long long)n * (n - 1) / 2) return {};

    // Start with identity
    deque<int> dq;
    for (int i = 1; i <= n; i++) dq.push_back(i);

    vector<int> result;
    result.reserve(n);

    // Build from left to right.
    // At each step, we can place the j-th element (0-indexed) from
    // the sorted remaining list, contributing j inversions.
    for (int i = n; i >= 1; i--) {
        long long take = min(k, (long long)(i - 1));
        // take-th largest remaining element from the back
        // Equivalently: element at index (i - 1 - take) in the deque
        int idx = (int)(i - 1 - take);
        result.push_back(dq[idx]);
        dq.erase(dq.begin() + idx);
        k -= take;
    }
    return result;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    long long k;
    cin >> n >> k;

    auto ans = perm_with_inversions(n, k);
    if (ans.empty()) {
        cout << -1 << "\n";
    } else {
        for (int i = 0; i < n; i++)
            cout << ans[i] << " \n"[i == n - 1];
    }
}
    Complexity note. The naïve deque approach above is O(n²) due to the erase. For n ≤ 10&sup5; this is fine. For larger n, use a Fenwick tree to binary-search for the k-th remaining element in O(n log n).

    Example: n = 5, k = 3. Identity is [1,2,3,4,5]. We process positions 5 → 1:

    • i=5: take min(3,4) = 3 inversions → pick element at index 4−3=1 in deque [1,2,3,4,5] → pick 2. Remaining: [1,3,4,5]. k=0.
    • i=4: take 0 → pick front 1. Remaining: [3,4,5].
    • i=3: take 0 → pick 3. Remaining: [4,5].
    • i=2: take 0 → pick 4. Remaining: [5].
    • i=1: take 0 → pick 5.

    Result: [2, 1, 3, 4, 5] — wait, that has only 1 inversion. Let’s re-examine. Actually the result is [2, 1, 3, 4, 5]? No — the deque ordering means we output in the order we pick: [2, 1, 3, 4, 5] has 1 inversion. The issue is we should think of it differently.

    A cleaner formulation: build the result right-to-left. For position n down to 1, decide how many of the remaining elements should be smaller than the one placed here. Let’s use the standard approach:

    vector<int> perm_with_inversions_v2(int n, long long k) {
    if (k > (long long)n * (n - 1) / 2) return {};

    vector<int> ans(n);
    vector<int> avail; // sorted available values
    for (int i = 1; i <= n; i++) avail.push_back(i);

    for (int i = 0; i < n; i++) {
        // At position i, we can contribute at most (n - 1 - i) inversions
        // by placing the largest available value here.
        long long can = n - 1 - i;
        long long take = min(k, can);
        // Place the element that is (take)-th from the largest in avail
        // i.e., index = avail.size() - 1 - take
        int idx = (int)(avail.size() - 1 - take);
        ans[i] = avail[idx];
        avail.erase(avail.begin() + idx);
        k -= take;
    }
    return ans;
}

    Now with n=5, k=3: position 0 can contribute at most 4 inversions; take 3. Pick avail[5−1−3] = avail[1] = 2 — no, pick from index 4−3=1, which is value 2. That creates 1 inversion (2 is ahead of only 1). Actually, placing value v at position i creates inversions equal to the number of remaining values smaller than v, which is idx values below it, i.e., avail.size() - 1 - idx = take. So placing value 4 at position 0 (index = 5-1-3=1... let me re-derive). With avail = [1,2,3,4,5], index = 4 - 3 = 1 means avail[1] = 2. But 2 at position 0 is only larger than {1} among remaining, so it contributes 1 inversion — not 3.

    The correct index calculation: to add take inversions, we need take values smaller than our pick to still be available. So we pick the element at index take from the available list (0-indexed, sorted ascending). All elements before it are smaller and will appear later → each becomes an inversion.

    vector<int> perm_with_inversions_correct(int n, long long k) {
    if (k > (long long)n * (n - 1) / 2) return {};

    vector<int> ans(n);
    vector<int> avail;
    for (int i = 1; i <= n; i++) avail.push_back(i);

    for (int i = 0; i < n; i++) {
        long long can = (long long)(avail.size() - 1);
        long long take = min(k, can);
        // Pick element at index (avail.size() - 1 - take) to get
        // 'take' elements that are larger? No...
        // Pick element at index 'take' (0-based, ascending).
        // Then 'take' elements are smaller and placed later = take inversions.
        // Wait: elements AFTER position i that are SMALLER cause inversions
        // with position i only if p[i] > p[j] for j > i.
        // So if we pick the (take)-th smallest remaining value,
        // 'take' smaller values remain and all go after i. Each gives an inversion.
        int idx = (int)take;
        // Hmm, but we want inversions, not non-inversions. Let me think again.
        // Inversions with position i = number of later positions j with p[j] < p[i].
        // If we pick the (take)-th smallest remaining, there are 'take' smaller
        // values still to be placed. So inversions from position i = take. Correct!
        // But we want to minimize perturbation: pick from the end for lex largest,
        // from position take for the canonical construction.
        ans[i] = avail[avail.size() - 1 - take];
        avail.erase(avail.begin() + (avail.size() - 1 - take));
        k -= take;
    }
    return ans;
}

    Alright, let’s stop going in circles and present the clean, correct version:

    The correct construction. Maintain a sorted list of available values. At each position i (left to right), you can contribute at most avail.size() − 1 inversions. Let t = min(k, avail.size() − 1). Pick the value at index (avail.size() − 1 − t) from the sorted available list. This value has exactly t available values larger than it that have already… no. Let’s just think about it as: pick the (t+1)-th largest remaining value. Then t values larger than it are already placed? No, they’re not placed yet. The inversions come from values smaller than the picked value that are placed after it. Since all remaining values will appear after, the number of inversions from this position equals the number of remaining values smaller than our pick. If we pick the (t+1)-th largest, there are avail.size() − 1 − t values smaller remaining, which is not t. So we should pick the (t+1)-th smallest remaining value (0-indexed: index t). Then t smaller values remain → t inversions. ✓ 
    vector<int> build_perm(int n, long long k) {
    if (k > (long long)n * (n - 1) / 2) return {};
    vector<int> avail, ans;
    for (int i = 1; i <= n; i++) avail.push_back(i);
    for (int i = 0; i < n; i++) {
        int rem = (int)avail.size();
        int take = (int)min(k, (long long)(rem - 1));
        // Pick the (take+1)-th smallest remaining = index 'take' (0-based ascending)
        // Nope: we need 'take' smaller vals after us. Pick index such that
        // (number of elements below it in avail) = take.
        // In a 0-indexed sorted avail, element at index j has j elements below it.
        // So pick index j = take... but then the element is the (take+1)-th smallest,
        // and 'take' smaller elements remain after it. But wait, we remove it from avail.
        // The remaining smaller elements = take. They all appear after position i.
        // So inversions from position i = take. Correct!
        ans.push_back(avail[take]);
        avail.erase(avail.begin() + take);
        k -= take;
    }
    return ans;
}

    Let’s verify: n=5, k=3.

    • Pos 0: avail=[1,2,3,4,5], take=3, pick avail[3]=4. Inversions from here = 3 (values 1,2,3 placed later). k=0.
    • Pos 1: avail=[1,2,3,5], take=0, pick avail[0]=1. k=0.
    • Pos 2: avail=[2,3,5], take=0, pick 2.
    • Pos 3: pick 3. Pos 4: pick 5.

    Result: [4, 1, 2, 3, 5]. Inversions: (4,1), (4,2), (4,3) = 3. ✓

    Constructing Derangements

    A derangement is a permutation with no fixed points: p[i] ≠ i for all i. Given n, construct any derangement of [1..n].

    Simple Swap Construction

    The easiest approach: swap adjacent pairs. For even n, swap (1,2), (3,4), …. For odd n, do the same for the first n−3 elements, then rotate the last 3 into a cycle: (n−2) → (n−1) → (n) → (n−2).

    vector<int> derangement(int n) {
    if (n == 1) return {}; // impossible
    vector<int> p(n + 1);
    for (int i = 1; i <= n; i++) p[i] = i;

    int i = 1;
    if (n % 2 == 1) {
        // Handle last 3 as a 3-cycle: n-2 -> n-1 -> n -> n-2
        p[n - 2] = n - 1;
        p[n - 1] = n;
        p[n]     = n - 2;
        // Pair-swap the rest
        for (; i <= n - 3; i += 2)
            swap(p[i], p[i + 1]);
    } else {
        for (; i <= n; i += 2)
            swap(p[i], p[i + 1]);
    }
    return p; // 1-indexed
}
    Derangement existence. Derangements exist for all n ≥ 2. The number of derangements is D(n) = n! · ∑ (-1)^k / k! for k = 0..n, which is approximately n!/e.

    Derangements with Additional Constraints

    A common extension: construct a derangement that also has no pair (i, j) with p[i] = j and p[j] = i (no 2-cycles). This means we need all cycles of length ≥ 3. For n ≥ 3 (and n ≠ 4 for full generality), a single n-cycle works: p[i] = (i % n) + 1.

    // Derangement with no 2-cycles (single n-cycle)
vector<int> derangement_no_transpositions(int n) {
    if (n <= 2) return {}; // impossible without 2-cycles for n=2
    vector<int> p(n + 1);
    for (int i = 1; i < n; i++) p[i] = i + 1;
    p[n] = 1;
    return p;
}

    Arrays with Given MEX Values

    The MEX (minimum excludant) of a set is the smallest non-negative integer not in the set. MEX-based construction problems appear frequently on Codeforces:

    • Construct a permutation of [0..n−1] where the prefix MEX sequence matches a given array.
    • Build an array of length n with values in [0..n] where MEX of the whole array is exactly m.
    • Partition an array so that each part has a given MEX.

    Problem: Construct Array with MEX = m

    Given n and m (0 ≤ m ≤ n), build an array of n non-negative integers with MEX exactly m.

    Solution: We must include all values 0, 1, …, m−1 and exclude m. The remaining n − m slots can be any value except m (we can reuse values 0..m−1 or use values > m).

    vector<int> build_array_with_mex(int n, int m) {
    // Need m <= n (we need at least m slots for 0..m-1)
    if (m > n) return {}; // impossible

    vector<int> a;
    // First m elements: 0, 1, ..., m-1
    for (int i = 0; i < m; i++) a.push_back(i);
    // Remaining n - m elements: anything != m. Use 0 for simplicity.
    for (int i = m; i < n; i++) a.push_back(0);
    return a;
}

    Problem: Prefix MEX Sequence

    Given an array b[1..n] representing the desired MEX of each prefix a[1..i], construct a valid permutation a of [0..n−1], or report it’s impossible.

    Key observations:

    • The MEX sequence b must be non-decreasing (adding elements can only increase or maintain MEX).
    • If b[i] = b[i−1], the element a[i] must not be b[i] (otherwise MEX would increase). We can use any unused value > current MEX.
    • If b[i] = b[i−1] + 1, then a[i] = b[i−1] (we must place exactly the value that was the MEX).
    • If b[i] > b[i−1] + 1, it’s impossible (MEX can increase by at most 1 per element in a permutation).
    #include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> b(n + 1);
    for (int i = 1; i <= n; i++) cin >> b[i];
    b[0] = 0; // MEX of empty prefix

    vector<int> a(n + 1);
    set<int> unused_large; // values > current MEX, not yet placed
    // Initially all values 0..n-1 are unused
    // We'll track which "large" values are available for non-MEX-increasing steps

    // First pass: determine which positions are forced
    bool ok = true;
    int cur_mex = 0;
    set<int> pool; // unused values > current mex at each step
    for (int v = 0; v < n; v++) pool.insert(v);

    for (int i = 1; i <= n; i++) {
        if (b[i] < b[i - 1] || b[i] > b[i - 1] + 1) {
            ok = false;
            break;
        }
        if (b[i] == b[i - 1] + 1) {
            // Must place value b[i-1] (the old MEX)
            a[i] = b[i - 1];
            pool.erase(b[i - 1]);
        } else {
            // b[i] == b[i-1]: place any unused value != b[i]
            // Pick the largest unused value that isn't b[i]
            auto it = pool.end();
            --it;
            if (*it == b[i]) {
                if (it == pool.begin()) { ok = false; break; }
                --it;
            }
            a[i] = *it;
            pool.erase(it);
        }
    }

    if (!ok) {
        cout << -1 << "\n";
    } else {
        for (int i = 1; i <= n; i++)
            cout << a[i] << " \n"[i == n];
    }
}
    Watch the edge cases. When b[i] = b[i−1] and the only unused value equals b[i], construction is impossible. Also verify that b[n] equals n if the problem requires a permutation of [0..n−1].

    Sequences with a Target Sum

    A very common pattern: construct an array of n positive integers with sum exactly S, subject to constraints (distinct values, bounded range, specific parity, etc.).

    Distinct Positive Integers with Sum S

    Find n distinct positive integers summing to S. The minimum possible sum using {1, 2, …, n} is n(n+1)/2. If S < n(n+1)/2, impossible. Otherwise:

    vector<int> distinct_with_sum(int n, long long S) {
    long long min_sum = (long long)n * (n + 1) / 2;
    if (S < min_sum) return {};

    vector<int> a(n);
    for (int i = 0; i < n; i++) a[i] = i + 1;

    long long deficit = S - min_sum;
    // Add the entire deficit to the last element.
    // This keeps all values distinct (last element becomes n + deficit,
    // which is > n >= all other elements).
    a[n - 1] += (int)deficit;
    return a;
}

    If we also need values bounded by some maximum M, we distribute the deficit greedily from the largest element downward:

    vector<int> distinct_bounded_sum(int n, long long S, int M) {
    long long min_sum = (long long)n * (n + 1) / 2;
    long long max_sum = (long long)n * M - (long long)n * (n - 1) / 2;
    // max_sum = M + (M-1) + ... + (M-n+1)
    if (S < min_sum || S > max_sum) return {};

    vector<int> a(n);
    for (int i = 0; i < n; i++) a[i] = i + 1;

    long long deficit = S - min_sum;
    // Greedily increase from the largest element
    for (int i = n - 1; i >= 0 && deficit > 0; i--) {
        int room = M - (n - 1 - i) - a[i];
        // a[i] currently = i+1, max it can be = M - (n-1-i)
        // (to keep room for larger values above it)
        // Actually, since we go from largest to smallest, max for a[i]
        // is M - (n-1-i) only if we haven't moved others. Let's simplify:
        // We want a[i] <= M - (number of elements after i that need to be > a[i])
        // There are (n-1-i) elements after i. But since we're going backwards
        // and increasing the largest first, let's compute max for a[i]:
        int max_val = M - (n - 1 - i);
        long long add = min(deficit, (long long)(max_val - a[i]));
        a[i] += (int)add;
        deficit -= add;
    }
    if (deficit > 0) return {}; // shouldn't happen if bounds checked above
    return a;
}

    Equal Elements with Target Sum and Parity

    Sometimes we need all elements to have the same parity. For example: construct n positive odd integers summing to S. This is possible iff S and n have the same parity (since sum of n odd numbers has parity n mod 2), and S ≥ n.

    vector<int> odd_integers_with_sum(int n, long long S) {
    // n odd numbers sum to S. Possible iff S >= n and S % 2 == n % 2.
    if (S < n || S % 2 != n % 2) return {};

    vector<int> a(n, 1); // start with all 1s (odd), sum = n
    long long rem = S - n;
    // rem is even (since S - n has same parity as n - n = 0, wait:
    // S % 2 == n % 2 implies (S - n) % 2 == 0). So we can add rem
    // in chunks of 2 to keep values odd.
    // Add all to last element for simplicity:
    a[n - 1] += (int)rem; // still odd since rem is even
    return a;
}

    Lexicographically Smallest Constructions

    Many problems ask: among all valid constructions, find the lexicographically smallest. The general strategy is greedy from left to right: at each position, pick the smallest possible value that still allows the remaining positions to be filled validly.

    Feasibility Check

    The heart of these problems is a fast feasibility check: “Given the choices so far, can the remaining positions be filled?” This usually reduces to checking whether the remaining “budget” (inversions, sum, etc.) falls within [min_possible, max_possible].

    Problem: Lex-Smallest Permutation with k Inversions

    We already saw how to build a permutation with k inversions. To get the lex-smallest one, process left to right and at each position try the smallest available value first:

    #include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    long long k;
    cin >> n >> k;

    if (k > (long long)n * (n - 1) / 2) {
        cout << -1 << "\n";
        return 0;
    }

    // avail sorted ascending; we want lex smallest permutation with k inversions.
    // At position i, if we pick the j-th smallest remaining (0-indexed),
    // it contributes j inversions. We need the remaining positions to be
    // fillable with (k - j) inversions. The max inversions from remaining
    // (n - 1 - i) elements is (n-1-i)(n-2-i)/2.
    // So: j inversions now, need k - j <= (n-1-i)(n-2-i)/2.
    // Pick smallest j (= 0 if possible, else increase).

    vector<int> avail;
    for (int i = 1; i <= n; i++) avail.push_back(i);

    vector<int> ans;
    for (int i = 0; i < n; i++) {
        int rem = n - 1 - i; // remaining positions after this one
        long long max_later = (long long)rem * (rem - 1) / 2;

        // We need: k - j <= max_later, so j >= k - max_later
        int min_j = (int)max(0LL, k - max_later);
        // Also j < avail.size()
        // Pick j = min_j (smallest possible value)
        ans.push_back(avail[min_j]);
        avail.erase(avail.begin() + min_j);
        k -= min_j;
    }

    for (int i = 0; i < n; i++)
        cout << ans[i] << " \n"[i == n - 1];
}

    This is clean and correct. At each step, the minimum index j we must use ensures the remaining budget is achievable. Example: n=4, k=3:

    • Pos 0: rem=3, max_later=3. min_j = max(0, 3−3) = 0. Pick avail[0]=1. k=3.
    • Pos 1: rem=2, max_later=1. min_j = max(0, 3−1) = 2. Pick avail[2]=4. k=1.
    • Pos 2: rem=1, max_later=0. min_j = max(0, 1−0) = 1. Pick avail[1]=3. k=0.
    • Pos 3: min_j=0. Pick avail[0]=2.

    Result: [1, 4, 3, 2]. Inversions: (4,3), (4,2), (3,2) = 3. ✓ And it is indeed lex-smallest.

    General lex-smallest pattern. For any constructive problem asking for the lex-smallest answer: iterate left to right, try the smallest candidate, check feasibility of the suffix, and take it if feasible. The feasibility check is problem-specific but often involves a simple min/max bound.

    Greedy Sequence Building Techniques

    Beyond the specific problems above, several greedy patterns recur across constructive sequence problems.

    Pattern 1: Fill from Extremes

    When constructing a permutation satisfying constraints on adjacent differences, a common trick is to alternate placing the smallest and largest remaining values. This maximizes or controls adjacent differences.

    // Construct permutation where all adjacent differences > n/2
// Strategy: place values in order small, large, small, large...
// [1, n, 2, n-1, 3, n-2, ...]
vector<int> zigzag_permutation(int n) {
    vector<int> a(n);
    int lo = 1, hi = n;
    for (int i = 0; i < n; i++) {
        if (i % 2 == 0) a[i] = lo++;
        else             a[i] = hi--;
    }
    return a;
}
// Adjacent diffs: |n-1|, |n-2|, |n-3|, ... all >= n/2 for the first half.

    Pattern 2: Greedy Remainder Distribution

    When you need to distribute a total sum across elements with per-element bounds, use a “fill to capacity” approach:

    // Distribute sum S across n elements, each in [lo, hi].
// Returns lex-smallest valid array, or empty if impossible.
vector<int> distribute(int n, long long S, int lo, int hi) {
    if (S < (long long)n * lo || S > (long long)n * hi) return {};

    vector<int> a(n, lo);
    long long rem = S - (long long)n * lo;

    // For lex-smallest: put excess at the end
    for (int i = n - 1; i >= 0 && rem > 0; i--) {
        int add = (int)min(rem, (long long)(hi - lo));
        a[i] += add;
        rem -= add;
    }
    return a;
}

    Pattern 3: Constructing by Invariant Maintenance

    Some problems require maintaining an invariant (like a running XOR, GCD, or modular sum). Build the sequence one element at a time, ensuring the invariant holds at each step.

    Example: Construct a permutation of [1..n] where the prefix XOR is never zero (except possibly the full array).

    // Construct permutation where no proper prefix has XOR = 0.
// Observation: 1 XOR 2 XOR ... XOR n = 0 if n % 4 == 3 or n % 4 == 0.
// Strategy: for most n, the identity works except for specific cases.
// A general approach: greedily pick the smallest unused value
// that doesn't make prefix XOR = 0.
vector<int> no_prefix_xor_zero(int n) {
    vector<int> ans;
    set<int> unused;
    for (int i = 1; i <= n; i++) unused.insert(i);

    int cur_xor = 0;
    for (int i = 0; i < n; i++) {
        for (auto it = unused.begin(); it != unused.end(); ++it) {
            int v = *it;
            int new_xor = cur_xor ^ v;
            if (i < n - 1 && new_xor == 0) continue; // skip if prefix XOR becomes 0
            ans.push_back(v);
            cur_xor = new_xor;
            unused.erase(it);
            break;
        }
    }
    return ans;
}

    Complete Competition Problems

    Let’s walk through two full problems that combine the techniques above.

    Problem A: Construct Permutation with Given Fixed Points

    Given n and k, construct a permutation of [1..n] with exactly k fixed points (p[i] = i). The constraint is 0 ≤ k ≤ n and if k ≠ n then n − k ≥ 2 (you need at least 2 non-fixed elements to form a derangement among them).

    #include <bits/stdc++.h>
using namespace std;

int main() {
    int t;
    cin >> t;
    while (t--) {
        int n, k;
        cin >> n >> k;

        if (k != n && n - k < 2) {
            // n - k == 1: exactly one non-fixed point, impossible
            // (a single element must map to itself)
            cout << -1 << "\n";
            continue;
        }

        vector<int> p(n + 1);
        // First k elements are fixed points
        for (int i = 1; i <= k; i++) p[i] = i;

        // Remaining n - k elements form a derangement among themselves.
        // Use pair-swaps (and a 3-cycle if n-k is odd, but n-k >= 2
        // and we checked n-k != 1, so if n-k is odd it's >= 3).
        int m = n - k;
        if (m == 0) {
            // all fixed
        } else if (m % 2 == 0) {
            for (int i = k + 1; i <= n; i += 2) {
                p[i] = i + 1;
                p[i + 1] = i;
            }
        } else {
            // m >= 3 and odd: 3-cycle at the end, pair-swap the rest
            p[n - 2] = n - 1;
            p[n - 1] = n;
            p[n]     = n - 2;
            for (int i = k + 1; i <= n - 3; i += 2) {
                p[i] = i + 1;
                p[i + 1] = i;
            }
        }

        for (int i = 1; i <= n; i++)
            cout << p[i] << " \n"[i == n];
    }
}

    Problem B: Array with Given Subarray Sums

    Construct an array a[1..n] of integers (possibly negative) such that for every subarray a[l..r], the sum lies in [−C, C], and the total sum equals a given S. This is solvable by a clever prefix-sum argument.

    Key idea: Let pre[0] = 0, pre[i] = a[1] + … + a[i]. The constraint “every subarray sum in [−C, C]” means |pre[i] − pre[j]| ≤ C for all i, j. This means all prefix sums lie in a window of width C. Specifically, max(pre) − min(pre) ≤ C.

    We need pre[n] = S and pre[0] = 0, and all prefix sums in some interval [L, L+C] that contains both 0 and S.

    #include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    long long S, C;
    cin >> n >> S >> C;

    if (abs(S) > C) {
        cout << "NO\n";
        return 0;
    }

    // All prefix sums must be in [lo, hi] where hi - lo <= C.
    // pre[0] = 0 and pre[n] = S must both be in [lo, hi].
    // Choose lo = min(0, S), hi = lo + C.
    // Check: hi = min(0,S) + C >= max(0,S)? Yes iff |S| <= C. Already checked.

    // Construct prefix sums: any path from 0 to S staying in [lo, hi].
    // Simplest: linear interpolation.
    long long lo = min(0LL, S);

    cout << "YES\n";
    long long prev = 0;
    for (int i = 1; i <= n; i++) {
        // pre[i] = S * i / n (rounded), clamped to [lo, lo + C]
        long long target = S * i / n;
        target = max(target, lo);
        target = min(target, lo + C);
        cout << target - prev << " \n"[i == n];
        prev = target;
    }
}

    Tips and Common Pitfalls

    • Always verify your construction. In contests, write a brute-force checker that validates your output against all constraints. Many constructive solutions have off-by-one bugs that a checker catches instantly.
    • Think about small cases. Constructive problems often have special cases for n = 1, n = 2, or k = 0. Handle these explicitly before the general algorithm.
    • Impossibility detection. Always start by deriving the exact conditions under which a solution exists. Common impossibility reasons: parity mismatch, sum out of range, or not enough elements to satisfy a minimum requirement.
    • Don’t overthink. Many constructive problems have simple solutions. If you find yourself writing 100+ lines, step back and look for a simpler pattern. Classic hint: “What happens if I start with the identity permutation and modify it?”
    • Lexicographic order ≠ sorted order. When asked for lex-smallest, you cannot just sort your answer. You must build it greedily from the first position, ensuring each choice is the smallest that leaves the rest feasible.
    • Watch for 1-indexed vs 0-indexed. Constructive problems frequently mix indexing. Decide on a convention at the start and stick with it. Most permutation problems on Codeforces use 1-indexing.
    • Large answers. If elements can be up to 1018, use long long everywhere. If the array length can be up to 2 × 105, your O(n²) construction might TLE — look for an O(n log n) approach.
    Stress testing recipe for constructive problems. Write a brute-force solver that tries all permutations/arrays (feasible for small n), then compare outputs with your greedy construction on random inputs. This catches almost every bug.
