    Constructive Algorithms — Parity, Invariants & Greedy Construction

    July 2026 · 30 min read

    What Are Constructive Algorithms?

    A constructive problem asks you to build an object satisfying a given set of constraints, or to prove that no such object exists. Unlike decision problems (“does X exist?”) or optimization problems (“find the best X”), constructive problems demand an explicit witness. On Codeforces, they carry the constructive algorithms tag and appear at every difficulty level from Div 2 A to Div 1 E.

    The typical workflow for a constructive problem is:

    1. Find a necessary condition — often a parity or divisibility check — that determines when the answer is -1 (impossible).
    2. Show the condition is sufficient by designing an explicit construction that always produces a valid answer.
    3. Implement the construction efficiently, usually in O(n) or O(n log n).

    There is no single “algorithm” that solves all constructive tasks. Instead, there is a toolbox of recurring ideas. This post covers the three most important ones: parity arguments, invariants, and greedy construction.

    Contest tip. If the problem says “print any valid answer,” a constructive approach almost certainly exists. Focus on finding a pattern that works for small cases, then prove it generalizes.

    Parity Arguments

    Parity is the most common reason a constructive answer is impossible. The idea: if every operation preserves some quantity modulo 2, and the target has the wrong parity, no sequence of operations can reach it.

    Classic Example — Tiling a Mutilated Chessboard

    Remove two opposite corners of an 8×8 chessboard (both the same colour). Can you tile the remaining 62 squares with 31 dominoes? Each domino covers exactly one black and one white square. After removing two squares of the same colour, the board has 32 squares of one colour and 30 of the other, so a perfect tiling is impossible — the parity of the colour imbalance is an invariant of the domino placement.

    Problem — Make Array Zero (CF-style)

    Statement. You are given an array a of n positive integers. In one move you may pick any two elements and decrease both by 1 (they must remain ≥ 0). Can you make every element zero?

    Analysis. Each move decreases the total sum by exactly 2, so the sum must be even. Moreover, no single element can be decreased more than ⌊S/2⌋ times, where S is the total sum — meaning the maximum element must satisfy max ≤ S − max. Both conditions together are necessary and sufficient.

    #include <bits/stdc++.h>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t;
    cin >> t;
    while(t--){
        int n;
        cin >> n;
        vector<int> a(n);
        long long sum = 0;
        int mx = 0;
        for(int i = 0; i < n; i++){
            cin >> a[i];
            sum += a[i];
            mx = max(mx, a[i]);
        }
        if(sum % 2 == 0 && mx <= sum - mx){
            // Construct a valid pairing sequence
            // Greedy: always pair the two largest elements
            priority_queue<int> pq(a.begin(), a.end());
            cout << "YES\n";
            while(pq.top() > 0){
                int x = pq.top(); pq.pop();
                int y = pq.top(); pq.pop();
                // decrease both by 1
                pq.push(x - 1);
                pq.push(y - 1);
            }
        } else {
            cout << "NO\n";
        }
    }
    return 0;
}
    Key insight. Every move changes the sum by −2 (even change). So if the sum is odd, the answer is immediately NO. This “parity filter” appears in a huge number of constructive problems.

    Parity in Permutation Swaps

    A permutation is even if it can be expressed as a product of an even number of transpositions, and odd otherwise. This is equivalent to saying the number of inversions is even or odd. If a problem asks you to reach a target permutation using only adjacent swaps, note that each adjacent swap changes the inversion count by exactly ±1 (changes parity). So if the current and target permutations have the same parity, you need an even number of swaps; if different, an odd number.

    Invariant-Based Reasoning

    An invariant is a property that every legal operation preserves. If the initial state satisfies the invariant but the target does not, the problem is impossible. Otherwise, we must construct a sequence of operations that reaches the target while maintaining the invariant at each step.

    Monovariant (Potential Function)

    Sometimes a quantity doesn’t stay constant but always moves in one direction (e.g., strictly decreases). This is a monovariant. If you can show your construction decreases a non-negative monovariant by at least 1 each step, the process terminates in at most V steps, where V is the initial value of the monovariant.

    Problem — Sort by Reverse (CF 1638B style)

    Statement. Given a permutation p of length n, in one operation you can choose any subarray and reverse it. Sort the permutation in at most n operations. Print the sequence of operations.

    Construction. We use a selection-sort style approach. In step i (from n down to 1), we place element i into position i:

    1. Find position j where element i currently sits.
    2. If j ≠ i, first reverse p[1..j] (this moves element i to position 1), then reverse p[1..i] (this moves element i to position i).

    Each step places one element correctly, and a correctly placed element at the right end is never disturbed. The monovariant is the number of elements not yet in their correct position — it decreases by at least 1 per step. Total operations: at most 2n.

    #include <bits/stdc++.h>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    vector<int> p(n + 1);
    for(int i = 1; i <= n; i++) cin >> p[i];

    vector<pair<int,int>> ops;
    for(int i = n; i >= 1; i--){
        // find where element i currently is
        int j = -1;
        for(int k = 1; k <= i; k++){
            if(p[k] == i){ j = k; break; }
        }
        if(j == i) continue; // already in place
        if(j != 1){
            // reverse p[1..j] to bring element i to front
            reverse(p.begin() + 1, p.begin() + j + 1);
            ops.push_back({1, j});
        }
        // reverse p[1..i] to place element i at position i
        reverse(p.begin() + 1, p.begin() + i + 1);
        ops.push_back({1, i});
    }

    cout << ops.size() << "\n";
    for(auto [l, r] : ops){
        cout << l << " " << r << "\n";
    }
    return 0;
}
    Invariant. After processing step i, positions i, i+1, …, n contain their correct values. This invariant guarantees correctness.

    Greedy Construction Strategies

    Many constructive problems admit a greedy solution: process elements in some order (left to right, largest first, etc.) and make a locally optimal choice at each step. The trick is choosing the right processing order and proving the greedy choice never paints you into a corner.

    Common Patterns

    • Build from the ends. Place the largest remaining element at whichever end of the array is available. This avoids conflicts in problems where adjacent elements have constraints.
    • Alternating placement. For problems requiring adjacent elements to differ in some property (e.g., parity), alternate between two pools of elements.
    • Lexicographically smallest. If the problem asks for the lex-smallest answer, greedily pick the smallest valid choice for each position from left to right.
    • Bipartite colouring. Split elements into two groups based on some binary property and interleave them.

    Problem — Construct Anti-Sorted Permutation

    Statement. Given n, construct a permutation p of {1, 2, …, n} such that for every i, p[i] ≠ i (a derangement). Print any valid derangement, or -1 if impossible.

    Observation. A derangement of {1} does not exist (we’d need p[1] ≠ 1, but 1 is the only option). For n ≥ 2, the simplest construction is a cyclic shift: set p[i] = (i % n) + 1. This maps 1 → 2, 2 → 3, …, n → 1, which is always a valid derangement.

    #include <bits/stdc++.h>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t;
    cin >> t;
    while(t--){
        int n;
        cin >> n;
        if(n == 1){
            cout << -1 << "\n";
            continue;
        }
        // cyclic shift: 2 3 4 ... n 1
        for(int i = 2; i <= n; i++) cout << i << " ";
        cout << 1 << "\n";
    }
    return 0;
}

    Problem — Alternating Parity Array

    Statement. Given an array of n integers, rearrange them so that no two adjacent elements have the same parity (both even or both odd). Print any valid arrangement, or -1 if impossible.

    Analysis. Let e be the count of even numbers and o the count of odd numbers. For an alternating arrangement to exist, we need |e − o| ≤ 1. If o > e, start with odd; if e > o, start with even; if equal, either works.

    #include <bits/stdc++.h>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    vector<int> a(n);
    vector<int> evens, odds;
    for(int i = 0; i < n; i++){
        cin >> a[i];
        if(a[i] % 2 == 0) evens.push_back(a[i]);
        else odds.push_back(a[i]);
    }

    int e = evens.size(), o = odds.size();
    if(abs(e - o) > 1){
        cout << -1 << "\n";
        return 0;
    }

    // Decide who goes first
    vector<int> *first, *second;
    if(o >= e){
        first = &odds;
        second = &evens;
    } else {
        first = &evens;
        second = &odds;
    }

    vector<int> result;
    int fi = 0, si = 0;
    for(int i = 0; i < n; i++){
        if(i % 2 == 0)
            result.push_back((*first)[fi++]);
        else
            result.push_back((*second)[si++]);
    }

    for(int i = 0; i < n; i++){
        cout << result[i];
        if(i + 1 < n) cout << " ";
    }
    cout << "\n";
    return 0;
}
    Greedy correctness proof. At each position we have at least one valid element available. Proof: the pool with more elements (or equal) is assigned the first slot. After placing one element, the other pool gets the next slot. The counts stay balanced throughout because |e − o| ≤ 1.

    Advanced Techniques

    Constructing from Small Cases

    Many constructive problems have a recursive structure: if you can solve the problem for n = 2 and n = 3, you can extend to arbitrary n by combining small solutions. For example, if a problem asks you to partition {1, 2, …, n} into pairs that satisfy some property, solve it for n = 2 first, then show how to extend by adding a new pair of elements.

    Divide and Conquer Construction

    Split the problem in half, solve each half recursively, then merge. This is particularly useful for constructing permutations with specific inversion counts or specific descent sets. The merge step must carefully interleave the two halves to maintain the desired property.

    Using Graph Theory

    Some constructive problems reduce to finding an Eulerian path, a Hamiltonian path, or a specific matching in a graph. If the problem says “visit every edge exactly once,” check the Eulerian path conditions (all vertices have even degree for a circuit, or exactly two odd-degree vertices for a path). If the problem involves pairing elements, model it as a bipartite matching.

    Problem — Construct Permutation with Given Inversion Count

    Statement. Given n and k, construct a permutation of {1, 2, …, n} with exactly k inversions. It is guaranteed that 0 ≤ k ≤ n(n−1)/2.

    Construction. Process positions from left to right. At position i, we can create at most n − i new inversions by placing the largest remaining element, or 0 new inversions by placing the smallest remaining element. Greedily place the largest when we still need inversions:

    #include <bits/stdc++.h>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    long long k;
    cin >> n >> k;

    // Use a deque with values 1..n sorted
    deque<int> dq;
    for(int i = 1; i <= n; i++) dq.push_back(i);

    vector<int> result;
    for(int i = 0; i < n; i++){
        long long can = (long long)(n - 1 - i);
        // placing dq.back() at position i creates 'can' inversions
        // (all remaining smaller elements come after)
        if(k >= can){
            result.push_back(dq.back());
            dq.pop_back();
            k -= can;
        } else {
            // place smallest: creates 0 new inversions
            result.push_back(dq.front());
            dq.pop_front();
        }
    }

    for(int i = 0; i < n; i++){
        cout << result[i];
        if(i + 1 < n) cout << " ";
    }
    cout << "\n";
    return 0;
}

    This runs in O(n) and always produces a valid permutation. The correctness follows from the fact that at each step, we either consume all available inversions from this position (place largest) or consume none (place smallest), and the remaining count k is always within the achievable range for the remaining positions.

    Problem-Solving Checklist

    When you encounter a constructive problem in a contest, work through this checklist:

    1. Small cases. Write out the answer for n = 1, 2, 3, 4. Look for patterns. Often the construction for general n is visible in the first few cases.
    2. Parity / modular check. Does the sum, product, or some other aggregate have a parity constraint? If the operation changes the sum by a fixed amount, check divisibility.
    3. Invariant search. What quantity is preserved by each operation? XOR of all elements? Sum modulo k? The set of elements modulo some small number?
    4. Greedy order. Is there a natural order (sort by value, process from ends, handle largest first) that simplifies the construction?
    5. Reduction to known problem. Can you reduce to graph colouring, Eulerian path, matching, or another well-known constructive result?
    6. Proof of sufficiency. Once you have a construction, prove it always works. A formal proof isn’t necessary during the contest — convincing yourself via 5–10 random test cases is often enough — but it should be airtight before you submit.
    Common pitfall. Don’t confuse “I can’t find a construction” with “no construction exists.” If you suspect the answer is -1, you must prove impossibility (typically via an invariant or parity argument). If you can’t prove it, keep looking for a valid construction.

    Summary Table

    Technique When to Use Classic Signal
    Parity argument Operation changes sum/count by fixed even number “each move changes exactly two elements”
    Invariant Need to show impossibility or bound steps “in one operation you may …” (look at what’s preserved)
    Greedy (left to right) Local choice doesn’t affect future positions “construct any valid X” or “lex-smallest”
    Build from ends Adjacent constraints, extremal placement “no two adjacent elements …”
    Graph reduction Pairing, path-finding, degree constraints “visit every edge/node exactly once”

    Practice Problems

    Sharpen your constructive skills with these curated problems, ordered by difficulty:

    1. CF 1560C — Infinity Table. Construct the coordinates of the k-th element in a spiral. Pure construction, no impossibility case.
    2. CF 1constructive — 1A Make it Equal. Reduce all towers to the same height using a specific cut operation. Greedy from top to bottom.
    3. CF 1638B — Odd Swap Sort. Determine if an array can be sorted using swaps of adjacent elements that sum to an odd number. Parity invariant on the relative order of even/odd elements.
    4. CF 1designers — 1A Two Operations. Reach a target using +1 and ×2 operations. Work backwards from the target (divide by 2 when even, subtract 1 when odd).
    5. CF 1980D — GCD Sequence. Remove one element to make the GCD sequence non-decreasing. Greedy check with prefix/suffix GCD arrays.

    For each problem, start by identifying the impossibility condition (if any), then design a construction, and finally verify on the sample test cases before coding.
