Interactive Binary Search — Guessing Game, Comparator Oracle & Searching with Lies

July 2026 · 20 min read

Introduction

Interactive problems form one of the most satisfying genres in competitive programming. Instead of reading all input upfront, your program converses with a judge — asking queries and receiving answers — to deduce a hidden value. The most fundamental technique behind the majority of these problems is binary search: halving the search space with each query to pin down the answer in logarithmic time.

In this post we build up from the classic number-guessing game, formalize the comparator oracle abstraction, handle the tricky case where the oracle can lie (Ulam’s problem), extend to ternary search and multi-dimensional settings, and finish with full C++ solutions to representative Codeforces and ICPC problems.

Prerequisite. You should be comfortable with standard (non-interactive) binary search and understand flush semantics in C++ I/O. We will cover the interactive protocol briefly, but the focus is on search strategies, not I/O plumbing.

Classic Guessing Game

Problem statement

The judge picks an integer x in [1, n]. You may ask queries of the form ? m. The judge replies:

  • < — the hidden number is less than m
  • > — the hidden number is greater than m
  • = — you found it

You must find x using at most ⌈log2 n⌉ queries.

Why ⌈log2 n⌉ suffices

Maintain an interval [lo, hi] initially [1, n]. Each query with m = (lo + hi) / 2 either finds x or halves the interval. After k queries the remaining interval has length at most n / 2k. Setting this ≤ 1 gives k = ⌈log2 n⌉.

Information-theoretic lower bound

Each three-outcome response carries at most log2 3 ≈ 1.585 bits of information, but for the standard comparator game each response eliminates exactly one half of the remaining candidates, yielding exactly 1 bit. Since we start with log2 n bits of uncertainty, ⌈log2 n⌉ queries are both sufficient and necessary in the worst case.

Complete C++ solution — CF 1167A “Guess the Number”

#include <bits/stdc++.h>
using namespace std;

int main() {
    int lo = 1, hi = 1000000;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        cout << "? " << mid << endl;   // endl flushes

        string resp;
        cin >> resp;

        if (resp == "<") {
            hi = mid - 1;
        } else if (resp == ">") {
            lo = mid + 1;
        } else {
            // resp == "="
            cout << "! " << mid << endl;
            return 0;
        }
    }
    // lo == hi, only one candidate left
    cout << "! " << lo << endl;
    return 0;
}
Key detail: Use endl (or cout.flush()) after every query. If you forget to flush, the judge never sees your query and the program hangs until time limit. This is the single most common cause of Wrong Answer on interactive problems.

Comparator Oracle Model

We can abstract the guessing game into a formal model. A comparator oracle O is a black box that, given a query element q, returns the relation between q and the hidden target x:

O(q) ∈ { <, =, > }

The query complexity of a search algorithm is the number of oracle calls it makes in the worst case. Standard binary search achieves query complexity ⌈log2 n⌉ for a search space of size n, and this is optimal for a deterministic algorithm against an adversarial oracle.

Sorted vs. unsorted domains

Sorted array

Given a sorted array a[0..n-1] and access to an oracle that compares your query index against the hidden target index, binary search finds the target in ⌈log2 n⌉ queries. This is the standard setting.

Unsorted domain

If the domain has no order structure, the oracle response carries no information about other elements. You need Θ(n) queries in the worst case — essentially linear scan. Binary search only helps when the oracle response partitions the remaining candidates monotonically.

Monotone predicate formulation

Many interactive problems don’t literally ask “guess a number.” Instead, you can define a monotone predicate P(m) over the query space such that P flips from false to true at the boundary. Then the oracle’s response tells you which side of the boundary m falls on. 

// Generic interactive binary search on a monotone predicate.
// query(m) asks the judge about value m and returns true/false.
// Returns the smallest m in [lo, hi] for which query(m) == true.
int interactive_bisect(int lo, int hi, function<bool(int)> query) {
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (query(mid))
            hi = mid;
        else
            lo = mid + 1;
    }
    return lo;
}

This template is surprisingly versatile. Problems like “find the broken version” (a hidden threshold in a sequence of builds), “find the city” (a hidden node in a tree with distance queries), and “find the temperature” (a hidden real value with approximate comparisons) all reduce to instantiating query appropriately.

Searching with Lies — Ulam’s Problem

In 1976, Stanisław Ulam posed the following question: “Someone thinks of a number between 1 and 1,000,000. What is the minimum number of yes/no questions needed to determine the number if the answerer is allowed to lie at most once?”

This is dramatically harder than the honest game. A single lie can steer you into the wrong half of the search space, and you won’t discover the mistake until much later (or never). The key insight is that you need redundancy in your queries — enough extra information to detect and correct the lie.

Berlekamp’s framework

Elwyn Berlekamp (1968) gave an elegant coding-theoretic treatment. Assign each candidate number a “state” that tracks how many lies could have been told so far on queries that would affect it. A candidate in state s has been “touched” by s queries whose answer, if a lie, would make this candidate consistent.

For at most e lies among q queries over a domain of size n, a necessary condition is:

∑i=0e C(q, i) ≥ n

where C(q, i) = qCi is the binomial coefficient. This comes from the Hamming bound: each query response is one bit, and with e lies the answers form a codeword at Hamming distance ≤ e from the true answer string.

Strategy for one lie (e = 1)

With at most one lie, each candidate can be in one of three states:

  • State 0: no “suspicious” query yet — the candidate is fully trusted.
  • State 1: one query has been asked whose lie would make this candidate survive — the candidate has used its lie budget.
  • Eliminated: two or more suspicious queries — impossible even with one lie.

The strategy is: partition the remaining candidates into two groups as evenly as possible by total weight, where a state-0 candidate has weight 2 (it can still “absorb” a lie) and a state-1 candidate has weight 1. Ask a query that splits the total weight roughly in half. Update states based on the response.

C++ solution — one-lie binary search

#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;

    // state[i] = number of suspicious queries for candidate i
    // 0 = clean, 1 = one suspicious query, 2+ = eliminated
    vector<int> state(n + 1, 0);  // 1-indexed

    int queries_left = 25;  // generous budget for n up to 10^6 with 1 lie

    while (queries_left-- > 0) {
        // Collect alive candidates with their weights
        // weight = 2 - state[i] for state 0 or 1
        long long total_weight = 0;
        vector<int> alive;
        for (int i = 1; i <= n; i++) {
            if (state[i] <= 1) {
                alive.push_back(i);
                total_weight += (2 - state[i]);
            }
        }

        if (alive.size() == 1) {
            cout << "! " << alive[0] << endl;
            return 0;
        }

        // Find the split point: first index where prefix weight >= total/2
        long long half = (total_weight + 1) / 2;
        long long prefix = 0;
        int split = alive.back();
        for (int i : alive) {
            prefix += (2 - state[i]);
            if (prefix >= half) {
                split = i;
                break;
            }
        }

        // Ask: "Is x <= split?"
        cout << "? " << split << endl;
        string resp;
        cin >> resp;

        if (resp == "<=" || resp == "YES") {
            // Claimed x <= split.
            // Candidates > split: if this is a lie, they survive.
            // So increment their state.
            for (int i : alive) {
                if (i > split) {
                    state[i]++;
                }
            }
        } else {
            // Claimed x > split.
            // Candidates <= split: if this is a lie, they survive.
            for (int i : alive) {
                if (i <= split) {
                    state[i]++;
                }
            }
        }
    }

    // Should not reach here with correct budget
    // Output the only remaining candidate
    for (int i = 1; i <= n; i++) {
        if (state[i] <= 1) {
            cout << "! " << i << endl;
            return 0;
        }
    }
    return 0;
}
Complexity note. The naïve implementation above is O(n) per query for bookkeeping. For large n (up to 109), you must switch to an interval-based representation: maintain a set of intervals tagged with states rather than per-element arrays. Each query splits at most one interval, keeping the total number of intervals O(q).

General e-lie search

For e ≥ 2 lies, the required number of queries q satisfies:

q ≥ ⌈log2 n⌉ + 2e·⌈log2 log2 n⌉ + O(e)

The extra queries pay for error-correction overhead. Each lie costs roughly 2·⌈log2 log2 n⌉ additional queries — far less than the naïve approach of repeating every query e+1 times (which would cost (e+1)·⌈log2 n⌉).

The practical takeaway: for Codeforces interactive problems with lies, count the lie budget e and compute the minimum q from the Hamming bound. If the problem grants you that many queries (or more), the weighted-partition strategy works.

Ternary Search with Queries

Some interactive problems feature a unimodal hidden function rather than a hidden value. Instead of “is x < m?” you ask “what is f(m)?” and receive the function value. The goal is to find the peak (or valley) of f with few queries.

How ternary search works

Maintain an interval [lo, hi]. Pick two probe points m1 = lo + (hi − lo)/3 and m2 = hi − (hi − lo)/3. Query both. If f(m1) < f(m2), the peak is in [m1, hi]; otherwise it is in [lo, m2]. Each iteration reduces the interval by a factor of 2/3, giving query complexity 2·⌈log3/2 n⌉ ≈ 2·log2 n / log2(3/2) ≈ 3.42·log2 n.

Ternary vs. golden-section. Golden-section search reuses one probe point from the previous iteration, needing only 1 query per iteration instead of 2. It uses ratio φ = (1+√5)/2 ≈ 1.618 and achieves ⌈logφ n⌉ ≈ 1.44·log2 n queries. In interactive problems where each query is expensive (counted), golden-section is strictly better.

Interactive ternary search — finding a peak

#include <bits/stdc++.h>
using namespace std;

// Ask the judge for f(pos), return the value.
long long ask(int pos) {
    cout << "? " << pos << endl;
    long long val;
    cin >> val;
    return val;
}

int main() {
    int n;
    cin >> n;

    int lo = 1, hi = n;
    while (hi - lo > 2) {
        int m1 = lo + (hi - lo) / 3;
        int m2 = hi - (hi - lo) / 3;
        long long f1 = ask(m1);
        long long f2 = ask(m2);
        if (f1 < f2)
            lo = m1 + 1;
        else
            hi = m2 - 1;
    }

    // Brute-force the remaining 1-3 elements
    long long best_val = LLONG_MIN;
    int best_pos = lo;
    for (int i = lo; i <= hi; i++) {
        long long v = ask(i);
        if (v > best_val) {
            best_val = v;
            best_pos = i;
        }
    }

    cout << "! " << best_pos << endl;
    return 0;
}

When ternary search applies in CP

Typical scenarios: “find the highest point on a hidden mountain”, “find the optimal parameter value where cost is unimodal”, or “find the breakpoint in a bitonic sequence.” The critical assumption is unimodality — the function must increase then decrease (or vice versa) with no plateaus.

Multi-Dimensional Binary Search

Some interactive problems hide a point in a 2D or higher-dimensional space. The judge answers distance queries, direction queries, or region-membership queries. The goal is to locate the hidden point with few queries.

2D binary search with directional oracle

Suppose the hidden point (x, y) lies in an n × n grid. You query a point (a, b) and the judge responds with the direction towards the target: one of 8 compass directions or “here.” A clean approach:

  1. Binary-search the x-coordinate independently: ask points along the row mid-y and use the east/west component of the response to narrow the column.
  2. Then binary-search the y-coordinate with the x-coordinate fixed.

Total queries: 2·⌈log2 n⌉. More sophisticated strategies can do both dimensions simultaneously in ⌈log2(n2)⌉ = 2·⌈log2 n⌉ queries, which is optimal.

Coordinate binary search in 2D

#include <bits/stdc++.h>
using namespace std;

// Ask direction from (a, b) to target.
// Returns a pair (dx, dy) where dx, dy in {-1, 0, 1}.
pair<int,int> ask(int a, int b) {
    cout << "? " << a << " " << b << endl;
    string resp;
    cin >> resp;
    if (resp == "HERE") return {0, 0};

    int dx = 0, dy = 0;
    if (resp.find("N") != string::npos) dy = -1;
    if (resp.find("S") != string::npos) dy = 1;
    if (resp.find("W") != string::npos) dx = -1;
    if (resp.find("E") != string::npos) dx = 1;
    return {dx, dy};
}

int main() {
    int n;
    cin >> n;

    // Phase 1: find x-coordinate
    int xlo = 1, xhi = n;
    int y_probe = (1 + n) / 2;  // arbitrary row
    while (xlo < xhi) {
        int xmid = xlo + (xhi - xlo) / 2;
        auto [dx, dy] = ask(xmid, y_probe);
        if (dx == 0 && dy == 0) {
            cout << "! " << xmid << " " << y_probe << endl;
            return 0;
        }
        if (dx > 0)       // target is East
            xlo = xmid + 1;
        else if (dx < 0)   // target is West
            xhi = xmid - 1;
        else                // dx == 0 means same column
            { xlo = xhi = xmid; break; }
    }

    int target_x = xlo;

    // Phase 2: find y-coordinate
    int ylo = 1, yhi = n;
    while (ylo < yhi) {
        int ymid = ylo + (yhi - ylo) / 2;
        auto [dx, dy] = ask(target_x, ymid);
        if (dx == 0 && dy == 0) {
            cout << "! " << target_x << " " << ymid << endl;
            return 0;
        }
        if (dy > 0)
            ylo = ymid + 1;
        else
            yhi = ymid - 1;
    }

    cout << "! " << target_x << " " << ylo << endl;
    return 0;
}

Higher dimensions

The same decomposition works in d dimensions: binary-search each coordinate independently for a total of d·⌈log2 n⌉ queries. If the oracle returns a single distance (not direction), you need more queries: typically O(d·log n) with a randomized approach using random hyperplanes.

Interactive Protocol & I/O

Getting the search strategy right is only half the battle. The other half is the I/O protocol. Here are the essential rules for Codeforces and similar judges:

Flushing

After every output line, you must flush. In C++: 

// Option 1: use endl (flushes automatically)
cout << "? " << mid << endl;

// Option 2: explicit flush
cout << "? " << mid << "\n";
cout.flush();

// Option 3: disable buffering entirely at the start
ios_base::sync_with_stdio(false);
cout << unitbuf;  // auto-flush after every << operation

Detecting wrong answers

Many judges return -1 if you ask an invalid query or exceed the query limit. Always check for this: 

int response;
cin >> response;
if (response == -1) {
    // Judge signaled error; exit immediately.
    exit(0);
}

Multiple test cases

Interactive problems with multiple test cases (common on Codeforces) require you to read T upfront, then loop. Make sure you reset all state between cases and flush after the final answer of each case before reading the next one.

Debugging tip. You cannot use cerr to print debug output on Codeforces — the judge ignores stderr. Locally, write a custom interactor script that plays the judge role, or use Codeforces’ built-in “custom invocation” feature.

Template for interactive binary search

#include <bits/stdc++.h>
using namespace std;

int main() {
    int T;
    cin >> T;

    while (T--) {
        int n;
        cin >> n;

        int lo = 1, hi = n;
        bool found = false;

        while (lo <= hi && !found) {
            int mid = lo + (hi - lo) / 2;
            cout << "? " << mid << endl;

            int resp;
            cin >> resp;

            if (resp == -1) return 0;  // judge error
            if (resp == 0) {
                // Found it
                cout << "! " << mid << endl;
                found = true;
            } else if (resp == 1) {
                lo = mid + 1;  // target is higher
            } else {
                hi = mid - 1;  // target is lower
            }
        }

        if (!found) {
            cout << "! " << lo << endl;
        }
    }

    return 0;
}

General tips for solving interactive search problems

  • Count your queries. Before coding, compute the information-theoretic lower bound ⌈log2 n⌉ and compare with the allowed query count. The gap tells you how many lies or wasted queries you can afford.
  • Think in bits. Each query response carries at most log2 k bits of information, where k is the number of possible responses. Design queries that maximize the information gained.
  • Handle edge cases. What happens when lo == hi? When the search space has size 1 or 2? Off-by-one errors are the most common bug.
  • Write a local interactor. For any non-trivial interactive problem, writing a small Python or C++ interactor script that simulates the judge saves enormous debugging time.
  • Stress test with adversarial inputs. An adaptive adversary that always picks the worst-case response (keeping the largest remaining search space) is the true test of your strategy.
Summary. Interactive binary search is the workhorse of interactive competitive programming problems. The honest game requires ⌈log2 n⌉ queries. Lies add an error-correction overhead of roughly 2e·log log n extra queries. Ternary/golden-section search handles unimodal functions. Multi-dimensional problems decompose into independent 1D searches. Master the I/O protocol, and these problems become free points in contests.
