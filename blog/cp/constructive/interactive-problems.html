    Interactive Problems

    July 2026

    Most competitive programming problems follow a simple pattern: read the entire input, compute an answer, print it. Interactive problems break this model entirely. Instead of receiving all the data up front, your program engages in a conversation with a judge — asking questions (queries) and receiving answers — and must deduce the hidden information within a limited query budget. This paradigm appears regularly on Codeforces, Google Code Jam, and ICPC regional contests.

    In this post we will cover the mechanics of interactive I/O, the most important algorithmic pattern (binary search with queries), how to reason about query limits using information theory, and practical techniques for debugging interactive solutions on your own machine.

    The Query-Response Model

    An interactive problem defines a hidden state that the judge knows but you do not. Your program may ask the judge questions (queries) in a specific format. After each query, the judge replies with information that partially reveals the hidden state. Your goal is to determine the hidden state — or produce a correct output derived from it — using at most Q queries.

    Anatomy of an Interactive Problem Statement

    A typical Codeforces interactive problem contains these sections:

    • Interaction protocol – the exact format of your queries and the judge’s responses.
    • Query limit – the maximum number of queries allowed (e.g., “at most 30 queries”).
    • Termination – how to output the final answer (often prefixed with !).
    • Flush requirement – a reminder to flush stdout after every output line.

    Here is the simplest possible example — the classic “guess my number” game:

    Problem. The judge has picked an integer x in [1, n]. You may ask queries of the form ? y. The judge replies 1 if x > y, -1 if x < y, or 0 if x = y. Find x in at most ⌈log2 n⌉ queries, then print ! x.

    The key insight is that each query partitions the remaining candidates into two sets, and the judge’s response tells you which set contains the answer. This is exactly binary search.

    Flushing Output in C++

    The single most common mistake in interactive problems is forgetting to flush. Standard I/O libraries buffer output for performance. In a normal problem this is fine because the judge reads your output only after your program terminates. In an interactive problem, the judge needs to read your query immediately so it can send a response. If your query is sitting in a buffer, both your program and the judge are waiting for each other — a deadlock.

    Method 1: std::endl

    std::endl writes a newline and flushes the stream. This is the simplest approach but slightly slower than a bare '\n' because of the forced flush on every line. For interactive problems the overhead is negligible since you only print a few dozen lines. 

    std::cout << "? " << mid << std::endl;

    Method 2: cout.flush()

    Explicitly flush after printing with '\n':

    std::cout << "? " << mid << '\n';
std::cout.flush();

    Method 3: printf + fflush(stdout)

    If you prefer C-style I/O:

    printf("? %d\n", mid);
fflush(stdout);

    Method 4: Disable Buffering Globally

    You can turn off buffering entirely at the start of main. After this, every output operation flushes automatically: 

    int main() {
    std::ios_base::sync_with_stdio(false);
    std::cin.tie(nullptr);
    std::cout << std::unitbuf;  // auto-flush after every insertion
    // ... rest of solution
}
    Tip. Using std::cout << std::unitbuf; at the top of main is the safest approach — you never have to think about flushing again. Many competitive programmers add this to their interactive template and forget about it.

    What About cin.tie?

    By default, std::cin is tied to std::cout, meaning cout is flushed automatically before every cin read. This sounds like it would save you, but competitive programmers often write cin.tie(nullptr) to speed up I/O in non-interactive problems. If you keep that line, you must flush manually. The safest pattern is to keep cin.tie(nullptr) for speed but add unitbuf to guarantee flushing.

    Binary Search with Queries

    Binary search is the bread and butter of interactive problems. The “guess the number” problem is the purest form, but the technique generalises to any setting where each query can eliminate a constant fraction of candidates.

    Solution: Guess the Number (CF 1–n)

    Here is a complete solution for the problem described above: find a hidden number in [1, n] using at most ⌈log2 n⌉ queries. 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    cout << unitbuf;  // auto-flush

    int n;
    cin >> n;

    int lo = 1, hi = n;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        cout << "? " << mid << '\n';

        int resp;
        cin >> resp;

        if (resp == 0) {
            // lucky: we found it exactly
            cout << "! " << mid << '\n';
            return 0;
        } else if (resp == 1) {
            // x > mid
            lo = mid + 1;
        } else {
            // x < mid  (resp == -1)
            hi = mid - 1;
        }
    }

    cout << "! " << lo << '\n';
    return 0;
}

    This uses exactly ⌈log2 n⌉ queries in the worst case. For n = 109, that is only 30 queries — remarkably efficient.

    Variant: Three-Way Response

    Some problems give a three-way response: “too low”, “too high”, or “correct”. Others give only two-way: ≤ or >. The two-way version is identical to standard binary search. The three-way version lets you short-circuit when you hit the answer exactly, but the worst-case query count is the same.

    Variant: Binary Search on a Real-Valued Answer

    Occasionally an interactive problem asks you to find a hidden real number within some tolerance ε. You simply run binary search for a fixed number of iterations (60–100 is standard) to guarantee precision well beyond 10-9: 

    double lo = 0, hi = 1e18;
for (int iter = 0; iter < 100; iter++) {
    double mid = (lo + hi) / 2.0;
    cout << "? " << fixed << setprecision(9) << mid << '\n';

    string resp;
    cin >> resp;

    if (resp == "HIGH") hi = mid;
    else lo = mid;
}
cout << "! " << fixed << setprecision(9) << (lo + hi) / 2.0 << '\n';

    Information-Theoretic Lower Bounds

    Before designing your query strategy, it pays to ask: how many queries do I actually need? Information theory gives a clean answer.

    The Counting Argument

    Suppose the hidden state is one of N possibilities. Each query receives one of k possible responses. After Q queries, you have seen one of kQ possible response sequences. To distinguish all N candidates, you need:

    kQ ≥ N  ⇒  Q ≥ ⌈logk N⌉

    For binary responses (k = 2):

    • N = 106 ⇒ Q ≥ 20
    • N = 109 ⇒ Q ≥ 30
    • N = 1018 ⇒ Q ≥ 60
    Rule of Thumb. If the query limit is ⌈log2 N⌉, the intended solution is almost certainly binary search. If the limit is 2 ⌈log2 N⌉ or slightly more, there may be noise, errors, or a more complex structure to navigate.

    Ternary Responses

    If each query has three possible responses (k = 3), you can find one of N candidates in ⌈log3 N⌉ queries. This arises in problems where you can compare two elements and get “less”, “equal”, or “greater”. For N = 109, ternary search needs only 19 queries.

    When the Bound Is Not Tight

    The information-theoretic bound is a lower bound. It tells you the minimum number of queries required by any algorithm. Your algorithm might need more if:

    • The query format restricts what you can ask (e.g., you can only ask about contiguous ranges).
    • The judge is adaptive — it changes the hidden state to maximise your query count, as long as its answers remain consistent.
    • Some queries are “wasted” due to the problem structure.

    Adaptive vs Non-Adaptive Strategies

    The terms “adaptive” and “non-adaptive” have two meanings in this context, and it is crucial not to confuse them.

    Adaptive Judge (Adversarial)

    An adaptive judge does not commit to the hidden state in advance. Instead, it answers each query in a way that is consistent with all previous answers but maximises the difficulty for your algorithm. This means the judge effectively picks the worst case on the fly.

    Codeforces problems sometimes warn: “the interactor is adaptive.” In practice, this means your solution must work against every consistent sequence of responses. You cannot rely on the hidden value being fixed.

    Key Insight. Against an adaptive judge, your algorithm must be correct for every possible response sequence — not just the one corresponding to some fixed hidden value. Think of it as a game against an adversary who sees your queries in real time.

    Non-Adaptive Strategy (Your Side)

    A non-adaptive strategy is one where you decide all your queries in advance, without looking at the responses. This is sometimes called a “batch” or “offline” strategy. For example, you could ask all Q queries up front, then read all Q responses and compute the answer.

    Non-adaptive strategies are rarely optimal (they cannot exploit information gained from early queries), but they are sometimes forced by the problem format. For example, in some Google Code Jam problems, you send all queries at once and receive all answers at once.

    Adaptive Strategy (Your Side)

    An adaptive strategy chooses each query based on the responses to all previous queries. Binary search is the canonical example: you pick the midpoint, see which half the answer is in, and recurse on that half. Each query is informed by all prior responses. Adaptive strategies are almost always more efficient than non-adaptive ones.

    The power of adaptivity is quantified by a classic result: for finding one of N items with yes/no queries, an adaptive strategy needs ⌈log2 N⌉ queries, while a non-adaptive strategy needs N - 1 queries (just ask about each item individually).

    Common Patterns in CF Interactive Problems

    After solving many interactive problems on Codeforces, certain patterns emerge repeatedly. Recognising these patterns lets you quickly identify the right approach.

    Pattern 1: Binary Search on a Hidden Value

    The simplest and most common pattern. The hidden state is a single value in a known range, and each query narrows the range. Query limit is O(log N).

    Pattern 2: Finding a Hidden Permutation

    The judge holds a hidden permutation of [1, n]. You can query comparisons like “is p[i] < p[j]?” and must reconstruct the permutation. This often reduces to sorting with a limited number of comparisons. The lower bound is ⌈log2(n!)⌉ ≈ n log2 n.

    Pattern 3: Graph Exploration

    You are placed on a node of a hidden graph and can query edges or neighbours. Common tasks include finding a specific node, a shortest path, or the entire graph structure. These problems often require BFS or DFS adapted to the interactive setting.

    Pattern 4: Coin Weighing / Lie Detection

    A subset of elements is “defective” (e.g., a fake coin). Queries are comparisons or subset tests. Some variants allow the judge to lie a bounded number of times, requiring redundancy in your queries. These are closely related to coding theory and error-correcting codes.

    Pattern 5: Narrowing a Set

    You start with a large candidate set and each query eliminates some candidates. The goal is to reduce the set to a single element. The key is to choose queries that eliminate as many candidates as possible in the worst case — ideally splitting the set roughly in half.

    Complete Example: Finding a Hidden Edge in a Graph

    Here is a classic interactive problem: a tree on n nodes has one hidden edge. You may query a node v and the judge tells you the distance from v to the nearest endpoint of the hidden edge. Find the edge in O(log n) queries.

    The approach: root the tree, then binary search on the path from the root to a leaf. A query at node v tells you whether the hidden edge is in the subtree of v or not. By binary searching on the Euler tour, we can locate the edge. 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    cout << unitbuf;

    int n;
    cin >> n;

    // Read the tree
    vector<vector<int>> adj(n + 1);
    for (int i = 0; i < n - 1; i++) {
        int u, v;
        cin >> u >> v;
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    // BFS to build parent array and order nodes by depth
    vector<int> parent(n + 1, 0), depth(n + 1, 0);
    vector<int> order;
    vector<bool> visited(n + 1, false);
    queue<int> bfs;
    bfs.push(1);
    visited[1] = true;
    while (!bfs.empty()) {
        int u = bfs.front(); bfs.pop();
        order.push_back(u);
        for (int v : adj[u]) {
            if (!visited[v]) {
                visited[v] = true;
                parent[v] = u;
                depth[v] = depth[u] + 1;
                bfs.push(v);
            }
        }
    }

    // Binary search: find the deeper endpoint of the hidden edge
    // Query a node v; judge returns distance to nearest endpoint
    // If dist == 0, the edge is incident to v
    int lo = 0, hi = (int)order.size() - 1;
    int best = 1;  // closest known node to the edge
    while (lo <= hi) {
        int mid = (lo + hi) / 2;
        int v = order[mid];
        cout << "? " << v << '\n';

        int dist;
        cin >> dist;

        if (dist == 0) {
            best = v;
            // v is an endpoint; find the other by querying neighbours
            for (int u : adj[v]) {
                cout << "? " << u << '\n';
                int d2;
                cin >> d2;
                if (d2 == 0) {
                    cout << "! " << v << " " << u << '\n';
                    return 0;
                }
            }
            break;
        } else if (depth[v] + dist <= depth[best] + 0) {
            // edge is deeper; search deeper half
            best = v;
            lo = mid + 1;
        } else {
            hi = mid - 1;
        }
    }

    return 0;
}
    Note. The exact details of this solution depend on the precise problem statement (what the judge returns and how the edge is encoded). The key idea — using BFS order and binary search to locate a hidden feature in a tree — is the transferable technique.

    Handling Multiple Test Cases

    Many Codeforces interactive problems have multiple test cases in a single run. The judge sends the number of test cases T, and you solve each one interactively before moving to the next. A common pitfall is failing to reset state between test cases or reading leftover data from the previous case. 

    #include <bits/stdc++.h>
using namespace std;

void solve() {
    int n;
    cin >> n;

    int lo = 1, hi = n;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        cout << "? " << mid << '\n';

        int resp;
        cin >> resp;

        if (resp == 1)
            lo = mid + 1;
        else
            hi = mid;
    }

    cout << "! " << lo << '\n';

    // Some judges send a confirmation; read it if the problem says so
    // int verdict; cin >> verdict;
    // if (verdict == -1) exit(0);  // wrong answer, terminate
}

int main() {
    cout << unitbuf;

    int T;
    cin >> T;

    while (T--) {
        solve();
    }

    return 0;
}
    Important. Some judges send -1 if your answer is wrong, and expect you to terminate immediately. Always read the problem statement carefully to see if there is a “wrong answer” signal. If you ignore it, you may get a “Runtime Error” or “Idleness Limit Exceeded” verdict instead of “Wrong Answer”.

    Debugging Interactive Solutions Locally

    Interactive problems are notoriously hard to debug because you cannot just run your program on a static input file. Here are several practical techniques.

    Technique 1: Write Your Own Judge

    The most reliable approach is to write a separate “judge” program that simulates the interactor. Then pipe the two programs together: 

    // judge.cpp — simple judge for "guess the number"
#include <bits/stdc++.h>
using namespace std;

int main(int argc, char* argv[]) {
    srand(atoi(argv[1]));  // seed from command line
    int n = 1000000;
    int x = rand() % n + 1;

    cout << n << endl;

    for (int q = 0; q < 30; q++) {
        string type;
        int val;
        cin >> type >> val;

        if (type == "!") {
            if (val == x)
                cerr << "Correct!" << endl;
            else
                cerr << "Wrong! Hidden was " << x << endl;
            return 0;
        }

        // type == "?"
        if (val < x)
            cout << 1 << endl;   // x > val
        else if (val > x)
            cout << -1 << endl;  // x < val
        else
            cout << 0 << endl;   // found it
    }

    cerr << "Query limit exceeded!" << endl;
    return 1;
}

    On Linux/macOS, you can connect the two programs with a named pipe or using a helper script:

    // run_interactive.sh (bash)
// mkfifo pipe
// ./judge 42 < pipe | ./solution > pipe
// rm pipe

    On Windows, you can use a Python script to mediate:

    // run_interactive.py
// import subprocess
// judge = subprocess.Popen(['judge.exe', '42'],
//     stdin=subprocess.PIPE, stdout=subprocess.PIPE)
// sol = subprocess.Popen(['solution.exe'],
//     stdin=judge.stdout, stdout=judge.stdin)
// sol.wait()
// judge.wait()

    Technique 2: Self-Contained Debug Mode

    Embed the judge inside your solution using a preprocessor flag. When you submit to the judge, the debug code is compiled away: 

    #include <bits/stdc++.h>
using namespace std;

#ifdef LOCAL
int hidden_x;  // the secret value

int ask(int y) {
    if (y == hidden_x) return 0;
    return (hidden_x > y) ? 1 : -1;
}

void answer(int y) {
    if (y == hidden_x)
        cerr << "AC: found " << y << endl;
    else
        cerr << "WA: guessed " << y
             << " but hidden was " << hidden_x << endl;
}
#else
int ask(int y) {
    cout << "? " << y << '\n';
    int resp;
    cin >> resp;
    return resp;
}

void answer(int y) {
    cout << "! " << y << '\n';
}
#endif

int main() {
    cout << unitbuf;

#ifdef LOCAL
    srand(42);
    int n = 1000000;
    hidden_x = rand() % n + 1;
    cerr << "Hidden: " << hidden_x << endl;
#else
    int n;
    cin >> n;
#endif

    int lo = 1, hi = n;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        int resp = ask(mid);

        if (resp == 0) { answer(mid); return 0; }
        else if (resp == 1) lo = mid + 1;
        else hi = mid - 1;
    }

    answer(lo);
    return 0;
}

    Compile with g++ -DLOCAL -o sol sol.cpp for local testing and g++ -o sol sol.cpp for submission.

    Stress Testing. Wrap the self-contained solution in a loop over random seeds to stress-test thousands of cases quickly. This is by far the most effective way to find bugs in interactive solutions.

    Technique 3: Logging Queries

    Print all queries and responses to stderr so they appear on your terminal but not in the judge’s input. This is always safe since judges ignore stderr: 

    int ask(int y) {
    cout << "? " << y << '\n';
    cerr << "  [query] ? " << y << endl;
    int resp;
    cin >> resp;
    cerr << "  [response] " << resp << endl;
    return resp;
}

    Common Pitfalls and Verdicts

    Interactive problems produce some verdicts you rarely see in standard problems. Understanding them saves hours of debugging.

    Idleness Limit Exceeded (ILE)

    This means your program and the judge are both waiting for input from each other — a deadlock. The cause is almost always a missing flush. Check that you flush after every output line, including the final answer.

    Wrong Answer on Query

    Some judges validate your queries. If you query a value outside the allowed range, or use an invalid format, the judge may respond with -1 and terminate. Always bounds-check your queries.

    Exceeding the Query Limit

    If you make more than Q queries, the judge terminates with Wrong Answer. Always add a query counter during debugging and assert that it stays within bounds: 

    int query_count = 0;
const int MAX_QUERIES = 30;

int ask(int y) {
    assert(++query_count <= MAX_QUERIES);
    cout << "? " << y << '\n';
    int resp;
    cin >> resp;
    if (resp == -1) exit(0);  // judge signalled error
    return resp;
}

    Off-by-One in Binary Search

    The most subtle bug. If your binary search bounds are wrong, you might converge to a value that is off by one, wasting the final query. Always trace through small examples (e.g., n = 2 and n = 3) by hand to verify your loop invariant.

    VerdictLikely CauseFix
    Idleness Limit Exceeded Missing flush Add unitbuf or endl
    Wrong Answer Exceeded query limit Check algorithm complexity
    Runtime Error Ignoring -1 response Check for error signal from judge
    Time Limit Exceeded Deadlock (variant of ILE) Ensure flush after every output

    Summary and Checklist

    Interactive problems test a different skill set than standard problems: you must think about information flow, query efficiency, and I/O mechanics. Here is a checklist for every interactive problem you solve:

    1. Flush output — add cout << unitbuf; at the top of main.
    2. Count queries — add an assertion to catch query limit violations during testing.
    3. Handle the error signal — if the judge sends -1, terminate immediately.
    4. Information-theoretic sanity check — verify that ⌈logk N⌉ ≤ Q for your approach.
    5. Small cases — trace through n = 1, 2, 3 by hand to catch off-by-one errors.
    6. Stress test — use the self-contained debug mode to run thousands of random cases.
    7. Adaptive adversary — if the interactor is adaptive, ensure your algorithm works for all consistent response sequences.
    8. Reset state — in multi-test-case problems, clear all data structures between cases.

    With these principles internalised, interactive problems become less mysterious and more like a structured conversation — one where the right questions, asked in the right order, always lead to the answer.
