Interactive Graph Problems — Exploring Hidden Structures with Queries

July 2026

Introduction

Interactive graph problems are among the trickiest in competitive programming. The judge holds a secret graph — a tree, a general graph, or even a single hidden vertex — and you must discover its structure by asking queries. Your total number of queries is bounded, so brute force is not an option.

These problems test graph theory knowledge, information-theoretic reasoning (how many bits does each query yield?), and algorithmic design under communication constraints. In this post we cover the core techniques that recur on Codeforces, AtCoder, and ICPC.

Interactive Protocol Reminder. Always flush output after every query (cout.flush() or endl). Forgetting this causes Idleness Limit Exceeded, not WA or TLE.

The typical interactive skeleton:

#include <bits/stdc++.h>
using namespace std;
int ask(int u, int v) {
    cout << "? " << u << " " << v << endl;
    int r; cin >> r; return r;
}
void answer(int x) { cout << "! " << x << endl; }
int main() {
    int n; cin >> n;
    // ... interact, then output answer
}

Adjacency Query Model

The simplest model: query “? u v” returns 1 if edge (u, v) exists, 0 otherwise. With n vertices there are n(n−1)/2 possible edges, so naïvely you need O(n2) queries to reconstruct the graph. The key is to exploit structural constraints.

Exploiting Degree Bounds

If the hidden graph has max degree Δ, stop querying a vertex once you’ve found Δ neighbours. This brings total queries to O(nΔ). 

// Find all edges of a hidden graph with max degree D
#include <bits/stdc++.h>
using namespace std;
int n, D;
vector<int> adj[100005];
int ask(int u, int v) {
    cout << "? " << u << " " << v << endl;
    int r; cin >> r; return r;
}
void solve() {
    cin >> n >> D;
    for (int u = 1; u <= n; u++) {
        if ((int)adj[u].size() >= D) continue;
        for (int v = u + 1; v <= n; v++) {
            if ((int)adj[v].size() >= D) continue;
            if (ask(u, v)) {
                adj[u].push_back(v);
                adj[v].push_back(u);
            }
            if ((int)adj[u].size() >= D) break;
        }
    }
    int m = 0;
    for (int u = 1; u <= n; u++) m += adj[u].size();
    cout << "! " << m / 2 << endl;
}
int main() { solve(); }

Information-Theoretic Lower Bounds

Each yes/no query gives 1 bit. If the hidden graph family has size F, you need ⌈log2F⌉ queries. For general graphs, F = 2n(n−1)/2 ⇒ Θ(n2) lower bound. For trees, F = nn−2 (Cayley) ⇒ roughly (n−2) log2n queries.

Key Insight. When the hidden structure is constrained (tree, matching, path), you can beat O(n2) dramatically. Always ask: “How many bits per query vs. how many bits total?”

Finding a Hidden Vertex

Classic setup: you know the tree structure; the judge picks a hidden vertex x. Query “? v” returns dist(v, x). Find x in as few queries as possible.

Binary Search on a Path

If the tree is a path, query the midpoint; the distance tells you which half contains x. This uses O(log n) queries.

General Trees: Centroid Decomposition

Find the centroid c of the current component, query dist(c, x) = d, then keep only vertices at distance exactly d from c. Mark c as removed and recurse. Each step halves the candidate set ⇒ O(log n) queries total. 

// Find hidden vertex on a known tree — centroid decomposition
#include <bits/stdc++.h>
using namespace std;
const int MAXN = 200005;
vector<int> adj[MAXN];
int sz[MAXN]; bool removed[MAXN]; int n;

int ask(int v) { cout << "? " << v << endl; int d; cin >> d; return d; }

void calc_sz(int u, int p) {
    sz[u] = 1;
    for (int v : adj[u]) if (v != p && !removed[v]) { calc_sz(v, u); sz[u] += sz[v]; }
}
int centroid(int u, int p, int ts) {
    for (int v : adj[u]) if (v != p && !removed[v] && sz[v] > ts/2) return centroid(v, u, ts);
    return u;
}
// BFS in alive component
vector<int> bfs(int src, vector<int>& alive) {
    map<int,int> dist;
    for (int v : alive) dist[v] = -1;
    dist[src] = 0;
    queue<int> q; q.push(src);
    while (!q.empty()) {
        int u = q.front(); q.pop();
        for (int w : adj[u]) if (dist.count(w) && dist[w] == -1) { dist[w] = dist[u]+1; q.push(w); }
    }
    vector<int> res(n+1, -1);
    for (auto& [v,d] : dist) res[v] = d;
    return res;
}
void solve() {
    cin >> n;
    for (int i = 0; i < n-1; i++) { int u,v; cin >> u >> v; adj[u].push_back(v); adj[v].push_back(u); }
    vector<int> cands(n); iota(cands.begin(), cands.end(), 1);
    while (true) {
        if (cands.size() == 1) { cout << "! " << cands[0] << endl; return; }
        calc_sz(cands[0], -1);
        int c = centroid(cands[0], -1, sz[cands[0]]);
        int d = ask(c);
        if (d == 0) { cout << "! " << c << endl; return; }
        auto dist_c = bfs(c, cands);
        vector<int> nxt;
        for (int v : cands) if (dist_c[v] == d) nxt.push_back(v);
        removed[c] = true;
        cands = nxt;
    }
}
int main() { solve(); }
Query Complexity. At most O(log n) queries. For n = 200 000 that’s ≈ 18 queries — well within typical limits of 20–30.

Two-Query Trick

If you query dist(u, x) and dist(v, x) for known u, v, the branch point where x’s subtree attaches to the u–v path is at distance (D + du − dv)/2 from u, where D = dist(u,v) is known from the tree structure. This dramatically narrows the search space.

Reconstructing Trees from Distance Queries

The judge hides a tree on n vertices. Query “? u v” returns dist(u, v). Reconstruct the tree.

BFS Layering + Parent Search

Phase 1: query dist(1, v) for all v to get the BFS layering (n−1 queries). Phase 2: for each vertex v at depth d, find its parent among depth-(d−1) vertices by querying until dist(u, v) = 1. 

// Reconstruct a hidden tree using distance queries
#include <bits/stdc++.h>
using namespace std;
int n, dist1[5005], par[5005];
int ask(int u, int v) {
    if (u == v) return 0;
    cout << "? " << u << " " << v << endl;
    int d; cin >> d; return d;
}
void solve() {
    cin >> n;
    for (int v = 2; v <= n; v++) dist1[v] = ask(1, v);
    vector<vector<int>> by_d(n + 1);
    for (int v = 1; v <= n; v++) by_d[dist1[v]].push_back(v);
    for (int d = 1; d < n; d++)
        for (int v : by_d[d])
            for (int u : by_d[d-1])
                if (ask(u, v) == 1) { par[v] = u; break; }
    cout << "!" << endl;
    for (int v = 2; v <= n; v++) cout << par[v] << " " << v << "\n";
    cout.flush();
}
int main() { solve(); }
Query Budget. The above is O(n2) worst-case (star graph). For stricter limits, use the LCA trick: depth(LCA(u,v)) = (dist(root,u) + dist(root,v) − dist(u,v)) / 2. This deduces parent relationships from fewer queries.

The LCA Distance Formula

For a rooted tree with known root distances, a single query dist(u, v) reveals the LCA depth, pinning down where u and v diverge. Combined with an Euler-tour ordering, this reconstructs the entire tree in O(n log n) queries.

BFS & DFS with Queries

Many interactive problems reduce to BFS or DFS on a hidden graph where you discover neighbours one query at a time.

Interactive BFS

Start from source s, discover its neighbours via queries, enqueue unvisited ones, repeat. Pruning: if the graph has max degree Δ, stop querying a vertex after finding Δ neighbours. For trees, stop once you find parent + all children.

Interactive DFS

Track the DFS stack explicitly. When a query reveals a back-edge (to an already-visited vertex), record it but don’t recurse. Early termination on degree bounds applies here too. 

// Interactive DFS — discover edges of a hidden graph (max degree D)
#include <bits/stdc++.h>
using namespace std;
int n, D; bool vis[1005];
vector<pair<int,int>> tree_e, back_e;
int ask(int u, int v) { cout << "? " << u << " " << v << endl; int r; cin >> r; return r; }
void dfs(int u) {
    vis[u] = true;
    int found = 0;
    for (int v = 1; v <= n && found < D; v++) {
        if (v == u) continue;
        if (ask(u, v)) {
            found++;
            if (!vis[v]) { tree_e.push_back({u, v}); dfs(v); }
            else back_e.push_back({u, v});
        }
    }
}
int main() {
    cin >> n >> D;
    memset(vis, 0, sizeof vis);
    dfs(1);
    cout << "! " << tree_e.size() << endl;
    for (auto [u,v] : tree_e) cout << u << " " << v << "\n";
    cout.flush();
}
Query Count. At most O(n · min(n, Δ)) queries. For sparse graphs (Δ « n) this is far better than the naïve O(n2).

Finding a Hamiltonian Path Interactively

Given a tournament (complete directed graph), find a Hamiltonian path. Query “? u v” returns 1 if the edge goes u→v, 0 otherwise. Every tournament has a Hamiltonian path; the challenge is the query budget.

Merge-Sort Approach: O(n log n)

Recursively find Hamiltonian paths in the left and right halves, then merge them like merge-sort. Merging two paths of lengths a and b costs O(a + b) queries. Total: O(n log n). 

// Hamiltonian path in a tournament — merge sort
#include <bits/stdc++.h>
using namespace std;
int ask(int u, int v) { cout << "? " << u << " " << v << endl; int r; cin >> r; return r; }
vector<int> merge_p(vector<int>& P, vector<int>& Q) {
    vector<int> res;
    int i = 0, j = 0;
    while (i < (int)P.size() && j < (int)Q.size()) {
        if (ask(P[i], Q[j])) res.push_back(P[i++]);
        else res.push_back(Q[j++]);
    }
    while (i < (int)P.size()) res.push_back(P[i++]);
    while (j < (int)Q.size()) res.push_back(Q[j++]);
    return res;
}
vector<int> build(vector<int>& v, int l, int r) {
    if (r - l == 1) return {v[l]};
    int m = (l + r) / 2;
    auto L = build(v, l, m), R = build(v, m, r);
    return merge_p(L, R);
}
int main() {
    int n; cin >> n;
    vector<int> v(n); iota(v.begin(), v.end(), 1);
    auto path = build(v, 0, n);
    cout << "!";
    for (int x : path) cout << " " << x;
    cout << endl;
}

Binary-Search Insertion Alternative

Maintain a path and insert vertices one at a time. To insert v into path [p1,…,pk], binary search: query v vs. pmid to decide left or right. Each insertion costs O(log k) queries; overall O(n log n).

Optimality. A tournament on n vertices has n! possible Hamiltonian paths. Each 1-bit query ⇒ lower bound log2(n!) = Θ(n log n). Both approaches match this.

Finding Edges Efficiently

A different query type: “? S” (subset query) returns 1 if any edge has both endpoints in set S, 0 otherwise. Goal: find a single edge in O(log2 n) queries.

Binary Search on Subsets

  1. Start with S = all vertices.
  2. Split into halves L, R. Query L.
  3. If 1, recurse on L. Else query R. If 1, recurse on R.
  4. Otherwise the edge crosses L–R: binary search each side for the endpoints.
// Find a hidden edge using subset queries
#include <bits/stdc++.h>
using namespace std;
int ask(vector<int>& S) {
    cout << "? " << S.size();
    for (int v : S) cout << " " << v;
    cout << endl;
    int r; cin >> r; return r;
}
int find_ep(vector<int>& cands, vector<int>& other) {
    if (cands.size() == 1) return cands[0];
    int m = cands.size() / 2;
    vector<int> L(cands.begin(), cands.begin() + m);
    vector<int> q = L;
    q.insert(q.end(), other.begin(), other.end());
    if (ask(q)) return find_ep(L, other);
    vector<int> R(cands.begin() + m, cands.end());
    return find_ep(R, other);
}
pair<int,int> find_edge(vector<int>& V) {
    if (V.size() <= 2) return {V[0], V[1]};
    int m = V.size() / 2;
    vector<int> L(V.begin(), V.begin() + m), R(V.begin() + m, V.end());
    if (ask(L)) return find_edge(L);
    if (ask(R)) return find_edge(R);
    int u = find_ep(L, R);
    vector<int> uv = {u};
    int v = find_ep(R, uv);
    return {u, v};
}
int main() {
    int n; cin >> n;
    vector<int> V(n); iota(V.begin(), V.end(), 1);
    auto [u, v] = find_edge(V);
    cout << "! " << u << " " << v << endl;
}

Finding All Edges of a Hidden Tree

With subset queries, find one edge, contract it (merge endpoints), repeat. Each edge-find costs O(log2 n); total for the tree: O(n log2 n) queries.

Classic CF Problems & Techniques

A survey of key interactive graph problems and their core ideas.

CF 1634D — Finding Zero

Query three indices at a time; the response reveals which contains the zero. A tournament-style elimination narrows candidates in O(n) queries.

CF 1498E — Two Houses

Given in-degree/out-degree info of a tournament, find two vertices in the same SCC. Sort by in-degree descending, then query adjacent pairs; the SCC structure guarantees a hit within O(n) queries.

CF 1081E — Missing Number

Hidden permutation, subset-sum queries. Binary search on prefixes to isolate individual elements. Think of it as discovering a hidden bipartite matching between positions and values.

General Technique: Divide & Conquer on Vertices

The single most recurring idea: split vertices into halves, query to determine which half contains the target, recurse. This converts O(n2) problems into O(n log n) or O(log2 n).

Common Patterns in Interactive Graph Problems:
  • Centroid decomposition — finding hidden vertices in trees.
  • Binary search on subsets — locating edges.
  • Merge-sort on comparisons — ordering / Hamiltonian paths.
  • BFS layering + LCA formula — tree reconstruction.
  • Randomisation — breaking adversarial structure.

Adaptive vs. Non-Adaptive Judges

Most CF problems use a non-adaptive judge (structure fixed beforehand). Some ICPC problems use an adaptive judge that picks answers to maximise your query count while staying consistent. Against adaptive judges: randomise query order, and ensure correctness against any consistent answer sequence.
