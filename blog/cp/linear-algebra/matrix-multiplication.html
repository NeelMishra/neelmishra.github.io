    Matrix Multiplication

    June 2026 · 18 min read

    Matrix multiplication is arguably the single most important linear algebra operation in competitive programming. It powers matrix exponentiation (which we'll cover next), graph reachability, DP optimization, and dozens of other techniques. In this post we'll build up from the textbook definition through several increasingly optimized implementations, all battle-tested for contest use.

    Prerequisites: You should be comfortable with the basics from Part 1 — Vectors & Matrices, especially how matrices are stored and indexed. We'll use 0-based indexing throughout.

    Naive Matrix Multiplication

    Recall the definition: given an A of size n × m and B of size m × p, the product C = A × B has size n × p where 

    C[i][j] = Σ (k = 0 to m-1) A[i][k] * B[k][j]

    This directly gives us an O(n · m · p) algorithm. When all dimensions are n, the complexity is O(n³). Here's the straightforward C++ implementation: 

    typedef vector<vector<long long>> Matrix;

Matrix multiply(const Matrix& A, const Matrix& B) {
    int n = A.size();
    int m = A[0].size();
    int p = B[0].size();
    Matrix C(n, vector<long long>(p, 0));
    for (int i = 0; i < n; i++)
        for (int j = 0; j < p; j++)
            for (int k = 0; k < m; k++)
                C[i][j] += A[i][k] * B[k][j];
    return C;
}

    When is this sufficient? For matrices up to about n = 500, the naive approach runs in roughly 125 million operations — tight but feasible within a 2-second time limit. For n ≤ 200 you'll have no issues at all. Most CP matrix multiplication problems actually use small matrices (2×2 to 10×10) inside matrix exponentiation, where the cubic cost per multiplication is negligible and the exponentiation's O(log k) factor dominates.

    Complexity summary:

    • Time: O(n · m · p), typically O(n³)
    • Space: O(n · p) for the result matrix

    Modular Matrix Multiplication

    In competitive programming, matrix entries almost always live in ℤ/MOD — typically MOD = 10⁹ + 7. This introduces two concerns: correctness (taking mod at the right time) and overflow (intermediate products can be huge).

    The Overflow Trap

    If matrix entries can be up to MOD − 1 ≈ 10⁹, then a single product A[i][k] * B[k][j] can be up to ~10¹⁸, which fits in a long long (max ~9.2 × 10¹⁸). But when we accumulate a sum of m such products without reducing, the running total can reach m × 10¹⁸. For m ≥ 9, that overflows long long.

    Solution: Take the modulus after each addition, or at least frequently enough that the accumulator never exceeds long long range. 

    const long long MOD = 1e9 + 7;

typedef vector<vector<long long>> Matrix;

Matrix multiply_mod(const Matrix& A, const Matrix& B) {
    int n = A.size();
    int m = A[0].size();
    int p = B[0].size();
    Matrix C(n, vector<long long>(p, 0));
    for (int i = 0; i < n; i++)
        for (int k = 0; k < m; k++)          // note: ikj order (see optimization section)
            for (int j = 0; j < p; j++) {
                C[i][j] += A[i][k] * B[k][j] % MOD;
                if (C[i][j] >= MOD) C[i][j] -= MOD;
            }
    return C;
}

    Why the Conditional Subtract?

    The expression A[i][k] * B[k][j] % MOD produces a value in [0, MOD−1]. Adding it to C[i][j] (which is already in [0, MOD−1]) gives at most 2·MOD − 2, which fits in long long. A single conditional subtraction is faster than a full % operation on most architectures.

    When MOD Is Larger

    If MOD can be up to ~10¹⁸ (rare, but it happens), then even a single a * b overflows long long. Options:

    • Use __int128 (GCC extension, available on most judges)
    • Use the __int128-based modmul: (__int128)a * b % MOD
    • Binary multiplication (multiply via repeated doubling), O(log b) per product — slow but safe
    // Safe modular multiplication for large MOD (up to ~4.6 * 10^18)
long long mulmod(long long a, long long b, long long mod) {
    return (__int128)a * b % mod;
}

Matrix multiply_large_mod(const Matrix& A, const Matrix& B, long long mod) {
    int n = A.size(), m = A[0].size(), p = B[0].size();
    Matrix C(n, vector<long long>(p, 0));
    for (int i = 0; i < n; i++)
        for (int k = 0; k < m; k++)
            for (int j = 0; j < p; j++)
                C[i][j] = (C[i][j] + mulmod(A[i][k], B[k][j], mod)) % mod;
    return C;
}

    Strassen's Algorithm

    Strassen's algorithm is the classic divide-and-conquer approach that reduces the exponent from 3 to approximately 2.807. The key insight: multiplying two 2×2 matrices normally requires 8 multiplications, but Strassen showed it can be done with only 7 (at the cost of more additions).

    The Idea

    Partition each n × n matrix into four (n/2) × (n/2) blocks: 

    A = | A11  A12 |    B = | B11  B12 |    C = | C11  C12 |
    | A21  A22 |        | B21  B22 |        | C21  C22 |

    Strassen computes seven intermediate products: 

    M1 = (A11 + A22)(B11 + B22)
M2 = (A21 + A22) B11
M3 = A11 (B12 - B22)
M4 = A22 (B21 - B11)
M5 = (A11 + A12) B22
M6 = (A21 - A11)(B11 + B12)
M7 = (A12 - A22)(B21 + B22)

    Then assembles the result: 

    C11 = M1 + M4 - M5 + M7
C12 = M3 + M5
C21 = M2 + M4
C22 = M1 - M2 + M3 + M6

    C++ Sketch

    A full implementation is lengthy. Here's a simplified sketch for square matrices whose dimensions are powers of 2: 

    typedef vector<vector<long long>> Matrix;

Matrix add(const Matrix& A, const Matrix& B) {
    int n = A.size();
    Matrix C(n, vector<long long>(n));
    for (int i = 0; i < n; i++)
        for (int j = 0; j < n; j++)
            C[i][j] = A[i][j] + B[i][j];
    return C;
}

Matrix sub(const Matrix& A, const Matrix& B) {
    int n = A.size();
    Matrix C(n, vector<long long>(n));
    for (int i = 0; i < n; i++)
        for (int j = 0; j < n; j++)
            C[i][j] = A[i][j] - B[i][j];
    return C;
}

Matrix strassen(const Matrix& A, const Matrix& B) {
    int n = A.size();
    if (n <= 64) return multiply(A, B);  // base case: naive

    int half = n / 2;
    // Split into quadrants
    Matrix A11(half, vector<long long>(half)), A12(half, vector<long long>(half)),
           A21(half, vector<long long>(half)), A22(half, vector<long long>(half));
    Matrix B11(half, vector<long long>(half)), B12(half, vector<long long>(half)),
           B21(half, vector<long long>(half)), B22(half, vector<long long>(half));

    for (int i = 0; i < half; i++)
        for (int j = 0; j < half; j++) {
            A11[i][j] = A[i][j];           A12[i][j] = A[i][j + half];
            A21[i][j] = A[i + half][j];    A22[i][j] = A[i + half][j + half];
            B11[i][j] = B[i][j];           B12[i][j] = B[i][j + half];
            B21[i][j] = B[i + half][j];    B22[i][j] = B[i + half][j + half];
        }

    Matrix M1 = strassen(add(A11, A22), add(B11, B22));
    Matrix M2 = strassen(add(A21, A22), B11);
    Matrix M3 = strassen(A11, sub(B12, B22));
    Matrix M4 = strassen(A22, sub(B21, B11));
    Matrix M5 = strassen(add(A11, A12), B22);
    Matrix M6 = strassen(sub(A21, A11), add(B11, B12));
    Matrix M7 = strassen(sub(A12, A22), add(B21, B22));

    // Assemble result
    Matrix C(n, vector<long long>(n));
    for (int i = 0; i < half; i++)
        for (int j = 0; j < half; j++) {
            C[i][j]                 = M1[i][j] + M4[i][j] - M5[i][j] + M7[i][j];
            C[i][j + half]          = M3[i][j] + M5[i][j];
            C[i + half][j]          = M2[i][j] + M4[i][j];
            C[i + half][j + half]   = M1[i][j] - M2[i][j] + M3[i][j] + M6[i][j];
        }
    return C;
}

    Why Strassen Rarely Appears in CP

    • Constant factor: The 18 matrix additions/subtractions per recursive step make it slower than naive for matrices under ~128×128 in practice.
    • Memory overhead: All those temporary sub-matrices eat memory and thrash cache.
    • Numerical stability: With modular arithmetic this isn't a concern, but the extra complexity isn't worth it for the typical small matrices in CP.
    • Implementation time: In a contest, correctness and speed of coding matter. The naive approach with minor optimizations is almost always the right call.

    Complexity: O(nlog₂7) ≈ O(n2.807). The crossover point where Strassen beats naive is typically around n = 100–256 depending on the machine and implementation quality. For completeness it's great to know, but don't reach for it in a contest.

    Optimizing for Competitive Programming

    When you're multiplying matrices of size n ≈ 500 or working under tight time limits, constant-factor optimizations matter. Here are the most impactful tricks.

    Loop Order: ikj vs ijk

    The textbook ijk loop accesses B[k][j] with k varying in the innermost loop — this jumps across rows of B, causing cache misses. Swapping to ikj order makes the innermost loop iterate over j, accessing both C[i][j] and B[k][j] sequentially along rows. 

    // Cache-friendly ikj loop order — typically 2-4x faster than ijk for large n
for (int i = 0; i < n; i++)
    for (int k = 0; k < m; k++)
        for (int j = 0; j < p; j++)
            C[i][j] += A[i][k] * B[k][j];

    This single change can give a 2–4× speedup for n ≥ 200. Always use ikj order.

    Fixed-Size Arrays vs Vectors

    vector<vector<long long>> stores each row in a separate heap allocation. For performance-critical code, use a flat fixed-size array: 

    const int MAXN = 512;
const long long MOD = 1e9 + 7;

struct Matrix {
    long long a[MAXN][MAXN];
    int n, m;

    Matrix() : n(0), m(0) { memset(a, 0, sizeof(a)); }
    Matrix(int _n, int _m) : n(_n), m(_m) { memset(a, 0, sizeof(a)); }
};

Matrix multiply(const Matrix& A, const Matrix& B) {
    Matrix C(A.n, B.m);
    for (int i = 0; i < A.n; i++)
        for (int k = 0; k < A.m; k++) {
            long long aik = A.a[i][k];
            if (aik == 0) continue;   // skip zero entries
            for (int j = 0; j < B.m; j++) {
                C.a[i][j] += aik * B.a[k][j] % MOD;
                if (C.a[i][j] >= MOD) C.a[i][j] -= MOD;
            }
        }
    return C;
}

    Note the if (aik == 0) continue; — for sparse matrices, this can skip a huge number of inner iterations. Even for dense matrices it costs almost nothing.

    Small Matrix Template (for Matrix Exponentiation)

    When your matrix is tiny (say 2×2 or 5×5), templates or compile-time sizes let the compiler unroll loops and optimize aggressively: 

    template<int N>
struct Mat {
    long long a[N][N];
    Mat() { memset(a, 0, sizeof(a)); }

    Mat operator*(const Mat& o) const {
        Mat res;
        for (int i = 0; i < N; i++)
            for (int k = 0; k < N; k++) {
                if (a[i][k] == 0) continue;
                for (int j = 0; j < N; j++)
                    res.a[i][j] = (res.a[i][j] + a[i][k] * o.a[k][j]) % MOD;
            }
        return res;
    }

    static Mat identity() {
        Mat I;
        for (int i = 0; i < N; i++) I.a[i][i] = 1;
        return I;
    }
};

template<int N>
Mat<N> power(Mat<N> base, long long exp) {
    Mat<N> result = Mat<N>::identity();
    while (exp > 0) {
        if (exp & 1) result = result * base;
        base = base * base;
        exp >>= 1;
    }
    return result;
}

// Usage: Fibonacci via 2x2 matrix exponentiation
// | F(n+1) |   | 1 1 |^n   | F(1) |
// | F(n)   | = | 1 0 |   * | F(0) |
const long long MOD = 1e9 + 7;

long long fibonacci(long long n) {
    if (n <= 1) return n;
    Mat<2> M;
    M.a[0][0] = 1; M.a[0][1] = 1;
    M.a[1][0] = 1; M.a[1][1] = 0;
    Mat<2> result = power(M, n - 1);
    return result.a[0][0];
}

    Compiler Pragmas

    These GCC pragmas can squeeze extra performance out of tight loops: 

    #pragma GCC optimize("O3,unroll-loops")
#pragma GCC target("avx2,bmi,bmi2,popcnt")

    The avx2 target enables 256-bit SIMD instructions. Combined with the ikj loop order (which gives the compiler a nice sequential inner loop), this can auto-vectorize the multiplication and deliver another 2–4× speedup on judges that support it (most Codeforces judges do).

    Boolean Matrix Multiplication

    A special but frequently encountered case: both matrices contain only 0s and 1s, and we want C[i][j] = OR over k of (A[i][k] AND B[k][j]). Equivalently, C[i][j] = 1 if and only if there exists some k such that A[i][k] = 1 and B[k][j] = 1.

    Application: Transitive Closure

    Given an adjacency matrix of a directed graph, we want to know which vertices can reach which. The transitive closure is: 

    T = I ∨ A ∨ A² ∨ A³ ∨ ... ∨ A^(n-1)

    Or equivalently, compute (I + A)^n using Boolean matrix multiplication (where + is OR and × is AND). This is the basis of the Floyd–Warshall reachability algorithm, but matrix multiplication lets us use repeated squaring for O(n³ log n).

    Bitset Optimization: O(n³ / 64)

    The key insight: store each row of A and each column of B as a bitset. Then C[i][j] = 1 iff the bitwise AND of row i of A and column j of B is nonzero. A 64-bit AND-and-compare replaces 64 individual checks. 

    #include <bitset>
const int MAXN = 2000;

bitset<MAXN> A[MAXN], B[MAXN], C[MAXN];
// A[i] = row i of matrix A
// B[j] = column j of matrix B (stored transposed for efficiency)

void bool_multiply(int n) {
    // Multiply A * B where B is stored as columns
    for (int i = 0; i < n; i++) {
        C[i].reset();
        for (int j = 0; j < n; j++) {
            if ((A[i] & B[j]).any()) {
                C[i].set(j);
            }
        }
    }
}

    But we can do better. The above is still O(n³ / 64) but has a bad inner loop. A more efficient approach avoids the per-(i,j) check entirely: 

    // Faster: for each bit k set in row i of A, OR in row k of B
void bool_multiply_fast(int n) {
    for (int i = 0; i < n; i++) {
        C[i].reset();
        for (int k = 0; k < n; k++) {
            if (A[i][k]) {
                C[i] |= B[k];   // B[k] = row k of original B (not transposed)
            }
        }
    }
}
// Here B[k] stores row k of the second matrix.
// The |= operation processes all n bits in O(n/64) time.

    This version is cleaner and usually faster in practice. The inner |= on bitsets of size n runs in O(n / 64) time, and we do it at most n times per row, giving O(n³ / 64) total — a 64× speedup over naive Boolean multiplication.

    Transitive Closure with Bitsets

    // Compute transitive closure of adjacency matrix adj
// adj[i] = bitset representing edges from vertex i
bitset<MAXN> adj[MAXN];

void transitive_closure(int n) {
    // Repeated squaring: O(n^3 / 64 * log n)
    // Or use Floyd-Warshall style:
    for (int k = 0; k < n; k++)
        for (int i = 0; i < n; i++)
            if (adj[i][k])
                adj[i] |= adj[k];
    // This is O(n^3 / 64) — same complexity, simpler code
}

    The bitset Floyd–Warshall is one of the most elegant tricks in CP. It handles n ≤ 2000 effortlessly within typical time limits, whereas the naive O(n³) Boolean version would TLE at around n = 1500.

    Common Mistakes and Edge Cases

    1. Dimension Mismatch

    You can only multiply n × m by m × p. The number of columns of A must equal the number of rows of B. In contest code this usually isn't checked — make sure you set up your matrices with matching inner dimensions. 

    // WRONG: assuming both matrices are n x n when they might not be
Matrix multiply(const Matrix& A, const Matrix& B) {
    int n = A.size();
    // BUG: using n for all dimensions
    // FIX: use A.size(), A[0].size(), B[0].size() separately
}

    2. Overflow in Modular Multiplication

    The most common bug. If A[i][k] and B[k][j] are both up to 10⁹ + 6, their product is about 10¹⁸, which fits in long long. But if you forget to mod the product before adding it to the accumulator, a few additions can overflow. 

    // DANGEROUS: may overflow if m is large
C[i][j] += A[i][k] * B[k][j];
C[i][j] %= MOD;

// SAFE: mod the product first, then add
C[i][j] += A[i][k] * B[k][j] % MOD;
if (C[i][j] >= MOD) C[i][j] -= MOD;

// ALSO SAFE: cast to ensure no intermediate overflow
C[i][j] = (C[i][j] + (long long)A[i][k] * B[k][j] % MOD) % MOD;

    3. Forgetting the Identity Matrix

    When implementing matrix exponentiation, the base case (exponent = 0) must return the identity matrix, not a zero matrix. The identity matrix has 1s on the main diagonal and 0s elsewhere: 

    Matrix identity(int n) {
    Matrix I(n, vector<long long>(n, 0));
    for (int i = 0; i < n; i++) I[i][i] = 1;
    return I;
}

    4. Using int Instead of long long

    If your matrix entries are int (say up to 10⁹), the product a[i][k] * b[k][j] overflows int. Always use long long for the multiplication, or store everything as long long from the start. 

    // BUG: int * int overflows
int a[N][N], b[N][N], c[N][N];
c[i][j] += a[i][k] * b[k][j];  // overflow!

// FIX: cast one operand
c[i][j] += (long long)a[i][k] * b[k][j] % MOD;

    5. Not Clearing the Result Matrix

    When reusing a matrix struct, make sure the result is zeroed out before accumulating products. Using memset(C.a, 0, sizeof(C.a)) or initializing with the constructor is essential.

    6. Off-by-One in Matrix Dimensions

    If your recurrence has states numbered 1 through k, your matrix should be k × k. A common mistake is making it (k+1) × (k+1) or (k−1) × (k−1). Double-check by manually verifying the first few terms of your recurrence against the matrix multiplication result.
