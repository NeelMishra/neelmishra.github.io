    Matrix Exponentiation

    March 2026 · 35 min read

    Why Matrix Exponentiation?

    Consider the Fibonacci recurrence: F(n) = F(n-1) + F(n-2). A naive loop computes F(n) in O(n) time. That is fine for n ≤ 107, but competitive programming regularly asks for n up to 1018. We need a way to jump ahead in the recurrence without iterating through every single term.

    The key insight is that any linear recurrence can be expressed as repeated multiplication by a constant transformation matrix. Applying that matrix k times gives us the k-th term. And because matrix multiplication is associative, we can compute Mk using binary exponentiation in only O(log k) multiplications—each costing O(n³) for n×n matrices.

    For Fibonacci, the transformation is:

    | F(n+1) |   | 1  1 |   | F(n)   |
| F(n)   | = | 1  0 | * | F(n-1) |

    So F(n) is the top-left entry of:

    | 1  1 |^n     | F(1) |
| 1  0 |   *   | F(0) |

    This gives us F(n) in O(8 · log n) = O(log n) operations (the matrix is 2×2, so each multiply is 2³ = 8 operations). Matrix exponentiation is one of the most versatile techniques in competitive programming and appears in hundreds of problems across Codeforces, AtCoder, and ICPC.

    When to reach for matrix exponentiation:

    • Linear recurrences with constant coefficients and very large n (≥ 109)
    • Counting paths of exact length k in a graph
    • State-machine transitions iterated k times
    • Any DP where the transition can be written as a matrix–vector product

    The Algorithm

    Matrix exponentiation is simply binary exponentiation (fast power) with the base being a matrix instead of a scalar. Recall the scalar version:

    // Scalar binary exponentiation: compute base^exp mod m
long long power(long long base, long long exp, long long m) {
    long long result = 1;
    base %= m;
    while (exp > 0) {
        if (exp & 1) result = result * base % m;
        base = base * base % m;
        exp >>= 1;
    }
    return result;
}

    For matrices we replace:

    • 1 → the identity matrix I (ones on the diagonal)
    • * → matrix multiplication (O(n³) per multiply)
    • % m → take every entry mod m after each multiply

    The overall complexity is O(n³ log k) where n is the matrix dimension and k is the exponent. For competitive programming, n is typically small (2–200) while k can be up to 1018.

    C++ Implementation

    Here is a clean, reusable matrix exponentiation template. The matrix size is set at compile time via the N constant, and all arithmetic is done modulo MOD.

    #include <bits/stdc++.h>
using namespace std;

const int MOD = 1e9 + 7;

struct Matrix {
    int n;
    vector<vector<long long>> a;

    Matrix(int _n) : n(_n), a(_n, vector<long long>(_n, 0)) {}

    // Identity matrix
    static Matrix identity(int n) {
        Matrix I(n);
        for (int i = 0; i < n; i++) I.a[i][i] = 1;
        return I;
    }

    Matrix operator*(const Matrix& other) const {
        Matrix res(n);
        for (int i = 0; i < n; i++)
            for (int k = 0; k < n; k++) {
                if (a[i][k] == 0) continue;  // small optimization
                for (int j = 0; j < n; j++)
                    res.a[i][j] = (res.a[i][j] + a[i][k] * other.a[k][j]) % MOD;
            }
        return res;
    }
};

// Compute M^exp mod MOD using binary exponentiation
Matrix mat_pow(Matrix M, long long exp) {
    Matrix result = Matrix::identity(M.n);
    while (exp > 0) {
        if (exp & 1) result = result * M;
        M = M * M;
        exp >>= 1;
    }
    return result;
}
    Loop order matters. The i-k-j loop order above (instead of the textbook i-j-k) is cache-friendly: we fix row i and column k of the left matrix, then sweep across columns of the right matrix. This can give a 2–3× speedup for larger matrices. The if (a[i][k] == 0) continue check skips an entire inner loop when the entry is zero, which helps for sparse transformation matrices.

    Fixed-Size Version (Faster)

    When you know the matrix size at compile time, use arrays instead of vectors for better cache performance:

    const int N = 2;  // change per problem
const int MOD = 1e9 + 7;

struct Mat {
    long long a[N][N];
    Mat() { memset(a, 0, sizeof(a)); }
};

Mat multiply(const Mat& A, const Mat& B) {
    Mat C;
    for (int i = 0; i < N; i++)
        for (int k = 0; k < N; k++) {
            if (A.a[i][k] == 0) continue;
            for (int j = 0; j < N; j++)
                C.a[i][j] = (C.a[i][j] + A.a[i][k] * B.a[k][j]) % MOD;
        }
    return C;
}

Mat mat_pow(Mat M, long long p) {
    Mat result;
    for (int i = 0; i < N; i++) result.a[i][i] = 1; // identity
    while (p > 0) {
        if (p & 1) result = multiply(result, M);
        M = multiply(M, M);
        p >>= 1;
    }
    return result;
}

    Fibonacci and Linear Recurrences

    Fibonacci: The Canonical Example

    Given F(0) = 0, F(1) = 1, F(n) = F(n-1) + F(n-2), compute F(n) mod 109+7 for n up to 1018.

    long long fibonacci(long long n) {
    if (n <= 1) return n;
    Mat M;
    M.a[0][0] = 1; M.a[0][1] = 1;
    M.a[1][0] = 1; M.a[1][1] = 0;

    Mat res = mat_pow(M, n - 1);
    // res.a[0][0] = F(n), res.a[0][1] = F(n-1)
    return res.a[0][0];
}

int main() {
    long long n;
    cin >> n;
    cout << fibonacci(n) << "\n";
}

    General k-th Order Linear Recurrences

    A k-th order recurrence has the form:

    f(n) = c₁·f(n-1) + c₂·f(n-2) + ... + cₖ·f(n-k)

    The transformation matrix is k × k:

    | f(n)   |   | c₁ c₂ c₃ ... cₖ |   | f(n-1) |
| f(n-1) |   | 1  0  0  ... 0  |   | f(n-2) |
| f(n-2) | = | 0  1  0  ... 0  | * | f(n-3) |
|  ...   |   | ...             |   |  ...   |
| f(n-k+1)|  | 0  0  ... 1  0  |   | f(n-k) |

    The first row contains the recurrence coefficients. Each subsequent row shifts the state down by one position (a 1 on the subdiagonal). Here is a complete solver:

    // Solve: f(n) = c[0]*f(n-1) + c[1]*f(n-2) + ... + c[k-1]*f(n-k)
// init[0] = f(0), init[1] = f(1), ..., init[k-1] = f(k-1)
long long solve_recurrence(vector<long long>& c,
                           vector<long long>& init, long long n) {
    int k = c.size();
    if (n < k) return init[n] % MOD;

    Matrix M(k);
    for (int j = 0; j < k; j++)
        M.a[0][j] = (c[j] % MOD + MOD) % MOD;
    for (int i = 1; i < k; i++)
        M.a[i][i - 1] = 1;

    Matrix res = mat_pow(M, n - k + 1);

    long long ans = 0;
    for (int j = 0; j < k; j++)
        ans = (ans + res.a[0][j] * (init[k - 1 - j] % MOD)) % MOD;
    return ans;
}

    Graph Problems

    Counting Paths of Exact Length k

    A classic result from algebraic graph theory: if A is the adjacency matrix of a graph with V vertices, then Ak[i][j] equals the number of walks of length exactly k from vertex i to vertex j.

    This is extremely useful in CP. Compute Ak using matrix exponentiation in O(V³ log k).

    // Given adjacency matrix adj[V][V] and length k,
// count walks of length k between all pairs
int V;  // number of vertices

void solve() {
    cin >> V;
    long long k;
    cin >> k;

    Matrix A(V);
    for (int i = 0; i < V; i++)
        for (int j = 0; j < V; j++)
            cin >> A.a[i][j];

    Matrix result = mat_pow(A, k);

    // result.a[i][j] = number of walks of length k from i to j (mod MOD)
    for (int i = 0; i < V; i++) {
        for (int j = 0; j < V; j++)
            cout << result.a[i][j] << " ";
        cout << "\n";
    }
}
    Walks vs paths. Matrix exponentiation counts walks (vertices and edges may repeat), not simple paths (no repetition). Counting simple paths of length k is NP-hard in general. For most CP problems, walks are what's asked for.

    Shortest Paths Variant

    Replace the multiplication ring (*, +) with (+, min) to compute shortest paths of exactly k edges. The “matrix product” becomes:

    C[i][j] = min over all m of (A[i][m] + B[m][j])

    This is the min-plus semiring (also called the tropical semiring). The identity matrix has 0 on the diagonal and ∞ elsewhere. Applying binary exponentiation with this product computes shortest paths with exactly k edges in O(V³ log k).

    const long long INF = 1e18;

struct MinPlusMat {
    int n;
    vector<vector<long long>> a;
    MinPlusMat(int _n) : n(_n), a(_n, vector<long long>(_n, INF)) {}

    static MinPlusMat identity(int n) {
        MinPlusMat I(n);
        for (int i = 0; i < n; i++) I.a[i][i] = 0;
        return I;
    }

    MinPlusMat operator*(const MinPlusMat& o) const {
        MinPlusMat res(n);
        for (int i = 0; i < n; i++)
            for (int k = 0; k < n; k++) {
                if (a[i][k] == INF) continue;
                for (int j = 0; j < n; j++)
                    res.a[i][j] = min(res.a[i][j], a[i][k] + o.a[k][j]);
            }
        return res;
    }
};

    Reachability in At Most k Steps

    To count walks of length at most k (instead of exactly k), augment the adjacency matrix with a self-loop column. Add a “sink” vertex that has a self-loop and is reachable from every vertex. Alternatively, compute (I + A)k: the identity allows “staying put” at each step, effectively summing A0 + A1 + … + Ak.

    Advanced Applications

    Recurrences with Constant Terms

    What if the recurrence has a constant additive term? For example:

    f(n) = 2·f(n-1) + 3·f(n-2) + 5

    Extend the state vector with an extra entry that is always 1:

    | f(n)   |   | 2  3  5 |   | f(n-1) |
| f(n-1) | = | 1  0  0 | * | f(n-2) |
|   1    |   | 0  0  1 |   |   1    |

    The third row preserves the constant 1. The first row’s third entry (5) adds the constant term at each step. This extends naturally to polynomial terms—see below.

    Recurrences with Polynomial Terms

    For f(n) = a·f(n-1) + P(n) where P(n) is a polynomial of degree d, we can track n0, n1, …, nd as extra state variables. The key identity is:

    (n+1)^k = sum of C(k,j) * n^j  for j = 0..k

    So we build the transformation matrix using binomial coefficients to advance each power of n. For example, with f(n) = 2·f(n-1) + n²:

    State: [f(n), n², n, 1]

| f(n+1)  |   | 2  1  0  0 |   | f(n) |
| (n+1)²  | = | 0  1  2  1 | * | n²   |
| (n+1)   |   | 0  0  1  1 |   | n    |
|    1     |   | 0  0  0  1 |   | 1    |

    Block Matrix Technique

    Sometimes the DP state has two components, e.g., you track both a sum and a count. Group them into a “block matrix” where each entry is itself a small matrix. This keeps the overall matrix size manageable while encoding richer transitions.

    A common pattern: computing ∑ f(i) for i = 1 to n. Track both f(n) and the prefix sum S(n) = S(n-1) + f(n):

    | S(n) |   | 1  1  1 |   | S(n-1) |
| f(n) | = | 0  1  1 | * | f(n-1) |
| f(n-1)| | 0  1  0 |   | f(n-2) |

    Cayley–Hamilton Optimization

    By the Cayley–Hamilton theorem, every n×n matrix satisfies its own characteristic polynomial of degree n. This means Mn can be expressed as a linear combination of M0, M1, …, Mn-1. For very large exponents, we can:

    1. Compute the characteristic polynomial (degree n).
    2. Reduce the exponent modulo this polynomial using polynomial exponentiation.
    3. Evaluate the resulting polynomial of degree < n on the matrix.

    This reduces the cost from O(n³ log k) to O(n² log k + n³). For large n and huge k, this is a significant improvement. The technique is also known as Berlekamp–Massey + Kitamasa in the CP community for computing linear recurrences in O(n² log k) or even O(n log n log k) with polynomial multiplication via NTT.

    Common Pitfalls

    Integer Overflow

    The most common bug. When multiplying two entries that are each up to MOD-1 ≈ 109, the product can reach ~1018, which fits in long long but not in int. Always use long long for the matrix entries.

    // WRONG: overflow when a[i][k] and b[k][j] are both ~1e9
int res = (res + a[i][k] * b[k][j]) % MOD;  // int * int can overflow!

// CORRECT: use long long
long long res = (res + (long long)a[i][k] * b[k][j]) % MOD;

    Matrix Size Limits

    Matrix exponentiation is efficient only when the matrix is small. A rough guideline:

    • n ≤ 100: Very comfortable. Even 1018 exponents finish in milliseconds.
    • n ≈ 200–300: Borderline. Each multiplication is ~107 operations; 60 multiplications for exponent 1018 gives ~6×108 operations. Tight but possible with fast I/O and optimized code.
    • n ≥ 500: Too slow for most time limits. Consider Cayley–Hamilton or a different approach.

    Identity Matrix Initialization

    Forgetting to initialize the result as the identity matrix (not the zero matrix) is a common mistake. The identity matrix is the multiplicative neutral element:

    // WRONG
Mat result;  // zero matrix
// result * M = zero, not M!

// CORRECT
Mat result;
for (int i = 0; i < N; i++) result.a[i][i] = 1;

    Edge Cases

    • Exponent 0: M0 = I (identity). Make sure your code handles this.
    • Exponent 1: M1 = M. No multiplication needed.
    • Negative coefficients: If recurrence coefficients can be negative, take care with modular arithmetic: add MOD before taking the remainder to keep values positive.
    // Handling negative coefficients in mod arithmetic
long long coeff = -3;
long long safe_coeff = ((coeff % MOD) + MOD) % MOD;  // ensures positive

    When Matrix Exponentiation is Overkill

    Not every recurrence needs matrix exponentiation:

    • If n ≤ 107, a simple loop is faster and simpler.
    • If the recurrence has non-constant coefficients (coefficients depend on n), matrix exponentiation does not directly apply.
    • If the recurrence is over a non-ring structure (e.g., max instead of addition), check whether the semiring version works.
    • For higher-order recurrences with n ≤ 106, simple memoization or iterative computation is usually sufficient.
