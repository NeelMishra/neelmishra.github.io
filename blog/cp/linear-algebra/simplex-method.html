Simplex Method — Linear Programming for CP

January 2026 · 25 min read

What Is Linear Programming?

A Linear Program (LP) is an optimisation problem where you maximise (or minimise) a linear objective function subject to linear inequality and equality constraints. Despite sounding abstract, LPs appear surprisingly often in competitive programming — from network flow to matching to resource allocation.

Standard Form

Every LP can be written in standard form:

Maximise c1x1 + c2x2 + … + cnxn Subject to a11x1 + a12x2 + … + a1nxn ≤ b1 a21x1 + a22x2 + … + a2nxn ≤ b2 ⋮ am1x1 + am2x2 + … + amnxn ≤ bm x1, x2, …, xn ≥ 0

In compact matrix notation: maximise cTx subject to Ax ≤ b, x ≥ 0. The vector x contains the decision variables, c is the objective vector, A is the constraint matrix, and b is the right-hand side.

Converting to Standard Form

  • Minimise cTx → Maximise −cTx
  • ≥ constraint: multiply both sides by −1 to get ≤
  • Equality constraint: replace with one ≤ and one ≥ constraint
  • Unrestricted variable: replace xi with xi+ − xi− where both ≥ 0
Geometry. The feasible region of an LP is a convex polytope. The simplex method works by moving along edges of this polytope from vertex to vertex, always improving the objective, until it reaches the optimum — which is always at a vertex (or along an edge between two optimal vertices).

The Simplex Algorithm

The simplex method is the workhorse algorithm for solving LPs. It operates on the simplex tableau — an augmented matrix that encodes the LP along with slack variables.

Slack Variables & Initial Tableau

Convert each inequality aiTx ≤ bi into an equality by adding a slack variable si ≥ 0:

aiTx + si = bi

The initial basic feasible solution (BFS) sets all original variables to 0 and each slack variable to bi. This is feasible when all bi ≥ 0.

Pivot Selection

Each iteration of the simplex method consists of:

  1. Choose entering variable: Pick a non-basic variable with a positive coefficient in the objective row (it can improve the objective).
  2. Choose leaving variable: Among rows with positive coefficients in the entering column, pick the one with the smallest ratio bi / aij (minimum ratio test — ensures feasibility).
  3. Pivot: Use row operations to make the entering variable basic in the leaving variable's row.

Bland's Rule (Anti-Cycling)

Degenerate LPs can cause the simplex method to cycle — revisiting the same BFS repeatedly without improving the objective. Bland's rule prevents cycling:

  • Always choose the entering variable with the smallest index among those with positive objective coefficients.
  • Always choose the leaving variable with the smallest index among those with the minimum ratio.
Cycling is rare in practice but can appear in carefully constructed contest problems. Always implement Bland's rule or use perturbation (lexicographic ordering) to be safe.

Termination

The algorithm terminates when:

  • Optimal: No non-basic variable has a positive objective coefficient — the current BFS is optimal.
  • Unbounded: An entering variable has no positive coefficient in any constraint row — the objective can grow without bound.

C++ Implementation

Below is a complete, contest-ready simplex solver. It handles maximisation in standard form with ≤ constraints. The implementation uses Bland's rule for cycling prevention and an epsilon tolerance for numerical stability. 

#include <bits/stdc++.h>
using namespace std;

typedef double T;
const T EPS = 1e-9;
const T INF = 1e18;

// Solves: maximise c^T x subject to Ax <= b, x >= 0
// Returns: 0 = optimal, 1 = infeasible, -1 = unbounded
// On optimal: res = optimal value, x = solution vector
struct Simplex {
    int m, n;           // m constraints, n variables
    vector<vector<T>> tab; // tableau: (m+1) x (n+m+1)
    vector<int> basis;     // basis[i] = variable index in row i

    // A: m x n constraint matrix, b: m-vector RHS, c: n-vector objective
    Simplex(vector<vector<T>>& A, vector<T>& b, vector<T>& c)
        : m(A.size()), n(A[0].size()), tab(m + 1, vector<T>(n + m + 1, 0)),
          basis(m) {
        // Build tableau: [A | I | b] with objective row [-c | 0 | 0]
        for (int i = 0; i < m; i++) {
            for (int j = 0; j < n; j++)
                tab[i][j] = A[i][j];
            tab[i][n + i] = 1;        // slack variable
            tab[i][n + m] = b[i];     // RHS
            basis[i] = n + i;
        }
        for (int j = 0; j < n; j++)
            tab[m][j] = -c[j];        // objective row (negated for max)
    }

    void pivot(int r, int s) {
        T inv = 1.0 / tab[r][s];
        for (int j = 0; j <= n + m; j++)
            tab[r][j] *= inv;
        for (int i = 0; i <= m; i++) {
            if (i == r) continue;
            T f = tab[i][s];
            for (int j = 0; j <= n + m; j++)
                tab[i][j] -= f * tab[r][j];
        }
        basis[r] = s;
    }

    int solve(T& res, vector<T>& x) {
        // Phase 1: ensure all b[i] >= 0
        while (true) {
            int r = -1;
            for (int i = 0; i < m; i++)
                if (tab[i][n + m] < -EPS && (r == -1 || tab[i][n + m] < tab[r][n + m]))
                    r = i;
            if (r == -1) break;
            int s = -1;
            for (int j = 0; j < n + m; j++)
                if (tab[r][j] < -EPS && (s == -1 || j < s))
                    s = j;   // Bland's rule: smallest index
            if (s == -1) return 1; // infeasible
            pivot(r, s);
        }
        // Phase 2: optimise
        while (true) {
            int s = -1;
            for (int j = 0; j < n + m; j++)
                if (tab[m][j] < -EPS && (s == -1 || j < s))
                    s = j;   // Bland's rule: smallest index
            if (s == -1) break;  // optimal
            int r = -1;
            for (int i = 0; i < m; i++) {
                if (tab[i][s] < EPS) continue;
                if (r == -1 || tab[i][n + m] / tab[i][s]
                             < tab[r][n + m] / tab[r][s] - EPS
                    || (abs(tab[i][n + m] / tab[i][s]
                          - tab[r][n + m] / tab[r][s]) < EPS
                        && basis[i] < basis[r]))
                    r = i;   // min ratio, Bland's tie-break
            }
            if (r == -1) return -1; // unbounded
            pivot(r, s);
        }
        res = -tab[m][n + m]; // negate because we stored -c
        x.assign(n, 0);
        for (int i = 0; i < m; i++)
            if (basis[i] < n)
                x[basis[i]] = tab[i][n + m];
        return 0;
    }
};

int main() {
    // Example: maximise 5x + 4y subject to
    //   6x + 4y <= 24
    //   x + 2y  <= 6
    //   x, y    >= 0
    int m = 2, n = 2;
    vector<vector<T>> A = {{6, 4}, {1, 2}};
    vector<T> b = {24, 6}, c = {5, 4};
    T res;
    vector<T> x;
    Simplex lp(A, b, c);
    int status = lp.solve(res, x);
    if (status == 0) {
        printf("Optimal value: %.6f\n", res);
        for (int i = 0; i < n; i++)
            printf("x[%d] = %.6f\n", i, x[i]);
    } else if (status == 1) {
        puts("Infeasible");
    } else {
        puts("Unbounded");
    }
    return 0;
}
Numeric precision. Use long double or rational arithmetic if the problem demands exact answers. The EPS value of 10−9 works for most contest problems, but tighten or loosen it as needed. For integer LPs on small inputs, consider rounding the result.

Handling Minimisation & Equality Constraints

To minimise, negate the objective vector c and negate the result. For an equality constraint aTx = b, add it as both aTx ≤ b and aTx ≥ b (i.e., −aTx ≤ −b). If bi < 0 for some constraint, the Phase 1 logic in the code above handles it by pivoting to restore feasibility.

Duality

Every LP (the primal) has an associated dual LP. Understanding duality gives you powerful reasoning tools and sometimes lets you solve the dual instead when it has fewer variables or simpler structure.

Primal-Dual Pair

Primal

Maximise cTx

Subject to Ax ≤ b, x ≥ 0

Dual

Minimise bTy

Subject to ATy ≥ c, y ≥ 0

The dual has one variable yi per primal constraint and one constraint per primal variable. The dual of the dual is the primal.

Weak Duality

For any feasible primal solution x and feasible dual solution y:

cTx ≤ bTy

This means any dual feasible solution provides an upper bound on the primal optimum. This is extremely useful for proving optimality or bounding answers in CP.

Strong Duality

If either the primal or dual has a finite optimal solution, then both do, and their optimal values are equal:

cTx* = bTy*

Complementary Slackness

At optimality, for each primal variable xj and corresponding dual constraint:

  • If xj > 0, then the dual constraint is tight (equality holds).
  • If the dual constraint is slack (strict inequality), then xj = 0.

Similarly for dual variables and primal constraints. This relationship is key to understanding the structure of optimal solutions and is used in proofs that min-cost flow equals the LP relaxation, and that bipartite matching has integral LP solutions.

Contest tip. Many max-flow / min-cut theorems, König's theorem, and min-cost matching results are consequences of LP duality. Understanding duality often reveals why a greedy or flow-based solution is correct.

LP Tricks in CP

The real power of LP in competitive programming is modelling — recognising that a problem can be expressed as an LP, which then either admits a known efficient algorithm or can be solved directly with simplex.

Max-Flow as LP

Given a flow network with source s, sink t, and edge capacities ce, the max-flow problem is:

Maximise Σe out of s fe Subject to 0 ≤ fe ≤ ce for each edge e Σe into v fe = Σe out of v fe for each v ≠ s, t

The constraint matrix is the incidence matrix of the graph, which is totally unimodular (TU). By the integrality theorem, when the capacities are integers, the LP optimal solution is automatically integral — so solving the LP gives the same answer as a combinatorial max-flow algorithm.

Bipartite Matching as LP

Maximum bipartite matching can be formulated as an LP on the bipartite graph's incidence matrix (also TU). The LP relaxation has integral optimal solutions, which is why König's theorem works:

max matching = min vertex cover (in bipartite graphs)

This is exactly the LP duality relationship: the primal gives matching, the dual gives vertex cover.

Assignment Problem

Given an n × n cost matrix, assign each worker to exactly one job to minimise total cost. This is a min-cost LP whose constraint matrix is TU, so the LP solution is integral. While the Hungarian algorithm solves it in O(n3), formulating it as LP clarifies why the algorithm works and lets you handle generalised versions.

LP Relaxation for Approximation

For NP-hard problems (set cover, vertex cover on general graphs), the LP relaxation gives a fractional optimum that is a lower (or upper) bound on the integer optimum. Rounding the fractional solution gives approximation algorithms. In CP, this is occasionally useful for proving bounds or for problems where the LP relaxation happens to be integral. 

// Example: modelling min vertex cover on bipartite graph as LP dual
// Primal = max matching, Dual = min vertex cover
// For bipartite graph with adjacency:
//   maximise sum of x_e for edges e
//   s.t. sum of x_e incident to v <= 1  for each vertex v
//        x_e >= 0
// Dual:
//   minimise sum of y_v for vertices v
//   s.t. y_u + y_v >= 1  for each edge (u,v)
//        y_v >= 0
// Strong duality + TU => both have integral optima

Handling Special Cases

A robust simplex implementation must handle several edge cases that arise in contest problems.

Infeasible LPs

An LP is infeasible when no point satisfies all constraints simultaneously. In the simplex method, this is detected when Phase 1 cannot find a basic feasible solution — the artificial objective remains positive. In the implementation above, Phase 1 detects infeasibility when a row with negative RHS has no suitable pivot column.

Unbounded LPs

An LP is unbounded when the objective can be made arbitrarily large. In simplex, this occurs when an entering variable has no positive coefficient in any constraint row — you can increase it indefinitely without violating any constraint. The code detects this and returns −1.

Degeneracy

A BFS is degenerate when one or more basic variables have value zero. This means a pivot may not improve the objective (the step size is zero). Degeneracy itself is harmless, but it can lead to cycling.

Cycling Prevention

Three common approaches to prevent cycling:

  • Bland's rule: Always pick the smallest-index entering and leaving variables. Simple, reliable, implemented in our code above.
  • Lexicographic method: Break ties in the minimum ratio test using lexicographic comparison of tableau rows. Slightly more complex but also guarantees finite termination.
  • Perturbation: Add tiny distinct perturbations to the RHS vector b to make all BFS non-degenerate. Simple in theory, needs care with floating-point precision.
// Bland's rule implementation (used in our solver)
// Entering variable: smallest index j with tab[m][j] < -EPS
int entering = -1;
for (int j = 0; j < n + m; j++)
    if (tab[m][j] < -EPS) { entering = j; break; }

// Leaving variable: min ratio, ties broken by smallest basis index
int leaving = -1;
for (int i = 0; i < m; i++) {
    if (tab[i][entering] < EPS) continue;
    if (leaving == -1
        || tab[i][n+m] / tab[i][entering]
         < tab[leaving][n+m] / tab[leaving][entering] - EPS
        || (fabs(tab[i][n+m] / tab[i][entering]
              - tab[leaving][n+m] / tab[leaving][entering]) < EPS
            && basis[i] < basis[leaving]))
        leaving = i;
}
Numerical stability. Contest simplex implementations are notoriously sensitive to floating-point errors. Use long double if available, keep EPS small but not too small (10−9 is typical), and avoid catastrophic cancellation by using scaled pivoting if needed.

When to Use Simplex in Contests

The simplex method has exponential worst-case time complexity (there exist pathological inputs requiring 2n pivots), but in practice it is extremely fast — typically O(m) pivots for m constraints. Understanding when to reach for simplex versus a specialised algorithm is crucial.

Time Complexity

AlgorithmWorst CasePracticalNotes
SimplexO(2n)Very fastEmpirically near-linear
Interior PointO(n3.5)FastPolynomial, harder to implement
EllipsoidO(n4)SlowTheoretical importance only

Problem Indicators

Consider using simplex when you see:

  • Optimise a linear function with linear constraints — direct LP.
  • Small constraint count — m ≤ ~500 and n ≤ ~500 is very comfortable for contest simplex.
  • Fractional solutions allowed — if the problem asks for real-valued answers or the constraint matrix is TU.
  • No known combinatorial algorithm — if the problem doesn't reduce to flow, matching, or shortest paths, LP might be the intended approach.
  • "Minimise maximum" or "maximise minimum" — these often reduce to LP by introducing an auxiliary variable.

When NOT to Use Simplex

  • Integer constraints required: LP relaxation may give fractional answers; you'd need ILP which is NP-hard in general.
  • Large n and m: Beyond ~2000 variables/constraints, contest time limits are tight. Use specialised algorithms (network simplex for flow LPs).
  • Known better algorithm: If the problem reduces to max-flow, shortest path, or bipartite matching, use those — they're faster and have better worst-case bounds.
Rule of thumb: If m, n ≤ 300–500 and the problem is clearly an LP, simplex will run in time. Above that, look for problem structure (network, TU matrix) that allows specialised solvers.

Practice Problems

  1. Basic LP: Solve a small LP with 2–3 variables manually using the tableau method, then verify with the C++ implementation. Try the example in the code above.
  2. Diet Problem: Given nutritional constraints and food costs, find the cheapest diet meeting all requirements. Classic introductory LP.
  3. Max-Flow via LP: Formulate a max-flow instance as an LP and solve with simplex. Verify the answer matches a Ford-Fulkerson / Dinic solution.
  4. Assignment Problem: Solve a small assignment problem (n ≤ 8) with simplex and verify integrality of the solution.
  5. Codeforces 375C — "Circling Round Dancing": Practice reducing a constraint optimisation to LP form.
  6. Unbounded Detection: Construct an LP that is unbounded and verify your solver reports it correctly. Example: maximise x subject to x ≥ 0 (no upper bound).
  7. Infeasibility Detection: Construct an LP with contradictory constraints (e.g., x ≤ 1 and x ≥ 2) and verify your solver detects infeasibility.
  8. Dual Construction: For a given primal LP, write out the dual by hand. Solve both and verify strong duality (equal optimal values).
  9. LP Relaxation: Take a small integer programming problem (e.g., 0-1 knapsack) and solve the LP relaxation. Compare the fractional bound with the true integer optimum.
