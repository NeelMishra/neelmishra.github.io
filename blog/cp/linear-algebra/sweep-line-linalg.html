    Sweep Line & Linear Algebra

    July 2026 · 22 min read

    Sweep line is a classic algorithmic paradigm: process events left-to-right (or top-to-bottom) and maintain some state that evolves incrementally. Linear algebra provides a different toolkit: vector spaces, bases, rank, and Gaussian elimination. At first glance, these two worlds seem unrelated. But a surprisingly rich family of competitive programming problems lives at their intersection — problems where we sweep through data while maintaining a linear-algebraic structure such as a basis or a row echelon form.

    This post develops the key ideas: how to perform Gaussian elimination incrementally as new vectors arrive during a sweep, how to track rank online, and how to solve two classic problems — maximum XOR subarray and range linear independence queries — using these techniques.

    When Sweep Line Meets Linear Algebra

    The sweep line technique processes a sorted sequence of events and maintains a state that is updated at each event. Typical examples include computing the area of a union of rectangles or finding the closest pair of points. The key idea is that the state changes locally at each event, so we never need to recompute from scratch.

    Linear algebra enters when the "state" we maintain is a vector space basis. Consider these motivating scenarios:

    • XOR subarray problems: Given an array, find the subarray with maximum XOR. As we sweep from left to right, we insert prefix XOR values into a basis one-by-one, querying for the maximum XOR achievable with the current prefix.
    • Interval independence queries: Given vectors associated with positions, answer queries of the form "are the vectors in range [l, r] linearly independent?" — efficiently handled by sweeping with an evolving basis.
    • Constraint satisfaction: Linear constraints arrive in sorted order. We maintain a row echelon form and check consistency or count degrees of freedom at each step.

    The common thread is that we need an online version of Gaussian elimination — one that can incorporate a new vector in O(B) time, where B is the dimension of the space (often 30 or 60 for XOR problems, or the number of variables in a linear system).

    Incremental Gaussian Elimination

    Standard Gaussian elimination takes an m × n matrix and reduces it to row echelon form in O(m · n · min(m, n)) time. But in sweep line problems, vectors arrive one at a time. We need an incremental (online) variant.

    The idea is simple: maintain a set of basis vectors in row echelon form. When a new vector v arrives, try to reduce it using existing basis vectors. If v reduces to zero, it is linearly dependent on the current basis. If it does not reduce to zero, it extends the basis.

    Over GF(2) — The XOR Basis

    The most common competitive programming setting is GF(2), where addition is XOR and the "vectors" are integers. Each bit position is a coordinate. The basis is maintained as an array basis[b] where basis[b] is the unique basis vector whose highest set bit is bit b (or 0 if no such vector exists). 

    struct XorBasis {
    long long basis[60];
    int sz = 0; // number of basis vectors

    XorBasis() { memset(basis, 0, sizeof(basis)); }

    // Try to insert v into the basis.
    // Returns true if v was linearly independent (basis grew).
    bool insert(long long v) {
        for (int i = 59; i >= 0; i--) {
            if (!(v >> i & 1)) continue;
            if (!basis[i]) {
                basis[i] = v;
                sz++;
                return true;
            }
            v ^= basis[i];
        }
        return false; // v is in the span of existing basis
    }

    // Query the maximum XOR achievable from the basis.
    long long queryMax() const {
        long long res = 0;
        for (int i = 59; i >= 0; i--) {
            res = max(res, res ^ basis[i]);
        }
        return res;
    }

    // Query the maximum XOR of val with some subset of the basis.
    long long queryMax(long long val) const {
        for (int i = 59; i >= 0; i--) {
            val = max(val, val ^ basis[i]);
        }
        return val;
    }
};

    Each insert call takes O(B) time where B = 60 (the number of bit positions). The basis never holds more than B vectors, so the space is O(B).

    Over General Fields

    Over ℝ or ℤₚ the same idea works. Maintain an array of basis vectors in echelon form. To insert a new vector, reduce it by subtracting multiples of existing basis vectors to zero out pivot positions. If a non-zero entry remains, it becomes a new pivot. This costs O(rank · dim) per insertion.

    Online Rank Maintenance

    A direct application of incremental Gaussian elimination is online rank maintenance. As we sweep through events (e.g., reading array elements left to right), we insert each new vector into the basis and track the current rank — the number of linearly independent vectors seen so far.

    Key observations:

    • Rank only increases. Each insertion either increases the rank by 1 (vector was independent) or leaves it unchanged (dependent). The rank never decreases during a left-to-right sweep.
    • Independence detection is free. The return value of insert() already tells us whether the new vector was independent.
    • Rank at each prefix can be recorded in an array for later use in answering queries. 
    // Track rank as we process n vectors
int n;
long long a[200005]; // input values for XOR basis example
int prefix_rank[200005];

void computePrefixRanks() {
    XorBasis B;
    for (int i = 1; i <= n; i++) {
        B.insert(a[i]);
        prefix_rank[i] = B.sz;
    }
}

    If we need to know the rank of the set {a[1], …, a[i]} for every i, one sweep suffices: O(n · B) total time.

    When Do We Need Deletions?

    The incremental basis supports insertions but not deletions. If the sweep line problem requires removing old vectors (e.g., when events expire as we move the sweep line forward), we need a different approach:

    • Offline processing: Sort queries and use a technique like the one in the "Range Linear Independence" problem below, where we cleverly choose which vectors to keep in the basis so that deletions are never needed.
    • Segment tree of bases: Build a segment tree where each node stores a basis for its range. Merging two bases takes O(B²) time. This supports arbitrary range queries but with higher constant factors.
    • Divide and conquer: Process the events using a D&C scheme that avoids explicit deletion.

    Sweep Line Over Intervals with Linear Constraints

    A general pattern in competitive programming: we have n constraints that are "active" during intervals [lᵢ, rᵢ], and we sweep a vertical line from left to right. At each position x, the active constraints form a linear system. We want to answer questions such as:

    • At each position, what is the rank of the active system?
    • Is the active system consistent?
    • How many degrees of freedom exist?

    The approach uses an event-based sweep: at position lᵢ we "insert" constraint i, and at position rᵢ + 1 we "remove" it. The challenge is that standard Gaussian elimination does not support removal well. Two strategies handle this:

    Strategy 1: Segment Tree of Bases

    Build a segment tree over positions. For each constraint active on [l, r], add it to all O(log n) nodes that cover [l, r]. Then, for a query at position x, walk from the root to the leaf for x, merging the bases at each node along the path. The merge of two bases of dimension B costs O(B²), and the path has O(log n) nodes, giving O(B² log n) per query.

    Strategy 2: Offline + Smart Replacement

    If we process queries offline (sorted by right endpoint), we can maintain a single basis where each basis vector remembers the rightmost starting index among the original vectors used to form it. When inserting a new vector whose interval starts at index l, we can perform a modified insertion: if the new vector can replace a basis vector with a smaller starting index, we swap them. This ensures the basis always contains vectors with the latest possible starting indices, which is exactly what we need for range queries — a technique we will detail in the Range Linear Independence problem below.

    Problem: Maximum XOR Subarray

    Statement: Given an array a[1..n] of non-negative integers, find a contiguous subarray a[l..r] (l ≤ r) whose XOR a[l] ⊕ a[l+1] ⊕ … ⊕ a[r] is maximized.

    Key insight: Let prefix[i] = a[1] ⊕ a[2] ⊕ … ⊕ a[i] and prefix[0] = 0. Then the XOR of subarray [l, r] equals prefix[r] ⊕ prefix[l−1]. So we want to maximize prefix[r] ⊕ prefix[j] over all 0 ≤ j < r, for the best choice of r.

    Sweep Line + XOR Basis Approach

    Sweep r from 1 to n. Before processing r, the basis contains (some representation of) {prefix[0], prefix[1], …, prefix[r−1]}. We query the basis for the maximum XOR with prefix[r], then insert prefix[r] into the basis. 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    vector<long long> a(n);
    for (auto &x : a) cin >> x;

    // Compute prefix XOR
    vector<long long> pre(n + 1);
    pre[0] = 0;
    for (int i = 0; i < n; i++) {
        pre[i + 1] = pre[i] ^ a[i];
    }

    // Sweep: maintain XOR basis of prefix[0..r-1], query with prefix[r]
    long long basis[60] = {};
    long long ans = 0;

    // Insert prefix[0] = 0 into basis (it contributes nothing, but
    // it represents the option of taking subarray [1..r]).
    // Actually 0 never enters a non-trivial basis, so we just start.

    for (int r = 1; r <= n; r++) {
        // Query: best XOR of prefix[r] with span of basis
        long long cur = pre[r];
        for (int i = 59; i >= 0; i--) {
            cur = max(cur, cur ^ basis[i]);
        }
        ans = max(ans, cur);

        // Insert prefix[r] into basis for future queries
        long long v = pre[r];
        for (int i = 59; i >= 0; i--) {
            if (!(v >> i & 1)) continue;
            if (!basis[i]) { basis[i] = v; break; }
            v ^= basis[i];
        }
    }

    cout << ans << "\n";
    return 0;
}

    Complexity: O(n · B) where B = 60. This is effectively O(n) for all practical purposes and handles n up to 10⁶ with ease.

    Why This Works

    The XOR basis represents the span of all prefix XOR values seen so far. The query queryMax(prefix[r]) finds the value in this span that maximizes the XOR with prefix[r]. Since prefix[r] ⊕ prefix[j] gives the XOR of subarray [j+1, r], and the span of the basis includes all achievable prefix[j] values, we get the optimal subarray ending at or before position r.

    Note a subtlety: the basis does not literally store every prefix value, but it stores a basis whose span includes all of them. This is sufficient because queryMax optimizes over the entire span.

    Problem: Range Linear Independence

    Statement: Given n integers a[1..n] and q queries, each query gives a range [l, r]. Determine the rank of the set {a[l], a[l+1], …, a[r]} over GF(2) (equivalently, the size of the XOR basis of these values). Some variants ask whether a target value T is achievable as a XOR of a subset within [l, r].

    Offline Sweep with Latest-Index Basis

    The key technique: sweep from left to right, maintaining a basis where each basis vector is tagged with the latest (rightmost) index it can be attributed to. When inserting a[i], if it conflicts with a basis vector tagged with index j < i, we swap: the new vector (tagged i) takes the basis slot, and the old one continues the reduction. This greedy swap ensures that the basis always contains vectors with the most recent indices possible.

    After sweeping to position r, the basis vectors with tag ≥ l exactly span the subset {a[l], …, a[r]}. So the rank for query [l, r] is the number of basis vectors with tag ≥ l. 

    #include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;
    vector<long long> a(n + 1);
    for (int i = 1; i <= n; i++) cin >> a[i];

    // For each query [l, r], store it at position r
    struct Query { int l, id; };
    vector<vector<Query>> queries(n + 1);
    vector<int> ans(q);
    for (int i = 0; i < q; i++) {
        int l, r;
        cin >> l >> r;
        queries[r].push_back({l, i});
    }

    // basis[b] = value with highest bit b
    // tag[b]   = the latest index this basis vector can be attributed to
    long long basis[60] = {};
    int tag[60] = {};

    for (int r = 1; r <= n; r++) {
        // Insert a[r] with tag r, using greedy swap
        long long v = a[r];
        int cur_tag = r;
        for (int i = 59; i >= 0; i--) {
            if (!(v >> i & 1)) continue;
            if (!basis[i]) {
                basis[i] = v;
                tag[i] = cur_tag;
                break;
            }
            // If current tag is more recent, swap
            if (cur_tag > tag[i]) {
                swap(v, basis[i]);
                swap(cur_tag, tag[i]);
            }
            v ^= basis[i];
        }

        // Answer queries with right endpoint = r
        for (auto &[l, id] : queries[r]) {
            int rank = 0;
            for (int i = 59; i >= 0; i--) {
                if (basis[i] && tag[i] >= l) rank++;
            }
            ans[id] = rank;
        }
    }

    for (int i = 0; i < q; i++) {
        cout << ans[i] << "\n";
    }
    return 0;
}

    Complexity Analysis

    The sweep processes each element in O(B) time. Each query is answered in O(B) by scanning the 60-element basis array. Total: O((n + q) · B), which is very fast.

    Why the Greedy Swap Works

    When we swap a basis vector tagged j with the incoming vector tagged i > j, we are replacing a vector that represents an older element with one that represents a newer element — without changing the overall span. The displaced old vector continues the reduction and may end up in a different basis slot (also with tag j) or be eliminated entirely. The invariant is:

    • The span of basis vectors equals the span of {a[1], …, a[r]}.
    • Each basis slot has the maximum possible tag — no rearrangement of the same vectors could produce a basis where any slot has a larger tag.

    This means that for any query [l, r], the number of basis vectors with tag ≥ l is exactly the rank of {a[l], …, a[r]}. Intuitively, vectors tagged < l are "too old" for the query range, and the greedy swap ensures we never use an old vector when a newer one could serve in that slot.

    Implementation Tips

    Choosing the Right Basis Size

    • XOR basis: Use B = 30 if values fit in 32-bit integers, B = 60 for 64-bit. Using long long with B = 60 is the safe default.
    • General linear algebra: B equals the number of variables. For systems over ℤₚ, use modular arithmetic carefully — always reduce mod p after every arithmetic operation.

    Avoiding Common Pitfalls

    • Forgetting prefix[0] = 0: In the maximum XOR subarray problem, prefix[0] = 0 is part of the candidate set. Since 0 never enters a non-trivial basis, this is handled naturally — but be explicit about it in your reasoning.
    • Off-by-one in tag comparison: When answering range queries, ensure the comparison is tag[i] >= l (not > l). The element at position l is included in the range.
    • Not swapping during insertion: The greedy swap is essential for the range query technique. Without it, basis vectors may have suboptimal tags, leading to wrong answers.
    • Integer overflow: When working over ℤₚ with large primes, multiplications can overflow 64-bit integers. Use __int128 or reduce intermediate products.

    Combining with Other Data Structures

    The sweep line + basis pattern composes well with other techniques:

    • Persistent basis: Store the basis at each sweep step using a persistent array, enabling online queries without sorting by right endpoint.
    • Mo's algorithm with basis: Adapt Mo's to maintain a basis with insertions on both ends; deletions use rollback via a stack of operations.

    Practice Problems

    • Codeforces 895C — Square Subsets: Count subsets whose product is a perfect square — reducible to XOR-sum-zero counting in exponent-parity space.
    • SPOJ XMAX — XOR Maximization: Find the maximum XOR of a subset. Direct XOR basis application.
    • Codeforces 1100F — Ivan and Burgers: Offline range maximum XOR queries — the exact range basis technique with greedy swaps.
    • AtCoder ABC 283 Ex — Constraints: Maintaining linear constraints during an interval sweep.
    • BZOJ 2115 — XOR: Max-XOR path in a weighted graph. Combines XOR basis with graph traversal.
