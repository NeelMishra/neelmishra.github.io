Misère Games — Misère Nim & Sprague-Grundy Under Misère Play

June 2026 · 28 min read

Normal vs Misère Play

Every combinatorial game must specify what happens when a player has no moves left. The two standard conventions are normal play and misère play.

Normal Play

The player who makes the last move wins. Equivalently, the first player unable to move loses.

This is the convention used in standard Nim, standard Sprague-Grundy theory, and the vast majority of competitive programming game problems.

Misère Play

The player who makes the last move loses. Equivalently, the first player unable to move wins.

This "opposite" convention appears in misère Nim, last-player-loses variants, and certain real-world games.

At first glance, misère seems like a trivial modification — just flip the winner at the end. For Nim specifically, the adjustment is elegant (a single extra condition). But for general combinatorial games, misère analysis is vastly harder than normal play.

Key insight: Under normal play, every impartial game is equivalent to a single Nim heap (Sprague-Grundy theorem), and sums of games reduce to XOR of Grundy values. Under misère play, no such clean reduction exists in general. The misère quotient of a game can be exponentially larger than its normal-play Grundy value.

Formally, in a two-player impartial game with positions G = (P, \mathcal{M}) where P is the set of positions and \mathcal{M}(p) gives the set of positions reachable from p:

  • Terminal position: p with \mathcal{M}(p) = \emptyset
  • Normal play: Terminal position is a loss for the player to move (previous player wins)
  • Misère play: Terminal position is a win for the player to move (previous player loses)

Why is misère harder? Under normal play, the Sprague-Grundy theorem gives us an equivalence: every game position has a unique Grundy value (nimber), and the Grundy value of a sum of games is the XOR (nim-sum) of the components' Grundy values. This clean algebraic structure breaks down under misère play. Two positions with the same Grundy value under normal play can behave very differently under misère play.

Misère Nim

Despite the general difficulty of misère games, Misère Nim has a clean characterization, first proven by Charles Bouton in 1901.

Theorem (Misère Nim)

Consider a Nim game with n heaps of sizes a_1, a_2, \ldots, a_n, played under misère convention. Let S = a_1 \oplus a_2 \oplus \cdots \oplus a_n (nim-sum / XOR).

The first player wins if and only if:

  • S \neq 0 and at least one heap has size \ge 2, or
  • S = 0 and all heaps have size \le 1 (i.e., an even number of size-1 heaps).

Equivalently, the first player loses (second player wins) if and only if:

  • S = 0 and at least one heap has size \ge 2, or
  • S \neq 0 and all heaps have size \le 1 (i.e., an odd number of size-1 heaps).

Notice that when all heaps have size \le 1, misère Nim is trivial: with k heaps of size 1, the players alternate taking one stone each. The last player to move loses, so the first player loses when k is odd (XOR = 1) and wins when k is even (XOR = 0). This is the opposite of normal Nim.

When at least one heap has size \ge 2, the condition is the same as normal Nim: first player wins iff XOR ≠ 0.

Proof of the Misère Nim Theorem

We prove the theorem by providing a winning strategy for the appropriate player. Define the configuration as the multiset of heap sizes, and let:

  • S = a_1 \oplus a_2 \oplus \cdots \oplus a_n
  • m = \max(a_1, a_2, \ldots, a_n) (size of the largest heap)

Case 1: All heaps have size ≤ 1.

This is a simple parity game. There are k heaps of size 1 (the rest are 0, which are irrelevant). Players alternate removing one stone from a size-1 heap. The game lasts exactly k moves, and the player making move k (the last move) loses. So:

  • If k is even, the first player wins (second player is forced to make the last move). Note S = k \bmod 2 = 0.
  • If k is odd, the first player loses. Note S = 1.

So with all heaps ≤ 1: first player wins iff S = 0. ✓

Case 2: At least one heap has size ≥ 2 and S \neq 0.

We show the first player can win. The strategy mirrors normal Nim with a twist at the end.

As in normal Nim, when S \neq 0, there exists a heap a_i with a_i \oplus S < a_i (take the heap whose highest bit matches the highest bit of S). The first player can reduce a_i to a_i \oplus S, making the new XOR equal to 0.

Key refinement: The first player plays the normal Nim strategy (maintaining XOR = 0 for the opponent) until reaching a position where all remaining heaps have size ≤ 1. At that critical moment, the player ensures an odd number of size-1 heaps remain (leaving the opponent in the losing all-heaps-≤-1 position with odd parity).

Why can the first player always arrange this? When S \neq 0 and some heap has size ≥ 2, the first player can always make a move to achieve S = 0. Critically, when exactly one heap has size ≥ 2 (say heap a_j \ge 2) and the rest are 0 or 1, with current S \neq 0, the first player can reduce a_j to either 0 or 1 to leave an odd number of 1s. If there are currently k heaps of size 1 among the others:

  • If k is even, reduce a_j to 1, leaving k + 1 (odd) heaps of size 1.
  • If k is odd, reduce a_j to 0, leaving k (odd) heaps of size 1.

In both cases, the opponent faces an odd number of size-1 heaps and loses. The first player can always reach this endgame because the normal-Nim strategy of maintaining XOR = 0 guarantees the opponent never has the last "big" move.

Case 3: At least one heap has size ≥ 2 and S = 0.

We show the first player loses (the second player can win). This mirrors the normal Nim argument.

When S = 0, any move by the first player changes some a_i to a_i' < a_i, making S' = S \oplus a_i \oplus a_i' = a_i \oplus a_i' \neq 0. Since m \ge 2 initially and at least two heaps must be non-zero (otherwise a single non-zero heap gives S = a_i \neq 0, contradicting S = 0), after the first player's move we are in a Case 2 situation from the second player's perspective. The second player then applies the Case 2 strategy and wins.

Together, Cases 1–3 establish the theorem completely.

∎
Summary formula. Let \text{allSmall} be true when every heap has size ≤ 1. Then the first player wins misère Nim iff:

(S \neq 0 \;\wedge\; \neg\text{allSmall}) \;\;\lor\;\; (S = 0 \;\wedge\; \text{allSmall})

Observe how close this is to normal Nim: the condition differs only in the all-heaps-≤-1 edge case. For large heap sizes, normal and misère Nim have the same winning condition.

Why Misère Is Hard

The elegance of misère Nim might suggest that misère analysis is always a minor tweak. This is profoundly wrong.

The Normal-Play Miracle

Under normal play, the Sprague-Grundy theorem gives us:

  1. Every position has a Grundy value \mathcal{G}(p) \in \{0, 1, 2, \ldots\}.
  2. A position is losing iff \mathcal{G}(p) = 0.
  3. For a sum of games: \mathcal{G}(G_1 + G_2) = \mathcal{G}(G_1) \oplus \mathcal{G}(G_2).

This means we can analyze each component independently, then combine with XOR.

The Misère Catastrophe

Under misère play, properties 1 and 3 break down for general games:

  • Grundy values don't determine the winner. Two positions with the same normal-play Grundy value can have different misère outcomes. For example, in Nim, the single heap *1 (one stone) loses under misère (you must take it and lose), but under normal play \mathcal{G}(*1) = 1, a winning position.
  • XOR doesn't work for sums. Consider the game where each component can be either "take one stone" or "take from a Nim heap." In misère, the sum's outcome might not be determined by XOR-ing the individual Grundy values.
  • Exponential blowup. The misère quotient — the algebraic structure needed to solve misère sums — can be exponentially larger than the corresponding normal-play structure. For the game of Dawson's Chess, the normal-play period is 34, but the misère quotient has 226 elements.
Counterexample. Consider a game with three components, each being a position with normal-play Grundy value 1. Under normal play, 1 \oplus 1 \oplus 1 = 1, so the first player wins. Under misère play with the same Grundy values, the answer might be different depending on the specific structure of each component — knowing only the Grundy value is insufficient.

What Does Work?

For misère analysis of general games, we need richer structures: genus values (normal/misère Grundy pairs), misère quotients (commutative monoids for misère equivalence classes), and the tame/wild classification.

The most common competitive programming misère game is Misère Nim, which has the clean solution above. But understanding the theory helps with non-standard variants.

Misère Sprague-Grundy

The standard Sprague-Grundy theory assigns each position a single nimber (Grundy value). For misère analysis, we need a richer invariant: the genus.

Genus Values

The genus of a game position G is a pair:

\text{genus}(G) = \mathcal{G}^+(G) \;|\; \mathcal{G}^-(G)

where:

  • \mathcal{G}^+(G) is the normal-play Grundy value (standard nimber)
  • \mathcal{G}^-(G) is the misère Grundy value, computed by the same mex rule but with terminal positions assigned value 1 (instead of 0)
Misère Grundy value. Computed identically to the normal Grundy value (mex over successors, terminal = 0). The interpretation differs: under misère, \mathcal{G}^- = 0 is a winning position (being unable to move means you win).

Let us be precise. The genus is formally written as:

\text{genus}(G) = g^+ \,\hat{}\, g^-

where g^+ is the normal Grundy value and g^- is the misère Grundy value. The genus of a Nim heap of size n:

Heap size n Normal \mathcal{G}^+ Misère \mathcal{G}^- Genus g^+\hat{\;}g^-
0000\hat{\;}0
1111\hat{\;}1
2222\hat{\;}2
3333\hat{\;}3
n \ge 1nnn\hat{\;}n

For Nim heaps, the normal and misère Grundy values are identical! This is why Nim is special — a Nim heap is a tame game (defined below).

Genus Addition

For sums of games, we'd like "XOR the genus values," but this doesn't work in general. For tame games however (where g^+ = g^- for all non-terminal positions), we get a workable theory:

\text{genus}(G_1 + G_2) \text{ depends on } \text{genus}(G_1) \text{ and } \text{genus}(G_2)

The normal component combines by XOR as usual: \mathcal{G}^+(G_1 + G_2) = \mathcal{G}^+(G_1) \oplus \mathcal{G}^+(G_2). For the misère component, the situation is more nuanced, leading to the tame vs. wild classification.

Misère Quotients

The fully general theory uses misère quotients (Plambeck & Siegel) — commutative monoids capturing misère equivalence classes. Key results:

  • Every impartial game ruleset has a finite misère quotient.
  • Nim's misère quotient is \mathbb{N}_0 \cup \{\infty\} — just one element larger than normal play.
  • Dawson's Chess has a misère quotient with 226 elements.
  • Some games have misère quotients of size > 106.

Tame Games

A game is tame if every position except the terminal position (the zero game) has \mathcal{G}^+ = \mathcal{G}^-. In other words, the normal and misère Grundy values agree everywhere except at the endgame.

Theorem (Tame Game Misère Sum)

If G_1, G_2, \ldots, G_k are all tame games and S = \mathcal{G}^+(G_1) \oplus \cdots \oplus \mathcal{G}^+(G_k), then the sum G_1 + G_2 + \cdots + G_k under misère play:

  • If at least one G_i has Grundy value ≥ 2: the first player wins iff S \neq 0 (same as normal play).
  • If all G_i have Grundy value ≤ 1: the first player wins iff S = 0 (opposite of normal play).

This is exactly the misère Nim theorem! This is not a coincidence — Nim heaps are tame games, and the tame game theorem generalizes misère Nim to any sum of tame components.

Examples of Tame Games

  • Nim — each heap is trivially tame since \mathcal{G}^+(n) = \mathcal{G}^-(n) = n for all n \ge 1.
  • Wythoff's game — tame, so misère Wythoff follows the same pattern.
  • Turning Turtles — many "Turning" games are tame.
  • Any game with Grundy values {0, 1} only — automatically tame (trivially, since all non-zero values are 1).

Wild Games

A game is wild if it is not tame — some non-terminal position has \mathcal{G}^+ \neq \mathcal{G}^-. Wild games require the full misère quotient machinery for analysis.

Examples of wild games include:

  • Dawson's Chess (Dawson's Kayles) — misère quotient has 226 elements.
  • Treblecross (0.007) — wild with complex misère structure.
  • Grundy's Game — split a heap into two unequal heaps; wild in misère.
Competitive programming tip: If a problem asks you to solve a misère game, first check: is it Nim (or a sum of tame games)? If yes, use the misère Nim theorem. If the game is wild, you likely need to precompute the misère Grundy values via brute-force DP and look for patterns (periods, special structure).

Detecting Tameness

To verify tameness, compute both \mathcal{G}^+ and \mathcal{G}^- for all reachable positions and check they agree except at terminals. For games with periodic Grundy values, verify within one period.

C++ Implementation

Misère Nim Solver

The simplest and most commonly needed implementation — solving misère Nim: 

#include <bits/stdc++.h>
using namespace std;

// Returns true if the first player wins misère Nim.
// Heaps: vector of heap sizes.
bool misereNimFirstWins(const vector<int>& heaps) {
    int xorSum = 0;
    bool allSmall = true; // all heaps <= 1?

    for (int h : heaps) {
        xorSum ^= h;
        if (h >= 2) allSmall = false;
    }

    if (allSmall) {
        // Parity game: first player wins iff even number of 1s
        return xorSum == 0;
    } else {
        // Same as normal Nim
        return xorSum != 0;
    }
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t;
    cin >> t;
    while (t--) {
        int n;
        cin >> n;
        vector<int> heaps(n);
        for (auto& h : heaps) cin >> h;

        cout << (misereNimFirstWins(heaps) ? "First" : "Second") << "\n";
    }
    return 0;
}

Computing Genus Values & Detecting Tameness

For a general impartial game given as a DAG, compute both Grundy values and check tameness: 

#include <bits/stdc++.h>
using namespace std;

struct Genus { int gPlus, gMinus; };

vector<Genus> computeGenus(const vector<vector<int>>& adj) {
    int n = adj.size();
    vector<Genus> g(n);
    for (int v = n - 1; v >= 0; --v) {
        if (adj[v].empty()) { g[v] = {0, 0}; continue; }
        set<int> rN, rM;
        for (int u : adj[v]) {
            rN.insert(g[u].gPlus);
            rM.insert(g[u].gMinus);
        }
        int mexN = 0, mexM = 0;
        while (rN.count(mexN)) ++mexN;
        while (rM.count(mexM)) ++mexM;
        g[v] = {mexN, mexM};
    }
    return g;
}

bool isTame(const vector<vector<int>>& adj, const vector<Genus>& g) {
    for (int v = 0; v < (int)adj.size(); ++v)
        if (!adj[v].empty() && g[v].gPlus != g[v].gMinus)
            return false;
    return true;
}

Misère Sum of Tame Games

When you've confirmed all components are tame, solving the misère sum: 

// Solve misère sum of tame game components.
bool misereTameSumFirstWins(const vector<int>& grundy) {
    int xorSum = 0;
    bool allSmall = true;
    for (int g : grundy) {
        xorSum ^= g;
        if (g >= 2) allSmall = false;
    }
    return allSmall ? (xorSum == 0) : (xorSum != 0);
}
Complexity. Misère Nim is O(n) where n is the number of heaps. Computing genus values for a game DAG is O(V + E) where V is the number of positions and E is the number of edges. The tameness check is an additional O(V).

Handling Non-Tame (Wild) Games

For wild games in competitive programming, the typical approach is:

  1. Compute misère Grundy values by brute-force DP for small cases.
  2. Look for periodicity or patterns in the sequence.
  3. If found, use the pattern to answer large inputs.
  4. If not, the problem likely has additional structure to exploit.
// Brute-force Grundy for a subtraction game (works for both conventions).
vector<int> computeGrundy(int maxN, const vector<int>& moves) {
    vector<int> g(maxN + 1, 0);
    for (int n = 1; n <= maxN; ++n) {
        set<int> reachable;
        for (int m : moves)
            if (m <= n) reachable.insert(g[n - m]);
        int mex = 0;
        while (reachable.count(mex)) ++mex;
        g[n] = mex;
    }
    return g;
}

// Detect period in a Grundy sequence.
pair<int,int> detectPeriod(const vector<int>& g) {
    int n = g.size();
    for (int p = 1; p <= n / 3; ++p) {
        for (int start = 0; start <= n / 2; ++start) {
            bool ok = true;
            for (int i = start; i + p < n; ++i)
                if (g[i] != g[i + p]) { ok = false; break; }
            if (ok) return {start, p};
        }
    }
    return {-1, -1};
}
