Strategy Stealing Argument

June 2026 · 20 min read

The Strategy Stealing Argument

The strategy stealing argument is one of the most elegant non-constructive proof techniques in combinatorial game theory. The core idea is deceptively simple: if the second player had a winning strategy, the first player could "steal" that strategy by making an arbitrary opening move and then following the second player's supposed winning strategy for the rest of the game.

This argument applies to a broad class of two-player, perfect-information, finite games where having an extra piece on the board (or an extra move already played) is never a disadvantage. The result is a powerful existential proof that the first player has a winning strategy — without revealing what that strategy actually is.

Formal Setup

Consider a two-player game $\mathcal{G}$ satisfying:

  1. Perfect information: Both players see the entire game state.
  2. Finite: The game terminates after a finite number of moves.
  3. Deterministic outcome: The game ends in a win for one player or a draw (no chance elements).
  4. No passing: A player cannot voluntarily skip a turn.

By Zermelo's theorem, at least one player has a winning strategy, or both players can force a draw. The strategy stealing argument narrows this further for games with the monotonicity property.

Monotonicity Property. A game has the monotonicity property if having an extra stone/piece on the board is never a disadvantage. Formally, if position $P$ is winning for a player, then any position $P'$ obtained by adding one of that player's pieces to $P$ is also winning (or at least not losing).

The Argument (Informal)

Suppose, for contradiction, that the second player (Player 2) has a winning strategy $S_2$. Then:

  1. Player 1 makes an arbitrary first move $m_0$.
  2. Player 1 now pretends to be Player 2 and follows $S_2$.
  3. Whenever $S_2$ says "play at position $p$," if $p$ is already occupied by $m_0$, Player 1 makes a new arbitrary move $m_0'$ instead (and tracks this as the new "extra" move).
  4. Since the extra move is never a disadvantage (monotonicity), Player 1's stolen strategy performs at least as well as $S_2$ would for Player 2.
  5. But $S_2$ was a winning strategy for Player 2, so Player 1 now also wins — contradiction, since both cannot win.

Therefore, Player 2 cannot have a winning strategy, implying that Player 1 has a winning strategy (or can at least force a draw). In games where draws are impossible, Player 1 has a winning strategy.

Proof Template

We now formalize the argument into a reusable proof skeleton. For any specific game, you fill in the game-specific details (what constitutes a "move," what "extra piece" means, and why monotonicity holds).

Theorem (Strategy Stealing — General)

Let $\mathcal{G}$ be a finite, two-player, perfect-information game with no draws and with the monotonicity property. Then the first player has a winning strategy.

Proof.

Assume for contradiction that Player 2 has a winning strategy $S_2$.

Step 1 (Arbitrary opening). Player 1 plays an arbitrary legal move $m_0$. After this move, it is Player 2's turn.

Step 2 (Role reversal). From this point on, Player 1 mentally swaps roles. Whenever it is Player 1's turn, Player 1 consults $S_2$ as if Player 1 were Player 2 responding to the current board position (ignoring the extra piece $m_0$ for the purpose of strategy consultation).

Step 3 (Collision handling). If $S_2$ recommends placing a piece at a cell already occupied by the "extra" piece, Player 1 replaces the role of the extra piece: the old extra piece is now a legitimate part of the strategy, and Player 1 plays a new arbitrary move $m_0'$, which becomes the new extra piece.

Step 4 (Monotonicity application). At every point during the game, Player 1's board position is identical to what Player 2 would have under $S_2$, plus one extra piece. By the monotonicity property, this extra piece does not hurt Player 1. Therefore, if $S_2$ would have led Player 2 to victory, it leads Player 1 to at least as good an outcome.

Step 5 (Contradiction). Strategy $S_2$ guarantees a win for its user. Player 1, following the stolen $S_2$, also wins. But the game has no draws and exactly one winner per game — contradiction.

Hence our assumption is false: Player 2 does not have a winning strategy. By Zermelo's theorem (every finite game of perfect information has a determined outcome), Player 1 must have a winning strategy.

$\blacksquare$

When the Argument Applies

The strategy stealing argument works when all of these conditions hold:

ConditionWhy It's Needed
No drawsOtherwise, the argument only proves Player 1 can force at least a draw
MonotonicityAn extra piece must never hurt; otherwise the stolen strategy may backfire
Perfect informationBoth players must be able to evaluate the full board state
Symmetric starting positionThe "role swap" must make sense — if Player 2 has different powers, the swap fails

Non-Constructive Nature

A crucial feature of strategy stealing is that it is non-constructive: it proves that a winning strategy exists without revealing what it is. For games like Hex on large boards, we know the first player wins, but no explicit winning strategy is known for boards larger than about $9 \times 9$. This makes strategy stealing a fascinating example of a pure existence proof in combinatorics.

Constructive vs. Non-Constructive. In a constructive proof, you exhibit the winning strategy explicitly. In strategy stealing, you only show that the assumption "Player 2 wins" leads to contradiction. The actual winning moves remain a mystery. This parallels the non-constructive proofs in the probabilistic method: we know a good object exists, but we may not know how to find it.

Hex Game Application

Hex is the classic showcase for the strategy stealing argument. Invented independently by Piet Hein (1942) and John Nash (1948), Hex is played on a rhombus-shaped board of hexagonal cells. Two players (Black and White) alternate placing stones; Black tries to connect the top and bottom edges, White tries to connect the left and right edges.

Key Properties of Hex

  1. No draws: On a completely filled board, exactly one player has a winning connection. This follows from the topological fact that on a hex board, you cannot tile the rhombus without creating a path for one of the players — a consequence of the Hex theorem (related to the Brouwer fixed-point theorem).
  2. Monotonicity: An extra stone never hurts. If you have a winning connection with a set $S$ of stones, adding another stone to get $S \cup \{s\}$ preserves that connection.
  3. Symmetry: Both players have identical rules (place one stone per turn), just with different winning conditions.

Theorem (Nash, 1949). The first player wins Hex.

Proof.

No draws. On a completely filled $n \times n$ hex board, exactly one player has a connecting path (by a topological argument: two crossing paths would share a cell, impossible since each cell has one colour).

Monotonicity. An extra Black stone never destroys a Black connection. If Black wins from position $P$, Black also wins from $P \cup \{c\}$ for any cell $c$.

Strategy stealing. Suppose White has a winning strategy $S_W$. Black plays an arbitrary stone $m_0$, then follows $S_W$ (swapping roles). If $S_W$ recommends an occupied cell, Black plays elsewhere (new "extra" stone). At every stage, Black's board is White's board under $S_W$ plus one extra stone. By monotonicity, Black wins — contradiction.

Therefore White has no winning strategy, and Black (first player) wins.

$\blacksquare$

Non-constructive! For $n \times n$ Hex with $n \ge 10$, no explicit winning strategy is known. Hex has been solved only up to $9 \times 9$.

C++: Hex Board Evaluation by Minimax

For small boards, we can compute the game-theoretic value by exhaustive search. The following implementation uses minimax with alpha-beta pruning on a small Hex board. We use a union-find structure to detect winning connections efficiently. 

#include <bits/stdc++.h>
using namespace std;

// Hex board of size N x N (small N for exhaustive search)
const int N = 4;
const int TOTAL = N * N;

// Cell states
const int EMPTY = 0, BLACK = 1, WHITE = 2;

int board[N][N];

// Union-Find for connectivity detection
struct DSU {
    int par[TOTAL + 4]; // +4 for virtual nodes
    int rnk[TOTAL + 4];
    void init(int n) {
        for (int i = 0; i < n; i++) par[i] = i, rnk[i] = 0;
    }
    int find(int x) {
        return par[x] == x ? x : par[x] = find(par[x]);
    }
    void unite(int a, int b) {
        a = find(a); b = find(b);
        if (a == b) return;
        if (rnk[a] < rnk[b]) swap(a, b);
        par[b] = a;
        if (rnk[a] == rnk[b]) rnk[a]++;
    }
};

// Virtual node indices: BLACK_TOP, BLACK_BOT, WHITE_LEFT, WHITE_RIGHT
const int BT = TOTAL, BB = TOTAL + 1, WL = TOTAL + 2, WR = TOTAL + 3;

int idx(int r, int c) { return r * N + c; }

// Hex neighbor offsets (6 directions)
const int dr[] = {-1, -1, 0, 0, 1, 1};
const int dc[] = {0, 1, -1, 1, -1, 0};

// Build DSU for a given player and check connectivity
bool checkWin(int player) {
    DSU dsu;
    dsu.init(TOTAL + 4);
    for (int r = 0; r < N; r++)
        for (int c = 0; c < N; c++) {
            if (board[r][c] != player) continue;
            int u = idx(r, c);
            // Connect to virtual nodes
            if (player == BLACK) {
                if (r == 0) dsu.unite(u, BT);
                if (r == N-1) dsu.unite(u, BB);
            } else {
                if (c == 0) dsu.unite(u, WL);
                if (c == N-1) dsu.unite(u, WR);
            }
            // Connect to same-color hex neighbors
            for (int d = 0; d < 6; d++) {
                int nr = r + dr[d], nc = c + dc[d];
                if (nr < 0 || nr >= N || nc < 0 || nc >= N) continue;
                if (board[nr][nc] == player)
                    dsu.unite(u, idx(nr, nc));
            }
        }
    if (player == BLACK) return dsu.find(BT) == dsu.find(BB);
    return dsu.find(WL) == dsu.find(WR);
}

// Minimax with alpha-beta pruning
// Returns +1 if current player to move wins, -1 if loses
int minimax(int turn, int alpha, int beta) {
    // Check if previous player won
    int prev = (turn == BLACK) ? WHITE : BLACK;
    if (checkWin(prev)) return -1; // previous player made winning connection

    // Collect empty cells
    vector<int> moves;
    for (int r = 0; r < N; r++)
        for (int c = 0; c < N; c++)
            if (board[r][c] == EMPTY)
                moves.push_back(idx(r, c));

    if (moves.empty()) return 0; // board full, no winner (shouldn't happen in Hex)

    int best = -2;
    for (int m : moves) {
        board[m / N][m % N] = turn;
        int val = -minimax(turn == BLACK ? WHITE : BLACK, -beta, -alpha);
        board[m / N][m % N] = EMPTY;
        best = max(best, val);
        alpha = max(alpha, val);
        if (alpha >= beta) break;
    }
    return best;
}

int main() {
    memset(board, 0, sizeof(board));
    int result = minimax(BLACK, -2, 2);
    if (result > 0)
        cout << "First player (BLACK) wins " << N << "x" << N << " Hex\n";
    else if (result < 0)
        cout << "Second player (WHITE) wins\n";
    else
        cout << "Draw (impossible in Hex)\n";
    return 0;
}
Complexity. Minimax on a $4 \times 4$ board is already quite expensive ($16!$ in the worst case). With alpha-beta pruning, many branches are cut, but this approach is only feasible for small boards. The strategy stealing argument bypasses all computation by proving the result abstractly.

Chomp Application

Chomp is a two-player game played on a rectangular $m \times n$ grid of "chocolate squares." The bottom-left square is "poisoned." Players alternate choosing a square and removing it along with all squares above and to the right. The player forced to take the poisoned square loses.

Formal Definition

The game state is a Young diagram: a non-increasing sequence $(a_1, a_2, \ldots, a_m)$ where $a_i$ is the number of remaining squares in row $i$, with $n \ge a_1 \ge a_2 \ge \cdots \ge a_m \ge 0$. The initial state is $(n, n, \ldots, n)$. A move consists of choosing cell $(r, c)$ and setting $a_i \leftarrow \min(a_i, c-1)$ for all $i \ge r$. The terminal losing position is $(1, 0, 0, \ldots, 0)$ — only the poisoned square remains.

Why Monotonicity Holds

In Chomp, having fewer squares available is never an advantage (from the perspective of the player about to move). More precisely, Chomp is a misère-like game with the property that the first player can always "waste" a move by chomping a single corner square, effectively passing the advantage. The key insight: if Player 1 can win from position $P$, then Player 1 can also win from any position $P'$ that contains $P$ as a subset, because Player 1 can first reduce $P'$ to $P$ (a legal move in Chomp) and then follow the winning strategy for $P$.

Theorem. The first player wins Chomp on any $m \times n$ board with $m \cdot n > 1$.

Proof.

Assume for contradiction that Player 2 has a winning strategy $S_2$. By Zermelo's theorem this is well-defined (no draws: the board strictly shrinks each turn).

Consider Player 1's opening move: chomp only the top-right square $(1, n)$, removing that single cell. Let the resulting position be $P_1$. Strategy $S_2$ provides Player 2's reply $R$, leading to position $P_2$ from which the mover (Player 1) loses.

Key observation: Player 1 could have played move $R$ as the first move instead. Move $R$ is legal from the full board (it was legal from the smaller board $P_1 \subset$ full board). After Player 1 plays $R$ on the full board, the resulting position $P_2'$ satisfies $P_2 \subseteq P_2'$ (they differ by at most the top-right square).

If $P_2' = P_2$: Player 2 now faces position $P_2$ — the very position that is losing for the mover under $S_2$. Player 1 wins. Contradiction.

If $P_2' \supsetneq P_2$: the extra square makes $P_2'$ a larger board. In Chomp, a larger board is never better for the mover (the mover can always reduce to the smaller board in one move). So $P_2'$ is at least as bad for Player 2 as $P_2$ would be. Player 1 still wins. Contradiction.

Therefore Player 2 cannot have a winning strategy, and Player 1 wins.

$\blacksquare$

Remarkable non-constructivity. Despite proving that Player 1 wins, the proof gives no hint about the actual winning first move (except for the trivial $1 \times n$ and $m \times 1$ cases, where the winning move is to chomp everything except the poisoned square). For a $10 \times 10$ board, finding the winning first move requires computation.

C++: Chomp Game Solver

We solve Chomp exactly for small boards using dynamic programming on game states represented as Young diagrams. A position is a losing position (P-position) if every move leads to a winning position (N-position), and vice versa. 

#include <bits/stdc++.h>
using namespace std;

// Chomp on an M x N board
const int M = 5, N = 5;

// State: rows[0..M-1], rows[i] = number of columns remaining in row i
// Constraint: rows[0] >= rows[1] >= ... >= rows[M-1] >= 0

map<vector<int>, int> memo; // 1 = N-position (current player wins), 0 = P-position

// Terminal losing state: only (0,0) remains => rows = {1, 0, 0, ...}
bool isTerminal(const vector<int>& rows) {
    if (rows[0] != 1) return false;
    for (int i = 1; i < M; i++)
        if (rows[i] != 0) return false;
    return true;
}

// Returns 1 if current player wins, 0 if current player loses
int solve(vector<int> rows) {
    if (isTerminal(rows)) return 0; // forced to eat poison -> lose

    auto it = memo.find(rows);
    if (it != memo.end()) return it->second;

    // Try all valid chomp moves: pick cell (r, c) and remove everything
    // above-right. This sets rows[i] = min(rows[i], c) for all i >= r.
    for (int r = 0; r < M; r++) {
        for (int c = (r == 0 ? 1 : 0); c < rows[r]; c++) {
            // Chomp at (r, c): remove (r,c) and everything above-right
            vector<int> nrows = rows;
            for (int i = r; i < M; i++)
                nrows[i] = min(nrows[i], c);

            if (solve(nrows) == 0) {
                // Found a move to a P-position -> current position is N-position
                memo[rows] = 1;
                return 1;
            }
        }
    }

    memo[rows] = 0;
    return 0;
}

int main() {
    vector<int> start(M, N); // full M x N board
    int result = solve(start);
    cout << "Chomp " << M << "x" << N << ": ";
    cout << (result ? "First player wins" : "Second player wins") << "\n";

    // Find the winning first move (if first player wins)
    if (result) {
        cout << "Winning first moves:\n";
        for (int r = 0; r < M; r++) {
            for (int c = (r == 0 ? 1 : 0); c < start[r]; c++) {
                vector<int> nrows = start;
                for (int i = r; i < M; i++)
                    nrows[i] = min(nrows[i], c);
                if (solve(nrows) == 0)
                    cout << "  Chomp at (" << r << "," << c << ")\n";
            }
        }
    }
    return 0;
}
State space. The number of Chomp positions on an $m \times n$ board is the number of Young diagrams fitting in an $m \times n$ box, which equals $\binom{m+n}{m}$. For $5 \times 5$, that's $\binom{10}{5} = 252$ states — very manageable. For $20 \times 20$, it's $\binom{40}{20} \approx 1.4 \times 10^{11}$, which is infeasible without further tricks.

Other Applications

Strategy stealing applies to many games beyond Hex and Chomp.

Tic-Tac-Toe Variants

In standard $3 \times 3$ tic-tac-toe, strategy stealing proves Player 1 can at least draw (an extra mark never hurts). For $n$-in-a-row on an infinite board:

  • Maker-Breaker: No draws exist, so strategy stealing proves Maker wins (extra stone never hurts Maker).
  • Strong game: Strategy stealing proves Player 1 draws or wins. For $n \le 4$ Player 1 wins; for $n \ge 8$ it's a draw.

Connect Four

Strategy stealing does not apply to Connect Four because the gravity mechanic breaks monotonicity: an extra disc raises the column floor, potentially blocking your own plays. The game was solved computationally (Victor Allis, 1988): first player wins by opening in the center column.

Maker-Breaker Games

In a Maker-Breaker game on a hypergraph $(V, \mathcal{F})$, Maker and Breaker alternately claim vertices. Maker wins by claiming all vertices of some $F \in \mathcal{F}$. An extra vertex never hurts Maker, so if Maker can draw, Maker wins. This relates to the Erdős–Selfridge theorem:

$$\text{If Breaker wins the Maker-Breaker game, then the corresponding strong game is a draw.}$$

Bridg-It (Shannon Switching Game)

Bridg-It is played on two interlocked rectangular lattices. Like Hex it has no draws and satisfies monotonicity, so strategy stealing proves the first player wins. Unlike Hex, Bridg-It has a known explicit pairing strategy (Oliver Gross).

Limitations

The strategy stealing argument is not universally applicable. Several important classes of games resist the technique.

When Extra Moves Hurt (Zugzwang)

In many games, being forced to move is a disadvantage. This is called zugzwang (German: "compulsion to move"). In chess, there are positions where the player to move would prefer to pass but cannot — every legal move worsens their position. Since an extra move can hurt, the monotonicity property fails, and strategy stealing does not apply.

Similarly, in Nim and other impartial games analyzed by Sprague-Grundy theory, having more stones in a pile can be either an advantage or a disadvantage depending on the XOR of the Grundy values. Strategy stealing does not apply to Nim.

Games with Draws

When draws are possible, strategy stealing only proves Player 1 can at least draw. For chess, this suggests White can at least draw (widely believed, not proven). For tic-tac-toe, it proves Player 1 draws or wins — which is tight.

Asymmetric Games & Monotonicity Failures

Strategy stealing requires symmetric rules. If players have different move sets, the role-swap step fails. Even in placement games, monotonicity can fail:

  • Go: Captured groups give the opponent points; ko rules break monotonicity.
  • Othello: A disc in the wrong position gets flipped against you.
  • Connect Four: Gravity means an extra disc blocks your own column.
A practical test. Before attempting a strategy-stealing proof, ask: "Can I always ignore an extra piece of my own colour?" If yes, monotonicity likely holds. If you can construct a scenario where an extra piece hurts you, strategy stealing will not work for that game.

C++: Detecting Monotonicity Violations

We can computationally check whether adding a piece ever worsens a position. This example demonstrates that Nim violates monotonicity. 

#include <bits/stdc++.h>
using namespace std;

// Nim: current player wins iff XOR of pile sizes != 0
int evaluateNim(const vector<int>& piles) {
    int x = 0;
    for (int p : piles) x ^= p;
    return (x != 0) ? 1 : -1;
}

bool checkMonotonicityNim() {
    for (int a = 0; a <= 5; a++) {
        for (int b = 0; b <= 5; b++) {
            int val = evaluateNim({a, b});
            if (a < 5) {
                int valPlus = evaluateNim({a + 1, b});
                if (val == 1 && valPlus == -1) {
                    cout << "VIOLATED: Nim(" << a << "," << b
                         << ") wins, Nim(" << a+1 << "," << b << ") loses\n";
                    return false;
                }
            }
        }
    }
    return true;
}

int main() {
    checkMonotonicityNim();
    // Nim(2,3): XOR=1 (win). Nim(3,3): XOR=0 (loss).
    // Adding a stone turned winning into losing!
    cout << "Nim(2,3) XOR=" << (2^3) << " (win), "
         << "Nim(3,3) XOR=" << (3^3) << " (loss)\n";
    cout << "Strategy stealing fails for Nim.\n";
}

Practice Problems

  1. Symmetric Placement Game: Two players place stones on an $n \times n$ grid, winning by completing a connected path between opposite edges. Prove first player wins. Hint: Verify no draws + monotonicity + symmetry, then apply the template.
  2. 3D Chomp: Prove first player wins Chomp on $\ell \times m \times n$ with $\ell m n > 1$. Hint: Generalize the corner-chomp argument.
  3. Anti-Strategy-Stealing: Give an example where the second player wins a no-draw game. Explain why strategy stealing fails. Hint: Consider pure zugzwang games where moving is always bad.
  4. Small Hex: For $3 \times 3$ Hex, find the unique winning first move among the 3 distinct positions (center, edge, corner).
