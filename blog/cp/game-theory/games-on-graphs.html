    Games on Graphs

    June 2026 · 25 min read

    Many combinatorial games can be modelled as a token sitting on a vertex of a graph, with players alternating moves along edges. The player unable to move loses. This framing unifies classics like Nim, Green Hackenbush, and countless competitive-programming problems under one roof: graph game theory.

    In this post we develop the theory of token-moving games on both directed acyclic graphs (DAGs) and general graphs, prove the key theorems, and implement efficient C++ solvers ready for contest use.

    Token Games on Graphs

    A graph game is a two-player, normal-play, combinatorial game defined by:

    • A directed graph $G = (V, E)$ (the game graph).
    • A starting vertex $v_0 \in V$ (the token's initial position).
    • Two players who alternate turns. On each turn the current player must move the token along an outgoing edge from the current vertex.
    • Normal play convention: the first player unable to move loses.

    Definition (Position types). A vertex $v$ is a P-position (previous player wins) if the player who just moved there wins — i.e., the player whose turn it is to move from $v$ loses with optimal play. A vertex is an N-position (next player wins) otherwise.

    The recursive characterisation is immediate:

    • A vertex with no outgoing edges (terminal) is a P-position (the player to move has no move and loses).
    • A vertex is an N-position if at least one successor is a P-position (the current player can move to a losing position for the opponent).
    • A vertex is a P-position if all successors are N-positions (every move leaves the opponent in a winning position).
    Formal rule. Let $F(v) = \{u : (v,u) \in E\}$ denote the set of followers of $v$. Then: $$ \text{type}(v) = \begin{cases} P & \text{if } F(v) = \emptyset, \\ P & \text{if } \forall u \in F(v),\; \text{type}(u) = N, \\ N & \text{if } \exists\, u \in F(v),\; \text{type}(u) = P. \end{cases} $$

    Example. Consider a path $v_0 \to v_1 \to v_2 \to v_3$ (three edges). The terminal vertex $v_3$ is P. Then $v_2$ has a successor that is P, so $v_2$ is N. Vertex $v_1$'s only successor is N, so $v_1$ is P. Finally $v_0$ has a P-successor, so $v_0$ is N — the first player wins by moving to $v_1$.

    This classification can be computed in linear time on a DAG (next section). On general directed graphs the situation is more nuanced, because cycles mean the recursion may never terminate. We handle that later.

    Games on DAGs

    When the game graph is a directed acyclic graph (DAG), every play is guaranteed to terminate (the token moves along a path in a finite DAG, which must end at a sink). This is the cleanest setting for Sprague–Grundy theory.

    Grundy Values via Reverse Topological Order

    Recall the Sprague–Grundy value (or nimber) of a position $v$:

    $$ \mathcal{G}(v) = \operatorname{mex}\bigl(\{\mathcal{G}(u) : u \in F(v)\}\bigr), $$

    where $\operatorname{mex}(S)$ is the minimum excludant — the smallest non-negative integer not in $S$.

    On a DAG we compute Grundy values by processing vertices in reverse topological order (sinks first). This ensures that when we process $v$, every successor $u \in F(v)$ has already been assigned a value.

    Theorem (DAG Game Analysis). Let $G = (V, E)$ be a finite DAG. Then for every vertex $v$:
    (i) $\mathcal{G}(v) = 0$ if and only if $v$ is a P-position.
    (ii) $\mathcal{G}(v) > 0$ if and only if $v$ is an N-position.
    (iii) The Grundy value can be computed in $O(|V| + |E|)$ time.

    Proof.

    (i)–(ii) follow from the mex definition by induction on the topological order:

    • Base case. A sink $v$ has $F(v) = \emptyset$, so $\mathcal{G}(v) = \operatorname{mex}(\emptyset) = 0$. A sink is indeed a P-position.
    • Inductive step. Suppose the claim holds for all successors of $v$.
      • If $\mathcal{G}(v) = 0$, then $0 \notin \{\mathcal{G}(u) : u \in F(v)\}$ is impossible since $\operatorname{mex}$ returns $0$ only when $0$ is absent… Actually, $\operatorname{mex}(S) = 0$ means $0 \notin S$, which means no successor has Grundy value $0$, which means no successor is a P-position. Hence every successor is an N-position, and $v$ is a P-position. ■
      • If $\mathcal{G}(v) > 0$, then since $\operatorname{mex}(S) > 0$ means $0 \in S$, there exists a successor $u$ with $\mathcal{G}(u) = 0$, which is a P-position by induction. Hence $v$ is an N-position. ■

    (iii) Complexity. Topological sort runs in $O(|V| + |E|)$. For each vertex we compute $\operatorname{mex}$ over its successors. Using a boolean marker array that we reset per-vertex, the total work across all vertices is $O\bigl(\sum_{v} |F(v)|\bigr) = O(|E|)$. Total: $O(|V| + |E|)$. ■

    Algorithm: Grundy on DAG

    // Compute Grundy values on a DAG in reverse topological order
// adj[v] = list of successors of v
vector<int> grundy_dag(int n, const vector<vector<int>>& adj) {
    // 1. Compute in-degrees and topological order (Kahn's algorithm)
    vector<int> indeg(n, 0);
    for (int v = 0; v < n; v++)
        for (int u : adj[v])
            indeg[u]++;

    queue<int> q;
    for (int v = 0; v < n; v++)
        if (indeg[v] == 0)
            q.push(v);

    vector<int> topo;
    topo.reserve(n);
    while (!q.empty()) {
        int v = q.front(); q.pop();
        topo.push_back(v);
        for (int u : adj[v])
            if (--indeg[u] == 0)
                q.push(u);
    }

    // 2. Process in reverse topological order
    vector<int> g(n, 0);
    for (int i = (int)topo.size() - 1; i >= 0; i--) {
        int v = topo[i];
        // Collect Grundy values of successors
        vector<bool> seen(adj[v].size() + 1, false);
        for (int u : adj[v]) {
            if (g[u] < (int)seen.size())
                seen[g[u]] = true;
        }
        // Compute mex
        g[v] = 0;
        while (g[v] < (int)seen.size() && seen[g[v]])
            g[v]++;
    }
    return g;
}
    Optimisation note. The $\operatorname{mex}$ computation above sizes the seen array per-vertex. In competitive programming, a common trick is to use a single global array with a generation counter to avoid re-allocation: 
    // Fast mex with global marker array (avoids per-vertex allocation)
int mex_stamp = 0;
int stamp[MAXN]; // initialised to 0

int fast_mex(const vector<int>& vals) {
    mex_stamp++;
    for (int x : vals)
        if (x < MAXN)
            stamp[x] = mex_stamp;
    for (int i = 0; ; i++)
        if (stamp[i] != mex_stamp)
            return i;
}

    Multiple Tokens

    The real power of Sprague–Grundy theory appears when the game consists of multiple independent tokens, each moving on its own game graph (or on the same graph independently). By the Sprague–Grundy theorem, the combined game is equivalent to a single Nim heap whose size equals the XOR of the individual Grundy values.

    Theorem (Sprague–Grundy for Game Sums). Let $G_1, G_2, \ldots, G_k$ be independent combinatorial games with Grundy values $\mathcal{G}_1, \mathcal{G}_2, \ldots, \mathcal{G}_k$ for their respective starting positions. Then the combined game (where each turn a player picks one component and makes a move) has Grundy value: $$ \mathcal{G}(G_1 + G_2 + \cdots + G_k) = \mathcal{G}_1 \oplus \mathcal{G}_2 \oplus \cdots \oplus \mathcal{G}_k, $$ where $\oplus$ denotes bitwise XOR. In particular, the first player wins if and only if the XOR is non-zero.

    Proof sketch. We verify the two conditions that characterise Grundy values:

    1. Reachability. For any target value $g < \mathcal{G}_1 \oplus \cdots \oplus \mathcal{G}_k$, there exists a move in some component that changes the overall XOR to $g$. This follows from the standard Nim strategy argument: find the highest bit where $g$ and the current XOR differ, pick a component whose Grundy value has that bit set, and move it to the appropriate value (which is reachable since the target value is smaller than the component's current Grundy value).
    2. Non-reachability. No single move can leave the XOR unchanged. If only one component changes, and all others stay the same, the XOR changes if and only if the moved component's Grundy value changes — which it must, since a move always changes the position. ■

    Application: Multiple Tokens on a DAG

    Suppose we have $k$ tokens at positions $v_1, v_2, \ldots, v_k$ on a DAG $G$. Each turn, a player picks one token and moves it along an edge. The player who cannot move (all tokens stuck at sinks) loses.

    This decomposes into $k$ independent single-token games. First compute $\mathcal{G}(v)$ for all $v$ in $O(|V| + |E|)$, then:

    $$ \text{First player wins} \iff \mathcal{G}(v_1) \oplus \mathcal{G}(v_2) \oplus \cdots \oplus \mathcal{G}(v_k) \neq 0. $$ 
    // Solve a multi-token game on a DAG
void solve() {
    int n, m, k;
    scanf("%d%d%d", &n, &m, &k);

    vector<vector<int>> adj(n);
    for (int i = 0; i < m; i++) {
        int u, v;
        scanf("%d%d", &u, &v);
        adj[u].push_back(v);
    }

    vector<int> g = grundy_dag(n, adj);

    int xor_sum = 0;
    for (int i = 0; i < k; i++) {
        int v;
        scanf("%d", &v);
        xor_sum ^= g[v];
    }

    puts(xor_sum ? "First" : "Second");
}

    Finding a Winning Move

    When the XOR is non-zero, we can find an actual winning move. The idea mirrors Nim strategy: pick a component whose Grundy value, when changed, makes the total XOR zero.

    // Find a winning move: returns {token_index, target_vertex}
pair<int,int> find_winning_move(
    const vector<int>& tokens,
    const vector<int>& g,
    const vector<vector<int>>& adj)
{
    int xor_sum = 0;
    for (int v : tokens) xor_sum ^= g[v];

    for (int i = 0; i < (int)tokens.size(); i++) {
        int need = xor_sum ^ g[tokens[i]];
        // We need to move token i to a vertex u with g[u] == need
        for (int u : adj[tokens[i]]) {
            if (g[u] == need)
                return {i, u};
        }
    }
    return {-1, -1}; // no winning move (xor_sum was 0)
}

    Games on Undirected Graphs

    When the game graph is undirected, cycles become possible: the token can revisit vertices. This causes two fundamental complications:

    1. Games may not terminate. If both players can keep the token cycling forever, the game has no natural end under the normal play convention. To resolve this, we typically add a rule: no vertex may be visited twice (the token leaves a trail and edges/vertices are consumed).
    2. The Sprague–Grundy recursion breaks down. With cycles, the P/N classification and mex computation cannot proceed via topological order since none exists.

    The "No Revisit" Model

    The most common undirected graph game in competitive programming uses the rule: each vertex can be visited at most once. The game state is then $(v, S)$ where $v$ is the current vertex and $S \subseteq V$ is the set of visited vertices. This state space is exponential in general, but many problems have special structure.

    When Sprague–Grundy applies to undirected games:

    • Trees. On trees (acyclic undirected graphs), once we orient edges away from the starting vertex, we get a DAG. Classical Sprague–Grundy applies. The Grundy value of a subtree rooted at $v$ satisfies $\mathcal{G}(v) = \mathcal{G}(c_1) \oplus \mathcal{G}(c_2) \oplus \cdots \oplus \mathcal{G}(c_d)$ where $c_1, \ldots, c_d$ are $v$'s children, with each child's value incremented by one edge (the Green Hackenbush rule for trees).
    • Bipartite graphs. A deep result connects the game on bipartite graphs to maximum matchings: the first player wins if and only if the starting vertex is in every maximum matching. This is decidable in polynomial time.
    Theorem (Undirected Games on Bipartite Graphs). In a token game on an undirected bipartite graph with the no-revisit rule, starting from vertex $v$: the first player wins if and only if $v$ belongs to every maximum matching of the graph.

    Proof outline.

    1. If $v$ is not in some maximum matching $M$, the second player can use a strategy based on $M$: whenever the first player moves along an unmatched edge to vertex $u$, the second player responds by moving along the matched edge from $u$ (which exists since $u$ is matched in $M$). Since $M$ is a matching in a bipartite graph, this strategy never revisits a vertex and the first player runs out of moves first.
    2. Conversely, if $v$ is in every maximum matching, the first player can move to the other endpoint of $v$'s matched edge in some maximum matching $M$. The argument then reduces by symmetry: the opponent is now at a vertex that is not in every maximum matching of the residual graph. ■

    Game on a Tree

    For the standard game on a tree (token moves along edges, no revisits), we compute Grundy values bottom-up. Each subtree rooted at a child contributes an independent game of depth equal to the edge plus the child's subtree game. The Grundy value of a bamboo (path) of length $\ell$ is simply $\ell$, so:

    $$ \mathcal{G}(\text{tree rooted at } v) = \bigoplus_{c \in \text{children}(v)} \bigl(\mathcal{G}(\text{subtree}(c)) + 1\bigr) $$

    Wait — that's not quite right for token games (vs Green Hackenbush). For the simple token game on a tree with the no-revisit rule, the analysis is simpler:

    Fact. In the token game on a tree (no-revisit rule, starting at root $r$), the first player wins if and only if $r$ is matched in a maximum matching of the tree.

    Since maximum matching on a tree is computable in $O(|V|)$ via a greedy bottom-up algorithm, this gives a linear-time solution.

    Kernel of a Graph

    The kernel of a directed graph provides an elegant characterisation of P-positions in graph games.

    Definition. A kernel of a directed graph $G = (V, E)$ is a set $K \subseteq V$ such that:

    1. Independence: No two vertices in $K$ are connected by an edge. That is, for all $u, v \in K$, $(u, v) \notin E$.
    2. Absorbance: Every vertex outside $K$ has an edge to some vertex in $K$. That is, for all $v \in V \setminus K$, there exists $u \in K$ with $(v, u) \in E$.
    Theorem (Kernel = P-positions). Let $G = (V, E)$ be a directed graph that admits a kernel $K$. Then $K$ is exactly the set of P-positions of the token game on $G$.

    Proof.

    We verify that $K$ satisfies the recursive characterisation of P-positions:

    • Every terminal vertex is in $K$: A terminal vertex $v$ has $F(v) = \emptyset$. Since $v$ has no outgoing edges, it vacuously satisfies independence. For absorbance, $v$ itself doesn't need to absorb anything (it has no outgoing edges, but absorbance requires vertices outside $K$ to reach $K$). If $v \notin K$, then absorbance demands $(v, u) \in E$ for some $u \in K$, but $F(v) = \emptyset$, a contradiction. So $v \in K$. ■
    • From any vertex in $K$, all moves go outside $K$: This is exactly the independence property. ■
    • From any vertex outside $K$, there is a move into $K$: This is exactly the absorbance property. ■

    These three conditions are precisely the defining properties of P-positions.

    Existence and Uniqueness

    Theorem (Richardson, 1953). Every finite DAG has a unique kernel.

    Proof. We prove existence and uniqueness simultaneously by induction on the topological order.

    Base case. Sinks (vertices with no outgoing edges) must be in the kernel by the argument above.

    Inductive step. Process vertices in reverse topological order. For vertex $v$:

    • If all successors of $v$ are not in the kernel, then $v$ must be in the kernel (otherwise absorbance from $v$ fails, since no successor is in $K$… wait, we need to be more careful). Actually: if some successor of $v$ is in $K$, then $v$ cannot be in $K$ (independence would be violated). If no successor of $v$ is in $K$, then $v$ must be in $K$ (absorbance for $v$'s predecessors or for $v$ itself requires this).

    More precisely, define $K$ by the rule:

    $$v \in K \iff \forall u \in F(v),\; u \notin K.$$

    In a DAG, this is well-defined when processing in reverse topological order. We verify:

    1. Independence: If $v \in K$ and $(v, u) \in E$, then by construction $u \notin K$. ■
    2. Absorbance: If $v \notin K$, then by the contrapositive of the defining rule, there exists $u \in F(v)$ with $u \in K$. ■

    Uniqueness follows because the rule forces the assignment at each step: a vertex is in $K$ if and only if none of its successors are, leaving no freedom. ■

    General Directed Graphs

    For general directed graphs (with cycles), kernels may not exist, and when they do, they may not be unique.

    • No kernel: A directed 3-cycle $v_1 \to v_2 \to v_3 \to v_1$ has no kernel. (Check: any single vertex fails absorbance for its predecessor, and any pair fails independence.)
    • Non-unique kernels: An even cycle $v_1 \to v_2 \to v_3 \to v_4 \to v_1$ has two kernels: $\{v_1, v_3\}$ and $\{v_2, v_4\}$.
    Theorem (Richardson, 1953). Every directed graph in which every directed cycle has even length admits a kernel. In particular, every DAG (no cycles at all) has a kernel.

    The condition "every directed cycle has even length" is equivalent to the graph being kernel-perfect. For games on graphs with odd cycles, the game-theoretic analysis requires different techniques (e.g., the full theory of loopy games by Fraenkel and others).

    C++ Implementation

    Below is a full contest-ready implementation covering the key algorithms for graph games.

    BFS-Based P/N Classification

    For games where only the win/loss status is needed (not Grundy values), a backward BFS from terminal positions is efficient and avoids topological sorting.

    #include <bits/stdc++.h>
using namespace std;

// Classify positions as P (0) or N (1) via backward BFS
// adj = forward adjacency, radj = reverse adjacency
// Returns: status[v] = 0 (P-position) or 1 (N-position), -1 if undetermined (cycles)
vector<int> classify_positions(int n,
    const vector<vector<int>>& adj,
    const vector<vector<int>>& radj)
{
    vector<int> outdeg(n), status(n, -1);
    queue<int> q;

    for (int v = 0; v < n; v++) {
        outdeg[v] = (int)adj[v].size();
        if (outdeg[v] == 0) {
            status[v] = 0; // terminal = P-position
            q.push(v);
        }
    }

    while (!q.empty()) {
        int v = q.front(); q.pop();

        for (int u : radj[v]) {
            if (status[u] != -1) continue;

            if (status[v] == 0) {
                // v is P, so u is N (u can move to a P-position)
                status[u] = 1;
                q.push(u);
            } else {
                // v is N, so u might be P if ALL successors are N
                outdeg[u]--; // one more successor classified as N
                if (outdeg[u] == 0) {
                    status[u] = 0;
                    q.push(u);
                }
            }
        }
    }
    return status;
}
    Key insight: We maintain an outdeg counter that counts unresolved successors. When a successor is classified as N, we decrement the counter. A vertex becomes P only when all successors are N (counter hits 0). A vertex becomes N as soon as any successor is P.

    Full Grundy Computation on DAG

    const int MAXN = 200005;
int stamp[MAXN], mex_gen = 0;

int compute_mex(const vector<int>& vals) {
    ++mex_gen;
    for (int x : vals)
        if (x < MAXN)
            stamp[x] = mex_gen;
    for (int i = 0; ; i++)
        if (stamp[i] != mex_gen)
            return i;
}

vector<int> grundy_values(int n, const vector<vector<int>>& adj) {
    vector<int> indeg(n, 0), g(n, 0);
    for (int v = 0; v < n; v++)
        for (int u : adj[v])
            indeg[u]++;

    // Kahn's algorithm for topological order
    queue<int> q;
    for (int v = 0; v < n; v++)
        if (indeg[v] == 0)
            q.push(v);

    vector<int> topo;
    topo.reserve(n);
    while (!q.empty()) {
        int v = q.front(); q.pop();
        topo.push_back(v);
        for (int u : adj[v])
            if (--indeg[u] == 0)
                q.push(u);
    }

    // Reverse topological order: process sinks first
    for (int i = (int)topo.size() - 1; i >= 0; i--) {
        int v = topo[i];
        vector<int> succ_grundy;
        succ_grundy.reserve(adj[v].size());
        for (int u : adj[v])
            succ_grundy.push_back(g[u]);
        g[v] = compute_mex(succ_grundy);
    }
    return g;
}

int main() {
    memset(stamp, 0, sizeof(stamp));
    int n, m, k;
    scanf("%d%d%d", &n, &m, &k);

    vector<vector<int>> adj(n);
    for (int i = 0; i < m; i++) {
        int u, v;
        scanf("%d%d", &u, &v);
        adj[u].push_back(v);
    }

    auto g = grundy_values(n, adj);

    int xor_sum = 0;
    for (int i = 0; i < k; i++) {
        int v;
        scanf("%d", &v);
        xor_sum ^= g[v];
    }

    puts(xor_sum ? "First" : "Second");
    return 0;
}

    Handling Large State Spaces

    Many competitive programming problems encode the game graph implicitly: the vertices are game states and edges are valid moves, but the graph is never stored explicitly. Instead, we compute Grundy values via DFS with memoisation.

    // Memoised DFS for implicit game graphs
unordered_map<long long, int> memo;

// Encode state as a long long (problem-specific)
int grundy_memo(long long state, /* ... game parameters ... */) {
    auto it = memo.find(state);
    if (it != memo.end()) return it->second;

    vector<int> reachable;
    // Generate all successor states from 'state'
    for (auto next_state : get_successors(state)) {
        reachable.push_back(grundy_memo(next_state));
    }

    int g = compute_mex(reachable);
    memo[state] = g;
    return g;
}
    Contest tip: When the state space is large but the Grundy values exhibit a pattern (periodicity, dependence on a small parameter), compute values for small cases, spot the pattern, and prove or assume it holds. Many game-theory problems on Codeforces/AtCoder are designed this way.

    Kernel Computation

    // Compute the kernel of a DAG (= set of P-positions)
vector<bool> compute_kernel(int n, const vector<vector<int>>& adj) {
    // Topological sort
    vector<int> indeg(n, 0);
    for (int v = 0; v < n; v++)
        for (int u : adj[v])
            indeg[u]++;

    queue<int> q;
    for (int v = 0; v < n; v++)
        if (indeg[v] == 0)
            q.push(v);

    vector<int> topo;
    topo.reserve(n);
    while (!q.empty()) {
        int v = q.front(); q.pop();
        topo.push_back(v);
        for (int u : adj[v])
            if (--indeg[u] == 0)
                q.push(u);
    }

    // Assign kernel membership in reverse topological order
    vector<bool> in_kernel(n, false);
    for (int i = (int)topo.size() - 1; i >= 0; i--) {
        int v = topo[i];
        bool all_non_kernel = true;
        for (int u : adj[v]) {
            if (in_kernel[u]) {
                all_non_kernel = false;
                break;
            }
        }
        in_kernel[v] = all_non_kernel;
    }
    return in_kernel;
}

    Tips for Graph Game Problems

    • Identify the game graph. The first step is always to model the problem as a token-moving game. What are the vertices (states)? What are the edges (moves)?
    • Check for DAG structure. If the game graph is a DAG (e.g., moves always decrease some quantity), apply Sprague–Grundy directly.
    • Look for decomposition. If the game splits into independent components, compute Grundy values for each and XOR them.
    • Spot patterns. For implicit game graphs, compute Grundy values for small cases and look for periodicity or closed-form expressions.
    • Maximum matching for undirected games. If the game is on an undirected graph with no-revisit rule, check if matching theory applies (especially on bipartite graphs and trees).
