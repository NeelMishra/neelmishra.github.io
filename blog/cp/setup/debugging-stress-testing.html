Stress Testing Framework

Stress testing is the single most effective debugging technique in competitive programming. The idea is simple: generate thousands of small random inputs, run both your solution and a brute force solution, and compare outputs. When they disagree, you have a failing test case to debug.

The typical stress test pipeline has three components:

  1. gen.cpp — generates random valid input
  2. brute.cpp — correct but slow O(n²) or O(n³) solution
  3. sol.cpp — your optimized solution being tested

The Stress Test Script (Bash)

#!/bin/bash # stress.sh — compile and run stress test g++ -O2 -o gen gen.cpp g++ -O2 -o brute brute.cpp g++ -O2 -o sol sol.cpp for i in $(seq 1 10000); do ./gen $i > input.txt # generate test case ./brute < input.txt > out1.txt # brute force answer ./sol < input.txt > out2.txt # your solution's answer if ! diff -q out1.txt out2.txt > /dev/null; then echo "MISMATCH on test $i" echo "Input:" cat input.txt echo "Expected:" cat out1.txt echo "Got:" cat out2.txt break fi if (( i % 1000 == 0 )); then echo "Passed $i tests" fi done

All-in-One C++ Stress Test

For speed, you can combine all three in a single file:

#include <bits/stdc++.h> using namespace std; mt19937 rng(42); // fixed seed for reproducibility int randint(int lo, int hi) { return uniform_int_distribution<int>(lo, hi)(rng); } // Your fast solution int solve(vector<int>& a) { // ... your O(n log n) algorithm ... return 0; } // Brute force (correct but slow) int brute(vector<int>& a) { // ... O(n^2) or O(n^3) solution ... return 0; } int main() { for (int iter = 0; iter < 100000; iter++) { int n = randint(1, 20); // small size for brute vector<int> a(n); for (int i = 0; i < n; i++) a[i] = randint(-100, 100); int expected = brute(a); int got = solve(a); if (expected != got) { cerr << "FAIL on iter " << iter << '\n'; cerr << "n = " << n << '\n'; for (int x : a) cerr << x << ' '; cerr << '\n'; cerr << "expected: " << expected << '\n'; cerr << "got: " << got << '\n'; return 1; } } cerr << "All tests passed!\n"; return 0; }
Pro tip: Use a fixed seed (like 42) during debugging so failures are reproducible. Switch to a time-based seed for more thorough testing: mt19937 rng(chrono::steady_clock::now().time_since_epoch().count())

Test Case Generators

Writing good generators is an art. The generator must produce valid inputs that respect all constraints in the problem statement.

Random Array Generator

// gen_array.cpp — generates random array #include <bits/stdc++.h> using namespace std; int main(int argc, char* argv[]) { mt19937 rng(atoi(argv[1])); // seed from command line auto randint = [&](int lo, int hi) { return uniform_int_distribution<int>(lo, hi)(rng); }; int n = randint(1, 15); cout << n << '\n'; for (int i = 0; i < n; i++) cout << randint(1, 1000) << " \n"[i == n - 1]; return 0; }

Random Tree Generator

// gen_tree.cpp — generates random tree on n vertices #include <bits/stdc++.h> using namespace std; int main(int argc, char* argv[]) { mt19937 rng(atoi(argv[1])); auto randint = [&](int lo, int hi) { return uniform_int_distribution<int>(lo, hi)(rng); }; int n = randint(2, 12); cout << n << '\n'; // Prüfer sequence method: generates uniformly random trees vector<int> prufer(n - 2); for (int& x : prufer) x = randint(1, n); // Convert Prüfer sequence to tree edges vector<int> degree(n + 1, 1); for (int x : prufer) degree[x]++; set<int> leaves; for (int i = 1; i <= n; i++) if (degree[i] == 1) leaves.insert(i); for (int x : prufer) { int leaf = *leaves.begin(); leaves.erase(leaves.begin()); cout << leaf << " " << x << '\n'; if (--degree[x] == 1) leaves.insert(x); } // Last edge connects the two remaining nodes auto it = leaves.begin(); int u = *it++, v = *it; cout << u << " " << v << '\n'; return 0; }

Random Graph Generator

// gen_graph.cpp — generates random connected graph #include <bits/stdc++.h> using namespace std; int main(int argc, char* argv[]) { mt19937 rng(atoi(argv[1])); auto randint = [&](int lo, int hi) { return uniform_int_distribution<int>(lo, hi)(rng); }; int n = randint(2, 8); int max_edges = n * (n - 1) / 2; int m = randint(n - 1, min(max_edges, 2 * n)); // Start with a spanning tree set<pair<int,int>> edges; vector<int> perm(n); iota(perm.begin(), perm.end(), 1); shuffle(perm.begin(), perm.end(), rng); for (int i = 1; i < n; i++) { int u = perm[randint(0, i - 1)], v = perm[i]; edges.insert({min(u, v), max(u, v)}); } // Add random extra edges while ((int)edges.size() < m) { int u = randint(1, n), v = randint(1, n); if (u != v) edges.insert({min(u, v), max(u, v)}); } cout << n << " " << edges.size() << '\n'; for (auto [u, v] : edges) cout << u << " " << v << '\n'; return 0; }
Generator pitfall: Your generator must produce inputs that satisfy ALL problem constraints. A generator producing disconnected graphs when the problem guarantees connectivity will find "bugs" that don't exist. Read the constraints carefully.

Brute Force Comparators

The brute force solution must be obviously correct, even if extremely slow. The goal is zero chance of bugs in the brute.

Writing Good Brute Force

✅ Good Brute

  • Simple nested loops
  • No clever optimizations
  • Direct translation of problem statement
  • Try all possibilities (permutations, subsets)

❌ Bad Brute

  • Uses same algorithm as main solution
  • Has its own optimizations that might be wrong
  • Too complex to verify by inspection
  • Shares code with the main solution

Example: Brute Force for Maximum Subarray

// Brute: O(n^2) — check all subarrays long long brute_max_subarray(vector<int>& a) { int n = a.size(); long long best = LLONG_MIN; for (int i = 0; i < n; i++) { long long sum = 0; for (int j = i; j < n; j++) { sum += a[j]; best = max(best, sum); } } return best; } // Your solution: O(n) Kadane's algorithm long long solve_max_subarray(vector<int>& a) { long long best = LLONG_MIN, cur = 0; for (int x : a) { cur = max((long long)x, cur + x); best = max(best, cur); } return best; }

Systematic Edge Case Construction

Many bugs only appear on specific edge cases. Instead of hoping random tests catch them, systematically construct inputs that target common failure modes.

Edge Case Checklist

Array Problems

  • n = 1 (single element)
  • All elements equal
  • Sorted ascending/descending
  • Maximum values (109, 1018)
  • All zeros, all negative
  • Alternating pattern
  • n at maximum constraint

Graph Problems

  • Single vertex (n = 1)
  • Single edge (n = 2)
  • Star graph (one center, n-1 leaves)
  • Path graph (chain)
  • Complete graph
  • Disconnected components
  • Self-loops and multi-edges (if allowed)

Constructing Adversarial Inputs

Some algorithms have worst-case inputs that differ from random ones:

// Adversarial input for quicksort (triggers O(n^2)) // Already sorted array with naive pivot selection // Adversarial input for hash maps (collision attack) // See "blowing up unordered_map" — use custom hash! // Adversarial input for DFS on deep tree (stack overflow) // Chain of n=200000 nodes — use iterative DFS // Adversarial input for string matching // Pattern: "aaa...ab", Text: "aaa...a" — triggers naive O(nm)
Stack overflow check: On Codeforces, the default stack is ~4MB. A recursive DFS on a path of 200,000 nodes will overflow. Always either increase the stack size (#pragma comment(linker, "/STACK:1000000000")) or use iterative DFS/BFS.

Debug Printing Macros

Effective debug output is crucial during contests. Here's a debug macro system that prints variable names and values, and is automatically disabled when you submit:

#ifdef LOCAL #define dbg(...) cerr << "[" << #__VA_ARGS__ << "]: ", \ debug_out(__VA_ARGS__) #else #define dbg(...) // nothing in submission #endif void debug_out() { cerr << '\n'; } template<typename T, typename... Args> void debug_out(T first, Args... rest) { cerr << first; if (sizeof...(rest)) cerr << ", "; debug_out(rest...); } // Overload for pairs template<typename A, typename B> ostream& operator<<(ostream& os, pair<A,B> p) { return os << "(" << p.first << "," << p.second << ")"; } // Overload for vectors template<typename T> ostream& operator<<(ostream& os, vector<T> v) { os << "["; for (int i = 0; i < (int)v.size(); i++) os << (i ? "," : "") << v[i]; return os << "]"; }

Compile locally with g++ -DLOCAL -O2 sol.cpp to enable debug output. On the judge, LOCAL is not defined, so dbg compiles to nothing.

// Usage in your solution int n = 5; vector<int> a = {3, 1, 4, 1, 5}; dbg(n, a); // Output: [n, a]: 5, [3,1,4,1,5] pair<int,int> p = {42, 17}; dbg(p, n * 2); // Output: [p, n * 2]: (42,17), 10
Contest tip: Never use cout for debug output—it might interfere with your solution's output. Always use cerr. And always wrap debug printing in the LOCAL macro to avoid accidental output on the judge.

Common Bugs Checklist

Before submitting, mentally check for these common bugs. They account for the vast majority of Wrong Answer and Runtime Error verdicts.

Integer Overflow

// Bug: int overflow when multiplying int a = 100000, b = 100000; int c = a * b; // OVERFLOW! result > 2^31 // Fix: cast to long long long long c = 1LL * a * b;

Array Index Out of Bounds

// Bug: accessing a[-1] or a[n] for (int i = 0; i <= n; i++) // should be i < n! sum += a[i]; // Bug: 1-indexed array not allocated properly vector<int> a(n); // indices 0..n-1 a[n] = 5; // OUT OF BOUNDS

Uninitialized Variables

// Bug: global arrays with multiple test cases int dp[1005][1005]; // not reset between test cases! // Fix: explicitly clear memset(dp, 0, sizeof(dp)); // Or use fill_n / assign

Off-by-One Errors

// Binary search: classic off-by-one while (lo <= hi) // vs while (lo < hi) mid = (lo + hi) / 2; // vs mid = lo + (hi - lo) / 2 // String indexing: s.substr(pos, len) vs s.substr(pos, end) // Segment tree: which is [l, r] vs [l, r)?

Full Debugging Workflow

  1. Read the problem statement again (many bugs come from misreading)
  2. Check all the common bugs above
  3. Test on the sample inputs manually
  4. Construct 2–3 edge cases and test
  5. If still failing: run the stress test
  6. If stress test passes but judge fails: your edge cases aren't extreme enough—try larger inputs
Time allocation: In a 2-hour contest, spending 10 minutes on stress testing after solving C is often worth more than spending those 10 minutes trying to solve D. A WA on C costs more rating than not attempting D.
