Template Philosophy

A good competitive programming template saves 2–5 minutes per problem. Over a contest with 5–6 problems, that's 10–30 minutes saved—often the difference between solving one more problem or not. But a template that's too bloated becomes hard to navigate and debug.

The golden rules:

  • Include everything you use in >50% of problems. If you only use it once a month, leave it in a separate snippet file.
  • Macros should save keystrokes without hiding logic. pb for push_back is fine. A macro that silently changes loop semantics is dangerous.
  • Debug output should be zero-cost in submission. Use #ifdef LOCAL guards.
  • Know your template by heart. You should be able to type it from memory in under 2 minutes.
Anti-pattern: Some people use templates with 200+ lines including segment tree, FFT, and max-flow implementations. During a contest, this slows you down. Keep the template minimal and copy-paste specific algorithms as needed.

The Core Template

Here's a complete, battle-tested template used by many competitive programmers rated 2000+ on Codeforces:

#include <bits/stdc++.h> using namespace std; // ─── Type aliases ─────────────────────────────────── using ll = long long; using ull = unsigned long long; using ld = long double; using pii = pair<int, int>; using pll = pair<ll, ll>; using vi = vector<int>; using vll = vector<ll>; using vvi = vector<vi>; using vpii = vector<pii>; // ─── Macros ───────────────────────────────────────── #define pb push_back #define eb emplace_back #define all(x) (x).begin(), (x).end() #define rall(x) (x).rbegin(), (x).rend() #define sz(x) (int)(x).size() #define each(x, a) for (auto& x : a) #define rep(i, a, b) for (int i = (a); i < (b); i++) #define per(i, a, b) for (int i = (b) - 1; i >= (a); i--) // ─── Constants ────────────────────────────────────── const int MOD = 1e9 + 7; const int INF = 1e9 + 5; const ll LINF = 1e18 + 5; const ld EPS = 1e-9; const ld PI = acos((ld)-1); // ─── Debug (only active locally) ──────────────────── #ifdef LOCAL #define dbg(...) cerr << "[" << #__VA_ARGS__ << "]: ", \ _dbg(__VA_ARGS__) #else #define dbg(...) #endif void _dbg() { cerr << '\n'; } template<typename T, typename... A> void _dbg(T x, A... a) { cerr << x; if(sizeof...(a)) cerr << ", "; _dbg(a...); } template<typename A, typename B> ostream& operator<<(ostream& o, pair<A,B> p) { return o << "(" << p.first << "," << p.second << ")"; } template<typename T> ostream& operator<<(ostream& o, vector<T> v) { o << "["; rep(i,0,sz(v)) o << (i?",":"") << v[i]; return o << "]"; } // ─── Solve ────────────────────────────────────────── void solve() { // your code here } int main() { ios_base::sync_with_stdio(false); cin.tie(nullptr); int t = 1; cin >> t; while (t--) solve(); return 0; }
Single test case? Comment out cin >> t; and the while loop, or just set t = 1. Having the multi-test-case loop by default saves time since most Codeforces problems now use multiple test cases.

Essential Macros

Let's break down what each macro does and when to use it.

Container Macros

// all(x) — pass entire container to STL algorithms sort(all(v)); // instead of sort(v.begin(), v.end()) reverse(all(v)); auto it = lower_bound(all(v), x); int uniq = unique(all(v)) - v.begin(); // rall(x) — reverse iterators sort(rall(v)); // sort descending // sz(x) — size as int (avoids signed/unsigned warnings) rep(i, 0, sz(v)) cout << v[i] << ' ';

Loop Macros

// rep(i, a, b) — for i from a to b-1 rep(i, 0, n) // equivalent to: for (int i = 0; i < n; i++) rep(i, 1, n+1) // 1-indexed: for (int i = 1; i <= n; i++) // per(i, a, b) — for i from b-1 down to a per(i, 0, n) // equivalent to: for (int i = n-1; i >= 0; i--) // each(x, a) — range-based for with reference each(x, v) cout << x << ' '; // prints all elements each(x, v) x *= 2; // doubles all elements in-place

Input/Output Helpers

// Read a vector in one shot template<typename T> void read(vector<T>& v) { each(x, v) cin >> x; } // Print a vector with separator template<typename T> void print(vector<T>& v, char sep = ' ') { rep(i, 0, sz(v)) cout << v[i] << (i + 1 < sz(v) ? sep : '\n'); } // Usage: vi a(n); read(a); sort(all(a)); print(a);
Macro danger zone: Avoid macros that change control flow or have side effects. For example, #define min(a,b) ((a)<(b)?(a):(b)) is dangerous because it evaluates arguments twice. Use std::min instead. Keep macros simple and predictable.

Debug Printing System

A comprehensive debug system that handles all common types: pairs, vectors, sets, maps, and nested containers.

#ifdef LOCAL #include <sstream> // Base case printer for known types template<typename T> string to_str(T x) { ostringstream os; os << x; return os.str(); } // Pair template<typename A, typename B> string to_str(pair<A,B> p) { return "(" + to_str(p.first) + "," + to_str(p.second) + ")"; } // Iterable container (vector, set, multiset, deque) template<typename T> string to_str(vector<T> v) { string s = "["; rep(i, 0, sz(v)) s += (i ? "," : "") + to_str(v[i]); return s + "]"; } template<typename T> string to_str(set<T> s) { string r = "{"; bool first = true; each(x, s) { if (!first) r += ","; r += to_str(x); first = false; } return r + "}"; } // Map template<typename K, typename V> string to_str(map<K,V> m) { string r = "{"; bool first = true; each(p, m) { if (!first) r += ", "; r += to_str(p.first) + ":" + to_str(p.second); first = false; } return r + "}"; } // 2D vector template<typename T> string to_str(vector<vector<T>> v) { string s = "[\n"; each(row, v) s += " " + to_str(row) + "\n"; return s + "]"; } #define dbg(...) cerr << "\033[33m[" << #__VA_ARGS__ << "]\033[0m = ", \ _print(__VA_ARGS__) void _print() { cerr << '\n'; } template<typename T, typename... A> void _print(T x, A... a) { cerr << to_str(x); if (sizeof...(a)) cerr << ", "; _print(a...); } #else #define dbg(...) #endif

This handles nested structures gracefully:

// Example outputs: vector<vector<int>> grid = {{1,2},{3,4}}; dbg(grid); // [grid] = [ // [1,2] // [3,4] // ] map<string, vi> m = {{"a",{1,2}}, {"b",{3}}}; dbg(m); // [m] = {a:[1,2], b:[3]}

Timing & Profiling

During practice, knowing how fast your solution runs helps you decide if an optimization is needed before submitting.

// Timer class for profiling struct Timer { chrono::high_resolution_clock::time_point start; Timer() : start(chrono::high_resolution_clock::now()) {} ~Timer() { auto end = chrono::high_resolution_clock::now(); auto ms = chrono::duration_cast<chrono::milliseconds>(end - start).count(); cerr << "Time: " << ms << "ms\n"; } }; // Usage: add at the beginning of main() int main() { #ifdef LOCAL Timer _timer; // prints elapsed time when main exits #endif // ... your solution ... }

Per-Section Timing

// Time specific sections of your code #ifdef LOCAL #define TIME_START(name) auto _t_##name = chrono::high_resolution_clock::now() #define TIME_END(name) cerr << #name << ": " << \ chrono::duration_cast<chrono::milliseconds>( \ chrono::high_resolution_clock::now() - _t_##name).count() << "ms\n" #else #define TIME_START(name) #define TIME_END(name) #endif // Usage: TIME_START(sorting); sort(all(v)); TIME_END(sorting); // "sorting: 12ms" TIME_START(dp); // ... DP computation ... TIME_END(dp); // "dp: 450ms"
Rule of thumb: On Codeforces, C++ does ~3–5 × 108 simple operations per second. If your algorithm is O(n²) with n = 105, that's 1010 operations—way too slow. Use the timer to confirm before trying to optimize.

Compiler Flags

# Local compilation (debug mode) g++ -std=c++17 -DLOCAL -O2 -Wall -Wextra -Wshadow \ -fsanitize=address,undefined -fno-sanitize-recover=all \ -o sol sol.cpp # What each flag does: # -DLOCAL → enables debug macros # -O2 → optimization (close to judge speed) # -Wall -Wextra → extra warnings catch common bugs # -Wshadow → warns when variable shadows another # -fsanitize=... → catches memory errors and UB at runtime # Competition mode (clean submission) g++ -std=c++17 -O2 -o sol sol.cpp
Always use sanitizers locally. -fsanitize=address catches out-of-bounds access, use-after-free, and memory leaks. -fsanitize=undefined catches integer overflow, null dereference, and other UB. These find bugs that would otherwise be invisible.

Code Snippets Library

Beyond the template, maintain a library of tested code snippets for common algorithms. Organize them by category and keep them in a searchable format.

Recommended Snippet Categories

Data Structures

  • Segment tree (point/range update)
  • Lazy propagation segment tree
  • Fenwick tree (BIT)
  • Disjoint Set Union (DSU)
  • Sparse table (RMQ)
  • Ordered set (pb_ds)

Algorithms

  • Dijkstra / Bellman-Ford
  • KMP / Z-function
  • FFT / NTT
  • Convex hull trick
  • HLD / Centroid decomposition
  • Max flow (Dinic's)

Snippet Organization

// Suggested directory structure: cp-library/ ├── data-structures/ │ ├── segtree.cpp │ ├── lazy-segtree.cpp │ ├── fenwick.cpp │ ├── dsu.cpp │ └── sparse-table.cpp ├── graphs/ │ ├── dijkstra.cpp │ ├── bellman-ford.cpp │ ├── hld.cpp │ └── max-flow.cpp ├── math/ │ ├── mint.cpp │ ├── ntt.cpp │ ├── sieve.cpp │ └── matrix-expo.cpp ├── strings/ │ ├── kmp.cpp │ ├── z-func.cpp │ ├── suffix-array.cpp │ └── aho-corasick.cpp └── template.cpp

Each snippet should be self-contained, well-tested, and include a usage example as a comment. Before a contest, review which snippets you might need based on the problem tags.

Editor setup: Configure your editor (VS Code, Vim, Sublime) with snippets that auto-expand. For example, typing segtree + Tab should insert your full segment tree implementation. This saves 3–5 minutes per algorithm insertion.

Compile Script

Create a one-command compile-and-run workflow:

#!/bin/bash # run.sh — compile with debug flags and run set -e g++ -std=c++17 -DLOCAL -O2 -Wall -Wextra -Wshadow \ -fsanitize=address,undefined \ -o sol "$1" if [ -f input.txt ]; then ./sol < input.txt else ./sol fi # Usage: ./run.sh sol.cpp
