I/O Optimization for Competitive Programming

CP Math Series CP Foundations

Why I/O Speed Matters

In competitive programming, time limits are typically 1–2 seconds. When a problem has 106 or more integers to read, naive I/O can consume 30–50% of your time budget. Many beginners get TLE (Time Limit Exceeded) not because their algorithm is wrong, but because their I/O is slow.

The root cause is synchronization overhead. By default, C++ synchronizes its iostream library with C's stdio library, adding a mutex lock on every I/O operation. Additionally, cin is tied to cout, forcing a flush before every read. These two defaults alone can make cin/cout 5–10× slower than necessary.

Real example: A problem reading 106 integers with default cin takes ~0.9s. The same input with sync_with_stdio(false) takes ~0.15s. With a custom reader: ~0.05s. That's an 18× difference.

sync_with_stdio & cin.tie

The single most important I/O optimization in C++ competitive programming is these two lines:

#include <bits/stdc++.h> using namespace std; int main() { ios_base::sync_with_stdio(false); // disable C/C++ sync cin.tie(nullptr); // untie cin from cout int n; cin >> n; // ... rest of solution return 0; }

What does sync_with_stdio(false) do?

By default, C++ iostream operations are synchronized with C stdio operations (printf, scanf, etc.). This means every cin/cout call acquires a lock to ensure interleaved C and C++ I/O produces correct output. Disabling this removes the lock overhead but means you cannot mix cin/cout with scanf/printf in the same program.

What does cin.tie(nullptr) do?

By default, cin is tied to cout, meaning cout.flush() is called before every cin operation. This is useful for interactive programs where you need output to appear before reading input, but in CP batch problems, it's pure overhead. Setting cin.tie(nullptr) removes this automatic flushing.

Rule of thumb: Always include both lines in your template. The only exception is interactive problems where the judge reads your output in real-time—there you might need to keep cin.tie or manually flush with cout.flush() or endl.

Common Pitfall: '\n' vs endl

endl does two things: outputs a newline and flushes the buffer. In CP, flushing after every line is expensive. Always use '\n' instead of endl:

❌ Slow

for (int i = 0; i < n; i++) cout << a[i] << endl; // flushes buffer n times!

✅ Fast

for (int i = 0; i < n; i++) cout << a[i] << '\n'; // single flush at program exit

scanf/printf vs cin/cout

Before sync_with_stdio(false) was widely known, competitive programmers used C-style scanf/printf exclusively. These are still useful in certain scenarios.

#include <cstdio> int main() { int n; scanf("%d", &n); int a[n]; for (int i = 0; i < n; i++) scanf("%d", &a[i]); long long sum = 0; for (int i = 0; i < n; i++) sum += a[i]; printf("%lld\n", sum); return 0; }

When to use scanf/printf

  • Formatted output: printf("%.10f\n", ans) is cleaner than cout << fixed << setprecision(10) << ans
  • Compatibility: Some online judges have older compilers where sync_with_stdio doesn't work perfectly
  • Parsing: scanf("%d:%d", &h, &m) can parse formatted input directly
Never mix: If you use sync_with_stdio(false), do NOT mix cin/cout with scanf/printf. The behavior is undefined and will cause wrong answers on some judges.

Custom Fast Reader

For problems with extremely large input (107+ integers), even optimized cin may not be fast enough. A custom reader using getchar_unlocked (Linux) or getchar can be 3–5× faster than scanf. The idea: read character by character and parse integers manually.

// Ultra-fast integer reader inline int readInt() { int x = 0, neg = 0; char c = getchar_unlocked(); while (c < '0' || c > '9') { if (c == '-') neg = 1; c = getchar_unlocked(); } while (c >= '0' && c <= '9') { x = x * 10 + (c - '0'); c = getchar_unlocked(); } return neg ? -x : x; } // Usage int n = readInt(); for (int i = 0; i < n; i++) a[i] = readInt();

Buffer-based Reader (Even Faster)

For maximum speed, read the entire input at once into a buffer, then parse from memory. This eliminates system call overhead entirely:

const int BUFSIZE = 1 << 22; // 4MB buffer char buf[BUFSIZE]; int bufPos = 0, bufLen = 0; inline char gc() { if (bufPos == bufLen) { bufLen = fread(buf, 1, BUFSIZE, stdin); bufPos = 0; } return buf[bufPos++]; } inline int readInt() { int x = 0, neg = 0; char c = gc(); while (c < '0' || c > '9') { if (c == '-') neg = 1; c = gc(); } while (c >= '0' && c <= '9') { x = x * 10 + (c - '0'); c = gc(); } return neg ? -x : x; }
When to use custom readers: Only for problems where TLE is tight and input is massive (107+ numbers). For 99% of problems, sync_with_stdio(false) + cin.tie(nullptr) is sufficient. Don't over-engineer your I/O unless you need to.

Reading Strings and Lines

Custom readers become trickier with strings. For line-based input, prefer getline with sync off:

ios_base::sync_with_stdio(false); cin.tie(nullptr); int n; cin >> n; cin.ignore(); // consume the newline after the integer string line; while (n--) { getline(cin, line); // process line }
Common bug: Forgetting cin.ignore() after reading an integer before getline. The leftover newline from cin >> n gets consumed by the first getline, giving you an empty string. This causes WA (Wrong Answer) on countless submissions.

Output Optimization

Output is usually less of a bottleneck than input, but large outputs can still be slow. Here are key techniques:

putchar_unlocked for Characters

// Fast integer output inline void writeInt(int x) { if (x < 0) { putchar_unlocked('-'); x = -x; } if (x > 9) writeInt(x / 10); putchar_unlocked('0' + x % 10); } // Usage writeInt(ans); putchar_unlocked('\n');

Output Buffering with puts

For problems requiring massive output, build a string buffer and output at once:

char outBuf[1 << 22]; int outPos = 0; inline void writeChar(char c) { outBuf[outPos++] = c; } inline void writeInt(int x) { char tmp[12]; int len = 0; if (x < 0) { writeChar('-'); x = -x; } do { tmp[len++] = '0' + x % 10; x /= 10; } while (x); while (len--) writeChar(tmp[len]); } inline void flushOut() { fwrite(outBuf, 1, outPos, stdout); }

Floating Point Output

For problems requiring decimal output, avoid cout << fixed << setprecision(10) in tight loops. Use printf("%.10f\n", x) or build your own formatter:

// Clean floating-point output double ans = 3.141592653589793; printf("%.10f\n", ans); // Or with cout (set once at top) cout << fixed << setprecision(10); cout << ans << '\n';

Practical recommendation: For 99% of CP problems, this is all you need:

ios_base::sync_with_stdio(false); cin.tie(nullptr);

Add the custom fread reader to your template as a fallback for the rare problem that needs it. Don't waste contest time implementing it from scratch.

Interactive Problem Caveat

For interactive problems, you must flush output after each query:

// Interactive problem pattern int query(int x) { cout << "? " << x << endl; // endl forces flush int res; cin >> res; return res; } // Or equivalently: cout << "? " << x << '\n'; cout.flush(); // explicit flush

In interactive problems, forgetting to flush is the #1 cause of "Idleness Limit Exceeded" verdicts. When in doubt, use endl for interactive problems—the small performance cost is irrelevant when you're making O(log n) queries.
