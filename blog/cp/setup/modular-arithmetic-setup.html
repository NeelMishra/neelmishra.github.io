Modular Arithmetic Setup

CP Math Series CP Foundations

Why Modular Arithmetic

A huge number of competitive programming problems end with "output the answer modulo 109+7" or "modulo 998244353." This isn't arbitrary—these numbers are chosen because they're prime and fit comfortably in a 32-bit integer, while their squares fit in a 64-bit integer.

Why these specific primes?

  • 109+7 (1000000007): The most common modulus. It's prime, so modular inverses exist for all non-zero elements. (109+7)² ≈ 1018 < 263, so a*b mod p doesn't overflow long long.
  • 998244353: Equal to 119 × 223 + 1. This specific form allows NTT (Number Theoretic Transform) up to size 223, making it ideal for polynomial multiplication problems.
Key property: Since p is prime, Fermat's little theorem gives us ap−1 ≡ 1 (mod p), so a−1 ≡ ap−2 (mod p). This is how we compute modular inverses.

Overflow Prevention

The most dangerous operation in modular arithmetic is multiplication. If a, b < 109+7, then a*b can be up to ~1018, which fits in long long. But if you're not careful, intermediate computations can overflow.

The __int128 Approach

For cases where even long long might overflow (e.g., when the modulus is close to 262, or when multiplying three numbers), use __int128:

// Safe multiplication for any modulus up to 2^62 long long mulmod(long long a, long long b, long long m) { return (__int128)a * b % m; } // Example: computing a*b*c mod m safely long long a = 1e18, b = 1e18, c = 1e18; long long m = 1e18 + 9; long long result = mulmod(mulmod(a, b, m), c, m);

When __int128 Isn't Available

On some judges (rare), __int128 isn't supported. Alternatives:

// Method 1: Binary multiplication (Russian peasant) long long mulmod_slow(long long a, long long b, long long m) { a %= m; b %= m; long long res = 0; while (b > 0) { if (b & 1) res = (res + a) % m; a = (a + a) % m; b >>= 1; } return res; } // Method 2: long double trick (faster but less precise) long long mulmod_ld(long long a, long long b, long long m) { long long q = (long long)((long double)a * b / m); long long r = a * b - q * m; if (r < 0) r += m; if (r >= m) r -= m; return r; }
Subtraction trap: (a - b) % m can produce negative results in C++! Always use ((a - b) % m + m) % m or ensure a ≥ b before subtracting.

The Mint Template Class

Writing ((a % MOD) + (b % MOD)) % MOD everywhere is error-prone and ugly. The standard competitive programming solution is a modular integer class (often called "Mint" or "ModInt") that handles mod operations automatically through operator overloading.

template<int MOD> struct Mint { int v; Mint() : v(0) {} Mint(long long x) : v((x % MOD + MOD) % MOD) {} Mint operator+(const Mint& o) const { int r = v + o.v; return {r >= MOD ? r - MOD : r}; } Mint operator-(const Mint& o) const { int r = v - o.v; return {r < 0 ? r + MOD : r}; } Mint operator*(const Mint& o) const { return {(long long)v * o.v % MOD}; } Mint power(long long e) const { Mint res = 1, base = *this; while (e > 0) { if (e & 1) res = res * base; base = base * base; e >>= 1; } return res; } Mint inv() const { return power(MOD - 2); } Mint operator/(const Mint& o) const { return *this * o.inv(); } Mint& operator+=(const Mint& o) { return *this = *this + o; } Mint& operator-=(const Mint& o) { return *this = *this - o; } Mint& operator*=(const Mint& o) { return *this = *this * o; } Mint& operator/=(const Mint& o) { return *this = *this / o; } bool operator==(const Mint& o) const { return v == o.v; } bool operator!=(const Mint& o) const { return v != o.v; } friend ostream& operator<<(ostream& os, const Mint& m) { return os << m.v; } friend istream& operator>>(istream& is, Mint& m) { long long x; is >> x; m = Mint(x); return is; } }; using mint = Mint<1000000007>;

Now your code reads naturally:

mint a = 5, b = 3; mint c = a * b + a / b - mint(2).power(100); cout << c << '\n'; // automatically reduced mod 10^9+7
Template tip: Keep this Mint class in your template. For problems with mod 998244353, just change the using line: using mint = Mint<998244353>;

Modular Inverse

Division in modular arithmetic is multiplication by the modular inverse. There are multiple ways to compute a−1 mod p:

Method 1: Fermat's Little Theorem

When p is prime: a−1 ≡ ap−2 (mod p). Computed via fast exponentiation in O(log p). This is what our Mint class uses.

Method 2: Extended Euclidean Algorithm

Works for any modulus (not just prime). Finds x, y such that ax + my = gcd(a, m). If gcd(a, m) = 1, then x is the inverse.

long long extgcd(long long a, long long b, long long& x, long long& y) { if (b == 0) { x = 1; y = 0; return a; } long long x1, y1; long long g = extgcd(b, a % b, x1, y1); x = y1; y = x1 - (a / b) * y1; return g; } long long modinv(long long a, long long m) { long long x, y; long long g = extgcd(a, m, x, y); // g must be 1 for inverse to exist return (x % m + m) % m; }

Method 3: Batch Inverse Computation

Computing inverses of 1, 2, ..., n in O(n) using the recurrence:

// Compute inv[1..n] where inv[i] = i^(-1) mod p const int MOD = 1000000007; vector<long long> inv(n + 1); inv[1] = 1; for (int i = 2; i <= n; i++) inv[i] = -(MOD / i) * inv[MOD % i] % MOD + MOD; // Proof: Let p = q*i + r where q = p/i, r = p%i // Then q*i + r ≡ 0 (mod p) // So i^(-1) ≡ -q * r^(-1) (mod p) // i.e., inv[i] = -(p/i) * inv[p%i] mod p
When to use which: Use Fermat for one-off inverses. Use extended GCD when modulus isn't prime. Use batch computation when you need all inverses from 1 to n (e.g., for computing all factorials' inverses).

Combinatorics Utilities

Many CP problems require computing C(n, k) mod p. Here's the complete setup:

const int MAXN = 2000005; mint fact[MAXN], ifact[MAXN]; void precompute() { fact[0] = 1; for (int i = 1; i < MAXN; i++) fact[i] = fact[i - 1] * i; ifact[MAXN - 1] = fact[MAXN - 1].inv(); for (int i = MAXN - 2; i >= 0; i--) ifact[i] = ifact[i + 1] * (i + 1); } mint C(int n, int k) { if (k < 0 || k > n) return 0; return fact[n] * ifact[k] * ifact[n - k]; } mint P(int n, int k) { if (k < 0 || k > n) return 0; return fact[n] * ifact[n - k]; } // Catalan number: C_n = C(2n,n) / (n+1) mint catalan(int n) { return C(2 * n, n) / (n + 1); }

Key optimization: We only compute one modular inverse (for (MAXN-1)!) and derive all others by multiplication: ifact[i] = ifact[i+1] × (i+1). This is O(n) + O(log p) instead of O(n log p).

Common Formulas

  • C(n,k) = n! / (k!(n-k)!)
  • Catalan(n) = C(2n,n)/(n+1)
  • Stars & bars: C(n+k-1, k-1)
  • Derangements: n! × Σ(-1)^k/k!

Typical Problems

  • Grid paths: C(n+m-2, n-1)
  • Balanced parentheses: Catalan(n)
  • Distributing balls into boxes
  • Inclusion-exclusion counts

Common Mod Bugs

These bugs cause Wrong Answer in thousands of CP submissions. Memorize them.

Bug 1: Negative Modular Results

// BUG: C++ % can return negative for negative operands int a = -7, m = 5; cout << a % m; // outputs -2, NOT 3! // FIX: always normalize int result = ((a % m) + m) % m; // correctly gives 3

Bug 2: Overflow Before Mod

// BUG: a*b overflows before % is applied int a = 1000000006, b = 1000000006; int MOD = 1000000007; cout << (a * b) % MOD; // OVERFLOW! int can't hold a*b // FIX: cast to long long cout << (1LL * a * b) % MOD; // correct

Bug 3: Forgetting to Mod Intermediate Results

// BUG: sum grows without bound long long sum = 0; for (int i = 0; i < n; i++) sum += a[i] * b[i]; // can overflow even with long long // FIX: mod at every step long long sum = 0; for (int i = 0; i < n; i++) sum = (sum + 1LL * a[i] % MOD * b[i] % MOD) % MOD;

Bug 4: Division Without Inverse

// BUG: regular division doesn't work in modular arithmetic long long ans = (a * b / c) % MOD; // WRONG! // FIX: multiply by modular inverse of c long long ans = (a % MOD * b % MOD) % MOD * power(c, MOD - 2, MOD) % MOD;
The #1 rule: Never perform integer division when working in modular arithmetic. Always use modular inverse. The Mint class handles this automatically, which is why you should always use it.
Testing tip: When debugging mod problems, test with small inputs where you can compute the exact (non-modular) answer. If the modular answer differs from exact_answer % MOD, you have a mod bug. The Mint class eliminates most of these bugs by design.
