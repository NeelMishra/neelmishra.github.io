Why Proofs Matter in CP

Many competitive programmers treat proof techniques as "academic"—something for math courses, not contests. This is a costly mistake. The difference between a 1600-rated and a 2000-rated programmer is often the ability to prove that a greedy approach works before coding it, rather than hoping it passes.

Proof techniques serve three practical purposes in CP:

  • Validating greedy algorithms: Greedy solutions are easy to code but hard to verify. A proof by exchange argument or contradiction confirms your greedy choice is optimal.
  • Finding the right approach: Trying to prove a wrong approach often reveals why it fails, guiding you to the correct one.
  • Constructing solutions: Many problems ask you to "construct" a valid arrangement. The proof of existence often directly gives you the construction algorithm.
Mental model: Think of proofs as "debugging your algorithm before you code it." A 5-minute proof saves you 30 minutes of debugging a wrong greedy.

Proof by Contradiction

Template: Assume the statement is false. Derive a logical contradiction. Therefore the statement must be true.

This is the most common proof technique in CP, especially for proving greedy algorithms are optimal.

Example: Proving a Greedy Scheduling Algorithm

Problem: Given n intervals [li, ri], select the maximum number of non-overlapping intervals.

Greedy claim: Always pick the interval with the smallest ri (earliest ending time) among those that don't overlap with already selected intervals.

// Greedy: sort by right endpoint, pick greedily sort(intervals.begin(), intervals.end(), [](auto& a, auto& b) { return a.r < b.r; }); int count = 0, last_end = -1; for (auto& [l, r] : intervals) { if (l >= last_end) { count++; last_end = r; } } // count is the maximum number of non-overlapping intervals

Proof by contradiction:

Assume there exists an optimal solution OPT that selects more intervals than our greedy solution G. Let g1, g2, ..., gk be the intervals chosen by G (sorted by end time), and o1, o2, ..., om be those chosen by OPT, with m > k.

Consider the first position i where gi ≠ oi. Since G picks the earliest-ending interval, we know gi.r ≤ oi.r. We can replace oi with gi in OPT without breaking any constraints (since gi ends no later). Repeating this argument, we can transform OPT to match G through the first k intervals. But G stops at k because no more intervals fit—so OPT can't have more than k intervals either. Contradiction with m > k. ∎

Common mistake: Saying "my greedy passed the samples, so it must be correct." Samples are weak tests. Many wrong greedy solutions pass samples but fail on systests. Always prove correctness or stress test against brute force.

Example: √2 is Irrational

This classic proof illustrates the contradiction pattern clearly:

Assume √2 = p/q where p, q are integers with no common factor. Then 2q² = p², so p² is even, so p is even (p = 2k). Then 2q² = 4k², so q² = 2k², so q is also even. But then p and q share factor 2—contradiction with "no common factor." ∎

Proof by Induction

Template: (1) Prove for base case n = 0 or n = 1. (2) Assume true for n = k. (3) Prove true for n = k + 1.

Induction is essential for proving properties of recursive algorithms, tree-based solutions, and divide-and-conquer approaches.

Example: Proving Merge Sort Correctness

Claim: Merge sort correctly sorts any array of n elements.

Base case (n = 1): A single element is already sorted. ✓

Inductive step: Assume merge sort correctly sorts arrays of size ≤ k. For an array of size k + 1, we split into two halves of size ≤ ⌈(k+1)/2⌉ ≤ k. By the inductive hypothesis, each half is correctly sorted. The merge operation correctly combines two sorted arrays into one sorted array (provable by a simple loop invariant). Therefore merge sort correctly sorts arrays of size k + 1. ✓

Strong Induction in CP

Strong induction assumes the statement holds for ALL values ≤ k (not just k) when proving for k + 1. This is particularly useful for:

  • Tree DP: proving a property for a tree of size n by assuming it holds for all subtrees (size < n)
  • Divide and conquer: proving correctness when subproblems have arbitrary sizes < n
  • Game theory: proving Sprague-Grundy values by assuming all smaller game states are correct
// Example: prove that every amount ≥ 12 can be // represented using coins of value 4 and 5 // Base cases: // 12 = 3×4, 13 = 2×4 + 1×5, 14 = 1×4 + 2×5, 15 = 3×5 // Inductive step (strong induction): // Assume true for all values from 12 to k (where k ≥ 15). // For k+1: since k+1 ≥ 16, we have k+1-4 ≥ 12. // By hypothesis, k+1-4 is representable. // Add one more coin of value 4 → k+1 is representable. ∎
CP insight: When a problem says "prove that a solution always exists," strong induction on the problem size is usually the way. The inductive step often directly suggests a constructive algorithm.

Loop Invariants

A loop invariant is a property that is true before and after every iteration of a loop. It's the key technique for proving iterative algorithms correct.

The three parts of a loop invariant proof:

  1. Initialization: The invariant is true before the first iteration.
  2. Maintenance: If the invariant is true before iteration i, it's true after iteration i.
  3. Termination: When the loop ends, the invariant (plus the exit condition) implies correctness.

Example: Binary Search Invariant

// Find the leftmost position where a[pos] >= target int lo = 0, hi = n; // answer is in [lo, hi] while (lo < hi) { int mid = lo + (hi - lo) / 2; if (a[mid] < target) lo = mid + 1; else hi = mid; } // lo == hi == answer // INVARIANT: a[lo-1] < target and a[hi] >= target // (with boundary conventions a[-1] = -∞, a[n] = +∞) // // Init: lo=0, hi=n → a[-1] = -∞ < target ✓, a[n] = +∞ ≥ target ✓ // Maintenance: // If a[mid] < target: set lo = mid+1. // a[lo-1] = a[mid] < target ✓ // If a[mid] >= target: set hi = mid. // a[hi] = a[mid] >= target ✓ // Termination: lo == hi, so a[lo-1] < target ≤ a[lo] → lo is the answer ✓

Invariants in Two-Pointer Technique

The two-pointer technique works because of carefully maintained invariants. For example, finding two elements that sum to a target in a sorted array:

int l = 0, r = n - 1; while (l < r) { int s = a[l] + a[r]; if (s == target) { /* found */ break; } else if (s < target) l++; else r--; } // INVARIANT: For any pair (i,j) with i < l or j > r, // we have already ruled out a[i]+a[j] != target. // Proof: When we increment l, a[l-1]+a[r'] < target // for all r' ≤ r (since array is sorted, a[r'] ≤ a[r]). // So all pairs involving l-1 with any remaining r' are too small. // Similar argument for decrementing r.
Debugging with invariants: If your two-pointer solution gives wrong answers, write down the invariant and check if each branch of your if/else maintains it. The bug is almost always a branch that violates the invariant.

Constructive Proofs

A constructive proof shows that something exists by building it. In CP, this means the proof directly gives you the algorithm. Many problems tagged "constructive algorithms" on Codeforces require this mindset.

Example: Graph Coloring

Claim: Every graph with maximum degree Δ can be properly (Δ+1)-colored.

Constructive proof: Process vertices in any order. For each vertex v, it has at most Δ neighbors, so at most Δ colors are "forbidden." Since we have Δ+1 colors available, there's always at least one valid color for v. This greedy algorithm IS the proof. ✓

// Constructive (Δ+1)-coloring vector<int> color(n, -1); for (int v = 0; v < n; v++) { set<int> used; for (int u : adj[v]) if (color[u] != -1) used.insert(color[u]); // Find smallest available color for (int c = 0; ; c++) { if (!used.count(c)) { color[v] = c; break; } } }

Example: Euler's Theorem Construction

Claim: A connected graph has an Eulerian circuit iff every vertex has even degree.

The constructive proof (Hierholzer's algorithm) builds the circuit: start anywhere, follow edges (removing them), and when stuck, you must be back at the start (by degree parity). Then find any unvisited edge adjacent to the circuit and extend. The proof of existence IS the algorithm.

Pattern recognition: When a CP problem says "output any valid answer" or "construct a sequence satisfying...", think: what existence proof gives me a construction? The proof technique often maps directly to an O(n) or O(n log n) algorithm.

The Exchange Argument

The exchange argument is the gold standard for proving greedy algorithms optimal. The idea: take any optimal solution and show you can transform it into your greedy solution without making it worse.

Template

  1. Let OPT be any optimal solution and G be your greedy solution.
  2. If OPT ≠ G, find a specific difference (an "exchange point").
  3. Show that swapping to match G at that point doesn't make OPT worse.
  4. Repeat until OPT = G, proving G is also optimal.

Example: Minimum Coins (Canonical Systems)

Problem: Given US coins {25, 10, 5, 1}, make change for amount N using fewest coins.

Greedy: Always use the largest coin that fits.

Exchange argument: Suppose OPT uses fewer 25¢ coins than greedy. Then OPT must represent the "missing" 25¢ using smaller coins: at most 2×10 + 1×5 = 25, which uses 3 coins instead of 1. So OPT uses ≥2 more coins. Replacing those 3 coins with one 25¢ coin reduces the count. Contradiction with OPT being optimal (or shows OPT must use same number of 25¢ coins as greedy). Apply similar argument for 10¢ and 5¢. ∎

When exchange fails: Not all coin systems are "canonical." For coins {6, 4, 1}, making 8: greedy gives 6+1+1 (3 coins), but 4+4 (2 coins) is better. The exchange argument fails here because you can't always improve by using the larger coin. Always verify your greedy before assuming it works!

Exchange Argument for Job Scheduling

Problem: Schedule jobs to minimize weighted completion time. Job i takes time ti and has weight wi.

Greedy: Sort by wi/ti in decreasing order.

Exchange proof: Consider two adjacent jobs i, j in optimal schedule with wi/ti < wj/tj (violating greedy order). Swapping them:

  • Job j finishes earlier by ti, saving wj·ti
  • Job i finishes later by tj, costing wi·tj
  • Net change: wj·ti − wi·tj > 0 (since wj/tj > wi/ti)

So the swap improves the solution—contradiction with OPT being optimal. ∎

Key takeaway: Before coding any greedy, spend 3–5 minutes trying an exchange argument. If you can't prove it, the greedy is probably wrong. Fall back to DP or another approach.
