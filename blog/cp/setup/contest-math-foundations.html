Floor & Ceiling Identities

The floor function ⌊x⌋ and ceiling function ⌈x⌉ appear constantly in CP—whenever you divide integers, allocate items into groups, or compute block decompositions. Mastering their identities saves you from off-by-one errors.

Core Identities

// Floor and ceiling of integer division // ⌊a/b⌋ = a / b (C++ integer division for a,b > 0) // ⌈a/b⌉ = ⌊(a + b - 1) / b⌋ = (a + b - 1) / b // Key identities: // ⌊a/b⌋ + ⌈a/b⌉ = ⌊a/b⌋ + ⌊a/b⌋ + (a%b != 0) (*) // ⌈a/b⌉ = ⌊a/b⌋ + 1 if b does not divide a // ⌈a/b⌉ = ⌊a/b⌋ if b divides a // Negative division in C++: truncates toward zero // -7 / 2 = -3 (not -4, which is ⌊-3.5⌋) // For true floor division with negative numbers: long long floordiv(long long a, long long b) { return a / b - (a % b != 0 && (a ^ b) < 0); } long long ceildiv(long long a, long long b) { return a / b + (a % b != 0 && (a ^ b) > 0); }
C++ division pitfall: -7 / 2 = -3 in C++, but mathematically ⌊−7/2⌋ = −4. If your problem involves negative values, always use the floordiv function above. This is a common source of WA in problems involving coordinate geometry or modular arithmetic with negative numbers.

Block Decomposition

Dividing n items into groups of size k:

// Number of complete blocks: ⌊n/k⌋ // Total blocks needed: ⌈n/k⌉ = (n + k - 1) / k // Size of last (possibly incomplete) block: n - k * ⌊n/k⌋ = n % k // (or k if n is divisible by k) // Example: partition array of size n into blocks of size B int B = (int)sqrt(n); // typical for sqrt decomposition int num_blocks = (n + B - 1) / B; for (int block = 0; block < num_blocks; block++) { int lo = block * B; int hi = min(lo + B, n); // exclusive // process elements [lo, hi) }

Floor Division Value Grouping

A crucial identity: ⌊n/i⌋ takes at most O(√n) distinct values as i ranges from 1 to n. This is the foundation of many number theory optimizations.

// Enumerate all distinct values of ⌊n/i⌋ for i = 1..n // For each value v = ⌊n/i⌋, find the range [lo, hi] where // ⌊n/j⌋ = v for all j in [lo, hi] long long n; for (long long lo = 1, hi; lo <= n; lo = hi + 1) { long long val = n / lo; // ⌊n/lo⌋ hi = n / val; // largest i where ⌊n/i⌋ = val // val = ⌊n/i⌋ for all i in [lo, hi] // Process this group: contributes val * (hi - lo + 1) } // Total iterations: O(√n)
Applications: This floor grouping technique is used in computing the sum Σ⌊n/i⌋ for divisor-counting problems, Euler's totient summation, and many Dirichlet series problems. It reduces O(n) computation to O(√n).

Harmonic Series & Divisor Sums

The harmonic series Hn = 1 + 1/2 + 1/3 + ... + 1/n = Θ(ln n) appears everywhere in CP algorithm analysis. Understanding it is essential for proving time complexity.

Why Harmonic Numbers Matter

Consider this common pattern: iterating over all multiples of each number:

// Sieve of Eratosthenes — mark multiples for (int i = 2; i <= n; i++) for (int j = i; j <= n; j += i) // do something with j (a multiple of i) // Total iterations = n/1 + n/2 + n/3 + ... + n/n // = n × (1 + 1/2 + 1/3 + ... + 1/n) // = n × H_n // = O(n log n)

This O(n log n) complexity is not O(n²)! The harmonic sum converges slowly, giving us a much better bound than the naive analysis of n × n.

Common Sums Involving Harmonic Numbers

// Sum of ⌊n/i⌋ for i = 1 to n // Σ ⌊n/i⌋ = number of pairs (i,j) with 1 ≤ j ≤ n, i | j // = sum of d(j) for j = 1 to n (d = divisor count) // ≈ n ln n + (2γ-1)n + O(√n) where γ ≈ 0.5772 // Sum of d(i) for i = 1 to n (average number of divisors) // Avg d(i) ≈ ln n, so total ≈ n ln n // Number of distinct values of ⌊n/i⌋: exactly O(2√n)

Divisor Counting Trick

Many CP problems ask: for each value, count something across its divisors or multiples. The harmonic sum tells us the total work is O(n log n):

// Count the number of divisors for each number 1..n vector<int> divisor_count(n + 1, 0); for (int i = 1; i <= n; i++) for (int j = i; j <= n; j += i) divisor_count[j]++; // O(n log n) total — NOT O(n²) // Sum of all divisors for each number 1..n vector<long long> divisor_sum(n + 1, 0); for (int i = 1; i <= n; i++) for (int j = i; j <= n; j += i) divisor_sum[j] += i; // Also O(n log n)
Pro insight: Whenever you see a nested loop where the inner loop steps by i, the total work is harmonic: Σ(n/i) = O(n log n). This pattern appears in sieves, GCD-related algorithms, and Möbius function computations.

Geometric Series

The geometric series formula is one of the most used in CP, especially for analyzing algorithms with divide-and-conquer structure.

Core Formula

// Finite geometric series: // 1 + r + r² + ... + r^(n-1) = (r^n - 1) / (r - 1) for r ≠ 1 // Special cases: // r = 2: 1 + 2 + 4 + ... + 2^(n-1) = 2^n - 1 // r = 1/2: 1 + 1/2 + 1/4 + ... = 2 (converges!) // Power of 2 identity (extremely common in CP): // 2^0 + 2^1 + 2^2 + ... + 2^k = 2^(k+1) - 1 // Computing geometric sum mod p: mint geom_sum(mint r, long long n) { // Returns 1 + r + r^2 + ... + r^(n-1) if (n == 0) return 0; if (n == 1) return 1; if (r.v == 1) return n; // special case: r = 1 return (r.power(n) - 1) / (r - 1); }

Applications in Complexity Analysis

Geometric series explain why many divide-and-conquer algorithms are efficient:

// Merge sort: T(n) = 2T(n/2) + O(n) // Work at each level: n (total across all subproblems) // Number of levels: log n // Total: O(n log n) // Segment tree build: T(n) = 2T(n/2) + O(1) // Work at level k: 2^k nodes, O(1) each // Total: 1 + 2 + 4 + ... + n = 2n - 1 = O(n) // (geometric series with r = 2) // Binary lifting precomputation: // For each of n nodes, compute LOG ancestors // Total: O(n log n) time and space

Geometric Series in Probability

Expected number of trials in randomized algorithms often involves geometric series:

// Expected coin flips until heads (p = 1/2): // E = 1·(1/2) + 2·(1/2)² + 3·(1/2)³ + ... = 2 // Expected number of random picks until new element // (coupon collector): H_n = 1 + 1/2 + ... + 1/n ≈ ln n // Birthday paradox: expect collision after O(√n) random picks // from n values. Used in hashing analysis.

Summation Tricks

These summation identities come up repeatedly in CP problems and complexity analysis.

Arithmetic Sums

// 1 + 2 + 3 + ... + n = n(n+1)/2 // 1² + 2² + ... + n² = n(n+1)(2n+1)/6 // 1³ + 2³ + ... + n³ = [n(n+1)/2]² // Sum of first n odd numbers: 1 + 3 + 5 + ... + (2n-1) = n² // Sum of arithmetic progression: n/2 × (first + last) // In code, beware of overflow: long long sum_1_to_n(long long n) { return n * (n + 1) / 2; // one of n, n+1 is even, so no precision loss } long long sum_squares(long long n) { // n(n+1)(2n+1)/6 — need careful ordering to avoid overflow long long a = n, b = n + 1, c = 2 * n + 1; // Exactly one of a,b is even; and one of a,b,c is divisible by 3 if (a % 2 == 0) a /= 2; else b /= 2; if (a % 3 == 0) a /= 3; else if (b % 3 == 0) b /= 3; else c /= 3; return a * b * c; }

Double Counting / Contribution Technique

Instead of computing a sum directly, count each element's contribution:

// Problem: Given array a[], compute Σ Σ |a[i] - a[j]| for all i < j // Naive: O(n²) — too slow for n = 10^5 // Contribution technique: Sort the array, then for a[i]: // - a[i] is subtracted by all j < i: contributes -a[i] × i // - a[i] is added by all j > i: contributes +a[i] × (n-1-i) // Total contribution of a[i] = a[i] × (2i - n + 1) after sorting sort(all(a)); long long ans = 0; for (int i = 0; i < n; i++) ans += 1LL * a[i] * (2 * i - n + 1); // O(n log n) — the sorting dominates

Swapping Summation Order

A powerful technique: change the order of summation to simplify computation.

// Σ_{i=1}^{n} Σ_{j=i}^{n} f(i,j) // = Σ_{j=1}^{n} Σ_{i=1}^{j} f(i,j) (swap order) // Example: Σ_{i=1}^{n} Σ_{j=i}^{n} 1 = Σ_{i=1}^{n} (n - i + 1) // = n + (n-1) + ... + 1 = n(n+1)/2 // Key insight: Σ_{d|n} φ(d) = n (Euler's product identity) // Useful when you need to sum over divisors: swap to sum over multiples // Σ_{i=1}^{n} Σ_{d|i} f(d) = Σ_{d=1}^{n} f(d) × ⌊n/d⌋ // (each d contributes to ⌊n/d⌋ multiples)
When to swap: If the inner sum depends on the outer variable in a complex way but becomes simple when you swap, do it. The classic sign: "for each i, sum over divisors of i" becomes "for each d, iterate over multiples of d" — turning Σ_divisors into harmonic iteration.

Asymptotic Analysis

Knowing the exact complexity of your algorithm determines whether it'll pass within the time limit. Here's how to do quick back-of-envelope analysis.

The 108 Rule

A rough guide: C++ does approximately 108–5×108 simple operations per second on a competitive programming judge. So:

Time Limit: 1 second

  • O(n): n ≤ 108
  • O(n log n): n ≤ 106–107
  • O(n√n): n ≤ 105
  • O(n²): n ≤ 5×103
  • O(n² log n): n ≤ 2×103
  • O(n³): n ≤ 500
  • O(2n): n ≤ 25
  • O(n!): n ≤ 11

Time Limit: 2 seconds

  • O(n): n ≤ 2×108
  • O(n log n): n ≤ 2×107
  • O(n√n): n ≤ 2×105
  • O(n²): n ≤ 104
  • O(n² log n): n ≤ 5×103
  • O(n³): n ≤ 700
  • O(2n): n ≤ 26
  • O(n!): n ≤ 12

Master Theorem (Quick Reference)

For recurrences T(n) = aT(n/b) + O(nc):

// Master Theorem for T(n) = a·T(n/b) + O(n^c): // Let p = log_b(a) // // Case 1: c < p → T(n) = O(n^p) // Case 2: c = p → T(n) = O(n^p · log n) // Case 3: c > p → T(n) = O(n^c) // // Examples: // Merge sort: a=2, b=2, c=1 → p=1, c=p → O(n log n) ✓ // Binary search: a=1, b=2, c=0 → p=0, c=p → O(log n) ✓ // Karatsuba: a=3, b=2, c=1 → p≈1.58, c<p → O(n^1.58) ✓ // Strassen: a=7, b=2, c=2 → p≈2.81, c<p → O(n^2.81) ✓
Master theorem limitations: It doesn't handle all recurrences. For example, T(n) = 2T(n/2) + O(n log n) (used in some tree algorithms) gives O(n log² n), which requires the Akra-Bazzi method or direct analysis.

CP Complexity Patterns

Recognizing complexity patterns helps you identify the right algorithm from the constraints.

Constraint → Algorithm Map

// n ≤ 10 → O(n!), brute force all permutations // n ≤ 20 → O(2^n), bitmask DP / meet in the middle // n ≤ 500 → O(n³), Floyd-Warshall, cubic DP // n ≤ 5000 → O(n²), quadratic DP, all-pairs // n ≤ 10^5 → O(n√n), Mo's algorithm, sqrt decomposition // n ≤ 10^5 → O(n log² n), merge sort tree, persistent // n ≤ 10^6 → O(n log n), sorting, segment tree // n ≤ 10^7 → O(n), linear sieve, linear DP // n ≤ 10^9 → O(√n) or O(log n), math formula, binary search // n ≤ 10^18 → O(log n), matrix exponentiation, binary lifting

Amortized Analysis Patterns

Some algorithms look slow per-operation but are fast amortized:

// Two pointers: Each pointer moves at most n times total // → O(n) amortized even though the loop looks O(n²) // DSU with path compression + union by rank: // O(α(n)) ≈ O(1) amortized per operation // (α = inverse Ackermann, practically ≤ 4) // Stack-based algorithms (next greater element, histogram): // Each element pushed and popped at most once → O(n) total // Splay tree: O(log n) amortized per operation // Individual operations can be O(n), but averaged: O(log n)

Hidden Constants

Big-O hides constant factors. In practice, these matter:

Fast Constants

  • Array access: ~1 ns
  • Integer arithmetic: ~1 ns
  • Bitwise operations: ~1 ns
  • Comparison: ~1 ns

Slow Constants

  • Cache miss: ~100 ns
  • Map lookup: ~100-200 ns
  • Division/modulo: ~5-20 ns
  • Memory allocation: ~100+ ns
Practical wisdom: An O(n log n) algorithm with heavy constant (e.g., using map inside) can be slower than an O(n√n) algorithm with light constant (e.g., using arrays). When in doubt, use the timer from the Template Setup post to measure actual performance.

Armed with these mathematical foundations, you have the tools to analyze any algorithm you encounter in competitive programming. The key is practice: every time you solve a problem, verify your complexity analysis matches the constraints. Over time, the constraint-to-algorithm mapping becomes second nature.
