Stochastic Games & Expected Value

October 2026 · 24 min read

Games with Randomness

A stochastic game is a game where the outcome depends not only on the players' decisions but also on random events — dice rolls, card draws, coin flips, or any probabilistic mechanism. These games sit at the intersection of game theory and probability, and they appear frequently in competitive programming.

The key challenge: a player must choose a strategy before knowing the random outcome. The rational approach is to choose the strategy that optimizes the expected value of the payoff.

Types of Stochastic Games in CP

  • Single-player: You make decisions to maximize your expected score. Examples: optimal stopping, casino games, random walks with choices.
  • Two-player zero-sum: You maximize, your opponent minimizes. Random events add a third "player" — Nature. Examples: dice battles, card games with incomplete information.
  • Cooperative/Team: Rare in CP, but sometimes two players collaborate against randomness.
Key principle: In a stochastic game, the optimal strategy at each decision point is the one that maximizes (or minimizes) the expected payoff over all possible random outcomes from that point forward.

Formal Setup

A stochastic game can be modeled as a tree (or DAG) where each node is one of:

  • MAX node: The maximizing player chooses which child to visit.
  • MIN node: The minimizing player chooses which child to visit.
  • CHANCE node: Nature randomly selects a child with given probabilities.
  • LEAF node: A terminal state with a known payoff.

Optimal Expected Play

The value of a game tree is computed recursively:

function value(node): if node is LEAF: return node.payoff if node is MAX: return max(value(child) for child in node.children) if node is MIN: return min(value(child) for child in node.children) if node is CHANCE: return Σ prob[child] * value(child)

This is the expectimax algorithm — an extension of minimax that handles chance nodes. Unlike minimax, we cannot prune chance nodes with alpha-beta, because we need the exact expected value.

Why Expected Value is the Right Objective

The von Neumann–Morgenstern theorem shows that any rational decision-maker whose preferences satisfy basic consistency axioms will behave as if maximizing expected utility. In CP, the "utility" is usually the score or payoff directly, so maximizing expected value is the standard approach.

Warning: In real life, risk aversion means people don't always maximize expected value. But in CP, always assume risk-neutral play unless stated otherwise. The problem asks for the expected value under optimal play, period.

A concrete example: you can either take a guaranteed $5 or flip a coin for $0/$12. The expected value of the coin flip is $6, so under optimal play you should flip. In CP, the answer would be 6.

Minimax with Chance Nodes

The expectimax algorithm generalizes minimax to handle chance events. Here is a full C++ implementation for a game tree represented explicitly:

#include <bits/stdc++.h>
using namespace std;

enum NodeType { LEAF, MAX_NODE, MIN_NODE, CHANCE_NODE };

struct Node {
    NodeType type;
    double payoff;                    // only for LEAF
    vector<int> children;            // indices of children
    vector<double> prob;             // probabilities (CHANCE only)
};

vector<Node> tree;

double expectimax(int u) {
    Node &nd = tree[u];
    if (nd.type == LEAF) return nd.payoff;

    if (nd.type == MAX_NODE) {
        double best = -1e18;
        for (int c : nd.children)
            best = max(best, expectimax(c));
        return best;
    }
    if (nd.type == MIN_NODE) {
        double best = 1e18;
        for (int c : nd.children)
            best = min(best, expectimax(c));
        return best;
    }
    // CHANCE_NODE
    double ev = 0;
    for (int i = 0; i < (int)nd.children.size(); i++)
        ev += nd.prob[i] * expectimax(nd.children[i]);
    return ev;
}

Complexity Analysis

Without pruning, expectimax visits every node in the tree: O(bd) where b is the branching factor and d is the depth. For chance nodes we cannot do alpha-beta pruning because we need the weighted average of all children, not just the best one.

However, we can optimize with memoization when the game has repeated states (i.e., it's really a DAG, not a tree). This is where the DP approach shines.

Tip: If the game state can be encoded as a small integer (bitmask, position, score), convert the tree search into a DP. This transforms exponential-time tree search into polynomial-time DP.

Dice Game Analysis

Problem: Two-Player Dice Battle

Alice and Bob alternate rolling a fair 6-sided die. The first player whose cumulative score reaches or exceeds T wins. Alice goes first. What is the probability that Alice wins, under optimal play?

Since there are no choices (both players must roll), "optimal play" here is trivial. This is a pure chance game. We solve it with DP.

Let dp[a][b][turn] = probability that Alice wins when Alice has score a, Bob has score b, and it's turn's move (0 = Alice, 1 = Bob).

#include <bits/stdc++.h>
using namespace std;

const int MAXS = 105;
double dp[MAXS][MAXS][2];
bool vis[MAXS][MAXS][2];
int T;

double solve(int a, int b, int turn) {
    if (a >= T) return 1.0;   // Alice already won
    if (b >= T) return 0.0;   // Bob already won
    if (vis[a][b][turn]) return dp[a][b][turn];
    vis[a][b][turn] = true;

    double &res = dp[a][b][turn];
    res = 0;
    for (int die = 1; die <= 6; die++) {
        if (turn == 0)
            res += solve(a + die, b, 1) / 6.0;
        else
            res += solve(a, b + die, 0) / 6.0;
    }
    return res;
}

int main() {
    cin >> T;
    printf("%.9f\n", solve(0, 0, 0));
    return 0;
}

Optimal Stopping: The "Quit or Continue" Dice Game

You roll a die repeatedly. After each roll, you can stop and keep your cumulative score, or continue rolling. If your cumulative score exceeds K, you lose everything (score = 0). What is the optimal strategy and the expected value?

This is a single-player stochastic game with MAX and CHANCE nodes. Let dp[s] = expected payoff with current score s under optimal play.

#include <bits/stdc++.h>
using namespace std;

int main() {
    int K;
    cin >> K;

    vector<double> dp(K + 7, 0);
    // dp[s] = max(s, E[continue from s])
    // E[continue from s] = (1/6) * sum_{d=1}^{6} dp[s+d]
    // but dp[s+d] = 0 if s+d > K (bust)

    for (int s = K; s >= 0; s--) {
        double ev_continue = 0;
        for (int d = 1; d <= 6; d++) {
            if (s + d > K)
                ev_continue += 0;  // bust
            else
                ev_continue += dp[s + d];
        }
        ev_continue /= 6.0;
        dp[s] = max((double)s, ev_continue);
    }

    printf("%.9f\n", dp[0]);
    return 0;
}
Key insight: The optimal strategy has a simple threshold structure: stop if s ≥ threshold, continue otherwise. This is because the expected gain from continuing decreases as s approaches K (higher bust risk), while the guaranteed payoff s increases.

Card Game Strategy

Problem: Secretary Problem Variant

A deck has n cards with values 1, 2, …, n in random order. You reveal cards one at a time. After seeing each card, you can either take it (game ends, your score = card value) or skip it (irreversible). If you skip all n cards, you get the last card. Maximize the expected score under optimal play.

This is the classic optimal stopping problem. The DP state is (number of cards remaining, current best available information). However, the exact solution requires tracking which values have been seen.

Simplified Analysis: Threshold Strategy

It turns out the optimal strategy has a threshold that depends on how many cards remain. Let dp[k] = expected score when k cards remain and you play optimally. If the revealed card has value v:

  • Take it if v ≥ dp[k − 1] (the card beats continuing)
  • Skip it otherwise
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;

    // dp[k] = expected score with k cards remaining, optimal play
    // Cards are a random permutation of 1..n.
    // When k cards remain, the revealed card is uniform over unseen values.
    // For simplicity, model as uniform over [1..n] (approximation).

    vector<double> dp(n + 1, 0);
    dp[1] = (n + 1.0) / 2.0;  // forced to take the last card

    for (int k = 2; k <= n; k++) {
        // Threshold: take if value >= dp[k-1]
        double threshold = dp[k - 1];
        double ev = 0;
        int count = 0;
        for (int v = 1; v <= n; v++) {
            if (v >= threshold) {
                ev += v;
                count++;
            }
        }
        if (count > 0)
            dp[k] = (ev / n) + (1.0 - (double)count / n) * dp[k - 1];
        else
            dp[k] = dp[k - 1];
    }

    printf("%.9f\n", dp[n]);
    return 0;
}
Warning: The classic secretary problem asks for the probability of picking the best card (not maximizing expected value). Those are different objectives with different optimal strategies! The classic 1/e rule applies to the "pick the best" version. For expected value maximization, the thresholds are different.

Card Game with Two Players

When two players compete over randomly drawn cards, the game becomes a two-player stochastic game. Each player sees the drawn card and decides whether to claim it. The key technique: model as a game tree with chance nodes (card draws) and MAX/MIN nodes (player decisions), then solve with expectimax or DP.

DP on Game States

The most powerful technique for stochastic games in CP is backward induction via DP. We define game states, identify terminal states, and work backward.

General Framework

  1. Define the state: What information fully determines the future of the game? (Scores, cards remaining, whose turn, etc.)
  2. Identify base cases: Terminal states where the outcome is known.
  3. Write the recurrence: For MAX nodes, take the max over children. For MIN nodes, take the min. For CHANCE nodes, take the weighted average.
  4. Implement: Use top-down memoization or bottom-up DP.

Example: Coin Flip Duel

Alice and Bob start with scores 0. They alternate flipping a fair coin. Heads: your score increases by 1. Tails: opponent's score increases by 1. First to reach N wins. What is Alice's winning probability?

#include <bits/stdc++.h>
using namespace std;

const int MAXN = 55;
double dp[MAXN][MAXN][2];
bool vis[MAXN][MAXN][2];
int N;

double solve(int a, int b, int turn) {
    if (a >= N) return 1.0;
    if (b >= N) return 0.0;
    if (vis[a][b][turn]) return dp[a][b][turn];
    vis[a][b][turn] = true;

    double &res = dp[a][b][turn];
    if (turn == 0) { // Alice flips
        // Heads: Alice +1, Tails: Bob +1
        res = 0.5 * solve(a + 1, b, 1) + 0.5 * solve(a, b + 1, 1);
    } else { // Bob flips
        res = 0.5 * solve(a, b + 1, 0) + 0.5 * solve(a + 1, b, 0);
    }
    return res;
}

int main() {
    cin >> N;
    printf("%.9f\n", solve(0, 0, 0));
    return 0;
}

Modular Arithmetic Version

When the problem asks for the answer modulo 998244353, replace doubles with long long and use modular inverses:

const long long MOD = 998244353;

long long power(long long b, long long e, long long m) {
    long long r = 1; b %= m;
    while (e > 0) {
        if (e & 1) r = r * b % m;
        b = b * b % m;
        e >>= 1;
    }
    return r;
}

const long long INV2 = power(2, MOD - 2, MOD);

long long dp_mod[MAXN][MAXN][2];
bool vis_mod[MAXN][MAXN][2];

long long solve_mod(int a, int b, int turn) {
    if (a >= N) return 1;
    if (b >= N) return 0;
    if (vis_mod[a][b][turn]) return dp_mod[a][b][turn];
    vis_mod[a][b][turn] = true;

    long long &res = dp_mod[a][b][turn];
    if (turn == 0) {
        res = INV2 % MOD * (solve_mod(a + 1, b, 1)
            + solve_mod(a, b + 1, 1)) % MOD;
    } else {
        res = INV2 % MOD * (solve_mod(a, b + 1, 0)
            + solve_mod(a + 1, b, 0)) % MOD;
    }
    return res;
}
Pattern: To convert any stochastic game DP from floating-point to modular arithmetic: (1) replace double with long long, (2) replace division by k with multiplication by power(k, MOD-2, MOD), (3) add % MOD after every arithmetic operation.

Backward Induction for Sequential Games

For games played over discrete rounds (e.g., n rounds of dice rolling), process rounds from the last to the first. At each round, compute the expected value assuming optimal play in all future rounds. This is backward induction and guarantees the globally optimal strategy.

CP Applications

Application 1: Gambler's Ruin with Choice

Problem: A gambler starts with $a. Each round, they choose to bet $1 or $2. With probability p they win the bet amount, with probability 1−p they lose it. Target: reach $n. What bet strategy maximizes the probability of reaching $n before going broke?

State: current wealth w. Transitions depend on the chosen bet.

// dp[w] = max probability of reaching n from wealth w
vector<double> dp(n + 3, 0);
dp[n] = 1.0; // target reached

for (int w = n - 1; w >= 1; w--) {
    // Option 1: bet $1
    double ev1 = p * dp[min(w + 1, n)] + (1 - p) * dp[max(w - 1, 0)];
    // Option 2: bet $2
    double ev2 = p * dp[min(w + 2, n)] + (1 - p) * dp[max(w - 2, 0)];
    dp[w] = max(ev1, ev2);
}

Application 2: Expected Game Length

Problem: Two players play a game that ends when some condition is met. Each turn involves a random event. Find the expected number of turns.

This reduces to solving a system of linear equations. Let E[s] = expected turns from state s. For each state:

E[s] = 1 + Σt P(s → t) · E[t]

Terminal states have E[s] = 0. Solve via DP (if no cycles) or Gaussian elimination (if the state graph has cycles).

Application 3: Stochastic Shortest Path

Problem: In a graph, each edge has a random weight drawn from a known distribution. Find the path that minimizes expected total weight.

If edge weights are independent, the expected weight of a path is the sum of expected edge weights. So the stochastic shortest path reduces to a deterministic shortest path on the expected-weight graph!

Tip: Linearity of expectation is your best friend in stochastic games. Before diving into complex DP, check whether the expected value decomposes into a simple sum.

Technique When to Use
Expectimax tree search Small game trees, explicit structure
DP with backward induction Games with polynomial state space

Final thought: The hardest part of stochastic game problems is not the implementation — it's defining the right state and recognizing whether the game has choices (MAX/MIN nodes) or is purely random (CHANCE nodes only). Once you get the model right, the code follows naturally.
