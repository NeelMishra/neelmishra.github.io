Probability Basics for Competitive Programming

June 2026 · 22 min read

Probability is one of the most rewarding topics in competitive programming. Unlike combinatorics, where you count how many, probability asks how likely. Many Codeforces Div 1 and AtCoder problems rely on probability reasoning, so a solid foundation here pays dividends. In this opening post of the Probability & Expected Value module we will cover the core definitions, the algebra of events, conditional probability, and the indispensable inclusion-exclusion principle.

Sample Space & Probability Axioms

Every probability problem starts with a sample space Ω, the set of all possible outcomes of an experiment. Rolling a fair six-sided die gives Ω = {1, 2, 3, 4, 5, 6}. Flipping two coins gives Ω = {HH, HT, TH, TT}.

An event is any subset of Ω. The event “roll an even number” is A = {2, 4, 6}. The entire sample space is the certain event, and the empty set ∅ is the impossible event.

Kolmogorov’s probability axioms give us the formal foundation:

  1. Non-negativity: For every event A, P(A) ≥ 0.
  2. Normalization: P(Ω) = 1.
  3. Countable additivity: If A1, A2, … are mutually exclusive events, then P(A1 ∪ A2 ∪ …) = ∑ P(Ai).
CP Tip: In contest problems the sample space is almost always finite and uniform (every outcome equally likely). That means P(A) = |A| / |Ω|, reducing probability to counting.

From the axioms several useful corollaries follow immediately:

  • P(∅) = 0.
  • P(Ac) = 1 − P(A) — the complement rule, hugely useful in CP.
  • If A ⊆ B then P(A) ≤ P(B).
  • For any two events: P(A ∪ B) = P(A) + P(B) − P(A ∩ B).

Example – dice sum. Two fair dice are rolled. Find the probability the sum is at least 10. The sample space has 36 equally likely outcomes. The favourable outcomes for sum ≥ 10 are: (4,6),(5,5),(5,6),(6,4),(6,5),(6,6) — six outcomes. So P(sum ≥ 10) = 6/36 = 1/6.

Events & Set Operations

Because events are sets we can combine them with the usual set operations, and each has a probabilistic interpretation:

  • Union A ∪ B — “A or B (or both) happens”.
  • Intersection A ∩ B — “both A and B happen”.
  • Complement Ac = Ω \ A — “A does not happen”.
  • Difference A \ B = A ∩ Bc — “A happens but not B”.

Two events are mutually exclusive (disjoint) if A ∩ B = ∅. In that case P(A ∪ B) = P(A) + P(B), which is axiom 3 for two events.

Example – cards. A card is drawn from a standard 52-card deck. Let A = “card is a heart” and B = “card is a face card”. Then |A| = 13, |B| = 12, |A ∩ B| = 3 (J, Q, K of hearts). So P(A ∪ B) = 13/52 + 12/52 − 3/52 = 22/52 = 11/26.

Warning: A common contest mistake is to add probabilities of overlapping events without subtracting the intersection. Always check whether events are disjoint first.

De Morgan’s laws are the bridge between unions and intersections under complementation. They are essential in inclusion-exclusion reasoning:

  • (A ∪ B)c = Ac ∩ Bc
  • (A ∩ B)c = Ac ∪ Bc

In CP you often want “none of the bad things happen” which is the complement of the union of all bad events — exactly De Morgan applied.

Conditional Probability & Independence

The conditional probability of A given B is defined as:

P(A | B) = P(A ∩ B) / P(B),   provided P(B) > 0.

Intuitively we “shrink” the sample space to the outcomes where B occurred and ask how much of that is also in A.

Example – dice conditioning. A fair die is rolled. Given the result is odd, what is the probability it is at least 5? Here B = {1, 3, 5} and A = {5, 6}. So A ∩ B = {5}, P(A ∩ B) = 1/6, P(B) = 3/6 = 1/2. Therefore P(A | B) = (1/6)/(1/2) = 1/3.

Two events are independent if knowing one gives no information about the other: P(A ∩ B) = P(A) · P(B). Equivalently, P(A | B) = P(A). Independence is a property, not a definition — you must verify it, not assume it.

CP Tip: Many contest problems give you independent trials (e.g., “each edge exists with probability p independently”). The word “independently” lets you multiply probabilities across trials. When it’s absent, be careful!

The multiplication rule extends to chains of events:

P(A1 ∩ A2 ∩ … ∩ An) = P(A1) · P(A2 | A1) · P(A3 | A1 ∩ A2) · …

This chain rule is the backbone of many DP-on-probability approaches. You process events one at a time, conditioning on everything you already know.

C++ – Monte Carlo probability simulation

When an exact formula is hard to derive, a quick Monte Carlo simulation can sanity-check your answer. The following snippet estimates the probability two random integers in [1, N] are coprime: 

#include <bits/stdc++.h>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    mt19937_64 rng(chrono::steady_clock::now().time_since_epoch().count());
    int N = 1000000;
    long long trials = 10000000, hits = 0;
    for(long long t = 0; t < trials; t++){
        long long a = rng() % N + 1;
        long long b = rng() % N + 1;
        if(__gcd(a, b) == 1) hits++;
    }
    // Should converge to 6 / pi^2 ≈ 0.6079
    cout << fixed << setprecision(6);
    cout << "P(coprime) ≈ " << (double)hits / trials << "\n";
    return 0;
}

The Inclusion-Exclusion Principle

Inclusion-exclusion (IE) is arguably the most powerful counting/probability tool in CP. It generalises the two-event formula to any number of events.

Statement. For events A1, …, An:

P(A1 ∪ … ∪ An) = ∑i P(Ai) − ∑i<j P(Ai ∩ Aj) + ∑i<j<k P(Ai ∩ Aj ∩ Ak) − …

Proof sketch (indicator-variable method)

For each outcome ω ∈ Ω let Xi(ω) = 1 if ω ∈ Ai else 0. The indicator for the union is 1 − ∏(1 − Xi). Expanding the product and taking expectations gives exactly the alternating sum above. The key identity:

1 − (1−X1)(1−X2) = X1 + X2 − X1X2

generalises to n terms with alternating signs by the binomial-like expansion.

CP Application – coprime counting with IE

Problem. Given N and a set of primes {p1, …, pk}, count integers in [1, N] divisible by at least one pi. By IE the count is:

∑ ⌊N/pi⌋ − ∑i<j ⌊N/(pipj)⌋ + …

Subtracting from N gives the count coprime to all listed primes — the sieve of Eratosthenes is exactly this!

#include <bits/stdc++.h>
using namespace std;

// Count integers in [1..N] coprime to all primes in ps[]
long long coprime_count(long long N, vector<long long>& ps){
    int k = ps.size();
    long long total = 0;
    for(int mask = 1; mask < (1 << k); mask++){
        long long prod = 1;
        int bits = __builtin_popcount(mask);
        for(int i = 0; i < k; i++){
            if(mask >> i & 1){
                prod *= ps[i];
                if(prod > N) break;  // overflow guard
            }
        }
        if(bits & 1) total += N / prod;
        else          total -= N / prod;
    }
    return N - total;  // coprime count
}

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long long N; int k;
    cin >> N >> k;
    vector<long long> primes(k);
    for(auto &p : primes) cin >> p;
    cout << coprime_count(N, primes) << "\n";
    return 0;
}
Complexity note: The bitmask loop runs in O(2k), so this is efficient when k is small (typically k ≤ 20). For Euler’s totient of a single number, k is at most ~15 (number of distinct prime factors of numbers up to 1018).

Preview: Bayes’ Theorem

Conditional probability leads directly to one of the most important results in all of probability: Bayes’ theorem. We will cover it in depth in the next post, but here is the statement for reference:

P(A | B) = P(B | A) · P(A) / P(B)

This “reversal” formula lets you update beliefs: given that you observed B, what can you say about A? It is the foundation of Bayesian reasoning and appears in CP problems involving sequential observations (e.g., “after drawing k red balls, what is the probability the urn is type 1?”).

We also need the law of total probability to compute P(B) when B can happen through several mutually exclusive pathways:

P(B) = ∑i P(B | Ai) · P(Ai)

Together, total probability + Bayes give you a systematic recipe for solving any “reverse conditioning” problem. Details and code in Part 2.

Practice Problems

Try these to solidify your understanding of the basics:

  1. Dice game. Roll two dice. You win if the sum is 7 or 11. What is P(win)? (Hint: count favourable outcomes.)
  2. Birthday paradox. In a group of n people, what is the probability at least two share a birthday? Use the complement: compute the probability all birthdays are distinct. For n = 23 the answer exceeds 0.5.
  3. Coprime pair (Codeforces-style). Given N ≤ 1012 and M ≤ 20 primes, count integers in [1, N] coprime to all of them. Use the IE bitmask approach above.
  4. Card draw. From a standard deck, 5 cards are drawn without replacement. What is the probability of getting exactly one ace? Use hypergeometric counting.
  5. Conditional dice. A fair die is rolled twice. Given that the first roll is at most 3, find the probability the sum of both rolls is at least 8.
Common pitfall: Don’t confuse “at least one” with “exactly one.” For “at least one” use the complement: P(at least one) = 1 − P(none). For “exactly one” you usually need a direct count or IE with careful parity.

Solutions sketch for Problem 1

Sum of 7: (1,6),(2,5),(3,4),(4,3),(5,2),(6,1) — 6 outcomes.
Sum of 11: (5,6),(6,5) — 2 outcomes.
These are disjoint, so P(win) = (6+2)/36 = 8/36 = 2/9.

C++ – Birthday paradox probability

#include <bits/stdc++.h>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    double prob_all_distinct = 1.0;
    for(int i = 1; i < n; i++){
        prob_all_distinct *= (365.0 - i) / 365.0;
    }
    cout << fixed << setprecision(9);
    cout << "P(at least one shared birthday) = "
         << 1.0 - prob_all_distinct << "\n";
    return 0;
}
