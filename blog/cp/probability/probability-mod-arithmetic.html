Probability with Modular Arithmetic

October 2026 · 22 min read

Why Modular Probability?

If you've solved enough Codeforces problems, you've seen the dreaded line: "Print the answer modulo 998244353." At first this seems bizarre — probability is a real number between 0 and 1, so how can we take it modulo a prime? The answer lies in a beautiful connection between fractions and modular arithmetic.

The core issue is precision. Floating-point numbers accumulate rounding errors, and for large inputs the answer could differ by enough to be marked wrong. By expressing the probability as a fraction a / b and computing a · b−1 mod p, we get an exact integer that the judge can compare bit-for-bit.

Key insight: Every rational probability P = a/b (with gcd(a, b) = 1 and b ≠ 0 mod p) has a unique representation in ℤp. Two different rational numbers map to different residues, so no information is lost.

The prime 998244353 is chosen because it is an NTT-friendly prime (998244353 = 119 · 223 + 1), which allows efficient polynomial multiplication via Number Theoretic Transform. Many problems that combine probability with counting or convolutions benefit from this property.

Another popular modulus is 109 + 7 (= 1000000007). Both are large primes, ensuring that every non-zero integer in the range [1, p−1] has a multiplicative inverse.

When does this NOT work?

If the denominator b is divisible by p, then b−1 mod p does not exist. In practice this almost never happens because p is astronomically large, and problem setters guarantee the answer is well-defined. But you should still watch out for edge cases where the probability is 0 or 1 — handle those separately before computing the inverse.

Modular Inverse Review

The modular inverse of an integer b modulo a prime p is the unique integer b−1 such that:

b · b−1 ≡ 1 (mod p)

Method 1: Fermat's Little Theorem

Since p is prime and gcd(b, p) = 1, Fermat's little theorem gives us:

bp−1 ≡ 1 (mod p)  ⇒  b−1 ≡ bp−2 (mod p)

This is the most commonly used method in competitive programming because it requires only a single fast exponentiation call, which runs in O(log p) time.

#include <bits/stdc++.h>
using namespace std;

const long long MOD = 998244353;

long long power(long long base, long long exp, long long mod) {
    long long result = 1;
    base %= mod;
    while (exp > 0) {
        if (exp & 1) result = result * base % mod;
        base = base * base % mod;
        exp >>= 1;
    }
    return result;
}

long long inv(long long b) {
    return power(b, MOD - 2, MOD);
}

Method 2: Extended Euclidean Algorithm

The extended GCD computes integers x, y such that b·x + p·y = 1. Then x mod p is b−1. This works for any modulus (not just primes) as long as gcd(b, p) = 1.

long long extgcd(long long a, long long b, long long &x, long long &y) {
    if (b == 0) { x = 1; y = 0; return a; }
    long long x1, y1;
    long long g = extgcd(b, a % b, x1, y1);
    x = y1;
    y = x1 - (a / b) * y1;
    return g;
}

long long inv_ext(long long b, long long mod) {
    long long x, y;
    extgcd(b, mod, x, y);
    return (x % mod + mod) % mod;
}
Warning: In competitive programming, prefer Fermat's method for prime moduli — it's shorter and less error-prone. Use extended GCD only when the modulus is not prime or when you need the inverse for a composite modulus.

Fraction Representation mod p

A probability P = a/b is represented in modular arithmetic as:

P ≡ a · b−1 (mod p)

This representation preserves all the algebraic properties we need:

  • Addition: a/b + c/d ≡ (a·d + c·b) · (b·d)−1 (mod p)
  • Multiplication: (a/b) · (c/d) ≡ (a·c) · (b·d)−1 (mod p)
  • Subtraction: a/b − c/d ≡ (a·d − c·b) · (b·d)−1 (mod p). Remember to add p if the result is negative.

Worked Example

Suppose P = 3/7 and we want P mod 998244353.

  1. Compute 7−1 mod 998244353 = 7998244351 mod 998244353 = 855638017
  2. Answer = 3 · 855638017 mod 998244353 = 569914698

We can verify: 569914698 · 7 mod 998244353 = 3. ✓

A useful mental model: think of the residue 569914698 as "3/7 in disguise." All arithmetic on these disguised fractions produces the correct disguised result.

Building a Modular Fraction Struct

struct ModFrac {
    long long val;
    static const long long MOD = 998244353;

    ModFrac(long long v = 0) : val((v % MOD + MOD) % MOD) {}
    ModFrac(long long num, long long den)
        : val(num % MOD * power(den, MOD - 2, MOD) % MOD) {}

    ModFrac operator+(const ModFrac &o) const {
        return ModFrac((val + o.val) % MOD);
    }
    ModFrac operator-(const ModFrac &o) const {
        return ModFrac((val - o.val + MOD) % MOD);
    }
    ModFrac operator*(const ModFrac &o) const {
        return ModFrac(val * o.val % MOD);
    }
    ModFrac operator/(const ModFrac &o) const {
        return ModFrac(val * power(o.val, MOD - 2, MOD) % MOD);
    }
};

Computing Probabilities mod p

Let's walk through the full pipeline for a typical CP probability problem.

Step 1: Identify the Probability as a Fraction

Most probability problems in CP reduce to counting: P = (favorable outcomes) / (total outcomes). Both numerator and denominator are integers, often involving factorials and binomial coefficients.

Step 2: Compute Numerator and Denominator mod p

Compute each separately using modular arithmetic. For combinatorial quantities, use precomputed factorials (covered in the next section).

Step 3: Multiply by the Inverse of the Denominator

answer = numerator · power(denominator, p − 2, p) % p.

Full Example: Random Pair Sum

Problem: Given an array of n integers, pick two elements uniformly at random (without replacement). What is the expected value of their sum, modulo 998244353?

Solution: By linearity of expectation, E[ai + aj] = E[ai] + E[aj]. Each element is chosen with probability 2/(n) in the sum (each element appears in (n−1) of the C(n,2) pairs). So E[sum] = 2 · S / n, where S = Σ ai.

#include <bits/stdc++.h>
using namespace std;

const long long MOD = 998244353;

long long power(long long b, long long e, long long m) {
    long long r = 1; b %= m;
    while (e > 0) {
        if (e & 1) r = r * b % m;
        b = b * b % m;
        e >>= 1;
    }
    return r;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;

    long long total = 0;
    for (int i = 0; i < n; i++) {
        long long a; cin >> a;
        total = (total + a) % MOD;
    }

    // E[sum] = 2 * total / n  mod p
    long long ans = 2 * total % MOD * power(n, MOD - 2, MOD) % MOD;
    cout << ans << "\n";
    return 0;
}
Tip: When the problem says "expected value modulo p," treat every division as multiplication by a modular inverse. The entire computation stays in integer arithmetic — no floats needed.

Precomputing Inverses

Many probability problems require binomial coefficients C(n, k), which involve factorials. Computing the inverse of each factorial on-the-fly via fast exponentiation is O(n log p). We can do better.

Precomputing Factorials and Inverse Factorials in O(n)

The trick: compute inv_fact[n] = (n!)−1 using a single fast-exponentiation call, then build the rest via:

inv_fact[i−1] = inv_fact[i] · i 

const int MAXN = 2000005;
long long fact[MAXN], inv_fact[MAXN];

void precompute() {
    fact[0] = 1;
    for (int i = 1; i < MAXN; i++)
        fact[i] = fact[i - 1] * i % MOD;

    inv_fact[MAXN - 1] = power(fact[MAXN - 1], MOD - 2, MOD);
    for (int i = MAXN - 2; i >= 0; i--)
        inv_fact[i] = inv_fact[i + 1] * (i + 1) % MOD;
}

long long C(int n, int k) {
    if (k < 0 || k > n) return 0;
    return fact[n] % MOD * inv_fact[k] % MOD * inv_fact[n - k] % MOD;
}

This runs the expensive power() call only once. Everything else is O(1) per entry. Total precomputation: O(n + log p).

Precomputing Individual Inverses in O(n)

Sometimes you need inv[i] = i−1 mod p for i = 1, 2, …, n. There's an elegant recurrence:

inv[i] = −(p / i) · inv[p mod i] mod p 

long long inv_arr[MAXN];

void precompute_inv() {
    inv_arr[1] = 1;
    for (int i = 2; i < MAXN; i++)
        inv_arr[i] = (MOD - MOD / i) * inv_arr[MOD % i] % MOD;
}
Warning: The recurrence for inv_arr relies on p being prime. It produces incorrect results for composite moduli. Also, inv_arr[0] is undefined — never access it.

Common Patterns

Pattern 1: Binomial Probability mod p

If each trial succeeds with probability a/b independently, the probability of exactly k successes in n trials is:

P(k) = C(n, k) · (a/b)k · (1 − a/b)n−k

In modular arithmetic:

long long binom_prob(int n, int k, long long a, long long b) {
    long long p_succ = a % MOD * power(b, MOD - 2, MOD) % MOD;
    long long p_fail = (1 - p_succ + MOD) % MOD;
    return C(n, k) % MOD
         * power(p_succ, k, MOD) % MOD
         * power(p_fail, n - k, MOD) % MOD;
}

Pattern 2: Expected Value as Sum of Probabilities

By linearity, E[X] = Σi P(event i). Each P(event i) is computed as a modular fraction, and we just add them up mod p.

Pattern 3: Probability DP with Modular Fractions

In DP problems, each state stores a probability as a modular integer. Transitions multiply by modular fractions. For example, if from state s we go to state t with probability 1/3:

dp[t] = (dp[t] + dp[s] * power(3, MOD - 2, MOD)) % MOD;

Pattern 4: Conditional Probability

P(A|B) = P(A ∩ B) / P(B). Compute both numerator and denominator mod p, then multiply the numerator by the inverse of the denominator.

CP Applications

Application 1: Random Permutation Statistics

Problem: A random permutation of [1..n] is generated. Find the expected number of fixed points modulo 998244353.

Solution: Let Xi = 1 if π(i) = i. Then P(Xi = 1) = 1/n. By linearity, E[X] = n · (1/n) = 1. The answer is simply 1, regardless of n. Elegant!

Application 2: Coupon Collector

Problem: There are n distinct coupons. Each draw gives a uniformly random coupon. Expected number of draws to collect all n, modulo 998244353.

Solution: E = n · H(n) where H(n) = Σi=1..n 1/i is the harmonic series. In modular arithmetic:

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    precompute();

    int n;
    cin >> n;

    long long ans = 0;
    for (int i = 1; i <= n; i++)
        ans = (ans + power(i, MOD - 2, MOD)) % MOD;
    ans = ans * n % MOD;
    cout << ans << "\n";
    return 0;
}

Application 3: Dice Probability

Problem: Roll a fair 6-sided die n times. Find the probability that the sum is exactly S, modulo 998244353.

Solution: Use DP. Let dp[i][j] = number of ways to get sum j with i dice. Answer = dp[n][S] · (6n)−1 mod p.

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, S;
    cin >> n >> S;

    vector<vector<long long>> dp(n + 1, vector<long long>(S + 1, 0));
    dp[0][0] = 1;
    for (int i = 1; i <= n; i++)
        for (int j = 1; j <= S; j++)
            for (int f = 1; f <= 6 && f <= j; f++)
                dp[i][j] = (dp[i][j] + dp[i - 1][j - f]) % MOD;

    long long total = power(6, n, MOD);
    long long ans = dp[n][S] % MOD * power(total, MOD - 2, MOD) % MOD;
    cout << ans << "\n";
    return 0;
}
Tip: When the total number of outcomes is kn, precompute its inverse once outside the main loop rather than calling power() repeatedly.
