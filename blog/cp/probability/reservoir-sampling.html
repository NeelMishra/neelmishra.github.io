Reservoir Sampling

September 2026 · 20 min read

Problem Statement

Consider the following fundamental problem: you are reading elements from a data stream one at a time. You do not know the total length n of the stream in advance—it could be billions of elements, and you cannot store them all. Your task is to select exactly k elements uniformly at random from the stream, using only O(k) memory.

This is the reservoir sampling problem. It arises in many practical scenarios: sampling log entries from a continuously growing log file, selecting random records from a database query of unknown size, or choosing random nodes from a graph traversal.

The simplest special case is k = 1: maintain a single element, and at the end of the stream, that element should be a uniformly random selection from all n elements seen. Let us start with this case before generalizing.

The Constraint: We process elements one at a time in a single pass, using O(k) memory. We cannot go back and re-read previous elements. When the stream ends, our reservoir must contain k elements chosen uniformly at random from all n elements that appeared.

Why uniform randomness matters: Each of the C(n, k) possible subsets of size k must be equally likely. Equivalently, each individual element must appear in the reservoir with probability exactly k/n. But we do not know n in advance—this is what makes the problem non-trivial.

A naive approach—store all elements then sample—uses O(n) memory. Another idea—accept each element with probability k/n—does not work because we do not know n. Reservoir sampling elegantly resolves this with a single-pass O(k)-space algorithm.

Algorithm R

Algorithm R, introduced by Jeffrey Vitter in 1985, is the classic reservoir sampling algorithm. Here is the procedure for selecting k items from a stream:

// Algorithm R: Reservoir Sampling
Initialize reservoir R[0..k-1] with the first k items
For i = k, k+1, k+2, ... (each new stream element):
  j = random_int(0, i)  // uniform in [0, i]
  If j < k:
    R[j] = stream[i]  // replace element at position j

That is the entire algorithm. When processing the i-th element (0-indexed), generate a random integer j uniformly in [0, i]. If j < k, replace R[j] with the new element. Otherwise, discard the new element.

For the special case k = 1, this simplifies beautifully:

  • Start with the first element as the selected item
  • When seeing the i-th element (1-indexed), replace the selected item with probability 1/i
  • After seeing all n elements, the selected item is uniformly random
#include <bits/stdc++.h>
using namespace std;

// Reservoir sampling for k = 1
// Returns a uniformly random element from the stream
int reservoir_sample_one(vector<int>& stream) {
    int result = stream[0];
    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

    for (int i = 1; i < (int)stream.size(); i++) {
        // Replace with probability 1/(i+1)
        int j = uniform_int_distribution<int>(0, i)(rng);
        if (j == 0) {
            result = stream[i];
        }
    }
    return result;
}

// Reservoir sampling for general k
vector<int> reservoir_sample_k(vector<int>& stream, int k) {
    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
    vector<int> reservoir(stream.begin(), stream.begin() + k);

    for (int i = k; i < (int)stream.size(); i++) {
        int j = uniform_int_distribution<int>(0, i)(rng);
        if (j < k) {
            reservoir[j] = stream[i];
        }
    }
    return reservoir;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, k;
    cin >> n >> k;
    vector<int> stream(n);
    for (int i = 0; i < n; i++) cin >> stream[i];

    vector<int> sample = reservoir_sample_k(stream, k);
    for (int x : sample) cout << x << " ";
    cout << "\n";
    return 0;
}
Common Mistake: Using rand() % (i+1) introduces modulo bias for large i. Always use uniform_int_distribution with a proper RNG engine like mt19937 for correct uniform sampling in competitive programming.

Correctness Proof

We prove that after processing all n elements, each element is in the reservoir with probability exactly k/n. The proof proceeds by induction on the stream length.

Base case (i = k): The reservoir contains the first k elements, each with probability k/k = 1. This is trivially correct since all k elements are present.

Inductive step: Assume that after processing i elements (i ≥ k), each element in the reservoir has probability k/i of being present. We must show that after processing element i + 1, each element has probability k/(i + 1).

For the new element (i + 1): It is included in the reservoir iff the random number j ∈ [0, i] satisfies j < k. This happens with probability k/(i + 1). ✓

For an old element already in the reservoir: It survives if either:

  • The new element is not selected (probability (i + 1 − k)/(i + 1)), OR
  • The new element is selected but replaces a different position (probability k/(i + 1) · (k − 1)/k = (k − 1)/(i + 1))

The survival probability is: (i + 1 − k)/(i + 1) + (k − 1)/(i + 1) = i/(i + 1).

By the inductive hypothesis, the old element was in the reservoir with probability k/i. After the (i + 1)-th step, its probability becomes:

(k/i) · (i/(i + 1)) = k/(i + 1) ✓

Key Insight: The beauty of reservoir sampling is that the replacement probability automatically adjusts as the stream grows. Each old element loses probability at exactly the rate needed to accommodate new elements, maintaining uniformity throughout.

We can verify this empirically. The following code runs the reservoir sampling algorithm many times and checks that each element appears with approximately equal frequency: 

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n = 100, k = 5, trials = 1000000;
    mt19937 rng(42);
    vector<int> count(n, 0);

    for (int t = 0; t < trials; t++) {
        vector<int> reservoir(k);
        iota(reservoir.begin(), reservoir.end(), 0);

        for (int i = k; i < n; i++) {
            int j = uniform_int_distribution<int>(0, i)(rng);
            if (j < k) reservoir[j] = i;
        }

        for (int x : reservoir) count[x]++;
    }

    // Each element should appear ~k/n * trials = 50000 times
    double expected = (double)k / n * trials;
    double max_dev = 0;
    for (int i = 0; i < n; i++) {
        double dev = abs(count[i] - expected) / expected;
        max_dev = max(max_dev, dev);
    }
    cout << "Expected count per element: " << expected << "\n";
    cout << "Max relative deviation: " << fixed << setprecision(4)
         << max_dev * 100 << "%\n";
    return 0;
}

Weighted Reservoir Sampling

In many applications, elements are not equally important—each element xi has an associated weight wi > 0. We want to sample k elements such that the probability of including element i is proportional to its weight.

The A-Res (Algorithm with Reservoir) by Efraimidis and Spirakis (2006) elegantly solves this. The key idea is to assign each element a priority key:

keyi = ui1/wi

where ui is drawn uniformly from (0, 1). Then keep the k elements with the largest keys in a min-heap.

// A-Res: Weighted Reservoir Sampling
Initialize min-heap H of capacity k
For each element x_i with weight w_i:
  u = random_double(0, 1)
  key = u1/w_i
  If |H| < k:
    H.push(x_i, key)
  Else if key > H.top().key:
    H.pop(); H.push(x_i, key)

Why this works: For a single element selection (k = 1), element i is selected when its key is the maximum. The probability that ui1/wi > uj1/wj for all j ≠ i works out to wi / ∑ wj, which is exactly weighted sampling. 

#include <bits/stdc++.h>
using namespace std;

// Weighted reservoir sampling using A-Res algorithm
vector<int> weighted_reservoir(vector<pair<int,double>>& items, int k) {
    // items[i] = {value, weight}
    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
    uniform_real_distribution<double> udist(0.0, 1.0);

    // min-heap of (key, value)
    priority_queue<pair<double,int>, vector<pair<double,int>>,
                   greater<pair<double,int>>> pq;

    for (auto& [val, w] : items) {
        double u = udist(rng);
        double key = pow(u, 1.0 / w);
        if ((int)pq.size() < k) {
            pq.push({key, val});
        } else if (key > pq.top().first) {
            pq.pop();
            pq.push({key, val});
        }
    }

    vector<int> result;
    while (!pq.empty()) {
        result.push_back(pq.top().second);
        pq.pop();
    }
    return result;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // Example: 5 items with different weights
    vector<pair<int,double>> items = {
        {1, 1.0}, {2, 2.0}, {3, 5.0}, {4, 10.0}, {5, 0.5}
    };
    int k = 2;

    // Verify distribution over many trials
    map<int, int> freq;
    int trials = 100000;
    for (int t = 0; t < trials; t++) {
        auto sample = weighted_reservoir(items, k);
        for (int x : sample) freq[x]++;
    }

    double total_weight = 0;
    for (auto& [v, w] : items) total_weight += w;

    cout << "Element | Empirical | Expected\n";
    for (auto& [v, w] : items) {
        double emp = (double)freq[v] / trials;
        double exp_prob = min(1.0, (double)k * w / total_weight);
        cout << "   " << v << "    |  " << fixed << setprecision(4)
             << emp << "   |  " << exp_prob << "\n";
    }
    return 0;
}
Numerical Warning: When weights are very large, u1/w approaches 1.0 and floating-point precision is lost. Use log-space arithmetic instead: log(key) = log(u) / w, and compare log-keys. This avoids underflow and overflow issues in competitive programming.

Reservoir for k Items

Let us examine the general k-item reservoir more carefully and discuss optimizations that are relevant for competitive programming.

Time complexity: Algorithm R processes each element in O(1) time (one random number generation and at most one swap), for a total of O(n) time. The space usage is O(k).

Optimization: skip generation. For large streams where k « n, most elements are discarded. Vitter’s Algorithm Z (1985) precomputes how many elements to skip before the next replacement, reducing the expected number of random variates from n to O(k(1 + log(n/k))).

The skip count S follows a specific distribution. After processing i items, the probability that the next replacement occurs at item i + S + 1 is:

P(S = s) = (k / (i + s + 1)) · ∏j=1s (1 − k/(i + j))

For CP purposes, Algorithm R is usually fast enough since n ≤ 106 or 107 is typical. But it is good to know that Algorithm Z exists for interview questions and system design.

Connection to Fisher-Yates shuffle: Reservoir sampling is essentially the Fisher-Yates shuffle applied to a stream. If we could store all n elements, the Fisher-Yates shuffle would produce a random permutation in O(n). Reservoir sampling achieves the same effect for the first k positions without storing the full array. 

#include <bits/stdc++.h>
using namespace std;

// Fisher-Yates shuffle (for comparison)
void fisher_yates(vector<int>& arr) {
    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
    int n = arr.size();
    for (int i = n - 1; i > 0; i--) {
        int j = uniform_int_distribution<int>(0, i)(rng);
        swap(arr[i], arr[j]);
    }
}

// Select k random elements using Fisher-Yates partial shuffle
// Requires O(n) memory but only O(k) swaps
vector<int> partial_shuffle(vector<int> arr, int k) {
    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
    int n = arr.size();
    for (int i = 0; i < k; i++) {
        int j = uniform_int_distribution<int>(i, n - 1)(rng);
        swap(arr[i], arr[j]);
    }
    return vector<int>(arr.begin(), arr.begin() + k);
}

// Reservoir sampling: O(k) memory, O(n) time, single pass
vector<int> reservoir(int n, int k) {
    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
    vector<int> res(k);
    iota(res.begin(), res.end(), 0);

    for (int i = k; i < n; i++) {
        int j = uniform_int_distribution<int>(0, i)(rng);
        if (j < k) res[j] = i;
    }
    return res;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n = 1000000, k = 10;
    auto sample = reservoir(n, k);
    sort(sample.begin(), sample.end());
    cout << "Random sample of " << k << " from [0, " << n << "):\n";
    for (int x : sample) cout << x << " ";
    cout << "\n";
    return 0;
}

Online Random Selection

Reservoir sampling for k = 1 has a beautiful application in linked list problems: select a random node from a singly linked list in one pass.

This is LeetCode 382 (Linked List Random Node). Since we cannot compute the length without a full traversal, and we want to avoid two passes, reservoir sampling gives us an elegant one-pass solution: 

#include <bits/stdc++.h>
using namespace std;

struct ListNode {
    int val;
    ListNode* next;
    ListNode(int x) : val(x), next(nullptr) {}
};

class Solution {
    ListNode* head;
    mt19937 rng;

public:
    Solution(ListNode* h) : head(h),
        rng(chrono::steady_clock::now().time_since_epoch().count()) {}

    int getRandom() {
        int result = head->val;
        ListNode* cur = head->next;
        int i = 2;
        while (cur) {
            // Keep current element with probability 1/i
            if (uniform_int_distribution<int>(1, i)(rng) == 1) {
                result = cur->val;
            }
            cur = cur->next;
            i++;
        }
        return result;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // Build list: 1 -> 2 -> 3 -> 4 -> 5
    ListNode* head = new ListNode(1);
    ListNode* cur = head;
    for (int i = 2; i <= 5; i++) {
        cur->next = new ListNode(i);
        cur = cur->next;
    }

    Solution sol(head);

    // Verify uniformity
    map<int, int> freq;
    int trials = 500000;
    for (int t = 0; t < trials; t++) {
        freq[sol.getRandom()]++;
    }
    for (auto& [val, cnt] : freq) {
        cout << val << ": " << fixed << setprecision(4)
             << (double)cnt / trials << "\n";
    }
    return 0;
}
Streaming Paradigm: Reservoir sampling exemplifies the streaming algorithm paradigm: process data in one pass with sublinear memory. Related streaming algorithms include Count-Min Sketch (frequency estimation), HyperLogLog (cardinality estimation), and Misra-Gries (heavy hitters).

CP Applications

Reservoir sampling and its underlying probabilistic ideas appear in competitive programming in several ways:

1. Random Node Selection in Graphs

When you need to select a uniformly random node satisfying a condition (e.g., random leaf in a tree), and you discover qualifying nodes during traversal, use k=1 reservoir sampling. This avoids the need to first count qualifying nodes and then pick one.

2. Randomized Algorithms with Provable Guarantees

Many CP problems have randomized solutions where you pick a random element and check if it has some property. If at least a fraction f of elements have the property, then after 1/f trials, you find one with constant probability. Reservoir sampling lets you draw these random elements from a stream or complicated data structure.

3. Interactive Problems

In interactive problems where you query elements one at a time and must maintain a random sample, reservoir sampling is the natural choice. You process responses as a stream and keep a running sample.

4. Generating Random Permutations Online

If elements arrive one at a time and you must maintain a uniformly random permutation of all elements seen so far, insert each new element at a random position. This is the online Fisher-Yates shuffle, closely related to reservoir sampling.

5. Load Balancing and Fair Distribution

Some problems require distributing items among servers or buckets such that each server gets a fair share. Reservoir sampling provides a principled way to do this when the total count is unknown.

6. Random Subset Queries

Given a dynamic data structure (e.g., a set with insertions and deletions), maintain the ability to return a uniformly random element in O(1). While reservoir sampling alone does not solve this (deletions are tricky), the core idea of maintaining a random sample inspires the solution: keep elements in an array, swap-and-pop for deletions, random index for queries.

Practice Problems

Here are problems to practice reservoir sampling and related techniques:

  1. LeetCode 382 – Linked List Random Node
    The classic k = 1 reservoir sampling problem. Select a random node from a singly linked list in one pass without knowing the length. Direct application of the algorithm.
  2. LeetCode 398 – Random Pick Index
    Given an array with duplicates, return a random index of a given target value. Use reservoir sampling to select among matching indices in a single scan.
  3. LeetCode 497 – Random Point in Non-overlapping Rectangles
    Weighted random selection among rectangles (weight = area), then uniform point within the selected rectangle. Combines weighted sampling with geometry.
  4. Codeforces 364D – Ghd
    Find the greatest GCD of a subset of size ≥ n/2. The key insight is to randomly pick an element (with probability ≥ 1/2 it belongs to the optimal subset) and check divisors. Reservoir sampling enables the random selection.
  5. Codeforces 1523D – Love-Hate
    Find a bitmask liked by at least half the population. Random sampling to find a representative element, then bitwise analysis. Beautiful application of randomized approach.
  6. Codeforces 1110H – Modest Substrings
    Requires careful use of randomized techniques. Good for advanced practice.
  7. AtCoder ABC232 G – Modulo Shortest Path
    While not directly about reservoir sampling, the problem benefits from randomized approaches and understanding of streaming algorithms.
Study Strategy: Master the k = 1 case first (LeetCode 382/398), then implement the general k-item version. Once comfortable, study the weighted variant (A-Res) and think about how reservoir sampling can be a building block in randomized CP solutions.
