Random Hashing & Birthday Paradox

September 2026 · 22 min read

The Birthday Paradox

The birthday paradox is one of the most counter-intuitive results in probability theory, and it underpins nearly every collision analysis in hashing. The question is deceptively simple: how many people must be in a room before there is a >50% chance that two of them share a birthday?

Most people guess around 180—after all, there are 365 possible birthdays, so surely you need roughly half that many people. The real answer is shockingly small: just 23 people. With 23 people in a room, the probability that at least two share a birthday exceeds 50%.

The key insight is that we are not asking whether a specific person shares a birthday with someone else. We are asking whether any pair among all people shares a birthday. With n people, there are C(n, 2) = n(n−1)/2 pairs. For n = 23, that is 253 pairs—each an independent opportunity for a collision.

Quick Approximation: For n items hashed into m slots, the probability of at least one collision is approximately:

P(collision) ≈ 1 − e−n²/(2m)

Setting this to 0.5 and solving: n ≈ √(2m · ln 2) ≈ 1.177√m. For m = 365: n ≈ 1.177 × √365 ≈ 22.5, confirming the birthday paradox result.

This Θ(√m) scaling is what makes the birthday paradox so powerful—and so dangerous for hash-based data structures. If your hash function maps to m possible values, you only need about √m elements before collisions become likely. For a 32-bit hash (m = 232 ≈ 4 × 109), collisions are expected after roughly 216 = 65,536 insertions.

Exact Derivation & Proof

Let us derive the exact probability of no collision when n items are placed uniformly and independently into m slots. This is the complement of what we want (at least one collision), and it is easier to compute directly.

The first item can go into any of the m slots. The second item must avoid the first item's slot, so it has (m − 1) valid slots out of m. The third must avoid two occupied slots: (m − 2)/m. Continuing this pattern:

P(no collision) = 1 · (1 − 1/m) · (1 − 2/m) · … · (1 − (n−1)/m)

In product notation:

P(no collision) = ∏i=0n−1 (1 − i/m) = m! / ((m − n)! · mn)

And therefore: P(at least one collision) = 1 − m! / ((m − n)! · mn).

To obtain the approximation, we use the inequality ln(1 − x) ≈ −x for small x:

ln P(no collision) = ∑i=0n−1 ln(1 − i/m) ≈ −∑i=0n−1 i/m = −n(n−1)/(2m)

So P(no collision) ≈ e−n(n−1)/(2m) ≈ e−n²/(2m) for large n, which gives us the standard approximation.

Here is C++ code that computes the exact birthday probability for given n and m:

#include <bits/stdc++.h>
using namespace std;

// Exact probability of at least one collision
// n items, m slots
double birthday_exact(int n, int m) {
    if (n > m) return 1.0;
    double p_no_collision = 1.0;
    for (int i = 1; i < n; i++) {
        p_no_collision *= (double)(m - i) / m;
    }
    return 1.0 - p_no_collision;
}

// Approximation: 1 - exp(-n^2 / (2m))
double birthday_approx(int n, int m) {
    return 1.0 - exp(-(double)n * n / (2.0 * m));
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // Classic birthday problem: m = 365
    int m = 365;
    for (int n = 1; n <= 60; n++) {
        double exact = birthday_exact(n, m);
        double approx = birthday_approx(n, m);
        if (n % 5 == 0 || n == 23) {
            cout << "n=" << n
                 << " exact=" << fixed << setprecision(6) << exact
                 << " approx=" << approx << "\n";
        }
    }
    return 0;
}
Precision Warning: For large n and m, the product of many terms close to 1.0 can lose precision due to floating-point arithmetic. Use logarithms (sum of log terms) for better numerical stability in competitive programming.

Hash Collision Probability

The birthday paradox has direct implications for hash tables and hash-based algorithms. When we insert n keys into a hash table with m slots using a uniform hash function, the probability of at least one collision follows the birthday formula exactly.

Key thresholds for hash sizes:

  • 32-bit hash (m = 232): Expect collisions after ~216 = 65,536 items
  • 64-bit hash (m = 264): Expect collisions after ~232 ≈ 4 × 109 items
  • 128-bit hash (m = 2128): Expect collisions after ~264 items

For competitive programming, where n is typically up to 105 or 106, a single modulus of around 109 gives collision probability roughly n2/(2 × 109). For n = 105, this is about 1010/(2 × 109) = 5, meaning collisions are almost certain. For n = 103, it is about 106/(2 × 109) = 0.0005, which is usually acceptable.

This analysis motivates the use of larger moduli or multiple independent hash functions, as we will discuss in the double hashing section.

When using hash maps in CP (e.g., unordered_map), the default hash function is often weak. Here is how to estimate the expected number of collisions. If n keys are hashed into m buckets uniformly, the expected number of colliding pairs is:

E[collisions] = C(n, 2) / m = n(n − 1) / (2m) 

#include <bits/stdc++.h>
using namespace std;

// Simulate hash collisions to verify birthday analysis
int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    mt19937_64 rng(chrono::steady_clock::now().time_since_epoch().count());

    int trials = 100000;
    long long m = 1000000007LL; // hash space size
    int n = 1000;               // number of items

    int collision_count = 0;
    for (int t = 0; t < trials; t++) {
        set<long long> seen;
        bool collision = false;
        for (int i = 0; i < n; i++) {
            long long h = rng() % m;
            if (seen.count(h)) {
                collision = true;
                break;
            }
            seen.insert(h);
        }
        if (collision) collision_count++;
    }

    double empirical = (double)collision_count / trials;
    double theoretical = 1.0 - exp(-(double)n * n / (2.0 * m));
    cout << "Empirical collision rate: " << empirical << "\n";
    cout << "Theoretical (approx):    " << theoretical << "\n";
    return 0;
}

Polynomial Hashing

Polynomial hashing (also called Rabin fingerprinting) is the workhorse of string hashing in competitive programming. Given a string s of length n, we define:

h(s) = (∑i=0n−1 s[i] · pi) mod M

where p is the base (a prime, typically chosen from the range [31, 109]) and M is a large modulus (commonly 109 + 7 or 109 + 9).

The hash of a substring s[l..r] can be extracted in O(1) using prefix hashes. Let H[i] = h(s[0..i−1]). Then:

h(s[l..r]) = (H[r+1] − H[l] · pr−l+1) mod M

Why polynomial hashing works: If two distinct strings of length L collide under a random base p chosen uniformly from ZM, then p is a root of a polynomial of degree at most L − 1 over ZM. By the Schwartz-Zippel lemma, such a polynomial has at most L − 1 roots. So the collision probability for two specific strings is at most (L − 1)/M, which is very small when M is large.

Schwartz-Zippel Bound: For a non-zero polynomial f of degree d over a field of size q, the probability that f(r) = 0 for a uniformly random r is at most d/q. This is the theoretical backbone of polynomial hashing security. 
#include <bits/stdc++.h>
using namespace std;

const long long MOD = 1e9 + 7;

struct PolyHash {
    int n;
    vector<long long> h, pw;
    long long base;

    PolyHash(const string& s, long long b) : n(s.size()), h(s.size()+1), pw(s.size()+1), base(b) {
        pw[0] = 1;
        for (int i = 0; i < n; i++) {
            h[i+1] = (h[i] + (long long)s[i] * pw[i]) % MOD;
            pw[i+1] = pw[i] * base % MOD;
        }
    }

    // hash of s[l..r] inclusive
    long long query(int l, int r) {
        long long val = (h[r+1] - h[l] + MOD) % MOD;
        // normalize by dividing out p^l (multiply by inverse)
        // alternatively, compare: hash(l1,r1)*pw[l2] == hash(l2,r2)*pw[l1]
        return val;
    }

    // compare substrings without modular inverse
    bool equal(int l1, int r1, int l2, int r2) {
        if (r1 - l1 != r2 - l2) return false;
        long long h1 = (h[r1+1] - h[l1] + MOD) % MOD;
        long long h2 = (h[r2+1] - h[l2] + MOD) % MOD;
        // h1 * pw[l2] == h2 * pw[l1] means same substring
        return h1 % MOD * pw[l2] % MOD == h2 % MOD * pw[l1] % MOD;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
    long long base = uniform_int_distribution<long long>(256, MOD - 1)(rng);

    string s;
    cin >> s;
    PolyHash ph(s, base);

    int q;
    cin >> q;
    while (q--) {
        int l1, r1, l2, r2;
        cin >> l1 >> r1 >> l2 >> r2;
        cout << (ph.equal(l1, r1, l2, r2) ? "Yes" : "No") << "\n";
    }
    return 0;
}

Anti-Hash Tests

In competitive programming, problem setters often create anti-hash tests—adversarial inputs designed to cause hash collisions and force solutions using weak hashing into TLE or WA. Understanding how these attacks work is essential for writing robust solutions.

How adversaries break polynomial hashing:

  1. Known base attack: If the base p is fixed and known (e.g., p = 31), the adversary can construct pairs of strings that collide. Find two characters a, b such that a · pi ≡ b · pi (mod M), then swap them to create collisions.
  2. Thue-Morse sequence: The classic technique generates 2k strings that pairwise have the same hash for a specific fixed base, by exploiting the Thue-Morse binary sequence.
  3. Birthday attack on small modulus: Generate √M random strings and check for collisions. For M = 109 + 7, only ~31,623 random strings are needed to find a collision with high probability.
Warning: Never use a fixed, publicly known hash base in competitive programming. Problem setters will hack your solution. Always randomize the base at runtime.

Countermeasures against anti-hash tests:

  • Randomize the base p at runtime using a random number generator seeded with time or address
  • Use two independent hash functions (double hashing) to reduce collision probability to ~1/M2
  • Use a prime modulus to ensure the hash polynomial is over a field (guaranteeing the Schwartz-Zippel bound)
  • Avoid moduli that are powers of 2—these are especially vulnerable because low-order bits of the polynomial dominate
#include <bits/stdc++.h>
using namespace std;

// Demonstration: generating colliding strings for a FIXED base
// This shows WHY you must randomize your base
void generate_collision_pair(long long base, long long mod) {
    // Find two 2-char strings with the same hash
    // h("ab") = a + b*base, h("cd") = c + d*base
    // Collision: a + b*base = c + d*base (mod mod)
    // (a - c) = (d - b)*base (mod mod)
    for (int a = 'a'; a <= 'z'; a++)
    for (int b = 'a'; b <= 'z'; b++)
    for (int c = 'a'; c <= 'z'; c++)
    for (int d = 'a'; d <= 'z'; d++) {
        if (a == c && b == d) continue;
        long long h1 = (a + (long long)b * base) % mod;
        long long h2 = (c + (long long)d * base) % mod;
        if (h1 == h2) {
            cout << "Collision: \"" << (char)a << (char)b
                 << "\" and \"" << (char)c << (char)d << "\"\n";
            return;
        }
    }
    cout << "No collision found among 2-char strings\n";
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long long fixed_base = 31;
    long long mod = 1e9 + 7;
    generate_collision_pair(fixed_base, mod);
    return 0;
}

Double Hashing

Double hashing is the standard defense against birthday-bound collisions and adversarial inputs. The idea is simple: compute two independent polynomial hashes with different bases and/or different moduli, and declare two strings equal only if both hashes match.

If hash function 1 has collision probability p1 ≤ L/M1 and hash function 2 has collision probability p2 ≤ L/M2, and they are independent, then the combined collision probability is at most p1 · p2 ≤ L2/(M1 · M2).

With two moduli of ~109, the effective hash space is ~1018, and by the birthday bound, collisions are not expected until ~109 strings—far beyond typical CP constraints. 

#include <bits/stdc++.h>
using namespace std;

const long long MOD1 = 1e9 + 7;
const long long MOD2 = 1e9 + 9;

struct DoubleHash {
    int n;
    vector<long long> h1, h2, pw1, pw2;
    long long b1, b2;

    DoubleHash(const string& s, long long base1, long long base2)
        : n(s.size()), h1(s.size()+1), h2(s.size()+1),
          pw1(s.size()+1), pw2(s.size()+1), b1(base1), b2(base2)
    {
        pw1[0] = pw2[0] = 1;
        h1[0] = h2[0] = 0;
        for (int i = 0; i < n; i++) {
            h1[i+1] = (h1[i] + (long long)s[i] * pw1[i]) % MOD1;
            h2[i+1] = (h2[i] + (long long)s[i] * pw2[i]) % MOD2;
            pw1[i+1] = pw1[i] * b1 % MOD1;
            pw2[i+1] = pw2[i] * b2 % MOD2;
        }
    }

    pair<long long, long long> get(int l, int r) {
        long long v1 = (h1[r+1] - h1[l] + MOD1) % MOD1;
        long long v2 = (h2[r+1] - h2[l] + MOD2) % MOD2;
        return {v1, v2};
    }

    bool equal(int l1, int r1, int l2, int r2) {
        if (r1 - l1 != r2 - l2) return false;
        long long a1 = (h1[r1+1] - h1[l1] + MOD1) % MOD1;
        long long a2 = (h1[r2+1] - h1[l2] + MOD1) % MOD1;
        long long b1v = (h2[r1+1] - h2[l1] + MOD2) % MOD2;
        long long b2v = (h2[r2+1] - h2[l2] + MOD2) % MOD2;
        return a1 * pw1[l2] % MOD1 == a2 * pw1[l1] % MOD1
            && b1v * pw2[l2] % MOD2 == b2v * pw2[l1] % MOD2;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());
    long long base1 = uniform_int_distribution<long long>(256, MOD1 - 1)(rng);
    long long base2 = uniform_int_distribution<long long>(256, MOD2 - 1)(rng);

    string s;
    cin >> s;
    DoubleHash dh(s, base1, base2);

    int q;
    cin >> q;
    while (q--) {
        int l1, r1, l2, r2;
        cin >> l1 >> r1 >> l2 >> r2;
        cout << (dh.equal(l1, r1, l2, r2) ? "Yes" : "No") << "\n";
    }
    return 0;
}
Rule of Thumb: In competitive programming, always use double hashing with randomized bases unless you can prove a single hash suffices (e.g., small n where birthday probability is negligible). The overhead is just 2× and the safety margin is enormous.

CP Applications

Hash collision analysis and the birthday paradox show up in many competitive programming contexts beyond straightforward string matching. Here are the most important applications:

1. String Matching (Rabin-Karp)

Given a text T and a pattern P, compute the polynomial hash of P and slide a window over T, comparing hashes. Expected O(n + m) with high probability when using randomized double hashing. This is the canonical application of polynomial hashing.

2. Number of Distinct Substrings

Hash all O(n2) substrings and count distinct hash values. With double hashing, the false positive rate is negligible. Can also be solved with suffix arrays, but hashing is simpler to implement.

3. Longest Common Substring

Binary search on the length L. For each L, hash all length-L substrings of both strings and check for a common hash. O(n log n) with hashing.

4. Palindrome Detection

Maintain both a forward hash and a reverse hash. A substring s[l..r] is a palindrome iff hash_forward(l, r) == hash_reverse(l, r). Useful for problems like longest palindromic substring.

5. Tree Isomorphism via Hashing

Hash rooted trees by combining children hashes. Two subtrees are isomorphic iff their hashes match. The birthday analysis tells us the probability of a false positive and guides our choice of hash space size.

6. Cycle Detection in Functional Graphs

The birthday paradox directly relates to Floyd’s cycle detection algorithm. In a random functional graph on m nodes, the expected cycle length is Θ(√m)—the same birthday bound.
