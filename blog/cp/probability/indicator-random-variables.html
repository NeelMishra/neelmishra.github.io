Indicator Random Variables

June 2026 · 18 min read

Definition & Basics

An indicator random variable (also called a Bernoulli random variable) is the simplest possible random variable. It takes only two values — 0 or 1 — acting as a binary flag for whether some event occurs.

Formally, given an event A in a sample space Ω, we define:

XA(ω) = 1   if ω ∈ A
XA(ω) = 0   if ω ∉ A

The beauty of indicator variables lies in their expectation. Since XA only takes values 0 and 1:

E[XA] = 1 · P(XA = 1) + 0 · P(XA = 0)
E[XA] = P(A)
Key Identity: The expected value of an indicator random variable equals the probability of its underlying event. This one-line identity, E[XA] = P(A), is the engine behind every technique in this post.

Indicator variables also have neat algebraic properties. If X is an indicator for event A, then:

  • X² = X — since 0² = 0 and 1² = 1.
  • 1 − X is the indicator for the complement Ac.
  • X · Y is the indicator for A ∩ B when Y indicates event B.
  • Var(X) = P(A)(1 − P(A)) = p(1 − p), which is maximized at p = 1/2.

These properties make indicators the ideal building blocks for decomposing complex counting problems into simple, manageable pieces.

The Indicator Trick

The indicator trick is arguably the single most powerful technique in probabilistic combinatorics. The idea is disarmingly simple: take a complicated count, express it as a sum of indicator variables, then apply linearity of expectation.

Suppose we want to find the expected number of “interesting” objects in some random configuration. We define:

X = total count of interesting objects
Xi = 1 if object i is interesting, 0 otherwise

Then: X = X1 + X2 + … + Xn

By linearity of expectation:
E[X] = E[X1] + E[X2] + … + E[Xn]
E[X] = P(event 1) + P(event 2) + … + P(event n)
E[X] = Σi=1..n P(object i is interesting)
Why is this powerful? The Xi's do not need to be independent! Linearity of expectation holds for any random variables, regardless of dependence. This is what makes the trick so broadly applicable — we can decompose sums without worrying about correlation.

The workflow for applying the indicator trick in competitive programming is always the same three steps:

  1. Identify what to count. What quantity X do you need the expectation of?
  2. Decompose into indicators. Write X = Σ Xi, where each Xi flags one “unit” contribution.
  3. Compute each P(Xi = 1). This is usually a straightforward combinatorial probability.

Let’s see this in action with several classic examples, starting from the most fundamental — counting inversions.

Counting Inversions

Problem: Given a uniformly random permutation of {1, 2, …, n}, what is the expected number of inversions? (An inversion is a pair (i, j) with i < j but π(i) > π(j).)

This is a textbook application of indicator variables. For each pair (i, j) with 1 ≤ i < j ≤ n, define:

Xij = 1 if π(i) > π(j),   0 otherwise

The total number of inversions is:

X = Σi<j Xij

Now, in a uniformly random permutation, for any pair of distinct positions (i, j), the relative order of π(i) and π(j) is equally likely to be either way. Therefore P(Xij = 1) = 1/2 for every pair.

E[X] = Σi<j E[Xij]
     = Σi<j 1/2
     = C(n, 2) · 1/2
     = n(n−1)/4
Result: A random permutation of n elements has n(n−1)/4 expected inversions. For n = 100, that’s 2475 inversions on average. Note this also equals half the maximum number of inversions C(n,2).

Here is a C++ solution that computes the expected inversions modulo a prime, handling the division by 4 with modular inverse:

#include <bits/stdc++.h>
using namespace std;

const long long MOD = 998244353;

long long power(long long base, long long exp, long long mod) {
    long long result = 1;
    base %= mod;
    while (exp > 0) {
        if (exp & 1) result = result * base % mod;
        base = base * base % mod;
        exp >>= 1;
    }
    return result;
}

long long modinv(long long a, long long mod) {
    return power(a, mod - 2, mod);
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long long n;
    cin >> n;

    // E[inversions] = n*(n-1)/4 mod p
    long long num = n % MOD * ((n - 1) % MOD) % MOD;
    long long ans = num % MOD * modinv(4, MOD) % MOD;
    cout << ans << "\n";

    return 0;
}

Verification by simulation: We can also verify this result empirically. The following code counts inversions via merge sort on random permutations and averages the result: 

#include <bits/stdc++.h>
using namespace std;

long long merge_count(vector<int>& arr, int l, int r) {
    if (r - l <= 1) return 0;
    int mid = (l + r) / 2;
    long long cnt = merge_count(arr, l, mid) + merge_count(arr, mid, r);
    vector<int> tmp;
    int i = l, j = mid;
    while (i < mid && j < r) {
        if (arr[i] <= arr[j]) tmp.push_back(arr[i++]);
        else { tmp.push_back(arr[j++]); cnt += mid - i; }
    }
    while (i < mid) tmp.push_back(arr[i++]);
    while (j < r) tmp.push_back(arr[j++]);
    copy(tmp.begin(), tmp.end(), arr.begin() + l);
    return cnt;
}

int main() {
    mt19937 rng(42);
    int n = 100, trials = 100000;
    double total = 0;
    for (int t = 0; t < trials; t++) {
        vector<int> perm(n);
        iota(perm.begin(), perm.end(), 1);
        shuffle(perm.begin(), perm.end(), rng);
        total += merge_count(perm, 0, n);
    }
    cout << fixed << setprecision(2);
    cout << "Empirical mean: " << total / trials << "\n";
    cout << "Theoretical:    " << n * (n - 1) / 4.0 << "\n";
    return 0;
}

Expected Number of Records

Problem: In a random permutation π of {1, …, n}, element π(i) is called a record (or left-to-right maximum) if π(i) > π(j) for all j < i. What is the expected number of records?

Define the indicator:

Xi = 1 if π(i) is a record, 0 otherwise

Element π(i) is a record if and only if it is the maximum of {π(1), π(2), …, π(i)}. In a random permutation, each of these i elements is equally likely to be the maximum. Therefore:

P(Xi = 1) = 1/i

By linearity:

E[number of records] = Σi=1..n 1/i = Hn

where Hn = 1 + 1/2 + 1/3 + … + 1/n is the n-th harmonic number. Asymptotically, Hn ≈ ln(n) + γ where γ ≈ 0.5772 is the Euler–Mascheroni constant.

Insight: Out of a million elements in random order, the expected number of records is only about H1000000 ≈ 14.39. Records are logarithmically rare! This result appears frequently in the analysis of algorithms (e.g., expected comparisons in quicksort relate to harmonic numbers).

Hat-check / Derangement Problem: Another beautiful application of indicators. Suppose n people check their hats, and hats are returned uniformly at random. Let X be the number of people who get their own hat back.

Xi = 1 if person i gets their own hat
P(Xi = 1) = 1/n

E[X] = Σi=1..n 1/n = 1

Remarkably, regardless of n, exactly 1 person is expected to get their own hat back. This holds for n = 10 and n = 109 alike. The indicators Xi are not independent (if person 1 gets hat 2, person 2 is less likely to get hat 2), but linearity of expectation doesn’t care. 

#include <bits/stdc++.h>
using namespace std;

const long long MOD = 998244353;

long long power(long long b, long long e, long long m) {
    long long r = 1; b %= m;
    while (e > 0) { if (e & 1) r = r * b % m; b = b * b % m; e >>= 1; }
    return r;
}

// Expected number of records = H_n mod p
int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;

    long long ans = 0;
    for (int i = 1; i <= n; i++) {
        ans = (ans + power(i, MOD - 2, MOD)) % MOD;
    }
    cout << ans << "\n";
    return 0;
}

Birthday Problem via Indicators

The classic birthday problem asks: among n people with birthdays uniformly distributed over m = 365 days, what is the expected number of colliding pairs?

Most presentations focus on the probability that some collision exists. But with indicators, we can directly compute the expected number of collisions — a much more informative quantity.

For each pair (i, j) with i < j, define:

Xij = 1 if person i and person j share a birthday

Each person’s birthday is uniform over m days, so:

P(Xij = 1) = 1/m

The expected number of colliding pairs is:

E[collisions] = Σi<j 1/m = C(n, 2) / m = n(n−1) / (2m)
Birthday Numbers: With 23 people and m = 365 days, the expected number of colliding pairs is 23 · 22 / 730 ≈ 0.693. The fact that this is close to ln(2) is no coincidence — it connects to the threshold where the probability of at least one collision crosses 50%.

Generalizing to hash collisions: In competitive programming, birthday-style analysis appears when estimating collision probabilities for hashing. If you insert n elements into a hash table of size m, the expected number of collisions is n(n−1)/(2m). To keep collisions below 1 in expectation, you need m > n(n−1)/2 ≈ n²/2.

Multi-collision variant: What if we want the expected number of days that have at least 2 birthdays? Define Yd = 1 if day d has ≥ 2 people. Then P(Yd = 1) = 1 − P(0 people on day d) − P(1 person on day d).

P(Yd = 1) = 1 − ((m−1)/m)n − n · (1/m) · ((m−1)/m)n−1

E[days with collision] = m · P(Yd = 1) 
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m;
    cin >> n >> m;

    // Expected colliding pairs = n*(n-1) / (2*m)
    // Simulate to verify
    mt19937 rng(42);
    int trials = 500000;
    double total_pairs = 0;

    for (int t = 0; t < trials; t++) {
        vector<int> cnt(m, 0);
        for (int i = 0; i < n; i++) cnt[rng() % m]++;
        long long pairs = 0;
        for (int c : cnt) pairs += (long long)c * (c - 1) / 2;
        total_pairs += pairs;
    }

    cout << fixed << setprecision(4);
    cout << "Empirical:   " << total_pairs / trials << "\n";
    cout << "Theoretical: " << (double)n * (n - 1) / (2.0 * m) << "\n";
    return 0;
}

CP Applications

Indicator random variables appear constantly in competitive programming expected-value problems. Here are the major patterns you should recognize.

Pattern 1: Expected Matching Pairs

Given an array of n elements, if you randomly shuffle it, what is the expected number of positions i where a[i] equals some target value? If there are k copies of the target, then E[matches at position i] = k/n, and E[total matches] = n · k/n = k. Trivial but the technique generalizes to harder settings.

Pattern 2: Expected Distinct Values After Sampling

You draw k items uniformly at random (with replacement) from {1, …, n}. What is the expected number of distinct values? Define Xv = 1 if value v appears at least once. Then P(Xv = 1) = 1 − ((n−1)/n)k and:

E[distinct] = n · (1 − ((n−1)/n)k)

Pattern 3: Expected Edges in Random Graphs

In G(n, p), each possible edge exists independently with probability p. Define Xe for each potential edge. Then E[edges] = C(n,2) · p. Similarly, E[triangles] = C(n,3) · p³, since each triangle needs all 3 edges.

Pattern 4: Contribution Technique

Many CP problems ask: “compute the sum of f(S) over all subsets S.” If f(S) can be written as a sum of per-element indicators, you can swap the order of summation. Each element contributes independently, and the total sum becomes Σi (number of subsets where element i contributes). 

#include <bits/stdc++.h>
using namespace std;

const int MOD = 998244353;

long long power(long long b, long long e, long long m) {
    long long r = 1; b %= m;
    while (e > 0) { if (e & 1) r = r * b % m; b = b * b % m; e >>= 1; }
    return r;
}

// Expected distinct values when sampling k times from {1..n}
// E = n * (1 - ((n-1)/n)^k) mod p
int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long long n, k;
    cin >> n >> k;

    // Compute ((n-1)/n)^k = (n-1)^k * inv(n)^k mod p
    long long nm1_k = power(n - 1, k, MOD);
    long long n_k = power(n, k, MOD);
    long long inv_nk = power(n_k, MOD - 2, MOD);
    long long ratio = nm1_k % MOD * inv_nk % MOD;
    long long ans = n % MOD * ((1 - ratio + MOD) % MOD) % MOD;
    cout << ans << "\n";

    return 0;
}
Common Pitfall: When computing expected values modulo a prime, remember that “expected value mod p” means computing the exact rational answer a/b and outputting a · b−1 mod p. Do not try to take expectations of modular quantities — that is mathematically meaningless.

Practice Problems

Work through these problems to build fluency with indicator random variables:

  1. Random Permutation Cycles: In a uniformly random permutation of n elements, what is the expected number of cycles? (Hint: element i starts a new cycle iff it is the smallest in its cycle. P = 1/i by symmetry. Answer: Hn.)
  2. Expected Fixed Points of Composed Permutations: Let π and σ be independent random permutations of {1,…,n}. What is E[number of i such that π(σ(i)) = i]? (Hint: indicator for each i.)
  3. CF 1Expected Value Problem (CF 280C): Tree problem where you pick random nodes and delete subtrees. Model each node’s survival with an indicator variable.
  4. Random Binary String: A string of n bits is chosen uniformly at random. What is the expected number of substrings equal to “01”? Use indicators over each position pair.
  5. Expected Inversions After k Swaps: Starting from identity permutation, perform k random adjacent swaps. Each swap either creates or removes an inversion with certain probability. Track E[inversions] as a recurrence.
  6. Hashing Collisions: You have n strings hashed into a table of size m. Using indicators over pairs, compute the expected number of collisions. Determine the minimum m so that E[collisions] < 1.
Study Strategy: For each problem, follow the three-step workflow: (1) identify what to count, (2) decompose into indicator variables, (3) compute each indicator’s probability. With practice, this becomes second nature and you’ll spot indicator decompositions instantly during contests.
