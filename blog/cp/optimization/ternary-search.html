    Ternary Search

    July 2025 · 18 min read

    What Is Ternary Search?

    Binary search divides a monotonic search space in half each iteration. Ternary search extends this idea to unimodal functions — functions that first increase then decrease (or vice-versa), having a single peak or valley. By evaluating the function at two interior points each iteration, we can discard a third of the remaining interval, converging on the extremum.

    Unimodal function: A function f defined on [a, b] is unimodal if there exists a point m in [a, b] such that f is strictly increasing on [a, m] and strictly decreasing on [m, b] (for a maximum). The goal is to find m.

    Consider a mountain-shaped curve. We pick two probe points, m1 and m2, that trisect the interval [lo, hi]:

    • m1 = lo + (hi - lo) / 3
    • m2 = hi - (hi - lo) / 3

    Then we compare f(m1) vs f(m2):

    • If f(m1) < f(m2), the maximum cannot lie in [lo, m1], so we set lo = m1.
    • If f(m1) > f(m2), the maximum cannot lie in [m2, hi], so we set hi = m2.
    • If f(m1) == f(m2), both sides can be trimmed: lo = m1, hi = m2.

    Each iteration reduces the interval to at most 2/3 of its previous size, giving O(log1.5 n) convergence — slightly slower than binary search's O(log2 n) but applicable to a wider class of problems.

    Ternary Search on Integers

    When the domain is discrete (e.g., finding the peak index in an array), we apply ternary search on integer indices. The loop terminates when the interval is small enough (typically hi - lo < 3), and we check the remaining candidates directly.

    Peak of a Unimodal Array

    Given an array where values increase up to some peak index then decrease, find that peak:

    #include <bits/stdc++.h>
using namespace std;

// Find peak index of a unimodal array a[0..n-1]
int ternarySearchPeak(const vector<int>& a) {
    int lo = 0, hi = (int)a.size() - 1;
    while (hi - lo > 2) {
        int m1 = lo + (hi - lo) / 3;
        int m2 = hi - (hi - lo) / 3;
        if (a[m1] < a[m2])
            lo = m1 + 1;   // peak is to the right of m1
        else
            hi = m2 - 1;   // peak is to the left of m2
    }
    // Check remaining 1–3 elements
    int best = lo;
    for (int i = lo + 1; i <= hi; i++)
        if (a[i] > a[best]) best = i;
    return best;
}

int main() {
    vector<int> a = {1, 4, 7, 11, 15, 13, 9, 6, 2};
    cout << "Peak index: " << ternarySearchPeak(a) << "\n";
    // Output: Peak index: 4 (value 15)
    return 0;
}
    Tip: For the special case of a bitonic array (strictly increasing then strictly decreasing), you can also use binary search on the derivative: check if a[mid] < a[mid+1]. But ternary search generalizes to arbitrary unimodal evaluation functions where you cannot compute a "derivative."

    Minimizing a Discrete Function

    Suppose you need to minimize a cost function cost(x) that is convex on integers in [0, N]. The same template works — just flip the comparison:

    // Minimize cost(x) for x in [lo, hi] where cost is unimodal (valley-shaped)
long long ternarySearchMin(int lo, int hi, function<long long(int)> cost) {
    while (hi - lo > 2) {
        int m1 = lo + (hi - lo) / 3;
        int m2 = hi - (hi - lo) / 3;
        if (cost(m1) < cost(m2))
            hi = m2 - 1;   // minimum is in [lo, m2-1]
        else
            lo = m1 + 1;   // minimum is in [m1+1, hi]
    }
    long long best = LLONG_MAX;
    int bestX = lo;
    for (int x = lo; x <= hi; x++) {
        long long c = cost(x);
        if (c < best) { best = c; bestX = x; }
    }
    return bestX;
}

    Ternary Search on Reals

    On a continuous domain, we iterate until the interval is smaller than a desired precision ε. This is the most common competitive-programming variant.

    // Find x in [lo, hi] that maximizes f(x) (unimodal, continuous)
double ternarySearchReal(double lo, double hi,
                         function<double(double)> f,
                         double eps = 1e-9) {
    // Run for a fixed number of iterations for safety
    for (int iter = 0; iter < 200; iter++) {
        if (hi - lo < eps) break;
        double m1 = lo + (hi - lo) / 3.0;
        double m2 = hi - (hi - lo) / 3.0;
        if (f(m1) < f(m2))
            lo = m1;
        else
            hi = m2;
    }
    return (lo + hi) / 2.0;
}
    Why 200 iterations? Each iteration shrinks the interval by a factor of 2/3. Starting from a range of 1018 and running 200 iterations gives roughly (2/3)200 · 1018 ≈ 10-17, far beyond the precision of double. Using a fixed iteration count avoids subtle infinite-loop bugs from floating-point edge cases.

    Convergence Rate

    After k iterations the interval width is (2/3)k · (hi − lo). To reach precision ε:

    k = ⌈log(ε / (hi − lo)) / log(2/3)⌉ ≈ 2.41 · log₂((hi − lo) / ε)

    Compared to binary search which needs log₂((hi − lo) / ε) iterations with one function call each, ternary search needs ~2.41× as many iterations with two calls each — so roughly 4.82× more function evaluations. This is acceptable for O(1) or O(log n) evaluation functions but can matter for expensive evaluations.

    Choosing ε Carefully

    Common choices:

    • Absolute ε: Use when the answer range is known. For example, if the answer is between 0 and 106, use eps = 1e-9.
    • Relative ε: Use while ((hi - lo) / max(1.0, abs(lo + hi)) > 1e-9) when the answer magnitude varies.
    • Fixed iterations: Safest. 100–200 iterations is almost always enough for double precision.

    Common Pitfalls

    1. Function Is Not Truly Unimodal

    Ternary search gives wrong answers on multimodal functions — those with multiple local peaks. If your function has flat regions or multiple extrema, ternary search may converge to a local, not global, optimum.

    Warning: Always verify that the problem guarantees unimodality. Common sources of unimodality: convex/concave cost functions, distance-based minimization, and problems with a single "turning point."

    2. Precision and Infinite Loops

    Using while (hi - lo > eps) alone with very small ε can cause infinite loops due to floating-point rounding. The interval might never actually shrink below ε. Always add an iteration cap:

    // BAD: may loop forever with tiny eps
while (hi - lo > 1e-12) { ... }

// GOOD: iteration cap guarantees termination
for (int iter = 0; iter < 300 && hi - lo > 1e-12; iter++) { ... }

    3. Off-by-One in Integer Ternary Search

    Be careful with m1 + 1 vs m1. If you set lo = m1 instead of lo = m1 + 1, the interval might not shrink when hi - lo == 3, causing an infinite loop. Test with small arrays (size 3, 4) to verify correctness.

    4. Equal Probe Values

    When f(m1) == f(m2), you can safely shrink from both sides: lo = m1 + 1; hi = m2 - 1; (for max search on integers). On reals, set lo = m1; hi = m2;. Some implementations only shrink one side, which is safe but slower.

    5. Expensive Function Evaluation

    Each ternary search iteration calls f twice. If f itself is O(n), and the domain has range R, the total cost is O(n · log R). Consider whether a different approach (e.g., golden section search, which reuses one evaluation) would be faster.

    Classic Problems

    Problem 1: Minimize Maximum Distance (Codeforces 578C – "Weakness and Poorness")

    Given a sequence of values, find a parameter t that minimizes the maximum-sum subarray of (ai − t). The function "max subarray sum as a function of t" is unimodal (convex), so we can ternary search on t.

    #include <bits/stdc++.h>
using namespace std;

int n;
double a[200005];

// Max subarray sum of (a[i] - t) via Kadane's algorithm
double maxSubarraySum(double t) {
    double cur = 0, best = -1e18;
    for (int i = 0; i < n; i++) {
        cur = max(a[i] - t, cur + a[i] - t);
        best = max(best, cur);
    }
    return best;
}

// "Weakness" = max(maxSubarraySum(t), -minSubarraySum(t))
// This is a convex function of t, so ternary search works.
double weakness(double t) {
    double cur = 0, best = -1e18;
    double curNeg = 0, bestNeg = -1e18;
    for (int i = 0; i < n; i++) {
        double v = a[i] - t;
        cur = max(v, cur + v);
        best = max(best, cur);
        curNeg = max(-v, curNeg - v);
        bestNeg = max(bestNeg, curNeg);
    }
    return max(best, bestNeg);
}

int main() {
    scanf("%d", &n);
    for (int i = 0; i < n; i++) scanf("%lf", &a[i]);

    double lo = -1e5, hi = 1e5;
    for (int iter = 0; iter < 200; iter++) {
        double m1 = lo + (hi - lo) / 3.0;
        double m2 = hi - (hi - lo) / 3.0;
        if (weakness(m1) < weakness(m2))
            hi = m2;
        else
            lo = m1;
    }
    printf("%.15f\n", weakness((lo + hi) / 2.0));
    return 0;
}

    Problem 2: Farthest Point on a Path (Geometry)

    Given a convex polygon and a point P outside it, find the point on the polygon boundary closest to P. The distance function along the boundary is unimodal, so ternary search applies.

    // Ternary search on a parameterized path
// path(t) returns the point at parameter t in [0, totalLength]
// dist(t) = ||path(t) - P|| is unimodal for convex shapes
double closestPointOnPath(double totalLen, Point P,
                          function<Point(double)> path) {
    double lo = 0, hi = totalLen;
    for (int iter = 0; iter < 200; iter++) {
        double m1 = lo + (hi - lo) / 3.0;
        double m2 = hi - (hi - lo) / 3.0;
        double d1 = dist(path(m1), P);
        double d2 = dist(path(m2), P);
        if (d1 < d2) hi = m2;
        else lo = m1;
    }
    return (lo + hi) / 2.0;
}

    Problem 3: Optimal Split Point

    Split an array into two parts to minimize max(sum of left part, sum of right part). The function "max(prefix_sum[i], total − prefix_sum[i])" is unimodal — it decreases then increases. Ternary search on the split point:

    #include <bits/stdc++.h>
using namespace std;

int main() {
    int n; cin >> n;
    vector<long long> a(n), pre(n + 1, 0);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
        pre[i + 1] = pre[i] + a[i];
    }
    long long total = pre[n];

    // cost(i) = max(pre[i], total - pre[i]), unimodal on [0, n]
    auto cost = [&](int i) -> long long {
        return max(pre[i], total - pre[i]);
    };

    int lo = 0, hi = n;
    while (hi - lo > 2) {
        int m1 = lo + (hi - lo) / 3;
        int m2 = hi - (hi - lo) / 3;
        if (cost(m1) < cost(m2))
            hi = m2;
        else
            lo = m1;
    }
    long long best = LLONG_MAX;
    for (int i = lo; i <= hi; i++)
        best = min(best, cost(i));
    cout << best << "\n";
    return 0;
}

    Complexity Analysis

    VariantIterationsFunction CallsTotal Complexity
    Integer, range N O(log1.5 N) ≈ O(1.71 · log N) 2 per iteration O(log N · T(f))
    Real, precision ε O(log1.5(R/ε)) 2 per iteration O(log(R/ε) · T(f))
    Golden Section Search O(logφ(R/ε)) 1 per iteration O(log(R/ε) · T(f)) — fewer evals

    Where T(f) is the cost of one function evaluation and R = hi − lo is the initial range.

    Ternary vs Golden Section: Golden section search reuses one probe point from the previous iteration, needing only 1 new evaluation per iteration instead of 2. For expensive functions, golden section is strictly better. In competitive programming, ternary search is preferred for its simplicity — the constant-factor difference rarely matters when T(f) is small.

    Space Complexity

    Ternary search uses O(1) extra memory — only the interval bounds and probe values. Even the recursive version has O(log N) stack depth, which is negligible.

    When to Prefer Ternary Search

    • Over binary search: When the function is unimodal but not monotonic.
    • Over gradient descent: When you need a guaranteed convergence without learning-rate tuning, or when the function isn't differentiable.
    • Over brute force: When the domain is large but the function is unimodal and evaluable in reasonable time.
    Contest tip: If you're unsure whether ternary search applies, check: does moving left from the answer always decrease f, and moving right always decrease f? If yes, it's unimodal. Plotting a few values can quickly confirm this.
