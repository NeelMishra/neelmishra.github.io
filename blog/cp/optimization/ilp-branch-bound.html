    Integer LP & Branch and Bound

    July 2026 · 22 min read

    In the previous posts we solved linear programs where variables could take any real value. But many real-world and contest problems demand integer solutions: you cannot build half a warehouse, assign 0.7 of a task to a machine, or pack 2.3 items into a knapsack. The moment we add the constraint that some or all variables must be integers, the problem jumps from polynomial-time solvable to NP-hard. This post covers the theory and practice of Integer Linear Programming (ILP) and the Branch and Bound algorithm — the workhorse method for solving ILP exactly.

    Integer Linear Programming

    An Integer Linear Program (ILP) has the same form as an LP — minimize or maximize a linear objective subject to linear constraints — but with the additional requirement that some or all decision variables take integer values.

    Formal Definition

    A pure integer linear program looks like:

    maximize    c^T x
subject to  A x <= b
            x >= 0
            x ∈ ℤⁿ          // all variables must be integers

    If only a subset of variables is required to be integer, we call it a Mixed-Integer Linear Program (MILP). A special and extremely common case is the 0/1 (binary) integer program, where every variable is restricted to {0, 1}.

    Why Integrality Makes LP Hard

    A continuous LP can be solved in polynomial time (via interior-point or ellipsoid methods) and in practice very efficiently by the simplex method. The feasible region is a convex polytope, and the optimum always sits at a vertex.

    Once we require integer solutions, the feasible region becomes a discrete set of lattice points inside the polytope. There is no efficient gradient or pivoting strategy that moves between neighbouring integer solutions. In fact, ILP is NP-hard — many classic NP-complete problems (vertex cover, set cover, traveling salesman) can be modeled as ILPs.

    Key Insight: The LP relaxation (dropping the integrality constraint) gives an upper bound for maximization problems and a lower bound for minimization problems. Branch and Bound exploits this by repeatedly solving LP relaxations.

    Classic ILP Examples

    • 0/1 Knapsack: Given items with weights wᵢ and values vᵢ, maximize ∑ vᵢxᵢ subject to ∑ wᵢxᵢ ≤ W, xᵢ ∈ {0,1}.
    • Set Cover: Given a universe and subsets, find the minimum number of subsets whose union is the universe. Each subset has a binary indicator variable.
    • Assignment Problem: Assign n workers to n tasks with a cost matrix, minimizing total cost, where each assignment variable is binary.
    • TSP (subtour elimination): The classic traveling salesman problem can be formulated as an ILP with binary edge-selection variables and subtour elimination constraints.

    Branch and Bound Framework

    Branch and Bound (B&B) is a systematic enumeration strategy that avoids examining every possible integer solution by using bounds to prune large portions of the search space. The three core ideas are:

    1. Branching

    When the LP relaxation of a subproblem yields a non-integer solution, we pick a fractional variable xᵢ with value f and create two child subproblems: one adding xᵢ ≤ ⌊f⌋ and one adding xᵢ ≥ ⌈f⌉.

    2. Bounding

    For each subproblem, we solve its LP relaxation to get an upper bound (for maximization). If this bound is no better than the best integer solution found so far (the incumbent), we can discard the entire subtree.

    3. Pruning

    A node is pruned (not explored further) in three cases:

    • Bound pruning: The relaxation's optimal value ≤ incumbent.
    • Infeasibility pruning: The relaxation is infeasible (adding the branching constraint made the polytope empty).
    • Integrality pruning: The relaxation gives an all-integer solution. We update the incumbent if this solution is better, and prune (no need to branch).
    B&B is an exact algorithm. It is guaranteed to find the optimal integer solution (or prove infeasibility) — it just may take exponential time in the worst case. In practice, good bounding and branching strategies make it fast for many instances.

    High-Level Pseudocode

    BranchAndBound(ILP):
    Q ← priority queue with root node (the original LP relaxation)
    incumbent ← -∞
    best_solution ← null

    while Q is not empty:
        node ← Q.pop()             // pick node with best bound
        solve LP relaxation of node

        if infeasible:
            continue                // prune by infeasibility
        if LP_opt <= incumbent:
            continue                // prune by bound
        if solution is all-integer:
            incumbent ← LP_opt
            best_solution ← solution
            continue                // prune by integrality

        // Branch: pick fractional variable x_i = f
        child1 ← node + constraint (x_i <= floor(f))
        child2 ← node + constraint (x_i >= ceil(f))
        Q.push(child1)
        Q.push(child2)

    return best_solution

    LP Relaxation as Bound

    The LP relaxation of an ILP is obtained by dropping all integrality constraints, turning xᵢ ∈ ℤ into xᵢ ∈ ℝ. Since the feasible set of the ILP is a subset of the LP's feasible set, the LP relaxation always provides a bound:

    • For maximization: LP_opt ≥ ILP_opt (upper bound).
    • For minimization: LP_opt ≤ ILP_opt (lower bound).

    The Integrality Gap

    The integrality gap is the worst-case ratio between the LP relaxation optimum and the ILP optimum. For some problems (like minimum vertex cover) the gap is bounded by a constant factor; for others (like maximum independent set) it can be arbitrarily large.

    A small integrality gap means the LP relaxation is a tight bound, which makes B&B converge quickly because nodes are pruned early. A large gap means B&B will explore many nodes before pruning — and this is where cutting planes help (discussed later).

    Fractional Knapsack as LP Relaxation

    Consider 0/1 knapsack. The LP relaxation allows fractional item selection (xᵢ ∈ [0,1]). The optimal fractional solution has a clean structure: sort items by value-to-weight ratio, take items greedily until the knapsack is full, and split the last item fractionally. This gives a tight upper bound for branch and bound. 

    // LP relaxation bound for 0/1 knapsack
// Items sorted by value/weight ratio (descending)
double lp_relaxation_bound(const vector<pair<int,int>>& items,
                           int capacity, int start_idx) {
    double bound = 0.0;
    int remaining = capacity;
    for (int i = start_idx; i < (int)items.size(); i++) {
        int v = items[i].first, w = items[i].second;
        if (w <= remaining) {
            bound += v;
            remaining -= w;
        } else {
            // Take fractional part
            bound += (double)v * remaining / w;
            break;
        }
    }
    return bound;
}
    Practical tip: For the knapsack LP relaxation, the fractional bound can be computed in O(n) after sorting. This makes each B&B node cheap to evaluate, which is critical for performance.

    Implementation: B&B for 0/1 Knapsack

    Let us implement a complete Branch and Bound solver for the 0/1 knapsack problem. Each node in the search tree represents a partial assignment: some items are fixed (taken or not taken) and the rest are free. We use the LP relaxation bound to prune unpromising branches.

    Node Representation

    #include <bits/stdc++.h>
using namespace std;

struct Item {
    int value, weight;
    double ratio;
    bool operator>(const Item& o) const {
        return ratio > o.ratio;
    }
};

struct Node {
    int level;       // index of next item to decide on
    int profit;      // profit so far
    int weight;      // weight so far
    double bound;    // LP relaxation upper bound
};

// Comparator: best-first search (highest bound first)
struct NodeCmp {
    bool operator()(const Node& a, const Node& b) {
        return a.bound < b.bound;
    }
};

    Computing the Bound

    double compute_bound(const vector<Item>& items, int n,
                     int capacity, const Node& u) {
    if (u.weight >= capacity) return 0;

    double bound = u.profit;
    int remaining = capacity - u.weight;
    int j = u.level;

    while (j < n && items[j].weight <= remaining) {
        remaining -= items[j].weight;
        bound += items[j].value;
        j++;
    }
    // Add fractional part of next item
    if (j < n) {
        bound += (double)items[j].value * remaining / items[j].weight;
    }
    return bound;
}

    Full B&B Solver

    int knapsack_bnb(vector<Item>& items, int capacity) {
    int n = items.size();
    sort(items.begin(), items.end(), greater<Item>());

    priority_queue<Node, vector<Node>, NodeCmp> pq;

    Node root;
    root.level = 0;
    root.profit = 0;
    root.weight = 0;
    root.bound = compute_bound(items, n, capacity, root);

    int best = 0;
    pq.push(root);

    while (!pq.empty()) {
        Node u = pq.top(); pq.pop();

        // Prune by bound
        if (u.bound <= best) continue;
        if (u.level == n) continue;

        // Branch: take item u.level
        Node take;
        take.level = u.level + 1;
        take.weight = u.weight + items[u.level].weight;
        take.profit = u.profit + items[u.level].value;

        if (take.weight <= capacity && take.profit > best) {
            best = take.profit;
        }
        take.bound = compute_bound(items, n, capacity, take);
        if (take.bound > best) {
            pq.push(take);
        }

        // Branch: skip item u.level
        Node skip;
        skip.level = u.level + 1;
        skip.weight = u.weight;
        skip.profit = u.profit;
        skip.bound = compute_bound(items, n, capacity, skip);
        if (skip.bound > best) {
            pq.push(skip);
        }
    }
    return best;
}

int main() {
    int n, W;
    cin >> n >> W;
    vector<Item> items(n);
    for (int i = 0; i < n; i++) {
        cin >> items[i].value >> items[i].weight;
        items[i].ratio = (double)items[i].value / items[i].weight;
    }
    cout << knapsack_bnb(items, W) << "\n";
    return 0;
}
    Complexity: Worst case is O(2ⁿ), but in practice the LP bound prunes the vast majority of nodes. For knapsack instances with n ≈ 40–50 and well-spread value/weight ratios, B&B typically explores only a few thousand nodes.

    DFS-Based B&B (Lower Memory)

    The best-first approach above uses a priority queue that can grow large. An alternative is depth-first B&B, which uses O(n) stack space and finds feasible solutions quickly (useful for getting a good incumbent early). 

    int best_profit;

void dfs_bnb(const vector<Item>& items, int n, int capacity,
             int level, int profit, int weight) {
    if (level == n) {
        best_profit = max(best_profit, profit);
        return;
    }

    // Compute LP bound for remaining items
    double bound = profit;
    int rem = capacity - weight;
    int j = level;
    while (j < n && items[j].weight <= rem) {
        rem -= items[j].weight;
        bound += items[j].value;
        j++;
    }
    if (j < n) bound += (double)items[j].value * rem / items[j].weight;

    // Prune
    if (bound <= best_profit) return;

    // Branch: take item
    if (weight + items[level].weight <= capacity) {
        dfs_bnb(items, n, capacity, level + 1,
                profit + items[level].value,
                weight + items[level].weight);
    }
    // Branch: skip item
    dfs_bnb(items, n, capacity, level + 1, profit, weight);
}

int knapsack_dfs_bnb(vector<Item>& items, int capacity) {
    sort(items.begin(), items.end(), greater<Item>());
    best_profit = 0;
    dfs_bnb(items, items.size(), capacity, 0, 0, 0);
    return best_profit;
}

    Branching Strategies

    The choice of which fractional variable to branch on has a dramatic impact on the size of the B&B tree. A good branching variable creates children whose LP bounds are significantly lower than the parent's, leading to more pruning.

    Most Fractional Variable

    The simplest heuristic: pick the variable whose LP relaxation value is closest to 0.5 (furthest from any integer). The intuition is that this variable is the "most undecided," so fixing it should reduce the search space most evenly. 

    // Returns index of the most fractional variable
// x[i] contains the LP relaxation values
int most_fractional(const vector<double>& x, int n) {
    int best_idx = -1;
    double best_frac = -1.0;
    for (int i = 0; i < n; i++) {
        double frac = x[i] - floor(x[i]);
        double dist = min(frac, 1.0 - frac);
        // Only consider non-integral variables
        if (dist > 1e-6 && (0.5 - dist) < (0.5 - best_frac)) {
            best_frac = dist;
            best_idx = i;
        }
    }
    return best_idx;
}

    Most Infeasible Branching

    A variant: pick the variable whose fractional part fᵢ = xᵢ − ⌊xᵢ⌋ minimizes |fᵢ − 0.5|. For binary variables this coincides with "most fractional."

    Strong Branching

    Strong branching is more expensive but often produces much smaller B&B trees. For each candidate branching variable, we tentatively solve both child LP relaxations and choose the variable that produces the greatest improvement (reduction in the bound). 

    // Pseudocode for strong branching
int strong_branching(const vector<double>& x, int n,
                     /* LP solver state */) {
    int best_var = -1;
    double best_score = -1.0;

    for (int i = 0; i < n; i++) {
        double frac = x[i] - floor(x[i]);
        if (frac < 1e-6 || frac > 1.0 - 1e-6) continue;

        // Solve LP relaxation with x_i <= floor(x_i)
        double down_obj = solve_lp_with_bound(i, floor(x[i]), /* <= */);
        // Solve LP relaxation with x_i >= ceil(x_i)
        double up_obj   = solve_lp_with_bound(i, ceil(x[i]),  /* >= */);

        // Score: product of degradations (common choice)
        double d_down = parent_obj - down_obj;
        double d_up   = parent_obj - up_obj;
        double score  = max(d_down, 1e-6) * max(d_up, 1e-6);

        if (score > best_score) {
            best_score = score;
            best_var = i;
        }
    }
    return best_var;
}
    Trade-off: Strong branching solves 2k LPs per node (for k fractional variables), which is expensive. Commercial solvers like CPLEX and Gurobi use reliability branching — strong branching for the first few nodes, then switching to a cheaper heuristic once they have estimated each variable's branching impact.

    Node Selection Strategies

    Besides branching variable selection, the order in which nodes are explored matters:

    • Best-first: Pick the node with the best bound. Minimizes nodes explored but uses more memory.
    • Depth-first: Explore children before siblings. Uses O(depth) memory and finds feasible solutions quickly.
    • Best-estimate: Pick the node with the best estimated integer objective via pseudocost estimates. A good compromise.

    Cutting Planes

    Cutting planes tighten the LP relaxation by adding linear inequalities that are valid for all integer solutions but cut off fractional LP optima. When combined with B&B, this yields the Branch-and-Cut algorithm used by all modern ILP solvers.

    Gomory Cuts

    A Gomory cut is derived directly from the simplex tableau. Suppose a basic variable xᵢ has a fractional value fᵢ in the LP optimum. From the tableau row for xᵢ: 

    xᵢ + Σⱼ āᵢⱼ xⱼ = b̄ᵢ      (tableau row)

Let fᵢ = b̄ᵢ - floor(b̄ᵢ),   fⱼ = āᵢⱼ - floor(āᵢⱼ)

Gomory cut:  Σⱼ fⱼ xⱼ  ≥  fᵢ

    This inequality is satisfied by every integer solution but violated by the current fractional LP optimum, so adding it tightens the relaxation.

    Other Cut Families

    Beyond Gomory cuts, solvers employ cover cuts (for knapsack-type constraints), clique cuts (for conflict graphs where at most one variable in a clique can be 1), and more sophisticated families like flow cover cuts and mixed-integer rounding.

    Branch-and-Cut Overview

    The Branch-and-Cut algorithm is simply B&B where, at each node, we also try to add cutting planes before branching: 

    BranchAndCut(node):
    solve LP relaxation
    while cuts can be found:
        add cutting planes
        re-solve LP relaxation
    if solution is all-integer:
        update incumbent
    else:
        branch on a fractional variable
    Note: In competitive programming, you rarely implement a full branch-and-cut solver. However, understanding the concept helps you recognize when a problem's LP relaxation is tight (making B&B fast) and when you might need problem-specific cuts or tighter formulations.

    B&B in Competitive Programming

    While industrial ILP solvers are complex beasts, the B&B paradigm shows up in competitive programming more often than you might expect — usually in the form of backtracking with pruning, which is really just B&B with a problem-specific bound.

    When to Use B&B in Contests

    • Small n with exponential search space: Problems with n ≤ 20–40 where brute force is too slow but a good bound function prunes most branches.
    • Knapsack variants: Non-standard knapsacks (multi-dimensional, with dependencies) where DP is too slow but B&B with LP bounds works.
    • Subset selection / assignment: Finding an optimal subset or matching when the constraints don't fit standard algorithms.
    • Graph problems on small instances: Maximum clique, minimum vertex cover, graph coloring on graphs with ≤ 30–50 vertices.

    Pruning Heuristics for Contests

    In a contest you typically implement DFS-based B&B with hand-crafted bounds. Key techniques:

    • Greedy initial solution: Compute a greedy solution first for a strong initial incumbent.
    • Sort for tighter bounds: Order items by value/weight ratio or degree for better pruning.
    • Prefix/suffix sums: Precompute running sums so each bound evaluates in O(1).
    • Constraint propagation: After fixing a variable, propagate implications to eliminate infeasible choices.

    Contest Example: Maximum Weight Independent Set (Small n)

    Given a graph with n ≤ 30 vertices and weights, find the maximum weight independent set. We use B&B where the bound is the sum of weights of all remaining vertices (an optimistic bound since we ignore edges). 

    #include <bits/stdc++.h>
using namespace std;

int n;
long long w[35];
long long adj[35];     // bitmask adjacency
long long best;

// suffix_sum[i] = sum of w[i..n-1]
long long suffix_sum[36];

void solve(int idx, long long cur_weight, long long mask) {
    if (idx == n) {
        best = max(best, cur_weight);
        return;
    }

    // Bound: current weight + all remaining vertices
    if (cur_weight + suffix_sum[idx] <= best) return;

    // Branch: skip vertex idx
    solve(idx + 1, cur_weight, mask);

    // Branch: take vertex idx (if no conflict)
    if ((mask & adj[idx]) == 0) {
        solve(idx + 1, cur_weight + w[idx], mask | (1LL << idx));
    }
}

int main() {
    int m;
    cin >> n >> m;
    for (int i = 0; i < n; i++) cin >> w[i];
    memset(adj, 0, sizeof(adj));
    for (int i = 0; i < m; i++) {
        int u, v; cin >> u >> v;
        adj[u] |= (1LL << v);
        adj[v] |= (1LL << u);
    }

    // Precompute suffix sums
    suffix_sum[n] = 0;
    for (int i = n - 1; i >= 0; i--)
        suffix_sum[i] = suffix_sum[i + 1] + w[i];

    best = 0;
    solve(0, 0, 0);
    cout << best << "\n";
    return 0;
}

    Improving the Bound

    The simple suffix-sum bound ignores edge conflicts. A tighter greedy bound only includes vertices that do not conflict with already-selected ones: 

    // Tighter bound: for each remaining vertex, include it
// only if it does not conflict with already-selected vertices
long long tighter_bound(int idx, long long cur_weight, long long mask) {
    long long bound = cur_weight;
    long long test_mask = mask;
    for (int i = idx; i < n; i++) {
        if ((test_mask & adj[i]) == 0) {
            bound += w[i];
            test_mask |= (1LL << i);
        }
    }
    return bound;
}
    Contest reality check: In competitive programming, you rarely say "I am implementing Branch and Bound." Instead, you write a backtracking search with bounds — but the underlying algorithm is exactly B&B. Recognizing this lets you apply the theory (LP relaxation bounds, branching order, constraint propagation) to write faster solutions.

    Summary

    Integer Linear Programming turns polynomial-time LPs into NP-hard problems, and Branch and Bound is the foundational exact algorithm for tackling them. The effectiveness of B&B depends on three things: (1) the tightness of the LP relaxation bound, (2) the choice of branching variable, and (3) pruning heuristics that avoid exploring hopeless subtrees. In competitive programming, B&B appears as backtracking with bounds — and understanding the theory helps you write dramatically faster exponential-time solutions.
