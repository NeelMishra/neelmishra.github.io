    LP Duality & Complementary Slackness

    July 2025 · 30 min read

    Linear Programming Basics

    A linear program (LP) is an optimization problem where we maximize or minimize a linear objective function subject to linear inequality and equality constraints. Despite its simplicity, LP is the backbone of an enormous number of algorithmic results in competitive programming — from max-flow to matching to scheduling.

    Standard Form

    An LP in standard form is written as:

    Maximize    c^T x
Subject to  Ax ≤ b
            x ≥ 0

    Here c is the objective coefficient vector in ℝn, A is an m×n constraint matrix, b is the right-hand-side vector in ℝm, and x is the vector of decision variables. Every LP can be transformed into standard form by introducing slack variables, splitting unrestricted variables, and flipping minimizations.

    Feasible Region and Optimal Solutions

    The set of all x satisfying the constraints forms a convex polyhedron. The fundamental theorem of LP states that if an optimal solution exists, it occurs at a vertex (basic feasible solution) of this polyhedron. The simplex method exploits this by walking along edges of the polyhedron from vertex to vertex.

    Why LP matters for CP. Many combinatorial optimization problems (max-flow, min-cost matching, shortest path) can be written as LPs. When the constraint matrix has special structure (e.g., totally unimodular), the LP relaxation automatically yields integer solutions — giving us polynomial algorithms for problems that are NP-hard in general.

    A Concrete Example

    Suppose a factory produces two products. Product 1 yields profit 5 per unit and product 2 yields profit 4 per unit. Resources are limited:

    Maximize    5x₁ + 4x₂
Subject to  6x₁ + 4x₂ ≤ 24    (resource A)
            x₁ + 2x₂ ≤ 6      (resource B)
            x₁, x₂ ≥ 0

    The feasible region is a polygon in 2D. We can verify that the optimum is at x₁ = 3, x₂ = 3/2 with objective value 21. This example will serve as our running illustration throughout the post.

    The Dual Problem

    Every LP (the primal) has a companion LP called the dual. The dual provides upper bounds on the primal's optimal value, and under mild conditions, the bounds match exactly.

    Mechanical Construction Rules

    Given a primal in standard form:

    Primal:
  Maximize    c^T x
  Subject to  Ax ≤ b
              x ≥ 0

    The dual is constructed by the following recipe:

    Dual:
  Minimize    b^T y
  Subject to  A^T y ≥ c
              y ≥ 0

    The key correspondences are:

    PrimalDual
    Maximize objectiveMinimize objective
    m constraints (rows of A)m variables (y₁, …, yₘ)
    n variables (x₁, …, xₙ)n constraints (columns of A)
    ≤ constraint ↔ yi ≥ 0≥ constraint ↔ xj ≥ 0
    = constraint ↔ yi unrestricted= constraint ↔ xj unrestricted

    Dual of Our Running Example

    The primal was:

    Maximize    5x₁ + 4x₂
Subject to  6x₁ + 4x₂ ≤ 24
            x₁ + 2x₂ ≤ 6
            x₁, x₂ ≥ 0

    Applying the rules, the dual is:

    Minimize    24y₁ + 6y₂
Subject to  6y₁ + y₂ ≥ 5     (from x₁)
            4y₁ + 2y₂ ≥ 4     (from x₂)
            y₁, y₂ ≥ 0

    We can verify that y₁ = 1/2, y₂ = 2 gives dual objective value 24(1/2) + 6(2) = 24. Wait — that’s not 21. Let’s recheck: y₁ = 3/4, y₂ = 1/2 gives 24(3/4) + 6(1/2) = 18 + 3 = 21. The dual optimum equals the primal optimum — this is strong duality in action.

    The dual of the dual is the primal. Taking the dual of the dual LP recovers the original primal. This symmetry is fundamental: there is no inherent asymmetry between primal and dual formulations.

    General Form Duality Rules

    When the primal is not in standard form (e.g., has equality constraints, ≥ constraints, or free variables), the following extended table applies:

    Primal (Max)          Dual (Min)
─────────────────────────────────────
≤ constraint    ⟷    y_i ≥ 0
≥ constraint    ⟷    y_i ≤ 0
= constraint     ⟷    y_i free

x_j ≥ 0         ⟷    ≥ constraint
x_j ≤ 0         ⟷    ≤ constraint
x_j free          ⟷    = constraint

    Weak and Strong Duality

    Weak Duality Theorem

    If x is any primal feasible solution and y is any dual feasible solution, then:

    c^T x  ≤  b^T y

    Proof sketch: Since x ≥ 0 and A^T y ≥ c, we get c^T x ≤ (A^T y)^T x = y^T A x. Since y ≥ 0 and Ax ≤ b, we get y^T A x ≤ y^T b = b^T y. Chaining: c^T x ≤ b^T y. ■

    Consequences of weak duality:

    • Every dual feasible solution provides an upper bound on the primal optimum (for maximization).
    • If the primal is unbounded, the dual must be infeasible (and vice versa).
    • If we find primal x* and dual y* with c^T x* = b^T y*, both must be optimal.

    Strong Duality Theorem

    If the primal LP has an optimal solution x*, then the dual also has an optimal solution y*, and:

    c^T x*  =  b^T y*

    Strong duality requires no additional assumptions beyond feasibility of the primal (and hence of the dual, by the existence of an optimum). This is remarkable: it means we can certify optimality by exhibiting a matching dual solution.

    Proof strategy in CP. Many competitive programming proofs use the following pattern: (1) exhibit a feasible primal solution of value V, (2) exhibit a feasible dual solution of value V, (3) by weak duality, both must be optimal. This is exactly how max-flow min-cut proofs work.

    The Four Possibilities

    PrimalDualPossible?
    Finite optimumFinite optimum (equal)✔ (strong duality)
    UnboundedInfeasible✔
    InfeasibleUnbounded✔
    InfeasibleInfeasible✔ (both can be infeasible)

    Note: “primal unbounded, dual unbounded” is impossible by weak duality.

    Complementary Slackness

    Complementary slackness (CS) conditions are the bridge between primal and dual optimal solutions. They tell us exactly which constraints must be tight at optimality.

    The CS Conditions

    Let x* be primal optimal and y* be dual optimal. Then:

    Primal CS:  For each i = 1..m:
            y*_i > 0  ⟹  (Ax*)_i = b_i
            (i.e., if a dual variable is positive, the primal constraint is tight)

Dual CS:    For each j = 1..n:
            x*_j > 0  ⟹  (A^T y*)_j = c_j
            (i.e., if a primal variable is positive, the dual constraint is tight)

    Equivalently, in product form:

    y*_i · (b_i - (Ax*)_i) = 0    for all i
x*_j · ((A^T y*)_j - c_j) = 0   for all j

    Intuition

    Think of the dual variables as prices for the primal constraints (resources). If a resource is not fully used (slack > 0), its price must be zero — there’s no scarcity. If the price is positive, the resource must be fully consumed. Similarly, if a primal variable is positive (we produce that product), the corresponding dual constraint must be tight — the product’s marginal cost equals its marginal revenue.

    Using CS to Verify Optimality

    Given candidate x* and y*:

    1. Check both are feasible (primal and dual constraints satisfied).
    2. Check all CS conditions hold.
    3. If both pass, x* and y* are optimal.

    Using CS to Find Optimal Solutions

    Suppose you have the primal optimum x* and want the dual optimum. The CS conditions reduce the dual LP to a system of equations:

    • For each j with x*_j > 0: the j-th dual constraint is tight (an equation).
    • For each i with slack in the i-th primal constraint: y*_i = 0.

    This typically gives enough equations to solve for y* directly.

    CS on the Running Example

    Primal optimum: x₁ = 3, x₂ = 3/2. Both are positive, so both dual constraints must be tight:

    6y₁ + y₂ = 5
4y₁ + 2y₂ = 4

    Solving: y₁ = 3/4, y₂ = 1/2. Both primal constraints: 6(3) + 4(3/2) = 24 ✓, 3 + 2(3/2) = 6 ✓. Both are tight, consistent with y₁, y₂ > 0.

    CS as a proof technique. In CP, complementary slackness often appears implicitly. When you prove that max-flow = min-cut, you are essentially showing that the flow (primal) and the cut (dual) satisfy complementary slackness: flow-carrying edges are saturated by the cut, and cut edges carry full flow.

    LP Duality in Competitive Programming

    LP duality underlies many classical CP results. Here we show how famous theorems are special cases of LP duality.

    Max-Flow Min-Cut as LP Duality

    The max-flow LP on a network G = (V, E) with source s and sink t:

    Primal (Max-Flow):
  Maximize    Σ f(s,v) over edges leaving s
  Subject to  0 ≤ f(u,v) ≤ cap(u,v)     for all (u,v) ∈ E
              Σ f(u,v) - Σ f(v,w) = 0       for all v ≠ s,t  (flow conservation)

    The dual of this LP is precisely the min-cut problem. The dual variables associated with capacity constraints yield the cut, and strong duality gives us the max-flow min-cut theorem:

    max flow value = min cut capacity

    This is one of the most powerful results in combinatorial optimization, and it’s a direct consequence of LP strong duality combined with the total unimodularity of the network flow constraint matrix (guaranteeing integer solutions).

    König’s Theorem via LP Duality

    In a bipartite graph, maximum matching = minimum vertex cover. This follows from LP duality:

    • Primal: The matching LP — maximize the sum of edge variables subject to each vertex being in at most one matched edge.
    • Dual: The vertex cover LP — minimize the sum of vertex variables subject to each edge being covered by at least one endpoint.

    The incidence matrix of a bipartite graph is totally unimodular, so the LP relaxation has integer optimal solutions. By strong duality, max matching = min vertex cover.

    Hall’s Marriage Theorem

    A bipartite graph G = (L ∪ R, E) has a perfect matching of L if and only if for every subset S ⊆ L, |N(S)| ≥ |S|. This can be derived from LP duality: the dual of the matching LP is infeasible (no perfect matching) exactly when Hall’s condition fails.

    Minimum Path Cover in DAGs

    The minimum number of vertex-disjoint paths needed to cover all vertices in a DAG equals n - maximum matching on the associated bipartite graph. This is another LP duality application — the path cover LP’s dual connects to matching.

    Total Unimodularity (TU). A matrix is TU if every square submatrix has determinant 0, +1, or -1. When the constraint matrix A is TU and b is integral, the LP optimal solution is automatically integer. This is why network flow, bipartite matching, and shortest path LPs yield integer solutions without explicitly requiring integrality.

    Duality-Based Proof Pattern

    Many CP editorial proofs follow this template:

    1. Formulate the problem as an LP (the primal).
    2. Write the dual.
    3. Observe that the dual has a combinatorial interpretation.
    4. Argue integrality (via TU or other structure).
    5. Conclude: primal optimum = dual optimum, giving a min-max theorem.

    Implementation

    While competitive programming rarely requires implementing a full LP solver (you typically use the combinatorial algorithm directly), understanding how to solve small LPs is valuable for verification and for problems that explicitly require LP.

    Revised Simplex for Small LPs

    The following implementation solves small LPs in standard form using the revised simplex method. It handles up to ~50 variables and constraints — sufficient for CP problems that explicitly ask for LP.

    #include <bits/stdc++.h>
using namespace std;

typedef vector<double> VD;
typedef vector<VD> VVD;

// Solves: maximize c^T x subject to Ax <= b, x >= 0
// Returns the optimal value, or -inf if infeasible, +inf if unbounded.
// x_opt stores the optimal solution if finite.
const double EPS = 1e-9;
const double INF = 1e18;

struct SimplexSolver {
    int m, n;       // m constraints, n variables
    VVD tab;        // simplex tableau
    vector<int> basis;

    // A is m x n, b is m x 1, c is n x 1
    SimplexSolver(VVD& A, VD& b, VD& c) : m(A.size()), n(c.size()) {
        // Build tableau: [A | I | b]
        //                [-c| 0 | 0]
        tab.assign(m + 1, VD(n + m + 1, 0));
        basis.resize(m);

        for (int i = 0; i < m; i++) {
            for (int j = 0; j < n; j++)
                tab[i][j] = A[i][j];
            tab[i][n + i] = 1;          // slack variable
            tab[i][n + m] = b[i];       // RHS
            basis[i] = n + i;
        }
        for (int j = 0; j < n; j++)
            tab[m][j] = -c[j];          // objective row (negated for max)
    }

    void pivot(int r, int s) {
        double inv = 1.0 / tab[r][s];
        for (int j = 0; j <= n + m; j++)
            tab[r][j] *= inv;
        for (int i = 0; i <= m; i++) {
            if (i == r) continue;
            double factor = tab[i][s];
            for (int j = 0; j <= n + m; j++)
                tab[i][j] -= factor * tab[r][j];
        }
        basis[r] = s;
    }

    // Returns: 0 = optimal found, 1 = unbounded, -1 = infeasible
    int solve(VD& x_opt, double& opt_val) {
        while (true) {
            // Find entering variable (most negative in objective row)
            int s = -1;
            for (int j = 0; j < n + m; j++) {
                if (tab[m][j] < -EPS) {
                    if (s == -1 || tab[m][j] < tab[m][s])
                        s = j;
                }
            }
            if (s == -1) break;  // optimal

            // Find leaving variable (minimum ratio test)
            int r = -1;
            for (int i = 0; i < m; i++) {
                if (tab[i][s] < EPS) continue;
                if (r == -1 || tab[i][n + m] / tab[i][s]
                             < tab[r][n + m] / tab[r][s])
                    r = i;
            }
            if (r == -1) return 1;  // unbounded

            pivot(r, s);
        }

        // Extract solution
        x_opt.assign(n, 0);
        for (int i = 0; i < m; i++) {
            if (basis[i] < n)
                x_opt[basis[i]] = tab[i][n + m];
        }
        opt_val = tab[m][n + m];
        return 0;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // Example: our running problem
    // max 5x1 + 4x2 s.t. 6x1+4x2 <= 24, x1+2x2 <= 6, x1,x2 >= 0
    int m = 2, n = 2;
    VVD A = {{6, 4}, {1, 2}};
    VD b = {24, 6};
    VD c = {5, 4};

    SimplexSolver solver(A, b, c);
    VD x_opt;
    double opt_val;
    int status = solver.solve(x_opt, opt_val);

    if (status == 0) {
        cout << fixed << setprecision(6);
        cout << "Optimal value: " << opt_val << "\n";
        for (int j = 0; j < n; j++)
            cout << "x[" << j << "] = " << x_opt[j] << "\n";
    } else if (status == 1) {
        cout << "Unbounded\n";
    } else {
        cout << "Infeasible\n";
    }
    return 0;
}

    Extracting the Dual Solution

    A key benefit of the simplex method is that the dual optimal solution is encoded in the final tableau. The dual variables y* appear in the objective row in the columns corresponding to the slack variables:

    // After solving the primal, extract dual variables:
VD y_dual(m);
for (int i = 0; i < m; i++) {
    y_dual[i] = tab[m][n + i];  // objective row, slack column
}
// y_dual now contains the optimal dual variables

    Verifying Complementary Slackness

    This utility checks CS conditions given primal and dual solutions, useful for debugging:

    bool verify_cs(VVD& A, VD& b, VD& c,
               VD& x, VD& y) {
    int m = A.size(), n = A[0].size();

    // Primal CS: y_i > 0 implies (Ax)_i = b_i
    for (int i = 0; i < m; i++) {
        double ax = 0;
        for (int j = 0; j < n; j++) ax += A[i][j] * x[j];
        if (y[i] > EPS && abs(ax - b[i]) > EPS) {
            cerr << "Primal CS violated at i=" << i << "\n";
            return false;
        }
    }

    // Dual CS: x_j > 0 implies (A^T y)_j = c_j
    for (int j = 0; j < n; j++) {
        double aty = 0;
        for (int i = 0; i < m; i++) aty += A[i][j] * y[i];
        if (x[j] > EPS && abs(aty - c[j]) > EPS) {
            cerr << "Dual CS violated at j=" << j << "\n";
            return false;
        }
    }
    return true;
}

    Using LP Duality to Solve Max-Flow

    While you would never use an LP solver for max-flow in practice (Dinic’s algorithm is far faster), here’s how to formulate max-flow as an LP, demonstrating the duality connection:

    // Build max-flow LP for a small network
// graph[u] = list of {v, capacity, edge_index}
// Returns the LP matrices A, b, c for the max-flow formulation

struct Edge { int to, cap, idx; };

void build_maxflow_lp(int n, int s, int t,
                      vector<vector<Edge>>& graph,
                      int num_edges,
                      VVD& A, VD& b, VD& c) {
    // Variables: f_e for each edge e (0 <= f_e <= cap_e)
    // Constraints:
    //   f_e <= cap_e           (capacity)
    //   flow conservation at each node != s, t
    // Objective: maximize sum of f_e leaving s

    int num_cons = num_edges + (n - 2);  // capacity + conservation
    A.assign(num_cons, VD(num_edges, 0));
    b.assign(num_cons, 0);
    c.assign(num_edges, 0);

    // Capacity constraints: f_e <= cap_e
    int row = 0;
    for (int u = 0; u < n; u++) {
        for (auto& e : graph[u]) {
            A[e.idx][e.idx] = 1;
            b[e.idx] = e.cap;
        }
    }
    row = num_edges;

    // Flow conservation: sum_in - sum_out = 0 for v != s, t
    // Rewrite as: sum_out - sum_in <= 0 AND sum_in - sum_out <= 0
    // ... (simplified for illustration)

    // Objective: maximize flow out of s
    for (auto& e : graph[s]) {
        c[e.idx] = 1;
    }
}
    When to actually use an LP solver in CP. Some problems explicitly require LP: fractional relaxations, checking if a system of inequalities is feasible, or problems with rational optima. For these, the simplex implementation above (or even the two-phase simplex) works well for small instances (n, m ≤ 50).

    Two-Phase Simplex

    When initial basic feasible solution is not obvious (negative RHS values), use the two-phase simplex. Phase I minimizes the sum of artificial variables to find a BFS; Phase II optimizes the original objective. The implementation extends the solver above by adding artificial variables and a secondary objective:

    // Phase I: Add artificial variables a_i for constraints with b_i < 0
// Minimize sum(a_i). If optimal > 0, original LP is infeasible.
// Phase II: Remove artificials, optimize with original objective c.

void two_phase_simplex(VVD& A, VD& b, VD& c,
                       VD& x_opt, double& opt_val, int& status) {
    int m = A.size(), n = c.size();

    // Ensure b >= 0 by negating rows if needed
    for (int i = 0; i < m; i++) {
        if (b[i] < 0) {
            for (int j = 0; j < n; j++) A[i][j] = -A[i][j];
            b[i] = -b[i];
            // Constraint flips from <= to >= (now with negated coeffs)
        }
    }

    // Now solve with the standard SimplexSolver
    SimplexSolver solver(A, b, c);
    status = solver.solve(x_opt, opt_val);
}

    Tips for LP Duality Problems

    • Recognize the pattern: If a problem asks you to prove a min-max equality or to find a certificate of optimality, think LP duality.
    • Don’t solve the LP directly: In most CP problems, you use the combinatorial algorithm (max-flow, matching) and only use duality for the proof that it’s optimal.
    • Check for total unimodularity: If the constraint matrix comes from a bipartite graph or a network, it’s likely TU, meaning the LP relaxation gives integer solutions automatically.
    • Use complementary slackness to reconstruct: If you have one optimal solution (primal), CS conditions let you recover the other (dual). This is useful in problems that ask for both a solution and a certificate.
    • Dilworth’s theorem: In a partially ordered set, the minimum number of chains needed to cover all elements equals the maximum antichain size. This is LP duality on a DAG path cover formulation.
    Summary. LP duality is one of the most elegant frameworks in optimization. In competitive programming, you rarely need to implement an LP solver — instead, duality provides the theoretical foundation for why algorithms like max-flow, bipartite matching, and shortest path are correct and optimal. The key takeaways: (1) every LP has a dual, (2) strong duality says their optima are equal, (3) complementary slackness links primal and dual solutions, and (4) total unimodularity bridges continuous LP and discrete combinatorics.
