    Simulated Annealing for Competitive Programming

    July 2026 · 22 min read

    What Is Simulated Annealing?

    Simulated Annealing (SA) borrows its name from metallurgy. When a blacksmith heats metal to a high temperature and then cools it slowly, atoms settle into a low-energy crystalline structure far more ordered than if the metal were quenched rapidly. SA mimics this process: we start with a "hot" random search that accepts even bad moves, then gradually cool down so the algorithm converges toward a near-optimal solution.

    More formally, SA is a probabilistic meta-heuristic for approximating the global optimum of a function over a large search space. It belongs to the family of local-search algorithms but escapes local optima by occasionally accepting worse solutions—with a probability that decreases over time as the "temperature" drops.

    Why SA in contests? Many competitive-programming problems (especially on platforms like AtCoder, Google Hash Code, or marathon-style contests) have enormous search spaces where exact algorithms are too slow. SA often produces near-optimal answers within the tight time limits of 2–5 seconds.

    The core idea in three bullets:

    • State — a candidate solution (e.g., a permutation, a set of coordinates, a partition).
    • Energy (cost) — the objective function you want to minimize (or maximize).
    • Temperature — a control parameter that governs the probability of accepting worse states. Starts high, ends near zero.

    At high temperature, the algorithm is essentially a random walk; at low temperature, it behaves like a greedy hill-climber. The magic is in the transition between these two regimes.

    The SA Algorithm

    The pseudocode is deceptively simple:

    s ← initial_solution()
E ← cost(s)
best ← s
best_E ← E
T ← T_initial

while T > T_min:
    s' ← neighbor(s)
    E' ← cost(s')
    ΔE ← E' - E

    if ΔE < 0:              // improvement → always accept
        s ← s'
        E ← E'
    else if random() < exp(-ΔE / T):  // worse → accept with probability
        s ← s'
        E ← E'

    if E < best_E:
        best ← s
        best_E ← E

    T ← cool(T)

return best

    Acceptance Probability

    The heart of SA is the Boltzmann acceptance criterion:

    P(accept) = e−ΔE / T

    • When ΔE < 0 (improving move), the exponent is positive, so P > 1 — we always accept.
    • When ΔE > 0 (worsening move) and T is large, the exponent is close to 0, so P ≈ 1 — we almost always accept.
    • When ΔE > 0 and T is small, the exponent is very negative, so P ≈ 0 — we almost never accept.

    Basic C++ Implementation

    #include <bits/stdc++.h>
using namespace std;

mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

double rand01() {
    return uniform_real_distribution<double>(0.0, 1.0)(rng);
}

int randint(int lo, int hi) {
    return uniform_int_distribution<int>(lo, hi)(rng);
}

struct State {
    vector<int> perm;
    double cost;
};

double compute_cost(const vector<int>& perm) {
    // problem-specific cost function
    double c = 0;
    for (int i = 0; i + 1 < (int)perm.size(); i++) {
        c += abs(perm[i] - perm[i + 1]);
    }
    return c;
}

State neighbor(State s) {
    int n = s.perm.size();
    int i = randint(0, n - 1);
    int j = randint(0, n - 1);
    swap(s.perm[i], s.perm[j]);
    s.cost = compute_cost(s.perm);
    return s;
}

State simulated_annealing(int n) {
    // Build initial solution
    State cur;
    cur.perm.resize(n);
    iota(cur.perm.begin(), cur.perm.end(), 0);
    shuffle(cur.perm.begin(), cur.perm.end(), rng);
    cur.cost = compute_cost(cur.perm);

    State best = cur;
    double T = 1e4;
    double T_min = 1e-6;
    double alpha = 0.999;

    while (T > T_min) {
        State nxt = neighbor(cur);
        double dE = nxt.cost - cur.cost;

        if (dE < 0 || rand01() < exp(-dE / T)) {
            cur = nxt;
        }
        if (cur.cost < best.cost) {
            best = cur;
        }
        T *= alpha;
    }
    return best;
}
    Always track the best. SA can wander away from good solutions near the end. Always keep a separate best variable and return it, not cur.

    Temperature Schedules

    The cooling schedule is arguably the most important design choice. It controls how quickly the algorithm transitions from exploration (random walk) to exploitation (greedy descent).

    Geometric (Exponential) Cooling

    The most common schedule in contests:

    T = T * alpha;    // alpha in [0.99, 0.99999]

    Temperature drops by a constant factor each iteration. With α = 0.9999 and 106 iterations, the temperature drops by a factor of about e−100 ≈ 0. This is the go-to schedule because it's simple and effective.

    Linear Cooling

    T = T_init * (1.0 - iteration / max_iterations);

    Temperature decreases linearly to zero. Easy to implement but spends too much time at intermediate temperatures and too little at low temperatures for fine-tuning.

    Logarithmic Cooling

    T = T_init / log(1 + iteration);

    Theoretically guarantees convergence to the global optimum, but is impractically slow for contests. Mentioned for completeness.

    Adaptive / Time-Based Cooling (Contest Favorite)

    Instead of counting iterations, we compute temperature as a function of elapsed time. This automatically adapts to the judge's machine speed:

    double time_limit = 1.95;  // seconds (leave margin from 2s limit)
auto start = chrono::steady_clock::now();

auto elapsed = [&]() -> double {
    auto now = chrono::steady_clock::now();
    return chrono::duration<double>(now - start).count();
};

double T_init = 1e4;
double T_end  = 1e-2;

while (true) {
    double t = elapsed();
    if (t >= time_limit) break;

    double progress = t / time_limit;              // 0 → 1
    double T = T_init * pow(T_end / T_init, progress); // geometric interpolation

    State nxt = neighbor(cur);
    double dE = nxt.cost - cur.cost;
    if (dE < 0 || rand01() < exp(-dE / T)) {
        cur = nxt;
    }
    if (cur.cost < best.cost) best = cur;
}
    Tuning initial and final temperature. A good heuristic: set T_init so that roughly 80% of random worsening moves are accepted at the start. Set T_end so that virtually no worsening moves are accepted at the end. You can estimate the typical ΔE by running a few hundred random neighbors and computing the average absolute cost change.

    How to Choose α

    Cooling Rate α Iterations to cool by 10× Use When
    0.99~230Fast cooling, small search space
    0.999~2300Medium problems, general default
    0.9999~23000Large search spaces, long time limits
    0.99999~230000Marathon contests (≥ 5s)

    SA in Competitive Programming

    Contest SA differs from textbook SA in several practical ways. Here are the patterns that matter.

    Time-Based Loops with clock()

    Calling chrono every iteration is expensive. A common trick is to check time only every K iterations:

    auto start = chrono::steady_clock::now();
double time_limit = 1.95;
int iter = 0;
double t = 0;

while (t < time_limit) {
    // Check clock every 256 iterations (bitwise trick)
    if ((iter & 255) == 0) {
        auto now = chrono::steady_clock::now();
        t = chrono::duration<double>(now - start).count();
    }

    double progress = t / time_limit;
    double T = T_init * pow(T_end / T_init, progress);

    // ... SA step ...
    iter++;
}
    Why 255? Checking (iter & 255) == 0 is a branchless bitwise test that fires every 256 iterations. This is much faster than iter % 256 == 0 and keeps the hot loop tight.

    Neighbor Generation Strategies

    The choice of neighbor function has a huge impact on SA performance. Here are common patterns:

    Problem Type Neighbor Move
    PermutationSwap two elements, or reverse a subarray (2-opt)
    Subset selectionAdd/remove one element, or swap one in/out
    Coordinate placementMove one point by a small random delta
    Assignment/coloringChange assignment of one item
    Binary stringFlip one or two bits

    Incremental Cost Updates

    Recomputing the entire cost function from scratch is often the bottleneck. Whenever possible, compute the delta cost from the neighbor move alone:

    // For a permutation problem with distance-based cost:
// Swapping positions i and j — only recalculate affected edges
double delta_cost_swap(const vector<int>& perm, int i, int j, 
                       const vector<vector<int>>& dist) {
    int n = perm.size();
    double delta = 0;

    // Remove old edges touching i and j
    auto edge_cost = [&](int a, int b) -> double {
        if (a < 0 || a >= n || b < 0 || b >= n) return 0;
        return dist[perm[a]][perm[b]];
    };

    // Subtract old contributions
    for (int d : {-1, 1}) {
        delta -= edge_cost(i, i + d);
        delta -= edge_cost(j, j + d);
    }

    // Perform the swap
    // (caller should swap perm[i] and perm[j] after accepting)

    // Add new contributions (with swapped values)
    swap(perm[i], perm[j]); // temporary
    for (int d : {-1, 1}) {
        delta += edge_cost(i, i + d);
        delta += edge_cost(j, j + d);
    }
    swap(perm[i], perm[j]); // undo

    // Handle the case where |i - j| == 1 (adjacent swap)
    if (abs(i - j) == 1) {
        delta += dist[perm[i]][perm[j]]; // double-counted edge
        delta -= dist[perm[i]][perm[j]];
    }
    return delta;
}

    Incremental updates can speed up SA by 10–100×, letting you run millions more iterations in the same time budget.

    Classic CP Problems

    1. Travelling Salesman (TSP) Approximation

    TSP is the quintessential SA problem. Given N cities and pairwise distances, find the shortest tour visiting all cities exactly once.

    // Neighbor: 2-opt reversal
State neighbor_tsp(State s) {
    int n = s.perm.size();
    int i = randint(0, n - 1);
    int j = randint(0, n - 1);
    if (i > j) swap(i, j);
    if (i == j) return s;

    // Reverse the segment [i, j]
    reverse(s.perm.begin() + i, s.perm.begin() + j + 1);
    s.cost = compute_tour_length(s.perm);
    return s;
}

// With incremental cost: only 2 edges change in a 2-opt move
double delta_2opt(const vector<int>& tour, int i, int j,
                  const vector<vector<double>>& dist) {
    int n = tour.size();
    int a = tour[i], b = tour[(i - 1 + n) % n];
    int c = tour[j], d = tour[(j + 1) % n];
    // Old edges: (b,a) + (c,d) → New edges: (b,c) + (a,d)
    return dist[b][c] + dist[a][d] - dist[b][a] - dist[c][d];
}

    2. Rectangle Placement / Packing

    Given rectangles to place on a grid, minimize overlap or maximize coverage. This appears in Google Hash Code and similar competitions.

    // State: each rectangle has (x, y) position
struct Placement {
    vector<pair<int,int>> positions;
    double cost; // overlap penalty + coverage bonus
};

Placement neighbor_placement(Placement s) {
    int idx = randint(0, (int)s.positions.size() - 1);
    int dx = randint(-5, 5);
    int dy = randint(-5, 5);
    s.positions[idx].first  += dx;
    s.positions[idx].second += dy;
    // Clamp to grid bounds
    s.positions[idx].first  = clamp(s.positions[idx].first,  0, W - 1);
    s.positions[idx].second = clamp(s.positions[idx].second, 0, H - 1);
    s.cost = compute_placement_cost(s);
    return s;
}

    3. Job Scheduling

    Minimize makespan or total weighted completion time. The state is typically a permutation of jobs, and neighbors are pairwise swaps or insert-moves.

    // Neighbor: move job at position i to position j (insert move)
State neighbor_schedule(State s) {
    int n = s.perm.size();
    int i = randint(0, n - 1);
    int j = randint(0, n - 1);
    if (i == j) return s;

    int val = s.perm[i];
    s.perm.erase(s.perm.begin() + i);
    s.perm.insert(s.perm.begin() + j, val);
    s.cost = compute_makespan(s.perm);
    return s;
}

double compute_makespan(const vector<int>& order) {
    // Simulate schedule: each job has processing time and machine assignment
    vector<double> machine_time(num_machines, 0);
    for (int job : order) {
        int m = job_machine[job];
        machine_time[m] += processing_time[job];
    }
    return *max_element(machine_time.begin(), machine_time.end());
}
    Marathon contest tip: For Hash Code–style problems, combine SA with a greedy initial solution. A good starting point lets SA focus on local improvements rather than wasting iterations escaping terrible states.

    Tuning Tips and Tricks

    How to Debug SA

    SA can feel like a black box. Use these techniques to understand what's happening:

    1. Print statistics every N iterations: Current cost, best cost, temperature, acceptance rate. If acceptance rate is always 1.0, your initial temperature is too high. If it drops to 0 immediately, it's too low.
    2. Plot cost over time: Pipe output to a file and plot with Python/gnuplot. You should see a noisy descent that flattens out.
    3. Track acceptance rate: Aim for ~80% at the start and ~1–5% at the end.
    4. Verify your neighbor function: Print a few neighbors and visually check they make sense. A bug in the neighbor function is the #1 cause of SA failures.
    // Debug output every 100000 iterations
if (iter % 100000 == 0) {
    int accepts_last_block = ...; // track in a counter
    double accept_rate = (double)accepts_last_block / 100000;
    fprintf(stderr, "iter=%d T=%.4f cur=%.2f best=%.2f accept=%.3f\n",
            iter, T, cur.cost, best.cost, accept_rate);
}

    Parameter Tuning Heuristics

    • T_init: Compute the average absolute ΔE over ~1000 random neighbors. Set T_init ≈ −ΔEavg / ln(0.8) so ~80% of worsening moves are initially accepted.
    • T_end: Set T_end ≈ −ΔEmin / ln(0.01) where ΔEmin is the smallest meaningful cost difference.
    • Neighbor size: Start with small moves. If you're stuck in local optima, try occasionally making larger moves (e.g., 3-opt instead of 2-opt).
    • Restarts: If time permits, run SA multiple times from different starting points and keep the best result. This is often more effective than one long run.

    When SA Works vs When It Doesn't

    SA Works Well SA Struggles
    Smooth cost landscapeHighly discrete / combinatorial with cliffs
    Good neighbor function existsHard to define meaningful neighbors
    Approximate answers acceptedExact optimum required
    Large connected search spaceFragmented feasible region
    Fast cost evaluation (or incremental)Expensive cost function, few iterations
    Key insight: SA is not magic. It works when you can (1) evaluate cost quickly, (2) generate neighbors that make small, meaningful changes, and (3) afford enough iterations to explore the space. If any of these fail, consider hill climbing, beam search, or genetic algorithms instead.

    Template Usage Checklist

    1. Define State to hold your solution representation.
    2. Implement initial_solution() — greedy is usually better than random.
    3. Implement gen_neighbor() — keep it fast, use incremental cost if possible.
    4. Implement evaluate() — or better, compute cost incrementally in gen_neighbor().
    5. Tune T_INIT and T_END based on your cost function's scale.
    6. Adjust TIME_LIMIT to your problem's time limit (leave 50–200ms margin).
