    Hill Climbing — Random Restart & Plateau Handling

    July 2025 · 22 min read

    What Is Hill Climbing?

    Hill climbing is the simplest local-search optimization strategy. Starting from an arbitrary solution, you repeatedly move to a neighboring solution that improves the objective function. When no neighbor is better, you stop — you've reached a local optimum.

    Think of it literally: you're blindfolded on a hilly landscape and can only feel the slope at your feet. You step uphill until every direction leads down. If you're lucky the hill you climbed is the tallest peak (global optimum). If not, you're stuck on a foothill.

    More formally, for a maximization problem with objective function f and a neighborhood function N(s) that produces the set of neighbors of state s:

    procedure HILL-CLIMB(initial_state):
    current ← initial_state
    loop:
        neighbor ← best state in N(current) with highest f(neighbor)
        if f(neighbor) ≤ f(current):
            return current          // local optimum
        current ← neighbor

    This "steepest-ascent" variant always picks the single best neighbor. Alternatives pick the first improving neighbor (first-choice) or a random improving neighbor (stochastic). We'll cover those later.

    Terminology note. Hill climbing is described for maximization. For minimization, simply negate the objective or flip the comparison. In CP you'll encounter both; just keep the direction consistent.

    Key properties of basic hill climbing:

    • Greedy & memoryless — no backtracking, no history.
    • O(1) memory — you only store the current state.
    • Fast per iteration — just evaluate neighbors.
    • Incomplete — may miss the global optimum.

    Basic Implementation

    Let's start with a concrete C++ template. We'll optimize a 1-D function to keep things clear, then generalize.

    #include <bits/stdc++.h>
using namespace std;

mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

// Objective: maximize f(x) = -(x-3)^2 + 25  (peak at x=3, value 25)
double f(double x) {
    return -(x - 3.0) * (x - 3.0) + 25.0;
}

// Generate neighbors: small perturbations
vector<double> neighbors(double x, double step) {
    return {x - step, x + step};
}

// Steepest-ascent hill climbing
double hill_climb(double start, double step, int max_iter) {
    double cur = start;
    for (int i = 0; i < max_iter; i++) {
        double best_val = f(cur);
        double best_nb = cur;
        for (double nb : neighbors(cur, step)) {
            double val = f(nb);
            if (val > best_val) {
                best_val = val;
                best_nb = nb;
            }
        }
        if (best_nb == cur) break;   // no improvement → local optimum
        cur = best_nb;
    }
    return cur;
}

int main() {
    double result = hill_climb(/*start=*/-10.0, /*step=*/0.01, /*max_iter=*/100000);
    printf("Optimum at x = %.4f, f(x) = %.4f\n", result, f(result));
    // Output: Optimum at x = 3.0000, f(x) = 25.0000
}

    For discrete combinatorial problems (common in CP), the pattern is identical but the state is a permutation or assignment, and neighbors are generated by swaps, 2-opt reversals, or insertions. We'll see full combinatorial examples in the sections below.

    Neighborhood design is everything. The quality of hill climbing hinges on how you define neighbors. Too small a neighborhood and you'll miss good moves; too large and each iteration becomes expensive. For permutations, 2-opt (reverse a subarray) and swap (exchange two elements) are the most common choices in CP.

    The Local Optima Problem

    The fundamental weakness of hill climbing is getting trapped in local optima. Consider a function with many peaks:

    // Multi-modal function: many local maxima
double f_multimodal(double x) {
    return sin(x) * sin(x) + 0.5 * cos(3.7 * x) + 0.3 * sin(7.1 * x + 1.0);
}

    Starting at a random x, steepest-ascent will climb to the nearest peak and stop. With dozens of peaks in any reasonable interval, the chance of landing on the global maximum is slim.

    Three categories of problematic landscapes:

    Landscape FeatureProblemExample
    Local maximumAll neighbors are worse, but better states exist elsewhere.A small peak next to a tall mountain.
    Plateau (flat region)All neighbors have the same value. No gradient to follow.A flat mesa — the optimum might be at the edge.
    RidgeA narrow ascending path that is not aligned with any single neighbor direction.Diagonal ascent when you can only move N/S/E/W.

    In competitive programming, local-optima traps manifest frequently:

    • Scheduling problems — swapping two jobs improves locally but a three-way rotation would be much better.
    • Graph coloring — reducing conflicts for one node may increase conflicts elsewhere.
    • Geometric optimization — moving one point improves distance locally but misses the true configuration.

    The takeaway: basic hill climbing is a building block, not a complete algorithm. We need strategies to escape local optima.

    Random Restart Hill Climbing

    The simplest and often most effective fix: run hill climbing many times from random starting points and keep the best result. This is sometimes called "shotgun hill climbing."

    The intuition is probabilistic. If each run has a probability p of finding the global optimum, then after k independent runs the probability of success is 1 - (1-p)^k. Even with a tiny p, enough restarts make success near-certain.

    // Random-restart hill climbing for a continuous function
double random_restart_hc(double lo, double hi, double step,
                         int max_iter, int restarts) {
    double global_best_x = lo;
    double global_best_val = -1e18;

    uniform_real_distribution<double> dist(lo, hi);

    for (int r = 0; r < restarts; r++) {
        double start = dist(rng);
        double x = hill_climb(start, step, max_iter);
        double val = f(x);
        if (val > global_best_val) {
            global_best_val = val;
            global_best_x = x;
        }
    }
    return global_best_x;
}

    For combinatorial problems in CP, generating a random starting permutation (or assignment) is straightforward:

    // Random-restart for permutation problems
State random_restart_perm(int n, int restarts, int max_iter) {
    State global_best;
    global_best.cost = LLONG_MAX;

    for (int r = 0; r < restarts; r++) {
        // Generate random permutation
        vector<int> perm(n);
        iota(perm.begin(), perm.end(), 0);
        shuffle(perm.begin(), perm.end(), rng);

        State local = hill_climb_perm(perm, max_iter);
        if (local.cost < global_best.cost) {
            global_best = local;
        }
    }
    return global_best;
}
    Time budget strategy. In CP contests you often have 2–3 seconds. A practical pattern is to run restarts in a time loop rather than a fixed count: 
    auto deadline = chrono::steady_clock::now() + chrono::milliseconds(1800);
while (chrono::steady_clock::now() < deadline) {
    // one restart iteration
}
    This automatically adapts to different machine speeds and problem sizes.

    How many restarts do you need? It depends on the landscape. Some guidelines for CP:

    • Few deep local optima (smooth landscape): 10–50 restarts often suffice.
    • Many shallow local optima (rugged landscape): hundreds or thousands of short climbs can outperform a few long ones.
    • Mixed strategy: allocate ~80% of time to restarts, ~20% to the climb iterations per restart. Tune empirically.

    A complete random-restart solver for a competitive-programming contest:

    #include <bits/stdc++.h>
using namespace std;

mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

int n;
int cost[505][505];  // adjacency cost matrix

long long evaluate(const vector<int>& p) {
    long long c = 0;
    for (int i = 0; i + 1 < n; i++)
        c += cost[p[i]][p[i + 1]];
    c += cost[p[n - 1]][p[0]];  // return to start
    return c;
}

// Delta evaluation for 2-opt: O(1) instead of O(n)
long long delta_2opt(const vector<int>& p, int i, int j) {
    int a = p[i], b = p[(i + 1) % n];
    int c = p[j], d = p[(j + 1) % n];
    return (long long)cost[a][c] + cost[b][d]
         - (long long)cost[a][b] - cost[c][d];
}

vector<int> hill_climb_2opt(vector<int> p, int max_iter) {
    long long cur_cost = evaluate(p);
    for (int it = 0; it < max_iter; it++) {
        bool improved = false;
        for (int i = 0; i < n - 1 && !improved; i++) {
            for (int j = i + 2; j < n; j++) {
                if (i == 0 && j == n - 1) continue;
                long long d = delta_2opt(p, i, j);
                if (d < 0) {
                    reverse(p.begin() + i + 1, p.begin() + j + 1);
                    cur_cost += d;
                    improved = true;
                    break;
                }
            }
        }
        if (!improved) break;
    }
    return p;
}

int main() {
    scanf("%d", &n);
    for (int i = 0; i < n; i++)
        for (int j = 0; j < n; j++)
            scanf("%d", &cost[i][j]);

    vector<int> best_perm(n);
    iota(best_perm.begin(), best_perm.end(), 0);
    long long best_cost = evaluate(best_perm);

    auto deadline = chrono::steady_clock::now() + chrono::milliseconds(1800);
    int restarts = 0;

    while (chrono::steady_clock::now() < deadline) {
        vector<int> p(n);
        iota(p.begin(), p.end(), 0);
        shuffle(p.begin(), p.end(), rng);

        p = hill_climb_2opt(p, 5000);
        long long c = evaluate(p);
        if (c < best_cost) {
            best_cost = c;
            best_perm = p;
        }
        restarts++;
    }

    printf("Best cost: %lld after %d restarts\n", best_cost, restarts);
    for (int i = 0; i < n; i++)
        printf("%d ", best_perm[i] + 1);
    printf("\n");
}

    Plateau Handling

    Plateaus are regions where all neighbors have the same objective value as the current state. The basic algorithm sees no improving move and stops — but the global optimum might be just across the flat region.

    Three strategies for handling plateaus:

    Strategy 1: Sideways Moves

    Allow moves to neighbors with equal value (not just strictly better). Cap the number of consecutive sideways moves to avoid infinite loops on true flat regions.

    double hill_climb_sideways(double start, double step,
                           int max_iter, int max_sideways) {
    double cur = start;
    int sideways_count = 0;

    for (int i = 0; i < max_iter; i++) {
        double best_val = f(cur);
        double best_nb = cur;
        for (double nb : neighbors(cur, step)) {
            double val = f(nb);
            if (val > best_val) {
                best_val = val;
                best_nb = nb;
            }
        }

        if (best_nb == cur) {
            // No improving neighbor — try sideways
            vector<double> sideways;
            for (double nb : neighbors(cur, step)) {
                if (abs(f(nb) - f(cur)) < 1e-12)
                    sideways.push_back(nb);
            }
            if (sideways.empty() || sideways_count >= max_sideways)
                break;
            // Pick a random sideways move
            cur = sideways[rng() % sideways.size()];
            sideways_count++;
        } else {
            cur = best_nb;
            sideways_count = 0;  // reset counter on uphill move
        }
    }
    return cur;
}
    How many sideways moves? Russell & Norvig report that allowing up to 100 consecutive sideways moves raises the success rate of hill climbing on the 8-queens problem from 14% to 94%. The right limit is problem-specific, but 50–200 is a good starting range.

    Strategy 2: Random Walk on Plateaus

    When stuck on a plateau, take a sequence of random steps (ignoring the objective) for a fixed number of moves, then resume climbing. This can teleport you to a different region of the plateau that might have an exit.

    // Combinatorial version: random walk escape
vector<int> random_walk_escape(vector<int> perm, int walk_len) {
    int n = perm.size();
    for (int w = 0; w < walk_len; w++) {
        int i = rng() % n;
        int j = rng() % n;
        if (i != j) swap(perm[i], perm[j]);
    }
    return perm;
}

State hill_climb_with_escape(vector<int> init, int max_iter,
                             int max_plateau, int walk_len) {
    State cur = {init, evaluate(init)};
    State best = cur;
    int plateau_count = 0;

    for (int it = 0; it < max_iter; it++) {
        // Try all 2-opt neighbors for best improvement
        bool improved = false;
        for (int i = 0; i < (int)cur.perm.size() - 1; i++) {
            for (int j = i + 2; j < (int)cur.perm.size(); j++) {
                vector<int> nb = cur.perm;
                reverse(nb.begin() + i, nb.begin() + j + 1);
                long long nc = evaluate(nb);
                if (nc < cur.cost) {
                    cur = {nb, nc};
                    improved = true;
                    plateau_count = 0;
                    break;
                }
            }
            if (improved) break;
        }

        if (!improved) {
            plateau_count++;
            if (plateau_count >= max_plateau) {
                // Escape via random walk
                cur.perm = random_walk_escape(cur.perm, walk_len);
                cur.cost = evaluate(cur.perm);
                plateau_count = 0;
            }
        }

        if (cur.cost < best.cost) best = cur;
    }
    return best;
}

    Strategy 3: Increasing Step Size

    For continuous optimization, adaptively grow the step size when stuck. This lets you "jump" off a flat region:

    double hill_climb_adaptive(double start, double init_step,
                           int max_iter) {
    double cur = start;
    double step = init_step;
    int stall = 0;

    for (int i = 0; i < max_iter; i++) {
        double best_val = f(cur);
        double best_nb = cur;
        for (double nb : neighbors(cur, step)) {
            if (f(nb) > best_val) {
                best_val = f(nb);
                best_nb = nb;
            }
        }
        if (best_nb == cur) {
            stall++;
            if (stall > 10) {
                step *= 2.0;   // widen search
                stall = 0;
            }
        } else {
            cur = best_nb;
            step = init_step;  // reset to fine-grained
            stall = 0;
        }
    }
    return cur;
}

    Variants

    Several modifications of basic hill climbing are worth knowing for CP.

    Stochastic Hill Climbing

    Instead of always picking the best neighbor, pick randomly among improving neighbors. This adds diversity and can help escape ridges where steepest-ascent always picks the same direction.

    double stochastic_hc(double start, double step, int max_iter) {
    double cur = start;
    for (int i = 0; i < max_iter; i++) {
        vector<double> improving;
        for (double nb : neighbors(cur, step)) {
            if (f(nb) > f(cur))
                improving.push_back(nb);
        }
        if (improving.empty()) break;
        cur = improving[rng() % improving.size()];
    }
    return cur;
}

    First-Choice Hill Climbing

    When the neighborhood is very large (e.g., all possible 2-opt swaps in a large permutation), evaluating every neighbor is expensive. First-choice generates random neighbors one at a time and accepts the first improvement:

    State first_choice_hc(vector<int> init, int max_iter, int max_tries) {
    State cur = {init, evaluate(init)};
    for (int it = 0; it < max_iter; it++) {
        bool found = false;
        for (int t = 0; t < max_tries; t++) {
            // Generate one random neighbor
            int n = cur.perm.size();
            int i = rng() % n, j = rng() % n;
            if (i == j) continue;
            if (i > j) swap(i, j);

            vector<int> nb = cur.perm;
            reverse(nb.begin() + i, nb.begin() + j + 1);
            long long nc = evaluate(nb);

            if (nc < cur.cost) {
                cur = {nb, nc};
                found = true;
                break;
            }
        }
        if (!found) break;  // no improving neighbor found
    }
    return cur;
}
    First-choice is often the best default for CP. In contests, neighborhoods are large and you're time-constrained. Generating a random neighbor and accepting any improvement gives you many more iterations per second than exhaustive neighborhood scans.

    Tabu Search (Lite)

    Tabu search prevents cycling by maintaining a short-term memory of recently visited states (or recently made moves). Any move that reverses a recent action is "tabu" (forbidden). This forces the search to explore new territory even if it means accepting a slightly worse solution.

    State tabu_search_lite(vector<int> init, int max_iter, int tabu_tenure) {
    State cur = {init, evaluate(init)};
    State best = cur;

    // tabu_list[i][j] = iteration when swap(i,j) becomes non-tabu
    int n = cur.perm.size();
    vector<vector<int>> tabu_list(n, vector<int>(n, 0));

    for (int it = 0; it < max_iter; it++) {
        int best_i = -1, best_j = -1;
        long long best_delta = LLONG_MAX;

        for (int i = 0; i < n; i++) {
            for (int j = i + 1; j < n; j++) {
                // Skip tabu moves (unless aspiration criterion met)
                vector<int> nb = cur.perm;
                swap(nb[i], nb[j]);
                long long nc = evaluate(nb);
                long long delta = nc - cur.cost;

                bool is_tabu = (tabu_list[i][j] > it);
                bool aspiration = (nc < best.cost);

                if ((!is_tabu || aspiration) && delta < best_delta) {
                    best_delta = delta;
                    best_i = i;
                    best_j = j;
                }
            }
        }

        if (best_i == -1) break;

        // Apply the move
        swap(cur.perm[best_i], cur.perm[best_j]);
        cur.cost += best_delta;

        // Mark reverse move as tabu
        tabu_list[best_i][best_j] = it + tabu_tenure;
        tabu_list[best_j][best_i] = it + tabu_tenure;

        if (cur.cost < best.cost) best = cur;
    }
    return best;
}

    The aspiration criterion overrides the tabu restriction: if a tabu move leads to the best-ever solution, allow it. This prevents the tabu list from blocking genuinely good moves.

    VariantAdvantageBest For
    Steepest ascentFinds best local optimum quicklySmall neighborhoods
    First-choiceFast iterations, good for large neighborhoodsPermutation/assignment problems
    StochasticMore diversity in search pathRidge-heavy landscapes
    Tabu searchEscapes local optima via memoryProblems with cycling

    Hill Climbing in Competitive Programming

    Hill climbing (especially with random restarts) is a competitive programmer's secret weapon for optimization problems that don't have clean polynomial algorithms. Here's when to reach for it:

    When to Use Hill Climbing

    • The problem asks for "minimize/maximize" with no obvious greedy/DP structure. Problems tagged "heuristic" or "approximate" on platforms like Codeforces or AtCoder are prime candidates.
    • The search space is too large for brute force but has local structure. Small changes to a good solution tend to produce nearby-quality solutions.
    • Time limits are generous (2–5 seconds) and the scoring is relative (your score vs. other participants).
    • The neighborhood is natural and cheap to evaluate. Swap two elements, toggle a bit, move one point — if these make sense for the problem, hill climbing will work.

    Hill Climbing vs. Simulated Annealing

    Simulated annealing (SA) extends hill climbing by accepting worse solutions with a probability that decreases over time. When should you pick one over the other?

    CriterionHill Climbing + RestartSimulated Annealing
    ImplementationSimpler, fewer parametersNeeds temperature schedule tuning
    Local optima escapeVia restarts (blunt but effective)Via probabilistic acceptance (gradual)
    Deep local optimaStruggles; restarts are independentCan tunnel through barriers
    Tuning effortMinimal (just time budget)Moderate (T₀, T_final, cooling rate)
    Contest recommendationStart here, upgrade if neededUse when HC+restart plateaus
    Contest tip: Start with random-restart first-choice hill climbing. If it plateaus too early (you can tell by printing the best score every few hundred restarts), switch to simulated annealing. Hill climbing is your fast prototype; SA is the refined version.

    Practical Example: Minimizing Maximum Distance

    A common CP pattern: place k facilities among n candidate locations to minimize the maximum distance from any city to its nearest facility. This is NP-hard in general, perfect for hill climbing.

    #include <bits/stdc++.h>
using namespace std;
mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

int n, m, k;
int d[305][305];     // d[i][j] = distance from city i to candidate j

// State: which k candidates are selected
struct Sol {
    vector<int> chosen;  // sorted indices of chosen candidates
    int obj;             // max over all cities of min-dist-to-chosen
};

int calc_obj(const vector<int>& chosen) {
    int mx = 0;
    for (int i = 0; i < n; i++) {
        int mn = INT_MAX;
        for (int c : chosen)
            mn = min(mn, d[i][c]);
        mx = max(mx, mn);
    }
    return mx;
}

Sol hill_climb_facility(int max_iter) {
    // Random initial: pick k out of m candidates
    vector<int> all(m);
    iota(all.begin(), all.end(), 0);
    shuffle(all.begin(), all.end(), rng);

    Sol cur;
    cur.chosen.assign(all.begin(), all.begin() + k);
    sort(cur.chosen.begin(), cur.chosen.end());
    cur.obj = calc_obj(cur.chosen);

    set<int> in_set(cur.chosen.begin(), cur.chosen.end());

    for (int it = 0; it < max_iter; it++) {
        // Neighborhood: swap one chosen candidate with one unchosen
        bool improved = false;
        for (int ci = 0; ci < k && !improved; ci++) {
            for (int j = 0; j < m && !improved; j++) {
                if (in_set.count(j)) continue;
                // Swap chosen[ci] with j
                int old_val = cur.chosen[ci];
                cur.chosen[ci] = j;
                int new_obj = calc_obj(cur.chosen);
                if (new_obj < cur.obj) {
                    in_set.erase(old_val);
                    in_set.insert(j);
                    cur.obj = new_obj;
                    sort(cur.chosen.begin(), cur.chosen.end());
                    improved = true;
                } else {
                    cur.chosen[ci] = old_val;  // revert
                }
            }
        }
        if (!improved) break;
    }
    return cur;
}

int main() {
    scanf("%d %d %d", &n, &m, &k);
    for (int i = 0; i < n; i++)
        for (int j = 0; j < m; j++)
            scanf("%d", &d[i][j]);

    Sol best;
    best.obj = INT_MAX;

    auto deadline = chrono::steady_clock::now() + chrono::milliseconds(2500);
    while (chrono::steady_clock::now() < deadline) {
        Sol s = hill_climb_facility(500);
        if (s.obj < best.obj) best = s;
    }

    printf("%d\n", best.obj);
    for (int c : best.chosen)
        printf("%d ", c + 1);
    printf("\n");
}

    Summary Checklist

    When implementing hill climbing in a contest, work through this checklist:

    1. Define the state representation — permutation, subset, vector of values, etc.
    2. Define the objective function — what are you maximizing or minimizing?
    3. Design the neighborhood — swap, 2-opt, insert, flip? Keep it small enough for fast evaluation.
    4. Implement delta evaluation if possible — O(1) re-evaluation instead of O(n) makes a huge difference.
    5. Use first-choice to maximize iterations per second.
    6. Add random restarts with a time-based loop.
    7. Handle plateaus — allow sideways moves or random-walk escapes.
    8. Track global best across all restarts.
    9. Test locally with multiple random seeds to ensure robustness.
    Remember: Hill climbing is not a substitute for exact algorithms when they exist. Always look for greedy, DP, or flow solutions first. Hill climbing is your fallback for problems where the optimal structure isn't apparent or the problem is explicitly NP-hard.
