Optimization Patterns from CF/ICPC

June 2025 · 35 min read

Competitive programming is, at its core, about optimization. Nearly every problem asks you to minimize cost, maximize profit, or count something subject to constraints. Over the years a rich toolkit of optimization meta-techniques has crystallized from thousands of Codeforces rounds and ICPC regionals. This post is a practitioner's field guide to the most important ones — each section includes the key idea, when to reach for the technique, a clean C++ implementation, and a curated list of practice problems.

Recognizing Optimization Problems

Before diving into specific techniques, let's build intuition for which tool to grab. Optimization problems in CP tend to fall into a few recurring shapes:

  • Minimize / Maximize a single value — Classic DP, greedy, or binary search.
  • Minimize subject to a cardinality constraint — "Pick exactly k items to minimize cost" → think WQS binary search.
  • Monotonic feasibility — "Can we achieve cost ≤ X?" is monotone in X → binary search on the answer.
  • Pairwise ordering decisions — "In what order should we process items?" → exchange argument.
  • 1D DP with convex structure — The cost landscape is convex/concave → slope trick or SMAWK.
  • Batch transitions in DP — Transition cost satisfies the quadrangle inequality → divide-and-conquer optimization.
Rule of Thumb: If you see "minimize the maximum" or "maximize the minimum," your first instinct should be binary search on the answer. If you see "arrange items in some order to minimize total cost," think exchange argument. If the DP has a convex cost function that shifts with each operation, think slope trick.

The rest of this post develops each technique in depth. We progress from the most broadly applicable (binary search) to the most specialized (WQS binary search), and close with a curated problem list organized by technique.

Binary Search on the Answer

The idea is deceptively simple: instead of computing the optimal answer directly, guess the answer and check whether that guess is feasible. If feasibility is monotone (once it becomes feasible it stays feasible, or vice versa), a binary search over the answer space finds the optimum in O(log(range) · Tcheck).

When It Applies

  • The answer lives in a bounded numeric range [lo, hi].
  • You can write a predicate feasible(mid) that runs in polynomial time.
  • The predicate is monotone: feasible(x) ⇒ feasible(x+1) (or the reverse).

Classic Example — Minimize the Maximum

"Split array a into k contiguous subarrays so that the maximum subarray sum is minimized." The check "can we split so that no subarray exceeds mid?" is a simple greedy scan. 

// Minimize the maximum subarray sum when splitting into at most k parts
#include <bits/stdc++.h>
using namespace std;

bool feasible(const vector<long long>& a, int k, long long maxSum) {
    int parts = 1;
    long long cur = 0;
    for (long long x : a) {
        if (x > maxSum) return false;   // single element exceeds limit
        if (cur + x > maxSum) {
            ++parts;
            cur = x;
        } else {
            cur += x;
        }
    }
    return parts <= k;
}

long long solve(const vector<long long>& a, int k) {
    long long lo = *max_element(a.begin(), a.end());
    long long hi = accumulate(a.begin(), a.end(), 0LL);
    while (lo < hi) {
        long long mid = lo + (hi - lo) / 2;
        if (feasible(a, k, mid))
            hi = mid;
        else
            lo = mid + 1;
    }
    return lo;
}

Parametric Search on Real Values

For problems where the answer is a real number (e.g., "maximize the minimum distance between points"), use floating-point binary search. Typically 60–80 iterations of halving suffice for full double precision. 

// Parametric binary search over real-valued answer
double solve(/* problem params */) {
    double lo = 0, hi = 1e18;
    for (int iter = 0; iter < 100; ++iter) {
        double mid = (lo + hi) / 2;
        if (feasible(mid))
            lo = mid;     // try to push higher
        else
            hi = mid;
    }
    return lo;
}
Pitfall: Make sure feasible truly is monotone. A common mistake is binary-searching when feasibility is not monotonic (e.g., problems with non-convex structure). Prove monotonicity before coding.

Exchange Argument

The exchange argument is the standard technique for proving that a greedy ordering is optimal. The recipe: take any optimal solution, find two adjacent elements that are out of order relative to your greedy criterion, swap them, and show the cost doesn't increase.

The Recipe

  1. Define a comparator cmp(i, j) that decides whether item i should come before item j.
  2. Take any optimal ordering. Find adjacent pair (i, j) where cmp(j, i) holds but j appears after i.
  3. Swap i and j. Show cost(new) ≤ cost(old).
  4. By repeated swaps (bubble sort), the greedy ordering is at least as good as any ordering.

Classic: Weighted Job Scheduling (Smith's Rule)

n jobs with processing times pi and weights wi. Minimize weighted completion time Σ wi · Ci. The optimal order is by increasing pi / wi. 

// Smith's Rule: minimize weighted completion time
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<pair<int,int>> jobs(n); // {processing_time, weight}
    for (auto& [p, w] : jobs) cin >> p >> w;

    // Sort by p/w ratio ascending  ⟺  p_i * w_j < p_j * w_i
    sort(jobs.begin(), jobs.end(), [](auto& a, auto& b) {
        return (long long)a.first * b.second < (long long)b.first * a.second;
    });

    long long time = 0, cost = 0;
    for (auto& [p, w] : jobs) {
        time += p;
        cost += (long long)w * time;
    }
    cout << cost << "\n";
}

Proof Sketch via Exchange

Consider adjacent jobs i, j in some optimal schedule. If we swap them, the change in cost is:

Δ = wj · pi − wi · pj

The swap improves (or keeps equal) cost when Δ ≥ 0, i.e., pi/wi ≤ pj/wj. So the greedy order by ascending p/w ratio is optimal. ∎

Key Insight: Exchange arguments work whenever the contribution of two adjacent elements depends only on each other (not on the rest of the sequence). This "local" property is what makes the swap analysis tractable.

More Exchange Patterns

  • Deadline scheduling: Sort by deadline (EDF) to maximize number of jobs completed on time.
  • Minimum penalty: Jobs with deadlines and penalties — sort by penalty/time ratio (similar to Smith's rule).
  • CF 1203F2 — Microtransactions: Exchange argument on purchase timings.

Slope Trick

Slope trick is an elegant technique for DP problems whose cost function is piecewise-linear and convex. Instead of tracking the full DP table, we maintain the set of breakpoints (slope-change positions) of the cost function using two priority queues.

Core Idea

Suppose f(x) is convex and piecewise-linear with slopes that increase by 1 at each breakpoint. We can represent f by:

  • A min-cost value (the minimum of f).
  • A max-heap L of breakpoints on the left side (slopes go from −∞ to 0).
  • A min-heap R of breakpoints on the right side (slopes go from 0 to +∞).

Common operations — adding |x − a|, shifting the function, taking prefix/suffix minimums — each become O(log n) push/pop operations on the heaps.

Classic Problem: Make Array Non-Decreasing

Given array a[1..n], change elements to make it non-decreasing. Minimize Σ|a[i] − b[i]|. This is the prototypical slope trick problem. 

// Slope trick: minimum cost to make array non-decreasing
// Cost = sum of |a[i] - b[i]| where b is non-decreasing
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;

    // L = max-heap of left breakpoints, R = min-heap of right breakpoints
    priority_queue<long long> L;                              // max-heap
    priority_queue<long long, vector<long long>, greater<>> R; // min-heap
    long long cost = 0;

    for (int i = 0; i < n; ++i) {
        long long a;
        cin >> a;

        // Add |x - a| to current cost function
        L.push(a);
        R.push(a);

        // Merge: if leftmost right breakpoint < rightmost left breakpoint, swap
        if (L.top() > R.top()) {
            long long l = L.top(), r = R.top();
            L.pop(); R.pop();
            cost += l - r;     // cost of fixing the overlap
            L.push(r);
            R.push(l);
        }
    }

    cout << cost << "\n";
}

When to Use Slope Trick: Look for DP recurrences of the form dp[i][x] = min over y { dp[i-1][y] + cost(x,y) + |x - a[i]| } where the transition preserves convexity. The telltale sign is absolute-value cost terms and constraints like "non-decreasing" or "bounded difference."

SMAWK and Divide-and-Conquer Optimization

Both techniques exploit structure in the transition matrix of a DP. When the "optimal split point" (argmin) is monotone, we can dramatically reduce the time complexity.

Divide-and-Conquer DP Optimization

Applies to recurrences of the form:

dp[i][j] = mink ≤ j { dp[i−1][k] + C(k+1, j) }

If the optimal k for dp[i][j] is ≤ the optimal k for dp[i][j+1] (monotonicity of the argmin), then we can solve each "layer" i in O(n log n) instead of O(n²) via divide-and-conquer on the range of j.

Sufficient Condition: Quadrangle Inequality

The cost function C satisfies the quadrangle inequality if for all a ≤ b ≤ c ≤ d:

C(a, c) + C(b, d) ≤ C(a, d) + C(b, c)

This guarantees the monotonicity of optimal split points.

// Divide-and-conquer DP optimization
// dp[j] = min over k in [0, j) of { old_dp[k] + cost(k, j) }
// opt[j] is monotone non-decreasing in j.
#include <bits/stdc++.h>
using namespace std;

const long long INF = 1e18;
int n, K;
vector<long long> dp, old_dp;

// cost(l, r): cost of segment [l, r) — problem-specific
long long cost(int l, int r);

void solve(int lo, int hi, int opt_lo, int opt_hi) {
    if (lo > hi) return;
    int mid = (lo + hi) / 2;
    long long best = INF;
    int opt = opt_lo;
    for (int k = opt_lo; k <= min(mid - 1, opt_hi); ++k) {
        long long val = old_dp[k] + cost(k, mid);
        if (val < best) {
            best = val;
            opt = k;
        }
    }
    dp[mid] = best;
    solve(lo, mid - 1, opt_lo, opt);
    solve(mid + 1, hi, opt, opt_hi);
}

int main() {
    cin >> n >> K;
    dp.assign(n + 1, INF);
    old_dp.assign(n + 1, INF);
    dp[0] = 0;

    for (int i = 1; i <= K; ++i) {
        old_dp = dp;
        fill(dp.begin(), dp.end(), INF);
        solve(1, n, 0, n - 1);
    }
    cout << dp[n] << "\n";
}

SMAWK Algorithm

SMAWK finds the row minima of a totally monotone n × m matrix in O(n + m) time. A matrix is totally monotone if every 2×2 submatrix has its column minimum in the left column when the row minimum is in the top row. This is a stronger condition than the quadrangle inequality.

D&C vs. SMAWK: Divide-and-conquer optimization gives O(n log n) per DP layer and is much simpler to implement. SMAWK gives O(n) per layer but is rarely needed in contests. Reach for D&C optimization first; only consider SMAWK if the time limit is extremely tight or you have many DP layers.

The full SMAWK implementation is relatively involved (REDUCE step + recursive odd-row solve + interleave). In practice, the D&C optimization above covers 99% of contest use cases. Reserve SMAWK for problems with many DP layers and extremely tight time limits.

Aliens / WQS Binary Search

Also known as the Lagrangian relaxation trick or lambda optimization, WQS binary search is named after the IOI 2016 problem "Aliens." It converts a constrained optimization problem into an unconstrained one by penalizing the constraint.

The Setup

You have a DP that computes f(k) = minimum cost using exactly k items. Computing f(k) for a single k might be O(n²) or worse, but the unconstrained version (use any number of items) is much easier — perhaps O(n log n).

Key requirement: f(k) must be convex (or concave) in k.

The Trick

Define g(λ) = mink { f(k) + λ · k }. For each fixed λ, this is an unconstrained problem. By convexity, there exists a λ* such that the unconstrained optimum uses exactly k items. We binary search on λ. 

// WQS Binary Search (Aliens trick)
// Suppose we want f(k) = min cost using exactly k segments.
// f is convex in k. unconstrained_solve(penalty) returns {cost, count}
// where cost = min over all k of { f(k) + penalty * k }, and count = k* used.
#include <bits/stdc++.h>
using namespace std;

// Returns {total_cost_with_penalty, number_of_segments_used}
pair<long long, int> unconstrained_solve(const vector<int>& a,
                                           long long penalty) {
    int n = a.size();
    // Example: dp[i] = min cost to cover a[0..i-1] with penalty per segment
    vector<long long> dp(n + 1, LLONG_MAX);
    vector<int> cnt(n + 1, 0);
    dp[0] = 0;
    cnt[0] = 0;
    for (int i = 1; i <= n; ++i) {
        for (int j = 0; j < i; ++j) {
            long long seg_cost = /* cost of segment [j+1, i] */ 0;
            long long val = dp[j] + seg_cost + penalty;
            if (val < dp[i] || (val == dp[i] && cnt[j] + 1 < cnt[i])) {
                dp[i] = val;
                cnt[i] = cnt[j] + 1;
            }
        }
    }
    return {dp[n], cnt[n]};
}

long long solve(const vector<int>& a, int target_k) {
    long long lo = 0, hi = 1e15;
    long long ans = LLONG_MAX;
    while (lo <= hi) {
        long long mid = lo + (hi - lo) / 2;
        auto [cost, k] = unconstrained_solve(a, mid);
        // cost = f(k) + mid * k, so f(k) = cost - mid * k
        if (k <= target_k) {
            ans = cost - mid * target_k;
            hi = mid - 1;
        } else {
            lo = mid + 1;
        }
    }
    return ans;
}

Why Convexity Matters

If f(k) is convex, then g(λ) is the Legendre transform of f. Each tangent line of slope −λ touches the convex hull of f at some point k, and binary-searching λ lets us find the tangent that touches at exactly our target k.

Tie-breaking is critical. When multiple values of k achieve the same penalized cost, you must track the smallest (or largest) k to ensure the binary search converges correctly. Failing to handle ties is the #1 source of WA on WQS problems.
Complexity Gain: If unconstrained_solve is O(n log n) and the answer range has size A, then WQS gives O(n log n · log A) total — often dramatically faster than O(n · k) for the constrained DP.
