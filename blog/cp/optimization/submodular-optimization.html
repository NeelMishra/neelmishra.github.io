Submodular Functions — Greedy Guarantees & Optimization

June 2025 · 20 min read

What Are Submodular Functions?

A set function f : 2V → ℝ defined on a ground set V is submodular if it satisfies the diminishing returns property: for every A ⊆ B ⊆ V and every element x ∈ V \ B,

f(A ∪ {x}) − f(A)  ≥  f(B ∪ {x}) − f(B)

In words: adding element x to a smaller set gives at least as much marginal gain as adding it to a larger set that contains the smaller one. This is exactly the "diminishing returns" intuition from economics.

Equivalent Definition

An equivalent characterization uses the submodular inequality: for all A, B ⊆ V,

f(A) + f(B)  ≥  f(A ∪ B) + f(A ∩ B)

These two definitions are provably equivalent. Use whichever is more convenient for the proof at hand.

Classic Examples

Submodular functions appear naturally in many settings:

  • Coverage function: Given a universe U and a family of sets S₁, S₂, …, Sₙ ⊆ U, define f(A) = |⋃i∈A Sᵢ|. The number of elements covered grows with diminishing returns because later sets overlap earlier ones.
  • Cut function: For a graph G = (V, E), define f(S) = |{(u,v) ∈ E : u ∈ S, v ∉ S}|, the number of edges crossing the cut. This is submodular (but not monotone).
  • Matroid rank function: For any matroid (V, I), the rank function r(A) = max{|I| : I ⊆ A, I ∈ I} is submodular and monotone.
  • Entropy: The entropy H(XA) of a subset of random variables is submodular — conditioning on more variables gives diminishing information gain.
Key intuition: If you can argue that "the marginal benefit of adding an element shrinks as the current set grows," you likely have a submodular function. This intuition is the most useful tool for recognizing submodularity in contest problems.

Monotone vs Non-monotone

A submodular function f is monotone if A ⊆ B implies f(A) ≤ f(B) — adding elements never decreases the function value. This is a critical distinction because the algorithmic guarantees differ dramatically.

Monotone Submodular

  • Coverage functions
  • Matroid rank
  • Influence spread
  • Greedy gives (1 − 1/e) ≈ 0.632 approximation under cardinality constraint
  • Tight: no polynomial algorithm can do better (unless P = NP)

Non-monotone Submodular

  • Cut functions
  • Graph bisection objectives
  • Certain diversity measures
  • Greedy alone gives only 1/4 approximation
  • Randomized algorithms achieve 1/e ≈ 0.368

For competitive programming, the vast majority of problems involve monotone submodular maximization under a cardinality constraint (pick at most k elements to maximize f). This is where the greedy algorithm truly shines.

Watch out: If the function is non-monotone, the plain greedy may perform very poorly. You need more sophisticated techniques (double-greedy, random sampling) that are beyond what most CP problems require.

Greedy Algorithm & the (1 − 1/e) Guarantee

The crown jewel of submodular optimization is the following theorem by Nemhauser, Wolsey, and Fisher (1978):

Theorem. For a non-negative, monotone submodular function f with f(∅) = 0, the greedy algorithm that iteratively picks the element with the largest marginal gain achieves:

f(Sgreedy)  ≥  (1 − 1/e) · f(SOPT)

where |Sgreedy| = |SOPT| = k.

The Greedy Algorithm

The algorithm is remarkably simple:

  1. Start with S = ∅.
  2. For i = 1 to k:
    • Pick x* = argmaxx ∈ V \ S [f(S ∪ {x}) − f(S)]
    • Set S ← S ∪ {x*}
  3. Return S.

Proof Sketch

Let OPT = {o₁, o₂, …, oₖ} be the optimal solution, and let Sᵢ denote the greedy solution after i steps. Define δᵢ = f(Sᵢ) − f(Sᵢ₋₁) as the marginal gain at step i.

Step 1: By submodularity and the greedy choice:

For each oⱼ ∈ OPT, we have f(Sᵢ₋₁ ∪ {oⱼ}) − f(Sᵢ₋₁) ≤ δᵢ (greedy picked the best).

Step 2: By submodularity and telescoping over all elements of OPT:

f(OPT ∪ Sᵢ₋₁) − f(Sᵢ₋₁) ≤ Σⱼ [f(Sᵢ₋₁ ∪ {oⱼ}) − f(Sᵢ₋₁)] ≤ k · δᵢ

Step 3: Since f is monotone, f(OPT) ≤ f(OPT ∪ Sᵢ₋₁), so:

f(OPT) − f(Sᵢ₋₁) ≤ k · δᵢ

Step 4: Let Δᵢ = f(OPT) − f(Sᵢ). Then:

Δᵢ = Δᵢ₋₁ − δᵢ ≤ Δᵢ₋₁ − Δᵢ₋₁/k = Δᵢ₋₁ · (1 − 1/k)

Step 5: Unrolling the recurrence:

Δₖ ≤ Δ₀ · (1 − 1/k)ᵏ < f(OPT) · e⁻¹

Therefore f(Sₖ) = f(OPT) − Δₖ > f(OPT) · (1 − 1/e). ∎

Tightness: This bound is tight. The maximum coverage problem is (1 − 1/e + ε)-hard to approximate for any ε > 0 under standard complexity assumptions, meaning the greedy algorithm is essentially the best we can do in polynomial time.

Lazy Evaluations (Accelerated Greedy)

A practical optimization: since marginal gains are non-increasing (by submodularity), we can use a priority queue and re-evaluate elements lazily. If the top element's marginal gain hasn't changed since its last evaluation, it must still be the best. This is called the lazy greedy or accelerated greedy algorithm, and it often runs orders of magnitude faster in practice while giving the exact same result.

Implementation

Generic Greedy Submodular Maximization

Here is a clean C++ implementation of the greedy algorithm for monotone submodular maximization under a cardinality constraint. The function f is passed as a callable that evaluates the objective on a bitmask. 

#include <bits/stdc++.h>
using namespace std;

// Greedy submodular maximization under cardinality constraint k.
// f(mask) returns the value of the submodular function on the subset
// encoded by the bitmask. n = ground set size.
pair<int, double> greedySubmodular(int n, int k,
                                     function<double(int)> f) {
    int chosen = 0;
    for (int step = 0; step < k; step++) {
        int bestElem = -1;
        double bestGain = -1e18;
        double curVal = f(chosen);
        for (int i = 0; i < n; i++) {
            if (chosen >> i & 1) continue;
            double gain = f(chosen | (1 << i)) - curVal;
            if (gain > bestGain) {
                bestGain = gain;
                bestElem = i;
            }
        }
        if (bestElem == -1) break;
        chosen |= (1 << bestElem);
    }
    return {chosen, f(chosen)};
}
Bitmask limitation: This version uses bitmasks, so it only works for n ≤ 20 or so. For larger ground sets, use a set or vector<bool> to represent the chosen subset.

Maximum Coverage (Scalable Version)

The most common CP application: given n sets over a universe, pick k sets to maximize the number of covered elements. This avoids bitmasks entirely. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m, k;
    cin >> n >> m >> k;  // n sets, m universe size, k to pick

    vector<vector<int>> sets(n);
    for (int i = 0; i < n; i++) {
        int sz; cin >> sz;
        sets[i].resize(sz);
        for (int &x : sets[i]) cin >> x;
    }

    vector<bool> covered(m + 1, false);
    vector<bool> used(n, false);
    int totalCovered = 0;
    vector<int> chosen;

    for (int step = 0; step < k; step++) {
        int bestSet = -1, bestGain = 0;
        for (int i = 0; i < n; i++) {
            if (used[i]) continue;
            int gain = 0;
            for (int x : sets[i])
                if (!covered[x]) gain++;
            if (gain > bestGain) {
                bestGain = gain;
                bestSet = i;
            }
        }
        if (bestSet == -1) break;
        used[bestSet] = true;
        chosen.push_back(bestSet);
        for (int x : sets[bestSet]) {
            if (!covered[x]) {
                covered[x] = true;
                totalCovered++;
            }
        }
    }

    cout << totalCovered << "\n";
    for (int s : chosen) cout << s + 1 << " ";
    cout << "\n";
    return 0;
}

Time complexity: O(k · n · max_set_size). For each of the k steps, we evaluate the marginal gain of every unused set, which takes time proportional to the set size.

Lazy Greedy with Priority Queue

For large inputs, the lazy greedy optimization can give dramatic speedups. We maintain a max-heap of (marginal_gain, set_index) pairs and only re-evaluate when an element reaches the top of the heap. 

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m, k;
    cin >> n >> m >> k;

    vector<vector<int>> sets(n);
    for (int i = 0; i < n; i++) {
        int sz; cin >> sz;
        sets[i].resize(sz);
        for (int &x : sets[i]) cin >> x;
    }

    vector<bool> covered(m + 1, false);
    // (gain, index) — gains are upper bounds until validated
    priority_queue<pair<int,int>> pq;
    for (int i = 0; i < n; i++)
        pq.push({(int)sets[i].size(), i});

    vector<int> lastEval(n, -1);  // step at which gain was computed
    int totalCovered = 0;

    for (int step = 0; step < k; step++) {
        while (true) {
            auto [gain, idx] = pq.top();
            pq.pop();
            if (lastEval[idx] == step) {
                // gain is current — pick this set
                for (int x : sets[idx]) {
                    if (!covered[x]) {
                        covered[x] = true;
                        totalCovered++;
                    }
                }
                break;
            }
            // recompute gain
            int newGain = 0;
            for (int x : sets[idx])
                if (!covered[x]) newGain++;
            lastEval[idx] = step;
            pq.push({newGain, idx});
        }
    }

    cout << totalCovered << "\n";
    return 0;
}
Why this works: By submodularity, the marginal gain of any element can only decrease as we add more elements to our solution. So the gain stored in the heap is always an upper bound on the true current gain. If the top element's gain is still valid (computed in the current step), it must be the true maximum.

Submodular Minimization

While submodular maximization is NP-hard (and we settle for approximation algorithms), submodular minimization is a completely different story:

Theorem (Grötschel, Lovász, Schrijver 1981): Unconstrained submodular function minimization can be solved in polynomial time. The first combinatorial strongly-polynomial algorithms were given independently by Iwata, Fleischer, and Fujishige (2001) and Schrijver (2000).

This is one of the most remarkable results in combinatorial optimization. The key idea is that the Lovász extension of a submodular function is convex, so minimizing it reduces to convex optimization.

Connection to Min-Cut

The most important special case: the graph cut function f(S) = |δ(S)| (number of edges crossing the cut) is submodular. Minimizing it over non-trivial partitions gives the minimum cut, which we know can be solved efficiently via max-flow.

More generally, any symmetric submodular function minimization can often be reduced to a sequence of minimum s-t cut computations. This is the basis of the Queyranne (1998) algorithm for symmetric submodular minimization without a flow network.

When Does This Help in CP?

Directly applying general submodular minimization algorithms in a contest is rare — they are complex to implement and the constants are large. However, recognizing that a problem's objective is a submodular function to be minimized tells you:

  • A polynomial-time solution exists (don't search for NP-hardness reductions).
  • It may reduce to min-cut / max-flow.
  • For graph-based objectives, model it as a flow problem.
// Minimum cut via Dinic's max-flow — the most common
// "submodular minimization" in CP.
#include <bits/stdc++.h>
using namespace std;

struct Edge { int to, rev; long long cap; };

struct Dinic {
    vector<vector<Edge>> g;
    vector<int> level, iter;
    Dinic(int n) : g(n), level(n), iter(n) {}

    void addEdge(int u, int v, long long c) {
        g[u].push_back({v, (int)g[v].size(), c});
        g[v].push_back({u, (int)g[u].size()-1, 0});
    }
    bool bfs(int s, int t) {
        fill(level.begin(), level.end(), -1);
        queue<int> q; level[s]=0; q.push(s);
        while (!q.empty()) {
            int v=q.front(); q.pop();
            for (auto &e:g[v]) if (e.cap>0 && level[e.to]<0)
                level[e.to]=level[v]+1, q.push(e.to);
        }
        return level[t]>=0;
    }
    long long dfs(int v, int t, long long f) {
        if (v==t) return f;
        for (int &i=iter[v]; i<(int)g[v].size(); i++) {
            Edge &e=g[v][i];
            if (e.cap>0 && level[v]<level[e.to]) {
                long long d=dfs(e.to,t,min(f,e.cap));
                if (d>0) { e.cap-=d; g[e.to][e.rev].cap+=d; return d; }
            }
        }
        return 0;
    }
    long long maxflow(int s, int t) {
        long long flow=0;
        while (bfs(s,t)) {
            fill(iter.begin(),iter.end(),0);
            long long d;
            while ((d=dfs(s,t,LLONG_MAX))>0) flow+=d;
        }
        return flow;
    }
};

Applications in CP

1. Maximum Coverage

Problem: Given n sets, choose at most k to maximize the number of distinct elements covered.

This is the textbook application. The coverage function is monotone submodular, so greedy gives a (1 − 1/e)-approximation. In many contest variants, the greedy solution is actually optimal due to special structure (e.g., laminar families).

2. Influence Maximization

Problem: In a social network, pick k seed nodes to maximize the expected number of nodes reached by a diffusion process. The expected influence under the Independent Cascade model is monotone submodular (Kempe, Kleinberg, Tardos 2003), so greedy gives a (1 − 1/e)-approximation. 

// Greedy influence maximization with Monte Carlo estimation
#include <bits/stdc++.h>
using namespace std;
mt19937 rng(42);

double estimateInfluence(int n,
    const vector<vector<pair<int,double>>> &adj,
    const vector<int> &seeds, int T) {
    double total = 0;
    for (int t = 0; t < T; t++) {
        vector<bool> active(n, false);
        queue<int> q;
        for (int s : seeds) { active[s]=true; q.push(s); }
        int cnt = seeds.size();
        while (!q.empty()) {
            int u=q.front(); q.pop();
            for (auto [v,p] : adj[u])
                if (!active[v] && uniform_real_distribution<>(0,1)(rng)<p)
                    { active[v]=true; cnt++; q.push(v); }
        }
        total += cnt;
    }
    return total / T;
}

3. Facility Location

Problem: Choose k facility locations from n candidates to minimize the maximum (or total) distance from any client to its nearest facility.

The "utility" version — maximize the sum of how much closer clients get — is monotone submodular. Greedy gives a (1 − 1/e)-approximation. 

// Greedy facility location (maximize total "closeness" utility)
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n, m, k;
    cin >> n >> m >> k;  // n facilities, m clients, k to open

    // dist[i][j] = distance from facility i to client j
    vector<vector<double>> dist(n, vector<double>(m));
    for (int i = 0; i < n; i++)
        for (int j = 0; j < m; j++)
            cin >> dist[i][j];

    double DMAX = 1e18;
    vector<double> nearest(m, DMAX);  // nearest facility distance for each client
    vector<bool> opened(n, false);
    double totalUtility = 0;

    for (int step = 0; step < k; step++) {
        int bestFac = -1;
        double bestGain = -1;
        for (int i = 0; i < n; i++) {
            if (opened[i]) continue;
            double gain = 0;
            for (int j = 0; j < m; j++) {
                double improvement = max(0.0, nearest[j] - dist[i][j]);
                gain += improvement;
            }
            if (gain > bestGain) { bestGain = gain; bestFac = i; }
        }
        if (bestFac == -1) break;
        opened[bestFac] = true;
        for (int j = 0; j < m; j++)
            nearest[j] = min(nearest[j], dist[bestFac][j]);
        totalUtility += bestGain;
    }

    cout << fixed << setprecision(6) << totalUtility << "\n";
    return 0;
}

4. Sensor Placement / Information Gain

Choose k sensor locations to maximize information gathered about an environment. The mutual information function is monotone submodular, making greedy optimal up to (1 − 1/e). Contest problems sometimes dress this up as "choose observation points."

Pattern recognition tip: When a contest problem asks you to "select k items to maximize some aggregate benefit" and you suspect diminishing returns, try the greedy algorithm. Even if you can't formally prove submodularity during the contest, the greedy is often the intended solution.

Tips for Contest Day

  • Recognize the structure: If the problem says "pick k things to maximize coverage/benefit" and you can argue diminishing returns, go greedy.
  • Check monotonicity: If elements can hurt the objective, you don't have monotone submodularity and plain greedy may fail.
  • Exact vs approximate: Some contest problems want an exact answer, not an approximation. In that case, submodularity might guide your DP formulation but the greedy alone won't suffice.
  • Small n: If n ≤ 20, just enumerate all 2n subsets. Submodular tricks are for when n is too large for brute force.
  • Min-cut connection: If the objective looks like a cut function, model it as a flow problem. This is the most common "submodular minimization" in CP.
Summary: Submodular optimization gives you a powerful lens for a class of problems where greedy works provably well. The (1 − 1/e) guarantee for monotone submodular maximization is one of the most beautiful results in combinatorial optimization, and it appears — often in disguise — across competitive programming, algorithms research, and machine learning.
