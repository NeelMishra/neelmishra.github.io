    Beam Search for Optimization

    July 2025 · 28 min read

    What Is Beam Search?

    Beam search is a heuristic search algorithm that explores a graph by expanding only the most promising nodes at each level. Think of it as BFS with a width limit: instead of keeping every node at the current depth, you keep only the top B candidates, where B is the beam width.

    This places beam search on a spectrum between two extremes:

    • Greedy search (beam width = 1) — always picks the single best candidate. Fast but myopic; easily trapped in local optima.
    • Exhaustive BFS (beam width = ∞) — explores every possibility. Optimal but exponentially expensive.

    By tuning the beam width, you control the trade-off between solution quality and computational cost. In competitive programming marathon/optimization contests, beam search is one of the most popular and effective techniques because it systematically explores many states while staying within tight time limits.

    Key insight: Beam search does not guarantee an optimal solution. It is a pruning heuristic — it bets that the globally best solution passes through locally promising intermediate states. This bet pays off surprisingly often in practice.

    The high-level idea is straightforward:

    1. Start with an initial state (or a set of initial states).
    2. At each step, generate all successor states from the current beam.
    3. Evaluate each successor with a scoring function.
    4. Keep only the top B successors — discard the rest.
    5. Repeat until a termination condition is met (depth limit, time limit, or no improvement).

    The scoring function (also called the evaluation function or heuristic) is what makes or breaks beam search. A good evaluation function ranks partial solutions so that high-scoring partial solutions tend to lead to high-scoring complete solutions.

    Algorithm and Implementation

    Let’s implement a generic beam search framework in C++. The core data structure is simple: at each depth level, maintain a vector of states sorted by their evaluation score, truncated to the beam width.

    State Representation

    A beam search state typically contains:

    • The current partial solution (e.g., a sequence of decisions made so far).
    • Enough bookkeeping to generate successors efficiently.
    • A score for ranking against other states in the beam.
    struct State {
    vector<int> decisions;   // sequence of choices made
    double score;             // evaluation score (higher = better)
    // ... problem-specific fields

    bool operator<(const State& o) const {
        return score < o.score;  // for max-heap / sorting
    }
};

    Core Beam Search

    The algorithm iterates through depth levels, expanding the beam at each step:

    #include <bits/stdc++.h>
using namespace std;

struct State {
    vector<int> decisions;
    double score;
    // problem-specific data
    int current_pos;

    bool operator<(const State& o) const {
        return score < o.score;
    }
};

vector<State> generate_successors(const State& s) {
    vector<State> next;
    // enumerate all possible next decisions from state s
    for (int choice : get_valid_choices(s)) {
        State ns = s;
        ns.decisions.push_back(choice);
        ns.current_pos = apply_choice(s, choice);
        ns.score = evaluate(ns);
        next.push_back(ns);
    }
    return next;
}

State beam_search(State initial, int beam_width, int max_depth) {
    vector<State> beam = {initial};
    State best = initial;

    for (int depth = 0; depth < max_depth; ++depth) {
        vector<State> candidates;
        for (const auto& s : beam) {
            auto successors = generate_successors(s);
            for (auto& ns : successors) {
                candidates.push_back(move(ns));
            }
        }

        if (candidates.empty()) break;

        // sort descending by score, keep top beam_width
        sort(candidates.begin(), candidates.end(),
             [](const State& a, const State& b) {
                 return a.score > b.score;
             });

        if (candidates.size() > (size_t)beam_width) {
            candidates.resize(beam_width);
        }

        beam = move(candidates);

        // track overall best
        if (beam[0].score > best.score) {
            best = beam[0];
        }
    }

    return best;
}

    Choosing an Initial Beam Width

    The right beam width depends on two factors:

    • Branching factor: If each state produces k successors, then at each level you generate up to B × k candidates. The sorting step is O(Bk log Bk).
    • Time budget: In a contest with a 2-second time limit and D depth levels, the beam width must satisfy B × k × D × (cost per state) ≤ time budget.
    Rule of thumb for contests: Start with B = 1000, measure your runtime, and binary-search for the largest B that fits within the time limit. Many top-rated contestants use this exact approach.

    Beam Search vs Other Methods

    Understanding where beam search fits relative to other search strategies helps you decide when to reach for it.

    Method Space Completeness Optimality Typical Use
    BFS Exponential Yes Yes (uniform cost) Short paths, small state spaces
    DFS Linear No (in infinite graphs) No Exhaustive enumeration, backtracking
    A* Exponential Yes Yes (admissible heuristic) Shortest path with good heuristics
    Greedy Constant No No Simple, fast approximation
    Beam Search O(B) No No Large state spaces, optimization contests

    Beam search vs Greedy: Greedy commits to a single path and never reconsiders. Beam search maintains B parallel paths, which provides natural diversity. Even a beam width of 5–10 can dramatically outperform greedy.

    Beam search vs A*: A* is optimal but needs to store the entire open set. When the state space is enormous (as in marathon contests), A* runs out of memory long before finding a solution. Beam search bounds memory at O(B).

    Beam search vs Simulated Annealing: SA works well when the search space has a single connected component and moves are reversible. Beam search is preferred when the problem has a natural sequential/layered structure — placing items one at a time, assigning tasks step by step, building a solution incrementally.

    Beam search vs Genetic Algorithms: Both maintain a population. But beam search has a structured depth progression, while GAs evolve a population stochastically. Beam search is simpler to implement and debug, and often faster to converge for sequential decision problems.

    Beam Search in CP Contests

    Beam search is a workhorse technique in marathon-style and optimization competitive programming contests. Platforms like AtCoder Heuristic Contest (AHC), Google Hash Code, CodinGame, and Topcoder Marathon frequently feature problems where beam search is a top strategy.

    When to Use Beam Search

    Beam search shines when the problem has these characteristics:

    • Sequential decisions: The solution is built step by step (place blocks, assign jobs, route vehicles).
    • Good intermediate evaluation: You can estimate how good a partial solution is.
    • Large branching factor: Too many choices for exhaustive search, but many are clearly bad and can be pruned by the beam.
    • Fixed depth: The number of steps is known or bounded (e.g., N items to place on a grid).

    AtCoder Heuristic Contest Strategies

    Many AHC problems are perfectly suited for beam search. Here is a pattern that frequently appears in top solutions:

    #include <bits/stdc++.h>
using namespace std;

struct Timer {
    chrono::high_resolution_clock::time_point start;
    Timer() : start(chrono::high_resolution_clock::now()) {}
    double elapsed() const {
        auto now = chrono::high_resolution_clock::now();
        return chrono::duration<double>(now - start).count();
    }
};

// Adaptive beam search: expand beam width to fill the time limit
State adaptive_beam_search(State initial, double time_limit) {
    Timer timer;

    // Phase 1: estimate cost per state with a small beam
    int probe_width = 100;
    // ... run a few steps, measure time per expansion

    // Phase 2: compute largest feasible beam width
    double time_per_state = /* measured */;
    int max_depth = /* problem-specific */;
    int beam_width = (int)(time_limit * 0.9 / (time_per_state * max_depth));
    beam_width = max(beam_width, 1);

    // Phase 3: run full beam search
    vector<State> beam = {initial};
    State best = initial;

    for (int depth = 0; depth < max_depth; ++depth) {
        if (timer.elapsed() > time_limit * 0.95) break;

        vector<State> candidates;
        for (const auto& s : beam) {
            for (auto& ns : generate_successors(s)) {
                candidates.push_back(move(ns));
            }
        }

        if (candidates.empty()) break;

        // partial_sort is faster than full sort when beam_width << candidates.size()
        int keep = min((int)candidates.size(), beam_width);
        partial_sort(candidates.begin(), candidates.begin() + keep,
                     candidates.end(),
                     [](const State& a, const State& b) {
                         return a.score > b.score;
                     });
        candidates.resize(keep);
        beam = move(candidates);

        if (beam[0].score > best.score) {
            best = beam[0];
        }
    }

    return best;
}
    Contest tip: Use partial_sort instead of sort when beam_width is much smaller than the candidate count. partial_sort runs in O(n log B) vs O(n log n), which can save significant time when the branching factor is large.

    Hash-Based Deduplication

    A common pitfall is having many nearly identical states in the beam. This wastes capacity. Use Zobrist hashing to detect and remove duplicates:

    // Deduplicate beam using hash
void deduplicate_beam(vector<State>& beam) {
    unordered_set<uint64_t> seen;
    vector<State> unique_beam;
    for (auto& s : beam) {
        if (seen.insert(s.hash).second) {
            unique_beam.push_back(move(s));
        }
    }
    beam = move(unique_beam);
}

    Stochastic Beam Search

    Standard beam search is deterministic: given the same input and beam width, it always produces the same result. This means it can get stuck in the same local optimum every time. Stochastic beam search adds randomness to escape this trap.

    Sampling from Top-k Candidates

    Instead of keeping the top B candidates by score, stochastic beam search samples B candidates with probability proportional to their scores. Higher-scoring states are more likely to survive, but lower-scoring states have a chance too.

    // Stochastic beam selection using softmax-style sampling
vector<State> stochastic_select(vector<State>& candidates, int beam_width,
                                  double temperature, mt19937& rng) {
    if ((int)candidates.size() <= beam_width) return candidates;

    // compute selection probabilities via softmax
    double max_score = candidates[0].score;
    for (auto& c : candidates) max_score = max(max_score, c.score);

    vector<double> weights(candidates.size());
    double total = 0.0;
    for (int i = 0; i < (int)candidates.size(); ++i) {
        weights[i] = exp((candidates[i].score - max_score) / temperature);
        total += weights[i];
    }

    // sample without replacement
    vector<State> selected;
    vector<bool> used(candidates.size(), false);
    for (int i = 0; i < beam_width; ++i) {
        double r = uniform_real_distribution<double>(0.0, total)(rng);
        double cumulative = 0.0;
        for (int j = 0; j < (int)candidates.size(); ++j) {
            if (used[j]) continue;
            cumulative += weights[j];
            if (cumulative >= r) {
                selected.push_back(move(candidates[j]));
                total -= weights[j];
                used[j] = true;
                break;
            }
        }
    }

    return selected;
}

    Temperature Control

    The temperature parameter controls how random the selection is:

    • Low temperature (e.g., 0.01) → nearly deterministic, almost identical to standard beam search.
    • High temperature (e.g., 10.0) → nearly uniform random selection, ignores scores.
    • Moderate temperature (e.g., 0.5–2.0) → a good middle ground that prefers high scores while allowing diversity.

    A common strategy is annealing the temperature: start with a high temperature (more exploration) and gradually decrease it (more exploitation) as the search progresses through depth levels.

    for (int depth = 0; depth < max_depth; ++depth) {
    double progress = (double)depth / max_depth;
    double temperature = T_start * pow(T_end / T_start, progress);

    // generate candidates ...
    beam = stochastic_select(candidates, beam_width, temperature, rng);
}
    Hybrid approach: Reserve a fraction of the beam (e.g., top 20%) for deterministic selection (always keep the best states), and fill the rest stochastically. This guarantees you never lose your best partial solutions while still maintaining diversity.

    Example: Solving a Grid Optimization Problem

    Let’s work through a concrete problem from start to finish. This is a simplified version of problems that frequently appear in AHC.

    Problem Statement

    You have an N × N grid. Each cell has a value v[i][j]. You start at (0, 0) and must reach (N−1, N−1) in exactly 2(N−1) steps, moving only right or down. Maximize the sum of values of visited cells. Additionally, visiting two adjacent cells with the same value gives a bonus of +10.

    With greedy, you’d just pick the better of right/down at each step. But a look-ahead is valuable because accepting a slightly worse cell now might unlock a much better path later.

    Full C++ Solution

    #include <bits/stdc++.h>
using namespace std;

const int MAXN = 50;
int N;
int grid[MAXN][MAXN];

struct State {
    int r, c;
    int total_score;
    int last_val;
    uint64_t hash;
    vector<char> path;  // 'R' or 'D'

    double eval_score() const {
        // base score + look-ahead heuristic
        double base = total_score;
        // heuristic: average of remaining cells on shortest path to goal
        int remaining = (N - 1 - r) + (N - 1 - c);
        if (remaining == 0) return base;

        // estimate future value: diagonal average
        double future = 0;
        int steps = 0;
        int cr = r, cc = c;
        while (cr < N - 1 || cc < N - 1) {
            if (cr < N - 1 && cc < N - 1) {
                future += max(grid[cr + 1][cc], grid[cr][cc + 1]);
                // move toward the better one
                if (grid[cr + 1][cc] >= grid[cr][cc + 1]) cr++;
                else cc++;
            } else if (cr < N - 1) {
                future += grid[++cr][cc];
            } else {
                future += grid[cr][++cc];
            }
            steps++;
        }

        return base + future * 0.5;  // discount future estimate
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    cin >> N;
    for (int i = 0; i < N; i++)
        for (int j = 0; j < N; j++)
            cin >> grid[i][j];

    int BEAM_WIDTH = 2000;
    int max_depth = 2 * (N - 1);

    State initial;
    initial.r = initial.c = 0;
    initial.total_score = grid[0][0];
    initial.last_val = grid[0][0];
    initial.hash = 0;

    vector<State> beam = {initial};
    State best = initial;

    for (int depth = 0; depth < max_depth; ++depth) {
        vector<pair<double, int>> scored;  // (eval, index in candidates)
        vector<State> candidates;

        for (const auto& s : beam) {
            // try moving right
            if (s.c + 1 < N) {
                State ns = s;
                ns.c++;
                int val = grid[ns.r][ns.c];
                ns.total_score += val;
                if (val == ns.last_val) ns.total_score += 10; // adjacency bonus
                ns.last_val = val;
                ns.path.push_back('R');
                int idx = (int)candidates.size();
                scored.push_back({ns.eval_score(), idx});
                candidates.push_back(move(ns));
            }
            // try moving down
            if (s.r + 1 < N) {
                State ns = s;
                ns.r++;
                int val = grid[ns.r][ns.c];
                ns.total_score += val;
                if (val == ns.last_val) ns.total_score += 10;
                ns.last_val = val;
                ns.path.push_back('D');
                int idx = (int)candidates.size();
                scored.push_back({ns.eval_score(), idx});
                candidates.push_back(move(ns));
            }
        }

        if (scored.empty()) break;

        // keep top BEAM_WIDTH
        int keep = min((int)scored.size(), BEAM_WIDTH);
        partial_sort(scored.begin(), scored.begin() + keep,
                     scored.end(),
                     [](const auto& a, const auto& b) {
                         return a.first > b.first;
                     });

        vector<State> new_beam;
        new_beam.reserve(keep);
        for (int i = 0; i < keep; ++i) {
            new_beam.push_back(move(candidates[scored[i].second]));
        }

        beam = move(new_beam);

        // update best based on actual score (not eval)
        for (const auto& s : beam) {
            if (s.total_score > best.total_score) {
                best = s;
            }
        }
    }

    // output result
    cout << best.total_score << "\n";
    for (char c : best.path) cout << c;
    cout << "\n";

    return 0;
}

    Key implementation details to note:

    • The eval_score() function combines the actual score so far with a greedy look-ahead estimate of future value. This is crucial — without the look-ahead, beam search degenerates toward greedy.
    • We use partial_sort since we only need the top B elements, not a fully sorted list.
    • We track the total_score (actual) separately from eval_score() (heuristic). The final answer uses the actual score.
    • We store the path to reconstruct the solution at the end.

    Tuning Beam Width

    The beam width is the single most important hyperparameter. Too narrow and you miss good solutions; too wide and you waste time on mediocre states or run out of time entirely.

    Time-Based Tuning

    The most robust approach in contests: measure how long one iteration takes and compute the maximum beam width.

    // Measure cost and auto-tune beam width
int auto_tune_beam_width(State initial, int max_depth, double time_limit) {
    Timer timer;

    // warm-up: run depth 0 with beam_width=1, measure time
    vector<State> beam = {initial};
    auto successors = generate_successors(beam[0]);
    double t1 = timer.elapsed();

    // cost per candidate ≈ t1 / successors.size()
    double cost_per_candidate = t1 / max(1, (int)successors.size());

    // branching factor estimate
    int branch_factor = (int)successors.size();

    // total work ≈ beam_width * branch_factor * max_depth * cost_per_candidate
    // solve for beam_width:
    double budget = time_limit * 0.85;  // leave 15% margin
    int beam_width = (int)(budget / (branch_factor * max_depth * cost_per_candidate));
    beam_width = max(beam_width, 1);
    beam_width = min(beam_width, 100000);  // cap to avoid memory issues

    return beam_width;
}

    Quality vs Speed Trade-offs

    In practice, the relationship between beam width and solution quality follows a curve of diminishing returns:

    • B = 1–10: Rapid improvement over greedy. Every additional beam slot contributes meaningfully.
    • B = 100–1000: Steady improvement. This is the sweet spot for most contest problems.
    • B = 10,000+: Marginal gains. You’re spending most of the time on states that are very similar to each other.

    Multi-Start vs Wide Beam

    Given a fixed time budget, you have a choice: run one beam search with a large width, or run multiple beam searches with smaller widths (possibly with different random seeds for stochastic selection). The latter is called multi-start beam search.

    State multi_start_beam(State initial, int num_runs, double total_time) {
    double time_per_run = total_time / num_runs;
    State global_best = initial;
    mt19937 rng(42);

    for (int run = 0; run < num_runs; ++run) {
        // vary the initial state or use stochastic selection
        State result = beam_search_with_stochastic(
            initial,
            /* beam_width */ 500,
            /* max_depth */ max_depth,
            /* temperature */ 1.0,
            /* time_limit */ time_per_run,
            rng
        );

        if (result.total_score > global_best.total_score) {
            global_best = result;
        }
    }

    return global_best;
}
    Empirical guideline: For problems where the evaluation function is strong (well-correlated with final score), prefer a single wide beam. For problems where the evaluation function is noisy, multi-start with stochastic selection tends to perform better.

    Memory Optimization

    Storing full state objects for every candidate can be expensive. Common tricks to reduce memory usage:

    • Store only deltas: Instead of copying the full state, store a parent pointer and the decision that generated the child. Reconstruct the full solution at the end by following parent pointers.
    • Pool allocators: Pre-allocate a fixed-size array and reuse slots across depth levels.
    • Compact state representation: Use bitsets or packed integers instead of vectors where possible.
    // Memory-efficient beam search with parent pointers
struct CompactState {
    int parent_idx;   // index in previous beam (-1 for root)
    int decision;     // what choice was made
    int score;
    // minimal problem-specific data
    int r, c;
};

// After beam search completes, reconstruct solution:
vector<int> reconstruct(vector<vector<CompactState>>& beams, int depth, int idx) {
    vector<int> path;
    for (int d = depth; d >= 1; --d) {
        path.push_back(beams[d][idx].decision);
        idx = beams[d][idx].parent_idx;
    }
    reverse(path.begin(), path.end());
    return path;
}

    Practical Tips and Common Pitfalls

    Evaluation Function Design

    The evaluation function is the most critical component. A few guidelines:

    • Combine actual score with estimated future score. Pure actual-score ranking is too greedy; pure estimated-future is too speculative.
    • Normalize components. If your score has multiple factors (e.g., total value + penalty for distance), ensure they’re on the same scale.
    • Test your evaluation function. Run beam search with a very large beam width and check if the top-ranked state at the end is actually the best solution. If not, your evaluation function has a bias.

    Common Mistakes

    • Forgetting to deduplicate: Many states in the beam may represent the same underlying solution reached via different orderings. This wastes beam capacity.
    • Copying large states: Use move semantics aggressively. State copying is often the bottleneck.
    • Not using partial_sort: When B << total candidates, partial_sort or nth_element + sort can be 2–5x faster than full sort.
    • Ignoring time limits: Always check elapsed time inside the main loop. Beam search with a fixed width may exceed the time limit on large inputs.
    // nth_element + sort: fastest when B is much smaller than candidates
int keep = min((int)candidates.size(), beam_width);
nth_element(candidates.begin(), candidates.begin() + keep,
            candidates.end(),
            [](const State& a, const State& b) {
                return a.score > b.score;
            });
candidates.resize(keep);
// optionally sort the kept elements if order matters
sort(candidates.begin(), candidates.end(),
     [](const State& a, const State& b) {
         return a.score > b.score;
     });

    Summary

    Beam search is a flexible and powerful heuristic search technique. Its strength lies in the balance it strikes between greedy speed and exhaustive thoroughness. To use it effectively:

    1. Design a strong evaluation function that estimates the quality of partial solutions.
    2. Auto-tune the beam width to fill the available time budget.
    3. Use partial_sort or nth_element to avoid sorting overhead.
    4. Consider stochastic variants or multi-start for problems with noisy evaluation.
    5. Optimize memory by storing deltas instead of full states.
    6. Always track and return the best solution seen across all depth levels.
