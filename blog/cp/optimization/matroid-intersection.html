    Matroid Intersection Algorithm

    July 2025 · 25 min read

    Many combinatorial optimization problems that appear intractable at first glance can be solved in polynomial time when the underlying structure turns out to be a matroid. The matroid intersection algorithm takes this a step further: given two matroids on the same ground set, it finds a maximum-cardinality (or maximum-weight) set that is independent in both matroids simultaneously. This single framework unifies bipartite matching, colorful spanning trees, arborescences, and a surprising number of contest problems.

    What Is a Matroid?

    A matroid is a pair M = (E, I) where E is a finite ground set and I ⊆ 2E is a family of independent sets satisfying three axioms:

    1. (I1) Non-emptiness: The empty set is independent: ∅ ∈ I.
    2. (I2) Hereditary property: If A ∈ I and B ⊆ A, then B ∈ I. Every subset of an independent set is independent.
    3. (I3) Exchange property: If A, B ∈ I and |A| < |B|, then there exists e ∈ B \ A such that A ∪ {e} ∈ I.

    Axiom (I3) is the powerful one — it guarantees that the greedy algorithm works for maximizing a linear objective over I, and it ensures all maximal independent sets (called bases) have the same cardinality, called the rank of the matroid.

    Classic Examples of Matroids

    Graphic (Cycle) Matroid. Given a graph G = (V, E), the ground set is E and a set of edges is independent if and only if it contains no cycle (i.e., it forms a forest). The rank equals |V| − (number of connected components). Kruskal's MST algorithm is just the greedy algorithm on a graphic matroid.

    Partition Matroid. The ground set E is partitioned into groups E₁, E₂, …, Eₖ with capacities d₁, d₂, …, dₖ. A set S is independent if |S ∩ Eᵢ| ≤ dᵢ for every i. This is the matroid behind many assignment and scheduling constraints.

    Transversal Matroid. Given a bipartite graph with left vertices L and right vertices R, the ground set is L. A set S ⊆ L is independent if there exists a matching in the bipartite graph that saturates every vertex in S. The rank equals the size of a maximum matching.

    Uniform Matroid Uk,n. The ground set has n elements and a set is independent if its size is at most k. This is the simplest non-trivial matroid. All sets of size ≤ k are independent.

    Linear (Representable) Matroid. Given a matrix A over some field, the ground set is the set of columns and a set of columns is independent if they are linearly independent. Many matroids encountered in CP are linear matroids over GF(2) or ℤₚ.

    Key insight: To use the matroid intersection algorithm, you must be able to implement an independence oracle for each matroid — a function that, given a set S, returns whether S ∈ I. The algorithm's efficiency depends heavily on how fast this oracle runs.

    The Matroid Intersection Problem

    Given two matroids M₁ = (E, I₁) and M₂ = (E, I₂) on the same ground set E, the matroid intersection problem asks for a set S of maximum cardinality such that S ∈ I₁ and S ∈ I₂ simultaneously.

    This is a remarkably general framework. Here are some classical problems that reduce to matroid intersection:

    • Bipartite matching — Intersect two partition matroids. Partition the edges by left endpoint (matroid M₁: at most 1 edge per left vertex) and by right endpoint (matroid M₂: at most 1 edge per right vertex).
    • Colorful spanning tree — Given a graph where each edge has a color, find a spanning tree using at most one edge of each color. Intersect the graphic matroid with a partition matroid on colors.
    • Spanning arborescence with degree constraints — Various arborescence problems can be formulated as intersection of a graphic matroid with a partition or transversal matroid.
    Important: While intersection of two matroids is solvable in polynomial time, intersection of three or more matroids is NP-hard in general. Always check that your problem decomposes into exactly two matroid structures.

    The maximum cardinality of a common independent set satisfies a min-max theorem analogous to König's theorem for bipartite matching:

    Matroid Intersection Theorem (Edmonds, 1970): max{|S| : S ∈ I₁ ∩ I₂} = min{r₁(U) + r₂(E \ U) : U ⊆ E}.

    The Augmenting Path Algorithm

    The standard algorithm for matroid intersection is based on augmenting paths in an exchange graph, analogous to how augmenting paths solve bipartite matching. The algorithm was developed by Lawler (1975) and later refined by several authors.

    Exchange Graph Construction

    Given a current common independent set J (J ∈ I₁ ∩ I₂), we construct a directed bipartite graph DJ = (E, A) as follows:

    • The vertex set is all elements of E.
    • For each y ∈ J and x ∈ E \ J: add arc (y → x) if (J \ {y}) ∪ {x} ∈ I₁ (i.e., swapping y for x keeps independence in M₁).
    • For each x ∈ E \ J and y ∈ J: add arc (x → y) if (J \ {y}) ∪ {x} ∈ I₂ (swapping y for x keeps independence in M₂).

    Additionally, define two special sets:

    • X₁ = {x ∈ E \ J : J ∪ {x} ∈ I₁} — elements that can be freely added to J in M₁.
    • X₂ = {x ∈ E \ J : J ∪ {x} ∈ I₂} — elements that can be freely added to J in M₂.

    Augmenting Path

    An augmenting path is a shortest path from any vertex in X₁ to any vertex in X₂ in DJ. If such a path P exists, set J ← J △ V(P) (symmetric difference). The new J has size |J| + 1 and remains a common independent set. If no augmenting path exists, J is already optimal.

    Algorithm Summary

    1. Start with J = ∅.
    2. Construct the exchange graph DJ.
    3. Compute sets X₁ and X₂.
    4. Find a shortest X₁–X₂ path using BFS.
    5. If a path exists, augment J along it and go to step 2.
    6. If no path exists, return J.

    Using shortest augmenting paths (BFS), the algorithm runs in at most r iterations (where r is the rank of the intersection), and each iteration requires O(|E|²) oracle calls in the worst case. The total complexity is O(r · |E|²) oracle calls, plus the cost of each oracle query.

    Implementation

    Below is a generic C++ matroid intersection template. We represent each matroid by an independence oracle — a function that checks whether a given set is independent. The algorithm builds the exchange graph implicitly during BFS.

    Matroid Oracle Interface

    // Abstract matroid oracle: checks independence of a set
// For efficiency, we use incremental checks:
//   can_add(J, e)  — can element e be added to current independent set J?
//   can_swap(J, rem, add) — is (J \ {rem}) ∪ {add} independent?
struct Matroid {
    int n; // ground set size
    // Returns true if J ∪ {e} is independent, given J is independent
    virtual bool can_add(const vector<bool>& in_set, int e) = 0;
    // Returns true if (J \ {rem}) ∪ {add} is independent
    virtual bool can_swap(const vector<bool>& in_set, int rem, int add) = 0;
    virtual ~Matroid() = default;
};

    Graphic Matroid (Union-Find Based)

    struct GraphicMatroid : Matroid {
    int V;
    vector<pair<int,int>> edges; // edge list

    GraphicMatroid(int V, vector<pair<int,int>> edges)
        : V(V), edges(edges) { n = edges.size(); }

    // DSU helpers
    vector<int> par, rnk;
    void init_dsu() { par.assign(V, 0); rnk.assign(V, 0); iota(par.begin(), par.end(), 0); }
    int find(int x) { return par[x] == x ? x : par[x] = find(par[x]); }
    bool unite(int a, int b) {
        a = find(a); b = find(b);
        if (a == b) return false;
        if (rnk[a] < rnk[b]) swap(a, b);
        par[b] = a;
        if (rnk[a] == rnk[b]) rnk[a]++;
        return true;
    }

    void build_dsu(const vector<bool>& in_set, int skip = -1) {
        init_dsu();
        for (int i = 0; i < n; i++) {
            if (i == skip || !in_set[i]) continue;
            auto [u, v] = edges[i];
            unite(u, v);
        }
    }

    bool can_add(const vector<bool>& in_set, int e) override {
        build_dsu(in_set);
        auto [u, v] = edges[e];
        return find(u) != find(v);
    }

    bool can_swap(const vector<bool>& in_set, int rem, int add) override {
        build_dsu(in_set, rem);
        auto [u, v] = edges[add];
        return find(u) != find(v);
    }
};

    Partition Matroid

    struct PartitionMatroid : Matroid {
    // color[i] = which group element i belongs to
    // cap[c]   = max elements from group c
    vector<int> color;
    vector<int> cap;

    PartitionMatroid(int n, vector<int> color, vector<int> cap)
        : color(color), cap(cap) { this->n = n; }

    int count_color(const vector<bool>& in_set, int c, int skip = -1) {
        int cnt = 0;
        for (int i = 0; i < n; i++)
            if (in_set[i] && i != skip && color[i] == c) cnt++;
        return cnt;
    }

    bool can_add(const vector<bool>& in_set, int e) override {
        return count_color(in_set, color[e]) < cap[color[e]];
    }

    bool can_swap(const vector<bool>& in_set, int rem, int add) override {
        if (color[rem] == color[add])
            return true; // removing one and adding one of same color: count unchanged
        return count_color(in_set, color[add], rem) < cap[color[add]];
    }
};

    Matroid Intersection Algorithm (Unweighted)

    // Returns maximum common independent set of M1 and M2
// Ground set = {0, 1, ..., n-1}
vector<int> matroid_intersection(Matroid& M1, Matroid& M2, int n) {
    vector<bool> in_set(n, false);

    while (true) {
        // Build exchange graph via BFS
        // par[e] = parent in BFS tree, -1 = unvisited, -2 = source
        vector<int> par(n, -1);
        queue<int> q;

        // X1: elements outside J that can be freely added in M1
        for (int e = 0; e < n; e++) {
            if (!in_set[e] && M1.can_add(in_set, e)) {
                par[e] = -2; // source node
                q.push(e);
            }
        }

        int target = -1;
        while (!q.empty() && target == -1) {
            int v = q.front(); q.pop();

            if (in_set[v]) {
                // v is in J: try arcs v -> x for M1-swaps
                for (int x = 0; x < n; x++) {
                    if (par[x] != -1 || in_set[x]) continue;
                    if (M1.can_swap(in_set, v, x)) {
                        par[x] = v;
                        q.push(x);
                    }
                }
            } else {
                // v is outside J: check if v is in X2 (can be added to M2)
                if (M2.can_add(in_set, v)) {
                    target = v;
                    break;
                }
                // try arcs v -> y for M2-swaps
                for (int y = 0; y < n; y++) {
                    if (par[y] != -1 || !in_set[y]) continue;
                    if (M2.can_swap(in_set, y, v)) {
                        par[y] = v;
                        q.push(y);
                    }
                }
            }
        }

        if (target == -1) break; // no augmenting path

        // Augment along the path: flip membership of each vertex
        int cur = target;
        while (cur != -2) {
            in_set[cur] = !in_set[cur];
            cur = par[cur];
        }
    }

    vector<int> result;
    for (int i = 0; i < n; i++)
        if (in_set[i]) result.push_back(i);
    return result;
}
    Complexity: The algorithm performs at most r augmentation rounds (r = size of the optimal solution). Each round does a BFS that makes O(n²) oracle calls. Total: O(r · n²) oracle calls. For graphic matroids, each oracle call is O(n α(n)) ≈ O(n), giving O(r · n³) overall. For partition matroids, oracle calls are O(n), yielding the same bound.

    Examples and Applications

    Colorful Spanning Tree

    Problem: Given a connected graph where each edge has one of k colors, determine whether there exists a spanning tree that uses at most one edge of each color. If it exists, find one.

    Solution: Let the ground set be the edge set E. Define M₁ as the graphic matroid (independent = acyclic) and M₂ as the partition matroid (color groups, each with capacity 1). A common independent set of size |V| − 1 is a rainbow spanning tree. 

    // Colorful spanning tree via matroid intersection
int main() {
    int V, E;
    cin >> V >> E;

    vector<pair<int,int>> edges(E);
    vector<int> color(E);
    int num_colors = 0;

    for (int i = 0; i < E; i++) {
        cin >> edges[i].first >> edges[i].second >> color[i];
        num_colors = max(num_colors, color[i] + 1);
    }

    GraphicMatroid M1(V, edges);
    PartitionMatroid M2(E, color, vector<int>(num_colors, 1));

    auto result = matroid_intersection(M1, M2, E);

    if ((int)result.size() == V - 1) {
        cout << "Rainbow spanning tree exists!\n";
        for (int idx : result) {
            auto [u, v] = edges[idx];
            cout << u << " -- " << v << " (color " << color[idx] << ")\n";
        }
    } else {
        cout << "No rainbow spanning tree.\n";
    }
}

    Bipartite Matching as Matroid Intersection

    While we normally use Hopcroft-Karp for bipartite matching, it is instructive to see how it fits the matroid intersection framework. Given a bipartite graph (L, R, E):

    • Ground set: edges E.
    • M₁: partition matroid — group edges by left endpoint, capacity 1 each.
    • M₂: partition matroid — group edges by right endpoint, capacity 1 each.

    A common independent set of maximum cardinality is a maximum matching. The augmenting path algorithm on these two partition matroids essentially reduces to the standard augmenting path algorithm for bipartite matching.

    Scheduling on Parallel Machines

    Problem: Schedule n unit-time jobs on k identical machines to maximize the number of jobs meeting their deadlines. Formulation: M₁ = partition matroid on jobs grouped by feasible time slots; M₂ = partition matroid on slots with capacity 1. Matroid intersection yields the optimal schedule.

    Weighted Matroid Intersection

    The unweighted version finds a common independent set of maximum cardinality. The weighted version assigns a weight w(e) to each element e ∈ E and seeks a common independent set of maximum total weight among all common independent sets of maximum cardinality. Alternatively, one can maximize weight without the cardinality constraint.

    Algorithm Sketch

    The weighted matroid intersection algorithm modifies the augmenting path approach by using shortest paths with respect to weights (Bellman-Ford or potential-based Dijkstra) instead of plain BFS:

    1. Assign weights to arcs in the exchange graph: for an arc involving element e, use weight +w(e) if e is being added and −w(e) if e is being removed.
    2. Find a minimum-weight augmenting path (this maximizes the net weight gain per augmentation).
    3. Use a potential function (dual variables) to eliminate negative weights, enabling Dijkstra instead of Bellman-Ford.

    The complexity becomes O(r · n² · log n) with Dijkstra, or O(r · n³) with Bellman-Ford.

    Weighted Implementation

    // Weighted matroid intersection: finds max-weight common independent set
// of maximum cardinality. Uses shortest-path augmentation with potentials.
vector<int> weighted_matroid_intersection(
    Matroid& M1, Matroid& M2, int n, const vector<long long>& w)
{
    const long long INF = 1e18;
    vector<bool> in_set(n, false);
    vector<long long> pot(n, 0); // dual potentials

    while (true) {
        // Bellman-Ford / shortest path in exchange graph
        vector<long long> dist(n, INF);
        vector<int> par(n, -1);
        vector<bool> in_queue(n, false);

        // X1 sources: elements outside J freely addable to M1
        for (int e = 0; e < n; e++) {
            if (!in_set[e] && M1.can_add(in_set, e)) {
                dist[e] = -w[e] + pot[e];
                par[e] = -2;
                in_queue[e] = true;
            }
        }

        // Relaxation rounds (Bellman-Ford style, at most n rounds)
        for (int iter = 0; iter < n; iter++) {
            bool updated = false;
            for (int v = 0; v < n; v++) {
                if (dist[v] == INF) continue;
                if (in_set[v]) {
                    // v in J: arcs to elements outside J via M1-swap
                    for (int x = 0; x < n; x++) {
                        if (in_set[x] || x == v) continue;
                        if (!M1.can_swap(in_set, v, x)) continue;
                        long long nd = dist[v] - w[x] + pot[x] - pot[v];
                        if (nd < dist[x]) {
                            dist[x] = nd;
                            par[x] = v;
                            updated = true;
                        }
                    }
                } else {
                    // v outside J: arcs to elements in J via M2-swap
                    for (int y = 0; y < n; y++) {
                        if (!in_set[y] || y == v) continue;
                        if (!M2.can_swap(in_set, y, v)) continue;
                        long long nd = dist[v] + w[y] + pot[y] - pot[v];
                        if (nd < dist[y]) {
                            dist[y] = nd;
                            par[y] = v;
                            updated = true;
                        }
                    }
                }
            }
            if (!updated) break;
        }

        // Find best target in X2
        int target = -1;
        long long best = INF;
        for (int e = 0; e < n; e++) {
            if (!in_set[e] && dist[e] < INF && M2.can_add(in_set, e)) {
                if (dist[e] < best) {
                    best = dist[e];
                    target = e;
                }
            }
        }

        if (target == -1 || best >= 0) break; // no improving path

        // Update potentials
        for (int i = 0; i < n; i++) {
            if (dist[i] < INF) pot[i] -= dist[i];
        }

        // Augment along the path
        int cur = target;
        while (cur != -2) {
            in_set[cur] = !in_set[cur];
            cur = par[cur];
        }
    }

    vector<int> result;
    for (int i = 0; i < n; i++)
        if (in_set[i]) result.push_back(i);
    return result;
}
    When to use weighted vs. unweighted: If the problem only asks for a maximum-size common independent set, use the simpler BFS-based unweighted version. If you need to maximize a sum of weights (e.g., maximum weight rainbow spanning tree), use the weighted variant.

    Common Pitfalls and Optimization Tips

    Matroid intersection is not as common as segment trees in contests, but when it appears, getting the implementation right is crucial. Here are common issues:

    • Oracle correctness: The most common bug is an incorrect independence oracle. Test your oracles independently on small cases before plugging into the algorithm.
    • Oracle efficiency: The algorithm makes O(r · n²) oracle calls. If each call is O(n) (e.g., graphic matroid rebuilds DSU), the total is O(r · n³). For n ≈ 500, this is ~1.25 × 10⁸ — tight but often feasible.
    • Incremental oracles: For graphic matroids, consider link-cut trees for O(log n) per query instead of rebuilding DSU from scratch.
    • BFS not DFS: Always use BFS (shortest augmenting path), not DFS. Shortest paths give the tightest bound on augmentation rounds.
    • Three-matroid trap: If your problem seems to need three matroids, look for a different formulation. Sometimes one constraint can be folded into another.

    How to Identify Matroid Intersection Problems

    Look for these patterns in problem statements:

    • "Select a subset satisfying two independent structural constraints" — e.g., acyclic AND colorful, spanning AND degree-bounded.
    • Problems that generalize bipartite matching — if the matching has additional structure (like spanning tree requirements), matroid intersection may apply.
    • Exchange property hints — if swapping elements in a feasible solution always yields another feasible solution (matroid axiom), you are likely dealing with a matroid.
    • Greedy works for one constraint but not two — greedy solves single matroid optimization; if you have two greedy-solvable constraints that conflict, try matroid intersection.
    Contest tip: In a timed contest, matroid intersection is a "last resort" technique. First check whether the problem has a simpler flow/matching formulation. Matroid intersection shines when the problem genuinely requires two incompatible greedy structures that cannot be reduced to a network flow.
