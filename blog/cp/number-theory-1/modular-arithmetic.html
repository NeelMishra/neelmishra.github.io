Modular Arithmetic & Overflow Handling

CP Math SeriesNumber Theory

Almost every competitive programming problem involving counting, combinatorics, or number theory requires computing results modulo a large prime (typically 10⁹ + 7). Getting modular arithmetic right—especially avoiding overflow—is a fundamental skill. This post covers every operation you need, with battle-tested C++ implementations.

Modular Addition & Subtraction

The basic properties that make modular arithmetic work for addition and subtraction are:

  • (a + b) mod m = ((a mod m) + (b mod m)) mod m
  • (a − b) mod m = ((a mod m) − (b mod m) + m) mod m
const long long MOD = 1e9 + 7; long long mod_add(long long a, long long b) { return ((a % MOD) + (b % MOD)) % MOD; } long long mod_sub(long long a, long long b) { return ((a % MOD) - (b % MOD) + MOD) % MOD; }
Why the + m in subtraction? In C++, the result of a % m when a is negative is implementation-defined before C++11 and truncates toward zero from C++11 onward. Either way, (a - b) % m can be negative. Adding m before the final mod ensures a non-negative result.

Overflow risk for addition: If a and b are both less than m (about 10⁹), their sum fits in a 64-bit integer (max ~1.8 × 10¹⁸). So addition of two reduced values is always safe with long long.

Modular Multiplication

The key property is: (a × b) mod m = ((a mod m) × (b mod m)) mod m.

long long mod_mul(long long a, long long b) { return ((long long)(a % MOD) * (b % MOD)) % MOD; }
Overflow Danger! If a and b are both up to 10⁹, then a × b can be up to 10¹⁸, which fits in long long (max ≈ 9.2 × 10¹⁸). But if either value approaches 10¹⁸ itself (e.g., summing products), the multiplication overflows! Always reduce before multiplying.
Example of silent overflow:
a = 2 × 10⁹, b = 5 × 10⁹
a × b = 10¹⁹ > 9.2 × 10¹⁸ (max long long)
⟹ undefined behavior in C++! The result wraps around silently.

The safe pattern is to always reduce mod m before multiplying:

// SAFE: both operands reduced to [0, MOD) before multiply long long safe_mul(long long a, long long b) { a %= MOD; if (a < 0) a += MOD; b %= MOD; if (b < 0) b += MOD; return a * b % MOD; }

Overflow Handling Techniques

When the modulus is large (up to 10¹⁸) or when you must multiply before reducing, you need specialized techniques:

Strategy 1: __int128

  • Cast to 128-bit before multiply
  • Fastest approach (hardware support on 64-bit)
  • GCC/Clang only; not in MSVC
  • Works for mod up to ~10¹⁸

Strategy 2: Binary Multiplication

  • Analogous to binary exponentiation
  • O(log b) additions instead of multiplication
  • Portable across all compilers
  • Slower due to loop overhead

Strategy 3: long double Trick

  • Compute ⌊a·b/m⌋ using floating point
  • Then a·b − ⌊a·b/m⌋·m ± m
  • Tricky to get right; precision issues
  • Used in some old competitive solutions

Strategy 4: Reduce Early & Often

  • Keep all values < m before every multiply
  • Works when m < 2³¹ (~2 × 10⁹)
  • No special types needed
  • The default for m = 10⁹ + 7
Never assume values stay small! A common WA pattern: you compute a = (x + y) % MOD and b = (u + v) % MOD, then do a * b % MOD. This is fine for MOD = 10⁹ + 7, but if MOD > 3 × 10⁹, even the product of two reduced values overflows long long.

128-Bit Tricks (__int128)

The __int128 type in GCC/Clang provides native 128-bit integer arithmetic. It can represent values up to about 1.7 × 10³⁸, making overflow essentially impossible for any two 64-bit operands.

// Multiply a * b mod m, safe for m up to ~10^18 long long mul_mod(long long a, long long b, long long m) { return (__int128)a * b % m; }

This is the recommended approach whenever the judge uses GCC (Codeforces, AtCoder, most online judges). It compiles to efficient machine code on x86-64.

For binary (Russian peasant) multiplication, used when __int128 is unavailable:

// O(log b) multiplication mod m, no overflow long long mul_mod_safe(long long a, long long b, long long m) { a %= m; b %= m; if (a < 0) a += m; if (b < 0) b += m; long long result = 0; while (b > 0) { if (b & 1) result = (result + a) % m; a = (a + a) % m; // double a (mod m) b >>= 1; } return result; }

Handling Negative Mods

In mathematics, the modulo operation always returns a non-negative result: (−7) mod 3 = 2 (since −7 = 3·(−3) + 2). In C++ however:

// C++ behavior (truncation toward zero): // -7 % 3 == -1 (not 2!) // -7 / 3 == -2 (truncated toward zero) // Verify: -2 * 3 + (-1) == -7 ✓
This is the #1 source of Wrong Answers in mod problems! Subtraction can produce negative values, and taking % on a negative number gives a negative result in C++. Always add m and take % again.

The standard fix:

// Safe modular reduction for possibly negative values long long safe_mod(long long a, long long m) { return (a % m + m) % m; } // Example: safe_mod(-7, 3) = (-7 % 3 + 3) % 3 = (-1 + 3) % 3 = 2 ✓

Be especially careful when computing differences inside a mod context:

// WRONG: prefix_sum[r] - prefix_sum[l] might be negative long long ans = (prefix[r] - prefix[l]) % MOD; // RIGHT: ensure non-negative long long ans = ((prefix[r] - prefix[l]) % MOD + MOD) % MOD;
Tip: Some competitive programmers always wrap results in ((x % MOD) + MOD) % MOD as a habit, even when they believe x is non-negative. This defensive style prevents subtle bugs in long coding sessions.

Safe Mod Template

Here is a complete modular arithmetic struct used by many competitive programmers. It encapsulates all operations and prevents overflow/negative mod bugs:

const long long MOD = 1e9 + 7; struct Mint { long long v; Mint(long long x = 0) : v((x % MOD + MOD) % MOD) {} Mint operator+(const Mint& o) const { return Mint(v + o.v); } Mint operator-(const Mint& o) const { return Mint(v - o.v + MOD); } Mint operator*(const Mint& o) const { return Mint(v * o.v); // safe since v < 10^9+7 } Mint power(long long p) const { Mint res(1), base(v); for (; p > 0; p >>= 1) { if (p & 1) res = res * base; base = base * base; } return res; } Mint inv() const { return power(MOD - 2); // Fermat's little theorem } Mint operator/(const Mint& o) const { return *this * o.inv(); } };
Usage in contests: With the Mint struct, you can write Mint ans = Mint(n) * Mint(k); and never worry about overflow or negative mods again. Many top competitive programmers use a similar template.

CP Patterns with Mod

Pattern 1: DP with Mod

When computing dp transitions, apply mod at every step:

// Count paths in a grid (mod 10^9+7) long long dp[1001][1001]; dp[0][0] = 1; for (int i = 0; i < n; i++) for (int j = 0; j < m; j++) { if (i + 1 < n) dp[i+1][j] = (dp[i+1][j] + dp[i][j]) % MOD; if (j + 1 < m) dp[i][j+1] = (dp[i][j+1] + dp[i][j]) % MOD; }

Pattern 2: Prefix Sums with Mod

// Safe range-sum with modular subtraction long long prefix[200001]; prefix[0] = 0; for (int i = 1; i <= n; i++) prefix[i] = (prefix[i-1] + a[i]) % MOD; // Query sum [l, r] long long range_sum(int l, int r) { return ((prefix[r] - prefix[l-1]) % MOD + MOD) % MOD; }

Pattern 3: Modular Exponentiation Chains

When computing expressions like a^(b^c) mod m, remember that you apply mod to the exponent using Euler's theorem: a^k ≡ a^(k mod φ(m)) (mod m) when gcd(a, m) = 1.

// Compute a^(b^c) mod m, where m is prime // First compute exp = b^c mod (m-1) [since phi(m) = m-1] // Then compute a^exp mod m long long exp = power(b, c, MOD - 1); // b^c mod (MOD-1) long long ans = power(a, exp, MOD); // a^exp mod MOD
Common mistake: Computing a^(b^c) mod m by first computing b^c as a regular integer. Since b^c can be astronomically large, you must reduce the exponent modulo φ(m), not compute it directly.

Pattern 4: Avoiding Repeated Mod in Tight Loops

When m = 10⁹ + 7 and values are bounded by m, you can sometimes avoid the expensive % operator:

// Instead of: val = (val + x) % MOD; // Use branchless subtraction: val += x; if (val >= MOD) val -= MOD; // This avoids the modulo operation (division is slow) // Only valid when val, x are in [0, MOD)
Performance note: The % operator compiles to a division instruction, which takes ~30 CPU cycles on modern x86. The branch-based subtraction takes ~1 cycle. In tight inner loops (10⁸ iterations), this can shave off 1-2 seconds.
