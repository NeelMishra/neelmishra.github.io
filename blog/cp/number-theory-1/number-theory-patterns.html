Number Theory Problem Patterns for CP

CP Math Series — Post 20Number Theory

This capstone post ties together the entire number theory series. After 19 posts covering primes, sieves, GCD/LCM, modular arithmetic, Euler’s theorem, CRT, multiplicative functions, Möbius inversion, primitive roots, discrete logarithms, and quadratic residues, we now focus on recognising which technique to apply and executing efficiently under contest pressure.

Problem Classification

Number theory problems in CF/ICPC broadly fall into five archetypes. Recognising the type early is half the battle.

Type A: Divisor / Factor

Signal: “divisors of”, “GCD/LCM”, “factorise”, “prime factors”

Techniques: Sieve of Eratosthenes, trial division, Pollard rho, Möbius function, divisor-sum functions

Series refs: Posts 1–4 (primes, sieves), Posts 11–13 (multiplicative functions)

Type B: Modular Arithmetic

Signal: “mod 109+7”, “modular inverse”, “power mod”, “CRT”

Techniques: Fast exponentiation, Fermat/Euler inverse, extended GCD, CRT

Series refs: Posts 5–8 (modular arithmetic, CRT, Euler’s theorem)

Type C: Counting / Enumeration

Signal: “how many pairs”, “count integers with property”, “sum over all”

Techniques: Inclusion-exclusion, Möbius inversion, Euler’s totient summation

Series refs: Posts 9–13 (totient, Möbius, multiplicative functions)

Type D: Construction / Existence

Signal: “find an x such that”, “does there exist”, “construct”

Techniques: Primitive roots, discrete log (BSGS), Tonelli-Shanks, CRT reconstruction

Series refs: Posts 14–16 (CRT), Posts 17–19 (primitive roots, BSGS, QR)

Type E: Hybrid / Ad-hoc

Signal: Combines multiple areas, often with a data-structure or DP layer on top.

Techniques: Combine NT with segment trees, DP, greedy, or graph algorithms.

Series refs: All posts — the key is identifying the NT sub-problem.

Technique Selector

Step 1: Identify if the core is divisibility, modular, or counting.

Step 2: Check constraints — sieve range? single large prime? many queries?

Step 3: Match to the appropriate algorithm from the series.

Divisor & Factor Problems

These problems revolve around the structure of divisors, GCD, LCM, or prime factorisations.

Archetype 1: GCD Queries on Arrays

Pattern: Given an array, answer queries about GCD of subarrays or count pairs with GCD = k.

Solution sketch: For subarray GCD queries, use a sparse table (O(1) per query via associativity of GCD). For counting pairs with gcd = k, use Möbius inversion: count pairs with k | gcd, then Möbius-invert to get exact GCD = k.

// Count pairs (i,j) with gcd(a[i], a[j]) == k // Step 1: Count multiples of each d for (int d = 1; d <= MAX; d++) { cnt[d] = 0; for (int j = d; j <= MAX; j += d) cnt[d] += freq[j]; // freq[v] = # elements equal to v } // Step 2: f[d] = C(cnt[d], 2) = pairs whose gcd is divisible by d for (int d = 1; d <= MAX; d++) f[d] = cnt[d] * (cnt[d] - 1) / 2; // Step 3: Möbius inversion to get exact gcd = d for (int d = MAX; d >= 1; d--) for (int j = 2 * d; j <= MAX; j += d) f[d] -= f[j]; // Answer for gcd == k is f[k]

Archetype 2: Factorise Large Numbers

Pattern: Given n up to 1018, find its prime factorisation.

Solution: Trial division up to 106, then Pollard’s rho for any remaining composite factor. This combination handles all contest constraints.

Tip: For problems with many factorisations of numbers ≤ 107, precompute a smallest-prime-factor (SPF) sieve. Factorisation then takes O(log n) per query.

Modular Arithmetic Problems

These problems require careful modular computation: inverses, large powers, or combining residues from different moduli.

Archetype 3: Modular Inverse and Combinatorics

Pattern: Compute C(n, k) mod p for large n.

Solutions by constraint:

  • p is prime, n < p: Precompute factorials and inverse factorials.
  • n ≥ p, p prime: Lucas’ theorem.
  • p is prime power: Generalised Lucas / Granville’s method.
  • General m: Factor m, compute mod each prime power, CRT to combine.

Archetype 4: Systems of Congruences

Pattern: “Find x ≡ ai (mod mi) for all i.”

Solution: Apply CRT iteratively. If moduli are not pairwise coprime, use the generalised CRT (checking consistency via GCD). Posts 14–16 covered this in detail.

Pitfall: When moduli are not coprime, the system may have no solution. Always check the compatibility condition: ai ≡ aj (mod gcd(mi, mj)) for all pairs.

Counting & Enumeration

Problems asking “how many integers / pairs / tuples satisfy a number-theoretic condition?”

Archetype 5: Sum/Count over GCD Conditions

Pattern: Compute ∑i,j f(gcd(i, j)) for 1 ≤ i,j ≤ n.

Solution: Möbius inversion. The standard reduction: ∑d=1n f(d) ∑k | d μ(d/k) ⌊n/k⌋2. After re-indexing, this can be evaluated in O(n log n) or even O(n√n) with block decomposition.

Archetype 6: Euler Totient Summation

Pattern: Compute ∑i=1n φ(i) or related totient sums.

Solution: Use the identity ∑d | n φ(d) = n and Möbius inversion, or the Meissel-like approach for sub-linear computation of totient prefix sums in O(n2/3).

// Compute sum of phi(i) for i = 1..n using sieve const int MAXN = 1000001; int phi[MAXN]; long long phi_sum[MAXN]; void compute_phi_sums() { iota(phi, phi + MAXN, 0); // phi[i] = i initially for (int i = 2; i < MAXN; i++) { if (phi[i] == i) { // i is prime for (int j = i; j < MAXN; j += i) phi[j] -= phi[j] / i; } } phi_sum[0] = 0; for (int i = 1; i < MAXN; i++) phi_sum[i] = phi_sum[i - 1] + phi[i]; }

Construction Problems

These problems ask you to find or construct an object with specific number-theoretic properties.

Archetype 7: Find x with xk ≡ a (mod p)

Solution: Find a primitive root g (Post 17), compute the discrete log logg(a) via BSGS (Post 18), then solve kt ≡ logg(a) (mod p−1) as a linear congruence. If k = 2, you can shortcut with Tonelli-Shanks or Euler’s criterion (Post 19).

Archetype 8: Construct a Number with Given Divisors

Pattern: “Find the smallest n with exactly d divisors.”

Solution: Since τ(n) = ∏ (ei + 1) for n = ∏ piei, this is an integer partition problem: write d as a product of integers ≥ 2, assign exponents in decreasing order to primes 2, 3, 5, …, and minimise the result. DFS/backtracking with pruning works well.

// Find smallest n with exactly d divisors long long best; int primes[] = {2, 3, 5, 7, 11, 13, 17, 19, 23}; void dfs(int idx, long long cur, int rem, int max_exp) { if (rem == 1) { best = min(best, cur); return; } if (idx >= 9) return; long long pw = primes[idx]; for (int e = 1; e <= max_exp && rem % (e + 1) == 0; e++) { pw *= primes[idx]; // pw = primes[idx]^(e+1) but we want ^e if (cur > best / pw) break; // pruning dfs(idx + 1, cur * pw / primes[idx], rem / (e + 1), e); } // Note: simplified sketch; actual implementation needs care }
Tip: For construction problems, always verify your answer by checking the claimed property. A 2-line verifier can catch subtle bugs and save hours of debugging.

Contest Strategy

Recognition Heuristics

Here is a decision flowchart for NT problems:

The Complete Technique Map

Problem Cue → Technique

  • “GCD of subarray” → Sparse table / Segment tree
  • “Coprime pairs” → Möbius inversion
  • “Factorial mod p” → Precompute + inverse
  • “C(n,k) mod p, n≥p” → Lucas’ theorem
  • “xk ≡ a” → Primitive root + BSGS
  • “Square root mod p” → Tonelli-Shanks

Constraint → Approach

  • n ≤ 107 → Sieve-based
  • n ≤ 1012 → Trial division + Pollard rho
  • n ≤ 1018 → Miller-Rabin + Pollard rho
  • Multiple moduli → CRT
  • “Modulo 998244353” → Likely needs NTT
  • “Modulo 109+7” → Standard modular arithmetic

Time Management

  • First 5 minutes: Read all problems, tag the NT ones.
  • NT classification: Spend 2 minutes identifying the archetype before coding.
  • Template usage: Keep pre-written templates for sieve, mod-inverse, CRT, BSGS, Tonelli-Shanks. Typing these from scratch wastes 10–15 minutes.
  • Edge cases first: Write the core logic, then immediately handle edge cases (n=0, n=1, p=2, gcd conditions).

Pitfalls & Debugging

The Top 10 NT Bugs

  1. Integer overflow: Multiplying two long long values without __int128 or modular reduction.
  2. Negative modular arithmetic: (a - b) % p can be negative in C++. Always add p and re-mod.
  3. Off-by-one in sieve: Sieve up to n but array has size n (needs n+1).
  4. Forgetting gcd check: Modular inverse exists only when gcd(a, m) = 1.
  5. Wrong Euler totient: φ(1) = 1, not 0. φ(p) = p−1, not p.
  6. CRT with non-coprime moduli: Using standard CRT formula when moduli share factors.
  7. BSGS with non-primitive-root base: The discrete log may not exist; check before assuming.
  8. Tonelli-Shanks on non-QR: Always verify a(p−1)/2 = 1 before proceeding.
  9. Sieve not clearing: Forgetting to reinitialise between test cases in multi-test problems.
  10. Mixing 32-bit and 64-bit: Using int for sieve arrays but long long for computation, causing silent truncation.
Critical pitfall: In C++, a % m preserves the sign of a. The expression (a * b) % m can overflow even with long long when a, b > 3 × 109. Always use (__int128)a * b % m or implement Barrett/Montgomery reduction.

Debugging Strategy

  1. Small cases: Generate brute-force answers for n ≤ 20 and compare.
  2. Stress test: Random input generator + brute force + your solution in a loop.
  3. Print intermediate values: For modular computations, print before and after each operation to find where the value diverges.
  4. Edge cases: Always test n = 0, 1, 2, a prime, a prime power, and a product of two primes.
// Stress testing template for NT problems #include <bits/stdc++.h> using namespace std; ll brute(ll n) { /* naive O(n^2) solution */ } ll fast(ll n) { /* your optimised solution */ } int main() { srand(time(0)); for (int iter = 0; iter < 100000; iter++) { ll n = rand() % 1000 + 1; ll ans1 = brute(n), ans2 = fast(n); if (ans1 != ans2) { cout << "MISMATCH n=" << n << " brute=" << ans1 << " fast=" << ans2 << "\n"; return 1; } } cout << "All tests passed\n"; }
