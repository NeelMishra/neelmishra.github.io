Linear Diophantine Equations

CP Math SeriesNumber Theory

Existence Conditions

A linear Diophantine equation has the form ax + by = c, where a, b, c are given integers and we seek integer solutions (x, y). The fundamental question is: when does a solution exist?

Theorem (Bézout's Identity). The equation ax + by = c has integer solutions if and only if gcd(a, b) divides c.

Proof

(⇒ Necessity) Suppose (x₀, y₀) is a solution, so ax₀ + by₀ = c. Let g = gcd(a, b). Then g | a and g | b, so g | (ax₀ + by₀) = c. Therefore g | c.

(⇐ Sufficiency) Suppose g | c. By Bézout's identity, there exist integers u, v such that au + bv = g. Since g | c, let c = g·k for some integer k. Then a(u·k) + b(v·k) = g·k = c, so (x₀, y₀) = (u·k, v·k) is a solution. QED.

Examples:

  • 6x + 10y = 14: gcd(6,10) = 2, and 2 | 14. ✓ Solutions exist.
  • 6x + 10y = 15: gcd(6,10) = 2, and 2 ∤ 15. ✗ No solutions.
  • 3x + 7y = 1: gcd(3,7) = 1, and 1 | 1. ✓ Solutions exist (e.g., x=5, y=−2).

Finding One Solution

Once we know a solution exists, we find one using the extended Euclidean algorithm. The extended GCD gives us u, v such that au + bv = gcd(a, b) = g. Then we scale: x₀ = u · (c/g), y₀ = v · (c/g).

long long ext_gcd(long long a, long long b, long long &x, long long &y) { if (b == 0) { x = 1; y = 0; return a; } long long x1, y1; long long g = ext_gcd(b, a % b, x1, y1); x = y1; y = x1 - (a / b) * y1; return g; } // Find one solution to ax + by = c // Returns true if solution exists, sets x0 and y0 bool find_solution(long long a, long long b, long long c, long long &x0, long long &y0) { long long g = ext_gcd(abs(a), abs(b), x0, y0); if (c % g != 0) return false; // no solution x0 *= c / g; y0 *= c / g; // Adjust signs for negative a, b if (a < 0) x0 = -x0; if (b < 0) y0 = -y0; return true; }
⚠ Edge case — negative coefficients: The extended GCD works on positive inputs. When a or b is negative, compute ext_gcd(|a|, |b|, …) and negate the corresponding variable. Forgetting this is a very common bug!

Worked Example

Solve 6x + 10y = 14.

Step 1: ext_gcd(6, 10) gives g = 2 with 6·(2) + 10·(−1) = 2.

Step 2: Scale by c/g = 14/2 = 7: x₀ = 2·7 = 14, y₀ = (−1)·7 = −7.

Verify: 6·14 + 10·(−7) = 84 − 70 = 14 ✓.

⚠ Pitfall with a = 0 or b = 0: If a = 0, the equation becomes by = c. A solution exists iff b | c, giving y = c/b and x arbitrary. Handle these degenerate cases before calling the general solver. Similarly for b = 0.

General Solution

Once we have a particular solution (x₀, y₀), the complete solution set is parameterized by an integer t:

x = x₀ + (b/g) · t      y = y₀ − (a/g) · t

where g = gcd(a, b) and t ranges over all integers.

Proof

If (x₀, y₀) and (x₁, y₁) are both solutions, then a(x₁ − x₀) + b(y₁ − y₀) = 0, which gives a(x₁ − x₀) = −b(y₁ − y₀). Dividing by g: (a/g)(x₁ − x₀) = −(b/g)(y₁ − y₀). Since gcd(a/g, b/g) = 1, we must have (b/g) | (x₁ − x₀), so x₁ − x₀ = (b/g)·t for some integer t. Then y₁ − y₀ = −(a/g)·t. QED.

Example: For 6x + 10y = 14 with particular solution (14, −7) and g = 2:

  • x = 14 + 5t, y = −7 − 3t for any integer t.
  • t = −2: x = 4, y = −1. Verify: 6·4 + 10·(−1) = 24 − 10 = 14 ✓.
  • t = −3: x = −1, y = 2. Verify: 6·(−1) + 10·2 = −6 + 20 = 14 ✓.

Counting Solutions in a Range

A common CP problem: count the number of solutions (x, y) to ax + by = c where xmin ≤ x ≤ xmax and ymin ≤ y ≤ ymax.

Since x = x₀ + (b/g)·t, the constraint xmin ≤ x ≤ xmax becomes:

(xmin − x₀) / (b/g) ≤ t ≤ (xmax − x₀) / (b/g)

Similarly from the y constraint. The valid range of t is the intersection of both intervals. The count is max(0, tmax − tmin + 1).

// Count solutions to ax + by = c with x in [xlo, xhi], y in [ylo, yhi] long long count_solutions(long long a, long long b, long long c, long long xlo, long long xhi, long long ylo, long long yhi) { long long x0, y0; if (!find_solution(a, b, c, x0, y0)) return 0; long long g = __gcd(abs(a), abs(b)); long long dx = b / g; // step for x long long dy = a / g; // step for y (negative direction) // Ensure dx > 0 (if not, negate t direction) if (dx < 0) { dx = -dx; dy = -dy; } // Range of t from x constraint auto ceil_div = [](long long num, long long den) -> long long { return num / den + (num % den != 0 && (num ^ den) > 0); }; auto floor_div = [](long long num, long long den) -> long long { return num / den - (num % den != 0 && (num ^ den) < 0); }; long long t_lo = ceil_div(xlo - x0, dx); long long t_hi = floor_div(xhi - x0, dx); // Range of t from y constraint: y = y0 - dy * t // ylo <= y0 - dy*t <= yhi if (dy > 0) { t_lo = max(t_lo, ceil_div(y0 - yhi, dy)); t_hi = min(t_hi, floor_div(y0 - ylo, dy)); } else if (dy < 0) { t_lo = max(t_lo, ceil_div(y0 - ylo, dy)); t_hi = min(t_hi, floor_div(y0 - yhi, dy)); } else { // dy == 0 means a == 0; y0 is fixed if (y0 < ylo || y0 > yhi) return 0; } return max(0LL, t_hi - t_lo + 1); }
⚠ Floor/ceiling division with negatives: C++ truncates towards zero, not towards negative infinity. You must implement proper floor and ceiling division functions. The XOR-based sign check (num ^ den) < 0 detects when numerator and denominator have different signs.

C++ Truncation Division

  • −7 / 2 = −3 (truncates toward 0)
  • −7 % 2 = −1
  • Not equivalent to floor division
  • Causes off-by-one errors in ranges

Mathematical Floor Division

  • ⌊−7/2⌋ = −4 (rounds toward −∞)
  • Remainder always non-negative
  • Needed for correct range counting
  • Must implement manually in C++

Multi-Variable Extensions

The theory extends to equations with more than two variables. The equation a₁x₁ + a₂x₂ + … + aₖxₖ = c has integer solutions if and only if gcd(a₁, a₂, …, aₖ) | c.

To solve, we reduce iteratively. First solve a₁x₁ + a₂x₂ = g₂ where g₂ = gcd(a₁, a₂). Then solve g₂·t₂ + a₃x₃ = g₃ where g₃ = gcd(g₂, a₃). Continue until we've processed all variables.

// Check if a₁x₁ + a₂x₂ + ... + aₖxₖ = c has solutions bool multi_diophantine_exists(vector<long long> &a, long long c) { long long g = 0; for (long long ai : a) g = __gcd(g, abs(ai)); return g != 0 && c % g == 0; } // Find one solution to a₁x₁ + a₂x₂ + ... + aₖxₖ = c bool multi_diophantine(vector<long long> &a, long long c, vector<long long> &x) { int k = a.size(); x.assign(k, 0); long long g = a[0]; x[0] = 1; // initially: a[0]*1 = a[0] for (int i = 1; i < k; i++) { long long u, v; g = ext_gcd(abs(g), abs(a[i]), u, v); // Scale all previous x's by u for (int j = 0; j < i; j++) x[j] *= u; x[i] = v; if (a[i] < 0) x[i] = -x[i]; } if (g == 0) return c == 0; if (c % g) return false; long long scale = c / g; for (int j = 0; j < k; j++) x[j] *= scale; return true; }
Degrees of freedom: An equation in k variables has a (k−1)-dimensional solution space. For 2 variables we get a line of solutions (parameterized by t). For 3 variables we get a plane, and so on. In CP, you typically add constraints (e.g., non-negativity) to make the count finite.

CP Problem Patterns

Pattern 1: Coin Change Feasibility

"Given coins of denominations a and b, can you make exactly c cents?" This is exactly ax + by = c with x, y ≥ 0. Find if a non-negative solution exists.

// Can we represent c using coins of value a and b? bool coin_possible(long long a, long long b, long long c) { long long x0, y0; if (!find_solution(a, b, c, x0, y0)) return false; // Check if a non-negative solution exists return count_solutions(a, b, c, 0, c / a, 0, c / b) > 0; }

Pattern 2: Chicken McNugget Theorem

If gcd(a, b) = 1, the largest number NOT representable as ax + by (x, y ≥ 0) is ab − a − b. This is the Frobenius number for two coprime denominations. All integers greater than ab − a − b are representable.

Pattern 3: Grid Walk / Lattice Points

"Starting at origin, each step goes (+a, +b) or (+c, +d). Can you reach (X, Y)?" This gives the system: a·s + c·t = X, b·s + d·t = Y. Solve one equation for t and substitute into the other.

Pattern 4: Minimum Positive x

To find the smallest positive x satisfying ax + by = c: find x₀, then compute xmin = x₀ mod (b/g). If xmin ≤ 0, add b/g.

// Find smallest positive x such that ax + by = c has a solution long long min_positive_x(long long a, long long b, long long c) { long long x0, y0; if (!find_solution(a, b, c, x0, y0)) return -1; // no solution long long g = __gcd(abs(a), abs(b)); long long step = abs(b / g); x0 %= step; if (x0 <= 0) x0 += step; return x0; }
⚠ Watch for a = 0 or b = 0: When a = 0, the equation is by = c. Then x is unconstrained — the "minimum positive x" is 1 (or any positive integer). When b = 0, then x = c/a must be exact. Always handle these degenerate cases first.

Pattern 5: Modular Arithmetic as a Special Case

The congruence ax ≡ c (mod m) is equivalent to ax + my = c. So solving linear congruences is just a special case of linear Diophantine equations. This connects directly to the extended GCD-based modular inverse.
