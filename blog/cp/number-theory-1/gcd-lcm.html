GCD & LCM Algorithms

CP Math SeriesNumber Theory

Naive GCD

The greatest common divisor (GCD) of two integers a and b is the largest positive integer that divides both. The simplest approach: iterate from min(a, b) downward until you find a common divisor.

// Naive GCD: O(min(a, b)) int naiveGCD(int a, int b) { int result = 1; for (int d = min(a, b); d >= 1; d--) { if (a % d == 0 && b % d == 0) { result = d; break; } } return result; }

Complexity: O(min(a, b)) in the worst case. This is terrible — for a = 109 and b = 109 + 1, we check nearly 109 values. We need something dramatically better.

An alternative naive approach uses the factorization-based definition: find the prime factorization of both numbers, then take the minimum exponent for each prime. While conceptually clean, factorization itself is O(√n), which is already much better but still not optimal.

Historical note: The Euclidean algorithm (next section) was described in Euclid’s Elements (~300 BC) and is one of the oldest known algorithms. It reduces GCD computation to O(log(min(a, b))) — an exponential improvement over the naive approach.

Euclidean Algorithm

The key insight: gcd(a, b) = gcd(b, a mod b). Applying this repeatedly until one argument becomes zero gives the GCD, since gcd(d, 0) = d.

// Euclidean algorithm: O(log(min(a, b))) long long gcd(long long a, long long b) { while (b) { a %= b; swap(a, b); } return a; } // Recursive version (cleaner but uses stack) long long gcdRec(long long a, long long b) { return b == 0 ? a : gcdRec(b, a % b); }

Proof of correctness: Let d = gcd(a, b). We show gcd(a, b) = gcd(b, a mod b).

  • Write a = bq + r where r = a mod b.
  • Since d | a and d | b, we have d | (a − bq) = r. So d divides both b and r.
  • Conversely, any common divisor of b and r also divides bq + r = a. So the set of common divisors is the same, hence the greatest is the same. ∎

Proof of O(log(min(a, b))) complexity:

We claim that after every two steps, the larger argument decreases by at least half. Consider two consecutive steps starting with (a, b) where a ≥ b:

  • Step 1: (a, b) → (b, a mod b). Let r = a mod b.
  • If r ≤ b/2, then after one step the second argument is already ≤ b/2.
  • If r > b/2, then in step 2: (b, r) → (r, b mod r). Since r > b/2, we have b mod r = b − r < b/2.

In either case, within two steps, the smaller argument drops below b/2. Since b starts at most min(a, b) and halves every two steps, we need at most 2 · log2(min(a, b)) steps total. Each step is O(1) (one modular reduction), giving O(log(min(a, b))) overall. ∎

Fun fact: The worst case for the Euclidean algorithm is consecutive Fibonacci numbers. gcd(Fn, Fn−1) requires exactly n − 2 steps, and Fn ∼ φn/√5, confirming the logarithmic bound. This was proved by Lamé in 1844.
Pitfall: In C++, __gcd(a, b) and C++17’s std::gcd(a, b) are available, but be careful: __gcd can return negative values if inputs are negative. Always use abs or ensure non-negative inputs. std::gcd handles this correctly.

Extended Euclidean Algorithm

The extended Euclidean algorithm finds not only gcd(a, b) but also integers x, y such that ax + by = gcd(a, b) (Bézout’s identity). This is essential for modular inverse computation.

// Extended Euclidean: returns gcd, sets x and y long long extgcd(long long a, long long b, long long &x, long long &y) { if (b == 0) { x = 1; y = 0; return a; } long long x1, y1; long long g = extgcd(b, a % b, x1, y1); x = y1; y = x1 - (a / b) * y1; return g; } // Modular inverse: a^(-1) mod m, exists iff gcd(a,m)=1 long long modInverse(long long a, long long m) { long long x, y; long long g = extgcd(a, m, x, y); if (g != 1) return -1; // no inverse return (x % m + m) % m; }

Derivation: At the base case (b = 0), gcd(a, 0) = a, and a · 1 + 0 · 0 = a. For the recursive step, suppose we have gcd(b, a%b) = b · x1 + (a%b) · y1. Since a%b = a − ⌊a/b⌋ · b, we get:

g = b · x1 + (a − ⌊a/b⌋ · b) · y1 = a · y1 + b · (x1 − ⌊a/b⌋ · y1)

So x = y1 and y = x1 − ⌊a/b⌋ · y1.

Tip: The modular inverse of a modulo m is the most common application of extgcd in CP. An alternative when m is prime: use Fermat’s little theorem, a−1 ≡ am−2 (mod m), computed via fast exponentiation.

Binary GCD (Stein’s Algorithm)

The binary GCD replaces expensive modular reductions with bit shifts and subtractions. On hardware with fast bit operations, it can be faster than the Euclidean algorithm in practice.

Key identities:

  • gcd(0, b) = b, gcd(a, 0) = a
  • If both a, b are even: gcd(a, b) = 2 · gcd(a/2, b/2)
  • If a is even and b is odd: gcd(a, b) = gcd(a/2, b)
  • If both odd and a ≥ b: gcd(a, b) = gcd((a − b)/2, b)
// Binary GCD (Stein's algorithm) long long binaryGCD(long long a, long long b) { if (a == 0) return b; if (b == 0) return a; // Extract common factor of 2 int shift = __builtin_ctzll(a | b); a >>= __builtin_ctzll(a); do { b >>= __builtin_ctzll(b); if (a > b) swap(a, b); b -= a; } while (b != 0); return a << shift; }

Complexity: O(log(a · b)) = O(log a + log b). Each iteration reduces the total bit-length by at least 1 (the subtraction and right-shift each remove at least one bit). In practice, this is comparable to or slightly better than the Euclidean algorithm.

Why use it? On some architectures, the modulo operation (%) is expensive (taking 20–40 CPU cycles), while bit shifts and count-trailing-zeros take 1–3 cycles. Binary GCD can be 2–3× faster in tight inner loops.

Pitfall: __builtin_ctzll(0) is undefined behavior! Always check for zero before calling. The implementation above handles this by checking a == 0 and b == 0 upfront, and the loop terminates before b reaches zero (the do...while condition catches it).

In modern C++, GCC’s __gcd already uses an optimized implementation. On most online judges, the built-in is fast enough. Binary GCD is worth knowing for systems programming and when you need to squeeze every nanosecond.

LCM Computation

The least common multiple of a and b is defined as the smallest positive integer divisible by both. The fundamental identity connecting GCD and LCM is:

lcm(a, b) = a / gcd(a, b) · b

Proof: Let d = gcd(a, b). Write a = d·m, b = d·n where gcd(m, n) = 1. Any common multiple of a and b must be divisible by d·m·n (since it must contain all prime factors of both a and b). The smallest such is d·m·n = a·b/d = a/d·b. ∎

// LCM using GCD identity long long lcm(long long a, long long b) { return a / gcd(a, b) * b; // divide first to avoid overflow }
Pitfall: Always compute a / gcd(a,b) * b, not a * b / gcd(a,b). The latter overflows for large values. For example, with a = b = 109, a*b = 1018 which barely fits in long long, but if gcd = 1 you lose precision. Dividing first keeps intermediate values smaller.

LCM of multiple numbers: Compute iteratively — lcm(a, b, c) = lcm(lcm(a, b), c). The order doesn’t matter since LCM is associative and commutative.

// LCM of an array of numbers long long arrayLCM(vector<long long> &v) { long long result = 1; for (long long x : v) { result = lcm(result, x); if (result <= 0) return -1; // overflow check } return result; } // GCD of an array long long arrayGCD(vector<long long> &v) { long long result = 0; for (long long x : v) result = gcd(result, x); return result; }
Tip: For arrayGCD, initializing result to 0 works because gcd(0, x) = x. This elegantly handles the first element without a special case.

Using the factorization-based definition: if a = ∏ piai and b = ∏ pibi, then:

  • gcd(a, b) = ∏ pimin(ai, bi)
  • lcm(a, b) = ∏ pimax(ai, bi)

This confirms the identity lcm · gcd = a · b, since min(x,y) + max(x,y) = x + y for all x, y.

Algorithm Comparison

Euclidean Algorithm

Time: O(log(min(a, b)))

Operations: Modulo (%)

Pros: Simple, fast, built into STL (__gcd)

Cons: Modulo can be slow on some hardware (20–40 cycles)

Best for: General purpose. Use this by default in CP.

Binary GCD (Stein’s)

Time: O(log a + log b)

Operations: Shifts, subtraction, ctz

Pros: No division, faster on some architectures

Cons: More complex, edge cases with zero

Best for: Performance-critical code, embedded systems, when profiling shows % is a bottleneck.

In practice, on modern x86-64 processors, the Euclidean algorithm and binary GCD have similar performance. The compiler’s __gcd is already well-optimized. Only use binary GCD if benchmarks show a measurable improvement for your specific workload.

CP Applications

GCD and LCM are among the most frequently used tools in competitive programming. Here are the essential patterns.

Pattern 1: Reduce fractions. To simplify a/b, compute g = gcd(a, b) and return (a/g) / (b/g). This is the basis for many problems involving rational numbers.

// Reduce a fraction a/b pair<long long,long long> reduce(long long a, long long b) { long long g = gcd(abs(a), abs(b)); if (b < 0) { a = -a; b = -b; } return {a / g, b / g}; }

Pattern 2: Modular inverse via extgcd. To find a−1 mod m, use the extended Euclidean algorithm. This works for any m (not just primes), as long as gcd(a, m) = 1.

Pattern 3: Linear Diophantine equations. The equation ax + by = c has integer solutions iff gcd(a, b) | c. If g = gcd(a, b) and (x0, y0) is one solution, the general solution is x = x0 + (b/g)t, y = y0 − (a/g)t for any integer t.

// Solve ax + by = c; returns false if no solution bool solveDiophantine(long long a, long long b, long long c, long long &x, long long &y) { long long g = extgcd(abs(a), abs(b), x, y); if (c % g != 0) return false; x *= c / g; y *= c / g; if (a < 0) x = -x; if (b < 0) y = -y; return true; } // Shift solution to find smallest non-negative x void shiftSolution(long long &x, long long &y, long long a, long long b, long long g) { long long step_x = b / g, step_y = a / g; if (x < 0) { long long t = (-x + step_x - 1) / step_x; x += t * step_x; y -= t * step_y; } }

Pattern 4: GCD queries on arrays. Maintain gcd of a subarray using a sparse table (O(1) query) or segment tree (supports updates). GCD is associative and commutative, making it compatible with both structures.

// Sparse table for range GCD queries in O(1) const int MAXN = 200001, LOG = 18; long long sparse[LOG][MAXN]; void buildSparse(long long a[], int n) { for (int i = 0; i < n; i++) sparse[0][i] = a[i]; for (int j = 1; j < LOG; j++) for (int i = 0; i + (1 << j) <= n; i++) sparse[j][i] = gcd(sparse[j-1][i], sparse[j-1][i + (1 << (j-1))]); } long long queryGCD(int l, int r) { int k = __lg(r - l + 1); return gcd(sparse[k][l], sparse[k][r - (1 << k) + 1]); }
Key insight: GCD is idempotent: gcd(a, a) = a. This is why sparse table works — overlapping ranges don’t cause problems, just like with min/max queries.

Pattern 5: Counting pairs with given GCD. Count pairs (i, j) in an array with gcd(ai, aj) = k. Use the standard technique: let f(d) = number of elements divisible by d. Then the number of pairs with d | gcd is C(f(d), 2). Apply Möbius inversion or inclusion-exclusion to extract exactly-k counts.

Pitfall: When computing LCM of many numbers, the result can grow extremely fast (exponentially in the number of elements). Always check for overflow. In many problems, you’re asked for LCM mod some prime — but LCM mod p is tricky because LCM isn’t compatible with modular arithmetic. You may need to track the factorization instead.

Pattern 6: Chicken McNugget theorem (Frobenius). For coprime a, b, the largest number that cannot be represented as xa + yb (x, y ≥ 0) is ab − a − b. This elegant result connects GCD to the coin problem and appears in various DP + number theory hybrids.

GCD and LCM are deceptively simple operations that underpin a vast range of competitive programming techniques. Combined with modular arithmetic and prime factorization, they form the core toolkit for number-theoretic problem solving.
