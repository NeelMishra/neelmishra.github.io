Chinese Remainder Theorem

CP Math SeriesNumber Theory

CRT Statement & Proof

The Chinese Remainder Theorem (CRT) is a cornerstone of number theory that lets us solve systems of simultaneous congruences with pairwise coprime moduli.

Theorem. Let m₁, m₂, …, mₖ be pairwise coprime positive integers. Then for any integers a₁, a₂, …, aₖ, the system:

x ≡ a₁ (mod m₁)
x ≡ a₂ (mod m₂)
    ⋮
x ≡ aₖ (mod mₖ)

has a unique solution modulo M = m₁ × m₂ × … × mₖ.

Proof

Existence: Define Mᵢ = M / mᵢ for each i. Since the moduli are pairwise coprime, gcd(Mᵢ, mᵢ) = 1, so the modular inverse yᵢ = Mᵢ⁻¹ mod mᵢ exists. Then:

x = ∑ᵢ aᵢ · Mᵢ · yᵢ

satisfies all congruences. To verify: for each j, every term with i ≠ j has Mᵢ ≡ 0 (mod mⱼ) (because mⱼ divides Mᵢ when i ≠ j). The j-th term gives aⱼ · Mⱼ · yⱼ ≡ aⱼ · 1 ≡ aⱼ (mod mⱼ). ✓

Uniqueness: If x and x' are two solutions, then x − x' ≡ 0 (mod mᵢ) for all i. Since the mᵢ are pairwise coprime, M divides x − x', so x ≡ x' (mod M). QED.

Constructive Algorithm with Step-by-Step Example

Let's solve a concrete system to see the construction in action.

Problem: Find x such that x ≡ 2 (mod 3), x ≡ 3 (mod 5), x ≡ 2 (mod 7).

Step 1: Compute M = 3 × 5 × 7 = 105.

Step 2: Compute Mᵢ values:

  • M₁ = 105/3 = 35
  • M₂ = 105/5 = 21
  • M₃ = 105/7 = 15

Step 3: Find modular inverses yᵢ = Mᵢ⁻¹ mod mᵢ:

  • y₁ = 35⁻¹ mod 3: 35 ≡ 2 (mod 3), and 2⁻¹ ≡ 2 (mod 3). So y₁ = 2.
  • y₂ = 21⁻¹ mod 5: 21 ≡ 1 (mod 5), and 1⁻¹ ≡ 1 (mod 5). So y₂ = 1.
  • y₃ = 15⁻¹ mod 7: 15 ≡ 1 (mod 7), and 1⁻¹ ≡ 1 (mod 7). So y₃ = 1.

Step 4: Compute x = (2 × 35 × 2 + 3 × 21 × 1 + 2 × 15 × 1) mod 105 = (140 + 63 + 30) mod 105 = 233 mod 105 = 23.

Verification: 23 mod 3 = 2 ✓, 23 mod 5 = 3 ✓, 23 mod 7 = 2 ✓.

Tip: In implementation, reduce intermediate sums modulo M to avoid overflow. Each term aᵢ × Mᵢ × yᵢ can be large, so use __int128 or modular multiplication.

Direct CRT Implementation

long long extended_gcd(long long a, long long b, long long &x, long long &y) { if (b == 0) { x = 1; y = 0; return a; } long long x1, y1; long long g = extended_gcd(b, a % b, x1, y1); x = y1; y = x1 - (a / b) * y1; return g; } long long mod_inverse_ext(long long a, long long m) { long long x, y; extended_gcd(a, m, x, y); return (x % m + m) % m; } // CRT for pairwise coprime moduli long long crt(vector<long long> &a, vector<long long> &m) { int k = a.size(); long long M = 1; for (int i = 0; i < k; i++) M *= m[i]; long long x = 0; for (int i = 0; i < k; i++) { long long Mi = M / m[i]; long long yi = mod_inverse_ext(Mi, m[i]); x = (x + (__int128)a[i] * Mi % M * yi) % M; } return (x + M) % M; }

Garner's Algorithm

Garner's algorithm is a more numerically stable way to apply CRT. Instead of computing a single large sum, it builds the solution incrementally using a mixed-radix representation.

The idea is to represent x in the form:

x = c₁ + c₂·m₁ + c₃·m₁·m₂ + … + cₖ·m₁·m₂·…·mₖ₋₁

where each coefficient cᵢ is determined by the congruence x ≡ aᵢ (mod mᵢ). This avoids computing the full product M and large intermediate values.

// Garner's Algorithm: numerically stable CRT // Returns x mod final_mod such that x ≡ a[i] (mod m[i]) long long garner(vector<long long> &a, vector<long long> &m, long long final_mod) { int k = a.size(); vector<long long> coeffs(k); // mixed-radix coefficients for (int i = 0; i < k; i++) { coeffs[i] = a[i]; for (int j = 0; j < i; j++) { coeffs[i] = mod_inverse_ext(m[j], m[i]) * (coeffs[i] - coeffs[j]) % m[i]; coeffs[i] = (coeffs[i] + m[i]) % m[i]; } } // Reconstruct: x = c₁ + c₂·m₁ + c₃·m₁·m₂ + ... long long result = 0, prefix = 1; for (int i = 0; i < k; i++) { result = (result + coeffs[i] * prefix) % final_mod; prefix = prefix * (m[i] % final_mod) % final_mod; } return result; }

Direct CRT

  • Computes full product M upfront
  • Needs __int128 for large moduli
  • Simpler conceptually
  • O(k) multiplications of big numbers

Garner's Algorithm

  • Avoids computing large product M
  • Works with a final modulus directly
  • Better numerical stability
  • O(k²) modular operations
When to use Garner: If the answer itself must be computed modulo some prime p (e.g., "output the answer mod 10⁹+7"), Garner's algorithm is ideal because it never computes M = m₁·m₂·…·mₖ explicitly — it goes directly to the result mod p.

CRT for Non-Coprime Moduli

When the moduli are not pairwise coprime, the standard CRT does not apply directly. However, we can extend the technique by merging two congruences at a time.

Given x ≡ a₁ (mod m₁) and x ≡ a₂ (mod m₂), a solution exists if and only if:

a₁ ≡ a₂ (mod gcd(m₁, m₂))

If it exists, we can merge into a single congruence x ≡ a (mod lcm(m₁, m₂)).

Merging Two Congruences

We need to find x = a₁ + k·m₁ such that a₁ + k·m₁ ≡ a₂ (mod m₂). This gives:

k·m₁ ≡ (a₂ − a₁) (mod m₂)

Let g = gcd(m₁, m₂). A solution exists iff g | (a₂ − a₁). If it does, divide through by g:

k·(m₁/g) ≡ (a₂ − a₁)/g (mod m₂/g)

Since gcd(m₁/g, m₂/g) = 1, we can find k = ((a₂ − a₁)/g) · (m₁/g)⁻¹ mod (m₂/g). The merged congruence is x ≡ a₁ + k·m₁ (mod lcm(m₁, m₂)).

// Merge two congruences: x ≡ a1 (mod m1), x ≡ a2 (mod m2) // Returns {new_a, new_m} or {-1, -1} if no solution pair<long long, long long> merge_congruences(long long a1, long long m1, long long a2, long long m2) { long long x, y; long long g = extended_gcd(m1, m2, x, y); if ((a2 - a1) % g != 0) return {-1, -1}; // no solution long long lcm = m1 / g * m2; long long diff = (a2 - a1) / g; long long k = (__int128)diff * x % (m2 / g); long long new_a = a1 + (__int128)k * m1 % lcm; new_a = (new_a % lcm + lcm) % lcm; return {new_a, lcm}; } // General CRT: works for non-coprime moduli too pair<long long, long long> general_crt(vector<long long> &a, vector<long long> &m) { long long cur_a = a[0], cur_m = m[0]; for (int i = 1; i < (int)a.size(); i++) { auto [na, nm] = merge_congruences(cur_a, cur_m, a[i], m[i]); if (nm == -1) return {-1, -1}; cur_a = na; cur_m = nm; } return {cur_a, cur_m}; }
⚠ Pitfall: When merging congruences, overflow is a serious risk. The intermediate value k·m₁ can overflow long long. Always use __int128 for multiplication or implement a safe modular multiplication function.

Full C++ Implementation

Here's a complete, contest-ready CRT implementation that handles both coprime and non-coprime cases.

#include <bits/stdc++.h> using namespace std; typedef long long ll; ll ext_gcd(ll a, ll b, ll &x, ll &y) { if (b == 0) { x = 1; y = 0; return a; } ll x1, y1; ll g = ext_gcd(b, a % b, x1, y1); x = y1; y = x1 - (a / b) * y1; return g; } // Safe modular multiplication using __int128 ll safe_mul(ll a, ll b, ll mod) { return (__int128)a * b % mod; } // Merge x≡a1(mod m1) and x≡a2(mod m2) pair<ll, ll> merge(ll a1, ll m1, ll a2, ll m2) { ll x, y; ll g = ext_gcd(m1, m2, x, y); if ((a2 - a1) % g) return {-1, -1}; ll lcm = m1 / g * m2; ll k = safe_mul((a2 - a1) / g, x, m2 / g); ll ans = (a1 + safe_mul(k, m1, lcm)) % lcm; return {(ans + lcm) % lcm, lcm}; } int main() { int n; scanf("%d", &n); ll cur_a, cur_m; scanf("%lld %lld", &cur_a, &cur_m); for (int i = 1; i < n; i++) { ll a, m; scanf("%lld %lld", &a, &m); auto [na, nm] = merge(cur_a, cur_m, a, m); if (nm == -1) { puts("-1"); return 0; } cur_a = na; cur_m = nm; } printf("%lld\n", cur_a); return 0; }

Complexity: O(k · log(max mᵢ)) — dominated by the extended GCD computations.

CP Problem Patterns

Pattern 1: Recovering a Number from Remainders

Given "x mod m₁ = a₁, x mod m₂ = a₂, …", find the smallest non-negative x. This is the direct application of CRT. If moduli aren't coprime, use the general version.

Pattern 2: Big Number Computation via CRT

Compute f(n) mod M where M is composite. Factor M = p₁a₁ · … · pₖaₖ, compute f(n) mod each pᵢaᵢ separately, and reconstruct via CRT. This often avoids needing to work with the full modulus.

Pattern 3: Interpolation over Multiple Primes

In problems requiring polynomial interpolation, compute the interpolated value modulo several primes and combine with CRT. This is the basis of multi-precision arithmetic in competitive programming.

Pattern 4: Hash Collision Avoidance

Multi-hash techniques use CRT implicitly. If two strings have equal hashes modulo several different primes p₁, …, pₖ, by CRT they agree modulo p₁·p₂·…·pₖ, making false positives extremely unlikely.
