Modular Equations — Solving ax ≡ b (mod m)

CP Math SeriesNumber Theory

Linear congruences are the modular-arithmetic analog of linear equations. Many competitive-programming problems — from coin puzzles to scheduling — reduce to asking: does ax ≡ b (mod m) have a solution, and if so, what are they? We will derive the full theory, prove every claim, and build C++ code you can paste straight into a contest.

Linear Congruences

A linear congruence is an equation of the form

ax ≡ b  (mod m)

where a, b, m are given integers with m > 0, and we seek integer solutions x. Two solutions that are congruent modulo m are considered identical.

This is equivalent to asking for integer pairs (x, y) satisfying

ax + my = b

which is a linear Diophantine equation in two unknowns. Bézout's identity tells us exactly when such an equation is solvable.

Key link: Every linear-congruence question is really a question about gcd(a, m) and the extended Euclidean algorithm.

Solving ax ≡ b (mod m)

Let g = gcd(a, m). The congruence ax ≡ b (mod m) is solvable if and only if g | b.

Proof

(⇒) Suppose x₀ is a solution. Then m | (ax₀ − b), so ax₀ − b = mk for some integer k, giving ax₀ − mk = b. Since g divides both a and m it divides the left side, hence g | b.

(⇐) Suppose g | b. Write a = gA, m = gM, b = gB. We need Ax ≡ B (mod M) with gcd(A, M) = 1. Because A is invertible modulo M, x₀ = A⁻¹B mod M is a solution, and lifting back gives x₀ as a solution of the original congruence. □

Reduction to Extended GCD

Run extgcd(a, m) to obtain g, x₀, y₀ with ax₀ + my₀ = g. If g ∤ b, report no solution. Otherwise multiply through by b/g:

x = x₀ · (b / g) mod m

// Extended GCD: returns gcd and sets x, y long long extgcd(long long a, long long b, long long &x, long long &y) { if (b == 0) { x = 1; y = 0; return a; } long long x1, y1; long long g = extgcd(b, a % b, x1, y1); x = y1; y = x1 - (a / b) * y1; return g; } // Solve ax ≡ b (mod m). Returns -1 if no solution. long long solve_linear_cong(long long a, long long b, long long m) { long long x, y; long long g = extgcd(a, m, x, y); if (b % g != 0) return -1; // no solution x = (x % m + m) % m; // normalize x = x * (b / g) % m; return (x % m + m) % m; }
Pitfall: Don't forget to normalize x into [0, m) after the extended-GCD call — the raw x₀ can be negative.

Existence & Uniqueness

When solutions exist there are exactly g = gcd(a, m) distinct solutions modulo m. They form an arithmetic progression with common difference m/g.

Theorem (Count & Structure)

If g | b, the full solution set modulo m is

x ≡ x₀ + k · (m / g)  (mod m),   k = 0, 1, …, g − 1

Proof sketch

After dividing through by g we get A x ≡ B (mod M) with gcd(A,M)=1, which has a unique solution x₀ modulo M = m/g. Lifting this to modulo m gives g equally-spaced representatives. □

Special case: gcd(a,m) = 1

Exactly one solution: x = a⁻¹·b mod m. This is the most common CP scenario — m is prime and a ≢ 0.

General case: g = gcd(a,m) > 1

Either no solution (g ∤ b) or g solutions spaced m/g apart. Enumerate them explicitly.

Enumerating All Solutions

Once we have the base solution x₀ we can enumerate all g solutions in O(g) time. In CP problems you usually need the smallest non-negative solution, or all solutions below some bound N.

// Enumerate all solutions of ax ≡ b (mod m) in [0, m) vector<long long> all_solutions(long long a, long long b, long long m) { vector<long long> res; long long x, y; long long g = extgcd(a, m, x, y); if (b % g != 0) return res; // empty x = (x % m + m) % m; x = x * (b / g) % m; x = (x % m + m) % m; long long step = m / g; for (int k = 0; k < g; ++k) { res.push_back((x + (long long)k * step) % m); } sort(res.begin(), res.end()); return res; }

Counting solutions below N

Given the smallest non-negative solution x₀ and step s = m/g, the count of solutions in [0, N) is

⌊(N − 1 − x₀) / s⌋ + 1  if x₀ < N, else 0.

Overflow trap: When m can be up to 1018, the product x * (b/g) overflows 64-bit integers. Use __int128 or a safe-multiply function.

Complexity

The extended-GCD call is O(log min(a, m)). Enumeration is O(g) which can be up to O(m) in the worst case. For most CP tasks g is tiny or you only need the smallest solution, making the overall cost O(log m).

Systems of Congruences

A system of linear congruences asks for x satisfying several equations simultaneously:

x ≡ r₁ (mod m₁),   x ≡ r₂ (mod m₂),   …

Chinese Remainder Theorem (CRT)

If all moduli are pairwise coprime, a unique solution exists modulo M = m₁·m₂·…·mₖ. For the general (non-coprime) case, we merge two congruences at a time.

Merging two congruences

Given x ≡ r₁ (mod m₁) and x ≡ r₂ (mod m₂), set g = gcd(m₁, m₂). A solution exists iff g | (r₂ − r₁). If so, the merged congruence is

x ≡ r₁ + m₁ · ((r₂ − r₁)/g · m₁⁻¹ mod (m₂/g))  (mod lcm(m₁,m₂))

// Merge x ≡ r1 (mod m1) and x ≡ r2 (mod m2) // Returns {new_r, new_m} or {-1,-1} if impossible pair<long long,long long> merge_cong( long long r1, long long m1, long long r2, long long m2) { long long x, y; long long g = extgcd(m1, m2, x, y); if ((r2 - r1) % g != 0) return {-1, -1}; long long lcm = m1 / g * m2; long long diff = (r2 - r1) / g; // Use __int128 to avoid overflow long long t = (__int128)diff * x % (m2 / g); long long nr = ((__int128)t * m1 + r1) % lcm; if (nr < 0) nr += lcm; return {nr, lcm}; }
CRT connection: When all moduli are pairwise coprime, merge_cong simplifies to the classical CRT construction. In CP you almost always see the general case, so keep the full version.

Iterative system solver

// Solve a system of congruences x ≡ r[i] (mod m[i]) pair<long long,long long> solve_system( vector<long long> &r, vector<long long> &m) { long long cur_r = r[0], cur_m = m[0]; for (int i = 1; i < (int)r.size(); ++i) { auto [nr, nm] = merge_cong(cur_r, cur_m, r[i], m[i]); if (nm == -1) return {-1, -1}; cur_r = nr; cur_m = nm; } return {cur_r, cur_m}; }

CP Applications

Pattern 1 — "Find smallest x"

Given ax ≡ b (mod m), the smallest non-negative x is returned directly by solve_linear_cong. If you need the k-th smallest, add k·(m/g).

Pattern 2 — Counting solutions in a range

How many x ∈ [L, R] satisfy ax ≡ b (mod m)? Get the smallest solution x₀ ≥ L, then count how many multiples of m/g fit in [L, R].

// Count solutions of ax ≡ b (mod m) in [L, R] long long count_in_range(long long a, long long b, long long m, long long L, long long R) { long long x0 = solve_linear_cong(a, b, m); if (x0 == -1) return 0; long long x, y; long long g = extgcd(a, m, x, y); long long step = m / g; // Shift x0 into [L, ..) if (x0 < L) { long long skip = (L - x0 + step - 1) / step; x0 += skip * step; } if (x0 > R) return 0; return (R - x0) / step + 1; }

Pattern 3 — Systems (multi-machine scheduling)

Classic problem: Machine A fires every m₁ seconds starting at r₁; machine B fires every m₂ seconds starting at r₂. When do they first fire together? This is exactly the two-congruence merge.

Pattern 4 — Modular inverse queries

Finding a⁻¹ mod m is the special case b = 1. When m is prime, Fermat's little theorem gives a⁻¹ = am−2 mod m, but for composite m the extended-GCD approach is necessary.

Fermat (prime m only)

a⁻¹ = am−2 mod m. O(log m) via fast exponentiation. Simpler to code, but limited to prime moduli.

Extended GCD (any m)

Works whenever gcd(a, m) = 1. O(log m). Gives extra info (the Bézout coefficients) useful for system solving.

Contest tip: Keep both inverse methods in your template. Use Fermat for the common "mod 10⁹+7" problems; switch to extgcd when the modulus is composite or you need to solve systems.
