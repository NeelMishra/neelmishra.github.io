Extended Euclidean Algorithm & Bézout's Identity

CP Math SeriesNumber Theory

The standard Euclidean algorithm computes gcd(a, b) efficiently, but in competitive programming we frequently need more: we need integers x and y such that ax + by = gcd(a, b). The Extended Euclidean Algorithm provides exactly this. It is the backbone of modular inverse computation, Diophantine equation solving, and the Chinese Remainder Theorem.

Bézout's Identity

Theorem (Bézout). For any integers a and b, not both zero, there exist integers x and y such that:

ax + by = gcd(a, b)

Moreover, gcd(a, b) is the smallest positive integer that can be expressed as a linear combination of a and b. Every integer of the form ax + by is a multiple of gcd(a, b).

Key Insight: Bézout's identity guarantees that the equation ax + by = d has integer solutions if and only if gcd(a, b) divides d. This is the foundation of linear Diophantine equations.

For example, gcd(35, 15) = 5, and indeed 35·(1) + 15·(−2) = 35 − 30 = 5. The coefficients x = 1, y = −2 are one particular solution; infinitely many others exist.

Proof of Bézout's Identity

Proof (by the Well-Ordering Principle)

Let S = { ax + by : x, y ∈ ℤ, ax + by > 0 }. Since a and b are not both zero, S is non-empty (take x = sign(a), y = 0 or similar). By the Well-Ordering Principle, S has a smallest element d = ax₀ + by₀.

Claim: d = gcd(a, b).

Step 1: d divides a. By the division algorithm, a = qd + r where 0 ≤ r < d. Then:

r = a − qd = a − q(ax₀ + by₀) = a(1 − qx₀) + b(−qy₀)

So r is a linear combination of a and b. If r > 0, then r ∈ S and r < d, contradicting the minimality of d. Therefore r = 0 and d | a.

Step 2: d divides b. By an identical argument, d | b.

Step 3: d is the greatest such divisor. If c | a and c | b, then c | (ax₀ + by₀) = d. Hence c ≤ d, proving d = gcd(a, b). ∎

Consequence: The set of all integers representable as ax + by is exactly the set of all multiples of gcd(a, b). This is the subgroup dℤ of ℤ.

Extended Euclidean Algorithm

The algorithm works by augmenting the standard Euclidean algorithm to track the coefficients x and y at each step. The key recurrence is:

If gcd(b, a mod b) = bx₁ + (a mod b)y₁, then since a mod b = a − ⌊a/b⌋·b, we get:

gcd(a, b) = a·y₁ + b·(x₁ − ⌊a/b⌋·y₁)

So the coefficients transform as: x = y₁ and y = x₁ − ⌊a/b⌋·y₁.

The base case is gcd(d, 0) = d, with x = 1, y = 0 (since d·1 + 0·0 = d).

C++ Implementation

The recursive implementation is clean and directly mirrors the mathematical recurrence:

// Returns gcd(a, b) and sets x, y such that a*x + b*y = gcd(a, b) long long extgcd(long long a, long long b, long long &x, long long &y) { if (b == 0) { x = 1; y = 0; return a; } long long x1, y1; long long g = extgcd(b, a % b, x1, y1); // Back-substitute: x = y1, y = x1 - (a/b)*y1 x = y1; y = x1 - (a / b) * y1; return g; }

An iterative version avoids stack overhead for very deep recursions:

long long extgcd_iter(long long a, long long b, long long &x, long long &y) { long long old_x = 1, old_y = 0; x = 0; y = 1; while (b != 0) { long long q = a / b; // Update (a, b) = (b, a - q*b) long long tmp = b; b = a - q * b; a = tmp; // Update coefficients tmp = x; x = old_x - q * x; old_x = tmp; tmp = y; y = old_y - q * y; old_y = tmp; } x = old_x; y = old_y; return a; }
Pitfall: The extgcd function may return a negative gcd if the input a is negative. Many implementations assume a, b > 0. If you need to handle negative inputs, take the absolute value first or adjust signs afterward.

Complexity: Same as the standard Euclidean algorithm: O(log(min(a, b))) divisions. Each step reduces the problem size by at least half, giving at most 2·log₂(min(a, b)) + 1 iterations. The constant factor is slightly higher due to extra arithmetic on x and y.

Modular Inverse via ExtGCD

To find a⁻¹ mod m (assuming gcd(a, m) = 1), we solve ax ≡ 1 (mod m), i.e., ax + my = 1 for some integer y. This is exactly what extgcd gives us.

long long mod_inverse(long long a, long long m) { long long x, y; long long g = extgcd(a, m, x, y); if (g != 1) return -1; // inverse doesn't exist return (x % m + m) % m; // ensure positive result }
Key advantage over Fermat's method: Extended GCD works for any modulus m, not just primes. If gcd(a, m) = 1, the inverse exists regardless of whether m is prime.

Fermat's Little Theorem

a⁻¹ ≡ a^(p−2) mod p

  • Only works when m is prime
  • Requires fast exponentiation
  • O(log p) multiplications
  • Simpler to code

Extended GCD

Solve ax + my = 1

  • Works for any m with gcd(a,m)=1
  • No exponentiation needed
  • O(log m) divisions
  • Slightly more complex code

CP Applications

Linear Diophantine Equations

Given ax + by = c, solutions exist iff gcd(a,b) | c. If (x₀, y₀) is a particular solution, then all solutions are:

x = x₀ + (b/g)·t,   y = y₀ − (a/g)·t   for any integer t

// Solve ax + by = c; returns false if no solution bool solve_diophantine(long long a, long long b, long long c, long long &x, long long &y, long long &g) { g = extgcd(abs(a), abs(b), x, y); if (c % g != 0) return false; x *= c / g; y *= c / g; if (a < 0) x = -x; if (b < 0) y = -y; return true; }

Chinese Remainder Theorem (CRT)

Given x ≡ r₁ (mod m₁) and x ≡ r₂ (mod m₂), we need to find x. This reduces to solving m₁·k + r₁ = m₂·l + r₂, which is a linear Diophantine equation solvable by extgcd.

// Merge x ≡ r1 (mod m1) and x ≡ r2 (mod m2) // Returns (new_r, new_m) or (-1, -1) if no solution pair<long long, long long> crt(long long r1, long long m1, long long r2, long long m2) { long long x, y; long long g = extgcd(m1, m2, x, y); if ((r2 - r1) % g != 0) return {-1, -1}; long long lcm = m1 / g * m2; long long diff = (r2 - r1) / g; long long k = (__int128)x * diff % (m2 / g); long long r = (((__int128)m1 * k + r1) % lcm + lcm) % lcm; return {r, lcm}; }
Overflow Warning: In CRT, the intermediate product m1 × k can overflow long long. Use __int128 or carefully divide before multiplying. This is a frequent source of WA in contests.

Finding the Minimum Positive Solution

For ax + by = c, if you need the smallest non-negative x:

// After finding particular solution x0: long long shift = b / g; x = ((x0 % shift) + shift) % shift; // smallest non-negative x y = (c - a * x) / b; // corresponding y
Contest tip: When a problem asks "find the minimum number of operations" or "smallest positive value," it often reduces to a linear Diophantine equation. Look for constraints of the form ax + by = target.

Typical CP Problem Patterns

  • Coin problem: Given coin denominations a and b, find if amount c is reachable → linear Diophantine equation
  • Grid movement: Can a piece at (0,0) reach (x,y) with moves (a,b) and (c,d)? → two Diophantine equations
  • Modular systems: Solving simultaneous congruences → CRT with extgcd
  • Lattice point counting: Count integer solutions in a range → parametric form from extgcd

Counting Integer Solutions in a Range

A frequent competitive programming task asks: given ax + by = c with x ∈ [x_lo, x_hi] and y ∈ [y_lo, y_hi], count the number of valid integer solutions. The approach uses the parametric representation from extgcd.

// Count solutions of ax + by = c with x in [x1,x2], y in [y1,y2] long long count_solutions(long long a, long long b, long long c, long long x1, long long x2, long long y1, long long y2) { long long x0, y0, g; if (!solve_diophantine(a, b, c, x0, y0, g)) return 0; // General solution: x = x0 + (b/g)*t, y = y0 - (a/g)*t long long dx = b / g, dy = a / g; // Find range of t from x-constraints long long t_lo_x, t_hi_x; if (dx > 0) { t_lo_x = ceil_div(x1 - x0, dx); t_hi_x = floor_div(x2 - x0, dx); } else { t_lo_x = ceil_div(x2 - x0, dx); t_hi_x = floor_div(x1 - x0, dx); } // Similarly for y-constraints, then intersect ranges long long t_lo = max(t_lo_x, t_lo_y); long long t_hi = min(t_hi_x, t_hi_y); return max(0LL, t_hi - t_lo + 1); }
Implementation detail: The ceil_div and floor_div functions must handle negative dividends correctly. In C++, integer division truncates toward zero, so you need custom functions for correct ceiling/floor division with negative values.
