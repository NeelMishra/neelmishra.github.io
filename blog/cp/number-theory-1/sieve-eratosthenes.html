Sieve of Eratosthenes & Variants

CP Math SeriesNumber Theory

Classic Sieve of Eratosthenes

The Sieve of Eratosthenes is the oldest and most intuitive algorithm for finding all primes up to a limit N. The idea: start with all numbers marked as prime, then for each prime p, mark all multiples of p (starting from p²) as composite.

Algorithm:

  1. Create a boolean array is_prime[0..N], initialized to true.
  2. Set is_prime[0] = is_prime[1] = false.
  3. For each i from 2 to √N: if is_prime[i], mark all multiples j = i², i²+i, i²+2i, … as composite.
  4. All indices still marked true are prime.
const int MAXN = 10000001; bool is_prime[MAXN]; vector<int> primes; void sieve() { fill(is_prime, is_prime + MAXN, true); is_prime[0] = is_prime[1] = false; for (int i = 2; (long long)i * i < MAXN; i++) { if (is_prime[i]) { for (int j = i * i; j < MAXN; j += i) is_prime[j] = false; } } for (int i = 2; i < MAXN; i++) if (is_prime[i]) primes.push_back(i); }
Why start from i²? Every composite number less than i² that has i as a factor also has a smaller prime factor, and was already marked in an earlier iteration. Starting at i² avoids redundant work.

Memory optimization: Use a bitset instead of a bool array to reduce memory by 8×. For N = 108, a bool array uses ~100 MB, while a bitset uses ~12.5 MB.

#include <bitset> const int MAXN = 100000001; bitset<MAXN> is_prime; void sieveBitset() { is_prime.set(); // set all to 1 is_prime[0] = is_prime[1] = 0; for (int i = 2; (long long)i * i < MAXN; i++) if (is_prime[i]) for (int j = i * i; j < MAXN; j += i) is_prime[j] = 0; }

Complexity: O(n log log n)

The number of operations performed by the sieve is:

∑p prime, p≤N N/p  =  N · ∑p prime, p≤N 1/p

By Mertens’ theorem, the sum of reciprocals of primes up to N is:

∑p prime, p≤N 1/p  =  ln(ln N) + M + o(1)

where M ≈ 0.2615 is the Meissel–Mertens constant. Therefore, the total work is O(n log log n).

Intuition: log log n grows extremely slowly. For n = 109, log log n ≈ 3. In practice, the sieve is nearly linear and runs in well under a second for n = 107.

Space complexity: O(n) for the boolean/bitset array. This is the main bottleneck for very large N, which motivates the segmented sieve.

Pitfall: When N > ~5 × 107, a plain bool array may cause memory limit exceeded (MLE) on many judges. Switch to bitset or segmented sieve.

Linear Sieve (Euler’s Sieve)

The linear sieve guarantees that each composite number is crossed off exactly once, by its smallest prime factor. This gives true O(n) time complexity.

Key insight: For each number i, multiply it by every prime p in our list, but stop as soon as p divides i. This ensures that the number i · p is always eliminated by its smallest prime factor p.

const int MAXN = 10000001; int spf[MAXN]; // smallest prime factor vector<int> primes; void linearSieve() { for (int i = 2; i < MAXN; i++) { if (spf[i] == 0) { // i is prime spf[i] = i; primes.push_back(i); } for (int j = 0; j < primes.size() && primes[j] <= spf[i] && (long long)i * primes[j] < MAXN; j++) { spf[i * primes[j]] = primes[j]; } } }

Why is this O(n)? Each composite number m has a unique representation m = spf(m) · (m / spf(m)). The inner loop for i = m / spf(m) will set spf[m] exactly once, because we stop when primes[j] > spf[i]. So each composite is touched exactly once.

Bonus: The linear sieve also computes the smallest prime factor (SPF) for every number — which is exactly what we need for O(log n) factorization (see Prime Factorization).

Practical note: Despite being O(n) vs O(n log log n), the linear sieve is not always faster in practice due to less cache-friendly memory access. For pure prime generation, the classic sieve often wins. The linear sieve shines when you also need SPF values.

Segmented Sieve

When N is very large (e.g., N = 1012), we cannot allocate an array of size N. The segmented sieve processes the range [2, N] in blocks of size Δ ≤ √N, using only O(√N) memory.

Algorithm:

  1. Sieve all primes up to √N using the classic sieve. Store them.
  2. Process the range [2, N] in segments [lo, lo+Δ).
  3. For each segment, create a local boolean array of size Δ. For each small prime p, mark its multiples in the segment as composite.
  4. Remaining unmarked values in the segment are prime.
// Count primes in [L, R] using segmented sieve long long segmentedCount(long long L, long long R) { int lim = (int)sqrt((double)R) + 1; vector<bool> small_sieve(lim + 1, true); small_sieve[0] = small_sieve[1] = false; for (int i = 2; i * i <= lim; i++) if (small_sieve[i]) for (int j = i * i; j <= lim; j += i) small_sieve[j] = false; vector<int> small_primes; for (int i = 2; i <= lim; i++) if (small_sieve[i]) small_primes.push_back(i); long long count = 0; const int BLOCK = 1 << 18; // ~256K, fits in L1 cache vector<bool> block(BLOCK); for (long long lo = L; lo <= R; lo += BLOCK) { long long hi = min(lo + BLOCK - 1, R); fill(block.begin(), block.end(), true); for (int p : small_primes) { long long start = max((long long)p * p, (lo + p - 1) / p * p); for (long long j = start; j <= hi; j += p) block[j - lo] = false; } for (long long i = max(lo, 2LL); i <= hi; i++) if (block[i - lo]) count++; } return count; }
Pitfall: Computing the first multiple of p in [lo, hi] is tricky. Use ((lo + p - 1) / p) * p to round up to the nearest multiple. Also ensure you skip p itself if p ≥ lo (it’s prime, not composite).

Complexity: Time is the same O(n log log n) as the classic sieve, but memory drops to O(√n + Δ). Choosing Δ to fit in L1 cache (∼256 KB) gives excellent real-world performance.

Smallest Prime Factor Sieve

A variant of the sieve that stores the smallest prime factor (SPF) for every number. This enables O(log n) factorization by repeatedly dividing by the SPF.

const int MAXN = 10000001; int spf[MAXN]; void computeSPF() { for (int i = 2; i < MAXN; i++) { if (spf[i] == 0) { // i is prime for (int j = i; j < MAXN; j += i) { if (spf[j] == 0) spf[j] = i; } } } } // Factorize n in O(log n) using precomputed SPF vector<pair<int,int>> factorize(int n) { vector<pair<int,int>> factors; while (n > 1) { int p = spf[n], cnt = 0; while (n % p == 0) { n /= p; cnt++; } factors.push_back({p, cnt}); } return factors; }

The SPF sieve has the same O(n log log n) complexity as the classic sieve. However, it stores integers instead of booleans, using 4× more memory per element. For N = 107, this is ~40 MB, which is usually acceptable.

Tip: The SPF sieve is arguably the most useful sieve variant in CP. It answers “is n prime?” (check if spf[n] == n), gives you factorization in O(log n), and lets you compute any multiplicative function quickly.

Sieve Comparison

Classic Sieve

Time: O(n log log n)

Space: O(n) bits with bitset

Output: Boolean prime array

Best for: Pure prime generation up to ~108. Fastest in practice due to cache-friendly access.

Linear Sieve (Euler)

Time: O(n)

Space: O(n) integers

Output: Prime list + SPF array

Best for: When you need SPF values and true linear guarantee. Slightly slower in practice for pure generation.

Segmented Sieve

Time: O(n log log n)

Space: O(√n)

Output: Primes in range [L, R]

Best for: Huge ranges (n > 108) or counting primes in [L, R] where R ≤ 1012.

SPF Sieve

Time: O(n log log n)

Space: O(n) integers

Output: SPF array for O(log n) factorization

Best for: Many factorization queries (Q up to 106) with n ≤ 107. The workhorse of CP number theory.

CP Applications

Sieves are foundational in competitive programming. Here are the most common patterns:

Pattern 1: Prime counting / generation. The classic problem: “How many primes are there up to N?” For N ≤ 107, use the classic sieve. For larger N, use a segmented sieve. The prime-counting function π(n) ∼ n / ln(n) by the Prime Number Theorem.

Pattern 2: Sieve-based multiplicative function computation. The Euler totient, Möbius function, divisor count, and divisor sum can all be computed sieve-style. The linear sieve is ideal for this because it identifies each number’s SPF and highest prime-power component.

// Möbius function via linear sieve int mu[MAXN]; void mobiusSieve() { mu[1] = 1; for (int i = 2; i < MAXN; i++) { if (spf[i] == i) { // i is prime mu[i] = -1; } else { int p = spf[i]; if ((i / p) % p == 0) mu[i] = 0; // p^2 | i else mu[i] = -mu[i / p]; } } }

Pattern 3: Goldbach-type problems. “Express N as a sum of two primes.” Sieve primes up to N, then for each prime p check if N − p is also prime. O(n log log n) precompute, O(n / ln n) per query.

Pattern 4: Prime gaps and twin primes. Find consecutive primes with gap exactly k. Generate primes via sieve, iterate through pairs. The segmented sieve handles ranges where R − L is manageable but L is large.

Contest tip: If a problem says N ≤ 107 and has many queries, your first thought should be “can I sieve something?” Sieve-based precomputation turns O(Q · √n) solutions into O(n log log n + Q) solutions.
Pitfall: Some judges have tight memory limits (64 MB or less). A bool sieve up to 107 is ~10 MB, fine. An int SPF sieve up to 107 is ~40 MB. An int sieve to 108 is ~400 MB — use bitset or segmented approaches instead.

Pattern 5: Sieve on function values. Compute the smallest prime factor, number of distinct prime factors, sum of prime factors, etc., for all numbers in a range. These are all sieve-like: iterate over primes, update all multiples. Time is O(n log log n) for one pass per prime.

// Number of distinct prime factors for all n ≤ MAXN int omega[MAXN]; // omega[n] = number of distinct prime factors of n void omegaSieve() { for (int i = 2; i < MAXN; i++) { if (omega[i] == 0) { // i is prime for (int j = i; j < MAXN; j += i) omega[j]++; } } }

Mastering these sieve variants gives you a powerful toolkit for a wide range of number theory and combinatorics problems in competitive programming.
