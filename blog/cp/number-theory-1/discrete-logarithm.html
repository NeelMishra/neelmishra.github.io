Discrete Logarithm: BSGS & Pohlig-Hellman

CP Math Series — Post 18Number Theory

The discrete logarithm problem is the inverse of modular exponentiation: given g, h, and p, find x such that gx ≡ h (mod p). While modular exponentiation is easy (O(log p)), the reverse is believed to be hard in general—the foundation of Diffie-Hellman and ElGamal cryptography. In competitive programming, however, the moduli are small enough that elegant algorithms solve it efficiently.

Hardness warning: For general groups of order n, no polynomial-time (in log n) algorithm is known. The best generic algorithms run in O(√n) time. For CP problems the modulus is at most ~1012, making √n ≈ 106—perfectly tractable.

The Discrete Logarithm Problem

Problem statement: Given a prime p, a generator g of (ℤ/pℤ)×, and a target h with gcd(h, p) = 1, find the unique integer x ∈ [0, p−2] such that gx ≡ h (mod p).

More generally, g need not be a primitive root. If ordp(g) = d, then gx ≡ h has a solution if and only if h(p−1)/d ≡ 1 (mod p), and solutions are unique modulo d.

The brute-force approach simply iterates x = 0, 1, 2, … checking each power. This takes O(p) time—far too slow for large primes.

Baby-Step Giant-Step

The Baby-step Giant-step (BSGS) algorithm, due to Daniel Shanks (1971), is a classic time–space trade-off. It solves the discrete log in O(√n) time and space, where n is the group order.

The Core Idea

Write x = i · m + j, where m = ⌈√n⌉ and 0 ≤ j < m, 0 ≤ i < m. Then:

gim + j ≡ h  ⇒  gj ≡ h · (g−m)i

Baby steps: Compute gj for j = 0, 1, …, m−1 and store them in a hash table mapping value → j.

Giant steps: Compute h · (g−m)i for i = 0, 1, …, m−1 and look up each value in the hash table. A match gives x = i · m + j.

Correctness & Complexity Proof

Theorem. BSGS correctly finds the discrete logarithm in O(√n) time and space.

Proof of correctness:

Since x ∈ [0, n−1], writing x = im + j with m = ⌈√n⌉ gives 0 ≤ j < m and 0 ≤ i ≤ m−1 (since im + j ≤ (m−1)m + (m−1) = m2 − 1 ≥ n−1). Such a decomposition always exists.

From gx = gim+j ≡ h, we get gj ≡ h · g−im. The baby-step table contains gj for the correct j, and the giant-step loop computes h · g−im for the correct i. Thus a match is guaranteed. ▮

Proof of complexity:

Baby-step phase: m multiplications and m hash insertions → O(m) = O(√n). Giant-step phase: at most m multiplications and m hash lookups → O(m) = O(√n). Space: O(m) = O(√n) for the hash table. ▮

Tip: In practice, use unordered_map for O(1) average-case lookups. If you hit adversarial hash collisions on Codeforces, use a custom hash or gp_hash_table from the PB-DS library.

Full C++ Implementation

#include <bits/stdc++.h> using namespace std; typedef long long ll; ll power_mod(ll b, ll e, ll m) { ll r = 1; b %= m; while (e > 0) { if (e & 1) r = (__int128)r * b % m; b = (__int128)b * b % m; e >>= 1; } return r; } // Baby-step Giant-step: find x such that g^x ≡ h (mod p) // Returns -1 if no solution exists ll bsgs(ll g, ll h, ll p) { if (h % p == 1) return 0; ll n = p - 1; // group order (p prime) ll m = (ll)ceil(sqrt((double)n)); // Baby steps: store g^j -> j unordered_map<ll, ll> table; ll cur = 1; for (ll j = 0; j < m; j++) { table[cur] = j; cur = (__int128)cur * g % p; } // Giant step factor: g^{-m} mod p ll giant = power_mod(g, p - 1 - m, p); // Giant steps: check h * giant^i cur = h; for (ll i = 0; i < m; i++) { auto it = table.find(cur); if (it != table.end()) { ll x = i * m + it->second; if (x > 0) return x; } cur = (__int128)cur * giant % p; } return -1; // no solution } int main() { // Example: find x such that 3^x ≡ 13 (mod 17) ll g = 3, h = 13, p = 17; ll x = bsgs(g, h, p); if (x != -1) cout << "x = " << x << "\n"; else cout << "No solution\n"; // Verify: 3^4 = 81 ≡ 13 (mod 17) ✓ }
Pitfall: When computing g−m, use power_mod(g, p - 1 - m, p) instead of computing the inverse separately. This avoids a common off-by-one when m divides p−1 evenly. Also ensure m is large enough: use ceil(sqrt(n)), not truncating integer sqrt.

Pohlig-Hellman Algorithm

When the group order n = p−1 has only small prime factors (i.e., n is smooth), the Pohlig-Hellman algorithm exploits this structure to solve the discrete log much faster than BSGS alone.

Key Idea

Factor n = q1e1 · … · qkek. For each prime power qiei, reduce the discrete log to a subproblem of order qiei, solve it (using BSGS on groups of order qi), then combine via CRT.

Algorithm Steps

  1. Factorise n = p−1.
  2. For each prime power qe dividing n:
    • Set g′ = gn/qe and h′ = hn/qe.
    • Find xi ≡ x (mod qe) by “lifting” digit by digit: extract digits d0, d1, … of xi in base q, each via a BSGS call on a group of order q.
  3. Combine results using CRT: x ≡ xi (mod qiei) for all i.

Complexity: O(∑i ei(√qi + log n)). When all prime factors are small (say < 106), this is dramatically faster than plain BSGS on the full group.

// Pohlig-Hellman: discrete log when p-1 is smooth ll pohlig_hellman(ll g, ll h, ll p) { ll n = p - 1; auto factors = factorise(n); vector<ll> rems, mods; for (auto& [q, e] : factors) { ll qe = 1; for (int i = 0; i < e; i++) qe *= q; ll gamma = power_mod(g, n / q, p); // order-q element ll x_i = 0, q_pow = 1; for (int k = 0; k < e; k++) { // h_k = (h * g^{-x_i})^{n/(q^{k+1})} ll h_k = power_mod( (__int128)h * power_mod(g, n - x_i % n, p) % p, n / (q_pow * q), p); ll d_k = bsgs(gamma, h_k, p); // solve in order-q group x_i += d_k * q_pow; q_pow *= q; } rems.push_back(x_i); mods.push_back(qe); } return crt(rems, mods); // Chinese Remainder Theorem }

Algorithm Comparison

Brute Force

Time: O(n)

Space: O(1)

When to use: n < 106, or quick sanity check.

Pros: Zero setup, trivial to code.

Cons: Far too slow for large moduli.

Baby-Step Giant-Step

Time: O(√n)

Space: O(√n)

When to use: n ≤ 1012, general purpose.

Pros: Simple, general, no factoring needed.

Cons: Memory-heavy for very large n.

Pohlig-Hellman

Time: O(∑ ei √qi)

Space: O(max √qi)

When to use: n is smooth (all prime factors small).

Pros: Exploits group structure, can handle huge n.

Cons: Requires factoring n; useless if n has large prime factor.

Index Calculus (beyond CP)

Time: sub-exponential

Space: sub-exponential

When to use: Cryptographic-size moduli (2048+ bits).

Pros: Fastest known for ℤ/pℤ.

Cons: Complex, not useful in CP.

CP Applications

Pattern 1: Solve ax ≡ b (mod p)

Direct BSGS application. Common in problems asking “after how many steps does state a reach state b under multiplication by a constant?”

Pattern 2: k-th Root Modulo p

To find x such that xk ≡ a (mod p): let g be a primitive root, write x = gt. Then gkt ≡ a, so kt ≡ logg(a) (mod p−1). Use BSGS to find logg(a), then solve the linear congruence.

Pattern 3: Functional Iteration

Some problems define f(x) = ax + b mod p and ask for the smallest n with fn(x0) = y. The closed form is anx0 + b(an−1)/(a−1) ≡ y, which rearranges to a discrete log.

Contest tip: Always check the edge case g = 0 or h = 0. Also handle the case a = 1 separately in functional iteration problems (it degenerates to a linear equation rather than a discrete log).

Example: CF 1106F

A classic problem where you need to find the discrete log of a matrix entry. The key insight is that after reducing the problem modulo a prime, BSGS gives the answer in O(√p) time. Combined with the primitive root from Post 17, this forms a powerful toolkit for cyclic-structure problems.

Common mistake: Forgetting that BSGS finds the smallest non-negative solution. If the problem asks for the smallest positive solution, you may need to add the group order when the result is 0. Similarly, if solutions should be in a specific range, reduce modulo ord(g).
