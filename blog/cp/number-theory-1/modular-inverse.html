Modular Inverse — Fermat, ExtGCD & Precomputation

CP Math SeriesNumber Theory

Division doesn't directly exist in modular arithmetic. Instead, dividing by a is equivalent to multiplying by a's modular inverse. Knowing how to compute modular inverses—efficiently and correctly—is essential for combinatorics, probability, and virtually any CP problem that says "output the answer modulo 10⁹ + 7."

What Is Modular Inverse?

The modular inverse of a (modulo m) is an integer x such that:

a · x ≡ 1 (mod m)

We write x = a⁻¹ mod m. It exists if and only if gcd(a, m) = 1 (a and m are coprime).

Intuition: Just as 5 × (1/5) = 1 in real numbers, in modular arithmetic 5 × 5⁻¹ ≡ 1 (mod m). For m = 7, we have 5⁻¹ = 3 since 5 × 3 = 15 ≡ 1 (mod 7).

Existence condition: By Bézout's identity, ax + my = gcd(a, m). If gcd(a, m) = 1, then ax + my = 1, meaning ax ≡ 1 (mod m). If gcd(a, m) > 1, no solution exists because ax mod m is always a multiple of gcd(a, m), which can never equal 1.

Pitfall: If m is prime, every non-zero element has an inverse. But if m is composite (e.g., m = 12), then elements like 2, 3, 4, 6, 8, 9, 10 do NOT have inverses mod 12.

Fermat's Little Theorem Method

Fermat's Little Theorem

If p is prime and gcd(a, p) = 1, then: a^(p−1) ≡ 1 (mod p)

Therefore: a · a^(p−2) ≡ 1 (mod p), so a⁻¹ ≡ a^(p−2) (mod p).

This gives us a simple formula: compute a^(p−2) mod p using fast exponentiation.

long long power(long long base, long long exp, long long mod) { long long result = 1; base %= mod; while (exp > 0) { if (exp & 1) result = result * base % mod; base = base * base % mod; exp >>= 1; } return result; } // Modular inverse using Fermat's little theorem (p must be prime) long long mod_inv_fermat(long long a, long long p) { return power(a, p - 2, p); }

Complexity: O(log p) for a single inverse. Simple and elegant, but requires p to be prime.

When to use: Fermat's method is the go-to when your modulus is a prime (the overwhelmingly common case in CP). It's one line of code on top of a binary exponentiation function you'll already have.

Extended GCD Method

As discussed in the Extended Euclidean post, we solve ax + my = 1 to find x = a⁻¹ mod m. This method works for any modulus m, not just primes.

long long extgcd(long long a, long long b, long long &x, long long &y) { if (b == 0) { x = 1; y = 0; return a; } long long x1, y1; long long g = extgcd(b, a % b, x1, y1); x = y1; y = x1 - (a / b) * y1; return g; } // Works for any m where gcd(a, m) = 1 long long mod_inv_extgcd(long long a, long long m) { long long x, y; long long g = extgcd(a, m, x, y); if (g != 1) return -1; // no inverse return (x % m + m) % m; }

Complexity: O(log m). Slightly faster than Fermat in practice since it uses only additions and subtractions (no multiplications in the loop). However the difference is negligible.

Watch out: The extgcd can return negative x values. Always normalize with (x % m + m) % m to get the result in [0, m).

Precomputing Inverses

When you need inverses of 1, 2, 3, …, n (e.g., for computing binomial coefficients), computing each individually takes O(n log p) total. We can do better.

Method 1: Precompute Factorial Inverses

The standard approach for nCr mod p:

const int MAXN = 2000001; const long long MOD = 1e9 + 7; long long fact[MAXN], inv_fact[MAXN]; void precompute() { fact[0] = 1; for (int i = 1; i < MAXN; i++) fact[i] = fact[i-1] * i % MOD; // Compute inverse of (MAXN-1)! using Fermat inv_fact[MAXN-1] = power(fact[MAXN-1], MOD - 2, MOD); // Backfill: (i-1)!⁻¹ = i!⁻¹ × i for (int i = MAXN - 2; i >= 0; i--) inv_fact[i] = inv_fact[i+1] * (i+1) % MOD; } long long nCr(int n, int r) { if (r < 0 || r > n) return 0; return fact[n] % MOD * inv_fact[r] % MOD * inv_fact[n-r] % MOD; }
Complexity: O(n) precomputation + O(1) per query. Only one modular exponentiation is needed (for the largest factorial inverse). Everything else uses simple multiplication.

Method 2: Linear Sieve of Inverses

We can compute i⁻¹ mod p for i = 1, 2, …, n using the recurrence:

i⁻¹ ≡ −⌊p/i⌋ · (p mod i)⁻¹ (mod p)

This arises from writing p = ⌊p/i⌋·i + (p mod i), then rearranging mod p.

long long inv[MAXN]; void precompute_inv(int n, long long p) { inv[1] = 1; for (int i = 2; i <= n; i++) inv[i] = (p - (p / i) * inv[p % i] % p) % p; }

Complexity: O(n) time and space. Each inverse computed in O(1) from previously computed values.

Subtle bug: The formula inv[i] = -(p / i) * inv[p % i] % p can give a negative intermediate. The expression (p - (p/i) * inv[p%i] % p) % p carefully avoids this by adding p before the subtraction.

Method Comparison

Fermat's Little Theorem

  • Requirement: m is prime
  • Single inverse: O(log m)
  • n inverses: O(n log m)
  • Simplicity: Very easy
  • Use case: One-off inverses with prime mod

Extended GCD

  • Requirement: gcd(a, m) = 1
  • Single inverse: O(log m)
  • n inverses: O(n log m)
  • Simplicity: Moderate
  • Use case: Non-prime mod, CRT steps

Factorial Inverse Precomputation

  • Requirement: m is prime
  • Setup: O(n) + one O(log m) call
  • Per query: O(1) for nCr
  • Simplicity: Moderate
  • Use case: Many nCr queries

Linear Sieve of Inverses

  • Requirement: m is prime, i < m
  • Setup: O(n)
  • Per query: O(1)
  • Simplicity: Easy once derived
  • Use case: Need i⁻¹ for all i ≤ n

CP Applications — nCr mod p

Computing nCr mod p (p prime, n < p)

This is the most common application. Using the precomputed factorials and inverse factorials:

// After calling precompute(): long long nCr(int n, int r) { if (r < 0 || r > n) return 0; return fact[n] % MOD * inv_fact[r] % MOD * inv_fact[n - r] % MOD; } // nPr (permutations) long long nPr(int n, int r) { if (r < 0 || r > n) return 0; return fact[n] % MOD * inv_fact[n - r] % MOD; }

Lucas' Theorem (n ≥ p)

When n can be larger than p, direct factorials don't work (p! ≡ 0 mod p). Lucas' theorem handles this:

C(n, r) ≡ ∏ C(nᵢ, rᵢ) (mod p), where nᵢ, rᵢ are digits of n, r in base p.

long long lucas(long long n, long long r, long long p) { if (r == 0) return 1; long long ni = n % p, ri = r % p; if (ri > ni) return 0; // C(ni, ri) = 0 if ri > ni return lucas(n / p, r / p, p) % p * nCr(ni, ri) % p; }
When to use Lucas: Only when n ≥ p. For the typical CP case where p = 10⁹ + 7 and n ≤ 10⁶ or 10⁷, standard factorial precomputation is sufficient and faster.

Catalan Numbers mod p

The nth Catalan number C_n = C(2n, n) / (n + 1). With modular inverse:

long long catalan(int n) { return nCr(2 * n, n) % MOD * mod_inv_fermat(n + 1, MOD) % MOD; } // Or equivalently using precomputed inverses: long long catalan_v2(int n) { return fact[2*n] % MOD * inv_fact[n] % MOD * inv_fact[n+1] % MOD; }

Derangements mod p

D(n) = n! · Σ (-1)^k / k! for k = 0 to n. Using modular inverse of factorials:

long long derangement(int n) { long long ans = 0; for (int k = 0; k <= n; k++) { if (k % 2 == 0) ans = (ans + inv_fact[k]) % MOD; else ans = (ans - inv_fact[k] + MOD) % MOD; } return ans % MOD * fact[n] % MOD; }
Contest checklist: (1) Is the mod prime? → Use Fermat. (2) Need many nCr values? → Precompute factorials. (3) Mod is not prime? → Use ExtGCD. (4) n ≥ p? → Use Lucas. Following this decision tree avoids most WA from using the wrong method.
