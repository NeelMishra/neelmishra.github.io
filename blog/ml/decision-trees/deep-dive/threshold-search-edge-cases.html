    Decision Tree Threshold Search & Edge Cases

    July 2026 · 18 min read

    When a decision tree searches for a numeric split, it does not try every possible number between the minimum and maximum feature value. That would be impossible for real-valued features and disastrous when the range is huge. Instead, the tree only tries thresholds that can change which training rows go left and right.

    This post is the implementation companion to Impurity & Information Gain and CART, Complexity & Pruning. It focuses on the engineering details needed when writing a tree from scratch.

    Only Gaps Between Observed Values Matter

    Suppose a node has one feature with sorted values

    $$[2,\ 2,\ 5,\ 11,\ 1000000000].$$

    It does not matter that the maximum is enormous. The split can only change when the threshold passes an observed training value. With a test like $x_j < t$, every threshold between $5$ and $11$ sends exactly the same rows left and right. So the implementation can choose one representative threshold in that open interval, usually the midpoint.

    The candidate thresholds are therefore:

    $$\frac{2+5}{2}=3.5,\qquad \frac{5+11}{2}=8,\qquad \frac{11+1000000000}{2}=500000005.5.$$

    There are only three candidates, not one billion candidates. The cost depends on the number of rows at the node, not on the numeric range of the feature.

    Key implementation rule. Never iterate through numeric values from min to max. Sort the observed values and try only boundaries between adjacent distinct values.

    ▶ Sorted Sweep over Observed Values Only

    The feature range is huge, but only adjacent observed gaps create different partitions. Duplicate values are moved together before a candidate threshold is scored.

    The Sorted Sweep

    For one feature, sort the row indices by that feature. Start with all rows in the right child and no rows in the left child. Sweep from left to right. Each time rows cross the boundary, update the class counts, then score the candidate if the next feature value is different.

    sort rows by feature f
leftCounts = all zeros
rightCounts = parent class counts

for each group of equal feature value v:
    move the whole group from rightCounts to leftCounts

    if there is no next distinct value:
        stop, because a split after the maximum would leave right empty

    threshold = midpoint(v, nextValue)
    if both sides satisfy minLeaf:
        score weighted impurity(leftCounts, rightCounts)
        keep the best threshold

    Moving a whole equal-value group is the cleanest way to reason about ties. The common shorter loop moves one row at a time and skips scoring while the next value is equal. Both approaches produce the same candidates: score only after the final copy of a duplicated value has crossed.

    Duplicates and Conflicting Labels

    If several rows have the same feature value, no threshold on that feature can split those rows apart. If two rows both have $x_j=2$ but different labels, this feature alone cannot separate them at this node. A correct implementation must not invent a threshold like $2.0001$ between identical values, because both rows still go to the same side for any threshold.

    This is why the split search checks adjacent sorted values:

    if value[current] == value[next]:
    continue  # no valid threshold inside a tie

    Large Ranges and Huge Values

    Large feature ranges do not change the number of candidate thresholds. A feature with values $1$, $2$, and $10^{12}$ has only two candidate gaps. The only special concern is numeric safety when computing the midpoint. Prefer

    mid = low + (high - low) / 2

    instead of

    mid = (low + high) / 2

    The safer form avoids overflow for integer-like values and reduces avoidable numeric trouble for very large floating-point values. Another robust option is to store the split as "feature value is less than the boundary between sorted positions" internally, but the midpoint is simpler for a first implementation.

    Edge Cases Checklist

    Case What can go wrong Implementation choice
    All values identical No threshold can create two nonempty children. Skip the feature at this node.
    Duplicate values with mixed labels The feature cannot separate those rows. Move duplicates together and score only between distinct values.
    Huge numeric range Naive iteration from min to max is impossible. Use sorted observed gaps only. Complexity depends on row count.
    Midpoint overflow (low + high) / 2 can overflow for large values. Use low + (high - low) / 2.
    NaN or missing values Sorting and comparisons become ambiguous. For a first implementation, clean or impute before training. Later, learn a default missing branch.
    Infinite values Midpoints and comparisons can become invalid. Reject or cap non-finite values during preprocessing.
    Too-small child A split may isolate one row and overfit. Require minSamplesLeaf on both sides before scoring.
    Tiny impurity improvement Floating-point noise can look like a real gain. Require gain > bestGain + eps or a minimum gain threshold.
    Categorical feature encoded as integers Sorting category IDs invents a fake order. Use one-hot encoding first, or implement category-subset splits separately.

    Classification versus Regression

    The threshold candidates are the same for classification and regression. What changes is the bookkeeping used to score a candidate.

    • Classification: maintain class counts on the left and right, then compute Gini or entropy.
    • Regression: maintain count, sum, and sum of squares on each side, then compute squared-error reduction.

    The sweep shape stays identical: sort once, move rows across once, score only between distinct adjacent values.

    Practical C++ Shape

    A robust first implementation can follow this shape:

    bestGain = 0
bestFeature = -1

for each feature f:
    sort row indices by X[row][f]
    initialize running left/right statistics

    pos = 0
    while pos < n:
        value = X[ord[pos]][f]
        move every row with this same value from right to left

        if pos == n:
            break
        nextValue = X[ord[pos]][f]
        if leftSize < minLeaf or rightSize < minLeaf:
            continue

        threshold = value + (nextValue - value) / 2
        gain = parentImpurity - weightedChildImpurity
        if gain > bestGain + eps:
            save feature, threshold, gain
    The mental model. A threshold is not a number guessed from the feature's range. It is a boundary between two neighboring observed training values. That one sentence prevents most implementation mistakes.
