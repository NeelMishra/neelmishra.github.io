    Bagging & Random Forests

    June 2026 · 17 min read

    Part 1 showed why a single unrestricted decision tree is a high-variance learner. It can fit the training set almost perfectly, but small changes in the sampled rows can change early splits, leaf shapes, and the final prediction. Bagging reduces that variance by averaging many trees trained on many slightly different versions of the data. Instead of asking one deep tree to be stable, the method asks a committee of deep trees to cancel out one another's sample-specific mistakes.

    The central idea is deliberately simple: keep the individual trees flexible, make their errors as independent as possible, then combine their predictions. The resulting model is still built from ordinary decision trees, so it keeps the non-linear, interaction-friendly behavior of trees, but the final predictor is much smoother than any one member of the ensemble.

    Bagging: Bootstrap Aggregating

    Bagging is short for bootstrap aggregating. Given a training set with $N$ rows, a bootstrap sample is produced by drawing $N$ rows with replacement from those same $N$ training rows. Because the draw is with replacement, some rows appear multiple times and some rows do not appear at all. One tree is trained on each bootstrap sample.

    For classification, each tree votes for a class and the ensemble returns the majority vote. For regression, each tree outputs a number and the ensemble returns the average. In both cases, the aggregation step makes the final prediction less sensitive to any one strange training row, any one unlucky split, or any one overgrown tree.

    The cleanest variance story starts with an idealization. If $B$ estimators are independent and identically distributed, each with variance $\\sigma^2$, then their average has variance

    $$\\operatorname{Var}\\left(\\frac{1}{B}\\sum_{b=1}^{B} T_b\\right)=\\frac{\\sigma^2}{B}.$$

    That is the dream: doubling the number of trees roughly halves the variance. Real bagged trees are not independent, because they are all trained from the same original data distribution and often discover similar strong splits. If every pair of trees has positive correlation $\\rho$, the variance of their average becomes

    $$\\rho\\sigma^2 + \\frac{1-\\rho}{B}\\sigma^2.$$

    The first term, $\\rho\\sigma^2$, is the variance floor. No matter how many trees are added, the correlated part of their behavior does not average away. The second term, $\\frac{1-\\rho}{B}\\sigma^2$, shrinks as $B$ grows, but only the uncorrelated portion benefits from the number of trees. This is why bagging needs two ingredients: many trees, and trees whose mistakes are not all the same.

    Bootstrap sampling supplies some diversity because each tree sees a different multiset of rows. A row that is duplicated may exert extra influence on one tree; a row that is absent cannot influence that tree at all. However, if a dataset has one or two dominant features, many bootstrapped trees may still choose those same features near the top, making their predictions highly correlated.

    Random Forests: Decorrelating the Trees

    A random forest adds one decisive source of randomness to bagging. At each split, the tree is not allowed to examine every feature. Instead, it considers only a random subset of mtry features and chooses the best split inside that subset. For classification, a common default is $\\texttt{mtry} = \\lfloor\\sqrt{D}\\rfloor$, where $D$ is the total number of features.

    This feature subsampling happens at each split, not once per tree. A strong feature might be available at the root of one tree, missing at the root of another tree, available again lower down, and absent from a different branch. This prevents every tree from keying on the same one or two strong predictors. Weaker but still useful features get chances to shape some trees, which lowers the tree-to-tree correlation $\\rho$.

    Returning to the variance formula, lowering $\\rho$ lowers both the long-run floor $\\rho\\sigma^2$ and increases the portion that can be averaged away. This is the one idea that turns bagged trees into a random forest: grow many bootstrap trees, but decorrelate their split choices by hiding most features from each split.

    ▶ Bootstrap, Train, Vote

    Each step draws one bootstrap sample, marks the out-of-bag rows for that tree, and adds the tree's vote on a query point whose true class is A.

    A single tree might be wrong on the query point, but the committee corrects it. In the example above, one tree votes B, four trees vote A, and the majority recovers the true class. This is variance reduction in action: unstable individual decisions become a stable aggregate decision.

    Out-of-Bag Error: Free Validation

    Bootstrap sampling also creates a useful validation signal. In one bootstrap sample, the probability that a particular training row is not selected on a single draw is $1 - 1/N$. The bootstrap makes $N$ independent draws with replacement, so the probability that the row is never selected is

    $$(1 - 1/N)^N \\to e^{-1} \\approx 0.37.$$

    So each bootstrap sample omits about $37\%$ of the original rows. These omitted rows are called out-of-bag rows for that tree. Equivalently, each training row is out-of-bag for about $37\%$ of the trees in a large forest.

    To estimate the error for a row, collect only the trees for which that row was out-of-bag, predict the row using that smaller committee, and compare the prediction with the known label. Repeating this for all rows gives the out-of-bag error. It acts like a built-in test-error estimate because the row being evaluated was not used to train the trees that vote on it. No separate held-out set is required.

    Out-of-bag error is especially convenient when data is limited. The full training set can be used to grow the forest, while the unused rows inside each bootstrap sample provide a nearly free validation mechanism. It is not magic, and it can still differ from the final test error, but it is usually a much better guide than training accuracy.

    In Practice

    The companion C++ experiment shows the expected pattern. A single deep tree overfits, reaching about $100\%$ training accuracy but only about $70\%$ test accuracy. A 120-tree forest, using mtry = 3 out of 10 features, reaches about $78.7\%$ test accuracy with an out-of-bag estimate near $84.7\%$. The full walkthrough is available in From Scratch · C++ · Random Forest.

    Bagging versus boosting. Bagging reduces variance by training independent deep trees in parallel, then averaging or voting over them. Boosting, covered in Part 6, instead trains trees sequentially, with each new model focusing on examples the previous models handled poorly.

    For the parallel bagging recipe, the most important knobs are the number of trees, tree depth or leaf-size constraints, and mtry. More trees usually improve stability until the uncorrelated variance has mostly averaged away. Smaller mtry can reduce correlation, but if it is too small, individual trees may become weak because good split candidates are often hidden. Random forests work well because this tradeoff is forgiving across many tabular problems.
