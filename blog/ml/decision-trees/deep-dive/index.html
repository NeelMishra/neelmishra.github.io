    Decision Trees: Foundations

    June 2026 · 18 min read

    This is the typeset, illustrated companion to the Handwritten Notes series. It covers the same ground, the decision-tree family from first principles, but every idea is drawn out in full and paired with an animation you can step through. This first part answers four questions: what a decision tree is, what it looks like as a partition of feature space, how a prediction is produced, and how the tree is grown. The companion C++ implementation lives in From Scratch · C++.

    What a Decision Tree Is

    A decision tree is a model made of three kinds of nodes arranged in a hierarchy:

    • A single root node, where every prediction begins.
    • Decision (internal) nodes, each asking one yes/no question about a single feature, for example "is $x_1 < 5$?". The answer selects which child to descend into.
    • Leaf nodes, each holding a final answer: a class label for classification, or a number for regression.

    It is a supervised model, learned from labelled examples, and the same structure serves both classification and regression. The only things that differ between the two are what the leaves store (a label versus a number) and how the quality of a split is scored. Because each question touches just one feature and compares it against a threshold, every decision is cheap to evaluate and easy to read off as a rule.

    The Geometry: Axis-Aligned Partitions

    There is a second, more revealing way to see a tree. Each internal node tests one feature against a threshold, so it draws a line perpendicular to that feature's axis, an axis-aligned cut. The root splits the whole feature space into two half-planes; each child splits its half again; and so on. A tree with $L$ leaves therefore carves the feature space into $L$ rectangular boxes, and every point inside a given box receives the same prediction, the label of the leaf that owns it.

    Growing the tree and carving the plane are the same process seen from two angles. The animation below makes that explicit: at each step it adds one split, drawing the cut on the left and the corresponding node on the right at the same time. The data here is a deliberately awkward "checkerboard" that a single straight line cannot separate, yet three axis-aligned cuts isolate all four groups perfectly.

    ▶ Growing a Tree = Carving the Plane

    Left: a 2-D feature space with two classes (green = A, orange = B). Right: the tree being built. Each step adds one axis-aligned split and the matching node, until every region (leaf) holds a single class.

    Two properties fall straight out of this picture. First, decision boundaries are always made of axis-aligned steps, so a diagonal or curved boundary can only ever be approximated by a staircase of rectangles. Second, the boxes tile the space completely and without overlap, so every possible input lands in exactly one leaf and receives exactly one prediction.

    How a Prediction Is Made

    Prediction is a walk from the root to a leaf. Starting at the root, evaluate its question on the query point, follow the matching branch to a child, and repeat until a leaf is reached; the leaf's stored answer is the prediction. For the tree above, classifying the point $(x_1, x_2) = (7, 2)$ runs "$x_1 < 5$? no" then "$x_2 < 5$? yes", landing in the bottom-right box, class B. The cost of a single prediction is just the depth of the tree, one comparison per level, which is why even large trees predict almost instantly.

    Reading a tree as rules. Each root-to-leaf path is an if chain: "if $x_1 \ge 5$ and $x_2 < 5$ then B". A whole tree is just a set of such rules that happen to share prefixes. This transparency is a big part of why trees are popular when a model has to be explained.

    Training: Greedy Recursive Splitting

    The tree in the animation looks obvious in hindsight, but training has to discover those splits from data. Nearly every tree learner uses the same greedy, top-down recipe: at the current node, look at every feature and every candidate threshold, score how much each proposed split would improve things, take the single best split, and then recurse on the two children. The recursion stops when a node is already pure, too small to split, or at a depth limit, at which point it becomes a leaf.

    function BuildTree(rows):
    if stopping_rule(rows):           # pure, too small, or max depth
        return Leaf(summarize(rows))    # majority class or mean target
    best = none
    for feature f, threshold t:        # every candidate split
        score = impurity(rows) − weighted_impurity(split(rows, f, t))
        if score > best.score: best = (f, t)
    (left, right) = split(rows, best.f, best.t)
    return Node(best.f, best.t,
                BuildTree(left), BuildTree(right))   # recurse

    What "improve things" means precisely, the impurity measure and the information gain that scores a split, is the subject of Part 2. The crucial word here is greedy: the algorithm commits to the locally best split at each node and never reconsiders it. This is fast, but it is not guaranteed to produce the smallest or best possible tree; a split that looks mediocre now might have enabled a much cleaner split two levels down, and the greedy learner will never find out.

    Greedy, not optimal. Finding the globally optimal tree is NP-hard, so practical learners settle for the greedy choice. The variance this introduces, two similar datasets can yield very different trees, is not a bug to be fixed at the single-tree level; it is exactly what bagging and random forests later turn to their advantage by averaging many such trees.

    Why Unrestricted Trees Overfit

    If splitting is allowed to continue unchecked, a tree will keep carving the plane until every leaf contains a single training point. Such a tree achieves $100\%$ training accuracy by construction, but it has memorized the noise: each tiny box is drawn around one example, so a new point that falls near a boundary inherits a label set by a single, possibly mislabelled, neighbour. The boundary becomes a jagged fence that traces the training set rather than the underlying pattern.

    This is the classic bias-variance tension. A shallow tree is high-bias (too coarse to capture the structure); a fully grown tree is high-variance (exquisitely sensitive to the particular sample). The remedies come in two flavours, both covered later: limit or prune a single tree (Part 4), or keep the trees deep but average many decorrelated ones (Part 5). Either way, the foundations are now in place: a tree is a recursive, axis-aligned partition of feature space, grown greedily and predicting in a single root-to-leaf walk.
