    CART, Complexity & Pruning

    June 2026 · 19 min read

    Parts 2 and 3 settled how a split is scored. This part is about the algorithm that uses that score in practice, CART (Classification And Regression Trees), and the two issues that decide whether a tree is any good on new data: how expensive it is to find the best split, and how to stop the tree from memorizing the training set. The remedy, cost-complexity pruning, is the centerpiece, and the animation below shows it trading tree size against error in real time.

    What CART Does Differently

    CART makes three specific commitments that distinguish it from the older ID3 of Part 2:

    • Binary splits only. Every internal node has exactly two children. A categorical feature with many values is handled by asking membership questions ("is colour in $\{$red, blue$\}$?") rather than fanning out one branch per value.
    • Numeric thresholds. Continuous features are split at a threshold $x_j < t$, with $t$ chosen from the midpoints between consecutive sorted values.
    • Gini for classification, squared error for regression. Classification nodes minimize the Gini index; regression nodes minimize within-node squared error $\sum_i (y_i - \bar y)^2$ and predict the mean $\bar y$ at each leaf.

    Because every split is binary, a CART tree is a clean binary tree, which is what makes the pruning theory below so tidy.

    The Cost of Finding the Best Split

    At a node holding $n$ points and $D$ features, the learner must, in principle, evaluate every candidate threshold on every feature. The naive reading of that is expensive, but a single idea makes it cheap. For one feature, sort the $n$ points by that feature once, in $O(n \log n)$, then sweep a threshold from left to right. As each point crosses from the right child to the left child, only that one point's class count changes, so the impurity of both sides updates in $O(1)$. The whole sweep for one feature is therefore $O(n \log n)$ dominated by the sort, and across $D$ features a node costs $O(D\, n \log n)$.

    for each feature f:
    sort the n points by f                 # O(n log n)
    leftCount = 0, rightCount = full counts
    for each adjacent pair in sorted order:   # O(n) sweep
        move one point: leftCount++, rightCount--   # O(1) update
        gain = impurity(parent) − weighted(left, right)
        keep the best (f, threshold)

    Summed over the whole tree, a balanced tree of depth $O(\log n)$ does $O(D\, n \log n)$ work per level for $O(\log n)$ levels, giving roughly $O(D\, n \log^2 n)$ to build. The lesson is that the sort is the only superlinear step; the impurity bookkeeping is linear because it is incremental. The companion C++ implementation is exactly this sorted sweep.

    Why incremental counts matter. Recomputing each child's impurity from scratch at every candidate threshold would make the sweep $O(n^2)$ per feature. Keeping running class counts and editing them by one as the threshold advances is what collapses it to $O(n)$. The same incremental trick reappears in the regression tree as a running sum and sum of squares.

    Overfitting, Concretely

    Left unconstrained, CART keeps splitting until every leaf is pure, often a single training point per leaf. Training error then drops to zero, but each leaf has been drawn around one example, so its prediction is dictated by that example's noise. On fresh data the error climbs. The cure is to prefer a smaller tree, and the principled way to choose "how much smaller" is cost-complexity pruning.

    Cost-Complexity Pruning

    Grow the full tree $T_0$, then judge any subtree $T$ by a penalized error that adds a price $\alpha$ for every leaf it keeps:

    $$R_\alpha(T) = R(T) + \alpha\,|T|,$$

    where $R(T)$ is the tree's training error (misclassification or squared error) and $|T|$ is its number of leaves. With $\alpha = 0$ the full tree wins; as $\alpha$ grows, each extra leaf must justify its cost, and the minimizing subtree shrinks. Weakest-link pruning raises $\alpha$ continuously and, at each threshold where a subtree stops paying for itself, collapses the internal node whose removal raises $R$ the least per leaf eliminated. This produces a nested sequence of trees $T_0 \supset T_1 \supset \dots \supset \{\text{root}\}$, and cross-validation picks the $\alpha$ (hence the tree) with the lowest validation error.

    The animation prunes such a sequence one step at a time. The left panel collapses the weakest subtree as $\alpha$ rises; the right panel plots training error (which can only rise as leaves are removed) against test error (which falls, then rises), the U-shape whose bottom is the tree worth keeping.

    ▶ Pruning a Tree as the Penalty Rises

    Left: the tree, collapsing its weakest subtree into a leaf at each step as $\alpha$ increases. Right: training error (green) keeps falling with more leaves, test error (orange) is U-shaped. The marker tracks the current tree; the star marks the test-error minimum.

    Reading the curve right to left (the direction of pruning), the full six-leaf tree overfits: its training error is lowest but its test error is high. Collapsing the weakest subtrees first improves test error until the four-leaf tree, after which further pruning throws away real structure and both errors climb. The four-leaf tree is the one cross-validation would keep.

    Pre-pruning versus post-pruning. Stopping early (a depth cap or a minimum-samples rule) is pre-pruning: cheap, but it can halt above a split that would have paid off. Cost-complexity pruning is post-pruning: grow fully, then cut back. Post-pruning is more reliable precisely because it can see the splits that early stopping would have missed.

    Regression Trees in One Line

    Everything above carries over to regression by swapping the impurity. A regression node's "error" is the within-node sum of squared deviations from the mean, and the sweep tracks a running sum and sum of squares so each candidate split is still scored in $O(1)$ via $\sum (y_i - \bar y)^2 = \sum y_i^2 - \tfrac{1}{n}\big(\sum y_i\big)^2$. Leaves store the mean target. Pruning uses the same $R_\alpha(T) = R(T) + \alpha|T|$ with $R$ now the total squared error. The full regression treatment, with code, is in From Scratch · C++ · Regression Tree.

    Pruning tames a single tree, but it leaves performance on the table: a lone tree, however well pruned, is still high-variance. Part 5 keeps the trees deep and attacks the variance a different way, by averaging many of them. The companion handwritten treatment of this material is in Handwritten Notes.
