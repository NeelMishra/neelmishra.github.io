The Problem

In PyTorch (Python), you write self.linear = nn.Linear(32, 16) and Python's __setattr__ magic automatically registers the layer. The optimizer finds all parameters by walking the module tree.

C++ has no such magic. If you just write torch::nn::Linear layer(32, 16); as a member variable, the parent Module has no idea it exists. It will not appear in parameters(), it will not be saved by torch::save, and the optimizer will not update its weights.

register_module is the C++ equivalent of Python's automatic registration. You must call it explicitly.

What register_module Does

Under the hood, register_module does three things:

The parent Module maintains an ordered map of name → child Module. Calling register_module("Q", linear_layer) inserts the linear layer into this map with key "Q". When you later call named_modules() or iterate children, it appears.

When you call parameters() on the parent, it recursively walks all registered children and collects their parameters. The linear layer's weight tensor (and bias, if any) become part of the parent's parameter list. This is what the optimizer iterates over during training.

When you call torch::save(module, "model.pt"), it serializes all registered modules and their parameters. The name you passed ("Q") becomes the key in the saved file. Without registration, the layer's weights silently disappear on save.

The signature:

// Returns a shared_ptr to the registered module template<typename M> std::shared_ptr<M> register_module( std::string name, // unique name within this module std::shared_ptr<M> module // the child module to register );

Parameter Tracking in Action

Let us see exactly what happens with and without registration:

class MyModel : public torch::nn::Module { torch::nn::Linear layer{nullptr}; public: MyModel() { layer = register_module("layer", torch::nn::Linear(32, 16)); } }; auto model = std::make_shared<MyModel>(); // This works because layer was registered: for (auto& p : model->parameters()) { std::cout << p.sizes() << std::endl; } // Output: [16, 32] (weight matrix) // The optimizer can now find and update these weights: torch::optim::Adam optimizer(model->parameters(), 0.001);

What Happens Without It

Without register_module

class Broken : public torch::nn::Module { torch::nn::Linear layer{nullptr}; public: Broken() { // Just construct, no register layer = torch::nn::Linear(32, 16); } }; auto m = std::make_shared<Broken>(); m->parameters().size(); // Returns 0! No parameters found.

The optimizer sees nothing. Training does nothing. Weights stay at their initial random values forever.

With register_module

class Working : public torch::nn::Module { torch::nn::Linear layer{nullptr}; public: Working() { layer = register_module( "layer", torch::nn::Linear(32, 16)); } }; auto m = std::make_shared<Working>(); m->parameters().size(); // Returns 1 (the weight matrix).

Optimizer updates these weights. Model learns. Model can be saved and loaded.

This is the most common LibTorch bug. Your model trains, loss stays flat, and you spend hours debugging. The fix is always: did you call register_module?

Named Modules and Serialization

The name you pass to register_module is not just for debugging. It defines the key hierarchy for saving and loading:

// Save torch::save(model, "attention_head.pt"); // The saved file contains: // "Q.weight" -> tensor [16, 32] // "K.weight" -> tensor [16, 32] // "V.weight" -> tensor [16, 32] // Load torch::load(model, "attention_head.pt"); // Names must match exactly. If you renamed "Q" to "query", // loading will fail with a key mismatch error.
Convention: Use short, descriptive names. "Q", "K", "V" for attention projections. "fc1", "fc2" for feed-forward layers. "norm" for layer normalization. These match PyTorch naming conventions.

register_module vs register_parameter

LibTorch has two registration functions:

  • register_module: For sub-modules (Linear, Conv2d, LayerNorm, or your own nn::Module subclass). Recursively registers all parameters inside the child.
  • register_parameter: For standalone tensors that should be treated as learnable parameters (e.g., a custom embedding matrix or a scalar bias). Takes a name and a torch::Tensor.
// register_module: for nn::Module children this->Q = register_module("Q", torch::nn::Linear(32, 16)); // register_parameter: for raw tensors auto scale = register_parameter( "scale", torch::ones({16}) );

When in doubt, use register_module for anything that is a Module, and register_parameter for standalone tensors.

Common Patterns

Registering in a loop

When you have multiple identical layers (like in multi-head attention), register them in a loop:

for (int i = 0; i < num_heads; i++) { heads.push_back(register_module( "head_" + std::to_string(i), std::make_shared<SingleAttentionHead>(n_emb, head_size) )); }

Using ModuleList

LibTorch provides torch::nn::ModuleList for collections of modules, similar to PyTorch's nn.ModuleList:

torch::nn::ModuleList layers; MyModel() { for (int i = 0; i < 6; i++) { layers->push_back( torch::nn::Linear(32, 32)); } register_module("layers", layers); }
