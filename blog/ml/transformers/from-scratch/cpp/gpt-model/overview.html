The Full GPT Model in C++ (LibTorch)

ML Series Transformers > From Scratch > C++ > GPT Model

Over the previous posts we built each piece of the transformer from the ground up: single attention heads, multi-head attention, the feed-forward network, and the transformer block. This page stitches everything together into a complete GPT model, introducing the final new components: token embeddings, positional encoding, the language model head, and the glue that connects them.

What We Have Built So Far

SingleAttentionHead Q, K, V projections, scaled dot-product, causal mask | MultiHeadAttention N heads in parallel, concatenate, project | FeedForward Expand, ReLU, compress (position-wise MLP) | TransformerBlock LayerNorm + MHA + residual, LayerNorm + FFN + residual | [This page] | GPTModel Embeddings + N blocks + final LN + output head

Each block transforms (B, T, n_embd) to (B, T, n_embd). But the raw input to a language model is not a 3D float tensor. It is a 2D tensor of integer token IDs: (B, T). And the output we need is not a 3D embedding. It is a (B, T, vocab_size) tensor of logits predicting the next token. The GPTModel class handles both ends of this conversion.

Class Overview

class GPTModel : public torch::nn::Module { vector<shared_ptr<TransformerBlock>> blocks; int vocab_size, n_embd, num_heads, head_size, d_ff, block_size, num_layers; torch::nn::Embedding token_emb{nullptr}; // token ID -> vector torch::Tensor pe; // positional encoding (not learned) torch::nn::LayerNorm final_ln{nullptr}; // normalize after all blocks torch::nn::Linear lm_head{nullptr}; // project to vocabulary logits public: torch::Tensor positional_encoding(int block_size, int n_embd); GPTModel(int vocab_size, int n_embd, int num_heads, int head_size, int d_ff, int block_size, int num_layers); torch::Tensor forward(torch::Tensor token_ids); };

The model has five categories of members. Each serves a distinct purpose in the pipeline:

MemberTypeRole
token_embEmbedding (learned)Converts integer token IDs into n_embd-dimensional vectors
peTensor (fixed buffer)Adds position information to token embeddings
blocksVector of TransformerBlocksThe core of the model: N stacked attention + FFN layers
final_lnLayerNorm (learned)Normalizes the output of the last block
lm_headLinear (learned)Projects from n_embd to vocab_size to produce next-token logits

The Seven Hyperparameters

The constructor takes seven integers that fully define the model architecture:

ParameterMeaningExample
vocab_sizeNumber of unique tokens in the vocabulary50257 (GPT-2)
n_embdEmbedding dimension (model width)768
num_headsNumber of attention heads per block12
head_sizeDimension per head (usually n_embd / num_heads)64
d_ffFFN inner dimension (usually 4 * n_embd)3072
block_sizeMaximum sequence length the model supports1024
num_layersNumber of stacked transformer blocks12

block_size is the maximum sequence length. The positional encoding is precomputed for block_size positions. During inference, you can pass shorter sequences (the forward pass handles this with slicing), but you cannot exceed block_size. If your input has 1024 tokens and block_size is 512, you must truncate or use a sliding window.

Several parameters are not truly independent:

  • head_size = n_embd / num_heads (so concatenation recovers n_embd)
  • d_ff = 4 * n_embd (standard ratio from the original paper)

We pass them explicitly rather than computing them internally. This gives flexibility: you could use a non-standard FFN expansion ratio or a head_size that does not evenly divide n_embd (though that would break concatenation in our MultiHeadAttention).

GPT-2 Small config: vocab_size = 50257, n_embd = 768, num_heads = 12, head_size = 64, d_ff = 3072, block_size = 1024, num_layers = 12. This gives roughly 124M parameters.

Token Embeddings

token_emb = register_module("token_emb", torch::nn::Embedding(vocab_size, n_embd));

torch::nn::Embedding is a lookup table. It stores a matrix of shape (vocab_size, n_embd) where each row is a learnable vector for one token in the vocabulary.

How it works

Embedding table shape: (vocab_size, n_embd) = (50257, 768) Input token IDs: [42, 1337, 0, 7] (a sequence of 4 tokens) Embedding lookup: token_emb(42) -> row 42 of the table -> [0.12, -0.34, ..., 0.78] (768 values) token_emb(1337) -> row 1337 of the table -> [0.56, 0.01, ..., -0.22] (768 values) token_emb(0) -> row 0 of the table -> [-0.91, 0.43, ..., 0.15] (768 values) token_emb(7) -> row 7 of the table -> [0.33, -0.67, ..., 0.89] (768 values) Output shape: (4, 768)

Given a 2D input of token IDs with shape (B, T), the embedding layer looks up each ID and returns a 3D tensor of shape (B, T, n_embd). This is the conversion from integer IDs to the floating-point tensors that the transformer blocks expect.

Under the hood, torch::nn::Embedding is not a matrix multiplication. It is an indexing operation. For token ID 42, it returns weight[42], which is a direct memory access. This is much faster than encoding the ID as a one-hot vector and multiplying by the weight matrix (though mathematically they are equivalent).

The embedding table is initialized randomly and updated during training via backpropagation. Over time, tokens with similar meanings end up with similar embedding vectors. The famous "king - man + woman = queen" relationship emerges from this learned embedding space.

Memory cost: With vocab_size = 50257 and n_embd = 768, the embedding table alone has 50257 * 768 = 38,597,376 parameters. That is 38.6M parameters just for the lookup table, roughly 31% of GPT-2 Small's total 124M parameters.

Positional Encoding

Attention is permutation-invariant. If you shuffle the order of tokens, the attention scores change (because the causal mask changes), but the model has no inherent notion of "this token is at position 3." Positional encoding injects position information into each token's representation.

pe = positional_encoding(block_size, n_embd); register_buffer("pe", pe);

Our model uses sinusoidal positional encoding: a fixed (not learned) matrix of shape (block_size, n_embd) where each row encodes a position using sine and cosine functions at different frequencies.

pe shape: (block_size, n_embd) = (1024, 768) Row 0: encoding for position 0 Row 1: encoding for position 1 ... Row 1023: encoding for position 1023 Each row has 768 values computed from sin/cos at various frequencies.

The encoding is added element-wise to the token embeddings: x = tok + pe. After this addition, each token's vector contains both its semantic meaning (from the embedding) and its position (from the PE).

Deep dive: The sinusoidal encoding math (the div_term, why sin for even indices and cos for odd, the frequency spectrum, and what makes positions distinguishable) is covered in detail on the dedicated Positional Encoding deep dive page.

GPT-2 and GPT-3 actually use learned positional embeddings (another torch::nn::Embedding(block_size, n_embd)). Our code uses fixed sinusoidal encoding from the original "Attention is All You Need" paper. Both approaches work well. The fixed version has the advantage of not adding learnable parameters and theoretically generalizing to longer sequences (though in practice, both approaches struggle beyond the trained context length).

register_buffer vs register_module

// Module registration (has learnable parameters) token_emb = register_module("token_emb", torch::nn::Embedding(...)); // Buffer registration (no learnable parameters) pe = positional_encoding(block_size, n_embd); register_buffer("pe", pe);

This is a critical distinction in LibTorch. Both register_module and register_buffer integrate a tensor into the model, but they serve fundamentally different purposes.

Propertyregister_moduleregister_buffer
What it registersA module with its own parametersA raw tensor
Included in parameters()?Yes (the module's weights)No
Updated by optimizer?Yes, gradients flow and weights updateNo, the tensor stays fixed
Moved by to(device)?Yes, all module tensors moveYes, the buffer tensor moves
Included in state_dict?YesYes
Typical useLinear layers, embeddings, normsFixed tensors: masks, PE, constants

Why register the PE as a buffer?

The positional encoding is a precomputed constant. We do not want the optimizer to modify it during training. But we do want it to:

  1. Move to GPU with the model. When you call model->to(torch::kCUDA), the PE must move to GPU alongside the token embeddings. Without registration, the PE stays on CPU, causing a device mismatch error when you try tok + pe.
  2. Be saved and loaded with the model. register_buffer includes the tensor in the state dict, so saving and loading the model preserves the PE.
  3. Not consume optimizer memory. The optimizer tracks momentum, variance, and other per-parameter state. For a (1024, 768) PE, that would waste 1.5M floats of optimizer state for a tensor that never changes.

If you declare pe as a plain member tensor without calling register_buffer, it becomes invisible to the model infrastructure. model->to(torch::kCUDA) does not move it. model->save() does not save it. It is a dangling tensor that happens to live inside the class. This is one of the most common bugs in LibTorch code: the forward pass works on CPU but crashes on GPU because an unregistered tensor was not moved.

A common mistake: Registering the PE with register_module instead of register_buffer. You cannot register a raw tensor as a module. register_module expects a module (something that inherits from torch::nn::Module). Passing a tensor would not compile. The two functions accept fundamentally different types.

Stacking Transformer Blocks

for (int i = 0; i < num_layers; i++) { shared_ptr<TransformerBlock> b = make_shared<TransformerBlock>(n_embd, num_heads, head_size, d_ff); register_module("block_" + to_string(i), b); blocks.push_back(b); }

This is the same pattern we used for attention heads in MultiHeadAttention: create in a loop, register with unique names, store in a vector. Each block is fully independent with its own set of parameters (its own MHA, FFN, and two LayerNorms).

Parameter hierarchy

With num_layers = 12, the registration creates a deep hierarchy:

block_0.mha.head_0.Q.weight block_0.mha.head_0.K.weight block_0.mha.head_0.V.weight block_0.mha.head_1.Q.weight ... block_0.mha.projection.weight block_0.mha.projection.bias block_0.ff.linear_1.weight block_0.ff.linear_1.bias block_0.ff.linear_2.weight block_0.ff.linear_2.bias block_0.ln1.weight block_0.ln1.bias block_0.ln2.weight block_0.ln2.bias block_1.mha.head_0.Q.weight ... block_11.ln2.bias

This naming hierarchy is critical for debugging (you can inspect any specific parameter), for transfer learning (you can freeze or load specific blocks), and for checkpoint saving (every tensor has a unique path).

Every block has the same structure: MHA with the same num_heads and head_size, FFN with the same d_ff. But each block creates its own fresh weight matrices initialized randomly. Block 0 and block 11 are structurally identical but learn completely different transformations during training. Early blocks tend to learn syntactic patterns, while later blocks learn more abstract semantic patterns.

Final LayerNorm

final_ln = register_module("final_ln", torch::nn::LayerNorm(torch::nn::LayerNormOptions({n_embd})));

There is an extra LayerNorm after the last transformer block and before the output head. This is a consequence of using pre-norm architecture.

Why is this necessary?

In our pre-norm design, each block normalizes before the sub-layer:

Block: x = x + MHA(LN(x)) x = x + FFN(LN(x))

The output of the last block has passed through MHA and FFN but has not been normalized afterward. The residual addition means the final output is the sum of many unnormalized sub-layer outputs. Without a final LayerNorm, this sum could have an unpredictable scale, making the linear head's job harder.

In post-norm architecture, this extra LayerNorm is not needed because every sub-layer's output is already normalized. The final LayerNorm is specific to pre-norm transformers.

The Language Model Head

lm_head = register_module("lm_head", torch::nn::Linear(n_embd, vocab_size));

lm_head is the final linear layer that converts the transformer's internal representation into a prediction over the vocabulary. It maps from n_embd dimensions to vocab_size dimensions.

lm_head: Linear(n_embd, vocab_size) = Linear(768, 50257) Input: (B, T, 768) one 768-dim vector per token position Output: (B, T, 50257) one score per vocabulary token, per position

What are logits?

The output of lm_head is called "logits": raw, unnormalized scores. For each position in the sequence, the model outputs 50257 numbers (one per vocabulary token). Higher numbers mean "the model thinks this token is more likely to come next."

For position t, the model outputs: logits[t] = [2.1, -0.5, 0.3, ..., 1.8] (50257 values) To get probabilities, apply softmax: probs[t] = softmax(logits[t]) probs[t] = [0.04, 0.003, 0.007, ..., 0.03] (sums to 1.0) The predicted next token is: argmax(logits[t])

During training, the logits are fed into a cross-entropy loss function along with the actual next tokens. The loss function internally applies softmax and computes the negative log probability of the correct token. We do not apply softmax in the model itself because cross-entropy loss is numerically more stable when computed directly from logits.

LibTorch's torch::nn::functional::cross_entropy expects raw logits, not probabilities. It computes log_softmax internally using the log-sum-exp trick, which avoids the numerical overflow that can happen when you compute softmax (which involves exp) followed by log separately. Returning logits gives the caller flexibility: they can apply softmax for inference, or pass logits directly to the loss for training.

Weight tying: Some architectures (GPT-2 included) share the lm_head weight matrix with the token embedding matrix (transposed). This is called "weight tying." The intuition is that the embedding maps token ID to vector, and the lm_head maps vector back to token ID. They are inverse operations, so sharing the weight makes sense. Our code does not implement weight tying for clarity, but it is a common optimization that reduces parameter count significantly.

The Forward Pass

torch::Tensor forward(torch::Tensor token_ids) { // token_ids shape: (B, T) where T <= block_size int T = token_ids.size(1); // Step 1: Token embedding lookup torch::Tensor tok = token_emb(token_ids); // (B, T) -> (B, T, n_embd) // Step 2: Add positional encoding torch::Tensor x = tok + pe.slice(0, 0, T); // (B, T, n_embd) // Step 3: Pass through all transformer blocks for (auto& b : blocks) x = b->forward(x); // (B, T, n_embd) each time // Step 4: Final layer norm x = final_ln(x); // (B, T, n_embd) // Step 5: Project to vocabulary auto logits = lm_head(x); // (B, T, vocab_size) return logits; }

Five steps. Let us walk through each with concrete dimensions (using GPT-2 Small config: vocab_size = 50257, n_embd = 768, block_size = 1024, num_layers = 12, and an input of B = 4 sequences of T = 64 tokens).

Step 1: Token embedding lookup

torch::Tensor tok = token_emb(token_ids); // token_ids: (4, 64) integers in range [0, 50256] // tok: (4, 64, 768) float vectors

Each integer token ID is replaced by its corresponding 768-dimensional embedding vector. This is a pure lookup, no computation.

Step 2: Add positional encoding

torch::Tensor x = tok + pe.slice(0, 0, T); // tok: (4, 64, 768) // pe.slice: (64, 768) sliced from (1024, 768) // Broadcasting: (64, 768) -> (1, 64, 768) -> (4, 64, 768) // x: (4, 64, 768)

Position information is injected. Token at position 0 gets PE row 0 added, token at position 5 gets PE row 5 added, and so on. We cover the slice operation in the next section.

Step 3: Transformer blocks

for (auto& b : blocks) x = b->forward(x); // x stays (4, 64, 768) through all 12 blocks

The tensor passes through all 12 blocks sequentially. Each block refines the representation: first through attention (tokens communicate), then through the FFN (each token processes independently). The shape never changes.

Step 4: Final LayerNorm

x = final_ln(x); // x: (4, 64, 768) -> (4, 64, 768) normalized

Normalize the output of the last block before projection.

Step 5: Language model head

auto logits = lm_head(x); // x: (4, 64, 768) // logits: (4, 64, 50257)

Each token position now has a score for every token in the vocabulary. Position 63's logits predict what token comes at position 64.

pe.slice(0, 0, T) Explained

torch::Tensor x = tok + pe.slice(0, 0, T);

The PE tensor has shape (block_size, n_embd) = (1024, 768). It has entries for all 1024 possible positions. But our current input might only have T = 64 tokens. We need only the first 64 rows.

pe.slice(dim, start, end) pe.slice(0, 0, T) dim = 0: slice along the first dimension (rows = positions) start = 0: start from position 0 end = T: stop at position T (exclusive) pe shape: (1024, 768) pe.slice(0,0,64): (64, 768) only the first 64 positions

Why not just use the full PE?

If the input has T = 64 tokens and we try to add the full (1024, 768) PE to a (B, 64, 768) embedding, the dimensions would not match. Broadcasting cannot reconcile dimension 1 (size 64) with dimension 0 (size 1024). We must slice the PE to match the actual sequence length.

Broadcasting after slicing

tok shape: (B, T, n_embd) = (4, 64, 768) 3D pe.slice shape: (T, n_embd) = (64, 768) 2D Broadcasting adds a dimension at front: (64, 768) -> (1, 64, 768) -> (4, 64, 768) Addition is now element-wise: tok[b][t][d] + pe[t][d] for all b, t, d

Every sequence in the batch gets the same positional encoding. This is correct because position 5 is position 5 regardless of which sequence contains it.

pe.slice(0, 0, T) returns a view into the original PE tensor. No data is copied. The view shares the same underlying memory, just with different metadata (start offset and size). This is why slicing is essentially free in terms of performance.

Parameter Count

Full parameter breakdown for GPT-2 Small configuration:

Token embedding: vocab_size * n_embd = 50257 * 768 = 38,597,376 Per transformer block: MHA: 8 heads * 3 * (768 * 64) = 1,179,648 Projection: 768 * 768 + 768 = 590,592 FFN: linear_1: 768 * 3072 + 3072 = 2,362,368 linear_2: 3072 * 768 + 768 = 2,360,064 LayerNorm * 2: 2 * (768 + 768) = 3,072 Block total: 6,495,744 12 blocks: 12 * 6,495,744 = 77,948,928 Final LayerNorm: 768 + 768 = 1,536 LM Head: 768 * 50257 + 50257 = 38,647,633 Positional encoding: (registered as buffer, NOT a parameter) = 0 Grand total: 38,597,376 + 77,948,928 + 1,536 + 38,647,633 = 155,195,473 parameters (without weight tying; with tying, subtract 38.6M)
Where do the parameters live? Token embedding: 25%. Transformer blocks: 50%. LM head: 25%. The blocks are the core, but the input and output layers together use half the parameters. With weight tying (sharing embedding and lm_head weights), the total drops to about 116M, closer to the commonly cited 124M figure (the difference is in exact config details).
