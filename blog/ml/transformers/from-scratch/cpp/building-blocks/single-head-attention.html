SingleHeadAttention

April 2026 · C++ / LibTorch · Transformers

What Does This Class Do?

A SingleHeadAttention module takes a batch of token embeddings and produces a new representation for every token in which each token has "attended" to every other token that came before it (and itself). In concrete terms:

  • Input: a tensor of shape (batch, seq_len, n_embd). Each of the seq_len tokens is represented by a vector of size n_embd.
  • Output: a tensor of shape (batch, seq_len, head_size). Every token now carries information gathered from the tokens it paid attention to, compressed into a vector of size head_size.

The transformation happens in five stages: project the input into Query, Key, and Value vectors, compute pairwise attention scores, scale them, mask out future positions (causal masking), normalize with softmax, and finally use those weights to produce a weighted sum of Value vectors.

Input: (batch, seq_len, n_embd) | Q, K, V projections (three separate linear layers) | Q, K, V each: (batch, seq_len, head_size) | raw_attn = Q @ K^T / sqrt(head_size) -> (batch, seq_len, seq_len) | causal mask + softmax | output = attn_weights @ V -> (batch, seq_len, head_size)

If the input embedding dimension n_embd is 64 and head_size is 16, the class projects each 64-dimensional token into a 16-dimensional attended representation. The dimensionality reduction is intentional: in a full Transformer, multiple heads run in parallel, each with a smaller head_size, and their outputs are concatenated back to the full embedding size.

The Class Structure

Here is the complete class declaration and constructor. Every line is doing something deliberate.

class SingleHeadAttention : public torch::nn::Module{ /* The input embedding vector comes in (size n_embd) and the output vector is of size head_size */ int n_embd, head_size; torch::nn::Linear K{nullptr}, Q{nullptr}, V{nullptr}; public: SingleHeadAttention(int n_embd, int head_size) : n_embd(n_embd), head_size(head_size) { K = register_module( "K", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false) ) ); Q = register_module( "Q", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false) ) ); V = register_module( "V", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false) ) ); }

Let us break down the non-obvious parts of this code.

Why torch::nn::Linear K{nullptr}?

In C++, member variables are constructed when the object is created. But torch::nn::Linear requires dimensions (n_embd and head_size) at construction time. Those values come from the constructor parameters, which are not available during member initialization in the class body. So we initialize K, Q, and V as null holders first, then assign real layers inside the constructor body where the dimensions are known.

The {nullptr} syntax creates the torch::nn::Linear wrapper in an "empty" state. It holds no weight matrix and no parameters. Think of it as a placeholder that says "I will be a linear layer, but I do not know my size yet."

Why register_module?

register_module does three critical things:

  1. Parameter discovery. It adds every parameter inside the linear layer (the weight matrix) to the module's parameter list. When you call model->parameters() to pass parameters to an optimizer, these weights will be found and updated during training. Without registration, the optimizer has no way to discover them.
  2. Named hierarchy. The string argument ("K", "Q", "V") gives each sub-module a name. When saving checkpoints, the weights are stored under keys like K.weight, Q.weight, V.weight. Loading a checkpoint matches by name, so the names must be consistent.
  3. Device management. When you call model->to(torch::kCUDA), every registered sub-module moves to GPU. An unregistered linear layer stays on CPU, causing a device mismatch crash at runtime.

Why .bias(false)?

The Q, K, V projections are pure matrix multiplications: Q = X * W_Q, K = X * W_K, V = X * W_V. No bias vector is added. This follows the original "Attention Is All You Need" paper. The rationale: the input embeddings already encode position and meaning. Adding a constant bias to every query, key, and value would introduce a uniform shift that provides no useful information for computing which tokens should attend to which. In practice, models trained with and without bias in Q/K/V projections converge to nearly identical performance, so the bias is omitted to save parameters.

What is torch::nn::LinearOptions?

LibTorch uses the options pattern for module configuration. Instead of passing multiple arguments directly to the Linear constructor, you create a LinearOptions object that specifies the input features, output features, and optional settings like .bias(false). This is a C++ convention in LibTorch to mirror PyTorch's keyword arguments. torch::nn::LinearOptions(n_embd, head_size) says "create a linear layer that maps from n_embd input features to head_size output features." The .bias(false) chain disables the bias term.

Internally, torch::nn::Linear(LinearOptions(n_embd, head_size).bias(false)) creates a weight tensor of shape (head_size, n_embd) with Kaiming uniform initialization and requires_grad = true. When called as Q(x), it computes x @ weight.T. The weight is stored transposed for memory layout efficiency, but the math is equivalent to a standard matrix multiply.

The Forward Pass Line by Line

This is where the attention computation happens. Here is the complete forward method with every line highlighted:

torch::Tensor forward(torch::Tensor x){ int seq_len = x.size(1); torch::Tensor output, k_vec, q_vec, v_vec, raw_attn, mask; q_vec = Q(x); k_vec = K(x); v_vec = V(x); raw_attn = torch::matmul(q_vec, k_vec.transpose(-2, -1)) / ((float) sqrt(head_size)); mask = torch::tril(torch::ones({seq_len, seq_len}, x.options())); raw_attn.masked_fill_(mask == 0, -1e9f); output = torch::softmax(raw_attn, -1); output = torch::matmul(output, v_vec); return output; }

Let us trace through each line. The input x has shape (batch, seq_len, n_embd).

int seq_len = x.size(1);

x.size(1) extracts the second dimension of the tensor. For a 3D tensor of shape (batch, seq_len, n_embd), dimension 0 is batch, dimension 1 is seq_len, and dimension 2 is n_embd. We store this because the causal mask needs to be seq_len x seq_len.

torch::Tensor output, k_vec, q_vec, v_vec, raw_attn, mask;

This declares all the intermediate tensors used in the forward pass. In C++, torch::Tensor is a smart pointer type. Declaring them without initialization creates empty tensors that will be assigned later. This is a style choice, declaring all variables at the top of the function to make the data flow visible at a glance.

The Projections: Q(x), K(x), V(x)

q_vec = Q(x); // (batch, seq_len, n_embd) -> (batch, seq_len, head_size) k_vec = K(x); // (batch, seq_len, n_embd) -> (batch, seq_len, head_size) v_vec = V(x); // (batch, seq_len, n_embd) -> (batch, seq_len, head_size)

Each call invokes the forward method of a torch::nn::Linear module. The linear layer stores a weight matrix of shape (head_size, n_embd). When called with input x of shape (batch, seq_len, n_embd), it computes x @ weight.T, which is a matrix multiplication of the last dimension:

(batch, seq_len, n_embd) @ (n_embd, head_size) = (batch, seq_len, head_size)

The operation is applied independently to every token in every sequence. Token 0 in batch 0 gets multiplied by the same weight matrix as token 5 in batch 3. The linear layer does not look at relationships between tokens. It simply projects each token's embedding from n_embd dimensions down to head_size dimensions.

Why three separate projections?

Each projection serves a different role in the attention mechanism:

The Query projection learns to extract "what is this token looking for?" For example, if a pronoun like "it" has learned its Query representation, that representation encodes "I am looking for the noun I refer to." The Query is the question that a token asks.

The Key projection learns to extract "what does this token contain?" Each token advertises its content through its Key. A noun like "mat" might have a Key that signals "I am a concrete noun." The Key is the label a token presents to incoming queries.

The Value projection learns to extract "what information should this token contribute?" When a token is attended to, its Value is what gets passed forward. The Value may encode different information than the Key. A word might advertise "I am a noun" through its Key, but its Value might carry semantic features like "physical object, flat surface, indoor item."

Having three separate weight matrices means the model learns three independent transformations. The Query and Key spaces are optimized for matching (computing which tokens are relevant to each other), while the Value space is optimized for content (what information to extract once relevance is determined). If we used the same projection for all three, the model would have to compromise between these competing objectives.

Concrete example: Consider the sentence "The cat sat on the mat because it was soft." The word "it" needs its Query to match the Key of "mat" (to determine reference). But the Value of "mat" should carry semantic content (softness, physical object), not the matching signal. Separate projections allow Q/K to specialize in matching and V to specialize in information transfer.

Why transpose(-2, -1)

This is the single most important mechanical detail in the attention computation. If you understand why we transpose, you understand how attention scores are computed. Let us build the reasoning from first principles.

What We Are Trying to Compute

We want to compute a score between every pair of tokens. Specifically, for token i and token j, the attention score is the dot product of token i's query vector and token j's key vector:

score[i][j] = dot(Q_i, K_j) = sum over d of (Q_i[d] * K_j[d])

If we have S tokens, we need to compute S x S scores: every query dotted with every key. This is exactly the definition of matrix multiplication. If we stack all query vectors as rows of a matrix Q and all key vectors as columns of a matrix K^T, the matrix product Q @ K^T gives us all S x S scores in one operation.

Q = [ Q_0 ] K^T = [ K_0 | K_1 | K_2 ] [ Q_1 ] [ Q_2 ] Q @ K^T = [ dot(Q_0,K_0) dot(Q_0,K_1) dot(Q_0,K_2) ] [ dot(Q_1,K_0) dot(Q_1,K_1) dot(Q_1,K_2) ] [ dot(Q_2,K_0) dot(Q_2,K_1) dot(Q_2,K_2) ]

Row i, column j of the result is exactly the attention score from token i to token j. This is what we want. But there is a problem with shapes.

The Shape Problem

After the projections, q_vec and k_vec both have shape (batch, S, H), where S is the sequence length and H is head_size. Let us try to multiply them directly:

q_vec: (batch, S, H) k_vec: (batch, S, H) matmul(q_vec, k_vec): Last two dims: (S, H) @ (S, H) Inner dimensions: H vs S H != S -> DIMENSION MISMATCH. Cannot multiply.

Matrix multiplication requires that the inner dimensions match. For (S, H) @ (?, ?), the second matrix must have its first dimension equal to H. But k_vec's last two dimensions are (S, H), so the first of those is S, not H. The multiplication is impossible as-is.

We need the second matrix to have shape (H, S) so that the inner dimensions align: (S, H) @ (H, S). The inner H dimensions match, and the result is (S, S), which is exactly the S x S attention score matrix we want.

What transpose(-2, -1) Does

k_vec.transpose(-2, -1) swaps the last two dimensions of the tensor. Before and after:

Before: k_vec shape = (batch, S, H) After: k_vec^T shape = (batch, H, S)

Now the matmul works:

matmul(q_vec, k_vec.transpose(-2, -1)): q_vec: (batch, S, H) k_vec^T: (batch, H, S) Last two dims: (S, H) @ (H, S) Inner dimension: H == H -> MATCH Result: (batch, S, S)

The result is a (batch, S, S) tensor. For each sequence in the batch, we get an S x S matrix where entry [i][j] is the dot product of query i with key j. That is the raw attention score.

An important detail: .transpose(-2, -1) does not copy or rearrange any data in memory. It changes the stride metadata of the tensor. LibTorch tensors store data as a flat 1D array plus a shape and stride descriptor. Transposing simply swaps the stride values for the two dimensions, so the same memory is read in a different order. This makes the transpose operation nearly free, just a metadata update, not a data copy.

Why -2 and -1, Not 0 and 1?

Negative indices count from the end of the dimension list:

For a 3D tensor with dims [0, 1, 2]: -1 = dimension 2 (the last dim, H) -2 = dimension 1 (second-to-last, S) -3 = dimension 0 (third-to-last, batch) transpose(-2, -1) swaps dim 1 and dim 2 -> swaps S and H

The reason for using negative indices instead of .transpose(1, 2) is generality. If the tensor has additional batch dimensions (for example, in multi-head attention the shape might be (batch, n_heads, S, H), a 4D tensor), .transpose(-2, -1) still swaps the correct pair (S and H). Using .transpose(1, 2) on a 4D tensor would swap n_heads and S, which is wrong.

3D case: (batch, S, H) .transpose(-2, -1) -> swaps dim 1, dim 2 -> (batch, H, S) CORRECT .transpose(1, 2) -> swaps dim 1, dim 2 -> (batch, H, S) CORRECT 4D case: (batch, n_heads, S, H) .transpose(-2, -1) -> swaps dim 2, dim 3 -> (batch, n_heads, H, S) CORRECT .transpose(1, 2) -> swaps dim 1, dim 2 -> (batch, S, n_heads, H) WRONG

By always using -2 and -1, the code says "swap the last two dimensions, no matter how many dimensions exist." This is robust and future-proof.

Batched matmul

When torch::matmul receives tensors with more than 2 dimensions, it treats all dimensions except the last two as batch dimensions. The actual matrix multiplication only happens on the last two dimensions. The batch dimensions are carried through unchanged.

matmul(A, B) where A is (batch, S, H) and B is (batch, H, S): For batch index 0: A[0] is (S, H), B[0] is (H, S) -> result[0] = A[0] @ B[0] = (S, S) For batch index 1: A[1] is (S, H), B[1] is (H, S) -> result[1] = A[1] @ B[1] = (S, S) ... Final result: (batch, S, S)

Each sequence in the batch gets its own independent matrix multiplication. The batch dimension is never mixed: tokens in batch 0 never interact with tokens in batch 1. This is correct, because each sequence is an independent input and should be processed independently.

Efficiency note: Batched matmul is not a loop. It dispatches all the multiplications to GPU in a single kernel call, running them in parallel. Processing a batch of 32 sequences costs almost the same time as processing one, because the GPU has thousands of cores that execute the independent matrix multiplications simultaneously.

The Scaling: / sqrt(head_size)

raw_attn = torch::matmul(q_vec, k_vec.transpose(-2, -1)) / ((float) sqrt(head_size));

After computing the raw dot products, we divide by the square root of head_size. This scaling is essential for stable training. Here is why.

The problem with large dot products

Each dot product sums over head_size dimensions. If the query and key vectors have elements drawn from a standard normal distribution (mean 0, variance 1), the dot product is the sum of head_size products of two standard normals. The variance of each product is 1, and the variance of their sum is head_size. So the dot product has standard deviation sqrt(head_size).

If head_size = 64: dot product std dev = sqrt(64) = 8 Typical scores range from roughly -16 to +16 If head_size = 512: dot product std dev = sqrt(512) = 22.6 Typical scores range from roughly -45 to +45

When these large values enter softmax, the function saturates. Softmax with large inputs pushes most of its probability mass onto the single largest value, making the output nearly one-hot (one entry close to 1, all others close to 0). This kills diversity in the attention pattern and, more critically, produces extremely small gradients. The model stops learning.

How scaling fixes it

Dividing by sqrt(head_size) normalizes the dot products back to unit variance:

Variance of raw dot product: head_size Divide by sqrt(head_size): variance becomes head_size / head_size = 1 Standard deviation after scaling: 1

Now the scores are in a moderate range regardless of the head dimension. Softmax receives well-behaved inputs, produces smooth probability distributions, and gradients flow normally.

The (float) cast

head_size is an int. The C++ sqrt function, when given an int, promotes it to double and returns double. Dividing a float tensor by a double can cause type promotion warnings or unexpected behavior. The (float) cast explicitly converts the result of sqrt to float, matching the tensor's data type. This is a defensive cast that ensures the division produces a float tensor, not a double tensor.

What happens without scaling? Training becomes unstable. Early in training the random Q/K vectors produce large dot products. Softmax saturates, attention becomes nearly one-hot, gradients vanish, and the model either fails to converge or converges much more slowly. The 1/sqrt(d_k) scaling was introduced in the original Transformer paper precisely to solve this problem.

The Causal Mask

mask = torch::tril(torch::ones({seq_len, seq_len}, x.options())); raw_attn.masked_fill_(mask == 0, -1e9f);

This is the mechanism that prevents tokens from "looking into the future." In a language model, when predicting what comes after token 3, the model must only see tokens 0, 1, 2, and 3. It must not see tokens 4, 5, 6, and beyond, because those have not been generated yet. The causal mask enforces this constraint.

torch::tril

torch::ones({seq_len, seq_len}) creates a matrix filled with 1s. torch::tril (triangular lower) zeroes out everything above the main diagonal:

torch::tril(torch::ones({4, 4})): [ 1 0 0 0 ] [ 1 1 0 0 ] [ 1 1 1 0 ] [ 1 1 1 1 ]

Row i has 1s in columns 0 through i and 0s everywhere else. This means token i is allowed to attend to tokens 0 through i (itself and all previous tokens) but not to tokens i+1 and beyond (future tokens).

x.options()

The x.options() call returns the tensor options (device, dtype) of the input tensor x. This ensures the mask is created on the same device (CPU or GPU) and with the same data type as the input. Without this, the mask would default to CPU float, and if x is on GPU, the masked_fill_ call would crash with a device mismatch error.

masked_fill_

raw_attn.masked_fill_(mask == 0, -1e9f) does two things in sequence:

  1. mask == 0 creates a boolean tensor that is true where the mask is 0 (the upper triangle) and false where the mask is 1 (the lower triangle and diagonal).
  2. .masked_fill_(..., -1e9f) sets every position where the boolean is true to -1e9 (negative one billion). Positions where the boolean is false are left unchanged.
raw_attn before masking (example 4x4): [ 0.5 0.3 0.8 0.1 ] [ 0.2 0.7 0.4 0.6 ] [ 0.9 0.1 0.5 0.3 ] [ 0.4 0.8 0.2 0.7 ] After masked_fill_(mask == 0, -1e9f): [ 0.5 -1e9 -1e9 -1e9 ] [ 0.2 0.7 -1e9 -1e9 ] [ 0.9 0.1 0.5 -1e9 ] [ 0.4 0.8 0.2 0.7 ]

The upper triangle is flooded with -1e9. When softmax processes each row, exp(-1e9) is effectively 0, so those positions receive zero attention weight. Token 0 can only attend to itself. Token 1 attends to tokens 0 and 1. Token 2 attends to tokens 0, 1, and 2. And so on.

-INFINITY would also work mathematically since exp(-inf) = 0. However, in edge cases (like when an entire row is masked, or during certain gradient computations), true infinity can produce NaN values. Using -1e9 (negative one billion) is numerically safer while being practically equivalent: exp(-1e9) is so close to 0 that it rounds to zero in float32.

masked_fill_ (with trailing underscore) modifies the tensor in place. masked_fill (without underscore) would return a new tensor, leaving the original unchanged. The in-place version is faster because it avoids allocating a new tensor. Since we do not need the unmasked scores again, in-place is the correct choice here.

Broadcasting the mask

The mask has shape (seq_len, seq_len), a 2D tensor. But raw_attn has shape (batch, seq_len, seq_len), a 3D tensor. How does a 2D mask apply to a 3D tensor?

LibTorch uses broadcasting. It automatically treats the mask as (1, seq_len, seq_len) and repeats it across the batch dimension. The same mask is applied to every sequence in the batch, which is correct because the causal constraint is the same for all sequences.

raw_attn shape: (batch, seq_len, seq_len) = (4, 8, 8) mask shape: (seq_len, seq_len) = (8, 8) Broadcasting treats mask as: (1, 8, 8) Effective mask: (4, 8, 8) -- same mask for each batch element

Softmax and Final Matmul

output = torch::softmax(raw_attn, -1); output = torch::matmul(output, v_vec);

torch::softmax(raw_attn, -1)

Softmax converts raw scores into a probability distribution. The -1 argument specifies the dimension along which to normalize. Since raw_attn has shape (batch, S, S), dimension -1 is the last dimension (the key/column dimension). This means softmax normalizes each row independently.

For one row of raw_attn (after masking), say token 2: raw scores: [ 0.9, 0.1, 0.5, -1e9 ] softmax: exp(0.9) = 2.46, exp(0.1) = 1.11, exp(0.5) = 1.65, exp(-1e9) = 0.0 sum = 2.46 + 1.11 + 1.65 + 0.0 = 5.22 weights: [ 2.46/5.22, 1.11/5.22, 1.65/5.22, 0.0/5.22 ] = [ 0.47, 0.21, 0.32, 0.00 ] sum = 1.00

After softmax, each row sums to 1.0. The masked positions (which were set to -1e9) become 0.0 because exp(-1e9) is effectively zero. The remaining weights form a valid probability distribution telling us how much each query token attends to each key token.

torch::matmul(output, v_vec)

This is the weighted sum of value vectors. The attention weights have shape (batch, S, S), and v_vec has shape (batch, S, head_size). The matmul computes:

(batch, S, S) @ (batch, S, head_size) = (batch, S, head_size) For each query token i: output[i] = sum over j of (weight[i][j] * V[j])

Each output row is a weighted combination of all value vectors, where the weights come from the attention distribution. Token i's output is mostly composed of the values of tokens it attended to strongly. Tokens it did not attend to (weight near 0, including future tokens) contribute nothing.

If token 2 has attention weights [0.47, 0.21, 0.32, 0.00] over tokens 0 through 3, then:
output[2] = 0.47 * V_0 + 0.21 * V_1 + 0.32 * V_2 + 0.00 * V_3
Token 2's output is mostly V_0 (47%), mixed with some V_2 (32%) and a bit of V_1 (21%). Token 3 contributes nothing because it is a future token and was masked out.

Why reuse the variable name output?

The code assigns to output twice: first the softmax result, then the final matmul result. This is a memory-efficient pattern. After the second assignment, the softmax tensor is no longer referenced and can be garbage collected. In a language model with long sequences, the (batch, S, S) attention weight matrix can be large (for S=2048, that is 4 million floats per batch element), so releasing it promptly matters.

Complete Shape Trace

Here is every variable in the forward pass with its shape, traced from input to output. Use S for sequence length and H for head_size.

Line Variable Shape
x (input) x (batch, S, n_embd)
q_vec = Q(x) q_vec (batch, S, head_size)
k_vec = K(x) k_vec (batch, S, head_size)
v_vec = V(x) v_vec (batch, S, head_size)
k_vec.transpose(-2, -1) (intermediate) (batch, head_size, S)
matmul(q_vec, k_vec^T) raw_attn (batch, S, S)
/ sqrt(head_size) raw_attn (batch, S, S)
torch::tril(torch::ones(...)) mask (S, S)
raw_attn.masked_fill_(...) raw_attn (batch, S, S)
torch::softmax(raw_attn, -1) output (batch, S, S)
matmul(output, v_vec) output (batch, S, head_size)
Input to output: The tensor enters as (batch, S, n_embd) and leaves as (batch, S, head_size). The sequence length S is preserved (every token still has its own representation). The embedding dimension changes from n_embd to head_size. This dimensionality reduction is intentional: in multi-head attention, each head works in a smaller subspace, and the outputs are concatenated to recover the full dimension.

Full Code

The complete SingleHeadAttention class, ready to compile with LibTorch:

class SingleHeadAttention : public torch::nn::Module{ /* The input embedding vector comes in (size n_embd) and the output vector is of size head_size */ int n_embd, head_size; torch::nn::Linear K{nullptr}, Q{nullptr}, V{nullptr}; public: SingleHeadAttention(int n_embd, int head_size) : n_embd(n_embd), head_size(head_size) { K = register_module( "K", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false) ) ); Q = register_module( "Q", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false) ) ); V = register_module( "V", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false) ) ); } torch::Tensor forward(torch::Tensor x){ int seq_len = x.size(1); torch::Tensor output, k_vec, q_vec, v_vec, raw_attn, mask; q_vec = Q(x); k_vec = K(x); v_vec = V(x); raw_attn = torch::matmul(q_vec, k_vec.transpose(-2, -1)) / ((float) sqrt(head_size)); mask = torch::tril(torch::ones({seq_len, seq_len}, x.options())); raw_attn.masked_fill_(mask == 0, -1e9f); output = torch::softmax(raw_attn, -1); output = torch::matmul(output, v_vec); return output; } };

Usage example

// Create a single-head attention with n_embd=64, head_size=16 auto head = std::make_shared<SingleHeadAttention>(64, 16); // Create a random input: batch=2, seq_len=8, n_embd=64 torch::Tensor x = torch::randn({2, 8, 64}); // Forward pass torch::Tensor out = head->forward(x); // out shape: (2, 8, 16) std::cout << out.sizes() << std::endl; // [2, 8, 16]

With n_embd=64 and head_size=16, each linear layer has a weight matrix of shape (16, 64) = 1024 parameters. Three layers (Q, K, V) give 3 * 1024 = 3072 total trainable parameters. No bias parameters since .bias(false) is set.

What Comes Next

A single attention head captures one type of relationship between tokens. But language has many simultaneous relationships: syntactic structure, coreference, semantic similarity, positional proximity, and more. A single head with a small head_size cannot capture all of these at once.

MultiHeadAttention solves this by running multiple SingleHeadAttention modules in parallel, each with its own Q, K, V weight matrices. Each head learns to focus on a different type of relationship. Their outputs are concatenated along the last dimension and projected back to the original embedding size:

n_heads = 4, head_size = 16, n_embd = 64 Head 0 output: (batch, S, 16) -- maybe learns syntactic dependencies Head 1 output: (batch, S, 16) -- maybe learns coreference Head 2 output: (batch, S, 16) -- maybe learns semantic similarity Head 3 output: (batch, S, 16) -- maybe learns positional patterns Concatenated: (batch, S, 64) -- all relationship types combined Final linear: (batch, S, 64) -- projected and mixed

The concatenation recovers the full embedding dimension: n_heads * head_size = 4 * 16 = 64 = n_embd. The final linear projection lets the model learn how to combine information from all the heads.

Key insight: The single-head attention you just built is the fundamental unit. Multi-head attention is just multiple instances of it running side by side. Understanding this one module means you understand the core of the Transformer.
