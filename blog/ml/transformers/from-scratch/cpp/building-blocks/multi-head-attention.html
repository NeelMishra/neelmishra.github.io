MultiHeadAttention in C++ (LibTorch)

ML Series ML · Transformers · From Scratch · C++ · Building Blocks

In the previous post, we built a single attention head. A single head learns one type of relationship between tokens in a sequence. But language is rich: in a single sentence, one word can relate to another through syntax, through coreference, through semantic similarity, and through positional proximity, all at the same time. A single set of attention weights cannot capture all of these patterns simultaneously.

MultiHeadAttention solves this by running several independent heads in parallel. Each head has its own Q, K, V weight matrices and learns its own attention pattern. Their outputs are then concatenated and projected back to the original embedding dimension. This page walks through the full C++ implementation using LibTorch, line by line.

1 · What Does MultiHeadAttention Do?

At a high level, multi-head attention does three things:

  1. Runs multiple SingleHeadAttention modules in parallel. Each head receives the same input tensor but processes it through its own learned Q, K, V projections. Because the weights differ, each head ends up focusing on different relationships.
  2. Concatenates all head outputs along the last dimension. If you have 8 heads, each producing a vector of size 32, the concatenated result is a vector of size 256.
  3. Projects the concatenated output through a linear layer. This learned projection mixes information across heads and maps the result back to the model's embedding dimension.

Why is this better than a single head with a larger head_size? Because each head operates in a smaller subspace, it is forced to specialize. One head might learn that pronouns attend to their antecedents. Another might learn that verbs attend to their subjects. A third might learn positional patterns like "the next token" or "the token two steps back." By running all of these in parallel and combining the results, the model captures richer, more diverse attention patterns than any single head could alone.

The original paper's formulation: Vaswani et al. define MultiHead(Q, K, V) = Concat(head_1, ..., head_h) * W_O, where each head_i = Attention(X * W_Q_i, X * W_K_i, X * W_V_i). Our implementation follows this exactly. The SHA_group vector holds the heads, and the projection_layer is W_O.

Input and output shapes

The module takes in a 3D tensor and produces a 3D tensor of the same shape:

Input: (batch_size, seq_len, n_embd) Output: (batch_size, seq_len, n_embd)

This shape preservation is essential. In a transformer, the output of multi-head attention feeds into a residual connection: x = x + MultiHeadAttention(x). If the shapes did not match, the element-wise addition would fail.

What each head sees

Every head receives the same input tensor x. There is no slicing or splitting of the input. The differentiation comes entirely from the weights. Each head's Q, K, V matrices are initialized randomly and trained independently, so they naturally diverge during optimization and learn to attend to different patterns.

Imagine giving the same document to eight different reviewers. Each reviewer has a different background: one is a grammarian, one is a fact-checker, one is a style editor, and so on. They all read the same text, but each highlights different things. Multi-head attention works the same way. Each head "reviews" the input through its own lens, and the projection layer synthesizes all their annotations into a final output.

2 · The Class Structure

Here is the complete class definition, exactly as written:

class MultiHeadAttention : public torch::nn::Module{ /* It has access to multiple singlheadattention, and then it concatenates the output of all of these layer into one vector which then it passes to a projection layer input : batch_size x seq_len x n_embd output : batch_size x seq_len x (#of heads x head_size => n_embd) */ int num_heads, head_size, n_embd; vector< shared_ptr<SingleHeadAttention> > SHA_group; torch::nn::Linear projection_layer{nullptr}; public: MultiHeadAttention(int num_heads, int head_size, int n_embd) : num_heads(num_heads), head_size(head_size), n_embd(n_embd){ for (int head_idx = 0; head_idx < num_heads; head_idx++){ shared_ptr<SingleHeadAttention> curr_head = make_shared<SingleHeadAttention>(n_embd, head_size); register_module( "SHA_" + to_string(head_idx), curr_head ); SHA_group.push_back(curr_head); } projection_layer = register_module( "projection_layer", torch::nn::Linear( torch::nn::LinearOptions( head_size * num_heads, n_embd ) ) ); } torch::Tensor forward(torch::Tensor x){ torch::Tensor output, concatenated_SHA_outputs; vector < torch::Tensor> SHA_outputs; for(int head_idx = 0; head_idx < num_heads; head_idx++){ SHA_outputs.push_back(SHA_group[head_idx]->forward(x)); } concatenated_SHA_outputs = torch::cat(SHA_outputs, -1); output = projection_layer(concatenated_SHA_outputs); return output; } };

Let us break down each member and design decision.

Member variables

Three integer fields store the configuration. num_heads is the number of parallel attention heads. head_size is the output dimension of each individual head. n_embd is the full embedding dimension of the model. The relationship head_size * num_heads = n_embd ensures that concatenation recovers the original dimension.

This is a std::vector of std::shared_ptrs, each pointing to a SingleHeadAttention object. Why a vector of shared pointers?

  • Vector: We need a dynamically-sized collection because the number of heads is a runtime parameter passed to the constructor. A fixed array would not work.
  • shared_ptr: LibTorch's register_module function accepts shared_ptr<Module>. It needs shared ownership because the module registry and our vector both need to reference the same head object. A raw pointer or unique_ptr would not satisfy this requirement.

The name SHA_group stands for "Single Head Attention group," a collection of all the attention heads that this multi-head module manages.

The projection layer is declared with {nullptr} initialization, the same pattern used in SingleHeadAttention for Q, K, V. We cannot construct the Linear layer here because we need head_size * num_heads and n_embd from the constructor parameters. The actual layer is created inside the constructor body and registered with register_module.

The constructor: registering heads in a loop

The constructor uses a loop to create and register each head:

for (int head_idx = 0; head_idx < num_heads; head_idx++){ shared_ptr<SingleHeadAttention> curr_head = make_shared<SingleHeadAttention>(n_embd, head_size); register_module( "SHA_" + to_string(head_idx), curr_head ); SHA_group.push_back(curr_head); }

Three things happen on each iteration:

  1. Create the head: make_shared<SingleHeadAttention>(n_embd, head_size) allocates a new SingleHeadAttention on the heap. Each head gets its own Q, K, V weight matrices of shape (n_embd, head_size).
  2. Register the head: register_module("SHA_" + to_string(head_idx), curr_head) adds this head to the LibTorch module tree. The name is unique per head: "SHA_0", "SHA_1", "SHA_2", etc. This registration is what makes the head's parameters visible to the optimizer. Without it, the Q, K, V weights inside the head would never be updated during training.
  3. Store the head: SHA_group.push_back(curr_head) keeps a reference in the vector for use during the forward pass.
Why register_module inside the loop? Each head is a separate torch::nn::Module with its own parameters (Q.weight, K.weight, V.weight). If you skip registration, those parameters become invisible. The optimizer will not update them. The model will not train. Calling model->to(torch::kCUDA) will leave unregistered heads on CPU while the rest of the model moves to GPU, causing device mismatch errors.

The string "SHA_" + to_string(head_idx) produces names like "SHA_0", "SHA_1", and so on. These names must be unique within the parent module. If two submodules share a name, LibTorch throws a runtime error. Using the loop index guarantees uniqueness. When you later call named_parameters(), you will see entries like SHA_0.Q.weight, SHA_0.K.weight, SHA_1.Q.weight, etc. This hierarchical naming makes debugging and checkpoint inspection straightforward.

The projection layer

projection_layer = register_module( "projection_layer", torch::nn::Linear( torch::nn::LinearOptions( head_size * num_heads, n_embd ) ) );

The projection layer is a standard linear layer that maps from head_size * num_heads to n_embd. Since head_size * num_heads equals n_embd by design, this is effectively an n_embd -> n_embd linear transformation. It is registered just like any other module, with a descriptive name "projection_layer".

Notice that unlike SingleHeadAttention's Q, K, V layers which use .bias(false), the projection layer uses the default torch::nn::LinearOptions which includes bias. This is standard. The Q, K, V projections typically omit bias (following the original paper), but the output projection W_O typically includes a bias term.

Parameter hierarchy after construction: For MultiHeadAttention(4, 64, 256) (4 heads, head_size=64, n_embd=256), calling named_parameters() produces:
SHA_0.Q.weight, SHA_0.K.weight, SHA_0.V.weight
SHA_1.Q.weight, SHA_1.K.weight, SHA_1.V.weight
SHA_2.Q.weight, SHA_2.K.weight, SHA_2.V.weight
SHA_3.Q.weight, SHA_3.K.weight, SHA_3.V.weight
projection_layer.weight, projection_layer.bias
That is 4 heads * 3 matrices + 1 weight + 1 bias = 14 parameter tensors.

3 · The Key Relationship: head_size * num_heads = n_embd

This is the central constraint that makes multi-head attention work. The embedding dimension n_embd is divided equally among the heads. Each head operates in a subspace of dimension head_size, and when their outputs are concatenated, we recover the full n_embd dimension.

n_embd = 256, num_heads = 8 head_size = n_embd / num_heads = 256 / 8 = 32 Each head output: (B, S, 32) Concatenated: (B, S, 32 * 8) = (B, S, 256) = (B, S, n_embd)

Some more examples to build intuition:

n_embdnum_headshead_sizeEach head outputConcatenated
256464(B, S, 64)(B, S, 256)
256832(B, S, 32)(B, S, 256)
512864(B, S, 64)(B, S, 512)
7681264(B, S, 64)(B, S, 768)
10241664(B, S, 64)(B, S, 1024)
n_embd must be divisible by num_heads. If n_embd = 256 and num_heads = 7, then head_size = 256 / 7 = 36.57, which is not an integer. The shapes will not align during concatenation, and you will get a runtime error. Always choose num_heads that evenly divides n_embd.

Why split into subspaces?

A single head with head_size = 256 would have the same total dimension as 8 heads with head_size = 32. The total computation is roughly the same. So why split?

A single large head can represent complex attention patterns, but it compresses everything into one set of attention weights per token pair. With 8 heads, you get 8 independent attention weight matrices. Head 0 might learn "attend to the previous word." Head 1 might learn "attend to the subject of the clause." Head 2 might learn "attend to tokens with similar meaning." These patterns would compete for representation in a single head, but in multi-head attention, they coexist.

The original transformer paper tested different head counts. With n_embd = 512, using 8 heads of size 64 consistently outperformed 1 head of size 512, even though the total parameter count for QKV is the same. The diversity of attention patterns matters more than the capacity of any single pattern.

Visualizing the subspace split

Full embedding vector (n_embd = 256): [v0, v1, v2, ..., v31, v32, v33, ..., v63, v64, ..., v255] |---- head 0 -----| |---- head 1 -----| |-- head 7 --| Each head projects from the FULL 256 dims into its own 32-dim subspace. Head 0: (256) -> (32) via its own Q, K, V matrices Head 1: (256) -> (32) via different Q, K, V matrices ... Head 7: (256) -> (32) via different Q, K, V matrices The projection layer then recombines: (32*8 = 256) -> (256)
Important distinction: Each head does not receive a slice of the input. Every head receives the complete 256-dimensional input vector and projects it down to 32 dimensions using its own learned weight matrices. The splitting happens in the output space, not the input space.

4 · The Forward Pass Line by Line

Here is the complete forward method:

torch::Tensor forward(torch::Tensor x){ torch::Tensor output, concatenated_SHA_outputs; vector < torch::Tensor> SHA_outputs; for(int head_idx = 0; head_idx < num_heads; head_idx++){ SHA_outputs.push_back(SHA_group[head_idx]->forward(x)); } concatenated_SHA_outputs = torch::cat(SHA_outputs, -1); output = projection_layer(concatenated_SHA_outputs); return output; }

Let us walk through every line.

Line 1: Variable declarations

torch::Tensor output, concatenated_SHA_outputs; vector < torch::Tensor> SHA_outputs;

output will hold the final result after projection. concatenated_SHA_outputs will hold the concatenated result before projection. SHA_outputs is a vector that collects the output tensor from each individual head. These are declared upfront in the C style.

Line 2: The head loop

for(int head_idx = 0; head_idx < num_heads; head_idx++){ SHA_outputs.push_back(SHA_group[head_idx]->forward(x)); }

This loop iterates over every head. On each iteration:

  1. SHA_group[head_idx] retrieves the shared pointer to the head at index head_idx.
  2. ->forward(x) calls the SingleHeadAttention's forward method, passing the same input x.
  3. The result, a tensor of shape (B, S, head_size), is pushed into the SHA_outputs vector.

Every head receives the exact same tensor x. There is no cloning or copying needed because forward only reads from x (through Q, K, V linear projections) and never modifies it in-place. Each head's different Q, K, V weights are what produce different outputs.

The code uses for(int head_idx = 0; head_idx < num_heads; head_idx++) with explicit indexing rather than a range-for loop like for(auto& h : SHA_group). Both work identically. The index-based loop matches the style used in the constructor and makes the iteration variable's meaning explicit. In performance terms, there is no difference for a small vector of shared pointers.

Line 3: Concatenation

concatenated_SHA_outputs = torch::cat(SHA_outputs, -1);

torch::cat takes a vector of tensors and joins them along a specified dimension. The second argument, -1, means "the last dimension." For 3D tensors of shape (B, S, head_size), the last dimension is dimension 2 (the feature dimension).

SHA_outputs[0]: (B, S, head_size) e.g. (4, 16, 32) SHA_outputs[1]: (B, S, head_size) e.g. (4, 16, 32) SHA_outputs[2]: (B, S, head_size) e.g. (4, 16, 32) ... torch::cat(SHA_outputs, -1): (B, S, head_size * num_heads) e.g. (4, 16, 256)

Visually, for a single token (one row of the S dimension) in one batch element:

Head 0 output: [a0, a1, ..., a31] 32 values Head 1 output: [b0, b1, ..., b31] 32 values Head 2 output: [c0, c1, ..., c31] 32 values ... Head 7 output: [h0, h1, ..., h31] 32 values After cat(dim=-1): [a0,...,a31, b0,...,b31, c0,...,c31, ..., h0,...,h31] 256 values total
All dimensions except the last must match. If one head's output shape were (4, 16, 32) and another's were (4, 15, 32) (different sequence length), torch::cat would throw a runtime error. Since all heads receive the same input, they will always produce matching batch and sequence dimensions.

Line 4: Projection

output = projection_layer(concatenated_SHA_outputs);

The projection layer is a torch::nn::Linear with input size head_size * num_heads and output size n_embd. Since these are equal, it maps (B, S, n_embd) -> (B, S, n_embd).

What does this projection actually do? Without it, the output is just a raw concatenation of independent head outputs. The first 32 values come from head 0, the next 32 from head 1, and so on. There is no interaction between heads. The projection layer applies a weight matrix of shape (n_embd, n_embd):

output = concatenated * W_proj + b_proj W_proj shape: (n_embd, n_embd) = (256, 256) b_proj shape: (n_embd,) = (256,)

Each value in the output is a weighted sum of all 256 concatenated values. This means information from head 0 can mix with information from head 7. The model learns the optimal way to combine the perspectives of different heads.

The transformer architecture requires that the output of multi-head attention has the same shape as the input, because it is added via a residual connection: x = x + MHA(x). Since the input has last dimension n_embd, the output must also have last dimension n_embd. The projection layer guarantees this.

Line 5: Return

return output;

The projected output, with shape (B, S, n_embd), is returned. This feeds into the next layer of the transformer: typically a residual addition followed by layer normalization and then a feed-forward network.

7 · Why shared_ptr and Not unique_ptr?

You might wonder why we use shared_ptr<SingleHeadAttention> instead of unique_ptr<SingleHeadAttention>. After all, each head is "owned" by one MultiHeadAttention object, which sounds like a unique ownership scenario.

The short answer

LibTorch's register_module function requires a shared_ptr. If you try to pass a unique_ptr, the code will not compile.

The longer answer

LibTorch's module system needs shared ownership for good reasons:

When you call register_module("SHA_0", curr_head), the internal module registry stores a copy of the shared_ptr. Now two things hold a reference to the head: your SHA_group vector and the module registry. A unique_ptr cannot be shared, by definition. It would need to be moved into the registry, which would leave your vector with a null pointer.

When you call model->parameters(), LibTorch walks the module registry (not your vector) to discover all parameter tensors. The registry needs a valid pointer to each submodule. If you had moved a unique_ptr into the registry, your forward pass code (which accesses heads through the vector) would have dangling null pointers.

LibTorch follows the same pattern as PyTorch: modules are always managed through shared pointers. This enables advanced use cases like module sharing (using the same module in multiple places), module cloning, and serialization. Even if you do not need these features now, the API requires shared_ptr for consistency.

What would happen with unique_ptr?

// This will NOT compile: unique_ptr<SingleHeadAttention> head = make_unique<SingleHeadAttention>(n_embd, head_size); register_module("SHA_0", head); // ERROR: no matching overload // Even if you try to move: register_module("SHA_0", std::move(head)); // ERROR: register_module does not accept unique_ptr
Rule of thumb in LibTorch: Any torch::nn::Module subclass that you want to register as a submodule must be managed with shared_ptr. Use make_shared to create it, then pass the shared pointer to both register_module and your own storage (vector, member variable, etc.).

Memory safety with shared_ptr

With shared_ptr, the head object is destroyed only when both references (the vector and the registry) release it. This happens automatically when the parent MultiHeadAttention object is destroyed, since both the vector and the registry are members of the same object. There is no risk of use-after-free or memory leaks.

8 · Parameter Count

Let us count every learnable parameter in MultiHeadAttention. We will use concrete numbers: n_embd = 256, num_heads = 8, head_size = 32.

Per-head parameters (SingleHeadAttention)

Each head has three linear layers: Q, K, V. All three map from n_embd to head_size with no bias.

Q weight: n_embd * head_size = 256 * 32 = 8,192 K weight: n_embd * head_size = 256 * 32 = 8,192 V weight: n_embd * head_size = 256 * 32 = 8,192 (no bias terms) Per head total: 3 * 8,192 = 24,576

All heads combined

num_heads * per_head = 8 * 24,576 = 196,608

Equivalently, the total QKV parameter count can be written as:

num_heads * (3 * n_embd * head_size) = 8 * (3 * 256 * 32) = 8 * 24,576 = 196,608

Projection layer parameters

The projection layer maps (head_size * num_heads) to n_embd. With the default bias enabled:

projection weight: (head_size * num_heads) * n_embd = 256 * 256 = 65,536 projection bias: n_embd = 256 Projection total: 65,536 + 256 = 65,792

Since head_size * num_heads = n_embd, the projection weight is always n_embd * n_embd. This can also be written as n_embd^2 + n_embd.

Grand total

QKV (all heads): 196,608 Projection: 65,792 ------- Grand total: 262,400 parameters

General formula

Total = num_heads * (3 * n_embd * head_size) + (n_embd * n_embd + n_embd) Simplifying with head_size = n_embd / num_heads: Total = num_heads * (3 * n_embd * n_embd / num_heads) + n_embd^2 + n_embd = 3 * n_embd^2 + n_embd^2 + n_embd = 4 * n_embd^2 + n_embd
Observation: The total parameter count is 4 * n_embd^2 + n_embd, which is independent of the number of heads. Whether you use 4 heads of size 64 or 8 heads of size 32 (both with n_embd = 256), the total parameter count is the same. The number of heads affects the diversity of attention patterns, not the model size.

9 · Full Code

Both classes together, ready to compile. The SingleHeadAttention class is required as a dependency.

SingleHeadAttention (dependency)

class SingleHeadAttention : public torch::nn::Module{ int n_embd, head_size; torch::nn::Linear K{nullptr}, Q{nullptr}, V{nullptr}; public: SingleHeadAttention(int n_embd, int head_size) : n_embd(n_embd), head_size(head_size) { K = register_module("K", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false))); Q = register_module("Q", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false))); V = register_module("V", torch::nn::Linear( torch::nn::LinearOptions(n_embd, head_size).bias(false))); } torch::Tensor forward(torch::Tensor x){ int seq_len = x.size(1); torch::Tensor output, k_vec, q_vec, v_vec, raw_attn, mask; q_vec = Q(x); k_vec = K(x); v_vec = V(x); raw_attn = torch::matmul(q_vec, k_vec.transpose(-2, -1)) / ((float) sqrt(head_size)); mask = torch::tril(torch::ones({seq_len, seq_len}, x.options())); raw_attn.masked_fill_(mask == 0, -1e9f); output = torch::softmax(raw_attn, -1); output = torch::matmul(output, v_vec); return output; } };

MultiHeadAttention

class MultiHeadAttention : public torch::nn::Module{ /* It has access to multiple singlheadattention, and then it concatenates the output of all of these layer into one vector which then it passes to a projection layer input : batch_size x seq_len x n_embd output : batch_size x seq_len x (#of heads x head_size => n_embd) */ int num_heads, head_size, n_embd; vector< shared_ptr<SingleHeadAttention> > SHA_group; torch::nn::Linear projection_layer{nullptr}; public: MultiHeadAttention(int num_heads, int head_size, int n_embd) : num_heads(num_heads), head_size(head_size), n_embd(n_embd){ for (int head_idx = 0; head_idx < num_heads; head_idx++){ shared_ptr<SingleHeadAttention> curr_head = make_shared<SingleHeadAttention>(n_embd, head_size); register_module( "SHA_" + to_string(head_idx), curr_head ); SHA_group.push_back(curr_head); } projection_layer = register_module( "projection_layer", torch::nn::Linear( torch::nn::LinearOptions( head_size * num_heads, n_embd ) ) ); } torch::Tensor forward(torch::Tensor x){ torch::Tensor output, concatenated_SHA_outputs; vector < torch::Tensor> SHA_outputs; for(int head_idx = 0; head_idx < num_heads; head_idx++){ SHA_outputs.push_back( SHA_group[head_idx]->forward(x)); } concatenated_SHA_outputs = torch::cat(SHA_outputs, -1); output = projection_layer(concatenated_SHA_outputs); return output; } };

10 · What Comes Next

We now have the two core attention building blocks: SingleHeadAttention and MultiHeadAttention. The next component in the transformer block is the FeedForward network: a position-wise two-layer MLP that is applied independently to each token after attention.

  • FeedForward network: A two-layer MLP with a hidden expansion factor (typically 4x). Takes each token's representation from n_embd to 4 * n_embd and back to n_embd, with a nonlinearity in between.
  • Layer normalization: Stabilizes activations before attention and before the feed-forward network. The pre-norm variant (used in GPT-2) applies LayerNorm before each sub-layer.
  • Residual connections: Skip connections that add the input back to the output of each sub-layer: x = x + Attention(LayerNorm(x)) and x = x + FFN(LayerNorm(x)).
  • Full transformer block: Combining multi-head attention, feed-forward, layer norm, and residuals into one reusable block that can be stacked to build the complete model.
The pattern so far: SingleHeadAttention learns one attention pattern in a subspace. MultiHeadAttention runs several in parallel, concatenates, and projects. Next, the FeedForward network provides the non-linear transformation capacity that attention alone cannot provide. Together, attention plus feed-forward form the backbone of every transformer layer.
