Multi-Head Attention in C++ (LibTorch)

ML Series Transformers > From Scratch > C++ > Basic Blocks

In the previous post, we built a single attention head. A single head learns one type of relationship between tokens. Multi-head attention runs several heads in parallel, each learning different relationships, then combines their outputs. This page walks through the C++ implementation using LibTorch.

Why Multiple Heads?

A single attention head computes one set of attention weights per token pair. But language has many simultaneous relationships. In the sentence "The cat sat on the mat because it was tired", the word "it" relates to "cat" (coreference), to "sat" (subject of action), and to "tired" (attribute). A single head would need to compress all of these into one set of weights.

Multi-head attention solves this by running multiple independent heads. Each head maintains its own Q, K, V weight matrices and produces its own attention pattern. One head might learn syntactic dependencies, another might learn coreference, another might learn positional patterns.

Key idea: Instead of one head with a large head_size, we use many heads with smaller head_sizes. The total computation is comparable, but the model can learn richer, more diverse attention patterns.

The head_size Relationship

In practice, head_size is derived from the embedding dimension and the number of heads:

head_size = n_embd / num_heads

For example, with n_embd = 512 and num_heads = 8, each head has head_size = 64. After all 8 heads produce their (B, T, 64) outputs, we concatenate them along the last dimension to get back to (B, T, 512).

n_embd = 512, num_heads = 8 head_size = 512 / 8 = 64 Each head output: (B, T, 64) Concatenated: (B, T, 64 * 8) = (B, T, 512) = (B, T, n_embd)

This is why the concatenated output has the same last dimension as the input. The total information capacity is preserved: instead of one 512-dimensional attention, we get eight 64-dimensional attentions that together span the same space.

n_embd must be divisible by num_heads. If n_embd = 512 and num_heads = 7, that gives head_size = 73.14, which does not work. The division must be exact.

Class Structure

class MultiHeadAttention : public torch::nn::Module { vector<shared_ptr<SingleAttentionHead>> heads; torch::nn::Linear projection{nullptr}; int n_embd, num_heads, head_size; public: MultiHeadAttention(int n_embd, int num_heads, int head_size); torch::Tensor forward(torch::Tensor x); };

Let us look at each member:

vector<shared_ptr<SingleAttentionHead>> heads stores all the attention heads. We use shared_ptr because LibTorch's register_module expects a shared pointer to a Module. Each element in this vector is a fully independent SingleAttentionHead with its own Q, K, V weights.

torch::nn::Linear projection{nullptr} is the output projection layer. After concatenating all head outputs, this linear layer projects the concatenated result back to the model dimension. It maps (n_embd) to (n_embd). This layer lets the model learn how to combine information from the different heads.

The concatenated output of all heads has dimension num_heads * head_size = n_embd. The projection layer takes this n_embd-dimensional vector and produces another n_embd-dimensional vector. This is a learned linear combination of the concatenated heads. Without this projection, the outputs of different heads would remain independent. The projection allows cross-head information mixing.

Registering Heads in a Loop

MultiHeadAttention(int n_embd, int num_heads, int head_size) : n_embd(n_embd), num_heads(num_heads), head_size(head_size) { projection = register_module("projection", torch::nn::Linear(n_embd, n_embd)); for (int i = 0; i < num_heads; i++) { auto h = make_shared<SingleAttentionHead>(n_embd, head_size); register_module("head_" + to_string(i), h); heads.push_back(h); } }

There are several important details in this constructor.

Projection registration

The projection layer is registered just like in the SingleAttentionHead. Note that unlike the single attention head where we used torch::nn::LinearOptions(...).bias(false), here we use the simpler constructor torch::nn::Linear(n_embd, n_embd), which includes a bias term by default. This is standard: the projection layer typically has bias.

Loop registration with unique names

Each head is created with make_shared and registered with a unique name: "head_0", "head_1", "head_2", and so on. This is critical for two reasons:

  1. Unique naming: Every registered module must have a unique name within its parent. Using "head_" + to_string(i) guarantees unique names. If you accidentally registered two modules with the same name, LibTorch would throw a runtime error.
  2. Recursive registration: When you register a SingleAttentionHead as a submodule, all of its own registered modules (Q, K, V) become part of the parameter tree. The full parameter names become head_0.Q.weight, head_0.K.weight, head_1.Q.weight, etc. This is how LibTorch builds a nested parameter hierarchy.
Parameter hierarchy: After constructing MultiHeadAttention(512, 8, 64), calling named_parameters() produces entries like:
projection.weight, projection.bias
head_0.Q.weight, head_0.K.weight, head_0.V.weight
head_1.Q.weight, head_1.K.weight, head_1.V.weight
... all the way to head_7.Q.weight, head_7.K.weight, head_7.V.weight
That is 8 heads * 3 matrices + projection weight + projection bias = 26 parameter tensors total.

make_shared<SingleAttentionHead>(n_embd, head_size) is preferred over shared_ptr<SingleAttentionHead>(new SingleAttentionHead(n_embd, head_size)). Both work, but make_shared performs a single memory allocation for both the control block and the object, while new + shared_ptr does two allocations. In tight training loops with many module constructions, this difference adds up.

Technically, the order of register_module and push_back does not matter for correctness. But registering first is good practice. If register_module throws (e.g., duplicate name), the head is never added to the vector, keeping the two data structures in sync.

The Forward Pass

torch::Tensor forward(torch::Tensor x) { vector<torch::Tensor> heads_output; for (auto& h : heads) { heads_output.push_back(h->forward(x)); } torch::Tensor concatenated_output = torch::cat(heads_output, -1); torch::Tensor projected_output = projection(concatenated_output); return projected_output; }

The forward pass has three logical phases: run all heads, concatenate, then project. Let us examine each.

Phase 1: Run each head independently

vector<torch::Tensor> heads_output; for (auto& h : heads) { heads_output.push_back(h->forward(x)); // each produces (B, T, head_size) }

Each head receives the same input x of shape (B, T, n_embd). Each head produces an output of shape (B, T, head_size). We collect all these outputs into a vector.

Same input, different weights. Every head sees the same input tensor, but each has its own Q, K, V weight matrices. So each head computes different queries, keys, and values, resulting in different attention patterns and different outputs.

The auto& in the range-for loop takes a reference to each shared_ptr in the vector. Without the reference (auto), a copy of each shared_ptr would be made on every iteration, incrementing and decrementing reference counts. Using auto& avoids that overhead.

Concatenation Along Last Dimension

torch::Tensor concatenated_output = torch::cat(heads_output, -1);

torch::cat joins a list of tensors along a specified dimension. The -1 means the last dimension (the feature/head_size dimension).

Concrete example with 4 heads

n_embd = 256, num_heads = 4, head_size = 64 head_0 output: (B, T, 64) head_1 output: (B, T, 64) head_2 output: (B, T, 64) head_3 output: (B, T, 64) torch::cat({head_0, head_1, head_2, head_3}, -1): Concatenate along last dim: (B, T, 64+64+64+64) = (B, T, 256) This recovers the original embedding dimension n_embd = 256.

Visually, for a single token in a single batch element:

head_0: [a0, a1, ..., a63] (64 values) head_1: [b0, b1, ..., b63] (64 values) head_2: [c0, c1, ..., c63] (64 values) head_3: [d0, d1, ..., d63] (64 values) concatenated: [a0, ..., a63, b0, ..., b63, c0, ..., c63, d0, ..., d63] (256 values)

Using -1 (last dimension) instead of 2 makes the code dimension-agnostic. If the tensor ever gains extra dimensions (e.g., from different batching strategies), -1 still correctly refers to the feature dimension. It is the same reason we used -2, -1 in transpose in the single attention head.

All dimensions except the concatenation dimension must match. If one head accidentally produced (B, T, 63) instead of (B, T, 64), torch::cat would throw a runtime error. This is another reason why n_embd % num_heads == 0 must hold.

The Projection Layer

torch::Tensor projected_output = projection(concatenated_output); // projection: Linear(n_embd, n_embd) // (B, T, n_embd) -> (B, T, n_embd)

The concatenated output already has the right shape (B, T, n_embd). So why add a linear projection that maps n_embd to n_embd?

What the projection does

Without the projection, each position in the output is simply the raw concatenation of individual head outputs. The 64 values from head_0 sit next to the 64 values from head_1, but there is no interaction between them. The projection layer applies a learned weight matrix:

projected_output = concatenated_output * W_proj + b_proj W_proj shape: (n_embd, n_embd) = (256, 256) b_proj shape: (n_embd,) = (256,)

Each output value in the projected tensor is a weighted sum of all 256 concatenated values. This means information from head_0 can mix with information from head_3. The projection learns the best way to combine the different heads' perspectives.

Think of each head as an expert giving a recommendation. Without the projection, you just stack all recommendations side by side. With the projection, you have a learned "committee chair" that weighs and combines all expert opinions into a final decision. The projection learns which heads' outputs are most useful and how they should interact.

In the original paper: The "Attention is All You Need" paper defines the projection explicitly: MultiHead(Q, K, V) = Concat(head_1, ..., head_h) * W_O, where W_O is the output projection matrix. Our projection layer is exactly W_O.

Parameter count

Let us count the total parameters:

Per head: Q weight: n_embd * head_size = 512 * 64 = 32,768 K weight: n_embd * head_size = 512 * 64 = 32,768 V weight: n_embd * head_size = 512 * 64 = 32,768 (no bias in Q, K, V) Per head total: 98,304 8 heads: 8 * 98,304 = 786,432 Projection layer: Weight: n_embd * n_embd = 512 * 512 = 262,144 Bias: n_embd = 512 Projection total: 262,656 Grand total: 786,432 + 262,656 = 1,049,088 parameters

Full Code

#include <torch/torch.h> #include <vector> #include <memory> #include <string> #include <iostream> using namespace std; // Assumes SingleAttentionHead is defined as in the previous blog post class MultiHeadAttention : public torch::nn::Module { vector<shared_ptr<SingleAttentionHead>> heads; torch::nn::Linear projection{nullptr}; int n_embd, num_heads, head_size; public: MultiHeadAttention(int n_embd, int num_heads, int head_size) : n_embd(n_embd), num_heads(num_heads), head_size(head_size) { projection = register_module("projection", torch::nn::Linear(n_embd, n_embd)); for (int i = 0; i < num_heads; i++) { auto h = make_shared<SingleAttentionHead>(n_embd, head_size); register_module("head_" + to_string(i), h); heads.push_back(h); } } torch::Tensor forward(torch::Tensor x) { vector<torch::Tensor> heads_output; for (auto& h : heads) { heads_output.push_back(h->forward(x)); } torch::Tensor concatenated_output = torch::cat(heads_output, -1); torch::Tensor projected_output = projection(concatenated_output); return projected_output; } }; int main() { int n_embd = 512; int num_heads = 8; int head_size = n_embd / num_heads; // 64 auto mha = make_shared<MultiHeadAttention>(n_embd, num_heads, head_size); // Input: batch=4, seq_len=32, n_embd=512 torch::Tensor x = torch::randn({4, 32, n_embd}); torch::Tensor out = mha->forward(x); cout << "Input shape: " << x.sizes() << endl; cout << "Output shape: " << out.sizes() << endl; // Input: [4, 32, 512] // Output: [4, 32, 512] // Print parameter names for (auto& p : mha->named_parameters()) { cout << p.key() << " : " << p.value().sizes() << endl; } }

What Comes Next

We now have the two core attention building blocks: a single head and multi-head attention. The next pieces of the transformer block are:

  • Feed-forward network: A position-wise two-layer MLP applied independently to each token after attention.
  • Layer normalization: Stabilizes activations. Pre-norm (before attention/FFN) vs post-norm (after).
  • Residual connections: Skip connections that add the input back to the output of each sub-layer.
  • Full transformer block: Combining multi-head attention + FFN + layer norm + residuals into one reusable block.
