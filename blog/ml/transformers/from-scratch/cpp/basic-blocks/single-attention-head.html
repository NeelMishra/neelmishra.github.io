Single Attention Head in C++ (LibTorch)

ML Series Transformers > From Scratch > C++ > Basic Blocks

What is an Attention Head?

Consider the sentence: "The cat sat on the mat because it was soft." When a human reads the word "it", they instantly know it refers to "mat" (not "cat" and not "sat"). An attention head is the mechanism that lets a model make that same connection.

Each token in a sequence asks a question: "Which other tokens should I pay attention to?" The attention head answers that question by computing a relevance score between every pair of tokens, then using those scores to blend information together.

The process has three steps: each token produces a Query (what am I looking for?), a Key (what do I contain?), and a Value (what information do I carry?). Queries and Keys are compared to get scores. Scores determine how much of each Value to take.

In code, all of this is a matrix multiplication, a mask, and a softmax. The rest of this page builds it from scratch in C++ using LibTorch.

Class Structure

class SingleAttentionHead : public torch::nn::Module { int n_emb = 32; int head_size = 16; torch::nn::Linear Q{nullptr}, K{nullptr}, V{nullptr}; public: SingleAttentionHead(int n_emb, int head_size) : n_emb(n_emb), head_size(head_size) { this->Q = register_module("Q", torch::nn::Linear( torch::nn::LinearOptions(n_emb, head_size).bias(false))); this->K = register_module("K", torch::nn::Linear( torch::nn::LinearOptions(n_emb, head_size).bias(false))); this->V = register_module("V", torch::nn::Linear( torch::nn::LinearOptions(n_emb, head_size).bias(false))); } torch::Tensor forward(torch::Tensor x); };

We declare Q{nullptr}, K{nullptr}, V{nullptr} because torch::nn::Linear requires dimensions at construction time. We cannot construct it as a member initializer without knowing n_emb and head_size. So we declare them as null first and assign real layers inside the constructor body.

.bias(false) matches the original "Attention is All You Need" paper. The Q, K, V projections compute Q = X * W_Q with no added bias vector. This is standard practice. The bias would add a constant offset to every query/key/value, which is redundant since the embeddings already capture position and meaning.

Why nn::Linear, Not Raw Tensors?

We could represent Q, K, V as plain torch::Tensor weight matrices instead of torch::nn::Linear modules. Both can perform the same matrix multiplication. But there are fundamental differences.

The raw tensor approach

// Using raw tensors as weights: torch::Tensor W_Q = torch::randn({n_emb, head_size}); torch::Tensor W_K = torch::randn({n_emb, head_size}); torch::Tensor W_V = torch::randn({n_emb, head_size}); // Forward pass: torch::Tensor Q_vec = torch::matmul(x, W_Q); // manual matmul torch::Tensor K_vec = torch::matmul(x, W_K); torch::Tensor V_vec = torch::matmul(x, W_V);

This works mathematically. torch::matmul(x, W_Q) multiplies the last dimension of x (n_emb) by the first dimension of W_Q (n_emb), producing a result with the last dimension as head_size. Same linear projection.

So why not use it?

A plain torch::Tensor created by torch::randn does not track gradients by default. You would need to explicitly set W_Q.set_requires_grad(true). Then you would need to manually pass each tensor to the optimizer. With nn::Linear, the weight tensor is created with requires_grad = true automatically.

torch::nn::Linear is a Module. When you register it, all its internal parameters become discoverable via model->parameters(). A raw tensor is just data. The optimizer and the module system have no way to find it. You would have to manually build your own parameter list and pass it to the optimizer.

nn::Linear initializes weights using Kaiming uniform initialization (by default), which is designed for stable training in deep networks. A raw torch::randn uses a standard normal distribution, which can cause exploding or vanishing activations in deep models. You would need to implement your own initialization.

When you call model->to(torch::kCUDA), all registered Linear modules move to GPU automatically. A raw tensor stays wherever it was created. You would need to manually track and move every weight tensor.

torch::save and torch::load work with modules. Every registered parameter gets a name and can be saved/loaded as part of a checkpoint. Raw tensors would need manual serialization logic. Missing one tensor during save means a corrupted checkpoint.

Bottom line: torch::nn::Linear wraps a weight tensor (and optional bias) inside a module that integrates with the entire LibTorch ecosystem: optimizers, checkpointing, device management, and the module hierarchy. Using raw tensors means reimplementing all of that yourself. For Q, K, V in attention, there is no reason to avoid nn::Linear.

What nn::Linear actually stores

Under the hood, torch::nn::Linear(n_emb, head_size) creates:

// Inside nn::Linear: torch::Tensor weight; // shape: (head_size, n_emb) -- NOTE: transposed torch::Tensor bias; // shape: (head_size,) -- omitted when .bias(false)

When you call Q(x), it computes x @ weight.T + bias. Notice the weight is stored as (head_size, n_emb) and transposed during the forward pass. This is a PyTorch convention for memory layout efficiency. With a raw tensor, you would store it as (n_emb, head_size) and compute x @ W_Q without transposing. Both produce the same result, but the internal layout differs.

Inside register_module

register_module is a critical LibTorch function. It does three things:

  1. Parameter registration: It adds all the weights of the linear layer to the module's parameter list. When you later call model->parameters(), the optimizer finds these weights and updates them during training. Without registration, the weights are invisible to the training loop.
  2. Named hierarchy: The string argument ("Q", "K", "V") creates a named hierarchy. When you print the model or save/load weights, each layer is identified by name. For example, saving produces keys like Q.weight, K.weight, V.weight.
  3. Device tracking: When you call model->to(torch::kCUDA), all registered modules are moved to GPU. Unregistered layers stay on CPU, causing device mismatch errors.
// What happens without register_module: torch::nn::Linear Q = torch::nn::Linear(32, 16); // NOT registered model->parameters(); // Q's weights are NOT in this list optimizer.step(); // Q's weights are NEVER updated model->to(torch::kCUDA); // Q stays on CPU, everything else on GPU // What happens with register_module: Q = register_module("Q", torch::nn::Linear(32, 16)); model->parameters(); // Q's weights ARE in this list optimizer.step(); // Q's weights ARE updated model->to(torch::kCUDA); // Q moves to GPU correctly
Deep dive: For a full explanation of register_module, named_parameters, and the LibTorch module system, see the dedicated register_module deep dive.

The Forward Pass

The forward method receives a 3D tensor x with shape (B, T, n_emb) where B is the batch size (number of sequences processed in parallel), T is the sequence length (number of tokens in each sequence), and n_emb is the embedding dimension (the size of each token's feature vector).

torch::Tensor forward(torch::Tensor x) { // Step 1: Project into Q, K, V torch::Tensor Q_vec = this->Q(x); // (B, T, n_emb) -> (B, T, head_size) torch::Tensor K_vec = this->K(x); // (B, T, n_emb) -> (B, T, head_size) torch::Tensor V_vec = this->V(x); // (B, T, n_emb) -> (B, T, head_size) // Step 2: Compute scaled attention scores torch::Tensor raw_attention = torch::matmul(Q_vec, K_vec.transpose(-2, -1)) // (B, T, T) / sqrt((float)(head_size)); // Step 3: Apply causal mask torch::Tensor mask = torch::tril( torch::ones({x.size(1), x.size(1)}) ); raw_attention.masked_fill_(mask == 0, -1e9f); // Step 4: Softmax over last dimension torch::Tensor weights = torch::softmax(raw_attention, -1); // Step 5: Weighted sum of values torch::Tensor out = torch::matmul(weights, V_vec); // (B, T, head_size) return out; }

Why transpose(-2, -1)?

We need to compute the dot product between every query and every key. If Q has shape (B, T, head_size), each row is a query vector for one token. To dot every query with every key, we need:

scores[i][j] = dot(Q[i], K[j]) This is exactly matrix multiplication: Q * K^T

But our tensors are 3D (they have a batch dimension). We only want to transpose the last two dimensions (the token and feature dimensions), not the batch dimension. That is what .transpose(-2, -1) does:

K_vec shape: (B, T, head_size) K_vec.transpose(-2, -1) shape: (B, head_size, T) matmul(Q_vec, K_vec.transpose(-2, -1)): (B, T, head_size) x (B, head_size, T) = (B, T, T)

Negative indices count from the end. For a 3D tensor with dimensions [0, 1, 2]:
-1 is dimension 2 (head_size)
-2 is dimension 1 (T)
So .transpose(-2, -1) swaps dimensions 1 and 2, leaving the batch dimension (0) untouched. This works regardless of whether the tensor is 3D, 4D, etc.

Why not .t()? The .t() method only works on 2D matrices. For batched tensors (3D+), you must use .transpose(dim0, dim1) to specify exactly which two dimensions to swap.

Why (float) Cast on head_size?

The line in question:

/ sqrt((float)(head_size))

head_size is an int. In C++, sqrt(int) can be ambiguous: the compiler might pick sqrt(double) or sqrt(float), or it might not compile at all depending on overload resolution.

More importantly, the division tensor / sqrt(int) can produce unexpected results. The sqrt of an int returns a double in C++. Dividing a float tensor by a double can trigger implicit type promotions. By explicitly casting to float:

  • We get sqrt(float) which returns a float
  • The division stays in float precision, matching the tensor's dtype (torch::kFloat32)
  • No ambiguous overload warnings from the compiler
The value itself: For head_size = 16, sqrt(16.0f) = 4.0f. We are dividing all attention scores by 4. This scaling prevents the dot products from growing proportionally with the dimension, which would push softmax into saturated regions where gradients vanish.

Why Softmax Along the Last Dimension

The softmax line is:

torch::Tensor weights = torch::softmax(raw_attention, -1); // -1 means the last dimension

raw_attention has shape (B, T, T). Think of it as: for each token (row), there is a score against every other token (columns). We want each row to become a probability distribution that sums to 1.

raw_attention for one sequence, one token (row 2): [0.4, 0.9, 0.6] (scores against tokens 0, 1, 2) After softmax along last dim (across columns): [0.22, 0.37, 0.27] (probabilities, sum = 1.0) Meaning: token 2 pays 22% attention to token 0, 37% to token 1, and 27% to itself.

What if we used a different dimension?

raw_attention shape: (B, T, T) dim = -1 (columns): Each ROW sums to 1 -> "For each token, how much should it attend to each other token?" -> This is what we want. dim = -2 (rows): Each COLUMN sums to 1 -> "For each token, how much attention does it RECEIVE from others?" -> Not useful. We want to control where each token LOOKS, not how much attention it receives. dim = 0 (batch): Normalizes across batch elements -> Completely wrong. Different sequences should be independent.

Row i of the attention matrix contains the raw scores of token i against all other tokens. Softmax along dim -1 converts that row into a probability distribution. The result is: token i attends to token j with probability weights[i][j]. These probabilities are then used to compute a weighted average of the Value vectors, which becomes token i's output.

The -1 convention: In LibTorch (and PyTorch), -1 always refers to the last dimension. For a (B, T, T) tensor, -1 is dimension 2 (the columns). This is the standard choice for attention softmax across virtually all transformer implementations.

How masked_fill_ Works Behind the Scenes

masked_fill_ is an in-place operation. The underscore suffix is LibTorch/PyTorch convention for in-place (mutating the tensor rather than creating a copy).

raw_attention.masked_fill_(mask == 0, -1e9f);

This is equivalent to the following element-wise operation:

for each element (i, j) in raw_attention: if mask[i][j] == 0: raw_attention[i][j] = -1e9f // else: leave it unchanged

Step by step for a 3-token sequence

raw_attention (before masking): mask (lower triangular): [[0.5, 0.3, 0.8], [[1, 0, 0], [0.2, 0.7, 0.1], [1, 1, 0], [0.4, 0.9, 0.6]] [1, 1, 1]] mask == 0: [[0, 1, 1], [0, 0, 1], [0, 0, 0]] raw_attention (after masked_fill_): [[0.5, -1e9, -1e9], [0.2, 0.7, -1e9], [0.4, 0.9, 0.6]]

After softmax, exp(-1e9) rounds to 0, so those positions contribute zero attention weight. Token 0 can only attend to itself. Token 1 can attend to tokens 0 and 1. Token 2 can attend to all three.

-INFINITY would also work mathematically (exp(-inf) = 0). However, in practice, some edge cases (like when an entire row is masked, or during gradient computation) can produce NaN with true infinity. Using -1e9 (a very large negative number) is numerically safer while being practically equivalent: exp(-1e9) is so close to 0 that it is zero in float32 representation.

The underscore matters. masked_fill_ (with underscore) modifies the tensor in place. masked_fill (without underscore) returns a new tensor. The in-place version is faster (no memory allocation) but means the original raw_attention is destroyed. In a forward pass where we do not need the unmasked scores again, in-place is the right choice.

Mask Dimensions and Broadcasting

The mask is created as a 2D tensor:

torch::Tensor mask = torch::tril(torch::ones({x.size(1), x.size(1)})); // x.size(1) = T (sequence length) // mask shape: (T, T)

But raw_attention has shape (B, T, T). How does a (T, T) mask work on a (B, T, T) tensor?

Broadcasting

LibTorch (like PyTorch and NumPy) uses broadcasting. When dimensions do not match, it automatically expands the smaller tensor by repeating it along the missing dimensions:

raw_attention shape: (B, T, T) = (4, 64, 64) mask shape: (T, T) = (64, 64) Broadcasting adds a batch dimension: mask is treated as: (1, T, T) = (1, 64, 64) Then repeated B times: effective mask: (B, T, T) = (4, 64, 64)

This means the same mask is applied to every sequence in the batch. This is correct because the causal constraint (do not look at future tokens) is the same for all sequences.

Why not create the mask as (B, T, T)? We could, but it would waste memory. Since the mask is identical for every batch element, a single (T, T) matrix that gets broadcast is more efficient. LibTorch handles the broadcast internally without physically copying the data B times.

Output Shape

The final output has shape:

out.shape = (B, T, head_size)

Let us trace why:

weights: (B, T, T) - attention weights (how much each token attends to each other) V_vec: (B, T, head_size) - value vectors matmul(weights, V_vec): (B, T, T) x (B, T, head_size) = (B, T, head_size)

So the output has the same batch and sequence dimensions as the input, but the last dimension changes from n_emb to head_size. Each token's output is a weighted average of all the value vectors, where the weights come from the attention scores.

TensorShapeMeaning
Input x(B, T, n_emb)Original token embeddings
Output(B, T, head_size)Context-aware token representations
In multi-head attention, we typically set head_size = n_emb / num_heads. With 8 heads and n_emb = 512, each head produces (B, T, 64). The 8 outputs are concatenated back to (B, T, 512), recovering the original dimension.

Full Code

#include <torch/torch.h> class SingleAttentionHead : public torch::nn::Module { int n_emb = 32; int head_size = 16; torch::nn::Linear Q{nullptr}, K{nullptr}, V{nullptr}; public: SingleAttentionHead(int n_emb, int head_size) : n_emb(n_emb), head_size(head_size) { this->Q = register_module("Q", torch::nn::Linear( torch::nn::LinearOptions(n_emb, head_size).bias(false))); this->K = register_module("K", torch::nn::Linear( torch::nn::LinearOptions(n_emb, head_size).bias(false))); this->V = register_module("V", torch::nn::Linear( torch::nn::LinearOptions(n_emb, head_size).bias(false))); } torch::Tensor forward(torch::Tensor x) { torch::Tensor Q_vec = this->Q(x); torch::Tensor K_vec = this->K(x); torch::Tensor V_vec = this->V(x); torch::Tensor raw_attention = torch::matmul(Q_vec, K_vec.transpose(-2, -1)) / sqrt((float)head_size); torch::Tensor mask = torch::tril( torch::ones({x.size(1), x.size(1)})); raw_attention.masked_fill_(mask == 0, -1e9f); torch::Tensor weights = torch::softmax(raw_attention, -1); torch::Tensor out = torch::matmul(weights, V_vec); return out; } }; int main() { auto head = std::make_shared<SingleAttentionHead>(32, 16); // Input: batch=2, seq_len=8, n_emb=32 torch::Tensor x = torch::randn({2, 8, 32}); torch::Tensor out = head->forward(x); std::cout << "Input shape: " << x.sizes() << std::endl; std::cout << "Output shape: " << out.sizes() << std::endl; // Input: [2, 8, 32] // Output: [2, 8, 16] }
