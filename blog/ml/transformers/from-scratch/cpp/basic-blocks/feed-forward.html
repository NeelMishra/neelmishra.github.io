Feed-Forward Network in C++ (LibTorch)

ML Series Transformers > From Scratch > C++ > Basic Blocks

In the previous post, we built multi-head attention. Attention lets tokens communicate with each other. But after tokens gather information from their neighbors, each token needs to individually process what it collected. That is the job of the feed-forward network (FFN). This page walks through the C++ implementation.

Role in the Transformer

Every transformer block has two sub-layers: multi-head attention followed by a feed-forward network. Attention handles inter-token communication (tokens talking to each other). The FFN handles intra-token computation (each token thinking on its own).

Transformer Block: 1. Multi-Head Attention (tokens communicate) 2. Feed-Forward Network (each token processes independently)

The FFN is applied to each token position independently and identically. The same weights are shared across all positions. Token 0 goes through the same FFN as token 42. This is why it is often called a "position-wise" feed-forward network.

Analogy: Attention is a meeting where everyone shares information. The FFN is private thinking time afterward, where each person individually processes what they heard using the same reasoning framework.

Why Expand Then Compress?

The FFN has a distinctive shape: it expands the dimension, applies an activation, then compresses back. The standard configuration from "Attention is All You Need" uses d_ff = 4 * n_embd.

n_embd = 512 linear_1: (512) -> (2048) expand by 4x ReLU activation linear_2: (2048) -> (512) compress back This is called a "bottleneck in reverse" or "expand-compress" architecture.

Why not just have a single linear layer from n_embd to n_embd? Because a single linear layer can only learn linear transformations. No matter how large the weight matrix, y = Wx + b is always a linear function. By expanding to a higher dimension, applying a non-linear activation (ReLU), and then compressing, the network can learn non-linear transformations.

In the higher-dimensional space (d_ff = 2048), the network has more "room" to represent complex patterns. Think of it as temporarily unfolding a compressed representation, manipulating it in a richer space, and then folding it back. ReLU selectively zeroes out some of the 2048 dimensions, effectively choosing which features matter for each token.

The original paper used d_ff = 4 * n_embd. This is an empirical choice, not a mathematical necessity. Research has shown that the 4x ratio provides a good trade-off between model capacity and computational cost. Some later architectures use different ratios: GPT-3 uses 4x, LLaMA uses 8/3x (with SwiGLU activation), and some efficient models use 2x.

Class Structure

class FeedForward : public torch::nn::Module { int n_embd, d_ff; torch::nn::Linear linear_1{nullptr}, linear_2{nullptr}; public: FeedForward(int d_ff, int n_embd); torch::Tensor forward(torch::Tensor x); };

The structure is refreshingly simple compared to multi-head attention. Just two linear layers and two dimension values.

linear_1 maps from n_embd to d_ff. This is the expansion. With the default 4x ratio, it takes a 512-dimensional vector and produces a 2048-dimensional vector. Each output value is a learned weighted sum of all 512 input values plus a bias term.

linear_2 maps from d_ff back to n_embd. This compresses the 2048-dimensional activated representation back to 512 dimensions. The output has the same dimension as the input, which is essential for the residual connection in the transformer block.

Both layers include bias by default. Unlike the Q, K, V projections in the attention head (where we explicitly set .bias(false)), the FFN layers use the default constructor torch::nn::Linear(in, out), which includes a bias term. This is standard practice for the FFN.

Registration Order

FeedForward(int d_ff, int n_embd) : d_ff(d_ff), n_embd(n_embd) { linear_1 = torch::nn::Linear(n_embd, d_ff); linear_2 = torch::nn::Linear(d_ff, n_embd); register_module("linear_1", linear_1); register_module("linear_2", linear_2); }

Notice a subtle difference from the previous blogs. In the SingleAttentionHead, we wrote:

// SingleAttentionHead style: register_module returns the module this->Q = register_module("Q", torch::nn::Linear(...));

Here, the FeedForward class constructs the layers first, then registers them separately:

// FeedForward style: construct first, then register linear_1 = torch::nn::Linear(n_embd, d_ff); register_module("linear_1", linear_1);

Are these equivalent?

Yes. Both approaches achieve the same result. register_module returns the module holder it receives, so you can use the return value for assignment (the attention head style) or ignore it and assign separately (the FFN style). The only requirement is that register_module is called at some point before training begins, so the parameters are visible to the optimizer.

What would go wrong without registration? If you skip register_module, the layers still work in the forward pass. But: (1) the optimizer never sees the weights, so they never update during training; (2) model->to(torch::kCUDA) does not move them to GPU; (3) saving the model does not include them. The model appears to "work" but never learns.

The Forward Pass

torch::Tensor forward(torch::Tensor x) { torch::Tensor output = linear_1(x); // (B, T, n_embd) -> (B, T, d_ff) output = torch::relu(output); // (B, T, d_ff) -> (B, T, d_ff) output = linear_2(output); // (B, T, d_ff) -> (B, T, n_embd) return output; }

Three lines. Expand, activate, compress. Let us walk through each.

Step 1: Expand via linear_1

linear_1(x) applies the first linear transformation. The input x has shape (B, T, n_embd). The linear layer only transforms the last dimension, so the output is (B, T, d_ff). Every token is independently projected from n_embd dimensions to d_ff dimensions.

output = x * W1^T + b1 W1 shape: (d_ff, n_embd) = (2048, 512) b1 shape: (d_ff,) = (2048,) For each token: [512 values] -> [2048 values]

Step 2: ReLU activation

torch::relu(output) applies the Rectified Linear Unit element-wise. This is the non-linearity that makes the FFN more than just a linear transformation. We cover ReLU in detail in the next section.

Step 3: Compress via linear_2

linear_2(output) projects from d_ff back to n_embd:

output = activated * W2^T + b2 W2 shape: (n_embd, d_ff) = (512, 2048) b2 shape: (n_embd,) = (512,) For each token: [2048 values] -> [512 values]

The code reuses the variable output across all three lines. In C++, each reassignment overwrites the previous tensor. LibTorch's autograd still tracks the computation graph correctly because the graph is built on tensor operations, not variable names. This is equivalent to writing return linear_2(torch::relu(linear_1(x))) as a single expression.

Why ReLU?

torch::relu applies the function:

ReLU(x) = max(0, x) If x > 0: output = x (pass through unchanged) If x <= 0: output = 0 (kill the value)

What ReLU does to the 2048-dimensional vector

After linear_1, we have a 2048-dimensional vector for each token. ReLU zeros out all negative values. Empirically, roughly half the values are negative (before training converges), so about half the dimensions are set to zero. This creates a sparse activation pattern.

Before ReLU: [0.3, -1.2, 0.8, -0.1, 2.5, -3.0, 0.0, 1.1, ...] After ReLU: [0.3, 0.0, 0.8, 0.0, 2.5, 0.0, 0.0, 1.1, ...]

Different tokens activate different subsets of the 2048 dimensions. This means the FFN learns a kind of "feature dictionary": each dimension in d_ff corresponds to a feature, and ReLU selects which features are relevant for each token.

ReLU is preferred over sigmoid and tanh for several reasons: (1) It does not saturate for positive values, so gradients flow freely during backpropagation. Sigmoid and tanh squash large values, causing vanishing gradients. (2) It is computationally cheap: just a comparison with zero. (3) It creates sparsity, which has been shown to help with feature learning in deep networks.

Modern alternatives: The original transformer uses ReLU. Newer architectures often replace it with GELU (Gaussian Error Linear Unit) or SwiGLU. GELU provides smoother gradients near zero. SwiGLU (used in LLaMA, PaLM) combines a gated mechanism with a smooth activation. Our implementation follows the original paper with ReLU.

Why the non-linearity is essential

Without the ReLU, the FFN degenerates to a single linear layer:

Without ReLU: output = linear_2(linear_1(x)) = (x * W1^T + b1) * W2^T + b2 = x * (W1^T * W2^T) + (b1 * W2^T + b2) = x * W_combined + b_combined This is just one linear transformation. Two linear layers without a non-linearity between them collapse into one. The expand-compress architecture would be meaningless.

ReLU breaks this collapse. The output becomes a non-linear function of the input, allowing the FFN to learn complex feature transformations that a single linear layer cannot represent.

Input shape = Output shape. The FFN takes (B, T, n_embd) and returns (B, T, n_embd). Just like multi-head attention, this is required for the residual connection: x = x + FFN(x). Every sub-layer in the transformer preserves the shape.

Position-wise: What It Really Means

The FFN is called "position-wise" because it operates independently on each token position. There is no interaction between tokens in the FFN. Token 5 does not see token 12. That is the attention layer's job.

For input x of shape (B, T, n_embd): Token at position 0: x[:, 0, :] -> FFN -> output[:, 0, :] Token at position 1: x[:, 1, :] -> FFN -> output[:, 1, :] ... Token at position T-1: x[:, T-1, :] -> FFN -> output[:, T-1, :] Each uses the SAME weights W1, b1, W2, b2. No information flows between positions.

This happens naturally because torch::nn::Linear only transforms the last dimension. The batch and sequence dimensions are preserved independently.

In convolutional neural network terms, a position-wise FFN is identical to a 1x1 convolution over the sequence dimension with d_ff filters. The 1x1 convolution processes each spatial position independently using shared weights. This is the same operation, just described in different frameworks.

Why is this useful? After attention, each token's representation is a mix of information from other tokens. But this mixing is purely linear (weighted sum of value vectors). The FFN adds non-linear processing to each token, allowing the model to transform the mixed representation into something more useful. The attention says "what to combine." The FFN says "how to process the combination."

Parameter Count

The FFN is the most parameter-heavy component in a transformer block. Let us count:

linear_1: Weight: d_ff * n_embd = 2048 * 512 = 1,048,576 Bias: d_ff = 2048 Subtotal: 1,050,624 linear_2: Weight: n_embd * d_ff = 512 * 2048 = 1,048,576 Bias: n_embd = 512 Subtotal: 1,049,088 FFN total: 2,099,712 parameters

Compare this to multi-head attention with the same n_embd = 512:

Multi-Head Attention: ~1,049,088 parameters Feed-Forward Network: ~2,099,712 parameters The FFN has roughly 2x the parameters of the attention layer.

This is a commonly overlooked fact. In transformer architectures, the FFN dominates the parameter count. About two-thirds of a transformer block's parameters are in the FFN. This is why techniques like mixture-of-experts (MoE) focus on making the FFN more efficient.

Full Code

#include <torch/torch.h> #include <iostream> using namespace std; class FeedForward : public torch::nn::Module { int n_embd, d_ff; torch::nn::Linear linear_1{nullptr}, linear_2{nullptr}; public: FeedForward(int d_ff, int n_embd) : d_ff(d_ff), n_embd(n_embd) { linear_1 = torch::nn::Linear(n_embd, d_ff); linear_2 = torch::nn::Linear(d_ff, n_embd); register_module("linear_1", linear_1); register_module("linear_2", linear_2); } torch::Tensor forward(torch::Tensor x) { torch::Tensor output = linear_1(x); output = torch::relu(output); output = linear_2(output); return output; } }; int main() { int n_embd = 512; int d_ff = 4 * n_embd; // 2048 auto ffn = make_shared<FeedForward>(d_ff, n_embd); // Input: batch=4, seq_len=32, n_embd=512 torch::Tensor x = torch::randn({4, 32, n_embd}); torch::Tensor out = ffn->forward(x); cout << "Input shape: " << x.sizes() << endl; cout << "Output shape: " << out.sizes() << endl; // Input: [4, 32, 512] // Output: [4, 32, 512] // Print parameter names and shapes for (auto& p : ffn->named_parameters()) { cout << p.key() << " : " << p.value().sizes() << endl; } // linear_1.weight : [2048, 512] // linear_1.bias : [2048] // linear_2.weight : [512, 2048] // linear_2.bias : [512] }

What Comes Next

We now have both sub-layers of a transformer block: multi-head attention and the feed-forward network. The remaining pieces are:

  • Layer normalization: Stabilizes training by normalizing activations before or after each sub-layer.
  • Residual connections: Skip connections that add the sub-layer input to its output, preventing gradient degradation.
  • Full transformer block: Combining attention + FFN + layer norm + residuals into one complete, stackable block.
