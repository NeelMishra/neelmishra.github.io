Transformer Block in C++ (LibTorch)

ML Series Transformers > From Scratch > C++ > Basic Blocks

In the previous posts we built multi-head attention and the feed-forward network. This page combines them into a complete transformer block: the fundamental repeatable unit that gets stacked to form a full transformer. We add layer normalization and residual connections, the two structural ingredients that make deep stacking work.

The Big Picture

A transformer is built by stacking identical blocks. Each block has the same internal structure:

Input x | +---> LayerNorm ---> Multi-Head Attention ---> + (add back x) | | v v x' = x + MHA(LN(x)) | +---> LayerNorm ---> Feed-Forward -----------> + (add back x') | | v v x'' = x' + FFN(LN(x')) | Output x''

The block takes a tensor of shape (B, T, n_embd) and returns a tensor of the same shape. This shape preservation is what allows blocks to be stacked: the output of block 1 feeds directly into block 2, then into block 3, and so on.

How many blocks? GPT-2 Small uses 12 blocks. GPT-2 Large uses 36. GPT-3 uses 96. The number of blocks is one of the primary scaling knobs. More blocks mean more capacity for learning complex patterns, at the cost of more parameters and compute.

Class Structure

class TransformerBlock : public torch::nn::Module { shared_ptr<MultiHeadAttention> mha; shared_ptr<FeedForward> ff; torch::nn::LayerNorm ln1{nullptr}, ln2{nullptr}; public: TransformerBlock(int n_embd, int num_heads, int head_size, int d_ff); torch::Tensor forward(torch::Tensor x); };

Four components, each with a distinct role:

shared_ptr<MultiHeadAttention> mha is the attention sub-layer. It lets tokens communicate with each other. We store it as a shared_ptr because our MultiHeadAttention class is registered as a submodule, and LibTorch's module registration system works with shared pointers.

shared_ptr<FeedForward> ff is the FFN sub-layer. It processes each token independently through expand-activate-compress. Same shared_ptr reasoning as above.

torch::nn::LayerNorm ln1, ln2 are two separate layer normalization modules. ln1 normalizes before attention. ln2 normalizes before the FFN. They have their own learned parameters (gamma and beta) and are initialized to nullptr, then constructed in the body.

Each LayerNorm learns its own scale (gamma) and shift (beta) parameters. The normalization before attention might learn different scaling than the normalization before the FFN. Using the same LayerNorm for both would force them to share these learned parameters, which would be a constraint with no benefit.

LayerNorm: What and Why

Layer normalization normalizes the features of each token independently. For a single token with feature vector [x_1, x_2, ..., x_n_embd]:

Step 1: Compute mean across features mean = (x_1 + x_2 + ... + x_n_embd) / n_embd Step 2: Compute variance across features var = sum((x_i - mean)^2) / n_embd Step 3: Normalize x_norm_i = (x_i - mean) / sqrt(var + epsilon) Step 4: Scale and shift (learned parameters) output_i = gamma_i * x_norm_i + beta_i

After normalization, the feature vector has approximately zero mean and unit variance. The learned gamma and beta allow the model to undo the normalization if that turns out to be better.

Why normalize at all?

As data flows through many layers, the distribution of activations can shift dramatically. One layer might produce values in the range [1000, 2000], while another produces [-0.001, 0.001]. This makes training unstable because:

  • The learning rate that works for one scale is too large or too small for another.
  • Gradients can explode or vanish based on activation scale.
  • Softmax and other non-linearities behave differently at different scales.

LayerNorm keeps activations in a consistent range, making the optimization landscape smoother and training more stable.

BatchNorm normalizes across the batch dimension: for each feature, compute mean and variance across all samples in the batch. LayerNorm normalizes across the feature dimension: for each sample (token), compute mean and variance across all features. LayerNorm is preferred in transformers because (1) it works with variable sequence lengths, (2) it does not depend on batch size, and (3) it behaves identically during training and inference (BatchNorm uses running statistics at inference time).

The epsilon value: The epsilon in sqrt(var + epsilon) prevents division by zero when the variance is extremely small. LibTorch defaults epsilon to 1e-5. This is a numerical stability safeguard, not a tunable hyperparameter.

LayerNormOptions Explained

ln1 = torch::nn::LayerNorm(torch::nn::LayerNormOptions({n_embd}));

The argument to LayerNormOptions specifies the normalized shape: the dimensions over which to compute the mean and variance.

Input shape: (B, T, n_embd) = (4, 32, 512) LayerNormOptions({512}) means: Normalize over the last dimension (size 512) Each token's 512-element feature vector is independently normalized LayerNormOptions({32, 512}) would mean: Normalize over the last two dimensions (Less common, but possible)

The {n_embd} argument is a vector (initializer list). This is because LayerNorm can normalize over multiple trailing dimensions. For transformers, we only normalize over the feature dimension, so we pass a single-element vector {n_embd}.

LayerNorm creates two learned parameter tensors: gamma (weight) with shape (n_embd,) and beta (bias) with shape (n_embd,). These are per-feature parameters. With n_embd = 512, each LayerNorm adds 512 + 512 = 1024 learnable parameters. This is tiny compared to the linear layers.

Why the braces? LayerNormOptions({n_embd}) needs the curly braces because the constructor expects a std::vector<int64_t>. Writing LayerNormOptions(n_embd) without braces would attempt to pass an int, which does not match the expected signature. The braces create an initializer list that converts to the vector.

Pre-Norm vs Post-Norm

There are two ways to arrange LayerNorm in a transformer block. Our code uses pre-norm:

Pre-norm (what our code does): x = x + MHA(LayerNorm(x)) x = x + FFN(LayerNorm(x)) Post-norm (original "Attention is All You Need" paper): x = LayerNorm(x + MHA(x)) x = LayerNorm(x + FFN(x))

The difference is whether LayerNorm is applied before or after the sub-layer.

PropertyPre-NormPost-Norm
LayerNorm placementBefore attention/FFNAfter residual addition
Training stabilityMore stable, easier to trainLess stable, needs warmup
Learning rate sensitivityLess sensitiveMore sensitive
Final performanceSlightly lower (debated)Slightly higher (debated)
Used byGPT-2, GPT-3, LLaMA, most modern modelsOriginal Transformer, BERT

Why pre-norm is more stable

In pre-norm, the residual path x + sublayer(LN(x)) always has a clean gradient path through the + x shortcut. The gradient can flow directly backward through the addition without passing through LayerNorm. In post-norm, the gradient must pass through LayerNorm at every layer, which can cause gradient scaling issues in very deep models.

Practical advice: Pre-norm is the default choice for most new transformer implementations. It trains reliably without learning rate warmup and works well across different model sizes. Our code follows this convention.

Residual Connections

The x + ... pattern in the forward pass is a residual (skip) connection. Instead of replacing x with the sub-layer output, we add the sub-layer output to x.

Without residual: x = MHA(LN(x)) (x is replaced) With residual: x = x + MHA(LN(x)) (x is preserved and augmented)

Why residuals are essential

Consider a 96-layer transformer (GPT-3 scale). Without residual connections, information must pass through 96 sequential transformations. If any layer slightly corrupts the signal, the corruption compounds. The gradient must also flow backward through all 96 layers, facing vanishing or exploding gradients.

With residual connections, the input has a direct shortcut path to the output. Even if a sub-layer outputs zeros (contributes nothing), the input passes through unchanged. This provides:

  • Identity initialization: At random initialization, each block approximately computes the identity function. The sub-layers produce small random outputs that barely change x. Training gradually learns what to add.
  • Clean gradient flow: The gradient of x + f(x) with respect to x includes a term of 1 (from the identity). This 1 keeps gradients alive regardless of how many layers deep we go.
  • Incremental refinement: Each block adds a small correction to the representation. Block 1 adds a little, block 2 adds a little more, and so on. The representation is incrementally refined rather than repeatedly overwritten.

For a single residual layer: y = x + f(x). The gradient is dy/dx = 1 + df/dx. Even if df/dx is very small (vanishing), the gradient is still at least 1. Over N layers: dy_N/dx = product of (1 + df_i/dx). Each factor is close to 1, so the product stays well-behaved. Without residuals: dy_N/dx = product of df_i/dx. Each factor may be less than 1, and the product can vanish exponentially.

The Forward Pass

torch::Tensor forward(torch::Tensor x) { x = x + mha->forward(ln1(x)); x = x + ff->forward(ln2(x)); return x; }

Two lines. That is the entire forward pass. But there is a lot happening in each line.

Line 1: Attention sub-layer

x = x + mha->forward(ln1(x));

Reading right to left:

  1. ln1(x) normalizes x (shape unchanged: B, T, n_embd)
  2. mha->forward(...) runs multi-head attention on the normalized x (shape unchanged: B, T, n_embd)
  3. x + ... adds the attention output back to the original (un-normalized) x
  4. x = ... reassigns x to the result

Line 2: FFN sub-layer

x = x + ff->forward(ln2(x));

Identical structure: normalize, transform, add residual. But now x is the output from line 1 (it already has attention information baked in).

x is reassigned, not mutated. x = x + ... creates a new tensor and binds the name x to it. The original input tensor is not modified. The autograd graph correctly tracks both the old x (for the residual) and the new x (after the addition).

Line-by-Line Trace

Let us trace concrete values with n_embd = 512, num_heads = 8, head_size = 64, d_ff = 2048, B = 4, T = 32.

StepExpressionShapeWhat Happens
1x(4, 32, 512)Input to block
2ln1(x)(4, 32, 512)Normalize each token's 512 features
3mha->forward(ln1(x))(4, 32, 512)8-head attention, project, concat
4x + mha->forward(ln1(x))(4, 32, 512)Residual addition (element-wise)
5ln2(x)(4, 32, 512)Normalize updated x
6ff->forward(ln2(x))(4, 32, 512)Expand to 2048, ReLU, compress to 512
7x + ff->forward(ln2(x))(4, 32, 512)Residual addition

Every intermediate tensor has the same shape (B, T, n_embd). This is by design. The entire block is shape-preserving, which is what makes stacking possible.

Complete Shape Walkthrough

Zooming into the internal shapes that our sub-layers produce (hidden from the block's perspective):

Block input: (4, 32, 512) === Attention sub-layer === ln1(x): (4, 32, 512) normalize features Inside MHA: Per head Q/K/V: (4, 32, 64) project to head_size Attention scores: (4, 32, 32) token-to-token weights Head output: (4, 32, 64) weighted sum of values Concatenated (8 heads):(4, 32, 512) cat along last dim Projected: (4, 32, 512) W_O projection x + MHA output: (4, 32, 512) residual add === FFN sub-layer === ln2(x): (4, 32, 512) normalize features Inside FFN: After linear_1: (4, 32, 2048) expand 4x After ReLU: (4, 32, 2048) sparsify After linear_2: (4, 32, 512) compress back x + FFN output: (4, 32, 512) residual add Block output: (4, 32, 512)

Parameter Count

Multi-Head Attention (n_embd=512, 8 heads, head_size=64): 8 heads * 3 matrices * (512 * 64) = 786,432 (Q, K, V per head) Projection: 512 * 512 + 512 = 262,656 MHA total: 1,049,088 Feed-Forward Network (d_ff=2048): linear_1: 512 * 2048 + 2048 = 1,050,624 linear_2: 2048 * 512 + 512 = 1,049,088 FFN total: 2,099,712 LayerNorm x 2: Each: 512 (gamma) + 512 (beta) = 1,024 Both: 2,048 Block total: 1,049,088 + 2,099,712 + 2,048 = 3,150,848 For a 12-block model: 12 * 3,150,848 = 37,810,176 parameters (just the blocks, excluding embeddings and output head)
Where the parameters live: The FFN accounts for about 67% of each block's parameters. MHA accounts for about 33%. LayerNorm is negligible (0.065%). This is why efficient transformer research often focuses on FFN optimization (mixture-of-experts, sparse FFN, etc.).

Full Code

#include <torch/torch.h> #include <vector> #include <memory> #include <string> #include <iostream> using namespace std; // Assumes SingleAttentionHead, MultiHeadAttention, and FeedForward // are defined as in the previous blog posts class TransformerBlock : public torch::nn::Module { shared_ptr<MultiHeadAttention> mha; shared_ptr<FeedForward> ff; torch::nn::LayerNorm ln1{nullptr}, ln2{nullptr}; public: TransformerBlock(int n_embd, int num_heads, int head_size, int d_ff) { mha = make_shared<MultiHeadAttention>(n_embd, num_heads, head_size); ff = make_shared<FeedForward>(d_ff, n_embd); ln1 = torch::nn::LayerNorm( torch::nn::LayerNormOptions({n_embd})); ln2 = torch::nn::LayerNorm( torch::nn::LayerNormOptions({n_embd})); register_module("mha", mha); register_module("ff", ff); register_module("ln1", ln1); register_module("ln2", ln2); } torch::Tensor forward(torch::Tensor x) { x = x + mha->forward(ln1(x)); x = x + ff->forward(ln2(x)); return x; } }; int main() { int n_embd = 512; int num_heads = 8; int head_size = n_embd / num_heads; // 64 int d_ff = 4 * n_embd; // 2048 auto block = make_shared<TransformerBlock>( n_embd, num_heads, head_size, d_ff); // Input: batch=4, seq_len=32, n_embd=512 torch::Tensor x = torch::randn({4, 32, n_embd}); torch::Tensor out = block->forward(x); cout << "Input shape: " << x.sizes() << endl; cout << "Output shape: " << out.sizes() << endl; // Input: [4, 32, 512] // Output: [4, 32, 512] // Count total parameters int64_t total = 0; for (auto& p : block->parameters()) { total += p.numel(); } cout << "Total parameters: " << total << endl; // Total parameters: 3,150,848 }

What Comes Next

We now have a complete, stackable transformer block. The remaining pieces to build a full language model:

  • Token and positional embeddings: Convert raw token IDs into the (B, T, n_embd) tensors that feed into the blocks.
  • Stacking blocks: Create a vector of N transformer blocks and run input through them sequentially.
  • Output head: A final linear layer that maps from n_embd to vocabulary size, producing logits for next-token prediction.
  • The full GPT model: Combining embeddings + N blocks + output head into one trainable model.
