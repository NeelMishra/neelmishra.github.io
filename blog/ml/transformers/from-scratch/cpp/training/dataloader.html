DataLoader in C++ (LibTorch)

ML Series · From Scratch in C++ · LibTorch

Before we can train a language model, we need to feed it data. A language model learns to predict the next token: given an input sequence, the target is the same sequence shifted right by one. The DataLoader takes a flat array of token IDs, carves it into fixed-length windows, and batches them into tensors ready for the forward pass.

Series roadmap. Single Head → Multi-Head Attention → Feed-Forward → Transformer Block → Full GPT Model → Cross-Entropy Loss → DataLoader (you are here) → Training Loop.

1 · What the DataLoader Does

Given a corpus encoded as token IDs [4, 17, 8, 92, 3, 55, 21, ...] and a context length (block_size), the DataLoader produces (input, target) pairs where each target is the input shifted right by one position. Multiple pairs are stacked into a batch of shape (B, T).

Each input window of length block_size is paired with a target window that starts one position later. A batch stacks batch_size such pairs into tensors of shape (B, T).

block_size is the sequence length. Throughout this series you’ll see block_size, seq_len, T, context_length, and n_ctx used interchangeably — they all name the same thing: the fixed context window the model attends over. Here the DataLoader carves every batch into windows of exactly block_size tokens, so x.shape = (batch_size, block_size) = (B, T) — the time dimension is block_size by construction. The only time they can differ is at inference, when you feed a prompt shorter than block_size; then T ≤ block_size.

Scaling Beyond One Text Stream

The implementation below intentionally uses one flat token stream because it makes the mechanics easy to see. Real datasets usually start as millions of rows — documents, web pages, chat turns, code files, support tickets, or product descriptions — and the production input pipeline turns those rows into the same kind of token windows.

  • Tokenize offline, not inside the training loop. Raw rows are cleaned, normalized, filtered, deduplicated, and tokenized in a preprocessing job. The training job usually reads token IDs from binary shards, Arrow/Parquet files, or memory-mapped arrays instead of repeatedly parsing text.
  • Add document boundaries. When rows are packed into one long stream, insert an <eos> or document-separator token between records. That tells the model where one example ends and prevents accidental blending of unrelated rows unless you explicitly want cross-document packing.
  • Shard the corpus. A single tensor in RAM works for a toy corpus, but large datasets are split into many token shards. Each worker reads a subset of shards, samples windows from local offsets, and periodically reshuffles shard order.
  • Sample spans, not whole rows. Long documents are sliced into many block_size spans. Short rows can be packed together with separators so GPU batches stay dense instead of wasting padding.
  • Keep I/O ahead of the GPU. Production loaders prefetch batches on CPU threads, use pinned host memory for faster host-to-device transfer, and avoid per-token Python/C++ work on the critical path.
  • Partition work for distributed training. With multiple GPUs or nodes, each rank gets different shards or offset ranges so workers do not all train on the same rows. A global seed plus rank-specific shard assignment keeps sampling reproducible.
// Real-world preprocessing shape raw rows/files -> clean + filter + deduplicate -> tokenize each row -> append <eos> between rows -> write token shards: shard_000.bin, shard_001.bin, ... // Training-time loader shape pick shard -> pick random token offset -> read [offset : offset + block_size + 1] input = tokens[0 : block_size] target = tokens[1 : block_size + 1]
Why the toy loader is still useful: after preprocessing, a massive row-based dataset and a tiny demo string both reduce to the same abstraction: sample contiguous token windows and train the model to predict the next token. The difference is engineering scale — sharding, streaming, prefetching, distributed partitioning, and careful handling of document boundaries.

2 · The DataLoader Class

Everything the loader needs is private: the token tensor and the two window sizes. The only public surface is the constructor and next_batch(). No state leaks, no cursor to manage, no copy of the token array floating around.

class DataLoader { torch::Tensor data; // all token IDs as a 1-D int64 tensor int block_size; // sequence length (context window) int batch_size; // sequences per batch public: DataLoader(const std::vector<int>& tokens, int block_size, int batch_size) : block_size(block_size), batch_size(batch_size) { // Embedding layers and cross-entropy targets expect int64 indices, // so we store the token stream as kLong once, up front. data = torch::tensor(tokens, torch::dtype(torch::kLong)); } std::pair<torch::Tensor, torch::Tensor> next_batch() { int n = data.size(0); // Random starts in [0, n - block_size). Each start s gives // x = data[s : s + block_size ] (length block_size) // y = data[s + 1 : s + block_size + 1] (same, shifted by 1) auto ix = torch::randint(n - block_size, {batch_size}, torch::dtype(torch::kLong)); std::vector<torch::Tensor> xs, ys; xs.reserve(batch_size); ys.reserve(batch_size); for (int i = 0; i < batch_size; i++) { int start = ix[i].item<int>(); xs.push_back(data.slice(0, start, start + block_size)); ys.push_back(data.slice(0, start + 1, start + block_size + 1)); } // stack along a new leading dim -> (B, T) for both inputs and targets return { torch::stack(xs), torch::stack(ys) }; } };

A few details worth calling out:

  • private by default. With class the members above the public: label are hidden. Callers can only touch the two-method API, which is exactly what we want.
  • const std::vector<int>& in the constructor — we don’t need to modify the caller’s vector, and we don’t copy it either; torch::tensor(...) reads it once and builds an owned tensor.
  • Dtype kLong (int64). Token IDs feed nn::Embedding as indices and nn::CrossEntropyLoss as class labels, both of which require int64. Picking the dtype once, here, means no casts later in the training loop.
  • Inline definition. Keeping next_batch() inside the class body keeps the header self-contained — no separate .cpp, no forward declarations, easier to drop into a single-file demo.

3 · How next_batch() Builds a Batch

Rather than stepping through the corpus with a cursor, we sample batch_size random start positions every call. Each start carves out a contiguous window for the input and a window shifted right by one for the target. Stacking the lists along a new axis gives the (B, T) tensors the model expects.

VariableShapeDescription
ix(B,)Random start positions, int64, each in [0, n - block_size)
xs[i](T,)data[start : start + block_size]
ys[i](T,)data[start + 1 : start + block_size + 1]
return.first(B, T)Stacked input sequences
return.second(B, T)Stacked target sequences
Why random sampling? Sequential reading means the model sees the same windows in the same order every epoch. Random sampling gives a fresh set of overlapping windows on every call — it’s both free data augmentation and a substitute for explicit shuffling and epoch bookkeeping.
Why n - block_size and not n - block_size - 1? torch::randint(high, ...) draws from [0, high), so the largest start it can return is n - block_size - 1. For that worst case, the target slice ends at index n, which is a valid exclusive upper bound — no out-of-range access.

Line-by-line walkthrough

A few lines in next_batch() rely on LibTorch-specific behaviour worth spelling out.

torch::randint(n - block_size, {batch_size}, torch::dtype(torch::kLong))

  • n - block_size is the exclusive upper bound. The signature is randint(high, size, options) with starts drawn from [0, high).
  • {batch_size} is an initializer list that converts to an IntArrayRef — the shape of the output tensor. Here it’s 1-D with length batch_size.
  • torch::dtype(torch::kLong) is a TensorOptions setting int64 storage, which is what indexing operations expect.

xs.reserve(batch_size)

We know the final size up front, so reserving avoids reallocations during push_back.

int start = ix[i].item<int>();

ix[i] returns a zero-dimensional torch::Tensor, not a plain integer. .item<int>() extracts the scalar and casts it to the requested C++ type.

If data lives on CUDA, each .item<int>() call forces a device→host synchronization. Doing that B times per batch can dominate runtime, so in a production loader you’d keep ix on CPU or move it there once before the loop.

data.slice(0, start, start + block_size)

slice(dim, start, end) mirrors Python’s tensor[start:end] along dim. end is exclusive, and the result is a view over the same storage — no token data is copied until the final stack. The target line shifts the window by one (start + 1 : start + block_size + 1) for next-token prediction.

return { torch::stack(xs), torch::stack(ys) };

torch::stack glues batch_size tensors of shape (T,) into a single (B, T) tensor along a new leading axis. This is where the views finally get copied into contiguous 2-D buffers. torch::cat would instead concatenate along an existing axis and yield a 1-D tensor of length B · T. The braces build the std::pair via list initialization.

4 · Index-Respecting Variant

A fair objection to the loader above: it does not respect index order. torch::randint draws batch_size independent starts every call, so a window can be picked twice while another is never picked at all. There is no notion of an epoch — the model just keeps drawing from a uniform distribution over [0, n - block_size) forever.

For pretraining on billions of tokens that’s usually fine: the law of large numbers takes over, the variance disappears, and skipping the bookkeeping buys real throughput. But on smaller corpora — fine-tuning, evaluation, ablations — you typically do want every window seen exactly once before any repeats. That is the standard PyTorch DataLoader contract: shuffle indices once per epoch, then iterate.

The variant below keeps the same std::pair<Tensor, Tensor> next_batch() signature so it is a drop-in replacement, but it now owns a permutation of valid start offsets and a cursor.

class EpochDataLoader { torch::Tensor data; torch::Tensor perm; // shuffled valid starts, shape (n - block_size,), int64 int64_t cursor = 0; // next index into perm int64_t epoch = 0; // completed epoch counter (handy for logging) int block_size; int batch_size; void reshuffle() { auto n = data.size(0); perm = torch::randperm(n - block_size, torch::dtype(torch::kLong)); cursor = 0; } public: EpochDataLoader(const std::vector<int>& tokens, int block_size, int batch_size) : block_size(block_size), batch_size(batch_size) { data = torch::tensor(tokens, torch::dtype(torch::kLong)); reshuffle(); } std::pair<torch::Tensor, torch::Tensor> next_batch() { // Drop-last: if we cannot fill a full batch from the remaining permutation, // finish the epoch and start a freshly shuffled one. This keeps the (B, T) // shape constant for the model and matches PyTorch's drop_last=True default. if (cursor + batch_size > perm.size(0)) { ++epoch; reshuffle(); } auto ix = perm.slice(0, cursor, cursor + batch_size); cursor += batch_size; std::vector<torch::Tensor> xs, ys; xs.reserve(batch_size); ys.reserve(batch_size); for (int i = 0; i < batch_size; i++) { int64_t start = ix[i].item<int64_t>(); xs.push_back(data.slice(0, start, start + block_size)); ys.push_back(data.slice(0, start + 1, start + block_size + 1)); } return { torch::stack(xs), torch::stack(ys) }; } int64_t current_epoch() const { return epoch; } };

The interesting line is torch::randperm(n - block_size, kLong). It returns a 1-D tensor containing every integer in [0, n - block_size) exactly once, in random order — the same primitive PyTorch’s RandomSampler builds on. Slicing batch_size entries off the front and advancing the cursor is what makes "every window once per epoch" hold by construction.

Trade-offs at a glance

AspectRandom sampling (section 3)Index-respecting (this section)
State per loaderO(1)O(n - block_size) for the permutation
Coverage per epochProbabilistic; some windows missed, others repeatedEvery valid start visited exactly once
Epoch signalNone — you train by step countCursor wrap = epoch boundary, easy to log
ReproducibilitySeed torch::Generator globallySame, plus the permutation is materialized so you can checkpoint perm + cursor
Implicit augmentationYes — new overlapping windows every callOnly between epochs (after reshuffle)
Tail handlingN/ADrop-last by default; alternatives: pad to fill, or wrap into next epoch
Rule of thumb. Use random sampling when the corpus is so large that a full pass is impractical and per-step variance is dominated by data scale — the canonical LM pretraining setting. Reach for the index-respecting variant when you actually care about epoch semantics: small or curated datasets, fine-tuning, evaluation, ablations, and any time you want a deterministic "did the model see every example?" answer.
Tail policy matters. Drop-last (shown above) is the safest default because it keeps (B, T) static, which avoids recompiling kernels and keeps tensor shapes friendly to torch.compile-style graph capture. If you need every token, prefer wrapping the leftover starts into the start of the next epoch’s permutation rather than emitting a runt batch.

5 · Usage Example

int main() { // token IDs from your BPE tokenizer std::vector<int> tokens = {4, 17, 8, 92, 3, 55, 21, 10, 44, 6, 73, 15, 88, 2, 61, 37, 49, 5, 80, 12}; int block_size = 4; int batch_size = 3; DataLoader loader(tokens, block_size, batch_size); auto [x, y] = loader.next_batch(); std::cout << "x shape: " << x.sizes() << std::endl; // [3, 4] std::cout << "y shape: " << y.sizes() << std::endl; // [3, 4] std::cout << "x:\n" << x << std::endl; std::cout << "y:\n" << y << std::endl; }

Each call to next_batch() returns a fresh random batch. Feed x into the model’s forward pass, compute cross-entropy loss against y, and backpropagate.
