BPE Tokenizer in C++

ML Series ML · Transformers · From Scratch · C++ · Training

What Is BPE

Byte-Pair Encoding (BPE) starts with individual characters (or bytes) as the base vocabulary, then repeatedly glues together the most frequent adjacent pair and gives that new glued-pair its own token ID. After enough merges the vocabulary contains subword units that land between character-level and word-level granularity—small enough to represent any text (no out-of-vocabulary tokens), large enough that common words become a single token. For the full theory, see the tokenization theory blog.

Tiny worked example. Train on the corpus "low low lower":

  1. Start with characters: l o w · l o w · l o w e r (spaces shown as ·).
  2. Count adjacent pairs. (l,o) appears 3 times—the winner.
  3. Merge it everywhere: lo w · lo w · lo w e r, and add "lo" to the vocabulary.
  4. Recount. (lo,w) now appears 3 times. Merge → "low" is born.
  5. Stop when we hit the target vocab size. The learned merge order is the tokenizer’s recipe.

Encoding later text means replaying the exact same recipe: split into characters, then apply each merge rule in the order it was learned. Decoding is even simpler: map IDs back to strings and concatenate. That’s the whole algorithm—the rest of this post is the C++ for it.

Data Structures

The whole tokenizer is just three containers: an ordered list of merge rules, and two maps that translate between strings and integer IDs. That’s it—no trees, no tries, no fancy state:

struct BPETokenizer { std::vector<std::pair<std::string, std::string>> merges; // ordered merge rules std::unordered_map<std::string, int> vocab; // token → ID std::unordered_map<int, std::string> inverse_vocab; // ID → token int vocab_size; // target size void train(const std::string& text, int target_vocab_size); std::vector<int> encode(const std::string& text); std::string decode(const std::vector<int>& ids); };
FieldPurpose
mergesPairs learned during training, in the order they were learned. Encoding replays them in this exact order.
vocab / inverse_vocabBidirectional string ↔ ID mapping. One is for encode, the other for decode.
vocab_sizeTarget size; training stops when we reach it.
Why a vector of merges and not a set? Order matters. The merge learned first is more “primitive” than later ones and must be applied first during encoding—otherwise we’d produce tokens that were never in training. A vector preserves the learning order for free; a set would lose it.

Training

Training is a greedy loop: count every adjacent pair, merge the most frequent one, repeat. The merge rules we record along the way are the tokenizer.

Before the code, a quick mental model of the four phases inside train():

  1. UTF-8 split. Turn the raw string into a list of one-character tokens. C++ std::string is just bytes, so we must respect UTF-8 boundaries (details below).
  2. Seed the vocabulary. Every unique character gets its own ID. These are the base tokens.
  3. Count pairs. Walk the sequence once, tallying every adjacent pair into a map.
  4. Pick the winner & merge. Rewrite the sequence replacing every occurrence of the top pair with its concatenation, add the new token to the vocab, record the rule, and loop until the target vocab size is reached.
void BPETokenizer::train(const std::string& text, int target_vocab_size) { // 1. Split text into individual UTF-8 characters as initial tokens std::vector<std::string> tokens; for (size_t i = 0; i < text.size(); ) { int len = 1; unsigned char c = text[i]; if (c >= 0xF0) len = 4; else if (c >= 0xE0) len = 3; else if (c >= 0xC0) len = 2; tokens.push_back(text.substr(i, len)); i += len; } // 2. Build base vocabulary from unique characters int next_id = 0; for (auto& t : tokens) { if (vocab.find(t) == vocab.end()) { vocab[t] = next_id; inverse_vocab[next_id] = t; next_id++; } } // 3. Iteratively merge the most frequent pair while (next_id < target_vocab_size) { // Count adjacent pairs std::map<std::pair<std::string,std::string>, int> pair_counts; for (size_t i = 0; i + 1 < tokens.size(); i++) pair_counts[{tokens[i], tokens[i+1]}]++; if (pair_counts.empty()) break; // Find the pair with the highest count auto best = std::max_element(pair_counts.begin(), pair_counts.end(), [](auto& a, auto& b) { return a.second < b.second; }); std::string merged = best->first.first + best->first.second; // Record the merge rule merges.push_back(best->first); vocab[merged] = next_id; inverse_vocab[next_id] = merged; next_id++; // Apply the merge to the token sequence std::vector<std::string> new_tokens; for (size_t i = 0; i < tokens.size(); i++) { if (i + 1 < tokens.size() && tokens[i] == best->first.first && tokens[i+1] == best->first.second) { new_tokens.push_back(merged); i++; // skip next token (consumed by merge) } else { new_tokens.push_back(tokens[i]); } } tokens = std::move(new_tokens); } vocab_size = next_id; }
Key point: The merge rules are order-dependent. During encoding we must replay them in the same order they were learned.

Reading the tricky bits

1. The UTF-8 split — the long version

The problem. In C++, std::string is just a sequence of char (which is 1 byte). But most of the world’s characters don’t fit in one byte. “A” fits. “é”, “中”, “😀” do not. UTF-8 solves this with a variable-length encoding: each character is stored as 1, 2, 3, or 4 bytes. If we naively wrote text[i] to iterate “characters”, we would cut a multi-byte character in half and produce garbage. So the first job of the tokenizer is: given a byte index i, how many bytes does the character starting there occupy?

How UTF-8 encodes length in the first byte. The designers used a brilliant trick: the top bits of the first byte tell you the length. Continuation bytes (the 2nd/3rd/4th byte of a multi-byte character) always start with the binary prefix 10, so they can never be mistaken for a start byte.

Notice the pattern: count the leading 1’s in the first byte and that is the total byte-length of the character. 0 leading ones = 1 byte, 110 = 2, 1110 = 3, 11110 = 4.

Why the thresholds 0xF0, 0xE0, 0xC0. Those hex values are exactly the binary prefixes above, padded with zeros:

0xF0 = 1111 0000 ← any byte ≥ this starts with "11110..." → 4-byte character 0xE0 = 1110 0000 ← any byte ≥ this starts with "1110..." → 3-byte character 0xC0 = 1100 0000 ← any byte ≥ this starts with "110..." → 2-byte character below 0xC0 → either plain ASCII (0–0x7F) or a continuation byte (0x80–0xBF)

The checks run from largest threshold first because a 4-byte start byte (say 0xF0) is also ≥ 0xE0 and ≥ 0xC0. If we checked 0xC0 first we would wrongly classify it as a 2-byte character. Descending order picks the most specific category that applies.

Walk through a concrete example. Take the string "Aé😀" (A, e-acute, grin emoji). The raw bytes are:

index: 0 1 2 3 4 5 6 byte: 0x41 0xC3 0xA9 0xF0 0x9F 0x98 0x80 └── é ──┘ └─────── 😀 ────────┘

Step-by-step loop execution:

i = 0: text[0] = 0x41 (= 65). Not ≥ 0xC0, so len = 1. push_back("A"); i = 1. i = 1: text[1] = 0xC3 (= 195). ≥ 0xC0, not ≥ 0xE0, so len = 2. push_back("é"); // = bytes [0xC3, 0xA9] i = 3. i = 3: text[3] = 0xF0 (= 240). ≥ 0xF0, so len = 4. push_back("😀"); // = bytes [0xF0, 0x9F, 0x98, 0x80] i = 7. Loop ends. tokens == [ "A", "é", "😀" ] // each element is a complete UTF-8 character

Why the unsigned char cast matters. On most compilers, plain char is signed, so a byte like 0xF0 would come out as the negative number -16. Comparing that to 0xF0 (which the compiler treats as 240) would give a wrong answer. Casting through unsigned char forces the byte to be read as 0–255, which is the only interpretation that makes sense here.

Finally, text.substr(i, len) copies exactly those bytes into a new std::string—a small, self-contained UTF-8 character—and we push it into tokens. After the loop, tokens contains one entry per visual character, not per byte, which is exactly what BPE needs as its starting alphabet.

2. The max_element lambda. std::max_element returns the largest element according to a “less-than” comparator. Our lambda a.second < b.second says “compare by pair-frequency”, so max_element returns the pair with the highest count. The ->first on the result is the key ((left,right) pair of strings) and ->second is the count.

3. The i++ after appending a merged token. When we find the pair (tokens[i], tokens[i+1]), we push one combined token merged and then i++ on top of the loop’s own i++—effectively skipping tokens[i+1] because it’s already been consumed. Without this, a sequence like l o l o merged on (l,o) would produce lo, then immediately try to merge (o, l) starting from index 1, which is a token that no longer exists in the rewritten sequence. The double increment keeps our reading position aligned with the rewritten output.

4. Why std::move(new_tokens)? new_tokens is a fresh vector we just built; after the assignment we don’t need it anymore. std::move hands its internal buffer to tokens in O(1) instead of copying every string element—a meaningful speed-up inside the merge loop.

Encoding

Encoding is where the “tokenizer is just a recipe” idea pays off. We take new text the model has never seen, split it into characters the same way we did during training, then apply every learned merge in the order it was learned. After all merges have been replayed, every surviving token is guaranteed to be in vocab—so we just look up its ID.

Why replay in training order? Imagine merge #5 was (lo, w) → low. That token “low” can only exist if (l, o) → lo (merge #1) was applied first. Running merges out of order would either skip valid tokens or produce a sequence that was never in the vocab—an instant crash at the vocab.at(t) lookup.

std::vector<int> BPETokenizer::encode(const std::string& text) { // Split into UTF-8 characters std::vector<std::string> tokens; for (size_t i = 0; i < text.size(); ) { int len = 1; unsigned char c = text[i]; if (c >= 0xF0) len = 4; else if (c >= 0xE0) len = 3; else if (c >= 0xC0) len = 2; tokens.push_back(text.substr(i, len)); i += len; } // Apply each merge rule in order for (auto& [left, right] : merges) { std::vector<std::string> new_tokens; for (size_t i = 0; i < tokens.size(); i++) { if (i + 1 < tokens.size() && tokens[i] == left && tokens[i+1] == right) { new_tokens.push_back(left + right); i++; } else { new_tokens.push_back(tokens[i]); } } tokens = std::move(new_tokens); } // Map tokens to IDs std::vector<int> ids; for (auto& t : tokens) ids.push_back(vocab.at(t)); return ids; }
Complexity note: This naïve approach is O(merges × sequence_length) per encode call—every learned rule does one full pass over the token sequence. Production tokenizers (e.g. tiktoken, SentencePiece) use priority queues keyed on merge rank to get O(n log n), but the greedy replay above is clearer and fine for learning.

A subtle point: the structured binding auto& [left, right] : merges unpacks each std::pair into the two halves of the merge rule. It’s the same data as merges[k].first / merges[k].second, just easier to read.

Decoding

Decoding is delightfully boring: walk the ID list, look each one up in inverse_vocab, concatenate. Because every token string already contains all of its characters (the merge "low" literally stores the three bytes l, o, w), no reverse-merge logic is needed—appending the strings reassembles the original text byte-for-byte.

std::string BPETokenizer::decode(const std::vector<int>& ids) { std::string result; for (int id : ids) result += inverse_vocab.at(id); return result; }

Scaling to Large Datasets

The code above works on one in-memory string. Real training corpora are very different—hundreds of gigabytes, possibly a trillion rows of (x, y) text pairs, stored across many shards on disk. You cannot std::string text = readAll(corpus); it will not fit. And even if it did, the inner merge loop rewrites the entire token sequence on every iteration, which for a trillion tokens and tens of thousands of merges is catastrophic.

The production trick is to never operate on the full sequence at all. Instead, operate on word frequencies:

Idea 1 — Word-frequency dictionary (the real BPE)

Text has huge redundancy: the word “the” may appear a billion times. Representing it as a billion separate character sequences is wasteful—it’s the same work, billion times over. The original BPE paper instead pre-tokenizes on whitespace/punctuation, stores each unique word exactly once, and keeps a count. Pair counts are then computed as a weighted sum over the dictionary.

// Compact representation: word → (character sequence, frequency) std::unordered_map<std::string, uint64_t> word_freq; void count_words_streaming(const std::string& path) { std::ifstream in(path); std::string line; while (std::getline(in, line)) { // one row at a time — O(1) memory per row for (auto& w : split_on_whitespace(line)) word_freq[w]++; } }

A trillion rows collapse into maybe a few million unique words. That fits comfortably in RAM. Now every operation that used to sweep a trillion tokens instead sweeps a few million dictionary entries, weighted by their frequency.

// Each word is stored as its current token split (starts as characters). std::unordered_map<std::string, std::vector<std::string>> word_tokens; for (auto& [w, f] : word_freq) word_tokens[w] = utf8_split(w); // Pair counts, weighted by word frequency. std::unordered_map<Pair, uint64_t, PairHash> pair_counts; for (auto& [w, f] : word_freq) { auto& toks = word_tokens[w]; for (size_t i = 0; i + 1 < toks.size(); ++i) pair_counts[{toks[i], toks[i+1]}] += f; // ← frequency-weighted }

The rewrite step only touches the words that contain the winning pair. You don’t need to scan the whole dictionary after every merge; keep an index pair → set<words containing it> and update incrementally. This alone turns each merge step from O(corpus) into O(affected words), usually < 1% of the dictionary.

Idea 2 — Priority queue instead of max_element

Our naive code called std::max_element over all pair counts on every iteration: O(P) per merge where P is the number of distinct pairs. With a max-heap keyed on count, picking the winner becomes O(log P), and each count update is O(log P) as well. For a 50k-merge vocabulary this is the difference between hours and minutes.

std::priority_queue<PairCount> heap; // top() = most frequent pair for (auto& [p, c] : pair_counts) heap.push({c, p}); while (next_id < target_vocab_size && !heap.empty()) { auto top = heap.top(); heap.pop(); // Lazy invalidation: the heap may hold stale counts from earlier updates. if (top.count != pair_counts[top.pair]) continue; // stale entry, discard apply_merge(top.pair); update_neighbors_and_heap(top.pair); }

The “lazy invalidation” pattern is the standard way to avoid expensive heap update-key operations: you re-push an entry with the new count and simply skip stale entries when they surface.

Idea 3 — Sharded counting (map-reduce)

If the corpus is truly massive (hundreds of GB of (x, y) rows), even the streaming word-count does not fit on one machine. Split the data into N shards, count word_freq per shard in parallel, then reduce:

// Map stage — one thread/process per shard, independent for (int s = 0; s < N; ++s) shard_counts[s] = count_words_streaming(shard_path(s)); // Reduce stage — merge dictionaries std::unordered_map<std::string, uint64_t> global; for (auto& sc : shard_counts) for (auto& [w, f] : sc) global[w] += f;

Once you have the global word_freq, the BPE training loop itself runs on a single machine (it’s operating on the compressed dictionary, which is small). Only the counting phase needs the map-reduce.

Idea 4 — You don’t need the whole corpus

A perhaps surprising point: BPE statistics saturate quickly. The merge ordering learned from a uniform 1–10 GB sample of a trillion-row dataset is essentially identical to what you’d learn from the full corpus—the top-frequency pairs are already stable. GPT-2’s tokenizer was trained on roughly 40 GB; it tokenizes trillion-token corpora just fine. For production, sample a representative slice, train on that, then apply the fixed tokenizer to the entire dataset in parallel.

Idea 5 — Handling (x, y) pairs

For supervised data with inputs x and targets y (translation, summarization, instruction tuning), the choice depends on goal:

  • Shared vocabulary (most common): concatenate the two columns during counting so both sides share the same tokenizer. Necessary when you want the model to reuse embeddings across input and output (e.g., GPT-style decoder-only, where y is appended to x).
  • Separate vocabularies: count x and y independently, train two tokenizers. Used by some translation systems with very different source/target languages.
  • Language-balanced sampling: if one column is mostly English and the other mostly code, uniform sampling will under-represent code. Weight the sampling so each language/domain contributes a target share of the counted tokens—this is what modern multilingual tokenizers do.

Putting the pieces together

A realistic training pipeline for a trillion-row dataset looks like:

  1. Stream each shard line-by-line; build a per-shard word_freq map. Spill to disk if a shard is too large.
  2. Reduce the per-shard maps into a global word_freq (or sample a ~10 GB representative slice and skip the reduce).
  3. Build word_tokens (UTF-8 split of each unique word) and the weighted pair_counts.
  4. Push pair counts into a heap. Loop: pop winner, apply merge to affected words only, update neighbor pair counts in the index, push updated entries back.
  5. Serialize merges and vocab to disk. Apply the tokenizer to the full dataset in parallel (encoding is embarrassingly parallel—each row is independent).
Takeaway: The algorithm in the rest of this post is conceptually correct—every production tokenizer does the same greedy-pair-merge thing. The difference is purely in the bookkeeping: word-frequency dictionaries, heap-based selection, incremental pair-count updates, and (optionally) sampled training data.

Putting It Together

A minimal driver that trains on a small corpus, encodes a string, and decodes back:

#include <iostream> #include <vector> #include <string> #include <map> #include <unordered_map> #include <algorithm> // ... BPETokenizer struct defined above ... int main() { BPETokenizer tok; std::string corpus = "low lower newest widest low lower"; tok.train(corpus, 30); std::cout << "Vocab size: " << tok.vocab_size << "\n"; std::cout << "Merges learned: " << tok.merges.size() << "\n\n"; // Print merge rules for (auto& [a, b] : tok.merges) std::cout << " (" << a << ", " << b << ") → " << a + b << "\n"; // Encode std::string input = "lower"; auto ids = tok.encode(input); std::cout << "\nEncode \"" << input << "\": ["; for (size_t i = 0; i < ids.size(); i++) std::cout << ids[i] << (i + 1 < ids.size() ? ", " : ""); std::cout << "]\n"; // Decode std::string decoded = tok.decode(ids); std::cout << "Decode back: \"" << decoded << "\"\n"; return 0; }

Expected output (exact merges depend on tie-breaking):

Vocab size: 30 Merges learned: 19 (l, o) → lo (lo, w) → low (e, r) → er (low, er) → lower ... Encode "lower": [17] Decode back: "lower"
That's it. Train, encode, decode—three functions, one data structure, zero dependencies. For the theory behind why BPE works well for language models, see the tokenization theory post.
