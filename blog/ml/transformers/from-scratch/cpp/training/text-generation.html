Text Generation in C++ (LibTorch)

ML Series ML · Transformers · From Scratch · C++ · Training

After training, we use the model to generate text one token at a time. Feed in a prompt, take the logits at the last position, sample a next token, append it, and repeat. This page implements the full generation pipeline: greedy decoding, temperature scaling, and top-k sampling.

1 · The Autoregressive Loop

Language models are autoregressive — they predict one token at a time, then feed their own output back as input. Given a prompt, the model produces logits over the entire vocabulary for the next position. We pick a token from those logits, append it to the context, and repeat until we hit a stop condition or a maximum length.

2 · Greedy Decoding

The simplest strategy: always pick the token with the highest logit (argmax). Deterministic, but tends to produce repetitive text.

torch::Tensor generate_greedy(GPT& model, torch::Tensor context, int max_new_tokens, int block_size) { for (int i = 0; i < max_new_tokens; i++) { // crop context to block_size if it exceeds the window auto ctx = context.size(1) > block_size ? context.index({Slice(), Slice(-block_size, None)}) : context; // forward pass → logits shape: (B, T, vocab_size) auto logits = model.forward(ctx); // take logits at last position → (B, vocab_size) logits = logits.index({Slice(), -1, Slice()}); // greedy: pick the token with highest logit auto next = logits.argmax(-1, /*keepdim=*/true); // append to running context context = torch::cat({context, next}, /*dim=*/1); } return context; }

Reading the crop: Slice, index(), and why

The first three lines inside the loop are the part most readers stumble on, so they deserve their own walkthrough.

auto ctx = context.size(1) > block_size ? context.index({Slice(), Slice(-block_size, None)}) : context;

context is a 2-D tensor of token IDs with shape (B, T):

  • B — batch size (number of sequences generated in parallel; usually 1 at inference).
  • T — current sequence length. It grows by one at the end of every loop iteration, because we just appended the predicted token.

So context.size(1) is just the current T, which we compare against block_size — the fixed context window the model was trained with.

Why crop at all?

The transformer’s positional embedding table is a learned (block_size, n_embd) matrix. Internally the forward pass computes x = tok_emb(idx) + pos_emb(arange(T)). If we pass in a sequence longer than block_size, arange(T) tries to index positions the model never learned — runtime error or garbage output. So before each forward pass we keep only the most recent block_size tokens. Older tokens fall out of the window. This is exactly the receptive field the model was trained on.

What Slice is

In Python you would write context[:, -block_size:]. C++ has no [:] syntax, so LibTorch exposes the same machinery through two pieces:

  • torch::indexing::Slice(start, stop, step) — a value object that represents a Python slice. Slice() with no arguments is equivalent to Python : (“everything along this axis”). Slice(-block_size, None) is equivalent to -block_size: (“from the block_size-th-to-last element through the end”). None here is torch::indexing::None, the C++ stand-in for Python None meaning “no bound”.
  • tensor.index({...}) — multi-dim advanced indexing. The braced list is one torch::indexing::TensorIndex per dimension; each entry can be a Slice, an integer, an ellipsis, a boolean mask, or another tensor.

So the call

context.index({Slice(), Slice(-block_size, None)})

reads dimension by dimension as:

  • dim 0 (batch): Slice() → keep all B rows.
  • dim 1 (time): Slice(-block_size, None) → keep only the last block_size columns.

which is exactly Python’s context[:, -block_size:].

Why the ternary

Early in generation T is small, so cropping is a no-op — we just hand context through unchanged. Once T exceeds the window we switch to the sliced view. Note that index() returns a view into the same storage, not a copy: it’s O(1) metadata (a stride and an offset). The forward pass then reads through that view without allocating new memory.

Mental model. The crop turns the autoregressive loop into a sliding window of width block_size. The model always sees the most recent block_size tokens and silently forgets older ones — a cheap form of “context truncation” with no copy, just a fresh view on the same buffer.
Greedy is deterministic. Given the same prompt, you always get the same output. This is useful for debugging but produces flat, repetitive text.

3 · Temperature Scaling

Divide logits by a temperature parameter before softmax. This controls how “sharp” the probability distribution is.

p(token_i) = softmax( logits / temperature ) temperature > 1.0 → flatter distribution (more random) temperature < 1.0 → peakier distribution (more confident) temperature = 1.0 → unchanged

In code, we divide the logits, apply softmax, and sample from the resulting distribution using torch::multinomial:

// temperature-scaled sampling logits = logits / temperature; auto probs = torch::softmax(logits, /*dim=*/-1); auto next = torch::multinomial(probs, /*num_samples=*/1);

4 · Top-k Sampling

Before sampling, zero out all logits except the top-k highest. This prevents the model from sampling unlikely tokens from the long tail of the distribution.

// keep only the top-k logits, set the rest to -inf auto [topk_vals, topk_idx] = torch::topk(logits, k); // threshold: smallest value that made it into top-k auto threshold = topk_vals.index({Slice(), -1}); // everything below threshold becomes -inf logits = torch::where( logits < threshold, torch::full_like(logits, -std::numeric_limits<float>::infinity()), logits );

After filtering, softmax redistributes all probability mass among the top-k candidates. Then sample normally with torch::multinomial.

5 · The Complete Generate Function

A single function combining temperature scaling and top-k filtering. When top_k = 0, we sample from the full distribution.

torch::Tensor generate(GPT& model, torch::Tensor context, int max_new_tokens, int block_size, float temperature = 1.0f, int top_k = 0) { model.eval(); torch::NoGradGuard no_grad; for (int i = 0; i < max_new_tokens; i++) { // crop to block_size auto ctx = context.size(1) > block_size ? context.index({Slice(), Slice(-block_size, None)}) : context; // forward → logits at last position auto logits = model.forward(ctx); logits = logits.index({Slice(), -1, Slice()}); // temperature scaling logits = logits / temperature; // optional top-k filtering if (top_k > 0) { auto [topk_vals, _] = torch::topk(logits, top_k); auto threshold = topk_vals.index({Slice(), -1}); logits = torch::where( logits < threshold, torch::full_like(logits, -std::numeric_limits<float>::infinity()), logits ); } // sample auto probs = torch::softmax(logits, -1); auto next = torch::multinomial(probs, 1); context = torch::cat({context, next}, 1); } return context; }
ParameterEffect
temperature = 0.8More focused, less random
temperature = 1.2More creative, more random
top_k = 50Sample only from the 50 most likely tokens
top_k = 0Sample from the full vocabulary
NoGradGuard. We wrap generation in torch::NoGradGuard to disable gradient tracking. This saves memory and speeds up inference since we never call backward during generation.

6 · Usage

Encode a prompt, generate tokens, decode back to text:

// encode prompt to token IDs std::vector<int> prompt_ids = tokenizer.encode("The meaning of life is"); // convert to tensor: shape (1, T) auto context = torch::tensor(prompt_ids, torch::kLong).unsqueeze(0).to(device); // generate 200 new tokens with temperature=0.8 and top_k=50 auto output = generate(model, context, 200, block_size, 0.8f, 50); // flatten and decode back to string std::vector<int> out_ids(output.data_ptr<int64_t>(), output.data_ptr<int64_t>() + output.numel()); std::string text = tokenizer.decode(out_ids); std::cout << text << std::endl;

That’s the complete text generation pipeline. The same loop powers every GPT-style model — the only differences in production systems are more sophisticated sampling strategies (top-p / nucleus, repetition penalties, beam search) layered on top of this core.
