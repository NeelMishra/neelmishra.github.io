Cross-Entropy Loss in C++ (LibTorch)

ML Series ML · Transformers · From Scratch · C++ · Training

In the previous post, we built the complete GPT model from scratch: token embedding, positional encoding, stacked transformer blocks, final LayerNorm, and a language model head. We ended with a model that takes token IDs of shape (B, T) and returns logits of shape (B, T, vocab_size). But logits are just raw, unnormalized scores. They do not tell the model whether it was right or wrong. For that, we need a loss function.

Series roadmap. Single Head → Multi-Head Attention → Feed-Forward → Transformer Block → Positional Encoding → Full GPT Model → Cross-Entropy Loss (you are here) → Training Loop.

The Bridge to Learning

Right now our GPT model is a function: you feed it token IDs and it spits out a tensor of numbers. But those numbers are random — the model was just initialized with random weights. For every position in the sequence, the model produces vocab_size scores, one for each possible next token. These scores are called logits.

The problem: logits have no inherent scale. A logit of 2.5 means nothing on its own. What we need is a way to compare the model's predictions against reality — against the actual next token that appeared in the training data. The loss function does exactly this. It takes two inputs:

  1. Predictions — the logits from the model, shape (B, T, vocab_size).
  2. Targets — the actual next tokens, shape (B, T), where each value is an integer index into the vocabulary.

And it produces a single number: the loss. This number quantifies how far the model's predictions are from reality. A high loss means the model is very wrong. A low loss means its predictions closely match the true data. Training is the process of adjusting the model's weights to make this number as small as possible.

The loss is the only feedback signal. Without it, the model has no way to improve. The loss function is the bridge between "here is what you predicted" and "here is what you should have predicted." Every weight update during training flows from this single number, through backpropagation, all the way back to every parameter in the model.

For language models, the standard loss function is cross-entropy loss. To understand it, we first need to understand how to convert raw logits into probabilities.

From Logits to Probabilities: Softmax

Logits are raw scores. They can be any real number: positive, negative, large, small. We need to turn them into a probability distribution: a set of non-negative numbers that sum to 1. The function that does this is called softmax.

softmax(zᵢ) = exp(zᵢ) / Σⱼ exp(zⱼ) For each element zᵢ in the input vector: 1. Exponentiate it: exp(zᵢ) — this makes it positive 2. Divide by the sum of all exponentiated values — this makes them sum to 1

Let us work through a concrete example. Suppose our vocabulary has just 3 tokens, and the model outputs these logits for one position:

Logits: [2.0, 1.0, 0.1]

Step 1 — Exponentiate:

  • exp(2.0) = 7.389
  • exp(1.0) = 2.718
  • exp(0.1) = 1.105

Step 2 — Sum: 7.389 + 2.718 + 1.105 = 11.212

Step 3 — Normalize:

  • 7.389 / 11.212 = 0.659
  • 2.718 / 11.212 = 0.242
  • 1.105 / 11.212 = 0.099

Probabilities: [0.659, 0.242, 0.099] — they sum to 1.0 ✓

Notice two important properties of softmax:

  • Order is preserved. The largest logit (2.0) gets the highest probability (0.659). Softmax does not change the ranking; it turns scores into a valid probability distribution while keeping the same ordering.
  • It amplifies differences. The gap between logits 2.0 and 0.1 is just 1.9, but the gap between probabilities 0.659 and 0.099 is 6.7x. The exponential function spreads things out. As logits grow larger, softmax becomes more "peaky" — concentrating almost all probability mass on the highest logit.

Numerical Stability: The Max Subtraction Trick

There is a practical problem with the naive softmax formula. If logits are large (say 1000.0), then exp(1000.0) is astronomically huge — it overflows to infinity in floating-point arithmetic. If logits are very negative (say -1000.0), then exp(-1000.0) underflows to zero, and you get 0/0.

The fix is simple and elegant. For any constant c, the following identity holds:

softmax(zᵢ) = exp(zᵢ - c) / Σⱼ exp(zⱼ - c) The standard choice is c = max(z).

Why does this work? Subtracting a constant from every logit does not change the softmax output (the constant cancels in the numerator and denominator). But choosing c = max(z) ensures that the largest exponent is exp(0) = 1, and all other exponents are between 0 and 1. No overflow, no underflow.

Every production implementation does this. PyTorch, LibTorch, TensorFlow, JAX — they all subtract the max before exponentiating. If you ever implement softmax yourself, always include this step. Without it, your code will produce NaN for any moderately large input.

In C++ with LibTorch, the stabilized softmax looks like this:

torch::Tensor stable_softmax(torch::Tensor logits) { torch::Tensor shifted = logits - std::get<0>(logits.max(-1, /*keepdim=*/true)); torch::Tensor exps = torch::exp(shifted); return exps / exps.sum(-1, /*keepdim=*/true); }

Cross-Entropy: Measuring Surprise

Now that we can convert logits to probabilities, we need a way to measure how bad those probabilities are. This is where information theory gives us a beautiful answer.

Self-Information: How Surprised Are You?

Imagine you are told "the sun rose today." You are not surprised at all — it happens every day. Now imagine you are told "it snowed in the Sahara desert." You are very surprised. Information theory formalizes this intuition:

I(x) = -log(p(x)) The rarer an event (lower p), the more "information" or "surprise" it carries.

If p(x) = 1.0 (certain event), then I(x) = -log(1) = 0. No surprise at all. If p(x) = 0.01 (rare event), then I(x) = -log(0.01) = 4.6. Highly surprising. The log ensures that surprise scales nicely: two independent events with probabilities p and q give combined surprise -log(p) + (-log(q)) = -log(pq).

Entropy: Expected Surprise

Entropy is the average surprise you expect from a distribution. If you know the true distribution p, the entropy is:

H(p) = -Σₓ p(x) log(p(x)) This is the minimum average number of bits needed to encode outcomes from p.

A fair coin has entropy log(2) ≈ 0.693 nats. A loaded coin (99% heads) has entropy near 0 — you are almost never surprised.

Cross-Entropy: Surprise Under the Wrong Model

Now the key idea. Suppose the true distribution is p (the actual next tokens from the training data), but you are using a different distribution q (the model's predicted probabilities) to make predictions. The cross-entropy measures your average surprise:

H(p, q) = -Σₓ p(x) log(q(x)) "How surprised is model q when it sees data drawn from distribution p?"

Cross-entropy is always greater than or equal to the entropy of the true distribution: H(p, q) ≥ H(p). The gap is called the KL divergence. When q = p (perfect model), cross-entropy equals entropy. Training minimizes cross-entropy, pushing q toward p.

The One-Hot Simplification

In language modeling, our targets are one-hot: for each position, there is exactly one correct next token. If the correct token is class c, then p(c) = 1 and p(x) = 0 for all x ≠ c. The sum collapses:

H(p, q) = -Σₓ p(x) log(q(x)) = -(1) · log(q(c)) - (0) · log(q(other)) - ... = -log(q(c)) Cross-entropy with one-hot targets = negative log probability of the correct class.

This is the formula you will see everywhere in practice. The loss for a single prediction is simply -log(qcorrect), where qcorrect is the probability the model assigned to the correct token.

Intuition checkpoint. If the model assigns probability 0.9 to the correct token, the loss is -log(0.9) = 0.105. Very small — the model is confident and correct. If the model assigns probability 0.01, the loss is -log(0.01) = 4.605. Very large — the model is essentially saying "I did not expect this at all." The loss measures how surprised the model is by the truth.

For a batch of predictions, we average the per-position losses:

Loss = -(1/N) Σᵢ log(qᵢ(cᵢ)) where N is the total number of predictions and cᵢ is the correct class for prediction i.

Why Cross-Entropy, Not MSE?

A natural question: why not just use Mean Squared Error? We could compute softmax probabilities, then measure the squared distance between the predicted distribution and the one-hot target. It would work, technically. But cross-entropy is dramatically better for classification. Here is why.

PropertyCross-Entropy LossMean Squared Error
Gradient when very wrong Large. If q(c) = 0.01, gradient ∝ (1 - 0.01) = 0.99. Strong push to fix the mistake. Small. Softmax saturation means gradients vanish. The model barely updates.
Gradient when almost right Small. If q(c) = 0.99, gradient ∝ (1 - 0.99) = 0.01. Gentle fine-tuning. Also small. Both loss functions agree here.
Gradient flow through softmax Clean. The gradient of CE w.r.t. logits simplifies to q - p (predicted minus target). Elegant and fast. Involves the Jacobian of softmax (a full matrix), making gradients more complex and slower.
Information-theoretic grounding Yes. Minimizing CE is equivalent to maximizing log-likelihood and minimizing KL divergence from the true distribution. No. MSE on probabilities has no principled interpretation for classification.
Standard in practice Universal. Used by GPT, BERT, image classifiers, and virtually all classification models. Rarely used for classification. Common for regression tasks.
The vanishing gradient problem with MSE is critical. When the model makes a confident wrong prediction (say, puts 0.99 probability on the wrong token), cross-entropy produces a huge gradient that forces rapid correction. MSE produces a tiny gradient because the softmax Jacobian squashes it. This means MSE-trained classifiers learn much more slowly from their biggest mistakes — exactly the opposite of what you want.

The key mathematical insight: the gradient of cross-entropy loss with respect to the logits is simply qᵢ - pᵢ, where q is the softmax output and p is the target. This means the gradient is directly proportional to the error. No vanishing, no exploding — just clean, linear feedback.

The Log-Softmax Trick

In practice, we never actually compute softmax and then take the log separately. Doing so is numerically unstable because softmax can output very small numbers (like 1e-38), and log of a very small number is a very large negative number that loses precision.

Instead, we compute log-softmax directly. Starting from the definition:

log(softmax(zᵢ)) = log( exp(zᵢ) / Σⱼ exp(zⱼ) ) = zᵢ - log( Σⱼ exp(zⱼ) )

We split the log of a fraction into a difference of logs. The numerator simplifies beautifully: log(exp(zᵢ)) = zᵢ. No exponentiation needed for the numerator at all!

But we still need to compute log(Σ exp(zⱼ)) — the LogSumExp. Here we apply the same max-subtraction trick:

log(Σⱼ exp(zⱼ)) = max(z) + log(Σⱼ exp(zⱼ - max(z))) Putting it all together: log_softmax(zᵢ) = zᵢ - max(z) - log(Σⱼ exp(zⱼ - max(z)))

Let us verify with our running example:

Logits: [2.0, 1.0, 0.1], max = 2.0

Shifted: [0.0, -1.0, -1.9]

Exponentiate shifted: [1.000, 0.368, 0.150]

Sum of exps: 1.000 + 0.368 + 0.150 = 1.518

log(sum): log(1.518) = 0.417

Log-softmax:

  • 0.0 - 0.417 = -0.417
  • -1.0 - 0.417 = -1.417
  • -1.9 - 0.417 = -2.317

Compare: log([0.659, 0.242, 0.099]) = [-0.417, -1.417, -2.313]. Matches! (The tiny difference is rounding.)

Notice what we gained: the entire computation stays in a numerically reasonable range. We never compute a probability that could be 1e-38, then take its log to get -87.5. Instead we work with shifted logits (bounded by 0 on top) and a moderate log-sum. This is exactly what torch::log_softmax and torch::nn::functional::cross_entropy do internally.

The full cross-entropy in one line. Since CE loss = -log(softmax(zcorrect)), and log-softmax gives us log(softmax(z)) directly, we just need to pick the element at the correct class index and negate it. That is it. No separate softmax, no separate log.

The Reshape Trick

There is one more practical detail before we can compute the loss. Our GPT model outputs logits of shape (B, T, vocab_size) — a three-dimensional tensor. But the cross-entropy function in LibTorch expects a two-dimensional input: (N, C) where N is the number of samples and C is the number of classes. Similarly, the targets should be a flat (N,) vector.

The solution is simple: reshape (flatten) the batch and time dimensions into one.

TensorOriginal ShapeAfter ReshapeOperation
logits (B, T, vocab_size) (B*T, vocab_size) logits.view({-1, vocab_size})
targets (B, T) (B*T,) targets.view({-1})

With concrete numbers: if B=4 and T=32, we go from (4, 32, vocab_size) to (128, vocab_size) for logits, and from (4, 32) to (128,) for targets. Each of the 128 rows is now an independent classification problem: "given these logits over the vocabulary, what is the correct next token?"

// The reshape trick in C++ int vocab_size = logits.size(-1); torch::Tensor logits_flat = logits.view({-1, vocab_size}); // (B*T, vocab_size) torch::Tensor targets_flat = targets.view({-1}); // (B*T,) // Now compute cross-entropy torch::Tensor loss = torch::nn::functional::cross_entropy(logits_flat, targets_flat);

Why does view({-1, vocab_size}) work? The -1 tells LibTorch to infer that dimension. If the total number of elements is B*T*vocab_size and the last dimension is vocab_size, then the first dimension must be B*T. This is a zero-cost operation — no data is copied, only the shape metadata changes.

Common bug alert. The order of arguments matters. LibTorch's cross_entropy expects (predictions, targets), not (targets, predictions). Getting this backwards will not crash (the shapes might even work out if vocab_size happens to equal B*T), but the loss will be nonsense and training will not converge.

Here is the complete shape trace for a typical training step:

#OperationShapeNotes
1 GPT forward pass (4, 32, 1000) B=4, T=32, vocab=1000
2 logits.view({-1, 1000}) (128, 1000) Flatten batch & time
3 targets.view({-1}) (128,) Integer class labels
4 cross_entropy(logits_flat, targets_flat) scalar Single loss value
5 loss.backward() — Gradients computed for all parameters

C++ Implementation from Scratch

Let us implement cross-entropy loss using only basic tensor operations — no torch::nn::functional::cross_entropy. We will write every step ourselves so you can see exactly what the library does under the hood.

torch::Tensor cross_entropy_from_scratch(torch::Tensor logits, torch::Tensor targets) { // logits: (N, C) where N = number of samples, C = number of classes // targets: (N,) integer class labels // Step 1: Compute log-softmax (numerically stable) // Subtract max for stability torch::Tensor max_vals = std::get<0>(logits.max(1, /*keepdim=*/true)); torch::Tensor shifted = logits - max_vals; // Compute log-sum-exp torch::Tensor log_sum_exp = torch::log( torch::exp(shifted).sum(1, /*keepdim=*/true) ); // Log-softmax = shifted - log_sum_exp torch::Tensor log_probs = shifted - log_sum_exp; // Step 2: Gather the log-prob at each target index // targets has shape (N,), we need (N, 1) for gather torch::Tensor targets_2d = targets.unsqueeze(1); // (N, 1) torch::Tensor picked = log_probs.gather(1, targets_2d); // (N, 1) // Step 3: Negate and average torch::Tensor loss = -picked.mean(); return loss; }

Let us walk through each piece:

We subtract the row-wise maximum, exponentiate, sum, and take the log. This gives us the log-sum-exp term. Then log_probs = shifted - log_sum_exp gives us the log-softmax values for every class.

The max(1, true) operates along dimension 1 (the class dimension) and keeps the dimension so broadcasting works. Similarly, sum(1, true) sums along the class dimension.

gather(1, targets_2d) is the key operation. For each row i, it picks the element at column targets[i]. If targets = [0, 2, 1] and log_probs has 3 rows, gather picks log_probs[0][0], log_probs[1][2], log_probs[2][1]. This is how we select the log-probability of the correct class.

We negate because cross-entropy is -log(pcorrect), and we average over all N samples to get a single scalar loss. The mean() call handles the averaging.

The One-Liner: LibTorch's Built-in Version

In practice, you would use the built-in function that does all of the above (with additional optimizations for numerical precision and GPU efficiency):

// The same thing, but let LibTorch handle the details torch::Tensor loss = torch::nn::functional::cross_entropy(logits, targets);

This single line computes log-softmax and negative-log-likelihood internally. It expects logits of shape (N, C) (raw scores, not probabilities) and targets of shape (N,) (integer labels, not one-hot).

Why implement it from scratch? Understanding every step makes debugging much easier. When your loss suddenly becomes NaN, you will know to check for overflow in exp(), not just stare at a mysterious one-liner. When your loss plateaus at log(vocab_size), you will recognize that the model is predicting a uniform distribution.

Verifying Correctness

You can verify your from-scratch implementation matches the built-in one:

torch::Tensor logits = torch::tensor({{2.0, 1.0, 0.1}}); torch::Tensor targets = torch::tensor({0}, torch::kLong); torch::Tensor loss_scratch = cross_entropy_from_scratch(logits, targets); torch::Tensor loss_builtin = torch::nn::functional::cross_entropy(logits, targets); cout << "From scratch: " << loss_scratch.item<float>() << endl; cout << "Built-in: " << loss_builtin.item<float>() << endl; // Both print: 0.4170

Full Code

Here is the complete, self-contained file. It implements cross-entropy loss from scratch, compares it against the LibTorch built-in, and demonstrates the reshape trick with GPT-shaped tensors.

#include <torch/torch.h> #include <iostream> using namespace std; // ===================================================== // Cross-Entropy Loss from Scratch // ===================================================== torch::Tensor cross_entropy_from_scratch(torch::Tensor logits, torch::Tensor targets) { // logits: (N, C) - raw unnormalized scores // targets: (N,) - integer class labels // --- Log-Softmax (numerically stable) --- // Subtract row-wise max for numerical stability torch::Tensor max_vals = std::get<0>(logits.max(1, /*keepdim=*/true)); torch::Tensor shifted = logits - max_vals; // log-sum-exp of shifted logits torch::Tensor log_sum_exp = torch::log( torch::exp(shifted).sum(1, /*keepdim=*/true) ); // log_softmax = shifted - log_sum_exp torch::Tensor log_probs = shifted - log_sum_exp; // --- Negative Log Likelihood --- // Gather the log-probability at each target index torch::Tensor targets_2d = targets.unsqueeze(1); torch::Tensor picked = log_probs.gather(1, targets_2d); // Negate and average over all samples torch::Tensor loss = -picked.mean(); return loss; } // ===================================================== // Main: Demonstrate and Verify // ===================================================== int main() { // ----- Test 1: Simple example ----- cout << "=== Test 1: Simple 1x3 logits ===" << endl; torch::Tensor logits1 = torch::tensor({{2.0, 1.0, 0.1}}); torch::Tensor targets1 = torch::tensor({0}, torch::kLong); torch::Tensor loss_scratch = cross_entropy_from_scratch(logits1, targets1); torch::Tensor loss_builtin = torch::nn::functional::cross_entropy( logits1, targets1 ); cout << "Logits: " << logits1 << endl; cout << "Target: " << targets1 << endl; cout << "From scratch: " << loss_scratch.item<float>() << endl; cout << "Built-in: " << loss_builtin.item<float>() << endl; cout << endl; // ----- Test 2: Batch of predictions ----- cout << "=== Test 2: Batch of 4 samples, 5 classes ===" << endl; torch::Tensor logits2 = torch::randn({4, 5}); torch::Tensor targets2 = torch::randint(0, 5, {4}, torch::kLong); torch::Tensor loss2_scratch = cross_entropy_from_scratch(logits2, targets2); torch::Tensor loss2_builtin = torch::nn::functional::cross_entropy( logits2, targets2 ); cout << "From scratch: " << loss2_scratch.item<float>() << endl; cout << "Built-in: " << loss2_builtin.item<float>() << endl; cout << "Match: " << (abs(loss2_scratch.item<float>() - loss2_builtin.item<float>()) < 1e-5 ? "YES" : "NO") << endl; cout << endl; // ----- Test 3: The reshape trick (GPT-shaped tensors) ----- cout << "=== Test 3: GPT reshape trick ===" << endl; int B = 4, T = 32, vocab_size = 1000; // Simulate GPT output torch::Tensor gpt_logits = torch::randn({B, T, vocab_size}); torch::Tensor gpt_targets = torch::randint(0, vocab_size, {B, T}, torch::kLong); cout << "GPT logits shape: " << gpt_logits.sizes() << endl; cout << "GPT targets shape: " << gpt_targets.sizes() << endl; // Reshape: (B, T, vocab_size) -> (B*T, vocab_size) torch::Tensor logits_flat = gpt_logits.view({-1, vocab_size}); torch::Tensor targets_flat = gpt_targets.view({-1}); cout << "Flattened logits: " << logits_flat.sizes() << endl; cout << "Flattened targets: " << targets_flat.sizes() << endl; // Compute loss both ways torch::Tensor loss3_scratch = cross_entropy_from_scratch( logits_flat, targets_flat ); torch::Tensor loss3_builtin = torch::nn::functional::cross_entropy( logits_flat, targets_flat ); cout << "From scratch: " << loss3_scratch.item<float>() << endl; cout << "Built-in: " << loss3_builtin.item<float>() << endl; cout << "Match: " << (abs(loss3_scratch.item<float>() - loss3_builtin.item<float>()) < 1e-4 ? "YES" : "NO") << endl; cout << endl; // ----- Test 4: Expected loss for random model ----- cout << "=== Test 4: Sanity check ===" << endl; float expected_loss = log((float)vocab_size); cout << "Expected loss for random model: -log(1/" << vocab_size << ") = " << expected_loss << endl; cout << "Actual loss: " << loss3_builtin.item<float>() << endl; cout << "(Should be close!)" << endl; return 0; }

Let us look at what each test verifies:

We use our running example: logits [2.0, 1.0, 0.1] with target class 0. Both implementations should return 0.4170. This is our ground-truth sanity check.

A batch of 4 random predictions over 5 classes. We verify that our from-scratch version matches the built-in version to within 1e-5. Any discrepancy larger than that would indicate a bug in our implementation.

The realistic case: GPT-shaped outputs with B=4, T=32, vocab_size=1000. We apply the reshape trick, then compute loss both ways. This tests that our implementation handles the shape that a real training loop would produce.

A randomly initialized model should assign roughly equal probability to every token. That probability is approximately 1/vocab_size, so the expected loss is -log(1/vocab_size) = log(vocab_size). For vocab_size=1000, that is about 6.908. If your initial loss is dramatically different from this, something is wrong with your model or loss computation.

The log(vocab_size) sanity check is invaluable. Before you train, print the loss on the first batch. If it is not close to log(vocab_size), investigate immediately. Common causes of wrong initial loss: incorrect reshape, transposed logits/targets arguments, wrong data type (float targets instead of long), or accidentally applying softmax before passing to cross_entropy.

What Comes Next

We now have all the pieces to evaluate a model:

  1. GPT model — takes token IDs, produces logits. (Previous post.)
  2. Cross-entropy loss — takes logits and targets, produces a scalar loss. (This post.)

But evaluation alone does not make the model learn. We need three more things to complete the training loop:

  • Backpropagation (loss.backward()) — computes the gradient of the loss with respect to every parameter in the model. LibTorch's autograd engine handles this automatically.
  • An optimizer (Adam, SGD, etc.) — uses the gradients to update the model's weights. We will use AdamW, the standard for transformers.
  • A data pipeline — loads text, tokenizes it, creates (input, target) pairs where the target is the input shifted by one position, and batches them for efficient GPU processing.

The next post will tie all of this together into a complete training loop: load data, forward pass, compute loss, backward pass, update weights, repeat.

for (int step = 0; step < max_steps; step++) { // 1. Get a batch of data auto [inputs, targets] = get_batch(data, batch_size, block_size); // 2. Compute loss (what we built today) torch::Tensor logits_flat = logits.view({-1, vocab_size}); torch::Tensor targets_flat = targets.view({-1}); torch::Tensor loss = torch::nn::functional::cross_entropy( logits_flat, targets_flat ); // 4. Backward pass: compute gradients optimizer.zero_grad(); loss.backward(); // 5. Update weights optimizer.step(); }
