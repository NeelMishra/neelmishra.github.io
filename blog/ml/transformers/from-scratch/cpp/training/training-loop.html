The Training Loop in C++ (LibTorch)

ML Series ML · Transformers · From Scratch · C++ · Training

In the previous posts we built a complete GPT model from scratch — the full architecture from single attention heads all the way up to the final language model head. We then built the cross-entropy loss function that measures how wrong the model’s predictions are. But a model that never learns is just an expensive random number generator. This post is about making it learn.

Series roadmap. Single Head → Multi-Head Attention → Feed-Forward → Transformer Block → Positional Encoding → Full GPT Model → The Training Loop (you are here).

The Paradigm Shift

Up to this point, everything we have built is static. The GPT class has millions of weights, but they are all random. If you feed in [The, cat, sat], the model will predict the next token with all the confidence of a coin flip — because it has never seen language before.

Training is the process that transforms a randomly initialized model into one that can actually predict the next word. The idea is deceptively simple:

  1. Show the model some data. Feed in a sequence of tokens and let it predict the next token at every position.
  2. Measure its mistakes. Compare the model’s predictions (logits) against the actual next tokens using cross-entropy loss. A high loss means the model is very wrong.
  3. Compute which direction to nudge each weight. This is backpropagation — using calculus (specifically, the chain rule) to compute exactly how much each weight contributed to the error.
  4. Nudge the weights. Adjust each weight by a tiny amount in the direction that reduces the loss. This is the optimizer’s job.
  5. Repeat. Thousands, millions, or billions of times.

That is the entire algorithm. Every neural network ever trained — from a two-layer perceptron in the 1980s to GPT-4 — follows this same loop. The differences are in the details: how you prepare the data, which optimizer you use, how you schedule the learning rate, and how you prevent the gradients from exploding. We will cover all of these in this post.

The core insight. Training is not a single clever algorithm. It is a loop. Each iteration makes the model slightly less wrong. After enough iterations, “slightly less wrong” compounds into “remarkably good.” GPT-3 was trained for approximately 300 billion tokens. Each token contributed one tiny nudge to the weights.

Data for Language Models

Before we can train, we need training data. For a language model, the data is simply text — lots of it. But raw text is not what the model sees. We need to convert it into (input, target) pairs that the model can learn from.

Tokenization

First, we convert raw text into a sequence of integer token IDs. A tokenizer maps each word (or subword) to a unique integer. For simplicity, assume a character-level tokenizer where each character gets its own ID:

// Raw text "the cat sat on the mat" // After tokenization (example IDs) [52, 19, 8, 3, 72, 44, 52, 3, 91, 44, 52, 3, 28, 16, 3, 52, 19, 8, 3, 15, 44, 52]

Creating (Input, Target) Pairs

A language model learns to predict the next token given the preceding tokens. So we create training examples by sliding a window of size block_size over our token sequence. The input is the first block_size tokens, and the target is the same window shifted right by one position:

The key insight: at every position t, the model sees all tokens from 0 to t (thanks to the causal mask) and must predict the token at position t+1. One training example of length block_size actually produces block_size separate prediction tasks, one per position. This is extremely data-efficient.

block_size: The Context Window

block_size (sometimes called context_length or max_seq_len) determines how many tokens the model can look back at when predicting the next one. GPT-2 uses block_size = 1024. Our toy model uses block_size = 32. The model literally cannot see anything beyond this window — earlier tokens might as well not exist.

The Data Loader in Pseudo-Code

// Prepare training data from a token sequence torch::Tensor data = tokenize(raw_text); // shape: (N,) — all tokens torch::Tensor get_batch(torch::Tensor data, int block_size, int batch_size) { int n = data.size(0); // Pick random starting positions torch::Tensor offsets = torch::randint(0, n - block_size, {batch_size}); torch::Tensor inputs = torch::zeros({batch_size, block_size}, torch::kLong); torch::Tensor targets = torch::zeros({batch_size, block_size}, torch::kLong); for (int i = 0; i < batch_size; i++) { int start = offsets[i].item<int>(); inputs[i] = data.slice(0, start, start + block_size); targets[i] = data.slice(0, start + 1, start + block_size + 1); } return {inputs, targets}; // both shape: (batch_size, block_size) }
Why shift by one, not by block_size? Each input position t creates a separate training signal: “given tokens 0..t, predict token t+1.” The causal mask inside the attention mechanism ensures that position t cannot peek at tokens after it. So a single block of length 32 gives us 32 training examples for free.

The Core Loop: Forward → Loss → Backward → Step

Every training iteration in every deep learning system follows the same four operations. Here they are, one by one.

Pass the input tokens through the model to get logits — raw, unnormalized scores over the vocabulary for each position in the sequence.

torch::Tensor logits = model->forward(input_tokens); // input_tokens shape: (B, T) — integer token IDs // logits shape: (B, T, V) — V = vocab_size

At this point, the logits are essentially random predictions. For an untrained model with a vocabulary of 1,000 tokens, every token has roughly equal probability — about 0.1%. The model is guessing.

Compare the model’s predictions against the actual targets using cross-entropy loss. This produces a single scalar number that quantifies how wrong the model is.

// Reshape for cross_entropy: (B*T, V) vs (B*T,) torch::Tensor loss = torch::nn::functional::cross_entropy( logits.view({-1, vocab_size}), targets.view({-1}) );

For a randomly initialized model with vocab_size = 1000, the initial loss is approximately ln(1000) ≈ 6.91. This is because the model assigns roughly equal probability to all 1,000 tokens. As training progresses, this number should steadily decrease.

What does the loss number mean? Cross-entropy loss of 6.91 means “the model is as surprised by the correct answer as if it were choosing uniformly at random from 1,000 options.” A loss of 4.0 means it has effectively narrowed down to about e&sup4; ≈ 55 candidates. A loss of 2.0 means about 7 candidates. Lower is always better.

Compute the gradient of the loss with respect to every learnable parameter in the model. This tells us, for each weight, which direction to move it to reduce the loss.

loss.backward(); // After this call, every parameter tensor has a .grad field // containing dLoss/dParameter

This single line invokes the entire backpropagation algorithm. LibTorch’s autograd engine traverses the computational graph in reverse, applying the chain rule at every node to compute gradients for every weight in the model. For our 220,776-parameter GPT, this computes 220,776 gradient values.

Use the computed gradients to update every weight in the model. The optimizer decides how to use the gradients — simple SGD just subtracts lr * gradient from each weight; Adam is much smarter (see below).

optimizer.step(); // Update all weights optimizer.zero_grad(); // Clear gradients for next iteration

The zero_grad() call is critical. PyTorch and LibTorch accumulate gradients by default — each .backward() call adds to the existing .grad values rather than replacing them. If you forget to zero the gradients, the model sees stale gradients from previous iterations mixed in with the current ones, and training goes haywire.

That is it. Four operations, repeated thousands of times. The entire training loop in its most distilled form is just seven lines:

for (int step = 0; step < max_steps; step++) { auto [input, target] = get_batch(data, block_size, batch_size); torch::Tensor logits = model->forward(input); torch::Tensor loss = torch::nn::functional::cross_entropy( logits.view({-1, vocab_size}), target.view({-1})); loss.backward(); optimizer.step(); optimizer.zero_grad(); }

Everything else we cover in this post — Adam, learning rate schedules, gradient clipping — is an enhancement to make this loop converge faster and more reliably. But the skeleton is always these four phases.

Backpropagation and Autograd

Backpropagation is the algorithm that makes step 3 possible. It computes the gradient of the loss with respect to every parameter in the model. Without it, we would not know which direction to nudge each weight.

The Chain Rule

Consider a simple chain of operations: loss = f(g(h(w))), where w is a weight. By the chain rule:

dLoss/dw = (dLoss/df) * (df/dg) * (dg/dh) * (dh/dw)

A neural network is just a very long chain of operations. The GPT model forward pass is: embedding → positional encoding → transformer block 0 → … → transformer block N → LayerNorm → linear head → cross-entropy. Each of these operations has well-defined local derivatives. The chain rule lets us multiply them all together to get dLoss/dWeight for every weight, no matter how deep it is in the network.

The Computational Graph

When you run the forward pass, LibTorch secretly builds a computational graph — a record of every operation and every intermediate tensor. When you call loss.backward(), it traverses this graph in reverse (hence “backpropagation”), computing gradients at every node using the chain rule.

Why zero_grad() Matters

LibTorch accumulates gradients by design. Each call to .backward() adds the new gradients to whatever is already in .grad. This is useful in certain advanced scenarios (like gradient accumulation across micro-batches), but for a standard training loop it means you must clear the gradients before each backward pass:

// Without zero_grad (BUG!): // step 1: grad = g₁ // step 2: grad = g₁ + g₂ ← stale gradient from step 1! // step 3: grad = g₁ + g₂ + g₃ ← even worse // With zero_grad (correct): // step 1: grad = g₁, then zero_grad → grad = 0 // step 2: grad = g₂, then zero_grad → grad = 0 // step 3: grad = g₃ ← clean, only current batch
Common beginner bug. Forgetting zero_grad() is one of the most common training bugs. The symptom: loss decreases at first, then starts oscillating wildly or diverges. The gradients grow without bound because old gradients keep accumulating.

The Adam Optimizer

The simplest optimizer is stochastic gradient descent (SGD): weight = weight - lr * gradient. This works, but it has two serious problems:

  1. Same learning rate for all parameters. A weight in the embedding layer and a weight in the last transformer block receive the same step size, even though they might need very different update magnitudes.
  2. No momentum. If the gradient oscillates (positive, negative, positive, negative), SGD oscillates with it instead of smoothing through the noise.

Adam (Adaptive Moment Estimation) solves both problems. It maintains two running averages for each parameter:

First Moment: Momentum

m_t = β₁ · m_{t-1} + (1 - β₁) · g_t

This is an exponential moving average of the gradients. If gradients have been consistently pointing in one direction, m_t will be large in that direction. If they oscillate, m_t will be small (the positives and negatives cancel out). This gives Adam momentum — it keeps moving in directions that are consistently useful.

Second Moment: Adaptive Learning Rate

v_t = β₂ · v_{t-1} + (1 - β₂) · g_t²

This is an exponential moving average of the squared gradients. It tracks the magnitude of recent gradients for each parameter independently. Parameters with large, frequent gradients get a smaller effective learning rate (the denominator is large). Parameters with small, infrequent gradients get a larger effective learning rate. This is the “adaptive” part of Adam.

Bias Correction

Both m_t and v_t are initialized to zero, which biases them toward zero in the early steps. Adam corrects for this:

m̂_t = m_t / (1 - β₁^t) v̂_t = v_t / (1 - β₂^t)

At step t=1, with β₁=0.9, the correction factor is 1/(1-0.9) = 10x. By step 100, it is 1/(1-0.9^100) ≈ 1.0x — the correction fades away as the running averages warm up.

The Update Rule

θ_t = θ_{t-1} - lr · m̂_t / (√v̂_t + ε)

The ε (typically 1e-8) prevents division by zero when v_t is very small.

Complete Adam Equations

m_t = β₁ · m_{t-1} + (1 - β₁) · g_t (momentum) v_t = β₂ · v_{t-1} + (1 - β₂) · g_t² (RMS of gradients) m̂_t = m_t / (1 - β₁^t) (bias correction) v̂_t = v_t / (1 - β₂^t) (bias correction) θ_t = θ_{t-1} - lr · m̂_t / (√v̂_t + ε) (weight update)

In LibTorch

torch::optim::Adam optimizer( model->parameters(), torch::optim::AdamOptions(3e-4) // learning rate .betas(std::make_tuple(0.9, 0.999)) .eps(1e-8) .weight_decay(0.01) );

AdamW: Decoupled Weight Decay

Standard Adam applies weight decay inside the gradient computation: it adds λ * weight to the gradient before computing the adaptive step. This means the weight decay is also scaled by the adaptive learning rate, which is not the intended behavior.

AdamW (the “W” stands for “decoupled Weight decay”) fixes this by applying weight decay directly to the weights, outside of the adaptive mechanism:

Adam: θ_t = θ_{t-1} - lr · m̂_t / (√v̂_t + ε) - lr · λ · m̂(regularized) AdamW: θ_t = θ_{t-1} - lr · m̂_t / (√v̂_t + ε) - lr · λ · θ_{t-1}

In LibTorch, setting .weight_decay(0.01) on AdamOptions gives you the standard L2 regularization variant. For true decoupled weight decay (AdamW), you would handle the decay manually or use the appropriate option. In practice, both variants produce good results for transformer training, and the difference matters most at large scale.

Why Adam dominates transformer training. Every major language model — GPT-2, GPT-3, LLaMA, PaLM — uses Adam or AdamW. SGD with momentum can work for vision models (ResNets), but transformers are notoriously sensitive to optimizer choice. Adam’s adaptive learning rates handle the wide variation in gradient magnitudes across attention weights, embedding matrices, and feed-forward layers.

Learning Rate Schedule

Using a constant learning rate for the entire training run is suboptimal. The optimal learning rate changes as training progresses:

  • Early training: The weights are random and gradients are noisy. A large learning rate would cause wild oscillations.
  • Mid training: The model has found the general direction. A higher learning rate helps it make rapid progress.
  • Late training: The model is close to a good solution. A large learning rate would overshoot. We need to slow down for fine-grained optimization.

The most common schedule for transformers is warmup + cosine decay:

Phase 1: Linear Warmup

lr(t) = max_lr · (t / warmup_steps) for t < warmup_steps

Start from zero (or near-zero) and linearly increase to the peak learning rate over the first warmup_steps steps. This prevents the random initial gradients from causing destructive weight updates.

Phase 2: Cosine Decay

lr(t) = min_lr + 0.5 · (max_lr - min_lr) · (1 + cos(π · progress)) where progress = (t - warmup_steps) / (max_steps - warmup_steps)

After warmup, the learning rate follows a cosine curve from max_lr down to min_lr. The cosine shape means the decay is slow at first, fast in the middle, and slow again near the end — a natural annealing schedule.

The Learning Rate Curve

Implementation in C++

float get_lr(int step, float max_lr, float min_lr, int warmup_steps, int max_steps) { if (step < warmup_steps) { // Linear warmup return max_lr * (static_cast<float>(step) / warmup_steps); } // Cosine decay float progress = static_cast<float>(step - warmup_steps) / (max_steps - warmup_steps); return min_lr + 0.5f * (max_lr - min_lr) * (1.0f + cos(M_PI * progress)); }
Why warmup? In the first few steps, the Adam optimizer’s second moment estimates (v_t) are nearly zero, which makes the effective step size enormous. The bias correction partially fixes this, but warmup provides additional stability. Without warmup, transformers often diverge in the first few hundred steps.

Gradient Clipping

Even with Adam and a good learning rate schedule, training can occasionally produce very large gradients. This happens when the loss landscape has a sharp cliff — a small change in weights causes a huge change in loss, producing enormous gradients. One step with these gradients can catapult the weights into a bad region, causing training to diverge.

Gradient Norm Clipping

The solution is simple: before the optimizer step, compute the total norm of all gradients. If it exceeds a threshold (max_norm), scale all gradients down proportionally so the total norm equals max_norm:

total_norm = √(Σ ||g_i||²) for all parameter gradients g_i if total_norm > max_norm: g_i = g_i · (max_norm / total_norm) for all i

This preserves the direction of the gradient update but caps its magnitude. The model still moves in the right direction; it just takes a smaller step.

In LibTorch

torch::nn::utils::clip_grad_norm_(model->parameters(), 1.0); // Call AFTER loss.backward() and BEFORE optimizer.step()

That is one line. A max_norm of 1.0 is the standard choice for transformer training — used by GPT-2, GPT-3, LLaMA, and most other large language models. The intuition: in a well-trained model, the gradient norm is typically between 0.1 and 1.0. A norm above 1.0 is unusually large and likely caused by a noisy batch or numerical instability.

Order matters. The correct sequence is: loss.backward() → clip_grad_norm_() → optimizer.step() → optimizer.zero_grad(). Clipping before backward does nothing (gradients do not exist yet). Clipping after step is too late (weights are already updated).

When Clipping Activates

During normal training, clipping rarely activates — the gradient norm stays below 1.0 most of the time. But during critical moments (early training with random weights, encountering an unusual data batch, or when the learning rate is too high), clipping acts as a safety net that prevents catastrophic weight updates. It is cheap insurance.

Putting It All Together: The Complete Training Script

Here is the full training script in C++ using LibTorch. It assumes the GPT class from the previous post is already defined. For demonstration, we use random tokens as training data — the model learns to memorize the pattern, proving the training loop works.

// ========================================================= // Complete Training Script for GPT in C++ (LibTorch) // ========================================================= // Assumes GPT class is defined above (from previous post) #include <torch/torch.h> #include <iostream> #include <cmath> using namespace std; // --------------------------------------------------------- // Hyperparameters // --------------------------------------------------------- const int vocab_size = 1000; const int n_embd = 64; const int num_heads = 4; const int head_size = 16; // n_embd / num_heads const int d_ff = 256; // 4 * n_embd const int block_size = 32; const int num_layers = 2; const int batch_size = 4; const int max_steps = 5000; const int warmup_steps = 100; const int eval_every = 100; const float max_lr = 3e-4; const float min_lr = 1e-5; const float grad_clip = 1.0; const float weight_decay = 0.01; // --------------------------------------------------------- // Learning Rate Schedule: Warmup + Cosine Decay // --------------------------------------------------------- float get_lr(int step) { if (step < warmup_steps) { return max_lr * (static_cast<float>(step) / warmup_steps); } float progress = static_cast<float>(step - warmup_steps) / (max_steps - warmup_steps); return min_lr + 0.5f * (max_lr - min_lr) * (1.0f + cos(M_PI * progress)); } // --------------------------------------------------------- // Data: Generate synthetic training data // In practice, this would be a real tokenized text corpus // --------------------------------------------------------- torch::Tensor train_data = torch::randint( 0, vocab_size, {10000}, torch::kLong ); torch::Tensor val_data = torch::randint( 0, vocab_size, {1000}, torch::kLong ); // --------------------------------------------------------- // Get a random batch of (input, target) pairs // --------------------------------------------------------- pair<torch::Tensor, torch::Tensor> get_batch(torch::Tensor& data, torch::Device device) { int n = data.size(0); torch::Tensor offsets = torch::randint( 0, n - block_size - 1, {batch_size} ); torch::Tensor inputs = torch::zeros( {batch_size, block_size}, torch::kLong ); torch::Tensor targets = torch::zeros( {batch_size, block_size}, torch::kLong ); for (int i = 0; i < batch_size; i++) { int s = offsets[i].item<int>(); inputs[i] = data.slice(0, s, s + block_size); targets[i] = data.slice(0, s + 1, s + block_size + 1); } return {inputs.to(device), targets.to(device)}; } // --------------------------------------------------------- // Evaluate loss on validation data (no gradients) // --------------------------------------------------------- float estimate_val_loss( shared_ptr<GPT>& model, torch::Tensor& data, torch::Device device, int eval_iters = 20 ) { model->eval(); // Switch to evaluation mode torch::NoGradGuard no_grad; // Disable gradient tracking float total_loss = 0.0f; for (int i = 0; i < eval_iters; i++) { auto [input, target] = get_batch(data, device); torch::Tensor logits = model->forward(input); torch::Tensor loss = torch::nn::functional::cross_entropy( logits.view({-1, vocab_size}), target.view({-1}) ); total_loss += loss.item<float>(); } model->train(); // Switch back to training mode return total_loss / eval_iters; } // --------------------------------------------------------- // Main: The Training Loop // --------------------------------------------------------- int main() { // Device selection torch::Device device = torch::cuda::is_available() ? torch::kCUDA : torch::kCPU; cout << "Using device: " << (device == torch::kCUDA ? "CUDA" : "CPU") << endl; // Create the model auto model = make_shared<GPT>( vocab_size, n_embd, num_heads, head_size, d_ff, block_size, num_layers ); model->to(device); // Count parameters int64_t total_params = 0; for (auto& p : model->parameters()) { total_params += p.numel(); } cout << "Total parameters: " << total_params << endl; // Create the optimizer torch::optim::Adam optimizer( model->parameters(), torch::optim::AdamOptions(max_lr) .betas(std::make_tuple(0.9, 0.999)) .eps(1e-8) .weight_decay(weight_decay) ); // ===================================================== // The Training Loop // ===================================================== for (int step = 0; step < max_steps; step++) { // 1. Update learning rate float lr = get_lr(step); for (auto& group : optimizer.param_groups()) { static_cast<torch::optim::AdamOptions&>( group.options() ).lr(lr); } // 2. Get a batch of training data auto [input, target] = get_batch(train_data, device); // 3. Forward pass torch::Tensor logits = model->forward(input); // 4. Compute loss torch::Tensor loss = torch::nn::functional::cross_entropy( logits.view({-1, vocab_size}), target.view({-1}) ); // 5. Backward pass optimizer.zero_grad(); loss.backward(); // 6. Gradient clipping torch::nn::utils::clip_grad_norm_( model->parameters(), grad_clip ); // 7. Optimizer step optimizer.step(); // 8. Logging if (step % eval_every == 0) { float train_loss = loss.item<float>(); float val_loss = estimate_val_loss( model, val_data, device ); cout << "step " << step << " | lr: " << lr << " | train loss: " << train_loss << " | val loss: " << val_loss << endl; } } cout << "Training complete." << endl; return 0; }

Let us walk through the key design decisions in this script:

At the start of each step, we compute the current learning rate using our warmup + cosine decay schedule and manually set it on the optimizer. LibTorch does not have a built-in CosineAnnealingWarmRestarts scheduler like PyTorch, so we update the lr field in the optimizer’s parameter groups directly.

Note that we call zero_grad() before backward(), not after step(). Both orderings work — the important thing is that gradients are zeroed before each backward() call. Placing it before backward makes the intent clearer: “start fresh, then compute gradients.”

The estimate_val_loss function switches the model to eval() mode and wraps computation in NoGradGuard. Eval mode disables dropout (if present) and changes BatchNorm behavior. NoGradGuard tells LibTorch not to build the computational graph, saving memory and compute. Always switch back to train() mode afterward.

We check for CUDA availability and move the model and data to the GPU if available. The model is moved once with model->to(device). Batches are moved per-iteration inside get_batch(). This pattern works for both CPU and GPU training without code changes.

For our small model, the default 0.999 works fine. If you scale up, consider lowering it.

What Comes Next

We now have a complete pipeline: architecture + loss + training. The model can learn from data. But it cannot yet generate text. That is the next step.

Autoregressive decoding. To generate text, we feed the model a prompt (a sequence of token IDs), take the logits at the last position, convert them to probabilities with softmax, sample the next token, append it to the sequence, and repeat. This is the core generation loop.

Temperature. A parameter that controls randomness. Temperature = 1.0 gives the raw probabilities. Temperature < 1.0 makes the distribution sharper (more deterministic). Temperature > 1.0 makes it flatter (more random).

Top-k sampling. Instead of sampling from the full vocabulary, restrict to the top k most probable tokens. This prevents the model from generating extremely unlikely tokens while still allowing diversity.

More data. Our toy model trains on 10,000 random tokens. GPT-2 was trained on 40GB of text (WebText). GPT-3 used 300 billion tokens from a mix of internet text, books, and Wikipedia.

Bigger models. GPT-2 Small has 117M parameters. GPT-2 XL has 1.5B. GPT-3 has 175B. The training loop is identical — only the hyperparameters and the scale of compute change.

Distributed training. At the GPT-3 scale, no single GPU can hold the model. Training is distributed across hundreds or thousands of GPUs using data parallelism, model parallelism, or pipeline parallelism. The core loop (forward → loss → backward → step) remains the same; it is just orchestrated across many machines.

The training loop we built in this post is the same loop used to train GPT-2, GPT-3, and every large language model that followed. The forward pass produces logits. The loss measures error. Backpropagation computes gradients. The optimizer updates weights. Repeat. The only things that change at scale are the numbers: more parameters, more data, more GPUs, more steps. The algorithm is the same.
