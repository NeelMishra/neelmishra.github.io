    Flash Attention: Accelerating Attention via IO-Aware Tiling

    January 2025 · 12 min read

    Standard scaled-dot-product attention computes $\mathrm{Softmax}(QK^T / \sqrt{d_k})V$. The math is simple; the implementation nightmare is memory I/O. For long sequences, the intermediate matrices $QK^T$ (shape $n \times n$, cost $O(n^2)$ memory and bandwidth) dominate wall-clock time. Flash Attention (Dao et al., 2022) rewires the computation to do most work in fast SRAM and minimizes traffic to slow HBM (high-bandwidth memory). The result: 2–4× speedup and 10–20× memory reduction on long sequences, with identical outputs.

    The IO Problem: Why Standard Attention Is Bandwidth-Bound

    Modern GPUs have fast compute (TFLOPS) but limited memory bandwidth (GB/s). Attention's roofline model:

    • Compute: $O(n^2 d_k)$ FLOPs (two matrices $Q \in \mathbb{R}^{n \times d_k}$ and $K^T \in \mathbb{R}^{d_k \times n}$).
    • Memory access: $O(n^2)$ elements (the intermediate $QK^T$ logits matrix), plus reads of $Q, K, V$ and writes of output. Total: $\Theta(n^2)$ bytes.
    • Arithmetic intensity (compute/memory): $O(n^2 d_k) / \Theta(n^2 d_k) = O(1)$ — constant, not increasing with $n$.

    With constant intensity, wall time is dominated by memory bandwidth. On modern A100s with ~2 TB/s bandwidth and TFLOPS in the thousands, this is a severe limitation. For $n = 4096$ and $d_k = 128$, the $16$ MB intermediate matrix is ~100× larger than typical SRAM ($\sim 100$ KB per streaming multiprocessor). It spills to HBM, and we spend most time waiting.

    The Tiling Idea: Work in SRAM-Sized Blocks

    The key insight: reshape the computation to operate on blocks that fit in SRAM (typically $\sim 16$–$32$ KB useful per SM). Load a block of $Q$ rows (say, $B_r$ rows), a block of $K$ columns ($B_c$ columns), compute the dot products, and write partial results back — all within SRAM. The intermediate $B_r \times B_c$ softmax matrix stays in cache; we never materialize the full $n \times n$ matrix.

    Concretely, partition $Q$ into row blocks (each $B_r \times d_k$) and $K, V$ into column blocks (each $d_k \times B_c$). For each (row-block, col-block) pair:

    1. Load $Q_i$ (shape $B_r \times d_k$) and $K_j$ (shape $d_k \times B_c$) into SRAM.
    2. Compute $S_{ij} = Q_i K_j^T / \sqrt{d_k}$ (shape $B_r \times B_c$, stays in SRAM).
    3. Compute softmax and weighted sum, updating running accumulators.
    4. Write output block back to HBM.
    5. Move to the next (col-block) — reuse $Q_i$, stream new $K_j, V_j$ in.
    Memory transfers: $O(n d_k)$ for $Q, K, V$ (one pass each, amortized) rather than $O(n^2)$ for the logits matrix. A $16–20×$ reduction in memory traffic.

    The Online Softmax: Incremental Stability

    The challenge: softmax requires a global max (for numerical stability) and a denominator sum over the entire row. But in a tiled computation, we process the row in chunks. We can't compute the full $\mathrm{Softmax}(Q_i K_j^T)$ until we've seen all $K_j$.

    Solution: online softmax with running statistics. Let $m_i^{(t)}$ be the running max of row $i$ through block $t$, and $\ell_i^{(t)}$ be the running denominator. After processing block $t+1$:

    $$m_i^{(t+1)} = \max(m_i^{(t)}, \max_k S_k^{(t+1)})$$ $$\ell_i^{(t+1)} = e^{m_i^{(t)} - m_i^{(t+1)}} \ell_i^{(t)} + \sum_k e^{S_k^{(t+1)} - m_i^{(t+1)}}$$ $$O_i^{(t+1)} = \frac{e^{m_i^{(t)} - m_i^{(t+1)}} O_i^{(t)} + e^{(S_i^{(t+1)} - m_i^{(t+1)}) \otimes \mathbf{1}^T} V_{(t+1)}}{{\ell_i^{(t+1)}}}$$

    These updates are numerically equivalent to the standard softmax computed all at once, but computed incrementally in SRAM, tile by tile. The rescaling $(m^{\text{old}} - m^{\text{new}})$ ensures underflow doesn't corrupt already-accumulated terms.

    Complexity and Memory Gains

    With tile size $B_r = B_c = B$:

    Metric Standard Attention Flash Attention
    FLOPs $2n^2 d_k$ $2n^2 d_k$ (same)
    Intermediate memory $O(n^2)$ — full logits $O(B^2)$ — one block
    HBM I/O $O(n^2 d_k)$ $O(n d_k + n^2 / B)$ (amortized)
    Numerical output $\mathrm{Softmax}(QK^T)V$ Identical (up to rounding)

    Typical gains on A100 (40 GB/s bandwidth, ~100 KB SRAM per SM):

    • $n = 1024$, $d_k = 64$: ~2.2× speedup.
    • $n = 4096$, $d_k = 128$: ~3.5× speedup.
    • Memory: standard needs $\sim 16$ MB for the logits; Flash needs $\sim 1.6$ MB total (SRAM blocks only).

    FlashAttention-2 and -3

    FlashAttention-2 (Dao et al., 2023) refines the tiling strategy, reducing redundant softmax reductions per SM and overlapping I/O with compute. Further 1.5–2× speedup.

    FlashAttention-3 (2024) adds even finer-grained kernel fusion and tensor-core optimizations. Marginal gains on newer hardware, but the principle remains: avoid the intermediate matrix, compute in tile-sized chunks, and keep hot data in SRAM.

    Implementation Sketch

    def flash_attention(Q, K, V, block_size=128):
    """
    Q, K, V: (batch, n_heads, seq_len, d_k)
    block_size: tile size (B_r = B_c)
    """
    N, d_k = Q.shape[-2:]
    # Normalize by sqrt(d_k) at outset
    Q = Q / math.sqrt(d_k)
    
    # Initialize running state
    m = torch.full((N,), -inf)  # running max per row
    ell = torch.zeros((N,))      # running softmax denominator
    O = torch.zeros((N, d_k))    # running output
    
    # Tile loop
    for block_start in range(0, N, block_size):
        block_end = min(block_start + block_size, N)
        Q_block = Q[block_start:block_end]  # (B_r, d_k)
        
        # Compute softmax row-by-row over K blocks
        for kv_start in range(0, N, block_size):
            kv_end = min(kv_start + block_size, N)
            K_block = K[kv_start:kv_end]     # (B_c, d_k)
            V_block = V[kv_start:kv_end]     # (B_c, d_k)
            
            # Scores: (B_r, B_c)
            S = Q_block @ K_block.T
            
            # Online softmax update
            m_new = torch.max(m[block_start:block_end, None], S.max(dim=1).values)
            ell_rescale = torch.exp(m[block_start:block_end] - m_new)
            ell[block_start:block_end] = (
                ell_rescale * ell[block_start:block_end] +
                torch.exp(S - m_new[:, None]).sum(dim=1)
            )
            m[block_start:block_end] = m_new
            
            # Update O: numerator
            P = torch.exp(S - m_new[:, None])  # softmax (numerator only)
            O_incr = P @ V_block
            O[block_start:block_end] = (
                (ell_rescale[:, None] * O[block_start:block_end]) + O_incr
            ) / ell[block_start:block_end, None]
    
    return O

    Practical Impact and Deployment

    Flash Attention powers the inference and training of nearly all modern LLMs. It's integrated into PyTorch's torch.nn.functional.scaled_dot_product_attention backend and is essential for long-context models (e.g., Claude's 100K context, LLaMA's extended contexts). On consumer hardware with limited bandwidth, the speedup is less dramatic but still substantial (2–3×). On enterprise GPUs (H100, B100), the gains are in parity with compute speedups.
