Multi-Head Attention

April 2026 · ML Series · Transformers

Why One Head Isn't Enough

Scaled dot-product attention gives every token a mechanism to look at every other token and pull information from the most relevant positions. That is a powerful idea. But a single attention head has a fundamental limitation: it produces exactly one set of attention weights per position. At each position, the model computes a single query, matches it against all keys, and blends the values according to that single pattern. It can only represent one notion of “relevance” at a time.

Language, however, is far more complex than that. Consider the sentence:

"The cat sat on the mat because it was tired."

To fully understand this sentence, a model processing the word “it” needs to simultaneously resolve multiple types of relationships:

  • Coreference resolution: “it” refers to “cat” (not “mat”). The model must attend strongly to “cat” to resolve the pronoun.
  • Syntactic dependency: “it” is the subject of “was”, so it needs to attend to the predicate that follows it.
  • Causal reasoning: The word “because” signals a causal link, and “it” needs to attend to “because” to understand why the clause exists.
  • Positional awareness: The local context around “it” (the words immediately before and after) provides additional syntactic cues.

A single attention head has only one query vector at each position. That one query vector gets compared to all keys via dot products, producing one distribution. The head must choose: does “it” primarily attend to “cat” (coreference), to “was” (syntax), to “because” (causality), or to adjacent tokens (local context)? It cannot do all of these simultaneously. The softmax normalization forces the weights to sum to 1, so giving more weight to one token necessarily takes weight away from others. A single head must compromise.

In practice, this compromise leads to blurred attention patterns. The head learns an average of several useful patterns, which means it does none of them well. It might attend somewhat to “cat” and somewhat to “was”, but the resulting blended representation is neither a good coreference signal nor a good syntactic signal. It is a muddy mixture.

The Representation Bottleneck

There is a deeper mathematical issue. A single attention head projects queries and keys into one shared d_k-dimensional subspace. All notions of similarity - syntactic, semantic, positional, discourse-level - must be encoded in the same set of learned projection matrices W_Q and W_K. These matrices define what “relevance” means, and with only one head, there is only one definition.

Think of it this way: the dot product q_i · k_j measures similarity in a particular vector space. A single head gets exactly one vector space. If syntactic relationships require one geometry and semantic relationships require a completely different geometry, one head cannot faithfully represent both. It is forced to learn a compromise geometry that partially captures each pattern but fully captures neither.

The Solution: Run Multiple Heads in Parallel

Multi-head attention solves this by running h independent attention operations in parallel, each with its own learned projections. Head 1 might specialize in local syntactic patterns. Head 2 might track coreference chains across long distances. Head 3 might focus on positional relationships. Each head operates in its own d_k-dimensional subspace, free to learn whatever notion of relevance is most useful for its particular role.

After all heads compute their outputs independently, the results are concatenated and projected back to the original dimension. The final output at each position is a rich blend of information gathered through multiple distinct lenses. The model no longer has to choose between tracking syntax and resolving coreference - it can do both at the same time, in separate heads.

The Multi-Head Mechanism

The complete formula for multi-head attention, as introduced in the “Attention Is All You Need” paper, is:

MultiHead(Q, K, V) = Concat(head_1, head_2, ..., head_h) · W_O where head_i = Attention(Q · W_Qi, K · W_Ki, V · W_Vi)

Let us dissect every piece of this formula.

The Per-Head Projections: W_Qi, W_Ki, W_Vi

Each head i has three private projection matrices:

  • W_Qi with shape (d_model × d_k) - projects queries into head i’s subspace
  • W_Ki with shape (d_model × d_k) - projects keys into head i’s subspace
  • W_Vi with shape (d_model × d_v) - projects values into head i’s subspace

These projections are the core of what makes multi-head attention work. Each head sees the same input data but through different learned lenses. The projection matrices are not shared between heads - they are independently learned parameters. This independence is what allows different heads to specialize in different tasks.

The Attention Function Inside Each Head

Once we have the projected queries, keys, and values, each head runs the standard scaled dot-product attention we covered in the previous article:

head_i = Attention(Q_i, K_i, V_i) = softmax(Q_i · K_i^T / sqrt(d_k)) · V_i

Nothing changes about the attention computation itself. The only difference is that each head operates on smaller, projected versions of Q, K, and V instead of the full d_model-dimensional vectors.

Why d_k = d_model / h

This is a critical design choice. In the original Transformer:

d_model = 512 h = 8 heads d_k = d_v = d_model / h = 512 / 8 = 64

Why this specific split? The answer is computational efficiency. A single head operating on the full d_model = 512 dimensions would involve projections of size 512 × 512 and attention matrices computed from 512-dimensional vectors. Running h = 8 heads of dimension d_k = 64 each involves h projections of size 512 × 64. The total projection work is:

Single head: d_model × d_model = 512 × 512 = 262,144 Multi-head: h × d_model × d_k = 8 × 512 × 64 = 262,144

The total computation is exactly the same. Multi-head attention does not add computational cost compared to single-head attention of the same total dimension. But it gains the enormous benefit of multiple independent attention patterns. This is the key insight: splitting into subspaces is both cheaper per head and more expressive overall.

The Output Projection: W_O

After all heads produce their outputs, they are concatenated along the feature dimension and multiplied by a final projection matrix W_O of shape (h · d_v × d_model). Since h · d_v = d_model in the standard setup, W_O is a square matrix of shape (d_model × d_model).

W_O serves two purposes:

  • Mixing head outputs: It allows the model to learn how to combine information from different heads. Maybe for some positions, head 3’s syntactic information should dominate, while for others, head 7’s semantic information matters more. W_O learns these combinations.
  • Dimensionality restoration: It maps the concatenated output back to d_model, ensuring the multi-head attention layer’s output has the same shape as its input. This is essential for residual connections.

The Split-Attend-Concat Pipeline

The multi-head attention computation follows a clean four-step pipeline. Let us trace through each step in detail, using concrete shapes to make the data flow completely transparent.

We will use the standard Transformer base configuration: d_model = 512, h = 8, d_k = d_v = 64, and a sequence length of n = 10 tokens.

Step 1 - Project Q, K, V into h Separate Subspaces

Start with the input matrices Q, K, V, each of shape (10, 512). For each head i, compute:

Q_i = Q · W_Qi shape: (10, 512) · (512, 64) = (10, 64) K_i = K · W_Ki shape: (10, 512) · (512, 64) = (10, 64) V_i = V · W_Vi shape: (10, 512) · (512, 64) = (10, 64)

This projection is the “split” step. Each head carves out its own 64-dimensional slice of the representation space. The 512-dimensional input is not literally split along its dimensions (which would be arbitrary). Instead, each head learns its own linear projection that selects the combination of features most useful for that head’s attention pattern.

In practice, all h projections are computed simultaneously as a single large matrix multiplication, then reshaped. The code computes Q · W_Q_all where W_Q_all has shape (512, 512), then reshapes the (10, 512) result into (10, 8, 64) and transposes to (8, 10, 64). This is much faster on GPUs than running 8 separate matrix multiplications.

Step 2 - Run Scaled Dot-Product Attention in Each Subspace

Each head independently computes attention using the standard formula:

scores_i = Q_i · K_i^T / sqrt(64) shape: (10, 64) · (64, 10) / 8 = (10, 10) weights_i = softmax(scores_i, dim=-1) shape: (10, 10) head_i = weights_i · V_i shape: (10, 10) · (10, 64) = (10, 64)

Each head produces a (10, 10) attention matrix, meaning every token has a full distribution over all other tokens. Crucially, these distributions are different across heads. Head 1 might give token 7 high weight to token 2, while Head 5 might give token 7 high weight to token 9. The independence of the heads is what creates diversity in the attention patterns.

The scaling factor sqrt(64) = 8 is the same for all heads, since all heads use the same d_k. This keeps the attention scores in a well-behaved range regardless of which head is computing them.

Step 3 - Concatenate h Output Vectors Back to d_model

After all 8 heads compute their outputs, we concatenate them along the feature dimension:

concat = Concat(head_1, head_2, ..., head_8) Each head_i: (10, 64) Concatenated: (10, 8 × 64) = (10, 512)

The concatenation reassembles the full d_model-dimensional representation. Each position now has a 512-dimensional vector that contains 64 dimensions of information from each of the 8 heads. The ordering matters: head 1’s 64 values occupy positions 0–63, head 2’s occupy 64–127, and so on. The subsequent W_O projection can learn to mix across these boundaries.

Step 4 - Apply Final Linear Projection W_O

The concatenated output is multiplied by the output projection:

output = concat · W_O shape: (10, 512) · (512, 512) = (10, 512)

W_O is a (512, 512) learned matrix. It performs a final linear transformation that can mix information across heads. Without W_O, the model could only use each head’s output independently. With W_O, the model learns how to combine and weight the different heads’ contributions. Research has shown that W_O captures important cross-head interactions and removing it hurts performance significantly.

The final output has shape (10, 512), identical to the input shape. This is essential because the output feeds into a residual connection: output = LayerNorm(X + MultiHead(Q, K, V)). The shapes must match for the addition to work.

Dimension Walkthrough

Tracking tensor shapes through multi-head attention is one of the best ways to solidify your understanding. Here is the complete shape trace for the standard configuration (d_model=512, h=8, d_k=d_v=64, n=10):

Stage Shape Notes
Input X (n, d_model) = (10, 512) n tokens, each a 512-d embedding
W_Qi, W_Ki (d_model, d_k) = (512, 64) One per head, 8 total each
W_Vi (d_model, d_v) = (512, 64) One per head, 8 total
Q_i, K_i, V_i (n, d_k) = (10, 64) Per head, after projection
Q_i · K_i^T (n, n) = (10, 10) Raw attention scores per head
softmax(scores_i / sqrt(d_k)) (n, n) = (10, 10) Attention weights per head
head_i = weights_i · V_i (n, d_v) = (10, 64) Weighted output per head
Concat(head_1, ..., head_8) (n, h·d_v) = (10, 512) All heads merged along feature dim
W_O (h·d_v, d_model) = (512, 512) Output projection matrix
Final output (n, d_model) = (10, 512) Same shape as input - ready for residual

Batched / GPU Shape

In practice, we add a batch dimension and reshape for efficient parallel computation across heads:

Stage Shape Notes
Input (B, n, d_model) Batch of B sequences
After projection (B, n, h, d_k) All heads computed at once
Transposed for attention (B, h, n, d_k) Heads become a batch dim
Attention scores (B, h, n, n) h separate n×n attention maps
Head outputs (B, h, n, d_v) h separate weighted sums
Transposed back (B, n, h, d_v) Ready for concatenation
Reshaped (concat) (B, n, h·d_v) Heads merged by reshape
After W_O (B, n, d_model) Final output

The key trick is that the head dimension h is treated as an extra batch dimension during the attention computation. The GPU runs all h attention operations in a single batched matrix multiply, which is enormously faster than running them sequentially.

Head Specialization

One of the most fascinating discoveries from analyzing trained Transformers is that different attention heads learn remarkably distinct and interpretable patterns. This specialization emerges naturally from training - no head is explicitly told what to focus on. The combination of independent projections and gradient descent leads each head to carve out its own niche.

Research by Clark et al. (2019), Voita et al. (2019), and others has identified several recurring head types. Here are the most commonly observed patterns:

🔵 Positional Heads (Local Attention)

These heads attend primarily to tokens in fixed relative positions: the immediately preceding token, the next token, or a token at a fixed offset. They act as learned n-gram detectors.

Example: A “previous-token” head consistently places most weight on position i−1 when computing the output for position i. Its attention matrix is nearly a shifted identity matrix.

Why it’s useful: Local context is critical for capturing bigram and trigram patterns, word order, and phrase boundaries. Without positional heads, the model would need to rely entirely on positional encodings for local structure.

🟠 Syntactic Heads (Dependency Tracking)

These heads learn to attend along syntactic dependency arcs. A verb token attends to its subject, an adjective attends to the noun it modifies, and a preposition attends to its object. The attention pattern mirrors the dependency parse tree of the sentence.

Example: In “The large dog chased the small cat,” the head at “chased” attends strongly to “dog” (subject) and “cat” (object), with minimal attention to “the”, “large”, and “small”.

Why it’s useful: Syntactic structure determines meaning. “Dog bites man” and “Man bites dog” have the same words but opposite meanings, determined entirely by subject-verb-object relationships.

🟣 Coreference Heads (Pronoun Resolution)

These heads specialize in connecting pronouns and other anaphoric expressions to their referents. When processing a pronoun like “it”, “they”, or “which”, the head attends strongly to the noun or noun phrase that the pronoun refers to.

Example: In “The cat sat on the mat because it was tired,” a coreference head at “it” places dominant weight on “cat.” This happens even though “mat” is closer and also a plausible referent syntactically.

Why it’s useful: Coreference is essential for understanding text beyond a single clause. Documents are full of pronouns, definite descriptions, and other referring expressions that require linking to earlier mentions.

🔴 Rare-Word Heads (Vocabulary Attention)

Some heads learn to attend preferentially to infrequent or informationally rich tokens. These tokens carry high surprise value and often contain the most important content in a sentence. Common function words (the, is, of) receive minimal attention.

Example: In “The mitochondria is the powerhouse of the cell,” a rare-word head broadly attends to “mitochondria” and “powerhouse” while largely ignoring “the”, “is”, “of.”

Why it’s useful: Rare words are often the most semantically meaningful tokens. By having heads that specialize in extracting information from them, the model can build better content representations.

🟢 Separator / Delimiter Heads

These heads attend heavily to sentence boundaries, punctuation, and special tokens like [SEP] or [CLS]. They help the model track document structure and segment boundaries.

Example: In a two-sentence input for BERT, several heads consistently attend to the [SEP] token regardless of position. This provides a “reset” signal that helps the model understand segment boundaries.

Why it’s useful: Structural boundaries carry important information. Knowing where one sentence ends and another begins is crucial for tasks like natural language inference and question answering.

Not All Heads Are Equal

An important finding from Voita et al. (2019) is that many heads are redundant. In their experiments with a trained 6-layer, 8-head Transformer, pruning the least important heads (removing 60-80% of heads across the model) caused only marginal drops in BLEU score for machine translation. This suggests that while multi-head attention provides the capacity for diverse patterns, not all heads find equally useful specializations. Some heads may end up learning nearly identical patterns or patterns of minimal value.

This observation has led to approaches like adaptive head pruning, where the model learns to disable unnecessary heads during training, and multi-query attention, where keys and values are shared across heads to reduce parameters while preserving most of the benefit.

Cross-Attention vs Self-Attention

Multi-head attention is used in two fundamentally different configurations within the Transformer architecture. Understanding the distinction is essential for grasping how encoder-decoder models work.

Self-Attention

In self-attention, the queries, keys, and values all come from the same sequence. When we write MultiHead(X, X, X), it means Q = X, K = X, V = X (before projection). Every token attends to every other token in the same sequence, including itself.

Self-attention is used in:

  • The encoder (each encoder layer applies self-attention over the input sequence)
  • The first attention sub-layer of each decoder layer (the decoder attends to its own previously generated tokens)
  • Encoder-only models like BERT (self-attention is the only attention type)
  • Decoder-only models like GPT (masked self-attention is the only attention type)

Cross-Attention

In cross-attention, the queries come from one sequence and the keys and values come from a different sequence. Specifically, in a Transformer encoder-decoder:

Q = decoder hidden states (from the decoder) K = encoder output (from the encoder) V = encoder output (from the encoder) MultiHead(Q_decoder, K_encoder, V_encoder)

The decoder “asks questions” (queries) about the encoder’s representation of the input, and the encoder “answers” through its keys and values. This is how the decoder knows what the input sequence contains when generating each output token.

Self-Attention

Source: Q, K, V from same sequence

Score matrix: (n, n) - tokens attend within themselves

Use case: Building contextual representations of a single sequence

Example: Understanding “The cat sat on the mat” by letting each word look at all other words

Cross-Attention

Source: Q from decoder, K and V from encoder

Score matrix: (m, n) - m decoder tokens attend to n encoder tokens

Use case: Connecting decoder output generation to encoder input representation

Example: Translating “Le chat” by letting each French token query the English source

The Shapes Differ

In self-attention, the attention score matrix is always square: (n, n). In cross-attention, it is rectangular: (m, n), where m is the decoder sequence length and n is the encoder sequence length. This is because decoder tokens (m of them) query against encoder tokens (n of them), and m does not need to equal n. In machine translation, the source sentence might have 15 words while the target has 20.

Masked Multi-Head Attention

In autoregressive generation (the decoder side of the Transformer, or in GPT-style models), each token should only attend to tokens that came before it in the sequence. It must not “peek” at future tokens, because at inference time those tokens have not been generated yet.

Masked multi-head attention enforces this constraint by applying a causal mask to the attention scores before the softmax, independently in each head:

scores_i = Q_i · K_i^T / sqrt(d_k) mask = upper_triangular_matrix_of(-∞) masked_scores_i = scores_i + mask weights_i = softmax(masked_scores_i)

The mask is an upper-triangular matrix filled with negative infinity (−∞). When added to the scores, any position where token i would attend to a future token j > i gets a score of −∞. After softmax, exp(−∞) = 0, so those positions receive zero attention weight. The token is effectively invisible.

Visualizing the Causal Mask

The mask is applied identically and independently in every head. Each head computes its own scores, adds the same causal mask, then runs softmax. The masking does not change across heads - the constraint is structural (no future information) rather than content-dependent.

Implementation Detail

In practice, the mask is precomputed as a boolean tensor and reused across all layers and all heads. The −∞ values are typically represented as a very large negative number (e.g., -1e9 or float('-inf')) to avoid numerical issues. Some frameworks use masked_fill to set positions to −∞ only where the mask is True:

mask = torch.triu(torch.ones(n, n), diagonal=1).bool() scores = scores.masked_fill(mask, float('-inf')) weights = softmax(scores, dim=-1)

Parameter Count

Understanding the parameter count of multi-head attention is important for model sizing, memory estimation, and comparing architectures. Let us derive it from first principles.

QKV Projection Parameters

Each head has three projection matrices: W_Qi, W_Ki, W_Vi. Each maps from d_model to d_k (or d_v). Across all heads:

QKV params = h × 3 × (d_model × d_k) = h × 3 × d_model × (d_model / h) = 3 × d_model × d_model = 3 × d_model²

Notice that the number of heads cancels out. Whether you use 1 head or 100 heads, the total QKV projection parameters are always 3 × d_model². This is because increasing h decreases d_k proportionally.

Output Projection Parameters

W_O params = (h × d_v) × d_model = d_model × d_model = d_model²

Total Parameters (Without Biases)

Total = 3 × d_model² + d_model² = 4 × d_model²

Concrete Example: Transformer Base

Component Formula Value (d_model=512, h=8)
W_Q (all heads) d_model × d_model 512 × 512 = 262,144
W_K (all heads) d_model × d_model 512 × 512 = 262,144
W_V (all heads) d_model × d_model 512 × 512 = 262,144
W_O d_model × d_model 512 × 512 = 262,144
Total 4 × d_model² 1,048,576 ≈ 1.05M

With Biases

If bias terms are included (some implementations include them, some do not):

QKV biases: 3 × d_model = 3 × 512 = 1,536 W_O bias: d_model = 512 Total with biases: 4 × d_model² + 4 × d_model = 1,048,576 + 2,048 = 1,050,624

The biases add a negligible 0.2% to the parameter count. This is why they are often ignored in rough calculations.

Scaling to Larger Models

Model d_model h MHA Params (per layer)
Transformer Base 512 8 ~1.05M
Transformer Big 1024 16 ~4.19M
GPT-2 1600 25 ~10.24M
GPT-3 (175B) 12288 96 ~603.98M

Parameter count grows quadratically with d_model. Doubling d_model quadruples the multi-head attention parameters per layer. This is one reason why large language models require enormous amounts of memory.

Implementation

Here is a clean, annotated pseudo-code implementation of multi-head attention. This follows the standard approach used in PyTorch and similar frameworks.

class MultiHeadAttention: def __init__(self, d_model, h): self.h = h self.d_k = d_model // h # Combined QKV projections for efficiency self.W_Q = Linear(d_model, d_model) # (d_model, d_model) self.W_K = Linear(d_model, d_model) # (d_model, d_model) self.W_V = Linear(d_model, d_model) # (d_model, d_model) self.W_O = Linear(d_model, d_model) # output projection def forward(self, Q, K, V, mask=None): B, n, _ = Q.shape # Step 1: Project and reshape for multi-head Q = self.W_Q(Q).view(B, n, self.h, self.d_k).transpose(1, 2) K = self.W_K(K).view(B, n, self.h, self.d_k).transpose(1, 2) V = self.W_V(V).view(B, n, self.h, self.d_k).transpose(1, 2) # Q, K, V shape: (B, h, n, d_k) # Step 2: Scaled dot-product attention per head scores = Q @ K.transpose(-2, -1) / math.sqrt(self.d_k) # scores shape: (B, h, n, n) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) weights = softmax(scores, dim=-1) # (B, h, n, n) attn = weights @ V # (B, h, n, d_k) # Step 3: Concatenate heads attn = attn.transpose(1, 2).contiguous().view(B, n, -1) # attn shape: (B, n, h * d_k) = (B, n, d_model) # Step 4: Output projection output = self.W_O(attn) # (B, n, d_model) return output, weights

Key Implementation Details

  • Combined projections: Instead of h separate small matrices, we use one large (d_model, d_model) matrix per QKV. The “splitting” into heads happens via reshape and transpose, which are free operations (no computation, just memory reinterpretation).
  • .contiguous(): After the transpose in step 3, the tensor’s memory layout may not be contiguous. The .contiguous() call ensures the subsequent .view() works correctly. This is a PyTorch-specific detail.
  • Mask broadcasting: The mask has shape (1, 1, n, n) and broadcasts across batch and head dimensions. This means the same mask is applied to all heads in all batch items.
  • Returning weights: We return the attention weights alongside the output for visualization and debugging. In production, you might drop this to save memory.

Efficient Batched Computation

The reshape-and-transpose trick is the key to efficiency. By moving the head dimension into the batch dimension, we turn h separate attention computations into a single batched matrix multiplication. The GPU sees (B × h) independent (n, d_k) matrix multiplies and parallelizes them across its cores. This is orders of magnitude faster than a Python loop over heads.

Single-Head vs Multi-Head Comparison

Single-Head Attention

Capacity: One attention pattern per layer per position. Must compromise between different types of relationships.

Focus: Can capture only one notion of “relevance.” Syntactic patterns interfere with semantic ones in the same weight matrix.

Subspaces: Operates in full d_model space. Similarity is measured in one geometry.

Computation: One QKV projection of size (d_model, d_model). One n×n attention matrix.

Parameters: 4 × d_model² (same as multi-head!).

Expressiveness: Limited. The single attention distribution must average over all desired patterns.

Pruning: Nothing to prune. All-or-nothing.

Multi-Head Attention (h heads)

Capacity: h independent attention patterns per layer per position. Each head can specialize.

Focus: Different heads learn different notions of relevance. Syntax, semantics, position, and discourse are separated.

Subspaces: Operates in h distinct d_k-dimensional subspaces. Multiple geometries for similarity.

Computation: h QKV projections of size (d_model, d_k). h separate n×n attention matrices. Same total FLOPs.

Parameters: 4 × d_model² (same as single-head!).

Expressiveness: Rich. Each head is a pure attention signal. W_O mixes them optimally.

Pruning: Redundant heads can be removed with minimal loss. Adaptive head counts are possible.

The critical insight from this comparison is that multi-head attention achieves strictly greater expressiveness for the same parameter count and computational cost. It is a free lunch in terms of capacity, which is why every modern Transformer uses it. The only reason to prefer a single large head would be if all the useful information genuinely required a single high-dimensional similarity metric - which, empirically, is never the case for natural language.

Analogy: Ensemble of Specialists

Think of multi-head attention as an ensemble of specialist analysts. A single-head model is like asking one analyst to evaluate a company: they must simultaneously consider financial health, market position, team quality, and regulatory risk, producing one blended score. Multi-head attention is like having four specialist analysts - one for each dimension - who each produce a focused assessment. Their reports are then combined by a manager (W_O) who weighs each specialist’s input based on what matters for the current decision. The ensemble almost always outperforms the single generalist, because specialization allows deeper expertise in each area.

Summary

Multi-head attention is one of the key innovations that makes Transformers so powerful. Here are the essential takeaways:

  1. Single heads are bottlenecked. One attention head produces one attention pattern per position, forcing the model to compromise between different types of linguistic relationships. This leads to blurred, suboptimal representations.
  2. Multiple heads run in parallel subspaces. Each head has its own projection matrices (W_Qi, W_Ki, W_Vi) that project inputs into a private d_k-dimensional space. Attention is computed independently in each subspace, allowing diverse patterns to coexist.
  3. Same cost, more expressiveness. Because d_k = d_model / h, the total computation and parameter count of multi-head attention equals that of a single head with the full d_model. Multi-head is strictly more expressive at no additional cost.
  4. The pipeline is Split → Attend → Concat → Project. Project Q, K, V into h subspaces, run scaled dot-product attention in each, concatenate the h outputs, and apply the output projection W_O to mix cross-head information.
  5. Heads specialize naturally. Without explicit supervision, heads learn distinct roles: positional patterns, syntactic dependencies, coreference resolution, rare-word attention, and more. Not all heads are equally important - many can be pruned with minimal performance loss.
  6. Cross-attention and masking are straightforward extensions. Cross-attention uses Q from one sequence and K, V from another. Masked attention applies a causal mask (negative infinity for future positions) before softmax, independently in each head.
  7. Parameters scale as 4 × d_model² per layer. The QKV projections contribute 3 × d_model² and the output projection adds d_model². The number of heads does not affect the total parameter count.
