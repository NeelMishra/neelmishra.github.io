Paged Attention (vLLM)

ML Series · Transformers · Variants

1 · The Fragmentation Problem

During LLM serving, each request generates tokens one at a time, growing its KV cache incrementally. Traditional serving systems pre-allocate a contiguous block of GPU memory for each request's maximum possible length:

Traditional KV cache allocation: Request arrives → allocate max_seq_len × n_layers × n_heads × d_head × 2 LLaMA-2 13B, max 2048 tokens: 2048 × 40 × 40 × 128 × 2 × 2 = 1.6 GB per request Problems: 1. Internal fragmentation: request only uses 100 tokens but 2048 allocated 2. External fragmentation: gaps between freed/allocated blocks 3. Reservation waste: 60-80% of allocated memory is unused on average

In practice, traditional systems waste 60–80% of GPU memory on fragmentation and over-allocation. This directly limits batch size and throughput.

2 · The OS Analogy

Operating systems solved the same problem for CPU memory decades ago with virtual memory paging. Instead of allocating contiguous physical memory, the OS:

  • Divides memory into fixed-size pages (e.g., 4 KB)
  • Maps virtual addresses to physical pages via a page table
  • Allocates pages on demand (no pre-allocation)
  • Pages can be non-contiguous in physical memory
Paged Attention applies this exact idea to KV cache: divide the KV cache into fixed-size blocks, track them with a block table, and allocate blocks on demand as the sequence grows. No more contiguous allocation, no more fragmentation.

3 · KV Cache Blocks

Instead of one contiguous tensor per sequence, the KV cache is divided into blocks, each storing K and V for a fixed number of tokens (the block size, typically 16):

Block size: B = 16 tokens Block contents: K[0:B], V[0:B] for all heads and layers Block memory: B × n_heads × d_head × 2 × 2 bytes For LLaMA-2 13B (40 layers, 40 heads, d_head=128): Block memory = 16 × 40 × 128 × 2 × 40 × 2 = 52 MB per block Sequence of 1000 tokens → ceil(1000/16) = 63 blocks Only 63 × 52 MB ≈ 3.3 GB allocated (vs 6.4 GB for 2048-token pre-allocation)

Blocks are stored in a block pool — a pre-allocated area of GPU memory divided into fixed-size slots. Free blocks are tracked in a free list, just like OS page frames.

4 · The Block Table

Each sequence has a block table — a mapping from logical block indices to physical block IDs:

Sequence "The cat sat on the mat" (6 tokens, block_size=4): Logical block 0: tokens 0-3 ("The cat sat on") → Physical block 7 Logical block 1: tokens 4-5 ("the mat") → Physical block 13 Block table: [7, 13] When a new token is generated: - If current block has space → append to physical block 13 - If full → allocate new physical block from free list, extend table to [7, 13, 22] Physical blocks 7 and 13 can be ANYWHERE in GPU memory — no contiguity required!

5 · Paged Attention Algorithm

The attention kernel is modified to fetch K/V from non-contiguous blocks using the block table:

def paged_attention(query, block_table, kv_cache_pool, block_size): # query: (n_heads, d_head) for the new token # block_table: list of physical block IDs for this sequence # kv_cache_pool: (num_blocks, 2, block_size, n_heads, d_head) output = zeros(n_heads, d_head) for head in range(n_heads): max_score = -inf sum_exp = 0 acc = zeros(d_head) for logical_idx, phys_block in enumerate(block_table): # Fetch K, V from physical block (non-contiguous!) K_block = kv_cache_pool[phys_block, 0] # (block_size, d_head) V_block = kv_cache_pool[phys_block, 1] # (block_size, d_head) # Standard attention with online softmax scores = query[head] @ K_block.T / sqrt(d_head) block_max = max(scores) new_max = max(max_score, block_max) # Rescale and accumulate scale = exp(max_score - new_max) sum_exp = scale * sum_exp + sum(exp(scores - new_max)) acc = scale * acc + exp(scores - new_max) @ V_block max_score = new_max output[head] = acc / sum_exp return output

7 · Memory Sharing

Paged Attention enables copy-on-write sharing between sequences with common prefixes:

When generating N completions from the same prompt, all N sequences share the prompt's KV blocks. Only new tokens (where outputs diverge) allocate fresh blocks. For a 1000-token prompt with 4 completions, this saves 3× the prompt's KV memory.

Beam search candidates share most of their KV cache (they diverge only at the last few tokens). Without sharing, each beam maintains a full copy — 4 beams = 4× memory. With paged sharing, overhead is near zero.

System prompts shared across requests can be cached once and referenced by all requests via shared block table entries. A 2000-token system prompt cached once saves ~100 MB per concurrent request for LLaMA-13B.

8 · Request Scheduling

vLLM uses an iteration-level scheduler that dynamically manages which requests run at each step:

  • Preemption: If GPU memory is exhausted, the scheduler can preempt (pause) a running request by either swapping its blocks to CPU memory or recomputing them later.
  • Dynamic batching: New requests join the batch as soon as there's memory for their first block, without waiting for running requests to finish.
  • Priority queues: Requests can be prioritized based on latency requirements, sequence length, or business logic.
Scheduling decisions per iteration: 1. Check free blocks in pool 2. For each waiting request: can we allocate ≥1 block? → admit 3. For each running request: enough blocks for next token? → continue 4. If pool exhausted: preempt lowest-priority request (swap blocks to CPU) 5. Run one forward pass for entire batch 6. Allocate new blocks for requests that filled their current block

9 · Implementation

Key data structures: BlockPool: physical_blocks: Tensor[num_blocks, 2, block_size, n_heads, d_head] free_list: List[int] — available physical block IDs BlockTable (per sequence): table: List[int] — logical → physical mapping num_filled: int — tokens in the last block PagedAttentionKernel: - Custom CUDA kernel that reads K/V from non-contiguous blocks - Uses block_table to compute physical addresses: phys_addr = block_table[token_pos // block_size] offset_in_block = token_pos % block_size - Compatible with Flash Attention's tiling (flash-attn with paged KV)

Why 2–4× throughput? It's not that Paged Attention computes faster — it computes the same attention. The gain is from higher batch sizes: near-zero memory waste → more concurrent requests → more tokens processed per second.

11 · Ecosystem

  • vLLM — the original Paged Attention serving engine. Open-source, supports most popular models.
  • SGLang — RadixAttention extends paging with a radix tree for automatic prefix caching across requests.
  • TensorRT-LLM (NVIDIA) — adopted paged KV cache for production inference.
  • llama.cpp — implements a simplified version of paged KV cache for CPU inference.
  • Flash Attention — FlashInfer and Flash-Decoding both support paged KV cache layouts for GPU-efficient attention kernels.
  • DeepSpeed-FastGen (Microsoft) — uses "SplitFuse" technique inspired by paged attention for dynamic batching.

12 · Summary

  • Paged Attention applies OS virtual memory paging to KV cache: fixed-size blocks, block tables, on-demand allocation.
  • Eliminates internal and external fragmentation, reducing memory waste from 60–80% to <4%.
  • Enables copy-on-write sharing for parallel sampling, beam search, and prefix caching.
  • Dynamic scheduling with preemption maximizes GPU utilization across variable-length requests.
  • Result: 2–4× serving throughput improvement, now standard in production LLM serving.
One-line summary: Paged Attention manages KV cache like an OS manages memory — fixed-size pages, block tables, on-demand allocation — eliminating fragmentation and doubling+ serving throughput.
