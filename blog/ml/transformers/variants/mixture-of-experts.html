Mixture of Experts (MoE)

ML Series · Transformers · Variants

1 · The Scaling Dilemma

Scaling laws tell us that larger models are better. But "larger" traditionally means more parameters and more compute per token. A 70B model processes every token through 70 billion parameters — expensive for training and inference.

The question: can we have the capacity of a large model with the compute cost of a small one?

The MoE answer: Yes. Give the model many parameters but only activate a sparse subset for each token. A 47B-parameter MoE model might only use 13B parameters per token — getting 47B-level quality at 13B-level cost.

2 · Core Idea: Sparse Experts

In a standard transformer, every token passes through the same FFN (Feed-Forward Network) in each layer. In an MoE transformer, the FFN is replaced by N expert networks plus a gating/router network that selects which experts to activate:

Standard Transformer layer: Attention → FFN → Add & Norm MoE Transformer layer: Attention → Router → Top-K Experts → Add & Norm Key insight: only K out of N experts run per token (K ≪ N) Mixtral 8×7B: N=8 experts, K=2 active → 2/8 = 25% of expert params per token

Each expert is a standard FFN (two linear layers with activation). The router is a small linear layer that produces a probability distribution over experts for each token.

3 · Architecture

An MoE layer replaces the FFN in select transformer layers (often every other layer). The structure:

Self-attention is shared — all tokens go through the same attention mechanism. Layer norm and residual connections are also shared. Only the FFN is replaced by experts.

N identical-architecture FFNs, each with its own weights. For Mixtral: each expert is a standard SwiGLU FFN with hidden dim 14336, giving 7B params per expert × 8 experts = 56B expert params (but only 2 activated per token).

A linear layer W_gate ∈ ℝ^(d_model × N) that maps each token to N logits, then softmax → top-K selection. The router is tiny (~0.01% of total params) but critically important.

4 · The Gating Network

Router computation for token x: logits = x · W_gate (d_model → N experts) probs = softmax(logits) (N probabilities) top_k = TopK(probs, K) (select K experts) gates = probs[top_k] / sum(probs[top_k]) (renormalize) Output: y = Σᵢ∈top_k gates[i] · Expert_i(x) (weighted sum of K expert outputs)

The gates serve dual purpose: they select which experts to run and weight their outputs. A token might route 60% to expert 3 and 40% to expert 7, getting a blend of their outputs.

Top-1 vs Top-2 routing: Switch Transformer uses K=1 (simplest, fastest). Mixtral and most modern MoEs use K=2 (better quality, slightly more compute). DeepSeek-V2 uses K=6 out of 160 experts with a shared expert alongside.

5 · The Math

Total parameters: P_total = P_attention + P_router + N × P_expert + P_other Active parameters per token: P_active = P_attention + P_router + K × P_expert + P_other Compute savings: FLOPs ratio = P_active / P_total ≈ K/N (for expert-dominated models) Example (Mixtral 8×7B): Total params: ~47B (8 experts × 7B each + shared attention + router) Active params: ~13B (2 experts × 7B + shared) FLOPs ≈ 13B-model level, quality ≈ 47B-model level
Memory requirement: All N experts must be loaded (even though only K are active) Mixtral 8×7B: needs ~47B params in memory (≈94 GB in FP16) But compute is only 13B level This is the MoE tradeoff: compute-efficient but memory-hungry

7 · Load Balancing

Without intervention, routing collapses: a few "popular" experts receive most tokens while others are starved. This wastes capacity and creates GPU load imbalance. Solutions:

Auxiliary load-balancing loss: f_i = fraction of tokens routed to expert i (across batch) p_i = average router probability for expert i (across batch) L_balance = α · N · Σᵢ f_i · p_i Ideal: f_i = p_i = 1/N for all experts (uniform distribution) α is typically 0.01–0.1
  • Auxiliary loss (above): penalizes uneven routing. Used by Switch Transformer, GShard, Mixtral.
  • Expert capacity: cap the number of tokens an expert can process per batch. Overflow tokens are dropped or sent to a shared "default" expert.
  • Noise injection: add Gaussian noise to router logits during training to encourage exploration.
  • Expert choice routing (Zhou et al., 2022): instead of tokens choosing experts, experts choose their top-K tokens. Guarantees perfect load balance.

8 · Expert Parallelism

MoE models need special parallelism strategies because experts on different GPUs receive different tokens:

Data parallelism: replicate all experts on all GPUs (memory-wasteful) Expert parallelism: distribute experts across GPUs (each GPU holds N/P experts) With expert parallelism (EP): 1. Each GPU computes attention (shared) for its tokens 2. All-to-all: tokens dispatched to the GPU holding their assigned experts 3. Each GPU runs its local experts on received tokens 4. All-to-all: outputs sent back to the originating GPU 5. Weighted combination of expert outputs
All-to-all cost: The two all-to-all communications are the main overhead of expert parallelism. For large clusters, this can be 10–20% of total time. Strategies like limiting the number of experts per device and using topology-aware routing help minimize this.

9 · Implementation

class MoELayer: def __init__(self, d_model, n_experts, top_k, expert_dim): self.experts = [FFN(d_model, expert_dim) for _ in range(n_experts)] self.gate = Linear(d_model, n_experts) self.top_k = top_k self.n_experts = n_experts def forward(self, x): B, S, D = x.shape x_flat = x.view(B * S, D) # Router: compute gating scores logits = self.gate(x_flat) # (B*S, N) probs = softmax(logits, dim=-1) top_k_probs, top_k_idx = topk(probs, self.top_k) top_k_probs = top_k_probs / top_k_probs.sum(dim=-1, keepdim=True) # Dispatch tokens to experts output = zeros_like(x_flat) for i in range(self.n_experts): mask = (top_k_idx == i).any(dim=-1) # which tokens go to expert i if mask.sum() == 0: continue expert_input = x_flat[mask] expert_output = self.experts[i](expert_input) # Weight by gating probabilities weight = top_k_probs[mask][top_k_idx[mask] == i] output[mask] += weight.unsqueeze(-1) * expert_output # Load-balancing auxiliary loss f = mask_counts / (B * S) # fraction per expert p = probs.mean(dim=0) # mean prob per expert self.aux_loss = self.n_experts * (f * p).sum() return output.view(B, S, D)

11 · Challenges

  • Memory: All N experts must reside in memory even if only K are active. An 8×7B MoE needs ~47B param memory, not 13B.
  • Training instability: Router gradients are noisy (discrete top-K selection). Experts can "die" (never receive tokens) or collapse (all tokens go to few experts).
  • Load imbalance: Despite auxiliary losses, some experts are consistently more popular, creating GPU idle time in expert-parallel setups.
  • Batch size sensitivity: Small batches make routing statistics noisy. MoE models benefit from large batch training.
  • Serving complexity: Expert parallelism requires all-to-all communication, limiting serving on small clusters.

12 · Summary

  • MoE replaces dense FFN with N expert FFNs + a router that activates only K per token.
  • This decouples model capacity (total params) from per-token compute (active params).
  • A well-designed MoE matches a dense model 3–4× its active size (e.g., Mixtral 13B-active ≈ LLaMA-2 70B).
  • Load balancing (auxiliary loss, expert capacity) is critical for training stability.
  • Expert parallelism enables training/serving but adds all-to-all communication overhead.
  • MoE is now mainstream: Mixtral, DeepSeek, Grok, DBRX all use it.
One-line summary: MoE scales model capacity by having many expert networks but only activating a few per token — big model quality at small model cost.
