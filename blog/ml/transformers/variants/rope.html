Rotary Position Embeddings (RoPE)

ML Series · Transformers · Variants

1 · The Position Problem

Self-attention is permutation-invariant. If you shuffle the tokens in a sequence, the attention scores for each pair remain the same — the mechanism has no notion of order. But language depends critically on word order: "the cat sat on the mat" and "mat the on sat cat the" contain the same tokens with completely different meanings.

Early transformers solved this by adding position embeddings to the token embeddings before attention. Two approaches dominated:

Fixed sine/cosine functions at different frequencies. No learned parameters, but the position signal gets diluted as it mixes with content through layers. Limited ability to generalize beyond the training context length.

A lookup table of trainable vectors, one per position. Expressive, but hard-coded to a maximum sequence length. To go from 2048 to 4096 tokens, you need to retrain or interpolate.

Both approaches encode absolute positions. But what matters for attention is usually relative position: how far apart two tokens are. A verb at position 5 attending to its subject at position 3 should behave the same as a verb at position 105 attending to a subject at position 103. The distance is 2 in both cases.

The ideal position encoding would: (1) inject absolute position information so each token "knows" where it is, (2) make the attention score between two tokens depend only on their relative distance, and (3) generalize to sequence lengths not seen during training. RoPE achieves all three.

2 · The Core Idea: Encode Position in Rotation

RoPE's insight is beautifully simple: instead of adding a position vector to each token embedding, rotate the query and key vectors by an angle proportional to their position. When you compute the dot product between a rotated query and a rotated key, the rotation angles partially cancel, and the result depends only on the difference in positions.

Think of it this way: if you stand at position m and rotate by angle mθ, and your friend stands at position n and rotates by angle nθ, the angle between you is (m-n)θ — which depends only on your relative distance.

Key intuition: RoPE encodes absolute position by applying a rotation, but the dot product between two rotated vectors depends only on the relative position. Absolute encoding in, relative information out.

3 · Rotation in 2D

Let's start with the simplest case: a 2-dimensional vector. To rotate a vector (x₁, x₂) by angle θ, we multiply by the standard rotation matrix:

R(θ) = | cos θ -sin θ | | sin θ cos θ | Rotated vector: R(θ) · [x₁, x₂]ᵀ = [x₁ cos θ - x₂ sin θ, x₁ sin θ + x₂ cos θ]

This rotation preserves the vector's magnitude (the length doesn't change) and shifts its direction by angle θ. RoPE sets θ = m · θbase, where m is the token's position and θbase is a frequency parameter.

For two vectors at positions m and n:

q̃ₘ = R(mθ) · qₘ (rotate query at position m) k̃ₙ = R(nθ) · kₙ (rotate key at position n) Dot product: q̃ₘᵀ · k̃ₙ = qₘᵀ · R(mθ)ᵀ · R(nθ) · kₙ = qₘᵀ · R((n - m)θ) · kₙ The dot product depends on (n - m), the relative distance!

This works because rotation matrices have the property R(a)ᵀ · R(b) = R(b - a). The individual absolute positions cancel out, leaving only the relative distance.

4 · Extending to Higher Dimensions

Real transformer heads use 64 or 128 dimensions, not 2. RoPE extends the 2D rotation by pairing up consecutive dimensions and rotating each pair at a different frequency:

For a d-dimensional vector, pair dimensions: (0,1), (2,3), (4,5), ..., (d-2, d-1) Each pair i gets its own frequency: θᵢ = 10000^(-2i/d) where i = 0, 1, ..., d/2 - 1 The full rotation matrix is block-diagonal: R(m) = diag(R₂(mθ₀), R₂(mθ₁), ..., R₂(mθ_{d/2-1})) where R₂(α) is the 2×2 rotation matrix for angle α.

Low-frequency pairs (small i) rotate slowly as position changes — they capture long-range patterns. High-frequency pairs (large i) rotate quickly — they capture fine-grained, local position information. This is analogous to the multi-scale frequency design in sinusoidal position embeddings.

With d = 64 and base 10000:

  • θ₀ = 10000⁰ = 1.0 (fast rotation, ~6.28 cycle every 2π positions)
  • θ₁₅ = 10000^(-30/64) ≈ 0.0056 (one cycle every ~1120 positions)
  • θ₃₁ = 10000^(-62/64) ≈ 0.000105 (one cycle every ~59,900 positions)

This gives the model sensitivity to both nearby and distant tokens.

5 · The RoPE Formula

For a query vector qₘ at position m with head dimension d:

q̃ₘ = RoPE(qₘ, m) For each dimension pair (2i, 2i+1): q̃ₘ[2i] = qₘ[2i] · cos(mθᵢ) - qₘ[2i+1] · sin(mθᵢ) q̃ₘ[2i+1] = qₘ[2i] · sin(mθᵢ) + qₘ[2i+1] · cos(mθᵢ) where θᵢ = 10000^(-2i/d) for i = 0, 1, ..., d/2-1

The same rotation is applied to key vectors. Value vectors are not rotated — only Q and K get position information, because position should influence which tokens to attend to (via Q·K), not what information to extract (via V).

Important: RoPE is applied after the linear Q and K projections, not to the input embeddings. This means the position encoding is injected directly into the attention computation, not mixed into the residual stream.

7 · The Relative Position Property

The mathematical proof is straightforward. For query at position m and key at position n, both in 2D:

q̃ₘᵀ · k̃ₙ = (R(mθ) qₘ)ᵀ (R(nθ) kₙ) = qₘᵀ R(mθ)ᵀ R(nθ) kₙ = qₘᵀ R(-mθ) R(nθ) kₙ (Rᵀ = R⁻¹ = R(-θ) for rotation) = qₘᵀ R((n-m)θ) kₙ (rotation composition)

The dot product depends on (n - m), not on m or n individually. This extends to the block-diagonal case because each 2D block satisfies this property independently, and the full dot product is the sum of per-block dot products.

A model trained on sequences of length 2048 learns attention patterns based on relative distances. When deployed on a sequence of length 4096, the relative distances are the same kinds of numbers (1, 2, 10, 100, etc.) — just occurring at higher absolute positions. Because the attention scores depend on relative distance, the learned patterns transfer naturally.

8 · Implementation

In practice, nobody constructs the full rotation matrix. Instead, the rotation is applied element-wise using cos/sin operations. The trick is to precompute cos(mθᵢ) and sin(mθᵢ) for all positions m and frequency indices i:

# Precompute frequency table (done once) def precompute_freqs(dim, max_seq_len, base=10000): i = arange(0, dim, 2) # [0, 2, 4, ..., dim-2] freqs = 1.0 / (base ** (i / dim)) # θᵢ values positions = arange(max_seq_len) # [0, 1, 2, ..., L-1] angles = outer(positions, freqs) # (L, dim/2): m * θᵢ return cos(angles), sin(angles) # each (L, dim/2) # Apply RoPE to a query or key vector def apply_rope(x, cos_table, sin_table): # x shape: (batch, seq_len, num_heads, head_dim) x_even = x[..., 0::2] # even-indexed dims x_odd = x[..., 1::2] # odd-indexed dims # Rotate each pair out_even = x_even * cos_table - x_odd * sin_table out_odd = x_even * sin_table + x_odd * cos_table # Interleave back return stack([out_even, out_odd], dim=-1).flatten(-2)

The total cost is 4 · d multiplications and 2 · d additions per token, per head — negligible compared to the Q, K, V linear projections.

9 · Context Length Extrapolation

One of RoPE's most practical benefits is that models can sometimes handle sequences longer than those seen during training. However, raw extrapolation degrades because the model has never seen the high-frequency rotation angles corresponding to very large positions. Several techniques address this:

Instead of using position m directly, scale it down: m' = m · (L_train / L_target). This maps all positions into the range the model has seen. Simple and effective. Used by Meta to extend LLaMA from 2K to 8K tokens.

Modify the base frequency: base' = base · α where α depends on the scaling factor. This adjusts the rotation speed so that high-frequency dimensions don't alias. More theoretically motivated than PI.

Combines NTK scaling with a temperature factor and attention scaling. Achieves strong extrapolation from 4K to 128K+ tokens with minimal fine-tuning. Used in many open-source models.

Note: All scaling methods require some fine-tuning on longer sequences to reach optimal quality. The advantage is that they need far less training than learning positions from scratch.

11 · Models Using RoPE

RoPE has become the dominant position encoding in modern LLMs:

  • LLaMA / LLaMA 2 / LLaMA 3 (Meta) — the model family that popularized RoPE in open-source
  • Mistral / Mixtral — combines RoPE with sliding window attention
  • GPT-NeoX / Pythia (EleutherAI) — early open-source adoption
  • PaLM / Gemma (Google) — large-scale production models
  • Qwen (Alibaba) — multilingual models
  • DeepSeek — uses RoPE as a foundation, extended with MLA
  • CodeLlama — extends RoPE with position interpolation for 16K–100K context

12 · Summary

  • RoPE encodes absolute position by rotating Q and K vectors, but the dot product between rotated vectors depends only on relative position.
  • The rotation is applied pair-wise across dimensions, with each pair rotating at a different frequency (θᵢ = 10000-2i/d).
  • No learnable parameters are added — RoPE is purely geometric.
  • Context length can be extended beyond training via interpolation (PI) or frequency scaling (NTK, YaRN).
  • RoPE is the de facto standard in modern LLMs: LLaMA, Mistral, PaLM, Qwen, and many others.
One-line summary: RoPE rotates queries and keys so that attention scores naturally encode relative distance — no extra parameters, no fixed length limit.
