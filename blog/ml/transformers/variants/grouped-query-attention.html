Grouped Query Attention (GQA)

ML Series · Transformers · Variants

1 · The KV Cache Bottleneck

During autoregressive inference, the KV cache stores key and value tensors for all previous tokens across all layers and all heads. For large models, this cache dominates GPU memory:

KV cache size = 2 × n_layers × n_heads × seq_len × head_dim × bytes_per_param LLaMA-2 70B (MHA): 2 × 80 × 64 × 4096 × 128 × 2 bytes = 10.7 GB per sequence

At batch size 32 with 4K context, that's 342 GB just for KV cache — more than most GPUs. The bottleneck isn't computation; it's memory bandwidth. During decoding, the model spends most of its time loading K and V from memory, not multiplying matrices.

The question becomes: do we really need separate K and V for every head?

2 · The MHA → GQA → MQA Spectrum

There's a natural spectrum of how many KV heads you can have relative to query heads:

n_kv_heads = n_query_heads. Every query head has its own dedicated K and V. Maximum expressiveness, maximum memory. This is the original transformer design.

n_kv_heads = n_query_heads / G. Groups of G query heads share a single K and V head. A middle ground: significant memory savings with minimal quality loss.

n_kv_heads = 1. All query heads share the same single K and V. Maximum memory savings, but some quality degradation. This is GQA with G = n_query_heads.

GQA is the Goldilocks solution: MHA uses too much memory, MQA compresses too aggressively. GQA with 8 KV heads (groups of 4–8 query heads each) matches MHA quality while using 4–8× less KV cache memory.

3 · The GQA Idea

In standard MHA with 32 heads, you compute 32 separate Q, K, V projections. In GQA with 8 KV groups, you compute 32 Q projections but only 8 K and 8 V projections. Each K/V pair is shared by a group of 4 query heads:

Query heads: Q₀ Q₁ Q₂ Q₃ | Q₄ Q₅ Q₆ Q₇ | ... | Q₂₈ Q₂₉ Q₃₀ Q₃₁ ↓ ↓ ↓ ↓ ↓ ↓ ↓ ↓ ↓ ↓ ↓ ↓ KV heads: K₀, V₀ | K₁, V₁ | ... | K₇, V₇ (shared by 4) (shared by 4) (shared by 4)

Each query head still computes its own unique attention pattern (because Q differs), but the keys and values they attend over are shared within the group. This is why the cache shrinks: instead of storing 32 K/V entries per layer, you store only 8.

4 · The Math

For a model with H query heads, G KV groups, and group size H/G:

Projections: Q: x → (B, S, H, d_head) via W_Q of shape (d_model, H × d_head) K: x → (B, S, G, d_head) via W_K of shape (d_model, G × d_head) V: x → (B, S, G, d_head) via W_V of shape (d_model, G × d_head) For query head h (in group g = h // (H/G)): attn_h = softmax(Q_h · K_g^T / √d_head) · V_g KV cache reduction: G / H (e.g., 8/32 = 4× smaller)

The key insight: the Q projection retains full rank (H heads), preserving the model's ability to compute diverse attention patterns. Only the K and V projections are reduced, which primarily affects what information is available to attend over — and empirically, sharing K/V across a small group causes minimal quality loss.

Parameter savings on K and V projections: MHA: W_K shape = (d_model, H × d_head), W_V shape = (d_model, H × d_head) GQA: W_K shape = (d_model, G × d_head), W_V shape = (d_model, G × d_head) Reduction ratio = G / H For H=32, G=8: only 25% of the KV parameters

The LLaMA-2 70B comparison is striking: GQA with 8 KV heads uses 8× less KV cache than MHA (1.3 GB vs 10.7 GB per sequence) while maintaining nearly identical benchmark scores.

7 · Quality vs Efficiency Tradeoff

The original GQA paper (Ainslie et al., 2023) showed:

  • GQA-8 (8 KV groups) matches MHA quality on almost all benchmarks while using 8× less KV cache.
  • MQA (1 KV group) shows measurable degradation on some tasks, especially tasks requiring fine-grained comparison between tokens.
  • The quality gap between GQA-8 and MHA shrinks further with uptraining (see next section).
Why does sharing K/V work? The query heads do most of the "thinking" — they decide what to look for. The keys and values provide the "memory" — what's available to look at. Having 8 distinct memories is enough for most tasks; the 32 query heads can still compute 32 different attention patterns over the same 8 sets of keys and values.

8 · Uptraining from MHA to GQA

You don't need to train a GQA model from scratch. The GQA paper introduced uptraining: take a pretrained MHA model and convert it to GQA by mean-pooling the KV heads within each group, then continuing training for a small fraction of the original compute (5% of pretraining tokens).

Uptraining procedure: 1. Start with MHA checkpoint (H query heads, H KV heads) 2. Choose G groups (e.g., G = H/8) 3. For each group g, average the K/V weights: W_K_new[g] = mean(W_K[g*(H/G) : (g+1)*(H/G)]) W_V_new[g] = mean(W_V[g*(H/G) : (g+1)*(H/G)]) 4. Continue training on ~5% of original tokens
Note: Uptraining is how Meta converted LLaMA-2 70B from MHA to GQA. The original 70B model was trained with MHA, then uptrained with GQA-8 for a fraction of the original compute.

9 · Implementation

class GroupedQueryAttention: def __init__(self, d_model, n_q_heads, n_kv_heads): self.n_q_heads = n_q_heads # e.g., 32 self.n_kv_heads = n_kv_heads # e.g., 8 self.group_size = n_q_heads // n_kv_heads # e.g., 4 self.head_dim = d_model // n_q_heads self.W_Q = Linear(d_model, n_q_heads * self.head_dim) self.W_K = Linear(d_model, n_kv_heads * self.head_dim) self.W_V = Linear(d_model, n_kv_heads * self.head_dim) self.W_O = Linear(d_model, d_model) def forward(self, x): B, S, _ = x.shape Q = self.W_Q(x).view(B, S, self.n_q_heads, self.head_dim) K = self.W_K(x).view(B, S, self.n_kv_heads, self.head_dim) V = self.W_V(x).view(B, S, self.n_kv_heads, self.head_dim) # Expand K, V to match Q head count by repeating within groups K = K.repeat_interleave(self.group_size, dim=2) # (B,S,H,d) V = V.repeat_interleave(self.group_size, dim=2) # (B,S,H,d) # Standard scaled dot-product attention scores = (Q @ K.transpose(-2, -1)) / sqrt(self.head_dim) weights = softmax(scores, dim=-1) out = weights @ V # (B, H, S, d) # Concatenate heads and project out = out.reshape(B, S, -1) return self.W_O(out)
Performance note: The repeat_interleave above is for clarity. Optimized implementations (e.g., Flash Attention with GQA support) avoid the explicit expansion and instead index into the KV heads directly during the attention kernel.

10 · Models Using GQA

  • LLaMA-2 70B (Meta) — GQA with 8 KV heads, uptrained from MHA
  • LLaMA-3 (Meta) — all sizes use GQA (8B: 8 KV heads; 70B: 8 KV heads)
  • Mistral 7B / Mixtral 8x7B — GQA with 8 KV heads
  • Gemma / Gemma 2 (Google) — GQA across all sizes
  • Qwen-2 (Alibaba) — GQA for larger variants
  • Command R+ (Cohere) — GQA for efficient serving
  • StarCoder 2 (BigCode) — GQA for code generation

GQA has become the default choice for models above ~7B parameters. MHA is still used for smaller models where the KV cache isn't a bottleneck.

11 · Summary

  • GQA shares K/V projections across groups of query heads, reducing KV cache by H / G×.
  • It sits between MHA (max quality, max memory) and MQA (max compression, some quality loss).
  • With 8 KV groups, GQA matches MHA quality while using 4–8× less memory during inference.
  • Existing MHA models can be converted via uptraining (mean-pool KV heads, train 5% more).
  • GQA is the de facto standard in modern LLMs: LLaMA-3, Mistral, Gemma, Qwen-2.
One-line summary: GQA groups query heads to share fewer KV heads — same quality, fraction of the inference memory.
