Sliding Window Attention

ML Series · Transformers · Variants

The self-attention mechanism is the beating heart of every Transformer model.  It lets each token look at every other token in the sequence — a property that gives Transformers their remarkable representational power.  But this global connectivity comes with a steep price: the compute and memory needed grow quadratically with sequence length.  For short sequences of a few hundred tokens the cost is tolerable, but when sequences stretch to 16 K, 64 K, or 128 K tokens the quadratic blow-up becomes a show-stopper.

Sliding Window Attention (SWA) is one of the cleanest ideas in the efficient-attention family: instead of letting every token attend to every other token, restrict each token to a local window of fixed size w centred on itself.  The result is an attention cost that scales as O(n · w) instead of O(n²) —  and with enough layers the model can still propagate information across the entire context.

In this post we will build up the full picture: the maths, the mask patterns, how information propagates across layers, the memory savings, how sliding-window attention is combined with global tokens in models like Longformer, how to implement it, the dilated variant, and which production models ship with SWA today.

1 · The Quadratic Cost Problem

Recall the standard self-attention computation.  Given an input sequence of length n, we compute queries Q, keys K, and values V — each of shape (n, d) — and then form the attention matrix:

Attention(Q, K, V) = softmax(Q KT / √d) · V Q KT has shape (n × n) → n² dot-products Softmax normalises each of the n rows → n² exponentiations Matrix multiply with V → n² multiply-adds Total FLOPs ≈ 4 · n² · d (per head) Memory for the attention matrix ≈ n² · sizeof(float)

Let's plug in some concrete numbers for a single attention head with d = 128 (a common per-head dimension):

Sequence length n Attention entries (n²) Memory (fp16) Approx. FLOPs
512 262 K 0.5 MB 134 M
2 048 4.2 M 8 MB 2.1 G
4 096 16.8 M 32 MB 8.6 G
16 384 268 M 512 MB 137 G
65 536 4.3 B 8 GB 2.2 T
131 072 (128K) 17.2 B 32 GB 8.8 T

With 32 attention heads the numbers above multiply by 32.  For a 128 K context that's over 1 TB just for the attention matrices of a single layer — far beyond what any current GPU can hold.  Even with FlashAttention (which avoids materialising the full matrix) the compute cost remains O(n²).

⚠ Why not just use bigger GPUs?  GPU memory grows linearly with hardware generations, but sequence lengths we want to process are growing much faster.  Throwing hardware at the problem cannot keep up — we need an algorithmic solution that changes the asymptotic scaling.

The fundamental observation is that in natural language (and many other sequential data), most of the useful context for a token lies nearby.  The verb in a sentence is usually close to its subject; a closing bracket is near the opening bracket.  Full attention treats the relationship between token 5 and token 50 000 with the same computational budget as token 5 and token 6, even though the latter is overwhelmingly more likely to be useful.  Sliding Window Attention exploits this locality.

2 · The Core Idea: Limit the Attention Window

The modification is almost embarrassingly simple: instead of letting query i attend to all keys 0 … n−1, restrict it to a window of size w centred on position i:

For query position i, attend to keys j where: max(0, i − ⌊w/2⌋) ≤ j ≤ min(n−1, i + ⌊w/2⌋) Or equivalently: |i − j| ≤ ⌊w/2⌋

Typical window sizes in practice are:

  • w = 256 — Longformer's default local window
  • w = 512 — common in early sparse-attention research
  • w = 4096 — Mistral 7B and Mistral-family models

Visually, a full attention mask and a sliding-window mask for the same 8-token sequence look like this:

Sliding Window (w = 3, causal)

In the full causal mask every query attends to all preceding keys (the lower triangle).  In the sliding-window version, only a narrow band around the diagonal survives.  The light-green cells mark the edge of the window — positions that are included but are the furthest from the query.

Key Insight:  Sliding window attention does not throw away information permanently.  It merely limits what a single layer can see.  Through multiple layers, information from distant tokens propagates hop by hop, much like how a CNN builds a large receptive field by stacking convolutional layers.

3 · The Attention Mask Pattern

We can formalise the sliding window mask as a binary matrix M ∈ {0, 1}n×n:

M[i, j] = 𝟙( |i − j| ≤ ⌊w/2⌋ ) where 𝟙(·) is the indicator function w is the window size (an odd number is simplest) i is the query position (row) j is the key position (column) For a causal sliding window we additionally require j ≤ i: Mcausal[i, j] = 𝟙( 0 ≤ i − j ≤ w − 1 )

This yields a band matrix — a matrix where all non-zero entries lie within a diagonal band of width  w.  Band matrices are one of the most well-studied structures in numerical linear algebra, and many efficient storage and multiplication algorithms exist for them.

The masked attention scores become:

A = softmax( (Q KT ⊙ M + (1 − M) · (−∞)) / √d ) where ⊙ is element-wise multiplication. Positions where M[i,j] = 0 get −∞ before the softmax, so their attention weight becomes exactly 0. In practice we never materialise the full n×n matrix. We only compute the w non-zero entries per row, giving a dense (n × w) score matrix — the band storage format.

The band-matrix view is important for implementation.  Rather than building the full n × n matrix and masking most of it to −∞, an efficient implementation computes only the n × w entries that survive the mask, avoiding both the wasted memory and the wasted FLOPs.

Boundary behaviour

At the beginning of the sequence, positions i < ⌊w/2⌋ have fewer than w keys to attend to because there are not enough preceding tokens.  In the causal variant, position 0 always attends only to itself.  This is analogous to the "same" padding behaviour in convolutions — the window is simply truncated at the edges.

4 · Effective Receptive Field Through Layers

A single sliding-window layer lets each token attend to at most w neighbours.  But what happens when we stack L such layers?

Token i aggregates information from tokens in the range [i − ⌊w/2⌋,  i + ⌊w/2⌋].

Each of those w neighbours has already aggregated information from its own window.  So the output representation of token i at layer 2 now carries indirect information from positions up to 2 × ⌊w/2⌋ away.

After L layers the effective receptive field spans up to L × w positions (approximately L × ⌊w/2⌋ in each direction for non-causal attention).

Effective receptive field after L layers ≈ L × w Example (Mistral 7B): w = 4096, L = 32 layers Effective field = 32 × 4096 = 131 072 tokens ≈ 128K This matches Mistral's advertised 128K context handling capability while each individual layer only ever computes 4096 attention entries per query.

This is the crucial insight that makes sliding-window attention practical.  You do not need each layer to see the full sequence.  You only need enough layers for the receptive field to cover the entire context.  Because modern Transformers already have 32–80 layers, this condition is easily satisfied even with a relatively small window.

Analogy — CNNs:  A 3×3 convolutional kernel sees only 9 pixels, but stacking 10 such layers gives a receptive field of 21×21 pixels.  Sliding-window attention is the sequence-domain equivalent.  The "kernel size" is w, and each Transformer layer acts as one convolutional step — except with data-dependent weights (attention scores) instead of fixed learned filters.

6 · Memory and Compute Savings

With sliding-window attention, each of the n queries only computes dot products with w keys.  The attention score matrix shrinks from n × n to n × w:

Full attention: Memory = O(n²) FLOPs = O(n² · d) Sliding window attention: Memory = O(n · w) FLOPs = O(n · w · d) Ratio (savings factor) = n / w n = 16 384, w = 512 → savings = 32× n = 65 536, w = 4096 → savings = 16× n = 131 072, w = 4096 → savings = 32×

Here is a concrete comparison table.  Memory values are for the attention matrix alone (fp16), per head:

Sequence (n) Window (w) Full Attn Memory SWA Memory Savings
4 096 256 32 MB 2 MB 16×
4 096 512 32 MB 4 MB 8×
16 384 512 512 MB 16 MB 32×
16 384 4 096 512 MB 128 MB 4×
65 536 4 096 8 GB 512 MB 16×
131 072 4 096 32 GB 1 GB 32×

The savings become especially dramatic at long sequence lengths.  At 128 K tokens with w = 4096, we go from 32 GB down to 1 GB per head — a factor that makes the difference between "impossible on any GPU" and "fits on a single A100."

KV-cache savings during inference

During autoregressive generation, the standard KV-cache stores all past key/value vectors and grows linearly with the generated sequence.  With sliding-window attention, only the most recent w key/value pairs are needed.  Older entries can be evicted from the cache, giving a constant-size KV-cache of w × d per layer per head, regardless of how long the generated sequence becomes.  This is a massive win for serving long-context models in production.

⚠ Trade-off:  Evicting old KV entries means the model truly cannot directly attend to tokens outside the window.  For tasks requiring very precise long-range retrieval (e.g., "what was the 5th word of the document?") a pure sliding window may lose information that full attention would preserve.  This is why many models combine SWA with global attention tokens (see Section 7).

7 · Combining with Global Attention

A purely local window can lose track of document-level context.  The Longformer (Beltagy et al., 2020) introduced an elegant solution: use sliding-window attention for most tokens, but designate a small number of global tokens that attend to (and are attended by) the entire sequence.

How global tokens work

Let G ⊂ {0, …, n−1} be the set of global-token positions (e.g., the [CLS] token, or the first token of each sentence).  The combined mask is:

M[i, j] = 1 if |i − j| ≤ ⌊w/2⌋ (local window) = 1 if i ∈ G or j ∈ G (global row / column) = 0 otherwise In matrix form, the mask looks like the sliding-window band plus full rows and columns at the global positions, forming a cross-shaped pattern overlaid on the band.

The global tokens act as information "hubs."  A distant local token can communicate with another distant local token by routing through a global token in just two hops: local → global → local.

Visual: sliding window + global

Global   Local window   Window edge   Masked

In the diagram above, token 0 (the [CLS] token) is global: its entire row and column are purple, meaning it attends to all tokens and all tokens attend to it.  Every other token uses a local sliding window (green band).  The complexity increases from O(n·w) to O(n·w + |G|·n), but since |G| is typically tiny (1–10 tokens), this is negligible.

BigBird's random attention

BigBird (Zaheer et al., 2020) extends this further by adding random attention edges on top of the sliding window and global tokens.  Each token randomly attends to a small set of r other tokens, which theoretically reduces the graph diameter and helps information mix faster.  In practice, the combination of local + global + random achieves strong performance on long-document benchmarks while remaining O(n) in total attention cost.

8 · Implementation

Below is pseudo-code for building the sliding-window attention mask and applying it during the forward pass.  The key optimisation is never to materialise the full n × n matrix.

# ── Build a causal sliding-window mask ────────────────── function sliding_window_mask(n, w): # Returns a boolean mask of shape (n, w) # mask[i, k] = True means query i attends to key (i − k) # k ∈ [0, w-1] is the offset within the window mask ← ones(n, w, dtype=bool) for i in 0 … n−1: for k in 0 … w−1: j ← i − k # actual key position if j < 0: mask[i, k] ← False # past start of sequence return mask # ── Sliding-window attention forward pass ─────────────── function swa_forward(Q, K, V, w): # Q, K, V : (n, d) w : window size n, d ← Q.shape # 1. Gather keys/values in a window around each query # K_win[i] = K[max(0,i-w+1) : i+1] shape → (n, w, d) K_win ← gather_windows(K, w) V_win ← gather_windows(V, w) # 2. Compute local dot-product scores (n, 1, d) × (n, d, w) → (n, w) scores ← einsum("nd, nwd → nw", Q, K_win) / sqrt(d) # 3. Apply the mask (set masked positions to −∞) mask ← sliding_window_mask(n, w) scores[~mask] ← −∞ # 4. Softmax over the window dimension attn_weights ← softmax(scores, dim=−1) # (n, w) # 5. Weighted sum of windowed values output ← einsum("nw, nwd → nd", attn_weights, V_win) return output # ── Helper: gather a sliding window of vectors ───────── function gather_windows(X, w): # X : (n, d) → X_win : (n, w, d) # X_win[i, k] = X[i − k] (zero-padded if i − k < 0) n, d ← X.shape X_pad ← pad(X, (w−1, 0), value=0) # (n+w−1, d) # Use strided view / unfold to create the windows X_win ← unfold(X_pad, size=w, step=1) # (n, w, d) return X_win

Practical notes

  • FlashAttention-2 has native support for sliding-window masks via a window_size parameter, so in practice you rarely need to build the mask yourself.
  • Hugging Face Transformers implements SWA for Mistral, Gemma, and others using the sliding_window config option, which is passed through to the attention kernel.
  • On TPUs, where FlashAttention is not available, the band-matrix approach (materialising only the n × w slice) is the standard implementation path, often using jax.lax.dynamic_slice.

9 · Dilated Sliding Window

Standard sliding-window attention attends to w contiguous positions.  Dilated (or strided) sliding-window attention introduces gaps of size d (the dilation factor) between attended positions:

Dilated sliding window: For query i, attend to key j if: (i − j) mod d = 0 AND 0 ≤ i − j ≤ (w−1) · d where w = number of positions attended (window size) d = dilation factor (gap between attended positions) Standard SWA is the special case d = 1. Effective span of the window = (w − 1) · d + 1 Example: w = 4, d = 3 Query 12 attends to keys: 12, 9, 6, 3 Span = (4 − 1) · 3 + 1 = 10 positions Compared to standard w = 4 (span = 4), dilated w = 4, d = 3 covers 2.5× more context with the same number of attention computations.

The idea is directly inspired by dilated convolutions (also called atrous convolutions) from computer vision.  By skipping positions, the window covers a much larger span without increasing the number of attended keys.  The downside is that the gaps introduce a form of aliasing —  tokens in the gaps are not directly attended to.

Multi-head dilation

A common strategy is to use different dilation factors across attention heads.  For example, in a model with 8 heads:

  • Heads 0–3: dilation d = 1 (standard local window, fine-grained detail)
  • Heads 4–5: dilation d = 2 (medium range)
  • Heads 6–7: dilation d = 4 (long range, coarser)

This lets the model simultaneously capture patterns at multiple scales, analogous to multi-scale feature extraction in vision architectures.  Longformer uses exactly this multi-head dilation strategy.

Design tip:  When choosing dilation factors, ensure that consecutive dilation values are not multiples of each other.  For instance, d ∈ {1, 2, 3} gives better coverage than {1, 2, 4} because the latter leaves more systematic gaps.  This mirrors best practices from dilated convolution research.

10 · Models Using Sliding Window

Sliding-window attention has moved from a research concept to a standard component in several high-profile production models:

Model Window Size Global Tokens? Dilation? Context Length Notes
Longformer 256–512 Yes Yes (per head) 4 096 Pioneered local+global pattern for long documents
BigBird Configurable Yes No 4 096 Adds random attention on top of local+global
Mistral 7B 4 096 No No 32K (128K effective) Pure SWA in every layer; rolling KV-cache
Mixtral 8×7B 4 096 No No 32K Inherits Mistral's SWA; adds MoE on top
Gemma / Gemma 2 4 096 No (interleaved) No 8K Alternates SWA and full-attention layers
Phi-3 Varies No (interleaved) No Up to 128K Uses SWA in a subset of layers for efficiency

Mistral's approach in detail

Mistral 7B is perhaps the cleanest example.  Every one of its 32 Transformer layers uses causal sliding-window attention with w = 4096.  There are no global tokens, no random connections, and no dilated heads — just a pure local window.  The model relies entirely on layer stacking to achieve long-range information flow (32 × 4096 = 131 072 effective receptive field).

During inference, Mistral uses a rolling KV-cache that overwrites the oldest entry once the cache reaches size w.  This keeps memory usage constant regardless of generation length — a property that is especially valuable for chat applications with very long conversations.

Gemma 2's interleaved strategy

Gemma 2 takes a hybrid approach: it alternates between layers with sliding-window attention and layers with full (global) attention.  The idea is that the global layers can capture long-range dependencies directly, while the SWA layers handle local patterns efficiently.  This strikes a balance between the speed of pure SWA and the expressiveness of full attention.

11 · Comparison: Full vs Sliding Window vs Sparse

The table below summarises the key trade-offs between full attention, sliding-window attention, and sparse attention methods (which include Longformer/BigBird-style combinations).

Property Full Attention Sliding Window Sparse (Local + Global + Random)
Memory per layer O(n²) O(n · w) O(n · (w + |G| + r))
FLOPs per layer O(n² · d) O(n · w · d) O(n · (w + |G| + r) · d)
Direct context Full sequence w tokens w + n (via global) + r
Effective context (L layers) Full sequence L × w Full (via global tokens)
KV-cache size O(n) — grows with sequence O(w) — constant O(n) — global tokens need full cache
Implementation complexity Simple Moderate (band ops) High (multiple patterns)
Quality on short sequences Best Comparable Comparable
Quality on long-range retrieval Best Degrades gracefully Good (global tokens help)
Practical max sequence ~8K–32K (GPU limited) 128K+ (Mistral) 16K–128K
Example models GPT-4, LLaMA 2 Mistral, Mixtral Longformer, BigBird

The choice between these approaches depends heavily on the use case.  For tasks where every token genuinely needs to attend to every other token (e.g., complex reasoning across a full document), full attention is still the gold standard.  For tasks dominated by local patterns (most language modelling, code generation, chat), sliding window provides a massive efficiency gain with minimal quality loss.  Sparse methods are the sweet spot when you need both long range and efficiency, at the cost of additional implementation complexity.

12 · Summary

Let's recap the key takeaways:

  1. Full attention is O(n²) in both memory and compute, making it impractical for long sequences (16K+ tokens).
  2. Sliding Window Attention restricts each token to a local window of size w, reducing the cost to O(n · w).
  3. The attention mask becomes a band matrix: M[i,j] = 1 iff |i−j| ≤ ⌊w/2⌋.
  4. Through L layers, the effective receptive field grows to L × w, recovering global context.
  5. KV-cache becomes constant-size (w entries), enabling efficient autoregressive generation.
  6. Global tokens (Longformer) and random attention (BigBird) can be layered on top for tasks that require explicit long-range connections.
  7. Dilated sliding windows increase the effective span without increasing the number of attended positions.
  8. Production models like Mistral 7B, Gemma 2, and Phi-3 use SWA as a core architectural component, achieving 32K–128K context with competitive quality.

Sliding Window Attention is a beautiful example of how a simple structural prior — "most useful context is local" — can be exploited to dramatically improve the scalability of Transformers without sacrificing much in terms of model quality.  As context windows continue to grow (we're already seeing models target 1M+ tokens), efficient attention mechanisms like SWA will only become more important.
