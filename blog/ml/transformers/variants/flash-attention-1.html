    Flash Attention v1: IO-Aware Exact Attention

    ML Series · June 2026

    Transformers changed deep learning, but their self-attention mechanism has a dirty secret: for long sequences, the runtime bottleneck is not how many floating-point operations the GPU performs — it is how many bytes it moves between slow global memory (HBM) and fast on-chip memory (SRAM). Flash Attention, introduced by Tri Dao et al. in 2022, eliminates this bottleneck by restructuring the exact attention computation so that the full N×N attention matrix is never materialized in HBM. The result is a 2–4× wall-clock speedup and a reduction in memory usage from O(N²) to O(N), all while computing the mathematically identical output as standard attention.

    This post walks through every layer of the algorithm: the GPU memory hierarchy that motivates it, the online softmax trick that enables it, the tiling strategy that implements it, and the IO-complexity analysis that proves why it works. We will also build an interactive animation of the tiling process and examine pseudo-code you can map directly to a CUDA kernel.

    1 · The Memory Bottleneck

    Standard self-attention for a sequence of length N with head dimension d proceeds in three stages: (1) compute the score matrix S = Q · KT, an N×N matrix; (2) apply row-wise softmax to obtain the attention weight matrix P; (3) compute the output O = P · V. Each of these stages reads from and writes to GPU global memory (HBM — High Bandwidth Memory).

    Consider the cost. For a sequence length of N = 8192 and head dimension d = 64, the score matrix S alone contains 8192² = 67 million entries. In FP16, that is 128 MB — just for one attention head in one layer. A model with 32 heads and 40 layers would need terabytes of HBM bandwidth per forward pass just to read and write these intermediate matrices.

    The problem is not FLOPs. Modern GPUs like the A100 can sustain 312 TFLOPS of FP16 compute. The problem is that HBM bandwidth is limited to roughly 2 TB/s, while the attention computation is memory-bound. Most of the GPU's execution time is spent waiting for data to arrive from HBM, not performing arithmetic.

    Standard Attention Memory Traffic (per head): Write S = Q K^T → N² elements written to HBM Read S for softmax → N² elements read from HBM Write P = softmax(S) → N² elements written to HBM Read P for P · V → N² elements read from HBM Total HBM access = 4 N² (in elements, ignoring Q, K, V, O reads/writes)

    This O(N²) memory footprint is what makes standard attention slow for long sequences. Doubling the sequence length quadruples both memory usage and HBM traffic. Flash Attention attacks this problem head-on: by restructuring the computation to use tiling and online softmax, it avoids ever writing the full N×N matrix to HBM.

    2 · The Core Insight: Tiling

    The fundamental idea behind Flash Attention is simple: instead of materializing the entire N×N score matrix in HBM, break Q, K, and V into small blocks (tiles) that fit in SRAM, compute the attention for each pair of tiles, and accumulate the result incrementally. At no point does the full N×N matrix exist in memory.

    Concretely, partition Q into Tr blocks of size Br × d, and K, V into Tc blocks of size Bc × d. For each pair of blocks (Qi, Kj), compute the local score block Sij = Qi · KjT, which has size Br × Bc. This small block fits comfortably in SRAM. Apply a partial softmax to this block, multiply by Vj, and accumulate into the output block Oi.

    The challenge is that softmax is a global operation — it requires the maximum and the sum over the entire row of S, not just one block. The key algorithmic trick that makes tiling possible is the online softmax algorithm, which computes softmax incrementally as new blocks arrive, without ever needing the complete row.

    Key insight: Flash Attention tiles the attention computation so that each tile pair fits in fast SRAM. The N×N attention matrix is computed piece by piece and never fully materialized. This changes the memory bottleneck from O(N²) to O(N).

    This tiling strategy is not a new approximation of attention. It computes the exact same result as standard attention, but reorganizes the order of operations so that intermediate values stay in fast memory. The idea is inspired by classical techniques in high-performance computing — loop tiling for cache optimization — applied to the specific structure of the attention computation.

    3 · The GPU Memory Hierarchy

    To understand why Flash Attention works, you need to understand the GPU memory hierarchy. Modern GPUs have two main levels of memory that matter for attention:

    • HBM (High Bandwidth Memory): The main GPU memory. On an A100, this is 40 GB or 80 GB, with a bandwidth of approximately 1.5–2.0 TB/s. This is where tensors like Q, K, V, and O live. It is "high bandwidth" compared to CPU RAM, but slow compared to SRAM.
    • SRAM (Static RAM / Shared Memory): On-chip memory local to each streaming multiprocessor (SM). On an A100, total SRAM across all SMs is about 20 MB, with an aggregate bandwidth of approximately 19 TB/s. This is roughly 10× faster than HBM, but roughly 1000× smaller.
    Property SRAM (On-Chip) HBM (Global)
    Capacity (A100) ~20 MB total 40–80 GB
    Bandwidth (A100) ~19 TB/s ~1.5–2.0 TB/s
    Latency ~28 cycles ~200–400 cycles
    Shared across Threads in one SM All SMs on the GPU
    Programmer control Explicit (shared mem) Implicit (global mem)
    Typical use Tile buffers, accumulators Model weights, activations

    The key ratio is the bandwidth gap: SRAM is roughly 10× faster than HBM. Any algorithm that can keep its working set in SRAM and minimize round-trips to HBM will be dramatically faster. This is exactly what Flash Attention does.

    Standard attention makes at least 4 full passes over the N×N matrix in HBM (write S, read S for softmax, write P, read P for output). Flash Attention restructures the computation so that each element of S is computed in SRAM, used immediately, and discarded — it never touches HBM. Only Q, K, V (input) and O (output) are read from / written to HBM, and each is only O(N · d) elements.

    Arithmetic Intensity (AI) = FLOPs / Bytes Moved Standard Attention AI ≈ O(1) → memory-bound Flash Attention AI ≈ O(d) → closer to compute-bound For d = 64, Flash Attention moves ~64× fewer bytes per FLOP.

    The arithmetic intensity of standard attention is low because every FLOP requires approximately one byte moved to/from HBM. Flash Attention increases this ratio by a factor proportional to d, pushing the computation from memory-bound toward compute-bound. This is why the wall-clock speedup is so dramatic even though the total number of FLOPs is the same (or even slightly higher, due to recomputation in the backward pass).

    4 · The Algorithm: Online Softmax

    The standard softmax over a vector x of length N is:

    softmax(x_i) = exp(x_i - m) / ℓ where: m = max(x_1, x_2, ..., x_N) (for numerical stability) ℓ = Σ_j exp(x_j - m) (normalization denominator)

    Computing this requires two passes over the data: one to find the maximum m, and one to compute the sum ℓ and the normalized values. In standard attention, this is fine because the entire row of S is in HBM and we just read it twice. But in Flash Attention, we are processing the row in blocks — when we process block j, we have not yet seen blocks j+1, j+2, …. We need a way to compute softmax incrementally.

    The online softmax algorithm (Milakov & Gimelshein, 2018) solves this. The idea is to maintain a running maximum m and a running sum ℓ, and update them as each new block arrives. When the maximum changes, we rescale the previously accumulated values.

    Online Softmax Update (processing block j): Given: running max m_old, running sum ℓ_old, running output O_old New block scores: s_j = Q_i · K_j^T (size B_r × B_c) Step 1: m_new = max(m_old, rowmax(s_j)) Step 2: ℓ_new = exp(m_old - m_new) · ℓ_old + rowsum(exp(s_j - m_new)) Step 3: P_j = diag(exp(s_j - m_new)) / ℓ_new (local attention weights) Step 4: O_new = (ℓ_old · exp(m_old - m_new) / ℓ_new) · O_old + P_j · V_j

    Let us unpack each step. In Step 1, we update the running maximum by comparing the old maximum with the maximum of the new block’s scores. In Step 2, we update the running sum: the old sum is rescaled by exp(mold − mnew) to account for the change in maximum, and we add the exponentiated scores of the new block. In Step 3, we compute the local attention weights for this block. In Step 4, we update the output: the old output is rescaled by the ratio of old-to-new normalizers, and we add the contribution of the current block.

    The critical property is that after processing all Tc blocks, the accumulated output Oi is exactly equal to what standard attention would produce. No approximation is involved. The online softmax is numerically equivalent to the two-pass softmax, just computed incrementally.

    Why this works: The exp function satisfies exp(a − c) = exp(a − b) · exp(b − c). This means when the running maximum changes from mold to mnew, we can exactly rescale all previously accumulated values by multiplying by exp(mold − mnew). No information is lost.

    The online softmax approach also handles numerical stability naturally. Because we always subtract the running maximum before exponentiating, the largest exponent is always 0 (i.e., exp(0) = 1), preventing overflow. The running maximum only increases or stays the same as we process more blocks, which means the rescaling factor exp(mold − mnew) is always ≤ 1, preventing the accumulated sum from growing unboundedly.

    5 · Tiled Forward Pass

    Now we can describe the full tiled forward pass. The algorithm has a two-level loop structure: the outer loop iterates over K/V blocks (columns of the attention matrix), and the inner loop iterates over Q blocks (rows).

    Divide Q into Tr = ⌈N / Br⌉ row-blocks, each of size Br × d. Divide K and V into Tc = ⌈N / Bc⌉ column-blocks, each of size Bc × d. Initialize output O as zeros (N × d), running maximums m as −∞ (N × 1), and running sums ℓ as zeros (N × 1) in HBM.

    For j = 1, 2, …, Tc: load Kj and Vj from HBM to SRAM. Each block is Bc × d elements. These stay in SRAM for the entire inner loop.

    For i = 1, 2, …, Tr: load Qi, Oi, mi, ℓi from HBM to SRAM. Compute the local score block Sij = Qi · KjT in SRAM (size Br × Bc). This is a small matrix multiply that the GPU tensor cores can execute at full throughput because both operands are in SRAM.

    Compute mnew = max(mi, rowmax(Sij)). Compute ℓnew = exp(mi − mnew) · ℓi + rowsum(exp(Sij − mnew)). Update the output: Oi ← (ℓi / ℓnew) · exp(mi − mnew) · Oi + (1 / ℓnew) · exp(Sij − mnew) · Vj. Update mi ← mnew, ℓi ← ℓnew.

    Write Oi, mi, ℓi back to HBM. Proceed to the next Q block (inner loop) or the next K/V block (outer loop).

    The key property is that Kj and Vj are loaded once per outer loop iteration and reused across all Tr Q blocks. Similarly, each Qi is loaded once per inner loop iteration. This reuse is what minimizes HBM traffic.

    Block sizes (A100, d = 64, FP16): SRAM per SM = 192 KB (max configurable shared memory) B_r = B_c = 128 (typical choice) Q_i block = 128 × 64 × 2 bytes = 16 KB K_j block = 128 × 64 × 2 bytes = 16 KB V_j block = 128 × 64 × 2 bytes = 16 KB S_ij block = 128 × 128 × 2 bytes = 32 KB O_i block = 128 × 64 × 2 bytes = 16 KB m_i, ℓ_i = 128 × 2 × 2 bytes = 0.5 KB Total SRAM ≈ 96.5 KB ✓ fits in 192 KB

    The block sizes are chosen so that all working data fits in SRAM. On the A100, each SM has up to 192 KB of configurable shared memory. With Br = Bc = 128 and d = 64, the total SRAM footprint is under 100 KB, well within budget. This leaves room for register spilling and other overhead.

    7 · IO Complexity Analysis

    The central theoretical contribution of Flash Attention is a formal IO-complexity analysis. Following the external-memory model of Aggarwal and Vitter (1988), we count the number of elements transferred between HBM and SRAM, treating SRAM as a fast cache of size M elements.

    Standard Attention HBM Accesses: Θ(N · d + N²) reads and writes Breakdown: • Read Q, K (N × d each) → 2Nd • Write S = QK^T (N × N) → N² • Read S for softmax → N² • Write P = softmax(S) (N × N) → N² • Read P for PV → N² • Read V (N × d) → Nd • Write O (N × d) → Nd Dominated by the 4N² terms for large N.
    Flash Attention HBM Accesses: Θ(N² · d² / M) where M = SRAM size (in elements). For A100: M ≈ 100K elements (in FP16), d = 64 → HBM accesses ≈ N² · 64² / 100K = N² · 4096 / 100K ≈ N² / 24 Speedup over standard attention ≈ 4N² / (N²/24) = 96× (in practice, 2-4× due to constants, compute overlap, etc.)

    The key insight is that M (SRAM size) appears in the denominator. The larger the SRAM, the bigger the tiles we can use, and the fewer HBM round-trips we need. Flash Attention also proves a lower bound: for any algorithm that computes exact attention, the HBM access complexity is Ω(N² d² / M). This means Flash Attention is asymptotically optimal — no exact attention algorithm can have fewer HBM accesses (up to constant factors).

    In practice, the wall-clock speedup depends on the ratio of compute time to memory access time. For short sequences (N < 512), the computation is already fast and the overhead of tiling may not help much. For long sequences (N ≥ 2048), the O(N²) HBM traffic of standard attention dominates, and Flash Attention’s reduction in IO yields dramatic speedups. The sweet spot is N = 1024–16384, where Flash Attention is typically 2–4× faster.

    Subtlety: Flash Attention does slightly more total FLOPs than standard attention because the online softmax requires extra multiplications for rescaling. However, these extra FLOPs happen in SRAM at 19 TB/s, while the saved HBM accesses would have cost 1.5 TB/s. The net effect is a large speedup because the computation was memory-bound, not compute-bound.

    8 · Backward Pass

    In the standard backward pass for attention, the gradient computation requires the attention weight matrix P (the softmax output). In standard attention, P is stored during the forward pass and retrieved during the backward pass. But P is N×N — storing it defeats the purpose of Flash Attention’s O(N) memory.

    Flash Attention solves this by recomputing P during the backward pass rather than storing it. The forward pass saves only the output O, the per-row statistics m and ℓ (both O(N)), and the inputs Q, K, V. During the backward pass, for each tile pair, the kernel reloads Qi, Kj from HBM, recomputes Sij = Qi · KjT, applies the saved m and ℓ to reconstruct the local Pij, and then computes the gradients.

    Backward pass recomputation: Saved from forward: O (N×d), m (N×1), ℓ (N×1), Q, K, V (N×d each) Total forward save: O(N·d) ← no N² term For each tile pair (i, j): Recompute S_ij = Q_i · K_j^T (in SRAM) Recompute P_ij = softmax(S_ij) using saved m_i (in SRAM) Compute dV_j += P_ij^T · dO_i (in SRAM) Compute dP_ij = dO_i · V_j^T (in SRAM) Compute dS_ij = P_ij ⊙ (dP_ij - rowsum(dO_i ⊙ O_i)) Compute dQ_i += dS_ij · K_j (in SRAM) Compute dK_j += dS_ij^T · Q_i (in SRAM)

    This recomputation strategy trades extra FLOPs for reduced memory. The extra FLOPs are the cost of recomputing each Sij block, which is the same matmul done in the forward pass. In total, the backward pass does roughly 2× the FLOPs of the forward pass (same as standard attention backward), plus the recomputation overhead. But critically, these extra FLOPs happen in SRAM, so they are essentially free compared to the cost of reading a stored N×N matrix from HBM.

    The memory savings are dramatic. Standard attention backward requires O(N²) memory to store P. Flash Attention backward requires O(N) memory. For N = 16384, that is the difference between 2 GB and 2 MB per attention head — a 1000× reduction.

    This recomputation-instead-of-storage approach is an instance of gradient checkpointing (also called activation checkpointing or rematerialization). Flash Attention applies it specifically to the attention matrix, where the savings are largest because the attention matrix is the only O(N²) intermediate in the transformer.

    9 · Implementation Details

    Below is pseudo-code for the Flash Attention forward pass. This corresponds closely to the actual CUDA kernel structure, though the real implementation uses warp-level primitives and carefully tuned shared memory layouts.

    // Flash Attention Forward Pass (single head) // Inputs: Q, K, V ∈ ℝ^{N×d} in HBM // Output: O ∈ ℝ^{N×d} in HBM function FlashAttentionForward(Q, K, V, N, d, B_r, B_c): // Initialize output and statistics in HBM O ← zeros(N, d) // output accumulator m ← fill(N, −∞) // running row-wise max ℓ ← zeros(N) // running row-wise sum T_c ← ⌈N / B_c⌉ // number of K/V blocks T_r ← ⌈N / B_r⌉ // number of Q blocks // Outer loop: iterate over K/V blocks for j ← 1 to T_c: // Load K_j, V_j from HBM → SRAM K_j ← load_to_sram(K[j·B_c : (j+1)·B_c, :]) V_j ← load_to_sram(V[j·B_c : (j+1)·B_c, :]) // Inner loop: iterate over Q blocks for i ← 1 to T_r: // Load Q_i, O_i, m_i, ℓ_i from HBM → SRAM Q_i ← load_to_sram(Q[i·B_r : (i+1)·B_r, :]) O_i ← load_to_sram(O[i·B_r : (i+1)·B_r, :]) m_i ← load_to_sram(m[i·B_r : (i+1)·B_r]) ℓ_i ← load_to_sram(ℓ[i·B_r : (i+1)·B_r]) // Compute local score block in SRAM S_ij ← Q_i · K_j^T // B_r × B_c matmul // Online softmax update m_new ← max(m_i, rowmax(S_ij)) P_ij ← exp(S_ij − m_new) // unnormalized weights ℓ_new ← exp(m_i − m_new) · ℓ_i + rowsum(P_ij) // Rescale old output and add new contribution α ← exp(m_i − m_new) · ℓ_i / ℓ_new β ← 1 / ℓ_new O_i ← α · O_i + β · P_ij · V_j // Update statistics m_i ← m_new ℓ_i ← ℓ_new // Write O_i, m_i, ℓ_i back to HBM write_to_hbm(O, O_i, i·B_r) write_to_hbm(m, m_i, i·B_r) write_to_hbm(ℓ, ℓ_i, i·B_r) return O, m, ℓ

    Several implementation details deserve attention:

    • Block size tuning: The optimal Br and Bc depend on the GPU architecture, head dimension d, and data type. On the A100 with d = 64 and FP16, Br = Bc = 128 is typical. For larger d (e.g., 128), smaller block sizes are needed to fit in SRAM.
    • Causal masking: For decoder-only models, Flash Attention applies causal masking by skipping tile pairs where Qi precedes Kj entirely (i.e., the block is fully masked). For partially masked blocks, the mask is applied to Sij before the softmax. This reduces computation by roughly half for causal attention.
    • Multi-head batching: In practice, the outer loop is parallelized across batch elements and attention heads. Each (batch, head) pair is assigned to a threadblock (or a group of threadblocks), and the tiling loops run within each threadblock using shared memory.
    • Dropout: Attention dropout is applied by generating a random mask for each tile pair in SRAM. The random state is saved for the backward pass (or re-seeded with a deterministic offset) so that the same mask can be regenerated during recomputation.
    • FP16 / BF16 precision: The Sij matmul is computed in FP16 (or BF16) using tensor cores. The online softmax accumulators (m, ℓ, and the rescaling) are kept in FP32 for numerical stability, then the final output is cast back to FP16 for storage.

    Several trends are visible. First, the speedup increases with sequence length because longer sequences have more N² HBM traffic to eliminate. Second, the memory savings are even more dramatic than the speed improvement — going from O(N²) to O(N) means Flash Attention can handle sequences 4× longer on the same GPU.     End-to-end training benchmarks show similar gains. GPT-2 training with Flash Attention is 2–3× faster than with standard attention. BERT pretraining reaches target accuracy 15% faster. The speedup translates directly to reduced training time and cost because attention is typically the bottleneck layer in transformers.

    Perhaps most importantly, Flash Attention enables training with longer contexts without any architectural changes. Models that were previously limited to 512 or 2048 tokens by memory constraints can now train on 4096, 8192, or even 16384 tokens. This has led to significant improvements in tasks that benefit from long context, such as document summarization, long-range dependency modeling, and retrieval-augmented generation.

    11 · Exact, Not Approximate

    A critical property of Flash Attention is that it computes the exact same result as standard attention. This distinguishes it from a long line of work on efficient attention approximations:

    • Sparse attention (BigBird, Longformer, etc.) restricts which token pairs can attend to each other, using local windows, global tokens, or random patterns. This changes the attention pattern and is an approximation.
    • Linear attention (Performers, Random Feature Attention, etc.) replaces the softmax kernel with a linear function, reducing complexity to O(N) but changing the attention mechanism fundamentally.
    • Low-rank attention (Linformer, etc.) projects K and V to a lower dimension, computing attention in a compressed space. This is an approximation whose quality depends on the effective rank of the attention matrix.

    Flash Attention takes a completely different approach. It does not modify what is computed — it modifies how it is computed. The mathematical operations are identical to standard attention; only the order of execution and the memory access pattern change. This means:

    • No loss of model quality or expressiveness
    • Drop-in replacement for any existing attention implementation
    • No need to retrain or fine-tune models
    • No hyperparameter tuning for approximation quality
    • Bit-for-bit identical output (up to floating-point associativity)
    Exactness guarantee: Flash Attention is a systems optimization, not a modeling approximation. The output is mathematically identical to standard softmax(Q KT / √d) · V. The only numerical difference comes from floating-point non-associativity (which also varies across standard implementations).

    This exactness property is what made Flash Attention immediately adoptable in production. Libraries like PyTorch, Hugging Face Transformers, and xformers integrated it as the default attention backend because it requires no model changes — it is purely a faster execution strategy for the same computation.

    The one caveat is floating-point non-associativity. Because Flash Attention processes blocks in a different order than standard attention, the accumulated floating-point results may differ at the level of machine epsilon. In practice, this difference is negligible (typically less than 1e-5 relative error in FP16) and does not affect model quality. All standard attention implementations have similar non-determinism depending on kernel launch order and hardware.

    12 · Summary

    Flash Attention v1 is a landmark contribution that reframes the attention efficiency problem. Instead of asking “how can we approximate attention with fewer FLOPs?”, it asks “how can we compute exact attention with fewer memory accesses?” The answer is a tiling strategy combined with online softmax that keeps the working set in fast SRAM and avoids materializing the N×N attention matrix in slow HBM.

    The key contributions:

    • IO-aware algorithm design: Flash Attention is the first attention algorithm to be explicitly designed around the GPU memory hierarchy. It treats HBM accesses as the primary cost metric, not FLOPs.
    • Tiling + online softmax: The combination of block-wise processing and incremental softmax computation enables exact attention without materializing the full attention matrix. This reduces memory from O(N²) to O(N).
    • Recomputation in backward: By saving only O(N) statistics from the forward pass and recomputing the attention matrix during the backward pass, Flash Attention eliminates the O(N²) memory cost of storing P.
    • Asymptotic optimality: The paper proves that Flash Attention’s IO complexity of O(N² d² / M) is tight — no exact attention algorithm can achieve fewer HBM accesses.
    • Practical impact: 2–4× wall-clock speedup, enabling 4× longer sequences, with zero loss in model quality. Now the default attention backend in major ML frameworks.

    Flash Attention v1 laid the foundation for Flash Attention v2 (which further optimized the tiling strategy with better work partitioning and reduced non-matmul FLOPs) and Flash Attention v3 (which exploits hardware features specific to Hopper GPUs like TMA and FP8). The core insight — that memory access, not arithmetic, is the bottleneck — has influenced algorithm design far beyond attention, including Flash FFT, Flash Linear Attention, and IO-aware approaches to other GPU kernels.

    Paper: Dao, T., Fu, D. Y., Ermon, S., Rudra, A., & Ré, C. (2022). FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness. NeurIPS 2022.
