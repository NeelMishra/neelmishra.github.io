Flash Attention 3

ML Series · Transformers · Variants

1 · Why FA3?

Flash Attention 2 achieved 73% of A100 peak throughput — an impressive result but one bounded by the A100's architecture. The NVIDIA H100 (Hopper) introduced three hardware features that FA2 couldn't exploit:

  • WGMMA — warpgroup matrix-multiply-accumulate instructions that operate on larger tiles
  • TMA — Tensor Memory Accelerator for asynchronous bulk data movement
  • FP8 tensor cores — double the throughput of FP16 at lower precision

FA3 is a ground-up rewrite targeting Hopper, achieving up to 85% of H100 peak in FP16 and 1.5–2× speedup over FA2 on H100. The algorithm is the same (tiled, IO-aware, online softmax); what changes is how the work is mapped to hardware.

2 · Hopper Architecture Overview

H100 vs A100: A100 H100 FP16 Tensor Core TFLOP/s 312 989 (3.2×) FP8 Tensor Core TFLOP/s — 1978 (new) HBM Bandwidth 2.0 TB/s 3.35 TB/s (1.7×) SRAM per SM 192 KB 228 KB (1.19×) New: TMA unit — ✓ (async DMA) New: WGMMA — ✓ (warpgroup matmul) New: Thread Block Clusters — ✓ (cross-SM cooperation)

The key observation: H100's tensor cores are 3.2× faster than A100's, but HBM bandwidth only grew 1.7×. This means attention is even more memory-bandwidth-bound on H100 than on A100. FA3 must maximize compute overlap with memory loads to stay efficient.

3 · WGMMA Instructions

On A100, tensor core operations use mma (matrix-multiply-accumulate) instructions that operate on 16×16 tiles within a single warp (32 threads). On H100, wgmma operates on a warpgroup (4 warps = 128 threads) with larger tiles:

A100 mma: M×N×K = 16 × 8 × 16 per instruction H100 wgmma: M×N×K = 64 × 256 × 16 per instruction (much larger tile) Key advantage: wgmma can read one operand directly from shared memory, avoiding the register file bottleneck. A100 flow: HBM → SRAM → registers → tensor core H100 flow: HBM → SRAM → tensor core (skip register load for one operand)
Why this matters for attention: In the inner loop, we compute S = Q · K^T and O += P · V. With WGMMA, K and V can stay in shared memory while Q and P are in registers. This halves the register pressure and allows larger tile sizes.

4 · TMA: Asynchronous Data Movement

The Tensor Memory Accelerator is a dedicated hardware unit that copies data between HBM and shared memory without involving any threads. On A100, threads must execute load instructions to copy data — tying up warps that could be doing math.

A100: Warp issues load instructions → occupies execution slots → stalls on memory H100: TMA engine copies data asynchronously → warps free for compute TMA supports: - Bulk 1D/2D/3D copies with automatic padding - Multicast to shared memory across thread block clusters - Arrives in shared memory with barrier synchronization

FA3 uses TMA to prefetch the next K/V block while the current block's matmul is running. This creates a software pipeline that hides memory latency behind compute.

5 · The 3-Stage Asynchronous Pipeline

FA3 structures each Q block's inner loop as a 3-stage pipeline. At any given moment, three different K/V blocks are in different stages of processing:

TMA prefetches K_{j+2}, V_{j+2} from HBM to shared memory. No threads involved — the TMA engine handles this asynchronously.

WGMMA computes S_{j+1} = Q · K_{j+1}^T using K_{j+1} already in shared memory. Softmax and rescaling happen here too.

WGMMA accumulates O += P_j · V_j using V_j in shared memory and P_j in registers. This is the final matmul for block j.

Pipeline timeline (each row = one iteration): Iter 0: [Load KV₂] [S = Q·K₁ᵀ] [O += P₀·V₀] Iter 1: [Load KV₃] [S = Q·K₂ᵀ] [O += P₁·V₁] Iter 2: [Load KV₄] [S = Q·K₃ᵀ] [O += P₂·V₂] ... All three stages execute simultaneously on different hardware units: TMA → shared memory load Tensor cores (warpgroup 1) → S matmul Tensor cores (warpgroup 2) → O matmul

6 · Ping-Pong Scheduling

To maximize SM utilization, FA3 uses ping-pong scheduling between two warpgroups within a thread block. While warpgroup A computes the matmul for its Q block, warpgroup B does softmax and rescaling (non-matmul) for its Q block, and vice versa:

Cycle Warpgroup A Warpgroup B 0 Matmul: S = Q_A·Kᵀ Softmax + rescale B 1 Softmax + rescale A Matmul: O_B += P_B·V 2 Matmul: O_A += P_A·V Matmul: S = Q_B·K'ᵀ 3 Matmul: S = Q_A·K'ᵀ Softmax + rescale B ... (alternating)

This ensures the tensor cores are never idle waiting for softmax — the non-matmul work for one Q block overlaps with the matmul work for another. Each warpgroup processes a different Q block, doubling the effective parallelism within an SM.

This is FA3's killer feature: On A100, non-matmul ops consumed ~30% of wall time despite being <5% of FLOPs. Ping-pong scheduling hides them completely behind matmuls, eliminating the non-matmul bottleneck that limited FA2 to 73% of peak.

7 · FP8 Attention

H100 supports FP8 (E4M3 and E5M2 formats) with double the tensor core throughput of FP16. FA3 is the first practical FP8 attention implementation:

FP8 challenges: - exp() in softmax needs higher precision → computed in FP32 - Accumulation (O += P·V) needs higher precision → FP32 accumulator - Q·K^T can overflow in FP8 → block-wise quantization FA3 approach: 1. Q, K, V stored in FP8 in HBM 2. WGMMA computes S = Q·K^T in FP8 → FP32 accumulator 3. Softmax computed in FP32 4. P quantized back to FP8 per-block 5. O += P·V in FP8 → FP32 accumulator 6. Final O converted to FP16/BF16 for output

To maintain numerical accuracy, FA3 uses incoherent processing (see next section) and per-block scaling factors. The result: ~1.5× speedup over FP16 FA3 with minimal quality loss.

9 · Incoherent Processing for FP8

FP8's limited range (E4M3: ±240 with 3 mantissa bits) makes quantization errors coherent — all values in a block are rounded in the same direction, causing systematic bias. FA3 mitigates this with incoherent processing:

Incoherent processing: 1. Generate random diagonal matrix D where D_ii ∈ {-1, +1} 2. Apply: Q' = Q · D, K' = K · D 3. Quantize Q' and K' to FP8 4. Attention is unchanged: Q'·K'^T = Q·D·D^T·K^T = Q·K^T (since D·D^T = I) Effect: randomizes the sign of each dimension before quantization, making rounding errors cancel out in expectation across dimensions.
Result: With incoherent processing, FP8 attention achieves the same perplexity as FP16 attention on LLaMA-style models, while running 1.5× faster. Without it, FP8 attention shows measurable quality degradation on long-context tasks.

11 · Limitations

  • Hopper-only: FA3 requires H100/H200 GPUs. It cannot run on A100 or older hardware — FA2 remains the best option there.
  • Kernel complexity: The WGMMA + TMA + ping-pong scheduling makes the CUDA kernel significantly more complex than FA2, harder to maintain and extend.
  • FP8 precision: While incoherent processing helps, FP8 attention may not be suitable for all applications (e.g., scientific computing, long-document retrieval with many near-duplicate passages).
  • Backward pass: The backward pass improvements are smaller (~1.3×) because the backward is more memory-bound and harder to pipeline.

12 · Summary

  • FA3 is a Hopper-native rewrite of Flash Attention that exploits WGMMA, TMA, and FP8.
  • The 3-stage async pipeline overlaps HBM loads (TMA), S matmul, and O matmul simultaneously.
  • Ping-pong scheduling between warpgroups hides non-matmul ops behind compute.
  • FP8 support with incoherent processing doubles throughput with minimal quality loss.
  • Result: 2× over FA2 on H100 (FP16), 3.4× with FP8, reaching 75% of H100 peak.
One-line summary: FA3 maps Flash Attention's IO-optimal algorithm onto Hopper's WGMMA + TMA + FP8 hardware to achieve 2–3× over FA2 on H100.
