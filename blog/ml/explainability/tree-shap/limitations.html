    Tree SHAP Limitations and Validation

    July 2026 · 15 min read

    Tree SHAP is exact for tree models under a chosen feature-missingness semantics, but exact does not mean assumption-free. The explanation still depends on what the baseline is, how absent features are handled, and how correlated features are interpreted.

    Two Semantics

    Mode What missing means Use when
    Tree-path-dependent Missing feature behavior follows training data counts through the tree paths. The tree's learned routing distribution is a reasonable background.
    Interventional Missing features are integrated using an explicit background dataset. A specific reference population should define the baseline.

    Validation Checks

    • Additivity check: baseline plus row SHAP values should match the model output.
    • Background check: changing the background population should be intentional and documented.
    • Correlation check: correlated features can split attribution in surprising ways.
    • Stability check: nearby rows with similar paths should have understandable attribution changes.

    ▶ Background Choice Changes the Baseline

    The same row can receive different signed contributions when the reference population changes, because the baseline changes.

    Best practice. Use Tree SHAP for row-level explanations of tree models, then validate additivity and document the background semantics. Do not interpret attributions as causal effects unless the data and model were designed for causal analysis.
