    Splitting & Stratification

    June 2026 · 12 min read

    Every model is judged on data it did not train on. That is why a test set must be held out before anything else in the preparation pipeline. Part 1 described the pipeline idea: split first, then learn every preprocessing parameter from the training rows only, then apply those learned parameters unchanged to held-out rows.

    The split is not just bookkeeping. It defines which examples are allowed to influence fitting, which examples guide design choices, and which examples remain untouched until the final evaluation. A clean split turns a training score into a more honest estimate of future performance.

    The Hold-Out Split

    A hold-out split partitions the dataset into separate roles. The training set fits model parameters such as weights, tree thresholds, or nearest-neighbor storage. The validation set tunes hyperparameters and choices such as regularization strength, feature options, model family, and early stopping. The test set is touched once, at the end, for an unbiased estimate of how the selected system performs on unseen data.

    Common ratios include 80/20 for train/test when a separate validation process is not needed, or 60/20/20 for train/validation/test when model selection is part of the workflow. Larger datasets can afford a smaller test fraction because the absolute number of held-out examples is still large. Smaller datasets often need careful validation because every row matters.

    For ordinary independent and identically distributed data, shuffle before cutting. Shuffling breaks accidental ordering by class, source, difficulty, or collection batch. Without it, a simple first-80-percent train and last-20-percent test split can accidentally place one kind of row in train and another kind in test.

    A Single Split Is Noisy

    A single hold-out score depends on which rows happened to land in the test set. If the test rows are a little easier than average, the score looks high. If they are a little harder than average, the score looks low. On small data, this swing can be several percentage points even when the model and pipeline are unchanged.

    This is sampling variance. The test score is an estimate, not the exact truth. The estimate improves as the test set grows, but holding out more data leaves less data for training. Cross-validation is the standard fix when a single split is too dependent on luck: it repeats the train/evaluate pattern across multiple folds and averages the results.

    Stratified Splitting

    Random splitting can drift on imbalanced data. Suppose the population has only 5% positive cases. A plain random split might leave the test set with 2% positives or 8% positives, especially when the dataset is small. The model would then be evaluated on a test distribution that does not represent the population it is meant to serve.

    A stratified split prevents that drift for classification labels. It partitions each class separately, takes the requested train/test fraction from each class, and then merges the pieces back together. If positives are 30% of the full dataset, both train and test should stay close to 30% positive.

    This matters most when a class is rare or the dataset is small. Rare classes have few examples, so moving just a handful of rows can change the measured class balance sharply. Stratification keeps the evaluation set representative, which makes metrics such as accuracy, precision, recall, F1, and ROC-AUC easier to interpret.

    On balanced, plentiful data, a plain split is fine, and that is what most examples use. Reach for stratification when a class is rare or data is scarce.

    Two Pitfalls

    The first pitfall is preprocessing before the split. If imputation, encoding, scaling, feature selection, or outlier filtering learns from the full dataset before rows are held out, then the test set has already influenced the transform parameters. This leaks the test set into training and violates the golden rule: fit transforms on train only, then apply them to validation and test.

    Time series are different. For time-ordered data, do not shuffle. Split by time so the model is trained only on the past and evaluated on the future. A random split can train on future patterns and then report an unrealistically strong score on older rows.

    The second pitfall is treating the test set like a validation set. If many models are repeatedly compared on the same test set, the team can accidentally overfit to that test set through decision making. The final test score is most meaningful when the test set is used once, after model choices are already fixed.
