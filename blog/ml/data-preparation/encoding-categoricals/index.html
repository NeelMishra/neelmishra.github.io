    Encoding Categorical Features

    June 2026 · 13 min read

    Most machine-learning models need numbers. A column such as color, city, or plan_type cannot usually enter a model as raw text, so each category must be encoded into a numeric representation. That conversion looks simple, but it changes the geometry that the model sees.

    The choice of encoding decides whether a false ordering is accidentally introduced. If red becomes 0, green becomes 1, and blue becomes 2, a numeric model may treat green as halfway between red and blue. That relationship is invented by the encoding, not present in the feature.

    Tree models, including decision trees, often tolerate integer labels better because a threshold simply partitions the integers into groups. A split such as color_id <= 1 separates one subset from another. Distance- and gradient-based models are less forgiving, because numeric gaps and directions directly influence their calculations.

    Label and Ordinal Encoding

    Label encoding maps each category to an integer id. For example, red, green, and blue may become 0, 1, and 2. The mapping is compact, easy to store, and useful when the encoded values are only identifiers.

    Label encoding is the natural fit for the target labels of a classifier. If a classifier predicts setosa, versicolor, or virginica, the target vector can store these classes as 0, 1, and 2. The model is not supposed to treat the target ids as measurements; they are class names written as integers.

    Ordinal encoding also maps categories to integers, but the order has meaning. A size feature with small < medium < large can be encoded as 0, 1, and 2 because the sequence reflects the domain. In that case, the increasing values communicate real information.

    The danger appears when label encoding is applied to an unordered input feature. It tells a numeric model that category 2 is "twice" category 1 and that category 1 sits halfway between 0 and 2. For colors, departments, product ids, or city names, that arithmetic is usually false.

    One-Hot Encoding

    One-hot encoding replaces a feature with $K$ categories by $K$ binary indicator columns. Exactly one of those columns is hot for each row. A color feature with red, green, and blue becomes three columns: is_red, is_green, and is_blue.

    One-hot encoding removes the false order. No category is numerically larger than another, and no category is placed between two others on a line. Each category gets its own coordinate direction, so the representation treats the names as distinct labels rather than measurements.

    The cost is width. A feature with $K$ categories adds $K$ columns, so high-cardinality features can blow up the dimension of the dataset. The representation is clean, but it may become expensive when $K$ is large.

    ▶ Label Encoding Invents an Order; One-Hot Does Not

    A label id places categories on a number line. A one-hot vector places them symmetrically, so no color becomes larger, smaller, or between the others.

    One-hot removes the artificial order by giving each category an independent indicator. The tradeoff is extra columns, especially when the category list is long.

    High Cardinality and Other Encoders

    Some categorical features have many possible values. Zip codes, user ids, product ids, and page urls can have hundreds, thousands, or millions of categories. One-hot encoding such a feature may create a matrix that is too wide, too sparse, or too easy for the model to memorize.

    Frequency encoding replaces a category by how often it appears in the training data. A common category receives a larger value than a rare category. This keeps the feature in one column, but it changes the meaning from identity to prevalence.

    Target encoding, also called mean encoding, replaces a category by a statistic of the target for that category. For binary classification, this may be the fraction of positive labels among training rows with that category. For regression, it may be the average target value. The idea is compact and powerful, but it is also easy to leak information.

    Target-encoding leakage. A target encoder must be fit on train only, following the golden rule. In cross-validation, the target statistic should be learned inside each fold from that fold's training rows, then applied to that fold's validation rows. If the validation target helps compute the encoding, the score is contaminated.

    These encoders are compromises. They reduce width, but they introduce statistics that must be learned carefully and regularized when categories are rare.

    Unseen Categories

    A category may appear at test time even though it never appeared during training. A new city, browser, product, or user id has no learned column in a one-hot encoder and no learned integer in a label map. The encoder must handle that case deliberately.

    The safe pattern is to fit the encoder on the training set, freeze the learned category list, then apply it to later data with an unknown-category policy. For one-hot encoding, an unknown value may become an all-zero row across the learned category columns. Another common choice is an explicit unknown bucket that collects rare or unseen values.

    The important part is not to crash, and not to refit on the test data. Relearning the mapping on test rows changes the feature space and breaks the separation between training information and evaluation information.

    Closing

    Categorical encoding is not just a formatting step. It chooses what relationships a model can see: a line with ordered ids, a set of separate indicator columns, or a statistic learned from the training data. The right choice depends on whether the category has order, how many values it has, and how sensitive the model is to numeric geometry.
