    Handling Missing Data

    June 2026 · 13 min read

    A model that expects a full numeric matrix cannot consume a gap. If one feature value is absent, arithmetic operations such as dot products, distances, gradients, and split criteria become undefined for that row. Missing entries must therefore be removed or filled before the model sees the data.

    The right choice depends less on the algorithm and more on why the data is missing. A random recording glitch is different from a skipped income field, and both are different from a sensor that fails only under certain weather conditions. Missingness has a mechanism, and that mechanism controls whether deletion is safe, imputation is reasonable, or the absence itself should become a feature.

    Why Data Is Missing: MCAR, MAR, MNAR

    Missing data is usually described by three mechanisms. They ask what determines the missingness indicator $M$, where $M=1$ means a value is missing and $M=0$ means it is observed.

    Missing completely at random (MCAR) means missingness is independent of everything: observed columns, unobserved columns, and the missing value itself. A file import may randomly drop a small number of cells, or a lab machine may fail on a random subset of samples. Under MCAR, dropping rows is statistically unbiased because the remaining rows are still representative of the full population. The practical cost is efficiency: data is thrown away, variance increases, and rare classes may become even rarer.

    Missing at random (MAR) means missingness depends on observed features, but not on the unobserved value after those features are known. For example, a sensor may fail more often in cold weather, and temperature is recorded in another column. The missingness is not random in the everyday sense, but it is explainable from columns already present in the dataset. Under MAR, imputation can use those observed columns: median fill may be acceptable for a baseline, while KNN or regression imputation can borrow information from correlated variables.

    Missing not at random (MNAR) means missingness depends on the unobserved value itself. High earners may decline to report income, patients with severe symptoms may skip optional surveys, or expensive houses may hide their sale price. This is the hardest case because the missing value is systematically different from the observed values. Naive imputation is biased, and the fact that a value is missing may carry predictive signal. In practice, MNAR often calls for a missingness indicator, domain assumptions, sensitivity analysis, or a model that treats the missing pattern as part of the data generating process.

    Deletion vs Imputation

    Listwise deletion drops every row with any missing value. It is simple, transparent, and unbiased under MCAR. It is also wasteful: one gap in one column can remove an otherwise useful row. As the number of columns grows, the chance that a row has at least one gap also grows, so listwise deletion can shrink the dataset dramatically.

    Deletion also fails under MAR and MNAR. If sensor failures are more common in cold weather, deleting missing sensor rows changes the temperature distribution. If high earners skip income, deleting those rows changes the income distribution. The resulting training set no longer represents the population the model will face.

    Imputation fills each gap with an estimate. Quick fills use the mean, median, or mode of the observed training values. The mean is common for roughly symmetric numeric columns, the median is safer for skewed numeric columns, and the mode is a simple choice for categorical columns. These methods are fast and easy to explain, but they do not recover row specific detail.

    Model-based imputation uses other columns to estimate the missing value. KNN imputation fills from similar rows, while regression imputation learns a relationship between the incomplete column and correlated complete columns. These methods can work well under MAR because the missingness is explainable from observed data, but they add complexity and must still be fit only on training data.

    A useful addition is a binary was-missing indicator column. It records whether the original value was absent before imputation. This preserves information that plain imputation erases, and it is especially important when MNAR is possible. The filled value lets the numeric pipeline run; the indicator lets the model learn that the gap itself may matter.

    ▶ Mean vs Median Imputation, and the Outlier Problem

    A single numeric column has values 1, 2, missing, 4, 100, missing. The outlier makes the mean move away from the real cluster, while the median stays near the typical values.

    The median is robust to outliers, which makes it a safer default than the mean for skewed columns.

    Fit on Train

    Golden rule. The fill value, whether it is a mean, median, mode, or a learned imputation model, must come from the training column only and then be applied to the test column. This is the same rule as the rest of the preprocessing pipeline in Part 1: fit on train, apply to test.

    If the test column helps choose the fill value, the evaluation has leaked information from the future into the preprocessing step. Even a simple median can leak when computed on the full dataset before splitting. The correct pattern is mechanical:

    train_fill = median(train_column_without_missing)
train_column = fill_missing(train_column, train_fill)
test_column  = fill_missing(test_column, train_fill)

    The same structure applies to model-based imputation. The imputer learns relationships from the training rows only, then transforms both training and test rows with those learned parameters.

    Closing the Loop

    Missing values are not just blanks to patch. They are clues about the data collection process. MCAR makes deletion defensible but inefficient, MAR makes imputation from observed columns useful, and MNAR warns that absence itself may be predictive. A practical baseline often starts with median or mode imputation plus a was-missing indicator, then improves only if the missingness mechanism and validation results justify more complexity.
