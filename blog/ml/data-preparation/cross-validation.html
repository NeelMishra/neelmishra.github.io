    Cross-Validation

    June 2026 · 14 min read

    A single train/test split, introduced in Splitting, gives one performance number. That number is useful, but it depends on which rows happened to land in the test set. If a few unusually easy or unusually hard examples fall there, the reported score can move even though the model and features did not change.

    Cross-validation reduces that dependence on luck. Instead of trusting one split, it averages performance across many train/validation splits. Each split asks the same question, how well does the model perform on rows it did not train on? The average becomes a lower-variance estimate of generalization, and the spread of the fold scores shows how much the estimate wobbles across different held-out rows.

    K-Fold Cross-Validation

    In k-fold cross-validation, the data is partitioned into $k$ roughly equal folds. The model trains on $k-1$ folds and validates on the one fold held out. Then the held-out fold rotates: fold 1 validates once, fold 2 validates once, and so on until every fold has been the validation fold exactly once.

    If $ ext{score}_i$ is the validation score from fold $i$, the cross-validation estimate is the average:

    $$\text{CV} = \frac{1}{k}\sum_{i=1}^{k}\text{score}_i$$

    The spread of the scores is just as important:

    $$\operatorname{std}(\text{score}_1,\dots,\text{score}_k)$$

    Every row is validated exactly once and trained on $k-1$ times. That means all rows contribute to evaluation, all rows contribute to training in most rounds, and the model is never tested on a row it trained on.

    Variants

    Leave-one-out cross-validation sets $k=N$, so each validation set contains exactly one row. It is nearly unbiased because each model trains on almost the whole dataset, but it is expensive and can have high variance because every score depends on a single example.

    Stratified k-fold preserves class ratios inside each fold. For classification, this is usually the default choice, and it is the fold-level counterpart of stratification in a single split. If the original dataset has 20 percent positives, each fold should stay near that ratio.

    Grouped k-fold keeps related rows together. If multiple rows come from the same patient, user, document, household, or device, they should not be split across train and validation. Otherwise the model can learn entity-specific clues and appear better than it really is.

    Time-series cross-validation respects chronology. An expanding-window scheme trains on earlier observations and validates on later observations, then expands the training window forward. It never validates on the past, because future information must not be used to predict earlier events.

    The choice of $k$ has a bias-variance tradeoff. A small $k$ trains each model on less data, which can make the estimate more biased. A large $k$ trains on more data, but it costs more and the fold scores can become more variable. Values of 5 or 10 are the usual compromise.

    The Leakage Rule, Again

    Cross-validation does not remove the preprocessing rule from Part 1. It makes the rule more important. Every transform that learns a parameter, such as an imputer median, a scaler mean, or an encoder category list, must be refit separately inside each fold on that fold's training portion.

    Inside-the-fold rule. For each fold, fit preprocessing on the training folds only, transform both the training folds and the validation fold with those fitted parameters, then train and score the model. Preprocessing the whole dataset once before folding leaks every validation fold into the parameters.

    This also affects hyperparameter selection. If cross-validation is used to choose the best model settings, the final reported performance should come from nested cross-validation: an inner loop tunes hyperparameters, and an outer loop evaluates the full selection process honestly.
