    Data Preparation: Foundations

    June 2026 · 15 min read

    Models get the headlines, but most of the work in a machine-learning project is getting the data into a state a model can use. Raw data arrives with text fields, gaps, wildly different scales, and a column order that may leak the answer. This series covers the preparation and preprocessing that comes first, the conceptual "why" behind each step. Every part has a hands-on counterpart that builds the same step from scratch in C++, starting with From Scratch · C++ · Data Preprocessing.

    The Preprocessing Pipeline

    Preprocessing is best seen as a pipeline: raw data flows through a fixed sequence of transforms and arrives as a clean numeric matrix the model can train on. A typical ordering is split, impute, encode, scale, then model. Two jobs run through it: shaping the data (filling gaps, turning categories into numbers, putting features on a comparable scale) and partitioning it so the model can be trained and judged honestly.

    The animation traces that pipeline. The crucial thing it highlights is where each transform's parameters come from: a median for imputation, a category list for encoding, a mean and standard deviation for scaling. Every one of them is learned from the training data only, then applied unchanged to the test data.

    ▶ The Pipeline, and Where Parameters Come From

    Each step lights up a stage. After the split, the train and test lanes are processed by the same transform, but its parameters (median, categories, mean/std) are always learned from the train lane and then applied to test.

    The Golden Rule: Fit on Train, Apply to Test

    The single most important principle in preprocessing is this: any statistic a transform needs must be computed from the training set, then applied unchanged to the test set. Formally, a transform has a fit step that learns parameters $\theta$ from the training data, and a transform step that applies $\theta$:

    $$\theta = \text{fit}(X_{\text{train}}), \qquad X'_{\text{train}} = \text{transform}(X_{\text{train}}, \theta), \qquad X'_{\text{test}} = \text{transform}(X_{\text{test}}, \theta).$$

    The test set is transformed with the training parameters; it never contributes to $\theta$. Breaking this rule is data leakage: information from the test set sneaks into the training process, and the reported score becomes optimistic in a way that will not survive contact with real, unseen data.

    A concrete leak. Suppose a feature is standardized using the mean and standard deviation of the entire dataset, train and test together. The test points have now influenced the scaling that the model sees, so the evaluation is no longer measuring performance on truly unseen data. The fix is mechanical: compute the mean and standard deviation on the training rows only, then scale both sets with those numbers.

    Leakage Hides in Cross-Validation

    The subtle version of this mistake appears with cross-validation. If preprocessing is done once, on the whole dataset, before the data is split into folds, then every validation fold has already leaked into the transform's parameters. The correct pattern refits the entire preprocessing pipeline inside each fold, on that fold's training portion only:

    for each fold:
    train_part, val_part = split_for_this_fold(data)
    theta = pipeline.fit(train_part)        # learn params on this fold's train only
    model.fit(pipeline.transform(train_part, theta))
    score = evaluate(model, pipeline.transform(val_part, theta))   # val never seen by fit

    This is why mature libraries wrap preprocessing and the model into a single "pipeline" object: it guarantees that fit only ever sees the training rows of whatever split it is handed, fold by fold.

    What the Series Covers

    The remaining parts take the pipeline stages one at a time, each pairing the concept here with the matching from-scratch implementation:

    • Splitting & Stratification, partitioning data so the test set is representative.
    • Feature Scaling, putting features on a comparable scale for distance- and gradient-based models.
    • Encoding Categorical Features, turning text labels into numbers without inventing a false order.
    • Handling Missing Data, the mechanisms behind gaps and how to fill them.
    • Cross-Validation, estimating performance with less luck than a single split.

    None of these needs a heavyweight library to understand: a split is a shuffle and a cut, scaling is a mean and a standard deviation, encoding is a lookup table. Building each by hand, as the C++ companion does, makes the golden rule concrete, because you can see exactly which numbers come from the training set.
